# yifanswe.github.io

Personal notes & essays site, served by GitHub Pages at
[yifanswe.github.io](https://yifanswe.github.io).

Remote: `git@github.com:yifanswe/yifanswe.github.io.git`

## How the site is built

This is a **hand-maintained static site**. GitHub Pages serves the committed
files as-is — there is **no Jekyll/Hugo build step** running on the server.
Every content page is a plain `index.html` that has been committed to the repo.

Content pages are **generated from Markdown source docs** and then committed.
The conversion is reproducible via [`tools/build_site.py`](tools/build_site.py),
which is how the `system-design-patterns/` section was produced and is the
documented recipe for the existing `backend-fundamentals/` and
`common-backend-systems/` sections (those were originally produced ad hoc with
the same template and Markdown settings).

### Directory layout

```
/                         repo root
├── index.html            homepage: post list + client-side search + year filter
├── backend-fundamentals/ section: <slug>/index.html per doc + section index.html
├── common-backend-systems/
├── system-design-patterns/   ← generated by tools/build_site.py
├── css/                  stylesheets only (style.css + style/*.min.css). NOT content.
├── js/                   jquery, highlight.js, toc.js, search.js, totop.js
├── fonts/  images/  2020/  archives/
└── tools/
    └── build_site.py     Markdown → HTML generator
```

> Note: content is **never** placed under `css/` — that folder holds only
> stylesheets. Each content doc lives at `/<section>/<slug>/index.html` so its
> URL is a clean `/<section>/<slug>/`.

### The Markdown → HTML conversion

`tools/build_site.py` uses [python-markdown](https://python-markdown.github.io/)
(verified with version 3.9) with the extensions:

| Extension    | Why |
| ------------ | --- |
| `extra`      | GitHub-style tables, fenced code blocks, footnotes, etc. |
| `toc`        | Assigns heading `id` slugs (`§1. What X Is` → `1-what-x-is`). The client-side `js/toc.js` reads these to build the in-page Contents nav. |
| `sane_lists` | Predictable ordered/unordered list handling. |

For each source doc the generator:

1. Strips the leading `# H1` (the template renders the title separately).
2. Converts the Markdown body to HTML.
3. Rewrites in-section sibling links so cross-references resolve on the live
   site: `foo.md` → `../foo/`, `foo.md#anchor` → `../foo/#anchor`,
   `index.md` → `../`. Links pointing outside the section (e.g.
   `../questions/...`) are left untouched.
4. Wraps the body in the article template (topbar nav, `/css/style.css`,
   highlight.js for code, sticky + inline TOC scaffolding, footer, to-top
   button).
5. Writes `<section>/<slug>/index.html`, and a `<section>/index.html` landing
   page listing every doc as a card.

### Code blocks & syntax highlighting

Code blocks are highlighted **client-side** by highlight.js; the generators do
not run a server-side highlighter. Highlighting is **selective** — only blocks
with an explicit language fence (` ```python `, ` ```sql `, ...) are themed.
Bare ` ``` ` blocks are left as plain monospace, because these docs use them
for ASCII architecture diagrams and highlight.js auto-detection mis-parses
those as code (coloring words like "table"/"insert"). The logic lives in
[`js/highlight-init.js`](js/highlight-init.js), loaded by every content page
instead of the old `hljs.initHighlightingOnLoad()`.

The theme is the dark **monokai-sublime** stylesheet (`/css/style/monokai-sublime.min.css`),
chosen to match the dark `pre` background (`#1a1a1a`) set in `style.css`. To
highlight a block, fence it with a language; to keep a diagram plain, leave the
fence bare or use ` ```text `.

### Regenerating / adding a section

1. Edit the `SECTIONS` list in `tools/build_site.py`: set the output `slug`,
   `title`, `nav_label`, the Markdown `src_dir`, an `intro` blurb, and the
   ordered `docs` list of `(filename, page_title, listing_description)`.
2. Run:

   ```bash
   pip install markdown          # once, if not already installed
   python3 tools/build_site.py
   ```

3. Register the section on the homepage `index.html`:
   - Add a `<a href="/<section>/" …>` link in `.topbar-nav`.
   - Add one `.post-item` per page in `#posts` (the homepage search and year
     filter operate on these items' `data-title` / `data-year` attributes).

### Bulk-importing a doc corpus (`tools/build_questions.py`)

`build_site.py` takes a curated, hand-written list of docs. For a *large*
corpus where hand-listing titles is impractical, [`tools/build_questions.py`](tools/build_questions.py)
auto-discovers docs instead: it scans source directories, derives each page
title from the first Markdown `# H1`, normalizes the filename into a slug, and
merges everything into one flat list under `common-backend-systems/`. It then
**injects** title-only cards into the existing section index and the homepage's
Common Backend Systems group (between `<!-- QUESTIONS:START/END -->` markers, so
re-runs are idempotent and the 5 hand-written component pages are preserved).
Run it the same way: `python3 tools/build_questions.py`.

### Deploying

GitHub Pages publishes whatever is on the default branch. Commit and push:

```bash
git add .
git commit -m "Add system-design-patterns section"
git push origin <default-branch>
```
