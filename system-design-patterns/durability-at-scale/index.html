Durability at Scale

2026/05/31  ·  System Design Patterns
Contents
Contents

Designing storage so that bytes you accepted never disappear, even at petabyte scale where disks die routinely, bits silently rot, and whole racks or regions can vanish at once.

This pattern is the durability-first lens applied to bulk storage. For terminology see the Staff System Design Glossary. Related question docs:

  • Amazon S3 — eleven nines, erasure coding, cross-region replication.
  • Hadoop Distributed File System (HDFS) / Google File System (GFS) — block replication, rack-aware placement, BlockScanner.
  • Google BigTable — durability delegated to GFS; SSTables + commit log.
  • Dropbox — file sync, chunk-level deduplication, durable backup.
  • YouTube — multi-region video storage with cold tiers.
  • Instagram — photo storage durability.
  • Amazon Dynamo — replication and anti-entropy in a quorum store.

1. The Fundamental Hardness

Eleven nines of durability — 99.999999999% — is the number Amazon Simple Storage Service (S3) advertises. The unit matters: it is a per-object, per-year expected loss probability of 10^-11, not a per-system uptime number. The literal claim is: if you store 10,000,000,000 (ten billion) objects, the expected number lost per year is 0.1 (10^10 * 10^-11). Equivalently, the expected time to lose a single object from a 10,000-object bucket is roughly 10,000,000 years. Note this is also distinct from availability — S3 advertises eleven nines of durability but only four nines of availability; durability says "we still have your bytes," availability says "you can read them right now." These are different design problems with different mitigations.

The naive answer is "store three copies." This works fine at gigabyte scale and is genuinely fine up to roughly tens of petabytes — 3x across three Availability Zones (AZs), with scrubbing, gets you to eleven nines on paper. Where it stops being the right answer is the storage bill. At 100 petabytes (PB) raw, 3x is 300 PB of disk; at exabyte scale 3x is hundreds of millions of dollars per year in disk that erasure coding could shave by half. The interview question is not "is replication wrong" — it is "at what scale does the cost math justify the complexity of erasure coding, and how do you defend the durability number under realistic failure correlations." Industry Annualized Failure Rate (AFR) for spinning disks is typically 1-4%; at 100,000 disks you lose roughly 3-11 per day, and the system must repair faster than the next coincident failure window. Failures are correlated, not independent. A rack switch dies and takes 40 nodes with it. A power phase trips and takes half a datacenter. A bad firmware push corrupts a model of drive across the fleet. Independent-failure math hides all the interesting risk.

Bit rot — silent data corruption on disk where bytes flip without any read error — is real. Studies of large fleets (Bairavasundaram et al., FAST 2008; Google's "What does an exabyte of storage smell like?") put silent corruption rates between 10^-14 and 10^-15 per bit read. At exabyte scale you encounter silent corruption every day. If you never read the data, you never notice — until you do, and then the parity you would have used to repair it is also rotted. Durability is not a one-time encode decision; it is a continuous verification job.

The interview question is therefore not "do you know what replication is." It is "how do you stack the layers — replication, erasure coding, geo-replication, scrubbing, anti-entropy, durable Write-Ahead Logs (WALs), durability classes — so that under realistic failure models (correlated rack failures, regional disasters, silent corruption, repair storms) you can defend a 1-10^-11 annual loss number to the interviewer with arithmetic, not vibes."

2. How to Recognize It in the Interview

Prompt phrases

  • "It must never lose data."
  • "Eleven nines of durability."
  • "Customers pay us to keep this safe forever."
  • "Petabyte" / "exabyte" / "billions of objects."
  • "Backup," "archive," "compliance retention," "legal hold."
  • "Survive a region failure."
  • "Storage cost is dominant" (signals erasure coding pressure).
  • "Append-mostly," "write-once-read-many" (signals object store, blob store).

Architectural smells

  • Storage is the bottleneck dollar-wise, not Central Processing Unit (CPU) or network.
  • Data is large and immutable or append-only (videos, photos, logs, backups).
  • Read traffic is bursty or cold; most bytes are rarely accessed.
  • A single object loss is a customer-visible failure or a compliance incident.
  • The system spans multiple Availability Zones (AZs) or regions by requirement.

Example interview prompts where this is core

  1. Design S3 / a planet-scale object store.
  2. Design GFS / HDFS / a distributed file system for analytics.
  3. Design YouTube or Netflix video storage (with cold tiers).
  4. Design a photo storage system like Instagram or Facebook Haystack.
  5. Design Dropbox / Google Drive (chunked file sync with durable backend).
  6. Design a backup / archive service (Glacier-style cold storage).
  7. Design a blockchain ledger or append-only audit log.
  8. Design a metrics / logs long-term store like Datadog's archival tier.

3. 60-Second Spoken Framing

The invariant is that any byte we acknowledged to the client must survive any single rack failure with very high probability and any single Availability Zone failure with high probability, and we should be able to recover from a region loss if the customer paid for it. I will hit eleven nines by stacking three layers: erasure coding across failure domains inside a region for cost-efficient redundancy, asynchronous cross-region replication for regional disasters, and continuous background scrubbing plus Merkle-tree anti-entropy to catch bit rot before the redundancy is exhausted. I will keep a durable Write-Ahead Log on the metadata path so an in-flight write is not lost during a failover, and I will tier cold data to cheaper media with the same durability target but worse latency. The math I want to walk through is the per-object annual loss probability under correlated rack failure plus repair time, because that is what drives the erasure coding rate M+K.

4. Canonical Toolkit

4.1 Replication (Nx copies)

Keep N identical copies on different failure domains. Reads can go to any copy; writes go to all N (or a quorum W).

  • Pros: simple, fast reads (no decode), fast repair (just copy one block), easy to reason about.
  • Cons: Nx storage overhead. At 3x and 100 PB that is 200 PB of "wasted" disk.
  • Where it shines: small objects (encoding overhead dominates the data), hot data (any replica can serve), low-latency writes (no parity compute).

4.2 Erasure Coding (Reed-Solomon M + K)

Split an object into M data chunks; compute K parity chunks; store all M+K chunks on different failure domains. Tolerate any K losses. Storage overhead is (M+K)/M.

Common configurations:

Scheme Overhead Tolerates Used by
3x replication 3.00x 2 losses HDFS default, Dynamo, Cassandra
Reed-Solomon 6+3 1.50x 3 losses HDFS Erasure Coding, Ceph
Reed-Solomon 10+4 1.40x 4 losses S3 large objects
Reed-Solomon 12+4 1.33x 4 losses Facebook f4 warm storage
Reed-Solomon 17+3 1.18x 3 losses Google ColossusFS (typical)

Pros: lower storage cost than replication for the same fault tolerance. 10+4 survives any 4-chunk loss with only 1.4x overhead; equivalent replication tolerance would be 5x.

Cons (this is not a free lunch — call these out explicitly in the interview):

  • Read tail latency. A read must reach M chunks; one slow shard makes the whole read wait. Replication can hedge to the fastest of N replicas — erasure coding cannot, unless you over-read (fetch M+1 and race), which costs network. EC read p99 is structurally worse than replicated read p99.
  • Repair amplification. Rebuilding one chunk requires reading M other chunks. At 10+4, a single disk-loss repair drags 10x the data of replicated repair across the network. At petabyte scale this can become the dominant network cost.
  • Write latency and CPU. Encode is a Galois-field matrix multiply (cheap per byte, but non-zero); writes block on M+K chunk placements across M+K failure domains instead of N replicas. The write commit-point is the slowest of M+K, not the fastest of N.
  • Small-object overhead. Chunk metadata and the M+K placement records can exceed the data for objects under ~1 MB. Production object stores route small objects through a replicated sidecar path.

4.3 Conceptual Python Sketch of Reed-Solomon M+K

Real implementations (Jerasure, ISA-L, Backblaze's reedsolomon) operate over Galois Field GF(2^8). The conceptual core is a Vandermonde matrix multiplication: encode treats data chunks as a vector and multiplies by an (M+K) × M matrix whose top M × M block is the identity (so the first M rows of output equal the original data, making it a systematic code). Any M of the M+K output rows form an invertible matrix, so any M surviving chunks can reconstruct the original.

# Conceptual only. Real code uses GF(2^8) arithmetic, not floats.
import numpy as np

def vandermonde_systematic(m: int, k: int) -> np.ndarray:
    """Return an (m+k) x m generator matrix whose top m rows are I_m."""
    # Top: identity so encoded chunks 0..m-1 equal original data (systematic).
    top = np.eye(m)
    # Bottom: a Vandermonde-style parity matrix. Any m rows of the
    # full (m+k) x m matrix must be invertible; Vandermonde guarantees this.
    bottom = np.array([[(i + 1) ** j for j in range(m)]
                       for i in range(k)], dtype=float)
    return np.vstack([top, bottom])

def encode(data_chunks: list[bytes], m: int, k: int) -> list[bytes]:
    """Encode m data chunks into m+k chunks. m original + k parity."""
    G = vandermonde_systematic(m, k)
    # Each chunk treated as a byte vector; multiply column-wise.
    data_matrix = np.stack([np.frombuffer(c, dtype=np.uint8)
                            for c in data_chunks])
    encoded = G @ data_matrix      # (m+k) x chunk_size
    return [row.astype(np.uint8).tobytes() for row in encoded]

def decode(surviving_chunks: dict[int, bytes], m: int, k: int) -> list[bytes]:
    """Recover the original m data chunks from any m of the m+k chunks."""
    G = vandermonde_systematic(m, k)
    indices = sorted(surviving_chunks)[:m]
    # Submatrix of the rows we received; invert it to solve for data.
    sub_G = G[indices, :]
    sub_data = np.stack([np.frombuffer(surviving_chunks[i], dtype=np.uint8)
                         for i in indices])
    recovered = np.linalg.solve(sub_G, sub_data)
    return [row.astype(np.uint8).tobytes() for row in recovered]

Production codes do not use floats — that loses information and is non-deterministic. They use GF(2^8) field arithmetic where addition is XOR and multiplication is a small table lookup. The conceptual takeaway: encode is a matrix multiply, decode is solving a linear system over the surviving rows.

Lesson: K should exceed the largest correlated failure domain you
want to lose. K=3 with 3 AZs is too tight; production uses K>=4 or
spreads chunks across MORE AZs to avoid losing K chunks at once.

4.5 Geo-Replication (Cross-Region, Async)

Replicate the durable object (after local encoding) to a second region asynchronously. Default is "eventually durable in two regions"; some classes commit synchronously to both for higher Recovery Point Objective (RPO) guarantees.

The async/sync split is a direct durability-vs-latency trade — don't paper over it:

  • Async cross-region. Cheap, write latency stays local (single-region commit). The cost is a non-zero RPO: in a sudden region loss, the trailing writes that had not yet been shipped are lost. Defensible only when the product can tolerate seconds-to-minutes of write loss on a region-loss event.
  • Sync cross-region. Write latency now includes inter-region Round-Trip Time (RTT) of 60-200 ms (US-east to US-west) on every commit. RPO is zero, but throughput drops and write tail latency grows substantially. Used for regulatory requirements (financial ledgers, healthcare) and for products that genuinely cannot tolerate a non-zero RPO.
  • Cross-region replication is not "extra nines" on the same durability number. It is a separate defense against a different failure mode (region loss). The advertised eleven-nines durability is a per-object disk/rack number; cross-region is what backs the RPO/Recovery Time Objective (RTO) for regional disasters.

4.6 Write-Ahead Log (WAL) + Checkpoints

Every state change is appended to a durable log before being applied to in-memory state. On crash, replay log forward from the last checkpoint. The WAL itself must be durable — usually 3x replicated or written to a durable log service (Apache BookKeeper, AWS Kinesis, Apache Kafka with acks=all).

  • BigTable: every write hits the GFS commit log before the memtable update is acknowledged. A tablet server crash replays the commit log on a new server.
  • Spanner, CockroachDB: the Raft log is the WAL.
  • The WAL ensures in-flight writes survive a failure — replication alone does not, because a write acknowledged from only one replica might be lost.

4.7 Background Scrubbing / Bit Rot Detection

Continuously read every chunk, verify its stored checksum (typically Cyclic Redundancy Check 32 (CRC32C) or 256-bit Secure Hash Algorithm (SHA-256)), and re-replicate on mismatch. Scrub the entire fleet every 14-30 days. Without scrubbing, parity also rots and you discover loss only on read — by which time it is too late.

Scrubbing is not free. A 30-day scrub cycle on 100 PB is a sustained ~38 Gbps of background reads (100 PB / 30 days), which competes with serving I/O on the same disks. Production systems throttle scrub to a small fraction of disk bandwidth (often < 5%), accepting longer cycle times for cold tiers. The interview honesty move: scrubbing is mandatory for durability but it is a real, persistent slice of the I/O budget, not a side process.

Scrub loop per disk:
  for chunk in chunks_on_disk:
      stored_hash = chunk.metadata.sha256
      computed = sha256(chunk.bytes)
      if stored_hash != computed:
          mark_chunk_corrupt(chunk)
          schedule_repair(chunk)        # reconstruct from peers
          alert("silent corruption", disk_id, chunk.id)

4.8 Merkle Trees / Anti-Entropy

Each replica or chunk-group maintains a Merkle tree of its contents. Peers exchange tree summaries; mismatches identify the specific divergent keys without exchanging full state. Dynamo and Cassandra use Merkle anti-entropy to repair long-lived divergence; S3 uses similar mechanisms to verify cross-region replicas match. See Amazon Dynamo for the canonical writeup.

4.9 Multi-AZ Writes with Quorum

A write is durable when W of N replicas, spread across N distinct AZs, have persisted it. Common: N=3, W=2 (write returns when 2 of 3 AZs have it) or N=5, W=3 (Spanner / Aurora style). The point is that the durability commit cannot complete from a single AZ.

4.10 Durability Classes / Tiering

Not every byte deserves eleven nines at hot-tier cost. Production object stores expose classes:

Class Durability Availability Retrieval Cost Use
S3 Standard 11 nines (multi-AZ) 4 nines ms 1.0x Active data
S3 Standard-IA (Infrequent Access) 11 nines (multi-AZ) 3 nines ms 0.55x Backups
S3 One Zone-IA 11 nines conditional on the AZ surviving 2 nines ms 0.45x Reproducible data
S3 Glacier Instant Retrieval 11 nines (multi-AZ) 4 nines ms 0.25x Archives, instant read
S3 Glacier Deep Archive 11 nines (multi-AZ) n/a hours 0.05x Cold archives

A subtlety to call out: S3 One Zone-IA does not survive an AZ loss. Its eleven-nines number applies only to per-object failure inside the surviving AZ; it is not equivalent durability to the multi-AZ classes. That is the explicit price for the 0.45x cost.

Cold-tier migration policies (lifecycle rules) move data automatically: "after 30 days no access, move to IA; after 90 days, to Glacier." The total dollar saving at 100 PB is dramatic; the operational risk is restoration latency from Glacier (minutes to hours) and the retrieval-cost spike if a customer suddenly needs cold data back. Glacier and Standard share the same durability class; what differs is retrieval latency and per-GB retrieval cost.

4.11 Lineage / Write-Fence for In-Flight Writes During Failover

When a primary fails and a standby takes over, you must not lose writes that the primary acknowledged but had not yet replicated. Two techniques work together:

  • Write fence. The old primary's writes are rejected the moment the new primary takes over (via lease expiration in Chubby / ZooKeeper / Etcd, or a fencing token bumped on the metadata side). This prevents split-brain — the old primary cannot keep acking writes after losing leadership — but on its own it does not save writes that were acked before the fence.
  • Lineage on the WAL. Every write carries a monotonic Log Sequence Number (LSN). A write is only acknowledged to the client after it is replicated to a quorum of WAL replicas. The new primary on takeover reads the highest committed LSN visible to the quorum and replays forward from the last checkpoint. Anything not present in a quorum of WAL replicas was never acknowledged, so the client must retry (which is safe under idempotency).

The combination matters: the fence stops the old primary from creating new acked writes, and the WAL quorum invariant guarantees that all previously acked writes are visible to whoever wins the election.

Pairs with Idempotency / Exactly-Once — clients retrying after a failover need idempotent writes to avoid duplicates.

4.12 Local Reconstruction Codes (LRC) — Tiered Parity

Pure Reed-Solomon repair is expensive: rebuilding one chunk costs reading M other chunks. At exabyte scale, repair bandwidth dominates the network budget. Local Reconstruction Codes (used by Azure Storage, Facebook's f4) add a layer of local parity over small subgroups of data chunks, plus global parity over the whole stripe.

LRC (12,2,2):  12 data chunks, 2 local parity, 2 global parity
                = 16 chunks total, overhead 1.33x

   group A: D0..D5  ─►  LA  (local parity for group A)
   group B: D6..D11 ─►  LB  (local parity for group B)
   all:     D0..D11 ─►  G0, G1  (global parity)

   Single chunk loss in group A: rebuild from D0..D5 + LA  (6 reads)
                                 instead of all 12 + globals (12 reads)
   Two chunk losses: must use global parity; still recoverable.

The single-chunk failure case (overwhelmingly the most common) reads M/2 chunks instead of M, halving repair bandwidth. The tradeoff vs pure Reed-Solomon: LRC trades a slightly weaker worst-case fault tolerance for dramatically cheaper common-case repair. Reed-Solomon 12+4 tolerates any 4 chunk losses; LRC (12,2,2) tolerates any 4 losses only in some configurations — adversarially placed losses spanning both local groups can be unrecoverable with fewer than 4 lost. LRC is the right choice when single-chunk-loss repair bandwidth is your bottleneck (typical at exabyte scale); pure Reed-Solomon is the right choice when worst-case fault tolerance dominates. The interview move is to name the tradeoff, not to claim LRC is strictly better.

5. Worked Example: Naive → Broken → Fixed

Scenario

A 100 PB object store, 100,000 spinning disks at 4 TB each, AFR 4%, distributed across 3 Availability Zones in one region. Repair bandwidth budget is 10 Gbps per node. Average object size 1 MB, so ~100 billion objects.

Naive: 3x replication across 3 AZs

Cost math:

  • Raw data: 100 PB.
  • Stored: 300 PB (3x).
  • At $0.02 / GB / month raw, monthly cost: 300 * 10^6 * $0.02 = $6M/month.
  • Network: writes amplify 3x; reads do not.

Durability math (back-of-envelope):

  • 100,000 disks, AFR 4% → expected 4,000 disk failures per year, ~11 per day.
  • A 4 TB disk holds ~4 * 10^6 chunks (1 MB each); each chunk has 2 surviving replicas.
  • Repair time for one disk: 4 TB / 10 Gbps ≈ 55 minutes of pure transfer; in practice contention and metadata work push it to several hours.
  • Per-disk hourly failure probability under AFR 4% is ~4.6 * 10^-6. For a specific chunk to be permanently lost, both surviving replicas must fail inside the repair window. Under the (false but useful for sanity) independent-failure assumption with a 1-hour window, that joint probability is ~(4.6 * 10^-6)^2 ≈ 2 * 10^-11 per chunk per failure event. Multiplied across ~4 * 10^9 total events per year per disk, the independent-failure number lands well below eleven nines.

So 3x looks fine on the independent-failure spreadsheet. Where the spreadsheet lies:

Broken: correlated failures + repair storms

  • Correlated failures: HDFS default placement puts replicas as (local, remote-rack, remote-rack-same). A rack power failure takes 2 of 3 replicas. If the third disk has even one stale chunk (bit rot you didn't notice), you have lost data. AFR is no longer the right model — rack failure rate is, and that is dominated by power, network, firmware, and human error, not disk mechanical failure.
  • Repair storms: A single datacenter network event kills 100 disks at once. You now need to repair 400 TB across the fleet, contending for the same network and CPU as live traffic. Repair window stretches from 1 hour to 24 hours; durability degrades by ~24x during the storm. Coincident failures during the storm are now likely.
  • Storage cost: $6M/month for just the bits. Hardware refresh and power-cooling pushes total cost above $10M/month. The CFO asks why we can't be more efficient.

Fixed: Reed-Solomon 6+3 + cross-AZ placement + scrub + cross-region

Local encoding: 6+3 Reed-Solomon, chunks placed 1 per rack across 9 racks distributed 3 racks per AZ.

  • Storage overhead: 1.5x instead of 3x. Saved 150 PB of disk. Direct cost saving: ~$3M/month.
  • Fault tolerance: survives any 3 chunk losses. Survives any rack loss (1 chunk). Survives an AZ loss (3 chunks) at the edge — but barely; one more rack failure during repair = data loss. So actually push to 6+4 or 10+4 (or spread chunks across more AZs) for AZ-loss-plus-repair tolerance.

Background scrub: every chunk is read and checksummed every 14 days. Bit rot caught and re-encoded before the parity is exhausted. Without scrub, you would only catch corruption on read — and 90% of objects are cold.

Cross-region async replication: opt-in per bucket; a second region holds a fully encoded copy. RPO: typically seconds. Survives region loss with one bucket-level config flip.

Durability math, fixed configuration:

  • Per-disk hourly failure probability: ~4.6 * 10^-6 (from AFR 4%).
  • Probability that 4 of the remaining 8 chunks (in 6+3, after one chunk loss) fail inside a 1-hour repair window, independent: C(8, 4) * (4.6 * 10^-6)^4 ≈ 3 * 10^-20. Annualized across 8760 hours and ~10^11 objects, this is still safely under 10^-11 per object per year.
  • That number is the floor under independent failures. The real risk is correlated rack/AZ failures, which is why placement spreads chunks across 9 distinct failure domains and K is chosen larger than the number of chunks that any single rack or AZ holds. With K=3 and 3 AZs (3 chunks per AZ), one full-AZ loss leaves zero margin; the design should move to 6+4 or spread across more AZs.
  • Cross-region async replication is a separate threat model — it defends against region loss with an explicit Recovery Point Objective (RPO) of seconds (the trailing un-replicated writes are at risk). It is not "extra nines on the same durability number"; it is a different number for a different failure mode.

Total: defensible eleven nines against disk/rack failures, with an explicit RPO for region loss. Cost is 1.5x in-region plus 1x cross-region = ~2.5x total, versus naive 3x single-region. Whether the complexity of erasure coding earns its keep depends on scale: at 100 PB the storage saving is ~$3M/month of disk, which pays for the engineering and operations of running EC. At 10 PB, the saving is ~$300K/month and 3x replication may be the staff-level answer.

When to switch from replication to erasure coding

The honest threshold is dollars, not nines — both schemes can hit eleven nines if placed correctly.

  • Scale floor. Under ~10 PB raw, 3x replication across AZs is genuinely fine; the storage saving from EC does not pay back the engineering, operational complexity, and worse read-tail latency. The "staff move" at single-digit PB is often to not introduce erasure coding.
  • Object size > ~1 MB. Below this, chunk metadata and the M+K placement records can exceed the data; small objects belong in a replicated sidecar path. S3 internally splits small and large object paths.
  • Read traffic not latency-critical. Decode adds latency and, more importantly, EC reads have a structurally worse tail (you wait for the slowest of M shards, with no fast-replica hedging). Hot, latency-sensitive data often stays replicated even at large scale.
  • Repair traffic can be absorbed. EC repair amplifies network use by ~M. Confirm the cluster network has headroom in the off-peak window for repair to complete inside the failure-correlation budget.
  • Storage cost dominates the budget. Once disk is the largest line item, 1.4x vs 3x at hundreds of petabytes is millions of dollars per year — at that point EC earns its complexity.

7. Failure Modes and Operations

Failure Metric to watch Mitigation
Disk failure (single drive) AFR, missing-chunk count Background re-replication; placement diversity.
Silent data corruption (bit rot) Scrub mismatch rate Continuous scrubbing; re-encode from parity.
Correlated rack failure Rack failure rate; under-replication count per rack Cross-rack placement constraint; K exceeds rack count in a failure domain.
AZ failure AZ availability, chunk-loss-per-AZ-event Cross-AZ placement; K >= chunks per AZ.
Region failure Cross-region replication lag Async geo-replication; tested DR runbooks.
Repair storm (mass disk loss) Repair backlog, repair bandwidth utilization Throttled repair; prioritize most-under-replicated chunks; reserve repair capacity.
Stale metadata (mismapped chunks) Metadata-vs-data audit divergence Merkle anti-entropy; periodic full audit.
In-flight write lost during failover WAL lag at failover, lost-write counter Durable WAL with lineage; client idempotency.
Hot disk overload during repair Per-disk QPS, repair-induced latency Spread repair source across many disks; rate-limit.
Bad firmware fleet-wide Fleet-wide error rate by drive model Diverse drive vendors / models in a placement group.
Operator deletion (fat-finger) Delete-rate spike Versioning + soft delete + delete protection + delayed garbage collection (GC).
Ransomware / poisoned encrypts Versioning churn Object lock, immutability windows, air-gapped backup.

Operational rules of thumb

  • Repair must complete faster than the Mean Time Between Failures (MTBF) of K coincident chunk losses; if not, you are durability-bankrupt.
  • Scrub interval should be shorter than the time for bit rot to exhaust your parity budget.
  • Placement engine must encode the failure-domain hierarchy: disk -> node -> rack -> AZ -> region. K must exceed the count of chunks in the largest failure domain you tolerate losing.
  • Always alert on under-replication count and on repair backlog growth, not on repair rate alone — a high repair rate that keeps up is fine; a low repair rate with a growing backlog is fatal.
  • Test failover at least quarterly. Untested failover is fictional failover. Many region-loss "DR plans" only exist on paper until the first real fire.
  • Track durability events at the per-customer level. A single bucket losing one object is a customer-trust incident even if the aggregate eleven-nines number is met.
  • Maintain a "lost object" runbook. When an actual object is unrecoverable, you owe the customer (a) which object, (b) the exact time it was lost, (c) which redundancy layers failed, (d) what is changing so it does not recur. S3 issues credits and writes an incident memo per lost-object event.

A worked metric set

Metric Definition Alert
Under-replicated chunks Chunks with fewer than target survivors Page if growing; ticket if non-zero for >1h
Repair backlog bytes Total bytes pending re-encode / re-replicate Page if growth rate exceeds throughput
Scrub coverage % Fraction of fleet scrubbed in last 30d Page if < 95%
Silent corruption rate Scrub mismatches per TB read Trend chart; investigate spikes
Cross-region replication lag (p99) Seconds between write and cross-region durable Page if > RPO_target

Annualized data loss rate Lost bytes per stored byte per year Track vs eleven-nines target

8. Combinations with Other Patterns

  • Idempotency / Exactly-Once — In-flight write recovery after failover requires idempotent client retries. The WAL gives the system a chance to recover; idempotency keys give the client a chance to retry without duplicate effects.
  • Strong Consistency Under Contention — A durable write that is not yet visible to readers is a consistency, not a durability, problem. S3 went from eventual to strong read-after-write in 2020 by tightening the metadata service while keeping the same durability stack.
  • Coordination / Exactly-One Execution — Chubby / ZooKeeper / Etcd leases provide the fencing tokens that make WAL-lineage-based failover safe.
  • Ordering / Causal Consistency — WAL ordering (LSN) is what makes failover safe. The same primitive supports causal consistency in higher-level systems.
  • Hot Key Skew — Durable storage with hot prefixes will see repair-traffic skew during a failure; prefix-aware placement and adaptive partitioning matter.
  • Fan-Out Write Amplification — 3x replication is 3x write amplification; erasure coding is (M+K)/M bytes but K independent parity calculations. Cross-region adds another 1x. Plan write-bandwidth headroom accordingly.
  • Cache Stampede / Read Amplification — Erasure-coded reads have inherent M-fold read amplification; a CDN or hot-replica tier in front of cold EC storage absorbs that.
  • Backpressure / Flow Control — Repair traffic must be backpressured against live traffic; otherwise a repair storm eats client SLO.

9. Likely Interviewer Probes

  1. "Walk me through why S3 advertises eleven nines, with arithmetic." Expect: state the unit first — eleven nines is 10^-11 per object per year, not a system uptime number, and it is separate from S3's four-nines availability. Then: encoding rate M+K, failure-domain diversity, repair-time math (P(K coincident failures during the repair window)), bit-rot scrubbing, and cross-region replication as a separate defense against region loss with its own RPO.

  2. "How do you choose M and K in Reed-Solomon?" Expect: storage overhead target ((M+K)/M), fault tolerance budget (K greater than the largest correlated failure domain), read amplification cost (smaller M = less read amplification but more overhead), and repair amplification (larger M = more chunks to read for each repair). Concrete examples: 10+4 for hot, 17+3 for cold.

  3. "Why not just use 5x or 7x replication instead of erasure coding?" Expect: at multi-PB scale, cost (5x vs ~1.4x) is the answer, plus the point that 5x replication only helps if those 5 copies live on 5 disjoint failure domains — at which point an equivalent erasure code achieves the same protection cheaper. Also acknowledge the inverse: at single-digit PB, 3x replication is the right answer because the storage saving doesn't pay back EC's repair amplification and read-tail cost.

  4. "How does scrubbing actually work? What does the production scrub schedule look like?" Expect: per-chunk CRC32C / SHA-256 stored at write; scrub reads chunks at a throttled rate, recomputes, compares; on mismatch, re-encode from peers and re-place. Scrub frequency 14-30 days; cold-tier scrub is less frequent. Throttle to < 5% of disk bandwidth.

  5. "A write was acknowledged to the client, then the primary died. How do you guarantee no data loss?" Expect: Write-Ahead Log on durable storage, written before ack; failover waits until the new primary catches up to the highest committed Log Sequence Number; fencing token prevents the old primary from acking more writes. Plus client-side idempotency for retries that crossed the failover.

  6. "What happens during a repair storm?" Expect: throttling, prioritization by current redundancy level (chunks with only M survivors get repaired first), repair-bandwidth reservation, alerting on repair backlog growth rate, and the math behind why repair must be faster than the time to the next coincident K-failure.

  7. "How does cross-region replication interact with strong consistency?" Expect: typically async (so cross-region is eventually consistent), with the regional layer providing strong consistency. Sync cross-region exists (Spanner) but pays inter-region RTT on every write. For object stores, async + monitoring + RPO < 1 minute is the norm.

  8. "How do you survive ransomware or a malicious insider?" Expect: object versioning (every overwrite is a new version, the prior version stays for a retention window), object lock / Write-Once-Read-Many (WORM) for compliance buckets, role-based delete restrictions, MFA-Delete (Multi-Factor Authentication required for deletes), and air-gapped backups whose credentials are not reachable from the production control plane. Durability has to defend against the operator and the attacker, not just hardware.

  9. "What if your placement engine itself has a bug?" Expect: a periodic placement-audit job that walks the metadata and verifies that each chunk's actual locations satisfy the placement constraint (cross-rack, cross-AZ). Bugs that secretly co-locate chunks are the most dangerous because the durability number lies until the correlated failure happens. Treat the placement audit as a first-class durability mechanism alongside scrubbing.

10. Common Traps → Stronger Moves

Trap Stronger move
"3x replication is enough." Distinguish independent vs correlated failures; show that 3x co-located gives ~1 nine of real durability under rack failure. Add cross-AZ placement, then erasure coding for cost.
"Just store more copies." At scale this is unaffordable. Reed-Solomon M+K gives the same fault tolerance at (M+K)/M overhead.
"Disks just fail; AFR explains it." Real risk is correlated failures: rack switch, power phase, bad firmware, datacenter event, human error. Independent-failure math hides these.
"Replication = durability." Replication of committed state is durability for stored data. In-flight data needs a durable Write-Ahead Log. Otherwise a primary crash can lose acked writes.
"We have replication, so we don't need scrubbing." Without scrubbing, bit rot silently exhausts your parity; you discover loss on read, too late. Scrubbing is a continuous-time component of durability, not a setup-time one.
"Erasure coding everywhere." Small objects pay encode overhead in metadata and chunk-management; replicate small objects, encode large ones. Hot data may want a replicated front to avoid M-fold read amplification.
"Cross-region replication = 2x durability." Async cross-region has RPO > 0 — the trailing seconds of writes can be lost in a region disaster. State this explicitly and have a runbook for it.
"Backup is durability." Backup snapshots are a defense against logical corruption, ransomware, and operator error; they do not replace replication / EC. Independent threat model.
"Glacier is just slow storage." Glacier has the same durability target as S3 Standard. The tradeoff is retrieval latency and cost, not durability. Pair with lifecycle policies.
"I will just use the cloud's eleven nines." The interviewer wants to see you design to eleven nines, not invoke a service. Walk the encoding rate, failure domain placement, scrub cadence, and repair math.
"We use Nx replication and don't worry about it." Quantify Mean Time To Data Loss (MTTDL): function of N, AFR, repair time, and failure-domain diversity. Show the arithmetic.
"Repair is automatic, so we are safe." Show repair-backlog metrics, repair-bandwidth budgeting, and the under-replication histogram. A growing repair backlog is the leading indicator of pending data loss.

Bonus probe: "What is the dominant cost in your durability story — disk, network, CPU, or operations?"

Strong answer: it depends on tier. Hot tier — disk dominates (replication or M+K overhead times petabyte-scale). Cold archive — tape and cooling dominate; encode CPU is once per object. Repair-heavy fleets — network dominates because each repair drags M chunks across racks. Operations dominates when scrub, repair, audit, and DR drills require dedicated SRE / Storage Reliability Engineering headcount, which they do at exabyte scale. Naming the cost dimension tells the interviewer you have operated, not just designed, this layer.

11. Staff-Level Answer Checklist

  • [ ] State the durability target in nines and convert to a per-object annual loss probability.
  • [ ] Identify the failure-domain hierarchy: disk → node → rack → AZ → region.
  • [ ] Choose a redundancy scheme (Nx replication or Reed-Solomon M+K) and justify the choice against storage cost and read amplification.
  • [ ] Make K exceed the largest correlated failure domain you commit to tolerating.
  • [ ] Place chunks across diverse failure domains; encode the placement constraint in the placement engine, not in a comment.
  • [ ] Add background scrubbing with a cadence shorter than your parity-exhaustion timeline.
  • [ ] Use a durable Write-Ahead Log for in-flight writes; failover waits on WAL catch-up.
  • [ ] Add cross-region replication with an explicit RPO target.
  • [ ] Define durability classes / tiering with lifecycle policies; cold tiers do not lose durability, only latency.
  • [ ] Track repair backlog and under-replication count; alert on growth rate, not raw rate.
  • [ ] Defend against operator error and ransomware via versioning, object lock, and immutability windows.
  • [ ] Walk the arithmetic: per-chunk failure probability, repair time, coincident-failure probability, and final per-object durability number.
  • [ ] Name the metric for each failure mode (correlated rack failure, bit rot, repair storm, region loss, in-flight loss).
  • [ ] Mention the operational practices: scrub schedule, repair throttling, DR drills, fleet-wide vendor / firmware diversity.
