Search and Ranking

2026/05/31  ·  System Design Patterns
Contents
Contents

Return the K most relevant documents out of billions in under 100 milliseconds by splitting the work into a cheap, high-recall candidate generation stage and an expensive, high-precision ranking stage, executed across a sharded inverted index plus an Approximate Nearest Neighbor (ANN) vector index, fronted by a query planner that scatters, gathers, ranks, and hydrates. See the Staff System Design Glossary for shared vocabulary. Closely related question docs: Design Google Search, Design Twitter Search, Design Typeahead / Autocomplete, Design Typeahead Suggestion, Design a Netflix-Style Recommendation System, Design Facebook's Newsfeed, Design Google Ads.

1. The fundamental hardness

Search and ranking is not one problem; it is two problems wearing the same UI. The first problem is recall: out of a corpus of N documents (where N is often 10^9 to 10^12), find the candidate set C that could plausibly answer the query. The second problem is ranking: out of C (typically 10^3 to 10^5 items), order them so the top K (often 10) match what the user actually wants. Recall is judged by "did we miss anything we should have shown?". Ranking is judged by "is the best result first?". These two metrics pull on different parts of the system, and conflating them is the most common failure mode at the whiteboard.

The naive design — store all documents in a database, run a LIKE '%query%' scan on each request — fails at three independent walls. Wall one: linear scan. Even at an aggressive ~10 µs per document for a substring match, 10^9 documents is ~10^4 seconds (hours) per query, and you cannot parallelize without already having sharded the data. Wall two: relevance. Substring match has no notion of which document is better; you would return the first match, not the best match. Wall three: semantics. The query "best laptop for coding" should match a result titled "developer notebook recommendations" even though they share zero tokens. Keyword indexes alone miss the second; vector embeddings alone miss exact matches and structured filters ("price under $50"). Whether a system needs both is workload-dependent — exact-match-dominated domains (e-commerce stock-keeping unit (SKU) search, log search, code search) can ship keyword-only and be fine. Semantic-intent domains (web search, recommendation, "find similar") need vectors. The honest answer in interview is to identify which side of that line your product is on.

Latency makes the hardness concrete. A typical budget for a consumer search is 200 ms end-to-end at p99, of which roughly 100 ms is the search backend and the rest is network + frontend rendering. Inside 100 ms you must: tokenize the query, look up postings lists across many shards, intersect / union them, score with Best Matching 25 (BM25), retrieve top-N candidates, optionally run a second-stage machine-learning (ML) ranker over N=1000 features per candidate, hydrate the document bodies, and merge. At 10^12 documents you cannot do this on one machine; you cannot do it serially across shards either. You must shard, scatter-gather in parallel, prune aggressively, and accept that the tail latency of the slowest shard becomes the latency of the query — and the p99 of S independent shards is roughly the per-shard p(1 - 0.01/S), i.e. p99.9 of a single shard at S=100, p99.99 at S=1000. That number is what justifies hedged requests, replica selection by latency, and per-shard timeouts; not a vague "tail latency is bad" hand-wave. The whole architecture is a study in latency budget allocation across a parallel fan-out, very similar in shape to fan-out write amplification but read-side.

The final twist: the system has to stay fresh. A tweet posted thirty seconds ago should be searchable; a product whose price just dropped should re-rank. But rebuilding a 10^12-document index takes hours. Real systems tier the index — a small, hot "live" tier indexed in near-real-time, plus a large, cheap "main" tier built in batch — and the query merges across tiers. Freshness vs. cost vs. index-build complexity is a permanent tradeoff that the interviewer will probe.

2. How to recognize it in the interview

Prompt phrases that signal this pattern

  • "Find documents / tweets / products / videos matching a query."
  • "Return top K most relevant ..."
  • "Rank by relevance, popularity, freshness, ..."
  • "Personalize the feed / results / recommendations."
  • "Auto-complete as the user types" (specialized prefix variant).
  • "Semantic search" / "find similar items" (vector-heavy variant).
  • "Show ads relevant to this query / this user." (ranking-heavy variant).
  • "We have billions of items and queries need to come back fast."

Architectural smells

  • Read-heavy workload with arbitrary predicates (full-text, filters, geo, time).
  • The corpus is too large to fit on one machine; queries must touch many shards.
  • "Most relevant" is application-defined and learned from feedback (clicks, dwell time).
  • Freshness matters and is not the same as the bulk index build cadence.
  • The product has both a head (popular queries, hot items) and a heavy long tail.
  • Read quality is judged by an offline metric (recall@K, normalized Discounted Cumulative Gain (nDCG), Mean Reciprocal Rank (MRR)) not just latency.

Example interview prompts where this is core

  1. Design Twitter search over the last decade of tweets at 100k Queries Per Second (QPS).
  2. Design YouTube / Netflix homepage recommendations for 1B users.
  3. Design Google Search at web scale.
  4. Design Yelp restaurant search with text, location, and ratings.
  5. Design e-commerce product search at Amazon scale.
  6. Design Google Ads ranking for a query auction at 100k QPS with sub-100 ms p99.
  7. Design LinkedIn People-You-May-Know (recommendation, not text search, but the same two-stage pattern).

3. 60-second spoken framing

Before I jump to architecture I'd size the problem. For a corpus under ~10M documents with low Queries Per Second (QPS), a single-node Elasticsearch with Best Matching 25 (BM25) and no learned ranker is genuinely fine — the staff-level move there is to not over-engineer. The interesting case is when the corpus exceeds one machine's memory, the latency budget can't accommodate a re-rank of the full corpus, or relevance quality is a product differentiator. In that regime I frame it as a two-stage retrieval problem. Stage one is candidate generation: I want high recall, low latency, and I'm willing to be sloppy on order. A sharded inverted index for keyword lookup, plus — if the queries are intent-heavy or semantic — an Approximate Nearest Neighbor index over learned embeddings, scatter-gather across shards, pull back a few hundred to a thousand candidates per query. Stage two is ranking: take those candidates and run a learned ranker — Gradient-Boosted Decision Tree (GBDT) like LambdaMART or a neural ranker — over features pulled from an online feature store, return the top ten. Around that I'd build a tiered index for freshness (live tier seconds-fresh, main tier rebuilt batch), caching of head queries, and an offline pipeline that retrains the ranker on click logs. The hard problems will be scatter-gather tail latency, freshness lag, and training/serving skew in the ranker, which I am happy to go deep on.

4. Canonical toolkit

Technique When to reach for it Cost
Inverted index (Lucene-style) Keyword retrieval, structured filters, the spine of every text-search system. Index build is expensive; query is logarithmic in postings list length.
Document-sharded scatter-gather Corpus too big for one machine; queries fanned out to every shard. Tail latency dominated by slowest shard.
Term-sharded indexing Massive corpus, very long postings lists for popular terms. Hot-shard risk on common terms; less common.
Vector embeddings + ANN (Hierarchical Navigable Small World (HNSW), Inverted File (IVF), ScaNN) Semantic search, recommendation, "find similar". Recall < 100%; embedding refresh cost; memory-heavy.
Two-stage retrieval (cheap recall + expensive rank) Anytime ranking signals are too expensive to compute over the whole corpus. Bug surface in the seam between stages.
Learn-to-rank (LambdaMART, neural rankers) Optimizing nDCG / Click-Through Rate (CTR) from labeled data and click logs. Training pipeline, feature drift, online/offline parity.
BM25 (Best Matching 25) baseline First-pass scoring inside the inverted index — the floor everyone competes against. Sensitive to tokenization, stemming, language analyzer choice; k1 / b hyperparameters need per-corpus tuning. Cheap to serve (microseconds per document).
Online feature store + offline feature pipeline Per-query / per-document / per-user features available in microseconds at rank time. Online/offline skew is a permanent source of bugs.
Query rewriting + expansion Bridge surface form to intent (synonyms, stemming, spell correction). Over-expansion blows up recall but adds latency.
Tiered index (live + main) Need second-level freshness over a corpus too big to rebuild often. Two indexing pipelines to operate; merge logic at query time.
Top-results cache Head queries hit hard; cache cuts most load. Cache invalidation on index updates, hot-key risk (see cache-stampede-read-amplification).

4.1 Inverted index build (sketch)

# Build a simple in-memory inverted index over a document corpus.
# In production, this is what Lucene / Tantivy does on disk with skip lists,
# block-compressed postings, term dictionaries, and positional info.

from collections import defaultdict
from dataclasses import dataclass, field
import math
import re

TOKEN_RE = re.compile(r"[A-Za-z0-9]+")

def tokenize(text: str) -> list[str]:
    # Real systems: language detect -> tokenizer -> lowercaser -> stemmer -> stopwords.
    return [t.lower() for t in TOKEN_RE.findall(text)]

@dataclass
class Posting:
    doc_id: int
    term_freq: int  # how many times the term appears in this doc

@dataclass
class InvertedIndex:
    # term -> list of postings (sorted by doc_id for fast intersection)
    postings: dict[str, list[Posting]] = field(default_factory=lambda: defaultdict(list))
    # doc_id -> document length in tokens (used by BM25)
    doc_len: dict[int, int] = field(default_factory=dict)
    avg_doc_len: float = 0.0
    n_docs: int = 0

    def add(self, doc_id: int, text: str) -> None:
        tokens = tokenize(text)
        self.doc_len[doc_id] = len(tokens)
        counts: dict[str, int] = defaultdict(int)
        for tok in tokens:
            counts[tok] += 1
        for term, tf in counts.items():
            self.postings[term].append(Posting(doc_id, tf))
        self.n_docs += 1

    def finalize(self) -> None:
        for term in self.postings:
            self.postings[term].sort(key=lambda p: p.doc_id)
        self.avg_doc_len = sum(self.doc_len.values()) / max(1, self.n_docs)

Two things to note in interview: postings are kept sorted by doc_id so multi-term queries can be answered by a linear merge (intersection / union in O(sum of postings list lengths)), and the index stores enough statistics (term frequency, document length, document count) to score BM25 without re-reading documents.

4.2 BM25 scoring (sketch)

def bm25_score(idx: InvertedIndex, query_terms: list[str], doc_id: int,
               k1: float = 1.2, b: float = 0.75) -> float:
    # BM25 = sum over query terms of: IDF(t) * (tf * (k1+1)) / (tf + k1 * (1 - b + b * dl/avg_dl))
    score = 0.0
    dl = idx.doc_len.get(doc_id, 0)
    if dl == 0:
        return 0.0
    for t in query_terms:
        postings = idx.postings.get(t, [])
        df = len(postings)
        if df == 0:
            continue
        # Lucene-style IDF; smoothed to stay non-negative.
        idf = math.log(1 + (idx.n_docs - df + 0.5) / (df + 0.5))
        # Find this doc's term frequency. In production, postings are skip-listed
        # for sublinear lookup; here we do a binary search.
        lo, hi = 0, len(postings) - 1
        tf = 0
        while lo <= hi:
            mid = (lo + hi) // 2
            if postings[mid].doc_id == doc_id:
                tf = postings[mid].term_freq
                break
            if postings[mid].doc_id < doc_id:
                lo = mid + 1
            else:
                hi = mid - 1
        if tf == 0:
            continue
        denom = tf + k1 * (1 - b + b * dl / max(1.0, idx.avg_doc_len))
        score += idf * (tf * (k1 + 1)) / denom
    return score

BM25 is a probabilistic refinement of Term Frequency / Inverse Document Frequency (TF-IDF). The interesting parameters: k1 controls term-frequency saturation (so the 100th occurrence of "the" does not score 100x the first), b controls length normalization (so a 10-word tweet does not lose to a 10,000-word page just on raw frequency). It is the floor that every learned ranker must beat.

4.3 Scatter-gather across document shards (sketch)

import asyncio
from dataclasses import dataclass

@dataclass(order=True)
class ScoredDoc:
    score: float
    doc_id: int
    shard_id: int

async def shard_search(shard_client, query: str, k: int) -> list[ScoredDoc]:
    # Each shard runs the inverted-index lookup + BM25 locally and returns its top-k.
    # In Lucene-speak: a per-shard `IndexSearcher.search(query, k)`.
    return await shard_client.search(query, k)

async def scatter_gather(shards: list, query: str, k: int,
                         per_shard_timeout_ms: int = 80) -> list[ScoredDoc]:
    # Fan out to all shards in parallel. Time-bound every shard call so one slow
    # shard cannot blow the global p99. Late shards are simply dropped; the
    # tradeoff is a small, bounded recall hit on the long tail.
    tasks = [
        asyncio.wait_for(shard_search(s, query, k), per_shard_timeout_ms / 1000.0)
        for s in shards
    ]
    results: list[ScoredDoc] = []
    for fut in asyncio.as_completed(tasks):
        try:
            results.extend(await fut)
        except asyncio.TimeoutError:
            # Record a shard-timeout metric; this is the most important search SLO signal.
            continue
    # Global merge: heap-merge top-k across all shards. We can use heapq.nlargest
    # because each shard already returned its sorted top-k.
    results.sort(reverse=True)
    return results[:k]

Three knobs for the interviewer: per-shard timeout (lower = tighter tail, higher = better recall), per-shard k (you typically ask each shard for k * over_fetch_factor because the global top-k is not evenly distributed across shards), and partial-result policy (return what you have, or fail closed). Partial results are usually the right choice for search; never for payments.

4.4 Two-stage ranking pipeline (sketch)

@dataclass
class Candidate:
    doc_id: int
    first_stage_score: float

def first_stage_retrieve(query: str, top_n: int = 1000) -> list[Candidate]:
    # Cheap: BM25 + ANN union, scatter-gather across shards, return ~1000 candidates.
    bm25_cands = bm25_search(query, k=top_n)
    ann_cands = ann_search(embed(query), k=top_n)
    # De-duplicate; keep max first-stage score as a feature for stage two.
    merged: dict[int, Candidate] = {}
    for c in bm25_cands + ann_cands:
        cur = merged.get(c.doc_id)
        if cur is None or c.first_stage_score > cur.first_stage_score:
            merged[c.doc_id] = c
    return list(merged.values())

def hydrate_features(candidates: list[Candidate], query: str, user_id: int) -> list[dict]:
    # Pull from an online feature store. Per-doc features (CTR, recency, quality),
    # per-(query, doc) features (BM25, semantic similarity), per-user features,
    # cross features (user x doc affinity).
    feature_store = OnlineFeatureStore()
    return feature_store.multi_get(
        doc_ids=[c.doc_id for c in candidates],
        query=query,
        user_id=user_id,
    )

def second_stage_rank(candidates: list[Candidate], features: list[dict],
                      ranker, k: int) -> list[Candidate]:
    # Expensive: GBDT or neural ranker scores each (query, doc, user) tuple.
    scores = ranker.predict(features)
    ranked = sorted(zip(candidates, scores), key=lambda x: x[1], reverse=True)
    return [c for c, _ in ranked[:k]]

def serve_query(query: str, user_id: int, k: int = 10) -> list[int]:
    cands = first_stage_retrieve(query, top_n=1000)
    feats = hydrate_features(cands, query, user_id)
    top = second_stage_rank(cands, feats, RANKER, k=k)
    return [c.doc_id for c in top]

The reason this two-stage pattern earns its complexity is concrete latency arithmetic, not a generic best-practice claim. Stage one (inverted-index lookup + BM25 across a sharded posting list) is on the order of 5-20 ms per query because BM25 scoring is microseconds per candidate document. A learned ranker with ~100 features per (query, doc, user) tuple typically costs 50-500 µs of inference per candidate (GBDT) or 1-10 ms (neural). If you tried to run the learned ranker over every matching document, a query that matches 10^6 documents would cost minutes; over a fixed 1000-candidate slate it costs 50 ms to a few hundred ms, fitting the budget. The break-even is: two-stage is worth the seam-bug surface when (candidates from stage one) × (per-candidate stage-two cost) would otherwise blow the latency budget. For small corpora (< 10M docs) or cheap rankers (linear model on a handful of features), single-stage BM25 with an ORDER BY is genuinely sufficient and shipping two stages is gold-plating. Every web-scale system — Google, Amazon, YouTube, Netflix, Twitter, TikTok, Yelp, LinkedIn Feed — uses some shape of two-stage because their candidate counts and ranker costs put them on the wrong side of that arithmetic.

5. Worked example: naive → broken → fixed

Workload. Twitter-style search: 100k QPS against 100 billion tweets (10 years of history at ~30M/day). Latency Service-Level Objective (SLO): p99 < 300 ms end-to-end, p50 < 80 ms. Freshness target: a tweet posted now should be searchable within 5 seconds. Each tweet has full text, author, timestamp, hashtags, language, engagement counts, and an embedding.

5.1 Naive design (interview anti-pattern)

Client --> Search Service --> Postgres
                                    SELECT id FROM tweets
                                    WHERE text LIKE '%query%'
                                    ORDER BY created_at DESC LIMIT 10;

Why every part breaks:

Problem Consequence
LIKE '%...%' cannot use a B-tree index. Full-table scan over 10^11 rows. Single query: hours.
One Postgres host. At 100k QPS, the host is overwhelmed at ~3 orders of magnitude under load.
ORDER BY created_at is recency-only ranking. No relevance; spam wins; user trust collapses.
No freshness pipeline. If we replicated to a read replica, the lag would be unbounded under write spikes.
No filters: language, blocked authors, deleted tweets. Visibility leaks.

This design fails on all of recall, ranking, latency, and correctness.

5.2 First-fix attempt (still broken)

A bright candidate jumps to: "OK, use Elasticsearch."

Client --> Search Service --> Elasticsearch (one cluster)
                                    GET /tweets/_search { query: ... }

This is better but still wrong at this scale, for reasons the interviewer will probe:

  • One cluster cannot hold 10^11 documents at reasonable cost. Elastic's practical per-shard sweet spot is 20-50 GB, so you need thousands of shards.
  • Scatter-gather across thousands of shards turns tail latency into a disaster: the slowest of 1000 shards at p99 is much worse than the cluster p99.
  • Ranking is still just BM25. You will lose to any competitor with a learned ranker.
  • No freshness tier. Indexing 30M tweets/day inline with the main index slows indexing and contends with query load.
  • No caching. Repeated head queries ("World Cup", "election") hit the cluster 100k * head_share times per second.
  • No degraded mode. When one shard dies, the query fails instead of returning partial results.

Why this works:

Decision Why
Document-sharded (not term-sharded) Even load across shards regardless of query; query fans out to all shards, but each shard's work is bounded. Term-sharded suffers hot-shard on "the".
Two-tier live + main Decouples freshness from index build cost. Live tier is small (~1B docs) and can be in memory. Main tier holds the long tail cheaply on disk.
Per-shard timeout = 80 ms Bounds tail. If a shard misses, we accept partial recall on the long tail rather than failing the query.
BM25 + ANN union for stage 1 Recall on both surface form and intent. BM25 alone misses paraphrases; ANN alone misses exact-string queries (handles, $cashtags, quoted phrases). Twitter search has both classes in the query mix, which is why the union is worth it here — for a domain that is exact-match-only (e.g., username lookup) ANN is overkill.
LambdaMART for stage 2 GBDT on ~100 features is a strong, cheap-to-serve baseline. Modern web-scale systems often migrate to a neural ranker (deep & cross, two-tower with cross-attention, transformer rerankers) once feature-engineering ROI plateaus; the right choice depends on team capacity for training infra and serving latency budget, not a universal "GBDT wins".
Online feature store Sub-millisecond feature reads at rank time; trained offline so online/offline parity is maintained.
Query-result cache, ~30 s TTL on head A small fraction of queries are most of the traffic; this cuts backend load roughly an order of magnitude.
Hydration after rank Don't pay full document I/O for 1000 candidates — only the top 10.
Visibility filter at query time, not at index time Deletes / blocks change too often to bake into the index.

5.4 Capacity math (interview-style napkin)

  • 10^11 tweets * 200 bytes posting overhead per term * 50 terms/tweet ~= 1 PB for the main inverted index. That is the budget that justifies tiering.
  • 100k QPS * 1000 shards = 10^8 sub-queries/sec of fan-out load against the cluster. Each shard sees 10^8 / 1000 = 10^5 sub-queries/sec, which is well above sustainable single-machine search QPS (a Lucene shard typically tops out at hundreds to low thousands of qps for non-trivial queries). The cache + cheap query rejection + over-provisioning shards is what makes the math close. If you do not name caching and the per-shard QPS target, the interviewer should push.
  • Scatter-gather tail: with 1000 shards fanned in parallel and a per-shard p99 of 20 ms, the aggregate p99 (slowest of 1000 independent draws) is roughly the per-shard p99.9 — typically 2-5x the per-shard p99. Hedged requests to a replica after 20 ms cut this materially, at the cost of ~10% extra cluster load.
  • Feature fetch budget: 1000 candidates × ~100 features = 10^5 feature reads per query. Batched as one multi-get into an in-memory feature store, this is ~1-5 ms wall-clock (not "< 1 µs / candidate" of compute — that arithmetic conflates wall time with per-item amortized cost). The honest claim is: the feature store must support batched multi-get with single-digit ms p99, which is what colocated in-memory key-value stores (Redis, RocksDB-in-memory, custom mmap) deliver.

6. Variants and how to choose

6.1 Keyword vs. vector vs. hybrid

Axis BM25 / keyword inverted index ANN over learned embeddings Hybrid
Strength Exact match, structured filters, explainable. Semantic match, "find similar", multilingual via shared embedding space. Both, when both classes of query exist in the mix.
Weakness Misses paraphrases, synonyms, intent. Misses exact phrases, struct filters, recall < 100%. Two indexing pipelines, two refresh stories, fusion-scoring complexity.
Recall@1000 on web queries High on head, low on intent-heavy long tail. Lower than BM25 on head, much better on long tail. Highest, but the gain over the better single mode is often modest (5-15% recall@1000) and not free.
Build cost Linear in tokens. Linear in docs + embedding model cost (often 10-100x the cost per doc of tokenization). Both.
Refresh cost Cheap (segment merges). Expensive (embedding refresh on model change rebuilds the whole index). Both.
Use when Exact-match dominates (SKU, code, log, username, legal); explainability/audit is required. Intent-dominant (recommendation, "more like this", multimodal); query mix is short/ambiguous. The query mix is genuinely mixed and you have the team capacity to operate two pipelines.

Hybrid is the web-search default because web queries are genuinely mixed. It is not a universal default: an e-commerce SKU lookup, a logs search, a code search, or a structured filter-first product (Yelp filters by cuisine + open-now + rating) can ship keyword-only and beat a hybrid system on latency, refresh cost, and predictability. Pick hybrid when you can name a class of queries that BM25 measurably misses.

6.2 Document-sharded vs. term-sharded

Axis Document-sharded (default) Term-sharded
How keys map to shards shard = hash(doc_id) % N; every shard has all terms, a slice of docs. shard = hash(term); one shard owns all postings for a given term.
Query fan-out All shards (high fan-out). Only the shards owning query terms (low fan-out).
Per-shard work Bounded, predictable. Unbounded for popular terms — hot shard on "the".
Tail latency Dominated by slowest shard; aggressive timeouts needed. Dominated by hottest term.
Indexing Each doc indexed on its single shard — simple, parallel. Each doc touches as many shards as it has distinct terms.
Industry use Lucene, Elasticsearch, Solr, Vespa — the standard. Mostly historical; some early web search systems.

For staff interviews: default to document-sharded and only mention term-sharded as the alternative you considered and rejected.

6.3 Offline-rank-and-store vs. online-rank

Axis Offline (precompute per user/query) Online (rank at request time)
Latency Single key lookup; sub-millisecond. Stage-1 + stage-2 + hydrate; 50-200 ms.
Freshness As stale as the last batch (hours to a day). Up-to-the-second.
Personalization Limited to keys you precomputed for. Arbitrary per-request signals (location, time, A/B bucket).
Cost Big batch job; cheap reads. Big online fleet; smaller batch.
Use when Stable corpus, predictable users (Netflix homepage, Facebook PYMK). Volatile corpus (Twitter search, news, ads).
Hybrid trick Precompute candidate pool; rank online. Same.

The hybrid (precompute candidates offline, rank online with fresh features) is the industry default for recommender feeds. Twitter / news search lean online because freshness wins.

7. Failure modes and operations

Failure Metric to alert on Mitigation
Shard timeout / drop shard_timeout_rate per shard, p99 query latency. Per-shard timeout; partial results; replica failover; auto-shed indexing load when query load spikes.
One shard hot (skewed doc_id hashing) per-shard QPS variance. Resharding; randomize doc_id (Twitter's snowflake IDs are good for this); upstream load balancer.
Index corruption (bad merge, disk bit-rot) Checksum mismatch, recall@K regression vs. canary index. Always keep N-1 prior segment generations; can roll back. Checksums on every segment.
Stale shards (replica missed updates) Replication lag, shard version drift. Pull from authoritative log (Kafka offset), not from sibling shards. Same shape as durability-at-scale.
Freshness lag (live tier behind) now - max(indexed_timestamp) per shard. Backpressure the indexer; scale up consumers; alert when lag > N seconds.
Ranker model drift (training/serving skew, distribution shift) Online vs. offline nDCG delta; CTR by bucket; calibration. Daily retrain; A/B + shadow eval before ramp; auto-rollback if online metric drops X%. Closely related to idempotency-exactly-once for click logging.
Feature store online/offline skew Per-feature distribution diff. Single source of truth in offline; replicate to online; emit feature snapshots at training time; checksum at serve time.
Hot query stampede ("when an event breaks") Cache miss rate spike, backend QPS spike. Top-results cache with negative TTL; request coalescing; admission control. Cross-link cache-stampede-read-amplification.
Visibility leak (deleted tweet still searchable) Time-to-purge SLA, sample audits. Deletion topic with strict ordering (ordering-causal-consistency); visibility filter at query time as a defense-in-depth.
Adversarial / spam queries Anomaly detection on per-query cost. Per-IP rate limits; query cost limits (backpressure-flow-control); query syntax allowlist.
Embedding model rollover Recall delta vs. old index; dual-write period length. Always dual-index for a window; switch reader after recall@K parity is verified.
Index build failure Pipeline SLO, freshness alarm. Keep prior generation; alert on age of "current" index.
ANN recall regression Offline recall@K on canary set; per-segment HNSW M / efConstruction drift. Lock ANN build params per version; re-evaluate on each model rev.
Slow / runaway query Per-query latency, posting-list length cap. Time budget per query; abandon on exceed; return partial.

7.1 Operational golden signals

  • p50 and p99 end-to-end latency (per region, per query class).
  • Per-shard p99; shard-timeout rate.
  • Recall@K and nDCG on a held-out judged set (offline, daily).
  • CTR / dwell-time / abandonment by bucket (online, hourly).
  • Indexing lag (live tier).
  • Index age / staleness (main tier).
  • Cache hit rate and hit-rate by query class.
  • Feature freshness (max age of any feature in the request).
  • Ranker prediction distribution (skew / drift detection).

8. Combinations with other patterns

  • Fan-out write amplification — search is the read-side mirror of newsfeed fan-out. The fan-out pattern pushes updates to many timelines; the search pattern pulls candidates from many shards. The latency budget allocation across many parallel calls is the same problem.
  • Hot key skew — head queries, breaking news, the term "the": all manifestations of skew at the query layer. Mitigations (caching, shard randomization, request coalescing) overlap heavily.
  • Cache stampede / read amplification — top-results cache, feature-store cache; both must defend against thundering-herd on miss.
  • Backpressure / flow control — when query volume spikes, you must shed expensive queries before the index falls behind. Per-query cost accounting is essential.
  • Geo-spatial queries — Yelp / Uber search adds a geo predicate on top of the relevance ranking. The two-stage pattern stays the same; the candidate generator changes from inverted index to geohash / S2 / quadtree.
  • Real-time aggregation — engagement features (likes, views, CTR) that feed ranking are themselves a real-time aggregation problem. The freshness of these features is often the binding constraint on the ranker's quality.
  • Idempotency / exactly-once — click-log ingestion must dedupe so ranker training is not poisoned. Two impressions of the same (query, doc, user) must merge cleanly.
  • Ordering / causal consistency — deletion events must be processed in order with the original publish; otherwise a deleted document re-appears.
  • Durability at scale — the index is derived state; the authoritative log is the input stream. You can always rebuild the index from the log; you cannot rebuild the log from the index.
  • Coordination / exactly-one execution — index build jobs (segment merges, ANN rebuilds) need leader election so we don't double-spend the compute budget.

9. Likely interviewer probes

  1. "Your fan-out is to 1000 shards and your p99 budget is 100 ms. Walk me through the math on tail latency. What is your shard-level p99 target and how do you stay there?" — push for per-shard timeout, hedged requests, replica selection by latency, and over-fetching.
  2. "What happens when one shard goes offline?" — replica failover, partial results, graceful degradation, query retry policy, why returning best-effort is the right call for search but wrong for transactions.
  3. "Your ranker just shipped a new model and online CTR dropped 3% overnight. How do you debug?" — A/B vs. shadow, feature drift, training/serving skew, calibration, auto-rollback, the role of holdouts.
  4. "How do you handle a tweet being deleted? At index time or query time?" — both. Index-time eventual purge for cost; query-time filter for correctness. The deletion topic and the audit story.
  5. "How do you decide between document-sharded and term-sharded?" — load distribution under realistic query mix; one slow term vs. one slow shard; how Lucene resolved this question 20 years ago.
  6. "Tell me about your embedding refresh. The model is retrained — what happens to the existing ANN index?" — dual-index period, recall parity gate, rolling switch; why you cannot just hot-swap.
  7. "Walk me through the offline → online feature pipeline. Where do bugs hide?" — training/serving skew, definition drift, log-and-wait validation, checksums, replay.
  8. "How would you A/B test a new ranker?" — interleaved evaluation vs. side-by-side; minimum detectable effect; guardrail metrics (latency, error rate, fairness); ramp policy.
  9. "What is your indexing pipeline's recovery story if the Kafka topic is corrupted?" — log is the source of truth; can we rebuild from the upstream system (the tweet store)? Backfill replays.
  10. "How do you handle multilingual queries?" — language-specific analyzers, multilingual embeddings, fallback strategies, query-language detection.

10. Common traps → stronger moves

Trap Stronger move
"I'll just put everything in Elasticsearch." Name the tiering, the shard count, the per-shard p99 target, the freshness pipeline, and the second-stage ranker. ES is one component, not a design.
Using one giant index for everything. Tier: live (in-memory, seconds-fresh) + main (on-disk, batch-built). Possibly per-doc-type sub-indexes.
Term-sharded as a default. Document-sharded; mention term-sharded only as the alternative you rejected.
Reflexively reaching for hybrid retrieval. Name the query classes you expect (exact-match, intent, structured filter, "more like this"). If only one class matters, ship one retrieval mode. If multiple classes exist and BM25 measurably misses one of them, add ANN and union; let the second-stage ranker arbitrate.
Ranking with ORDER BY created_at DESC. Learn-to-rank (LambdaMART, DNN); name the features and the training signal (clicks + dwell, not just clicks).
Computing features at query time from raw data. Online feature store; precompute aggregates; cache; emit feature snapshots at training time to preserve offline/online parity.
Failing the whole query on one shard failure. Per-shard timeout + partial results + degraded-mode banner. Document the recall hit.
Caching with infinite TTL on head queries. Short TTL (10-60 s); cache key includes user segment if personalized; negative TTL to soak up cache misses on tail; coalescing on stampede.
Indexing deletes by overwriting in place. Soft delete in index + visibility filter at query time + background purge. Hard delete is too slow at scale.
"We'll personalize later." Reserve the per-user feature space and the candidate-injection slot in stage 2 from day one; personalization retrofits are painful.
Treating the ranker as a static asset. The ranker is a service with its own lifecycle: train, shadow, A/B, ramp, monitor, rollback. Name an owner.
Ignoring offline metrics. nDCG / recall@K / MRR on a judged set; daily; with regression alarms. Online CTR alone leads to feedback loops.
Forgetting visibility / safety filters. Apply at query time after rank; treat as a hard requirement, not a flag. The cost is small; the leak is reputation-ending.

11. Staff-level answer checklist

  • [ ] Sized the problem first: corpus size, QPS, relevance-quality bar. Called out that single-node Elasticsearch with BM25 is the right answer below ~10M documents / low QPS / no semantic queries — and only escalated to two-stage when the latency or quality math actually requires it.
  • [ ] Named the recall vs. ranking split out loud within the first two minutes.
  • [ ] Picked two-stage retrieval (cheap candidate generation, expensive ranking) and justified it with latency math — specifically, that (candidate count × per-candidate ranker cost) over the full corpus would blow the budget.
  • [ ] Named the inverted index (Lucene/Elasticsearch) for keyword recall, and — if the query mix has intent / semantic / "find similar" classes — ANN (HNSW/IVF/ScaNN) for semantic recall, with the union justified by a named query class that keyword misses. If the corpus is exact-match-only, said so and skipped ANN.
  • [ ] Chose document-sharded scatter-gather; named the per-shard timeout and the partial-result policy.
  • [ ] Specified a live tier + main tier with the freshness target (seconds vs. nightly) and the merge story at query time.
  • [ ] Specified a learned ranker (LambdaMART or DNN), the feature sources, and where features live (online feature store).
  • [ ] Discussed BM25 as the baseline the learned ranker must beat, plus query rewriting / expansion.
  • [ ] Named caching for head queries with TTL and stampede protection.
  • [ ] Talked about failure modes: shard timeouts, model drift, freshness lag, deletion ordering, training/serving skew.
  • [ ] Talked about operations: nDCG/CTR/latency dashboards, A/B + shadow, ramp + rollback for ranker changes.
  • [ ] Drew a capacity napkin: corpus size, shard count, per-shard p99 budget, QPS per shard after caching.
  • [ ] Linked to adjacent patterns: fan-out, hot key, cache stampede, real-time aggregation, geo, ordering.
  • [ ] Said what is out of scope (crawl pipeline, ad auction internals) and named the trigger that would bring them back.
  • [ ] Closed with the biggest open risk: usually ranker drift or training/serving skew, not raw latency.
