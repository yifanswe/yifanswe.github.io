Strong Consistency Under Contention

2026/05/31  В·  System Design Patterns
Contents
Contents

When N actors race for one scarce resource, the system must produce exactly one winner and tell every loser "no" вҖ” never silently overcommit. See the Staff System Design Glossary for terms. Question docs where this is the load-bearing pattern: - design-ticketmaster.md - design-a-flash-sale-for-an-e-commerce-site.md - design-a-stock-exchange.md - design-a-payment-system.md - design-the-stripe-payment-gateway.md - design-a-distributed-lock-manager-like-chubby.md - design-amazon-shopping-cart.md

1. The fundamental hardness

Distributed systems default to availability. You replicate state across many machines so that any of them can serve a read or accept a write. That works beautifully for "post a tweet," "view a profile," or "increment a like counter," because the truth is additive and a few seconds of skew costs nothing.

This pattern is the opposite. The truth is subtractive and bounded. There is one seat at row 4, column 12, for the Tesla Cybertruck unveil. There is $143.27 in Yifan's checking account. There is one last unit of the limited-edition graphics card. Two requests at the same millisecond cannot both win. The Atomicity, Consistency, Isolation, Durability (ACID) "Isolation" property вҖ” that two transactions should appear to run one after the other вҖ” is precisely what every distributed-database trick tries to relax, because serializing concurrent writes is expensive. Under contention, you cannot relax it. You have to pay.

A staff-level caveat before going further: the bar for "strong consistency" is set by the product, not by reflex. A bank balance, a stock-exchange fill, and a username reservation cannot tolerate any overcommit. A concert seat or a flash-sale unit usually can вҖ” most ticketing and e-commerce platforms quietly absorb a small overselling rate (a fraction of a percent) by issuing refunds or upgrades, because the business cost of compensation is lower than the engineering cost of perfect serialization at peak. The pattern below assumes the strict case; when the interviewer's product actually allows compensation, the right answer is to call that out and pick a cheaper design.

The invariant is brutal in its simplicity:

For a contested resource of capacity C, the count of successful commits must equal min(C, demand). Never C + 1. Never a silent loss of a successful commit. And every loser must observe a deterministic "no" so they can stop, retry against a different key, or give up cleanly.

What makes this hard at staff level is not the lock itself. The hard part is doing this under three simultaneous pressures: (1) throughput вҖ” 10,000 requests per second (RPS) hitting a single key; (2) tail latency вҖ” your locking layer cannot turn a 50 millisecond (ms) p99 user click into a 5 second timeout; (3) failure вҖ” the process holding the lock can die mid-checkout, the lock-manager can lose quorum, the database can fail over mid-transaction. Any one of these turns a "just use a database transaction" design into either oversells, deadlocks, ghost reservations, or full outages.

A naive design under contention breaks in one of five ways. Lost update: two requests both read inventory=1, both decrement, both write inventory=0, both succeed; you sold two. Phantom reservation: a user holds a seat in a session, the session dies silently, the seat is held forever. Double-spend on retry: a payment client times out, retries; the original request actually committed; the user is charged twice. Lock holder dies: a node holds a lock, garbage collection (GC) pauses it for 12 seconds, by then another node grabbed the lock thinking it was abandoned, and now two nodes both "own" the resource. Cascading lock contention: 10,000 buyers all wait on the same row lock, your database connection pool exhausts, the rest of the site goes down.

2. How to recognize it in the interview

Prompt phrases that signal this pattern

  • "Limited inventory" / "last item" / "sold out" / "in stock count."
  • "Two users cannot book the same X."
  • "Exactly one winner" / "no oversell" / "no double-charge."
  • "Concert tickets" / "flash sale" / "drop" / "lottery."
  • "Match buyer with seller" / "order book" / "trade."
  • "Balance" / "transfer" / "debit" / "authorization."
  • "Reserve and confirm" / "hold for N minutes" / "checkout flow."
  • "First come first served" with high concurrency.

Architectural smells in stated requirements

  • A scalar field (stock_count, balance, seat_status) that is decremented rather than appended.
  • A unique constraint that is semantically a business rule, not just a database hint (UNIQUE(seat_id) in bookings).
  • A workflow with two phases (reserve вҶ’ confirm, authorize вҶ’ capture) where the gap can be seconds to minutes of user think-time.
  • Reads of stock count appearing on a high-traffic landing page while writes need to be exact. (This is the read-write split вҖ” see cache-stampede-read-amplification.md.)
  • A single hot key вҖ” one concert, one stock symbol, one celebrity-restock вҖ” concentrating contention. Overlaps with hot-key-skew.md.
  • An idempotency-key requirement. Overlaps with idempotency-exactly-once.md.

Example "design X" prompts where this is the core hardness

  1. Ticketmaster / Eventbrite seat selection вҖ” 50,000-seat stadium, 200,000 fans hitting "select" in the first 30 seconds.
  2. Flash sale on a limited-edition product вҖ” 1,000 units of a graphics card go on sale at noon, 100,000 people on the page.
  3. Stock exchange / matching engine вҖ” every order against the book must commit in deterministic price-time priority.
  4. Payment system / debit on a wallet balance вҖ” bank balance must never go negative even if two ATMs and a card terminal hit the same account at once.
  5. Uber driver dispatch вҖ” one driver can only accept one rider at a time; two riders can be assigned to two different drivers but never both to the same one.
  6. Hotel / Airbnb booking on a specific date range вҖ” overlapping date ranges for the same listing must not both succeed.
  7. Username / handle reservation вҖ” first person to claim @elonmusk2 wins, everyone else gets "taken."

3. 60-second spoken framing

The invariant here is exact. For a contested resource of capacity C, the system must produce exactly min(C, demand) successful commits, every loser must get a clear "no," and we cannot block the rest of the site while the contended key burns. My default approach is to push all writes for one key through a single serialization point вҖ” either a single-writer partition or a database row lock вҖ” and protect the user experience with a short-TTL reservation token so the seat is held during checkout but released automatically on abandon. On top of that I layer optimistic concurrency control for low-contention items, a database-level unique constraint as a final safety net so even a bug in the application layer cannot oversell, and an idempotency key on every write so retries are safe. The two failure modes I plan for are lock-holder death (use fencing tokens, never raw locks) and hot-key meltdown (cap concurrency per key, shed load early).

4. Canonical toolkit

4.1 Single-writer partition (the default at scale)

What: For every contested key K, route all writes to the same node. That node serializes writes through an in-memory queue. There is no lock, no compare-and-swap (CAS) retry storm вҖ” just a queue. This is how Kafka achieves per-partition ordering, how an order-matching engine handles a single stock symbol, and how LinkedIn Espresso handles per-key linearizability.

When to use: High contention on a finite set of keys, where the per-key throughput fits a single core. Realistic per-core ceilings: ~5-20k ops/sec in Python (GIL-bound), ~50-200k ops/sec in Java or Go with a tight in-memory state machine, 500k+ in C++ or Rust. The number that matters is "what does the per-key apply cost in CPU?" вҖ” if a decrement-plus-log-append is ~5 microseconds, one core saturates around 200k ops/sec, and most of the budget is the replicated log append. Use this for matching engines, ticket seat-allocation, ad auction floors.

Cost: You lose the ability to load-balance hot keys across machines. If one key is 100x hotter than the cluster average, that one node will melt. You also need a routing layer that consistently maps K to the same node вҖ” usually consistent hashing with a versioned partition map. Failover means draining the in-flight queue and replaying on a backup; you need replication (typically Raft, Paxos, or a Kafka-style log) for the per-partition state.

When it breaks: One key gets so hot that one node cannot keep up. Then you must shard the key itself, which usually means changing the product (allocate seats into sub-blocks, split a stock symbol across price ranges). Or the node dies and you have to fail over without losing the queue вҖ” you need synchronous replication of the in-memory queue's input.

# Sketch: single-writer partition, applied per key.
# In production this is the same idea as a Raft state machine.

import asyncio
from collections import defaultdict

class SeatPartition:
    def __init__(self, seat_ids):
        # source of truth, in memory, replicated via log
        self.seats = {s: "FREE" for s in seat_ids}
        self.queues = defaultdict(asyncio.Queue)

    async def reserve(self, seat_id, user_id, request_id):
        # Caller awaits this; per-seat queue serializes the apply step.
        future = asyncio.get_event_loop().create_future()
        await self.queues[seat_id].put((user_id, request_id, future))
        return await future

    async def run_partition(self, seat_id):
        # One coroutine per seat. No lock вҖ” there is exactly one writer.
        q = self.queues[seat_id]
        while True:
            user_id, request_id, future = await q.get()
            if self.seats[seat_id] == "FREE":
                self.seats[seat_id] = ("HELD", user_id, request_id)
                # Append to replicated log BEFORE acking the future.
                await replicate(("RESERVE", seat_id, user_id, request_id))
                future.set_result("ok")
            else:
                future.set_result("taken")

4.2 Optimistic concurrency control (CAS with a version)

What: Read the row with its current version number. Compute the new value. Write back with WHERE version = old_version. If another writer already incremented the version, your update affects zero rows; retry from the read.

-- read
SELECT stock_count, version FROM inventory WHERE sku = 'X';
-- write
UPDATE inventory
   SET stock_count = stock_count - 1, version = version + 1
 WHERE sku = 'X' AND version = :old_version AND stock_count > 0;
-- if 0 rows affected, retry

def try_decrement(sku):
    for attempt in range(MAX_RETRIES):
        row = db.fetch_one("SELECT stock_count, version FROM inventory WHERE sku=%s", sku)
        if row.stock_count <= 0:
            return "sold_out"
        affected = db.execute(
            "UPDATE inventory SET stock_count = stock_count - 1, "
            "version = version + 1 WHERE sku=%s AND version=%s AND stock_count > 0",
            sku, row.version,
        )
        if affected == 1:
            return "ok"
        # else: someone else won this round, retry
    return "give_up"

When to use: Low to moderate contention. Updates on user-owned data (profile, preferences). Inventory of items with many SKUs where any one SKU is rarely the hot one. The Dynamo paper's vector clock siblings are a generalized form of CAS вҖ” see amazon-dynamo.md.

Cost: Retry rate scales roughly with the number of concurrent writers on the same key during one round-trip. With round-trip latency t and N concurrent writers on one key, the expected number of CAS attempts before any individual writer succeeds is ~N/2, so total work scales as O(N^2). Concretely: at 10 concurrent writers on one row and a 2 ms round-trip, each request averages ~5 attempts (10 ms tail). At 100 concurrent writers, average ~50 attempts (100 ms tail) and the database is doing 50x the read+update work that a single-writer design would do. Past a few tens of concurrent writers on one key, optimistic CAS stops being cheaper than just locking the row.

When it breaks: Hot key, high contention. The crossover where pessimistic locking or single-writer is cheaper than CAS is usually in the range of ~10-50 concurrent writers per key, depending on round-trip latency and update cost. Above that, switch.

4.3 Pessimistic locking (SELECT ... FOR UPDATE)

What: Open a database transaction. Lock the row. Read it. Modify it. Commit. The database serializes everyone touching that row.

BEGIN;
SELECT stock_count FROM inventory WHERE sku = 'X' FOR UPDATE;  -- blocks others
-- application checks and decides
UPDATE inventory SET stock_count = stock_count - 1 WHERE sku = 'X';
COMMIT;

When to use: Single-region, single relational database, simple flows. The staff-level move is to recognize how far this goes: if the transaction body is just SELECT ... FOR UPDATE; UPDATE; COMMIT with no application-side think-time, the lock is held for ~1-3 ms (a couple of round-trips), so one row can sustain 300-1000 RPS before the lock-wait queue noticeably backs up. Even better, the entire workflow collapses to a single UPDATE inventory SET stock = stock - 1 WHERE sku = X AND stock > 0 вҖ” atomic in Postgres or MySQL, lock held for microseconds, scales to thousands of RPS per row. Most interview prompts at the "tens to low hundreds RPS per key" level genuinely do not need anything beyond this. Common for B2B order entry, hotel booking, internal admin tools.

Cost: The danger is holding the transaction across application-side work. A 100 ms application-side delay between FOR UPDATE and COMMIT, multiplied by a few hundred concurrent contenders, will saturate a typical 50-200 connection pool and freeze the rest of the site. Two rules of thumb: never await a user action inside the transaction; never await a network call to another service inside the transaction (use a reservation token outside the transaction instead). Deadlock probability also increases when transactions touch multiple rows in inconsistent order вҖ” always lock by a fixed key order.

When it breaks: Cross-shard or cross-region locking вҖ” Postgres or MySQL row locks do not extend across shards or regions. Massive sustained contention on one row вҖ” at ~5-10k RPS on one row with a 1 ms transaction, the lock-wait queue depth (queries Г— hold_time / request_interval) starts to exceed the connection pool and the queue itself becomes the bottleneck. At that point you have a true hot-key problem, not a locking problem, and the fix is single-writer partitioning, not a different lock primitive.

4.4 Distributed lock with fencing tokens (Chubby / ZooKeeper / etcd)

What: A separate coordination service grants a lock. The lock holder gets a monotonically increasing fencing token. Every downstream service that the lock protects must reject any operation whose token is older than the latest seen token. This protects against the lock-holder-pause failure mode that destroys naive distributed locks.

client A acquires lock вҶ’ token = 33
   вҶ“
client A is GC-paused for 12 seconds
   вҶ“
lock expires вҶ’ client B acquires lock вҶ’ token = 34
   вҶ“
client B writes to storage with token=34 вҶ’ storage records "last=34"
   вҶ“
client A wakes up, tries to write with token=33 вҶ’ storage rejects (33 < 34)

# Pseudocode, deliberately verbose to show the fencing step.
token = lock_service.acquire("seat:A12", ttl_sec=30)  # token = e.g. 33
if token is None:
    return "could_not_acquire"
try:
    # Storage must verify the token is the latest. The token is part of every write.
    response = storage.cas_update(
        key="seat:A12",
        expected_state="FREE",
        new_state=("HELD", user_id),
        fencing_token=token,   # storage compares to its own monotonic counter
    )
finally:
    lock_service.release("seat:A12", token=token)

When to use: When the resource lives outside the lock service (a database row, a file, a remote API call) and you need an external lock manager because the resource itself does not support strong leadership. Common for distributed job leadership, leader election, exclusive access to a shared file. See design-a-distributed-lock-manager-like-chubby.md for the deep dive.

Cost: Adds a dependency on a quorum-based system (ZooKeeper, etcd, Chubby) with its own availability, latency, and operational story. Cross-service token enforcement is invasive вҖ” every downstream API needs to accept and check tokens.

When it breaks: If you forget the fencing token, a paused-then-resumed client silently overwrites a newer holder's work. This is the single most common subtle bug in distributed-lock design. Martin Kleppmann's "How to do distributed locking" essay is the canonical reference вҖ” interviewers love to probe this.

4.5 Reservation tokens with TTL

What: Two-step booking. Step 1: "reserve" decrements available inventory by 1 and writes a row (resource_id, user_id, token, expires_at) with a short TTL (typically 60-600 seconds). Step 2: "confirm" turns the reservation into a permanent booking. A background job (or a lazy check on read) reclaims expired reservations.

t=0  user clicks "select seat A12"
     в”Җв”Җв–ә reservation_create(seat=A12, user=Yifan, expires_at=t+300)
     в”Җв”Җв–ә seat state = HELD-by-Yifan-until-t+300

t=10 Yifan enters payment info, confirms
     в”Җв”Җв–ә reservation_confirm(token=...) вҶ’ booking row, seat=BOOKED

OR

t=400 (Yifan abandoned tab)
     в”Җв”Җв–ә sweeper or next caller's lazy check вҶ’ seat = FREE again

def reserve(seat_id, user_id, ttl_sec=300):
    now = time.time()
    # Atomic: either we acquire the seat or someone else owns it.
    affected = db.execute(
        """
        UPDATE seats
           SET held_by = %s, held_until = %s, version = version + 1
         WHERE seat_id = %s
           AND (held_by IS NULL OR held_until < %s)
        """,
        user_id, now + ttl_sec, seat_id, now,
    )
    if affected == 1:
        return make_token(seat_id, user_id, now + ttl_sec)
    return None  # someone else holds it and the hold hasn't expired

def confirm(token):
    seat_id, user_id, expires_at = parse_token(token)
    if time.time() > expires_at:
        return "expired"
    affected = db.execute(
        """
        UPDATE seats
           SET booked_by = %s, status = 'BOOKED', held_by = NULL, held_until = NULL
         WHERE seat_id = %s AND held_by = %s AND held_until >= %s
        """,
        user_id, seat_id, user_id, time.time(),
    )
    return "ok" if affected == 1 else "lost_hold"

When to use: Any flow with user think-time between "I want this" and "I committed to this." Ticket purchasing, hotel booking, ride dispatch ("driver, you have 15 seconds to accept"). This is the de facto standard for Ticketmaster, StubHub, Airbnb.

Cost: TTL is a tuning problem disguised as a number, not a "pick a reasonable value" choice. Too short and you reclaim seats from users who are still entering their card; too long and during a flash sale you cap effective inventory at capacity Г— (TTL / median_checkout_time), suppressing real demand. You also need a sweeper and lazy reclaim on read, because either one alone has a failure mode (sweeper falls behind under load; lazy reclaim never fires for keys nobody re-reads). The state space вҖ” FREE, HELD, BOOKED, EXPIRED вҖ” has an asymmetry that is easy to miss: reserve and release are not symmetric. A reserve happens in one atomic write, but a release can happen via three different paths (explicit user release, sweeper expiry, lazy reclaim on the next reader's request). Each of those paths must agree on "is this hold actually expired right now?" using a single source of time (the storage's clock, not any client's clock) and a single source of truth for the token (the holder's identity plus expiry, not just the identity вҖ” otherwise a slow confirm can race a successful steal).

When it breaks: The classic zombie-reservation race: user A's hold expires at t=300; sweeper or lazy reclaim resets the seat to FREE at t=301; user B grabs the seat at t=302; user A's delayed confirm request arrives at t=303 with their old (now-expired) token. The confirm must reject вҖ” which is why the confirm query in the code above re-checks held_by = user_id AND held_until >= now() instead of trusting the token alone. The second failure: same user races their own retries (network flakes during reserve) and ends up holding two seats. Idempotency on reserve keyed by (user_id, event_id, request_id) fixes the retry case, but does not fix a legitimately-greedy user who reserves under multiple sessions вҖ” for that you need a per-user concurrent-hold cap inside the reservation service.

4.6 Two-phase reservation (intent вҶ’ commit) across services

What: When the commit requires multiple systems to all agree (e.g., debit my wallet AND mark seat as booked AND notify the venue), you cannot atomically update them all. The reservation step is the intent: write a Saga state row, debit nothing, mark seat HELD. The commit step is the apply: debit wallet, finalize booking, idempotent confirm on the venue. If any step fails, run the compensation: refund wallet, release seat. See idempotency-exactly-once.md and coordination-exactly-one-execution.md.

When to use: Multi-service commits. Payment + inventory + notification. Avoid distributed two-phase commit (2PC) вҖ” it blocks under coordinator failure. Use Saga with explicit compensation instead.

Cost: Visible intermediate states. Need careful idempotency on each step. Operationally complex.

When it breaks: Compensation can fail too. You need a permanent reconciliation job to detect and alarm on stuck Sagas.

4.7 Database-level constraint as last line of defense

What: Add a UNIQUE(seat_id, event_id) constraint on the bookings table. Even if every application-layer check is buggy, the database itself will refuse to insert a duplicate booking.

CREATE TABLE bookings (
  booking_id  BIGINT PRIMARY KEY,
  event_id    BIGINT NOT NULL,
  seat_id     BIGINT NOT NULL,
  user_id     BIGINT NOT NULL,
  created_at  TIMESTAMP NOT NULL,
  UNIQUE (event_id, seat_id)   -- the load-bearing constraint
);

When to use: Always, when the semantic invariant can be expressed as a uniqueness or check constraint. Cheap, simple, foolproof. The cost is essentially zero.

Cost: Negligible. Insertion is O(log n) on the index. The only downside is that constraint violations show up as exceptions, so the application must distinguish between "duplicate" (return "taken") and "real error" (retry or alarm).

When it breaks: Cannot express more complex invariants (e.g., "no overlapping date range"). For those, use exclusion constraints (Postgres EXCLUDE USING GIST) or a custom serialization layer.

5. Worked example: naive вҶ’ broken вҶ’ fixed

Scenario: A Tesla Cybertruck pre-order event. The Foundation Series has 10,000 units. The order page opens at 9:00 AM Pacific Time. We expect 500,000 unique users on the page in the first 60 seconds. Each user clicks "Reserve mine" essentially simultaneously. The product invariant: exactly 10,000 successful reservations, no oversells, every loser sees "sold out" within 5 seconds.

Naive version (broken)

# Every web request runs this. No locking.
def reserve():
    row = db.fetch_one("SELECT available FROM inventory WHERE sku='cybertruck-fs'")
    if row.available > 0:
        # ... user enters payment info, picks color, takes 30 seconds ...
        db.execute("UPDATE inventory SET available = available - 1 WHERE sku='cybertruck-fs'")
        db.execute("INSERT INTO orders (user_id, sku) VALUES (...)")
        return "confirmed"
    return "sold_out"

What goes wrong, in order of severity:

  1. Massive oversell. The UPDATE has no WHERE available > 0 guard, and the SELECT-then-UPDATE window straddles 30 seconds of user input. Every request that passed the SELECT proceeds to decrement, regardless of what the row has become. Even if only ~10,000 requests pass the SELECT concurrently (read at the same instant), each one decrements unconditionally; the counter goes deeply negative. The fix on this line alone is UPDATE ... WHERE available > 0 and rely on affected_rows == 1, which a single atomic statement guarantees. The real bug being illustrated is "application-side read, then unconditional write" вҖ” and that is what every technique in section 4 is designed to prevent.
  2. Database meltdown. The web tier can hold maybe 50-200 DB connections; the database accepts that many concurrent transactions, the rest queue at the connection pool layer. With 30-second user think-time held inside the transaction, throughput collapses to pool_size / hold_time вүҲ a handful of commits per second, while the queue grows without bound. The rest of the Tesla site sharing the same database also queues. Within a minute the entire pool is held by stuck checkout sessions.
  3. Reads from caches. The "available" count on the marketing page is cached. After "sold out" the cache says "92 available" for 15 minutes, generating angry users.
  4. Stuck checkouts. Some users got past the SELECT, lost network, never finished. Their slot is consumed in business terms but their order row never landed. Inventory is now mismatched from order intent.

Fixed version (production grade)

# Inventory service - single writer for sku.

class InventoryPartition:
    def __init__(self, sku, capacity):
        self.sku = sku
        self.available = capacity
        self.held = {}  # token -> (user_id, expires_at)

    def reserve(self, user_id, idempotency_key):
        # Single-threaded apply. No lock needed because there's one writer.

        # Idempotency: if same user retries, return their existing hold.
        for token, (uid, exp) in self.held.items():
            if uid == user_id and self._matches_idem(token, idempotency_key):
                return ("ok", token)

        if self.available <= 0:
            return ("sold_out", None)

        token = generate_token(self.sku, user_id, idempotency_key)
        self.available -= 1
        self.held[token] = (user_id, time.time() + 600)
        replicate_log_append({
            "op": "HELD",
            "sku": self.sku,
            "token": token,
            "user_id": user_id,
            "available_after": self.available,
        })
        return ("ok", token)

    def confirm(self, token, user_id):
        held = self.held.get(token)
        if not held or held[0] != user_id:
            return "invalid"
        if time.time() > held[1]:
            return "expired"  # will be reclaimed by sweeper
        del self.held[token]
        replicate_log_append({"op": "SOLD", "sku": self.sku, "token": token})
        return "ok"

    def release(self, token):
        held = self.held.pop(token, None)
        if held:
            self.available += 1
            replicate_log_append({"op": "RELEASE", "sku": self.sku, "token": token})

    def sweep(self):
        now = time.time()
        for token, (uid, exp) in list(self.held.items()):
            if exp < now:
                self.release(token)

Order finalization keeps the database constraint as the floor:

INSERT INTO orders (user_id, sku, hold_token, status)
VALUES (:user_id, :sku, :hold_token, 'PAID');
-- UNIQUE(hold_token) ensures even if the application layer retries, only one row lands.

Why this works:

Pressure How the design responds
Oversell Single writer for the SKU is the only entity that decrements. The atomic counter cannot go below zero.
Throughput One core can sustain ~50-200k decrement-and-log per second in a compiled language (Java/Go/Rust); the log-append round-trip dominates. We only need ~10k decrement events total over 60 seconds, so a single core is two orders of magnitude over-provisioned.
Database meltdown The database is no longer in the hot path of decrement. It only receives confirmed orders, at human-rate.
Marketing-page cache lie The reservation service emits a "sold_out" event; the marketing service consumes and updates the cache. Stale by seconds, not minutes.
Stuck checkouts TTL on the hold; sweeper releases; database UNIQUE constraint prevents double-commit if a stuck flow ever resumes.
Lock-holder dies Single-writer node's state is replicated synchronously to a follower. Failover restores the counter.

6. Variants and how to choose

Decision table

Technique Best fit Throughput per key Operational cost Failure mode when stressed
Single-writer partition Massive contention on a known key set (matching engine, flash sale, ticket master event) Highest (~50-200k/sec per core in compiled languages; log-append-bound) High setup (routing, replication) One key hotter than one core can serve
Optimistic CCC (CAS + version) Low to moderate contention; many keys, few collisions Good when collisions are rare; total work is O(N^2) in concurrent writers per key Low Retry storm past ~10-50 concurrent writers per key
Pessimistic lock (SELECT FOR UPDATE) Single relational DB, simple flows, no app-side think-time inside the transaction A single atomic UPDATE sustains thousands of RPS per row; transactions with round-trips hold ~ms locks вҶ’ low hundreds RPS per row before queueing dominates Low (built in) Connection-pool starvation when transactions hold across app-side work; deadlocks from inconsistent lock order
Distributed lock + fencing token Resource lives outside any one DB; need exclusive access across services Moderate (limited by lock service) Medium-high (lock service + fencing protocol) Forgetting the fence; lock service outage
Reservation token with TTL Multi-step user flow with think-time Same as the underlying atomic step Medium (TTL tuning, sweeper) TTL too short or too long, sweeper falls behind
Two-phase reservation (Saga) Cross-service commit (payment + inventory + notify) Limited by slowest step High (compensation, reconciliation) Compensation failure, stuck Sagas
Database UNIQUE constraint Anywhere uniqueness is the invariant Free (always-on) Zero Cannot express non-uniqueness invariants

If the interviewer says "single relational database, tens of orders per minute, must never oversell": pessimistic lock + UNIQUE constraint. Done.

If the interviewer says "10k requests per second on one SKU during a flash sale": single-writer partition + reservation TTL + UNIQUE.

If the interviewer says "payment must debit a wallet and create an order, across two services": Saga with compensation + idempotency keys + UNIQUE.

8. Combinations with other patterns

This pattern almost never stands alone. Common pairings:

  • hot-key-skew.md вҖ” the contended key is by definition hot. Pre-partition by sub-key (seat-block, price-bucket), shed load with a waiting room, or apply per-key rate limits. The flash-sale example combines both patterns.
  • idempotency-exactly-once.md вҖ” every write under contention must be safe to retry. The client cannot know if their first attempt committed when the network drops. Idempotency keys are mandatory.
  • cache-stampede-read-amplification.md вҖ” the read side often shows "12 left!" which must be approximately correct. Cache the count, but use the strong-consistency layer for the actual decrement.
  • coordination-exactly-one-execution.md вҖ” when the result of the contention must trigger exactly one downstream action (one confirmation email, one ledger entry). Outbox pattern + idempotent consumer.
  • durability-at-scale.md вҖ” the winner's commit must survive node loss. Synchronous replication of the partition log (Raft) or quorum write to a durable store.
  • backpressure-flow-control.md вҖ” when 500k users hit one URL, you must drop or queue early. The waiting-room pattern.
  • ordering-causal-consistency.md вҖ” a stock-exchange order book needs strict per-symbol ordering (price-time priority). The single-writer partition pattern is the same machinery, with an extra ordering constraint.

9. Likely interviewer probes

  1. "What happens when the node holding your single-writer partition dies mid-reservation?" Answer: synchronous log replication to at least one follower (Raft / Multi-Paxos); leader election triggered by failure detector; follower replays committed log entries; in-flight uncommitted requests are returned as "unknown" so client retries with idempotency key; bounded recovery time objective (RTO) of seconds, not minutes.

  2. "You said you use a distributed lock. What if the lock holder GC-pauses for 15 seconds?" Answer: that is exactly why we use fencing tokens. The lock service issues a monotonically increasing token on every acquire. Every downstream storage operation includes the token; storage rejects tokens lower than the highest it has seen. The paused client wakes up, tries to write with stale token, gets rejected. This is THE central probe for distributed locking вҖ” interviewer is checking if you know Martin Kleppmann's article.

  3. "How do you handle 500,000 users hitting the page at exactly 9:00:00.000?" Answer: shed load before it reaches the reservation service. Waiting-room pattern: edge servers issue queue tokens, slowly admit them into the checkout flow. Inventory service only sees demand at the rate it can confirm. CAPTCHA or proof-of-work at admission to defeat bots.

  4. "What is the right TTL for a seat hold?" Answer: derive it, do not pick it. Under normal demand (capacity not the binding constraint) TTL is set generously from the user flow вҖ” typical checkout p95 is 60-180 seconds, so TTL of 5-10 minutes gives margin for slow card entry. Under flash-sale conditions the math flips: effective sellable inventory equals capacity Г— (TTL / median_checkout_time), so a 10-minute TTL with 60-second checkouts effectively serializes inventory at 1/10 of capacity, suppressing demand. Compress TTL to ~60-120 seconds, display a visible countdown so users know they're racing, and measure two metrics post-event: hold_to_confirm_ratio (low means TTL too short or UX too slow) and held_seat_recycle_count (high means TTL too long for the demand level). Tune both.

  5. "How do you prevent a user from holding 100 seats by spamming the reserve API?" Answer: per-user rate limit at edge. Per-user concurrent-hold limit enforced inside the reservation service (max_holds_per_user). Idempotency on reserve by (user_id, request_id) so retries do not multiply holds.

  6. "Walk me through the payment flow when the user clicks confirm. What if the network drops mid-call?" Answer: idempotency key generated client-side at hold time, included on every confirm attempt. Payment provider de-duplicates by idempotency key. If the client times out and retries, the second call returns the result of the first. If the payment succeeds but our order finalize fails, a background reconciliation job catches the orphan payment and either finalizes the order or refunds.

  7. "How do you audit that you never oversold?" Answer: every state transition (FREE вҶ’ HELD вҶ’ SOLD) is logged to an append-only event log. A nightly reconciliation job replays the log, recomputes per-SKU counts, compares to the source-of-truth row. Alarm on mismatch. Also, daily report of count(orders WHERE sku=X) вүӨ capacity(X) per SKU; any violation is a Sev-1.

  8. "Why not just use a single atomic UPDATE inventory SET stock = stock - 1 WHERE sku = X AND stock > 0 and call it a day?" Answer: for many products, that is the right answer. A single guarded UPDATE in Postgres or MySQL holds the row lock for ~tens of microseconds and trivially handles up to thousands of RPS per row without any oversell, because the database itself serializes the writes. The technique stack in this doc earns its complexity only when one of four constraints binds: (a) per-key throughput exceeds what one database row can serialize (single-writer partition); (b) the workflow has user think-time between intent and commit (reservation TTL); (c) the commit spans services that cannot share a transaction (Saga); (d) the resource lives outside any one database and an external lock manager is the only option (distributed lock + fencing). If none of those bind, the staff-level move is to call out the single-row UPDATE as sufficient and add only the database UNIQUE constraint as a belt-and-suspenders floor. Reaching for Chubby on a B2B inventory system that does 10 orders per minute is a red flag.

10. Common traps вҶ’ stronger moves

Trap Stronger move
"Just use SELECT FOR UPDATE." Layer it: DB UNIQUE constraint as floor, reservation token with TTL for UX, single-writer partition for scale. State which layer absorbs which failure.
Distributed lock without fencing token. Always pair the lock with a monotonic token enforced at the storage layer; otherwise a paused holder corrupts state.
Treat retries as harmless. Idempotency key on every write; UNIQUE constraint on (user, idem_key); retries must converge to the same outcome.
Hide the contended key. Name it. Say: "the hot key is sku=cybertruck-fs. My design routes it to a dedicated single-writer node."
Optimistic CAS at any scale. Acceptable under low contention; under high contention, retries explode. Switch to single-writer for known hot keys.
Forget about phantom holds. Always pair holds with TTL + sweeper + lazy reclaim on the next read. State the TTL number and the recycle latency target.
Sweeper is the only reclaim mechanism. Add lazy reclaim on read so even if the sweeper falls behind, the next caller can claim an expired hold.
Stale "in stock" count on the marketing page. Use the contention layer as truth; let the marketing page be a derived projection that catches up within seconds. State the SLA on staleness.
Two-phase commit (2PC) across services. 2PC blocks on coordinator failure. Use Saga with explicit compensation and reconciliation.
No upper bound on holds per user. Per-user concurrent-hold cap. Per-user rate limit at edge.
Assume clocks agree across services. Use the storage's own clock for TTL expiry; never trust the client. Or rely on monotonic fencing tokens, not wall time.

11. Staff-level answer checklist

When you give a strong-consistency-under-contention answer, the interviewer should hear all of these in your spoken design:

  • [ ] You named the invariant precisely: commits == min(C, demand), no oversells, every loser sees "no" within a stated latency.
  • [ ] You identified the hot key by name (not "the row," but "the row for event_id=42, seat_id=A12").
  • [ ] You picked one primary serialization mechanism (single-writer, lock, CAS) and stated why.
  • [ ] You layered a database UNIQUE constraint as the last line of defense.
  • [ ] You used a reservation token with a TTL for any flow that has user think-time.
  • [ ] You named idempotency by key on every retry-prone write.
  • [ ] You discussed lock-holder pause and fencing tokens (if locks are in the design at all).
  • [ ] You described the failover story for the serialization node (replication, leader election, RTO).
  • [ ] You described how the read path stays fast (cache, projection) without being the source of truth.
  • [ ] You named one or two metrics that would let an on-call engineer detect oversells, stuck holds, or contention storms.
  • [ ] You discussed load shedding at the edge (waiting room, CAPTCHA, rate limit) so the contended layer is never overwhelmed.
  • [ ] You distinguished between the legal Saga states and stated how a stuck Saga is detected and resolved.
