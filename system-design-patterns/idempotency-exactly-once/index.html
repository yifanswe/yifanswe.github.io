Idempotency / Deduplicating Retries

2026/05/31  ·  System Design Patterns
Contents
Contents

A design discipline for making operations safe to retry: the system produces the same observable effect whether a request is delivered one time or one hundred times, by deduplicating at well-chosen seams rather than chasing the impossible promise of "exactly once" on the wire. See glossary for related terms (quorum, vector clock, write-ahead log). Related question docs: design-a-payment-system, design-the-stripe-payment-gateway, design-a-delayed-payment-system, design-a-notification-system, apache-kafka, design-a-flash-sale-for-an-e-commerce-site, design-a-reminder-alert-system.

1. The fundamental hardness

"Did my request succeed?" is the only question that matters in a distributed system, and it is the one question the network refuses to answer. A client sends a POST /charge. The server commits to the database, the database flushes the write-ahead log (WAL), the response is serialized and put on the wire — and then a transient network drop, a load-balancer reset, a Linux conntrack timeout, or a half-closed Transmission Control Protocol (TCP) connection eats the acknowledgment. From the client's vantage point, the request is in an indeterminate state: it might have succeeded, it might have failed, it might have succeeded and the database has already debited the user's card. The only rational client behavior is to retry. The only rational server behavior is to assume retries will happen, even ones it cannot see (the client may have died and been replaced; a sibling service may be retrying on its behalf; a queue may redeliver hours later).

This is the gap that every payment, signup, message-send, withdrawal, and order-placement system lives in. "Exactly once delivery" at the wire level does not exist: you get at-most-once (fire and forget; messages can be lost) or at-least-once (retry until acked; messages can duplicate). What you can build, by careful design at the application and storage layer, is exactly-once effect — meaning the user gets charged once, the email gets sent once, the row gets inserted once — even though the wire-level request may have arrived zero, one, two, or eleven times. The mechanism is always the same in spirit: assign a stable identifier to the intent of an operation, persist a record keyed by that identifier, and let the persistent record arbitrate whether to do the work or to return the previously computed result.

A staff-level caveat up front: this machinery is not free. A dedup table is an extra row per request, an extra index, a TTL policy, and a piece of state whose availability budget must match the business write. For low-stakes mutations whose accidental duplication is cheap to detect and reverse (e.g., a "favorite" toggle, an analytics event, a cache invalidation), the naive design plus a downstream reconciliation job is often the correct answer. The pattern earns its complexity when duplication is expensive — money moved, inventory committed, externally visible side effects like an SMS or a refund. Name the cost of a duplicate before you reach for the toolkit.

The Two Generals Problem captures the narrow version of why this is irreducible: two parties cannot reach common knowledge that a message was delivered over a lossy channel with a finite exchange, because the last sender can never know whether their final ack was received. The practical takeaway is more modest than the theorem is sometimes used to suggest — you can still build reliable systems, you just cannot rely on the sender's view of delivery to coordinate state. The escape is to make the operation's effect commutative under repetition. That is the whole game: make the second, third, and Nth attempts arithmetic no-ops with respect to externally visible state, while still returning a coherent answer to the caller.

Three forces conspire to make this hard at scale. First, the dedup state must itself be highly available and consistent — if your dedup table goes down or returns stale answers, your idempotency guarantee evaporates. Second, dedup state grows without bound unless time-to-live (TTL) or compaction is designed in, and getting TTL wrong creates a different failure mode (late retry → double effect because the dedup record expired). Third, the operation often spans multiple systems — debit Ledger A, credit Ledger B, send a Kafka event, call a Stripe Application Programming Interface (API) — and idempotency must compose across each boundary independently or via an outbox + transactional pattern that turns the multi-system commit into a single-system one.

2. How to recognize it in the interview

Prompt phrases that signal this pattern

  • "The client may retry."
  • "Network is unreliable" / "what if the response is lost"
  • "How do you prevent double charges / double sends / duplicate orders?"
  • "What happens when the consumer crashes mid-batch?"
  • "How do you guarantee a webhook is processed once?"
  • "We use at-least-once delivery — how do you handle duplicates?"
  • "What if the same payment authorization is submitted twice?"
  • "Exactly once" anywhere in the prompt is a giant flashing sign.

Architectural smells (things in the diagram that demand this pattern)

  • A queue (Kafka, Simple Queue Service (SQS), RabbitMQ) feeding any consumer that mutates external state. Queues are at-least-once.
  • A public Hypertext Transfer Protocol (HTTP) endpoint that mutates state and has any retry policy upstream (client retry, load balancer retry, service mesh retry, mobile network reconnection).
  • A workflow with multiple write side-effects — a Saga, a payment flow, a "create user + send welcome email" combo.
  • A webhook receiver: webhooks are at-least-once by every major provider's contract.
  • Any cross-region / cross-system replication with retries.
  • Cron jobs or schedulers that can fire twice on failover.
  • Anywhere a developer says "we'll just check before we insert" — that is a Time-Of-Check-To-Time-Of-Use (TOCTOU) race waiting to happen.

Example interview prompts where this is core

  1. "Design Stripe's payment gateway." Every public Stripe write API takes an Idempotency-Key header. This is the canonical interview moment for this pattern. See design-the-stripe-payment-gateway.md.
  2. "Design a payment system" / "Design a delayed payment system." Authoritative ledger writes must be idempotent. See design-a-payment-system.md and design-a-delayed-payment-system.md.
  3. "Design a notification system." Sending the same push notification twice on a Kafka rebalance is the bug. See design-a-notification-system.md.
  4. "Design Kafka" or "Design a message queue." You will be asked about idempotent producers, transactional commits, and consumer offset management. See apache-kafka.md.
  5. "Design Uber" / "Design Uber Eats." Driver-accepts-trip and order-placement must be single-effect under client retry. See design-the-uber-backend.md and design-uber-eats.md.
  6. "Design a flash sale." Decrementing inventory under retry storms must not over-sell. See design-a-flash-sale-for-an-e-commerce-site.md.
  7. "Design Ticketmaster." Seat reservation = intent record + commit; retries cannot double-reserve. See design-ticketmaster.md.
  8. "Design a job scheduler." Cron firing twice on leader failover. See design-a-distributed-job-scheduler-like-cron.md.

3. 60-second spoken framing

The invariant is exactly-once effect, not exactly-once delivery. At the network level, the client cannot tell whether a lost response means the server processed the request or not, so it must retry. I make retries safe by giving every state-changing request a stable idempotency key — usually a client-supplied Universally Unique Identifier (UUID) — and storing the operation outcome keyed by that UUID in a deduplication table inside the same transaction that mutates business state. The first request wins, computes the answer, and persists it; every later request with the same key short-circuits and replays the stored response. For multi-system effects, I use the transactional outbox pattern: the business write and an outbox row commit atomically in one database transaction, and a separate publisher with at-least-once semantics pushes the outbox row to Kafka or to downstream services, which themselves dedup by the outbox event identifier. The metric I care about is the duplicate-suppression rate, and the failure mode I budget for is the dedup table's TTL being shorter than the maximum retry horizon.

Seven techniques. Each is a piece of the puzzle; staff-level answers combine three or four.

4.1 Idempotency key handling

The client generates a Universally Unique Identifier (UUID) v4 (or a deterministic hash of the request body if it controls retries) and sends it as a header on every retry of the same logical request. The server stores (idempotency_key, request_fingerprint, response_body, status_code, created_at) in a dedup table, keyed by idempotency_key with a unique index. The transaction that performs the business work and the transaction that inserts the dedup row are the same transaction.

# Single-system idempotent write. Postgres-flavored pseudocode.

import hashlib
import json
import uuid
from dataclasses import dataclass

@dataclass
class IdempotentResult:
    status_code: int
    body: dict
    replayed: bool  # True if served from dedup cache

def charge(
    db,
    user_id: str,
    amount_cents: int,
    idempotency_key: str,
    request_body: dict,
) -> IdempotentResult:
    # Fingerprint guards against accidental key reuse on a *different* request.
    fingerprint = hashlib.sha256(
        json.dumps(request_body, sort_keys=True).encode()
    ).hexdigest()

    with db.transaction() as tx:
        # Try to insert the dedup row first. The unique index on idempotency_key
        # is what makes this safe against concurrent retries.
        row = tx.execute(
            """
            INSERT INTO idempotency_keys
                (key, fingerprint, status, created_at)
            VALUES (%s, %s, 'IN_FLIGHT', NOW())
            ON CONFLICT (key) DO NOTHING
            RETURNING id
            """,
            (idempotency_key, fingerprint),
        ).fetchone()

        if row is None:
            # A concurrent or prior request already owns this key.
            existing = tx.execute(
                "SELECT fingerprint, status, response_body, status_code "
                "FROM idempotency_keys WHERE key = %s",
                (idempotency_key,),
            ).fetchone()

            if existing.fingerprint != fingerprint:
                # Same key, different request body = client bug; reject loudly.
                raise IdempotencyKeyConflict(idempotency_key)

            if existing.status == 'IN_FLIGHT':
                # Concurrent retry while the first is still running.
                # Either block (with timeout) or return 409 and let client back off.
                raise IdempotencyInFlight(idempotency_key)

            return IdempotentResult(
                status_code=existing.status_code,
                body=json.loads(existing.response_body),
                replayed=True,
            )

        # We are the first writer. Do the business work in the same transaction.
        charge_id = str(uuid.uuid4())
        tx.execute(
            "INSERT INTO charges (id, user_id, amount_cents, status) "
            "VALUES (%s, %s, %s, 'AUTHORIZED')",
            (charge_id, user_id, amount_cents),
        )

        response = {"charge_id": charge_id, "status": "AUTHORIZED"}

        tx.execute(
            """
            UPDATE idempotency_keys
            SET status = 'DONE',
                response_body = %s,
                status_code = 200,
                completed_at = NOW()
            WHERE key = %s
            """,
            (json.dumps(response), idempotency_key),
        )

        return IdempotentResult(status_code=200, body=response, replayed=False)

Three subtleties that separate junior from staff answers:

  1. Fingerprint check. If the same idempotency key arrives with a different request body, that is a client bug — never replay the cached response, because the client thinks it is sending a new request. Return 409 Conflict. Stripe does this; it catches an enormous class of bugs.
  2. IN_FLIGHT state. Two retries can arrive concurrently (mobile network blips create bursts). The first one inserts with status IN_FLIGHT; the second one sees it, must not start a second write, and must either wait or return a 409 telling the client to retry after a backoff. Note: the code above shows a single-transaction flow where the IN_FLIGHT state is never observable to a concurrent retry because the row is not visible until commit. That is fine for the pure-database case (no external call between insert and commit). For a flow that calls an external service mid-transaction — see Section 5 — the IN_FLIGHT row must be committed before the external call so concurrent retries can observe it; the business writes then happen in a second transaction. Pick the shape that matches whether there is an external side effect in the middle.
  3. Same atomicity domain. If the dedup record commits and the business row does not (or vice versa), you have lost the guarantee. For purely-database operations, both writes belong to the same database transaction (or are collapsed into one row, e.g., the charge row itself is keyed by idempotency_key). For operations with an external side effect, you split into two transactions and accept that the IN_FLIGHT → DONE transition is the atomicity boundary the system can offer — with the operational consequence that a crash mid-call leaves a stuck IN_FLIGHT row that a reaper must resolve.

4.2 Dedup table schema with TTL

# Postgres DDL for the dedup table. Annotated.

"""
CREATE TABLE idempotency_keys (
    key            TEXT PRIMARY KEY,                  -- client-supplied UUID
    fingerprint    TEXT NOT NULL,                     -- sha256 of request body
    status         TEXT NOT NULL,                     -- 'IN_FLIGHT' | 'DONE' | 'FAILED'
    status_code    INT,
    response_body  JSONB,
    user_id        TEXT,                              -- for sharding / partition pruning
    created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at   TIMESTAMPTZ,
    expires_at     TIMESTAMPTZ NOT NULL                -- TTL horizon
);

-- Partial index to quickly find expired rows for cleanup.
CREATE INDEX idempotency_keys_expires_at ON idempotency_keys (expires_at)
  WHERE status IN ('DONE', 'FAILED');

-- Range-partition by created_at so TRUNCATE PARTITION can drop old data fast.
-- Daily partitions × 30-day retention = 30 partitions.
"""

# Recommended TTL: longer than the maximum retry horizon for the API.
# - Mobile clients: 24-72h is typical because that bounds offline app-resume retry.
#   The 24h floor comes from "user backgrounds the app overnight"; pick the upper
#   end if your client SDK persists pending requests across reinstalls.
# - Server-to-server with a queue: max queue retention + safety margin
#   (e.g., 7-day Kafka retention -> TTL >= 8 days).
# - Webhooks delivered by external providers: match the provider's documented
#   retry budget. Stripe retries webhooks for up to 3 days, so 7 days is the
#   common floor for receivers. GitHub, Twilio, SNS all publish their own budgets.
#
# Failure modes of getting it wrong:
# - TTL too short -> late retry arrives after the dedup row was evicted, the
#   server treats it as a brand-new request, and the effect happens twice. This
#   is the *worse* failure mode because it is silent.
# - TTL too long -> table grows large (manageable with partitioning) and the
#   chance of a client reusing the same UUID for a logically different request
#   over a long window goes up slightly (caught by fingerprint check).
# Default to "too long" when in doubt; partitioning makes storage cheap, but
# double-effect on a missed dedup is a customer-facing incident.

4.3 Outbox pattern table schema

The outbox pattern is how you make "write to database AND publish to Kafka" atomic without a distributed transaction. You write a row to an outbox table in the same DB transaction as the business write. A separate publisher reads the outbox and pushes to Kafka with at-least-once semantics; downstream consumers dedup by event ID.

Be honest about what this does and does not solve. It does solve the dual-write race: business state and the intent-to-publish commit atomically, so a crash between the database commit and the Kafka send no longer causes the event to be silently lost. It does not give you exactly-once delivery downstream — the publisher is at-least-once by design (it can crash after kafka.send succeeds but before UPDATE outbox SET published_at), so downstream consumers must still dedup by event_id. The benefit is that the system collapses to "at-least-once with idempotent consumers," which is the only shape that actually works in practice.

"""
CREATE TABLE outbox (
    id              BIGSERIAL PRIMARY KEY,         -- monotonic publish order
    event_id        UUID NOT NULL UNIQUE,          -- dedup key for consumers
    aggregate_type  TEXT NOT NULL,                 -- e.g. 'charge', 'order'
    aggregate_id    TEXT NOT NULL,                 -- e.g. charge_id
    event_type      TEXT NOT NULL,                 -- e.g. 'charge.authorized'
    payload         JSONB NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    published_at    TIMESTAMPTZ                    -- NULL until published
);

CREATE INDEX outbox_unpublished ON outbox (id) WHERE published_at IS NULL;
"""

def authorize_and_emit(db, user_id, amount_cents, idempotency_key, request_body):
    with db.transaction() as tx:
        charge_id = insert_charge(tx, user_id, amount_cents, idempotency_key)

        # Same transaction! This is the entire point.
        tx.execute(
            """
            INSERT INTO outbox
                (event_id, aggregate_type, aggregate_id, event_type, payload)
            VALUES (%s, 'charge', %s, 'charge.authorized', %s)
            """,
            (
                str(uuid.uuid4()),
                charge_id,
                json.dumps({"charge_id": charge_id, "amount": amount_cents}),
            ),
        )
        return charge_id

# A separate worker process polls or tails the outbox:
def publish_outbox_loop(db, kafka):
    while True:
        with db.transaction() as tx:
            rows = tx.execute(
                """
                SELECT id, event_id, event_type, payload
                FROM outbox
                WHERE published_at IS NULL
                ORDER BY id
                LIMIT 100
                FOR UPDATE SKIP LOCKED
                """,
            ).fetchall()

            for row in rows:
                # Kafka idempotent producer keys by event_id so a producer-side
                # retry within the same producer session does not duplicate.
                kafka.send(
                    topic='charges',
                    key=row.event_id,
                    value=row.payload,
                )

            tx.execute(
                "UPDATE outbox SET published_at = NOW() "
                "WHERE id = ANY(%s)",
                ([r.id for r in rows],),
            )

The outbox publisher is at-least-once: if it sends to Kafka but crashes before updating published_at, the next iteration republishes the same event_id. Consumers must dedup by event_id. That is fine — the system is at-least-once with idempotent consumers, which is the standard answer.

A note on the polling-vs-CDC choice. The poll loop above is operationally simple but adds publish latency equal to the poll interval (commonly 100ms-1s) and constant database load proportional to publisher count. The Change Data Capture alternative — Debezium / Maxwell tailing the database's replication log — eliminates polling and reduces commit-to-publish latency to tens of milliseconds, but in exchange you take on a non-trivial piece of infrastructure: a stateful connector that owns a database replication slot, a schema-evolution story for the change stream, a separate operational surface for connector lag, and the failure mode of "the connector is the only thing publishing your events." Most systems start with polling and only graduate to CDC when publish-latency or database-load measurements force the move. The doc's recommendation is to default to polling at 100-500ms and reserve CDC for the cases where you have already measured the constraint.

4.4 Monotonic state machine

Many domains can use the business state itself as the dedup mechanism, eliminating the need for an external dedup table. A charge moves through INITIATED → AUTHORIZED → CAPTURED → SETTLED, and the transition table only allows forward moves.

# State transition is the dedup mechanism.

ALLOWED_TRANSITIONS = {
    'INITIATED':  {'AUTHORIZED', 'FAILED'},
    'AUTHORIZED': {'CAPTURED', 'VOIDED'},
    'CAPTURED':   {'SETTLED', 'REFUNDED'},
    'SETTLED':    {'REFUNDED'},
    'FAILED':     set(),
    'VOIDED':     set(),
    'REFUNDED':   set(),
}

def transition(db, charge_id: str, target_status: str) -> bool:
    """
    Returns True if the transition happened on this call, False if it was a no-op
    because the row was already past this state.
    """
    with db.transaction() as tx:
        # Conditional update: only transition if current status allows it.
        row = tx.execute(
            """
            UPDATE charges
            SET status = %s, updated_at = NOW()
            WHERE id = %s
              AND status IN %s
            RETURNING status
            """,
            (
                target_status,
                charge_id,
                tuple(s for s, allowed in ALLOWED_TRANSITIONS.items()
                      if target_status in allowed),
            ),
        ).fetchone()

        if row is None:
            # Either the charge does not exist, or it is already in (or past)
            # target_status, or it is in a state from which this target is illegal.
            current = tx.execute(
                "SELECT status FROM charges WHERE id = %s", (charge_id,)
            ).fetchone()
            if current is None:
                raise ChargeNotFound(charge_id)
            if current.status == target_status:
                return False  # idempotent no-op
            # Else this is a state-machine violation, not a retry. Raise.
            raise IllegalTransition(current.status, target_status)

        return True

This is cleaner than a dedup table when the operation has a clear lifecycle. The retry is implicitly safe because the second call sees status = AUTHORIZED already and the conditional UPDATE ... WHERE status IN ('INITIATED',) matches zero rows. No external dedup state is needed.

When this is the right tool: the domain entity naturally has a monotonic lifecycle (payment, order, reservation, deployment), the operation is a state transition on that entity, and the entity ID is a stable client-visible identifier the caller will supply on retry. When it is over-engineered: the operation is a one-shot mutation with no follow-up state (a "like" toggle, a profile field update, an analytics event), or the entity has no natural ordering. Forcing an artificial 4-state machine onto a single-write endpoint adds review surface without changing correctness — for those, a unique constraint on (user_id, target_id) or an idempotency-key dedup row is simpler.

4.5 Two-phase intent → commit

For workflows where you must allocate resources before performing the externally visible side-effect (e.g., reserve a seat, place a hold on a card, lock inventory), separate the intent from the commit. Both phases are individually idempotent; the commit phase is gated on the intent record's existence.

def reserve_seat(db, user_id: str, seat_id: str, idem_key: str) -> str:
    """Phase 1: create the intent. Idempotent on idem_key."""
    with db.transaction() as tx:
        row = tx.execute(
            """
            INSERT INTO seat_reservations (id, user_id, seat_id, status, expires_at)
            VALUES (%s, %s, %s, 'HELD', NOW() + INTERVAL '10 minutes')
            ON CONFLICT (id) DO NOTHING
            RETURNING id
            """,
            (idem_key, user_id, seat_id),
        ).fetchone()

        if row is None:
            existing = tx.execute(
                "SELECT user_id, seat_id, status FROM seat_reservations WHERE id = %s",
                (idem_key,),
            ).fetchone()
            assert existing.user_id == user_id and existing.seat_id == seat_id
            return idem_key  # replay

        # Also conditionally mark seat as held (or fail if already held).
        seat = tx.execute(
            "UPDATE seats SET held_by = %s WHERE id = %s AND held_by IS NULL "
            "RETURNING id",
            (idem_key, seat_id),
        ).fetchone()
        if seat is None:
            raise SeatUnavailable(seat_id)

        return idem_key

def confirm_reservation(db, idem_key: str) -> None:
    """Phase 2: commit. Idempotent on the reservation_id state machine."""
    with db.transaction() as tx:
        result = tx.execute(
            "UPDATE seat_reservations SET status = 'CONFIRMED' "
            "WHERE id = %s AND status = 'HELD' RETURNING id",
            (idem_key,),
        ).fetchone()
        if result is None:
            current = tx.execute(
                "SELECT status FROM seat_reservations WHERE id = %s", (idem_key,)
            ).fetchone()
            if current and current.status == 'CONFIRMED':
                return  # idempotent replay
            raise IllegalConfirmation(current)

If the intent is created but the user never confirms (or the network drops), the expires_at reaper releases the seat. If the confirm is retried, the state-machine guard short-circuits the second attempt.

4.6 Kafka idempotent producer + transactional commits

Kafka markets these features as "exactly-once semantics" (EOS). The honest framing is exactly-once within Kafka's boundary — i.e., a record produced as part of a committed transaction is visible to read_committed consumers exactly once, and the consume-process-produce loop does not double-write within the cluster. The moment a side effect leaves Kafka (a database row, an HTTP call, a push notification), EOS provides nothing; you are back to at-least-once-plus-idempotent-receiver. Treat EOS as a useful internal-pipeline tool, not as a system-wide guarantee.

The two cooperating features:

  • Idempotent producer (enable.idempotence=true): every producer is assigned a producer ID (PID) and every batch carries a monotonic sequence number. The broker rejects duplicate sequence numbers from the same PID, so a producer-side retry of a previously-acked batch does not duplicate the message within a single producer session.
  • Transactional producer (transactional.id=...): groups produces across multiple topics/partitions and consumer offset commits into a single atomic transaction. A consumer reading with isolation.level=read_committed only sees messages from committed transactions.

The pattern is "consume → process → produce → commit offset" as one transaction:

# Pseudocode for the Kafka exactly-once-semantics (EOS) consumer pattern.

producer = KafkaProducer(
    bootstrap_servers=...,
    enable_idempotence=True,
    transactional_id='payment-enricher-v1',
)
producer.init_transactions()

consumer = KafkaConsumer(
    'charges',
    isolation_level='read_committed',
    enable_auto_commit=False,  # we commit offsets inside the transaction
    group_id='payment-enricher',
)

for batch in consumer:
    producer.begin_transaction()
    try:
        for msg in batch:
            enriched = enrich(msg.value)
            producer.send('enriched-charges', key=msg.key, value=enriched)
        # Commit consumer offsets as part of the producer transaction.
        producer.send_offsets_to_transaction(
            offsets={tp: offset_for(tp, batch) for tp in batch.partitions()},
            consumer_group_id='payment-enricher',
        )
        producer.commit_transaction()
    except Exception:
        producer.abort_transaction()
        raise

Two caveats that matter at staff level:

  • This only applies inside Kafka. The moment you have a side-effect outside Kafka (writing to a database, calling an external API), Kafka transactions do not cover it. You are back to the outbox pattern or idempotent receivers.
  • Performance cost is non-trivial — transactional producers add coordinator overhead and increase end-to-end latency (commits batch every commit.interval, typically 100ms+). Reserve EOS for pipelines whose downstream is also inside Kafka and where dedup downstream is genuinely impractical. For most "Kafka in, side effect out" pipelines, at-least-once-plus-idempotent-consumer is both simpler and gives the same observable guarantee.

4.7 At-least-once-plus-idempotent-consumer

The most common production pattern, and usually the right interview answer when EOS is not specifically required.

def handle_event(db, event):
    """Idempotent consumer. Safe to call N times for the same event_id."""
    with db.transaction() as tx:
        row = tx.execute(
            "INSERT INTO processed_events (event_id, processed_at) "
            "VALUES (%s, NOW()) ON CONFLICT (event_id) DO NOTHING "
            "RETURNING event_id",
            (event.id,),
        ).fetchone()

        if row is None:
            return  # already processed

        apply_effect(tx, event)  # business work in same transaction

    # Commit Kafka offset *after* the DB transaction commits.
    # If we crash between the two, the next read replays the message; the
    # INSERT ... ON CONFLICT short-circuits and apply_effect is not re-run.

The dedup table here is the consumer's responsibility. It can be a dedicated processed_events table, or the business write itself can be conditional (e.g., INSERT ... ON CONFLICT (event_id) DO NOTHING directly on the target table).

5. Worked example: naive → broken → fixed

Scenario

A user taps "Pay $50" in a mobile app. The app calls POST /charges on the payment service. The payment service writes a row to its charges table and calls Stripe to authorize the card. The mobile network is flaky.

Naive design

Mobile app  ──POST /charges──►  PaymentService  ──auth──►  Stripe
                                       │
                                       ▼
                                  charges table

# THE BROKEN VERSION. Do not ship this.
def naive_charge(db, user_id, amount_cents):
    stripe_charge_id = stripe.authorize(user_id, amount_cents)
    db.execute(
        "INSERT INTO charges (id, user_id, amount_cents, stripe_id, status) "
        "VALUES (%s, %s, %s, %s, 'AUTHORIZED')",
        (uuid.uuid4(), user_id, amount_cents, stripe_charge_id),
    )
    return {"status": "ok"}

How it breaks

t=0  app POSTs /charges                 [request id R1]
t=1  PaymentService calls stripe.authorize
t=2  Stripe debits the card, returns ch_abc
t=3  PaymentService inserts charges row
t=4  PaymentService returns 200 to app
t=5  Mobile radio drops; app never sees the 200
t=6  App's HTTP client times out, retries [request id R2]
t=7  PaymentService calls stripe.authorize AGAIN
t=8  Stripe debits the card AGAIN, returns ch_def
t=9  PaymentService inserts a second charges row
t=10 App sees 200, shows "Payment successful"

The user is charged twice. Two charges rows exist with different IDs. The reconciliation team finds the bug three days later when the customer complains.

This is not a hypothetical. This is what happens to every payment system before it learns this pattern.

Fixed design: idempotency key + state machine + outbox

Mobile app  ──POST /charges──►  PaymentService  ──auth──►  Stripe
   │  Idempotency-Key: <uuid>            │
   │                                     │  same transaction:
   │                                     ▼
   │                          ┌──────────────────┐
   │                          │ idempotency_keys │
   │                          │ charges          │
   │                          │ outbox           │
   │                          └──────────────────┘
   │                                     │
   │                                     ▼
   │                              Outbox publisher
   │                                     │
   │                                     ▼
   │                                   Kafka  (event_id = outbox.event_id)

def charge(db, user_id, amount_cents, idem_key, request_body):
    fingerprint = sha256_json(request_body)

    with db.transaction() as tx:
        row = tx.execute(
            "INSERT INTO idempotency_keys (key, fingerprint, status) "
            "VALUES (%s, %s, 'IN_FLIGHT') ON CONFLICT (key) DO NOTHING RETURNING id",
            (idem_key, fingerprint),
        ).fetchone()

        if row is None:
            existing = tx.execute(
                "SELECT fingerprint, status, response_body FROM idempotency_keys "
                "WHERE key = %s", (idem_key,),
            ).fetchone()
            if existing.fingerprint != fingerprint:
                raise IdempotencyKeyConflict()
            if existing.status == 'IN_FLIGHT':
                raise IdempotencyInFlight()
            return json.loads(existing.response_body)

        # First-time path. We are inside the transaction; Stripe call is OUTSIDE.
        # We must commit the IN_FLIGHT row first so a concurrent retry sees it.
        tx.commit()

    # Outside the DB transaction now. Stripe's API is itself idempotent on its
    # own key; we use the same idem_key so Stripe dedups on its side too.
    try:
        stripe_id = stripe.authorize(
            user_id, amount_cents, idempotency_key=idem_key,
        )
    except StripeError as e:
        with db.transaction() as tx:
            tx.execute(
                "UPDATE idempotency_keys SET status='FAILED', "
                "response_body=%s, status_code=502 WHERE key=%s",
                (json.dumps({"error": str(e)}), idem_key),
            )
        raise

    # Stripe succeeded. Persist business state and dedup result atomically.
    with db.transaction() as tx:
        charge_id = str(uuid.uuid4())
        tx.execute(
            "INSERT INTO charges (id, user_id, amount_cents, stripe_id, status) "
            "VALUES (%s, %s, %s, %s, 'AUTHORIZED')",
            (charge_id, user_id, amount_cents, stripe_id),
        )
        tx.execute(
            "INSERT INTO outbox (event_id, aggregate_type, aggregate_id, "
            "event_type, payload) VALUES (%s, 'charge', %s, 'charge.authorized', %s)",
            (str(uuid.uuid4()), charge_id, json.dumps({"charge_id": charge_id})),
        )
        response = {"charge_id": charge_id, "status": "AUTHORIZED"}
        tx.execute(
            "UPDATE idempotency_keys SET status='DONE', "
            "response_body=%s, status_code=200 WHERE key=%s",
            (json.dumps(response), idem_key),
        )

    return response

Now replay the failure timeline:

t=0  app POSTs /charges, Idempotency-Key=K1
t=1  PaymentService INSERTs idempotency_keys(K1, IN_FLIGHT), commits
t=2  PaymentService calls stripe.authorize(idempotency_key=K1)
t=3  Stripe debits the card, returns ch_abc
t=4  PaymentService INSERTs charges + outbox + updates K1 to DONE, commits
t=5  PaymentService returns 200; mobile network drops
t=6  App retries: POST /charges, Idempotency-Key=K1
t=7  PaymentService tries INSERT idempotency_keys(K1, ...) → conflict
t=8  PaymentService reads existing row, sees status=DONE, returns cached 200
t=9  Card is charged exactly once. App is happy.

Three things to call out at the whiteboard:

  1. Stripe's own idempotency layer is invoked with the same key, so even if our service calls Stripe twice (e.g., we crash between t=2 and t=4), Stripe dedups for us.
  2. The IN_FLIGHT row is committed before the Stripe call so concurrent retries can see it and refuse to issue a second authorize.
  3. The outbox row is inserted in the same transaction as the charge row, so a single Kafka event represents this charge — downstream services that dedup by event_id cannot get the event twice from us either.

One asymmetry the whiteboard answer should name explicitly. The flow has two commit boundaries (IN_FLIGHT commit, then final DONE commit), and the window between them is where Stripe is debited. If the service crashes inside that window, the next retry sees an IN_FLIGHT row and returns 409 — but the card is already charged. This is a stuck IN_FLIGHT state, and the system needs a reaper that resolves it: scan rows in IN_FLIGHT older than some threshold (longer than the worst-case Stripe call), then for each stuck row, query Stripe with the same idempotency key to learn whether the authorization succeeded, and transition the row to DONE (with the recovered response) or FAILED. The point is that exactly-once-effect across an external boundary is not free — you trade "rare lost-then-retried request" for "rare stuck-then-reconciled request," and you must own the reconciliation path. Symmetric idempotency between your service and Stripe is what makes the reaper safe to retry.

6. Variants and how to choose

Technique Best when Cost / drawback Skip when
Client-supplied idempotency key + dedup table Public API; client controls retries; arbitrary request bodies Dedup table growth, TTL tuning, schema for storing responses Operation is naturally addressable by a business key
Natural business key as dedup Operation maps to a unique business entity (e.g., order ID, transfer ID) Requires client to materialize a stable ID before calling Operation is anonymous / unstructured
Monotonic state machine Lifecycle has clear forward-only transitions Designing the state machine; harder to retrofit Operation is one-shot with no follow-up state
Two-phase intent → commit Multi-step workflow; need to reserve resources before commit Two round trips; intent-leak reaper Single-step operation
Transactional outbox Need to publish events alongside DB writes; cannot use distributed transaction Outbox table growth, publisher lag, consumer dedup still required Single-system operation with no fan-out
Outbox + Change Data Capture (CDC) Outbox publisher should not poll; want low latency from commit to publish CDC pipeline (Debezium, Maxwell) operational burden Polling at 100ms is acceptable
Kafka idempotent producer + transactions All side effects live inside Kafka; need pipeline-internal EOS Producer / transaction coordinator overhead, complexity Single producer + simple consumer; downstream dedup is fine
At-least-once + idempotent consumer Default; works everywhere; composes with everything Consumer must maintain dedup state Never skip this in interviews; it is the staple

Decision flow

What is the cost of a duplicate effect?
    └── Negligible (analytics, "like" toggle) ─► Do nothing structural.
                                                  Rely on best-effort retries +
                                                  downstream reconciliation.
    └── Material (money, inventory, SMS)  ─► Continue.

Is there a natural stable business identifier
the client materializes before calling
(order_id, transfer_id)?
    └── Yes ─► Use it as the dedup key directly. No header needed.
    └── No  ─► Require a client-supplied Idempotency-Key header + dedup table.

Does the operation also publish events?
    └── Yes ─► Add an outbox row in the same DB transaction.

Does the operation have a clear lifecycle state?
    └── Yes ─► Make the state machine monotonic; use it as a second dedup
              layer on the business write.

Does the operation require pre-allocating a resource?
    └── Yes ─► Two-phase intent → commit. Reaper for stuck intents.

Are all side effects inside Kafka?
    └── Yes ─► Idempotent producer + transactional commits + read_committed.

Anywhere else there is at-least-once delivery (queue, webhook, retry mesh)?
    └── Default ─► Dedup at the consumer by event_id.

Key operational metrics

The percent ranges below are rules of thumb from public-API patterns, not universal numbers. The right baseline for your system is whatever the metric stabilizes at during a steady week; alert on change, not on an absolute threshold.

  • Duplicate-suppression rate: replayed_requests / total_requests. For mobile-client public APIs, low single digits is typical baseline. The shape that matters is the spike — a sudden multi-x jump means a network event, an upstream client bug retrying aggressively, or a regression that started returning non-2xx on the first attempt.
  • Idempotency-conflict rate: 409 conflict / total_requests. Should be near zero; a non-zero rate means a client is reusing keys across different requests, which is a bug.
  • Dedup-table miss rate after retry: requires a separate event log; alert if you can detect a request that came after a TTL eviction (rare but devastating).
  • Outbox lag p99: time from outbox.created_at to outbox.published_at. Target depends on polling interval — sub-second for a 100ms poller, single-digit-seconds for a 1s poller. Alert on the backlog of unpublished rows growing monotonically, which catches publisher-stuck conditions independent of latency target.
  • Processed-event dedup rate at consumer: events_short_circuited / events_received. Baseline equals the upstream redelivery rate; the alert is on deviation from baseline, not on hitting a specific number.

8. Combinations with other patterns

Idempotency is a horizontal concern that shows up in nearly every other pattern. Cross-references:

  • coordination-exactly-one-execution.md — Idempotency makes individual operations safe to retry; coordination/exactly-one-execution makes a job run on exactly one worker at a time (leader election, locks, leases). Together they cover both axes of "exactly once."
  • ordering-causal-consistency.md — If retries arrive out of order, the state machine pattern (forward-only transitions) makes ordering hazards safe. If you need strict ordering, you typically need both a sequencer (ordering pattern) and idempotent application (this pattern).
  • strong-consistency-under-contention.md — Idempotency keys depend on a strongly consistent dedup store. A dedup table on an eventually consistent KV store is a footgun.
  • fan-out-write-amplification.md — When a single write fans out to N downstream consumers, each consumer must be idempotent on the upstream event ID, or you multiply duplicates by N.
  • durability-at-scale.md — The outbox pattern's correctness depends on the outbox being on durable storage alongside the business data; otherwise a crash drops both.
  • backpressure-flow-control.md — Under load, retries multiply. Idempotency keeps correctness while backpressure keeps the system from collapsing.
  • real-time-aggregation.md — Aggregators that double-count duplicate events corrupt metrics. Either dedup before aggregating or use a HyperLogLog / probabilistic structure that tolerates duplicates.
  • hot-key-skew.md — A popular idempotency key (e.g., a global "default" key from a bad client) becomes a hot row in the dedup table. Validate key cardinality.
  • cache-stampede-read-amplification.md — Single-flight / dogpile-prevention is a read-side cousin of idempotency: many concurrent calls collapse to one execution and share the result. Same shape, different goal (latency, not correctness).

9. Likely interviewer probes

  1. "What if two clients send the same idempotency key concurrently for the same request?" — Answer: the unique index on the dedup table serializes them. The second one sees IN_FLIGHT and either waits (with timeout) or returns 409. Discuss the choice: blocking is simpler for clients, returning 409 is friendlier to server resources.

  2. "What's your TTL on the dedup table and why?" — Answer: longer than the maximum retry horizon for that surface. For mobile APIs, 24-72 hours covers offline buffering. For webhooks delivered by Stripe / GitHub / Twilio, look up the provider's documented retry policy (Stripe retries up to 3 days). Never set TTL to "1 hour because storage is expensive" — that is how you ship a double-charge bug. If storage is a concern, partition the table by date and drop old partitions.

  3. "Why not just use a primary key constraint on the business table directly?" — Answer: it works fine when (a) the client supplies or can derive a stable business ID before the call and (b) the database write is what gates the externally visible effect. For example, an INSERT INTO orders (order_id, ...) ON CONFLICT DO NOTHING is a perfectly adequate dedup mechanism if your retry path is "client retries with same order_id" and the order being inserted is what triggers downstream work via the outbox. The case where a separate dedup table earns its keep is when there is an external side effect before the database write — e.g., you call Stripe, then write the row. There, the primary-key constraint cannot prevent a duplicate Stripe call because the dedup point is before the row exists, which is why you need the IN_FLIGHT row committed up front. So: natural-key dedup for "DB write is the side effect"; separate idempotency-key dedup table for "external call before DB write." Don't reach for the heavier machinery if the lighter mechanism actually fits.

  4. "How does the outbox pattern compare to two-phase commit (2PC) across the database and Kafka?" — Answer: 2PC requires a transaction coordinator and is operationally hostile (blocking, prepare-and-hang failure modes, very few systems support it cleanly). Outbox makes the multi-system commit into a single-system commit by writing the event to the database first; the publisher then has at-least-once semantics with downstream consumer dedup. It is the standard answer at scale.

  5. "You said outbox publisher is at-least-once. How do downstream consumers dedup?" — Answer: each outbox row has a UUID event_id. Consumers maintain a processed_events table keyed by event_id and use INSERT ... ON CONFLICT DO NOTHING as the dedup gate. The Kafka offset is committed after the consumer's database transaction. This composes idempotent producer (within Kafka) with idempotent consumer (across the system boundary).

  6. "What if the dedup table goes down?" — Answer: it is a critical-path dependency. Co-locate it with the business data (same database, same shard) so its availability matches the business write. Do not put it in a separate microservice with its own failure mode. If you must shard separately, consider colocating by user_id so a single user's writes always hit the same shard pair.

  7. "How do you handle a request that times out mid-flight from the server's perspective — neither client retry nor server completion is observable to the other?" — Answer: the IN_FLIGHT row is exactly the mechanism. Server crashes after IN_FLIGHT but before DONE → client retries → sees IN_FLIGHT → returns 409 "in flight" → client backs off and retries → eventually a reaper transitions stale IN_FLIGHT rows to FAILED (or a side-channel like Stripe's idempotency window resolves it). Discuss the reaper interval and the choice between aggressive vs conservative cleanup.

10. Common traps → stronger moves

Trap Stronger move
"We'll just check if the row exists before inserting." Use the unique index and INSERT ... ON CONFLICT DO NOTHING. Read-then-write is a Time-Of-Check-To-Time-Of-Use (TOCTOU) race.
Dedup table without a TTL or partitioning strategy. Range-partition by created_at; drop partitions on a cron; document the retention horizon as part of the API contract.
Dedup state on a store whose durability or atomicity does not match the business write (e.g., Redis with no Append-Only File (AOF) persistence and no replication for a payment dedup table; a dedup KV in a different database that cannot share a transaction with the business write). Match the dedup store's durability and availability to the business write, and place it where it can atomically commit with the business state — typically the same relational database. Redis with AOF + replication can work for non-financial dedup, but for money-moving flows, prefer the same database as the business row.
Same idempotency key for "create" and "update" (client treats it as session-level). Document key scope as per-logical-request, not per-session. Reject duplicate-key/different-fingerprint loudly with 409.
"Stripe is idempotent on its side, so I don't need to be." Layered idempotency. Your service still needs to dedup because the client retries against you, not against Stripe. Forward the key to Stripe too for defense in depth.
Outbox publisher does SELECT ... FOR UPDATE without SKIP LOCKED. Use FOR UPDATE SKIP LOCKED so multiple publisher workers can scan in parallel without blocking on each other.
Commit Kafka offset before the consumer's DB transaction. Commit DB first, then Kafka offset. The crash-between window leaves a redelivery, which the dedup table absorbs. The reverse leaves a lost message.
"Exactly once" claimed as a system property without naming the seam where dedup happens. Always specify: dedup at the API key, the business key, the event ID, or the state machine — and say where the dedup state lives.
Ignoring the IN_FLIGHT state; only modeling DONE vs absent. Concurrent retries on the same key will both try to do the work. Model IN_FLIGHT explicitly and have a policy.
Treating webhooks as exactly-once because "the provider guarantees it." No provider does. Stripe, GitHub, Twilio, Slack, AWS Simple Notification Service (SNS) — all at-least-once. Always dedup by the provider's event ID.
Putting the dedup table on a different database from the business write. The atomicity of "insert dedup row AND insert business row" is gone. Co-locate.
Using auto-incrementing IDs as event IDs in the outbox. Use UUIDs. Auto-increment leaks ordering, doesn't survive partitioning, and is not collision-safe across shards.

11. Staff-level answer checklist

  • Name the cost of a duplicate effect before reaching for the toolkit. Cheap-to-reverse duplicates may not warrant the machinery.
  • Name the invariant explicitly: "exactly-once effect, not exactly-once delivery."
  • Identify where retries originate: client, load balancer, service mesh, queue, webhook redelivery, scheduler.
  • Choose the dedup seam: HTTP header, business key, event ID, state machine.
  • Co-locate dedup state with the business write so both commit atomically.
  • Model an IN_FLIGHT state for concurrent retries; choose blocking vs 409.
  • Use the outbox pattern for cross-system fan-out; do not invoke distributed transactions.
  • Forward idempotency keys to downstream providers that support them (Stripe, payment networks, internal services).
  • State the TTL and tie it to the maximum retry horizon you support.
  • Name the consumer-side dedup mechanism for any at-least-once stream you produce.
  • Specify the operational metric you would alert on (replayed-request rate, outbox lag, conflict rate).
  • Distinguish "legitimate retry" (return cached result) from "key reuse on different request" (return 409 — client bug).
  • Discuss what breaks if the dedup store fails: it is critical-path, treat it as such.
  • For Kafka pipelines: know when EOS is worth the cost vs at-least-once-plus-idempotent-consumer.
  • Cite at least one production failure mode (dedup TTL eviction, outbox lag, stuck IN_FLIGHT) and its mitigation.
  • Be explicit about Kafka "exactly-once": it is exactly-once within the Kafka boundary only; external side effects still need idempotent receivers.
