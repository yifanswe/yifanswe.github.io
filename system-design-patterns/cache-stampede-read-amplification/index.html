Cache Stampede / Read Amplification

2026/05/31  ·  System Design Patterns
Contents
Contents

A cache miss on a hot key sends every concurrent reader to the origin at the same moment, converting one missing entry into a self-inflicted denial-of-service against the database. The pattern is the toolkit for collapsing N concurrent misses into one origin fetch and for keeping hot entries warm so the miss never happens in the first place. See the Staff System Design Glossary for foundational terms (Time-To-Live (TTL), Content Delivery Network (CDN), cache-aside, etc.).

Core question docs where this pattern is the center of gravity: - Design a Distributed Cache like Redis - Design a Flash Sale for an E-commerce Site - Design Google News as a Global Aggregator - Design Twitter Timeline - Design YouTube - Design Ticketmaster - Design Typeahead Autocomplete

1. The fundamental hardness

Caches exist because the origin — usually a relational database, a search index, or an object store — cannot service the steady-state read rate at acceptable latency or cost. A modern web property can run two to four orders of magnitude more reads per second than its database can sustain. The whole architecture is therefore implicitly a bet that the hit rate stays high. The bet is fine in steady state. The bet breaks the moment a hot entry is missing from cache and the front door does not know how to slow down.

The breakage shape is the thundering herd: at time T the hot key expires (or evicts, or has never been written, or was just invalidated). Between T and T + Round-Trip-Time (RTT)-to-origin, every concurrent reader sees a miss and independently issues an origin fetch. The damage is the product of two numbers — concurrent missers × per-miss origin cost — and it only matters when that product saturates a downstream resource. Worked example: a key receiving 50,000 QPS at the cache, with a 30-millisecond origin query, sees ~1,500 concurrent missers during the gap; if the database connection pool is 200 and steady-state QPS to the database is 50, that single missing key drives 30x the design budget into the pool. The database saturates, latency climbs, the front door times out and retries, retries pile on top of misses, and the system enters cascade failure. The cache that was supposed to protect the database is now the trigger that destroys it.

The corollary is the boundary condition: for cold keys with low concurrent miss counts, naive cache-aside is fine. A key that sees 5 QPS and a 10-millisecond database query produces ~0.05 concurrent missers at expiry — there is no stampede to mitigate. Stampede mitigation earns its complexity on the long tail of genuinely hot keys with expensive backing queries. The staff-level move is often to say "naive cache-aside is the right answer for 99% of keys; here is the small set where I'd add single-flight."

The variants of the breakage are the same shape with different triggers. A cold start after a deploy or a node reboot starts with an empty cache and every first request is a miss — sustained for several minutes until the working set is warm. A mass eviction (the cache cluster OOMs, evicts an entire generation, or fails over and the new node has cold local memory) is functionally a cold start without a deploy. A flash crowd brings a key into hot range for the first time — the cache may have it, but the upstream CDN tier does not, and the origin shield gets every CDN edge's first request simultaneously. A synchronized expiry — every entry written at deploy time with the same TTL — turns into a synchronized miss storm at T + TTL.

The deeper invariant the pattern protects is bounded origin work per unit of cache miss. Naively, one miss equals one origin fetch — but only if you collapse concurrent misses for the same key. The whole toolkit below is about making that statement true under contention, and about pushing the miss probability down so the toolkit rarely has to fire.

2. How to recognize it in the interview

Prompt phrases that signal this pattern

  • "Cache the result."
  • "Serve from cache, fall back to database."
  • "Read-heavy."
  • "Reduce database load."
  • "Viral content / trending / breaking news."
  • "Black Friday / Super Bowl / launch day traffic."
  • "Cold start after deploy."
  • "Hot row / hot key / hot URL."
  • "Edge cache / CDN / origin shield."
  • "TTL of N seconds."
  • "Eviction policy / Least Recently Used (LRU) / Least Frequently Used (LFU)."

Architectural smells in stated requirements

  • A read-to-write ratio above roughly 50:1, so caching is the obvious cost lever.
  • A single entity (post, video, product, ticker) is read by a large fraction of all readers — so a miss on that key has a herd waiting.
  • Origin latency is much higher than cache latency (database query is 20 ms, cache is 0.5 ms — the herd window is 40x larger than the cache hit window).
  • The product has a "going viral" or "trending" affordance — the workload distribution is heavy-tailed by definition.
  • Multiple cache tiers are mentioned (CDN, edge, application-local, distributed cache) — every boundary is a potential stampede surface.
  • The interviewer asks about "what happens when the cache fails" — they want to hear graceful degradation, not "we fall back to the database."
  • The interviewer mentions invalidation, deploys, or warmup — they want to hear about cold start handling.

Example "design X" prompts where this is the core hardness

3. 60-second spoken framing

The risk I am designing around is read amplification — one cache miss on a hot key turning into a flood of identical origin queries. But before reaching for techniques, I would scope the problem: for the long tail of keys with modest QPS and cheap backing queries, naive cache-aside is the right answer. The mitigations earn their complexity only on genuinely hot keys with expensive origin queries. For those, I'd defend in depth. First, in-process single-flight: a per-key lock so only one fetcher per application instance hits the origin and the rest wait — this caps origin QPS at N-application-instances per miss, not at concurrent-readers. Second, probabilistic early expiration so hot keys refresh asynchronously before they expire, not at the moment they expire — concretely justified when QPS-on-hot-key times origin-query-cost would saturate a downstream pool. Third, stale-while-revalidate so a miss still serves the previous value to clients while we refetch in the background — justified by a specific staleness SLO the product can name. Fourth, tiered caches — Content Delivery Network, edge, application-local Least Recently Used cache, distributed cache, origin — so each tier shields the next from herds. I'd only reach for distributed single-flight (Redis lock) if N-instances of in-process single-flight still overruns the origin. Operationally, I would alert on origin queries-per-second relative to baseline, cache hit rate by key class, and the fanout factor — concurrent waiters per single-flight slot. The thing I never want to see is a cache cluster failover triggering a mass-eviction cold start with no rate limit between the application and the database.

4. Canonical toolkit

4.1 Single-flight / request coalescing

What it is. When N concurrent requests miss the same key, only one of them — the leader — actually calls the origin. The other N-1 — the followers — block on a per-key future and receive the same response. The N origin fetches collapse to one.

T=0   100 readers all miss key "post:42"
        │
        ▼
   try acquire lock("post:42")
        │
        ├── reader 0 acquires lock ──► fetch from origin ──► populate cache ──► return
        │
        └── readers 1..99 wait on future ──► get same value when ready
                                              (zero origin queries from them)

Python sketch (asyncio). A per-key asyncio.Lock plus a per-key Future to broadcast the result. The lock keeps fetchers serialized; the future lets followers piggyback without re-entering the lock.

import asyncio
from typing import Awaitable, Callable, TypeVar

V = TypeVar("V")


class SingleFlight:
    """Coalesce concurrent calls for the same key into one origin fetch."""

    def __init__(self) -> None:
        # key -> Future holding the in-flight result. Presence of the entry
        # is itself the "someone is already fetching" signal.
        self._inflight: dict[str, asyncio.Future[V]] = {}
        # A coarse lock around the dict; per-key locking is via the future.
        self._mu = asyncio.Lock()

    async def do(self, key: str, fetch: Callable[[], Awaitable[V]]) -> V:
        async with self._mu:
            existing = self._inflight.get(key)
            if existing is not None:
                fut = existing
                is_leader = False
            else:
                fut = asyncio.get_running_loop().create_future()
                self._inflight[key] = fut
                is_leader = True

        if not is_leader:
            return await fut  # follower: just wait

        # Leader: actually do the work, then publish the result.
        try:
            value = await fetch()
            fut.set_result(value)
            return value
        except Exception as exc:
            fut.set_exception(exc)
            raise
        finally:
            async with self._mu:
                # Remove the inflight entry so the next miss starts a new fetch.
                self._inflight.pop(key, None)

Distributed variant — and the asymmetry to call out. A per-key asyncio.Lock only coalesces requests on one process. The in-process version is cheap: a dict and a lock, microseconds of overhead per call. The cross-process version is qualitatively different: it requires a distributed lock — typically a short-lived Redis SET key:lock NX PX 5000 with a unique token — which adds an extra cache round-trip on every miss path, plus a correctness story for lock TTL versus origin p99 latency, lock owner identification, and what happens when the lock holder dies. Followers either spin-poll the cache key (acceptable if cache RTT is single-digit ms) or subscribe to a pub/sub channel.

The honest framing: if you have 200 application servers and the same key misses on all of them simultaneously, in-process single-flight gives you 200 origin fetches (one per box), not 1. That may or may not be acceptable. If the origin can take 200 QPS bursts comfortably (cache miss on the median hot key, with a moderate-cost query), stop there — distributed coordination is unjustified complexity. Reach for the Redis lock only when 200 concurrent origin fetches would actually hurt, which usually means the backing query is expensive (hundreds of ms) or the origin is already near capacity.

Subtleties to call out in interview. - The leader can die. Set a finite lock TTL so a crashed leader unblocks the followers — but pick TTL > origin p99 latency or you'll get duplicate fetches. - The leader's fetch can be slow. Followers must have their own deadline; if the leader is taking too long, a follower should give up (and ideally serve stale) rather than blocking the user request indefinitely. - Single-flight does nothing for different keys missing concurrently — it only collapses identical key misses. Hot-key skew is the input; see hot-key-skew.md.

4.2 Probabilistic early expiration (XFetch / Vattani et al.)

What it is. A reader that finds a value in cache will, with probability that grows as the TTL approaches expiry, voluntarily refresh the value early instead of just returning it. By the time the entry actually expires, it has almost certainly been refreshed already, so there is no synchronized miss event.

The math (from Vattani, Chierichetti, Lowenstein, 2015 — "Optimal Probabilistic Cache Stampede Prevention"). Each cached entry stores the time it was last computed and the time the computation took (delta). On read:

should_refresh_early  iff   now - delta * beta * ln(rand()) >= expiry

where beta is a tuning constant (typically beta = 1), rand() is uniform in (0, 1), and delta is the historical fetch latency for this key. Because ln(rand()) is negative and unbounded below, the inequality is rarely true when now is far from expiry, and becomes true with rising probability as now approaches expiry. The expensive-to-fetch entries (large delta) refresh earlier, which is exactly the right policy — they cost the most to recompute on a hard miss.

import math
import random
import time
from dataclasses import dataclass
from typing import Awaitable, Callable, Optional


@dataclass
class CacheEntry:
    value: object
    delta: float       # how long the last fetch took, in seconds
    expiry: float      # absolute unix time at which this becomes hard-stale


async def get_with_early_expiration(
    cache,
    key: str,
    fetch: Callable[[], Awaitable[object]],
    ttl: float,
    beta: float = 1.0,
) -> object:
    entry: Optional[CacheEntry] = await cache.get(key)
    now = time.time()

    if entry is None or now >= entry.expiry:
        # Hard miss path. Combine with single-flight to avoid a herd here.
        return await _refresh(cache, key, fetch, ttl)

    # Early-expiration test. As now approaches entry.expiry, the chance of
    # this test firing grows; for an entry that just refreshed, it almost
    # never fires.
    if now - entry.delta * beta * math.log(random.random()) >= entry.expiry:
        # Refresh in background; serve the current (still-valid) value now.
        asyncio.create_task(_refresh(cache, key, fetch, ttl))

    return entry.value


async def _refresh(cache, key, fetch, ttl):
    started = time.time()
    value = await fetch()
    delta = time.time() - started
    await cache.set(
        key,
        CacheEntry(value=value, delta=delta, expiry=time.time() + ttl),
        ttl=ttl,
    )
    return value

Why this is the staff move. Early expiration is decoupled from the herd: each reader independently rolls the dice. There is no synchronization needed across processes or data centers. The aggregate effect is that the probability of any single reader triggering a hard miss approaches zero, while the steady-state refresh rate is one refresh per TTL per key. Combine with single-flight to guarantee that the rare hard-miss case still only fires one fetch.

4.3 Stale-while-revalidate (SWR) and stale-if-error

What it is. When a cache entry's TTL expires, do not delete it. Mark it stale. On read of a stale entry, serve the stale value to the user immediately, and trigger a background refresh. The user never blocks on the origin.

read at T=0 (entry fresh):    cache HIT, return fresh value.
read at T=TTL+1 (stale):      cache HIT (stale), return stale value
                              + kick off background refresh.
read at T=TTL+2 (refresh in flight): cache HIT (stale), return stale
                              + DO NOT kick off another refresh
                              (single-flight on the refresh job).
read at T=TTL+5 (refresh done): cache HIT (fresh again), return fresh.

Stale-if-error is the cousin: if the background refresh fails (origin down, timeout), keep serving the stale value past its nominal expiry rather than returning an error. The freshness budget you accept is the price of staying available.

Python sketch (decorator form).

import asyncio
import time
from functools import wraps


def swr_cache(cache, ttl: float, stale_extension: float):
    """
    Stale-while-revalidate decorator.

    - For `ttl` seconds, the cached value is fresh.
    - For an additional `stale_extension` seconds, the cached value is stale
      but still served while a background refresh runs.
    - After ttl + stale_extension, it is hard-expired.
    """
    inflight: dict[str, asyncio.Task] = {}

    def decorator(fetch):
        @wraps(fetch)
        async def wrapper(key: str, *args, **kwargs):
            entry = await cache.get(key)
            now = time.time()

            if entry is None:
                # Hard miss: must block on origin (still coalesce via single-flight).
                value = await fetch(key, *args, **kwargs)
                await cache.set(key, {"value": value, "stored_at": now}, ttl=ttl + stale_extension)
                return value

            age = now - entry["stored_at"]

            if age < ttl:
                # Fresh.
                return entry["value"]

            # Stale but still within stale_extension. Serve stale, refresh async.
            if key not in inflight:
                async def refresh():
                    try:
                        v = await fetch(key, *args, **kwargs)
                        await cache.set(key, {"value": v, "stored_at": time.time()},
                                        ttl=ttl + stale_extension)
                    finally:
                        inflight.pop(key, None)

                inflight[key] = asyncio.create_task(refresh())

            return entry["value"]
        return wrapper
    return decorator

When to use. Anywhere the value is recomputable but a few seconds of staleness is acceptable. News articles, social media post bodies, product detail pages, recommendation feeds, search result pages — all of these are SWR-friendly. The pattern is built into Hypertext Transfer Protocol (HTTP) Cache-Control directives (stale-while-revalidate, stale-if-error) and is supported natively by CDNs like Fastly and Cloudflare.

When to avoid. Inventory counts, balances, anything where stale-by-five-seconds is a correctness violation. Use strong-consistency-under-contention.md instead.

4.4 Background refresh / async warming

What it is. Decouple cache population from user requests entirely. A cron job, a Kafka consumer, or a long-running worker recomputes the hot keys on a schedule and writes them to cache. The user-facing read path is supposed to always hit; a miss is an alert-worthy event.

When to use. - The set of hot keys is known and bounded — top 1,000 trending topics, the seat-map for the next 50 events going on sale, the metadata for the homepage carousel. - The recomputation is expensive (seconds per key — too slow to do inline on a miss). - The freshness requirement is loose enough that the schedule is acceptable (every 10 seconds for trending, every 60 seconds for a leaderboard).

Architectural picture.

            ┌──────────────────────────┐
            │  Hot-key registry        │
            │  (Redis sorted set       │
            │   of top-N keys)         │
            └────────────┬─────────────┘
                         │ produces refresh jobs
                         ▼
            ┌──────────────────────────┐
            │  Refresh workers         │
            │  (consume Kafka /        │
            │   cron schedule)         │
            └────────────┬─────────────┘
                         │ writes
                         ▼
            ┌──────────────────────────┐
            │  Distributed cache       │
            └────────────┬─────────────┘
                         │
            ┌────────────▼─────────────┐
            │  User request path:      │
            │  cache MUST hit          │
            └──────────────────────────┘

Risk. Background refresh on its own is brittle. If the hot-key registry is wrong (some new key became hot but isn't on the list), the user path hits a hard miss and you're back to single-flight. If the refresh worker is slow or fails, the cache goes stale silently. Pair with SWR or single-flight as a fallback.

4.5 Tiered caching

What it is. Multiple caches stacked between the user and the origin. Each tier filters traffic from the next.

user ─► CDN edge (geo-near)
          │ miss
          ▼
       CDN origin shield (one POP per region)
          │ miss
          ▼
       Application local Least Recently Used cache (per process)
          │ miss
          ▼
       Distributed cache (Memcached / Redis cluster)
          │ miss
          ▼
       Origin database

Why it works. Each tier is at most one cache miss away from being a stampede surface, but each tier also collapses the herd of the tier above it. CDN edges in 200 Points of Presence (POPs) talking to one origin shield mean the database sees at most one fetch per region per miss, not 200. The application-local LRU cache further collapses within a single server: 100 in-flight requests on one process share the local hit.

Key staff-level point. The tiers must each have their own herd protection. It is not enough to single-flight the application tier. The CDN must have origin shield (also called "tiered cache" by Cloudflare, "shield" by Fastly) to collapse cross-edge concurrent misses. The distributed cache must have replication of hot keys so the request rate to any one Redis shard is bounded.

The tax tiering imposes. Each tier is its own TTL, its own invalidation story, and its own consistency frontier. A correction to an article body must propagate through CDN edge → CDN origin shield → application-local LRU → distributed cache → origin; if any tier's TTL is long and the invalidation channel doesn't reach it, that tier serves stale data. In practice, this means you pay for: explicit purge APIs at every tier (CDN purge, app-local cache busting, Redis DEL), or aligned TTLs that bound the worst-case staleness window, or per-version cache keys (key includes a content hash so a new version creates a new key and old versions naturally age out). Adding a tier is not free — it's a layer of operational surface, not just a layer of performance.

4.6 Negative caching

What it is. Cache the "not found" answer too. If a key doesn't exist in the origin, do not let every miss re-query the origin to confirm it still doesn't exist. Store a tombstone with a short TTL.

When it earns its place. Negative caching is cheap (it's just an extra SET with a sentinel value), so the cost-benefit bar is low — but it's not universally needed. The keys that benefit are those where misses are expensive (a full-text search that finds nothing still scans), or where the miss volume is driven adversarially (scrapers walking /user/<random>, attackers probing /api/v1/<garbage>). For a typical primary-key lookup on an indexed column, the "not found" path is already fast — adding a tombstone is fine but rarely the lever that saves you. The 404 path is also where a herd can be invisible — your hit rate looks fine but your database is being hammered by lookups that return zero rows, so it's worth instrumenting miss-by-result-type, not just miss rate.

NOT_FOUND_SENTINEL = "__NF__"

async def get_user(user_id: int):
    cached = await redis.get(f"user:{user_id}")
    if cached == NOT_FOUND_SENTINEL:
        return None
    if cached is not None:
        return deserialize(cached)

    user = await db.fetch_user(user_id)
    if user is None:
        # Short TTL — the user might be created soon.
        await redis.set(f"user:{user_id}", NOT_FOUND_SENTINEL, ex=60)
        return None

    await redis.set(f"user:{user_id}", serialize(user), ex=3600)
    return user

Subtleties. Short TTL on tombstones — if the entity is later created, you want the cache to learn about it within a minute. Sentinel value must be a sentinel, not an empty string, so you can distinguish "we know it does not exist" from "we have not asked yet."

4.7 Lease tokens (Memcached-style)

What it is. A protocol-level primitive that some caches expose to make single-flight a property of the cache itself, not of every client library. On a miss, the cache hands the first client a lease token — a permission slip to recompute. Subsequent missers either wait for the value or are told "someone else has the lease, back off and retry in N ms."

Memcached's gets/cas with leases (per Facebook's mcrouter paper, "Scaling Memcache at Facebook," NSDI 2013) does exactly this. The lease prevents N missers from all writing the same key, and the wait protocol prevents N missers from all calling the origin.

Why staff candidates know this. It's the canonical citation for "we built single-flight into the cache because client-side coalescing was too easy to get wrong at scale."

5. Worked example: naive → broken → fixed

Setup

A news site has just published a story that goes viral. The story body is cached in Redis as article:12345 with TTL 5 minutes. At noon, the application tier is serving 5,000 reads per second for this article (the upstream CDN absorbs the bulk of the 50,000 reads per second of public traffic — at 50K QPS on a single key, the first fix is to make sure the CDN and edge tier are actually hit; the app cache is a backstop, not the front door). The story was first cached at 11:55 AM. The query that builds the article body is a join across articles, authors, and topics — about 30 milliseconds at p50, 200 milliseconds at p99 under normal load.

(If the CDN tier isn't there — say this is a logged-in personalized view bypassing the CDN — the same analysis applies with the original numbers; the magnitudes scale but the techniques are the same.)

Naive design: vanilla cache-aside

async def get_article(article_id: int):
    cached = await redis.get(f"article:{article_id}")
    if cached is not None:
        return deserialize(cached)
    article = await db.query_article(article_id)
    await redis.set(f"article:{article_id}", serialize(article), ex=300)
    return article

What goes wrong

At 12:00:00 PM, the TTL of article:12345 fires. The next read (12:00:00.001) sees a miss, queries the database (30 ms), and writes the cache. During those 30 ms, 5,000 reads per second × 0.030 seconds = 150 concurrent readers also see the cache as empty and each issue their own database query.

The article-fetch query has been carefully tuned to handle the steady-state ~50 QPS to the database (mostly long-tail articles, not the hot one). 150 simultaneous fetches against a 200-connection pool already crowd out other queries, blow through the connection pool budget when combined with normal load, queue up on lock contention in the query planner, and the p99 of that query goes from 200 ms to 5 seconds. The application's HTTP timeout fires; clients retry; retries arrive at the load balancer; the load balancer pushes more concurrent requests through the application; the application opens more database connections; the database refuses connections; the application returns 503. The whole site goes down because one cache key expired.

(Note the calibration: 150 concurrent misses on a 30 ms query is the real stampede here. If the query were 5 ms instead of 30, you'd get 25 concurrent fetchers — annoying but probably survivable. The reason this stampede actually breaks the system is the multiplication: hot key × expensive query × shared pool with other traffic.)

It gets worse the next day. The operator, blaming the TTL, raises it to 30 minutes. Now the cache hits 30 minutes between misses, the stampede effect is the same when the miss does happen, and now the staleness window is also 30 minutes long — the breaking-news team complains that corrections take half an hour to propagate.

Fixed design: probabilistic early expiration + single-flight + SWR

We layer three of the toolkit pieces.

import asyncio
import math
import random
import time

from typing import Optional

SINGLE_FLIGHT = SingleFlight()  # from section 4.1


async def get_article(article_id: int):
    key = f"article:{article_id}"
    entry: Optional[dict] = await redis.get_json(key)
    now = time.time()

    if entry is None:
        # Hard miss. In-process single-flight only. With N application instances
        # the origin can still see up to N concurrent fetches on a hard miss;
        # that's intentional — we accept N origin queries to avoid the extra
        # round-trip and complexity of a distributed lock. Add a Redis lock here
        # only if N origin queries is still too many.
        return await SINGLE_FLIGHT.do(key, lambda: _fetch_and_store(article_id, key))

    age = now - entry["stored_at"]
    ttl = 300                # 5 minutes "fresh"
    stale_window = 60        # extra minute of "stale but servable"

    if age >= ttl + stale_window:
        # Hard stale, treat as miss.
        return await SINGLE_FLIGHT.do(key, lambda: _fetch_and_store(article_id, key))

    if age >= ttl:
        # Stale-while-revalidate: serve stale, refresh in background.
        asyncio.create_task(
            SINGLE_FLIGHT.do(key, lambda: _fetch_and_store(article_id, key))
        )
        return entry["value"]

    # Fresh. Probabilistic early expiration: refresh early with rising
    # probability as we approach the TTL.
    delta = entry["delta"]
    expiry = entry["stored_at"] + ttl
    if now - delta * math.log(random.random()) >= expiry:
        asyncio.create_task(
            SINGLE_FLIGHT.do(key, lambda: _fetch_and_store(article_id, key))
        )

    return entry["value"]


async def _fetch_and_store(article_id: int, key: str):
    started = time.time()
    article = await db.query_article(article_id)
    delta = time.time() - started
    await redis.set_json(
        key,
        {"value": article, "stored_at": time.time(), "delta": delta},
        ex=300 + 60,
    )
    return article

What changes

  • At 11:59:30, some lucky reader rolls a small random number and triggers an early refresh. The cache is rewritten with a fresh stored_at. No user blocks; no herd.
  • At 12:00:00, the previous "expiry" comes and goes, but most likely the value has already been refreshed via early expiration. If it hasn't, the SWR window kicks in: the next reader sees a stale entry, returns it immediately, and triggers one background refresh — also coalesced by single-flight.
  • The background refresh hits the database exactly once per application instance (since the SingleFlight shown is in-process). With N application servers, you get up to N concurrent fetches, not 1 — which is fine if N is small (say 20) and the query is 30 ms. If you need cross-process coalescing too, layer the Redis lock from Section 4.1 on top of the in-process single-flight.
  • 149 other concurrent readers on the leader's instance see the stale entry and serve happily; on the other N-1 instances, their followers behave the same way.
  • The database sees roughly N queries per refresh cycle (one per application instance), not 150. For N=20, that's a 7.5x reduction — enough to pull the database out of the cascade-failure zone.

If we further want to be paranoid — say this is a Super Bowl-grade event and one query per refresh is still too many to risk — we add a background warmer running every 30 seconds that touches the top-100 articles, so the cache is never close to expiry in steady state. Section 4.4.

What we paid

  • Code complexity went up — we now have three interacting mechanisms (PEE, SWR, single-flight) plus a worker.
  • Stale window: readers can see up to 60 seconds of staleness. The product team needs to be OK with that.
  • Bookkeeping: we store stored_at and delta per entry, which inflates the cache value slightly.

The tradeoff is well-bought. The system survives viral events.

6. Variants and how to choose

Technique When it's the right pick When it's wrong Hidden cost
Single-flight (per-key lock) Concurrent misses on the same key need to collapse. Default-on for any read-through cache with hot keys. Misses on different keys (you have hot-key skew, not stampede). Use hot-key-skew.md. Followers' tail latency grows: they wait the leader's full origin latency. Need follower-side deadlines.
Probabilistic early expiration TTL is the trigger; you want to spread the refresh load randomly across time and clients. Cold start (no entry to refresh early). Eviction-driven misses (cache forgot the entry). Each entry must store delta and stored_at; not all cache schemas support that without wrapping.
Stale-while-revalidate Product tolerates a few seconds to a minute of staleness; user-blocking on origin is unacceptable. Inventory, balances, anything where stale equals wrong. The "freshness budget" needs to be communicated to product owners and to observability so a refresh worker stuck at 10 minutes raises an alarm.
Background refresh / warming Hot key set is small, known, and expensive to recompute. Top trending topics, homepage modules, leaderboards. Hot-key set is unbounded or unpredictable (long-tail content goes viral). Worker fleet to operate; brittle if the hot-key registry diverges from reality.
Tiered caches Read traffic is geographically distributed; multiple cache surfaces are natural (CDN + edge + app + distributed cache). Single-region small system; tiers add latency and ops cost. Each tier has its own invalidation story; propagating a busted entry through four tiers is hard.
Negative caching Lookup pattern includes many "does this exist" queries; 404 path is hot. Per-user ACL checks where negative answers leak permission state. Coherence: if the entity is later created, cache must learn quickly.
Lease tokens (cache-level) You control the cache server (Memcached) or use one that exposes leases natively. Most off-the-shelf Redis deployments don't expose this. Application code must understand the lease protocol; harder to mock in tests.
Graceful degradation (stale-if-error) The right tail of cache misses is dominated by origin outages, not just expiry. Strong-consistency invariants. Need a hard upper bound on how stale you'll serve before failing the request.

A decision tree worth walking the interviewer through:

Is the workload read-heavy with a hot key set?
├─ No  ── caching is not the dominant lever; focus elsewhere.
└─ Yes ── you need stampede protection.
          │
          Is the hot key set bounded and known?
          ├─ Yes ── background refresh + warming, with single-flight as fallback.
          └─ No  ── probabilistic early expiration + single-flight + SWR.
                    │
                    Is the product OK with stale reads?
                    ├─ Yes ── add SWR for graceful degradation.
                    └─ No  ── single-flight only; accept tail latency on miss.
                             Then check if the consistency requirement is real;
                             consider [strong-consistency-under-contention.md].

7. Failure modes and operations

A note on categorization before reading the table: not all of these are "cache stampede" problems in the strict design-pattern sense. They are the family of incidents a cache-fronted system experiences, but their mitigations come from different toolboxes:

  • Design-pattern problems (this doc): cold start, synchronized expiry storm, slow leader in single-flight, stampede on first-time-viral key. Single-flight, jitter, SWR, and early expiration are the answers.
  • Operational problems: mass eviction, cache failover triggers cold path. The fix is right-sizing, replica warming, and incident playbooks — not a design change.
  • Hot-key-skew problems: hot-key amplification (one key serves >10% of traffic). Cross-ref hot-key-skew.md; replication and load-spreading, not single-flight, is the answer.
  • Security problems: cache poisoning. Validation, schema checks, and emergency invalidation — this is a correctness/trust problem, not a load problem.
  • CDN problems: origin shield miss on viral content. CDN configuration (tiered cache, shielding), not application-layer single-flight.

They're listed together because they all show up on the same on-call dashboard. Knowing which one you have is half the fix.

Failure mode What you observe Metric Mitigation
Cold start after deploy Cache hit rate collapses from 95 percent to near zero; origin QPS spikes 20x; p99 latency climbs. cache_hit_rate by tier, origin_qps. Pre-warm hot keys before serving traffic; staged rollout with traffic shifting; persistent local cache that survives restart.
Mass eviction Sudden hit rate cliff without a deploy; usually correlated with memory pressure on the cache cluster. cache_evictions_per_second, cache_memory_usage. Right-size the cache; partition hot tenants; admission control on what gets cached.
Hot-key amplification One key serves >10 percent of cache traffic; one Redis shard pegs at 100 percent CPU while others sit idle. per-key QPS histogram, per-shard CPU. Replicate hot keys to multiple shards; client-side load-balancing across replicas; see hot-key-skew.md.
Cache poisoning One reader writes a bad value (a stack trace, an error object, a zero count); every subsequent reader serves it. error rate on cache hits, schema validation failures. Validate values before caching; never cache exceptions or partial responses; use short TTL on first-write; have an emergency invalidate-by-pattern operation.
Synchronized expiry storm Every deploy writes entries with identical TTL; T + TTL minutes after each deploy, mass miss event. distribution of time_until_expiry across cache keys. Jitter the TTL (ttl = base + uniform(-jitter, jitter)); avoid simultaneous bulk warming.
Stampede on first-time-viral key A long-tail key that was never hot suddenly is; not yet in cache; herd hits origin. origin QPS by key, sudden new entries to the per-key-QPS histogram. Single-flight as defense in depth; rate limit per-key origin queries; trending-detection pipeline that warms keys preemptively.
Slow leader in single-flight One leader is slow (network blip to origin); 999 followers all blocked behind it; tail latency degrades. followers_per_leader histogram, single_flight_wait_p99. Per-follower deadline; have followers serve stale if the leader exceeds budget; circuit-break and let one follower take over.
Cache failover triggers cold path Redis primary fails over; new primary has cold local memory; backlog of writes; clients see misses; origin gets the herd. Redis failover_count, cache hit rate around failover events. Replica with warm memory; rate limit the origin from the application layer; circuit breaker that sheds load during failover.
Origin shield miss on viral content First request to CDN per region; shield doesn't have it; herd of all regions hits origin. CDN origin shield hit rate, origin QPS during viral events. Cloudflare "tiered cache" or Fastly "shielding" enabled; origin-side single-flight as backstop.

Metrics dashboard a staff candidate names

  • Cache hit rate per tier. Trend over time, with anomaly detection at deploy boundaries.
  • Cache miss QPS to origin. This is the number that goes 100x in a stampede. Alert on rate-of-change, not absolute value.
  • Per-key QPS distribution. A long tail of hot keys is normal; a single key >5 percent of total traffic is a hot key worth replicating.
  • Single-flight fanout. Followers-per-leader histogram. p99 > 100 means you're frequently collapsing big herds — good if it's working, but it means your TTL or eviction strategy is letting the miss happen.
  • Stale-serve count. How often you're serving past the fresh TTL. A spike means refresh workers are unhealthy.
  • Coalescing latency penalty. p99 of follower wait time. If followers wait longer than they would have spent calling origin themselves, single-flight is mis-tuned.

8. Combinations with other patterns

This pattern lives at the intersection of several others. Some combinations are natural; some require care.

  • hot-key-skew.md. Cache stampede and hot-key skew are siblings, not the same. Stampede is about temporal concentration of misses (concurrent missers on one key in one short window). Skew is about spatial concentration of traffic on one cache shard. Single-flight collapses the temporal burst at each application instance (and distributed single-flight further collapses across instances at the cost of a lock round-trip); replication and load-spreading address the spatial concentration on a cache shard. A celebrity post often needs both: single-flight to handle TTL-driven misses on a hot article body, plus hot-key replication so the cache shard that owns the key doesn't melt under the steady-state read QPS.

  • fan-out-write-amplification.md. Fan-out on write builds per-recipient caches (timeline:user_42). Those caches have the same stampede surface as any other: when the timeline cache expires or evicts, the rebuild has to query a bunch of upstream stores. The pull path of a hybrid push-pull system inherits cache stampede risk for celebrity posts.

  • strong-consistency-under-contention.md. SWR and stale-if-error are explicit decisions to give up consistency in exchange for availability and herd protection. If your invariant is strong consistency (inventory, payment authorization), you cannot use SWR; you have to single-flight only, and accept that misses block. Often the right move is to remove the read from the strict-consistency path entirely (use a separate display cache that lags) and keep the authority check on a different path.

  • idempotency-exactly-once.md. Single-flight requires that the fetch is safe to retry — if the leader fails partway through and a follower takes over, the origin fetch can run twice. For pure reads this is fine. For "fetch this expensive computation that has side effects" (e.g., debit-then-cache), you need idempotency on the origin side.

  • backpressure-flow-control.md. When the herd does break through, the rate at which the application calls the origin must be capped — admission control, semaphore, or token bucket. Single-flight is the optimistic case; rate limiting on the cache→origin path is the pessimistic case for when single-flight isn't enough.

  • coordination-exactly-one-execution.md. Distributed single-flight (cross-process leader election per key) is a special case of "elect one worker to run this job," and the patterns rhyme. Redis SETNX with TTL, ZooKeeper ephemeral nodes, and etcd leases are all variants.

  • durability-at-scale.md. Cache is not durable storage. If the cache cluster loses all state, the origin must be able to absorb the cold-start traffic for as long as it takes to repopulate. The capacity plan for the origin must be sized to "cache offline" mode, even if that's a degraded mode.

9. Likely interviewer probes

These are the deep-dive questions a staff-level interviewer is likely to push on. Have a real answer for each.

  1. "What TTL would you pick for the article body cache, and why?" Talk about the tradeoff between freshness and miss rate. Show that you'd jitter the TTL to avoid synchronized expiry. Mention that with SWR you decouple the fresh window from the serve window — you might pick 5 minutes fresh, 10 minutes serve-if-stale, 30 minutes hard-expire.

  2. "What happens when your Redis cluster fails over to a secondary?" Acknowledge that secondaries usually have cold local CPU cache (even if replicated keys are present, the JIT and connection pools are cold). Talk about read load to the secondary spiking during failover, circuit breaker between application and origin to prevent thunderhead, and validation that your application library does sane connection pooling.

  3. "How does single-flight work across N application servers?" This is the cross-process coordination question. Cover Redis SET key:lock NX PX 5000 with a unique token, follower spin-polling (cheap if cache RTT is low), or pub/sub for completion notification. Mention the failure mode where the leader dies and the lock TTL has to be longer than origin p99 but short enough that a dead leader unblocks followers in seconds.

  4. "Probabilistic early expiration with beta=1 — what's the expected refresh rate per key?" Higher QPS means more dice rolls per unit time, which shifts the expected refresh time earlier within the TTL window (closer to TTL/2 for very hot keys, closer to TTL for cold ones). The property the paper actually proves is not "one refresh per TTL regardless of QPS" — it's that the probability of any reader ever experiencing a hard miss (i.e., observing now >= expiry rather than triggering an early refresh first) decays exponentially with QPS. That's the desired property: hot keys self-protect, cold keys mostly don't bother. Showing you can distinguish "refresh rate" (grows slowly with QPS) from "hard-miss probability" (decays exponentially with QPS) is the staff signal.

  5. "What's the difference between a cache miss and a cache stampede?" A miss is fine — by itself it's just one origin fetch. A stampede is a miss with concurrency: N parallel misses on the same key. The relevant metric is not "miss rate" but "concurrent missers per unique key."

  6. "You mentioned negative caching. What's the security risk?" If user existence is a privacy-sensitive signal, a tombstone TTL that's different from the positive TTL leaks information. Also, an attacker who can mint nonexistent keys at high rate can fill cache memory with tombstones (Denial of Service via cache pollution). Limit by namespace and use cache admission policies (e.g., TinyLFU) that don't admit one-hit-wonders.

  7. "How do you protect the cache itself from being the bottleneck?" Hot-key replication across shards, client-side load balancing across replicas (round-robin or consistent-hash-with-bounded-loads), and an application-local LRU as a top-tier filter. The cache is just another distributed system and has the same hot-shard problems as any other.

10. Common traps → stronger moves

Trap Stronger move
"I'll just cache it." "I'll cache it with single-flight on miss, jittered TTL, and stale-while-revalidate, and I'll alert on origin QPS as a leading indicator of cache failure."
Uniform TTL across all keys. TTL with jitter (base + uniform(-base*0.1, base*0.1)) to prevent synchronized expiry storms.
Increase TTL to "reduce database load." Lengthening TTL doesn't fix stampedes — it just makes them rarer and bigger. Add probabilistic early expiration so the database load is constant regardless of TTL.
One global Redis cluster for all caching. Tiered caches: CDN, application-local LRU, distributed cache, origin. Each tier shields the next; failure of one doesn't take down the system.
"Cache invalidation is hard, so I'll just use short TTLs." Cache invalidation is hard, but short TTLs trade one problem (staleness) for two (staleness plus stampede plus origin load). Use targeted invalidation on write and modest TTL with SWR.
"Cache fallback to database on miss." Cache fallback to database on miss with single-flight, rate limiting, circuit breaker, and serve-stale on origin error. The fallback path must have its own herd protection or the cache is a liability, not an asset.
Cache the exception when origin fails. Never cache exceptions, error responses, or partial results. Cache only well-formed successful responses. Errors are temporary; caching them turns transient outages into long ones.
One cache server for the hot key. Replicate the hot key to N shards; clients pick a replica randomly or by hash; eliminates per-shard hot spotting.
Assuming in-process single-flight is sufficient when the math says it isn't. Compute N-instances × cost-per-fetch versus origin budget. If 200 application servers each issuing one origin query (instead of one query total) still saturates the database, reach for distributed single-flight via per-key Redis lock with a unique-token holder. If N concurrent fetches is fine, stop at in-process — the distributed lock has its own correctness and latency costs.
Cache aside, no SWR, no early expiration. Cache aside + early expiration + SWR + single-flight: a layered defense where any one layer being misconfigured doesn't take you down.
Test cache behavior at steady state only. Run a deploy-and-warmup load test (full restart, cold cache, full peak traffic) at least monthly. The bug you find that way is the one that takes your site down.
Treat CDN as a magic black box. Understand origin shielding and Vary headers; an unintentional Vary: Cookie reduces cache hit rate to near zero.

11. Staff-level answer checklist

When the interviewer pushes on read amplification or "what happens at viral scale," your answer should hit these points without prompting:

  • [ ] Name the failure mode by its real name: cache stampede, thundering herd, read amplification. The vocabulary is itself a staff signal.
  • [ ] Scope the problem before reaching for techniques: cache stampede mitigation is for keys where concurrent-missers times origin-cost actually saturates a resource. For median keys with cheap queries, naive cache-aside is fine — saying so is itself a staff move.
  • [ ] Distinguish temporal stampede (concurrent misses on the same key at expiry or eviction) from spatial hot-key skew (one shard owning a heavily-trafficked key). Mention they need different solutions and they often coexist.
  • [ ] Recommend in-process single-flight (asyncio lock) as the default for any hot key with expensive backing query; reach for cross-process single-flight (Redis lock or lease token) only when the math shows N-instances of in-process is still too much for the origin.
  • [ ] Recommend probabilistic early expiration to flatten the load curve, with the staff-level reference to Vattani et al. or Memcached's lease tokens.
  • [ ] Cite stale-while-revalidate and stale-if-error as availability tools, with the caveat that they trade staleness for availability — confirm the product tolerates it.
  • [ ] Sketch tiered caches (CDN → application-local LRU → distributed cache → origin) and note that each tier needs its own stampede protection.
  • [ ] Call out cold start as a distinct failure mode requiring pre-warming or staged traffic shift, not just steady-state cache design.
  • [ ] Name the metrics: hit rate, origin QPS, per-key QPS, single-flight fanout, stale-serve count. Alert on rate-of-change, not absolute.
  • [ ] Acknowledge cache poisoning: never cache exceptions, validate values, have an invalidate-by-pattern emergency operation.
  • [ ] Connect to siblings: hot-key skew, fan-out, strong consistency, backpressure.
  • [ ] State the tradeoff explicitly: "We are choosing availability over freshness here because the product tolerates 60 seconds of staleness on this read path; if it didn't, I would handle this differently."
  • [ ] Mention TTL jitter to break synchronized expiry, and bounded follower deadlines so a slow leader doesn't take down latency.
  • [ ] If the question is large enough, sketch the capacity plan with and without cache — show that the origin must survive in "cache cold" mode, possibly in degraded read-only state.
