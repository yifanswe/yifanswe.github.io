Ordering / Causal Consistency

2026/05/31  ·  System Design Patterns
Contents
Contents

Designing systems so that events respect "happens-before" without paying the cost of a single global sequencer for every write. See the Staff System Design Glossary for vocabulary (vector clock, Lamport clock, CRDT, quorum, hinted handoff). Closely tied to these question docs: Facebook Messenger, WhatsApp, Google Docs, Twitter, Stripe payment gateway, collaborative whiteboard, Amazon Dynamo.

1. The fundamental hardness

Distributed systems do not have a single shared clock. Two events that happen "at the same time" on two different machines have no objective ordering — physical wall clocks drift (typically tens of milliseconds across a data center, hundreds across regions even with Network Time Protocol (NTP)), packets travel different routes with different latencies, and message queues reorder. Yet users and product semantics demand order: a chat message must show up after the one it replies to, a Google Docs edit must apply to the right base version, a bank account debit must be visible before the dependent credit.

The simplest answer is "give everything a total order from a single sequencer." This works at small scale (a single Kafka partition with one producer, a single Spanner Paxos group). It breaks at scale because:

  1. A single sequencer is a per-key throughput cap — one machine's stamping rate (tens to low hundreds of thousands of stamps per second) bounds the whole keyspace it owns.
  2. A single sequencer requires geographically centralized coordination — cross-region round-trips for every write are unacceptable for messaging or collaborative editing. (Availability is not the problem most of the time — a logical sequencer can be made highly available via Raft. Throughput and geography are.)

The escape hatch is the distinction between total order (every event has a globally agreed-upon position, including events that never interacted) and causal order (events that depend on each other respect happens-before; independent events can be ordered arbitrarily). Causal order is strictly weaker and dramatically cheaper. Leslie Lamport's 1978 "Time, Clocks, and the Ordering of Events" defines happens-before (denoted →): event a → b if a happens before b on the same node, or a is a send and b is the matching receive, or there is a chain. Events with no chain between them are concurrent, and causal order does not require us to pick one over the other.

The staff-level move is to ask: what is the minimum ordering scope my product actually needs? Per-conversation? Per-document? Per-user-account? Then design a partition key that contains that scope inside a single sequencer, and let independent scopes scale horizontally. Where two scopes must merge (multi-region writes to the same key, or two users editing the same document), reach for the toolkit below.

2. How to recognize it in the interview

Prompt phrases

  • "Messages must appear in order on every device."
  • "Two clients edit the same document at the same time."
  • "Multi-region replication, but the user expects a consistent view."
  • "Out-of-order delivery should not corrupt the state."
  • "The conversation/feed/timeline must be deterministic."
  • "Read your own writes" across devices.
  • "Bank ledger" or "double-entry transactions."

Architectural smells

  • Two writers can touch the same logical entity (the same chat, the same doc, the same counter).
  • Wall-clock timestamps are being used as a sort key. (Suspicious — clock skew breaks this.)
  • The system spans regions and the prompt mentions "active-active."
  • A queue or fan-out is between the writer and the storage, so messages can arrive out of order.
  • The state is naturally a sequence (messages, events, edits, ledger entries) rather than a set.

Example interview prompts where this is core

  1. Design Facebook Messenger / WhatsApp — per-conversation ordering across devices and regions.
  2. Design Google Docs — concurrent edits to the same document.
  3. Design a collaborative whiteboard like Miro — same problem in 2D.
  4. Design a payment / ledger system — debits and credits must be causally consistent; double-entry invariants must hold.
  5. Design Twitter timeline — when multiple producers post and a fan-out reorders them, the timeline must still feel sane.
  6. Design an ad-click aggregator — out-of-order click events arriving across regions must aggregate correctly without double-counting.

If you hear any of these, the first 90 seconds of your answer should name the ordering scope and the technique you will use to enforce it.

3. 60-second spoken framing

The invariant I am designing around is causal consistency — if event B could have observed event A, the system must show them in that order. I am explicitly not promising total order across unrelated keys, because that requires a single sequencer and does not scale. My first move is to pick an ordering scope — per conversation, per document, per account — and route all writes for one scope to a single partition with a single writer, so within the scope I get free total order. Where two scopes must merge, like multi-region replication or concurrent edits, I reach for vector clocks, Conflict-Free Replicated Data Types (CRDTs), or Operational Transform (OT) depending on whether I need to detect conflicts, auto-merge them, or transform operations. I will tell you which one I picked and why.

4. Canonical toolkit

Technique What it gives you Cost Typical use
Per-key single writer / single partition Free total order within the partition. Per-key throughput cap; partition is a hotspot if the key is hot. Kafka partition key, Spanner directory, per-conversation log.
Lamport timestamp Total order consistent with causality (but not unique to causality). One scalar per event; ties broken by node ID. Lightweight event ordering; logs; simple replicated state machines.
Vector clock Detects happens-before vs concurrent precisely. One counter slot per ever-active writer in the object's history; at 1000 nodes that's 1000 (node_id, counter) entries on every replicated object until pruned. Dynamo siblings, Riak, multi-region replication where you actually need to detect concurrent writes (not just order them).
Conflict-Free Replicated Data Type (CRDT) Auto-merging data structures (G-Counter, OR-Set, Last-Writer-Wins Register, RGA for text). State per replica grows with the operation history — OR-Set carries one tombstone per ever-deleted element per replica until garbage collected; not every data type has a useful CRDT formulation (e.g., "set with cardinality constraint", arbitrary integer with multiplication). Multi-region counters, offline-first apps, collaborative editing without a central server.
Operational Transform (OT) Transform concurrent operations against each other so they converge. Requires a central server (typically); transform functions are subtle and bug-prone — TP1 must hold for every op-pair. Google Docs, Etherpad, Office Online.
Session sequence number from client Causal order within one client's session at near-zero cost. Only orders one client's own actions. Multi-device sync, idempotency keys, retry deduplication.
Monotonic ID from a sequencer Per-machine monotonic IDs that are roughly time-sortable across machines. Not a true global total order — two events at the same millisecond on different machines are ordered only by an arbitrary machine-ID tiebreaker. Depends on monotonic wall clock. Twitter Snowflake, Sonyflake, KSUID. Good for unique sortable IDs; not a substitute for consensus when you need real linearizability.
Hybrid Logical Clock (HLC) Lamport semantics plus wall-clock approximation, bounded skew. Slightly more code; depends on NTP for the wall-clock part. CockroachDB, YugabyteDB, modern multi-region stores.

4.1 Lamport timestamp update

class LamportClock:
    """One Lamport scalar per node. Monotonically increases with any event."""

    def __init__(self, node_id: int) -> None:
        self.node_id = node_id
        self.counter = 0

    def on_local_event(self) -> int:
        self.counter += 1
        return self.counter

    def on_send(self) -> int:
        # Stamp the outgoing message with our incremented counter.
        self.counter += 1
        return self.counter

    def on_receive(self, incoming_ts: int) -> int:
        # Lamport rule: counter = max(local, incoming) + 1.
        self.counter = max(self.counter, incoming_ts) + 1
        return self.counter

    def order_key(self, ts: int) -> tuple[int, int]:
        # Break ties with node_id so the total order is deterministic.
        return (ts, self.node_id)

Lamport gives a → b implies LC(a) < LC(b), but the converse is false: LC(a) < LC(b) does not mean a → b. Two concurrent events can still be totally ordered by Lamport; you just cannot detect the concurrency from the timestamp alone. That is fine when you only need a deterministic order and do not care about conflict detection.

4.2 Vector clock compare

from enum import Enum

class Causality(Enum):
    BEFORE = "before"        # A happened before B
    AFTER = "after"          # A happened after B
    EQUAL = "equal"          # Same event
    CONCURRENT = "concurrent"  # Conflict; neither observed the other

VectorClock = dict[str, int]  # node_id -> counter

def compare(a: VectorClock, b: VectorClock) -> Causality:
    a_less_eq_b = True
    b_less_eq_a = True
    for node in a.keys() | b.keys():
        ca, cb = a.get(node, 0), b.get(node, 0)
        if ca > cb:
            a_less_eq_b = False
        if cb > ca:
            b_less_eq_a = False
    if a_less_eq_b and b_less_eq_a:
        return Causality.EQUAL
    if a_less_eq_b:
        return Causality.BEFORE
    if b_less_eq_a:
        return Causality.AFTER
    return Causality.CONCURRENT

def merge(a: VectorClock, b: VectorClock) -> VectorClock:
    # Element-wise max; used when a node observes a remote version.
    out: VectorClock = {}
    for node in a.keys() | b.keys():
        out[node] = max(a.get(node, 0), b.get(node, 0))
    return out

def on_local_write(vc: VectorClock, node: str) -> VectorClock:
    out = dict(vc)
    out[node] = out.get(node, 0) + 1
    return out

The expensive part is not the comparison — it is the storage. Every replicated object carries a vector clock with one slot per writer that has ever touched it. In a Dynamo-style cluster with a few hundred nodes where any coordinator can stamp any key, the clock on a long-lived object can accumulate hundreds of (node_id, counter) pairs even though only a couple of them have written recently. Production systems prune entries that have not been touched in a long time, accepting a small probability of false-concurrent reports. Most interview prompts do not actually need vector clocks — they need Lamport stamps or a per-scope sequence number. Reach for vector clocks only when the design has multiple coordinators writing the same key and the application has to distinguish "concurrent, must resolve" from "one strictly followed the other." If you just need a deterministic order, Lamport is cheaper.

4.3 G-Counter CRDT merge

class GCounter:
    """Grow-only counter. Each node increments only its own slot.
    Merge is element-wise max; value is the sum.
    Used for things like "global like count" across regions without coordination.
    """

    def __init__(self, node_id: str) -> None:
        self.node_id = node_id
        self.slots: dict[str, int] = {}

    def increment(self, by: int = 1) -> None:
        if by < 0:
            raise ValueError("G-Counter cannot decrement. Use PN-Counter for that.")
        self.slots[self.node_id] = self.slots.get(self.node_id, 0) + by

    def value(self) -> int:
        return sum(self.slots.values())

    def merge(self, other: "GCounter") -> "GCounter":
        out = GCounter(self.node_id)
        for k in self.slots.keys() | other.slots.keys():
            out.slots[k] = max(self.slots.get(k, 0), other.slots.get(k, 0))
        return out

The CRDT property: merge is commutative, associative, and idempotent. So replicas can gossip in any order, multiple times, and still converge. The data structure pays for this with extra state — a G-Counter stores one integer per ever-active node, not just a single integer.

The honest trade-offs to call out:

  • G-Counter cannot decrement. If you need both increment and decrement (likes minus unlikes), you need a PN-Counter, which is two G-Counters — twice the state, and the displayed value is a difference, which can transiently go negative under partition recovery if a remove arrived without its matching add yet.
  • OR-Set carries tombstones. Every remove(x) must persist as a tombstone until every replica has observed it (causal stability), otherwise a concurrent add(x) on a lagging replica would resurrect the element. Without garbage collection, state grows with the cumulative number of removes, not the current set size.
  • Not every data type has a CRDT. Operations that aren't commutative (assign integer to a function of its current value, enforce "at most K elements") don't have a useful merge. CRDTs make commutative operations cheap, not non-commutative ones possible.

4.4 Operational Transform — insert vs insert

from dataclasses import dataclass

@dataclass
class InsertOp:
    position: int
    text: str
    author: str  # tiebreaker for same-position conflicts

def transform_insert_insert(local: InsertOp, remote: InsertOp) -> InsertOp:
    """Transform `local` so it can be applied after `remote` was applied.
    Both ops were generated against the same base document.
    """
    if remote.position < local.position:
        # Remote inserted to the left; shift our position right.
        return InsertOp(local.position + len(remote.text), local.text, local.author)
    if remote.position > local.position:
        # Remote inserted to the right; we are unaffected.
        return local
    # Same position. Break ties deterministically so both sides agree.
    if remote.author < local.author:
        return InsertOp(local.position + len(remote.text), local.text, local.author)
    return local

OT is what Google Docs uses. The transform function must satisfy the TP1 property: for any two concurrent operations a and b, transform(a, b) then apply b must produce the same document as transform(b, a) then apply a. Building correct OT for rich text (insert, delete, format, with cursors and selections) is famously hard. Many modern systems prefer CRDTs (specifically RGA — Replicated Growable Array, or Yjs/Automerge) to avoid the TP1 minefield.

4.5 Hybrid Logical Clock update

import time

class HLC:
    """Hybrid Logical Clock — Lamport behavior, but tracks wall-clock too.
    Two-tuple: (wall_ms, logical). Comparison is lexicographic.
    Used in CockroachDB, YugabyteDB.
    """

    def __init__(self) -> None:
        self.wall_ms = 0
        self.logical = 0

    def _now(self) -> int:
        return int(time.time() * 1000)

    def on_local_event(self) -> tuple[int, int]:
        now = self._now()
        if now > self.wall_ms:
            self.wall_ms = now
            self.logical = 0
        else:
            self.logical += 1
        return (self.wall_ms, self.logical)

    def on_receive(self, incoming_wall: int, incoming_logical: int) -> tuple[int, int]:
        now = self._now()
        new_wall = max(self.wall_ms, incoming_wall, now)
        if new_wall == self.wall_ms == incoming_wall:
            self.logical = max(self.logical, incoming_logical) + 1
        elif new_wall == self.wall_ms:
            self.logical += 1
        elif new_wall == incoming_wall:
            self.logical = incoming_logical + 1
        else:
            self.logical = 0
        self.wall_ms = new_wall
        return (self.wall_ms, self.logical)

HLC is the modern default when you want Lamport's safety plus the operational benefit of timestamps that approximate wall time (so humans reading logs and tooling building time-series indexes both work).

5. Worked example: naive → broken → fixed (Messenger)

The scenario

Two-person chat between Alice and Bob. Alice's phone is on a flaky cellular connection. Alice sends two messages back-to-back:

T = 100ms : Alice taps "Did you see the email?"      (msg1)
T = 101ms : Alice taps "Reply when you can"          (msg2)

In a single healthy TCP connection these would arrive in order — TCP itself is FIFO. The realistic ways msg2 reaches the server before msg1 are:

  • msg1's send failed and got queued for retry while msg2 went on a freshly re-opened socket. The two now race through different load-balanced application servers.
  • The client reconnected between msg1 and msg2 (cell-to-WiFi handoff, app backgrounded then resumed). The new connection lands on a different server.
  • Alice has two devices (phone and laptop) and they are independent — covered as a separate edge case below.

This is the common failure shape. "Single TCP connection delivered packets out of order" is not — assume FIFO inside one connection.

The naive design

The send service stamps each message with the application server's wall-clock time on receipt and writes to a globally sharded message store. Because msg1 was retried on a slower path, msg2 actually reaches the system first:

App server B receives msg2 at B's clock = 250ms  -> store as ts=250
App server A receives msg1 at A's clock = 280ms  -> store as ts=280
                                              ^ msg1's retry took longer than msg2's first send

Bob's client queries messages where conv_id = X order by ts asc and sees:

[250ms] "Reply when you can"
[280ms] "Did you see the email?"

Bob is confused. The reply appears before the question. Clock skew is not the root cause here — even with perfectly synchronized clocks, the receipt timestamp reflects arrival order, not Alice's send order. Wall clocks would make it strictly worse: a 50ms NTP drift between servers can flip ordering even when arrival order is correct.

Why naive timestamps fail

  1. Arrival time is not send time. Retries, reconnects, and routing variance mean a later-sent message can arrive earlier.
  2. Wall clocks drift independently on top of that. Even with NTP, ±5ms between hosts is normal; ±50ms during NTP outages. Across regions, hundreds of milliseconds.
  3. The events are causally ordered (Alice sent msg1 strictly before msg2 from her single device) but the storage layer threw that information away.
  4. There is no single sequencer, so no machine has authority to say "msg1 is first."

Fix 1: per-conversation single partition with monotonic sequence

The natural ordering scope here is the conversation. Partition the conversation log by conversation_id. All writes for one conversation go to one Kafka partition / one Spanner directory / one log shard. The shard owner stamps each message with a monotonic per-conversation sequence number.

                  +----------------------+
   Alice phone -->| Conversation Router  | -- hash(conv_id) --> shard 17
   Bob   phone -->|                      |
                  +----------------------+
                                                |
                                                v
                                +-------------------------------+
                                | Conversation shard 17 (leader)|
                                |                               |
                                |   seq=42 msg1 "Did you see..."|
                                |   seq=43 msg2 "Reply when..." |
                                |   ...                         |
                                +-------------------------------+

Within the shard, sequence numbers are assigned in arrival order at the leader. Bob reads where conv_id=X order by seq asc and sees the right order regardless of wall-clock drift across servers.

The remaining problem: what if msg1 and msg2 arrived at the leader out of order (msg2 first, because msg1's retry was slower)? Then seq=42 is "Reply when you can" and seq=43 is "Did you see the email?". The shard's own order is still consistent (Bob will see the same order on every device), but it does not match Alice's intent.

Fix 2: client session sequence number

Have the client stamp each message with (client_session_id, client_seq). The client increments client_seq for every message it sends in this session. The shard uses this to sort within a single sender's burst: if two messages from the same client session arrive at the shard, order by client_seq even if arrival order disagrees.

# Client side
class ChatClient:
    def __init__(self, session_id: str) -> None:
        self.session_id = session_id
        self.client_seq = 0

    def send(self, conv_id: str, body: str) -> dict:
        self.client_seq += 1
        return {
            "conv_id": conv_id,
            "body": body,
            "session_id": self.session_id,
            "client_seq": self.client_seq,
        }

# Shard side, when assigning the durable per-conversation sequence:
def assign_seq(shard_state, msg):
    # Buffer briefly to absorb out-of-order arrivals from the same session.
    # Then assign the next shard sequence in (session_id, client_seq) order.
    shard_state.pending.append(msg)
    ready = shard_state.flush_in_order()  # sorts by (session, client_seq)
    for m in ready:
        m["seq"] = shard_state.next_seq()
        shard_state.log.append(m)

Now Alice's two messages — even if delivered out of order to the shard — are reordered by client_seq before sequencing. Bob sees them in Alice's intended order.

Edge case: Alice has two devices

Now Alice has both her phone and her laptop sending into the same conversation. The two devices have independent client_seq counters. They are genuinely concurrent — there is no "right" answer for which Alice-message comes first. The shard breaks ties by arrival order at the leader (effectively a Lamport stamp), and the conversation is consistent across viewers even if a re-watch from a fresh device gets the same order. That is the best you can do without slowing every send down with cross-device coordination.

Why not Spanner / a single global sequencer?

You could use Spanner-style TrueTime to give every message a globally ordered timestamp. It works for the bank ledger case (where you need cross-account total order). For chat, it is overkill — the product only needs per-conversation order, and TrueTime adds ~10-14ms of commit-wait to every write. The per-conversation shard approach gives you what you need at one-tenth the latency.

Summary table

Layer Mechanism What it enforces
Client Session sequence number. Order within one device's burst.
Shard Per-conversation monotonic seq. Order across all writers into the conversation.
Cross-shard None needed. We accepted no cross-conversation order.
Cross-region Single leader region per conversation. One source of truth; followers replicate the seq.

6. Variants and how to choose

Technique Detects conflict? Auto-merges? Total order? Storage cost When to use
Single partition / single writer N/A — no conflict possible. N/A. Yes, within partition. Low. Per-conversation chat, per-account ledger, per-document edits with a server.
Lamport timestamp No (cannot distinguish concurrent from sequential). No. Yes, but arbitrary tie-breaks. One integer per event. Lightweight log ordering; replicated state machine input ordering.
Vector clock Yes (precise happens-before). No (you must merge in app code). No (only partial). O(writers) per object; needs pruning. Dynamo-style stores, multi-master KV with sibling resolution.
CRDT (G-Counter, OR-Set, LWW-Reg, RGA) Implicitly resolved. Yes (mathematically). No. Larger state; tombstones for delete-aware CRDTs. Offline-first apps, multi-region counters, peer-to-peer collaborative editors.
Operational Transform Yes (server resolves). Yes via transform. Yes (after server linearizes). Operation log + transform functions. Server-mediated rich-text editors (Google Docs, Etherpad).
Single sequencer (Snowflake) No. No. Approximate — sortable by time, but events at the same millisecond on different machines are ordered by an arbitrary machine-ID tiebreaker, not by causality. Tiny per-event ID. Globally unique sortable IDs (tweet IDs, ad-click IDs). Not a substitute for consensus when you need real total order under contention.
HLC Effectively yes (via Lamport part). No. Yes (lex order). Two integers per event. CockroachDB, YugabyteDB; modern multi-region SQL.
Consensus (Paxos, Raft) N/A. N/A. Yes, strong. Quorum write latency. Bank ledger, exactly-once invariants, leader election.

Decision flowchart

Does your product need a global total order across unrelated keys?
   YES -> Consensus per shard (Spanner / CockroachDB) or single sequencer.
   NO  -> Continue.

Can you partition by an ordering scope (conversation, document, account)?
   YES -> Per-scope single writer. A monotonic sequence number assigned by
          the leader is usually sufficient — STOP here. This covers most
          chat apps, per-document editing with a central server, per-account
          ledgers within one region.
   NO  -> Continue (true multi-master).

Do concurrent writes have a deterministic merge rule?
   YES, mathematical merge   -> CRDT.
   YES, transform-based      -> OT (need server) or RGA CRDT.
   NO, app must resolve      -> Vector clocks + sibling presentation.

The staff-level read of this flowchart is that most interview prompts terminate at the second box. Reaching vector clocks, CRDTs, or OT is a signal that the design has true multi-master writes, which is itself a choice (often forced by offline-first or cross-region active-active). If the prompt does not require those, don't pay for them.

7. Failure modes and operations

Failure What it looks like Metric to watch Mitigation
Vector clock unbounded growth Single object has VC with hundreds of writer IDs. p99_vector_clock_size. Prune entries older than retention window; cap at K writers; tombstone unused.
CRDT state explosion OR-Set carries millions of tombstones for deleted items. crdt_state_bytes_per_object. Periodic garbage collection with causal stability (Riak's "reap"); switch to delta-CRDTs.
Sequencer outage Kafka partition leader (or per-scope leader) down; writes to that scope blocked. (Snowflake is per-machine, so one instance dying does not block writes — clients route to another generator.) sequencer_unavailability_seconds. Fast leader election (Raft); multiple sequencers per key range with non-overlapping ID space; client-side retry with backoff.
Wall clock skew breaks HLC Clock jumps backward; HLC max(now, prev) masks the issue but operations look misordered in logs. clock_skew_ms_p99. NTP with chrony + multiple time sources; alert on skew > 50ms; reject events with future-dated wall clocks beyond a threshold.
Out-of-order delivery to single-partition log Client retry causes msg2 to arrive before msg1. out_of_order_arrival_rate. Client session seq + bounded reorder buffer at the leader.
Lamport tie collisions Two events with identical Lamport stamp; ties broken by node ID can starve a node. lamport_tie_rate. Periodic clock-jump to spread node IDs; randomize node ID at startup.
Concurrent CRDT semantics surprise users Add-then-delete in one region + delete-then-add in another converges to "exists" or "deleted" depending on CRDT choice. User-reported "my changes disappeared" tickets. Pick the CRDT that matches product expectations (Add-wins OR-Set vs Remove-wins) and document it; expose "this was concurrently edited" UI hint.
OT transform bug TP1 violated; two clients diverge. Diff between server log replay and client state. Property-based tests for every (op, op) pair; canary new transforms; fall back to server snapshot on drift.
Snowflake ID collision under clock rewind Two events with the same ID. id_collision_count. Refuse to issue IDs when wall clock goes backward; use machine-ID bits for safety; HLC inside the generator.
Cross-region replication lag breaks read-your-write User writes in us-east, reads from eu-west, doesn't see their write. replication_lag_p99_ms. Sticky session to the write region for a TTL; client-side write-through cache; explicit causal token in client to filter stale replicas.

Operational discipline

  • Always emit the ordering metadata (Lamport stamp, vector clock, HLC) into application logs, even when you do not act on it. Correctness drift is invisible without it.
  • Have a replay tool that can re-derive the current state from the event log. CRDTs and OT systems must support this for incident recovery.
  • Pre-decide your merge policy in product specs. "Last writer wins" is a product decision masquerading as an engineering one; the product manager owns the call.
  • Test out-of-order delivery in integration tests by deliberately reordering message arrivals at the boundary. Most ordering bugs are reproducible only under specific reorderings.

8. Combinations with other patterns

This pattern is rarely solved in isolation. The strongest staff answers chain it with:

  • strong-consistency-under-contention.md — When the ordering scope is one account or one document and you need not just order but linearizability, you graduate from per-partition single-writer to consensus (Paxos/Raft). Use this for bank ledgers, ticket inventory, and any "no double-spend" invariant.
  • idempotency-exactly-once.md — Causal order plus retries equals duplicate-prone systems. Every ordered write needs an idempotency key (client session + client seq is a natural fit). Without idempotency, retried writes show up as duplicate events at different positions in the order.
  • fan-out-write-amplification.md — When you fan out a message to many recipients (group chat, Twitter timeline), each fan-out destination has its own delivery ordering. Combine per-source order (this pattern) with per-destination merge order (fan-out pattern).
  • coordination-exactly-one-execution.md — A single sequencer / partition leader is itself an exactly-one-execution problem solved via lease or consensus. The two patterns share the lease primitive.
  • hot-key-skew.md — Per-key single writer is a hot-key bottleneck if one key receives most traffic (the "Justin Bieber" chat, the most-edited Google Doc). You may need to sub-shard within a logical key.
  • real-time-aggregation.md — Out-of-order arrival at aggregators (ad-click counts arriving late from a slow region) is solved with watermarks and event-time windows. The aggregation pipeline relies on the upstream ordering guarantee you established here.
  • backpressure-flow-control.md — A single-writer partition under load needs explicit backpressure; otherwise the reorder buffer at the leader grows without bound and ordering guarantees degrade.
  • durability-at-scale.md — The ordered log is the source of truth; durability is non-negotiable. Replicated log (Kafka, Bookkeeper) is the standard substrate.

9. Likely interviewer probes

  1. "You said per-conversation Kafka partition. What happens when one conversation gets so hot that one partition can't keep up?" — First check whether you actually have this problem: a single Kafka partition handles tens of thousands of messages per second, which covers all but a handful of real-world group chats. If you genuinely do (a 500K-member Discord channel during a livestream), sub-shard by (conv_id, sub_partition_id) and merge with a Lamport stamp at read time so per-sender order is preserved while cross-sender order is best-effort. Vector clocks are overkill here — the product only needs "messages from one sender are in order," not pairwise concurrency detection.
  2. "Two users edit the same Google Doc cell from different continents. Walk me through what happens with OT vs CRDT." — OT routes both ops to a central server which linearizes and transforms them against each other; the transform function T(a, b) is hand-written per op-pair and must satisfy TP1. CRDT lets each client apply locally then gossip, with merge baked into the data structure (RGA for text uses position identifiers chosen so insertions don't collide). The honest trade-off: OT has smaller per-document state but the transform code is bug-prone and a transform error silently diverges clients; CRDT has bigger per-document state (every character carries an identifier and characters deleted by anyone live on as tombstones until garbage-collected) but the convergence proof is structural, not code. Google Docs uses OT for historical reasons; newer collaborative editors (Figma, Linear, Notion blocks) lean CRDT because the libraries (Yjs, Automerge) have absorbed the state-management complexity and you avoid TP1 bug surface.
  3. "Your vector clock grows forever. How do you bound it?" — Prune entries with no activity in T (typically days). Accept rare false-concurrent reports for very-stale writers. Many production systems also cap at K entries and evict LRU.
  4. "What does Spanner's TrueTime buy you that HLC does not?" — TrueTime gives an externally consistent ordering across the entire database via commit-wait of the clock-uncertainty interval. HLC only orders events that interact (Lamport semantics); it does not give external consistency. Spanner pays ~10ms per commit; CockroachDB / HLC pay 0 commit-wait but cannot promise external consistency.
  5. "How do you implement read-your-own-write across regions when writes go to the primary region?" — Issue the client a causal token on write (Lamport stamp, HLC, or write commit position). Reads carry the token; replicas refuse to serve if their replication position is behind the token, redirecting to a fresher replica or stalling briefly.
  6. "Tell me about CRDT garbage collection. Why is it hard?" — A deletion in an OR-Set must persist as a tombstone until all replicas have seen and acknowledged it (causal stability). Tracking causal stability requires either central coordination or full vector-clock propagation. Without GC, tombstones grow forever and state explodes.
  7. "You're using Snowflake IDs sorted by time. What goes wrong on clock rewind?" — IDs become non-monotonic; sorting by ID produces a reordering that violates causal order. Mitigations: refuse to issue IDs when clock goes backward (block for the rewind duration); use HLC inside the generator; or pair the time-based ID with a Lamport stamp.

10. Common traps → stronger moves

Trap Stronger move
"I'll just timestamp everything with time.now() and sort." Name a partition key that maps to your ordering scope; assign monotonic sequence at a single writer per partition.
"I'll use Spanner everywhere for global order." Only pay for global order where the invariant requires it (ledger, inventory). Use cheaper per-scope ordering for chat / feed / docs.
"Vector clocks solve everything." Vector clocks detect conflict but don't resolve it. State the application's merge rule explicitly (LWW, set union, app callback).
"We'll use OT for collaborative editing." OT is correct but the transform functions are notoriously bug-prone; if the team is small, prefer a battle-tested CRDT library (Yjs, Automerge).
"Client retries are fine, idempotency keys not needed." Combine ordering with idempotency: (session_id, client_seq) as the dedup key; retried writes get the same slot, not a new one.
"I'll let wall-clock skew sort itself out with NTP." Treat clock skew as adversarial. Use HLC for any timestamp that influences ordering; alert on skew > 50ms.
"Single Kafka partition gives total order — done." Single partition is also a single bottleneck and a single point of failure for that key. Discuss sub-sharding and leader-failover plan.
"CRDTs auto-merge so the product is correct by construction." CRDTs converge mechanically but the convergence target may not match user intent (Add-wins vs Remove-wins). Validate semantics with product.
"We'll order by the database's auto-incrementing ID." DB auto-increment is per-database, not per-key, and breaks under multi-master. Use per-scope sequence or HLC instead.
"Sequencer is a single point of failure — let's not have one." A single logical sequencer can be highly available via Raft. The right question is throughput, not availability.

11. Staff-level answer checklist

  • [ ] You named the ordering scope the product requires (per-conversation, per-document, per-account), not "global order."
  • [ ] You distinguished total order from causal order and chose the cheaper option that meets the invariant.
  • [ ] You picked one of {single-partition, Lamport, vector clock, CRDT, OT, sequencer, HLC, consensus} with a justification grounded in the product, not the buzzword.
  • [ ] You discussed wall-clock skew as a real adversary and named your defense (HLC, monotonic seq, no wall clocks in sort keys).
  • [ ] You showed how client session sequence numbers combine with server sequencing to handle in-flight reordering.
  • [ ] You named the conflict-resolution policy (LWW, set union, application callback, CRDT semantics) and noted that this is a product decision.
  • [ ] You discussed read-your-own-write across regions via causal tokens or sticky writes.
  • [ ] You named at least one operational metric (vector clock size, replication lag, out-of-order arrival rate, sequencer availability).
  • [ ] You acknowledged the multi-device problem (two devices from the same user are genuinely concurrent) and gave a tie-break rule.
  • [ ] You combined this with idempotency for retries and exactly-one-execution for the sequencer / leader.
  • [ ] You called out a fallback path for when the primary mechanism degrades (sequencer down, vector clock grows too large, CRDT state explodes).
