How to Tackle an Unfamiliar System Design

2026/05/31  ·  System Design Patterns
Contents
Contents

Quick reference (read first): When you're stuck, ask one question — "The thing I'm trying to do right now: is it a verb, or an adjective?"

  • Verb → find a tool among the 8 functional verbs (Store / Read / Search / Compute / Move / Coordinate / Schedule / Notify); remember Compute is optional.
  • Adjective → find one among the 6 quality techniques (Fast / Scalable / Reliable / Consistent / Cheap / Observable).
  • The key judgment is at Step 4 (Profile = dominant verb × dominant adjective) — it decides your tool family.

This doc is a method, not an architecture. Use it when a prompt lands in a domain you've never built, and your mind goes blank or anchors on the first tool you happen to remember. It turns "I don't know how to do this" into a finite traversal that always terminates on a tool family.

The core fix: "verb or adjective?"

The failure mode this prevents: you read the prompt, grab the first tool you already know that vaguely fits, and never discover that a whole branch of better options existed — because you never traversed it. The usual tell is anchoring on a familiar batch / query / database tool when the requirement was quietly pointing at a different branch entirely (streaming, a workflow engine, a geospatial index, a fan-out strategy, ...).

Root cause: you didn't traverse your options; you grabbed the first match. The cure is to make the search explicit and exhaustive.

Whenever you're stuck on how to do something, the "something" is always either:

  • a verb — "I need to make the system do X" (store, search, compute, notify...) → traverse the 8 functional verbs.
  • an adjective — "I need to make it be Y" (faster, more durable, consistent...) → traverse the 6 quality techniques.

Most people skip this split and dive into detail. Do the split first; it routes you.

The 9-step template

Four phases, nine steps. Each step asks one question and produces one concrete output. By Step 4 you've decided the spine; by Step 7 the core framework and flow fall out.

Phase A — Understand (what, and how much)

Step 1 · One-sentence job. Ask: "In one breath, what does this system do? What is the one core entity + one core action?" → Output: a sentence + the primary noun & verb. The noun usually becomes your storage / partition key; the verb hints at the dominant functional action you'll traverse for in Step 2.

Step 2 · Functional requirements = VERBS. Ask: "Which of the 8 verbs does it need?" Mark each core / minor / none. → Output: a verb profile. (Remember: Compute is optional — many systems are just Store + Read.)

Step 3 · Non-functional = ADJECTIVES + NUMBERS. Ask: "Which qualities matter, and what's the number?" Attach QPS, p99, data size, freshness, consistency level. → Output: an adjective profile with numbers. (The numbers decide tools in Step 5.)

Phase B — Decide the spine ← the decision happens here

Step 4 · Name the PROFILE = dominant verb × dominant adjective. Ask: "Which 1–2 verbs and 1–2 adjectives dominate everything else?" This single judgment picks your tool family. A Compute × Fast system and a Coordinate × Consistent system share almost nothing — naming the profile is what routes you to the right family before you anchor on a familiar tool.

Step 5 · Map each core verb → a concrete tool. For every core verb, descend its sub-branches (the master sheet below) and pick the row that fits Step 3's numbers. - If Compute is core → choose sync / async / batch / stream. - Watch for the emergent tool: Store + Coordinate + Schedule + Notify all clustering on one long-lived entity ⇒ reach for a workflow engine. → Output: one tool per core verb.

Phase C — Assemble ← framework crystallizes here

Step 6 · Core entities + two views. Ask: "What do I store, keyed by what for writes, keyed by what for reads?" → Output: a write-keyed view + a read-keyed view, and the stage that converts one to the other. The write key and read key often differ; the converting stage is where most of the system's real work lives.

Step 7 · Draw write path, then read path. Trace one event in (ingress → core-verb tools → store) and one query out (request → store → serve). → Output: the end-to-end diagram = your core framework and flow.

Phase D — Harden

Step 8 · Dominant adjective → "what breaks?" → deep dives. Take the #1 adjective from Step 4 and attack it; the failures it implies are your deep-dive list. - Fast/Scalable → hot keys, sharding, cache miss/stampede - Consistent → races, idempotency, saga, failure recovery

Step 9 · Defer. State explicitly what you're cutting from scope.

Reference checklist 1 — the 8 functional verbs

Verb Intent Trigger words
Store put it down to keep persist, save, record
Read get it back by key look up by id, fetch detail
Search find by content, not key full-text, relevance, fuzzy, vector recall
Compute ⚠️optional transform into a new shape aggregate, count, rank, transform
Move ship data A→B decouple, buffer, sync, CDC
Coordinate agree / lock / elect / serialize distributed lock, leader election, config, rate limit
Schedule do it at a time / after a delay cron, timer, deadline, delayed job
Notify push an event to others push, subscribe, webhook, fan-out

Reference checklist 2 — the 6 non-functional adjectives

Adjective Intent Typical techniques
Fast cut latency cache, index, precompute, CDN, read replica, denormalize
Scalable sustain throughput shard/partition, horizontal scale, queue buffering, back-pressure
Reliable available + don't lose data replication, retries, idempotency, WAL, failover
Consistent everyone sees one truth transactions, quorum, consensus, saga
Cheap cut cost tiered hot/cold storage, batching, compression, approximation
Observable debuggable when it breaks metrics, logs, traces, audit log

Step 5 master sheet — every verb → every option → why

Within a verb, pick the row using the numbers from Step 3 (scale, latency, consistency, freshness).

1. STORE — persist data (pick by access pattern, not by name)

Option Why pick it Pick when / avoid
Relational SQL (Postgres, MySQL) ACID transactions, joins, secondary indexes ✅ relationships, multi-row transactions, strong consistency, rich/ad-hoc queries, fits one primary; ❌ write volume exceeds a single primary, global multi-region writes, large unstructured blobs
Key-Value (DynamoDB, RocksDB, Riak) Get/put by key, huge scale, low latency ✅ every access is by a known key, need predictable single-digit-ms at any scale; ❌ range scans, joins, querying by secondary attributes, analytics
Wide-column (Cassandra, ScyllaDB, HBase, Bigtable) Massive write throughput, horizontal scale, tunable consistency ✅ write-heavy, known query patterns, time-series/event logs, want linear scale + tunable quorum; ❌ ad-hoc queries, joins, multi-key transactions, high-cardinality secondary indexes
Document (MongoDB, Couchbase) Flexible schema, nested JSON, per-doc atomicity ✅ schema changes often, data is one self-contained aggregate read/written together; ❌ many-to-many relationships, cross-document transactions, heavy analytics
Distributed SQL / NewSQL (Spanner, CockroachDB, TiDB) ACID + SQL + horizontal/global scale ✅ outgrew a single SQL primary but still need ACID + joins, or global strong consistency; ❌ small scale (over-engineering), latency-sensitive single-region (extra hops + cost)
OLAP columnar (ClickHouse, Druid, Pinot, BigQuery, Snowflake) Fast scans/aggregations over huge data ✅ large scans, group-by/aggregations, dashboards, append-mostly; ❌ point lookups, frequent updates/deletes, OLTP, sub-100ms single-row reads
Object store (S3, GCS, Blob) Cheap, durable, huge blobs, archival ✅ large immutable blobs, backups, raw event archive, data-lake source, cheap cold tier; ❌ small random reads/writes, low latency, frequent in-place mutation, transactional metadata
Time-series DB (TimescaleDB, InfluxDB, Prometheus) Timestamp-optimized, downsampling, retention ✅ timestamped metrics with high ingest + retention/downsampling/rollups; ❌ relational queries, non-time-keyed data, exact financial records
Graph DB (Neo4j, Neptune, JanusGraph) Relationship traversals (friends-of-friends, fraud rings) ✅ value is in multi-hop traversals (recommendations, fraud rings, social graph); ❌ simple lookups/aggregations, or when 1–2 SQL joins suffice (don't add graph infra early)
Ledger / append-only (QLDB, event store, double-entry) Immutable, auditable history ✅ immutable audit trail, money movement, event sourcing, replay; ❌ mutable state with frequent in-place updates, or when history isn't needed
Embedded KV (RocksDB, LevelDB, SQLite) In-process local state ✅ local per-process state (stream-operator state, edge, on-disk cache); ❌ data that must be shared or queried across nodes

2. READ — serve reads / get back by key (pick by read-scaling need)

Option Why pick it Pick when / avoid
Primary-key read on the DB Simplest, no extra infra ✅ read volume the primary handles, want simplicity + freshness; ❌ reads saturate the primary, or many fan-out reads per request
Cache-aside (Redis, Memcached) Sub-ms hot reads, offload DB ✅ hot keys, high read:write ratio, tolerate seconds of staleness; ❌ strong read-after-write, low hit-rate (cold/uniform access), huge values; gotcha: invalidation + stampede
Read replicas (DB followers) Scale reads horizontally ✅ read-heavy relational, reads tolerate replica lag; ❌ read-your-writes / monotonic reads without routing to primary; gotcha: lag spikes during write bursts
CDN / edge cache Geographic latency + origin offload ✅ globally dispersed users, cacheable/immutable or short-TTL content; ❌ highly personalized or fast-changing per-user data; gotcha: purge/invalidation latency
Materialized view / CQRS read model Precompute the read shape, skip joins ✅ read shape needs expensive joins/aggregations, read >> write; ❌ write-heavy + rarely-read view, or projection staleness unacceptable; gotcha: rebuild/backfill
In-memory snapshot in service Zero-hop, fastest ✅ small, hot, slowly-changing data (a small ranked list, config, feature flags); ❌ large or per-user data, or every replica must agree instantly
Client/browser cache Cheapest, closest ✅ static assets, client-owned private data, offline tolerance; ❌ shared mutable data, security-sensitive freshness; gotcha: can't force-invalidate deployed clients

3. SEARCH — find by content, not by key

Option Why pick it Pick when / avoid
Inverted index (Elasticsearch, OpenSearch, Solr) Full-text relevance, filters, facets, aggregations ✅ full-text relevance + filters + facets at scale; ❌ as source of truth (it's a derived index), strong consistency, very-frequent single-field updates; gotcha: reindex/mapping changes
Vector / ANN search (FAISS, Pinecone, Milvus, pgvector, HNSW) Semantic "similar to" via embeddings (ANN = approximate nearest neighbor) ✅ semantic similarity, "more like this", embedding recall, RAG; ❌ exact keyword/boolean is enough, tiny datasets, need exact results; gotcha: recall-vs-latency tuning
Hybrid (BM25 + vector) Lexical precision + semantic recall ✅ need both exact-term precision and semantic recall (best quality); ❌ when one alone suffices (extra complexity + cost)
Postgres full-text / trigram (tsvector, pg_trgm) No new infra ✅ small–medium data, search is secondary, avoid new infra; ❌ high query volume, advanced relevance/facets, large corpus
Geospatial index (PostGIS, geohash, S2, Redis GEO) "Near me" radius/box queries ✅ radius / bounding-box / nearest queries (ride-hailing, "near me"); ❌ non-spatial queries; gotcha: choose cell size/precision for your density
Secondary index in KV (DynamoDB GSI, Cassandra index) Query by non-primary attribute at scale ✅ already on a KV store, need one extra query attribute; ❌ many access patterns or analytics (use a search engine); gotcha: GSI cost + eventual consistency

4. COMPUTE — transform data (⚠️ optional; the verb most people freeze on)

Option Why pick it Pick when / avoid
Synchronous in-request Simple, freshest, no infra ✅ cheap, deterministic, must be in the response; ❌ slow/heavy/fan-out work that blows the latency budget
Async job + workers (Celery, Sidekiq, queue + worker) Keep request fast, retriable ✅ heavy/deferrable, user can wait, needs retries/backpressure; ❌ result needed in the same response; gotcha: idempotency + a result-delivery channel
Batch (Spark, MapReduce, Hive, dbt) Throughput + complex transforms, cheap ✅ huge volume, multi-stage transforms, T+1/periodic, cost-efficient; ❌ minute/second freshness, per-event reactions
Stream (Flink, Kafka Streams, Spark Structured Streaming) Continuous, windowed, stateful, low-latency ✅ continuous freshness + windowed/keyed aggregation + low-latency reactions; ❌ one-shot compute, or batch freshness is fine (streaming adds state/ops cost)
Serverless (Lambda, Cloud Functions) Autoscale, low ops, event-driven ✅ spiky/low-volume, event-driven glue, minimal ops; ❌ sustained high throughput, long-running, latency-sensitive (cold starts), heavy local state
OLAP query engine (Trino/Presto, Druid) Compute = aggregate at query time ✅ arbitrary ad-hoc dimensions chosen at query time, analyst-driven; ❌ fixed high-QPS queries (precompute), sub-100ms user-facing reads
Sketches (Count-Min, HyperLogLog, Space-Saving, t-digest) Bounded memory at huge scale ✅ cardinality/top-k/quantiles/membership at scale with bounded memory, approx OK; ❌ exact counts required (billing, audit)
ML inference / GPU servers Heavy math (embeddings, ranking) ✅ ranking, embeddings, classification; ❌ a rule/heuristic is good enough; gotcha: latency, batching, model versioning

Stream sub-choice: Flink = long-lived keyed state + event-time + exactly-once; Kafka Streams = simpler library tied to Kafka; Spark Structured Streaming = micro-batch if already on Spark.

5. MOVE — ship data A→B / decouple

Option Why pick it Pick when / avoid
Log/stream (Kafka, Pulsar, Kinesis) High-throughput, durable, replayable, partitioned pub-sub ✅ high throughput, many consumer groups, replay, per-partition ordering, back-pressure; ❌ simple low-volume hand-off, or need per-message delay/priority; gotcha: partition key = ordering + hot-partition risk
Work queue (RabbitMQ, SQS, Redis Streams) Point-to-point job dispatch, per-msg ack ✅ distribute discrete jobs, per-message ack/retry/DLQ, delay/priority; ❌ need replay or many independent consumers (use a log); SQS = simplest managed, RabbitMQ = rich routing
RPC (gRPC, REST, Thrift) Immediate request/response ✅ need the result synchronously, low latency; ❌ producer must not block on consumer, traffic spikes, or fan-out (no buffering, tight coupling, cascading failure)
CDC — Change Data Capture (Debezium, Maxwell) Stream DB changes out, no dual-write ✅ propagate DB changes to search/cache/warehouse without dual writes; ❌ when an explicit domain event is cleaner than leaking table rows; gotcha: schema-change handling, ordering
Transactional outbox + relay Reliable publish inside a DB transaction ✅ must publish an event atomically with a DB write (no lost/ghost events); ❌ no DB transaction involved (publish directly); pairs with CDC/relay
Managed ETL (Fivetran, Airbyte, NiFi) Bulk system-to-system sync ✅ bulk SaaS/DB → warehouse sync, low engineering effort; ❌ low-latency or custom per-event logic
Lightweight bus (NATS, EventBridge) Simple eventing ✅ simple internal eventing, low overhead/ops; ❌ need durable replay, long retention, or high-volume log semantics

6. COORDINATE — agree / lock / elect / serialize

Option Why pick it Pick when / avoid
Coordination service (ZooKeeper, etcd, Consul) Strong-consistency leader election, config, discovery, membership ✅ leader election, dynamic config, discovery, membership with strong consistency; ❌ high-throughput data-plane writes (it's a control plane); gotcha: keep it off the hot path
Distributed lock (etcd lease, ZK ephemeral, Redis Redlock) Mutual exclusion / single writer ✅ short critical section, best-effort exclusion; ❌ correctness-critical exclusivity (prefer single-writer partition or a fencing token); gotcha: lock expiry vs long GC pauses
Consensus (Raft, Paxos — usually inside etcd/Spanner) Agree on order/value, replicated state machine ✅ building a replicated state machine / strongly-consistent store; ❌ rolling your own (use a system that embeds it); gotcha: quorum latency, majority availability
Optimistic concurrency (version / CAS) Lock-free conflict detection at write ✅ low write contention, conflicts rare, want lock-free throughput; ❌ high contention (constant retries/livelock); gotcha: must handle the conflict-retry path
Pessimistic lock (SELECT FOR UPDATE) Serialize transactions on a row ✅ high contention on a row that must serialize, short txn; ❌ long transactions or wide lock scope (deadlocks, throughput collapse)
Partition / single-writer-per-key Serialize by routing — no locks at all ✅ serialize all ops for an entity by routing to one owner (Kafka partition, Flink keyed); ❌ ops that span many keys atomically; gotcha: hot key / skew
Idempotency keys Dedupe retries → exactly-once effect ✅ any at-least-once path where retries must not double-apply (payments, webhooks); ❌ truly read-only ops; gotcha: dedupe-store TTL + key design
Saga / orchestration (Temporal, Cadence, Step Functions, Camunda) Multi-step distributed txn with compensation ✅ multi-step, multi-service, long-running consistency with compensations; ❌ a single-DB transaction would do (don't over-engineer); gotcha: design compensations + idempotency
2-phase commit / XA Strict cross-resource atomicity ✅ strict atomicity across a few XA-capable resources; ❌ across services / at scale — blocking, coordinator failure (prefer saga/outbox)

7. SCHEDULE — do something at a time / after a delay / recurring

Option Why pick it Pick when / avoid
Cron / CronJob (cron, K8s CronJob, Quartz, Airflow) Recurring at fixed wall-clock time ✅ fixed recurring wall-clock jobs; ❌ per-entity dynamic deadlines, sub-minute precision, huge fan-out of distinct timers; gotcha: overlap / missed-run handling
Delay queue (SQS delay, RabbitMQ TTL+DLX, Redis sorted-set by due-time) Fire a message after a delay ✅ per-message delay (retry backoff, short reminders) within the broker cap; ❌ very long (days/weeks) delays, or needing to cancel/query pending timers; gotcha: max-delay limits
Durable timers in workflow engine (Temporal/Cadence sleep) Reliable long timers (days–weeks), survive restarts ✅ long deadlines (days–weeks), survive restarts, tied to a process with retries/state; ❌ ultra-high-frequency short timers (engine overhead)
DB-backed scheduler (next_run_at + polling worker) Durable, queryable, wake only due rows ✅ need to query/cancel/modify timers, index by due-time and wake only due rows; ❌ when an off-the-shelf engine already fits (don't reinvent); gotcha: poll interval vs precision
In-process timer wheel / scheduled executor Low overhead ✅ many short-lived timers inside one process, lowest overhead; ❌ must survive crashes or coordinate across nodes (not durable / HA)
Distributed scheduler (db-scheduler, Quartz cluster, Dagster) HA scheduling ✅ many jobs needing HA / failover + visibility; ❌ a couple of simple cron jobs (overkill)
Event-time triggers / watermarks (Flink) Schedule by data time, not wall clock ✅ emit per data-time window, handle out-of-order / late events; ❌ wall-clock scheduling; gotcha: watermark lag vs completeness

8. NOTIFY — push an event to others / fan-out

Option Why pick it Pick when / avoid
Pub/Sub (Kafka, Google Pub/Sub, SNS, Redis pub/sub) Broadcast to many decoupled subscribers ✅ broadcast one event to many decoupled internal consumers; ❌ guaranteed per-user delivery with read state (use a notification service); gotcha: at-least-once → dedupe
WebSocket / SSE / long-poll Real-time push to connected clients ✅ server-push to connected clients in real time (chat, presence, live dashboards); ❌ offline delivery, millions of idle connections (cost); gotcha: connection state, reconnect/replay
Push services (APNs, FCM, Web Push) Mobile/browser notifications ✅ reach mobile/browser when the app is backgrounded/closed; ❌ in-app real-time (use WebSocket), guaranteed/ordered delivery; gotcha: token management, throttling
Webhooks (HTTP callbacks) Notify external/partner systems ✅ notify external/partner systems via HTTP callbacks; ❌ internal high-volume eventing (use a bus); gotcha: retries, signing, receiver downtime → DLQ
Email/SMS (SES, SendGrid, Twilio) User-facing comms ✅ durable user-facing comms, async, no app required; ❌ real-time UX, high-frequency updates; gotcha: deliverability, rate limits, cost
Fan-out-on-write vs fan-out-on-read Write = fast reads (costly for celebrities); read = cheap writes (costly reads) → hybrid ✅ write for fast reads on normal users, read for celebrities/heavy writers → hybrid; ❌ picking one globally at scale; gotcha: the celebrity / hot-fan-out problem
Notification service + in-app inbox Durable notifications with read/unread state ✅ persistent per-user notifications with read/unread, multi-channel; ❌ simple ephemeral broadcast (overkill)

Tools people forget (the "Flink moment")

When you traverse honestly, two tool families are the ones candidates most often miss because they aren't a single verb — they're an emergent cluster:

  • Stream processor (Flink / Kafka Streams). Surfaces when Compute is core and the adjective is Fast/fresh with windowed aggregation. The forgotten alternative to OLAP.
  • Workflow engine (Temporal / Cadence / Step Functions / Camunda). Surfaces when Store + Coordinate + Schedule + Notify all attach to one long-lived entity. The forgotten alternative to "a status column + a nightly cron."

Rule of thumb: if you ever catch yourself about to hand-roll durable timers, retries, and state transitions for a multi-step process → that's a workflow engine.

Use it under pressure (10-second version)

Job → Verbs → Numbers → Profile → Tools → Two views → Flow → What breaks → Defer. Everything pivots on Step 4 (Profile): dominant verb × dominant adjective = tool family.

Practice — run the method on real prompts

The only way this becomes reflex is to run it cold. Open a prompt from the question bank, cover the answer columns, and derive it yourself: Job → Verbs → Numbers → Profile → Tool family → first deep dive. Then open the linked doc and check what you missed. Each prompt below is chosen to land on a different dominant verb, so together they exercise the whole traversal.

Full walk-through (do this once in prose)

Prompt: Design an ad click aggregator

  1. Job: count clicks per (ad, time-window) and serve near-real-time totals. Entity = click event; action = aggregate / count.
  2. Verbs: Move (ingest a firehose), Compute (windowed counting — core), Store (counts), Read (dashboard). No real Search / Coordinate logic.
  3. Numbers: very high event rate; freshness ~1 min; approximate is fine for dashboards but billing wants exact-ish; writes vastly outnumber reads.
  4. Profile: Compute × Fast/fresh. This is the moment to not anchor on "dump it in a warehouse and run a query later" (the Fast-adjective failure). Fresh + windowed + keyed = the streaming branch.
  5. Tool family: ingest log (Kafka) → stream processor with windowed keyed counters (Flink / Kafka Streams) → a serving store (KV / OLAP) for reads. Add sketches if memory blows up; a periodic batch re-aggregation as the exact correction layer.
  6. Two views: write-keyed by ad_id (counters) ↔ read-keyed by (window, facet) (published totals).
  7. Flow: click → Kafka → stream aggregator updates window counters → publishes totals → dashboard reads the serving store.
  8. What breaks (deep dive): a viral ad = hot key (salt + merge); late / out-of-order clicks (watermark); double-counted clicks on the billing path (idempotency / dedupe).
  9. Defer: fraud filtering, per-advertiser custom dimensions, long-history analytics (push to batch / OLAP).

Notice steps 4–5 are where the whole design is decided; everything else falls out.

Core flow · core tradeoff · core design

For each prompt: the flow (the request path in one line), the tradeoff (the decision, both sides named), and the design (the one signature idea to say out loud). Links are in the table above.

Ad click aggregator · Compute × Fast - Flow: click → Kafka (keyed by ad_id) → windowed keyed counters → publish totals → dashboard read. - Tradeoff: stream (fresh, stateful, more ops) vs batch/OLAP (simple, stale) → stream for minute-freshness, batch re-aggregation as the exact backstop. - Design: tumbling / sliding window counters per ad_id; watermark for late events; sketches when memory-bound.

URL shortener · Store/Read × Fast + Scalable - Flow: write: long URL → generate code → KV put; read: code → KV get (cache) → 301 redirect. - Tradeoff: counter / base62 (short, sequential, guessable) vs random / hash (opaque, needs collision check) → choose by privacy needs. - Design: KV mapping + heavy read cache; no compute layer; codes from a partitioned counter or hash.

API rate limiter · Coordinate × Fast - Flow: request → atomic check-and-decrement of a per-key counter (Redis Lua) → allow / deny → refill over time. - Tradeoff: central counter (accurate, +latency) vs local token bucket (fast, drifts) → hybrid local + async central sync. - Design: token-bucket or sliding-window log keyed by client/route; one atomic script so concurrent requests can't race.

Notification system · Notify × Scalable / Reliable - Flow: event → fan-out to per-channel queues → workers call APNs / FCM / email / SMS → write inbox + read state. - Tradeoff: at-least-once (dup risk) vs exactly-once (expensive) → at-least-once + idempotency / dedupe key. - Design: queue + channel adapters + preference/dedup store; retries with a dead-letter queue; per-user rate limits.

Distributed job scheduler · Schedule × Reliable - Flow: enqueue job with due_time → store → wake due rows (or durable timer) → dispatch to queue → run → mark done. - Tradeoff: poll precision vs DB load → index by due_time and wake only due rows; durable timers for long sleeps. - Design: due-time index + leader/lock so each job fires once; idempotent execution; visibility timeout on dispatch.

Typeahead autocomplete · Search / Compute × Fast - Flow: keystroke → look up ranked suggestions for the prefix from a precomputed structure (cache) → return. - Tradeoff: precompute top-N per prefix (fast, stale, storage) vs compute at query (fresh, slow) → precompute + periodic rebuild. - Design: trie or prefix-indexed store holding ranked suggestions; an offline / stream job refreshes the rankings.

Payment system · Coordinate × Consistent - Flow: request + idempotency key → dedup check → write pending ledger entry → call provider (saga) → finalize / compensate → notify. - Tradeoff: 2PC / strong consistency (slow, brittle) vs saga + idempotency (eventual, resilient) → saga with compensating actions. - Design: double-entry ledger + idempotency keys + outbox; a reconciliation job catches drift.

Nearby places / friends · Search(geo) × Fast - Flow: location → map to geo cells (geohash / S2) → fetch entities in cell + neighbors → sort by exact distance. - Tradeoff: cell size — coarse (few queries, over-fetch) vs fine (precise, many cells) → tune to density. - Design: geospatial index keyed by cell; for live friends, a location-update stream + ephemeral store.

News feed / timeline · Notify(fan-out) + Read × Fast - Flow: post → fan-out-on-write to followers' timeline caches; read → fetch precomputed timeline → rank → hydrate. - Tradeoff: fan-out-on-write (fast reads, costly for celebrities) vs on-read (cheap writes, slow reads) → hybrid. - Design: per-user timeline cache + push for normal users + pull-merge for high-fan-out accounts.

Web crawler · Move / Coordinate × Scalable - Flow: seeds → frontier queue → fetch → parse → extract links → dedupe (seen?) → enqueue new → store pages. - Tradeoff: throughput vs politeness — crawl fast (overload hosts) vs per-host limit (slower) → per-host scheduling. - Design: URL frontier + bloom-filter dedupe + per-host rate scheduler; object store for content.

Distributed cache (Redis) · Store/Read × Fast - Flow: get → hash key → route to shard → in-memory hit, or miss → origin → populate; writes replicate to a replica. - Tradeoff: eviction (LRU vs LFU) and write policy (write-through vs write-back) → pick by hit-rate vs durability. - Design: consistent hashing across shards + a replica per shard; TTL + eviction; client- or proxy-side routing.

Object store (S3) · Store × Reliable / Scalable - Flow: PUT → chunk → replicate / erasure-code across nodes & AZs → record in metadata index; GET → metadata → reassemble. - Tradeoff: replication (simple, 3× cost) vs erasure coding (cheap, CPU / repair cost) → EC for cold, replicas for hot. - Design: split the metadata plane from the data plane; durability via redundancy + background scrubbing / repair.

WhatsApp / messenger · Notify / Move × Reliable - Flow: send → server persists → push to recipient's live connection, or store for offline → delivery / read receipts back. - Tradeoff: strict ordering (latency) vs best-effort → per-conversation ordering + at-least-once + message-id dedupe. - Design: persistent connection gateway + per-user inbox queue; message id for idempotency; group fan-out.

Google Docs (collab editing) · Coordinate × Consistent - Flow: edit → op to server → server orders / transforms → broadcast to other clients → each applies and converges. - Tradeoff: OT (central server, complex transforms) vs CRDT (P2P-friendly, metadata overhead) → choose per architecture. - Design: per-doc ordered op log with OT or CRDT for convergence; a separate presence / cursor channel.

Ticketmaster / booking · Coordinate × Consistent - Flow: browse → hold seat (reservation with TTL) → checkout → confirm, or auto-release on expiry. - Tradeoff: optimistic (oversell risk) vs pessimistic reservation / lock (lower throughput) → short-TTL per-seat reservation. - Design: per-seat state machine (available → held → sold) with atomic transition; a virtual waiting queue for flash sales.

Metrics & monitoring (Datadog) · Compute / Store × Scalable / Cheap - Flow: agents push metrics → ingest → stream rollup per series / time-bucket → time-series store → query & alert. - Tradeoff: raw retention (precise, costly) vs downsampling / rollups (cheap, lossy) → tiered resolution by age. - Design: time-series DB + pre-aggregated rollups; tag / label cardinality control; windowed threshold alerting.

Uber / Lyft (matching) · Search(geo) / Coordinate × Fast - Flow: driver locations stream into a geo index; rider request → find nearby drivers → match → dispatch → track trip state. - Tradeoff: match radius — wide (more supply, longer ETA) vs tight (fast, may fail) → adaptive radius. - Design: geo index of live drivers + matching service + per-trip state machine; absorb high-rate location updates.

What the spread teaches

Seventeen prompts that together touch all eight functional verbs — each landing on a different tool family, all produced by the same nine-step traversal. That is the entire point: you are not memorizing architectures, you are running a traversal that routes any unfamiliar prompt to the right family before you anchor on the first tool you happen to remember.
