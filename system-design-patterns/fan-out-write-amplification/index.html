Fan-out / Write Amplification

2026/05/31  ·  System Design Patterns
Contents
Contents

One producer write must become visible to N consumers. The pattern is the toolkit for choosing where to pay the cost — at write time (push), at read time (pull), or in a hybrid split — and for taming the celebrity case where N is unbounded. See the Staff System Design Glossary for foundational terms.

Core question docs where this pattern is the center of gravity: - Design Twitter - Design Twitter Timeline - Design Facebook Newsfeed - Design Instagram - Design a Notification System - Design Discord - Design a Live Comment Streaming Service like Twitch Chat

1. The fundamental hardness

The naive shape of any social, messaging, or notification system is a single user action that must be reflected in many places. A Tweet is one row written by one user; the product promise is that up to hundreds of millions of followers see it in their timeline. A new chat message is one insert; the promise is that every member of the channel — possibly tens of thousands in a Discord server, possibly two devices for a 1:1 conversation — gets the message delivered, ordered, and counted in unread state. An ad-click event is one record; the dashboards, billing aggregates, fraud detectors, and reporting indexes all need their own derived view of it.

The invariant that breaks under naive design is bounded per-write work. If the cost of one producer write is O(N) where N is the size of an unbounded social graph, then any single popular producer is a denial-of-service vector against your own database. A Beyoncé tweet at 100M followers and a 1KB timeline row is a 100GB write burst per tweet. A Discord announcement to a 500K-member server is a 500K-row fan-out. The CPU and I/O budget of the system is no longer dominated by users; it's dominated by the graph of who-follows-whom.

The opposite extreme — compute the read view from scratch by scanning everything a consumer follows — moves the cost to read time. That solves the celebrity problem but creates a symmetric read-amplification problem (see the sibling pattern cache-stampede-read-amplification.md). A user who follows 5,000 accounts now triggers 5,000 partition reads on every timeline refresh, and the average user refreshes more often than the average user posts. Both extremes are bad; the staff-level move is to know exactly when to use which, and how to combine them.

The pattern is also about decoupling latency from work. The user who wrote the Tweet must get a fast acknowledgment (the write to their own outbox must commit in tens of milliseconds). The 100M follower deliveries must complete eventually, with bounded freshness lag, but they do not have to complete in the request path. That's the role of Kafka, fan-out workers, and a clear split between the synchronous authoritative write and the asynchronous derived projection.

2. How to recognize it in the interview

Prompt phrases that signal this pattern

  • "Every follower sees the post."
  • "Timeline / feed / inbox."
  • "Notifications to all subscribers."
  • "Members of the channel."
  • "Broadcast to all clients connected to the room."
  • "Push the update to N devices."
  • "Activity stream / what's new."
  • "When a user posts, all their friends..."

Architectural smells in stated requirements

  • The producer and consumer counts are wildly asymmetric (one writer, many readers).
  • The follower / subscriber graph has heavy tail — a small number of nodes have orders of magnitude more edges than the median.
  • Read freshness target is on the order of seconds (not milliseconds, not minutes).
  • The product is read-heavy in aggregate (timeline refreshes >> tweets posted).
  • There is an "unread" or "delivery receipt" requirement, meaning per-recipient state has to exist somewhere.
  • The interviewer mentions "personalized" — each recipient's view differs, so a shared cache key doesn't work.

Example "design X" prompts where this is the core hardness

Prompt Why fan-out is the center

3. 60-second spoken framing

The hardness here is write amplification: one producer write has to reach N consumers, and N is unbounded for celebrities. I'd start with a hybrid push-pull model. For the median user — a few hundred followers — I push at write time by inserting into each follower's timeline cache, so the read is a single key lookup. For users above a celebrity threshold, say 10K followers, I do not push. Their posts live in a per-author store; the timeline read merges precomputed pushes plus a pull from the small set of celebrities the reader follows. Fan-out itself is async behind Kafka, so the producer's write latency is decoupled from delivery work, and fan-out workers are sharded by recipient to avoid hot writers. The thing I'd watch in production is the celebrity threshold drifting — recompute it from the follower distribution monthly.

4. Canonical toolkit

4.1 Push model (precompute on write, "fan-out on write")

What it is. When a producer writes, synchronously enqueue a job that inserts the produced item into every consumer's materialized inbox / timeline. Read time becomes O(1) — fetch the top K from your own inbox.

producer writes Tweet T
        │
        ▼
   tweets table  (authoritative store)
        │
        ▼
   Kafka topic  (fan-out events)
        │
        ▼
  fan-out workers
        │
        ├── insert (follower_1, T) into timeline:follower_1
        ├── insert (follower_2, T) into timeline:follower_2
        └── ... N times

When to use. Read-heavy systems where the cost of one read is more expensive than one write summed over the read frequency ratio. If users refresh their timeline 50x for every Tweet they post, you've already lost the math if reads are O(followees) and writes are O(1).

Cost. Storage proliferates — each Tweet is duplicated into every follower's timeline. Write fan-out is O(N) per produce. For N=100M, this is multi-minute work for one event.

When it breaks. Celebrity accounts. Inactive followers (you fan out to accounts that haven't logged in in two years). Burst posts (one popular event causes everyone with a million followers to post at once; queues back up).

def on_tweet(tweet):
    db.tweets.insert(tweet)               # authoritative
    kafka.publish("fanout", {
        "tweet_id": tweet.id,
        "author_id": tweet.author_id,
    })

def fanout_worker(event):
    author = event["author_id"]
    for follower_id in followers_of(author):    # may be 100M
        timeline_cache.lpush(
            f"timeline:{follower_id}",
            event["tweet_id"],
        )
        timeline_cache.ltrim(f"timeline:{follower_id}", 0, 999)

4.2 Pull model (compute on read, "fan-out on read")

What it is. Store only the producer's own outbox (their own tweets / posts). On read, the consumer queries every followee's outbox, merges by timestamp, returns top K.

read timeline for U:
   followees(U) = [a, b, c, ..., m]      # may be 5000
   for each f in followees(U):
       scan top K from tweets where author = f
   merge-sort by ts, take K

When to use. Write-heavy systems with few reads. Celebrity-only paths (when the producer has too many followers to push). Cold accounts that read so rarely it's cheaper to compute on demand than maintain a materialized view.

Cost. Read is O(followees) partition reads plus merge cost. Tail latency is bounded by the slowest shard. Cache hit rate is per-author-outbox, not per-reader, so cache benefits the popular producers but doesn't help readers who follow obscure accounts.

When it breaks. Power users who follow thousands of accounts. Read QPS spikes (everyone refreshing at 9am). Any per-reader personalization (ranking model) that has to score all candidates.

4.3 Hybrid push / pull (the Twitter "celebrity threshold")

What it is. Pick a follower-count threshold C (e.g., 10,000). Accounts below C are pushed normally. Accounts above C are not pushed; their tweets sit in a per-author celebrity store. On timeline read, the consumer merges (a) their pre-pushed timeline cache with (b) a pull-fan-out across the small set of celebrities they follow.

on tweet by author A:
   if followers(A) < C:
       push to every follower's timeline cache
   else:
       store in celebrities:{A} only (do not push)

on timeline read for U:
   cached    = read timeline:{U}                   # pre-pushed posts
   celebs    = followees(U) ∩ celebrity_set        # usually < 50
   recent    = [read celebrities:{c} top K for c in celebs]
   return merge_by_ts(cached, *recent)[:K]

When to use. The follower distribution is heavy-tailed (it always is). You want the cheap path for the 99% and a different path for the 1% that would otherwise drive write storms.

Cost. Two code paths. The celebrity set is itself state that must be maintained (graduating a user above the threshold means stopping pushes for them; demoting means restarting). Read path has to do a small fan-out per request.

When it breaks. A user just below the threshold suddenly goes viral. A timeline reader who follows 200 celebrities (the read-side fan-out is now non-trivial). The threshold is set wrong for the actual graph — too low and you waste reads, too high and write storms persist.

4.4 Kafka (or any durable log) for async decoupling

What it is. The producer's authoritative write commits synchronously to the source-of-truth store. A second event publishes to Kafka. Fan-out happens in workers that consume from Kafka. The producer never blocks on fan-out work.

producer ──► tweets DB  (sync, ack here)
         └─► Kafka      (async, decoupled)
                  │
                  ▼
            fan-out workers (scalable, sharded, retryable)

When to use. Always, in any serious design. Even if your fan-out is trivial today, decoupling lets you scale workers independently, retry independently, and add new consumers (search index, analytics, notifications) without modifying the write path.

Cost. End-to-end latency adds a few hundred ms of queue delay. Exactly-once semantics require idempotent workers (see idempotency-exactly-once.md).

When it breaks. Queue backs up under burst load and the freshness SLO is violated. Workers crash mid-fan-out and you re-deliver duplicates. Partition assignment causes one fan-out worker to own a celebrity and choke its partition.

4.5 Fan-out workers sharded by recipient

What it is. Instead of one Kafka partition per author (which puts all of Beyoncé's fan-out on one worker), shard fan-out work by recipient. The fan-out coordinator reads the author's follower list, hashes each follower to a partition, and emits one event per recipient (or per recipient-shard batch).

              "tweet T by author A"
                       │
                       ▼
          coordinator: load followers(A)
                       │
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
   partition 0     partition 1     partition N
   followers       followers       followers
   hash%N=0        hash%N=1        hash%N=N
       │               │               │
   worker 0        worker 1        worker N
   inserts into    inserts into    inserts into
   timelines for   timelines for   timelines for
   their recipients

When to use. Any push model where the producer's follower count can be large. Recipient-sharding spreads the load evenly across the cluster instead of concentrating it on whichever worker owns the author key.

Cost. Coordinator must read the entire follower list to dispatch. For 100M followers this is itself a streaming operation (you don't load the list into memory; you scan the follower index and emit batched events).

When it breaks. If your follower table itself isn't sharded for streaming reads, the coordinator becomes the bottleneck. If you batch too aggressively, individual recipient delivery latency degrades.

4.6 Priority queues for VIP recipients

What it is. Not all recipients are equal. The author's own device should see the post before strangers do. Verified accounts may have an SLO. Active users (currently in-app) get priority delivery; offline users wait.

fan-out queue tiers:
  P0: self + currently-active sessions     (< 1s SLO)
  P1: recently-active followers             (< 30s SLO)
  P2: long-tail dormant followers          (best effort, minutes)

When to use. When freshness perception varies sharply by recipient. Twitter's "you see your own tweet immediately" is a P0 path; the cousin who hasn't logged in for two months is P2.

Cost. Three queues, three sets of workers. Operational complexity. Need to classify recipients on the fly (cache of "active in last 24h").

When it breaks. Classification is stale and you under-prioritize someone who just opened the app. P0 starves itself (everyone is "active" during a major event).

4.7 Dedicated celebrity store, merged at read time

What it is. Beyond the push/pull split, store celebrity posts in a dedicated, heavily-cached, geo-replicated store optimized for fan-in reads. Every timeline read does a merge between the consumer's materialized timeline and a small fan-in across the celebrities they follow. The celebrity store is sized for the ~10,000 most-followed accounts and can be fully cached in memory globally.

When to use. When the celebrity tail is small enough to fit hot. When read-time merge cost is acceptable (a few extra partition reads).

Cost. Cache for the celebrity set must be kept warm and consistent globally. The set membership (who is a celebrity) changes over time.

When it breaks. A non-celebrity goes viral overnight and isn't in the celebrity store. The merge sort at read time is non-trivial when a user follows many celebrities and wants per-tweet ranking.

5. Worked example: naive → broken → fixed

Scenario. A Twitter-like system. 500M daily active users. Median user has 200 followers. p99 follower count is 50K. Top 1,000 accounts have between 1M and 200M followers. Posts per day: 500M. Timeline reads per day: 50B (each user refreshes ~100 times). Freshness SLO: a non-celebrity post should appear in followers' timelines within 5 seconds (p95).

Naive design

"On every Tweet, fan out to every follower synchronously. On read, fetch from a per-user timeline table."

Math: - Writes per day = 500M. - Average fan-out = 200, but the distribution is heavy-tailed. Total fan-out writes per day ≈ 500M × E[followers]. With heavy tail, E[followers] is dominated by celebrity posts. If 0.001% of posts come from 100M-follower accounts, that's ~5,000 posts × 100M = 500B follower-writes from celebrities alone. Plus 500B for the median. - Per-write payload: 100 bytes timeline entry. - Storage per day for the materialized timeline: 1T entries × 100B = 100TB. Per year: 36PB. - Sync fan-out blocks the producer for minutes when Beyoncé tweets — write latency p99 explodes.

Exact failure mode

  • Hot worker. The Kafka partition owning Beyoncé's userId gets every Beyoncé tweet; its single fan-out consumer is sequential. Lag grows monotonically until the consumer crashes.
  • Storage blow-up. Timelines for inactive users (those who haven't logged in for 6 months) are still being written to. Half the storage is for users who will never read it.
  • Write storms during major events. A breaking news tweet by a major news account at 9am Eastern triggers fan-out to 100M followers, all of whom are also refreshing — read QPS and write QPS spike together.
  • Tail latency. p99 write latency for the producer tracks fan-out completion. A Beyoncé tweet takes minutes; the user sees a spinner.

Proper design

  1. Decouple sync producer write from fan-out. Producer commits to tweets table; emits tweet_published to Kafka. Producer ACKs in < 50ms regardless of follower count.
  2. Hybrid push / pull. Compute a celebrity threshold C from the follower-count distribution monthly. Set C = 10,000. Authors below C are pushed; authors at or above C are not pushed — their tweets live in celebrity_tweets:{author_id} only.
  3. Recipient-sharded fan-out workers. The fan-out coordinator scans the follower index in chunks and emits one Kafka event per recipient-shard (where partitioning is by follower_id % N). Workers pull from their partition and write to the timeline cache for their recipients only.
  4. Priority tiers. P0 = the author's own device (push instantly). P1 = followers active in the last 24h (push within 5s). P2 = dormant followers (push best-effort, with a cap of 30 minutes of lag tolerated).
  5. Active-follower filter. Don't fan out to users who haven't logged in for 30+ days. Their timeline is computed on-demand from per-author outboxes if they ever return.
  6. Celebrity store at read time. Timeline read = (a) pull pre-pushed timeline from cache for the reader, plus (b) for each celebrity the reader follows (typically < 50), pull recent posts from celebrity_tweets:{author}. Merge by timestamp / rank, return top K.

Numbers after fix

  • Median-author fan-out: 200 timeline inserts per Tweet, mostly to active followers (~80 actually delivered). Bounded per-event work.
  • Celebrity-author posts: ~5,000 per day, zero fan-out writes. Each is a single insert into the celebrity store, replicated to all regions.
  • Sync write latency for producer: < 50ms p99, independent of follower count.
  • Fan-out worker lag: P0 < 1s, P1 < 5s, P2 < 5 minutes — measured per-tier, alertable.
  • Storage: only active timelines materialized; storage savings ~50% from dormant-follower exclusion.
  • Read path: timeline cache hit ≈ 95%; merge with celebrity store adds 5-20ms of read latency depending on how many celebrities the reader follows.

6. Variants and how to choose

Variant Write cost Read cost Storage Best for Example
Pure push O(N followers) O(1) High (N × posts) Read-heavy, bounded fan-out Discord channels with member cap
Pure pull O(1) O(F followees) Low (1 × posts) Write-heavy, low-read or fully personalized ranking Personalized news ranking with full re-rank per read
Hybrid (celebrity threshold) O(N) for normal, O(1) for celebs O(1) + O(few celebs) Medium Heavy-tailed follower graph Twitter, Instagram
Push to active only O(active followers) O(1) + on-demand backfill for dormant Lower Apps with large dormant tail LinkedIn feed
Pull with aggressive cache O(1) O(F) but cached per-author Medium Small fan-out, hot producers Reddit subreddit listings

Decision drivers: - Skew of follower distribution. Long-tail → hybrid. Uniform → push or pull. - Read:write ratio. > 10:1 → push. < 1:1 → pull. Otherwise hybrid. - Personalization at read. Per-user ranking on the candidate set → pull, because you can't precompute every reader's view anyway. - Freshness SLO. Seconds → push or hybrid. Minutes-acceptable → pull is fine. - Active user ratio. If only 30% of followers are active, don't push to the other 70%.

7. Failure modes and operations

Failure mode Symptom / metric Mitigation
Celebrity write storm Fan-out lag spikes; one Kafka partition's consumer lag grows Hybrid push/pull threshold; recipient-sharded workers; rate-limit fan-out per producer
Queue backlog under burst Kafka consumer lag > SLO Auto-scale workers on lag; spill P2 work to slower batch path
Dormant follower waste Storage growth faster than DAU growth Active-follower filter; on-demand backfill for returning users
Duplicate delivery Same tweet appears twice in some timelines Idempotent insert keyed by (recipient, tweet_id); see idempotency-exactly-once.md
Out-of-order delivery Older tweet appears above newer Sort by tweet timestamp (not insert time) in the timeline cache; ranking uses authoritative ts
Threshold misconfiguration Latency regression after a follower-distribution shift Recompute celebrity threshold monthly; alert on threshold drift
Cold cache after deploy Read-path latency spike Warm timeline cache from the materialized store on rollout; gradual cutover
Hot reader (power user follows everyone) Slow timeline reads for one user Cap per-user "followees" in read fan-out; offer "lists" UI to segment
Compaction / GC of old timeline entries Disk full Timeline ring buffer with fixed length (e.g., 1000 newest); evict cold
Cross-region fan-out Inter-region write storm Regionalize fan-out: each follower's home region runs the insert; cross-region pub/sub of post events only

On-call incident playbook

Symptom: "Fan-out lag alert, P1 tier, lag = 4 minutes."

  1. Identify the producer cohort. Query the lag-by-partition dashboard. If lag is concentrated on a few partitions, look at the authors whose follower-shard hashes to those partitions.
  2. Check for a celebrity bypass miss. Is there an author near the threshold who just crossed it? If yes, mark them as celebrity manually and replay the affected events through the celebrity path.
  3. Check worker health. CPU? GC pauses? Downstream timeline-cache shard hot? Often the timeline cache shard is the actual bottleneck, not the worker.
  4. Apply backpressure. Temporarily downgrade P2 fan-out (delay dormant deliveries) to free worker capacity for P0/P1.
  5. Hot-shard remediation. If one timeline-cache shard is the bottleneck, do a one-off re-shard or add a read-through cache layer in front of it.
  6. Post-incident. Update the celebrity-threshold computation. If a new product (e.g., breaking-news mentions) is driving spikes, route those through a separate fan-out class.

8. Combinations with other patterns

Fan-out / write amplification rarely lives alone. The typical stack:

  • hot-key-skew.md — Celebrities are the canonical hot key on the write side; their posts are the canonical hot key on the read side. The celebrity store is a hot-key mitigation. Fan-out worker sharding is hot-key mitigation in the queue layer.
  • idempotency-exactly-once.md — Fan-out workers retry on failure. Inserts into recipient timelines must be idempotent (deduplication keyed by (recipient_id, tweet_id)).
  • cache-stampede-read-amplification.md — Timeline reads hit a cache. Cache stampedes occur when many users refresh after a major event. Use request coalescing and stale-while-revalidate.
  • backpressure-flow-control.md — Fan-out queues need backpressure: when the consumer lag exceeds SLO, the system must throttle producers or degrade tiers, not blindly accept work.
  • ordering-causal-consistency.md — Chat messages must arrive in order per channel. Fan-out by channel partition; do not parallelize across partitions for a single channel.
  • durability-at-scale.md — The authoritative tweet store must be durable. The materialized timelines do not have to be — they are a derived projection and can be rebuilt from the authoritative store.
  • real-time-aggregation.md — Counters ("retweet count", "like count") are themselves fan-out problems where the writes go to a counter store; combine with fan-in aggregation.
  • coordination-exactly-one-execution.md — When the coordinator dispatches recipient-sharded fan-out, exactly-one dispatch per (author, tweet) must be guaranteed even if the coordinator restarts.
  • search-and-ranking.md — If timelines are ranked (not just chronological), the candidate set is built by fan-out, then ranked at read time per user. Ranking lives on top of fan-out, not instead of it.

9. Likely interviewer probes

Q1: Walk me through the celebrity threshold. How do you pick it, and what happens when someone crosses it?

Strong answer shape: The threshold is a function of the follower-count distribution and the per-write cost of fan-out. Empirically you pick it where the marginal cost of one more push exceeds the marginal cost of one more pull at read time. In practice, ~10K-50K for Twitter-scale. The transition is asymmetric:

  • Promotion (regular → celebrity). Stop pushing new tweets; followers start seeing this user's posts via the celebrity pull-merge at read time. No backfill needed — older tweets were already pushed and remain in follower timeline caches.
  • Demotion (celebrity → regular). Subtle: tweets posted during the celebrity period were never pushed; they were visible only through the celebrity pull-merge. If you turn off the pull-path the moment the user drops below threshold, those tweets vanish from follower timelines. Real systems mitigate this three ways: (1) keep the user in the celebrity-pull set for a grace period (e.g., 30 days) after demotion so followers continue merging until the tweets age out of the bounded timeline window naturally; (2) trigger a one-time backfill fan-out of the last N celebrity-era tweets at demotion (correct but expensive — itself a write storm); (3) add hysteresis on the threshold (promote at 10K followers, demote only at 5K) so flapping is rare and the demotion path is exercised infrequently.

The threshold itself is recomputed periodically (monthly) based on the current graph and current cost ratios. Most production systems combine grace period + hysteresis and rely on bounded timeline length to close the consistency window.

Q2: Your fan-out worker has crashed mid-job. There were 50M followers; 20M had been delivered. How do you recover?

Strong answer shape: Inserts must be idempotent (deduplicated by (recipient_id, tweet_id) at the timeline-cache layer). The fan-out job is checkpointed by follower-shard — you know which shards completed. On restart, re-emit only the uncompleted shards. Within a shard, the re-emission re-inserts the same tweet_id, which is dropped by the idempotency check. Cross-reference idempotency-exactly-once.md.

Q3: How do you bound the work for one author's tweet without bounding the size of their follower list?

Strong answer shape: Decouple write commit from fan-out completion. The author's write commits in O(1) — single insert to tweets. Fan-out runs async in worker pool, sharded by recipient, possibly across multiple priority tiers. The author never blocks on their own followers. Bounded work is a per-worker property, not a per-author property. If the author has 100M followers, the system does 100M units of work, but no single component does more than its slice.

Q4: How do you handle the case where a user posts a tweet, then immediately refreshes their own timeline and doesn't see it?

Strong answer shape: Self-visibility is a P0 path. On post, write to (a) the authoritative tweets table, (b) immediately insert into the author's own timeline cache before fan-out kicks off, (c) emit the fan-out event. The author sees their own tweet via path (b) within milliseconds of writing it. Read-your-own-write is a separate concern from general freshness.

Q5: Why not just do everything at read time and avoid the write storm entirely?

Strong answer shape: Read traffic dominates write traffic by 50-100x in social feed products. For a chronological feed with a 50-post display, naive pull is actually fine at p50 — 50 reads from in-memory cache complete in single-digit ms. Push earns its complexity in three specific cases: (1) ranked feeds where candidate generation across 5000 followees would re-score every refresh, (2) tight p99.9 SLO where scatter-gather tail latency hurts, (3) extreme refresh QPS where the per-refresh read budget exceeds cache capacity. The honest staff-level answer is: pull suffices for basic chronological feeds; push is a ranking / tail-latency / QPS optimization, not a generic "many followers" fix.

Q6: How does your design change if posts can be deleted?

Strong answer shape: Deletes are a second fan-out problem — they must also reach every recipient. Two common approaches: (a) lazy delete — keep the tweet_id in timelines but mark it deleted in the authoritative store; the read path filters out tombstoned tweet_ids; (b) eager delete — emit a delete event to the same fan-out workers, which remove from each recipient's timeline. Lazy is cheaper and simpler; eager gives faster removal but doubles the fan-out cost. For privacy-sensitive deletes (e.g., revoked posts), eager is required and SLO-bound.

10. Common traps → stronger moves

Trap Stronger move
Pure push, no celebrity carve-out. Hybrid push / pull above a threshold computed from the follower distribution.
Sync fan-out in the producer's write path. Authoritative write commits sync; fan-out runs async behind Kafka.
Partition fan-out work by author. Partition by recipient — celebrity authors no longer pin one worker.
Treat all followers equally. Priority tiers (P0 self, P1 active, P2 dormant) with separate SLOs.
Fan out to every follower including dormant. Active-follower filter; on-demand backfill for returning users.
Pull from every followee on every read. Cache the timeline; pull only the celebrity tail at read time.
Single Kafka topic for all fan-out. Topic per priority tier; topic per fan-out class (timeline vs notifications vs search index).
Retry on failure without dedup. Idempotent recipient inserts keyed by (recipient_id, post_id).
Materialize timeline forever. Ring buffer of newest N entries; older posts pulled from authoritative store on demand.
Same threshold globally for all features. Per-feature threshold — timelines vs notifications vs search may all want different splits.
Ignore cross-region. Regionalize fan-out — only post events cross regions; the per-recipient inserts stay local.
"We'll just use Redis." Pick the storage tier per access pattern: in-memory cache for hot timelines, durable KV for celebrity store, log for the authoritative tweets.

11. Staff-level answer checklist

  • [ ] Identified the heavy-tailed follower distribution as the source of write amplification.
  • [ ] Picked a model (push / pull / hybrid) and justified it from the read:write ratio.
  • [ ] Decoupled authoritative write from fan-out via Kafka.
  • [ ] Sharded fan-out workers by recipient, not by author.
  • [ ] Specified a celebrity threshold and how it's chosen / recomputed.
  • [ ] Designed a read path that merges pre-pushed timeline with a small celebrity pull.

  • Design Google News as a Global Aggregator — fan-out into topical, regional, and personalized streams.
