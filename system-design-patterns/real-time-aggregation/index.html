Real-Time Aggregation

2026/05/31  ·  System Design Patterns
Contents
Contents

Continuous, incremental computation over an unbounded firehose of events to produce up-to-the-second counts, rollups, top-K lists, or anomaly signals — built around windows, watermarks, checkpointed keyed state, an honest delivery contract (at-least-once + idempotent sink, effectively-once, or true Kafka-to-Kafka Exactly-Once Semantics (EOS)), and approximate sketches. See the Staff System Design Glossary. Core question docs: ad click aggregator, YouTube likes counter, Datadog-style metrics and monitoring, Google News aggregator, Google Ads, typeahead suggestion, Apache Kafka.

1. The fundamental hardness

A batch system answers "how many clicks did campaign 42 get yesterday?" with SELECT COUNT(*) FROM clicks WHERE campaign_id = 42 AND day = .... The data is finite, fully arrived, and indexed. The query runs once, scans, returns. The real-time version of the same question — "how many clicks in the last five minutes, refreshed every second, for every active campaign, across one million events per second" — cannot be answered that way. The dataset is unbounded; it has no end. You cannot scan it. You cannot wait for it to stop arriving. You must compute incrementally, holding only state, never history.

That single shift — from finite scan to infinite incremental compute — drags in three problems that batch never had to solve. The first is event time versus processing time. A click happened at 12:00:01 on a phone with a flaky network; it shows up at your ingestion pipeline at 12:00:47. The user belongs to the 12:00 minute window, but processing time says 12:00 closed 47 seconds ago. If you bucket by processing time you get wrong counts. If you bucket by event time you have to decide when a window is "done" given that more events for that window might still arrive. That decision is the watermark: a heuristic timestamp saying "I believe no more events older than this will arrive." Watermarks are wrong sometimes — they are a guess about the tail of a distribution — and the system has to choose between freshness (close the window early, lose late events) and completeness (close the window late, hurt freshness). There is no free answer.

The second problem is state. A counter has no problem; it is one integer per key. But "top 100 trending hashtags across the last hour with one-minute slides" requires keeping per-hashtag counts for the last 60 windows, indexed for fast top-K, replicated for fault tolerance, checkpointed so a worker crash does not lose the last 10 minutes of computation. State can grow to terabytes. It can become hotter than the event stream itself. State management — sharded, embedded (RocksDB) versus remote, checkpointed, restorable — is the real engineering of stream processing. Everything else is filters and maps.

The third problem is at-least-once delivery and its side effects. The naive pipeline is "Kafka consumer reads click, increments a database row." Now the consumer crashes after the increment but before committing the Kafka offset. On restart it reads the same click and increments again. Or the database commit fails and you retry, double-incrementing. The framework-marketing term for the fix is end-to-end exactly-once, but it is worth being precise about what that actually buys you. Flink and Kafka Streams give you exactly-once within Kafka: source offsets, operator state, and a transactional Kafka sink commit as one atomic unit. The moment the sink is something else — Cassandra, Redis, MySQL — the engine cannot give you a real distributed transaction with that store, and you fall back to one of two real-world strategies: (a) deterministic replay plus an idempotent upsert keyed by (aggregation_key, window), which yields effectively-once output as long as the aggregation is deterministic; or (b) at-least-once with downstream dedup. The choice is never "exactly-once everywhere" by magic; it is a design decision per pipeline. For many production aggregations — dashboards, trending lists, metrics — at-least-once plus idempotent sink is the honest answer and Flink's transactional commits are overkill.

These three constraints — event-time semantics, large managed state, and at-least-once-with-idempotent-sink (or effectively-once) delivery — define the pattern. Every other tool (Flink, Kafka Streams, Spark Structured Streaming, Beam) is a different opinion about how to handle them.

One framing trap worth flagging up front: "real-time" alone does not justify Flink. A 5-minute freshness budget is comfortably met by a 1-minute Spark Structured Streaming micro-batch, or even a periodic batch job, with far less operational surface. Streaming earns its complexity when (a) the freshness budget is seconds-to-sub-second, (b) the state is large enough to make recomputation from scratch infeasible, or (c) event-time correctness against late arrivals is required. If none of those apply, the staff-level move is to say so and pick the simpler tool.

2. How to recognize it in the interview

Prompt phrases that signal this pattern

  • "real-time" or "near real-time" with freshness budgets in seconds to minutes
  • "trending" / "top N" / "leaderboard" updated continuously
  • "click counter" / "impression counter" / "view count" at scale
  • "metrics rollup" / "p99 latency over the last 5 minutes per service"
  • "anomaly detection" / "fraud signals as events arrive"
  • "live aggregation" / "live dashboard"
  • "session reconstruction" from a stream of events
  • "windowed" computation, or "rolling X-minute average"

Architectural smells

  • Read QPS for "the count" dwarfs write QPS for the underlying event ("we have 1M clicks/sec but the dashboard only shows the rollup")
  • The same number must be wrong in the same way for every reader (you cannot compute it on demand per reader; it has to be precomputed and consistent)
  • The aggregate must be cheap and fast to serve, but the events behind it are too big and too fast to scan
  • The product tolerates a small freshness lag (seconds to a few minutes) but not staleness measured in hours, ruling out daily batch
  • The product can tolerate approximate answers within a small error bound (allowing sketches like HyperLogLog (HLL) and Count-Min Sketch (CMS))

Example interview prompts where this is core

  1. Design an ad click aggregator that counts impressions and clicks per campaign per minute, with five-minute freshness, at one million events per second. (doc)
  2. Design a Datadog-like metrics and monitoring system that ingests per-host counters and gauges and supports per-tag rollups at 10-second resolution. (doc)
  3. Design a YouTube likes counter that appears instant to the liker but reaches global consistency within seconds. (doc)
  4. Design Google News as a global aggregator that surfaces trending topics across regions in near real time. (doc)
  5. Design a typeahead suggestion service whose ranking incorporates the last hour of query frequency. (doc)
  6. Design a fraud detection pipeline that flags suspicious payment sequences within seconds of the events arriving. (Related: delayed payment system, payment system.)

3. 60-second spoken framing

I would treat this as a stream processing problem rather than a database problem. Events land in a partitioned log like Apache Kafka, partitioned by the aggregation key so that all events for one campaign or one host land on one consumer. A stream processor — Apache Flink in production, Kafka Streams for lighter workloads — consumes those partitions, maintains per-key windowed state in a local RocksDB-backed state backend, and emits rollups when watermarks advance past window boundaries. State and source offsets checkpoint together so we can replay from a known-good point on failure. For unique-count metrics like distinct users, I use HyperLogLog so memory stays bounded; for top-K I use Count-Min Sketch plus a heap. The serving layer reads completed rollups from a key-value store like Apache Cassandra or a time-series store. The invariant I design around is that no committed event is double-counted and no event is silently dropped, even under worker crashes, rebalances, and late arrivals.

4. Canonical toolkit

4.1 The processing engines

Engine Sweet spot State model Watermarks Exactly-once
Apache Flink Low-latency, large state, complex windows. Managed, RocksDB backend, incremental checkpoints to remote storage. First-class, per-operator. End-to-end with transactional sinks.

The default in a staff-level answer is Flink for "real-time with non-trivial state" and Kafka Streams for "stays inside Kafka and the state is small." Spark Structured Streaming is acceptable when the interviewer mentions an existing Spark batch pipeline; do not introduce a new engine just for streaming.

4.2 Windows

Three window types cover almost every interview prompt.

Tumbling (non-overlapping, fixed size):
   [---- W1 ----][---- W2 ----][---- W3 ----]
   t=0          5             10            15

Sliding (overlapping, fixed size, step < size):
   [---- W1 ----]
        [---- W2 ----]
             [---- W3 ----]
   t=0   2    4    6   8    10

Session (gap-based, dynamic boundaries):
   user A:   ev ev ev    [gap >30s closes window]   ev ev
            [-- S1 --]                              [- S2 -]
  • Tumbling: per-minute click counts. Cheap; each event belongs to one window. The default.
  • Sliding: "five-minute moving average updated every 30 seconds." Each event belongs to multiple windows; cost multiplies. Use when smoothness matters.
  • Session: "user activity session bounded by 30 minutes of inactivity." Window boundaries depend on data, not the clock. Expensive state because you must track an open session per key.

4.3 Watermarks

A watermark is a monotonic timestamp asserting "no event with event-time earlier than W will arrive in the future." Engines emit watermarks based on observed timestamps minus a configured allowed lateness. When the watermark crosses a window's end, the window fires.

event-time line ─────────────────────────────────────►
                  e1   e2  e3        e4       e5
                  10s  12s 11s       14s      18s
                                            watermark = max_seen - 5s = 13s
                                            window [0s, 10s] closes when watermark > 10s
                                            window [10s, 20s] still open

Late events arriving after watermark either: (1) are dropped, (2) trigger a window re-fire with updated state (Flink "allowed lateness" + side outputs), or (3) go to a dead-letter side output for offline reconciliation. The choice is a product decision; surface it.

4.4 Exactly-once via checkpointed barriers — and what it actually guarantees

The Flink-style mechanism, simplified: periodic barriers are injected into the stream by sources. Each operator, when it has seen the barrier on all its inputs, snapshots its state to durable storage and forwards the barrier downstream. When the sink acks the barrier and commits its transaction, the checkpoint is complete. On failure, all operators roll back to the last completed checkpoint and sources rewind to the recorded offsets.

The honest version of this story has three asymmetries the doc-writing brain wants to flatten:

  1. Within Flink, this is exactly-once. State is consistent with the checkpoint, and replay is deterministic.
  2. Kafka-to-Kafka EOS is exactly-once. When the sink is also a transactional Kafka topic, the source-offset commit and the sink-output commit are bundled in one Kafka transaction. This is the only case where the framework itself gives you end-to-end exactly-once.
  3. Kafka-to-external-store is not. Cassandra, Redis, MySQL, S3, ClickHouse — none of these participate in a Flink/Kafka transaction. You get end-to-end correctness only if the sink writes are idempotent and deterministic: same input window produces the same output row, and the upsert is keyed by (aggregation_key, window). This is "effectively-once," not exactly-once in the textbook sense; if any operator does anything non-deterministic (random sampling, calls a wall clock, reads an external feature store at runtime), the property breaks and you can re-emit different values for the same window after a crash.

See idempotency-exactly-once.md for the dual treatment.

4.5 State backends

Backend When Cost
In-memory (heap). State fits in RAM; tiny key cardinality. GC pressure; loss on crash unless checkpointed.
Embedded RocksDB. Large state per task, local-disk speed. Disk I/O; serialization overhead; the standard choice.
Remote (S3 / HDFS) checkpoints. Durable snapshots, not live reads. Snapshot cost; recovery latency.
Remote state store (e.g., FoundationDB, Tair). Disaggregated compute and state; rare. Network on every state access.

Default: RocksDB local with incremental remote checkpoints. Mention the alternative only if the interviewer probes recovery time or compute-state disaggregation.

4.6 Approximate algorithms

When exact counting is too expensive, sketches give probabilistic answers in bounded memory.

HyperLogLog (HLL) for unique counts

Counts distinct elements in roughly 1.04/sqrt(m) relative error using m registers. With p=14 (m=16384 registers), that is ~0.8% relative error in ~16KB. Crucially, HLLs are mergeable: union of two HLLs is the HLL of the union of their inputs (it is the elementwise max of the register arrays). This means per-shard parallelism is cheap — the merge cost is O(m), independent of cardinality — but it is not literally free: registers still have to be shipped across the wire at merge time, and at very small p the relative error blows up.

The error bound matters. ~0.8% sounds tight until you remember it is relative and probabilistic: at a true cardinality of 100M unique users, the typical error band is ±800K. That is fine for "distinct users who saw this campaign today" on a dashboard; it is not fine for "distinct payers we owe a refund to," where you need an exact COUNT DISTINCT against the source-of-truth event log. The rule: use HLL when the consumer of the number is a human looking at a graph or a ranking; use exact counting when the consumer is an accounting system, a regulator, or a deduplication path that must not miss.

import math
import hashlib

class HyperLogLog:
    def __init__(self, p=14):
        # p = log2(m). p=14 -> m=16384 registers, ~0.8% error, ~16KB.
        self.p = p
        self.m = 1 << p
        self.registers = [0] * self.m

    def _hash(self, item):
        h = hashlib.sha1(str(item).encode()).digest()
        return int.from_bytes(h[:8], "big")

    def add(self, item):
        x = self._hash(item)
        idx = x >> (64 - self.p)            # top p bits select register
        w = (x << self.p) & ((1 << 64) - 1) # remaining bits
        # rank = position of leftmost 1-bit + 1, in the remaining 64-p bits
        rank = 1
        mask = 1 << 63
        while rank <= 64 - self.p and (w & mask) == 0:
            rank += 1
            mask >>= 1
        if rank > self.registers[idx]:
            self.registers[idx] = rank

    def estimate(self):
        alpha = 0.7213 / (1 + 1.079 / self.m)
        z = sum(2.0 ** -r for r in self.registers)
        return alpha * self.m * self.m / z

    def merge(self, other):
        assert self.p == other.p
        self.registers = [max(a, b) for a, b in zip(self.registers, other.registers)]

Use cases: distinct users per campaign, unique IPs per minute, distinct query strings.

Count-Min Sketch (CMS) for heavy hitters and frequency estimation

CMS is a 2-D array of d rows by w columns. Each item is hashed d times; each hash increments one cell per row. Query returns the minimum of the d cells, which is an over-estimate bounded by epsilon = e/w with probability 1 - 1/e^d.

import hashlib

class CountMinSketch:
    def __init__(self, w=2048, d=5):
        # w controls error, d controls confidence.
        self.w = w
        self.d = d
        self.table = [[0] * w for _ in range(d)]

    def _hashes(self, item):
        b = str(item).encode()
        for i in range(self.d):
            h = hashlib.blake2b(b, digest_size=8, person=str(i).encode()).digest()
            yield int.from_bytes(h, "big") % self.w

    def add(self, item, count=1):
        for i, j in enumerate(self._hashes(item)):
            self.table[i][j] += count

    def estimate(self, item):
        return min(self.table[i][j] for i, j in enumerate(self._hashes(item)))

    def merge(self, other):
        assert self.w == other.w and self.d == other.d
        for i in range(self.d):
            for j in range(self.w):
                self.table[i][j] += other.table[i][j]

Pair CMS with a small min-heap of size K to get streaming top-K: when an item's estimated frequency exceeds the heap's min, evict and insert.

Use cases: trending hashtags, hot keys for cache pre-warm, fraud heavy hitters.

Bloom filter for streaming membership

For "have I seen this event ID before in this window?" Used to deduplicate at the edges of an at-least-once pipeline. False positives mean an event might be wrongly discarded; tune m and k accordingly.

Reservoir sampling for streaming uniform sample

To keep a uniformly random sample of k items from an unbounded stream without knowing the total size:

import random

class Reservoir:
    def __init__(self, k):
        self.k = k
        self.buf = []
        self.seen = 0

    def add(self, x):
        self.seen += 1
        if len(self.buf) < self.k:
            self.buf.append(x)
        else:
            j = random.randrange(self.seen)
            if j < self.k:
                self.buf[j] = x

Use cases: log sampling, anomaly investigation, training-data sampling from production traffic.

4.7 Tumbling-window aggregation with watermark (sketch)

A self-contained Python pseudocode of how an engine handles per-key tumbling windows with an event-time watermark. Real engines do this with much more care around state and checkpointing.

from collections import defaultdict

WINDOW_SIZE_MS = 60_000
ALLOWED_LATENESS_MS = 30_000

class TumblingAggregator:
    def __init__(self):
        # key -> window_start_ms -> count
        self.state = defaultdict(lambda: defaultdict(int))
        self.watermark = 0
        self.max_event_time = 0

    def window_start(self, ts):
        return (ts // WINDOW_SIZE_MS) * WINDOW_SIZE_MS

    def on_event(self, key, event_time_ms):
        if event_time_ms + ALLOWED_LATENESS_MS < self.watermark:
            # Too late: route to side output for offline reconciliation.
            return ("late", key, event_time_ms)
        ws = self.window_start(event_time_ms)
        self.state[key][ws] += 1
        self.max_event_time = max(self.max_event_time, event_time_ms)
        # Watermark = max observed event time minus configured lateness.
        self.watermark = max(self.watermark,
                             self.max_event_time - ALLOWED_LATENESS_MS)
        return self._fire_closed_windows()

    def _fire_closed_windows(self):
        emissions = []
        for key, windows in list(self.state.items()):
            for ws in list(windows.keys()):
                if ws + WINDOW_SIZE_MS <= self.watermark:
                    emissions.append((key, ws, windows[ws]))
                    del windows[ws]
            if not windows:
                del self.state[key]
        return emissions

The interview value of this sketch is to show that you understand: per-key state, window assignment by event time, watermark derived from max observed event time minus lateness, and firing on watermark advance. In a real Flink job all of this is hidden behind keyBy(...).window(...).aggregate(...) but you must be able to explain the machinery underneath.

5. Worked example: naive → broken → fixed

Scenario

An ad click aggregator. Inputs:

  • 1,000,000 click events per second at peak, partitioned across regions.
  • 5,000,000 impression events per second.
  • 100,000 active campaigns. Long tail: 1% of campaigns get 80% of traffic.
  • Freshness target: counts visible to the billing dashboard within 5 minutes of the event.
  • Correctness target: per-minute campaign counts must be accurate within 0.1% (this is billing-grade; double-counting means refunds and fraud risk).

Naive design (the trap)

Click producer ──► Kafka topic "clicks"
                       │
                       ▼
                  Consumer group
                       │
                       ▼
   For each event: UPDATE counters SET c = c + 1
                   WHERE campaign_id = X AND minute = Y
                       │
                       ▼
                   MySQL

A junior answer. It fails on every axis.

Why it breaks

  1. Lock contention on hot rows. The top campaign gets ~50,000 clicks per second. Each UPDATE counters SET c = c + 1 WHERE campaign_id = X AND minute = Y takes an exclusive row lock and an fsync at commit; on a single InnoDB row this realistically caps at a few thousand updates per second per row even with group commit, well below the offered load. The result is not just slow updates — it is unbounded lock-wait queue growth, the consumer falling behind on Kafka, and Lock wait timeout exceeded errors as transactions are killed. The fix is to never write per-event; pre-aggregate in memory and write per-window.
  2. Duplicate increments under retry. The consumer reads a batch, increments rows, then crashes before committing Kafka offsets. On restart it re-reads the batch, increments again. There is no idempotency key on the database side. Per-campaign counts drift up by a few percent over a day. Billing is wrong. See idempotency-exactly-once.md and coordination-exactly-one-execution.md.
  3. No event-time semantics. A click that happened at 12:00:30 but arrived at 12:05:01 is credited to the 12:05 minute, not 12:00. Per-minute reports are wrong at the boundaries.
  4. No backpressure handling. When MySQL slows down, the consumer can either drop events, OOM, or block — and Kafka does not push back on producers. See backpressure-flow-control.md.
  5. Hot key kills one MySQL shard. Even sharding by campaign ID does not help: the top campaign hash always lands on one shard and dominates it. See hot-key-skew.md.
  6. Scan-based dashboard. Every dashboard query joins by minute and campaign and aggregates. Reads of the most-recent minute are hot and synchronous with the writer.

Fixed design

                   (events partitioned by campaign_id mod N)
Click producer ──► Kafka "clicks" (200 partitions)
                       │
                       ▼
       ┌───────────────┴───────────────┐
       ▼                               ▼
  Flink job (taskmanager 1)       Flink job (taskmanager N)
   - keyBy(campaign_id)            (each owns a partition subset)
   - tumbling(1 minute, event time)
   - allowed lateness 2 minutes
   - state: RocksDB local
   - checkpoint every 30s to S3
       │
       ▼
   Per-minute (campaign_id, minute, count) records
       │
       ├──► Kafka "rollups" topic (transactional sink, exactly-once)
       │       │
       │       ▼
       │   Sink consumer with idempotent upsert (PK = campaign_id, minute)
       │       │
       │       ▼
       │   Apache Cassandra: counts_per_minute(campaign_id, minute) ── PK ──
       │       │
       │       ▼
       │   Dashboard / billing reads
       │
       └──► (optional) HyperLogLog merged across partitions for unique-user counts

Key design choices and why each one is right:

  1. Partition by campaign_id. All clicks for one campaign land on the same Flink keyed-stream slot. State is local; no cross-task coordination per event.
  2. Hot key salting where needed. The top 10 campaigns are pre-identified and rekeyed to (campaign_id, salt) where salt = hash(event_id) mod 16. Sixteen sub-counters are then merged in a second Flink stage. Trades latency (one extra hop) for parallelism on the hot key. See hot-key-skew.md.
  3. Event-time tumbling windows of 1 minute with 2-minute allowed lateness. Watermarks derived from per-partition max event time minus lateness. Late events past 2 minutes go to a side-output topic that a batch job reconciles nightly.
  4. Effectively-once into Cassandra. Flink barriers checkpoint operator state and Kafka source offsets together; that gets us exactly-once within Flink and into the intermediate Kafka rollups topic. From Kafka into Cassandra, the sink consumer upserts on (campaign_id, minute) as the primary key. Because the upstream window aggregation is deterministic on replay from a checkpoint, the upsert is idempotent by construction — a retry writes the same row with the same value. The honest label for this is "effectively-once": end-to-end correctness holds as long as the aggregation stays deterministic. There is no two-phase commit between Flink and Cassandra; we get there through replay determinism plus idempotent keys, not magic.
  5. Cassandra for serving. Wide-row layout: partition key campaign_id, clustering key minute. Dashboard query for the last 60 minutes of one campaign is a single-partition range scan. Hot-campaign reads are addressed with a thin read-through cache or per-campaign replicas. See apache-cassandra.md.
  6. Pre-aggregation before the sink. Within each 1-minute window, the Flink task emits one (campaign, minute, count) record at window close, not one per event. This is the entire reason the design works: 1M events/sec collapses to ~100K rollup records/minute, which Cassandra handles easily.
  7. Approximate unique users via HLL. Per-window HLL with p=14 (~16KB, 0.8% error). HLLs across partitions are merged into a global HLL at the second stage. Cardinality is read off the merged register.
  8. Backfill and reconciliation. A nightly Spark job reads the raw click topic from cold storage (S3 export from Kafka) and recomputes exact counts; the streaming counts are reconciled against the batch result with a tolerance. Drift exceeding 0.1% pages on-call. This is lambda architecture for billing-grade correctness; pure kappa is acceptable for dashboards but rarely for money.

6. Variants and how to choose

6.1 Exact versus approximate

Need Use exact Use approximate
Billing, regulatory, settlement. Yes — must be exact. Reconcile with batch. Never.
User-facing counters (likes, views). Acceptable to be exact. Acceptable; HLL on uniques, exact on totals.
Trending / top-K dashboards. Often overkill. CMS + heap is the right answer.
Distinct-user metrics at high cardinality. Cost-prohibitive. HLL, mergeable across shards.
Per-event audit and replay. Yes — keep raw events. Sketches lose the underlying events.

6.2 Tumbling versus sliding versus session

Window Choose when Cost
Tumbling. Per-minute / per-hour reports; default. One state entry per (key, window).
Sliding. Smooth dashboards; "moving average." N x tumbling cost where N = size/slide.
Session. Per-user sessionization; activity grouping. State per open session; gap timer per key.
Global. One result over all data. Single shared state; usually do not use.

A trap: candidates choose sliding windows by default because they sound more sophisticated. The cost is real. If the product only needs a per-minute number, tumbling is the right answer.

6.3 Lambda versus kappa

Lambda:                              Kappa:
  raw events ──► Kafka                  raw events ──► Kafka
                  ├─► stream layer ──► serving         └─► stream layer ──► serving
                  └─► batch layer  ──► serving
                       (HDFS, Spark, daily/hourly)

  Two pipelines, two codebases,         One pipeline.
  reconciled to one answer.             Replay from Kafka for backfill.
  • Lambda — keep batch as truth, stream as fast approximate. Use when the stream cannot be trusted for billing or compliance, or when historical reprocessing requires logic the streaming engine cannot easily express.
  • Kappa — stream is the only path. Reprocessing means re-reading Kafka from the beginning with a new job and switching the consumer. Use when stream correctness is sufficient and you do not want two codebases. Requires long Kafka retention or tiered storage.

In an interview, propose kappa first as the conceptual default — one pipeline, one codebase, one mental model — but be honest about the operational cost. Kappa's "just replay Kafka" is fine for replaying a few hours; replaying 90 days of click data through a Flink job to backfill a schema change can take days of cluster time, during which you are running two Flink jobs in parallel (the live one and the replay), and Kafka or tiered storage has to retain the entire window. Production teams often end up with lambda anyway because the batch path also supports ad-hoc analytical SQL on the same raw events — they get the batch layer "for free" from their existing data lake, so the marginal cost of running it as a reconciliation source of truth is low. Pick kappa when the use case is genuinely append-only and bounded in history; pick lambda when (a) you need billing-grade reconciliation, (b) you need analytical SQL over the raw events anyway, or (c) your retention window for stream replay is shorter than the history you need to reprocess.

Large state, complex windows, low latency required. Apache Flink.
Existing Spark batch pipelines, micro-batch latency acceptable. Spark Structured Streaming.
Portability across cloud runners. Apache Beam on Dataflow / Flink runner.

7. Failure modes and operations

Failure Metric to watch Mitigation
Late events arriving after watermark. Late-event rate (side output). Increase allowed lateness; route to reconciliation job; surface as known correctness lag.
State size explosion (per-user session never closes). RocksDB on-disk bytes per task; checkpoint size. TTL on state; explicit session-close events; cap per-key state with eviction.
Hot key in keyed stream. Per-task event rate; CPU per task. Salt the hot key, aggregate in two stages; see hot-key-skew.md.
Checkpoint failure / timeout. Checkpoint duration p99; failed-checkpoint count. Switch to incremental checkpoints; reduce state per task; increase checkpoint timeout.
Watermark stalls (one partition idle). Per-partition watermark age. Idle-source detection; emit synthetic watermarks; alarms on watermark age > N.
Skewed partitions (one Kafka partition hot). Per-partition lag. Repartition upstream; salt before producing; see hot-key-skew.md.
Exactly-once gap at sink. Output duplicate rate (idempotency key collisions vs expected). Transactional sink; idempotent upsert; dedupe with Bloom filter at boundary.
Consumer lag growing. Kafka lag per partition. Add Flink parallelism; check downstream sink throughput; backpressure visible. See backpressure-flow-control.md.
Backfill correctness drift vs batch. Streaming-vs-batch diff per (key, day). Reconciliation job; alert at threshold; investigate window misassignment, dropped late events, or sketch error.
Schema change in event payload. Deserialization failure rate. Schema registry; forward/backward compatible Avro/Protobuf; dead-letter queue for poison events.
Recovery time too long after crash. Time-to-resume after restart. Incremental checkpoints; warmer standby tasks; smaller state per task.
Cluster restart loses last-N seconds. Recovery lag (gap between crash and restart). Acceptable if exactly-once preserves correctness; lag becomes a freshness incident, not a correctness incident.

Operational metrics to call out by name

  • Per-partition watermark age: max(now - watermark) across partitions. The single most important freshness metric.
  • Checkpoint duration and size: leading indicator of state-management problems.
  • Records-late-after-watermark: correctness budget consumption.
  • Sink commit latency: where exactly-once contracts hide their cost.
  • Restart count and recovery time: operational health of the cluster.

8. Combinations with other patterns

Real-time aggregation rarely stands alone. Common pairings:

  • Idempotency / exactly-once — the sink commit is where end-to-end exactly-once is realized. Upserts on (key, window) are the cleanest mechanism.
  • Hot-key skew — the top-K campaigns or users dominate any keyed stream. Salting and two-stage aggregation are mandatory at scale.
  • Backpressure / flow control — when sinks slow down, the stream must propagate backpressure rather than buffer indefinitely. Flink does this natively; Kafka does not push back on producers.
  • Fan-out write amplification — if rollups feed many downstream subscribers (per-region dashboards, ML feature stores, alerting), the fan-out itself becomes the bottleneck.
  • Ordering and causal consistency — event-time semantics require careful timestamp assignment; out-of-order events within a key require either reordering buffers or commutative aggregation.
  • Durability at scale — Kafka retention plus checkpoint storage define the recovery RPO/RTO.
  • Coordination exactly-one execution — at-most-once side effects (sending an alert) on top of an at-least-once stream require dedup at the boundary.
  • Search and ranking — trending-topic pipelines feed ranking; the aggregation pattern produces the features, the ranking pattern consumes them.
  • Cache stampede / read amplification — pre-aggregated rollups are themselves a cache; a stampede on the most-recent window read is the typical failure mode if the cache layer is misdesigned.
  • Strong consistency under contention — when the aggregate must be transactionally consistent with the underlying writes (e.g., available budget for an ad campaign), the streaming-aggregate-as-derived-projection model breaks; you need synchronous decrements with reservation semantics instead.

9. Likely interviewer probes

  1. "Walk me through what happens when a Flink task crashes mid-window. Where does state come from on restart? Do we lose the partial count?" - State restored from last completed checkpoint in remote storage. Kafka offsets rewound to the offsets recorded in that checkpoint. The partial window is replayed deterministically from those offsets. No partial loss; the price is recovery time, bounded by checkpoint interval plus restore latency.

  2. "How do you handle a click that arrives 10 minutes late because the user was on a plane?" - Configure allowed lateness to 2 minutes for the primary window. Anything later goes to a side-output topic. A nightly reconciliation job reads the side output, looks up the original (campaign, minute) record in Cassandra, and applies a correction. The dashboard shows "preliminary" vs "final" for the most recent hour. Trade: freshness budget plus correctness budget jointly negotiated with the product.

  3. "Why HyperLogLog over a HashSet for distinct counts? When would you not use HLL?" - HashSet is exact but memory grows with cardinality; 100M unique users per day at 50 bytes each is 5 GB per key per day. HLL is ~16 KB per key with ~0.8% error and is mergeable across partitions. Do not use HLL for billing where exactness is required, or when cardinality is low enough that a HashSet is cheaper than the HLL constant overhead.

  4. "Your top campaign gets 50K clicks/sec. Walk me through what your Flink job actually does." - Without mitigation, one keyed-stream slot is doing 50K ops/sec while others are idle — that slot becomes the bottleneck. Mitigation: salt the campaign_id into (campaign_id, salt) where salt = event_hash mod 16, parallelizing across 16 slots. Stage two re-keys by campaign_id only, sums the 16 sub-counts, and emits the final per-minute record. Latency cost: one extra stage and one network shuffle.

  5. "What is the difference between processing-time and event-time, and why does it matter?" - Processing time is when the stream operator saw the event; event time is when the event happened in the real world. Bucketing by processing time means events that arrived late get counted in the wrong bucket; reports are wrong at minute boundaries, and replaying the stream produces a different result than the original run. Event time, anchored by watermarks, is deterministic across replays and correct under network delay. The cost is the watermark machinery and the need to handle late events.

  6. "How do you guarantee exactly-once when the sink is Cassandra, not Kafka?" - Cassandra does not support multi-row transactions across the Flink commit boundary. You make the sink commit idempotent: every emitted record has a deterministic primary key (campaign_id, minute) and the operation is UPSERT. A retry writes the same row with the same value (because the upstream window aggregation is deterministic on replay from a checkpoint). End-to-end exactly-once is achieved by deterministic replay plus idempotent upsert. The cost is that all aggregations must be deterministic — no random sampling, no wall-clock-dependent logic.

  7. "What metric tells you the pipeline is healthy?" - Three: watermark age per partition (freshness), checkpoint duration and success rate (state health), and reconciliation drift vs batch (correctness). Any one of them moving is an incident class. Records-late-after-watermark and Kafka lag per partition are secondary.

  8. "What if I tell you the dashboard must reflect a click within 1 second, not 5 minutes?" - That changes the design. Tumbling windows of 1 minute do not meet 1-second freshness; you need either continuous incremental output (emit on every event, not on window close — Flink supports this with aggregate + early triggers), or you abandon window semantics and use per-key running counters with periodic flushing. The cost is that "the count" is now a moving target during the window — readers must accept eventual values that may revise downward when late events arrive or a worker re-emits after a crash.

10. Common traps → stronger moves

Trap Stronger move
"We will just increment a Redis counter on every event." Pre-aggregate in a stream processor; one rollup write per window, not per event.
Bucketing by processing time. Event-time windows with watermarks and explicit allowed lateness.
Reaching for Flink because the prompt says "real-time." Quote the freshness budget. 5-minute SLO with modest state? A periodic batch or Spark micro-batch is simpler and cheaper. Flink earns its complexity at seconds-to-sub-second freshness or with large keyed state.
Treating exactly-once as a checkbox the framework provides for free. Identify the sink contract — transactional Kafka (true EOS), deterministic-replay + idempotent upsert (effectively-once), or at-least-once with downstream dedup — and explain which one you are buying.
Sliding windows by default. Tumbling by default; sliding only when smoothness is required, and budget the multiplicative state cost.
One global counter per metric. Partition by key; aggregate per partition; merge if needed at a second stage.
Storing every event in the state for "trending top 100." Count-Min Sketch plus a small heap; bounded memory, bounded error.
HashSet for distinct counts at scale. HyperLogLog with p=14 (~16KB, ~0.8% error) and mergeability across shards.
Single-stage aggregation on a hot key. Two-stage with salting on stage one and rekey on stage two.
Ignoring late events. Allowed-lateness window plus side-output topic plus reconciliation job.
Lambda everywhere by reflex. Kappa first; lambda only when batch is the source of truth for billing or compliance.
"We will scale by adding more Kafka partitions." Partitions are an upper bound on consumer parallelism, not a magic knob; rebalance cost is non-trivial; choose the partition count for the next 1-2 years, not for today.
Glossing over recovery time. Quote checkpoint interval, restore time, and Kafka rewind window as a concrete recovery budget.
Treating the streaming layer as truth. The Kafka log is truth; the stream is a derived projection; you can rebuild the projection by reprocessing.

11. Staff-level answer checklist

When you finish answering a real-time aggregation question, the interviewer should have heard you:

  • Name the freshness budget (seconds, minutes) and the correctness budget (exact vs approximate, error bound) as explicit numbers, not adjectives.
  • Identify the partition key and explain why it gives parallelism without cross-task coordination.
  • Choose a window type (tumbling / sliding / session) and justify it.
  • Choose event-time over processing-time, define the watermark heuristic, and pick an allowed-lateness value with a reason.
  • Pick a state backend and justify it against your state-size estimate.
  • Specify the delivery contract end to end and use the right label: exactly-once within Kafka (Flink transactional sink to a Kafka topic), effectively-once into an external store (deterministic replay plus idempotent upsert), or at-least-once with downstream dedup. Saying "exactly-once" without naming the sink is the staff-level red flag.
  • Estimate state size and checkpoint cost, not just throughput.
  • Identify the hot key(s) and propose salting / two-stage aggregation.
  • Choose an approximate algorithm (HLL, CMS, Bloom, reservoir) only when the product allows it, and quote the memory/error tradeoff.
  • Propose lambda vs kappa explicitly and choose based on whether the use case is billing-grade or dashboard-grade.
  • Name the three operational metrics: watermark age, checkpoint duration, reconciliation drift.
  • Acknowledge the reprocessing story: how do you re-run the pipeline against three months of history?
  • Acknowledge schema evolution: a new field in the event payload should not break the pipeline.
  • Tie the design back to a single invariant: "no event is counted twice and no event is silently dropped within the freshness budget."
