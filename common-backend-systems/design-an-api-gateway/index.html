1. Basic Knowledge And Solution

What This Question Is Testing

An API gateway is the single entry point in front of a service mesh. Every public request hits it before reaching any backend. The hard part is being fast (sub-millisecond overhead), safe (auth, abuse, rate limiting), and dynamically configurable (routes change without restarts).

The invariant is this:

Every request entering the system is authenticated, authorized, rate-limited, and routed to the correct backend with predictable latency overhead, and the gateway never becomes the single point of failure that takes the entire platform down.

At ICT5 / L6+ level, the interviewer is testing whether you can separate the data plane (per-request work) from the control plane (config, secrets, policies), handle config changes without dropping connections, integrate with observability and rate-limiting subsystems, and reason about which features live in the gateway vs in dedicated services.

60-Second Interview Framing

Spoken version:

I would design the API gateway as a horizontally scalable data plane plus a separate control plane. The data plane terminates TLS, authenticates requests, enforces rate limits, routes to backends, and emits observability data. The control plane owns route definitions, auth policies, and secrets, and pushes versioned configurations to data plane nodes. The interesting decisions are which features live in the gateway, how config updates propagate safely, and how to keep p99 overhead minimal.

Requirements To Clarify First

Functional requirements:

  • TLS termination; protocol support (HTTP/1.1, HTTP/2, gRPC, WebSocket).
  • Authentication: API keys, OAuth, JWT, mTLS.
  • Authorization: scope/role checks; per-route policies.
  • Rate limiting and quota enforcement per identity and route.
  • Request/response transformation: header rewrites, body remapping, schema validation.
  • Routing: path-based, header-based, weight-based for canary/A-B.
  • Service discovery integration; health checks.
  • Caching responses where appropriate.
  • Per-route logging and tracing context propagation.

Non-functional requirements:

  • Single-digit ms p99 overhead.
  • Million+ QPS per gateway cluster.
  • Zero-downtime config updates.
  • Survive individual data plane node failures with no user-visible impact.
  • Tenanted: different teams own different routes and policies.

Out of scope unless asked:

  • L4 load balancing (assume an L4 LB sits in front).
  • WAF rule authoring (integrate, don't build).
  • Service mesh sidecar (related but different problem).

Scale Assumptions

Area Assumption

Latency overhead < 5 ms p99 added by the gateway.

The bottleneck is per-request CPU work (TLS handshake amortization, JWT parse, route lookup) and the control-plane fan-out of config changes.

Core Components

Component Responsibility Key detail
L4 load balancer Spreads connections across data plane nodes; anycast for region steering. DSR, ECMP, or cloud LB.
Data plane node (Envoy/NGINX-style) Per-request processing through a filter chain. Stateless; mostly CPU-bound; high concurrency via async I/O.
Filter chain Ordered list of filters: TLS, auth, rate limit, route, transform. Pluggable; per-route configuration.
Control plane API CRUD on routes, policies, secrets. Versioned, audited, role-restricted.
Config distribution Pushes config snapshots to data plane nodes. xDS-style streaming, or polled with versioning.
Secret store Holds private keys, JWT verification keys, API key hashes. Rotated; never logged.
Rate limit subsystem Per-identity-per-route counters. See dedicated rate limiter design.
Service discovery Maps logical service to backend endpoints. Health-checked; weighted.
Observability sidecar Streams logs, metrics, traces. Decoupled; never blocks request path.
WAF integration Optional, runs early in filter chain. Can be in-line or out-of-band.

Request Path (Data Plane)

For each incoming request:

  1. Accept connection. L4 LB has steered TCP to a data plane node.
  2. TLS termination. Use SNI to choose the right certificate; session resumption to amortize handshake cost.
  3. Protocol parse. HTTP/1.1, HTTP/2, or gRPC.
  4. Route match. Compiled trie or radix tree of route patterns; O(log n) or better.
  5. Authentication filter. Verify JWT signature against cached JWKS, or look up API key hash, or validate mTLS cert chain. Reject on fail.
  6. Authorization filter. Check scopes/roles against route's policy. Reject on fail.
  7. Rate limit filter. Increment counters; deny if over quota. (See dedicated rate limiter design.)
  8. Transform filter. Add/remove headers, rewrite path, validate request schema.
  9. Caching filter. If response is cacheable and present, return immediately.
  10. Backend selection. Choose backend instance from service discovery; weighted/least-loaded.
  11. Forward request. Open or reuse upstream connection; send.
  12. Response. Apply response transforms; possibly cache; emit log/metric/trace.
  13. Return to client.

Per-filter latency budgets are sub-millisecond. The whole path should add under 5 ms p99.

Config Update Path (Control Plane)

When a developer updates a route:

  1. Control plane API receives change; validates against schema and policy.
  2. Writes to versioned config store (e.g., etcd or a database with snapshots).
  3. Generates a new ConfigSnapshot containing the full state.
  4. Pushes via streaming RPC (xDS-style) to all data plane nodes.
  5. Each node atomically swaps its compiled config; in-flight requests continue on old config; new requests use the new one.
  6. Nodes ack with their applied version; control plane tracks fleet coverage.

This is the standard control-plane/data-plane pattern from Envoy/Istio. The key guarantee: a config update never drops in-flight connections.

For risky changes, use staged rollout:

  • Push to 1% of data plane nodes first.
  • Monitor error rates.
  • Continue to 10%, 50%, 100% if healthy.
  • Auto-rollback on error spike.

Interview phrase:

Config distribution is the hardest correctness problem in the gateway. The data plane must never serve stale config during a critical update like a security policy change, but it also must not drop in-flight requests during config swaps.

Authentication Approaches

Approach How it works Tradeoffs
API key Pre-shared secret; gateway hashes and looks up in cache. Simple; key rotation is operational.
JWT Client presents signed token; gateway verifies signature with public key. Stateless verification; revocation requires extra check.
OAuth2 token introspection Gateway calls auth service per request. Authoritative but adds latency; cache short-lived.
mTLS Mutual TLS at the connection; identity from cert. Strong identity; complex client setup.

Caching:

  • JWKS public keys cached per issuer with TTL.
  • OAuth introspection results cached with short TTL.
  • API key hash table refreshed via control plane push.

Rate Limiting

Delegate to a dedicated rate limit service (see separate design) rather than implementing in the gateway:

  • Gateway calls rate limiter via fast RPC (or local sidecar) per request.
  • Gateway caches recent decisions locally to reduce calls for hot keys.
  • Fallback mode: if rate limiter unreachable, fail open or closed per route policy.

Observability

Every request produces:

  • Access log: sampled at 100% by default for low-traffic routes; sampled at adaptive rates for high-traffic.
  • Metric: request count, latency histogram, status code, route, identity, region.
  • Trace span: propagates trace context (traceparent); creates a span for the gateway hop.

These are emitted asynchronously to a sidecar or local agent. The request path never blocks on observability writes.

Scaling And Partitioning

Layer Scaling
Data plane Horizontal; stateless; L4 LB load-balances connections.
Control plane Stateful (config store); replicated via consensus. Reads are heavy from data plane; writes are infrequent.
Rate limit store Sharded by key; separate service.
Secret store Replicated read-only to data plane after authentication.

Consistency Model

Concern Consistency Reason
Route configuration Eventually consistent across data plane; bounded by config push lag Acceptable for non-emergency changes.
Emergency policy changes (e.g., block key) Bounded staleness target: under 10 seconds globally Security-critical; monitor propagation.
Auth secrets Eventually consistent within rotation window Old keys remain valid during overlap.
Rate limit counters Approximate (see rate limiter design) Tolerable.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Data plane node crashes mid-request. L4 LB removes node; client retries; idempotent ops safe. Per-node error rate.
Control plane down. Data plane runs on last known config indefinitely. Config version skew; control plane availability.
Bad config rolls out. Staged rollout; auto-rollback on error spike; manual revert via versioning. Error rate per version.
Backend down. Health checks remove from pool; circuit breaker stops sending traffic; failover to backup if configured. Backend health.
TLS cert expires. Auto-renewal pipeline; alerts well before expiry. Cert expiry time per route.
Rate limiter down. Per-route fail mode; fall back to local counters. Rate limiter availability.
Auth service down. Cached JWT verification continues; introspection fails to fail-open or fail-closed per route. Auth subsystem health.
DDoS hits gateway. L4 LB or CDN absorbs; gateway has connection limits; WAF blocks signatures. Connection count, TLS handshake rate.
Slow backend cascades. Per-route timeouts; circuit breakers; bulkheads. Backend p99; circuit open count.
Tail latency from one slow filter. Per-filter latency monitoring; filter timeouts. Filter histograms.

Operational Metrics

Correctness:

  • Config version skew across data plane.
  • Time from policy change to fleet-wide propagation.
  • Reject reason distribution (auth fail, rate limit, 4xx, 5xx).

Capacity:

  • QPS per node; CPU utilization.
  • Connection count; new connections/sec.
  • TLS handshake rate.
  • Backend connection pool utilization.

Per-route metrics:

  • Latency p50/p99 per route.
  • Error rate per route.
  • Cache hit ratio per route.

Multi-Region Strategy

  • Run independent gateway clusters per region.
  • L4 anycast or geo-DNS routes users to nearest region.
  • Routes and policies are global; replicated to each region's control plane.
  • Per-region failover via DNS or anycast withdrawal.
  • For active-active backends, the gateway considers cross-region routing only when local backends are unhealthy.

Tradeoffs To Call Out

  • Filter chain depth adds latency; minimize default filters and add per-route opt-ins.
  • Centralized rate limit gives accuracy at the cost of round trip; local fallback is faster but inaccurate.
  • Aggressive config push gives faster propagation but risks correlated rollout failures.
  • TLS termination at gateway centralizes cert management but pushes CPU load to gateway nodes.
  • Caching responses helps but invalidation across nodes is hard.
  • Including auth in gateway simplifies backends but couples auth changes to gateway deployment.

Common Traps And Stronger Moves

Trap Stronger move
One monolithic config file. Versioned snapshots with staged rollout.
Synchronous logging on request path. Async to sidecar/local agent.
Stateful gateway nodes (sessions). Stateless data plane; state in dedicated services.
Per-request auth introspection. Cached JWT verification; introspection only when needed.
One filter chain for all routes. Per-route filter list; lazily compiled.
Single control plane node. Replicated control plane with consensus.

Staff-Level Answer Checklist

  • Separate data plane from control plane.
  • Versioned config snapshots with atomic swaps.
  • Stateless data plane; horizontal scale.
  • Auth, rate limit, observability as filters with per-route config.
  • Staged rollout for risky changes.
  • Per-route timeouts, retries, circuit breakers.
  • Async observability.
  • Multi-region as independent clusters.

Interview Phrase

An API gateway is the classic data-plane / control-plane split. The data plane is stateless and stays fast by avoiding synchronous calls to anything that can fail; the control plane owns truth and propagates versioned snapshots. Every filter has a latency budget, and the gateway is never the bottleneck.

2. Simulated Interview

Interviewer: Design an API gateway.

Candidate: I will design two pieces: a stateless data plane that handles per-request work (TLS, auth, rate limit, route, transform), and a control plane that owns routes, policies, and secrets and pushes versioned config to the data plane. The invariant is sub-millisecond filter overhead and never being the single point of failure.

Interviewer: How does config propagation work?

Candidate: Control plane API writes to a versioned store. It generates a ConfigSnapshot containing the full state, signs it, and pushes via streaming RPC (xDS-style) to every data plane node. Each node atomically swaps its compiled config; in-flight requests continue on the old config, new requests use the new one. Nodes ack with their applied version, and the control plane tracks fleet coverage.

Interviewer: A developer pushes a bad route. How do you not melt the platform?

Candidate: Staged rollout. The change deploys first to 1% of data plane nodes; we monitor error rates for a few minutes. If healthy, expand to 10%, 50%, 100%. If error spike detected, auto-rollback to the previous version. Developers can also manually revert via the versioned config store; every snapshot has an id.

Interviewer: How fast can you propagate an emergency security change like "block this API key right now"?

Candidate: Target under 10 seconds globally. The control plane pushes immediately via streaming RPC; each data plane node applies on receipt. We monitor a config_propagation_latency metric explicitly. For blocking known-bad credentials, we maintain a deny list that propagates separately from the route config and is consulted on every auth check.

Interviewer: How does authentication work?

Candidate: Multiple modes, per route. JWT is most common: client sends a token, the gateway verifies the signature against a cached JWKS public key. JWKS cache is refreshed periodically. For API keys, the gateway hashes and looks up in a cache backed by the control plane. For mTLS, the L4 LB or gateway terminates and extracts the client cert. OAuth introspection is the most expensive — we call the auth service and cache the result for short TTL.

Interviewer: Where does rate limiting live?

Candidate: I delegate to a dedicated rate limit service rather than build it into the gateway. The gateway makes an RPC to the limiter per request. For hot keys, the gateway has local approximation: a small in-process token bucket that absorbs the load and reconciles asynchronously with the central limiter. If the limiter is unreachable, the route's fail-mode (open or closed) decides.

Interviewer: What about TLS at this QPS?

Candidate: TLS handshake is the most expensive thing the gateway does. We use session resumption and TLS 1.3 zero-round-trip resumption where possible. Hardware acceleration where available. Per-route certificate selection via SNI. Cert rotation is automated; the secret store pushes new certs, the gateway loads them without restart.

Interviewer: A backend service is slow — how do you contain the damage?

Candidate: Per-route timeout: if the backend doesn't respond in N ms, the gateway returns 504 and frees the connection. Circuit breaker: after a threshold of failures or slow responses, the gateway stops sending traffic to the unhealthy backend instances. Bulkhead: connection pool per backend is bounded, so one slow backend can't consume all the gateway's concurrency.

Interviewer: How do you handle WebSockets and long-lived connections?

Candidate: Different filter chain. The auth filter runs on connect; rate limiting may run on the initial upgrade and possibly per message. The gateway holds a persistent upstream connection to the backend; client disconnects propagate. Connection draining: during deploy, the gateway stops accepting new WebSocket connections and lets existing ones complete or migrate.

Interviewer: How do you observe a billion requests per day?

Candidate: Async observability: each request emits a structured event to a sidecar or local agent that batches and ships to the observability backend. We sample access logs at adaptive rates based on traffic volume; metrics are always emitted but pre-aggregated; traces are sampled at low percentages with tail-based sampling on errors. The request path never waits on observability.

Interviewer: Multi-region?

Candidate: Independent gateway clusters per region. Geo-DNS or anycast steers users to the nearest region. Routes and policies are global state; the control plane replicates them per region. A region can fail over by withdrawing anycast advertisements or shifting DNS. Cross-region backend calls are avoided unless explicitly configured for failover.

Interviewer: What lives in the gateway vs in dedicated services?

Candidate: In the gateway: TLS, auth verification, routing, rate-limit enforcement (with cached decisions), request/response transformations. In dedicated services: identity provider, authoritative rate limit store, service discovery, WAF rule engine, observability backends. The principle: keep the gateway lean and fast, push heavy or specialized logic to specialized services.

Interviewer: Summarize.

Candidate: Stateless data plane plus separate control plane. Versioned config snapshots distributed via streaming RPC; atomic config swap; staged rollouts for safety. Auth, rate limiting, transforms as filters with per-route configuration. Async observability. Per-route timeouts and circuit breakers contain backend failures. Multi-region as independent clusters with replicated control plane. The hardest correctness problem is config distribution; the hardest performance problem is keeping per-filter overhead sub-millisecond.

3. Practice Notes

Five-Minute Version

  1. Clarify: TLS, auth, rate limit, route, observe; per-tenant config.
  2. Invariant: sub-ms overhead, never a SPOF.
  3. Architecture: stateless data plane + control plane.
  4. Filter chain: TLS, auth, rate limit, route, transform, observe.
  5. Config: versioned snapshots, atomic swap, staged rollout.
  6. Delegate rate limit and identity to dedicated services.
  7. Per-route timeouts, circuit breakers, bulkheads.
  8. Multi-region: independent clusters; geo-routing.

One-Minute Closeout

An API gateway is the canonical data-plane / control-plane separation. The data plane is stateless, runs a per-route filter chain, and never blocks on anything that can fail. The control plane owns truth and propagates versioned snapshots with staged rollout. Auth and rate limiting integrate as filters that call dedicated subsystems. Per-route timeouts and circuit breakers contain backend failures. The hardest part is config distribution speed for security-critical changes.

Best Interview Phrases

  • "Data plane stays fast by never synchronously calling anything that can fail; failures fall back to cached or local state."
  • "Config distribution uses versioned snapshots with atomic swap so in-flight requests never see partial config."
  • "Auth verification is cached; only introspection touches the auth service on the request path."
  • "Rate limiting is delegated; the gateway calls a dedicated service with a local approximation fallback."
  • "Per-route timeouts and circuit breakers stop one slow backend from cascading."
