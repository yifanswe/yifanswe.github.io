1. Basic Knowledge And Solution

What This Question Is Testing

Google News aggregates articles from thousands of publishers globally, clusters them into stories, ranks for relevance and freshness, personalizes per user. The hard part is real-time clustering of articles into coherent stories across languages, freshness within minutes, and personalization at planet scale.

The invariant is this:

Every article published is crawled (or pushed) within bounded freshness, deduplicated, clustered with other articles about the same story, ranked, and surfaced to relevant users; story clusters remain coherent (no article in two stories at a time); breaking news appears within minutes.

At ICT5 / L6+ level, the interviewer is testing whether you can design real-time crawl + ingest, article clustering (NLP-based), per-user ranking, multilingual handling, and personalization.

60-Second Interview Framing

Spoken version:

I would design Google News as a pipeline from publisher → crawler → article processor → clustering engine → ranking → personalized feed. Articles are crawled in real time, deduplicated, clustered into stories via NLP similarity, ranked by recency, source authority, and user relevance. Multiple languages handled separately. Personalization combines user interest with editorial. The interesting decisions are clustering, real-time ingest, ranking, and personalization.

Requirements

Functional requirements:

  • Aggregate articles from publishers (RSS, sitemap, crawled, partner feed).
  • Cluster articles about the same story.
  • Show top stories per topic, region, user.
  • Personalize feed.
  • Multilingual.
  • Topic categorization (politics, tech, sports, etc.).

Non-functional requirements:

  • Article ingested → visible within minutes.
  • High availability.
  • Story clusters stable but evolve as new articles arrive.
  • Multi-region.

Out of scope unless asked:

  • Editorial UI tools.
  • Subscription / paywalls.
  • Newsletter delivery.

Crawl / Ingest

Multiple sources: - RSS / sitemap polling: check publishers' feeds; politeness similar to web crawler design. - Push: partner publishers send notifications. - Discovery: find new sources via web crawler signals.

Goal: minute-level freshness for top publishers; hourly for long tail.

Article Processing

For each new article:

  1. Extract main content (boilerplate removal).
  2. Detect language.
  3. Compute embedding (text → dense vector via transformer).
  4. Classify topic (multi-label).
  5. Score publisher authority.
  6. Hand to clustering.

Clustering Engine

Real-time online clustering.

Approach:

  1. For each new article, compute embedding.
  2. Compare to centroids of recent stories (last N hours).
  3. If similarity > threshold to some story's centroid, join that story.
  4. Else, create new story with this article as seed.
  5. Update story centroid.
  6. Stories age out after N days; closed.

For multilingual clustering: use cross-lingual embeddings or first translate to canonical language.

Edge cases: - Article fits multiple stories: pick highest similarity (or sub-cluster). - Story splits over time as articles diverge: rare; periodic re-cluster job handles. - Story merge: two clusters discovered to be the same → merge member articles.

Interview phrase:

Online clustering: each new article either joins an existing story by embedding similarity or starts a new one. Centroids update; stories age out. Periodic re-cluster fixes drift.

Story Ranker

For each topic / region / user, rank stories:

Signals: - Recency. - Authority of sources (high-reputation publishers boost). - Number of articles in story (signal of importance). - Engagement (user clicks). - User interest match. - Editorial signals.

Multi-stage: cheap pre-rank → ML ranker on top.

Personalization

User profile contains interests, region, language, read history. Online ranker incorporates: - Topic affinity (sports vs finance). - Source preference. - Local relevance. - Click history.

Daily batch updates user features.

Editorial Overrides

For breaking news or curation, editorial team can: - Promote a specific story to top. - Mark stories sensitive. - Adjust source authority.

Editorial state strong; visible immediately.

Multilingual

Two strategies: - Per-language pipelines: each language has its own clustering and ranking. Users see content in their preferred languages. - Cross-language: embeddings unified; same story across English and Spanish clusters together. Display in user's language.

Most production systems combine: cluster within language but cross-language for the same story can show both editions.

Operational Metrics

Correctness:

  • Cluster purity (sampled).
  • Time from publish to surface.
  • Misinformation rate.

Capacity:

  • Ingestion lag.
  • Clustering throughput.
  • Ranking p99.

  • DAU.
  • Click-through rate.

Tradeoffs To Call Out

  • Real-time clustering trades some accuracy for freshness.
  • More aggressive clustering threshold = more merging vs more splits.
  • Personalization improves engagement but risks filter bubble.
  • Cross-language unified clustering is complex but unified UX.
  • Authority weighting penalizes new publishers.
  • Editorial vs algorithmic balance.

2. Simulated Interview

Interviewer: Design Google News.

Candidate: Pipeline: publisher → crawler / ingest → article processor (parse, embed, classify) → online clustering engine → ranking → personalized feed. Stories are clusters of related articles, formed online by embedding similarity. Per-language pipelines with cross-language unification. Editorial overrides for breaking news. The invariant is real-time freshness with coherent story clusters and per-user personalized ranking.

Interviewer: Walk me through an article ingestion.

Candidate: Crawler fetches new article from publisher (RSS poll or push). Article processor extracts main text (boilerplate removal), detects language, computes embedding via transformer, classifies topic, scores publisher authority. Clustering engine compares embedding to centroids of recent stories (last N hours); if similarity above threshold, join that story; else, start a new story. Update story centroid. Story ranker re-scores affected stories. Visible in feed within minutes.

Interviewer: Clustering — algorithm?

Candidate: Online clustering. Maintain centroids for recent stories (rolling window, last 1-3 days typical). For each new article, compute cosine similarity to all active centroids; if max > threshold, assign to that story and update its centroid. Else, new story with this article as seed. Periodic re-cluster job handles drift (every few hours): re-evaluate borderline articles, merge accidentally-split clusters.

Interviewer: Two stories should be merged.

Candidate: Periodic re-cluster detects clusters whose centroids are very similar. Merge: union member articles, recompute centroid, retire one story_id (or pick canonical). Editorial can manually merge or split if needed.

Interviewer: Ranking signals?

Candidate: Multi-signal. Recency (newer better). Source authority (high-reputation publishers boost). Story size (more articles signals importance, with diminishing returns to avoid clickbait). Engagement (CTR, time spent). User interest match. Editorial boost. Local relevance. Multi-stage: cheap pre-rank prunes; ML ranker scores top candidates.

Interviewer: Personalization.

Candidate: User profile: topic interests (sports, finance), preferred sources, region, language, read history. Online ranker blends global story score with user-affinity score. Diversity injection prevents filter bubble — surface stories outside the user's typical interests periodically. Daily batch updates user features.

Interviewer: Multilingual.

Candidate: Per-language pipelines for clustering (English articles cluster with English; Spanish with Spanish). For cross-language unification, use multilingual embeddings: a Spanish article about the same event as an English article scores similar in shared embedding space. Or first translate to canonical language for clustering. Display in user's preferred language; if no article in their language, can show top article in another.

Interviewer: Source authority.

Candidate: Per-publisher reputation score. Based on: domain authority, editorial standards, publication history, factuality signals, user feedback. Higher-authority sources surface higher in rankings. Affects new publishers — they need time to build authority. Editorial can curate authority for breaking situations.

Interviewer: Misinformation.

Candidate: Source authority is the primary mitigation. Low-authority sources downranked. Specific misinformation patterns (deepfakes, well-known false claims) flagged by ML classifiers. Editorial intervention for emerging cases. Fact-check labels overlaid. Users can flag; reports feed editorial review.

Interviewer: Breaking news.

Candidate: Real-time pipeline ensures fast surfacing — articles ingested within minutes. New stories about breaking events form quickly as multiple publishers cover. Editorial may promote a story to top. Ranker boosts stories with high publish velocity. Push notifications can alert users to top stories.

Interviewer: Multi-region.

Candidate: Crawlers regional; partner pushes accepted globally. Story store replicated globally for cross-region viewing. User profile in home region. Ranking per region (regional interest weight) with user's language preference.

Interviewer: Privacy / personalization.

Candidate: User opt-out of personalization gives a non-personalized algorithmic feed. Read history bounded retention. Anonymized signals for global trends. Per-user opt-outs respected.

Interviewer: Summarize.

Candidate: Pipeline: crawl → process → cluster → rank → personalize. Online clustering groups articles into stories via embedding similarity. Multi-stage ranking with signals (recency, authority, engagement, personalization). Per-language clustering + cross-language unified for multilingual coverage. Editorial overrides for breaking news. Source authority weights against misinformation. Multi-region with global story replication. The hardest correctness problem is keeping clusters coherent at scale; periodic re-cluster handles drift.
