1. Basic Knowledge And Solution

What This Question Is Testing

Facebook Messenger (or WhatsApp / iMessage) is real-time 1:1 and group messaging. The hard part is delivering every message to every participant exactly once with correct ordering, working across devices, surviving offline recipients, and providing read receipts / typing indicators / presence without making the system fragile.

The invariant is this:

Every sent message is durably stored, delivered to every conversation participant at least once on each of their active devices, displayed in a consistent order across devices, and remains available offline; deletes and edits are eventually visible everywhere; read receipts and presence are best-effort but never claimed when not true.

At ICT5 / L6+ level, the interviewer is testing whether you can separate the messaging substrate (durable conversation log) from the delivery substrate (online push, offline fetch), handle multi-device sync, design end-to-end encryption (if asked), and reason about group fan-out vs 1:1 patterns.

60-Second Interview Framing

Spoken version:

I would design Messenger around per-conversation append-only logs as the source of truth. Clients sync from their position in the log; a real-time push channel notifies them of new messages when online. Multi-device sync is solved by per-device cursors and a shared mailbox. For groups, the conversation log is still append-only; we just have more participants pulling. The interesting decisions are per-conversation ordering, multi-device sync, delivery acknowledgment, and end-to-end encryption.

Requirements To Clarify First

Functional requirements:

  • Users send 1:1 and group messages (text, media, reactions).
  • Messages must be persisted and available offline.
  • Read receipts, typing indicators, delivery status.
  • Presence (online / last seen).
  • Multi-device support (phone + web simultaneously).
  • Edit and delete messages.
  • Message history syncs to new devices.
  • Optional: end-to-end encryption.

Non-functional requirements:

  • Sub-second message delivery for online users.
  • Months/years of message retention.
  • Strong per-conversation ordering.
  • Exactly-once user-visible delivery (with at-least-once underneath).
  • Multi-region; messages cross continents in seconds.
  • High availability for the messaging substrate.

Out of scope unless asked:

  • Voice/video calls (WebRTC; different problem).
  • Stories/status (separate).
  • Search across all messages (huge separate problem).
  • E2E encryption (mention; depth on request).

Scale Assumptions

Area Assumption
Users 2B+ MAU.
Messages/day 100B+ globally.
Average user Tens of conversations, hundreds of messages/day.
Group size Up to thousands (WhatsApp 1024+; Messenger varies).
Peak Millions of messages/sec at peak.
Devices per user 2-3 typical; up to 4-5 in some cases.
Latency Online delivery sub-second; offline pickup on reconnect.
Retention Indefinite by default.

The bottleneck is per-conversation throughput for very large group chats and the global presence/typing indicator overhead.

Core Product Model

Layer Responsibility Requirement
Conversation log Append-only ordered messages per conversation. Source of truth; strong per-conversation order.
Sync substrate Each device pulls from its cursor in each conversation it's in. At-least-once; cursor advances on ack.
Real-time push Notifies online clients of new messages to reduce poll lag. Best-effort; clients fall back to pull.
Presence Online / last-seen / typing. Best-effort; ephemeral; not stored long.
Receipts Delivered / read per device. Per-device state; conversation-aggregated.

Core Components

Component Responsibility Key detail
Send service Receives outbound message; writes to log; broadcasts. Idempotent on client-side message id.
Conversation log store Append-only per-conversation messages. Sharded by conversation_id; strong per-conv order.
Per-device inbox cursor Tracks last delivered seq per conversation per device. Updated by ack from device.
Push channel Persistent connection (WebSocket / MQTT) to online devices. Used for new-message notification + presence.
Pull / sync service Returns messages since cursor when device polls. Bounded batch; pagination.
Receipt service Stores delivered/read receipts per device per message. Aggregated for sender display.
Presence service Online/offline/typing state. In-memory with TTL; not durable.
Media service Image/video upload and serve. Separate from message metadata.
E2E encryption (optional) Per-conversation key management; per-message encryption. Server only stores ciphertext.

Write Path: Send a Message

  1. Client POSTs with client_message_id (idempotency key).
  2. Send service checks idempotency; if dup, returns prior result.
  3. Assign server-side message_id and a seq monotonic per conversation.
  4. Persist to conversation log store (replicated, durable).
  5. Update conversation's last_message_seq.
  6. Emit event to per-conversation stream.
  7. For each recipient device: - If online (presence service says so), push notification via WebSocket containing message metadata or full content. - If offline, the device fetches on next reconnect via pull.
  8. Return success to sender with message_id and seq.

The send response does not wait for delivery acks; the sender sees their message ack'd as soon as the log write completes.

Read / Sync Path

Each device maintains a cursor per conversation it's in.

On connect / wakeup:

  1. Device sends current cursor for each conversation (or one global sync cursor).
  2. Sync service queries log store for messages with seq > cursor per conversation.
  3. Returns messages, capped at batch size; client paginates if more.
  4. Device acks delivered_seq per conversation.
  5. Server updates DeviceCursor.

For real-time push:

  1. Client maintains WebSocket connection.
  2. New messages in conversations the device participates in are pushed.
  3. Push contains either full content (small messages) or metadata (so client pulls full).
  4. Client acks; cursor advances.

If WebSocket disconnects, the device falls back to pull on reconnect — no messages lost.

Interview phrase:

The push channel is an optimization, not the source of truth. The conversation log is. Clients can always recover by pulling from their cursor.

Per-Device State And Sync

A user with phone + web + tablet has three devices. Each must:

  • See the same conversation history.
  • Send messages from any device.
  • Receive new messages on all online devices.
  • Have independent read state (read on phone doesn't read on web).

Implementation:

  • Each device has its own DeviceCursor per conversation.
  • Sending from any device persists to the conversation log with the user as sender.
  • All recipient devices (including the sender's other devices) get the message.
  • Read receipts are per-device but aggregated for display.

New device sign-in:

  • Bootstraps cursor to "newest" or fetches recent history depending on policy.
  • Subsequent syncs are incremental.
  • For E2E encryption, key exchange to new device is a complex sub-problem (Signal Protocol-style).

Group Messaging

Same log structure, just more participants.

  • Conversation log is per group; one append-only stream.
  • Each participant's device has cursor over this log.
  • Send fan-out: write once to the group log; recipient devices pull or push from the same source.
  • Strong order per group (seq).

Large groups (1000+ members): - Push fan-out cost scales with group size and online member count. - Pre-compute online members; push only to them. - Offline members pull on reconnect.

Interview phrase:

Groups don't multiply storage; they multiply read amplification. One write, many readers — and reads are cheap because they're append-only log scans from a cursor.

Read Receipts And Delivery Status

Each message has potentially many receipts: delivered and read per recipient device.

Two views:

  • Sender view: "My message has been delivered to / read by N of M people."
  • System: Store receipts per (message_id, user_id, device_id).

For groups with thousands, storing all receipts is heavy. Some platforms:

  • Aggregate to counters: delivered_count, read_count.
  • Show "Seen by all" once all participants ack.
  • Skip per-recipient detail for very large groups.

Presence And Typing

Ephemeral, best-effort, in-memory.

  • Online state: TTL'd entry in a presence store, refreshed by WebSocket keepalive.
  • Typing: short-lived event (a few seconds), broadcast to conversation participants.
  • Last-seen: updated on disconnect, with privacy controls.

Not durable; restarts may lose presence state momentarily. Clients accept this.

Ordering

Per-conversation order is critical. Cross-conversation order doesn't matter.

  • Sequence numbers assigned per conversation by the log store (atomic increment or single leader).
  • All participants see the same order.
  • Clock skew at send time is irrelevant; server assigns seq and sent_at.

For multi-region: the conversation has a home region (typically the region of the first participant or the creator). Sends route to the home region for sequencing; replicated to other regions.

End-to-End Encryption (If Asked)

  • Each user has identity keys (Signal Protocol: identity key, signed pre-key, one-time pre-keys).
  • New conversations use X3DH key agreement.
  • Per-message keys via Double Ratchet for forward secrecy.
  • Server stores only ciphertext.
  • Server cannot read messages but still routes by conversation_id and orders by seq.
  • Multi-device: each device has its own key pair; sender encrypts to each recipient device separately, or uses Sender Keys for groups.

Scaling And Partitioning

Data Partition key Why
Conversation log conversation_id hash Strong per-conv ordering; cheap reads.
Device cursors user_id hash Per-user state.
Receipts conversation_id + message_id Sender-side aggregation.
Presence user_id hash Ephemeral; can be in Redis.
Push connections Sticky to a push gateway by device_id Maintain WebSocket affinity.

Consistency Model

Layer Consistency Reason
Conversation log writes Strong per-conversation (single seq order) Per-conv ordering required.
Cross-conversation order Eventually consistent Doesn't matter.
Device cursor Strong on update Avoid lost messages.
Receipts Eventually consistent UX hint.
Presence Eventually consistent, TTL'd Inherently ephemeral.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Push channel drops; user offline. Pull on reconnect from cursor; no messages lost. Reconnect rate; pull volume.
Send retried; duplicate sent. Idempotent on client_message_id. Duplicate send rate.
Conversation log shard hot (huge group). Sub-shard within conversation; or accept and scale shard. Per-shard QPS.
Out-of-order message arrival on client. Client buffers and applies in seq order. Client reorder count.
Cursor corruption / loss. Server keeps authoritative cursor; client can request full sync. Sync error rate.
Multi-device send race. Per-conversation server-side seq prevents conflict. Send conflict rate.
Group fan-out backpressure. Online-only push; pull for offline; backoff if overload. Push queue depth.
Presence service outage. Treat all as offline; messages still deliver via pull. Presence service uptime.
Receipt explosion (large groups). Aggregate to counters; cap detail to 1:1 and small groups. Receipt storage growth.
Cross-region latency on send. Route to conversation home region; accept latency for sender. Cross-region send count.

Operational Metrics

Correctness:

  • Lost message rate (must be near zero; cursors detect).
  • Out-of-order delivery rate.
  • Duplicate user-visible deliveries.
  • Receipt accuracy.

Capacity:

  • Send QPS by conversation type.
  • Push fan-out rate.
  • WebSocket connection count.
  • Sync request rate.

Business:

  • DAU; messages per user per day.
  • Group size distribution.
  • Read latency / delivery latency.

Multi-Region Strategy

  • Conversation has a home region (typically the one with most active participants).
  • Writes route to home region for sequencing.
  • Log replicated to other regions for local reads.
  • Push gateways are regional; the home region notifies remote push gateways for participants there.
  • Presence is regional with cross-region propagation as needed.

Tradeoffs To Call Out

  • Per-conversation strong ordering simplifies UX but limits per-conversation throughput.
  • Pulling from a cursor is robust but requires the cursor to be reliably tracked.
  • Storing full message content vs just metadata in pushes balances bandwidth vs latency.
  • Per-device receipts give precision but bloat storage for large groups.
  • Presence accuracy vs network cost: more frequent heartbeats = fresher presence = more battery.
  • E2E encryption prevents server-side moderation; tradeoff with safety features.

Common Traps And Stronger Moves

Trap Stronger move
Treating push as the delivery mechanism. Conversation log is source of truth; push is optimization.
Single global seq for all conversations. Per-conversation seq; no global coordination.
Per-recipient mailbox storage. One per-conversation log; cursors per device.
Synchronous ack from all recipients before sender sees sent. Sender sees sent on log write; ack is per-recipient async.
Ignoring multi-device. Device cursors per conversation; key management per device.
Receipts as full records for large groups. Aggregate counters for >N members.

Deep Dives Interviewer May Choose

Deep dive Strong answer shape
Multi-device sync Per-device cursor; new-device bootstrap; key exchange if E2E.
Group fan-out One log; online push + offline pull; pre-compute online list.
E2E encryption Signal protocol; Double Ratchet; Sender Keys for groups.

Staff-Level Answer Checklist

  • Per-conversation append-only log as source of truth.
  • Per-device cursors for sync.
  • Push is optimization; pull from cursor is fallback.
  • Idempotent send with client_message_id.
  • Multi-device handled via independent cursors.
  • Group fan-out is one write, many cursors.
  • Presence and typing are ephemeral, separate from log.
  • E2E encryption awareness if asked.

Interview Phrase

Messenger is a per-conversation append-only log with cursors. The log is the source of truth; push is an optimization. Multi-device sync is just per-device cursors over the same log. Groups don't multiply storage; they multiply readers, which is cheap because reads are cursor-based log scans.

2. Simulated Interview

Interviewer: Design Facebook Messenger.

Candidate: I will design around a per-conversation append-only log as the source of truth, per-device cursors for sync, and a real-time push channel for online delivery. The invariant is exactly-once user-visible delivery per device, with per-conversation strong ordering. I will walk through send, sync, multi-device, and groups.

Interviewer: Walk me through sending a message.

Candidate: Client POSTs with a client-generated message_id for idempotency. Server checks for duplicate. If new, assign a server-side seq monotonic per conversation, persist to the conversation log (replicated), update conversation metadata, emit event. For each recipient device, if presence says online, push via WebSocket; if offline, do nothing — they'll pull on reconnect. The sender's other devices also receive the push. Response goes back to sender as soon as the log write commits.

Interviewer: How does the recipient device know about new messages?

Candidate: Two paths. Online: WebSocket push from the server contains the message (or metadata). The device renders and acks. Offline: on reconnect, the device sends its cursor — last delivered seq per conversation — and pulls everything since. The push channel is just an optimization for low-latency online delivery; the conversation log plus cursor is the durable mechanism.

Interviewer: Multi-device — user on phone and web simultaneously.

Candidate: Each device has its own cursor per conversation. When a message arrives, it pushes to both online devices independently. If they read on phone, the phone acks read_seq; the web cursor's read_seq is separate. Sending from any device writes to the same conversation log with the user as sender; all recipient devices, including the sender's other devices, see it.

Interviewer: How do you handle ordering across devices?

Candidate: Server assigns per-conversation seq. All devices see the same seq. Client buffers and applies in seq order. Clock skew at send time doesn't matter because the server assigns seq and timestamp on write. Cross-conversation order isn't guaranteed; doesn't matter for UX.

Interviewer: Group with 1000 members.

Candidate: Same log structure. One write per send; the conversation log captures it once. Each member's devices have cursors over this log. Push fan-out scales with the number of online members. We pre-compute the online member list before fan-out and only push to online devices; offline members pull on reconnect. For huge groups, we may have a separate push fan-out service that batches.

Interviewer: Read receipts at that scale?

Candidate: For 1:1 and small groups, store per-recipient receipts. For large groups, aggregate to counters: delivered_count, read_count. Display "Seen by 47 of 1000" or "Seen by everyone" once all ack. We stop tracking per-recipient detail above some threshold to keep storage bounded.

Interviewer: Presence and typing?

Candidate: Ephemeral, in-memory in a Redis-like store with TTL. Online state refreshed by WebSocket keepalive. Typing event is short-lived (a few seconds), broadcast to conversation participants via push. Last-seen is updated on disconnect, with privacy controls. None of this is durable; service restart may lose state, which is OK.

Interviewer: End-to-end encryption?

Candidate: Signal Protocol. Each user has identity keys and pre-keys. X3DH key agreement starts a conversation; Double Ratchet provides forward secrecy. Per-message keys; server stores only ciphertext. Multi-device: each device has its own key pair, and either sender encrypts to each recipient device or uses Sender Keys for groups. The server still routes by conversation_id and orders by seq, even though it can't read content.

Interviewer: Multi-region?

Candidate: Each conversation has a home region — usually the one with the most active participants or where it was created. Writes route to home region for sequencing. The log is replicated to other regions for fast local reads. Push gateways are regional; the home region notifies remote push gateways which forward to local devices. Cross-region delivery latency is mostly the replication lag — seconds.

Interviewer: What happens if the conversation home region fails?

Candidate: Failover to a replica region. The seq is preserved via consensus replication. Brief unavailability for writes while failover happens. Reads continue from replicas. After failover, the new region takes over sequencing.

Interviewer: What about message edits and deletes?

Candidate: Edit is a new event in the log: MessageEdited(message_id, new_content, edited_at). Clients receive it and update their UI. Delete is similar: MessageDeleted(message_id, deleted_for_all). The original message in the log is marked or replaced with a tombstone. Clients respect the latest state.

Interviewer: Summarize.

Candidate: Per-conversation append-only log is the source of truth; per-device cursors track delivery. Push is an optimization for online delivery; pull from cursor is the fallback. Multi-device sync is per-device cursors over the same log. Groups are one log with many readers. Send is idempotent via client_message_id. Per-conversation strong ordering via server-assigned seq. Presence and typing are ephemeral. Multi-region uses per-conversation home regions with replicated reads. The hardest cases are large groups and multi-device sync, both handled by the log + cursor design.

3. Practice Notes

Five-Minute Version

  1. Clarify: real-time, multi-device, groups, retention, encryption.
  2. Invariant: per-conv ordered, exactly-once user-visible, durable.
  3. Per-conversation append-only log; per-device cursor.
  4. Push for online; pull from cursor for offline.
  5. Multi-device: independent cursors over same log.
  6. Group: one log, many cursors; online push + offline pull.
  7. Receipts: per-device for small, aggregate counters for large.
  8. Presence ephemeral; E2E optional via Signal.

One-Minute Closeout

Messenger is a per-conversation append-only log with per-device cursors. The log is the source of truth; push is an optimization for online delivery. Multi-device sync is just per-device cursors over the same log. Groups are one log with many readers. Send is idempotent; ordering is per-conversation via server-assigned seq. Presence and typing are ephemeral. Multi-region uses per-conversation home regions. End-to-end encryption uses Signal Protocol with per-device keys.

Best Interview Phrases

  • "The conversation log is the source of truth; push is an optimization."
  • "Per-device cursors over a shared log solve multi-device sync."
  • "Groups multiply readers, not storage; reads are cheap from a cursor."
  • "Server-assigned per-conversation seq gives consistent ordering across devices."
  • "Receipts at large group scale aggregate to counters; per-recipient detail is for 1:1 and small groups."
