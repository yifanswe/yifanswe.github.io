1. Basic Knowledge And Solution

What This Question Is Testing

Miro is real-time collaborative drawing and diagramming on an infinite canvas. The hard part combines Google Docs (concurrent editing convergence) with spatial canvas semantics: many shape objects, cursor presence, zoom levels, and undo across users.

The invariant is this:

Every operation performed by any user (create, move, edit, delete a shape) eventually appears on every other user's view of the same board, converging to the same canonical state; concurrent operations are merged deterministically; cursors and selections are visible to others in real time; the board can grow to thousands of objects without slowing edits.

At ICT5 / L6+ level, the interviewer is testing whether you can design CRDT-based or OT-based convergence on a shape-graph data model, handle real-time presence, efficient rendering of large boards, undo/redo across users, and reason about board sharding and scaling.

60-Second Interview Framing

Spoken version:

I would design Miro around a per-board operation log with CRDT-style merge semantics over a shape graph. Each board has a session server that serializes operations from connected clients and broadcasts to all peers. Cursors and presence are ephemeral, separate channels. The interesting decisions are CRDT design for spatial objects, real-time scaling per board, undo semantics across users, and rendering large boards.

Requirements To Clarify First

Functional requirements:

  • Multiple users collaborate on a shared canvas.
  • Create, move, resize, delete shapes.
  • Text, freehand drawing, sticky notes, frames.
  • See others' cursors and selections live.
  • Undo / redo.
  • Versions / history.
  • Board sharing with permissions.
  • Comments anchored to objects.

Non-functional requirements:

  • Sub-200ms latency for collaborator actions.
  • Boards with thousands of shapes.
  • Offline edits sync on reconnect.
  • Permissions enforced.
  • Multi-region.
  • Boards persist forever.

Out of scope unless asked:

  • Real-time video/audio.
  • AI-assisted features.
  • Connector/diagram intelligence.

CRDT For Shapes

Shapes are objects with properties; operations modify properties.

CRDT design choices:

  • Each shape is an object with LWW-per-property: create a shape, set properties. Concurrent updates to the same property pick winner by timestamp + tiebreaker.
  • Position as LWW or counters: when two users drag the same shape, last-writer-wins on final position. Intermediate cursors visible via presence.
  • Z-order as fractional indexing: each shape has a rational z-index. Inserting between two shapes uses a fractional value. Avoids conflicts on reordering.
  • Deletes use tombstones: a deleted shape is a tombstone; later modifications that didn't see the delete are ignored.

For freehand drawing: each stroke is its own shape; concurrent strokes don't conflict.

Interview phrase:

CRDTs for whiteboard shapes are mostly LWW per property plus fractional indexing for ordering plus tombstones for deletes. Yjs is a common library implementing this; Automerge is another option.

Sync Protocol

Similar to Google Docs:

  1. Client opens board; loads latest snapshot + ops since snapshot.
  2. Builds local CRDT state.
  3. Client makes a change; emits an op tagged with current state vector.
  4. Sends op to session server.
  5. Server appends to op log; broadcasts to all peers including originator.
  6. Peers apply op to their CRDT state.

For offline:

  • Client buffers ops while disconnected.
  • On reconnect, sends buffered ops; receives missed ops.
  • CRDT merge guarantees convergence regardless of order.

Session Server

Per-board leader running on a session server.

  • Maintains the current CRDT state in memory.
  • Persists ops to the log.
  • Broadcasts to connected clients.
  • Authoritative for op ordering.

Sharding: many boards across many session servers. Load balancer assigns boards to servers; sticky.

For very-popular boards (100+ concurrent editors): single session leader is the bottleneck. Mitigation: vertical scale; or accept latency penalty for very large rooms.

Snapshots

Replaying thousands of ops on open is slow. Periodically:

  • Materialize current state into a snapshot.
  • Persist snapshot.
  • On open, load snapshot + ops since.

Snapshot frequency: every N ops or M minutes.

Presence: Cursors And Selections

Real-time, ephemeral, broadcast over WebSocket without persistence.

  • Each user broadcasts cursor position every ~100ms while moving.
  • Each user broadcasts selection (which shapes selected).
  • Other clients render colored cursors with name labels.

Not persisted; restart loses cursors.

Undo / Redo

Tricky in multi-user scenarios. Options:

  • Per-user undo stack: Each user undoes their own operations only. Other users' ops are not affected. Most intuitive UX.
  • Global undo stack: Anyone can undo the last op. Confusing.

Per-user is the standard. Operationally: - Each user maintains a local stack of their ops. - Undo emits an inverse op that the CRDT applies.

Rendering Large Boards

A board with 10,000 shapes can't redraw all on every frame.

Client optimizations: - Viewport culling: only render shapes in view. - LOD (level of detail): simplify off-screen or zoomed-out shapes. - WebGL / canvas batched rendering. - Spatial indexing on client (R-tree or quadtree).

Server-side: send only ops, not the full state. Snapshot bandwidth bounded by snapshot size.

Permissions

Per-board ACL: users/groups with roles (view, comment, edit, admin).

Checked at session connect and per op. Cached at session server.

Multi-Region Strategy

  • Each board has a home region.
  • Session server in home region.
  • Remote clients connect via WebSocket; accept cross-region latency.
  • Snapshots and op logs replicated for DR.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Session server fails. Replica takes over; clients reconnect; replay from log. Session reconnects.
Concurrent edits create conflicts. CRDT merge converges. Convergence failures (should be zero).
Large board slow to load. Snapshot avoids op replay; lazy load distant shapes. Initial load latency.
Presence storm. Throttle broadcast rate. Presence QPS.
Offline user's ops massive on return. Bounded buffer; chunked sync. Pending op count on reconnect.
Permission revoke mid-session. Re-check on each op; eject. Revoke propagation.
Hot board with 100 editors. Vertical scale; degrade UX (read-only beyond N). Per-board QPS.
Op log corruption. CRC + replication. Integrity violations.
Render performance degrade. Viewport culling; LOD. Client FPS.
Cross-region replication lag. Async; bounded. Lag.

Operational Metrics

Correctness:

  • Convergence failures (should be zero).
  • Lost ops on reconnect.
  • Permission bypass attempts.

Capacity:

  • Sessions per server.
  • Ops per board per second.
  • Snapshot generation rate.

Business:

  • Active boards, editors per board.
  • Average board size.

Tradeoffs To Call Out

  • CRDTs converge automatically but use more metadata than OT.
  • Per-user undo is intuitive but doesn't undo cross-user conflicts.
  • Snapshot frequency: more frequent = faster load, more storage.
  • Sticky session server per board limits scale per board but simplifies design.
  • Aggressive presence broadcast feels live but costs bandwidth.
  • Multi-region per-board is hard; usually single-home.

2. Simulated Interview

Interviewer: Design Miro.

Candidate: Per-board CRDT shape graph backed by a session server. Clients connect via WebSocket; ops broadcast to peers; CRDT merges concurrent edits convergently. Operation log persists ops; periodic snapshots enable fast load. Presence (cursors, selections) is ephemeral. The invariant is that every editor's view converges to the same state regardless of concurrent operation ordering.

Interviewer: CRDT — explain the design.

Candidate: Shape graph. Each shape is an object with properties; operations create, modify, delete shapes. For shape properties (position, size, color), LWW with timestamp + tiebreaker (author id). For shape ordering (z-index), fractional indexing — inserting between two shapes uses a rational value between them. Deletes are tombstones; later modifications that didn't see the delete are ignored. For freehand drawing, each stroke is a separate shape so concurrent strokes don't conflict. Yjs and Automerge are libraries implementing this; custom implementations also common.

Interviewer: Two users drag the same shape simultaneously.

Candidate: Each user emits position update ops with timestamps. CRDT merge: last-write-wins on the final position. Intermediate positions are visible only via presence (live cursors and "user is moving X"), not via the persisted state. The final resting position is whoever's last op had the latest timestamp. Both users see the same final position.

Interviewer: Sync protocol.

Candidate: Client connects via WebSocket to the board's session server. Server returns latest snapshot + ops since snapshot. Client builds local CRDT state. User actions emit ops; client applies optimistically and sends to server. Server appends to op log and broadcasts to all peers. Peers apply op to their CRDT state. Convergence guaranteed by CRDT semantics.

Interviewer: Offline editing.

Candidate: Client buffers ops while disconnected. On reconnect, sends buffered ops; receives missed ops from server. CRDT merge handles all orderings — concurrent ops produced offline merge with online ops without conflict. The client's local state catches up to convergence.

Interviewer: A board has 10,000 shapes. Initial load time?

Candidate: Snapshot avoids replaying the full op log. The latest snapshot is downloaded; ops since snapshot are applied. Snapshot is megabytes-scale typically. To render, client uses viewport culling — only shapes in the visible area are rendered. LOD: zoomed-out views show simplified shapes. Spatial indexing (R-tree or quadtree) speeds picking and viewport queries.

Interviewer: Undo / redo with multiple users?

Candidate: Per-user undo. Each user has a local stack of their own ops. Undo emits an inverse op (move shape back; resurrect deleted shape; etc.) that the CRDT applies. Other users' ops are unaffected. Global undo where any user can undo any op is confusing and rare.

Interviewer: Presence — 50 users on a board moving cursors.

Candidate: Ephemeral. Each client broadcasts cursor position every 100ms while moving; selections on change. Server forwards via WebSocket without persisting. Throttle the broadcast rate per user to bound bandwidth. With 50 users, that's 500 cursor updates per second per board; manageable.

Interviewer: Permissions?

Candidate: Per-board ACL — users/groups with roles (view, comment, edit, admin). Checked at session connect. Cached at session server. Per-op revalidation cached. On revoke, session server ejects the user. Edit ops from view-only users are rejected.

Interviewer: Large board, 100 concurrent editors.

Candidate: Single session server handles ops in sequence. CPU and memory bound the concurrency. Vertical scale (more powerful server) can handle 50-100 editors. Beyond that, performance degrades; we might disable some features or set the board to "read-only beyond N viewers" UX. Sharding a single board across servers is hard because ordering is per-board.

Interviewer: Multi-region?

Candidate: Per-board home region. Session server in the home region. Remote editors connect via WebSocket, accepting cross-region latency (50-200ms). Op log and snapshots replicated globally for DR. We don't try to run an active session server in each region for the same board because the serialization invariant would break.

Interviewer: Summarize.

Candidate: CRDT shape graph with LWW per property, fractional indexing for order, tombstones for deletes. Per-board session server sequences ops and broadcasts via WebSocket. Op log persists; snapshots enable fast load. Presence ephemeral. Per-user undo via inverse ops. Client uses viewport culling and LOD for large boards. Multi-region by per-board home. The hardest correctness problem is convergence under concurrent edits; the hardest UX problem is rendering large boards smoothly.
