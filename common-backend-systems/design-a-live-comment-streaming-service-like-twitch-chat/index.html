1. Basic Knowledge And Solution

What This Question Is Testing

Twitch chat is a high-fanout live comment system: thousands to millions of viewers attached to one stream, each sending messages and seeing everyone else's messages with sub-second latency. The hard part is the asymmetric fan-out — one message must reach millions of viewers — combined with moderation, slow mode, emote rendering, and ephemeral retention.

The invariant is this:

Every chat message sent by a viewer reaches all other connected viewers of the same channel within sub-second latency, ordered consistently across recipients; moderation actions (timeouts, bans, deletes) apply within seconds; the system survives massive viewer count surges and channel-specific moderation rules.

At ICT5 / L6+ level, the interviewer is testing whether you can design websocket-based pub/sub fan-out, scale per-channel sessions to millions of viewers, moderation workflows, and reason about ephemeral storage and rate limiting.

60-Second Interview Framing

Spoken version:

I would design Twitch chat as per-channel pub/sub where every message published by a viewer fans out to all connected viewers via WebSocket. Channel sessions live on a fleet of chat servers; viewers connect via WebSocket to a server that handles their channel. Moderation, slow mode, and emote replacement are applied server-side. The interesting decisions are fan-out architecture for million-viewer channels, moderation, and rate limits.

Requirements

Functional requirements:

  • Viewers join a channel; send messages.
  • Messages broadcast to all viewers in real time.
  • Moderation: timeouts, bans, message deletes.
  • Emotes (image substitution by name).
  • Slow mode, subscribers-only, follower-only.
  • Chat history visible on (re)join (recent N messages).
  • Replies, mentions.

Non-functional requirements:

  • Sub-second message delivery.
  • Millions of viewers per popular channel.
  • Survive channel-specific spike (popular streamer goes live).
  • Moderation actions visible within seconds.
  • Multi-region.

Out of scope unless asked:

  • Voice/video chat.
  • Whispers (private DMs).
  • Channel point redemption details.

Send Path

  1. Viewer sends message via WebSocket.
  2. Gateway validates: viewer is in channel; not banned; not timed out.
  3. Rate limit check: per-viewer per-channel token bucket.
  4. Moderation: keyword filter; spam detection; emote substitution.
  5. Publish message to channel pub/sub topic.
  6. Persist to recent-messages buffer (bounded).
  7. All gateways subscribed to the channel forward the message to their local viewers' WebSockets.

Receive Path

  1. Each gateway subscribes to channels its viewers are in.
  2. Pub/sub broker delivers new messages.
  3. Gateway sends to all local viewer WebSockets for that channel.
  4. Client renders with emote substitution.

Fan-Out For Million-Viewer Channels

A 1M-viewer channel with 1000 messages/sec is 1B message-deliveries/sec. Manageable if distributed.

Strategy:

  • Multiple gateways per channel.
  • Each gateway holds a subset of viewers.
  • Pub/sub broadcasts to all gateways; each forwards to local viewers.
  • Gateway fanout: 1M / N gateways = ~10k per gateway. Manageable per-node.

Interview phrase:

Each gateway forwards to its local viewers only. Pub/sub broadcasts to all gateways; the fanout cost is distributed across the gateway fleet. A million-viewer channel is just 100 gateways with 10k viewers each.

Moderation

  • Bans: persistent block. User can't send. Enforced at send check.
  • Timeouts: temporary; expires after N seconds.
  • Slow mode: users can send one message per N seconds.
  • Subscribers-only: only paying viewers can send.
  • Follower-only: only followers (with min age).
  • Delete: remove specific message; broadcast delete event.
  • Auto-mod: AI flags suspicious content; mod approves before display.

Moderators (volunteer) can issue actions. Actions propagate via pub/sub to all gateways for the channel.

Rate Limiting

Per viewer per channel: - Normal: ~1 message per second. - Subscribers: maybe relaxed. - Spam detection: identical/similar messages flagged.

Per channel: - Slow mode applied uniformly.

Token bucket per (channel, user) in-memory at gateway, with periodic sync.

Emotes

Channel and global emotes. Text token like :LUL: substituted with image.

  • Emote dictionary cached per channel.
  • Substitution at client render or at gateway.
  • Custom subscriber emotes per channel.

Recent Message Buffer

Channel keeps last N messages (e.g., 100-500) in an in-memory buffer.

  • On viewer join, replay buffer so they see context.
  • Bounded; older drops.
  • Optional: ephemeral storage with TTL (hours).

Channel Coordinator

Per-channel state (mode, bans, mods, slow mode timer) held by a coordinator service.

  • Gateways read coordinator state.
  • Mod actions update coordinator + broadcast.

Multi-Region Strategy

  • Channels have a home region (streamer's).
  • Viewers from other regions connect to local gateway; gateway subscribes to channel's pub/sub which may be cross-region.
  • Cross-region latency adds ~50-100ms.
  • For huge global streams, regional fan-out: each region's pub/sub mirrors the channel; messages replicate cross-region.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Gateway crash. Viewers reconnect; resubscribe. Reconnect rate.
Pub/sub broker fail. Failover replica; resubscribe. Broker health.
Million-viewer surge. Auto-scale gateways; pre-warm. Per-channel viewer count.
Spam attack. Auto-mod + rate limits + ban. Spam rate.
Mod action lag. Pub/sub propagation; alert. Mod-to-effect lag.
Channel hot pub/sub topic. Shard topic; multi-region. Topic throughput.
Cross-region message loss. At-least-once; retry. Drop rate.
Order misorder. Channel-level monotonic seq. Out-of-order events.
Rate limit bypass. Aggregate at gateway + central sanity check. Bypass count.
Emote dictionary stale. Periodic refresh; invalidation on change. Stale emote rate.

Operational Metrics

Correctness:

  • Lost messages.
  • Out-of-order delivery.
  • Mod action propagation lag.

Capacity:

  • Connections per gateway.
  • Messages/sec per channel.
  • Pub/sub fanout latency.

Business:

  • Concurrent viewers per channel.
  • Chat engagement rate.
  • Mod actions per channel.

Tradeoffs To Call Out

  • Ordered delivery per channel adds complexity but matches UX expectations.
  • Persistence depth vs cost: more buffer = better catchup but more memory.
  • Aggressive auto-mod helps but blocks legitimate users.
  • Multi-region adds latency but resilience.
  • Per-channel pub/sub topic is simple; sharding hot channels adds complexity.

Common Traps And Stronger Moves

Trap Stronger move
Single broker per channel. Sharded broker for hot channels.
Synchronous moderation in send path. Inline fast filter + async deep auto-mod.
Persist all chat. Bounded recent buffer; ephemeral.
Single gateway. Distributed; pub/sub between.
No rate limits. Token bucket per (channel, user).
Trust client emote rendering. Server-side emote substitution.

  • Distributed fanout (each gateway forwards to local viewers).
  • Moderation inline + async auto-mod.
  • Per-viewer + per-channel rate limits.

2. Simulated Interview

Interviewer: Design Twitch chat.

Candidate: Per-channel pub/sub system. Viewers connect via WebSocket to a chat gateway. Each channel has a pub/sub topic; all gateways with viewers in that channel subscribe. When someone sends a message, gateway validates and publishes to the topic; pub/sub broadcasts to all subscribed gateways, each of which forwards to its local viewers. Million-viewer channels distribute the fanout across the gateway fleet. Moderation applied inline (fast filter, ban check) plus async deep auto-mod.

Interviewer: Walk me through sending a message.

Candidate: Viewer types message; gateway receives via WebSocket. Gateway checks: viewer in channel, not banned, not timed out, within rate limit (token bucket per-user-per-channel), satisfies channel mode (sub-only, slow). Inline content filter (banned words, spam). Publish to channel pub/sub topic with channel-level monotonic seq. Add to recent message buffer (bounded ~500). Pub/sub broadcasts to all gateways with this channel subscribed. Each gateway forwards to its local viewer WebSockets for the channel. Sub-second end-to-end.

Interviewer: A million-viewer channel.

Candidate: Each gateway forwards to its local viewers only. With 100 gateways each holding 10k viewers, each gateway sends 1000 msg/sec × 10k = 10M op/sec — heavy but feasible per gateway. Pub/sub broadcasts to all 100 gateways: 100 deliveries per message. The total fanout (1M deliveries per message) is amortized across the gateway fleet. Scale by adding gateways.

Interviewer: Order across viewers.

Candidate: Channel-level monotonic seq assigned by the channel coordinator (or the publishing gateway with a coordinator-issued range). All gateways forward in seq order. Recipients see consistent order. For multi-region, each region's local order matches the global; cross-region viewers may see slight delay but consistent order with their region's recipients.

Interviewer: Mod ban.

Candidate: Moderator clicks ban. REST request to chat service. Channel coordinator updates ban list. Pub/sub broadcast of ban event to all gateways. Gateways update local state; future sends from banned user rejected at gateway. Already-published messages may remain visible; optional broadcast a delete event to remove their recent messages. End-to-end propagation in seconds.

Interviewer: Slow mode.

Candidate: Per-channel setting: max one message per N seconds per user. Token bucket at gateway: each user has a bucket with N-second refill. Send rejects if bucket empty. Subscribers may be exempted.

Interviewer: Auto-mod.

Candidate: Two layers. Inline fast filter: banned words, regex, spam patterns. Cheap; runs on every send. Async deep classifier: ML model scores message; if unsafe, broadcast a delete event after the fact and may apply timeout. Async because deep models are expensive; we accept brief display of borderline messages.

Interviewer: Recent message buffer.

Candidate: Per-channel ring buffer of last ~500 messages. In-memory at coordinator or replicated. When new viewer joins, replay buffer so they see context. Bounded; older drops. Ephemeral; no long-term persistence except possibly hours-retention for moderation audit.

Interviewer: Connection drops mid-stream.

Candidate: Viewer reconnects via WebSocket. Gateway resubscribes the connection to the channel. Gateway sends the last N messages (replay buffer) so the viewer catches up. No state loss because chat is mostly ephemeral.

Interviewer: Multi-region.

Candidate: Channel home region. Pub/sub topic in home region. Viewers worldwide connect to local gateway; gateway subscribes to the home region's topic (or to a regional mirror). For huge global streams, each region runs its own pub/sub that mirrors home region's stream — viewers attached locally with cross-region replication adding ~50-100ms.

Interviewer: Spam attack: 10k bot accounts blast the channel.

Candidate: Per-user rate limit catches most. Pattern detection: similar messages from many new accounts → auto-mod kicks in; bans propagated. Channel mode can switch to follower-only or sub-only as defense. Pub/sub absorbs the spike but moderation reduces signal.

Interviewer: Summarize.

Candidate: Per-channel pub/sub with WebSocket gateways. Distributed fanout for million-viewer channels. Inline moderation (rate limit, banned words, ban check) + async deep auto-mod. Bounded recent message buffer. Slow mode and channel modes enforced at gateway. Multi-region with channel home + regional mirrors for global streams. Mod actions propagate via pub/sub in seconds.

3. Practice Notes

Five-Minute Version

  1. Clarify: scale, latency, moderation.
  2. Invariant: sub-second broadcast; ordered per channel; mod fast.
  3. WebSocket gateways + per-channel pub/sub.
  4. Distributed fanout: gateway → local viewers.
  5. Inline moderation + async auto-mod.
  6. Per-user per-channel rate limit.
  7. Bounded recent message buffer.
  8. Multi-region with channel home + mirrors.

One-Minute Closeout

Twitch chat is per-channel pub/sub with WebSocket gateways. Each gateway forwards to its local viewers; the fanout is distributed across the gateway fleet, so a million-viewer channel is just a hundred 10k-viewer gateways. Moderation is inline (rate limit, ban check, banned words) plus async deep auto-mod. Bounded recent buffer for new joiners. Channel-level seq preserves order. Multi-region with channel home.
