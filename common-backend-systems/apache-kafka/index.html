1. Basic Knowledge And Solution

What This Question Is Testing

Kafka is a distributed log: an append-only, partitioned, replicated stream. The hard part is delivering ordered messages within a partition, surviving broker failures without loss, scaling to millions of writes per second, and supporting consumer groups that share work without duplicating processing.

The invariant is this:

Every produced message that returns success is durably committed across a quorum of replicas, ordered within its partition, readable by any consumer at its assigned offset, and retained for the configured time; consumer groups divide partitions so that each partition is processed by at most one consumer in the group; rebalances are bounded and do not lose progress.

At ICT5 / L6+ level, the interviewer is testing whether you can design a partitioned replicated log, explain leader election and ISR (in-sync replicas), reason about producer acks levels, consumer group rebalancing, and exactly-once semantics via transactions or idempotent producers.

60-Second Interview Framing

Spoken version:

I would design Kafka as a cluster of brokers, each owning a subset of topic partitions. Each partition is a replicated append-only log with one leader serving reads and writes; followers replicate. Producers send batches to the partition leader; consumers track per-partition offsets. Consumer groups split partitions for parallelism. The interesting decisions are partition replication, ISR management, producer ack levels, consumer group rebalancing, and exactly-once.

Requirements To Clarify First

Functional requirements:

  • Topics with partitioned, ordered message logs.
  • Producers append messages; durable.
  • Consumers read from offsets; resumable.
  • Consumer groups: members share partitions.
  • Retention: by time or size.
  • Topic compaction (keep latest per key).
  • Transactions across topics for exactly-once.

Non-functional requirements:

  • Millions of writes/sec per cluster.
  • Petabytes of storage; weeks of retention.
  • High availability: tolerate broker failures.
  • Strong ordering per partition.
  • Producer durability tunable (acks=0, 1, all).
  • Consumer at-least-once or exactly-once.

Out of scope unless asked:

  • Connectors and Kafka Connect.
  • Streams processing API.
  • KSQL.

Replication And ISR

Each partition has a replication factor (e.g., 3). One replica is the leader; the others are followers.

  • Leader receives produce requests; appends to local log.
  • Followers fetch from leader and replicate.
  • A follower is in-sync (ISR) if it's caught up within a threshold.
  • The high-water mark (HW) is the highest offset replicated to all ISRs.
  • Consumers can only read up to the HW.

Producer acks levels:

  • acks=0: producer doesn't wait. Fastest, no durability.
  • acks=1: leader writes locally; doesn't wait for followers. Risk of loss if leader fails before replication.
  • acks=all: leader waits for all ISR members. Strongest durability.

If a follower lags too much, it's removed from ISR; the partition continues with remaining ISR members. If ISR shrinks below min.insync.replicas, the partition refuses writes (with acks=all).

Interview phrase:

The ISR is the dynamic set of replicas considered up-to-date. Writes commit when all ISR members ack; consumers see only committed offsets. This gives durability tunable to the cost.

Leader Election

The controller monitors broker health. On broker failure:

  1. Controller detects via missed heartbeats.
  2. For each partition that broker led, controller picks a new leader from the ISR.
  3. New leader takes over; new leader_epoch issued.
  4. Producers retry against the new leader.

If no ISR member is available, behavior depends on unclean.leader.election.enable:

  • Default false: partition is unavailable until an ISR member returns.
  • True: an out-of-sync replica may be elected (allows availability but risks data loss).

Modern Kafka uses KRaft (Raft-based controller) instead of ZooKeeper, simplifying deployment.

Producer Path

  1. Producer creates record with optional key.
  2. Partitioner picks partition: hash(key) % num_partitions, or round-robin if no key.
  3. Producer batches records by partition.
  4. Sends batch to partition leader.
  5. Leader appends to local log, replicates.
  6. Once acks level satisfied, leader returns success.

Idempotent producer:

  • Each producer has a producer_id and per-partition sequence number.
  • Broker dedupes by (producer_id, sequence_number).
  • Eliminates duplicates from retries.

Transactional producer:

  • Producer begins a transaction.
  • Writes to multiple partitions / topics.
  • Commits or aborts atomically.
  • Consumers configured read_committed see only committed messages.
  • Used to implement exactly-once semantics for read-process-write pipelines.

Consumer Path

  1. Consumer polls broker for messages from assigned partitions.
  2. Broker returns batches up to high-water mark.
  3. Consumer processes messages.
  4. Consumer commits offset (synchronously or async).
  5. On restart or rebalance, consumer resumes from last committed offset.

Default delivery: at-least-once. If consumer crashes after processing but before committing, it reprocesses on restart.

For exactly-once: use transactional producer to commit offsets atomically with output writes.

Consumer Group Rebalancing

Members of a group share partitions: each partition assigned to one member.

Membership protocol:

  1. New member joins; sends JoinGroup to group coordinator broker.
  2. Coordinator collects all members; selects a leader (one of the members).
  3. Leader computes assignment (e.g., range or round-robin).
  4. Coordinator broadcasts assignment to all members.
  5. Members start consuming their partitions.

On member leave or crash, rebalance triggers. During rebalance:

  • All members pause.
  • New assignment computed.
  • Members resume.

Modern Kafka has incremental cooperative rebalancing to reduce stop-the-world impact: only affected partitions move; others continue.

Interview phrase:

Consumer group rebalancing is the most operationally tricky part. Members pause during rebalance; long rebalances stall processing. Cooperative rebalancing reduces the disruption.

Storage And Retention

Each partition is a sequence of log segments on disk.

  • New writes append to the active segment.
  • When segment reaches size limit, it's rolled; a new active segment starts.
  • Old segments are retained per topic config (time or size).
  • Cleanup deletes expired segments.

Compaction (per-key retention):

  • For a topic configured with cleanup.policy=compact.
  • Older records with the same key as a newer record are eligible for deletion.
  • Useful for changelog topics where only latest value matters.

Consistency Model

Aspect Guarantee
Per-partition order Strong.
Cross-partition order None.
Producer ack Tunable (0, 1, all).
Consumer read Up to high-water mark (committed).
Idempotent producer Per-partition exactly-once.
Transactional producer Atomic across partitions/topics.

Scaling And Partitioning

Concern Strategy
Throughput per topic Increase partition count.
Throughput per partition Single-leader bottleneck; tune batch sizes.
Broker disk Distribute partitions across brokers.
Hot key Sub-partition by adding random suffix; consumers re-aggregate.
Consumer scale Add members up to partition count; one partition per member max.

Partition count is a key decision:

  • More partitions = more parallelism, but more controller load and longer rebalance.
  • Fewer partitions = less overhead, but consumer scale is capped.

Multi-Region (MirrorMaker / Cluster Linking)

  • Primary cluster in one region.
  • MirrorMaker replicates topics to a secondary cluster for DR or geo-locality.
  • Cluster linking (Confluent): native cross-cluster replication preserving offsets.
  • Bidirectional active-active is possible but requires careful conflict handling for the same topic.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Broker fails. Controller picks new leaders from ISR. Broker count; under-replicated partitions.
All ISR members fail. Partition unavailable; alert. ISR size.
Slow follower. Removed from ISR; rejoin on catch-up. ISR shrink frequency.
Producer retries duplicate. Idempotent producer dedupes. Producer retry rate.
Consumer lag growing. Add consumers; alert on lag. Consumer lag per partition.
Long rebalance stalls work. Cooperative rebalancing; static membership. Rebalance frequency and duration.
Disk full. Retention enforcement; alert on usage. Per-broker disk %.
Hot partition (one producer key). Sub-partition; spread by adding random suffix. Per-partition write rate.
Controller failure. KRaft consensus failover. Controller health.
Cross-region replication lag. Monitor; backpressure source. Mirror lag.

Operational Metrics

Correctness:

  • Under-replicated partitions (must be near zero).
  • Failed ISR shrinks.
  • Producer errors per topic.
  • Consumer commit failures.

Capacity:

  • Per-partition throughput.
  • Per-broker disk I/O.
  • Consumer lag.
  • Rebalance duration.

Business:

  • Topic throughput by team.
  • Retention compliance.
  • End-to-end latency.

Tradeoffs To Call Out

  • More partitions = more parallelism but more metadata overhead.
  • acks=all is safest but slowest.
  • Unclean leader election trades availability for potential data loss.
  • Idempotent + transactional producer adds latency but gives exactly-once.
  • Compaction saves space for keyed topics but adds complexity.
  • ISR shrink risk vs requiring high min.insync.replicas.

Common Traps And Stronger Moves

Trap Stronger move
Treat partitions as ordering across topic. Order is per partition; cross-partition is independent.
Single partition for total order. Sacrifices parallelism; use per-key ordering instead.
acks=1 everywhere. acks=all for correctness-critical topics.
Ignoring consumer lag. Alert on lag; auto-scale consumers.
No idempotency. Enable idempotent producer for safe retries.

Interview Phrase

Kafka is a partitioned replicated append-only log. Per-partition strong ordering, leader-follower replication, and tunable durability via producer acks make it the workhorse of streaming. The hardest correctness problems are exactly-once (solved with transactions) and consumer group rebalances (mitigated with cooperative rebalancing).

2. Simulated Interview

Interviewer: Design Kafka.

Candidate: Cluster of brokers; each topic partitioned across brokers; each partition replicated 3x typically with one leader and followers. Producers append; consumers poll from offsets. Consumer groups split partitions. Controller (KRaft these days) manages metadata. The invariant is per-partition strong ordering, durable to the configured ack level, with consumer groups dividing partitions for parallel processing.

Interviewer: Walk me through a produce.

Candidate: Producer batches records by partition. Picks partition via hash(key) % num_partitions if key set, else round-robin. Sends batch to partition leader. Leader appends to local log, replicates to followers. With acks=all, leader waits for all ISR members to ack, then returns success to producer. With acks=1, leader returns after local write. With acks=0, fire-and-forget.

Interviewer: A broker dies. What happens?

Candidate: Controller detects via missed heartbeats. For each partition that broker led, controller picks a new leader from the ISR — the in-sync replicas. New leader_epoch is issued; producers retry against the new leader. Followers continue replicating from the new leader. If the failed broker was also in followers of partitions led elsewhere, those partitions' ISRs shrink; if min.insync.replicas drops, those partitions refuse writes.

Interviewer: What's the ISR?

Candidate: In-Sync Replicas — the set of replicas considered up-to-date with the leader within a configured lag threshold. ISR is dynamic: slow followers are removed; recovering ones rejoin. The high-water mark is the highest offset replicated to all ISR members; consumers see only up to HW. acks=all waits for ISR ack, which gives strongest durability.

Interviewer: How does the consumer group know which partition to read?

Candidate: Membership protocol. Each consumer joins the group via JoinGroup to the group coordinator broker. Coordinator collects all members, picks one as leader, asks it for an assignment plan (range, round-robin, or sticky). Coordinator broadcasts the assignment. Each consumer starts reading its assigned partitions. On member change, rebalance triggers and a new assignment is computed.

Interviewer: Rebalances pause work?

Candidate: Traditionally yes — stop-the-world. All members pause, new assignment computed, members resume. For high-throughput consumers, this is painful. Modern Kafka has cooperative rebalancing: only partitions that move are paused; others keep flowing. Static membership lets a consumer rejoin without triggering rebalance if it crashes briefly.

Interviewer: Exactly-once?

Candidate: Two parts. First, idempotent producer: each producer has a producer_id; per-partition sequence numbers let the broker dedupe retries. Eliminates duplicates from a single producer's retries. Second, transactional producer: producer begins a transaction, writes to multiple partitions/topics, commits atomically. Consumers configured read_committed only see committed records. Combined with committing consumer offsets in the same transaction, you get exactly-once for read-process-write pipelines.

Interviewer: Compaction?

Candidate: Per-topic policy. For cleanup.policy=compact, the broker periodically rewrites segments to keep only the latest record per key. Older records with the same key are dropped. Useful for changelog topics where you only care about the current state. Tombstones (null values) signal deletion. Compaction runs in background; doesn't block writes.

Interviewer: A producer key generates 100,000 msg/sec, far more than one partition can handle.

Candidate: Single partition is the throughput ceiling for that key. Mitigation: add a sub-partition by appending a random suffix to the key, so the same logical key spreads across multiple partitions. Consumers must re-aggregate if they need per-key processing. The tradeoff: lose strict per-key ordering for higher throughput.

Interviewer: Storage?

Candidate: Each partition is a sequence of log segments on disk. The active segment receives writes; when it reaches size or time limit, it's rolled. Old segments are retained per retention policy (time or bytes). Cleanup deletes expired segments. Compaction (if enabled) keeps the latest per key. Brokers use page cache aggressively; sequential disk writes are very fast.

Interviewer: Multi-region?

Candidate: MirrorMaker (or cluster linking) replicates topics to a secondary cluster. Primary in one region, secondary in another for DR or read locality. Bidirectional active-active is possible but each side writes to its own topics; cross-region writes to the same topic require careful conflict handling. Most setups are active-passive with failover.

Interviewer: Summarize.

Candidate: Kafka is a partitioned replicated append-only log. Brokers store partitions on disk; leader-follower per partition with ISR. Producers tune durability via acks; idempotent producer dedupes retries. Consumers track per-partition offsets; consumer groups split partitions for parallelism. Controller (KRaft) manages metadata and leader elections. Compaction supports keyed-state topics. Multi-region via MirrorMaker. The hardest correctness problem is exactly-once; the hardest operational problem is rebalance impact.

Kafka is a partitioned replicated append-only log. Per-partition strong ordering; leader-follower per partition; tunable durability via producer acks.
