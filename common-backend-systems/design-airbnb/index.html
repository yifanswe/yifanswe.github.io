1. Basic Knowledge And Solution

What This Question Is Testing

Airbnb is a two-sided marketplace with date-based scarce inventory: each listing has unique nights for sale; once booked, that night cannot be sold again. The hard part is searching geographically with date availability, atomically reserving date ranges across listings, handling host cancellations, and managing payouts.

The invariant is this:

For every listing, each calendar night is sold to at most one confirmed reservation; search results reflect only available date ranges; bookings convert to confirmed reservations only after payment succeeds and the host accepts (or instant book applies); cancellations release dates back to inventory atomically.

At ICT5 / L6+ level, the interviewer is testing whether you can model date-range inventory (more complex than seat-based), design geospatial search with availability filtering, handle the booking state machine through payment, and reason about host-initiated cancellations and overbooking risks.

60-Second Interview Framing

Spoken version:

I would design Airbnb as four core systems: a listing service with geospatial search, an availability service that owns calendar nights per listing, a booking state machine with payment integration, and a messaging system between hosts and guests. The interesting decisions are date-range inventory modeling, atomic multi-night reservation, host vs instant-book flows, and cancellation handling with refund policies.

Requirements To Clarify First

Functional requirements:

  • Hosts list properties with photos, calendar, pricing, rules.
  • Guests search by location, dates, guests, filters.
  • Booking: instant book or request-to-book.
  • Payment, refunds per cancellation policy.
  • Messaging between guest and host.
  • Reviews after stay.
  • Host cancellation and reassignment.

Non-functional requirements:

  • No double-booking: a night belongs to at most one reservation.
  • Search returns results in sub-second.
  • High availability for browsing; correctness-critical for booking.
  • Multi-region (global users, regional listings).
  • Auditability for money and date ownership.

Out of scope unless asked:

  • Listing creation tools and verification.
  • Pricing optimization (Smart Pricing).
  • Hosts' superhost program.
  • Tax collection details.

Geospatial Search With Availability Filter

The core search query: "find listings within radius R of (lat, lng), available on dates [check_in, check_out], satisfying filters."

Approach:

  1. Geospatial filter: H3/S2/geohash cells around target location.
  2. Index lookup: listings in those cells.
  3. Availability filter: for each candidate listing, check if all dates in [check_in, check_out] are available.
  4. Apply other filters (price range, guests, amenities).
  5. Rank by relevance, price, rating, personalization.

Optimizations:

  • Pre-computed availability sets in the search index (e.g., bitmap per listing for next 365 days).
  • Geospatial sharding for parallel filter.
  • Cached results for popular queries (e.g., Paris, this weekend).

Interview phrase:

Search must filter by both location AND date availability. A bitmap encoding of each listing's availability lets us check N nights in O(1) per listing during search.

Booking State Machine

requested -> accepted -> paid -> confirmed -> completed
   |             |          |          |
   +-> canceled  +-> declined +-> failed  +-> canceled

Instant book skips the "accepted" / "declined" steps.

Transitions:

Transition Trigger Effect
requested -> accepted Host accepts (request-to-book). Hold dates.
requested -> declined Host declines. Release dates.
accepted -> paid Payment succeeds. Charge guest.
paid -> confirmed Final state. Dates owned by booking.
Any -> canceled Cancellation. Per policy refund; release dates.

Each transition is conditional CAS on (booking_id, from_state). Idempotent.

Atomic Date Reservation

For booking [check_in, check_out]:

  1. Booking service starts.
  2. Acquire a hold on each date in range: - Conditional update on ListingAvailability(listing_id, date): UPDATE status = held, reservation_id = ?, version = version + 1 WHERE status = available AND version = ?
  3. If any date fails (already booked or held), roll back all holds; return unavailable.
  4. If all succeed, create Booking in accepted (or requested if RTB).
  5. Initiate payment.
  6. On payment success, transition dates from held to booked.
  7. On payment failure or hold timeout, release dates back to available.

Hold TTL: short (e.g., 15-30 minutes) for instant book; longer for request-to-book where host has time to respond (24 hours typical).

Interview phrase:

Atomic multi-night reservation requires conditional writes on each date row. Roll back on first failure; release on payment fail or timeout. This is the same pattern as Ticketmaster but with date ranges instead of seats.

Payment Integration

Reuse the payment system design:

  • Idempotency key per booking.
  • Authorize at hold; capture at confirmation (or at check-in time for some policies).
  • Refunds per cancellation policy.
  • Host payout after check-in (or after dispute window).

Timeout case: payment ambiguous → keep dates held; reconcile via payment provider.

Cancellation Policy

Three common policies:

Policy Refund
Flexible Full refund up to 24h before check-in.
Moderate Full refund up to 5 days before.
Strict 50% up to 7 days before; no refund inside.

Implementation:

  • Cancellation request triggers state transition.
  • Refund amount computed from policy + time-to-check-in.
  • Dates released back to available.
  • Refund issued via payment system.

Host cancellation:

  • More punitive (host pays penalty, listing may be deprioritized).
  • Guest receives full refund + relocation help.
  • Dates released.

Search Indexing And Refresh

Search index is eventually consistent. Refresh strategies:

  • On listing CRUD: publish event; indexer updates.
  • On booking: publish availability change event; indexer updates affected dates.
  • Periodic full re-index for catch-up.

Lag tolerance: a listing freshly booked may briefly appear available in search. Final hold attempt catches this.

Multi-Region Strategy

  • Listings region-affine: usually associated with a country.
  • Availability and booking writes route to listing's region.
  • Search index replicated globally (read-only) for fast cross-region browsing.
  • Guests from anywhere can book any listing; the write routes home.

Tradeoffs To Call Out

  • Search availability filter trades freshness for speed (bitmap pre-computed lags new bookings by seconds).
  • Instant book speeds reservation but reduces host control.
  • Strict cancellation policies favor hosts; flexible favor guests; the platform decides default.
  • Pre-warming search caches helps popular queries; useless for long-tail.
  • Geographic sharding helps locally but cross-country search is awkward.
  • Long hold TTLs hurt inventory utilization; short TTLs frustrate slow hosts.

No hold during payment. Hold dates from request to payment success.

Synchronous host accept. Async with 24h timeout; auto-decline.

2. Simulated Interview

Interviewer: Design Airbnb.

Candidate: I will design four core systems: listing service with geospatial search, availability service with calendar nights per listing, booking state machine with payment integration, and messaging. The invariant is each calendar night is sold to at most one confirmed reservation. Search returns listings filtered by location and date availability; booking atomically reserves a date range.

Interviewer: How do you store availability?

Candidate: Per-night row: (listing_id, date, status). Status is available, held, booked, or blocked. For a 5-night booking, we conditionally update 5 rows from available to held in a transaction (or sequential CAS with rollback). The booking is atomic across all dates in the range.

Interviewer: Two guests book overlapping dates simultaneously.

Candidate: Both attempt CAS updates on the same date rows. Only one succeeds for each contested date. If guest A locks night N first, guest B's update fails. The booking service rolls back any partial holds for guest B and returns "unavailable" with the next available alternative. The losing guest sees an immediate error; we may suggest similar listings.

Interviewer: Search must filter by date availability. How fast?

Candidate: Pre-compute an availability bitmap per listing covering the next 365 days. Bit i = 1 if date i is available. To check "available July 15-22", we look up the 7-bit slice and confirm all bits are 1. Stored in the search index. So search is geospatial cell scan plus a fast bitmap AND. Eventually consistent — when a booking commits, the bitmap update lags by seconds. The final hold is the authoritative check.

Interviewer: Instant book vs request-to-book?

Candidate: Two flows in the state machine. Instant book: guest pays immediately; dates go from held to booked on payment success. Request-to-book: guest's request creates a hold and pending state; host has 24 hours to accept or decline. On accept, payment proceeds. On decline or timeout, the hold releases. The state machine branches at the start; the rest converges.

Interviewer: What about payment timeout?

Candidate: Same as Ticketmaster. The dates are held; the booking is in pending_payment. We do not release the hold until we know the payment result. A reconciliation worker queries the payment provider; on success, transition to confirmed; on failure, release dates. We accept that a night is locked during reconciliation rather than risk double-charging or double-booking.

Interviewer: Cancellation policies?

Candidate: Per-listing setting: flexible, moderate, strict. Cancellation triggers a state transition. Refund amount computed from policy and time-to-check-in. Dates released back to available. Payment service issues the refund. For host-initiated cancellations, the host pays a penalty, and the platform helps the guest relocate.

Interviewer: Search latency — Paris in July gets millions of queries.

Candidate: Cache popular query patterns: location + dates + popular filter combinations. Cache TTL of seconds to minutes; OK because we have the final hold as authoritative. Geospatial sharding by location cell parallelizes the filter. CDN-able for unauthenticated browse. Personalized ranking is online but lightweight.

Interviewer: Host updates their calendar — blocks a week.

Candidate: Host calls PUT /listings/{id}/availability with date range and status blocked. We CAS update the rows from available to blocked. If a date is already booked, we error; the host cannot unilaterally cancel a confirmed booking — they must go through the cancellation flow. Search index is updated via event.

Interviewer: Multi-platform host syncing calendars (Airbnb + VRBO).

Candidate: iCal sync. Host can import an iCal feed; we periodically poll and block dates marked in the external calendar. Two-way sync exports our calendar to the other platform. Conflict detection: if the same dates are booked on both platforms, alert the host. This is a known headache in the industry.

Interviewer: Reviews?

Candidate: Posted after stay. Both guest and host review each other. Reviews remain hidden until both submit (or a deadline passes), to avoid retaliation. Reviews aggregate to listing rating and host superhost status. Eventually consistent for the aggregate display.

Interviewer: Multi-region?

Candidate: Listings are region-affine to the country. Availability and booking writes route to the listing's region. Search index replicates globally for read so guests from anywhere search quickly. Cross-country booking writes hit the listing's home region — acceptable latency.

Interviewer: Summarize.

Candidate: Date-range inventory as per-night rows; atomic CAS for booking. Geospatial search with availability bitmap. Booking state machine handles instant and request-to-book. Payment integration with timeout handling preserves the hold during reconciliation. Cancellation per policy with refund. Search is eventually consistent; final hold is authoritative. Multi-region by listing home. The hardest correctness problem is no double-booking; the hardest search problem is date-availability filtering at scale.

3. Practice Notes

Five-Minute Version

  1. Clarify: scale, search, booking, cancellation.
  2. Invariant: each night sold to at most one reservation.
  3. Listing + availability as per-night rows.
  4. Geospatial search with availability bitmap.
  5. Booking state machine: requested → accepted → paid → confirmed.
  6. Atomic multi-night CAS reservation.
  7. Cancellation per policy; refund + date release.
  8. Multi-region with listing home; global search index.

One-Minute Closeout

Airbnb is Ticketmaster for date-range inventory. Per-night rows allow atomic CAS reservation. Geospatial search with pre-computed availability bitmap filters by location and dates fast. The booking state machine flows from request through payment to confirmation; instant book and request-to-book are branches. Cancellation policies map to refund formulas plus date release. Search is eventually consistent; the final hold is authoritative. Multi-region by listing home with a global search replica.

Best Interview Phrases

  • "Each night is a unit of inventory; bookings atomically reserve a range."
  • "Search uses availability bitmaps for fast date filtering; the final hold is authoritative."
  • "Booking is a state machine; instant book and request-to-book are two paths through it."
  • "Payment timeout preserves the hold and reconciles; never release scarce inventory on a guess."
  • "Cancellation is a state transition plus refund pipeline, not a delete."
