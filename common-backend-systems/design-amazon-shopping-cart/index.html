1. Basic Knowledge And Solution

What This Question Is Testing

Amazon's shopping cart is a deceptively-simple-looking feature: it's the canonical example used to motivate the original Dynamo paper. The hard part is keeping the cart available — every "add to cart" must succeed even during partitions — while reconciling conflicting concurrent updates from multiple devices.

The invariant is this:

Every "add to cart" action eventually appears in the user's cart on every device; deletes do not reappear unless re-added; concurrent updates from multiple devices merge without losing items; the cart is available even when some replicas are unreachable.

At ICT5 / L6+ level, the interviewer is testing whether you can articulate the Dynamo-style "always writable" tradeoff, design conflict resolution via vector clocks or CRDTs, handle multi-device cart merging, and reason about cart abandonment, inventory checks, and conversion to order.

60-Second Interview Framing

Spoken version:

I would design the cart around an always-writable Dynamo-style KV store keyed by user_id. Adds and removes are merged via vector clocks or CRDT semantics so concurrent updates from multiple devices never silently lose items. The interesting decisions are conflict resolution semantics (favor preservation), inventory check timing, and the cart-to-order conversion flow at checkout.

Requirements To Clarify First

Functional requirements:

  • Add item to cart with quantity.
  • Update quantity.
  • Remove item.
  • View cart from any device.
  • Persistent cart (survives session).
  • Apply coupons / promotions.
  • Save for later (separate list).
  • Checkout: convert cart to order.

Non-functional requirements:

  • High availability: cart writes must succeed even during partial outages.
  • Eventually consistent across devices.
  • Sub-second response.
  • Scale to hundreds of millions of users.
  • Conflict resolution: never silently lose items.

Out of scope unless asked:

  • Inventory enforcement at cart-add (hold inventory only at checkout).
  • Tax calculation (separate service).
  • Recommendation insertion (separate).

Conflict Resolution: The Dynamo Story

Scenario: User adds book A on phone, adds book B on laptop, simultaneously (or with one device offline).

Vector clock approach:

Each write carries a vector clock. The cart store accepts both writes; on read, if multiple versions exist (siblings), the client merges them.

For a cart, the natural merge is union: take all items from both versions. If both versions modify the quantity of the same item, take the max (preserve more, avoid loss).

Result: cart contains both A and B. User never "loses" an add.

For deletes, this is harder. If A was in the cart, then deleted on device 1, then device 2 (with stale state) adds quantity to A, naive union resurrects A.

Fix: use tombstones. Delete records a tombstone with version; later adds with older versions are merged but the tombstone wins.

CRDT approach:

Model the cart as a CRDT (e.g., OR-Set or 2P-Set):

  • OR-Set: each add has a unique tag; remove only tags it knows about.
  • Result: an add that didn't see the delete still appears in the merged set.

Interview phrase:

Dynamo's design favors availability over consistency. The cart is always writable; concurrent updates produce siblings that the client merges. The natural merge for a cart is set union with tombstones — favoring preservation over deletion.

Write Path

  1. User adds item on a device.
  2. Cart API writes to cart store with W (write quorum) replicas.
  3. Returns success once W replicas ack.
  4. Replicas propagate asynchronously to others.
  5. Read on another device may see eventually consistent state; if multiple versions exist, merge.

For high availability: W + R > N for strict consistency, or W < N and R < N for availability with eventual consistency.

Read Path

  1. User opens cart on a device.
  2. Cart API reads from R replicas.
  3. If versions diverge, client (or merge service) reconciles via vector clock.
  4. Return merged cart.

Optionally, write back the merged version to repair divergence (read repair).

Multi-Device Sync

When user opens cart on a different device:

  1. Read from cart store.
  2. Display merged state.
  3. User's actions (add, remove, update) write back.

Push update: when a write commits, notify subscribed devices to refresh. Optional; pull-on-open is sufficient.

Inventory At Cart Time

Amazon historically does NOT hold inventory at cart add. Reasons:

  • Cart abandonment is high; reserving inventory blocks others.
  • Inventory becomes a shopping-mall pre-purchase land grab.
  • The expensive coordination is moved to checkout.

So at cart-add, no inventory check. At checkout, inventory is verified and reserved. If out of stock by then, user sees "no longer available."

For very scarce items (flash sale-style), this changes — see Flash Sale design. But normal cart: lazy inventory.

Checkout

  1. User initiates checkout.
  2. Cart frozen for this checkout session.
  3. For each item, inventory check + reservation.
  4. Pricing computed (with current promos).
  5. Order created (state machine).
  6. Payment captured.
  7. Cart marked converted; items removed.

If any item is out of stock, surface error and let user adjust. Cart remains active.

Save For Later

Similar to cart but separate list. Same KV pattern; same conflict resolution.

Move between cart and save-for-later are atomic operations.

Operational Metrics

Correctness:

  • Lost-add rate (must be ~zero).
  • Resurrection rate (deleted items reappear).
  • Sibling rate during reads.

Capacity:

  • Cart writes per second.
  • Read QPS.
  • Merge operations per second.

Business:

  • Cart-to-checkout conversion.
  • Cart abandonment rate.
  • Average cart size.

Tradeoffs To Call Out

  • High availability sacrifices strict consistency; trade is acceptable for cart.
  • Set-union merge preserves items but may resurrect deletes; tombstones fix at cost of metadata.
  • No inventory hold at cart add saves coordination but disappoints users at checkout.
  • Tombstone GC requires deciding when it's safe to forget.
  • Per-user state simplifies partitioning.
  • CRDT design is more complex than vector clocks but compositional.

Hold inventory at cart-add. Lazy check at checkout.
No tombstones (delete loses to stale add). Tombstones with version.
Single replica. N replicas with W/R quorums.
Strict ordering. Concurrent OK; merge resolves.

2. Simulated Interview

Interviewer: Design Amazon's shopping cart.

Candidate: Dynamo-style always-writable KV store keyed by user_id. Cart writes succeed even during partial outages; reads resolve siblings via vector clocks. The natural merge for cart items is set union with max quantity per item; deletes use tombstones to prevent resurrection. No inventory hold at cart-add; inventory is checked and reserved at checkout. The invariant is that every add is preserved even under network partitions and concurrent device updates.

Interviewer: Walk me through a multi-device race.

Candidate: User has phone and laptop both online. Phone adds book A; laptop adds book B simultaneously, both based on a previous cart state. Cart store writes both with their respective vector clocks. Neither write knows about the other. When the user opens cart on either device, the read sees two siblings (concurrent versions). Merge: union the item sets, take max quantity per item. Result: cart has A and B. User never loses an add.

Interviewer: What if user deletes A on phone, then adds quantity to A on laptop with stale state?

Candidate: Tombstones. The delete writes a tombstone with version V. The laptop's add has version V' but doesn't know about the tombstone. On merge, the tombstone takes precedence per CRDT semantics (or per merge rule that tracks deletes). A stays deleted. The user's laptop refresh shows the cart without A. The alternative — naive union — would resurrect A, which is the "silent delete" bug.

Interviewer: Why no inventory hold at cart-add?

Candidate: Cart abandonment is the norm — most carts never check out. Holding inventory at cart-add reserves it from real buyers; under high traffic, you'd reserve more than you have actual buyers for. Coordination cost across hundreds of millions of carts is high. Amazon's choice: lazy inventory at checkout. If the item is out of stock by then, surface gracefully. The exception is flash sales, where scarcity is the point and the system holds inventory at add.

Interviewer: Checkout flow.

Candidate: State machine. User initiates → cart frozen for this checkout (lock optimistically) → for each item, inventory check + reservation → pricing computed with current promos → order created with idempotency key → payment captured → cart marked converted; items removed. If any inventory check fails, surface error and let user adjust; cart remains active.

Interviewer: What if a user checks out from two devices simultaneously?

Candidate: Idempotency. The checkout starts with a checkout_session_id. Two simultaneous starts produce two sessions; whichever commits first creates the order. The second session fails with "cart already checked out." UI on the second device refreshes. Alternatively, use a single CAS on cart state to checkout_in_progress.

Interviewer: A user adds 100 items rapidly.

Candidate: Per-user rate limit on adds. Reasonable thresholds. Bot detection if pattern looks automated. Cart still works for normal heavy users (e.g., bulk shopper) but flagged for review at extreme rates.

Interviewer: Multi-region?

Candidate: User home region for cart writes. Cross-region replication asynchronous. Travel: user accesses cart from another region — replica may have slight lag; read returns possibly-stale state; on write, merge resolves. Failover region warm.

Interviewer: Cart abandonment?

Candidate: Cart retained indefinitely by default (or per policy — many e-commerce sites keep 90 days). Marketing systems may email reminders about abandoned carts. Soft expiry (mark abandoned after N days) for metrics; hard expiry rare.

Interviewer: Promotions in cart?

Candidate: Promo codes recorded on cart items or cart-level. At checkout, validated and applied (eligibility, expiry, redemption count). Cart total computed at checkout; cart-display shows estimated total. Promo expired between cart-add and checkout: surface; let user adjust.

Interviewer: Save for later?

Candidate: Separate list, same pattern. Moving between cart and save-for-later is a tombstone in one + add in the other. Same conflict resolution applies.

Interviewer: Summarize.

Candidate: Dynamo-style always-writable cart store, keyed by user_id, replicated N ways. Vector clocks resolve concurrent writes; merge is set union with tombstones for deletes, max quantity for collisions. No inventory at cart-add; checked at checkout. Cart-to-order is a state machine with idempotency. Multi-region with user home. The hardest correctness problem is preserving every add while not resurrecting deletes; tombstones solve it.

3. Practice Notes

Five-Minute Version

  1. Clarify: scale, multi-device, inventory timing, conflict.
  2. Invariant: never lose adds; deletes stay deleted.
  3. Dynamo-style always-writable store.
  4. Vector clock or CRDT for conflict resolution.
  5. Union merge with tombstones.
  6. No inventory at add; lazy check at checkout.
  7. Cart-to-order state machine with idempotency.
  8. Multi-region with user home.

One-Minute Closeout

Amazon's cart is the canonical Dynamo paper use case. Availability is paramount; concurrent updates from multiple devices merge via vector clocks with set-union semantics and tombstones. No inventory hold at cart-add; checked lazily at checkout. The hardest correctness issue is preserving every add without resurrecting deletes — tombstones handle it. Multi-region with user home.
