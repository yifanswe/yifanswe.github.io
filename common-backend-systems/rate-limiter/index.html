Rate Limiter (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a distributed rate limiter that protects an API from abuse. Every incoming request hits the limiter first; it either lets it through or rejects with 429. Rules look like "100 req/min per user," "1000 req/sec per API key," "10 logins/hour per IP." Single-digit-millisecond overhead is the budget. 1M QPS across the fleet, hundreds of limiter instances.

1. Requirements (~5 min)

Functional - Decide allow/deny for each request, given one or more rules. - Support common policies: GCRA (Generic Cell Rate Algorithm) / token bucket / sliding window counter, keyed on (user, IP, key, route). - Surface remaining quota, reset time, and bucket scope in response headers so clients can back off intelligently. - Hot-reload rules without redeploying the gateway.

I'd defer policy authoring UI and OAuth-quota / billing-integrated metering.

Non-functional - Latency overhead p99 < 5 ms. - Available — counter-store outage must not take down the protected service. Fail mode is per-rule. - Reasonably accurate; small over-allowance is acceptable for soft rules, not for auth/abuse. - Scale to ≥1M decisions/sec across the cluster.

Capacity - 100M unique buckets (identity × rule × window). - State per bucket: ~50 bytes (counter + timestamp). - Raw counter state is ~5 GB, but Redis object overhead and replication make the real hot footprint closer to 30–60 GB. The dominant cost is not disk; it is memory plus one network round trip on the synchronous path.

Production calibration: a real rate limiter is a stack, not one knob.

Layer Question Window Example
Global shedder Is the fleet overloaded? sec Drop non-critical with 503 when busy
Per-route Is this endpoint hot? sec–min 10k QPS on /search
Per-tenant / identity Is one customer noisy? min–hour 100 req/min per user
Quota Did they exceed their plan? hour–day 1M API calls/day on Pro tier
Abuse Are they probing? rolling 10K invalid in 10min → ban

Each layer wants different state (memory-local for shedding, Redis-shared for per-tenant, long-term store for quotas, ML pipeline for abuse). Treating these as one component is the most common interview misstep.

2. Core Entities (~2 min)

  • Rule — name, matcher (a predicate over identity attributes), algorithm, limit, window, layer. Owned by the limiter service.
  • Identity — the flat set of dimensions describing a request: user_id, ip, api_key, route, method, anything else a rule might match on. The caller hands the whole bag over; rules pick which fields they key on.
  • Bucket — a (rule, identity_subset) pair mapping to one counter. The server-side identifier surfaced to clients as X-RateLimit-Bucket.
  • Decision — allowed, per-rule remaining / reset_at / scope, optional retry_after.

3. API (~5 min)

// Internal RPC the API gateway calls per request.
// The caller describes the request; the limiter owns rule lookup.
POST /v1/check {
  identity: { user_id?, ip?, api_key?, route, method, ... },
  cost: 1
} → {
  allowed,
  decisions: [ { rule_name, bucket_id, allowed, remaining, reset_at, scope } ],
  retry_after?
}

identity is one flat bag of request attributes — there's no useful distinction between "who" (user_id) and "what" (route) at this layer, because a rule like "5 logins/hour per IP per route" needs both, and only the rule knows which fields it cares about. The caller describes the request fully; the limiter projects identity down to the fields each matched rule keys on.

Why the caller doesn't pass rule_id. Rules are config owned by the limiter. Letting callers name rules couples every gateway to the rule catalog — a new policy means a gateway redeploy, and stale clients drift from the source of truth. By passing raw request context instead, the limiter can add, retire, or reshape rules without any client change.

Public response headers (when the gateway returns 429): X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset, X-RateLimit-Bucket, X-RateLimit-Scope (user / shared / global), Retry-After. Discord's X-RateLimit-Bucket is opaque so the server can re-shard bucketing without breaking clients — see Deep Dives.

Sidecar alternative. A library that talks to a Redis/Memcached pool directly — same algorithm, no extra hop. The sidecar pulls rules from the config service and matches locally; the public API surface to application code stays identical (it still passes the identity bag, not rule IDs). I'd offer both depending on overhead constraints.

5. High Level Design (~10–15 min)

  1. Limiter service / library. Receives the identity bag. Looks up matching rules from its in-memory rule table, projects identity to each rule's key dimensions, queries the counter store, aggregates decisions across layers. The caller never names a rule.
  2. Counter store. Redis Cluster. Each bucket has a string key (e.g., rl:user:42:postTweets:202605181847); Redis hashes the key into one of 16,384 slots and routes to the shard that owns that slot — so buckets spread evenly across shards and any single bucket lives on exactly one node. Counters live in memory with the algorithm's data structure (one TAT integer for GCRA, two counters for sliding window). Eviction by TTL = window length. Co-located with limiters (same AZ) to keep p99 under budget.
  3. Configuration service. Owns the rule catalog (source of truth). Pushes updates to limiter instances (long-poll or etcd/Consul). Limiter hot-reloads the matcher table atomically. New rules launch in observe-only mode first (see Deep Dives).
  4. Local pre-check / verdict cache. In-process state — see Deep Dives (Hot keys, Adaptive enforcement) — absorbs the easy cases without a network hop.
  5. Decision pipeline. Every decision emits (rule_name, allowed, layer, bucket_id, identity_hash) to a metrics + log pipeline. Operators see denial rates per rule; a sample lands in a structured log for offline tuning and dark-launch comparisons.

End-to-end narration. A request to POST /tweets arrives at the gateway. Gateway calls the limiter with identity={user_id:42, ip:1.2.3.4, route:/tweets, method:POST}. The limiter walks its layers in order — global shed (cheap, in-memory), per-route ("10k QPS on writes"), per-tenant ("100 req/min per user"), abuse ("recent invalid rate ok"). Each matched rule produces a bucket key (rl:user:42:postTweets:202605181847), updated atomically via a Lua script. If any layer denies, return 429 with Retry-After and bucket headers. Otherwise pass through.

6. Deep Dives (~10 min)

Concrete keys and memory.

Layer Key pattern Data structure / entry TTL Shard key
Global shed shed:{route}:{second} in-process counter, ~16 B 2–5 s local instance
GCRA bucket rl:{rule}:{identity_hash} Redis string = theoretical arrival time + metadata, ~50 B logical burst window + idle grace {rule}:{identity_hash}
Sliding counter rlsw:{rule}:{identity_hash}:{window} two Redis counters, ~32 B logical 2 × window {rule}:{identity_hash}
Local deny cache deny:{rule}:{identity_hash} in-process verdict + reset_at, ~64 B until reset local instance
Rule config ruleset:{version} serialized matcher table, KB–MB until superseded replicated to all

The identity_hash is server-generated from the dimensions the rule selects, so changing a rule's key shape creates a new bucket namespace without client changes.

Capacity by store / path.

Item Math Notes
Active shared buckets 100M × 50 B logical ≈ 5 GB; Redis overhead + replicas ≈ 30–60 GB dominant hot memory cost
Strict Redis ops 1M QPS × 1–3 matched strict rules = 1–3M ops/sec pipeline independent buckets
Local deny cache 1M hot denied buckets × 64 B ≈ 64 MB per large gateway tier attack traffic becomes cheap
Rule config thousands of rules × matcher data = KB–MB per instance replicated to every limiter
Decision stream 1M QPS × ~100 B ≈ 100 MB/sec if fully logged sample aggressively; metrics are aggregated
Quota exports day/hour aggregates keyed by tenant analytical view, not enforcement path

Check path with stores touched. 1. Gateway calls /v1/check with the flat identity bag and request cost. 2. Limiter matches rules in-memory; global shed and local deny-cache checks run first with no Redis hop. 3. For each strict matched rule, limiter builds a bucket key and pipelines Lua/GCRA commands to Redis. Independent rule buckets can run in parallel, but each individual bucket is atomic on one Redis shard. 4. Decisions are aggregated by most restrictive result. If any hard rule denies, return 429 with Retry-After; soft observe-only rules only emit metrics. 5. Decision events go async to metrics/logging; logging failure never blocks the request.

"My punch line: a rate limiter is a hierarchy of cheap local decisions and precise shared counters; making every rule pay the strongest consistency cost is the scalability bug."

Hot/cold bucket behavior. 1. New or dormant identity: Redis key is missing, so the Lua/GCRA script creates rl:{rule}:{identity_hash} with TTL = burst window + idle grace. No cleanup scan is needed. 2. Healthy cold bucket: optimistic local allow is permitted only for rules marked soft; async increment response carries the remaining quota signal. 3. Near-limit bucket: limiter flips to strict synchronous mode and updates the same Redis key atomically before allowing the request. 4. Exhausted bucket: limiter writes deny:{rule}:{identity_hash} locally until reset_at, so repeated abusive requests do not keep hitting Redis. 5. Rule shape changes: identity_hash namespace changes with ruleset:{version}, so old buckets expire naturally while new buckets start clean.

Two-view framing. The counter store is the enforcement index keyed by (rule, identity_subset). The decision stream is the analytical view keyed by rule, tenant, route, and time. Dashboards and quota exports can lag or sample; the Redis bucket decision cannot.

Algorithms. - GCRA (Generic Cell Rate Algorithm) — my default. One "theoretical arrival time" integer per bucket. Atomic in a single Redis command (CL.THROTTLE via redis-cell). Mathematically equivalent to leaky bucket but eliminates the drip-process failure mode and the non-atomic read-modify-write race that bites naive token buckets. - Sliding window counter — two fixed-window counters blended by elapsed fraction: rate = prev * (1 − elapsed/window) + curr. Cloudflare measured ~0.003% misclassification at production scale. Cheap and predictable. - Token bucket — natural for burst tolerance; needs careful Lua to stay atomic. Use when policy explicitly wants burst + steady-rate semantics. - Fixed window — only when the burst-at-boundary (2× limit possible) doesn't matter. - Sliding window log — exact but O(window × QPS) memory per bucket. Don't use at scale.

Counter synchronization — three models. - Centralized counter store (Redis). Stripe, Envoy, Kong. Single Redis Cluster, each bucket key hashed to one shard. Simple and exact. Cost: one network hop per check. My default for per-tenant and quota layers. - Anycast-pinned local counters. Cloudflare. Source-IP routing pins a flow to one PoP, so per-IP buckets live PoP-locally with no cross-PoP sync. Zero-hop, but unusable for per-user limits where the same user hits multiple PoPs. - Cooperative lease distribution. Google's Doorman. Limiters ask a master for a lease (e.g., 1000 req/s for 5s), refresh periodically. Trades per-request precision for the ability to enforce huge global limits without per-request hops. Good when limits are large and clients are many.

Why Redis rather than DynamoDB/Cassandra for the default counter store: the limiter needs single-digit-millisecond atomic increments with TTL, not durable query history. DynamoDB can do conditional writes but its tail latency and cost per request are a poor fit at 1M QPS; Cassandra is excellent for write logs, weaker for per-request atomic counters.

Hot keys. A famous user or scraper IP creates a hot bucket. Mitigations stack: - Cache the deny verdict locally once a bucket is over-limit (Cloudflare, Envoy freecache). Subsequent requests from that bucket skip the store entirely until TTL. Counterintuitive: backend load drops during an attack on the worst-case path. - Layer cheap before expensive. Per-IP at the edge (in-memory); per-user behind it on Redis. - Counter sharding for the absolute hottest buckets — hash into N sub-keys, sum on read. Trades a read for spreading the write hot-spot. - Client-side micro-cache of the allow decision (e.g., 50ms) on the gateway.

Edge stress tests.

Edge Failure mode Fallback / budget
Dormant identity bucket expired recreate on first request; one Redis hop
Exhausted hot bucket attacker hammers 429 path local deny cache until reset
Burst writes many identities become hot at once pipeline Redis, shed globally before per-user checks
Redis shard failure strict counters unavailable per-rule fail-closed or fail-open-with-floor
Multi-region drift same user hits multiple PoPs use central Redis for strict user limits or Doorman-style leases for large global quotas
Cold config rollout new rule too aggressive observe-only first, then enforce after dashboard review

Latency budget. 5 ms p99 means one Redis hop max. Co-locate Redis with gateways (same AZ). Connection pooling, pipelining, and Lua scripts keep wire round-trips to one.

Adaptive enforcement (predict instead of check). ≫99% of (identity, rule) pairs never approach their limit — paying a Redis hop on every request is wasteful. Maintain per-bucket state in the limiter: cold (counter well under limit, denial rate ≈ 0) and hot (near limit or denials rising): - Cold path: allow immediately, fire-and-forget the increment (batched / pipelined). Latency ≈ 0. - Hot path: synchronous check-then-allow via the Lua script. Strict.

Transitions: cold→hot when the counter crosses ~80% of the limit or async denials appear; hot→cold after a quiet window. The signal rides back on the async increment response — Redis tells us "you're at 95%," we flip to hot. Cost: brief over-allowance at the cold→hot edge. Acceptable for soft limits (pagination quotas); never for hard limits (auth, abuse). Per-rule opt-in.

Stage Mode Source Behavior
Cold optimistic allow local state + async Redis increment lowest latency, bounded over-allow
Warm sampled sync periodic Redis check detects drift before limit
Hot strict sync Redis Lua/GCRA exact decision before request passes
Store failed floor mode local safe-capacity bucket fail-open-with-floor or fail-closed by rule

Fail-open with a floor, not binary. If the counter store is unreachable, the binary choice (fail-open = abuse gets through; fail-closed = self-inflicted outage) is wrong for most rules. Each gateway carries a safe-capacity floor per identity (e.g., "if Redis is down, allow up to 10 req/s per user from local memory"). Hard rules (auth, payment) still fail-closed; soft rules fail-open-with-floor. Doorman models this explicitly; Stripe approximates it by fail-open + a concurrent-request limiter that keeps a faulty client from monopolizing workers.

Bucket discovery via response headers. Server returns X-RateLimit-Bucket: <opaque> so well-behaved clients group endpoints that share a bucket and back off correctly without hardcoding limits. Server controls bucketing, can re-shard, and clients adapt automatically. Pair with X-RateLimit-Scope so the client knows which layer fired (user / shared / global) and whether to back off this token or all tokens.

Cost model. Not all requests are equal: - Point system (GitHub: GET=1, mutation=5; 900 points/min). Counter increments by request cost — supported trivially by passing cost through the Lua script. - GraphQL query complexity — sum field costs at parse time, charge once. - Concurrent in-flight limiter — distinct from RPS. A user making 100 slow expensive requests can starve workers without exceeding RPS. Stripe runs this as its own layer; catches the case RPS limits miss.

Don't leak the limit (abuse layer). For adversarial rules, 429 + Retry-After tells the attacker exactly when to retry. Figma's pattern: shadow ban — return 200, silently drop or delay. The attacker can't tell they've been throttled and can't pivot to fresh credentials. Use shadow-ban only on suspected-bot layers; never on developer-facing API quotas where transparent feedback is the contract.

Dark-launch / shadow mode. Never enforce a new rule cold. Run it in observe-only mode for days first — count what would have been blocked, dashboard the verdicts, tune the threshold. Stripe and Envoy both build this in as a first-class mode. Cheap to add; saves you from accidentally 429-ing a critical batch job.

Multi-tenant fairness. Noisy tenant must not slow others. Partition Redis per-tenant (or per-tier) so one tenant's hot buckets don't evict another's. Combine with the per-tenant layer in the stack so an abusive tenant's denial work doesn't burn the shared budget.

Synchronizing config. Rule changes propagate via the config service. Limiter watches; on change, swaps the rule table atomically. In-flight requests finish on the old config — that's fine.

Clock skew. Counters live in one place at a time — the Redis server's clock (centralized) or the PoP-local clock (anycast). Don't trust client timestamps. For lease-based systems, leases are seconds-scale and NTP skew is negligible.

What I'd defer. ML-based novel-bot detection, CAPTCHA fallback, OAuth quota enforcement, billing-integrated metering, deeply hierarchical (org → team → user) quota allocation.
