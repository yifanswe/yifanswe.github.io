Price Tracking Service — Camelcamelcamel-Style (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a price tracking service. Users subscribe to products on Amazon/Walmart/etc., we crawl prices on a schedule, and we alert users when a product hits their target price. 10M users, 100M tracked products, prices crawled every 10 minutes for hot items, hourly for the long tail.

1. Requirements (~5 min)

Functional - Add a product URL to track; we extract the canonical product, current price, and start scraping it. - Set a target price; receive an alert (email/push) when current price drops to or below target. - View price history chart for a tracked product.

I'd defer cross-retailer matching ("is this the same product on Amazon and Walmart?"), browser extension, and affiliate revenue.

Non-functional - Crawl freshness within target cadence (10 min hot, 1 h long tail). - Alert delivery within 5 min of detected price drop. - Be polite to retailers — respect robots.txt, throttle per host, identify ourselves. - Survive retailer-side anti-bot measures.

Capacity - 100M products tracked. Each price record ~50 bytes. 100M × 100 (10-min crawls/day) = 10B price records/day if all hot — realistically a mix; say 1B/day → 50 GB/day. - 10M users; avg 5 alerts → 50M user-product subscriptions. - Price history dominates: 1B records/day × 50 B × 90 d × 3 replicas ≈ 13.5 TB for raw hot history before indexes. Rollups are much smaller and can live for years. - Catalog and subscriptions are modest: 100M products × ~1 KB × 3 ≈ 300 GB; 50M subscriptions × ~150 B × two indexes × 3 ≈ 45 GB.

2. Core Entities (~2 min)

  • Product — product_id, retailer, retailer_sku, canonical_url, name, image.
  • PriceRecord — product_id, ts, price, in_stock.
  • Subscription — user_id, product_id, target_price, alert_channel.
  • CrawlJob — product_id, next_crawl_at, cadence.

3. API (~5 min)

POST  /v1/products/track          { url, target_price? }   → { product_id }
GET   /v1/products/{id}           → { product, current_price, history_url }
GET   /v1/products/{id}/history?range=30d   → PriceRecord[]
DELETE /v1/subscriptions/{id}

User from token.

4. Data Flow (~5 min)

User adds URL → resolve canonical product → upsert into catalog → schedule crawl → crawler scrapes price → write PriceRecord → if price ≤ any subscription's target, fire alert event → notification service delivers.

5. High Level Design (~10–15 min)

  1. URL resolution service. When a user submits a URL, normalize it, fetch once, extract product identity (retailer + SKU). Dedupe against catalog — multiple users tracking the same product share one crawl.
  2. Catalog store. Products in MySQL (sharded by product_id). Subscriptions in a separate table indexed by both user_id (for "my list") and product_id (for fan-out on price change).
  3. Crawl scheduler. Priority queue (Redis sorted set by next_crawl_at). Each product has a cadence based on popularity (subscriber count, volatility).
  4. Crawler pool. Per-retailer workers pulling from the scheduler. Per-host rate limit. Uses headless browser for JS-heavy retailers; cheap HTTP for static pages. Rotates residential-IP proxy pool.
  5. Parser. Per-retailer parser modules. Versioned because retailers change HTML frequently.
  6. Price store. Time-series store for PriceRecord — InfluxDB or a sharded MySQL table keyed by (product_id, ts) with retention policies (raw 90 d, 1 h aggregates 1 y, daily aggregates forever).
  7. Alert pipeline. Each new PriceRecord publishes an event to Kafka. Alert worker: - Looks up subscriptions for the product (cached by product_id). - Filters to those with target_price >= new_price. - Deduplicates against last-alert-sent timestamp per subscription. - Emits alert events to the notification service.
  8. Notification service. Email / mobile push / browser push. Out of scope to design in detail.

End-to-end narration: Alice adds an Instant Pot listing → URL resolver extracts product PXC123 → catalog upsert → subscription row with target_price = $59. Crawler hits it on the next cadence, finds price $79, writes record. Two days later, price drops to $55 — alert worker reads the event, finds Alice's subscription with target $59 ≥ $55, hasn't alerted in 24 h, fires email + push.

6. Deep Dives (~10 min)

Concrete stores and shard keys.

Store Key pattern Data structure / entry TTL / retention Shard key
Product catalog product:{product_id} MySQL row, retailer SKU/name/image/canonical URL, ~1 KB forever product_id
User subscriptions subs_by_user:{user_id} rows/list of product targets, ~150 B/sub until deleted user_id
Product subscribers subs_by_product:{product_id} index of subscription IDs + target price, ~80 B/sub until deleted product_id
Crawl schedule crawl_due:{retailer} Redis ZSET, member=product_id, score=next_crawl_at, ~40 B no TTL; updated every crawl retailer
Price history price:{product_id}:{day} time-series partition, (ts, price, stock), ~50 B/record raw 90 d; hourly 1 y; daily forever product_id
Alert dedupe alerted:{subscription_id} last price + last sent timestamp, ~80 B TTL 30–90 d subscription_id

Redis stores scheduling indexes, not price history. The durable time-series store is the byte-heavy system.

Capacity by store.

Store / index Math Notes
Product catalog 100M products × 1 KB × 3 ≈ 300 GB canonical SKU content
User subscription index 50M subs × 150 B × 3 ≈ 22.5 GB user_id view
Product subscriber index 50M subs × 80 B × 3 ≈ 12 GB fan-out view on price drop
Crawl schedule 100M products × 40 B × Redis overhead ≈ 15–50 GB real memory mutable due-time index
Raw price history 1B/day × 50 B × 90 d × 3 ≈ 13.5 TB dominant durable store
Alert dedupe 50M subs × 80 B × 3 ≈ 12 GB TTL-bounded, avoids spam
Crawl throughput 1B crawls/day ≈ 11.6k/sec average per-retailer caps, not global capacity, usually bind first

Track-product path. 1. User posts a URL; resolver normalizes host/path and extracts (retailer, sku). 2. Catalog lookup uses unique key (retailer, sku); if absent, resolver fetches once and creates product:{product_id}. 3. Subscription service writes both subs_by_user:{user_id} and subs_by_product:{product_id} in one transaction or outbox-backed dual write. 4. Scheduler inserts/updates crawl_due:{retailer} with the product's cadence; if the product already had subscribers, no duplicate crawl is created.

Product read / history path. 1. GET /v1/products/{id} reads product:{product_id} and the latest price cache in parallel; if the latest cache is missing, it reads the tail of price:{product_id}:{today}. 2. GET /history?range=30d reads raw price:{product_id}:{day} partitions for the requested days, shard key product_id, and returns an opaque cursor {product_id, end_ts, granularity} for longer charts. 3. If the requested range extends beyond raw 90-day retention, the same cursor switches to hourly/daily rollups. The UI gets lower resolution, not an empty chart. 4. If a dormant product has no fresh crawl, the API returns the last good price with stale_at and optionally schedules a one-off refresh job; the page view should not synchronously scrape the retailer.

Two-index framing. Subscription is one logical dataset with two physical views: subs_by_user:{user_id} for "my tracked items" and subs_by_product:{product_id} for alert fan-out. The dual-write must be transactional or outbox-backed because a missing product index silently drops alerts.

Price-drop path. 1. Crawler pops due product IDs per retailer, respecting host token buckets. 2. Parser extracts price; writer upserts price:{product_id}:{day} with unique (product_id, ts_minute). 3. Price event goes to Kafka; alert worker reads subs_by_product:{product_id}, filters target_price >= price, and checks alerted:{subscription_id}. 4. Matching subscriptions are batched to the notification service; alert dedupe is updated after enqueue, not after email success, so retries do not spam.

Crawl scheduling and politeness. Per-host token bucket; never more than N concurrent requests to one retailer. Distribute work so multiple workers don't pile on the same host. Honor Retry-After and 429s.

Why Redis ZSET scheduler instead of scanning a SQL table: due-time pop is exactly a sorted-set operation, and scanning 100M products every minute is wasted work. A Kafka delay topic can work for immutable one-shot timers, but crawl cadence changes often with subscriber count and retailer backoff, so a mutable priority queue is cleaner.

Anti-bot defenses. Retailers actively block scrapers. Mitigations: residential proxy pool, realistic browser headers, randomized timing jitter, occasional cookie maintenance per host. Mention these as a known cat-and-mouse problem, not solved by us alone.

Cadence by popularity. A product with 10k subscribers gets crawled every 10 min. A product with 1 subscriber crawls hourly. Recompute cadence weekly based on subscriber count and historical price volatility.

Time-series storage and queries. Raw records keyed by (product_id, ts) — cheap append. For history charts, downsample on read or maintain rollups (1-min → 1-h → 1-d). InfluxDB or TimescaleDB native; on plain SQL, a partitioned table per month works.

For this workload I would choose TimescaleDB/Influx-style time-series storage if the team already operates it; otherwise a partitioned MySQL/Postgres table is acceptable because reads are almost always single-product range scans. Cassandra is a fit only if write throughput and multi-region active-active matter more than ad hoc chart queries.

Fan-out on price drop. A hot product (e.g., new iPhone) has tens of thousands of subscribers. A single price drop triggers ten of thousands of alerts. Defenses: - Batch alert dispatch into the notification queue. - Per-user rate-limit alerts (don't email someone 10 times in 5 min). - For very large subscriber sets, fan-out via a background worker pool rather than synchronously.

Idempotent crawls. Crawler may retry on transient failures; the price store should treat (product_id, ts_minute) as a unique key to avoid double-counting.

Detecting "real" drops. Avoid alerting on transient blips. Require N consecutive crawls below target, or 2 records ≥ 5 min apart, before firing.

Freshness state machine.

Stage Mode Source Transition
Fresh hot 10-min crawl crawl_due:{retailer} + latest price many subscribers or high volatility
Fresh long tail hourly crawl scheduler cadence low subscribers, stable price
Stale last good price price:{product_id} retailer 429/parser failure
On-demand refresh queued async refresh one-off crawl job user page view or manual refresh
Rollup-only history low-resolution chart hourly/daily aggregates raw 90-day window exhausted

Cross-retailer match (optional). "Same product, different retailer" requires product matching: name normalization + UPC if available + image embedding similarity. Out of scope to design in 45 min but worth mentioning.

Compliance and ToS. Retailers' terms often prohibit scraping. Mention the legal/ethical risk explicitly; real services like Honey use partner data feeds where possible.

Graceful degradation.

Case Mode Source Latency budget Fallback
Retailer 429 / block backoff Retry-After, per-host bucket next allowed crawl stretch cadence and mark price stale
Parser breaks stale price last good price:{product_id} alert delayed route sample pages to parser-fix queue
Dormant product cold crawl hourly/daily cadence best effort crawl on user page view to refresh
Hot product drop async fan-out subs_by_product:{product_id} alert within 5 min batch notification jobs; per-user cap
History range exhausted rollup read hourly/daily aggregates < 500 ms chart show lower-resolution history

"The interview punch line: users subscribe to products, but crawlers crawl canonical SKUs; that dedupe is what keeps 50M subscriptions from becoming 50M crawls."

What I'd defer. Affiliate-link monetization, browser extension architecture, "deal feed" curation, A/B testing alert templates.
