Game Leaderboard (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a multiplayer game leaderboard. Players post scores; we show the global top 100, the player's rank, and the player's neighbors (±5 around them). 50M monthly active players, peak ~100k score writes/sec during weekend tournaments. Leaderboards for multiple scopes: global, friends-only, region, time window (daily / weekly / all-time).

1. Requirements (~5 min)

Functional - Player submits a score; system records and updates leaderboard. - Query top-K globally / for a region / for the player's friends. - Query the player's own rank + their N neighbors.

I'd defer cheat detection depth and ranking tie-break rules beyond simple highest-score-wins.

Non-functional - Top-K query p99 < 100 ms. - Rank-of-player query p99 < 200 ms. - Eventual consistency on global ranks (< 5 s) is fine; player-self rank should be near-real-time. - Handle write spikes during weekend tournaments.

Capacity - 50M players; daily leaderboard ~5M active; weekly ~15M; all-time ~50M entries. - 100k score writes/sec peak. - Redis memory is manageable but not free: daily 5M × ~100 B × RF=2 is ~1 GB; weekly 15M is ~3 GB; all-time 50M is ~10 GB. The dominant long-term cost is score history, not the top-K index.

2. Core Entities (~2 min)

  • Player — player_id, region, friends.
  • Score — (player_id, scope, score, ts). Scope = daily, weekly, all, region:X, etc.
  • Leaderboard — derived, indexed by score.

3. API (~5 min)

POST  /v1/scores                     { score, mode }                          → { rank, percentile }
GET   /v1/leaderboards/{scope}?top=100                                        → { entries[] }
GET   /v1/leaderboards/{scope}/me                                             → { rank, score, neighbors[] }
GET   /v1/leaderboards/friends/me                                             → { friends_ranked[] }

User from token.

The rank-neighbor cursor is server-issued and encodes {scope, player_id, rank_snapshot_version, offset} so scrolling around a player does not drift while new scores arrive.

5. High Level Design (~10–15 min)

  1. Score ingest. Score POST → validated (basic sanity, anti-cheat hooks) → written to scores table (sharded by player_id) → published to Kafka for downstream.
  2. Leaderboard store: Redis sorted set per scope. ZADD leaderboard:daily:2026-05-18 score player_id. Per scope, one ZSET. ZADD GT keeps only the player's best score.
  3. Top-K queries. ZREVRANGE leaderboard:daily 0 99 WITHSCORES — O(K+log N). Cached briefly (1 s) for popular scopes.
  4. Player rank query. ZREVRANK leaderboard:daily player:42 — O(log N). Neighbors via ZREVRANGE around that rank.
  5. Friend leaderboards. Player's friend list intersected with a global ZSET; per-friend scores fetched in a single ZMSCORE call, then sorted client-side.
  6. Multiple scopes. Daily/weekly ZSETs roll over at boundary — drop yesterday's daily after the day ends. All-time keeps growing; periodically trim to top N + some buffer.
  7. Persistence. Authoritative scores in MySQL (sharded by player_id) — Redis is the index. On Redis loss, rebuild from MySQL.
  8. Region scopes. Separate ZSET per region (leaderboard:region:us-west). Lower cardinality, smaller sets, faster queries.

End-to-end narration: Alice scores 9500 → POST writes the score row, ZADD GT leaderboard:daily:2026-05-18 9500 alice, ZADD GT leaderboard:all 9500 alice. Service returns Alice's new rank (ZREVRANK leaderboard:daily alice → rank 5, ZREVRANGE 0 9 returns top 10). Friend leaderboard: pull Alice's friends [bob, carol, ...] → ZMSCORE leaderboard:daily alice bob carol ... → sort descending → render.

6. Deep Dives (~10 min)

Why Redis sorted sets. ZSET operations are O(log N) with very low constants — perfect for leaderboards. Range queries (ZREVRANGE) and rank lookups (ZREVRANK) at sub-millisecond. Memory cost: ~80 B per entry, so 50M players × 1 leaderboard ≈ 4 GB. Cluster shards by leaderboard key.

Why not MySQL-only or Cassandra-only: they can store the authoritative score, but rank and neighbor queries need ordered-set operations on every write. Redis ZSET is the read index; MySQL is the source of truth. If an interviewer pushes on Redis durability, the answer is rebuildability plus RF/AOF, not treating Redis as the only copy.

Capacity to say out loud: daily + weekly + all-time + 10 region boards are roughly (5M + 15M + 50M + 50M) × 100 B × RF=2, about 24 GB of Redis logical leaderboard entries before allocator overhead. MySQL score history is larger: 100k/sec peak is bursty, but even 1B score rows/year × 80 B × RF=3 is ~240 GB before indexes. The knobs are cap low-value windows, archive raw rows after season close, and keep all-time exact only for top ranks plus player-owned lookup.

Scaling a single hot leaderboard. Global daily leaderboard sees all 100k writes/sec. One Redis instance maxes out around 100k ops/sec. Mitigations: - Shard the leaderboard into N sub-ZSETs by player_id % N. Top-K reads merge sort across shards. - Write-batching: accumulate writes for 100 ms in the ingest tier, batch into one Lua-pipelined call. Lowers RTT cost; same total work but fewer round trips.

Top-K from sharded leaderboard. Each shard returns its local top K. Merge-sort to produce global top K. Same trick as YouTube Top-K (need local K' > K to guarantee correctness).

Top-K read path: client -> leaderboard API -> top-K cache lookup; on miss, parallel ZREVRANGE lb:{scope}:{window}:{0..N-1} 0 K' WITHSCORES; API merge-sorts, hydrates player profiles in one batch, writes top:{scope}:{window} for 1 s, returns.

Top-K cache miss
    API
     |-- shard 0: ZREVRANGE 0..K'
     |-- shard 1: ZREVRANGE 0..K'
     |-- shard N: ZREVRANGE 0..K'
     '-- profile MGET for winners
                -> merge by score/tie_ts -> top:{scope}:{window} TTL 1s -> client

This is a two-view design: the score row is content, while lb:{scope}:{window}:{shard} is the ranking index and top:{scope}:{window} is the read snapshot. They are derived from the same score event, so the service can rebuild every view from score history.

Rank of arbitrary player from sharded leaderboard. Harder. ZREVRANK player_42 in shard 5 only gives local rank. To get global rank, sum ZCOUNT score_above across all shards. Implement as a single Lua-pipelined call; still O(N_shards × log N).

Rank read path: find player_id % N for the local score, fetch score with ZSCORE, then parallel ZCOUNT shard (score,+inf) across all shards. Neighbor rows come from local shard plus nearby candidate ranges from adjacent score bands, then the API merge-sorts around the global rank.

The rank cursor stores {scope, window, player_id, score_watermark, shard_offsets[]}. If new scores arrive while the user scrolls neighbors, the cursor keeps the page anchored to the snapshot score rather than drifting.

Daily/weekly rollover. Daily ZSET key includes the date; at midnight, new key starts being used; old key kept for read access for 24 h then archived to S3. Weekly aggregates from daily — score = max over the week, stored in leaderboard:weekly:2026-W20.

All-time leaderboard size. 50M entries × 80 B = 4 GB Redis memory. Manageable. Trim to top 10k + each player's best (the player needs to know "where they rank" in tail too). Use a hybrid: top 10k in Redis, all others in a sharded MySQL index for the long tail.

State transition for the long tail:

Stage Mode Source Latency budget Cursor/state
Hot rank Exact Redis sharded ZSET + ZCOUNT <200 ms score_watermark
Tail rank Estimated percentile MySQL score histogram <300 ms score_bucket
Exact async Background recompute MySQL ordered index seconds job id returned internally
Rebuilt Fresh Redis view replayed score history minutes after incident versioned window

Friend leaderboards at scale. Player with 200 friends → ZMSCORE for 200 player_ids in one round trip. Sort 200 entries client-side. Fast (< 10 ms).

Anti-cheat / validation. Server-side score validation: signed score from the game client, sanity checks (impossible score patterns), rate-limit submissions per player. Cheaters' entries quarantined (separate ZSET) rather than removed.

Replay / disaster. Redis ZSETs are an index; authoritative store is MySQL. On Redis cluster loss, rebuild from MySQL by replaying all-time and recent scores.

Graceful degradation.

Case Mode Source User-visible behavior
Global ZSET hot Batch writes, serve cached top 100 Redis shards + 1 s cache top-K may lag 1-2 s
Player in trimmed all-time tail Estimate percentile, exact rank async MySQL score index show percentile first
Redis shard lost Freeze affected scope, rebuild from score history MySQL replay stale but available top-K
Friend list huge Cap to active friends, paginate friends graph + ZMSCORE exact within page
Tournament spike Contest queue isolated from casual modes dedicated Redis/Kafka pool casual ranks lag

Interview punch line: "The leaderboard itself is a secondary index; the durable fact is the player's best score, so losing Redis hurts freshness, not correctness."

What I'd defer. Detailed anti-cheat ML, season pass / battle pass progression, multi-game leaderboards, cross-platform identity, rewards distribution.
