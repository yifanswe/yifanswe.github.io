Design a Netflix-Style Recommendation System

2026/06/01  ·  Common Backend Systems

1. Basic Knowledge And Solution

What This Question Is Testing

A Netflix-style recommender chooses which titles to show, in what order, on the homepage. The hard part is computing personalized rankings for hundreds of millions of users across a 10k-title catalog, balancing recall, latency, freshness, and A/B experimentation.

The invariant is this:

Every user request to the homepage receives a ranked list of rows and titles within the latency budget, derived from a personalized candidate set plus an online ranker; signals (watch history, browsing, ratings) feed back into model retraining; A/B experiments isolate model changes for measurable evaluation.

At ICT5 / L6+ level, the interviewer is testing whether you can design offline + online recommendation pipelines, candidate generation (collaborative filtering, embeddings, popularity), ranking architecture, feature stores, A/B infrastructure, and reason about freshness vs cost.

60-Second Interview Framing

Spoken version:

I would design a two-stage recommender: an offline batch pipeline that pre-computes candidate sets per user (collaborative filtering, embedding similarity, popularity) and an online ranker that scores candidates at request time with personalization. A feature store provides offline and online features. A/B experimentation is pervasive. The interesting decisions are candidate generation, online inference latency, feature freshness, and experiment design.

Requirements To Clarify First

Functional requirements:

  • Generate personalized homepage with rows of titles per user.
  • Update recommendations daily and on engagement signals.
  • Support multiple rows: Trending, Continue Watching, Because You Watched X, For You.
  • Cold-start handling for new users.
  • A/B testing of models and UX.

Non-functional requirements:

  • Homepage render under 1 second.
  • Recommendations refreshed daily plus real-time signals.
  • 200M+ users.
  • 10k-title catalog.

Out of scope unless asked:

  • Search ranking specifics.
  • Auto-play decisions.
  • Trailer selection.

Offline Pipeline

Daily (or hourly) batch:

  1. Ingest engagement events from the last N days.
  2. Update user features: taste embedding from recent watches; recent genres, languages, completion patterns.
  3. Update title features: popularity, completion rate, watch trends.
  4. Train models: matrix factorization, deep collaborative filter, two-tower networks for embeddings.
  5. Compute candidates per user: - Collaborative filtering: titles watched by similar users. - Content-based: titles similar to recently watched (via embedding cosine). - Popularity in user's region. - "Because you watched X" row anchored on a recent watch.
  6. Store candidate sets in offline feature store.

Each user typically gets thousands of candidates across multiple row types.

Online Ranker

Per request:

  1. Fetch user's candidate sets from offline feature store.
  2. Fetch real-time features (just-watched, current session activity).
  3. Score candidates with online model (often a deep neural net).
  4. Apply diversity, deduplication, freshness.
  5. Pick rows + order titles.
  6. Return.

Latency budget: under 500ms total. Ranker inference targets 100-200ms.

Multi-stage ranking: - Cheap pre-rank prunes to top-K candidates per row. - Expensive ML ranker scores top-K.

Interview phrase:

Two-stage by design: offline pre-computes candidate sets per user from billions of events; online ranker scores top candidates with real-time features. Offline handles scale; online handles freshness.

Candidate Generation Strategies

Strategy How
Collaborative filtering "Users like you watched..." Matrix factorization or two-tower.
Content-based Embedding similarity to recent watches.
Popularity Top in region or category.
"Because you watched X" Anchored on specific recent watch.
Editorial / curated Hand-picked rows.
New releases Recent additions to catalog.
Continue watching In-progress titles.
Personalized search Query-driven.

Each contributes candidates; the homepage composer mixes and matches.

Feature Store

Two tiers:

Offline: Daily / hourly batch updates. User taste embeddings, title popularity, historical engagement. Large; bulk lookups.

Online: Real-time signals. Current session, just-watched, just-browsed. Low-latency point lookups.

Feature store provides consistent training/serving features (point-in-time correctness during training to avoid leakage).

A/B Experimentation

Pervasive at Netflix.

  • Experiment service assigns users to variants on stable hash of user_id.
  • Each variant tests a model change or UX change.
  • Engagement events tagged with experiment id + variant.
  • Daily analysis: A vs B on metrics (watch time, completion, retention).
  • Statistical significance test; ship winning variant.

Many experiments concurrent; clear factorial design and orthogonal variants.

Cold Start

New user has no watch history.

  • Fallback: popular titles in user's region.
  • Onboarding survey: "pick 3 titles you like" → seed taste.
  • Quickly converge: first few watches inform candidate gen within hours.

Engagement Capture

Client streams events: play, pause, skip, complete, rate, browse, hover. Events feed: - Offline pipeline (next day's candidate update). - Online feature store (real-time signal). - Experiment outcome metrics.

Diversity And Deduplication

Within a row: avoid showing similar titles (multiple sequels in a row).

Across rows: avoid same title in multiple rows.

Diversity injected by post-processing the ranker output: greedy selection with diversity penalty.

Multi-Region Strategy

  • User home region for profile.
  • Candidate sets replicated to nearest region.
  • Online ranker per region.
  • Engagement events per region with cross-region replication.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Ranker slow. Strict timeout; fallback to candidate set order. Ranker p99.
Feature store stale. Use cached; mark with timestamp. Feature freshness.
Cold start. Popular + onboarding seed. Cold-start request rate.
Model regression. A/B; auto-rollback on metric drop. Engagement delta.
Candidate set missing. Fallback to popularity. Missing candidate rate.
Event pipeline lag. Backpressure; alert. Pipeline lag.
Experiment leakage. Stable hash; per-experiment audit. Cross-variant contamination.
Diversity failure. Post-process check. Duplicate title rate.
Cross-region replication. Async; bounded. Lag.
Privacy violation. Per-user opt-outs; anonymized. Privacy audit.

Operational Metrics

Correctness:

  • Cold-start request rate.
  • Stale feature use.
  • Diversity violations.
  • Experiment contamination.

Capacity:

  • Ranker p99.
  • Feature store QPS.
  • Candidate set freshness.

Business:

  • Watch time per user.
  • Completion rate.
  • Click-through rate per row.
  • Retention.

Tradeoffs To Call Out

  • Offline candidates cheap to refresh; online ranker is the latency tax.
  • Deeper models give better ranking but more latency / cost.
  • Richer feature store helps but raises serving costs.
  • Aggressive personalization vs filter bubble risk.
  • Diversity hurts apparent relevance but improves long-term engagement.
  • Cold-start fallbacks vs forcing onboarding survey.

Interview Phrase

A Netflix-style recommender is two-stage: offline pre-computes per-user candidate sets from billions of events; online ranker scores top candidates with real-time features. Multiple candidate sources (collaborative filtering, content embeddings, popularity) feed different row types. Feature store has offline + online tiers. A/B experimentation is pervasive. 2. Simulated Interview

Interviewer: Design Netflix's recommender.

Candidate: Two-stage. Offline batch pipeline pre-computes per-user candidate sets daily from multiple sources: collaborative filtering, content embeddings, popularity. Online ranker scores top candidates at request time with real-time features. Feature store has offline + online tiers. A/B experimentation is pervasive. The invariant is sub-second personalized homepage with daily-fresh candidates plus real-time signals.

Interviewer: Walk me through serving the homepage.

Candidate: User requests /home. Backend looks up user's candidate sets in feature store — multiple sets, one per row type (Trending, For You, Because You Watched X, etc.). Each set has 1000s of titles. Online ranker scores top candidates per row using offline features (user taste embedding, title features) plus online features (current session, just-watched). Homepage composer picks rows and orders titles, applying diversity and dedup. Returns paginated page. Engagement events flow back into the pipeline.

Interviewer: Candidate generation strategies.

Candidate: Multiple. Collaborative filtering via matrix factorization or two-tower networks: "users like you watched X." Content-based: embedding similarity between titles I recently watched and the catalog. Regional popularity: top titles where I am. "Because you watched X" anchored on a recent watch. New releases. Continue watching. Editorial / curated. Each row type sources from one or more candidate generators. The mix is the design choice.

Interviewer: Online ranker.

Candidate: Multi-stage. Cheap pre-rank prunes candidate set to top-K (~100). Expensive ML ranker (deep model) scores top-K with extensive features. Returns ordered list. Strict timeout (~150ms); on timeout, fall back to pre-rank order. Features pre-computed where possible; computed inline if absolutely needed.

Interviewer: Feature store.

Candidate: Two tiers. Offline: daily batch updates. User taste embeddings, title popularity, historical engagement aggregates. Bulk lookups for ranker batch inference. Online: real-time signals — current session activity, just-watched. Sub-ms point lookups. Critical detail: training/serving consistency — features used at serving must be reproducible at training time (point-in-time correctness) to avoid leakage.

Interviewer: A/B testing.

Candidate: Every model change goes through A/B. Experiment service assigns users to variants via stable hash on user_id. Each variant tests a model or UX. Engagement events tagged with experiment id + variant. Daily analysis: A vs B on key metrics (watch time, completion, retention). Statistical significance tests determine winner. Multiple experiments concurrent; orthogonal designs reduce interference.

Interviewer: Cold start.

Candidate: New user has no watch history. Onboarding survey: pick a few titles you like — seeds taste features. Fallback to popular titles in their region. After first few watches, the standard pipeline takes over. Within a session or two, recommendations personalize.

Interviewer: Filter bubble?

Candidate: Diversity injected by post-processing the ranker output. Greedy selection with a diversity penalty: subsequent picks penalized for similarity to already-selected. Editorial rows surface content outside personalized predictions. We monitor diversity metrics; if a user's recommendations get too narrow, force in exploration.

Interviewer: A new title is added.

Candidate: Cold start for the title. Content-based candidates via title embedding (computed from genre, cast, synopsis). Initial popularity from marketing signals. After some watches, collaborative signals develop. Editorial promotion may help bootstrap.

Interviewer: Feature freshness.

Candidate: Offline features daily; online features within seconds. The just-watched signal is online and influences the very next ranker pass. Daily refresh updates taste embeddings and candidate sets. For very fresh signals (just clicked something), the online feature store carries them with sub-second lag.

Interviewer: Multi-region.

Candidate: User home region for profile and historical features. Candidate sets replicated to nearby regions. Online ranker per region. Engagement events per region with cross-region replication for global model training.

Interviewer: Summarize.

Candidate: Two-stage recommender: offline batch pre-computes per-user candidate sets from multiple sources (collaborative filter, content embedding, popularity); online ranker scores top candidates with real-time features and personalization. Feature store has offline + online tiers with training/serving consistency. A/B experimentation pervades. Cold start via onboarding + popular fallback. Diversity injected at post-processing. The hardest performance constraint is online ranker latency; the hardest correctness concern is feature freshness vs cost.
