Donations Website — GoFundMe Style (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Easy · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a donations site. Organizers create campaigns with a goal. Donors browse, contribute money, and a live progress bar updates. Spikes happen — a viral disaster campaign can attract 10k donations in an hour. Steady-state low, viral peaks the design driver.

1. Requirements (~5 min)

Functional - Organizer creates a campaign: title, story, goal amount. - Donors view campaign, contribute via card. - Campaign page shows live total + recent donor list.

I'd defer organizer payouts, identity verification, and refunds.

Non-functional - Correctness on money is the headline — no double-charge, no lost donations. - Campaign page p99 < 500 ms even during virality (10k concurrent viewers). - Strong consistency on amount-raised counter. - Eventual consistency fine for the "recent donors" list (within a few seconds).

Capacity - Baseline: 10k active campaigns, 1k donations/day. - Viral campaign: 10k donations in an hour → ~3/sec on a single campaign, 100k viewers concurrent. - The read spike is the real scale driver: 100k viewers polling every 2 s is 50k live-widget QPS for one campaign. Cache the live block aggressively, but keep the money ledger strongly consistent.

2. Core Entities (~2 min)

  • Campaign — campaign_id, organizer_id, title, story, goal_amount, raised_amount, status.
  • Donation — donation_id, campaign_id, donor_id, amount, message, anonymous, status.
  • Payment — handled by a payment gateway (Stripe). We hold the reference.

3. API (~5 min)

POST  /v1/campaigns             { title, story, goal }       → { campaign_id }
GET   /v1/campaigns/{id}        → { campaign, recent_donations[] }
POST  /v1/campaigns/{id}/donations    { amount, payment_method, anonymous? }    (Idempotency-Key)   → { donation_id, status }
GET   /v1/campaigns/{id}/donations?cursor=...

User from token. Idempotency-Key required on donation.

Donation responses are safe to retry because the idempotency key maps to a persisted donation attempt: {donor_id, idempotency_key} -> donation_id, status, gateway_charge_id}.

5. High Level Design (~10–15 min)

  1. Campaign service. CRUD on campaigns. Sharded MySQL by campaign_id.
  2. Donation service. Receives donation request, creates donation row (status=pending), invokes payment gateway, on success transitions to succeeded and atomically increments the campaign's raised_amount.
  3. Payment gateway integration. Stripe (or similar). We don't store PAN; we use Stripe tokens. Idempotency keys passed through.
  4. Live counter. raised_amount cached in Redis with write-through from the donation service. Campaign page reads the cached value plus recent donations.
  5. Recent donors list. Append-only per campaign — Redis list of last 50 donors, evicted to MySQL on overflow.
  6. Campaign page. CDN-cacheable for static parts (story, images). Dynamic parts (total raised, recent donors) served from a tiny API call that hits Redis.

End-to-end narration: Donor opens campaign page → static content from CDN; AJAX call to /campaigns/123/live → Redis returns {raised: 47600, recent: [...]} → progress bar renders. Donor donates $50 → API call (Idempotency-Key) → donation row created (pending) → Stripe charge succeeds → atomic update: campaign.raised += 50, donation.status = succeeded, recent list prepended → next page poll sees the new total.

Donation write path:

  1. API gateway authenticates donor and checks (donor_id, idempotency_key) in MySQL.
  2. Donation service inserts donation(status=pending) and a ledger intent row in the same transaction.
  3. It calls Stripe with the same idempotency key; Stripe may return success, failure, or timeout.
  4. On success, a DB transaction marks donation succeeded, inserts immutable ledger credit, and increments campaign.raised_amount.
  5. After commit, it updates Redis campaign_live:{campaign_id} and recent_donors:{campaign_id}; if Redis fails, the DB remains correct and the cache is rebuilt on next read.

Campaign live read path:

  1. Browser loads static campaign HTML/images from CDN using campaign_id as the cache key.
  2. It calls /v1/campaigns/{id}/live; the service reads Redis campaign_live:{campaign_id} and recent_donors:{campaign_id} in parallel.
  3. On cache miss, it reads MySQL campaign row by campaign_id, reads the most recent successful donations from the donation index, rebuilds Redis with a 1-5 s TTL, and returns.
  4. Cursor for donor pagination is {campaign_id, last_created_at, last_donation_id} rather than offset, so new donations do not shift page boundaries.
  5. If Redis is down during a viral event, the service serves a CDN/edge-cached live block for 1-2 s or falls back to MySQL with aggressive request coalescing per campaign.
Donate:  Client -> Donation Service -> MySQL donation pending -> Stripe
                                      -> MySQL ledger + raised_amount commit
                                      -> Redis live cache + recent donors

Read:    Browser -> CDN static page
                 -> Live API -> Redis live/recent -> MySQL rebuild on miss

6. Deep Dives (~10 min)

Counter consistency under spike. On viral campaigns, donations come in fast. The campaign row becomes a hot row. Two options: - Single-row CAS update: UPDATE campaigns SET raised = raised + 50 WHERE id = X. Fine up to a few hundred QPS on one row. - Sharded counter: N counter rows; sum on read. Scales further. For donations volumes (< 100/sec even for the hottest), the simpler single-row CAS is enough; mention sharded as the scale-out.

Concrete stores:

Store Key pattern Value / data structure Entry size TTL / retention Shard key
Campaign DB campaign_id campaign row + raised_amount ~2 KB forever campaign_id
Donation DB donation_id; unique (donor_id, idem_key) donation status row ~1 KB forever / audit campaign_id or donor_id by query need
Ledger DB ledger_entry_id immutable debit/credit entries ~200 B forever campaign_id
Live cache campaign_live:{campaign_id} raised amount, goal, status ~200 B 1-5 s TTL, write-through campaign_id
Recent donors recent_donors:{campaign_id} Redis LIST last 50 donor snippets ~150 B/item 1 h or refreshed campaign_id
Donor page index (campaign_id, created_at, donation_id) successful donation IDs ~40 B/index entry forever campaign_id
Idempotency idem:{donor_id}:{key} donation_id, status, gateway_charge_id ~200 B 24 h+ donor_id
Webhook queue stripe_webhook:{event_id} durable event payload ~1 KB 30 days event_id

Capacity math: 1M donations/day at ~1 KB donation row + 200 B ledger row + 40 B donor index entry ~= 1.24 GB/day before indexes and replicas, tiny compared with media/story assets. Idempotency at 1M/day x 200 B is only hundreds of MB/day with TTL. The viral read cache is the hot path: 50k QPS hitting a 200 B Redis value is easy; 50k QPS hitting MySQL is self-inflicted pain. Recent donors for 10k active campaigns x 50 x 150 B ~= 75 MB before Redis overhead. Dominant correctness cost is operational, not storage: reconciling Stripe, ledger, and campaign totals.

There are two views of donations. The immutable ledger is the money truth, keyed by ledger_entry_id and campaign. The campaign page is a serving view: campaign.raised_amount, campaign_live:{campaign_id}, and recent_donors:{campaign_id}. If the serving view is stale, rebuild it from the ledger/donation rows; never rebuild the ledger from the progress bar.

Idempotency. Donation submission idempotent on (donor, key). Critical because slow networks during checkout cause retries. Cached response prevents double-charge.

Payment integration. We hand off the card to Stripe Elements (client-side), get a token, then call our API with {token, amount}. We never touch PAN — PCI scope minimized.

Why MySQL + Stripe + Redis. MySQL is the right default for money because we need transactions, unique idempotency constraints, and audit queries. A distributed NoSQL counter would scale writes but make reconciliation harder. Stripe owns card handling and idempotent charging; building a processor is out of scope and changes the compliance problem. Redis is only a serving cache for the campaign page, never the source of truth for money.

Failure between charge and ledger update. Stripe says "charged"; our DB update fails. Two paths: - Webhook from Stripe confirms; our reconciliation handler retries the ledger update. - Polling sweeper looks for pending donations older than 5 min, queries Stripe, transitions correctly.

Donation state machine:

State Source of truth Transition User-visible result
pending donation row request accepted, Stripe call in flight spinner / pending receipt
succeeded_unposted Stripe charge DB update failed after charge hidden; webhook/sweeper repairs
succeeded ledger + donation row ledger committed, counter incremented thank-you + progress update
failed Stripe response card declined / fraud reject no counter change
refunded ledger reversal refund committed raised amount decremented

Live-page degradation state:

Stage Mode Source User-visible behavior
normal fresh live block Redis campaign_live + recent_donors progress updates within seconds
cache miss rebuild MySQL campaign + donation index one slower request, then refill
viral spike edge coalescing CDN caches live JSON for 1-2 s many viewers share one origin read
Redis down DB-safe fallback MySQL raised amount, recent list may hide progress lags, donations still accepted
payment drift reconciliation Stripe webhook/sweeper -> ledger repair user receipt may show pending

For a viral campaign, if Redis is exhausted, serve the campaign story from CDN and show the last committed total from MySQL with a short "updating" state. Never accept a donation without an idempotency key, and never trust the cache for money.

Edge stress tests:

Edge Design response Budget
Hot campaign read storm CDN static page + 1-2 s cached live block + Redis origin campaign page p99 <500 ms
Burst donations on one campaign single-row increment is fine to ~hundreds/sec; switch to sharded counter rows if needed no lost increments
Donor retries after timeout unique (donor_id, idempotency_key) returns same donation/gateway charge no double charge
Cursor exhausts recent list donor pagination falls through to MySQL index (campaign_id, created_at, donation_id) stable order
Multi-region drift payments and ledger writes route to campaign home region; read replicas serve campaign page money ordered in one region
Cold start after cache flush request coalescing lets one rebuild populate Redis while others serve stale edge JSON avoid DB stampede

Interview punch line: the live page is a cached projection of the ledger, not a participant in the money transaction.

Recent donors list. Per-campaign Redis list (LPUSH + LTRIM to 50). Reads from Redis are sub-millisecond. Anonymous donors show as "Anonymous" with amount.

Campaign page caching. Static story rendered as HTML and CDN-cached (5 min TTL, purged on edit). Dynamic block (counter + recent) loaded via JS — cached for 1–2 s at CDN edge to absorb viral spikes.

Fraud and abuse. Heuristics: stolen-card velocity, donation-then-refund patterns, organizer-self-donation. Webhook listeners watch for chargebacks; surface to ops.

Refunds. Stripe Refund API + reverse the raised_amount increment. Done synchronously in the donation service; donation row status → refunded.

What I'd defer. Organizer KYC and payouts, currency conversion, comment moderation, recommendation of "campaigns like this."

Interview punch line: treat donations as a ledger first and a live counter second. The progress bar can lag by a second; a double charge or missing ledger entry is the real failure.
