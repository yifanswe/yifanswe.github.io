1. Basic Knowledge And Solution

What This Question Is Testing

The Twitter home timeline is the canonical hybrid-fan-out problem at staff-level depth. The hard part is delivering a personalized, low-latency feed to hundreds of millions of users without paying push cost for celebrity tweets or read cost for normal users.

The invariant is this:

Every follower's home timeline eventually contains every tweet from authors they follow, ordered by recency or ranked score, subject to visibility filters; reads return in sub-second p99; writes don't fan out to more than the threshold number of inboxes.

At ICT5 / L6+ level, the interviewer is testing whether you can articulate the hybrid push/pull (fan-out) decision, design the inbox cache, merge celebrity author timelines at read, apply visibility filters, and reason about hot users and replication.

60-Second Interview Framing

Spoken version:

I would design the Twitter timeline as a per-user inbox cache holding recent tweet ids from followed authors plus a pull-at-read-time path for celebrity authors above a follower threshold. Read merges the inbox with celebrity author timelines, hydrates tweet content, applies visibility filters, optionally ranks. The interesting decisions are the celebrity threshold, inbox size cap, hydration cost, and visibility enforcement.

Requirements

Functional requirements:

  • User opens home; sees recent tweets from followed authors.
  • Order: reverse-chronological (or ranked).
  • Pagination; load more.
  • Filter blocked/muted authors.
  • Deletes propagate fast.

Non-functional requirements:

  • Read p99 < 200ms.
  • Bounded staleness (seconds).
  • Available even if the write path is degraded.
  • 500M+ MAU.

Out of scope unless asked:

  • Tweet creation (see Design Twitter).
  • Ranking ML in depth.
  • Search.

Data Model

Inbox Entry

InboxEntry(
  user_id,
  tweet_id,
  author_id,
  inserted_at
)

Stored in Redis sorted set per user, sorted by tweet_id (snowflake; time-ordered). Capped to last N entries (e.g., 800).

Author Timeline Cache

AuthorTimeline(
  author_id,
  recent_tweets: [(tweet_id, posted_at)],
  cache_until
)

For all authors but more critical for celebrities.

APIs

GET /home?cursor=&limit=
GET /users/{id}/timeline             # author timeline

Read Path

  1. Client requests /home.
  2. Timeline service fetches inbox cache for user (last N tweet ids).
  3. Looks up follow graph: which followed authors are celebrities?
  4. For each celebrity, fetch their recent author timeline cache.
  5. Merge: sort by tweet_id (which embeds time) — recent first.
  6. Take page of K results.
  7. Hydrate: batch GET on tweet store for tweet content.
  8. Visibility filter: drop tweets from blocked/muted, deleted tweets.
  9. Optional ranker pass with timeout.
  10. Return.

Latency budget: - Inbox fetch: 5-10ms (Redis). - Celebrity merge: 10-20ms (small set of caches). - Hydration: 30-50ms (batched fetch from tweet store). - Visibility: <10ms (cached). - Ranking: <100ms with timeout. - Total: <200ms.

Inbox Cap And Eviction

Inbox holds last N tweet ids. Cap chosen to cover typical "load more" pagination without overflow.

When user follows new authors, their inbox doesn't backfill — they see new tweets going forward. (Optionally, a backfill job can populate recent tweets from new follows.)

Cache eviction: LRU; cold inboxes evicted. On miss, reconstruct from follow graph + author timelines (slow path).

Celebrity Merge

For each celebrity followed: - Fetch their recent author timeline cache (small, cached). - Merge into the read result.

If user follows 200 normal authors + 10 celebrities, the read does 1 inbox read + 10 author timeline reads — manageable.

For a user following hundreds of celebrities, the merge cost grows linearly. Tradeoffs: cap the celebrity set considered per request, or treat top celebrities as inbox-eligible.

Hydration

Tweet ids → content. Batch fetch from tweet store. Hydration includes: - Tweet text and metadata. - Media URLs. - Author info (denormalized). - Engagement counts.

Hydration is the largest latency contributor. Cache hydrated tweets at the timeline service.

Visibility Filter

Applied at read, never baked into inbox.

  • Blocks: user X's blocks list; filter their tweets.
  • Mutes: similar.
  • Deletes: tweets marked deleted in tweet store — hydration skips.
  • Privacy: protected accounts visible only to approved followers.
  • Safety / policy: removed tweets.

The visibility filter is a final gate. Even if inbox contains a tweet id, the read service may skip it.

Ranking

Optional reorder of candidates by ML score.

  • Side service.
  • Strict timeout (e.g., 100ms).
  • On timeout, fall back to reverse-chronological.
  • Features: viewer-author affinity, tweet engagement, recency.

Tradeoffs To Call Out

  • Larger inbox cap supports deeper scroll but uses more memory.
  • Aggressive celebrity threshold reduces fan-out cost but increases per-read celebrity merges.
  • Pre-materialized inbox is fast but storage-heavy.
  • Read-time visibility is correct but adds lookup cost.
  • Ranking improves engagement but adds latency and complexity.

The Twitter timeline is hybrid fan-out: normal authors push tweet ids into follower inboxes; celebrities are pulled at read. Reads merge inbox with celebrity author timelines, hydrate, apply visibility filters, optionally rank. Tweet store is authoritative; inboxes hold ids only. 2. Simulated Interview

Interviewer: Design the Twitter home timeline.

Candidate: Hybrid fan-out: per-user inbox cache holds tweet ids from normal authors; celebrity authors are pulled at read time. Read merges the inbox with celebrity author timelines, sorts by time (or rank), hydrates tweet content, applies visibility filters. The invariant is sub-200ms reads with eventually-consistent freshness. Tweet writes feed into either the fan-out workers (for normal authors) or remain pull-only (for celebrities).

Interviewer: Why hybrid?

Candidate: Pure push fails for celebrities — one tweet from a 100M-follower account would mean 100M inbox writes, taking minutes. Pure pull fails for the read path — every read would aggregate from hundreds of followed authors. Hybrid splits at a threshold: small-following authors push; large-following authors pull. The read does inbox + small set of celebrity merges.

Interviewer: Where do you put the threshold?

Candidate: Tune to minimize total cost: fan-out write cost vs read amplification savings. Per author with F followers posting P tweets/day, push cost is F × P. Read cost for that author is R reads/day across followers × pull-per-read. Find the F where push = pull. Typically 10k-1M followers. Adjust dynamically as authors gain followers.

Interviewer: Read path.

Candidate: Inbox fetch from Redis (sorted set; last N entries). Identify which followed authors are celebrities (from follow graph). For each celebrity, fetch their cached recent author timeline. Merge inbox + celebrity tweets sorted by tweet_id (snowflake = time-ordered). Take page. Hydrate tweet content via batched tweet store fetch. Apply visibility filter — drop blocked/muted/deleted. Optional ranker pass. Return.

Interviewer: What's the latency budget?

Candidate: Inbox fetch ~5ms. Celebrity merges ~10-20ms. Hydration ~30-50ms (the biggest piece). Visibility ~10ms. Ranking 100ms with timeout. Total under 200ms p99. Hydration dominates; we cache hydrated tweets to reduce.

Interviewer: Inbox cap.

Candidate: ~800 entries typical. Covers typical "load more" pagination. When new authors are followed, inbox doesn't backfill historically — user sees new tweets going forward. Long-scroll past the cap requires either pre-cache extension or pulling from authors directly.

Interviewer: Deletes — propagation.

Candidate: Tweet store marks the tweet deleted. Hydration sees the status and returns nothing (or a tombstone). Inbox entries remain — tweet ids point at deleted records. The visibility filter handles deletes at read. Caches of hydrated content invalidate on delete via pub/sub.

Interviewer: Block — does the inbox need to be rewritten?

Candidate: No. Blocks are applied at read by the visibility filter. The inbox still has the blocked user's tweet id, but the read service drops it before returning. Saves billions of inbox writes on every block change.

Interviewer: A user follows 1000 celebrities.

Candidate: Read merges 1000 celebrity author timelines — too many. Mitigations: cap the celebrity set per request to top-K by interaction history. For the rest, treat as inbox-eligible (move them out of the celebrity bucket for this user). Or scope to recently active celebrities. Edge case; most users follow far fewer.

Interviewer: Ranking?

Candidate: Side service. After candidate generation, send candidate tweet ids + viewer context to ranker. Ranker scores and returns ordered ids. Strict timeout (100ms); fall back to chronological if it times out. Features pre-computed: viewer-author affinity, tweet engagement, recency.

Interviewer: Multi-region.

Candidate: Per-user home region. Inbox cache per region. Tweet store replicated globally. Celebrity author timelines replicated to nearby regions. Reads from local region. Fan-out workers in user's home region. Cross-region replication is async with bounded lag.

Interviewer: Summarize.

Candidate: Hybrid fan-out (push for normal, pull for celebrity). Per-user inbox cache in Redis holds tweet ids; sized to last N. Read merges inbox + celebrity author timelines, hydrates via batched tweet store fetch, applies visibility filter at read, optionally ranks with timeout. Multi-region per-user home. The hardest performance constraint is sub-200ms with hydration as the dominant cost; aggressive caching of hydrated tweets mitigates.

3. Practice Notes

Five-Minute Version

  1. Hybrid fan-out: push for normal, pull for celebrity.
  2. Inbox cache (Redis sorted set) per user.
  3. Celebrity author timelines cached.
  4. Read merges inbox + celebrities.
  5. Batched hydration from tweet store.
  6. Read-time visibility filter.
  7. Optional ranker with timeout.
  8. Multi-region with per-user home.

One-Minute Closeout

Hybrid fan-out splits tweet propagation: small-following authors push to follower inboxes; celebrities are pulled at read. Inbox is a Redis sorted set of tweet ids per user. Read merges inbox + celebrity author timelines, hydrates, applies visibility filter, optionally ranks. Tweet store is authoritative; inboxes are projections. The hardest performance constraint is sub-200ms reads; hydration dominates.

Best Interview Phrases

  • "Push for normal authors, pull for celebrities; thresholded by total cost."
  • "Inboxes hold tweet ids, not content; hydration at read is batched."
  • "Visibility — blocks, mutes, deletes — is applied at read, never baked into inbox."
  • "Ranker is a side service with strict timeout and chronological fallback."
  • "Per-user home region; caches per region with global tweet store."
