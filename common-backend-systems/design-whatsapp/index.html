1. Basic Knowledge And Solution

What This Question Is Testing

WhatsApp is similar to Facebook Messenger but with two-billion-user scale, mobile-first network conditions, and end-to-end encryption as a non-negotiable. The hard part is delivering every message exactly once to every device across flaky cellular networks worldwide, with E2E encryption that the server cannot read.

The invariant is this:

Every sent message is persisted server-side as ciphertext, delivered at least once to every recipient device, displayed in the same order across devices, and remains available until acknowledged; end-to-end encryption is enforced so the server never sees plaintext; group sends fan out without leaking conversation content to the server.

At ICT5 / L6+ level, the interviewer is testing whether you can design Signal-protocol-style E2E encryption, handle mobile network unreliability, support multi-device sync without sacrificing E2E security, and reason about group fan-out under encryption constraints.

60-Second Interview Framing

Spoken version:

I would design WhatsApp around per-user message inboxes on the server holding encrypted messages waiting for delivery, a long-poll or persistent push channel to mobile devices, and Signal Protocol for end-to-end encryption. The server stores and routes ciphertext but cannot read it. Multi-device sync requires key transfer between user devices. The interesting decisions are E2E with multi-device, group encryption via Sender Keys, mobile reliability, and offline delivery.

Requirements To Clarify First

Functional requirements:

  • 1:1 and group messages (text, media, voice notes).
  • Voice and video calls.
  • Status updates (24-hour ephemeral).
  • End-to-end encryption mandatory.
  • Read receipts, delivery indicators, typing.
  • Multi-device (phone primary + linked devices).
  • Offline message delivery; queue server-side.
  • Backup (encrypted) to cloud.

Non-functional requirements:

  • 2B+ users, hundreds of billions of messages/day.
  • Mobile-first: poor cellular, intermittent connectivity.
  • Low latency for online delivery (sub-second).
  • E2E encryption non-negotiable.
  • Survive carrier-level outages with local queueing.
  • Multi-region; cross-country sends quick.

Out of scope unless asked:

  • Voice/video call media servers (WebRTC; separate).
  • WhatsApp Business API features.
  • Payments.

Signal Protocol Basics

Every user has:

  • Identity key (long-term, public/private).
  • Signed pre-key (medium-term).
  • One-time pre-keys (uploaded in batches, consumed once each).

X3DH (start a conversation):

  1. Alice fetches Bob's identity + signed pre-key + one one-time pre-key from server.
  2. Alice computes shared secret using all these keys.
  3. Alice sends initial message + her ephemeral key to Bob.
  4. Bob computes the same shared secret.

Double Ratchet (per message keys):

  • Each message uses a unique symmetric key derived from a ratchet.
  • The ratchet advances with each message (chain key) and on receipt of new DH ephemeral (root key).
  • Provides forward secrecy: compromise of current key doesn't reveal past messages.

Multi-device: - Each device has its own identity key and pre-keys. - Sender fetches each recipient device's bundle and encrypts the message N times, once per device. - For groups, this scales as sender_devices × group_size × devices_per_member.

Sender Keys For Groups

Encrypting each message per recipient device for a 1000-member group is heavy. Sender Keys reduces this:

  • Sender generates a symmetric key for the group.
  • Sender encrypts the symmetric key to each recipient device via Double Ratchet (one-time setup or on member change).
  • For each subsequent message, sender encrypts message with the symmetric key; sends ciphertext + ratchet step to all members.
  • Each recipient decrypts using the previously received symmetric key.

This reduces per-message work from O(N × devices) to O(1) once the key is established.

Interview phrase:

Per-device encryption scales poorly for groups. Sender Keys lets one ciphertext serve all members, with key setup amortized at member-change events.

Send Flow

  1. Sender's client generates message.
  2. For each recipient device, encrypts using Signal session (or group symmetric key).
  3. Sends bundle to send service: {recipient_device: ciphertext}.
  4. Send service writes each (recipient_device, ciphertext) to that device's inbox.
  5. If device is online (push channel connected), push immediately.
  6. Otherwise, the ciphertext waits in inbox.
  7. Sender receives ack from server (received).
  8. Recipient device decrypts on push, displays.
  9. Recipient sends delivered receipt; later sends read receipt.
  10. Server can delete delivered ciphertext (or retain briefly for cross-device sync).

Mobile Reliability

WhatsApp's secret weapon is mobile network engineering.

  • Persistent push channel with adaptive heartbeats.
  • Background notification triggers app wakeup.
  • Aggressive compression and message batching.
  • Network-quality-aware retry strategies.
  • Bandwidth-efficient encoding (custom protocols).

The client maintains a small local DB of recent messages; on send, queue locally; on connect, drain queue.

Multi-Device

WhatsApp's multi-device is unusual: historically the phone was the primary; linked devices proxied through it. Modern multi-device:

  • Each device has its own identity.
  • Sender sends to all of recipient's devices.
  • Per-device key bundles in the directory.
  • New device must verify with primary (QR code or push notification).

History sync to new device:

  • On link, primary device sends historical message keys + ciphertext (or plaintext via E2E channel) to the new device.
  • Subsequent messages are sent to the new device by senders directly.

Group Messaging

Same model as Messenger but with E2E:

  • Group registry holds membership.
  • Sender fetches member device list.
  • Sender encrypts using Sender Keys (symmetric).
  • Server fans out ciphertext to each member device.
  • Member changes (add/remove): rekey the group (rotate sender key).

For 1024-member groups, fan-out is 1024 inbox writes per message — still manageable.

Voice And Video Calls

Out of scope mostly, but: signaling via the messaging substrate, media via WebRTC peer-to-peer or through TURN relays. E2E for calls uses SRTP with keys exchanged via the same Signal session.

Backup

User-encrypted backup to user's iCloud/Google Drive.

  • Local backup file encrypted with a key derived from user's password or hardware-bound key.
  • Cloud provider sees only ciphertext.
  • Restore requires the user's password or device biometric.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Recipient offline. Inbox retains ciphertext until delivered. Inbox age distribution.
One-time pre-keys exhausted. Client uploads more periodically. Pre-key exhaustion rate.
Device key compromise. Forward secrecy via Double Ratchet; re-key on detection. Reported compromises.
Multi-device key sync failure. New device requires re-verification. Linking failure rate.
Group fan-out lag. Parallel fan-out; backpressure. Per-message fan-out time.
Push channel drops. Background reconnect; fallback to pull. Push reconnect rate.
Server holds ciphertext indefinitely. Cleanup after delivery + TTL. Inbox storage growth.
Sender Key rotation needed (member removed). Rekey on remove; all members get new key. Rekey frequency.
Carrier-level outage. Local queueing on device; deliver when network returns. Per-region delivery rate.
Spam/scam at scale. Reputation; rate-limit; community reports. Reported messages.

Operational Metrics

Correctness:

  • Lost message rate (target zero).
  • Out-of-order delivery rate.
  • Decryption failure rate.
  • Receipt accuracy.

Capacity:

  • Send QPS by region.
  • Push connection count.
  • Fan-out latency for groups.
  • Inbox storage.

Business:

  • DAU; messages per user per day.
  • Group size distribution.
  • Multi-device adoption.

Multi-Region Strategy

  • User home region for inbox and identity.
  • Push gateways regional; cross-region routing if user is roaming.
  • Key directory replicated globally for fast bundle lookup.
  • Failover regions warm.

Tradeoffs To Call Out

  • E2E encryption blocks server-side moderation; safety relies on reports and metadata analysis.
  • Per-device encryption scales poorly for groups; Sender Keys amortize but add complexity.
  • Strong order per conversation is hard with multi-device; rely on timestamps + tiebreakers.
  • Aggressive mobile optimization reduces battery cost but requires custom protocols.
  • Server holding ciphertext is necessary for offline delivery but adds storage and risk.
  • Multi-device security requires careful linking and verification.

Common Traps And Stronger Moves

One identity per user. Per-device identities for multi-device.

Interview Phrase

WhatsApp is per-device encrypted message queueing with Signal Protocol. The server is a router that sees ciphertext only. Sender Keys make groups feasible; per-device inboxes make multi-device work. The hard parts are mobile reliability, group encryption efficiency, and safety without server-side content access.

2. Simulated Interview

Interviewer: Design WhatsApp.

Candidate: I will design around per-device ciphertext inboxes, Signal Protocol for E2E encryption, and persistent push channels for online delivery. The server stores and routes ciphertext; plaintext exists only on user devices. Multi-device is solved by per-device identities. Groups use Sender Keys to amortize encryption cost. The invariant is that every message is durably stored until delivered to all recipient devices, with E2E enforced.

Interviewer: Walk me through sending a 1:1 message.

Candidate: Sender's client gets the recipient device's pre-key bundle from the key directory. Runs X3DH to establish a shared secret, then Double Ratchet derives per-message keys. Encrypts the message. Sends to the server: (recipient_device_id, ciphertext, ratchet_metadata). Server writes to the recipient device's inbox. If the recipient device is online (push channel connected), push immediately. Recipient device decrypts using its Double Ratchet state. Sends delivered receipt back; later read receipt. Server clears the ciphertext after delivered ack.

Interviewer: Multi-device E2E.

Candidate: Each device has its own identity key and pre-key bundle. When sender sends a message, they fetch all recipient devices' bundles and encrypt the message separately to each device. So if recipient has 3 devices, sender sends 3 ciphertexts. On the sender's side, their other devices also need a copy — they encrypt to themselves too. Server fans out to all relevant inboxes.

Interviewer: That's expensive for groups.

Candidate: Yes — for a 1000-member group with 2-3 devices each, naive per-device encryption is 2000-3000 encryptions per message. Sender Keys fix this. The sender generates a symmetric key for the group; once it's been established (via per-device encryption at group creation or on member change), all subsequent messages use the symmetric key. Per message: encrypt once symmetrically; ratchet step. Send ciphertext to all members. They decrypt with the shared symmetric key. Amortizes the cost.

Interviewer: When does the symmetric key change?

Candidate: On membership change. When someone is removed, we rekey so they cannot decrypt future messages. When someone joins, we either rekey or transfer the current key encrypted to them. Sender Keys typically rotate on member change events.

Interviewer: Recipient is offline for 3 days.

Candidate: Ciphertext sits in their device inbox on the server. When they come online, their device connects via push channel, gets all queued messages, decrypts. Server gets delivered acks for each, clears them. We retain undelivered ciphertext for a window (e.g., 30 days) per WhatsApp policy; after that, drop with a "message expired" indicator.

Interviewer: How do you handle flaky mobile networks?

Candidate: Persistent push channel with adaptive heartbeats — pings less often on stable WiFi, more often on cellular. Aggressive compression; bandwidth-efficient protocol. Client keeps a local DB of outgoing messages; queues them when offline; drains on reconnect. Server-side, we retry pushes with exponential backoff. Background push notifications wake the app for delivery.

Interviewer: Multi-device linking — how does a new linked device get history?

Candidate: User scans a QR code on the primary device. The primary derives a shared secret with the new device, then sends historical messages — keyed with derived encryption keys — via the messaging substrate. The new device gets recent history; very old messages may not transfer to keep linking fast. After linking, the new device is in the key directory; senders send to it directly for new messages.

Interviewer: Group of 1024 members. How does fan-out work?

Candidate: The sender encrypts the message symmetrically using the Sender Key. Sends one outgoing request to the server containing the ciphertext and the recipient list (group membership). The server fans out — writes the ciphertext to each member's inbox in parallel. 1024 inbox writes per message. Online members get push; offline get queued. Each member's client decrypts independently.

Interviewer: What about server-side spam moderation? E2E means you can't see content.

Candidate: Right; we cannot inspect content. We rely on metadata: send rates, recipient acceptance, reports. If user X sends thousands of messages per minute to strangers, we throttle and may suspend. Community reports surface spam patterns. Forwarded message indicators discourage chain-spreading. The tradeoff: stronger E2E privacy means weaker content moderation; we lean on behavioral signals.

Interviewer: Backup?

Candidate: Encrypted backup to user's cloud (iCloud/Google Drive). Local backup file is encrypted with a key derived from a user-set password or from a device-bound credential. The cloud sees ciphertext only. Restore requires the password. If the user forgets, backup is lost.

Interviewer: Multi-region?

Candidate: User home region for inbox. Push gateways are regional; user roaming routes to their home region. Key directory replicated globally for fast bundle lookup. Failover regions warm. Cross-region message routing happens at the server layer; ciphertext goes from sender's region to recipient's region inbox.

Interviewer: Summarize.

Candidate: WhatsApp is per-device ciphertext inboxes with Signal Protocol E2E. Server routes ciphertext, never sees plaintext. Multi-device works via per-device identities and per-device encryption. Groups use Sender Keys to amortize fan-out. Mobile reliability via persistent push, local queueing, adaptive protocols. Spam moderation relies on metadata not content. Backup is user-key-encrypted to cloud. The hardest constraints are E2E with multi-device and group efficiency.

3. Practice Notes

Five-Minute Version

  1. Clarify: 1:1, group, E2E mandatory, multi-device, mobile.
  2. Invariant: ciphertext to recipient device, server never plaintext.
  3. Signal Protocol: X3DH, Double Ratchet.
  4. Per-device inboxes + persistent push.
  5. Sender Keys for groups.
  6. Multi-device: per-device identities; sender encrypts N times.
  7. Mobile reliability via persistent push + local queueing.
  8. Spam moderation via metadata; user backups encrypted client-side.

One-Minute Closeout

WhatsApp is Signal Protocol E2E messaging at planet scale. The server stores and routes ciphertext to per-device inboxes; the persistent push channel delivers when devices are online; the local queue absorbs flaky networks. Groups use Sender Keys to amortize per-device encryption. Multi-device is solved by per-device identities. Spam moderation relies on metadata, not content. Backups are user-key encrypted to cloud. The hardest constraints are E2E with multi-device and group efficiency.

Best Interview Phrases

  • "Server stores ciphertext only; the plaintext lives on devices."
  • "Sender Keys make groups feasible; without them, per-device encryption blows up."
  • "Multi-device works by per-device identities, not one key per user."
  • "Mobile reliability is custom protocols, not just better servers."
  • "E2E privacy and content moderation are in tension; we rely on metadata."
