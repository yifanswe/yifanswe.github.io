Food Review App — Restaurant Reviews (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a food review app — a focused subset of Yelp. Users write reviews of dishes (not just restaurants), browse top-rated dishes near them, and follow other reviewers. 10M users, 5M restaurants, 100M dishes, ~1M new reviews/month.

1. Requirements (~5 min)

Functional - Find dishes near me, sorted by rating. - Read and write a review of a specific dish (rating + text + photo). - Follow another reviewer; see their recent reviews in a feed.

I'd defer reservations, group dining suggestions, and recommendations beyond simple rating.

Non-functional - Search results p99 < 300 ms. - Eventual consistency on aggregate ratings (within seconds). - Photo storage at scale. - Review fraud filtering — out-of-scope to design but mentioned.

Capacity - 100M dishes; 1M reviews/month = ~12/sec average; spikes around dinner. - Photos: 70% of reviews include 1+ photo; ~1M photos/month at 500 KB each = 500 GB/month. - Dominant cost is media, not review text: review rows at ~1 KB × 12M/year × RF=3 are only ~36 GB/year; photos at 500 GB/month raw × 3 derived sizes × RF=2 are ~3 TB/month before CDN cache.

2. Core Entities (~2 min)

  • User — user_id, profile, follows.
  • Restaurant — restaurant_id, name, geo, hours.
  • Dish — dish_id, restaurant_id, name, price?, avg_rating, review_count.
  • Review — review_id, dish_id, user_id, rating, text, photos[], created_at.
  • Photo — photo_id, url, dish_id, uploader.

3. API (~5 min)

GET   /v1/dishes/search?q=&lat=&lng=&min_rating=4              → Dish[]
GET   /v1/dishes/{id}                                          → Dish + top_reviews
GET   /v1/dishes/{id}/reviews?cursor=...
POST  /v1/dishes/{id}/reviews             { rating, text, photo_ids[] }   → { review_id }
POST  /v1/photos                          (multipart upload)              → { photo_id }
GET   /v1/feed                                                            → Review[] from followees
POST  /v1/follows                         { user_id }

User from token.

Cursors are opaque and server-issued. Review cursors encode {dish_id, last_created_at, last_review_id}; feed cursors encode {user_id, last_score, last_review_id} so inserts do not shift pagination.

5. High Level Design (~10–15 min)

  1. Catalog service. Restaurants + dishes. Restaurants seeded from public datasets / partner feeds; dishes mostly user-contributed with merge tooling. Sharded by restaurant_id.
  2. Search index. Elasticsearch over dishes — full-text on dish name, restaurant name, tags; geo-distance filter on the parent restaurant location; sort by Bayesian rating + distance + recency.
  3. Review service. Writes reviews keyed by (dish_id, user_id) (each user has at most one review per dish). Publishes review.created events.
  4. Aggregator pipeline. Stream processor maintains (sum_rating, review_count) per dish; updates avg_rating back to the catalog and search index.
  5. Photo pipeline. Uploads to object store; resize worker generates 3 sizes; CDN-fronted.
  6. Feed service. Reviewer-follow feed. Hybrid push/pull like FB News Feed but simpler scale: push on follow → reviewer's reviews appear in followers' feed lists. Most users follow < 100 people; pull-on-read works.
  7. Notification. "Reviewer you follow posted a review" — out of scope to design.

End-to-end narration: Alice searches "ramen" near Mission St. → search service hits ES with full-text + geo-distance → returns top 20 dishes ranked by Bayesian-adjusted rating, distance, recency. Alice eats and writes a 5-star review of Mensho's miso ramen with a photo → photo upload returns photo_id → review POST writes the row → aggregator updates Mensho's miso ramen avg_rating from 4.6 (200 reviews) to 4.61 (201). Search index updated within seconds.

Concrete stores I would name out loud:

Store Key / structure Entry size TTL / retention Shard key
Review DB review:{dish_id}:{review_id} row, plus unique (dish_id,user_id) index ~1 KB without photos forever, soft-delete dish_id
Rating aggregate dish_rating:{dish_id} -> {sum,count,prior_version} ~32 B forever dish_id
Search index dish:{dish_id} ES document with dish, restaurant geo, rating fields ~600 B rebuilt from catalog dish_id / geo partitions
Feed cache feed:{user_id} Redis ZSET of review_id scored by rank time, cap 200 ~64 B/member 30 days user_id
Reviewer recent index reviewer_recent:{user_id} Redis list of review IDs ~16 B/id, cap 50 30 days user_id
Photo object photo/{hash}/{size}.jpg immutable object 50 KB-1 MB forever, tier after 90 days content hash

Capacity math: the review DB grows slowly, about 12M reviews/year × 1 KB × RF=3 = ~36 GB/year. Feed indexes are also small if capped: 10M users × 200 ids × 64 B is ~128 GB logical before replication, and most users are inactive so keep only active feeds hot. Photos dominate by orders of magnitude, so object-store tiering and CDN hit rate matter more than SQL sharding.

6. Deep Dives (~10 min)

Dish identity. Dishes are user-contributed within a restaurant. Two reviewers may add "Spicy tuna roll" and "Spicy Tuna Roll" — duplicate dish entries. Defenses: - Fuzzy match at create time; suggest existing dishes. - Periodic deduplication job using normalized name + restaurant_id. - Eventually-consistent merge of duplicate dishes (preserving review pointers).

Bayesian-adjusted rating. A 5-star dish with 1 review shouldn't beat a 4.8-star with 200. Apply Bayesian prior: adjusted = (C × m + sum) / (C + n) where m is global mean (~4.0), C is prior weight (~10). Recompute as the aggregator updates.

Photos. Upload via signed URL directly to S3. Async worker generates 200²/600²/1200² thumbnails. CDN-fronted with cache-forever URLs (URLs are content-hash-based, so changing the file changes the URL).

Follow feed. Each user's feed list is a small Redis sorted set (capped 200). On review.created, push review_id into followers' lists (skip pushes for celebrity reviewers with > 10k followers; pull-on-read for those). Feed read merges followee reviews + assembles full review payloads.

Read path trace: GET /feed reads feed:{user_id} and the user's celebrity-follow list in parallel; pulls recent celebrity reviews from reviewer_recent:{reviewer_id}; dedupes by review_id; batch-hydrates review rows and dish rows; then returns the first page. Cache miss on feed:{user_id} falls back to pulling the last ~20 reviews from each followee, capped at a 250 ms budget.

Interview punch line: "The feed cache is an index of review IDs, not the review content; that keeps fan-out cheap and lets me rehydrate with fresh moderation and rating state."

Write path for a review:

POST review -> validate one review per dish/user
            -> write review:{dish_id}:{review_id}
            -> publish review.created
            |-- aggregate dish_rating:{dish_id}
            |-- update dish search document
            '-- fan out review_id to feed:{follower_id}, except celebrity reviewers

This is the two-view pattern: review:{dish_id}:{review_id} is the content, feed:{user_id} is the reader-keyed index, and reviewer_recent:{user_id} is the writer-keyed index for pull fallback.

Search ranking. Inputs: text relevance (BM25), distance, Bayesian rating, review count, reviewer authority (followers of the reviewer), freshness. Linear combination tuned offline.

Why Elasticsearch rather than PostGIS-only: PostGIS is great for nearby restaurants, but this workload needs text relevance, typo tolerance, geo filtering, and ranking in one query. A pure SQL/PostGIS design works for v1 nearby browse; ES/OpenSearch is the better read index once dish-name search and ranking become first-class.

Search path trace: client -> search service -> ES query over {dish_name, restaurant_name, geo_cell, adjusted_rating} -> top 50 dish_ids -> parallel batch fetch to catalog cache and rating aggregate -> trim to 20 -> cursor contains {query_hash, last_score, last_dish_id}.

Cold search fallback: if the ES cluster is degraded, use top_dishes:{geo_cell}:{category} from Redis, then hydrate dish pages from the catalog. It loses keyword precision but keeps the app usable for nearby browse within ~500 ms.

Fraud filtering. Heuristics: IP overlap with restaurant owner, account age, review velocity, content similarity to known templates. Flag for review; don't auto-delete.

Pagination. Cursor-based (server-generated opaque cursor) for review lists and feeds. Prevents pagination drift on inserts/deletes.

Caching. Top dish lookups (popular dishes) cached for 60 s. Hot restaurants pre-warmed.

Graceful degradation.

Case Mode Source Latency budget
Hot restaurant Serve cached dish cards and stale rating Catalog cache + rating aggregate <100 ms
Search index lag Show stale ranking with fresh dish page on click ES + catalog DB <300 ms
Feed cache exhausted Pull recent followee reviews and rebuild async Review DB + follow graph <800 ms
Photo variant missing Serve original or lower-res variant Object store/CDN <1 s first hit
Aggregator delayed Display last aggregate with "count may lag" semantics Rating aggregate seconds of drift

Feed degradation state:

Stage Mode Source Latency budget
Hot feed push index feed:{user_id} + review hydrate <200 ms
Celebrity mix hybrid pull feed:{user_id} + reviewer_recent:{id} <300 ms
Feed exhausted cold pull follow graph + recent reviews per followee <800 ms
Dormant user lazy rebuild review DB + follow graph async job first page small, rebuild in background

Interview punch line: "The product looks social, but the hard boundary is still content versus indexes: reviews are durable content; feeds and search are disposable views."

What I'd defer. Reservation integration, social graph importing, ML-based "you might like" recommendations, multi-language reviews and translation, business owner verification.
