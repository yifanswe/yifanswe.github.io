1. Basic Knowledge And Solution

What This Question Is Testing

LeetCode runs user-submitted code against hidden test cases, returns verdict (Accepted, Wrong Answer, TLE, Runtime Error), tracks contest standings. The hard part is securely running untrusted code in isolation, fast verdict turnaround under contest load, and scoring contests in real time.

The invariant is this:

Every submission runs in an isolated sandbox with deterministic resource limits, is judged against the canonical test cases, produces a verdict deterministic from the run, and updates the user's profile and contest standing within seconds; malicious code cannot escape the sandbox.

At ICT5 / L6+ level, the interviewer is testing whether you can design sandboxed code execution (containers, gVisor, Firecracker), judging fleet scaling under contest bursts, contest scoring (penalty system), and reason about cheat detection.

60-Second Interview Framing

Spoken version:

I would design LeetCode as a submission service that queues code for judging, a fleet of isolated judge workers running submissions in sandboxes (containers + seccomp, or Firecracker), a verdict service producing the result, and a contest scoring service tracking standings. Sandboxing security and contest burst handling are the central challenges. The interesting decisions are sandbox tech, judge fleet scaling, contest scoring, and cheat detection.

Requirements To Clarify First

Functional requirements:

  • Users submit code to a problem.
  • System runs against test cases.
  • Returns verdict: Accepted, Wrong Answer, TLE, MLE, Runtime Error, Compilation Error.
  • Tracks user's accepted problems, ranking.
  • Contests with leaderboards.
  • Multiple languages.
  • Discussion forums (out of scope).

Non-functional requirements:

  • Verdict in seconds for normal submissions.
  • Contest burst: thousands of submissions per second at contest start.
  • Strong sandbox security (untrusted code).
  • Resource limits enforced.
  • Multi-region for users globally.

Out of scope unless asked:

  • Problem authoring tools.
  • Premium subscriptions.
  • Mock interviews.

Submission Flow

  1. User POSTs code.
  2. API validates: code size, language, problem accessibility.
  3. Persist Submission with state pending.
  4. Enqueue to judge queue.
  5. Return submission_id; user polls or subscribes via SSE.
  6. Judge worker picks up.
  7. Compile (if needed).
  8. Run against each test case in sandbox with limits.
  9. Verify output against expected.
  10. Aggregate verdict.
  11. Update Submission record.
  12. Update user profile and contest standings if applicable.
  13. Notify user via SSE.

Sandbox Security

Running arbitrary user code requires strong isolation.

Options (security increasing):

  1. Container with seccomp: restrict syscalls. Cheap to start, decent isolation. Vulnerable to kernel exploits.
  2. gVisor: user-space kernel; stronger isolation; some overhead.
  3. Firecracker: lightweight VM; hardware isolation; ~125ms boot.
  4. Full VM: strongest; slow to start.

For LeetCode scale: typically container + seccomp + restricted user + cgroups for resource limits is sufficient with hardening. Firecracker for paranoid setups.

Restrictions: - No network (typically). - Limited filesystem (only test inputs). - Time and memory cgroups. - No /proc access. - Read-only root filesystem. - Drop all capabilities.

Resource Limits

Each test case run with limits: - CPU time (e.g., 1 second for typical problems). - Wall clock time (often 2-3x CPU limit). - Memory (e.g., 256 MB). - Output size (prevent infinite output). - Process count.

Enforced via cgroups + rlimit. If exceeded: - CPU/wall: TLE (Time Limit Exceeded). - Memory: MLE (Memory Limit Exceeded).

Judging Logic

For each test case: 1. Set up sandbox with input file. 2. Run binary with input as stdin (or as args). 3. Capture stdout, stderr. 4. Compare stdout to expected output. 5. If matches: pass. If not: WA.

Comparison: exact string match for most problems. Custom checkers for problems with multiple valid outputs.

Compilation

For compiled languages: compile once per submission; use cached binary for each test case.

Compile-time errors → CE (Compilation Error).

For interpreted languages: skip compile step.

Contest Scoring

Real-time leaderboard updates.

ICPC-style: each problem is binary (solved or not); penalty = time of first AC + 20 minutes per wrong attempt.

IOI-style: partial credit per test case.

Implementation: - ContestSubmission per (user, problem) tracks attempts and best score. - On new submission, recompute user's contest score. - Sub-counter sharding for hot contests. - Leaderboard cached; refreshed every few seconds.

Burst Handling (Contest Start)

Thousands of submissions at contest start.

  • Judge queue absorbs burst.
  • Auto-scale judge fleet pre-contest (pre-warm).
  • Priority queue: contest submissions prioritized over practice.
  • Resource pool dedicated to contests.

Pre-contest activities: - Pre-warm sandboxes. - Pre-load problem test cases on judge workers. - Validate judge fleet capacity.

Cheat Detection

Code similarity across submissions.

  • Tokenize and fingerprint code.
  • Compare against other submissions.
  • Flag if similarity > threshold.
  • Manual review for contests.

Also: behavioral signals (extremely fast solve, identical timing patterns).

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Sandbox escape. Multiple layers; audit. Detected escapes (zero).
Judge worker crash. Retry submission on different worker. Worker health.
Contest burst overload. Pre-scale; priority queue; auto-scale. Queue depth.
Plagiarism / cheat. Async detection; manual review. Flag rate.
TLE due to slow worker (not user code). Re-run on different worker; standardize hardware. Per-worker performance.
Test case leak. Strict access control; no leakage in error messages. Audit access.
Submission lost. Durable queue; retry. Lost submission rate.
Verdict streaming dropped. Polling fallback. Stream failure rate.
Leaderboard lag. Sub-counter; cache. Lag distribution.
Cross-region replication. Async; bounded. Lag.

Tradeoffs To Call Out

  • Stronger sandbox (Firecracker) = more secure but slower start.
  • Per-test-case isolation vs single sandbox per submission (faster but slightly less isolated).
  • Real-time leaderboard vs eventually consistent (UX vs cost).
  • Pre-warming for contests vs cost.
  • Strict TLE vs allowing minor variance (hardware-dependent).
  • Cache test cases on workers vs always pull (privacy + perf).

2. Simulated Interview

Interviewer: Design LeetCode.

Candidate: Submission service queues code for judging; fleet of sandboxed judge workers runs against hidden test cases; verdict service aggregates; user profile and contest scoring updated. The invariant is every submission produces a deterministic verdict from sandboxed execution within configured resource limits, with user profile and contest leaderboard updated within seconds.

Interviewer: Sandboxing — how?

Candidate: Layered. Linux container with restricted user, seccomp filtering syscalls, cgroups for CPU/memory limits, read-only filesystem with only test inputs accessible, no network, drop all capabilities. This is sufficient for most cases. For stronger isolation (paranoid setups), gVisor (user-space kernel) or Firecracker (lightweight VM with hardware isolation). The container overhead is small; Firecracker adds ~125ms boot.

Interviewer: Walk me through a submission.

Candidate: User POSTs code. API validates (size, language). Persists Submission as pending. Enqueues to judge queue. Returns submission_id; user polls or subscribes via SSE for verdict. Judge worker picks up. Compiles (cached if compiled lang). For each test case: sets up sandbox with input, runs binary with CPU/memory limits, captures output, compares to expected. Aggregates verdict (AC if all pass, else WA/TLE/etc.). Updates Submission. Triggers user profile + contest score updates. Notifies user.

Interviewer: Contest burst — thousands per second.

Candidate: Pre-warm judge fleet before contest. Auto-scale during. Priority queue: contest submissions get higher priority than practice. Dedicated worker pool for contests (avoid practice traffic contention). Queue absorbs the burst; users see slight delay but submissions don't drop. We monitor queue depth; if too deep, surface degraded ETA to users.

Interviewer: TLE — how do you make it fair?

Candidate: Standardize hardware: all workers identical hardware, same kernel, same OS. CPU time measured (not just wall clock) to ignore other processes. Wall clock as safety net. Pinning CPU cores per sandbox to reduce noise. Periodically re-run reference solutions to verify worker calibration. If a worker reports significantly different times, flag and remove.

Interviewer: Leaderboard in real-time.

Candidate: Contest score per user computed from ContestSubmissions. On each verdict, async aggregation updates user's contest score. Leaderboard cached and refreshed every few seconds. For very-hot contests, sub-counter sharding on score storage. WebSocket or SSE pushes updates to clients viewing the leaderboard.

Interviewer: Cheat detection.

Candidate: Code similarity analysis async. Tokenize the submission, compute fingerprint. Compare against other recent submissions to the same problem. Plagiarism detectors like Moss-style. Flag for manual review if similarity > threshold. Behavioral signals: extremely fast solve, identical timing patterns across users, copy-paste markers. In contests, post-contest review can disqualify cheaters.

Interviewer: What about test cases — leak prevention?

Candidate: Test cases stored in protected service; only judge workers can access them. Workers fetch test cases per problem; cache locally. Test cases never appear in error messages or verdict responses (otherwise users would extract them). Diff output truncated. Only sample test cases visible to users (small subset).

Interviewer: Multi-language support.

Candidate: Per-language config: how to compile, how to run, file extension. Judge workers have all language toolchains pre-installed. Different time and memory limits per language (Python is slower than C, so allow more time). Some problems have language-specific limits.

Interviewer: Multi-region.

Candidate: Submission API and judge workers per region. Users submit locally for low latency. Problem test cases replicated globally (immutable, easy to replicate). User profiles sharded. Contests have one global leaderboard; submissions from any region feed into it. Slight cross-region replication lag is acceptable.

Interviewer: What if user code tries to break out?

Candidate: Defense in depth. Seccomp limits syscalls so the code can't exec arbitrary binaries, ptrace, modify network state. Read-only filesystem so no malicious file writes. Network disabled. cgroups prevent fork bombs. Even if a kernel exploit exists, attacker gets only sandbox privileges. Audit logs detect attempts. For high-assurance, Firecracker isolates at the hypervisor level.

Interviewer: Summarize.

Candidate: Submission queue + sandboxed judge fleet + verdict aggregator. Each submission runs in container with seccomp, cgroups, no network, read-only FS. Per-test-case execution with strict resource limits. Contest scoring async with sub-counter sharding for hot contests. Pre-warm and priority queue handle contest bursts. Cheat detection async. Multi-region with global contest state. The hardest security problem is sandbox escape; defense in depth is the answer.

3. Practice Notes

Five-Minute Version

  1. Clarify: submission, sandbox, contests, languages.
  2. Invariant: sandboxed; deterministic verdict; resource-limited.
  3. Submission queue + judge fleet + verdict aggregator.
  4. Sandbox: container + seccomp + cgroups + read-only FS, no network.
  5. Per-test-case resource limits.
  6. Contest scoring async; pre-warm for bursts.
  7. Cheat detection (code similarity + behavioral).
  8. Multi-region with global contests.

One-Minute Closeout

A code judging system is an async pipeline: queued submission → sandboxed judge worker → verdict → profile and contest score update. Sandbox uses container + seccomp + cgroups + read-only FS + no network; defense in depth against untrusted code. Per-test-case resource limits. Contest bursts handled by pre-warming and priority queue. Cheat detection runs async via code similarity. Multi-region for low latency; global contest state.
