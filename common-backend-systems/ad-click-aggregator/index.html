Ad Click Aggregator (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build the system that counts ad clicks. Every click on a display ad routes through us. We attribute it (which ad, which campaign), dedupe bots, and aggregate counts that advertisers see on a dashboard. Real-time dashboards within seconds, exact-ish counts within an hour. 10B clicks/day, ~115k/sec average, ~1M/sec during major events.

1. Requirements (~5 min)

Functional - Click goes through us; record click event with attributes. - Aggregate counts at per-ad / per-campaign / per-advertiser, per-minute / per-hour / per-day. - Dashboard shows near-real-time counts.

I'd defer impressions, attribution (last-touch vs multi-touch), and billing reconciliation depth.

Non-functional - Counts approximately correct in real-time (~5 s lag); exact counts within 1 hour. - Idempotent click ingestion — never double-count a deduped click. - High availability — every uncounted click is potentially uncollected revenue. - Survive 10× spike on Super Bowl day.

Capacity - 10B clicks/day × ~500 B per event = ~5 TB/day raw. - Aggregates: ~10M unique (ad_id × minute) buckets/day. Multiple roll-up granularities. - Average ingest is ~115k clicks/sec; design for ~1M/sec peak. At 3× replication, 7-day Kafka retention is ~105 TB before compression. Raw event storage dominates cost; aggregate buckets are small.

2. Core Entities (~2 min)

  • Click — click_id, ad_id, campaign_id, advertiser_id, user_id, ts, ip, ua, referrer.
  • Aggregate bucket — (ad_id, minute) → count. Same for (campaign, minute), (advertiser, minute), etc.
  • Dedup record — click_id (signed) to prevent reuse.

3. API (~5 min)

GET   /click/{signed_token}                  // user-facing redirect; we record then 302 to ad URL

// Internal dashboard:
GET /v1/advertisers/{id}/stats?range=1h     → { per_ad: { count }, totals }

Signed token contains ad_id, campaign_id, exp, nonce, HMAC.

The stats cursor is time-based rather than offset-based: {advertiser_id, start_minute, end_minute, rollup_level, last_bucket} so pagination does not drift while fresh buckets are still changing.

4. Data Flow (~5 min)

User clicks ad → redirect through us → click event published to Kafka → dedup → stream aggregator computes per-minute rollups → writes to OLAP (online analytical processing) store → dashboards read from OLAP + a Redis "live count" layer for sub-second freshness.

5. High Level Design (~10–15 min)

  1. Click ingest endpoint. Receives GET /click/{token} → validates HMAC → emits a Click event to Kafka → immediately 302-redirects to the advertiser's destination URL. The 302 is the user-visible action; counting is fire-and-forget within the same request.
  2. Kafka. Partitioned by ad_id so all clicks for one ad land in order on the same partition. Long retention (e.g., 7 days) gives us replay.
  3. Dedup service. Stream processor that filters out duplicate click_ids seen within a window (e.g., 24 h). Backed by a Redis set or RocksDB-backed Flink state.
  4. Aggregation pipeline. Flink job consumes deduped events, maintains tumbling windows (1 min) keyed by (ad_id, minute), emits to: - Real-time tier: Redis counters (INCRBY per minute) for the live dashboard. - Persistent tier: OLAP store (ClickHouse or Druid) for hourly+ aggregates and historical query.
  5. Dashboards. API queries Redis for the live window and ClickHouse for everything beyond. Time-range queries fan out across the two layers.
  6. Reconciliation. Batch job runs hourly from the raw Kafka topic → exact counts → upserted into ClickHouse (corrects any drift from the streaming layer).
  7. Bot filtering. ML model + heuristics on the dedup stream classify clicks as valid/bot. Bot clicks counted separately (kept for analysis, excluded from advertiser-facing totals).

End-to-end narration: a user clicks a banner → browser hits gad.com/click/eyJhbGciO... → ingest validates HMAC, emits Click event to Kafka, returns 302 to destination URL within 50 ms. Flink consumes, dedupes (click_id not seen), increments per-minute bucket in Redis and writes to ClickHouse. Advertiser opens dashboard → sees clicks-this-minute updating live (Redis), historical from last hour from ClickHouse.

Concrete write path:

  1. Browser calls GET /click/{token}; ingest validates HMAC and extracts click_id, ad_id, campaign_id, advertiser_id, and destination URL.
  2. Ingest writes click_id to the producer retry queue only if Kafka publish fails; the normal path publishes to Kafka topic click_raw partitioned by hash(ad_id).
  3. Flink consumes click_raw, checks dedup state keyed by click_id, and emits valid events to click_valid.
  4. Aggregator updates Redis live counters and writes minute rows into ClickHouse. The Redis and ClickHouse writes can run in parallel because ClickHouse rows are idempotent by (ad_id, minute, version).
  5. Dashboard reads ClickHouse for closed windows and Redis for the currently open minute, then adds them client-side.

Dashboard read path:

  1. Stats API authenticates the advertiser and parses {range, granularity, cursor}; the shard key is advertiser_id for query routing and authorization.
  2. It queries ClickHouse for closed buckets, partition-pruned by day and filtered by (advertiser_id, campaign_id?, ad_id?).
  3. In parallel, it reads Redis keys live:{ad_id}:{minute} for the open minute only; if there are many ads, it batches by Redis shard and caps the fan-out to the advertiser's active ad set.
  4. On a Redis miss, it returns ClickHouse-only results and marks the open bucket as provisional; the next poll usually fills from Redis or from the first flushed ClickHouse row.
  5. Cursor state is opaque and server-issued: {advertiser_id, range_start, range_end, granularity, last_bucket, freshness_cutoff}. The cursor pins freshness_cutoff so page 2 does not drift while minute buckets are still moving.

Hot vs cold dashboard path:

Hot recent range:  client -> Stats API -> Redis live counters
                                      \-> ClickHouse closed buckets

Cold old range:    client -> Stats API -> ClickHouse only -> object archive only for audit export

Replay repair:     Kafka/S3 raw -> batch recon job -> ClickHouse replace partition

6. Deep Dives (~10 min)

Idempotency / dedup. Every click is signed with a unique nonce in the token. Two routes for dedup: - At ingest: check the nonce against a Redis bloom filter of recent nonces; reject if seen. Bloom false-positives drop some legit clicks (rare, acceptable). - In the stream: Flink keys by click_id, dedupes within a 24-h window. More accurate but accepts the duplicate into Kafka first. I'd default to in-stream dedup for accuracy and use ingest-level dedup only for crude bot mitigation.

Concrete stores:

Store Key pattern Value / data structure Entry size TTL / retention Shard key
Kafka raw click_raw[{partition}] append-only click events ~500 B/event 7 days hot hash(ad_id) plus salt for hot ads
Flink dedup state dedup:{click_id} RocksDB bool + event minute ~40 B 24 h hash(click_id)
Redis live counter live:{ad_id}:{minute} integer counter ~32 B 2 h hash(ad_id)
ClickHouse minute row (day, advertiser_id, ad_id, minute) counts by valid/bot/source ~80-120 B 2 years aggregate advertiser_id, partition by day
Raw cold archive s3://clicks/dt=YYYY-MM-DD/hour=HH/ compressed Parquet event rows ~150-250 B compressed 90 days+ date/hour
Bot decision cache bot:{click_id} valid/bot reason code ~32 B 24 h hash(click_id)

Capacity math: 10B raw events/day x 500 B = 5 TB/day. Kafka x3 replicas x7 days = 105 TB uncompressed, usually ~35-60 TB with compression. Dedup state is one day of click IDs: 10B x 40 B x replication/overhead ~= 0.5-1 TB of RocksDB state. Bot decision cache is another bounded 10B x 32 B ~= 320 GB before overhead if kept outside Flink state, so I would usually fold it into stream state. Redis live counters are tiny: 10M buckets/day x 32 B x only 2 hours retained ~= a few GB. ClickHouse aggregates for 2 years at 10M buckets/day x 100 B x 730 ~= 730 GB before replication/compression. The raw event stream is the bill.

The two-view pattern is the main architectural move: raw clicks are the immutable content log, while Redis and ClickHouse are serving indexes over that same log. Redis is indexed by (ad_id, open_minute) for freshness; ClickHouse is indexed by (advertiser_id, day, minute) for dashboard scans. They are not competing sources of truth, and replay can rebuild both.

Exactly-once vs approximate. Sub-second dashboards: approximate counts from Redis. Reconciled-exact counts: hourly batch from Kafka → ClickHouse with upsert semantics. The advertising contract typically requires the reconciled number; the dashboard caveat is small print.

Why Kafka + Flink + ClickHouse. Kafka is the durable replay log; Redis Streams or Pub/Sub cannot give 7-day replay at this volume, and Kinesis/Pulsar are also fine if that is the company's standard. Flink is useful because the hard part is keyed window state and replay, not just queue consumption. ClickHouse is a better fit than MySQL for dashboard scans because queries are columnar ranges over (advertiser_id, time); Druid is the closest alternative and wins if you want built-in rollup/serving, while ClickHouse is simpler when you can own the schema.

Aggregation correctness under failures. Flink with RocksDB state + Kafka source replays from offsets on task failure. Exactly-once write to ClickHouse via two-phase commit or idempotent upserts keyed by (window, ad_id).

Bot filtering. Heuristics: same IP burst, no referrer, suspicious UA, click-without-prior-impression, click-too-fast-after-impression. Plus a model trained on labeled bot traffic. Filtered clicks are stored separately for visibility but excluded from billable counts.

Hot keys (Super Bowl ad). One ad gets 10M clicks in a minute. Kafka partitioning by ad_id puts them on one partition — could saturate. Mitigation: salt the partition key for hot ads (hash(ad_id) + small random) and merge on aggregation. Detection: monitor partition lag.

Hot-ad state machine:

Stage Mode Source / write pattern Latency target
normal unsalted partition = hash(ad_id) dashboard lag <5 s

extreme sampled live + exact later Redis live counter samples; Kafka keeps all events live approximate, hourly exact
recovery compact salts merge salted buckets into normal ClickHouse rows next closed window

Dashboard freshness state machine:

Stage Mode Source Fallback / latency budget

Edge stress tests:

Edge Design response Budget
Dormant advertiser opens a 1-year dashboard Skip Redis, scan ClickHouse by advertiser_id and day partitions; paginate with fixed freshness_cutoff first page <2 s
Dashboard paginates past cached live window Cursor naturally switches to ClickHouse-only closed buckets stable page order

Interview punch line: the dashboard is allowed to show a provisional live overlay, but billing should always be reproducible from the immutable click log.

Interview punch line: I would make the redirect path highly available and the counting path replayable. The user-visible click should finish fast; money accuracy comes from idempotent replay and reconciliation, not from trying to do every aggregate synchronously.

Latency budget. Ingest must be < 100 ms so the redirect is snappy. Kafka publish is async — acks=1 on the producer keeps the publish under 5 ms. Click event emission can be fire-and-forget with retry queue on producer-side failure (rare).

Backfill / replay. If aggregation breaks, replay from Kafka. Aggregations are deterministic given the same input stream and the same windows, so re-run produces the same numbers.

Multi-tenancy / sharding. ClickHouse partitioned by date + advertiser_id. Hot advertisers get dedicated shards if their query load is heavy. Smaller advertisers share.

Cost. Storage of raw events for 90 days at 5 TB/day = 450 TB. Aggregates a tiny fraction. Tier raw events to cold storage after 7 days; keep aggregates hot.

Click attribution (touch). A click on an ad now might lead to a purchase later. Attribution is a downstream concern: stream the click + the eventual conversion event through a join with a configurable window. Out of scope to design fully here.

What I'd defer. Impression counting (similar pipeline at higher volume), advanced attribution windows, ad-fraud detection depth, real-time budget pacing (cutting off campaigns mid-flight).
