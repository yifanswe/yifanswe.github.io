YouTube Top-K Videos (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a service that returns the Top-K most-viewed YouTube videos over a time window (last hour, last day, last 30 days). 100M videos in active circulation, 1B view events per hour at peak, K up to 1000. Response p99 < 200 ms.

1. Requirements (~5 min)

Functional - Query top-K videos for a given window: 1h, 1d, 7d, 30d. Plus "all time." - Optionally filter by category or region.

I'd defer per-user personalization and trending algorithm tweaks beyond raw counts.

Non-functional - Query p99 < 200 ms. - Ingestion sustains 1B events/h, ~280k events/s peak. - Eventual consistency — top-K reflects events within ~1 min (per-minute granularity is fine). - Approximate top-K acceptable; we'll use sketches if exact is too expensive.

Capacity - 100M active videos. Storing exact per-window counters for 4 windows = 400M counters in hot memory if all materialized. - View event ≈ 100 bytes; 1B/h × 24 = 2.4 TB/day raw before replication, but compressed minute counts are far smaller. - Exact bucket state can fit if we choose the right granularity: 1h at 60 minute buckets × 100M × 4 B ≈ 24 GB; 1d at 24 hourly buckets ≈ 9.6 GB; 30d at 30 daily buckets ≈ 12 GB. Replication and category/region facets multiply this, so the dominant hot cost is state size in the stream processor, not API reads.

2. Core Entities (~2 min)

  • ViewEvent — video_id, user_id, ts, region, category.
  • WindowCounter — (video_id, window) → count.
  • TopKList — (window, [video_id]_K), the precomputed ranking.

3. API (~5 min)

GET /v1/topk?window=1h&k=100&category=music&region=us   → { videos[] }

Window is enumerated (1h, 1d, 7d, 30d, all); no arbitrary windows.

The response cursor is only for paging within the precomputed list; it encodes (window, facet, version, offset) so every page comes from the same published snapshot.

4. Data Flow (~5 min)

View events → Kafka → stream processor → updates per-window counters (sliding) → maintains a heap of top-K per window → publishes top-K list to a read store → served from Redis on read.

5. High Level Design (~10–15 min)

  1. Event collection. YouTube player emits view events to an ingest layer. Events go to Kafka topic views.raw, partitioned by video_id (so all events for one video land on the same partition for easier aggregation).
  2. Stream aggregator. Flink job consumes views.raw, maintains counters in multiple windows. Two flavors: - Tumbling 1-min counters (always materialized). - Sliding windows of 1h/1d/7d/30d computed by summing the right number of tumbling buckets.
  3. Per-video state. Keyed state in Flink: video_id → ring buffer of last N minute-buckets. Memory cost = 100M videos × small constant.
  4. Top-K maintenance. Within each task slot (which owns a hash range of video_ids), keep a min-heap of the local top-K per window. On every minute boundary, emit the local top-K to a merge stage.
  5. Merge stage. Coordinator that takes local top-K from each task and computes the global top-K. With N partitions and K=1000, local top-K per partition can be larger (say 5K) to guarantee global correctness when items just below local-K could merge into global top-K.
  6. Read store. Global top-K written to Redis (one list per (window, region, category) combination). Read service simply fetches the precomputed list.
  7. Cold-storage backfill. Raw events also archived to S3 / a data warehouse for ad-hoc analytics.

End-to-end narration: a video gets viewed → event lands in Kafka partition for that video_id → Flink task increments the current minute bucket → at minute boundary, slide the buffer (drop oldest, add newest) → recompute sliding counts for each window → if updated count enters the local top-K, update the local heap → every minute, local heaps merged → global top-K written to Redis → API serves it directly.

6. Deep Dives (~10 min)

Concrete store contracts.

Capacity by store: Kafka hot retention at 2.4 TB/day × 3 replicas × 3 days is ~21.6 TB before compression. Exact active state is tens of GB for core windows but multiplies quickly with facets, so keep supported facets bounded. Redis serving lists are tiny: 400 facet/window lists × 1000 rows × 24 B is under 10 MB before overhead; the read path is not the cost center.

Ingest and query path trace. Player beacon → ingest validates and writes Kafka → Flink task keyed by video_id updates current bucket and total counter → at minute boundary it advances ring buffers and local heaps → merge stage combines local K' lists into global K → read store writes a new Redis version → API reads topk:{window}:{facet}:{version} and hydrates video metadata in parallel. Query path is intentionally a single Redis read plus metadata hydration; no scan happens on user request.

The cursor pins a published version; otherwise page 2 could come from a newer minute and duplicate or skip videos.

Exact vs approximate. Exact top-K at 100M keys/280k QPS is heavy but doable with proper sharding. Approximate alternatives: - Count-min sketch for counters with bounded error; saves memory but loses exactness. - Space-saving algorithm for streaming top-K: maintains k×c buckets, replacing the min when a new item dominates. Provably bounded error. - I'd default to exact (per-video state in Flink) since the question size fits, mention space-saving as the fallback if memory blows up.

Why Flink/RocksDB instead of Spark micro-batches: we need minute freshness, keyed state, watermarks, and failure recovery with low continuous latency. Spark Structured Streaming can work, but Flink is the more natural fit for long-lived keyed state and event-time windows.

Why Redis for serving instead of querying Flink state directly: Redis gives a stable, versioned snapshot with predictable p99 under 200 ms. Flink state is optimized for streaming updates and checkpointing, not thousands of API reads.

Top-K aggregation correctness. A naive "global top-K = union of local top-K" is wrong: an item just below K in every partition could globally exceed K. Solution: send local top-K' where K' > K (e.g., 5K) so global merge sees enough candidates. Or include the global count via a second pass.

Sliding window memory. For a 30-day window at minute resolution, that's 43200 buckets/video × 100M videos = 4.3 trillion entries. Too much. Mitigation: bucket coarsely for older windows — minute resolution for 1h, hour resolution for 1d, day resolution for 7d/30d. Trade precision at the tail of the window for memory.

Interview punch line: "The trick is not computing top-K; the trick is refusing to store 30 days at minute granularity when the product only needs minute freshness near now."

Hot key skew. A viral video gets a disproportionate share of events. The Kafka partition that owns it backpressures. Mitigation: salted partitioning for hot keys, with a downstream merge to recombine. Detection: an upstream tracker watches partition lag.

Late events. A view event from a flaky network might arrive 5 min late. Flink handles with watermarking — events outside the watermark are dropped or sent to a side output for reconciliation. The trade-off is freshness vs completeness; for top-K, dropping 0.01% of events is fine.

Multi-region. Compute region-local top-K within each region; for global top-K, run a single global Flink job that consumes a global Kafka mirror — or merge regional top-K's at a coordinator. Latency cost on the global view.

Faceted top-K (per category / region). Each (window, dimension) combo is a separate precomputed list. Cardinality is small (e.g., 10 categories × 10 regions × 4 windows = 400 lists). Compute cost is linear in dimensions.

This is another two-view pattern: counters are keyed by video_id for updates, while published lists are keyed by (window, facet) for reads. They are the same view data materialized two ways so writes and reads are both cheap.

Failure recovery. Flink with RocksDB-backed state and Kafka source replays from offsets on task failure. Exactly-once semantics via Flink's checkpoint + Kafka's transactional reads.

The two-view pattern is explicit here: counters are update-keyed by video_id, while lists are read-keyed by (window, facet, version). The merge stage is the only place that converts one view into the other.

Case Mode Source Latency budget Fallback
Viral hot key Salted counting Kafka salted partitions + merge <1 min Split video_id into salts, recombine before heap

What I'd defer. Per-user "top-K I'd like" personalization, trending = velocity-based not raw counts, fraud filtering of bot views.
