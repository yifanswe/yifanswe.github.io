Local Delivery Service — DoorDash-Style (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Easy · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a local food/parcel delivery service like DoorDash. A customer opens the app, sees nearby merchants, places an order, and a dasher picks it up and drops it at their address. We care about getting the order to the right place in 30–45 minutes with live tracking. Assume 10M daily orders across the US.

1. Requirements (~5 min)

Functional - Customer browses nearby merchants and places an order. - System matches the order to a nearby available dasher and routes them merchant → customer. - Customer sees live ETA and the dasher's location on a map.

I'd scope out menu management, payments deep-dive, and dasher onboarding.

Non-functional - Order placement must be transactional — never two charges, never a lost order. - Match latency p99 < 5 seconds; users won't wait longer with a finger on "Place order." - Live tracking updates every 3–5 seconds with sub-second freshness on the map. - Geographic isolation — a US-East outage cannot stall US-West orders. - Strong consistency on the order state machine (placed → accepted → picked → delivered). Eventual consistency is fine for browse and analytics.

Capacity - 10M orders/day → ~115/sec average, ~1000/sec peak at lunch/dinner. - ~1M concurrent active dashers at peak emitting GPS every 3s → ~350k GPS writes/sec. - Order rows at ~2 KB × 10M/day × RF=3 are ~60 GB/day. The live dasher index is small because it only stores latest state (~1M × 150 B × RF=2 ≈ 300 MB), but the GPS analytics log can dominate if retained raw: 350k/sec × ~50 B ≈ 1.5 TB/day before RF/compression.

2. Core Entities (~2 min)

  • User — customer.
  • Merchant — restaurant; has a fixed geo location and a menu.
  • Order — customer, merchant, items, status, addresses, dasher_id.
  • Dasher — current location, status (offline / available / on-trip), current order.
  • GPS ping — dasher_id, lat, lng, ts. High volume, ephemeral.

3. API (~5 min)

GET   /v1/merchants?lat=&lng=&radius=    → Merchant[]
POST  /v1/orders                          { merchant_id, items[], address } → { order_id }
GET   /v1/orders/{id}                     → Order (status, dasher, eta)
WS    /v1/orders/{id}/track               → push dasher location + ETA
POST  /v1/dashers/location                { lat, lng } (auth: dasher)
POST  /v1/dashers/orders/{id}/{action}    action ∈ {accept, picked_up, delivered}

Customer ID and dasher ID come from the auth token.

Order mutations carry idempotency keys. Tracking cursors encode {order_id, last_location_ts, last_state_version} so a reconnect receives only missed state and location updates.

5. High Level Design (~10–15 min)

I'd partition the world by city/region. Each region is largely self-contained.

  1. Merchant search. Client sends lat/lng. A geo-indexed merchant service (PostGIS, S2 cells, or geohash) returns nearby merchants ranked by distance + popularity. This is read-heavy and cacheable per cell + time of day.
  2. Order placement. Transactional service. Writes an order row in the regional order store (MySQL, sharded by order_id within the region) with status placed. Publishes an OrderPlaced event to Kafka.
  3. Matching service. Consumes OrderPlaced, queries the live dasher index for the region (nearby + available), runs a small optimization (closest dasher with low estimated finish time), tries to assign. Assignment is a Redis-backed lease so two simultaneous orders don't grab the same dasher.
  4. Dasher location pipeline. Dasher app posts GPS every 3s → ingest service writes to a regional Redis geo-index (GEOADD) keyed by region. Persistent log to Kafka for analytics.
  5. Tracking WebSocket. Once assigned, the customer's WS pulls from a per-order channel that fans out the dasher's location updates plus state changes.
  6. Order state service. Owns the state machine; emits events to Kafka for downstream (notifications, payments capture, analytics).
  7. Payments and notifications sit downstream of the order event stream — out of scope to design, in scope to draw.

End-to-end narration: customer taps "Place order" → order row written, event published → matcher picks Dasher #42 → leases Dasher 42 in Redis → publishes OrderAccepted → customer WS opens → dasher's GPS stream now also fans into the customer's WS → state advances on each dasher action → on delivered, capture payment, terminal state.

Concrete stores:

Store Key / structure Entry size TTL / retention Shard key
Order DB order:{region}:{order_id} state row ~2 KB 7 years/archive by policy region + order_id
Dasher geo index dashers:{region} Redis GEO member=dasher_id ~150 B/dasher 10 s freshness TTL region
Dasher lease dasher:{id}:lease -> order + fencing token ~100 B 10-30 s TTL dasher_id
Merchant index merchant_cell:{s2_cell} -> merchant IDs ~16 B/member hours, rebuilt S2/geohash cell
GPS log topic gps.{region} event=dasher_id, lat,lng,ts ~50 B 24 h raw, roll up region + dasher_id
Order event log topic orders.{region} state transitions ~500 B 7-30 days replay region + order_id

Store math: order data is ~60 GB/day replicated and easy to shard by region. The raw GPS stream is bigger: ~1.5 TB/day primary, ~4.5 TB/day at RF=3 before compression, so keep only 24h raw and roll up for analytics. The live index is tiny; its risk is freshness, not bytes.

6. Deep Dives (~10 min)

Geo-indexing for dashers and merchants. Merchants are static, so a precomputed S2 cell → merchant list works. Dashers move every 3s; Redis GEOADD with GEORADIUS is the practical choice — in-memory, sub-millisecond, naturally per-region.

Why Redis GEO for dashers but S2/PostGIS for merchants: dashers need latest-position reads and short TTLs, where an in-memory geo index wins. Merchants are static and richer, so precomputed S2 cells or PostGIS work well and are cheaper to cache by cell/time of day.

Be ready for "why not PostGIS for dashers too?" It can work at lower write rates, but 350k location writes/sec plus 5-second matching latency pushes us toward an ephemeral in-memory index. PostGIS remains the source for merchant truth and offline spatial queries.

The matching algorithm. Naive nearest-dasher creates clustering and unfair distribution. I'd run a greedy assignment with a small batch window (collect orders for 2s, run a bipartite match) to balance dasher utilization. Mention the Hungarian algorithm or auction-based matching as the production lever.

Concurrency on dasher assignment. Same dasher could be claimed for two orders if not careful. Use Redis SET dasher:42:lease order:99 NX EX 10 to lease for 10s while the dasher is being notified. If the dasher rejects or times out, lease expires and matcher reruns.

Order placement path trace: client -> order service validates cart and idempotency key -> payment authorization and order row write happen in the same saga step -> publish OrderPlaced -> matcher reads nearby available dashers from dashers:{region} -> leases one dasher -> writes assigned state with row lock -> emits state event to WS/notifications.

Tracking read path: customer WS subscribes to order:{id}; push layer reads latest order state from MySQL/cache and latest dasher position from Redis in parallel; after that it streams batched updates from gps.{region} and order events using {last_location_ts,last_state_version}.

Order placement hot path
    client -> Order API
        -> idempotency:{customer}:{key}
        -> MySQL order row status=placed
        -> Kafka orders.{region} OrderPlaced
        -> Matcher GEORADIUS dashers:{region}
        -> SET dasher:{id}:lease NX EX 10 with fencing token
        -> MySQL order status=assigned version+1
        -> WS/notification event

Two views, one trip: the order row is keyed by order_id for correctness, while the live dasher index is keyed by region/geo_cell for matching. The same dasher location event also feeds gps.{region} for replay/analytics, but only Redis is on the user-facing hot path.

Live tracking scale. 350k GPS writes/sec is too much to fan out per write. Instead: GPS writes go to Redis; the WS push layer batches updates per order every 3–5s rather than per ping.

Order state transitions. Single owner of the state machine, behind a row-level lock in MySQL keyed on order_id. Every transition emits a Kafka event for downstream services. Idempotency keys on dasher action endpoints prevent double-transitions on flaky networks.

Regional isolation. Per-region cells: MySQL, Redis, Kafka, services. Global services only for identity and merchant catalog. A US-East failure doesn't stop SF orders.

ETA prediction. Start with a heuristic (haversine distance / avg city speed + 5-min merchant prep). Later, plug a model fed by historical trip data. Be explicit that v1 is a heuristic.

Graceful degradation.

Case Mode Source User-visible behavior
Dasher GPS stale Keep assignment, mark location stale last Redis position map pauses, ETA widens
No nearby dashers Expand radius, then queue order geo index + matcher delayed assignment
Matcher overload Batch 1-2 s and prioritize paid orders order queue placement succeeds, assignment lags
Redis lease lost Reconcile with order DB fencing token order state + lease avoid double assignment
Region outage Stop taking new regional orders regional health other regions unaffected

State machine for an order:

Stage Mode Source of truth Latency budget Transition guard
placed durable order MySQL row + OrderPlaced event <300 ms idempotency key unique
matching ephemeral lease Redis GEO + lease token <5 s lease fencing token
assigned strong state MySQL versioned update <200 ms row lock / compare version
in_transit live signal Redis latest GPS + Kafka events 3-5 s update dasher action idempotency
delivered terminal MySQL row + payment capture event <1 s user-visible no backward transition

Interview punch line: "Orders need strong state transitions; locations are ephemeral signals. I would never put them in the same consistency bucket."

What I'd defer. Multi-stop orders, batched deliveries (a dasher carrying 2 orders), surge pricing, dasher scheduling and earnings.
