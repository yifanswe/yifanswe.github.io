Yelp — Local Business Search & Reviews (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build Yelp. A user opens the app, searches "ramen near me," and gets a ranked list of nearby restaurants with photos, ratings, and reviews. 100M monthly active users, ~10M businesses, ~200M reviews lifetime. Read-heavy; reviews trickle in.

1. Requirements (~5 min)

Functional - Search for businesses by keyword + location, optionally filtered (category, rating, price, open now). - View a business page: photos, hours, address, aggregate rating, reviews. - Write and read reviews.

I'd defer reservations, business owner tooling, and ad placement.

Non-functional - Search results p99 < 300 ms. - Read-heavy: ~10k searches/sec average. - Strong consistency on the business profile (hours, address); eventual is fine on aggregate ratings (cached). - Review fraud is real; flagging pipeline matters even if we don't implement it.

Capacity - 10M businesses × 5 KB profile = 50 GB. Fits one shard easily; sharded for read fan-out. - 200M reviews × 2 KB = 400 GB. - Photos: ~1B images, ~500 KB each → ~500 TB in object storage. - Search index is still modest: 10M businesses × ~2–5 KB indexed fields/terms ≈ 20–50 GB primary, maybe 3–5× with replicas and inverted-index overhead. The dominant cost is photos/CDN, not metadata. - Review writes are low compared with reads: even 5M reviews/day is only ~60 writes/sec average; the review store is designed for business-page locality and moderation, not write scale.

2. Core Entities (~2 min)

  • Business — business_id, name, geo, hours, categories, phone.
  • Review — review_id, business_id, user_id, rating, text, photos, created_at.
  • User — user_id, profile.
  • Photo — photo_id, business_id, url, uploader.

3. API (~5 min)

GET   /v1/search?q=ramen&lat=&lng=&filters=...&cursor=...   → Business[]
GET   /v1/businesses/{id}                                    → Business + recent_reviews
GET   /v1/businesses/{id}/reviews?cursor=...                 → Review[]
POST  /v1/businesses/{id}/reviews     { rating, text, photos[] }   → { review_id }

User from token.

Search uses cursor-based pagination; the cursor encodes query_hash, geo_cell, last_score, and last_business_id so page 2 is stable even while ratings drift slightly.

5. High Level Design (~10–15 min)

  1. Business catalog service. Authoritative store (MySQL or DynamoDB, sharded by business_id). Read-through cache for hot businesses.
  2. Search service. Elasticsearch (or OpenSearch) cluster indexing businesses with full-text on name + categories + tags + city, plus geospatial indexing on the location field.
  3. Geo + keyword query. ES query combines geo-distance filter + keyword + filters + a custom score (rating, review count, distance, recency).
  4. Review service. Writes reviews (sharded by business_id so all reviews for one business co-locate). Publishes review.created to Kafka.
  5. Aggregator pipeline. Stream processor consumes review events, recomputes per-business rating + count, writes back to the catalog. Also enqueues fraud scoring.
  6. Photo service. Upload through API, store in S3, generate thumbnails async, CDN-front everything.
  7. CDN + edge cache. Search results cacheable for short TTL by (query, geo cell, filters); business pages cacheable by business_id.

End-to-end narration: Alice searches "ramen near me" → API gateway → search service queries ES with geo-distance < 5 km, full-text "ramen," filter open_now=true → ES returns top 50 → service hydrates with cached business profiles → returns top 20 with photos. Alice taps a business → catalog lookup (likely cached) → 20 most recent reviews from review service.

6. Deep Dives (~10 min)

Concrete store contracts.

Store Key / data structure Entry size + TTL Shard key
Business catalog business:{business_id} document ~5 KB, retained business_id
Search index OpenSearch/ES doc with text fields + geo_point + filter fields ~2–5 KB indexed, rebuilt/replicated geo_cell/business_id routing if needed
Reviews reviews_by_business:{business_id} sorted by created_at ~2 KB/review, retained business_id
User review index reviews_by_user:{user_id} pointers ~32 B/review pointer user_id
Rating aggregate rating:{business_id} → {sum,count,bayes_avg} ~32 B, retained business_id
Recent-review cache recent_reviews:{business_id} Redis list ~40 KB for top 20, TTL 5 min business_id
Photos s3://photos/{business_id}/{photo_id}_{size} ~500 KB original, thumbnails retained business_id

Store math: catalog + reviews are hundreds of GB, while photos are hundreds of TB. Search is a read index: 10M × 5 KB × 3 replicas is roughly 150 GB before inverted-index overhead; still small compared with photo storage. The cost levers are thumbnail sizing, CDN cache hit rate, and photo lifecycle tiering, not shaving bytes from business rows.

Search path trace. Client → search API normalizes query and geo → OpenSearch executes geo-distance filter, keyword match, structured filters, and top-100 scoring → search service hydrates business documents and rating aggregates from cache/KV in parallel → fetches thumbnail URLs → returns top 20 with cursor. Page 2 reuses the same query hash and last_score so pagination is by score/cursor, not offset.

Cold search path: if the query cache misses and OpenSearch is slow, the API can fall back to popular:{geo_cell}:{category} from Redis, then hydrate those businesses. That result is less personalized but still useful within 500 ms. The cursor carries {mode=popular_fallback, geo_cell, category, offset} so pagination stays in the same degraded mode.

Geo + keyword in one query. Elasticsearch with a geo_point field handles this well — geo-distance filter applied first (cheap), then full-text BM25 scoring, then a script_score for custom ranking. Alternative: separate geo index (S2 cells → business_ids) intersected with a keyword index — more boxes, more failure modes, mention as an option.

Why ES/OpenSearch instead of a relational database with geo extensions: the workload combines full-text relevance, filters, geo radius, and top-K ranking in one low-latency path. PostGIS is excellent for spatial truth, but BM25/rescoring and inverted-index pagination are the core search problem here.

Be ready for "why not only S2 cells?" S2 is great for nearby candidate generation, but it cannot rank typo-tolerant text queries like "ramen," "ramn," and category synonyms without rebuilding a search engine around it.

Ranking. Inputs: text relevance, distance, rating (Bayesian-averaged with global prior so a single 5-star doesn't beat a 4.5-with-200-reviews), recency, review count, business open right now. Learn-to-rank model trained offline; serve as ES rescore script for the top 100.

Reviews — write path. Reviews are sharded by business_id so the business page query is a single shard read. Reviewer can amend their review; row keyed by (business_id, user_id) so each user has at most one review per business.

Write path: client → review service validates one-review-per-business with unique (business_id,user_id) → writes review row → writes pointer to reviews_by_user → emits review.created → stream aggregator updates rating:{business_id} and search index asynchronously → cache invalidation for recent_reviews:{business_id} is best-effort.

Two indexes, one review: reviews_by_business:{business_id} is optimized for the business page, while reviews_by_user:{user_id} is optimized for profile/history and moderation. Both point to the same review content; neither should duplicate photo blobs or large text beyond what the query needs.

Aggregate ratings without recomputation cost. Stream processor maintains sum(rating), count(review) per business; aggregate = sum/count. Update is a single key-value put. Avoid scanning all reviews to compute average.

Hot business pages. Trending restaurant has 1000 concurrent viewers. Cache the business profile in CDN (60s TTL) and recent reviews in Redis (5 min TTL). Cache invalidation on write is best-effort — eventual consistency is acceptable for reviews.

Interview punch line: "The business profile needs correctness; the rating and recent reviews need freshness, and those are different contracts."

Photo serving. Upload → S3 → image worker resizes to 4 sizes → publishes to CDN. Originals kept for super-zoom; CDN serves the right size by URL convention.

Fraud detection. Heuristics: same IP across multiple reviews, account age, review velocity, content similarity to known templates. Flagged reviews are hidden but stored; humans review the queue.

Filter on "open now." Per business, store weekly hours. Index open_hours_minute_of_week as a sparse bitmap in ES, query against current minute. Refresh per minute.

Graceful degradation state machine.

Case Mode Source Latency budget Fallback
Hot search query Cached result page query+geo+filter cache <100 ms Short TTL 30–60s; hydrate fresh profile on tap
OpenSearch slow Degraded search cached query or geo-only index <500 ms Return popular nearby businesses with a stale badge internally
Hot business page CDN/Redis catalog cache + recent reviews cache <200 ms Serve stale profile for 60s; reviews eventually catch up
Review write burst Async aggregates review row truth + Kafka Write ack <200 ms Rating count lags; never drop review row
Photo CDN miss Origin fetch object store Seconds okay Serve placeholder thumbnail; page still useful

Interview punch line: "Yelp search is a search-index problem, but Yelp correctness is a catalog-and-review problem; the index can lag, the user's review cannot disappear."

What I'd defer. Reservations system (OpenTable integration), business owner verification, structured ad placement, recommendations / "you might also like," local sponsored slots.
