1. Basic Knowledge And Solution

What This Question Is Testing

A web crawler is the canonical large-distributed-system problem: trillions of URLs, every host an independent rate-limited resource, the work never finishes. The hard part is being polite (not hammering hosts), efficient (not re-crawling unchanged pages), and respectful of robots.txt at the same time.

The invariant is this:

The crawler must visit URLs at a rate that respects each host's politeness budget and robots.txt, must not fetch a URL more than necessary for freshness, and must produce a deterministic, deduplicated, indexable record for every successfully fetched page.

At ICT5 / L6+ level, the interviewer is testing whether you can design per-host queueing, manage a frontier of trillions of URLs without re-fetching, handle traps (calendars, session ids), and prioritize freshness for important pages over completeness for everything.

60-Second Interview Framing

Spoken version:

I would design the crawler around a frontier of URLs to fetch, partitioned by host, with strict per-host politeness queues. A fleet of stateless fetchers pulls from the frontier, fetches HTML, extracts links, dedupes via content and URL fingerprints, and emits documents to downstream parsers and indexers. The interesting decisions are how to handle politeness, dedupe at trillion scale, prioritize the frontier, and detect spider traps.

Requirements To Clarify First

Functional requirements:

  • Crawl the public web starting from seed URLs.
  • Respect robots.txt and crawl-delay directives.
  • Re-crawl pages on a schedule informed by their estimated freshness.
  • Detect duplicate and near-duplicate pages.
  • Extract links and follow them subject to filters (depth, domain whitelist, content-type).
  • Produce content suitable for downstream indexing or analysis.

Non-functional requirements:

  • Scale to tens of billions of pages, trillions of known URLs.
  • Throughput: hundreds of thousands of fetches per second.
  • Politeness: never overload a small site.
  • Resilience to traps and adversarial content.
  • Resumable after crawler outages.

Out of scope unless asked:

  • Index serving (covered in "Design Google Search").
  • JavaScript rendering at full scale (mention as a heavy mode).
  • Real-time content discovery (push from publishers).

Scale Assumptions

Area Assumption
Pages 10B-100B documents in the corpus.
Known URLs 10x to 100x pages (most never fetched).
Hosts 200M+ active hosts.
Fetch rate 100k-1M URLs/sec across the fleet.
Average page size 50-200 KB raw, 10-30 KB extracted text.
Politeness 1 request/second/host typical; configurable.
Re-crawl cadence Hours for news; weeks for static pages.
Storage Petabytes for raw HTML; tens of petabytes for historical versions.

The bottleneck is host concurrency: a single hot host cannot be hit faster than its politeness budget, and the frontier must keep workers busy without violating any host's budget.

Core Components

Component Responsibility Key detail

Fetcher HTTP client; fetches URL; handles redirects, retries, robots. Stateless; horizontally scalable; pulls from frontier.

DNS resolver High-throughput DNS with internal cache. Avoid hammering recursive resolvers.

Content dedupe Tracks page content fingerprints. Simhash / minhash for near-duplicates.
Scheduler Decides re-crawl cadence per URL. Score = freshness × importance × cost.
Frontier prioritizer Reorders URLs in frontier by score. Hot pages get more attention.
Trap detector Bounds crawl depth and ID parameter cardinality per host. Avoid infinite loops.

URL Frontier Design

This is the central deep dive.

Two-level structure:

  • A front queue per priority class (high, medium, low).
  • A back queue per host, ensuring strict per-host FIFO and politeness gating.

Workflow:

  1. New URL arrives from extractor.
  2. Look up host; route to host's back queue.
  3. A scheduler thread per priority class peeks at front queue; selects host whose politeness window has elapsed; pops next URL.
  4. Hands URL to fetcher.

Storage:

  • Front and back queues live in persistent durable storage (a sharded log or KV store) because trillions of URLs do not fit in memory.
  • Hot working set is cached in memory for the active fetcher fleet.

Partitioning:

  • Frontier shards by hash(host). All URLs for a host live on the same shard.
  • Each shard manages its own per-host queues.

This guarantees: per-host politeness is enforced locally; cross-host parallelism is achieved by having many shards each managing many hosts.

Interview phrase:

The frontier is the heart of the system. The two-level design — back queues per host for politeness, front queues across hosts for priority — is what lets a single shard fetch from many hosts without violating any host's budget.

Politeness

For each host:

  • Track last_fetch_at. The next fetch starts after last_fetch_at + politeness_interval.
  • politeness_interval defaults to one second, can be overridden by robots.txt crawl-delay.
  • Concurrent fetches per host: usually 1; can be relaxed for large hosts after consultation.
  • On repeated 5xx or rate-limit responses, automatically increase the interval (multiplicative backoff).

Robots.txt Handling

Before fetching a URL:

  1. Check robots cache for the host. If missing or expired (TTL 24h typical), fetch host/robots.txt.
  2. If robots.txt is missing/200, parse and store.
  3. Match URL against rules. If disallowed for our user-agent, drop and emit a forbidden URLRecord update.
  4. Respect Crawl-delay if set.
  5. Cache sitemaps; enqueue them as discovery sources.

Many parsers exist; the rules are not formal but conventions matter (longest-match per user-agent, etc.). Treat robots.txt as authoritative.

Deduplication: URLs And Content

URL dedup:

  • Canonicalize URL: lowercase host, sort query parameters, strip session ids, remove fragments.
  • Hash to a 64-bit fingerprint.
  • Maintain a global bloom filter for membership probe; a persistent KV-backed exact set as the truth.
  • Bloom misses → query KV. Set membership informs whether to enqueue.

Content dedup:

  • Compute SHA-256 of normalized text.
  • Compare against a "seen content" index. Exact match → known duplicate.
  • For near-duplicates (boilerplate or templated pages), compute simhash or minhash signatures; cluster pages whose signatures are within a Hamming distance threshold.
  • One canonical URL per content cluster.

Refetch Scheduling

Not all URLs deserve the same attention.

Score factors:

  • Importance: domain authority, inbound link count, traffic estimates.
  • Estimated change rate: from prior fetch differences (news sites change hourly; static manuals weekly).
  • Last fetch age.
  • Capacity budget per host and per priority class.

Algorithm:

  • Each URL has a refetch_score. Higher score → fetched sooner.
  • Score recomputed after each fetch from observed change frequency.
  • Scheduler keeps a per-host priority heap of URLs awaiting re-crawl.

Spider Traps

Common traps:

  • Infinite calendar pages: /cal/2024/01/01, /cal/2024/01/02, ...
  • Faceted commerce URLs: query parameter combinations grow exponentially.
  • Session ids in URLs.
  • Broken pagination that loops.

Mitigations:

  • Cap crawl depth per host.
  • Limit URLs per host per day.
  • Detect parameter-cardinality explosions and quarantine the pattern.
  • Strip well-known session id parameters during canonicalization.
  • Reputation system: hosts with high duplicate ratios are throttled.

Adversarial Content

  • Pages that detect crawler user-agent and serve different content (cloaking).
  • Pages that respond with malware payloads.
  • Pages that try to exhaust the crawler's resources (decompression bombs).

Defenses:

  • Limit per-page bytes; abort if exceeded.
  • Sandbox parsers and renderers.
  • Random sampling: occasionally crawl as a normal user agent to detect cloaking.
  • Resource limits per fetch.

Write And Read Path

Write (fetch) path:

  1. Fetcher pulls URL from frontier.
  2. Check robots cache; if disallowed, drop and update URLRecord.
  3. Resolve host via DNS cache.
  4. HTTP GET with conditional headers (If-Modified-Since, ETag).
  5. On 304, update last_fetched_at; skip storage.
  6. On 200, stream response to content store; record metadata.
  7. Parse: extract links, text, canonical tags.
  8. For each link: canonicalize, dedup check, enqueue if new.
  9. Update content dedup index.
  10. Push document to downstream indexer.

Read path: Indexers and downstream consumers consume from the document stream or read from content store. Not part of the crawler proper.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Hammering a small site (politeness bug). Per-host single-flight gate; backoff on 429/5xx. Per-host QPS; complaint volume.
Spider trap eats fleet capacity. Depth caps, host caps, parameter cardinality detection. Per-host duplicate ratio; URL growth rate per host.
Robots.txt fetch fails. Default to disallow if recently unreachable; or default to allow with reduced rate. Robots fetch error rate.
Frontier shard imbalance. Re-shard hot hosts to dedicated shards. Per-shard backlog size.
Content store growth explosion. Dedup before storage; sample storage for unchanging pages. Storage growth rate vs new-content rate.
Fetcher crashes mid-stream. Idempotent re-fetch; URL stays in dispatched state with timeout. In-flight URL stale count.
DNS resolver overload. Local DNS cache; rate-limit by domain authority. DNS QPS, error rate.
Adversarial cloaking. Mixed user-agent sampling; reputation tracking. Per-host content-hash variance under different agents.

Operational Metrics

Correctness:

  • Politeness violations per host (must be ~zero).
  • robots.txt rule violations.
  • Duplicate fetches per URL within freshness window.

Capacity:

  • Fetch QPS by priority.
  • Frontier size and growth rate.
  • Bloom false-positive rate.
  • Per-host backlog distribution.

Business:

  • Coverage: fraction of important pages re-crawled within target SLA.
  • Index freshness for news domains.
  • Discovered URLs per day.

Multi-Region Strategy

  • Run regional fetcher fleets close to publishers (US, EU, Asia).
  • Frontier sharding is partitioned by host; each host is owned by one region (often picked by host's hosting location).
  • Content store is regional with cross-region replication for indexing.
  • Avoid double-crawling from two regions on the same host.

Tradeoffs To Call Out

  • Tight politeness limits cap throughput; lax limits anger webmasters.
  • Aggressive dedup reduces storage but risks missing legitimate updates.
  • Strict robots respect costs coverage; loose interpretation invites legal trouble.
  • Centralized priority is hard at trillion-URL scale; approximate per-shard priority is usually accepted.
  • Deep crawling discovers more but consumes budget; shallow crawling misses long-tail.
  • Re-crawl frequency should adapt to observed change rate; static schedules waste fetches.

Common Traps And Stronger Moves

Trap Stronger move
One big global queue. Two-level frontier: back queue per host, front queue by priority.
Ignoring robots.txt. First-class robots cache, consulted before every fetch.
URL dedup only at insert. Bloom filter + exact set; re-canonicalize before checking.
Treating all URLs as equal. Importance + change-rate scoring for re-crawl.
Ignoring spider traps. Depth and parameter caps per host; cardinality detection.
Synchronous indexing. Append-only document stream; indexer is downstream.

Staff-Level Answer Checklist

  • Politeness as a hard invariant, enforced per-host.
  • Two-level frontier: priority across hosts, FIFO per host.
  • URL canonicalization and dedup with bloom + exact set.
  • Content dedup with simhash / shingles.
  • Refetch scheduling driven by importance and change rate.
  • Spider trap detection and host reputation.
  • Robots.txt cache as a first-class service.
  • Append-only document store; indexer is downstream.

Interview Phrase

A web crawler's hardest constraint is politeness, not throughput. Two-level queues — back queue per host, front queue across hosts — let me parallelize without ever hammering a single host. Dedup with bloom filters plus content fingerprints prevents wasted work; importance and change-rate scoring focuses capacity on what matters.

2. Simulated Interview

Interviewer: Design a web crawler.

Candidate: I will design it around a URL frontier that enforces per-host politeness as a hard invariant, a stateless fetcher fleet, and dedup layers for both URLs and content. The hard parts are politeness, scale of the dedup set, spider traps, and choosing what to re-crawl.

Interviewer: How does the URL frontier work?

Candidate: Two levels. Each host gets a back queue in strict FIFO order; politeness is enforced as "next fetch from this back queue at least T ms after last fetch." A small set of front queues across priority classes selects which host to pop next. A scheduler thread checks "which host's politeness window has elapsed" and picks the highest-priority one. The whole frontier is sharded by hash(host), so all URLs for one host live on one shard.

Interviewer: How big is the frontier?

Candidate: Trillions of URLs. It cannot fit in memory. The frontier lives in persistent storage — a log-structured KV or sharded RocksDB — with hot working set cached. Workers stream URLs into and out of disk-resident queues.

Interviewer: How do you dedupe URLs at that scale?

Candidate: First canonicalize: lowercase host, sort query parameters, strip known session ids, drop fragments, normalize trailing slash. Hash to a 64-bit fingerprint. Check a sharded bloom filter for fast membership. On bloom hit, check the persistent exact set. On miss, insert into bloom and exact set, and enqueue the URL.

Interviewer: Bloom false positives — how do you handle them?

Candidate: They mean we occasionally skip a URL that we have not actually seen. With a properly sized bloom filter, the false-positive rate is well under 1%. Acceptable because the URL will appear again from another inbound link and be checked against the exact set then. The cost of missing a re-discovery is much lower than the cost of looking up the exact set on every URL.

Interviewer: What about content dedup?

Candidate: Compute SHA-256 over normalized text. Exact match → known duplicate, skip storage but update URL → content_hash mapping. For near-duplicates, compute simhash or minhash. Pages with simhash within a small Hamming distance are clustered; we keep one canonical URL per cluster.

Interviewer: How do you respect robots.txt?

Candidate: First-class robots service. Before fetching a URL, check the host's robots entry. If missing or expired (TTL 24h), fetch /robots.txt (which itself respects politeness for that host). Parse rules, store. Match URL against the rules for our user-agent. Disallowed URLs are marked forbidden and skipped. We also pull sitemap URLs from robots.txt as discovery seeds and respect Crawl-delay.

Interviewer: Spider trap: a calendar that links infinitely forward.

Candidate: Three defenses. First, per-host crawl depth cap (say 20). Second, per-host URL budget per day; once exceeded, lower-priority URLs from that host are dropped. Third, parameter-cardinality detection: if a query parameter takes thousands of distinct values within a host, we flag the pattern and quarantine new URLs that match. The host's reputation degrades, and future fetches are throttled.

Interviewer: What gets re-crawled and when?

Candidate: A refetch score per URL combining importance (PageRank-style), observed change frequency, and time since last fetch. News domains may have a one-hour refetch interval; static manuals may be monthly. The scheduler maintains a priority heap per host; capacity is allocated across priority classes with quotas. We also respect Last-Modified and ETag via conditional GETs; a 304 response is cheap and confirms freshness.

Interviewer: How do you handle a host that returns lots of 5xx errors?

Candidate: Multiplicative backoff on the host's politeness interval. After repeated errors, the host enters a degraded state — fewer fetches per hour. After a longer cooldown, we probe to see if it has recovered. If a host is consistently down, we deprioritize but don't permanently drop; outages happen.

Interviewer: How do you parallelize within a host for big sites?

Candidate: Carefully. By default, one fetch per host at a time. For very large hosts (major news sites, large e-commerce) we may negotiate higher rates or detect that they obviously can handle more. We can run multiple concurrent fetches to such hosts via separate IP addresses if we have a reciprocal agreement.

Interviewer: What about JS-rendered pages?

Candidate: Two lanes. The default lane fetches raw HTML and parses it. For sites known to require JS, a slower, more expensive lane uses a headless browser. The headless fleet is much smaller; we route only a small fraction of URLs to it, prioritized by importance.

Interviewer: Multi-region?

Candidate: Run fetcher fleets in each region close to publishers. Each host is owned by one region — usually the closest geographically — so we never double-fetch the same host from two regions. The content store replicates cross-region for downstream indexing.

Interviewer: Summarize.

Candidate: A two-level frontier sharded by host enforces strict politeness; stateless fetchers pull URLs and respect robots.txt. URL dedup uses bloom + exact set; content dedup uses simhash. Re-crawl scheduling uses an importance + change-rate score. Spider traps are bounded by depth and cardinality caps. The output is an append-only document stream consumed by downstream indexers. Politeness is the hard invariant, not throughput; everything else scales by sharding.

3. Practice Notes

Five-Minute Version

  1. Clarify: scale, politeness, robots, dedup, freshness.
  2. Invariant: respect per-host politeness; deterministic dedup.
  3. Two-level frontier: back queues per host, front queues by priority.
  4. Stateless fetchers; robots cache; DNS cache.
  5. URL dedup with bloom + exact set; canonicalize first.
  6. Content dedup with hash + simhash.
  7. Refetch scheduling by importance × change rate.
  8. Spider trap defenses; reputation; multi-region.

One-Minute Closeout

A web crawler is bounded by politeness, not throughput. The two-level frontier — back queue per host, front queue by priority — guarantees we never hammer a host while staying fully utilized. Trillion-URL dedup uses bloom filters with a persistent exact-set fallback. Content dedup uses simhash for near-duplicates. Refetch is score-driven, focusing capacity on important and frequently changing pages. Spider traps are bounded by depth, cardinality, and reputation. Output is an append-only document stream consumed by downstream indexers.
