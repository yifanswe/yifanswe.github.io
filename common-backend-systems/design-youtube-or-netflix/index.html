1. Basic Knowledge And Solution

What This Question Is Testing

"YouTube or Netflix" as a combined prompt is the canonical video streaming question. The hard part is the same regardless of which: upload + transcode, multi-tier CDN delivery, adaptive bitrate streaming, recommendation, and storage at exabyte scale. The systems differ in catalog size (YouTube unbounded, Netflix curated) and origination (UGC vs licensed).

The invariant is this:

Every video is durably stored, transcoded into multiple resolutions/codecs for adaptive bitrate streaming, served via CDN with sub-second start, and discoverable via search or recommendation; viewers experience smooth playback across network conditions and devices.

At ICT5 / L6+ level, the interviewer is testing whether you can articulate the shared architecture (upload, transcode, CDN, ABR), reason about the differences between UGC and curated catalogs, design recommendation pipelines, handle viral content, and discuss DRM (Netflix-specific) vs open content (YouTube).

60-Second Interview Framing

Spoken version:

I would describe the shared design as four core systems: an upload + transcoding pipeline that produces multiple resolutions, a multi-tier CDN for delivery, an adaptive bitrate streaming layer (HLS/DASH), and a recommendation engine for discovery. YouTube and Netflix differ in catalog scale (UGC vs curated), DRM (Netflix strict, YouTube optional), and CDN strategy (Netflix Open Connect inside ISPs, YouTube via Google's CDN). The interesting decisions are encoding ladder, CDN tiering, ABR, and recommendation.

Requirements

Functional requirements:

  • Creators / studios upload video.
  • Async transcoding to multiple resolutions and codecs.
  • Viewers watch on web, mobile, TV with adaptive quality.
  • Resume from last position.
  • Search and recommendations.
  • Comments / ratings (YouTube) or maybe-not (Netflix).
  • Downloads for offline (Netflix).

Non-functional requirements:

  • Sub-second video start.
  • Adaptive bitrate up to 4K HDR.
  • Petabyte to exabyte storage.
  • Multi-region delivery.
  • Strict DRM (Netflix); optional (YouTube monetized).
  • High availability.

APIs And Playback Paths

Core APIs:

PUT /uploads/{upload_id}/chunks/{chunk_id}
POST /videos/{video_id}/publish
GET /videos/{video_id}
GET /videos/{video_id}/manifest?profile_id=&device=&codec=
POST /drm/licenses
GET /segments/{video_id}/{rendition}/{segment_no}.m4s
PUT /profiles/{profile_id}/playheads/{video_id}

Hot playback path:

  1. Client asks for the manifest with profile_id, device_class, and video_id. Catalog reads shard by video_id; entitlement and playhead reads shard by profile_id.
  2. Edge cache checks manifest:{video_id}:{codec}:{device_class}:{region}. A hit returns in under 30ms; a miss falls through to the manifest service.
  3. Manifest service parallel reads video:{video_id}, ladder:{video_id}:{codec}, playhead:{profile_id}:{video_id}, and, for Netflix-style content, entitlement:{profile_id}:{title_id}.
  4. If DRM is required, the client calls the license service with session_id and device_id; the license response is cached only for the playback session.
  5. Player fetches immutable segment objects by segment:{video_id}:{rendition}:{segment_no} from edge CDN. Segment misses go edge -> regional cache -> origin object storage.
  6. Playhead writes are async and idempotent on (profile_id, video_id, updated_at_bucket) so retries do not rewind progress.

Cold playback path:

  1. Edge miss checks regional cache using the same segment key; regional miss pulls from origin object storage sharded by video_id.
  2. Origin returns the immutable segment plus cache headers. Edge stores popular segments for 7-30 days and may prefetch the next 3-5 segments for the same rendition.
  3. If origin pull exceeds 300ms, the manifest can bias the player toward lower initial bitrates; if a rendition is missing, the player falls back to the nearest lower ladder entry.

Server-side state that persists across requests: playback session id, DRM license lease, latest committed playhead, CDN request id, and optional recently-served segment window for abuse detection.

The read path is not a video service serving bytes; it is a manifest service plus a CDN key lookup, with origin object storage only on cold misses.

Per-Title Encoding Ladder

Both produce multiple variants per video: - Resolutions: 240p, 480p, 720p, 1080p, 4K. - Codecs: H.264 (compat), HEVC, AV1. - Audio: multiple languages, formats (stereo, 5.1, Atmos).

Netflix tunes per-title (saves bandwidth via content-aware bitrates); YouTube uses similar but for billions of videos optimizes per-video too.

Multi-tier: - Device cache - Edge POP - Regional cache - Origin (rare hit)

Adaptive Bitrate (ABR)

HLS or DASH manifest lists segments at multiple bitrates. Player measures network + buffer, picks bitrate per segment. Switches up or down between segments. Smooth playback regardless of network condition.

Playback degradation state:

Stage Mode Source Latency target Transition
1 Edge hit, target bitrate Edge POP <30ms segment fetch Normal steady state.
2 Regional hit Regional cache <100ms Edge miss or new POP.
3 Origin pull Object storage <300ms first miss Regional miss; cache segment after pull.
4 Lower bitrate Same CDN path Avoid rebuffer Buffer below threshold or origin/edge latency high.
5 Audio-only / error Manifest service Fast fail No playable rendition or entitlement failure.

The cursor-like state is the player's segment number plus rendition choice; the server can remain stateless for byte delivery because segment names encode position.

DRM

Netflix: strict (Widevine, FairPlay, PlayReady). License bound to device + session. Required by studios.

YouTube: optional. Premium content uses DRM; ads/free content doesn't.

Recommendation

See Design a Netflix-Style Recommendation System.

Two-stage: offline candidate generation per user + online ranking.

YouTube layers in subscription feed + trending. Netflix uses heavy A/B and per-title encoding decisions tied to viewer modeling.

Storage Tiering

Hot tier: recent / popular derivatives, CDN-fronted. Warm tier: older but still accessed, cheaper storage. Cold tier: originals + rarely-accessed; archival.

YouTube has more tier movement because of long-tail content.

Worked sizing:

  • Catalog: 50B videos * 4KB = 200TB raw; with 3 replicas across 2 regions, about 1.2PB. This is large but not the dominant cost.
  • Playheads: 1B profiles * 200 active titles * 120B = 24TB raw; with 3 replicas, about 72TB. Keep only the active window in the hot store and archive older history.
  • Segment egress dominates. If 20M peak streams average 2Mbps, peak egress is 40Tbps before overhead; shaving 10% bitrate with per-title encoding saves several Tbps.
  • Segment storage dominates persistent bytes. A 10-minute video at 5 renditions and 2Mbps average is roughly 750MB; 50B such videos would be tens of EB before replicas and cold-tier compression.

Main cost knobs: per-title bitrate tuning, CDN hit ratio, shorter edge TTL for unpopular derivatives, cold-tier originals, and not precomputing rare codec/rendition combinations until demand appears.

Multi-Region

Both rely heavily on global CDNs. Metadata replicated globally. Recommendation features per region. User playhead per home region.

Failure Modes And Mitigations

Transcode failure. Retry; degrade.
Hot video CDN miss. Multi-tier; pre-warm.
Celebrity livestream or trailer launch. Pre-position manifest and first segments by region; shard counters by video_id bucket.
Dormant long-tail video requested. Cold origin pull, lower initial bitrate, then lazy edge population.
Edge cache window exhausted by deep seek. Fetch by segment number from regional/origin; prefetch around new seek position.
Storage cost. Tier aggressively.

DRM down. Cached sessions; alert.

Stale content after delete. Cache invalidation.
Region capacity exhausted. Cross-region failover.
Multi-region catalog drift. Manifest uses catalog version; stale regions reject deleted/tombstoned videos at entitlement check.

Operational Metrics

Correctness: - Transcode success. - Stale serve after delete. - DRM compliance.

Capacity: - CDN hit ratio. - Egress bandwidth. - Player rebuffer rate.

Business: - Watch time. - Engagement per video. - Recommendation CTR.

Tradeoffs To Call Out

  • Per-title encoding ladders save bandwidth vs uniform ladders.
  • ISP-embedded CDN (Netflix) reduces transit but requires hardware deployment vs global CDN.
  • Object storage is right for immutable segments; a relational database would waste query features and collapse under byte-range egress.
  • CDN delivery beats app-server streaming because cache locality and TCP termination dominate cost, not business logic.
  • Precomputing every codec/rendition improves first play but multiplies storage; lazy rare-rendition encoding reduces bytes at the cost of cold-start delay.
  • DRM strict (Netflix) for content licensing vs open (YouTube most content).
  • Pre-positioning vs lazy load.
  • Curated catalog (Netflix) vs UGC (YouTube) drives storage strategy.

2. Simulated Interview

Interviewer: Design YouTube or Netflix.

Candidate: Shared architecture: upload + async transcode produces multi-resolution derivatives; multi-tier CDN serves via adaptive bitrate streaming. Differences: YouTube has billions of UGC videos with optional DRM; Netflix has thousands of curated titles with strict DRM and ISP-embedded Open Connect CDN. Recommendation drives discovery. The invariant is durable storage and sub-second playback start across network conditions.

Interviewer: Walk me through upload + transcode.

Candidate: Creator/studio submits video via resumable upload to object storage. Async transcoder splits into segments and parallel-transcodes to multiple resolutions and codecs (H.264, HEVC, AV1) plus audio tracks and HDR variants. Per-title bitrate ladders optimize bandwidth — a slow drama needs less than an action movie. First derivative ready → video publishable. Higher quality lands async. Derivative store backs the CDN.

Interviewer: CDN strategy?

Candidate: Multi-tier. Device cache → edge POP → regional cache → origin. Hit ratio target above 90%. Netflix runs Open Connect: physical appliances inside ISPs that pre-fill content during off-peak and serve subscribers locally during peak, keeping traffic off transit. YouTube uses Google's global CDN with massive edge presence and on-demand caching. Both pre-warm hot content.

Interviewer: Adaptive bitrate.

Candidate: HLS or DASH manifest lists segments at multiple bitrates. Player measures bandwidth and buffer, picks bitrate per segment. Between segments, may switch up (improving conditions) or down (degrading). Smooth playback regardless. Server just serves segments; intelligence is in the player.

Interviewer: DRM.

Candidate: Netflix: strict, mandatory. Widevine for Chrome/Android, FairPlay for Apple, PlayReady for Windows. Content encrypted; player gets a license bound to device + session. License has expiry; renewals during long playback. YouTube: DRM optional, used for premium content; free/ad-supported is open. The studio licensing relationship drives this difference.

Interviewer: Recommendation.

Candidate: Two-stage. Offline batch (daily/hourly) builds per-user candidate sets from collaborative filtering, content embeddings, popularity. Online ranker scores top candidates at request time with real-time features and personalization. YouTube layers subscription feed, trending, and watch history. Netflix uses heavy A/B and per-title performance modeling.

Interviewer: Storage at exabyte scale.

Candidate: Tiered. Hot derivatives in fast object storage with CDN backing. Warm content in cheaper storage. Cold originals in archival. Reference counts on derivatives; cleanup when unused. Re-transcode in newer codecs (AV1) over time to save egress bandwidth.

Interviewer: A viral video — sudden million views per minute.

Candidate: Multi-tier CDN absorbs. Pre-warm edges based on signals (recommendation surfacing, social shares). Request coalescing at edges. Engagement counters sub-shard for the hot video. View counter aggregator runs async; periodic recount.

Interviewer: YouTube vs Netflix differences.

Candidate: Catalog: YouTube is UGC, billions of videos, mostly long tail; Netflix is curated thousands of titles, top-N drives most viewing. Storage: YouTube spends more on cold tail; Netflix more on top-quality derivatives. DRM: Netflix strict; YouTube optional. CDN: Netflix Open Connect inside ISPs; YouTube via Google CDN. Recommendation: YouTube exploration-heavy for discovery; Netflix exploitation-heavy for proven titles.

Interviewer: Resume / playhead.

Candidate: Per-profile per-video playhead. Player heartbeats every 30s. Server-side authoritative. On reopen, "Continue Watching" row reads playhead. Synced across devices.

Interviewer: Engagement (likes, comments).

Candidate: YouTube has heavy engagement (likes, comments, view counts) — see Design a YouTube Likes Counter. Sub-counter sharding for hot videos. Periodic recount. Netflix has lighter engagement (thumbs up/down for personalization, not displayed publicly).

Interviewer: Multi-region.

Candidate: Both rely on global CDN. Derivatives geo-replicated for fast edge fill. Metadata replicated globally. User playhead in home region. Recommendation features per-region with cross-region global trending.

Interviewer: Summarize.

Candidate: Shared video pipeline: upload, async transcode with per-title ladder, multi-tier CDN with high hit ratio, ABR streaming via HLS/DASH. Differences: catalog (UGC vs curated), DRM (optional vs strict), CDN (global vs ISP-embedded), recommendation philosophy. Two-stage recommendation. Engagement counters sub-sharded. Storage tiering by age. Multi-region via CDN. The dominant cost is egress; both platforms invest heavily in CDN.

3. Practice Notes

Five-Minute Version

  1. Clarify: catalog (UGC vs curated), DRM, scale.
  2. Invariant: durable; sub-second start; smooth ABR.
  3. Upload + async transcode with per-title ladder.
  4. Multi-tier CDN (Open Connect vs global Google CDN).
  5. ABR via HLS/DASH.
  6. DRM: strict (Netflix) or optional (YouTube).
  7. Two-stage recommendation.
  8. Storage tiering; engagement counters (YT) sub-sharded.

One-Minute Closeout

YouTube and Netflix share a video pipeline: upload, async transcode with per-title bitrate ladders, multi-tier CDN with high hit ratio, adaptive bitrate streaming via HLS/DASH, two-stage recommendation. They differ in catalog (UGC vs curated), DRM (optional vs strict), and CDN (global vs ISP-embedded). The dominant cost is egress; both invest in CDN. The hardest correctness problem is durable storage at exabyte scale; the hardest performance problem is sub-second start.
