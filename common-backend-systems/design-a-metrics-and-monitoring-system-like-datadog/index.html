1. Basic Knowledge And Solution

What This Question Is Testing

A metrics platform ingests time-series data from millions of hosts and serves dashboards and alerts. The hard part is the cardinality explosion (metric × dimensions × hosts), efficient storage, low-latency queries over long ranges, and reliable alert evaluation.

The invariant is this:

Every metric data point sent to the system is durably stored within bounded latency, retained according to retention policy, queryable by metric name and tag filters across configured time ranges, and used in alert evaluation that fires reliably within the alert's evaluation window.

At ICT5 / L6+ level, the interviewer is testing whether you can design time-series ingestion at scale, downsampling and retention, query engine for aggregations across high-cardinality tag spaces, alert evaluation, and reason about cardinality control.

60-Second Interview Framing

Spoken version:

I would design the metrics system as three planes: ingestion that receives metric data points and writes them to time-series storage, storage that is column-oriented and downsampled over retention tiers, and query/alert that evaluates dashboards and alert rules. The interesting decisions are cardinality control, downsampling strategy, query latency over long ranges, and alert evaluation reliability.

Requirements To Clarify First

Functional requirements:

  • Hosts and services emit metrics (counters, gauges, histograms, distributions).
  • Metrics tagged with dimensions (host, service, region, etc.).
  • Dashboards query metrics across time ranges.
  • Alert rules: threshold, anomaly, rate-of-change.
  • Notifications when alerts fire.
  • Logs and traces (often integrated; we focus on metrics).

Non-functional requirements:

  • Ingest millions of data points per second.
  • Query latency: dashboards under a second; long-range queries seconds.
  • Retention: high-resolution recent (1s/10s), downsampled long-term (5min, 1h).
  • Alert evaluation: every minute typically.
  • Multi-region.
  • Auditability for alert state and silences.

Out of scope unless asked:

  • Log management (similar but distinct).
  • APM tracing.
  • RUM and synthetic monitoring.

Data Model

Metric Time-Series

TimeSeries(
  series_id,                // hash of (metric_name, tag_set)
  metric_name,
  tag_set: {key: value, ...},
  type,                     // counter, gauge, histogram, distribution
  data_points: [(ts, value, ...)]
)

A data point is (timestamp, value). Histograms and distributions have multiple values (buckets) per point.

Tag Index

TagPosting(
  tag_key,
  tag_value,
  series_ids: [series_id]
)

For query "metric X where region=us-east": look up region=us-east posting, intersect with metric_name index.

Alert Rule

AlertRule(
  rule_id,
  name,
  query,                    // metric query + threshold
  evaluation_interval,
  for_duration,             // alert fires only if condition true for X
  thresholds,
  notification_targets,
  silence_windows,
  state,                    // pending, alerting, ok, silenced
  version
)

Alert Event

AlertEvent(
  event_id,
  rule_id,
  state_change,             // ok -> alerting
  evaluated_at,
  trigger_value,
  metadata
)

Ingestion Pipeline

  1. Agent on host batches metrics every 10s, sends to ingestion gateway over HTTP/2 or gRPC.
  2. Gateway authenticates customer, rate-limits, validates schema.
  3. Writes to Kafka-like buffer for durability.
  4. Writer consumes; for each data point: - Compute series_id = hash(metric_name, sorted_tag_set). - Append point to series. - Update tag index if series is new.
  5. Writer is sharded by series_id so each writer handles a subset.

Storage

TSDB is columnar and compressed:

  • Time-series data is highly compressible: delta encoding for timestamps, Gorilla-style float compression for values.
  • Per-series compaction over time.
  • Indexed by series_id + time block.

Storage tiers:

  • Hot (recent days): high-resolution (1-10 second), in-memory or SSD.
  • Warm (recent months): lower resolution (1-5 minute), local disk.
  • Cold (long-term): hour-aggregated, object storage.

Downsampler runs periodically to roll up older data to lower resolutions.

Interview phrase:

Time-series compression is the secret weapon. Gorilla-style encoding gives 10-50x compression on typical metrics. Downsampling extends retention without exploding storage.

Cardinality Control

Cardinality = number of unique time-series. Adding a high-cardinality tag (like user_id) explodes cardinality and storage.

Controls:

  • Per-customer cardinality budget; alert and throttle when exceeded.
  • Tag whitelist/blacklist.
  • Allowlist for high-cardinality tags requiring approval.
  • Automatic cardinality detection: alert on metrics with rising cardinality.

The hardest production problem in metrics systems is cardinality explosion. A customer adding user_id to a counter can blow up the storage cost overnight.

Query Engine

Query language like PromQL or Datadog's: avg(rate(http_requests{service="api"}[5m])).

Execution:

  1. Parse query AST.
  2. Resolve series matching tag filters: tag index lookup.
  3. For each matching series, fetch data points in the time range.
  4. Apply functions (rate, avg, sum, max).
  5. Group by specified dimensions.
  6. Return time-series result.

Optimizations:

  • Pre-aggregate common rollups.
  • Cache query results.
  • Parallelize across series shards.
  • Distributed query plan for large ranges.

For long-range queries (15 months), use downsampled tier; for recent ranges, use high-resolution.

Alert Engine

Each alert rule has an evaluation interval (e.g., every minute).

  1. Scheduler triggers rule evaluation.
  2. Query engine evaluates the rule's expression for the eval window.
  3. Compare result to threshold.
  4. Update rule state: - ok -> pending if threshold crossed. - pending -> alerting after for_duration of sustained alerting. - alerting -> ok when threshold no longer crossed.
  5. On state change, emit alert event.
  6. Notification service pages on-call.

Reliability:

  • Alert evaluation must not silently fail.
  • Per-rule heartbeat; alert if rule hasn't evaluated in N intervals.
  • Idempotent evaluation; replay-safe.
  • Backfill on evaluator outage to catch missed firings.

Silences:

  • Per-rule silence windows.
  • Maintenance mode.
  • Suppress notifications without losing state.

Interview phrase:

An alert that doesn't fire is a worse bug than a false alarm. Alert evaluation must heartbeat and self-monitor.

Multi-Region Strategy

  • Per-region ingestion and storage; metrics generally consumed where generated.
  • Cross-region query federation for global dashboards.
  • Alert evaluation region-local; cross-region for global alerts.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Ingestion overload. Rate-limit per customer; buffer absorbs. Buffer depth.
Cardinality explosion. Per-customer budget; alert; throttle. Cardinality growth.
Hot series. Add shard tag; pre-aggregate. Per-series write rate.
Query timeout. Fall back to downsampled tier; partial result. Query p99.
Alert evaluator outage. Heartbeat; backfill on recovery. Eval lag.
Notification flap. Hysteresis (for_duration); silences. Flap count.
Storage growth. Downsampling; retention enforcement. Storage per customer.
Tag index hot key. Cache; partition further. Hot key QPS.
Cross-region query lag. Federate with bounded staleness. Federation latency.
Agent crash on host. Heartbeat detect; alert. Agent absence.

Operational Metrics

Correctness:

  • Lost data points (must be ~zero).
  • Alert evaluation lag.
  • Cardinality growth.

Capacity:

  • Ingest QPS by customer.
  • Query p99.
  • Storage growth.

Business:

  • Active metrics per customer.
  • Alert firings.
  • Dashboard usage.

Tradeoffs To Call Out

  • Higher resolution costs more storage; downsampling extends retention but loses detail.
  • Pre-aggregating rollups speeds queries but adds write work.
  • Strict cardinality limits help cost but frustrate customers.
  • Async ingestion improves throughput but adds query lag.
  • Alert evaluation interval tunes responsiveness vs cost.
  • Long for_duration reduces flap but delays notification.

Common Traps And Stronger Moves

Ignore cardinality. Budget; alert; throttle.
Synchronous ingest. Buffer + async writer.
One global query engine. Sharded; federated.

2. Simulated Interview

Interviewer: Design Datadog metrics.

Candidate: Three planes: ingestion that receives metric data points from agents and writes durably; storage that is columnar with downsampling tiers; query/alert that evaluates dashboards and alert rules. Invariant: every data point durable; queryable; alerts reliable. The hardest problem is cardinality combined with long-range queries.

Interviewer: How is data stored?

Candidate: Per-series time-ordered points. Series_id is a hash of metric_name + sorted tag_set. Storage is columnar — timestamps and values stored separately, both compressed with Gorilla-style encoding for 10-50x reduction. Compaction merges segments over time. Tiered: hot (1-10s resolution, in-memory/SSD), warm (1-5 min, local disk), cold (1 hour, object storage). Downsampler rolls up periodically.

Interviewer: Cardinality explosion?

Candidate: The biggest production issue. Adding a user_id tag to a counter can blow up storage. Defenses: per-customer cardinality budget; alert when exceeded; throttle ingestion if needed. Tag allowlist/blocklist. Cardinality detection: a metric whose unique series count is rising fast triggers alarms. Customer-facing tools to show high-cardinality offenders.

Interviewer: Query — avg(rate(http_requests{service="api"}[5m])).

Candidate: Parse the AST. Resolve service="api" from the tag index to get matching series. For each series, fetch data points for the requested window. Apply rate() per series (delta over time). Apply avg() across series. Return result series. Parallelized across series shards. For long ranges, use the downsampled tier automatically.

Interviewer: Alert evaluation?

Candidate: Scheduler runs each rule per its eval interval (typically every minute). For each rule, query engine evaluates the expression for the eval window. Compare to threshold. State machine: ok → pending (when crossed) → alerting (after for_duration sustained) → ok. On state change, emit event; notification service pages on-call. Each rule heartbeats; if a rule hasn't evaluated in N intervals, we alert about the evaluator. Replay-safe: re-running an interval doesn't double-fire.

Interviewer: Why heartbeat?

Candidate: A silent alert failure is the worst outcome. If the evaluator dies and we don't know, on-call has no signal about real production issues. Heartbeating ensures we detect evaluator death; backfill on recovery catches missed firings.

Interviewer: Hot metric — one metric receives 1M points/sec.

Candidate: Single series can't handle 1M points/sec — single writer is the bottleneck. Mitigation: add a shard tag (random suffix) that splits the metric into N sub-series. Customers re-aggregate at query time. Alternative: pre-aggregate at ingest with a stream processor.

Interviewer: Downsampling — explain.

Candidate: Old high-resolution data is rolled up. For example, after 7 days, 1-second points become 1-minute averages, mins, maxes, sums. Storage cost drops dramatically. Queries over long ranges use the downsampled tier automatically. Downsampler runs as a background job, idempotent.

Interviewer: Multi-tenant — how do you isolate noisy tenants?

Candidate: Per-tenant rate limits at ingestion. Cardinality budgets. Storage shards may be per-tenant for large customers; shared for small. Query engine bounds per-tenant resource usage. Alert engine per-tenant fairness.

Interviewer: Multi-region?

Candidate: Per-region ingestion and storage. Most metrics consumed locally. Global dashboards federate queries across regions. Cross-region alerts evaluate in a designated region. Cross-region latency tolerated for dashboards (seconds).

Interviewer: Summarize.

Candidate: Three planes: ingest, store, query/alert. Columnar Gorilla-compressed storage with downsampling tiers. Tag-based inverted index for series lookup. Alert engine with heartbeat and replay-safety. Cardinality control prevents the most common production headache. Multi-region with federation. The dominant scaling problem is cardinality; the dominant correctness problem is alert reliability.

3. Practice Notes

Five-Minute Version

  1. Clarify: scale, retention, alerts.
  2. Invariant: durable points; reliable alerts.
  3. Ingest → buffer → writer → columnar TSDB.
  4. Gorilla compression + downsampling tiers.
  5. Tag inverted index.
  6. Query engine with parallel + downsample tier.
  7. Alert engine with heartbeat.
  8. Cardinality control as continuous process.

One-Minute Closeout

A metrics platform is high-throughput time-series ingest plus columnar compressed storage plus query and alert engines. Gorilla compression keeps storage feasible; downsampling extends retention without exploding cost. Tag inverted index supports filtered queries. Alert engine evaluates per-rule with heartbeats so silent failures are detected. Cardinality is the dominant production concern; per-customer budgets and detection control it.

Best Interview Phrases

  • "Gorilla compression makes time-series storage feasible."
  • "Downsampling extends retention; recent data is high-resolution, older is rolled up."
  • "Cardinality explosion is the biggest production headache; budget and detect."
  • "Alert evaluation must heartbeat; silent failure is worse than false alarms."
  • "Multi-region uses per-region storage with federated queries."
