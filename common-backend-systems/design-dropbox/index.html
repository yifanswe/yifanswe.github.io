1. Basic Knowledge And Solution

What This Question Is Testing

Dropbox synchronizes file trees across many devices per user. The hard part is efficient sync (don't re-upload unchanged content), conflict resolution when multiple devices change a file, deduplication at scale, and offline-friendly UX.

The invariant is this:

Every committed file change is durably stored, eventually replicated to all of the user's devices, and a given file's content at any point matches one of a deterministic set of versions; concurrent edits produce conflict copies (or a merge) rather than silent data loss; deletes are reversible within a retention window; storage charges scale with deduplicated unique content.

At ICT5 / L6+ level, the interviewer is testing whether you can design chunk-based deduplication, an efficient sync protocol that minimizes bandwidth, a metadata service that owns the canonical file tree per user, and conflict handling under multi-device editing.

60-Second Interview Framing

Spoken version:

I would design Dropbox as three planes: a metadata service that owns each user's file tree and version history, a content-addressable block store for file chunks with cross-user dedup, and a sync protocol that exchanges deltas between clients and the metadata service. The interesting decisions are chunking strategy, delta sync, conflict resolution, and dedup that respects privacy.

Requirements To Clarify First

Functional requirements:

  • User uploads, modifies, deletes files in a folder tree.
  • Files sync across the user's devices.
  • Sharing: folders shared with other users with permissions.
  • File versioning: see and restore older versions.
  • Selective sync: not every device gets every file.
  • Web access: download files from a browser.
  • Comments, links, mobile preview.

Non-functional requirements:

  • Petabyte-scale storage with cross-user dedup.
  • Sync latency: changes propagate within seconds when devices are online.
  • Bandwidth-efficient: don't re-upload unchanged parts.
  • Durable: file content never lost.
  • Available offline: clients work without network.
  • Multi-region.

Out of scope unless asked:

  • Real-time collaborative editing (separate; see Google Docs).
  • Full-text search across all files.
  • DRM / managed file rights.

Chunking And Dedup

Each file is split into chunks. Choice of chunking strategy matters:

Fixed-size chunks (e.g., 4 MB): - Simple. - Bad for insertions: changing the start of a file shifts all chunks; dedup breaks.

Variable-size chunks via content-defined chunking (CDC, e.g., Rabin fingerprint): - Boundary determined by content; insertion only affects nearby chunks. - Better dedup for slightly-modified files. - More CPU at chunk time.

For each chunk:

  • block_id = SHA-256(content).
  • Upload if not exists (check via API).
  • Reference count incremented when a file version uses it.
  • When references drop to zero, eligible for garbage collection.

Dedup scope:

  • Per-user (always).
  • Cross-user (privacy-sensitive; requires careful key management).

For cross-user dedup with encryption:

  • Convergent encryption: derive encryption key from content hash; two users with the same file produce the same ciphertext. Enables dedup but leaks "do you have this file?"
  • Per-user encryption: prevents cross-user dedup but is privacy-safer.

Interview phrase:

Content-defined chunking with content-addressable blocks gives both bandwidth efficiency and dedup. Cross-user dedup is a privacy decision, not just an engineering one.

Sync Protocol

Each device has a local cursor over the user's event stream. The metadata service maintains a per-user append-only event log.

Sync flow:

  1. Client connects; sends its cursor.
  2. Server responds with events since the cursor: file added, modified, deleted, moved, shared.
  3. For modifications: client compares its local block list with the server's; downloads missing blocks; updates local file.
  4. Local change: client computes new block list; uploads new blocks (checking existence first); commits new file version with block_refs.
  5. Server emits event; other devices pull and sync.

Bandwidth efficiency:

  • Only changed blocks transferred.
  • Existence check (HEAD /blocks/{id}) before upload.
  • Server-side delta computation for small edits.

Conflict Resolution

Two devices edit the same file while offline. Both upload modified versions.

Server-side detection:

  • Each FileVersion has parent_version_id.
  • If a client commits with parent_version_id = V, but the current server version is V' (where V' != V), conflict.

Resolution policy:

  • Default: create a "conflict copy" — filename (Device A conflict copy).ext. Both versions preserved.
  • Three-way merge for text: optional in some products.
  • For binary files, no merge; conflict copy is the right answer.

Interview phrase:

Dropbox does not silently merge edits; it preserves both versions as conflict copies. Three-way merge is risky for binary files.

Move Vs Copy

Moving a file changes the metadata but does not re-upload content.

  • Move: metadata transaction changing parent_id or name.
  • Copy: new node, same block_refs, increment reference counts.

This makes large folder moves nearly free.

Sharing

Shared folders have a shared root node visible in multiple users' trees.

  • Shared folder has its own ACL.
  • Each member's tree references the shared folder by id.
  • Permissions checked at the metadata service.
  • Sync protocol delivers events for shared folders to all members.

Versioning

Each file modification creates a new FileVersion.

  • Old versions retained for a window (30 days typical) or longer for paid tiers.
  • Restore: copy old version's block_refs as a new current version.
  • Garbage collection removes old versions past retention.

Scaling And Partitioning

Data Partition key Why
Metadata nodes user_id (or workspace_id) Per-user tree; transactional within partition.
Event log user_id Per-user sync cursor.
Blocks block_id hash Even distribution.
Block ref counts block_id hash Same shard as block.
Shared folders shared_folder_id Independent of any one user's partition.

Hot partition: a single user with millions of files in one folder. Mitigation: sub-shard the folder if it grows beyond threshold; or simply accept and scale the user's partition.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Conflict between concurrent edits. Conflict copy. Conflict rate per user.
Upload incomplete (network drop). Resumable; partial chunks discarded. Incomplete-upload age.
Block corruption. Hash verification on read; replicate; alert. Hash mismatch rate.
Garbage collection deletes referenced block. Reference count + grace period; verify before delete. GC-deleted-but-referenced count.
Metadata store hot user partition. Sub-shard; or vertical scale. Per-partition QPS.
Sync stuck on bad event. Skip with quarantine; manual repair. Stuck-cursor count.
Cross-user dedup leaks existence. Per-user dedup as default; convergent encryption with care. Privacy audit.
Restore old version after GC. Maintain version history bounded by retention. Restore failures past retention.
Shared folder revoked. Metadata change; client removes local copy. Revoke propagation lag.
Massive folder upload (10M files). Throttle; batch events; progress reporting. Per-user upload rate.

Operational Metrics

Correctness:

  • Conflict rate per user.
  • Hash mismatch on block read.
  • Sync cursor lag.
  • Lost-block events (must be zero).

Capacity:

  • Upload/download QPS and bandwidth.
  • Metadata service QPS per user.
  • Event log fan-out latency.

Business:

  • Dedup ratio.
  • Storage per user.
  • Active devices per user.
  • Sharing activity.

Multi-Region Strategy

  • User home region for metadata.
  • Block store replicates globally for fast download.
  • Sync notification routes to user's home region; remote devices subscribe via local gateways.
  • Cross-region replication latency tolerated.

Tradeoffs To Call Out

  • Content-defined chunking saves bandwidth but costs CPU; fixed-size is simpler but less efficient.
  • Cross-user dedup saves storage but raises privacy concerns.
  • Storing every block forever wastes space; ref-count + GC adds complexity.
  • Three-way merge is nice for text but dangerous for binary; default to conflict copies.
  • Larger blocks reduce metadata overhead but reduce dedup benefit; smaller blocks the opposite.
  • Strong per-user consistency limits cross-user transactions; transactional shared folders are harder.

Common Traps And Stronger Moves

Trap Stronger move
Re-upload entire file on change. Chunked delta sync; check block existence.
Silent merge of concurrent edits. Conflict copy as default; ask user.
Last-writer-wins. Detect concurrent edit via parent_version_id.
Synchronous metadata + block writes. Two-phase: blocks first, metadata commit second.
Sharing as duplication. Shared folder referenced by both users.
Single global metadata service. Per-user sharding.

  • Metadata + blocks split.
  • Content-addressed blocks with dedup.
  • Chunked upload/download with existence check.
  • Per-user event log for sync.
  • Conflict copy as default for concurrent edits.
  • Sharing via shared folder nodes.

Interview Phrase

Dropbox is a per-user file tree with content-addressed block storage. Chunked delta sync minimizes bandwidth; conflict copies preserve both versions on concurrent edits; cross-user dedup is a privacy tradeoff. The metadata service is the source of truth; the block store is content-addressable and immutable.

2. Simulated Interview

Interviewer: Design Dropbox.

Candidate: I will design two planes plus a sync protocol. The metadata plane owns each user's file tree, versions, and sharing — strongly consistent per user. The block plane is content-addressable storage, immutable, deduplicated. The sync protocol exchanges events from a per-user log between clients and the metadata service, transferring only changed blocks.

Interviewer: How does a file upload work?

Candidate: Client chunks the file using content-defined chunking — boundaries determined by content fingerprint. Each chunk's block_id is the SHA-256 of its content. For each chunk, the client checks HEAD /blocks/{block_id}. If exists, skip. Otherwise, upload. After all blocks exist, the client commits a new FileVersion with the list of block_refs and the parent_version_id. The metadata service performs the commit transactionally; emits an event.

Interviewer: Why content-defined chunking?

Candidate: Insertions and small edits don't shift the chunks. If I insert text at the top of a file, fixed-size chunking shifts every chunk boundary and dedup breaks. CDC sets boundaries based on content, so only the chunks immediately around the edit change. The rest dedupes.

Interviewer: How does sync work?

Candidate: Each device has a cursor in the user's event log. On connect, it sends its cursor; the server returns all events since (created, modified, deleted, moved, shared). For modifications, the client compares the new version's block_refs against what it has locally; downloads only missing blocks; reconstructs the file. The event log is per-user; sync is bounded by the user's activity, not the global system.

Interviewer: Two devices edit the same file while offline.

Candidate: When device A comes online and commits a new version, the metadata service applies it and updates current_version_id. When device B comes online and tries to commit with parent_version_id = old_version, the server detects conflict: the current version is no longer the parent device B thinks. Server creates a conflict copy — a new node with name like file (Device B conflict copy).ext. Both versions are preserved. The user resolves manually.

Interviewer: Why not auto-merge?

Candidate: For text, three-way merge is feasible. For binary — most files — it's dangerous. A silent merge corrupts content. Conflict copies make both versions visible so the user decides. Some products do offer text merging as opt-in.

Interviewer: Storage — dedup helps?

Candidate: A lot. Same content uploaded by the same user dedupes by content hash. Cross-user dedup is more sensitive: if two users have the same file, do they share blocks? Yes saves storage but leaks existence information. Convergent encryption (encryption key derived from content) supports cross-user dedup, but exposes "do you have this file?" attacks. Per-user dedup is the safer default.

Interviewer: How do moves work?

Candidate: Metadata-only operation. Moving a 1 TB folder is a single transaction changing parent_id. No content re-uploaded. Same for renames. Copies create new nodes with the same block_refs and increment reference counts.

Interviewer: Versioning?

Candidate: Each modification creates a new FileVersion linked to the prior. Old versions retained for 30 days (or per tier). Restore copies an old version's block_refs as a new current. Garbage collection removes unreferenced versions and decrements block ref counts; blocks with zero refs and past a grace period are deleted from the block store.

Interviewer: Sharing?

Candidate: Shared folder is a node that multiple users' trees reference. ACL on the shared folder governs permissions. Sync events for the shared folder flow to all members. Adding a file to a shared folder publishes the event; all members' clients pick it up and download blocks.

Interviewer: Selective sync?

Candidate: Per-device configuration of which folders sync locally. Client honors the config: subscribe to events for those folders only; download blocks only for those. Metadata still has the full tree; the client just doesn't materialize unselected folders.

Interviewer: Multi-region?

Candidate: Per-user home region for metadata. Block store geo-replicated for fast download. Notification service has regional gateways; the user's home region publishes events; remote gateways subscribe and forward to local devices. Cross-region replication is async with bounded lag.

Interviewer: A user uploads 10 million files at once.

Candidate: Throttle. Per-user upload rate limit. Batched event emission to avoid notification spam. Progress reported to the client. Metadata transactions are bounded in size; the client uploads incrementally. The user's partition may become hot; we sub-shard if needed.

Interviewer: Summarize.

Candidate: Metadata service per user owns the file tree. Block service stores content-addressed immutable chunks with dedup. Chunked delta sync minimizes bandwidth via existence checks. Conflicts produce conflict copies. Sharing is a shared folder node with ACL. Versioning + reference counting + GC manage storage. Per-user home region with replicated blocks for multi-region. The hardest correctness problem is conflict handling; the hardest scale problem is large folder operations and per-user partition hotspots.

3. Practice Notes

Five-Minute Version

  1. Clarify: sync, sharing, versioning, devices.
  2. Invariant: per-user metadata authoritative; blocks immutable.
  3. Metadata + content-addressed block store.
  4. Content-defined chunking; existence check before upload.
  5. Per-user event log for sync.
  6. Conflict copies for concurrent edits.
  7. Sharing via shared folder node.
  8. Versioning + GC; multi-region per-user home.

One-Minute Closeout

Dropbox is a per-user file tree on top of content-addressed block storage. Chunked delta sync only transfers what changed; existence checks before upload exploit dedup. Concurrent edits become conflict copies; auto-merge is risky for binary files. Sharing uses a shared folder node referenced by multiple users. Versioning is append-only with reference-counted GC. Per-user home region with globally-replicated blocks gives both correctness and performance.

Best Interview Phrases

  • "Content-defined chunking with content-addressed blocks gives both bandwidth efficiency and dedup."
  • "Concurrent edits produce conflict copies; we don't silently merge binary files."
  • "Moves and copies are metadata-only operations; no content re-uploaded."
  • "Cross-user dedup is a privacy decision, not just an engineering one."
  • "Per-user event log makes sync bounded by activity, not by global scale."
