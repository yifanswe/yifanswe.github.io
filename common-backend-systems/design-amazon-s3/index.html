1. Basic Knowledge And Solution

What This Question Is Testing

S3 is the canonical planet-scale object store. The hard part is offering 11 nines of durability and near-100% availability while supporting trillions of objects across exabytes of storage, where individual disk and node failures happen constantly.

The invariant is this:

Every PUT that returns success has its data durably stored across multiple failure domains; every GET against a stored key returns the latest committed version (or the requested version for versioned buckets); listing, key uniqueness, and ACL enforcement are correct; data survives any single-AZ failure without loss.

At ICT5 / L6+ level, the interviewer is testing whether you can design erasure-coded storage with high durability, separate metadata from data, handle high-throughput key listing, manage hot keys, and reason about consistency (strong since the 2020 update, eventual prior).

60-Second Interview Framing

Spoken version:

I would design S3 as two services: a metadata service that maps object keys to data placements, and a data plane that stores object content using erasure coding or replication across availability zones. The interesting decisions are durability via erasure coding, key namespace partitioning, consistency (now strong read-after-write), hot-prefix handling, and multipart uploads for large objects.

Requirements To Clarify First

Functional requirements:

  • PUT, GET, DELETE objects by key in a bucket.
  • LIST objects by prefix; sorted lexicographically.
  • Multipart upload for large objects.
  • Bucket versioning (optional per bucket).
  • Lifecycle policies: tier or expire after N days.
  • Access control via IAM and bucket policies.
  • Presigned URLs.
  • Cross-region replication (optional).

Non-functional requirements:

  • 11 nines of durability (99.999999999%).
  • High availability per region.
  • Strong read-after-write consistency (modern S3).
  • Object size: 1 byte to 5 TB.
  • Throughput: thousands of requests per second per prefix.
  • Petabyte to exabyte scale per customer.
  • Geo-replication and DR.

Out of scope unless asked:

  • Direct integration with compute (e.g., S3 Select).
  • Querying object content (Athena, Glue).
  • Object transformations (Lambda triggers).

Durability And Erasure Coding

11 nines means losing one object per 10^11. Achieving this requires:

  • Spread chunks across availability zones (independent failure domains).
  • Erasure coding (e.g., 10+4 Reed-Solomon): split object into 10 data chunks + 4 parity chunks; survive any 4 chunk losses; 1.4× storage overhead vs 3× for triple replication.
  • For small objects, replication may be cheaper than EC overhead.

Placement:

  • For each object, pick N + M storage nodes across at least 3 AZs.
  • Ensure no two chunks of one object share a rack within an AZ.
  • Background repair: when a node fails, regenerate its chunks on a healthy node from the surviving chunks.

Interview phrase:

Durability is achieved by spreading chunks across failure domains plus erasure coding. The math: with chunks across 3 AZs and Reed-Solomon 10+4, the probability of losing more than 4 chunks before repair is astronomically small.

Write Path

For a typical PUT under 100 MB:

  1. Client PUT to frontend with key, content, headers.
  2. Frontend authenticates, checks ACL, generates request_id.
  3. Compute content hash (etag) as content is streamed in.
  4. Erasure-encode into k+m chunks.
  5. Placement service picks k+m storage nodes across AZs.
  6. Frontend writes chunks in parallel; waits for quorum (or all).
  7. After all chunks written and confirmed, frontend commits to metadata service: (bucket, key, version) -> chunk_refs.
  8. Metadata write returns; ACK to client.

For strong read-after-write consistency, the metadata commit must be visible to all readers before the client receives success. This typically means the metadata service writes through consensus.

Multipart Upload (For Large Objects)

For objects 100 MB to 5 TB:

  1. Client initiates multipart upload: gets uploadId.
  2. Uploads parts in parallel; each part is independently stored.
  3. Each part has its own placement and etag.
  4. After all parts uploaded, client calls complete with list of (partNumber, etag).
  5. Metadata service records the object as a composition of parts.
  6. Client GET retrieves the composition and streams parts sequentially.

Benefits:

  • Resumable: failed parts retried independently.
  • Parallel: many parts uploaded simultaneously.
  • Bounded memory: server doesn't buffer the whole object.

Read Path

  1. Client GET to frontend.
  2. Auth and ACL check.
  3. Frontend queries metadata service for object's chunk placements.
  4. Frontend reads chunks from storage nodes in parallel.
  5. If EC: reconstruct from any k of k+m chunks; survives slow or failed nodes.
  6. Stream bytes to client.

For hot objects, frontend may cache chunk content briefly; or a CDN sits in front.

Listing

LIST is the painful operation: ordered by key lexicographically, paginated.

Implementation:

  • Per-bucket B-tree-like index keyed by object key.
  • Range scan from marker for max-keys entries.
  • Sharded by key range; range scan spans shards if needed.

Performance:

  • LIST is much slower than GET/PUT.
  • Customers are encouraged to LIST sparingly and cache results.

Hot bucket (millions of keys): partition the bucket's key space into ranges; each range is its own metadata shard.

Consistency Model

Modern S3 (post-2020):

  • Strong read-after-write for new objects.
  • Strong consistency for overwrites and deletes.
  • No more "list shows old, get returns new" issues.

Achieved by:

  • Metadata service is strongly consistent (consensus per range).
  • LIST queries the strongly consistent index.
  • Frontend coordinates write-then-ack only after metadata commit.

Hot Prefix Handling

The metadata service partitions by key prefix. A bucket with millions of writes to keys starting with logs/2024/01/ pins one shard.

Mitigations:

  • Customer guidance: use hashed prefixes for high-throughput patterns.
  • Adaptive partitioning: S3 auto-splits hot prefixes.
  • Pre-warm: contact AWS for known high-throughput buckets.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Disk fails. Erasure coding survives; background repair. Disk failure rate; rebuild backlog.
AZ fails. Chunks across AZs; service continues from remaining. AZ availability; quorum size.
Hot prefix. Adaptive partitioning; customer guidance. Per-prefix QPS.
Metadata shard hot. Sub-shard; cache popular reads. Per-shard QPS.
Slow storage node. Read from EC quorum, skip slow node. Per-node tail latency.
Correlated failure (firmware bug). Diverse hardware; staged rollouts. Cross-AZ failure correlation.
Object versioning conflict. Versioning generates unique version_id. Version conflict count.
Multipart upload incomplete. Garbage collection of stale parts. Stale uploads age.
Cross-region replication lag. Async with monitoring; alert on lag. Replication lag distribution.
ACL bug exposes data. Defense-in-depth checks; audit logs. Unauthorized access attempts.

Operational Metrics

Durability:

  • Repair backlog; time-to-repair after disk failure.
  • Lost-object events (must be near zero).
  • Cross-AZ chunk distribution.

Capacity:

  • PUT/GET QPS by region.
  • Latency p50/p99 by operation.
  • Storage growth.
  • Per-prefix QPS.

Business:

  • Storage per customer.
  • Request volume per customer.
  • Egress bandwidth.

Multi-Region Strategy

  • Each region operates independently.
  • Cross-region replication is opt-in per bucket; PUT replicates to peer region async.
  • Replication can be one-way or bidirectional.
  • Buckets are region-scoped; key namespace per bucket per region.

Tradeoffs To Call Out

  • Erasure coding gives high durability with lower overhead than replication but adds CPU and read complexity.
  • Strong consistency adds metadata write latency but eliminates a class of user bugs.
  • Per-prefix partitioning makes listing fast but creates hot-prefix bottlenecks.
  • Versioning preserves history at storage cost.
  • Cross-region replication is expensive; usually opt-in.
  • Small objects pay metadata overhead disproportionately; aggregating tiny objects helps.

Common Traps And Stronger Moves

Trap Stronger move
Triple replication everywhere. Erasure coding for large objects; replication for small.

S3 is metadata plus data plane. The data plane achieves 11 nines via erasure coding across AZs and continuous background repair. The metadata service is strongly consistent per key range. The hardest scaling problems are hot prefixes and durability under correlated failures; the answers are adaptive partitioning and diverse failure domains.

2. Simulated Interview

Interviewer: Design S3.

Candidate: I will design two services: metadata that maps object keys to data placements, and a data plane that stores object bytes using erasure coding across availability zones. The invariant is durability — every successful PUT survives any single-AZ failure, with 11 nines target. Strong consistency for reads after writes since the 2020 update. I will walk through PUT, GET, multipart, and durability math.

Interviewer: Walk me through a PUT.

Candidate: Client sends PUT with key and bytes. Frontend authenticates, checks ACL, streams the data and computes the etag. The bytes are erasure-encoded into k+m chunks — say 10 data chunks plus 4 parity chunks. The placement service picks 14 storage nodes spread across at least 3 AZs. Frontend writes chunks in parallel; after all confirm, commits to the metadata service: (bucket, key, version_id) -> chunk_refs. Metadata commit goes through consensus per key range. Once committed, ACK to client.

Interviewer: Why erasure coding over triple replication?

Candidate: Storage overhead. Triple replication is 3× overhead. Reed-Solomon 10+4 is 1.4× overhead with comparable durability. For large objects, the difference is huge at petabyte scale. For small objects, EC has metadata overhead that may exceed the savings; replication is sometimes better. Modern S3 uses a hybrid.

Interviewer: 11 nines — how do you achieve it?

Candidate: Spread chunks across 3 AZs. With Reed-Solomon 10+4, we tolerate 4 chunk losses before data loss. AZs are independent failure domains. Probability of losing 5 chunks before repair completes is astronomically small with continuous background repair. Concrete math: assume disk MTBF 1M hours, repair time 1 hour, 14 chunks across 3 AZs — the combined probability over a year is well below 10^-11. Diverse hardware and firmware reduce correlated failure risk.

Interviewer: Walk me through a GET.

Candidate: Client GET to frontend. Auth + ACL. Metadata lookup returns the chunk_refs. Frontend reads chunks from storage nodes in parallel. Because of EC, any k chunks (10 of 14) are sufficient to reconstruct. We initiate reads to all 14 and accept the first 10 to complete, which gives natural tail-latency mitigation. Stream bytes to client.

Interviewer: A 1 TB object — how?

Candidate: Multipart upload. Client initiates and gets an uploadId. Uploads parts in parallel (e.g., 100 MB each, 10,000 parts). Each part is independently stored — its own erasure-coded chunks. The client retries failed parts. After all parts complete, client sends a finalize request with the list of (partNumber, etag). The metadata service records the object as a composition of parts. GET reconstructs by reading parts sequentially or in parallel.

Interviewer: Strong consistency — how?

Candidate: The metadata service is strongly consistent per key range, via consensus (Paxos or Raft within the range's replica set). After a successful PUT commit, any subsequent GET sees the new version. LIST queries the same strongly consistent index. Prior to 2020, S3 was eventually consistent for new keys after delete; modern S3 fixed this.

Interviewer: A customer writes 100,000 objects/sec all with keys starting with logs/2024/01/. What happens?

Candidate: All those writes hit the metadata shard owning that prefix range. Hot. Mitigations: adaptive partitioning — S3 auto-detects hot prefixes and splits the range across more shards. Customer guidance: use a hashed prefix to spread load. For known high-throughput buckets, pre-partition. We monitor per-prefix QPS and alert.

Interviewer: A disk dies. What happens?

Candidate: The chunks on that disk are missing. Background repair detects and rebuilds them on a healthy node from the surviving k chunks of each affected object. Repair is continuous; backlog is monitored. Until repair completes, GETs still work because we have remaining k+m-1 chunks (still above the k threshold).

Interviewer: Versioning?

Candidate: Per-bucket toggle. When enabled, every PUT to an existing key creates a new version_id rather than overwriting. The metadata service indexes by (bucket, key, version_id). DELETE creates a delete marker; the object can be restored. Lifecycle policies can expire old versions.

Interviewer: Lifecycle?

Candidate: Per-bucket policy: e.g., "after 30 days move to Infrequent Access, after 1 year to Glacier, after 7 years delete." Lifecycle service runs daily, scans bucket index, applies transitions. Moving to Glacier may involve re-encoding for different storage class with different durability/latency tradeoffs.

Interviewer: Cross-region replication?

Candidate: Per-bucket configuration: source → destination. On every PUT, after metadata commit, replication service enqueues a copy. The peer region receives the object via internal API. Async; lag typically seconds to minutes. Monitor replication lag.

Interviewer: Listing a bucket with billions of objects?

Candidate: LIST is the hardest operation. The metadata service has a per-bucket B-tree-like ordered index. Range scan from marker for max-keys entries. If keys span multiple shards, the frontend stitches. We strongly encourage customers to use specific prefixes to bound the scan and to paginate.

Interviewer: Summarize.

Candidate: S3 is metadata + data plane. Erasure coding across AZs gives 11 nines. Metadata is strongly consistent per key range via consensus. Multipart upload handles large objects. Adaptive partitioning handles hot prefixes. Background repair survives constant disk failures. Lifecycle and replication are async pipelines. Strong consistency was added in 2020 by making the metadata commit synchronous before ack.

3. Practice Notes

Five-Minute Version

  1. Clarify: durability, throughput, multipart, consistency.
  2. Invariant: 11 nines durability; strong R-A-W.
  3. Two services: metadata + data plane.
  4. Erasure coding across 3 AZs; e.g., RS 10+4.
  5. Multipart upload for large objects.
  6. Strong consistency via metadata consensus.
  7. Adaptive partitioning for hot prefixes.
  8. Background repair; lifecycle; cross-region replication.

One-Minute Closeout

S3 is metadata plus data plane. The data plane achieves 11 nines via erasure coding spread across availability zones with continuous background repair. The metadata service is strongly consistent per key range; PUT acks only after metadata commit. Multipart upload handles objects up to 5 TB. Adaptive partitioning handles hot prefixes; customer guidance to use hashed keys helps. Lifecycle and cross-region replication are async pipelines. The hardest correctness problem is durability under correlated failures; the answer is diverse failure domains and continuous repair.

Best Interview Phrases

  • "Erasure coding across AZs gives durability with lower storage overhead than replication."
  • "Strong consistency requires the metadata commit before the PUT ack."
  • "Multipart upload is mandatory for large objects; parallel and resumable."
  • "Hot prefixes are mitigated by adaptive partitioning and customer-side key hashing."
  • "Continuous background repair makes constant disk failures invisible to users."
