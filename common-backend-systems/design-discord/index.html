1. Basic Knowledge And Solution

What This Question Is Testing

Discord is real-time chat organized into servers, channels, and voice rooms. The hard part is the combination: persistent text channels with rich history (like Slack/IRC), real-time presence and voice (like Skype), and the scale of millions of concurrent connections per server.

The invariant is this:

Every sent message is durably stored in its channel, delivered to all online members of the channel, and remains retrievable for the channel's retention; presence reflects who is currently online and in which voice channel; voice and video traffic is routed peer-to-peer or through media servers with low latency; permissions are enforced at the channel and role level.

At ICT5 / L6+ level, the interviewer is testing whether you can design persistent channel-based chat, real-time presence at scale, voice/video routing via WebRTC + media servers (SFU), permissions modeled via roles, and reason about server (community) scale where one server may have millions of members.

60-Second Interview Framing

Spoken version:

I would design Discord as a chat substrate (channel-based persistent messages with WebSocket push), a presence/voice control plane (who's online, who's in which voice channel), and a media routing layer (SFU-based voice/video). Servers (communities) are groups of channels with role-based permissions. The interesting decisions are channel message storage, real-time fan-out to online members, voice room routing, and handling very-large servers (1M+ members).

Requirements To Clarify First

Functional requirements:

  • Users join servers (communities) with channels.
  • Text channels: persistent message history.
  • Voice channels: real-time audio + screen share + video.
  • Direct messages between users.
  • Presence (online, idle, dnd, offline).
  • Reactions, attachments, threads.
  • Roles and permissions per channel.
  • Notifications when mentioned.

Non-functional requirements:

  • Real-time message delivery (<200 ms).
  • Voice latency under 150 ms.
  • High availability per server.
  • Multi-region.
  • Very-large servers (1M+ members) supported.

Out of scope unless asked:

  • Bot platform and APIs.
  • Nitro / payment.
  • Stage channels (similar to voice).

Chat Path (Text)

Send:

  1. Client → WebSocket → gateway → chat service.
  2. Chat service authenticates, checks channel permissions.
  3. Assigns snowflake message_id (time-ordered).
  4. Persists to message store partitioned by (channel_id, time_bucket).
  5. Publishes MESSAGE_CREATE event.
  6. Gateway fan-out to all subscribed clients (online members of the channel).
  7. Notification service triggered for mentions.

Receive:

  • Clients subscribe to channels they have open / member of.
  • Receive events over WebSocket.
  • Read message history via REST (GET /channels/{id}/messages?before=).

Message ordering:

  • Snowflake message_id is monotonic per channel (or per shard with embedded timestamp).
  • Clients use message_id to order display.

Channel Fan-Out

Discord's notable design choice is fan-out only to online subscribers, not maintaining offline mailboxes per user.

  • Each gateway tracks which channels each connection subscribes to.
  • On message: gateway determines which connections are subscribed; push to those.
  • Offline users miss the real-time push; they pull via REST when they come back online.

For very-large channels (1M+ members), only a fraction are online at once. The fan-out is bounded by online subscribers, not total members.

Discord uses Elixir/Erlang's process model for managing millions of connections per gateway node, with a "guild process" per server handling routing.

Interview phrase:

Discord fan-out is to online subscribers only, not to per-user mailboxes. This scales the system to multi-million-member servers because most members are offline at any moment.

Presence At Scale

Presence is ephemeral — who's online, what they're doing.

Challenge: for a 1M-member server, even just sending "user X is now online" to all members is heavy.

Optimization:

  • Presence updates are batched.
  • For huge servers, only members in the same channel or recently active receive presence updates.
  • "Lazy presence": list of online users is fetched on demand; not pushed proactively to everyone.

Storage: in-memory with TTL; not durable.

Voice / Video Architecture

Voice and video use WebRTC over UDP.

SFU (Selective Forwarding Unit):

  • Each participant sends a single audio/video stream to the SFU.
  • SFU forwards each participant's stream to all other participants.
  • Each participant receives N-1 streams (decode locally).
  • Much cheaper than MCU (mixing at server) for moderate room sizes.

Flow:

  1. User joins voice channel via voice control service.
  2. Voice control assigns a media server (SFU) close to the user.
  3. User connects to SFU via WebRTC; ICE negotiation, DTLS handshake.
  4. SFU joins user to the channel's media room.
  5. Streams flow between user and SFU.

For large rooms (Stage channels with 1000s of listeners):

  • Active speakers send to SFU.
  • Listeners receive a forwarded stream.
  • SFU may transcode to reduce bandwidth.

Multi-region:

  • SFUs in multiple regions.
  • Cross-region SFU-to-SFU bridging for global voice rooms.
  • Voice control picks the SFU region based on participant locations.

Permissions And Roles

Per-server roles with permission bitfields:

  • E.g., MANAGE_MESSAGES, KICK_MEMBERS, SPEAK_IN_VOICE.

Per-channel permission overrides:

  • A specific role/user has permission X allowed or denied in this channel.

Computed effective permissions:

  • Server-level base from member's roles, OR-combined.
  • Apply channel overrides.

Evaluated on every operation (send, delete, kick, etc.). Cached at the gateway for hot paths.

Scaling And Partitioning

Data Partition key Why
Server metadata server_id Co-locate channels, roles, members.
Members (server_id, user_id) Membership lookup.
Channel messages (channel_id, time_bucket) Time-range scan.
Presence user_id In-memory; ephemeral.
Voice rooms channel_id → SFU Real-time routing.
Roles server_id Per-server.

Per-server sharding: each server's data (members, channels, messages) lives on a small set of shards. A "guild process" in Elixir manages live state.

For multi-million member servers: shard the server itself across multiple guild processes; or use a special architecture.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Gateway crashes. Client reconnects; resumes from sequence cursor. Reconnect rate.
Massive server fan-out. Online-only push; lazy presence. Per-server connection count.
Voice media server fails. Migrate participants; ICE renegotiation. SFU health.
Presence storm. Batched updates; throttle. Presence update rate.
Message store hot shard. Per-channel + time partitioning. Shard QPS.
Permission cache stale. Short TTL; explicit invalidation on role change. Permission revoke lag.
Voice cross-region latency. Pick SFU near majority of participants. Voice latency p99.
Snowflake clock skew. Use NTP; cap drift; tiebreak. Snowflake conflicts.
Spam in channel. Rate limit per channel per user; spam filter. Spam rate.
1M-member server burst. Online-only fan-out; pull-based offline. Fan-out latency.

Operational Metrics

Correctness:

  • Lost messages.
  • Permission bypass attempts.
  • Voice drop rate.

Capacity:

  • Active connections per gateway.
  • Message QPS per channel.
  • Voice room churn rate.

Business:

  • DAU; messages per channel; voice minutes.

Multi-Region Strategy

  • Server has a home region for guild process + chat state.
  • Members anywhere connect via local gateway, which routes to home region.
  • Voice SFUs distributed; chosen by participant locations.
  • Cross-region replication for DR.

Tradeoffs To Call Out

  • Online-only push scales fan-out but offline users see history via REST (acceptable).
  • SFU is cheaper than MCU but pushes decode cost to clients.
  • Lazy presence avoids storms but presence may lag.
  • Per-server guild process simplifies state but limits cross-server queries.
  • Aggressive permission caching speeds operations but staleness on revoke.

Common Traps And Stronger Moves

Trap Stronger move
Per-user mailbox for channel messages. Online-only fan-out; REST for history.
Synchronous presence broadcast to all members. Batched + lazy presence.
MCU for voice. SFU; lighter server load.
Single global gateway. Regional gateways.
Re-evaluate permissions per op without cache. Cached effective permissions; invalidate on change.
Single guild process for 1M-member server. Shard the server.

Staff-Level Answer Checklist

  • Chat persistent + presence ephemeral + voice real-time as three planes.
  • Online-only fan-out for channels.
  • SFU for voice with regional routing.
  • Role-based permissions with channel overrides.
  • Snowflake ids for time-ordered messages.
  • Erlang/Elixir-style process model for connections.
  • Multi-region with server home region.

Interview Phrase

Discord is three planes overlapping in one app: persistent text chat, ephemeral presence, and real-time voice/video. The key scaling insight is online-only fan-out — messages broadcast to currently subscribed connections, not stored in per-user inboxes. Voice rides through SFUs near participants. Permissions are role-based with channel overrides.

2. Simulated Interview

Interviewer: Design Discord.

Candidate: Three planes: persistent text channels with append-only history, ephemeral presence/typing, and real-time voice/video via SFUs. Each user has a WebSocket connection to a gateway; servers (communities) own channels with role-based permissions. The invariant is durable messages with per-channel order, low-latency voice, and permission-correct access.

Interviewer: Walk me through sending a message.

Candidate: Client sends via WebSocket. Gateway forwards to chat service. Chat service authenticates, checks channel permissions, assigns a snowflake message_id, persists to the message store partitioned by (channel_id, time_bucket). Publishes a MESSAGE_CREATE event. The gateway looks up subscribed connections for this channel and pushes the event. Notification service triggered for mentioned users.

Interviewer: A server has 2 million members. How does fan-out work?

Candidate: Online-only fan-out. The system tracks which connections subscribe to which channels — those are the only recipients of real-time push. Most of 2 million members are offline at any moment; fan-out is bounded by online subscribers (maybe 50k-100k). Offline users see history via REST pull when they reconnect. Discord uses Elixir's process model — a guild process per server handles routing — with sharding for very large servers.

Interviewer: Why no per-user mailbox like Messenger?

Candidate: Discord channels are pub-style: everyone in the channel sees the same messages. Storing one copy per user would explode storage for large channels. Channel-level history is shared. Users pull when they need history; real-time push goes only to online subscribers. The tradeoff: offline users miss live notification context but see history on reconnect.

Interviewer: Walk me through voice.

Candidate: User joins voice channel. Voice control service assigns an SFU close to the user. Client establishes WebRTC connection to SFU: ICE candidate exchange, DTLS handshake. SFU joins the user to the channel's media room. Each participant sends one audio stream to the SFU; the SFU forwards each participant's stream to every other participant. Each client decodes the N-1 streams it receives. For large stage channels, only active speakers send; listeners receive a single forwarded stream.

Interviewer: Why SFU and not MCU?

Candidate: SFU is just a packet forwarder — much cheaper than MCU which decodes, mixes, and re-encodes. For moderate room sizes (up to dozens of speakers), SFU scales better. The cost shifts to the client: each client decodes N-1 streams. For very large audience-style rooms, you might mix at the server to reduce client load, but that's a design choice.

Interviewer: Presence — 1M users online.

Candidate: Ephemeral in-memory state with TTL. The challenge is presence updates: sending "user X is now online" to every server member is heavy. Mitigations: lazy presence — clients pull the online list for a server when they open it, rather than pushing every change to everyone. Updates that do propagate are batched. For users in the same voice channel or currently chatting, presence pushes more aggressively.

Interviewer: Permissions?

Candidate: Server-level roles with permission bitfields (MANAGE_MESSAGES, KICK_MEMBERS, etc.). Each member has one or more roles; permissions OR-combined. Channels can override: "this role can't speak in this voice channel." Effective permission = (server-base from roles) ∪ (channel allow overrides) − (channel deny overrides). Cached at the gateway for performance; invalidated on role/permission change.

Interviewer: Multi-region?

Candidate: Each server has a home region — the region of its guild process and chat state. Members anywhere connect via local gateway, which routes to the server's home region. Voice SFUs distributed; chosen by participant geo. For globally-distributed servers, cross-region routing adds latency, accepted in chat but problematic for voice if the server's voice goes through a far SFU.

Interviewer: Snowflake ids?

Candidate: 64-bit ids combining timestamp + worker + sequence. Provides natural time-ordering across distributed message creators. Critical for displaying messages in order without sorting by wall clock (which would have skew). Discord uses snowflakes for messages, channels, servers, users.

Interviewer: A server gets DDoS'd via mass spam.

Candidate: Per-channel per-user rate limits enforced at the gateway. Anti-spam heuristics (repeated messages, links). Auto-moderation tools per server allow admins to set keyword filters, slowmode (one message per N seconds). Severe cases trigger temp mute or kick.

Interviewer: Summarize.

Candidate: Three planes: persistent text channels, ephemeral presence/typing, real-time voice/video via SFUs. Online-only fan-out scales channels to multi-million members. Voice rides through SFUs near participants. Role-based permissions with channel overrides, cached for performance. Server home region; SFUs regional. Erlang/Elixir process model handles millions of connections per node. The hardest scaling case is the 1M-member server; the answer is online-only fan-out plus per-server guild process sharding.
