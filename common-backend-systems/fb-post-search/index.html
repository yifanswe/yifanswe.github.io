Facebook Post Search — Full-Text Search Across the Firehose (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build Facebook (FB) post search — like Twitter's search bar. User types a query, gets ranked recent posts that match the keywords, filtered by their network/visibility. Index 1B new posts/day; serve 10k searches/sec; results within a few seconds of post creation.

1. Requirements (~5 min)

Functional - Search recent posts by keyword(s), optionally filtered by date/author/type. - Results respect post visibility (public / friends / private). - Index new posts near-real-time (target: post visible in search within ~30 s).

I'd defer ranking depth (we can mention ML reranking) and image/video search content.

Non-functional - Search p99 < 500 ms. - Indexing throughput keeps up with 1B posts/day (~12k posts/sec average). - Strong consistency on visibility — never leak a private post. - Eventual consistency on document availability — a 30 s delay between post and search-visibility is acceptable.

Capacity - 1B posts/day × 1 KB = 1 TB/day of post text, ~30 TB/month. - Inverted index ≈ 0.5× document size with term posting lists. - Multi-year retention → petabytes. - Query fan-out is bounded by time routing: most searches hit recent shards first and only expand older if fewer than K visible results come back.

2. Core Entities (~2 min)

  • Post — post_id, author_id, text, created_at, visibility (public/friends/private), entities (mentions, hashtags).
  • IndexShard — inverted index keyed by term → list of (post_id, score, ts).
  • User — needed for visibility checks (friend graph).

3. API (~5 min)

GET /v1/search?q=...&filters=...&cursor=...    → { posts[], next_cursor }

User from token. Cursor is opaque server state (term, last seen score+id) for pagination stability.

The cursor encodes {query_hash, time_bucket, shard_offsets, last_score, last_post_id, visibility_epoch} so page 2 uses the same visibility snapshot and does not reshuffle while new posts arrive.

4. Data Flow (~5 min)

Post created → published to Kafka posts.created → indexer consumes, tokenizes, applies analyzer, writes to inverted index shard → search service queries indexed shards on read.

5. High Level Design (~10–15 min)

  1. Post service publishes post.created events to Kafka. Out of scope to design; we're consumers.
  2. Indexer service. Stream processor (Flink/Kafka consumer pool): - Tokenize text (Unicode-aware, lowercase, stem optionally). - Strip noise; extract entities. - Compute initial static score (author authority, post type). - Write document to the index.
  3. Index storage. Elasticsearch (or a custom inverted index over a key-value store like RocksDB). Shard by time bucket (recent month vs older) and within bucket by term hash for spread. - Recent month sharded for write-heavy near-real-time indexing. - Older months are read-mostly, can be merged into fewer, larger shards.
  4. Visibility filtering. Two approaches: - Filter at query time: join results against user's friend list. Works if friend lists are reasonable size (a few thousand). - Indexed visibility tokens: index a synthetic term per allowed-viewer-group (public, friend-list-id). Query ANDs the user's tokens with the search terms. - I'd default to query-time filtering for friends and indexed tokens for public/private boolean.
  5. Search service. Parses query → fan-out to relevant shards (most recent first) → merge sort → rerank top N with a ranking model → return.
  6. Ranking pipeline. Lightweight learn-to-rank model (LightGBM): features include text relevance (BM25), recency, author affinity, engagement. Rerank top 200 down to top 20.
  7. Cache. Common queries cached for 30 s (hot trending searches dominate volume).

End-to-end narration: Alice posts "Niners just won the Super Bowl!" → indexer in 5 s adds the post to today's shard. Bob searches "Super Bowl" 30 s later → search service fans out across last-week shards → each shard returns its top 50 → merge to top 200 candidates → rerank model picks the freshest, most engaging, most network-relevant → top 20 returned, including Alice's post.

Concrete query path:

  1. Search service parses terms, filters, and user visibility context.
  2. It routes to recent time buckets first, shard key (time_bucket, term_hash).
  3. Shards fetch posting lists for each term, intersect/score locally, and return top 50 post IDs plus BM25 scores.
  4. In parallel, the service fetches user friend/block filters and ranking features for the candidate authors/posts.
  5. It applies fine-grained visibility checks, hydrates post snippets by batched post ID lookup, reranks top 200, and returns top 20 with cursor.

Write/index path:

  1. post.created is appended to Kafka with key (created_at_bucket, post_id); updates/deletes use the same post_id so indexers see versions in order per post.
  2. Indexer tokenizes text, emits one posting per unique term, and writes the document snippet under doc:{post_id}.
  3. Public visibility is written as a coarse token; friends/private posts keep enough metadata for query-time ACL checks and are never trusted solely from the index.
  4. The indexer writes postings to hot time-bucket shards keyed by (bucket, term_hash) and advances an index_watermark:{bucket}.
  5. Search only routes to shards whose watermark is healthy; a lagging shard is marked degraded and retried asynchronously rather than blocking all queries.

Hot vs cold query path: a trending public query first checks query:{hash}:public and may return in <100 ms. A personalized query misses cache, fans out to recent shards, over-fetches top 500, fetches visibility:{user_id}:{epoch} in parallel with ranking features, then filters and reranks. If fewer than K visible results survive, the cursor expands to the next older time bucket with the same query_hash and shard offsets.

6. Deep Dives (~10 min)

Near-real-time indexing. Elasticsearch refreshes new segments every ~1 s by default; tuned for our volume that's fine. New posts appear searchable within seconds.

Concrete index stores:

Inverted postings (bucket, term_hash) compressed list of (post_id, ts, term_freq, static_score) ~12-20 B/posting hot 30 days SSD; older tiered term_hash within time bucket
Document store post_id text snippet, author, visibility, entity ids ~1 KB same as post retention post_id

Hot query cache query:{hash}:visibility:{tier} top-N post IDs + scores ~10 KB 30 s query hash

Capacity math: 1B posts/day × 1 KB = 1 TB/day raw document text. If each post averages 20 unique terms and each posting is ~16 B compressed, postings are ~320 GB/day before replicas; doc snippets add ~1 TB/day. With 3 replicas and 30 hot days, hot search storage is roughly 120 TB before compression wins. Multi-year retention is object-store/cold-index territory. Dominant cost is postings/doc storage for hot recent shards plus query CPU on skewed terms.

Indexing queue math: 12k posts/sec average at ~1.5 KB/event is ~18 MB/sec raw through Kafka, ~54 MB/sec with RF=3. A 30 s freshness target means normal lag should stay under ~360k events; if indexers are down for 10 minutes, the backlog is ~7.2M posts, or ~10 GB raw event data, which Kafka can hold easily but shard refresh/merge CPU becomes the limiter. Hot query cache is modest: 1M cached query variants × 10 KB is ~10 GB raw with a 30 s TTL, but cache invalidation is intentionally time-based, not per-post.

Time-sharded index. Most searches are weighted to recent posts. Shard by time bucket so recent shards are tiny and indexed hot; older shards are larger and read-mostly. Query routing hits the freshest shards first, falls back to older shards only on insufficient results.

Two indexes, one answer. The inverted index answers "which posts contain these terms?" The graph/visibility index answers "which of those posts can this viewer see?" I would not build a per-user search index because the storage would explode; instead, over-fetch from the term index and apply the visibility view at read time before hydration.

Why Elasticsearch/OpenSearch. It gives inverted indexes, refresh/segment merge, query routing, and scoring out of the box. Solr is a close alternative. A custom RocksDB inverted index can be cheaper at FB scale, but it means owning analyzers, merges, ranking, and operations. Bigtable/Cassandra are not enough by themselves because the key operation is term posting-list intersection, not a single wide-row read.

Term skew. Hashtags like #superbowl produce massive posting lists. Mitigations: per-shard caching of top postings; pre-computing top-K hot terms with their best-scoring docs (a "hot phrase" cache).

Skew degradation:

Stage Mode Source Latency target
normal live shard fan-out recent postings <500 ms
hot term cached top-K query:{hash} / hot phrase cache <100 ms for first page
exhausted page continue cursor shard offsets + older bucket <700 ms acceptable
visibility-heavy over-fetch get top 500-1000 then filter more CPU, no leaks
shard failed partial results omit failed shard, mark degraded, retry async return some results

Ranking — text + behavior. BM25 for raw relevance. Boost by post engagement (likes/comments per minute). Personalize by author affinity (does the searcher interact with this author?). For staff-level depth, mention training data from clicks and "did this satisfy the query" signals.

Visibility correctness. The hard one. A search must never return a post the searcher isn't allowed to see. Two-stage: 1. Coarse filter at index time. Public posts indexed in a public shard; friends-only posts indexed with a synthetic term _friend_of_author_X. Searcher's query adds their author tokens. 2. Fine filter at result time. After retrieving, re-check each result against the searcher's friend graph for the small number of edge cases (recent unfriends, blocks).

The fine filter is non-negotiable. If the friend graph service is unavailable, fail closed for friends/private posts and return public-only results rather than risk a privacy leak.

The visibility cursor carries visibility_epoch; page 2 should use the same ACL snapshot or a newer fail-closed one. If Bob unfriends Alice between page 1 and page 2, page 2 may omit Alice's older result, but it must never resurrect it from a cached query page.

Privacy / blocking. If A blocks B, neither sees the other's posts. Maintain a block-list per user; cross-check at result time. Implement as a small bloom filter per user for O(1) check.

Update + delete propagation. Posts can be edited or deleted. Indexer subscribes to post.updated / post.deleted topics; updates inverted index (tombstone + later compaction). Stale results possible for a few seconds — acceptable.

Storage cost. Inverted index for 1 PB of raw text → ~500 TB. Tier by age: hot (SSD-backed ES nodes) for last 30 days, warm (HDD) for older, cold (snapshot to object store) for everything past a year.

Hot query caching. Trending queries (e.g., "Super Bowl") dominate traffic. Cache top-N results per query for 30 s. Cache key includes the searcher's coarse visibility tier (most users share the "public-only" view).

Multi-region and cold-start behavior. Indexing can happen in the post's home region and replicate segments to search regions; a remote region may be 5-30 s behind and that is acceptable for freshness. Visibility data is stricter: blocks and privacy changes replicate on a priority stream, and if the local ACL epoch is stale, search returns public-only or asks the home region for the fine check. A new time bucket starts empty, so routers keep querying the previous bucket until index_watermark:{bucket} proves the new shard is serving; after a shard restart, it replays Kafka from the bucket offset and search marks that shard partial until replay catches up.

Interview punch line for failure cases: I can return fewer or older results, but I cannot return one private result. The degradation path is partial/public-only results, not stale ACLs.

What I'd defer. Semantic / vector search, image/video search, autocomplete and query suggestion, multilingual analysis, query understanding (entity recognition, intent classification).

Interview punch line: search can be eventually consistent for freshness, but never eventually consistent for privacy. I would over-fetch and filter at read time before I would risk returning one private post.
