1. Basic Knowledge And Solution

What This Question Is Testing

A payment system moves money. Every bug is a regulator complaint, a chargeback, or a missing dollar that some accountant has to find. The hard part is not throughput; it is correctness over the lifetime of every transaction across many external systems that may time out, return ambiguous responses, or change state without telling you.

The invariant is this:

For every payment intent, the system records exactly one terminal outcome (settled, refunded, voided, declined), the money flow at each step is double-entry recorded in an immutable ledger, every external side effect uses idempotency, and reconciliation with provider statements detects drift that internal logs miss.

At ICT5 / L6+ level, the interviewer is testing whether you can model money as a state machine, design idempotent calls to providers, handle the "unknown state" outcome of timeouts, build a double-entry ledger, and run daily reconciliation against external sources of truth.

60-Second Interview Framing

Spoken version:

I would design the payment system around three things: a payment-intent state machine, a double-entry ledger as the source of truth for money movement, and idempotent integrations with payment providers wrapped in a reconciliation loop. The interesting decisions are how to handle provider timeouts, how to model refunds and chargebacks, how to enforce idempotency end-to-end, and how to reconcile when our records and the provider's disagree.

Requirements To Clarify First

Functional requirements:

  • Customers pay via various methods: cards, ACH, wallets, BNPL, bank transfer.
  • Merchants receive funds after settlement.
  • Refunds, partial refunds, voids, chargebacks.
  • Multi-currency.
  • Recurring billing / subscriptions.
  • Idempotent retries from clients and providers.
  • Webhooks from providers (success, failure, chargeback, dispute).
  • Reporting and reconciliation for finance teams.

Non-functional requirements:

  • Strong correctness: no double-charge, no lost transactions.
  • Resilient to provider outages.
  • Auditability: every state change traced.
  • Compliance: PCI-DSS, regional regulations (PSD2 SCA, KYC, AML).
  • Settlement may take seconds (cards) to days (ACH).
  • High availability for the authorize step (in-band with checkout).

Out of scope unless asked:

  • Fraud scoring algorithms (treat as input from a fraud service).
  • Tax calculation (separate system).
  • Merchant onboarding KYC (separate system).
  • Card vault / PAN storage (delegate to a vault).

Scale Assumptions

Area Assumption
Volume Up to millions of transactions per day at peak; tens of millions across markets.
Latency Authorize p99 under 2 seconds (provider round trip dominant).
Settlement window Card capture same-day; ACH 1-5 business days.
Refund window Up to 180 days post-charge.
Chargeback window Up to 540 days for some networks.
Multi-region Per-region issuers and acquirers; central ledger.
Reconciliation cadence Daily for provider settlements; near-real-time for high-risk.
Storage Years of transaction history; immutable.

The bottleneck is provider integration reliability and reconciliation — not raw QPS. A payment system processing 1000 TPS is huge.

Core Product Model

Three layers:

Layer Responsibility Requirement
Payment intent state machine Owns the lifecycle of each payment from creation to terminal state. Strongly consistent; explicit state transitions.
Money ledger Double-entry records of every credit/debit to internal accounts. Append-only, immutable, balanced.
Provider integrations Calls to card networks, banks, wallets. Idempotent; reconciled with provider statements.

The biggest design mistake is conflating the intent state with the ledger entries, or letting the application skip the ledger and trust the provider response directly.

Core Components

Component Responsibility Key detail
Payment API Receives intent creation, capture, refund requests from merchants. Idempotent on Idempotency-Key.

Payment Intent State Machine

                  +----->  authorized
                              |
                  +-----------+-----------+
                  |                       |
                  v                       v
              captured                  voided

                  |           |
                  v           v
              settled     refunded (full/partial)
                  |
                  v

Allowed transitions are explicit; every transition is conditional on current state and idempotent on a transition key. The ledger is updated only on specific transitions:

  • authorized: hold but no money moves (depending on rail).
  • captured: ledger entry: debit merchant pending / credit acquirer.
  • settled: ledger entry: provider funds received.
  • refunded: ledger entry: reversal.
  • charged_back: ledger entry: reverse + fee.

Interview phrase:

Money never moves implicitly. Every ledger entry corresponds to an explicit state transition with a recorded reason and idempotency key.

Idempotency End-to-End

Three layers of idempotency:

  1. Client → Payment API: Idempotency-Key header. The idempotency store caches the response keyed by (merchant_id, idempotency_key) for 24-48 hours. Retries with the same key return the same response.
  2. Payment API → Provider: Provider-specific idempotency mechanisms (Stripe's Idempotency-Key, others have request id). The adapter generates a deterministic key per attempt.
  3. State machine transitions: Each transition is a CAS update on (intent_id, from_state). Same transition request always produces same result.

Critical case: a charge times out from the provider. We do not know if the charge succeeded.

  • Mark the intent state processing with provider_status_unknown.
  • A reconciliation worker queries the provider with the same idempotency key.
  • If the provider confirms success, we transition to authorized or captured and write the ledger entry.
  • If failure, we transition to failed.
  • We never charge the customer twice for the same intent.

Interview phrase:

Timeouts are the central correctness problem. The system must treat "unknown" as a first-class state and resolve it via reconciliation, not by guessing.

Double-Entry Ledger

Every money movement is two entries:

debit  account_A   amount   transaction_id
credit account_B   amount   transaction_id

Internal accounts:

  • Merchant pending (between capture and settlement).
  • Merchant available (after settlement).
  • Provider in-transit.
  • Fees collected.
  • Refunds pending.
  • Chargebacks owed.

A charge of $100 by customer for merchant M:

At capture:
  debit  customer_funding             $100
  credit merchant_M_pending           $97
  credit platform_fee_revenue         $3

At settlement (provider deposits):
  debit  provider_in_transit          $97
  credit merchant_M_available         $97

At every point, debits == credits per transaction (double-entry balanced), and the sum of all balances equals zero. Any drift means a missing entry, which the reconciliation engine detects.

Properties:

  • Append-only; no updates or deletes.
  • Each entry has transaction_id, account, amount, currency, direction, timestamp, reference_intent.
  • Indexed by account for balance queries, by intent for traceability.

Write Path: Authorize / Capture

  1. Merchant POSTs intent with idempotency key.
  2. Payment API checks idempotency store; if cached response exists, return it.
  3. Validate amount, currency, method.
  4. Risk service scores; may decline upfront.
  5. Generate intent_id; persist with state=created.
  6. Call provider adapter with deterministic idempotency key.
  7. Provider response: - Success: transition to authorized; write ledger entry if capture mode is automatic. - Failure: transition to failed. - Timeout: transition to processing, schedule reconciliation. - SCA challenge: transition to requires_action; return challenge URL.
  8. Persist response in idempotency store.
  9. Return response to merchant.

For manual capture, a separate /capture call later triggers the actual money movement.

Webhook Path: Async Updates

Providers send webhooks for asynchronous events:

  1. Webhook receiver validates signature.
  2. Dedup by provider_event_id; ignore duplicates.
  3. Persist raw event.
  4. Look up affected intent; apply state transition with CAS.
  5. Write ledger entry if applicable.
  6. Notify merchant via merchant-side webhook.

Webhooks may arrive out of order or duplicated. Both are handled by idempotency on provider_event_id and CAS on intent state.

Reconciliation

Daily reconciliation against provider settlement files:

  1. Pull settlement file from provider (CSV / SFTP / API).
  2. For each provider record, find matching intent and ledger entry.
  3. Match on provider_charge_id and amount.
  4. Mismatches go to a manual review queue.
  5. Drift between our ledger sum and provider expected sum triggers an alert.

This catches:

  • Webhooks we missed.
  • Charges we recorded but provider didn't.
  • Currency conversion errors.
  • Fee discrepancies.

Interview phrase:

Reconciliation is the safety net for everything our event-based pipeline could miss. It is slow but authoritative. Drift between our ledger and the provider's settlement is the canary for systemic bugs.

Refunds, Voids, Chargebacks

Refund: money back to customer. New transition from captured or settled to refunded. Ledger entries reverse the original credits.

Void: cancel before capture. Transition authorized → voided. No ledger entry (no money moved).

Chargeback: customer dispute via card network. We receive a webhook; transition captured → disputed then to charged_back after resolution. Ledger entries reverse the original plus add chargeback fee. We engage dispute handler to provide evidence.

Multi-Currency

Each intent has a currency. Ledger entries record amounts in their native currency. Cross-currency conversion (e.g., USD merchant, EUR customer) involves a separate FX leg:

debit  customer_funding_EUR        €100
credit fx_holding_EUR              €100

debit  fx_holding_USD              $108
credit merchant_pending_USD        $108

The FX rate is locked at the time of capture; a separate FX service provides rates.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Provider timeout. Intent in processing state; reconciliation queries provider. Unknown-state count; reconciliation lag.
Provider outage. Circuit breaker; queue intents; merchant-visible error. Provider availability; queue depth.
Duplicate webhook. Dedup on provider_event_id. Duplicate event rate.
Out-of-order webhooks. CAS on intent state; idempotent transitions. Out-of-order transition attempts.
Ledger drift. Daily reconciliation; alerts; quarantine on mismatch. Drift per merchant/currency.
Lost webhook. Reconciliation catches via settlement file. Missing intent updates.
Refund without capture. State machine rejects invalid transition. Invalid transition attempts.
Double-capture from retries. Idempotency at API; provider-side idempotency. Double-charge complaints.
Currency mismatch in ledger. Schema enforcement; per-currency accounts. Cross-currency entry count.
Dispute volume spike. Risk score tuning; manual review queue. Chargeback rate per merchant.

Operational Metrics

Correctness:

  • Ledger drift per account per day.
  • Unknown-state intents older than threshold.
  • Unmatched provider settlement entries.
  • Duplicate webhook rate.
  • Refund without prior capture (impossible, alert if seen).

Capacity:

  • Intent creation rate.
  • Provider call latency and error rate per provider.
  • Reconciliation duration.
  • Webhook queue depth.

Business:

  • Authorization success rate.
  • Capture-to-settlement time.
  • Chargeback ratio.
  • Refund ratio.
  • Revenue per provider, per currency.

Multi-Region Strategy

  • Issuers and acquirers are regional; we maintain a provider per region.
  • Intent state and ledger writes route to merchant's home region.
  • Cross-region replication for read replicas and disaster recovery.
  • Webhooks land in the region the provider sends them to; route to merchant's region.

Tradeoffs To Call Out

  • Synchronous provider calls add latency but give immediate user feedback. Async webhooks add eventual updates.
  • Aggressive risk declines reduce fraud but hurt conversion.
  • Daily reconciliation is slow but authoritative; real-time reconciliation is fast but noisy.
  • Per-merchant idempotency is safe but stores grow; TTL bounded retention.
  • Single ledger gives auditability; sharded ledgers scale but complicate balance queries.
  • Tokenizing cards in vault adds latency but is a PCI requirement.

Common Traps And Stronger Moves

Trap Stronger move
Trusting provider success without writing ledger transactionally. Write ledger entry as part of state transition CAS.
Ignoring timeouts. First-class unknown state; reconciliation worker.
Single ledger row per transaction. Double-entry: every transaction has matched debit/credit.
Treating refunds as new charges. Refund is a state transition with reverse ledger entries.
Forgetting webhooks can be duplicated. Dedup on provider_event_id; idempotent transitions.
No reconciliation. Daily provider settlement match; drift alerts.

Staff-Level Answer Checklist

  • Model payment as a state machine with explicit transitions.
  • Double-entry ledger as source of truth for money movement.
  • End-to-end idempotency: API, provider, state CAS.
  • First-class unknown state for timeouts; reconciliation.
  • Webhook dedup and idempotent transitions.
  • Daily reconciliation against provider settlements.
  • Multi-currency via per-currency accounts and FX legs.
  • Compliance: PCI-DSS, SCA, KYC delegated appropriately.

Interview Phrase

A payment system is correctness, not scale. Every charge is a state machine transition that writes a balanced double-entry ledger; idempotency is end-to-end; unknown outcomes from provider timeouts are a first-class state resolved by reconciliation. Reconciliation is the canary for any drift the event-based pipeline misses.

2. Simulated Interview

Interviewer: Design a payment system.

Candidate: I will design three layers: a payment intent state machine, a double-entry ledger as the source of truth for money movement, and provider integrations wrapped in idempotency and reconciliation. The invariant is that every intent has exactly one terminal outcome, every money movement is two balanced ledger entries, and external timeouts resolve via reconciliation rather than retry-guess.

Interviewer: Walk me through a card charge.

Candidate: Merchant POSTs a payment intent with an idempotency key. The Payment API checks the idempotency store. If new, we create the intent in created state, run risk scoring, then call the card provider with a deterministic idempotency key. On success, transition to authorized and write a ledger entry (debit customer funding, credit merchant pending). If timeout, transition to processing and let a reconciliation worker query the provider. Webhook updates from the provider (capture confirmed, settlement) drive further transitions.

Interviewer: The provider times out. What now?

Candidate: This is the central correctness case. The intent state goes to processing with provider_status_unknown. A reconciliation worker queries the provider with the same idempotency key. If the provider says success, we transition to authorized and write the ledger entry. If failure, transition to failed. We never retry the charge blindly; that could double-charge. We never assume failure; that could leave the customer charged but our records empty.

Interviewer: Show me the double-entry ledger for a $100 charge.

Candidate: At capture, two entries: debit customer_funding $100, credit merchant_X_pending $97, credit platform_fee_revenue $3. At settlement: debit provider_in_transit $97, credit merchant_X_available $97. Every transaction is balanced; the sum of all balances is zero. A daily trial-balance check ensures no entry is missing.

Interviewer: How does reconciliation work?

Candidate: Daily, we pull the provider's settlement file. For each line, we find the matching intent and ledger entry by provider_charge_id. Mismatches go to a review queue. Aggregate drift between our ledger and the provider's expected sum triggers an alert. This catches anything our webhook pipeline missed — duplicates, lost events, fee discrepancies. The ledger is internal truth; the provider settlement is external truth; reconciliation makes them agree or escalates.

Interviewer: A webhook arrives twice.

Candidate: Dedup on provider_event_id. The webhook receiver stores each event id; duplicates are ignored after the first is processed. The state transition itself is also idempotent: CAS on (intent_id, from_state). Even if dedup is bypassed, the second transition fails because the state has already advanced.

Interviewer: A refund — how do you process it?

Candidate: The merchant calls /refund on a captured or settled intent. We transition state to refunded (or partial_refunded). The ledger entries reverse the original: credit customer funding, debit merchant pending or available. We call the provider's refund API with a new idempotency key. On confirmation, the refund completes. The original capture remains in the ledger; the refund is a new transaction referencing the original intent.

Interviewer: How do you prevent double-charging?

Candidate: Idempotency at every layer. Client → Payment API uses an Idempotency-Key header; the idempotency store caches the response. Payment API → provider uses provider-specific idempotency. Intent state transitions are CAS on current state. So a retried API call returns the same response without re-charging; a retried provider call lands on the same charge id; a retried state transition fails because the state already advanced. No path leads to two ledger entries for the same intent.

Interviewer: Chargebacks?

Candidate: A customer disputes a charge through their bank, which sends a chargeback to the card network. We receive a webhook. The intent transitions from settled to disputed, then to charged_back if we lose, or back to settled if we win. Ledger entries reverse the original plus add the chargeback fee. Dispute handlers gather evidence (delivery confirmation, customer communication) and submit through the provider's dispute API.

Interviewer: Multi-currency?

Candidate: Each ledger entry has a currency. We never mix currencies in one account; instead, FX is an explicit leg. A USD merchant capturing from a EUR customer: debit customer_funding_EUR €100, credit fx_holding_EUR €100, then in a paired transaction: debit fx_holding_USD $108, credit merchant_pending_USD $108. The FX rate is locked at capture time from an FX service. Reconciliation per currency.

Interviewer: How do you do high availability?

Candidate: Stateless API in multiple regions. Database is multi-region with the merchant's home region as primary. Cross-region read replicas for reporting. Provider integrations are inherently external; circuit breakers and retries handle provider blips. Webhooks land in any region and route to the merchant's home region for processing.

Interviewer: PCI?

Candidate: We never store raw card data. The card vault is a separate isolated system with PCI-DSS scope. Customers tokenize their card; we store only the token. Our broader systems are PCI-DSS-out-of-scope or limited scope. Compliance for SCA (PSD2) is handled by triggering the provider's 3DS flow when needed and transitioning the intent to requires_action.

Interviewer: Summarize.

Candidate: Payment intent state machine with explicit transitions; double-entry ledger as source of truth for money; idempotency at every layer; first-class unknown state for provider timeouts resolved by reconciliation; webhook dedup; daily settlement reconciliation. Multi-currency uses per-currency accounts with explicit FX legs. Compliance via card vault and provider-native SCA. The hardest correctness problem is timeouts; the safety net is daily reconciliation.

3. Practice Notes

Five-Minute Version

  1. Clarify: intent, capture, refund, chargeback; idempotency; compliance.
  2. Invariant: one terminal outcome per intent; balanced double-entry ledger.
  3. State machine: created → authorized → captured → settled, with branches for void/refund/dispute.
  4. Idempotency end-to-end: client → API → provider → state CAS.
  5. Double-entry ledger; per-currency accounts.
  6. Unknown state for timeouts; reconciliation worker.
  7. Webhook dedup; CAS state transitions.
  8. Daily reconciliation against provider settlement files.

One-Minute Closeout

A payment system is a state machine over a double-entry ledger. Every money movement is two balanced ledger entries triggered by an explicit state transition. End-to-end idempotency makes retries safe; first-class unknown state for provider timeouts is resolved by reconciliation rather than guessing. Webhooks deduplicate on event id; settlement reconciliation each day catches anything the event pipeline missed. Multi-currency is per-currency accounts with explicit FX legs. The hardest correctness problem is timeouts.
