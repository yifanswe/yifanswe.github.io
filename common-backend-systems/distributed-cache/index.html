Distributed Cache — Redis/Memcached Class (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a distributed in-memory cache from scratch. Clients PUT/GET keys, the cache shards across many nodes, evicts under memory pressure, survives node failure with minimal data loss, and serves at sub-millisecond latency. Target: 10M ops/sec across a 100-node cluster, 1 TB hot working set.

1. Requirements (~5 min)

Functional - GET(key) → value, PUT(key, value, ttl?), DELETE(key). - Optional INCR, EXPIRE, list/hash operations as scope allows. - Cluster auto-partitioning across N nodes. - Eviction policy when memory full.

I'd defer pub/sub, scripting, and persistence-as-DB modes.

Non-functional - p99 latency < 1 ms intra-DC, p99 < 5 ms cross-DC. - 10M ops/sec across the cluster. - Survive single-node failure without availability loss; acceptable data loss for purely-cache use. - Linear scaling: add a node, gain ~1% capacity.

Capacity - 100 nodes × 32 GB RAM = 3.2 TB total; ~1 TB hot working set. - Key/value sizes: small (avg key 50 B, avg value 500 B). ~2B objects in steady state. - With metadata overhead (hash entry, TTL, LRU pointers, allocator slack), a 550 B logical object is closer to 700-900 B in RAM. At 2B objects, the true footprint is ~1.4-1.8 TB before replication.

2. Core Entities (~2 min)

  • Key — opaque bytes, hashed to a partition.
  • Entry — { key, value, expires_at, last_access }.
  • Partition — range / hash bucket assigned to a node.
  • Node — process holding partitions in-memory.
  • Cluster topology — mapping from key → owning node.

3. API (~5 min)

Binary protocol over TCP (like RESP for Redis or memcached binary). Sketch:

GET    key                          → value | nil
SET    key value [EX seconds]       → OK
DEL    key                          → 1 | 0
INCR   key                          → integer
EXPIRE key seconds                  → 1 | 0

Clients hold the cluster topology and route directly to the owning node.

Clients send an optional client_topology_version. If a node receives a request for a vnode it no longer owns, it returns MOVED {node, topology_version} so the client refreshes and retries once.

5. High Level Design (~10–15 min)

  1. Cluster topology — consistent hashing. Key space mapped to a ring. Each node owns a contiguous range. Adding a node steals one range; removing returns one range. Implement with virtual nodes (each physical node owns ~100 vnodes) so load distributes evenly.
  2. Client routing. Clients fetch topology from a coordinator (gossip-based or a small etcd/ZooKeeper cluster), cache it, and route requests directly to the owning node. On MOVED redirect, refresh topology.
  3. Node internals. Each node runs an event-loop process: - In-memory hash table (open-addressing or chained) holding entries. - Eviction policy (LRU, LFU, allkeys-lru). LRU implemented via a doubly-linked list + hash map; LFU via approximate frequency counters. - TTL expiration: lazy on read + periodic scan. - Single-threaded execution per node for simplicity (Redis model) or sharded threads (Memcached threading).
  4. Replication for HA. Each partition has a primary + 1–2 replicas. Writes go to primary; async replicate to replicas. On primary failure, replica promotes via the coordinator.
  5. Cluster coordinator. Tracks node liveness (heartbeats / gossip), promotes replicas, rebalances on add/remove. Lightweight Raft cluster (etcd-style) holds authoritative topology.
  6. Multi-tier serving (optional). Client SDK has an L1 in-process cache (millisecond budget); cache cluster is L2.

End-to-end narration: client wants key user:42. Hashes to ring position 0x4A3, mapped to vnode V37, owned by node N7. Client TCP connects to N7 (already pooled), sends GET user:42 → N7 hash-table lookup → finds entry, checks expiry, updates LRU → returns value. Total ~200 µs.

Replication log vnode:{id}:offset recent writes for replicas value-sized seconds-minutes vnode

GET path:

  1. Client hashes key to vnode using cached topology and sends directly to primary N7.
  2. N7 verifies topology version and ownership, then does one hash-table lookup.
  3. If found, it checks expires_at; expired entries are deleted and returned as nil.
  4. If live, it updates approximate recency/frequency metadata and returns bytes.
  5. On MOVED or timeout, client refreshes topology and retries the new primary; it does not broadcast.

SET path:

  1. Client hashes key to vnode and sends SET key value ttl to the primary with its client_topology_version.
  2. Primary checks ownership, writes the entry table, installs/updates TTL metadata, and appends the mutation to the vnode replication log.
  3. It asynchronously ships the mutation to replicas keyed by vnode:{id}:offset; if the write durability mode is cache-fast, it returns after primary memory write.
  4. For cache-safe, it waits for one replica ack or a 1-2 ms budget, then returns OK or TRY_AGAIN.
  5. On topology change, the primary returns MOVED before mutating so the client retries against the new owner.

Hot vs cold lookup path:

Hot hit:   client L1 -> owner node hash table -> value
Cold miss: client L1 miss -> owner node miss -> caller backing store -> SET refill
Moved:     client -> old owner -> MOVED(version) -> topology refresh -> new owner
Failed:    client -> timeout -> replica/new primary after coordinator promotion

6. Deep Dives (~10 min)

Consistent hashing details. A single key maps to a 32-bit position via SHA-256 (truncated). Each node has ~100 vnodes at random positions. Adding a node: pick 100 random positions, the keys that fall in those positions migrate from their old owners. On removal, keys redistribute among remaining nodes.

Capacity math per node: 32 GB RAM - 20% reserved for allocator fragmentation, replication buffers, and OS leaves ~25 GB usable. At ~800 B physical entry size, that is ~31M entries/node. Across 100 nodes, usable primary capacity is ~3.1B entries; with one replica it is ~1.55B logical entries. The dominant cost is RAM overhead per entry, not raw value bytes, so compact metadata and sampling-based eviction matter.

Replication and metadata math: with one async replica, the 1 TB hot working set becomes 2 TB resident. Topology metadata is tiny: 10k vnodes x 3 node IDs x ~8 B is under 1 MB plus versioning. Replication logs are bounded by lag, not retention; at 10M ops/sec x 700 B average physical mutation, even a 10-second lag is ~70 GB cluster-wide, so each node needs backpressure before logs eat the memory reserved for cache values.

Why not just Redis Cluster? In production I would use Redis/Memcached unless the interview asks from scratch. The design here borrows Redis's single-threaded event loop for predictable latency and Memcached's client-side sharding for scale. etcd/ZooKeeper/Raft is for tiny topology metadata; putting every cache write through Raft would destroy throughput.

Be ready for "why not DynamoDB/Cassandra?" They are durable key-value stores with millisecond latencies and disk-backed replication; this system optimizes for sub-millisecond in-memory reads and accepts cache loss. Be ready for "why not pure client-side consistent hashing without a coordinator?" It is fast, but replica promotion and migration need one authoritative topology version or clients disagree during failure.

Migration during scaling. When a vnode moves N7 → N12, both nodes briefly own it. Migration protocol: 1. Coordinator marks vnode in migration N7→N12. 2. New writes go to N12. 3. N7 streams existing keys to N12. 4. On done, coordinator updates topology; clients see the change on next refresh. 5. Old N7 deletes the range.

Migration state machine:

Stage Owner behavior Client behavior Risk
stable N7 primary route to N7 normal
migrating writes dual-write or forward to N12; N7 streams snapshot old topology still works temporary extra latency
cutover coordinator publishes new version clients get MOVED and refresh short retry spike
cleanup N7 deletes vnode after grace period route only to N12 stale clients rejected

Serving degradation state:

Stage Mode Source Latency / correctness
hot L1 + primary hit client cache or owner hash table sub-ms, may be TTL-stale
warm primary miss caller backing store then SET refill caller pays backing-store latency
replica promotion new primary async replica state through last offset sub-second failover, latest writes may be lost
topology stale redirect MOVED with new version one retry
memory pressure eviction sampled LRU/LFU correctness preserved as miss

Eviction policies. - LRU: doubly linked list of entries by recency; O(1) update. Memory cost: 2 pointers per entry. - LFU: approximate counters (logarithmic counter à la Redis); resistant to scan attacks. - Random sampling LRU (Redis approximation): sample N entries, evict oldest. Lower memory cost, near-LRU quality.

TTL expiration. Two strategies, used together: - Lazy expiry: on each GET, check expires_at; if expired, delete and return nil. - Active expiry: a background task samples random keys with TTLs, deletes expired ones. Bounds memory growth from never-read expired keys.

Hot keys. A single key (feature_flag_X) drawing millions of QPS overwhelms its owning node. Mitigations: - Client-side caching with TTL: most recent value held for 100ms; absorbs hot reads. - Replicate hot keys to multiple nodes; clients spread reads. - Auto-detect hot keys server-side, broadcast to clients to enable replication.

Hot-key fallback is explicit: normal mode serves from one primary; detected-hot mode replicates read-only copies to hot:{key}:{replica_id} on several nodes; stale-safe mode lets clients use L1 cache for 100-500 ms; write-heavy hot keys like counters are rejected or redirected to a different service because cache nodes are not a general counter database.

The two-view pattern shows up in topology and data: topology has one strongly consistent view in the coordinator, while cached values are deliberately weak and per-vnode. I would never require data-plane consensus for every value; only the map of who owns the value needs consensus.

Persistence. Pure cache: no persistence; data loss on full restart accepted. Cache-with-snapshot: periodic RDB-style snapshot + AOF log of writes; replay on restart. Tradeoff is between memory write throughput and recovery time.

Cross-region. Each region runs its own cluster (cache locality matters for latency). Cross-region replication is opt-in per-key — most caches don't need it. For sessions that must be globally visible, use a different layer (replicated K-V store).

Concurrency model. Single-threaded per node (Redis-style) gives simple correctness and zero locking overhead. Multi-core utilization via running multiple processes per machine. Alternative: per-thread shards within a process; locks confined to per-shard structures.

Failure semantics. A node dying loses its in-memory contents. With replication, the replica promotes and serves; without, those keys become cold cache misses (acceptable for a cache). Clients receive errors during failover (sub-second), then succeed.

Failure modes table:

Case Behavior Latency budget
expired key lazy delete on read, return nil <1 ms
evicted key miss, caller falls through to backing store caller-owned
primary failed with replica coordinator promotes replica, clients refresh <1 s failover
primary failed without replica keys cold-miss until repopulated availability preserved
coordinator unavailable clients keep old topology; no rebalance existing traffic continues

Backpressure. Per-connection request limits; reject when CPU saturated. Out-of-memory triggers eviction; if eviction can't keep up, return errors rather than corrupt data.

What I'd defer. Multi-master writes, pub/sub, Lua scripting, secondary indexes, geospatial commands, cluster-wide transactions.

Interview punch line: a cache is allowed to lose values, but it is not allowed to become a distributed database by accident. Keep topology metadata strongly coordinated, keep data-plane reads direct and fast, and define miss/failover behavior clearly.
