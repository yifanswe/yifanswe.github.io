1. Basic Knowledge And Solution

What This Question Is Testing

A distributed lock manager looks simple — "give me the lock for key X" — but the system that backs it is one of the few pieces of infrastructure that must be both highly available and strictly consistent. The hard part is correctness when clients crash holding locks and when the network partitions the lock manager from those clients.

The invariant is this:

At any moment, for any lock identifier, at most one client believes it holds the lock in an action-permitting state, and the lock manager and downstream services agree on that belief through fencing tokens, not through wall-clock timeouts alone.

At ICT5 / L6+ level, the interviewer is testing whether you can build mutual exclusion on top of a replicated state machine, defend against the "client paused for 30 seconds" failure that breaks naive lock managers, and explain why Chubby chose coarse-grained locks plus session leases rather than fine-grained mutexes.

60-Second Interview Framing

Spoken version:

I would build a Chubby-like service as a small cluster of replicas running Paxos or Raft. Clients open sessions with a TTL lease; locks are reified as files in a tiny hierarchical namespace; every lock acquisition returns a monotonically increasing fence token. Downstream services must check the token. The interesting decisions are session expiry handling, fencing tokens versus pure TTLs, and why this is built for infrequent coarse-grained locks, not microsecond mutexes.

Requirements To Clarify First

Functional requirements:

  • Clients acquire and release named locks with optional exclusive or shared mode.
  • Clients maintain a session with the lock service via heartbeats.
  • A tiny file system supports small configuration files (Chubby's original use case).
  • Clients receive notifications on file or lock state changes.
  • The service supports leader election as a primary use case.

Non-functional requirements:

  • Strong consistency: a granted lock is not also granted to anyone else.
  • High availability: tolerates minority replica failures.
  • Survive client crashes without leaking locks.
  • Survive network partitions without dangerous double grants.
  • Latency in tens of milliseconds for acquisition; cheap heartbeats.
  • Auditability for every grant, release, and session expiry.

Out of scope unless interviewer asks:

  • High-throughput fine-grained per-row locking (use database transactions instead).
  • Application-level lock semantics like reentrancy beyond simple counter-style support.
  • Cross-cluster lock federation.

The bottleneck is not throughput; it is the Raft/Paxos commit latency and the heartbeat fan-in from many clients.

Acquire response includes the fence token (sometimes called sequencer): a monotonically increasing integer that downstream services check before honoring writes attributed to that lock holder.

Session Lease And The Jeopardy State

Sessions are leases with a TTL (e.g., 12 seconds typical Chubby default).

Lifecycle:

  1. Client opens session; server grants a lease until T.
  2. Client periodically sends KeepAlive. Server extends lease.
  3. If a KeepAlive does not arrive before T, the server marks the session expired.
  4. On expiry, all locks held by the session are released, all watches removed, all ephemeral nodes deleted.
  5. The client may be unable to reach the server but is still up. It enters jeopardy state and pauses application activity until it confirms it still holds the session.

Critical detail: the client enforces a local conservative deadline that is earlier than the server's. If the client cannot reach the server before its conservative deadline, it stops using the lock (rather than waiting until it might have expired server-side). This asymmetry prevents the "client thinks it holds the lock, server already gave it away" window.

Interview phrase:

Lease timeouts alone cannot make mutual exclusion safe across the network. The client must apply a conservative bound earlier than the server's, and downstream services must check a fence token rather than trusting the client's claim.

Fencing Tokens — The Crucial Mechanism

Without fencing, this scenario breaks mutual exclusion:

  1. Client A acquires lock; server grants until T=10.
  2. Client A pauses (GC, network) until T=12.
  3. Server expires A's session at T=10; grants lock to Client B at T=11.
  4. A resumes at T=12 unaware, sends write to storage.
  5. B also sends a write.
  6. Storage serves both — corruption.

Fencing fix:

  • Each grant increments a per-lock monotonic counter and returns it.
  • A gets token 7; B gets token 8.
  • Storage tracks the highest token it has seen for the resource; rejects writes with lower tokens.
  • A's stale write fails downstream.

This requires downstream cooperation. Lock services that don't fence — or services that don't check the token — cannot safely use lock-protected critical sections across the network.

Consensus Layer

All write operations go through Paxos or Raft:

  • Five replicas; one elected leader processes client writes.
  • Leader appends to local log, replicates to majority, applies to state machine, replies to client.
  • Reads default to the leader (linearizable); optional stale reads at replicas.
  • On leader failure, a new leader is elected; uncommitted entries from the previous term either commit or are overwritten according to the protocol.

Heartbeats and session keepalives also go through the leader so the state machine reflects current session liveness.

Read Caching In The Client Library

Most operations are reads. To keep load down:

  • Client library caches file contents, indexed by path, with the server-provided version.
  • The server tracks which clients hold which cached entries.
  • When data changes, the server invalidates the relevant clients before completing the write.
  • This effectively makes reads strongly consistent from the application's view, while offloading work to clients.

This is the Chubby caching design. It is one of the major reasons Chubby can serve thousands of clients with five replicas.

Watches And Notifications

  • Clients register watches on file paths.
  • After a change commits, the dispatcher pushes a one-shot notification.
  • The client re-reads to learn the new state.
  • Notifications carry no payload; the read is the truth, and avoids race conditions where the watch and the read disagree.

Watches are how leader election clients learn that the previous leader's lock has been released.

Leader Election Pattern

The canonical use case:

1. Each candidate opens session.
2. Each tries to acquire lock at /service/election.
3. The winner sets the file contents to their address; remaining candidates set watches.
4. Winner periodically heartbeats its session.
5. If winner's session expires, the lock is released; watches fire.
6. A new election begins.
7. Every successful acquire returns a higher fence token than the previous leader's.

Downstream services check the fence token on every write they receive; a delayed write from the previous leader fails.

Scaling Strategy

Chubby is intentionally not high-throughput. To scale across many tenants:

  • Run many independent Chubby cells per data center, sharded by application.
  • Each cell handles a few thousand clients.
  • Latency-sensitive applications use cells in their own region; cross-cell coordination is avoided.
  • Heartbeats are aggregated and batched on the client side when many sessions originate from one process.

If throughput becomes an issue, the typical answer is "you are using it wrong" — Chubby is for coarse coordination, not for serializing writes.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Client paused with lock held; lease expires. Fencing tokens checked downstream; client conservative deadline. Stale fence-token rejections at storage; jeopardy session count.
Leader replica fails. Raft election picks new leader; outage of a few seconds. Leader-change rate, election duration.
Network partition isolates minority replicas. Minority cannot commit; clients connected to it lose ability to acquire. Quorum size, partition detection.
Network partition isolates a client from the cell. Client enters jeopardy, then stops using the lock; server expires session. Jeopardy duration distribution.
Heartbeat storm. Client batching; conservative TTL; server rate limit. Heartbeat QPS, response latency.
Watch storm on popular file. Fan-out throttling; client-side debounce. Watch fire rate, dispatcher queue depth.
Replica clock skew. Use server-only clocks for lease decisions; do not rely on client time. Cross-replica clock drift.
Storage of replicated log fills. Snapshot + log compaction. Log size, snapshot age.

Operational Metrics

Correctness:

  • Fence token violations at downstream (must be zero or auditable).
  • Sessions expired with locks held — duration and count.
  • Double-grant alerts.

Capacity:

  • Consensus commit latency p99.
  • Active session count and heartbeat rate.
  • Active watch count.
  • Cache hit ratio at client.

Service health:

  • Leader stability; election frequency.
  • Quorum availability.
  • Log size and snapshot age.

Tradeoffs To Call Out

  • Strong consistency requires consensus, capping write throughput in the hundreds per second per cell.
  • Coarse-grained locks reduce frequency but require applications to design around longer critical sections.
  • TTL leases are conservative; setting them too short causes spurious failovers, too long delays recovery.
  • Fencing tokens require downstream cooperation; applications that ignore them cannot be safe.
  • Client-side caching offloads load but requires server-tracked invalidations.
  • Five replicas tolerate two failures; more replicas increase commit latency for limited extra safety.

Common Traps And Stronger Moves

Trap Stronger move
Treating it as a high-throughput mutex service. Frame it as coarse-grained coordination; locks per role, not per operation.
Trusting lease TTL alone. Fence tokens checked downstream.
Symmetric client/server expiry deadlines. Client uses an earlier conservative deadline.
Forgetting cache invalidation in the design. Server tracks cache holders and invalidates before commit.
Using Chubby as a config DB at scale. Small files only; for bigger config, point to external storage.
Cross-cell global locks. Avoid; partition the lock space by application/region.

Interview Phrase

A distributed lock manager is correctness-first infrastructure. The lock is not the safe primitive; the lock plus a fencing token is. Sessions, leases, and the jeopardy protocol prevent leaked locks, but the safety net is the monotonic token enforced by every service that honors the lock.

2. Simulated Interview

Interviewer: Design a distributed lock manager.

Candidate: I would build a Chubby-style cell of five replicas running Raft, exposing a tiny hierarchical namespace where files double as locks. Each client opens a session with a lease; acquiring a lock returns a monotonically increasing fence token. The invariant is at-most-one-holder, and the downstream services must check the fence token before honoring writes.

Interviewer: Why do I need fencing tokens? Aren't lease TTLs enough?

Candidate: No. Suppose client A holds the lock, then pauses for 20 seconds (GC, network blip). Its lease expires; the server grants the lock to client B. Then A resumes, still thinks it has the lock, and writes to storage. Without fencing, storage cannot tell A's write from B's. With fencing, A's token is 7, B's is 8; storage tracks the max token it has accepted and rejects A's stale write. Lease TTLs alone leave a window of unsafety; fencing closes it.

Interviewer: How big should the lease TTL be?

Candidate: A balance. Too short and minor pauses cause spurious failovers and re-elections. Too long and recovery from a real failure is slow. Chubby's default is around 12 seconds. The client uses a more conservative local deadline — maybe 8 seconds — so it stops using the lock before the server might revoke it. That asymmetry is critical.

Interviewer: Walk me through leader election.

Candidate: Each candidate opens a session and attempts Acquire on the election file. One wins, gets a fence token, writes its address into the file. Losers register watches. The winner heartbeats periodically. If the winner's session expires, the lock is released; watches fire on losers; the next acquire returns a higher token. Any writes the deposed leader sends downstream after expiry are rejected by token comparison.

Interviewer: How does the consensus layer work?

Candidate: Raft. Five replicas, one leader, majority quorum required to commit. Every write — acquire, release, file edit, session keepalive — is appended to the log, replicated to a majority, then applied to the deterministic state machine. Reads can go through the leader for linearizability or through replicas with explicit staleness.

Interviewer: Reads through the leader sound expensive.

Candidate: That is why the client library caches. The server tracks which clients hold which cached entries. When a write commits, the server sends invalidations to all cache holders before the write completes. The next read from a holder is forced to refetch. This gives applications linearizable reads with most of the work offloaded to the client.

Interviewer: Why is this not a high-QPS service?

Candidate: Every write goes through consensus, which takes tens of milliseconds in the best case. That bounds you to hundreds of writes per second per cell. Chubby was designed for coarse-grained coordination — a service grabs the lock for its entire master tenure, not per-request. If you need fine-grained locks, you use a database transaction or a Redis-style optimistic system instead.

Interviewer: A client's network breaks but the client process is up. What happens?

Candidate: The client enters jeopardy: it failed to refresh its lease. It pauses application activity using the lock, expecting to recover before the lease expires. It keeps trying to reach any replica. If it reconnects within the lease window, it confirms the session is still alive and resumes. If the conservative client-side deadline passes without confirmation, it gives up the lock locally even if technically the server has not expired the session. This way, even if the network heals and the server still thinks the client has the lock, the client refuses to use it.

Interviewer: What if the consensus minority is partitioned?

Candidate: The minority cannot commit writes. Clients connected to minority replicas see acquire failures. The majority continues serving writes through its leader. When the partition heals, the minority replicas replay log entries and rejoin. Importantly, the minority replicas never granted a conflicting lock during the partition because they did not have quorum.

Interviewer: How do you handle watch storms?

Candidate: Watches are one-shot and payload-free; after firing, the client must re-register if it wants further notifications. The fan-out is bounded by the dispatcher's rate limit. For popular files, we either reduce notification frequency or push watchers to poll. The application reads after a notification to learn the truth, which avoids races where the watch fires but the cached value is stale.

Interviewer: Multi-region or just one cell?

Candidate: Multiple cells, partitioned by application and region. One cell per application per data center is typical. Global coordination is a non-goal; if two services need to coordinate across regions, they do so above this layer (and accept the latency). Trying to make one cell global blows up commit latency from milliseconds to hundreds of milliseconds.

Interviewer: Summarize.

Candidate: A small replicated state machine — five replicas, Raft — exposes a tiny file system where files act as locks. Sessions with leases plus a jeopardy protocol prevent leaked locks; fencing tokens close the GC-pause hole. Client-side caches with server-driven invalidations make reads cheap. The system is intentionally coarse-grained; we scale by adding cells, not by raising throughput per cell.

3. Practice Notes

Five-Minute Version

  1. Clarify: coarse-grained, leader election main use, mutual exclusion.
  2. State invariant: at-most-one-holder + fence token correctness.
  3. Build on consensus (Raft or Paxos); five replicas.
  4. Show sessions, leases, jeopardy.
  5. Show fencing tokens and the GC-pause scenario.
  6. Explain client-side caching with invalidations.
  7. Explain watches; one-shot, payload-free.
  8. Close with multi-cell scaling.

One-Minute Closeout

A distributed lock manager is correctness-first infrastructure. Locks live in a replicated state machine; sessions with TTL leases prevent permanent leaks; fencing tokens prevent stale clients from corrupting downstream state. Client-side caches with server-driven invalidations carry the read load. The system is coarse-grained by design; scale comes from running many cells, not from raising throughput per cell.
