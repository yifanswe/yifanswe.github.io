1. Basic Knowledge And Solution

What This Question Is Testing

Gmail is email at billion-user scale. The hard part is SMTP interop with the rest of the internet, spam at petabyte scale, message threading, full-text search across decades of mail, durable per-user inbox storage, and attachment handling.

The invariant is this:

Every email accepted by SMTP intake (or sent by the user) is durably stored in the recipient's mailbox (or rejected with a clear reason), spam-classified before delivery, indexed for search, and accessible from every user device with consistent threading; sent mail is reliably delivered to external recipients or bounced with a clear status.

At ICT5 / L6+ level, the interviewer is testing whether you can design SMTP ingest, anti-spam, per-user mailbox storage with billions of users, full-text search at scale, and reason about threading, label-based organization, and attachment storage.

60-Second Interview Framing

Spoken version:

I would design Gmail around four core systems: SMTP intake with anti-spam, per-user mailbox storage with threading and labels, full-text search index, and SMTP outbound for sending. The interesting decisions are anti-spam architecture, per-user storage scale, search index strategy, and attachment handling. Most of the engineering complexity is anti-spam and search at this scale.

Requirements To Clarify First

Functional requirements:

  • Receive email via SMTP from internet.
  • Send email via SMTP to internet.
  • Per-user inbox with labels, folders, threads.
  • Search across all email.
  • Attachments.
  • Spam, phishing, malware filtering.
  • Auto-replies, vacation responders.
  • Multi-device sync (IMAP, web, mobile).

Non-functional requirements:

  • Billions of users.
  • Hundreds of billions of emails/day delivered.
  • Per-user storage in GBs to TBs.
  • High availability.
  • Strict spam filtering with low false positive rate.
  • Search across years of email under a second.
  • Multi-region.

Out of scope unless asked:

  • Real-time chat / Hangouts.
  • Calendar integration (separate).
  • Drive integration for attachments (touch on).
  • Enterprise (Workspace) features.

Bottleneck: anti-spam compute and per-user storage scale.

SMTP Intake

  1. External MX (mail.gmail.com) receives SMTP connection.
  2. Greylisting / rate limit for unknown senders.
  3. SPF check: is the sending IP authorized for the From domain?
  4. DKIM verification: is the signature valid?
  5. DMARC: does it align?
  6. Accept or temp-fail / reject.
  7. Pass to anti-spam pipeline.

If rejected at SMTP time (5xx response), sender's MTA bounces. If accepted (2xx) but later filtered, message lands in Spam folder (silently) or is dropped per policy.

Anti-Spam Pipeline

Multi-stage:

  1. Reputation: sender domain and IP reputation. Known spammers blocked.
  2. Content rules: signature matching, regex, URL block lists.
  3. ML classifier: trained on labeled spam; outputs spam_score.
  4. User signals: is recipient in addressbook / has prior conversation?
  5. Final decision: clean → inbox, suspicious → spam folder, malware → quarantine.

ML classifier features: - Sender behavior patterns. - Content features (text, URLs, attachments). - Recipient engagement patterns. - Cluster analysis across recipients (mass campaign detection).

Spam is at least 50% of inbound; Gmail's filtering reaches <0.1% spam in inbox.

Threading

Group emails into conversations using RFC 5322 References and In-Reply-To headers:

  • Each email has a Message-ID.
  • A reply includes the parent's Message-ID in In-Reply-To and prior thread in References.
  • Threading engine groups messages by transitive reference closure.
  • Subject-based threading as fallback (Re: original).

Per user, a thread is a Thread record with message_ids.

Labels And Filters

Gmail uses labels instead of folders. A message can have multiple labels. System labels: Inbox, Sent, Drafts, Spam, Trash. User labels: anything.

Filters: user-defined rules to auto-label, archive, delete based on sender/subject/content.

Smart labels: Gmail's ML auto-categorizes into Inbox, Social, Promotions, Updates, Forums.

Mailbox Storage

Per-user; sharded by user_id.

  • Each user's mailbox is a sequence of message records plus thread records plus label assignments.
  • Bigtable-style row store works well: row key = user_id:thread_id:message_id.
  • Attachments stored separately (content-addressed); referenced by ID.
  • Storage grows over time; cold messages tier to cheaper storage.

For dedup: attachment content-addressed. If many users receive the same file (e.g., a forwarded image), one blob with multiple references.

Search

Per-user inverted index:

  • Tokenize subject, body, sender, recipient, labels, attachments.
  • Index by user.
  • Real-time index for new mail; batch for catch-up.

Query: tokens + filters + range, return ranked messages. Sub-second per-user.

For 1.5B users with TB of mail each, search index is massive — but it's per-user, so sharded heavily.

Outbound SMTP

User sends an email:

  1. Compose validated; recipient list checked.
  2. DKIM signing (Gmail signs as authoritative for the user's domain).
  3. SPF/DMARC alignment.
  4. Submit to outbound SMTP queue.
  5. Worker connects to recipient's MX (looked up via DNS MX records).
  6. Deliver via SMTP.
  7. On 2xx ack: success.
  8. On 4xx temp fail: retry with exponential backoff.
  9. On 5xx perm fail: bounce back to sender.
  10. After retry timeout: bounce.

For Gmail-to-Gmail: bypass external SMTP, deliver internally for low latency.

Bounce Handling

Bounces (NDRs - Non-Delivery Reports) come back as DSN messages. Parsed and presented in sender's Sent folder.

Consistency Model

Multi-Region Strategy

  • Per-user home region for storage.
  • MX servers globally distributed.
  • Inbound mail delivered to user's region.
  • Sending from user's region.
  • Failover region warm.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Spam classified as ham. User reports; classifier retraining. False negative rate.
Ham classified as spam. User restores; classifier retraining; preview window. False positive rate.
Phishing slips through. Multi-stage detection; safety warnings; user education. Reported phishing rate.
Malware attachment. Sandboxed scan; quarantine. Malware detection rate.
DDoS on MX. Rate limit by IP; reputation. Per-IP connection count.
Per-user mailbox hot (e.g., a user with millions of messages). Sub-partition; index pruning. Per-user QPS.
Outbound delivery temp-fail. Retry queue with backoff. Retry queue depth.
Search index lag. Async; alert on lag. Index lag.
Bounce loop. Detect and stop forwarding. Bounce frequency.
Cross-region failover. Warm replica; bounded lag. Failover RTO.

Operational Metrics

Correctness:

  • Spam reaching inbox rate.
  • Ham filtered to spam rate.
  • Lost email rate (must be near zero).
  • Bounce rate.

Capacity:

  • SMTP intake rate.
  • Storage growth.
  • Search QPS / latency.

Business:

  • Active users.
  • Mail volume per user.
  • Engagement (open, reply).

Tradeoffs To Call Out

  • Aggressive spam filter reduces inbox noise but increases false positives.
  • Storing every email forever costs storage; tier or expire per policy.
  • Per-user search index is fast but storage-heavy.
  • Real-time threading vs batch (cached) trades off latency vs CPU.
  • Attachment dedup saves storage but adds metadata.
  • DKIM/SPF/DMARC enforcement protects users but blocks legitimate senders with bad config.

Common Traps And Stronger Moves

Trap Stronger move
Treat email as messaging (Discord-like). SMTP is the protocol; design around interop.
Single global search index. Per-user shard.
Synchronous spam check. Multi-stage with fast-path.
Folders not labels. Labels (multi-membership).
One ML model. Multi-stage classifiers + reputation.
In-place attachment storage. Content-addressed separate store.

2. Simulated Interview

Interviewer: Design Gmail.

Candidate: Four core systems: SMTP intake with anti-spoof, multi-stage anti-spam pipeline, per-user mailbox storage with threading and labels, full-text search index, and outbound SMTP for sending. The invariant is durable per-user storage of accepted mail, spam-filtered before user-visible, searchable, and reliably delivered to external recipients.

Interviewer: Walk me through receiving an email.

Candidate: External MX server accepts SMTP connection. Performs SPF check (is sending IP authorized for From domain?), DKIM verification (signature valid?), DMARC alignment. If checks pass and sender reputation is acceptable, accept the message. Pass to anti-spam pipeline. Multi-stage filtering — reputation, content rules, ML classifier, user signals — produces a spam score. Below threshold: deliver to user's inbox. Above threshold: spam folder. Malware: quarantine. Mailbox storage writes the message; threading engine attaches it to a conversation; search indexer indexes it.

Interviewer: Anti-spam at scale.

Candidate: Multi-stage. (1) Reputation: sender IP and domain reputation, known-spammer block lists, regional spam patterns. (2) Content rules: signature matches, regex, URL block lists. (3) ML classifier: deep model trained on labeled spam, features include text, URLs, sender behavior, attachment patterns. (4) User signals: is the sender in the recipient's addressbook? Have they corresponded before? (5) Cluster detection: mass campaigns identified across recipients. Multi-stage because no single check is perfect; combination gives high precision and recall.

Interviewer: Threading?

Candidate: RFC 5322 References and In-Reply-To headers. A reply includes the parent message's Message-ID. Threading engine takes the transitive closure of these references to group messages. Subject normalization (strip Re:, Fwd:) as fallback. Per-user thread records capture the group; users see a conversation rather than individual messages.

Interviewer: Storage scale?

Candidate: Per-user, sharded by user_id. Bigtable-style row store: row key user_id:thread_id:message_id makes per-user reads efficient. Each message record contains headers, body, labels, attachment refs. Attachments stored separately as content-addressed blobs with dedup — many users may receive the same file, stored once with reference counts. Old mail tiers to cheaper storage with longer access latency.

Interviewer: Search across years of email?

Candidate: Per-user inverted index. Tokens from subject, body, sender, recipient, labels, attachment names. Real-time indexing for new mail. Sharded per user — your search only touches your shard. Query: tokens + filters (label, date range, sender), score by relevance + recency, return paginated.

Interviewer: A user with 100 GB of mail searches "vacation."

Candidate: Their per-user search index is large but still bounded. The token "vacation" maps to a posting list of message ids. Filter by other criteria. Score. Return top N with snippet. Sub-second because the index is per-user and well-tuned.

Interviewer: Sending an email.

Candidate: User composes; client submits. Validate recipients. DKIM signing — Gmail signs the message as authoritative for the user's domain. Submit to outbound SMTP queue. Worker resolves the recipient's MX via DNS, opens SMTP connection, delivers. On 2xx: success. On 4xx temp-fail: retry queue with exponential backoff (minutes to hours, capped at days). On 5xx perm-fail or final retry exhaustion: bounce NDR back to sender.

Interviewer: Gmail to Gmail?

Candidate: Bypass external SMTP. The receiving side is the same system; deliver internally with lower latency. Same anti-spam pipeline still applies (intra-Gmail mail can still be spam — compromised account, bot account, etc.).

Interviewer: Attachment dedup?

Candidate: Content-addressed: blob_id = SHA-256 of content. Reference count. Multiple users referencing the same attachment increment count. When all references drop, GC. Saves significant storage for forwarded files. Caveat: cross-user dedup leaks "do you have this file" signal — usually fine for benign attachments, but special care for sensitive ones.

Interviewer: Phishing — a sophisticated email looks legitimate.

Candidate: Multi-signal detection: domain similarity (lookalike domains), unusual sender for this recipient, suspicious link patterns, prior phishing campaigns, ML model trained on phishing. Soft warnings in UI ("This email looks like phishing"). Strict cases go to spam or quarantine. User reports feed retraining.

Interviewer: Multi-region?

Candidate: Per-user home region. MX servers globally distributed; inbound mail routed to user's region. Outbound originates from user's region. Failover region warm with bounded replication lag. Search index regional with the mail.

Interviewer: A user reports a legitimate email as spam.

Candidate: User reports feed into the classifier. The specific message moves to spam folder. Similar messages from same sender may be retroactively re-classified for that user. Global classifier weights update over time based on aggregated reports.

Interviewer: Summarize.

Candidate: SMTP intake with anti-spoof checks; multi-stage anti-spam pipeline; per-user mailbox storage with threading and labels; per-user search index; content-addressed attachment storage with dedup; outbound SMTP with retry and bounce. Multi-region with per-user home. The dominant engineering challenge is anti-spam quality; the dominant scale challenge is per-user storage and search index volume.

  3. MX → anti-spam → mailbox → search index.
  4. Multi-stage anti-spam pipeline.
  5. Per-user sharded storage (Bigtable-style).
  6. Per-user inverted index for search.
  7. Attachment content-addressed; dedup.
  8. Outbound SMTP with retry + bounce.

Gmail is per-user mailbox storage plus SMTP interop plus aggressive anti-spam.
