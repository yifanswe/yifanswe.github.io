1. Basic Knowledge And Solution

What This Question Is Testing

AWS Lambda is serverless compute: deploy a function, get an endpoint, pay per invocation. The hard part is cold-start latency, secure multi-tenant isolation on shared hardware, fast scaling from zero to thousands of concurrent invocations, and reliable event-source integration.

The invariant is this:

Every invocation request matched by a deployed function executes in an isolated environment with the configured code and runtime, completes within the function's timeout, and returns a result or error to the caller; concurrent invocations scale independently; resource limits (memory, CPU, network) are enforced; secrets and IAM identity are correct per invocation.

At ICT5 / L6+ level, the interviewer is testing whether you can design micro-VM isolation (Firecracker), cold-start optimization, autoscaling from zero to N, event-source mapping (S3, SQS, Kinesis triggers), and reason about resource scheduling on shared fleet.

60-Second Interview Framing

Spoken version:

I would design Lambda as four core systems: a function registry holding code and config, an isolated execution layer with micro-VMs (Firecracker), a request router that scales invocations across a worker fleet, and event sources that trigger functions. The interesting decisions are cold-start optimization, isolation model, fleet scheduling, and integration with synchronous (API Gateway) and asynchronous (S3, SQS, EventBridge) triggers.

Requirements To Clarify First

Functional requirements:

  • Deploy a function: code, runtime, memory, timeout, env vars.
  • Invoke synchronously (API call) or asynchronously (events).
  • Auto-scale: zero idle cost; thousands concurrent.
  • Logging and metrics per invocation.
  • IAM identity per function.
  • VPC connectivity.
  • Multi-runtime support (Node, Python, Java, Go, custom).
  • Layers and dependencies.

Non-functional requirements:

  • Cold start: under a second; warm under 50 ms.
  • Concurrent: thousands per function.
  • Isolation: strong (different tenants share hardware).
  • Pay per millisecond billed.
  • High availability per region.
  • Limit enforcement.

Out of scope unless asked:

  • SnapStart pre-warming details.
  • Container image deployment (touch on).
  • Step Functions / orchestration.
  • Lambda@Edge.

Function Deployment

  1. Developer packages code (zip or container image).
  2. Uploads to code store.
  3. Creates/updates function in registry.
  4. Function metadata published.
  5. Optional: pre-warm via SnapStart or provisioned concurrency.

Invocation Path

Synchronous invoke (caller waits for result):

  1. Caller invokes via API or SDK.
  2. Invocation API authenticates, validates payload size.
  3. Routes to scheduler.
  4. Scheduler decides: - Warm: existing micro-VM for this function available → route invocation to it. - Cold: no warm env → pick worker, provision micro-VM, fetch code, init runtime, then invoke.
  5. Function executes within micro-VM.
  6. Result returned to caller via Invocation API.
  7. Execution env retained for reuse (warm) or terminated based on idle policy.

Async invoke (event-driven):

  1. Caller invokes; request queued.
  2. Async invocation queue.
  3. Worker picks from queue; same execution path.
  4. On failure: retries per policy; dead-letter on exhaustion.

Cold Start Optimization

Cold start = time to provision micro-VM, fetch code, initialize runtime, run init code.

Components: - Micro-VM provisioning: ~125ms (Firecracker). - Code download: depends on package size. - Runtime init: language-dependent (Java slowest). - User init code: depends on function.

Optimizations: - Worker pre-warming: keep pool of pre-booted micro-VMs ready to load any code. - Code caching: workers cache recently-used function code. - Snapshots (SnapStart): persist warm state of init; restore from snapshot rather than re-init. - Provisioned concurrency: customer pays to keep N environments warm. - Container image layer caching: for image-based functions.

Interview phrase:

Cold start is the central performance challenge. The micro-VM itself boots in ~125ms; the rest is code fetch and init. SnapStart restores from a snapshot of post-init state, shrinking the variable portion.

Isolation: Firecracker Micro-VMs

Multi-tenant on shared hardware requires strong isolation.

  • Linux containers alone are insufficient — kernel boundary too permeable for cross-tenant.
  • KVM-based micro-VMs give hardware-level isolation.
  • Firecracker is a slim VMM (Virtual Machine Monitor) purpose-built for serverless: ~125 MB memory overhead per VM, ~125ms boot.
  • Each invocation env is a separate VM with kernel, isolated network and filesystem.
  • Workers can host hundreds of micro-VMs.

Worker Fleet Scheduling

Workers are general-purpose hosts running Firecracker.

Scheduler decisions per invocation: - Find existing warm env for this function? - If yes, route there. - If no, pick a worker with capacity to provision new env. - Capacity = available memory, CPU, network.

Bin-packing: workers should be utilized efficiently without overcommitting.

Function isolation: - One worker can host many functions from many tenants. - Each tenant's micro-VMs are isolated. - No shared memory across tenants.

Event Source Mappings

For event triggers:

  • S3: S3 service invokes Lambda on object create/delete events.
  • SQS: Lambda polls SQS, batches messages, invokes function. Acks on success.
  • Kinesis: Lambda polls shards in order, processes batches.
  • DynamoDB Streams: similar to Kinesis.
  • EventBridge: rule-based event routing to Lambda.
  • API Gateway: synchronous HTTP-to-Lambda integration.

Event source mappings translate source events into invocations with proper payload format, retries, batching, and ordering (where applicable).

Concurrency Limits

Per-function: - Reserved concurrency: dedicated quota (no scaling beyond, but guaranteed). - Unreserved: shares regional pool. - Burst limits: scale up to N per region per minute.

Per-account: regional concurrency limit (default 1000, raisable).

Tradeoffs To Call Out

  • Firecracker overhead vs container speed: better isolation, slower start.
  • Pre-warming costs idle compute vs cold-start latency.
  • Strict concurrency limits prevent runaway costs but block bursts.
  • SnapStart saves cold start but limits flexibility.
  • Per-function reserved concurrency guarantees but partitions capacity.
  • More aggressive worker bin-packing improves utilization but risks noisy neighbors.

2. Simulated Interview

Interviewer: Design AWS Lambda.

Candidate: Four core systems: function registry holding code and config, invocation API receiving requests, scheduler placing invocations on workers, worker fleet hosting Firecracker micro-VMs. The invariant is every matched invocation runs in an isolated environment with the configured code, within resource limits, returning result or error. The performance challenge is cold start; the safety challenge is tenant isolation.

Interviewer: Walk me through an invocation.

Candidate: Caller invokes function by name. Invocation API authenticates, validates payload, routes to scheduler. Scheduler checks if a warm execution environment exists for this function and version. If yes, routes the invocation there — sub-50ms latency. If no, cold start: pick a worker with capacity, provision a Firecracker micro-VM (~125ms), fetch function code from code store, initialize runtime, run user init code, then invoke. Result returned to caller; execution env retained for reuse.

Interviewer: Why Firecracker?

Candidate: Multi-tenant isolation on shared hardware. Linux containers alone share the kernel — too permeable for cross-tenant. KVM-based micro-VMs give hardware-level isolation. Firecracker is a slim VMM purpose-built for serverless: ~125 MB memory overhead, ~125ms boot. Each invocation env is its own VM with kernel, isolated network, isolated filesystem. Workers can host hundreds.

Interviewer: Cold start optimization?

Candidate: Several techniques. Worker pre-warming: pool of pre-booted micro-VMs ready to load any function code (saves boot time). Code caching on workers: recently used function code stays cached. SnapStart: take a snapshot of post-init state, restore from snapshot rather than re-running init — eliminates the variable init time. Provisioned concurrency: customer pays to keep N envs warm always. For container images, layer caching reduces image fetch.

Interviewer: A function suddenly gets 10,000 concurrent invocations.

Candidate: Burst limits handle this. Initial pool of warm envs serves the early ones. New invocations trigger cold starts on workers with capacity. The scheduler bin-packs across the fleet. There's a burst limit per region per minute; beyond that, throttle. Customer can request limit raises. If reserved concurrency is set, that's the guaranteed quota.

Interviewer: Scheduling decisions?

Candidate: Per invocation, scheduler checks (1) is there a warm env for this function and version? If yes, send there. (2) If no, pick a worker with available capacity (memory, CPU, network). (3) Avoid noisy neighbors — don't pack too many high-memory tenants on one host. (4) Affinity for repeated invocations to similar function families. Decision is sub-millisecond.

Interviewer: Event sources?

Candidate: Event source mappings translate source events into invocations. S3 directly invokes on object events. SQS: Lambda service polls queues, batches messages, invokes function. Kinesis/DynamoDB Streams: polls shards in order. EventBridge: rule-based routing. API Gateway: synchronous HTTP-to-Lambda. Each integration handles retries, batching, ordering, and dead-letter semantics appropriate to the source.

Interviewer: A function fails repeatedly.

Candidate: For async invocations: retries with exponential backoff up to configured limit (typically 2x). After exhaustion, message goes to dead-letter queue (SQS or SNS) configured per function. For sync: error returned to caller; they handle. Metrics surface error rate; alerts notify customer.

Interviewer: VPC connectivity?

Candidate: Functions can run in customer VPC. Each function-version needs Elastic Network Interfaces (ENIs) in the VPC subnets. Originally ENI provisioning per cold start added significant latency; modern Lambda pre-provisions ENIs per function to avoid the penalty.

Interviewer: IAM per invocation?

Candidate: Function has an execution role (IAM role). On invocation, Lambda provides temporary credentials for that role to the function via environment variables. Function uses these to call other AWS services. Tied to the execution env lifecycle; each warm env has credentials it can refresh.

Interviewer: Multi-region?

Candidate: Each region is independent. Function deployed per region. Cross-region invocation goes via internet/API. Multi-region deployment requires the customer to deploy in each region (or use Lambda@Edge for global edge functions).

Interviewer: Summarize.

Candidate: Function registry + code store + scheduler + worker fleet hosting Firecracker micro-VMs. Cold start is the performance challenge, mitigated by warm pools, code caching, and SnapStart. Firecracker provides hardware-level isolation for multi-tenancy. Event source mappings translate triggers into invocations with retry and dead-letter. Concurrency limits per function and per account prevent runaway costs. The scheduler bin-packs invocations across the worker fleet.
