1. Basic Knowledge And Solution

What This Question Is Testing

Google Docs is real-time collaborative editing: multiple users typing in the same document simultaneously, seeing each other's cursors, and never losing changes. The hard part is the concurrency-control algorithm — operational transformation (OT) or conflict-free replicated data types (CRDT) — combined with persistence, offline edits, and presence.

The invariant is this:

Every editor's local view of a document converges to the same canonical state as everyone else's once they have seen the same set of operations; no operation is silently lost; the order of operations is causally consistent; offline edits merge correctly upon reconnection; document content has a durable version history.

At ICT5 / L6+ level, the interviewer is testing whether you can choose between OT and CRDT, design the synchronization protocol, handle presence and cursors, manage permissions, and reason about the per-document hot key (every editor pinned to the same doc).

60-Second Interview Framing

Spoken version:

I would design Google Docs around per-document operation logs serialized through a single authoritative service. Editors send operations; the server transforms them against concurrent operations using OT (or merges via CRDT), broadcasts to all peers, and persists. Presence and cursors ride alongside. The interesting decisions are OT vs CRDT, per-document scaling, offline handling, and permission enforcement.

Requirements To Clarify First

Functional requirements:

  • Multiple users edit the same document simultaneously.
  • See others' cursors and selections.
  • Comments and suggestions.
  • Edit history and revisions.
  • Offline edits sync on reconnect.
  • Permissions: view, comment, edit, owner.
  • Rich text formatting.
  • Share link or email-based invites.

Non-functional requirements:

  • Convergence: all editors see the same final state.
  • Latency: own keystroke visible immediately; peers' within 100-300 ms.
  • High availability per document.
  • Survive editors disconnecting.
  • Auditability: who changed what.
  • Multi-region with reasonable cross-region latency.

Out of scope unless asked:

  • Spreadsheet formulas (similar problem but specialized).
  • Slides (specialized).
  • Embedded media editing.
  • Translation, accessibility tools.

Operational Transformation (OT)

The classic approach.

Concept: each op is rewritten ("transformed") to remain consistent in the presence of concurrent ops.

Example: doc is HELLO. User A inserts "X" at position 3. User B inserts "Y" at position 1 concurrently.

  • Without transform: A's op as-is yields HELXLO; B's op yields HYELLO. After both: depends on order.
  • With OT: when A's op arrives at server after B's op of Insert "Y" at 1, transform A's position from 3 to 4 (shifted by Y). Result is HYELXLO regardless of order.

Implementation:

  • Server is the central sequencer.
  • Each op carries the seq it was generated against.
  • Server transforms incoming ops against any ops sequenced after that point.
  • Broadcasts the transformed op to all peers with its new seq.

Client side:

  • Apply local op immediately (optimistic).
  • Send to server.
  • When server acks with the transformed version, reconcile if needed.

Interview phrase:

OT requires a central sequencer because the transform function depends on knowing all ops in between. The server is the source of truth for ordering.

CRDTs (Alternative)

Conflict-free Replicated Data Types: ops carry enough metadata (timestamps, unique ids) to merge commutatively without coordination.

For text, common CRDTs are RGA, Yjs, Logoot, Treedoc.

Pros over OT:

  • Don't need a central sequencer; peers can merge directly.
  • Works offline; merge on reconnect.

Cons:

  • More metadata per op (can be expensive).
  • Implementation is harder.
  • Some CRDTs have "tombstone bloat" or non-trivial GC.

For a system like Google Docs, OT with a central server is the historical choice. Modern systems often use CRDTs.

Sync Protocol

Editor connects via WebSocket to the doc session server.

  1. Server returns the latest snapshot + ops since snapshot.
  2. Client materializes local state.
  3. Client makes a local edit; generates an op tagged with current seq.
  4. Client applies locally (optimistic UI).
  5. Client sends op to server.
  6. Server transforms against any concurrent ops; assigns new seq; persists to op log; broadcasts to all peers.
  7. Other peers receive op; transform their local state if needed.
  8. Originating client receives ack; reconciles if local seq differs.

For offline edits: - Buffer ops locally with monotonic client_op_id. - On reconnect, send buffered ops to server. - Server transforms each in turn against the current state.

Snapshots And History

Replaying millions of ops on doc open is expensive. Periodic snapshots:

  • Every N ops or M minutes, write a snapshot containing the materialized doc + the seq it represents.
  • On open, load snapshot + ops since snapshot.
  • Old ops compactable but typically retained for history.

Revisions:

  • The op log is the history.
  • "Revision points" can be marked at snapshots.
  • Named revisions are user-created bookmarks.

Presence And Cursors

Ephemeral; broadcast over the same WebSocket as ops.

  • Client sends cursor position every few hundred ms or on selection change.
  • Server forwards to all peers without persisting.
  • Each peer is shown as a colored cursor + name.

Cursor positions update as text shifts via OT/CRDT transformation.

Comments

Anchored to a span of text by character offset + position id (so they survive edits).

  • Stored separately from main text.
  • Threaded replies.
  • Resolve / unresolve state.
  • Notification when @mentioned.

Scaling And Partitioning

Data Partition key Why
Documents doc_id hash One doc, one session leader.
Operation log doc_id hash Per-doc append.
Snapshots doc_id hash Per-doc materialization.
Permissions doc_id Checked per doc.
Presence In-memory at session server Ephemeral.

Hot doc (e.g., a town-hall edit-along with thousands of editors):

  • The single session leader becomes the bottleneck.
  • Mitigations:
  • Read-only viewers don't go through the leader; they stream ops from a fan-out cache.
  • Vertical scaling of the session server.
  • Rate-limit ops per editor.
  • For very large collaborative sessions, fall back to a "read-only with one editor" mode.

Interview phrase:

The hardest scalability case for a doc is not the long tail; it is the one viral doc that thousands of people are editing simultaneously. The session leader is the central serializer and cannot be horizontally split per document.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Session server dies. Failover to replica; ops re-broadcast from log. Session reconnection rate.
Editor offline edits. Buffer ops; replay on reconnect with OT transform. Buffered op count distribution.
Conflicting concurrent edits. OT transform / CRDT merge ensures convergence. Convergence-check failures.
Op log shard hot for popular doc. Pre-shard; cap viewers via fan-out tier. Per-doc op rate.
Permission change mid-session. Re-check on op; eject editor if revoked. Revoke propagation lag.
Comment storm. Rate limit per user. Comment QPS per doc.
Snapshot service falls behind. More frequent snapshots; alert. Snapshot lag.
Cross-region editor on remote doc. Route to doc's home region; accept latency. Cross-region latency.
Network glitch causes op duplicates. Idempotent on client_op_id. Duplicate op rate.
Editor's local state diverges. Periodic state hash check; force re-sync on mismatch. Divergence-detection count.

Operational Metrics

Correctness:

  • State divergence between client and server.
  • Convergence delay distribution.
  • Failed transforms.

Capacity:

  • Active editing sessions per server.
  • Op QPS per doc.
  • WebSocket connection count.
  • Snapshot creation rate.

Business:

  • Concurrent editors distribution.
  • Editing duration.
  • Comments per doc.

Multi-Region Strategy

  • Each doc has a home region (typically created in user's region).
  • Session leader runs in home region.
  • Remote editors connect via WebSocket to home region.
  • Snapshots replicated globally for read-only access.
  • Failover region warm.

Tradeoffs To Call Out

  • OT requires central sequencer; CRDT is more distributed but more complex.
  • Central session leader is a per-doc bottleneck; scaling it is hard.
  • Frequent snapshots reduce replay cost but use storage.
  • Storing all keystrokes is huge but enables time-machine history.
  • Optimistic local apply gives instant feedback but requires reconciliation on transform.
  • Strong permission checks per op cost latency; cached checks risk staleness.

Common Traps And Stronger Moves

Trap Stronger move
Last-writer-wins. OT or CRDT for convergence.
Single op-log scan on every open. Snapshots + ops since.
Direct DB write per keystroke. Append-only op log; periodic snapshots.
Lock-based editing. Lock-free with transformation.
Sync via polling. WebSocket with broadcast.
Multi-region active session leader. Single-region per doc; cross-region read replicas.

Staff-Level Answer Checklist

  • OT or CRDT chosen and justified.
  • Per-doc session leader serializes ops.
  • Append-only op log; periodic snapshots.
  • Optimistic local apply with reconciliation.
  • Offline edits buffer + transform on reconnect.
  • Presence and comments separate from text ops.
  • Permissions checked per op.
  • Multi-region with per-doc home region.

Interview Phrase

Google Docs is fundamentally a per-document operation log serialized through a session leader. Operational transformation (or CRDT) keeps all editors convergent despite concurrent edits. Snapshots make load cheap; offline edits buffer and reconcile. Presence and comments ride alongside but don't go through the main op pipeline.

2. Simulated Interview

Interviewer: Design Google Docs.

Candidate: I will design around a per-document operation log serialized by a session leader. Editors connect via WebSocket; the leader sequences ops with OT, broadcasts to all peers, and persists. Snapshots periodically materialize the current state to avoid replaying full history. Presence and comments are separate. The invariant is that all editors converge to the same state and no op is silently lost.

Interviewer: OT or CRDT?

Candidate: Both are valid; I'll discuss OT since it's the historical choice for Google Docs. OT requires a central sequencer because transforming an op depends on knowing what ops came before. The server is the source of truth for order. CRDT is more peer-to-peer friendly because ops carry enough metadata (timestamps, unique ids) to merge commutatively without coordination. CRDTs are popular in modern systems but have more per-op overhead.

Interviewer: Walk me through a keystroke.

Candidate: User types "X" at position 5. Client generates an op Insert("X", at=5, parent_seq=42) and applies locally — UI shows X immediately. Sends op to server over WebSocket. Server sees current seq is 45 (three ops have arrived since the user's last sync). Transforms the user's op against ops 43, 44, 45: maybe their inserts shifted position 5 to 7. New op Insert("X", at=7) sequenced at 46. Server persists to op log, broadcasts to all peers. Originating client gets the ack with seq 46; reconciles if local position differs.

Interviewer: Two users edit the same line simultaneously.

Candidate: OT handles it. User A inserts "X" at position 3, parent_seq 42. User B inserts "Y" at position 3, parent_seq 42, concurrently. Server gets A first, assigns seq 43, broadcasts. Then gets B's op with parent_seq 42; transforms against op 43 (which is A's insert at 3); B's position shifts to 4. New op Insert("Y", at=4) sequenced at 44. All peers converge to "AB..." or "XY..." order depending on transform rule (typically based on author id tiebreak). The point: convergence is guaranteed.

Interviewer: A user goes offline, types 100 paragraphs, comes back online.

Candidate: Client buffers ops with monotonic client_op_id. On reconnect, sends the entire buffer in order. Server transforms each against the current state (which has moved forward), sequences them, broadcasts. The user's edits land at appropriate positions in the current doc. If parts of the doc the user edited offline were deleted by someone else, transform handles it: edits to deleted regions are either skipped or anchored to nearest surviving text per policy.

Interviewer: 5,000 people edit one doc at the same time.

Candidate: This is the hardest case. The session leader for the doc is the bottleneck. Mitigations: most of the 5,000 are likely viewers, not editors. Read-only viewers stream ops from a fan-out cache, not the leader. Among actual editors, rate-limit ops per editor. For truly viral edit sessions, we may degrade to one writer at a time (lock-mode) or limit editor count.

Interviewer: Permissions?

Candidate: Drive owns permissions. On connect, the session leader checks the user's access. Per op, a fast cached check confirms. On permission revoke, the leader receives a notification and ejects the user; further ops are rejected. The cache TTL is short for revoke propagation.

Interviewer: How is history stored?

Candidate: The op log itself is the history. Periodic snapshots compact: every N ops or M minutes, materialize the current state to a snapshot. On open, load snapshot + ops since. For named revisions ("Final draft"), users mark snapshots. The op log is retained indefinitely for time-machine restore.

Interviewer: Comments?

Candidate: Anchored to a text span by character offsets that survive edits. When the surrounding text shifts, the anchor moves with it (OT applies to anchors too). Comments are stored separately from text ops to avoid bloating the main log. Threaded replies. Resolve state.

Interviewer: Multi-region?

Candidate: Each doc has a home region. Session leader runs there. Editors from other regions connect via WebSocket, accepting the cross-region latency (a few hundred ms). Snapshots and op log replicate globally for fast read-only access. Failover region warm. We don't try to run active-active session leaders per doc because the serialization invariant breaks down.

Interviewer: Doc crashes during a save.

Candidate: Ops persist to the op log before being acked to the client. If the session leader dies, a replica takes over: it has the op log up to the last replicated op. Clients reconnect; replay any ops not yet acked. The append-only log is the recovery mechanism.

Interviewer: Summarize.

Candidate: Per-doc session leader serializes ops via OT. Append-only op log per doc with periodic snapshots. Optimistic local apply, transform on server, broadcast to peers. Offline edits buffer and reconcile. Presence and comments separate. Permissions checked per op via Drive. Multi-region uses per-doc home with replicated snapshots. The hardest case is a single very-popular doc whose session leader is a serialization bottleneck.

3. Practice Notes

Five-Minute Version

  1. Clarify: real-time collab, offline, history, permissions.
  2. Invariant: convergence; no lost ops.
  3. Per-doc session leader; OT or CRDT.
  4. Append-only op log; periodic snapshots.
  5. Optimistic local apply; transform on server.
  6. Offline buffer; reconcile on reconnect.
  7. Presence and comments separate.
  8. Permissions per op; multi-region per-doc home.

One-Minute Closeout

Google Docs is a per-document operation log serialized through a session leader. OT (or CRDT) keeps all editors convergent despite concurrent edits. Optimistic local apply gives instant feedback; server transforms reconcile differences. Snapshots make load cheap; offline edits buffer and reconcile on reconnect. Presence and comments are separate from the main op pipeline. Multi-region uses per-doc home regions. The hardest scaling case is one very-popular doc.

Best Interview Phrases

  • "OT requires a central sequencer; CRDT supports peer-to-peer merge."
  • "The session leader for each doc is the central serializer for that doc."
  • "Optimistic local apply gives instant feedback; transform on the server reconciles."
  • "Offline edits buffer locally and replay through transform on reconnect."
  • "Per-doc home region; replication across regions is read-only."
