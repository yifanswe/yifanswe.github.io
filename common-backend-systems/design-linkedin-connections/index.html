Design LinkedIn Connections

Contents

1. Basic Knowledge And Solution

What This Question Is Testing

LinkedIn connections form a professional social graph with explicit distance semantics: 1st degree (direct), 2nd degree (friend-of-friend), 3rd degree (further). The hard part is computing distances on demand at billion-user scale (typical query: "how am I connected to X?") with low latency, while supporting connection invites, recommendations, and search filtered by distance.

The invariant is this:

The connection graph is bidirectional (a connection is mutual); distance queries between any two users return their shortest-path distance up to a threshold or "out of network"; recommendations leverage 2nd/3rd-degree connections; invite acceptance creates an atomic bidirectional edge; the graph supports queries within tight latency.

At ICT5 / L6+ level, the interviewer is testing whether you can design a sharded social graph store, support 1st/2nd/3rd-degree distance queries at scale (with bounded BFS), connection invite flows, and reason about graph storage tradeoffs.

60-Second Interview Framing

Spoken version:

I would design LinkedIn connections as a distributed graph store sharded by user_id, with bidirectional edges representing mutual connections. Distance queries use bounded BFS (limited to 3 hops). Per-user 1st-degree connection lists are cached. Invite acceptance creates atomic bidirectional edges. PYMK and search ride on this graph. The interesting decisions are graph storage, distance computation, edge-list caching, and the BFS bound.

Requirements

Functional requirements:

  • Users connect with each other (bidirectional).
  • Send / accept / decline invites.
  • View 1st-degree connections list.
  • Distance to any other user (1st, 2nd, 3rd, out of network).
  • "Who can introduce me" via 2nd degree.
  • Connection recommendations (see PYMK).
  • Connection-aware search filters ("2nd degree only").

Non-functional requirements:

  • Distance query < 200ms.
  • 1B+ users; hundreds of avg connections.
  • Some users have 30k+ connections (sales people, recruiters).
  • High availability.
  • Multi-region.

Out of scope unless asked:

  • Premium features (InMail, advanced search depth).
  • Profile / experience search.

Distance Query

The classic LinkedIn query: "what's my distance to user X?"

BFS bounded to depth 3:

  1. Start at U; check if X is in 1st degree → return 1.
  2. Otherwise, expand 1st degree to 2nd: union of friends of U's friends. Is X in the set? → return 2.
  3. Otherwise, expand to 3rd degree. Is X in the set? → return 3.
  4. Otherwise, return "out of network."

For users with average connections (~500), 2nd degree is ~50k-100k users; 3rd degree explodes. Bounded BFS is essential.

Implementation:

  • Fetch U's 1st-degree list.
  • Check membership of X.
  • If not, fetch 2nd-degree set: union of friends of each of U's friends. Membership check.
  • For 3rd, similar expansion.

For power users with 30k connections, 2nd-degree expansion is heavy (30k friends × ~500 each = 15M edges). Pre-compute 2nd-degree sets.

Interview phrase:

Distance queries are bounded BFS to depth 3. For users with normal degree, on-demand BFS is fine. For power users, pre-compute 2nd-degree sets and use Bloom filters for fast "in network" tests.

Pre-Computed 2nd-Degree Sets

For high-degree users (>1000 connections), pre-compute their 2nd-degree set as a Bloom filter or compressed bitmap.

  • Updated incrementally on new connections.
  • Used as fast "is X in my 2nd degree?" check.
  • Avoids on-demand expansion for power users.

For 3rd degree, even more aggressive caching; or accept "out of network" if not in cached set.

Invite Flow

  1. User U sends invite to V.
  2. Invite service creates Pending Invite.
  3. V receives notification.
  4. V accepts: invite state → accepted.
  5. Atomic bidirectional connection creation: - Write Connection(U, V) with canonical order. - Update U's connection list cache. - Update V's connection list cache.
  6. Invalidate distance caches involving U and V.

Atomicity: use a transaction across both users' shards if needed, or a state machine that ensures both updates eventually happen with retry.

Connection List Cache

Per-user 1st-degree list cached. Refreshed on connection change.

For power users with 30k connections, the list itself is large — paginate. The full list lives in storage; the cache may hold a compressed bitmap of connected user_ids.

Recommendations

See PYMK design. Leverages 2nd-degree connections plus shared employer, school, interests.

Search Integration

Search queries can filter "only my 2nd degree":

  • Distance service called per result candidate.
  • For batch result sets, batched distance queries.
  • Pre-computed 2nd-degree set helps when the user is power-connected.

Privacy

  • Users can hide their connections from non-connections.
  • "Visible to: My connections only" controls who sees the list.
  • Distance queries respect privacy: if V hides connections, U can know V is at distance 2 (via a mutual) but can't list V's other connections.
  • Blocked users excluded from queries.

Scaling And Partitioning

Multi-Region Strategy

  • User home region for connection data.
  • Connection edges replicated globally.
  • Distance service per region.
  • Pre-computed 2nd-degree sets per region.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
2nd-degree expansion slow. Pre-compute for power users. Distance query p99.
Power user hot partition. Sub-partition; aggregate. Per-shard QPS.
Stale connection cache after accept. Invalidation events. Cache lag.
Atomic bidirectional create race. Transaction or saga. Inconsistency count.
Block bypass. Filter at query. Bypass count.
Privacy leak. Visibility checks. Privacy reports.
Spam invites. Rate limit; auto-detect. Per-user invite rate.
Cross-region replication. Async; bounded. Lag.
Bloom filter false positive (in 2nd degree). Confirm with actual lookup. False positive rate.
Search filter slow. Pre-cache or batch distance. Search-filter latency.

Tradeoffs To Call Out

  • On-demand BFS is flexible but slow for power users; pre-compute trades storage for latency.
  • Bidirectional storage means writes hit two partitions; atomicity is a concern.
  • Bounded BFS at 3 hops misses long-distance connections; accept "out of network."
  • More aggressive caching helps but adds invalidation complexity.
  • Strict privacy reduces matchmaking utility.
  • Invite rate limits prevent spam but frustrate legitimate networkers.

Common Traps And Stronger Moves

Trap Stronger move
Unbounded BFS. Cap at 3 hops; pre-compute for power users.
Edges as one-sided. Bidirectional with atomic creation.
No power-user handling. Pre-compute 2nd-degree set; Bloom filter.
Synchronous cache invalidation. Async via pub/sub.
Trust client claims for distance. Server-computed distance.
Ignore privacy in distance UI. Respect connection visibility settings.

LinkedIn connections is a bidirectional social graph with 1st/2nd/3rd-degree distance queries. 2. Simulated Interview

Interviewer: Design LinkedIn connections.

Candidate: Distributed graph store with bidirectional edges sharded by user_id. Connection list cached per user. Distance service performs bounded BFS up to 3 hops. For power users with huge connection counts, pre-compute 2nd-degree sets as Bloom filters or compressed bitmaps. Invite flow creates atomic bidirectional edges on accept. The invariant is bidirectional symmetric connections, distance queries within latency, privacy respected.

Interviewer: Distance query.

Candidate: Bounded BFS up to depth 3. Step 1: is X in U's 1st-degree list? Return 1 if yes. Step 2: expand 1st-degree friends' friends; is X in that set? Return 2 if yes. Step 3: similar for 3rd. Return "out of network" otherwise. The hard case is when U has many connections — 2nd-degree expansion of 500 friends × 500 each = 250k users, doable. For 30k-connection power users, expansion is heavy — pre-compute and use Bloom filter for membership check.

Interviewer: Power user with 30k connections.

Candidate: Pre-compute their 2nd-degree set as a Bloom filter. Updated incrementally on new connections (their friend's friends accumulate). Distance query consults Bloom filter for "is X in 2nd degree?" — sub-millisecond. Bloom filter false positives are confirmed by actual lookup. For 3rd degree, even larger; either pre-compute or accept "out of network" beyond a depth bound.

Interviewer: Invite accept — atomicity.

Candidate: Bidirectional edge must be created atomically. The Connection table stores once (canonical order: smaller id first). Both users' connection list caches must be updated. Options: transaction across both user's partitions, or saga with idempotent steps. Saga: create Connection record; on success, two events go to update each user's cache; if cache update fails, retry; eventually consistent within seconds.

Interviewer: Privacy.

Candidate: Several rules. Users can hide their connection list from non-connections. Distance queries respect this: U can know V is at distance 2 (via a mutual), but cannot list V's other connections. Blocked users are filtered out — appear as "out of network" even if technically connected via mutuals. Privacy filter applied at every query.

Interviewer: Hot partition for power user.

Candidate: Power user's edges + queries can saturate their partition. Mitigations: sub-partition the connection list (one row per N edges); pre-aggregate 2nd-degree state at update time so queries don't trigger full expansion; cache hot reads aggressively.

Interviewer: Invite spam.

Candidate: Per-user invite rate limits. Anti-spam detection: invites to many non-related people, low acceptance rate, account age signals. Block sending after exceeded thresholds. Manual review for severe cases. Visibility into invite-to-accept ratio per user.

Interviewer: Search with distance filter.

Candidate: Query: "find candidates with skill X within my 2nd degree." Search returns initial candidates; distance service called per candidate (or batched). For power user, pre-computed 2nd-degree set makes filter fast. For typical user, on-demand BFS works.

Interviewer: Stale connection cache.

Candidate: Connection list cache invalidated by pub/sub on connection change. Both users' caches updated. Distance caches that involved either user are invalidated. Eventual consistency in the seconds range; acceptable for distance display (users tolerate "1 hour ago you connected" lag).

Interviewer: Multi-region.

Candidate: User home region for connection data. Edges replicated globally. Distance service per region; uses local replicas. 2nd-degree pre-compute per region. Cross-region replication async.

Interviewer: Summarize.

Candidate: Distributed graph with bidirectional edges sharded by user_id. Connection list cached per user. Distance queries via bounded BFS to depth 3; for power users, pre-computed 2nd-degree Bloom filters keep queries fast. Atomic bidirectional create on invite accept. Privacy and blocks apply at every query. Search integration uses the same distance service. Multi-region with user home. The hardest scaling problem is power users; the hardest correctness problem is bidirectional atomicity.

LinkedIn connections is a bidirectional social graph with bounded distance queries (1st/2nd/3rd degree). Sharded graph store with per-user connection list cache.
