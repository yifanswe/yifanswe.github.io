1. Basic Knowledge And Solution

What This Question Is Testing

A planet-scale KV store like DynamoDB sits at the foundation of many production systems. The hard part is offering predictable single-digit-millisecond latency and 99.999% availability across regions while letting the operator pick the consistency mode they need per request.

The invariant is this:

For any key, a successful write is durable across at least a quorum of replicas, eventually visible to all consistent readers, and the system continues to accept writes during single-node, single-zone, and even partial-region failures, with conflict resolution rules that converge deterministically.

At ICT5 / L6+ level, the interviewer is testing whether you can choose between leaderless quorum-based replication (Dynamo paper style) and single-leader partition replication (modern DynamoDB style), explain consistent hashing and partition splits, handle vector clocks or last-write-wins, and design hot-partition mitigations.

60-Second Interview Framing

Spoken version:

I would design a managed KV store with three planes: a request router, a partition layer that owns ranges of the key space, and a replicated storage layer per partition. The interesting decisions are partitioning strategy (consistent hashing vs range partitioning with auto-split), replication (leaderless quorum vs leader-based), consistency tunability (strong vs eventual), and handling hot partitions. I will clarify requirements, pick a partition + replication scheme, walk write and read paths, then go deep on quorum semantics and hot keys.

Requirements To Clarify First

Functional requirements:

  • Put, Get, Delete by primary key (single-key operations).
  • Optional sort key for range scans within a partition.
  • Conditional writes (CAS): write iff version = X or item does not exist.
  • Batch get/put, paginated scans.
  • Secondary indexes (global and local).
  • Streams of changes for downstream consumers.
  • Per-table TTL for automatic deletion.

Non-functional requirements:

  • Single-digit ms p99 reads and writes.
  • 99.99% to 99.999% availability per region.
  • Tunable consistency per request: strong or eventual.
  • Petabyte scale per table; tens of millions of QPS.
  • Multi-region with active-active option.
  • Per-tenant isolation and quotas.

Out of scope unless asked:

  • Full SQL query planner.
  • Multi-key ACID transactions across partitions (mention as a separate feature).
  • Backup and PITR mechanics.

Scale Assumptions

Area Assumption
Tables Hundreds of thousands across tenants.
Items per table Billions to trillions.
Item size KB-range typical; max 400 KB.
Throughput Up to tens of millions of read/write units per second per table.
Latency p50 < 5 ms, p99 < 10 ms for in-region.
Replication 3 replicas per partition by default; cross-AZ within region.
Multi-region Optional active-active with last-writer-wins or CRDT.
Hot partition Single partition limited to ~3000 RCU / 1000 WCU; need to detect and split.

The bottleneck is hot partitions and tail latency under replica failure.

Core Product Model

Surface Operation Requirement
Single-key write Put/Update/Delete by primary key. Atomic per key; tunable durability.
Single-key read Get by primary key. Tunable consistency; very low latency.
Range query Items sharing partition key, sort key range. Bounded results; pagination via cursor.
Conditional write Compare-and-swap. Atomic check-and-set within a partition.
Streams Change log per partition. Ordered per partition; at-least-once delivery.

The system is intentionally constrained to per-key (or per-partition for range) operations. Multi-partition transactions are the exception, not the rule.

Streams Per-partition change log exposed to consumers. Ordered per partition; sharded for parallel consumption.
Cross-region replicator Replicates writes between regions for global tables. Last-writer-wins by default; CRDTs for counters.

Two Architectural Choices

Choice A: Leaderless quorum (original Dynamo paper)

  • N replicas per key, no designated leader.
  • Writes go to W of N replicas, reads from R of N. R + W > N for strong consistency.
  • Conflicts resolved via vector clocks and sibling reconciliation at read time.
  • Hinted handoff: if a replica is down, a temporary stand-in stores the write.
  • Anti-entropy via Merkle trees periodically reconciles divergent replicas.

Pros: write-available even with replica failures; no leader bottleneck. Cons: sibling resolution is complex; clients must merge or system must apply LWW.

Choice B: Leader-based per partition (modern DynamoDB)

  • Each partition has one leader; replicas via consensus (Multi-Paxos / Raft style).
  • Writes ack only after majority log replication.
  • Strong consistency on reads from leader; eventual from replicas.
  • Failover is a leader re-election; takes seconds.

Pros: simpler client model; strong consistency natural; no sibling problem. Cons: leader is bottleneck per partition; failover causes brief write unavailability.

Modern DynamoDB is closer to leader-based per partition with Multi-Paxos. The interview answer should pick one and justify; if asked, contrast both.

Interview phrase:

I would default to leader-based partition replication because the operator experience is simpler — no siblings to merge, strong consistency on demand, and per-partition leaders give natural sharding. Leaderless quorum is more available under replica loss but pushes complexity to clients.

Partitioning

DynamoDB uses hashed partition key as primary partition strategy:

  • Apply hash function to partition key, mod into the key space.
  • Each partition owns a contiguous range of the hashed space.
  • Item with (partition_key, sort_key) goes to the partition whose range contains hash(partition_key).
  • Within a partition, items are sorted by sort_key (B-tree or LSM-organized).

Auto-split: When a partition's storage or throughput exceeds threshold, it splits at the median key. The split is online: the split table picks a midpoint, replication continues on both new partitions, and the metadata service flips the routing once the new partitions are caught up.

Hot keys: A single partition key (one user, one tenant) can saturate one partition. Mitigations: - Adaptive capacity: temporarily allow the partition more throughput by borrowing from idle siblings. - Write sharding: client appends a suffix to the partition key (user_42#0, user_42#1, ...) to spread across partitions, then queries all suffixes. - Burst capacity: short-term over-provisioning.

Interview phrase:

Auto-split handles uneven data volume, but the same hot partition key still routes everything to one partition. The fix has to be on the access pattern — write sharding or caching above the store.

Replication

Per-partition replication (leader-based):

  1. Client write reaches router → partition leader.
  2. Leader appends to local WAL.
  3. Leader sends log entry to followers; followers persist to WAL.
  4. Once majority (e.g., 2 of 3) acked, leader applies to local state and responds.
  5. Followers apply asynchronously and confirm.

Consensus details:

  • Raft-style with terms and log indices, or Multi-Paxos with a stable leader.
  • Leader election on failure within seconds; the metadata service may participate or it may be intrinsic to the consensus group.

For leaderless (Dynamo paper):

  1. Client writes to coordinator; coordinator forwards to N replicas in parallel.
  2. Wait for W acks; respond.
  3. On read, coordinator queries R replicas; resolves conflicts.
  4. Vector clocks track causal history; on conflict, return all siblings to client or merge by LWW.

Consistency Model

Read mode Behavior
Eventually consistent Read from any replica; may be stale by milliseconds to seconds.
Strongly consistent (leader-based) Read from leader; reflects all committed writes.
Linearizable read (leaderless) Quorum read of R replicas with R+W>N.
Write mode Behavior
Standard put Acked after majority replicas persist.
Conditional put Same plus condition evaluated atomically on partition leader.
Transactional write Two-phase commit across partitions involved.

Write Path

  1. Client → request router with partition key.
  2. Router resolves partition via cached map; forwards to leader.
  3. Leader checks conditional expression (if any) against current state.
  4. Append to WAL; replicate to followers.
  5. Wait for majority ack.
  6. Apply to local LSM/Btree.
  7. Emit to per-partition stream; update GSI asynchronously.
  8. Respond.

Failure cases:

  • Conditional fails: return ConditionalCheckFailed; nothing applied.
  • Replica unreachable: leader uses remaining replicas if quorum still achievable; else fail with retryable error.
  • Leader fails mid-write: client retries; new leader has the log up to last committed entry.

Read Path

  1. Client → router → leader (strong) or any replica (eventual).
  2. Replica returns item from its local store.
  3. For eventual, no further coordination.
  4. For strong, the leader confirms it has applied all committed entries (it always has, by definition of being leader).

For range queries, the leader scans the sort-key range, applies the filter, and returns up to the result limit plus a cursor.

Secondary Indexes

Local secondary index (LSI): alternate sort key within the same partition. Updated synchronously with the base table; consistent with base.

Global secondary index (GSI): different partition key entirely. Maintained asynchronously by a GSI writer that consumes the partition's change log. Eventually consistent.

GSI writes can fail or lag without affecting base writes. The application tolerates that by querying GSIs with eventual consistency.

Streams (Change Data Capture)

Per-partition append-only log of operations:

  • Same sequence numbers as the replication log.
  • Exposed as shards corresponding to partitions; consumers iterate cursors.
  • Retention typically 24 hours.
  • Order guaranteed per partition; not across partitions.

Useful for triggers, materialized views, search indexes, replication to other systems.

Multi-Region Strategy

Two main modes:

Global tables (active-active): - Each region accepts writes locally. - Cross-region replication is asynchronous. - Conflict resolution: last-writer-wins by timestamp + region tiebreaker, OR CRDT semantics for counters and sets. - Reads are eventually consistent across regions. - Required SLA: convergence within seconds.

Single-region with cross-region read replica: - One region accepts writes. - Other regions have read-only replicas. - Simpler; failover requires explicit promotion.

For most workloads, single-region is sufficient. Active-active is for low-latency global writes (and the price is the conflict model).

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Replica down Continue with remaining quorum; rebuild on recovery. Replica health, quorum size.
Leader down Re-elect within seconds; writes pause briefly. Leader election rate, write unavailability.
Partition runs hot. Adaptive capacity; split partition; rate-limit client. Per-partition throughput, throttled requests.
Storage volume fills. Auto-split + rebalance. Per-host storage utilization.
Cross-region replication lag. Backpressure to writers; alert on lag. Replication lag p99.
Conflicting concurrent writes (active-active). LWW with region tiebreaker; CRDT for counters. Conflict count per partition.
GSI lag. Async backfill; bounded staleness alerts. GSI lag per index.
Network partition isolates a region. Continue locally; converge on heal. Partition detection time, reconvergence.
Tail latency from one slow replica. Hedged reads to a second replica with timeout. Read p99 vs p50 gap.
Client retries amplify load. Server-side admission control; exponential backoff hints. Retry-after rate, throttled QPS.

Operational Metrics

Correctness:

  • Stale reads after strong-consistency request (must be zero).
  • Conflict resolution events per region.
  • GSI sync lag.

Capacity:

  • Per-partition read/write QPS and throughput consumption.
  • Storage growth.
  • Replication lag intra-region and cross-region.
  • Throttled-request rate per table.

Business:

  • p50/p99 latency per table.
  • Cost per operation.
  • Hot key detection events.

Tradeoffs To Call Out

  • Leader-based replication gives strong consistency easily but loses brief availability on failover. Leaderless gives availability but pushes conflict resolution to the client.
  • Hashed partitioning gives uniform distribution but breaks range scans across partition keys; that is by design.
  • Auto-split rebalances data but cannot fix hot partition key access patterns.
  • Strong reads cost an extra round trip (or leader hop). Eventual reads are cheaper and usually fine.
  • Active-active multi-region helps latency but constrains the data model (LWW, CRDT).
  • LSM provides high write throughput at cost of compaction overhead; B-tree gives predictable reads but writes are slower.

Common Traps And Stronger Moves

Trap Stronger move
Hash everything by user id without thinking. Identify hot users and apply write sharding above the store.
Promise strong reads everywhere. Differentiate eventual vs strong per request; charge for strong.
Forget secondary index consistency. State GSI is eventually consistent; LSI is consistent.
Multi-key transactions everywhere. Limit transactions to small sets of partitions; design data model to colocate.
Treat global tables as "free magic." Explain LWW / CRDT and the consequences.
Ignore tail latency. Hedged reads, P99 alarms, replica health checks.

Staff-Level Answer Checklist

  • Pick a partitioning + replication scheme and justify.
  • State per-key atomicity invariant and replication quorum.
  • Walk write and read paths with consistency modes.
  • Handle hot partitions explicitly.
  • Cover GSIs as async derived state.
  • Address multi-region active-active with conflict semantics.
  • Streams as the change capture surface.
  • Operational metrics and tail latency mitigations.

Interview Phrase

A planet-scale KV store is fundamentally a partitioned, replicated log per partition with a key-range routing layer. The hard decisions are not the data structure; they are partitioning (hash with auto-split), replication (leader vs leaderless), and multi-region conflict semantics. Hot partitions and tail latency are the operational headaches that take the most engineering.

2. Simulated Interview

Interviewer: Design a key-value store like DynamoDB.

Candidate: I will design a three-layer system: a request router, a partition layer that owns key ranges, and a replicated storage layer per partition. The core invariant is that successful writes are durable across a majority of replicas and reads can be tuned per request between strong and eventual.

Interviewer: How do you partition data?

Candidate: I hash the partition key and assign each partition a contiguous range of the hashed space. Each partition is replicated three ways across availability zones. When a partition's storage or throughput exceeds threshold, it auto-splits at the median key; the split happens online by spinning up the two new partition leaders, catching them up from the log, then flipping the metadata routing.

Interviewer: Leader-based or leaderless replication?

Candidate: Leader-based per partition. Each partition has a leader elected via Raft-style consensus, plus two followers. Writes go through the leader, replicate to a majority quorum (2 of 3), then ack. Strong reads come from the leader; eventual reads from any replica. The reason I prefer this over the original Dynamo paper's leaderless quorum is simpler client semantics — no sibling resolution to push to applications.

Interviewer: A single partition key becomes hot — one user generates 100,000 requests per second. What do you do?

Candidate: Three layers of defense. First, adaptive capacity: the partition borrows throughput budget from idle sibling partitions. Second, hot-partition splitting cannot help because all the load is on one key; auto-split splits by storage, not by traffic per key. Third, the application has to shard the key: append a small random suffix to the partition key on write (user_42#0 through user_42#9), and fan out queries across all suffixes. That trades a small read amplification for spreading writes.

Interviewer: What if a leader dies?

Candidate: Raft election kicks in; a follower with the most up-to-date log becomes leader in a few seconds. Writes briefly pause for the affected partitions. Followers continue serving eventual reads throughout. The metadata service is informed of the new leader and updates the routing cache; clients with stale routes get a "not leader" error and retry against the metadata service.

Interviewer: How do conditional writes work?

Candidate: Single-key CAS is straightforward — the leader reads current value, evaluates the condition, and if it passes, writes the new value atomically by including the check in the replication log entry. For multi-item transactions across partitions, two-phase commit: a coordinator prepares each partition leader (which locks the items and writes a prepare entry), then commits once all prepare. Slower; only used when needed.

Interviewer: How do secondary indexes work?

Candidate: Two kinds. Local secondary index is within the same partition; updated synchronously with the base item, consistent. Global secondary index uses a different partition key, so updates must be cross-partition. The GSI maintainer consumes the change stream from the base partition, computes the GSI entry, writes it to the target partition. GSIs are eventually consistent; lag is usually under a second but spikes during reindex.

Interviewer: How do streams work?

Candidate: Every partition has an append-only change log with monotonic sequence numbers. Stream consumers iterate per partition shard with a cursor. We retain 24 hours. Order is guaranteed per partition, not across partitions. Consumers include GSI maintainers, cross-region replicators, and application triggers.

Interviewer: How do global tables work — active-active across regions?

Candidate: Each region accepts writes for the same table. Cross-region replication is asynchronous through the stream. Conflicts use last-writer-wins by timestamp, with region id as tiebreaker. For counter attributes, a CRDT counter would be more correct but most products live with LWW. Convergence target is a few seconds. The cost is that two simultaneous writes in different regions can lose one's data; users have to design around this (e.g., per-region partition keys for "owned" data).

Interviewer: What about tail latency?

Candidate: Several techniques. Hedged reads: send the request to the leader and a replica simultaneously, take the first response; the slow one is canceled or just ignored. Speculative retries. Capacity headroom: target 60-70% utilization so a slow node does not cascade. Per-replica health checks demote slow replicas from the read path. Compaction scheduling avoids spiking I/O during peak.

Interviewer: Storage engine choice?

Candidate: LSM tree underneath. Writes append to memtable then flush to sorted runs on disk; merge in background. Predictable write throughput, good compression, slightly worse read tail than B-tree due to merge passes. Bloom filters per SSTable speed up negative lookups.

Interviewer: Multi-key transaction across partitions — how often does it happen?

Candidate: Rarely, by design. The data model encourages single-partition operations because that is where ACID is cheap. Cross-partition transactions use 2PC and are charged at a higher cost. Application architects are encouraged to redesign rather than reach for transactions.

Interviewer: Summarize.

Candidate: Hashed partition keys with auto-split; three-replica Raft-style leader per partition; tunable consistency per request; async GSIs via change stream; active-active multi-region via LWW or CRDT; hot partitions handled with adaptive capacity and client-side sharding; tail latency mitigated with hedged reads and capacity headroom. The architecture is intentionally simple at the operation level (single-key) so the system can deliver predictable performance at planet scale.

3. Practice Notes

Five-Minute Version

  1. Clarify: single-key ops; tunable consistency; range scans within partition; GSIs.
  2. Invariant: durable write to quorum; tunable reads.
  3. Architecture: router + partition layer + replicated storage.
  4. Hashed partitioning with auto-split.
  5. Leader-based per partition; Raft-style consensus.
  6. Write path: WAL → quorum → apply → respond.
  7. Hot key handling: adaptive capacity + client write sharding.
  8. Multi-region: LWW or CRDT; GSI async; streams for CDC.

One-Minute Closeout

A planet-scale KV store is a partitioned, replicated log per partition with a routing layer above it. I choose hashed partitioning with online auto-split, three-replica Raft-style leader per partition, and tunable read consistency. Hot partitions get adaptive capacity and client-side sharding; multi-region uses async replication with last-writer-wins; secondary indexes are eventually consistent via the per-partition change stream. The hardest production problems are hot keys and tail latency, addressed with adaptive capacity and hedged reads.
