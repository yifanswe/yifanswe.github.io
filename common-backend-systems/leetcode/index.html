LeetCode — Code Judging Platform (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build LeetCode. Users browse problems, write code in the browser, submit, and within a few seconds see verdict + runtime stats. We run hostile code from strangers safely. Assume 2k problems, 1M daily active users (DAU), ~5M submissions/day with weekly contest spikes of 100k concurrent submissions.

1. Requirements (~5 min)

Functional - Browse problems and view problem statement. - Submit code; get verdict (accepted, wrong answer, time limit exceeded, memory limit exceeded, runtime error). - See submission history and contest leaderboards.

I'd defer discussions/forum, premium subscriptions, and contest organization.

Non-functional - Sandboxed execution — running arbitrary code, must not let it touch our infra. - p95 verdict latency < 5 s for normal load; < 15 s during contest spikes. - Submission durability — never lose a contest submission. - Eventual consistency on leaderboards is fine (within 5–10 s). - Fairness during contests — same problem, same machine class, deterministic timing.

Capacity - 5M submissions/day → ~60/sec average; contest peak ~100k in 5 min → ~330/sec. - Each submission: small source code (<10 KB) + small judge output. - Submission archive dominates metadata: assume 20 KB code/result/log per submission × 5M/day × RF=3 ≈ 300 GB/day, or ~100 TB/year before compression/tiering. Hidden tests are comparatively small: 2k problems × 50 MB/problem is ~100 GB.

2. Core Entities (~2 min)

  • Problem — problem_id, statement, test cases (hidden), constraints.
  • Submission — submission_id, user_id, problem_id, language, code, status, runtime_ms, memory_kb.
  • TestCase — problem_id, idx, input, expected_output, is_sample.
  • Contest — contest_id, problems[], start, end, participants.

3. API (~5 min)

GET   /v1/problems                          → Problem[]
GET   /v1/problems/{id}                     → Problem (with sample cases)
POST  /v1/submissions                        { problem_id, language, code, contest_id? }  → { submission_id }
GET   /v1/submissions/{id}                   → { status, runtime, memory, failed_case? }
WS    /v1/submissions/{id}/stream            → status updates
GET   /v1/contests/{id}/leaderboard

User from token. Idempotency-key optional but recommended on submissions.

Submission POST is idempotent on {user_id, problem_id, contest_id, idempotency_key}. Leaderboard cursors encode {contest_id, score, penalty, user_id}.

4. Data Flow (~5 min)

Submit → API writes submission row (status=queued) → enqueue to judge queue → worker picks up → spin up sandbox → compile → run against hidden test cases until first failure or all pass → write verdict + stats → push to client via WS → update leaderboard if contest.

5. High Level Design (~10–15 min)

  1. API gateway / submission service. Validates input, writes the submission row (sharded by submission_id) in queued state, enqueues a job to a priority queue (Kafka or SQS), pushes a submission_created event for leaderboard subscribers.
  2. Judge queue. Separate queues per language and per priority (contest vs practice). Contest submissions get a dedicated worker pool to prevent practice traffic from starving them.
  3. Judge workers (the hard part). Pull a job, spin up an isolated execution environment (gVisor, Firecracker microVM, or Docker with strict seccomp + cgroups + no network). Compile the source, run against the hidden test cases in order, capture stdout, compare against expected output, measure CPU and memory.
  4. Test-case storage. Object store (S3) with a CDN-fronted cache; problems are immutable so caching is trivial.
  5. Verdict service. Writes the final status, runtime, memory to the submission row; pushes update to WS subscribers.
  6. Leaderboard service. Per-contest sorted set in Redis keyed by (score, time_penalty) — score = problems solved, penalty = time of first AC. Updated by the judge result event.
  7. Discussion/comments/etc. out of scope.

End-to-end narration: Alice submits her solution → API writes row → Kafka topic judge.python → worker pulls, starts a fresh Firecracker VM (pre-warmed), compiles, runs case 1...30, case 18 fails on wrong answer → worker writes verdict, pushes WS update → Alice's browser shows "WA on test 18, expected [1,2] got [2,1]."

6. Deep Dives (~10 min)

Sandboxing — the centerpiece. Three layers: - Process isolation: Firecracker microVM or gVisor (lightweight VM-class isolation per submission). Cheaper than Docker for security; faster than VMs for startup. - Resource limits: cgroups for CPU (one core), memory (hard cap), pids, file descriptors. Timeout enforced by the worker, not by the guest. - No network, no host filesystem. seccomp denies all networking syscalls; root filesystem is read-only with a small tmpfs scratch. - Submission and result piped in/out via stdin/stdout — no shared volumes.

Why Firecracker/gVisor over plain Docker: Docker with seccomp and cgroups is fast but has a larger shared-kernel attack surface. Firecracker gives VM-class isolation with acceptable startup if pre-warmed; gVisor is a middle ground. For hostile code, I would spend extra milliseconds on isolation.

Judge path trace: API writes submission:{id} as queued and publishes to judge.{language}.{priority}; worker pulls, fetches code and test manifest in parallel, starts a pre-warmed sandbox, compiles, runs cases sequentially with CPU/memory limits, streams progress events, writes terminal verdict with compare output, then emits judge.result for leaderboard updates.

Hot submission path, step by step: 1. Client sends POST /v1/submissions with an idempotency key; API shards submission:{submission_id} by the generated id and writes queued durably before returning. 2. API publishes only {submission_id, language, priority} to judge.{language}.{priority}; code stays in the submission store so queue messages remain tiny. 3. Worker pulls from the language/priority partition, reads code and test manifest in parallel, and checks compile:{lang}:{hash}. 4. If the compile cache hits, it starts execution immediately; otherwise it compiles inside the sandbox image and may write a cache object. 5. Worker runs tests in deterministic order, stops on first visible failure for normal problems, writes the terminal verdict, and emits judge.result for leaderboard and websocket subscribers.

Cold path: a cache-miss worker fetches the language image, hidden tests, and compiler toolchain from object storage; this is why contest pools keep pre-warmed sandboxes per language. If the websocket subscriber is gone, the submission row remains the source of truth and the client can poll GET /v1/submissions/{id}.

Queue and store choice defense: Kafka is attractive for partitioned, replayable contest traffic; SQS is acceptable for simpler operation but weaker for ordered replay and custom priority. Redis fits leaderboards because each contest is a small, hot sorted view derived from submissions; the submission database remains authoritative. A relational database can store problems and contests, but submission history should be sharded by submission_id because writes dominate.

Worker pool sizing. Each VM is short-lived (~2 s). At 60 QPS steady, ~120 concurrent workers; contest peak 330 QPS → ~700 workers. Pre-warm 50 spare VMs per language at all times to absorb bursts; autoscale on queue depth.

Fairness during contests. Deterministic test machine class for contest submissions — same instance type, same OS image. Worker only judges one submission at a time (no CPU sharing). Time limits are wall-clock plus a generous slack to absorb scheduler jitter.

Verdict-stream UX. Client opens a WS to /submissions/{id}/stream after submitting. Worker can publish progress (compiling, running case 5/30) for long-running submissions. Cleaner than polling.

Compiler caching. Compilation is the slow part for C++ submissions. Cache compiled binaries by hash(code + language + compiler flags). On a re-submit of identical code, skip the compile step.

Cheating detection. Plagiarism via code similarity hashing across recent submissions for the same problem; flag for review, don't auto-penalize. Rate-limit submissions per user (token bucket).

Leaderboard at contest scale. 100k concurrent submitters → 100k judge results in 5 min. Redis ZADD is fine for the rate; sharding by contest_id is natural since each contest is independent.

Leaderboard update path: judge.result -> contest service checks first accepted submission per {contest_id,user_id,problem_id} in the submission index -> computes score and penalty -> ZADD contest:{contest_id}:lb -> cache top page for 1 s. Practice submissions bypass this path.

Submission state machine:

Stage Mode Source Transition User-visible state
Queued durable wait submission DB + judge queue worker claims message "Queued" with ETA from queue depth
Compiling isolated build sandbox + compiler cache compile success/fail "Compiling"
Running test execution sandbox + hidden tests first failure or all pass progress optional
Retrying infra worker/sandbox lost submission row + queue replay retry once with same id still pending, no duplicate score
Terminal verdict written submission DB AC/WA/TLE/MLE/RE final verdict + stats

Two-view framing. The submission row is the write-ahead truth, while the websocket stream and leaderboard are derived views. If the stream or Redis leaderboard is lost, replaying judge.result from terminal submission rows can rebuild the user-visible views.

Edge stress tests. Contest spikes route to dedicated contest queues so practice traffic degrades first. A hot problem with a large hidden-test file should be pinned in worker-local cache; a cold miss pays object-store latency but stays under the contest p95 by prewarming. Malicious code gets no network, capped processes, read-only filesystem, and a hard wall-clock kill. If a user paginates deep into submission history, the API switches from hot recent partitions to archived rows using the {last_submission_id, created_at} cursor. Multi-region fairness is handled by judging a contest in one region/machine class and replicating results outward.

Graceful degradation.

Case Mode Source User-visible behavior
Contest spike Dedicated contest queues and workers judge queue practice latency rises first
Language worker exhausted Queue by language, show ETA queue depth verdict delayed, submission durable
Test object cache miss Fetch from encrypted object store test store first judge slower
Sandbox lost mid-run Mark attempt lost and retry once worker heartbeat + submission DB no lost submission
Leaderboard lag Show verdict immediately, rank catches up submission DB + Redis LB rank lags 5-10 s

Interview punch line: "The source of truth is the submission row; everything else, including the contest leaderboard, is a derived view fed by judge results."

What I'd defer. Problem authoring tools, contest registration, discussion threading, language interop (custom judges, interactive problems), premium subscription billing.
