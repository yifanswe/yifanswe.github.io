GitHub Actions — CI Runner Platform (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build GitHub Actions. Developers commit code; a workflow YAML defines jobs that run on ephemeral VMs/containers; results report back. 100M repos, ~10M jobs/day, peak ~100k jobs/min, jobs vary from 30 s to 6 hours, on hosted Linux/macOS/Windows VMs.

1. Requirements (~5 min)

Functional - Developer pushes commit → matching workflows trigger jobs. - Each job runs in an isolated, ephemeral environment. - Stream logs in near-real-time; final status reported back to GitHub UI / API. - Support matrix jobs (multiple OS + version combos).

I'd defer self-hosted runners, complex artifact storage, and advanced dependency caching beyond mentioning.

Non-functional - Isolation — running arbitrary user code; must not leak across jobs or to our infra. - Job start latency p95 < 30 s (queue + provision). - Sustain 100k jobs/min peak, ~1500 concurrent VMs starting. - Durable logs and status; never lose a build result. - Multi-OS support.

Capacity - 10M jobs/day × avg 15 min = ~150M VM-minutes/day. With 24×60 = 1440 min/day, that's ~100k concurrent VMs at any time. - Logs: 10M jobs × ~5 MB ≈ 50 TB/day. - Metadata is small compared with logs: run/job rows at ~2 KB/job × 10M/day × RF=3 are ~60 GB/day, while compressed logs at 5 TB/day with 90-day retention are ~450 TB. Artifacts can dwarf both, so enforce TTL and quota early.

2. Core Entities (~2 min)

  • Workflow — YAML file in the repo.
  • WorkflowRun — run_id, repo, commit, status, triggered_by.
  • Job — job_id, run_id, name, runs_on, status, started_at, ended_at.
  • Step — sub-units within a job, log lines streamed.
  • Runner — VM/container available to execute a job.

3. API (~5 min)

// Internal triggers:
POST  /v1/runs                       { repo, commit, workflow }  → { run_id }

// Runner protocol (long-poll / WS):
GET   /v1/runner/jobs                 → { job_id, image, steps[] }  // pulls a job
POST  /v1/runner/jobs/{id}/logs       (streaming)
POST  /v1/runner/jobs/{id}/status     { state, exit_code }

// User-facing:
GET   /v1/runs/{id}                   → run + jobs + statuses
WS    /v1/jobs/{id}/logs              → stream logs to UI

Runner status updates are idempotent on {job_id, attempt, state_seq}. Log upload chunks carry {job_id, attempt, offset} so retries do not duplicate log bytes.

4. Data Flow (~5 min)

Commit event → workflow trigger → parse YAML → enqueue jobs → runner picks up → provision VM → execute steps → stream logs → write status → mark complete → emit checks back to GitHub.

5. High Level Design (~10–15 min)

  1. Webhook ingestion. GitHub events (push, PR) arrive at our service. Parse the affected workflow YAML for the repo. Materialize one WorkflowRun and N Jobs.
  2. Job queue. Per-runner-class queues: linux-ubuntu, linux-self-hosted, macos, windows. Priority based on plan tier (paid > free). Implemented as Kafka topics with shard keys per OS.
  3. Runner provisioner. Manages a pool of ephemeral runners. For Linux: Firecracker/Kata microVMs spun from a pre-baked image. For macOS: dedicated Mac hardware (real machines), pooled. For Windows: pre-warmed VMs. Pre-warm to absorb start-time latency.
  4. Runner agent. Inside each runner, a small agent polls the job queue, downloads job spec, executes steps, streams logs.
  5. Log streaming. Agent batches log lines (~1 s windows), POSTs to log service. Log service writes to a streaming log store (object storage with append-style writes; or a logs cluster like ClickHouse).
  6. Status service. Owns job state machine: queued → in_progress → success | failure | canceled. Emits events on transition (used by UI and back to GitHub Checks API).
  7. Artifact storage. Optional artifacts produced by jobs uploaded to object storage with TTL.
  8. Secrets injection. Per-repo encrypted secrets stored in a KMS-backed vault. Runner agent receives only the secrets the job needs, decrypted in-VM, redacted from logs.
  9. Concurrency limits per org. Max concurrent jobs per plan tier; surplus queues.

End-to-end narration: Alice pushes to main → webhook → workflow parsed, 4 jobs created (lint, test-linux, test-mac, build) → jobs enqueued in respective OS queues → runner provisioner spins up a Firecracker microVM with the Linux image, hands it the lint job → runner agent boots in 8 s, clones repo, runs steps, streams logs → on completion, status set to success, VM destroyed. Meanwhile, tests run in parallel on other VMs.

Concrete stores:

Run creation path:

  1. Trigger service receives the push event, reads workflow YAML at the commit SHA, and expands matrix jobs.
  2. In one DB transaction, it writes run:{repo_id}:{run_id} and job:{run_id}:{job_id} rows in queued state.
  3. It publishes one queue message per job to jobs.{os}.{priority} with {job_id, repo_id, image, attempt=1}; queue partition key is (os, priority, repo_id) to spread large organizations.
  4. Status service emits the initial Checks API state and the UI can now read the run even before a runner exists.

Runner claim path:

  1. Provisioner watches queue depth and idle pool size for each OS; if jobs.linux.paid grows, it boots or reserves Linux microVMs.
  2. Runner agent long-polls /runner/jobs; scheduler checks org concurrency tokens and assigns a job attempt.
  3. Scheduler writes runner:{runner_id} lease with 30 s TTL and moves the job to in_progress using compare-and-set on {job_id, attempt}.
  4. Runner fetches secrets, repo checkout metadata, and cache manifests in parallel; then steps execute and logs stream by chunk offset.
  5. Terminal status updates the Job DB, releases the org token, emits UI/Checks events, and destroys the VM.

Capacity by store: Run DB is ~10M jobs/day × 2 KB × RF=3 = ~60 GB/day, or ~5.4 TB for a 90-day hot window. Queue traffic at 100k jobs/minute is ~1.7k messages/sec; at 1 KB/message and RF=3, that is only ~5 MB/sec, so queue throughput is not the dominant cost. Logs are: 50 TB/day raw, roughly 5-10 TB/day compressed, and 90 days is 450-900 TB in object storage. Runner leases are tiny, ~100k active runners × 200 B = 20 MB, but heartbeat correctness matters. Artifacts can dwarf logs; even 20% of jobs uploading a 50 MB artifact is 100 TB/day, so TTLs and per-org quotas are product requirements, not cleanup details.

6. Deep Dives (~10 min)

Isolation — the centerpiece. Each job runs in a fresh ephemeral VM, never reused. Firecracker (or Kata Containers) provides VM-class isolation with container-class startup time (~1–2 s). After the job, the VM is destroyed. No filesystem persistence between jobs.

Why not plain containers: they start fastest, but arbitrary customer code makes kernel sharing a scary default. Why not full cloud VMs only: stronger isolation, but cold-start and cost are worse at 100k concurrent jobs. MicroVMs are the middle: hardware isolation boundary, image pre-warming, and fast teardown. For macOS, the alternative is not Firecracker; it is a finite pool of real Mac hardware, so scheduling policy matters more than virtualization cleverness.

Pre-warmed pool. Cold-boot of a runner is ~10–15 s. To hit p95 < 30 s job start, keep a pool of N pre-warmed runners idle per OS class. Sized to absorb bursts; auto-scales on queue depth.

Provisioning path trace: scheduler writes job_id to jobs.linux.paid; provisioner watches queue depth and keeps idle microVMs in runner_pool:linux; runner long-polls, receives the job spec, writes a lease heartbeat every 10 s, streams logs by chunk offset, then posts terminal status. Queue pull, secrets fetch, and repo checkout start metadata can run in parallel after the runner is assigned.

Job scheduling. Per-OS queues; per-org concurrency limits enforced at dequeue. A runner that pulls a job checks "does the org still have capacity?" — if not, returns the job and picks another. Better: enforce at the scheduler before dequeue, with org-level token bucket.

Why Kafka-like durable queues rather than Redis lists: Redis is simpler for a small worker pool, but this workload needs replay, per-OS partitioning, plan-tier priority, and auditability when a result is disputed. SQS/PubSub also work; Kafka is easiest to explain when ordering and replay matter.

macOS scarcity. Mac hardware is finite (legally required to run on Apple silicon). Smaller pool, longer queue times, often a tier surcharge. Acknowledge this is a real product constraint.

Log streaming and storage. Each step's stdout streams from the runner via batched POSTs (every 1 s). Logs land in a hot store for the live UI; after job completion, flushed to object storage with retention (e.g., 90 days). 50 TB/day raw → compressed ~10×.

Live log read path: browser WS -> log service subscribes to log_tail:{job_id}; in parallel it fetches existing chunks from the hot store starting at the requested offset; new chunks arrive from the runner and are pushed to all subscribers; after completion the same API reads from object storage by byte range.

Two views of logs and jobs. The Job DB is the durable status view; the queue is only the work-to-do view and may contain duplicates. The log hot store is the live-tail view; object storage is the durable archive. The UI should tolerate switching sources at completion by using {job_id, attempt, offset} as the cursor, so a browser refresh can continue from hot chunks or archived byte ranges without duplicate lines.

Live log fan-out. When a user is watching a running job, the UI WS subscribes to log events for that job. Log service holds the open connection and pushes new chunks as they arrive. Browser sees logs ~1 s after they happen.

Secrets handling. Repository secrets encrypted at rest (KMS). When provisioning a runner for a job, fetch only the secrets referenced by the job, send over TLS, decrypt in the VM's memory, never write to disk. Log scrubber masks secret values in stdout.

Cancel / re-run. Cancel sends a signal to the runner's job; agent stops the step, marks status. Re-run: same job spec re-enqueued; fresh VM; idempotent on re-run id.

Caching (touched). Dependency caches (node_modules, .gradle) are huge. Per-repo cache service stores key → blob in object storage. Jobs declare cache keys; on hit, runner downloads the blob before steps run. Cache eviction is age + size based.

Matrix jobs. YAML strategy.matrix expands into N jobs, scheduled independently. Failed jobs don't block siblings unless fail-fast: true is set.

Failure modes. Runner dies mid-job: status timer fires after lack of heartbeat → job marked failed: lost runner → user sees the failure with a clear cause. Optionally auto-retry once.

Duplicate queue delivery is expected: the runner must acquire {job_id, attempt} with compare-and-set before executing. If two runners race, one wins and the loser drops the job. If a region fails, the scheduler replays queued jobs in another region, but in-progress jobs are marked lost_runner because their VM state is not portable. Cold image starts are handled by pre-warmed pools and image-layer caches; if those are empty, the job remains queued with an ETA rather than starting on a reused VM. Secrets-service degradation fails closed: jobs that need secrets stay queued or fail explicitly, while public jobs can continue.

Graceful degradation.

Case Mode Source Latency / behavior
Linux burst Drain pre-warm pool, then cold-start runner pool + queue p95 may rise toward 30 s
macOS exhausted Queue by plan tier, show ETA mac runner queue minutes, but explicit
Log hot store down Buffer chunks on runner disk and Kafka runner local spool live logs lag, final logs durable

Secrets service degraded Do not start secret-requiring jobs vault/KMS fail closed

Interview punch line: "For CI, isolation and durable status beat raw scheduling cleverness; every optimization has to preserve the fact that hostile code ran in a disposable box and still produced an auditable result."

Self-hosted runners (briefly). Customers run our runner agent on their hardware. It long-polls our job queue with their token; runs jobs in their environment. Same protocol, their isolation responsibility.

What I'd defer. Reusable actions ecosystem and registry, full marketplace, organization billing, compliance certifications (SOC 2), advanced workflow composition (reusable workflows).
