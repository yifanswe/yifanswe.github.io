Hadoop Distributed File System (HDFS / GFS)

2026/06/01  ·  Common Backend Systems
Contents
Contents

New to these terms? See the Staff System Design Glossary. HDFS is the open-source reimplementation of Google File System (GFS); the design is essentially the same.

1. Basic Knowledge And Solution

What This Question Is Testing

HDFS / GFS is the canonical distributed file system for large data: petabyte-scale storage, batch read/write of large files, commodity hardware tolerance. The hard part is achieving durability and high throughput on cheap, failure-prone hardware while keeping the metadata service simple.

The invariant is this:

Every file written to HDFS is stored as a sequence of blocks; each block is replicated across multiple datanodes (typically 3); reads can come from any replica; the namespace and block locations are tracked by a namenode; the system tolerates routine datanode failure without data loss.

At ICT5 / L6+ level, the interviewer is testing whether you can articulate the namenode/datanode split, block replication strategy, write pipelining, the original single-namenode bottleneck and how HA / federation address it, and the workload assumptions (append-mostly, large files).

60-Second Interview Framing

Spoken version:

I would describe HDFS as a namenode plus a fleet of datanodes. Files are split into large blocks (128 MB typical); each block is replicated to three datanodes across racks. The namenode owns the filesystem namespace and block-to-datanode mapping. Datanodes store and serve blocks. Writes pipeline through replicas. The design assumes append-mostly large-file workloads. The interesting decisions are block size, replication placement, the namenode bottleneck, and HA via standby namenode.

Requirements The Paper Sets

Functional requirements:

  • Hierarchical filesystem namespace.
  • Files are sequences of blocks; large files dominant.
  • Read and append (no random write in original GFS / HDFS).
  • High throughput batch reads/writes.
  • Replication for durability.

Non-functional requirements:

  • Petabytes of data.
  • Thousands of datanodes.
  • Commodity hardware (frequent disk/server failures).
  • High aggregate throughput.
  • Simple coherence model.

Out of scope of the original design:

  • Low-latency small reads.
  • Random in-place writes.
  • POSIX compliance.

Core Concepts

Blocks

A file is split into blocks of fixed size (default 128 MB in HDFS).

  • Large block size reduces metadata pressure on namenode.
  • Each block stored as a Unix file on a datanode's local disk.
  • Blocks are first-class objects in HDFS.
  • Replication factor specified per file (default 3).

Namenode (Master)

Single namenode (originally; HA later) holds the metadata:

  • Namespace: directory tree, file metadata, permissions.
  • Block-to-datanode mapping (constructed at startup from datanode heartbeats).
  • Operations log (EditLog) and image of the namespace (FsImage).

All metadata is in memory for speed. EditLog persists changes; FsImage is periodic checkpoint.

Single namenode is a SPOF and a scale ceiling. HDFS HA via standby; HDFS Federation splits namespaces across multiple namenodes.

Datanodes (Workers)

Each datanode:

  • Stores blocks on local disk (one file per block).
  • Reports stored blocks to namenode periodically (block reports).
  • Heartbeats to namenode regularly.
  • Serves reads and writes from clients.

Replication Placement

For replication factor 3: - Replica 1: usually on the writer's local node (if datanode) or a random node. - Replica 2: on a different rack from replica 1. - Replica 3: on the same rack as replica 2 (different node).

This balances: - Durability (rack failure tolerated). - Write bandwidth (one rack-crossing write). - Read locality (most blocks reachable in one rack).

Write Path

  1. Client requests to create a file: create(/path/to/file).
  2. Namenode creates file metadata, returns lease to client.
  3. Client writes data; buffered locally until a block-size chunk is ready.
  4. Client requests addBlock from namenode.
  5. Namenode picks 3 datanodes (per placement policy), returns to client.
  6. Client opens a pipeline: client → datanode 1 → datanode 2 → datanode 3.
  7. Data flows through the pipeline as small packets (~64 KB).
  8. Each datanode persists locally and forwards to next.
  9. Acks flow back through the pipeline.
  10. Once all three ack, block is committed.
  11. Repeat for subsequent blocks.
  12. On close: client tells namenode; namenode finalizes file.

Interview phrase:

Writes pipeline through datanodes so we only consume one cross-rack bandwidth per replica trio. No central coordinator on the data path.

Read Path

  1. Client opens file: namenode returns block IDs + replica locations.
  2. For each block, client picks the closest replica (rack-aware).
  3. Reads directly from that datanode.
  4. If datanode unreachable or block corrupt, try next replica.

Reads bypass the namenode entirely after the initial metadata fetch.

Append

HDFS supports append after a file is closed: re-open in append mode; namenode grants lease; new data pipelined to the file's last block (or new block).

Original GFS supports atomic record append: many writers can concurrently append; system serializes. HDFS append is single-writer.

Replication And Failure Handling

Each datanode periodically heartbeats. Namenode flags a datanode dead if no heartbeat for ~10 minutes.

On datanode death: - All blocks that datanode held are under-replicated. - Namenode schedules re-replication: pick another datanode, copy the block. - Targets per-block replication factor.

Block integrity: - Datanode periodically scans blocks for checksum failures. - Bad blocks reported to namenode; namenode triggers re-replication.

Namenode HA

Original single namenode is a SPOF.

HDFS HA solution: - Active + standby namenodes. - Shared edit log via JournalNodes (Quorum-based). - Active writes to JournalNodes; standby reads and applies. - ZooKeeper or similar coordinates failover. - On active failure, standby promotes within seconds.

Namenode Federation

Single namenode's memory caps the cluster scale (one entry per file + block).

Federation: - Multiple namenodes, each owning a subset of the namespace. - Datanodes report to all namenodes. - Clients route requests to the right namenode based on path. - Linear namespace scaling.

Secondary Namenode

Despite the name, NOT a hot standby. It periodically merges EditLog into FsImage to keep checkpoint files small. Without it, EditLog grows unbounded, slowing startup.

Rack Awareness

Datanodes are configured with rack info. The namenode uses rack info for: - Replica placement (across racks). - Read locality (prefer closer replicas). - Re-replication (avoid placing two replicas on same rack).

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Datanode dies. Re-replicate from surviving replicas. Under-replicated block count.
Disk failure. Block scanner detects; re-replicate. Bad block count.
Network partition. Datanode marked dead; later re-discovered with stale blocks. Stale datanode count.
Namenode dies (no HA). Cluster unavailable until restart. Namenode uptime.

Slow datanode in pipeline. Pipeline blocks; client may retry. Slow write detection.
Hot read. Cache; increase replication on the file. Per-file read rate.
Block-report storm. Stagger; throttle. Block report rate.
Imbalanced disk usage. Rebalancer. Disk usage spread.

Operational Metrics

Correctness:

  • Under-replicated blocks.
  • Missing blocks (zero).
  • Stale block reports.

Capacity:

  • Cluster used vs free.
  • Per-datanode utilization.
  • Namenode memory.

Business:

  • Files; total size.
  • Read/write throughput.

Tradeoffs To Call Out

  • Large block size reduces namenode load but wastes space for small files.
  • Single namenode is simple but a SPOF/bottleneck; HA + federation fixes both.
  • Append-only model simplifies coherence; random write would need locking.
  • Cross-rack write costs bandwidth but provides rack-failure resilience.
  • 3x replication is durable but expensive; erasure coding offers space savings at compute cost.
  • Strong consistency on namespace but eventual block-location knowledge.

Common Traps And Stronger Moves

Trap Stronger move
Treating HDFS like a POSIX FS. Append-only; large files; no random writes.
Many small files. Concatenate or use HAR/SequenceFile.
Single namenode without HA. Standby namenode + JournalNodes.
Random replica placement. Rack-aware.
Single namespace forever. HDFS Federation as scale increases.
No erasure coding for cold data. EC for cold tier; 3x for hot.

2. Simulated Interview

Interviewer: Describe HDFS.

Candidate: Master-slave design: namenode for metadata, datanodes for data. Files split into 128 MB blocks; each replicated 3x across racks. Namenode holds the namespace and block-to-datanode mapping in memory; datanodes heartbeat block reports periodically. Clients talk to namenode for metadata, then directly to datanodes for data. Writes pipeline through replicas. Reads come from closest replica. Append-only; designed for petabytes of large files on commodity hardware.

Interviewer: Walk me through a write.

Candidate: Client calls create on namenode; gets a lease. As the client fills a 128 MB block, it calls addBlock; namenode picks 3 datanodes (rack-aware: 1 local, 2 on a different rack). Client opens a pipeline — client → datanode 1 → datanode 2 → datanode 3. Data flows as small packets; each datanode persists locally and forwards. Acks flow back. Block committed once all 3 ack. Subsequent blocks repeat. Close finalizes the file with the namenode.

Interviewer: Block placement?

Candidate: Rack-aware. Replica 1 typically on the writer's local node (or random if writer isn't a datanode). Replica 2 on a different rack from replica 1. Replica 3 on the same rack as replica 2, different node. Balances durability (rack failure tolerated) with bandwidth (one cross-rack write) and read locality.

Interviewer: Why large blocks?

Candidate: Reduces namenode metadata. Namenode holds one entry per file + per block; with 4 KB blocks, namenode metadata explodes. With 128 MB blocks, a TB file is just 8000 blocks. Tradeoff: small files waste a block worth of metadata each. HDFS doesn't handle many small files well.

Interviewer: Namenode is a SPOF.

Candidate: Originally yes. HA solution: active + standby namenodes. Shared edit log via JournalNodes (Quorum-based — write to majority of journals before ack). Active writes EditLog entries to journals; standby reads them and replays. ZooKeeper or similar coordinates failover. On active failure, standby promotes within seconds.

Interviewer: Scaling namenode beyond memory.

Candidate: HDFS Federation. Split the namespace across multiple independent namenodes — e.g., /user on namenode 1, /data on namenode 2. Each namenode owns its subset; datanodes report to all namenodes. Clients route by path. Linear namespace scaling.

Interviewer: Read path.

Candidate: Client asks namenode for the file's block IDs and replica locations. For each block, client picks the closest replica (rack-aware), reads directly from that datanode. If unreachable or checksum fails, try next replica. The namenode is out of the data path entirely once metadata is fetched.

Interviewer: Datanode dies.

Candidate: Namenode flags it dead after missing heartbeats for ~10 minutes. All blocks on that datanode are now under-replicated. Namenode schedules re-replication: for each block, pick a healthy datanode, copy from a surviving replica. Continues until target replication factor restored. Background; doesn't block reads/writes.

Interviewer: Append?

Candidate: HDFS supports append after a file is closed: re-open in append mode; namenode grants a write lease; new data pipelined as if continuing the file. Single writer per file. Original GFS supports atomic record append — many concurrent appenders, system serializes — HDFS doesn't.

Interviewer: Small files problem.

Candidate: Each file + block takes namenode memory. Many small files explode the namenode metadata. Mitigations: HAR (Hadoop Archive) concatenates small files; SequenceFile or Avro container files store many records in one HDFS file; HBase as a layer above HDFS for many small records.

Interviewer: Erasure coding?

Candidate: Reed-Solomon style. Instead of 3x replication, encode each block into k+m chunks; tolerate m losses. E.g., 6+3 gives 1.5x overhead vs 3x. Used for cold data; hot data stays on 3x replication for read parallelism. Adds compute on writes (encoding) and reads (when reconstruction needed).

Interviewer: Block integrity.

Candidate: Each block has a checksum. Datanode's BlockScanner periodically reads blocks and verifies checksums. On corruption: report to namenode; trigger re-replication from another replica. Reads also verify checksums; on mismatch, try another replica.

Interviewer: Summarize.

Candidate: Namenode (metadata) + datanodes (data). Large blocks reduce metadata. Rack-aware 3x replication. Pipeline writes. Read closest replica. Append-only. HA via standby + JournalNodes; federation for namespace scale. The strengths are scale, simplicity, and resilience to commodity hardware failure; the weaknesses are small files and the inherently single-writer model.

3. Practice Notes

Five-Minute Version

  1. Namenode + datanodes.
  2. Large blocks (128 MB), 3x replication.
  3. Rack-aware placement.
  4. Pipeline writes; read closest replica.
  5. Append-only large files.
  6. Single namenode SPOF → HA via standby + journals.
  7. Federation for namespace scale.
  8. Small files problem; erasure coding for cold.

One-Minute Closeout

HDFS is master + workers: namenode holds metadata, datanodes hold data. Large blocks (128 MB) and 3x rack-aware replication. Writes pipeline through replicas. Reads bypass namenode. Append-only model. Original namenode SPOF resolved by HA (standby + JournalNodes); namespace scale resolved by federation. The strengths are scale on commodity hardware and resilience to disk failure; the weaknesses are small files and lack of random writes.
