1. Basic Knowledge And Solution

What This Question Is Testing

Typeahead is a deceptively-simple-looking feature: as the user types, suggest completions ranked by popularity. The hard part is sub-50ms latency on every keystroke, ranking based on shifting popularity, supporting personalization, and updating the suggestion corpus continuously.

The invariant is this:

For every keystroke, the system returns relevant completions ranked by a combination of popularity, recency, and personalization within tens of milliseconds; the suggestion corpus reflects recent query trends within bounded staleness; misspellings and typos still produce relevant results.

At ICT5 / L6+ level, the interviewer is testing whether you can design a trie or inverted-index structure for prefix lookup, scale it across regions, integrate freshness from query logs, handle misspellings, and reason about the latency budget per keystroke.

60-Second Interview Framing

Spoken version:

I would design typeahead as a prefix-indexed structure (trie or inverted prefix index) backing a low-latency suggest service. As the user types, each keystroke fires a request that returns top-N completions. Background pipelines aggregate query logs to update suggestion popularity. Personalization adjusts rank at the edge. The interesting decisions are the index data structure, freshness pipeline, fuzzy/typo handling, and latency at scale.

Requirements To Clarify First

Functional requirements:

  • User types into a search box; suggestions appear after each keystroke.
  • Suggestions are ranked by popularity, freshness, and personalization.
  • Top 5-10 suggestions returned.
  • Support typos and misspellings.
  • Multi-language.

Non-functional requirements:

  • Sub-50ms latency per request (target for "instant" feel).
  • Hundreds of thousands of requests per second peak.
  • Suggestion corpus updated daily or hourly with new query trends.
  • Multi-region serving.

Out of scope unless asked:

  • Showing actual search results below suggestions (separate search).
  • Knowledge-graph-style entity completions.
  • Voice-driven suggestions.

Scale Assumptions

Area Assumption
Queries indexed Hundreds of millions of distinct queries.
QPS 1M+ keystrokes/sec globally.
Latency budget 30-50 ms p99 from server.
Index size Few GB compressed in memory.
Update frequency Hourly batch + real-time delta.
Per-user personalization Lightweight; cached.

Bottleneck: serving latency at high QPS plus distinct-suggestion index size.

Core Product Model

Personalization Re-ranks for the user. Optional; lightweight.
Fuzzy match Handles typos. Edit-distance or n-gram.

Data Model

Suggestion

  text,
  popularity_score,
  category,
  language,
  last_updated
)

Trie Node (logical)

TrieNode(
  prefix,
  top_n_suggestions: [(text, score), ...]
)

Each trie node caches the top-N suggestions for that prefix, so lookup is O(prefix_length) plus reading the cached list.

APIs

GET /suggest?q={prefix}&lang=&user_token=
  -> [
       { "text": "...", "score": 0.95, "highlight": "..." },
       ...
     ]

Headers typically include locale and user context (for personalization).

Prefix Index Design

Two viable structures:

Trie (in-memory):

  • Each node represents a prefix.
  • Each node stores top-N completions for that prefix.
  • Lookup: walk prefix character-by-character; return node's top-N.
  • O(prefix_length) time.
  • Memory-efficient with compression (PATRICIA trie, ternary search trie).

Inverted prefix index:

  • For each suggestion, generate all prefixes; map prefix → list of (suggestion, score).
  • Lookup: prefix key → list, sorted by score.
  • Easy to shard; each shard handles a key range.
  • More storage but predictable distribution.

For top-N at every prefix node, pre-compute and cache:

  • Walk the trie once offline.
  • At each node, take the top-N highest-scored suggestions in its subtree.
  • Store in the node.

Interview phrase:

I pre-compute top-N at every prefix node so lookup is O(prefix_length) regardless of the subtree size. Updates rebuild the affected ancestors when popularity shifts.

Lookup Path

  1. Client emits keystroke "app".
  2. Edge cache hit: return cached suggestions immediately.
  3. On miss, route to suggest service.
  4. Service queries prefix index for "app" → returns pre-computed top-N.
  5. Apply personalization re-rank if user_token present.
  6. Return suggestions, highlight matching part.
  7. Round trip target: under 50ms.

For very-common prefixes ("a", "the"), the CDN handles most traffic.

Personalization

Lightweight, per-user:

  • Stored as a small feature vector (recent queries, click patterns).
  • Cached at the edge near the user.
  • Re-rank: blend popularity score with personal-affinity score.
  • Cap latency impact: <5ms.

Privacy: respect opt-outs; bound retention.

Fuzzy / Typo Tolerance

When the prefix doesn't match anything popular, try edit-distance.

  • Generate candidates within edit distance 1 or 2.
  • Look up each candidate in the index; collect suggestions.
  • Slower than exact match; invoked on low-confidence matches.

Alternative: n-gram index where each suggestion is indexed by its constituent n-grams; query n-gram overlap finds approximate matches.

Freshness Pipeline

Suggestions update over time:

  • Query logs from search service feed an aggregation pipeline.
  • Daily batch: aggregate query counts per term over the last N days.
  • Streaming layer: real-time tracking of breaking trends.
  • Combined: score = popularity_baseline + recent_velocity.
  • Re-index periodically or apply deltas in place.

For breaking events (e.g., "lottery winner"), real-time signals can push a suggestion into the top quickly.

Scaling And Partitioning

Concern Strategy
Prefix index Sharded by first 1-2 characters of prefix.
QPS Stateless service replicas; CDN absorbs bulk.
Memory Compress trie; cap top-N per node; tier rarely-used.
Personalization Edge-cached per user; bounded.
Hot prefix ("a") CDN-cacheable; pre-warm.

For multi-language: separate indexes per language, routed by Accept-Language.

Consistency Model

Layer Consistency Reason
Suggestion popularity Eventually consistent (hourly batch + streaming) Trends move slowly.
Personalization Eventually consistent at edge Bounded staleness.
Index lookup Strong within a replica Read from local in-memory state.
Editorial overrides Strong Manual changes take effect quickly.

Multi-Region Strategy

  • Index replicated to every region.
  • Edge CDN caches common prefixes per region.
  • Personalization per user, regional cache.
  • Updates pushed to all regions from central pipeline.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Index load slow. Pre-warm replicas; staged rollout. Replica readiness.
Suggest service slow. Aggressive timeout; fall back to popular suggestions. p99 latency.
Stale index missing trending term. Streaming layer catches up; editorial push. Trend detection lag.
Personalization service down. Drop personalization; return popularity-ranked. Personalization availability.
Spam suggestion. Filter pipeline; manual review. Spam-suggestion reports.
Typo not resolved. Fuzzy fallback; spell correction. Fuzzy hit rate.
Hot prefix saturation. CDN cache; pre-warm. Hot-prefix QPS.
Multi-language mixup. Locale routing; language detection. Cross-locale results.
Adversarial popularity manipulation. Anomaly detection; baseline floor. Sudden trend signals.
Regional index divergence. Centralized indexing pipeline; replica health. Divergence count.

Operational Metrics

Correctness:

  • Top-N quality (offline evaluation).
  • Stale-trend lag.
  • Personalization accuracy.

Capacity:

  • QPS by region.
  • p99 latency.
  • Index memory per shard.

Business:

  • Click-through rate on suggestions.
  • Suggestion-to-search conversion.

Tradeoffs To Call Out

  • Pre-computed top-N speeds reads but requires rebuild on updates.
  • Tries are memory-efficient but harder to update; inverted index easier to update but uses more memory.
  • Personalization improves relevance but adds latency.
  • Fuzzy matching helps typos but is slow; fallback path.
  • Aggressive caching at edge saves origin cost but staleness for hot trends.
  • More languages = more indexes = more memory.

Common Traps And Stronger Moves

Trap Stronger move
Database lookup per keystroke. In-memory index; sub-millisecond lookup.
Compute top-N per request. Pre-compute at each trie node.
Synchronous personalization. Edge-cached; lightweight blend.
No typo handling. Edit-distance or n-gram fallback.
Single global index. Sharded; multi-language; replicated.
Static suggestions. Continuous freshness pipeline.

Staff-Level Answer Checklist

  • In-memory prefix index (trie or inverted).
  • Pre-computed top-N per node.
  • Stateless suggest service; CDN-fronted.
  • Freshness via batch + streaming pipeline.
  • Edge-cached personalization.
  • Fuzzy fallback for typos.
  • Sharded and replicated globally.
  • Operational metrics for latency, freshness, CTR.

Interview Phrase

Typeahead is sub-50ms prefix lookup with pre-computed top-N suggestions. The trie or inverted prefix index lives in memory; suggest service is stateless and CDN-fronted. Freshness comes from a continuous aggregation pipeline; personalization blends at the edge. The hardest performance constraint is the per-keystroke latency budget; the hardest correctness concern is trend freshness.

2. Simulated Interview

Interviewer: Design typeahead.

Candidate: Sub-50ms per-keystroke suggest service backed by an in-memory prefix index — typically a compressed trie with pre-computed top-N suggestions at each node. Edge CDN caches common prefixes. Background pipeline aggregates query logs to update suggestion popularity. Personalization re-ranks lightly at the edge. The invariant is fast, relevant, fresh suggestions on every keystroke.

Interviewer: Why pre-compute top-N at each node?

Candidate: Lookup latency. If I had to walk the subtree of every queried prefix and rank, it would scale with subtree size — bad for short prefixes like "a" with millions of completions underneath. Pre-computing top-N per node makes lookup O(prefix_length) plus reading a small list. Updates rebuild affected ancestor nodes.

Interviewer: How big is the index?

Candidate: Hundreds of millions of distinct queries, each with metadata. A compressed trie holds prefixes as edges with shared paths; top-N at each node adds memory. Estimate: a few GB to tens of GB compressed in memory. Fits per replica.

Interviewer: How fresh are the suggestions?

Candidate: Two pipelines. Batch: daily or hourly, aggregate query counts over recent windows from logs, update popularity scores, regenerate top-N per trie node. Streaming: real-time signal for breaking trends (e.g., a sudden viral query). Streaming layer applies deltas to the index without full rebuild. So a brand-new viral query can surface within minutes.

Interviewer: Personalization?

Candidate: Edge-cached per-user vector summarizing recent queries and clicks. When a request arrives with user_token, the edge fetches the user's vector and blends a personal-affinity score with popularity score. Cap latency at 5ms. Privacy: bounded retention, opt-out, anonymized.

Interviewer: A user mistypes — "appl" instead of "apple."

Candidate: Exact prefix lookup returns suggestions for "appl" if any exist. If results are sparse or low-confidence, fuzzy fallback: generate candidate prefixes within edit distance 1 ("appe", "apple", etc.), look up each, merge results. The fallback is slower so we only invoke it when needed. Alternative: maintain an n-gram index where each suggestion is indexed by its 2-3-grams; query n-gram overlap finds approximate matches.

Interviewer: "a" gets 100k QPS — how does the system survive?

Candidate: CDN absorbs most of it. The top-N for prefix "a" is small and globally stable; cache it at the edge with a short TTL. Origin sees only the misses. We pre-warm popular prefixes to keep the CDN hot. The suggest service for cache misses has stateless replicas behind a load balancer.

Interviewer: Multi-language?

Candidate: Separate index per language. Locale from request determines routing. Some languages share prefixes (e.g., proper names in Latin script); we handle by indexing in both languages with locale-aware ranking.

Interviewer: How do you ensure trending terms surface fast?

Candidate: Streaming layer monitors query velocity per term. When velocity spikes (e.g., a celebrity event), the term's popularity score is boosted. Streaming applies the delta to the trie's top-N for affected prefixes within seconds to minutes. Editorial overrides exist for known events (e.g., elections).

Interviewer: Spam suggestions?

Candidate: Filter pipeline before indexing. Blocklists, heuristics (very short non-words, gibberish), ML classifiers. Suggestions below a quality floor are excluded. Manual review queue for borderline cases. Anomaly detection catches sudden manipulation attempts.

Interviewer: Multi-region?

Candidate: Index replicated to each region from a central indexing pipeline. Edge CDN per region caches common prefixes. Personalization cache per region. Failover: if a region's suggest service is degraded, route to the next-nearest region or serve from CDN with stale data.

Interviewer: Summarize.

Candidate: In-memory prefix index — trie with pre-computed top-N per node — serves sub-50ms lookups via stateless suggest service fronted by CDN. Background pipeline updates popularity from query logs in batch plus streaming for trends. Personalization blends lightly at the edge. Fuzzy matching handles typos. Sharded per language, replicated per region. The hardest performance constraint is per-keystroke latency; the hardest correctness concern is trend freshness.
