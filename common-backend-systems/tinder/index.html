Tinder — Dating App (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build Tinder. A user opens the app, gets a stack of nearby candidate profiles, swipes left/right, and when two users mutually swipe right they "match" and can chat. 100M daily active users (DAU), median ~50 swipes/day, ~10M new matches/day.

1. Requirements (~5 min)

Functional - Browse a stack of candidate profiles filtered by location, age range, gender preference. - Swipe left (pass) or right (like). - When both users have swiped right, create a match and open a chat thread.

I'd defer chat/messaging implementation depth, premium features (Super Like, Boost), profile photo moderation.

Non-functional - Card-stack open p99 < 300 ms; nothing kills the experience like a slow next-card. - Strong consistency on match detection — both users must see the match exactly once. - A user must never see the same candidate twice (no re-shows). - Read-heavy: 100M × 50 = 5B swipes/day → ~60k swipes/sec average; candidate fetches roughly 1:5 with that.

Capacity - 100M DAU × 50 swipes = 5B swipes/day. - Match table: ~3.6B matches lifetime. - Profile store: 200M profiles total at 5 KB each ≈ 1 TB. - Swipe write volume: 5B/day × ~40 B compact row (from,to,dir,ts) ≈ 200 GB/day before replication; with 3 replicas and 1-year retention, this is ~220 TB. The dominant hot cost is write throughput and per-user exclusion state, not profile storage. - Active geo index: if 30M users are active in the last 30 min and each entry is ~24 B, the live location index is <1 GB primary; it is small but extremely latency-sensitive.

2. Core Entities (~2 min)

  • User / Profile — user_id, location, age, photos, preferences.
  • Swipe — from, to, direction (like/pass), ts. Append-only.
  • Match — user_a, user_b, created_at, chat_id. Symmetric pair.
  • Chat thread — out of scope to design fully; just touch the entry point.

3. API (~5 min)

GET   /v1/cards?lat=&lng=&n=10               → Profile[]    // candidate stack
POST  /v1/swipes                              { target_id, direction }  → { matched?: bool, match_id?: ... }
GET   /v1/matches                             → Match[]
POST  /v1/location                            { lat, lng }

User from token. Swipes are idempotent by (swiper, target).

The cards cursor is server-issued and encodes user_id, geo_cell, candidate_batch_id, and rank_offset; it prevents duplicate cards across retries without exposing ranking internals.

5. High Level Design (~10–15 min)

  1. Profile service. Sharded by user_id. Read-heavy; fronted by a per-profile cache.
  2. Location service. User reports location → write to a geo-index (S2 cells; Redis or a geo-aware store like ElasticSearch).
  3. Candidate generation. For a card request, query nearby users in the user's S2 cell (and adjacent cells) with matching age/gender preferences. Subtract already-swiped users. Cap at ~50 candidates, rank by a lightweight quality model.
  4. Swipe service. Write the swipe row (sharded by swiper_id). If direction = like, check whether (target, swiper, like) exists — if so, create a match.
  5. Match detection. Two paths: (a) synchronous on swipe — single read of the reciprocal swipe; if found, write match row; (b) async via Kafka if we don't want to add a read on the swipe path. I'd default to sync for the snappy "It's a Match!" UX.
  6. Match store. Symmetric — store one row keyed by sorted (min_id, max_id) to avoid duplicate matches; unique constraint enforces "match at most once."
  7. Chat service. When a match is created, provision a chat thread (just a chat_id); messages are out of scope.

End-to-end narration: Alice opens the app → cards request → candidate gen finds 50 nearby singles, subtracts Alice's 1000 prior swipes, returns top 10. She swipes right on Bob → swipe service writes the row, looks up (Bob, Alice, like) → not found, just write swipe. Later Bob swipes right on Alice → swipe service writes, looks up (Alice, Bob, like) → found. Insert match row (unique constraint), return matched=true. Both phones get a push.

6. Deep Dives (~10 min)

Concrete store contracts.

Store Key / data structure Entry size + TTL Shard key
Profile profile:{user_id} document in KV store + CDN photo URLs ~5 KB, retained until delete user_id
Active geo index geo:{s2_cell} sorted set of user_id, score=last_seen ~24 B/member, TTL 30 min s2_cell
Swipe log swipes_from:{swiper_id}:{month} append rows ~40 B/row, 1–2 years hot/warm swiper_id
Reciprocal-like index likes_to:{target_id} set of swiper_id ~8–16 B/id, same retention as swipes target_id
Already-shown filter shown:{user_id} bloom/cuckoo filter 16–64 KB/user, rolling 6–12 months user_id
Match match:{min_user}:{max_user} unique row ~80 B, retained forever sorted pair hash

This is the two-index pattern: the same swipe event is stored once as the user's history and once as a reciprocal lookup. The first supports "what have I done?"; the second supports "did they already like me?"

Store math: swipe history at 5B/day is the heavyweight write path, roughly 200 GB/day compact and ~600 GB/day at RF=3 before indexes. Per-user shown filters are the sneaky RAM cost: only active users should be loaded, e.g., 30M active × 32 KB is ~1 TB if kept hot, so keep it in a tiered cache and rebuild from swipe logs for dormant users.

Cards path trace. Client → cards API → read user preferences/profile → query geo:{cell} plus adjacent cells in parallel → remove users from shown:{user_id} and block lists → hydrate top ~50 profiles from profile cache in parallel → rank → return top 10 with a cursor → mark returned candidates in shown:{user_id}. If profile hydration misses cache, fetch from profile KV; if too many misses, return fewer cards rather than exceeding 300 ms.

Swipe path trace. Client sends (target_id, direction, client_uuid) → idempotency check on (swiper,target) → append to swipes_from:{swiper} → update shown:{swiper} → if like, check likes_to:{swiper} for target_id in parallel with writing likes_to:{target} → if reciprocal exists, insert match:{min,max} unique row and provision chat id. Duplicate concurrent matches lose the unique insert and read back the winning match row.

Named tech choice: DynamoDB/Cassandra-style wide rows fit swipes_from:{swiper} because the write volume is high and access is append-by-user. A relational store can still own match:{min,max} because uniqueness and exactly-once match creation matter more there. Redis/S2 active sets are for candidates; they are not the source of truth for swipes.

Never showing the same profile twice. Naive: subtract a list of swipes from candidates — expensive at 50 swipes/day × users. Better: maintain a bloom filter per user of "already shown," small enough to keep in-memory in the candidate service. Bloom false positives just mean a profile occasionally doesn't show; never the reverse.

Interview punch line: "For candidate generation, a false positive is acceptable because skipping one possible profile is cheap; showing the same profile twice is the product bug."

Geo-search at scale. S2 cells at appropriate level (e.g., level 12, ~1 km²). Index cell_id → set of recently-active user_ids. Query: nearby cells (1 + 8 ring) for users matching preference filters. Refresh user-to-cell mapping every location ping (debounced to once per 5 min unless user moves >1 km).

Why S2 + Redis-style active sets instead of Elasticsearch geo search: the dating stack asks for fresh, nearby, active users, not arbitrary text + geo ranking. Elasticsearch works for richer profile search, but S2 active sets make freshness, TTL, and adjacent-cell expansion explicit and cheap.

The match race. Both users swipe right on each other within the same second. Two concurrent inserts. The match table must use a deterministic primary key like LEAST(a,b)||GREATEST(a,b) and INSERT IGNORE so only one row wins. The losing writer reads back the existing row and returns the same match_id.

Hot-key on prolific swipers. A user swiping 500/day creates a hot shard. Sharding by swiper_id is generally fine; for the heaviest tail, batch swipes client-side and submit in a single call.

Candidate ranking quality. Pure recency in cell ignores attractiveness/compatibility signals. Add a small ranking model (LightGBM, ~50 features) at the candidate service. Mention online learning from swipe outcomes as the production lever.

Photo storage. Profile photos in object store (S3) behind a CDN with signed URLs. Thumbnails (3 sizes) generated async on upload.

Privacy / safety. Profile visibility is bidirectional: if Alice blocks Bob, neither appears in the other's stack. Block list lives in the user profile; candidate gen filters against it.

ELO / matchmaking score (optional). Some products use a desirability score derived from swipe-right rates. Compute offline; bucket users by score to keep matches balanced. Be ready to discuss the ethical tradeoff.

Graceful degradation state machine.

Case Mode Source Latency budget Fallback
Dense city Normal ranked stack S2 active cells + profile cache <300 ms Expand 1 ring, rank top 50
Sparse/cold area Exhaustion mode Wider S2 rings + older active users <500 ms Show fewer cards, widen distance, relax recency
Dormant user returns Lazy filter load Persisted swipe log rebuilds shown filter First request <1s Return small batch while async rebuild completes
Hot swiper Batched writes Swipe log + reciprocal index Swipe ack <100 ms Buffer client swipes and flush in small batches
Geo index degraded Conservative candidates Last-known profile location <500 ms Disable distance-sensitive ranking; never break no-repeat
Match insert race Unique-row winner match:{min,max} <100 ms Losing writer reads existing match and returns same id

Server-side state that must persist across requests: the opaque card cursor {user_id, geo_cell, candidate_batch_id, rank_offset}, the shown:{user_id} filter, idempotency for (swiper,target), and the sorted-pair match key. Losing the cursor is tolerable; losing shown risks repeats, so rebuild it before returning a full stack.

What I'd defer. Chat infrastructure (presence, typing indicators), Super Like and rewinds, abuse moderation, machine-learning ranking deep-dive, premium subscription logic.
