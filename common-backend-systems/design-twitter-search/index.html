1. Basic Knowledge And Solution

What This Question Is Testing

Twitter Search is real-time search over a high-velocity stream — millions of tweets per day, indexed within seconds, queryable by text, hashtag, user, location. The hard part is keeping the index fresh enough that breaking news is searchable within seconds, while keeping queries sub-second over years of historical data.

The invariant is this:

Every published tweet is indexed within bounded freshness (typically seconds for recent / minutes for the long tail); search queries return relevant tweets ordered by recency and engagement signals; visibility filters (blocks, deletes, privacy) applied at query time; the index scales with corpus size and query volume.

At ICT5 / L6+ level, the interviewer is testing whether you can design a sharded inverted index over a high-write stream, real-time vs batch indexing tiers, scoring and ranking, and the read-side query planner.

60-Second Interview Framing

Spoken version:

I would design Twitter Search as a two-tier inverted index: a live tier for breaking content indexed within seconds, and a main tier for historical content built in batch. Queries scatter-gather across both tiers, merge by score and time, hydrate tweet content from the tweet store, apply visibility filters, return. The interesting decisions are real-time indexing, sharding strategy, ranking, and visibility.

Requirements To Clarify First

Functional requirements:

  • Search by text, hashtag, user, language, location.
  • Sort by recency, popularity, or relevance.
  • Filter by date range, language, media type.
  • Real-time results for breaking news.
  • Trending topics (related but separate).

Non-functional requirements:

  • New tweet indexed within seconds.
  • Query p99 < 500ms.
  • Years of history searchable.
  • Multi-region.

Out of scope unless asked:

  • Tweet creation (see Design Twitter).
  • Personalization beyond basics.
  • DM search.

Index Shard

Each shard owns a subset of the index. Sharding strategies:

  • By time: each shard holds tweets in a time range. Good for time-range queries; risk of hot recent shard.
  • By hash: distribute by hash of tweet_id. Balanced load; every query hits every shard.
  • Hybrid: time-tier + hash within tier.

In practice, the live tier shards by hash for write throughput; main tier shards by time + hash for query patterns.

Concrete Index Stores And Session State

Visibility cache visibility:{viewer_id} block/mute/protected-author ids 10k IDs cap, 80-160KB worst case 30-60 second cache over strong store viewer_id
Search cursor opaque cursor_id or signed blob query plan, last score, shard offsets, time window 0.5-2KB 10-30 minutes viewer_id + query hash

Worked sizing:

  • If 500M tweets/day produce 20 indexed terms each and each live posting is 24B, live index growth is 240GB/day raw. Seven days with 3 replicas is about 5TB, which fits in memory only after compression and active-term filtering.
  • The main index stores the same tweet corpus as immutable compressed segments. At 8B/posting compressed, one day is about 80GB raw; five years with 3 replicas is roughly 438TB before doc store and metadata.
  • Tweet content is a separate hydrated store: 500M/day * 1.5KB = 750GB/day raw. Search indexes should return ids and scores, not full tweet bodies.

This is a two-index pattern, not two sources of truth: live and main indexes are two views over the same tweet stream. The live view is mutable and recency-heavy; the main view is compact and historical.

Tech-choice defense:

  • Use an inverted index because the access pattern is term -> candidate tweet ids; scanning a key-value store by tweet id would be the wrong direction.
  • Use Lucene-style immutable segments for the main tier because segment merge and compression are first-class. A wide-column store can hold posting rows, but it makes scoring, compression, and top-K retrieval application work.
  • Use an append stream for indexing because tweet writes should not block on every term index update. Synchronous indexing gives fresher results but couples post latency to the slowest index shard.

Be ready for "why not Cassandra?" The answer is that Cassandra stores rows well, but search needs compressed posting blocks, skip lists, scoring, and segment merges.

APIs

GET /search?q=&from=&until=&type=&lang=&geo=

Indexing Pipeline

For each new tweet:

  1. Tweet posted; event emitted to stream.
  2. Live indexer consumes event.
  3. Tokenize text: lowercase, normalize, split, stop words.
  4. Extract entities: hashtags, mentions, URLs, language.
  5. Update posting lists in live index shard(s).
  6. Tweet searchable in seconds.

Periodically (hourly or daily):

  1. Live tier rolls off older content.
  2. Batch indexer processes rolled-off tweets.
  3. Builds compressed main index segment.
  4. Merge into main tier.

Query Path

For "italian restaurants near NYC":

  1. Frontend parses query: terms, operators, filters.
  2. Query rewriter expands: synonyms, spelling correction.
  3. Query planner identifies which shards to query (e.g., live + recent main shards for recency-sensitive queries).
  4. Scatter to selected shards.
  5. Each shard intersects posting lists for query terms.
  6. Returns top-K candidates with scoring data.
  7. Merger aggregates across shards, scores, returns top-K.
  8. Hydrate tweet content from tweet store.
  9. Visibility filter.
  10. Return.

Latency budget: - Shard query: ~50-100ms. - Merge: ~20ms. - Hydration: ~50ms. - Visibility: <10ms. - Total: under 500ms.

Hot breaking-news query path:

  1. Client sends q="#oscars", viewer_id, filters, and optional cursor. The gateway hashes the normalized query to check a 5-15 second result cache.
  2. Query planner chooses live shards plus the newest main time buckets. Shard key is term_hash; fan-out calls run in parallel.
  3. Each shard reads live:{term_hash}:{shard_id} from memory and recent main:{time_bucket}:{term_hash}:{segment_id} blocks from SSD cache.
  4. Shards return top 1k tweet ids and cheap scores. The merger keeps top 2k globally, then calls the ranker.
  5. Hydration batches tweet:{tweet_id} reads by tweet-id shard while visibility reads visibility:{viewer_id} in parallel.
  6. Slow shards are dropped after the per-shard timeout; the response carries partial=true and a cursor that records included tiers and shard offsets.

Cold historical query path:

  1. Query planner narrows by from/until, then maps time buckets to main index segments.
  2. Segment cache miss loads compressed posting blocks from object/SSD store. This is the only path allowed to exceed the normal 100ms shard budget.
  3. If the user paginates beyond the hot top-K window, the cursor resumes with (query_hash, time_bucket, shard_offsets, last_score, visibility_version) instead of using offset pagination.
  4. Exhausted cursors return an empty page and a terminal cursor; they do not rescan from the beginning.

Read-path sketch:

Client -> Gateway -> Query Planner

Query degradation state:

Stage Mode Source Latency target Cursor behavior
1 Full hot results live + cached main segments <300ms Cursor records shard offsets.
2 Partial results live + successful main shards <500ms Cursor marks missing shards for retry.
3 Cold historical main segments from SSD/object <1.5s Cursor pins time buckets.
4 Exhausted window no more postings past cursor <100ms Terminal cursor returned.

The search service can degrade relevance before it degrades correctness: drop slow shards or old tiers, but never skip hydration and visibility checks.

Tail Latency

Scatter-gather: slowest shard bounds latency.

Mitigations: - Hedged requests to replicas. - Per-shard timeouts. - Tolerate partial results (drop slow 1%). - Capacity headroom.

Ranking

Multi-signal: - Term match relevance (BM25). - Author authority. - Engagement (likes, retweets). - Recency. - Personalization (viewer's interaction history). - Verified accounts boosted.

Multi-stage: cheap per-shard scoring; expensive ranker on top.

For real-time / breaking content: time-decay heavy.

Trending Topics

Related but separate pipeline:

  • Aggregate hashtag mentions per region per minute.
  • Compare to baseline (prior period).
  • Top movers = trending.
  • Personalized per region and user interest.

Visibility Filter

Apply at query result time:

  • Blocks: drop blocked authors.
  • Mutes: drop muted accounts/keywords.
  • Deletes: drop deleted tweets (hydration sees status).
  • Privacy: protected accounts visible only to approved followers.
  • Policy: removed tweets.

Never bake visibility into the index; filter at read.

Tradeoffs To Call Out

  • Live tier is fast to update but expensive memory-wise; rolling off is critical.
  • Main tier is cheap but slower to update.
  • Time-sharded queries are fast for recency-sorted but every shard hit for general queries.
  • Hash-sharded balances load but every query hits every shard.
  • Aggressive ranking improves relevance but adds latency.
  • Personalization vs determinism.

2. Simulated Interview

Interviewer: Design Twitter Search.

Candidate: Two-tier inverted index. Live tier indexes new tweets within seconds — in-memory or SSD, sharded by hash for write throughput, retains recent hours. Main tier is long-term: batch-built from rolled-off tweets, compressed, sharded by time + hash. Queries scatter to relevant shards (live + recent main for recency-sensitive; broader for general), merge by score, hydrate, apply visibility filter, return.

Interviewer: Why two tiers?

Candidate: Different optimization points. Live tier needs sub-second indexing; can't afford the batch process. Main tier needs efficient storage for trillions of tweets; batch processing compresses better. Combining one tier with both properties would be too expensive. The query merge is straightforward — recent results from live tier, older from main, sort.

Interviewer: Indexing latency for breaking news.

Candidate: Stream consumer (Kafka) feeds live indexer. Tokenize, extract entities, update posting lists in live shards. End-to-end seconds. For very-hot terms ("breaking"), shard the posting list further to handle high write rate.

Interviewer: Query path.

Candidate: Frontend parses, rewrites query (synonyms, spell correction). Planner picks shards: live + recent main for time-sensitive; broader for general. Scatter to shards. Each shard intersects posting lists for query terms, scores candidates with cheap features (BM25 + static), returns top-K. Aggregator merges, applies multi-stage ranking (cheap → expensive ML), hydrates tweet content, applies visibility filter, returns.

Interviewer: Tail latency from scatter-gather.

Candidate: Hedged requests: send to two replicas of the same shard, take first response. Per-shard timeout; on miss, drop slow shard but return partial results (with bounded quality loss). Capacity headroom around 60-70% so slow nodes don't cascade. Aggressive per-shard p99 monitoring.

Interviewer: Ranking?

Candidate: Multi-stage. Per-shard cheap scoring: BM25 + static signals (author authority, recency). Returns top-K per shard. Aggregator merges; expensive ML ranker scores top results with features (viewer affinity, engagement, recency, personalization). Final ordering.

Interviewer: Visibility — blocks and deletes.

Candidate: Read-time gate. After hydration, drop blocked authors' tweets, deleted tweets, protected accounts' tweets (if not approved follower), policy-removed tweets. Never bake visibility into the index — that would require massive re-indexing on every block change.

Interviewer: Trending topics.

Candidate: Related but separate pipeline. Aggregate hashtag mentions per region per minute. Compare to recent baseline; top movers are trending. Personalized per region and user interest. Independent of the search index.

Interviewer: A user follows 500 hashtags and queries for one.

Candidate: Search is general — anyone can query any term. The "follow hashtag" feature is more like a saved filter. The query path is the same; personalization for the user's follows can boost results in ranking.

Interviewer: Multi-region.

Candidate: Per-region live + main indexes. Tweet stream replicates globally so each region has the full corpus. Queries served locally. Personalization data per-user home region.

Interviewer: Scale of posting lists for hot terms ("the", "a").

Candidate: Stop words filtered. Common terms like "a" not indexed. Mid-frequency terms with huge posting lists are compressed (delta encoding on tweet_ids). For very-hot terms during events, the posting list can be sharded further so per-shard write rate stays manageable.

Interviewer: Spam in results.

Candidate: Spam scoring per tweet. Filtered or down-ranked. Author reputation feeds into static scoring. Pattern detection on suspicious clusters (similar text across many accounts). Manual review for severe cases.

Interviewer: Summarize.

Candidate: Two-tier inverted index: live (seconds, in-memory, hash-sharded) + main (hours, compressed, time+hash-sharded). Async indexer consumes tweet stream. Queries scatter-gather, multi-stage rank with hedged requests for tail. Hydration from tweet store. Visibility filter at read. Multi-region per-region indexes. The hardest constraint is indexing latency for breaking news; the two-tier design isolates it from the long tail.

One-Minute Closeout

Twitter Search is a two-tier inverted index: live for breaking content (seconds), main for historical (hours). Async pipeline consumes the tweet stream. Queries scatter-gather across relevant shards, multi-stage rank, hydrate, apply visibility filters. Hedged requests handle tail latency. Visibility at read prevents massive re-indexing. Multi-region per-region indexes. The hardest constraint is real-time indexing without sacrificing historical query speed.
