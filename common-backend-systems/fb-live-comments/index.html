Facebook Live Comments — Live-Stream Chat Fan-Out (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build the live-comment system for a Facebook (FB) Live broadcast or Twitch-style stream. One streamer broadcasts, millions of viewers watch, and viewers post comments that everyone sees in near real-time. A popular live event has 1M concurrent viewers and 5k comments/sec; comments must appear within ~1 s.

1. Requirements (~5 min)

Functional - Viewers post comments on a live stream. - All viewers see new comments within ~1 s, in roughly the order posted. - New viewers joining see the last few hundred comments as backfill.

I'd defer reactions, polls, monetization, and moderator tools.

Non-functional - p95 comment-post-to-display < 1 s. - Survive 1M+ concurrent connections per stream; many streams concurrent overall. - Eventual ordering is fine — strict global order across all viewers is not required. - Loss tolerance: dropping the rare comment is acceptable; failing to deliver any comment for 10 s is not. - Backpressure: when comments outpace fan-out, drop oldest rather than melt the system.

Capacity - A mega-stream: 1M viewers × 1 comment/min = 17k comments/sec write; read fan-out 1M × 5k = 5B msg/sec — unrealistic to send every comment to every viewer at high volume, see deep dives. - The system must cap displayed comments per viewer. If each viewer receives 5 batched updates/sec and each batch is ~2 KB, that is still ~10 GB/sec egress for one mega-stream.

2. Core Entities (~2 min)

  • Stream — stream_id, broadcaster, started_at, status.
  • Comment — comment_id, stream_id, user_id, text, ts.
  • Viewer connection — connection_id, stream_id, user_id.

3. API / Protocol (~5 min)

WS  /v1/streams/{id}/comments
  ← client sends:  { post: text }   (idempotency-key)
  → server pushes: { comment }      (batched)

HTTP:
GET /v1/streams/{id}/comments/recent?limit=200    → backfill for join

User from token; rate-limit per user per stream.

The WS connection carries a server-side cursor {stream_id, last_comment_id, sample_policy}. On reconnect, the client asks for recent comments after last_comment_id, then resumes live push.

5. High Level Design (~10–15 min)

  1. Edge WS gateway. Holds the millions of viewer connections. Per-stream sharding: hash(stream_id) → set of edge gateways.
  2. Comment ingestion. Comment hits any edge gateway → forwarded to the stream owner shard (the partition responsible for this stream) → ingestion service assigns server timestamp + comment_id, writes to persistent log (Kafka topic per stream or sharded by stream).
  3. Fan-out tier. Each stream's comment topic is consumed by a fan-out service that pushes to all edge gateways holding viewers for that stream. Edge gateways then push to the local WS connections.
  4. Persistent store. Append-only — Cassandra keyed by (stream_id, ts) or just the Kafka topic with long retention. Drives backfill and replay.
  5. Backfill. When a viewer joins, edge gateway gives them the last ~200 comments via a separate HTTP call before opening the WS push.
  6. Rate-limit / spam pipeline. Per-user token bucket on post. Profanity / blocklist filter pre-broadcast. Moderator actions go through the same pipeline.

End-to-end narration: Alice is watching → her phone has a WS to edge gateway G1 for stream S → Bob posts "🔥🔥🔥" → his edge gateway G3 forwards to stream owner shard for S → ingestion stamps it with ts=…, comment_id=C99 → publishes to Kafka topic stream.S.comments → fan-out service consumes, pushes to G1, G2, G3 (every gateway holding S viewers) → each gateway broadcasts to its local WS connections → within 500 ms Alice sees Bob's comment.

Post path:

  1. Edge gateway validates auth, rate limit, and message size.
  2. It forwards to the stream owner shard keyed by stream_id, which assigns comment_id = (stream_epoch, sequence).
  3. Ingest writes Kafka and recent-backfill Redis in parallel after moderation passes.
  4. Fan-out workers consume the stream partition and push batches to all edge gateways in stream_edges:{stream_id}.
  5. Each gateway writes to local WS ring buffers; slow clients drop old batches and keep the newest.

Join/read path:

  1. Viewer calls GET /comments/recent?limit=200; edge first reads recent:{stream_id} from Redis.
  2. On Redis miss, edge falls back to CommentStore[(stream_id, comment_id DESC)], over-fetches 300, filters deleted/moderated comments, and repopulates Redis.
  3. Viewer opens WS with {stream_id, last_comment_id}; gateway registers itself in stream_edges:{stream_id}:{region} with a 10-30 s heartbeat TTL.
  4. Live batches arrive from fan-out; gateway dedupes against last_comment_id, appends to the per-connection ring buffer, and advances the server-side cursor.
  5. On reconnect, the cursor asks for comments after last_comment_id; if the id is older than retention, the server returns lossy_resume plus the newest 200 comments.

6. Deep Dives (~10 min)

Fan-out at million-viewer scale. Single broker pushing to 1M connections falls over. Hierarchical fan-out: one broker per region/AZ; each broker holds a chunk of viewers (say 10k each); a Kafka topic with N partitions feeds N brokers. Brokers don't broadcast individual comments — they batch and broadcast every 200 ms with the accumulated comments. Trade off freshness for throughput.

Capacity math: 17k comments/sec × 200 B is only ~3.4 MB/sec into Kafka for the mega-stream; fan-out is the monster. Sending every comment to 1M viewers would be 17B deliveries/sec. Capping each viewer to 5 batches/sec × 2 KB makes egress ~10 GB/sec, still large but feasible with regional edge gateways. Dominant cost is network fan-out and open connection memory, not comment storage.

Store math for one sustained mega-stream: live-topic writes are ~3.4 MB/sec raw, ~900 GB/day with RF=3, so 7-day replay is ~6 TB. Durable comments at ~300 B/comment are ~440 GB/day raw, ~1.3 TB/day with RF=3; most streams are short, so retention policy matters more than peak size. recent:{stream_id} is tiny: 1,000 comments × 300 B is ~300 KB per stream, or ~30 GB raw for 100k active streams. Connection memory is the sharp edge: 1M viewers × a 100 KB ring cap is 100 GB spread across gateways, so the ring buffer must be bounded and batch objects should be shared per stream on each gateway instead of copied per connection. Rate-limit state for 1M active commenters is only ~32 MB.

Why Kafka, not Redis Pub/Sub. Redis Pub/Sub is fast but not replayable and makes backfill/reconnect awkward. Kafka/Pulsar give ordered partitions and replay from offsets. Cassandra is fine for (stream_id, comment_id) backfill because reads are recent range scans; MySQL would need heavy sharding for hot streams.

Be ready for "why not push directly from Kafka to every gateway?" The answer is that Kafka is the durable stream, not the per-connection delivery system. A fan-out tier can batch, sample, prioritize streamer/mod lanes, and apply regional backpressure without turning Kafka partitions into connection managers.

Lossy delivery is OK. If a viewer's WS can't keep up, drop oldest comments rather than buffer. The newest 100 are what matters in a live stream. Per-connection ring buffer with a tail-drop policy.

Ordering. Strict per-stream order is impossible at multi-million scale without serializing through one machine. Approximate: ingestion assigns a server timestamp per comment, fan-out brokers batch in timestamp order. Clients display in arrival order with timestamps — small reorderings are visually fine. Don't promise totally-ordered.

Backpressure on hot streams. When comment rate exceeds display rate, sample. Show 1 in 5 comments to viewers (random sampling); show all to moderators. Mention "Twitch chat slow mode" as a precedent.

Two views of comments. The durable log (stream_id, comment_id) is the truth for replay, moderation, and post-stream archive. The recent Redis list recent:{stream_id} is the read-optimized window for joins and reconnects. The live fan-out batches are a third, lossy delivery view derived from the same log; losing a batch affects one viewer's experience but not the canonical comment record.

Hot-stream degradation:

Stage Mode Source Viewer effect
normal full batch all comments in 200 ms batches near-complete chat
hot sampled batch random/sample by score representative chat
hotter slow mode per-user post bucket tightened fewer writes
extreme priority lanes streamer/VIP/mod comments full fidelity; public sampled important comments survive
edge slow tail drop per-connection ring buffer drops oldest latest chat remains live

Backfill on join. Last 200 comments from Cassandra (keyed by stream_id) before WS opens. After WS opens, edge gateway begins live push, deduping any overlap by comment_id.

Reconnect path: HTTP fetch recent:{stream_id} or Cassandra range where comment_id > last_seen; WS opens only after the client has a new cursor. If the cursor is older than retention, return the last 200 and mark the stream as lossy_resume.

Hot streamer / VIP comments. Streamer's own comments and pinned messages are pushed at full fidelity regardless of sampling. Implemented as a separate channel viewers subscribe to.

Persistence. Kafka topic per popular stream with 7-day retention. For long tail, a single Kafka cluster handles many streams partitioned. Cold storage in object store after 7 days for compliance / replays.

Latency budget. Post→display 1 s budget: client→edge 50 ms, edge→shard 50 ms, ingest write 20 ms, fan-out batch window 200 ms, broker→edge 50 ms, edge→clients batched 200 ms. Fits.

Connection scaling. 1M concurrent WS connections on commodity gateways: ~50k-100k per box; ~20 gateway boxes per stream-region. Tuning kernel limits (file descriptors, ephemeral ports) is real work in practice.

Hot keys, regions, and cold starts. A mega-stream makes stream_id a hot key, so split stream_edges:{stream_id} by region and maybe by gateway bucket: stream_edges:{stream_id}:{region}:{bucket}. If one owner shard cannot stamp 17k comments/sec, split ingestion into lanes keyed by (stream_id, lane) and generate comment_id = timestamp + lane + sequence; clients display by timestamp and tolerate tiny reorderings. Regional viewers should receive local fan-out first; cross-region replication of comments can lag a few hundred milliseconds, which is acceptable for public chat but not for moderator deletes, so moderation/control events use a priority lane. At stream start, pre-create the live topic partitions, recent list, and edge registry entry when the broadcaster goes live; do not wait for the first million viewers to trigger cold allocation.

Moderation. Streamer-side block list + ML toxicity score. Flagged comments hidden from public broadcast but visible to mods.

What I'd defer. Reactions/emote bursts, super-chats / paid pins, multi-language translation, real-time analytics, persistent post-stream comment archive UI.

Interview punch line: comment writes are easy; fan-out is the product. I would design the system around bounded per-viewer delivery and make degradation intentional: sample public chat, never drop streamer/mod messages, and keep the stream feeling live.
