1. Basic Knowledge And Solution

What This Question Is Testing

Cassandra is a wide-column distributed database that combines Dynamo's leaderless cluster design (consistent hashing, sloppy quorum, gossip) with Bigtable's storage model (SSTables, LSM tree, column families). The hard part is the trade between the strict-as-needed consistency (CL ALL, QUORUM, ONE) and high availability, plus the data model based on wide rows with clustering columns.

The invariant is this:

Every write is durably persisted to a commit log on at least one replica and propagated to all N replicas; reads at the configured consistency level reflect the latest write (within tunability); the wide-column model allows efficient per-partition queries with clustering-column-ordered reads; the cluster scales linearly by adding nodes.

At ICT5 / L6+ level, the interviewer is testing whether you can articulate Cassandra's data model (partition key + clustering columns), the LSM-tree storage engine, consistency levels, lightweight transactions (Paxos), and the operational complexity (compaction, repair).

60-Second Interview Framing

Spoken version:

I would describe Cassandra as Dynamo's cluster model plus Bigtable's storage engine. Consistent hashing for partitioning; configurable replication and consistency level per query; commit log + memtable + SSTables for storage; LSM compaction; gossip for membership. The data model is wide rows with a partition key plus clustering columns for sorted ranges within a partition. Lightweight transactions via Paxos for compare-and-set. The interesting decisions are data model, consistency levels, compaction strategy, and repair.

Requirements Cassandra Addresses

Functional requirements:

  • Distributed wide-column store.
  • Wide row queries with clustering order.
  • Tunable consistency per query.
  • Multi-data-center replication.
  • Lightweight transactions (compare-and-set).
  • Time-series and IoT workloads.

Non-functional requirements:

  • High write throughput.
  • High availability.
  • Linear scalability.
  • Operationally simple cluster expansion.
  • Multi-region (active-active).

Out of scope:

  • Multi-row transactions across partitions.
  • Strict global ordering.
  • Ad-hoc full-table joins (not designed for relational workloads).

Core Concepts

Data Model

Keyspace
  └── Table
       └── Row
            ├── Partition key (determines node placement)
            ├── Clustering columns (sort order within partition)
            └── Regular columns (values)

Example: sensor time series.

CREATE TABLE sensor_data (
  sensor_id text,
  reading_ts timestamp,
  value double,
  PRIMARY KEY (sensor_id, reading_ts)
);
  • sensor_id = partition key → hashed to a node.
  • reading_ts = clustering column → sorted within the partition.
  • Query: SELECT * FROM sensor_data WHERE sensor_id = ? AND reading_ts > ? ORDER BY reading_ts.

A "row" in Cassandra is identified by (partition_key, clustering_columns); a "partition" holds many rows with the same partition key, sorted by clustering columns.

This data model is partition-local: efficient queries are within a single partition. Queries spanning partitions are expensive (require scatter-gather).

Interview phrase:

The Cassandra data model is partition-key + clustering columns. Query patterns must be designed first; tables are denormalized per query pattern.

Storage Engine: LSM Tree

Inherited from Bigtable.

  • Write path: commit log (durability) + memtable (in-memory sorted structure).
  • When memtable fills: flush to SSTable on disk.
  • SSTables are immutable sorted files.
  • Compaction merges SSTables periodically.
  • Bloom filters per SSTable speed up negative lookups.

Compaction strategies: - SizeTieredCompactionStrategy (STCS): merge similar-sized SSTables. Good for write-heavy. - LeveledCompactionStrategy (LCS): maintain levels with size limits. Good for read-heavy. - TimeWindowCompactionStrategy (TWCS): group by time window. Good for time-series.

Consistency Levels (per query)

Level Behavior
ONE Wait for 1 replica.
QUORUM Wait for floor(N/2) + 1.
ALL Wait for all N.
LOCAL_QUORUM Quorum within local DC.
EACH_QUORUM Quorum in each DC.
ANY Coordinator accepts via hinted handoff if all replicas down.

Strong consistency: R + W > N (e.g., QUORUM + QUORUM with N=3). Eventual consistency: lower levels.

Cluster Design (from Dynamo)

  • Consistent hashing ring with vnodes.
  • Replication factor N per keyspace.
  • Gossip for membership.
  • Hinted handoff for short failures.
  • Anti-entropy repair via Merkle trees.
  • Sloppy quorum optional.

High-Level Architecture

Client -> Coordinator (any node)
              |
              v
     Consistent Hashing Ring
              |
              v
       Preference List (N nodes)
              |
        Replicate write to N (wait for CL)
              |
              v
       Commit log + Memtable
              |
        Flush to SSTable

              |
        Gossip / Hints / Repair

Core Components

Component Responsibility Key detail
Coordinator Per-request node. Any node.
Commit log Append-only durable log. First write target.
Memtable In-memory sorted by clustering. Flushes on size/time.
SSTable Immutable on-disk sorted file. Many per partition.
Compactor Merges SSTables. Background; strategy-driven.
Gossip Cluster membership. From Dynamo.
Hinted handoff Temp store for unreachable replicas. Replay on recovery.
Repair Merkle-tree anti-entropy. Manual or scheduled.
Bloom filter Per-SSTable negative lookup. In memory.
Snitch Topology awareness for replica placement. E.g., DC-aware.

Data Model Continued

Counters

Special column type for atomic increment/decrement. Implementation uses different replication semantics (delta-based). Limited consistency guarantees.

Materialized Views

Server-side denormalization: secondary tables auto-maintained from a base table. Per-write overhead; useful when you query by different keys.

Secondary Indexes

Local per-node index on a non-partition column. Query requires scatter-gather across all partitions; expensive.

Write Path Details

  1. Coordinator receives write.
  2. Computes partition key hash → preference list (N replicas).
  3. Sends write to all N replicas in parallel.
  4. Each replica: append to commit log → insert into memtable.
  5. Wait for CL responses.
  6. Return success.

Memtable flush: when size or time threshold reached, write to a new SSTable. Commit log truncated after flush.

Read Path Details

  1. Coordinator receives read.
  2. Computes partition key hash → preference list.
  3. Sends read request to closest replicas (per CL).
  4. Each replica: query memtable + relevant SSTables (filtered by bloom).
  5. Coordinator merges responses (latest timestamp wins).
  6. If CL > 1, return when CL responses received.
  7. Read repair (optional): if responses disagree, write merged value back.

For range queries within a partition: read SSTables in clustering order, merge, return rows in clustering order.

Lightweight Transactions (CAS)

Paxos-based compare-and-set for single-partition atomic operations.

UPDATE users SET email = 'new' WHERE id = 1 IF email = 'old';
INSERT INTO users (...) VALUES (...) IF NOT EXISTS;

Implementation: four-phase Paxos within the partition. Slower (4× round trips) than normal writes. Useful for cases where strict atomicity is required (e.g., unique username).

Tombstones

Deletes don't physically remove data; they write a tombstone marking the row deleted.

  • Tombstone has a timestamp.
  • On read, tombstone wins over older data.
  • Compaction eventually drops tombstones older than gc_grace_seconds.

Why grace period: prevents resurrection of deleted data after a missed replica catches up. If a replica was offline during the delete and tombstone was already GC'd, the replica's old value could resurrect the data via repair.

Common operational pitfall: high tombstone count slows reads.

Compaction Strategies

Strategy Best for
SizeTieredCompactionStrategy Write-heavy; tolerates higher read amplification.
LeveledCompactionStrategy Read-heavy; bounded read amplification.
TimeWindowCompactionStrategy Time-series; drops old data efficiently.

Pick based on workload.

Repair

Cassandra requires regular repair to ensure replicas converge.

  • Merkle-tree comparison between replicas.
  • Streams differing data.
  • Run on a schedule (typically weekly).
  • Heavy operation; usually one node at a time.

Without repair: stale replicas drift, deleted data may resurrect after gc_grace_seconds.

Multi-Data-Center

Snitches make placement DC-aware.

  • Replication factor per DC: {"DC1": 3, "DC2": 3}.
  • Writes propagate to all DCs; coordinator can wait for local quorum (LOCAL_QUORUM) — fast — and async to other DCs.
  • For strict cross-DC consistency: EACH_QUORUM.

Cassandra is known for multi-DC active-active.

Scaling And Partitioning

Concern Strategy
Add node Bootstrap from existing; streams responsible ranges.

Hot partition Bound partition size; model around access pattern.
Tombstone overhead Tune gc_grace; design for fewer deletes.
Read amplification LCS compaction; smaller SSTables.

Operational Metrics

Correctness:

  • Repair lag.
  • Tombstone count per read.
  • Hint backlog.
  • Stale reads from low CL.

Capacity:

  • Per-node QPS.
  • Compaction backlog.
  • Read latency p99.

Business:

  • Throughput.
  • Storage per node.

Tradeoffs To Call Out

  • Cassandra trades multi-row transactions for high write throughput and linear scaling.
  • Wide rows enable efficient per-partition queries but require modeling per query.
  • LCS gives bounded read amplification but more write amplification than STCS.
  • LWT gives atomic CAS at significant latency cost.
  • Tombstones cleanup time vs delete resurrection risk.
  • Multi-DC active-active is great for read locality but eventual consistency cross-DC.
  • Counters have weaker consistency than regular columns.

Common Traps And Stronger Moves

Trap Stronger move
Use as relational DB with joins. Denormalize per query; design data model first.
Default consistency without thought. Pick CL per query; QUORUM is common.
Frequent deletes. Design to minimize; understand tombstone cost.
Random partition keys for time-series. TimeWindowCompactionStrategy.
Ignore repair. Schedule regular repairs.
LWT for everything. Only when atomicity truly needed.

2. Simulated Interview

Interviewer: What is Cassandra?

Candidate: Distributed wide-column database. Cluster model from Dynamo: consistent hashing, gossip, sloppy quorum, anti-entropy repair. Storage engine from Bigtable: commit log, memtable, SSTables, compaction. Data model: keyspaces → tables → partitions → rows; each row is (partition_key, clustering_columns) → values. Queries optimized per partition. Tunable consistency per query. The invariant is high availability, linear scaling, and efficient per-partition queries.

Interviewer: Walk me through a write.

Candidate: Client sends write to a coordinator. Coordinator hashes partition_key, finds the N replicas in the preference list, sends the write to all in parallel. Each replica appends to commit log (durability) and inserts into memtable (sorted in-memory by clustering columns). Coordinator waits for CL replicas to ack; returns success. Memtable flushes to a new immutable SSTable when full. Background compaction merges SSTables.

Interviewer: Read path.

Candidate: Coordinator hashes partition_key, picks closest replicas per CL. Each replica queries memtable + SSTables (Bloom filter avoids unnecessary disk seeks). Returns rows for the requested partition + clustering range. Coordinator merges responses (latest timestamp wins per column). Optional read repair if replicas disagree.

Interviewer: Data model.

Candidate: Wide rows. A partition_key determines node placement. Clustering columns define sort order within the partition. Example: (user_id, post_ts) — all of a user's posts are in one partition, sorted by time, so range query "user X's posts in time range" is efficient (single partition read). The art is designing the partition_key + clustering to match query patterns. Tables are denormalized per query.

Interviewer: Why no joins?

Candidate: Joining across partitions would require cross-node coordination, hurting Cassandra's linear scaling. Instead, denormalize: store related data in the same partition or duplicate across tables. For a user's posts and comments, store comments embedded in the same partition or in a parallel table per (user, post).

Interviewer: LWT — when to use?

Candidate: Compare-and-set when atomicity matters: unique username, leader election, sequence numbers. Uses Paxos under the hood — 4 round-trips. Much slower than normal writes (10x). Use sparingly. For most use cases, accept eventual consistency and design around races.

Interviewer: Tombstones — what's the issue?

Candidate: Delete doesn't physically remove; writes a tombstone with timestamp. Reads must check tombstones to know what's deleted. Many tombstones slow reads. Compaction eventually drops tombstones after gc_grace_seconds (default 10 days). Before that, the tombstone must be retained so an offline replica receiving an old value during repair sees the tombstone and doesn't resurrect deleted data.

Interviewer: Compaction strategy.

Candidate: Three main strategies. SizeTieredCompactionStrategy: merge similar-sized SSTables; write-heavy. LeveledCompactionStrategy: maintain levels with size budgets; bounded read amplification; better for read-heavy. TimeWindowCompactionStrategy: group SSTables by time window; ideal for time-series where old data is dropped wholesale. Pick based on workload.

Interviewer: Multi-DC?

Candidate: Snitch makes placement DC-aware. Replication factor per DC. Writes propagate to all DCs. LOCAL_QUORUM waits for local DC quorum (fast); EACH_QUORUM for cross-DC strict consistency. Common pattern: active-active across DCs with LOCAL_QUORUM; cross-DC lag in seconds is OK.

Interviewer: Repair.

Candidate: Anti-entropy. Merkle-tree comparison between replicas of a partition range. Differences streamed to converge. Required to ensure eventual consistency, especially after long failures. Run regularly (weekly typical). Heavy operation; usually one node at a time.

Interviewer: A hot partition?

Candidate: All traffic for one partition_key lands on the same N replicas. Bottleneck. Fix at the application: sub-partition by adding a random or time-based suffix to partition_key, then query all suffixes for aggregation. Alternative: caching layer above. Cassandra itself can't shard a single partition_key across nodes.

Interviewer: Counter columns?

Candidate: Special column type for atomic increment/decrement. Internally uses delta-based replication different from regular columns. Eventually consistent; some loss possible in failure scenarios. Useful for "likes counter" semantics where exactness isn't critical. For exact counts, use periodic recomputation from event logs.

Interviewer: Bootstrap a new node.

Candidate: Pre-assigned vnode ranges. New node joins ring; existing nodes stream the relevant ranges. Parallelized across vnodes — multiple nodes stream simultaneously. Throttle to avoid saturating. Gossip propagates the new membership. Once bootstrap completes, the new node serves reads/writes for its ranges.

Interviewer: Summarize.

Candidate: Dynamo cluster + Bigtable storage. Wide-column data model with partition_key + clustering columns; query-pattern-driven design. LSM tree write path; SSTable compaction. Tunable consistency per query; LWT via Paxos for CAS. Multi-DC active-active. Anti-entropy repair required. Hot partitions and tombstones are operational pitfalls. The strength is linear scale and high write throughput; the weakness is multi-row transactions.

3. Practice Notes

Five-Minute Version

  1. Dynamo cluster + Bigtable storage.
  2. Wide rows: partition_key + clustering columns.
  3. LSM: commit log + memtable + SSTable + compaction.
  4. Tunable CL per query.
  5. LWT via Paxos for CAS.
  6. Multi-DC active-active.
  7. Repair (Merkle); tombstone gc.
  8. Compaction strategy per workload.

One-Minute Closeout

Cassandra combines Dynamo's cluster model with Bigtable's LSM storage. Wide rows with partition_key + clustering columns enable efficient per-partition queries. Tunable consistency per query. LWT via Paxos for compare-and-set. LSM compaction (STCS, LCS, TWCS) chosen per workload. Multi-DC active-active. Anti-entropy repair required. The strengths are linear write scale and HA; the weaknesses are no joins, no multi-row transactions, and operational complexity around tombstones and compaction.
