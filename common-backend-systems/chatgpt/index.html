ChatGPT — Conversational LLM Service (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build ChatGPT. Users type a prompt; we route it to a Large Language Model (LLM), stream the response token-by-token, persist the conversation. Each request consumes substantial GPU compute. 100M users, 100M conversations/day, peak ~10k concurrent generation streams. GPU economics and time-to-first-token (TTFT) drive the design.

1. Requirements (~5 min)

Functional - User sends a prompt within a conversation; receives a streamed response. - Conversation history persisted; resume any past conversation. - Apply conversation-level context (history + system prompt).

I'd defer per-user fine-tuning, function calling / tools, deep multi-modal inputs.

Non-functional - Time-to-first-token (TTFT) p95 < 1 s — user must see something quickly. - Tokens-per-second (TPS) ≥ 30 during streaming — feels fluid. - Sustain 10k concurrent active streams; spike to 30k. - Strong consistency on conversation history (turn ordering). - Multi-tenant isolation — one user's request doesn't see another's data.

Capacity - 100M conversations/day × ~5 turns × ~500 generated tokens = 250B tokens/day. - GPU cost dominates everything else. - Conversation storage is not the scary part: 500M turns/day × ~2 KB average persisted text/metadata is ~1 TB/day before replicas. The dominant cost is GPU memory/compute for active token generation.

2. Core Entities (~2 min)

  • User.
  • Conversation — conversation_id, user_id, system_prompt, created_at.
  • Turn — (conversation_id, idx, role, content, tokens). Append-only.
  • GenerationRequest — request_id, prompt_tokens, response_tokens, model, status.
  • Model — model_id, params, deployment, version.

3. API (~5 min)

POST /v1/conversations                                                          → { conversation_id }
POST /v1/conversations/{id}/messages   { content, model? }   (Server-Sent Events) → streamed tokens
GET  /v1/conversations/{id}                                                     → { turns[] }

SSE or chunked HTTP for streaming; WebSocket as alternative. Idempotency key on the message POST so retries don't double-bill or double-generate.

The message POST stores the user turn before routing inference and returns streamed events with monotonically increasing token_seq. A retry with the same idempotency key either reattaches to the in-flight stream or returns the finalized assistant turn.

  1. API gateway. Authenticates, applies per-tenant rate limit, runs input-side moderation, holds the SSE connection to the client. Idempotency keys deduplicate retries.
  2. Conversation service. Persists conversation metadata + turns in a sharded LSM store — Cassandra or DynamoDB, keyed by conversation_id with turn_idx as the clustering/sort key. The choice is driven by the access pattern: append-heavy writes (LSM > B-tree), reads are always "last N turns of one conversation" (single-partition range scan, <10 ms inside the TTFT budget), no joins, no cross-conversation transactions, and we need horizontal scale across billions of conversations. PostgreSQL/MySQL would need manual sharding; S3 can't hit the read latency; Redis isn't durable. Cassandra and Dynamo are interchangeable here — pick by infra.
  3. Prompt assembly. Combines system prompt + relevant conversation history + new user message into a prompt that fits the model's context window. Truncates oldest turns; optionally summarizes via a background offline job.
  4. Inference router. Picks a model deployment, finds a GPU worker. Decision uses worker load, conversation stickiness, and the prefix-cache index (which workers already hold the prompt's prefix in their KV cache).
  5. GPU worker pool. Each worker hosts a deployed model. Runs prefill (encode prompt into KV cache), then a continuous-batching decode loop generating one token per forward pass per active request. Streams each token back.
  6. Streaming safety filter. Small model classifies the output with a lookback window; passes / masks / aborts.
  7. Streaming proxy. Tokens flow from GPU worker → safety filter → router → gateway → client SSE. Gateway buffers up to ~1 MB if client is slow.
  8. Token usage / billing. Counted on stream completion (input tokens + output tokens × per-token cost). Emitted to usage pipeline.
  9. Storage. Conversation turns written as they finalize. On completion, summary + token counts persisted.

End-to-end narration. Alice opens a chat → POSTs first message with an idempotency key → conversation service writes user turn → prompt assembled (system prompt + user turn) → router consults the prefix-cache index: "system prompt blocks are on workers W3 and W7" → picks W7 (lower load) → W7 prefills (mostly cache hits on the shared system prompt blocks) → decode joins W7's continuous batch → tokens stream through the safety filter at ~50 TPS → SSE to Alice's browser. On end-of-stream, assistant turn written to conversation history; usage event emitted.

Concrete API path:

  1. Gateway authenticates, checks tenant token budget, and reserves request:{request_id} using the idempotency key.
  2. Conversation service appends the user turn to (conversation_id, turn_idx) with a conditional write so turn order is strict.
  3. Prompt assembly reads the last N turns and optional summary in parallel with feature/quota reads.
  4. Router checks conv_sticky:{conversation_id} and prefix:{prefix_hash}; it picks a worker with both KV locality and enough token budget.
  5. Worker streams tokens through the safety filter. Gateway persists a small ring buffer {request_id -> last K tokens} for mid-stream replay.
  6. On EOS, assistant turn, usage counters, and billing event are committed; the idempotency record points to the final response.

Resume / history read path:

  1. Gateway authenticates user_id and routes GET /v1/conversations/{id} by conversation_id.
  2. Conversation service reads the conversation metadata row and the latest page of turns from the same partition key conversation_id.
  3. Older pages use an opaque cursor {conversation_id, last_turn_idx, page_size, snapshot_version} so edits/deletes do not shift pagination.
  4. In parallel, the UI can fetch generated assets or attachments from object storage by turn attachment IDs; text history should not wait on large blobs.
  5. If the hot conversation partition is temporarily unavailable, the read path can serve a cached snapshot for display, but message append must fail closed or queue behind the per-conversation turn lock.

6. Deep Dives (~10 min)

GPU scheduling — continuous batching is the heart. Each GPU forward pass produces one token for every request in the batch. Static batching wastes the GPU — slowest request stalls the rest; finished requests leave empty slots. Iteration-level scheduling (vLLM's term): at every forward pass the scheduler can (a) drop a request that emitted EOS, (b) admit a new request whose prefill is ready, (c) demote a request whose KV cache no longer fits. Net effect: GPU at ~80%+ utilization vs ~30% for static batching.

Capacity math: 10k active streams × 8k context × ~300 KB/token is 24 TB of KV cache if every stream used a full 8k context, which is why admission control, prefix sharing, and shorter effective prompts matter. A 4×H100 pod with ~180 GB usable KV cache supports roughly 75 full 8k-context streams; 10k streams means ~130 such pods before headroom. This is the dominant cost; Cassandra/Dynamo storage is noise compared with GPUs.

Storage math beyond GPUs: 500M turns/day x ~2 KB x 3 replicas ~= 3 TB/day before compression, so a 90-day hot history window is a few hundred TB, still cheaper than GPU serving. Idempotency is 100M message requests/day x 200 B x 24 h ~= 20 GB before overhead. Replay buffers are bounded by concurrency: 30k active streams x 100 KB ~= 3 GB plus replication. Prefix index state is tiny; the expensive part is the worker-local KV blocks it points to.

There are two views of the same conversation data. The append-only turn log is the source of truth for audit, ordering, and user-visible history. The prompt view is a materialized serving view made from latest turns + summary + retrieved snippets; it is allowed to be lossy and regenerated, but it must always cite a snapshot_version so retries use the same context.

Subtle wrinkle: prefill (encode the whole prompt at once) and decode (one token at a time) have very different compute profiles. Mixing them naively is bad. Chunked prefill splits a long prompt's prefill across N forward passes so each iteration's compute is balanced — decode of other requests doesn't stall waiting for one giant prefill.

KV cache management (PagedAttention). During decode each request holds a key/value cache: roughly 2 × layers × kv_heads × head_dim × seq_len × dtype_bytes ≈ ~300 KB per token on a 70B FP16 model with Grouped Query Attention. At 8k context that's ~2.4 GB per request. Naive contiguous allocation (one max-sized block per request) wastes 60–80% of GPU memory.

PagedAttention (vLLM) treats KV cache like virtual memory: - Fixed-size blocks of 16 tokens in physical GPU memory. - Each request carries a page table mapping logical positions → physical blocks. - Blocks allocated on demand as generation proceeds. No external fragmentation; ~4× more concurrent requests on the same GPU. - Prefix sharing via copy-on-write. If two requests share a prompt prefix (e.g., the same 2k-token system prompt across 10k users), the prefix blocks are physically shared. Hash the prefix tokens to look up shared blocks. When a request needs to diverge (write its own token), copy that block. Net: O(1) prefix storage instead of O(users).

Eviction (evict a request's blocks, re-prefill later) is expensive — admission control should prevent it from happening.

Time-to-first-token (TTFT) — budget breakdown. Target p95 < 1 s. Components: - Auth + rate limit + input moderation: ~20 ms - Conversation history read: ~10 ms (replica read) - Prompt assembly: ~5 ms - Router → worker handoff: ~5 ms - Queue wait on worker: variable; the dominant term under load - Prefill compute: ~50 ms per 1k prompt tokens on a 70B model (linear in length) - First-token decode: ~30 ms

That leaves ~900 ms for queue + prefill. The next deep dives are the levers to keep queue + prefill within that.

Pre-warming workers — how, not just "no cold start." - Image-baked weights. Container image (or a sidecar volume) carries the model weights so they end up on node-local NVMe at boot once. Avoids pulling 60+ GB from object storage on every scale-out (which is the difference between a 30 s and a 10 min spin-up). - Warm pool / floor. Autoscaler keeps a minimum N workers always loaded per model — never scaled to zero. Floor sized to off-peak QPS. - Synthetic warmup prompts. After model load, the worker runs a handful of dummy prompts (a few hundred tokens) to JIT-compile CUDA kernels, populate the cuBLAS heuristic cache, and trigger TensorRT autotuning. vLLM does this on startup by default. Skipping warmup → the first real prompt pays 2–5× the steady-state prefill cost. - Health gate. Worker doesn't register with the router until warmup completes and a steady-state benchmark passes. New workers can't poison the latency distribution. - Pre-emptive scale-out. Autoscaler triggers a new worker when the pool hits ~70% token-budget pressure. Model load + warmup takes ~30–60 s — provisioning ahead of saturation means the worker is ready before the existing pool is full. - Candidate pool. A small set of nodes with the model already loaded but excluded from the active LB rotation. Promotion is a config flip — sub-second — versus a full provision.

Sticky routing for KV-cache and prefix locality. - Conversation stickiness. Router keeps conv_id → worker_id in a fast store (Redis, ~few-second TTL). Subsequent turns route to the same worker so its KV cache for prior turns can be reused — saves a full prefill on continuation. Sticky window is intentionally short (workers churn); on miss, the new worker re-prefills. - Prefix-cache index. Router maintains a small global index prefix_hash → [workers holding it]. On a new request, hash the prompt's prefix in 16-token windows, look up which workers already hold those blocks, and prefer one of them. Saves prefill cost even for fresh conversations that share a system prompt (most of them). - Migration. If the owner worker is overloaded, do prefix-aware migration: the new target worker likely already has part of the prefix (system-prompt blocks are universal). Way cheaper than a full request restart.

Speculative decoding (2–3× throughput on suitable workloads). A small draft model (e.g., 1B params) proposes the next k tokens (k ≈ 4–8). The target model (e.g., 70B) verifies all k in a single forward pass — the same pass it would otherwise do for one token. Rejection sampling keeps the longest prefix that matches the target's distribution. If average accepted length is n, throughput multiplier ≈ n / (k+1) × cost(target) / cost(target + draft). Big wins on structured outputs (code, factual answers); less on creative writing. Medusa variant: add multiple decoding heads to the target model itself — no separate draft model — for ~2× speedup with less infra.

Admission control by token budget, not request count. The unit of GPU work is tokens, not requests. Each worker exposes "remaining token budget" = (free KV blocks) × (block size). Router admits a request iff (input_tokens + max_output_tokens) ≤ remaining budget. If no worker has budget and the projected queue wait would breach TTFT, return 503 + Retry-After immediately — better than holding the client connection until timeout. Two queues: short-output (<200 tokens, fast lane) vs long-form (slow lane) so a chat doesn't queue behind a 4k-token essay.

Load degradation state:

normal cache-local route sticky worker or prefix hit TTFT <1 s

Cold and edge-state handling:

Edge Design response Budget
Dormant conversation resumed after summaries expired Read turn log, rebuild summary asynchronously, generate from last N turns first extra 100-500 ms before generation
Very long prompt exhausts context window deterministic truncation: system prompt, recent turns, pinned facts, then summary no silent overflow

Interview punch line: the same turn log feeds two products: exact history for the user and a compact prompt view for the model; separating those lets us optimize latency without corrupting conversation truth.

Why SSE. Server-Sent Events are enough because the server sends a one-way token stream and cancellation is just connection close. WebSocket is useful for richer bidirectional tools, but it adds connection-state complexity without improving token latency for basic chat.

Streaming protocol — SSE specifics. HTTP response with Content-Type: text/event-stream. Each token emits as data: {...}\n\n. Keepalive comments (:\n\n) every 15 s prevent intermediate proxies (cloud LBs, corporate proxies — most idle-timeout at 30–60 s) from closing the connection. Cancellation: when the client closes the connection, the gateway notifies the router, which signals the worker to drop the request from the next iteration — frees its KV cache immediately. This matters: an idle GPU slot is wasted; a freed slot serves another user. Backpressure: if the client buffer fills, TCP backpressure propagates upstream, but you cannot pause GPU decode without wasting the batch slot. Gateway buffers up to ~1 MB; on overflow, abort with reason — better than burning the slot on an unreachable client.

Streaming safety filter — how it works in band. Output passes through a small classifier (~8B-class model or distilled head) before reaching the gateway. It maintains a sliding lookback window of the last ~50 tokens, classifying every ~10 tokens. Filter latency must be smaller than the inter-token interval (~30 ms) or the stream stalls. On a hit: - Soft (mild policy violation): mask the offending span ([redacted]), continue. - Hard (severe): abort the stream, emit a safety message, log for review.

Default failure mode is closed: if the filter times out or is unreachable, mask the chunk.

Conversation context window. Context window is finite (e.g., 128k tokens). For long conversations: - Keep last N turns verbatim. - Summarize older turns into a compressed system message — done offline by a background job on a cheaper model. - For very long chats, retrieve only the most relevant past turns via vector search (retrieval-augmented).

Per-tenant fairness on shared GPUs. Token bucket on requests is wrong — the resource is GPU-seconds (i.e., tokens), not requests. Use a token-rate quota per tenant (tokens-out per second). Enforcement is in the scheduler: when a tenant exceeds its share, the scheduler deprioritizes its in-flight requests (skips them in the next iteration's decode list, costing ~1 token of latency per skip) so other tenants get serviced. Hard cap: tenant burns through 2× quota for >10 s → admission control rejects until cooldown. Priority lanes for paid tiers (paid > free; in-conversation continuation > new conversation because cache is hot).

Autoscaling signal. Not QPS. The right signal is token-budget pressure = sum(active KV bytes across workers) / total KV capacity. Scale out at ~70% pressure; lead time = pre-warm latency (~30–60 s with image-baked weights). Scale in conservatively: workers drain (no new assignments, finish in-flight, then deprovision) over a ~10 min idle threshold because spin-up is expensive.

Failure modes — real mid-stream recovery. - GPU worker dies mid-stream. Tokens already streamed are gone (client received them). Router gets a worker_disconnected event. Choices: - Replay on new worker: gateway holds the last K emitted tokens in a small ring buffer; on failure, concat (original prompt + emitted tokens) into the new prompt; new worker prefills, resumes decode where the stream left off. User sees a brief stall. - Regenerate UI: emit a [stream interrupted] sentinel and let the client click regenerate. - Production typically replays under TTFT budget; regenerates otherwise. - Router crash. Gateway sees timeout; conversation_id is durable; client retries with the same idempotency key. No double-billing. - Conversation store unavailable. Skip persistence on the write path; generate from in-flight context; queue the turn-write to a durable retry queue. Read path falls back to a cached snapshot. - Safety filter timeout. Fail closed — mask the chunk, log a filter-outage event.

Multi-model. Different requests route to different models (free vs paid tier, "fast" vs "thinking" mode). Named worker pools share the same router; routing picks the pool based on user tier + request hint. Prefix indexes and KV caches are per-pool.

Cost optimization — the ROI levers, ranked. GPU-hour is the dominant cost. 1. Continuous batching (no idle slots) — table stakes. 2. PagedAttention + prefix sharing (shared system prompt → near-zero incremental prefill). 3. Speculative decoding (2–3× decode throughput on structured outputs). 4. Quantization (INT8 / FP8) — ~2× memory, ~1.5× speed, small accuracy loss. 5. Multi-tier serving — small model handles easy queries; large model handles hard ones (router uses a query classifier).

Critical numbers worth memorizing. - KV cache per token, 70B FP16 (GQA): ~300 KB. At 8k context: ~2.4 GB per request. - 70B FP16 weights: ~140 GB → tensor-parallel across 4× H100 80GB; ~180 GB free for KV cache on that pod → ~75 concurrent 8k-context requests. - Prefill rate: ~10k input tokens/s per pod; decode rate: ~50 tokens/s per request inside a busy batch. - Model load: ~30 s from local NVMe (image-baked); 5–15 min from object storage (cold) — argues for image baking. - SSE keepalive: 15 s (most LB idle timeouts are 30–60 s).

What I'd defer. Tool / function calling protocol, plugins ecosystem, multi-modal (image input/output), per-customer fine-tuning, RLHF training pipeline, cross-region failover semantics.

Interview punch line: the product looks like chat, but the system is a GPU scheduler with a conversation store attached. Optimize for token budget, KV locality, and graceful early rejection; the text rows are the easy part.
