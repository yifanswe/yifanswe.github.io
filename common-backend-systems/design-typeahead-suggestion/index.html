Design Typeahead Suggestion

2026/06/01  ·  Common Backend Systems
Contents
Contents

New to these terms? See the Staff System Design Glossary. For an alternate framing see Design Typeahead / Autocomplete; this doc covers the same problem and design space.

1. Basic Knowledge And Solution

What This Question Is Testing

Typeahead suggestion is keystroke-driven completion: the user types a prefix and the system surfaces likely completions ranked by popularity and personalization. The hard part is sub-50ms latency per keystroke at very high QPS, refreshed popularity from query logs, and typo tolerance.

The invariant is this:

For every keystroke, the system returns relevant completions ranked by combined popularity, recency, and personalization within tens of milliseconds; the suggestion corpus reflects recent query trends within bounded staleness; typos surface relevant results.

At ICT5 / L6+ level, the interviewer is testing whether you can articulate the prefix index (trie or inverted), pre-computed top-N at each node, freshness pipeline, personalization, and fuzzy matching, plus reason about latency at scale and multi-language deployment.

60-Second Interview Framing

Spoken version:

I would design typeahead as a prefix-indexed structure (trie or inverted prefix index) backing a low-latency suggest service. Each trie node caches its top-N completions for fast lookup. Background pipelines update popularity from query logs in batch plus streaming for trends. Personalization re-ranks at the edge. The interesting decisions are the index data structure, freshness pipeline, fuzzy matching, and latency at scale.

Requirements

Functional requirements:

  • User types prefix; suggestions appear.
  • Top 5-10 ranked completions.
  • Typo tolerance.
  • Personalization.
  • Multi-language.

Non-functional requirements:

  • Sub-50ms p99.
  • Millions of QPS globally.
  • Hourly trend updates plus real-time streaming.
  • Multi-region.

Out of scope unless asked:

  • Showing search results below.
  • Knowledge graph completions.

Area Assumption

Core Product Model

Same as Design Typeahead / Autocomplete:

Fuzzy match Handles typos. Edit-distance / n-gram.

Data Model

Suggestion

Suggestion(
  text,
  popularity_score,
  category,
  language,
  is_editorial,
  last_updated
)

Trie Node

TrieNode(
  prefix,
  top_n: [(text, score)]      // pre-computed for fast lookup
)

APIs

GET /suggest?q=&lang=&user_token=
  -> [{text, score, highlight}, ...]

Lookup Path

  1. Client emits keystroke.
  2. Edge cache hit returns immediately.
  3. On miss, suggest service queries prefix index.
  4. Walk the trie character-by-character; return top-N at the matched node.
  5. Apply personalization re-rank if user_token present.
  6. Return.

Latency: under 50ms total.

Prefix Index Design

Trie with pre-computed top-N:

  • Each node represents a prefix.
  • Each node stores top-N completions for that subtree.
  • Lookup: walk prefix; return node's top-N.
  • O(prefix_length) time.

Updates rebuild affected ancestors:

  • New or updated suggestion's prefix path's nodes refresh their top-N.
  • Done by aggregation pipeline.

Compressed (PATRICIA / radix) for memory:

  • Common single-child chains collapsed.
  • Reduces memory.

Freshness Pipeline

Daily batch: - Aggregate query counts over recent window from logs. - Update popularity scores. - Regenerate top-N per trie node.

Streaming: - Real-time signal for emerging trends. - Streaming layer applies score deltas to trie. - Trending term can surface within minutes.

Personalization

Lightweight, edge-cached per-user vector.

  • Features: recent queries, locale, clicks.
  • Blend personal_affinity_score with popularity_score.
  • Latency cap: a few ms.

Privacy: opt-out respect; anonymized.

Fuzzy Matching

When exact prefix has low confidence: - Generate candidates within edit-distance 1 or 2. - Look up each; merge. - Slower; used as fallback.

Or: n-gram index where each suggestion indexed by n-grams; query n-gram overlap finds approximate matches.

Editorial Overrides

For breaking events, manual overrides take priority. Editorial layer applied before final ranking.

Multi-Region Strategy

  • Index replicated to every region.
  • Edge cache per region.
  • Personalization per-user regional cache.
  • Updates from central pipeline.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Index load slow. Pre-warm. Replica readiness.
Service timeout. Strict timeout; fall back. p99.
Stale trend. Streaming layer. Trend lag.
Personalization down. Drop personalization. Personalization availability.
Spam suggestion. Filter; manual review. Spam reports.
Typo unresolved. Fuzzy fallback. Fuzzy hit rate.
Hot prefix saturation. CDN cache. Hot prefix QPS.
Adversarial manipulation. Anomaly detection. Sudden trend signals.
Regional divergence. Centralized indexing. Divergence count.
Multi-language mixup. Locale routing. Cross-locale results.

Operational Metrics

Correctness:

  • Top-N quality.
  • Trend freshness lag.

Capacity:

  • QPS by region.
  • p99 latency.
  • Index memory.

Business:

  • Click-through rate.
  • Suggestion conversion.

Tradeoffs To Call Out

  • Trie compact; inverted easier to update.
  • Pre-computed top-N is fast but requires rebuild on updates.
  • Personalization adds latency.
  • Fuzzy is slow; fallback only.
  • More languages = more memory.

Common Traps And Stronger Moves

Trap Stronger move
DB lookup per keystroke. In-memory; sub-ms lookup.
Compute top-N per request. Pre-compute.
Synchronous personalization. Edge-cached.
No typo handling. Fuzzy or n-gram fallback.
Single global index. Sharded; multi-language.
Static suggestions. Continuous freshness.

  • In-memory prefix index (trie or inverted).
  • Pre-computed top-N per node.
  • CDN-fronted suggest service.
  • Batch + streaming popularity pipeline.
  • Edge-cached personalization.
  • Fuzzy fallback.
  • Sharded and replicated.

Interview Phrase

Typeahead is sub-50ms prefix lookup with pre-computed top-N suggestions. Trie or inverted index in memory; stateless suggest service CDN-fronted. Batch + streaming popularity pipeline keeps it fresh. Personalization blends at the edge. Fuzzy fallback for typos. The hardest constraint is per-keystroke latency at high QPS.

2. Simulated Interview

Interviewer: Design typeahead.

Candidate: Sub-50ms per-keystroke suggest backed by an in-memory prefix index (compressed trie with pre-computed top-N per node). CDN caches popular prefixes. Background pipeline aggregates query logs; popularity refresh via batch + streaming. Personalization blends at the edge. Fuzzy fallback for typos. The invariant is fast, relevant, fresh suggestions on every keystroke.

Interviewer: Why pre-compute top-N?

Candidate: Lookup latency. Walking a subtree to score and rank on every query scales with subtree size — bad for short prefixes with millions of completions underneath. Pre-computing top-N per node makes lookup O(prefix_length) plus reading a small list.

Interviewer: Updates?

Candidate: Batch aggregation runs nightly: scan recent queries, update popularity scores, regenerate top-N per trie node. Streaming layer applies real-time deltas for breaking trends. So a new viral query can surface within minutes via the streaming path.

Interviewer: Personalization?

Candidate: Edge-cached per-user vector — recent queries, locale, preferences. Blend personal-affinity score with popularity. Latency capped at a few ms. Privacy: opt-out respect, anonymized.

Interviewer: Typo — "appl" instead of "apple."

Candidate: Exact prefix lookup returns whatever exists for "appl." If results are sparse, fuzzy fallback: generate candidates within edit distance 1 ("apple"), look up each, merge. Slower path; only invoked when needed. Alternative: n-gram index.

Interviewer: Hot prefix "a" at 100k QPS.

Candidate: CDN absorbs most. Top-N for "a" is small and globally stable; cache aggressively. Origin sees only misses. Pre-warm popular prefixes.

Interviewer: Multi-language.

Candidate: Separate index per language. Locale from request determines routing. Some shared prefixes (proper names); handle by indexing in both languages.

Interviewer: Trending — how fast?

Candidate: Streaming layer monitors query velocity per term. When velocity spikes, popularity score boosted. Streaming applies delta to trie within seconds to minutes. Editorial overrides for known events.

Interviewer: Multi-region.

Candidate: Index replicated to each region from a central pipeline. Edge CDN per region. Personalization cache per region. Failover to next region with stale data if local degraded.

Interviewer: Summarize.

Candidate: In-memory prefix index — trie with pre-computed top-N — serves sub-50ms via stateless suggest service fronted by CDN. Background pipeline updates popularity (batch + streaming). Personalization blends at the edge. Fuzzy fallback for typos. Per-language and per-region replication. The hardest performance constraint is per-keystroke latency.

3. Practice Notes

Five-Minute Version

  1. Clarify: latency, freshness, typos.
  2. Invariant: sub-50ms; relevant; fresh.
  3. In-memory trie with pre-computed top-N.
  4. CDN + stateless suggest service.
  5. Batch + streaming popularity pipeline.
  6. Edge personalization; fuzzy fallback.
  7. Per-language and per-region.
  8. CTR + latency metrics.

One-Minute Closeout

Typeahead is sub-50ms prefix lookup. In-memory trie with pre-computed top-N per node makes lookup O(prefix_length). Stateless suggest service backed by CDN absorbs popular prefixes. Background pipeline updates popularity from query logs in batch plus streaming for breaking trends. Personalization at the edge. Fuzzy fallback. Per-language and per-region replication.
