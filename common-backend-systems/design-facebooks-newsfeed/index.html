1. Basic Knowledge And Solution

What This Question Is Testing

Facebook's newsfeed is the canonical personalized social feed: billions of users, complex relationships (friends, follows, groups, pages), heavy ranking with ML, and the requirement that opening the app shows fresh, relevant content in under a second.

The invariant is this:

Every post is durably stored with attribution; the home feed of each user eventually includes every post from sources they are connected to (subject to ranking, visibility, blocks, and policy); the feed reads in sub-second p99 with ranked content; deletes, blocks, and policy decisions propagate within seconds and are enforced at read.

At ICT5 / L6+ level, the interviewer is testing whether you can design hybrid fan-out for a more complex relationship graph than Twitter (friends are bidirectional, plus pages, groups, recommendations), heavy ML ranking with feature stores, integrate engagement signals, and reason about objectionable content moderation at scale.

60-Second Interview Framing

Spoken version:

I would design Facebook's newsfeed as a feed generation pipeline: aggregate candidate posts from friends, pages, and groups the user is connected to, score them with an ML ranker, apply visibility filters, return. Authoritative storage for posts is separate from the feed projection. Fan-out is hybrid like Twitter but more nuanced because relationships include pages and groups. Heavy investment in ranking and the feature pipeline. The interesting decisions are candidate generation, ranking architecture, feature freshness, and content moderation.

Requirements To Clarify First

Functional requirements:

  • User has friends (bidirectional), follows pages, joins groups.
  • Posts from friends, pages, groups appear in home feed.
  • Posts can be text, photo, video, link, check-in.
  • Like, comment, share, react.
  • Ranked by relevance, not strict chronological.
  • Visibility filters: blocks, mutes, hidden posts.
  • Stories surface separately (like Instagram).

Non-functional requirements:

  • 3B+ MAU; billions DAU.
  • Feed read p99 < 500 ms.
  • Bounded freshness (seconds to minutes).
  • Heavy ranking with sub-200 ms ML inference.
  • Multi-region.

Out of scope unless asked:

  • Marketplace.
  • Watch (videos).
  • Dating, gaming.
  • Ads (treat as separate insertion).

Candidate Generation

For each user's feed request:

  1. Gather candidate post ids from multiple sources: - Inbox-style fan-out: posts from friends pushed to user's inbox (Twitter-like, for normal friends). - Pull from celebrities/pages: large-following pages pulled at request time. - Group posts: from user's joined groups. - Recommendations: posts from non-connected sources predicted relevant.
  2. Combine into candidate set (typically thousands).
  3. Hand off to ranker.

For users with very dense graphs (1000+ friends + 200 pages + 100 groups), the candidate set is large; we prune by recency and source-level scoring before full ranking.

Ranking

Multi-stage ML pipeline:

  1. Lightweight pre-ranker: Cheap features per candidate (age, source type, basic engagement). Prune to top hundreds.
  2. Main ranker: Deep model (neural network) with extensive features: - Viewer-author affinity (interaction history). - Post engagement (likes, comments, shares). - Post type performance for this viewer. - Recency. - Personalization vectors.
  3. Post-processing: - Diversity (don't show 5 posts from same author). - Visibility filters. - Sponsored insertion (ads interleaved per policy). - Final ordering.

Latency: - Pre-rank: <50ms. - Main rank: <200ms. - Total feed gen: <500ms p99.

Interview phrase:

Ranking is two-stage: a cheap pre-ranker prunes candidates to a manageable set; an expensive deep model ranks the survivors. The feature store provides pre-computed features so inference doesn't compute from scratch.

Feature Store

Online + offline.

  • Offline: Daily/hourly batch jobs compute user embeddings, content embeddings, affinity scores.
  • Online: Real-time signals (just liked, just commented) update online feature store.
  • Ranker reads features for each (user, candidate) pair at request time.
  • Pre-computed where possible; computed inline if necessary.

Visibility Gate

Always applied AFTER candidate generation and ranking, BEFORE returning to client:

  • Blocks: filter posts from blocked users.
  • Mutes: filter posts from muted sources.
  • Hidden: posts the user hid.
  • Privacy: ensure viewer matches post's audience.
  • Moderation: removed/policy-blocked posts excluded.

Read-time enforcement; not baked into inbox.

Write Path: Post

  1. Client submits post.
  2. Post service authenticates, validates, runs auto-moderation.
  3. Assign snowflake post_id.
  4. Persist to post store.
  5. Compute audience: who can see this post (friends, public, custom).
  6. Emit PostCreated event.
  7. Fan-out workers: - For normal author: push post_id to friends' inbox candidate cache. - For high-follower author / page: skip; pull at read time.
  8. Update feature store: new content available.

Read Path: Newsfeed

  1. Client requests /home.
  2. Feed service fetches user's inbox candidates + pulls from large pages/groups.
  3. Combine candidate set (~1000s).
  4. Pre-ranker prunes to ~200.
  5. Main ranker scores and orders top ~50.
  6. Visibility gate filters.
  7. Hydrate post content from post store.
  8. Render with media URLs.
  9. Return page.

Engagement: Reactions, Comments

Same patterns as YouTube likes and Reddit comments:

  • Per-user per-post reaction (one active).
  • Sub-counter sharded reaction counters.
  • Comments flat with parent_id.
  • Engagement feeds back into feature store for future ranking.

Multi-Region Strategy

  • Per-user home region.
  • Posts replicated globally.
  • Inbox caches per region.
  • Feature store regional with cross-region rollup for global features.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Celebrity post fan-out storm. Hybrid push/pull threshold; backpressure. Fan-out lag.
Ranker model down. Fallback to pre-ranker or reverse-chronological. Ranker error rate.
Feature store stale. Use cached features; mark with timestamp. Feature freshness.
Stale deletion in inbox. Read-time visibility gate. Stale-display count.
Block bypass. Final visibility filter. Block escape count.
Inbox cache eviction. Reconstruct from post store + connections. Cache miss rate.
Cross-region replication. Async; bounded. Replication lag.
Moderation lag. Async + strong on action propagation. Time-to-block.
Cold start user. Recommendation fallback; popular content. Cold-start request rate.
Sponsored content failure. Skip ad if rendering fails; don't block feed. Ad error rate.

Operational Metrics

Correctness:

  • Visibility violations.
  • Stale post display.
  • Feature freshness.

Capacity:

  • Feed read QPS.
  • Ranker p99.
  • Inbox write rate.

Business:

  • Time spent in feed.
  • Engagement rate.
  • Sessions per DAU.

Tradeoffs To Call Out

  • Hybrid fan-out balances write/read cost.
  • Deeper ML model gives better ranking but adds latency.
  • Larger candidate set finds more relevant content but costs ranking compute.
  • Pre-computed features speed inference but add storage.
  • Aggressive personalization may create filter bubbles.
  • Strict moderation may suppress legitimate speech; lenient lets bad content through.

Common Traps And Stronger Moves

Trap Stronger move
Reverse-chronological. ML-ranked with diversification.
Single ranking pass. Multi-stage: pre-ranker + main ranker.
Filter blocks at write. Read-time visibility gate.
Synchronous feature computation. Feature store with online + offline.
Pure fan-out on write. Hybrid for celebrities/pages.
Synchronous moderation. Auto-mod at write + async deeper scan.

Staff-Level Answer Checklist

  • Hybrid fan-out with multi-source candidate generation.
  • Multi-stage ML ranking with feature store.
  • Read-time visibility gate.
  • Engagement aggregation separate from feed.
  • Multi-region with per-user home.
  • Moderation as continuous pipeline.

Interview Phrase

Facebook's newsfeed is hybrid fan-out plus heavy ML ranking. The feed isn't reverse-chronological — it's an ML-ranked composition of candidate posts from friends, pages, groups, and recommendations. Multi-stage ranking prunes candidates and then scores them with a deep model. Visibility filtering happens at read, never baked into the inbox.

2. Simulated Interview

Interviewer: Design Facebook's newsfeed.

Candidate: Pipeline: candidate generation, multi-stage ranking, visibility gate, render. Candidates come from multiple sources — friends' posts, pages followed, groups joined, recommended content from non-connected sources. Pre-ranker prunes; main ML ranker scores; visibility gate filters; render hydrates. The invariant is bounded-staleness display of ranked content from sources the user is connected to, with policy and privacy correctly enforced.

Interviewer: Candidate generation — how does it work?

Candidate: Multi-source. Friends' posts arrive via hybrid fan-out (normal friends push to inbox; very-active friends with thousands of friends-of-friends pull at read). Pages and groups: push for small followings, pull for large. Recommendations: a separate service surfaces predicted-relevant posts from non-connected sources. Combined candidate set is thousands of posts; pruned to manageable size before full ranking.

Interviewer: Ranking?

Candidate: Two stages plus post-processing. Pre-ranker is cheap: features per candidate (age, source type, basic engagement). Prunes to top ~200. Main ranker is a deep neural network with extensive features from the feature store: viewer-author affinity, post engagement, recency, post type performance for this viewer, personalization vectors. Scores top ~50. Post-processing: diversity (don't repeat same author), visibility filters, ad insertion, final ordering.

Interviewer: Feature store?

Candidate: Online + offline. Offline: daily batch builds user embeddings, content embeddings, affinity scores. Online: real-time signals (just liked, just commented) update features within seconds. Ranker reads features for each (user, candidate) pair at request time. Pre-computed where possible; computed inline if necessary. Feature freshness directly affects ranking quality.

Interviewer: Visibility — how do blocks work?

Candidate: Read-time gate. After ranking but before returning to the client, the visibility service filters: blocked users' posts, muted sources, posts the user hid, audience-mismatched posts (e.g., friends-only post from a non-friend due to a graph inconsistency), policy-removed posts. Never bake blocks into the inbox; the inbox is just candidate ids, and visibility is final-authority.

Interviewer: Posts arrive while you're scrolling. How do you keep the feed fresh?

Candidate: Inbox cache continuously updated as friends post. The next feed pull picks up fresh candidates. For active sessions, the client can poll for "new content" indicator that triggers a re-fetch. For real-time feels, push notifications can announce significant posts.

Interviewer: Celebrity page with 100M followers posts.

Candidate: Pulled at read time, not pushed. The page's recent posts are cached centrally; followers' inboxes are not written for celebrity content. When a follower opens the feed, candidate generator queries the celebrity's recent posts cache. Saves 100M inbox writes per post.

Interviewer: Posts that violate policy?

Candidate: Auto-moderation runs at write — basic checks (banned content, hate speech classifiers). Higher-confidence detections suppress immediately. Async deeper scan + human review for subtle cases. On takedown, post status transitions to removed; cache invalidates; read-time visibility gate filters. Time-to-takedown is a critical operational metric.

Interviewer: Engagement counters at this scale?

Candidate: Same pattern as YouTube likes. Per-user per-post reaction state is authoritative; per-post counter is derived. Sub-counter sharding for hot posts. Streaming aggregation. Periodic recount for correctness.

Interviewer: Multi-region?

Candidate: Per-user home region for primary writes. Posts replicated globally. Inbox caches per region. Feature store regional with cross-region rollup for global features (e.g., global trending). Reads come from local region.

Interviewer: Cold start user — just joined, no friends, no engagement history.

Candidate: Default to popular content within their region and stated interests. Recommendation engine surfaces "people you may know" and pages to follow. Heavy emphasis on bootstrapping affinity through early interactions. The feature store has cold-start fallbacks.

Interviewer: Summarize.

Candidate: Hybrid fan-out into per-user inbox candidate cache, plus pull from large sources. Multi-stage ML ranking — cheap pre-rank then deep model — with feature store providing pre-computed features. Read-time visibility gate enforces blocks, privacy, moderation. Engagement counters sub-sharded. Multi-region with per-user home. The hardest scaling problem is candidate generation across a dense graph; the hardest correctness problem is visibility enforcement at read.

Facebook's newsfeed is hybrid fan-out plus heavy ML ranking. Candidate generation pulls from friends, pages, groups, and recommendations.
