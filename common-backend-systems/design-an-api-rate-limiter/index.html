1. Basic Knowledge And Solution

What This Question Is Testing

A rate limiter sits on the hot path of every request and must decide allow-or-deny in single-digit milliseconds. The hard part is keeping that decision globally consistent enough to protect downstreams while not becoming the bottleneck it is supposed to prevent.

The invariant is this:

For any rate limit key (user, IP, API key, route, tenant, or combination), the number of allowed requests in any sliding window must not exceed the configured quota, even when traffic is distributed across many edge nodes and clients retry aggressively.

At ICT5 / L6+ level, the interviewer is testing whether you can pick the right counting algorithm for the freshness vs cost tradeoff, design a distributed counter that survives hot keys, and explain what happens when the limiter itself becomes unavailable.

60-Second Interview Framing

Spoken version:

I would design the rate limiter as a sidecar or inline middleware on every API node, backed by a fast distributed counter store. The interesting decisions are which counting algorithm to use (fixed window, sliding window, token bucket, leaky bucket), how to keep counts consistent across edges without paying a full round trip per request, and how the system fails open or closed when the counter store is degraded. I will clarify requirements, pick token bucket as default, walk the data path, then go deep on hot keys and consistency.

Requirements To Clarify First

Functional requirements:

  • Limit requests per identity (user, API key, IP, tenant) and per route.
  • Support multiple limit windows simultaneously (e.g., 100/sec, 1000/min, 10000/hour).
  • Different identities may have different quotas; enterprise customers get higher limits.
  • Return clear feedback: 429 Too Many Requests with Retry-After and quota headers.
  • Allow short bursts above sustained rate (burst allowance).
  • Allow admins to configure, raise, or revoke quotas at runtime without redeploy.

Non-functional requirements:

  • Allow-or-deny decision latency in single-digit ms at p99.
  • Must not become a single point of failure for the API.
  • Must scale to millions of distinct keys.
  • Must handle hot keys (one tenant or IP receiving disproportionate traffic).
  • Auditability for denied requests and quota changes.

Out of scope unless interviewer asks:

  • DDoS at the network layer (handled before reaching API rate limiter).
  • WAF rules and content-based blocking.
  • Business-tier billing and metering.
  • Concurrency limits (max in-flight) vs rate limits (per unit time) — clarify whether both are needed.

Scale Assumptions

Area Assumption
API traffic 1M requests/sec global peak.
Distinct keys 100M+ active rate limit keys (per user × per route combinations).
Hot keys Top 1% of keys account for 50%+ of traffic.
Quota updates Thousands per day; mostly automated tier changes.
Limiter latency budget Under 5 ms p99 added to the API path.
Accuracy tolerance Within 1-5% of the configured limit at the boundary, depending on algorithm.
Fail mode Configurable per route: fail open (allow) for non-critical, fail closed (deny) for sensitive.

The bottleneck is not the count store; it is hot keys. A single user driving 100k requests per second pins one shard.

Downstream service Trusts that traffic is within agreed quota. Limiter must not over-allow under normal operation.

Core Components

Component Responsibility Key detail
Edge / API gateway Terminates TLS, applies coarse IP rate limits before reaching the application. Cheap first-level filter; uses simple fixed-window counters in memory.
Rate limiter middleware Per-request allow/deny decision; runs in API process or as sidecar. Local token bucket replicas for hot keys; falls back to counter store.
Counter store Authoritative distributed counter for each (key, window). Typically a sharded in-memory KV like Redis cluster; sub-ms reads/writes.
Quota config service Stores per-identity and per-route quotas; supports updates. Pushes via pub/sub or polled with versioning.
Algorithm Token bucket, leaky bucket, fixed window, or sliding window log/counter. Choose per route based on burst tolerance and accuracy needs.
Denial log Captures every 429 with key, reason, quota, current count. Async; used for analytics and abuse detection.
Admin API CRUD on quotas; emergency overrides. Audited, signed, role-restricted.
Fallback decision What happens when counter store is unreachable. Configurable per route: fail open or fail closed.

Algorithms (Pick The Right One)

Algorithm How it works Pros Cons
Fixed window Count requests per fixed time bucket (e.g., per minute). Reset on bucket boundary. Simplest; one counter per key. Burst at window boundary can deliver 2x quota across two windows.
Sliding window log Store timestamp of every request; count those within the window. Most accurate. Memory cost scales with traffic.
Sliding window counter Weighted blend of current and previous fixed windows. Cheap; approximates sliding window well. Slight inaccuracy near limits.
Token bucket Bucket of N tokens refilled at rate R; each request consumes a token. Allows controlled bursts up to bucket size; intuitive for "max sustained rate". Need atomic decrement plus refill calculation.
Leaky bucket Queue with fixed leak rate; requests join queue or get dropped. Smooths outgoing rate; good for shaping. Adds latency by queueing.

Default recommendation: token bucket for client-facing APIs because the policy is human-readable and burst is a first-class concept. Combine with multiple windows for the same key: e.g., a 100/sec bucket and a 10000/hour bucket, both must allow.

Interview phrase:

I would use token bucket as the default because the policy is human-readable and burst is a first-class concept. I add multiple buckets at different windows for the same key when both short and long protections are needed.

Data Model

Quota Configuration

Quota(
  scope,                // global, route, tenant, user, ip, api_key
  key_pattern,          // route_id, tenant_id, user_id, ...
  algorithm,            // token_bucket, sliding_window, ...
  capacity,             // bucket size or window count
  refill_rate,          // tokens per second
  window_size,          // for window-based algorithms
  priority,             // tie-break when multiple rules match
  fail_mode,            // open, closed
  version
)

Token Bucket State

Stored in the counter store, keyed by bucket_key = hash(quota_id, identity):

BucketState(
  bucket_key,
  tokens,               // float
  last_refill_ts,       // milliseconds
  version               // optional CAS token
)

For sliding window counter:

WindowState(
  bucket_key,
  current_window_id,
  current_count,
  previous_count
)

Denial Record

DenialEvent(
  ts,
  key,
  identity,
  route,
  quota_id,
  reason,                // exceeded_burst, exceeded_sustained, ...
  retry_after_ms
)

APIs

Internal limiter check (RPC or in-process):

CheckRequest {
  identities: [user_id?, api_key?, ip?, tenant_id?]
  route_id
  cost?            // default 1; expensive endpoints can charge more
}

CheckResponse {
  decision: ALLOW | DENY
  remaining: integer
  reset_at: timestamp
  retry_after_ms: integer  // when DENY
  reason: string           // when DENY
}

External response on deny:

HTTP 429 Too Many Requests
Headers:
  Retry-After: 12
  X-RateLimit-Limit: 100
  X-RateLimit-Remaining: 0
  X-RateLimit-Reset: 1700000000

Admin API:

PUT /quotas/{scope}/{key}
GET /quotas/{scope}/{key}
GET /denials?identity=&route=&since=

Token Bucket Implementation Detail

The classic atomic operation:

on each request:
  state = read BucketState(bucket_key)
  now = current_time_ms
  elapsed = now - state.last_refill_ts
  tokens_to_add = elapsed * refill_rate / 1000
  state.tokens = min(capacity, state.tokens + tokens_to_add)
  state.last_refill_ts = now
  if state.tokens >= cost:
    state.tokens -= cost
    write BucketState (CAS or atomic Lua)
    return ALLOW
  else:
    return DENY with retry_after = (cost - state.tokens) / refill_rate

Two implementation choices:

  • Redis with Lua script: atomically read-modify-write the bucket in a single round trip. Each API node calls the script.
  • Local approximation + async sync: each node maintains a local token bucket per hot key and periodically syncs to the central counter store with a delta. Sacrifices accuracy for latency and counter store load.

For hot keys: local approximation is essential. The synchronous central-store version cannot handle 100k QPS on one key.

Hot Key Strategy

A single tenant burning their quota at 100k req/sec pins the Redis shard holding that key.

Mitigations:

  • Local rate limiting first. Each API node holds a local token bucket per key. Sized as local_capacity = total_capacity / N where N is the number of nodes. Bursts allowed locally without coordinating.
  • Per-node sub-quotas. Pre-distribute quota across nodes statically or via gossip; nodes only contact the central store when their slice approaches exhaustion.
  • Hierarchical limiters. Edge node enforces 80% of the global quota locally; central enforces the last 20% for cross-node bursts.
  • Sharding the key. For a single very-hot key, split it into N sub-keys at the limiter; each sub-key carries capacity/N, and requests are hashed to a sub-key. Accept that one heavy client might be slightly throttled even when their global quota is not exhausted.
  • Replica reads. For algorithms tolerant of slight lag, read from local replicas and reconcile in the background.

Interview phrase:

The central counter store is a fallback for accuracy, not the inline path for hot keys. I push enforcement to the edge with local buckets and reconcile asynchronously, accepting bounded over-allow to protect availability.

Write And Read Path

Request path (every API call):

  1. API node receives request.
  2. Resolve identities: extract user from auth token, route, tenant, IP.
  3. Look up applicable quotas in local quota cache (refreshed from quota service every few seconds).
  4. For each quota, check the local token bucket; if any returns DENY, return 429.
  5. If all locals allow, optionally sync with central counter store for the high-priority quotas.
  6. Forward request to the API.
  7. Emit a denial event if denied; otherwise no log on the synchronous path.

Quota config update path:

  1. Admin updates quota via admin API.
  2. Quota service writes to a strongly consistent store.
  3. Publishes a quota-version-bumped event on pub/sub.
  4. Limiter nodes pick up the new version on their next poll or push, with a version number, then atomically swap their local rules.

Scaling And Partitioning

Data Partition key Why
Counter store buckets Hash of bucket_key across Redis cluster shards. Even distribution; supports atomic per-key operations.
Quota config Replicated across all limiter nodes. Read-heavy, small, infrequent changes.
Denial log Time-bucketed; partitioned by route or tenant. Volume can be huge under attack; useful for after-the-fact analysis.
Hot key splits Logical key bucket_key:shard_id for known hot keys. Spreads load across shards.

Consistency Model

Decision Consistency Reason
Allow/deny on a normal key Eventually consistent across nodes (drift of milliseconds to seconds) Acceptable to over-allow by a few percent.
Allow/deny on a hot key Local-first; central as backstop Cannot afford a synchronous round trip per request.
Quota configuration Strong via versioned config Operators expect changes to take effect predictably.
Denial logging Eventually consistent Used for analytics, not decision making.

Accept that distributed rate limiting is approximate by nature. The contract is "we will not allow significantly more than the configured rate," not "we will not allow exactly one more request."

Failure Modes And Fail Mode Choice

Failure Mitigation Failure mode default
Counter store unreachable Local buckets continue serving; central counts catch up on recovery. Fail open for low-tier routes; fail closed for billing/auth/security.
Counter store hot shard Pre-shard hot keys; local approximation. Fail open with degraded accuracy.
Quota config service down Limiters use last known config. Effectively frozen but functional.
Clock skew between nodes Use server-time for refill; cap drift; rely on monotonic clock for local. Slight inaccuracy at refill boundaries.
Limiter middleware crash Sidecar restart; in-process middleware fails as part of API. Same fate as API process.
Coordinated burst at window boundary (fixed window flaw) Switch to sliding window or token bucket. Algorithm choice.

Choosing fail open vs fail closed is a product policy decision: payment endpoints fail closed; status pages fail open.

Multi-Region Strategy

For a global API, quota typically means a single global rate.

Two patterns:

  • Per-region quota (split N ways): simple, no cross-region coordination, but a quiet region "donates" capacity to a busy region only if you reallocate.
  • Global quota with hierarchical enforcement: each region enforces its baseline share locally; a global reconciler tracks total usage. If one region's share is unused, others can request a temporary lease.

For most APIs, per-region splits are good enough. Cross-region accuracy under bursts is hard and rarely required.

Operational Metrics

Correctness metrics:

  • Actual rate vs configured limit per key (should hover at or under limit).
  • Over-allow ratio: ratio of actual / configured during sustained traffic.
  • Quota config version skew across limiter nodes.

Capacity metrics:

  • Counter store QPS, latency, shard distribution.
  • Limiter middleware decision latency.
  • Top-N hot keys by QPS and denial rate.

Business and security metrics:

  • Denial rate by route, tenant, identity.
  • 429-to-200 retry success rate.
  • Spike detection: sudden increase in denials for one identity (possible bot or compromise).
  • Quota change audit trail.

Tradeoffs To Call Out

  • Sliding window log is most accurate but expensive; sliding window counter trades a few percent of accuracy for orders of magnitude less memory.
  • Token bucket allows bursts; leaky bucket smooths them. Pick based on whether burstiness is welcome.
  • Synchronous central counter is accurate but limits throughput; local approximation is fast but slightly over-allows.
  • Aggressive limits protect downstream but generate user complaints; lenient limits protect UX but expose downstream.
  • Per-IP limits hurt users behind NAT or large corporate proxies; per-API-key limits require authentication on the rate-limit path.
  • Fail open is friendly but lets abuse through; fail closed protects downstream but causes incidents to widen.

Common Traps And Stronger Moves

Trap Stronger move
Using fixed window only. Mention the boundary burst flaw; pick sliding window or token bucket.
One Redis instance for all counters. Sharded cluster; treat the cluster itself as a tier and design for shard saturation.
Synchronously round-tripping to Redis per request. Local buckets with async reconcile; hierarchical enforcement.
Ignoring the quota update path. Versioned config with explicit propagation and audit.
Forgetting failure modes. State fail open or closed per route and justify.
Treating all keys equally. Recognize hot keys, pre-shard or use approximate local buckets.

Deep Dives Interviewer May Choose

Hot key handling Local-first enforcement; pre-shard; gossip-based capacity reallocation.

Staff-Level Answer Checklist

  • State the invariant: do not exceed configured rate per key, even under retries.
  • Pick an algorithm and justify it.
  • Distinguish hot path enforcement from accurate central counting.
  • Explain hot key handling with local buckets and pre-sharding.
  • Cover the quota configuration plane separately from the data plane.
  • State fail open or fail closed per route.
  • Discuss multi-region quota splitting.
  • Include denial logging as an async side channel for security.

Interview Phrase

A distributed rate limiter is an approximation problem, not an exact-counting problem. I would accept a few percent over-allow in exchange for sub-millisecond decisions and resilience to counter-store failures, and I would pay for accuracy only where it matters: billing, security, and abuse blocking.

2. Simulated Interview

Interviewer: Design an API rate limiter.

Candidate: I will design it around a per-request allow/deny middleware backed by a distributed counter. The invariant is that for any rate limit key — user, API key, IP, tenant, route, or combination — we do not significantly exceed the configured quota across the fleet. I will start by clarifying requirements, pick an algorithm, walk the data path, and dive into hot keys and consistency.

Interviewer: What algorithm would you use?

Candidate: Token bucket as the default. The product semantics are intuitive: a sustained refill rate and a burst capacity. For accuracy near the limit at long windows, I sometimes layer in a sliding window counter as a second bucket. Fixed window is simplest but has the boundary burst problem where you can get 2x the limit across the window boundary, so I avoid it for client-facing APIs.

Interviewer: Where does the counter live?

Candidate: Logically, in a sharded fast KV like Redis cluster, accessed via a Lua script that atomically performs the read-modify-write. Practically, each API node holds a local token bucket per active key, and the central counter is a backstop for accuracy. For hot keys, every node enforces from its local slice without round-tripping; otherwise, the central store would be the bottleneck.

Interviewer: What if a single tenant generates 100,000 requests per second?

Candidate: That is the hot key problem. A few mitigations: first, distribute the quota across API nodes so each node enforces its share locally. Second, split the key into sub-shards: hash the request to one of N logical buckets, each with capacity/N. The user might be slightly throttled even when not at global cap, which is an acceptable tradeoff. Third, if the central store starts saturating, the limiter falls back to local-only enforcement and accepts the over-allow.

Interviewer: How accurate is this?

Candidate: With local-first enforcement and asynchronous reconciliation, you can over-allow by a few percent during bursts, especially right after a sudden traffic spike. That is acceptable for almost all rate-limit use cases. If a route needs strict counting — billing meters, security blocks — I send those through a synchronous Redis Lua script and accept higher latency.

Interviewer: What if Redis goes down?

Candidate: The middleware falls back to local buckets. Configuration matters here: for non-critical routes, fail open and accept some over-allow; for billing and auth, fail closed. Each route has a fail-mode policy. The denial log captures the fallback so we can audit after the fact.

Interviewer: How do you handle quota updates?

Candidate: Quota config is a separate plane from the data plane. An admin API writes to a strongly-consistent store, bumps a version, and publishes a change event on pub/sub. Limiter nodes either subscribe or poll; they swap to the new quota atomically with the new version. Operators can verify propagation by querying the active version on each node.

Interviewer: What status code and headers do you return?

Candidate: 429 Too Many Requests with Retry-After, X-RateLimit-Limit, X-RateLimit-Remaining, and X-RateLimit-Reset headers. The retry-after is computed from the token bucket as (cost - tokens) / refill_rate. Clients can use it to back off precisely.

Interviewer: What about IPs behind NAT or corporate proxies?

Candidate: Per-IP limits unfairly hurt users behind NAT — a whole office gets one quota. So I would compose limits: a coarse per-IP guard for anonymous abuse protection, and a per-authenticated-identity limit for the real product quota. Inside the API, I prefer identity-keyed limits whenever possible.

Interviewer: Multi-region?

Candidate: Easiest is per-region quota splits: each region gets a slice of the global quota and enforces locally. No cross-region coordination, but a quiet region's unused capacity is wasted. If the product requires a single global quota, I add a global reconciler that tracks usage and reallocates leases across regions periodically — adds operational complexity for marginal accuracy gain. Most APIs are fine with per-region splits.

Interviewer: How do you support different tiers — free, pro, enterprise?

Candidate: The quota config matches identity attributes to algorithms and capacities. When a request comes in, the limiter resolves the tier from the auth context and looks up the applicable quotas. Tier changes are config changes; the next config refresh picks them up. For very high-volume enterprise customers, I might pre-shard their key to avoid pinning a single Redis shard.

Interviewer: How do you detect a customer abusing their quota?

Candidate: Denial events stream to an analytics pipeline. Patterns include sustained 100% denial rate (they are constantly at limit), sudden burst from a previously quiet identity (compromise), or denial concentration on suspicious routes. Alerts feed into security review and possibly auto-revoke API keys.

Interviewer: Summarize the design.

Candidate: Token bucket as the algorithm, local-first enforcement on every API node with a sharded Redis backstop for accuracy. Hot keys split into sub-buckets. Quota configuration is a separate versioned control plane. Fail-mode policy per route. Multi-region uses per-region quota splits by default. The whole design accepts that distributed rate limiting is approximate by nature and optimizes for fast decisions and survivability, not exact counts.

3. Practice Notes

Five-Minute Version

  1. Clarify: limit per identity, per route, multiple windows, fail mode.
  2. State invariant: do not significantly exceed configured rate.
  3. Pick token bucket; justify vs fixed/sliding window.
  4. Show the data path: local bucket → optional central Redis check.
  5. Address hot keys with local-first and key sharding.
  6. Show quota config plane separate from data plane.
  7. Discuss fail-open vs fail-closed per route.
  8. Close with multi-region per-region split and denial logging.

One-Minute Closeout

A rate limiter is an approximation system. I would use token buckets for human-readable quotas, push enforcement to local in-process buckets to keep the hot path fast, and use a sharded Redis cluster as a backstop. Hot keys get pre-sharded; quota config has its own versioned plane. Fail mode is route-specific. The main tradeoff is exact counting vs availability, and I optimize for availability with a few percent acceptable over-allow.

Best Interview Phrases

  • "Distributed rate limiting is an approximation problem, not an exact-count problem."
  • "Token bucket is the default because burst is a first-class concept."
  • "Local-first enforcement; the central store is a backstop, not the inline path."
  • "Hot keys break naive designs; pre-shard the key or split it logically."
  • "Fail-mode is a per-route policy decision, not a system-wide setting."
