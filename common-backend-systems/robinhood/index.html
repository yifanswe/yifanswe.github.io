Robinhood — Trading Platform (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build Robinhood. Users place stock orders (market, limit), see live price quotes, view portfolio P&L, and the system routes orders to a market maker / exchange. 30M users, 100k orders/sec peak, real-time quotes for ~10k tickers from market data feeds at thousands of ticks/sec per popular ticker.

1. Requirements (~5 min)

Functional - Place market and limit orders for a listed stock. - See live quote and your portfolio's live P&L. - View order history and execution details.

I'd defer options trading, crypto, fractional shares deep dive, and tax-lot accounting.

Non-functional - Correctness is the headline — no lost orders, no double-fills, no incorrect balances. - Quote latency to client < 200 ms; order acknowledgment < 500 ms; execution + settlement is exchange-bound. - Strong consistency on cash/position balances at all times. - Regulatory: durable audit log of every action. - Survive market-open spike: 100k orders/sec is the design target.

Capacity - 30M users; ~1M trading on any day; 5–10M orders/day peak; 100k/sec at open. - Market data: 10k tickers × ~10 ticks/sec average, hot tickers thousands/sec → ~1M ticks/sec aggregate ingest. - Order rows are large but manageable: 10M orders/day × ~1 KB × 7 years × 3 replicas ≈ 76 TB before indexes. - Raw market data can dominate if retained: 1M ticks/sec × ~100 B ≈ 8.6 TB/day raw, so keep hot quotes in Redis and compact/archive tick history aggressively.

2. Core Entities (~2 min)

  • User / Account — user_id, cash_balance, positions.
  • Position — (user_id, ticker) → shares, avg_cost.
  • Order — order_id, user_id, ticker, side, qty, type, limit_price, status, created_at.
  • Execution / Fill — fill_id, order_id, qty, price, ts.
  • Quote — (ticker) → bid, ask, last, ts.

3. API (~5 min)

POST  /v1/orders                       { ticker, side, qty, type, limit?, idempotency_key }   → { order_id, status }
GET   /v1/orders/{id}
DELETE /v1/orders/{id}                  // cancel
GET   /v1/portfolio                     → { cash, positions[], pnl_live }
WS    /v1/quotes                        → subscribe { tickers[] }
WS    /v1/orders                        → subscribe to user's order events

Idempotency key mandatory on order placement.

5. High Level Design (~10–15 min)

  1. Order entry service. Accepts orders, validates (enough buying power, market hours, position size limits), writes order row (status=accepted), publishes to order routing.
  2. Pre-trade risk check. Single transactional step against the user's cash and positions: cash_balance ≥ order_value for buys; shares_held ≥ qty for sells. Implemented as a row update with WHERE clause to atomically reserve cash/shares. This is the most important consistency boundary.
  3. Order routing. Routes the order to a broker-dealer or exchange via standard protocols (FIX). For Robinhood specifically, routes to market makers for payment for order flow (PFOF). The route response can be: immediate fill, queued, partial fill, or rejected.
  4. Execution events. Fills come back asynchronously. Each fill updates the order (partially_filled or filled) and adjusts position + cash atomically.
  5. Position service. Owns positions and cash. Single-writer per account ensures correctness. On each fill, applies the delta in one transaction.
  6. Market data pipeline. Ingest from exchange / SIP feed (FIX/ITCH) → publish to Kafka topic per ticker → quote service maintains current best bid/ask in Redis → fan out to client WS subscribers.
  7. Portfolio + P&L. Computed real-time on read: cash + Σ(position.qty × current_price). Current price comes from the quote service. Cached briefly to dampen churn.
  8. Audit log. Every order event appended to an immutable log (Kafka with very long retention + S3 cold storage) for regulatory replay.

End-to-end narration: market opens at 9:30:00 ET → Alice places "buy 10 AAPL market" → order service validates buying power → atomic reserve $1750 from cash, write order row, publish to router → router sends FIX message to market maker → fill comes back in 200 ms at $174.95 × 10 → execution event updates position (+10 AAPL @ $174.95) and cash (release $50 over-reserved). Alice's portfolio WS sees the order fill, the position update, and updated P&L within 1 s.

6. Deep Dives (~10 min)

Concrete stores and shard keys.

Store Key pattern Data structure / entry TTL / retention Shard key
Account balance acct:{user_id} strongly consistent row, cash/buying power/version, ~1 KB 7 y+ user_id
Positions pos:{user_id}:{ticker} row with qty/cost basis/reserved qty, ~150 B 7 y+ user_id
Orders orders:{user_id}:{order_id} order state row, ~1 KB 7 y user_id
Executions fills:{order_id}:{fill_id} immutable fill row, ~500 B 7 y order_id
Idempotency order_idem:{user_id}:{key} response/status pointer, ~300 B 24–72 h user_id
Current quote quote:{ticker} Redis hash bid/ask/last/ts, ~100 B TTL 5–30 s ticker
Quote subscribers subs:{ticker} ephemeral connection IDs, ~16 B/sub while connected ticker
Audit log audit/orders/yyyy/mm/dd/{partition} append-only event, ~500 B 7 y+ date + partition

Cash/positions are authoritative content; quote Redis is a freshness cache. If quotes go stale, balances do not become stale.

Capacity by store. Account state is not the size problem; correctness is. 30M accounts × 1 KB × 3 replicas is only ~90 GB, and even 30M users × 20 positions × 150 B × 3 is ~270 GB. The heavier stores are histories: 10M orders/day × 1 KB × 7 years × 3 is ~76 TB, and if fills average two per order, 20M fills/day × 500 B × 7 years × 3 is another ~76 TB. Audit can be larger because every order has multiple lifecycle events: 10M orders/day × 6 events × 500 B × 7 years × 3 is ~230 TB. Market data dwarfs all of that if retained raw: 1M ticks/sec × 100 B is ~8.6 TB/day before replication, so I keep only current quotes hot, retain short tick windows for replay, and compact/archive historical ticks separately.

Place-order path. 1. Client posts order with idempotency key; gateway authenticates and checks order_idem:{user_id}:{key}. 2. Order entry loads acct:{user_id} and relevant pos:{user_id}:{ticker} on the same account shard. 3. Pre-trade risk reserves cash/shares with a conditional update. This is the consistency boundary; no FIX message is sent before reservation commits. 4. Order row orders:{user_id}:{order_id} is written as accepted, and an outbox event routes to the FIX adapter. 5. Router sends to market maker/exchange. Fills arrive async and append fills:{order_id}:{fill_id}. 6. Position service applies each fill to account cash/position in account-shard order, releases over-reservation, and emits user WS events.

Quote read path. 1. Market data feed handler normalizes ticks and publishes by ticker. 2. Quote service updates quote:{ticker} and pushes latest ticks to regional fan-out. 3. Portfolio read fetches positions and current quotes in parallel, computes unrealized P&L, and caches the result for ~1 s.

Hot and cold API paths.

The order cursor is the order_id; the streaming cursor is a per-connection last event sequence so reconnect can replay missed order events from the user-event log.

Pre-trade risk — the correctness centerpiece. All cash and position reservations happen on a single row per account with optimistic concurrency. A buy order:

UPDATE accounts
SET cash = cash - 1750, pending_orders = pending_orders + 1
WHERE user_id = X AND cash >= 1750;

On fill, release the reservation against actuals. On reject, refund.

Account sharding. Sharded by user_id so all of a user's orders, positions, cash live in one shard. Avoids distributed transactions across shards.

Why single-writer/account-shard instead of a globally distributed transaction: most orders touch exactly one account and one ticker, so localizing cash and position state by user_id gives stronger correctness with less latency. Cross-account settlement and broker reconciliation happen asynchronously; they should not be in the order-entry p99 path.

Tech choice defense. For account and position state, I want a strongly consistent SQL/Spanner-style store because conditional updates, unique order IDs, and audit joins are first-class. Cassandra can absorb writes, but lightweight transactions would become the risky part of the money path. Redis is useful for quotes and ephemeral state, not as the source of truth for balances. For routing, a durable outbox/Kafka bridge beats direct synchronous FIX calls from the order transaction because a broker timeout should not erase an accepted order; the outbox gives replay with exactly-once effect at the router boundary.

Be ready for "why not one giant event log?" The log is perfect for audit and replay, but the serving path still needs current account rows so a pre-trade risk check does not scan history.

Market data fan-out at 1M ticks/sec. Single quote-service node can't push to millions of WS clients. Pyramid: - Ingest tier writes ticks to Kafka per ticker. - Fan-out tier (regional) subscribes to relevant tickers, pushes to WS gateways. - WS gateways hold connections and forward the right tickers to each subscriber. Client subscribes to specific tickers, not the firehose.

Quote update batching. AAPL during open updates 1000×/sec. No human needs that refresh rate. WS layer rate-limits per-ticker push to clients to ~5 Hz, taking the most recent tick. Mid-trip ticks are dropped — fine.

Order routing protocol. FIX over TLS to broker-dealer / market maker. Latency-sensitive: keep persistent FIX connections, route by ticker class. For self-clearing brokers, would also handle the clearing/settlement (T+1 cycle).

Cancellation race. User cancels right as fill arrives. Atomic order row update with state check: UPDATE orders SET status='canceled' WHERE id=X AND status='accepted' — only succeeds if not already filled. If fill came first, cancel returns "too late."

Hot tickers at market open. All users want AAPL/TSLA at 9:30 sharp. Order entry has to absorb the spike. Mitigations: - Pre-warm worker pools and Redis cache. - Queue orders briefly at the routing tier; emit 202 Accepted with order_id, let the WS push the eventual fill. - For limit orders, return immediately and let the exchange handle the queueing.

P&L computation. Realized P&L from filled orders (deterministic, tax-lot aware). Unrealized P&L = current price × shares - cost basis. Recomputed on each quote update for the user's open positions; cached for ~1 s.

Two-index framing. Order events are the same dataset presented two ways: user-keyed rows (orders:{user_id}:{order_id}) for app history and time/partition-keyed audit logs for regulatory replay. Quotes also have two views: content by ticker (quote:{ticker}) and subscriber membership by ticker (subs:{ticker}). Calling this out matters because I am not inventing separate truths; I am keeping one truth and multiple serving indexes.

Regulatory and audit. Every order event written to an append-only Kafka topic with 7-year retention (regulatory minimum). Reconciliation jobs compare positions vs broker statements daily.

Outages and degradation. If market data fails, freeze quote-display age timer in UI and refuse market orders (limit orders still safe). If router fails, queue orders with explicit user notice.

Case Mode Source User-visible behavior
Quote stale quote-safe quote:{ticker} age show stale badge; reject market orders, allow limits
Router down accepted-not-routed order outbox return 202 with clear pending status, drain later if allowed
Account shard degraded fail-closed for orders account DB block new orders; portfolio may show last cached state
Hot ticker open dampened fan-out latest tick per ticker push 5 Hz latest quote, drop intermediate ticks
Cancel/fill race state CAS order row + fill log whichever commits first wins; no ambiguous cancel

"The punch line: market data can degrade to stale display, but order and balance state must fail closed because incorrect money is worse than unavailable trading."

What I'd defer. Options chains, fractional share aggregation, recurring investments, crypto trading rails, tax-lot accounting depth, regulatory filings pipeline (e.g., CAT reporting).
