1. Basic Knowledge And Solution

What This Question Is Testing

Twitter (X) is the canonical "social feed" problem: short posts, follow graph, home timeline, all at planet scale. The hard part is the asymmetry — a celebrity with 100M followers posts once, but a normal user's timeline reads need to be sub-second. The naive fan-out-on-write or fan-out-on-read both fail at the extremes.

The invariant is this:

Every tweet is stored exactly once in an authoritative store; the home timeline of each follower eventually includes every tweet from authors they follow (respecting visibility, blocks, deletes); the home timeline reads in sub-second p99; and the system handles celebrity and viral cases without blocking normal users.

At ICT5 / L6+ level, the interviewer is testing whether you can articulate the hybrid fan-out strategy (push for normal authors, pull for celebrities), separate authoritative storage from timeline projections, handle deletions and visibility changes correctly, and design ranking/search as separate concerns from feed delivery.

60-Second Interview Framing

Spoken version:

I would design Twitter around a hybrid fan-out strategy. The authoritative tweet store and follow graph are the source of truth. For normal authors, we push (fan-out on write) tweet ids into follower timeline inboxes. For celebrities with millions of followers, we pull (fan-out on read) at home-timeline construction time, merging celebrity tweets with the precomputed inbox. The interesting decisions are the celebrity threshold, deletion and visibility propagation, ranking, and hot-event surge handling.

Requirements To Clarify First

Functional requirements:

  • Users post tweets (text, optional media); reply, retweet, quote, like.
  • Users follow each other; unfollow; block; mute.
  • Home timeline shows tweets from followed users.
  • User timeline shows a user's own posts.
  • Search by text, hashtag, user.
  • Real-time notifications on mentions, likes, replies.
  • Delete a tweet; edits if supported.

Non-functional requirements:

  • Hundreds of millions of daily active users.
  • Home timeline read latency under 200 ms p99.
  • Tweet post latency under 1 second.
  • Some lag in timeline freshness is acceptable (seconds).
  • High availability: timeline reads must work during write outages.
  • Deletes and blocks must take effect quickly (seconds, not minutes).

Out of scope unless asked:

  • Ad insertion and ranking.
  • Trending topic detection (separate).
  • DM messaging (separate, see Facebook Messenger design).
  • Spaces / live audio (separate).

Scale Assumptions

Area Assumption
Users 500M monthly active.
Posts/day 500M tweets/day.
Average follow count 100; long tail to millions.
Celebrities 10k+ users with >1M followers.
Home timeline reads 100B+/day; 1M+/sec peak.

Tweet size ~1 KB text; media in separate storage.
Timeline page size 20-50 tweets.
Latency Read p99 200 ms; write p99 1 s.

The bottleneck is the read path: home timeline reads outnumber writes by 4 orders of magnitude.

Core Components

Component Responsibility Key detail
Tweet store Source of truth for tweet content + metadata. Sharded by tweet_id.
Follow graph Who follows whom; supports follower lookup. Sharded by user_id; specialized for fan-out.
Fan-out workers Push new tweets from normal authors to follower inboxes. Skip celebrities (over follower threshold).
Timeline inbox Per-user precomputed list of recent tweet ids. Cap to last N (e.g., 800).
Home timeline service On read: merge inbox + celebrity authors' recent tweets; rank; return. Composite; pulls from cache and store.
Author timeline Per-author recent tweets, fetched directly. Cached for popular authors.
Search service Inverted index of tweet content. Eventual; consumes tweet event stream.
Ranking service Optional reorder of timeline candidates. ML-driven; sub-100 ms timeout.
Visibility/safety service Apply blocks, mutes, deletes, content policy. Final gate before returning.
Media service Image/video storage and serving. CDN-fronted.
Event stream Tweet, follow, like, delete events for derived systems. Kafka-style.

Stored in a fast KV (Redis), one entry per user. Capped to recent N entries.

Write Path: Post a Tweet

  1. Client POSTs /tweets.
  2. API authenticates, validates, runs policy filters (spam, content).
  3. Generate tweet_id (snowflake: timestamp + machine + sequence).
  4. Write to tweet store, partitioned by tweet_id.
  5. Emit TweetCreated event to stream.
  6. Update author's AuthorTimeline cache.
  7. Fan-out decision: - If author is "normal" (followers under threshold, e.g., 10k-100k): fan-out workers consume the event and append tweet_id to each follower's TimelineInbox. - If author is "celebrity" (over threshold): skip fan-out; rely on read-time merge.
  8. Return tweet_id.

The post response does not wait for fan-out to complete. Fan-out is async; followers may see the tweet seconds later.

Read Path: Home Timeline

  1. Client requests /home.
  2. Home timeline service fetches user's TimelineInbox from cache (most recent N tweet ids).
  3. For users the viewer follows who are celebrities, fetch their recent tweets from AuthorTimeline cache.
  4. Merge inbox + celebrity tweets, sorted by time (or by ranker score).
  5. Hydrate tweet ids to full tweet content via tweet store (batched).
  6. Apply visibility filters: deletes, blocks, mutes, safety.
  7. Optionally rank via ranking service (with timeout fallback to chronological).
  8. Return page.

The composite read is fast because the inbox is precomputed and celebrity author timelines are small and cached. Tweet hydration is a batched key-value read.

Interview phrase:

The home timeline is a merge of a precomputed inbox plus a small set of celebrity author timelines. The inbox absorbs the fan-out cost; the celebrity merge keeps the post path cheap for big accounts.

Hybrid Fan-out: Why The Split

Pure fan-out on write (push): - Cheap reads (just read inbox). - Expensive writes for celebrities (one tweet → 100M inbox appends).

Pure fan-out on read (pull): - Cheap writes (just store). - Expensive reads (must query authors-followed-by-viewer for each request).

Hybrid: - Normal authors push. - Celebrities pull at read time. - A user following 200 normal accounts and 10 celebrities gets: precomputed inbox + 10 small celebrity timeline reads.

Threshold tuning: - Set so that fan-out write cost equals read amplification savings. - Typically 10k-1M followers; depends on platform.

Edge case: a non-celebrity user goes viral. Move them to celebrity bucket dynamically based on follower growth.

Deletes And Visibility

A deleted tweet must disappear quickly. Naive: scan all timelines and remove. Better:

  • Mark tweet as deleted in tweet store.
  • Hydration step in read path returns nothing (or a tombstone) for deleted tweets.
  • Inbox entries remain (small storage overhead) but display nothing.

For blocks: when A blocks B, the read path filters B's tweets from A's home timeline at hydration. No need to rewrite inboxes.

For protected accounts: visibility flag on tweet; read path checks viewer is approved follower.

Interview phrase:

Deletes and blocks are applied at read time, not at write time. The inbox is a fast index; the truth lives in the tweet store and visibility service.

Ranking

The default home timeline is reverse-chronological. Ranking optionally reorders based on engagement signals:

  • Recency.
  • Author affinity (how often viewer engages with author).
  • Tweet engagement (likes, replies, retweets).
  • Topic relevance.

Implementation:

  • Ranking is a side service called after candidate generation.
  • Strict timeout (e.g., 100 ms). On timeout, fall back to chronological.
  • Cached features per user.

Search

Inverted index built by consuming the tweet event stream:

  • Tokenize tweet content; index by token to (tweet_id, posted_at).
  • Shard by token hash or term + time.
  • Real-time near-online indexing (seconds lag).
  • Separate from feed delivery; runs in its own cluster.

Data Partition key Why

Consistency Model

Layer Consistency Reason
Tweet store Strong write Source of truth.
Follow graph Strong write Used to drive fan-out and visibility.

Search index Eventually consistent Indexing lag of seconds.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Fan-out lag for popular post. Auto-scale fan-out workers; sample for celebrities; threshold-based push/pull split. Fan-out backlog; per-author lag.
Celebrity tweet floods inboxes if mis-classified. Dynamic celebrity detection; threshold adjustment; back-off on burst. Inbox write rate spikes.
Deleted tweet still visible. Read-time hydration check; soft delete in tweet store; cache TTLs. Stale-read alerts.
Block bypass. Final visibility gate in read path. Block escape count.
Ranking service slow. Hard timeout + chronological fallback. Ranker timeout rate.
Tweet store hot shard (one popular tweet). Replicas; cache layer; CDN for embedded media. Per-shard QPS.
Inbox cache eviction loses entries. Reconstruct from author timelines + follow graph (slow path). Cache miss rate.
Hot trending event. Backpressure on fan-out; degrade gracefully (no ranking, fewer recommendations). Event QPS by topic.
Search lag. Async indexing; show "no results" or partial results. Indexing lag.

Operational Metrics

Correctness:

  • Stale post visibility (deleted tweets shown).
  • Block-violation count.
  • Inbox vs ground-truth divergence (sampled).
  • Fan-out lag distribution.

Capacity:

  • Read p50/p99 by region.
  • Write p50/p99.
  • Fan-out worker backlog.
  • Cache hit ratio.

Business:

  • Tweets per user per day.
  • Home timeline DAU.
  • Engagement rate (like/reply/retweet per timeline impression).

Multi-Region Strategy

  • Tweet store and follow graph replicated globally; each user has a home region for primary writes.
  • Inbox caches regional; updated via the event stream.
  • Reads from local region; cross-region tail only if needed.
  • Search indexes per region with replicated tweets.

Tradeoffs To Call Out

  • Hybrid fan-out balances write and read cost but is more complex than either pure approach.
  • Storing inboxes pre-materialized uses lots of memory; capped to recent N reduces but doesn't eliminate cost.
  • Ranking improves engagement but adds latency; strict timeouts protect feeds.
  • Read-time visibility filtering is correct but adds lookup cost.
  • Snowflake ids enable time-ordering without coordination but require careful clock management.
  • Eventually consistent inbox means a fresh tweet may not appear immediately; usually acceptable.

Common Traps And Stronger Moves

Trap Stronger move
Pure fan-out on write. Hybrid with celebrity pull path.
Storing full tweet in inbox. Store tweet id; hydrate at read time.
Apply blocks at write time. Apply at read time; inbox is content-id only.
Synchronous ranking. Strict timeout with chronological fallback.
No celebrity threshold. Dynamic threshold based on follower count + posting frequency.
Replicating tweet content across all inboxes. Tweet store is single source; inboxes hold ids.

Interview Phrase

Twitter's design center is the fan-out strategy. Normal authors push their tweets into precomputed inboxes; celebrities pull at read time. Tweet content lives in one authoritative store; inboxes are id lists. Visibility, deletes, and blocks are applied at read time, not write time. Ranking and search are separate side services.

2. Simulated Interview

Interviewer: Design Twitter.

Candidate: I will design around a hybrid fan-out strategy. The authoritative tweet store and follow graph are the source of truth. For normal authors, we push tweet ids into follower inboxes (fan-out on write). For celebrities with huge follower counts, we skip fan-out and pull at read time. The home timeline merges the precomputed inbox with the celebrity author timelines. The invariant is that every follower eventually sees every authored tweet, subject to visibility.

Interviewer: Walk me through posting a tweet.

Candidate: Client POSTs the tweet text. We authenticate, run policy checks, assign a snowflake id (timestamp-prefixed for natural time ordering), persist to the tweet store sharded by tweet_id. Emit a TweetCreated event to a stream. Update the author's recent-tweets cache. If the author is below the celebrity follower threshold, fan-out workers append the tweet_id to each follower's inbox. If the author is a celebrity, we skip fan-out. Response returns the tweet_id without waiting for fan-out completion.

Interviewer: Walk me through reading the home timeline.

Candidate: Fetch the user's inbox from Redis — a list of recent tweet ids capped at, say, 800. Identify which followed users are celebrities; fetch their recent author timelines from cache. Merge the inbox and celebrity tweets sorted by time. Hydrate tweet ids to content via batched reads from the tweet store. Apply visibility filters: deletes, blocks, mutes, age-gating. Optionally call the ranking service with a strict timeout; fall back to chronological order on timeout. Return the page.

Interviewer: A celebrity has 100 million followers. What happens at post time?

Candidate: With pure fan-out on write, we'd append 100M times — minutes of work and massive write amplification. The hybrid avoids it: we don't fan out for celebrities. Their followers' home timelines fetch celebrity tweets at read time from a small per-author cache. The cost is read amplification — for a follower of 10 celebrities, that's 10 small cache lookups — which is much cheaper than 100M writes per post.

Interviewer: How do you set the celebrity threshold?

Candidate: Tune so total work is minimized. If a user has F followers and posts P tweets per day, fan-out cost is F * P writes/day. If they have R followers reading the timeline R_r times per day, pull cost is R * R_r reads/day. Find the F where push cost exceeds pull cost. In practice, somewhere between 10k and 1M. Adjust dynamically: a user gaining followers fast gets promoted; idle accounts demoted.

Interviewer: A user goes viral suddenly. They're still classified as normal.

Candidate: Detect rapid follower growth and promote them to celebrity status. Existing fan-out backlog from before promotion completes naturally; new posts use pull. Also: if a single tweet from a normal user goes viral (retweets cascading), the fan-out workers might back up. We backpressure: queue depth alerts, dynamic worker scaling, and a fallback where we stop pushing if the queue is too deep and rely on pull for those followers.

Interviewer: How do you handle deletes?

Candidate: Mark the tweet as deleted in the tweet store. Don't scan and clean inboxes — they hold tweet_ids, and the read-time hydration sees the deleted status and returns nothing for that entry. Caches respect a TTL or invalidation event. So a deleted tweet stops appearing in feeds within seconds of the delete, without touching billions of inboxes.

Interviewer: Blocks?

Candidate: Same pattern. A block is a state on the follow edge or a separate block list. At read-time hydration, the visibility service checks: is the author in my block list? Is the author protected and I'm not approved? Filter out. The inbox doesn't need to be rewritten when blocks change; the next read picks up the new block.

Interviewer: Search?

Candidate: Separate service. Indexer consumes the tweet event stream and builds an inverted index — tokens → posting list of (tweet_id, posted_at). Sharded by token hash or by term + time. Eventual consistency: a tweet is searchable a few seconds after posting. Read pattern is query → token lookup → score → return tweet ids → hydrate via tweet store. The search service does not interact with the feed delivery path.

Interviewer: Ranking?

Candidate: Side service with strict timeout. The home timeline service generates candidates (inbox + celebrity tweets). Sends candidate ids + viewer context to the ranking service. Ranker returns scored ids. If timeout, fall back to chronological. The ranker uses features like viewer-author affinity, tweet engagement, recency. Heavy features are precomputed.

Interviewer: Multi-region?

Candidate: Tweet store and follow graph replicate globally with each user having a home region for primary writes. Inbox caches are regional and rebuilt from the event stream. Reads come from the local region. Cross-region replication has bounded lag, typically seconds. For posts, we accept a small window where users in different regions see the new tweet at slightly different times.

Interviewer: Trending event — World Cup goal. Now what?

Candidate: Spike of posts, retweets, likes. Fan-out workers scale up. We backpressure: if queues are backing up, defer some fan-outs and switch to pull for affected authors temporarily. Ranking service may degrade to chronological under load. Search indexing prioritizes the hot terms. Caches around popular tweets absorb hot reads. We monitor and degrade gracefully — feeds slow slightly but stay correct.

Interviewer: Summarize.

Candidate: Hybrid fan-out. Normal authors push tweet ids to follower inboxes; celebrities are pulled at read time. Tweet content lives in one authoritative store; inboxes hold ids only. Read path hydrates ids, applies visibility filters, optionally ranks with a timeout. Snowflake ids give time-ordered tweets without coordination. Search and ranking are side services. Multi-region uses per-user home regions with eventual replication. The hardest cases are celebrities and viral spikes, handled by dynamic threshold promotion and backpressure.

3. Practice Notes

Five-Minute Version

  1. Clarify: scale, latency, viral cases, deletes.
  2. Invariant: every authored tweet eventually visible to followers, subject to visibility.
  3. Hybrid fan-out: push normal, pull celebrity.
  4. Authoritative tweet store; inbox holds ids only.
  5. Snowflake ids for time-ordering.
  6. Read-time visibility: deletes, blocks, mutes.
  7. Ranking + search as side services with timeouts.
  8. Multi-region with per-user home regions.

One-Minute Closeout

Twitter is the canonical hybrid fan-out problem. Normal authors push tweet ids into precomputed inboxes; celebrities pull at read time. The tweet store is the single source of truth; inboxes and caches are projections. Deletes and blocks are applied at read time via the visibility gate, not by rewriting inboxes. Ranking and search are side services with timeouts and async indexing. Multi-region uses per-user home regions with eventually-replicated inbox caches. The hardest cases — celebrities, viral spikes — are handled by dynamic thresholds and backpressure.
