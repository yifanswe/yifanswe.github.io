1. Basic Knowledge And Solution

What This Question Is Testing

A ride-hailing platform matches riders with drivers in real time. The hard part is the geospatial dispatch problem: find the right driver among millions of moving locations, hand off the trip atomically so no two riders get the same driver, then track the trip to completion.

The invariant is this:

A driver is assigned to at most one in-progress trip at a time; each rider request is matched to at most one driver who accepts; once matched, both sides agree on the trip state through its lifecycle (en route, in trip, completed); location and ETA updates are timely; fares are computed deterministically from the trip's recorded events.

At ICT5 / L6+ level, the interviewer is testing whether you can model geospatial indexing, design real-time matching under contention, handle driver state transitions atomically, manage trip lifecycle as a state machine, and reason about surge pricing and ETA prediction.

60-Second Interview Framing

Spoken version:

I would design Uber as three core systems: a real-time location index of available drivers (geospatial), a matching service that finds and reserves the right driver for each request, and a trip lifecycle service that owns the state machine from request to completion. The interesting decisions are geospatial sharding, atomic driver assignment (no double-booking), trip state transitions, surge pricing, and ETA prediction.

Requirements To Clarify First

Functional requirements:

  • Rider requests a ride from A to B.
  • System matches a driver; rider sees ETA, driver info.
  • Driver receives, accepts or declines; on decline, system tries next.
  • Real-time location updates from rider and driver during trip.
  • Trip lifecycle: requested → matched → en_route → in_trip → completed.
  • Fare calculation based on distance, time, surge.
  • Cancel by rider or driver.
  • Payments after trip.
  • Ratings and review.

Non-functional requirements:

  • Matching latency under a few seconds.
  • Location updates at high frequency (every 4-10 seconds per driver).
  • Survive sudden surges (event end, weather, airport).
  • No double-booking of drivers.
  • Accurate trip records for billing.
  • Multi-region (each city is essentially independent).

Out of scope unless asked:

  • Pool / shared rides matching algorithm.
  • Long-haul logistics (Uber Freight).
  • Food delivery (Uber Eats; similar but distinct).

Geospatial Index

Drivers move; the index must update fast.

Approach: divide the world into cells (S2, H3, or geohash) at appropriate resolution.

  • H3 resolution 9: ~100m hex cells.
  • Each cell stores a set of available driver_ids.
  • On location update, if the driver's cell changes, remove from old cell, add to new.
  • Each cell lives in memory in a service; shard by cell_id across nodes.

For "find drivers within R of point P":

  • Compute the cells covering the disc of radius R around P.
  • Query each cell's available-driver set in parallel.
  • Filter by exact distance and vehicle compatibility.
  • Rank by score (distance, ETA, driver rating).

Interview phrase:

The geo index is fundamentally "for each cell, who is available." Location updates are cell-membership changes. Spatial queries are cell scans plus exact-distance filtering.

Matching Algorithm

When a rider requests:

  1. Compute candidate cells around origin (e.g., 1-2 km radius).
  2. Query geo index for available drivers.
  3. Score candidates: ETA, driver rating, vehicle match, recent decline rate.
  4. Pick top candidate.
  5. Atomically attempt to reserve the driver — conditional update on Driver.status = online_available, current_trip_id = null.
  6. If reserved successfully, offer the trip to the driver.
  7. If driver accepts, transition trip to matched.
  8. If driver declines or times out, release reservation and pick next candidate.

Atomic reservation is the key:

  • Use CAS on Driver.status and Driver.current_trip_id.
  • Only one trip can hold a driver at a time.
  • The driver's "offer pending" is a short-lived hold (~15-30 seconds).

Interview phrase:

The match is an atomic conditional state transition on the driver. No two trips can race-acquire the same driver because only one CAS wins.

Trip State Machine

requested -> matched -> en_route -> arrived -> in_trip -> completed
     |             |          |          |          |
     +-> canceled  +-> driver_canceled   +-> canceled

Transitions:

Transition Trigger Effect
requested -> matched Driver accepts. Set driver_id; lock driver.
matched -> en_route Driver starts driving toward rider. Update display.
en_route -> arrived Driver arrives at pickup. Start wait timer.
arrived -> in_trip Driver begins trip. Start meter.
in_trip -> completed Driver ends trip. Compute fare; charge.
Any -> canceled Cancel action. Release driver; possibly fee.

Each transition is CAS on (trip_id, from_state).

Surge Pricing

A heatmap by cell of supply vs demand.

  • Demand = open requests per cell per minute.
  • Supply = available drivers per cell.
  • Multiplier = function of supply/demand ratio.
  • Updated every 30 seconds to a few minutes.
  • Cached client-side for display.

The surge value at the time of request is recorded in the trip and used for fare calculation.

ETA Prediction

Two-level:

  • Routing ETA: based on road network and current traffic.
  • ML ETA: correction based on historical patterns, driver behavior, time of day.

Pre-computed for common O-D pairs; on-demand for arbitrary.

Used at:

  • Match time: ETA to pickup for driver scoring.
  • Trip preview: ETA for rider.
  • En route: live ETA updates.

Location Update Pipeline

Drivers send location every 4-10 seconds. At 1M drivers online, that's 100k-250k updates per second.

  • Each driver's location updates partition by driver_id.
  • A streaming pipeline updates the geo index in near-real time.
  • Updates are tolerant of slight lag (a few seconds).
  • We do not store every location update forever; only the route polyline for completed trips.

For real-time display:

  • Rider's app subscribes to driver location during en_route and in_trip.
  • Server pushes location at lower frequency than driver reports (e.g., every 2-5 seconds).

Per-city sharding:

  • Cities are largely independent operationally.
  • A driver in San Francisco does not match with a rider in New York.
  • Each city has its own geo index, dispatch service, and (often) data center.

Consistency Model

Layer Consistency Reason
Driver state Strong (CAS) No double-booking.
Trip state Strong (CAS) Single authoritative state.
Geo index Eventually consistent (seconds lag) Acceptable for matching.
Surge heatmap Eventually consistent Updated periodically.
Trip history Eventually consistent Reporting.

Multi-Region Strategy

  • Cities are the natural sharding unit.
  • Each city's geo index and dispatch run in the nearest region.
  • Driver state and trip state in same region.
  • Cross-city handoffs (long airport trip) are rare and handled with specific logic.
  • Global services (payments, user accounts) replicated globally.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Two riders request same driver simultaneously. CAS on driver state; only one wins. Conditional-write conflict rate.
Driver app goes offline during trip. Trip continues based on rider's reports; reconciliation when reconnect. Driver-offline-mid-trip count.
Location update lost. Smooth client-side; resume on next update. Update gap distribution.
Surge spike at venue end. Pre-allocate dispatch capacity; surge pricing adjusts demand. Per-cell demand vs supply ratio.
Match offer not responded (driver away from phone). Offer expires in 15s; reassign. Offer expiry rate.
Driver accepts but doesn't move. Time-bounded; rider can cancel without penalty after threshold. Match-but-no-move count.
Fraudulent location spoofing. GPS plausibility checks; behavior analytics. Anomaly score per driver.
Geo index hot cell. Sub-shard the cell; or accept and scale node. Per-cell QPS.
Trip stuck in in_trip. Reconciliation worker; max trip duration. Trips > N hours.
Payment fails post-trip. Trip already recorded; payment retried; reuse payment system. Failed payment rate.

Operational Metrics

Correctness:

  • Double-booking attempts (must equal zero conflicts that succeed).
  • Trip state inconsistency events.
  • Cancellations within first N seconds (UX problem signal).

Capacity:

  • Location updates per second per region.
  • Match latency p99.
  • Geo index node CPU and memory.

Business:

  • Active trips, active drivers, requests per minute by city.
  • Match success rate.
  • Surge multiplier distribution.
  • Average wait time.

Tradeoffs To Call Out

  • Larger geo cells make spatial queries cheaper but reduce match precision.
  • More frequent location updates give live ETA but cost battery and bandwidth.
  • Shorter offer timeout matches faster but causes more reassignment.
  • Strict driver state CAS prevents double-booking but limits throughput per driver.
  • Per-city sharding is natural but cross-city trips become awkward.
  • Aggressive surge dampens demand but generates user complaints.

Common Traps And Stronger Moves

Trap Stronger move
Global location index. Per-city geo index sharded by cell.
Linear scan of all drivers. H3/S2 cell-based spatial index.
First-come-first-serve dispatch. Score-based matching; CAS reservation.
No offer timeout. Short-bounded offer with fallback to next candidate.
Persistent storage for every location update. Stream-only with bounded retention; persist polylines on trip end.
Surge as a simple slider. Per-cell heatmap updated frequently.

Uber is a geospatial dispatch problem layered over a trip state machine. The geo index makes "find nearby drivers" cheap; atomic CAS on driver state guarantees no double-booking; the trip is a state machine from request to completion. Surge and ETA ride alongside as supporting services.

2. Simulated Interview

Interviewer: Design Uber.

Candidate: I will design three core systems. First, a real-time geospatial index of available drivers organized into H3 cells. Second, a matching service that queries the index and atomically reserves the chosen driver. Third, a trip state machine that owns the lifecycle from request to completion. The invariant is that a driver is assigned to at most one in-progress trip at a time. Surge pricing, ETA, and payments are supporting services.

Interviewer: How do you find the nearest available driver?

Candidate: Divide the world into H3 cells at resolution 9 — about 100m hex cells. Each cell maintains a set of available driver_ids. On a request, I compute the cells covering a small disc around the origin (1-2 km), query each cell in parallel, get the candidate driver_ids, filter by exact distance and vehicle compatibility, and score them by ETA and other factors. The geo index is sharded by cell_id and held in memory.

Interviewer: Two riders request a ride simultaneously and both get the same nearest driver. How do you prevent double-booking?

Candidate: Atomic conditional update on the driver's state. After scoring, the matching service does UPDATE Driver SET status=on_trip_pending, current_trip_id=? WHERE driver_id=? AND status=online_available AND current_trip_id IS NULL. Only one of the two requests' CAS succeeds. The losing request picks the next best candidate. This is the canonical "use database atomicity instead of a distributed lock" approach.

Interviewer: The driver accepts. What happens?

Candidate: Trip transitions to matched. We notify both apps. The driver's state is now on_trip_pending. The driver starts navigation; the rider sees the driver approaching. As the driver moves, location updates flow through the geo index and the per-trip subscription. Trip state advances: matched → en_route → arrived → in_trip → completed. Each transition is a CAS on (trip_id, from_state).

Interviewer: The driver doesn't accept after 15 seconds.

Candidate: Offer expires. We release the driver's on_trip_pending back to online_available. The matching service picks the next candidate and offers. We track the driver's decline rate as a signal for future matching scoring.

Interviewer: Driver app goes offline mid-trip.

Candidate: Two scenarios. If the trip is in_trip, the rider can still report progress (the rider's app sends location). The trip continues based on rider's reports; on driver app reconnect, we reconcile. The driver's location updates pick back up. If the driver disconnect is long, support can intervene. For trip completion, either side can trigger; we have safeguards against unilateral premature completion.

Interviewer: How does surge pricing work?

Candidate: Per-cell heatmap. Demand per cell (recent open requests) divided by supply (available drivers). When ratio exceeds threshold, multiplier kicks in. Updated every 30 seconds or so. The surge multiplier at request time is captured in the trip record and used for fare calculation. It's deterministic given the recorded surge value.

Interviewer: Location updates — 10M drivers reporting every 5 seconds is 2M updates per second.

Candidate: Sharded by driver_id. A streaming pipeline (Kafka + stateful consumer) receives updates and applies them to the geo index. If the driver crossed a cell boundary, we remove from old cell and add to new. The geo index is in-memory; each node owns some cells. Update latency is sub-second. We don't persist every location update; only the route polyline for completed trips.

Interviewer: ETA?

Candidate: Routing engine computes ETA from current location to destination based on road network and traffic data. ML model adjusts for historical patterns, time of day, driver behavior. For match scoring, we use ETA to pickup. For rider display during en_route and in_trip, we use ETA to destination, updated periodically.

Interviewer: Multi-region?

Candidate: Cities are the natural sharding unit. Each city has its own geo index, matching service, and trip state — usually in the nearest data center. Cross-city handoffs (long airport trip) are rare and handled with explicit support. Global services like payments and user accounts replicate globally.

Interviewer: What about a venue letting out — 50,000 people requesting rides simultaneously.

Candidate: That cell becomes hot. Geo index node for that cell may saturate; we pre-shard the hot cell into sub-shards. Surge pricing kicks in, dampening demand. Drivers are notified of high demand and migrate. We also pre-allocate dispatch capacity for known events. Trip backlog is handled with explicit "you'll be matched soon" UX rather than failing.

Interviewer: Trip stuck in in_trip for 12 hours.

Candidate: Reconciliation worker scans for trips in non-terminal states older than threshold. Likely scenarios: driver finished trip but didn't tap complete, or driver app crashed. We auto-complete based on detected motion patterns or driver outreach. Fare is computed from recorded events.

Interviewer: Summarize.

Candidate: Geo index with H3 cells, sharded per city. Atomic CAS on driver state prevents double-booking. Trip state machine with explicit transitions. Surge pricing as per-cell heatmap. ETA from routing + ML. Location updates streamed; geo index updated in-memory. Reconciliation for stuck trips. The hardest correctness problem is the atomic match; the hardest scale problem is the location update firehose.

3. Practice Notes

Five-Minute Version

  1. Clarify: matching, real-time location, trip lifecycle, surge.
  2. Invariant: at most one trip per driver.
  3. Geo index with H3 cells; per-city sharding.
  4. Matching: score candidates; atomic CAS reservation.
  5. Trip state machine with explicit transitions.
  6. Surge as per-cell heatmap; ETA from routing + ML.
  7. Location update streaming; bounded retention.
  8. Reconciliation for stuck trips.

One-Minute Closeout

Uber is a geospatial dispatch problem layered over a trip state machine. H3-cell geo index makes nearby-driver queries cheap; atomic CAS on driver state prevents double-booking; the trip lifecycle is a state machine of CAS transitions. Surge pricing and ETA ride alongside. Cities are the natural sharding unit. The hardest scale problem is the location update firehose; the hardest correctness problem is the atomic match.
