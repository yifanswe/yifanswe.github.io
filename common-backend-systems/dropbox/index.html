Dropbox — File Sync Across Devices (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Easy · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build Dropbox: a desktop client and web app where a user drops a file into a folder on one device and it appears on all their other devices within seconds. 500M users, average user has 100 GB, files range from a 10 KB text note to a 5 GB video.

1. Requirements (~5 min)

Functional - Upload a file from any device and have it persist. - See and download every file the user owns from any other device. - Sync changes automatically — modify locally, the modification propagates.

I'd defer sharing, comments, and selective sync; mention them so the interviewer knows they're scoped out.

Non-functional - Durability is the headline — Dropbox losing your file is the end of trust. Eleven nines for file bytes. - Sync latency target: seconds, not minutes, for small files. - Bandwidth efficiency — never re-upload bytes the server already has. - Support files up to ~50 GB; chunking is a hard requirement. - Strong read-after-write for metadata so the file you just uploaded shows up everywhere immediately.

Capacity - 500M users × 100 GB = 50 EB. Hot tier is far smaller — most bytes are cold. - Average file ~1 MB, chunked at 4 MB; 100B chunks lifetime. - Metadata is large but manageable; bytes dominate. If chunk metadata is ~100 B × 100B chunks × 3 replicas, that is ~30 TB. Object storage for 50 EB is the cost center.

2. Core Entities (~2 min)

  • User — owns files, devices.
  • File — file_id, owner_id, path, size, version, content_hash.
  • Chunk — chunk_id (= content hash), size, storage_url. Deduped globally.
  • Device — device_id, last sync cursor.
  • Event — append-only log of file changes per user, drives sync.

3. API (~5 min)

POST  /v1/upload/initiate       { path, size, chunk_hashes[] } → { upload_id, missing_chunks[] }
PUT   /v1/upload/{upload_id}/chunk/{idx}  (binary)             → { ok }
POST  /v1/upload/{upload_id}/commit                            → { file_id, version }
GET   /v1/files?cursor=...      → { changes[], next_cursor }   // long-poll or WebSocket
GET   /v1/files/{file_id}/download/{idx}                        → 302 to chunk URL

The two-phase upload (initiate → chunk PUTs → commit) lets the server tell the client which chunks it already has so the client only ships novel bytes.

The sync cursor is opaque and server-issued: {user_id, event_seq, device_id}. The server never trusts a client-provided path diff; clients replay the per-user event log from the cursor.

4. Data Flow (~5 min)

Upload: client chunks the file → hashes each chunk → calls initiate with the hash list → server replies with which chunks are missing → client uploads only those → calls commit → server writes a new file version and appends an event to the owner's event log.

Download/sync: each client holds a cursor into the user's event log. A long-poll connection wakes when new events arrive; the client fetches the file metadata, sees which chunks it doesn't have, and downloads only those.

5. High Level Design (~10–15 min)

  1. Client does all chunking and hashing locally (block-level deduplication, ~4 MB chunks).
  2. API gateway / metadata service handles auth, upload coordination, and the events feed.
  3. Metadata store (MySQL, sharded by user_id) holds files, versions, and chunk-id lists. Sharding by user keeps a user's data co-located.
  4. Chunk store (object store like S3) holds raw chunks keyed by content hash. Eleven nines, geo-replicated.
  5. Event log (per user) in Kafka or a sharded MySQL table; sync clients read from their cursor.
  6. Notification service holds long-poll/WebSocket connections to clients and pushes "new event" pokes; clients then re-pull.
  7. CDN sits in front of the chunk store so popular chunks (rare in Dropbox but real for shared files) don't always hit origin.

End-to-end I'd narrate: Alice edits a doc on her laptop → laptop computes new chunk hashes → initiate shows server has 95% of them → laptop uploads the 5% delta → commit writes version 7 → event lands in Alice's log → Alice's phone is long-polling, wakes up, fetches version 7 metadata, downloads the same 5% delta. Phone is now in sync.

Concrete stores:

Upload path, stores touched:

  1. Client hashes chunks locally and calls upload/initiate with chunk_hashes[].
  2. Metadata service checks chunk index by chunk_id in parallel and creates upload:{upload_id} listing only missing chunks.
  3. Client uploads missing chunks directly to object storage using signed URLs; each PUT is idempotent on (upload_id, idx).
  4. Commit transaction writes file metadata, version manifest, chunk refcount increments, and the next (user_id, seq) event.
  5. Notification service wakes devices; each device reads /files?cursor=seq, then downloads only chunks absent from local disk.

Capacity sanity by store: object bytes dominate at 50 EB logical; even with erasure coding around 1.4x, storage and egress are the headline. At 4 MB target chunks, the upper bound is 12.5T logical chunk references; a 32 B manifest reference is ~400 TB raw, and a 100 B chunk-index row is ~1.25 PB raw before replication and dedup wins. File rows can be PB-scale if users really average tiny files, so old versions and deleted metadata need compaction/lifecycle rules. The event log is smaller: 500M users × 5 metadata changes/day × 200 B × RF=3 is ~1.5 TB/day, or ~135 TB for a 90-day hot window. Upload sessions are tiny by comparison: 10M concurrent sessions × 1 KB is ~10 GB with a 24 h TTL.

6. Deep Dives (~10 min)

Chunking and deduplication. Fixed-size 4 MB chunks are simpler and survive arbitrary insertions poorly. Content-defined chunking (Rabin fingerprint) survives inserts in the middle of a file but is more expensive client-side. I'd default to fixed chunks for the interview and call out CDC as the production answer for documents that get edited in place.

Why object store + sharded MySQL. The byte store wants durability, lifecycle tiers, and cheap massive capacity; S3/GCS-style object storage beats HDFS here because clients need signed object URLs and cross-region durability without operating storage nodes. Metadata wants transactions around path rename/version/event append for one user, so sharded MySQL by user_id is honest. Cassandra/DynamoDB can work for metadata if you accept weaker multi-row transactions or add an application transaction log.

Be ready for "why not put metadata in the object store too?" The answer is that path rename, version conflict detection, and event append are small transactional writes keyed by user_id; object stores are excellent for immutable chunks but clumsy for ordered per-user metadata.

Conflict resolution. Two devices edit the same file offline. Server keeps both versions: file.txt and file (Bob's conflicted copy).txt. Last-writer-wins on metadata, but never silently discard bytes. Vector clocks per device would let us detect the conflict precisely.

Sync protocol. Long-poll with cursor is the simple answer. For active users, upgrade to a WebSocket. The event log per user means a client coming back online after a week just replays everything from its cursor — no diff computation needed.

Two views of the same user state. The path tree (user_id, path) is the current view, optimized for "show me my Dropbox." The event log (user_id, seq) is the change view, optimized for "tell this device what changed since seq 900." They are not competing sources of truth: commit writes both in one metadata transaction, and the snapshot fallback stitches them together with a high-water mark.

Sync degradation state:

Stage Mode Source Latency
active WebSocket poke + cursor replay event log hot partition seconds
dormant cursor older than hot log metadata tree snapshot + recent events tens of seconds for large trees
exhausted local state unknown/corrupt full manifest walk by (user_id, path) minutes, resumable
cold bytes chunk in archive tier restore request then delayed download minutes-hours

The cursor carries event_seq; if it falls behind retention, the server returns cursor_expired and the client switches to snapshot sync instead of silently missing changes.

Hot vs cold storage. 90% of bytes are never read after upload. Tier chunks: hot (S3 standard) for recent and frequently accessed, cold (Glacier-style) for chunks untouched for 90 days. Metadata says where each chunk lives; rehydrate on first read.

Capacity math: 50 EB logical bytes can drop with global dedup, but end-to-end encryption or per-user keys reduce cross-user dedup. With 4 MB chunks, 50 EB is roughly 12.5 trillion logical chunk references; the manifest/reference metadata is tens to hundreds of TB, while object bytes are exabytes. Dominant cost is object storage and egress, not the metadata DB.

Large file uploads. Resumable: each chunk PUT is idempotent on (upload_id, idx). Client retries any chunk that failed. commit is the atomic point — partial uploads are garbage-collected after a TTL.

Edge stress tests. A rename storm in a large folder is metadata-heavy, so the server batches events under one parent-operation id and lets devices apply them in order from the event log. A hot chunk from a widely shared file is served through CDN and object-store range cache; the chunk index stays keyed by chunk_id, so it does not become a per-user hot key. A burst of uploads from one user is fenced by per-user commit sequencing: chunks can upload in parallel, but metadata commits serialize by (user_id, seq). Multi-region drift is handled by assigning a metadata home region per user and replicating chunks globally; on failover, the new region resumes from the last replicated event sequence and rejects stale commits with an older metadata epoch. A brand-new device never replays years of events; it starts from the path snapshot and only then follows the live log.

Interview punch line for the edge cases: bytes can be parallel and content-addressed, but metadata must be ordered per user. That is the line that keeps sync predictable when devices go offline, regions fail over, or files are renamed in bulk.

End-to-end encryption (optional). Client encrypts chunks before upload with a per-file key wrapped by the user's master key. Server stores ciphertext only. Loses server-side dedup across users — call that out as the explicit tradeoff.

What I'd defer. Sharing and permissions, real-time collaborative editing on the same file, mobile bandwidth optimization, abuse detection on shared links.

Interview punch line: Dropbox is two systems joined by a cursor: immutable content-addressed bytes, and strongly consistent per-user metadata/events. Sync stays simple because devices replay ordered server events rather than guessing diffs.
