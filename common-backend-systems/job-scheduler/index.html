Distributed Job Scheduler — Cron at Scale (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a distributed cron / job scheduler. Users register jobs with a schedule (cron expression or one-shot timestamp) and a payload (URL to POST or function to invoke). The system fires each job at its scheduled time, exactly once, retries on failure. 10M registered jobs, peak ~100k jobs fire per second at top-of-minute, average job runtime 5 s.

1. Requirements (~5 min)

Functional - Register a recurring or one-shot job with cron expression + handler. - Fire each job at its scheduled time within ~1 s. - At-most-once or at-least-once semantics (configurable); never silently miss. - Retry on failure with backoff. - Visibility: see job history, success/failure.

I'd defer multi-tenancy depth and the handler runtime sandbox (assume HTTP webhook target).

Non-functional - Fire-time accuracy: p99 within 1 s of scheduled time. - Survive scheduler node failure without dropping firings. - No silent misses. Better to double-fire (and require idempotent handlers) than to miss. - Linear scale: 10M jobs, 100k fires/sec peak.

Capacity - 10M registered jobs × ~200 B metadata = 2 GB. - 100k fires/sec × ~5 s handler runtime = 500k concurrent in-flight executions. - Schedule metadata is tiny after replication (~2 GB × RF=3 = 6 GB). The dominant cost is run history and retry/audit events: if peak 100k fires/sec lasted an hour, at ~300 B/event × RF=3 that hour alone is ~324 GB before indexes.

2. Core Entities (~2 min)

  • Job — job_id, schedule (cron / one-shot), target (URL + payload), retry policy, owner.
  • JobInstance — a specific firing: instance_id, job_id, scheduled_at, status (queued/running/done/failed).
  • Lease — claim that prevents two workers from running the same instance.

3. API (~5 min)

POST  /v1/jobs              { schedule, target, retry_policy }  → { job_id }
GET   /v1/jobs/{id}/runs?cursor=...
DELETE /v1/jobs/{id}
POST  /v1/jobs/{id}/pause / resume

User from token. Job creation is idempotent on optional client_id.

Run history cursors encode {job_id, last_scheduled_at, last_instance_id}. Handler invocations include fire_id so customers can dedupe retries on their side.

4. Data Flow (~5 min)

Job registered → next_fire_at computed → indexed in time-bucket. Timing wheel walks buckets second-by-second → due jobs claimed by workers via lease → handler invoked → result recorded → next_fire_at recomputed.

5. High Level Design (~10–15 min)

  1. Job catalog. Sharded MySQL keyed by job_id. Row holds the schedule, target, and next_fire_at.
  2. Scheduling index. A time-bucketed index for "due in next N seconds." Two flavors that fit the question: - Timing wheel: in-memory bucket array, slot per second (or finer), each slot holds job_ids due in that slot. Cheap O(1) per slot tick. Persisted state in a backing store so a crash doesn't lose state. - Sorted set / heap by next_fire_at: Redis ZSET keyed on next_fire_at. Workers ZRANGEBYSCORE 0 now and consume.
  3. Dispatcher. Walks the timing wheel/ZSET each tick (1 Hz). For each due job, writes a JobInstance row (status=queued) and pushes a fire event to the job queue (Kafka or SQS).
  4. Worker pool. Pulls fire events. Each worker: - Acquires a lease (Redis SET instance:I worker:W NX EX 60). - Invokes the handler (HTTP POST to target with payload + signed retry token). - Records result; releases lease.
  5. Retry policy. On failure, compute next retry time = now + backoff(attempt). Schedule a new JobInstance with that fire time. Cap attempts.
  6. Next-fire recomputation. On successful fire, recompute next_fire_at from the cron expression and update both the catalog row and the scheduling index.
  7. Coordination. Multiple dispatcher instances. Each is responsible for a shard of job_ids (consistent hash). Leader election within a shard for the active dispatcher; standbys take over on failure.

End-to-end narration: at 09:59:55, dispatcher walks the timing wheel for 10:00:00 slot, finds 100k jobs due → writes 100k JobInstance rows (batched) → pushes to Kafka. At 10:00:00, workers pull from Kafka, lease each instance, fire HTTP POST to the target. Handler responds 200 in 200 ms → result recorded → dispatcher recomputes next fire (e.g., 10:01:00 for a every-minute job) and reinserts into the timing wheel.

6. Deep Dives (~10 min)

Why a timing wheel. With 10M jobs and 100k firing per second peak, scanning a heap for due jobs every second is expensive. Timing wheel costs O(active jobs per slot) per tick. For a 1-hour wheel at 1 Hz, that's 3600 slots; jobs landing beyond the wheel are handled by a hierarchical second wheel (e.g., day wheel feeds the hour wheel). Hierarchical timing wheels are the classic answer.

Why not only Redis ZSET: a ZSET by next_fire_at is a good persistent index and simpler v1, but at top-of-minute it creates hot range scans and lock contention. The timing wheel gives predictable per-slot work; the ZSET/Kafka log is the recovery source.

Tech choice defense: MySQL/Postgres is fine for the job catalog because updates are per-job and need transactional create/pause/delete semantics; DynamoDB also works if the interview leans cloud-native, but secondary access by owner/history is more awkward. Kafka is better than direct worker RPC because dispatch bursts need buffering and replay; SQS is simpler operationally but gives less control over partition ownership and ordered replay by job_id % N. Redis leases are fast enough for second-level coordination, while etcd/ZooKeeper should only own dispatcher shard leadership, not every job instance.

Persistence + recovery. Timing wheel is in-memory; if the process dies, in-flight slots are lost. Two defenses: - Backed by a sorted set in Redis (so on restart the wheel rebuilds). - Periodic snapshot to durable storage + replay of recent events from a Kafka topic.

Register path trace: API validates cron and owner quota -> writes job:{job_id} with next_fire_at -> writes due:{epoch_second}:{shard} -> emits job.registered audit event. The catalog row and schedule index update must be atomic or reconciled by a repair job that scans for catalog rows missing from the index.

Fire path trace: dispatcher owns shard -> reads wheel slot -> creates instance:{job_id}:{scheduled_at} with idempotent insert -> publishes fires.{shard} -> worker acquires lease:{instance_id} -> invokes target -> writes terminal status -> recomputes next fire and reinserts job.

Hot fire path, step by step: 1. Dispatcher owns hash range job_id % N = shard through a coordinator lease. 2. At tick T, it drains due:{T}:{shard} from memory; if the wheel slot is missing, it falls back to the Redis/ZSET recovery index for [T - grace, T]. 3. It batch-inserts instance:{job_id}:{scheduled_at} with idempotency on (job_id, scheduled_at, attempt). 4. It publishes compact fire messages to fires.{shard} and commits the slot watermark. 5. Worker pulls by shard, reads the instance row, acquires lease:{instance_id} with a fencing token, calls the target, and writes the terminal status only if its fencing token is current.

Cold / recovery path: after dispatcher failure, a standby claims the shard, reads the last committed watermark, scans the schedule index for [watermark, now], and recreates missing instance rows. Duplicate fire events are allowed because the instance insert and customer-visible fire_id are idempotent; misses are not allowed.

Run-history pagination uses {job_id, last_scheduled_at, last_instance_id} so the UI walks the job_id shard in time order without offset scans.

Exactly-once vs at-least-once. True exactly-once across a distributed scheduler is fool's gold. Promise at-least-once + require idempotent handlers (via a fire_id the handler can dedupe on). Document this clearly in the SDK.

Lease-based double-fire prevention. Worker acquires SET instance:I worker:W NX EX 60. If worker dies, lease expires and a different worker can re-claim. Worker periodically renews the lease (heartbeat) for long-running jobs.

Hot-second problem. Top-of-minute is a thundering herd — everyone's */1 * * * * job lands on :00. Two mitigations: - Jitter at registration: automatically jitter the schedule by ±5 s (with user opt-out for strict schedules). - Smear within slot: during dispatch, spread firings uniformly within the second using a small jitter buffer.

Scaling the dispatch. Shard job_id across N dispatcher instances. Each owns a hash range; jobs only fire from one owner. Coordinator (etcd / ZooKeeper) tracks ownership; on failure, range fails over.

Backpressure on the worker pool. If targets are slow, the queue grows. Auto-scale workers on queue depth. For burst targets, allow per-target concurrency limits (don't pile 10k concurrent calls on a single endpoint).

Two-view model. The scheduler keeps the same future firing in two views: the catalog row (job:{job_id}.next_fire_at) is the source of truth for repair, while the schedule index (due:{second}:{shard} or ZSET) is the serving index for the next tick. The repair loop continuously reconciles them so a lost bucket does not become a silent miss.

Job instance state machine:

Stage Mode Source Transition Latency / behavior
Scheduled future firing schedule index + catalog tick reaches scheduled_at p99 within 1 s target
Queued fire event emitted fire queue + instance row worker pull seconds under normal load
Running lease held lease + worker heartbeat target returns or timeout renew every 15-30 s
Retrying failed attempt instance DB + retry policy backoff expires jittered retry
Terminal done/failed/dead-letter instance DB success or attempts exhausted retained 30-90 d

Edge stress tests. Top-of-minute bursts are handled by jitter, slot smearing, and batched instance writes. Dormant jobs whose wheel entry aged out are recovered from job:{job_id}.next_fire_at by the repair scan. If a user pages through very old run history, the API reads archived instance partitions instead of hot rows and returns a slower cursor. Multi-region active-active scheduling is risky because two regions can fire the same job; I would keep a single write/dispatch region per job and replicate history globally. Cold-starting a new dispatcher should rebuild only its shard window, not all 10M jobs.

Webhook reliability. Target may be flaky. Retry with exponential backoff + jitter, cap at N attempts. After exhausting retries, route to a dead-letter queue and surface in the UI.

Cron drift / time zone. Schedules stored in user-specified time zones. Server computes next_fire_at by converting to UTC at parse time. Daylight saving transitions need care (skipped hour, repeated hour) — document the convention.

Auditability. Every fire emits a structured event. JobInstance history retained for 30+ days. Useful for debugging and SLA reporting.

Graceful degradation.

Case Mode Source Latency / behavior
Dispatcher dies Standby claims shard and replays due range schedule index + Kafka duplicate possible, no miss
Top-of-minute hot slot Smear and batch instance writes timing wheel slot fire within 1-5 s depending SLA
Worker pool saturated Queue grows, autoscale, per-target caps fire queue delayed, visible backlog
Lease expires mid-run New worker may retry with same fire_id lease + instance DB at-least-once semantics
Schedule index corrupted Rebuild from catalog next_fire_at job catalog degraded accuracy during rebuild

Interview punch line: "I would rather be at-least-once with a stable fire_id than pretend exactly-once; the product promise is no silent misses."

What I'd defer. Sandboxed code execution for inline handlers (vs webhook targets), workflows/DAGs of jobs (cross-job dependencies), pricing/quotas, tenant isolation depth.
