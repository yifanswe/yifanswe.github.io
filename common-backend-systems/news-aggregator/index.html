News Aggregator — Google News Style (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Easy · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a news aggregator like Google News. We continuously crawl thousands of news sites, group articles about the same story into a cluster, and serve a homepage of "top stories" personalized by user interest. 100M daily active users (DAU), ~1M new articles/day from 10k publishers.

1. Requirements (~5 min)

Functional - Crawl publisher RSS/feeds and full pages on a schedule. - Cluster articles that are about the same story. - Serve a ranked homepage per user — fresh, diverse, personalized.

I'd defer paywalled content negotiation, deep editorial categorization, and notifications.

Non-functional - Freshness — a story breaking now should appear within ~minutes. - Homepage p99 < 300 ms. - Read-dominated: ~1M articles/day in, but 100M users × 10 page views = 1B reads/day → ~12k QPS average, ~50k peak. - Eventual consistency on cluster membership is fine — we can reshuffle clusters as new articles arrive.

Capacity - 1M articles/day × 5 KB metadata = 5 GB/day, plus raw HTML in object storage. - Cluster index: ~50k active clusters at any time (a story dies in 24–48h). - Raw HTML is the dominant store: 1M/day × ~50 KB = ~50 GB/day before compression; keep 30 days hot (~1.5 TB) and archive older pages cheaply. - Vector working set is smaller: 2M articles in the 48h index × 512 floats × 4 B ≈ 4 GB raw vectors, maybe 2–3× with HNSW graph overhead.

2. Core Entities (~2 min)

  • Publisher — feed_url, crawl policy, last fetched.
  • Article — url, publisher_id, title, published_at, embedding, cluster_id.
  • Cluster — cluster_id, articles, canonical title, topic tags, freshness score.
  • User profile — interest vector / topic preferences.

3. API (~5 min)

GET /v1/home?cursor=...        → { stories[], next_cursor }
GET /v1/stories/{cluster_id}   → { canonical, articles[], related[] }
GET /v1/topics/{topic}         → { stories[] }
POST /v1/interactions          { cluster_id, action }  // implicit signals for personalization

User derived from token.

4. Data Flow (~5 min)

  1. Crawler pulls publisher feeds on a per-publisher cadence (respect robots, rate limits).
  2. Fetcher retrieves full article HTML, extracts main content (readability).
  3. Enricher computes embedding, extracts entities, runs language detection.
  4. Clusterer assigns the article to an existing cluster (cosine similarity > threshold) or creates a new one.
  5. Ranker computes per-cluster freshness + quality + diversity scores.
  6. Personalizer intersects clusters with the user's interest vector at request time.

5. High Level Design (~10–15 min)

  1. Crawl scheduler. Per-publisher cron with backoff. Tier-1 publishers polled every 60s, tail every 30 min. Frontier in a priority queue (Redis sorted set by next-fetch time).
  2. Fetcher workers. Pull URLs, dedupe by canonical URL, write raw HTML to object storage. Push article record to Kafka topic articles.raw.
  3. Enrichment pipeline. Stream processor (Flink) consumes articles.raw, computes a sentence embedding (text encoder), writes enriched record to articles.enriched and a vector store (FAISS or Pinecone shard per topic).
  4. Clustering service. For each enriched article, search vector store for nearest neighbors among recent articles (last 48h). If best match > threshold, assign to that cluster; else create a new cluster. Cluster metadata in a primary store (MySQL or DynamoDB).
  5. Ranking pipeline. Periodic (every 30s) batch job recomputes per-cluster scores: freshness decay + publisher quality + total coverage + diversity penalty. Writes top-N clusters per topic into a Redis cache.
  6. Homepage service. On request: pull user's interest vector, fetch top clusters from the per-topic Redis cache, rerank by user fit, return top 20.
  7. Feedback loop. interactions events go to Kafka → updates the user interest model (offline retraining + lightweight online update).

End-to-end narration: a fire breaks at noon. NYT, Reuters, AP each publish within minutes. The crawler picks each up within its publisher cadence. Each article is enriched, vector-search finds the others, and a new cluster forms. The ranker on its next 30s tick scores this cluster highly (high freshness + 3 high-quality publishers). Within 5 minutes of the first article, the cluster is on the homepage for users whose interest vector includes "world news."

6. Deep Dives (~10 min)

Concrete stores and shard keys.

The interview point is that Redis holds indexes and ranked pointers, not article content. The big bytes are raw pages and article history; the low-latency path moves only cluster IDs and compact metadata.

Homepage read path. 1. Client calls GET /v1/home?cursor=C; API gateway authenticates and forwards user_id. 2. Homepage service decodes cursor {topic_offsets, seen_cluster_ids, rank_version} so pagination is stable across refreshes. 3. In parallel, it fetches profile:{user_id} from Redis/MySQL fallback and top lists top:{locale}:{topic} for the user's top 5–10 topics. 4. It merges ~500 candidate cluster_ids, drops seen_cluster_ids, applies diversity (MMR), then hydrates the top 20 from cluster:{cluster_id} and article snippets from article:{article_id}. 5. Cache miss path: if a topic list is missing, fall back to regional top:{locale}:frontpage; if cluster metadata is missing, skip that candidate rather than blocking the page. 6. Return stories plus an opaque cursor carrying offsets and dedupe state, not raw scores.

Article ingest path. 1. Scheduler pops due URLs from crawl:due:{host} and writes fetch attempts keyed by url_hash for dedupe. 2. Fetcher stores raw HTML in object storage, writes metadata row article:{article_id}, and emits articles.raw. 3. Enricher computes embedding/entities and writes articles.enriched; entity extraction and embedding can run in parallel after main-text extraction. 4. Clusterer searches vec:{locale}:{topic}:48h, updates cluster:{cluster_id} with a CAS on the centroid version, then inserts the vector. 5. Ranking job updates top:{locale}:{topic}; homepage reads the new pointer list on the next refresh window.

Hot homepage path vs cold path: 1. Hot path reads profile:{user_id} and the user's top topic lists in parallel, merges cluster IDs, and hydrates only the final 20 clusters. The shard keys are user_id, (locale, topic), and cluster_id. 2. Cold profile path falls back to top:{locale}:frontpage, returns a stable cursor, and queues an async profile rebuild from recent interactions. 3. Exhausted pagination path uses the cursor's {topic_offsets, seen_cluster_ids, rank_version} to continue within the same rank snapshot; if the Redis top list expired, it rebuilds from the cluster DB's rank snapshot and may return fewer stories. 4. Cache-miss hydration skips missing clusters rather than blocking the page, because freshness matters more than complete recall on the homepage.

Pipeline choice defense: Kafka/Flink-style streaming is a better fit than a pure cron batch because freshness is a product requirement measured in minutes. MySQL/DynamoDB works for cluster metadata because reads are by cluster_id and writes are small CAS updates; Elasticsearch is useful for keyword search, but it is not the primary clustering store. Redis top lists are pointer caches; using Redis as article storage would waste memory and complicate archival.

Clustering — the hard part. Naive K-means doesn't fit: clusters are unbounded and streaming. Use online clustering: each new article queries an HNSW index of recent articles (48h window); if cosine similarity to a centroid > 0.85, join; else seed a new cluster. Centroid is the running mean of member embeddings. Old articles age out of the index.

Why HNSW / FAISS instead of a SQL scan or periodic K-means: SQL cannot scan millions of recent vectors inside the ingest latency budget, and K-means assumes a mostly fixed cluster set. HNSW gives approximate nearest neighbor lookup in milliseconds and matches the streaming workload. Be ready for "why not Elasticsearch vector search?" — it is reasonable if search is already your operational base, but a dedicated vector index is easier to tune for recall/latency on the 48h working set.

Cluster quality. Bad clusters mix unrelated stories. Defense: hard cap similarity threshold, require ≥2 distinct publishers before promoting to homepage, decay centroids slowly so they don't drift.

Personalization without latency cost. Interest vectors are tiny (≤512 floats). Compute them offline, write to a per-user cache (Redis or in-process). At request time, the homepage service does a cheap dot product against precomputed top-K clusters per topic.

Two-view framing. Articles are accessed two ways: article:{article_id} is the content/metadata view, while top:{locale}:{topic} and cluster:{cluster_id}.members are pointer views over the same article set. The homepage should never scan articles directly; it merges ranked cluster pointers, then hydrates content at the end.

Freshness vs ranking stability. Pure freshness yields flicker — the top story changes every refresh. Use a hysteresis term so a cluster has to beat the current top by a margin before swapping in.

Diversity. Without a penalty, all top slots become the same story. Maximal Marginal Relevance (MMR) at rank time: each pick discounts clusters similar to already-picked clusters.

Crawl politeness and freshness budget. Per-publisher rate limits enforced in the scheduler. Use HTTP If-Modified-Since to avoid refetching unchanged pages. Sitemaps + RSS hubs (PubSubHubbub) push us toward zero polling latency for participating publishers.

Serving state machine:

Stage Mode Source Cursor/session state Latency target
Hot personalized profile + topic lists profile:{user_id}, top:{locale}:{topic} rank version + seen clusters < 300 ms
Cold user regional default top:{locale}:frontpage new cursor, empty profile < 300 ms
Exhausted deep pagination rank snapshot + cluster DB topic offsets + seen set < 800 ms
Fresh breaking promote cluster cluster metadata + trusted publisher rule rank version advances minutes to homepage
Degraded clustering lexical fallback SimHash/entity index low-confidence marker internal < 10 min freshness

Edge stress tests. A single breaking story can create write contention on one cluster:{cluster_id}; use centroid-version CAS with retry and batch member appends. Dormant users should not force profile recomputation on the request path. Publisher bursts are bounded by per-host crawl queues and stream backpressure. Multi-region drift is acceptable if each region shows its own rank_version and converges within a few minutes. Cold start after vector-index loss uses lexical clustering first, then repairs clusters when the HNSW index is rebuilt.

Graceful degradation.

Case Mode Source Latency budget Fallback
Hot breaking story push into fresh cluster vector index + cluster CAS < 5 min to homepage promote with 2-publisher rule relaxed for trusted wires
Dormant user profile cold personalize default regional top lists < 300 ms serve frontpage, rebuild profile async
Cursor past cached window cold pagination cluster DB by rank snapshot < 800 ms return fewer stories + new cursor
Vector index degraded lexical clustering title/entity SimHash < 10 min freshness mark low confidence, repair clusters later
Publisher throttles polite backoff Retry-After + crawl frontier next allowed slot rely on feeds/sitemaps from other publishers

"My punch line would be: the homepage is a pointer-serving problem; clustering is the expensive async problem, so I keep those failure domains separate."

Duplicate detection. Same article published on syndicated outlets. SimHash on title + first paragraph; if SimHash within 3 bits, treat as duplicate and merge metadata rather than creating a new article.

What I'd defer. Comment/discussion threading, paywall handling, video stories, fact-check integration, multilingual cluster merging.
