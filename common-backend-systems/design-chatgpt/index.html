1. Basic Knowledge And Solution

What This Question Is Testing

ChatGPT is a streaming chat interface over a large language model. The hard part is the GPU economics — every token generated costs GPU compute — combined with conversation state, streaming token delivery, safety filtering, and scaling across millions of concurrent sessions on a finite GPU fleet.

The invariant is this:

Every chat request is routed to an inference worker with the requested model loaded, generates tokens within the model's latency budget, streams them to the client, applies safety filtering before display, persists the conversation history per user, and is billed/quota-tracked deterministically; GPU capacity is shared across users with fairness controls.

At ICT5 / L6+ level, the interviewer is testing whether you can design GPU inference serving with batching, streaming token delivery via SSE/WebSocket, conversation state management, safety filtering pipelines, and reason about GPU cost optimization (continuous batching, KV cache, quantization).

60-Second Interview Framing

Spoken version:

I would design ChatGPT around four core systems: a chat session service holding conversation state, an inference fleet of GPU workers serving models with continuous batching, a streaming gateway that delivers tokens as they're generated, and safety filtering pipelines on both input and output. The interesting decisions are GPU scheduling and batching, conversation context management, streaming protocols, safety integration, and cost optimization.

Requirements To Clarify First

Functional requirements:

  • User sends prompts; assistant streams responses.
  • Multi-turn conversations with history.
  • Multiple models (GPT-4, GPT-4o, etc.).
  • File and image uploads (vision).
  • Function calling / tool use.
  • Conversation history saved per user.
  • Edit / regenerate messages.
  • Custom instructions per user.

Non-functional requirements:

  • Time-to-first-token under 1 second.
  • Token-per-second throughput acceptable for streaming UX.
  • Tens of millions of concurrent sessions.
  • High availability.
  • Strict safety: filter unsafe content.
  • Multi-region.
  • Pay-per-token billing accuracy.

Out of scope unless asked:

  • Model training pipeline (separate; massive distinct system).
  • Custom GPTs / agent platform internals.
  • Memory feature internals.
  • Plugin / browsing details.

Inference Flow

  1. User sends message.
  2. API auth + quota check.
  3. Input safety filter: classify prompt; reject if violates policy.
  4. Append user message to conversation in session store.
  5. Build inference request: full conversation history + system prompt.
  6. Send to inference scheduler.
  7. Scheduler picks GPU worker with the requested model loaded.
  8. Worker enqueues request into continuous batch.
  9. Forward pass generates one token at a time.
  10. Each token streams back to client via SSE.
  11. Output safety filter applied per token or per chunk.
  12. On completion, persist assistant message; update quota.

Continuous Batching

The core GPU optimization. Naive: one request at a time per GPU. Wasteful — GPUs are massively parallel.

Continuous batching: - GPU runs inference in batches of N requests. - Each forward pass produces one token for each active request in the batch. - Completed requests leave the batch; new ones join. - KV cache (key-value attention cache) maintained per request. - Effective throughput much higher than sequential.

Implementation: vLLM, TGI (HuggingFace Text Generation Inference), or custom serving stack.

Interview phrase:

Continuous batching is the GPU economy. Without it, a single 8xH100 server serves a handful of users; with it, hundreds. KV cache management determines how many concurrent streams the worker holds.

KV Cache

Transformer attention requires the keys and values of all prior tokens. Recomputing per token is wasteful.

  • KV cache stores K, V tensors for each token in the active conversation.
  • Memory cost scales with context length and concurrent requests.
  • KV cache eviction: when memory pressured, complete a request or swap to CPU.
  • Pre-allocated cache pools per worker; bounded concurrency per worker.

Streaming Protocol

Server-Sent Events (SSE) is the typical choice:

  • Server sends data: {token}\n\n per token.
  • Client receives via EventSource API.
  • Half-duplex; client → server is separate request.

WebSocket is an alternative for full-duplex; useful for cancel mid-stream.

The streaming connection holds open for the duration of the response (seconds to minutes for long generations).

Safety Filtering

Two phases:

Input filter: - Classifier checks user prompt for policy violations. - Reject obvious unsafe (CSAM, terrorism, etc.). - Soft-warn or transform for borderline (e.g., medical, legal advice).

Output filter: - Per-token or per-chunk classifier. - If output looks unsafe, abort stream + apologize. - Some checks happen on full response (jailbreak detection).

For sensitive topics: model fine-tuning enforces refusals + RLHF; safety classifier is the belt-and-suspenders.

Hybrid: small fast classifier inline + larger model for ambiguous cases.

Conversation State Management

Conversation history is the model's context. As history grows:

  • Context window limit (8k, 32k, 200k tokens depending on model).
  • For long conversations, truncate or summarize older messages.
  • Summarization: periodically summarize and replace early messages with summary.
  • "Memory" feature: extract long-term facts about the user, store separately, inject into system prompt.

Persist conversation messages durably so user can resume.

GPU Fleet Scheduling

Workers in the fleet:

  • Each worker has one or more GPUs.
  • A worker loads one (or sometimes a few) models.
  • Workers grouped by model; scheduler routes to the right pool.
  • Within a pool, scheduler picks workers with capacity.
  • Bin-packing across the fleet.

Auto-scaling:

  • Track active streams per model.
  • Add workers when demand rises; drain idle workers.
  • Cold start: load model weights (large, slow).
  • Mitigation: keep a warm pool; pre-load popular models.

Multi-model deployment: - Larger models (GPT-4) need multi-GPU per worker (model parallelism). - Smaller models (3.5) fit on one GPU. - Quantization (FP8, int8) reduces memory and increases throughput.

Tool / Function Calling

Model decides to call a tool (function with defined schema):

  1. Model output: tool_call(name, args).
  2. Streaming pauses; client or server executes tool.
  3. Tool result fed back as a new message.
  4. Inference resumes with the result in context.

Supports browsing, code execution, image generation, custom tools.

Vision / Multimodal

Images are tokenized into image embeddings:

  1. Image uploaded; vision preprocessor extracts patches.
  2. Embeddings prepended to text tokens in the prompt.
  3. Model handles unified attention.

Adds latency and memory cost; pre-cached for repeat use.

Multi-Region Strategy

  • Conversation storage per user home region.
  • GPU fleets in multiple regions.
  • Inference routed to nearest region with capacity for the requested model.
  • Cross-region failover possible.

Operational Metrics

Correctness:

  • Safety filter false positives/negatives.
  • Conversation persistence success.
  • Quota drift vs billed.

Capacity:

  • TTFT (time to first token) p99.
  • Tokens/sec per stream.
  • GPU utilization.
  • Queue depth.

Business:

  • Active users; tokens per user.
  • Tool call usage.
  • Conversion to paid.

Tradeoffs To Call Out

  • Larger batch = better GPU utilization but worse TTFT.
  • Larger context = better quality but more memory and slower.
  • Tighter safety = fewer incidents but more refusals (user complaints).
  • Quantization saves memory but degrades quality slightly.
  • More aggressive auto-scaling = lower latency but higher cost.
  • Sticky streaming connections vs reconnect-on-failure.

2. Simulated Interview

Interviewer: Design ChatGPT.

Candidate: Four core systems: chat session service holding conversation history, inference fleet of GPU workers with continuous batching, streaming gateway delivering tokens via SSE, and safety filtering on both input and output. The invariant is every request runs through safety, generates within budget, streams to client, persists history, and decrements quota. The dominant cost is GPU; the dominant performance challenge is TTFT.

Interviewer: Walk me through a chat message.

Candidate: User submits prompt. Gateway authenticates, checks quota. Input safety classifier scans for policy violations — obvious unsafe is rejected. Session store appends the user message to conversation. Inference request is built: full message history + system prompt. Scheduler picks a GPU worker pool for the requested model, picks a worker with capacity. Worker enqueues into continuous batch. Forward pass starts generating tokens. Each token streams back via SSE; output safety filter validates per chunk. On done, persist assistant message and update quota.

Interviewer: Continuous batching — explain.

Candidate: GPUs are massively parallel; sequential per-request inference wastes compute. Continuous batching combines concurrent requests into a single forward pass. Each pass produces one token for every active request in the batch. KV cache maintained per request so subsequent passes don't recompute prior tokens. Completed requests leave the batch; new ones join. Result: a single H100 server handles hundreds of concurrent streams instead of a handful. vLLM and TGI implement this pattern.

Interviewer: KV cache?

Candidate: Transformer attention requires keys and values of all prior tokens. Recomputing per token is O(n²). KV cache stores K, V tensors after each token, so the next token's attention is O(n) marginal. Memory cost scales with context length × concurrent requests. Workers pre-allocate KV cache pools; concurrency is bounded by total cache memory.

Interviewer: TTFT — how do you keep it low?

Candidate: Several factors. Time-to-first-token includes: queue wait, prefill (process all input tokens in parallel), then first decode step. Mitigations: keep GPU fleet warm with model loaded; bound queue depth; prioritize TTFT over batch size. Continuous batching helps even TTFT because new requests join active batches without waiting for the batch to drain.

Interviewer: Streaming protocol?

Candidate: SSE typically — server sends data: {token}\n\n per token. Client uses EventSource API. Half-duplex; client-to-server is a separate POST. WebSocket as alternative for full-duplex (cancel mid-stream). Connection stays open for the response duration — seconds to minutes for long generations. Reconnect-on-drop with cursor for resumption.

Interviewer: Safety?

Candidate: Two-stage. Input filter: classifier scans prompt; rejects obvious unsafe (CSAM, terrorism, etc.). Soft-warn for borderline. Output filter: per-token or per-chunk classifier — if output drifts unsafe, abort with apology. Belt-and-suspenders: model fine-tuning enforces refusals via RLHF, safety classifier is a secondary check. Some checks evaluate full response post-stream for jailbreak patterns.

Interviewer: Long conversation — 100 messages, 50k tokens.

Candidate: Context window limit (depending on model: 8k, 32k, 128k, 200k). For longer: truncate or summarize. Summarization: periodically replace early messages with a summary message, keeping the recent N verbatim. "Memory" feature: extract long-term facts about the user separately, inject into system prompt. The full history is persisted for user reference; only the windowed/summarized version goes to the model.

Interviewer: Tool calling?

Candidate: Model output may include tool_call(name, args) in a structured format. Streaming pauses; the server or client executes the tool (browse the web, run code, call an API). Tool result fed back as a new message. Inference resumes with result in context. Supports plugins, browsing, code execution, image generation, custom tools.

Interviewer: GPU fleet — how do you scale?

Candidate: Workers grouped by model. Each worker loads one (or a few) models; bigger models use multi-GPU model parallelism. Auto-scaling per model based on demand. Warm pool pre-loaded with popular models. Cold start is slow (loading multi-GB weights) — mitigated by keeping minimum warm capacity. Quantization (FP8, int8) reduces memory and increases throughput at small quality cost.

Interviewer: A worker crashes mid-stream.

Candidate: The streaming connection breaks. Client receives partial response. On reconnect, server can resume from the persisted partial: re-prefill the context up to the last delivered token and continue generation on a new worker. Or just present the partial result with a regenerate button. Depends on UX choice.

Interviewer: Quota and billing?

Candidate: Per-user token budget. Each token decrement counter (input + output tokens). Pre-flight check: estimate response size; reject if grossly over budget. During streaming, real-time decrement; if depleted, stop generation and surface "quota exhausted." Billing aggregates usage periodically; charges per plan.

Interviewer: Multi-region?

Candidate: Conversation storage per user home region. GPU fleets in multiple regions. Inference routes to nearest region with capacity for the requested model. Cross-region failover if local capacity exhausted. Some users may experience higher latency if their home region is heavily loaded.

Interviewer: Summarize.

Candidate: Streaming chat over GPU inference. Continuous batching is the GPU efficiency lever. KV cache management bounds concurrency per worker. SSE delivers tokens to client. Two-stage safety (input + output) filters policy violations. Conversation history persisted per user with truncation/summarization for long contexts. Tool calling integrates external execution. Per-user quota and billing. Multi-region fleets. The hardest economic problem is GPU cost; the hardest performance problem is TTFT.

3. Practice Notes

Five-Minute Version

  1. Clarify: chat, streaming, models, safety, scale.
  2. Invariant: safe input/output; streamed tokens; persisted history.
  3. Session + inference + streaming + safety.
  4. Continuous batching on GPUs with KV cache.
  5. SSE streaming protocol.
  6. Two-stage safety filtering.
  7. Conversation truncation/summarization for long contexts.
  8. Per-user quota; multi-region.

One-Minute Closeout

ChatGPT is streaming chat over GPU inference. Continuous batching with KV cache is the GPU efficiency lever — it turns each GPU from a few-stream server into a hundred-stream server. SSE delivers tokens. Two-stage safety filters input and output. Conversation history persisted per user; long contexts truncated or summarized. Tool calling integrates external execution. Per-user quota tracks tokens. Multi-region with GPU fleet auto-scaling. The dominant cost is GPU; the dominant performance metric is time-to-first-token.
