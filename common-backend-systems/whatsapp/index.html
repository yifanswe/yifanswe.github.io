WhatsApp — Messaging (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build WhatsApp. Two billion users send 100B messages/day. One-on-one and group chats, end-to-end encrypted, with delivery and read receipts, across flaky mobile networks and offline devices. Multiple devices per user.

1. Requirements (~5 min)

Functional - Send a text message in a one-on-one chat. - Send a message in a group (up to ~1000 members). - See delivery and read receipts (✓ sent, ✓✓ delivered, blue ✓✓ read). - Receive messages while offline, see them when device wakes up.

I'd defer voice/video calling and media-storage deep-dive.

Non-functional - End-to-end encryption — server stores only ciphertext. - Delivery is exactly-once per recipient semantically (no dupes shown to user). - Per-conversation message order preserved. - p99 message-send-to-recipient < 1 s on healthy networks. - Survive offline; deliver on reconnect. - Multi-device per user (phone + web + desktop) — message must reach all devices.

Capacity - 100B msgs/day → ~1.2M msgs/sec average, 3–5M peak. Each msg ~500 bytes ciphertext. - 2B users; assume 3 devices each → 6B active session tokens. - Ciphertext volume: 100B/day × 500 B ≈ 50 TB/day before replication. If offline queues retain 30 days, that is ~1.5 PB primary, ~4.5 PB with 3 replicas; the offline queue is the dominant storage cost. - Connection map is much smaller: 600M online devices × ~64 B mapping ≈ 38 GB primary, but it needs very low-latency lookups and constant churn handling.

2. Core Entities (~2 min)

  • User — user_id, phone_number, devices.
  • Device — device_id, public key, last seen.
  • Chat — one-on-one (implicit, derived from user pair) or group (chat_id, members).
  • Message — msg_id, chat_id, sender_device_id, ciphertext, ts, server_ts.
  • DeliveryReceipt — msg_id, recipient_device_id, state (sent/delivered/read).

3. API / Protocol (~5 min)

Mostly a custom protocol over a persistent connection. Sketch:

WS  /v1/connect                   // long-lived, auth via token + device_id
  → server pushes: { msg, receipt, presence }
  ← client sends:  { send_msg, ack_received, mark_read }

HTTP fallback for media:
POST /v1/media/upload             → { media_id }
GET  /v1/media/{id}               → bytes

Each message carries a client-generated UUID for idempotency.

The reconnect cursor is server-issued per device and encodes the last acknowledged per-device queue sequence; clients do not ask for arbitrary offsets.

5. High Level Design (~10–15 min)

  1. Edge connection servers. Long-lived WebSocket (or XMPP-like) terminations, sharded by device_id. A device is "online" iff a connection server holds its WS. Connection-server-to-device map kept in a Redis cluster.
  2. Routing service. When user A's connection server receives a send, it looks up each recipient's device_id → connection_server and forwards. If a device is offline, write the message to its offline queue.
  3. Message store / offline queue. Per-device FIFO queue (Kafka topic per shard, or a sharded Cassandra/HBase table) for messages a device hasn't yet ACK'd. On reconnect, the device drains its queue.
  4. Group fan-out service. For groups, expand chat_id → member device_ids, enqueue one entry per device. Server never decrypts — it just routes ciphertext.
  5. Key directory. Per-device public keys (Signal-style identity + signed prekey + one-time prekeys). Senders pull keys to establish sessions.
  6. Media service. End-to-end encrypted media is uploaded as encrypted bytes; the chat message carries the media URL + decryption key in the encrypted payload.
  7. Presence / receipts. Receipt updates flow back over the same connection — delivered on device receipt, read on user view. Sender's clients are updated by the same routing path.

End-to-end narration: Alice's phone sends {msg_id: U1, chat: A-B, ciphertext} over her WS → her connection server looks up Bob's device list and connection servers → Bob's phone is online (one connection server), Bob's web is offline. Forward to phone connection server, which delivers; phone returns delivered receipt. Web device's offline queue gets the message ciphertext; when web reconnects in 3 hours, it drains the queue. Bob's phone shows the message; tapping the chat sends a read receipt back through the routing service to Alice's devices.

6. Deep Dives (~10 min)

Concrete store contracts.

Store Key / data structure Entry size + TTL Shard key
Connection map conn:{device_id} → {edge_id, session_id} in Redis ~64 B, TTL heartbeat 60s device_id
Offline inbox wide-column inbox(device_id, seq) → ciphertext envelope ~600 B/message, TTL 30d or until ACK device_id
Dedupe dedupe:{device_id}:{msg_uuid} ~32 B, TTL 30d device_id
Group membership group:{chat_id} → member user/device ids + version 10 KB–1 MB, retained chat_id
Key directory keys:{device_id} identity/signed/one-time prekeys ~1–5 KB, retained/rotated device_id
Receipts receipt:{msg_id}:{device_id} state + ts ~32 B, TTL or compacted msg_id

Capacity by store. Offline inbox is the dominant cost: 100B messages/day × 600 B is ~60 TB/day primary, and 30 days at 3 replicas is ~5.4 PB if every message waits out the TTL. In practice ACK compaction deletes most entries earlier, but the design must budget for mobile devices being offline. Connection map is large but manageable: 600M online devices × 64 B × 3 is ~115 GB. Dedupe cannot naively keep every message UUID for 30 days in memory; keep short hot dedupe for delivered messages and align longer dedupe with inbox retention for undelivered devices. Group membership is read-hot but small compared with inbox bytes; receipt streams are append/compact, not permanent per-message rows forever.

Send path trace. Alice's edge server receives {msg_uuid, chat_id, ciphertexts} → idempotency check on Alice device → chat service reads recipient devices/group membership version → routing looks up conn:{device_id} for online devices in parallel → online devices get pushed over WS → offline devices get appended to inbox(device_id, seq) → each receiving device ACKs seq → receipt event routes back to Alice's devices. Group fan-out parallelizes by recipient-device shard; message ordering is per chat_id sequence, while storage delivery is per-device FIFO.

Multi-device fan-out. A user with N devices means N copies of every message. Sender encrypts independently for each recipient device using the Signal Double-Ratchet protocol (or Sender Keys for groups). Server treats each (message, device) as an independent routing target. Sender's own devices also receive copies (so phone and web stay in sync).

Group messages without quadratic blowup. For a 1000-member group, naive per-device encryption gives sender 1000+ ciphertexts to upload. Signal's Sender Keys: sender encrypts the message once with a symmetric chain key, distributes the chain key per-recipient out-of-band when needed (when membership changes or initially). Tradeoff: forward-secrecy is weaker than per-message keys, but it makes 1000-person groups feasible.

Per-conversation ordering. Server assigns a monotonic per-chat sequence number on receipt (timestamp + tiebreaker). Recipient devices order by that. Sender's client-side timestamp is shown but server seq is the source of truth.

Two-index framing. Message order is chat-keyed, while delivery is device-keyed. The server assigns a per-chat sequence so every participant sees the same conversation order, then writes per-device inbox entries so each phone/web client can reconnect independently. Those are two indexes over encrypted envelopes, not two competing message stores.

Offline delivery. Per-device offline queue is the contract. Capped at a TTL (e.g., 30 days). On reconnect, the device pulls everything since its last ACK'd offset. Once the device ACKs delivery, server drops from the queue.

Why a wide-column inbox instead of Kafka per device: Kafka is excellent for service-to-service streams, but 6B devices need random per-device replay, TTL, and independent ACK offsets. A Cassandra/HBase-style inbox keyed by device_id makes reconnect cheap. Kafka still fits for internal receipt and analytics streams.

Compared with Redis-only queues, a wide-column inbox survives long offline windows and regional restarts without keeping petabytes in memory. Compared with a relational table, the access pattern is almost always device_id + increasing seq, so wide-column storage keeps writes and reconnect scans predictable.

At-most-once vs at-least-once. Server delivers at-least-once; client deduplicates by msg_id (UUID from sender). Net effect: exactly-once shown to user.

Connection scaling. 2B devices × 30% online = 600M concurrent WS connections. Each connection server holds ~500k connections; we need 1200+ machines just for edges. Use efficient async I/O (epoll/kqueue) and minimal per-connection memory.

End-to-end encryption — server's role. Server never sees plaintext. Server stores (ciphertext, recipient_device_id). Receipt metadata (delivered/read flags) is plaintext server-side. Acknowledge that abuse moderation is therefore client-side reports plus metadata signals.

Interview punch line: "The server is a durable encrypted router; it owns delivery semantics, not message meaning."

Hot-shard mitigation. Famous user broadcast lists. Cap broadcast list size; rate-limit per-sender; for businesses, use a separate API tier.

Graceful degradation state machine.

Case Mode Source Latency budget Fallback
Recipient online Push Connection map + WS <1s Store then push if routing uncertain
Recipient offline Queue inbox(device_id, seq) Reconnect-dependent Retain 30d, notify sender only as sent
Device dormant > TTL Expired Inbox TTL N/A Sender sees sent, not delivered; client may request history sync if allowed
Edge server fails Reconnect Connection map heartbeat expires Seconds Device reconnects to another edge and resumes from cursor
Group fan-out hot Async fan-out group membership version + inbox shards Seconds for large groups Accept send, enqueue fan-out job, deliver progressively
Duplicate retry Dedupe msg_uuid per device <50 ms Return prior ACK; client shows one bubble

What I'd defer. Voice/video calls (WebRTC), status/stories, business APIs, backup encryption to iCloud/Google Drive, machine learning on metadata for spam.
