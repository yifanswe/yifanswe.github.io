1. Basic Knowledge And Solution

What This Question Is Testing

Stripe is the canonical merchant payment gateway: API-first, developer-friendly, abstracts away the complexity of card networks, banks, regulatory regimes. It builds on Design a Payment System but emphasizes the gateway layer: API design, idempotency, webhooks, SDK integration, and multi-method support.

The invariant is this:

Every API call to charge a customer produces exactly one terminal outcome (succeeded, failed, requires_action) for that payment intent; idempotency keys make retries safe; webhooks reliably notify merchants of every state change; the API behaves predictably across hundreds of payment methods and regions.

At ICT5 / L6+ level, the interviewer is testing whether you can design the developer-facing API surface, idempotency end-to-end, webhook reliability, payment method abstraction, and reason about regulatory complexity (PSD2 SCA, KYC, AML).

60-Second Interview Framing

Spoken version:

I would design Stripe as a developer-facing payment gateway. Four core systems: API gateway with idempotency, payment intent state machine, integration adapters to card networks and alternative payment methods, and webhook delivery to merchants. The interesting decisions are idempotency end-to-end, payment method abstraction, webhook reliability, and 3DS / SCA flows.

Requirements To Clarify First

Functional requirements:

  • Merchants integrate via REST API and SDKs.
  • Charge a customer (one-time, subscription).
  • Multiple payment methods: cards, ACH, SEPA, wallets (Apple Pay, Google Pay), local methods.
  • Idempotency on all writes.
  • Webhooks notify merchants of events.
  • Refunds, voids, disputes.
  • 3DS / SCA flows for European compliance.
  • Customer / Payment Method tokenization.
  • Multi-currency.

Non-functional requirements:

  • API latency under 500ms p99.
  • High availability per region.
  • PCI-DSS compliance.
  • Idempotent retries.
  • Webhook delivery within seconds with retries.
  • Auditability.

Out of scope unless asked:

  • Connect platform (multi-tenant marketplace flows).
  • Issuing (Stripe Issuing).
  • Treasury / banking-as-a-service.

Payment Intent Flow

created -> requires_payment_method -> requires_confirmation -> processing -> succeeded
   |                |                          |                |
   v                v                          v                v
 canceled        requires_action          requires_action    failed

Sub-states for 3DS / SCA: requires_action means the customer must complete a challenge (3DS redirect, push notification, etc.).

Idempotency

Every write API call accepts Idempotency-Key header.

  • Per-merchant scope.
  • TTL: 24 hours typically.
  • First request with a key stores response.
  • Subsequent requests with same key return the same response (and don't re-execute).
  • Different requests (different body) with same key return error.

The idempotency store is a fast KV: (merchant_id, idempotency_key) → response.

For provider calls: Stripe maintains its own idempotency to the provider. Different from merchant's key. This means if a merchant retries with the same key, Stripe doesn't double-charge at the provider level.

Interview phrase:

Idempotency is end-to-end. The merchant's idempotency key dedupes at our API; we maintain a separate provider-side idempotency key so the upstream network doesn't see duplicates. Three layers: merchant→Stripe, Stripe→provider, and the payment intent state machine itself.

Tokenization And Card Vault

Raw card data never leaves the PCI-isolated vault.

  • Merchant collects card via Stripe.js or mobile SDK; raw card goes directly to Stripe (never touches merchant's servers).
  • Stripe vaults the card; returns a payment_method_id.
  • Merchant uses pm_id for charges; can't access raw card.
  • Vault is in heavily-isolated infrastructure with PCI DSS Level 1 audit.

This is what brings most merchants into PCI scope reduction.

3DS / SCA Flow

European PSD2 requires Strong Customer Authentication (SCA) for most transactions.

  1. PaymentIntent created with payment_method requiring SCA.
  2. API returns requires_action with next_action describing the challenge.
  3. Frontend SDK loads the challenge (typically iframe or redirect to issuing bank).
  4. Customer completes (password, biometric, OTP).
  5. Bank issues authentication result.
  6. Frontend confirms PaymentIntent.
  7. PaymentIntent transitions to processing then succeeded or failed.

The client_secret is used by the frontend to access the PaymentIntent without exposing the API key.

Provider Integration

Each payment method has its own provider adapter:

  • Cards: Visa, Mastercard via acquirer banks; Stripe is the acquirer.
  • ACH: US bank transfers.
  • SEPA: European bank transfers.
  • Apple Pay / Google Pay: wallet tokens, processed as cards.
  • Local methods: iDEAL (NL), Bancontact (BE), Alipay, WeChat Pay, etc.

Adapters are pluggable. Each handles its own retry, idempotency, error mapping.

Webhook Delivery

Stripe events fire webhooks to merchants. Reliability is critical.

  1. Event published to internal event stream.
  2. Webhook delivery service consumes; looks up merchant's webhook endpoints subscribed to this event type.
  3. For each endpoint, POST event payload (signed with merchant's webhook secret).
  4. On 2xx response within timeout: success.
  5. On failure: retry with exponential backoff for up to 3 days.
  6. After exhaustion: mark dead-letter; merchant can replay from dashboard.

Signing prevents forgery:

  • Stripe computes HMAC of (timestamp + body) with merchant's webhook secret.
  • Adds Stripe-Signature header.
  • Merchant verifies signature before processing.

Event Stream

All state changes emit events. Event types are versioned (API version of the event).

Examples: - payment_intent.succeeded - payment_intent.payment_failed - charge.refunded - customer.created - invoice.paid

Fraud (Stripe Radar)

Inline ML scoring per charge:

  • Features: card patterns, behavioral, geographic, prior history.
  • Returns risk score.
  • High-risk: decline or require additional verification.
  • Async deeper analysis for patterns across merchants.

Disputes And Chargebacks

When customer disputes a charge:

  1. Card network sends chargeback to Stripe.
  2. Webhook fires to merchant.
  3. Merchant provides evidence (receipts, communication, delivery proof).
  4. Stripe submits to card network.
  5. Resolution: lost (refund + fee) or won.
  6. Status visible in dashboard.

Reconciliation

Daily, Stripe reconciles its internal records against card network settlement files. Drift triggers investigation.

  • Provider integrations regional (different acquirers per region).

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Provider timeout. Mark intent processing; reconcile. Unknown-state count.
Duplicate API call. Idempotency key. Dup retry rate.
Webhook endpoint down. Retry queue with backoff. Per-endpoint failure rate.
Card decline. Standard reason codes; surface to merchant. Decline rate.
Fraud false positive. Review queue; merchant can override per case. False positive rate.

Tradeoffs To Call Out

  • Synchronous provider call adds latency but gives merchant immediate result.
  • Async confirmation simplifies the API but adds webhook complexity.
  • Aggressive fraud scoring reduces losses but increases false positives.
  • Strict 3DS improves security but adds friction.
  • Token vault centralizes PCI scope but is critical infrastructure.
  • Multi-region per data residency vs cross-region failover ease.

Common Traps And Stronger Moves

2. Simulated Interview

Interviewer: Design Stripe.

Candidate: Stripe is a payment gateway built around the PaymentIntent abstraction — a state machine for one charge from creation through confirmation, SCA challenge, processing, and success. Four core systems: developer-facing API with idempotency, payment intent state machine, provider adapter layer abstracting card networks and alternative methods, and webhook delivery to merchants. The invariant is exactly one terminal outcome per payment intent and idempotent retries throughout.

Interviewer: Walk me through a charge.

Candidate: Merchant POSTs /v1/payment_intents with Idempotency-Key and amount, currency, payment method. Idempotency check: if seen, return cached response. New intent created in requires_payment_method or directly in processing if confirm=true. Provider adapter is invoked with the payment method (e.g., card via Visa/Mastercard acquirer). Provider returns success, failure, or 3DS challenge. On success: intent transitions to succeeded; event emitted; webhook fired. On 3DS: intent moves to requires_action; frontend completes challenge; merchant calls confirm; flow resumes.

Interviewer: Idempotency — explain the three layers.

Candidate: First layer: merchant's API call uses Idempotency-Key header. Stripe's idempotency store caches the response by (merchant_id, key). Retries return the same response without re-executing. Second layer: Stripe's call to the provider (card network) uses Stripe's own idempotency mechanism — different from merchant's. This means even if merchant retries with same key, the provider sees one charge. Third layer: payment intent state machine — transitions are CAS on (intent_id, from_state). A retried transition fails because state has advanced.

Interviewer: Webhooks.

Candidate: Events emit to an internal stream for every state change. Webhook delivery service consumes events, looks up merchant's subscribed endpoints, POSTs payload with HMAC signature header. On 2xx within timeout: success. On failure: retry with exponential backoff up to 3 days. After exhaustion: dead-letter; merchant can replay from dashboard. Signing uses merchant's webhook secret; merchant verifies before processing.

Interviewer: 3DS / SCA?

Candidate: PaymentIntent transitions to requires_action with next_action describing the challenge. The API returns client_secret which the frontend uses to access the intent. Frontend SDK presents the challenge (iframe to issuing bank). Customer completes (password, biometric, OTP). Bank issues authentication result. Frontend calls confirm; intent transitions to processing then succeeded or failed.

Interviewer: Card vault.

Candidate: Raw card data never leaves the PCI-isolated vault. Merchants integrate via Stripe.js or mobile SDK; raw card goes directly to Stripe (skipping merchant servers, reducing PCI scope). Vault tokenizes; returns payment_method_id. Merchants use pm_id for charges. The vault is in heavily-audited infrastructure (PCI DSS Level 1).

Interviewer: Provider abstraction.

Candidate: Pluggable adapter per payment method. Cards via Visa/Mastercard acquirer. ACH for US bank. SEPA for Euro bank. Wallets (Apple Pay, Google Pay) processed as cards. Local methods (iDEAL, Alipay, etc.) each with their own adapter. Each adapter handles its own retry, idempotency mapping, error code translation. The PaymentIntent state machine is provider-agnostic.

Interviewer: Provider times out.

Candidate: Intent stays in processing with unknown_state indicator. A reconciliation worker queries the provider with the same idempotency key. On confirmed success: transition to succeeded. On confirmed failure: transition to failed. Never retry blindly — that could double-charge.

Interviewer: Disputes / chargebacks.

Candidate: Card network sends chargeback to Stripe. Webhook fires charge.dispute.created to merchant. Merchant provides evidence (receipt, communication, delivery proof) via API. Stripe submits to network. Resolution: lost (refund + fee) or won. Webhook fires final resolution. Merchant sees dispute lifecycle in dashboard.

Interviewer: Fraud (Radar)?

Candidate: Inline ML scoring per charge. Features: card patterns, behavioral signals, geographic, account history. Returns risk score. High-risk: decline or require additional verification. Custom rules per merchant via dashboard. Async deeper analysis across merchants for emerging fraud patterns.

Interviewer: Multi-region?

Candidate: API endpoints per region. Card vault per region for data residency compliance (EU data stays in EU). Provider integrations regional (different acquirers per geography). Webhook delivery from merchant's region. Cross-region replication for DR. Data residency is a strict constraint, not just a perf optimization.

Interviewer: Reconciliation?

Candidate: Daily. Stripe reconciles internal payment records against card network settlement files. Mismatches go to review. Aggregate drift triggers investigation. The cold path (reconciliation) is authoritative for billing and reporting; the hot path (real-time intent state) is operational.

Interviewer: Summarize.

Candidate: PaymentIntent state machine as central abstraction; pluggable provider adapters; three-layer idempotency; PCI-isolated card vault; 3DS via client_secret + next_action; at-least-once webhook delivery with HMAC signing and retry; daily reconciliation; multi-region with data residency. The hardest correctness problem is exactly-one terminal outcome per intent under retries and timeouts; idempotency end-to-end is the answer.
