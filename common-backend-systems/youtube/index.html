YouTube — Video Upload, Storage, Playback (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build YouTube. Creators upload videos; viewers search/discover and stream them at appropriate bitrate for their network. 500M creators, 2B daily active viewers, 500 hours of video uploaded per minute, hundreds of millions of concurrent streams. Storage and CDN-egress economics drive the design.

1. Requirements (~5 min)

Functional - Upload a video. - Stream a video at a quality matched to the viewer's network (adaptive bitrate). - View counts and a watch page (title, thumbnail, owner, basic metadata).

I'd defer recommendation algorithm, comments, live streaming depth, and monetization.

Non-functional - Playback start time < 2 s — the dominant UX metric. - Cost-efficient at exabyte scale; storage and egress are the budget. - Multi-region, low-latency video serving via CDN. - Eventual consistency on view counts and metadata. - Durability — eleven nines for source videos.

Capacity - 500 hr/min × 60 min × 24 = 720k hr uploaded/day. At 1 GB/hour (post-encode, multi-bitrate ladder): ~720 TB/day uploaded; multi-ladder per video pushes this 5–10× → ~5 PB/day of stored bytes. - 2B viewers × 60 min × 5 Mbps avg = ~75 Pb/s aggregate egress. - Metadata is tiny beside media: even 10B video rows × 2 KB is ~20 TB primary. Media dominates: 5 PB/day of new ladder bytes means 150 PB/month before replication/tiering. - CDN egress is the business cost: a single popular 10-minute video at 5 Mbps watched 100M times is ~375 TB delivered. The design is mainly about avoiding origin reads and storing fewer bits per minute.

2. Core Entities (~2 min)

  • Video — video_id, owner, title, description, duration, status, source_url, ladder.
  • Ladder — set of (resolution, bitrate, codec) → segment_url_list. The streamable artifact.
  • Upload — multipart upload state.
  • Viewer event — view, watch-time ping. Drives ranking + analytics.

3. API (~5 min)

POST  /v1/uploads                       → { upload_id, multipart_urls[] }
PUT   <multipart_url>                   (binary chunks)
POST  /v1/uploads/{upload_id}/complete  → { video_id, status: processing }
GET   /v1/videos/{id}                   → metadata + ladder manifest URL
GET   <manifest_url>                    → HLS/DASH manifest
GET   <segment_url>                     → media segment

Manifest URLs are CDN-fronted; signed for private videos.

The upload cursor is the multipart upload id plus completed part numbers; the playback cursor lives in the player manifest/segment sequence, not in the API server.

4. Data Flow (~5 min)

Upload (multipart) → original stored in object store → transcoding pipeline produces ABR ladder → manifest published → CDN warmed → video listed.

Viewer requests manifest → player picks bitrate → fetches segments via CDN → adapts up/down based on buffer health.

5. High Level Design (~10–15 min)

  1. Upload service. Generates a multipart upload to object storage (S3). Client uploads directly. On complete, writes a Video row (status=processing) and publishes video.uploaded.
  2. Transcoding pipeline. - Job scheduler distributes segments of the source to a worker fleet (GPU-accelerated, ffmpeg or x264/x265/AV1). - For each bitrate × resolution × codec combination in the ladder, produce HLS/DASH-compatible segments (typically 2–6 s segments). - Segments and manifests stored in object store. Multi-CDN distribution.
  3. Catalog / metadata. Video metadata in a sharded data store (Spanner or Cassandra). Search index in Elasticsearch.
  4. Playback path. Viewer requests watch page → metadata service returns title, owner, manifest URL. Player loads HLS/DASH manifest from CDN → fetches segments.
  5. CDN strategy. Multi-CDN with cache hierarchy: edge → regional shield → origin (object store). Pre-warm popular videos to edges; long-tail videos cold-fetch from origin on demand.
  6. Recommendation / discovery (out of scope to design here). Mention it sits alongside as a separate ranking service.
  7. View accounting. Player sends watch-time beacons → Kafka → counters aggregated in ClickHouse + Redis live counters.

End-to-end narration: a creator uploads a 1 GB video → multipart upload to S3 → transcoder fans out, produces 1080p/720p/480p/240p × H.264 and AV1 ladders (~5× source size total) → manifest published → metadata status=ready. A viewer in Tokyo opens the watch page → metadata fetched from regional cache → manifest fetched from Tokyo CDN edge → player negotiates 720p initially (~3 Mbps) → segments served from Tokyo edge → after first 10s the buffer is healthy, player bumps to 1080p.

6. Deep Dives (~10 min)

Concrete store contracts.

Store Key / data structure Entry size + TTL Shard key
Upload state upload:{upload_id} → part map, owner, expiry 1–10 KB, TTL 7 days incomplete upload_id
Source object s3://video-source/{video_id} immutable original MB–GB, retained durable/cold video_id
Segment object s3://video-segments/{video_id}/{codec}/{rendition}/{segment} 0.5–4 MB/segment, tiered video_id, rendition
Manifest manifest:{video_id}:{codec} HLS/DASH text 10–100 KB, immutable per version video_id
Metadata video:{video_id} row {owner,title,status,manifest_version} ~2 KB, retained video_id
Transcode jobs transcode:{video_id}:{rendition}:{segment} queue item ~200 B, until completed video_id
View counters views:{video_id}:{minute} + rollups 8–32 B/counter, TTL by granularity video_id

Capacity by store. Upload state is small even for millions of incomplete uploads: 1M × 10 KB is ~10 GB. Metadata is also small: if 500 hours/minute averages 10-minute videos, that is ~4.3M uploads/day; 4.3M × 2 KB × 3 is ~26 GB/day. Manifests at 100 KB each are hundreds of GB/day. Media dominates: ~720 TB/day of originals and ~5 PB/day of encoded ladder bytes before origin replication/tiering. View events are big but not media-big: 2B viewers × 60 min/day × one beacon/10s × 100 B is ~1.2 TB/day before replicas. The cost knob is bytes delivered and stored per watched minute, not rows.

Playback path trace. Viewer → watch API reads video:{id} and permission state → returns signed manifest URL → CDN edge serves manifest or fetches from regional shield → player requests first low/mid-bitrate segment → CDN edge/shield/origin lookup by immutable segment key → player adapts bitrate based on buffer. Metadata/profile/recommendation calls can run in parallel with manifest fetch; media segment serving should bypass app servers entirely.

Transcoding economics. Most expensive infrastructure. Process video in parallel segments (split into 6 s chunks, transcode independently, recombine). Two key levers: - GPU pipelines (NVENC) for fast common ladders. - Software encoders (libx264, libaom-av1) for higher-quality / new codecs at long latency. - Per-title encoding: tune bitrate ladder based on content complexity (animation vs sports) — saves 20–40% storage.

Codec strategy. Maintain multiple codec ladders: H.264 for compatibility, VP9/AV1 for modern devices. Storage trade — AV1 saves ~30% bytes at the cost of compute. Compute amortizes over months/years of egress savings.

Storage tiering. Recently-uploaded and trending: hot object storage. Long-tail (years old, rarely watched): warm/cold. Re-promote to hot on demand spike.

Why object storage over HDFS-style clusters: segments are immutable, massive, and CDN-origin friendly, so object storage gives durability and lifecycle tiering without managing storage nodes. HDFS can work for internal processing, but serving and retention want object semantics.

For metadata, Spanner/sharded SQL is attractive when privacy state, ownership, and publish status need strong transitions; Cassandra is also plausible for global read scale if I keep state transitions simple and event-driven. I would not put media bytes in either one because the media access pattern is immutable object reads through CDN.

CDN cache hierarchy. Edge nodes (close to user) → regional shields → origin. Hot videos sit at the edge; long-tail cold-fetches from shield then origin. Cache key includes segment URL (immutable, content-hashed).

Interview punch line: "The app server starts playback, but the CDN carries the product; every cache miss that reaches origin is a cost bug at YouTube scale."

Multi-CDN. YouTube runs its own CDN (Google's edge) but acknowledges: multi-CDN strategy for redundancy, with traffic-shifting based on health and cost.

Live streaming (touch lightly). Lower-latency pipeline: chunked CMAF segments (1–2 s), LL-HLS or DASH. Ingest via RTMP → live transcoder → CDN. Targets glass-to-glass < 5 s. Different infrastructure path than VOD.

Search. Inverted index on title/description/tags. Stream new metadata into ES on publish. Combine with engagement signals at rank time.

Watch counts. Don't increment per view in a relational DB. Beacon → Kafka → aggregator → Redis counter + ClickHouse rollup. Counts shown approximate live, eventually consistent.

Why Kafka + ClickHouse/OLAP instead of transactional counters: view events are append-only analytics, not user-facing ownership. Kafka absorbs bursty beacons; Redis gives live approximate counts; ClickHouse gives cheap rollups. A relational row per video would hot-spot on viral videos.

Two-index framing. Video has content-addressed media objects for playback, metadata rows for watch-page state, a search index for discovery, and analytics rollups for counters. Those are separate views over the same video_id; the media segment is immutable content, while metadata/search/counters are serving indexes that can lag or rebuild.

Recommendations (briefly). Two-tower model: video tower + user tower → embeddings → ANN search at serve time. Out of scope to design fully.

Resumable upload. Multipart uploads with chunk-level retries; long-lived uploads survive cell network drops. Server discards incomplete uploads after a TTL.

Failure modes. Transcoder dies mid-job: jobs are idempotent on segment; restart on a different worker. CDN edge unavailable: traffic shifts to next-nearest edge.
