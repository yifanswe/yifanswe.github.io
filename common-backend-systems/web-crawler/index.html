Web Crawler — Googlebot Class (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a polite, large-scale web crawler. Given a seed set, crawl reachable pages, extract content for indexing, and continuously refresh known pages. Target: crawl 1B unique URLs/day, respect robots.txt and per-host rate limits, store raw HTML + extracted text.

1. Requirements (~5 min)

Functional - Discover and fetch reachable URLs starting from a seed set. - Extract links, text content, and metadata; store for downstream indexing. - Re-crawl known pages on a refresh policy (popular pages frequently, long tail rarely).

I'd defer JavaScript rendering depth, image/video crawling, and PageRank-style ranking.

Non-functional - Politeness — respect robots.txt; throttle per host; identify ourselves. - Throughput: 1B URLs/day → ~12k fetches/sec. - Avoid duplicate work (URL dedup) and infinite loops (cycles, spider traps). - Resumable — if a worker dies, no URL is lost. - Eventual consistency — no real-time SLA.

Capacity - 100B known URLs in the frontier index. - Avg page = 50 KB → 50 TB/day of fetched content. - URL dedup index: 100B URLs × 32 B canonical key = ~3 TB. - Frontier metadata: 100B URLs × ~80 B (url_hash, host_hash, next_fetch_at, priority, status) ≈ 8 TB primary, ~24 TB with 3 replicas. This is large but manageable as a partitioned key-value/index store. - Raw content is the dominant cost: 50 TB/day fetched, maybe 15–25 TB/day after compression; 90-day hot retention is ~1.5–2.2 PB before replication, so tiering matters more than queue storage.

2. Core Entities (~2 min)

  • URL — url, normalized canonical form, host, scheme.
  • Frontier entry — url, next_fetch_at, priority.
  • FetchResult — url, status, body, headers, fetched_at, content_hash.
  • HostPolicy — robots.txt parsed, per-host crawl delay.

3. API / Interface (~5 min)

Mostly internal. Externals if any:

POST /v1/seeds            { urls[] }
GET  /v1/pages/{hash}     → fetched page
GET  /v1/stats            → crawl progress

The crawler's "API" is really the frontier-to-worker contract.

4. Data Flow (~5 min)

Seed URLs → frontier → fetcher → parser (extract links + content) → URL filter (dedup, policy check) → write new URLs to frontier; write content to storage. Loop.

5. High Level Design (~10–15 min)

  1. URL frontier. Priority queue of URLs to fetch, partitioned by host. Per-host queue ensures politeness: workers pull at most 1 URL per host per crawl delay. Implemented as Kafka with per-host partitioning + Redis sorted set per host for delay enforcement.
  2. DNS resolver service. Cached resolution; without this, DNS becomes the bottleneck.
  3. Fetcher workers. Pull a URL → consult host policy (robots.txt + crawl-delay) → HTTP GET → store result. For JS-heavy pages, send to a headless-browser pool (more expensive, used sparingly).
  4. Parser pipeline. From the body: extract links, canonicalize, normalize text (readability extraction), compute content hash, detect language.
  5. URL filter. - Seen-URL check: Bloom filter (per-shard) for O(1) dedup of already-known URLs. Persistent backing in a key-value store for evidence. - Canonicalization: strip fragments, normalize trailing slashes, sort query params (allowlisted set), resolve relative paths. - Robots / policy: skip disallowed paths, off-site links (depending on scope), known traps.
  6. Frontier writer. Inserts new URLs with priority + next_fetch_at.
  7. Content store. Raw HTML in object storage (compressed); extracted text in a downstream indexing pipeline (out of scope).
  8. Scheduler / coordinator. Tracks host queues, balances workers, applies global politeness limits.

End-to-end narration: seed URL nytimes.com/ enqueued → fetcher pulls (host queue empty, can fetch now) → GETs robots.txt first (cached), then page → parser extracts 200 links → URL filter dedups against bloom filter → 50 are new → frontier writer enqueues them with priority = page authority × freshness. Meanwhile, the host queue for nytimes.com has a 1 s crawl delay; the next URL from that host waits 1 s before being eligible.

6. Deep Dives (~10 min)

Concrete store contracts.

Store Key / data structure Entry size + TTL Shard key
Frontier index frontier:{host_hash}:{url_hash} → {url, next_fetch_at, priority} ~80–150 B + URL, retained while known host_hash
Ready queues Kafka partitions by host_hash; Redis zset ready:{partition} score=next_fetch_at pointer ~32 B, until fetched host_hash
Host policy host_policy:{host} robots + crawl delay 1–10 KB, TTL 24h host_hash
Seen filter Bloom filter per URL shard + KV evidence seen:{url_hash} 32 B hash evidence, retained years url_hash
Raw content s3://crawl/{date}/{url_hash} compressed HTML 10–50 KB avg, tiered 30/90/365d date,url_hash
Parsed metadata page:{url_hash} → status, content_hash, outlink count ~200 B, retained years url_hash

Capacity by store. The frontier metadata is 100B URLs × ~80 B × 3 or ~24 TB; storing full normalized URLs can push the primary footprint into the 15–25 TB range before replication depending on average URL length. The exact seen evidence is 100B × 32 B × 3 or ~9 TB, while a 10-bit/item bloom filter is only ~125 GB primary. Raw content dominates: 1B pages/day × 50 KB is 50 TB/day fetched, roughly 15–25 TB/day compressed, and 90 days at 3 replicas is several PB. Parsed metadata is moderate: 100B × 200 B × 3 is ~60 TB. Host policies can be large but bounded by active hosts and TTL, not by URL count.

Fetch path trace. Scheduler reads eligible URLs from ready:{partition} → worker leases one URL with a visibility timeout → reads host_policy:{host} and host:next_ok_at → if allowed, atomically advances next_ok_at → fetches HTTP → writes raw HTML to object storage and parsed metadata to KV → parser emits outlinks → URL filter canonicalizes and checks bloom/KV in parallel → new URLs are written to frontier and their host partitions. If the worker dies before ack, the lease expires and the URL is visible again.

Politeness — the cardinal rule. Per-host crawl delay (default 1 s, or from robots.txt). Implementation: Redis key host:nytimes.com:next_ok_at. Worker checks this before fetching; sets it to now + delay on dispatch. If conflicting (multiple workers contending for the same host), the loser requeues.

Interview punch line: "The crawler is fast globally by being slow locally; politeness is a per-host invariant, not a fleet-wide average."

Robots.txt. Fetched once per host per day. Parsed and cached. Worker consults before each fetch. Honor Crawl-delay, User-agent, Disallow, Sitemap directives.

URL canonicalization. Two URLs that point to the same page should hash to the same key. Rules: - Lowercase scheme + host. - Resolve .. and . in path. - Strip default ports. - Sort query params; drop session-id-like params from an allowlist. - Strip fragment. Mention <link rel="canonical"> as the server-side hint we'd also honor.

Seen-URL dedup at scale. 100B URLs is too many for an in-memory hash set. A scalable bloom filter (per partition) gives O(1) checks with bounded false-positive rate (~0.1%). Persistent backing in a key-value store for hard checks when bloom says "maybe seen."

Why Bloom + KV instead of only a database uniqueness check: the bloom filter rejects most repeats before they hit storage, while the KV gives evidence and repairability. A pure exact index is simpler but costs too much random I/O at 12k fetches/sec plus outlink expansion; a pure bloom filter is cheap but silently loses URLs on false positives.

Frontier prioritization. Not all URLs equal. Priority = f(page_rank_estimate, freshness_need, newness). Newly discovered URLs from popular hubs get high priority. Re-crawl frequency: tier-1 (nytimes) every hour, tier-2 daily, long tail weekly+.

Spider traps. Infinite calendars, generated URLs with continuously incrementing query params. Defenses: - Cap depth per host (don't crawl > 1000 levels deep). - Cap pages per host per day. - Detect content sameness — if content_hash repeats for many URLs from a host, blacklist a pattern.

Distributed coordination. Hash URLs by host → assign to one of N partitions. Each partition has its own frontier + workers, so per-host politeness lives in one place. Cross-partition: only the URL discovery emits to other partitions (a parsed page from host A may have outlinks to host B).

Kafka is good for durable handoff and replay, but it is not a priority queue by itself. Redis zsets or a frontier DB hold next_fetch_at; Kafka carries ready work and parser outputs. Be ready for "why not just Kafka?" — because delayed priority scheduling and per-host politeness need indexed time, not only append order.

Two-index framing. URLs need two indexes over the same canonical URL set: host-keyed queues for politeness (frontier:{host_hash}:{url_hash}) and URL-hash evidence for dedupe (seen:{url_hash}). Host-keyed state answers "when may I crawl this host next?" URL-keyed state answers "have I already discovered this page?" Keeping that duality explicit prevents a design that optimizes throughput but violates robots/politeness.

JavaScript rendering. Modern web is JS-heavy. Selective rendering: only render when the static page yields no useful content. Headless Chrome workers, expensive (~2 s / page vs 50 ms for static HTML).

Recrawl scheduling. Each page has next_recrawl_at. Tier by hub-vs-leaf, page change rate, importance. Important pages (high in-link count or news sites) recrawl daily; long-tail recrawl monthly.

Failure handling. Fetcher retries with backoff on 5xx; respects Retry-After. On persistent 4xx, mark URL dead. On timeouts, demote priority.

What I'd defer. Image/video crawling, structured data (JSON-LD) extraction, anti-fingerprinting evasion (we identify ourselves), specialized vertical crawlers (news, shopping), distributed PageRank.
