1. Basic Knowledge And Solution

What This Question Is Testing

Google Search is the canonical web-scale information retrieval problem. The hard part is indexing the entire public web, answering arbitrary queries in under a second, ranking results by relevance and authority, and continuously updating to reflect a changing web.

The invariant is this:

Every indexed page is discoverable via the appropriate query; query results are ranked deterministically by a stable scoring function; the index reflects the corpus within a bounded staleness; query latency is under a few hundred milliseconds at p99; the system survives partial index outages by returning best-effort results.

At ICT5 / L6+ level, the interviewer is testing whether you can design the indexing pipeline (crawl → parse → index), the inverted index sharded for query serving, the ranking pipeline with PageRank-style signals, query planning and result merging, and reason about freshness vs cost tradeoffs.

60-Second Interview Framing

Spoken version:

I would design Google Search as a pipeline from crawler to index to query serving. The crawler fetches the web; the indexer builds an inverted index sharded by term; query servers scatter-gather across shards, rank, and return. Ranking combines static signals (PageRank, content quality) with query-time signals (relevance, freshness, personalization). The interesting decisions are index sharding strategy, scatter-gather query latency, ranking architecture, freshness pipeline, and serving capacity.

Requirements To Clarify First

Functional requirements:

  • User submits a query, gets ranked list of relevant web pages.
  • Support boolean operators, phrases, filters (site:, intext:).
  • Personalization based on user history, location.
  • Auto-suggest as the user types.
  • Knowledge panels and direct answers (snippets).
  • Image and video search (mention; separate indexes).

Non-functional requirements:

  • Query p99 under 500 ms.
  • Index covers tens of billions of pages.
  • Freshness: minutes for news, hours/days for static pages.
  • High availability of query serving.
  • Multi-region; queries served from local region.

Out of scope unless asked:

  • Crawling specifics (see Web Crawler design).
  • Ad serving and auctions.
  • Spam detection details.

Inverted Index Shard

A shard owns posting lists for a subset of terms (sharded by term hash) — or owns a subset of documents (sharded by doc_id). The latter is common at Google's scale.

APIs

GET /search?q=
GET /complete?q=                 # autosuggest
GET /image-search?q=             # separate index
GET /news-search?q=              # freshness-focused

Indexing Pipeline

  1. Crawler stores raw HTML in page store.
  2. Parser extracts text, links, structured data.
  3. Tokenizer normalizes (lowercase, stemming, language-specific).
  4. Posting list builder: for each (term, doc_id), append entry with frequency and positions.
  5. Index builder writes inverted index in sorted segments per shard.
  6. Periodic merge (LSM-style) keeps index efficient.

Static signals computed offline:

  • PageRank from link graph.
  • Content quality scores.
  • Spam / penalty scores.

Each document has a static_rank precomputed; used as a query-independent prior.

Freshness pipeline:

  • News and high-priority pages indexed in minutes.
  • A separate, smaller "live index" serves recent content.
  • Query merges results from main index + live index.

Interview phrase:

The index is two-tier: a giant batch-built main index covering the bulk of the web, plus a small live index that captures the freshest content. Queries merge both.

Index Sharding

Two strategies:

Document-based sharding:

  • Each shard owns a partition of documents.
  • For a query, all shards process the query against their docs and return top-K.
  • Aggregator merges.
  • Pros: easy to add/remove shards; per-shard work is bounded.
  • Cons: every query hits every shard.

Term-based sharding:

  • Each shard owns posting lists for a partition of terms.
  • For a query, only shards holding query terms are contacted.
  • Pros: smaller fan-out for narrow queries.
  • Cons: hot terms become hot shards; cross-shard intersection is expensive.

In practice, document-based is common at Google scale because every query hits roughly equal load. Term-based is used in some specialized cases.

Query Serving

For a query like "best italian restaurants in NYC":

  1. Frontend receives query.
  2. Query rewriter expands: synonyms, spell correction, query understanding.
  3. Scatter to all index shards (or relevant subset).
  4. Each shard returns top-N candidate doc_ids with scoring data.
  5. Aggregator merges candidates, dedupes.
  6. Ranker pass: combine static rank + query-time features (BM25, semantic similarity, freshness, personalization) → final score.
  7. Top-K results: fetch titles, snippets from metadata store.
  8. Snippet generator creates contextual summaries.
  9. Knowledge graph: check if query has entity answer; add knowledge panel if so.
  10. Return result page.

All within ~500 ms.

Latency budget:

  • Network round trip ~10 ms.
  • Index shard query ~100-200 ms.
  • Aggregation ~50 ms.
  • Ranking pass ~50-100 ms.
  • Snippet + metadata fetch ~50 ms.
  • Total ~300-400 ms.

Scatter-Gather Tail Latency

With thousands of shards, the slowest shard determines query latency.

Mitigations:

  • Hedged requests: send to two replicas of the same shard, take first.
  • Tail-tolerance: respond after, say, 99% of shards return, drop slowest 1%.
  • Per-shard timeouts.
  • Health-aware routing.

Interview phrase:

Tail latency is the dominant cost in scatter-gather. Hedged requests and partial-response tolerance keep p99 reasonable.

Ranking

Multi-stage:

  1. Lightweight per-shard scoring: Each shard scores its candidates with cheap features (BM25, static rank). Returns top-K per shard.
  2. Aggregated scoring: Aggregator combines, applies more features (entity match, freshness).
  3. ML ranker: Final pass over top results with a deep model (BERT-style or transformer-based) for the highest fidelity ranking.

Signals:

  • PageRank, authority, content quality.
  • Query-document relevance (BM25, semantic similarity).
  • Freshness for time-sensitive queries.
  • Personalization (location, prior clicks).
  • Click-through rate priors.
  • Mobile-friendliness, page speed.

Knowledge Graph

For entity queries ("Barack Obama height"), the system retrieves from a structured knowledge graph rather than just text matching.

  • Entity recognition in query.
  • Knowledge graph lookup.
  • Rendered as a knowledge panel + direct answer snippet.

Auto-Suggest

  • Trie of popular queries with frequency.
  • Bucketed by prefix.
  • Personalized re-rank.
  • Sub-50ms latency.

Consistency Model

Layer Consistency Reason
Inverted index Eventually consistent (hourly batch + live tier) Web is not real-time.
Document metadata Eventually consistent UI hint.
Knowledge graph Strong on entity facts Direct answers must be correct.
Ranking Stable function over current index Determinism.

Multi-Region Strategy

  • Index replicated to each region for fast local serving.
  • Query routed to nearest region.
  • Updates flow from primary indexing to all regions.
  • Freshness lag: minutes between primary and replica.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Slow shard. Hedged requests; partial response. Per-shard tail latency.
Failed shard. Replica fallback; degrade with missing coverage. Shard health.
Index serving overload. Shed; rate limit; prioritize logged-in users. QPS vs capacity.
Bad rank model. Canary rollout; auto-rollback. Click-through rate.
Spam injection. Spam detection layer; manual review. Anomalous URL appearance.
Index lag for breaking news. Live index tier; pre-prioritized crawl. Time-from-publish-to-indexed.
Personalization bug exposes private data. Opt-in personalization; clear separation. Privacy audit.
Knowledge graph fact wrong. Manual override; fact provenance. User report rate.
Cross-region replication lag. Async; bounded; alert. Replication lag.

Operational Metrics

Correctness:

  • Result quality (offline evaluation, click signals).
  • Spam in results.
  • Knowledge graph accuracy.

Capacity:

  • Query QPS by region.
  • Index shard QPS and latency.
  • Tail latency p99/p999.

Business:

  • Click-through rate.
  • Top-N click distribution.
  • Query reformulation rate.

Tradeoffs To Call Out

  • Larger index covers more pages but adds query latency.
  • Frequent re-indexing improves freshness but costs compute.
  • More personalization improves relevance but raises privacy concerns.
  • Deeper ranking models improve quality but cost compute.
  • Hedged requests reduce tail latency but double cost.
  • Two-tier index (main + live) gives freshness but increases complexity.

Common Traps And Stronger Moves

Trap Stronger move
One big monolithic index. Sharded inverted index; document-based partitioning.
Per-query full ranking. Multi-stage ranking; cheap then expensive.
Ignore tail latency. Hedged requests; partial-response tolerance.
Fresh + main as one index. Two-tier: small live index + large main.
Synchronous ranking model. Lightweight pass before ML model.
Forget knowledge graph. Entity recognition + structured lookup.

  • Pipeline: crawl → parse → index → serve.
  • Document-based shard partitioning.
  • Scatter-gather with tail-latency mitigation.
  • Multi-stage ranking.
  • Two-tier index for freshness.
  • Knowledge graph for entity answers.
  • Auto-suggest as a separate fast path.
  • Multi-region replicas.

Interview Phrase

Google Search is a scatter-gather query over an inverted index, layered with multi-stage ranking. The index is sharded by document with replicas for tail-latency mitigation; ranking goes from cheap per-shard scoring to expensive ML at the top of the result set. Freshness comes from a live tier merged with the main index. The hardest performance problem is tail latency in scatter-gather; the hardest correctness problem is ranking quality.

2. Simulated Interview

Interviewer: Design Google Search.

Candidate: Pipeline: crawler fetches the web, parser tokenizes, indexer builds a sharded inverted index, query servers scatter-gather, ranker scores, results returned. The invariant is sub-second query latency over an index covering tens of billions of pages, with relevance ranking and bounded freshness.

Interviewer: How is the index sharded?

Candidate: Document-based. Each shard owns a partition of documents. For a query, all shards process the query against their docs and return top-K candidates. Aggregator merges. The alternative is term-based — each shard owns posting lists for some terms — but at Google's scale, document-based is more common because every query hits roughly equal load.

Interviewer: Walk me through a query.

Candidate: User types "best italian restaurants nyc". Frontend rewrites: spelling, synonyms, intent detection. Scatter to thousands of index shards. Each shard scans its posting lists for "italian," "restaurants," "nyc," intersects, computes a lightweight score (BM25 + static rank), returns top-K (say 100) candidates. Aggregator merges across shards into top-1000. Multi-stage ranking: a mid-tier model scores top-1000 with more features; final ML model scores top-50 with heavy features (BERT, personalization, freshness). Top-10 are returned with snippets. Knowledge graph adds a knowledge panel if entity detected.

Interviewer: Tail latency — thousands of shards, one slow shard kills the query.

Candidate: Hedged requests: each shard has multiple replicas; send to two simultaneously, take first to respond. Partial-response tolerance: aggregator returns once 99% of shards respond, drops slow 1% with bounded quality loss. Per-shard timeouts. Health-aware routing demotes slow replicas. Capacity headroom around 60-70% utilization so a slow node doesn't cascade.

Interviewer: Ranking?

Candidate: Multi-stage. Cheap per-shard scoring with BM25 and static signals (PageRank, authority) returns top-K candidates per shard. Aggregator merges and applies more features (entity match, freshness). Top results pass through a deep ML model (transformer-based) for the highest-fidelity ranking. The two-stage structure controls compute: cheap features prune to a manageable set; expensive features evaluate only the top.

Interviewer: PageRank?

Candidate: Static, query-independent rank based on the link graph. Pages with many high-authority inbound links get higher PageRank. Computed offline in periodic batch jobs over the web link graph. Stored per document as a feature, used in ranking. Modern Google has many more signals beyond PageRank, but the same principle: pre-computed authority.

Interviewer: Freshness — a breaking news story.

Candidate: Two-tier index. The main index is built in batches (hourly or daily); covers the bulk of the web. A live index is much smaller, fed by priority crawl of news sites and recent updates, indexed in minutes. Queries merge results from both. The ranker boosts fresh results for time-sensitive queries. Editorial signals can promote specific stories.

Interviewer: Knowledge graph?

Candidate: Separate structured store of entities, attributes, relations. Entity recognition in the query (NER) detects e.g. "Barack Obama." Lookup in the knowledge graph retrieves attributes. The result page shows a knowledge panel with structured info, plus a direct-answer snippet for simple factual queries. Fact provenance — what source supports each fact — is critical to handle errors.

Interviewer: Auto-suggest?

Candidate: Trie of popular queries with frequency, sharded by prefix. As user types, server returns top completions. Personalization layer re-ranks based on user history and location. Sub-50ms latency. Cached client-side for common prefixes.

Interviewer: Multi-region?

Candidate: Index replicated to each region. Queries routed to nearest region. Index updates flow from primary indexing pipeline to all regions; replication lag is minutes. Knowledge graph replicated globally. Per-region capacity sized for local QPS.

Interviewer: Spam?

Candidate: Spam detection runs at parse time and indexing time. Signals: cloaking, doorway pages, link spam, content quality. Penalty scores reduce the page's rank. Outright spam is excluded from index. Manual review for edge cases; webmaster tools allow appeals.

Interviewer: Personalization?

Candidate: Opt-in. User's recent searches, location, language preferences influence ranking. Personalization features computed online from user profile, applied at the final ranking stage. Bounded — not so heavy that ranking becomes unpredictable. Privacy considerations: explicit user controls; anonymized signals.

Interviewer: Summarize.

Candidate: Pipeline from crawler to inverted index to query serving. Document-based shard partitioning. Scatter-gather with hedged requests and partial-response tolerance for tail latency. Multi-stage ranking from cheap per-shard scoring to expensive ML at the top. Two-tier index for freshness. Knowledge graph for entity answers. Auto-suggest separate path. Multi-region replicas. The hardest performance problem is tail latency; the hardest correctness problem is ranking quality and spam handling.

3. Practice Notes

Five-Minute Version

  1. Clarify: query latency, freshness, scale.
  2. Invariant: sub-second p99 over tens of billions of pages.
  3. Pipeline: crawl → parse → index → serve.
  4. Document-based shard partitioning.
  5. Scatter-gather with hedged requests.
  6. Multi-stage ranking (cheap → expensive).
  7. Two-tier index for freshness.
  8. Knowledge graph for entity answers.

One-Minute Closeout

Google Search is scatter-gather over a document-sharded inverted index with multi-stage ranking. Cheap per-shard scoring prunes candidates; expensive ML ranks the top. Hedged requests and partial-response tolerance keep tail latency tame. A two-tier index merges live and main for freshness. Knowledge graph powers entity answers. Auto-suggest is a separate fast path. The dominant performance problem is scatter-gather tail latency; the dominant correctness problem is ranking quality.
