1. Basic Knowledge And Solution

What This Question Is Testing

Calendar manages events across users, with invitations, RRULE recurrences, attendee responses, free/busy lookups, and notifications. The hard part is timezone correctness, recurring event handling (single instance edits within a series), conflict-free updates across attendees, and free/busy queries at scale.

The invariant is this:

Every event is durably stored, visible to its organizer and invited attendees per their RSVP state; recurring events expand deterministically into individual occurrences accounting for timezone and DST; modifications to a single occurrence override the recurrence rule only for that instance; notifications fire at scheduled local time on every user's devices; free/busy queries reflect current event state.

At ICT5 / L6+ level, the interviewer is testing whether you can model recurring events with overrides, handle invitations and attendee responses, do free/busy at scale, integrate with reminders, and reason about cross-calendar sharing.

60-Second Interview Framing

Spoken version:

I would design Google Calendar as an event store with RRULE recurrence, attendee management with RSVPs, free/busy queries, and a reminder pipeline. Events are stored as a master template plus per-instance overrides for exceptions. Free/busy is computed by occurrence materialization. The interesting decisions are recurrence storage, timezone handling, attendee invitation flow, and integration with the reminder system.

Requirements To Clarify First

Functional requirements:

  • Create, edit, delete events.
  • Recurring events (RRULE).
  • Edit single occurrence vs entire series.
  • Invite attendees; RSVP (yes, no, maybe).
  • Find free/busy for a list of users.
  • Multiple calendars per user (work, personal, holidays).
  • Sharing calendars with permissions.
  • Reminders before events.
  • Sync with mobile clients (CalDAV-like).

Non-functional requirements:

  • Sub-second event read.
  • Reminder timing accurate to seconds.
  • Multi-region support for global teams.
  • Strong consistency on event edits.
  • Bidirectional iCal sync.

Out of scope unless asked:

  • Meeting room booking (Resource Calendar).
  • Smart scheduling AI.
  • Email integration beyond invites.

Recurrence Model

Each recurring event has a single Event row with RRULE. Occurrences are not pre-materialized; they're computed on the fly for the requested view range.

For exceptions:

  • A single occurrence modified: create an EventOverride row pointing to the series, specifying the modified data.
  • A single occurrence cancelled: EventOverride with is_cancelled=true.
  • Edit "this and future": split the series — set RRULE END on the original, create a new Event series starting from the cutover.
  • Edit "all": modify the master event (and possibly drop overrides for changed fields).

Reading a date range:

  1. Fetch all Event rows where the series overlaps the range.
  2. Expand RRULE to get occurrences within range.
  3. Apply overrides: substitute or remove specific occurrences.
  4. Return materialized occurrence list.

Interview phrase:

Recurring events store the rule, not the instances. Overrides for exceptions point back to the series. Expand at read time within the requested range.

Timezone Handling

Master event stores start_time and timezone (e.g., "9 AM America/Los_Angeles"). Recurrence rule operates in the event's timezone.

  • When user crosses timezones, displayed times shift but the event's local time stays.
  • DST transitions: events at 2:30 AM on a fall-back day get the first occurrence (or per policy).
  • All-day events have no timezone; they span "the day" in the user's local view.

Free/busy: events are converted to UTC for cross-user queries.

Invitations

Organizer adds attendees to an event:

  1. Each attendee added with initial response=needs_action.
  2. Notification (email + in-app) sent.
  3. Attendee responds: yes, no, maybe.
  4. Event's attendee list updated; organizer notified.
  5. The event appears in the attendee's calendar with their RSVP status.

For cross-organization invites (different Google Workspace orgs): iCal-based email invites for interop.

Each attendee technically has their own view of the event (their RSVP). The "source of truth" event is in the organizer's calendar; attendees see a copy linked by event_id. Edits propagate via invitations.

Free/Busy Service

For "show me when these 5 users are free this week":

  1. For each user, fetch events overlapping the time range.
  2. Expand recurrences.
  3. Apply privacy: free/busy doesn't reveal event titles (unless explicitly shared).
  4. Convert each event to a UTC busy window.
  5. Return the union per user.

Optimization:

  • Materialize per-user free/busy intervals for the next 30 days.
  • Update on event create/edit.
  • Query reads the materialized index.

For scheduling assistant ("find a time that works for all 5"): intersect the inverse (free windows).

Reminder Integration

Each event has reminders configured (e.g., 10 minutes before).

  • When event is created/edited, compute the reminder times.
  • Submit to the reminder pipeline (see reminder system design).
  • Reminder pipeline materializes per-occurrence reminders, accounting for recurrence and timezone.
  • On fire, notification system sends push/email/in-app alert.

Sharing And ACL

Each calendar has an ACL: users/groups with roles (free/busy only, read, write, manage).

  • Sharing a calendar grants read or write access.
  • Edits by writers are visible to all viewers.
  • ACL checked on every operation.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Override conflicts with edited series. Conflict resolution: prefer override or master per policy. Override conflict count.
Timezone mismatch on travel. Display in user's current TZ; events stay in their TZ. TZ-display issues.
DST ambiguity. Explicit policy at materialization. DST-ambiguity events.
Free/busy lag. Re-materialize on event change; alert on stale. Free/busy freshness.
Recurring event with many occurrences. Cap expansion range; lazy materialize. Expansion size.
Invitation flood spam. Rate limit; sender verification. Invite spam reports.
Reminder miss. Reminder system handles via expected-runs. Pending past-due reminders.
Cross-org sync issue. iCal fallback; reconciliation. Cross-org sync errors.
Calendar share revoked while user views. ACL revalidation on read; cache short TTL. Revoke propagation.
Master event delete with overrides. Cascade-delete overrides. Orphan override count.

Tradeoffs To Call Out

  • Storing recurrence as rule vs materialized: rule is compact, materialized is fast for arbitrary queries.
  • Per-occurrence reminders use storage but enable single-occurrence edits.
  • Free/busy materialization speeds reads but adds write work.
  • Calendar sharing per user vs per group: per-group simplifies but adds resolution lookups.
  • Strong consistency on invitations vs propagation latency.

Common Traps And Stronger Moves

Trap Stronger move
Materialize all recurrences. Store RRULE; expand at read with cap.
Edit recurrence in place destroys overrides. Split-series approach for "this and future."
Single global timezone. Store TZ per event; materialize UTC at occurrence.
Free/busy from raw event scan. Materialize per-user intervals.
Invite all attendees synchronously. Async notification queue.
ACL check at write only. Strong ACL revalidation.

2. Simulated Interview

Interviewer: Design Google Calendar.

Candidate: I will design four core systems: event store with RRULE recurrence, override store for exceptions, free/busy service, and reminder integration. The invariant is durable events visible to attendees with correct RSVP state, recurrence expanded deterministically with timezone correctness, and reminders firing on time.

Interviewer: How do you store recurring events?

Candidate: Single Event row with the master fields plus an RRULE (RFC 5545 recurrence rule). I don't pre-materialize occurrences in storage. When reading a date range, I expand the rule on-demand to find occurrences. For exceptions — a single occurrence moved or cancelled — I store an EventOverride linked to the series; the override specifies the modified data or is_cancelled=true.

Interviewer: What about "edit this and future"?

Candidate: Split-series. Set an END on the original event's RRULE at the cutover date; create a new Event series starting from the cutover with the new properties. Past occurrences remain in the original series; future ones are in the new series. This preserves history and lets each series have its own RRULE.

Interviewer: Timezone handling?

Candidate: Each event stores timezone and a local-time start_time. The recurrence operates in that timezone. When materializing occurrences for free/busy or display, convert to UTC for the user's current viewing TZ. DST transitions are handled at materialization — "every day at 2:30 AM" on a fall-back day picks the first occurrence per policy.

Interviewer: Invitations?

Candidate: Organizer's event lists attendees with initial response=needs_action. Invitation service sends notification (email + in-app) to each attendee. Attendees respond; their RSVP is updated on the event's attendee list. Each attendee sees the event in their calendar with their RSVP. Cross-organization invites use iCal email for interop.

Interviewer: Free/busy — show me when 5 users are busy this week.

Candidate: Materialized per-user free/busy index. For each user, store the busy intervals for the next 30 days, derived from their events (with recurrence expansion). Free/busy query reads the materialized indices. For find-a-meeting-time: invert each user's busy intervals to free, intersect across users. Privacy: free/busy reveals only times, not titles.

Interviewer: Update the free/busy when an event is created.

Candidate: On event create/edit, compute the new busy intervals (expand recurrence for the next 30 days) and update the user's materialized index. Eventually consistent — a fresh event may take seconds to reflect in others' free/busy view. Strong consistency for the event itself; the index is a derived projection.

Interviewer: Reminders?

Candidate: Integrate with the reminder system. When event is created with reminders=[{10 minutes before, push}], compute the reminder times per occurrence and submit to the reminder pipeline. Pipeline materializes per-occurrence reminders, accounts for timezone, fires via notification system at fire time. See reminder system design for details.

Interviewer: A user has a 5-year recurring meeting. How do you handle reminders?

Candidate: Don't materialize all reminders for 5 years. Materialize a rolling horizon — say, 30 days ahead. A nightly horizon-extension job materializes the next day's reminders. This bounds storage and lets edits to the series propagate cleanly to future reminders.

Interviewer: Sharing a calendar.

Candidate: Per-calendar ACL: users/groups with roles (free/busy only, read, write, manage). On every event op, check ACL. If a calendar is shared read-only, the user sees events but can't edit. Sharing a calendar lets the user see it merged into their views.

Interviewer: A meeting attendee changes the time of just their copy.

Candidate: Generally not allowed — attendees can RSVP but not modify the event. The organizer's event is the source of truth; attendees see a copy. If an attendee declines, the event is hidden from their calendar but remains in the organizer's. Some products allow attendee-side notes (visible to self only), but the event time is the organizer's.

Interviewer: Multi-region?

Candidate: Per-user home region. Events and overrides written there. Replicated globally for cross-org viewing and free/busy lookups by non-home users. Reminders fire in user's home region.

Interviewer: A single-occurrence cancel of a recurring event.

Candidate: Create EventOverride with is_cancelled=true pointing to the series and the specific occurrence date. When expanding the recurrence at read, this occurrence is skipped. Reminders for this occurrence are also cancelled (the reminder pipeline picks up the change).

Interviewer: Summarize.

Candidate: Events with RRULE + per-occurrence overrides. Each event has a timezone; UTC computed at materialization. Recurrence expanded at read, not stored materialized. Split-series for "this and future" edits. Attendee RSVPs are per (event, user). Free/busy materialized per-user. Reminders integrated with the reminder pipeline. Calendar sharing via ACL. The hardest correctness problem is single-occurrence overrides in a recurring series.
