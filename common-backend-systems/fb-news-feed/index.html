Facebook News Feed (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build the Facebook (FB) News Feed. A user opens the app and sees a ranked, personalized feed of posts from friends and pages they follow. 2B daily active users (DAU), average user follows ~300 entities, sessions are short and frequent.

1. Requirements (~5 min)

Functional - Post text/photo/video updates (write side). - Open the app and see a ranked feed (read side — the focus). - Like/comment/share on a post; basic engagement signals back to ranking.

I'd defer chat, Marketplace, Stories, and groups.

Non-functional - Feed open p99 < 500 ms. - Read-heavy: ~100:1 reads to writes. - Ranking should incorporate fresh signals — a post your best friend just made should rise quickly. - Eventual consistency is fine — a like takes a second to count, no one notices. - Survive celebrity asymmetry — a poster with 50M followers vs. a poster with 50.

Capacity - 2B DAU × 10 feed opens = 20B feed loads/day → ~230k QPS average, ~1M peak. - Average user posts 0.5/day → 1B posts/day → ~12k writes/sec average.

2. Core Entities (~2 min)

  • User — user_id, follows.
  • Post — post_id, author_id, content, media, created_at, signals (likes, comments).
  • Edge (follow) — follower_id, followee_id.
  • FeedEntry — user_id, post_id, score, inserted_at.

3. API (~5 min)

POST  /v1/posts                       { content, media }     → { post_id }
GET   /v1/feed?cursor=...&limit=20    → { posts[], next_cursor }
POST  /v1/posts/{id}/like
POST  /v1/posts/{id}/comments         { text }

User from token; pagination by cursor (server-issued opaque cursor, not offset).

5. High Level Design (~10–15 min)

The central question: fan-out on write vs. fan-out on read. I'd default to hybrid.

  1. Post service. Write the post to a primary store (sharded by post_id). Publish a post.created event.
  2. Fan-out service. For most authors (≤10k followers), push the post_id into each follower's feed list (Redis sorted set, score = ranking time). For celebrities, skip fan-out — leave their posts in a global "celebrity recent posts" cache.
  3. Feed assembly service. On feed open: - Pull the user's precomputed feed list from Redis (sorted set keyed by user_id) — this is ~300 candidate post_ids only, not post bodies. - Merge in post_ids from celebrity authors this user follows, read at request time from per-author posts_by:{author_id} Redis lists. - Hydrate post bodies via batched MGET post:{post_id} against the post-body cache; cache misses fall through to the source-of-truth store. - Rerank with the live ranking model. - Return top 20 with cursor.
  4. Ranking service. Online inference. Features include freshness, author affinity, engagement velocity, post type. Model server (e.g., TensorFlow Serving) with a feature store fetch in parallel with feed pull.
  5. Engagement pipeline. Likes/comments → Kafka → counter aggregation (Redis + ClickHouse for analytics) + ranking signal updates.

Concrete stores in the hot path:

Store Key pattern Value / data structure Entry size TTL / retention Shard key
Reader feed index feed:{user_id} Redis ZSET of post_id + score ~64 B/entry 7 days / cap 200-300 user_id
Author recent index posts_by:{author_id} Redis LIST of recent post_ids ~8-16 B/entry 7 days / cap 20-100 author_id
Post body cache post:{post_id} content pointer + metadata ~1-5 KB 1-7 days LRU post_id
Source of truth posts.post_id durable post row/media refs ~1-5 KB forever post_id
Dedupe set recently_shown:{user_id} Redis SET / compact ids ~8 B/id + overhead 24 h user_id
Active users active_users:{day} Bloom filter / set ~10 bits/user 7 days user hash

Feed read trace: ZREVRANGE feed:{user} and feature-store reads run in parallel. Celebrity LRANGE posts_by:{author} calls are pipelined by Redis shard. Hydration is a batched MGET post:{id...}; only misses hit TAO/MySQL. Ranking runs after hydration because the model needs content type and fresh counters. The cursor stores {mode, score_watermark, last_seen_ts, seen_set_id}.

End-to-end narration: Alice posts. Fan-out service pushes the post_id into the Redis feed-lists of her 200 friends, and also LPUSHes it onto posts_by:alice (the per-author index, used for pull-mode reads). Bob opens FB → assembly fetches feed:bob (300 candidate post_ids), reads posts_by:{celeb_id} for the celebrities Bob follows (5 more post_ids), batch-hydrates all 305 post bodies from the post cache, sends them to the ranker, returns the top 20.

Two indexes, one dataset. Push and pull aren't different stores — they're the same posts indexed two ways. feed:{reader_id} pre-joins on the reader side (push). posts_by:{author_id} pre-joins on the writer side (pull). Hybrid keeps both indexes and chooses which to query based on whether the reader's feed list is hot.

Write-side capacity: 1B posts/day is ~12k post.created events/sec average; at a 200 B event payload, Kafka is only ~2.4 MB/sec raw, ~7 MB/sec with RF=3, so the event log is not the cost center. Fan-out writes are. If a normal author fans out to 200 active followers, the average is ~2.4M Redis ZADDs/sec; at peak this can be 5-10x, so workers batch by target Redis shard and pipeline writes. Each fan-out task carries {author_id, post_id, follower_page_token} so retries resume from a follower-page boundary and do not duplicate feed:{user_id} entries.

Fan-out mode state:

Stage Mode Trigger Index written Read behavior
normal author push follower count <= ~10k feed:{reader_id} + posts_by:{author_id} fast Redis feed read
large author active-only push 10k-1M followers active followers only + author index inactive readers pull later
celebrity pull-only >1M followers or burst detected posts_by:{author_id} only merge at read time
dormant reader rebuild feed:{reader_id} missing async rebuild after pull-mode read first open slower, next open hot
exhausted reader pull-back/cold cursor past feed cap no new feed write required cursor moves to author/cold indexes

6. Deep Dives (~10 min)

Why hybrid push/pull. Pure push falls over on celebrities (one post → 50M Redis writes). Pure pull doesn't scale either — on every feed open, do a query like "give me top 300 recent posts from the 300 people I follow" which is hot-key heaven. Hybrid: push for normals, pull-on-read for celebrities. The cutoff is a tuning knob (~10k followers).

Be ready for "why not pure precompute?" The honest answer is asymmetry: the median author makes push cheap, but the largest author turns one write into tens of millions of writes. Hybrid keeps the cheap case fast and makes the pathological case a read-time merge.

The Redis feed list. Per-user sorted set keyed by user_id, capped at ~300 entries. Each entry ≈ 64 bytes (8B post_id + 8B score + skiplist overhead). Per user ~20 KB. Reads are O(log N) range reads via ZREVRANGE. Time-to-live (TTL) on individual entries (~7 days) so the set self-trims.

Capacity — this is the dominant cost. 3B monthly active users (MAU) × 20 KB ≈ 60 TB of feed lists before replication. Replicated ×3 across regions → ~1 PB of Redis RAM. The post-body cache is a rounding error by comparison (~2 TB for the hot window), because most feed reads serve posts <24h old and post bodies are small.

The author index is cheaper: 1B active authors × 50 ids × 8 B is ~400 GB raw, a few TB with Redis overhead/replication. recently_shown can become sneaky: 2B daily active users × 200 ids × ~16 B physical is several TB, so keep TTL short and cap hard. Dominant cost remains feed:{user_id} RAM and fan-out writes.

Knobs to cut footprint: (a) maintain an active set (Redis set or bloom filter of users active in the last ~7 days) and only pre-fan-out to members — drops live working set to ~30 TB; (b) per-entry TTL ~7 days so dormant lists evict; (c) compact encoding (int64 post_ids, not strings); (d) smaller cap (200) if the pull-mode fallback is cheap enough.

When a dormant user opens the app, their list is gone — assembly falls back to pull mode (see below) and an async job rebuilds feed:{user_id} + adds the user back to the active set. Their next open hits the fast push path.

Ranking at sub-300 ms. Pre-fetch candidate features into a feature store keyed by user_id and post_id. Two parallel calls from assembly: feed list fetch + feature fetch. Model inference is a single batch call. Cache the user's feed result for 5–10 seconds so refresh-spamming is cheap.

Ranking signal freshness. Engagement events stream into a "recent signals" online feature store (Redis or a key-value cache) with second-level latency. Velocity (likes per minute since post) is the strongest freshness signal.

Post storage. The workload is pure point-lookup by post_id at high write throughput with uniform sharding — any horizontally sharded store works. The canonical FB answer is sharded MySQL behind TAO (FB's graph-aware caching layer): strong per-shard consistency, well-understood ops, and the actual production architecture. Cassandra, HBase, Bigtable, or DynamoDB also fit — reach for them if you want multi-DC active-active writes and can tolerate eventual consistency.

A write-through post-body cache (Redis/Memcached keyed by post_id) sits in front and absorbs the ~230k QPS read path. Hit rate is very high in practice — most feed reads are <24h old. Misses fall through to the source-of-truth store; the back end sees a tiny fraction of read traffic.

In the interview, naming Cassandra reflexively is fine, but be ready for "why not sharded MySQL?" The honest answer: this workload doesn't use any of Cassandra's interesting features (wide rows, tunable consistency). Pick it for write throughput and multi-DC active-active, not because it's a wide-column store.

Multi-region drift. Writes land in the author's home region, then post.created replicates to reader regions. For normal authors, fan-out can happen in the reader's home region once the event arrives; a 1-3 second delay is acceptable. For celebrities, only posts_by:{author_id} needs cross-region replication, so the blast radius is smaller. The feed cursor should include a feed_epoch or region_watermark so page 2 does not mix candidates from two ranking snapshots after a failover. Engagement counts are explicitly eventually consistent; visibility and blocks are checked against the freshest graph cache available and fail closed if stale.

Edge stress tests. A celebrity post is handled by pull-only indexing, not by trying to push to 50M lists. A dormant user's missing feed:{user_id} triggers pull-mode and async rebuild, not a cold 500 ms miss on every request. A power scroller carries the cursor into pull-back and cold-history stages instead of asking Redis for entries it no longer has. Burst writes from a highly active author are coalesced into fan-out batches by follower shard; if the queue lags, newest posts win and old low-score candidates can be skipped because ranking would bury them anyway. A cold start of a new Redis cluster is handled by serving pull-mode reads while fan-out workers lazily repopulate active users.

Interview punch line for reliability: the feed should be stale before it is slow. If Redis is cold or fan-out lags, I fall back to the author index and ranking can still produce a reasonable page.

Celebrity reads. Per-author "recent posts" Redis list at posts_by:{author_id} — last ~20 post_ids, TTL ~7 days. For celebrities, fan-out service skips writing into followers' feed lists and only updates this list; assembly reads it at request time. Cap a single user's celebrity fan-in to the most-followed N to avoid pathological reads. This per-author index isn't celebrity-only — it exists for every author and is the foundation of pull-mode fallback (below).

Why Redis ZSET/LIST instead of only the DB: the feed open has a 500 ms p99 and needs hundreds of tiny ordered reads. Redis gives predictable memory-resident range reads; MySQL/TAO stays the source of truth for post bodies and cold history. A pure Cassandra wide-row feed also works, but then every feed open pays storage read latency and compaction costs instead of RAM cost.

Pull-mode fallback. When a dormant user opens the app and feed:{user_id} is empty, assemble at read time by fanning in from the same posts_by:{author_id} index used for celebrities:

  1. Check the active set (bloom filter) — miss → pull mode.
  2. Get top-K followees by affinity (precomputed, cap to ~500) rather than scanning all 3000.
  3. Scatter-gather pipelined LRANGE posts_by:{author_id} 0 20 across those 500 author shards in parallel.
  4. Merge ~10,000 candidate post_ids, take top 300 by timestamp, hydrate post bodies, rank, return top 20.
  5. Async rebuild: enqueue a job to populate feed:{user_id} with the top-ranked post_ids and re-add the user to the active set. Their next open is fast.

Cost: ~75-125 ms for the cold path (still under the 500ms p99 budget). The trick is the per-author index — it converts "scan posts where author IN (300 followees)" (hot-key heaven) into 300 O(1) Redis reads.

Pagination beyond the feed list cap. feed:{user_id} is a cache of the top-of-feed, capped at ~300. When a power-scroller exhausts it, the cursor walks a state machine:

Stage Mode Source Latency
1 feed_list feed:{user_id} Redis sorted set <100 ms
2 pull_back posts_by:{author_id} scatter-gather with time watermark ~150 ms
3 cold_history Source-of-truth store with index on (author_id, created_at DESC) 200-500 ms
4 discovery Precomputed trending / friends-of-friends / suggested ~50 ms

The cursor is an opaque blob: {mode, last_seen_ts, seen_set_id}. The seen_set_id points to recently_shown:{user_id} (Redis set, TTL ~24h, capped) so dedupe happens server-side across pages. Discovery fill is typically interleaved before stage 3 in production — pure personal feed thins out fast and ads/suggested keep the app sticky.

Stage 3 needs a secondary index on (author_id, created_at DESC). In Cassandra that's a natural clustering key; in sharded MySQL/Vitess it's a composite index on each author shard. Reads are scatter-gather across author shards — slow but rare enough to tolerate.

Punch line for the interview: the feed list is a cache of the top of the feed, not the whole feed. Push and pull are two indexes of the same dataset. The cursor decides which tier to query, and the system degrades gracefully through them instead of cliffing when Redis runs out.

Backfill for new users. A new follow doesn't backfill historical posts into the feed list — too expensive. Instead, the assembly path pulls recent posts from the new followee at read time for the next few hours, then the steady-state hybrid takes over.

Consistency on engagement counts. Approximate counts shown immediately from a Redis counter; exact counts reconciled from ClickHouse hourly. Users never notice an off-by-3 like count.

What I'd defer. Ranking model training pipeline, ads injection, story/short video unification, integrity (spam/hate), notifications.
