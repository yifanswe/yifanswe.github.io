Metrics & Monitoring — Datadog / Prometheus Class (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a metrics & monitoring system. Hosts emit time-series metrics (counters, gauges, histograms) every 10 s. Engineers query for charts and alerts on conditions. 1M hosts × ~500 metrics each = 500M unique series × 6 datapoints/min = 3B datapoints/minute (~50M/sec). Queries scan minutes to months.

1. Requirements (~5 min)

Functional - Hosts push metric datapoints (name, tags{}, value, ts). - Query: arbitrary tag filters + aggregations (avg, p99) over time windows. - Alerts: rules that evaluate against the metric store and fire on condition.

I'd defer logging, tracing, and APM correlation — sibling systems.

Non-functional - Ingest 50M datapoints/sec sustained. - Query p99 < 1 s for typical dashboard queries (last 1 h aggregated). - Survive a single zone failure with no data loss. - Retention: 15 days raw, 1 year downsampled. - Eventual consistency on query — slight lag (~15 s) is fine.

Capacity - 50M points/sec × 24 B/point ≈ 1.2 GB/sec raw → ~100 TB/day → 1.5 PB at 15-day raw retention. - After compression (typical 10×): ~150 TB. - Cardinality: 500M unique series. Cardinality is the silent killer. - The inverted index is smaller than raw samples but still material: if each series has ~6 tag postings at ~8 B/series_id, 500M series costs ~24 GB per full index copy before compression/overhead. The dominant cost remains raw/warm sample storage.

2. Core Entities (~2 min)

  • Series — (metric_name, tags{}) → series_id. Unique combination.
  • Datapoint — (series_id, ts, value).
  • Aggregation tier — downsampled (e.g., 1-min, 1-h, 1-d).
  • Alert rule — query, threshold, window.

3. API (~5 min)

POST  /v1/ingest                    // host agent → batched metrics
GET   /v1/query?q=...&from=...&to=...    → time-series result
POST  /v1/alerts                    { rule }   → { alert_id }

Query language: PromQL-style — avg_over_time(http_latency{service="checkout"}[5m]).

Query cursors are rarely user-facing, but long table queries should carry {tenant, query_hash, shard_checkpoint, time_block} so partial scans can resume without repeating all index expansion.

4. Data Flow (~5 min)

Host agent batches metrics every 10 s → ingest → write to time-series store (sharded by series_id + time) → downsampling rolls up old data → query layer reads from the right tier → alerts run scheduled queries.

5. High Level Design (~10–15 min)

  1. Host agent. Lightweight daemon collects from local exporters, batches, ships every 10 s. Backoff on ingest failure; local disk buffer (limited).
  2. Ingest tier. Stateless ingest fleet behind a load balancer. Validates, normalizes tags, looks up/creates series_id (in a Redis-backed cardinality cache), forwards to Kafka.
  3. Kafka. Partitioned by series_id (or hash(metric_name + tags)). Long enough retention to replay on storage failure (24 h).
  4. Time-series storage. Custom or off-the-shelf (Prometheus' TSDB, InfluxDB, M3, Cortex). Storage layout: - Hot tier: recent 24 h on SSDs, fine-grained (10 s). - Warm tier: 1 min downsamples for 15 days. - Cold tier: 1 h downsamples for 1 year+ in cheaper storage (object store).
  5. Index. Inverted index per shard: tag=value → set of series_ids. Powers tag-filtered queries.
  6. Downsampler. Stream job consumes raw Kafka, writes 1-min and 1-h aggregates to warm/cold tiers.
  7. Query engine. Parses PromQL, expands tag filters via the index, scans relevant blocks, aggregates. Pushes down filters/aggregates into shards.
  8. Alert evaluator. Cron-driven worker that periodically runs each rule's query, applies threshold, fires notifications. Out of scope: notification delivery.

End-to-end narration: host emits http_latency{service=checkout, host=web-42, route=/buy} = 0.230 → ingest assigns series_id S99 → writes to Kafka partition for S99. Storage node consumes, appends datapoint to S99's block in the hot tier. An engineer queries avg by service of http_latency{service=checkout} over last 5 min → query engine looks up tag service=checkout in the index → gets ~500 series → scans last 5 min of each → aggregates avg → returns in 300 ms.

6. Deep Dives (~10 min)

Cardinality — the silent killer. Putting request_id or user_id into tags blows up series count. Defenses: - Cardinality limits per-metric (refuse new series past threshold). - Quarantine suspicious tag keys (auto-detect explosion, alert). - Schema: treat tags as bounded sets; high-cardinality data belongs in logs/traces, not metrics.

Storage layout. Series rows are columnar by time. Layout in a single block: - Time column (delta-encoded, then varint). - Value column (Gorilla XOR encoding — sub-byte/datapoint typically). - Compression yields 10× over naive layout. Per-series blocks sealed and immutable after their time window; written once, read many.

Why a time-series database rather than generic Cassandra rows: Cassandra can ingest high write volume, but query-time scans over tag filters and time windows need columnar compression, block pruning, downsampling, and a purpose-built inverted index. Prometheus alone is excellent per-cluster, but at 500M series I would use a horizontally sharded system like M3/Cortex/Thanos-style architecture.

Why Kafka in the middle: direct writes from ingest to TSDB minimize hops, but they make replay and zone failure recovery harder. Kafka gives a 24 h shock absorber, partitions by tenant + series_id, and lets downsampling consume the same stream. The tradeoff is a very expensive transient buffer, so retention must be short and compressed. Be ready for "why not ClickHouse only?" -- ClickHouse is strong for analytics scans, but alerting on fresh per-series data and high-cardinality tag indexes usually need a TSDB-shaped hot tier.

Sharding. Hash series_id to a shard. Each shard owns a slice of series. Replication factor 3 across zones for HA. Query fans out across shards; per-shard work is parallel.

Hot writes vs immutable reads. Recent data goes to a memtable; sealed every few minutes into immutable on-disk blocks (LSM-tree pattern). Query layer reads memtable + blocks, deduping.

Downsampling. Tumbling 1-min windows aggregate raw points → min/max/avg/count/sum/p50/p95/p99 per series per minute. Same again to 1-h. Aggregates retain enough statistical signal for most queries but are much smaller.

Query optimization. Push aggregations down to shards: instead of returning raw datapoints, shards return per-series aggregates. Reduces network bytes. For large tag-filter results, parallelize the per-series scan.

Query path trace: query engine parses PromQL -> expands tag filters through the inverted index -> intersects posting lists to get series_ids -> groups by shard -> parallel shard RPCs scan only matching time blocks -> shard returns partial aggregates -> root merges and down-samples for chart resolution. For a 1-year query, the planner uses 1h rollups instead of raw blocks.

Hot dashboard path, step by step: 1. Client sends a PromQL-style query; the planner hashes {tenant, query_text, time_range} and checks a short query cache. 2. Planner expands tag filters through tag:{tenant}:{key}:{value} posting lists, intersects them, and caps the resulting series set against tenant limits. 3. It groups series IDs by TSDB shard (series_id shard key) and sends parallel RPCs for only the requested time blocks. 4. Shards read memtable plus sealed raw blocks for the last hour, compute partial aggregates locally, and return compact vectors. 5. Root merges partials and returns chart-resolution points; cursor state for huge table views carries {tenant, query_hash, shard_checkpoint, time_block}.

Cold long-range path: for windows beyond raw retention, the planner skips raw blocks and reads rollup/{tier}/{series_id}/{day}. If a user asks for more series than the cap, the API returns a partial result with a continuation cursor instead of expanding an unbounded posting list.

Ingest path trace: agent sends a compressed batch -> ingest validates tenant budget and normalizes tags -> looks up or creates series_id in the registry -> appends the batch to Kafka keyed by series_id -> storage consumers write memtable entries -> sealed blocks update the inverted index and object storage manifest.

Alerts. Each rule has an evaluation period (e.g., every 60 s). Worker queries metric store with the same engine. State machine per rule (OK / alerting / resolved) with hysteresis to avoid flapping. Notification fan-out is async via a separate queue.

High-cardinality "ad-hoc" labels. If users insist on putting URLs in tags, route those metrics to a separate higher-cost store with cardinality caps, not the main TSDB.

Multi-tenant. Per-tenant namespacing on metric names; per-tenant cardinality budgets; rate-limit ingest per tenant.

Cross-region. Each region runs its own TSDB cluster for local ingest + dashboards. Cross-region queries (global dashboards) federate across regions at the query layer.

Two-view framing. The same metric stream is stored as raw per-series blocks for fresh troubleshooting and downsampled rollup blocks for long-range trend queries. The inverted index is a pointer view into those blocks, not another copy of metric content; protecting that pointer view from cardinality explosions is as important as compressing samples.

Query degradation state machine:

Stage Mode Source Transition Latency target
Hot raw last minutes/hours memtable + raw blocks normal dashboard < 1 s typical
Warm rollup days/weeks 1-min rollups range exceeds raw/detail budget < 1-2 s
Cold rollup months/year 1-h object-store blocks long range seconds, async ok
Partial too many series capped posting expansion series cap hit return partial + cursor
Degraded raw shard down replicated shard or rollup shard timeout lower resolution, explicit warning

Edge stress tests. A hot dashboard should hit cached partial aggregates for 10-30 s. Dormant series keep tombstones so recycled series_ids do not corrupt old blocks. A tenant that adds user_id as a tag gets new series rejected before the inverted index explodes. Burst writes are absorbed by Kafka and then shed by tenant budget if the buffer crosses a threshold. Cross-region drift is visible as per-region lag metadata; global dashboards should not pretend all regions are equally fresh.

Replay / disaster. Kafka retention covers a few hours; in catastrophe, hosts also retain a local disk buffer of recent points to replay.

Graceful degradation.

Case Mode Source User-visible behavior
Ingest burst Shed over-budget tenants first tenant rate limits partial data with explicit drops
New-series explosion Quarantine tag key / reject series cardinality cache existing metrics continue
Hot dashboard Cache query result and partial aggregates query cache + rollups slightly stale chart
Raw tier unavailable Serve downsampled data warm rollups lower resolution
Alert evaluator lag Keep last state, mark delayed alert state no false resolve

Interview punch line: "Metrics systems fail from cardinality before they fail from bytes; the design has to protect the index as aggressively as the storage tier."

Cost levers. Most cost is storage. Aggressive downsampling, columnar compression, and short raw retention are the big wins.

What I'd defer. Logs and traces (separate problems with similar shapes), exemplar tracking (linking metric points to traces), per-tenant cost attribution and billing, anomaly detection beyond static thresholds.
