Strava — Activity Tracking & Segments (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build Strava. Users start a workout, their phone streams GPS points throughout, and at the end we save the activity with map, splits, and segment leaderboards. 100M users, ~10M activities/day, each activity ~1k GPS points.

1. Requirements (~5 min)

Functional - Record a workout: stream GPS points from the phone, save as an Activity. - View your activities — map, pace, distance, elevation, splits. - Match an activity's path against pre-defined "segments" (a section of road/trail) and update segment leaderboards.

I'd defer the social feed, training plans, and Strava Premium analytics.

Non-functional - GPS ingestion is high-volume during peak afternoons but bursty per user; durability matters more than latency. - Activity save end-to-end p95 < 5 s after the user taps "Finish." - Segment leaderboard updates should appear within a minute of activity upload. - Eventual consistency on leaderboards is fine.

Capacity - 10M activities/day × 1k points × 50 bytes = 500 GB raw GPS/day, ~6 GB/s during peak afternoons globally. - Compressed time-series storage cuts that ~10×. - Compressed raw GPS is still the dominant long-term store: ~50 GB/day × 3 replicas ≈ 150 GB/day, or ~55 TB/year before lifecycle moves to colder storage. - Activity metadata is comparable but query-hot: 10M/day × ~5 KB = 50 GB/day before replicas. Segment efforts are smaller: if each activity matches 5 segments, 50M efforts/day × ~120 B × 3 ≈ 18 GB/day.

2. Core Entities (~2 min)

  • User — user_id.
  • Activity — activity_id, user_id, type, started_at, duration, distance, polyline (compressed path), stats.
  • GPSPoint — activity_id, ts, lat, lng, elev, hr?. Stored as a time-series within an activity blob.
  • Segment — segment_id, geometry (a polyline), creator.
  • SegmentEffort — segment_id, activity_id, user_id, time_ms. Drives leaderboards.

3. API (~5 min)

POST  /v1/activities                          → { activity_id, upload_url }    // server allocates id
POST  /v1/activities/{id}/points              { points[] }                     // batched, every ~30s
POST  /v1/activities/{id}/finish              → final activity object
GET   /v1/activities/{id}
GET   /v1/segments/{id}/leaderboard?period=all|year

User from token. Point uploads are idempotent on (activity_id, ts).

4. Data Flow (~5 min)

Phone collects GPS at 1 Hz → batches every 30 s → POST to ingest service → ingest writes to a per-activity blob (or appends to a time-series store) → on finish, run a background job: compute stats, match against segments, write Activity row and SegmentEfforts → push leaderboard updates.

5. High Level Design (~10–15 min)

  1. Ingest service. Accepts batched point uploads. Writes append-only to a per-activity object in object storage (one blob per activity) or to a time-series DB (InfluxDB / TimescaleDB) keyed by activity_id.
  2. Activity service. Owns the activity metadata row in MySQL (sharded by user_id). At finish, kicks off a processing job.
  3. Processing pipeline. Stream/job worker: - Reads all points for the activity. - Computes distance, duration, pace, elevation, splits. - Builds a compressed polyline (Google polyline encoding) and stores on the activity row. - Runs segment matching. - Writes the activity object back to the catalog.
  4. Segment matching. For each segment, geo-index its bounding box in S2 cells. For each activity, find segments whose bbox intersects the activity's bbox. Then run a precise match: does the activity's polyline pass through the segment's start, follow the path, and reach the end?
  5. Leaderboard service. Per-segment Redis sorted set: score = effort_time_ms (ascending), one entry per user (best time). Pruned to top N + the requester's neighborhood.
  6. Storage tiers. Raw GPS in object store (cold-friendly). Activity metadata + polylines in MySQL/Cassandra. Leaderboards in Redis.

End-to-end narration: Alice starts a ride at 8 AM → phone POSTs /activities → server gives activity_id A123 → during the ride, batches of 30 points stream in every 30 s → ingest appends to gs://strava/activity/A123/points.pb → at 9:30 she taps Finish → server enqueues processing job → worker reads all 5400 points, computes 28.4 km/1h28m, encodes polyline, finds her ride passed through segments S55 and S77, computes her effort times, writes to leaderboards → push notification "You got 3rd on Tunitas Climb."

6. Deep Dives (~10 min)

Concrete stores and shard keys.

Store Key pattern Data structure / entry TTL / retention Shard key
Raw points activities/{user_hash}/{activity_id}/points.pb protobuf/columnar blob, ~10 KB for 1k points compressed forever; cold after 30–90 d user_hash + activity_id
Activity metadata activity:{user_id}:{activity_id} row with stats/polyline/blob pointer, 2–5 KB forever user_id
Segment geo-index s2:{level}:{cell_id} list/set of segment_ids, KB–MB per busy cell no TTL cell_id
Segment effort effort:{segment_id}:{user_id}:{activity_id} immutable effort row, ~120 B forever segment_id
Leaderboard leader:{segment_id}:{period} Redis ZSET score=time_ms, member=user_id, ~30–60 B/entry hot forever; rebuildable segment_id
Upload dedupe point_seen:{activity_id}:{ts} short-lived seen marker or chunk checksum, ~32 B active + 24 h activity_id

Object storage holds the GPS content; MySQL/Cassandra stores searchable metadata and pointers. Redis leaderboards are a serving index that can be rebuilt from SegmentEffort.

Capacity by store. Raw point blobs dominate long-term retention: 10M activities/day × 5–10 KB compressed is 50–100 GB/day primary, or 55–110 TB/year with 3 replicas before cold tiering. Activity metadata is hotter for user history: 10M/day × 5 KB × 3 is ~150 GB/day. Segment efforts are smaller but high-write: 50M efforts/day × 120 B × 3 is ~18 GB/day. The S2 segment index is modest: even 10M segments × 3 cell refs × 100 B × 3 is under 10 GB. Leaderboards are capped serving indexes; if a hot segment stores top 10k plus neighborhoods at ~60 B/member, that is ~600 KB per hot segment, and the uncapped truth remains in SegmentEffort.

Point upload and finish path. 1. Phone creates activity, receives activity_id, and uploads chunks with sequence numbers/checksums. 2. Ingest validates ownership, dedupes by (activity_id, chunk_seq) or (activity_id, ts), and appends to the raw points blob/staging object. 3. On finish, Activity service marks activity processing and enqueues a job with the blob pointer. 4. Worker reads raw points once, computes stats/polyline/splits, writes activity:{user_id}:{activity_id}, then queries S2 cells for candidate segments. 5. Precise matcher writes SegmentEffort rows and updates leader:{segment_id}:{period} with best effort per user. 6. If leaderboard update fails, the effort row remains durable and a repair job rebuilds the Redis ZSET.

Segment matching at scale. Naive: for every activity, check every segment. 10M activities/day × 10M segments = 10¹⁴ — impossible. Defense: - Geo-index segments by S2 cell (level ~12, ~1 km²). - For each activity, compute the set of S2 cells the polyline touches. - Candidate segments = union of segments in those cells. - Run precise polyline match only on candidates (a few hundred typically).

Precise polyline matching. The activity must enter the segment near its start point (within ~10 m), follow the path (Frechet distance below threshold), and exit near the end. Implementation: walk the activity polyline and look for windows whose endpoints align with the segment.

Time-series storage choice. Raw points per activity are append-only, immutable after finish, queried as a unit. Object storage (S3) wins on cost; serialize the points as protobuf or columnar (Parquet) for size. Avoid putting them in a row store.

Why S3/object storage instead of InfluxDB/Timescale for raw points: the common read is "load one finished activity," not "aggregate all points across users by time." A time-series DB is useful for live telemetry dashboards, but object storage is cheaper and simpler for immutable per-activity blobs. S2 beats plain geohash here because cell coverings handle paths and boundaries more predictably.

For activity metadata, MySQL works if I need strong user-history pagination and simple transactions; Cassandra works if global write volume and multi-region writes dominate. I would choose MySQL/sharded SQL for the interview baseline because the high-volume GPS content is already outside the row store, and user activity lists want predictable ordered reads by (user_id, started_at).

Polyline encoding. Google polyline encoding with precision 5 gets ~10 bytes per point; a 1000-point ride → 10 KB instead of 50 KB raw.

Live tracking (optional). "Beacon" feature: friends can watch a ride live. Add a per-user Redis cell with the most recent point + an SSE channel that fans out the 30-s batches as they arrive.

Leaderboard cardinality. A popular segment may have hundreds of thousands of efforts. Redis sorted set with ZADD per upload is fine — keep only the best per user (ZADD GT). For "top N this year" we can keep one Redis ZSET per time window.

Two-index framing. The same ride is stored by activity for ownership (activity:{user_id}:{activity_id} + raw blob) and by segment for competition (effort:{segment_id}:{user_id}:{activity_id}). Those are not competing truths: the activity blob is the content, and segment efforts are secondary indexes derived from it. If a matcher bug is found, I rebuild the segment index from immutable activity content.

Leaderboard exhaustion and cold ranks. Redis only needs hot windows: top N, friends, and the viewer's neighborhood. When a user paginates beyond that, the cursor carries the last score and user_id into the durable SegmentEffort table. The latency budget can stretch from <100 ms to ~500 ms for deep pages because the user is already exploring long-tail ranks, not loading the main activity screen.

Cheating detection. GPS spoofing, motorized vehicle on a bike activity. Heuristics: implausible pace, sharp altitude jumps, average HR mismatched with workload. Flagged efforts hidden from leaderboards but stored.

Phone offline / spotty network. Phone buffers points locally; upload retries with exponential backoff. Ingest is idempotent on (activity_id, ts) so re-uploads are safe.

Graceful degradation.

Case Mode Source Latency budget Fallback
Phone offline local buffer phone storage until reconnect upload chunks later with same activity_id
Duplicate chunks idempotent ingest chunk checksum / timestamp key immediate ignore duplicates, keep first valid point
Huge activity chunked processing raw blob + streaming worker > 5 s accepted show activity processing, compute async
Hot segment rebuildable leaderboard SegmentEffort truth + Redis ZSET < 1 min normally queue ZADDs; rebuild ZSET if corrupted
Matcher failure partial activity activity row + retry job stats visible first segment efforts appear later

"The punch line: raw GPS is immutable content, leaderboards are rebuildable indexes; that separation lets me be cheap on storage and aggressive on serving."

What I'd defer. Social feed and kudos, training load calculations, route planning, group rides / live races.
