Google Docs — Collaborative Editing (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build Google Docs. Up to ~50 people edit the same document at once, each typing keystroke by keystroke. Everyone sees each other's cursors and edits within a second. Offline edits sync on reconnect. 100M daily documents touched, peak ~100 collaborators on the busiest docs.

1. Requirements (~5 min)

Functional - Multiple users edit the same document in real-time. - See other users' cursors and edits live. - Document is durable; reload returns the same content. - Offline edits sync on reconnect without manual conflict resolution.

I'd defer comments, suggestion mode, version history depth, sharing/ACL beyond mentioning, and rich embedded objects.

Non-functional - Real-time updates: typing → other clients see < 500 ms. - Strong convergence — all participants eventually see the same final document for a given set of edits. - Survive offline editing; merge gracefully. - Durability — never lose committed edits.

Capacity - 100M docs touched/day; concurrent active sessions ~10M. - Hot docs: ~100 concurrent editors → 100 × ~5 ops/s = 500 ops/s per doc. - Total ops: dominated by long tail; assume 1B ops/day globally. - Op log storage is modest but latency-sensitive: 1B ops/day × ~150 B/op × RF=3 ≈ 450 GB/day. Snapshots at ~50 KB for 100M touched docs are ~5 TB per snapshot generation before RF; snapshot/object storage is the dominant durable byte cost.

2. Core Entities (~2 min)

  • Document — doc_id, owner, ACL, current snapshot, version.
  • Op — edit operation (insert / delete with position + content + author + logical timestamp).
  • Session — session_id, doc_id, user_id, cursor_position.
  • Snapshot — periodic materialization of doc state for fast load.

3. API / Protocol (~5 min)

WS  /v1/docs/{doc_id}/sync
  ← client: { op, base_version }
  → server: { op, version, author }    // op stream from peers
  → server: { presence, cursor }       // presence beacons

HTTP:
POST /v1/docs                              → { doc_id }
GET  /v1/docs/{doc_id}/snapshot            → snapshot + version

User from token; doc_id and base_version on each op for ordering.

The server issues monotonically increasing versions per doc. Reconnect cursors encode {doc_id, last_acked_version, client_session_id}; presence has no cursor because it is best-effort.

5. High Level Design (~10–15 min)

This problem is about conflict resolution. Two viable models:

Operational Transformation (OT)

  • Each op references a base version.
  • Server has the canonical sequence; when a new op arrives based on an older version, the server transforms it against intervening ops before applying.
  • Client similarly transforms received ops against any local in-flight ops.

Conflict-free Replicated Data Types (CRDT)

  • Each character has a unique, globally ordered ID (Logoot, RGA, Yjs).
  • Insert specifies "after which character"; delete refers to character ID.
  • Commutative — any order of applying ops yields the same result. No transformation needed.

I'd default to OT for an interview because it matches what Google Docs actually uses and the server-as-authority model is easier to explain; mention CRDT as the alternative used by Figma and Yjs.

System layout

  1. Edit session router. Hash doc_id to a session server (single-owner pattern). All edits for one doc converge on one server for ordering. This is the heart.
  2. Session server. Holds the canonical op log in memory + a streaming queue per connected client. Applies OT to incoming ops, broadcasts transformed ops to peers, persists ops to durable storage.
  3. Op log storage. Append-only — Cassandra/Spanner keyed by (doc_id, seq). Replicated for durability.
  4. Snapshot service. Every N ops or M seconds, fold the op log into a doc snapshot and persist. Loaders fetch latest snapshot + tail of ops.
  5. Presence / cursors. Separate channel from edits — ephemeral, lossy. Each client pushes cursor position; session server broadcasts. No persistence needed.
  6. WebSocket gateway. Persistent connection per client; routes to the right session server.
  7. Offline buffer. Client retains uncommitted ops locally. On reconnect, replays from its last server-acknowledged version.

End-to-end narration: Alice and Bob both have the doc open. Alice types "hello" → her client sends op insert(pos=0, "hello", base_version=42) → session server is at version 47 already → server transforms Alice's op against ops 43–47 → applies, broadcasts to all sessions including Bob. Bob's client receives insert(pos=5, "hello", version=48), applies it (also transforming against his unacknowledged local op if any), and renders.

6. Deep Dives (~10 min)

OT in practice. Each op carries (type, position, content, base_version, author). Server maintains canonical op log. On receipt: 1. If base_version == current_version, apply and broadcast. 2. Else transform op against ops (base_version + 1 ... current_version) using transformation rules (a delete shifts positions of subsequent inserts, etc.). 3. Apply, increment version, broadcast.

Client mirrors this for in-flight local ops it hasn't gotten the server ACK for.

Write path trace: client -> WebSocket gateway -> owner lookup by doc_id -> session server validates ACL and client_op_id; session server fetches missing tail ops if base_version is stale; transforms, appends op:{doc_id}:{version} durably, updates in-memory snapshot, then broadcasts. Append and broadcast are ordered: if durable append fails, the op is not acknowledged.

Hot write path, step by step: 1. Client sends {doc_id, client_op_id, base_version, op} over the existing WebSocket. 2. Gateway hashes doc_id, checks doc_owner:{doc_id}, and forwards to the current session owner; this is the shard key at every hop. 3. Owner validates ACL and seen_op:{doc_id}:{client_op_id} in parallel with loading the short op tail from memory. 4. If base_version is current, the owner applies immediately; if it is stale, it reads op:{doc_id}:{base_version+1...current} from memory or the durable log and transforms first. 5. Owner appends op:{doc_id}:{new_version} durably, then ACKs the author and broadcasts to peer streams. Presence updates never block this path.

Cold load / recovery path: 1. Client loads snapshot:{doc_id}:{snapshot_version} from the snapshot store and asks the gateway for the live owner. 2. If owner exists, it streams ops after snapshot_version; if owner is cold, the gateway elects one with a doc_owner:{doc_id} lease. 3. New owner replays durable ops after the snapshot, rebuilds the in-memory document, and only then admits writers. 4. Cursor state persists as {doc_id, last_acked_version, client_session_id} so reconnect and pagination through the op tail resume from the right version.

CRDT alternative. Each character gets a unique ID (a fractional position). Insert says "between X and Y, insert with ID Z." Concurrent inserts produce a consistent order without server transformation. Server can be a simple relay. Storage cost is higher per character.

Why OT here rather than CRDT: with a central session owner, OT keeps operations small and gives the server one canonical version for ACL, audit, and undo. CRDT is stronger for offline-first peer-to-peer editing, but the per-character IDs and tombstones increase storage and make rich document operations harder to explain in 45 minutes.

Storage choice defense: Spanner fits if I want strongly consistent per-document transactions and cross-region replicas, but it is expensive for a pure append log. Cassandra fits the append-heavy (doc_id, version) access pattern if the session owner is the consistency point. Kafka alone is not enough because users need random tail replay by doc and long-term durability beyond log retention. Be ready for "why not only object storage?" -- object storage is perfect for snapshots, but too slow and coarse for per-keystroke tail reads.

Single-owner per doc. All edits for a doc must serialize at one node so the OT canonical order is well-defined. Achieved by consistent-hashing doc_id to a session server. On owner failure, leader election + log replay from durable op store.

Snapshotting. Reading 6 months of ops to load a doc is prohibitive. Periodic snapshots: every 1000 ops or 5 min, fold ops into a snapshot. Load = snapshot + ops since snapshot.

Load path trace: GET snapshot reads the latest snapshot:{doc_id}:{version} and, in parallel, asks the owner for the op tail after that version. If the owner is cold, the gateway replays the tail from the durable op log and elects a new owner before admitting editors.

Offline / disconnect. Client keeps a local op log. On reconnect, sends {since_version: N}. Server replays missed ops to client; client transforms and replays its buffered local ops, sending each as a new op based on the new version.

Cursor / presence. Different channel: high frequency, no persistence. Cursors flicker is fine. Throttle to ~10 Hz per user.

ACL / sharing. Permissions checked at session connect (read-only vs editor). Server tags ops with author; clients enforce read-only by refusing to send.

Conflict patterns OT handles cleanly. - Insert + insert at same position: deterministic by author tiebreak. - Insert + delete on the same range: insert wins (text preserved), delete shifted. - Concurrent identical edits: deduped by op ID.

What it doesn't handle gracefully. Schema-level conflicts (one user inserting a table, another deleting the surrounding paragraph). These require op types beyond text — Google Docs uses a richer op model with structural ops.

Hot doc scaling. 100 collaborators on one doc → 500 ops/s on a single session server. Comfortable for one node. Beyond that (1000s of viewers reading), separate read replicas pull from the op log; only writers go through the owner.

Two-view model. The durable dataset is the same document history, exposed in two views: op:{doc_id}:{version} is the fine-grained edit index for sync/replay, while snapshot:{doc_id}:{version} is the hydrated content view for fast load. The snapshot is derived; if it is stale or missing, replaying the op index recovers correctness.

Edge stress tests. A dormant document whose owner lease expired cold-starts from snapshot plus log tail and should load in under 2 s for normal docs. A user paginating far back in version history leaves the hot op tail and reads compacted audit storage, so the UI should switch to a history-loading state. A celebrity/shared public doc with thousands of viewers should fan out through read-only replicas while the writer path remains single-owner. Multi-region drift is handled by routing all active edits for a doc to one write region; remote regions serve snapshots and subscribe to the op stream with visible lag.

Session state machine:

Stage Mode Source Transition Latency target
Active owner in memory owner doc state + hot op tail normal edits < 500 ms peer visibility
Warm reconnect owner alive, client stale owner memory + op tail client cursor behind < 1 s catch-up
Cold doc no owner lease snapshot + durable op log first editor opens < 2 s before editable
Recovery owner failed durable op log + lease election heartbeat miss seconds; freeze writes
Archive history compacted versions snapshot chain + archived ops user opens old version slower, read-only first

Durability. Every accepted op fsynced to the op log before broadcasting. Snapshots are async.

Graceful degradation.

Case Mode Source User-visible behavior
Owner fails Freeze writes, elect owner, replay log tail op log + owner map seconds of reconnect
Client offline Local optimistic edits browser local log sync on reconnect
Snapshot stale Snapshot + longer op tail snapshot store + op log slower load, correct doc
Presence fan-out hot Drop/throttle cursor updates presence TTL store text edits unaffected
Op log store degraded Reject new edits after small buffer session memory + durable log fail closed to avoid lost edits

Interview punch line: "The single-owner server is not a scalability weakness here; it is the mechanism that makes OT understandable, durable, and auditable per document."

What I'd defer. Comments and suggestions, version history UI, rich object embeds (images, tables), import/export, fine-grained sharing.
