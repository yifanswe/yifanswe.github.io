Notification System — Multi-Channel Delivery (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build the notification fan-out service. Backend services trigger notifications; we deliver via push (iOS/Android), email, SMS, and in-app. 1B users, ~100M notifications/day average, big bursts (e.g., 50M push pings in 30 s for a breaking news alert). Respect user preferences and rate limits.

1. Requirements (~5 min)

Functional - Backend services submit notification events (recipient, template, payload, channels). - System renders messages per channel, applies user preferences, and dispatches. - Track delivery status (sent / delivered / failed) for retries and reporting.

I'd defer in-depth marketing analytics and segmentation/audience-building tools.

Non-functional - High availability — undelivered transactional notifications are real failures. - Throughput: sustain 100M/day = ~1200/sec average; absorb 50M-in-30s = ~1.6M/sec bursts. - Latency targets per channel: push p95 < 5 s, email p95 < 30 s, SMS p95 < 30 s. - Retry transient failures; never silently drop. - Respect user preferences (channel opt-outs, quiet hours).

Capacity - 1B users; ~3B device tokens (Android/iOS/Web). - Burst notification: 50M sends in 30 s ≈ 1.6M/sec. - If the average event fans out to 1.4 channels, 100M events/day creates ~140M delivery rows/day. At ~700 B/row × 90 d × 3 replicas, delivery status is ~26 TB and is the dominant hot store. - Device tokens are large but stable: 3B × ~200 B × 3 replicas ≈ 1.8 TB before indexes.

2. Core Entities (~2 min)

  • NotificationEvent — event_id, user_id, template_id, payload, channels[], priority.
  • Delivery — delivery_id, event_id, channel, recipient_address, status, attempts.
  • UserPreferences — user_id, channel_opt_ins, quiet_hours, device_tokens[].
  • Template — template_id, channel-specific bodies, locales.

3. API (~5 min)

POST  /v1/notifications              { user_id, template_id, payload, channels[] }   → { event_id }
POST  /v1/notifications/bulk         { events[] }                                    → { batch_id }
GET   /v1/deliveries/{id}            → status

// Outgoing webhooks from providers (Apple, FCM, SES, Twilio):
POST  /v1/webhooks/provider/{name}   → delivery status update

Internal callers; idempotent on event_id.

4. Data Flow (~5 min)

Event submitted → enriched with user preferences → fanned out per channel → per-channel queue → per-channel worker pool → provider API call → status recorded → retry on failure.

5. High Level Design (~10–15 min)

  1. Ingestion service. Accepts events. Validates, writes the event row (NotificationEvent in Cassandra, sharded by event_id), publishes to a router topic.
  2. Router service. Stream consumer that: - Looks up user preferences (cached). - Filters out opted-out channels. - Applies quiet-hours (defer or drop based on policy). - Splits the event into one Delivery per surviving channel. - Publishes each delivery to that channel's queue.
  3. Per-channel queues. Kafka topics: dispatch.push, dispatch.email, dispatch.sms, dispatch.in_app. Sized differently — push the largest.
  4. Per-channel worker pools. - Push workers: Group by token type, batch APNs/FCM calls (both support bulk), respect per-provider rate limits. - Email workers: Use email provider (SES, Sendgrid), respect per-domain throttling. - SMS workers: Per-carrier and per-country rate limits. - In-app workers: Write to user's in-app notification feed (Cassandra row), push via WS if connected.
  5. Provider callbacks. Each provider returns status (sent/delivered/bounced/etc.). Webhook handlers update Delivery rows.
  6. Retry policy. Per channel: exponential backoff with cap. Push transient: retry 3 times over 1 hour. Email: retry 3 days. After give-up, dead-letter for ops review.
  7. Rate limit per user. "Don't email someone 10× a day." Token bucket per (user_id, channel) in Redis. Drops or coalesces if exceeded.
  8. Templates & rendering. Templates with locale variants + payload variables. Rendered at dispatch time (per locale, per channel).

End-to-end narration: a friend-request service emits a notification event for user U. Router pulls U's prefs — opted in to push + in_app, opted out of email; quiet hours not active. Two deliveries written: push and in_app. Push delivery routes to APNs worker, which batches with other deliveries to the same push gateway, calls APNs. Status sent; later APNs webhook says delivered. In-app delivery writes to U's feed; if U has an open WS, immediate broadcast.

6. Deep Dives (~10 min)

Concrete stores and shard keys.

Store Key pattern Data structure / entry TTL / retention Shard key
Event record notif_event:{event_id} Cassandra/Dynamo row, payload pointer + template + priority, ~0.5–1 KB 30 d, transactional 1 y event_id
Delivery status delivery:{delivery_id} row with channel/status/attempts/provider_ref, ~700 B 30–90 d by channel delivery_id or (user_id, day) for user lookup
User prefs cache pref:{user_id} Redis hash, opt-ins/quiet hours/time zone, ~300 B TTL 1 h; invalidate on write user_id
Device tokens tokens:{user_id} set/list of token records, ~200 B/token until provider invalidates user_id
In-app feed inapp:{user_id}:{yyyy_mm} wide row/list of notification IDs + snippet, ~150 B/item 90 d hot, archive older user_id
Idempotency notif_idem:{producer}:{event_id} response pointer/status, ~200 B 24–72 h (producer, event_id)
Delayed delivery delay:{deliver_at_minute} queue/ZSET of delivery_ids until fired deliver_at_minute
User/channel cap notif_cap:{user_id}:{channel}:{yyyy_mm_dd} Redis token bucket, ~40 B 2 d user_id

Kafka queues are transport and buffering; Cassandra/Dynamo rows are the durable queryable state. I would call that out because interviewers often probe whether the queue is accidentally becoming the source of truth.

Capacity by store.

Store / index Math What dominates
Event records 100M/day × 0.8 KB × 30 d × 3 replicas ≈ 7.2 TB payload pointers, not the rendered body
Delivery status 140M/day × 700 B × 90 d × 3 ≈ 26 TB dominant hot durable store
Preference cache 100M hot users × 300 B × Redis overhead/replica ≈ 100–200 GB real memory active users only; cold falls through
Device token DB 3B tokens × 200 B × 3 ≈ 1.8 TB stable but write-light
In-app feed 40M/day × 150 B × 90 d × 3 ≈ 1.6 TB user-keyed index, not full content
Burst queue 50M delivery pointers × 200 B × 3 ≈ 30 GB raw broker data short-lived buffer, not long-term state

Why Cassandra/Dynamo-style rows instead of a single relational cluster: delivery writes are append-heavy, TTL-friendly, and naturally partition by delivery_id or (user_id, day). MySQL can work at smaller scale, but the 26 TB hot-status footprint and burst write rate push me toward a wide-row or Dynamo-style store. Elasticsearch is a reporting index only; it should not be the source of truth for delivery status.

Submit notification path, hot and cold. 1. Producer calls POST /v1/notifications with event_id; ingestion checks notif_idem:{producer}:{event_id} on shard (producer, event_id) and returns the old response on duplicate submit. 2. Ingestion writes notif_event:{event_id} on shard event_id, then publishes a small pointer to router.normal or router.high with Kafka partition key user_id so per-user coalescing and rate caps stay ordered. 3. Router fetches pref:{user_id}, tokens:{user_id}, and notif_cap:{user_id}:{channel}:{day} in parallel. On a preference/token cache miss, it falls through to the preference/token DB by user_id, repopulates Redis, and marks missing tokens as explicit failed_no_token deliveries rather than silently dropping push. 4. Router filters opt-outs, quiet hours, and per-user caps; each surviving channel gets a delivery:{delivery_id} row before enqueue, with secondary lookup fields (user_id, day) for support/debug views. 5. Channel workers consume dispatch.{channel}.{priority}. They batch by provider/token type, call APNs/FCM/SES/Twilio, and update delivery:{delivery_id} with attempt/status/provider_ref. 6. Provider webhooks update the same delivery row idempotently using (provider_ref, provider_event_id); retries use the durable attempt counter, not worker memory. 7. Cold-start path: if the router cache is empty after deploy, it degrades to DB-backed preference/token reads and protects the DB with a small per-user local cache plus queue backpressure.

Two-index framing. Notification content is written once as notif_event:{event_id}. Delivery status is indexed by delivery_id for provider callbacks and by (user_id, day) for support/user-history reads. In-app feed is another user-keyed view over the same event/delivery IDs; it should store snippets and pointers, not a second copy of every rendered message.

Burst handling (50M in 30 s). Three pressure points: - Ingest: stateless; scale horizontally. Bounded by network. - Router: stateless; reads prefs from a cache; scale on demand. Pre-warm before known events. - Provider rate limits: can't be exceeded. APNs and FCM accept hundreds of thousands/sec per app; you still queue and drain.

For mega-bursts (50M in seconds), the queue absorbs and the worker pools drain over a longer window. Acceptable as long as priority traffic isn't blocked.

Why Kafka instead of a simple Redis queue or SQS-only design: Kafka gives ordered partitions, replay for router bugs, and consumer-group scaling across the 1.6M/sec burst. SQS/Pub/Sub can work for simpler systems, but replay and per-priority partition control are weaker. Redis queues are too memory-fragile for a burst that must survive worker downtime.

Edge stress tests.

Edge Failure mode Fallback / budget
Dormant user prefs/tokens not cached DB read by user_id, populate Redis; keep submit p95 under a few hundred ms
Hot celebrity / broadcast one campaign creates 50M rows segment fan-out into jobs; preserve transactional priority queues
In-app feed past 90 d hot row window exhausted cursor switches from inapp:{user_id}:{yyyy_mm} to archive store; chart/history can be slower
Provider cap APNs/SMS throttles defer in delay:{deliver_at_minute} and drain by priority
Multi-region drift duplicate webhook or retry in another region idempotent delivery_id update; provider reference is globally unique

Priority queues. Critical transactional (password reset, auth code) vs marketing (weekly digest). Separate queues + workers prevent marketing surge from delaying transactional. Each channel has at least two priorities.

De-duplication. Idempotency on event_id end-to-end. Router stores (event_id, channel) -> delivery_id so re-submit doesn't double-send.

User preference cache. 1B users × small pref row = a few hundred GB. Cache hot users in a Redis cluster; cold users hit MySQL/Cassandra. Updates propagate via cache invalidation on pref change.

Quiet hours and time zones. User's time zone in preferences. Router computes "is now within user's quiet hours?" If yes, defer the event (write to a delayed queue with deliver_at set to end-of-quiet-hours). Mark transactional events as quiet-hour-overriding.

Coalescing. Friend likes 5 photos in 1 minute → one notification "5 likes" rather than 5 separate. Implemented at the router: collapse same-template events within a window into one. Bumps engagement, lowers user fatigue.

Per-provider failover. Multiple providers for SMS (Twilio + alternates) and email (SES + Sendgrid). Per-provider success-rate monitoring; auto-shift traffic if one degrades. Routing config hot-reloads.

Status tracking. Each delivery row updates on provider callback. Status enables sender-side analytics ("Was my password reset delivered?"). Per-channel retention varies — push status often only kept short-term; transactional kept 90 days.

Graceful degradation state machine.

Stage Mode Source Latency / retention Action
Accepted durable event notif_event:{event_id} immediate event is safe before routing
Deferred quiet hours / provider cap delay:{deliver_at_minute} minutes–hours wake later with same delivery ID
Provider degraded queued drain Kafka + delivery attempts channel SLA may stretch preserve priority queues; do not block transactional
Duplicate submit idempotent return notif_idem:{producer}:{event_id} < 10 ms return original event_id / status
Permanent failure dead letter delivery:{id} + DLQ retained 30–90 d expose failure, no silent drop

"The line I want the interviewer to remember: providers are best-effort, but my internal state is not; every notification is either delivered, deferred, or explicitly failed."

Token rot. Mobile tokens expire on app uninstall. Provider returns Unregistered → invalidate the token in our DB. Periodic sweeper removes very old tokens never refreshed.

Multi-region. Each region routes to its local providers; cross-region failover if a region's providers all degrade. Provider rate limits often global, so coordinate via shared counters.

What I'd defer. Marketing segmentation, A/B testing of templates, machine-learning send-time optimization, content moderation, internationalization tooling beyond locale templates.
