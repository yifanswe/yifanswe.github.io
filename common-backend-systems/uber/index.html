Uber — Rideshare Backend (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build Uber. A rider requests a ride; the system matches them with a nearby driver; both see each other's live location; on drop-off, payment captures. 100M monthly riders, 5M drivers, 20M rides/day, peak ~1500 rides/sec, ~1M concurrent drivers reporting GPS at 4 Hz.

1. Requirements (~5 min)

Functional - Rider requests a ride from origin to destination; system matches an available driver. - Both sides see each other's live location during the trip. - On completion, capture payment.

I'd defer Uber Eats, surge pricing depth, driver onboarding/background checks, and ride pooling (UberPool) beyond mentioning it.

Non-functional - Match latency p99 < 5 s — riders won't wait. - Live location updates within 1 s. - Strong consistency on ride state machine (requested → matched → in-progress → completed). - Regional isolation — one city's outage doesn't stall others. - Highly available — money on the line.

Capacity - 1M concurrent drivers × 4 Hz GPS = 4M GPS writes/sec. - 1500 rides/sec peak. - Per ride: ~30 min of GPS pings × 240 pings/min = 7200 GPS pings persisted per ride. - Live location state is tiny: 1M drivers × ~80 B latest-location record ≈ 80 MB primary, a few hundred MB with replicas. The dominant cost is the stream: 4M pings/sec × ~100 B ≈ 400 MB/sec, ~35 TB/day before compression/replication. - Ride state is small: 20M rides/day × ~1 KB/ride ≈ 20 GB/day primary; it must be transactionally correct, not huge.

2. Core Entities (~2 min)

  • Rider — rider_id.
  • Driver — driver_id, current location, status (offline / available / on-trip), current ride.
  • Ride — ride_id, rider, driver, origin, destination, status, fare.
  • GPSPing — (driver_id or rider_id), lat, lng, ts.

3. API (~5 min)

POST  /v1/rides                  { origin, destination }   → { ride_id, status: requested }
GET   /v1/rides/{id}             → Ride
WS    /v1/rides/{id}/stream      → push: state changes + counterparty location

POST  /v1/drivers/location       { lat, lng }   (auth: driver)
POST  /v1/drivers/rides/{id}/{action}    // accept, pickup, dropoff

User from token. Ride state transitions idempotent on (ride_id, action).

Ride cursors are not exposed; the server owns the ride state machine. Clients only submit actions, and each action is accepted if the current persisted state allows it.

5. High Level Design (~10–15 min)

I'd partition the world by city/region. Each region is mostly autonomous.

  1. Ride service. Owns the ride state machine. Sharded by ride_id, but routing ensures all rides for one region land on a regional cluster.
  2. Driver location service. Drivers POST GPS at 4 Hz → ingest writes to a regional Redis geospatial index (GEOADD) and to Kafka for downstream consumers (analytics, anti-fraud, trip records).
  3. Matching service. Triggered on ride request: - Query nearby drivers in the rider's S2 cell + ring (cheap GEORADIUS). - Filter to available drivers (status = available). - Score: ETA to pickup × driver acceptance rate × surge willingness. - Offer to top candidate; on reject/timeout, move to next.
  4. Notification. Driver app receives offer over a persistent WS or push channel. Driver accepts within ~15 s or the offer reroutes.
  5. Trip tracker. Once matched, both rider and driver share a WS connected to the ride's channel; pings from one side fan out to the other.
  6. Pricing service. On ride creation, estimate fare from a heuristic (base + per-km + per-min + surge multiplier for the cell). On dropoff, finalize from actual trip data.
  7. Payment. Out of scope to design, in scope to draw. Capture after dropoff via the payment gateway.

End-to-end narration: Alice taps "Request UberX" → ride service writes ride row (status=requested) and asks matcher → matcher GEORADIUSes for nearby drivers, picks Driver 42 → notification pushes offer → 42 accepts → ride state → matched, driver assigned. Both apps open WS to the ride channel. Driver location pings stream from 42 → ingest → fan-out to Alice's WS. At pickup, driver hits "Pickup" → ride state → in_progress. At dropoff, ride state → completed, fare finalized, payment captured.

6. Deep Dives (~10 min)

Concrete store contracts.

Store Key / data structure Entry size + TTL Shard key
Driver geo geo:{region}:{s2_cell} Redis geo/zset member=driver_id ~40 B/member, TTL 10s region+s2_cell
Driver status driver:{driver_id}:status KV {available,on_trip,ride_id,version} ~100 B, TTL while online driver_id
Offer lease offer:{driver_id} → ride_id with NX EX 15 ~40 B, TTL 15s driver_id
Ride truth rides:{region} SQL/Spanner row ~1 KB, retained years region, ride_id
Ride event log Kafka ride_events ~200 B/event, 7–30 days hot ride_id
GPS history object/columnar store partitioned by date + region 20–100 B/ping compressed, 30–90 days hot region, date

Capacity by store. The hot matcher state is tiny compared with the stream: 1M online drivers × 40 B × 3 is ~120 MB for geo entries, and status rows are ~300 MB with replicas. Offer leases are tiny: even 1500 rides/sec × 15 s × 40 B × 3 is under 3 MB. Ride truth is 20M rides/day × 1 KB × 3 or ~60 GB/day. The expensive path is GPS history and Kafka retention: 4M pings/sec × 100 B is ~35 TB/day before compression/replication, so I keep latest location in Redis, short retention in Kafka, and compact trip traces into columnar object storage.

Ride request path trace. Client → ride service creates rides row with requested → matcher queries nearby S2 cells in parallel from Redis geo → reads driver status for top candidates in parallel → writes offer:{driver_id} lease with NX EX 15 → notification pushes offer → driver accept calls ride service → ride service conditionally updates ride requested→matched and driver available→on_trip in one transaction/compare-and-set → emits ride.matched to Kafka. If the first driver times out, the lease expires and the matcher tries the next candidate.

Geo-indexing at 4M GPS writes/sec. Redis geospatial index per region. Each region's Redis sharded into multiple cells if a single Redis maxes out. Writes are GEOADD driver:region:cell; reads are GEOSEARCH. Sub-millisecond per op, fits in-memory.

Why Redis/S2 instead of PostGIS or Elasticsearch: this path is high-write, low-retention, and needs nearest active drivers within seconds. PostGIS is great for durable spatial queries but not 4M/sec volatile writes; Elasticsearch adds indexing lag and heavier refresh costs. The durable GPS trail goes to Kafka/object storage, not the hot matcher.

For ride truth, I would use a regional SQL/Spanner-style store rather than Cassandra because the state machine needs conditional transitions and driver assignment must not double-book. Cassandra is fine for append-only event history, but the source of truth for requested→matched→completed benefits from transactions or strong compare-and-set semantics.

Matching algorithm. Naive nearest-driver is greedy and unfair. Better: batched matching every 1–2 s collects pending requests + available drivers in a cell, runs a bipartite assignment minimizing total ETA. Mention Hungarian or auction-style matching as the production lever. Also: drivers can be offered to multiple riders but lock the first to accept.

Concurrency on driver assignment. Two ride requests in the same area could pick the same driver. Use Redis SET driver:42:offered ride:99 NX EX 15 to lease the driver for 15 s while the offer is outstanding. If they reject, lease expires and they're eligible again.

GPS fan-out. A trip has 1 driver and 1 rider — fan-out is 1:1 per trip. The challenge is the ingest rate (4M/sec), not the fan-out. Use ingest workers per region with stateless write-through to Redis. Persist to Kafka for analytics, then to cold storage.

Ride state machine. Owned by the ride service. Each transition is a single row update with a state check (UPDATE rides SET status='completed' WHERE id=X AND status='in_progress') to enforce legal transitions. Emit Kafka events on each transition for downstream services.

Stage Allowed next state Source of truth Latency target
requested matched, canceled ride row + matcher attempt <5 s to first offer
matched driver_arrived, canceled ride row + driver status immediate CAS
driver_arrived in_progress, canceled_no_show ride row <500 ms
in_progress completed, support_intervention ride row + trip pings <500 ms
completed terminal ride row + payment idempotency payment may settle async

Interview punch line: "Location can be eventually consistent; the ride state machine cannot, because that is where money, safety, and driver assignment meet."

ETA prediction. v1: haversine distance × city avg speed + buffer. v2: routed ETA from a map service that knows current traffic. v3: ML model fed by historical trip times. Each level adds latency and complexity; we'd start at v1 and call out the upgrade path.

Surge pricing. Per-S2-cell rolling demand/supply ratio; surge multiplier = piecewise function of the ratio. Recomputed every minute, written to a Redis cache the pricing service reads on ride creation. Show surge to rider before they request.

Two-index framing. Drivers are indexed by location for matching (geo:{region}:{s2_cell}) and by identity for correctness (driver:{driver_id}:status). Matching reads both: the geo index finds candidates, but the driver status row decides whether a driver is actually assignable. This is the same active-driver dataset optimized for two access patterns.

Failure modes. - Driver app dies mid-trip: server's last_ping > 30s triggers a check; rider sees "trying to reconnect." - Network split mid-payment: payment is idempotent on ride_id, retried. - Rider cancels post-match but pre-pickup: small cancellation fee, driver freed.

Regional isolation. Each city has its own ride/matching/Redis stack. Cross-region failures are contained. Global services only for identity, ride history aggregation, and payment.

What I'd defer. UberPool matching (multi-stop optimization), driver heatmaps, fare splitting, scheduled rides, driver routing optimization, fraud detection.
