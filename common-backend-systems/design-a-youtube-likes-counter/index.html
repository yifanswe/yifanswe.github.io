1. Basic Knowledge And Solution

What This Question Is Testing

A likes counter sounds trivial — increment a number — until you account for billions of videos, viral hot keys with millions of likes per minute, idempotency for double-tap, dislike toggling, fraud, and the user expecting their own like to appear immediately while everyone else can tolerate seconds of lag.

The invariant is this:

Each (user, video) like is recorded at most once and is reversible; the displayed count converges within bounded staleness to count(active likes) for that video; the user's own like state is consistent from their perspective immediately after they tap; and fraudulent or abusive likes are filtered before the count is publicly visible.

At ICT5 / L6+ level, the interviewer is testing whether you can model likes as a per-user state (not just a counter), handle hot videos without locking on one row, design read-your-own-write semantics, and integrate fraud filtering without slowing the like action.

60-Second Interview Framing

Spoken version:

I would design the likes system in two layers: an authoritative per-user like state that records who liked which video and when, and a denormalized counter per video that is updated asynchronously from the state changes. The interesting decisions are hot-video counter sharding, read-your-own-write semantics for the user's own like, idempotency for double-tap, and fraud filtering before the count goes public.

Requirements To Clarify First

Functional requirements:

  • User can like a video; one like per user per video.
  • Toggle off (unlike) reverses the action.
  • Display total like count per video.
  • Show "you liked this" indicator to the user who liked.
  • Per-user history: "videos I liked."
  • Filter spam, bots, click farms.
  • Optional dislike (or treat as separate; modern YouTube hides dislikes).

Non-functional requirements:

  • Sub-second response for the like action.
  • Counter freshness: seconds to minutes for normal videos; tolerate longer for viral.
  • Read-your-own-write: the liking user sees their like instantly.
  • Strong correctness on per-user state (one like per user per video).
  • Approximate display count is acceptable.
  • Petabyte scale: billions of videos, trillions of like records over time.

Out of scope unless asked:

  • Recommendation impact of likes (separate system).
  • Display of "people you know liked" (separate social system).
  • Comment likes (similar design, separate scale).

Scale Assumptions

Area Assumption
Videos 5B existing; 500M+ active.
Users 2B+.
Like events 10B+ per day; 100k-1M/sec peak.
Viral video 10k-100k likes/sec on one video.
Read QPS Likes count on every video page; tens of millions/sec.
Latency Like action sub-second; count freshness seconds.
Storage Trillions of (user, video) records over time.

The bottleneck is the viral video: one video receiving 100k likes per second pins any single counter row.

Core Product Model

Layer Purpose Requirement
Per-user like state Records (user, video, action, ts). Strongly consistent per user; the user-visible truth.
Per-video counter Denormalized total for display. Eventually consistent; approximate OK.
Fraud filter Removes inauthentic likes from public count. Async; runs before count is finalized for display.

Three reasons to separate state from counter: 1. The counter is hot; the per-user state is naturally sharded by user. 2. Toggling on/off is a per-user state change; counter must derive from current state. 3. Fraud filtering happens after the like is recorded but before the count is publicly visible.

Data Model

Per-User Like

UserLike(
  user_id,
  video_id,
  status,                // active, removed
  liked_at,
  removed_at,
  fraud_status,          // unknown, valid, suspicious, removed
  version
)

Partitioned by user_id. Read pattern: "did this user like this video?" → point lookup by (user_id, video_id). List pattern: "what videos did this user like?" → range scan within partition.

Video Counter

VideoCounter(
  video_id,
  count,                 // approximate, derived from valid likes
  raw_count,             // total likes including fraud-pending
  last_updated,
  version
)

Hot videos use sharded sub-counters: VideoCounter(video_id, shard_id, count); the display is the sum.

Like Event

LikeEvent(
  event_id,              // deterministic: hash(user_id, video_id, action, ts_window)
  user_id,
  video_id,
  action,                // like, unlike
  timestamp,
  source                 // client, server-side replay
)

Fraud Score

FraudScore(
  user_id,
  video_id,
  score,
  reasons[],
  decided_at
)

APIs

GET  /videos/{video_id}

GET  /users/{user_id}/liked-videos?cursor=

Write Path: Like Action

  1. Client POSTs /videos/{video_id}/like with action.
  2. API authenticates user; validates.
  3. Conditional update on UserLike(user_id, video_id): - If no record: insert with status=active. - If status=active and action is like: no-op (idempotent). - If status=active and action is unlike: update to removed, set removed_at. - If status=removed and action is like: update to active, set new liked_at.
  4. Emit LikeEvent to stream.
  5. Update user's local cache so they see their like immediately.
  6. Return success.

Latency: one DB write per like, plus async event emission. Sub-second easily.

Read Path: Display Likes On Video Page

  1. Client requests /videos/{video_id}.
  2. Read service queries: - VideoCounter(video_id) for the count (multi-tier cache; CDN, in-memory). - UserLike(user_id, video_id) for whether the current user has liked (only if user is authenticated).
  3. Merge: return { count, user_has_liked }.
  4. Display.

For unauthenticated users, only the count is fetched.

Read-Your-Own-Write

When the user taps like, the UI immediately shows their like before the server even responds:

  • Client optimistically updates UI on tap.
  • API call confirms.
  • On confirmation, the count displayed may not yet reflect their like (because the counter aggregator is async). The UI handles this by adding +1 to the displayed count locally if the user's like is active.

Server-side, the user's UserLike row is strongly consistent: if they refresh, they see user_has_liked = true immediately, and the displayed count includes their +1 client-side overlay until the counter aggregator catches up.

Interview phrase:

The user must see their own like instantly. The counter can lag. So we composite the displayed count from the eventually-consistent counter plus a client-side +1 for the user's own action.

Counter Aggregation

Counter updates are async to keep the like action fast and to absorb viral spikes.

Approach: stream processor consumes like events and updates per-video counters.

For normal videos:

  1. Consume event.
  2. Atomically increment or decrement video counter in counter store.

For viral (hot) videos:

  1. Hash by (video_id, random_shard) to a sub-counter.
  2. Atomically increment the sub-counter.
  3. Read path aggregates sub-counters: count = sum(sub_counter[i] for i in 0..N).

Sub-counter strategy:

  • Detect hot videos (incoming rate > threshold).
  • Promote them to use N sub-counters (e.g., 100).
  • Demote when traffic cools down.

Periodic full recount from per-user like DB serves as ground truth:

  • Background job per video aggregates all UserLike(video_id, status=active, fraud_status=valid) rows.
  • Overwrites counter with authoritative count.
  • Runs hourly or daily for high-traffic videos.

Fraud Filtering

Detecting inauthentic likes:

  • Click farms: many accounts liking the same video in short time.
  • Bots: synthetic accounts liking.
  • Reward fraud: users liking for incentives outside the platform.

Pipeline:

  1. Fraud service consumes like events.
  2. Per-user signals: account age, prior fraud, IP reputation, like rate.
  3. Per-video signals: sudden burst, geographic skew, follower overlap.
  4. ML scoring produces a fraud_score per (user, video).
  5. Scores above threshold mark the UserLike as fraud_status = suspicious.
  6. Counter aggregator only counts valid likes; suspicious likes are excluded.

Effect: the publicly displayed count reflects only validated likes. The user who placed the like may still see "you liked this" on their own UI even if fraud-flagged, depending on policy (silent removal vs explicit notification).

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Double-tap from client. Idempotent on (user_id, video_id, action) plus client debouncing. Duplicate like attempts.
Counter aggregator falls behind. Backpressure; auto-scale; periodic full recount. Aggregator lag; counter drift from ground truth.
Viral video pins one counter shard. Auto-promote to sub-counter sharding. Per-counter QPS; lag per video.
Counter drift from missed events. Daily full recount from UserLike. Drift % between counter and recount.
Stream loses events. Durable Kafka; idempotent consumers. Consumer lag; replay coverage.
Fraud filter false positives. Manual review queue; user appeal. Reversal rate per fraud reason.
Bot wave inflates count before fraud kicks in. Display delay for new videos; fraud filter latency monitoring. Suspicious like rate by video.
User's like not visible immediately. Strong read of UserLike; client-side +1 overlay. Read-your-write violations.
Storage explosion. Coldness-based tiering; archive old liked videos. Storage growth rate.

Operational Metrics

Correctness:

  • Counter vs recount drift per video.
  • Duplicate UserLike rows (must be zero).
  • Read-your-write failures (must be near zero).

Capacity:

  • Like QPS by region.
  • Aggregator lag by video.
  • Counter store p99 latency.

Business:

  • Likes per video distribution.
  • Fraud detection rate.
  • User engagement metrics tied to likes.

Multi-Region Strategy

  • UserLike partitioned by user; user's home region owns writes.
  • Like events replicate cross-region for counter aggregation in any region.
  • Counter store replicates eventually; reads from local region.
  • Cross-region eventual convergence within seconds is acceptable for displayed count.

For a global viral event, multi-region sub-counter sharding spreads the write load. Periodic recount reconciles across regions.

Tradeoffs To Call Out

  • Storing per-user like state at trillion scale is expensive but is the only way to support unlike and fraud removal correctly.
  • Sub-counter sharding for hot videos adds complexity; periodic recount keeps it honest.
  • Async counter update sacrifices freshness for performance; client-side +1 overlay hides the lag.
  • Aggressive fraud filtering may hide legitimate likes; appeal mechanism exists.
  • Strong user state is per-partition cheap; global counter strong consistency would not scale.
  • Dislike support roughly doubles state size; YouTube hides them publicly for that reason among others.

Common Traps And Stronger Moves

Trap Stronger move
Just incrementing a counter. Per-user state is the source of truth; counter is derived.
Synchronous counter update. Async via event stream; periodic recount for correctness.
Single counter row per video. Sub-counter sharding for hot videos.
Ignoring fraud. Async ML scoring; exclude suspicious from public count.
Forgetting read-your-write. Strong UserLike read; client-side +1 overlay.
Counting events not state. Toggling on/off makes event count meaningless; use current state.

Staff-Level Answer Checklist

  • Separate per-user state from per-video counter.
  • Per-user state is strongly consistent; counter is eventually consistent.
  • Sub-counter sharding for hot videos.
  • Async counter update via event stream.
  • Periodic full recount for correctness.
  • Read-your-write via strong state read + client overlay.
  • Fraud filtering as async pipeline excluding suspicious from public count.
  • Per-user partitioning for scale.

Interview Phrase

A likes counter is two things: per-user state and per-video count. The state is the truth; the count is a fast projection. Hot videos break naive counters; sub-counter sharding plus periodic recount keeps them honest. The user must see their own like instantly; the count can lag.

2. Simulated Interview

Interviewer: Design YouTube's likes counter.

Candidate: Two layers. First, per-user like state — a record per (user, video) with status active or removed. This is the source of truth; it's strongly consistent and partitioned by user. Second, a per-video counter that's updated asynchronously from like events. The counter is eventually consistent and exists for fast display. The hardest cases are hot videos and read-your-own-write.

Interviewer: Why not just increment a counter on like?

Candidate: Two reasons. First, likes are toggleable — unlike must reverse the count, which requires knowing the per-user state. If we only stored a counter, we couldn't tell whether a click is a new like or a re-tap on an existing like. Second, fraud filtering removes inauthentic likes from the public count; we need per-user records to identify and exclude them. The counter is derived from the per-user state, not the other way around.

Interviewer: A video goes viral and gets 100,000 likes per second. What happens to the counter?

Candidate: A single counter row would be pinned and bottlenecked. We auto-detect hot videos (incoming rate above threshold) and promote them to sub-counter sharding: split into, say, 100 sub-counters. Each like event hashes to a random sub-counter for atomic increment. The displayed count sums all sub-counters. When the rate cools, we demote and consolidate.

Interviewer: How do you keep the counter from drifting?

Candidate: A periodic recount job reads the per-user state. For each video, count UserLike rows with status=active, fraud_status=valid. Overwrite the counter. Runs hourly for high-traffic videos, daily for everything else. This is the ground truth. The async event-driven counter is an approximation; the recount keeps it honest.

Interviewer: When the user taps like, they should see their like immediately. How?

Candidate: Two pieces. Server side: the UserLike state is written synchronously in the like request and is strongly consistent. So if the user refreshes, the user_has_liked field is correct. Client side: the UI optimistically updates count = count + 1 and you_liked = true on tap. This bridges the gap between the synchronous state write and the async counter update. The user perceives instant feedback even though the global counter takes seconds to catch up.

Interviewer: Double-tap — how do you handle idempotency?

Candidate: The like operation is a conditional state transition: if status is already active and action is like, it's a no-op. The DB write doesn't change anything. If the client sends three taps in a second, only the first matters; the others are no-ops on the server. Clients also debounce locally.

Interviewer: Fraud — how is it detected and applied?

Candidate: Async fraud pipeline consumes the event stream. It scores likes using ML on signals like account age, IP reputation, geographic anomalies, follower overlap with the video creator, like rate per account. Scores above threshold flag the UserLike row's fraud_status as suspicious. The counter aggregator and the periodic recount only count fraud_status=valid likes. So the public count reflects only validated likes. The user who placed the like may still see their own action in their history.

Interviewer: Storage — trillions of UserLike rows.

Candidate: Partition by user_id. The natural query patterns are (a) "did user X like video Y" — point lookup within user's partition; (b) "what videos has user X liked" — range scan within user's partition. Both are cheap. The expensive query — "who liked this video" — happens during fraud analysis and periodic recount, where we tolerate large scans. We can maintain a secondary index by video_id for hot videos if needed, but it's not the default.

Interviewer: Multi-region?

Candidate: User's home region owns their UserLike. Like events replicate cross-region for counter aggregation, which can happen anywhere. Counter store replicates eventually; reads come from local region. For global viral events, sub-counter sharding spans regions naturally — each region updates its local sub-counter, and aggregation sums them.

Interviewer: How fresh is the count?

Candidate: Normal videos: a few seconds lag from the async aggregator. Viral videos: depends on aggregator throughput and sub-counter shard count. Periodic recount catches up to actual within minutes to hours. For UX, the user sees their own like instantly via client overlay; the displayed total may show a small lag for everyone else, which is unnoticeable for billions but tolerable for viral spikes.

Interviewer: What if a user is shadowbanned? Their likes should not count.

Candidate: Their UserLike rows have fraud_status=suspicious (or a separate flag). The counter aggregator and recount exclude them. From the user's perspective, their like appears (they see "you liked this"); from the world's, the count doesn't change. This is policy-driven.

Interviewer: Summarize.

Candidate: Per-user state is the source of truth; per-video counter is a derived projection. State is strongly consistent and partitioned by user; counter is eventually consistent and sharded for hot videos. Async aggregation via event stream; periodic recount for correctness. Read-your-write via strong state plus client overlay. Async fraud pipeline excludes suspicious likes from the public count. The hardest case is viral hot videos, handled by sub-counter sharding plus periodic recount.

3. Practice Notes

Five-Minute Version

  1. Clarify: toggleable, one per user, viral hot videos, fraud.
  2. Invariant: per-user state authoritative; counter derived.
  3. Per-user state + per-video counter split.
  4. Sub-counter sharding for hot videos.
  5. Async counter update via event stream.
  6. Periodic recount for correctness.
  7. Read-your-write via strong state + client overlay.
  8. Async fraud filter; exclude suspicious from public count.

One-Minute Closeout

A likes counter is per-user state plus a derived counter. The state is the truth, strongly consistent and per-user partitioned; the counter is a fast eventually-consistent projection. Hot videos use sub-counter sharding; periodic recount keeps the counter honest. The user sees their own like instantly via strong state read plus client-side overlay; the displayed total catches up within seconds. Fraud filtering is async and excludes suspicious likes from the public count.
