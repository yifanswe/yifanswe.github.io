Instagram — Photo Sharing + Feed (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build Instagram. Users upload photos and short videos, follow other users, and scroll a ranked feed of posts from people they follow. 500M daily active users (DAU), 100M new posts/day, average user opens the app 8× a day. Media-heavy + ranking-heavy.

1. Requirements (~5 min)

Functional - Post photo/short video. - Follow other users. - Scroll a ranked feed of recent posts from people you follow. - Like and comment on posts.

I'd defer Stories, Reels (algorithmic discovery), Direct Messages, and Shopping.

Non-functional - Feed open p99 < 500 ms. - Media must serve globally with low latency — CDN-fronted. - Eventual consistency on feed and engagement counts. - Asymmetric: a few accounts have 100M+ followers (celebrity problem). - Read-heavy: ~100:1 reads to writes.

Capacity - 100M posts/day; avg 2 MB per post (after compression / multiple sizes) → 200 TB/day raw → ~73 PB/year. - 500M DAU × 8 opens × 20 posts = 80B post views/day → ~1M QPS at peak. - 1B follow relationships. - Dominant cost is media: even post metadata at ~1 KB × 100M/day × RF=3 is ~300 GB/day, while media is ~200 TB/day before replication/tiering. Feed Redis is large but bounded: 500M users × 500 post IDs × ~32 B × RF=2 is ~16 TB plus overhead.

2. Core Entities (~2 min)

  • User — user_id, follows.
  • Post — post_id, author_id, media_refs[], caption, created_at, like_count, comment_count.
  • Media — media_id, url_variants{sizes}, format, owner.
  • Edge — (follower_id, followee_id).
  • FeedEntry — (user_id, post_id, score).

3. API (~5 min)

POST  /v1/media/upload                     → { media_id, upload_url }
POST  /v1/posts                            { media_ids[], caption }   → { post_id }
GET   /v1/feed?cursor=...&limit=20         → { posts[], next_cursor }
POST  /v1/posts/{id}/like
POST  /v1/posts/{id}/comments              { text }
POST  /v1/follows                          { user_id }

Two-phase upload — request signed URL, upload to object store directly, then commit.

Feed cursors are opaque and encode {user_id, rank_session_id, last_score, last_post_id}. A rank session stores the candidate set for a few minutes so page 2 does not reshuffle under the user.

5. High Level Design (~10–15 min)

  1. Upload path. - Client requests upload URL → media service signs an S3 URL. - Client uploads directly to S3. - On commit, media service kicks off image-processing pipeline: generate sizes (thumbnail, medium, large), optionally HEIF→JPEG, EXIF strip. For videos: transcode to multiple bitrates. - Post is written to the post catalog (Cassandra, sharded by post_id) with refs to media variants. Publish post.created event.
  2. Feed assembly. Same hybrid push/pull as the FB News Feed problem: - Push fan-out for normal users (followers ≤ 10k): on post.created, append post_id to each follower's Redis feed list. - Pull-on-read for celebrities (followers > 10k): when assembling feed, also fetch recent celebrity posts from per-celebrity caches. - Merge + rank at read time.
  3. Ranking. Online model service. Features: recency, author affinity, post engagement velocity, media type, content category. Reranks the candidate set.
  4. Media serving. CDN (CloudFront/Akamai) in front of S3. Aggressive caching by URL path; signed URLs for private accounts. Adaptive resolution served based on device class.
  5. Engagement pipeline. Likes/comments → Kafka → counter aggregation (Redis for live counts, ClickHouse for analytics) + ranking signal updates.
  6. Storage hierarchy. Recent posts and metadata hot (Cassandra + Redis caching). Old media in cheaper tiers (S3 Glacier-style) but still accessible via CDN with cache miss penalty.

End-to-end narration: Alice posts a photo → upload to S3, commit triggers processing → media variants generated → post row in Cassandra → fan-out pushes post_id into the Redis feed lists of her 200 friends. Bob opens the app → feed assembly pulls Bob's feed list (300 candidate post_ids), merges in 5 celebrity-posts candidates → fetches post metadata in a batch → ranking service reranks → returns top 20. Bob sees Alice's photo, served from a CDN edge.

6. Deep Dives (~10 min)

Media pipeline. Upload → S3 → SQS notification → image worker → generates 4 sizes (thumb 150², medium 600², large 1080², detail 2160²) and a CDN-warm preview. For videos: transcode to 480p/720p/1080p H.264, generate poster image, MP4 + HLS. Pipeline is async; the post can publish before full processing (use a fallback to the original until variants ready).

Why hybrid feed. Pure push breaks on celebs (Kim Kardashian post → 200M Redis writes). Pure pull breaks on average users (each feed open fans out across 300 friends' recent posts). Hybrid uses push for the 99.9% and pull for the 0.1% with the huge follower counts. Cutoff is a config knob.

Why Redis for feed IDs rather than Cassandra-only: Cassandra can store durable timelines, but feed open needs a small, hot, ordered candidate set in <50 ms. Redis stores IDs only; Cassandra remains the durable post catalog. If Redis is lost, rebuild from author timelines and recent post.created events.

Feed list maintenance. Redis sorted set per user, capped at last few hundred entries with a sliding TTL. On a new follow, don't backfill historical posts (too expensive). Instead, assembly augments with read-time pulls for new follows for the first few days.

Feed write path: post.created -> fan-out worker reads followers:{author_id}; for normal authors it batches ZADD feed:{follower_id} score post_id and trims with ZREMRANGEBYRANK; for celebrity authors it only writes author_posts:{author_id} and invalidates celebrity caches. The post catalog write and media processing are already committed before fan-out begins.

Feed read path: client -> feed service reads feed:{user_id} and following_celeb:{user_id} in parallel; pulls recent celebrity posts from author_posts:{author_id}; batch-fetches post metadata, counters, viewer like state, and media variants in parallel; ranking service reranks the candidate list; cursor stores the rank session.

Hot feed page trace: 1. Client calls GET /v1/feed?cursor=C; the feed service decodes {user_id, rank_session_id, last_score, last_post_id}. 2. It reads feed:{user_id} from Redis using user_id as the shard key and, in parallel, reads following_celeb:{user_id} plus a cached viewer feature vector. 3. For celebrity followees, it pulls recent ids from author_posts:{author_id} keyed by each celebrity author_id. 4. It merges 300-800 candidate post IDs, drops ids already stored in the rank session, and batch-fetches post:{post_id}, counter:{post_id}:*, viewer-like state, and media variants in parallel. 5. Ranking runs as one batch call; the service returns 20 hydrated posts and updates the rank session so page 2 is stable.

Cold feed rebuild path: if feed:{user_id} is missing or exhausted, the service reads following:{user_id}, pulls the last N ids from each normal author's author_posts:{author_id} with a tight cap, merges with celebrity pulls, writes a fresh feed:{user_id} asynchronously, and serves a smaller page if the rebuild exceeds ~700 ms.

Why the main stores: Cassandra/DynamoDB-style storage is a better fit for post:{post_id} than a single relational database because the workload is append-heavy, globally sharded, and mostly point reads. Redis is used only for bounded ID indexes; putting full posts there would multiply the media-adjacent metadata cost. Kafka separates upload commit from fan-out so a slow follower shard does not make posting slow. Be ready for "why not pure pull?" -- pure pull burns read latency for average users following hundreds of accounts.

Ranking model serving. Pre-fetch features into a feature store (user_features, post_features). On feed open, parallel calls: feed list fetch + feature fetch + media metadata fetch. Single batch inference call. Total budget < 200 ms server-side.

CDN strategy. Multi-CDN to handle outages. Cache key includes size variant. TTL ~1 day for media (immutable URLs with content hash). Invalidation: when a post is deleted, purge its URLs.

Celebrity reads at scale. Per-celebrity "recent posts" Redis hash + per-feature precomputed in-process cache on assembly nodes (since the same celebrities show up in millions of feeds). Update on each new celebrity post.

Engagement count consistency. Approximate count from Redis incremented on each like; reconciled hourly from the Kafka-derived rollup in ClickHouse. Users never notice the slight inaccuracy mid-hour.

Storage cost. Photos are mostly cold after 30 days. Tier to S3 Infrequent Access after 30 d, Glacier after 1 y. Rehydrate on access — first hit pays the latency, subsequent hits CDN-cached.

Hot post bottleneck. Viral post → 10M views in an hour. CDN absorbs the media reads; post metadata is small, easily cached at the API gateway. The risk is the engagement counter — use multiple shard keys and sum on read.

Two-index framing. There is one post dataset and two indexes over it. feed:{user_id} is reader-keyed and optimized for normal authors; author_posts:{author_id} is writer-keyed and optimized for celebrities, new follows, and feed rebuilds. The read path merges both views and then ranks, so the product still feels like one feed.

Edge stress tests. Dormant users may have no warm feed list; they get a pull-based rebuild from author timelines with a slower first page. Users who paginate beyond the 500-id Redis window switch to author-timeline pull plus a new cursor rather than growing Redis unbounded. Burst writes from a celebrity never fan out to all followers; only the author timeline and celebrity cache update. Multi-region drift is acceptable for feed freshness, but post creation should write the catalog in the author's home region and replicate before cross-region ranking uses it. Cold media variants fall back to the original object while workers finish transcoding.

Graceful degradation.

Case Mode Source User-visible behavior
Feed cache cold Pull from author timelines and rebuild async follow graph + author posts slower first feed open
Celebrity fan-out Pull-on-read, no follower writes celebrity recent cache fresh within seconds
Ranking service down Recency + affinity fallback rank feed index + cached features less personalized feed
Media variant missing Serve lower-res/original variant object store/CDN quality drops, post still loads
Counter hot key Sharded counters, approximate sum Redis shards + rollup counts may lag

Interview punch line: "The feed is two indexes over the same posts: reader-keyed for normal authors and author-keyed for celebrities; the read path merges them into one ranked experience."
