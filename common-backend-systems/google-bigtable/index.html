1. Basic Knowledge And Solution

What This Question Is Testing

Bigtable (the 2006 paper) is the canonical column-family distributed store. The hard part is partitioning a sorted-by-row-key table across thousands of servers, supporting fast random reads and sequential scans, while maintaining strong per-row consistency.

The invariant is this:

Every row is owned by exactly one tablet at any moment; reads and writes for a row go to its current tablet server; tablets split and migrate without losing data or duplicating ownership; the underlying GFS storage provides durability; Chubby coordinates ownership.

At ICT5 / L6+ level, the interviewer is testing whether you can articulate the row-key-sharded tablet model, the SSTable + commit log + GFS storage, tablet server architecture, Chubby's role in coordination, and the column-family data model.

60-Second Interview Framing

Spoken version:

I would describe Bigtable as a sparse, distributed, persistent multi-dimensional sorted map keyed by row+column+timestamp. The table is partitioned into tablets — contiguous row ranges — each owned by one tablet server. Tablets are stored as SSTables in GFS for durability. A master assigns tablets to servers; Chubby coordinates membership and exclusive ownership via leases. The interesting decisions are tablet ownership semantics, SSTable storage on GFS, locality groups, and Chubby's role.

Requirements The Paper Sets

Functional requirements:

  • Multi-dimensional sorted map: (row_key, column_family:column_qualifier, timestamp) → value.
  • Row-key range queries.
  • Per-row atomic updates (read-modify-write within a row).
  • Versioning by timestamp.
  • Column families with shared compression.

Non-functional requirements:

  • Petabytes of data across thousands of servers.
  • Low-latency random reads/writes.
  • High throughput sequential scans.
  • Strong per-row consistency.
  • Distributed but single-region in original paper.

Scale Assumptions

Area Assumption
Data Petabytes per table.
Servers Thousands of tablet servers.
Tablets per table Millions.
Tablet size ~100-200 MB before split.
Cells per row Up to billions (though usually fewer).

Core Concepts

Data Model

Table
  └── Row (uniquely identified by row_key, sorted lex)
       └── Column Family
            └── Column Qualifier
                 └── Timestamp
                      └── Value

Each cell can hold multiple timestamped versions; old versions GC'd per policy.

Example (Webtable):

row_key:        com.google.www
column family:  contents
column quals:   t1, t2, t3 (different timestamps)
value:          HTML content at each time

Row keys sort lexicographically. Reversing the URL ("com.google.www") groups related URLs together, enabling efficient prefix scans.

Interview phrase:

Row key design is the single biggest decision in Bigtable. Lexicographic sorting means related rows live in the same tablet — efficient scans, but careful to avoid hot tablets from sequential keys.

Column Families

Defined at table creation. Within a family, columns share: - Storage (one SSTable per family on disk). - Compression. - Access controls.

Column qualifiers within a family are dynamic — you can add new ones at write time without schema change.

This design separates frequently-accessed columns from rarely-accessed ones at the storage layer.

Tablets

A tablet is a contiguous range of rows: [start_row, end_row).

  • Initially one tablet per table.
  • When tablet grows beyond threshold (~100-200 MB), it splits into two.
  • Each tablet is owned by exactly one tablet server.
  • Tablets are the unit of distribution.

A tablet is composed of: - A commit log on GFS. - A memtable (in-memory recent writes). - Multiple SSTables on GFS.

SSTables

Persistent, sorted, immutable string-to-string maps. Stored as files on GFS.

  • Indexed by row key + column family + qualifier + timestamp.
  • Block-organized; bloom filters for fast negative lookups.
  • Immutable after creation; compaction creates new SSTables.

Compaction

  • Minor compaction: memtable flushed to a new SSTable on disk.
  • Merging compaction: merge a few SSTables to reduce count.
  • Major compaction: merge all SSTables into one; removes tombstones.

Core Components

Component Responsibility Key detail
Master Assigns tablets to tablet servers; load balancing; schema changes. One per cluster.
Tablet server Owns a set of tablets; serves reads/writes. Many.
GFS Stores SSTables and commit logs. Underlying durable storage.
Chubby Stores master/tablet-server liveness; root tablet location; schemas. Single coordination service.
Client library Caches tablet location; talks directly to tablet servers. Heavy caching.
Compaction Background SSTable merging. Per tablet.

Tablet Location Hierarchy

To locate the tablet owning a given row key, Bigtable uses a three-level hierarchy:

Chubby file -> root tablet -> METADATA tablets -> user tablets
  • Chubby file contains location of root tablet.
  • Root tablet (special, never splits) stores locations of METADATA tablets.
  • METADATA tablets store locations of all user tablets.

Clients cache aggressively; on cache miss, walk down the hierarchy. Three-level lookup, but cached after first.

Write Path

  1. Client sends mutation to the appropriate tablet server (cached lookup).
  2. Tablet server writes to commit log on GFS (durability).
  3. Inserts into memtable (in-memory sorted map).
  4. Returns success.

Memtable flush (minor compaction): - When memtable reaches size threshold, flush to a new SSTable on GFS. - Truncate commit log up to the flushed point.

Read Path

  1. Client sends read to tablet server.
  2. Tablet server reads from memtable + all relevant SSTables.
  3. Merges results (latest timestamp per cell).
  4. Applies any visible tombstones.
  5. Returns rows.

Bloom filters skip SSTables that don't contain the queried keys.

For range scans: sequential read of SSTables in key order, merged.

Tablet Split

When a tablet grows beyond threshold:

  1. Tablet server picks a split point (median row key).
  2. Updates METADATA to reflect two new tablet ranges.
  3. Original SSTables still apply to both (they're indexed by row range).
  4. Client cache invalidates on next access.

Split is fast — just a metadata change. Physical data isn't copied; SSTables are shared until compaction.

Tablet Reassignment

When a tablet server fails, the master detects (via Chubby) and reassigns tablets to other servers.

  1. Master detects failed tablet server (Chubby lease expiry).
  2. For each tablet on failed server, master assigns to a new server.
  3. New server reads the tablet's commit log + SSTables from GFS.
  4. Replays uncommitted log entries into a new memtable.
  5. Ready to serve.

Chubby's role: ensures only one master, ensures only one server owns a tablet at a time.

Chubby's Role

  • Master election (only one master).
  • Tablet server membership (each server holds a unique Chubby lock; dropping the lock = dead).
  • Schema storage.
  • Root tablet location.

Without Chubby, Bigtable can't operate.

Locality Groups

A locality group is a set of column families stored together on disk.

  • Default: each column family is its own locality group.
  • Frequently-co-accessed families can share a locality group for less disk IO.
  • Locality groups can be memory-loaded for very fast access.

Compression

Per-column-family compression. Two-stage: 1. First, a key-encoding step that exploits sorted nature. 2. Then a content-encoding step (Bentley-McIlroy, Snappy, etc.).

Compression ratios of 10:1 common.

Single-Row Transactions

Bigtable supports atomic read-modify-write within a single row:

ApplyMutation(row_key, mutations, condition)

Tablet server holds a lock on the row, applies mutations atomically. No cross-row atomicity.

Multi-Row Transactions

Not native. Layered services (Megastore, Spanner) added this on top with extra coordination.

Consistency Model

Layer Consistency Reason
Per-row writes Strong (tablet server serializes) Single owner.
Cross-row None (no distributed transactions) Design choice.
GFS storage Strong (replicated 3x) Underlying.
Tablet ownership Strong via Chubby At-most-one.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Tablet server fails. Master reassigns; new server loads from GFS. Reassign time.
Master fails. Chubby election picks new master. Election time.
Chubby outage. Bigtable unavailable for new ops; existing leases run out. Chubby availability.
GFS partial failure. GFS handles replication; Bigtable continues. GFS health.
Hot tablet. Auto-split; app-level sharding. Per-tablet QPS.
Compaction storm. Throttle; pace. Pending compactions.
Split race. METADATA write coordinated. Conflict count.
Lease expiry mid-op. Detect; abort; retry. Lease expiry events.
Schema change races. Chubby coordinates. Schema version.

Operational Metrics

Correctness:

  • Tablet ownership uniqueness (zero double-owners).
  • Lost writes (zero).

Capacity:

  • Tablets per server.
  • Pending compactions.
  • GFS bandwidth.

Business:

  • Read/write QPS per table.
  • Latency.

Tradeoffs To Call Out

  • Strong per-row consistency, no multi-row transactions: clean design choice.
  • Row-key design determines everything; sequential keys risk hot tablets.
  • Column families separate storage but require upfront schema.
  • Compaction is necessary but expensive; tunable.
  • Three-level location lookup adds initial latency; aggressive client caching mitigates.
  • Single master is a SPOF in lite versions; failover via Chubby.
  • Relies on GFS and Chubby — both must work for Bigtable to work.

Common Traps And Stronger Moves

Trap Stronger move
Sequential row keys for time-series. Hash prefix or reverse-time to avoid hot tablets.
One giant column family. Locality groups per access pattern.
Multi-row transactions in app. Acknowledge limitation; layer Megastore/Spanner.
Forget compaction. Major compaction periodically; tombstone cleanup.
Hard-coded tablet locations. Use METADATA hierarchy with client cache.
Single Chubby file. Multiple lock files per cluster.

2. Simulated Interview

Interviewer: Describe Bigtable.

Candidate: Distributed sorted map keyed by (row, column, timestamp). Sparse, persistent, multi-dimensional. Tables partition into tablets — contiguous row ranges, ~100-200 MB each. Each tablet owned by one tablet server. Storage is SSTables on GFS. Commit logs ensure durability. Chubby coordinates master election and tablet ownership. Per-row writes are strongly consistent; no cross-row transactions.

Interviewer: Walk me through a write.

Candidate: Client sends mutation to the tablet server owning the row's tablet (looked up via METADATA hierarchy, cached). Tablet server writes to commit log on GFS (durability), inserts into in-memory memtable (sorted). Returns success. Memtable flushes to a new SSTable when full (minor compaction). Commit log is truncated up to flushed point. Multiple SSTables exist per tablet over time; compaction merges them.

Interviewer: Read path.

Candidate: Read goes to the tablet server. Server reads from memtable + relevant SSTables (Bloom filter avoids SSTables that don't have the key). Merges all data by timestamp (latest wins per cell). Applies tombstones. Returns rows. For range queries, sequential read in row order; merge across SSTables maintains sorted output.

Interviewer: Tablet split.

Candidate: When a tablet's data exceeds threshold (~200 MB), the tablet server picks a split point — typically the median row key. Updates METADATA: the original tablet range splits into two. SSTables aren't physically copied; they're indexed by row range and apply to both halves until compaction. Split is fast — just a metadata operation.

Interviewer: Tablet server fails.

Candidate: Tablet server holds a Chubby lock. When it crashes or its lease expires, the lock releases. Master detects this. For each tablet on the failed server, master assigns to another tablet server. The new server reads the tablet's commit log + SSTables from GFS, replays unflushed log entries into a new memtable, and starts serving. The whole takeover happens in seconds; GFS provides durability so no data is lost.

Interviewer: Chubby's role.

Candidate: Coordination only — not in the data path. Master election: only one master at a time. Tablet server membership: each holds a unique Chubby lock. Lock loss = server dead. Schema storage. Root tablet location (the entry point of the METADATA hierarchy). Bigtable can't operate without Chubby.

Interviewer: Row key design.

Candidate: The single biggest decision. Row keys sort lexicographically; related rows live in the same or adjacent tablets, enabling efficient range scans. Webtable famously uses reversed URLs ("com.google.www") to group related sites. But sequential keys (like timestamps) create hot tablets — all writes go to the same tablet server. Fix: hash prefix or reverse-time to distribute.

Interviewer: Column families.

Candidate: Defined at table creation. Within a family, columns share storage (one SSTable per family per tablet), compression, and access control. Column qualifiers within a family are dynamic — added at write time without schema change. Separates frequently from rarely accessed columns at the storage layer. Locality groups can collocate multiple families for shared disk IO.

Interviewer: Compaction.

Candidate: Three types. Minor compaction: memtable flushes to a new SSTable. Merging compaction: combine several SSTables into one to reduce count. Major compaction: merge all SSTables into one, removing tombstones. Trade-off: too few SSTables = more disk usage but better reads; too many = slower reads. Tuned per workload.

Interviewer: No cross-row transactions?

Candidate: By design. Layered services (Megastore, Spanner) provide them via Paxos / Two-Phase Commit on top. Bigtable keeps the per-tablet design simple. For most use cases, single-row atomicity is enough; apps that need more layer it on.

Interviewer: Locality groups.

Candidate: A locality group is a set of column families stored together. Default: one group per family. Configurable: group families that are typically accessed together — e.g., page metadata and contents — to reduce disk IO. Memory-loaded groups stay in RAM for very fast access.

Interviewer: Compression.

Candidate: Per column family or per locality group. Two-stage: key encoding exploiting sorted keys, then content compression (Bentley-McIlroy + Snappy or similar). Compression ratios of 10:1 are common, especially for highly-redundant data like web pages.

Interviewer: Summarize.

Candidate: Bigtable is a row-key-sharded distributed sorted map. Tablets partition the table; each owned by one tablet server. SSTables on GFS provide durable storage; commit log + memtable architecture handles writes; compaction maintains read performance. Chubby coordinates ownership. Per-row strong consistency; no multi-row transactions natively. Row-key design is critical. The strengths are linear scale and efficient range queries; the limitations are no cross-row transactions and hot-tablet risk from poor row-key design.

3. Practice Notes

Five-Minute Version

  1. Sparse multi-dimensional map: (row, col_fam:qual, ts).
  2. Tablets (row ranges) partition the table.
  3. One tablet server per tablet.
  4. SSTables on GFS; commit log + memtable.
  5. Compaction (minor / merging / major).
  6. Chubby for coordination; master assigns tablets.
  7. Column families + locality groups for storage layout.
  8. Per-row strong; no cross-row transactions.

One-Minute Closeout

Bigtable is a row-key-sharded distributed sorted map. Tablets are contiguous row ranges, each owned by one tablet server. Storage is SSTables on GFS; commit logs ensure durability; memtables hold recent writes; compaction merges SSTables. Chubby coordinates master election and tablet ownership. Per-row strong consistency; no multi-row transactions natively. Row-key design determines performance — sequential keys risk hot tablets.
