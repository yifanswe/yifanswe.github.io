Bitly — URL Shortener (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Easy · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a Bitly-style URL shortener. Anyone pastes a long URL, gets back a short code, and anyone who visits that short code is 302-redirected to the original. Owners can see basic click counts. Assume 100M new links/month and a 100:1 read-to-write ratio.

1. Requirements (~5 min)

Functional — "Users should be able to..." - Shorten a long URL into a short URL, optionally with a custom alias. - Visit a short URL and be redirected to the original. - See click count and recent referrers for links they own.

I'd explicitly drop expiring links and full analytics dashboards to focus on the redirect path.

Non-functional — "The system should..." - Redirect with p99 < 100 ms; every extra millisecond is felt on click. - Stay available for redirects even during write outages — once a link exists, it must work. - Be durable for the short-code-to-URL mapping; a lost mapping is unrecoverable from the user's side. - Make codes unique and non-guessable enough that random scanning is uneconomical.

Capacity (only because it shapes sharding): - 100M links/mo × 60 mo = ~6B links. Row ≈ 200 bytes. Total ~1.2 TB metadata — fits sharded MySQL. - 100:1 read ratio → ~4k redirects/sec average, ~40k/sec peak. - 7-char base62 = 62⁷ ≈ 3.5 trillion codes — plenty of room. - Click events dominate write volume: ~400k/sec peak if every redirect emits analytics. Keep them out of MySQL; Kafka + rollups absorb that path.

2. Core Entities (~2 min)

  • User — owns links, authenticates writes.
  • Link — short_code, long_url, owner_id, created_at, custom?.
  • ClickEvent — short_code, ts, ip, ua, referrer. High volume, lives outside the main store.

3. API (~5 min)

POST /v1/links              { long_url, custom_alias? }  → { short_url }
GET  /{short_code}          → 302 Location: long_url
GET  /v1/links/{short_code}/stats → { count_24h, count_total, recent[] }

Auth via Bearer token; owner_id derived from the token, never from the body. The redirect endpoint is the only unauthenticated path. Plural links resource for collection routes.

POST /v1/links accepts an Idempotency-Key so a retry after timeout returns the same {short_url} rather than burning a second code. Stats pagination uses a cursor like {short_code, bucket_start, last_event_id}.

5. High Level Design (~10–15 min)

Boxes I'd draw, in order:

  1. Client → API Gateway — TLS termination, auth, per-IP rate limit on writes.
  2. Write service — validates URL, generates a short code, writes (short_code, long_url, owner_id) to the metadata store. Returns short URL.
  3. Read service (redirect path) — GET /{code} → Redis lookup → fall through to metadata store on miss → cache → return 302. Fires a click event to Kafka before responding (or as the response is written).
  4. Metadata store — sharded MySQL keyed by short_code (consistent-hash on the code). Single-row lookups, no joins.
  5. Cache — Redis cluster, write-through, LRU eviction. Hot codes stay in memory.
  6. Click pipeline — Kafka → stream processor (Flink) → rollups in ClickHouse (per-minute, per-hour, per-day buckets keyed by short_code).
  7. Stats service — reads ClickHouse rollups for the /stats endpoint.

I'd walk the interviewer through one write and one read end-to-end so the data path is concrete.

Write path:

  1. API gateway authenticates, rate-limits by owner_id, and validates long_url.
  2. Write service reserves a code from its local ID block, or reserves alias:{custom} with SETNX for custom aliases.
  3. It inserts one metadata row keyed by short_code; on duplicate custom alias, return 409.
  4. It writes code:{short_code} into Redis with the long URL and owner fields, then returns the short URL.

Redirect path:

  1. Read service parses /{short_code} and checks the edge bloom filter for definitely-missing codes.
  2. It does GET code:{short_code} in Redis. Hit returns a 302 immediately.
  3. On miss, it does a single-row lookup in MySQL shard hash(short_code), repopulates Redis, and returns 302.
  4. In parallel with the response write, it publishes ClickEvent to Kafka and increments live_clicks:{short_code}:{minute} in Redis. Analytics failure must not block the redirect.

Stats path:

  1. Stats service authenticates owner and checks ownership via MySQL or a cached owner:{owner_id}:codes index.
  2. It queries ClickHouse rollups for closed buckets keyed by (short_code, minute).
  3. In parallel, it reads Redis live_clicks:{short_code}:{minute} for the current minute.
  4. It returns an opaque cursor {short_code, bucket_start, bucket_end, last_bucket, freshness_cutoff}. The cutoff freezes the result set for pagination while new clicks arrive.
  5. If ClickHouse is behind, the response includes closed buckets through the last committed watermark plus a provisional live count from Redis.

6. Deep Dives (~10 min)

Short code generation. Three options: - Random base62 + collision check on insert. Simple but every write costs an existence check. - Hash(long_url) truncated. Deterministic dedupe but you still need collision probing. - Counter + base62, per-shard ID ranges. Each writer pulls a 10k-id block from a coordinator and converts to base62. No collision, no extra read on the write path. I'd default to this and accept that codes leak rough creation order.

Hot redirect path. Goal is to never touch MySQL for popular codes. Redis is the front line; a small bloom filter at the edge short-circuits known-bad codes. For codes going viral right now, I'd let the CDN cache the 302 for 30–60s — a stale redirect is fine, a stale destination is the worst case but I'd document the tradeoff.

Concrete stores:

Alias reservation alias:{alias} owner_id / request_id ~64 B 5 s while creating hash(alias)
Create idempotency idem:{owner_id}:{key} short_code, status ~100 B 24 h owner_id
Owner link index owner:{owner_id}:codes paged list of short_codes ~12 B/code forever owner_id
Bad-code bloom daily filter membership bits for known codes ~10 bits/code 24 h rebuild edge-local
Click stream clicks[{partition}] append-only ClickEvent ~150 B 7 days hot hash(short_code)
Rollups (short_code, minute) count, referrer sketch ~80 B 2 years hash(short_code)

Metadata capacity is modest: 6B links x 500 B x 3 replicas ~= 9 TB, still ordinary sharded MySQL. The owner index is 6B codes x 12 B x 3 replicas ~= 216 GB before page overhead, small enough to keep beside metadata or in a separate index table. Redirect cache for the hottest 1B links at ~500 B is ~500 GB before Redis overhead, maybe 1-2 TB replicated. Click analytics is the larger moving stream: 40k/sec peak x 150 B ~= 6 MB/sec peak, with Kafka 7-day x3 retention under ~11 TB before compression. Rollups at 40k/sec collapsed into (short_code, minute) buckets are much smaller; the cost knob is how much referrer/user-agent detail we keep.

The two-index pattern is simple but important: the canonical Link row is addressed by short_code for redirects, while the owner dashboard needs owner_id -> short_codes. I would keep the redirect row as the source of truth and maintain the owner index transactionally on create, because listing owned links should never add a join to the redirect path.

Interview punch line: the read path has two jobs with different guarantees: redirect must be fast and durable, while analytics can lag and be rebuilt from the click log.

Click analytics without writing per click to MySQL. Every redirect fires an event to Kafka and increments a Redis counter for the live UI. Flink rolls up to ClickHouse on a tumbling 1-min window. The /stats endpoint only ever reads ClickHouse. MySQL never sees click traffic.

Why MySQL + Redis. The metadata access pattern is a primary-key lookup and a rare insert; sharded MySQL gives durable rows, unique constraints for aliases, and simple backups. DynamoDB/Cassandra also work, but we do not need wide rows or multi-region active-active unless the interview pushes global writes. Redis beats an in-process cache because redirect traffic is shared across many stateless read-service nodes; CDN caching is an extra edge optimization for ultra-hot codes, not the source of truth.

Custom-alias contention. Two users racing for bit.ly/launch: use SETNX on alias:launch with a 5-second TTL to reserve before the MySQL insert. First writer wins; second gets a clean 409.

Abuse and safety. Token-bucket rate limit per owner_id on writes (Redis). After insert, push the URL through Google Safe Browsing asynchronously; if flagged, soft-delete and serve a warning interstitial instead of the 302.

What I'd defer if I had more time. Multi-region active-active for the redirect path, per-link TTLs, geo-aware analytics, and link previews. I'd mention them so the interviewer knows I see them.

Interview punch line: the redirect is a tiny primary-key read, so protect that path from everything else. Link metadata is durable and boring; analytics is high-volume and lossy enough to be streamed and reconciled separately.
