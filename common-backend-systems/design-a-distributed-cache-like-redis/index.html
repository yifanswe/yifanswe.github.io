1. Basic Knowledge And Solution

What This Question Is Testing

A distributed cache sits between the application and its source of truth — typically a slower datastore (database, object store, downstream service) — and absorbs the read load. The hard part is not the in-memory hash table; it is keeping a large pool of memory across many machines coordinated, evicting the right things, surviving failures, and shielding the source of truth from correlated traffic spikes.

The invariant is this:

For any cached key, a read returns either a value that was valid at some recent point in time (the freshness contract) or a clean miss that triggers exactly one origin fetch per stampede window, even when nodes fail, keys are hot, and clients retry aggressively.

At Industry Career Track Level 5 (ICT5) or Senior Staff equivalent (L6+), the interviewer is testing whether you can pick a partitioning scheme, an eviction policy, a replication and failover story, and — most importantly — explain what happens to the origin when the cache misses, fails, or restarts.

60-Second Interview Framing

Spoken version:

I would design the cache as a sharded in-memory key-value (KV) cluster, sitting on the read path between application servers and the source of truth. The interesting decisions are how to partition keys, how to replicate for availability, how to evict when memory is full, and how to protect the origin from cache stampede on hot misses and cold restarts. I will clarify requirements, pick consistent hashing with hash slots, walk the read and write paths, then go deep on eviction, hot keys, and failure modes.

Requirements To Clarify First

Functional requirements:

  • Operations supported: GET, SET, DEL, with Time To Live (TTL) on SET.
  • Data types: just byte-string values, or richer types like lists, sets, sorted sets, hashes? Redis itself exposes structures; Memcached does not.
  • Pub/Sub, atomic counters, distributed locks — are these in scope?
  • Multi-key operations: pipelines, transactions, scripts that touch multiple keys.
  • Consistency contract: read-your-writes? Eventual? Best-effort?

Non-functional requirements:

  • Read latency budget — typically sub-millisecond at p99 inside a datacenter.
  • Hit rate target — usually north of 90% for the design to be worth it.
  • Memory size — terabytes across the fleet.
  • Throughput — millions of operations per second per cluster.
  • Durability — is the cache allowed to lose data on restart, or must it survive? If it must survive, you are in KV-store territory, not cache.
  • Multi-region — replicated cluster per region, or one global cluster?

Out of scope unless interviewer asks:

  • Full disk persistence and point-in-time recovery (that is a database problem).
  • Query languages, indexes, secondary indexes.
  • Strong cross-shard transactions (caches do not do this well, and you should say so).
  • Search and analytics workloads (use Elasticsearch or a column store).

Scale Assumptions

Area Assumption
Read traffic 10M operations/sec at peak across the cluster.
Write traffic 1M operations/sec — typically 10:1 read:write.
Working set 1 TB hot data across 100+ shards.
Value sizes Median 1 KB, p99 50 KB, hard cap 1 MB (large values blow up the event loop).
TTL distribution Most keys 5 min to 1 hour; some sticky session keys 24 hours.
Hit rate Target 95%+; below 80% means the cache is not earning its keep.
Latency budget p99 GET under 1 ms inside a region; p99 SET under 2 ms.
Restart blast radius Full cluster restart must not knock over the origin database.

The bottleneck is rarely raw queries-per-second per node — a single Redis instance can do 100k+ operations per second on commodity hardware. The bottleneck is hot keys, hot shards, and origin protection during cache misses.

Capacity math to say out loud:

  • If the hot value set is 1 TB and the median value is 1 KB, the cluster stores roughly 1B keys.
  • Per-key metadata is not free: key bytes, expiry timestamp, LRU/LFU metadata, allocator overhead, hash-table pointers. Budget 100-150 B/key for a simple string value.
  • 1B keys × 1 KB value ≈ 1 TB value bytes; 1B × 120 B metadata ≈ 120 GB overhead.
  • Add 30% allocator fragmentation and free headroom: (1.12 TB × 1.3) ≈ 1.46 TB per full copy.
  • With primary + one replica, the fleet needs about 3 TB RAM before growth margin. With 100 primary shards, each primary owns about 15 GB logical; in practice use 40-80 GB nodes so failover and resharding have breathing room.

The dominant cost is RAM, not CPU. The knobs to cut cost are TTL, smaller value caps, compact encodings, dropping cold namespaces, and moving large blobs out of cache into object storage with only metadata cached.

Core Product Model

The client (or a proxy in front of the client) is responsible for hashing the key to a shard. Each shard is a primary plus one or more replicas. A topology service tracks who owns which key range and coordinates failover.

Core Components

  • Smart client or proxy: hashes keys to shards, handles redirects when slots move, retries on failure. Smart-client examples: Jedis, Lettuce, redis-py-cluster. Proxy examples: Twemproxy, Envoy, Redis Cluster Proxy. Smart client is faster (one less network hop) but couples the client to topology; proxy is easier to operate.
  • Shard primary: owns a slice of the keyspace, serves reads and writes, replicates to followers.
  • Shard replica: takes async replication from the primary, can serve reads in some configurations, becomes the new primary on failover.
  • Topology service: maintains the slot-to-shard map and orchestrates failover. Can be Redis Sentinel, Redis Cluster gossip, ZooKeeper, etcd, or a custom controller.
  • Origin database: the source of truth. Cache misses fall through here.

Concrete Data Model And Client API

Even though the cache is "just keys," interviewers expect you to name the metadata that makes eviction, expiry, and failover work.

Data Key pattern / shard Value TTL
Cached value {namespace}:{logical_id}; slot = crc16(key) % 16384 Byte string up to 1 MB, plus metadata {expire_at_ms, last_access_sample, lfu_counter, version?} Caller-specified, usually 5 min-1 hour. No TTL only for explicitly configured namespaces.
Negative cache neg:{namespace}:{logical_id} Tiny marker: NOT_FOUND, source version, created_at 30-120 seconds to avoid hiding newly-created objects too long.
Single-flight lease sf:{hash(key)} Fetch owner request id + deadline 5-30 seconds; must expire automatically if the owner dies.
Slot map slot:{0..16383} in topology service Primary node id, replica ids, config epoch, migrating/importing state No TTL; changes through quorum-managed config update.
Hot-key sketch per-shard Count-Min Sketch + top-N heap Key fingerprints and approximate counts 1-minute rolling windows, exported to metrics.
Persistence snapshot per-shard RDB/AOF segment if enabled Serialized key/value data or append log Operational, not application TTL.

Minimal client API:

GET key                         -> value | MISS | ERROR
MGET [key...]                   -> [{key, value|MISS}]
SET key value ttl_ms [NX|XX]     -> OK | NOT_STORED | ERROR
DEL key                         -> deleted_count
INCR key delta ttl_ms?           -> integer
TOPOLOGY                         -> slot map + epoch

MGET is only efficient when keys map to a small number of shards; otherwise the smart client fans out in parallel and merges responses. Multi-key atomic operations should require a hash tag, like user:{42}:profile and user:{42}:cart, so all touched keys live in one slot.

Technology choice framing: Redis Cluster is the default if the product wants rich data structures, Lua/functions, streams, and mature operations. Memcached is better for pure string GET/SET at huge CPU scale because it is simpler and multi-threaded. Aerospike or DynamoDB DAX becomes relevant if the "cache" needs durable KV behavior. Be ready for "why not Memcached?" — answer: use it when the workload is simple; pick Redis when the data-structure and replication features are worth the operational cost.

Partitioning Strategy

Two real options:

Scheme How it works Tradeoff
Consistent hashing with virtual nodes Each physical node owns many points on the hash ring. Add or remove a node, only its neighbors are affected. Smooth resize, no central authority needed. Hot keys still pin one node.
Hash slots (Redis Cluster style) Fixed 16,384 slots, each slot maps to a shard. Moving a slot moves all its keys. Simpler operational model; reshard is explicit and observable. The choice Redis Cluster actually made.

Pick hash slots as the default — they make reshards explicit, support multi-key operations via hash tags (user:{42}:profile and user:{42}:cart go to the same slot), and integrate cleanly with online migration. Be ready to defend why over consistent hashing: predictable slot movement, MOVED and ASK redirections give the client a clear protocol, and hash tags solve the multi-key problem cleanly.

Modulo-N hashing (hash(key) % N) is the trap answer — every resize remaps every key.

Eviction Policy

When memory is full, you must evict. The policy choices and when each is right:

Policy When to use Cost
Least Recently Used (LRU) General-purpose default; recent access predicts future access. Approximate LRU is cheap; exact LRU costs a doubly-linked list per key.
Least Frequently Used (LFU) Scan-resistant workloads where a one-time large scan would evict the hot set under LRU. Needs a counter per key; access patterns shift slowly.
TTL only When every key has a meaningful expiry and you do not want to evict early. Memory may fill if TTLs are long; then SETs start failing.
First In First Out (FIFO) Almost never; LRU is strictly better at similar cost. —
Adaptive Replacement Cache (ARC) / Window TinyLFU (W-TinyLFU) When hit rate matters more than implementation simplicity. More complex; great for benchmark wins, harder to operate.

Redis defaults to allkeys-lru or volatile-lru. The deeper point: approximate LRU is fine. Redis samples a few random keys and evicts the oldest among them. With sample size 10, you get within a percent of true LRU at a fraction of the cost.

The harder design question is what to do when a SET fails because of memory pressure: return an error to the application, evict eagerly, or accept the miss on the next read? Most caches return an error and let the application decide. Some pre-evict on a background thread when memory crosses a high-water mark.

Replication And Consistency

Default model: asynchronous replication from primary to one or more replicas.

  • Writes go to the primary, which acknowledges before the replica has the data. The replication stream is a logical operation log (in Redis terms, it is the same protocol the primary would speak to a client).
  • Replica lag is typically sub-millisecond inside a datacenter, low tens of milliseconds across regions.
  • Reads can go to the primary (read-your-writes) or to replicas (cheaper, can be stale).
  • Failover promotes a replica when the primary is unreachable. Whoever runs the topology service decides this; in Redis Sentinel a quorum of sentinels agrees; in Redis Cluster the other primaries vote.

The consistency contract for caches is eventual with read-from-primary for read-your-writes. Cache invalidation is a hint, not a guarantee — applications must treat cache misses as the failure mode that drives correctness, with the origin database as the authoritative store.

If the interviewer asks about stronger consistency: caches with strong consistency exist (Aerospike, FoundationDB) but pay for it with extra round trips. Most caching use cases do not need strong consistency; the application using the cache is already comfortable with stale reads up to a TTL.

Read And Write Path

Read (cache-aside, the most common pattern):

1. App constructs `{namespace}:{logical_id}` and asks the smart client for `GET`.
2. Smart client hashes the key to a slot and reads the current slot owner from its local topology cache.
3. It dispatches to the shard primary for read-your-writes namespaces, or to a replica for stale-tolerant namespaces.
4. Cache hit: shard checks `expire_at_ms`, updates approximate LRU/LFU metadata, returns the byte string.
5. Clean miss: app tries to acquire `sf:{hash(key)}` single-flight lease for 5-30s.
6. Lease winner reads the origin, writes `SET key value ttl+jitter`, and releases by expiry. Concurrent missers wait briefly or return stale-if-available.
7. If origin returns not found, app writes `neg:{namespace}:{logical_id}` for 30-120s, not a long-lived empty value.

Write (cache-aside, write-around):

1. App writes to origin database and commits the durable version.
2. App invalidates cache with `DEL key` or writes `SET key new_value ttl` if the new value is already available.
3. Invalidation is best effort; correctness still comes from TTL and origin truth.
4. For high-value namespaces, origin write publishes an invalidation event so every region deletes the key.

Topology-move path:

1. Cluster manager marks slot S as MIGRATING from shard A to shard B with config epoch E+1.
2. Smart clients seeing MOVED update the slot map; clients seeing ASK send the next request to B for that one key.
3. A streams keys in slot S to B while both shards continue serving under migration rules.
4. Cluster manager flips ownership when copied keys and replication catch up.

Interview phrase:

A cache miss is not just a miss; it is a controlled origin fetch with a lease, jittered refill, and negative-cache option. That is the difference between a cache and a database amplifier.

Other patterns to mention:

  • Write-through: cache and database written together; cache always consistent with database but writes pay for both. Use when read-your-writes matters and origin writes are cheap.
  • Write-back (write-behind): write to cache, flush to database asynchronously. Great for write-heavy workloads, but the cache becomes a durability single point of failure. Almost never the right call unless you have invested in cache durability.
  • Refresh-ahead: proactively refresh hot keys before TTL expires. Good for predictable hot keys; wasted work for cold ones.

Default to cache-aside with write-around. State this clearly to the interviewer and only deviate if requirements force it.

Cache Stampede Protection

The single most important topic. A cache stampede happens when a hot key expires (or the cache restarts) and thousands of clients all miss simultaneously, all call the origin, and the origin falls over.

Mitigations, from cheap to expensive:

Technique How it works When to use
TTL jitter Add ±10% random jitter to TTLs so they do not all expire together. Always. Free.
Request coalescing (single-flight) First miss takes a per-key lock, fetches from origin, populates cache; concurrent misses wait. When the origin call is expensive. Implement at the application layer or in a sidecar.
Probabilistic early expiration (XFetch) Each reader rolls a die; as TTL approaches, probability of "refresh now" rises. One reader hits origin a few seconds early, repopulating before the cliff. Hot keys where downtime at expiration is unacceptable.
Background refresh A worker refreshes hot keys before they expire; readers always see fresh data. Top-N hot keys, where you can identify them.
Negative caching Cache misses themselves with a short TTL ("this key has no value"). When clients repeatedly request keys that do not exist (think 404 floods).
Bloom filter in front of cache Reject impossible lookups before they hit the cache or origin. Cache penetration attacks where attackers query random nonexistent keys.

The deepest answer: distinguish stampede on hot key expiration from stampede on cache cold start. The first is per-key; coalescing handles it. The second is fleet-wide; only origin protection (origin-side rate limiting, gradual cluster warm-up, sticky pre-warming from a snapshot) handles it.

Hot Key And Hot Shard Strategy

A hot key is one that a single shard cannot serve. Strategies:

  • Read replicas: spread reads across replicas. Works for read-hot keys, not for write-hot keys.
  • Client-side caching: clients hold a short-TTL in-process cache for the very hottest keys. Redis 6 supports server-assisted client-side caching with invalidation notifications.
  • Key splitting: split leaderboard:global into leaderboard:global:{0..15} and have clients merge results. Trades extra reads for distributed load.
  • Dedicated hot-key tier: route known hot keys to a separate replica fleet sized for the traffic.
  • Application-level dedupe: if many requests need the same data, batch them at the application layer before they reach the cache.

A hot shard is one whose key range concentrates traffic regardless of any single key. Symptoms: shard CPU pegged, neighbor shards idle. Strategies: pre-sharding (allocate more slots to the hot range from day one), online slot migration, or restructuring keys to spread better (avoid tenant_id:{...} if one tenant dominates).

Failure Modes

Failure What happens Mitigation
Single replica down Reads to that replica fail; clients retry or fall back to primary. Health checks, fast topology refresh.
Primary down Failover promotes a replica. Some uncommitted writes lost (async replication). Quorum-based promotion, fence the old primary.
Network partition Split brain risk: two primaries serving the same slot. Quorum prevents promotion when majority is unreachable; fencing tokens on writes.
Cluster cold start Cache is empty; every read misses; origin overwhelmed. Restore from a snapshot, gradual traffic ramp, origin-side rate limit.
Slow operation blocks event loop A KEYS * or large SUNION pegs the single thread; everything else times out. Forbid slow operations in code review; use SCAN; cap value sizes.
Out of memory (OOM) Cache evicts faster, hit rate drops, origin load rises. Alert on memory pressure well before hard limit; eviction policy that prefers cold data.
Replica lag spikes Replica reads return very stale data. Bound the lag; drop replica from rotation if lag exceeds threshold.
Hot key kills one shard One shard at 100% CPU, others idle. Detect via per-key sampling; apply key splitting or client-side cache for the offender.

Graceful Degradation State Table

Stage Mode Source Latency budget Fallback
Hot hit Normal cache read Shard primary / replica <1 ms p99 in-region None.
Local topology stale Client routed to old owner MOVED / ASK redirect One extra network round trip Refresh slot map by epoch.
Clean miss Origin fetch controlled by lease Origin DB + single-flight Origin p99 plus waiters capped at a few hundred ms Serve stale value if namespace allows; otherwise bounded miss.
Lease owner dies Single-flight lease expires sf:{hash(key)} TTL 5-30s worst-case wait if clients choose to wait Next waiter becomes owner; origin rate limit still applies.
Shard primary down Failover Replica + topology quorum Seconds; client retries with backoff Promote replica, fence old primary, serve stale from local app cache if allowed.
Cluster cold Low hit rate after restart Snapshot + origin Minutes to warm Traffic ramp, origin circuit breaker, pre-warm top keys.
Origin overloaded Cache miss path unsafe Origin rate limiter Serve degraded response quickly Stale-if-error, negative admission, shed low-priority requests.

The cursor-like state that persists across requests is the topology epoch in the client and, for paged bulk operations such as SCAN, the server-issued scan cursor. A SCAN cursor encodes shard, bucket position, and topology epoch; if the epoch changes during resharding, the client should tolerate duplicates and dedupe by key.

Interview phrase:

The graceful state machine is hit → controlled miss → stale-if-allowed → shed. I never let a cache outage turn into an unbounded origin stampede.

State the fail-mode policy out loud: cache failures must not become origin failures. The application must treat cache errors as cache misses and have origin-side protection (rate limit, circuit breaker) to absorb the resulting spike.

Persistence (Cache vs. KV Store)

Pure caches do not persist. If the interviewer says "we cannot afford a cold start," you have two options:

  • Snapshot to disk: periodic point-in-time dumps (Redis RDB). Cheap, but you lose anything since the last dump.
  • Append-only log: every write also appends to a log on disk (Redis AOF). More durable, slower writes, log compaction needed.

Be honest with the interviewer: enabling persistence on a cache makes it a slower KV store. If the application cannot tolerate cold-start latency, the better answer is usually:

  1. Replica in another availability zone, promoted on full primary failure.
  2. Origin protection so cold start is bearable, not invisible.
  3. Snapshot on shutdown and reload on startup (best-effort warm cache).

Do not promise zero-loss durability from a cache layer. That is a database.

Memory Management

The implementation detail that distinguishes a serious answer:

  • Memory allocator: jemalloc is the default in Redis; it dramatically reduces fragmentation versus glibc malloc. Mention it.
  • Fragmentation ratio: used_memory_rss / used_memory. Above 1.5 means you are wasting memory; consider ACTIVEDEFRAG or a restart.
  • Object sharing: small integers are shared across keys in Redis. Useful for high-cardinality counter caches.
  • Encoding optimization: short hashes, lists, sorted sets use compact encodings (ziplist, listpack); they cross over to full hash table at configurable thresholds. Tuning these saves memory at the cost of operation latency for large values.
  • Eviction granularity: per-key, not per-bucket. Big values evict fewer items per memory reclaimed.
  • maxmemory and maxmemory-policy: the two configs that decide eviction behavior at memory cap. Get them wrong and the cache starts returning errors instead of evicting.

Threading Model

Redis is famously single-threaded for command execution. The implications:

  • No locks needed inside the engine — operations are atomic by virtue of serial execution.
  • One slow command stalls everything. KEYS * over a million keys is a multi-second stall.
  • Use the event loop for I/O, single thread for command logic. Redis 6 added I/O threads for parsing and writing replies, keeping command execution single-threaded.
  • Pipelines batch commands so the round-trip cost amortizes. Always use pipelines for bulk operations.
  • Lua and Function scripts run atomically on the single thread; the same "do not do slow things" rule applies.

The frequent interview question: "Why is Redis single-threaded and still fast?" Answer: the bottleneck is network and memory bandwidth, not CPU; serialized execution avoids locking overhead; the event loop handles many connections concurrently. The threading model is a fit for the workload, not a workaround.

Multi-Region Strategy

Three options, in order of operational simplicity:

  1. Independent regional clusters. Each region has its own cache, populated lazily from its own origin replicas. No cross-region cache coordination. Cache may temporarily disagree across regions; eventual consistency from the origin replication brings them together.
  2. Active-passive global cluster. One region writes, others read. Reads in the passive region pay cross-region latency on miss only. Failover promotes another region.
  3. Active-active with Conflict-free Replicated Data Types (CRDTs). Redis Enterprise supports this. Writes in any region converge. Expensive operationally; only use if the product demands it.

Default to option 1. Push the consistency problem down to the origin database, where it belongs.

Operational Metrics

What you alert on:

  • Hit rate (per route, per shard, global). Drop indicates origin pressure incoming.
  • p99 GET / SET latency.
  • Connections per shard. Connection pool exhaustion is a common failure mode.
  • Memory usage and fragmentation ratio per shard.
  • Evictions per second. Spike means working set just exceeded memory.
  • Replication lag per replica.
  • Commands per second per shard. Hot shard detection.
  • Slowlog entries. Anything over 10ms gets a human eyeball.

What you sample (not alert on, but log):

  • Top N hot keys per shard.
  • Top N largest values.
  • Origin call rate from cache-miss path.

Tradeoffs To Call Out

  • Cache freshness vs origin load: shorter TTL means fresher data and more origin traffic. Pick TTL per data type, not globally.
  • Consistency vs availability: stronger consistency means cross-shard coordination, which kills latency. Caches choose availability and pay with staleness.
  • Memory vs compute: bigger cache, higher hit rate, lower origin cost — until you cannot afford the cache. Find the sweet spot via hit-rate / cost-per-GB analysis, not gut feeling.
  • Smart client vs proxy: smart client is one hop faster; proxy is easier to operate and lets you change topology without redeploying clients.
  • Persistence on or off: on costs latency and disk; off costs cold-start pain. Default off, mitigate cold-start with replicas and origin protection.

Common Traps And Stronger Moves

  • Saying "just use Redis" without picking partitioning, replication, or failover. The interviewer wants to see the components, not the product name.
  • Forgetting cache stampede entirely. The biggest "miss" in answering this question.
  • Promising strong consistency from an async-replicated cache. The contract is eventual; own it.
  • Modulo-N hashing for partitioning. Reshard is a full data movement.
  • Ignoring hot keys and hot shards. Senior interviewers will deliberately steer here.
  • Treating the origin as infinitely scalable. The cache exists to protect the origin; if you forget the origin in your design, you have not solved the actual problem.
  • Promising sub-millisecond cross-region cache reads. Speed of light says no.

Deep Dives Interviewer May Choose

  • Distributed locking with Redis: the Redlock algorithm, Martin Kleppmann's critique, when redlock is OK (best-effort coordination) versus when you need a real consensus system (correctness).
  • Pub/Sub vs Streams: Redis Pub/Sub is fire-and-forget, no delivery guarantee. Redis Streams are an append log with consumer groups, persistence, and at-least-once semantics. Pick the right one.
  • Geo, HyperLogLog, sorted sets: when these earn their keep. Sorted sets for leaderboards, HyperLogLog for unique-count estimation under tight memory.
  • Sharded counter vs Redis INCR: when do you need sharded counters to avoid hot-key pinning?
  • Cluster resharding mechanics: how a slot migrates while still serving traffic (the MIGRATING and IMPORTING states, MOVED versus ASK redirects).
  • Memcached vs Redis: Memcached is multi-threaded, simpler data model, no persistence; Redis has data structures, persistence, replication. Memcached is still right for the simplest workload.

Staff-Level Answer Checklist

  • Stated the invariant: cache returns fresh-enough or clean miss, origin is protected from stampede.
  • Picked partitioning (hash slots) with justification.
  • Picked eviction (approximate LRU) with justification.
  • Walked the cache-aside read and write paths.
  • Addressed cache stampede (jitter, coalescing, probabilistic refresh).
  • Addressed hot keys and hot shards (splitting, client-side cache, replicas).
  • Discussed failure modes and stated that cache failures must not become origin failures.
  • Picked replication model (async primary-replica with quorum failover) and named the consistency contract.
  • Multi-region strategy (independent regional clusters by default).
  • Operational metrics and what to alert on.

Interview Phrase

"The cache exists to protect the origin and to serve fast reads. Every design decision — partitioning, eviction, replication, stampede protection — flows from those two goals. I default to hash-slot partitioning, approximate LRU, async replication with quorum failover, and cache-aside with TTL jitter and request coalescing. The most common failure I design against is correlated cache miss, not a single node crash."

2. Simulated Interview

Interviewer: Design a distributed cache like Redis.

Candidate: I will design a sharded in-memory KV cluster on the read path between application servers and an origin database. The two goals are sub-millisecond reads with high hit rate, and protecting the origin from correlated cache misses. I will clarify requirements, pick partitioning, walk the data path, and dive into eviction and stampede protection.

Interviewer: How do you partition keys?

Candidate: Hash slots, like Redis Cluster. A fixed number of logical slots — say 16,384 — each mapped to a shard. Clients hash the key, look up the slot in a small map, and route directly. Resharding moves slots between shards, which is explicit and observable. Hash tags like user:{42}:profile let me put related keys on the same shard for multi-key operations. I avoid modulo-N hashing because resize remaps everything, and I avoid pure consistent hashing because hash slots have a cleaner operational story.

Interviewer: How do you handle replication and failover?

Candidate: Each shard is a primary plus one or more async replicas. Writes go to the primary, replication is sub-millisecond inside a region. A topology service — Redis Cluster gossip, Sentinel, or etcd-backed controller — watches health and promotes a replica on primary failure. Quorum is required for promotion, which prevents split brain during a partition. The consistency contract is eventual; reads from replicas can be slightly stale.

Interviewer: What eviction policy do you use?

Candidate: Approximate Least Recently Used as the default. Redis samples a handful of random keys on each eviction and removes the oldest among them; with a sample size around 10 it is within a percent of true LRU at much lower cost. For scan-heavy workloads where LRU evicts the hot set, I switch to Least Frequently Used. The deeper choice is what happens on memory pressure: by default, SETs return an error; I would also alert at 80% memory so an operator can resize before it hurts.

Interviewer: What about cache stampede?

Candidate: The biggest topic on this question. Three layers. First, TTL jitter — add ten percent random noise so keys do not all expire at the same instant. Free win. Second, request coalescing per key — the first miss takes an in-process lock or short Redis lock, calls the origin, populates the cache; concurrent misses wait. Third, probabilistic early refresh for the hottest keys — readers stochastically refresh slightly before expiration, so the cliff never happens. Separately, I plan for cold-start stampede where the whole cache is empty: snapshot-on-shutdown plus restore-on-startup, plus origin-side rate limiting that absorbs the inevitable spike.

Interviewer: What if a single key is so hot it pins one shard?

Candidate: That is the hot-key problem. A few moves: enable server-assisted client-side caching so each client keeps a short-TTL local copy; spread reads across replicas; split the key — leaderboard:global becomes leaderboard:global:{0..15} and clients union the slices. For write-hot keys, key splitting is the only real answer; client caching does not help.

Interviewer: Cache-aside or write-through?

Candidate: Cache-aside with write-around as the default. Application reads check cache, miss falls through to origin, populates cache. Application writes go to origin then invalidate the cache key. Write-through is tempting because it keeps cache and origin consistent, but it doubles the write path latency and the failure modes get nasty when the cache write succeeds and the origin write fails. Write-back is almost never right unless you have built real durability into the cache, at which point it is a database.

Interviewer: What happens when the cache cluster is unavailable?

Candidate: Cache failures must not become origin failures. The application treats cache errors as cache misses, falls through to origin. Without protection that is the worst case for the origin — every read goes through. So the application also has a circuit breaker that opens on sustained cache errors, with origin-side rate limiting and request shedding behind it. Better to serve a fraction of users a clean error than to take the origin down for everyone.

Interviewer: How does the cache restart without taking down the origin?

Candidate: Two strategies. First, replicas are in a separate availability zone so a single-zone outage does not take the whole cluster cold. Second, snapshot to disk on shutdown, load on startup; the cluster comes back warm-ish. Third, if both fail and we are doing a real cold start, the application's circuit breaker and origin-side rate limit absorb the spike. The cache being slow to warm is acceptable; the database falling over is not.

Interviewer: Multi-region?

Candidate: Default is independent regional clusters, each populated from its regional origin replicas. No cross-region cache coordination, lower complexity, eventual consistency from the database replication. If the product genuinely needs a single global cache, Redis Enterprise active-active with CRDTs exists, but the operational cost is real and most products do not need it. I would push hard on whether the requirement is actually a global cache or just per-region consistency with the global database.

Interviewer: Tell me about persistence.

Candidate: A pure cache does not persist. The moment you turn on persistence, the cache is slower and the failure modes are more complex; you are paying KV-store costs for cache benefits. Redis offers RDB snapshots and AOF append logs. I use RDB snapshots on shutdown as a best-effort warm reload, not as durability. If the product cannot tolerate the cold-start period, the right answer is usually a multi-AZ replica setup plus origin protection, not full persistence.

Interviewer: What metrics do you alert on?

Candidate: Hit rate per route — a drop means origin load is rising. P99 latency for GET and SET. Memory pressure and fragmentation ratio per shard — eviction rate spikes when working set crosses memory cap. Replication lag — old replicas should not serve reads. Slowlog entries — anything over ten milliseconds in a single-threaded engine is a real problem. Connections per shard — pool exhaustion is a quiet failure mode that does not look like a cache outage until everything times out.

Interviewer: Summarize the design.

Candidate: Hash-slot partitioning with smart clients, async primary-replica replication per shard, quorum-based failover. Approximate LRU eviction. Cache-aside with TTL jitter, request coalescing, and probabilistic early refresh for stampede protection. Hot keys handled with client-side caching, replica reads, and key splitting. Multi-region uses independent regional clusters. Failures are designed so the cache going down does not take the origin with it — circuit breakers, origin-side rate limits, and the application treating cache errors as cache misses. The whole design optimizes for fast reads and protected origin, not for exact consistency.

3. Practice Notes

Five-Minute Version

  1. Clarify: data types, TTL behavior, consistency contract, persistence required or not.
  2. State the invariant: cache returns fresh-enough or clean miss, origin protected from stampede.
  3. Pick hash-slot partitioning; explain why over modulo and consistent hashing.
  4. Pick async primary-replica with quorum failover; name the consistency contract.
  5. Pick approximate LRU; cover eviction at memory pressure.
  6. Walk cache-aside read and write paths.
  7. Stampede protection: jitter, coalescing, probabilistic refresh.
  8. Hot keys: client-side cache, replica reads, key splitting.
  9. Failure: cache errors become cache misses, not origin failures.
  10. Close with metrics and multi-region as independent regional clusters.

One-Minute Closeout

A cache is an approximation system whose first job is to protect the origin and whose second job is to be fast. Hash slots for partitioning, async replication for availability, approximate LRU for eviction, and cache-aside with stampede protection on top. The interesting questions are not the data structures; they are what happens when keys are hot, when the cache is cold, and when a shard fails. I design those paths first and let the steady-state design fall out.

Best Interview Phrases

  • "The cache exists to protect the origin, not just to be fast."
  • "Cache failures must not become origin failures."
  • "Stampede is the failure mode I design against first; node death is easy."
  • "Async replication, eventual consistency, read-from-primary for read-your-writes — that is the contract."
  • "Approximate LRU is good enough; do not pay for exact."
  • "Hot keys break naive designs; client-side caching, replica reads, and key splitting are the three moves."
  • "A cache with persistence turned on is a slower KV store. Be explicit about which one you are building."
