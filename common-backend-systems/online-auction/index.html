Online Auction — eBay-Style Bidding (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build an online auction system. Sellers list items with a start price, duration, and reserve. Buyers place bids; the highest bid at close wins. Right before close, "sniping" creates a write-heavy spike. Assume 100M auctions/year, 10M concurrent active auctions, ~1M bids/sec at peak across the platform during major events.

1. Requirements (~5 min)

Functional - Seller creates an auction with start time, end time, start bid, reserve. - Buyers place bids; the system enforces minimum increment and rejects late/low bids. - All viewers see live current price; on close, the highest bidder wins.

I'd defer payments deep dive, dispute resolution, and seller storefronts.

Non-functional - Strong correctness: highest valid bid at close wins, deterministically. - Per-auction total ordering on bids — the "current high bid" is unambiguous. - Bid placement p99 < 500 ms even during the last-minute spike. - Live price updates broadcast to all watchers within ~1 s. - Highly available — every dropped bid is a real complaint.

Capacity - 10M concurrent auctions; ~1M bids/sec at peak. - Read fan-out: a hot auction may have 100k viewers watching live. - Bid records are not the dominant cost unless a major event sustains peak: 1M bids/sec × 10 min × ~180 B × 3 replicas ≈ 324 GB for that spike. Over a normal year, 1B bids × 180 B × 3 ≈ 540 GB before indexes. - The hotter constraint is per-auction serialization and live fan-out: a 100k-viewer auction needs ~1.6 MB of watcher IDs, but the broadcast path must push every accepted high-bid update within ~1 s.

2. Core Entities (~2 min)

  • Auction — auction_id, seller_id, item, start_bid, reserve, start_at, end_at, status, current_high_bid_id.
  • Bid — bid_id, auction_id, bidder_id, amount, ts.
  • Watcher — user_id, auction_id. Drives live updates and notifications.
  • User — buyer or seller.

3. API (~5 min)

POST  /v1/auctions                          { item, start_bid, end_at, reserve }  → { auction_id }
GET   /v1/auctions/{id}                     → Auction + recent bids
POST  /v1/auctions/{id}/bids                { amount }   (idempotency-key required)  → { bid_id, accepted, new_high }
WS    /v1/auctions/{id}/stream              → live high-bid + countdown

User from token. Idempotency key on bids is mandatory.

5. High Level Design (~10–15 min)

  1. Auction service. Owns auction metadata. Reads served from a write-through cache.
  2. Bid service. The heart of the system. Receives bid → validates (auction active, amount > current + min increment, bidder not seller) → atomically promotes if higher → writes the bid row → broadcasts.
  3. Per-auction sharding. All writes for one auction go to the same shard so we can enforce ordering with a single in-memory queue or a row lock. Hot auctions still hit a single shard.
  4. In-memory current-high state. Each auction's current_high_bid_id, amount lives in Redis (or in-process state) for the bid service to compare against without a DB round-trip. Source of truth backed by the bid row in MySQL with a unique constraint per auction on (auction_id, bid_id).
  5. Bid persistence. Append-only Cassandra or sharded MySQL keyed by auction_id — all bids for one auction co-locate. Bids are immutable.
  6. Closer service. A timing wheel or scheduled jobs keyed by end_at per auction. At close, the service marks the auction closed, freezes the high bid, fires the auction.closed event.
  7. Live updates. Each bid emits an event to a per-auction channel. WS gateway holds connections and fans out. For mega-auctions with 100k watchers, fan-out goes through a hierarchical bus.
  8. Sniping defense. "Soft close": if a bid lands in the last 30 s, extend end_at by 30 s. Atomic update on the auction row.

End-to-end narration: Auction A1 closes at 10:00:00. At 09:59:58 Alice bids $120 (current high $115). Request hits the partition for A1 → in-memory state shows current = $115 → atomic CAS to ($120, Alice's bid_id) → write bid row → publish event → all watchers get the update within 1 s → soft-close extends end to 10:00:28. Bob bids $130 at 10:00:25 → similar flow. No more bids → at 10:00:28 closer service marks A1 closed, Bob wins.

6. Deep Dives (~10 min)

Concrete stores and shard keys.

Store Key pattern Data structure / entry TTL / retention Shard key
Auction metadata auction:{auction_id} MySQL/Dynamo row, item/end/status/current_high pointer, ~1 KB active + 90 d hot, archive years auction_id
Current high auction_high:{auction_id} Redis hash {amount,bid_id,bidder,version}, ~120 B until end_at + 7 d auction_id
Bid log bids:{auction_id} append-only partition ordered by server sequence, ~150–180 B/bid years auction_id
Idempotency bid_idem:{user_id}:{key} accepted/rejected response, ~300 B 24 h user_id
Close wheel close:{epoch_second} set/list of auction_ids, ~16 B/id until slot processed + 2 h epoch_second
Watcher channel watch:{auction_id} ephemeral connection IDs, ~16 B/watcher while connected auction_id

The bid log is content; Redis is only the current-high index. If Redis loses a key, the owner replays the last bids for that auction from bids:{auction_id} and reconstructs the high bid.

Capacity by store.

Store / index Math Notes
Auction metadata 10M active × 1 KB × 3 ≈ 30 GB hot; 100M/year ≈ 300 GB/year archive closed auctions cheaply
Current high 10M active × 120 B × Redis overhead/replicas ≈ 5–10 GB real memory index only; rebuildable
Bid log 1B bids/year × 180 B × 3 ≈ 540 GB/year; 10-min 1M/sec spike ≈ 324 GB durable content store
Idempotency 100M bid attempts/day × 300 B × 3 ≈ 90 GB if kept 24 h TTL keeps it bounded
Close wheel 10M IDs × 16 B × 3 ≈ 480 MB time index, not content
Watcher channel hot auction 100k watchers × 16 B ≈ 1.6 MB per region ephemeral connection index

Why sharded MySQL/Dynamo for auction metadata but append storage for bids: the auction row needs conditional state transitions (active → closed, soft-close extension), while the bid log is immutable content keyed by auction_id. A single relational table for all bids can work at lower scale, but the hot path should not make live fan-out or bid append compete with metadata row locks.

Place-bid path. 1. Client sends POST /v1/auctions/{id}/bids with an idempotency key; API gateway authenticates and forwards to the bid router. 2. Router hashes auction_id to the current single owner; all bid requests for that auction hit the same in-memory queue. 3. Owner checks bid_idem:{user_id}:{key}; if absent, it reads auction status/end time and current high from memory or auction_high:{auction_id}. 4. Owner validates active auction, bidder != seller, amount >= current + increment, and soft-close condition. 5. In one serialized step, it assigns a sequence, appends to bids:{auction_id}, updates auction_high:{auction_id}, and updates auction:{auction_id}.current_high_bid_id. 6. Broadcast is async: publish bid.accepted to the auction channel; slow WS clients get the newest high bid and may miss intermediate bids.

Auction read path, hot and cold. 1. GET /v1/auctions/{id} reads auction:{auction_id} and auction_high:{auction_id} in parallel; recent bids come from the tail of bids:{auction_id} with cursor {auction_id, last_sequence}. 2. Hot live clients should stay on WS /stream; the WS gateway keeps only connection IDs in watch:{auction_id} and receives accepted-high events from regional pub/sub. 3. If auction_high:{auction_id} is missing, the service reads the auction row, replays the last bid-log page for auction_id, reconstructs the high bid, and repopulates Redis before returning. 4. If the cursor asks for bids older than the hot partition, the API switches to the archived bid log; latency can be seconds because bid acceptance is already over.

Two-index framing. The same bid dataset has three views: append-only bids:{auction_id} as content, auction_high:{auction_id} as the current-high index, and close:{epoch_second} as a time index for closure. Keeping that distinction clear prevents the common mistake of treating Redis current-high as the legal audit trail.

Per-auction ordering under spike. The whole interview hinges here. Two viable approaches: - Single writer per auction. Route all bids for auction_id = X to one Bid service instance (consistent hashing). That instance serializes via an in-memory mutex/queue. Easy ordering. Fault: instance death loses in-flight state — back it with a Kafka topic so on failover the new owner can replay. - Optimistic CAS in Redis. WATCH/MULTI/EXEC or a Lua script: read current high; if new bid > current, replace; else reject. Naturally scales but loses fairness when multiple bids arrive in the same RTT (one wins arbitrarily).

I'd lead with single-writer-per-auction for clarity, mention Redis CAS as the alternative.

Why not just lock a database row for every bid: it is correct, but the hottest auction pins one DB primary and makes fan-out retries compete with bid acceptance. The single-owner design keeps the critical comparison in memory, uses the DB as durable append-only history, and can replay after failover.

Sniping spike. Last 30 s of a popular auction: thousands of bids, all for the same key. The hot-key is unavoidable since bids must be ordered against the current high. Mitigations: scale the single-owner's bid service for that auction; use a tightly-coupled in-memory queue; ensure WS fan-out doesn't bottleneck the bid path (separate concern).

Multi-region and clocks. Active bidding for an auction is pinned to one home region selected by auction_id; other regions serve read replicas and route bids back to the owner. The owner stamps bids with server receive time, and the close service rejects any bid whose receive time is after the final end_at, even if a remote client sent it earlier. The tradeoff is a little extra cross-region latency for some bidders in exchange for deterministic close semantics.

Soft close. Prevents true snipe-at-zero: any bid in the last 30 s extends the auction by 30 s. Encoded as end_at = max(end_at, now + 30s) in the same transaction as the bid acceptance.

Auction closing. Don't scan all auctions every second. Timing wheel keyed by end_at resolution = 1 s, slot holds auction_ids. The closer service walks the current slot, fires close events. For 10M concurrent auctions this is manageable.

Bid latency. End-to-end < 500 ms. Steps: API gateway (10 ms), routing to owner (10 ms), in-memory accept (1 ms), Cassandra write (~10 ms), publish (1 ms). Lots of headroom; the heavy cost is the WS fan-out to all watchers, which is decoupled.

Live updates at fan-out. 100k watchers on one auction. Hierarchical pub/sub: bid event → regional brokers → WS gateways → clients. Drop intermediate updates if a client is slow (only need the latest current high).

Graceful degradation.

Case Mode Source Latency budget Fallback
Cold auction DB + cache auction:{id}, auction_high:{id} < 500 ms bid lazy-load owner state on first bid
Hot auction single owner queue in-memory high + bid log append < 500 ms bid shed live-update work, never shed accepted bid persistence
Owner failure replay bids:{auction_id} + last sequence seconds freeze accepts briefly, elect new owner, replay tail
Close wheel delayed deterministic close auction.end_at + owner sequence close within 1–2 s reject bids with server receive time > final end_at
Watcher exhausted latest-only stream pub/sub + WS gateway < 1 s latest drop intermediate updates for slow clients

"My punch line: every auction is intentionally a single hot key; the design is about isolating that hot key, not pretending we can shard one auction's bid ordering."

Idempotency. Bids retried during slow networks must not double-charge or double-rank. Server stores (idempotency_key, response) for 24 h.

Fraud and shill bidding. Detect bidders with very high cancel rates, IP overlap with seller, sudden spike just before close. Hide flagged bids from leaderboard; resolve manually.

What I'd defer. Buy-it-now, second-chance offers, multi-item auctions (Dutch), bid retraction, payment capture deep dive, dispute resolution.
