Payment System — Stripe / PayPal Class (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Hard · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build a payment system. Merchants charge customers' cards via our API; we route to card networks (Visa/MC) or wallets, settle funds, handle refunds and disputes, and never lose or double-charge a customer. 100M transactions/day, peak ~5k/sec, average txn $30. Failure modes around money are the whole interview.

1. Requirements (~5 min)

Functional - Merchant submits a charge intent (customer payment method + amount). - System authorizes with the card network, captures funds, returns success/failure. - Refund and partial-refund. - Webhook callbacks to merchants on terminal events.

I'd defer recurring billing, subscriptions, multi-currency conversion, and fraud-model details.

Non-functional - Correctness above all. Never double-charge, never silently drop a charge. - Idempotency on all writes — networks retry; we must too without duplicating money movement. - Strong consistency on ledger balances. - Durable audit log of every state transition (regulatory). - p99 charge < 2 s (most of that is the card network). - High availability — payments down means real revenue loss for every merchant.

Capacity - 100M txns/day = 1200/sec average, 5k/sec peak. - ~7-year retention for transaction records (PCI / regulatory). - Hot traffic is not huge; retained money history is. 100M/day × 7 years ≈ 255B transactions. - If each transaction creates ~4 ledger entries at ~250 B, ledger storage is ~255 TB raw, ~765 TB with 3 replicas before indexes/cold compression. Ledger + audit are the dominant costs.

2. Core Entities (~2 min)

  • PaymentIntent — the merchant's order: intent_id, merchant_id, amount, currency, customer_id, status.
  • Charge — an attempt to move money: charge_id, intent_id, status, network_ref.
  • LedgerEntry — double-entry: (account, debit/credit, amount, ref). Authoritative.
  • PaymentMethod — tokenized card / wallet handle.
  • Refund — partial or full reverse of a charge.

3. API (~5 min)

POST  /v1/payment_intents                { amount, currency, payment_method, customer }   (Idempotency-Key header required)   → PaymentIntent
POST  /v1/payment_intents/{id}/confirm                                                                                       → Charge
POST  /v1/charges/{id}/refunds            { amount }                                                                          → Refund
GET   /v1/charges/{id}
POST  /v1/webhooks/{merchant_id}/events                                                                                       // outgoing

Idempotency-Key is mandatory on every state-changing request. That's the cornerstone.

5. High Level Design (~10–15 min)

  1. API gateway. Auth, rate-limit, idempotency-key middleware: looks up (merchant_id, key) → returns cached response if seen, else passes through.
  2. PaymentIntent service. Writes the intent row in a strong-consistency store (Spanner-class, or sharded MySQL with a master per shard). State machine: requires_payment_method → requires_confirmation → processing → succeeded | failed.
  3. Authorization service. On confirm, build the network message (ISO 8583 or HTTP for processor APIs), route to a card-network-adapter. Persist the outgoing attempt before sending; persist the response. On timeout, mark unknown (not failed) and reconcile later — never assume failure on timeout, that's how you double-charge.
  4. Ledger service. Double-entry bookkeeping. Every money movement = a debit on one account + a credit on another, totals must balance. Examples: - Authorization: pending_authorization (customer) ↔ pending_capture (merchant payable). - Capture: pending_capture → merchant_balance; customer pending → settled. - Refund: reverse the entries. Stored in an append-only ledger table, partitioned by account.
  5. Webhook service. On terminal state, enqueue a delivery to the merchant's webhook URL. Retry with backoff up to 3 days; persisted queue.
  6. Network adapters. Per-card-network or processor (e.g., Visa, Stripe Connect). Each owns its connection pool, message format, and retry policy.
  7. Reconciliation. Daily batch job pulls the day's network settlement file, compares against our ledger, flags discrepancies.
  8. Audit log. Every state transition appended to an immutable log (Kafka with very long retention + cold storage).

End-to-end narration: merchant calls POST /payment_intents with idempotency key K. Gateway checks; never seen. Intent row written (status=requires_confirmation). Merchant calls /confirm → auth service sends ISO 8583 message to Visa → Visa responds approved in 800 ms → ledger writes double-entry (customer pending: −$30; merchant payable: +$30) → intent.status=succeeded → webhook fired. Two days later, settlement file from Visa arrives → reconciliation confirms our records match → ledger entries become settled.

6. Deep Dives (~10 min)

Concrete stores and shard keys.

Store Key pattern Data structure / entry TTL / retention Shard key
Idempotency idem:{merchant_id}:{key} status + response hash/body pointer, 1–4 KB 24–72 h hot, 7 d durable (merchant_id, key)
PaymentIntent pi:{intent_id} strongly consistent row, amount/status/customer, ~1 KB 7 y merchant_id + intent_id
Charge attempt ch:{charge_id} network request/response refs/status, ~1 KB 7 y charge_id
Ledger ledger:{account_id}:{seq} append-only debit/credit entry, ~250 B 7 y hot/warm, archive longer account_id
Outbox/webhook webhook:{merchant_id}:{event_id} destination/status/attempts/body pointer, ~1 KB retry 3 d, status 90 d merchant_id
Audit log audit/yyyy/mm/dd/{partition} immutable event stream/object records, ~500 B/event 7 y+ date + partition

The ledger is content and source of truth; caches and idempotency keys are control-plane safety rails. I would explicitly avoid making Redis authoritative for any money movement.

Capacity by store.

Store / index Math Notes
Idempotency 100M/day × 2 KB avg × 3 ≈ 600 GB for 24 h; up to ~1.8 TB for 72 h hot safety rail, TTL-bounded
PaymentIntent 255B over 7 y × 1 KB × 3 ≈ 765 TB compress/archive cold years
Charge attempt 255B × 1 KB × 3 ≈ 765 TB includes network refs, not full PAN/card data
Ledger 255B txns × 4 entries × 250 B × 3 ≈ 765 TB authoritative money history
Webhook outbox 100M/day × 1 KB × 90 d × 3 ≈ 27 TB if status retained hot active retry set is much smaller
Audit log 255B txns × ~5 events × 500 B × 3 ≈ 1.9 PB raw cold object storage + compression dominates savings
Settlement files 100M/day × 200 B ≈ 20 GB/day raw batch reconciliation input

Kafka is useful for audit/event fan-out, but it is not the ledger. The outbox row is committed with the state transition first, then a broker delivery can be retried. That keeps the durable payment state ahead of asynchronous notifications.

Confirm charge path. 1. Gateway authenticates merchant and checks idem:{merchant_id}:{key}; if another request is in flight on the same key, it waits or returns the eventual result. 2. PaymentIntent service CASes requires_confirmation → processing on pi:{intent_id}. 3. In parallel where allowed, it loads the tokenized payment method from the vault and runs fraud/risk checks; if either fails, the intent transitions to failed with an audit event. 4. Authorization service writes ch:{charge_id} as network_request_persisted before sending anything to the network. 5. Network adapter sends the auth/capture request. Approved response triggers a ledger transaction: customer pending debit and merchant payable credit must commit together. 6. Intent/charge transition to terminal state, outbox writes webhook:{merchant_id}:{event_id}, and the idempotency record stores the exact response. 7. If the adapter times out, the charge remains unknown; retry/reconcile is keyed by network_ref, not by creating a second charge.

Refund and read paths. 1. Refund request checks idem:{merchant_id}:{refund_key} and reads ch:{charge_id} on shard charge_id. 2. Service verifies refunded_amount + requested_amount <= charge.amount, then creates refund:{refund_id} and persists the outgoing network refund attempt before sending. 3. On network approval, ledger writes the reversing double-entry rows on shard account_id; the charge's materialized refunded_amount updates with a CAS. 4. GET /v1/charges/{id} reads ch:{charge_id}, pi:{intent_id}, and a ledger summary in parallel. If the charge is older than the hot retention window, the API falls through to the cold archive and may return slower but still audit-complete history. 5. The only session state clients carry is the idempotency key. Pagination for merchant transaction exports uses an opaque cursor {merchant_id, created_at, charge_id} so exports do not rely on offset scans over money history.

Two-index framing. The ledger is account-keyed because balances are computed by account. Merchant dashboards are transaction-keyed by merchant_id and created_at. Those are two indexes over the same money events; the dashboard index can lag, but the ledger cannot.

Idempotency — the heart. Every mutating endpoint accepts an Idempotency-Key. Server stores (merchant_id, key) → response, status for ≥24 h. Repeat requests return the cached response without re-executing. Key store is a Redis cluster + a durable backing store. Critically, the key locks the operation: in-flight requests with the same key wait (with a timeout); subsequent requests return the result. Networks WILL retry; we must absorb retries safely.

Timeout = "unknown", not "failed". If the card network call times out, the truth is unknown. Marking the charge failed and accepting a retry could double-charge. Mark processing/unknown; reconcile later via the network's stand-in advice or settlement file. The merchant sees pending; the customer doesn't get billed twice.

Double-entry ledger. Every money movement is a balanced pair of entries. Total debits = total credits, always. Account balances are derived (sum of entries) or materialized for performance. Audit invariant: at any snapshot, the sum across all accounts is zero. This catches bugs that would otherwise hide.

State machine. PaymentIntent state transitions are tightly controlled by a CAS update: UPDATE intent SET status='succeeded' WHERE id=X AND status='processing'. Illegal transitions return error. Every transition emits an event for the audit log.

State Source of truth Allowed next states Recovery behavior
requires_confirmation pi:{intent_id} processing, canceled safe to retry confirm with same key
processing intent + charge row succeeded, failed, unknown do not expose as failed on timeout
unknown charge row + network ref succeeded, failed reconcile via network advice/settlement file
succeeded ledger committed refund/dispute flows only immutable success; no backward transition
failed charge terminal reason new intent or new confirm attempt no ledger movement

Why Spanner-class storage instead of Cassandra: we need transactional state transitions and ledger invariants more than write throughput. Sharded MySQL with a primary per shard is an honest alternative if we can keep account/merchant writes local; Cassandra is better for append-only audit, not authoritative balances.

Exactly-once webhook delivery. Merchants' webhook endpoints can also fail. Outgoing deliveries are persisted; retried with exponential backoff up to 3 days; merchants are expected to dedupe on the event ID (we include it).

PCI compliance. Card numbers (PAN) only ever pass through the tokenization service, which stores them encrypted in an isolated vault. Application code only ever sees tokens. Reduces our PCI scope.

Failover and durability. Single-region failure must not lose any intent. Multi-region active-active with conflict-free state model: intent_id includes region + random component; ledger is per-region for hot writes with global reconciliation.

Refunds. Refund = a separate state machine that mirrors the original charge in reverse. Idempotent on a refund-specific key. Network refund call → ledger entries that reverse the original. Partial refunds need careful arithmetic — track refunded_amount, never exceed charge.amount.

Disputes / chargebacks. Customer reports fraud → network sends a dispute event. We hold the disputed amount from merchant balance, kick off the evidence collection flow. Out of scope to design fully.

Reconciliation. Daily batch: pull network settlement file → join with our charges → assert match → flag mismatches for human review. This is the primary correctness check; an unreconciled day is an incident.

Hot merchant. Black Friday: one merchant doing 1000/sec. Their intents shard on merchant_id, so one shard takes the load. Pre-shard hot merchants across multiple sub-shards if needed.

Edge stress tests.

Edge Failure mode Fallback / invariant
Network timeout charge truth unknown keep unknown; reconcile by network_ref, never create a second charge blindly
Expired idempotency key merchant retries after TTL require client to query by intent/order reference before re-confirming
Hot merchant one shard overloaded pre-shard by (merchant_id, sub_shard) while ledger accounts remain balanced
Multi-region drift same intent confirmed in two regions region-scoped intent owner or global CAS on pi:{intent_id} fences duplicates
Cold archive read 5-year-old charge lookup fetch charge + ledger events from archive; p99 can be seconds, not on checkout path

"The staff-level punch line: the network can be ambiguous, but my ledger cannot be; timeout creates an unknown state, never a second attempt that might move money twice."

What I'd defer. Subscriptions/recurring, multi-currency FX, payment method storage and reuse, fraud scoring depth, fees & revenue share, sub-merchant marketplace flows (Connect).
