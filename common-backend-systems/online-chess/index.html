Online Chess — Chess.com / Lichess Class (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build an online chess platform. Players match into games by skill rating, play with a clock, see opponent's move within ~200 ms, and finished games update Elo. 10M monthly active users, ~500k concurrent during peak, ~50k games/min at peak.

1. Requirements (~5 min)

Functional - Player enters matchmaking; pairs with someone close in rating. - Two players play moves in real-time with a clock per side. - On game end (checkmate, resign, timeout, draw), update ratings.

I'd defer tournaments, anti-cheat depth (engine detection), tutorials, and stream viewing.

Non-functional - Move propagation < 200 ms p95. - Clock fairness — both sides see the same elapsed time; no one loses on time due to lag. - Strong consistency on game state — both sides agree on the position. - Highly available — tournament games are real to people.

Capacity - 500k concurrent connections. - 50k games/min × 30 moves/game = 1.5M moves/min ≈ 25k moves/sec. - Move writes are modest: 25k moves/sec × ~80 B/move × 3 replicas ≈ 6 MB/sec at peak. Connection state and fan-out are the larger operational pressure. - If 5M games finish/day, game metadata at ~1 KB/game is ~5 GB/day; the long-term archive is cheap compared with video/social systems.

2. Core Entities (~2 min)

  • User — user_id, rating, peak_rating.
  • Game — game_id, white_id, black_id, moves[], clock_white, clock_black, status, result.
  • Move — (game_id, ply, uci_or_san, ts_server, time_left).
  • Matchmaking ticket — user_id, rating, time_control, queued_at.

3. API / Protocol (~5 min)

POST  /v1/queue                       { time_control, rating_range }    → { ticket_id }
WS    /v1/queue/{ticket_id}            → push: { matched: game_id }
WS    /v1/games/{id}                   ← move: { uci }
                                       → push: { move, clock_left }
POST  /v1/games/{id}/resign

User from token.

5. High Level Design (~10–15 min)

  1. Matchmaking service. Each time control has its own queue (10s blitz, 5+5, 15+10, classical). Within a queue, players are sorted by rating. A matcher process samples pairs whose ratings are within an expanding window (starts narrow, widens with wait time). On pair: create game row, push WS messages to both clients.
  2. Game session service. All moves for one game funnel to one owner node (consistent hash on game_id). Owner holds the game state in memory + persists to durable storage.
  3. Move validation. Server validates each move against the current position. Rejection on illegal move. Server is source of truth on position and clocks.
  4. Clock authority. Server tracks clocks with monotonic time. Time decremented from the moving player's clock based on server timestamps. Client UI uses its own clock for display but server arbitrates timeout.
  5. Move broadcast. Owner pushes the validated move to both clients over WS.
  6. Game end + rating update. On terminal state, the owner runs Elo update, writes the final game record, pushes game_over to both clients.
  7. Persistence. Move list appended per move (Cassandra keyed by game_id). Game metadata in MySQL (sharded by game_id).
  8. Spectator path (optional). Other users can subscribe to a game's WS in read-only mode; owner fans out broadcast to them with the existing channel.

End-to-end narration: Alice queues for 5-min blitz → matchmaker pairs her with Bob (within 50-rating window) → game G created → both clients open WS to game owner → Alice plays e4 → server validates, decrements her clock by 0.3 s (network), pushes e4 to Bob with clock_white: 4:59.7. Game proceeds for 30 moves; on Bob's checkmate, owner runs Elo, updates both users' ratings, writes the game record.

6. Deep Dives (~10 min)

Concrete stores and shard keys.

Store Key pattern Data structure / entry TTL / retention Shard key
Matchmaking queue queue:{time_control} Redis ZSET, score=rating, member=ticket_id, ~80 B 10 min or until matched time_control
Game owner map game_owner:{game_id} Redis string {owner_node, epoch}, ~60 B active game + 5 min game_id
Game metadata game:{game_id} MySQL row, players/status/result/clocks, ~1 KB forever game_id
Move log moves:{game_id} append-only list/partition ordered by ply, ~50–80 B/move forever, cold archive later game_id
Position checkpoint state:{game_id}:{checkpoint_ply} FEN/clock snapshot, ~200 B active + 7 d hot game_id
Reconnect cursor ack:{game_id}:{user_id} last acknowledged ply, ~40 B active + 1 h (game_id, user_id)
Rating row rating:{user_id} rating/provisional flags, ~100 B forever user_id

The position itself is not stored as a separate authoritative blob on every move; it is derived in memory from the move log and checkpointed occasionally for fast reconnect/replay.

Capacity by store.

Store / index Math Notes
Matchmaking queues 500k waiting/concurrent users × 80 B × Redis overhead ≈ tens of MB split by time control
Game owner map 500k active games × 60 B × 3 ≈ 90 MB logical ephemeral routing index
Game metadata 5M games/day × 1 KB × 3 ≈ 15 GB/day long-term archive is still modest
Move log 25k moves/sec × 80 B × 86,400 × 3 ≈ 518 GB/day if peak sustained actual daily average lower
Checkpoints 500k active × 10 checkpoints × 200 B × 3 ≈ 3 GB speeds recovery, not source of truth
Ratings 10M users × 100 B × 3 ≈ 3 GB tiny but consistency-sensitive

Why Redis ZSET for matchmaking instead of SQL or Kafka: the operation is "find nearby rating in this time-control queue" and remove two tickets atomically. SQL can do it, but row contention around hot rating bands is awkward; Kafka is ordered but not searchable by rating. For moves, Cassandra or an append-only SQL table both work because reads are single-game range scans; the important choice is serializing by game_id, not the brand of storage.

Move path. 1. Client sends {game_id, uci, client_seq} over the player WebSocket. 2. WS gateway routes by game_owner:{game_id}; if missing, it resolves owner via consistent hash and creates an epoch. 3. Owner verifies it is that player's turn, validates the move against in-memory board state, and computes elapsed clock time from monotonic server timestamps. 4. Owner appends moves:{game_id} with (ply, move, server_ts, time_left) and updates game:{game_id} status/clocks. 5. Owner pushes the accepted move to both players; spectator broadcast runs on a separate channel so it cannot delay the player ACK. 6. On terminal state, owner writes final result, updates rating:{white_id} and rating:{black_id} in one transaction or outbox-backed workflow, then emits game_over.

Reconnect / cold game path. 1. Client reconnects with {game_id, last_seen_ply}; gateway routes by game_owner:{game_id} or elects a new owner if the map is absent. 2. Hot owner returns an in-memory snapshot plus moves after last_seen_ply; it also updates ack:{game_id}:{user_id} so later reconnects know the cursor. 3. Cold owner loads the latest state:{game_id}:{checkpoint_ply} and replays moves:{game_id} from that ply. If no checkpoint exists, replay starts at ply 0; that is still cheap for normal games. 4. If the game is finished and archived, GET /games/{id} reads game:{game_id} plus the move archive; latency can be slower because it is no longer on the player move path.

Two-index framing. moves:{game_id} is the durable content log. game_owner:{game_id} is only a routing index, ack:{game_id}:{user_id} is a per-client replay cursor, and rating:{user_id} is a materialized view derived after terminal results. None of those indexes replace the legal move log.

Single-owner per game. All moves and clock decisions serialize at one node. Hash game_id to a session server. On failure, leader election + replay from durable log of moves. The owner enforces ordering, validates moves, and arbitrates clocks.

Why single-owner instead of distributed locks around every move: chess has tiny per-game throughput and very strict ordering, so a colocated owner is simpler and faster. A DB row lock would be correct but adds latency to the hot path and makes clock fairness depend on storage tail latency. Be ready for "why not CRDT?" — collaborative merge semantics do not help because there is only one legal next move.

Clock under lag. Each player has a wall clock the server tracks. When a player's move arrives, server subtracts (server_time - last_move_received) from the player's clock. If clock would go negative, the player has flagged (lost on time). - Server includes time_left with each move push so the opponent UI shows accurate countdown. - Increment time-controls (e.g., +5 s per move) added on each move acknowledgment. - Anti-griefing: small grace window (~150 ms) before flagging to absorb network spikes.

Move validation. Server holds the position; validates each move against chess rules (legal squares, no leaving king in check, castling rights, en passant). On illegal move, reject with reason. Client should also validate locally to prevent obviously illegal submissions.

Disconnects. WS drops mid-game: client has a 30-second grace; on reconnect, server replays moves since the client's last ACK. Server's clock continues to run on the moving player during disconnect.

Matchmaking with rating bands. Maintain a Redis sorted set per time control, keyed by rating. Match scans for pairs within ±N rating; N starts at 50 and widens over time. Trade-off between fast match and fair match.

Rating update. Elo: new = old + K × (actual - expected). K varies by player provisional status. Done in a transaction with the game record write.

Anti-cheat (touched). Move analysis: compare to top-engine moves; consistent agreement at high depth is suspicious. Out of scope to design fully; real platforms use offline batch jobs + human review.

Spectator scaling. A popular game (Carlsen vs Niemann) draws 100k spectators. Server's WS layer separates "player" connections (state-changing) from "spectator" (read-only). Hierarchical fan-out for spectators.

Region choice and cold starts. A game is assigned to one region at creation, usually closest to both players or to the tournament's home region. If players are far apart, both pay some latency, but clocks remain fair because the region's server timestamp is authoritative. After a full owner-pool restart, game owners rebuild active state from checkpoints + move logs and pause clocks briefly with pause credit so nobody flags during recovery.

Game storage. Move list in PGN-like format, compressed (~50 B/game on disk). Long-term retention is cheap — a billion-game archive is < 100 GB.

Graceful degradation.

Case Mode Source Latency budget Fallback
Player reconnect replay moves:{game_id} after ack ply < 1 s resend board snapshot + tail moves
Owner failure cold owner move log + checkpoint 1–3 s pause fence old epoch, replay, resume clocks with pause credit
Spectator hot game latest-only fan-out regional pub/sub < 500 ms latest drop spectator frames; player stream remains strict
Matchmaking sparse expanding bands queue:{time_control} seconds widen rating window, then ask user to continue waiting
Clock dispute server authority move log server timestamps immediate client clock is display only

"The interview punch line: for chess I optimize around per-game correctness, not global write throughput; one game is a tiny serial log with a very demanding clock."

What I'd defer. Tournaments (Swiss/round-robin), analysis engine on demand, computer opponents (Stockfish integration), variant chess rules (960, three-check), club/league features.
