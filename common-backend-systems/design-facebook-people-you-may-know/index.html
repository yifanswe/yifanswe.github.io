1. Basic Knowledge And Solution

What This Question Is Testing

People You May Know (PYMK) suggests friend connections based on graph signals (shared friends, mutual schools/employers, shared interests). The hard part is computing personalized friend candidates for billions of users via expensive graph operations, refreshing as the graph changes, and balancing relevance with privacy.

The invariant is this:

For every user, the system surfaces a personalized ranked list of candidate friends derived from the social graph and additional signals, refreshed regularly, respecting privacy controls and blocks; recommendations don't reveal hidden information (e.g., friend-of-friend should not leak a private friend).

At ICT5 / L6+ level, the interviewer is testing whether you can design offline batch graph processing for friend-of-friend computation, ranking, privacy enforcement, and reason about the scaling and freshness tradeoffs.

60-Second Interview Framing

Spoken version:

I would design PYMK as an offline batch pipeline that computes friend-of-friend candidates per user with shared-attribute signals, plus an online ranker for personalization. Privacy filters apply at delivery so we never reveal hidden friendships. Graph storage is sharded by user_id; pipeline scales via distributed graph computation. The interesting decisions are candidate computation, ranking, privacy enforcement, and freshness.

Requirements

Functional requirements:

  • Surface a personalized list of candidate friends per user.
  • Signals: friend-of-friend, mutual schools/employers, mutual interests, contacts imported.
  • Filter: already-friends, blocked, hidden suggestions.
  • Refresh as graph changes.

Non-functional requirements:

  • Daily refresh (or more frequent for high-activity users).
  • 3B+ users.
  • Recommendations served in sub-second.
  • Privacy: no leak of hidden info.

Out of scope unless asked:

  • Phone book / email contact import privacy details.
  • Detailed ranking model.

Scale Assumptions

Bottleneck: friend-of-friend computation at billion-user scale; expensive but parallelizable.

Offline Pipeline

Daily batch:

  1. Load social graph snapshot.
  2. Compute friend-of-friend per user: for each friend of user U, take their friends. Aggregate counts (number of mutual friends).
  3. Aggregate other signals: shared school, employer, location, mutual interests.
  4. Score candidates per user: weighted combination of signals.
  5. Top-K per user stored in candidate set.

Graph processing frameworks: Spark, Apache Giraph, Google Pregel, custom MapReduce.

For 3B users with hundreds of friends each, total friend-of-friend pairs are massive (trillions). Pruning aggressive: bound at top-K per user.

Friend-Of-Friend Computation

Pseudocode:

for each user U:
  candidates = {}
  for friend F of U:
    for friend G of F (where G != U and G not already U's friend):
      candidates[G] += weight(F)
  top_K = sort_and_truncate(candidates)

Weighted by "strength" of F's connection or recent interaction.

Bipartite from (U, F) to (F, G): matrix-product-like operation; efficient in graph frameworks.

Other Signals

  • Shared school: users in same school within same years.
  • Shared employer: similar pattern.
  • Shared location: in same city/region.
  • Contact import: if both sides imported each other's contact (with consent).
  • Profile interactions: profile visits, group co-membership.

Each signal contributes to candidate score with calibrated weight.

Online Ranker

At request time:

  1. Fetch user's candidate set.
  2. Fetch real-time features (recent profile visits, current page context).
  3. Score candidates with online model.
  4. Filter: already-friends, blocked, hidden.
  5. Apply privacy gate.
  6. Return top N.

Latency: sub-second.

Privacy

Critical. Several rules:

  • Do not show that A is suggested because B (a private friend) connects them.
  • Hidden friendships excluded from friend-of-friend.
  • Blocked users excluded.
  • "People who don't want to be discovered" excluded.
  • "Why this suggestion" must respect display rules — show count of mutual friends but not list, depending on visibility.

Privacy filter is applied at serve time; the candidate set may include private info, but display redacts.

Interview phrase:

Privacy gates apply at display. Hidden friendships influence ranking but never appear in the explanation. The candidate set might know A and B share a hidden friend, but the UI must redact that fact.

Feedback Loop

User dismisses suggestion → "hide" stored; future candidate set excludes.

User accepts → friend edge created.

Strong signals: dismiss reduces score of similar candidates; accept boosts.

Failure Modes And Mitigations

Candidate set stale. Daily refresh; alert on lag. Candidate freshness.
Privacy leak. Strict filter; audit. Privacy incident reports.

Hidden user shown. Filter; respect hide. Hide-but-shown count.
Spam suggestions. Rate limit accepts; suspicious patterns. Spam reports.

Operational Metrics

Correctness:

  • Privacy leaks (zero).
  • Stale-suggestion rate.
  • Block bypasses.

Capacity:

  • Job runtime.
  • Candidate set storage.
  • Service QPS.

Business:

  • Accept rate.
  • Dismiss rate.
  • Friend additions per DAU.

Tradeoffs To Call Out

  • Daily refresh is acceptable; real-time would be expensive.
  • Friend-of-friend is the primary signal; adding more signals adds compute.
  • Aggressive privacy reduces relevance display.
  • More candidates per user = more storage but better online ranking.
  • Contact imports raise privacy concerns.
  • Highly-connected nodes can dominate candidate sets — cap influence.

Common Traps And Stronger Moves

Single signal (FoF only). Multiple signals weighted.
No feedback loop. Hide / accept update candidate scoring.
Show hidden friends in explanation. Redact source if private.
Same candidate forever. Hide TTL or permanent exclusion per user choice.

Interview Phrase

PYMK is offline graph batch + online ranker. Friend-of-friend with weighted shared signals (schools, employers, location) generates candidates daily; online ranker personalizes. Privacy filter at display ensures hidden friends never leak into explanations. Feedback from dismisses and accepts loops back into the pipeline.

2. Simulated Interview

Interviewer: Design People You May Know.

Candidate: Offline batch pipeline computes friend-of-friend candidates per user daily, adding shared-attribute signals (school, employer, location). Top-K per user stored. Online ranker scores at request time with real-time features. Privacy filter at display drops hidden friendships, blocks, and redacts explanations. Feedback from dismiss / accept loops back. The invariant is personalized candidates respecting privacy and blocks.

Interviewer: Friend-of-friend computation.

Candidate: For each user, walk their friends; for each friend's friends, count and weight. Pseudocode: candidates[G] += weight(F) for G in F's friends. Truncate to top-K per user. Done in batch with Spark or Giraph; bipartite matrix-product-like operation. Trillions of (U, G) pairs total across users; pruning aggressive.

Interviewer: Signals beyond FoF.

Candidate: Shared school in same years, shared employer, shared location, contact import (with consent), mutual interests, profile visits. Each signal scored and combined with calibrated weights. Multi-signal blending makes candidates more relevant than pure friend-of-friend.

Interviewer: Privacy.

Candidate: Several rules. Hidden friendships not used in FoF (or used but redacted in explanation). Blocked users excluded. Users opting out of discoverability excluded. "Why this suggestion" UI redacts private friends — show "X mutual friends" but not their names if they're hidden. The candidate set may know the private connection, but display must hide. Hard-coded filter at display; security review verifies.

Interviewer: What about a user with 5000 friends?

Candidate: Highly connected nodes can dominate candidate sets — every one of their friends gets suggested to many. Cap the contribution: weight each friend's influence by inverse of their friend count, or simply cap at a maximum count of friends to consider for FoF.

Interviewer: Hide an unwanted suggestion.

Candidate: User clicks "hide." Hide event recorded; user's hide list updated. Future candidate sets exclude that user. The hide may be permanent or TTL'd. We may also use hides as negative signals to demote similar candidates.

Interviewer: Cold start.

Candidate: New user has no friends yet. Signup signals: school (from form), location, contact import (if granted). Initial candidates from these. Within a few connections, FoF kicks in. Onboarding flow surfaces likely connections aggressively.

Interviewer: A user adds a friend. Should PYMK refresh?

Candidate: New friend should influence future PYMK. Daily refresh picks this up. For high-activity users, more frequent refresh (e.g., every few hours). Real-time incremental update is possible but expensive — usually daily is fine.

Interviewer: Contact import — privacy concern.

Candidate: Sensitive. Requires explicit user consent. The contacts uploaded are matched against the platform's users via hashed identifiers; only matched users surface as candidates. Non-matched contacts not stored beyond the matching session. Both sides should ideally have imported each other for the highest-confidence suggestion. Recent privacy scrutiny has restricted this signal significantly.

Interviewer: Online ranker.

Candidate: Takes candidate set + real-time features (recent visits, current context). Scores with ML model. Filters already-friend, blocked, hidden. Applies privacy redaction to "why" text. Returns top N. Sub-second latency.

Interviewer: Feedback loop.

Candidate: Dismiss → recorded; future candidates exclude; similar candidates may be demoted. Accept → friend edge; the new connection feeds back into next FoF cycle. Other engagement (profile visit, conversation) provides positive signals.

Interviewer: Multi-region.

Candidate: Per-user home region for graph data. Candidate sets replicated to nearby regions. PYMK service per region. Pipeline runs once globally or per-region — depends on graph storage.

Interviewer: Summarize.

Candidate: Offline batch graph pipeline computes friend-of-friend candidates per user daily, blended with shared-attribute signals. Candidate set stored per user. Online ranker scores at request time. Privacy filter at display drops hidden, blocked, and redacts explanations. Feedback loops update via dismiss / accept. The hardest correctness issue is privacy (never leak hidden friendships); the hardest scale issue is the FoF computation at trillions of pairs.

3. Practice Notes

Five-Minute Version

  1. Clarify: signals, freshness, privacy.
  2. Invariant: personalized; private info redacted.
  3. Offline batch FoF + online ranker.
  4. Multi-signal weighted scoring.
  5. Strict privacy filter at display.
  6. Feedback loop from dismiss / accept.
  7. Cold start via signup signals.
  8. Cap influence of super-connected nodes.

One-Minute Closeout

PYMK is offline batch friend-of-friend + online ranker. Daily pipeline computes candidates per user from FoF plus shared-attribute signals. Online ranker personalizes. Privacy filter at display: hidden friendships excluded, blocks excluded, explanations redacted. Feedback from dismiss / accept loops back. Cold start uses signup signals. The hardest correctness issue is privacy — never leak hidden info into explanations.
