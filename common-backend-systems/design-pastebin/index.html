1. Basic Knowledge And Solution

What This Question Is Testing

Pastebin lets users paste a block of text and get a short URL. It is structurally similar to TinyURL but with a few important differences: payloads are larger (KB to MB), expiry is common, syntax highlighting / formatting is expected, and privacy / abuse is real.

The invariant is this:

Every successfully posted paste is durably stored and retrievable via its unique URL until expiry; the URL is unguessable enough to provide reasonable privacy for unlisted pastes; reads are fast; abuse (malware, credentials dumps) is filtered or taken down on detection.

At ICT5 / L6+ level, the interviewer is testing whether you can design a write-once read-many text store, handle expiry, separate metadata from content storage, manage abuse, and reason about hot reads when a paste goes viral.

60-Second Interview Framing

Spoken version:

I would design Pastebin as a write-once content store keyed by a generated paste_id, with metadata in a fast KV and content in object storage. Reads are CDN-fronted because pastes are immutable. The interesting decisions are id generation, expiry handling, privacy levels (public / unlisted / private), abuse detection, and viral paste handling.

Requirements To Clarify First

Functional requirements:

  • User pastes text, optionally with syntax language hint.
  • Gets a short unique URL.
  • Visit URL to see the paste, syntax-highlighted.
  • Optional expiry (10 min, 1 day, 1 month, never).
  • Privacy levels: public (indexed), unlisted (URL only), private (requires login).
  • Edit (optional) or delete by author.
  • Comments (sometimes).

Non-functional requirements:

  • Sub-second paste read.
  • Durable storage; no loss.
  • Scales to billions of pastes over time.
  • Hot-paste read amplification handled.
  • Abuse handling: malware, credentials.
  • Multi-region.

Out of scope unless asked:

  • Collaborative editing.
  • Account-based folders.
  • Advanced ACLs.

Scale Assumptions

Area Assumption
Pastes/day Millions.
Reads/day 100x writes.
Average paste 5-50 KB; some MB.
Active pastes Billions over time.
Latency Read < 200 ms; write < 500 ms.
Hot paste Can get millions of views (e.g., shared on Twitter).
Retention Forever (with optional expiry).

The bottleneck is hot paste reads and total storage growth.

Core Components

Component Responsibility Key detail
API gateway Auth, rate limit. Standard.
Create service Validate, generate id, write. Idempotent.
Id generator Unique short paste_id. Counter-with-range or random.
Metadata store paste_id -> {content_ref, expiry, owner, privacy, syntax} Fast KV.
Content store Actual paste content. Object storage.
CDN Cache rendered/raw pastes. Aggressive TTL because pastes are immutable.
Read service Fetch metadata; serve content; check expiry. Stateless.
Abuse pipeline Scan for malware, credentials, copyrighted material. Async.
Expiry sweeper Remove expired pastes. Background.
Search (optional) Public paste index. Eventually consistent.

Id Generation

Same as TinyURL: counter-with-range-claim or random with conditional insert.

  • Base62 encoding for compact URLs.
  • 7-8 chars typically (3.5T to 218T addressable).
  • Random preferred for unguessable (unlisted privacy).
  • For public, sequential is acceptable but reveals volume.

Write Path

  1. Client POSTs paste content + options.
  2. API gateway authenticates (if logged in), rate-limits anonymous.
  3. Create service generates paste_id.
  4. Compute content hash.
  5. Upload content to object storage.
  6. Insert PasteMeta with content_ref pointing to object.
  7. Enqueue abuse scan.
  8. Return paste_id and URL.

For large pastes, use multipart upload directly to object storage with signed URL, then commit metadata.

Read Path

  1. Client visits /{paste_id}.
  2. CDN checks cache; on hit, returns rendered/raw paste.
  3. On miss, hits read service.
  4. Read service: - Looks up PasteMeta by paste_id. - Checks expiry and abuse status. - Fetches content from object storage. - Renders (syntax-highlight if requested) or serves raw.
  5. CDN caches with TTL up to expiry.

For immutable pastes, CDN TTL can be very long (hours to days).

Caching Strategy

Pastes are immutable; caching is straightforward.

  • Edge CDN caches paste content with Cache-Control: max-age=N.
  • TTL bounded by expiry: min(default_ttl, expires_at - now).
  • Invalidation events for:
  • Author deletion.
  • Abuse takedown.
  • Edit (if supported).

Multi-tier: CDN → regional read service → metadata store + content store.

Expiry

  • expires_at set at creation.
  • Read path checks expiry; expired pastes return 410 Gone.
  • Background sweeper deletes expired metadata and content periodically.
  • Eventually consistent: expired pastes may still appear briefly until sweep runs.

Privacy

Privacy Behavior
Public Indexed for search; appears in lists.
Unlisted Accessible only via direct URL; not indexed.
Private Requires authentication; owner only.

For public, the search index periodically ingests public pastes.

Abuse Pipeline

Pastebin is regularly used for malware, credential dumps, leaked code. Defenses:

  • Content scanner runs after upload: known-bad signatures, credential regex (passwords, tokens), URLs against block lists.
  • Reputation: low-trust users (anonymous, new accounts) get aggressive scanning.
  • User reports trigger manual review.
  • Auto-block on high-confidence detection: status blocked; read returns interstitial.
  • Takedown notices respected (DMCA, etc.).
  • Cache invalidation on block.

Scaling And Partitioning

Consistency Model

Layer Consistency Reason
Metadata write Strong Source of truth.
Content immutable Strong on first write No edits (or new version).
CDN cache Eventually consistent Acceptable for immutable; explicit invalidation on takedown.
Abuse decision Eventually consistent Async scan.
Expiry Eventually consistent Sweep lag tolerable.

Multi-Region Strategy

  • Metadata replicated globally.
  • Content stored in primary region; replicated to others.
  • CDN edges everywhere.
  • Region-local creates with global replication.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Viral paste hot read. CDN absorbs; multi-tier. Origin egress; CDN hit ratio.
Storage growth. Expiry sweeper; cold tiering. Per-tenant storage.
Abuse content. Scanner + manual review; takedown. Time-to-block.
Cache stale after takedown. Invalidation events; short TTL near takedown. Stale-serve count.
Id collision (random). Conditional insert + retry. Collision rate.
Author deletion races with view. Soft delete + invalidation. Delete-vs-view ordering.
Large paste upload fails. Resumable multipart upload. Upload failure rate.
Spam pastes. Rate limit anonymous; CAPTCHA. Anonymous create rate.
Search index lag. Async; bounded. Index lag.
Cross-region replication. Async; bounded. Replication lag.

Operational Metrics

Correctness:

  • Successful read of unexpired pastes (~100%).
  • Stale serve after takedown.
  • Storage growth.
  • Abuse time-to-block.

Capacity:

  • Create QPS.
  • Read QPS, CDN hit ratio.
  • Storage volume.

Business:

  • Active pastes; views per paste.
  • Abuse rate.

Tradeoffs To Call Out

  • Random ids vs sequential: unguessable vs cost predictability.
  • Aggressive CDN TTL vs takedown speed; explicit invalidation bridges.
  • Long retention vs storage cost.
  • Aggressive abuse filtering vs false positives.
  • Strict rate limits vs UX for legitimate users.
  • Stronger privacy modes vs UX (login requirement).

Common Traps And Stronger Moves

Trap Stronger move
Store content in metadata DB. Separate content store (object storage).
Synchronous abuse scan. Async with stub interstitial; takedown after.
Sequential ids for unlisted privacy. Random ids for unguessability.
Forget expiry. Background sweeper + read-time check.
Single CDN tier. Multi-tier with invalidation.
No abuse handling. Scanner + reports + manual review pipeline.

  • CDN for immutable content.
  • Async abuse scan with takedown invalidation.

Pastebin is a write-once content store with metadata in fast KV, content in object storage, CDN for reads. Pastes are immutable so caching is easy; the operational challenge is abuse handling and hot-read amplification when a paste goes viral.

2. Simulated Interview

Interviewer: Design Pastebin.

Candidate: Three pieces: metadata KV mapping paste_id to {content_ref, expiry, privacy, owner}; content object storage holding the actual text; CDN serving reads. Create writes both; read fetches from CDN if cached, falls back to origin. Abuse scanning runs async after create; takedowns invalidate cache. The invariant is durable storage and retrievability until expiry, with abuse content blocked promptly.

Interviewer: Id generation?

Candidate: Two options. Counter-with-range-claim like TinyURL: each create node pre-claims a range of integers, encodes to base62. Sequential, predictable, low collision risk. Or random 7-8 base62 characters with conditional insert. Random is preferred for Pastebin because unlisted privacy assumes unguessable URLs. Sequential ids would let anyone enumerate.

Interviewer: A paste goes viral on Twitter — 100k reads per second.

Candidate: CDN absorbs essentially all of it. The paste is immutable; cache TTL can be hours. Edge POP caches the rendered or raw content. Origin sees a tiny fraction. We monitor per-paste read rate and pre-warm POPs if needed. If origin is hit, it's a single metadata + content read.

Interviewer: Abuse — credentials dump.

Candidate: Async content scanner. Regex for credentials, API keys, known leaked-credential patterns. ML classifier for malware-style obfuscated text. On high-confidence detection, set abuse_status=blocked. Issue CDN invalidation. Read service returns an interstitial instead of content. Manual review for borderline. User-reported pastes go to a review queue. We measure time-to-block.

Interviewer: Expiry?

Candidate: expires_at set at creation. Read service checks expiry on every read — if expired, return 410 Gone and don't fetch content. Background sweeper periodically scans by time bucket and deletes expired metadata and content. The read-time check protects against sweeper lag; the sweeper reclaims storage.

Interviewer: Privacy modes?

Candidate: Public: indexed for search, listed on home pages. Unlisted: accessible by URL only, not indexed. Private: requires authentication, owner-only. The privacy flag is on PasteMeta; read service enforces. Random ids make unlisted reasonably private because URLs are unguessable.

Interviewer: What about edits?

Candidate: Two choices. Either pastes are write-once immutable (and editing means creating a new paste with a forwarding link), or each paste has version history. Pastebin traditionally is immutable; for editable, store a chain of versions and the read serves the latest. Versions makes caching slightly harder.

Interviewer: Storage growth?

Candidate: Billions of pastes over time. Tier old, rarely-accessed pastes to cold object storage. Expiry sweeper removes opt-in-expired ones. Soft delete for author-deleted (retained briefly for moderation). Periodic GC of orphan content blobs.

Interviewer: Multi-region?

Candidate: Metadata replicated globally for fast read of paste existence and access checks. Content in primary region with cross-region replication for high-traffic pastes. CDN everywhere. Creates land in nearest region with async replication.

Interviewer: What stops someone from spamming a million pastes?

Candidate: Rate limits at the gateway: anonymous users get aggressive limits; logged-in users more lenient. CAPTCHA on anonymous burst. Per-IP and per-account quotas. Pattern detection on content for known-spam signatures. Account reputation feeds back into rate limits.

Interviewer: Summarize.

Candidate: Pastebin is a write-once content store. Metadata in fast KV, content in object storage, CDN for reads. Random ids for unguessable unlisted. Async abuse scan with takedown invalidation. Expiry via read-time check plus background sweep. Privacy levels enforce access at read. Multi-region with global metadata and region-primary content. The hardest operational issues are abuse and viral hot reads, both addressed with async scanning and aggressive CDN caching.

3. Practice Notes

Five-Minute Version

  1. Clarify: text payload, expiry, privacy, abuse.
  2. Invariant: durable + retrievable until expiry; abuse blocked.
  3. Metadata KV + content object storage.
  4. Random ids for unguessability.
  5. CDN aggressive caching (immutable content).
  6. Async abuse scan + takedown invalidation.
  7. Expiry: read-time check + sweeper.
  8. Privacy levels enforced at read.

One-Minute Closeout

Pastebin is a write-once content store with metadata in fast KV and content in object storage, fronted by CDN. Random ids give unlisted privacy reasonable unguessability. Abuse is filtered async with cache invalidation on takedown. Expiry is checked at read time and swept in background. Viral hot reads are absorbed by CDN. Multi-region replication keeps the hot path fast everywhere.

Best Interview Phrases

  • "Immutable content makes caching easy; the operational challenge is abuse and viral hot reads."
  • "Random ids for unguessability; counter-based leaks volume."
  • "Async abuse scan with cache invalidation on takedown."
  • "Expiry is read-time check plus background sweep."
  • "Metadata + content split lets us scale storage independently."
