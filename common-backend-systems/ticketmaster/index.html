Ticketmaster — Concert Ticket Sales (walkthrough)

2026/06/01  ·  Common Backend Systems
Contents
Contents

Difficulty: Medium · Total: ~45 min · Framework: Hello Interview Delivery

Scenario in one breath: Build Ticketmaster. An event goes on sale at 10 AM, 100k people hit the queue, and there are 20k seats. Each seat is unique — once sold, it's gone. We need fair queueing, no double-sells, and a snappy seat-picker UI. Steady-state load is small; flash-sale spikes are the whole interview.

1. Requirements (~5 min)

Functional - Browse events and pick a seat from a visual map. - Hold a selected seat for ~10 minutes while the user enters payment. - Complete purchase; seat becomes permanently sold.

I'd defer reselling, subscriptions, and venue-side admin.

Non-functional - No double-sells, ever. Stronger than "five nines" — it's a correctness contract. - Fairness during flash sales — first-in-queue, first-served. - Seat-map page p99 < 500 ms even during a 100k-user surge. - Read-after-write on hold/purchase: the user must see their own hold immediately. - Eventual consistency is fine on availability counts shown to other users (slightly stale is OK; oversold is not).

Capacity - Baseline: 10k events live, 100 reads/sec average. - Flash sale: 100k concurrent users → 100k queue entries, seat-map polled or pushed at ~1Hz → 100k QPS spikes lasting minutes. - Storage is small compared with burst coordination: 10k events × 20k seats/event × ~200 B/seat row ≈ 40 GB primary seat data, ~120 GB with 3 replicas. A single 100k-person queue at ~80 B/member is only ~8 MB; the hard part is fairness and hot-event fan-out, not bytes. - The dominant live cost is the seat-map overlay channel: if 100k clients get a 2 KB diff every 2s, that is ~100 MB/s for one hot sale before CDN/edge fan-out.

2. Core Entities (~2 min)

  • Event — event_id, venue, start, sale_window.
  • Seat — seat_id, event_id, section/row/num, status (available/held/sold), price.
  • Hold — seat_id, user_id, expires_at. TTL-based.
  • Order — order_id, user_id, event_id, seat_ids[], payment_id.
  • QueueTicket — event_id, user_id, position.

3. API (~5 min)

GET   /v1/events/{id}/map           → SeatMap (compressed)
POST  /v1/events/{id}/queue         → { ticket_id, position }
GET   /v1/queue/{ticket_id}         → { position, ready, allowed_until }
POST  /v1/events/{id}/seats/{seat_id}/hold     (idempotency-key required) → { hold_id, expires_at }
POST  /v1/orders                     { holds[], payment } → { order_id, status }
DELETE /v1/holds/{hold_id}

Idempotency keys are mandatory on hold and orders — flaky networks during a sale will retry.

The queue ticket is opaque and signed; it encodes event_id, user_id, queue_epoch, and allowed_until, so the seat service can reject users who skip the waiting room without a synchronous queue lookup on every click.

5. High Level Design (~10–15 min)

  1. Static event catalog. Cached at the edge. Browsing events doesn't touch primary stores.
  2. Seat map service. Renders the venue map. The map is mostly static; only the per-seat status overlay is dynamic. I'd serve the static skeleton from CDN and only fetch a compact {seat_id: status} diff from the API.
  3. Queue service. When an event goes on sale, every arriving user gets a ticket and a position. Implemented as a Redis sorted set keyed by event_id, score = arrival timestamp. The service drips users into "allowed to purchase" state at a rate the seat service can handle (e.g., 1000/sec). Users hold a SSE/WS connection that pushes their position.
  4. Seat inventory service. Owns seat status. Holds are TTL-keyed in Redis (seat:{event}:{seat} value = user_id, TTL 600s). Purchase converts the Redis hold + writes a permanent row in MySQL (sharded by event_id).
  5. Order service. Coordinates payment + finalizes the seat status. Two-phase: reserve payment authorization → finalize seat row → capture payment. On payment failure, rollback the Redis hold.
  6. Notification service. Hold-expiring soon, queue-ready, payment-confirmed.

End-to-end narration: Alice opens the app at 9:59 AM. At 10:00 she taps "Buy tickets" → queue service issues ticket #4520 of 100k → SSE pushes her position every second. At ~10:03 she's allowed in → seat map renders → she picks seat A14 → POST /hold with idempotency key → Redis SETNX with TTL 600 succeeds → she enters payment → POST /orders → payment captured, MySQL row seat A14 sold to Alice committed, Redis key updated to terminal state. If she'd timed out, the Redis TTL would auto-release and the seat returns to available pool, broadcast via the seat-map push channel.

6. Deep Dives (~10 min)

Concrete store contracts.

Store Key / data structure Entry size + TTL Shard key
Queue queue:{event_id} Redis sorted set, member=ticket_id, score=arrival_ts ~80 B/member, TTL sale + 24h event_id
Allowed window allowed:{event_id} Redis set of ticket_ids ~32 B/member, TTL 15 min event_id
Hold hold:{event_id}:{seat_id} Redis string/json {user_id, hold_id} via SET NX EX 600 ~120 B, TTL 10 min event_id
Seat truth MySQL seats(event_id, seat_id, status, version) ~200 B, retained forever event_id, optionally section_id for huge venues
Sold uniqueness MySQL order_seats(event_id, seat_id, order_id) unique (event_id, seat_id) ~80 B, retained forever event_id
Idempotency idem:{user_id}:{key} → prior response ~1 KB, TTL 24h user_id
Seat overlay seat_status:{event_id} Redis bitmap/hash + version 1–4 B/seat, TTL active sale + 1h event_id

Capacity by store. The durable inventory is small: 10k events × 20k seats × 200 B × 3 is ~120 GB. A hot event queue is also small: 100k members × 80 B × 3 is ~24 MB. Holds for a sold-out 20k-seat event are ~2.4 MB primary. The live pressure is network and coordination: a 2 KB seat diff every 2 seconds to 100k clients is ~100 MB/s before edge fan-out, so batching and per-event channels matter more than database bytes. Idempotency is bounded by sale traffic: 100k attempts × 1 KB is ~100 MB for 24 hours on one hot event.

Hold and purchase path trace. Hold path: client → queue entitlement check using signed ticket → idempotency lookup → Redis SET hold:{event}:{seat} NX EX 600 → update seat overlay version → publish batched diff on event:{id}:seat_updates. Purchase path: client → order service checks idempotency → payment authorization → MySQL transaction inserts order_seats(event_id, seat_id) with a unique key → updates seats.status='sold' → captures payment → deletes/marks Redis hold terminal. The Redis hold makes the UI fair; the MySQL unique insert is the correctness line.

Hot and cold API paths.

API path Hot sale path Cold/fallback path Cursor/session state
POST /queue Add signed ticket to queue:{event_id} sorted set; SSE streams position If SSE drops, client polls GET /queue/{ticket_id} with jitter ticket encodes event_id,user_id,queue_epoch,allowed_until
GET /map CDN static venue map + Redis overlay diff by version Dormant event lazily builds overlay from MySQL seat rows since_version for overlay diffs
POST /hold Signed entitlement → idempotency → Redis SET NX EX → overlay diff If overlay is stale, Redis/MySQL truth rejects unavailable seat; user picks again hold_id, 10-min TTL, idempotency key
POST /orders Payment auth → MySQL unique insert/update → payment capture Payment slow extends hold once; DB conflict releases auth and returns sold-out seat order idempotency, payment auth id, seat version

If clients paginate or zoom the seat map past the cached section overlay, they fetch a section-scoped snapshot from MySQL and resume diffs with the returned overlay_version.

Why Redis + MySQL, not only one store. Redis is the right fit for queue positions, expiring holds, and high-frequency overlay diffs because the data is short-lived and latency-sensitive. MySQL is the right fit for sold seats because we need durable transactions, row locks, and unique constraints. A pure Redis design is fast but scary for money; a pure MySQL design is correct but would melt under 100k users polling seat state. Be ready for: "Redis coordinates the sale; MySQL decides ownership."

Compared with Spanner-only, Redis + MySQL keeps the hot, expiring interaction state cheap and close to users while leaving final ownership in a transactional store. Compared with ZooKeeper/etcd locks, Redis holds are simpler at sale scale because the lease is user-facing inventory state, not just a coordination primitive.

Preventing double-sell. The single source of truth is MySQL with a unique constraint on (seat_id, sold=true). Holds are best-effort coordination in Redis to keep users from grabbing the same seat — they're an optimization, not the safety net. A correctly-implemented INSERT ... ON DUPLICATE KEY (or SELECT ... FOR UPDATE) at finalization is the actual guarantee.

Interview punch line: "I don't trust a cache to sell a seat; Redis improves the buying experience, but the database constraint is what prevents the lawsuit."

Queueing at scale. A Redis sorted set per event is the simplest. For very large events, shard the queue into N sub-queues by hash(user_id); each sub-queue drips at rate/N. The seat service rate-limit is the actual bottleneck — the queue is the polite waiting room.

Hot rows in MySQL. Front-row seats are everyone's first click. Shard by event_id so all seats for one event land on the same shard; within the shard, the row-level lock per seat is brief (sub-ms). For the largest events, sharding further within an event by section is reasonable.

Hold expiration. Don't rely on a sweeper to clean up holds. Redis TTL handles it. The seat-map push channel publishes "seat A14 released" when the TTL expires (Redis keyspace notifications or an explicit unhold call from a timer service).

Seat-map updates without melting the network. Don't broadcast every status change to every user. Each open seat-map subscribes to a per-event channel that batches updates every 1–2s. Send compact diffs (only changed seats) not full maps.

Two-view framing. The venue map is static content plus a dynamic availability overlay. The same seat also has two state views: Redis hold state for temporary UX coordination and MySQL sold state for permanent ownership. The overlay is derived from those states; if it is wrong, the hold/purchase APIs still enforce truth.

Bot prevention. Captcha at queue entry; rate-limit by IP + device fingerprint; queue position is signed (HMAC) so it can't be forged; flag accounts that consistently hit the API faster than humans.

Idempotency. Idempotency keys on hold and order survive client retries during flaky cell connections. Server stores (idempotency_key, response) for 24h and returns the cached response on dupes.

Graceful degradation state machine.

Case Mode Source of truth Latency budget Fallback
Hot event Queue drip + batched overlays Redis queue + MySQL seat rows Queue updates 1s, seat holds <100 ms Slow drip rate; show stale availability counts but reject invalid holds
Cold/dormant event Direct seat map load MySQL snapshot + Redis overlay lazily populated <500 ms Build overlay on first request and cache for sale window
Seat-map channel exhausted Pull instead of push seat_status:{event_id} versioned diff 1–2s Clients poll ?since_version= with jitter
Payment provider slow Hold extension once Redis hold + payment auth state +2 min max Extend hold once, then release to avoid inventory hoarding

What I'd defer. Resale market, dynamic pricing, season-ticket bundles, partial refunds, fan-club presales.
