1. Basic Knowledge And Solution

What This Question Is Testing

Ticketmaster is a scarce-inventory booking system under extreme burst traffic.

The hard part is not listing events or rendering venue maps. The hard part is this invariant:

One seat can be sold at most once, even when millions of users arrive at sale start, clients retry, payments time out, caches are stale, and bots attack the system.

At ICT5 / L6+ level, the interviewer is testing whether you can separate fast read projections from authoritative booking state, control burst traffic before it reaches inventory, and explain the exact state transition that prevents overselling.

60-Second Interview Framing

Spoken version:

I would design Ticketmaster around scarce seat inventory and burst control. Browsing and seat-map reads can be served from caches and search indexes, but holds, checkout, and ticket issuance must go through an authoritative state machine. I will first clarify requirements, then cover the read path, reservation path, waiting room, data model, scaling strategy, and failure handling. The deep dive I would expect is seat hold correctness under payment retries and hold expiry.

Requirements To Clarify First

Functional requirements:

  • Users can browse events by artist, venue, city, date, and category.
  • Users can view event details, ticket prices, and a seat map.
  • Users can select one or more seats and place a short-lived hold.
  • Users can complete checkout and receive tickets.
  • Users can see order history and ticket status.
  • The system supports event organizers creating events, venue maps, sale windows, pricing tiers, and ticket inventory.
  • The system needs bot protection, purchase limits, and fraud checks.

Non-functional requirements:

  • Prevent double-selling a seat.
  • Survive massive sale-start bursts for popular events.
  • Keep browsing and search low-latency even during sales.
  • Keep checkout reliable and idempotent under retries.
  • Allow stale availability in browsing views, but never in final booking decisions.
  • Provide auditability for money, inventory, ticket issuance, refunds, and disputes.

Out of scope unless interviewer asks:

  • Full resale marketplace.
  • Dynamic pricing optimization.
  • Barcode scanning at venue entry.
  • Organizer analytics dashboards.

Scale Assumptions

State assumptions out loud so the design has a shape:

Area Assumption
Users Hundreds of millions of registered users.
Events Millions of events, but only a small fraction are hot at once.
Popular sale 1 to 10 million users arrive near sale start.
Inventory A venue has 10,000 to 100,000 seats. Some events use general admission instead of assigned seats.
Read/write ratio Browsing and seat-map reads dominate; hold and purchase writes are much smaller but correctness-critical.
Latency Search under a few hundred milliseconds; hold request around hundreds of milliseconds to a few seconds during bursts; checkout may take seconds because payment providers are involved.
Consistency Strong per-seat consistency for hold and sale. Event search and displayed availability can be eventually consistent.

The bottleneck is not normal database throughput. The bottleneck is a hot event where many users compete for the same small set of seats at the same time.

Core Product Model

Think in terms of three separate surfaces:

Surface User experience System requirement
Discovery Search and browse events. Fast, cacheable, eventually consistent.
Selection View seat map and pick seats. Mostly read-heavy, can show slightly stale availability with clear final validation.
Booking Hold seats, pay, issue tickets. Strongly consistent per seat, idempotent, auditable.

The main design mistake is letting the discovery or seat-map layer become the source of truth. It should only be a projection.

Core Components

Component Responsibility Key detail
Edge, CDN, WAF Serve static assets, absorb traffic, block obvious abuse. Event pages and venue map metadata should be cacheable.
API gateway Authentication, authorization, rate limits, request routing. Enforces per-user and per-device limits before traffic reaches inventory.
Event catalog service Owns event metadata, performers, venue, sale window, price tiers. Not the owner of seat sale state.
Search service Indexes events for discovery by text, location, time, and category. Eventually consistent projection from event catalog.
Seat map service Serves venue layout, sections, seat labels, accessibility info, and price zones. Venue geometry changes rarely; availability changes often. Keep them separate.
Availability projection Shows approximate available, held, sold, or low inventory status. Derived from inventory events; useful for UI, not final authority.
Waiting room Controls admission into hot event purchase flow. Protects inventory service and creates a fairness story.
Seat hold service Creates and expires short-lived holds. Authoritative path for seat state transitions.
Checkout service Validates hold ownership, charges payment, converts hold to sold. Must be idempotent because payment calls time out and users retry.
Ticket ledger Issues immutable ticket records, transfer status, barcodes, and audit log. The ticket is created only after successful checkout.
Event stream Publishes inventory, order, payment, and ticket events. Used to rebuild projections and reconcile failures.
Bot and fraud service Scores users, devices, IPs, velocity, payment risk, and account age. Should run before scarce inventory is touched.

Seat State Machine

This is the core of the design.

available -> held -> sold
     |         |
     |         +-> expired -> available
     |
     +-> blocked

Allowed transitions:

Transition Who can perform it Condition
available -> held Seat hold service Seat is available, sale window open, user has valid admission token, conditional write succeeds.
held -> sold Checkout service Hold belongs to user, hold not expired, payment succeeded, idempotency key accepted.
held -> expired Expiry worker or hold service Current time is after hold_expires_at, hold not converted.
expired -> available Expiry worker Release seat for future holds.
available -> blocked Organizer/admin Seat is reserved, removed, damaged, or held for compliance.

The important implementation detail is conditional update:

UPDATE Seat
SET status = 'held', hold_id = ?, hold_expires_at = ?, version = version + 1
WHERE event_id = ?
  AND seat_id = ?
  AND status = 'available'
  AND version = ?

For multiple adjacent seats, options are:

  • Use a single partition and transaction if all selected seats are colocated.
  • Use a deterministic seat ordering and conditional writes, then roll back if any seat fails.
  • Prefer holding seat groups from the same shard to avoid distributed transactions in the hot path.

Interview phrase:

The seat map may be stale, but the hold transition cannot be stale. The final decision is a conditional state transition on the authoritative seat record.

Write Path: Create Hold

  1. User enters event page and tries to buy.
  2. API gateway verifies auth, sale window, rate limit, and bot score.
  3. If the event is hot, user must present a waiting-room admission token.
  4. Seat hold service validates event_id, seat_ids, user purchase limits, and idempotency key.
  5. Service routes to the inventory shard for event_id or event_id + section.
  6. Inventory store performs conditional writes from available to held.
  7. Hold record is created with short expiry, for example 2 to 10 minutes.
  8. Inventory event is emitted to update availability projections and analytics.
  9. Client receives hold_id and expires_at.

Key point: the response waits for the authoritative hold, not for search index or availability cache refresh.

Write Path: Checkout

  1. Client submits hold_id, payment method, and checkout idempotency key.
  2. Checkout service verifies the hold belongs to the user and is still active.
  3. Create payment intent with the payment provider using an idempotency key.
  4. If payment succeeds, transition hold to converted and seats from held to sold.
  5. Create order and ticket records in the ticket ledger.
  6. Emit order paid and tickets issued events.
  7. Notify user by email, push, and order page update.

Payment timeout handling:

  • If provider returns success, complete the order.
  • If provider returns failure, release the hold or let it expire.
  • If provider times out, mark order as payment_unknown or pending_payment and reconcile with provider before releasing seats.
  • Do not immediately sell the same seats to someone else if the payment may have succeeded.

Interview phrase:

Payment creates an unknown-state problem. I would rather temporarily keep the hold and reconcile than risk charging one user and selling the seat to another.

Read Path: Search And Event Browsing

  1. Client searches for events by query, location, category, and time.
  2. API gateway routes to search service.
  3. Search service reads from event index, not the booking database.
  4. Event detail page reads event metadata from catalog cache.
  5. Seat map geometry is loaded from a CDN or cache because it changes rarely.
  6. Availability overlay is loaded from an availability projection with short TTL.

This path can be eventually consistent. If the UI says a seat looks available but the hold fails, the system returns a clear unavailable response and refreshes the availability overlay.

Waiting Room Design

The waiting room protects the authority path from a traffic spike.

Goals:

  • Admit users at a rate the seat hold service can handle.
  • Prevent queue bypass.
  • Preserve reasonable fairness.
  • Give users progress feedback.
  • Let operators pause, slow, or stop admissions.

Flow:

  1. Before sale start, users enter the waiting room.
  2. System assigns a queue token with user id, event id, issued time, randomization seed, and signature.
  3. At sale start, users are ordered by fairness policy, for example randomized users already in the room plus first-come order afterward.
  4. Admission controller releases signed admission tokens at a controlled rate.
  5. Booking APIs require valid admission tokens for hot events.
  6. Token expires quickly and is bound to user, event, and device/session.

Why not pure first-come-first-served?

  • It rewards bots and fastest networks.
  • It creates an edge traffic stampede exactly at sale start.
  • A randomized waiting room can be more fair and easier to operate.

Admission token:

AdmissionToken(
  token_id,
  event_id,
  user_id,
  admitted_at,
  expires_at,
  max_hold_attempts,
  signature
)

Scaling And Partitioning

Partitioning choices:

Data Partition key Why
Event catalog event_id or organizer Normal metadata reads/writes.
Search index term/location/time shards Query serving and ranking.
Seat inventory event_id + section_id for large events, event_id for smaller events Keeps hot event inventory isolated.
Holds event_id + hold_id or user_id depending on lookup path Need lookup by hold during checkout and by event during expiry.
Orders user_id and order_id indexes Users read order history; support staff uses order lookup.
Tickets event_id + seat_id and ticket_id Venue entry and ownership queries.

Hot-event strategy:

  • Precompute event page, venue geometry, price tiers, and section-level availability before sale start.
  • Put hot events into dedicated inventory shards or isolated capacity pools.
  • Use the waiting room to shape write traffic.
  • Keep search, analytics, and recommendation traffic away from the booking database.
  • For very large venues, split inventory by section so users selecting balcony seats do not contend with floor seats.
  • Use backpressure from inventory shard latency to slow waiting-room admission.

Caching:

  • Cache event metadata and seat map geometry aggressively.
  • Cache availability overlays with short TTL and version numbers.
  • Never trust cache for final hold or checkout.
  • Use event stream updates to invalidate hot sections quickly.

Consistency Model

Strong consistency required:

  • Seat hold creation.
  • Seat sale conversion.
  • Ticket issuance.
  • Refund or void operations that affect ticket validity.

Eventual consistency acceptable:

  • Search results.
  • Event detail page counts.
  • Seat-map availability overlay.
  • Recommendation and popularity ranking.
  • Analytics dashboards.

This is a classic authority/projection split:

Layer Consistency Reason
Inventory store Strong per seat or inventory bucket Prevent oversell.
Hold/order state machine Strong state transitions Retries must converge.
Availability projection Eventually consistent UI hint only.
Search index Eventually consistent Discovery can tolerate lag.

Expiry And Reconciliation

Holds cannot last forever.

Expiry worker responsibilities:

  • Scan or consume hold-expiry tasks by time bucket.
  • Transition active expired holds to expired.
  • Release seats back to available if no checkout is in progress.
  • Emit inventory events to refresh projections.

Avoid a single giant scan:

  • Store holds in time buckets by expires_at.
  • Use delayed queues or a timing-wheel style scheduler.
  • Make expiry idempotent because workers can crash and retry.

Reconciliation jobs:

  • Compare seats marked sold with tickets issued.
  • Compare paid orders with payment provider settlement records.
  • Compare active holds with expiry deadline.
  • Rebuild availability projection from inventory events or current seat records.
  • Quarantine inconsistent orders for manual review.

Multi-Region Strategy

For the first version, use one primary write region per event.

Why:

  • The invariant is per-seat correctness.
  • Active-active writes for the same event create conflict-resolution risk.
  • Most popular events are naturally tied to a venue and sale region.

Design:

  • Route all hold and checkout writes for an event to its owner region.
  • Replicate read projections globally for browsing.
  • Keep CDN and search close to users.
  • Use active-passive failover for the authority region, with clear recovery point and recovery time goals.

If the interviewer requires active-active:

  • Assign seat sections to regional owners, not all regions writing all seats.
  • Or pre-allocate inventory slices per region for general admission.
  • Do not merge conflicting seat sales after the fact; avoid creating the conflict.

Bot Protection And Fairness

Bot protection needs to run before inventory.

Signals:

  • Account age and verification.
  • Device fingerprint and session history.
  • IP reputation and proxy detection.
  • Request velocity and navigation pattern.
  • Payment method reuse.
  • Queue token integrity.
  • Purchase limit per user, household, card, or device.

Controls:

  • Rate limiting at edge and gateway.
  • CAPTCHA or challenge only when risk is high.
  • Signed waiting-room tokens.
  • Per-event purchase caps.
  • Fraud review before ticket transfer or resale.

Tradeoff:

  • Aggressive bot controls may block legitimate users.
  • Weak bot controls let automated buyers consume scarce inventory.
  • The practical answer is progressive friction: low-risk users move quickly, suspicious users face more checks.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Seat is sold twice. Conditional writes on seat state, explicit state machine, idempotency keys, reconciliation alerts. State-transition conflict rate, duplicate event_id + seat_id sold records, ticket issuance mismatch.
Seat map shows stale availability. Treat availability as projection, use TTL/versioning, re-check on hold. Hold failure due to stale UI, projection lag, cache version skew.
Payment times out after provider charged user. Use payment idempotency key, pending/unknown state, provider reconciliation before releasing seats. Unknown payment age, provider mismatch count, delayed conversion rate.
Hold expires while checkout is in progress. Use checkout state transition or short extension with fencing token. Checkout-after-expiry conflicts, expired-hold conversion attempts.
Waiting room bypass overloads inventory. Require signed admission token for hot event booking APIs. Requests without valid token, inventory shard saturation, admission rate vs hold latency.
Expiry worker falls behind. Time-bucketed expiry queues, idempotent workers, alert on expiry lag. Expired active holds, expiry backlog age, artificially low availability.
Event stream drops or delays updates. Durable outbox pattern, replay from audit log, rebuild projections. Outbox lag, consumer lag, projection rebuild time.
Bot attack floods login or queue. Edge rate limits, WAF, device reputation, progressive challenge. Bot score distribution, challenge rate, blocked traffic, false-positive reports.
Primary write region fails. Event ownership failover, freeze hot holds during failover, recover from authoritative log. Failover duration, write unavailability, replication lag.

Operational Metrics

Correctness metrics:

  • Duplicate sold seat count should be zero.
  • Paid order without ticket count.
  • Ticket without paid order count.
  • Active hold past expiry count.
  • Payment unknown-state age.

Traffic and capacity metrics:

  • Waiting-room queue size and admission rate.
  • Seat hold success/failure rate by reason.
  • Inventory shard p95 and p99 latency.
  • Conditional write conflict rate.
  • Checkout conversion rate.
  • Availability projection lag.

Business and trust metrics:

  • Bot challenge rate and pass rate.
  • Purchase-limit violation attempts.
  • Refund/dispute rate.
  • Customer support contacts per hot event.

Tradeoffs To Call Out

  • Waiting room reduces raw immediacy, but it protects the system and improves fairness.
  • Availability overlays can be stale, but final holds must be authoritative.
  • Per-seat strong consistency is necessary; global strong consistency across all events is unnecessary.
  • Short holds reduce inventory lockup, but increase checkout pressure and user frustration.
  • Longer holds improve checkout success, but make inventory look unavailable longer.
  • Single write region per event simplifies correctness, but adds latency for faraway users.
  • Active-active writes sound attractive, but are dangerous for assigned seats unless ownership is partitioned.

Common Traps And Stronger Moves

Trap Stronger move
Treating seat availability cache as truth. Say the cache is a hint; the hold service performs the authoritative conditional transition.
Ignoring the waiting room. Use admission control so hot-event demand does not melt the inventory service.
Saying payment success immediately sells the seat without unknown-state handling. Model payment as an idempotent state machine with pending and reconciliation states.
Using one global lock for an event. Partition by section or seat key; coordinate only the seats involved.
Forgetting general admission. Use inventory buckets and atomic reservation counters instead of seat records.
Assuming exactly-once messages. Use durable outbox, idempotent consumers, and replayable audit events.

Deep Dives Interviewer May Choose

Deep dive Strong answer shape
Seat hold correctness Draw the state machine, conditional writes, idempotency, and expiry behavior.
Waiting room fairness Explain signed tokens, admission rate control, anti-bypass checks, and operator controls.
Payment timeout Explain pending state, provider idempotency, reconciliation, and why seats are not immediately released.
Hot partition Split by event section, isolate hot events, and use backpressure to slow admission.
Multi-region Use event ownership for writes and global replication for reads.
General admission Use atomic inventory bucket reservation with held/sold counters and expiry.

Staff-Level Answer Checklist

  • Start with the invariant: one seat sold at most once.
  • Separate browsing, seat-map display, hold, checkout, and ticket issuance.
  • Name the authoritative store and every derived projection.
  • Draw the seat state machine.
  • Explain waiting-room admission and queue bypass prevention.
  • Explain idempotency for hold and checkout retries.
  • Explain payment unknown-state handling.
  • Explain hold expiry and reconciliation.
  • Show how hot events are isolated from normal browsing traffic.
  • Include operational metrics that detect correctness drift.

Interview Phrase

For Ticketmaster, I would not make the seat map the source of truth. The seat map is a fast projection. The authoritative operation is a conditional transition from available to held, then from held to sold, with idempotency and reconciliation around payment.

2. Simulated Interview

Interviewer: Design Ticketmaster.

Candidate: I will design it around scarce inventory and burst control. The core invariant is that one assigned seat can be sold at most once. Browsing and seat-map views can be eventually consistent, but hold, checkout, and ticket issuance need authoritative state transitions.

Interviewer: What are the main requirements?

Candidate: Users should search events, view event details and seat maps, hold seats, pay, and receive tickets. Organizers create events and inventory. Non-functionally, the system must survive huge sale-start bursts, prevent double sales, handle payment retries safely, and provide auditability for orders and tickets.

Interviewer: What is your high-level design?

Candidate: I would split the system into a read-heavy discovery path and a correctness-critical booking path. Search, event details, and seat-map geometry are served from indexes, caches, and CDN. For hot events, users enter a waiting room. Once admitted, they can call the seat hold service, which performs conditional writes against the authoritative inventory store. Checkout validates the hold, processes payment idempotently, converts seats to sold, and issues tickets through a ledger.

Interviewer: How do you prevent double-selling?

Candidate: Each assigned seat has a state machine: available, held, sold, blocked, and expired. The transition from available to held is a conditional write on the authoritative seat record. If two users race for the same seat, only one conditional write succeeds. The transition from held to sold also checks hold ownership, hold expiry, and order idempotency.

Interviewer: What if a user retries the hold request?

Candidate: The client sends an idempotency key. If the first request succeeded but the response was lost, the retry returns the same hold. If the first request failed because the seat was unavailable, the retry returns the same failure for that key. This prevents duplicate holds and makes client retries safe.

Interviewer: How does the waiting room work?

Candidate: Users enter before sale start. The system assigns signed queue tokens. At sale start, the admission controller releases users at a rate the inventory service can handle. Booking APIs for hot events require a signed admission token bound to user, event, and expiration time. If inventory latency rises, the waiting room slows admission.

Interviewer: What about stale seat maps?

Candidate: That is expected. Availability overlays are projections and may lag. The UI can show a seat as available, but the hold service still performs the authoritative check. If the hold fails, the client refreshes the section availability. This gives low-latency browsing without sacrificing correctness.

Interviewer: Payment succeeds but your service times out. What happens?

Candidate: That is an unknown payment state. I would use a payment idempotency key and mark the order as pending or payment_unknown. I would not immediately release the seats. A reconciliation worker asks the provider for the final payment state. If payment succeeded, we convert the hold to sold and issue tickets. If payment failed, we release or expire the hold.

Interviewer: How do holds expire?

Candidate: Holds have a short TTL, for example two to ten minutes. We store holds by expiry time bucket or delayed queue. Expiry workers idempotently transition active expired holds to expired and release seats back to available. We monitor expired active holds and expiry backlog age because stale holds artificially reduce inventory.

Interviewer: How do you scale for one extremely popular event?

Candidate: I would isolate that event into dedicated capacity, precompute its event page and seat map, and partition inventory by section for large venues. The waiting room controls admission based on inventory shard latency. Search, recommendations, analytics, and notifications are bulkheaded away from the booking authority path.

Interviewer: How would you handle multi-region?

Candidate: I would use one owner write region per event for hold and checkout. Read projections can be replicated globally. For active-active requirements, I would partition ownership by section or pre-allocate general admission inventory slices. I would avoid letting two regions write the same assigned seat.

Interviewer: What would you monitor?

Candidate: Correctness first: duplicate sold seats, paid orders without tickets, tickets without paid orders, active holds past expiry, and payment unknown-state age. Then capacity: waiting-room admission rate, inventory p99 latency, conditional write conflicts, projection lag, and checkout conversion.

Interviewer: Summarize the design.

Candidate: The design separates fast but stale discovery from authoritative booking. A waiting room controls bursts. The hold service owns the available-to-held transition, checkout owns held-to-sold, and ticket ledger records issued tickets. Idempotency, explicit state transitions, expiry workers, and reconciliation keep the system correct under retries, timeouts, and partial failures.

3. Practice Notes

Five-Minute Version

  1. Clarify: assigned seats, hot sale, no double sell, checkout required.
  2. State invariant: one seat sold at most once.
  3. Split read path from authority path.
  4. Draw waiting room before seat hold service.
  5. Draw seat state machine.
  6. Explain conditional writes and idempotency.
  7. Explain payment unknown state and reconciliation.
  8. Close with scaling and observability.

One-Minute Closeout

Ticketmaster is a scarce-inventory system. I would cache event discovery and seat-map views, but keep holds and sales in an authoritative inventory state machine. The waiting room protects hot events, conditional writes prevent double-selling, checkout is idempotent around payment, and reconciliation repairs unknown states. The main tradeoff is that users may see stale availability, but final booking remains correct.

Best Interview Phrases

  • "Availability display is a hint; hold creation is the authority."
  • "I only need strong consistency per seat or per inventory bucket, not globally across the whole platform."
  • "The waiting room is not just user experience. It is admission control for the inventory write path."
  • "Payment timeout is an unknown-state problem, so I need reconciliation before releasing scarce inventory."
  • "Every derived view should be rebuildable from inventory and order events."
