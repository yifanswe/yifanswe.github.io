1. Basic Knowledge And Solution

What This Question Is Testing

A code deployment system pushes code changes from a repo to production servers. The hard part is doing this reliably across thousands of services with various deployment patterns (rolling, blue/green, canary), zero downtime, fast rollback, and integration with CI, monitoring, and feature flags.

The invariant is this:

Every deployed change is traceable to a specific source commit and build artifact; deployments roll out per the configured strategy without dropping traffic; rollback to any prior version completes within minutes; production state always matches a known deployment record.

At ICT5 / L6+ level, the interviewer is testing whether you can design build pipelines, artifact storage, deployment orchestration with strategies, rollback, and reason about safe production changes at organizational scale.

60-Second Interview Framing

Spoken version:

I would design the deployment system as a pipeline from source repo → build → artifact registry → deploy orchestrator → target hosts/clusters. The orchestrator supports rolling, blue/green, canary strategies, with health checks gating progression. Rollback is treated as a re-deploy of a prior artifact. The interesting decisions are artifact immutability, deployment strategies, automated rollback signals, and integration with feature flags.

Requirements To Clarify First

Functional requirements:

  • Trigger deploy from commit/PR/manual.
  • Build artifact from source.
  • Deploy to environments (dev/staging/prod).
  • Strategies: rolling, blue/green, canary.
  • Rollback to prior version.
  • Visibility: who deployed what when.
  • Manual approval gates.
  • Integration with CI tests.

Non-functional requirements:

  • Zero downtime in production.
  • Fast: minutes to deploy.
  • Reliable: failed deploys rollback automatically or surface clearly.
  • Auditable: every deploy traced.
  • Multi-region.
  • Per-service customization.

Out of scope unless asked:

  • Source code review tools.
  • Infrastructure-as-code provisioning.
  • Database migrations beyond mention.

Deployment Strategies

Rolling: - Update N hosts at a time; wait for health; proceed. - Most common; gradually replaces. - Continuous service.

Blue / Green: - Two identical environments. - Deploy to inactive (green). - Switch traffic via LB. - Old environment kept warm for fast rollback. - More resource-expensive.

Canary: - Deploy to small subset (1-5% of fleet). - Monitor metrics. - Gradually expand if healthy. - Automated decision based on success metrics.

Per-service config picks the strategy.

Deployment Flow

  1. CI builds artifact, pushes to registry, triggers deploy (or developer manually triggers).
  2. Orchestrator creates Deployment record in pending.
  3. If env requires approval, wait for human.
  4. Orchestrator validates: artifact exists, dependencies OK, no in-flight deploy for this service.
  5. Begin rollout per strategy: - For rolling: batch update; health-check; proceed. - For canary: deploy to 1%; monitor; expand. - For blue/green: deploy to inactive; switch.
  6. Throughout: health checks; metrics monitoring.
  7. On regression: auto-rollback (if configured).
  8. On success: mark succeeded.
  9. Update service current version.

Interview phrase:

Deploy is a state machine over the artifact's rollout. Strategy determines pacing; health checks gate progression; rollback is just re-deploying the prior artifact.

Health Checks

Multiple layers:

  • Process health: is the new binary running?
  • Service health: does the endpoint respond?
  • Application health: business metrics (latency, error rate).
  • Synthetic checks: scripted user flows.

A deploy step waits for healthy signal before proceeding.

Rollback

Treated as a re-deploy of the prior artifact:

  • Same strategy applies (rolling, etc.).
  • Often faster because prior version is still in artifact registry.
  • For blue/green: just flip the LB back.
  • For rolling: redeploy old version.

Auto-rollback triggers on: - Error rate spike post-deploy. - Latency degradation. - Synthetic check failures. - Manual operator decision.

Decision boundary: many teams require human approval for rollback to prevent flapping.

Canary Analysis

For canary deploys, automated analysis:

  • Deploy to 1% of fleet.
  • Compare metrics (error rate, latency, throughput) between canary and rest.
  • Statistical significance test.
  • If canary is significantly worse: rollback.
  • If similar: expand to 10%, repeat. Then 50%, 100%.

Time window: minutes to hours per stage.

Feature Flags Integration

Feature flags decouple code deploy from feature release:

  • Deploy code with feature flag off.
  • Independently flip flag on for users.
  • If issue: flip flag off; no re-deploy.

This is the most powerful tool for safe production changes — but requires app-level support.

Multi-Region

  • Deploy can target one region or all.
  • Per-region orchestrator.
  • "Wave" strategy: deploy region 1, monitor, then region 2, etc.

Approvals

Manual gates for sensitive environments (prod):

  • Configured per service.
  • Approver receives notification.
  • Approves or rejects with reason.
  • Audit log records.

For repetitive low-risk deploys, can be auto-approved.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Bad artifact deployed. Auto-rollback on health degradation. Error rate post-deploy.
Partial rollout stuck. Timeout per step; alert. Stuck deployment count.
Health check flaky. Threshold over time; not single failure. Flap rate.
Concurrent deploys for same service. Lock per service. Concurrent deploy attempts.
Approval delayed. SLA alerts; auto-cancel after timeout. Pending approval age.
Auto-rollback storm. Throttle; require human after N rollbacks. Rollback frequency.
Artifact registry unreachable. Caching at deploy controllers. Registry availability.
Target host crash mid-deploy. Replace; retry. Host failures.
Cross-region partial deploy. Track per-region state; manual reconcile. Region divergence.
Configuration drift (post-deploy manual changes). Detect; alert; auto-revert. Drift count.

Operational Metrics

Correctness:

  • Failed deploy count.
  • Auto-rollback triggers.
  • Production-state vs deploy-record drift.

Capacity:

  • Deploys per day per service.
  • Deploy duration p95.
  • Rollback duration.

Business:

  • Mean time to deploy.
  • Mean time to recovery.
  • Deploy frequency.

Tradeoffs To Call Out

  • Aggressive canary (1% start) is safer but slower than full rolling.
  • Blue/green is fast switch but resource-expensive.
  • Auto-rollback is fast but can flap; manual is slower but stable.
  • Approvals slow deploys but catch mistakes.
  • Per-service deploy config is flexible but adds complexity.
  • Feature flags decouple but require app support.

2. Simulated Interview

Interviewer: Design a code deployment system.

Candidate: Pipeline from source repo → CI build → artifact registry → deploy orchestrator → target fleet. The orchestrator supports per-service strategies (rolling, blue/green, canary). Health checks gate progression. Rollback is re-deploying the prior artifact. Feature flags decouple code from feature release. The invariant is every deployed change is traceable to a commit, rolls out per strategy, and can be rolled back.

Interviewer: Walk me through a typical deploy.

Candidate: Developer merges PR. CI runs tests, builds artifact (container image), pushes to artifact registry with a content-hash tag. CI triggers deploy via API or pipeline. Orchestrator creates Deployment record. For prod, may wait for approval. Orchestrator picks strategy (say canary). Deploys to 1% of fleet. Waits 10 minutes; checks metrics (error rate, latency). If healthy: expand to 10%, 50%, 100%. If unhealthy: auto-rollback. Each step gated by health checks.

Interviewer: Canary analysis — how decided?

Candidate: Compare metrics between canary hosts and the rest. Error rate, p99 latency, request throughput. Statistical significance test (e.g., is canary's error rate significantly higher than baseline?). If significant degradation: rollback. If similar: proceed to next stage. Time window per stage: 5-30 minutes typically.

Interviewer: Rollback.

Candidate: Same orchestrator infrastructure. Identify prior healthy artifact for the service. Deploy it. For blue/green: just flip the LB back. For rolling: redeploy the prior artifact incrementally. Faster than initial deploy because artifact is cached. Auto-rollback on metric regression; manual rollback by operator command.

Interviewer: A deploy auto-rolled back, then re-deployed, then rolled back again.

Candidate: Flapping. Throttle auto-rollback: after N rollbacks for the same service in a window, require human intervention. This prevents thrashing — sometimes the rollback target also has issues, or the issue is downstream. Alert operators on flap detection.

Interviewer: Feature flags?

Candidate: Decouple code deploy from feature release. Deploy code with new feature behind a flag, flag default off. Test in production with low risk because the new path is inactive. Independently flip the flag on for a percentage of users, monitor. If issue, flip flag off without re-deploying. This is the most powerful safe-change tool — but requires app-level support (feature flag client SDK).

Interviewer: Multi-region deploy.

Candidate: Wave strategy. Deploy to region 1 first; monitor for a window; if healthy, deploy to region 2; etc. Each region uses its own strategy (canary within region). If region 2 fails, region 1 is unaffected and stable. For services with global state (databases), special handling: deploy schema changes first, code that reads new schema later.

Interviewer: Health checks?

Candidate: Layered. Process health (is the binary running?). Service health (does the endpoint respond?). Application health (business metrics: error rate, latency, throughput compared to baseline). Synthetic checks (scripted user flows). All must pass for deploy step to be considered healthy. A flaky single check shouldn't block; threshold over time.

Interviewer: Approvals?

Candidate: Per-service config: which environments require approval, who approves. For prod, often a teammate or release manager approves. SLA on pending approval — if not responded in N hours, alert; possibly auto-cancel after timeout. Audit log records who approved what.

Interviewer: Concurrent deploys for same service.

Candidate: Lock per service. Only one deploy in progress at a time. New deploy request either queues or is rejected with an error message. Aborting the current deploy is possible if operator decides to roll back urgently.

Interviewer: Artifact storage.

Candidate: Immutable, content-addressed (artifact_id is hash of content). Stored in registry (e.g., container image registry or zip storage). Edge-cached near deploy controllers for fast pulls. Kept indefinitely or per retention policy (deletes only after no rollback target needs it).

Interviewer: A deploy fails mid-rollout — half the hosts have new code, half old.

Candidate: Stuck deployment alert. Orchestrator decides: continue (if the few failed hosts are deterministic, e.g., bad host hardware), or rollback. For most cases, auto-rollback. Until resolved, the service runs in a mixed state — which is why backward-compatibility between adjacent versions is important.

Interviewer: Summarize.

Candidate: Pipeline from source → CI build → immutable artifact registry → deploy orchestrator → target fleet. Per-service strategy (rolling, blue/green, canary). Health checks gate progression. Auto-rollback on metric regression with flap prevention. Feature flags decouple code from feature release. Multi-region as wave. Approval gates for sensitive environments. Every deploy traceable end-to-end. The hardest correctness problem is detecting bad deploys fast; the answer is canary analysis with auto-rollback.

3. Practice Notes

Five-Minute Version

  1. Clarify: build, deploy, rollback, environments.
  2. Invariant: traceable; strategy-driven; rollback-able.
  3. Pipeline: source → CI → artifact → orchestrator → fleet.
  4. Immutable artifacts.
  5. Strategies per service: rolling, blue/green, canary.
  6. Health checks + auto-rollback.
  7. Feature flags decouple.
  8. Multi-region wave; approval gates.

One-Minute Closeout

A code deployment system is a pipeline with immutable artifacts and a strategy-driven orchestrator. Rolling, blue/green, or canary per service, gated by health checks. Auto-rollback on regression with flap prevention. Feature flags decouple code from feature release. Multi-region as wave; approvals for sensitive environments. Every deploy traceable to commit and artifact. The hardest correctness problem is detecting bad deploys fast; canary analysis with auto-rollback handles it.

Best Interview Phrases

  • "Artifacts are immutable; deploys are re-pointers, rollback re-deploys prior."
  • "Canary analysis statistically compares the new fleet subset to baseline."
  • "Feature flags decouple code release from feature release."
  • "Auto-rollback on metric regression; throttle to prevent flap."
  • "Multi-region wave: deploy region-by-region with monitoring between."
