1. Basic Knowledge And Solution

What This Question Is Testing

A reminder alert system fires user-scheduled events at the right moment, sometimes years in advance, often with recurrence rules and snooze. The hard part is reliable wakeup at the exact scheduled time, handling timezones, and recovering from outages without flooding users with delayed reminders.

The invariant is this:

Every active reminder fires at most once at or near its scheduled local time on every user device that owns it; missed reminders due to outages either fire promptly with a "delayed" indicator or are explicitly marked missed per policy; recurrence rules expand into discrete occurrences each handled independently; user edits and deletes propagate within seconds.

At ICT5 / L6+ level, the interviewer is testing whether you can model reminders as scheduled occurrences with deterministic ids (similar to cron), handle timezones correctly, scale to billions of reminders, and integrate with the notification delivery system.

60-Second Interview Framing

Spoken version:

I would design the reminder system as a scheduling layer over a notification delivery layer. Each reminder is stored as a definition plus expanded occurrences pre-materialized to a horizon. A time-bucketed scheduler scans for due occurrences and dispatches them to the notification system. The interesting decisions are timezone handling, recurrence expansion, edit/delete propagation, and recovery from missed reminders.

Requirements To Clarify First

Functional requirements:

  • User creates a reminder with title, notes, scheduled time, recurrence, devices.
  • Recurrence: daily, weekly, monthly, custom RRULE.
  • Snooze: postpone by N minutes or until a specific time.
  • Edit and delete (single occurrence or entire series).
  • Per-device delivery: phone push, email, in-app.
  • Timezone-aware; respects user's current timezone.
  • Acknowledgment / dismiss tracking.

Non-functional requirements:

  • Trigger within seconds of scheduled time.
  • Scale to billions of reminders.
  • High availability of scheduling.
  • Edits visible within seconds.
  • Survive scheduler outages without losing reminders.
  • Multi-device sync.

Out of scope unless asked:

  • Calendar event semantics (attendees, invites, RSVP).
  • Smart suggestions based on context.

Component Responsibility Key detail

Reminder store Persistent reminders + recurrence. Per-user partitioned.

Occurrence(

Occurrence Materialization

When a reminder is created or modified:

  1. Compute occurrences for the next horizon (e.g., 30 or 365 days).
  2. For each occurrence, generate occurrence_id = hash(reminder_id, due_at_utc).
  3. Insert into Occurrence table with status=pending if not exists (idempotent on occurrence_id).
  4. Schedule a background task to extend the horizon periodically (e.g., daily nightly job extends each reminder's occurrences forward).

For very long-tail reminders (one-time, far future), insert single occurrence at the scheduled time.

Interview phrase:

Like distributed cron, occurrences are durable rows with deterministic ids. The scheduler is a state machine over those rows. Edits update the rows; fires transition state.

Scheduling And Dispatch

A time-bucketed scheduler (similar to distributed cron):

  1. Time buckets per minute (or finer).
  2. Scheduler scans for occurrences with due_at in the current bucket and status=pending.
  3. Conditional update status = dispatched.
  4. Sends to notification system with idempotency key derived from occurrence_id.
  5. On ack from notification system, status fired.

If the scheduler is briefly down, the next scan picks up missed buckets and dispatches with a "delayed" flag if the lag is significant.

Timezone Handling

Reminders are stored with user's timezone. A "9am Monday" reminder fires at 9am in the user's local timezone, accounting for travel and DST.

  • Reminder stored with timezone and recurrence rule.
  • When materializing occurrences, compute due_at_utc based on the user's current timezone at occurrence time.
  • If user changes timezone, re-materialize future occurrences from new TZ.
  • Edge cases: ambiguous local times during DST transitions (1:30am on fall-back day exists twice) — pick one explicitly.

Interview phrase:

Timezone correctness is the most easily overlooked detail. The reminder "every Monday at 9am" must fire at 9am local even if the user crosses timezones or DST changes.

Snooze

User taps snooze:

  1. Snooze API receives (occurrence_id, until_time).
  2. Update occurrence: status = snoozed, snoozed_until = until_time.
  3. Create a new occurrence (or re-schedule same one) with due_at = until_time.
  4. The scheduler picks it up at the new time.

Snooze does not affect the series; future occurrences from the recurrence rule fire normally.

Edits And Deletes

Edit a single occurrence: change the one row. Edit the series: change Reminder, then re-materialize future occurrences. Existing scheduled occurrences may need to be canceled or rescheduled. Delete the series: set Reminder status to deleted, cancel all pending occurrences.

Edits propagate fast because:

  • Reminder change publishes an event.
  • The materializer rebuilds future occurrences within seconds.
  • The scheduler picks up new occurrences.

Multi-Device Delivery

A user with phone + watch + computer wants all to alert.

  • Reminder definition lists devices (or "all my devices").
  • Dispatch creates one notification per device via the notification system.
  • Each device acks independently.
  • Dismiss from one device propagates to others via sync.

Sync mechanism:

  • Each device subscribes to user's reminder change stream.
  • On dismiss/snooze from one device, server broadcasts to other online devices.

Scaling And Partitioning

Data Partition key Why
Reminders user_id hash Per-user CRUD.
Occurrences due_at time bucket + hash Time-range scan for due occurrences.
Snooze state occurrence_id Lookup by id.
Device subscriptions user_id hash Sync per-user.

Top-of-hour burst (lots of users set 9am reminders): jitter occurrences by a few seconds based on hash(reminder_id) to spread the fire rate.

Consistency Model

Layer Consistency Reason
Reminder definitions Strong User expects edits to take effect.
Occurrences Strong on transitions Idempotency, no double-fire.
Sync to devices Eventually consistent Online devices update fast; offline catch up on reconnect.
Notification delivery At-least-once + dedupe See notification design.

Multi-Region Strategy

  • Per-user home region.
  • Reminder edits route to home region.
  • Occurrences materialized in home region.
  • Scheduler runs in user's home region.
  • Failover: warm replica in alternate region.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Scheduler outage causes missed fires. Backfill from pending occurrences with due_at < now. Pending-past-due count.
Duplicate fire from retry. Idempotent dispatch on occurrence_id. Duplicate fire rate.
Timezone change misses fire. Re-materialize on TZ change event. TZ-mismatch alerts.
Edit not propagated before fire. Edit cancels pending occurrence and re-creates. Edit-vs-fire race count.
User has 1000 reminders all set for 9am. Per-user fan-out rate limit; jitter. Per-user fire rate.
Notification delivery fails. Notification system retries; mark occurrence based on system result. Delivery failure rate.
Recurrence horizon not extended. Nightly horizon-extension job; alert on stale reminders. Reminders without future occurrences.
DST transition. Recompute occurrences around DST boundary; pick policy for ambiguous times. DST-ambiguity alerts.
Snooze followed by delete. Cancel via state transition; idempotent. Snooze-then-delete count.

Operational Metrics

Correctness:

  • Fire-time accuracy distribution.
  • Missed reminders (fired more than threshold late).
  • Duplicate fires.
  • TZ mismatch.

Capacity:

  • Per-minute fire rate.
  • Scheduler scan latency.
  • Notification system lag.

Business:

  • Reminders set per user.
  • Dismiss / snooze ratios.

Tradeoffs To Call Out

  • Pre-materializing occurrences uses storage but makes scheduling robust.
  • Long horizons (1 year) waste storage on never-fired occurrences; short horizons (30 days) require frequent extension.
  • Strict fire-time accuracy requires frequent scheduler ticks; coarse ticks are cheaper but allow lateness.
  • Per-device delivery gives flexibility but requires sync across devices.
  • TZ-aware scheduling adds complexity but is correctness-critical.

Common Traps And Stronger Moves

Trap Stronger move
Single global timer. Time-bucketed scheduler with sharded buckets.
Storing only the recurrence rule. Materialize occurrences; rule is just the source.
Storing UTC only. Store user TZ + recurrence; materialize UTC per occurrence.
Synchronous notification on fire. Async dispatch; notification system owns delivery.
Recompute on every edit. Only recompute future occurrences; past are immutable.
Forgetting horizon extension. Nightly extend; monitor reminders without future occurrences.

Staff-Level Answer Checklist

  • Reminders are definitions; occurrences are durable scheduled events.
  • Time-bucketed scheduler with deterministic occurrence ids.
  • Timezone correctness via stored TZ + UTC materialization.
  • Edits and deletes propagate via occurrence cancellation + re-materialization.
  • Snooze creates a new occurrence; idempotent.
  • Per-device delivery via notification system.
  • Multi-device sync of dismiss/snooze.
  • Burst handling via jitter and pre-materialization.

2. Simulated Interview

Interviewer: Design a reminder alert system.

Candidate: I will design two layers: a scheduling layer that owns reminder definitions and pre-materializes occurrences, and the notification delivery layer that handles per-device push. The invariant is that every active reminder fires at or near its scheduled local time on every user device, with no double-fires and reliable recovery from outages.

Interviewer: Why materialize occurrences instead of computing on the fly?

Candidate: Two reasons. First, it makes the scheduler a state machine over durable rows — same as distributed cron. The scheduler scans for due occurrences and dispatches; recovery from outages is just "find pending past due." Second, it lets edits and deletes propagate cleanly: cancel the pending occurrences and re-materialize. Computing on the fly couples scheduling to the recurrence interpretation on every tick.

Interviewer: Timezones?

Candidate: Store reminder with user's timezone and the recurrence rule. When materializing each occurrence, compute the UTC time based on that local time and the user's current TZ. If the user changes timezones, we re-materialize future occurrences from the new TZ. DST transitions are handled at materialization: "every day at 2:30am" handles fall-back ambiguity per policy (we pick the first occurrence).

Interviewer: A user has 100 reminders, all set for 9am Monday. How do you not fire 100 push notifications at the same instant?

Candidate: Jitter. Each occurrence's due_at is offset by hash(reminder_id) mod jitter_window (say, 30 seconds). The user gets 100 reminders within a 30-second window. Per-user notification rate limits also help; the notification system may digest if the user has too many fires.

Interviewer: What if the scheduler is down for 30 minutes?

Candidate: On recovery, the scheduler scans for occurrences with due_at < now AND status = pending. It dispatches them, marking each as delayed. The notification system shows "Was due at X" so the user knows. We avoid flooding by rate-limiting and digesting if the backlog is large. Per-reminder policy can be "fire late," "skip if more than N minutes late," or "fire next occurrence only."

Interviewer: Snooze?

Candidate: Snooze creates a new occurrence (or updates the existing one) with the new due_at = snooze_until. The state transitions to snoozed. The scheduler picks it up at the new time. Original recurrence series is unaffected.

Interviewer: Edit the recurrence — "change weekly Monday 9am to Tuesday 10am."

Candidate: Update the reminder definition. Cancel pending future occurrences for this reminder. Re-materialize future occurrences from the new rule. Past occurrences are immutable. The materializer respects idempotency: re-running on the same data produces the same occurrence_ids; cancellation marks them.

Interviewer: Multi-device dismiss?

Candidate: When the user dismisses on phone, the phone acks the occurrence. The server updates the occurrence state and broadcasts a sync event to user's other devices via the notification system or a sync channel. Other devices remove the notification from their UI. If a device was offline, it gets the dismiss on reconnect.

Interviewer: Scale — billions of reminders.

Candidate: Reminders partitioned by user_id; occurrences partitioned by (due_at time bucket + hash). The scheduler scans buckets, which keeps the scan size manageable per tick. Hot users with thousands of reminders are bounded by per-user rate limits. The horizon for materialization is bounded (30-365 days) to cap storage.

Interviewer: What about a reminder set for "in 5 years"?

Candidate: Stored with single occurrence at the scheduled time, no materialization beyond the existing horizon. The nightly horizon-extension job picks it up when it falls into the materialization window. Until then, it's just a row in the reminder store waiting for the scheduler to claim it within horizon.

Interviewer: Summarize.

Candidate: Reminders are definitions; occurrences are durable scheduled rows with deterministic ids. A time-bucketed scheduler dispatches due occurrences to the notification system. Timezones are stored with the reminder and respected at materialization. Edits cancel and re-materialize. Snooze creates a new occurrence. Multi-device dismiss syncs across devices. Top-of-hour bursts are handled with jitter. Recovery from outages backfills past-due occurrences with delayed flag.

3. Practice Notes

Five-Minute Version

  1. Clarify: scale, recurrence, timezone, snooze, multi-device.
  2. Invariant: fire at scheduled local time, no double-fire, recovery.
  3. Reminder definitions + materialized occurrences.
  4. Deterministic occurrence_id from reminder_id + due_at.
  5. Time-bucketed scheduler + horizon extension.
  6. Timezone via stored TZ + UTC materialization.
  7. Edits cancel and re-materialize; snooze creates new occurrence.
  8. Multi-device delivery via notification system; sync for dismiss.

One-Minute Closeout

A reminder system is a scheduling layer over notification delivery. Reminders are definitions; occurrences are pre-materialized durable rows. A time-bucketed scheduler dispatches them at the right UTC time, derived from the user's stored timezone. Edits cancel and re-materialize. Snooze creates a new occurrence. Multi-device delivery rides the notification system; dismiss syncs across devices. Top-of-hour bursts get jittered. Outage recovery backfills past-due occurrences with a delayed flag.

Best Interview Phrases

  • "Occurrences are durable rows with deterministic ids; the scheduler is a state machine over them."
  • "Timezone is stored with the reminder; UTC is derived per occurrence."
  • "Edits cancel pending occurrences and re-materialize; past is immutable."
  • "Snooze is just a new occurrence at the new time."
  • "Recovery from scheduler outage is a backfill, not a re-scheduling problem."
