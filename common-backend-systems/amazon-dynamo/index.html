1. Basic Knowledge And Solution

What This Question Is Testing

Dynamo (the 2007 paper, not modern DynamoDB) is the canonical eventually-consistent distributed KV store. It is the proof-of-existence for "always writable" systems built from consistent hashing, sloppy quorums, vector clocks, hinted handoff, anti-entropy via Merkle trees, and gossip. Modern systems (Cassandra, Riak, Voldemort, modern DynamoDB) inherit pieces from it.

The invariant is this:

Every write succeeds on at least W replicas; every read examines at least R replicas; concurrent writes produce siblings reconciled deterministically; the system stays writable even during partitions and node failures, accepting eventual consistency as the tradeoff.

At ICT5 / L6+ level, the interviewer is testing whether you can articulate consistent hashing, the (N, R, W) quorum tunable, vector clocks for causality, hinted handoff and Merkle-tree anti-entropy, and the architectural philosophy of "always writable."

60-Second Interview Framing

Spoken version:

I would describe Dynamo as a leaderless KV store built on consistent hashing for partitioning and replication. Writes go to W of N preference list replicas; reads gather from R. (N, R, W) is tunable per request. Conflicts produce vector-clock siblings that the client merges. Hinted handoff handles short failures; Merkle-tree anti-entropy fixes longer divergence. The architectural choice is availability over strict consistency — always writable.

Requirements The Paper Sets

Functional requirements:

  • Simple Put / Get API by key.
  • Highly available writes during failures and partitions.
  • Multi-data-center.
  • Configurable durability and latency tradeoffs.
  • Used at Amazon for shopping cart, session, customer preferences, etc.

Non-functional requirements:

  • 99.9th-percentile latency under hundreds of ms.
  • High availability — write must succeed even when some replicas are unreachable.
  • Eventually consistent (within seconds typically).
  • Conflict resolution at read time.

Out of scope of the original paper:

  • Strong consistency.
  • Multi-key transactions.
  • Range queries.
  • Secondary indexes.

Scale Assumptions

Area Assumption
Cluster size 10s to 100s of nodes per cluster.
Replication factor N Typically 3.
Quorum W, R Tunable per request; common: W=R=2.
Latency p99 in tens of ms.
Throughput Thousands of ops/sec.

Core Concepts

Consistent Hashing

Nodes and keys live on a ring (hash space).

  • Each node placed at one or more positions (virtual nodes / vnodes) for load balance.
  • A key's coordinator is the first node clockwise from the key's hash.
  • Replicas are the next N-1 nodes on the ring after the coordinator.

Adding/removing nodes shifts only adjacent ranges.

Why vnodes: - Per-node load smoother. - Heterogeneous nodes can be assigned more vnodes. - Bootstrapping a new node is parallel.

Preference List

For each key, the preference list is the ordered list of N+ nodes that should hold replicas. The first N reachable form the actual quorum.

(N, R, W) Tunable

  • N: total replicas.
  • W: minimum replicas that must ack a write.
  • R: minimum replicas that must respond to a read.

If R + W > N: at least one replica overlap between any read and any write — strong consistency (subject to failures).

If R + W ≤ N: higher availability, weaker guarantees.

Common tuning: - N=3, W=2, R=2: read-your-own-write under normal conditions. - N=3, W=3, R=1: strong consistency on read; writes slow. - N=3, W=1, R=1: maximum availability; very eventual.

Data Model

Stored Object

StoredObject(
  key,
  value,
  version_vector,           // vector clock of writes
  context                   // opaque token passed back on next write
)

Vector Clock

A list of (node_id, counter) pairs. Each write by node N increments N's counter.

Comparison: - If vc_A's counters are all ≤ vc_B's, vc_A < vc_B (causally before). - If neither is ≤ the other, they're concurrent (siblings).

Example: - Write 1 by A: vc = [(A, 1)] - Write 2 by A (after read of write 1): vc = [(A, 2)] - Write 3 by B (after read of write 1): vc = [(A, 1), (B, 1)] - Write 2 and Write 3 are concurrent.

Write Path

  1. Client sends PUT(key, value, context) to a coordinator.
  2. Coordinator looks up preference list.
  3. Updates vector clock: take input context's vc, increment coordinator's counter.
  4. Sends Put to N nodes in preference list.
  5. Waits for W acks.
  6. Returns success.

If a replica is down: - Send to next node on ring as a temporary hint. - Hint includes original target. - When original recovers, replay hint to original.

Read Path

  1. Client sends GET(key) to coordinator.
  2. Coordinator queries N nodes in preference list.
  3. Waits for R responses.
  4. If responses have divergent vector clocks, return all siblings + merged context.
  5. Client (or coordinator-side resolver) merges.
  6. Optional read repair: write merged value back.

For sibling merge: - For cart: union of items with max quantity per item. - For session: most recent fields (LWW per field). - For generic: application-defined.

Hinted Handoff

When a write target is unreachable:

  1. Coordinator picks the next reachable node on the ring as a hint holder.
  2. Hint includes original target as metadata.
  3. Hint holder stores the data with a "this is for X" tag.
  4. Periodically tries to replay hints to original.
  5. On success, hint deleted.

If hint holder dies before replay: data lost (unless N>3 and other replicas have it). For high durability, use larger N.

Anti-Entropy: Merkle Trees

For longer-lived divergence (hint holder lost data, partition lasted hours):

  1. Each node maintains a Merkle tree per partition's key range.
  2. Leaves are hashes of (key, value) tuples.
  3. Internal nodes are hashes of children.
  4. Periodically, replicas exchange tree summaries.
  5. Tree-diff: walk down the tree comparing hashes; mismatches identify divergent keys.
  6. For each divergent key, exchange and reconcile.

Cost: tree build is expensive; usually done periodically and incrementally.

Membership: Gossip

Membership changes (node add, remove, fail) propagate via gossip:

  1. Each node periodically picks a peer at random.
  2. Exchanges membership state.
  3. Reconciles differences.
  4. Membership info eventually converges.

Failure detection: gossip heartbeats; a node not seen in T seconds is suspected dead, marked unreachable.

Sloppy Quorum

A strict quorum (W out of N preference list) might block writes if some preference list members are down. Sloppy quorum allows the next N reachable nodes to take the write.

  • Preserves write availability during failures.
  • The "real" replicas catch up via hinted handoff later.

Multi-Data-Center

The paper covers replication across data centers:

  • Replication factor includes nodes in different DCs.
  • Writes typically wait for local DC quorum; replication to other DCs is async.
  • DR via second-region replicas.

Operational Metrics

Correctness:

  • Sibling rate.
  • Hinted handoff backlog.
  • Anti-entropy convergence time.
  • Lost-write count.

Capacity:

  • Per-node QPS.
  • Replication latency.
  • Ring rebalance time.

Tradeoffs To Call Out

  • Availability over consistency: writes always succeed; reads may diverge.
  • Vector clocks expressive but grow without pruning.
  • Sloppy quorum maintains availability but allows temporary divergence.
  • Application-level merge is flexible but error-prone.
  • Hinted handoff trades simplicity for durability concerns.
  • Anti-entropy is heavy but catches all divergence eventually.

Common Traps And Stronger Moves

2. Simulated Interview

Interviewer: Walk me through Dynamo's design.

Candidate: Leaderless KV store on a consistent hashing ring with vnodes. Each key's preference list is the N next nodes clockwise. Writes go to any coordinator, replicate to W of N; reads gather R. Conflicts produce vector-clock siblings reconciled by the client at read. Hinted handoff handles short failures; Merkle-tree anti-entropy handles longer divergence. Gossip propagates membership. The invariant is always writable, eventually consistent.

Interviewer: Walk me through a PUT.

Candidate: Client sends PUT(key, value, context) where context is the vector clock from the last read. Coordinator (any node) hashes the key, finds the preference list, increments its own counter in the vector clock, and forwards the write to N replicas. Waits for W acks. Returns success. If some replicas are unreachable, sends to the next reachable nodes (sloppy quorum) with hints noting the original target. Original target gets the hint replayed when it recovers.

Interviewer: What about a GET?

Candidate: Coordinator queries N replicas; waits for R responses. If all responses have causally comparable vector clocks (one dominates), return the winner. If they're concurrent (siblings), return all of them with a merged context. Client merges per application semantics — for cart, union with max quantity per item.

Interviewer: Vector clocks.

Candidate: List of (node_id, counter) pairs. Each write by node N increments N's counter. To compare: if all counters in A ≤ all in B, A is causally before B; if neither dominates, they're concurrent. Siblings result from concurrent writes. The client passes the context (vc) back on the next write so the system knows what version was last seen.

Interviewer: Vector clocks grow over time.

Candidate: Right. Prune by capping size and dropping oldest entries — but careful, dropping loses causal information. Alternative: prune older-than-N entries by timestamp. The original paper notes this is a real concern. CRDTs offer an alternative for merge-friendly types.

Interviewer: Hinted handoff.

Candidate: When a replica is down, coordinator sends the write to the next reachable node on the ring as a temporary hint. Hint is tagged with the original target. The hint holder periodically tries to replay to the original. On success, hint is deleted. If the hint holder dies before replay, the hint is lost — durability depends on having other replicas.

Interviewer: Anti-entropy via Merkle trees.

Candidate: Each node maintains a Merkle tree per key-range partition. Leaves hash (key, value); internal nodes hash children. Replicas exchange tree summaries; walking down the tree, mismatched hashes localize divergent keys. Exchange the differing keys, reconcile. Used for longer-lived divergence that hinted handoff didn't catch.

Interviewer: Sloppy quorum vs strict.

Candidate: Strict: writes go to exactly N preference list members; if some unreachable, fewer than W ack, write fails. Sloppy: when preference list members are unreachable, sub in the next reachable nodes. Write succeeds but on potentially temporary holders. Trade: write availability for temporary divergence (which anti-entropy eventually resolves).

Interviewer: (N, R, W) trade-offs.

Candidate: N=3 typical. W=2, R=2 gives R+W>N — at least one overlap → strong consistency under no-failure. W=1: write fast, very eventually consistent. W=3: write slow but durable. R=1, W=N=3: read-fast strong consistency. Tuning per workload.

Interviewer: Multi-data-center?

Candidate: Preference list can span DCs. Common pattern: writes wait for local DC quorum (fast), async replicate to other DCs (eventual). DR fails over to another DC. Some strict-consistency configurations wait for cross-DC quorum, paying latency.

Interviewer: Adding a node.

Candidate: Pre-assigned vnode positions. New node joins ring at those positions. Bootstrap: stream key ranges from neighbors. Parallelism via vnodes — many neighbors stream simultaneously. Membership change propagates via gossip. Throttle bootstrap to not saturate.

Interviewer: Gossip.

Candidate: Each node periodically picks a peer at random, exchanges membership state, reconciles. Eventually consistent membership. Failure detection via heartbeats; a node not seen in T seconds is suspected dead. Suspected nodes can recover; the system errs toward continuing rather than declaring dead.

Interviewer: Summarize.

Candidate: Consistent hashing partitions and replicates; (N, R, W) tunes durability and latency per request; vector clocks make causality explicit so conflicts surface as siblings for app-level merge; hinted handoff and Merkle-tree anti-entropy reconcile divergence; gossip propagates membership. Sloppy quorum maintains write availability under failures. The whole design favors availability over strict consistency — an explicit choice for use cases like the cart.

Dynamo is the canonical eventually-consistent leaderless KV store. Consistent hashing with vnodes partitions and replicates. (N, R, W) tunes per-request durability and latency. Vector clocks surface concurrent writes as siblings for app-level merge. Hinted handoff handles short failures; Merkle-tree anti-entropy handles longer divergence. Sloppy quorum keeps writes available under partial failures. The architectural choice is availability over strict consistency.
