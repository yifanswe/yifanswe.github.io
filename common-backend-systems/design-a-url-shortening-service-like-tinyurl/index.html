1. Basic Knowledge And Solution

What This Question Is Testing

A URL shortener looks trivial — accept a long URL, return a short code, redirect on lookup. The hard part is keeping the system correct and fast at scale.

The invariant is this:

Every short code maps to exactly one canonical long URL for its entire lifetime, and the same long URL can be re-used by many distinct short codes; lookups must be cheap, idempotent, and globally consistent enough that no two creators ever receive the same code.

At ICT5 / L6+ level, the interviewer is testing whether you can pick a code generation strategy that avoids collisions at scale, separate the write path from a read-mostly redirect path, and explain how analytics, abuse, and link expiry are layered on without slowing the hot redirect.

60-Second Interview Framing

Spoken version:

I would design TinyURL around two paths: a low-volume create path that allocates a globally unique short code, and an extremely high-volume redirect path that does a single key lookup. The interesting decisions are how I generate codes without collisions, how I cache the redirect path, and how I protect the system from malware and analytics-driven hot keys. I will clarify requirements, define the data model, walk the write and read paths, then go deeper on code generation and cache strategy.

Requirements To Clarify First

Functional requirements:

  • Users can submit a long URL and receive a short code.
  • Users can optionally request a custom alias.
  • Visiting a short URL redirects to the long URL.
  • Users can see basic analytics for their links: total clicks, recent clicks, geographic distribution.
  • Links can optionally have an expiration time or a one-time-use flag.
  • Admins can disable malicious links.

Non-functional requirements:

  • Redirect latency well under 100 ms at the edge.
  • No two creators ever receive the same code for different long URLs.
  • The same long URL submitted twice may produce different codes; deduping is optional.
  • The redirect path must keep working even when the create path is degraded.
  • Analytics may lag by seconds to minutes; redirects cannot lag.
  • Auditability for who created which link and when, for abuse review.

Out of scope unless interviewer asks:

  • Branded vanity domains and per-tenant white labeling.
  • Deep link routing into mobile apps.
  • A/B testing or dynamic destinations.
  • Full link-in-bio profile pages.

Scale Assumptions

State assumptions out loud so the design has a shape:

Area Assumption
Writes 100M new links/day at peak, roughly 1k/s average, 10k/s spike.
Reads 100x to 1000x writes; ~100k-1M redirects/sec peak globally.
Object size Long URL up to ~2 KB; short code 6–8 chars.
Retention Default 5 years; some links never expire.
Hot keys Viral links can dominate; one URL can receive most traffic for hours.
Latency Redirect p99 under 50 ms from edge; create p99 under 300 ms.
Consistency Strong uniqueness for code allocation. Eventual consistency for analytics.

The bottleneck is not write throughput. The bottleneck is the global uniqueness guarantee on code generation and the read-side cache hit rate for viral links.

Core Product Model

Think in terms of three separate surfaces:

Surface User experience System requirement
Create Submit long URL, get short code. Strongly consistent code allocation, low write volume, idempotent on client retry.
Redirect Visit short URL, land at long URL. Extreme read scale, cacheable, must work read-only during create outages.
Analytics View click counts and trends. Eventually consistent, heavy aggregation, separate from the redirect hot path.

The main design mistake is letting analytics or feature flags sit on the synchronous redirect path. The redirect should be a single key lookup plus a 301/302 response.

Core Components

Component Responsibility Key detail
Edge / CDN Cache 301/302 responses for hot short codes; absorb DDoS. Hot viral links should rarely reach origin.
API gateway Authentication, rate limits, abuse checks on create path. Anonymous create may be allowed but heavily limited.
Create service Validate long URL, allocate code, write to link store. Idempotent on Idempotency-Key header.
Code allocator Produces a globally unique short code. The critical correctness component. Counter-based, hash-based, or pre-allocated pool.
Link store Authoritative short_code -> long_url mapping plus metadata. Read-heavy. Key-value or columnar store.
Redirect service Resolves a short code to a long URL and emits a click event. Stateless; horizontally scalable; reads from cache then store.
KV cache In-memory LRU for the hot working set. Short codes have a long-tail and a fat head — caching is very effective.
Click stream Async pipeline for click events: timestamp, code, geo, referrer, agent. Never blocks the redirect response.
Analytics service Aggregates clicks for dashboards and exports. OLAP store, hourly/daily rollups.
Abuse and safety Scans newly created and lookup-time URLs against blocklists. Async on create, with a denylist consulted by redirect service.

Data Model

Link

Link(
  short_code,         // primary key, 6-8 chars base62
  long_url,
  owner_user_id,      // nullable for anonymous
  created_at,
  expires_at,         // nullable
  status,             // active, disabled, expired
  is_custom_alias,    // bool
  click_count_approx, // denormalized counter, eventually consistent
  safety_status,      // unknown, safe, flagged, blocked
  version

short_code is the partition and primary key. The store does not need a secondary index for long_url unless deduplication is required.

Idempotency: the same Idempotency-Key for the same user returns the same short_code. This makes client retries safe and prevents allocating two codes for what the user perceives as one create.

Code Generation Strategy

This is the core deep dive. Three viable approaches:

Approach A: Counter + base62 encoding

  • A central or per-shard counter increments on each create.
  • Encode the integer to base62 (0-9 a-z A-Z); a 6-char code covers ~56 billion; 7 chars covers ~3.5 trillion.
  • To avoid contention, each create node pre-claims a range (e.g. 10,000 integers) and serves locally until empty.
  • The range claim is the only synchronous step; it uses a transactional counter or ZooKeeper-style sequencer.

Pros: deterministic, no collision, predictable code length growth. Cons: codes are guessable and enumerable; competitors and bots can scrape sequentially.

Approach B: Random + collision check

  • Generate 7-8 random base62 characters.
  • Conditional insert: INSERT IF NOT EXISTS short_code = ?.
  • On collision, generate a new random code and retry.

Pros: codes are unguessable; no coordination. Cons: collision rate rises as keyspace fills; bounded by birthday paradox; requires conditional writes.

Approach C: Hash of (long_url, salt) truncated

  • Hash with SHA-256, truncate to 7-8 base62 characters.
  • Same long URL with same salt gives same code, so it provides automatic dedup.
  • Collisions are still possible; resolve with extra bits or counter suffix.

Pros: built-in dedup for the same submitter; no central counter. Cons: collision handling is awkward; the same long URL across users may yield the same code, which leaks information about other users' links.

Interview phrase:

I would use a counter-with-range-claim approach by default for predictability, plus a random shuffle to mask the sequence so scraping does not reveal active codes. If the product requires unguessable codes, I switch to random with conditional insert and accept the retry cost.

Write Path: Create Link

  1. Client POST /links with Idempotency-Key.
  2. Gateway enforces auth, rate limit, blocklist on the long URL host.
  3. Create service checks the idempotency table; if present, return the prior short_code.
  4. Allocate a code: - If the local range has integers, take the next one and encode. - If exhausted, claim a new range from the global allocator.
  5. Insert (short_code, long_url, owner, ...) into the link store. For random codes, use conditional insert and retry on collision.
  6. Write the idempotency record.
  7. Push the new link to safety scanners asynchronously.
  8. Return short_code and full short_url.

The synchronous critical path is one counter increment plus one insert. Both are O(1).

Read Path: Redirect

  1. Client visits https://short.example/abc123.
  2. Edge CDN checks its cached redirect. On hit, return 301 immediately and emit an edge click event.
  3. On miss, hit a regional redirect service.
  4. Redirect service checks KV cache.
  5. On cache miss, read from link store.
  6. Check status and safety_status. If disabled or blocked, return a safety interstitial, not a redirect.
  7. Respond with 301 Location: <long_url> (or 302 to force analytics on every hit).
  8. Fire and forget a click event to the stream.

The response wait is one cache lookup. Click writing is fully asynchronous.

301 vs 302:

Code Behavior Analytics impact
301 Permanent Browsers and intermediaries cache aggressively. Lowest latency. First click counted; subsequent clicks from same client may bypass origin.
302 Found No caching by default. Every click reaches origin and is counted.

Most production short link services choose 302 for analytics fidelity and tolerate the lost edge caching. If the system needs both, you can serve 301 with a short Cache-Control: max-age=N and accept bounded analytics loss.

Caching Strategy

The redirect workload has extreme skew: a small percentage of links generate the majority of traffic.

  • Multi-tier caching: browser HTTP cache, CDN edge cache, regional KV cache, in-process caches.
  • Cache key is the short code. Cache value is the long URL plus status and safety flag.
  • TTLs are long (hours to days) because mappings are immutable. The only invalidation cases are status changes (disable, expire, safety block) and updates.
  • On status change, publish an invalidation event to a fan-out channel that each cache node listens to.
  • For viral links, pre-warm caches when a sudden burst is detected.

Interview phrase:

Caching here is unusually effective because the mapping is immutable in the common case. The interesting question is invalidation when a link is disabled for safety, which should happen within seconds globally.

Scaling And Partitioning

Data Partition key Why
Link store short_code (hash partitioned) Even distribution; the natural lookup key.
Counter range claims Single small table, conditional update Low volume, but must be transactional.
Click events short_code for short-window aggregation, then daily by date Hot-key spread for viral links uses sampling or sub-partitioning.
Analytics rollups short_code + bucket Pre-aggregated counters.
Owner index owner_user_id secondary index or separate table For "my links" lookups.

Hot-key handling:

  • For a viral link generating millions of redirects per minute, cache near the user and serve from CDN.
  • Sample click events to avoid overloading the click pipeline; sampled events are scaled up for aggregates.
  • Per-IP and per-user-agent rate limits at the edge.

Consistency Model

Layer Consistency Reason
Code allocation Strong (within allocator) Prevent duplicate codes.
Link store insert Strong primary-key conditional write Prevent code reuse.
Cache Eventually consistent with explicit invalidation events Acceptable lag for the immutable mapping; safety updates need fast propagation.
Click counters Eventually consistent Dashboard freshness is per-minute or per-hour.
Safety decisions Eventually consistent on create, strong on lookup once decided A link found malicious must stop redirecting quickly.

Expiry And Lifecycle

  • Links may have expires_at. The redirect service checks the field on every miss read from the store; the cache record includes the expiry so it can self-invalidate.
  • Soft delete on user request: set status = disabled, broadcast invalidation.
  • Garbage collection of expired links runs as a slow background job; the rows can stay indefinitely but be marked.
  • One-time-use links: redirect service performs a conditional UPDATE status FROM active TO used before responding; second visitor sees a "link already used" page.

Multi-Region Strategy

The redirect path is read-mostly and benefits from active-active across regions.

  • Asynchronously replicate the link store globally. Stale replica reads are acceptable for redirects because the mapping is immutable after creation.
  • Code allocation: each region claims independent counter ranges (e.g., region-tagged prefixes or interleaved integer ranges) so creates can be local without cross-region coordination.
  • Invalidation events for disabled or blocked codes broadcast through a global pub/sub channel with low end-to-end latency target (seconds, not minutes).
  • Failover: if a region loses its store, redirect traffic continues from cache; creates fail or fall back to a peer region.

If active-active causes problems for custom aliases (two users picking the same alias in different regions), funnel custom-alias creates through a single region or take a global lock per alias.

Abuse, Safety, And Privacy

URL shorteners are weaponized for phishing and malware.

Controls:

  • Blocklist check on submitted host at create time (known phishing domains, sanctioned hosts).
  • Async deeper scan after create: fetch the destination, render in a sandbox, classify.
  • Reputation scoring of submitter (account age, prior abuse).
  • Rate limit anonymous creates aggressively.
  • Interstitial warning page for newly scanned but unverified links.
  • Manual review queue for user reports.
  • Quick global takedown: a denylist that every redirect node consults; a redis-backed bloom filter for low-latency negative cache; explicit invalidation on add.

Privacy:

  • Hash IP addresses before storing click events.
  • Honor Do Not Track and regional privacy regimes for analytics.
  • Owner-only access to click streams; do not leak click info to third parties.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Two creators receive the same code. Conditional inserts on short_code; range allocator transactional; collision retry on random scheme. Insert-collision rate, duplicate-code alerts.
Code generation contention storm. Per-node range pre-claim; lock-free allocator; multiple parallel sequencers. Allocator p99 latency, range-claim error rate.
Cache poisoning with malicious redirect. Signed cache entries; invalidation requires write through the store. Mismatch between cache and store, unauthorized invalidation events.
Viral link saturates origin. Aggressive edge caching; cache-as-301 with TTL; rate limit at edge. Origin QPS per code, edge cache hit ratio, top-N hot codes.
Click pipeline backs up and loses events. Sampled ingest; durable buffer; replay from logs. Pipeline lag, drop count, replay coverage.
Phishing link not blocked fast enough. Global denylist with pub/sub invalidation, cached negatively. Time-to-blocklist after report, false-positive rate.
Store hot partition on short_code. Hash partition keyspace; pre-split ranges; check for sequential-code skew. Per-partition QPS distribution, p99 by partition.
Counter region claim collisions across regions. Region-tagged prefix bits or non-overlapping integer ranges. Duplicate code rate per region pair.
Analytics dashboard shows wrong numbers. Idempotent aggregation; periodic reconciliation from raw event store. Daily total mismatch between live dashboard and recompute.

Operational Metrics

Correctness metrics:

  • Duplicate short codes: must be zero.
  • Inserts failing due to collision (should be very low for counter, bounded for random).
  • Redirect responses with stale safety status.
  • Disabled link still redirecting (cache invalidation lag).

Capacity metrics:

  • Redirect QPS by region, percentile latency.
  • Edge cache hit ratio overall and for top-100 codes.
  • Allocator range-claim rate and contention.
  • Click pipeline lag.

Business metrics:

  • Creates per day, retention of created links, share rate.
  • Abuse takedown time-to-block.
  • Per-creator anomaly detection (sudden burst of submissions).

Tradeoffs To Call Out

  • Counter-based codes are predictable and easy; they leak link volume to attackers.
  • Random codes are unguessable but require collision handling and waste keyspace.
  • 301 redirects are fast but lose per-click analytics; 302 captures every click but costs every hit at origin.
  • Edge caching speeds redirects but slows safety takedowns; explicit invalidation is required.
  • Dedup of the same long URL saves space but couples unrelated users; usually not worth it.
  • Multi-region active-active for reads is easy; for custom alias creates it is risky.

Common Traps And Stronger Moves

Trap Stronger move
Single global atomic counter for every create. Range-claim allocator: amortize coordination across thousands of creates.
Storing click counts in the same row as the link mapping. Separate counter store; aggregate from event stream.
Treating the cache as the source of truth. Cache is a hint; the store is authoritative. Invalidations are explicit.
Forgetting safety state in the redirect path. Each cached entry includes safety status; takedown publishes invalidation.
Using base64 instead of base62. URL-safe charset matters: avoid +, /, and = padding.
Letting custom aliases collide with auto-generated ones. Separate namespaces or reserved prefix; or check both spaces on create.

Deep dive Strong answer shape

Staff-Level Answer Checklist

  • Start with the invariant: unique short code, immutable mapping.
  • Distinguish the create path (low volume, strong consistency) from the redirect path (extreme reads, cacheable).
  • Pick a code generation scheme and justify it.
  • Show the cache hierarchy and how invalidation works.
  • Explain idempotent create.
  • Explain abuse handling and safety takedown.
  • Show how multi-region works for reads and what gets harder for custom aliases.
  • Include analytics as a non-blocking side path.
  • Identify viral links as the hot-key risk and address it.

Interview Phrase

For TinyURL the trick is to recognize the workload asymmetry. The create path is rare and needs correctness around uniqueness; the redirect path is the majority of traffic and needs to be a single cached key lookup. Everything else, from analytics to safety scanning, layers off the redirect path asynchronously.

2. Simulated Interview

Interviewer: Design TinyURL.

Candidate: I will design it around two distinct paths: a low-volume create path that allocates a globally unique short code, and a very high-volume redirect path that is essentially a key lookup. The core invariant is that every short code maps to exactly one canonical long URL forever.

Interviewer: What are your scale assumptions?

Candidate: I would assume around 100 million creates per day, peak roughly 10,000 per second, but reads are 100 to 1000 times higher, so up to a million redirects per second globally. Most traffic concentrates on a small number of viral links.

Interviewer: How do you generate the short codes?

Candidate: I would use a counter-based allocator with range pre-claiming. Each create node reserves, say, 10,000 integers from a transactional counter, then encodes them to base62 locally. That makes the synchronous coordination one range claim per ten thousand creates, not one per create. If unguessability matters, I would switch to random 7-character codes with conditional insert, accepting a small retry rate due to collisions.

Interviewer: How long should the codes be?

Candidate: Base62 with 7 characters gives about 3.5 trillion codes, which is enough for many decades of growth at 100 million per day. Six characters covers about 56 billion and could last roughly two decades. I would start at 7 and reserve the option to grow.

Interviewer: What does the redirect path look like?

Candidate: Client hits the short URL at the edge. The CDN may have a cached 301 response; if so, it serves immediately. On a miss, the request reaches a regional redirect service that checks a KV cache and falls back to the link store. The response is a redirect to the long URL plus a fire-and-forget click event to an async stream.

Interviewer: 301 or 302?

Candidate: That is a real tradeoff. 301 is permanent and aggressively cached by browsers and intermediaries, which gives the lowest latency but loses analytics fidelity because repeat clicks bypass the origin. 302 is uncached by default and ensures every click is counted but pushes load to origin. Most production shorteners pick 302 for analytics and absorb the cost with caching at the redirect tier. I would go with 302 and a strong cache layer.

Interviewer: A link goes viral and gets 100,000 hits per second. How does the system survive?

Candidate: Edge cache absorbs most of it because the mapping is immutable. Each CDN PoP stores a 301-or-302-with-short-TTL for the hot code; the origin sees maybe a few thousand QPS instead of 100,000. We monitor top-N hot codes and pre-warm regional caches when burst detection fires. The click event stream uses sampling for hot codes and corrects with a scale factor during aggregation, so the analytics pipeline does not back up.

Interviewer: A link is reported as phishing. How fast can you stop redirects?

Candidate: A global denylist with pub/sub invalidation. The takedown writes status to the link store, publishes an invalidation event, and every cache node and CDN POP receives it within a few seconds. Each cached entry carries safety status, so on the next request after invalidation it shows the interstitial. Targeting under five seconds globally is realistic.

Interviewer: What about custom aliases?

Candidate: Custom aliases use a conditional insert on the short_code column. If the alias is taken, return a clear conflict. Custom aliases share the same namespace as auto-generated codes, so I would reserve a prefix or character pattern to keep allocator codes from accidentally producing something a user wanted. In multi-region, custom alias creates go through a single owner region or use a global lock to avoid two regions claiming the same alias.

Interviewer: How does idempotency work on create?

Candidate: The client sends an Idempotency-Key header. The create service has an idempotency table keyed by user plus key. If a duplicate arrives, it returns the previously allocated short code rather than allocating a new one. This makes client retries safe and prevents the user from getting two short codes for what they think is one create.

Interviewer: How do you partition the link store?

Candidate: Hash partition on short_code. The lookup pattern is exact-match on the key, so a hash partition gives even distribution. Sequential codes from the counter would create hot partitions if I used range partitioning, but hashing flattens that. Secondary indexes for owner queries go in a separate per-owner table.

Interviewer: What if a region goes down?

Candidate: Redirects continue from cache and from any healthy region's replica. Creates fail or are routed to a peer region. Because each region pre-allocates its own counter ranges or uses region-tagged prefix bits, there is no risk of two regions producing the same code during the outage. When the region recovers, replication catches up and we replay any safety invalidations through the pub/sub channel.

Interviewer: How do you handle the click event pipeline at scale?

Candidate: Click events go to a high-throughput append-only log. For viral codes, the edge samples events at a known rate and the aggregation pipeline multiplies by the sample factor. The pipeline batches events into hourly and daily rollups stored in an analytical store. The redirect path never blocks on writing a click; if the pipeline is degraded, redirects still work.

Interviewer: Summarize.

Candidate: Two paths: rare creates that must produce unique codes, and frequent redirects that are key lookups. A range-claim counter allocator gives uniqueness without coordination on every write. The redirect path is dominated by cache; safety takedown rides a pub/sub invalidation. Analytics is async and sampled for hot keys. Custom aliases and multi-region creates need careful conflict handling, but everything else scales horizontally.

3. Practice Notes

Five-Minute Version

  1. Clarify: short codes, redirect, analytics optional, abuse handling.
  2. State invariant: unique immutable mapping.
  3. Pick code generation: counter with range claim, base62, 7 chars.
  4. Draw two paths: create (rare, strong) and redirect (frequent, cached).
  5. Explain idempotent create and conditional insert.
  6. Explain edge cache, KV cache, store.
  7. Explain safety takedown with pub/sub invalidation.
  8. Close with multi-region (active-active reads, region-tagged code ranges).

One-Minute Closeout

TinyURL is workload-asymmetric: rare creates need correctness, frequent redirects need speed. I would use a range-claim counter for unique base62 codes, cache aggressively because mappings are immutable, separate the click pipeline from the redirect hot path, and use pub/sub invalidation for fast safety takedowns. The main tradeoff is 301 vs 302: I would pick 302 for analytics fidelity and absorb the cost with caching.

Best Interview Phrases

  • "The mapping is immutable; that is why caching is so effective."
  • "I do not coordinate on every create; I pre-claim a range and serve locally."
  • "The redirect path is a single key lookup. Everything else, including analytics, is asynchronous."
  • "Safety takedown is the one case where stale caches matter; pub/sub invalidation gives me a few-second SLA."
  • "Custom aliases and multi-region creates are where the simple design breaks; everything else is easy."
