1. Basic Knowledge And Solution

What This Question Is Testing

Instagram is a media-heavy social feed: photos, videos, stories, reels. It combines the fan-out problem from Twitter with the storage and serving cost of media at scale. The hard part is delivering personalized feeds with media at hundreds of milliseconds while storing exabytes of user content and respecting visibility rules.

The invariant is this:

Every uploaded media is durably stored, viewable by authorized followers/public; the home feed of each follower eventually includes every post from accounts they follow (subject to ranking, visibility, blocks, and policy); media renders at the right resolution for each viewer's device and network; deletes and visibility changes are visible everywhere within seconds.

At ICT5 / L6+ level, the interviewer is testing whether you can design photo/video upload and CDN serving, handle the same hybrid fan-out as Twitter, integrate ranking and stories, and reason about transcoding pipelines and storage tiering.

60-Second Interview Framing

Spoken version:

I would design Instagram as a media-heavy feed system: upload pipeline writes original media to object storage, an async transcoding pipeline generates multiple resolutions, CDN serves to viewers. Feed delivery uses hybrid fan-out (push for normal accounts, pull for celebrities) like Twitter. Stories are time-limited, separate from feed. Ranking, search, and recommendations are side services. The interesting decisions are upload + transcoding, CDN sizing, fan-out, stories, and ranking timeouts.

Requirements To Clarify First

Functional requirements:

  • User uploads photos and videos (single, carousel, reels).
  • User follows others, sees a feed.
  • Stories: 24-hour ephemeral posts.
  • Like, comment, save, share, DM.
  • Search by user, hashtag, location.
  • Explore page with recommendations.
  • Profile page with all posts.
  • Editing tools (filters, crops); applied client-side.

Non-functional requirements:

  • Billions of users; hundreds of millions of daily uploads.
  • Feed read p99 under 500 ms including media URLs.
  • Media URL → first byte under 100 ms via CDN.
  • Storage durability 11 nines.
  • Stories expire automatically.
  • Multi-region.
  • Deletes propagate fast.

Out of scope unless asked:

  • Live video streaming.
  • Shopping checkout.
  • Ad targeting and bidding.

Upload Flow

  1. Client requests /uploads/init with media metadata.
  2. Server returns upload_url and media_id.
  3. Client uploads media in chunks (resumable) directly to a storage gateway or signed URL.
  4. Client calls /uploads/{media_id}/complete.
  5. Server records media as transcoder_status=pending; enqueues transcoder job.
  6. Client calls POST /posts with media_ids and caption to create the post.
  7. Post is created with status=processing until media has at least one usable derivative.
  8. Transcoder generates multiple resolutions (e.g., 240p, 480p, 720p, 1080p for video; thumbnail + multi-res for photo).
  9. On first derivative ready, post transitions to active and visible.
  10. Fan-out to follower inboxes begins (for non-celebrities).

Interview phrase:

Upload and transcode is the bedrock. The post becomes visible the moment a usable derivative exists; subsequent derivatives improve quality for higher bandwidth viewers.

Transcoding Pipeline

For video especially:

  • Split source into segments.
  • Parallel transcode segments to each target resolution.
  • Assemble.
  • Generate adaptive streaming manifest (HLS / DASH).
  • Thumbnails for scrubbing.

Storage:

  • Originals kept for ability to re-transcode (e.g., new format).
  • Derivatives stored separately; can be tier-aged.
  • Reference counts; deletes cascade.

Cost optimization:

  • Spot capacity for transcode workers.
  • Skip rare-resolution derivatives until requested ("just-in-time" derivatives).
  • Tier old originals to cold storage.

Feed Delivery (Same Hybrid Fan-out As Twitter)

Same model: push for normal accounts, pull for celebrities. Inbox stores post_id only; hydration at read time fetches post + media URLs.

Ranking is more central to Instagram than to Twitter; the default home feed is heavily ranked rather than chronological.

  • Ranking service receives candidate post_ids + viewer context.
  • Returns scored ids with strict timeout.
  • Features: viewer-author affinity, post engagement, recency, media type, freshness.

Read path:

  1. Fetch inbox + celebrity author timelines.
  2. Apply visibility filters.
  3. Rank.
  4. Hydrate post metadata + media URLs.
  5. Return.

Media URLs point to CDN; client fetches.

Stories

Stories are not in the main feed. They have their own surface (top of app, horizontal scroll).

  • Per-user story list (recent 24h).
  • Sorted by recency or by viewer's affinity.
  • TTL-driven cleanup: stories older than 24h disappear.
  • Viewer state tracked: who viewed which story (for "seen by" indicator).
  • High view rate: counter sharding for celebrity stories.

Storage:

  • Stories in their own table partitioned by user_id.
  • Background job removes expired stories.
  • Media may be retained longer than 24h if user opts in (Highlights).

CDN And Media Serving

  • Media served via CDN with global edge POPs.
  • URLs signed for short-lived access (especially for private accounts).
  • Adaptive bitrate for video.
  • Cache hit ratio is critical for cost; >90% target.
  • Per-region warm caches for trending content.

Multi-Region Strategy

  • Object storage geo-replicated.
  • Originals in primary region; derivatives replicated globally.
  • Feed inbox caches per region.
  • Per-user home region for writes.
  • CDN edges everywhere.

Failure Modes And Mitigations

Failure mode Mitigation Signal to monitor
Transcode fails on a video. Retry; degrade to lower quality; mark post processing longer. Transcode failure rate.
Hot post overwhelms engagement counter. Sub-counter sharding (same as YouTube likes). Per-counter QPS.
CDN miss on viral content. Pre-warm; multi-tier cache. Origin egress; CDN hit ratio.
Stories expiry job lags. TTL enforced at read time as final gate. Expired-still-visible count.
Upload incomplete (user closes app). Resumable; stale partial uploads garbage-collected. Incomplete-upload age.
Inbox cache loss. Rebuild from author timelines + follow graph. Cache miss rate.
Mod fails to catch policy violation. Async re-classification; user reports. Moderation latency.
Celebrity post fan-out storm. Hybrid threshold; backpressure; pull path. Fan-out backlog.
Storage cost growth. Cold tiering of old originals; reference-counted dedup. Storage growth rate.
Multi-region replication lag. Async; bounded; alert. Replication lag.

Operational Metrics

Correctness:

  • Stale post visibility after delete.
  • Block/visibility violations.
  • Story expired-but-shown count.
  • Transcode failure rate.

Capacity:

  • Upload QPS; transcode queue depth.
  • Feed read p99.
  • CDN hit ratio, origin egress.
  • Engagement counter lag.

Business:

  • DAU; posts per user; engagement rates.
  • Stories DAU.
  • Explore engagement.

Tradeoffs To Call Out

  • Pre-generating all resolutions wastes storage; on-demand is slower first-view.
  • Ranking improves engagement but adds latency; strict timeouts protect feed.
  • Per-user inbox materialization is expensive but speeds read.
  • Stories 24h TTL is product-defining; cleanup must be reliable.
  • Heavy media at scale dominates infrastructure cost; CDN strategy is critical.
  • Privacy controls (close friends, private accounts) require filter at read.

Common Traps And Stronger Moves

Trap Stronger move
Synchronous transcode before post visibility. Post visible at first derivative; higher quality lands later.
Storing media in the post table. Separate media store; post holds refs.
One CDN cache layer. Multi-tier: device → edge POP → regional → origin.
Treating stories like posts. Separate surface; TTL; expiry job + read-time gate.
No ranking timeout. Strict timeout + chronological fallback.
Original media in hot storage forever. Tier to cold storage with time.

Staff-Level Answer Checklist

  • Separate upload, transcode, post metadata, feed, stories surfaces.
  • First-derivative-ready visibility.
  • Hybrid fan-out for feed.
  • Stories with TTL + read-time gate.
  • CDN multi-tier with hit ratio focus.
  • Engagement counters sub-sharded.
  • Search and recommendation as side systems.
  • Multi-region with per-user home + CDN edges.

Interview Phrase

Instagram is Twitter plus a media pipeline. The fan-out is the same hybrid; the wrinkle is the upload-and-transcode flow that delays full quality but allows fast publication, plus stories with 24h TTL. The dominant infrastructure cost is media storage and CDN serving; the dominant correctness concern is feed delivery and visibility filtering.

2. Simulated Interview

Interviewer: Design Instagram.

Candidate: I will design it as a media-heavy social feed: upload pipeline writes media to object storage; transcoder generates multiple resolutions async; post metadata is separate from media; feed delivery uses hybrid fan-out like Twitter; stories are a separate ephemeral surface. The invariant is that uploaded media is durable, posts become visible once at least one derivative is ready, and the feed delivers ranked content from followed accounts within hundreds of milliseconds.

Interviewer: Walk me through upload.

Candidate: Client requests an upload URL and media_id. Uploads chunks directly to storage via signed URLs. When done, calls complete. Server marks media transcoder_status=pending and enqueues a transcoder job. Client posts the post metadata referencing the media_id. The post is created in processing state, becoming active once any derivative is ready — usually within seconds for photos, longer for video. Higher quality derivatives land later but the post is already visible.

Interviewer: Why first-derivative visibility?

Candidate: Two reasons. First, perceived publish latency. Users expect their post to appear immediately, not after a 30-second transcode. Second, the lowest quality is enough to render in feed previews; high-quality versions are only needed when the post is viewed full-screen on capable bandwidth. We optimize for "good enough to show" and improve quality asynchronously.

Interviewer: Feed delivery — same as Twitter?

Candidate: Yes, hybrid fan-out. Normal accounts push post_ids to follower inboxes; celebrities pull at read time. The feed service merges and ranks. Ranking is more aggressive than Twitter — Instagram's default is heavily ranked rather than chronological. Strict timeout on the ranker with chronological fallback.

Interviewer: Stories?

Candidate: Separate surface. Stories table partitioned by user_id, with expires_at = posted_at + 24h. The story bar at the top of the app shows recent stories from followed users, sorted by recency or affinity. A background job removes expired stories, and the read path filters anything past expiry as a final gate. Story media goes through the same transcoder.

Interviewer: A celebrity posts a viral story watched by millions. How does the counter not melt?

Candidate: Same pattern as YouTube likes. View counter for the story is sharded into sub-counters; views hash to a random sub-counter for atomic increment. Display sum aggregates. Per-viewer "seen" state is recorded in a separate per-story-per-viewer table if needed for "seen by" indicators, or aggregated to a count for very popular stories.

Interviewer: Storage cost?

Candidate: Exabytes of media. We tier aggressively. Hot tier (recent posts) on fast object storage. Warm tier (older posts that still get views) in cheaper storage. Cold tier (originals older than X months) in archival storage. Derivatives are smaller and stay in hot tier longer because they're served via CDN. Reference counting handles deletes: when no post references a media_id, it's eligible for cleanup.

Interviewer: CDN strategy?

Candidate: Multi-tier. Device cache (browser/app) → edge POPs → regional caches → origin. Hit ratio target above 90%. URLs are signed for private accounts. Trending content pre-warms edge caches via prediction. Adaptive bitrate for video using HLS/DASH manifests.

Interviewer: Search?

Candidate: Separate search service consuming post events. Inverted index on caption tokens, hashtags, location, username. Sharded by token. Eventually consistent — a post becomes searchable within seconds. Queries fan out to relevant shards, score, return.

Interviewer: Explore page?

Candidate: Offline + online recommendation pipeline. Daily offline job generates candidate set per user from collaborative filtering, embedding similarity, and trending. Online ranker scores candidates at request time with strict timeout. The candidate set may be 1000s; ranker returns top 50.

Interviewer: Deletes?

Candidate: Same as Twitter. Mark post deleted; the feed read path's hydration sees the deletion and skips. Inbox entries remain but display nothing. Media reference count decrements; storage cleanup is async. The deleting user's feed reflects the deletion immediately; followers see it within seconds.

Interviewer: Multi-region?

Candidate: Object storage geo-replicated. Originals in user's home region; derivatives replicated globally for fast CDN warm-up. Post metadata replicated. Per-user home region for writes. CDN serves locally everywhere. Cross-region replication lag is seconds; users may briefly see slightly stale feeds across regions, acceptable.

Interviewer: Summarize.

Candidate: Instagram is Twitter plus a media pipeline. Upload writes original, async transcoder generates derivatives, post becomes visible at first derivative. Hybrid fan-out for feed; stories are a separate ephemeral surface with TTL. CDN multi-tier serves media; storage tiered by age. Engagement counters sub-sharded. Search and Explore are side systems. The dominant cost is media storage and CDN; the dominant correctness concern is feed delivery and visibility.

3. Practice Notes

Five-Minute Version

  1. Clarify: media types, feed, stories, scale.
  2. Invariant: durable media, visible on first derivative, feed delivers.
  3. Upload + async transcode; first-derivative visibility.
  4. Post metadata separate from media.
  5. Hybrid fan-out for feed like Twitter; ranking with timeout.
  6. Stories separate surface with TTL.
  7. CDN multi-tier; storage tiering by age.
  8. Engagement counters sub-sharded; search/explore side.

One-Minute Closeout

Instagram is a media-heavy social feed. Upload writes the original; an async transcoder produces multiple resolutions; the post becomes visible at first derivative. The feed uses hybrid fan-out — push for normal accounts, pull for celebrities. Stories are a separate surface with 24h TTL. CDN multi-tier serves media at edge. Storage tiers age content to cheaper layers. Engagement counters are sub-sharded for hot posts. The dominant cost is media; the dominant correctness concern is feed visibility.
