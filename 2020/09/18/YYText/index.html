Let's start with the architecture diagram from the GitHub repo.

The README's interpretation of TextKit is wrong. Apple's official TextKit interpretation is:

Structurally, YYText and TextKit share the same class boundaries — both are MVC, with the only difference being that YYText lacks NSTextStorage's ability to reuse an attributed string across multiple layouts and containers. NSTextStorage is a subclass of NSAttributedString; storage, in my view, exists to extend the reuse capability of a string so it can be shared across multiple layouts and containers.

Another structural difference: in TextKit, an NSLayoutManager controls multiple NSTextContainers, whereas one YYTextLayout corresponds to exactly one YYTextContainer. My guess is that YYText is iOS-only and there is no scenario where one layout drives multiple containers.

How do you choose between YYLabel and YYTextView? See the diagram below.

If you need a password field, use UITextField. Otherwise, UILabel → YYLabel and UITextView → YYTextView.

YYText and CoreText interaction

The overall flow:

  1. First, configure your NSAttributedString.
  2. Use it to create a CTFramesetter.
  3. Provide a CGPath to the CTFramesetter to generate a CTFrame.
  4. The CTFrame contains multiple CTLines.
  5. Each CTLine contains other CTLines or CTRuns.
  6. Call Core Graphics to render the CTRun.

The CoreText concepts involved are CTFramesetter, CTFrame, CTLine, and CTRun.

We'll start with CTLine to understand the CTRun rendering flow, then go back to how CTLines are laid out.

Laying out attachments

Positioning attachments:

An attachment has no font to drive its position; we must manually provide its ascent, descent, and leading.

For a CTRun containing text, CTRunDraw ignores baselineOffset. That's why YYText does not support baselineOffset — UILabel does, presumably by pulling out glyphs and adding the offset manually when calling setTextPosition.

But for a CTRun containing an attachment, we have full layout control because an attachment is just a rect to place, and a CTRun is already positioned inside a CTLine — we control the rect's position ourselves.

By default in mixed text + image layout, images and text are aligned to the top — that's YYTextAttachmentVerticalAlignTop. To support Center and Bottom, we re-set the baseline so each attachment can have its own vertical attribute. Insets also work.

Below we walk through:

How this API uses a font plus alignment mode to set ascent and descent:

Top case:

Take the font's ascender; the attachment's extra height goes into descent.

If descent would be negative, clamp it; set ascent to the height.

Note: if the attachment is taller than the font's ascent, we use the font's ascent as the image's ascent and stuff the rest into descent. How we fill descent doesn't really matter — it's just a definition, as long as bounds calculation is correct.

If the attachment is shorter than the font's ascent, since descent cannot be negative, we set descent = 0 and ascent = attachment height — equivalent to placing the position at the bottom-left.

If you treat YYTextAttachment as ordinary text, you'll notice it misses baseline handling, but in fact we don't need the baseline concept for attachments.

Why is runPosition.y = _position.y - runPosition.y?

Recap:

_position is the line's baseline position — the bottom-left of the line in UIKit, i.e. the distance from the top of the label.

runPosition is in CoreText coordinates, relative to the line.

So _position.y - runPosition.y is the baseline position (bottom-left) of this run in the label's UIKit coordinates.

With the baseline position known, the run's UIKit rect.origin.y is simply position.y - ascent. When rendering, we draw directly with this UIKit-coordinate rect.

As noted, runPosition is the attachment's origin. So when the height in yy_attachmentStringWithContent is less than the font ascent and we set descent = 0, runPosition shifts up — equivalent to baselineOffset for ordinary fonts. The UIKit-coordinate y for runTypoBounds remains runPosition.y - ascent.

CTRun

After the attachment layout above, it's clear that a CTRun is a range whose characters all share the same attributes. YYText does not wrap CTRun because the interface is simple enough.

Main APIs:

CTRunGetGlyphCount — get the run's glyph count.

CTRunGetAttributes — return the attribute dictionary used to create the glyph run.

CTRunGetStatus — knowing direction and ordering can help string-index analysis; knowing whether positions reference the identity text matrix can avoid expensive comparisons.

CTRunGetGlyphs — get glyphs.

CTRunGetPositions — get every glyph's position.

CTRunGetAdvancesPtr — get every glyph's advance.

CTRunGetTypographicBounds — get this run's ascent, descent, leading, width.

CTRunDraw — draw a run. The docs note this is a convenience method: you could fetch each glyph and render individually. After this operation, the context may be left in an arbitrary state and is not reset.

Note: the CoreText attributes that YYText supports are exactly the attributes CTRunDraw supports. Attributes like baselineOffset that CTRunDraw does not support require fetching each glyph manually and laying it out yourself. CTRunDraw also does not support vertical layout natively — that's why YYText only manually lays out and renders each glyph for vertical layout, and does not support baselineOffset.

CTFrame

A frame contains multiple lines. CTFrame is the text-framing result of CTFramesetter.

You can draw a CTFrame directly via the Core Graphics interface. CTFrame contains an array of CTLines; you can get glyph info per line and render each line individually.

YYText asynchronous rendering

YYText's other celebrated feature is async. Good write-up on Jianshu. TLDR:

There's a function YYAsyncLayerGetDisplayQueue that decides which queue to async-submit render work to. Internally it maintains an array of queues sized to the CPU core count; each queue is serial. If YYKit's QoS file is included, QoS can be enabled in Release builds.

YYTextSentinel *sentinel is a YYLabel property used as a counter. Every update calls setNeedsDisplay or otherwise cancels existing display tasks and bumps this sentinel. During async work, before significant steps, the value is compared against the copy taken at the start of the task — equal means no new task or cancellation, unequal means abandon this work. Significant steps include: creating CGContext, closing CGContext, setting layer.contents, getting a UIImage from a context.

YYTextAsyncLayerDisplayTask has three blocks:

willDisplay — cleanup; check and diff attachments.

Display — perform the main text drawing.

  1. Check cancelled.
  2. Check whether layout needs update — rebuild layout; should shrink layout.
  3. Adjust the rendered image's position based on YYTextVerticalAlignment.
  4. Draw in context with the size & position.

didDisplay — update YYLabel properties.

  1. If shrunk, assign shrinkLayout.
  2. If this render task was cancelled, clear all attachments.
  3. Add a fade-in / fade-out CATransition animation.

In practice, "async" doesn't help much: YYText itself can't handle very heavy tasks (limited by CoreText), and for light tasks the difference between async and sync is small. Also remember to disable the fade animation, otherwise flicker may occur.
