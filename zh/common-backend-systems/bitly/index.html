Bitly — URL Shortener (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Easy · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建一个 Bitly 风格的短链接服务。用户粘贴一个长 URL，获得一个短码，访问该短码后 302 重定向到原始链接。创建者可以查看基础点击统计。假设每月新建 1 亿条链接，读写比为 100:1。

1. 需求（约 5 分钟）

功能性需求——"用户应能够..." - 将长 URL 缩短为短 URL，可选自定义别名。\n- 访问短 URL 并跳转至原始链接。\n- 查看其名下链接的点击数及近期来源信息。

我会明确舍弃链接过期和完整的分析看板功能，专注于重定向路径。

非功能性需求——"系统应..." - 重定向 p99 延迟 < 100 ms；每一毫秒的增加都能被用户感知。\n- 即使写入服务发生故障，重定向也应保持可用——链接一旦存在，必须可访问。\n- 短码到 URL 的映射必须持久化；一旦丢失，用户侧无法恢复。\n- 短码需具备足够的唯一性和不可猜测性，使随机扫描在经济上不可行。

容量估算（仅因其决定分片策略）： - 每月 1 亿条链接 × 60 个月 = 约 60 亿条链接。每行约 200 字节，元数据总量约 1.2 TB——适合分片 MySQL 存储。\n- 100:1 读写比 → 平均约 4k 次重定向/秒，峰值约 40k 次/秒。\n- 7 位 base62 = 62⁷ ≈ 3.5 万亿个码——空间充裕。\n- 点击事件主导写入量：若每次重定向都触发分析，峰值约 40 万次/秒写入。应将其排除在 MySQL 之外，由 Kafka + 聚合计算来承载该路径。

2. 核心实体（约 2 分钟）

  • User ——拥有链接，负责写操作的认证。
  • Link — short_code, long_url, owner_id, created_at, custom?.
  • ClickEvent — short_code, ts, ip, ua, referrer。高写入量，存储在主库之外。

3. API（约 5 分钟）

POST /v1/links              { long_url, custom_alias? }  → { short_url }
GET  /{short_code}          → 302 Location: long_url
GET  /v1/links/{short_code}/stats → { count_24h, count_total, recent[] }

通过 Bearer token 进行认证； owner_id 从 token 中派生，不来自请求体。重定向端点是唯一无需认证的路径。集合路由使用复数形式的 links 资源路由。

POST /v1/links 接受一个 Idempotency-Key 以便超时后重试时返回相同的 {short_url} 而不会消耗第二个码。统计分页使用类似 {short_code, bucket_start, last_event_id}.

5. 高层设计（约 10–15 分钟）

我会按顺序画出以下几个模块：

  1. Client → API Gateway ——TLS 终止、认证、写操作的按 IP 限流。
  2. Write service ——校验 URL，生成短码，将 (short_code, long_url, owner_id) 写入元数据存储。返回短 URL。
  3. Read service（重定向路径） — GET /{code} → Redis 查询 → 未命中时穿透至元数据存储 → 写缓存 → 返回 302。在响应写出之前（或同时）向 Kafka 发送点击事件。
  4. Metadata store ——以 short_code 为键的分片 MySQL（对短码做一致性哈希）。单行查询，无需 join。
  5. 缓存 ——Redis 集群，写透模式，LRU 淘汰。热门短码常驻内存。
  6. Click pipeline ——Kafka → 流处理器（Flink）→ 汇总写入 ClickHouse（按 short_code 为键，按分钟/小时/天分桶聚合）。
  7. Stats service ——读取 ClickHouse 聚合数据，服务于 /stats 端点。

我会带面试官完整走一遍一次写入和一次读取的数据路径，使数据流路径具体可见。

6. 深度探讨（约 10 分钟）

短码生成。 三种方案：\n- 随机 base62 + 插入时碰撞检测。简单，但每次写入都需检查是否存在。\n- Hash(long_url) 截断。确定性去重，但仍需碰撞探测。\n- Counter + base62，按分片划分 ID 范围。 每个写入方从协调器拉取 1 万个 ID 的块并转换为 base62。无碰撞，写路径无需额外读取。我会默认采用此方案，并接受短码会泄露大致创建顺序的权衡。

热门重定向路径。 目标是对热门短码永不触达 MySQL。Redis 是第一道防线；边缘层的小型 Bloom 过滤器可短路已知不存在的码。对于当下正在病毒式传播的短码，我会让 CDN 缓存 302 响应 30–60 秒——稍旧的重定向可以接受，目标 URL 本身过期才是最坏情况，但我会明确记录该权衡。

Metadata DB links.short_code 行内容：long_url, owner_id, created_at, flags ~200-500 B ， hash(short_code)

Rollups (short_code, minute) count, referrer sketch ~80 B 2 years hash(short_code)

元数据容量并不夸张：60 亿条链接 × 500 B × 3 副本 ≈ 9 TB，仍属常规分片 MySQL 的处理范围。所有者索引为 60 亿条 × 12 B × 3 副本 ≈ 216 GB（不含页开销），足够小，可与元数据共存或存放于独立索引表。1B 条最热链接的重定向缓存按约 500 B 计算约为 500 GB，加上 Redis 开销后约为 1–2 TB（含副本）。点击分析是更大的流量来源：峰值 40k/秒 × 150 B ≈ 6 MB/秒，Kafka 以 7 天 × 3 副本保留，未压缩约 11 TB。聚合后的 (short_code, minute) 时间桶体积小得多；成本调节点在于保留多少 referrer/user-agent 详情。

双索引模式简单但重要：Link 记录以 short_code 为键用于重定向，而所有者仪表盘则需要 owner_id -> short_codes。我会将重定向行作为数据来源，并在创建时以事务方式维护所有者索引，因为查询用户名下的链接列表不应给重定向路径增加任何 join 操作。

阶段 模式 数据源 用户影响

面试核心要点： 读路径承担两项不同保障级别的工作：重定向必须快速且持久，而分析可以有延迟，并可从点击日志中重建。

不将每次点击写入 MySQL 的点击分析方案。 每次重定向都向 Kafka 发送一个事件，并递增 Redis 中的计数器用于实时展示。Flink 以 1 分钟滚动窗口聚合写入 ClickHouse。 /stats 端点仅读取 ClickHouse，MySQL 永远不会看到点击流量。

为什么选择 MySQL + Redis。 元数据的访问模式是主键查询加偶发写入；分片 MySQL 提供持久化行存储、别名的唯一约束以及简单的备份机制。DynamoDB/Cassandra 也可行，但除非面试官将问题推向全球多活写入，否则我们不需要宽行或多区域 active-active。Redis 优于进程内缓存，原因是重定向流量分布在大量无状态读服务节点上；CDN 缓存是针对超热短码的边缘层额外优化，而非数据来源。

自定义别名竞争。 两个用户同时争抢 bit.ly/launch：使用 SETNX 对于 alias:launch 并设置 5 秒 TTL，在 MySQL 插入前先行预留。先到者胜；后到者收到干净的 409 响应。

滥用与安全。 对写操作按 owner_id 实施令牌桶限流（Redis）。插入后，异步将 URL 推送至 Google Safe Browsing 检测；若被标记，则软删除并在重定向前展示风险提示页面，而非直接返回 302。

如果有更多时间，我会延伸讨论的内容。 重定向路径的多区域 active-active、按链接 TTL 过期、地理感知分析以及链接预览。我会提及这些，让面试官知道我已意识到它们的存在。

面试核心要点： 重定向本质上是一次微小的主键读操作，因此要将该路径与其他一切隔离保护。链接元数据持久且访问模式简单；分析是高吞吐量且容许丢失的，适合通过流处理独立消化并进行对账。
