1. 基础知识与解题思路

本题考察点

Dynamo（2007 年论文，而非现代 DynamoDB）是最具代表性的最终一致性分布式 KV 存储。它证明了"始终可写"系统的可行性，该系统基于一致性哈希、宽松 quorum、向量时钟、Hinted Handoff、基于 Merkle 树的反熵以及 Gossip 协议构建。现代系统（Cassandra、Riak、Voldemort、现代 DynamoDB）均继承了其中的思想。

其不变量如下：

每次写入至少在 W 个副本上成功；每次读取至少检查 R 个副本；并发写入产生兄弟版本并以确定性方式进行对账；系统在发生分区和节点故障时仍保持可写，以最终一致性作为权衡。

在 ICT5 / L6+ 级别，面试官考察的是你能否阐述一致性哈希、可调节的 (N, R, W) quorum、用于因果追踪的向量时钟、Hinted Handoff 与 Merkle 树反熵，以及"始终可写"的架构哲学。

60 秒面试框架

口述版本：

我会将 Dynamo 描述为一个无主节点的 KV 存储，基于一致性哈希进行分区和复制。写入操作发送至 N 个偏好列表副本中的 W 个；读取从 R 个副本汇聚结果。(N, R, W) 可按请求调节。冲突会产生向量时钟兄弟版本，由客户端进行合并。Hinted Handoff 处理短暂故障；Merkle 树反熵修复长期分歧。其架构选择是可用性优先于严格一致性——始终可写。

论文设定的需求

功能性需求：

  • 通过 key 提供简单的 Put / Get API。
  • 在故障和分区期间保持高可用的写入能力。
  • 多数据中心。
  • 可配置的持久性与延迟权衡。
  • 在 Amazon 用于购物车、会话、用户偏好等场景。

非功能性需求：

  • 99.9 百分位延迟在数百毫秒以内。
  • 高可用性——即使部分副本不可达，写入也必须成功。
  • 最终一致性（通常在数秒内）。
  • 在读取时进行冲突解决。

原始论文范围之外：

  • 强一致性。
  • 多键事务。
  • 范围查询。
  • 二级索引。

规模假设

领域 假设
集群规模 每个集群数十至数百个节点。
复制因子 N 通常为 3。
Quorum W, R 可按请求调节；常见配置：W=R=2。
Latency（延迟） p99 在数十毫秒级别。
吞吐量 每秒数千次操作。

核心概念

一致性哈希

节点和键分布在一个环（哈希空间）上。

  • 每个节点放置在一个或多个位置（虚拟节点 / vnodes）以实现负载均衡。
  • 某个 key 的协调节点是环上从该 key 哈希值顺时针方向的第一个节点。
  • 副本是环上协调节点之后的 N-1 个节点。

添加/移除节点只影响相邻范围。

为什么使用 vnodes：\n- 每个节点的负载更均匀。\n- 异构节点可分配更多 vnodes。\n- 引导新节点时可并行处理。

偏好列表

对于每个 key，偏好列表是应持有副本的 N+ 个节点的有序列表。前 N 个可达节点构成实际 quorum。

可调节的 (N, R, W)

  • N：副本总数。
  • W：写入必须确认的最少副本数。
  • R：读取必须响应的最少副本数。

若 R + W > N：任意读写之间至少有一个副本重叠——强一致性（受故障影响）。

若 R + W ≤ N：更高可用性，较弱的一致性保证。

常见调节：\n- N=3, W=2, R=2：正常情况下可读取自己的写入。\n- N=3, W=3, R=1：读取时强一致性；写入较慢。\n- N=3, W=1, R=1：最大可用性；一致性极弱。

数据模型

存储对象

StoredObject(
  key,
  value,
  version_vector,           // vector clock of writes
  context                   // opaque token passed back on next write
)

向量时钟

由 (node_id, counter) 对组成的列表。节点 N 的每次写入会递增 N 的计数器。

比较规则：\n- 若 vc_A 的所有计数器均 ≤ vc_B，则 vc_A < vc_B（因果先序）。\n- 若两者均不满足 ≤ 关系，则它们是并发的（兄弟版本）。

示例：\n- A 的写入 1：vc = [(A, 1)]\n- A 的写入 2（在读取写入 1 之后）：vc = [(A, 2)]\n- B 的写入 3（在读取写入 1 之后）：vc = [(A, 1), (B, 1)]\n- 写入 2 和写入 3 是并发的。

写入路径

  1. 客户端向协调节点发送 PUT(key, value, context)。
  2. 协调节点查找偏好列表。
  3. 更新向量时钟：取输入 context 的 vc，递增协调节点的计数器。
  4. 向偏好列表中的 N 个节点发送 Put 请求。
  5. 等待 W 个确认。
  6. 返回成功。

若某副本节点宕机：\n- 将数据发送至环上的下一个节点作为临时 hint。\n- Hint 中包含原始目标节点信息。\n- 原始节点恢复后，将 hint 重放至原始节点。

读取路径

  1. 客户端向协调节点发送 GET(key)。
  2. 协调节点查询偏好列表中的 N 个节点。
  3. 等待 R 个响应。
  4. 若响应的向量时钟存在分歧，返回所有兄弟版本及合并后的 context。
  5. 客户端（或协调节点侧的解析器）进行合并。
  6. 可选的读取修复：将合并后的值写回。

兄弟版本合并策略：\n- 购物车：对每个商品取最大数量并取并集。\n- 会话：取最近的字段（按字段 LWW）。\n- 通用：由应用层定义。

Hinted Handoff

当写入目标节点不可达时：

  1. 协调节点选择环上下一个可达节点作为 hint 持有者。
  2. Hint 中包含原始目标节点作为元数据。
  3. Hint 持有者以"此数据属于 X"的标签存储数据。
  4. 定期尝试将 hint 重放至原始节点。
  5. 重放成功后删除 hint。

若 hint 持有者在重放前宕机：数据丢失（除非 N>3 且其他副本持有该数据）。对于高持久性要求，应使用更大的 N 值。

反熵：Merkle 树

用于处理长期存在的分歧（hint 持有者数据丢失，分区持续数小时）：

  1. 每个节点为其分区的 key 范围维护一棵 Merkle 树。
  2. 叶节点是 (key, value) 元组的哈希值。
  3. 内部节点是子节点的哈希值。
  4. 副本节点定期交换树摘要。
  5. 树差分：沿树向下比较哈希值；不匹配处即为分歧的 key。
  6. 对每个分歧的 key 进行交换和对账。

代价：树的构建开销较大；通常定期增量执行。

成员管理：Gossip

成员变更（节点添加、移除、故障）通过 Gossip 协议传播：

  1. 每个节点定期随机选择一个对等节点。
  2. 交换成员状态。
  3. 协调差异。
  4. 成员信息最终收敛。

故障检测：gossip 心跳；某节点在 T 秒内未被观测到则被怀疑宕机，标记为不可达。

Sloppy Quorum

严格 quorum（偏好列表中 N 选 W）在部分偏好列表成员宕机时可能阻塞写入。Sloppy quorum 允许下 N 个可达节点接受写入。

  • 在故障期间保留写入可用性。
  • "真实"副本稍后通过 hinted handoff 追赶。

多数据中心

该论文涵盖跨数据中心的复制：

  • 复制因子包含不同数据中心的节点。
  • 写入通常等待本地数据中心 quorum；向其他数据中心的复制是异步的。
  • 通过第二区域副本实现灾难恢复。

关注点 策略

移除节点 排空至邻节点。

冲突 将 sibling 呈现给客户端；由应用层合并。

运维指标

正确性：

  • Sibling 速率。
  • Hinted handoff 积压量。
  • 反熵收敛时间。
  • 丢失写入数量。

容量：

  • 每节点 QPS。
  • 复制延迟。
  • 环重平衡时间。

需要指出的权衡

  • 可用性优先于一致性：写入始终成功；读取可能出现分叉。
  • Vector clock 表达力强，但若不裁剪会持续增长。
  • Sloppy quorum 维持可用性，但允许临时分叉。
  • 应用层合并灵活但易出错。
  • Hinted handoff 以简单性换取持久性方面的隐患。
  • 反熵开销较大，但最终能捕获所有分叉。

常见陷阱与更优做法

陷阱 更优做法
始终使用最后写入者获胜。 使用 vector clock sibling；让应用层合并。
严格 quorum。 Sloppy quorum + hinted handoff。
等待所有副本。 N 中取 W 的 quorum；可调。
静态节点放置。 Vnode；引导并行化。
忽略分叉。 通过 Merkle 树进行反熵。
中心化协调者。 任意节点均可作为协调节点。

面试用语

Dynamo 是典型的无主节点最终一致性 KV 存储。一致性哈希负责分区与复制；(N, R, W) 调节持久性与延迟的权衡；vector clock 追踪因果关系，读取时通过 sibling 解决冲突；hinted handoff 和 Merkle 树反熵在故障后协调；gossip 传播成员信息。架构选择是可用性优先于严格一致性——始终可写。

2. 模拟面试

面试官： 带我了解一下 Dynamo 的设计。

候选人： 无主节点 KV 存储，基于带 vnode 的一致性哈希环。每个 key 的偏好列表是顺时针方向的 N 个后继节点。写入发送至任意协调者，复制到 N 中的 W 个副本；读取收集 R 个响应。冲突产生 vector clock sibling，在读取时由客户端协调。Hinted handoff 处理短暂故障；Merkle 树反熵处理更长时间的分叉。Gossip 传播成员信息。不变量是始终可写、最终一致。

面试官： 带我了解一次 PUT 操作。

候选人： 客户端发送 PUT(key, value, context)，其中 context 是上次读取的 vector clock。协调者（任意节点）对 key 进行哈希，找到偏好列表，在 vector clock 中递增自身计数器，并将写入转发给 N 个副本。等待 W 个 ack。返回成功。若某些副本不可达，则发送给下一批可达节点（sloppy quorum），并附带注明原始目标的 hint。原始目标在恢复后收到 hint 重放。

面试官： 那 GET 操作呢？

候选人： 协调者查询 N 个副本；等待 R 个响应。若所有响应的 vector clock 具有因果可比性（某一个占主导），则返回胜者。若它们是并发的（sibling），则将所有响应连同合并的 context 一并返回。客户端按应用语义合并——对于购物车，取并集并对每个商品取最大数量。

面试官： Vector clocks。

候选人： 由 (node_id, counter) 对组成的列表。节点 N 的每次写入递增 N 的计数器。比较方式：若 A 中所有计数器 ≤ B 中所有计数器，则 A 因果先于 B；若互不占主导，则为并发。Sibling 来源于并发写入。客户端在下次写入时将 context（vc）传回，使系统知晓上次读取的版本。

面试官： Vector clock 会随时间增长。

候选人： 是的。通过限制大小并丢弃最旧的条目来裁剪——但需注意，丢弃会损失因果信息。替代方案：按时间戳裁剪超过 N 个条目的旧条目。原始论文指出这是一个真实存在的问题。CRDT 为支持合并的类型提供了另一种替代方案。

面试官： Hinted handoff。

候选人： 当副本宕机时，协调者将写入作为临时 hint 发送给环上下一个可达节点。Hint 标记了原始目标。Hint 持有者定期尝试向原始目标重放。成功后删除 hint。若 hint 持有者在重放前宕机，hint 将丢失——持久性取决于是否有其他副本。

面试官： 通过 Merkle 树进行反熵。

候选人： 每个节点为每个 key 范围分区维护一棵 Merkle 树。叶节点对 (key, value) 进行哈希；内部节点对子节点进行哈希。副本交换树摘要；沿树向下遍历，哈希不匹配的部分定位到分叉的 key。交换不同的 key，完成协调。用于处理 hinted handoff 未能捕获的长期分叉。

面试官： Sloppy quorum 与严格 quorum。

候选人： 严格：写入必须发送到恰好 N 个偏好列表成员；若某些不可达，则少于 W 个 ack，写入失败。Sloppy：当偏好列表成员不可达时，用下一批可达节点替代。写入成功，但可能在临时持有者上。权衡：以写入可用性换取临时分叉（反熵最终解决）。

面试官： (N, R, W) 权衡分析。

候选人： N=3 是典型值。W=2，R=2 使 R+W>N——至少有一个重叠 → 无故障时强一致。W=1：写入快速，非常最终一致。W=3：写入慢但持久。R=1，W=N=3：读取快速的强一致性。按工作负载调优。

面试官： 多数据中心？

候选人： 偏好列表可以跨数据中心。常见模式：写入等待本地数据中心 quorum（快速），异步复制到其他数据中心（最终）。灾难恢复切换到另一个数据中心。某些严格一致性配置等待跨数据中心 quorum，但需付出延迟代价。

面试官： 添加节点。

候选人： 预分配的 vnode 位置。新节点在这些位置加入环。引导：从邻节点流式传输 key 范围。通过 vnode 实现并行化——多个邻节点同时流式传输。成员变更通过 gossip 传播。限制引导速率以避免饱和。

面试官： Gossip。

候选人： 每个节点定期随机选择一个对等节点，交换成员状态，进行协调。最终一致的成员信息。通过心跳进行故障检测；某节点在 T 秒内未被观测到则被怀疑宕机。被怀疑的节点可以恢复；系统倾向于继续运行而非宣告死亡。

面试官： 总结。

候选人： 一致性哈希负责分区与复制；(N, R, W) 按请求调节持久性和延迟；vector clock 使因果关系显式化，冲突以 sibling 形式呈现供应用层合并；hinted handoff 和 Merkle 树反熵协调分叉；gossip 传播成员信息。Sloppy quorum 在故障时维持写入可用性。整个设计倾向于可用性优先于严格一致性——这是针对购物车等用例的显式选择。

3. 练习备注

五分钟版本

  1. 始终可写；最终一致。
  2. 使用 vnode 的一致性哈希用于分区 + 复制。
  3. (N, R, W) 可调 quorum。
  4. Vector clock + sibling 解决方案。
  5. Sloppy quorum + hinted handoff 保障可用性。
  6. Merkle 树反熵用于分叉修复。
  7. Gossip 用于成员管理。
  8. 跨数据中心异步的多数据中心复制。

一分钟结尾陈述

Dynamo 是典型的最终一致无主节点 KV 存储。带 vnode 的一致性哈希负责分区与复制。(N, R, W) 按请求调节持久性和延迟。Vector clock 将并发写入以 sibling 形式呈现供应用层合并。Hinted handoff 处理短暂故障；Merkle 树反熵处理更长时间的分叉。Sloppy quorum 在部分故障时保持写入可用性。架构选择是可用性优先于严格一致性。
