Food Review App — Restaurant Reviews (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Medium · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建一个美食点评应用——Yelp的一个聚焦子集。用户撰写菜品（而非仅餐厅）的评论，浏览附近评分最高的菜品，并关注其他评论者。1000万用户，500万家餐厅，1亿道菜品，每月约100万条新评论。

1. 需求（约 5 分钟）

功能需求 - 查找附近的菜品，按评分排序。\n- 查看和撰写特定菜品的评论（评分+文字+照片）。\n- 关注其他评论者；在动态中查看他们的最新评论。

我会将预订、拼桌建议以及超出简单评分范围的推荐功能放到后续再考虑。

非功能需求 - 搜索结果p99 < 300毫秒。\n- 聚合评分最终一致性（秒级）。\n- 规模化照片存储。\n- 评论欺诈过滤——超出设计范围但会提及。

容量估算 - 1亿道菜品；每月100万条评论 = 平均约12条/秒；晚餐时段出现峰值。- 照片：70%的评论包含1张以上照片；每月约100万张照片，每张500 KB = 每月500 GB。- 主要成本是媒体，而非评论文本：评论行约1 KB × 1200万/年 × RF=3，每年仅约36 GB；照片每月500 GB原始 × 3种派生尺寸 × RF=2，在CDN缓存之前约为每月3 TB。

2. 核心实体（约 2 分钟）

  • User — user_id、profile、follows。
  • 餐厅 — restaurant_id、name、geo、hours。
  • Dish — dish_id, restaurant_id, name, price?, avg_rating, review_count.
  • 评价 — review_id, dish_id, user_id, rating, text, photos[], created_at.
  • 图片 — photo_id, url, dish_id, uploader.

3. API（约 5 分钟）

GET   /v1/dishes/search?q=&lat=&lng=&min_rating=4              → Dish[]
GET   /v1/dishes/{id}                                          → Dish + top_reviews
GET   /v1/dishes/{id}/reviews?cursor=...
POST  /v1/dishes/{id}/reviews             { rating, text, photo_ids[] }   → { review_id }
POST  /v1/photos                          (multipart upload)              → { photo_id }
GET   /v1/feed                                                            → Review[] from followees
POST  /v1/follows                         { user_id }

从令牌中获取 User。

游标是不透明的，由服务器签发。评论游标编码 {dish_id, last_created_at, last_review_id}；feed 游标编码 {user_id, last_score, last_review_id} 以确保插入操作不会导致分页偏移。

5. 高层设计（约 10–15 分钟）

  1. 目录服务。 餐厅和菜品。餐厅数据来自公开数据集/合作伙伴数据源；菜品主要由用户贡献，并配有合并工具。按 restaurant_id.
  2. 分片。 搜索索引。
  3. 基于 Elasticsearch 的菜品检索——对菜品名称、餐厅名称、标签进行全文搜索；对父餐厅位置进行地理距离过滤；按贝叶斯评分 + 距离 + 时效性排序。 评论服务。 (dish_id, user_id) 以 review.created 为键写入评论（每位用户对每道菜品最多只有一条评论）。发布
  4. 事件。 聚合管道。 (sum_rating, review_count) 流处理器维护每道菜品的 avg_rating ；将
  5. 更新回目录和搜索索引。 照片管道。
  6. 上传至对象存储；resize worker 生成3种尺寸；通过 CDN 前置提供服务。 Feed 服务。
  7. 基于关注关系的 feed。采用类似 FB News Feed 的混合推/拉模式，但规模更小：关注时推送 → 被关注者的评论出现在关注者的 feed 列表中。大多数用户关注人数 < 100；按需读取方案可行。 通知。

"你关注的评论者发布了一条新评论"——不在本次设计范围内。

端到端叙述：Alice 在 Mission St 附近搜索"ramen" → 搜索服务向 ES 发起全文 + 地理距离查询 → 返回按贝叶斯调整评分、距离、时效性排序的前20道菜品。Alice 就餐后为 Mensho 的味噌拉面写了一条5星评论并附上照片 → 照片上传返回 photo_id → review POST 写入数据行 → 聚合器将 Mensho 味噌拉面的 avg_rating 从4.6（200条评论）更新为4.61（201条）。搜索索引在数秒内完成更新。

Key / 结构 review:{dish_id}:{review_id} Review DB (dish_id,user_id) 行，加上唯一 索引 不含照片约1 KB dish_id
评分聚合 dish_rating:{dish_id} -> {sum,count,prior_version} ~32 B ， dish_id

容量估算：评论数据库增长缓慢，约1200万条评论/年 × 1 KB × RF=3 = 约36 GB/年。若设置上限，feed 索引也很小：1000万用户 × 200个 id × 64 B 逻辑存储约128 GB（复制之前），且大多数用户不活跃，因此只需将活跃 feed 保留为热数据。照片的存储量高出数个数量级，因此对象存储分层和 CDN 命中率比 SQL 分片更为重要。

6. 深度探讨（约 10 分钟）

菜品唯一性。 菜品由用户在餐厅内贡献。两位评论者可能分别添加"Spicy tuna roll"和"Spicy Tuna Roll"——产生重复菜品条目。防御措施：- 创建时进行模糊匹配；推荐已有菜品。- 定期使用规范化名称 + restaurant_id 进行去重任务。- 对重复菜品进行最终一致性合并（保留评论指针）。

贝叶斯调整评分。 一道只有1条评论的5星菜品不应排在拥有200条评论的4.8星菜品之前。应用贝叶斯先验： adjusted = (C × m + sum) / (C + n) 其中 m 为全局均值（约4.0）， C 为先验权重（约10）。由聚合器在更新时重新计算。

照片。 通过签名 URL 直接上传至 S3。异步 worker 生成 200²/600²/1200² 缩略图。使用永久缓存 URL 通过 CDN 提供服务（URL 基于内容哈希，更改文件即更改 URL）。

关注 Feed。 每位用户的 feed 列表是一个小型 Redis 有序集合（上限200条）。发生 review.created事件时，将 review_id 推送到关注者的列表中（跳过粉丝数 > 10k 的头部评论者的推送；对这些人采用按需读取）。Feed 读取合并被关注者的评论并组装完整评论载荷。

读取路径追踪： GET /feed 读取 feed:{user_id} 与用户的头部关注列表并行执行；从 reviewer_recent:{reviewer_id}拉取近期头部评论者的评论；按 review_id去重；批量补全评论行和菜品行；然后返回第一页。 feed:{user_id} 的缓存未命中时，退化为从每个被关注者拉取最近约20条评论，时间预算上限为250 ms。

面试要点："Feed 缓存是评论 ID 的索引，而非评论内容本身；这使扇出成本低廉，同时允许我用最新的审核状态和评分数据重新补全。"

评论写入路径：

POST review -> validate one review per dish/user
            -> write review:{dish_id}:{review_id}
            -> publish review.created
            |-- aggregate dish_rating:{dish_id}
            |-- update dish search document
            '-- fan out review_id to feed:{follower_id}, except celebrity reviewers

这是双视图模式： review:{dish_id}:{review_id} 是内容， feed:{user_id} 是以读者为键的索引， reviewer_recent:{user_id} 是以作者为键的索引，用于拉取回退。

搜索排序。 输入因素：文本相关性（BM25）、距离、贝叶斯评分、评论数量、评论者权威度（评论者的粉丝数）、时效性。线性组合，离线调参。

为何选择 Elasticsearch 而非纯 PostGIS：PostGIS 非常适合 nearby restaurants，但此工作负载在一次查询中需要文本相关性、错误容忍、地理过滤和排序。纯 SQL/PostGIS 方案在 v1 附近浏览时可行；一旦菜品名称搜索和排序成为一等功能，ES/OpenSearch 是更好的读取索引。

搜索路径追踪：client -> 搜索服务 -> 对 {dish_name, restaurant_name, geo_cell, adjusted_rating} 的 ES 查询 -> 前50条 dish_id记录 -> 并行批量获取目录缓存和评分聚合 -> 裁剪至20条 -> 游标包含 {query_hash, last_score, last_dish_id}.

冷搜索降级：若 ES 集群出现故障，使用 Redis 中的 top_dishes:{geo_cell}:{category} ，然后从目录补全菜品页面。这会损失关键词精度，但能在约500 ms 内保持应用可用于附近浏览。

欺诈过滤。 启发式方法：IP 与餐厅所有者重叠、账户年龄、评论发布速度、内容与已知模板的相似度。标记供人工审核；不自动删除。

分页。 评论列表和 feed 采用游标分页（服务器生成的不透明游标），防止插入/删除导致的分页漂移。

缓存。 热门菜品的查询缓存60秒。热门餐厅预热。

优雅降级。

情形 模式 数据源 延迟预算
热门餐厅 提供缓存的菜品卡片和过期评分 目录缓存 + 评分聚合 Redis 有序集合
搜索索引延迟 点击时显示过期排名并展示最新菜品页面 ES + 目录数据库 <300 ms
Feed 缓存耗尽 拉取近期被关注者的评论并异步重建 评论数据库 + 关注关系图 <800 ms
照片变体缺失 提供原始图片或低分辨率变体 对象存储/CDN 首次命中 <1 秒
聚合器延迟 显示最后一次聚合，附"数量可能存在延迟"语义 评分聚合 数秒的漂移

Feed 降级状态：

阶段 模式 数据源 延迟预算
热 feed 推送索引 feed:{user_id} + 评论补全 <200 ms
头部账号混合 混合拉取 feed:{user_id} + reviewer_recent:{id} <300 ms
Feed 耗尽 冷拉取 关注关系图 + 每被关注者近期评论 <800 ms
不活跃用户 惰性重建 评论数据库 + 关注关系图异步任务 首页数据量少，后台重建

面试要点："这个产品看起来是社交应用，但核心边界仍然是内容与索引之分：评论是持久内容；feed 和搜索是可丢弃的视图。"

我会推迟的内容。 预订集成、社交关系图导入、基于 ML 的"你可能喜欢"推荐、多语言评论与翻译、商家认证。
