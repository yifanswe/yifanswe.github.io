Rate Limiter (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Medium · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建一个分布式限流器，保护 API 免受滥用。每个入站请求首先经过限流器；限流器要么放行，要么以 429 拒绝。规则形如"每用户每分钟 100 次请求"、"每 API key 每秒 1000 次请求"、"每 IP 每小时 10 次登录"。预算为个位数毫秒级开销。整个集群 1M QPS，数百个限流器实例。

1. 需求（约 5 分钟）

功能需求 - 根据一条或多条规则，对每个请求作出允许/拒绝决策。 - 支持常见策略：GCRA（通用信元速率算法）/ 令牌桶 / 滑动窗口计数器，按 (user, IP, key, route) 作为键。 - 在响应头中暴露剩余配额、重置时间和桶范围，使客户端能够智能退避。 - 支持热重载规则，无需重新部署网关。

我会暂缓实现策略配置 UI 以及 OAuth 配额/计费集成计量功能。

非功能需求 - 延迟开销 p99 < 5 毫秒。 - 高可用——计数器存储故障不得导致被保护服务宕机。故障模式按规则粒度处理。 - 合理准确；对于软规则，少量超额允许是可接受的，但对于认证/滥用场景则不行。 - 扩展至整个集群 ≥ 1M 次决策/秒。

容量估算 - 1 亿个唯一桶（身份 × 规则 × 窗口）。 - 每个桶的状态：约 50 字节（计数器 + 时间戳）。 - 原始计数器状态约 5 GB，但 Redis 对象开销和副本使实际热数据占用接近 30–60 GB。主要成本不是磁盘，而是内存加上同步路径上的一次网络往返。

生产标定：真正的限流器是一个分层体系，而不是一个旋钮。

层级 问题 窗口 示例
全局卸载器 集群是否过载？ 秒 繁忙时以 503 丢弃非关键请求
按路由 该端点是否过热？ 秒–分钟 10k QPS 时 /search
按租户/身份 某个客户是否存在噪声请求？ 分钟–小时 每用户 100 req/min
配额 他们是否超出了套餐配额？ 小时–天 Pro 套餐每天 1M API 调用
滥用 他们是否在探测系统？ 滚动窗口 10 分钟内 10K 次无效请求 → 封禁

每一层需要不同的状态（用于流量卸载的内存本地状态、用于按租户限流的 Redis 共享状态、用于配额管理的长期存储，以及用于滥用检测的 ML 流水线）。将这些视为单一组件是面试中最常见的失误。

2. 核心实体（约 2 分钟）

  • Rule — name，匹配器（对身份属性的谓词）、算法、限制值、窗口、层级。由限流服务持有。
  • Identity ——描述一个请求的一组扁平维度： user_id, ip, api_key, route, method，以及规则可能匹配的任何其他字段。调用方将完整的属性包传递过来；规则从中选择自己关注的字段进行键控。
  • Bucket ——一个 (rule, identity_subset) 将某个键映射到一个计数器的键值对。对外通过以下方式暴露给客户端： X-RateLimit-Bucket.
  • 决策 — allowed，按规则 remaining / reset_at / scope，可选 retry_after.

3. API（约 5 分钟）

// Internal RPC the API gateway calls per request.
// The caller describes the request; the limiter owns rule lookup.
POST /v1/check {
  identity: { user_id?, ip?, api_key?, route, method, ... },
  cost: 1
} → {
  allowed,
  decisions: [ { rule_name, bucket_id, allowed, remaining, reset_at, scope } ],
  retry_after?
}

identity 是一个包含请求所有属性的扁平包——在这一层区分"谁"（user_id）和"什么"（路由）没有实际意义，因为类似"每 IP 每路由 5 次登录/小时"的规则同时需要两者，而且只有规则本身才知道它关注哪些字段。调用方完整描述请求，限流器将身份投影到每条匹配规则所键控的字段上。

为什么调用方不传递 rule_id. 规则是限流器持有的配置。允许调用方命名规则，会将每个网关与规则目录耦合在一起——新策略意味着网关需要重新部署，而过时的客户端会与真实配置产生偏差。通过传递原始请求上下文，限流器可以在不修改任何客户端的情况下新增、废弃或调整规则。

公共响应头 （当网关返回 429 时）： X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset, X-RateLimit-Bucket, X-RateLimit-Scope (user / shared / global), Retry-After。Discord 的 X-RateLimit-Bucket 是不透明的，以便服务器可以在不破坏客户端的情况下重新分片分桶策略——详见深度解析。

Sidecar 替代方案。 一个直接与 Redis/Memcached 连接池通信的库——算法相同，无需额外的网络跳转。Sidecar 从配置服务拉取规则并在本地进行匹配；对应用代码的公开 API 接口保持不变（仍然传递身份包，而非规则 ID）。根据开销限制，我会同时提供两种方案。

5. 高层设计（约 10–15 分钟）

  1. 限流服务/库。 接收身份包。从内存规则表中查找匹配规则，将身份投影到每条规则的键维度，查询计数器存储，并跨层聚合决策结果。调用方不需要指定规则名称。
  2. 计数器存储。 Redis Cluster。每个桶对应一个字符串键（例如 rl:user:42:postTweets:202605181847）；Redis 将该键哈希到 16,384 个槽之一，并路由到拥有该槽的分片——因此桶均匀分布在各分片上，每个桶恰好存在于一个节点上。计数器以算法对应的数据结构存储在内存中（GCRA 使用一个 TAT 整数，滑动窗口使用两个计数器）。通过 TTL = 窗口长度进行淘汰。与限流器共置于同一可用区，以将 p99 延迟控制在预算范围内。
  3. 配置服务。 持有规则目录（单一真实来源）。将更新推送给限流器实例（长轮询或通过 etcd/Consul）。限流器以原子方式热重载匹配器表。新规则首先以 仅观察模式 启动（详见深度解析）。
  4. 本地预检/判决缓存。 进程内状态——详见深度解析（热键、自适应执行）——无需网络跳转即可处理简单情况。
  5. 决策流水线。 每次决策都会向指标和日志流水线发出 (rule_name, allowed, layer, bucket_id, identity_hash) 信号。运维人员可以按规则查看拒绝率；部分决策会写入结构化日志，用于离线调优和暗启动对比。

端到端流程叙述。 一个对 POST /tweets 的请求到达网关。网关调用限流器并传入 identity={user_id:42, ip:1.2.3.4, route:/tweets, method:POST}。限流器按顺序遍历各层——全局流量卸载（廉价，内存计算）、按路由（"写操作 10k QPS"）、按租户（"每用户 100 req/min"）、滥用检测（"最近无效请求率是否正常"）。每条匹配规则生成一个桶键（rl:user:42:postTweets:202605181847），通过 Lua 脚本以原子方式更新。若任一层拒绝，则返回 429 并携带 Retry-After 及桶头部信息。否则放行。

6. 深度探讨（约 10 分钟）

具体的键和内存占用。

全局流量卸载 shed:{route}:{second} 进程内计数器，约 16 B 2–5 s 本地实例

触及各存储的检查路径。 1. 网关调用 /v1/check 并传入扁平身份包和请求代价。\n2. 限流器在内存中匹配规则；全局流量卸载和本地拒绝缓存检查优先执行，无需 Redis 跳转。\n3. 对每条匹配的严格规则，限流器构建桶键并将 Lua/GCRA 命令流水线发送到 Redis。独立规则的桶可以并行执行，但每个桶在一个 Redis 分片上以原子方式更新。\n4. 按最严格结果聚合决策。若任何硬规则拒绝，则返回 429 并携带 Retry-After；仅观察的软规则只发出指标。\n5. 决策事件异步发送到指标/日志；日志记录失败不会阻塞请求。

"我的核心观点：限流器是廉价本地决策与精确共享计数器的层级结构；让每条规则都承担最强一致性代价，才是可扩展性的根本缺陷。"

热/冷桶行为。 1. 新建或休眠的身份：Redis 键不存在，Lua/GCRA 脚本创建 rl:{rule}:{identity_hash} ，TTL = 突发窗口 + 空闲宽限期。无需清理扫描。\n2. 健康的冷桶：乐观本地放行仅允许用于标记为软模式的规则；异步增量响应携带剩余配额信号。\n3. 接近限制的桶：限流器切换到严格同步模式，在允许请求前原子更新同一 Redis 键。\n4. 耗尽的桶：限流器将 deny:{rule}:{identity_hash} 写入本地，直到 reset_at，从而使重复的恶意请求不再持续命中 Redis。\n5. 规则形状变更： identity_hash 命名空间随 ruleset:{version}变化，旧桶自然过期，新桶以干净状态开始。

双视图框架。 计数器存储是以 (rule, identity_subset)为键的执行索引。决策流是以规则、租户、路由和时间为键的分析视图。仪表盘和配额导出可以有延迟或采样；Redis 桶决策不允许。

算法。 - GCRA（通用信元速率算法） ——我的默认选择。每个桶一个"理论到达时间"整数。通过单条 Redis 命令实现原子操作（CL.THROTTLE 通过 redis-cell）。数学上等价于漏桶，但消除了滴漏进程故障模式和朴素令牌桶中的非原子读-修改-写竞争问题。\n- 滑动窗口计数器 ——两个固定窗口计数器按经过时间比例混合： rate = prev * (1 − elapsed/window) + curr。Cloudflare 在生产规模下测量到约 0.003% 的误分类率。廉价且可预测。\n- 令牌桶 ——天然适合突发容忍；需要精心编写 Lua 以保持原子性。当策略明确需要突发 + 稳定速率语义时使用。\n- 固定窗口 ——仅当边界处的突发（可能达到 2× 限制）不重要时使用。\n- 滑动窗口日志 ——精确但每桶内存为 O(窗口 × QPS)。不要在大规模场景下使用。

计数器同步——三种模型。 - 集中式计数器存储（Redis）。 Stripe、Envoy、Kong。单个 Redis Cluster，每个桶键哈希到一个分片。简单且精确。代价：每次检查一次网络跳转。这是我对按租户和配额层的默认方案。\n- Anycast 固定本地计数器。 Cloudflare。源 IP 路由将流量固定到一个 PoP，因此按 IP 的桶本地存储在 PoP 上，无需跨 PoP 同步。零跳转，但无法用于按用户限流（同一用户可能访问多个 PoP）。\n- 协作式租约分发。 Google 的 Doorman。限流器向主节点申请 租约 （例如 5 秒内 1000 req/s），并定期刷新。以每请求精度换取在无每请求跳转的情况下执行大规模全局限制的能力。适用于限制值较大且客户端较多的场景。

为什么默认计数器存储选 Redis 而非 DynamoDB/Cassandra：限流器需要个位数毫秒级的原子增量加 TTL，而不是持久查询历史。DynamoDB 支持条件写入，但其尾延迟和每请求成本在 1M QPS 下并不适合；Cassandra 擅长写日志，但在按请求原子计数器方面较弱。

热键。 知名用户或爬虫 IP 会产生热桶。应对措施叠加使用：\n- 在本地缓存拒绝判决 一旦某个桶超出限制（Cloudflare、Envoy freecache）。来自该桶的后续请求完全跳过存储，直到 TTL 到期。反直觉的是：在最坏情况路径下，攻击期间后端负载 下降 。\n- 廉价层先于昂贵层。 边缘按 IP（内存）；后端按用户（Redis）。\n- 计数器分片 用于绝对热点桶——哈希到 N 个子键，读时求和。以一次读操作换取写热点的分散。\n- 客户端微缓存 对放行决策（例如 50ms 窗口）缓存在网关端。

边缘压力测试。

边缘 故障模式 降级 / 预算
休眠身份 桶已过期 在第一次请求时重建；一次 Redis 跳转
耗尽的热桶 攻击者持续命中 429 路径 本地拒绝缓存直到重置

延迟预算。 5 ms p99 意味着最多一次 Redis 跳转。将 Redis 与网关共置于同一可用区。连接池、流水线和 Lua 脚本将网络往返次数控制在一次。

自适应执行（预测而非检查）。 ≫99% 的 (identity, rule) 键值对从未接近其限制——对每个请求都付出 Redis 跳转的代价是浪费的。在限流器中维护每个桶的状态： 冷启动 （计数器远低于限制，拒绝率 ≈ 0）和 hot （接近限制或拒绝率上升）：\n- 冷路径： 立即放行，异步触发增量（批量/流水线）。延迟 ≈ 0。\n- 热路径： 通过 Lua 脚本进行同步先检查后放行。严格模式。

状态转换：当计数器超过限制的约 80% 或出现异步拒绝时，由冷转热；经过一段安静窗口后，由热转冷。信号通过异步增量响应返回——Redis 告知"你已达到 95%"，我们切换到热模式。代价：在冷转热边界处存在短暂的过量放行。对软限制（分页配额）可接受；对硬限制（认证、滥用检测）绝不适用。按规则选择开启。

阶段 模式 数据源 行为
冷层 乐观放行 本地状态 + 异步 Redis 增量 最低延迟，有界过量放行
温层 采样同步 周期性 Redis 检查 在达到限制前检测漂移
热层 严格同步 Redis Lua/GCRA 请求通过前的精确决策
存储故障 下限模式 本地安全容量桶 按规则选择带下限的 fail-open 或 fail-closed

带下限的 fail-open，而非二元选择。 如果计数器存储不可达，二元选择（fail-open = 滥用流量放行；fail-closed = 自我引发故障）对大多数规则来说都是错误的。每个网关对每个身份携带一个 安全容量下限 （例如，"如果 Redis 不可用，每用户每秒最多允许 10 次请求，从本地内存执行"）。硬规则（认证、支付）仍然 fail-closed；软规则则采用带下限的 fail-open。Doorman 对此进行了明确建模；Stripe 通过 fail-open 加上一个并发请求限制器来近似实现，防止故障客户端独占工作线程。

通过响应头实现桶发现。 服务器返回 X-RateLimit-Bucket: <opaque> 以便行为良好的客户端可以对共享一个桶的端点进行分组并正确退避，而无需硬编码限制值。服务器控制分桶策略，可以重新分片，客户端自动适应。与 X-RateLimit-Scope 配合使用，让客户端知道哪一层触发了限流（用户/共享/全局），以及是应该退避当前令牌还是所有令牌。

代价模型。 并非所有请求代价相同：\n- 积分制 （GitHub：GET=1，mutation=5；每分钟 900 积分）。计数器按请求代价增量——通过在 Lua 脚本中传递 cost 来轻松支持。\n- GraphQL 查询复杂度 ——在解析时累加字段代价，一次性计费。\n- 并发在飞请求限制器 — 与 RPS 限制不同的 概念。一个用户发出 100 个慢速昂贵请求可能耗尽工作线程而不超过 RPS 限制。Stripe 将此作为独立层运行，用于捕获 RPS 限制遗漏的情况。

不暴露限制值（滥用层）。 对于对抗性规则，429 + Retry-After 会告诉攻击者确切的重试时机。Figma 的模式： 影子封禁 ——返回 200，悄悄丢弃或延迟请求。攻击者无法判断自己是否被限流，也无法切换到新凭证。仅在疑似机器人层使用影子封禁；对面向开发者的 API 配额绝不使用，因为透明反馈是服务契约的一部分。

暗启动/影子模式。 永远不要冷启动执行一条新规则。先以 仅观察 模式运行数天——统计 本会 被拦截的请求数量，在仪表盘上查看判决结果，调整阈值。Stripe 和 Envoy 都将此作为一等模式内置。成本低廉；能避免意外地对关键批处理任务返回 429。

多租户公平性。 嘈杂的租户不得拖慢其他租户。按租户（或按层级）对 Redis 进行分区，使一个租户的热桶不会驱逐另一个租户的数据。结合堆栈中的按租户层，确保滥用租户的拒绝工作不会消耗共享预算。

配置同步。 规则变更通过配置服务传播。限流器监听变化；收到变更时，以原子方式替换规则表。飞行中的请求按旧配置完成处理——这是可以接受的。

时钟偏差。 计数器在同一时刻只存在于一个地方——Redis 服务器的时钟（集中式）或 PoP 本地时钟（任播）。不要信任客户端时间戳。对于基于租约的系统，租约是秒级别的，NTP 偏差可以忽略不计。
