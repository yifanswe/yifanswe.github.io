Ticketmaster — Concert Ticket Sales (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Medium · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建 Ticketmaster。某活动于上午 10 点开售，10 万人涌入排队，共有 2 万个座位。每个座位是唯一的——一旦售出即消失。我们需要公平排队、不重复销售，以及流畅的选座 UI。平时负载较小；闪购峰值才是面试的核心所在。

1. 需求（约 5 分钟）

功能需求 - 浏览活动并从可视化地图中选择座位。- 在用户输入支付信息时，将选中的座位保留约 10 分钟。- 完成购买；座位变为永久售出状态。

我会推迟转售、订阅和场馆管理端的功能。

非功能需求 - 永不重复销售。 比“五个九”更强——这是一份正确性契约。- 闪购期间的公平性——先到先得。- 10 万用户涌入时，选座地图页面 p99 < 500 ms。- 持有/购买后的读己写：用户必须立即看到自己的持有状态。- 对其他用户显示的可用数量可最终一致（轻微过时可以；超卖不行）。

容量估算 - 基准：1 万个活跃活动，平均 100 次/秒读取。- 闪购：10 万并发用户 → 10 万个排队条目，座位图轮询或推送频率约 1Hz → 持续数分钟的 10 万 QPS 峰值。- 存储量相对于突发协调来说较小：1 万个活动 × 每活动 2 万个座位 × 约 200 B/座位行 ≈ 40 GB 主座位数据，含 3 个副本约 120 GB。单个 10 万人排队约 80 B/成员，仅约 8 MB；难点在于公平性和热门活动扇出，而非字节数。- 主要实时成本是座位图叠加层通道：若 10 万客户端每 2 秒获得 2 KB 差异，在 CDN/边缘扇出之前，单次热门销售约为 ~100 MB/s。

2. 核心实体（约 2 分钟）

  • 活动 — event_id、场馆、开始时间、销售窗口。
  • 座位 — seat_id、event_id、区/排/号、状态（available/held/sold）、价格。
  • 锁定（Hold） — seat_id、user_id、expires_at。基于 TTL。
  • 订单 — order_id、user_id、event_id、seat_ids[]、payment_id。
  • QueueTicket — event_id、user_id、position。

3. API（约 5 分钟）

GET   /v1/events/{id}/map           → SeatMap (compressed)
POST  /v1/events/{id}/queue         → { ticket_id, position }
GET   /v1/queue/{ticket_id}         → { position, ready, allowed_until }
POST  /v1/events/{id}/seats/{seat_id}/hold     (idempotency-key required) → { hold_id, expires_at }
POST  /v1/orders                     { holds[], payment } → { order_id, status }
DELETE /v1/holds/{hold_id}

幂等性键在 hold 。 orders 上是必需的——销售期间不稳定的网络会进行重试。

排队票据是不透明且经过签名的；它编码了 event_id, user_id, queue_epoch，以及 allowed_until，因此座位服务可以拒绝绕过等候室的用户，而无需在每次点击时同步查询排队状态。

5. 高层设计（约 10–15 分钟）

  1. 静态活动目录。 在边缘缓存。浏览活动不访问主存储。
  2. 座位图服务。 渲染场馆地图。地图主体是静态的；只有每个座位的状态叠加层是动态的。我会从 CDN 提供静态骨架，仅从 API 获取紧凑的 {seat_id: status} 差异。
  3. 排队服务。 活动开售时，每个到来的用户都会获得一张票据和一个位置号。实现为以 event_id为键的 Redis 有序集合，分数 = 到达时间戳。服务以座位服务可处理的速率（例如 1000/秒）将用户滴入“允许购买”状态。用户持有一个 SSE/WS 连接，该连接推送其位置号。
  4. 座位库存服务。 管理座位状态。持有状态通过 Redis 以 TTL 为键（seat:{event}:{seat} 值 = user_id，TTL 600s）。购买操作将 Redis 持有状态转换并在 MySQL 中写入永久行（按 event_id 分片）。
  5. 订单服务。 协调支付并最终确定座位状态。两阶段：预授权支付 → 最终确定座位行 → 捕获支付。支付失败时，回滚 Redis 持有状态。
  6. 通知服务。 持有即将到期、排队就绪、支付已确认。

端到端叙述：Alice 在上午 9:59 打开应用。10:00 时她点击“购买票”→ 排队服务颁发第 4520 号票（共 10 万）→ SSE 每秒推送她的位置。约 10:03 时她被允许进入 → 座位图渲染 → 她选择 A14 座位 → POST /hold 携带幂等性键 → Redis SETNX 含 TTL 600 成功 → 她进入支付流程 → POST /orders → 支付捕获，MySQL 行 seat A14 sold to Alice 已提交，Redis 键更新为终态。若她超时，Redis TTL 将自动释放座位并通过座位图推送通道将其返回可用池。

6. 深度探讨（约 10 分钟）

具体存储契约。

存储 键/数据结构 条目大小 + TTL 分片键
Queue queue:{event_id} Redis 有序集合，成员=ticket_id，分数=arrival_ts 约 80 B/成员，TTL 为销售期 + 24h event_id
允许窗口 allowed:{event_id} ticket_id 的 Redis 集合 约 32 B/成员，TTL 15 分钟 event_id
锁定（Hold） hold:{event_id}:{seat_id} Redis string/json {user_id, hold_id} 通过 SET NX EX 600 约 120 B，TTL 10 分钟 event_id
座位真实数据 MySQL seats(event_id, seat_id, status, version) 约 200 B，永久保留 event_id，可选 section_id 用于大型场馆
售出唯一性 MySQL order_seats(event_id, seat_id, order_id) unique (event_id, seat_id) 约 80 B，永久保留 event_id
幂等性 idem:{user_id}:{key} → 先前响应 约 1 KB，TTL 24h user_id
座位叠加层 seat_status:{event_id} Redis bitmap/hash + 版本号 1–4 B/座位，TTL 活跃销售期 + 1h event_id

各存储容量估算。 持久化库存量较小： 10k events × 20k seats × 200 B × 3 约为 120 GB。热门活动排队也较小： 100k members × 80 B × 3 约为 24 MB。已售罄 2 万座位活动的持有状态约为 2.4 MB 主数据。实时压力来自网络和协调： 2 KB 座位差异每 2 秒推送给 10 万客户端，在边缘扇出前约为 ~100 MB/s，因此批量处理和按活动通道比数据库字节数更重要。幂等性受销售流量限制： 100k attempts × 1 KB 在单个热门活动的 24 小时内约为 100 MB。

持有与购买路径追踪。 持有路径：客户端 → 使用签名票据进行排队权限检查 → 幂等性查询 → Redis SET hold:{event}:{seat} NX EX 600 → 更新座位叠加层版本 → 在 event:{id}:seat_updates上发布批量差异。购买路径：客户端 → 订单服务检查幂等性 → 支付预授权 → MySQL 事务以唯一键插入 order_seats(event_id, seat_id) → 更新 seats.status='sold' → 捕获支付 → 删除/标记 Redis 持有为终态。Redis 持有使 UI 公平；MySQL 唯一插入是正确性的保障线。

Hot and cold API paths。

API path 热门销售路径 Cold/fallback path 游标/会话状态
POST /queue 将签名票据添加到 queue:{event_id} 有序集合；SSE 流式传输位置号 若 SSE 断开，客户端带抖动轮询 GET /queue/{ticket_id} （带抖动） 票据编码 event_id,user_id,queue_epoch,allowed_until
GET /map CDN 静态场馆地图 + 按版本号的 Redis 叠加层差异 休眠活动从 MySQL 座位行懒加载叠加层 since_version 用于叠加层差异
POST /hold 签名权限 → 幂等性 → Redis SET NX EX → 叠加层差异 若叠加层过时，Redis/MySQL 真实数据会拒绝不可用座位；用户重新选择 hold_id，10 分钟 TTL，幂等性键
POST /orders 支付预授权 → MySQL 唯一插入/更新 → 支付捕获 支付缓慢时延长一次持有；数据库冲突释放预授权并返回售罄座位 订单幂等性、支付预授权 id、座位版本

若客户端对座位图进行翻页或缩放超出缓存的区段叠加层，则从 MySQL 获取区段范围快照，并使用返回的 overlay_version.

为什么使用 Redis + MySQL，而非单一存储。 Redis 非常适合排队位置、过期持有和高频叠加层差异，因为这些数据是短暂的且对延迟敏感。MySQL 非常适合已售座位，因为我们需要持久化事务、行锁和唯一约束。纯 Redis 方案速度快但对金融交易来说风险高；纯 MySQL 方案正确但在 10 万用户轮询座位状态时会崩溃。面试备答：“Redis 协调销售；MySQL 决定所有权。”

与仅使用 Spanner 相比，Redis + MySQL 将热门的、过期的交互状态保持在靠近用户的低成本位置，同时将最终所有权留在事务存储中。与 ZooKeeper/etcd 锁相比，Redis 持有在销售规模下更简单，因为租约是面向用户的库存状态，而非仅是协调原语。

防止重复销售。 单一事实来源是 MySQL，在 (seat_id, sold=true)上具有唯一约束。Redis 中的持有是尽力而为的协调，防止用户抢占同一座位——它们是优化手段，而非安全网。正确实现 INSERT ... ON DUPLICATE KEY （或 SELECT ... FOR UPDATE）才是真正的保障。

面试核心观点：“我不信任缓存来销售座位；Redis 改善了购买体验，但防止诉讼的是数据库约束。”

排队的规模化。 每个活动一个 Redis 有序集合是最简单的方案。对于非常大型的活动，按 hash(user_id) 将排队分片为 N 个子队列；每个子队列以 rate/N 的速率滴入。座位服务的速率限制才是真正的瓶颈——排队是有礼貌的等候室。

MySQL 中的热门行。 前排座位是所有人的第一点击目标。按 event_id 分片，使一个活动的所有座位落在同一分片上；在分片内，每个座位的行级锁是短暂的（亚毫秒级）。对于最大型的活动，在活动内按区段进一步分片是合理的。

持有过期。 不要依赖清理程序来清除持有状态。Redis TTL 会处理它。当 TTL 过期时，座位图推送通道发布“A14 座位已释放”（Redis 键空间通知或来自定时器服务的显式释放调用）。

座位图更新而不耗尽网络。 不要将每次状态变更广播给每个用户。每个打开的座位图订阅一个按活动划分的通道，该通道每 1–2 秒批量更新一次。发送紧凑差异（仅变更的座位）而非完整地图。

双视图框架。 场馆地图是静态内容加上动态可用性叠加层。同一座位也有两种状态视图：Redis 持有状态用于临时 UX 协调，MySQL 售出状态用于永久所有权。叠加层由这些状态派生；即使叠加层有误，持有/购买 API 仍然执行真实数据。

防机器人。 在排队入口处使用验证码；按 IP + 设备指纹进行速率限制；排队位置经过签名（HMAC）因此无法伪造；标记持续以超人速度访问 API 的账号。

幂等性。 持有和订单上的幂等性键能够在不稳定的移动网络连接期间存活客户端重试。服务器存储 (idempotency_key, response) 24 小时并对重复请求返回缓存响应。

优雅降级状态机。

情形 模式 数据真实来源 延迟预算 回退
热门活动 排队滴入 + 批量叠加层 Redis 排队 + MySQL 座位行 排队更新 1 秒，座位持有 <100 ms 降低滴入速率；显示过时的可用数量但拒绝无效持有
冷门/休眠活动 直接加载座位图 MySQL 快照 + Redis 叠加层懒加载 <500 毫秒 在首次请求时构建叠加层并在销售窗口期间缓存
座位图通道耗尽 改为拉取而非推送 seat_status:{event_id} 带版本号的差异 1–2s 客户端轮询 ?since_version= （带抖动）
支付提供商缓慢 延长一次持有 Redis 持有 + 支付预授权状态 最多 +2 分钟 延长一次持有，然后释放以避免库存囤积
Redis 故障 停止受影响活动的新持有 MySQL 售出行 快速失败（fail closed） 对已授权的预扣款保留购买记录；不接受盲目预扣
