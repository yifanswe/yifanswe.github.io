1. 基础知识与解题思路

本题考察点

消息通知系统负责通过推送、邮件、短信和站内消息等渠道向数十亿设备投递消息，通常伴随突发式扇出和大量第三方服务提供商集成。难点在于：在保证正确性约束（退订后不再通知、不重复投递、遵守免打扰时段）的同时保持高性能，并能抵御服务提供商故障。

其不变量如下：

对于每一条被触发的通知，系统应在每个符合条件的渠道上对每位用户恰好投递一次，以发送时（而非触发时）的当前偏好和授权状态为准，并生成持久化的投递回执，区分"已投递"、"失败"和"被策略抑制"三种状态。

在 ICT5 / L6+ 级别，面试官考察的是你能否将触发与投递分离，设计幂等的跨渠道扇出，管理每个服务提供商对每位接收者的速率限制，并处理一个事件可能产生数百万条通知这一不对称问题。

60 秒面试框架

口述版本：

我会将通知系统设计为三个平面：事件摄入层（捕获触发器）、策略与模板层（解析接收者及每用户偏好）、以及渠道特定投递层（集成 APNs、FCM、SMTP、短信网关和站内消息 WebSocket）。关键设计决策包括：扇出时的幂等性、以发送时而非触发时应用偏好、服务提供商速率限制，以及如何处理单次触发产生数百万条发送的突发场景。

首先明确需求

功能性需求：

  • 触发器来自产品服务：新消息、点赞、优惠提醒。
  • 系统支持推送（iOS/Android）、邮件、短信、站内消息和 webhook 渠道。
  • 用户可设置每渠道和每分类的偏好。
  • 支持免打扰时段、频率上限和摘要模式。
  • 投递回执对产品团队可见。
  • 支持营销活动的定时发送和受众定向。
  • 运营人员可通过模板快速屏蔽通知（熔断开关）。

非功能性需求：

  • 高吞吐量：每天数十亿条通知。
  • 突发扇出：单次触发可产生 1 亿条以上的发送。
  • 事务型通知低延迟：端到端不超过 5 秒。
  • 高投递成功率；服务提供商故障时重试。
  • 发送时必须遵守每位接收者的授权状态。
  • 合规审计要求（CAN-SPAM、TCPA、GDPR）。

以下内容超出范围，除非被特别询问：

  • 内容创作工具。
  • 推送通知中的富媒体内容渲染（视频、动画）。
  • 深度个性化评分（视为输入）。

规模假设

领域 假设
用户 跨区域 10 亿以上用户。
设备数量 2-3 倍于用户数；一个用户拥有多台设备。
每日体量 每日 100 亿条通知；平均约 10 万条/秒，峰值 100 万条/秒。
突发场景 一次营销活动在数分钟内向 1 亿用户发送通知。
Latency（延迟） 事务型延迟 < 5 秒 p95；营销型尽力而为，分钟级。
服务提供商限制 APNs 每连接允许数千条；FCM 每批最多 500 条；短信服务提供商有每号码 TPS 上限。
一致性 发送时偏好强一致性；投递回执最终一致性。

瓶颈在于服务提供商速率限制以及将单次触发映射到大量接收者的扇出成本。

核心产品模型

平面 职责 需求
触发平面 捕获可能需要通知某人的事件。 持久化摄入，幂等处理。
受众与策略平面 解析接收者；应用用户偏好和速率上限。 偏好处理强一致性。
投递平面 渲染内容；通过渠道特定的服务提供商发送；追踪回执。 至少一次投递，每接收者去重。

最常见的错误是将系统视为单一流水线。产品正确性（授权、频率）与投递机制（服务提供商重试、速率限制）是相互独立的。

核心组件

组件 职责 关键细节
触发服务 接收事件，通过 trigger_id 去重，持久化。 基于 trigger_id 的幂等摄入。
受众解析器 从事件元数据计算接收者列表。 从用户/关注者/分群服务读取数据。
偏好服务 存储每用户每渠道每分类的偏好和授权状态。 发送时强一致性读取。
频率上限服务 追踪每用户最近的通知量以执行上限限制。 每用户计数器；通常使用 Redis。
模板引擎 使用个性化占位符渲染内容；支持本地化。 版本化模板；预览工具。
渠道 worker 每个渠道的发送逻辑：推送、邮件、短信、站内消息、webhook。 每个 worker 集成对应服务提供商的协议。
Provider 限流器 对每个 Provider 的出站调用进行限速。 按 app-id、IP、账户的 TPS 上限。
投递账本 存储发送尝试、回执、退信、打开记录。 仅追加写入；按日期分区。
屏蔽列表 硬退信、取消订阅、滥用投诉。 权威数据源；每次发送前均需检查。
熔断开关 按模板/按类别的紧急拦截。 在 Worker 中缓存；更新在数秒内传播。

具体的存储与队列

存储/队列 键模式 值 容量/上限 TTL/生命周期 分片键
Trigger 存储 trigger:{trigger_id} source、受众规格、模板、状态 1–4KB 热数据保留 30–90 天；之后归档 trigger_id
发送尝试存储 attempt:{yyyyMMdd}:{user_id}:{attempt_id} channel、状态、provider id、尝试次数 300–700B 热数据保留 30 天；审计归档 13 个月 user_id
偏好存储 pref:{user_id} channel/category 偏好、免打扰时段、版本 1–2KB 永久保留直至账户删除；缓存 5–30s user_id
屏蔽存储 suppress:{channel}:{contact_hash} 原因、来源、created_at 100–250B 永久保留，除非用户明确重新订阅 contact_hash
频率限制 freq:{user_id}:{category}:{yyyyMMdd} 按优先级/模板的计数器 50–200B 32 天 user_id
Channel 队列 queue:{channel}:{priority}:{partition} attempt id + 渲染上下文指针 200–500B/消息 保留 7–14 天或进入 DLQ channel + partition
Provider 限流器 bucket:{provider}:{region}:{sub_key} 令牌桶状态 <200B 内存存储；每隔几秒做一次 checkpoint provider + sub_key

容量估算：

  • 1 亿日活用户 (DAU) * 每天 20 次通知尝试 = 每天 20 亿次尝试。以 500B/次计算，热尝试存储每天写入约 1TB 原始数据；加上 3 个副本，归档压缩前约为每天 3TB。
  • 平均入队速率为 20 亿 / 86,400 ≈ 每秒 2.3 万次尝试。10 倍产品突发流量为每秒 23 万次；以 500B/消息计算，一小时积压约 414GB 原始数据，因此队列需要按 channel 分区并设置保留上限。
  • 偏好数据量较小，但对正确性至关重要：10 亿用户 * 1.5KB * 3 个副本约为 4.5TB。按用户分片保持强一致性的成本完全可接受。
  • 屏蔽列表增长缓慢，但不能自动过期，因为法律退订和硬退信是需要持久保存的记录。

技术选型依据：

  • 对 Channel Worker 使用持久队列/日志，因为 Provider 故障可能持续数小时。仅用 Redis 的队列速度虽快，但重放和审计能力较弱。
  • 对偏好/屏蔽存储使用强一致性，因为读到过期的取消订阅记录属于正确性违规。短时缓存只有在 Worker 在调用 Provider 前重新检查的情况下才可接受。
  • 使用按 channel 独立的队列而非一个全局队列，这样 SMS 限速或 APNs 故障就不会阻塞邮件或站内信的投递。

最大的存储不是偏好数据，而是每用户每 channel 的尝试日志——它是审计链路和幂等性的载体。

扇出流水线

这是热路径：

  1. Trigger 摄入。 产品服务发出 (trigger_id, audience_spec, template_id, ...)。Trigger 服务以 trigger_id为键执行条件写入。重复请求返回相同响应。
  2. 受众解析。 Resolver 从用户/关注者/分群服务读取数据。对于超大受众（数百万用户），以流式方式将 user_id 分块传入下一阶段，而非将全量数据加载到内存。
  3. 偏好过滤。 对每个用户查询偏好、检查全局屏蔽列表、检查熔断开关、检查频率限制。丢弃被屏蔽的用户，并写入一条 suppressed 用于审计的 SendAttempt。
  4. Channel 选择。 根据剩余偏好和 Trigger 允许的 channel，为每个用户选择一组 channel。
  5. 按 channel 入队。 生成一条带有确定性 id 的 SendAttempt(trigger_id, user_id, channel) 记录，并入队到特定 channel 的 Worker 队列。
  6. Channel Worker。 从队列拉取消息、用个性化内容渲染模板、调用 Provider、等待确认、写入回执。

确定性的 SendAttempt id 使扇出具有幂等性：如果扇出 Worker 在解析中途崩溃并重试，会生成相同的 attempt id，条件写入会自动去重。

面试用语：

幂等性作用在每用户每 channel 的粒度，而非 Trigger 粒度。Trigger id 与用户 id 共同生成一个确定性的 attempt id，因此在扇出任意阶段发生重试都能收敛。

Trigger 写路径：

  1. 产品服务调用 POST /triggers 与 trigger_id 作为幂等性键。
  2. Trigger 服务条件写入 trigger:{trigger_id}。重复写入返回已有的 Trigger 状态。
  3. Resolver 将 audience_spec 展开为用户 id 分块。分块任务以 (trigger_id, chunk_id) 为键，重试不会产生重复进度。
  4. 扇出 Worker 并行读取偏好 pref:{user_id}、屏蔽列表 suppress:{channel}:{contact_hash}、频率限制 freq:{user_id}:{category}:{yyyyMMdd}以及模板熔断状态。
  5. 对每个允许的 channel，Worker 条件写入 attempt:{date}:{user_id}:{attempt_id} 并入队 queue:{channel}:{priority}:{partition} ，其中 partition 通常为 hash(user_id).
  6. Channel Worker 拉取队列，重新检查偏好/屏蔽以满足合规要求，渲染模板，等待 Provider 令牌桶，发送，然后更新 attempt 行。

站内信读路径：

  1. 客户端调用 GET /users/{user_id}/notifications?cursor=；分片键为 user_id.
  2. 热读命中按 (user_id, created_at desc, notification_id)排序的每用户收件箱表，游标携带最后时间戳和通知 id。
  3. 缓存未命中或不活跃用户回退到尝试/收件箱存储；首页可以稍慢，但仍须应用已删除模板和屏蔽状态。
  4. 游标耗尽时返回空页面和终止游标。避免使用 offset 分页，因为用户滚动时收件箱可能发生变更。

写路径示意：

Product service -> Trigger store -> Audience resolver -> Fan-out workers
                |              |\
                |              | +--> Preference / suppression / frequency
                |              +----> Attempt store
                v
              Channel queues -> Provider workers -> APNs/FCM/Email/SMS

发送时检查偏好，而非 Trigger 时检查

计划营销 Trigger 可能在发送前数小时入队，用户可能在此期间取消订阅。

  • 偏好和屏蔽列表在 发送时检查，而非 Trigger 时检查。
  • 发送 Worker 在调用 Provider 之前从强一致性偏好存储读取数据。
  • 短时缓存可以接受（秒级），但不能更长；屏蔽列表的读取必须反映最近的取消订阅操作。

这将更多工作推迟到发送时，但这是唯一安全的做法。频率限制和免打扰时段同样在发送时生效。

各 Channel 的特殊注意事项

Push (APNs/FCM)： - 维护设备 token 注册表；token 会过期，必须刷新。\n- 对 APNs 使用 HTTP/2 多路复用连接；FCM 提供批量发送 API。\n- 追踪每台设备的投递情况；一个用户持有五台设备将产生五次尝试。\n- 收到 Provider 返回的 Unregistered 时，将该设备标记为屏蔽。

邮件 (SMTP / SendGrid / SES)： - 遵守 SPF/DKIM/DMARC；声誉管理至关重要。\n- 硬退信加入屏蔽列表；软退信按退避策略重试。\n- 退订链接必须包含签名 token；主流邮箱服务商要求支持一键退订。\n- 按域名的声誉限速。

SMS： - 各国法规（美国 TCPA、opt-in 要求）。\n- 运营商对每个号码的 TPS 上限。\n- 长号码与短号码；定价影响模板选择。\n- 部分国家（印度等）要求 DLT/监管模板。

站内信： - WebSocket 扇出或由每用户通知收件箱表驱动的 pub/sub。\n- 即使用户离线也会存储；客户端在下次会话时轮询或订阅。

Webhook： - 客户提供的 URL，使用签名 payload。\n- 重试策略；针对每个目标 URL 的熔断器。

Provider 限流

每个 Provider 都有自己的约束：

  • APNs：每个 HTTP/2 连接数千个并发流；每个 token 授权机构的连接数有限。
  • FCM：基础每项目每秒 600 条消息；批量 API 可放大吞吐量。
  • SMS Provider：每号码 TPS、每国 MPS 限制。

实现方式：

  • 按 Provider 设置专属令牌桶 (provider, sub-key).
  • Worker 在调用 Provider 前阻塞等待令牌桶。
  • 超出负载在 Worker 处排队；若队列过深，则丢弃非事务性 Trigger。

投递降级状态：

阶段 模式 数据源 延迟预算 状态转换
1 正常发送 Channel 队列 + Provider 事务型 <5s，营销型数分钟 Provider 健康且令牌桶有余量。
2 限速中 队列等待令牌桶 事务型 <30s，营销型数小时 Provider 配额已达上限。
3 Provider 故障 重试队列 + 熔断器 取决于优先级 错误率超过阈值。
4 跨 Provider 故障转移 若模板/channel 允许，切换到备用 Provider 事务型 <2 分钟 主 Provider 故障且策略允许故障转移。
5 进入死信队列/屏蔽 DLQ 或屏蔽存储 仅用于审计 重试预算耗尽或发生硬退信。

压力边界场景：

  • 热点名人 Trigger：以分块方式流式处理受众，限制每个 Trigger 的扇出速率，并通过已完成分块数暴露进度。
  • 不活跃用户：偏好缓存未命中时读取强一致性存储；站内信首页可冷读归档，但发送时的屏蔽检查仍然适用。
  • 突发写入：事务型优先级获得预留队列容量；营销 Trigger 在消耗 Provider 配额前被延迟或丢弃。
  • 多地域漂移：扇出在用户的归属地域执行，远端地域转发 Trigger 分块而非读取过期的偏好副本。

优雅降级是优先级感知的：事务型通知等待更长时间并可能触发故障转移；营销通知是卸压阀。

扩展与分区

数据 分区键 原因
Trigger trigger_id 哈希值 均匀分布；幂等写入。
发送尝试 user_id 当前数据按 hash 分片；归档按日期分区 按用户查询；长尾历史记录。
偏好 user_id 哈希值 每次发送时读取。

对于大规模扇出（一个 Trigger -> 1 亿用户），受众 Resolver 以流式方式分发给一组并行的扇出 Worker。Trigger 记录携带已完成分片的计数器，使进度可观测。

一致性模型

层级 一致性 原因
偏好 强一致性 发送时检查同意状态。
屏蔽列表 强一致性 取消订阅后不得再发通知。
发送尝试去重 强去重（per-attempt CAS） 至少投递一次，用户可见最多一次。
投递回执 最终一致 用于分析。
站内信收件箱 每用户强一致性（读己写） 用户可立即看到自己的通知。
频率限制 近似一致（Redis 计数器） 轻微超计数可以接受。

故障模式与缓解措施

故障模式 缓解措施 监控指标
Provider 故障 退避重试；每个 Provider 设置熔断器；尽可能跨 Provider 故障转移。 Provider 错误率；按 channel 统计的队列积压。
受众规模爆炸（关注者数量激增）。 流式解析；按 Trigger 限制扇出速率。 每 Trigger 扇出速率。

需要熔断开关（模板损坏）。 缓存中按模板的熔断标志，在发送时检查。 熔断传播延迟。
有问题的产品导致对单用户的垃圾通知。 频率上限；按模板按用户去重。 频率上限拒绝数。
Webhook 客户端接口失败。 重试 + 熔断器；死信队列。 按客户成功率。
违反免打扰时段。 Worker 在发送时检查；推迟到下一个允许的时间窗口。 免打扰时段违规次数。
应用内收件箱无限增长。 按类别设置保留窗口；用户可手动清除。 收件箱大小分布。
休眠用户缓存未命中。 从权威存储读取偏好/收件箱；接受首次发送或首次翻页较慢。 偏好缓存未命中延迟。
Provider 故障期间队列保留耗尽。 丢弃营销消息，延长事务性消息保留时间，在 DLQ 老化限制前告警。 按队列统计最旧消息的存龄。
多区域偏好数据漂移。 将 fan-out 路由至用户所属区域；Worker 在发送前重新检查。 跨区域转发延迟。

运维指标

正确性指标：

  • 退订后仍发送通知数（必须为零）。
  • 每 (trigger_id, user_id, channel) 重复发送数（必须为零）。
  • Kill-switch 传播延迟。
  • 免打扰时段违规次数。

容量指标：

  • 按优先级分类的端到端延迟。
  • 按 channel 的队列深度。
  • 按 provider 的吞吐量与上限对比。
  • Fan-out Worker 利用率。

业务指标：

  • 按 channel 和模板的打开/点击率。
  • 按模板的退订率。
  • 按 provider 的投递成功率。

多区域策略

  • 用户偏好和抑制列表在全球范围内复制，每个用户（或每个分片）拥有唯一的所属区域。
  • Channel Worker 运行在靠近 provider 的区域；APNs/FCM 在美国，SMS provider 按国家分布。
  • 触发事件可在任意区域产生；fan-out 在用户所属区域执行，以保证偏好一致性。

需要指出的权衡

  • 发送时偏好检查会增加延迟，但这是唯一安全的做法。
  • 按 channel 独立的 Worker 增加了复杂度，但允许针对每个 channel 进行优化。
  • 至少一次投递是不可避免的；用户可见的去重依赖 attempt id。
  • 强一致性的偏好存储限制了吞吐量；缓存 TTL 故意设置得很短。
  • 激进的频率上限保护用户但会降低互动率；按类别调优。
  • 按 provider 的限速以绝对吞吐量换取合规性。

常见陷阱与更优做法

陷阱 更优做法
所有 channel 共用单一队列。 按 channel 独立队列，这样 APNs 抖动不会阻塞邮件发送。
在触发时检查偏好。 在发送时检查；偏好在此处具有权威性。
仅在触发级别保证幂等。 按用户按 channel 的确定性 attempt id。
将 provider 故障视为服务端问题。 熔断器、跨 provider 故障转移、排队并消费。
遗漏抑制列表。 每次发送都查询权威存储；硬退回自动加入列表。
手动 Kill Switch。 版本化、缓存、Worker 内检查；传播在秒级完成。

Staff 级别回答检查清单

  • 三层平面：触发、受众+策略、投递。
  • 来自 (trigger_id, user_id, channel) 的确定性 attempt id。
  • 在发送时检查偏好和抑制列表。
  • 按 channel 独立的 Worker 和速率限制。
  • Kill switch 与模板版本管理。
  • 针对海量受众的 fan-out 流式处理。
  • 多区域部署，按偏好所属区域划分。
  • 合规：opt-in、退订、免打扰时段。

面试用语

通知系统是 fan-out 加策略。fan-out 问题通过确定性 attempt id 和按 channel 独立队列解决；策略问题通过在发送时（而非触发时）读取偏好和抑制列表来解决。最难的场景是一次触发产生 1 亿次发送，同时不违反每用户频率上限或退订设置。

2. 模拟面试

面试官： 设计一个通知系统。

候选人： 我将把它设计为三层平面：触发摄入、受众与策略解析，以及按 channel 的投递。不变量是：对于每一个触发的通知，我们对每个用户每个 channel 尝试一次投递，在发送时尊重用户同意，并产生持久化的回执。我将从需求开始，勾勒数据流，然后深入探讨 fan-out 和 provider 速率限制。

面试官： 有一个触发事件说"通知所有关注 @user_x 的人"，他们有 5000 万粉丝。这是如何工作的？

候选人： 触发记录很小——它携带的是受众规格而非物化列表。受众解析器以分块方式从社交图谱服务中流式获取粉丝 id。每个分块被并行 fan-out 到按 channel 的 Worker。每次 fan-out 会发出一个具有确定性 id 的 SendAttempt hash(trigger_id, user_id, channel)。如果某个 fan-out 分片崩溃并重试，相同的 id 会重新生成，条件插入完成去重。触发事件逐步推进 fanned_out_count 直到完成。

面试官： 如果用户在 fan-out 仍在运行时退订，会怎样？

候选人： 偏好和抑制列表在发送时检查，而非在触发时。按 channel 的发送 Worker 在调用 provider 之前读取当前偏好状态和抑制列表。如果用户在最后一秒退订，我们会发出一条 suppressed SendAttempt 并跳过 provider 调用。这意味着我们接受一些浪费的 fan-out 工作，但用户可见的行为是正确的。

面试官： 你如何端到端地处理幂等性？

候选人： 三个层次。首先，触发事件有幂等键；来自上游的重复触发返回相同的 trigger_id。其次，每个 SendAttempt 有确定性 id；fan-out 处的重试会收敛。第三，各 provider 均支持某种形式的消息 id 或去重；我们将 attempt id 作为 provider 的请求 id 传递，以便关联回执。用户每个 channel 最多看到一条通知。

面试官： APNs 宕机了。会发生什么？

候选人： Push channel Worker 在达到错误阈值后触发熔断器。新的 push 尝试进入队列等待。我们可以路由到备用方案（某些产品使用 APNs + 应用内消息组合）。队列可吸收短暂中断；对于较长时间的中断，我们根据优先级丢弃非事务性触发。事务性发送在队列未满的情况下持续重试。

面试官： 一个 provider 有每秒 1000 的速率上限。如何不超过它？

候选人： 在 channel Worker 前端设置按 provider 的令牌桶。Worker 在调用 provider 之前获取令牌；如果桶为空，则阻塞等待。令牌桶通过 Redis 计数器在各 Worker 分片间复制，使全局速率遵守上限。超出负载在 Worker 队列中排队；如果队列增长超过阈值，触发告警并丢弃低优先级流量。

面试官： 你如何支持免打扰时段？

候选人： 每用户偏好：quiet_hours_start、quiet_hours_end，以用户本地时区为准。在发送时，Worker 检查"用户时区中的当前时间"是否在免打扰时段内。如果是，则推迟：将 SendAttempt 重新入队，延迟到下一个允许的时间窗口，或通过不打扰的 channel（如应用内消息）发送。事务性类别可豁免。

面试官： 你如何处理频率上限？

候选人： Redis 中的按用户按类别计数器。在发送时，Worker 执行自增并检查。如果用户超出上限，则抑制并发出审计记录。计数器按滚动窗口重置。对于超高并发系统，轻微的超量计数是可接受的，因此我们使用近似计数器而非事务性计数器。

面试官： 如何关停一个正在给用户发垃圾消息的坏模板？

候选人： 每模板 Kill switch。运维人员在 Kill-switch 存储中设置标志；更改通过 pub/sub 在秒级传播到 Worker。Worker 在调用 provider 之前检查该标志。已传递给 provider 的进行中发送无法撤回——这就是为什么快速传播至关重要。我们将 Kill 传播延迟作为正确性指标进行监控。

面试官： 多区域？

候选人： 偏好和抑制列表按用户所有——每个用户有一个归属区域。触发事件可在任意区域摄入，但路由到用户归属区域进行 fan-out。Channel Worker 运行在靠近 provider 的区域：APNs/FCM 在美国，SMS 在符合各国监管的区域。投递回执复制回归属区域。这使得 fan-out 期间偏好读取开销低，同时允许 Worker 靠近 provider。

面试官： 你如何支持应用内通知？

候选人： 每用户的收件箱表，存储近期通知。当用户在线时，Worker 写入收件箱并通过 WebSocket / pub/sub 推送。当用户离线时，收件箱等待；客户端在下次会话时拉取。收件箱按类别支持保留策略——促销消息可能保留 30 天，告警类可能保留 90 天。

面试官： 合规方面怎么处理？

候选人： 抑制列表被视为法律记录。硬退回和通过一键退订（邮件使用 RFC 8058）产生的退订会自动加入。我们记录每次发送以供审计。对于 SMS 等受监管 channel，我们在偏好层强制执行国家特定的同意规则，并拒绝针对未同意号码的触发事件。

面试官： 总结。

候选人： 触发平面以幂等方式捕获事件。受众与策略平面流式处理接收者，在发送时应用偏好和抑制列表。按 channel 的 Worker 与 provider 集成，并按 provider 执行速率限制。确定性 SendAttempt id 使端到端重试安全。多区域按用户归属区域划分偏好，按 provider 近邻划分投递。合规性体现在抑制列表和发送时偏好检查中。

3. 练习备注

五分钟版本

  1. 明确：channel、偏好、频率、合规。
  2. 不变量：每个用户每个 channel 发送一次，在发送时获取同意。
  3. 三层平面：触发、受众+策略、投递。
  4. 来自 (trigger_id, user_id, channel) 的确定性 SendAttempt id。
  5. 在发送时检查偏好，而非触发时。
  6. 按 provider 的限速器，使用令牌桶。
  7. 按 channel 独立 Worker；Kill switch；多区域。
  8. 以指标收尾：抑制/发送比，provider 成功率。

一分钟结尾陈述

通知系统是 fan-out 加策略。触发事件以幂等方式摄入；受众以流式方式解析；偏好和抑制列表在发送时应用。按 channel 的 Worker 在按 provider 的速率限制下与 provider 集成。确定性 attempt id 防止重试中出现重复。合规和 Kill switch 存在于策略平面。最难的场景是 1 亿用户 fan-out 同时不违反每用户同意或频率上限，通过流式解析和发送时检查来解决。

最佳面试用语

  • "偏好和抑制列表在发送时应用，而非触发时。"
  • "幂等性存在于按用户按 channel 的 attempt 级别，而非触发级别。"
  • "每个 provider 有其自己的令牌桶和自己的熔断器。"
  • "抑制列表是法律记录，而非缓存。"
  • "Channel Worker 做了舱壁隔离，因此一个 provider 故障不会阻塞其他 provider。"
