设计延迟支付系统

相关文档： 设计支付系统 涵盖通用的授权-扣款流程。 设计类似 cron 的分布式任务调度器 涵盖在未来某时刻触发的问题。本文是二者的交集：一笔计划在未来时间 T 执行的支付，T 可以是从现在起几秒到几个月后。

1. 基础知识与解题思路

本题考察点

延迟支付系统允许用户（或系统）现在承诺一笔支付，但在未来某个时间执行。示例："在每月 15 日扣我的卡"、"在到期日自动支付此发票"、"下周二给朋友转账"、先买后付（BNPL）分期付款、计划商户打款、订阅计费。

难点不在于调度本身，而在于"现在"到"T"之间可能发生的一切变化：

  • 银行卡可能过期或被注销。
  • 用户在 T 时可能没有足够的资金。
  • 用户可能在此期间取消支付。
  • 支付处理商在 T 时可能宕机。
  • 授权（预授权冻结）可能在 T 之前过期。
  • 调度器可能触发两次，或完全错过触发。

其不变量如下：

对于每一笔计划支付，必须且只能发生 {已扣款、已失败、已取消} 中的一种最终状态——绝不能是零次（静默丢弃），也不能是两次（重复扣款），且用户必须收到结果通知。

在行业职业通道 Level 5（ICT5）或高级员工（L6+）级别，面试官考查的是你能否在保证幂等性、承受部分故障、遵守银行卡网络七天授权窗口等约束的前提下，将任务调度器与支付状态机组合起来。

60 秒面试框架

口述版本：

我会将此设计为：支付意图被调度进一个持久化定时器系统，通过支付执行 worker 触发，并以端到端的幂等键遍历严格状态机。关键决策在于如何处理授权时间与扣款时间之间的空隙——预授权会过期，银行卡会过期，资金可能不足。我将依次澄清需求、设计状态机、走查授权和执行路径，并深入探讨重试、取消竞争以及与支付处理商的对账。

首先明确需求

功能性需求：

  • "延迟"的含义：固定的未来时间戳、周期性计划（每月）还是条件触发事件（货到付款）？
  • 最大延迟窗口：秒、小时、天、月、年？这决定了授权策略。
  • 支付方式：银行卡（借记/信用）、自动清算所（ACH）、钱包、先买后付——每种方式有不同的时间约束。
  • 用户能否在计划与执行之间取消？截止时间是什么？
  • 计划后能否修改金额或计划时间？
  • 是否包含周期性计费（月度订阅），还是仅限一次性延迟？
  • 由谁发起：终端用户、商户还是系统？
  • 结果如何通知：邮件、推送、商户 webhook？

非功能性需求：

  • 从用户角度看的精确一次执行语义（零重复扣款）。
  • 可接受的执行延迟：必须在计划时间几秒内触发，还是几分钟内，还是一天内？
  • 吞吐量：每秒峰值计划支付数。
  • 持久性：计划支付必须在任意单个组件故障后存活。
  • 保留期：历史支付记录保留多久（监管最低要求为数年）。

以下内容超出范围，除非被特别询问：

  • 欺诈检测（独立系统提供信号输入）。
  • 货币转换（独立外汇系统）。
  • 税务计算（税务引擎）。
  • 结算与商户打款（扣款的下游流程）。

规模假设

领域 假设
待执行的计划支付 在任意时刻有 1 亿笔（订阅、发票、BNPL 分期）。
峰值每日执行量 每天 1000 万笔，集中在每小时热点（午夜、月末）。
每秒峰值 在每月 1 日午夜峰值期间达 1k 次执行/秒。
最大延迟 最长 1 年后（年度订阅自动续费）。
最小延迟 1 秒（延迟确认流程）。
取消窗口 直到计划执行时间为止，保留一小段截止时间。
银行卡授权窗口 7 天（Visa/Mastercard 默认值）——驱动重新授权策略。
软拒绝重试策略 72 小时内最多重试 3 次。
可接受的执行漂移 p99 在计划时间 60 秒以内。

瓶颈不在于执行时的吞吐量，而在于状态机在漫长时间跨度内的正确性——期间任何事物都可能失败或改变。

核心产品模型

界面 用户体验 系统需求
调度 用户选择未来日期；收到支付已计划的确认。 幂等的调度 API；立即持久化存储。
取消 用户在执行前取消；查看状态更新。 取消与执行器之间的竞态必须由状态机解决，而非靠运气。
执行 资金在正确时刻转移，且仅转移一次。 调度器 + 执行器 + 支付处理器必须协同工作，不得重复触发或静默丢失。
失败 卡片被拒；通知用户；清晰的后续操作 UX。 重试策略；终态失败状态；附带原因的用户通知。
商户 通过 webhook 获知支付结果。 至少一次的 webhook 投递；商户必须幂等地消费。

将基础设施与业务逻辑分离

在绘制架构图之前，先划清以下界限： 由基础设施提供的内容 。 支付团队实际构建的内容。这是该问题中最重要的框架定位——遗漏这一点的 staff 级候选人最终会在支付服务内部重新发明一套工作流引擎。

关注点 归属于 原因
"在时间 T 执行此函数，支持故障恢复，重启后重放，支持取消信号，在步骤间持久化其状态" 基础设施 （持久执行平台：Temporal、AWS Step Functions、Cadence、Airflow、Netflix Conductor） 这是一个通用问题。公司中每个需要调度未来任务的团队都有这个需求。一次解决，复用即可。
"当卡片被拒绝并返回代码 51（余额不足）时，安排 24 小时后重试；当被代码 14（无效卡片）拒绝时，不重试" 业务逻辑 （支付服务） 卡网络响应码、软性拒绝与硬性拒绝的分类、重试预算——这些都是支付领域的知识。任何基础设施平台都不了解这些，也不应该了解。
"持久化工作流的中间状态，使其在 worker 崩溃后能够恢复" 基础设施 通用持久性。
"持久化带有外键指向支付方式及幂等键约束的 Payment Intent 状态机" 业务逻辑 领域 schema。
"对失败步骤进行指数退避重试" 基础设施 通用重试原语——适用于任何有副作用的步骤。
"决定是否应该重试 这个特定的处理器响应 根本" 业务逻辑 只有支付服务才能理解 Stripe 响应码的含义。
"向下游消费者至少一次地投递消息" 基础设施 （队列、事件总线） 通用。
"每日对账我们对支付结果的本地视图与处理器的视图" 业务逻辑 领域特定的对账规则。

清晰的职责划分： 支付服务负责状态机及每次状态转换的语义；基础设施平台负责"稍后触发函数、支持故障恢复、支持取消信号"。 如果你发现自己在支付服务内设计了一个名为 scheduled_jobs 加上 next_run_at 的数据库列，说明你已经越界——该表属于基础设施层（或者你正在选择自行构建基础设施层，因为你没有现成的）。

这一框架定位有两方面的意义。第一，它改变了"如何处理重试/崩溃/时间偏移"的答案：正确答案是"由工作流引擎处理"加上上层的领域特定决策。第二，它改变了运维叙事：你不需要值守一个自定义调度器；平台团队负责 Temporal 的值守，而你负责状态机的值守。

工作流引擎是架构图上唯一的基础设施组件；其余一切都属于支付服务。Schedule API 是一个轻量的入口——它持久化意图，然后将控制权移交给工作流引擎，由后者负责"在时间 T 触发 activity、对瞬时失败进行重试、投递取消信号"。 ExecutePaymentActivity 是系统中唯一与支付处理器通信的代码；它对 Intent DB 行执行 CAS 状态转换，使用正确的幂等键向处理器发起调用，并返回一个领域类型化的裁决，工作流引擎据此决定下一步。来自异步支付通道（如 ACH）的 Webhook 通过 Webhook Receiver 回传，后者向正在运行的工作流发送信号，而非直接操作 Intent DB——从而使工作流引擎始终作为状态转换的唯一协调者。对账任务置于旁侧，因为它不参与实时请求路径；它每日批量与处理器进行核对，并将修正写回 Intent DB。

值得大声说出来的边界： 工作流引擎是由基础设施提供的，而非支付服务自行构建的。 "休眠至时间 T、支持崩溃恢复、失败时重试、投递取消信号"是一个通用问题；它属于公司共享的 Temporal / Step Functions / Cadence 集群。支付服务只需编写一个 PaymentWorkflow 定义和一个 activity；该框内的其余一切都是平台团队的职责。如果你的公司没有这样的平台，这个框就需要自行构建——但那是独立于本次设计讨论的另一个设计练习（参见分布式作业调度器文档).

值得特别指出的三条反馈路径：

  • (a) 初始调度路径 — Client → Schedule API → Intent DB + start workflow。在 SCHEDULED中创建一条 intent 行；以 intent_id 为键启动一个工作流实例。
  • (b) 重试循环 —— activity 返回的软性拒绝会触发工作流执行 sleep_until(now + backoff) 并重新调用该 activity。Intent DB 行在两次尝试之间保持 RETRY_PENDING 状态。重试由工作流管理；领域决策（是否重试以及重试多久）由 activity 作出，并通过 activity 的返回值传递。
  • (c) 周期循环 —— 对于周期性计划的 intent，在 CAPTURED 时工作流会为下一个周期启动一个 新的 工作流实例（或在 Temporal 的术语中调用 continue_as_new）。activity 为下一周期插入新的 intent 行，原工作流终止。

右侧的 webhook 路径是第四条反馈通道：对于 ACH 等异步结算通道，同步处理器响应仅表示"已提交"。Webhook Receiver 随后通过信号（或启动一个结果更新工作流）调用正在运行的工作流，将状态机推进至终态。

核心组件

按平面分组以保持边界清晰。

业务逻辑（支付团队负责）：

  • Schedule API —— 接受带幂等键的请求，在 SCHEDULED，为每个 intent 启动一个工作流实例。
  • Payment Intent Database —— 每笔支付对应一条持久行：状态、金额、支付方式、计划时间、尝试次数、幂等键、最后错误。这是 领域的真相来源；工作流引擎有其自身的 工作流执行状态.
  • ExecutePaymentActivity —— 有副作用的原子单元：对 intent 状态执行 CAS 转换，使用正确的幂等键调用处理器，对响应进行分类，返回裁决（captured, soft_decline_retryable, hard_decline_terminal).
  • Webhook Receiver —— 接收异步处理器事件，按事件 ID 去重，向相应工作流发送信号以推进状态机。
  • Outcome Updater —— 状态机的合规护栏（合法转换规则），由 activity 和 webhook receiver 共同调用。纯业务逻辑。
  • Reconciliation Job —— 每日运行，将 Intent DB 与处理器视图进行差异比对并解决偏差。纯领域逻辑；没有对应的基础设施等价物，因为只有支付服务才知道"一致"的含义。
  • Notifier —— 向平台事件总线发布领域事件（PaymentCaptured, PaymentFailed）；消费者负责处理邮件/推送/商户 webhook。

基础设施（平台团队负责；支付服务使用）：

  • 持久执行平台 （Temporal、AWS Step Functions、Cadence、Conductor）—— 运行 PaymentWorkflow 实例。提供： sleep_until(T) 在 worker 重启后依然存活的持久状态、带重试的 activity 调用、通过信号实现的取消、通过幂等 token 实现的恰好一次 activity 执行、持久化工作流状态。
  • 数据库 —— Intent DB 本身运行在托管数据库上（Postgres、Spanner、CockroachDB）。支付服务负责 schema，而非数据库运维。
  • 事件总线 / 队列 —— 领域事件的至少一次投递；由下游 Notifier 和 webhook 订阅者消费。
  • 幂等 Webhook 摄取 —— 通用 webhook 接收基础设施，负责接收、重试并持久化任意服务的入站 webhook。支付服务接入领域特定的事件处理器。

如果你的公司没有持久执行平台，你需要自行构建——那是独立的设计练习（参见 设计类似 cron 的分布式任务调度器）。关键在于，这是一个与"设计延迟支付系统" 独立 独立的设计练习。将两者混为一谈是该问题中最常见的单一错误。

具体存储、键与容量估算

存储被有意拆分为 领域真相 。 工作流执行状态。它们通过 intent_id关联，但并非同一张表。

存储 键 / 分片 值 TTL / 保留期 容量估算
Payment Intent DB 主键 intent_id；按 hash(intent_id) Intent 行：金额、支付方式 token、计划时间、状态、尝试次数、最后错误、处理器 charge id。约 1.2 KB/行。 活跃 intent 在线保留；终态行热数据保留 90 天，之后归档多年。 1 亿活跃行 × 1.2 KB × 3 副本 × 2 区域 ≈ 720 GB。历史行占主导；1000 万/天 × 365 × 5 年 × 1.2 KB ≈ 22 TB 主数据（不含副本），因此需要积极归档。
Intent 二级索引 (state, schedule_at) 用于兜底扫描， (user_id, state) 用于用户历史，唯一 (user_id, idempotency_key) 索引条目保存行指针和少量状态字段，不含完整 intent 数据。约 40–80 字节/条目。 与热 intent 窗口相同。 1 亿活跃行 × 3 个索引 × 64 B × 3 副本 ≈ 58 GB。索引相比历史审计数据而言体量很小。
工作流引擎状态 PaymentWorkflow:{intent_id}；引擎按工作流 ID 和定时器桶分片 序列化后的工作流调用栈、下一个定时器、重试策略、取消信号状态。约 1–3 KB。 终态后 30 天删除或压缩。 1 亿活跃 × 2 KB × 3 副本 ≈ 600 GB 在工作流平台中。这正是应由平台而非支付服务来运维它的原因。
支付尝试 / 审计日志 追加于 (intent_id, attempt_number)；按 created_at 处理器请求/响应摘要、转换审计、操作者、原因。约 800 B/次尝试。 热数据 90 天；冷数据 5–7 年用于审计。 1000 万次执行/天 × 1.1 次尝试 × 800 B ≈ 8.8 GB/天主数据；加上 3 副本 ≈ 26 GB/天。多年下来这远超活跃 intent 的存储量。
API 幂等表 idem:{user_id}:{client_key} → {intent_id, request_hash, status} 小行，约 300 B。 保留至 max(schedule_at) + 30 days；此处上限为 13 个月。 若保留 1 亿活跃请求：1 亿 × 300 B × 3 副本 ≈ 90 GB。
Webhook 去重 processor_event:{event_id} → 终态结果指针 事件 ID、charge id、接收时间戳。约 150 B。 90 天足以覆盖处理器重试窗口。 1000 万/天 × 90 × 150 B × 3 副本 ≈ 405 GB。
Outbox outbox_id 按提交时间戳排序 领域事件负载指针，非完整的邮件/webhook 正文。约 500 B。 投递后 7–30 天。 1000 万终态事件/天 × 500 B × 30 × 3 ≈ 450 GB。

技术选型： 在历史数据和多区域需求占主导之前，Postgres 可满足需求；之后使用 Spanner 或 CockroachDB 以实现跨区域的强一致性状态转换。我不会为 Payment Intent 真相来源选择 Cassandra，因为我需要 CAS 状态转换、幂等唯一性约束和事务性 outbox 语义，这些比原始写吞吐量更重要。Cassandra 可以存储审计日志，但不应掌管资金状态机。

准备好应对"为何不直接用 Kafka？"这一问题。Kafka 是事件日志和重试传输通道，而非判断支付是否 CAPTURED 或 CANCELLED。合法的状态转换必须存在于事务性行中。

状态：

  • SCHEDULED —— 已提交，将在时间 T 触发。可取消。
  • EXECUTING —— 正在与处理器通信。不可取消；取消请求已入队，待处理器调用以某种方式完成后再执行。（这是关键的竞态解决规则。）
  • RETRY_PENDING —— 软性拒绝（余额不足、网络抖动）；已安排重试。
  • CAPTURED —— 终态成功。
  • FAILED —— 终态失败（硬性拒绝、重试预算耗尽、支付方式过期）。通知用户。
  • CANCELLED —— 终态取消。

终态永不再转换。每次状态变更都写入一条审计行。

端到端幂等性

三层幂等键相互叠加；缺少任何一层都可能导致重复扣款。

层级 键 防护内容
Schedule API 客户端提供的幂等键 两个相同的调度请求不会创建两个 intent。
执行器 → 处理器 intent_id:attempt_n 如果执行器内部重试，同一次执行尝试不会扣款两次。
Webhook 处理 处理器事件 ID 同一处理器 webhook 的两次投递不会导致状态被双重更新。

状态机本身是最深层的幂等保障：尝试将 CAPTURED → CAPTURED 转换是空操作而非报错；尝试将 CANCELLED → EXECUTING 转换会被拒绝。

调度策略

默认立场：不要自行构建调度器。使用持久执行平台。

"在时间 T 触发函数、支持故障恢复、支持取消"是通用基础设施问题。现代技术栈将其作为服务提供：

平台 提供能力
Temporal / Cadence 以代码形式编写的工作流，具备持久状态， sleep_until(T) 可在重启后存活，支持取消信号，自动重试 activity。
AWS Step Functions 声明式状态机，带有 Wait 状态、集成的重试/捕获、通过任务 token 实现的信号。
Netflix Conductor 以 JSON 定义的工作流，具备类似的原语。
Apache Airflow 为批处理而生；对于一次性延迟不够理想，但适用于 cron 风格的周期性调度。

一个 PaymentWorkflow(intent_id) 为以上任意平台编写的代码都可以简化为几行：休眠至 schedule_at，调用 ExecutePaymentActivity，软性拒绝时再次休眠并重试，终态时发出领域事件。平台负责持久性、重放和 worker 亲和性。

请清晰地阐明默认立场： "我不会自己构建调度器。公司有 Temporal（或 AWS Step Functions）；我编写一个 PaymentWorkflow 由平台负责定时触发、崩溃恢复和信号投递。"

回退方案：构建一个时间排序扫描器（Time-Sorted Sweeper）。

如果公司没有持久执行平台——小公司、全新项目，或平台团队明确将此推给你——那么你在构建基础设施。要对这种范围扩展保持清醒认识。

在这种情况下，最低成本且可信的方案是在 Intent DB 之上构建一个时间排序扫描器：

  • 一列 schedule_at 在 intent 行上。
  • 一个周期性 worker 扫描 WHERE state IN ('SCHEDULED','RETRY_PENDING') AND schedule_at <= now() LIMIT N 并通过队列将每行分发给执行 worker。
  • 按 hash(intent_id) % N对扫描器进行分片；每个 worker 在租约下拥有一个分区（参见 Chubby 文档），以防两个扫描器重复触发。
  • 对于亚分钟级精度，添加一个内存定时轮，从数据库中加载近期触发任务。

扫描器作为临时方案是可以的，但要认清其权衡：你现在在运营一个工作流引擎。最终你将需要取消信号、部分执行的崩溃恢复、指数退避重试，以及"工作流 X 卡在哪里"的可观测性——最终你会拙劣地重新发明 Temporal。在面试中明确指出："我在这里构建的是一个最小化工作流引擎。如果系统增长，我会迁移到真正的平台。"

其他选项及不适用的场景：

  • 延迟队列（Amazon SQS、RabbitMQ delayed exchange）。 上限为 15 分钟（SQS），或存在顺序/取消的缺口。适合短暂的即发即忘延迟；不适合延迟可达数月且取消必须可靠的定时付款。
  • 分布式 cron（Quartz、Chronos）。 为挂钟时间的周期性调度而设计，对于具有每条 intent 时间戳的一次性延迟处理起来很别扭。

授权窗口问题

卡网络（Visa、Mastercard）的授权有效期约为 7 天。如果用户将付款调度在 30 天后，而你在调度时就授权了卡，到执行时授权已过期。三种实际策略：

策略 工作方式 权衡
仅在执行时授权 调度时跳过授权；在 T 时刻进行授权+扣款。 简单。风险：在 T 时刻，卡可能被拒；用户事后收到通知。
调度时授权，执行时扣款 立即对卡进行预授权，在 T 时刻扣款。 仅当延迟 < 授权窗口（卡约为 7 天）时有效。
执行前重新授权 在 T - X 分钟时，系统重新授权卡以验证其仍然有效；如有效则扣款。 更早发现过期卡；增加一次额外的处理器调用。适合高价值定时付款。

最简洁的答案： default to authorize-at-execution-time 对超出授权窗口的延迟；叠加 re-auth-at-T-minus-X 用于高价值付款（预检验证重要的场景）。明确说明权衡：代码路径更简单，但失败只在执行时才暴露。

对于 ACH（较慢的支付通道，无授权概念），没有授权与扣款的区别；你在 T 时刻发起，数天后通过 webhook 得知结果。

重试策略

沿同样的基础设施与业务逻辑分界线清晰地拆分。

基础设施层提供重试机制：在 now + backoff 调度下一次尝试（带抖动），在尝试间崩溃后存活，持久化尝试计数，暴露进行中状态。在工作流引擎中，这只需在 activity 上进行几行配置。

业务逻辑决定是否重试以及按什么计划重试：哪些处理器响应码可重试，尝试间延迟应为多少（24 小时而非秒级，因为我们在等待用户银行账户余额刷新），总重试预算是多少，何时放弃。

activity 返回一个领域类型化的结果：

activity result -> workflow action
  CAPTURED                  -> terminal success, emit event
  HARD_DECLINE (code 14)    -> terminal failure, emit event
  SOFT_DECLINE (code 51)    -> if attempts < 3: sleep 24h ±10% jitter, retry
                               else: terminal failure
  PROCESSOR_TIMEOUT         -> query processor by idempotency key to learn outcome,
                               then map to one of the above

软拒绝的默认计划：第 1 次尝试在 T，第 2 次在 T+24h，第 3 次在 T+72h。抖动 ±10%，使来自众多用户的重试不会集中在同一秒。这些值是领域选择（银行隔夜重新提交透支，因此 24h 有意义）；休眠和重新调用的 机制 是平台的职责。

重试计数器存在于 Payment Intent 行（领域状态），也由工作流引擎（执行状态）跟踪。它们保持一致，因为每次尝试都通过 activity，activity 在调用处理器之前将计数器写回 Intent 行。处理器幂等键是 intent_id:attempt_n 使处理器能够区分各次尝试。

为什么不直接使用平台内置的重试策略？因为 Temporal 的默认指数退避（秒到分钟）对于卡片来说是错误的——你不希望在 10 分钟内对一张被拒的卡发起 30 次请求。你对该 activity 关闭平台的重试策略，让工作流本身在尝试之间休眠具有领域意义的时长。重试策略在业务代码中 表达 ；重试 机器 是平台。

取消竞争

设计中最难的正确性问题。用户取消的同时调度器触发。系统必须避免以下可能结果：

  • 用户已取消，付款仍然执行（从用户角度看是重复扣款）。
  • 取消已被接受，付款也被处理；需要退款（运营混乱）。
  • 付款已处理，取消也被记录；状态机处于无效状态。

解决规则： 取消是一个异步请求，只在安全状态转换时生效.

Cancel request arrives:
  if state == SCHEDULED:
    atomic CAS (compare-and-swap): SCHEDULED -> CANCELLED. Done.
  if state == EXECUTING:
    write cancel_requested = true. Do NOT abort the in-flight processor call.
    When executor finishes:
      if processor returned CAPTURED:
        state -> CAPTURED, then auto-issue a refund (separate operation)
        notify user: "payment processed but auto-refunded due to late cancel"
      if processor returned FAILED:
        state -> CANCELLED (the cancel won the race)
  if state in {CAPTURED, FAILED, CANCELLED}:
    cancellation no-op (already terminal)

更深层的观点： 永远不要中断对支付处理器的进行中调用。处理器要么提交要么失败；你得知结果并进行对账。中断它（例如在调用过程中杀死执行器进程）会产生孤儿扣款，处理器已扣款但本地状态认为没有。对账过程则需要通过轮询来找到这些记录。

产品层面应设置"取消截止时间" T - X 分钟，使用户不会遭遇这种竞争。在截止时间之后，系统处理；在截止时间之前，UI 显示"您在接下来 5 分钟内无法取消。"

对账

现实总比设计更复杂。对账过程每日运行（对高吞吐量系统则每小时）：

  1. 拉取本地状态显示为终态但处理器显示为不同状态的所有 intent，反之亦然。
  2. 对每个分歧进行分类：我们是否错过了 webhook？处理器数据是否过期？是否存在 bug？
  3. 将两侧驱动至一致；如果解决方案需要人工覆盖，则发出告警。

漂移的典型来源：

  • webhook 在传输中丢失（处理器发送三次重试；如果全部失败，我们已失步）。
  • 本地执行器在调用中途崩溃；处理器已扣款；我们从未得知。
  • 处理器通过其仪表板记录了人工退款；我们从未得知。
  • webhook 到达与我们自己处理处理器响应之间的竞争。

对账作业才是使系统真正正确的关键，而非状态机本身。明确提到它；这是区分教科书设计与生产设计的所在。

分区：按 hash(intent_id) 对 intent 表分区。审计日志可按 created_at （时间序列）分区。

API

POST /payments/schedule
  Idempotency-Key: <client-generated UUID>
  body: {
    amount_cents, currency, payment_method_id, schedule_at, metadata
  }
  → 201 { intent_id, state: "SCHEDULED" }

GET /payments/{intent_id}
  → 200 { intent_id, state, attempts, last_error, ... }

POST /payments/{intent_id}/cancel
  → 200 { state: "CANCELLED" | "EXECUTING with cancel_requested" }
  → 409 if terminal state already reached

POST /webhooks/processor (internal endpoint)
  body: { event_id, charge_id, status, ... }
  → 200; deduplicated by event_id

逐步 API 路径

调度路径——客户端承诺一笔未来付款。

  1. 客户端调用 POST /payments/schedule 与 Idempotency-Key ，并经过认证的 user_id.
  2. 调度 API 验证金额、货币、支付方式令牌，以及 schedule_at 在允许的未来窗口内。
  3. 它在分片 hash(intent_id)上启动一个数据库事务：插入 PaymentIntent(state=SCHEDULED)，插入幂等行，插入一条 outbox 事件 PaymentScheduled.
  4. 它在持久执行平台上以 PaymentWorkflow:{intent_id} 启动 {intent_id, schedule_at}。如果工作流启动在 DB 提交后失败，outbox 修复 worker 从持久行中启动它。
  5. 它返回 {intent_id, state}。长延迟时不调用处理器；除非延迟在授权窗口内且产品明确需要预授权，否则在执行时授权。

执行路径——工作流在 T 时刻触发。

  1. 工作流定时器触发并调用 ExecutePaymentActivity(intent_id).
  2. Activity 从分片 PaymentIntent 读取 hash(intent_id) ，并 CAS SCHEDULED -> EXECUTING 或 RETRY_PENDING -> EXECUTING。如果 CAS 失败，重复的 activity 退出。
  3. 并行地，它加载支付方式令牌和风险标志；在状态转换持久化之前不调用处理器。
  4. 它以幂等键调用处理器 intent_id:attempt_number.
  5. 成功转换到 CAPTURED，硬拒绝转换到 FAILED，软拒绝转换到 RETRY_PENDING 与 next_retry_at，超时则先按幂等键查询处理器，再选择转换。
  6. 终态转换插入 outbox 事件；通知器和商户 webhook 消费者至少投递一次，并按 (intent_id, event_type).

取消路径——用户与执行器竞争。

  1. 客户端调用 POST /payments/{intent_id}/cancel.
  2. API 路由到分片 hash(intent_id) ，并在 CAS 下读取当前状态。
  3. 如果 SCHEDULED，它转换到 CANCELLED，向 PaymentWorkflow:{intent_id} 发出信号以停止，并发出 PaymentCancelled.
  4. 如果 EXECUTING，它设置 cancel_requested=true；工作流接收到信号，但处理器调用不被中断。
  5. 如果为终态，返回终态。游标或重试状态不由客户端拥有；服务器行是权威来源。

Webhook 路径——外部处理器稍后同步。

  1. 处理器发送 event_id, charge_id、状态和签名。
  2. Webhook 接收器验证签名，插入 processor_event:{event_id} （TTL 90 天），并丢弃重复项。
  3. 它将 charge_id 或处理器幂等键映射回 intent_id 并向工作流发出信号。
  4. 工作流应用与 activity 相同的合法转换表。Webhook 永远不绕过状态机。

面试用语：

有两个持久状态来源：用于领域真相的支付 intent 行，以及用于定时器和重试的工作流状态。它们之间唯一安全的桥梁是 intent_id、CAS 转换以及每个副作用边界上的幂等键。

扩展与分区

  • Schedule API：无状态；在负载均衡器后水平扩展。
  • Payment Intent Database：按 hash(intent_id)分区。热用户（调度数千笔付款的人）很少见；如果发生，按 user_id 子分片。
  • 工作流引擎/调度器：工作流 id PaymentWorkflow:{intent_id} 按 hash(intent_id) 和定时器桶分发定时器。支付团队消费此平台；它不拥有定时器表。
  • 回退扫描器：仅在没有工作流平台时使用。按 intent 分片分区；每个分片 worker 使用租约扫描自己的切片。
  • 执行 Worker：无状态；水平扩展。瓶颈是处理器速率限制；与处理器协商每秒上限，超出时退避。
  • Notifier：标准扇出队列。

每月 1 日午夜的流量峰值对订阅计费工作负载来说是真实存在的。预热执行器容量；在调度时为每个用户添加小的抖动（例如 0-30 分钟偏移）来分散调度时间。用户同意的是"在 1 日计费"，而非"在 1 日 00:00:00 计费"。

一致性模型

  • Payment Intent 数据库内部：状态转换使用严格可序列化事务。使用行级锁或乐观 CAS。
  • Payment Intent 数据库与支付处理器之间：最终一致，通过 webhook 加每日对账来协调。
  • Payment Intent 数据库与通知器之间：通过 outbox 模式实现至少一次投递。通知器消费者是幂等的（按 (intent_id, event_type)).

强一致性用于重要的地方（你自己的状态机）；最终一致性用于你别无选择的地方（外部处理器）。

故障模式与缓解措施

失败 影响 缓解措施
调度器宕机 付款未触发 多个调度器 worker；基于租约的所有权；在 count(intents WHERE schedule_at < now() AND state == SCHEDULED).
执行器在调用中途崩溃 Intent 卡在 EXECUTING 幂等键加对账；EXECUTING 超时（例如 5 分钟）将其翻转为 RETRY_PENDING。
处理器返回超时 结果未知 视为软错误；切勿盲目重试。查询处理器（GET /charges/{idempotency_key}）以了解实际结果，再决定下一步操作。
Webhook 丢失 本地状态过期 对账过程捕获；处理器自身的重试策略覆盖瞬时丢失。
Webhook 重复投递 双重状态更新风险 Webhook 接收器按 event_id 去重。
调度与执行之间卡片过期 T 时刻硬拒绝 向用户发送可操作的消息（"更新支付方式"）；FAILED 状态；对于周期性付款，暂停订阅。
调度器 worker 时钟偏差 付款提前或延迟触发 使用 NTP 校准时钟；扫描器依赖数据库 now()，而非 worker 本地时间。
大规模定时峰值（所有人在 00:00 被计费） 执行器过载，处理器速率限制 用每条 intent 的抖动分散调度时间；对执行器限速；队列溢出时施加背压。

优雅降级状态表

系统应通过转入更慢、更明确的状态来降级，而不是对资金流动进行猜测。

阶段 模式 数据真实来源 用户可见延迟预算 回退
正常调度 工作流定时器负责唤醒 Intent 行 + 工作流状态 在 schedule_at 无。
工作流定时器延迟 已过期但未执行 Intent 行 state=SCHEDULED、工作流指标 告警前最多等待 5 分钟 修复 worker 扫描 (state, schedule_at) 索引并启动/向缺失工作流发出信号。
处理器超时 结果未知 按幂等键查询处理器 用户看到"处理中"数分钟，而非成功/失败 不要盲目重试；先对账处理器结果，再进行状态转换。
Webhook 缺失 异步通道本地状态过期 处理器 API + 对账作业 ACH 类通道需数小时；卡片授权/扣款需数秒 对近期意图进行每小时对账；每日全量差异比对。
重试预算已激活 软拒绝恢复 Intent 行 RETRY_PENDING(next_retry_at) 24-72小时，具体取决于策略 首次失败后通知用户；达到有限次尝试后停止。
数据库主节点故障转移 状态机不可用 复制 DB 仲裁 在秒级到分钟级故障转移期间，API 对 schedule/cancel 请求返回可重试的 503 没有成功的 CAS 就不调用处理器。

面试用语：

当资金结果未知时，系统必须选择一个可见的待处理状态并进行对账；不应凭空捏造成功状态，也不应仅因为 worker 超时就重试处理器调用。

运维指标

告警指标：

  • 意图计数（状态为 schedule_at < now() - 60s AND state == SCHEDULED （sweeper 处理落后）。
  • 意图在 EXECUTING 状态停留超过 5 分钟的计数（执行器挂起）。
  • 每处理器每小时的硬拒绝率（处理器性能下降）。
  • 软拒绝率（欺诈或处理器问题的信号）。
  • 每日对账差异数。
  • Webhook 投递延迟 p99。

记录但不告警：

  • 取消竞争发生次数（罕见，但每次都是用户体验教育时机）。
  • 因延迟取消触发的自动退款数量。
  • 各支付方式的成功率。

多区域策略

大多数支付系统采用单区域（或主备）架构，原因如下：

  • 支付处理器通常按区域或国家划分。
  • 监管规定（支付卡行业数据安全标准 — PCI-DSS，通用数据保护条例 — GDPR）限制持卡人数据的跨区域复制。
  • 吞吐量完全在单个区域的容量范围内。

默认方案：单区域主节点加另一区域热备；故障转移需要同步支付意图数据库。仅当产品真正跨多个区域且各区域有独立支付处理器时，才考虑使用全球分布式数据库（Spanner、CockroachDB）。

需要指出的权衡

  • 调度时授权 vs 执行时授权：调度时授权可提前发现无效卡，但任何超过 7 天的延迟都会导致授权过期。执行时授权更简单，但失败只会在 T 时刻才暴露。
  • 严格调度截止时间 vs 延迟取消加自动退款：截止时间方案 UX 更简单；自动退款方案以更高运营成本换取用户信任。
  • 单处理器 vs 多处理器：单处理器更简单；多处理器提供故障回退和价格套利，但增加了每个处理器的对账复杂度。
  • 同步 Schedule API vs 异步：同步对客户端更简单；异步通过队列吸收峰值流量，但使 UX 更复杂。
  • 慷慨重试 vs 谨慎重试：慷慨重试会掩盖瞬时问题；谨慎重试尊重用户体验（一张卡连续被拒三次会让用户不满）。按处理器响应码制定策略是正确答案。

常见陷阱与更优做法

  • 忘记授权窗口。强候选人会立即提到 7 天卡授权限制。
  • 并非每层都做幂等。无论是 schedule 调用、执行器到处理器的调用，还是 webhook 处理，都可能成为重复来源。
  • 取消时中断进行中的处理器调用。会产生孤儿请求。
  • 将 webhook 视为权威来源。Webhook 是至少一次投递，可能会丢失；对账才是保证系统正确性的手段。
  • 忽略午夜峰值。定时支付会集中触发；需要将其分散开。
  • 跳过对账。每位资深面试官都会追问；请在他们开口之前主动提及。
  • 承诺精确时间执行。处理器和网络存在抖动；承诺"p99 在 60 秒以内"，而非"精确到秒"。

面试官可能深入探讨的方向

  • 先买后付（BNPL）：如何对分期付款建模（多个意图关联同一笔购买），某期失败时如何处理，如何终止整个计划 vs 重试失败的那期。
  • 周期性订阅：如何建模"每月 5 日"；当 5 日不存在时如何处理（二月）；计划变更时的按比例计费。
  • 软拒绝重试策略：哪些处理器响应码可重试，带抖动的指数退避，何时标记为 FAILED。
  • 催收（Dunning）：周期性支付失败时的邮件/通知策略——如何在减少打扰客户与实际重试扣款之间取得平衡。
  • 多币种：在调度时还是执行时锁定汇率？权衡：费率确定性 vs 商户承受的外汇（FX）风险。
  • 合规性：PCI-DSS 范围界定如何影响架构？通过令牌化支付方式确保卡号永远不经过服务器。
  • 大规模分布式调度：sweeper 如何处理数亿条意图；亚秒级粒度；跨分片再平衡。

  • 明确不变量：每笔定时支付有且仅有一个终态结果。
  • 设计状态机并命名合法转换。
  • 识别三个幂等层（schedule、execute、webhook）。
  • 选择调度策略并说明理由（默认：时间排序 sweeper）。
  • 选择调度策略并说明理由（默认：持久化工作流引擎；回退方案：时间排序 sweeper）。
  • 解决授权窗口问题（7 天卡授权），并选择授权策略。
  • 设计重试策略（软拒绝 vs 硬拒绝、带抖动的退避、重试预算）。
  • 解决取消竞争问题（绝不中断进行中的处理器调用）。
  • 包含对账流程并说明其必要性。
  • 运营指标与具体告警阈值。

面试用语

"状态机就是设计本身。其他一切——调度器、执行器、webhook、对账——都是保持状态机正确运转的管道。最难的正确性问题是取消竞争，我通过绝不中断进行中的处理器调用、将取消视为异步请求并在安全转换时生效来解决。最难的实际问题是 7 天卡授权窗口，这迫使我要么在执行时授权，要么在捕获前重新授权。"

2. 模拟面试

面试官： 设计一个延迟支付系统。

候选人： 我将把它设计为一个带有严格状态机的支付意图，以持久化数据库为支撑，配合一个在预定执行时间触发的调度器，以及一个以幂等方式调用支付处理器的执行器。关键决策围绕调度时间与执行时间之间的间隔展开——授权保留会过期、卡会过期、用户可能取消、处理器可能失败。我将澄清需求、梳理状态机，并深入探讨重试和取消竞争问题。

面试官： 最大延迟支持多长时间？

候选人： 最长一年，用于年度订阅续费。延迟越长，调度到执行之间可能发生变化的状态就越多。具体而言，Visa 和 Mastercard 的卡授权保留约在 7 天后过期，因此对于超过一周的延迟，不能依赖预先授权——必须在执行时授权。这是影响设计的最重要约束。

面试官： 调度器如何工作？

候选人： 我默认选择持久化工作流引擎，如 Temporal 或 Step Functions，因为"睡眠到 T、在崩溃后恢复、接收取消信号、重试活动"是通用基础设施。支付服务启动 PaymentWorkflow:{intent_id} 与 schedule_at；工作流睡眠到 T 并调用 ExecutePaymentActivity。意图行仍保留 (state, schedule_at) ，以便修复 worker 可以找到缺失或逾期的工作流。如果公司没有工作流平台，则回退到基于同一索引的时间排序 sweeper，但我会指出此时是在设计调度器基础设施，而不仅仅是支付系统。

面试官： 状态机是什么？

候选人： SCHEDULED、EXECUTING、RETRY_PENDING、CAPTURED、FAILED、CANCELLED。SCHEDULED 是初始状态，可取消。EXECUTING 表示执行器正在主动调用处理器，通常意义上不可取消——取消请求会在处理器调用完成后才被响应。RETRY_PENDING 用于应该稍后重试的软拒绝。CAPTURED、FAILED、CANCELLED 是终态。非法转换在数据库层通过行约束来拒绝，而不仅仅依赖应用代码。

面试官： 带我走一遍执行时刻的流程。

候选人： sweeper 拾取一个 schedule_at 已过期的意图并分发一个执行器任务。执行器首先使用 compare-and-swap 将行从 SCHEDULED 转换到 EXECUTING——如果失败，说明另一个执行器已经持有它，当前执行器丢弃该任务。然后以 intent_id:attempt_number 作为处理器幂等键，调用处理器的授权并捕获接口。处理器返回成功、软拒绝、硬拒绝或超时。成功则转换到 CAPTURED；硬拒绝则转换到 FAILED；软拒绝则转换到 RETRY_PENDING 并安排下次重试。超时时，执行器绝不能盲目重试——它通过幂等键向处理器查询实际结果，再据此进行转换。终态转换完成后，发出通知事件。

面试官： 如果用户恰好在调度器触发时取消会怎样？

候选人： 这是取消竞争，也是最难的正确性问题。规则：取消是一个异步请求，在安全状态转换时生效。如果行处于 SCHEDULED，compare-and-swap 到 CANCELLED——完成。如果行处于 EXECUTING，在行上写入 cancel_requested = true 但绝不中断进行中的处理器调用。执行器完成后检查该标志：若处理器成功捕获，状态转到 CAPTURED，然后自动发起退款并通知用户其延迟取消已以退款形式兑现；若处理器失败，状态转到 CANCELLED，取消赢得了竞争。绝不中断处理器调用的原因是：中断会产生孤儿扣款——处理器已捕获，但本地状态没有记录。不如让其完成并对账。

面试官： 如何处理卡授权窗口？

候选人： 默认在执行时授权，而非在调度时授权。卡授权保留约 7 天，因此对于更长的延迟无法依赖预授权。代价是无效卡只有在执行时才会暴露，而非在调度时——用户直到 T 时刻才知道卡已过期。对于高价值支付，我会在 T 前 X 小时增加一个重新授权步骤，做一次小额授权来验证卡是否仍然有效，实际捕获在 T 时进行。对于 ACH，没有授权的概念；发起后通过 webhook 在数天后才能知道结果。

面试官： 重试策略？

候选人： 默认三次，在 72 小时内进行——T、T+24h、T+72h——带抖动。仅对处理器分类为软拒绝的情况重试，如余额不足或网络抖动。硬拒绝（如卡被盗）直接转到 FAILED。重试计数器存储在意图行上，每次尝试使用 intent_id:attempt_n 作为处理器幂等键，以便处理器区分各次尝试。重试预算耗尽后，转换到 FAILED 并通知用户。

面试官： 如果处理器发来的 webhook 丢失了怎么办？

候选人： 系统不能仅依赖 webhook 来保证正确性。每日对账任务拉取处于非终态的意图以及近期状态发生变化的意图，通过处理器 API 获取其视图，并解决任何不一致。Webhook 是至少一次投递，可能在传输中丢失；对账才是让系统真正正确的手段。我会在一开始就提到这个任务——这正是区分教科书设计与生产支付系统的关键。

面试官： 1 号凌晨的峰值怎么处理？

候选人： 订阅会集中触发。用户合同写的是"每月 1 日扣款"，而非"1 日 00:00:00 扣款"。因此在调度时，我为每个用户添加 0 到 30 分钟的随机抖动。峰值被分散到半小时内，执行器容量保持合理，处理器也不会对我进行频率限制。在可预见的峰值窗口前预热执行器。

面试官： 多区域？

候选人： 默认方案是单区域加另一区域热备。支付处理器通常按区域划分，且持卡人数据法规（如支付卡行业数据安全标准）限制跨区域复制。大多数产品本来就只服务一个区域的支付通道。如果产品需要真正的多区域支持——例如全球市场——我会为意图表使用全球分布式数据库，并在每个区域选择一个处理器，但运营复杂度很高，大多数产品并不需要。

面试官： 总结设计。

候选人： 支付意图采用六状态机，持久化存储在作为真相来源的持久化数据库中。持久化工作流引擎负责管理计时器、重试和取消信号；执行活动使用分层幂等键调用处理器。Webhook 加对账使本地状态与处理器的实际状态保持同步。取消绝不中断进行中的处理器调用。长延迟时授权在执行时进行。重试带抖动且有固定预算。整个设计接受这样一个事实：资金去向的真相来源是处理器，意图数据库是我们对其的最佳理解，并持续进行对账。

3. 练习备注

五分钟版本

  1. 澄清：最大延迟、支付方式、取消规则、周期性或一次性、通知渠道。
  2. 明确不变量：每笔定时支付有且仅有一个终态结果。
  3. 绘制状态机：SCHEDULED、EXECUTING、RETRY_PENDING、CAPTURED、FAILED、CANCELLED。
  4. 选择调度策略（时间排序 sweeper）并说明理由。
  5. 解决授权窗口问题：超过 7 天的延迟在执行时授权。
  6. 走通执行流程：CAS 到 EXECUTING，使用幂等键调用处理器，转换到终态。
  7. 解决取消竞争：取消是异步的，绝不中断进行中的处理器调用。
  8. 重试策略：带抖动、仅软拒绝、有限预算。
  9. 对账：每日执行，处理器是真相来源，意图 DB 是我们对其的建模。
  10. 最后总结指标（sweeper 滞后、卡在 EXECUTING 中、对账差异）。

一分钟结尾陈述

延迟支付系统是一个状态机，必须在调度和执行之间可能发生的一切情况中保持正确。状态机就是设计；其余——调度器、执行器、webhook、对账——都是保持状态机正确运转的管道。难点在于授权窗口、取消竞争和 webhook 丢失。我通过默认在执行时授权、绝不中断进行中的调用，以及每日将处理器视为实际资金流动真相来源的对账来解决这些问题。
