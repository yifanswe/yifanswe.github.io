Instagram — Photo Sharing + Feed (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Hard · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 设计 Instagram。用户上传图片和短视频，关注其他用户，并浏览来自所关注用户的帖子的排序信息流。日活跃用户（DAU）5 亿，每天新增帖子 1 亿条，平均每用户每天打开应用 8 次。媒体内容密集 + 排序计算密集。

1. 需求（约 5 分钟）

功能需求 - 发布图片/短视频。\n- 关注其他用户。\n- 浏览来自所关注用户近期帖子的排序信息流。\n- 对帖子点赞和评论。

我会推迟实现 Stories、Reels（算法推荐发现）、私信和购物功能。

非功能需求 - 信息流打开 p99 延迟 < 500 ms。\n- 媒体资源必须在全球以低延迟提供服务——由 CDN 前置。\n- 信息流和互动计数最终一致。\n- 非对称：少数账号拥有 1 亿以上粉丝（名人问题）。\n- 读多写少：读写比约 100:1。

容量估算 - 每天 1 亿条帖子；每条帖子平均 2 MB（压缩/多尺寸后）→ 每天原始数据 200 TB → 每年约 73 PB。\n- 5 亿 DAU × 8 次打开 × 20 条帖子 = 每天 800 亿次帖子浏览 → 峰值约 100 万 QPS。\n- 10 亿关注关系。\n- 主要成本是媒体：仅帖子元数据约 1 KB × 1 亿/天 × RF=3 约为 300 GB/天，而媒体在复制/分层之前约为 200 TB/天。Feed Redis 容量大但有上限：5 亿用户 × 500 个帖子 ID × 约 32 B × RF=2 约为 16 TB 加上开销。

2. 核心实体（约 2 分钟）

  • User — user_id，关注关系。
  • 帖子 — post_id, author_id, media_refs[], caption, created_at, like_count, comment_count.
  • 媒体 — media_id, url_variants{sizes}, format, owner.
  • 边缘 — (follower_id, followee_id).
  • FeedEntry — (user_id, post_id, score).

3. API（约 5 分钟）

POST  /v1/media/upload                     → { media_id, upload_url }
POST  /v1/posts                            { media_ids[], caption }   → { post_id }
GET   /v1/feed?cursor=...&limit=20         → { posts[], next_cursor }
POST  /v1/posts/{id}/like
POST  /v1/posts/{id}/comments              { text }
POST  /v1/follows                          { user_id }

两阶段上传——请求签名 URL，直接上传到对象存储，然后提交。

Feed 游标是不透明的，编码了 {user_id, rank_session_id, last_score, last_post_id}。排序会话存储候选集数分钟，以确保第 2 页不会在用户浏览时重新洗牌。

5. 高层设计（约 10–15 分钟）

  1. 上传路径。 - 客户端请求上传 URL → 媒体服务签发一个 S3 URL。\n - 客户端直接上传到 S3。\n - 提交时，媒体服务启动图片处理 pipeline：生成多种尺寸（缩略图、中等、大图），可选 HEIF→JPEG 转换，剥除 EXIF。对于视频：转码为多种码率。\n - 帖子写入帖子目录（Cassandra，按 post_id 分片），包含媒体变体的引用。发布 post.created 事件。
  2. 信息流组装。 与 FB News Feed 问题相同的混合推/拉策略：\n - 对普通用户（粉丝 ≤ 10k）进行推送扩散：在 post.created，追加 post_id 追加到每个粉丝的 Redis feed 列表中。\n - 对名人（粉丝 > 10k）在读取时拉取：组装信息流时，同时从各名人的缓存中获取其近期帖子。\n - 读取时合并并排序。
  3. 排序。 在线模型服务。特征：时间新鲜度、作者亲密度、帖子互动速度、媒体类型、内容分类。对候选集重新排序。
  4. 媒体服务。 S3 前置 CDN（CloudFront/Akamai）。按 URL 路径强制缓存；私人账号使用签名 URL。根据设备类型自适应分辨率。
  5. 互动管道。 点赞/评论 → Kafka → 计数聚合（Redis 用于实时计数，ClickHouse 用于分析）+ 排序信号更新。
  6. 存储层次结构。 近期帖子和元数据为热数据（Cassandra + Redis 缓存）。旧媒体存储在更廉价的层级（S3 Glacier 风格），但仍可通过 CDN 访问（存在缓存未命中代价）。

端到端叙述：Alice 发布一张图片 → 上传到 S3，提交触发处理 → 生成媒体变体 → 帖子行写入 Cassandra → 扩散将 post_id 推入她 200 个好友的 Redis feed 列表。Bob 打开应用 → 信息流组装拉取 Bob 的 feed 列表（300 个候选 post_id），合并 5 个名人帖子候选 → 批量获取帖子元数据 → 排序服务重新排序 → 返回 top 20。Bob 看到 Alice 的图片，由 CDN 边缘节点提供服务。

具体存储方案：

存储 我会明确说出的具体存储方案： 条目大小 TTL / 保留期 分片键
帖子目录 post:{post_id} 带有作者、媒体引用、计数器的行 ~1 KB 永久保留，旧行分层 post_id
作者时间线 author_posts:{author_id} 最近的 ZSET/列表 post_ids 约 32 B/成员 上限 1k-5k author_id

按存储容量估算：

存储 Math 主要调节项
媒体对象 100M posts/day * 2 MB ~= 200 TB/day before replicas; 30 d hot ~= 6 PB 压缩变体、分层存储、CDN 命中率
帖子目录 100M/day * 1 KB * RF=3 ~= 300 GB/day 元数据压缩和旧行分层
Feed 索引 500M users * 500 ids * 32-80 B * RF=2 ~= 16-40 TB plus Redis overhead 限制长度、TTL、活跃用户过滤
作者时间线 100M posts/day * 32 B/id * RF=2 ~= 6.4 GB/day 按作者设置上限，归档旧 ID
关注图 1B edges * 16 B * RF=3 ~= 48 GB before index overhead 邻接压缩
热计数器 3B hot posts * 20 shards * 16 B ~= 960 GB before overhead 分片数量和 30 天汇总

6. 深度探讨（约 10 分钟）

媒体处理 pipeline。 上传 → S3 → SQS 通知 → 图片处理 worker → 生成 4 种尺寸（缩略图 150²、中等 600²、大图 1080²、详情 2160²）和 CDN 预热预览图。对于视频：转码为 480p/720p/1080p H.264，生成海报图片，MP4 + HLS。Pipeline 是异步的；帖子可在完整处理前发布（在变体就绪前回退到原图）。

为什么使用混合信息流。 纯推送在名人场景下会崩溃（Kim Kardashian 发帖 → 2 亿次 Redis 写入）。纯拉取在普通用户场景下会崩溃（每次打开信息流需要扩散读取 300 个好友的近期帖子）。混合方案对 99.9% 的用户使用推送，对 0.1% 拥有庞大粉丝群的用户使用拉取。截止阈值是一个配置参数。

为什么用 Redis 存储 feed ID 而非仅用 Cassandra：Cassandra 可以存储持久的时间线，但信息流打开需要在 50 ms 内获取一个小型、热点、有序的候选集。Redis 仅存储 ID；Cassandra 作为持久的帖子目录。如果 Redis 丢失，可从作者时间线和近期 post.created 为键写入评论（每位用户对每道菜品最多只有一条评论）。发布

Feed 列表维护。 每个用户一个 Redis sorted set，以滑动 TTL 保留最近几百条记录。新增关注时，不回填历史帖子（成本太高）。相反，在新关注后的最初几天，组装时通过实时拉取来补充。

Feed 写入路径： post.created -> 扩散 worker 读取 followers:{author_id}；对于普通作者，批量执行 ZADD feed:{follower_id} score post_id 并用 ZREMRANGEBYRANK修剪；对于名人作者，仅写入 author_posts:{author_id} 并使名人缓存失效。帖子目录写入和媒体处理在扩散开始前已经提交。

Feed 读取路径：client -> feed service 并行读取 feed:{user_id} 。 following_celeb:{user_id} ；从 author_posts:{author_id}拉取近期名人帖子；并行批量获取帖子元数据、计数器、用户点赞状态和媒体变体；排序服务对候选列表重新排序；游标存储排序会话。

热信息流页面追踪：\n1. 客户端调用 GET /v1/feed?cursor=C；feed service 解码 {user_id, rank_session_id, last_score, last_post_id}。\n2. 读取 feed:{user_id} ，以 user_id 作为分片键从 Redis 中获取，并行读取 following_celeb:{user_id} 加上缓存的用户特征向量。\n3. 对于名人关注者，从 author_posts:{author_id} （按各名人 author_id为键）拉取近期 ID。\n4. 合并 300-800 个候选帖子 ID，丢弃已存储在排序会话中的 ID，并行批量获取 post:{post_id}, counter:{post_id}:*、用户点赞状态和媒体变体。\n5. 排序作为一次批量调用执行；服务返回 20 条已填充内容的帖子，并更新排序会话以保证第 2 页稳定。

冷信息流重建路径：如果 feed:{user_id} 缺失或已耗尽，服务读取 following:{user_id}，从每个普通作者的 author_posts:{author_id} 中以严格上限拉取最后 N 个 ID，与名人拉取合并，异步写入新的 feed:{user_id} ，若重建超过约 700 ms 则返回较小的页面。

为什么主要存储选用 Cassandra/DynamoDB 风格的存储更适合 post:{post_id} 相比单一关系型数据库，因为该工作负载以追加写入为主，需要全局分片，且大多数为点读取。Redis 仅用于有界 ID 索引；将完整帖子存入 Redis 会成倍增加媒体相关元数据的成本。Kafka 将上传提交与扇出解耦，避免慢速从节点分片导致发帖变慢。要做好回答"为什么不使用纯拉取模式？"的准备——纯拉取会为关注了数百个账号的普通用户带来较高的读取延迟。

排名模型服务。 将特征预取到特征存储（user_features, post_features）。在 Feed 打开时并行调用：Feed 列表拉取 + 特征拉取 + 媒体元数据拉取。单次批量推理调用。服务端总预算 < 200 ms。

CDN 策略。 采用多 CDN 应对故障。缓存键包含尺寸变体。媒体 TTL 约 1 天（使用含内容哈希的不可变 URL）。失效策略：当帖子被删除时，清除其 URL。

名人读取的规模问题。 每位名人维护"近期帖子" Redis 哈希 + 在聚合节点上对每个特征进行进程内预计算缓存（因为同一批名人会出现在数百万个 Feed 中）。每次名人发新帖时更新。

互动计数的一致性。 Redis 中对每次点赞递增的近似计数；每小时通过 ClickHouse 中 Kafka 派生的汇总进行对账。用户几乎不会注意到整点之间的轻微误差。

存储成本。 图片在 30 天后大多进入冷数据。30 天后分层至 S3 低频访问，1 年后迁至 Glacier。按需重新加载——首次命中承受延迟，后续命中由 CDN 缓存处理。

热门帖子瓶颈。 病毒式帖子 → 1 小时内 1000 万次浏览。CDN 吸收媒体读取压力；帖子元数据体量小，可在 API 网关轻松缓存。风险在于互动计数器——使用多个分片键，读取时求和。

双索引框架。 同一份帖子数据集上建立两个索引。 feed:{user_id} 以读者为键，针对普通作者进行优化； author_posts:{author_id} 以写者为键，针对名人、新关注以及 Feed 重建进行优化。读取路径合并两个视图后再排名，因此产品呈现的依然是统一的 Feed 体验。

边缘压力测试。 休眠用户可能没有预热的 Feed 列表；他们会通过基于拉取的方式从作者时间线重建，首页加载较慢。分页超出 Redis 500 ID 窗口的用户会切换到作者时间线拉取加新游标，而不是让 Redis 无限增长。名人的突发写入不会扇出到所有关注者；只更新作者时间线和名人缓存。多区域漂移对 Feed 新鲜度可以接受，但帖子创建应在作者归属区域写入目录并在跨区域排名使用前完成复制。冷媒体变体在转码 Worker 完成前会回退到原始对象。

Feed 服务状态机：

阶段 模式 数据源 游标/会话状态 延迟目标
热层 已推送 Feed feed:{user_id} + 排名会话 rank_session_id，最后得分/ID < 500 ms
混合 推送 + 名人拉取 Feed 索引 + author_posts:* 同一会话，去重集合 < 600 ms
冷层 读时重建 关注图 + 作者时间线 新会话，部分候选集 < 800 ms（首页）
已耗尽 深度翻页 按时间戳读取作者时间线 游标存储作者偏移量 速度较慢，帖子减少可接受
降级 排名下线 时近性/亲密度启发式算法 游标形状不变 < 500 ms

优雅降级。

情形 模式 数据源 用户可见行为
Feed 缓存冷启动 从作者时间线拉取并异步重建 关注图 + 作者帖子 首次打开 Feed 较慢
名人扇出 读时拉取，不写入关注者 名人近期缓存 秒级新鲜度
排名服务宕机 时近性 + 亲密度降级排名 Feed 索引 + 缓存特征 个性化程度降低
媒体变体缺失 提供低分辨率/原始变体 对象存储/CDN 质量下降，帖子仍可加载
计数器热点键 分片计数器，近似求和 Redis 分片 + 汇总 计数可能有延迟

面试要点："Feed 是同一批帖子上的两个索引：以读者为键面向普通作者，以作者为键面向名人；读取路径将二者合并为一个有排名的统一体验。"
