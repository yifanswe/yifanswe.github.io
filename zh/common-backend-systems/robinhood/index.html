Robinhood — Trading Platform (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Hard · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建 Robinhood。用户下达股票订单（市价单、限价单），查看实时报价，查看投资组合的 P&L，系统将订单路由至做市商或交易所。3000 万用户，峰值 10 万笔订单/秒，来自行情数据源的约 1 万个 ticker 的实时报价，热门 ticker 每秒数千个 tick。

1. 需求（约 5 分钟）

功能需求 - 对上市股票下达市价单和限价单。- 查看实时报价及投资组合的实时 P&L。- 查看订单历史和成交详情。

我会推迟处理期权交易、加密货币、零股深度分析以及税务批次会计。

非功能需求 - 正确性是首要目标——不丢失订单、不重复成交、不出现错误余额。- 报价到达客户端的延迟 < 200 ms；订单确认 < 500 ms；成交与结算受交易所约束。- 随时保持现金/持仓余额的强一致性。- 合规要求：对每个操作保留持久化审计日志。- 承受开市峰值：设计目标为 10 万笔订单/秒。

容量估算 - 3000 万用户；任意交易日约 100 万活跃用户；峰值每天 500–1000 万笔订单；开市时 10 万笔/秒。- 行情数据：1 万个 ticker × 平均约 10 个 tick/秒，热门 ticker 数千个/秒 → 聚合摄入约 100 万个 tick/秒。- 订单行较大但可管理：1000 万笔订单/天 × 约 1 KB × 7 年 × 3 个副本 ≈ 76 TB（不含索引）。- 若保留原始行情数据则会占据主导：100 万 tick/秒 × 约 100 B ≈ 每天 8.6 TB 原始数据，因此将热门报价保存在 Redis 中，并对 tick 历史积极压缩/归档。

2. 核心实体（约 2 分钟）

  • User / Account — user_id, cash_balance, positions。
  • Position — (user_id, ticker) → shares, avg_cost.
  • 订单 — order_id, user_id, ticker, side, qty, type, limit_price, status, created_at.
  • Execution / Fill — fill_id, order_id, qty, price, ts.
  • Quote — (ticker) → bid, ask, last, ts.

3. API（约 5 分钟）

POST  /v1/orders                       { ticker, side, qty, type, limit?, idempotency_key }   → { order_id, status }
GET   /v1/orders/{id}
DELETE /v1/orders/{id}                  // cancel
GET   /v1/portfolio                     → { cash, positions[], pnl_live }
WS    /v1/quotes                        → subscribe { tickers[] }
WS    /v1/orders                        → subscribe to user's order events

下单时必须携带幂等键。

5. 高层设计（约 10–15 分钟）

  1. Order entry service。 接收订单，验证（购买力充足、在市场交易时间内、持仓规模限制），写入订单行（status=accepted），并发布至订单路由。
  2. Pre-trade risk check。 针对用户现金和持仓的单一事务步骤：买入时 cash_balance ≥ order_value；卖出时 shares_held ≥ qty。以带有 WHERE 子句的行更新方式实现，以原子方式预留现金/股票。这是最重要的一致性边界。
  3. Order routing。 通过标准协议（FIX）将订单路由至经纪商或交易所。对于 Robinhood 而言，专门路由至做市商以获取订单流付款（PFOF）。路由响应可以是：立即成交、排队等待、部分成交或被拒绝。
  4. Execution events。 成交异步回传。每次成交更新订单（partially_filled 或 filled）并原子地调整持仓和现金。
  5. Position service。 管理持仓和现金。每个账户的单一写入者确保正确性。每次成交时，在同一事务中应用变化量。
  6. Market data pipeline。 从交易所/SIP 数据源（FIX/ITCH）摄入 → 按 ticker 发布到 Kafka topic → 报价服务在 Redis 中维护当前最优买卖价 → 扇出至客户端 WS 订阅者。
  7. Portfolio + P&L。 在读取时实时计算：现金 + Σ(position.qty × current_price)。当前价格来自报价服务。短暂缓存以减少抖动。
  8. 审计日志。 每个订单事件都追加到不可变日志（Kafka 超长保留期 + S3 冷存储）以满足合规重放需求。

端到端叙述：市场在 9:30:00 ET 开市 → Alice 下达"买入 10 股 AAPL 市价单" → 订单服务验证购买力 → 从现金中原子预留 $1750，写入订单行，发布至路由器 → 路由器向做市商发送 FIX 消息 → 200 ms 后以 $174.95 × 10 的价格成交 → 成交事件更新持仓（+10 AAPL @ $174.95）和现金（释放超额预留的 $50）。Alice 的投资组合 WS 在 1 秒内看到订单成交、持仓更新和 P&L 更新。

6. 深度探讨（约 10 分钟）

具体存储与分片键。

存储 键模式 数据结构 / 条目 TTL / 保留期 分片键
Account balance acct:{user_id} 强一致性行，现金/购买力/版本，约 1 KB 7 年以上 user_id
Positions pos:{user_id}:{ticker} 含数量/成本基础/预留数量的行，约 150 B 7 年以上 user_id
订单 orders:{user_id}:{order_id} 订单状态行，约 1 KB 7 年 user_id
Executions fills:{order_id}:{fill_id} 不可变成交行，约 500 B 7 年 order_id
幂等性 order_idem:{user_id}:{key} 响应/状态指针，约 300 B 24–72 小时 user_id
Current quote quote:{ticker} Redis hash 买价/卖价/最新价/时间戳，约 100 B TTL 5–30 s ticker
Quote subscribers subs:{ticker} 临时连接 ID，约 16 B/订阅者 连接期间 ticker
审计日志 audit/orders/yyyy/mm/dd/{partition} 仅追加事件，约 500 B 7 年以上 日期 + 分区

现金/持仓是权威内容；报价 Redis 是新鲜度缓存。若报价变旧，余额不会随之变旧。

各存储容量估算。 账户状态不是规模问题，正确性才是。 30M accounts × 1 KB × 3 replicas 仅约 90 GB，甚至 30M users × 20 positions × 150 B × 3 约 270 GB。更重的存储是历史数据： 10M orders/day × 1 KB × 7 years × 3 约 76 TB，若每笔订单平均两次成交， 20M fills/day × 500 B × 7 years × 3 还需另外约 76 TB。审计可能更大，因为每笔订单有多个生命周期事件： 10M orders/day × 6 events × 500 B × 7 years × 3 约 230 TB。若保留原始行情数据则会使上述所有容量相形见绌： 1M ticks/sec × 100 B 每天约 8.6 TB（复制前），因此我仅保持当前热门报价，保留短时间窗口的 tick 以供回放，并单独对历史 tick 进行压缩/归档。

Place-order path。 1. 客户端携带幂等键提交订单；网关进行身份验证并检查 order_idem:{user_id}:{key}。2. Order entry 加载 acct:{user_id} 以及相关的 pos:{user_id}:{ticker} ，均在同一账户分片上。3. 预交易风险以条件更新方式预留现金/股票。这是一致性边界；在预留提交之前不发送 FIX 消息。4. 订单行 orders:{user_id}:{order_id} 被写为 accepted，并通过 outbox 事件路由至 FIX 适配器。5. 路由器发送至做市商/交易所。成交异步到达并追加 fills:{order_id}:{fill_id}。6. Position service 按账户分片顺序将每次成交应用于账户现金/持仓，释放超额预留，并向用户 WS 发出事件。

Quote read path。 1. 行情数据馈送处理程序规范化 tick 并按 ticker 发布。2. 报价服务更新 quote:{ticker} 并将最新 tick 推送至区域扇出层。3. 投资组合读取并行获取持仓和当前报价，计算未实现 P&L，并将结果缓存约 1 秒。

Hot and cold API paths。

API path Hot path Cold/fallback path Server-side state
POST /v1/orders 幂等键查找 → 在 user_id 分片上进行账户/持仓条件预留 → 订单行/outbox → FIX 路由器 若幂等键已存在，返回之前的响应；若账户分片不可用，则在路由之前快速失败 order_idem:{user_id}:{key}，订单状态，预留版本

订单游标是 order_id；流式游标是每个连接的最后事件序列号，以便断线重连后可从用户事件日志中回放错过的订单事件。

Pre-trade risk — the correctness centerpiece。 所有现金和持仓预留都在每个账户的单一行上以乐观并发方式进行。买入订单：

UPDATE accounts
SET cash = cash - 1750, pending_orders = pending_orders + 1
WHERE user_id = X AND cash >= 1750;

成交时，根据实际情况释放预留。拒绝时，退还预留。

Account sharding。 按以下方式分片 user_id 这样用户的所有订单、持仓、现金都在同一分片中。避免跨分片的分布式事务。

为何选择单写入者/账户分片而不是全局分布式事务：大多数订单只涉及一个账户和一个 ticker，因此按 user_id 对现金和持仓状态进行本地化，可在更低延迟下获得更强的正确性。跨账户结算和经纪商对账异步进行；它们不应出现在订单录入的 p99 路径中。

Tech choice defense。 对于账户和持仓状态，我需要强一致性的 SQL/Spanner 风格存储，因为条件更新、唯一订单 ID 和审计联合查询都是核心需求。Cassandra 可以吸收大量写入，但轻量级事务会成为资金路径中的风险点。Redis 适合报价和临时状态，而非余额的真相来源。对于路由，持久化 outbox/Kafka 桥接优于在订单事务中直接进行同步 FIX 调用，因为经纪商超时不应抹去已接受的订单；outbox 在路由边界提供了具有恰好一次效果的重放能力。

准备好回答"为何不用一个巨大的事件日志？"——日志非常适合审计和回放，但服务路径仍然需要当前账户行，以便预交易风险检查无需扫描历史记录。

Market data fan-out at 1M ticks/sec。 单个报价服务节点无法向数百万 WS 客户端推送。金字塔架构：- 摄入层按 ticker 将 tick 写入 Kafka。- 扇出层（区域级）订阅相关 ticker，推送至 WS 网关。- WS 网关持有连接，向每个订阅者转发正确的 ticker。客户端订阅特定 ticker，而非全量数据流。

Quote update batching。 AAPL 开市期间每秒更新 1000 次。没有人需要如此高的刷新率。WS 层将每个 ticker 向客户端的推送频率限制为约 5 Hz，取最新的 tick。中途的 tick 会被丢弃——这完全可以接受。

Order routing protocol。 通过 TLS 的 FIX 协议连接经纪商/做市商。对延迟敏感：保持持久的 FIX 连接，按 ticker 类别路由。对于自清算经纪商，还需处理清算/结算（T+1 周期）。

Cancellation race。 用户在成交到达的同时取消订单。对订单行进行带状态检查的原子更新： UPDATE orders SET status='canceled' WHERE id=X AND status='accepted' ——仅在尚未成交的情况下才能成功。若成交先到，取消操作返回"为时已晚"。

Hot tickers at market open。 所有用户都在 9:30 整抢购 AAPL/TSLA。订单录入必须吸收这次峰值。缓解措施：- 预热 worker 池和 Redis 缓存。- 在路由层短暂排队订单；返回带有 order_id 的 202 Accepted，让 WS 推送最终成交。- 对于限价单，立即返回并让交易所处理排队。

P&L computation。 已实现 P&L 来自已成交的订单（确定性，税务批次感知）。未实现 P&L = 当前价格 × 股数 - 成本基础。针对用户的开仓持仓，每次报价更新时重新计算；缓存约 1 秒。

双索引框架。 订单事件是同一数据集的两种呈现方式：按用户键控的行（orders:{user_id}:{order_id}）用于应用历史记录，按时间/分区键控的审计日志用于合规回放。报价也有两种视图：按 ticker 的内容（quote:{ticker}）和按 ticker 的订阅者成员关系（subs:{ticker}）。指出这一点很重要，因为我并非在创造独立的事实来源，而是保持一份事实，构建多个服务索引。

Regulatory and audit。 每个订单事件写入仅追加的 Kafka topic，保留 7 年（合规最低要求）。对账作业每天将持仓与经纪商报表进行比对。

Outages and degradation。 若行情数据失败，在 UI 中冻结报价显示时效计时器并拒绝市价单（限价单仍然安全）。若路由器失败，将订单排队并向用户明确提示。

情形 模式 数据源 用户可见行为
Quote stale quote-safe quote:{ticker} age 显示过时徽章；拒绝市价单，允许限价单
Router down accepted-not-routed order outbox 返回 202 并明确标注待处理状态，如允许则稍后处理
Account shard degraded fail-closed for orders account DB 阻止新订单；投资组合可能显示上次缓存状态
Hot ticker open dampened fan-out latest tick per ticker 以 5 Hz 推送最新报价，丢弃中间 tick
Cancel/fill race state CAS order row + fill log 先提交者获胜；不存在模糊的取消状态

"重点总结：行情数据可以降级为显示过时数据，但订单和余额状态必须快速失败，因为资金出错比交易不可用更为严重。"

我会推迟的内容。 期权链、零股聚合、定期投资、加密货币交易轨道、税务批次会计深度处理、合规申报流水线（例如 CAT 报告）。
