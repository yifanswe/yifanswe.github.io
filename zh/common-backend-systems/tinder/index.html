Tinder — Dating App (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Medium · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建 Tinder。用户打开应用，获得一叠附近候选人资料，向左/右滑动，当两位用户互相向右滑动时即"匹配"成功并可以聊天。日活跃用户（DAU）1亿，每人每天中位数约50次滑动，每天约1000万次新匹配。

1. 需求（约 5 分钟）

功能需求 - 浏览经位置、年龄范围、性别偏好筛选后的候选人资料栈。 - 向左滑动（跳过）或向右滑动（喜欢）。 - 当双方均向右滑动时，创建匹配并开启聊天线程。

我会推迟聊天/消息实现的深度设计、付费功能（Super Like、Boost）以及资料照片审核。

非功能需求 - 卡片栈打开 p99 < 300 ms；没有什么比下一张卡片加载缓慢更影响体验了。 - 匹配检测需要强一致性——双方用户必须各自且仅看到一次匹配通知。 - 用户绝不能看到同一个候选人两次（不重复展示）。 - 读多写少：1亿 × 50 = 50亿次滑动/天 → 平均约6万次滑动/秒；候选人获取量与此大致为1:5。

容量估算 - 1亿 DAU × 50次滑动 = 50亿次滑动/天。 - 匹配表：终身约36亿条匹配记录。 - 资料存储：共2亿份资料，每份约5 KB ≈ 1 TB。 - 滑动写入量：50亿/天 × 约40 B紧凑行（from,to,dir,ts） ≈ 复制前约200 GB/天；3副本、1年保留期后约220 TB。主要的热数据成本在于写入吞吐量和每用户排除状态，而非资料存储。 - 活跃地理索引：若3000万用户在过去30分钟内活跃，每条记录约24 B，则主节点活跃位置索引 < 1 GB；数据量小，但对延迟极为敏感。

2. 核心实体（约 2 分钟）

  • 用户 / 资料 — user_id，位置、年龄、照片、偏好。
  • 滑动 — from, to, direction (like/pass), ts。仅追加。
  • 匹配 — user_a, user_b, created_at, chat_id。对称对。
  • 聊天线程 ——不在完整设计范围内；仅涉及入口点。

3. API（约 5 分钟）

GET   /v1/cards?lat=&lng=&n=10               → Profile[]    // candidate stack
POST  /v1/swipes                              { target_id, direction }  → { matched?: bool, match_id?: ... }
GET   /v1/matches                             → Match[]
POST  /v1/location                            { lat, lng }

从令牌获取用户。滑动操作按 (swiper, target).

卡片游标由服务端下发并编码 user_id, geo_cell, candidate_batch_id，以及 rank_offset；防止重试时出现重复卡片，同时不暴露排名内部逻辑。

5. 高层设计（约 10–15 分钟）

  1. 资料服务。 按以下方式分片 user_id。读多写少；前置每份资料的缓存。
  2. 位置服务。 用户上报位置 → 写入地理索引（S2 单元格；Redis 或 ElasticSearch 等支持地理感知的存储）。
  3. 候选人生成。 对于卡片请求，查询用户所在 S2 单元格（及相邻单元格）中符合年龄/性别偏好的用户。减去已滑动的用户。上限约50个候选人，按轻量质量模型排序。
  4. 滑动服务。 写入滑动行（按 swiper_id分片）。若方向为喜欢，检查 (target, swiper, like) 是否存在——如存在，则创建匹配。
  5. 匹配检测。 两条路径：(a) 滑动时同步检测——单次读取互相滑动记录；若找到，写入匹配行；(b) 若不希望在滑动路径上增加读操作，则通过 Kafka 异步处理。为获得流畅的"匹配成功！"用户体验，我默认选择同步方式。
  6. 匹配存储。 对称存储——按排序后的 (min_id, max_id) 为键存储一行，以避免重复匹配；唯一约束确保"最多匹配一次"。
  7. 聊天服务。 匹配创建后，创建一个聊天线程（仅为 chat_id）；消息不在设计范围内。

端到端叙述：Alice 打开应用 → 卡片请求 → 候选人生成找到50位附近单身用户，减去 Alice 之前的1000次滑动记录，返回前10位。她向右滑动 Bob → 滑动服务写入该行，查找 (Bob, Alice, like) → 未找到，仅写入滑动记录。之后 Bob 向右滑动 Alice → 滑动服务写入，查找 (Alice, Bob, like) → 找到。插入匹配行（唯一约束），返回 matched=true。双方手机收到推送通知。

6. 深度探讨（约 10 分钟）

具体存储契约。

存储 键/数据结构 条目大小 + TTL 分片键
个人主页 profile:{user_id} KV 存储中的文档 + CDN 照片 URL 约5 KB，删除前保留 user_id
活跃地理索引 geo:{s2_cell} 排序集，包含 user_id，分数=last_seen 约24 B/成员，TTL 30分钟 s2_cell
滑动日志 swipes_from:{swiper_id}:{month} 追加行 约40 B/行，热/温数据保留1–2年 swiper_id
互相喜欢索引 likes_to:{target_id} 集合，包含 swiper_id 约8–16 B/id，保留期与滑动记录相同 target_id
已展示过滤器 shown:{user_id} 布隆/布谷鸟过滤器 每用户16–64 KB，滚动6–12个月 user_id
匹配 match:{min_user}:{max_user} 唯一行 约 80 B，永久保留 排序对哈希

这是双索引模式：同一次滑动事件既作为用户历史记录存储一次，又作为互相查找索引存储一次。前者支持"我做过什么？"；后者支持"他们已经喜欢我了吗？"

存储估算：每天50亿次滑动的历史记录是重量级写入路径，紧凑格式约200 GB/天，RF=3后约600 GB/天（不含索引）。每用户 shown 过滤器是容易被忽视的内存成本：仅活跃用户才应加载，例如3000万活跃用户 × 32 KB ≈ 1 TB（若全量热存储），因此应置于分层缓存中，并对休眠用户从滑动日志重建。

卡片路径追踪。 客户端 → 卡片 API → 读取用户偏好/资料 → 查询 geo:{cell} 及相邻单元格（并行） → 从 shown:{user_id} 和黑名单中移除用户 → 并行从资料缓存中获取前约50个资料 → 排序 → 返回前10个及游标 → 在 shown:{user_id}中标记已返回的候选人。若资料缓存未命中，则从资料 KV 获取；若未命中过多，宁可返回较少卡片也不超过300 ms。

Cards request
    |-- profile/preferences MGET
    |-- geo:{cell + neighbors} reads in parallel
    |-- shown:{user_id} + block lists
    '-- profile hydrate for candidates
                -> rank -> return 10 -> update shown:{user_id}

滑动路径追踪。 客户端发送 (target_id, direction, client_uuid) → 对 (swiper,target) 进行幂等性检查 → 追加到 swipes_from:{swiper} → 更新 shown:{swiper} → 若 like，检查 likes_to:{swiper} 处对 target_id 与写入 likes_to:{target} 并行 → 若互相记录存在，则插入 match:{min,max} 唯一行并分配聊天 id。并发重复匹配会在唯一插入时失败，并读回已有的匹配行。

命名技术选型：DynamoDB/Cassandra 风格的宽行适用于 swipes_from:{swiper} ，因为写入量大且访问模式为按用户追加。关系型存储仍可用于 match:{min,max} ，因为那里更看重唯一性和精确一次的匹配创建。Redis/S2 活跃集合用于候选人，不是滑动记录的真实来源。

永不重复展示同一资料。 朴素方案：从候选人中减去滑动列表——以每用户50次滑动计算代价高昂。更优方案：在候选人服务中为每个用户维护一个"已展示"布隆过滤器，小到足以在内存中保存。布隆过滤器的假阳性仅意味着某个资料偶尔不会出现；绝不会出现相反情况。

面试要点："对于候选人生成，假阳性是可接受的，因为跳过一个可能的资料代价低廉；而两次展示同一资料才是产品缺陷。"

地理搜索的规模扩展。 S2 单元格选取合适的级别（如第12级，约1 km²）。索引 cell_id → set of recently-active user_ids。查询：查找附近单元格（1 + 8环）中符合偏好筛选条件的用户。每次位置 ping 刷新用户到单元格的映射（防抖至每5分钟一次，除非用户移动超过1 km）。

为何选择 S2 + Redis 风格活跃集合而非 Elasticsearch 地理搜索：约会应用需要新鲜、附近、活跃的用户，而非任意文本 + 地理排名。Elasticsearch 适用于更丰富的资料搜索，但 S2 活跃集合让新鲜度、TTL 和相邻单元格扩展变得直观且高效。

匹配竞态问题。 两个用户在同一秒内互相向右滑动。两次并发插入。匹配表必须使用确定性主键（如 LEAST(a,b)||GREATEST(a,b) 。 INSERT IGNORE ），从而只有一行胜出。失败的写入方读回已有行并返回相同的 match_id。

高频滑动用户的热键问题。 每天滑动500次的用户会产生热分片。按 swiper_id 分片通常没问题；对于尾部最重的用户，在客户端批量积累滑动记录，然后单次调用提交。

候选人排序质量。 纯粹按单元格内新鲜度排序会忽略吸引力/兼容性信号。在候选人服务中添加一个小型排序模型（LightGBM，约50个特征）。在生产杠杆上提及从滑动结果进行在线学习。

照片存储。 资料照片存储在对象存储（S3）中，通过 CDN 提供带签名 URL 的访问。缩略图（3种尺寸）在上传时异步生成。

隐私 / 安全。 资料可见性是双向的：如果 Alice 屏蔽了 Bob，则双方都不会出现在对方的卡片栈中。黑名单存储在用户资料中；候选人生成时进行过滤。

ELO / 匹配评分（可选）。 某些产品使用基于向右滑动率得出的吸引力评分。离线计算；按评分对用户分桶以保持匹配平衡。需做好讨论伦理权衡的准备。

优雅降级状态机。

情形 模式 数据源 延迟预算 回退
人口密集城市 正常排序卡片栈 S2 活跃单元格 + 资料缓存 <300 ms 扩展1环，排序前50
稀疏/冷门地区 耗尽模式 更宽的 S2 环 + 较早活跃的用户 <500 毫秒 展示更少卡片、扩大距离、放宽新鲜度要求
休眠用户回归 延迟过滤器加载 从持久化滑动日志重建 shown 过滤 首次请求 < 1秒 返回小批量同时异步重建完成
高频滑动用户 批量写入 滑动日志 + 互相喜欢索引 滑动确认 < 100 ms 在客户端缓冲滑动记录并小批量刷新
地理索引降级 保守候选人策略 最后已知资料位置 <500 毫秒 禁用距离敏感排序；永不违反不重复原则
匹配插入竞态 唯一行胜出 match:{min,max} Redis 有序集合 失败写入方读回已有匹配并返回相同 id

跨请求必须持久化的服务端状态：不透明的卡片游标 {user_id, geo_cell, candidate_batch_id, rank_offset}，则可使用 shown:{user_id} 过滤器、 (swiper,target)的幂等性，以及排序对匹配键。丢失游标是可以接受的；丢失 shown 会导致重复展示，因此在返回完整卡片栈前需重建。
