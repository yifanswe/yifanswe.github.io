1. 基础知识与解题思路

本题考察点

Dropbox 在用户的多台设备间同步文件树。难点在于高效同步（不重新上传未更改的内容）、多设备同时修改文件时的冲突解决、规模化去重，以及对离线友好的用户体验。

其不变量如下：

每次已提交的文件变更都被持久化存储，最终复制到用户的所有设备；任意时刻某文件的内容与确定性版本集合中的某个版本一致；并发编辑产生冲突副本（或合并）而非静默数据丢失；删除操作在保留窗口内可撤销；存储费用随去重后的唯一内容缩放。

在 ICT5 / L6+ 级别，面试官考察你是否能设计基于分块的去重、最小化带宽的高效同步协议、拥有每用户规范文件树的元数据服务，以及多设备编辑下的冲突处理。

60 秒面试框架

口述版本：

我会将 Dropbox 设计为三个平面：一个拥有每用户文件树和版本历史的元数据服务，一个用于文件分块且支持跨用户去重的内容寻址块存储，以及一个在客户端和元数据服务之间交换增量的同步协议。有趣的决策包括分块策略、增量同步、冲突解决，以及兼顾隐私的去重。

首先明确需求

功能性需求：

  • 用户在文件夹树中上传、修改、删除文件。
  • 文件在用户的设备间同步。
  • 共享：文件夹与其他用户共享，带权限控制。
  • 文件版本控制：查看并恢复旧版本。
  • 选择性同步：并非每台设备都获取每个文件。
  • Web 访问：从浏览器下载文件。
  • 评论、链接、移动端预览。

非功能性需求：

  • PB 级存储，支持跨用户去重。
  • 同步延迟：设备在线时变更在数秒内传播。
  • 带宽高效：不重新上传未更改的部分。
  • 持久性：文件内容永不丢失。
  • 离线可用：客户端无网络时仍可工作。
  • 多地区。

以下内容超出范围，除非被特别询问：

  • 实时协作编辑（独立功能；参见 Google Docs）。
  • 全文检索所有文件。
  • DRM / 受管文件权限。

规模假设

领域 假设
用户 5 亿以上。
文件 千亿级。
文件大小 KB 至 GB；平均几 MB。
日上传量 PB 级。
每用户设备数 2-5 台（典型值）。
同步延迟 在线时秒级；批量时分钟级。
存储 EB 级；每日持续增长。
去重比例 典型工作负载降低 30-60%。

瓶颈在于元数据服务的每用户扩展以及同步协议的带宽。

核心产品模型

客户端 监听本地文件系统；与元数据服务同步。 弹性；处理离线情况。

分块与去重

每个文件被切分为多个分块。分块策略的选择很重要：

固定大小分块（例如 4 MB）： - 简单。\n- 对插入操作不友好：在文件开头修改会移位所有分块，去重失效。

基于内容定义分块的可变大小分块（CDC，例如 Rabin 指纹）： - 边界由内容决定；插入只影响附近的分块。\n- 对略有修改的文件去重效果更好。\n- 分块时 CPU 开销更高。

对于每个分块：

  • block_id = SHA-256(content).
  • 若不存在则上传（通过 API 检查）。
  • 当某个文件版本引用该分块时，引用计数加一。
  • 引用计数降为零时，可进行垃圾回收。

去重范围：

  • 每用户（始终）。
  • 跨用户（隐私敏感；需要谨慎的密钥管理）。

针对带加密的跨用户去重：

  • 收敛加密：从内容哈希派生加密密钥；拥有相同文件的两个用户产生相同的密文。支持去重，但会泄露「你是否有这个文件？」的信息。
  • 每用户加密：阻止跨用户去重，但隐私更安全。

面试用语：

基于内容定义的分块加内容寻址块，同时实现带宽效率和去重。跨用户去重是隐私决策，而不仅仅是工程决策。

同步协议

每台设备在用户的事件流上维护一个本地游标。元数据服务维护每用户的仅追加事件日志。

同步流程：

  1. 客户端连接；发送其游标。
  2. 服务器响应自该游标以来的事件：文件新增、修改、删除、移动、共享。
  3. 对于修改：客户端将本地块列表与服务器的进行比对；下载缺失的块；更新本地文件。
  4. 本地变更：客户端计算新的块列表；上传新块（先检查是否存在）；以 block_refs 提交新的文件版本。
  5. 服务器发出事件；其他设备拉取并同步。

带宽效率：

  • 只传输变更的块。
  • 上传前先做存在性检查（HEAD /blocks/{id}）。
  • 针对小幅编辑的服务端增量计算。

冲突解决

两台设备在离线时编辑同一文件。两者都上传了修改后的版本。

服务端检测：

  • 每个 FileVersion 都有 parent_version_id.
  • 如果客户端提交时带有 parent_version_id = V，但当前服务器版本是 V' （其中 V' != V），冲突。

解决策略：

  • 默认：创建「冲突副本」—— filename (Device A conflict copy).ext。两个版本均保留。
  • 文本的三路合并：某些产品中可选。
  • 对于二进制文件，无法合并；冲突副本是正确答案。

面试用语：

Dropbox 不会静默合并编辑；它将两个版本保留为冲突副本。三路合并对二进制文件有风险。

移动与复制

移动文件只改变元数据，不重新上传内容。

  • 移动：更改 的元数据事务 parent_id 或 name.
  • 复制：新节点，相同的 block_refs，递增引用计数。

这使得大型文件夹移动几乎零成本。

共享

共享文件夹有一个在多个用户树中可见的共享根节点。

  • 共享文件夹拥有自己的 ACL。
  • 每个成员的树通过 id 引用共享文件夹。
  • 权限在元数据服务处检查。
  • 同步协议向所有成员传递共享文件夹的事件。

版本控制

每次文件修改都会创建一个新的 FileVersion。

  • 旧版本在窗口期（通常 30 天）或付费层更长时间内保留。
  • 恢复：将旧版本的 block_refs 复制为新的当前版本。
  • 垃圾回收删除超过保留期的旧版本。

扩展与分区

数据 分区键 原因
元数据节点 user_id （或 workspace_id） 每用户树；分区内事务性。
事件日志 user_id 每用户同步游标。
块 block_id 哈希值 均匀分布。
块引用计数 block_id 哈希值 与块相同的分片。
共享文件夹 shared_folder_id 独立于任何单个用户的分区。

热分区：单个用户在一个文件夹中有数百万个文件。缓解措施：若文件夹增长超过阈值则进行子分片；或直接接受并扩展用户分区。

一致性模型

层级 一致性 原因
每用户元数据 强一致性 真实数据来源。
块存在性 强一致性 提交前必须检查。
块内容 不可变（内容寻址） 一旦写入不可更改。
同步到其他设备 最终一致（秒级） 正常。
共享事件 最终一致 成员在数秒内看到新增内容。

故障模式与缓解措施

故障模式 缓解措施 监控指标
并发编辑之间的冲突。 冲突副本。 每用户冲突率。
上传未完成（网络中断）。 可恢复；丢弃部分块。 未完成上传的时长。
块损坏。 读取时哈希校验；副本；告警。 哈希不匹配率。
垃圾回收删除被引用的块。 引用计数 + 宽限期；删除前验证。 GC 删除但仍被引用的块数量。
元数据存储热用户分区。 子分片；或垂直扩容。 每分区 QPS。
同步卡在错误事件上。 隔离跳过；人工修复。 卡住游标的数量。
跨用户去重泄露存在性信息。 默认使用每用户去重；谨慎使用收敛加密。 隐私审计。
GC 后恢复旧版本。 按保留期维护有界的版本历史。 超出保留期的恢复失败次数。
共享文件夹被撤销。 元数据变更；客户端删除本地副本。 撤销传播延迟。
大规模文件夹上传（1000 万个文件）。 限流；批量处理事件；进度上报。 每用户上传速率。

运维指标

正确性：

  • 每用户冲突率。
  • 块读取时哈希不匹配。
  • 同步游标滞后。
  • 丢块事件（必须为零）。

容量：

  • 上传/下载 QPS 与带宽。
  • 元数据服务每用户 QPS。
  • 事件日志扇出延迟。

业务层面：

  • 去重比率。
  • 每用户存储量。
  • 每用户活跃设备数。
  • 共享活动。

多区域策略

  • 元数据的用户归属区域。
  • 块存储在全球范围内复制以加速下载。
  • 同步通知路由至用户的归属区域；远端设备通过本地网关订阅。
  • 可容忍的跨区域复制延迟。

需要指出的权衡

  • 内容定义分块节省带宽但消耗 CPU；固定大小分块更简单但效率较低。
  • 跨用户去重节省存储空间，但引发隐私顾虑。
  • 永久保存每个块会浪费空间；引用计数 + GC 增加了复杂性。
  • 三路合并对文本很有效，但对二进制文件很危险；默认采用冲突副本。
  • 较大的块降低元数据开销但减少去重收益；较小的块则相反。
  • 强每用户一致性限制了跨用户事务；事务性共享文件夹更难实现。

常见陷阱与更优做法

陷阱 更优做法
变更时重新上传整个文件。 分块增量同步；检查块是否存在。
静默合并并发编辑。 默认采用冲突副本；询问用户。
最后写入者优先。 通过 parent_version_id 检测并发编辑。
元数据与块的同步写入。 两阶段：先写块，再提交元数据。
共享作为复制。 共享文件夹由两位用户共同引用。
单一全局元数据服务。 按用户分片。

面试官可能深入探讨的方向

深入方向 优秀回答要点
分块与去重 CDC 与固定大小对比；内容寻址；引用计数。
同步协议 基于游标的事件日志；增量块；存在性检查。
冲突解决 通过 parent_version 检测；创建冲突副本。
共享 共享文件夹作为独立节点；ACL；同步至所有成员。
多区域 每用户归属区域；复制的块；区域网关。
存储效率 去重 + 分层存储 + 引用计数。

Staff 级别回答检查清单

  • 元数据与块分离。
  • 带去重的内容寻址块。
  • 带存在性检查的分块上传/下载。
  • 每用户事件日志用于同步。
  • 并发编辑默认采用冲突副本。
  • 通过共享文件夹节点实现共享。
  • 版本控制 + 对未引用块进行 GC。
  • 多区域部署，每用户有归属区域。

面试用语

Dropbox 是基于内容寻址块存储的每用户文件树。分块增量同步最小化带宽；冲突副本在并发编辑时保留两个版本；跨用户去重涉及隐私权衡。元数据服务是事实来源；块存储是内容可寻址且不可变的。

2. 模拟面试

面试官： 设计 Dropbox。

候选人： 我将设计两个平面以及一个同步协议。元数据平面管理每个用户的文件树、版本和共享——每用户强一致性。块平面是内容可寻址存储，不可变、已去重。同步协议在客户端与元数据服务之间通过每用户日志交换事件，仅传输已变更的块。

面试官： 文件上传如何工作？

候选人： 客户端使用内容定义分块对文件进行分块——边界由内容指纹确定。每个块的 block_id 是其内容的 SHA-256 值。对于每个块，客户端检查 HEAD /blocks/{block_id}。若存在则跳过，否则上传。所有块存在后，客户端提交一个新的 FileVersion，包含 block_refs 列表和 parent_version_id。元数据服务以事务方式执行提交并发出事件。

面试官： 为何使用内容定义分块？

候选人： 插入和小幅编辑不会移动分块边界。如果在文件顶部插入文本，固定大小分块会移动所有块边界，导致去重失效。CDC 基于内容设置边界，因此只有紧邻编辑位置的块会改变，其余块仍可去重。

面试官： 同步如何工作？

候选人： 每个设备在用户的事件日志中持有一个游标。连接时，设备发送其游标；服务器返回自该游标以来的所有事件（创建、修改、删除、移动、共享）。对于修改，客户端将新版本的 block_refs 与本地已有的进行对比，仅下载缺失的块并重建文件。事件日志按用户维护；同步范围受限于该用户的活动，而非整个全局系统。

面试官： 两台设备在离线状态下编辑同一文件。

候选人： 当设备 A 上线并提交新版本时，元数据服务应用该版本并更新 current_version_id。当设备 B 上线并尝试以 parent_version_id = old_version提交时，服务器检测到冲突：当前版本已不再是设备 B 所认为的父版本。服务器创建冲突副本——一个新节点，名称类似 file (Device B conflict copy).ext。两个版本均被保留，由用户手动解决。

面试官： 为何不自动合并？

候选人： 对于文本，三路合并是可行的。但对于二进制文件——绝大多数文件——这很危险。静默合并会损坏内容。冲突副本使两个版本均可见，由用户决策。部分产品确实将文本合并作为可选功能提供。

面试官： 存储——去重有帮助吗？

候选人： 帮助很大。同一用户上传的相同内容通过内容哈希去重。跨用户去重更为敏感：若两位用户拥有相同文件，他们是否共享块？共享可节省存储，但会泄露存在性信息。收敛加密（加密密钥从内容派生）支持跨用户去重，但会暴露"你是否拥有该文件？"类型的攻击。默认采用每用户去重更为安全。

面试官： 移动操作如何工作？

候选人： 仅元数据操作。移动 1 TB 的文件夹只需一次事务，修改 parent_id。无需重新上传内容。重命名同理。复制操作创建具有相同 block_refs 的新节点并递增引用计数。

面试官： 版本控制？

候选人： 每次修改都会创建一个新的 FileVersion，链接到前一个版本。旧版本保留 30 天（或按套餐）。恢复时，将旧版本的 block_refs 复制为新的当前版本。垃圾回收删除未引用的版本并递减块的引用计数；引用计数为零且超过宽限期的块将从块存储中删除。

面试官： 共享？

候选人： 共享文件夹是多个用户文件树引用的节点。共享文件夹上的 ACL 控制权限。共享文件夹的同步事件流向所有成员。向共享文件夹添加文件会发布事件；所有成员的客户端接收该事件并下载块。

面试官： 选择性同步？

候选人： 按设备配置哪些文件夹在本地同步。客户端遵循该配置：仅订阅这些文件夹的事件；仅为这些文件夹下载块。元数据仍保有完整文件树；客户端只是不实体化未选中的文件夹。

面试官： 多区域？

候选人： 元数据采用每用户归属区域。块存储跨地理区域复制以加速下载。通知服务设有区域网关；用户的归属区域发布事件；远端网关订阅并转发至本地设备。跨区域复制为异步，具有有界延迟。

面试官： 用户一次性上传 1000 万个文件。

候选人： 限流。每用户上传速率限制。批量发出事件以避免通知风暴。向客户端上报进度。元数据事务大小有界；客户端增量上传。该用户的分区可能变热；必要时进行子分片。

面试官： 总结。

候选人： 每用户元数据服务管理文件树。块服务以去重方式存储内容寻址的不可变块。分块增量同步通过存在性检查最小化带宽。冲突产生冲突副本。共享是带 ACL 的共享文件夹节点。版本控制 + 引用计数 + GC 管理存储。每用户归属区域配合复制块实现多区域部署。最难的正确性问题是冲突处理；最难的规模问题是大型文件夹操作和每用户分区热点。

3. 练习备注

五分钟版本

  1. 澄清：同步、共享、版本控制、设备。
  2. 不变量：每用户元数据权威；块不可变。
  3. 元数据 + 内容寻址块存储。
  4. 内容定义分块；上传前检查块是否存在。
  5. 每用户事件日志用于同步。
  6. 并发编辑采用冲突副本。
  7. 通过共享文件夹节点实现共享。
  8. 版本控制 + GC；多区域每用户归属区域。

一分钟结尾陈述

Dropbox 是基于内容寻址块存储构建的每用户文件树。分块增量同步仅传输变更内容；上传前的存在性检查利用了去重优势。并发编辑生成冲突副本；对二进制文件自动合并具有风险。共享使用被多个用户引用的共享文件夹节点。版本控制是追加式的，配合引用计数 GC。每用户归属区域加上全球复制的块，兼顾了正确性与性能。

最佳面试用语

  • "内容定义分块结合内容寻址块，同时实现带宽效率和去重。"
  • "并发编辑产生冲突副本；我们不静默合并二进制文件。"
  • "移动和复制是纯元数据操作；不重新上传内容。"
  • "跨用户去重是隐私决策，而不仅仅是工程决策。"
  • "每用户事件日志使同步范围受限于活动量，而非全局规模。"
