1. 基础知识与解题思路

本题考察点

指标平台从数百万台主机摄取时序数据，并为仪表盘和告警提供服务。难点在于基数爆炸（指标 × 维度 × 主机）、高效存储、长时间范围内的低延迟查询，以及可靠的告警评估。

其不变量如下：

发送到系统的每个指标数据点都在有界延迟内持久化存储，按照保留策略进行保存，可通过指标名称和标签过滤器跨配置时间范围查询，并用于在告警评估窗口内可靠触发的告警评估。

在 ICT5 / L6+ 级别，面试官考查的是你能否设计大规模时序摄取、降采样与保留、跨高基数标签空间的聚合查询引擎、告警评估，以及基数控制的推理能力。

60 秒面试框架

口述版本：

我会将指标系统设计为三个平面：摄取平面（接收指标数据点并写入时序存储）、存储平面（列式存储并按保留层级降采样），以及查询/告警平面（评估仪表盘和告警规则）。关键决策包括基数控制、降采样策略、长时间范围内的查询延迟，以及告警评估的可靠性。

首先明确需求

功能性需求：

  • 主机和服务发出指标（计数器、仪表、直方图、分布）。
  • 指标带有维度标签（主机、服务、区域等）。
  • 仪表盘按时间范围查询指标。
  • 告警规则：阈值、异常、变化率。
  • 告警触发时发送通知。
  • 日志和追踪（通常集成；此处聚焦于指标）。

非功能性需求：

  • 每秒摄取数百万个数据点。
  • 查询延迟：仪表盘低于 1 秒；长范围查询数秒。
  • 保留策略：近期高分辨率（1s/10s），长期降采样（5min、1h）。
  • 告警评估：通常每分钟一次。
  • 多地区。
  • 告警状态和静默的可审计性。

以下内容超出范围，除非被特别询问：

  • 日志管理（相似但有所区别）。
  • APM 追踪。
  • RUM 与合成监控。

规模假设

领域 假设
主机数量 大型客户超过 100k；跨客户达数百万。
每主机指标数 数百至数千个。
每秒数据点数 每集群 10M+。
基数 数亿个唯一时序。
留存期 典型保留 15 个月，含降采样。
查询 QPS 数千个。
告警规则 每客户数千；跨客户数千万。

瓶颈在于高基数与长范围查询的组合，以及在高流失率下可靠的告警评估。

核心产品模型

层级 用途 需求
摄取 接收指标数据点；去重；路由。 高吞吐量。
存储 持久化时序；降采样层级。 压缩；可查询。
查询引擎 聚合、过滤、范围查询。 亚秒级响应仪表盘。
告警引擎 评估告警规则；触发通知。 可靠；有界评估延迟。
通知 通知值班人员。 复用通知系统。

核心组件

组件 职责 关键细节
Agent 运行于主机；采集指标；发送至摄取层。 轻量级；本地聚合。
摄取网关 接收突发流量；认证；限流；路由。 无状态；水平扩展。
缓冲区 摄取层与存储层之间的持久化队列。 类 Kafka。
写入器 将时序数据写入 TSDB。 按指标 + 标签哈希分片。
索引 基于标签的搜索：标签 → 时序 ID。 倒排索引。
TSDB 时序数据库。 列式；压缩。
降采样器 汇总为更长时间桶。 旧数据采用更低分辨率。
查询引擎 解析查询；获取时序；聚合；返回结果。 分布式。
告警引擎 按计划评估告警规则。 可靠；支持回放安全。
仪表盘服务 图表和表格的 UI。 缓存查询结果。
通知服务 通知值班人员。 复用独立系统。

数据模型

指标时序

TimeSeries(
  series_id,                // hash of (metric_name, tag_set)
  metric_name,
  tag_set: {key: value, ...},
  type,                     // counter, gauge, histogram, distribution
  data_points: [(ts, value, ...)]
)

一个数据点为 (timestamp, value)。直方图和分布每个数据点包含多个值（桶）。

标签索引

TagPosting(
  tag_key,
  tag_value,
  series_ids: [series_id]
)

对于查询"指标 X，其中 region=us-east"：查找 region=us-east posting，与 metric_name 索引取交集。

告警规则

AlertRule(
  rule_id,
  name,
  query,                    // metric query + threshold
  evaluation_interval,
  for_duration,             // alert fires only if condition true for X
  thresholds,
  notification_targets,
  silence_windows,
  state,                    // pending, alerting, ok, silenced
  version
)

告警事件

AlertEvent(
  event_id,
  rule_id,
  state_change,             // ok -> alerting
  evaluated_at,
  trigger_value,
  metadata
)

摄取流水线

  1. 主机上的 Agent 每 10s 批量发送指标，通过 HTTP/2 或 gRPC 发送至摄取网关。
  2. 网关对客户进行认证、限流、校验 schema。
  3. 写入类 Kafka 缓冲区以保证持久性。
  4. 写入器消费数据；对每个数据点：\n - 计算 series_id = hash(metric_name, sorted_tag_set)。\n - 将数据点追加到时序。\n - 若时序为新，则更新标签索引。
  5. 写入器按 series_id 分片，每个写入器处理一个子集。

存储

TSDB 采用列式存储和压缩：

  • 时序数据具有极高压缩率：时间戳使用 delta 编码，值使用 Gorilla 风格浮点压缩。
  • 按时序逐步压缩。
  • 按 series_id + 时间块建立索引。

存储层级：

  • 热层（近期数天）： 高分辨率（1-10 秒），内存或 SSD。
  • 温层（近期数月）： 较低分辨率（1-5 分钟），本地磁盘。
  • 冷层（长期）： 按小时聚合，对象存储。

降采样器定期运行，将旧数据汇总为更低分辨率。

面试用语：

时序压缩是秘密武器。Gorilla 风格编码对典型指标可实现 10-50 倍压缩比。降采样可在不爆炸存储的前提下延长保留期。

基数控制

基数 = 唯一时序的数量。添加高基数标签（如 user_id）会导致基数和存储爆炸。

控制措施：

  • 每客户基数预算；超出时告警并限流。
  • 标签白名单/黑名单。
  • 高基数标签需经审批的允许名单。
  • 自动基数检测：对基数持续增长的指标发出告警。

指标系统中最棘手的生产问题是基数爆炸。客户在计数器上添加 user_id 可能在一夜之间导致存储成本飙升。

查询引擎

类似 PromQL 或 Datadog 的查询语言： avg(rate(http_requests{service="api"}[5m])).

执行过程：

  1. 解析查询 AST。
  2. 通过标签过滤器解析匹配时序：标签索引查找。
  3. 对每条匹配时序，在时间范围内获取数据点。
  4. 应用函数（rate、avg、sum、max）。
  5. 按指定维度分组。
  6. 返回时序结果。

优化措施：

  • 预聚合常见汇总。
  • 缓存查询结果。
  • 跨时序分片并行化。
  • 大范围查询采用分布式查询计划。

长范围查询（15 个月）使用降采样层；近期范围使用高分辨率层。

告警引擎

每条告警规则有一个评估间隔（例如每分钟一次）。

  1. 调度器触发规则评估。
  2. 查询引擎在评估窗口内对规则表达式进行求值。
  3. 将结果与阈值进行比较。
  4. 更新规则状态：\n - ok -> pending 阈值被触发时。\n - pending -> alerting after for_duration 持续告警中。\n - alerting -> ok 阈值不再被触发时。
  5. 状态变更时，发出告警事件。
  6. 通知服务通知值班人员。

可靠性：

  • 告警评估不能静默失败。
  • 每条规则心跳检测；若规则在 N 个间隔内未评估则告警。
  • 幂等评估；支持回放安全。
  • 评估器宕机后回填以捕获漏报。

静默：

  • 每规则静默窗口。
  • 维护模式。
  • 在不丢失状态的前提下抑制通知。

面试用语：

一个不触发的告警比误报更严重。告警评估必须进行心跳检测和自我监控。

扩展与分区

数据 分区键 原因
时序数据 series_id 哈希值 分散写入；本地化读取。
标签索引 (tag_key, tag_value) 哈希值 时序查找。
告警规则 rule_id 哈希值 独立评估。
告警状态 rule_id 哈希值 状态转换。
客户配置 customer_id 哈希值 租户隔离。

热指标（高写入速率）：单时序吞吐量成为瓶颈；必要时通过添加分片标签进行分区。

一致性模型

层级 一致性 原因
摄取 持久写入缓冲区；最终写入完成 有界延迟。
时序存储 最终一致 对监控可接受。
标签索引 最终一致 时序发现。
告警状态 强一致性（CAS 用于状态转换） 避免重复触发。
告警评估 每规则强一致性 可靠。

多区域策略

  • 按区域摄取和存储；指标通常在产生地消费。
  • 跨区域查询联邦用于全局仪表盘。
  • 告警评估区域本地化；全局告警支持跨区域。

故障模式与缓解措施

故障模式 缓解措施 监控指标
摄取过载。 按客户限流；缓冲区吸收。 缓冲区深度。
基数爆炸。 按客户预算；告警；限流。 基数增长。
热时序。 添加分片标签；预聚合。 每时序写入速率。
查询超时。 回退至降采样层；返回部分结果。 查询 p99。
告警评估器宕机。 心跳检测；恢复后回填。 评估延迟。
通知抖动。 迟滞（for_duration）；静默。 抖动次数。
存储增长。 降采样；保留策略执行。 每客户存储量。
标签索引热键。 缓存；进一步分区。 热键 QPS。
跨区域查询延迟。 以有界陈旧性联邦查询。 联邦延迟。
主机上 Agent 崩溃。 心跳检测；告警。 Agent 缺失。

运维指标

正确性：

  • 数据点丢失（必须趋近于零）。
  • 告警评估延迟。
  • 基数增长。

容量：

  • 按客户划分的接入 QPS。
  • 查询 p99。
  • 存储增长。

业务层面：

  • 每个客户的活跃指标数。
  • 告警触发次数。
  • 仪表板使用情况。

需要指出的权衡

  • 分辨率越高，存储成本越大；降采样可延长保留时间，但会损失细节。
  • 预聚合 rollup 可加速查询，但会增加写入开销。
  • 严格的基数限制有助于控制成本，但会令客户感到受限。
  • 异步接入提高吞吐量，但会增加查询延迟。
  • 告警评估间隔在响应性与成本之间取得平衡。
  • 较长的 for_duration 可减少抖动，但会延迟通知。

常见陷阱与更优做法

陷阱 更优做法
基于行的时序存储。 列式存储加压缩。
永久保存每个数据点。 降采样分层。
所有告警使用单线程评估。 分布式评估；已分片。
忽略基数。 预算；告警；限流。
同步接入。 缓冲区 + 异步写入器。
单一全局查询引擎。 分片；联邦化。

面试官可能深入探讨的方向

深入方向 优秀回答要点
时序存储 列式存储 + Gorilla 压缩 + 降采样。
基数 按租户预算；高基数标签检测。
查询引擎 分布式；并行；部分结果。
告警可靠性 心跳；回填；重放安全。
标签索引 倒排索引；已分片。
多区域 按 region 存储；联邦查询。

Staff 级别回答检查清单

  • 接入 → 缓冲区 → 写入器 → TSDB。
  • 使用 Gorilla 压缩的列式存储。
  • 用于保留策略的降采样分层。
  • 基于标签的倒排索引。
  • 带心跳的告警引擎。
  • 基数控制。
  • 带联邦的多 region 架构。

面试用语

指标平台是高吞吐量时序接入、列式压缩存储与告警引擎的组合。最主要的扩展问题是基数——添加一个高基数标签可能导致存储成本爆炸式增长。降采样可延长数据保留时间。告警评估必须具备心跳机制，因为告警静默失效比误报更危险。

2. 模拟面试

面试官： 设计 Datadog 指标系统。

候选人： 三个平面：接入层负责从 agent 接收指标数据点并持久化写入；存储层采用带降采样分层的列式存储；查询/告警层负责评估仪表板和告警规则。不变式：每个数据点持久化；可查询；告警可靠。最难的问题是基数与长范围查询的结合。

面试官： 数据如何存储？

候选人： 按序列按时间排列的数据点。Series_id 是 metric_name 与排序后 tag_set 的哈希值。存储采用列式——时间戳与数值分开存储，均使用 Gorilla 风格编码压缩，压缩比达 10-50x。Compaction 随时间合并 segment。分层：热层（1-10s 分辨率，内存/SSD）、温层（1-5 分钟，本地磁盘）、冷层（1 小时，对象存储）。Downsampler 定期执行 rollup。

面试官： 基数爆炸问题？

候选人： 最常见的生产问题。向一个 counter 添加一个 user_id 标签可能导致存储爆炸。防御措施：按客户设置基数预算；超限时告警；必要时限流接入。标签白名单/黑名单。基数检测：某个指标的唯一序列数快速增长时触发告警。面向客户的工具可展示高基数的"罪魁祸首"。

面试官： 查询—— avg(rate(http_requests{service="api"}[5m])).

候选人： 解析 AST。从标签索引中解析 service="api" 以获取匹配的序列。对于每个序列，获取请求窗口内的数据点。对每个序列执行 rate()（时间差分）。对所有序列执行 avg() 聚合。返回结果序列。在序列分片间并行执行。对于长范围查询，自动使用降采样分层。

面试官： 告警评估？

候选人： 调度器按每条规则配置的评估间隔（通常每分钟一次）运行。对于每条规则，查询引擎在评估窗口内对表达式求值。与阈值比较。状态机：ok → pending（触发条件时）→ alerting（持续 for_duration 后）→ ok。状态变更时发出事件；通知服务通知值班人员。每条规则发送心跳；若某条规则在 N 个间隔内未完成评估，则对评估器发出告警。重放安全：重复运行某个区间不会导致重复触发。

面试官： 为什么需要心跳？

候选人： 告警静默失效是最糟糕的结果。如果评估器宕机而我们无从知晓，值班人员对真实的生产问题将毫无感知。心跳机制确保我们能检测到评估器宕机；恢复后回填可补全遗漏的触发记录。

面试官： 热指标——某个指标每秒接收 1M 个数据点。

候选人： 单个序列无法处理每秒 1M 个数据点——单写入器成为瓶颈。缓解方案：添加一个分片标签（随机后缀），将该指标拆分为 N 个子序列。客户在查询时重新聚合。替代方案：在接入时使用流处理器进行预聚合。

面试官： 降采样——请解释。

候选人： 历史高分辨率数据会被 rollup。例如，7 天后，1 秒级数据点变为 1 分钟的平均值、最小值、最大值和总和。存储成本显著下降。长范围查询自动使用降采样分层。Downsampler 以后台任务方式运行，幂等执行。

面试官： 多租户——如何隔离嘈杂的租户？

候选人： 接入层的按租户限速。基数预算。对于大客户，存储分片可能按租户划分；小客户则共享分片。查询引擎限制每个租户的资源使用。告警引擎保证按租户公平调度。

面试官： 多区域？

候选人： 按 region 进行接入和存储。大多数指标在本地消费。全局仪表板通过跨 region 联邦查询获取数据。跨 region 告警在指定 region 中进行评估。仪表板可容忍跨 region 延迟（秒级）。

面试官： 总结。

候选人： 三个平面：接入、存储、查询/告警。带降采样分层的列式 Gorilla 压缩存储。基于标签的倒排索引用于序列查找。带心跳与重放安全的告警引擎。基数控制可防止最常见的生产问题。带联邦的多 region 架构。最主要的扩展问题是基数；最主要的正确性问题是告警可靠性。

3. 练习备注

五分钟版本

  1. 明确范围：规模、保留时间、告警需求。
  2. 不变式：数据点持久化；告警可靠。
  3. 接入 → 缓冲区 → 写入器 → 列式 TSDB。
  4. Gorilla 压缩 + 降采样分层。
  5. 标签倒排索引。
  6. 支持并行与降采样分层的查询引擎。
  7. 带心跳的告警引擎。
  8. 将基数控制作为持续过程。

一分钟结尾陈述

指标平台是高吞吐量时序接入、列式压缩存储以及查询与告警引擎的组合。Gorilla 压缩使存储具有可行性；降采样在不导致成本爆炸的前提下延长了保留时间。标签倒排索引支持过滤查询。告警引擎带心跳机制按规则评估，以便检测静默失效。基数是最主要的生产关注点；按客户预算与检测机制对其加以控制。

最佳面试用语

  • "Gorilla 压缩使时序存储具有可行性。"
  • "降采样延长了保留时间；近期数据高分辨率，历史数据被 rollup。"
  • "基数爆炸是最大的生产难题；设置预算并主动检测。"
  • "告警评估必须具备心跳机制；静默失效比误报更糟糕。"
  • "多 region 采用按 region 存储与联邦查询的方式。"
