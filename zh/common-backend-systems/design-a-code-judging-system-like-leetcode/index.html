1. 基础知识与解题思路

本题考察点

LeetCode 运行用户提交的代码，对照隐藏测试用例进行评测，返回评测结果（Accepted、Wrong Answer、TLE、Runtime Error），并追踪竞赛排名。难点在于在隔离沙箱中安全执行不可信代码、在竞赛负载下快速返回评测结果，以及实时计算竞赛得分。

其不变量如下：

每次提交均在具有确定性资源限制的隔离沙箱中运行，对照标准测试用例进行评测，产生由运行结果确定的评测结论，并在数秒内更新用户的个人档案和竞赛排名；恶意代码无法逃逸沙箱。

在 ICT5 / L6+ 级别，面试官考察的是你能否设计沙箱化代码执行（容器、gVisor、Firecracker）、应对竞赛突发流量的评测机群扩缩容、竞赛计分（罚时制度），以及作弊检测的推理能力。

60 秒面试框架

口述版本：

我会将 LeetCode 设计为：一个提交服务，将代码入队待评测；一个隔离评测 worker 机群，在沙箱（容器 + seccomp 或 Firecracker）中运行提交；一个评测结果服务，生成评测结论；以及一个竞赛计分服务，追踪排名。沙箱安全和竞赛突发流量处理是核心挑战。关键设计决策包括沙箱技术选型、评测机群扩缩容、竞赛计分和作弊检测。

首先明确需求

功能性需求：

  • 用户向题目提交代码。
  • 系统对照测试用例运行代码。
  • 返回评测结果：Accepted、Wrong Answer、TLE、MLE、Runtime Error、Compilation Error。
  • 追踪用户已通过的题目及排名。
  • 含排行榜的竞赛功能。
  • 支持多种编程语言。
  • 讨论论坛（超出范围）。

非功能性需求：

  • 普通提交的评测结果在数秒内返回。
  • 竞赛突发流量：竞赛开始时每秒数千次提交。
  • 强沙箱安全（不可信代码）。
  • 强制执行资源限制。
  • 为全球用户提供多区域服务。

以下内容超出范围，除非被特别询问：

  • 题目创作工具。
  • 高级订阅服务。
  • 模拟面试。

提交流程

  1. 用户 POST 代码。
  2. API 校验：代码大小、编程语言、题目访问权限。
  3. 持久化提交记录，设置初始状态 pending.
  4. 入队至评测队列。
  5. 返回 submission_id；用户通过轮询或 SSE 订阅获取结果。
  6. 评测 worker 取任务。
  7. 编译（如需要）。
  8. 在沙箱中对照每个测试用例运行，并施加资源限制。
  9. 将输出与预期结果对比验证。
  10. 汇总评测结论。
  11. 更新提交记录。
  12. 如适用，更新用户档案和竞赛排名。
  13. 通过 SSE 通知用户。

沙箱安全

运行任意用户代码需要强隔离。

安全性递增的选项：

  1. 容器 + seccomp： 限制系统调用。启动开销低，隔离性尚可。存在内核漏洞利用风险。
  2. gVisor： 用户态内核；更强隔离；有一定性能开销。
  3. Firecracker： 轻量级虚拟机；硬件级隔离；启动约 125ms。
  4. 完整虚拟机： 隔离性最强；启动慢。

对于 LeetCode 规模：通常容器 + seccomp + 受限用户 + cgroups 资源限制加固已足够。对安全要求极高的场景使用 Firecracker。

限制条件：- 通常禁用网络。- 文件系统受限（仅测试输入）。- 时间和内存 cgroups。- 禁止访问 /proc。- 只读根文件系统。- 放弃所有 capabilities。

资源限制

每个测试用例运行时的限制：- CPU 时间（如典型题目 1 秒）。- 墙钟时间（通常为 CPU 限制的 2-3 倍）。- 内存（如 256 MB）。- 输出大小（防止无限输出）。- 进程数量。

通过 cgroups + rlimit 强制执行。超出限制时：- CPU/墙钟时间：TLE（超时）。- 内存：MLE（内存超限）。

评测逻辑

对每个测试用例：1. 设置沙箱并准备输入文件。2. 以输入作为 stdin（或参数）运行二进制程序。3. 捕获 stdout、stderr。4. 将 stdout 与预期输出对比。5. 若匹配则通过，否则判定为 WA。

对比方式：大多数题目采用精确字符串匹配。对有多个合法输出的题目使用自定义 checker。

编译

编译型语言：每次提交编译一次；对每个测试用例复用缓存的二进制文件。

编译时错误 → CE（编译错误）。

解释型语言：跳过编译步骤。

竞赛计分

实时更新排行榜。

ICPC 风格：每题二值判定（通过或未通过）；罚时 = 首次 AC 时间 + 每次错误提交 20 分钟。

IOI 风格：按测试用例部分得分。

实现方案：- ContestSubmission 按（用户, 题目）记录尝试次数和最佳得分。- 每次新提交时重新计算用户竞赛得分。- 热门竞赛使用子计数器分片。- 排行榜缓存，每隔几秒刷新一次。

突发流量处理（竞赛开始）

竞赛开始时数千次并发提交。

  • 评测队列吸收突发流量。
  • 竞赛前预先自动扩容评测机群（预热）。
  • 优先队列：竞赛提交优先于练习提交。
  • 为竞赛专设独立资源池。

竞赛前准备工作：- 预热沙箱。- 在评测 worker 上预加载题目测试用例。- 验证评测机群容量。

作弊检测

跨提交代码相似度检测。

  • 对代码进行 token 化和指纹提取。
  • 与其他提交进行比对。
  • 相似度超过阈值时标记。
  • 竞赛提交进行人工审核。

同时关注行为信号（极快的解题速度、相同的时间模式）。

扩展与分区

数据 分区策略
提交记录 user_id；时间
题目 problem_id
测试用例 problem_id
用户档案 user_id
竞赛提交记录 contest_id
评测队列 按哈希分区以实现负载均衡

一致性模型

层级 一致性
提交写入 强一致性
评测结果 强一致性（唯一最终结果）
用户档案聚合 最终一致
竞赛排行榜 最终一致性（秒级延迟）
作弊检测 最终一致

多区域策略

  • 每区域部署提交 API + 评测 worker。
  • 题目测试用例全球复制（不可变）。
  • 用户档案分片存储。
  • 竞赛通常使用单一全球排行榜；跨区域复制。

故障模式与缓解措施

故障模式 缓解措施 监控指标
沙箱逃逸。 多层防御；审计。 检测到的逃逸事件（零次）。
评测 worker 崩溃。 在不同 worker 上重试提交。 Worker 健康状态。
竞赛突发流量过载。 预先扩容；优先队列；自动扩容。 队列深度。
抄袭/作弊。 异步检测；人工审核。 标记率。
因 worker 缓慢（非用户代码问题）导致的 TLE。 在不同 worker 上重新运行；标准化硬件。 每 worker 性能指标。
测试用例泄露。 严格访问控制；错误消息中不泄露信息。 审计访问记录。
提交丢失。 持久化队列；重试。 提交丢失率。
评测结果流推送中断。 降级为轮询。 流传输失败率。
排行榜延迟。 子计数器；缓存。 延迟分布。
跨区域复制。 异步；有界。 延迟。

运维指标

正确性：

  • 判题准确率。
  • 沙箱逃逸尝试。
  • 作弊检测率。

容量：

  • 判题队列深度。
  • 每秒提交数。
  • 每个 worker 的利用率。

业务层面：

  • 每道题的提交次数。
  • 竞赛参与人数。
  • 用户留存率。

需要指出的权衡

  • 更强的沙箱（Firecracker）= 更安全但启动更慢。
  • 每个测试用例独立沙箱 vs 每次提交单一沙箱（更快但隔离性略低）。
  • 实时排行榜 vs 最终一致（用户体验 vs 成本）。
  • 为竞赛预热 vs 成本。
  • 严格 TLE vs 允许小幅误差（依赖硬件）。
  • 在 worker 上缓存测试用例 vs 始终拉取（隐私 + 性能）。

常见陷阱与更优做法

陷阱 更优做法
直接在宿主机上运行代码。 沙箱隔离。
每次提交启动一个独立的判题 worker。 Worker 池；复用。
在 API 中同步判题。 异步 + 队列。
同步更新排行榜。 异步聚合。
所有题目使用纯字符串匹配。 针对模糊输出使用自定义 checker。
竞赛与练习共用同一集群。 竞赛使用专用资源池。

面试官可能深入探讨的方向

深入方向 优秀回答要点
沙箱 容器 + seccomp + cgroups；或 Firecracker。
竞赛弹性扩容 预热；优先级队列；自动扩容。
排行榜 子计数器；已缓存；亚秒级更新。
作弊检测 代码相似度；行为信号。
判题结果流式推送 SSE；轮询降级。
多区域 各区域独立部署判题节点；全局排行榜。

Staff 级别回答检查清单

  • 提交队列 + 判题集群 + 判题结果聚合器。
  • 沙箱化执行（容器或虚拟机）。
  • 每个测试用例设置资源限制。
  • 通过聚合实现实时排行榜。
  • 作弊检测异步进行。
  • 竞赛突发流量处理。
  • 多区域部署，全局状态同步。

面试用语

代码判题系统是一个异步流水线：提交入队，判题 worker 在沙箱中运行，聚合判题结果，更新用户档案。沙箱安全性和竞赛突发流量处理是核心挑战。沙箱采用容器 + seccomp + cgroups，或使用 Firecracker 提供更高安全保证。竞赛排行榜通过子计数器聚合在数秒内完成更新。

2. 模拟面试

面试官： 设计 LeetCode。

候选人： 提交服务将代码排队等待判题；由沙箱化的判题 worker 集群对隐藏测试用例进行测试；判题结果服务负责聚合；随后更新用户档案和竞赛评分。不变式为：每次提交在配置的资源限制内经过沙箱化执行后产生确定性判题结果，用户档案和竞赛排行榜在数秒内完成更新。

面试官： 沙箱——如何实现？

候选人： 分层设计。Linux 容器配合受限用户、seccomp 过滤系统调用、cgroups 限制 CPU/内存、只读文件系统（仅允许访问测试输入）、禁用网络、丢弃所有 capabilities。对大多数场景已足够。如需更强隔离（高安全场景），可使用 gVisor（用户态内核）或 Firecracker（轻量级 VM，提供硬件级隔离）。容器开销较小；Firecracker 约增加 ~125ms 启动时间。

面试官： 请介绍一次提交的完整流程。

候选人： 用户 POST 代码。API 校验（大小、语言）。将 Submission 持久化为 pending。入队到判题队列。返回 submission_id；用户通过轮询或 SSE 订阅获取判题结果。判题 worker 拾取任务。编译（编译型语言若已缓存则跳过编译）。针对每个测试用例：配置沙箱并注入输入，在 CPU/内存限制下运行二进制文件，捕获输出，与预期结果比对。聚合判题结果（全部通过则为 AC，否则为 WA/TLE 等）。更新 Submission。触发用户档案及竞赛评分更新。通知用户。

面试官： 竞赛突发流量——每秒数千次提交。

候选人： 竞赛前预热判题集群。竞赛期间自动扩容。优先级队列：竞赛提交优先级高于练习。为竞赛设立专用 worker 池（避免与练习流量竞争）。队列吸收突发流量；用户体验轻微延迟但提交不丢失。监控队列深度；若队列过深，向用户展示降级的预计等待时间。

面试官： TLE——如何保证公平性？

候选人： 统一硬件：所有 worker 使用相同硬件、相同内核、相同操作系统。测量 CPU 时间（而非挂钟时间）以排除其他进程干扰。挂钟时间作为安全兜底。为每个沙箱绑定 CPU 核心以降低噪声。定期重跑参考解以验证 worker 校准情况。若某 worker 报告的时间与基准差异显著，则标记并下线。

面试官： 实时排行榜。

候选人： 基于每位用户的 ContestSubmissions 计算竞赛得分。每次产生判题结果后，异步聚合更新用户的竞赛得分。排行榜每隔数秒缓存并刷新一次。对于热门竞赛，采用子计数器分片存储得分。通过 WebSocket 或 SSE 向正在查看排行榜的客户端推送更新。

面试官： 作弊检测。

候选人： 异步进行代码相似度分析。对提交内容做 token 化处理，计算指纹，与同一题目的其他近期提交进行比对。使用类 Moss 风格的抄袭检测器。相似度超过阈值则标记为人工审核。行为信号：极快的解题速度、多用户间相同的时序模式、复制粘贴标记。竞赛结束后可对作弊者进行复查并取消资格。

面试官： 测试用例泄露防护怎么做？

候选人： 测试用例存储在受保护的服务中，仅判题 worker 可访问。Worker 按题目拉取测试用例并在本地缓存。测试用例不会出现在错误消息或判题结果响应中（否则用户可提取）。diff 输出做截断处理。用户仅可见样例测试用例（少量子集）。

面试官： 多语言支持。

候选人： 按语言配置：如何编译、如何运行、文件扩展名。判题 worker 预装所有语言工具链。不同语言设置不同的时间和内存限制（Python 比 C 慢，因此给予更多时间）。部分题目支持语言专属限制。

面试官： 多地区。

候选人： 各区域独立部署提交 API 和判题 worker。用户就近提交以降低延迟。题目测试用例全球同步复制（不可变，易于复制）。用户档案按分片存储。竞赛使用唯一的全局排行榜；来自任意区域的提交均汇入其中。轻微的跨区域复制延迟可接受。

面试官： 如果用户代码尝试逃逸沙箱怎么办？

候选人： 纵深防御。seccomp 限制系统调用，使代码无法执行任意二进制文件、ptrace 或修改网络状态。只读文件系统防止恶意文件写入。禁用网络。cgroups 防止 fork bomb。即使存在内核漏洞，攻击者也仅获得沙箱权限。审计日志记录逃逸尝试。对于高安全场景，Firecracker 在 hypervisor 层提供隔离。

面试官： 总结。

候选人： 提交队列 + 沙箱化判题集群 + 判题结果聚合器。每次提交在配置了 seccomp、cgroups、禁用网络、只读文件系统的容器中运行。按测试用例执行并严格限制资源。竞赛评分异步进行，热门竞赛采用子计数器分片。预热和优先级队列应对竞赛突发流量。作弊检测异步运行。多区域部署，全局竞赛状态同步。最难的安全问题是沙箱逃逸；纵深防御是答案。

3. 练习备注

五分钟版本

  1. 澄清：提交、沙箱、竞赛、语言支持。
  2. 不变式：沙箱化；确定性判题结果；资源受限。
  3. 提交队列 + 判题集群 + 判题结果聚合器。
  4. 沙箱：容器 + seccomp + cgroups + 只读文件系统，禁用网络。
  5. 每个测试用例设置资源限制。
  6. 竞赛评分异步进行；为突发流量预热。
  7. 作弊检测（代码相似度 + 行为信号）。
  8. 多区域部署，全局竞赛状态同步。

一分钟结尾陈述

代码判题系统是一个异步流水线：提交入队 → 沙箱化判题 worker → 判题结果 → 用户档案和竞赛评分更新。沙箱采用容器 + seccomp + cgroups + 只读文件系统 + 禁用网络；对不可信代码实施纵深防御。按测试用例设置资源限制。竞赛突发流量通过预热和优先级队列处理。作弊检测通过代码相似度异步运行。多区域部署降低延迟；全局竞赛状态同步。

最佳面试用语

  • "沙箱是纵深防御：seccomp、cgroups、禁用网络、只读文件系统。"
  • "通过 cgroups 实现每个测试用例的资源限制；测量 CPU 时间而非挂钟时间。"
  • "竞赛突发流量通过预热和优先级队列处理。"
  • "作弊检测是异步的——代码相似度加行为信号。"
  • "测试用例不会在错误消息中泄露；防止提取的防御措施。"
