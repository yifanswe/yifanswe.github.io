1. 基础知识与解题思路

本题考察点

Airbnb 是一个基于日期的稀缺库存双边市场：每个房源有独特的夜晚可供出售；一旦被预订，该夜晚就不能再次出售。难点在于按日期可用性进行地理搜索、跨房源原子性地预留日期范围、处理房东取消预订，以及管理款项支付。

其不变量如下：

对于每个房源，每个日历夜最多卖给一个已确认预订；搜索结果只反映可用的日期范围；预订仅在付款成功且房东接受（或适用即时预订）后才转换为已确认预订；取消预订以原子方式将日期释放回库存。

在 ICT5 / L6+ 级别，面试官测试的是你能否对日期范围库存进行建模（比基于座位的更复杂），设计带可用性过滤的地理空间搜索，处理经过付款的预订状态机，以及分析房东发起的取消和超额预订风险。

60 秒面试框架

口述版本：

我会将 Airbnb 设计为四个核心系统：带地理空间搜索的房源服务、拥有每房源日历夜晚的可用性服务、集成付款的预订状态机，以及房东与房客之间的消息系统。关键决策在于日期范围库存建模、多夜原子预订、房东流程与即时预订流程，以及带退款政策的取消处理。

首先明确需求

功能性需求：

  • 房东发布带照片、日历、定价和规则的房源。
  • 房客按位置、日期、人数和筛选条件进行搜索。
  • 预订：即时预订或申请预订。
  • 付款，按取消政策退款。
  • 房客与房东之间的消息沟通。
  • 入住后评价。
  • 房东取消预订及重新安排。

非功能性需求：

  • 不允许双重预订：一个夜晚最多属于一个预订。
  • 搜索结果在亚秒级内返回。
  • 浏览高可用；预订环节正确性至关重要。
  • 多区域（全球用户，区域性房源）。
  • 资金和日期所有权的可审计性。

以下内容超出范围，除非被特别询问：

  • 房源创建工具和验证。
  • 定价优化（智能定价）。
  • 房东超级房东计划。
  • 税收收取详情。

瓶颈在于日期范围库存：对"巴黎，7月15日至22日"的搜索必须快速按日期可用性过滤数百万房源。

带可用性过滤的地理空间搜索

核心搜索查询："找出位于 (lat, lng) 半径 R 范围内、在日期 [check_in, check_out] 可用、满足筛选条件的房源。"

解决方案：

  1. 地理空间过滤：目标位置周围的 H3/S2/geohash 单元格。
  2. 索引查找：这些单元格中的房源。
  3. 可用性过滤：对每个候选房源，检查 [check_in, check_out] 范围内的所有日期是否可用。
  4. 应用其他过滤条件（价格范围、入住人数、设施）。
  5. 按相关性、价格、评分和个性化推荐排序。

优化措施：

  • 在搜索索引中预计算可用性集合（例如，每个房源未来 365 天的 bitmap）。
  • 地理空间分片以实现并行过滤。
  • 缓存热门查询结果（如巴黎本周末）。

面试用语：

搜索必须同时按位置和日期可用性过滤。对每个房源的可用性进行 bitmap 编码，可在搜索时以 O(1) 的复杂度检查 N 晚的可用性。

预订状态机

requested -> accepted -> paid -> confirmed -> completed
   |             |          |          |
   +-> canceled  +-> declined +-> failed  +-> canceled

即时预订跳过"已接受"/"已拒绝"步骤。

状态转换：

状态转换 Trigger 影响
requested -> accepted 房东接受（请求预订）。 锁定日期。
requested -> declined 房东拒绝。 释放日期。
accepted -> paid 支付成功。 向房客收费。
paid -> confirmed 最终状态。 日期归预订所有。
Any -> canceled 取消。 按政策退款；释放日期。

每次状态转换均基于 (booking_id, from_state) 的条件 CAS。幂等操作。

原子性日期预留

对于预订 [check_in, check_out]：

  1. 预订服务启动。
  2. 对范围内每个日期执行锁定： - 对以下字段进行条件更新： ListingAvailability(listing_id, date): UPDATE status = held, reservation_id = ?, version = version + 1 WHERE status = available AND version = ?
  3. 若任意日期失败（已预订或已锁定），回滚所有锁定；返回不可用。
  4. 若全部成功，在以下位置创建预订记录： accepted （或 requested 若为 RTB）。
  5. 发起支付。
  6. 支付成功后，将日期状态从以下状态转换： held 与 booked.
  7. 支付失败或锁定超时时，将日期释放回以下状态： available.

锁定 TTL：即时预订较短（例如 15-30 分钟）；请求预订（房东有时间响应，通常 24 小时）较长。

面试用语：

多晚原子预留需要对每个日期行执行条件写入。首次失败即回滚；支付失败或超时时释放。该模式与 Ticketmaster 相同，区别在于用日期范围代替座位。

支付集成

复用支付系统设计：

  • 每次预订对应一个幂等键。
  • 锁定时授权；确认时（或某些策略下在入住时）扣款。
  • 按取消政策执行退款。
  • 入住后（或争议窗口期后）向房东付款。

超时情况：支付结果不明确 → 继续持有日期锁定；通过支付提供商进行对账。

取消政策

三种常见政策：

策略 退款
灵活 入住前 24 小时内可全额退款。
Moderate（中级） 入住前 5 天内可全额退款。
严格 入住前 7 天内退 50%；此后不退款。

实现方式：

  • 取消请求触发状态转换。
  • 退款金额根据政策和距入住时间计算。
  • 日期释放回以下状态： available.
  • 通过支付系统发放退款。

房东取消：

  • 惩罚更严重（房东需支付违约金，房源可能被降权）。
  • 房客获得全额退款及换房协助。
  • 日期释放。

搜索索引与刷新

搜索索引具有最终一致性。刷新策略：

  • 房源 CRUD 时：发布事件；索引器更新。
  • 预订时：发布可用性变更事件；索引器更新受影响日期。
  • 定期全量重建索引以补偿滞后。

滞后容忍：刚被预订的房源可能在搜索中短暂显示为可用。最终锁定尝试会捕获此情况。

扩展与分区

数据 分区键 原因
房源 listing_id 哈希值 按房源读写。
可用性 listing_id + 日期范围 与房源共置。
预订量 booking_id hash；按房客、房东索引 查找。
搜索索引 地理空间格元 本地查询模式。
评论 listing_id 哈希值 按房源聚合。
消息 conversation_id 哈希值 复用消息系统设计。

热门日期/热门地点（如 Coachella 周末）：预热搜索缓存；预期热门房源上存在竞争。

一致性模型

层级 一致性 原因
房源元数据 编辑时强一致性 房东期望立即生效。
可用性（预订写入） 强一致性（CAS） 不得出现重复预订。
搜索索引 最终一致 有界时效性。
预订状态 强一致性（CAS） 资金 + 日期。
评论 最终一致 有界。

多区域策略

  • 房源区域亲和性：通常与所在国家关联。
  • 可用性和预订写入路由至房源所在区域。
  • 搜索索引在全球复制（只读），以实现跨区域快速浏览。
  • 来自任意地区的房客均可预订任意房源；写操作路由回归属地。

故障模式与缓解措施

故障模式 缓解措施 监控指标
重复预订尝试。 对每个日期执行条件 CAS。 冲突率。
搜索显示可用但锁定失败。 最终锁定为权威依据；UI 显示错误。 锁定失败率。
支付超时。 锁定保留；通过提供商对账。 待支付账龄。
房东临时取消。 状态转换 + 换房流程；违约金。 房东取消率。
房客延迟取消。 按政策退款；释放日期。 延迟取消率。
日历同步（多平台房东）。 iCal 同步；冲突检测。 同步冲突。
搜索延迟峰值。 缓存热门查询；背压。 搜索 p99。
预订确认中途中断。 幂等重试；状态 CAS。 卡死预订数量。
欺诈预订。 风险评分；审核队列。 欺诈检测率。
地理数据有误。 人工核实；地图提供商。 地理争议数量。

运维指标

正确性：

  • 重复预订事件数（必须为零）。
  • 搜索与可用性不一致情况。
  • 支付丢失数量。

容量：

  • 按区域搜索 QPS。
  • 预订转化率。
  • 可用性写入速率。

业务层面：

  • 转化漏斗：浏览 → 点击 → 请求 → 确认。
  • 平均预订金额。
  • 房东取消率。
  • 评价提交率。

需要指出的权衡

  • 搜索可用性过滤以速度换时效性（bitmap 预计算相较新预订滞后数秒）。
  • 即时预订加快预留速度，但降低了房东的控制权。
  • 严格取消政策有利于房东；灵活政策有利于房客；平台决定默认策略。
  • 预热搜索缓存对热门查询有效；对长尾查询无效。
  • 地理分片在本地有效，但跨国搜索较为尴尬。
  • 较长的锁定 TTL 会降低库存利用率；较短的 TTL 则让响应慢的房东感到沮丧。

常见陷阱与更优做法

陷阱 更优做法
将每次预订视为独立处理，不考虑日期冲突。 每个日期执行原子条件 CAS。
搜索返回房源时不进行可用性过滤。 在搜索索引中预计算可用性 bitmap。
支付期间不锁定日期。 从请求到支付成功期间锁定日期。
取消操作视为删除。 含退款流程的状态转换。
同步等待房东接受。 异步，24 小时超时；自动拒绝。
单一全局数据库。 区域分片 + 全局搜索索引。

面试官可能深入探讨的方向

深入方向 优秀回答要点
日期范围库存 每晚一行；每日期执行 CAS；冲突时回滚。
含可用性的搜索 地理空间格元 + 可用性 bitmap。
预订流程 状态机；支付集成；幂等性。
取消政策 退款计算；日期释放；房东违约金。
多区域 按房源归属区域；全局搜索副本。
房东日历同步 iCal 同步；冲突检测。

Staff 级别回答检查清单

  • 日期范围库存采用每晚行 + CAS。
  • 含可用性过滤的地理空间搜索。
  • 含支付集成的预订状态机。
  • 取消政策对应状态转换 + 退款。
  • 搜索索引最终一致；最终锁定为权威依据。
  • 多区域，按房源归属区域划分。
  • 锁定 TTL；支付超时处理。

面试用语

Airbnb 是日期范围库存的 Ticketmaster。每个日历夜晚是一个库存单位；预订原子性地预留一段范围。预订状态机通过支付处理请求预订和即时预订路径。搜索按位置和日期可用性过滤；即使搜索结果显示为过期可用性，最终锁定仍为权威依据。

2. 模拟面试

面试官： 设计 Airbnb。

候选人： 我将设计四个核心系统：含地理空间搜索的房源服务、含每晚日历行的可用性服务、含支付集成的预订状态机，以及消息系统。不变量是每个日历夜晚最多售给一个已确认预订。搜索返回按位置和日期可用性过滤的房源；预订原子性地预留一段日期范围。

面试官： 如何存储可用性？

候选人： 每晚一行： (listing_id, date, status)。状态为可用、已锁定、已预订或已屏蔽。对于 5 晚预订，我们在事务中（或通过顺序 CAS + 回滚）将 5 行从以下状态条件更新： available 与 held 预订对范围内所有日期具有原子性。

面试官： 两位房客同时预订重叠日期。

候选人： 两者均尝试对相同日期行执行 CAS 更新，每个争用日期只有一方成功。如果房客 A 首先锁定第 N 晚，房客 B 的更新将失败。预订服务回滚房客 B 的所有部分锁定，并返回"不可用"及下一个可用的替代选项。失败的房客会立即看到错误提示；我们可能会推荐类似房源。

面试官： 搜索必须按日期可用性过滤，速度如何保证？

候选人： 预计算每个房源未来 365 天的可用性 bitmap，第 i 位为 1 表示第 i 天可用。要检查"7 月 15-22 日是否可用"，查找该 7 位切片并确认所有位均为 1。存储于搜索索引中。因此搜索 = 地理空间格元扫描 + 快速 bitmap AND。最终一致——预订提交后，bitmap 更新滞后数秒。最终锁定为权威检查依据。

面试官： 即时预订与请求预订的区别？

候选人： 状态机中的两条流程。即时预订：房客立即支付；支付成功后日期从已锁定转为已预订。请求预订：房客的请求创建锁定和待处理状态；房东有 24 小时接受或拒绝。接受后进行支付。拒绝或超时后锁定释放。状态机在入口处分叉，其余流程汇聚。

面试官： 支付超时怎么处理？

候选人： 与 Ticketmaster 相同。日期处于锁定状态，预订处于以下状态： pending_payment在确认支付结果之前，我们不释放锁定。对账 worker 查询支付提供商；成功则转为已确认；失败则释放日期。我们接受对账期间某晚被锁定，以避免重复扣款或重复预订的风险。

面试官： 取消政策？

候选人： 按房源设置：灵活、适中、严格。取消触发状态转换。退款金额根据政策和距入住时间计算。日期释放回可用状态。支付服务处理退款。对于房东发起的取消，房东需支付违约金，平台协助房客换房。

面试官： 搜索延迟——巴黎七月可能有数百万次查询。

候选人： 缓存热门查询模式：位置 + 日期 + 热门过滤条件组合。缓存 TTL 为秒到分钟级；可接受，因为我们有最终锁定作为权威依据。按位置格元进行地理空间分片，并行执行过滤。未认证浏览可由 CDN 承载。个性化排序在线进行但轻量。

面试官： 房东更新日历——屏蔽某一周。

候选人： 房东调用以下接口： PUT /listings/{id}/availability 传入日期范围和状态参数。 blocked我们将对应行从以下状态执行 CAS 更新： available 与 blocked如果某日期已处于以下状态： booked则返回错误；房东不得单方面取消已确认预订——必须走取消流程。搜索索引通过事件更新。

面试官： 多平台房东同步日历（Airbnb + VRBO）。

候选人： iCal 同步。房东可导入 iCal feed；我们定期轮询并屏蔽外部日历中标记的日期。双向同步将我们的日历导出到其他平台。冲突检测：如果同一日期在两个平台均被预订，则提醒房东。这是业内公认的难题。

面试官： 评价系统？

候选人： 住宿结束后发布。房客和房东互相评价。在双方均提交（或截止日期过后）之前，评价保持隐藏，以避免报复行为。评价汇总至房源评分和房东 Superhost 状态。聚合展示最终一致。

面试官： 多区域？

候选人： 房源按国家具有区域亲和性。可用性和预订写入路由至房源所在区域。搜索索引在全球复制，以便所有地区的房客快速搜索。跨国预订写操作命中房源归属区域——延迟可接受。

面试官： 总结。

候选人： 日期范围库存采用每晚行；预订使用原子 CAS。含可用性 bitmap 的地理空间搜索。预订状态机处理即时预订和请求预订。含超时处理的支付集成在对账期间保持锁定。按政策取消并退款。搜索最终一致；最终锁定为权威依据。按房源归属区域实现多区域部署。最难的正确性问题是杜绝重复预订；最难的搜索问题是大规模日期可用性过滤。

3. 练习备注

五分钟版本

  1. 澄清：规模、搜索、预订、取消。
  2. 不变量：每晚最多售给一个预订。
  3. 房源 + 可用性采用每晚行。
  4. 含可用性 bitmap 的地理空间搜索。
  5. 预订状态机：已请求 → 已接受 → 已支付 → 已确认。
  6. 原子多晚 CAS 预留。
  7. 按政策取消；退款 + 日期释放。
  8. 多区域，按房源归属区域；全局搜索索引。

一分钟结尾陈述

Airbnb 是日期范围库存的 Ticketmaster。每晚行支持原子 CAS 预留。预计算可用性 bitmap 的地理空间搜索可按位置和日期快速过滤。预订状态机从请求流转至支付再到确认；即时预订和请求预订是两条分支。取消政策映射为退款公式加日期释放。搜索最终一致；最终锁定为权威依据。按房源归属区域实现多区域部署，配备全局搜索副本。

最佳面试用语

  • "每晚是一个库存单位；预订操作以原子方式保留一段日期区间。"
  • "搜索使用可用性位图进行快速日期过滤；最终持有状态具有权威性。"
  • "预订是一个状态机；即时预订与申请预订是其中的两条路径。"
  • "支付超时后保留持有状态并进行对账；切勿基于猜测释放稀缺库存。"
  • "取消是一次状态转换加退款流水线，而非删除操作。"
