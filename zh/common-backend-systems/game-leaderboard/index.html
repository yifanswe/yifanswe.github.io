Game Leaderboard（演练）

2026/06/01 · Common Backend Systems
目录
目录

难度： Medium · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建一个多人游戏排行榜。玩家提交分数；系统展示全球前100名、玩家自身排名以及玩家周边排名（前后各5名）。5000万月活跃玩家，周末锦标赛期间写入峰值约10万次/秒。支持多个范围的排行榜：全球、仅好友、地区、时间窗口（日榜/周榜/全时段榜）。

1. 需求（约 5 分钟）

功能需求 - 玩家提交分数；系统记录并更新排行榜。- 查询全球/地区/玩家好友的前K名。- 查询玩家自身排名及其N个相邻玩家。

我会推迟深入设计作弊检测和超越简单最高分获胜规则之外的排名并列规则。

非功能需求 - 前K名查询 p99 < 100 ms。- 玩家排名查询 p99 < 200 ms。- 全球排名最终一致性（< 5 秒）可接受；玩家自身排名应接近实时。- 能应对周末锦标赛期间的写入峰值。

容量估算 - 5000万玩家；日榜约500万活跃；周榜约1500万；全时段榜约5000万条目。- 写入峰值10万次/秒。- Redis 内存可管理但并非免费：日榜500万 × 约100 B × RF=2 约1 GB；周榜1500万约3 GB；全时段榜5000万约10 GB。长期主要成本是分数历史，而非前K名索引。

2. 核心实体（约 2 分钟）

  • 播放器 — player_id、region、friends。
  • Score — (player_id, scope, score, ts)。Scope = daily, weekly, all, region:X等。
  • 排行榜 ——派生值，按分数索引。

3. API（约 5 分钟）

POST  /v1/scores                     { score, mode }                          → { rank, percentile }
GET   /v1/leaderboards/{scope}?top=100                                        → { entries[] }
GET   /v1/leaderboards/{scope}/me                                             → { rank, score, neighbors[] }
GET   /v1/leaderboards/friends/me                                             → { friends_ranked[] }

从令牌中获取 User。

排名邻居游标由服务器签发并编码 {scope, player_id, rank_snapshot_version, offset} ，这样即使有新分数到来，在玩家周围翻页时也不会漂移。

5. 高层设计（约 10–15 分钟）

  1. 分数写入。 Score POST → 验证（基本合法性检查、反作弊钩子）→ 写入 scores 表（按 player_id分片）→ 发布到 Kafka 供下游消费。
  2. 排行榜存储：每个 scope 对应一个 Redis 有序集合。 ZADD leaderboard:daily:2026-05-18 score player_id。每个 scope 对应一个 ZSET。 ZADD GT 只保留玩家的最高分。
  3. 前K名查询。 ZREVRANGE leaderboard:daily 0 99 WITHSCORES ——O(K+log N)。热门 scope 短暂缓存（1秒）。
  4. 玩家排名查询。 ZREVRANK leaderboard:daily player:42 ——O(log N)。通过在该排名前后执行 ZREVRANGE 获取相邻玩家。
  5. 好友排行榜。 玩家的好友列表与全局 ZSET 取交集；通过一次 ZMSCORE 调用批量获取每位好友的分数，然后在客户端排序。
  6. 多 scope 管理。 日榜/周榜在边界时刻滚动——当天结束后丢弃昨日日榜。全时段榜持续增长；定期裁剪至前N名加一定缓冲。
  7. 持久性。 MySQL（按 player_id分片）中保存权威分数——Redis 是索引。Redis 丢失时，从 MySQL 重建。
  8. 地区 scope。 每个地区对应一个独立 ZSET（leaderboard:region:us-west）。基数更低，集合更小，查询更快。

端到端叙述：Alice 提交9500分 → POST 写入分数行， ZADD GT leaderboard:daily:2026-05-18 9500 alice, ZADD GT leaderboard:all 9500 alice。服务返回 Alice 的新排名（ZREVRANK leaderboard:daily alice → 排名第5，ZREVRANGE 0 9 返回前10名）。好友排行榜：拉取 Alice 的好友列表 [bob, carol, ...] → ZMSCORE leaderboard:daily alice bob carol ... → 降序排序 → 渲染。

6. 深度探讨（约 10 分钟）

为何选择 Redis 有序集合。 ZSET 操作是 O(log N)，常数极小——非常适合排行榜。范围查询（ZREVRANGE）和排名查询（ZREVRANK）在亚毫秒级完成。内存成本：约80 B/条目，5000万玩家 × 1个排行榜 ≈ 4 GB。集群按排行榜键分片。

为何不选择纯 MySQL 或纯 Cassandra：它们可以存储权威分数，但排名和邻居查询在每次写入时都需要对有序集合进行操作。Redis ZSET 是读取索引；MySQL 是数据真实来源。如果面试官追问 Redis 持久性，答案是可重建性加上 RF/AOF，而不是将 Redis 视为唯一副本。

需要当场说出的容量估算：日榜 + 周榜 + 全时段榜 + 10个地区排行榜合计约 (5M + 15M + 50M + 50M) × 100 B × RF=2，分配器开销前约24 GB Redis 逻辑排行榜数据。MySQL 分数历史更大：10万/秒的写入峰值是突发性的，但即使每年10亿条分数行 × 80 B × RF=3 也有约240 GB（不含索引）。可调节的旋钮是：压缩低价值时间窗口、赛季结束后归档原始行、全时段榜仅对头部排名保持精确，其余提供玩家自有查询。

扩展单个热点排行榜。 全球日榜承受所有10万次写入/秒。单个 Redis 实例上限约10万次操作/秒。缓解措施：- 对排行榜分片 ，按 player_id % N拆分为N个子 ZSET。前K名读取跨分片进行归并排序。- 写入批处理：在写入层积累100 ms 的写入，批量合并为一次 Lua 流水线调用。降低 RTT 成本；总工作量不变但减少往返次数。

分片排行榜的前K名查询。 每个分片返回其本地前K名。归并排序得出全局前K名。与 YouTube Top-K 同样的技巧（需要本地 K' > K 以保证正确性）。

前K名读取路径：client -> 排行榜 API -> 前K名缓存查找；未命中时，并行执行 ZREVRANGE lb:{scope}:{window}:{0..N-1} 0 K' WITHSCORES；API 归并排序，在一次批量调用中补全玩家 profile，写入 top:{scope}:{window} 缓存1秒后返回。

Top-K cache miss
    API
     |-- shard 0: ZREVRANGE 0..K'
     |-- shard 1: ZREVRANGE 0..K'
     |-- shard N: ZREVRANGE 0..K'
     '-- profile MGET for winners
                -> merge by score/tie_ts -> top:{scope}:{window} TTL 1s -> client

这是一个双视图设计：分数行是内容， lb:{scope}:{window}:{shard} 是排名索引， top:{scope}:{window} 是读取快照。它们派生自同一个分数事件，因此服务可以从分数历史重建所有视图。

从分片排行榜查询任意玩家的排名。 更难处理。 ZREVRANK player_42 in shard 5 只能给出本地排名。要获取全局排名，需要对所有分片求 ZCOUNT score_above 之和。以单次 Lua 流水线调用实现；仍为 O(N_shards × log N)。

排名读取路径：查找 player_id % N 用于本地分数，通过以下方式获取分数 ZSCORE，然后并行 ZCOUNT shard (score,+inf) 跨所有分片。邻近行来自本地分片加上相邻分数段的候选范围，然后 API 在全局排名附近进行归并排序。

排名游标存储 {scope, window, player_id, score_watermark, shard_offsets[]}。如果用户滚动邻近记录时有新分数进来，游标会将页面锚定在快照分数上，而不是随之漂移。

每日/每周滚动。 每日 ZSET 键包含日期；在午夜时开始使用新键；旧键保留 24 小时供读取访问，之后归档至 S3。每周聚合来自每日数据——分数取一周内最大值，存储于 leaderboard:weekly:2026-W20.

全时段排行榜大小。 5000 万条目 × 80 B = 4 GB Redis 内存，可管理。裁剪至前 10k 加每位玩家最佳成绩（玩家也需要知道自己在末尾的排名）。采用混合方案：前 10k 存储在 Redis 中，其余所有玩家存储在分片 MySQL 索引中用于长尾部分。

长尾的状态转换：

阶段 模式 数据源 延迟预算 游标/状态
热门排名 精确 Redis 分片 ZSET + ZCOUNT <200 ms score_watermark
尾部排名 估算百分位 MySQL 分数直方图 <300 ms score_bucket
精确异步 后台重新计算 MySQL 有序索引 秒级 内部返回的 job id
已重建 新鲜的 Redis 视图 重放分数历史 故障后数分钟 版本化窗口

好友排行榜的扩展。 拥有 200 位好友的玩家 → ZMSCORE 一次往返中获取 200 个 player_id 的分数。客户端排序 200 条记录。速度快（< 10 ms）。

防作弊/验证。 服务端分数验证：来自游戏客户端的签名分数、合理性检查（不可能出现的分数模式）、每个玩家的提交速率限制。作弊者的记录被隔离（单独的 ZSET）而非删除。

回放/灾难恢复。 Redis ZSET 是索引；权威数据存储是 MySQL。在 Redis 集群丢失时，通过重放全时段和近期分数从 MySQL 重建。

优雅降级。

情形 模式 数据源 用户可见行为
全局 ZSET 热数据 批量写入，提供缓存的前 100 名 Redis 分片 + 1 秒缓存 top-K 可能滞后 1-2 秒
玩家在裁剪后的全时段尾部 估算百分位，异步获取精确排名 MySQL 分数索引 先显示百分位
Redis 分片丢失 冻结受影响范围，从分数历史重建 MySQL 回放 数据陈旧但 top-K 仍可用
好友列表过大 限制为活跃好友，分页处理 好友图 + ZMSCORE 页面内精确
赛事峰值 竞赛队列与休闲模式隔离 专用 Redis/Kafka 池 休闲排名有延迟

面试要点："排行榜本身是一个二级索引；持久化的事实是玩家的最佳分数，因此丢失 Redis 只影响数据新鲜度，不影响正确性。"
