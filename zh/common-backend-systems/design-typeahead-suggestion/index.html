1. 基础知识与解题思路

本题考察点

Typeahead 建议是按键驱动的补全功能：用户输入一个前缀，系统根据热度和个性化返回最可能的补全结果。难点在于：每次按键的延迟低于 50ms 且 QPS 极高、从查询日志持续刷新热度，以及拼写容错。

其不变量如下：

对于每次按键，系统在数十毫秒内返回按热度、时效性和个性化综合排序的相关补全结果；建议语料库在有界过期时间内反映近期查询趋势；拼写错误也能返回相关结果。

在 ICT5 / L6+ 级别，面试官考察的是：你能否清晰阐述前缀索引（trie 或倒排索引）、每个节点预计算的 top-N、数据新鲜度 pipeline、个性化、模糊匹配，并能推理大规模场景下的延迟与多语言部署。

60 秒面试框架

口述版本：

我会将 typeahead 设计为一个前缀索引结构（trie 或倒排前缀索引），作为低延迟建议服务的后端。每个 trie 节点缓存其 top-N 补全结果以加快查找。后台 pipeline 通过批量处理查询日志更新热度，并通过流处理捕捉趋势。个性化在边缘节点完成重排序。关键决策包括：索引数据结构、数据新鲜度 pipeline、模糊匹配，以及大规模场景下的延迟。

功能需求

功能性需求：

  • 用户输入前缀，建议随即呈现。
  • 排名靠前的 5-10 条补全结果。
  • 容忍拼写错误。
  • 个性化。
  • 多语言支持。

非功能性需求：

  • p99 延迟低于 50ms。
  • 全球百万级 QPS。
  • 每小时趋势更新，加上实时流处理。
  • 多地区。

领域 假设

数据模型

建议

Suggestion(
  text,
  popularity_score,
  category,
  language,
  is_editorial,
  last_updated
)

Trie 节点

TrieNode(
  prefix,
  top_n: [(text, score)]      // pre-computed for fast lookup
)

API

GET /suggest?q=&lang=&user_token=
  -> [{text, score, highlight}, ...]

查找路径

  1. 客户端发出按键事件。
  2. 边缘缓存命中则立即返回。
  3. 缓存未命中时，建议服务查询前缀索引。
  4. 逐字符遍历 trie；返回匹配节点的 top-N。
  5. 如果存在 user_token，则应用个性化重排序。
  6. 返回结果。

延迟：总计低于 50ms。

前缀索引设计

带预计算 top-N 的 trie：

  • 每个节点表示一个前缀。
  • 每个节点存储该子树的 top-N 补全结果。
  • 查找：遍历前缀；返回该节点的 top-N。
  • 时间复杂度 O(prefix_length)。

更新时重建受影响的祖先节点：

  • 新增或更新建议的前缀路径上的节点刷新其 top-N。
  • 由聚合 pipeline 完成。

压缩（PATRICIA / radix）以节省内存：

  • 合并常见的单子节点链。
  • 减少内存占用。

新鲜度 Pipeline

每日批处理：从日志中聚合近期窗口内的查询计数。更新热度分数。重新生成每个 trie 节点的 top-N。

流处理：为新兴趋势提供实时信号。流处理层将分数增量应用到 trie。热门词可在数分钟内浮现。

个性化

轻量级、边缘缓存的每用户向量。

  • 特征：最近查询、地区、点击记录。
  • 混合 personal_affinity_score 与 popularity_score.
  • 延迟上限：几毫秒。

隐私：遵守退出选项；匿名化处理。

模糊匹配

当精确前缀置信度较低时：生成编辑距离为 1 或 2 的候选项。逐一查找；合并结果。速度较慢；作为兜底方案。

或者：使用 n-gram 索引，每条建议按 n-gram 建立索引；通过查询 n-gram 重叠度来寻找近似匹配。

编辑覆盖

对于突发事件，手动覆盖具有最高优先级。编辑层在最终排序前应用。

扩展与分区

关注点 策略
前缀索引 按前缀前 1-2 个字符分片
QPS 无状态副本 + CDN
查找： 压缩 trie；限制每节点 top-N 数量
热门前缀（"a"） 可被 CDN 缓存
多语言 每种语言独立索引

一致性模型

层级 一致性
热门内容 最终一致
个性化 最终一致
索引查找 副本内强一致性
编辑 强一致性

多区域策略

  • 索引复制到每个区域。
  • 每个地区的边缘缓存。
  • 每用户的地区个性化缓存。
  • 来自中央 pipeline 的更新。

故障模式与缓解措施

故障模式 缓解措施 监控指标
索引加载缓慢。 预热缓存。 副本就绪状态。
服务超时。 严格超时；降级兜底。 p99。
趋势数据过期。 流处理层。 趋势延迟。
个性化服务下线。 跳过个性化。 个性化可用性。
垃圾建议。 过滤；人工审核。 垃圾举报。
拼写错误未解决。 模糊兜底。 模糊命中率。
热门前缀过载。 CDN 缓存。 热门前缀 QPS 峰值。
恶意操控。 异常检测。 突发趋势信号。
地区数据分歧。 集中式索引。 不一致数量。
多语言混淆。 地区路由。 跨 locale 结果。

运维指标

正确性：

  • Top-N 质量。
  • 趋势新鲜度延迟。

容量：

  • 按区域统计的 QPS。
  • p99 延迟。
  • 索引内存占用。

业务层面：

  • 点击率（Click-through rate）。
  • 建议转化率。

需要指出的权衡

  • Trie 结构紧凑；倒排索引更新更容易。
  • 预计算 top-N 查找速度快，但更新时需要重建。
  • 个性化会增加延迟。
  • 模糊匹配较慢；仅作兜底。
  • 语言越多，内存占用越大。

常见陷阱与更优做法

陷阱 更优做法
每次按键都进行 DB 查询。 内存存储；亚毫秒级查找。
每次请求计算 top-N。 预计算。
同步个性化。 边缘缓存。
不处理拼写错误。 模糊或 n-gram 兜底。
单一全局索引。 分片；多语言支持。
静态建议。 持续保持数据新鲜度。

面试官可能深入探讨的方向

深入方向 优秀回答要点
Trie 设计 压缩；预计算 top-N。
新鲜度 批量 + 流式处理。
个性化 边缘缓存混合。
模糊 编辑距离 / n-gram。

多区域 索引已复制。

Staff 级别回答检查清单

  • 内存内前缀索引（Trie 或倒排索引）。
  • 每节点预计算 top-N。
  • CDN 前置的建议服务。
  • 批处理 + 流处理热度 pipeline。
  • 边缘缓存个性化。
  • 模糊兜底。
  • 分片且多副本。

面试用语

Typeahead 是亚 50ms 的前缀查找，配合预计算的 top-N 建议。内存中的 trie 或倒排索引；无状态建议服务以 CDN 为前置。批处理 + 流处理热度 pipeline 保持数据新鲜。个性化在边缘节点混合。模糊兜底应对拼写错误。最难的约束是每次按键的高 QPS 延迟要求。

2. 模拟面试

面试官： 设计 Typeahead 自动补全。

候选人： 亚 50ms 每次按键建议，由内存前缀索引（带预计算 top-N 的压缩 trie）提供支撑。CDN 缓存热门前缀。后台 pipeline 通过批处理 + 流处理聚合查询日志更新热度。个性化在边缘节点混合。模糊兜底应对拼写错误。不变量是每次按键都能快速、相关、新鲜地给出建议。

面试官： 为什么要预计算 top-N？

候选人： 查找延迟。在每次查询时遍历子树进行评分和排序，其代价随子树大小线性增长——对于下方有数百万补全结果的短前缀来说代价极高。对每个节点预计算 top-N 可使查找复杂度为 O(prefix_length) 加上读取一个小列表。

面试官： 如何更新？

候选人： 批量聚合每晚运行：扫描近期查询，更新热度分数，重新生成每个 trie 节点的 top-N。流处理层为突发趋势应用实时增量。因此，一个新的病毒式查询可在数分钟内通过流处理路径浮现。

面试官： 个性化？

候选人： 边缘缓存的每用户向量——最近查询、地区、偏好。将个人兴趣分数与热度分数混合。延迟上限为几毫秒。隐私：遵守退出选项，匿名化处理。

面试官： 拼写错误——输入 "appl" 而非 "apple"。

候选人： 精确前缀查找返回 "appl" 的所有结果。若结果稀少，则触发模糊兜底：生成编辑距离为 1 的候选项（"apple"），逐一查找，合并结果。速度较慢；仅在需要时触发。替代方案：n-gram 索引。

面试官： 热门前缀 "a" 达到 100k QPS。

候选人： CDN 承载大部分流量。"a" 的 top-N 较小且全球稳定；积极缓存。源站仅处理缓存未命中。预热热门前缀。

面试官： 多语言支持。

候选人： 每种语言独立索引。通过请求中的地区信息决定路由。部分共享前缀（专有名词）通过在两种语言中都建立索引来处理。

面试官： 趋势——多快？

候选人： 流处理层监控每个词语的查询速度。当速度激增时，热度分数提升。流处理在数秒到数分钟内将增量应用到 trie。对已知事件提供编辑覆盖。

面试官： 多地区。

候选人： 索引从中央 pipeline 复制到每个地区。每个地区部署边缘 CDN。每个地区维护个性化缓存。若本地降级，则故障转移到下一个地区（使用过期数据）。

面试官： 总结。

候选人： 内存前缀索引——带预计算 top-N 的 trie——通过以 CDN 为前置的无状态建议服务实现亚 50ms 响应。后台 pipeline 更新热度（批处理 + 流处理）。个性化在边缘节点混合。模糊兜底应对拼写错误。按语言和地区进行复制。最难的性能约束是每次按键的延迟。

3. 练习备注

五分钟版本

  1. 澄清：延迟、新鲜度、拼写错误。
  2. 不变量：50ms 以内；相关；新鲜。
  3. 内存 trie，带预计算 top-N。
  4. CDN + 无状态建议服务。
  5. 批处理 + 流处理热度 pipeline。
  6. 边缘个性化；模糊回退。
  7. 按语言和地区分区。
  8. CTR + 延迟指标。

一分钟结尾陈述

Typeahead 是亚 50ms 的前缀查找。每节点预计算 top-N 的内存 trie 使查找复杂度为 O(prefix_length)。以 CDN 为后盾的无状态建议服务承载热门前缀。后台 pipeline 通过批处理 + 流处理从查询日志更新热度以捕捉突发趋势。边缘节点个性化。模糊兜底。按语言和地区进行复制。

最佳面试用语

  • "每个 trie 节点预计算 top-N 使查找复杂度为 O(prefix_length)。"
  • "CDN 承载热门前缀；建议服务处理长尾请求。"
  • "批处理 + 流处理保持索引新鲜；趋势可在数分钟内浮现。"
  • "个性化在边缘混合，设有严格的延迟上限。"
  • "仅当精确匹配置信度较低时，才触发模糊兜底。"
