WhatsApp — Messaging (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Medium · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建 WhatsApp。20 亿用户每天发送 1000 亿条消息。一对一聊天和群聊，端到端加密，支持消息送达和已读回执，适用于不稳定的移动网络和离线设备。每位用户可使用多台设备。

1. 需求（约 5 分钟）

功能需求 - 在一对一聊天中发送文本消息。 - 在群组中发送消息（最多约 1000 名成员）。 - 查看送达和已读回执（✓ 已发送，✓✓ 已送达，蓝色 ✓✓ 已读）。 - 离线时接收消息，设备唤醒后查看。

我会暂不考虑语音/视频通话及媒体存储的深入设计。

非功能需求 - 端到端加密——服务端仅存储密文。 - 送达保证为精确一次 每位接收方 语义上（用户不会看到重复消息）。 - 每段对话的消息顺序保持一致。 - p99 消息从发送到接收方到达时间 < 1 秒（网络正常时）。 - 支持离线；重连后送达。 - 每用户多设备（手机 + Web + 桌面）——消息必须到达所有设备。

容量估算 - 每天 1000 亿条消息 → 平均约 120 万条/秒，峰值 300 万至 500 万条/秒。每条消息约 500 字节密文。 - 20 亿用户，假设每人 3 台设备 → 60 亿个活跃会话令牌。 - 密文体量：每天 1000 亿条 × 500 B ≈ 50 TB/天（复制前）。若离线队列保留 30 天，约 1.5 PB 主数据，3 副本则约 4.5 PB；离线队列是主要存储成本。 - 连接映射小得多：6 亿在线设备 × ~64 B 映射 ≈ 38 GB 主数据，但需要极低延迟的查询和持续的抖动处理。

2. 核心实体（约 2 分钟）

  • User — user_id，phone_number，devices。
  • 设备 — device_id，public key，last seen。
  • ：双方独立发送消息，顺序交错。任意一方均可结束其半侧的流。 ——一对一（隐式，由用户对派生）或群组（chat_id，members）。
  • 消息 — msg_id, chat_id, sender_device_id, ciphertext, ts, server_ts.
  • DeliveryReceipt — msg_id, recipient_device_id, state (sent/delivered/read).

3. API / 协议（约 5 分钟）

主要是在持久连接上运行的自定义协议。概要如下：

WS  /v1/connect                   // long-lived, auth via token + device_id
  → server pushes: { msg, receipt, presence }
  ← client sends:  { send_msg, ack_received, mark_read }

HTTP fallback for media:
POST /v1/media/upload             → { media_id }
GET  /v1/media/{id}               → bytes

每条消息携带客户端生成的 UUID 用于幂等性。

重连游标由服务端按设备签发，编码最后一次已确认的每设备队列序列号；客户端不请求任意偏移量。

5. 高层设计（约 10–15 分钟）

  1. 边缘连接服务器。 长连接 WebSocket（或类 XMPP）终端，按 device_id 分片。设备"在线"当且仅当某连接服务器持有其 WS。连接服务器到设备的映射保存在 Redis 集群中。
  2. 路由服务。 当用户 A 的连接服务器收到发送请求时，它查询每位接收方的 device_id → connection_server 并转发。如果设备离线，则将消息写入其 离线队列.
  3. 消息存储 / 离线队列。 每设备 FIFO 队列（每分片一个 Kafka topic，或分片的 Cassandra/HBase 表），存储设备尚未 ACK 的消息。重连时，设备清空其队列。
  4. 群组扇出服务。 对于群组，展开 chat_id → member device_ids，为每台设备入队一条记录。服务端不解密——只路由密文。
  5. 密钥目录。 每设备公钥（Signal 风格的身份密钥 + 已签名预密钥 + 一次性预密钥）。发送方拉取密钥以建立会话。
  6. 媒体服务。 端到端加密的媒体以加密字节上传；聊天消息在加密载荷中携带媒体 URL + 解密密钥。
  7. 在线状态 / 回执。 回执更新通过同一连接回传—— delivered 设备收到消息时， read 用户查看时。发送方客户端通过相同路由路径更新。

端到端叙述：Alice 的手机通过其 WS 发送 {msg_id: U1, chat: A-B, ciphertext} → 她的连接服务器查询 Bob 的设备列表和连接服务器 → Bob 的手机在线（一台连接服务器），Bob 的 Web 离线。转发至手机连接服务器并送达；手机返回 delivered 回执。Web 设备的离线队列收到消息密文；3 小时后 Web 重连时清空队列。Bob 的手机显示消息；点击聊天界面发送 read 回执，经路由服务送回 Alice 的设备。

6. 深度探讨（约 10 分钟）

具体存储契约。

存储 键/数据结构 条目大小 + TTL 分片键
连接映射 conn:{device_id} → {edge_id, session_id} in Redis 约 64 B，TTL 心跳 60 秒 device_id
离线收件箱 宽列 inbox(device_id, seq) → 密文信封 约 600 B/消息，TTL 30 天或直至 ACK device_id
Dedupe dedupe:{device_id}:{msg_uuid} 约 32 B，TTL 30 天 device_id
群组成员关系 group:{chat_id} → 成员 user/device id + 版本 10 KB–1 MB，持久保留 chat_id
密钥目录 keys:{device_id} identity/signed/one-time prekeys 约 1–5 KB，持久保留/轮换 device_id
回执 receipt:{msg_id}:{device_id} 状态 + 时间戳 约 32 B，TTL 或压缩 msg_id

各存储容量估算。 离线收件箱是主要存储成本： 100B messages/day × 600 B 主数据约 60 TB/天，若每条消息都等待 TTL 到期，30 天 3 副本约 5.4 PB。实际中 ACK 压缩会提前删除大多数条目，但设计必须为移动设备长时间离线预留预算。连接映射体量大但可控： 600M online devices × 64 B × 3 约 115 GB。去重不能简单地将每条消息 UUID 在内存中保留 30 天；对已送达消息保持短时热去重，对未送达设备的去重与收件箱保留期对齐。群组成员关系读热但与收件箱字节量相比很小；回执流是追加/压缩模式，不是每条消息永久保留一行。

发送路径追踪。 Alice 的边缘服务器收到 {msg_uuid, chat_id, ciphertexts} → 对 Alice 设备进行幂等性检查 → 聊天服务读取接收方设备/群组成员版本 → 路由并行查询在线设备的 conn:{device_id} → 在线设备通过 WS 推送 → 离线设备追加到 inbox(device_id, seq) → 每台接收设备 ACK seq → 回执事件路由回 Alice 的设备。群组扇出按接收方设备分片并行化；消息排序基于 chat_id 序列，而存储送达为每设备 FIFO。

热路径与冷路径协议。

路径 Hot path Cold/fallback path 游标/会话状态
一对一发送 边缘 → 路由 → 并行 conn:{device_id} 查询 → 在线推送；离线追加 若在线路由不确定，先追加到收件箱，再推送以避免丢失 客户端消息 UUID，每对话序列号
设备重连 边缘认证 → 读取游标 → 清空队列 inbox(device_id, seq > cursor) 休眠超过 TTL 的设备无排队消息；可选加密历史同步处理支持的场景 已签名游标 {device_id,last_seq,inbox_epoch}
群组发送 读取成员版本 → 按接收方设备分片扇出 → 逐步入队/推送 大型/热点群组先接受发送再异步扇出；发送方看到已发送早于全部送达 群组版本，扇出任务 id
回执更新 设备 ACK/已读 → 回执事件 → 发送方设备 若发送方离线，回执进入发送方设备的收件箱 每...的回执状态版本 (msg_id,device_id)

多设备扇出。 拥有 N 台设备的用户意味着每条消息有 N 份副本。发送方使用 Signal Double-Ratchet 协议（群组使用 Sender Keys）为每台接收方设备独立加密。服务端将每台 (message, device) 视为独立的路由目标。发送方自己的设备也会收到副本（从而保持手机和 Web 同步）。

群组消息避免二次方级别的膨胀。 对于 1000 人群组，朴素的每设备加密需要发送方上传 1000+ 份密文。Signal 的 Sender Keys：发送方使用对称链密钥对消息加密一次，在需要时（成员变更或初始化时）带外分发链密钥给每位接收方。权衡：前向保密性弱于每消息密钥，但使千人群组变得可行。

每对话消息排序。 服务端在收到时分配单调递增的每对话序列号（时间戳 + 决胜项）。接收方设备按此排序。发送方客户端时间戳显示出来，但服务端序列号才是真相来源。

双索引框架。 消息顺序按对话键控，而送达按设备键控。服务端分配每对话序列号，确保所有参与者看到相同的对话顺序，然后写入每设备收件箱条目，使每部手机/Web 客户端可以独立重连。这是对加密信封的两个索引，而非两个相互竞争的消息存储。

离线送达。 每设备离线队列是合约。设有 TTL 上限（如 30 天）。重连时，设备拉取自最后一个已 ACK 偏移量以来的所有消息。设备 ACK 送达后，服务端从队列中删除。

为何使用宽列收件箱而非每设备 Kafka：Kafka 非常适合服务间流，但 60 亿台设备需要随机的每设备重放、TTL 和独立的 ACK 偏移量。以 device_id 为键的 Cassandra/HBase 风格收件箱使重连成本低廉。Kafka 仍然适合内部回执和分析流。

与纯 Redis 队列相比，宽列收件箱在长时间离线窗口和区域重启后无需在内存中保存 PB 级数据。与关系型表相比，访问模式几乎始终是 device_id + increasing seq，因此宽列存储使写入和重连扫描具有可预测性。

至多一次 vs 至少一次。 服务端保证至少一次送达；客户端通过 msg_id （发送方生成的 UUID）去重。净效果：用户侧精确一次展示。

连接扩展。 20 亿设备 × 30% 在线 = 6 亿并发 WS 连接。每台连接服务器持有约 50 万连接；仅边缘层就需要 1200 台以上机器。使用高效异步 I/O（epoll/kqueue）并最小化每连接内存占用。

端到端加密——服务端的角色。 服务端永远看不到明文。服务端存储 (ciphertext, recipient_device_id)。回执元数据（已送达/已读标志）在服务端是明文。需承认滥用内容审核因此依赖客户端举报和元数据信号。

面试要点："服务端是一个持久化的加密路由器；它拥有送达语义，而非消息含义。"

热分片缓解。 知名用户广播列表。限制广播列表大小；按发送方限速；对企业用户使用单独的 API 层级。
