1. 基础知识与解题思路

本题考察点

Facebook 信息流是个性化社交信息流的经典案例：数十亿用户、复杂的关系（好友、关注、群组、主页），结合机器学习的重度排名，以及在不到一秒内展示新鲜相关内容的要求。

其不变量如下：

每篇帖子都被持久化存储并标注归属；每位用户的主页信息流最终包含其所有关联来源的帖子（受排名、可见性、拉黑和策略约束）；信息流以亚秒级 p99 延迟读取已排序内容；删除、拉黑和策略决定在数秒内传播并在读取时强制执行。

在 ICT5 / L6+ 级别，面试官在考察你能否为比 Twitter 更复杂的关系图谱设计混合扇出（好友是双向的，还有主页、群组、推荐），带有特征存储的重度 ML 排名、整合互动信号，以及大规模推理不良内容审核。

60 秒面试框架

口述版本：

我会将 Facebook 信息流设计为一个信息流生成流水线：从用户所关联的好友、主页和群组中汇总候选帖子，用 ML 排名器对其评分，应用可见性过滤器后返回。帖子的权威存储与信息流投影分离。扇出方式类似 Twitter 的混合模式，但因为关系包含主页和群组而更加精细。需要对排名和特征流水线进行大量投入。有趣的决策在于候选生成、排名架构、特征时效性和内容审核。

首先明确需求

功能性需求：

  • 用户拥有好友（双向）、关注主页、加入群组。
  • 来自好友、主页、群组的帖子出现在主页信息流中。
  • 帖子可以是文字、图片、视频、链接、签到。
  • 点赞、评论、分享、表情回应。
  • 按相关性排序，而非严格按时间顺序。
  • 可见性过滤器：拉黑、静音、隐藏帖子。
  • 动态（Stories）单独展示（类似 Instagram）。

非功能性需求：

  • MAU 超过 30 亿。
  • 信息流读取 p99 < 500 毫秒。
  • 有界时效性（秒到分钟级）。
  • ML 推理延迟低于 200 毫秒的重度排名。
  • 多地区。

以下内容超出范围，除非被特别询问：

  • Marketplace。
  • Watch（视频）。
  • Dating、游戏。
  • 广告（视为单独插入处理）。

规模假设

领域 假设
用户 MAU 30 亿。
好友平均数量 300-500.
已关注主页/群组 50-200.
每日帖子数 数十亿。
信息流读取数 每日数千亿次。
延迟预算 500 毫秒 p99 信息流读取。
每次信息流的候选集 数千篇候选帖子。

瓶颈在于复杂图谱上的候选生成，以及严格延迟限制下的 ML 排名。

候选生成

对于每位用户的信息流请求：

  1. 从多个来源收集候选帖子 ID： 收件箱式扇出： 来自好友的帖子被推送到用户收件箱（类似 Twitter，适用于普通好友）。 从名人/主页拉取： 关注者众多的主页在请求时拉取。 群组帖子： 来自用户已加入的群组。 推荐： 来自非关联来源的预测相关帖子。
  2. 合并为候选集（通常为数千篇）。
  3. 移交给排名器。

对于关系图谱非常密集的用户（1000+ 好友 + 200 主页 + 100 群组），候选集规模较大；在进行完整排名之前，我们会按时效性和来源级评分进行剪枝。

排序

多阶段 ML 流水线：

  1. 轻量级预排名器： 每个候选使用廉价特征（时效、来源类型、基础互动）。剪枝至前数百个。
  2. 主排名器： 具有丰富特征的深度模型（神经网络）： - 查看者与作者的亲密度（互动历史）。 - 帖子互动量（点赞、评论、分享）。 - 该查看者对帖子类型的偏好。 - 时效性。 - 个性化向量。
  3. 后处理： - 多样性（不展示同一作者的 5 篇帖子）。 - 可见性过滤器。 - 广告插入（按策略插入赞助内容）。 - 最终排序。

延迟：- 预排名：< 50 毫秒。- 主排名：< 200 毫秒。- 信息流总生成时间：p99 < 500 毫秒。

面试用语：

排名分两阶段：廉价的预排名器将候选剪枝至可控规模；昂贵的深度模型对剩余候选排名。特征存储提供预计算特征，使推理无需从头计算。

特征存储

在线 + 离线。

  • 离线层： 每日/每小时批量任务计算用户向量、内容向量和亲密度评分。
  • 在线层： 实时信号（刚点赞、刚评论）更新在线特征存储。
  • 排名器在请求时读取每个（用户，候选）对的特征。
  • 尽可能预计算；必要时内联计算。

可见性门控

始终在候选生成和排名之后、返回给客户端之前应用：

  • 拉黑：过滤被拉黑用户的帖子。
  • 静音：过滤被静音来源的帖子。
  • 隐藏：用户已隐藏的帖子。
  • 隐私：确保查看者符合帖子的受众设置。
  • 审核：已移除/策略封禁的帖子被排除。

读取时强制执行；不预先写入收件箱。

写入路径：发帖

  1. 客户端提交帖子。
  2. 帖子服务进行身份验证、校验，并运行自动审核。
  3. 分配雪花算法 post_id。
  4. 持久化到帖子存储。
  5. 计算受众：谁可以看到该帖子（好友、公开、自定义）。
  6. 向流发送 PostCreated 事件。
  7. 扇出工作器：- 对于普通作者：将 post_id 推送到好友的收件箱候选缓存。 - 对于高关注者作者/主页：跳过；读取时拉取。
  8. 更新特征存储：新内容可用。

读取路径：信息流

  1. 客户端请求 /home.
  2. 信息流服务获取用户收件箱候选 + 从大型主页/群组拉取。
  3. 合并候选集（约数千篇）。
  4. 预排名器剪枝至约 200 篇。
  5. 主排名器对前约 50 篇进行评分和排序。
  6. 可见性门控过滤。
  7. 从帖子存储填充帖子内容。
  8. 附带媒体 URL 渲染。
  9. 返回页面。

互动：表情回应、评论

与 YouTube 点赞和 Reddit 评论相同的模式：

  • 每用户每帖子的表情回应（一个有效）。
  • 子计数器分片的表情回应计数器。
  • 带 parent_id 的扁平评论结构。
  • 互动反馈到特征存储，用于未来排名。

扩展与分区

数据 分区键 原因
帖子 post_id 哈希值 直接查找。
连接数 source_id 用于出向； target_id 用于入向 两者均有。
收件箱缓存 user_id 哈希值 按用户存储。

一致性模型

层级 一致性 原因
帖子写入 强一致性 真实数据来源。
收件箱 最终一致 有界时效性。
特征存储 最终一致 在线 + 批量刷新。
互动计数器 最终一致 用户体验。
可见性决策 数据填充时强一致性 隐私。

多区域策略

  • 按用户归属地区部署。
  • 帖子全局复制。
  • 每区域的收件箱缓存。
  • 特征存储按区域划分，并跨区域汇总全局特征。

故障模式与缓解措施

故障模式 缓解措施 监控指标
名人帖子扇出风暴。 混合推/拉阈值；背压。 扇出延迟。
排名器模型宕机。 降级为预排名器或按时间倒序。 排名器错误率。
特征存储数据过期。 使用缓存特征；附带时间戳标记。 特征新鲜度。
收件箱中的过期删除。 读取时可见性门控。 过期展示计数。
屏蔽绕过。 最终可见性过滤器。 拉黑逃逸计数。
收件箱缓存驱逐。 从帖子存储 + 关系重建。 缓存未命中率。
跨区域复制。 异步；有界。 复制延迟。
审核延迟。 异步 + 操作传播时强一致性。 封禁生效时间。
冷启动用户。 推荐回退；热门内容。 冷启动请求率。
赞助内容故障。 渲染失败时跳过广告；不阻塞信息流。 广告错误率。

运维指标

正确性：

  • 可见性违规。
  • 过期帖子展示。
  • 特征新鲜度。

容量：

  • 信息流读取 QPS。
  • 排序器 p99 延迟。
  • 收件箱写入速率。

业务层面：

  • 信息流停留时间。
  • 互动率。
  • 每 DAU 会话数。

需要指出的权衡

  • 混合扇出平衡写/读成本。
  • 更深的 ML 模型提升排名质量，但会增加延迟。
  • 更大的候选集能发现更多相关内容，但会消耗更多排名计算资源。
  • 预计算特征加快推理速度，但会增加存储开销。
  • 激进的个性化可能造成信息茧房。
  • 严格审核可能压制合法言论；宽松则放任不良内容通过。

常见陷阱与更优做法

陷阱 更优做法
按时间倒序排列。 带多样化处理的 ML 排名。
单次排名。 多阶段：预排名器 + 主排名器。
写入时过滤拉黑。 读取时可见性门控。
同步特征计算。 带在线 + 离线的特征存储。
纯写入时扇出。 针对名人/主页的混合扇出。
同步审核。 写入时自动审核 + 异步深度扫描。

面试官可能深入探讨的方向

深入方向 优秀回答要点
排序 多阶段；特征存储；延迟预算。
候选生成 混合推/拉；多源。
可观测性 读取时过滤；屏蔽、静音、隐私。

Staff 级别回答检查清单

  • 混合扇出与多源候选生成。
  • 多阶段 ML 排序与特征存储。
  • 读取时可见性门控。
  • 互动聚合与信息流分离。
  • 多地区，按用户划分主域。
  • 审核作为持续流水线。

面试用语

Facebook 的 newsfeed 采用混合扇出加重度 ML 排序。信息流并非按时间倒序排列——而是通过 ML 对来自好友、主页、群组及推荐内容的候选帖子进行排序后组合而成。多阶段排序对候选集进行剪枝，再用深度模型评分。可见性过滤在读取时进行，从不预先写入收件箱。

2. 模拟面试

面试官： 设计 Facebook 的 newsfeed。

候选人： 流水线：候选生成、多阶段排序、可见性过滤、渲染。候选来源多样——好友帖子、关注的主页、加入的群组、来自非连接源的推荐内容。预排序器剪枝；主 ML 排序器评分；可见性过滤器筛选；渲染器填充数据。不变量是：对用户已关联源的排序内容进行有界过时展示，并正确执行策略与隐私规则。

面试官： 候选生成——它是如何工作的？

候选人： 多源。好友帖子通过混合扇出到达（普通好友推送到收件箱；有数千好友之友的超活跃用户在读取时拉取）。主页和群组：小规模关注者推送，大规模关注者拉取。推荐：独立服务从非连接源挖掘预测相关帖子。合并后的候选集有数千篇帖子，在完整排序前剪枝至可管理规模。

面试官： 排序？

候选人： 两阶段加后处理。预排序器开销低：每个候选的特征（时效、来源类型、基础互动量）。剪枝至约 200 个。主排序器是深度神经网络，利用特征存储中的大量特征：浏览者-作者亲密度、帖子互动量、时效性、该浏览者的帖子类型表现、个性化向量。对约 50 个帖子评分。后处理：多样性（避免同一作者重复出现）、可见性过滤、广告插入、最终排序。

面试官： 特征存储？

候选人： 线上 + 线下。线下：每日批量构建用户嵌入、内容嵌入、亲密度分数。线上：实时信号（刚点赞、刚评论）在数秒内更新特征。排序器在请求时读取每个（用户，候选）对的特征。尽可能预计算；必要时内联计算。特征新鲜度直接影响排序质量。

面试官： 可见性——屏蔽是如何工作的？

候选人： 读取时过滤。排序后、返回客户端前，可见性服务进行过滤：被屏蔽用户的帖子、已静音的来源、用户隐藏的帖子、受众不匹配的帖子（例如，因图谱不一致导致非好友看到仅好友可见的帖子）、违规删除的帖子。永远不要将屏蔽预写入收件箱；收件箱只是候选 ID，可见性是最终权威。

面试官： 你在浏览时帖子不断到来，如何保持信息流的新鲜度？

候选人： 收件箱缓存随好友发帖持续更新。下次拉取信息流时会获取新鲜候选。对于活跃会话，客户端可轮询"新内容"指示器，触发重新拉取。为了实时感，推送通知可宣告重要帖子。

面试官： 拥有 1 亿粉丝的明星主页发帖。

候选人： 在读取时拉取，而非推送。该主页的近期帖子集中缓存；粉丝收件箱不会因明星内容而被写入。当粉丝打开信息流时，候选生成器查询该明星近期帖子缓存。每篇帖子节省了 1 亿次收件箱写入。

面试官： 违反政策的帖子？

候选人： 写入时运行自动审核——基础检查（违禁内容、仇恨言论分类器）。高置信度检测立即屏蔽。对于微妙案例进行异步深度扫描 + 人工审核。一旦下架，帖子状态转为 removed；缓存失效；读取时可见性过滤器生效。下架时间是关键运营指标。

面试官： 这种规模下的互动计数器？

候选人： 与 YouTube 点赞相同模式。每用户每帖子的反应状态是权威来源；每帖子计数器是派生值。热门帖子使用子计数器分片。流式聚合。定期重新计数以保证正确性。

面试官： 多区域？

候选人： 按用户划分主写入地区。帖子全球复制。每地区收件箱缓存。特征存储按地区部署，跨地区汇总用于全局特征（如全球热门）。读取来自本地地区。

面试官： 冷启动用户——刚加入，无好友，无互动历史。

候选人： 默认展示其地区内的热门内容和所填写的兴趣。推荐引擎推出"你可能认识的人"和可关注的主页。通过早期互动大力引导亲密度积累。特征存储有冷启动兜底方案。

面试官： 总结。

候选人： 混合扇出到按用户的收件箱候选缓存，加上从大型来源拉取。多阶段 ML 排序——廉价预排序再用深度模型——特征存储提供预计算特征。读取时可见性过滤执行屏蔽、隐私、审核。互动计数器子分片。多地区按用户划分主域。最难的扩展问题是在稠密图上的候选生成；最难的正确性问题是读取时的可见性强制执行。

3. 练习备注

五分钟版本

  1. 明确：好友、主页、群组、排序、可见性。
  2. 不变量：来自已连接来源的排序信息流；强制执行可见性。
  3. 混合扇出用于候选生成。
  4. 多阶段 ML 排序器与特征存储。
  5. 读取时可见性门控。
  6. 互动子计数器分片。
  7. 多地区按用户划分主域。
  8. 持续审核流水线。

一分钟结尾陈述

Facebook 的 newsfeed 采用混合扇出加重度 ML 排序。候选生成从好友、主页、群组和推荐内容拉取。多阶段排序——廉价预排序再用深度模型——对顶部候选评分。特征存储提供预计算特征。可见性过滤在读取时执行屏蔽、隐私、审核。互动计数器子分片。多地区按用户划分主域。最难的扩展问题是在稠密图上的候选生成。
