News Aggregator — Google News Style (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Easy · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建一个类似 Google News 的新闻聚合器。我们持续爬取数千个新闻网站，将同一事件的文章聚合成簇，并为用户提供按兴趣个性化排序的"热点新闻"首页。日活跃用户（DAU）1 亿，来自 1 万家媒体的新文章约 100 万篇/天。

1. 需求（约 5 分钟）

功能需求 - 按计划爬取媒体 RSS/Feed 及完整页面。- 将相同事件的文章聚合成簇。- 为每位用户提供新鲜、多样、个性化的排序首页。

我会推迟处理付费内容谈判、深度编辑分类和通知功能。

非功能需求 - 新鲜度——当前爆发的事件应在约几分钟内出现。- 首页 p99 < 300 毫秒。- 读多写少：~100 万篇文章/天写入，但 1 亿用户 × 10 次页面浏览 = 10 亿次读/天 → 平均约 1.2 万 QPS，峰值约 5 万。- 簇成员关系的最终一致性可接受——随新文章到来可重新分配簇。

容量估算 - 100 万篇文章/天 × 5 KB 元数据 = 5 GB/天，加上对象存储中的原始 HTML。- 簇索引：任意时刻约 5 万个活跃簇（一个事件在 24-48 小时后消亡）。- 原始 HTML 是主要存储：100 万/天 × ~50 KB = ~50 GB/天（压缩前）；保留 30 天热数据（~1.5 TB），旧页面廉价归档。- 向量工作集更小：48 小时索引中 200 万篇文章 × 512 个 float × 4 B ≈ 4 GB 原始向量，加上 HNSW 图开销约为 2-3 倍。

2. 核心实体（约 2 分钟）

  • 媒体 — feed_url、爬取策略、上次抓取时间。
  • 文章 — url, publisher_id, title, published_at, embedding, cluster_id.
  • 簇 — cluster_id、文章列表、规范标题、话题标签、新鲜度得分。
  • 用户档案 ——兴趣向量 / 话题偏好。

3. API（约 5 分钟）

GET /v1/home?cursor=...        → { stories[], next_cursor }
GET /v1/stories/{cluster_id}   → { canonical, articles[], related[] }
GET /v1/topics/{topic}         → { stories[] }
POST /v1/interactions          { cluster_id, action }  // implicit signals for personalization

用户由 token 派生。

4. 数据流（约 5 分钟）

  1. 爬虫 按每家媒体的节奏拉取媒体 Feed（遵守 robots 协议和速率限制）。
  2. 抓取器 获取完整文章 HTML，提取正文内容（可读性处理）。
  3. Enricher 计算嵌入向量，提取实体，执行语言检测。
  4. Clusterer 将文章分配到已有簇（余弦相似度 > 阈值）或创建新簇。
  5. 排序器 计算每簇的新鲜度 + 质量 + 多样性得分。
  6. Personalizer 在请求时将簇与用户兴趣向量取交集。

5. 高层设计（约 10–15 分钟）

  1. 爬取调度器。 基于每家媒体的定时任务，带退避机制。一级媒体每 60 秒轮询一次，长尾媒体每 30 分钟一次。调度队列为优先级队列（Redis sorted set，按下次抓取时间排序）。
  2. Fetcher Worker。 拉取 URL，按规范 URL 去重，将原始 HTML 写入对象存储。将文章记录推送至 Kafka topic articles.raw.
  3. 富化管道。 流处理器（Flink）消费 articles.raw，计算句子嵌入（文本编码器），将富化后的记录写入 articles.enriched 及向量存储（每个话题一个 FAISS 或 Pinecone 分片）。
  4. 聚类服务。 对每篇富化文章，在向量存储中搜索近期文章（最近 48 小时）的最近邻。若最佳匹配 > 阈值，则分配到该簇；否则创建新簇。簇元数据存储在主数据库（MySQL 或 DynamoDB）中。
  5. 解析查询 → 扇出到相关分片（最近的优先）→ 归并排序 → 用排序模型对 Top N 重排 → 返回。 每隔 30 秒的周期性批处理任务重新计算每簇得分：新鲜度衰减 + 媒体质量 + 总覆盖率 + 多样性惩罚。将每话题前 N 个簇写入 Redis 缓存。
  6. 首页服务。 收到请求时：获取用户兴趣向量，从每话题 Redis 缓存中取出顶级簇，按用户契合度重排，返回前 20 条。
  7. 反馈循环。 interactions 事件进入 Kafka → 更新用户兴趣模型（离线重训练 + 轻量在线更新）。

端到端叙述：中午爆发一场火灾。纽约时报、路透社、美联社各自在数分钟内发布。爬虫按各媒体节奏分别抓取。每篇文章经富化后，向量搜索找到其他相关文章，一个新簇形成。排序器在下一个 30 秒周期对该簇打出高分（高新鲜度 + 3 家高质量媒体）。从第一篇文章发布起 5 分钟内，该簇出现在兴趣向量包含"世界新闻"的用户首页上。

6. 深度探讨（约 10 分钟）

具体存储与分片键。

存储 键模式 数据结构 / 条目 TTL / 保留期 分片键
爬取调度队列 crawl:due:{host} Redis ZSET，member=url_id，分数=next_fetch_at，~80 B/member 无 TTL；抓取后删除 host
原始文章 s3://news/raw/yyyy/mm/dd/{article_id}.html 压缩 HTML blob，~20-50 KB 热存 30 天，归档 1 年 日期 + article_id
文章元数据 article:{article_id} 标题/url/媒体/实体/cluster_id，~1-2 KB 热存 30 天，冷存永久 article_id
近期向量索引 vec:{locale}:{topic}:48h HNSW 条目 = article_id + 512 维向量，~2 KB 原始加图开销 48 小时滑动窗口 (locale, topic)
簇元数据 cluster:{cluster_id} 规范标题、成员、质心、得分，~5 KB 活跃 7 天，归档 90 天 cluster_id
话题热榜 top:{locale}:{topic} Redis ZSET，约 5k 条 cluster_ids，score=排名，~100 KB/榜单 每 30 秒刷新，TTL 5 分钟 (locale, topic)
用户档案 profile:{user_id} 话题权重 + 512 维兴趣向量，~1-2 KB 活跃 30 天缓存，持久化 profile user_id

按存储容量估算：

存储 Math 主要调节项
原始文章 HTML 100 万/天 * 50 KB ~= 50 GB/天；30 天热存 ~= 归档前 1.5 TB 压缩与原始保留
文章元数据 100 万/天 * 2 KB * RF=3 ~= 6 GB/天 冷归档策略
向量索引 200 万活跃文章 * ~2 KB * 2-3 倍图开销 ~= 每语言/话题工作集切片 8-12 GB 48 小时 TTL 与话题分区
簇元数据 5 万活跃簇 * 5 KB * RF=3 ~= 750 MB 体量小；热点簇的写竞争更为关键
话题热榜 100 个地区 * 100 个话题 * 100 KB * RF=2 ~= 2 GB 限制榜单长度
用户档案 1 亿 DAU * 2 KB ~= 全部缓存时 200 GB 热存 活跃用户缓存与首页兜底

面试要点在于：Redis 存储的是索引和排序指针，而非文章内容。大体量数据是原始页面和文章历史；低延迟路径仅传递簇 ID 和紧凑元数据。

首页读取路径。 1. 客户端调用 GET /v1/home?cursor=C；API 网关鉴权并转发 user_id。2. 首页服务解码游标 {topic_offsets, seen_cluster_ids, rank_version} ，使分页在刷新时保持稳定。3. 并行从 Redis/MySQL 兜底获取 profile:{user_id} 以及用户前 5-10 个话题的热榜 top:{locale}:{topic} 。4. 合并约 500 个候选 cluster_ids，过滤 seen_cluster_ids，应用多样性（MMR），再从 cluster:{cluster_id} 和文章摘要 article:{article_id}中补全前 20 条详情。5. 缓存未命中路径：若话题榜单缺失，回退至区域 top:{locale}:frontpage；若簇元数据缺失，跳过该候选而非阻塞页面。6. 返回文章列表及一个不透明游标（携带偏移量和去重状态，而非原始分数）。

文章摄入路径。 1. 调度器从 crawl:due:{host} 弹出到期 URL，并以 url_hash 为键写入抓取记录用于去重。2. Fetcher 将原始 HTML 存储至对象存储，写入元数据行 article:{article_id}，并发出 articles.raw。\n3. Enricher 计算 embedding/实体并写入 articles.enriched；实体提取和 embedding 可在主文本提取完成后并行运行。\n4. Clusterer 搜索 vec:{locale}:{topic}:48h，更新 cluster:{cluster_id} 对质心版本执行 CAS，然后插入向量。\n5. 排名任务更新 top:{locale}:{topic}；首页在下一个刷新窗口读取新的指针列表。

热首页路径与冷路径：\n1. 热路径并行读取 profile:{user_id} 和用户的热门话题列表，合并 cluster ID，仅对最终 20 个集群进行数据填充。分片键为 user_id, (locale, topic)，以及 cluster_id。\n2. 冷用户画像路径回退到 top:{locale}:frontpage，返回稳定的游标，并将异步用户画像重建任务加入队列，基于近期交互进行重建。\n3. 翻页耗尽路径使用游标的 {topic_offsets, seen_cluster_ids, rank_version} 在同一排名快照内继续翻页；若 Redis 热门列表已过期，则从集群数据库的排名快照重建，并可能返回较少的内容。\n4. 缓存未命中时，数据填充会跳过缺失的集群而非阻塞页面加载，因为对首页而言，内容的时效性比完整召回更重要。

管道选型的理由：相比纯 cron 批处理，Kafka/Flink 式流处理更适合本场景，因为时效性是以分钟为单位衡量的产品需求。MySQL/DynamoDB 适用于集群元数据，因为读取按 cluster_id 和写入是小型 CAS 更新；Elasticsearch 适合关键词搜索，但不是主要的聚类存储。Redis 热门列表是指针缓存；将 Redis 用作文章存储会浪费内存并增加归档复杂度。

聚类——最难的部分。 朴素 K-means 不适用：集群数量无界且为流式场景。采用在线聚类：每篇新文章在近期文章（48 小时窗口）的 HNSW 索引中查询；若与某质心的余弦相似度 > 0.85，则加入该集群；否则新建一个集群。质心为成员 embedding 的滑动均值。旧文章会从索引中自然淘汰。

为何选用 HNSW / FAISS 而非 SQL 扫描或周期性 K-means：SQL 无法在摄取延迟预算内扫描数百万近期向量，而 K-means 假设集群集合基本固定。HNSW 可在毫秒级完成近似最近邻查找，契合流式工作负载。要准备好回答"为何不用 Elasticsearch 向量搜索"——若搜索本就是你的运维基础，这是合理选择，但对于 48 小时工作集而言，专用向量索引在召回率/延迟调优上更容易。

集群质量。 劣质集群会混入不相关的内容。应对策略：设置严格的相似度阈值下限，要求至少来自 2 个不同发布者才可晋升至首页，质心衰减要缓慢以防漂移。

无额外延迟成本的个性化。 兴趣向量体积极小（≤512 个浮点数）。离线计算后写入用户粒度的缓存（Redis 或进程内）。请求时，首页服务对每个话题预计算的 Top-K 集群做廉价的点积运算。

双视图框架。 文章通过两种方式被访问： article:{article_id} 是内容/元数据视图，而 top:{locale}:{topic} 。 cluster:{cluster_id}.members 是同一文章集的指针视图。首页不应直接扫描文章；它应先合并排序后的集群指针，最后再填充内容。

时效性与排名稳定性。 纯时效性策略会导致内容抖动——每次刷新头条都在变化。引入滞后项，使一个集群必须以一定差距超越当前头条才会换位。

多样性。 若无惩罚机制，所有头部位置都会被同一篇内容占据。在排名时采用最大边际相关性（MMR）：每次选取时对与已选集群相似的集群进行折价。

爬取礼让与时效性预算。 在调度器中对每个发布者执行速率限制。使用 HTTP If-Modified-Since 避免重复抓取未变更的页面。Sitemaps + RSS hubs（PubSubHubbub）使参与发布者实现接近零延迟的推送。

服务状态机：

阶段 模式 数据源 游标/会话状态 延迟目标
热个性化 用户画像 + 话题列表 profile:{user_id}, top:{locale}:{topic} 排名版本 + 已见集群 < 300 ms
冷用户 区域默认值 top:{locale}:frontpage 新游标，空用户画像 < 300 ms
已耗尽 深度翻页 排名快照 + 集群 DB 话题偏移量 + 已见集合 < 800 ms
实时突发新闻 晋升集群 集群元数据 + 可信发布者规则 排名版本递进 分钟级送达首页
聚类降级 词法回退 SimHash/实体索引 低置信度标记（内部） < 10 分钟时效性

边缘压力测试。 单条突发新闻可能在某个 cluster:{cluster_id}上产生写竞争；使用带重试的质心版本 CAS 以及批量成员追加。休眠用户不应在请求路径上强制触发用户画像重计算。发布者的流量突增受每台主机的爬取队列和流背压限制。多区域数据漂移是可接受的，只要每个区域展示自己的 rank_version 并在数分钟内收敛即可。向量索引丢失后的冷启动先使用词法聚类，待 HNSW 索引重建后再修复集群。

优雅降级。

情形 模式 数据源 延迟预算 回退
热门突发新闻 推入新集群 向量索引 + 集群 CAS < 5 分钟到首页 以放宽 2 发布者规则（针对可信通讯社）的方式晋升
休眠用户画像 冷个性化 区域默认热门列表 < 300 ms 展示首页，异步重建用户画像

向量索引降级 词法聚类 标题/实体 SimHash < 10 分钟时效性 标记低置信度，稍后修复集群
发布者限流 礼让退避 Retry-After + 爬取前沿队列 下一个允许的时间槽 依赖其他发布者的 feeds/sitemaps

"我的核心结论是：首页是一个指针服务问题；聚类是代价高昂的异步问题，因此我将这两个故障域分开处理。"

重复检测。 同一篇文章在联合分发媒体上发布多次。对标题+首段做 SimHash；若 SimHash 距离在 3 bit 以内，则视为重复，合并元数据而非创建新文章。
