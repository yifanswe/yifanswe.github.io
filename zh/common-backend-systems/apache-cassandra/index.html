1. 基础知识与解题思路

本题考察点

Cassandra 是一个宽列分布式数据库，将 Dynamo 的无主节点集群设计（一致性哈希、sloppy quorum、gossip）与 Bigtable 的存储模型（SSTable、LSM 树、列族）相结合。难点在于严格按需一致性（CL ALL、QUORUM、ONE）与高可用性之间的权衡，以及基于带聚簇列的宽行的数据模型。

其不变量如下：

每次写入都持久化到至少一个副本的提交日志，并传播到所有 N 个副本；在配置的一致性级别下读取能反映最新写入（在可调范围内）；宽列模型支持高效的按分区查询，并以聚簇列顺序读取；集群通过添加节点实现线性扩展。

在 ICT5/L6+ 级别，面试官考察你是否能清晰阐述 Cassandra 的数据模型（分区键 + 聚簇列）、LSM 树存储引擎、一致性级别、轻量级事务（Paxos）以及运维复杂性（压缩、修复）。

60 秒面试框架

口述版本：

我会将 Cassandra 描述为 Dynamo 的集群模型加上 Bigtable 的存储引擎。一致性哈希用于分区；每次查询可配置复制和一致性级别；提交日志 + memtable + SSTable 用于存储；LSM 压缩；gossip 用于成员管理。数据模型是带分区键和聚簇列的宽行，在分区内实现有序范围查询。通过 Paxos 实现轻量级事务（比较并设置）。关键决策点是数据模型、一致性级别、压缩策略和修复。

Cassandra 满足的需求

功能性需求：

  • 分布式宽列存储。
  • 带聚簇顺序的宽行查询。
  • 每次查询可调的一致性。
  • 多数据中心复制。
  • 轻量级事务（比较并设置）。
  • 时间序列和 IoT 工作负载。

非功能性需求：

  • 高写入吞吐量。
  • 高可用性。
  • 线性可扩展性。
  • 运维简单的集群扩展。
  • 多区域（主动-主动）。

超出范围：

  • 跨分区的多行事务。
  • 严格的全局排序。
  • 即席全表 join（不适用于关系型工作负载）。

核心概念

数据模型

Keyspace
  └── Table
       └── Row
            ├── Partition key (determines node placement)
            ├── Clustering columns (sort order within partition)
            └── Regular columns (values)

示例：传感器时间序列。

CREATE TABLE sensor_data (
  sensor_id text,
  reading_ts timestamp,
  value double,
  PRIMARY KEY (sensor_id, reading_ts)
);
  • sensor_id = 分区键 → 哈希到某个节点。
  • reading_ts = 聚簇列 → 在分区内排序。
  • 查询： SELECT * FROM sensor_data WHERE sensor_id = ? AND reading_ts > ? ORDER BY reading_ts.

Cassandra 中的"行"由 (partition_key, clustering_columns) 标识；"分区"持有具有相同分区键的多行，按聚簇列排序。

该数据模型是 分区局部性：高效查询在单个分区内进行。跨分区的查询代价高昂（需要 scatter-gather）。

面试用语：

Cassandra 数据模型是分区键 + 聚簇列。查询模式必须优先设计；表按查询模式去规范化。

存储引擎：LSM 树

继承自 Bigtable。

  • 写入路径： 提交日志（持久性）+ memtable（内存中的有序结构）。
  • 当 memtable 写满时：刷新到磁盘上的 SSTable。
  • SSTable 是不可变的有序文件。
  • 压缩定期合并 SSTable。
  • 每个 SSTable 的 Bloom filter 加速了负向查找。

压缩策略： - SizeTieredCompactionStrategy (STCS)： 合并大小相近的 SSTable，适合写密集型场景。 - LeveledCompactionStrategy (LCS)： 按大小限制维护多个层级，适合读密集型场景。 - TimeWindowCompactionStrategy (TWCS)： 按时间窗口分组，适合时序数据。

一致性级别（按查询）

级别 行为
ONE 等待 1 个副本响应。
QUORUM 等待 floor(N/2) + 1.
ALL 等待全部 N 个副本响应。
LOCAL_QUORUM 本地数据中心内的 Quorum。
EACH_QUORUM 每个数据中心内的 Quorum。
ANY 若所有副本不可用，协调节点通过 hinted handoff 接受写入。

强一致性： R + W > N （例如，N=3 时 QUORUM + QUORUM）。最终一致性：使用较低的级别。

集群设计（源自 Dynamo）

  • 使用虚节点的一致性哈希环。
  • 每个 keyspace 的复制因子为 N。
  • Gossip 用于成员管理。
  • 短暂故障使用 Hinted handoff。
  • 通过 Merkle 树进行反熵修复。
  • 可选的 Sloppy quorum。

高层架构

Client -> Coordinator (any node)
              |
              v
     Consistent Hashing Ring
              |
              v
       Preference List (N nodes)
              |
        Replicate write to N (wait for CL)
              |
              v
       Commit log + Memtable
              |
        Flush to SSTable
              |
        Compaction (background)
              |
        Gossip / Hints / Repair

核心组件

组件 职责 关键细节
协调节点 单次请求节点。 任意节点。
Commit log 仅追加的持久化日志。 首次写入目标。
Memtable 按聚簇列排序的内存结构。 达到大小/时间阈值时刷写。
SSTable 磁盘上不可变的有序文件。 每个分区对应多个。
Compactor 合并 SSTable。 后台运行，由压缩策略驱动。
Gossip 集群成员管理。 源自 Dynamo。
Hinted Handoff 临时存储不可达副本的数据。 节点恢复后重放。
修复 基于 Merkle 树的反熵。 手动或定时执行。
Bloom filter 每个 SSTable 的负向查找。 存于内存。
Snitch 副本放置的拓扑感知。 例如，DC 感知模式。

数据模型（续）

计数器

用于原子自增/自减的特殊列类型，采用不同的复制语义（基于增量）。一致性保证有限。

物化视图

服务端去规范化：从基表自动维护的二级表，每次写入时有额外开销；适用于需要按不同键查询的场景。

二级索引

非分区列上的节点本地索引，查询需要对所有分区进行散射-聚合，开销较大。

写入路径详解

  1. 协调节点接收写入请求。
  2. 计算分区键哈希值 → 得到偏好列表（N 个副本）。
  3. 并行向所有 N 个副本发送写入请求。
  4. 每个副本：追加到 commit log → 插入 memtable。
  5. 等待达到 CL 所需的响应数。
  6. 返回成功。

Memtable 刷写：当大小或时间阈值触发时，写入新的 SSTable，刷写完成后截断 commit log。

读取路径详解

  1. 协调节点接收读取请求。
  2. 计算分区键哈希值 → 得到偏好列表。
  3. 按 CL 要求向最近的副本发送读取请求。
  4. 每个副本：查询 memtable 和相关 SSTable（通过 Bloom filter 过滤）。
  5. 协调节点合并响应（时间戳最新的列值获胜）。
  6. 若 CL > 1，收到 CL 个响应后返回。
  7. 读修复 （可选）：若响应不一致，将合并后的值写回各副本。

对于分区内的范围查询：按聚簇顺序读取 SSTable，合并后按聚簇顺序返回行。

轻量级事务（CAS）

基于 Paxos 的比较并交换，用于单分区原子操作。

UPDATE users SET email = 'new' WHERE id = 1 IF email = 'old';
INSERT INTO users (...) VALUES (...) IF NOT EXISTS;

实现方式：在分区内执行四阶段 Paxos，比普通写入慢（4 倍往返时延）。适用于需要严格原子性的场景（例如，唯一用户名）。

墓碑（Tombstone）

删除操作不会物理移除数据，而是写入一个墓碑标记该行已删除。

  • 墓碑带有时间戳。
  • 读取时，墓碑优先于旧数据。
  • 压缩最终会丢弃超过一定时间的墓碑 gc_grace_seconds.

为何需要宽限期：防止已删除数据在错过的副本追上来后重新出现。若副本在删除期间下线且墓碑已被 GC，该副本的旧值可能通过修复使数据复活。

常见运维陷阱：墓碑数量过多会导致读性能下降。

压缩策略

策略 适用场景
SizeTieredCompactionStrategy 写密集型场景，可容忍较高的读放大。
LeveledCompactionStrategy 读密集型场景，读放大有界。
TimeWindowCompactionStrategy 时序数据，可高效清除旧数据。

根据工作负载进行选择。

修复

Cassandra 需要定期修复以确保副本收敛。

  • 在副本之间进行 Merkle 树比较。
  • 流式传输差异数据。
  • 按计划运行（通常每周一次）。
  • 开销较重，通常一次只对一个节点执行。

若不进行修复：陈旧副本会发生漂移，已删除数据可能在以下情况后复活： gc_grace_seconds.

多数据中心

Snitch 使副本放置具备数据中心感知能力。

  • 每个数据中心的复制因子： {"DC1": 3, "DC2": 3}.
  • 写入会传播到所有数据中心；协调节点可等待本地 Quorum（LOCAL_QUORUM）——速度快——并异步复制到其他数据中心。
  • 若需严格的跨数据中心一致性：使用 EACH_QUORUM。

Cassandra 以多数据中心主-主架构著称。

扩展与分区

关注点 策略
添加节点 从现有节点 Bootstrap，流式传输负责的数据范围。
移除节点 下线节点，重新分配数据。
热点分区 限制分区大小，根据访问模式建模。
墓碑开销 调整 gc_grace，设计时减少删除操作。
读放大 LCS 压缩，较小的 SSTable。

一致性模型

层级 一致性 原因
每次写入 在配置的 CL 时 可调。
每次读取 在配置的 CL 时 可调。
修复 最终一致 周期性。
LWT 强一致性（Paxos） 以延迟为代价。

故障模式与缓解措施

故障模式 缓解措施 监控指标
副本宕机。 Hinted handoff，较低的 CL 维持可用性。 Hint 积压。
永久丢失。 修复，副本替换。 修复延迟。
压缩积压。 调整策略，纵向扩展。 待处理的压缩任务。
墓碑过载。 gc_grace 调优，优化设计。 每次读取的墓碑数量。
热点分区。 在应用层重新分片。 分区大小。
宽分区。 限制分区键。 分区大小。
陈旧读（低 CL）。 读修复，提高 CL。 陈旧读比率。
网络分区。 Sloppy quorum，设置告警。 分区事件。
多数据中心延迟。 LOCAL_QUORUM，跨数据中心监控。 跨数据中心延迟。
引导缓慢。 流量限速，并行执行。 Bootstrap 时长。

运维指标

正确性：

  • 修复延迟。
  • 每次读取的墓碑数量。
  • Hint 积压。
  • 低 CL 导致的陈旧读。

容量：

  • 每节点 QPS。
  • 压缩积压。
  • 读延迟 p99。

业务层面：

  • 吞吐量。
  • 每节点存储。

需要指出的权衡

  • Cassandra 以牺牲多行事务换取高写吞吐量和线性扩展能力。
  • 宽行可高效支持按分区查询，但要求按查询进行数据建模。
  • LCS 的读放大有界，但写放大高于 STCS。
  • LWT 提供原子 CAS，但延迟代价较高。
  • 墓碑清理时间与已删除数据复活风险之间的权衡。
  • 多数据中心主-主架构非常适合读本地化，但跨数据中心为最终一致性。
  • 计数器的一致性比普通列弱。

常见陷阱与更优做法

陷阱 更优做法
当作关系型数据库使用（含 JOIN）。 按查询去规范化，优先设计数据模型。
不加考虑地使用默认一致性级别。 按查询选择 CL，QUORUM 较为常见。
频繁执行删除操作。 尽量减少删除，理解墓碑的代价。
时序数据使用随机分区键。 TimeWindowCompactionStrategy。
忽略修复操作。 定期安排修复。
对所有场景使用 LWT。 仅在真正需要原子性时使用。

面试官可能深入探讨的方向

深入方向 优秀回答要点
数据模型 分区键 + 聚簇列，查询驱动的设计。
存储引擎 Commit log + memtable + SSTable + 压缩。
一致性 按查询选择 CL，R+W>N 保证强一致性。

  • 分区键 + 聚簇列。
  • LSM tree 写入路径，SSTable + 压缩。

面试用语

Cassandra 是 Dynamo 的集群架构加上 Bigtable 的存储引擎。使用一致性哈希进行分区，使用 LSM tree（commit log + memtable + SSTable + 压缩）进行存储，支持按查询调整一致性级别，通过 Paxos 实现轻量级事务 CAS。数据模型为分区键 + 聚簇列——以查询模式优先，每种模式对应一张去规范化的表。多数据中心主-主架构是其优势所在，多行事务则不是。

2. 模拟面试

面试官： 什么是 Cassandra？

候选人： 分布式宽列数据库。集群模型源自 Dynamo：一致性哈希、Gossip 协议、Sloppy quorum、反熵修复。存储引擎源自 Bigtable：commit log、memtable、SSTable、压缩。数据模型：keyspace → 表 → 分区 → 行，每行由（partition_key, clustering_columns）映射到值。查询按分区优化。支持按查询调整一致性级别。其不变量是高可用性、线性扩展以及高效的按分区查询。

面试官： 介绍一次写入的完整流程。

候选人： 客户端将写入请求发送给协调节点。协调节点对 partition_key 进行哈希，在偏好列表中找到 N 个副本，并行向所有副本发送写入请求。每个副本追加到 commit log（持久化）并插入 memtable（按聚簇列在内存中排序）。协调节点等待 CL 个副本确认后返回成功。Memtable 写满后刷写为新的不可变 SSTable，后台压缩负责合并 SSTable。

面试官： 读取路径。

候选人： 协调节点对 partition_key 进行哈希，按 CL 要求选择最近的副本。每个副本查询 memtable 和 SSTable（Bloom filter 避免不必要的磁盘寻址），返回所请求分区和聚簇范围内的行。协调节点合并响应（每列取时间戳最新的值）。若副本之间存在差异，可选择执行读修复。

面试官： 数据模型。

候选人： 宽行。partition_key 决定节点的分配。聚簇列定义分区内的排序顺序。示例： (user_id, post_ts) ——某用户的所有帖子存于同一分区，按时间排序，因此范围查询"用户 X 在某时间段内的帖子"非常高效（单分区读取）。关键在于设计 partition_key + 聚簇列以匹配查询模式，表按查询进行去规范化。

面试官： 为何不支持 JOIN？

候选人： 跨分区的 JOIN 需要跨节点协调，会损害 Cassandra 的线性扩展能力。取而代之的做法是反范式化：将相关数据存储在同一分区，或在多张表中冗余存储。对于用户的帖子和评论，可以将评论嵌入同一分区，或为每个 (user, post) 维护一张并行表。

面试官： LWT — 何时使用？

候选人： 当需要原子性时使用 Compare-and-set：唯一用户名、主节点选举、序列号。底层使用 Paxos 实现，需要 4 次往返。比普通写入慢得多（约 10 倍）。应谨慎使用。大多数场景下，接受最终一致性并围绕竞态条件进行设计即可。

面试官： Tombstone — 有什么问题？

候选人： 删除操作不会物理移除数据，而是写入带时间戳的 tombstone。读取时必须检查 tombstone 以判断哪些数据已被删除。大量 tombstone 会拖慢读取性能。Compaction 最终会在以下时间后丢弃 tombstone： gc_grace_seconds （默认 10 天）。在此之前，tombstone 必须保留，以便在修复过程中，处于离线状态的副本接收到旧值时能看到 tombstone，而不会将已删除的数据重新复活。

面试官： Compaction 策略。

候选人： 三种主要策略。SizeTieredCompactionStrategy：合并大小相近的 SSTable，适合写密集型场景。LeveledCompactionStrategy：按大小预算维护多层级，读放大有界，适合读密集型场景。TimeWindowCompactionStrategy：按时间窗口对 SSTable 分组，适合批量丢弃旧数据的时序场景。根据工作负载选择合适策略。

面试官： 多数据中心（Multi-DC）？

候选人： Snitch 使数据放置具备数据中心感知能力。每个数据中心独立配置副本因子。写入会传播到所有数据中心。LOCAL_QUORUM 等待本地数据中心达到 quorum（速度快）；EACH_QUORUM 要求跨数据中心的严格一致性。常见模式：各数据中心采用 LOCAL_QUORUM 的 active-active 部署，跨数据中心延迟在秒级可接受。

面试官： 修复（Repair）。

候选人： 反熵机制。通过 Merkle 树比对分区范围内各副本的数据差异，并流式传输差异以使数据收敛。这是确保最终一致性的必要手段，尤其是在长时间故障之后。建议定期执行（通常每周一次）。属于重量级操作，通常每次只对一个节点执行。

面试官： 热点分区？

候选人： 某个 partition_key 的所有流量都落在相同的 N 个副本上，形成瓶颈。在应用层修复：通过为 partition_key 添加随机或基于时间的后缀来进行子分区，聚合时查询所有后缀。另一种方案是在上层添加缓存层。Cassandra 本身无法将单个 partition_key 拆分到多个节点上。

面试官： Counter 列？

候选人： 用于原子自增/自减的特殊列类型。内部采用与普通列不同的基于增量的副本复制机制。具有最终一致性；在故障场景下可能存在少量数据丢失。适用于对精确性要求不高的"点赞计数"等场景。如需精确计数，应使用从事件日志周期性重新计算的方式。

面试官： 为集群添加新节点。

候选人： 预先分配 vnode 范围。新节点加入环后，现有节点流式传输相关范围的数据。通过 vnodes 并行化——多个节点同时传输。应控制速率以避免带宽饱和。Gossip 协议负责传播新的成员关系。Bootstrap 完成后，新节点开始为其负责的范围提供读写服务。

面试官： 总结。

候选人： Dynamo 集群 + Bigtable 存储。基于 partition_key + clustering column 的宽列数据模型，围绕查询模式进行设计。LSM 树写入路径；SSTable compaction。每次查询可调节一致性；通过 Paxos 实现 LWT CAS 操作。多数据中心 active-active 部署。需要反熵修复。热点分区和 tombstone 是运维陷阱。优势在于线性扩展和高写入吞吐量；劣势是不支持多行事务。

3. 练习备注

五分钟版本

  1. Dynamo 集群 + Bigtable 存储。
  2. 宽行：partition_key + clustering column。
  3. LSM：commit log + memtable + SSTable + compaction。
  4. 每次查询可调节一致性级别（CL）。
  5. 通过 Paxos 实现 LWT CAS。
  6. 多数据中心 active-active。
  7. Repair（Merkle 树）；tombstone 垃圾回收。
  8. 按工作负载选择 compaction 策略。

一分钟结尾陈述

Cassandra 将 Dynamo 的集群模型与 Bigtable 的 LSM 存储相结合。使用 partition_key + clustering column 的宽行结构，支持高效的分区内查询。每次查询可调节一致性级别。通过 Paxos 实现 LWT 的 compare-and-set 操作。LSM compaction（STCS、LCS、TWCS）根据工作负载选择。多数据中心 active-active 部署。需要反熵修复。优势是线性写入扩展和高可用性；劣势是不支持 JOIN、不支持多行事务，以及围绕 tombstone 和 compaction 的运维复杂性。

最佳面试用语

  • "数据模型优先：查询模式决定分区键和 clustering 键。"
  • "每次查询可调节一致性；对极少数需要严格原子性的场景使用 LWT。"
  • "LSM 树写入路径；SSTable compaction 策略取决于工作负载。"
  • "Tombstone 存活至 gc_grace 过期；大量 tombstone 会拖慢读取性能。"
  • "多数据中心 active-active 是优势；不支持多行事务是劣势。"
