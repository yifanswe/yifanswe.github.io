Facebook News Feed（详解）

2026/06/01 · Common Backend Systems
目录
目录

难度： Medium · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建 Facebook (FB) 新鲜事。用户打开应用，看到来自好友和关注主页的排名个性化信息流。日活跃用户 (DAU) 20 亿，平均用户关注约 300 个实体，会话短暂且频繁。

1. 需求（约 5 分钟）

功能需求 - 发布文字/图片/视频更新（写入侧）。\n- 打开应用查看排名信息流（读取侧——重点）。\n- 对帖子点赞/评论/分享；基本互动信号反馈给排名系统。

我会推迟聊天、Marketplace、Stories 和群组功能。

非功能需求 - 信息流打开 p99 < 500 ms。\n- 读写比重约 100:1，读取为主。\n- 排名应融入新鲜信号——好友刚发的帖子应快速上升。\n- 最终一致性可接受——点赞计数延迟一秒，没人会注意到。\n- 能应对名人不对称——拥有 5000 万关注者的发帖者 vs. 只有 50 个关注者的发帖者。

容量估算 - 20 亿 DAU × 10 次信息流打开 = 每天 200 亿次信息流加载 → 平均约 23 万 QPS，峰值约 100 万。\n- 平均用户每天发帖 0.5 次 → 每天 10 亿帖子 → 平均约 1.2 万次写入/秒。

2. 核心实体（约 2 分钟）

  • User — user_id，关注关系。
  • 帖子 — post_id，author_id，content，media，created_at，signals（点赞数、评论数）。
  • 边缘（关注关系） — follower_id, followee_id.
  • FeedEntry — user_id, post_id, score, inserted_at.

3. API（约 5 分钟）

POST  /v1/posts                       { content, media }     → { post_id }
GET   /v1/feed?cursor=...&limit=20    → { posts[], next_cursor }
POST  /v1/posts/{id}/like
POST  /v1/posts/{id}/comments         { text }

从 token 获取用户；通过游标分页（服务端颁发的不透明游标，非偏移量）。

5. 高层设计（约 10–15 分钟）

核心问题： 写时扇出 vs. 读时扇出。 我默认选择 混合.

  1. 帖子服务。 将帖子写入主存储（按 post_id分片）。发布一条 post.created 事件。
  2. 扇出服务。 对于大多数作者（关注者 ≤1 万），将 post_id 推入每个关注者的信息流列表（Redis 有序集合，score = 排名时间）。对于名人，跳过扇出——将其帖子保留在全局"名人近期帖子"缓存中。
  3. 信息流组装服务。 信息流打开时：\n - 从 Redis 拉取用户的预计算信息流列表（以 user_id为键的有序集合）——这只是约 300 个候选 post_id，而非帖子正文。\n - 合并此用户关注的名人作者的 post_id，在请求时从每位作者的 posts_by:{author_id} Redis 列表中读取。\n - 通过批量 MGET post:{post_id} 对帖子正文进行水合；缓存未命中时回退到事实来源存储。\n - 使用在线排名模型重新排序。\n - 返回前 20 条及游标。
  4. 排名服务。 在线推理。特征包括新鲜度、作者亲密度、互动速度、帖子类型。模型服务器（如 TensorFlow Serving），与信息流拉取并行进行特征存储获取。
  5. 互动管道。 点赞/评论 → Kafka → 计数器聚合（Redis + ClickHouse 用于分析）+ 排名信号更新。

热路径中的具体存储：

信息流读取链路： ZREVRANGE feed:{user} 和特征存储读取并行运行。名人 LRANGE posts_by:{author} 调用按 Redis 分片流水线化。水合是一次批量 MGET post:{id...}；只有未命中才访问 TAO/MySQL。排名在水合之后运行，因为模型需要内容类型和最新计数器。游标存储 {mode, score_watermark, last_seen_ts, seen_set_id}.

端到端叙述：Alice 发帖。扇出服务将 post_id 推入她 200 个好友的 Redis 信息流列表，同时 LPUSH 到 posts_by:alice （每作者索引，用于拉取模式读取）。Bob 打开 FB → 组装服务获取 feed:bob （300 个候选 post_id），为 Bob 关注的名人读取 posts_by:{celeb_id} （另外 5 个 post_id），批量水合全部 305 条帖子正文（来自帖子缓存），发送给排名服务，返回前 20 条。

两种索引，一份数据集。 推送和拉取并非不同的存储——它们是对相同帖子的两种索引方式。 feed:{reader_id} 在 读取 侧预先连接（推送）。 posts_by:{author_id} 在 写入 侧（拉取）。混合模式同时维护两种索引，并根据读取者的信息流列表是否热存来选择查询哪一个。

写入侧容量：每天 10 亿帖子约合每秒 1.2 万 post.created 事件；以 200 B 的事件负载计算，Kafka 原始数据仅约 2.4 MB/秒，RF=3 时约 7 MB/秒，因此事件日志不是成本中心。扇出写入才是。若普通作者扇出给 200 个活跃关注者，平均约每秒 240 万次 Redis ZADD；峰值可达 5-10 倍，因此工作节点按目标 Redis 分片批处理并流水线化写入。每个扇出任务携带 {author_id, post_id, follower_page_token} ，使重试从关注者分页边界恢复，不会产生重复的 feed:{user_id} 条目。

6. 深度探讨（约 10 分钟）

为什么选混合推/拉模式。 纯推送在名人场景下会崩溃（一条帖子 → 5000 万次 Redis 写入）。纯拉取同样无法扩展——每次打开信息流都要执行类似"给我我关注的 300 人中最近 300 条热帖"的查询，热键问题严重。混合方案：普通用户推送，名人读取时拉取。阈值是一个可调参数（约 1 万关注者）。

要做好回答"为什么不纯预计算？"的准备。诚实的答案是不对称性：中位数作者使推送成本低廉，但最大的作者会将一次写入变成数千万次写入。混合方案使常见情况保持快速，并将极端情况变为读取时合并。

Redis 信息流列表。 以 user_id为键的每用户有序集合，上限约 300 条。每条约 64 字节（8B post_id + 8B score + 跳表开销）。每用户约 20 KB。读取通过 ZREVRANGE进行 O(log N) 范围查询。对单个条目设置 TTL（约 7 天），使集合自动裁剪。

容量——这是主导成本。 30 亿月活跃用户 (MAU) × 20 KB ≈ 60 TB 信息流列表（复制前）。跨区域复制 ×3 → 约 1 PB 的 Redis 内存。帖子正文缓存相比之下微不足道（热窗口约 2 TB），因为大多数信息流读取服务的是 24 小时内的帖子，且帖子正文较小。

作者索引更便宜：10 亿活跃作者 × 50 个 id × 8 B 约 400 GB 原始数据，加上 Redis 开销/复制后为几 TB。 recently_shown 可能出人意料地大：20 亿日活跃用户 × 200 个 id × 约 16 B 物理大小 = 数 TB，因此保持 TTL 短并严格设置上限。主要成本仍是 feed:{user_id} 内存和扇出写入。

缩减占用的调节手段： (a) 维护一个 活跃集合 （近 7 天内活跃用户的 Redis 集合或布隆过滤器），仅向成员预扇出——将实时工作集降至约 30 TB；(b) 每条目 TTL 约 7 天，使休眠列表自动逐出；(c) 紧凑编码（int64 的 post_id，非字符串）；(d) 若拉取模式回退足够廉价，则缩小上限至 200 条。

当休眠用户打开应用时， 其列表已消失——组装服务回退到 拉取模式 （见下文），异步任务重建 feed:{user_id} 并将用户重新加入活跃集合。其下次打开将命中快速推送路径。

在 300 毫秒以内完成排序。 将候选特征预取到以下键索引的特征存储中： user_id 。 post_id。来自聚合层的两路并行调用：Feed 列表获取 + 特征获取。模型推理为单次批量调用。将用户的 Feed 结果缓存 5–10 秒，使得频繁刷新的代价极低。

排序信号的新鲜度。 互动事件流入"近期信号"在线特征存储（Redis 或键值缓存），延迟在秒级。速率（每分钟点赞数）是最强的新鲜度信号。

帖子存储。 该工作负载纯粹是按以下方式进行点查询： post_id 在高写入吞吐量与均匀分片下——任何水平分片存储均可胜任。Facebook 的经典答案是 分片 MySQL 配合 TAO （FB 的图感知缓存层）：强分片内一致性、运维成熟、也是实际生产架构。Cassandra、HBase、Bigtable 或 DynamoDB 同样适用——若需要多数据中心主-主写入且能接受最终一致性，可选用它们。

写穿透型 帖子正文缓存 （Redis/Memcached，以 post_id为键）位于前端，吸收约 23 万 QPS 的读请求路径。实际命中率极高——大多数 Feed 读取的内容均在 24 小时以内。缓存未命中时透传至真实存储，后端仅需处理极小比例的读流量。

面试时，直接提 Cassandra 无妨，但要准备好回答"为什么不用分片 MySQL？"。实话实说：这个工作负载根本用不上 Cassandra 的亮点特性（宽行、可调一致性）。选 Cassandra 是为了写入吞吐量和多数据中心主-主写入，而不是因为它是宽列存储。

多区域漂移。 写入落地于作者所在的主区域，然后 post.created 向读取区域复制。对于普通作者，扇出可在读取区域收到事件后在本地发生；1–3 秒的延迟可接受。对于名人作者，只有 posts_by:{author_id} 需要跨区域复制，因此影响范围更小。Feed 游标应包含 feed_epoch 或 region_watermark ，以避免故障转移后第 2 页混入来自两个不同排序快照的候选结果。互动计数明确为最终一致；可见性与屏蔽操作需对照最新可用的图缓存进行检查，若数据过期则保守拒绝。

边缘压力测试。 对于名人帖子，采用纯拉取索引，而非尝试推送至 5000 万个列表。某个不活跃用户的 feed:{user_id} 缺失时，触发拉取模式和异步重建，而不是每次请求都遭遇 500 毫秒的冷命中。深度滚动的用户通过游标进入回拉和冷历史阶段，而不是让 Redis 去查找它已不再持有的条目。来自高活跃作者的突发写入按从节点分片合并为扇出批次；若队列滞后，最新帖子优先，旧的低分候选可以跳过，因为排序无论如何都会将其埋没。新 Redis 集群的冷启动通过提供拉取模式读取来处理，同时扇出工作进程懒加载地重新填充活跃用户的数据。

面试关键结论（可靠性方面）： Feed 宁可过时，也不能变慢。如果 Redis 数据冷、扇出滞后，我会降级到作者索引，排序仍能返回合理的页面。

名人读取。 按作者的"近期帖子"Redis 列表，位于 posts_by:{author_id} ——最近约 20 个 post_id，TTL 约 7 天。对于名人，扇出服务跳过向粉丝 Feed 列表写入，只更新此列表；聚合层在请求时读取它。将单个用户的名人扇入上限设为关注最多的前 N 人，以避免异常读取。这个按作者的索引并非仅用于名人——它对每个 每个 作者都存在，也是拉取模式降级（见下文）的基础。

为什么用 Redis ZSET/LIST 而不是只用数据库：Feed 打开的 p99 延迟为 500 毫秒，需要数百次小型有序读取。Redis 提供可预期的内存驻留范围读取；MySQL/TAO 保持作为帖子正文和冷历史的真实存储。纯 Cassandra 宽行 Feed 同样可行，但那样每次 Feed 打开都要支付存储读取延迟和合并代价，而非内存代价。

Feed 列表是 Feed 顶部的缓存，而不是完整 Feed。推送和拉取是同一数据集的两个索引。游标决定查询哪个层级，系统通过各层逐步降级，而非在 Redis 耗尽时直接崩溃。 新用户的历史回填。

新关注操作不会将历史帖子回填到 Feed 列表中——代价太高。取而代之的是，聚合路径在接下来几小时内于读取时从新被关注者拉取近期帖子，之后由稳态混合模式接管。 互动计数的一致性。

我会推迟的内容。 立即从 Redis 计数器展示近似计数；每小时通过 ClickHouse 核对精确计数。用户永远不会注意到差了 3 个点赞数。
