1. 基础知识与解题思路

本题考察点

Kafka 是一个分布式日志系统：一个仅追加、分区、副本复制的数据流。难点在于：在分区内保证消息有序、在 broker 故障时做到零数据丢失、扩展至每秒数百万次写入，以及支持消费者组在不重复处理的情况下共享工作负载。

其不变量如下：

每条生产成功的消息都会在 quorum 副本上持久化提交，在其所在分区内有序排列，可由任何消费者按分配的 offset 读取，并按配置时间保留；消费者组将分区分配给各成员，使每个分区最多由组内一个消费者处理；rebalance 有界，不会丢失进度。

在 ICT5 / L6+ 级别，面试官考查的是：能否设计一个分区副本日志、解释主节点选举和 ISR（同步副本集），推导 producer acks 级别的含义，理解消费者组 rebalance 机制，以及通过事务或幂等 producer 实现 exactly-once 语义。

60 秒面试框架

口述版本：

我会将 Kafka 设计为一个 broker 集群，每个 broker 持有一部分 topic 分区。每个分区是一个带副本的仅追加日志，由一个主节点（leader）处理读写，从节点（follower）负责复制。Producer 向分区 leader 发送批量消息；Consumer 按分区追踪 offset。消费者组将分区拆分以实现并行消费。核心设计决策包括：分区副本、ISR 管理、producer ack 级别、消费者组 rebalance，以及 exactly-once 语义。

首先明确需求

功能性需求：

  • 带分区的有序消息日志 topic。
  • Producer 追加消息；持久化。
  • Consumer 从 offset 开始读取；可恢复。
  • 消费者组：成员共享分区。
  • 数据保留：按时间或大小。
  • Topic 压缩（按 key 保留最新值）。
  • 跨 topic 的事务，以实现 exactly-once 语义。

非功能性需求：

  • 每集群每秒百万级写入。
  • PB 级存储；数周的保留期。
  • 高可用性：能容忍 broker 故障。
  • 分区内强顺序保证。
  • Producer 持久性可调（acks=0、1、all）。
  • Consumer 支持 at-least-once 或 exactly-once。

以下内容超出范围，除非被特别询问：

  • Connector 和 Kafka Connect。
  • Streams 处理 API。
  • KSQL。

副本复制与 ISR

每个分区有一个副本因子（例如 3）。一个副本为 leader，其余为 follower。

  • Leader 接收生产请求，追加到本地日志。
  • Follower 从 leader 拉取数据并复制。
  • Follower 满足以下条件时被认为是 同步副本（ISR） ：其 offset 在阈值范围内追上了 leader。
  • 高水位线（HW）是所有 ISR 成员都已复制的最高 offset。
  • Consumer 只能读取到 HW 之前的数据。

Producer acks 级别：

  • acks=0：producer 不等待确认。速度最快，无持久性保证。
  • acks=1：leader 写入本地后立即返回，不等待 follower。若 leader 在复制前故障，存在数据丢失风险。
  • acks=all：leader 等待所有 ISR 成员确认。持久性最强。

若某个 follower 滞后过多，将被移出 ISR；分区继续以剩余 ISR 成员运行。若 ISR 缩减至低于 min.insync.replicas，分区将拒绝写入（附带 acks=all).

面试用语：

ISR 是被认为处于最新状态的动态副本集。当所有 ISR 成员确认后，写入才算提交；Consumer 只能看到已提交的 offset。这使得持久性可以以成本为代价进行调节。

主节点选举

Controller 监控 broker 健康状态。当某个 broker 故障时：

  1. Controller 通过心跳超时检测到故障。
  2. 对于该 broker 所负责的每个分区，controller 从 ISR 中选出新的 leader。
  3. 新 leader 接管；发出新的 leader_epoch。
  4. Producer 向新 leader 重试。

若没有可用的 ISR 成员，行为取决于 unclean.leader.election.enable:

  • 默认 false：分区将不可用，直到某个 ISR 成员恢复。
  • True：允许选举一个不同步的副本（提高可用性，但存在数据丢失风险）。

现代 Kafka 使用 KRaft （基于 Raft 的 controller），取代 ZooKeeper，简化了部署。

Producer 路径

  1. Producer 创建带可选 key 的消息记录。
  2. Partitioner 选择分区：hash(key) % num_partitions，或在无 key 时使用轮询。
  3. Producer 按分区批量聚合消息。
  4. 将批量消息发送给分区 leader。
  5. Leader 追加到本地日志并复制。
  6. 一旦满足 acks 级别要求，leader 返回成功。

幂等 producer：

  • 每个 producer 有一个 producer_id 及每个分区的序列号。
  • Broker 通过以下方式去重： (producer_id, sequence_number).
  • 消除重试导致的重复消息。

事务性 producer：

  • Producer 开启一个事务。
  • 向多个分区/topic 写入数据。
  • 原子性地提交或中止。
  • 配置了 read_committed 的 Consumer 只能看到已提交的消息。
  • 用于实现 read-process-write 流水线的 exactly-once 语义。

Consumer 路径

  1. Consumer 从已分配的分区轮询 broker 获取消息。
  2. Broker 返回不超过高水位线的批量消息。
  3. Consumer 处理消息。
  4. Consumer 提交 offset（同步或异步）。
  5. 重启或 rebalance 后，Consumer 从上次提交的 offset 恢复消费。

默认投递语义： 至少一次（at-least-once）。若 Consumer 在处理后、提交前崩溃，重启后会重新处理。

如需 exactly-once：使用事务性 producer 将 offset 提交与输出写入原子性地捆绑。

消费者组 Rebalance

组内成员共享分区：每个分区分配给一个成员。

成员协议：

  1. 新成员加入；向组协调器 broker 发送 JoinGroup 向组协调器 broker。
  2. 协调器收集所有成员；选出一个 leader（成员之一）。
  3. Leader 计算分配方案（例如范围分配或轮询）。
  4. 协调器将分配结果广播给所有成员。
  5. 成员开始消费各自的分区。

成员离开或崩溃时触发 rebalance。Rebalance 期间：

  • 所有成员暂停消费。
  • 计算新的分配方案。
  • 成员恢复消费。

现代 Kafka 提供 增量协作式 rebalance（incremental cooperative rebalancing） 以减少全停顿影响：只有受影响的分区发生迁移，其余分区继续正常消费。

面试用语：

消费者组 rebalance 是运维上最棘手的部分。成员在 rebalance 期间暂停消费；长时间的 rebalance 会阻塞消息处理。协作式 rebalance 可降低中断程度。

存储与数据保留

每个分区在磁盘上由一系列日志段（log segment）组成。

  • 新写入追加到活跃段。
  • 当 segment 达到大小限制时进行滚动，并开始新的活跃段。
  • 旧 segment 按 topic 配置（时间或大小）保留。
  • 清理操作删除过期 segment。

压缩（按键保留）：

  • 对于配置了 cleanup.policy=compact.
  • 具有与较新记录相同键的旧记录可被删除。
  • 适用于只关心最新值的变更日志主题。

一致性模型

维度 保证
分区内顺序 强一致性。
跨分区顺序 无。
生产者 ack 可调（0、1、all）。
消费者读取 不超过高水位线（已提交）。
幂等生产者 分区内精确一次。
事务性生产者 跨分区/主题原子写入。

扩展与分区

关注点 策略
每主题吞吐量 增加分区数量。
每分区吞吐量 单主节点瓶颈；调整批处理大小。
Broker 磁盘 将分区分布到各 Broker 上。
热键 添加随机后缀进行子分区；消费者重新聚合。
消费者扩展 成员数最多增加至分区数；每个成员最多对应一个分区。

分区数量是关键决策：

  • 更多分区 = 更高并行度，但 Controller 负载更大、Rebalance 时间更长。
  • 更少分区 = 更低开销，但消费者扩展受限。

多区域（MirrorMaker / Cluster Linking）

  • 主集群位于一个区域。
  • MirrorMaker 将主题复制到从集群，用于灾难恢复或地理就近访问。
  • Cluster Linking（Confluent）：保留偏移量的原生跨集群复制。
  • 双向主主模式可行，但同一主题的冲突处理需格外谨慎。

故障模式与缓解措施

故障模式 缓解措施 监控指标
Broker 故障。 Controller 从 ISR 中选举新的主节点。 Broker 数量；副本不足的分区数。
所有 ISR 成员故障。 分区不可用；触发告警。 ISR 大小。
慢速从节点。 从 ISR 中移除；追上进度后重新加入。 ISR 收缩频率。
生产者重试产生重复消息。 幂等生产者去重。 生产者重试率。
消费者积压持续增长。 增加消费者；对积压触发告警。 每分区消费者积压。
长时间 Rebalance 阻塞工作。 协作式 Rebalance；静态成员资格。 Rebalance 频率与持续时间。
磁盘满。 执行保留策略；对使用率触发告警。 每 Broker 磁盘使用率 %。
热分区（单一生产者键）。 子分区；添加随机后缀分散写入。 每分区写入速率。
Controller 故障。 KRaft 共识故障转移。 Controller 健康状态。
跨区域复制延迟。 监控；定位背压来源。 镜像延迟。

运维指标

正确性：

  • 副本不足的分区数（应接近零）。
  • ISR 收缩失败次数。
  • 每主题生产者错误数。
  • 消费者提交失败次数。

容量：

  • 每分区吞吐量。
  • 每 Broker 磁盘 I/O。
  • 消费者积压。
  • Rebalance 持续时间。

业务层面：

  • 按团队划分的主题吞吐量。
  • 保留策略合规性。
  • 端到端延迟。

需要指出的权衡

  • 更多分区 = 更高并行度，但元数据开销更大。
  • acks=all 是最安全但最慢的选项。
  • 不干净主节点选举以可能丢失数据为代价换取可用性。
  • 幂等 + 事务性生产者增加延迟，但实现精确一次语义。
  • 压缩为键控主题节省空间，但增加了复杂性。
  • ISR 收缩风险 vs 要求高 min.insync.replicas.

常见陷阱与更优做法

陷阱 更优做法
将分区视为主题内的全局顺序。 顺序是分区内的；跨分区互相独立。
使用单分区实现全局顺序。 牺牲并行度；建议改用按键排序。
acks=1 到处都是。 acks=all 适用于正确性关键型主题。
忽视消费者积压。 对积压触发告警；自动扩展消费者。
无幂等性保障。 启用幂等生产者以确保安全重试。
Stop-the-world 式 Rebalance。 协作式 Rebalance；静态成员资格。

深入方向 优秀回答要点

  • 分区复制日志模型。
  • 每分区主从结构；ISR。
  • 生产者 ack 级别可调，用于控制持久性。
  • 每消费者组每分区的消费者偏移量。

面试用语

Kafka 是一个分区复制的仅追加日志。分区内强顺序保证、主从复制以及通过生产者 acks 可调的持久性，使其成为流处理的核心引擎。最难的正确性问题是精确一次（通过事务解决）和消费者组 Rebalance（通过协作式 Rebalance 缓解）。

2. 模拟面试

面试官： 设计 Kafka。

候选人： 由多个 Broker 组成的集群；每个主题跨 Broker 分区；每个分区通常以 3 副本复制，含一个主节点和若干从节点。生产者追加数据；消费者从偏移量轮询。消费者组分摊分区。Controller（现在是 KRaft）管理元数据。不变量是分区内强顺序，持久性取决于配置的 ack 级别，消费者组通过分配分区实现并行处理。

面试官： 带我走一遍生产流程。

候选人： 生产者按分区批量发送记录。若设置了键，则通过 hash(key) % num_partitions 选择分区，否则轮询。将批次发送到分区主节点。主节点追加到本地日志，复制到从节点。使用 acks=all时，主节点等待所有 ISR 成员确认后再向生产者返回成功。使用 acks=1时，主节点本地写入后即返回。使用 acks=0时，直接发送不等待确认。

面试官： 某个 Broker 宕机，会发生什么？

候选人： Controller 通过心跳超时检测到故障。对于该 Broker 担任主节点的每个分区，Controller 从 ISR（同步副本集）中选举新的主节点。发布新的 leader_epoch；生产者向新主节点重试。从节点继续从新主节点复制数据。如果故障 Broker 也是其他分区的从节点，那些分区的 ISR 会收缩；若低于 min.insync.replicas，这些分区将拒绝写入。

面试官： 什么是 ISR？

候选人： In-Sync Replicas——在配置的延迟阈值内与主节点保持同步的副本集合。ISR 是动态的：慢速从节点会被移除；恢复后重新加入。高水位线是所有 ISR 成员均已复制的最高偏移量；消费者只能看到高水位线以下的数据。 acks=all 等待 ISR 确认，提供最强持久性保障。

面试官： 消费者组如何知道读取哪个分区？

候选人： 成员资格协议。每个消费者通过 JoinGroup 向消费者组协调者 Broker 加入群组。协调者收集所有成员，选出一个作为 Leader，让它制定分配方案（range、round-robin 或 sticky）。协调者广播分配结果。每个消费者开始读取被分配的分区。成员变化时，触发 Rebalance 并重新计算分配。

面试官： Rebalance 会暂停工作吗？

候选人： 传统上是的——stop-the-world。所有成员暂停，计算新分配，成员恢复。对于高吞吐量消费者来说代价很大。现代 Kafka 支持协作式 Rebalance：只有发生迁移的分区才会暂停，其他分区继续处理。静态成员资格允许消费者在短暂崩溃后重新加入而不触发 Rebalance。

面试官： 精确一次？

候选人： 两个部分。第一，幂等生产者：每个生产者有一个 producer_id；每分区的序列号使 Broker 能够去重重试。消除单个生产者重试产生的重复消息。第二，事务性生产者：生产者开启事务，向多个分区/主题写入，原子提交。消费者配置为 read_committed 时只能看到已提交的记录。结合在同一事务中提交消费者偏移量，可实现读-处理-写流水线的精确一次语义。

面试官： 压缩？

候选人： 按主题策略。对于 cleanup.policy=compact，Broker 定期重写日志段，每个键只保留最新记录。相同键的旧记录被删除。适用于只关心当前状态的变更日志主题。墓碑记录（null 值）表示删除。压缩在后台运行，不阻塞写入。

面试官： 某个生产者键每秒产生 100,000 条消息，远超单个分区的处理能力。

候选人： 单个分区是该键的吞吐量上限。缓解方案：在键上追加随机后缀进行子分区，使同一逻辑键分散到多个分区。若需要按键处理，消费者必须重新聚合。权衡：以牺牲严格的按键顺序换取更高的吞吐量。

面试官： 存储？

候选人： 每个分区是磁盘上的一系列日志段。活跃段接收写入；当达到大小或时间限制时进行滚动。旧段按保留策略（时间或字节）保留。清理操作删除过期段。压缩（若启用）每个键只保留最新记录。Broker 大量使用页缓存；顺序磁盘写入速度非常快。

面试官： 多区域？

候选人： MirrorMaker（或 Cluster Linking）将主题复制到从集群。一个区域为主集群，另一个为灾难恢复或读就近访问的从集群。双向主主模式可行，但每一侧写入自己的主题；跨区域写入同一主题需要仔细处理冲突。大多数部署采用主备模式，并在需要时进行故障转移。

面试官： 总结。

候选人： Kafka 是一个分区复制的仅追加日志。Broker 将分区存储在磁盘上；每分区主从结构，带 ISR。生产者通过 acks 调节持久性；幂等生产者去重重试。消费者跟踪每分区偏移量；消费者组分摊分区实现并行。Controller（KRaft）管理元数据和主节点选举。压缩支持键控状态主题。通过 MirrorMaker 实现多区域复制。最难的正确性问题是精确一次；最难的运维问题是 Rebalance 影响。

3. 练习备注

五分钟版本

  1. 澄清：分区日志、复制、消费者组、保留策略。
  2. 不变量：分区内顺序；持久性取决于 ack 级别。
  3. 主题 + 分区；每个分区主从复制。
  4. ISR + acks 级别用于持久性调优。
  5. 消费者组 Rebalance 分配分区。
  6. 幂等 + 事务性生产者实现精确一次。
  7. 压缩用于键控状态主题。
  8. KRaft 用于元数据；MirrorMaker 用于多区域。

一分钟结尾陈述

Kafka 是一个分区复制的仅追加日志。分区内强顺序；每分区主从结构；通过生产者 acks 可调的持久性。消费者组分摊分区实现并行处理。幂等和事务性生产者实现精确一次。压缩支持基于键的保留。Controller（KRaft）管理元数据。MirrorMaker 处理多区域复制。最难的正确性问题是精确一次；最难的运维问题是 Rebalance。

最佳面试用语

  • "分区内强顺序；跨分区互相独立。"
  • "ISR 是同步副本的动态集合；HW 是已提交的偏移量。"
  • "acks=all 等待 ISR；每主题可调的持久性。"
  • "幂等 + 事务性生产者为读-处理-写流水线提供精确一次语义。"
  • "协作式 Rebalance 减少 stop-the-world 的影响。"
