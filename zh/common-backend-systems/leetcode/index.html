LeetCode — Code Judging Platform (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Medium · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建 LeetCode。用户在浏览器中浏览题目、编写代码、提交，并在数秒内看到评测结果和运行时统计。我们需要安全地运行来自陌生人的恶意代码。假设有 2000 道题目，100 万日活用户（DAU），约 500 万次/天的提交量，以及每周竞赛期间 10 万并发提交的峰值。

1. 需求（约 5 分钟）

功能需求 - 浏览题目并查看题目描述。 - 提交代码，获取评测结果（通过、错误答案、超时、内存超限、运行错误）。 - 查看提交历史和竞赛排行榜。

我会推迟讨论/论坛、付费订阅和竞赛组织功能。

非功能需求 - 沙盒化执行 — 运行任意代码，绝不能让其接触我们的基础设施。 - p95 评测延迟：正常负载 < 5 秒；竞赛峰值 < 15 秒。 - 提交持久性——绝不丢失竞赛提交。 - 排行榜最终一致性可接受（5-10 秒内）。 - 竞赛期间的公平性——相同题目、相同机器类型、确定性计时。

容量估算 - 500 万次提交/天 → 平均约 60 次/秒；竞赛峰值约 5 分钟内 10 万次 → 约 330 次/秒。 - 每次提交：少量源代码（< 10 KB）+ 少量评测输出。 - 提交归档主导元数据：假设每次提交 20 KB（代码/结果/日志）× 500 万/天 × RF=3 ≈ 300 GB/天，约 100 TB/年（压缩/分层前）。隐藏测试数据相对较小：2000 道题 × 50 MB/题约 100 GB。

2. 核心实体（约 2 分钟）

  • 问题 — problem_id、题目描述、测试用例（隐藏）、约束条件。
  • 提交 — submission_id、user_id、problem_id、language、code、status、runtime_ms、memory_kb。
  • TestCase — problem_id, idx, input, expected_output, is_sample.
  • 竞赛 — contest_id, problems[], start, end, participants.

3. API（约 5 分钟）

GET   /v1/problems                          → Problem[]
GET   /v1/problems/{id}                     → Problem (with sample cases)
POST  /v1/submissions                        { problem_id, language, code, contest_id? }  → { submission_id }
GET   /v1/submissions/{id}                   → { status, runtime, memory, failed_case? }
WS    /v1/submissions/{id}/stream            → status updates
GET   /v1/contests/{id}/leaderboard

从 token 中获取用户。提交时建议使用幂等键（可选）。

提交 POST 在 {user_id, problem_id, contest_id, idempotency_key}上是幂等的。排行榜游标编码 {contest_id, score, penalty, user_id}.

4. 数据流（约 5 分钟）

提交 → API 写入提交行（status=queued）→ 入队评测队列 → worker 取出 → 启动沙盒 → 编译 → 对隐藏测试用例逐一运行直到首次失败或全部通过 → 写入评测结果和统计 → 通过 WS 推送给客户端 → 若为竞赛则更新排行榜。

5. 高层设计（约 10–15 分钟）

  1. API 网关 / 提交服务。 验证输入，以 queued 状态，将任务入队至优先队列（Kafka 或 SQS），并推送一条 submission_created 排行榜订阅者的事件。
  2. 评测队列。 按语言和优先级（竞赛 vs 练习）分设独立队列。竞赛提交使用专用 worker 池，以防止练习流量抢占资源。
  3. 评测 worker（核心难点）。 取出任务，启动隔离的执行环境（gVisor、Firecracker microVM 或带严格 seccomp + cgroups + 无网络的 Docker）。编译源码，按顺序对隐藏测试用例运行，捕获 stdout，与预期输出比较，并测量 CPU 和内存使用。
  4. 测试用例存储。 使用带 CDN 前置缓存的对象存储（S3）；题目不可变，因此缓存非常简单。
  5. 评测结果服务。 将最终状态、运行时间、内存写入提交行，并将更新推送给 WS 订阅者。
  6. 排行榜服务。 Redis 中按 (score, time_penalty) 为键的每场竞赛有序集合——分数 = 通过题数，罚时 = 首次 AC 时间。由评测结果事件更新。
  7. 讨论/评论等功能 超出范围。

端到端叙述：Alice 提交解法 → API 写入行 → Kafka topic judge.python → worker 取出，启动预热的 Firecracker VM，编译，运行用例 1...30，用例 18 答案错误 → worker 写入评测结果，推送 WS 更新 → Alice 的浏览器显示「第 18 个用例 WA，期望 [1,2] 实际 [2,1]."

6. 深度探讨（约 10 分钟）

沙盒化——核心所在。 三层防护： - 进程隔离： Firecracker microVM 或 gVisor（每次提交使用轻量级 VM 级隔离）。安全性优于 Docker；启动速度快于虚拟机。 - 资源限制： cgroups 限制 CPU（单核）、内存（硬上限）、进程数、文件描述符。超时由 worker 而非 guest 强制执行。 - 无网络，无宿主文件系统。 seccomp 拒绝所有网络系统调用；根文件系统为只读，附带小型 tmpfs 临时空间。 - 提交和结果通过 stdin/stdout 传递——无共享卷。

为何选择 Firecracker/gVisor 而非普通 Docker：带 seccomp 和 cgroups 的 Docker 速度快，但共享内核攻击面更大。Firecracker 在预热后能以可接受的启动时间提供 VM 级隔离；gVisor 是两者的折中。面对恶意代码，我愿意多花几毫秒换取更强的隔离。

评测路径追踪：API 将提交写为 queued 并发布至 submission:{id} ；worker 取出，并行获取代码和测试清单，启动预热沙盒，编译，按顺序运行用例并施加 CPU/内存限制，流式传输进度事件，写入终态评测结果和对比输出，然后发出 judge.{language}.{priority}用于排行榜更新。 judge.result 热提交路径，逐步说明： 1. 客户端发送

并附带幂等键；API 按生成的 id 对 POST /v1/submissions 进行分片，并将 submission:{submission_id} 持久化写入后返回。 2. API 只发布 queued ；代码保留在提交存储中，以确保队列消息保持精简。 3. Worker 从语言/优先级分区中取出，并行读取代码和测试清单，并检查 {submission_id, language, priority} 与 judge.{language}.{priority}。 4. 若编译缓存命中，则立即开始执行；否则在沙盒镜像内编译，并可能写入缓存对象。 5. Worker 按确定性顺序运行测试，普通题在首次可见失败时停止，写入终态评测结果，并发出 compile:{lang}:{hash}供排行榜和 WebSocket 订阅者使用。 judge.result 冷路径：缓存未命中的 worker 从对象存储获取语言镜像、隐藏测试和编译工具链；这正是竞赛池需要为每种语言保持预热沙盒的原因。若 WebSocket 订阅者已断开，提交行仍是事实来源，客户端可通过轮询

获取结果。 GET /v1/submissions/{id}.

队列与存储选择的理由：Kafka 适合分区、可重放的竞赛流量；SQS 操作更简单，但有序重放和自定义优先级能力较弱。Redis 适合排行榜，因为每场竞赛都是从提交数据派生的小型热视图；提交数据库仍是权威来源。关系型数据库可存储题目和竞赛，但提交历史应按 submission_id 分片，因为写入占主导。

Worker 池大小。 每个 VM 生命周期较短（约 2 秒）。稳定状态 60 QPS 时，约需 120 个并发 worker；竞赛峰值 330 QPS 时需约 700 个。每种语言始终预热 50 个备用 VM 以吸收突发流量；按队列深度自动扩容。

竞赛期间的公平性。 竞赛提交使用确定性的测试机器类型——相同实例类型、相同 OS 镜像。Worker 每次只评测一个提交（不共享 CPU）。时间限制为挂钟时间加上充裕的余量，以吸收调度抖动。

评测结果流式 UX。 客户端提交后向 /submissions/{id}/stream 打开 WS 连接。Worker 可为长时间运行的提交发布进度（编译中、运行第 5/30 个用例）。优于轮询。

编译器缓存。 对于 C++ 提交，编译是耗时部分。以 hash(code + language + compiler flags) 为键缓存编译后的二进制文件。相同代码重新提交时跳过编译步骤。

作弊检测。 通过代码相似度哈希对同一题目的近期提交进行抄袭检测；标记审查，不自动处罚。对每个用户的提交频率进行限制（令牌桶算法）。

竞赛规模下的排行榜。 10 万并发提交者 → 5 分钟内产生 10 万条评测结果。Redis ZADD 能满足此速率；按 contest_id 分片是自然的选择，因为每场竞赛相互独立。

排行榜更新路径： judge.result -> 竞赛服务检查每位 {contest_id,user_id,problem_id} 在提交索引中的首次 AC 提交 -> 计算 score 。 penalty -> ZADD contest:{contest_id}:lb -> 缓存顶部页面 1 秒。练习提交绕过此路径。

提交状态机：

阶段 模式 数据源 状态转换 用户可见状态
已入队 持久等待 提交 DB + 评测队列 worker 接收消息 根据队列深度显示「已排队」及预计等待时间
编译中 隔离构建 沙盒 + 编译器缓存 编译成功/失败 「编译中」
运行中 测试执行 沙盒 + 隐藏测试 首次失败或全部通过 进度可选显示
基础设施重试 worker/沙盒丢失 提交行 + 队列重放 以相同 id 重试一次 仍处于等待中，不重复计分
终态 评测结果已写入 提交 DB AC/WA/TLE/MLE/RE 最终评测结果 + 统计数据

双视图框架。 提交行是预写事实，而 WebSocket 流和排行榜是派生视图。若流或 Redis 排行榜丢失，通过重放终态提交行的 judge.result 事件可重建用户可见视图。

边缘压力测试。 竞赛流量峰值路由到专用竞赛队列，练习流量优先降级。包含大型隐藏测试文件的热门题目应在 worker 本地缓存中固定；冷缓存未命中需支付对象存储延迟，但通过预热可保持在竞赛 p95 以内。恶意代码无网络访问、进程数受限、文件系统只读，并有硬挂钟时间强制终止。若用户深度翻阅提交历史，API 将使用 {last_submission_id, created_at} 游标从热近期分区切换到归档行。多区域公平性通过在单一区域/机器类型中评测竞赛并向外复制结果来保障。

面试要点："事实来源是提交行；其他所有内容，包括竞赛排行榜，都是由判题结果驱动的衍生视图。"

我会推迟的内容。 题目编写工具、竞赛报名、讨论串、语言互操作性（自定义判题器、交互式题目）、付费订阅计费。
