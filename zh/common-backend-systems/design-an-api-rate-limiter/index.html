1. 基础知识与解题思路

本题考察点

限流器处于每个请求的热路径上，必须在个位数毫秒内做出允许或拒绝的决策。难点在于：在保持决策全局一致性以保护下游的同时，自身不能成为它本应防止的瓶颈。

其不变量如下：

对于任何限流键（用户、IP、API key、路由、租户或其组合），在任意滑动窗口内允许的请求数不得超过配置的配额，即便流量分散在众多边缘节点且客户端频繁重试。

在 ICT5 / L6+ 级别，面试官考察的是：能否针对新鲜度与成本的权衡选择合适的计数算法、设计能应对热点键的分布式计数器，以及解释限流器自身不可用时的处理方案。

60 秒面试框架

口述版本：

我会将限流器设计为每个 API 节点上的 sidecar 或内联中间件，由快速的分布式计数器存储作为后端。关键决策包括：选用哪种计数算法（固定窗口、滑动窗口、令牌桶、漏桶）、如何在不为每次请求支付完整往返开销的情况下跨边缘节点保持计数一致性，以及计数器存储降级时系统如何选择 fail-open 或 fail-closed。我将先明确需求，选定令牌桶作为默认算法，梳理数据路径，再深入讨论热点键与一致性问题。

首先明确需求

功能性需求：

  • 按身份（用户、API key、IP、租户）和路由对请求进行限流。
  • 同时支持多个限流窗口（例如：100/秒、1000/分钟、10000/小时）。
  • 不同身份可以拥有不同的配额；企业客户享有更高限额。
  • 返回清晰的反馈： 429 Too Many Requests 与 Retry-After 以及配额相关的响应头。
  • 允许短暂突发流量超过持续速率（突发配额）。
  • 允许管理员在不重新部署的情况下，在运行时配置、提升或撤销配额。

非功能性需求：

  • 允许/拒绝的决策延迟在 p99 个位数毫秒级。
  • 不能成为 API 的单点故障。
  • 必须能扩展至数百万个不同的键。
  • 必须能处理热点键（某个租户或 IP 接收不成比例的流量）。
  • 对被拒绝请求和配额变更具备可审计性。

除非面试官要求，否则超出范围：

  • 网络层的 DDoS 攻击（在到达 API 限流器之前已处理）。
  • WAF 规则及基于内容的封锁。
  • 业务层的计费与计量。
  • 并发限制（最大在途请求数）与速率限制（每单位时间）的区别——需确认是否两者都需要。

规模假设

领域 假设
API 流量 全球峰值 1M 请求/秒。
独立键数量 超过 100M 个活跃的限流键（每用户 × 每路由的组合）。
热点 key 前 1% 的键占据超过 50% 的流量。
配额更新 每天数千次；大多为自动化的等级变更。
限流器延迟预算 在 API 路径上增加的延迟 p99 低于 5 ms。
精度容忍度 在边界处，根据算法不同，误差在配置限额的 1-5% 以内。
故障模式 按路由配置：非关键路由 fail-open（允许），敏感路由 fail-closed（拒绝）。

瓶颈不在计数器存储，而在热点键。单个用户以每秒 10 万次请求的速度驱动，会锁定某一个分片。

核心产品模型

界面 用户体验 系统需求
API 调用方 请求要么被处理，要么收到带有明确重试信号的 429 响应。 决策必须足够快且准确，确保合法流量不被拦截。
管理员 配置配额、等级、覆盖规则；查看拒绝请求的仪表盘。 配置需强一致性；并复制到所有限流节点。
下游服务 信任流量处于约定配额之内。 限流器在正常运行时不得超额放行。

核心组件

组件 职责 关键细节
边缘层 / API 网关 负责 TLS 终止，在到达应用层之前实施粗粒度的 IP 级限流。 低成本的第一级过滤器；在内存中使用简单的固定窗口计数器。
限流中间件 对每个请求做出允许/拒绝决策；运行在 API 进程中或作为 sidecar。 对热点键使用本地令牌桶副本；降级时回退到计数器存储。
计数存储 每个键的权威分布式计数器 (key, window). 通常是分片的内存 KV 存储，如 Redis 集群；读写延迟低于 1 ms。
配额配置服务 存储每个身份和每个路由的配额；支持更新。 通过 pub/sub 推送或通过版本控制轮询。
算法 令牌桶、漏桶、固定窗口或滑动窗口日志/计数器。 根据突发容忍度和精度需求，按路由选择。
拒绝日志 记录每一个 429 响应，包含键、原因、配额和当前计数。 异步写入；用于分析和滥用检测。
管理 API 对配额进行增删改查；紧急覆盖操作。 已审计、签名且限角色访问。
兜底决策 计数器存储不可达时的处理方案。 按路由配置：fail-open 或 fail-closed。

算法选择（选对适合的算法）

算法 工作方式 优点 缺点
固定窗口 按固定时间桶（如每分钟）统计请求数，在桶边界处重置。 最简单；每个键只需一个计数器。 窗口边界处的突发流量可能导致跨两个窗口的放行量达到配额的 2 倍。
滑动窗口日志 存储每个请求的时间戳；统计窗口内的请求数。 最精确。 内存开销随流量线性增长。
滑动窗口计数器 对当前和上一个固定窗口进行加权混合。 开销低；对滑动窗口的近似效果好。 在接近限额时存在轻微的不准确性。
令牌桶 大小为 N 个令牌的桶，以速率 R 补充；每个请求消耗一个令牌。 支持受控突发，最多可达桶容量；对于"最大持续速率"的策略直观易懂。 需要原子性的扣减加上补充计算。
漏桶 具有固定泄漏速率的队列；请求加入队列或被丢弃。 平滑输出速率；适合流量整形。 因排队引入额外延迟。

默认推荐： 令牌桶 适用于面向客户端的 API，因为其策略易于人类理解，且突发是一等概念。对同一个键组合使用多个窗口：例如，一个 100/秒的桶和一个 10000/小时的桶，两者同时允许才放行。

面试用语：

我会将令牌桶作为默认选择，因为其策略对人类可读，且突发是一等概念。当同一个键需要同时应对短期和长期保护时，我会在不同窗口上叠加多个桶。

API

  identities: [user_id?, api_key?, ip?, tenant_id?]

令牌桶实现细节

经典的原子操作：

on each request:
  state = read BucketState(bucket_key)
  now = current_time_ms
  elapsed = now - state.last_refill_ts
  tokens_to_add = elapsed * refill_rate / 1000
  state.tokens = min(capacity, state.tokens + tokens_to_add)
  state.last_refill_ts = now
  if state.tokens >= cost:
    state.tokens -= cost
    write BucketState (CAS or atomic Lua)
    return ALLOW
  else:
    return DENY with retry_after = (cost - state.tokens) / refill_rate

两种实现选择：

  • Redis 加 Lua 脚本：在单次往返中原子性地对桶执行读-改-写操作。每个 API 节点调用该脚本。
  • 本地近似值 + 异步同步：每个节点为热点键维护一个本地令牌桶，并定期以 delta 的方式同步到中央计数器存储。以牺牲精度换取低延迟和更低的计数器存储压力。

对于热点键：本地近似值至关重要。同步调用中央存储的方案无法处理单个键每秒 10 万 QPS 的场景。

热点键策略

单个租户以每秒 10 万请求的速度消耗其配额，会锁定持有该键的 Redis 分片。

缓解措施：

  • 优先本地限流。 每个 API 节点为每个键维护一个本地令牌桶，容量为 local_capacity = total_capacity / N 其中 N 即节点数量。突发流量在本地处理，无需协调。
  • 节点级子配额。 通过静态分配或 gossip 协议将配额预分发到各节点；节点仅在其份额接近耗尽时才联系中央存储。
  • 分层限流器。 边缘节点在本地强制执行全局配额的 80%；中央节点对跨节点突发流量执行剩余的 20%。
  • 对键进行分片。 对于单个极热的键，在限流器层将其拆分为 N 个子键；每个子键承载 capacity/N，请求根据哈希映射到子键。接受一种情况：即便某个大客户的全局配额未耗尽，仍可能因子键层面被略微限流。
  • 副本读取。 对于能容忍轻微滞后的算法，从本地副本读取，并在后台进行调和。

面试用语：

中央计数器存储是精度的兜底手段，而非热点键的内联处理路径。我将执行逻辑下推到边缘，使用本地桶进行异步调和，以有界的超额放行换取高可用性。

写路径与读路径

请求路径（每次 API 调用）：

  1. API 节点接收请求。
  2. 解析身份：从鉴权 token 中提取用户信息、路由、租户和 IP。
  3. 从本地配额缓存（每隔几秒从配额服务刷新）中查找适用的配额。
  4. 对每个配额，检查本地令牌桶；若任意一个返回 DENY，则返回 429。
  5. 若所有本地检查均放行，可选择性地与中央计数器存储同步，用于高优先级配额。
  6. 将请求转发给 API。
  7. 若被拒绝，发出拒绝事件；否则在同步路径上不记录日志。

配额配置更新路径：

  1. 管理员通过管理 API 更新配额。
  2. 配额服务写入强一致性存储。
  3. 在 pub/sub 上发布配额版本变更事件。
  4. 限流节点在下一次轮询或推送时获取新版本（附带版本号），然后原子性地替换本地规则。

扩展与分区

数据 分区键 原因
计数器存储的桶 对 bucket_key 跨 Redis 集群分片进行哈希。 分布均匀；支持每个键的原子操作。
配额配置 复制到所有限流节点。 读多写少，数据量小，变更频率低。
拒绝日志 按时间分桶；按路由或租户分区。 在攻击期间数据量可能极大；适用于事后分析。
热点键拆分 逻辑键 bucket_key:shard_id 针对已知热点键。 将负载分散到多个分片。

一致性模型

决策 一致性 原因
正常键的允许/拒绝 跨节点最终一致（漂移范围为毫秒到秒级） 可接受少量百分比的超额放行。
热点键的允许/拒绝 本地优先；中央作为兜底 无法承受每次请求的同步往返开销。
配额配置 通过版本化配置实现强一致性 运营人员期望变更能够可预期地生效。
拒绝日志 最终一致 用于分析，而非决策。

接受分布式限流本质上是近似的这一现实。合同是"我们不会显著超过配置速率的放行"，而非"我们不会多放行哪怕一个请求"。

故障模式及故障模式选择

失败 缓解措施 故障模式默认行为
计数器存储不可达 本地桶继续服务；恢复后中央计数同步跟上。 低级别路由 fail-open；计费/鉴权/安全路由 fail-closed。
计数器存储热点分片 提前对热点键进行分片；使用本地近似值。 以降级精度 fail-open。
配额配置服务宕机 限流器使用最后已知的配置。 实际上是冻结状态，但功能正常。
节点间时钟偏差 使用服务器时间进行补充；限制漂移；本地使用单调时钟。 在补充边界处存在轻微不准确。
限流中间件崩溃 Sidecar 重启；进程内中间件随 API 进程一同失败。 与 API 进程同命运。
在窗口边界处的协调突发问题（固定窗口缺陷） 切换到滑动窗口或令牌桶。 算法选择问题。

选择 fail-open 还是 fail-closed 是产品策略决定：支付接口选 fail-closed；状态页面选 fail-open。

多区域策略

对于全球 API，配额通常意味着单一全局速率。

两种方案：

  • 按区域配额 （拆分为 N 份）：简单，无需跨区域协调，但空闲区域的"冗余容量"只有在重新分配时才能被繁忙区域使用。
  • 全局配额加分层执行：每个区域在本地执行其基准份额；全局调和器追踪总使用量。若某区域的份额未使用，其他区域可申请临时租约。

对大多数 API 而言，按区域拆分已足够好。突发情况下的跨区域精度极难保证，且鲜少被要求。

运维指标

正确性指标：

  • 每个键的实际速率与配置限额的对比（应保持在限额以内或以下）。
  • 超额放行比率：在持续流量期间 actual / configured 的比值。
  • 各限流节点间的配额配置版本偏差。

容量指标：

  • 计数器存储的 QPS、延迟和分片分布情况。
  • 限流中间件的决策延迟。
  • 按 QPS 和拒绝率排名的 Top-N 热点键。

业务与安全指标：

  • 按路由、租户、身份划分的拒绝率。
  • 429 转 200 的重试成功率。
  • 峰值检测：某一身份的拒绝数突然激增（可能是机器人或账号被盗）。
  • 配额变更审计跟踪。

需要指出的权衡

  • 滑动窗口日志最为精确，但开销较大；滑动窗口计数器以牺牲若干百分比的精度为代价，将内存占用降低数个数量级。
  • Token bucket 允许突发流量；leaky bucket 则将其平滑处理。根据是否欢迎突发性来做选择。
  • 同步中心计数器精度高，但限制吞吐量；本地近似方案速度快，但允许量略有超出。
  • 过于严格的限制能保护下游，但会引发用户投诉；过于宽松的限制能保护用户体验，但会暴露下游。
  • 按 IP 限流会伤害 NAT 后或大型企业代理后的用户；按 API key 限流需要在限流路径上进行身份认证。
  • 失败开放对用户友好，但会放过滥用行为；失败关闭保护下游，但会导致故障范围扩大。

常见陷阱与更优做法

陷阱 更优做法
仅使用固定窗口。 指出边界突发缺陷；选择滑动窗口或 token bucket。
用单个 Redis 实例存储所有计数器。 使用分片集群；将集群本身作为一个分层并针对分片饱和进行设计。
每次请求都同步往返 Redis。 本地桶 + 异步协调；分层执行。
忽视配额更新路径。 带显式传播与审计的版本化配置。
忘记故障模式。 声明每条路由的失败开放或关闭策略并说明理由。
平等对待所有 key。 识别热点 key，提前分片或使用近似本地桶。

面试官可能深入探讨的方向

深入方向 优秀回答要点
算法选择 对比 token bucket、滑动窗口计数器、固定窗口，说明为何以 token bucket 作为默认选择。
热点 key 处理 本地优先执行；预分片；基于 gossip 的容量重新分配。
分布式一致性 接受有界误差；解释本地 + 中心协调如何工作。
故障模式 按路由的失败开放与关闭；如何检测与恢复。
多区域 按区域拆分与分层全局方案的对比；权衡分析。
实现 展示用于原子 token bucket 的 Redis Lua 脚本。

Staff 级别回答检查清单

  • 声明不变量：即使在重试情况下，也不得超过每个 key 的配置速率。
  • 选择一种算法并说明理由。
  • 区分热路径执行与精确中心计数。
  • 用本地桶和预分片解释热点 key 处理。
  • 将配额配置平面与数据平面分开介绍。
  • 声明每条路由的失败开放或关闭策略。
  • 讨论多区域配额拆分。
  • 将拒绝日志作为安全审计的异步旁路通道。

面试用语

分布式限流是一个近似问题，而非精确计数问题。我愿意接受几个百分点的超限，以换取亚毫秒级决策和对计数器存储故障的韧性，并只在真正重要的地方为精度付出代价：计费、安全和滥用拦截。

2. 模拟面试

面试官： 设计一个 API 限流器。

候选人： 我将以每请求允许/拒绝中间件为核心进行设计，后端由分布式计数器支撑。不变量是：对于任意限流 key——用户、API key、IP、租户、路由或其组合——我们在整个集群中不会显著超过配置的配额。我将从澄清需求开始，选择算法，梳理数据路径，并深入讨论热点 key 与一致性问题。

面试官： 你会选择哪种算法？

候选人： 默认选用 token bucket。其语义直观易懂：固定的补充速率和突发容量。对于长窗口下接近限制时的精度要求，我有时会叠加一个滑动窗口计数器作为第二个桶。固定窗口最简单，但存在边界突发问题——在窗口边界附近可能获得 2 倍的限额，因此对面向客户端的 API 我会避免使用。

面试官： 计数器存放在哪里？

候选人： 逻辑上存放在分片的高速 KV 存储（如 Redis 集群）中，通过 Lua 脚本原子执行读-改-写。实践中，每个 API 节点针对活跃 key 维护本地 token bucket，中心计数器作为精度的兜底。对于热点 key，每个节点从本地份额执行，无需往返；否则中心存储将成为瓶颈。

面试官： 如果单个租户每秒产生 100,000 个请求怎么办？

候选人： 这就是热点 key 问题。有几种缓解方案：第一，将配额分配给各 API 节点，让每个节点在本地执行其份额。第二，将 key 拆分为子分片：将请求哈希到 N 个逻辑桶中的某一个，每个桶容量为 capacity/N。用户在未达到全局上限时可能会被轻微限流，这是可接受的权衡。第三，如果中心存储开始饱和，限流器退回到纯本地执行，并接受超限。

面试官： 精度如何？

候选人： 在本地优先执行和异步协调下，突发期间可能超限几个百分点，尤其是在流量突然激增之后。这对几乎所有限流场景来说都是可接受的。如果某路由需要严格计数——计费计量、安全拦截——我会通过同步 Redis Lua 脚本处理，并接受更高延迟。

面试官： 如果 Redis 宕机怎么办？

候选人： 中间件回退到本地桶。这里的配置很关键：对非关键路由，失败开放并接受部分超限；对计费和认证，失败关闭。每条路由都有一个失败模式策略。拒绝日志会记录回退情况，以便事后审计。

面试官： 如何处理配额更新？

候选人： 配额配置是独立于数据平面的控制平面。管理 API 写入强一致性存储，递增版本号，并在 pub/sub 上发布变更事件。限流节点通过订阅或轮询获取更新，并原子地切换到带有新版本号的新配额。运维人员可通过查询每个节点上的活跃版本来验证传播情况。

面试官： 你会返回什么状态码和响应头？

候选人： 429 Too Many Requests 与 Retry-After, X-RateLimit-Limit, X-RateLimit-Remaining，以及 X-RateLimit-Reset 响应头。retry-after 根据 token bucket 计算得出 (cost - tokens) / refill_rate。客户端可以据此精确退避。

面试官： NAT 后或企业代理后的 IP 如何处理？

候选人： 按 IP 限流会不公平地伤害 NAT 后的用户——整个办公室共享一个配额。因此我会组合限制：一个粗粒度的按 IP 防护用于匿名滥用保护，一个按已认证身份的限制用于实际产品配额。在 API 内部，我尽可能优先使用基于身份的限制。

面试官： 多区域？

候选人： 最简单的方式是按区域拆分配额：每个区域获得全局配额的一份，在本地执行。无需跨区域协调，但安静区域未使用的容量会被浪费。如果产品需要单一全局配额，我会添加一个全局协调器，定期跟踪用量并在区域间重新分配租约——以边际精度提升换取运维复杂度。大多数 API 使用按区域拆分就足够了。

面试官： 如何支持不同的服务层级——免费、专业、企业？

候选人： 配额配置将身份属性映射到算法和容量。请求到来时，限流器从认证上下文中解析服务层级并查询适用的配额。层级变更即配置变更，下次配置刷新时生效。对于极高并发的企业客户，我可能会预先对其 key 进行分片，以避免单个 Redis 分片成为瓶颈。

面试官： 如何检测客户滥用配额？

候选人： 拒绝事件流向分析管道。模式包括：持续 100% 拒绝率（始终处于上限）、以前安静的身份突然爆发（账号被盗），或拒绝集中在可疑路由上。告警会触发安全审查，并可能自动吊销 API key。

面试官： 总结设计。

候选人： 算法选用 token bucket，每个 API 节点本地优先执行，分片 Redis 集群作为精度兜底。热点 key 拆分为子桶。配额配置是独立的版本化控制平面。每路由配置失败模式策略。多区域默认使用按区域配额拆分。整体设计接受分布式限流本质上是近似的，以快速决策和高可用性为优先目标，而非精确计数。

3. 练习备注

五分钟版本

  1. 澄清：按身份、按路由、多窗口、失败模式的限制。
  2. 声明不变量：不得显著超过配置速率。
  3. 选择 token bucket；与固定/滑动窗口对比说明理由。
  4. 展示数据路径：本地桶 → 可选的中心 Redis 检查。
  5. 用本地优先和 key 分片解决热点 key 问题。
  6. 展示独立于数据平面的配额配置平面。
  7. 讨论每路由的失败开放与失败关闭。
  8. 以多区域按区域拆分和拒绝日志作为收尾。

一分钟结尾陈述

限流器是一个近似系统。我会使用 token bucket 提供人类可读的配额，将执行下沉至进程内本地桶以保持热路径快速，并以分片 Redis 集群作为兜底。热点 key 预先分片；配额配置有独立的版本化平面。失败模式按路由配置。核心权衡是精确计数与可用性，我优化可用性，接受几个百分点的超限。

最佳面试用语

  • "分布式限流是一个近似问题，而非精确计数问题。"
  • "Token bucket 是默认选择，因为突发性是一等概念。"
  • "本地优先执行；中心存储是兜底，而非内联路径。"
  • "热点 key 会破坏朴素设计；预先对 key 分片或从逻辑上拆分它。"
  • "失败模式是按路由的策略决策，而非全系统设置。"
