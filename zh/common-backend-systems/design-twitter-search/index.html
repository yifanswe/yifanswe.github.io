1. 基础知识与解题思路

本题考察点

Twitter Search 是针对高速推文流的实时搜索——每天数百万条推文，数秒内完成索引，支持按文本、话题标签、用户、位置查询。难点在于保持索引足够新鲜以使突发新闻在数秒内可被搜索，同时确保对多年历史数据的查询能在亚秒级完成。

其不变量如下：

每条已发布推文在有界新鲜度内完成索引（近期数据通常为数秒，长尾数据为数分钟）；搜索查询按时效性和互动信号返回相关推文；在查询时应用可见性过滤（屏蔽、删除、隐私）；索引随数据量和查询量扩展。

在 ICT5 / L6+ 级别，面试官在考察你能否设计出针对高写入流的分片倒排索引、实时与批量索引分层、评分与排序，以及读取侧查询规划器。

60 秒面试框架

口述版本：

我会将 Twitter Search 设计为两层倒排索引：实时层用于在数秒内索引突发内容，主层用于批量构建历史内容。查询在两层之间进行 scatter-gather，按分数和时间合并，从推文存储填充推文内容，应用可见性过滤，然后返回。关键决策在于实时索引、分片策略、排序和可见性。

首先明确需求

功能性需求：

  • 支持按文本、话题标签、用户、语言、位置搜索。
  • 按时效性、热度或相关性排序。
  • 按日期范围、语言、媒体类型过滤。
  • 突发新闻的实时结果。
  • 热门话题（相关但独立）。

非功能性需求：

  • 新推文在数秒内完成索引。
  • 查询 p99 < 500ms。
  • 历史数据可被搜索，跨越多年。
  • 多地区。

以下内容超出范围，除非被特别询问：

  • 推文创建（参见 设计 Twitter).
  • 超出基础范畴的个性化推荐。
  • 私信搜索。

索引分片

每个分片持有索引的一个子集。分片策略：

  • 按时间： 每个分片存储某一时间范围内的推文。适合时间范围查询；近期热点分片存在热点风险。
  • 按哈希： 按 tweet_id 哈希分布。负载均衡；每次查询命中所有分片。
  • 混合式： 时间分层 + 层内哈希。

实践中，实时层按哈希分片以获得写入吞吐量；主层按时间 + 哈希分片以适应查询模式。

具体的索引存储与会话状态

存储 / 索引 键模式 值 容量/上限 TTL/生命周期 分片键
推文内容 tweet:{tweet_id} 作者、文本、媒体引用、状态 每条推文 1-2KB 永久保存，直至删除/标记墓碑 tweet_id
实时倒排列表 live:{term_hash}:{shard_id} 近期 (tweet_id, timestamp, score) 倒排列表 每条倒排记录 20-32B；按时间限制内存上限 3-7 天 term_hash + 实时分片
主索引段 main:{time_bucket}:{term_hash}:{segment_id} 压缩倒排块 每条倒排记录压缩后 4-16B 不可变；合并压缩 time_bucket + term_hash
词条统计 term_stats:{term_hash}:{day} 文档频率、语言、垃圾先验 每词条每天 100-300B 热点数据保留 30-90 天 term_hash
可见性缓存 visibility:{viewer_id} 屏蔽/静音/保护作者 ID 最多 1 万个 ID，最坏情况 80-160KB 对强一致性存储的 30-60 秒缓存 viewer_id
搜索游标 不透明 cursor_id 或签名 blob 查询计划、最后分数、分片偏移量、时间窗口 0.5-2KB 10-30 分钟 viewer_id + 查询哈希

容量估算：

  • 若每天 5 亿条推文每条索引 20 个词条，且每条实时倒排记录为 24B，实时索引的增长量为每天 240GB 原始数据。7 天 3 副本约为 5TB，仅在压缩和活跃词条过滤后才能放入内存。
  • 主索引以不可变压缩段的形式存储相同的推文语料库。按每条倒排记录压缩后 8B 计算，一天约 80GB 原始数据；5 年 3 副本（不含文档存储和元数据）约为 438TB。
  • 推文内容是独立的填充存储：每天 5 亿条 * 1.5KB = 每天 750GB 原始数据。搜索索引应返回 ID 和分数，而非完整的推文正文。

这是两索引模式，而非两个数据源：实时索引和主索引是同一推文流的两个视图。实时视图是可变的，侧重于近期数据；主视图是紧凑的，侧重于历史数据。

技术选型依据：

  • 使用倒排索引，因为访问模式是词条 → 候选推文 ID；按推文 ID 扫描键值存储方向相反。
  • 主层使用 Lucene 风格的不可变段，因为段合并和压缩是一等公民特性。宽列存储可以保存倒排行，但评分、压缩和 Top-K 检索都需要在应用层实现。
  • 索引使用追加流，因为推文写入不应阻塞在每次词条索引更新上。同步索引可获得更高的新鲜度，但会将发帖延迟与最慢的索引分片耦合在一起。

做好回答"为什么不用 Cassandra？"的准备。答案是 Cassandra 擅长存储行数据，但搜索需要压缩倒排块、跳表、评分和段合并。

索引构建流水线

每条新推文处理流程：

  1. 推文发布；事件发送至数据流。
  2. 实时索引器消费事件。
  3. 对文本进行分词：小写化、规范化、分割、去停用词。
  4. 提取实体：话题标签、@提及、URL、语言。
  5. 更新实时索引分片中的倒排列表。
  6. 推文在数秒内可被搜索。

定期执行（每小时或每天）：

  1. 实时层淘汰旧内容。
  2. 批量索引器处理已淘汰的推文。
  3. 构建压缩主索引段。
  4. 合并至主层。

查询路径

以"italian restaurants near NYC"为例：

  1. 前端解析查询：词条、运算符、过滤器。
  2. 查询改写器扩展：同义词、拼写纠正。
  3. 查询规划器确定查询哪些分片（例如，对时效敏感的查询，命中实时分片 + 近期主分片）。
  4. 分散至选定的分片。
  5. 每个分片对查询词条进行倒排列表求交。
  6. 返回带有评分数据的 top-K 候选结果。
  7. Merger 跨分片聚合、打分，返回 top-K。
  8. 从推文存储中填充推文内容。
  9. 可见性过滤。
  10. 返回结果。

延迟预算：\n- 分片查询：~50-100ms。\n- 合并：~20ms。\n- 填充：~50ms。\n- 可见性：<10ms。\n- 总计：500ms 以内。

热点突发新闻查询路径：

  1. 客户端发送 q="#oscars", viewer_id、过滤条件及可选游标。网关对规范化查询进行哈希，以检查 5-15 秒的结果缓存。
  2. 查询规划器选择活跃分片加上最新的主时间桶。分片键为 term_hash；扇出调用并行执行。
  3. 每个分片从内存中读取 live:{term_hash}:{shard_id} 以及最近的 main:{time_bucket}:{term_hash}:{segment_id} 块（来自 SSD 缓存）。
  4. 各分片返回 top 1k 推文 id 及低成本评分。Merger 在全局保留 top 2k，然后调用排序器。
  5. 填充操作按 tweet-id 分片批量执行 tweet:{tweet_id} 读取，同时并行读取可见性数据 visibility:{viewer_id} 。
  6. 超过单分片超时的慢速分片将被丢弃；响应携带 partial=true 以及记录了已包含分层和分片偏移量的游标。

冷历史查询路径：

  1. 查询规划器按 from/until缩小范围，再将时间桶映射到主索引段。
  2. 段缓存未命中时，从对象存储/SSD 加载压缩的倒排块。这是唯一允许超过正常 100ms 分片预算的路径。
  3. 若用户翻页超出热点 top-K 窗口，游标将从 (query_hash, time_bucket, shard_offsets, last_score, visibility_version) 继续，而非使用偏移量分页。
  4. 已耗尽的游标返回空页和终止游标；不会从头重新扫描。

查询降级状态：

阶段 模式 数据源 延迟目标 游标行为
1 完整热点结果 live + 已缓存的主段 <300ms 游标记录分片偏移量。
2 部分结果 live + 成功响应的主分片 <500ms 游标标记缺失分片以供重试。
3 冷历史 来自 SSD/对象存储的主段 <1.5s 游标固定时间桶。
4 窗口已耗尽 游标之后无更多倒排列表 <100ms 返回终止游标。

搜索服务可以在降低相关性之前先降级正确性：丢弃慢速分片或旧分层，但绝不跳过填充和可见性检查。

尾部延迟

Scatter-gather：最慢的分片决定延迟上限。

缓解措施：\n- 向副本发送对冲请求。\n- 单分片超时。\n- 容忍部分结果（丢弃最慢的 1%）。\n- 保留容量余量。

排序

多信号：\n- 词项匹配相关性（BM25）。\n- 作者权威度。\n- 互动量（点赞、转推）。\n- 时效性。\n- 个性化（浏览者的互动历史）。\n- 认证账号加权。

多阶段：每分片低成本打分；在此基础上进行高成本排序。

对于实时/突发内容：时间衰减权重较高。

热门话题

相关但独立的流水线：

  • 按区域、每分钟聚合话题标签提及次数。
  • 与基线（前一时段）对比。
  • 涨幅最大者 = 热门。
  • 按区域和用户兴趣个性化呈现。

可见性过滤器

在查询结果时应用：

  • 屏蔽：过滤被屏蔽作者的推文。
  • 静音：过滤被静音账号/关键词的推文。
  • 删除：过滤已删除推文（填充时可见状态）。
  • 隐私：受保护账号仅对已批准的关注者可见。
  • 政策：已下架推文。

绝不将可见性规则写入索引；在读取时进行过滤。

多区域策略

  • 每个区域拥有各自的 live 索引和主索引。
  • 推文流在全球范围内复制。
  • 查询在本地提供服务。
  • 个性化数据按用户所属主区域存储。

运维指标

正确性：

  • 可见性违规。
  • 陈旧结果率。

容量：

  • 按区域统计的查询 QPS。
  • 索引延迟。
  • 分片 p99。

业务层面：

  • 搜索结果 CTR。
  • 查询模式。

需要指出的权衡

  • Live 分层更新快但内存开销大；滚动下线至关重要。
  • 主分层成本低但更新较慢。
  • 按时间分片的查询对时效性排序很快，但通用查询会命中每个分片。
  • 哈希分片能均衡负载，但每次查询都会命中所有分片。
  • 激进排序能提升相关性，但会增加延迟。
  • 个性化与确定性之间的权衡。

常见陷阱与更优做法

陷阱 更优做法
单一索引分层。 Live + 主两级分层。
写入推文时同步更新索引。 通过流异步更新。
仅按时间分片。 时间 + 哈希混合分片。
在索引时应用可见性规则。 在读取时应用。
一次性排序。 多阶段排序。
无尾部延迟缓解。 对冲请求 + 部分响应。

面试用语

Twitter Search 是一个两级倒排索引：live 级用于突发内容（秒级），主级用于历史内容（小时级）。异步流水线从推文流中建立索引。查询跨相关分片 scatter-gather，多阶段排序，填充推文内容，应用可见性过滤器。对冲请求处理尾部延迟。最难的约束是在支撑数万亿历史推文的同时，保证突发新闻的索引延迟。

2. 模拟面试

面试官： 设计 Twitter Search。

候选人： 两级倒排索引。Live 分层在秒级内对新推文建索引——驻内存或 SSD，按哈希分片以提高写吞吐量，保留最近数小时数据。主分层用于长期存储：从滚动下线的推文批量构建，压缩存储，按时间+哈希分片。查询分散到相关分片（时效性敏感时用 live + 近期主索引；通用查询范围更广），按评分合并，填充内容，应用可见性过滤器后返回。

面试官： 为何要两级分层？

候选人： 优化目标不同。Live 分层需要亚秒级索引，无法承受批处理开销。主分层需要高效存储数万亿推文，批处理压缩效果更好。若将两者合并为一级并同时满足两种特性，代价过高。查询合并非常直接——recent 结果来自 live 分层，older 结果来自主分层，排序即可。

面试官： 突发新闻的索引延迟。

候选人： 流消费者（Kafka）向 live 索引器推送数据。分词、提取实体、更新 live 分片中的倒排列表。端到端仅需数秒。对于极热词项（如"breaking"），可进一步对倒排列表分片，以应对高写入速率。

面试官： 查询路径。

候选人： 前端解析、改写查询（同义词、拼写纠正）。规划器选择分片：时效性敏感时选 live + 近期主索引；通用查询范围更广。分散到各分片。每个分片对查询词项的倒排列表求交集，用低成本特征（BM25 + 静态信号）对候选结果打分，返回 top-K。聚合器合并结果，应用多阶段排序（低成本 → 高成本 ML），填充推文内容，应用可见性过滤器后返回。

面试官： scatter-gather 带来的尾部延迟。

候选人： 对冲请求：向同一分片的两个副本发送请求，取先返回的响应。单分片超时；超时时丢弃慢速分片但返回部分结果（质量损失可控）。容量余量保持在 60-70% 左右，避免慢节点级联。对单分片 p99 进行积极监控。

面试官： 排序？

候选人： 多阶段排序。每分片低成本打分：BM25 + 静态信号（作者权威度、时效性）。每分片返回 top-K。聚合器合并；高成本 ML 排序器对顶部结果按特征（浏览者亲和度、互动量、时效性、个性化）打分。最终排序输出。

面试官： 可见性——屏蔽与删除。

候选人： 读取时门控。填充后，过滤被屏蔽作者的推文、已删除推文、受保护账号的推文（若非已批准关注者）以及违规下架推文。绝不将可见性规则写入索引——否则每次屏蔽变更都需要大规模重建索引。

面试官： 热门话题。

候选人： 相关但独立的流水线。按区域、每分钟聚合话题标签提及次数。与近期基线对比；涨幅最大者为热门。按区域和用户兴趣个性化呈现。独立于搜索索引。

面试官： 某用户关注了 500 个话题标签并查询其中一个。

候选人： Search 是通用的——任何人都可以查询任意词项。"关注话题标签"功能更类似于已保存的过滤器。查询路径相同；排序时可针对该用户的关注内容进行个性化提权。

面试官： 多地区。

候选人： 每区域独立的 live + 主索引。推文流在全球复制，因此每个区域拥有完整语料库。查询在本地提供服务。个性化数据按用户所属主区域存储。

面试官： 热词（"the"、"a"）的倒排列表规模。

候选人： 停用词已过滤。"a" 等常用词不参与索引。中等频率词项的超大倒排列表采用压缩存储（对 tweet_id 进行差值编码）。对于活动期间极热的词项，可进一步对倒排列表分片，使每分片写入速率保持可控。

面试官： 结果中的垃圾内容。

候选人： 对每条推文进行垃圾信息评分。过滤或降权。作者信誉纳入静态打分。对可疑集群（多账号发布相似内容）进行模式检测。严重情况进行人工审核。

面试官： 总结。

候选人： 两级倒排索引：live（秒级，驻内存，哈希分片）+ 主（小时级，压缩，时间+哈希分片）。异步索引器消费推文流。查询 scatter-gather，通过对冲请求进行多阶段排序处理尾部延迟。从推文存储填充内容。读取时进行可见性过滤。多区域各自独立索引。最难的约束是突发新闻的索引延迟；两级设计将其与长尾问题隔离开来。

3. 练习备注

五分钟版本

  1. 明确需求：规模、时效性、排序。
  2. 不变式：索引延迟有界；可见性正确。
  3. 两级倒排索引：live + 主。
  4. 从推文流异步建索引。
  5. live 按哈希分片；主索引按时间+哈希分片。
  6. Scatter-gather + 对冲请求 + 部分结果。
  7. 多阶段排名。
  8. 读取时可见性过滤。

一分钟结尾陈述

Twitter Search 是一个两级倒排索引：live 用于突发内容（秒级），主用于历史内容（小时级）。异步流水线消费推文流。查询跨相关分片 scatter-gather，多阶段排序，填充内容，应用可见性过滤器。对冲请求处理尾部延迟。读取时可见性过滤避免大规模重建索引。多区域各自独立索引。最难的约束是在不牺牲历史查询速度的前提下实现实时索引。

最佳面试用语

  • "两级分层：live 保证秒级时效性，主索引存储压缩历史数据。"
  • "通过流异步索引；推文在数秒内可被搜索到。"
  • "Scatter-gather 配合对冲请求 + 部分响应处理尾部延迟。"
  • "可见性——屏蔽、删除、隐私——在读取时应用，绝不写入索引。"
  • "多阶段排序：每分片低成本打分，顶部结果用高成本 ML 排序。"
