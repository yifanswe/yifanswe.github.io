§1. 什么是容器编排

一个 container orchestrator 是一个运行时，它将容器化工作负载调度到机器集群上，管理其生命周期，将其以服务形式暴露，并处理故障——所有这些都由对期望状态的声明式描述驱动。你向系统提交一个清单，声明"我需要三个 Web 应用副本，每个分配 500m CPU 和 512Mi 内存，暴露在 8080 端口，部署在标有 'web-tier' 的节点上"，编排器的工作就是让现实匹配该描述，并在节点故障、流量突增、新版本部署时持续保持匹配。

其核心特性是 期望状态收敛。你不发出命令式指令（"在节点 Y 上启动容器 X"）。你声明应当为真的状态，控制循环持续观察当前状态，将其与期望状态比较，并采取行动来弥合差距。这就是系统具有弹性的原因：每隔一段时间，最初放置容器的那个循环都会询问"三个副本还在运行吗？它们在健康节点上吗？它们仍然通过健康检查吗？"——如果答案是否，它会采取必要措施使答案变为是。

容器编排处于设计空间的一个角落：

  • 容器编排（Kubernetes、Nomad、ECS、Docker Swarm） ——将 OCI（开放容器倡议）容器调度到集群上，以服务形式暴露，管理生命周期。Pod 或任务是短暂的；编排器负责替换。是无状态及大多数有状态工作负载的现代默认选择。
  • VM（虚拟机）编排（VMware vSphere、OpenStack、AWS EC2 Auto Scaling Groups） ——更重的单元（VM 启动需 30–120 秒，容器仅需 1–5 秒），更强的隔离性（hypervisor），每 VM 工作负载更密集（多进程）。仍用于遗留应用、依赖内核特性的工作负载或严格隔离要求。
  • Serverless 平台（AWS Lambda、Google Cloud Run、Azure Functions、Fargate） ——完全抽象掉集群。你向平台提交一个函数或容器，平台按需运行，缩容到零，按调用计费。无需管理节点；执行环境受限（15 分钟超时、临时文件系统、内存有限）。
  • 裸金属配置（Foreman、MAAS、Tinkerbell） ——在物理机上安装操作系统和配置。是 VM 和容器之下的基础层。变更周期慢（数小时到数天），无原生调度。
  • 配置管理（Ansible、Puppet、Chef、SaltStack） ——通过 SSH 或 agent 对长期存活的主机应用配置。早于容器出现；与容器正交，但常用于容器层下面的主机层。

本文重点介绍容器编排——主要是 Kubernetes，因为 90% 以上的新部署都在其上，且字节层面的机制最为重要——同时沿途对其他变体进行比较。

容器编排不擅长的领域：

  • 持久状态保证。 Pod 是短暂的。编排器会在认为必要时将你的数据库 pod 重新调度到另一个节点，不会发出警告。持久化是由存储层（PV——持久卷、PVC——持久卷声明、CSI——容器存储接口驱动）单独处理的关注点，编排器本身对你的数据不作任何承诺。
  • 跨集群原子性。 对集群 A 的 Kubernetes 部署与对集群 B 的部署是相互独立的。跨集群没有内置的分布式事务。多集群系统必须在上层自行实现。
  • 业务逻辑正确性。 编排器运行你的代码，不验证代码是否正确。只要健康检查返回 200 OK，有 bug 的 pod 就会被欢快地永远保活。
  • 小规模工作负载。 运行一个 50 个 pod 的集群需要约 3 个控制平面节点加上存储和监控——其开销可能超过工作负载本身的成本。低于某个阈值（约 100 个 pod、约 5 个服务）时，使用 systemd 的单台 VM 确实更简单。
  • 硬实时。 Pod 调度、CNI（容器网络接口）配置和 kubelet 记账会引入可变的启动延迟。如果你需要工作负载在触发后 50 ms 内运行，请使用预热的 serverless 平台或固定一个长期运行的 pod。
  • 需要直接控制硬件的 GPU 密集型遗留训练任务。 在 Kubernetes 中可以实现（通过 NVIDIA 的设备插件），但遗留工具（Slurm）在科学 HPC（高性能计算）和部分 AI 训练场景中仍占主导。

心智模型：容器编排器是 调度与生命周期层 ，位于应用代码与物理基础设施之间。其上层的所有内容（负载均衡规则、部署策略、密钥、配置）均以声明式方式表达；其下层的所有内容（CPU、内存、磁盘、网络）都被抽象为资源模型。编排器的工作是调和这两者。

§2. 固有保证

容器编排按设计提供的能力，以及仍需在其上层叠加的能力。

设计上已提供

  • 期望状态收敛。 系统中的每个控制器都运行一个 reconcile 循环，给定一个声明式 spec，采取行动使实际状态与其匹配。如果你声明"三个副本"，只要集群有容量，系统就会持续保持三个副本存活。
  • 通过 pod 替换实现自愈。 崩溃的 pod 会被重启（Deployments 的默认行为 restartPolicy: Always ）。节点故障后，其上的 pod 会在约 5 分钟内（pod 驱逐超时）被重新调度到其他节点。健康检查失败的 pod 会在数秒内从服务端点中剔除。
  • 有界中断的滚动更新。 更新镜像时，每次滚动一个（或 N 个）pod，遵守 maxUnavailable 。 maxSurge。流量在 pod 被终止前通过就绪探针和端点控制器切换走。
  • 服务发现。 每个 Service 对象都有一个稳定的虚拟 IP（ClusterIP）和 DNS 名称（my-service.my-ns.svc.cluster.local）。Pod 来来去去；Service 抽象了这种变动。
  • Secret 与配置注入。 Secret（base64 编码，若配置则在 etcd 中加密存储）和 ConfigMap 以文件或环境变量的形式挂载到 pod 中。支持轮换（有注意事项——见 §7）。
  • 资源隔离。 Pod 声明 CPU 和内存的 requests 。 limits。调度器根据 requests 进行放置；内核（通过 cgroups）在运行时强制执行 limits。
  • 基于指标的水平扩缩容。 HPA（Horizontal Pod Autoscaler）和 VPA（Vertical Pod Autoscaler）根据指标调整副本数或每 pod 资源。KEDA（Kubernetes 事件驱动自动扩缩容）将其扩展到非指标触发器，如队列深度。

需在其上叠加

  • 持久状态。 Pod 对编排器而言是无状态的。持久存储由 PV 提供，后端为 EBS（Elastic Block Store）、GCE（Google Compute Engine）持久磁盘、Ceph、NFS 等。编排器决定 pod 运行在哪里；存储层决定数据是否跟随。Operator（如 CloudNativePG、Strimzi、Vitess）将有状态工作负载与自定义 reconcile 逻辑封装在一起。
  • 跨区域/多集群协调。 单个 Kubernetes 集群局限于一个（或少数耦合的）故障域。跨区域主动/主动架构需要联邦工具（KubeFed、Cluster API、Argo CD ApplicationSet、Karmada）加上外部服务网格或全局负载均衡器。
  • 网络策略执行。 默认的 Kubernetes 网络是扁平的——每个 pod 都可以访问其他所有 pod。NetworkPolicy 对象是 请求；CNI 插件必须负责执行（Calico、Cilium 可以；部分插件不行）。零信任需要服务网格层（Istio、Linkerd）或支持 L7 过滤的 CNI（带 eBPF 的 Cilium）。
  • 滚动更新之外的部署策略。 原生 Deployment 支持滚动更新和重建。金丝雀、蓝绿、流量镜像均需要额外机制——Argo Rollouts、Flagger、服务网格流量分割，或手工脚本。
  • GitOps / CI/CD。 Kubernetes 接受 manifest；它不从 git 拉取 manifest、构建镜像或编排发布顺序。ArgoCD、FluxCD、Spinnaker 和 Tekton 等工具在上层添加了交付流水线。
  • 备份与灾难恢复。 Velero、Stash 等负责 etcd 快照和 PV 备份。编排器本身只知道当前状态，不维护历史记录。
  • 多租户隔离。 Namespace 是 软 边界——RBAC（基于角色的访问控制）、ResourceQuota、LimitRange 在其上层叠加。 硬 多租户（不受信任的工作负载）需要额外手段（gVisor、Kata Containers、vCluster 或独立集群）。
  • 成本分配、费用分摊、FinOps。 编排器本身不按团队归因计算成本。Kubecost、OpenCost 等工具从标签中计算出这些数据。

总结：编排器保证的是 在足够同质的机器集群上对短暂工作负载的调度、生命周期管理和服务抽象。超出此范围的一切——持久状态、跨集群、安全策略执行、交付流水线、多租户——都是系统设计者需要解决的问题。

§3. 设计空间

编排器设计空间以 Kubernetes 为主导，但有若干有意义的变体。

维度 1：通用编排器 vs 云托管

  • 通用、可自托管（Kubernetes、Nomad、Docker Swarm、Mesos）： 可运行在任何地方——本地裸金属、EC2（Elastic Compute Cloud）、裸金属云、边缘。你自己运营控制平面（或使用同款软件的托管服务）。
  • 云私有（Amazon ECS、AWS Fargate、Azure Container Instances）： 与某一云的服务紧密耦合。运维更简单（无需管理 etcd 和控制平面），但存在厂商锁定。

维度 2：仅容器 vs 多工作负载

  • 容器专注型（Kubernetes、ECS、Docker Swarm）： pod/任务是主要单元。其他工作负载（原始二进制、VM）是二等公民或不受支持。
  • 多工作负载型（Nomad）： 调度容器、原始可执行文件（ exec 驱动）、QEMU VM、Java JAR、Windows 任务。一个调度器，多种运行时。当你的集群中有无法轻易容器化的遗留可执行文件时非常有用。

维度 3：全功能 vs 最小化

  • 全功能型（Kubernetes）： 内置几乎所有资源的控制器——Deployment、StatefulSet、DaemonSet、Job、CronJob、HPA、NetworkPolicy、Ingress（通过控制器）、Service、PVC 等。学习曲线陡峭；生态系统丰富。
  • 最小核心型（Nomad）： 约 50 MB 的单一二进制。调度、服务发现、简单的 stanza 风格作业定义。生态系统较薄；运维负担较低。

维度 4：部署交付模型

与编排器本身正交，但实践中紧密耦合：

  • 推式（Spinnaker、传统 Jenkins 驱动的 kubectl apply): CI/CD 系统持有集群凭证，将 manifest 推送进去。熟悉；密钥在 CI 中；集群不知道其配置来源。
  • 拉式 GitOps（ArgoCD、FluxCD）： 集群内控制器监听 git 仓库，拉取 manifest，应用它们，报告漂移。集群无需入站凭证；git 是唯一事实来源。
  • 渐进式交付（Argo Rollouts、Flagger）： 原生支持金丝雀和蓝绿发布的部署控制器，以 Prometheus / Datadog 等的指标作为每个步骤的门控。

4.2 etcd——期望状态存储

etcd 是集群的唯一事实来源。 其他所有组件均为无状态或拥有可重建的状态；所有必须持久化的内容（对象规格、当前状态、主节点选举）都存储在 etcd 中。它是一个基于 Raft 复制的键值存储，具备以下特性：

  • MVCC（多版本并发控制）： 每次写入都会创建一个新版本；旧版本会保留 --auto-compaction-retention （通常为 1 小时），以便 watcher 可以从近期历史的任意位置恢复。
  • Watch 流： 客户端打开长连接的 gRPC 流并订阅某个键前缀。该前缀下的任何变更都会以事件形式推送。这是 API server 用来将变更事件推送给集群中每个控制器的机制。
  • 共识： 写入需要 etcd 成员的多数派确认（5 节点中的 3 个，3 节点中的 2 个）。读取默认为 线性化 （也需要多数派确认），但也可以设置为 读；从主节点读取（带 read-index）可实现线性化读。etcd 默认为线性化读。 （从本地成员读取，可能存在数据陈旧，但速度更快）。

  → value: protobuf-encoded PodSpec + PodStatus, ~5-50 KB

API server 负责读写 etcd；其他组件不直接访问 etcd。这种隔离是有意为之——这意味着 etcd 的权限模型可以极为严格（仅使用 TLS——传输层安全——客户端证书），集群的复杂度可以持续增长，而 etcd 无需了解 Pod、Service 或任何 Kubernetes 概念。对 etcd 而言，这一切都是不透明的字节。

规模上限：etcd 在总数据量约 8 GB、约 15 万个对象时通常运行良好，可持续承受每秒数百次写入。超过此限制，Raft 日志会变得很大，从节点的复制将开始落后；集群运维人员必须调整 --quota-backend-bytes、定期运行 etcdctl defrag，并限制最大对象大小（ kube-apiserver --max-request-bytes 默认为 1.5 MB——这暗示非常大的对象并非设计目标）。Stripe 曾报告在接近此极限的环境下运行 etcd；Google 的 k8s 托管服务则通过在每个逻辑集群内跨多个 etcd 集群分片来绕过此限制。

4.3 kube-apiserver——验证与准入层

API server 是一个无状态的 HTTPS 服务器，以 REST 形式暴露完整的 Kubernetes API。它承担四项职责：

  1. 认证 ——验证客户端身份（TLS 证书、Token、OIDC——OpenID Connect、Webhook）。
  2. 授权 ——执行 RBAC 检查。"用户 U 是否可以在命名空间 N 中对资源 R 执行操作 V？"
  3. 准入控制 ——运行 mutating 和 validating Webhook。这里负责完成默认值填充、验证和策略执行。示例：内置的 LimitRanger 准入控制器填充默认的 CPU/内存请求。一个 PodSecurityPolicy （或其后继者 PodSecurity 准入）拒绝请求以 root 运行的 Pod。外部 mutating Webhook（例如 Istio 的 sidecar 注入器）会向带标签命名空间中的每个 Pod 注入一个 Envoy sidecar。
  4. 持久化到 etcd ——通过准入后，对象被写入 etcd。只有在 etcd 确认后，才会向客户端返回响应。

API server 是 水平可扩展的 ——你可以在负载均衡器后面运行 3、5、10 个实例，所有实例读写同一个 etcd。Watch 扇出是主要的扩展瓶颈：每个 watcher 都是 API server 必须持续推送的长连接流。在大规模场景（5000 节点集群）下，API server 使用内部 watch 缓存 来去重 etcd watch——每个资源类型只有一个 etcd watcher，在进程内扇出给数千个 API 客户端。

4.4 控制器模式——声明式的核心

一个 控制器 是一个能够执行以下操作的进程：

  1. 通过 API server 的 watch 协议监听一个或多个 API 资源。
  2. 维护当前状态的本地缓存。
  3. 定期（或在事件触发时）将当前状态与期望状态进行对比。
  4. 计算出收敛所需的操作。
  5. 通过 API server 执行这些操作。
  6. 无限循环重复以上过程。

这就是 协调循环:

                  ┌─────────────────┐
                  │  Watch events   │  ← from API server (apiserver feeds
                  │   queue up      │     watch events into client-go's
                  └────────┬────────┘    SharedInformer cache)
                           │
                           ▼
              ┌────────────────────────┐
              │   workqueue (rate-     │
              │   limited, dedup)      │
              └──────────┬─────────────┘
                         │ pop a key
                         ▼
              ┌────────────────────────┐
              │  Reconcile(key)         │
              │   1. fetch current state│
              │   2. fetch desired spec │
              │   3. compute delta      │
              │   4. take action via    │
              │      apiserver API call │
              │   5. update status      │
              └────────────────────────┘
                         │
                  ┌──────┴──────┐
                  │             │
              (success)     (error / requeue)

Kubernetes 中的每个"原语"都是一个控制器。Deployment 控制器通过创建 ReplicaSet 来协调 Deployment。ReplicaSet 控制器通过创建 Pod 来协调 ReplicaSet。Endpoints 控制器通过列举 Pod 来协调 Endpoints。自定义资源定义（CRD）扩展了这一模式——编写一个 CRD 加上一个自定义控制器，就能在不修改核心 Kubernetes 代码的情况下添加新的原语。这正是各类 Operator（面向 Postgres 的 CloudNativePG、面向 Kafka 的 Strimzi、面向监控的 Prometheus Operator）的工作原理。

核心原则： 控制器是水平触发的，而非边缘触发的。控制器不是消费单条"创建 Pod X"消息后便将其遗忘，而是持续观测"我需要 3 个名为 X-* 的 Pod"并不断确保其成立。即使某个事件丢失，下一次定期同步仍会纠正它。这正是系统能够对抗网络抖动、控制器重启和 etcd 偶发问题的原因。

4.5 调度器——基于谓词与优先级的装箱算法

默认调度器（kube-scheduler）是一个控制器，它监听 spec.nodeName 未设置的 Pod，并将其分配到某个节点。该算法为 先过滤，再评分:

For each pending pod:
  Step 1: FILTER (a.k.a. predicates)
    For each node N in cluster:
      pass = true
      check NodeAffinity / NodeSelector match    ← skip if mismatch
      check Taints / Tolerations                 ← skip if untolerated
      check resource fits (CPU, memory, ephemeral storage)
      check pod fits port requirements
      check volume zone matches node zone
      check existing pod (anti-)affinity rules
      ...
      if any check fails, drop N from candidates

  Step 2: SCORE (a.k.a. priorities)
    For each candidate node N that passed filter:
      score = 0
      score += LeastRequested      (prefer emptier nodes — bin-pack tightness)
      score += BalancedAllocation  (prefer balance between CPU/memory)
      score += NodeAffinityWeight  (weighted by soft affinity rules)
      score += ImageLocality       (prefer nodes that already have the image)
      score += InterPodAffinity    (prefer near/far from labeled pods)
      ...

  Step 3: BIND
    Pick the highest-scoring node.
    Issue Binding API call → apiserver writes spec.nodeName = N to etcd.
    The pod is now "assigned." kubelet on node N will pick it up.

调度器在绑定阶段默认为 单线程 （一次处理一个 Pod），但过滤和评分阶段会在多个 goroutine 上并行执行。调度器在 5000 节点集群上可持续处理约 100 个 Pod/秒，这是快速扩容的实际上限。

不存在"某个 Pod 在哪个节点上"的中央数据库——这一信息是 etcd 中各 Pod 对象上 spec.nodeName 字段的集合。调度器不具备对已完成调度决策的追溯权。一旦 Pod 被绑定，只有驱逐或删除才能移动它。这是有意为之的：Kubernetes 不会 非 为了装箱优化而重新调度运行中的 Pod，除非明确要求（通过 descheduler 或 PodDisruptionBudget感知驱逐）。

4.6 kubelet——每节点代理

每个工作节点运行一个 kubelet，即将 Pod 规格转化为运行中容器的代理。kubelet 的职责：

  1. 监听 API server 中 spec.nodeName == thisNode.
  2. 通过 CRI（容器运行时接口） ——由 containerd 或 CRI-O 实现的 gRPC API——来创建 Pod 沙箱、拉取镜像并启动每个容器。
  3. 通过 CNI（容器网络接口） 插件以挂载网络接口。
  4. 通过 CSI（容器存储接口） 插件以挂载持久化卷。
  5. 定期将节点和 Pod 状态上报给 API server，后者写入 etcd。
  6. 运行存活探针和就绪探针；重启失败的容器。
  7. 将日志流式写入磁盘；并提供给 kubectl logs.

kubelet 本身不运行容器。容器由 containerd 运行（自 Kubernetes 1.24（2022 年）弃用 Docker 作为集群内运行时后，containerd 已成为主流的底层运行时）。kubelet 与 containerd 之间的边界即为 CRI：

kubelet                                    containerd
  │     CreateContainerRequest               │
  │ ───────────────────────────────────────► │   shim process forks runc
  │                                           │   runc sets up namespaces
  │     CreateContainerResponse(id=xyz)       │   (PID, NET, MNT, IPC, UTS, USER)
  │ ◄─────────────────────────────────────── │   cgroups limit resources
  │     StartContainerRequest                 │   container is now running
  │ ───────────────────────────────────────► │
  │     StartContainerResponse                │
  │ ◄─────────────────────────────────────── │

containerd 之下是 runc （或替代方案——用于沙箱隔离的 gVisor、用于 VM 级隔离容器的 Kata Containers、以及更轻量的 C 实现 crun）。runc 才是真正执行 clone() 。 unshare() 系统调用来创建 Linux 命名空间以及 cgroups v2 写入以强制执行 CPU/内存限制的程序。

CRI 的分离使得编排器可以独立于运行时演进。Kubernetes 最初直接对接"Docker"；2022 年弃用 dockershim 正是长期推迟的、标准化到 CRI 的最终结果。

4.7 CNI（容器网络接口）——Pod 网络的工作原理

CNI 是一份契约："给定一个网络命名空间和 Pod 的标识，挂载一个网络接口并配置路由。"其实现由 CNI 插件完成，而插件的选择是每个集群中影响最深远的决策之一。

三种主流 CNI 插件：

  • Cilium ——基于 eBPF。通过 BPF（Berkeley Packet Filter）映射直接对 Linux 内核的网络子系统进行编程，无需 iptables，也不需要 kube-proxy。支持 L7（应用层）策略，在高 Pod 密度下性能最佳。是 2026 年新建大型集群的事实之选。
  • Calico ——支持 BGP（边界网关协议）在节点间路由 Pod IP，或在非 BGP 环境下使用 VXLAN/IPIP 覆盖网络。长期以来是生产环境的默认选择，策略模型非常灵活。
  • Flannel ——仅支持 VXLAN 覆盖网络，操作最为简单，不支持策略执行。常见于开发集群，在大规模生产环境中少见。

在 Cilium 下（使用 bpf_lxc 模式）Pod 间网络的实际工作原理：

Pod A on Node 1 wants to send a packet to Pod B on Node 2.

1. Pod A's process calls sendmsg(). The socket is in Pod A's network namespace.
2. The kernel's network stack consults the routing table inside Pod A's netns.
   Default route → veth-A-host (the host-side end of a veth pair).
3. Packet arrives in the host's main network namespace via the veth.
4. Cilium's eBPF program attached to veth-A-host inspects the packet:
   - Checks NetworkPolicy: is Pod A allowed to talk to Pod B?
   - Looks up Pod B's identity in a BPF map (keyed by destination IP).
   - Determines the egress route: which physical interface, which encap.
5. If Node 2 is on the same L2 (Layer 2) segment, the packet is encapsulated
   in VXLAN (or routed natively if BGP is configured) and sent out eth0.
6. Node 2's eth0 receives. Cilium's ingress eBPF program decapsulates,
   checks NetworkPolicy again on the destination side, and delivers to
   veth-B-host.
7. Packet flows through veth-B-host → veth-B-pod → Pod B's netns → Pod B's
   socket.

关键点： CNI 工作在 L3+（三层及以上）。Pod 获得 IP 地址；其余部分由内核路由加上可选的封装来处理。编排器并不发明新的网络协议栈；它只是配置内核现有的协议栈。

4.8 CSI（容器存储接口）——PVC 如何映射到磁盘

CSI 是存储领域的 CNI 类比物。每个存储后端（EBS、GCE PD、Ceph、Portworx、OpenEBS、vSAN、NFS）都提供一个 CSI 驱动，以两个组件的形式运行：

  • 一个 控制器插件 （Deployment 形式），负责集群级别的操作：创建卷、删除卷、快照。
  • 一个 节点插件 （DaemonSet 形式），负责每节点的操作：将卷挂载到该节点、在 Pod 内挂载。

当 Pod 申请 PersistentVolumeClaim（PVC）时的流程：

1. User creates a PVC:
     kind: PersistentVolumeClaim
     spec:
       resources: { requests: { storage: 100Gi } }
       storageClassName: ebs-gp3

2. PVC controller sees the PVC. Looks up the StorageClass to find the provisioner.
3. CSI controller plugin gets a CreateVolume call:
     csi-ebs creates a 100 GiB gp3 EBS volume via AWS API.
     Returns volumeHandle = "vol-0123abc".
4. A PersistentVolume (PV) object is created with that handle.
5. The PVC binds to the PV.
6. A pod referencing the PVC is scheduled to node N.
7. The kubelet on node N calls the CSI node plugin: NodePublishVolume.
8. CSI node plugin:
     a. Attaches vol-0123abc to node N via AWS API (~5-20 seconds).
     b. Detects the resulting /dev/nvme1n1.
     c. Mounts /dev/nvme1n1 onto /var/lib/kubelet/pods/<pod-uid>/volumes/...
9. kubelet bind-mounts that directory into the container's mount namespace
   at the path specified in volumeMounts.
10. Container starts. Its filesystem now includes the persistent volume.

当 Pod 被删除或重新调度时，以上过程逆向发生：卸载、解除挂载，以及根据 PersistentVolumeReclaimPolicy).

脆弱之处：attach/detach 速度较慢（5-30 秒），跨可用区的卷只能挂载到同一可用区的节点，且只有某些后端（处于 mt-attach 模式的 EBS gp3、AWS EFS、Ceph RBD）支持多重挂载。这是"我的有状态 Pod 一直卡在 Terminating"这一痛点的根源——CSI 驱动正在等待 AWS 解除对持有陈旧租约的卷的挂载。

4.9 端到端演练 kubectl apply pod.yaml 全流程

深度演练。开发者输入 kubectl apply -f pod.yaml，以下是所有发生的事情。

Pod manifest:
  apiVersion: v1
  kind: Pod
  metadata: { name: web-app, namespace: prod }
  spec:
    containers:
    - name: app
      image: registry.example.com/web-app:v1.4.2
      resources: { requests: { cpu: 500m, memory: 512Mi } }

Step 1: kubectl
  Reads ~/.kube/config to find the apiserver URL and credentials.
  Parses the YAML; converts to JSON.
  Computes a 3-way merge patch vs the last-applied-config annotation.
  Sends HTTPS PATCH https://apiserver:6443/api/v1/namespaces/prod/pods/web-app.

Step 2: kube-apiserver — request routing
  TLS handshake; client cert validated.
  Authn: extracts user identity from cert (Common Name + Organization).
  Authz: RBAC check — can user 'alice' verb 'patch' resource 'pods' in 'prod'?
  Routes to the v1.Pod handler.

Step 3: kube-apiserver — admission
  Mutating admission webhooks run in order:
    - LimitRanger fills in default memory limit (e.g., 1Gi).
    - NamespaceDefaulter sets serviceAccountName: default.
    - Istio sidecar injector adds an envoy sidecar container.
  Validating admission webhooks run:
    - PodSecurity admission checks privilege escalation.
    - ResourceQuota controller checks remaining quota in 'prod' namespace.

Step 4: kube-apiserver — persistence
  apiserver encodes the Pod as Protobuf (default storage format).
  Issues etcd transaction:
    if revision('pods/prod/web-app') == expected_revision:
      put('pods/prod/web-app', encoded_pod)
  etcd's Raft leader appends to WAL, replicates to followers, applies to
  state machine, returns success after quorum.
  apiserver returns 201/200 to kubectl. Pod is now persisted.

Step 5: Scheduler
  Scheduler's pod-informer cache receives a watch event: new Pod with
  spec.nodeName == "".
  Scheduler enqueues web-app.
  Filter phase: iterates over the 100 nodes in the cluster.
    Node 1: requests 500m CPU; has 200m free. FILTER OUT.
    Node 2: 800m free. PASS.
    ...
    47 nodes pass.
  Score phase: scores each. Node 17 wins (most free capacity + image already cached).
  Binding: scheduler issues a POST /pods/web-app/binding with target Node 17.
  apiserver writes spec.nodeName = "node17" to etcd.

Step 6: kubelet on node17
  kubelet's pod informer sees Pod with spec.nodeName == node17.
  Pulls Pod into its pod-manager.
  Calls CRI: RunPodSandbox — containerd creates the pod sandbox (a pause
    container that holds the network namespace).
  Calls CNI ADD: Cilium attaches a veth pair, allocates pod IP 10.244.17.42,
    programs eBPF maps with the pod's identity.
  Calls CRI: PullImage if not present. containerd pulls registry.example.com/
    web-app:v1.4.2 via the configured credentials (~5-50 seconds depending
    on image size and registry latency).
  Calls CRI: CreateContainer for the 'app' container — containerd creates
    the OCI bundle and runc config.
  Calls CRI: StartContainer — runc clone()s the process inside the namespaces;
    cgroups v2 enforce 500m CPU shares and 1Gi memory limit.
  Container is now running with PID 1 = the app process.

Step 7: Readiness probe
  kubelet runs the readiness probe (e.g., HTTP GET /healthz on port 8080).
  Once it returns 200, kubelet patches the Pod's status.conditions to set
  Ready: True.
  apiserver writes the status update to etcd.

Step 8: Endpoints controller
  The Endpoints controller watches Pods for the Service that selects this pod.
  Sees the pod become Ready.
  Adds 10.244.17.42:8080 to the Endpoints object for service 'web-app'.
  apiserver writes the Endpoints update to etcd.

Step 9: kube-proxy / Cilium dataplane
  kube-proxy on every node sees the Endpoints change.
  Updates its iptables/IPVS rules (or, with Cilium, eBPF maps) so that
  packets to the service ClusterIP load-balance to include 10.244.17.42:8080.

Step 10: Traffic flows
  Other pods in the cluster doing DNS for web-app.prod.svc.cluster.local
  resolve to the Service ClusterIP. Their packets are DNAT'd to 10.244.17.42:8080.

DURABILITY POINT:
  The cluster's authoritative state lives in etcd. If every kubelet, scheduler,
  and apiserver dies, the data in etcd is sufficient to reconstruct the cluster.
  If etcd loses quorum but a quorum's worth of disks survive, a restored member
  can rejoin and apply the WAL. If etcd is irrecoverably lost, recovery requires
  restoring from a snapshot (typically taken via `etcdctl snapshot save` on a
  cron).

大多数工程师对此的理解止步于"kubectl 提交 Pod，它就运行了"。深度在于控制器-watch-响应链，正是这一链路使系统保持水平触发且具有弹性：每一步都是控制器观察状态并采取行动的结果，而非一系列命令式指令的链式调用。

§5. 容量上限

容器编排所覆盖的吞吐量与规模范围。

小规模——3 节点集群的初创公司，约 50 个 Pod

典型的种子轮 SaaS 公司。三台小型 EC2 实例或 Droplet，每台同时运行控制平面和工作负载（使用 k3s 等单集群模式或 Linode Kubernetes Engine 等托管 k8s）。约 5 个服务（Web 应用、API、Worker、Postgres、Redis）共 50 个 Pod。每个节点的资源利用率约 30%。集群配置变更频率极低。编排器的开销在比例上较为显著——控制平面 CPU 约为 100m-500m，etcd 占用约 200 MB 内存，kubelet+containerd 再占约 300 MB——但绝对数值很小。

下一个瓶颈 在这个规模下，瓶颈不在编排器，而在应用本身。集群在不费力的情况下可以再承载 10 倍的 Pod。

中等规模——Spotify, Backstage，约 1 万个 Pod

Spotify 的内部平台（Backstage 在此诞生）在 Kubernetes 上运行数千个微服务。集群规模约为 1 万个 Pod，每个 Region 有 500-1000 个节点，并为爆炸半径隔离部署了多个集群。在这个规模下：

  • 控制平面为独占式（3 或 5 个节点，由 GKE——Google Kubernetes Engine 管理）。
  • etcd 持有约 3-8 GB 数据。Compaction 和碎片整理是运维关注点。
  • 每个服务每天约 5-10 次部署，数百个服务，每天共计数千次部署。
  • 每个集群最多约 150 个节点；多集群联邦处理其余部分。

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 下一个瓶颈 在于运维：控制 API server 的请求速率（大型 kubectl get -A 查询可能导致停顿）、保持 etcd 紧凑，以及保持 CNI 策略一致。

大规模——Stripe，约 10 万个 Pod

Stripe 在 AWS 上跨多个集群运行低几十万量级的 Pod。其编排器主要为 Kubernetes，但配有大量内部自动化。相关数字如下：

  • 每个集群为安全起见上限约为 1000-3000 个节点。
  • etcd 采用 --snapshot-count 激进调优以保持 WAL 较短。
  • Pod 启动延迟 p99 在 30-60 秒范围内（冷缓存时镜像拉取占主导；热缓存约为 5-10 秒）。
  • 部署频率为每天数千次；发布采用渐进式并依赖指标门控。

超大规模——Google Borg → Kubernetes，数百万核心

Google 的 Borg（Kubernetes 的直接前身；2015 年的 Borg 论文是必读文献）在数百万核心的规模下运行。单个 Borg Cell 容纳数万台机器。Kubernetes 目前并未运行在如此大的规模——Google 内部对最大工作负载仍使用 Borg——但 Kubernetes 的设计基准是"单集群 5000 节点、共 15 万个 Pod 的规模测试"。对于更大的集群群组，答案是 多集群，而非更大的单集群。

§7. 该技术固有的难题

任何使用容器编排的人都会遇到的五个问题。每个问题包含：朴素解法、其失效方式以及真正的修复方案。

7.1 异构资源约束下的装箱问题

一句话描述问题： 调度器必须将 Pod 放置到满足其 CPU、内存及其他资源请求的节点上，而装箱问题在一般情况下是 NP-hard 的。

具体失效场景：一个拥有 100 个节点、每节点 8 核 32 GiB RAM 的集群，托管 1500 个 Pod，平均每个 Pod 约使用 500m CPU 和 1.5 GiB RAM。聚合利用率约为 75% CPU 和 70% RAM——纸面上看很充裕。此时一个新 Pod 申请 4 CPU 和 16 GiB RAM。 它无法被调度到任何节点，因为没有任何节点有 4 个连续空闲 CPU 核，尽管聚合容量充足。这就是"每个节点都半满，但没有节点能容纳这个 Pod"的问题。

朴素修复：通过添加节点进行扩容。浪费——你已经有约 25% 的容量空闲。

实际修复方案，按干预顺序：

  1. 对 Pod 请求进行合理调整。 许多 Pod 申请 1 CPU 却只使用 100m。使用 VPA（Vertical Pod Autoscaler）推荐模式对整个集群的请求进行合理调整。典型结果是回收 30-50% 的容量。
  2. 拓扑感知调度。 对于需要靠近（低延迟通信）或分散（高可用）的 Pod，使用 topologySpreadConstraints 来影响调度位置。
  3. 运行 descheduler。 一个单独的控制器，用于驱逐"应该"调度到其他位置的 Pod——包括过载节点上的 Pod、违反（已更新的）亲和性规则的 Pod、低利用率 Pod。编排器不会自动重新均衡；descheduler 弥补了这一差距。
  4. Cluster Autoscaler。 检测无法适配当前任何节点的 Pending Pod，并从实例类型匹配的节点组中供给新节点。当使用率下降时回收节点。弹性伸缩的标准工具。
  5. Karpenter（AWS）。 新一代自动扩缩器，动态选择实例类型（而非从预设节点组中选取），在已发布的案例研究中，与传统 Cluster Autoscaler 相比通常节省 30-60% 的成本。

来自另一个领域的示例——ML 训练。一个训练任务需要 8 个 GPU（图形处理单元），最好在同一节点上以利用 NVLink 带宽。调度器必须找到一个有 8 个空闲 GPU 的节点 。 尊重任何 taint（例如，GPU 节点被 taint 以阻止通用工作负载）。如果没有 GPU 感知调度（nvidia.com/gpu 资源类型、NVIDIA device plugin、GPU-feature-discovery 标签），即使节点上有 GPU，调度器也会拒绝调度 Pod，因为它不知道这些 GPU 的存在。有了它，调度器将 GPU 视为另一种普通资源——但 8 个 GPU 必须在同一节点上的约束是一个 requiredDuringSchedulingIgnoredDuringExecution Pod 亲和性规则。

7.2 StatefulSet vs Deployment vs DaemonSet vs Job——选择正确的原语

一句话描述问题： Kubernetes 提供了四种（加上 CronJob 实际上是五种）用于管理 Pod 生命周期的原语，选错会导致正确性或可运维性问题。

这四种：

  • Deployment ——无状态副本。Pod 可互换。滚动更新替换随机 Pod；身份无关紧要。
  • StatefulSet ——有状态副本。Pod 具有稳定身份（my-db-0, my-db-1, my-db-2），稳定存储（每个 Pod 有自己的 PVC），以及有序的启动/关闭。集群数据库、Kafka broker 等的默认选择。
  • DaemonSet ——每节点一个 Pod。用于节点本地代理：日志采集器（Fluentd、Vector）、指标导出器（node-exporter）、CNI 插件、CSI 节点插件。
  • Job ——运行至完成。一次性任务：数据库迁移、ML 训练运行、ETL。
  • CronJob ——按计划执行的 Job。

具体失效场景：一个团队通过 Deployment 部署 Postgres 副本，配置了 replicas: 3。每个 Pod 挂载了一个 PersistentVolumeClaim ，路径为 /var/lib/postgresql。集群缩容到一个 Pod，然后再扩容到三个。 三个新 Pod 无法判断哪个是主节点。 它们的 PVC 未绑定到特定副本；Deployment 不保证稳定身份。数据不一致。

朴素修复：通过设置 metadata.name手工实现身份标识。行不通——Deployment 生成带随机后缀的名称。

实际修复：使用 StatefulSet。一个有 serviceName: my-db 。 replicas: 3 个副本的 StatefulSet 会创建名为 my-db-0, my-db-1, my-db-2的 Pod。每个 Pod 有自己的 PVC（my-data-my-db-0等），在重启后保持稳定。Headless Service my-db 解析到特定 Pod 的 DNS 名称（my-db-0.my-db.ns.svc.cluster.local）。编排器现在保证"身份为 0 的 Pod"始终挂载相同的数据。

然后在上层叠加一个 operator ——即自定义控制器——来处理运维关切：主节点选举、故障转移、备份、版本升级。CloudNativePG、Strimzi（Kafka）、Vitess（分片 MySQL）、Crunchy Postgres Operator 是广泛使用的示例。Operator 模式正是 Kubernetes 在 2020-2024 年间最终成为有状态工作负载可行选择的原因。

批处理场景示例——ML 训练流水线。训练脚本天然适合用 Job：执行一次，成功或失败，无需复制。Argo Workflows 或 Tekton（或 Kubeflow Pipelines）将 Job 封装为具有依赖、重试和产物传递的多步骤 DAG（有向无环图）。编排器运行 Job；工作流工具对其进行排序。

7.3 持久化存储——"我的 Pod 迁移到新节点了，数据在哪里？"

一句话描述问题： Pod 是短暂的；编排器可以移动它们；如果没有显式的存储配置，旧节点上的数据将会丢失。

具体失效场景：一个 Pod 将一个 5 GiB 的 SQLite 文件写入容器文件系统。托管该 Pod 的节点因内核补丁而被 drain。Pod 被重新调度到不同的节点。 SQLite 文件消失了。 容器文件系统是每个实例独立的。

朴素修复：使用 emptyDir 卷。没有帮助—— emptyDir 也与 Pod 的生命周期绑定；Pod 消亡时它也消失。

实际修复：使用由网络附加 PersistentVolume（PV）支持的 PersistentVolumeClaim （PVC）——EBS、GCE PD、Azure Disk、Ceph RBD 等。PV 独立于任何节点；它可以挂载到 Pod 运行的任何节点上。

但这带来了新问题：

  • 区域卷。 EBS 卷是区域性的（us-east-1a 或 us-east-1b，而非同时支持两者）。如果 Pod 被调度到 us-east-1c 的节点上，卷无法挂载。StorageClass 的 topology.kubernetes.io/zone 标签和 WaitForFirstConsumer 绑定模式共同确保卷在 Pod 运行的区域中被供给。
  • 单挂载（RWO——ReadWriteOnce）。 大多数块存储一次只能挂载到一个节点。如果旧节点挂起而未释放卷，新节点无法挂载。恢复需要手动干预（强制卸载），或越来越多地依赖能处理此情况的 CSI 驱动（AWS CSI 驱动在新版本中已支持）。
  • 多挂载（RWX——ReadWriteMany）。 共享存储所需（例如有状态的 Jenkins）。需要网络文件系统（EFS、NFS、CephFS）——比块存储慢。并非所有 CSI 驱动都支持。

数据库使用场景示例：CloudNativePG 在 Kubernetes 上部署 Postgres。每个副本是一个带有自己 PVC（通常为 50–500 GiB 的 EBS gp3 卷）的 Pod。当主节点故障时，operator 通过修改 leader Pod 标签来晋升副本；副本的 PVC 继续存在。应用通过选中新主节点的 Service 重新连接。存储的连续性是这一切能够运作的关键；没有它，每次故障转移都需要完整的数据重建。

7.4 网络策略与零信任 Pod 网络

一句话描述问题： 默认情况下，Kubernetes 集群中的每个 Pod 都可以通过网络访问其他任何 Pod，这是一个潜在的安全灾难。

具体失效场景：开发者推送了一个包含已知 CVE（Common Vulnerabilities and Exposure）的 Docker 镜像。该镜像作为 marketing-service Deployment 的一部分运行。攻击者利用该 CVE 获取 shell，然后 curl访问 payments-service 的内部端点。payments service 信任集群内流量；攻击者窃取支付数据。

朴素修复：依赖集群边界。一旦集群内部有任何组件被攻陷，这种方法就失效了。

实际修复：NetworkPolicy。NetworkPolicy 对象表达"标签为 X 的 Pod 只能被标签为 Y 的 Pod 通过端口 Z 访问"。常见的默认策略是"全部拒绝"加上选择性允许：

kind: NetworkPolicy
metadata: { name: default-deny, namespace: payments }
spec:
  podSelector: {}        # selects all pods in this namespace
  policyTypes: [Ingress, Egress]
  # no ingress/egress rules = deny all

然后针对合法的跨服务流量添加显式允许规则。NetworkPolicy 由 CNI 插件强制执行——Calico 和 Cilium 支持；Flannel 不支持。对于 L7（应用层——HTTP 方法、路径、请求头）策略，需要 Service mesh（Istio AuthorizationPolicy）或 Cilium L7 策略。

在此之上再叠加： 服务间的 mTLS（双向 TLS） ，由 Service mesh sidecar 终止。每个服务有自己的短期证书（SPIFFE——Secure Production Identity Framework for Everyone——身份标识是现代化模式），流量在连接层面进行加密和认证。即使 NetworkPolicy 配置错误，没有有效证书也无法建立连接。

Feed 服务栈示例：推荐服务必须调用用户资料服务进行数据补全，但不应访问计费服务。NetworkPolicy 表达了这一约束。如果没有它，推荐服务中的漏洞就会危及计费服务。有了它，横向移动在网络层被阻断。

7.5 多租户——namespace 隔离 vs 独立集群

一句话描述问题： namespace 是 软 隔离；对于 硬 多租户（不受信任的工作负载、监管边界、严格的资源隔离），namespace 是不够的。

具体失效场景：一个工程组织在一个集群上运行 50 个团队的工作负载，每个团队在自己的 namespace 中。团队 A 部署了一个有问题的 Pod，分配了 100 GiB 的 swap 支持内存；节点 OOM-killer（内存溢出杀手）开始驱逐 Pod，包括团队 B 的数据库。内核不尊重 namespace 边界的内存压力。

朴素修复：收紧 ResourceQuota。有助于配额统计，但当限制无界或设置不当时，无法阻止单个贪婪 Pod 造成的节点级资源竞争。

实际修复是多层次的：

  • ResourceQuota 按 namespace 限制请求资源。在准入时防止一个团队消耗所有 CPU/内存。
  • LimitRange 为单个 Pod 的请求/限制设置默认值和最大值。
  • Pod Security Standards （Restricted/Baseline/Privileged）限制内核能力。
  • NetworkPolicy 隔离每个 namespace 的流量。
  • PriorityClass + PodDisruptionBudget 防止低优先级工作负载抢占高优先级工作负载。
  • 对于 硬 硬多租户：
  • 独立集群 按租户划分。最干净的方案。
  • vCluster ——在宿主集群内运行的虚拟 Kubernetes 集群，每个集群有自己的 apiserver 和控制器。
  • gVisor / Kata Containers ——运行时沙箱，为每个 Pod 提供更强的内核隔离。

SaaS 场景示例：托管客户代码的 B2B SaaS（例如 Pipedream 这样的 webhook 平台或 Hex 这样的 notebook 主机）运行不受信任的用户代码。硬多租户是强制要求。大多数此类产品对每个租户使用 Firecracker microVM（AWS Fargate 的底层技术）或 gVisor，而非依赖 namespace 隔离。

7.6 不停机集群升级

一句话描述问题： Kubernetes 每 4 个月发布一个新版本，每个版本支持约 14 个月——这意味着升级是一个持续的运维关切，而失败的升级可能导致集群宕机。

具体失效场景：一个团队运行 Kubernetes 1.24，API server 中断了他们依赖的某个 CRD 的 kubectl apply ，升级到 1.28 时废弃了他们在脚本中硬编码的 policy/v1beta1 PodDisruptionBudget API。升级过程中，他们一半的 Deployment 无法应用。

朴素修复：原地升级。风险高——控制平面停机，废弃 API 立即暴露。

实际解决方案：

  1. 阅读发布说明。每个 Kubernetes 次版本都有"已移除 API"章节。将你的 manifest 与之进行审计。
  2. 模拟程序启动时会进行的重定基址操作。 kubectl convert （或使用 kubent) 等工具）查找已废弃 API 的使用情况。
  3. 先原地升级控制平面 （托管 Kubernetes 会处理这一步——GKE/EKS/AKS 通过滚动替换控制平面节点来升级控制平面）。工作负载继续运行。
  4. 然后逐节点升级工作节点，通过 drain（优雅地驱逐 Pod，遵守 PodDisruptionBudget）然后替换节点。
  5. 对于风险较高的升级， 供给一个新集群，并行部署工作负载，切换流量，下线旧集群。即"蓝绿集群"模式。代价高昂但万无一失。

高可用服务示例：支付集群不能承受哪怕几秒钟的停机。运维人员在集群层面执行蓝绿升级——在新 Kubernetes 版本上供给新集群（"绿"），通过 GitOps 部署所有工作负载，在负载均衡器层面切换流量，一周后下线旧集群（"蓝"）。升级总成本：一周内 2 倍基础设施费用。总停机时间：零。

，而非整个存储桶。

故障发生时会怎样，以及哪些内容能够存活。

8.1 工作节点丢失

节点消失（内核崩溃、硬件故障、EC2 实例终止、网络分区）。

  • 约 5 分钟内 （默认 pod-eviction-timeout），node-lifecycle 控制器将节点标记为 NotReady 并开始驱逐其 Pod。驱逐是逻辑性的——节点上的 Pod 被标记为待删除，但节点可能仍在运行它们；这是有状态工作负载"脑裂"风险的根源。
  • 无状态 Pod 由 Deployment/ReplicaSet 控制器重新调度到其他节点。总中断时间：几分钟，取决于 Pod 启动时间。
  • 有状态 Pod （StatefulSet）仅在 operator 确认安全后才重新调度。如果仲裁面临风险，operator 可能暂停；可能需要人工干预。
  • 存活的内容： etcd 状态完整。死亡节点上的 Pod 在其他地方重启。PV 中的数据重新附加到新 Pod（在卷从死亡节点卸载后，如果云端不知道节点已死亡，这个过程可能很慢）。

恢复流程：等待。编排器被设计为处理这种情况。如果卷挂住，通过云控制台手动强制卸载。

8.2 控制平面节点丢失

三个（或五个）控制平面节点中的一个宕机。

  • etcd： 失去三个成员中的一个。剩余的两个形成 2/3 的仲裁并继续服务。如果第二个成员也宕机，写入仲裁丢失。
  • apiserver： 存活的实例继续服务。负载均衡器绕开死亡实例路由。
  • Scheduler / controller-manager： 这些组件具有主节点选举语义。死亡节点上的主节点丢失其租约（约 15 秒），另一个实例成为主节点。
  • 工作负载： 正在运行的 Pod 继续运行。新 Pod 调度、扩缩容和滚动更新短暂暂停（约 30 秒），直到新的 scheduler 主节点选出。

恢复流程：替换故障节点。添加新的 etcd 成员。新成员通过 Raft 复制追赶进度。

8.3 etcd 失去仲裁

三个（或五个）etcd 成员中的两个（或三个）同时宕机。 这是灾难性情况。

  • apiserver 变为只读 （线性化读取也会阻塞；根据配置，只有来自本地 etcd 的过期读取可能可用）。
  • 无法调度新 Pod。无法应用新的 Deployment。
  • 正在运行的 Pod 继续运行。 这至关重要：即使控制平面宕机，数据平面仍然存活。编排器的电平触发设计在此体现价值。

恢复流程：

  1. 识别仍然存活的成员。
  2. 从其磁盘上的 WAL 中获取快照（或使用最近一次计划快照）。
  3. 从快照恢复 ，方法是用快照初始化一个新的 etcd 集群并启动它。
  4. 重新配置 apiserver 指向新集群。
  5. 验证状态。根据快照时效，最近的一些变更（最后几分钟）可能会丢失。

这就是为什么 定期（通常每 30 分钟）进行 etcd 快照是不可或缺的。持久性的关键点就是快照文件。

8.4 CNI 故障

CNI 插件（或其底层数据平面——VXLAN、BGP、eBPF 程序）在整个集群范围内发生故障。这种情况较为罕见，但爆炸半径非常大。

  • 现有连接 可能继续正常工作（内核套接字已建立）。
  • 新连接失败。 Pod 到 Pod、Pod 到 Service、Ingress 到 Pod——所有通信都可能失败。
  • DNS 解析失败 ，因为 CoreDNS Pod 无法被访问。

恢复：取决于故障模式。如果 Cilium daemon 因错误的配置推送在整个集群范围内崩溃，回滚配置。如果 BGP 会话崩溃，重启 BGP daemon。如果是内核 bug，逐个节点重启。

教训：CNI 变更的爆炸半径仅次于 etcd 变更。先在较小的集群中测试；逐步推出。

8.5 镜像拉取失败

Pod 已调度但 ImagePullBackOff ，因为镜像仓库不可达或请求的标签处不存在该镜像。

  • Pod 停留在 Pending → ImagePullBackOff 状态 ，以指数退避方式持续重试（两次尝试间隔最长 5 分钟）。
  • 其他 Pod 继续运行。 这是单个 Pod 的故障，而非集群故障。
  • 恢复： 修复镜像引用，或修复镜像仓库。使用 digest 固定镜像，以避免"镜像已移动"的意外。

8.6 CrashLoopBackOff

容器启动、崩溃、被重启、再次崩溃。Kubernetes 以指数方式递增退避时间（10s、20s、40s、80s，上限为 5 分钟）。

  • 该 pod 在技术层面上处于运行状态， 但处于无用状态。
  • 就绪探针（Readiness probes） 是关键所在——如果就绪探针失败，该 pod 将从 Service 端点中移除，流量便不会打到一个异常的 pod 上。
  • 恢复： 查看容器日志（kubectl logs --previous），修复 bug，推送新镜像，重新部署。

8.7 PodDisruptionBudget 阻塞节点排空

某节点需要进行维护而被排空。PodDisruptionBudget（PDB）规定"至少 2 个副本（共 3 个）必须始终可用"。若多个副本位于同一节点上，排空该节点将违反 PDB。

  • 排空操作挂起。 kubectl drain 会阻塞，直到 PDB 条件可被满足。
  • 恢复： 首先分散副本（topologySpreadConstraints 会自动处理此问题），或使用 --ignore-daemonsets --delete-emptydir-data --force 进行紧急排空，接受短暂的不可用期。

§9. 为什么不直接 SSH 到服务器用 systemd 管理？

容器编排为何存在？最朴素的替代方案是什么，它又为何会失效？

朴素的替代方案：一批虚拟机，每台运行 systemd，服务通过手工或配置管理工具（Ansible、Chef）进行管理。部署方式是 scp 加重启。服务发现依赖硬编码的 IP 或 DNS 轮询。扩容方式是"新增虚拟机并告知负载均衡器"。

这种方式在小规模下有效，但会在可预见的阶段逐步变得痛苦：

  1. 约 20 个服务时，手动扩容成为持续不断的琐事。 需要增加 10% 的容量？SSH 登录 50 台机器，修改配置，重启服务。配置漂移悄然而至——部分机器更新了新配置，部分没有。

  2. 没有自愈能力。 服务崩溃后，systemd 会将其重启。但若机器本身出现故障，systemd 无法将其调度到其他机器上。机器所有者必须自行发现并介入处理。工程师的 on-call 负担随机群规模线性增长。

  3. 滚动更新依赖各自为战的脚本。 每个团队各自编写"每次部署 20% 节点"的逻辑。该逻辑中的 bug 会导致故障。有些团队跳过这一步，一次性全量部署；有些团队则过度工程化。

  4. 服务发现质量下降。 机器更换后，硬编码的 IP 会失效。DNS 虽然有效但收敛速度慢。专用服务注册中心（Consul、etcd、ZooKeeper）有帮助，但会带来额外的运维负担。

  5. 资源隔离完全依赖内核提供的能力。 同一台虚拟机上的两个服务会竞争内存；OOM killer 可能误杀错误的那个。要么过度配置（每台虚拟机只运行一个服务，成本高昂），要么接受"嘈杂邻居"问题。

  6. "1000 台服务器"的临界点。 大约在 100 到 500 台服务器之间，"SSH + systemd"方式的运维负担累积超过了运行 Kubernetes 的负担。临界点取决于团队规模和技术能力，但这一趋势是真实存在的。

实例演练：一个团队在两年内从 10 台虚拟机上运行 5 个服务，扩展到 200 台虚拟机上运行 50 个服务。起初他们使用 Ansible playbook 和手写部署脚本。如今每季度需要花半个 sprint 在部署基础设施上：编写排空替换脚本、改造服务发现、调试"为什么这个服务在 box-37 上仍然运行旧版本"。最终他们迁移到 Kubernetes，部署基础设施变成了几百行 YAML 和一条 CI 流水线。团队在实际产品工作上的效率大幅提升。

编排器并不会消除复杂性；它 将复杂性转移到共享的、被广泛理解的、声明式的基础原语中。一旦服务数量达到一定规模，"自建部署脚本"的单服务开销超过"学习 Kubernetes"的平台开销时，这种权衡就是值得的。

§10. 扩展维度

编排器沿两个不同维度扩展；其失效模式也各不相同。

类型 1——均匀增长：更多服务，更多节点

基本上是线性扩展。增加节点即可扩容。通过调优控制平面来适应更高的 pod 数量。

  • 最多约 150 个节点（每个集群） 使用默认配置：无需特殊调优。标准托管 Kubernetes（GKE/EKS/AKS）可透明处理。
  • 150 到 1000 个节点：专用控制平面需要扩容（apiserver 配置 4 至 8 核，etcd 配置约 16 GiB 内存）。调优 --max-pods-per-node （默认值为 110；在大节点上可提高到 250）。调优 --kube-api-qps 。 --kube-api-burst 以避免控制器遭受速率限制。
  • 1000 到 5000 个节点：这是单个 Kubernetes 集群的软性上限。超过约 5000 个节点后，调度延迟、etcd 写入吞吐量以及 apiserver 内存均开始触及瓶颈。据报道，Spotify、Uber、Stripe 均以此规模为上限，并采用多集群方案。
  • 超过 5000 个节点：需要多集群联邦。可使用 Cluster API、Karmada，或通过 GitOps ApplicationSet（在 ArgoCD 中）或 Flux 的 Kustomization 树管理的按环境/按区域集群。没有单集群解决方案；只能拆分。

类型 2——热点强化：服务不变，流量激增

单个服务需要从 100 QPS 扩展到 100,000 QPS。其机制是 针对单个服务的:

  • HPA（Horizontal Pod Autoscaler，水平 Pod 自动伸缩器） 根据 CPU 利用率、内存或自定义指标来调整副本数量。结合 metrics-server （用于 CPU/内存）以及 custom-metrics-adapter （用于 Prometheus 派生指标），HPA 可以设定"将 CPU 维持在 70%"或"将 p95 延迟维持在 200ms"等目标。这是标准工具。
  • KEDA（Kubernetes Event-Driven Autoscaler，Kubernetes 事件驱动自动伸缩器） 将 HPA 的能力扩展到非指标触发场景——例如 Kafka 消费积压、SQS 队列深度、Postgres 查询结果、CloudWatch 告警。KEDA 的设计点是事件驱动型工作负载：处理队列消息的服务应按队列深度扩缩容，而不是按 CPU。
  • VPA（Vertical Pod Autoscaler，垂直 Pod 自动伸缩器） 调整单个 pod 的资源请求。对热点服务（需要更多副本而非更大的 pod）用处不大，但对稳定状态下的资源量合理调整非常有效。

扩展曲线：

  • 低于 1k QPS / 50 个副本：使用默认配置的 HPA 即可。扩容大约需要 1 到 3 分钟（pod 调度 + 镜像拉取 + 预热）。对大多数工作负载而言可以接受。
  • 1k 至 10k QPS / 50 到 500 个副本：HPA 依然适用，但需要调优 behavior.scaleUp.policies 以更激进地扩容。考虑预留一定的容量余量（--min-replicas 设置为当前负载的约 130%），以便在 HPA 响应之前吸收流量峰值。
  • 10k 至 100k QPS：副本数量的重要性低于节点容量。单个 Service 对象的 iptables/IPVS 路由规则可能成为瓶颈；可考虑使用 headless Service + 客户端负载均衡，或迁移到服务网格。
  • 100k+ QPS：对服务进行分片。不同分片部署在不同的命名空间或集群中；客户端按 key 路由。编排器管理单个分片；分片逻辑在编排器之上。

约 1000 万用户

  • 约 150 个节点时： 需要设立专职平台团队；集群运维负担超出"副业任务"的范畴。
  • 约 5000 个节点时： 多集群联邦成为必须。
  • 约每个服务 10k QPS 时： 可观测性投入成为必须；在这一规模下无法通过查看日志来排查问题。

§12. 部署模式

这是技术与规范相结合的地方。同一个 Kubernetes 集群，根据所采用的模式不同，可以实现截然不同的风险特征。

12.1 滚动更新——默认方式

Kubernetes Deployment 默认采用 RollingUpdate 策略，由 maxUnavailable 。 maxSurge进行参数化配置。两者均可设为绝对数量或百分比。

maxUnavailable: 25%   ← at most 25% of replicas may be unavailable at a time
maxSurge: 25%         ← at most 25% extra replicas may be created beyond
                        the desired count during the rollout

For a Deployment with replicas: 8:
  - At most 2 pods are unavailable at any time
  - At most 10 pods exist transiently (8 + 2 surge)

发布流程：

Time 0:  ┌─────────────────────────────────────┐
         │ v1: 8 replicas                       │  ← steady state
         └─────────────────────────────────────┘

Time 1:  ┌─────────────────────────────────────┐
         │ v1: 6 ready, 2 terminating         │
         │ v2: 2 ready                         │  ← 25% replaced
         └─────────────────────────────────────┘

Time 2:  ┌─────────────────────────────────────┐
         │ v1: 4 ready, 2 terminating          │
         │ v2: 4 ready                         │
         └─────────────────────────────────────┘

...

Time N:  ┌─────────────────────────────────────┐
         │ v2: 8 ready                          │  ← rolled out
         └─────────────────────────────────────┘

优点： 简单，内置支持，无额外成本，若流量切换平滑则实现零停机。 缺点： v1 和 v2 同时运行。任何不兼容（数据库 schema、API 契约）必须保持向后兼容性。回滚同样是一次滚动更新——速度较慢。 适用场景： 变更向后兼容的中小型服务。适用于大多数工作负载。

12.2 蓝绿部署

两套完整的生产环境，"蓝"（当前）和"绿"（新版）。在蓝版本继续承接流量的同时，将 v2 部署到绿环境。绿环境验证通过后，切换负载均衡器。

Step 1:  ┌─────────────┐         ┌─────────────┐
         │ Blue (v1)   │ ← all   │ Green       │  (empty / being built)
         │ serving     │  traffic│             │
         └─────────────┘         └─────────────┘

Step 2:  ┌─────────────┐         ┌─────────────┐
         │ Blue (v1)   │ ← all   │ Green (v2)  │
         │ serving     │  traffic│ smoke tests │
         └─────────────┘         └─────────────┘

Step 3:  ┌─────────────┐         ┌─────────────┐
         │ Blue (v1)   │         │ Green (v2)  │ ← all traffic
         │ idle        │         │ serving     │
         └─────────────┘         └─────────────┘
         (kept for rollback)

Step 4:  ┌─────────────┐         ┌─────────────┐
         │ Blue        │         │ Green (v2)  │
         │ decommiss.  │         │ now "blue"  │
         └─────────────┘         └─────────────┘

优点： 原子切换。回滚是即时的（反向切换负载均衡器）。部署期间无版本混用。 缺点： 切换期间需要 2 倍的基础设施。长连接会话、WebSocket、流式请求必须优雅排空，否则将被中断。 适用场景： 适用于回滚延迟敏感的无状态服务（例如，凌晨 3 点的部署出现故障——你希望 30 秒内完成回滚，而不是 30 分钟）。

在 Kubernetes 中的实现：部署两个 Deployment（web-app-blue 使用 v1， web-app-green 使用 v2），通过不同的标签进行区分。Service 选择器指向"生产"Deployment 对应的标签。切换 Service 选择器即可完成流量切换。

12.3 金丝雀部署

逐步将流量从 v1 切换到 v2，在每个步骤监控指标。指标健康则继续推进；指标异常则回滚。

Phase 1 (1%):    ┌────────┐  ┌──┐
                 │ v1 99% │  │v2│
                 │        │  │1%│
                 └────────┘  └──┘
                 Wait 15min. Check error rate, p95 latency, CPU.

Phase 2 (10%):   ┌─────────┐ ┌────┐
                 │ v1 90%  │ │ v2 │
                 │         │ │10% │
                 └─────────┘ └────┘

Phase 3 (50%):   ┌──────┐    ┌──────┐
                 │ v1   │    │ v2   │
                 │ 50%  │    │ 50%  │
                 └──────┘    └──────┘

Phase 4 (100%):              ┌────────┐
                              │ v2 100%│
                              └────────┘

优点： 故障影响半径被限制在金丝雀流量比例之内。能够以较小的用户影响范围提前发现 bug。基于指标的回滚可以自动化。 缺点： 发布过程较慢（对于风险敏感的部署可能需要数小时至数天）。需要完善的指标体系和流量分割基础设施。 适用场景： 适用于回归代价高昂的面向用户的服务。Netflix、Lyft、Stripe 均将其作为标准做法。

实现方式：

  • Argo Rollouts （Kubernetes 原生渐进式交付控制器）原生支持金丝雀部署。可声明式地定义发布阶段、流量权重、暂停时间和指标检查。
  • Flagger （Flux 生态系统的类似工具）具备相同功能。
  • 服务网格流量分割 （Istio VirtualService、Linkerd）在 L7 层进行流量切换。
  • Ingress 控制器权重 适用于更简单的场景（Nginx、Traefik）。

指标门控是核心所在。一个不查询 Prometheus/Datadog 且不在指标回退时自动中止的金丝雀发布，本质上只是一次慢速滚动更新。

12.4 暗发布（Dark launch）

部署代码，但通过 功能开关 （参见 13_feature_flag.md）进行隔离。流量仍然打到 v2 的 pod，但新代码路径处于休眠状态。一旦验证安全，翻转开关即可启用。

Step 1: Deploy v2 pods. Flag = OFF.
        Behavior: identical to v1 (new code path not entered).

Step 2: Enable flag for internal users only.
        Behavior: internal users see new code path; everyone else: old.

Step 3: Enable for 1% of external users.

Step 4: Enable for 100%.

Step 5: Remove the flag (and dead code) in a later release.

优点： 将部署与发布解耦。回滚就是"翻转开关"——耗时毫秒级。 缺点： 代码库中会积累功能开关的冗余代码。需要严格的纪律来清理历史开关。 适用场景： 适用于任何具有不可忽视影响半径的面向用户的变更。是"每天多次部署"文化的基石。

12.7 "周五部署"反模式

这是一个文化层面的问题，而非技术问题。其背后的思维是："不要在周五部署，因为一旦出问题会毁掉你的周末。"典型症状：

  • 部署集中在周一至周四。
  • 长期存在的功能分支会不断积累。
  • 周一大规模部署所承担的风险，远超其所取代的每日小规模部署。

解决方案不是"反正周五也要部署"，而是让 每个 部署低风险：

  • 基于主干的开发。短生命周期分支。
  • 小规模高频部署，而非大规模低频部署（Accelerate / DORA — DevOps Research and Assessment — 研究证实，部署频率与变更失败率呈负相关）。
  • 使用功能标志将部署与发布解耦。
  • 自动化金丝雀发布配合指标门控，让有问题的部署无需人工介入即可自动回滚。
  • 全天候 on-call 轮值，且不因周末事故受到惩罚。

做到位时，部署将成为一件平常小事。Stripe、Etsy 及类似工程文化的公司每天全时段部署数百次，且充满信心。

§14. 使用场景展示

14.1 典型 SaaS Web 应用部署

典型案例。一家 SaaS 公司运行约 50 个微服务：Web 前端、API 网关、若干后端服务、异步工作者、缓存、可观测性 sidecar。每个服务是一个基于 CPU 进行 HPA 扩缩的 Deployment。总计约 500 个 pod。每个环境（dev、staging、prod）一个 Kubernetes 集群。ArgoCD 负责 GitOps。Argo Rollouts 处理生产环境的金丝雀发布。标准的 Prometheus + Grafana 用于监控。服务网格（Istio）提供 mTLS + 链路追踪。

技术要求：零停机部署、快速回滚、通过命名空间实现多团队隔离。

技术变体：Kubernetes（托管版——EKS 或 GKE），无状态服务使用 Deployment，自托管的 Postgres 使用 StatefulSet，外加若干用于夜间批处理的 CronJob。

14.3 ML 训练任务

训练模型与服务模型从根本上不同。训练任务具有以下特点：

  • 长时间运行（数小时到数周）。
  • 资源密集型（多 GPU，通常通过 Horovod、PyTorch DDP 或 DeepSpeed 进行分布式训练）。
  • 批处理形态（运行至完成后退出）。
  • 多步骤（预处理 → 训练 → 评估 → 注册）。

Kubernetes 原生 ML 技术栈：

  • Argo Workflows 或 Kubeflow Pipelines 用于 DAG。每个步骤是一个容器；工作流工具对其进行排序。
  • Volcano 调度器（或 Kubernetes 自身的 gang-scheduling 支持）——协同调度分布式训练任务的所有 pod 以同时启动。若没有此机制，等待所有副本可运行时可能发生死锁。
  • NVIDIA GPU operator + nvidia.com/gpu 用于 GPU 调度的资源类型。
  • Cluster autoscaler 扩缩 GPU 节点组。GPU 节点价格昂贵（约 $3-30/小时）；空闲时缩容至零。

一个典型训练运行：包含五个顺序步骤的 Argo Workflow；训练步骤是一个运行 8 小时的 4-GPU 任务；工作流处理重试、制品传递（模型检查点到 S3）和清理。

技术要求：GPU 调度、gang 调度、持久中间制品、与实验追踪系统集成（MLflow、Weights & Biases）。

变体：Job + Volcano 调度器 + NVIDIA device plugin + 上层工作流工具。

14.4 批处理

经典 ETL 工作负载。夜间任务从数据仓库读取数据，进行转换，写入另一个数据仓库。每天运行一次，耗时 2 小时。

Kubernetes 原生批处理：

  • CronJob 用于调度。
  • Job 由 CronJob 生成，配合 backoffLimit: 3, activeDeadlineSeconds: 7200.
  • Argo Workflows 或 Tekton （若批处理为多步骤或有扇出，例如以 100 并发处理 10,000 个文件的工作流）。

一个典型模式：每天 UTC 02:00 执行 CronJob 。它创建一个 Job 与 parallelism: 50, completions: 10000。Kubernetes 并发调度 50 个 pod；每个完成后，另一个启动，直到全部 10,000 个完成。

技术要求：定时执行、扇出并行、带重试的失败处理、资源公平性（不饿死交互式工作负载）。

变体：CronJob + Job + 用于编排的工作流工具。

14.5 边缘部署

在靠近用户的地方运行计算——零售店、工厂、电信、车辆。约束条件：

  • 每个站点计算资源有限（通常是单台小型服务器、Raspberry Pi 集群或少量节点）。
  • 与中央云的网络连接不稳定。
  • 由非专业人员操作（店长、技术人员）。
  • 站点数量庞大（数千到数万个）。

解决方案：

  • k3s ——Rancher 出品的 50 MB 单二进制 Kubernetes 发行版。专为边缘设计。单节点部署默认以 SQLite 替代 etcd。可在 Raspberry Pi 上运行。
  • KubeEdge ——将 Kubernetes 扩展以管理可能离线的节点。连接可用时，Pod 同步到边缘节点；断连期间，边缘节点继续运行 Pod。
  • Nomad ——Cloudflare 使用 Nomad 管理全球 200 多个 POP（Points of Presence）。每个 POP 运行一个 Nomad agent；中央 Nomad 集群在其间进行调度。
  • Argo CD ApplicationSet ——将相同的 manifest 部署到从列表生成的 N 个目标集群（例如，每家店一个集群）。

技术要求：极小的资源占用、容忍断连、便捷的批量管理、简单的单站点升级。

变体：k3s + GitOps + 精心调整的资源请求。

§15. 真实世界中的实现与数字

以具名系统为例，展示容器编排在不同场景下的大规模落地。

Google Borg → Kubernetes

鼻祖。Google 内部的 Borg 系统（论文："Large-scale cluster management at Google with Borg"，EuroSys 2015）几乎支撑着 Google 的全部业务。一个典型的 Borg cell 包含 10,000 台机器；Google 拥有数百个 cell。总规模达数百万台机器、数十亿个容器。

Borg 首创：声明式作业规范、两级调度（Borg → 任务 → Linux 进程）、优先级/抢占机制、基于 cgroups 的隔离。Kubernetes（2014 年开源）是从 Borg 中提炼出的外化经验成果。

Kubernetes 本身目标规模为每集群 5,000 个节点；Google 内部将 Kubernetes 用于部分工作负载，但最大规模场景仍依赖 Borg。

Spotify

Spotify 在 2018-2020 年间从自研系统迁移到 Kubernetes。平台团队开源了 Backstage ——一个将 Kubernetes 部署、可观测性和所有权关联在一起的开发者门户。公开数据：数千个服务，约 1 万个 pod，每天数百次部署。每个地区多个集群，通过 Flux 实现 GitOps。

Airbnb

Airbnb 在多个地区的 Kubernetes（主要是 EKS）上运行约 100 个生产服务。从自研部署系统迁移而来。公开演讲中提到，在对 HPA + Cluster Autoscaler 进行适当调优后，部分工作负载迁移到 Kubernetes 后成本降低了约 70%，这得益于更好的装箱效率。

Lyft

Lyft 是公开讨论最多的 Envoy 用户之一（Envoy 诞生于 Lyft）。他们在 Kubernetes 上运行包含数千个服务的服务网格，每天处理数十亿次请求。Lyft 平台团队为云原生生态系统做出了大量贡献。

Pinterest

Pinterest 在 Kubernetes（约 1 万个 pod）上加上自有平台抽象层运行。他们已发表过关于多集群 ingress、大规模 GitOps 以及采用 Argo Rollouts 的文章。

Stripe

Stripe 在多个集群的 Kubernetes 上大规模运行。他们发表过关于大规模 Argo CD、精心调优 pod 启动延迟（镜像拉取是主要因素，通过积极缓存缓解）以及数十万个 pod 运营模型的文章。

Netflix

Netflix 部分运行在 Kubernetes 上（多年来通过其内部平台"Titus"，近期更多地引入 Kubernetes），部分直接运行在 EC2 上。Spinnaker 在 Netflix 创建，作为跨两者的部署编排器。其使用场景是典型的"数千个服务、每天数百次部署、多地区主动-主动"模式。Open Connect（Netflix 的 CDN 边缘服务器）运行 Nomad。

Cloudflare

Cloudflare 在全球 200 多个 POP 上运行 Nomad。他们公开讨论了为何 Nomad 更小的资源占用（相对于 Kubernetes）适合边缘场景：每个 POP 节点数量有限，多工作负载模型允许混合运行容器和原始二进制文件，而单一二进制文件更易于在如此多的站点上部署。

AWS — ECS 和 Fargate

Amazon ECS 被大量 AWS 客户使用——Slack、Lyft（除 Kubernetes 外）、众多 SaaS 初创公司。ECS 比 Kubernetes 更简单（无需管理集群），与 AWS 深度集成（IAM — Identity and Access Management — 角色通过任务角色 ARN 映射到任务，VPC — Virtual Private Cloud — 网络原生支持，ALB — Application Load Balancer — 目标是一等公民）。底层的 Fargate 类似 AWS Lambda：AWS 分配微型虚拟机（Firecracker），运行容器，按 vCPU 小时计费。被超过 10 万个客户使用。

§16. 总结

容器编排是运行时层，它将"我需要此服务的 N 个副本"转化为现实：控制平面将期望状态存储在 Raft 复制的 KV 存储中，控制器运行协调循环将意图转化为行动，调度器将 pod 装箱到节点上，每节点 agent 通过用于计算（CRI）、网络（CNI）和存储（CSI）的标准化接口驱动容器运行时。在该运行时之上，部署模式——滚动、金丝雀、蓝绿、暗启动、影子——以及交付系统——使用 Argo 或 Flux 的 GitOps、使用 Argo Rollouts 的渐进式交付、使用 Spinnaker 的推送流水线——将软件发布从英雄式事件转变为持续的、声明式的、基于水平触发的过程。该技术的契约比表面看起来更窄：它承诺在机器集群上对临时工作负载进行调度、生命周期管理和抽象，而非持久状态、跨集群原子性或业务正确性——能够出色运用 Kubernetes 的团队与不能的团队之间的差距，很大程度上在于前者理解了这一窄契约而后者没有。
