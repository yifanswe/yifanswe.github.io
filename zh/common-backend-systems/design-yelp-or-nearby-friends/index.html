1. 基础知识与解题思路

本题考察点

Yelp（或附近的朋友）是地理空间搜索："找到距我 R 范围内的地点（或朋友），按类别过滤，按相关性排序。"难点在于对数百万个位置建立索引、支持亚秒级查询、根据评论质量进行排名，以及处理更新（新评论、商家信息变更）。

其不变量如下：

目录中的每家商家都可通过地理查询和类别过滤发现；评论被持久存储、归属于作者，并汇总为每商家评分；垃圾评论被过滤；营业时间和特殊状态（已关闭、已认领）反映真实情况。

对于附近的朋友：每个用户的当前位置与选定好友私密共享；R 范围内的好友会收到在线指示符；位置数据是临时性的。

在 ICT5/L6+ 级别，面试官在考察你是否能设计地理空间索引（geohash、S2、H3）、商家和评论数据模型、排名算法、图片/媒体处理，以及对位置共享中隐私问题的推理能力。

60 秒面试框架

口述版本：

我会围绕地理空间商家索引、具备垃圾过滤和每商家汇总的评论服务，以及图片媒体服务来设计 Yelp。地理空间查询使用 H3 或 geohash 单元格。搜索将地理过滤与类别和排名信号相结合。对于附近的朋友，位置信息是临时性的，基于每个好友的同意授权；相同的地理索引服务于两者，但访问模型不同。有趣的设计决策在于地理索引、排名、评论汇总和隐私保护。

首先明确需求

功能性需求：

  • 按位置和类别搜索商家。
  • 查看商家详情：营业时间、图片、评论、评分。
  • 发布带评分和图片的评论。
  • 签到（Yelp）；在线状态（附近的朋友）。
  • 按价格、评分、距离过滤。
  • 图片、菜单、预订（超出核心范围）。
  • 商家认领和管理。

非功能性需求：

  • 亚秒级搜索。
  • PB 级图片存储。
  • 评论写入吞吐量适中；读取量非常高。
  • 评分更新在数秒内反映。
  • 多地区。

以下内容超出范围，除非被特别询问：

  • 预订系统（独立）。
  • 配送集成。
  • 基于评论的广告定向。

规模假设

领域 假设
商家数量 全球 1000 万以上。
评论 1 亿以上；持续增长。
用户 数千万。
搜索 QPS 每秒 1 万以上。
照片 数十亿条；数十 PB。
Latency（延迟） 搜索 < 500 毫秒；评论发布亚秒级。

瓶颈：带相关性排名的地理空间查询，以及图片存储成本。

地理空间索引

H3（或 S2）六边形单元格层级：

  • 每个位置在各个分辨率下都映射到一个单元格。
  • 分辨率 9 约 100 米，分辨率 8 约 250 米，以此类推。
  • 对于"查找距经纬度 1 千米范围内的商家"：计算覆盖该圆盘的单元格；查询每个单元格的商家集合。

索引存储：

  • 每个单元格对应一个 business_id 列表。
  • 热门地区存于内存；冷数据有持久化备份。
  • 更新：商家位置变更时在单元格之间移动。

对于感知类别的查询：按（单元格, 类别）建立索引，使"1 千米内的咖啡店"成为快速查询。

面试用语：

H3 单元格将"附近"转化为"在这些单元格中"，再进行过滤和排名。

搜索流程

对于"旧金山 SoMa 最好的意大利餐厅"：

  1. 将位置（旧金山 SoMa）地理编码为经纬度或单元格。
  2. 地理过滤：目标区域所覆盖单元格内的商家。
  3. 类别过滤：意大利餐厅。
  4. 其他过滤：价格、评分、当前是否营业。
  5. 排名：综合距离、评分、评论数、热度、个性化的相关性评分。
  6. 返回前 N 个结果，附带摘要（名称、评分、图片、距离）。

延迟预算：500ms 以内。主要耗时分布：- 单元扫描：数十毫秒。- 过滤与排序：数百毫秒（取决于结果集大小）。- 照片 URL 填充：数十毫秒。

排序

多信号排序：

  • 距离。
  • 平均评分（针对低评论数进行贝叶斯调整）。
  • 评论数量（评论越多 = 越可靠）。
  • 评论的时效性。
  • 照片质量。
  • 个性化（用户过往访问记录、偏好）。
  • 赞助加权（付费排名）。

多阶段：廉价地理 + 类别过滤 → 宽泛候选集；ML 排序器对顶部候选项评分；返回结果。

评论聚合

当一条评论被发布时：

  1. 持久化评论。
  2. 更新商户信息 rating_avg 。 review_count （最终一致）。
  3. 异步发送事件进行垃圾检测。
  4. 如果被标记，则标记 spam_status=suspicious；聚合重新计算时排除该评论。

评分聚合：

  • 存储在商户记录中： rating_avg, review_count.
  • 由消费评论事件的聚合服务负责更新。
  • 如果单个热点商户出现流量激增，则使用子计数器分片。

贝叶斯平滑：

  • 一个只有 1 条 5 星评论的商户，不应排在拥有 100 条评论且平均 4.8 星的商户之上。
  • 调整后评分 = (avg * count + prior * confidence) / (count + confidence)。

垃圾评论与虚假评论

持续性对抗。

信号：

  • 评论者的账户注册时长与历史记录。
  • IP/设备模式（大量评论来自同一 IP）。
  • 文本模式（模板化内容、可疑措辞）。
  • 评分异常（短时间内出现 5 星集群）。
  • 竞争对手的 1 星恶意刷评。

流水线：

  • 对每条评论进行异步 ML 评分。
  • 高置信度垃圾评论自动删除。
  • 边界情况进入人工审核。
  • 可疑评论从聚合中排除，但根据策略可见（或隐藏）。

照片

  • 通过单独的上传流水线上传。
  • 存储在对象存储中；缩略图异步生成。
  • 通过 CDN 提供访问服务。
  • 内容审核：自动化分类器 + 针对被标记内容进行人工审核。

附近好友

独立平面，叠加在同一地理空间索引之上。

  • 用户选择加入；设置可见时长。
  • 客户端定期（约 30 秒）向服务器发送位置信息。
  • 服务器以带 TTL 的方式在内存中存储位置。
  • 基于好友的 ACL：控制谁可以查看该用户的位置。
  • 在半径 R 范围内的好友可在其应用中收到在线状态。

隐私：

  • 始终需要选择加入。
  • 时间限制（例如，"分享 1 小时"）。
  • 基于好友的精细化控制。
  • 位置精度可适当粗化（使用城市级别而非街道级别）。
  • 审计日志。

扩展与分区

数据 分区键 原因
商家目录 business_id 哈希值 直接查找。
地理索引 地理单元或区域 空间查询。
评论 business_id 哈希值 按商户聚合。
照片 business_id hash；CDN 按商户热点。
好友位置 user_id hash；内存 临时性。
搜索索引 按区域部署 本地查询模式。

热点商户（病毒式传播的地点）：为评论聚合使用子计数器；积极缓存。

一致性模型

层级 一致性 原因
商户信息编辑 强一致性 商户主期望变更立即生效。
评论写入 强一致性 可审计性。
评分聚合 最终一致 用户体验。
地理索引 最终一致（秒级） 商户地址变更较为罕见。
搜索结果 最终一致 有界时效性。
好友位置 最终一致（内存中） 隐私 + TTL。

故障模式与缓解措施

故障模式 缓解措施 监控指标
垃圾评论泛滥。 ML 过滤 + 人工审核；速率限制。 可疑速率峰值。
商户位置信息有误。 商户主纠正；地图服务商集成。 地理争议数量。
聚合延迟。 从评论重新计算。 聚合与重新计数之间的数据漂移。
照片滥用。 内容审核流水线。 举报机制。
搜索延迟。 缓存热门查询；背压。 搜索 p99。
好友隐私泄露。 严格 ACL；审计日志；漏洞赏金计划。 隐私投诉。
地理编码器错误。 多服务提供商；降级回退。 地理编码失败率。
热点商户（病毒式传播）。 缓存聚合数据；为点赞/访问使用子计数器。 每商户 QPS。
已关闭的商户仍被展示。 状态标志；定期核验。 已关闭但仍可见的商户数量。
多区域复制延迟。 异步；有界。 复制延迟。

运维指标

正确性：

  • 垃圾检测率；误报率。
  • 聚合数据漂移。
  • 地理编码精度。

容量：

  • 按区域搜索 QPS。
  • 评论写入速率。
  • 照片出口流量。

业务层面：

  • 每商户评论数；用户贡献量。
  • 转化率（查看 → 访问）。
  • 认领率。

多区域策略

  • 为本地搜索构建分区域地理索引。
  • 商户目录在全球范围内复制。
  • 评论写入以区域为主；复制至全局聚合。
  • 照片通过 CDN 提供服务。

需要指出的权衡

  • 贝叶斯平滑在响应速度与新商户公平性之间进行权衡。
  • 激进的垃圾过滤可减少噪音，但会误伤合法的差评。
  • 详细的个性化可提升相关性，但引发隐私顾虑。
  • 缓存聚合数据可加速读取，但增加了缓存失效的复杂性。
  • 地理单元分辨率越高，精度越好，但内存开销越大。
  • 实时位置共享会增加负载；粗化位置可降低精度。

常见陷阱与更优做法

陷阱 更优做法
将地理信息仅作为经纬度过滤处理。 基于 H3/S2 单元的索引；空间扫描。
简单平均评分。 贝叶斯平滑。
忽略垃圾评论。 ML 流水线 + 人工审核。
对所有用户使用单一排序分数。 结合个性化的多信号排序。
照片存储在应用服务器上。 独立媒体服务 + CDN。
好友位置长期存储。 临时存储 + TTL + 基于好友的 ACL。

面试官可能深入探讨的方向

深入方向 优秀回答要点
地理索引 H3 单元；多分辨率；空间查询。
排序 贝叶斯评分 + 多信号 + 个性化。
垃圾评论 异步 ML 流水线；包含原因说明；人工审核。
聚合 热点商户使用子计数器；定期重新计数。
附近好友隐私 选择加入；TTL；基于好友的 ACL；位置粗化。
多区域 按区域划分的地理索引；全局目录；复制聚合数据。

Staff 级别回答检查清单

  • 基于 H3/S2 的地理索引。
  • 商户目录与地理索引分离。
  • 带聚合反规范化的评论存储。
  • 垃圾过滤作为异步 ML 任务处理。
  • 贝叶斯评分平滑。
  • 照片服务独立部署。
  • 附近好友采用注重隐私保护的设计。
  • 多区域部署，配备区域级索引。

面试用语

Yelp 的核心是地理空间搜索加评论聚合。H3 单元使"附近"查询成为快速索引查找。贝叶斯平滑评分可防止低评论量的商户主导排名。垃圾过滤异步运行，以保持评论发布路径的高速响应。对于附近好友功能，地理索引服务于不同的访问模型，并配备严格的隐私保护机制。

2. 模拟面试

面试官： 设计 Yelp。

候选人： 地理空间商户索引加评论服务加照片服务，由一个复合搜索服务统一查询。不变式是：每个商户均可按位置和类别被发现；评论正确聚合为每商户评分；垃圾评论被过滤。

面试官： 地理索引方案？

候选人： H3 六边形单元。每个商户以精细分辨率（约 100 米）映射到一个单元。按位置搜索：计算目标周围圆盘覆盖的单元，查询每个单元的商户集合，按类别和其他条件过滤，排序，返回结果。对于类别感知查询，按 (cell, category) 建立索引以跳过过滤步骤。

面试官： 排序？

候选人： 多信号排序。距查询点的距离、贝叶斯平滑评分、评论数量、时效性、热度、个性化。多阶段：廉价的地理 + 类别过滤生成宽泛候选集；ML 排序器对顶部候选项评分。贝叶斯平滑可防止 1 条评论 5 星的商户超过 100 条评论 4.8 星的商户。

面试官： 评论——如何处理聚合？

候选人： 每商户的 rating_avg 。 review_count 存储为反规范化数据。聚合服务消费评论事件并更新这些数据。最终一致——新评论可能需要数秒才能反映。对于热点商户（新近病毒式传播），子计数器分片可应对写入速率。定期从评论进行全量重新计数，以保持聚合数据的准确性。

面试官： 垃圾评论怎么处理？

候选人： 异步 ML 流水线。信号：账户注册时长、IP/设备聚类、文本模式、评分聚类、竞争对手 1 星恶意刷评。高置信度垃圾评论自动删除。边界情况进入人工审核。可疑评论从聚合中排除，但根据策略可能对用户仍然可见（或隐藏）。流水线在发布路径之后运行，使评论能够快速出现；如果事后被标记，聚合数据随之调整。

面试官： 一个用户在旧金山 SoMa 搜索"italian"，会发生什么？

候选人： 前端对"SoMa"进行地理编码，或使用用户 GPS 定位。搜索服务计算目标周围约 1-2 公里覆盖的 H3 单元。查询每个单元的 (cell, category=italian) 索引。获取候选商户列表。按其他条件过滤（是否营业中、价格、评分）。按综合分数排序。为顶部结果填充照片和摘要。返回结果。总耗时低于 1 秒。

面试官： 一张照片随评论一起上传。

候选人： 照片通过独立的媒体服务上传；使用签名 URL 上传至对象存储。缩略图异步生成。审核流程运行（自动化 + 如被标记则进行人工审核）。照片与评论和商户关联。通过 CDN 提供服务。

面试官： 附近好友——如何设计？

候选人： 叠加在同一地理空间索引之上的独立平面，但配备严格的隐私保护。用户选择加入并设置可见时长（"分享 1 小时"）。客户端每约 30 秒发送一次位置信息。服务器以带 TTL 的方式在内存中存储位置。基于好友的 ACL 控制谁可以查看谁的位置。在半径 R 范围内的好友会收到在线状态指示。位置可根据用户偏好进行粗化（城市级别）。记录访问的审计日志。

面试官： 附近好友的隐私顾虑？

候选人： 有几点：仅限选择加入，默认关闭。时间限制共享。基于好友的精细化控制（不同好友看到不同精度）。在服务器端进行位置粗化。不进行长期留存，设有 TTL。严格的 ACL 强制执行。用户可访问审计日志。设有漏洞赏金计划以应对任何泄漏。我们将位置视为最敏感的数据。

面试官： 多区域？

候选人： 按区域划分的地理索引——商户按区域存储于本地。商户目录在全球范围内复制，用于存储权威信息。评论写入以区域为主；进行复制。每区域聚合，并汇总至全局聚合。照片在全球各地通过 CDN 提供服务。好友位置是临时性的，仅存储于本地区域。

面试官： 一个商户有 5 星但只有 2 条评论，它应该排名靠前吗？

候选人： 不应该。贝叶斯平滑可以处理这种情况。有效评分 = (5 * 2 + prior * confidence) / (2 + confidence)。当 prior=3.5 且 confidence=10 时，计算结果为 (10 + 35) / 12 = 3.75，远低于一个拥有大量评论、平均 4.8 星的成熟商户。随着商户积累更多评论，平滑效果会逐渐减弱。

面试官： 总结。

候选人： H3 单元地理索引加商户目录加评论服务加照片服务。搜索按位置和类别过滤，按贝叶斯平滑多信号分数排序，并填充结果。评论聚合为最终一致；垃圾评论异步过滤。附近好友功能基于同一索引，配备严格隐私保护：选择加入、TTL、ACL、位置粗化。多区域部署，配备区域级索引和全局目录。最难的正确性问题是评论垃圾；最难的扩展性问题是热点商户聚合。

3. 练习备注

五分钟版本

  1. 明确范围：搜索、评论、照片、好友。
  2. 不变式：可被发现；有评分；垃圾已过滤。
  3. H3 地理索引加商户目录。
  4. 多信号贝叶斯排序。
  5. 异步垃圾过滤流水线。
  6. 照片 + CDN。
  7. 附近好友：选择加入、TTL、ACL、位置粗化。
  8. 多区域索引；全局目录。

一分钟结尾陈述

Yelp 是带排序评论的地理空间搜索系统。H3 单元使"附近"查询高效；贝叶斯平滑评分可处理低评论量的商户；多信号排序融合了距离、热度和个性化因素。评论聚合为最终一致；垃圾过滤采用异步 ML。照片通过 CDN 提供服务。附近好友功能基于同一地理索引，配备严格的隐私保护：选择加入、临时存储、基于好友的 ACL。
