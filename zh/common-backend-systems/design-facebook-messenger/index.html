1. 基础知识与解题思路

本题考察点

Facebook Messenger（或 WhatsApp / iMessage）是实时的一对一和群组消息系统。难点在于：将每条消息精确传递给每位参与者一次，保证正确的顺序，跨设备工作，支持离线接收方，并提供已读回执/正在输入指示/在线状态，同时不使系统变得脆弱。

其不变量如下：

每条已发送的消息均被持久存储，至少在每位会话参与者的每台活跃设备上投递一次，跨设备显示顺序一致，且离线时仍可访问；删除和编辑最终在所有地方可见；已读回执和在线状态尽力而为，但绝不在未真实发生时声称已发生。

在 ICT5 / L6+ 级别，面试官在考察你是否能将消息基础设施（持久化会话日志）与投递基础设施（在线推送、离线拉取）分离，处理多设备同步，设计端到端加密（若被要求），以及对群组扇出与一对一模式进行权衡分析。

60 秒面试框架

口述版本：

我会围绕每会话追加式日志作为事实来源来设计 Messenger。客户端从其在日志中的位置开始同步；实时推送通道在在线时通知新消息。多设备同步通过每设备游标和共享邮箱解决。对于群组，会话日志仍是追加式的；只是有更多参与者拉取。关键决策在于：每会话排序、多设备同步、投递确认以及端到端加密。

首先明确需求

功能性需求：

  • 用户发送一对一和群组消息（文本、媒体、表情回应）。
  • 消息必须被持久化，并在离线时可用。
  • 已读回执、正在输入指示、投递状态。
  • 在线状态（在线 / 最近上线时间）。
  • 多设备支持（手机与网页同时在线）。
  • 编辑和删除消息。
  • 消息历史同步到新设备。
  • 可选：端到端加密。

非功能性需求：

  • 在线用户的消息投递延迟低于一秒。
  • 消息保留数月/数年。
  • 强每会话排序。
  • 用户可见层面精确投递一次（底层为至少一次）。
  • 多区域；消息跨洲际在数秒内送达。
  • 消息基础设施高可用。

以下内容超出范围，除非被特别询问：

  • 语音/视频通话（WebRTC；属于不同问题）。
  • 动态/状态（独立功能）。
  • 全消息搜索（庞大的独立问题）。
  • 端到端加密（提及；按需深入）。

规模假设

领域 假设
用户 20 亿以上月活用户。
每日消息数 全球超过 1000 亿条消息。
平均用户 数十个会话，每天数百条消息。
群组规模 最多数千人（WhatsApp 1024+；Messenger 不等）。
峰值 峰值每秒数百万条消息。
每用户设备数 通常 2-3 台；某些情况下最多 4-5 台。
Latency（延迟） 在线投递低于一秒；离线时重连后接收。
留存期 默认无限期保留。

瓶颈在于超大群聊的每会话吞吐量以及全局在线状态/正在输入指示的开销。

回执 每设备的已投递/已读状态。 每设备状态；会话级聚合。

核心组件

组件 职责 关键细节
发送服务 接收出站消息；写入日志；广播。 基于客户端侧消息 id 实现幂等。
会话日志存储 每会话追加式消息。 按 conversation_id 分片；强每会话顺序。
每设备收件箱游标 追踪每设备每会话的最后已投递序号。 由设备确认后更新。
推送通道 与在线设备的持久连接（WebSocket / MQTT）。 用于新消息通知和在线状态。
拉取 / 同步服务 当设备轮询时返回自游标以来的消息。 有界批次；分页。
回执服务 存储每条消息每设备的已投递/已读回执。 汇聚后用于发送方展示。
存在感知服务 在线/离线/正在输入状态。 内存存储带 TTL；不持久化。
媒体服务 图片/视频上传与访问。 与消息元数据分离。
端到端加密（可选） 每会话密钥管理；每消息加密。 服务器仅存储密文。

WebSocket: persistent connection for real-time push

写入路径：发送消息

  1. 客户端以 client_message_id （幂等键）发起 POST 请求。
  2. 发送服务检查幂等性；若为重复请求，返回先前结果。
  3. 分配服务端 message_id 以及 seq 每会话单调递增的序号。
  4. 持久化到会话日志存储（已复制，持久化）。
  5. 更新会话的最新序号。 last_message_seq.
  6. 向每会话流发出事件。
  7. 对于每位接收方设备：若在线（在线状态服务确认），通过 WebSocket 推送包含消息元数据或完整内容的通知；若离线，设备在下次重连时通过拉取获取。
  8. 向发送方返回成功，附带已分配的序号。 message_id 。 seq.

发送响应不等待投递确认；日志写入完成后发送方即收到消息确认。

读取 / 同步路径

每台设备对其所在的每个会话维护一个游标。

连接 / 唤醒时：

  1. 设备发送每个会话的当前游标（或一个全局同步游标）。
  2. 同步服务查询日志存储，获取满足以下条件的消息： seq > cursor 每会话。
  3. 返回消息，上限为批次大小；客户端在有更多消息时分页获取。
  4. 设备确认 delivered_seq 每会话。
  5. 服务器更新 DeviceCursor.

实时推送：

  1. 客户端维护 WebSocket 连接。
  2. 设备参与的会话中的新消息会被推送。
  3. 推送包含完整内容（小消息）或元数据（客户端再拉取完整内容）。
  4. 客户端确认；游标推进。

若 WebSocket 断开，设备在重连时回退到拉取——不丢失任何消息。

面试用语：

推送通道是优化手段，而非事实来源。事实来源是会话日志。客户端始终可以通过从游标拉取来恢复。

每台设备的状态与同步

一个拥有手机、网页端和平板的用户有三台设备。每台设备必须：

  • 查看相同的对话历史。
  • 从任意设备发送消息。
  • 在所有在线设备上接收新消息。
  • 拥有独立的已读状态（在手机上已读不等于在网页端已读）。

实现方式：

  • 每台设备都有各自的 DeviceCursor 每会话。
  • 从任意设备发送的消息都会以该用户作为发件人持久化到对话日志中。
  • 所有收件人设备（包括发件人的其他设备）都会收到该消息。
  • 已读回执按设备维度记录，但会聚合后再展示。

新设备登录时：

  • 根据策略，将游标初始化到"最新"位置或拉取近期历史。
  • 后续同步为增量同步。
  • 对于端对端加密，向新设备进行密钥交换是一个复杂的子问题（类似 Signal Protocol 的方式）。

群组消息

日志结构相同，只是参与者更多。

  • 对话日志以群组为单位，是一个仅追加的数据流。
  • 每个参与者的设备在该日志上维护各自的游标。
  • 发送扇出：只需向群组日志写入一次；收件人设备从同一来源拉取或推送。
  • 按群组强序（seq）。

大型群组（1000+ 成员）：- 推送扇出成本随群组规模和在线成员数量线性增长。- 预计算在线成员列表；仅向他们推送。- 离线成员在重连时拉取。

面试用语：

群组不会使存储量成倍增加，而是使读放大成倍增加。一次写入，多个读者——而读操作代价低廉，因为它们是从游标开始的仅追加日志扫描。

已读回执与投递状态

每条消息可能有大量回执：每个收件人设备的"已投递"和"已读"状态。

两种视图：

  • 发件人视图： "我的消息已被 M 人中的 N 人投递/已读。"
  • 系统： 按 (message_id, user_id, device_id) 存储回执。

对于拥有数千名成员的群组，存储所有回执开销很大。部分平台会：

  • 聚合为计数器：delivered_count、read_count。
  • 一旦所有参与者确认，显示"所有人已读"。
  • 对于超大型群组，跳过每个收件人的详细信息。

在线状态与正在输入

短暂性的、尽力而为的、内存中的。

  • 在线状态：在 presence 存储中维护带 TTL 的条目，通过 WebSocket 心跳刷新。
  • 正在输入：短暂事件（持续几秒），广播给对话参与者。
  • 最后在线时间：在断开连接时更新，并受隐私控制。

该数据不持久；重启可能导致在线状态短暂丢失，客户端对此能接受。

顺序性

单个对话内的顺序至关重要；跨对话的顺序则无关紧要。

  • 序列号由日志存储按对话维度分配（原子递增或单主节点）。
  • 所有参与者看到相同的顺序。
  • 发送时的时钟偏差无关紧要；由服务端分配 seq 。 sent_at.

多地区方案：对话有一个归属地区（通常是第一个参与者或创建者所在的地区）。发送操作会路由到归属地区进行排序，然后复制到其他地区。

端对端加密（如被问及）

  • 每个用户拥有身份密钥（Signal Protocol：身份密钥、签名预密钥、一次性预密钥）。
  • 新对话使用 X3DH 密钥协商。
  • 通过 Double Ratchet 生成每条消息的密钥，实现前向保密。
  • 服务端仅存储密文。
  • 服务端无法读取消息内容，但仍按 conversation_id 路由并按 seq 排序。
  • 多设备：每台设备拥有独立的密钥对；发件人分别对每个收件人设备加密，或对群组使用 Sender Keys。

扩展与分区

数据 分区键 原因
会话日志 conversation_id 哈希值 每个对话的强排序；低成本读取。
设备游标 user_id 哈希值 每用户状态。
回执 conversation_id + message_id 发件人侧聚合。
存在感知 user_id 哈希值 短暂性；可存储在 Redis 中。
推送连接 通过以下方式绑定到推送网关 device_id 维护 WebSocket 亲和性。

一致性模型

层级 一致性 原因
对话日志写入 每个对话的强序（单一 seq 顺序） 需要按对话排序。
跨对话顺序 最终一致 无关紧要。
设备游标 更新时强一致 避免消息丢失。
回执 最终一致 用户体验提示。
存在感知 最终一致，带 TTL 天然短暂性。

故障模式与缓解措施

故障模式 缓解措施 监控指标
推送通道断开；用户离线。 从游标处重连时拉取；不会丢失消息。 重连频率；拉取量。
重试发送；出现重复发送。 幂等，基于 client_message_id. 重复发送率。
对话日志分片过热（超大群组）。 在对话内进行子分片；或接受现状并扩展分片。 每分片 QPS。
消息到达客户端时顺序混乱。 客户端缓冲并按 seq 顺序应用。 客户端重排序次数。
游标损坏/丢失。 服务端保留权威游标；客户端可请求全量同步。 同步错误率。
多设备发送竞争。 每个对话的服务端 seq 可防止冲突。 发送冲突率。
群组扇出背压。 仅向在线用户推送；离线用户拉取；过载时进行退避。 推送队列深度。
在线状态服务宕机。 将所有用户视为离线；消息仍通过拉取方式投递。 在线状态服务可用性。
回执爆炸（大型群组）。 聚合为计数器；将详细信息限制在 1:1 和小型群组。 回执存储增长。
发送时的跨地区延迟。 将请求路由到对话归属地区；发件人接受延迟。 跨地区发送次数。

运维指标

正确性：

  • 消息丢失率（必须接近零；游标可检测）。
  • 乱序投递率。
  • 用户可见的重复投递次数。
  • 回执准确率。

容量：

  • 按对话类型统计的发送 QPS。
  • 推送扇出速率。
  • WebSocket 连接数。
  • 同步请求速率。

业务层面：

  • DAU；每用户每天的消息数。
  • 群组规模分布。
  • 读延迟/投递延迟。

多区域策略

  • 对话有一个归属地区（通常是活跃参与者最多的地区）。
  • 写入操作路由到归属地区进行排序。
  • 日志复制到其他地区以支持本地读取。
  • 推送网关是区域性的；归属地区向当地参与者所在的远程推送网关发出通知。
  • 在线状态是区域性的，并按需跨地区传播。

需要指出的权衡

  • 每个对话的强排序简化了用户体验，但限制了单个对话的吞吐量。
  • 从游标拉取的方式健壮，但要求游标被可靠地追踪。
  • 在推送消息中存储完整内容与仅存储元数据之间，需要在带宽和延迟之间权衡。
  • 按设备维护回执可提高精度，但会使大型群组的存储膨胀。
  • 在线状态准确性与网络成本之间的权衡：心跳越频繁 = 在线状态越新鲜 = 耗电越多。
  • 端对端加密阻止了服务端侧的内容审核；需要与安全功能进行权衡。

常见陷阱与更优做法

陷阱 更优做法
将推送视为投递机制。 对话日志是真实来源；推送是优化手段。
所有对话共用单一全局 seq。 每个对话独立的 seq；无需全局协调。
按收件人的邮箱存储。 每个对话一个日志；每台设备维护游标。
发件人看到"已发送"前需等待所有收件人同步确认。 发件人在日志写入后即看到"已发送"；确认是按收件人异步的。
忽略多设备场景。 每个对话的设备游标；每台设备的密钥管理。
大型群组使用完整记录存储回执。 超过 N 个成员时聚合为计数器。

面试官可能深入探讨的方向

深入方向 优秀回答要点
多设备同步 每设备游标；新设备引导；端对端加密时的密钥交换。
群组扇出 单一日志；在线推送 + 离线拉取；预计算在线列表。
端对端加密 Signal 协议；Double Ratchet；群组使用 Sender Keys。
大规模回执 聚合计数器；1:1 和小型群组使用每条回执记录。
存在感知 带 TTL 的内存存储；区域性；不持久。
多区域 每个对话的归属地区；复制读取；跨地区推送。

Staff 级别回答检查清单

  • 以每个对话的仅追加日志作为真实来源。
  • 每台设备的游标用于同步。
  • 推送是优化手段；从游标拉取是回退方案。
  • 带有 client_message_id 的幂等发送。
  • 多设备通过独立游标处理。
  • 群组扇出是一次写入，多个游标。
  • 在线状态和正在输入是短暂性的，与日志分离。
  • 如被问及，需考虑端对端加密。

面试用语

Messenger 是一个每个对话的仅追加日志，带有游标。日志是真实来源；推送是一种优化手段。多设备同步只需在同一日志上维护每台设备的游标。群组不会使存储量成倍增加，而是使读者数量成倍增加，而由于读取是基于游标的日志扫描，所以代价低廉。

2. 模拟面试

面试官： 设计 Facebook Messenger。

候选人： 我将围绕每个对话的仅追加日志作为真实来源、每台设备的游标用于同步，以及在线投递的实时推送通道进行设计。不变量是每台设备对用户可见的消息恰好投递一次，并保证每个对话内的强排序。我将逐步介绍发送、同步、多设备和群组。

面试官： 带我走一遍消息发送的流程。

候选人： 客户端使用客户端生成的以下字段 POST 请求 message_id 用于幂等性。服务端检查是否重复。如果是新消息，则分配服务端的 seq（每个对话单调递增），持久化到对话日志（已复制），更新对话元数据，发出事件。对于每个收件人设备，如果在线状态显示在线则通过 WebSocket 推送；如果离线则不做任何操作——他们将在重连时拉取。发件人的其他设备也会收到推送。一旦日志写入提交，响应就会返回给发件人。

面试官： 收件人设备如何知道有新消息？

候选人： 两条路径。在线：服务端通过 WebSocket 推送包含消息（或元数据）的内容。设备渲染后确认。离线：重连后，设备发送其游标——每个对话的最后投递 seq——并拉取此后的所有内容。推送通道只是低延迟在线投递的优化手段；对话日志加游标才是持久机制。

面试官： 多设备——用户同时使用手机和网页端。

候选人： 每台设备在每个对话上有各自的游标。当消息到达时，它会独立推送到两台在线设备。如果用户在手机上已读，手机会确认 read_seq；网页端游标的 read_seq 是独立的。从任意设备发送都会以该用户作为发件人写入同一个对话日志；所有收件人设备，包括发件人的其他设备，都能看到该消息。

面试官： 如何处理跨设备的排序？

候选人： 服务端分配每个对话的 seq。所有设备看到相同的 seq。客户端缓冲并按 seq 顺序应用。发送时的时钟偏差无关紧要，因为服务端在写入时分配 seq 和时间戳。跨对话的顺序不作保证；对用户体验无影响。

面试官： 拥有 1000 名成员的群组。

候选人： 日志结构相同。每次发送只写入一次；对话日志只捕获一次。每个成员的设备都在该日志上维护游标。推送扇出随在线成员数量扩展。我们在扇出前预计算在线成员列表，仅向在线设备推送；离线成员在重连时拉取。对于超大型群组，我们可能会有一个单独的推送扇出服务来进行批处理。

面试官： 在这种规模下，已读回执怎么处理？

候选人： 对于 1:1 和小型群组，存储每个收件人的回执。对于大型群组，聚合为计数器： delivered_count, read_count。一旦所有人确认，显示"47/1000 人已读"或"所有人已读"。超过某个阈值后，我们停止追踪每个收件人的详细信息，以控制存储量。

面试官： 在线状态和正在输入怎么处理？

候选人： 短暂性的，存储在类似 Redis 的带 TTL 的内存存储中。在线状态通过 WebSocket 心跳刷新。正在输入事件是短暂的（持续几秒），通过推送广播给对话参与者。最后在线时间在断开连接时更新，并受隐私控制。这些数据都不是持久的；服务重启可能导致状态丢失，这是可以接受的。

面试官： 端对端加密怎么处理？

候选人： Signal Protocol。每个用户拥有身份密钥和预密钥。X3DH 密钥协商开始一段对话；Double Ratchet 提供前向保密。每条消息使用独立密钥；服务端只存储密文。多设备：每台设备拥有自己的密钥对，发件人要么分别对每个收件人设备加密，要么对群组使用 Sender Keys。服务端仍按 conversation_id 路由并按 seq 排序，即使无法读取内容。

面试官： 多区域？

候选人： 每个对话有一个归属地区——通常是活跃参与者最多的地区，或对话创建所在的地区。写入操作路由到归属地区进行排序。日志复制到其他地区以支持快速本地读取。推送网关是区域性的；归属地区通知远程推送网关，后者再转发给本地设备。跨地区投递延迟主要是复制延迟——秒级别。

面试官： 如果对话归属地区发生故障怎么办？

候选人： 故障转移到副本地区。seq 通过共识复制得以保留。故障转移期间写操作会短暂不可用。读操作从副本继续进行。故障转移后，新地区接管排序工作。

面试官： 消息编辑和删除怎么处理？

候选人： 编辑是日志中的一个新事件： MessageEdited(message_id, new_content, edited_at)。客户端收到后更新其 UI。删除类似： MessageDeleted(message_id, deleted_for_all)。日志中的原始消息被标记或替换为墓碑记录。客户端遵从最新状态。

面试官： 总结。

候选人： 每个对话的仅追加日志是真实来源；每台设备的游标追踪投递进度。推送是在线投递的优化手段；从游标拉取是回退方案。多设备同步是在同一日志上使用每台设备的游标。群组是一个日志，多个读者。发送通过 client_message_id 实现幂等。通过服务端分配的 seq 实现每个对话的强排序。在线状态和正在输入是短暂性的。多地区使用每个对话的归属地区，并支持复制读取。最难处理的场景是大型群组和多设备同步，两者都通过日志 + 游标设计来应对。

3. 练习备注

五分钟版本

  1. 澄清：实时性、多设备、群组、数据保留、加密。
  2. 不变量：每个对话有序，用户可见恰好一次，持久。
  3. 每个对话的仅追加日志；每台设备的游标。
  4. 在线时推送；离线时从游标拉取。
  5. 多设备：在同一日志上维护独立游标。
  6. 群组：一个日志，多个游标；在线推送 + 离线拉取。
  7. 回执：小型群组按设备记录，大型群组聚合为计数器。
  8. 在线状态短暂性；端对端加密通过 Signal 可选实现。

一分钟结尾陈述

Messenger 是一个每个对话的仅追加日志，带有每台设备的游标。日志是真实来源；推送是在线投递的优化手段。多设备同步只需在同一日志上维护每台设备的游标。群组是一个日志，多个读者。发送是幂等的；排序是通过服务端分配的 seq 按对话维度进行的。在线状态和正在输入是短暂性的。多地区使用每个对话的归属地区。端对端加密使用 Signal Protocol，每台设备拥有独立密钥。

最佳面试用语

  • "对话日志是真实来源；推送是一种优化手段。"
  • "在共享日志上的每台设备游标解决了多设备同步问题。"
  • "群组使读者数量成倍增加，而非存储量；基于游标的读取代价低廉。"
  • "服务端分配的每个对话 seq 在各设备间保证了一致的排序。"
  • "大型群组规模下，回执聚合为计数器；每个收件人的详细信息仅适用于 1:1 和小型群组。"
