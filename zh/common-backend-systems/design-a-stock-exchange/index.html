1. 基础知识与解题思路

本题考察点

股票交易所在严格的延迟、公平性和审计要求下撮合证券的买卖双方。难点在于 撮合引擎：一个单线程的确定性系统，必须在微秒级时间内完成交易执行并保证完美的顺序保持，同时向数千名订阅者推送行情数据，并在整个交易日内无故障运行。

其不变量如下：

接受订单、验证并在每个价格档位内以严格的时间优先级使用标准撮合算法（通常为价格-时间优先）进行匹配；每笔成交都有可从订单流推导出的确定性对手方配对、价格和数量；行情数据推送反映所有订阅者相同的状态；订单簿和成交历史可精确到微秒级审计。

在 ICT5 / L6+ 级别，面试官考察的是你能否设计确定性单线程撮合引擎、高吞吐量订单摄取及行情数据推送，处理交易前风控，并对灾难恢复和审计进行系统性思考。

60 秒面试框架

口述版本：

我会将交易所设计为三个核心系统：带有交易前风控的订单摄取、每个品种的确定性撮合引擎（为保证正确性采用单线程），以及广播订单簿更新和成交信息的行情数据推送层。关键决策包括撮合算法、单线程确定性、排序、审计日志，以及如何在数千个品种间扩展而不破坏各品种内部的确定性。

首先明确需求

功能性需求：

  • 接受限价单和市价单。
  • 按价格-时间优先撮合订单。
  • 取消和修改订单。
  • 推送 Level 1（最优买/卖价）和 Level 2（深度）行情数据。
  • 向监管机构报告成交。
  • 竞价阶段（开盘、连续交易、收盘）。
  • 熔断和停牌。

非功能性需求：

  • 从订单录入到成交：微秒级（典型值 10-100 μs）。
  • 吞吐量：峰值每秒数百万笔订单。
  • 确定性：相同的输入顺序产生逐字节相同的成交结果。
  • 高可用性：交易时间内零停机。
  • 灾难恢复：无消息丢失的故障转移。
  • 符合监管要求的审计追踪。

以下内容超出范围，除非被特别询问：

  • 清算和结算（独立系统）。
  • KYC 及账户开户。
  • 超出要求的税务/合规报告。

规模假设

领域 假设
品种 10k+ 个交易标的。
每秒订单数 全交易所峰值 1M+。
每品种 活跃品种每秒数百笔订单。
每秒成交数 数万笔。
行情数据订阅者 数千家机构。
Latency（延迟） 10-100 μs 从订单到确认。
审计保留期 多年。

瓶颈在于每品种撮合引擎加上行情数据广播延迟。

订单录入 API

通常采用 FIX 协议或私有二进制协议：

NewOrderSingle(client_order_id, symbol, side, quantity, price, ...)
CancelRequest(client_order_id, order_id)
ModifyRequest(...)

响应（执行报告）：

Ack            -- order accepted
Reject         -- with reason
Fill           -- partial or full
Cancel ack

撮合算法：价格-时间优先

对每个标的的订单簿：

  • 买单按价格降序排列，同价位按时间升序（先进先出）。
  • 卖单按价格升序排列，同价位按时间升序。

当新订单到达时：

  • 主动成交（穿透订单簿）： 从最优价格开始与对手方逐级撮合，在每个价位按先进先出原则消耗，直至订单被完全成交或无更多对手方订单可穿透。
  • 被动挂单： 在其价位挂入订单簿，并打上时间戳。

每次撮合产生一条成交记录，包含 price = price of the resting order （哪方先到；被动方获得向主动方的价格改善）。

这是标准算法；其他算法包括按比例分配（pro-rata）、价格-数量优先等。

单线程确定性

单个标的的撮合引擎是单线程的。

  • 所有输入（新订单、撤单、改单、时间滴答）均通过一个队列流入。
  • 引擎逐条按顺序处理。
  • 给定输入，输出是确定性的。

为何采用单线程？

  • 并发是确定性的敌人。两个线程在订单簿上竞争可能因时序不同而产生不同结果。
  • 多线程引入锁，反而会降低速度。
  • 单个 CPU 核每秒可处理数百万次简单操作，对单个标的已绰绰有余。

扩展方式：每个标的（或每组标的）一个撮合引擎。不同标的相互独立——可运行在不同核心/主机上。

面试用语：

单线程撮合不是限制，而是一种特性。确定性至关重要，单核单线程足以应对任意单个标的的吞吐量。

排序器

全局定序器按到达顺序为每个订单事件分配单调递增的序列号，这样做的作用：

  • 提供全交易所范围内的全序关系。
  • 支持确定性回放。
  • 是公平性的基础——序列号越小，优先级越高。

实现上通常涉及硬件（FPGA）或专用低延迟网络设备。

交易前风控

在发送到撮合引擎前进行内联检查：

  • 会员持仓限额。
  • 信用额度/抵押品。
  • 误触检测（订单数量或价格严重偏离正常范围）。
  • 熔断开关（会员主动触发的紧急停止）。
  • 限流（按会员的速率限制）。

必须非常快——微秒级。

行情数据分发

两种推送类型：

  • Level 1： 最优买价、最优卖价、最新成交价。
  • Level 2： 深度——多个价位及对应数量。
  • 全深度 / Level 3： 每一笔订单。

分发方式：

  • 采用 UDP 多播以实现低延迟和广播效率。
  • 每个数据包附带序列号；订阅方检测到缺口后通过独立的 TCP/UDP 通道请求重传。
  • "快照 + 增量"模型：订阅方定期获取完整快照，之后接收增量更新。

审计日志

每个输入事件和每个输出事件均记录序列号和微秒级时间戳。

  • 可回放：将输入流输入到一个全新的撮合引擎，可复现完全相同的输出。
  • 持久化到磁盘并复制到多个站点。
  • 监管机构可请求追溯分析。

故障转移

撮合引擎是核心，在交易期间不得宕机。

  • 热备节点与主节点保持同步运行：相同输入流，相同状态。
  • 通过有序事件日志进行复制。
  • 主节点故障时，备用节点接管且不丢失任何消息。
  • 故障转移对会员透明（通常在毫秒级完成）。

Aeron、LMAX Disruptor 等技术支撑了这一模式。

竞价阶段

  • 开盘前： 订单排队；发布订单簿失衡信息；开盘时计算开盘价。
  • 连续竞价： 标准价格-时间撮合。
  • 收盘前/收盘： 通过收盘竞价算法确定当日收盘价。
  • 停牌： 所有撮合暂停；订单按规则拒绝或排队。

熔断机制：价格剧烈波动时自动停牌。

扩展与分区

数据 分区策略
撮合引擎 按标的（或按组）
订单网关 无状态；水平扩展
交易前风控 按会员；复制状态
行情数据 基于标的的多播组
审计日志 追加只写；已复制

独立标的水平扩展；标的内部单线程。

故障模式与缓解措施

故障模式 缓解措施 监控指标

运维指标

正确性：

  • 订单因无效原因被拒（应接近零）。
  • 站点间成交不匹配（必须为零）。
  • 审计日志缺口。

容量：

  • 订单录入延迟 p99 / p999。
  • 每个标的的撮合吞吐量。
  • 行情数据发布延迟。

业务层面：

  • 按标的的交易量；市场份额。
  • 会员延迟。
  • 失败订单数。

多地域 / 灾难恢复

  • 主数据中心承担撮合。
  • 在邻近数据中心运行热备节点。
  • 第三灾备站点。
  • 严格延迟要求：备用节点必须跟上主节点。
  • 定期进行故障转移演练。

需要指出的权衡

  • 单线程速度快且正确，但限制了单标的吞吐量；通常不是问题。
  • 交易前风控增加延迟；与拦截错误订单之间存在权衡。
  • 多播 UDP 速度快但不可靠；重传通道增加了复杂性。
  • 严格的确定性要求每个细节（包括时间）都必须是确定性输入。
  • 更严格的熔断保护市场，但会触怒高频交易者。
  • 审计日志永久保留会消耗存储，但具有法律强制性。

常见陷阱与更优做法

陷阱 更优做法
多线程撮合。 按标的单线程；按标的扩展。
基于墙钟的排序。 基于定序器的单调排序。
TCP 传输行情数据。 UDP 多播 + 重传通道。
仅内存状态。 可复制的事件日志；可回放。
无交易前风控。 每笔订单内联检查。
同步监管报告。 异步 + 强审计日志。

面试官可能深入探讨的方向

深入方向 优秀回答要点
撮合算法 价格-时间优先；每价位先进先出。
单线程确定性 每个标的单线程；无并发。
故障转移 具有可回放日志的热备节点。
行情数据 UDP 多播 + 重传 + 快照。
交易前风控 内联检查；极低延迟。
竞价阶段 开收盘竞价算法。

Staff 级别回答检查清单

  • 基于定序器的全序。
  • 每个标的单线程撮合。
  • 价格-时间优先。
  • 交易前风控内联执行。
  • 带重传的多播行情数据。
  • 可回放审计日志。
  • 热备故障转移。
  • 竞价阶段与熔断机制。

面试用语

股票交易所是基于全局有序订单事件序列的确定性状态机。每个标的的撮合引擎采用单线程以保证正确性；并发是确定性的敌人。行情数据通过带重传通道的 UDP 多播分发。审计日志可精确到微秒级回放。故障转移采用与主节点同步的热备节点。

2. 模拟面试

面试官： 设计一个股票交易所。

候选人： 三大核心系统：带交易前风控的订单录入、每个标的采用价格-时间优先的单线程撮合引擎，以及多播行情数据分发。所有事件通过全局定序器进行排序以保证全序。核心不变量是确定性执行：相同的输入流精确到微秒级产生相同的成交结果。审计日志达到监管级别。

面试官： 带我过一遍订单流程。

候选人： 会员通过 FIX 协议将订单发送到网关。网关进行身份验证、解析，并转发至定序器。定序器分配下一个序列号，并以纳秒精度打上时间戳。订单流入交易前风控：检查会员持仓限额、信用额度和误触规则。通过后，路由至该标的对应的撮合引擎。撮合引擎尝试以当前最优价格与对手方撮合；每次匹配产生成交记录；剩余数量挂入订单簿。执行报告通过 drop-copy 返回给会员。

面试官： 为何采用单线程？

候选人： 确定性。多线程引擎处理相同输入流时可能因线程调度不同而产生不同输出——这与交易所公平性规则不相容。单线程 = 确定性输出 = 可回放审计 = 可恢复状态。同时，单个 CPU 核能处理充足的吞吐量：即使每个标的每秒 10 万笔订单，单线程也完全应付自如。

面试官： 故障转移如何工作？

候选人： 热备节点与主节点同步运行。定序器的输出同时复制到主节点和备用节点；两者将事件应用于各自的撮合引擎，状态应完全一致。主节点故障时，备用节点已处于相同状态，可在毫秒级内接管。会员通常通过会话管理器路由，实现无缝故障转移。审计日志是恢复基础：回放日志可复现完全相同的状态。

面试官： 行情数据呢？

候选人： UDP 多播推送——Level 1（最优买卖报价 + 最新成交），Level 2（深度），全深度。每个数据包附带序列号；订阅方检测到缺口后通过 TCP 缺口通道请求重传。定期快照使订阅方能从大缺口中恢复。多播是面向数千订阅方扩播的正确选择，因为路由器会将同一数据包分发给所有人。

面试官： 交易前风控呢？

候选人： 在撮合前内联执行。包括按会员的持仓限额、信用检查、误触阈值（订单数量或价格超出合理区间）以及熔断开关。必须非常快——微秒级。风控数据复制到所有网关/风控节点；更新持续流入。会员可通过熔断开关自主限流。

面试官： 一笔订单的价格是合理水平的 1000 倍，会怎样？

候选人： 交易前风控中的误触检测会将其拦截。该订单被以明确原因拒绝。如果规则阈值未能拦截（例如复杂的操纵行为），订单进入订单簿；由此产生的成交可能因极端价格波动而触发熔断。事后，交易所可在规定时间窗口内撤销错误成交（错误成交撤销规则）。

面试官： 竞价阶段呢？

候选人： 开盘前：订单排队；发布订单簿失衡信息；开盘价由最大化单一价格撮合量的算法确定。连续竞价：标准价格-时间撮合。收盘前：类似开盘前的收盘竞价。停牌：所有撮合暂停；订单可被拒绝或排队。熔断机制在极端价格波动时自动停牌。

面试官： 审计日志呢？

候选人： 每个输入事件（订单、撤单、改单）和每个输出事件（成交、拒绝、确认）均记录序列号和纳秒级时间戳。追加只写；跨数据中心复制。监管机构可请求历史数据；回放日志可逐字节复现整个交易日的状态。日志是唯一可信来源——即使撮合引擎的内存状态丢失也可重建。

面试官： 多区域？

候选人： 大多数交易所在单一数据中心内运营以降低延迟；多地域主要用于灾难恢复。在邻近站点运行热备节点，更远处设第三灾备站点。备用节点通过有序事件流跟上主节点。定期进行故障转移演练。通常不做跨地域双活撮合，因为延迟代价会使其失去意义。

面试官： 如何应对突然 100 倍的订单速率？

候选人： 按会员速率限制遏制单一滥用行为。对网关实施聚合流量控制和背压。交易前风控对超限情况走快速路径拒绝。若撮合引擎饱和，订单在定序器处排队，延迟增加有界。最坏情况：宣布临时停牌，发出全员撤单指令后恢复。

面试官： 总结。

候选人： 带交易前风控的订单录入、用于全局全序的定序器、每个标的采用价格-时间优先的单线程撮合、带重传的 UDP 多播行情数据、监管级审计日志。具有可回放日志的热备故障转移。竞价阶段和熔断机制应对市场压力。确定性是核心不变量；单线程撮合实现了这一点。通过增加标的数量而非增加每个标的的线程数来扩展。

3. 练习备注

五分钟版本

  1. 明确：延迟、公平性、确定性、审计。
  2. 不变量：确定性；价格-时间优先；可回放。
  3. 定序器实现全局全序。
  4. 每个标的单线程撮合。
  5. 交易前风控内联执行。
  6. UDP 多播行情数据 + 重传通道。
  7. 热备故障转移；可回放审计日志。
  8. 竞价阶段与熔断机制。

一分钟结尾陈述

股票交易所是一台确定性状态机。定序器提供全局全序；每个标的单线程撮合产生逐字节确定性的输出；价格-时间优先是标准算法。交易前风控内联执行。行情数据采用带重传通道的 UDP 多播。审计日志可回放。故障转移采用与主节点同步的热备节点。确定性至上；并发是其敌人。跨标的扩展，而非在标的内部扩展。

最佳面试用语

  • "单线程撮合是一种特性，而非局限。"
  • "Sequencer 提供全局全序；所有下游处理均为确定性的。"
  • "价格-时间优先是经典算法：最优价格优先，相同价格内按 FIFO 排序。"
  • "审计日志是真相之源；内存状态可通过回放重建。"
  • "多播 UDP 用于快速扇出；重传通道用于间隙恢复。"
