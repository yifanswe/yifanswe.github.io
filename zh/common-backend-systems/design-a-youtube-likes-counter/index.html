1. 基础知识与解题思路

本题考察点

点赞计数器听起来很简单——就是对一个数字做自增——但一旦考虑到数十亿个视频、每分钟数百万次点赞的病毒热点、双击的幂等性、取消点赞的切换、欺诈行为，以及用户期望自己的点赞立即显示而其他人可以接受数秒的延迟，问题就复杂了。

其不变量如下：

每个 (user, video) 点赞最多被记录一次且可撤销；显示的计数在有界旧性范围内收敛至 count(active likes) 对应视频的真实值；用户自己的点赞状态在点击后立即从其视角保持一致；欺诈或滥用点赞在计数公开前被过滤掉。

在 ICT5 / L6+ 级别，面试官考察的是：你能否将点赞建模为每用户状态（而不仅仅是一个计数器）、在不锁定单行的情况下处理热门视频、设计读己所写语义，以及在不影响点赞操作速度的情况下集成欺诈过滤。

60 秒面试框架

口述版本：

我会将点赞系统设计为两层：一层是记录谁点赞了哪个视频以及时间的权威性每用户点赞状态，另一层是从状态变更异步更新的每视频非规范化计数器。关键决策在于热门视频计数器分片、用户自身点赞的读己所写语义、双击的幂等性，以及计数公开前的欺诈过滤。

首先明确需求

功能性需求：

  • 用户可以为视频点赞；每位用户每个视频只能点赞一次。
  • 切换取消（取消点赞）可撤销该操作。
  • 每个视频显示总点赞数。
  • 向点赞过的用户显示"你已点赞"标识。
  • 每用户历史记录："我点赞的视频"。
  • 过滤垃圾信息、机器人和刷量行为。
  • 可选的踩功能（或作为独立功能处理；现代 YouTube 隐藏了踩数）。

非功能性需求：

  • 点赞操作的响应时间在亚秒级。
  • 计数新鲜度：普通视频秒到分钟级；病毒视频可容忍更长时间。
  • 读己所写：点赞用户立即看到自己的点赞。
  • 每用户状态的强正确性（每用户每视频只能点赞一次）。
  • 显示的计数允许近似值。
  • PB 级规模：数十亿个视频，随时间累积的万亿级点赞记录。

以下内容超出范围，除非被特别询问：

  • 点赞对推荐的影响（独立系统）。
  • "你认识的人点赞了"的展示（独立社交系统）。
  • 评论点赞（类似设计，独立规模）。

规模假设

领域 假设
视频 现有 50 亿个；5 亿以上活跃。
用户 20 亿以上。
点赞事件 每天超过 100 亿次；峰值每秒 10 万到 100 万次。
病毒视频 单个视频每秒 1 万到 10 万次点赞。
读 QPS 每个视频页面均展示点赞数；每秒数千万次。
Latency（延迟） 点赞操作亚秒级；计数新鲜度秒级。
存储 随时间累积的万亿级 (user, video) 记录。

瓶颈在于病毒视频：单个视频每秒接收 10 万次点赞会导致任何单行计数器被打满。

核心产品模型

层级 用途 需求
每用户点赞状态 记录 (user, video, action, ts)。 每用户强一致性；用户可见的真实状态。
每视频计数器 用于展示的非规范化总计数。 最终一致；近似值可接受。
欺诈过滤器 从公开计数中剔除不真实的点赞。 异步；在计数最终公开前运行。

将状态与计数器分离的三个原因：1. 计数器是热点；每用户状态天然按用户分片。2. 点赞/取消点赞是每用户状态变更；计数器必须从当前状态推导。3. 欺诈过滤在点赞被记录后、计数公开前执行。

核心组件

组件 职责 关键细节
点赞 API 接受点赞/取消点赞操作。 幂等，基于 (user_id, video_id, action).
每用户点赞数据库 事实来源：哪个用户点赞了哪个视频。 按用户分区。
事件流 点赞和取消点赞作为事件。 至少一次交付；按视频分区。
计数聚合器 从事件更新每视频计数。 定期刷新；针对热门视频的分片计数器。
计数存储 每视频点赞数。 快速 KV；多层缓存。
欺诈过滤器 对点赞评分；标记不真实行为。 ML + 规则；异步运行。
个人历史记录 用户点赞的视频列表。 按 user_id 建立索引。
读服务 返回视频计数、用户自身点赞状态，并为 UI 合并展示。 以有界旧性合并计数器与用户状态。

数据模型

每用户点赞

UserLike(
  user_id,
  video_id,
  status,                // active, removed
  liked_at,
  removed_at,
  fraud_status,          // unknown, valid, suspicious, removed
  version
)

按 user_id。读取模式：该用户是否点赞了这个视频？按 (user_id, video_id)进行点查询。列表模式：该用户点赞了哪些视频？在分区内做范围扫描。

视频计数器

VideoCounter(
  video_id,
  count,                 // approximate, derived from valid likes
  raw_count,             // total likes including fraud-pending
  last_updated,
  version
)

热门视频使用分片子计数器： VideoCounter(video_id, shard_id, count)；展示值为各子计数器之和。

点赞事件

LikeEvent(
  event_id,              // deterministic: hash(user_id, video_id, action, ts_window)
  user_id,
  video_id,
  action,                // like, unlike
  timestamp,
  source                 // client, server-side replay
)

欺诈评分

FraudScore(
  user_id,
  video_id,
  score,
  reasons[],
  decided_at
)

API

POST /videos/{video_id}/like
Headers:
  Idempotency-Key (optional; client-generated)

GET  /users/{user_id}/liked-videos?cursor=

写路径：点赞操作

  1. 客户端 POST /videos/{video_id}/like 附带操作。
  2. API 认证用户；进行校验。
  3. 对 UserLike(user_id, video_id)：- 若无记录：以 status=active插入。- 若 status=active 且操作为 like：空操作（幂等）。- 若 status=active 且操作为 unlike：更新为 removed设置为约100 ms，在阻塞时向上游调用方施加背压。(2) Broker 配额（ removed_at插入。- 若 status=removed 且操作为 like：更新为 active，设置新的 liked_at.
  4. 向流发送 LikeEvent 到流。
  5. 更新用户本地缓存，使其立即看到自己的点赞。
  6. 返回成功。

延迟：每次点赞一次数据库写入，加上异步事件发送。轻松达到亚秒级。

读路径：在视频页面展示点赞数

  1. 客户端请求 /videos/{video_id}.
  2. 读服务查询：- VideoCounter(video_id) 获取计数（多层缓存；CDN、内存）。- UserLike(user_id, video_id) 判断当前用户是否已点赞（仅限已认证用户）。
  3. 合并：返回 { count, user_has_liked }.
  4. 展示。

对于未认证用户，仅获取计数。

读己所写

用户点击点赞时，UI 在服务器响应前立即展示其点赞状态：

  • 客户端在点击时乐观更新 UI。
  • API 调用确认。
  • 确认后，显示的计数可能尚未反映其点赞（因为计数聚合器是异步的）。UI 通过在本地对显示计数加 +1 来处理这一问题（当用户点赞已激活时）。

服务端，用户的 UserLike 行是强一致的：若用户刷新，立即看到 user_has_liked = true 紧接着，显示的计数在客户端侧叠加 +1，直到计数聚合器追上。

面试用语：

用户必须立即看到自己的点赞。计数可以滞后。因此，显示的计数由最终一致的计数器加上用户自身操作的客户端 +1 合成。

计数聚合

计数更新是异步的，以保持点赞操作的快速响应并吸收病毒式峰值。

方案：流处理器消费点赞事件，并更新每视频计数器。

对于普通视频：

  1. 消费事件。
  2. 在计数存储中原子地对视频计数器做自增或自减。

对于病毒（热点）视频：

  1. 按 (video_id, random_shard) 写入某个子计数器。
  2. 原子地递增该子计数器。
  3. 读路径聚合各子计数器： count = sum(sub_counter[i] for i in 0..N).

子计数器策略：

  • 检测热门视频（入站速率 > 阈值）。
  • 将其提升为使用 N 个子计数器（例如 100 个）。
  • 流量降温后将其降级。

从每用户点赞数据库定期全量重新计数，作为真实依据：

  • 后台作业按视频聚合所有 UserLike(video_id, status=active, fraud_status=valid) 行。
  • 用权威计数覆写计数器。
  • 对高流量视频每小时或每天运行一次。

欺诈过滤

检测不真实点赞的方法：

  • 刷量工厂：大量账号在短时间内点赞同一视频。
  • 机器人：由合成账号发起的点赞。
  • 激励欺诈：用户为平台外的激励而点赞。

流水线：

  1. 欺诈服务消费点赞事件。
  2. 每用户信号：账号年龄、历史欺诈记录、IP 信誉、点赞速率。
  3. 每视频信号：突发涌入、地理偏斜、粉丝重叠度。
  4. ML 评分为每个 (user, video) 产生一个 fraud_score。
  5. 超过阈值的评分将 UserLike 标记为 fraud_status = suspicious.
  6. 计数聚合器仅统计有效点赞；可疑点赞被排除在外。

效果：公开显示的计数仅反映经过验证的点赞。被欺诈标记的用户在自己的 UI 上可能仍然看到"你已点赞"，具体取决于策略（静默删除还是明确通知）。

扩展与分区

数据 分区键 原因
UserLike user_id 哈希值 用户范围查询；万亿级记录。
VideoCounter video_id 哈希值 每个视频页面一次读取。
点赞事件 video_id 哈希值 每视频计数聚合。
用户点赞视频索引 user_id 哈希值 列表查询。
欺诈评分 video_id 哈希值 每视频聚合。

对于 UserLike，"谁点赞了这个视频"的查询较为罕见（由欺诈系统和聚合使用）。若有需要，可建立按 video_id 的二级索引，但规模较大。

一致性模型

层级 一致性 原因
每用户点赞状态 强一致（读己所写） 用户体验要求如此。
计数器 最终一致 展示允许滞后。
点赞历史列表 在用户分区内最终一致 可接受的延迟。
欺诈评分 最终一致 异步 ML。

故障模式与缓解措施

故障模式 缓解措施 监控指标
客户端重复点击。 幂等，基于 (user_id, video_id, action) 加上客户端防抖。 重复点赞尝试。
计数聚合器处理滞后。 背压；自动扩容；定期全量重新统计。 聚合器延迟；计数器与真实值偏差。
热门视频将单个计数器分片压垮。 自动晋升为子计数器分片。 每个计数器的 QPS；每个视频的延迟。
因漏事件导致计数器偏差。 每日从 UserLike 进行全量重新统计。 计数器与重新统计结果之间的偏差百分比。
流丢失事件。 持久化 Kafka；幂等消费者。 消费者延迟；回放覆盖率。
欺诈过滤误报。 人工审核队列；用户申诉。 每种欺诈原因对应的撤销率。
机器人刷量在欺诈检测介入前虚增计数。 新视频的展示延迟；欺诈过滤延迟监控。 每个视频的可疑点赞率。
用户的点赞不会立即可见。 强读 UserLike；客户端 +1 覆盖层。 读己写违规。
存储空间爆炸。 基于冷热度的分级存储；归档旧的被点赞视频。 存储增长速率。

运维指标

正确性：

  • 每个视频的计数器与重新统计结果的偏差。
  • 重复的 UserLike 行（必须为零）。
  • 读己写失败（必须接近零）。

容量：

  • 按地区划分的点赞 QPS。
  • 每个视频的聚合器延迟。
  • 计数器存储 p99 延迟。

业务层面：

  • 每个视频的点赞量分布。
  • 欺诈检测率。
  • 与点赞关联的用户互动指标。

多区域策略

  • UserLike 按用户分区；用户所在主区域负责写入。
  • 点赞事件跨区域复制，供任意区域进行计数聚合。
  • 计数器存储最终复制；从本地区域读取。
  • 跨区域最终在数秒内收敛，对于展示计数是可接受的。

对于全球性病毒事件，多区域子计数器分片可分散写入负载。定期重新统计可跨区域进行对账。

需要指出的权衡

  • 在万亿规模下存储每用户点赞状态成本高昂，但这是正确支持取消点赞和欺诈移除的唯一方式。
  • 热门视频的子计数器分片增加了复杂度；定期重新统计保证其准确性。
  • 异步计数器更新以牺牲时效性换取性能；客户端 +1 覆盖层掩盖了延迟。
  • 激进的欺诈过滤可能隐藏合法点赞；申诉机制仍然存在。
  • 强一致的用户状态按分区成本低廉；全局计数器强一致性无法扩展。
  • 支持踩功能会大致使状态规模翻倍；YouTube 出于这个原因及其他因素对外隐藏了踩数。

常见陷阱与更优做法

陷阱 更优做法
只是递增一个计数器。 每用户状态是事实来源；计数器是派生的。
同步计数器更新。 通过事件流异步更新；定期重新统计以保证正确性。
每个视频单行计数器。 热门视频的子计数器分片。
忽略欺诈。 异步 ML 评分；从公开计数中排除可疑点赞。
忘记读己写。 强读 UserLike；客户端 +1 覆盖层。
统计事件而非状态。 开/关切换使事件计数失去意义；使用当前状态。

面试官可能深入探讨的方向

深入方向 优秀回答要点
热门视频计数器 子计数器分片；自动晋升；定期重新统计。
读己写 强读 UserLike；乐观客户端覆盖层。
欺诈过滤 异步 ML；从公开计数中排除可疑点赞；撤销。
存储规模 按用户分区；冷数据分级存储；万亿级记录索引。
幂等性 条件状态转换；客户端防抖。
定期重新统计 后台任务；将计数器与权威状态进行比较。

Staff 级别回答检查清单

  • 将每用户状态与每视频计数器分离。
  • 每用户状态强一致；计数器最终一致。
  • 热门视频的子计数器分片。
  • 通过事件流异步更新计数器。
  • 定期全量重新统计以保证正确性。
  • 通过强状态读取 + 客户端覆盖层实现读己写。
  • 欺诈过滤作为异步流水线，将可疑点赞从公开计数中排除。
  • 按用户分区以支持规模扩展。

面试用语

点赞计数器是两件事：每用户状态和每视频计数。状态是事实；计数是快速投影。热门视频会压垮朴素计数器；子计数器分片加上定期重新统计保持其准确性。用户必须立即看到自己的点赞；计数可以滞后。

2. 模拟面试

面试官： 设计 YouTube 的点赞计数器。

候选人： 两层架构。第一层，每用户点赞状态——每条 (user, video) 记录对应状态 active 或 removed。这是事实来源；它是强一致的，并按用户分区。第二层，每视频计数器，从点赞事件异步更新。计数器最终一致，用于快速展示。最难处理的情况是热门视频和读己写。

面试官： 为什么不直接在点赞时递增计数器？

候选人： 两个原因。第一，点赞是可切换的——取消点赞必须回退计数，这需要知道每用户状态。如果只存储计数器，就无法判断一次点击是新点赞还是重复点击已有点赞。第二，欺诈过滤会从公开计数中移除非真实点赞；我们需要每用户记录来识别并排除它们。计数器派生自每用户状态，而非反过来。

面试官： 一个视频走红，每秒获得 100,000 个点赞。计数器会发生什么？

候选人： 单行计数器会成为瓶颈被压垮。我们自动检测热门视频（传入速率超过阈值），并将其晋升为子计数器分片：拆分为（例如）100 个子计数器。每个点赞事件哈希到一个随机子计数器进行原子递增。展示计数汇总所有子计数器。速率降低后，降级并合并。

面试官： 如何防止计数器偏差？

候选人： 定期重新统计任务读取每用户状态。对每个视频，统计 UserLike 的行，其中 status=active, fraud_status=valid。覆盖写入计数器。高流量视频每小时运行一次，其他视频每天运行一次。这是真实数据。事件驱动的异步计数器是近似值；重新统计保持其准确性。

面试官： 用户点击点赞后，应立即看到自己的点赞。如何实现？

候选人： 两部分。服务端：UserLike 状态在点赞请求中同步写入，强一致。因此用户刷新后， user_has_liked 字段是正确的。客户端：UI 在点击时乐观更新 count = count + 1 。 you_liked = true 。这填补了同步状态写入与异步计数器更新之间的空白。即使全局计数器需要数秒才能追上，用户也能感知到即时反馈。

面试官： 双击——如何处理幂等性？

候选人： 点赞操作是条件状态转换：如果状态已经是 active 且操作是 like，则为空操作。DB 写入不会改变任何内容。如果客户端在一秒内发送三次点击，只有第一次有效；其余的在服务端是空操作。客户端也在本地进行防抖。

面试官： 欺诈——如何检测和应用？

候选人： 异步欺诈流水线消费事件流。它使用 ML 对账户年龄、IP 声誉、地理异常、与视频创作者的粉丝重叠度、每账户点赞速率等信号对点赞进行评分。评分超过阈值的点赞会标记 UserLike 行的 fraud_status ：将 suspicious。计数聚合器和定期重新统计只统计 fraud_status=valid 的点赞。因此公开计数仅反映经过验证的点赞。放置点赞的用户仍可在其历史记录中看到自己的操作。

面试官： 存储——万亿级 UserLike 行。

候选人： 按 user_id分区。自然查询模式有：(a) "用户 X 是否点赞了视频 Y"——用户分区内的点查询；(b) "用户 X 点赞了哪些视频"——用户分区内的范围扫描。两者都很高效。昂贵的查询——"谁点赞了这个视频"——在欺诈分析和定期重新统计期间发生，我们可以容忍大扫描。如有需要，可为热门视频维护按 video_id 建立的二级索引，但这不是默认行为。

面试官： 多区域？

候选人： 用户的主区域拥有其 UserLike。点赞事件跨区域复制，用于计数聚合（可在任意区域进行）。计数器存储最终复制；读取来自本地区域。对于全球病毒事件，子计数器分片自然跨区域——每个区域更新其本地子计数器，聚合时对其求和。

面试官： 计数有多新鲜？

候选人： 普通视频：异步聚合器产生数秒延迟。热门视频：取决于聚合器吞吐量和子计数器分片数量。定期重新统计在数分钟到数小时内追上实际值。从用户体验角度，用户通过客户端覆盖层立即看到自己的点赞；展示的总数对其他所有人可能有小幅延迟，这对数十亿用户来说无法察觉，对于病毒式飙升也是可接受的。

面试官： 如果用户被影子封禁怎么办？他们的点赞不应被计入。

候选人： 他们的 UserLike 行有 fraud_status=suspicious （或单独的标志位）。计数聚合器和重新统计将其排除。从用户角度，他们的点赞会显示（他们看到"您已点赞"）；但在世界范围内，计数不会改变。这是由策略驱动的。

面试官： 总结。

候选人： 每用户状态是事实来源；每视频计数器是派生投影。状态是强一致的，按用户分区；计数器是最终一致的，为热门视频分片。通过事件流异步聚合；定期重新统计保证正确性。通过强状态加客户端覆盖层实现读己写。异步欺诈流水线将可疑点赞从公开计数中排除。最难处理的情况是病毒式热门视频，通过子计数器分片加定期重新统计来处理。

3. 练习备注

五分钟版本

  1. 澄清：可切换、每用户唯一、病毒式热门视频、欺诈。
  2. 不变量：每用户状态权威；计数器派生。
  3. 每用户状态 + 每视频计数器分离。
  4. 热门视频的子计数器分片。
  5. 通过事件流异步更新计数器。
  6. 定期重新统计以保证正确性。
  7. 通过强状态 + 客户端覆盖层实现读己写。
  8. 异步欺诈过滤；从公开计数中排除可疑点赞。

一分钟结尾陈述

点赞计数器是每用户状态加上一个派生计数器。状态是事实，强一致且按用户分区；计数器是一个快速的最终一致投影。热门视频使用子计数器分片；定期重新统计保持计数器准确。用户通过强状态读取加客户端覆盖层立即看到自己的点赞；展示的总数在数秒内追上。欺诈过滤是异步的，将可疑点赞从公开计数中排除。

最佳面试用语

  • "每用户状态是事实；计数器是允许滞后的快速投影。"
  • "热门视频会压垮单行计数器；子计数器分片加定期重新统计来处理它们。"
  • "读己写通过强状态读取加客户端覆盖层来解决。"
  • "可切换意味着我们不能只统计事件；我们统计当前状态。"
  • "欺诈过滤将非真实点赞从公开计数中排除，而不向用户透露该操作。"
