Dropbox — 跨设备文件同步（详解）

2026/06/01 · Common Backend Systems
目录
目录

难度： Easy · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建 Dropbox：一款桌面客户端与网页应用，用户将文件拖入某台设备的文件夹后，文件将在数秒内出现在其所有其他设备上。5 亿用户，平均每位用户拥有 100 GB 数据，文件大小从 10 KB 的文本笔记到 5 GB 的视频不等。

1. 需求（约 5 分钟）

功能需求 - 从任意设备上传文件并持久保存。 - 从任意其他设备查看并下载用户所拥有的所有文件。 - 自动同步变更——在本地修改后，修改内容自动传播。

我会推迟实现共享、评论和选择性同步；提及它们以便面试官知道这些功能已被纳入范围但暂不实现。

非功能需求 - 持久性是核心要求——Dropbox 丢失文件将彻底失去用户信任。文件字节需达到 11 个 9 的持久性。 - 同步延迟目标：小文件应在秒级内完成，而非分钟级。 - 带宽效率——绝不重新上传服务器已有的字节。 - 支持最大约 50 GB 的文件；分块是硬性要求。 - 元数据的强读后写一致性，确保刚上传的文件立即在所有设备上显示。

容量估算 - 5 亿用户 × 100 GB = 50 EB。热数据层远小于此——大多数字节是冷数据。 - 平均文件约 1 MB，以 4 MB 为单位分块；生命周期内共 1000 亿个分块。 - 元数据量大但可管理；字节量占主导。若分块元数据约 100 B × 1000 亿分块 × 3 副本，约为 30 TB。50 EB 对象存储是主要成本中心。

2. 核心实体（约 2 分钟）

  • User — 拥有文件和设备。
  • File — file_id, owner_id, path, size, version, content_hash.
  • Chunk — chunk_id （= 内容哈希）， size, storage_url。全局去重。
  • 设备 — device_id，上次同步游标。
  • 活动 — 每用户文件变更的追加日志，驱动同步。

3. API（约 5 分钟）

POST  /v1/upload/initiate       { path, size, chunk_hashes[] } → { upload_id, missing_chunks[] }
PUT   /v1/upload/{upload_id}/chunk/{idx}  (binary)             → { ok }
POST  /v1/upload/{upload_id}/commit                            → { file_id, version }
GET   /v1/files?cursor=...      → { changes[], next_cursor }   // long-poll or WebSocket
GET   /v1/files/{file_id}/download/{idx}                        → 302 to chunk URL

两阶段上传（发起 → 分块 PUT → 提交）使服务器能够告知客户端哪些分块已存在，从而客户端只上传新字节。

同步游标由服务端颁发且对客户端不透明： {user_id, event_seq, device_id}。服务器从不信任客户端提供的路径差异；客户端从游标位置重放每用户事件日志。

4. 数据流（约 5 分钟）

上传：客户端对文件分块 → 对每个分块计算哈希 → 调用 initiate 并附带哈希列表 → 服务器回复缺失的分块 → 客户端仅上传那些分块 → 调用 commit → 服务器写入新版本文件并向所有者的事件日志中追加一条事件。

下载/同步：每个客户端持有用户事件日志的游标。长轮询连接在新事件到达时唤醒；客户端拉取文件元数据，查看本地缺失哪些分块，然后仅下载那些分块。

5. 高层设计（约 10–15 分钟）

  1. 客户端 在本地完成所有分块和哈希计算（块级去重，约 4 MB 分块）。
  2. API 网关 / 元数据服务 负责认证、上传协调以及事件流。
  3. 元数据存储（MySQL，按 user_id 分片） 存储文件、版本和分块 ID 列表。按用户分片使一个用户的数据保持共置。
  4. 分块存储（对象存储，如 S3） 存储以内容哈希为键的原始分块。11 个 9 持久性，跨地域复制。
  5. 事件日志（每用户） 存储在 Kafka 或分片的 MySQL 表中；同步客户端从各自的游标读取。
  6. 通知服务 持有到客户端的长轮询/WebSocket 连接，并推送"有新事件"通知；客户端随后重新拉取数据。
  7. CDN 位于分块存储前端，使热门分块（在 Dropbox 中较少见，但共享文件中确实存在）不总是打到源站。

端到端叙述：Alice 在笔记本上编辑一个文档 → 笔记本计算新分块哈希 → 发起请求显示服务器已有 95% 的分块 → 笔记本上传 5% 的增量 → 提交写入第 7 版本 → 事件落入 Alice 的日志 → Alice 的手机在长轮询中被唤醒，拉取版本 7 元数据，下载同样 5% 的增量。手机现已同步。

具体存储方案：

存储 键模式 值 / 数据结构 条目大小 TTL / 保留期 分片键
文件元数据 (user_id, path) 。 file_id 文件行，当前版本 ~500 B ， user_id
版本表 (file_id, version) 分块清单列表 ~100 B + 32 B/分块 永久保留/压缩旧版本 file_id
分块索引 chunk_id = sha256(bytes) 引用计数、大小、存储类 ~100 B 当引用计数 >0 时保留 chunk_id
对象存储 chunks/{hash_prefix}/{chunk_id} 加密分块字节 目标 4 MB 永久保留/生命周期分层 chunk_id
事件日志 (user_id, seq) 文件变更事件 约 200 B 热点数据保留 30-90 天 user_id
上传会话 upload:{upload_id} 缺失分块位图、路径、所有者 ~1 KB 24 小时 upload_id

上传路径，涉及的存储层：

  1. 客户端在本地对分块计算哈希，然后调用 upload/initiate 与 chunk_hashes[].
  2. 元数据服务按 chunk_id 并行检查分块索引，并创建 upload:{upload_id} 仅列出缺失分块。
  3. 客户端使用签名 URL 将缺失分块直接上传到对象存储；每个 PUT 在以下情况下具有幂等性 (upload_id, idx).
  4. 提交事务写入文件元数据、版本清单、分块引用计数自增以及下一个 (user_id, seq) 事件。
  5. 通知服务唤醒各设备；每台设备读取 /files?cursor=seq，然后仅下载本地磁盘中不存在的分块。

同步/下载路径，热数据与冷数据对比：

 HOT DEVICE — cursor inside event-log retention

 Device ─▶ GET /v1/files?cursor={user_id, seq=900, device_id}
          │
          ▼
 Metadata service reads EventLog[(user_id, seq > 900)]
          │
          ├──▶ batch-read File metadata / Version manifests by file_id
          ├──▶ create signed chunk URLs for chunk_id misses
          └──▶ return changes[] + next_cursor={seq=917}

 COLD DEVICE — cursor expired or brand-new device

 Device ─▶ GET /v1/files?cursor=expired
          │
          ▼
 Server returns cursor_expired + snapshot_cursor={user_id, path_after}
          │
          ▼
 Device paginates MetadataDB[(user_id, path)] in path order,
 receives current manifests, downloads missing chunk_ids, then switches
 back to EventLog from the snapshot high-water mark.

按存储层估算容量：对象字节在逻辑上占主导地位，达 50 EB；即使使用纠删码约 1.4 倍，存储和出口带宽仍是主要成本。以 4 MB 目标分块计算，逻辑分块引用的上限为 12.5 万亿个；32 B 的清单引用约为 400 TB 原始大小，100 B 的分块索引行约为 1.25 PB 原始大小（不含复制和去重收益）。若用户平均文件确实很小，文件行可达 PB 级别，因此旧版本和已删除的元数据需要压缩/生命周期策略。事件日志较小：5 亿用户 × 每天 5 次元数据变更 × 200 B × RF=3 约为每天 1.5 TB，或 90 天热窗口约 135 TB。上传会话相比之下很小：1000 万并发会话 × 1 KB，TTL 24 小时约为 10 GB。

6. 深度探讨（约 10 分钟）

分块与去重。 固定大小 4 MB 分块实现更简单，但对任意位置的插入支持较差。内容定义分块（Rabin 指纹）能够应对文件中间的插入，但客户端计算成本更高。面试中我会默认选用固定分块，并指出 CDC 是文档原地编辑场景的生产级答案。

为什么选择对象存储 + 分片 MySQL。 字节存储需要持久性、生命周期分层和廉价的大容量；S3/GCS 风格的对象存储优于 HDFS，因为客户端需要签名对象 URL 和跨地域持久性，而无需自己运维存储节点。元数据需要针对单个用户的路径重命名/版本/事件追加的事务，因此按 user_id 分片的 MySQL 是诚实的选择。如果你能接受较弱的多行事务或添加应用层事务日志，Cassandra/DynamoDB 也可用于元数据。

准备好回答"为什么不把元数据也放进对象存储？"答案是：路径重命名、版本冲突检测和事件追加都是以 user_id为键的小型事务性写入；对象存储非常适合不可变分块，但对有序的每用户元数据却很笨拙。

冲突解决。 两台设备离线编辑同一个文件。服务器保留两个版本： file.txt 。 file (Bob's conflicted copy).txt。元数据采用最后写入者胜出策略，但绝不静默丢弃字节。每设备的向量时钟可以让我们精确检测冲突。

同步协议。 带游标的长轮询是简单答案。对活跃用户，升级为 WebSocket。每用户事件日志意味着客户端离线一周后重新上线，只需从游标位置重放所有事件——无需差异计算。

同一用户状态的两种视图。 路径树 (user_id, path) 是当前视图，针对"展示我的 Dropbox"进行了优化。事件日志 (user_id, seq) 是变更视图，针对"告诉这台设备自序列号 900 以来发生了什么变化"进行了优化。它们不是竞争的真实来源：提交操作在一次元数据事务中同时写入两者，而快照回退通过高水位线将它们拼接在一起。

同步降级状态：

阶段 模式 数据源 Latency（延迟）
active WebSocket 通知 + 游标重放 事件日志热分区 秒级
dormant 游标早于热日志 元数据树快照 + 近期事件 大型目录树需数十秒
exhausted 本地状态未知/损坏 按...进行完整清单遍历 (user_id, path) 需数分钟，可恢复
冷数据字节 分块在归档层 发出恢复请求后延迟下载 需数分钟至数小时

游标携带 event_seq；若落后于保留窗口，服务器返回 cursor_expired ，客户端切换为快照同步而非静默丢失变更。

热数据与冷数据存储。 90% 的字节在上传后从未被读取。对分块进行分层：近期及频繁访问的使用热层（S3 标准），90 天未触及的使用冷层（Glacier 风格）。元数据记录每个分块的存储位置；首次读取时进行数据恢复。

容量估算：50 EB 逻辑字节在全局去重后可减少，但端到端加密或每用户密钥会降低跨用户去重效果。以 4 MB 分块计算，50 EB 约有 12.5 万亿个逻辑分块引用；清单/引用元数据为数十至数百 TB，而对象字节则达到 EB 级别。主要成本是对象存储和出口带宽，而非元数据库。

大文件上传。 可断点续传：每个分块 PUT 在 (upload_id, idx)上是幂等的。客户端可重试任何失败的分块。 commit 是原子提交点——部分上传会在 TTL 到期后被垃圾回收。

边缘压力测试。 大型文件夹的重命名风暴是元数据密集型操作，因此服务器将事件批量合并在一个父操作 ID 下，让设备按事件日志中的顺序依次应用。来自广泛共享文件的热门分块通过 CDN 和对象存储区间缓存提供服务；分块索引仍以 chunk_id为键，不会变成每用户热点键。来自单个用户的突发上传通过每用户提交串行化来限制：分块可并行上传，但元数据提交按 (user_id, seq)串行化。多地域漂移通过为每用户分配一个元数据归属地域并全局复制分块来处理；故障转移时，新地域从最后一个已复制的事件序列号恢复，并以旧元数据 epoch 拒绝过期提交。全新设备不会重放多年的事件；它从路径快照开始，之后才跟随实时日志。

边缘案例的面试要点： 字节可以并行处理且是内容寻址的，但元数据必须按用户有序处理。这条原则在设备离线、地域故障转移或批量文件重命名时保持同步的可预测性。

端到端加密（可选）。 客户端在上传前使用每文件密钥对分块加密，该密钥由用户主密钥封装。服务器只存储密文。这会失去跨用户的服务端去重能力——需明确指出这是显式的权衡取舍。

我会推迟的内容。 共享与权限、同一文件的实时协作编辑、移动端带宽优化、共享链接的滥用检测。

面试核心要点： Dropbox 是由游标连接起来的两个系统：不可变的内容寻址字节，以及强一致性的每用户元数据/事件。同步保持简单，是因为设备重放服务器有序事件，而非猜测差异。
