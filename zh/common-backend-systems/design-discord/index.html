1. 基础知识与解题思路

本题考察点

Discord 是以服务器、频道和语音房间为组织形式的实时聊天应用。难点在于其组合性：带有丰富历史记录的持久化文字频道（类似 Slack/IRC）、实时在线状态与语音（类似 Skype），以及每个服务器数百万并发连接的规模。

其不变量如下：

每条发送的消息都被持久化存储在其频道中，投递给频道所有在线成员，并可在频道保留期内检索；在线状态反映当前谁在线以及在哪个语音频道；语音和视频流量通过点对点或媒体服务器以低延迟路由；权限在频道和角色级别强制执行。

在 ICT5 / L6+ 级别，面试官测试的是：你能否设计持久化的基于频道的聊天、大规模实时在线状态、通过 WebRTC + 媒体服务器（SFU）的语音/视频路由、基于角色建模的权限，以及针对社区规模（单个服务器可能有数百万成员）的推理能力。

60 秒面试框架

口述版本：

我会将 Discord 设计为一个聊天基础设施（基于频道的持久化消息 + WebSocket 推送）、一个在线状态/语音控制平面（谁在线、谁在哪个语音频道）以及一个媒体路由层（基于 SFU 的语音/视频）。服务器（社区）是具有基于角色权限的频道组合。核心设计决策包括：频道消息存储、向在线成员实时扇出、语音房间路由，以及处理超大规模服务器（100 万+成员）。

首先明确需求

功能性需求：

  • 用户加入带有频道的服务器（社区）。
  • 文字频道：持久化消息历史记录。
  • 语音频道：实时音频 + 屏幕共享 + 视频。
  • 用户之间的私信。
  • 在线状态（在线、空闲、请勿打扰、离线）。
  • 表情回应、附件、话题串。
  • 每个频道的角色与权限。
  • 被提及时推送通知。

非功能性需求：

  • 实时消息投递（<200 毫秒）。
  • 语音延迟低于 150 毫秒。
  • 每个服务器高可用。
  • 多地区。
  • 支持超大规模服务器（100 万+成员）。

以下内容超出范围，除非被特别询问：

  • 机器人平台与 API。
  • Nitro / 付费功能。
  • 舞台频道（类似语音频道）。

规模假设

领域 假设
用户 数亿用户；数千万 DAU。
服务器 数百万个；部分服务器拥有数百万成员。
频道 数千万。
并发语音 数百万。
每日消息数 数百亿条。
Latency（延迟） 文字消息亚秒级；语音 RTT 低于 200 毫秒。

瓶颈在于向大型频道的扇出以及在成员频繁变动下的语音房间路由。

聊天路径（文字）

发送：

  1. 客户端 → WebSocket → 网关 → 聊天服务。
  2. 聊天服务进行身份验证，检查频道权限。
  3. 分配 snowflake message_id（按时间排序）。
  4. 持久化到按以下字段分区的消息存储 (channel_id, time_bucket).
  5. 发布 MESSAGE_CREATE 事件。
  6. 网关将消息扇出至所有已订阅的客户端（频道的在线成员）。
  7. 通知服务在被提及时触发。

接收：

  • 客户端订阅其已打开或所属的频道。
  • 通过 WebSocket 接收事件。
  • 通过 REST 读取消息历史（GET /channels/{id}/messages?before=).

消息排序：

  • Snowflake message_id 在每个频道内单调递增（或在嵌入时间戳的每个分片内单调递增）。
  • 客户端使用 message_id 对显示内容排序。

频道扇出

Discord 的一个显著设计选择是 仅扇出给在线订阅者，而不为每个用户维护离线邮箱。

  • 每个网关跟踪每条连接订阅了哪些频道。
  • 收到消息时：网关确定哪些连接已订阅，并向其推送。
  • 离线用户会错过实时推送；他们重新上线后通过 REST 拉取消息。

对于超大频道（成员数超过 100 万），同一时刻在线的成员只占少数。扇出的范围受限于在线订阅者，而非总成员数。

Discord 使用 Elixir/Erlang 的进程模型管理每个网关节点上的数百万条连接，并为每个服务器维护一个"guild 进程"来处理路由。

面试用语：

Discord 的扇出仅面向在线订阅者，而非每用户邮箱。这使系统能够支撑拥有数百万成员的服务器，因为大多数成员在任意时刻都处于离线状态。

大规模在线状态

在线状态是短暂的——谁在线、他们在做什么。

挑战：对于一个拥有 100 万成员的服务器，即便只是向所有成员发送"用户 X 现在上线"这条消息也是沉重的负担。

优化：

  • 在线状态更新会被批量处理。
  • 对于超大型服务器，只有同一频道中的成员或最近活跃的成员才会收到在线状态更新。
  • "懒加载在线状态"：在线用户列表按需获取；不主动推送给所有人。

存储：内存存储，带 TTL；不保证持久性。

语音/视频架构

语音和视频通过 UDP 上的 WebRTC 传输。

SFU（选择性转发单元）：

  • 每位参与者向 SFU 发送一路音视频流。
  • SFU 将每位参与者的流转发给所有其他参与者。
  • 每位参与者接收 N-1 路流（在本地解码）。
  • 对于中等规模房间，相比 MCU（服务端混流）代价更低。

流程：

  1. 用户通过语音控制服务加入语音频道。
  2. 语音控制服务分配距用户较近的媒体服务器（SFU）。
  3. 用户通过 WebRTC 连接 SFU；进行 ICE 协商和 DTLS 握手。
  4. SFU 将用户加入该频道的媒体房间。
  5. 流在用户与 SFU 之间传输。

对于大型房间（拥有数千名听众的舞台频道）：

  • 活跃发言者向 SFU 发送流。
  • 听众接收转发的流。
  • SFU 可能会进行转码以降低带宽消耗。

多地区：

  • 在多个地区部署 SFU。
  • 跨地区 SFU 间互联，支持全球语音房间。
  • 语音控制服务根据参与者所在位置选择 SFU 地区。

权限与角色

每个服务器的角色使用权限位域表示：

  • 例如， MANAGE_MESSAGES, KICK_MEMBERS, SPEAK_IN_VOICE.

按频道的权限覆盖：

  • 特定角色/用户在该频道中对权限 X 的允许或拒绝设置。

计算有效权限：

  • 以成员所有角色的服务器级基础权限进行 OR 运算。
  • 应用频道覆盖规则。

每次操作时求值（发送、删除、踢出等）。热路径在网关处缓存。

扩展与分区

数据 分区键 原因
服务器元数据 server_id 将频道、角色、成员统一存放。
成员 (server_id, user_id) 成员资格查询。
频道消息 (channel_id, time_bucket) 按时间范围扫描。
存在感知 user_id 内存存储；短暂。
语音房间 channel_id → SFU 实时路由。
角色 server_id 按服务器划分。

按服务器分片：每个服务器的数据（成员、频道、消息）存放在少量分片上。Elixir 中的"guild 进程"管理实时状态。

对于拥有数百万成员的服务器：将服务器本身拆分到多个 guild 进程；或采用特殊架构。

一致性模型

层级 一致性 原因
消息存储 通过 snowflake 排序实现按频道强一致性 需要按频道排序。
权限 编辑时强一致性 安全性。
存在感知 最终一致（内存存储，带 TTL） 临时性。
语音房间成员资格 强一致（单一事实来源） 路由正确性。
服务器元数据 强一致性 真实数据来源。

故障模式与缓解措施

故障模式 缓解措施 监控指标
网关崩溃。 客户端重连；从序列游标处恢复。 重连率。
超大服务器扇出。 仅在线推送；懒加载在线状态。 每服务器连接数。
语音媒体服务器故障。 迁移参与者；重新进行 ICE 协商。 SFU 健康状况。
存在感风暴。 批量更新；限流。 在线状态更新速率。
消息存储热分片。 按频道 + 时间分区。 分片 QPS。
权限缓存过期。 短 TTL；角色变更时主动失效。 权限撤销延迟。
语音跨地区延迟。 选择距大多数参与者最近的 SFU。 语音延迟 p99。
Snowflake 时钟偏移。 使用 NTP；限制漂移；打破平局。 Snowflake 冲突。
频道内垃圾消息。 按频道按用户限流；垃圾过滤器。 垃圾信息发送率。
100 万成员服务器突发流量。 仅在线扇出；离线基于拉取。 扇出延迟。

运维指标

正确性：

  • 消息丢失。
  • 尝试绕过权限。
  • 语音掉话率。

容量：

  • 每网关活跃连接数。
  • 每频道消息 QPS。
  • 语音房间流动率。

业务层面：

  • DAU；每频道消息数；语音分钟数。

多区域策略

  • 服务器有一个主地区，用于 guild 进程和聊天状态。
  • 成员从任意地区通过本地网关连接，本地网关将流量路由至主地区。
  • 语音 SFU 分布部署；根据参与者位置选择。
  • 跨地区复制用于容灾。

需要指出的权衡

  • 仅在线推送可扩展扇出，离线用户通过 REST 查看历史（可接受）。
  • SFU 比 MCU 代价更低，但将解码负担转移至客户端。
  • 懒加载在线状态可避免风暴，但在线状态可能有所滞后。
  • 按服务器的 guild 进程简化了状态管理，但限制了跨服务器查询。
  • 激进的权限缓存加速了操作，但在撤销权限时存在过期问题。

常见陷阱与更优做法

陷阱 更优做法
按用户维护频道消息邮箱。 仅在线扇出；REST 获取历史。
向所有成员同步广播在线状态。 批量处理 + 懒加载在线状态。
MCU 用于语音。 SFU；降低服务器负载。
单一全球网关。 按地区部署网关。
每次操作时不使用缓存重新计算权限。 缓存有效权限；变更时失效。
100 万成员服务器使用单一 guild 进程。 对服务器进行分片。

面试官可能深入探讨的方向

深入方向 优秀回答要点
大规模扇出 仅在线推送；懒加载在线状态。
语音架构 SFU、WebRTC、ICE、按地区路由。
权限 角色位域 + 频道覆盖；缓存有效权限。
超大服务器 对 guild 进程分片；非活跃用户基于拉取。
多区域 每服务器主地区；SFU 靠近参与者。
Erlang/Elixir 选型 每连接一个进程的模型；监督树。

Staff 级别回答检查清单

  • 聊天持久化 + 在线状态短暂 + 语音实时，作为三个独立平面。
  • 仅对频道进行在线扇出。
  • 语音使用 SFU 并按地区路由。
  • 基于角色的权限，支持频道覆盖。
  • 使用 Snowflake ID 实现按时间排序的消息。
  • 使用类 Erlang/Elixir 风格的进程模型管理连接。
  • 多地区部署，每个服务器有主地区。

面试用语

Discord 是三个平面在一个应用中的叠加：持久化文字聊天、短暂在线状态和实时语音/视频。关键的扩展思路是仅在线扇出——消息广播至当前已订阅的连接，而非存储在每用户收件箱中。语音流量通过靠近参与者的 SFU 传输。权限基于角色并支持频道覆盖。

2. 模拟面试

面试官： 设计 Discord。

候选人： 三个平面：带追加式历史记录的持久化文字频道、短暂的在线状态/输入状态，以及通过 SFU 实现的实时语音/视频。每个用户通过 WebSocket 连接网关；服务器（社区）拥有带基于角色权限的频道。不变量是：消息持久化且按频道有序、语音低延迟、访问控制权限正确。

面试官： 带我走一遍消息发送的流程。

候选人： 客户端通过 WebSocket 发送消息。网关将消息转发至聊天服务。聊天服务进行身份验证，检查频道权限，分配 snowflake message_id，将消息持久化到按以下字段分区的消息存储 (channel_id, time_bucket)。发布一个 MESSAGE_CREATE 事件。网关查找该频道的已订阅连接并推送事件。被提及用户的通知服务被触发。

面试官： 一个服务器有 200 万成员，扇出是如何工作的？

候选人： 仅在线扇出。系统跟踪哪些连接订阅了哪些频道——这些连接是实时推送的唯一接收方。200 万成员中，大多数在任意时刻都处于离线状态；扇出受限于在线订阅者（可能为 5 万至 10 万）。离线用户重新连接时通过 REST 拉取查看历史。Discord 使用 Elixir 的进程模型——每个服务器有一个 guild 进程负责路由——对于超大型服务器则采用分片。

面试官： 为什么不像 Messenger 那样使用每用户邮箱？

候选人： Discord 频道是发布订阅模式：频道内的所有人看到相同的消息。对大型频道，为每个用户存储一份副本会使存储爆炸式增长。频道级历史是共享的。用户需要时拉取历史；实时推送仅发送给在线订阅者。权衡之处在于：离线用户会错过实时通知上下文，但重新连接后可查看历史。

面试官： 请介绍一下语音功能的实现。

候选人： 用户加入语音频道。语音控制服务为用户分配最近的 SFU。客户端与 SFU 建立 WebRTC 连接：ICE 候选交换、DTLS 握手。SFU 将用户加入频道的媒体房间。每位参与者向 SFU 发送一路音频流；SFU 将每位参与者的流转发给其他所有参与者。每个客户端解码收到的 N-1 路流。对于大型舞台频道，只有活跃发言者发送；听众接收单路转发流。

面试官： 为什么用 SFU 而不用 MCU？

候选人： SFU 只是一个数据包转发器——成本远低于需要解码、混音和重新编码的 MCU。对于中等规模的房间（最多数十位发言者），SFU 扩展性更好。成本转移到客户端：每个客户端需要解码 N-1 路流。对于非常大的观众式房间，可以在服务器端混音以降低客户端负载，但这是设计上的权衡。

面试官： 在线用户数——100 万。

候选人： 短暂的内存状态，带有 TTL。挑战在于在线状态更新：向服务器每个成员发送"用户 X 现在在线"的消息代价高昂。缓解措施：懒加载在线状态——客户端在打开服务器时拉取在线列表，而不是将每次变更推送给所有人。需要传播的更新会被批量处理。对于在同一语音频道或正在聊天的用户，在线状态会更积极地推送。

面试官： 权限？

候选人： 服务器级别的角色与权限位字段（MANAGE_MESSAGES、KICK_MEMBERS 等）。每个成员拥有一个或多个角色；权限取 OR 合并。频道可以覆盖：「此角色不能在该语音频道中发言。」有效权限 = （角色的服务器基础权限）∪（频道允许覆盖）−（频道拒绝覆盖）。在网关处缓存以提升性能；角色/权限变更时失效。

面试官： 多区域？

候选人： 每个服务器有一个主区域——其 guild 进程和聊天状态所在的区域。任何地方的成员都通过本地网关连接，网关路由到服务器的主区域。语音 SFU 分布式部署，按参与者地理位置选择。对于全球分布的服务器，跨区域路由会增加延迟，对聊天尚可接受，但如果服务器语音需要经过远处的 SFU 则对语音有问题。

面试官： Snowflake id？

候选人： 64 位 id，由时间戳 + worker + 序列号组成。在分布式消息创建者之间提供自然的时间排序。对于按顺序显示消息至关重要，无需按挂钟时间排序（挂钟时间存在偏差）。Discord 对消息、频道、服务器、用户使用 snowflake。

面试官： 一个服务器遭遇大量垃圾消息的 DDoS 攻击。

候选人： 在网关处强制执行每频道每用户速率限制。反垃圾启发式算法（重复消息、链接）。每个服务器的自动审核工具允许管理员设置关键词过滤器、慢速模式（每 N 秒一条消息）。严重情况触发临时静音或踢出。

面试官： 总结。

候选人： 三个平面：持久化文本频道、短暂的在线状态/输入状态、通过 SFU 实现的实时语音/视频。仅在线扇出使频道扩展到数百万成员。语音通过靠近参与者的 SFU 传输。基于角色的权限支持频道覆盖，在网关处缓存以提升性能。服务器主区域；SFU 按区域部署。Erlang/Elixir 进程模型每节点处理数百万连接。最难的扩展场景是 100 万成员服务器；答案是仅在线扇出加上每服务器 guild 进程分片。

3. 练习备注

五分钟版本

  1. 明确需求：文本、语音、在线状态、服务器规模。
  2. 不变量：持久化文本、实时扇出、低延迟语音、基于角色的正确访问。
  3. 三个平面：聊天 / 在线状态 / 语音。
  4. 频道消息仅在线扇出。
  5. 语音使用 SFU 并按地区路由。
  6. 基于角色的权限，支持频道覆盖。
  7. Snowflake id 用于排序。
  8. 多区域每服务器主节点 + 区域 SFU。

一分钟结尾陈述

Discord 是持久化文本频道加短暂在线状态加实时语音/视频的结合。扩展的关键洞察是仅在线扇出——消息广播给当前连接的订阅者，而不是存储在每用户收件箱中。语音使用靠近参与者的 SFU 媒体服务器。权限基于角色并支持频道覆盖，在网关处缓存。每个服务器有聊天状态的主区域；SFU 按区域部署。Erlang/Elixir 进程模型每节点处理数百万连接。

最佳面试用语

  • 「仅在线扇出使数百万成员服务器成为可能。」
  • 「SFU 路由数据包；MCU 混合数据包——SFU 对中等规模房间更经济。」
  • 「权限是角色加每频道覆盖；缓存有效位字段。」
  • 「Snowflake id 无需按挂钟时间排序即可提供时序消息。」
  • 「Erlang 的进程模型天生为此而设计——每连接一个进程，监督树。」
