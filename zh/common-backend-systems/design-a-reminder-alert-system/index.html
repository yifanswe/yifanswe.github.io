1. 基础知识与解题思路

本题考察点

提醒告警系统在正确的时刻触发用户调度的事件，有时提前数年，通常带有循环规则和延后提醒。难点在于在确切调度时间可靠唤醒、处理时区，以及在中断后恢复时不向用户大量推送延迟提醒。

其不变量如下：

每条活跃提醒在其调度本地时间附近最多触发一次，作用于每台拥有它的用户设备；因中断遗漏的提醒要么及时触发并附带"延迟"标识，要么按策略明确标记为已错过；循环规则展开为独立处理的离散触发时机；用户的编辑和删除在数秒内传播。

在 ICT5 / L6+ 级别，面试官考察的是你能否将提醒建模为具有确定性 id 的调度触发时机（类似 cron），正确处理时区，扩展到数十亿条提醒，并与通知投递系统集成。

60 秒面试框架

口述版本：

我会将提醒系统设计为通知投递层之上的调度层。每条提醒存储为定义加上预物化到某一时间跨度的展开触发时机。基于时间桶的调度器扫描到期触发时机并将其分发到通知系统。关键决策点在于时区处理、循环规则展开、编辑/删除传播，以及从遗漏提醒中恢复。

首先明确需求

功能性需求：

  • 用户创建提醒，包含标题、备注、调度时间、循环规则、设备。
  • 循环：每日、每周、每月、自定义 RRULE。
  • 延后：推迟 N 分钟或推迟到指定时间。
  • 编辑和删除（单次触发时机或整个系列）。
  • 按设备投递：手机推送、邮件、应用内。
  • 时区感知；遵从用户当前时区。
  • 确认/关闭追踪。

非功能性需求：

  • 在调度时间的数秒内触发。
  • 扩展到数十亿条提醒。
  • 调度的高可用性。
  • 编辑在数秒内生效。
  • 调度器中断时不丢失提醒。
  • 多设备同步。

以下内容超出范围，除非被特别询问：

  • 日历事件语义（参与者、邀请、RSVP）。
  • 基于上下文的智能建议。

规模假设

领域 假设
用户 1亿至10亿。
每用户提醒数 典型 10-100 条活跃；长尾可达数千条。
循环时间跨度 预物化 30-365 天。
触发速率 峰值每分钟 10 万至 100 万条（整点效应）。
触发精度 在调度时间的数秒内。
到达设备延迟 推送投递数秒。

瓶颈与 cron 相同：整点突发和可靠唤醒。

核心产品模型

界面 用户体验 需求
创建 设置提醒；选择时间/循环规则。 同步到所有用户设备。
触发 在正确时间发送通知。 可靠，时区正确。
延后 推迟。 幂等；更新下次触发时机。
编辑/删除 修改或取消。 快速传播到调度器。
列表 查看即将触发的提醒。 跨设备同步。

核心组件

组件 职责 关键细节
Reminder API 提醒定义的 CRUD 操作。 跨用户设备同步。
Reminder store 持久化提醒及循环规则。 按用户分区。
Occurrence materializer 将循环规则展开为独立的到期时间戳。 预物化一段时间跨度。
Scheduled queue 按时间桶存储的到期触发时机列表。 类似分布式 cron 的预期运行记录。
Dispatcher 在触发时间将事件发送到通知系统。 基于 occurrence_id 保证幂等。
Snooze handler 更新下次触发时机；不改变系列。 基于 snooze_id 保证幂等。
Notification system 通过推送、邮件、应用内方式投递。 参见独立的通知系统设计。
Sync service 将提醒列表变更推送到用户设备。 使用 WebSocket/推送实现实时同步。

数据模型

Reminder Definition

Reminder(
  reminder_id,
  user_id,
  title,
  notes,
  start_time,            // first occurrence
  recurrence_rule,       // RRULE string
  timezone,
  devices: [device_id],
  status,                // active, paused, deleted
  created_at,
  updated_at,
  version
)

Occurrence

Occurrence(
  occurrence_id,         // hash(reminder_id, due_at)
  reminder_id,
  user_id,
  due_at,                // in UTC
  due_local,             // in user tz at occurrence time
  status,                // pending, fired, snoozed, dismissed, missed
  snoozed_until?,
  fired_at?,
  ack_devices: [device_id],
  version
)

主键 occurrence_id 具有确定性；按 due_at 时间桶分区以提升扫描效率。

Occurrence 物化

当提醒被创建或修改时：

  1. 计算未来时间跨度内的触发时机（如 30 或 365 天）。
  2. 对每个触发时机，生成 occurrence_id = hash(reminder_id, due_at_utc).
  3. 插入 Occurrence 表，使用 status=pending 以 if not exists 方式插入 Occurrence 表（基于 occurrence_id 幂等）。
  4. 调度后台任务定期延伸时间跨度（如每日夜间任务将每条提醒的触发时机向前延伸）。

对于非常长尾的提醒（一次性、远期），在调度时间插入单条触发时机。

面试用语：

与分布式 cron 类似，触发时机是具有确定性 id 的持久行。调度器是作用于这些行的状态机。编辑更新这些行，触发则转换状态。

调度与分发

基于时间桶的调度器（类似分布式 cron）：

  1. 每分钟（或更细粒度）划分时间桶。
  2. 调度器扫描当前桶中 due_at 且 status=pending.
  3. 条件更新 status = dispatched.
  4. 携带由 occurrence_id 派生的幂等键发送到通知系统。
  5. 收到通知系统的确认后，状态变为 fired.

若调度器短暂宕机，下次扫描将拾取遗漏的桶，若延迟显著则携带"延迟"标志进行分发。

时区处理

提醒存储时附带用户时区。"周一上午 9 点"的提醒在用户本地时区的上午 9 点触发，考虑出行和夏令时因素。

  • 提醒存储时附带 timezone 和循环规则。
  • 物化触发时机时，根据触发时刻用户的当前时区计算 due_at_utc 基于用户在触发时刻的当前时区。
  • 若用户更改时区，则从新时区重新物化未来的触发时机。
  • 边界情况：夏令时转换期间的本地时间歧义（秋季回拨当天凌晨 1:30 存在两次）——明确选取其中一个。

面试用语：

时区正确性是最容易被忽视的细节。"每周一上午 9 点"的提醒必须在本地时间 9 点触发，即使用户跨时区或夏令时发生变化。

延后

用户点击延后：

  1. Snooze API 接收 (occurrence_id, until_time).
  2. 更新触发时机： status = snoozed, snoozed_until = until_time.
  3. 创建新触发时机（或重新调度同一条），设定 due_at = until_time.
  4. 调度器在新时间拾取该触发时机。

延后不影响系列；循环规则中的未来触发时机正常触发。

编辑与删除

编辑单次触发时机：修改对应行。编辑整个系列：修改 Reminder，然后重新物化未来触发时机，已调度的触发时机可能需要取消或重新调度。删除整个系列：将 Reminder 状态设为 deleted，取消所有待处理触发时机。

编辑传播迅速，因为：

  • Reminder 变更发布一个事件。
  • 物化器在数秒内重建未来触发时机。
  • 调度器拾取新的触发时机。

多设备投递

拥有手机 + 手表 + 电脑的用户希望所有设备都能收到提醒。

  • 提醒定义列出设备（或"我的所有设备"）。
  • 分发器通过通知系统为每台设备创建一条通知。
  • 每台设备独立确认。
  • 在一台设备上关闭后，通过同步传播到其他设备。

同步机制：

  • 每台设备订阅用户的提醒变更流。
  • 当一台设备关闭/延后时，服务端广播到其他在线设备。

扩展与分区

数据 分区键 原因
Reminders user_id 哈希值 按用户的 CRUD 操作。
Occurrences due_at 时间桶 + 哈希 到期触发时机的时间范围扫描。
Snooze state occurrence_id 按 id 查找。
Device subscriptions user_id 哈希值 按用户同步。

整点突发（大量用户设置上午 9 点提醒）：基于 hash(reminder_id) 对触发时机加入数秒抖动以分散触发速率。

一致性模型

层级 一致性 原因
提醒定义 强一致性 用户期望编辑立即生效。
Occurrences 状态转换的强一致性 幂等性，不重复触发。
同步到设备 最终一致 在线设备快速更新；离线设备在重连后追补。
通知投递 至少一次 + 去重 参见通知系统设计。

多区域策略

  • 按用户归属地区部署。
  • 提醒编辑路由至主区域。
  • 发生记录在主区域物化。
  • 调度器运行在用户的主区域。
  • 故障转移：在备用区域保留热副本。

故障模式与缓解措施

故障模式 缓解措施 监控指标
调度器宕机导致漏触发。 从待处理的发生记录中回填，附带 due_at < now. 待处理逾期数量。
重试导致重复触发。 基于 occurrence_id 的幂等派发。 重复触发率。
时区变更导致漏触发。 在时区变更事件时重新物化。 时区不匹配告警。
触发前编辑未能传播。 编辑取消待处理的发生记录并重新创建。 编辑与触发的竞争计数。
用户有 1000 条提醒全部设置为上午 9 点。 按用户限制扇出速率；添加抖动。 每用户触发速率。
通知投递失败。 通知系统重试；根据系统结果标记发生记录。 投递失败率。
循环周期的时间窗口未延伸。 每晚运行时间窗口延伸任务；对陈旧提醒发出告警。 没有未来发生记录的提醒数量。
夏令时切换。 在夏令时边界重新计算发生记录；为模糊时间制定处理策略。 夏令时模糊告警。
暂停后删除。 通过状态转换取消；幂等操作。 先暂停后删除的计数。

运维指标

正确性：

  • 触发时间精度分布。
  • 漏触发的提醒（触发时间超过阈值的延迟量）。
  • 重复触发次数。
  • 时区不匹配。

容量：

  • 每分钟触发速率。
  • 调度器扫描延迟。
  • 通知系统滞后。

业务层面：

  • 每用户设置的提醒数量。
  • 关闭/暂停比率。

需要指出的权衡

  • 预物化发生记录会消耗存储，但能使调度更加健壮。
  • 较长的时间窗口（1 年）会因从未触发的发生记录浪费存储；较短的时间窗口（30 天）则需要频繁延伸。
  • 严格的触发时间精度要求调度器频繁扫描；粗粒度扫描成本更低，但会允许延迟。
  • 按设备投递提供了灵活性，但需要跨设备同步。
  • 感知时区的调度增加了复杂性，但对正确性至关重要。

常见陷阱与更优做法

陷阱 更优做法
单一全局定时器。 使用分片桶的时间分桶调度器。
仅存储循环规则。 物化发生记录；规则仅作为来源。
仅存储 UTC 时间。 存储用户时区和循环规则；按发生记录物化 UTC 时间。
触发时同步发送通知。 异步派发；通知系统负责投递。
每次编辑时重新计算。 仅重新计算未来的发生记录；过去的记录不可变。
忘记延伸时间窗口。 每晚延伸；监控没有未来发生记录的提醒。

面试官可能深入探讨的方向

深入方向 优秀回答要点
时区处理 存储时区；在发生记录时物化 UTC；时区变更时重新物化。
循环展开 RRULE 解析；时间窗口策略；延伸任务。
漏触发恢复 从待处理记录回填；标记为延迟；避免向用户发送大量通知。
多设备同步 推送至所有已注册设备；关闭操作需传播。
编辑传播 取消待处理记录并重新物化。
整点突发 抖动；预物化；调度器水平扩展。

Staff 级别回答检查清单

  • 提醒是定义；发生记录是持久的已调度事件。
  • 使用确定性 occurrence_id 的时间分桶调度器。
  • 通过存储时区和 UTC 物化保证时区正确性。
  • 编辑和删除通过取消发生记录并重新物化来传播。
  • 暂停创建新的发生记录；幂等操作。
  • 通过通知系统实现按设备投递。
  • 关闭/暂停的多设备同步。
  • 通过抖动和预物化处理整点突发。

面试用语

提醒系统是构建在通知系统之上的调度层。提醒是定义；发生记录是预物化的持久行，具有确定性 ID。时间分桶调度器在正确时刻派发它们，并处理时区问题。编辑取消并重新物化；暂停创建新的发生记录。多设备投递依托通知基础设施。

2. 模拟面试

面试官： 设计一个提醒告警系统。

候选人： 我将设计两个层：一个调度层负责管理提醒定义并预物化发生记录，以及一个通知投递层负责处理按设备推送。不变量是每个活跃提醒在每台用户设备上都在其调度的本地时间前后准时触发，不发生重复触发，并能从宕机中可靠恢复。

面试官： 为什么要物化发生记录而不是动态计算？

候选人： 有两个原因。第一，这使调度器成为基于持久行的状态机——与分布式 cron 相同。调度器扫描到期的发生记录并派发；从宕机中恢复只需"找到待处理的逾期记录"。第二，这让编辑和删除能够干净地传播：取消待处理的发生记录并重新物化。动态计算则在每次扫描时都将调度与循环解释耦合在一起。

面试官： 如何处理时区？

候选人： 将提醒与用户的时区和循环规则一起存储。在物化每个发生记录时，根据该本地时间和用户当前时区计算 UTC 时间。如果用户更改了时区，则从新时区重新物化未来的发生记录。夏令时切换在物化时处理："每天凌晨 2:30" 按策略处理回拨模糊性（选择第一个发生时间）。

面试官： 一个用户有 100 条提醒，全部设置为周一上午 9 点。如何避免 100 条推送通知在同一时刻触发？

候选人： 抖动。每个发生记录的 due_at 偏移量为 hash(reminder_id) mod jitter_window （例如 30 秒）。用户在 30 秒的时间窗口内收到 100 条提醒。按用户的通知速率限制也有帮助；如果用户有太多触发，通知系统可以对其进行摘要聚合。

面试官： 如果调度器宕机 30 分钟怎么办？

候选人： 恢复后，调度器扫描具有 due_at < now AND status = pending的发生记录，将每条标记为 delayed后进行派发。通知系统显示"原定于 X 触发"以便用户知晓。如果积压较大，我们通过速率限制和摘要聚合来避免泛洪。每条提醒的策略可以是"延迟触发"、"超过 N 分钟则跳过"或"仅触发下一个发生记录"。

面试官： 暂停功能如何实现？

候选人： 暂停使用新的 due_at = snooze_until创建一个新的发生记录（或更新现有记录），状态转换为 snoozed。调度器在新时间点处理它。原有的循环序列不受影响。

面试官： 编辑循环规则——"将每周一上午 9 点改为每周二上午 10 点"。

候选人： 更新提醒定义。取消该提醒待处理的未来发生记录。从新规则重新物化未来的发生记录。过去的发生记录不可变。物化器保证幂等性：对相同数据重新运行会产生相同的 occurrence_id；取消操作会对其进行标记。

面试官： 多设备关闭如何处理？

候选人： 当用户在手机上关闭时，手机确认该发生记录。服务器更新发生记录的状态，并通过通知系统或同步通道向用户的其他设备广播同步事件。其他设备从界面上移除该通知。如果设备处于离线状态，则在重新连接时收到关闭事件。

面试官： 规模——数十亿条提醒。

候选人： 提醒按 user_id 分区；发生记录按（due_at 时间桶 + 哈希）分区。调度器扫描各桶，使每次扫描的数据量在每个扫描周期内保持可控。拥有数千条提醒的热点用户受到按用户速率限制的约束。物化的时间窗口有界（30-365 天），以限制存储消耗。

面试官： 如果一条提醒设置为"5 年后"怎么办？

候选人： 以单条发生记录存储在调度时间处，不在现有时间窗口之外进行物化。每晚运行的时间窗口延伸任务在其进入物化窗口时处理它。在此之前，它只是提醒存储中的一行，等待调度器在时间窗口内将其认领。

面试官： 总结。

候选人： 提醒是定义；发生记录是具有确定性 ID 的持久调度行。时间分桶调度器将到期的发生记录派发至通知系统。时区随提醒一起存储，并在物化时被遵守。编辑取消并重新物化。暂停创建新的发生记录。多设备关闭跨设备同步。整点突发通过抖动处理。从宕机中恢复通过回填逾期发生记录并标记延迟标志来实现。

3. 练习备注

五分钟版本

  1. 明确：规模、循环、时区、暂停、多设备。
  2. 不变量：在调度的本地时间触发，不重复触发，可恢复。
  3. 提醒定义 + 物化的发生记录。
  4. 从 reminder_id + due_at 派生的确定性 occurrence_id。
  5. 时间分桶调度器 + 时间窗口延伸。
  6. 通过存储时区 + UTC 物化处理时区。
  7. 编辑取消并重新物化；暂停创建新的发生记录。
  8. 通过通知系统实现多设备投递；关闭时同步。

一分钟结尾陈述

提醒系统是构建在通知投递之上的调度层。提醒是定义；发生记录是预物化的持久行。时间分桶调度器在正确的 UTC 时间（由用户存储的时区推导）派发它们。编辑取消并重新物化。暂停创建新的发生记录。多设备投递依托通知系统；关闭操作跨设备同步。整点突发通过抖动处理。宕机恢复通过回填带有延迟标志的逾期发生记录来实现。

最佳面试用语

  • "发生记录是具有确定性 ID 的持久行；调度器是其上的状态机。"
  • "时区随提醒一起存储；UTC 按发生记录推导。"
  • "编辑取消待处理的发生记录并重新物化；过去的记录不可变。"
  • "暂停只是在新时间创建一个新的发生记录。"
  • "从调度器宕机中恢复是一个回填问题，而非重新调度问题。"
