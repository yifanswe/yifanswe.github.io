Notification System — Multi-Channel Delivery (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Medium · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建通知扇出服务。后端服务触发通知；通过推送（iOS/Android）、邮件、SMS 和应用内通知进行投递。10 亿用户，日均约 1 亿条通知，存在大流量突发（例如突发新闻推送在 30 秒内 5000 万次推送）。遵守用户偏好和限流规则。

1. 需求（约 5 分钟）

功能需求 - 后端服务提交通知事件（接收方、模板、载荷、渠道）。- 系统按渠道渲染消息，应用用户偏好并分发。- 跟踪投递状态（已发送 / 已送达 / 失败）用于重试和上报。

我会推迟深入讨论营销分析以及用户分群和受众构建工具。

非功能需求 - 高可用性 — 未投递的事务性通知属于真实故障。- 吞吐量：持续支撑 100M/天 = 平均约 1200/秒；吸收 50M/30秒 = 约 1.6M/秒的突发。- 各渠道延迟目标：推送 p95 < 5 秒，邮件 p95 < 30 秒，SMS p95 < 30 秒。- 重试瞬时故障；不允许无声丢弃。- 遵守用户偏好（渠道退订、免打扰时段）。

容量估算 - 10 亿用户；约 30 亿设备 token（Android/iOS/Web）。- 突发通知：30 秒内发送 5000 万条 ≈ 1.6M/秒。- 若平均每个事件扇出到 1.4 个渠道，100M 事件/天会产生约 1.4 亿条投递记录/天。以约 700 B/行 × 90 天 × 3 副本计算，投递状态约占 26 TB，是最大的热数据存储。- 设备 token 量大但变化缓慢：3B × ~200 B × 3 副本 ≈ 1.8 TB（索引前）。

2. 核心实体（约 2 分钟）

  • NotificationEvent — event_id, user_id, template_id, payload, channels[], priority.
  • Delivery — delivery_id, event_id, channel, recipient_address, status, attempts.
  • UserPreferences — user_id, channel_opt_ins, quiet_hours, device_tokens[].
  • Template — template_id, channel-specific bodies, locales.

3. API（约 5 分钟）

POST  /v1/notifications              { user_id, template_id, payload, channels[] }   → { event_id }
POST  /v1/notifications/bulk         { events[] }                                    → { batch_id }
GET   /v1/deliveries/{id}            → status

// Outgoing webhooks from providers (Apple, FCM, SES, Twilio):
POST  /v1/webhooks/provider/{name}   → delivery status update

内部调用方；幂等键为 event_id.

4. 数据流（约 5 分钟）

事件提交 → 用用户偏好进行富化 → 按渠道扇出 → 进入各渠道队列 → 各渠道 worker 池 → 调用供应商 API → 记录状态 → 失败时重试。

5. 高层设计（约 10–15 分钟）

  1. Ingestion 服务。 接收事件。校验后写入事件行（NotificationEvent 存入 Cassandra，按 event_id 分片），并发布到 router 主题。
  2. Router 服务。 流消费者，执行以下操作：- 查询用户偏好（使用缓存）。- 过滤已退订渠道。- 应用免打扰时段（根据策略延迟或丢弃）。- 将事件拆分为一条 Delivery 每个存活渠道各一条投递记录。- 将每条投递发布到对应渠道的队列。
  3. 各渠道队列。 Kafka 主题： dispatch.push, dispatch.email, dispatch.sms, dispatch.in_app。各渠道大小不同——推送队列最大。
  4. 各渠道 worker 池。 - 推送 worker： 按 token 类型分组，批量调用 APNs/FCM（均支持批量），遵守各供应商限流。- 邮件 worker： 使用邮件供应商（SES、Sendgrid），遵守每域名限流。- SMS worker： 遵守每运营商和每国家限流。- 应用内 worker： 写入用户的应用内通知 Feed（Cassandra 行），若用户已连接则通过 WS 推送。
  5. 供应商回调。 各供应商返回状态（已发送/已送达/退回等）。Webhook 处理器更新 Delivery 行。
  6. 重试策略。 各渠道：指数退避加上限。推送瞬时错误：1 小时内重试 3 次。邮件：重试 3 天。放弃后进入死信队列供运维审查。
  7. 按用户限制速率。 "不要每天给同一个人发 10 封邮件。"在 Redis 中按 (user_id, channel) 存储令牌桶。超限则丢弃或合并。
  8. 模板与渲染。 带本地化变体和载荷变量的模板。在分发时（按语言环境、按渠道）进行渲染。

端到端叙述：好友请求服务为用户 U 发出通知事件。Router 读取 U 的偏好——已启用推送 + 应用内通知，已关闭邮件；免打扰时段未激活。写入两条投递记录：push 和 in_app。推送投递路由到 APNs worker，该 worker 与其他发往同一推送网关的投递合批，调用 APNs。状态为 sent；随后 APNs webhook 返回 delivered。应用内投递写入 U 的 Feed；若 U 有打开的 WS 连接，则立即广播。

6. 深度探讨（约 10 分钟）

具体存储与分片键。

存储 键模式 数据结构 / 条目 TTL / 保留期 分片键
事件记录 notif_event:{event_id} Cassandra/Dynamo 行，载荷指针 + 模板 + 优先级，约 0.5–1 KB 保留 30 天，事务类保留 1 年 event_id
投递状态 delivery:{delivery_id} 包含 channel/status/attempts/provider_ref 的行，约 700 B 按渠道保留 30–90 天 delivery_id 或 (user_id, day) 用于用户查询
用户偏好缓存 pref:{user_id} Redis hash，包含 opt-in/免打扰时段/时区，约 300 B TTL 1 小时；写入时失效 user_id
设备 token tokens:{user_id} token 记录的集合/列表，约 200 B/token 直到供应商使其失效 user_id
应用内 Feed inapp:{user_id}:{yyyy_mm} 通知 ID + 摘要的宽行/列表，约 150 B/条 热数据保留 90 天，更早的归档 user_id
幂等性 notif_idem:{producer}:{event_id} 响应指针/状态，约 200 B 24–72 小时 (producer, event_id)
延迟投递 delay:{deliver_at_minute} 队列/ZSET，存储 delivery_ids 直到触发 deliver_at_minute
用户/渠道上限 notif_cap:{user_id}:{channel}:{yyyy_mm_dd} Redis 令牌桶，约 40 B 2 天 user_id

Kafka 队列是传输和缓冲层；Cassandra/Dynamo 行才是持久的可查询状态。我会特别指出这一点，因为面试官常常会追问队列是否意外地成了真实来源。

各存储容量估算。

存储 / 索引 Math 主要占用者
事件记录 100M/天 × 0.8 KB × 30 天 × 3 副本 ≈ 7.2 TB 存储的是载荷指针，而非渲染后的正文
投递状态 140M/天 × 700 B × 90 天 × 3 ≈ 26 TB 最大的热持久存储
Preference cache 1 亿活跃用户 × 300 B × Redis 开销/副本 ≈ 100–200 GB 实际内存 仅活跃用户；冷数据穿透到数据库
设备 token 数据库 3B token × 200 B × 3 ≈ 1.8 TB 稳定但写入量少
应用内 Feed 40M/天 × 150 B × 90 天 × 3 ≈ 1.6 TB 按用户键的索引，非完整内容
突发队列 5000 万条投递指针 × 200 B × 3 ≈ 30 GB 原始 Broker 数据 短暂缓冲，非长期状态

为什么使用 Cassandra/Dynamo 式行存储而非单一关系型集群：投递写入以追加为主，对 TTL 友好，且天然按 delivery_id 或 (user_id, day)。MySQL 在较小规模下可行，但 26 TB 热状态数据量和突发写入速率促使我倾向宽行或 Dynamo 式存储。Elasticsearch 仅作为报表索引，不应作为投递状态的真实来源。

提交通知路径（热路径与冷路径）。 1. 生产者调用 POST /v1/notifications 与 event_id；Ingestion 在分片 notif_idem:{producer}:{event_id} 上检查 (producer, event_id) ，如为重复提交则返回旧响应。2. Ingestion 写入 notif_event:{event_id} 上检查 event_id，然后将一个小指针发布到 router.normal 或 router.high ，Kafka 分区键为 user_id ，以便每用户的合并与限流保持有序。3. Router 并行获取 pref:{user_id}, tokens:{user_id}，以及 notif_cap:{user_id}:{channel}:{day} 。若偏好/token 缓存未命中，则按 user_id穿透到偏好/token 数据库，回填 Redis，并将缺失 token 标记为显式 failed_no_token 投递，而非无声丢弃推送。4. Router 过滤退订、免打扰时段和每用户上限；每个存活渠道在入队前写入一条 delivery:{delivery_id} 行，并带有辅助查询字段 (user_id, day) 供支持/调试使用。5. 渠道 worker 消费 dispatch.{channel}.{priority}。按供应商/token 类型合批，调用 APNs/FCM/SES/Twilio，并更新 delivery:{delivery_id} 中的 attempt/status/provider_ref。6. 供应商 webhook 使用 (provider_ref, provider_event_id)幂等地更新同一投递行；重试使用持久的 attempt 计数器，而非 worker 内存。7. 冷启动路径：若部署后 router 缓存为空，则降级为数据库读取偏好/token，并通过每用户本地小缓存加队列背压保护数据库。

双索引框架。 通知内容以 notif_event:{event_id}的形式写入一次。投递状态按 delivery_id 索引以支持供应商回调，按 (user_id, day) 索引以支持客服/用户历史查询。应用内 Feed 是基于同一 event/delivery ID 的另一个按用户键的视图；应存储摘要和指针，而非每条渲染消息的完整副本。

突发处理（30 秒内 5000 万条）。 三个压力点：- 摄入： 无状态；水平扩展。受网络带宽限制。- Router： 无状态；从缓存读取偏好；按需扩展。在已知活动前预热。- 供应商限流： 无法突破。APNs 和 FCM 每个应用每秒可接受数十万请求；仍需排队后逐步消耗。

对于超大突发（数秒内 5000 万条），队列负责吸收，worker 池在更长时间窗口内消耗。只要优先级流量不被阻塞，这是可接受的。

为什么选择 Kafka 而非简单的 Redis 队列或纯 SQS 方案：Kafka 提供有序分区、针对 router bug 的重放能力，以及应对 1.6M/秒突发的消费组扩展能力。SQS/Pub/Sub 适用于更简单的系统，但重放和按优先级分区控制较弱。Redis 队列内存脆弱性过高，无法在 worker 宕机期间保证突发数据不丢失。

边缘压力测试。

边缘 故障模式 降级 / 预算
不活跃用户 偏好/token 未缓存 按 user_id进行数据库读取，回填 Redis；确保提交 p95 延迟在几百毫秒以内
热门名人 / 广播 一个活动产生 5000 万行 将扇出拆分为 job；保留事务性优先队列
应用内 Feed 超过 90 天 热行窗口耗尽 游标从 inapp:{user_id}:{yyyy_mm} 切换到归档存储；图表/历史记录可以更慢
供应商上限 APNs/SMS 触发限流 在 delay:{deliver_at_minute} 中延迟，按优先级消耗
多区域漂移 重复 webhook 或另一区域的重试 幂等 delivery_id 更新；provider reference 全局唯一

优先队列。 关键事务类（密码重置、验证码）vs 营销类（每周摘要）。独立队列 + worker 防止营销高峰延迟事务通知。每个渠道至少有两个优先级。

去重。 端到端的幂等性基于 event_id 。Router 存储 (event_id, channel) -> delivery_id ，确保重复提交不会重复发送。

用户偏好缓存。 10 亿用户 × 小偏好行 = 数百 GB。在 Redis 集群中缓存热用户；冷用户穿透到 MySQL/Cassandra。偏好变更时通过缓存失效传播更新。

免打扰时段与时区。 用户时区存于偏好设置中。Router 计算"当前是否在用户的免打扰时段内？"若是，则推迟事件（写入延迟队列， deliver_at 设为免打扰时段结束时间）。将事务性事件标记为可覆盖免打扰时段。

合并通知。 好友 1 分钟内点赞 5 张照片 → 发送一条"5 个赞"通知，而非 5 条独立通知。在 router 层实现：将同一模板的事件在时间窗口内合并为一条。提升互动率，降低用户疲劳。

每供应商故障转移。 SMS（Twilio + 备选）和邮件（SES + Sendgrid）均有多个供应商。监控每供应商成功率；若某供应商降级则自动切换流量。路由配置支持热重载。

状态跟踪。 每条投递行在供应商回调时更新。状态支持发送方侧分析（"我的密码重置邮件送达了吗？"）。各渠道保留时长不同——推送状态通常只保留较短时间；事务类保留 90 天。

优雅降级状态机。

阶段 模式 数据源 延迟 / 保留时长 操作
已接受 持久化事件 notif_event:{event_id} 立即 事件在路由前已安全落盘
已推迟 免打扰时段 / 供应商上限 delay:{deliver_at_minute} 分钟到小时 稍后用相同的投递 ID 唤醒
供应商降级 队列排空 Kafka + 投递尝试次数 渠道 SLA 可能延长 保留优先队列；不阻塞事务通知
重复提交 幂等返回 notif_idem:{producer}:{event_id} < 10 ms 返回原始 event_id / 状态
永久失败 死信 delivery:{id} + DLQ 保留 30–90 天 暴露失败，不允许无声丢弃

"我希望面试官记住这句话：供应商是尽力而为的，但我的内部状态不是；每条通知要么已投递，要么已推迟，要么已明确失败。"

Token 过期。 移动端 token 在应用卸载后失效。供应商返回 Unregistered → 从数据库中使该 token 失效。定期清理器删除长期未刷新的旧 token。

多地区。 每个区域路由到本地供应商；若某区域所有供应商全部降级，则进行跨区域故障转移。供应商限流通常为全局配置，因此需通过共享计数器进行协调。

我会推迟的内容。 营销分群、模板 A/B 测试、基于机器学习的最优发送时间、内容审核、超出 locale 模板范围的国际化工具。
