1. 基础知识与解题思路

本题考察点

"YouTube 或 Netflix"作为一道综合题，是视频流领域的经典面试问题。无论选哪个，难点都相同：上传 + 转码、多级 CDN 传输、自适应码率流式传输、推荐以及 EB 级别的存储。两个系统的差异在于内容目录规模（YouTube 无边界，Netflix 精选）和内容来源（UGC vs 授权内容）。

其不变量如下：

每个视频都经过持久存储、转码为多种分辨率/编解码器以支持自适应码率流式传输，通过 CDN 以秒级以内启播的方式提供服务，并可通过搜索或推荐被发现；无论网络状况和设备如何，观众都能获得流畅的播放体验。

在 ICT5 / L6+ 级别，面试官考察的是候选人能否清晰阐述共享架构（上传、转码、CDN、ABR），分析 UGC 与精选内容目录的差异，设计推荐管道，处理爆火内容，并讨论 DRM（Netflix 特有）与开放内容（YouTube）的区别。

60 秒面试框架

口述版本：

我会将共享设计描述为四个核心系统：一个生成多分辨率输出的上传 + 转码管道、一个用于传输的多级 CDN、一个自适应码率流传输层（HLS/DASH），以及一个用于内容发现的推荐引擎。YouTube 与 Netflix 的差异在于内容目录规模（UGC vs 精选）、DRM（Netflix 严格，YouTube 可选）以及 CDN 策略（Netflix Open Connect 内嵌于 ISP，YouTube 使用 Google CDN）。值得深入探讨的设计决策是编码阶梯、CDN 分层、ABR 以及推荐系统。

功能需求

功能性需求：

  • 创作者/制片公司上传视频。
  • 异步转码为多种分辨率和编解码器。
  • 观众在 Web、移动端、电视上以自适应画质观看。
  • 从上次位置恢复播放。
  • 搜索与推荐。
  • 评论/评分（YouTube）或不支持（Netflix）。
  • 离线下载（Netflix）。

非功能性需求：

  • 秒级以内启播。
  • 自适应码率最高支持 4K HDR。
  • PB 到 EB 级别的存储。
  • 多区域分发。
  • 严格 DRM（Netflix）；可选（YouTube 付费内容）。
  • 高可用性。

API 与播放路径

核心 API：

PUT /uploads/{upload_id}/chunks/{chunk_id}
POST /videos/{video_id}/publish
GET /videos/{video_id}
GET /videos/{video_id}/manifest?profile_id=&device=&codec=
POST /drm/licenses
GET /segments/{video_id}/{rendition}/{segment_no}.m4s
PUT /profiles/{profile_id}/playheads/{video_id}

热播放路径：

  1. 客户端使用 profile_id, device_class，以及 video_id。内容目录按 video_id；权益和播放进度按 profile_id.
  2. 边缘缓存检查 manifest:{video_id}:{codec}:{device_class}:{region}。命中时30ms以内返回；未命中则穿透至清单服务。
  3. 清单服务并行读取 video:{video_id}, ladder:{video_id}:{codec}, playhead:{profile_id}:{video_id}，对于 Netflix 风格的内容，还读取 entitlement:{profile_id}:{title_id}.
  4. 若需要 DRM，客户端使用 session_id 。 device_id；许可证响应仅在当前播放会话中缓存。
  5. 播放器通过 segment:{video_id}:{rendition}:{segment_no} 从边缘 CDN 获取不可变片段对象。片段未命中时依次走 edge -> regional cache -> origin object storage。
  6. 播放进度写入在 (profile_id, video_id, updated_at_bucket) 因此重试不会倒退进度。

冷播放路径：

  1. 边缘未命中时，使用相同的片段键检查区域缓存；区域缓存未命中则从按 video_id.
  2. 源站返回不可变片段及缓存头信息。边缘为热门片段缓存7-30天，并可能为相同清晰度预取后续3-5个片段。
  3. 若源站拉取超过300ms，清单可引导播放器选择较低的初始码率；若某个清晰度缺失，播放器回退到最近的低一级阶梯条目。

跨请求持久保存的服务端状态：播放会话 id、DRM 许可证租约、最新已提交的播放进度、CDN 请求 id，以及用于滥用检测的可选近期服务片段窗口。

读路径不是一个负责传输字节的视频服务；它是一个清单服务加 CDN 键查找，只有冷未命中时才访问源站对象存储。

每片库编码阶梯

两者都为每个视频生成多个变体：分辨率：240p、480p、720p、1080p、4K。编解码器：H.264（兼容性）、HEVC、AV1。音频：多语言、多格式（立体声、5.1、Atmos）。

Netflix 按片库调优（通过内容感知码率节省带宽）；YouTube 做法类似，但针对数十亿视频也进行按视频优化。

CDN 策略

策略 YouTube Netflix
CDN Google 全球 CDN Open Connect（内嵌于 ISP）
出站流量费 规模庞大；Google 自有基础设施 ISP 本地，绕过骨干网
预置 一定程度；基于需求 激进；每晚填充
命中率 90%+ 95%以上（ISP 本地）

多级：设备缓存、边缘 POP、区域缓存、源站（极少命中）

自适应码率（ABR）

HLS 或 DASH 清单列出多码率的片段。播放器测量网络状况和缓冲区，按片段选择码率。在片段之间动态切换。无论网络状况如何均可流畅播放。

播放降级状态：

阶段 模式 数据源 延迟目标 状态转换
1 边缘命中，目标码率 边缘 POP 片段获取 <30ms 正常稳态。
2 区域命中 区域缓存 <100ms 边缘未命中或新 POP。
3 源站拉取 对象存储 首次未命中 <300ms 区域未命中；拉取后缓存片段。
4 降低码率 相同 CDN 路径 避免重缓冲 缓冲区低于阈值，或源站/边缘延迟高。
5 仅音频/错误 清单服务 快速失败 无可播放的清晰度或权益校验失败。

类似游标的状态是播放器的片段序号加清晰度选择；由于片段名称中编码了位置信息，服务器在字节传输上可以保持无状态。

DRM

Netflix：严格（Widevine、FairPlay、PlayReady）。许可证与设备和会话绑定。制片公司要求。

YouTube：可选。付费内容使用 DRM；广告/免费内容不使用。

推荐

参见 设计 Netflix 风格的推荐系统.

两阶段：每用户离线候选生成 + 在线排序。

YouTube 叠加了订阅动态和热门趋势。Netflix 大量使用 A/B 测试，并将每片库编码决策与观众建模绑定。

存储分层

热层：近期/热门派生文件，有 CDN 前置。暖层：较旧但仍有访问，使用更廉价的存储。冷层：原始文件 + 极少访问的内容；归档存储。

YouTube 因长尾内容更多，分层流动也更频繁。

具体存储及容量估算

存储 / 索引 键模式 值 容量/上限 TTL/生命周期 分片键
内容目录元数据 video:{video_id} 片库标题、所有者、状态、派生文件指针 每个视频2-4KB 永久保存；删除时打墓碑标记 video_id
编码阶梯 ladder:{video_id}:{codec} 各清晰度、片段数、码率 每种编解码器5-20KB 发布后不可变；重新编码时生成新版本 video_id
片段对象 segment:{video_id}:{rendition}:{segment_no} 4秒媒体片段 0.5-8MB 源站永久保存；边缘 TTL 7-30天 video_id
清单缓存 manifest:{video_id}:{codec}:{device_class}:{region} HLS/DASH 清单 10-100KB 边缘缓存5-30分钟 video_id

热度索引 popular:{region}:{hour} 顶级 video_ids 及评分 每地区每小时约10k个 ID，约200KB 保留48小时 region

容量估算：

  • 内容目录：50B 个视频 * 4KB = 200TB 原始数据；3副本跨2个地区，约1.2PB。数据量大，但不是主要成本。
  • 播放进度：10亿个 profile * 200个活跃片库 * 120B = 24TB 原始数据；3副本约72TB。仅在热存储中保留活跃窗口，归档旧历史记录。
  • 片段出流量是主要开销。若2000万峰值流平均码率2Mbps，峰值出流量在开销之前已达40Tbps；通过每片库编码将码率降低10%可节省数个 Tbps。
  • 片段存储占持久化字节的主要部分。一个10分钟视频按5种清晰度和平均2Mbps估算约为750MB；500亿个此类视频在副本和冷层压缩之前将达到数十 EB。

主要成本调节手段：每片库码率调优、CDN 命中率、降低冷门派生文件的边缘 TTL、冷层存储原始文件，以及按需延迟计算稀少的编解码器/清晰度组合。

多区域

两者都严重依赖全球 CDN。元数据全球多副本复制。推荐特征按地区维护。用户播放进度存储在归属主区域。

运维指标

正确性：- 转码成功。- 删除后不提供过期内容。- DRM 合规。

容量：- CDN 命中率。- 出口带宽。- 播放器重缓冲率。

业务：- 观看时长。- 每视频互动量。- 推荐 CTR。

需要指出的权衡

  • 按片名的编码阶梯相比统一阶梯可节省带宽。
  • ISP 内嵌 CDN（Netflix）减少中转流量，但需要硬件部署，而全球 CDN 则无此要求。
  • 对象存储适合不可变分片；关系型数据库会浪费查询能力，并在字节范围出口下崩溃。
  • CDN 分发优于应用服务器流式传输，因为缓存局部性和 TCP 终止是主要成本驱动因素，而非业务逻辑。
  • 预计算所有编解码器/分辨率版本可改善首次播放体验，但会成倍增加存储；对稀有分辨率进行懒惰编码可减少字节数，代价是冷启动延迟。
  • Netflix 采用严格 DRM 用于内容授权，YouTube 大部分内容则为开放模式。
  • 预置内容 vs 懒加载。
  • 精选目录（Netflix）vs UGC（YouTube）驱动不同的存储策略。

常见陷阱与更优做法

面试用语

YouTube 和 Netflix 共享视频流水线：上传、异步转码、多层 CDN、ABR 分发、推荐。差异在于目录（UGC vs 精选）、DRM（可选 vs 强制）和 CDN（全球 vs ISP 内嵌）。主要成本是出口带宽；主要正确性问题是持久存储和流畅播放。

2. 模拟面试

面试官： 设计 YouTube 或 Netflix。

候选人： 共享架构：上传 + 异步转码生成多分辨率派生版本；多层 CDN 通过自适应码率流式传输提供服务。差异：YouTube 拥有数十亿 UGC 视频，DRM 可选；Netflix 拥有数千个精选片名，DRM 严格，并使用 ISP 内嵌的 Open Connect CDN。推荐系统驱动内容发现。不变量是持久存储和在各种网络条件下的亚秒级播放启动。

面试官： 带我了解上传 + 转码流程。

候选人： 创作者/工作室通过可断点续传上传将视频提交至对象存储。异步转码器将视频拆分为分片，并并行转码为多种分辨率和编解码器（H.264、HEVC、AV1）以及音轨和 HDR 变体。按片名的码率阶梯优化带宽——一部节奏缓慢的剧情片比动作片所需更少。第一个派生版本就绪后视频即可发布，更高质量版本异步后续生成。派生版本存储作为 CDN 的数据源。

面试官： CDN 策略？

候选人： 多层。设备缓存 → 边缘 POP → 区域缓存 → 源站。命中率目标 90% 以上。Netflix 运行 Open Connect：部署在 ISP 内部的物理设备，在非高峰时段预填充内容，在高峰时段本地为用户提供服务，将流量留在 ISP 内部。YouTube 使用 Google 的全球 CDN，具有大量边缘节点和按需缓存。两者都会对热门内容进行预热。

面试官： 自适应码率。

候选人： HLS 或 DASH Manifest 列出多种码率的分片。播放器测量带宽和缓冲区，逐分片选择码率。在分片之间可向上切换（条件改善）或向下切换（条件变差）。无论如何都能保证流畅播放。服务器只负责提供分片，智能逻辑在播放器端。

面试官： DRM。

候选人： Netflix：严格，强制执行。Chrome/Android 使用 Widevine，Apple 使用 FairPlay，Windows 使用 PlayReady。内容加密；播放器获取绑定到设备 + 会话的许可证。许可证有有效期；长时间播放需要续期。YouTube：DRM 可选，用于付费内容；免费/广告支持内容为开放模式。版权方授权关系驱动了这一差异。

面试官： 推荐。

候选人： 两阶段。离线批处理（每日/每小时）通过协同过滤、内容嵌入、流行度为每位用户构建候选集。在线排序器在请求时结合实时特征和个性化对候选进行打分。YouTube 叠加订阅动态、热门内容和观看历史。Netflix 大量使用 A/B 测试和按片名的效果建模。

面试官： EB 级存储。

候选人： 分层。热门派生版本存储在带 CDN 支撑的高速对象存储中。温热内容存储在成本较低的存储中。冷门原始文件存入归档存储。派生版本有引用计数；不再使用时清理。随时间推移使用更新的编解码器（AV1）重新转码以节省出口带宽。

面试官： 一个视频突然爆红——每分钟百万次播放。

候选人： 多层 CDN 可吸收流量。根据信号（推荐曝光、社交分享）预热边缘节点。边缘节点进行请求合并。对热门视频进行互动计数器子分片。观看计数聚合器异步运行；定期重新统计。

面试官： YouTube vs Netflix 差异。

候选人： 目录：YouTube 是 UGC，数十亿视频，以长尾为主；Netflix 是精选数千片名，头部内容贡献大部分观看量。存储：YouTube 在冷尾存储上花费更多；Netflix 在顶级质量派生版本上投入更多。DRM：Netflix 严格；YouTube 可选。CDN：Netflix Open Connect 内嵌于 ISP；YouTube 使用 Google CDN。推荐：YouTube 以探索为主用于内容发现；Netflix 以挖掘为主聚焦经过验证的片名。

面试官： 断点续播 / 播放进度。

候选人： 每个用户每个视频的播放进度。播放器每 30 秒发送心跳。服务端权威存储。重新打开时，「继续观看」行读取播放进度。跨设备同步。

面试官： 互动（点赞、评论）。

候选人： YouTube 有大量互动（点赞、评论、观看计数）——参见 设计 YouTube 点赞计数器。对热门视频进行子计数器分片。定期重新统计。Netflix 的互动较轻量（大拇指点赞/踩用于个性化，不公开显示）。

面试官： 多地区。

候选人： 两者都依赖全球 CDN。派生版本跨地域复制以加快边缘填充。元数据全球复制。用户播放进度存储在归属区域。推荐特征按区域维护，跨区域共享全球热门内容。

面试官： 总结。

候选人： 共享视频流水线：上传、带按片名阶梯的异步转码、高命中率多层 CDN、通过 HLS/DASH 的 ABR 流式传输。差异：目录（UGC vs 精选）、DRM（可选 vs 严格）、CDN（全球 vs ISP 内嵌）、推荐理念。两阶段推荐。互动计数器子分片。按年龄的存储分层。通过 CDN 实现多区域。主要成本是出口带宽；两个平台都在 CDN 上大量投入。

3. 练习备注

五分钟版本

  1. 明确：目录（UGC vs 精选）、DRM、规模。
  2. 不变量：持久性；亚秒级启动；流畅 ABR。
  3. 上传 + 带按片名阶梯的异步转码。
  4. 多层 CDN（Open Connect vs 全球 Google CDN）。
  5. 通过 HLS/DASH 实现 ABR。
  6. DRM：严格（Netflix）或可选（YouTube）。
  7. 两阶段推荐。
  8. 存储分层；互动计数器（YT）子分片。

一分钟结尾陈述

YouTube 和 Netflix 共享视频流水线：上传、带按片名码率阶梯的异步转码、高命中率多层 CDN、通过 HLS/DASH 的自适应码率流式传输、两阶段推荐。差异在于目录（UGC vs 精选）、DRM（可选 vs 严格）和 CDN（全球 vs ISP 内嵌）。主要成本是出口带宽；两者都在 CDN 上大量投入。最难的正确性问题是 EB 级持久存储；最难的性能问题是亚秒级启动。

最佳面试用语

  • "共享视频流水线：上传 → 异步转码 → 多层 CDN → ABR 播放器。"
  • "按内容独立的编码码率阶梯在不损失画质的情况下节省带宽。"
  • "Open Connect（Netflix）将 CDN 部署在 ISP 内部；YouTube 使用 Google 的全球 CDN。"
  • "Netflix 采用严格 DRM，YouTube 可选——由授权驱动。"
  • "两阶段推荐：离线候选集 + 在线排序器。"
