1. 基础知识与解题思路

本题考察点

Uber Eats 是一个三方市场——食客、餐厅（商家）和配送员——建立在地理空间调度核心之上。与 Uber/Lyft不同，配送员匹配必须与 食物准备完成时间对齐：派单过早，配送员在餐厅空等；过晚，食物已变凉。此外还有目录/发现层（菜单、搜索、图片），这在网约车中是缺失的，且订单生命周期涉及三个独立参与方，必须干净地完成交接。

其不变量如下：

每个订单流经：购物车 → 已下单 → 商家已接单 → 准备中 → 已就绪 → 已取餐 → 已送达，每个订单（或每批次）最多对应一名配送员，付款在下单时完成幂等授权并在送达时完成结算，食客看到的菜单与商家当前可用状态在数秒内保持一致，且费用可从已记录事件中确定性地计算得出。

在 ICT5 / L6+ 级别，面试官考察的是：你能否将地理空间调度系统与目录/搜索系统以及多方订单状态机组合起来——并能推理出备餐时间预测、批量配送、单品库存（"86'd" 下架项），以及食品配送特有的故障模式（餐厅拒单、配送员未出现、缺少单品、食物变凉）。

60 秒面试框架

口述版本：

我会将 Uber Eats 设计为四层。第一层是目录与发现层——餐厅、菜单、按配送范围和备餐时间过滤的搜索。第二层是订单服务，负责购物车和三方状态机。第三层是配送员调度层，复用 Uber 地理索引，但增加了感知备餐时间的调度和批量配送功能。第四层是支付和定价，包括配送费和小费。关键决策包括：如何将派单时机与预测的食物就绪时间对齐、如何对附近订单进行批量处理、如何保持商家菜单与实时可用状态一致，以及如何处理三方中某一方退出的故障模式。

首先明确需求

功能性需求：

  • 食客按位置、菜系、评分、送达时间浏览餐厅。
  • 食客从一家餐厅的菜单中构建购物车；填写地址、付款信息和小费后结账。
  • 餐厅在平板/POS 上收到订单，进行接单（或拒单），并标记为已备好。
  • 系统匹配一名配送员；配送员取餐，完成配送。
  • 全程生命周期实时追踪。
  • 评分：食客对餐厅和配送员进行评分；配送员对食客进行评分。
  • 针对缺少单品、延迟送达、订单错误提供退款和积分。
  • 促销活动、拼团订单、预约订单、重新下单。

非功能性需求：

  • 目录读多；食客浏览必须在亚秒级响应。
  • 下单强一致性（不允许未接单就扣款）。
  • 配送员匹配延迟几秒；派单时机需与备餐就绪时间对齐。
  • 单品可用状态传播延迟在数秒到一分钟内。
  • 多地区；城市是天然的分片单元。
  • 订单、支付、退款的可审计性。

以下内容超出范围，除非被特别询问：

  • 杂货 / 便利商品（不同的履单形态）。
  • 餐厅 POS 硬件规格细节。
  • 地图 / 路由引擎内部实现。
  • 推荐 ML 的深入细节（仅在高层提及）。

具体存储、键、TTL 与容量

Uber Eats 看起来是一个统一应用，但热数据分为目录快照、实时覆盖层、订单、骑手状态和索引。内容存储与索引应保持分离：索引主要存储 ID 和指针；内容存储存储已填充的完整对象。

存储 键 / 分片 值 TTL / 保留期 容量估算
Restaurant DB restaurant_id；按城市分片后再哈希 餐厅档案、营业时间、配送多边形、收款账户指针。约 4 KB/行。 持久化；变更审计保留多年。 1M 家餐厅 × 4 KB × 3 副本 × 3 读取区域 ≈ 36 GB，相比订单数据很小。
Menu item DB (restaurant_id, item_id) 菜品文字、修改组、基础价格、图片指针、版本号。约 1 KB/项。 持久化；旧版本归档。 1M 家餐厅 × 200 项 × 1 KB ≈ 200 GB 主存储；含 3 副本和全球读取副本 ≈ 1-2 TB。
卡片快照缓存 card:{restaurant_id}:v{version} in Redis 去规范化的浏览卡片 JSON：名称、菜系标签、评分、图片 URL、基础 ETA 字段。1-2 KB。 24h TTL 加上目录写入时的显式失效。 1M × 2 KB × RF2 × 3 区域 ≈ 12 GB，成本低且访问极为频繁。
实时库存覆盖层 avail:{restaurant_id} Redis hash item_id -> {available, version, reason} 仅存储实时状态；不含完整菜单内容。 餐厅营业期间无 TTL；餐厅关闭后 24h 过期，并从 DB 重建。 200M 条目标记 × ~60 B 有效 Redis 开销 ≈ 每份完整副本 12 GB；RF2 ≈ 每区域 24 GB。
地理正向索引 geo:{h3_cell}:restaurants 及所选 geo:{h3_cell}:{filter} 集合 restaurant_id 或 (restaurant_id, zone_id) 指针。 在多边形/菜单/状态变更时重建；无用户端 TTL。 若 1M 个格子平均 500 个 ID × 8 B 原始大小为 4 GB；Redis/set 开销使实际大小接近每区域 20-40 GB。
Order DB order:{order_id}；按 hash(order_id)，辅助索引基于 eater_id, restaurant_id, courier_id 包含状态、菜品快照、定价快照、支付意图的订单行。4-8 KB。 近 90 天保持热存储；冷数据归档用于客服/税务。 30M 订单/天 × 90 天 × 6 KB ≈ 16 TB 主热存储；含 3 副本 ≈ 48 TB，是占主导地位的存储。
订单事件日志 (order_id, seq) 只追加 状态转换、操作者、时间戳、定价输入。300-600 B/事件。 近 90 天热存储；冷数据保留多年。 30M/天 × ~12 个事件 × 500 B ≈ 180 GB/天主库；快速归档。
骑手实时状态 courier:{courier_id} 。 couriers:{h3_cell} 按新鲜度/ETA 排序的 sorted set 位置、状态、活跃订单、最近心跳。 30-60s TTL；骑手应用每 4-10s 刷新一次。 500k 在线骑手 × 300 B × RF3 不足 500 MB；地理索引开销为主但仍在 GB 级而非 TB 级。
浏览会话状态 不透明游标编码 {request_id, cell, rank_score, tie_breaker, index_version}；可选 shown:{request_id} set 游标和去重 ID，不含已填充的卡片。 10-30 分钟。 1M 活跃浏览会话 × 2 KB ≈ 2 GB。

技术选型：订单使用关系型或全球分布式 SQL 存储，因为状态转换需要 compare-and-swap、辅助查找和支付对账。发现服务使用 Elasticsearch / Solr，因为排名需要地理 + 过滤 + 文本；单靠 Redis 不够。快照和覆盖层使用 Redis，因为数据量小、访问热且可重建。准备好回答"为什么不直接把菜单放入 Elasticsearch？"——搜索索引文档是指针和排名特征；菜单 DB 仍是唯一事实来源，卡片快照提供已填充的读取模型。

逐步热路径分析

浏览餐厅——读密集路径。

  1. 用餐者应用调用 GET /restaurants?lat=&lng=&filters=&cursor= （若已登录则携带用户身份验证）。
  2. Browse API 将经纬度映射到 H3 格子并读取 geo:{cell}:restaurants 加上相邻格子并行处理。
  3. 并行加载实时覆盖层（restaurant status, busy level，库存摘要）、用餐者特征、赞助候选项和缓存的 ETA 特征。
  4. 按"当前营业中"、配送多边形、硬性过滤条件和库存对候选 ID 进行筛选。此阶段操作的是 ID 和位集，而非完整菜单。
  5. L1 排序器对数百个 ID 打分；L2 排序器对前 ~50 个使用更丰富特征打分。
  6. 卡片构建器 MGETs card:{restaurant_id}:v{version} 从 Redis 批量获取，叠加实时状态，注入仍通过资格审查的赞助位，并返回 10-15 张卡片。
  7. 游标编码排名分数、决胜字段、请求/索引版本，以及可选的去重 token。后续分页会重新检查实时状态，但不会重新执行完整全局排名。

结账——一致性热点。

  1. 用餐者将购物车提交到 POST /orders 并附带幂等键。
  2. 订单服务在短暂的校验锁下读取菜单项版本和 avail:{restaurant_id} ；购物车价格过期将触发"确认后下单"响应。
  3. 定价服务根据确定性输入计算费用、税费、促销和小费占位符；促销核销是原子预留操作。
  4. 支付服务预授权 estimated_total 使用 order_id 作为幂等键。
  5. Order DB 事务插入 Order(state=placed)，菜品/定价快照、状态历史事件，以及发给商家适配器和调度器的 outbox 事件。
  6. 商家适配器至少一次将订单投递到平板/POS；商家确认通过以下方式实现幂等 order_id.

调度——双视图模式。

同一在途订单以两种方式建立索引：

视图 键 用途
订单自有视图 order:{order_id} 权威状态机、定价、支付、客服历史。
调度视图 dispatch:{h3_cell}:{ready_bucket} → 订单 ID 按地理位置和预计就绪时间快速匹配。
  1. 在 placed，调度器计算 food_ready_time 并将订单 ID 插入 dispatch:{pickup_cell}:{ready_bucket}.
  2. 匹配工作进程扫描 dispatch_deadline <= now的桶，读取附近的 couriers:{h3_cell} sorted set 并对候选骑手评分。
  3. 工作进程通过 CAS 操作预留骑手 courier:{courier_id} 并通过 CAS 在以下位置完成调度分配。 order:{order_id}.
  4. 若商家提前标记就绪，订单移至更早的桶；若备餐超时，出价可在取餐前被延迟或重新分配。

面试用语：

浏览是最终卡片构建前的 ID 与索引问题；结账是事务性状态机问题；调度是同一订单按格子和就绪时间桶二次建立索引的问题。

订单生命周期

三方状态机：

cart -> placed -> accepted -> preparing -> ready -> picked_up -> delivered
          |          |           |           |          |
          v          v           v           v          v
        canceled  rejected   canceled     canceled  undelivered
                  (by merchant)                     (return / refund)

状态转换与归属：

状态转换 操作者 CAS 条件
cart -> placed 用餐者（结账） 支付预授权成功。
placed -> accepted 商户 在接单 SLA 内（~2 分钟）。
placed -> rejected 商户 退款授权。
accepted -> preparing 商家（接单时自动触发） —
preparing -> ready 商户 触发骑手交接。
ready -> picked_up 配送员 在餐厅位置。
picked_up -> delivered 配送员 到达配送地址 + 用餐者确认或拍照。
any -> canceled 用餐者 / 系统 依据退款政策。

每次转换均通过 CAS 操作 (order_id, from_state)。状态历史追加记录用于审计。

                       |
                       v
                courier_offered -> courier_accepted -> en_route_pickup
                                                              |
                                                              v
                                                       at_restaurant
                                                              |
                                                       (waits for ready)
                                                              |
                                                              v
                                                        picked_up

商家轨道与骑手轨道在以下时刻汇合 picked_up.

面试用语：

订单状态和骑手状态是两个并行状态机，在取餐时汇合。核心时序难题是将骑手到达时间与食物就绪时间对齐。

目录与搜索

目录读取密集且变更缓慢；搜索受地理边界约束。

  • 餐厅和菜单数据存储在关系型存储（Postgres / Spanner）中，按 restaurant_id.
  • 菜单读取通过缓存（Redis）+ CDN（用于图片）提供服务。
  • 搜索索引（Elasticsearch）以格子 + 餐厅为键；文档包含菜系、评分、备餐时间、配送时间估计。
  • 浏览查询 = 配送半径内的地理格子 + 过滤条件。按 score(eta, rating, popularity, sponsored).
  • 库存变更（"下架三文鱼"、"餐厅暂停营业"）通过流式管道传播，在数秒内更新搜索索引并使缓存失效。

面试用语：

浏览即"附近能配送到我地址的餐厅，经过过滤和排名"。它是基于格子的 ES 查询，叠加实时库存和 ETA，并因读远多于写而进行积极缓存。

深度解析：向用餐者展示餐厅列表

这是核心读取路径。读写比约为 1000:1，峰值时达数十万 QPS，且响应必须感觉即时，同时仍能在数秒内反映实时变更。

查询重述

"哪些餐厅现在能配送到我的地址，按个性化排名，附带当前 ETA 和实时状态——p99 延迟在 300 ms 以内。"

拆解：

  1. 地理空间资格 ——用餐者地址位于各餐厅配送区域内。
  2. 当前营业 ——当前本地时间在营业时间内，且餐厅未暂停。
  3. 容量估算 ——未处于软限流/热门餐厅模式。
  4. 过滤条件 ——用餐者的菜系、饮食标签、价格档位、评分下限。
  5. 排序 ——ETA、评分、热度、个性化、赞助位、业务规则。
  6. 装饰信息 ——名称、图片 URL、评分、配送费、ETA 范围、徽章、当前状态。

为何比看起来更难

  • 地理方向与 Uber 相反。 每家餐厅都有自己的配送多边形。问题是"用餐者是否在 这家餐厅的 多边形内？"——而非"用餐者 X 公里内有什么？"自然索引方向发生翻转。
  • ETA 是动态的。 每张卡片的"32-45 分钟"取决于餐厅当前的备餐负载和该格子的路况，无法固化为每日快照。
  • 个性化。 同一地址的两位用餐者应看到不同的排序。
  • 实时库存。 餐厅暂停、营业时间变更和新上线必须在数秒内呈现出来。
  • 冷启动延迟。 应用首次打开必须感觉即时；我们无法将地理 + 过滤 + 排名 + ETA + 广告 + 装饰信息串行处理。

地理候选集：按格子正向建立索引

资格建模的两种方式：

方式 读取形态 写入形态
多边形挂在餐厅侧 （对每个候选执行点在多边形内测试） 读取较慢——扫描 + PIP 测试。 写入成本低。
格子正向索引 cell_id -> [restaurant_id] 每个 Redis SMEMBERS。快速。 写入成本更高；每次多边形变更都需更新。

我们为热路径选择正向索引；多边形是事实来源，用于重建索引。餐厅写入或多边形变更时，索引器枚举多边形覆盖的 H3 格子，更新正向索引并使受影响的用餐者格子缓存失效。

边缘细节：

  • 处于格子边界的用餐者： 查询用餐者所在格子加上紧邻的 H3 邻居并去重。
  • 多区域餐厅 （不同子区域收取不同费用）：条目为 (restaurant_id, zone_id) 因此取数时即可获知费用。
  • 分辨率选择： H3 精度 ~8-9（~100-400 m）；比调度更精细，因为用户体验需要更高精度。

卡片快照缓存

获取候选 ID 后，我们需要装饰数据。在读取路径上我们不执行 非 关联查询目录 DB。

  • 快照服务在每次目录写入时为每家餐厅物化一个去规范化的"卡片"（~1-2 KB JSON）：名称、图片 URL、评分、编译后的营业时间、基础费用、标签位集、prep_time_p50、状态， version/etag.
  • 以 restaurant_id为键存储在 Redis 集群中。单次 MGET 即可在一次往返中取回所有候选卡片。
  • 本地 LRU 在浏览服务层缓存每个城市排名前 N 的餐厅——这是极热的尾部流量。
  • 照片 通过 URL 从 CDN 提供；食客 app 在文字内容渲染完成后再加载图片。
  • 食客 app 在刷新时发送 If-None-Match: <etag> ；未更改的卡片返回 304。

实时状态 不能 存储在快照中的实时状态 —— status （已暂停/已关闭），当前 busy 级别——在卡片构建时以独立的 Redis hash 叠加。

营业时段与营业时间

每家餐厅有每周营业时间及特殊日期。写入时，目录服务将营业时间编译为本地时区未来 7 天的 (open_ts, close_ts) 区间列表，存入快照。读取时，"当前营业"通过区间查找实现，"即将打烊"以类似方式预计算。预约订单翻转时间戳：同一数据结构，未来 now.

卡片上的 ETA

浏览时显示的 ETA 是 近似值 ，这是设计决策：

eta_min = prep_time(restaurant, current_load) + drive_time(restaurant_cell, eater_cell)
eta_max = eta_min + uncertainty_pad
  • prep_time 来自与调度系统相同的模型，每约 30 秒刷新一次，当餐厅待处理订单数超过阈值时由订单服务更新。
  • drive_time 是 非 每次浏览每家餐厅都调用一次路由引擎。我们预先计算一个 cell -> cell 基于观测配送数据、每隔几分钟刷新的平均行程时间矩阵。粗略、快速，对卡片来说已足够好。
  • 缓存 (eater_cell, restaurant_id) -> eta 持续约 60 秒。

在餐厅菜单页面我们会精化 ETA，那里只有一家餐厅，可以承担一次路由调用的开销。

排名：两阶段

延迟预算不允许对数百个候选项运行重型模型。两阶段策略：

  1. L1（轻量、快速）： 对预计算特征进行线性混合——区域内热度、评分、ETA、配送费、食客菜系偏好粗向量。将候选从约数百个缩减至约前 50 个。
  2. L2（ML 模型）： 更丰富的特征——食客嵌入（离线每日更新，在线增量）、餐厅嵌入、时段、天气、历史交互。GBDT 或轻量 transformer 打分 P(order | eater, restaurant, context)。对前 50 个重新排序。

特征来自缓存；排名过程中不同步调用底层存储。食客嵌入由 eater_id缓存；餐厅嵌入已烘焙进快照。

冷启动食客 （无历史记录）：跳过个性化，回退到区域热度排名器。首次打开时无个性化往返调用。

推广位

独立的竞价服务返回 (restaurant_id, bid) 对。在固定位置（如第 1 位、第 5 位）注入并附上"推广"标签。竞价系统遵守投放节奏和每个广告主的每日预算上限。广告相关性须通过相同的地理位置 + 当前营业 + 筛选项检查，确保不展示无法送达的广告。

过滤条件

筛选项是排名前应用的硬约束。两种形式：

  • Bitset 筛选 存储在快照中：每家餐厅携带一个标签 bitset（素食、清真、快速配送、免配送费促销等）。对候选集进行 CPU 级位运算，耗时微秒级。
  • 索引级预筛选 用于最热门的维度： cell_id:vegetarian -> [restaurant_id]。读取开销更低，但占用更多索引空间。仅用于少数高基数筛选项。

实时失效管道

商家暂停、修改营业时间或新上线，必须在秒级内反映出来。

Merchant action -> Kafka(merchant_events)
                       |
        +--------------+---------------+--------------------+
        v              v               v                    v
  Forward-index   Card snapshot   Live status hash    Local-LRU
  updater         updater         (paused/open)       fan-out invalidate

食客 app 不订阅每一个变更；下一次浏览请求将获取最新状态。对于正在查看 feed 的食客，可以通过实时通道推送"餐厅已暂停"提示，使卡片变灰，无需刷新。

分页

  • 首屏：约 10-15 张卡片。
  • API 使用不透明游标分页，编码 L2 排名器的分数断点和平局打破键。
  • 后续页面对该页餐厅重新验证可用性，但不重新排名。
  • 我们不在索引层分页——那会破坏全局顺序。

冷流量/匿名流量的边缘缓存

对于未登录或新用户请求，响应具有很高的可缓存性。以 (cell_id, filter_hash, user_segment) 为键缓存响应，TTL 约 10-30 秒。已个性化的用户跳过缓存，直接命中 feed 构建器。这一层单独吸收了晚餐高峰流量的大部分。

浏览功能特有的故障模式

失败 缓解措施 信号
前向索引 Redis 宕机。 回退到更冷的"城市热门"列表；优雅降级。 索引缺失率。
L2 排名器慢。 仅 L1 回退；发出告警。 排序器 p99 延迟。
ETA 服务慢。 使用带"近似"标识的过期 ETA。 ETA 获取 p99。

浏览功能运营指标

  • 浏览延迟 p50 / p99（目标 p99 低于 300 ms）。
  • 边缘缓存命中率；Redis 命中率；本地 LRU 命中率。
  • 排名器 p99 及模型新鲜度滞后。
  • 过期卡片率（快照与实时状态不一致）。
  • 空 Feed 率（用户体验失败信号）。
  • 前排位置和推广位的点击率（CTR）。
  • 滚动深度和从浏览下单的转化率（用于训练排名器）。

浏览功能特有的权衡

  • 区域前向索引 vs. 每家餐厅独立多边形： 前向索引更快，但在多边形编辑时更难失效——这是可接受的代价。
  • 浏览时近似 ETA vs. 精确 ETA： 在此 QPS 下近似 ETA 是必要的；精确 ETA 在菜单打开时提供。
  • 两阶段排名器 vs. 单一复杂模型： 两阶段引入了复杂性，但将 p99 控制在预算内。
  • 匿名流量边缘缓存 vs. 始终个性化： 以少量个性化损失换取大幅 QPS 提升。
  • 推送实时失效 vs. 下次请求时拉取： 主要采用拉取方式；仅对活跃 feed 视图推送。

面试用语

浏览查询的本质是："哪些餐厅现在能送到我这里、按我的偏好排序、并附有实时状态。"我们通过区域键控前向索引、Redis 中的反范式化卡片快照、两阶段排名器和近似 ETA 装饰器来回答这个问题。实时状态通过 Kafka 失效管道流转。匿名和冷启动流量命中边缘缓存 feed；已个性化用户命中 feed 构建器。核心在于将 p99 控制在 300 ms 以内，同时在秒级内反映实时可用性。

商品缺货（"86'd"）

餐厅在班次内会出现商品缺货。系统必须快速反映这一情况，否则食客会下单无法获得的商品。

  • 商家平板电脑在商品缺货点击时推送 AvailabilityEvent 。
  • 可用性服务写入每家餐厅的 Redis hash，并向 Kafka topic 发出消息。
  • 搜索索引消费者更新商品的 available 字段。
  • 目录读取在返回菜单前查询 Redis 获取实时可用性。
  • 下单时，订单服务在短暂加锁下重新验证每个商品的可用性；若有任何商品不可用，提示食客替换或移除。

定价与费用

定价由购物车和上下文确定性计算得出：

subtotal      = sum(item.unit_price * qty + modifier prices)
delivery_fee  = base + distance_fee + surge_multiplier * peak_fee
service_fee   = round(subtotal * service_rate)
tax           = tax_engine(subtotal, jurisdiction)
discount      = promo_engine(eater, cart)
tip           = eater-entered, editable up to N minutes after delivery
total         = subtotal + delivery_fee + service_fee + tax - discount + tip

merchant_payout = subtotal - commission - adjustments
courier_payout  = base_payout + distance + time + tips + incentives

每个组成部分都记录在 Order.pricing中。对账作业从快照重新计算总额以检测漂移。

骑手调度（考虑备餐时间）

这是与网约车最大的差异。我们不希望骑手提前 10 分钟到达餐厅（浪费时间，骑手离开）或晚到 10 分钟（食物变凉，食客体验差）。

调度器输入：

  • prep_time_estimate(restaurant, order_size, current_load) ——基于餐厅 × 星期几时段历史备餐时间训练的 ML 模型。
  • pickup_eta(courier, restaurant) ——路由引擎。
  • placed_at 时间戳。

目标： courier_arrival_at_restaurant ≈ food_ready_time.

算法概要：

  1. 在 placed，估算 food_ready_time = placed_at + prep_time_estimate.
  2. 计算 dispatch_deadline = food_ready_time - pickup_eta(best_candidate).
  3. 推迟发单至 now >= dispatch_deadline - safety_margin.
  4. 调度时刻，在地理索引上运行匹配：候选骑手在半径范围内，按 ETA + 评分 + 批单潜力打分，通过 Courier.status 上的 CAS 原子预留（与 Uber 相同）。
  5. 若商家提前或延迟发出 ready ，重新规划：提前调度或回退到更近的骑手。
  6. 若最佳候选骑手的 ETA 超过食物准备好时间超过阈值，立即调度（食物变冷的风险优先于等待时间浪费的风险）。

面试用语：

调度是调度问题，而非匹配问题。我们故意延迟匹配，使骑手恰好在食物准备好时到达。备餐时间模型是核心。

批单

一名骑手在同一家餐厅（或相邻餐厅）取多个订单并路线重叠，是重要的效率提升。

批单条件：

  • 相同或相邻餐厅。
  • 取件地点在小绕路范围内。
  • 两个订单的 ETA SLA 均满足。
  • 食客未选择退出。

实现方式：

  • 每个区域维护一个流式"可批单队列"。
  • 在线装箱：每隔几秒，尝试将当前待处理订单两两配对。
  • 形成 batch_id；作为整体调度。
  • 骑手 app 显示有序行程：取 A、取 B、送 A、送 B（路线优化）。

权衡对比：

  • 在密集市场减少约 10-30% 的骑手供需需求。
  • 第二个送达订单多等几分钟；通过路线优化缓解。
  • 增加了骑手 UI 和状态机的复杂性。

实时追踪

食客看到：餐厅已接单、备餐中、骑手出发、骑手已到店、骑手正在前往、已送达。

  • 基于持久连接（WebSocket / SSE）实现。
  • 订单服务发出状态变更事件。
  • 在取单/送单阶段，来自地理索引管道的骑手位置更新被推送给订阅的食客。
  • 为省电，骑手 app 每 4-10 秒上报一次位置；服务端以更低频率向食客扇出。

支付集成

复用 支付系统:

  1. 下单时：预授权 estimated_total （含小费占位符）。
  2. 商家接单时：确认授权仍然有效。
  3. 送达时：扣款实际金额 total （含最终小费）。
  4. 退款：部分退款（商品缺失）或全额退款（未送达）。通过退款管道处理。
  5. 商家结算：按周期（每日/每周）汇总；与已接受订单对账。
  6. 骑手结算：按周期汇总；小费 100% 转给骑手。

幂等性： order_id 是支付提供商操作的幂等键。

  • 各城市操作在最近区域运行。
  • 目录数据全球复制为只读；写入以区域为主节点。
  • 跨区域食客（出行中）较少；解析到本地目录。
  • 全球服务（账户、支付）全球复制，区域主节点。

优雅降级规则因界面而异：浏览可以减少个性化，ETA 可以放宽，但结账和支付必须停止而不是猜测。

运维指标

正确性：

  • 订单状态不一致。
  • 支付与订单之间的漂移。
  • 重新计算的定价与已扣款总额匹配。
  • 骑手双重分配（必须为零）。

容量：

  • 浏览 QPS，p99 延迟。
  • 下单速率。
  • 每秒位置更新次数。
  • 搜索索引延迟。

业务层面：

  • 每城市每小时订单数。
  • 接单率（商家）。
  • 重新分配率（骑手）。
  • 下单到送达的平均总时长。
  • 平均食物就绪到取餐的时间差（≈0 为理想值）。
  • 退款率/类别（缺少商品、超时、食物变凉）。
  • 合单率。
  • 峰时触发频率。

需要指出的权衡

  • 早匹配 vs 晚匹配： 晚匹配与食物就绪时间对齐，但若最近骑手离开，则会减少调度选项。
  • 合单： 提高效率，但增加第二个配送点的用餐者侧延迟。
  • 严格的商品可用性传播： 减少错误订单，但增加目录变更流量。
  • 餐厅接单 SLA： 短 SLA 改善用餐者体验，但对备餐较慢的商家不利。
  • 按区域分片： 自然划分，但跨区域配送（长半径）需要显式逻辑处理。
  • 激进式价格激增： 平衡供需，但影响客户满意度。
  • 单区域支付授权 vs 配送时扣款： 下单时授权可防止欺诈，但需要精确的最终扣款逻辑来处理小费。

常见陷阱与更优做法

面试用语

Uber Eats 是一个三边市场，由目录/发现层、订单状态机和备餐时间感知的骑手调度系统拼接而成。与打车的最大区别在于，我们希望骑手恰好在食物备好时到达，而不是尽可能快地到达。目录读取占主导地位；订单状态是一致性热点；合单是效率杠杆；定价和佣金结算由已记录事件确定性计算得出。

2. 模拟面试

面试官： 设计 Uber Eats。

候选人： 我将其设计为一个三边市场——用餐者、商家、骑手——构建在四个层次之上。目录/发现层用于浏览餐厅和菜单。订单服务拥有一个三方状态机。骑手调度层复用 Uber 地理索引，并增加备餐时间感知调度和订单合单功能。支付/定价层位于顶部。不变量是每个订单都流经 cart → placed → accepted → preparing → ready → picked_up → delivered，支付具有幂等性，每个订单（或合单）最多一名骑手，菜单在几秒内反映商家实时可用性。

面试官： 从浏览和搜索开始。

候选人： 目录读取占主导，因此我将其与写路径分离。餐厅和菜单存储在按 restaurant_id分区的关系型数据库中。菜单缓存在带 TTL 的 Redis 中；图片从 CDN 提供。搜索在按地理区域分片的 Elasticsearch 索引上运行。浏览查询获取用餐者位置，扩展到配送半径内的区域，按菜系和评分过滤，并按结合 ETA、评分和热度的综合评分排序。商品可用性变更通过流式管道同步到 Redis 和 ES，使菜单在几秒内保持准确。

面试官： 一家餐厅中途售罄三文鱼。会发生什么？

候选人： 商家在平板上点击"停售三文鱼"。这会触发一个 AvailabilityEvent。可用性服务写入每家餐厅的 Redis 哈希并向 Kafka 主题发送消息。两个消费者：一个更新 ES 文档，使搜索和浏览不再将其显示为可用；另一个使菜单缓存失效。几秒内，正在浏览菜单的用餐者会看到三文鱼显示为不可用。为了处理进行中结账的竞态条件，订单服务在结账时短暂加锁重新验证商品可用性；若三文鱼此时已不可用，系统提示用餐者替换或移除。

面试官： 用餐者结账。请逐步讲解。

候选人： 购物车进入订单服务。我们重新验证商品和价格。调用定价服务计算小计 + 配送费 + 服务费 + 税 - 折扣 + 小费占位符。通过支付服务使用 order_id 作为幂等键对总金额进行预授权。授权成功后，我们转换 cart → placed，追加到状态历史记录，并向商家适配器和调度调度器发出 OrderPlaced 。商家平板收到订单。用餐者看到"等待餐厅确认"。

面试官： 您何时调度骑手？

候选人： 这是核心调度问题。我不会立即调度。我使用 food_ready_time = placed_at + prep_time_estimate(restaurant, order_size, current_load) 通过基于历史就绪时间训练的 ML 模型进行估算。我计算 dispatch_deadline = food_ready_time - pickup_eta(best_nearby_courier)。我将订单提供推迟到该截止时间前留有一小段安全余量。目标是让骑手在食物就绪时大约到达餐厅——而不是提前 10 分钟，也不是晚 10 分钟。

面试官： 如何避免骑手被双重预订？

候选人： 与 Uber 相同的原子 CAS 模式。从地理索引对候选人评分后，调度器对骑手状态执行条件更新： UPDATE Courier SET status=offer_pending, active_orders=active_orders+order_id WHERE courier_id=? AND status=online_available。只有一个 CAS 成功。失败的调度器继续尝试下一位候选人。对于合单，CAS 条件为"骑手空闲或已持有同一餐厅的可合单订单"。

面试官： 请详细介绍合单。

候选人： 合单是主要的效率杠杆。在每个区域中，缓冲区积累待处理订单。每隔几秒，我们运行一次在线装箱算法：将来自相同或相邻餐厅、配送地点在小绕路范围内且 SLA 仍允许合单的订单配对。若找到配对，我们生成一个 batch_id 并将两个订单都调度给一名骑手，附带路线优化的行程安排：取 A，取 B，送 A，送 B。对时间敏感的订单，用餐者可选择退出。代价是第二个配送点增加几分钟，但换来的是每个订单所需骑手行程大幅减少。

面试官： 商家标记订单已就绪。

候选人： CAS 将状态转换为 preparing → ready 并发出 OrderReady。骑手状态很可能为 en_route_pickup 或 at_restaurant。若骑手已在餐厅，则开始交接；骑手扫描袋子，转换 ready → picked_up。若骑手尚未到达，订单在柜台等待——这正是我们通过备餐时间感知调度努力最小化的代价。若就绪事件比预测早很多，调度器重新规划，可能将调度提前至最近的可用骑手。

面试官： 骑手完成配送。

候选人： 骑手转换 picked_up → delivered，可选附上交接照片或用餐者确认。订单服务触发对最终总金额的支付扣款（包含最终小费，用餐者可在时间窗口内编辑）。支付服务使用 order_id 作为幂等键。商家和骑手的佣金结算汇总到下一个结算周期。系统提示用餐者评分；骑手对用餐者评分。

面试官： 用餐者反映缺少一件商品。会发生什么？

候选人： 用餐者在应用中提交问题。退款管道进行分类——缺少商品、商品错误、食物变凉、送达超时——并运行资格规则。在阈值内，退款自动批准（缺少商品部分退款，以积分或原始支付方式返还）。超出阈值，由客服处理工单。订单转换到配送后流程的子状态；商家的缺少商品率被追踪，并影响排名，可能还会向商家扣除相关费用。

面试官： 高峰期商家平板离线。

候选人： 商家适配器监控心跳。检测到丢失时，为该餐厅暂停新订单（搜索将其标记为暂停）。进行中的订单升级为电话联系商家确认/拒绝。重新连接时重试推送已接受的订单。有进行中订单的用餐者收到可能延迟的通知。接单 SLA 计时器在尝试带外联系期间暂停。

面试官： 热门餐厅——一家病毒式爆红的餐厅收到正常订单量 10 倍的订单。

候选人： 首先，我们让商家发出容量信号（例如"厨房繁忙，+20 分钟"）。我们将更长的备餐 ETA 传播到浏览和结账界面。若订单速率超出容量，我们限流：减少接受新订单，显示更长的排队 ETA，推荐替代选项。备餐时间模型通过当前负载特征快速适应新负载。该区域的骑手将看到激励并迁移过来。热门餐厅变成热点分片；若持续高热，我们可以进行预分片。

面试官： 多区域？

候选人： 城市是自然分片单位，与 Uber 相同。每个城市的目录、订单和调度在最近的区域运行。目录是唯一受益于全球读副本（服务于旅行中的用餐者）的部分。支付和账户是全局的，设有区域主节点。跨城市情况罕见，通过显式逻辑处理——大多数配送在同城内完成，配送半径对此有约束。

面试官： 对账。

候选人： 夜间任务。从快照重新计算订单定价；标记漂移。将支付扣款与已送达订单进行比较。从已接受订单中扣除佣金和退款后重新计算商家佣金结算；与下一个结算金额比较。对骑手佣金结算与完成的配送和小费进行对账。标记卡住的订单（超过 2 小时仍在 preparing ，超过 30 分钟仍在 ready ）供客服介入。

面试官： 总结。

候选人： 三边市场。目录/搜索层用于浏览，按城市分片，积极缓存。订单服务具有三方状态机——商家和骑手轨道并行运行，在取餐时汇合。调度是备餐时间感知的：故意晚匹配，使骑手在食物就绪时到达。合单是效率杠杆。商品可用性在几秒内流式同步。支付在下单时授权，送达时扣款，以 order_id为幂等键。按城市分片；对账捕获漂移。与 Uber 最大的区别在于时机——我们不追求最快匹配，而是与就绪时间对齐。

3. 练习备注

五分钟版本

  1. 明确：浏览、购物车、三方订单、骑手调度、支付、评分。
  2. 不变量：订单流经状态机；每个订单最多一名骑手；菜单反映实时可用性；支付幂等。
  3. 分层：目录/搜索、订单服务、调度（备餐时间感知）+ 合单、支付。
  4. 目录读取占主导 → CDN + Redis + ES；可用性在几秒内流式同步。
  5. 订单状态机 + 并行骑手状态机，在取餐时汇合。
  6. 调度推迟至≈食物就绪时间；骑手状态使用 CAS；合单引擎。
  7. 下单时授权、送达时扣款的支付，以 order_id 保证幂等性。
  8. 按城市分片；对账任务。

一分钟结尾陈述

Uber Eats 是一个三边市场，构建在地理空间调度核心之上。目录和搜索读取占主导，被积极缓存。下单具有强一致性，并重新验证实时商品可用性。骑手调度是备餐时间感知的：我们故意等待匹配，使骑手在食物就绪时到达，并对附近订单进行合单以提高效率。商家和骑手轨道是并行状态机，在取餐时汇合。支付是下单时授权、送达时扣款，以 order_id 保证幂等性。城市是自然分片单位。对账捕获定价、佣金结算和卡住订单的漂移。最难的正确性问题是三方交接；最难的规模问题是浏览扇出；最具特色的设计选择是故意晚匹配。
