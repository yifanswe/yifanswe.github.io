Local Delivery Service — DoorDash-Style (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Easy · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建一个类似 DoorDash 的本地餐饮/包裹配送服务。顾客打开应用，查看附近商家，下单后由配送员取货并送至指定地址。我们的目标是在 30–45 分钟内将订单送达，并提供实时追踪。假设全美每日订单量为 1000 万。

1. 需求（约 5 分钟）

功能需求 - 顾客浏览附近商家并下单。 - 系统将订单匹配给附近的空闲配送员，并规划商家 → 顾客的路线。 - 顾客可在地图上看到实时预计到达时间和配送员位置。

我会将菜单管理、支付深度分析和配送员入驻排除在范围之外。

非功能需求 - 订单创建必须是事务性的——绝不重复扣款，绝不丢失订单。 - 匹配延迟 p99 < 5 秒；用户不会在点击"下单"后等待更长时间。 - 实时追踪每 3–5 秒更新一次，地图刷新延迟低于 1 秒。 - 地理隔离——美东故障不能阻塞美西订单。 - 订单状态机（已下单 → 已接受 → 已取货 → 已送达）要求强一致性；浏览和分析允许最终一致性。

容量估算 - 每日 1000 万订单 → 平均约 115 单/秒，午餐/晚餐高峰约 1000 单/秒。 - 高峰时约 100 万配送员同时在线，每 3 秒上传一次 GPS → 约 35 万次 GPS 写入/秒。 - 订单行约 2 KB × 1000 万/天 × RF=3，约 60 GB/天。配送员实时索引较小，因为只存储最新状态（约 100 万 × 150 B × RF=2 ≈ 300 MB），但如果原始 GPS 分析日志不压缩全量保留，其数据量可能远超其他：35 万/秒 × 约 50 B ≈ 每天 1.5 TB（RF/压缩前）。

2. 核心实体（约 2 分钟）

  • User — 顾客。
  • 商户 — 餐厅；有固定地理位置和菜单。
  • 订单 — 顾客、商家、商品、状态、地址、dasher_id。
  • Dasher — 当前位置、状态（离线 / 空闲 / 配送中）、当前订单。
  • GPS ping — dasher_id, lat, lng, ts。高频、短暂。

3. API（约 5 分钟）

GET   /v1/merchants?lat=&lng=&radius=    → Merchant[]
POST  /v1/orders                          { merchant_id, items[], address } → { order_id }
GET   /v1/orders/{id}                     → Order (status, dasher, eta)
WS    /v1/orders/{id}/track               → push dasher location + ETA
POST  /v1/dashers/location                { lat, lng } (auth: dasher)
POST  /v1/dashers/orders/{id}/{action}    action ∈ {accept, picked_up, delivered}

顾客 ID 和配送员 ID 来自认证令牌。

订单变更携带幂等键。追踪游标编码 {order_id, last_location_ts, last_state_version} 以便重连后只接收未收到的状态和位置更新。

5. 高层设计（约 10–15 分钟）

我会按城市/区域对整个世界进行分区，每个区域基本自成一体。

  1. 商家搜索。 客户端发送经纬度坐标，地理索引商家服务（PostGIS、S2 单元格或 geohash）返回按距离和热度排序的附近商家。该服务读多写少，可按单元格和时段进行缓存。
  2. 订单创建。 事务性服务。在区域订单存储（MySQL，按 order_id 在区域内分片）中写入状态为 placed的订单行，并向 Kafka 发布 OrderPlaced 事件。
  3. Matching 服务。 消费 OrderPlaced，查询该区域的配送员实时索引（附近且空闲），运行小型优化算法（选取预计完成时间最短的最近配送员），尝试进行分配。分配使用 Redis 支持的租约，确保两个并发订单不会争抢同一配送员。
  4. 配送员位置采集管道。 配送员 App 每 3 秒上报 GPS → 采集服务将数据写入按区域划分的 Redis 地理索引（GEOADD）以区域为键。同时持久化日志写入 Kafka 用于分析。
  5. 追踪 WebSocket。 分配完成后，顾客的 WebSocket 连接订阅每个订单的专属频道，该频道扇出配送员位置更新和状态变更。
  6. 订单状态服务。 管理状态机；向 Kafka 发布事件，供下游（通知、支付捕获、分析）消费。
  7. 支付和通知 位于订单事件流的下游——不在设计范围内，但需在架构图中体现。

端到端流程：顾客点击"下单" → 订单行写入，事件发布 → 匹配器选中配送员 #42 → 在 Redis 中租约配送员 42 → 发布 OrderAccepted → 顾客 WebSocket 建立连接 → 配送员 GPS 流同时扇入顾客 WebSocket → 每次配送员操作推进状态 → 触发 delivered时，捕获支付，进入终态。

具体存储方案：

存储 我会明确说出的具体存储方案： 条目大小 TTL / 保留期 分片键
Order DB order:{region}:{order_id} 状态行 ~2 KB 7 年/按策略归档 region + order_id
配送员地理索引 dashers:{region} Redis GEO 成员=dasher_id 约 150 B/配送员 10 秒新鲜度 TTL region
配送员租约 dasher:{id}:lease -> 订单 + 隔离令牌 ~100 B 10-30 秒 TTL dasher_id
商家索引 merchant_cell:{s2_cell} -> 商家 ID 约 16 B/成员 小时，重建 S2/geohash 单元格
GPS 日志 主题 gps.{region} 事件=dasher_id, lat,lng,ts ~50 B 24 小时原始数据，汇总后归档 region + dasher_id
订单事件日志 主题 orders.{region} 状态转换 ~500 B 7-30 天重放保留 region + order_id

存储估算：订单数据约 60 GB/天（已复制），按区域分片轻松扩展。原始 GPS 流数据量更大：主节点约 1.5 TB/天，RF=3 时约 4.5 TB/天（压缩前），因此仅保留 24 小时原始数据并做汇总供分析使用。实时索引体量极小，风险在于数据新鲜度，而非存储字节数。

6. 深度探讨（约 10 分钟）

配送员和商家的地理索引。 商家位置固定，因此预计算的 S2 单元格 → 商家列表方案可行。配送员每 3 秒移动一次；Redis GEOADD 与 GEORADIUS 是实际最优选择——内存存储、亚毫秒延迟、天然支持按区域划分。

为何配送员用 Redis GEO 而商家用 S2/PostGIS：配送员需要读取最新位置并设置短 TTL，内存地理索引更具优势；商家信息静态且更丰富，预计算的 S2 单元格或 PostGIS 效果好且按单元格/时段缓存成本更低。

面对"为何不对配送员也用 PostGIS？"需有备选答案：在低写入频率下 PostGIS 可行，但每秒 35 万次位置写入加上 5 秒匹配延迟要求，促使我们选择短暂存活的内存索引。PostGIS 仍是商家数据的权威来源，也适用于离线空间查询。

匹配算法。 简单的最近配送员算法会导致聚集和分配不均。我会采用带小批量窗口的贪心分配（收集 2 秒内的订单，运行二部图匹配）来平衡配送员负载。可提及匈牙利算法或基于拍卖的匹配作为生产级调优手段。

配送员分配的并发控制。 若不加以控制，同一配送员可能被两个订单同时抢占。使用 Redis SET dasher:42:lease order:99 NX EX 10 进行 10 秒租约，同时通知配送员。若配送员拒绝或超时，租约过期后重新匹配。

订单创建路径追踪：客户端 -> 订单服务校验购物车和幂等键 -> 支付授权与订单行写入在同一 saga 步骤中执行 -> 发布 OrderPlaced -> 匹配器从 dashers:{region} 读取附近空闲配送员 -> 租约一名配送员 -> 使用行锁写入 assigned 状态 -> 向 WebSocket/通知服务发送状态事件。

追踪读路径：顾客 WebSocket 订阅 order:{id}；推送层并行从 MySQL/缓存读取最新订单状态，从 Redis 读取最新配送员位置；之后通过 gps.{region} 和订单事件使用 {last_location_ts,last_state_version}.

Order placement hot path
    client -> Order API
        -> idempotency:{customer}:{key}
        -> MySQL order row status=placed
        -> Kafka orders.{region} OrderPlaced
        -> Matcher GEORADIUS dashers:{region}
        -> SET dasher:{id}:lease NX EX 10 with fencing token
        -> MySQL order status=assigned version+1
        -> WS/notification event

两种视图，一次配送：订单行以 order_id 为键保证正确性，而配送员实时索引以 region/geo_cell 为键用于匹配。同一配送员位置事件同时写入 gps.{region} 用于重放/分析，但只有 Redis 处于面向用户的热路径上。

实时追踪规模。 每秒 35 万次 GPS 写入不适合逐次扇出。改为：GPS 写入 Redis，WebSocket 推送层每 3–5 秒按订单批量更新，而非逐次 ping 触发。

订单状态转换。 状态机由单一所有者管理，在 MySQL 中以 order_id为键的行级锁保护。每次转换向 Kafka 发送事件供下游服务消费。配送员操作接口的幂等键可防止在不稳定网络下重复转换。

区域隔离。 按区域划分：MySQL、Redis、Kafka、服务各自独立。仅身份认证和商家目录使用全局服务。美东故障不影响旧金山的订单处理。

预计到达时间预测。 从启发式算法起步（Haversine 距离 / 城市平均速度 + 5 分钟商家备餐时间）。后续接入基于历史行程数据训练的模型。明确指出 v1 为启发式方案。

优雅降级。

情形 模式 数据源 用户可见行为
配送员 GPS 数据过期 保留分配，标记位置为过期 最后一次 Redis 位置 地图暂停，预计到达时间拓宽
附近无配送员 扩大搜索半径，然后将订单排队 地理索引 + 匹配器 分配延迟

订单状态机：

阶段 模式 数据真实来源 延迟预算 转换守卫
placed 持久化订单 MySQL 行 + OrderPlaced 事件 <300 ms 幂等键唯一性约束
matching 短暂租约 Redis GEO + 租约令牌 <5 s 租约隔离令牌

面试要点："订单需要强状态转换；位置是短暂信号。我绝不会将它们放入同一个一致性桶中。"

我会推迟的内容。 多站点订单、批量配送（一名配送员携带 2 个订单）、激增定价、配送员排班与收益。
