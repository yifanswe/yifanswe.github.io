Donations Website — GoFundMe Style (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Easy · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建一个捐款网站。组织者创建带有目标金额的活动。捐款人浏览活动、捐款，进度条实时更新。会出现流量尖峰——一个病毒式传播的灾难活动可能在一小时内吸引 1 万笔捐款。稳态流量低，病毒式峰值是设计的关键驱动因素。

1. 需求（约 5 分钟）

功能需求 - 组织者创建活动：标题、故事、目标金额。\n- 捐款人查看活动，通过银行卡捐款。\n- 活动页面显示实时总额和最近捐款人列表。

我会推迟处理组织者提现、身份验证和退款。

非功能需求 - 资金的正确性是首要要求——不能重复扣款，不能丢失捐款。\n- 活动页面 p99 延迟 < 500 ms，即使在病毒式传播期间（1 万并发访问者）。\n- 已筹金额计数器强一致性。\n- "最近捐款人"列表最终一致性可接受（几秒内同步）。

容量估算 - 基准：1 万个活跃活动，每天 1000 笔捐款。\n- 病毒式活动：1 小时内 1 万笔捐款 → 单个活动约 3 笔/秒，10 万并发访问者。\n- 读取尖峰才是真正的规模驱动因素：10 万访问者每 2 秒轮询一次，单个活动的实时组件 QPS 高达 5 万。对实时区块积极缓存，但保持资金账本的强一致性。

2. 核心实体（约 2 分钟）

  • 广告活动 — campaign_id, organizer_id, title, story, goal_amount, raised_amount, status.
  • Donation — donation_id, campaign_id, donor_id, amount, message, anonymous, status.
  • 支付 — 由支付网关（Stripe）处理，我们保存引用。

3. API（约 5 分钟）

POST  /v1/campaigns             { title, story, goal }       → { campaign_id }
GET   /v1/campaigns/{id}        → { campaign, recent_donations[] }
POST  /v1/campaigns/{id}/donations    { amount, payment_method, anonymous? }    (Idempotency-Key)   → { donation_id, status }
GET   /v1/campaigns/{id}/donations?cursor=...

从 token 获取用户。捐款时必须携带 Idempotency-Key。

捐款响应可安全重试，因为幂等键映射到一个持久化的捐款尝试： {donor_id, idempotency_key} -> donation_id, status, gateway_charge_id}.

5. 高层设计（约 10–15 分钟）

  1. 活动服务。 对活动进行 CRUD 操作，按 campaign_id.
  2. 捐款服务。 接收捐款请求，创建 donation 行（status=pending），调用支付网关，成功后转换为 succeeded 并原子性地递增活动的 raised_amount.
  3. 支付网关集成。 Stripe（或类似服务）。我们不存储 PAN，使用 Stripe token。幂等键透传。
  4. 实时计数器。 raised_amount 通过捐款服务写透缓存到 Redis 中。活动页面读取缓存值加上最近的捐款记录。
  5. 最近捐款人列表。 每个活动追加写入——Redis list 保存最近 50 个捐款人，溢出时淘汰到 MySQL。
  6. 活动页面。 静态部分（故事、图片）可由 CDN 缓存。动态部分（已筹总额、最近捐款人）由访问 Redis 的轻量 API 调用提供。

端到端叙述：捐款人打开活动页面 → 静态内容来自 CDN；AJAX 调用 /campaigns/123/live → Redis 返回 {raised: 47600, recent: [...]} → 进度条渲染。捐款人捐款 $50 → API 调用（携带 Idempotency-Key）→ 创建捐款行（pending）→ Stripe 扣款成功 → 原子更新：campaign.raised += 50，donation.status = succeeded，最近列表前置 → 下次页面轮询看到新总额。

捐款写入路径：

  1. API 网关验证捐款人身份并检查 MySQL 中的 (donor_id, idempotency_key) 。
  2. 捐款服务在同一事务中插入 donation(status=pending) 和账本意向行。
  3. 使用相同的幂等键调用 Stripe；Stripe 可能返回成功、失败或超时。
  4. 成功后，DB 事务将捐款标记为 succeeded，插入不可变账本贷记，并递增 campaign.raised_amount.
  5. 提交后，更新 Redis campaign_live:{campaign_id} 。 recent_donors:{campaign_id}；若 Redis 失败，DB 仍保持正确，缓存在下次读取时重建。

活动实时读取路径：

  1. 浏览器以 campaign_id 为缓存键，从 CDN 加载活动静态 HTML/图片。
  2. 调用 /v1/campaigns/{id}/live，服务读取 Redis campaign_live:{campaign_id} 。 recent_donors:{campaign_id} 。
  3. 缓存缺失时，按 campaign_id读取 MySQL 活动行，从捐款索引读取最近成功的捐款，用 1-5 s TTL 重建 Redis 并返回。
  4. 捐款人分页的游标使用 {campaign_id, last_created_at, last_donation_id} 而非偏移量，使新捐款不会改变分页边界。
  5. 若 Redis 在病毒式事件期间宕机，服务提供 CDN/边缘缓存的实时区块（1-2 s），或回退到 MySQL 并对每个活动进行积极的请求合并。
Donate:  Client -> Donation Service -> MySQL donation pending -> Stripe
                                      -> MySQL ledger + raised_amount commit
                                      -> Redis live cache + recent donors

Read:    Browser -> CDN static page
                 -> Live API -> Redis live/recent -> MySQL rebuild on miss

6. 深度探讨（约 10 分钟）

尖峰下的计数器一致性。 病毒式活动中捐款涌入速度很快，活动行成为热行。两种方案：\n- 单行 CAS 更新： UPDATE campaigns SET raised = raised + 50 WHERE id = X。在单行数百 QPS 以内没有问题。\n- 分片计数器： N 个计数行，读取时求和。可进一步扩展。对于捐款量（即使最热的活动也不超过 100 笔/秒），更简单的单行 CAS 已足够；分片方案作为扩展选项提及。

具体存储方案：

存储 键模式 值 / 数据结构 条目大小 TTL / 保留期 分片键
活动 DB campaign_id 活动行 + raised_amount ~2 KB ， campaign_id
捐款 DB donation_id；唯一 (donor_id, idem_key) 捐款状态行 ~1 KB 永久 / 审计 campaign_id 或 donor_id 按查询需求
账本 DB ledger_entry_id 不可变借贷条目 约 200 B ， campaign_id
实时缓存 campaign_live:{campaign_id} 已筹金额、目标金额、状态 约 200 B 1-5 s TTL，写透 campaign_id
最近捐款人 recent_donors:{campaign_id} Redis LIST 最近 50 个捐款人片段 ~150 B/条目 1 h 或刷新 campaign_id
捐款人页面索引 (campaign_id, created_at, donation_id) 成功的捐款 ID ~40 B/索引条目 ， campaign_id
幂等性 idem:{donor_id}:{key} donation_id, status, gateway_charge_id 约 200 B 24 h+ donor_id
Webhook 队列 stripe_webhook:{event_id} 持久化事件载荷 ~1 KB 30 天 event_id

容量估算：每天 100 万笔捐款，每笔约 1 KB 捐款行 + 200 B 账本行 + 40 B 捐款人索引条目 ≈ 1.24 GB/天（不含索引和副本），与媒体/故事资源相比微不足道。每天 100 万次的幂等性数据以 200 B 计，加上 TTL 仅数百 MB/天。病毒式读取缓存是热路径：5 万 QPS 访问 200 B Redis 值轻而易举；5 万 QPS 访问 MySQL 是自找麻烦。1 万个活跃活动 × 50 × 150 B ≈ 75 MB（不含 Redis 开销）。最大的正确性成本是运营层面，而非存储：协调 Stripe、账本和活动总额。

捐款有两个视图。不可变账本是资金真相，以 ledger_entry_id 和活动为键。活动页面是服务视图： campaign.raised_amount, campaign_live:{campaign_id}，以及 recent_donors:{campaign_id}。若服务视图陈旧，从账本/捐款行重建；永远不要从进度条重建账本。

幂等性。 捐款提交在 (donor, key)上幂等。这至关重要，因为结账时的慢网络会导致重试，缓存响应可防止重复扣款。

支付集成。 我们将银行卡交给 Stripe Elements（客户端），获取 token，然后携带 {token, amount}调用我们的 API。我们从不接触 PAN——PCI 范围最小化。

为何选择 MySQL + Stripe + Redis。 MySQL 是处理资金的正确默认选择，因为我们需要事务、唯一幂等约束和审计查询。分布式 NoSQL 计数器可以扩展写入，但会使对账更困难。Stripe 负责卡处理和幂等扣款；自建支付处理器超出范围且会引入合规问题。Redis 仅作为活动页面的服务缓存，永远不是资金的真相来源。

扣款与账本更新之间的故障。 Stripe 返回"已扣款"，但我们的 DB 更新失败。两条路径：\n- Stripe 的 Webhook 确认；我们的对账处理器重试账本更新。\n- 轮询扫描器查找超过 5 分钟的 pending 状态捐款，查询 Stripe，正确转换状态。

捐款状态机：

状态 数据真实来源 状态转换 用户可见结果
pending 捐款行 请求已接受，Stripe 调用进行中 显示转圈 / 待处理收据
succeeded_unposted Stripe 扣款 DB 更新在扣款后失败 对用户隐藏；Webhook/扫描器修复
succeeded 账本 + 捐款行 账本已提交，计数器已递增 感谢页面 + 进度更新
failed Stripe 响应 银行卡拒绝 / 欺诈拒绝 计数器不变
refunded 账本冲销 退款已提交 已筹金额递减

实时页面降级状态：

阶段 模式 数据源 用户可见行为
正常 新鲜的实时区块 Redis campaign_live + recent_donors 进度在几秒内更新
缓存缺失 重建 MySQL 活动 + 捐款索引 一次较慢的请求，然后回填
病毒式尖峰 边缘合并 CDN 缓存实时 JSON 1-2 s 众多访问者共享一次源站读取
Redis 宕机 DB 安全回退 MySQL 已筹金额，最近列表可能隐藏 进度滞后，捐款仍可接受
支付漂移 对账 Stripe webhook/sweeper -> 账本修复 用户收据可能显示待处理状态

对于病毒式传播的活动，若 Redis 耗尽，则从 CDN 提供活动页面，并展示 MySQL 中最后一次已提交的总额，同时显示简短的"更新中"状态。绝不接受没有幂等键的捐款，也绝不信任缓存中的资金数据。

边缘压力测试：

边缘 设计方案 预算
热门活动读取风暴 CDN 静态页面 + 1-2 秒缓存的实时数据块 + Redis 源站 活动页面 p99 延迟 <500 ms
单次活动突发捐款 单行自增在每秒数百次以内没有问题；如有需要可切换为分片计数器行 无丢失自增
捐款人超时后重试 unique (donor_id, idempotency_key) 返回相同的捐款/支付网关扣款 不会重复扣款
游标耗尽最近列表 捐款人分页回退到 MySQL 索引 (campaign_id, created_at, donation_id) 稳定顺序
多区域漂移 支付和账本写入路由到活动归属地域；读副本服务活动页面 资金在单一地域内有序处理
缓存刷新后的冷启动 请求合并让一次重建即可填充 Redis，其余请求继续服务过期的边缘 JSON 避免数据库雪崩

面试核心要点： 实时页面是账本的缓存投影，而非资金交易的参与方。

最近捐款人列表。 每活动 Redis 列表（LPUSH + LTRIM 保留 50 条）。从 Redis 读取耗时低于毫秒级。匿名捐款者显示为"Anonymous"并附带金额。

活动页面缓存。 静态故事内容渲染为 HTML 并缓存到 CDN（TTL 5 分钟，编辑时清除）。动态块（计数器 + 最近记录）通过 JS 加载——在 CDN 边缘缓存 1–2 秒以吸收病毒式流量高峰。

欺诈与滥用。 启发式规则：被盗卡高频操作、先捐款后退款的模式、组织者自我捐款。Webhook 监听器监控拒付情况并上报给运营团队。

退款。 Stripe Refund API + 回滚 raised_amount 自增。同步在捐款服务中完成；捐款行状态 → refunded.

我会推迟的内容。 组织者 KYC 和打款、货币转换、评论审核、"类似活动"推荐。

面试核心要点： 将捐款首先视为账本，其次才是实时计数器。进度条延迟一秒是可以接受的；重复扣款或账本条目缺失才是真正的故障。
