1. 基础知识与解题思路

本题考察点

广告点击聚合器统计广告事件，用于计费、报告和预算控制。难点在于每次点击都涉及金钱：少计会损失收入，多计会破坏广告主信任，欺诈是持续的攻击手段，预算上限必须在多一次点击泄露之前停止投放广告。

其不变量如下：

对于每一个可计费的点击事件，系统精确地摄入一次，对重试和欺诈进行去重，聚合为每个活动和每个广告的计数器，最终收敛至权威总量，并在活动达到预算上限的瞬间停止投放广告——同时在快速计数器与权威账本之间进行对账。

在 ICT5 / L6+ 级别，面试官考察的是你能否将快速但近似的聚合路径与缓慢但正确的计费账本分离开来，处理迟到事件，在大规模场景下去重，在突发流量下执行预算上限，以及在资金流动前推理欺诈过滤。

60 秒面试框架

口述版本：

我会在同一点击流上设计两条路径：一条快速聚合路径，在数秒内更新每个活动的计数器，用于预算上限控制和仪表盘展示；一条慢速正确性路径，生成具有完整去重、欺诈过滤和迟到事件处理的权威计费记录。关键决策在于如何在高吞吐量下去重、如何在突发流量下执行预算上限、如何对账两条路径，以及如何在不拖慢广告服务器的情况下进行欺诈过滤。

首先明确需求

功能性需求：

  • 捕获每个广告上的每一次点击及其归因信息（用户、活动、广告、投放位置、时间戳）。
  • 以亚分钟级粒度，按广告、活动、广告主聚合点击数据。
  • 执行活动预算上限控制；预算耗尽时停止投放。
  • 为广告主提供近实时仪表盘。
  • 生成用于开具发票的权威计费记录。
  • 过滤欺诈性点击（机器人流量、点击农场、误触）。
  • 处理归因窗口（导致数小时后发生转化的点击）。

非功能性需求：

  • 峰值每秒摄入 100 万以上点击事件。
  • 在数秒内更新计数器；预算上限执行延迟低于 5 秒。
  • 计费的精确一次（exactly-once）语义。
  • 可审计性：每一笔计费点击均可追溯至其原始事件。
  • 多区域捕获；集中计费。
  • 欺诈过滤不得拖慢广告投放。

以下内容超出范围，除非被特别询问：

  • 广告定向 / 竞价系统（独立模块）。
  • 点击归因以外的转化归因算法。
  • 长期数据仓库 / OLAP。

规模假设

领域 假设
点击事件 峰值 100 万/秒；每日 100 亿。
曝光量 点击量的 100 倍。
活跃活动数 全球数百万，峰值时数万个热点活动。
预算执行延迟 全球范围内低于 5 秒。
仪表盘延迟 分钟级可接受。
计费对账 每小时或每日权威汇总。
事件大小 经过丰富处理后约 1 KB。
留存期 原始点击数据保留 90 天；聚合数据保留数年。
欺诈过滤器 过滤掉原始事件的 5-20%。

瓶颈在于：在足够快地执行预算上限以防止广告主超支的同时，仍需完成去重和欺诈过滤。

核心产品模型

路径 用途 需求
热路径（实时聚合） 用于预算上限控制和实时仪表盘的每活动计数器。 5 秒内新鲜度；近似值可接受。
冷路径（权威计费） 包含去重和欺诈过滤的每事件账本。 精确一次语义；用于开具发票的正确结果。
欺诈路径 在计费前检测并过滤欺诈性点击。 异步执行；无需阻塞广告投放。

这是经典的 lambda 架构 （热路径 + 冷路径），或者 kappa 架构 （以两种速度读取单一有序日志）。两条路径并存的原因：预算执行需要速度，计费需要正确性。

核心组件

组件 职责 关键细节
点击摄入端点 接收来自广告投放基础设施的点击事件。 轻量级；追加至日志；接近零延迟。
分布式日志（Kafka） 对点击流进行缓冲和分区。 分区键为 campaign_id，以实现亲和性。
流处理器（热路径） 按窗口聚合；更新活动计数器。 通过 Bloom filter 进行近似去重；亚秒级更新。
计数器存储（KV） 每个活动和每个广告的运行总量。 原子性递增；分片；由广告服务器读取。
批处理器（冷路径） 精确去重、欺诈过滤、归因处理。 每小时运行，或以更长窗口作为连续流运行。
计费账本 每次点击的权威记录。 仅追加；按日期 + 活动分区。
欺诈检测 识别机器人/农场流量。 ML 模型 + 规则引擎；在原始流上异步运行。
预算执行器 为广告服务器判断"活动已超预算"。 读取计数器；翻转活动状态。
对账器 将热路径计数器与冷路径账本进行比对；纠正偏差。 每小时执行；解释差异原因。
报表 API 提供仪表盘和报表服务。 从物化聚合存储中读取数据。

点击捕获与去重

每次点击由广告服务器生成。点击 URL 包含：

  • request_id：广告服务器签发的签名令牌，用于标识本次曝光。
  • campaign_id, ad_id, placement.
  • 带签名的时间戳，用于检测过期/重放的点击。

摄入端点负责：

  1. 校验签名。
  2. 分配服务端生成的 click_id （UUID）。
  3. 按 campaign_id.
  4. 返回重定向至目标 URL。

去重策略：

  • 热路径： 每个分区使用带近期窗口 TTL 的 Bloom filter。误报会丢弃计数器递增，但在统计上有界。
  • 冷路径： 以 request_id 为键在持久化集合中进行精确去重。若同一个 request_id 出现两次，只对其中一次计费。

为什么两者都需要？因为热路径需要快速且近似，冷路径需要精确但可以较慢。

热路径：实时聚合

流处理器（Flink、Spark Streaming、Kafka Streams）消费点击流：

  1. 从分区读取事件。
  2. Bloom filter 去重检查；若在近期窗口内为重复则跳过。
  3. 应用轻量级欺诈过滤（封禁列表 IP、明显的机器人模式）。
  4. 在计数器 KV 中原子性地递增 (campaign_id, time_bucket) 的计数器。
  5. 若活动的运行总量超过预算 × 阈值（例如 95%），则将活动状态设置为 near_budget。若超出预算，则设置 over_budget 并通知广告服务器。

广告服务器从快速缓存（约每秒刷新一次）中读取活动状态。当状态翻转为 over_budget时，广告服务器停止对该活动的出价/投放。

面试用语：

热路径优化的是预算执行速度。它是近似的——轻微的多计是可接受的，因为冷路径会进行对账；但少计是有害的，因为它会让广告在超出预算后继续投放。

预算上限执行

一个关键的正确性问题：预算为 $1000、每次点击 $1 的活动不应超出太多。

挑战：某个热点活动以 10k/秒的速度产生点击；即使计数器每秒更新一次，广告服务器也可能多投放数千次。

缓解措施：

  • 节奏控制： 限制每秒支出。平滑导致超支的流量突发。
  • 预测性截止： 在计数器达到预算的 95% 时停止投放，而非 100%，以应对在途点击。
  • 按区域分配预算： 跨区域预先分配预算，以避免全局计数器争用。
  • 快速计数器： 计数器 KV 驻于内存；亚毫秒级递增。
  • 直接通知广告服务器： 当预算阈值触发时，推送通知而非轮询。

即便如此，仍会出现轻微的超量投放。平台将其视为正常的运营成本——通常控制在一个较小的百分比以内。

冷路径：权威计费

批处理器或连续流处理器：

  1. 在较大窗口（例如 1 小时）内从 Kafka（或备份的 S3 归档）中读取数据。
  2. 使用 request_id 对持久化集合进行精确去重。
  3. 应用来自 ML 流水线的欺诈过滤结果（该流水线可能已提前处理过这些事件）。
  4. 如有需要，进行归因处理。
  5. 将计费记录写入带有 billing_status = valid 或 fraud 或 duplicate.
  6. 更新 budget_spent_authoritative 活动上的数据。

账本是开票的唯一权威来源。与热路径计数器的对账是自动进行的。

欺诈过滤

需检测的模式：

  • 来自单个用户/设备/IP 的高频点击。
  • 没有前置曝光的点击。
  • 来自已知机器人 User-Agent 的点击。
  • 转化率低于预期的点击。
  • 地理异常。
  • 点击农场（来自大量设备的同步点击）。

实现方式：

  • 一个独立的欺诈服务消费点击流。
  • 运行 ML 模型和基于规则的检查。
  • 写入 fraud_score 每个 click_id 到边表。
  • 冷路径使用 fraud_score 判断计费状态。
  • 热路径可能仅使用快速规则（封禁列表），因为等待 ML 结果会增加延迟。

欺诈路径是异步的，因为计费本身也是异步的。热路径会轻微多计；对账负责纠正。

迟到事件与乱序

事件迟到的场景：

  • 网络状况差的移动客户端。
  • 网络缓冲。
  • 跨区域复制延迟。

流处理器使用 水位线（watermarks） 来处理乱序问题：

  • 水位线的含义是"我们已接收到截至时间 T 的事件"。
  • 早于水位线的时间桶聚合结果将被最终确认。
  • 早于水位线的事件进入独立的"迟到"管道，用于调整历史聚合。

在计费场景中，冷路径会延迟运行（例如 1 小时），以等待大部分迟到事件到达。更晚到达的事件计入下一周期的调整。

扩展与分区

数据 分区键 原因
Click stream campaign_id 将同一活动的事件集中分配，用于聚合。
计数器 KV campaign_id 每个活动的原子性递增；每个热点活动一个分片。
计费账本 date + campaign_id 用于开票的时间范围查询。
去重集合（冷路径） hash(request_id) 均匀分布。
聚合 campaign_id + time_bucket 查询模式。

热点活动缓解策略：非常热的活动可能会固定单个计数器分片。缓解方案：进一步分片计数器（campaign_id + region_id 个子计数器），然后再聚合。

一致性模型

层级 一致性 原因
摄入日志 持久化、分区、至少一次 基础；不丢失任何事件。
热点计数器 近似；近实时更新 预算上限执行。
冷路径计费账本 通过去重实现精确一次语义 资金相关。
欺诈评分 最终一致 异步 ML。
聚合报表 最终一致 每小时/每日汇总。

多区域策略

  • 在用户所在区域摄入点击事件。
  • 各区域的流汇入全局 Kafka 集群，或区域 Kafka 向中心节点复制数据，供冷路径使用。
  • 热路径计数器可以是区域级的（配合跨区域对账），也可以是全局的（配合分片活动）。
  • 计费集中进行：唯一权威账本。

故障模式与缓解措施

故障模式 缓解措施 监控指标
摄入时丢失点击事件。 Kafka 持久化写入；广告服务器使用相同的 request_id. 摄入错误率，日志延迟。
同一次点击被计算两次（重试、重复 request_id）。 Bloom filter（热路径），精确去重（冷路径）。 重复率；计数器漂移。
预算上限大幅超出。 预测性截止；节奏控制；更快地向广告服务器传播状态。 每个活动的超预算百分比。
欺诈在计费前未被检测到。 在写入权威账本前运行欺诈过滤；允许在下一周期进行追溯性撤销。 欺诈检测延迟；冲销量。
延迟事件在账单关闭后才到达。 在下一个账单周期生成调整记录。 延迟事件率；调整量。
流处理器延迟。 自动扩容；对延迟设置告警；从日志回填。 消费者积压。
计数器 KV 热分片。 按地区或子键进一步分片。 每分片 QPS。
对账发现大量漂移。 排查；通过权威路径修正。 每周期漂移百分比。
突发流量期间地区下线。 在备用地区缓冲；恢复后对账。 各地区的摄入速率。

运维指标

正确性：

  • 热计数器与冷账本的漂移（目标：每小时检查点时漂移率在几个百分点以内）。
  • 重复检测率。
  • 欺诈检测率与误报率。
  • 每个活动的超预算百分比。

容量：

  • 各地区的摄入 QPS。
  • 流消费者延迟。
  • 计数器存储的 QPS 和延迟。

业务层面：

  • 各广告系列/广告的点击量。
  • 各广告系列的费用；营收。
  • 转化率（若包含归因）。
  • 广告主仪表盘数据新鲜度。

需要指出的权衡

  • Lambda 架构需要维护两条流水线；Kappa 更简单，但需要更长的重处理时间。
  • 近似热路径速度快，但可能导致预算超投；严格执行则需要更快的传播速度。
  • 按地区拆分预算可避免竞争，但在某地区繁忙时可能浪费预算。
  • 账单后的欺诈过滤需要冲销；账单前过滤会增加延迟。
  • Bloom filter 去重开销低，但会放过少量重复请求。
  • 按分钟聚合数据更新鲜，但增加存储压力；按小时聚合效率更高。

常见陷阱与更优做法

陷阱 更优做法
一条流水线试图同时做到快速和权威。 拆分热路径和冷路径；进行对账。
在点击时同步进行欺诈检查。 异步欺诈检测，事后修正。
将预算上限作为硬性"到 100% 即停止"规则。 考虑在途请求，在 95% 时预测性截止。
无分片的全局计数器。 按地区或子广告系列分片。
信任客户端的点击 ID。 服务端 click_id 加签名 request_id 来自广告服务器。
忽略延迟事件。 水位线；调整周期；重处理预算。

面试官可能深入探讨的方向

深入方向 优秀回答要点
预算上限执行 预测性截止；按地区拆分；快速计数器；广告服务器缓存。
去重策略 热路径使用 Bloom filter；冷路径对 request_id 使用精确集合。
欺诈过滤 异步 ML 流水线；规则加评分；事后冲销。
延迟事件 水位线；窗口延迟后才最终化；调整记录。
对账 每小时比对；漂移告警；冷账本为事实来源。
多区域 地区摄入；中央账单；按地区分配预算。

Staff 级别回答检查清单

  • 两条流水线：热路径（快速、近似）和冷路径（慢速、精确）。
  • 服务端 click_id；签名 request_id；冷路径中的精确去重。
  • 带在途余量的预测性预算截止。
  • 带事后冲销的异步欺诈过滤。
  • 用于延迟事件的水位线；调整周期。
  • 计数器与账本之间的对账。
  • 多地区摄入，集中账单。
  • 热路径用 Bloom filter 去重；冷路径用精确集合。

面试用语

广告点击聚合器是一个经典的 Lambda 架构问题。热路径快速近似，用于预算执行和仪表盘；冷路径慢速精确，用于账单。两者定期对账。最难的正确性问题是突发流量下的预算超投；最难的扩展性问题是每天数十亿事件的去重。

2. 模拟面试

面试官： 设计一个广告点击聚合器。

候选人： 我将在同一点击流上设计两条流水线：热路径实时更新各广告系列计数器，用于预算执行和仪表盘；冷路径生成带精确去重和欺诈过滤的权威账单账本。不变量是：每个可计费点击在账本中被精确计数一次，且预算上限的执行速度足以防止大额超支。

面试官： 当预算耗尽时，如何停止投放广告？

候选人： 热路径流处理器在每次点击时更新快速 KV 存储中的各广告系列计数器。当计数器达到预算的 95%——考虑在途点击——广告系列状态翻转为 over_budget 并将事件推送至广告服务器缓存。广告服务器在每次投放请求时检查状态；客户端每隔约一秒刷新本地缓存。因此存在几秒的窗口期会发生少量超投，这是可以接受的。

面试官： 为什么是 95%，而不是 100%？

候选人： 预测性截止。"计数器读到 100%"与"广告服务器停止竞价"之间始终存在一个时间窗口。对于热门广告系列，10k 点击/秒的情况下，即使是 1 秒的窗口也意味着 10k 个额外点击。将截止点设为 95% 可为在途负载预留缓冲。具体百分比取决于点击率和传播延迟。

面试官： 如何去重？

候选人： 两层机制。广告服务器在点击 URL 中包含一个签名的 request_id ——同一次展示的每次点击共享相同的 request_id。在热路径中，每个分区的 Bloom filter 可在近期窗口内拒绝重复请求；速度快，但偶尔会有漏网之鱼，这没问题，因为冷路径会修正。在冷路径中，以 request_id 为键的精确去重集合是权威的去重器。账单账本只记录首次出现；后续重复项被标记。

面试官： 欺诈——如何过滤？

候选人： 异步欺诈服务消费点击流。它运行规则（IP 封锁列表、机器人 User-Agent、每设备点击率）和 ML 评分。每个 click_id 的欺诈评分写入一张侧表。冷路径在决定账单状态时使用此评分。热路径只应用最快的规则以避免拖慢速度；热路径中的少量误计事后通过对账修正。

面试官： 延迟到达的点击怎么处理？

候选人： 流处理器使用水位线。我们声明"已收到截至时间 T 的事件"，更旧的事件会被处理，但进入"调整"路径以更新过去的聚合结果。冷路径在挂钟时间过去约一小时后才最终确定该小时的账单记录，为延迟事件留出到达时间。真正迟到的事件计入下一周期的调整记录。

面试官： 冷路径如何与热路径对账？

候选人： 每小时，一个对账器按广告系列比较热计数器总量与冷账本总量。若漂移在容忍范围内（几个百分点），则接受并更新权威的 budget_spent。若漂移超过阈值，则告警并排查。冷账本始终是开票的事实来源；热计数器仅用于运营。

面试官： 某广告系列有 100 万美元预算且点击率极高，如何避免单个分片成为瓶颈？

候选人： 按地区分配预算。我们将 100 万美元按预期流量比例拆分到各地区——例如美国 30 万、欧盟 20 万等。每个地区有自己的计数器分片。耗尽本地分配的地区可通过较慢的协调服务向其他地区申请转移剩余预算。这样既避免了全局计数器竞争，又保留了整体预算控制。

面试官： 多区域？

候选人： 点击事件在其发源地区完成摄入。每个地区有 Kafka 和流处理器。热计数器是区域性的，跨地区对账。冷路径从中央 Kafka（跨地区复制）消费，写入一个权威账本。这使热路径延迟保持较低，同时保持账单的集中管理。

面试官： Kafka 短暂宕机怎么办？

候选人： 广告服务器摄入端使用小型有界队列本地缓冲，若 Kafka 不可达时间过长则丢弃数据。在灾难性故障期间，我们接受一小部分点击丢失。广告服务器也可以回退到另一地区的备用摄入通道。我们监控点击丢失率并设置告警。

面试官： 仪表盘——如何提供服务？

候选人： 流处理器在查询存储（例如时序数据库或列式存储）中维护各广告系列的每分钟聚合数据。仪表盘直接从中读取，延迟约 30-60 秒。对于长时间范围的查询，我们使用冷路径生成的预聚合每小时和每日汇总数据。

面试官： 总结。

候选人： Lambda 架构。热路径用于预算执行和仪表盘，冷路径用于权威账单。签名的 request_id 来自广告服务器支持去重；热路径用 Bloom filter，冷路径用精确集合。预测性截止在 95% 可防止大多数超投。欺诈过滤为异步，事后修正。延迟事件通过水位线和调整周期处理。按地区拆分预算可避免热分片竞争。每小时对账验证热/冷路径的收敛性。

3. 练习备注

五分钟版本

  1. 明确：预算上限要快；账单要精确；欺诈过滤；延迟事件处理。
  2. 不变量：可计费点击在账本中被精确计数一次；上限以近实时方式执行。
  3. Lambda 架构：热路径 + 冷路径。
  4. 服务端 click_id + 签名 request_id 用于去重。
  5. 热路径：流处理器 + 计数器 KV；在 95% 时预测性截止。
  6. 冷路径：精确去重、欺诈过滤、账单账本。
  7. 用水位线处理延迟事件；路径间对账。
  8. 热门广告系列按地区分配预算；集中账单。

一分钟结尾陈述

广告点击聚合器分为两条流水线，处理同一点击流。热路径将各广告系列计数器的刷新速度控制在五秒内以执行预算上限；冷路径生成带精确去重和欺诈过滤的权威账单账本。两者每小时对账。最难的正确性问题是防止突发流量下的大额超投，通过预测性截止和按地区分配预算解决。最难的扩展性问题是对每天数十亿事件进行精确去重，通过热路径 Bloom filter 和以签名 request_id 为键的冷路径精确集合解决。

最佳面试用语

  • "两条路径：热路径追求速度，冷路径追求正确性。对账保证两者的一致性。"
  • "预算执行是预测性的，而非被动反应式的——在 95% 时停止，以应对在途点击。"
  • "去重：热路径使用 Bloom filter，冷路径使用以 request_id 在冷路径中。"
  • "欺诈过滤为异步，在账单账本中进行事后冲销。"
  • "按地区分配预算是 1000 万美元广告系列不让单个计数器分片成为瓶颈的方法。"
