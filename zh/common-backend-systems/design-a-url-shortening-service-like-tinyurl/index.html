1. 基础知识与解题思路

本题考察点

URL 缩短器看似简单——接受一个长 URL，返回一个短码，查询时重定向。难点在于如何在大规模场景下保证系统的正确性与高性能。

其不变量如下：

每个短码在其整个生命周期内唯一映射到一个规范的长 URL，同一个长 URL 可以被多个不同的短码引用；查询必须廉价、幂等，并在全局范围内保持足够的一致性，确保不会有两个创建者获得相同的短码。

在 ICT5 / L6+ 级别，面试官考察的是你能否选择一种在大规模场景下避免冲突的短码生成策略，将写路径与以读为主的重定向路径分离，并说明如何在不拖慢热重定向的前提下叠加分析、防滥用和链接过期功能。

60 秒面试框架

口述版本：

我将围绕两条路径设计 TinyURL：一条是分配全局唯一短码的低流量创建路径，以及一条执行单次 key 查询的极高流量重定向路径。关键决策在于：如何在不发生冲突的情况下生成短码，如何缓存重定向路径，以及如何保护系统免受恶意软件和分析驱动的热点 key 影响。我将依次阐明需求、定义数据模型、梳理写路径和读路径，然后深入探讨短码生成和缓存策略。

首先明确需求

功能性需求：

  • 用户可以提交一个长 URL 并获得一个短码。
  • 用户可以选择请求自定义别名。
  • 访问短 URL 会重定向到对应的长 URL。
  • 用户可以查看其链接的基本分析数据：总点击量、近期点击、地理分布。
  • 链接可以选择设置过期时间或一次性使用标志。
  • 管理员可以禁用恶意链接。

非功能性需求：

  • 在边缘节点的重定向延迟远低于 100 ms。
  • 不会有两个创建者针对不同的长 URL 获得相同的短码。
  • 同一个长 URL 提交两次可能产生不同的短码；去重是可选的。
  • 即使创建路径降级，重定向路径也必须保持正常运行。
  • 分析数据可以滞后数秒到数分钟；重定向不能滞后。
  • 可审计性：记录谁在何时创建了哪个链接，用于滥用审查。

除非面试官要求，否则超出范围：

  • 品牌专属域名与租户级白标能力。
  • 移动应用的深度链接路由。
  • A/B 测试或动态目标地址。
  • 完整的 link-in-bio 个人主页。

规模假设

大声说出假设条件，让设计有具体的形态：

领域 假设
写操作 峰值每天新增 1 亿条链接，平均约 1k/s，峰值 10k/s。
读操作 写入量的 100 到 1000 倍；全球峰值约 100k–1M 次重定向/秒。
对象大小 长 URL 最多约 2 KB；短码 6–8 个字符。
留存期 默认 5 年；部分链接永不过期。
热点 key 病毒式链接可能主导流量；单个 URL 可能在数小时内接收大部分流量。
Latency（延迟） 边缘节点重定向 p99 低于 50 ms；创建 p99 低于 300 ms。
一致性 短码分配需强一致性，分析数据可最终一致。

瓶颈不在写入吞吐量，而在短码生成的全局唯一性保证以及病毒式链接的读侧缓存命中率。

核心产品模型

从三个独立界面来考量：

界面 用户体验 系统需求
创建 提交长 URL，获取短码。 短码分配需强一致性，写入量低，客户端重试时幂等。
重定向 访问短 URL，跳转到长 URL。 极致读取规模，可缓存，必须在创建中断期间以只读方式运行。
分析 查看点击量和趋势。 最终一致性，重度聚合，与重定向热路径分离。

最常见的设计错误是将分析或特性标志放在同步重定向路径上。重定向应仅是一次单 key 查询加上一个 301/302 响应。

核心组件

组件 职责 关键细节
边缘 / CDN 对热门短码缓存 301/302 响应；吸收 DDoS 流量。 病毒式热门链接应极少到达源站。
API 网关 在创建路径上进行身份验证、速率限制和滥用检查。 匿名创建可以允许，但须受到严格限制。
创建服务 验证长 URL，分配短码，写入链接存储。 幂等，基于 Idempotency-Key 头部选择。
短码分配器 生成一个全局唯一的短码。 关键的正确性组件。基于计数器、哈希或预分配池。
链接存储 权威的 short_code -> long_url 映射及元数据。 以读为主，key-value 或列式存储。
重定向服务 将短码解析为长 URL 并发出点击事件。 无状态；水平可扩展；先读缓存，再读存储。
KV 缓存 针对热点工作集的内存 LRU 缓存。 短码呈现长尾分布和胖头部——缓存非常有效。
Click stream 用于点击事件的异步管道：时间戳、短码、地理位置、来源页、代理信息。 永远不阻塞重定向响应。
分析服务 聚合点击数据，用于仪表盘和导出。 OLAP 存储，按小时/天进行汇总。
滥用与安全 对新创建和查询时的 URL 进行黑名单扫描。 创建时异步处理，重定向服务查询拒绝名单。

数据模型

Link

Link(
  short_code,         // primary key, 6-8 chars base62
  long_url,
  owner_user_id,      // nullable for anonymous
  created_at,
  expires_at,         // nullable
  status,             // active, disabled, expired
  is_custom_alias,    // bool
  click_count_approx, // denormalized counter, eventually consistent
  safety_status,      // unknown, safe, flagged, blocked
  version
)

short_code 是分区键和主键。该存储不需要针对 long_url 的二级索引，除非需要去重。

短码分配记录

针对基于计数器的分配器：

CounterRange(
  region_id,
  worker_id,
  range_start,
  range_end,
  claimed_at
)

单个计数器不需要在每次写入时保持全局同步；每个创建节点可预先保留一段整数范围，然后在本地转换为 base62 编码。

点击事件（仅追加）

ClickEvent(
  event_id,
  short_code,
  ts,
  ip_hash,
  geo_country,
  referrer,
  user_agent_family
)

滥用决策

SafetyDecision(
  short_code,
  decided_at,
  source,             // scanner_v1, manual_review, partner_feed
  status,             // safe, suspicious, blocked
  reason_code
)

API

幂等性：对同一用户的相同 Idempotency-Key 请求返回相同的 short_code。这使客户端重试安全，并防止用户在感知上只有一次创建时却分配了两个短码。

短码生成策略

这是核心深度探讨。三种可行方案：

方案 A：计数器 + base62 编码

  • 一个中央或分片计数器在每次创建时递增。
  • 将整数编码为 base62（0-9 a-z A-Z）；6 字符短码可覆盖约 560 亿个；7 字符可覆盖约 3.5 万亿个。
  • 为避免竞争，每个创建节点预先认领一段范围（例如 10,000 个整数），在本地用完之前持续使用。
  • 范围认领是唯一的同步步骤；使用事务性计数器或 ZooKeeper 风格的 sequencer。

优点：确定性，无冲突，短码长度增长可预测。缺点：短码可被猜测和枚举；竞争对手和爬虫可按顺序抓取。

方案 B：随机 + 冲突检查

  • 生成 7–8 个随机 base62 字符。
  • 条件插入： INSERT IF NOT EXISTS short_code = ?.
  • 发生冲突时，生成新的随机短码并重试。

优点：短码不可猜测；无需协调。缺点：随着 keyspace 填满，冲突率上升；受生日悖论约束；需要条件写入。

方案 C：对 (long_url, salt) 进行截断哈希

  • 使用 SHA-256 哈希，截取为 7–8 个 base62 字符。
  • 相同的长 URL 加相同的盐值生成相同的短码，因此提供自动去重。
  • 冲突仍然可能发生；通过额外位或计数器后缀解决。

优点：对同一提交者的相同长 URL 内置去重；无需中央计数器。缺点：冲突处理较繁琐；不同用户提交相同长 URL 可能得到相同短码，从而泄露其他用户链接的信息。

面试用语：

我默认会采用带范围认领的计数器方案以保证可预测性，同时加入随机混淆以掩盖序列，防止爬取时暴露活跃短码。如果产品要求短码不可猜测，则切换为随机方案加条件插入，并接受重试开销。

写路径：创建链接

  1. 客户端 POST /links 与 Idempotency-Key.
  2. 网关在长 URL 主机上执行身份验证、速率限制和黑名单检查。
  3. 创建服务检查幂等性表；如已存在，返回之前的 short_code.
  4. 分配短码：如果本地范围内有整数，取下一个并编码；如果已耗尽，从全局分配器认领新范围。
  5. 插入 (short_code, long_url, owner, ...) 写入链接存储。对于随机短码，使用条件插入，冲突时重试。
  6. 写入幂等性记录。
  7. 异步将新链接推送给安全扫描器。
  8. 返回 short_code 以及完整的 short_url.

同步关键路径仅为一次计数器递增加上一次插入，均为 O(1)。

读路径：重定向

  1. 客户端访问 https://short.example/abc123.
  2. 边缘 CDN 检查其缓存的重定向。命中时，立即返回 301 并发出边缘点击事件。
  3. 未命中时，请求区域重定向服务。
  4. 重定向服务检查 KV 缓存。
  5. 缓存未命中时，从链接存储读取。
  6. 检查 status 。 safety_status。如果已禁用或被封锁，返回安全拦截页，而非重定向。
  7. 响应 301 Location: <long_url> （或 302 以强制对每次访问进行分析统计）。
  8. 异步发送点击事件到流。

响应等待仅为一次缓存查询，点击写入完全异步。

301 vs 302：

编码 行为 分析影响
301 永久重定向 浏览器和中间件积极缓存，延迟最低。 仅统计首次点击；同一客户端后续点击可能绕过源站。
302 临时重定向 默认不缓存。 每次点击都到达源站并被统计。

大多数生产短链接服务选择 302 以保证分析数据的准确性，并容忍边缘缓存的损失。如果系统两者都需要，可以使用带有较短 Cache-Control: max-age=N 并接受有界的分析损失。

缓存策略

重定向工作负载具有极端倾斜：少量链接产生绝大多数流量。

  • 多层缓存：浏览器 HTTP 缓存、CDN 边缘缓存、区域 KV 缓存、进程内缓存。
  • 缓存键是短码，缓存值是长 URL 加状态和安全标志。
  • TTL 较长（数小时到数天），因为映射是不可变的。唯一需要失效的情况是状态变更（禁用、过期、安全封锁）和更新。
  • 状态变更时，向每个缓存节点监听的扇出通道发布失效事件。
  • 对于病毒式链接，在检测到突发流量时预热缓存。

面试用语：

此处缓存异常有效，因为映射在通常情况下是不可变的。有趣的问题是当链接因安全原因被禁用时的失效处理，该操作应在数秒内全局生效。

扩展与分区

数据 分区键 原因
链接存储 short_code （哈希分区） 均匀分布；是天然的查询键。
计数器范围认领 单张小表，条件更新 写入量低，但必须具有事务性。
点击事件 short_code 用于短窗口聚合，然后按日期进行每日汇总 病毒式链接的热点 key 分散采用采样或子分区。
分析汇总 short_code + bucket 预聚合计数器。
拥有者索引 owner_user_id 二级索引或单独的表 用于"我的链接"查询。

热点 key 处理：

  • 对于每分钟产生数百万次重定向的病毒式链接，在靠近用户的位置缓存并从 CDN 提供服务。
  • 对点击事件进行采样，避免压垮点击管道；采样事件在聚合时按比例放大。
  • 在边缘节点执行每 IP 和每 user-agent 速率限制。

一致性模型

层级 一致性 原因
短码分配 强一致性（在分配器内部） 防止重复短码。
链接存储插入 强主键条件写入 防止短码重用。
缓存 最终一致性，附带显式失效事件 对不可变映射可接受延迟；安全更新需要快速传播。
点击计数器 最终一致 仪表盘刷新频率为每分钟或每小时。
安全决策 创建时最终一致，决策后查询时强一致 被判定为恶意的链接必须快速停止重定向。

过期与生命周期

  • 链接可以设置 expires_at。重定向服务在每次从存储中读取未命中时检查该字段；缓存记录包含过期时间以便自动失效。
  • 用户请求时软删除：设置 status = disabled，广播失效事件。
  • 过期链接的垃圾回收作为后台慢作业运行；行可以无限期保留，但标记为已过期。
  • 一次性链接：重定向服务在响应前执行条件 UPDATE status FROM active TO used ；第二次访问者看到"链接已被使用"页面。

多区域策略

重定向路径以读为主，受益于跨区域的主动-主动部署。

  • 全局异步复制链接存储。副本读取存在滞后对重定向是可接受的，因为映射在创建后是不可变的。
  • 短码分配：每个区域认领独立的计数器范围（例如区域标记的前缀或交错的整数范围），使创建操作可在本地完成，无需跨区域协调。
  • 针对 disabled 或 blocked 短码的失效事件通过全局 pub/sub 通道广播，端到端延迟目标较低（秒级，而非分钟级）。
  • 故障转移：如果某区域的存储不可用，重定向流量从缓存继续服务；创建操作失败或回退到对等区域。

如果主动-主动方案导致自定义别名出现问题（两个用户在不同区域选择了相同的别名），可将自定义别名创建路由到单一区域或对每个别名加全局锁。

滥用、安全与隐私

URL 缩短器被武器化用于网络钓鱼和恶意软件传播。

控制措施：

  • 在创建时对提交的主机进行黑名单检查（已知钓鱼域名、被制裁的主机）。
  • 创建后异步进行更深入扫描：抓取目标页面，在沙箱中渲染，进行分类。
  • 对提交者的信誉评分（账号注册时长、历史滥用记录）。
  • 对匿名创建进行积极的速率限制。
  • 对尚未完成扫描验证的新链接显示拦截警告页。
  • 针对用户举报的人工审核队列。
  • 快速全局下架：每个重定向节点查询的拒绝名单；基于 Redis 的布隆过滤器用于低延迟的否定缓存；添加时显式触发失效。

隐私：

  • 在存储点击事件之前，对 IP 地址进行哈希处理。
  • 遵守 Do Not Track 信号及各地区的隐私法规，用于分析目的。
  • 点击流仅限所有者访问，不得将点击信息泄露给第三方。

故障模式与缓解措施

故障模式 缓解措施 监控指标
两个创建者获得了相同的短码。 对以下字段进行条件插入 short_code；范围分配器使用事务操作；随机方案下发生冲突时进行重试。 插入冲突率，重复短码告警。
短码生成竞争风暴。 每节点预申请范围；无锁分配器；多个并行序列生成器。 分配器 p99 延迟，范围申请错误率。
缓存投毒与恶意重定向。 对缓存条目进行签名；失效操作须通过存储层写入。 缓存与存储之间的不一致，未授权的失效事件。
爆红链接压垮源站。 激进的边缘缓存；以 301 缓存并设置 TTL；在边缘限流。 每个短码的源站 QPS，边缘缓存命中率，Top-N 热点短码。
点击处理管道积压并丢失事件。 采样摄入；持久化缓冲区；从日志重放。 管道延迟，丢弃数量，重放覆盖率。
钓鱼链接屏蔽不够及时。 全局拒绝列表配合 pub/sub 失效通知，支持负向缓存。 从举报到加入黑名单的时间，误报率。
存储层热分区，发生在 short_code. 对键空间进行哈希分区；预先分裂范围；检查顺序短码倾斜问题。 每分区 QPS 分布，按分区统计的 p99。
跨 Region 的计数器 Region 申请冲突。 带 Region 标签的前缀位或不重叠的整数范围。 每对 Region 的重复短码率。
分析仪表盘显示数据错误。 幂等聚合；定期与原始事件存储进行核对。 实时仪表盘与重新计算之间每日总数的偏差。

运维指标

正确性指标：

  • 重复短码：必须为零。
  • 由于冲突导致的插入失败（计数器方式应极低，随机方式应有上界）。
  • 带有过时安全状态的重定向响应。
  • 已禁用的链接仍在重定向（缓存失效延迟）。

容量指标：

  • 按 Region 统计的重定向 QPS，百分位延迟。
  • 总体及 Top-100 短码的边缘缓存命中率。
  • 分配器范围申请速率与竞争情况。
  • 点击处理管道延迟。

业务指标：

  • 每日创建量，已创建链接的留存率，分享率。
  • 滥用处置：从举报到屏蔽的时间。
  • 针对每个创建者的异常检测（突发大量提交）。

需要指出的权衡

  • 基于计数器的短码可预测性强、易于实现；但会向攻击者泄露链接总量信息。
  • 随机短码不可猜测，但需要处理冲突并浪费键空间。
  • 301 重定向速度快，但会丢失单次点击分析；302 每次点击均可捕获，但每次都会产生源站访问开销。
  • 边缘缓存加速重定向，但会拖慢安全处置速度；必须使用显式失效机制。
  • 对相同长 URL 进行去重可节省空间，但会将无关用户耦合在一起；通常不值得这样做。
  • 多 Region 主动-主动架构对读操作容易实现；对于自定义别名的创建则存在风险。

常见陷阱与更优做法

陷阱 更优做法
所有创建操作共用单个全局原子计数器。 范围申请分配器：将协调开销摊销到数千次创建操作中。
将点击计数存储在与链接映射相同的行中。 独立的计数器存储；从事件流中聚合。
将缓存视为数据来源。 缓存是提示；存储层才是权威来源。失效操作须显式触发。
在重定向路径中忽略了安全状态。 每个缓存条目包含安全状态；下线操作会发布失效通知。
使用 base64 而非 base62。 URL 安全字符集很重要：避免使用 +, /，以及 = 填充字符。
允许自定义别名与自动生成的别名发生冲突。 独立的命名空间或保留前缀；或在创建时同时检查两个空间。

Staff 级别回答检查清单

  • 从不变量出发：唯一短码，不可变映射。
  • 区分创建路径（低频，强一致性）和重定向路径（极高读量，可缓存）。
  • 选择短码生成方案并说明理由。
  • 展示缓存层级及失效机制的工作原理。
  • 解释幂等创建。
  • 解释滥用处理与安全下线。
  • 说明多 Region 如何支持读操作，以及为何自定义别名更加复杂。
  • 将分析作为非阻塞的旁路。
  • 将爆红链接识别为热键风险并加以应对。

面试用语

TinyURL 的关键在于认识到负载的不对称性。创建路径频率低，需要在唯一性方面保证正确性；重定向路径占绝大多数流量，本质上是一次缓存键查询。其余所有内容，包括分析和安全扫描，都以异步方式挂载在重定向路径上。

2. 模拟面试

面试官： 设计 TinyURL。

候选人： 我会围绕两条截然不同的路径进行设计：一条是低频的创建路径，用于分配全局唯一的短码；另一条是极高频的重定向路径，本质上是一次键查询。核心不变量是：每个短码永久唯一映射到一个规范的长 URL。

面试官： 你的规模假设是什么？

候选人： 我假设每天约 1 亿次创建，峰值约每秒 10,000 次，但读操作是创建的 100 到 1000 倍，全球范围内每秒最高可达 100 万次重定向。大部分流量集中在少数几个爆红链接上。

面试官： 你如何生成短码？

候选人： 我会使用带范围预申请的基于计数器的分配器。每个创建节点从事务性计数器中预留约 10,000 个整数，然后在本地将其编码为 base62。这样同步协调的频率变为每一万次创建一次范围申请，而非每次创建都协调。如果不可猜测性很重要，我会切换为随机 7 字符短码加条件插入，接受因冲突而产生少量重试。

面试官： 短码应该多长？

候选人： base62 加 7 个字符可提供约 3.5 万亿个短码，以每天 1 亿次的速度足以使用数十年。6 个字符约覆盖 560 亿个，大约可以用二十年。我会从 7 位开始，并保留扩展的选项。

面试官： 重定向路径是什么样的？

候选人： 客户端在边缘节点访问短 URL。CDN 可能缓存了一个 301 响应；如果有，则直接返回。未命中时，请求到达区域重定向服务，该服务检查 KV 缓存，未命中则回退到链接存储。响应是重定向到长 URL，同时以"发出即忘"的方式将点击事件发送到异步流。

面试官： 301 还是 302？

候选人： 这是一个真实的权衡。301 是永久重定向，会被浏览器和中间件激进地缓存，延迟最低，但重复点击会绕过源站，导致分析数据失真。302 默认不被缓存，可确保每次点击都被统计，但会将负载推向源站。大多数生产级短链接系统选择 302 以保证分析准确性，并通过重定向层的缓存来承担开销。我会选择 302 加强缓存层。

面试官： 一个链接爆红，每秒 100,000 次访问。系统如何应对？

候选人： 边缘缓存可以吸收大部分流量，因为映射是不可变的。每个 CDN PoP 为热点短码存储一个带短 TTL 的 301 或 302 响应；源站看到的 QPS 可能只有几千，而非 100,000。我们监控 Top-N 热点短码，并在爆发检测触发时预热区域缓存。点击事件流对热点短码采用采样，并在聚合时用缩放因子修正，使分析管道不会积压。

面试官： 一个链接被举报为钓鱼链接。你能多快停止重定向？

候选人： 使用带 pub/sub 失效通知的全局拒绝列表。下线操作将状态写入链接存储，发布失效事件，每个缓存节点和 CDN POP 在几秒内收到通知。每个缓存条目携带安全状态，因此在失效后的下一次请求时会显示插屏页面。全球范围内控制在五秒以内是可以实现的目标。

面试官： 自定义别名怎么处理？

候选人： 自定义别名在 short_code 列上使用条件插入。如果别名已被占用，则返回明确的冲突提示。自定义别名与自动生成的短码共享同一命名空间，因此我会保留一个前缀或字符模式，防止分配器生成的短码恰好与用户想要的别名重名。在多 Region 场景下，自定义别名的创建操作须经过单一所有者 Region，或使用全局锁，以避免两个 Region 申请同一别名。

面试官： 幂等创建是如何工作的？

候选人： 客户端发送一个 Idempotency-Key 请求头。创建服务维护一张以用户加密钥为键的幂等表。如果收到重复请求，则返回之前已分配的短码，而不是重新分配。这使客户端重试变得安全，并防止用户对同一次创建操作获得两个不同的短码。

面试官： 你如何对链接存储进行分区？

候选人： 对以下字段进行哈希分区 short_code。查询模式是对键的精确匹配，因此哈希分区可以实现均匀分布。如果使用范围分区，计数器生成的顺序短码会产生热点分区，但哈希可以消除这一问题。所有者查询的二级索引存放在独立的每所有者表中。

面试官： 某个 Region 宕机怎么办？

候选人： 重定向可从缓存和任意健康 Region 的副本继续服务。创建操作失败或被路由到对等 Region。由于每个 Region 预先分配了自己的计数器范围，或使用带 Region 标签的前缀位，在中断期间不存在两个 Region 生成相同短码的风险。当 Region 恢复后，副本完成追赶，并通过 pub/sub 通道重放所有安全失效通知。

面试官： 你如何在大规模下处理点击事件管道？

候选人： 点击事件写入高吞吐量的追加-only 日志。对于热点短码，边缘以已知速率对事件进行采样，聚合管道将结果乘以采样因子。管道将事件批量写入存储在分析存储中的小时和每日汇总数据。重定向路径不会因写入点击事件而阻塞；即使管道降级，重定向仍然正常工作。

面试官： 总结。

候选人： 两条路径：低频的创建操作必须生成唯一短码，高频的重定向操作是键查询。范围申请计数器分配器在无需每次写入都协调的情况下保证唯一性。重定向路径由缓存主导；安全下线通过 pub/sub 失效通知实现。分析是异步的，热键采用采样。自定义别名和多 Region 创建需要仔细处理冲突，但其他一切都可以水平扩展。

3. 练习备注

五分钟版本

  1. 明确范围：短码、重定向、可选的分析、滥用处理。
  2. 声明不变量：唯一的不可变映射。
  3. 选择短码生成方案：带范围申请的计数器，base62，7 位字符。
  4. 画出两条路径：创建路径（低频，强一致性）和重定向路径（高频，缓存）。
  5. 解释幂等创建和条件插入。
  6. 解释边缘缓存、KV 缓存、存储层。
  7. 解释带 pub/sub 失效通知的安全下线。
  8. 以多 Region 结尾（主动-主动读取，带 Region 标签的短码范围）。

一分钟结尾陈述

TinyURL 的负载是不对称的：低频创建需要正确性，高频重定向需要速度。我会使用范围申请计数器生成唯一的 base62 短码，利用映射不可变性激进缓存，将点击管道与重定向热路径分离，并使用 pub/sub 失效通知实现快速安全下线。主要权衡是 301 vs 302：我会选择 302 以保证分析准确性，并用缓存来承担开销。

最佳面试用语

  • "映射是不可变的；这正是缓存如此高效的原因。"
  • "我不会在每次创建时都进行协调；我预申请一个范围并在本地服务。"
  • "重定向路径是一次单键查询。包括分析在内的其他一切都是异步的。"
  • "安全下线是过时缓存真正重要的唯一场景；pub/sub 失效通知为我提供了几秒级的 SLA。"
  • "自定义别名和多 Region 创建是简单设计被打破的地方；其他一切都很容易。"
