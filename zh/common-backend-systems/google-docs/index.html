Google Docs — 协作编辑（详解）

2026/06/01 · Common Backend Systems
目录
目录

难度： Hard · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 设计 Google Docs：最多约 50 人同时编辑同一文档，每人逐键输入，所有人能在一秒内看到彼此的光标和编辑内容。离线编辑在重新连接后同步。每日触达 1 亿个文档，最繁忙的文档峰值约有 100 名协作者。

1. 需求（约 5 分钟）

功能需求 - 多用户实时编辑同一文档。 - 实时看到其他用户的光标和编辑内容。 - 文档具有持久性；重新加载后内容一致。 - 离线编辑在重新连接后自动同步，无需手动解决冲突。

我会推迟讨论评论、建议模式、版本历史深度、共享/ACL 细节以及富嵌入对象等功能。

非功能需求 - 实时更新：输入后其他客户端在 500 ms 内可见。 - 强收敛性——对于给定的一组编辑，所有参与者最终看到相同的最终文档。 - 支持离线编辑；优雅合并。 - 持久性——绝不丢失已提交的编辑。

容量估算 - 每日触达 1 亿个文档；并发活跃会话约 1000 万。 - 热门文档：约 100 名并发编辑者 → 100 × 约 5 ops/s = 每文档 500 ops/s。 - 总操作量：由长尾主导；全球假设每日 10 亿次操作。 - Op log 存储适中但对延迟敏感：10 亿 ops/天 × 约 150 B/op × RF=3 ≈ 每日 450 GB。对于 1 亿个触达文档，每份快照约 50 KB，RF 前约 5 TB；快照/对象存储是主要的持久字节成本。

2. 核心实体（约 2 分钟）

  • 文档 — doc_id，所有者、ACL、当前快照、版本。
  • Op — 编辑操作（带位置 + 内容 + 作者 + 逻辑时间戳的插入/删除）。
  • Session（会话模式） — session_id, doc_id, user_id, cursor_position.
  • 快照 — 文档状态的定期物化，用于快速加载。

3. API / 协议（约 5 分钟）

WS  /v1/docs/{doc_id}/sync
  ← client: { op, base_version }
  → server: { op, version, author }    // op stream from peers
  → server: { presence, cursor }       // presence beacons

HTTP:
POST /v1/docs                              → { doc_id }
GET  /v1/docs/{doc_id}/snapshot            → snapshot + version

通过 token 识别用户；每个 op 携带 doc_id 和 base_version 用于排序。

服务器为每个文档发放单调递增的版本号。重连游标编码 {doc_id, last_acked_version, client_session_id}；presence 无游标，因为它是尽力而为的。

5. 高层设计（约 10–15 分钟）

这个问题的核心是 冲突解决。两种可行模型：

操作转换（OT）

  • 每个 op 引用一个 base version。
  • 服务器持有权威序列；当新 op 基于旧版本到达时，服务器在应用前将其与中间 op 进行变换。
  • 客户端同样将接收到的 op 与本地在途 op 进行变换。

Conflict-free Replicated Data Types (CRDT)

  • 每个字符拥有唯一的全局有序 ID（Logoot、RGA、Yjs）。
  • 插入操作指定"在哪个字符之后"；删除操作引用字符 ID。
  • 可交换——任意顺序应用 op 结果相同，无需变换。

我默认选择 OT 用于面试，因为它与 Google Docs 实际使用的方式一致，且以服务器为权威的模型更易解释；提及 CRDT 作为 Figma 和 Yjs 所采用的替代方案。

系统架构

  1. 编辑会话路由器。 Hash doc_id 到某个 session server（单一所有者模式）。同一文档的所有编辑汇聚到一个 server 进行排序，这是核心所在。
  2. Session server。 在内存中维护权威 op log，并为每个连接的客户端维护一个流式队列。对传入 op 应用 OT，将变换后的 op 广播给其他参与者，并将 op 持久化到持久存储。
  3. Op log 存储。 仅追加——以 (doc_id, seq)为键的 Cassandra/Spanner。为持久性而做副本。
  4. 快照服务。 每 N 个 op 或 M 秒，将 op log 折叠为文档快照并持久化。加载时获取最新快照 + 尾部 op。
  5. Presence / 光标。 与编辑分离的独立通道——短暂、有损。每个客户端推送光标位置；session server 广播。无需持久化。
  6. WebSocket 网关。 每个客户端一个持久连接；路由到正确的 session server。
  7. 离线缓冲区。 客户端在本地保留未提交的 op。重连时，从最后一个服务器确认的版本开始重放。

端到端叙述：Alice 和 Bob 都打开了文档。Alice 输入"hello" → 客户端发送 op insert(pos=0, "hello", base_version=42) → session server 当前已在版本 47 → 服务器将 Alice 的 op 与 op 43-47 进行变换 → 应用后广播给所有会话包括 Bob。Bob 的客户端收到 insert(pos=5, "hello", version=48)，应用它（同时对其未确认的本地 op 进行变换），并渲染。

具体存储方案：

存储 我会明确说出的具体存储方案： 条目大小 TTL / 保留期 分片键
Op log op:{doc_id}:{version} 追加行 约 150 B 文本 op 永久保存或快照后压缩 doc_id
快照存储 snapshot:{doc_id}:{version} 压缩的文档正文 10-200 KB 最新版本 + 版本历史策略 doc_id

按存储容量估算：

存储 Math 主要调节项
Op log 1B ops/天 * 150 B/op * RF=3 ~= 450 GB/天；30 天热尾约 13.5 TB 将旧 op 压缩为快照，仅保留审计窗口的热数据
快照 1 亿触达文档 * 50 KB 平均 * RF=3 ~= 15 TB（一次最新快照生成） 版本历史策略和快照间隔是字节主导因素
存在感知 1000 万活跃会话 * 200 B * RF=2 ~= 4 GB 10 s TTL；高负载下丢弃更新
Owner map 1 亿近期触达文档 * 100 B * RF=2 ~= 最坏情况 20 GB 租约 TTL 和懒创建
Dedupe 1B 客户端 op id/天 * 32 B * RF=2 ~= 64 GB/天 24 h TTL；shard by doc_id

6. 深度探讨（约 10 分钟）

OT 实践。 每个 op 携带 (type, position, content, base_version, author)。服务器维护权威 op log。收到 op 时：1. 若 base_version == current_version，直接应用并广播。2. 否则，将 op 与 op (base_version + 1 ... current_version) 进行变换，使用变换规则（删除会偏移后续插入的位置等）。3. 应用，递增版本，广播。

客户端对其尚未收到服务器 ACK 的在途本地 op 做同样处理。

写路径追踪：客户端 -> WebSocket 网关 -> 通过 doc_id 查找所有者 -> session server 验证 ACL 和 client_op_id；若 base_version 过旧，session server 获取缺失的尾部 op；变换，持久化追加 op:{doc_id}:{version} ，更新内存快照，然后广播。追加和广播是有序的：若持久化追加失败，则 op 不被确认。

热写路径，逐步说明：1. 客户端通过已有的 WebSocket 发送 {doc_id, client_op_id, base_version, op} 。2. 网关对 doc_id进行哈希，检查 doc_owner:{doc_id}，并转发给当前 session 所有者；这是每一跳的分片键。3. 所有者在从内存加载短 op 尾的同时并行验证 ACL 和 seen_op:{doc_id}:{client_op_id} 。4. 若 base_version 是最新的，所有者立即应用；若过旧，则先从内存或持久日志读取 op:{doc_id}:{base_version+1...current} 并进行变换。5. 所有者持久化追加 op:{doc_id}:{new_version} ，然后向作者返回 ACK 并广播到对等流。Presence 更新不会阻塞此路径。

冷加载/恢复路径：1. 客户端从快照存储加载 snapshot:{doc_id}:{snapshot_version} ，并向网关请求当前活跃所有者。2. 若所有者存在，则流式传输 snapshot_version之后的 op；若所有者是冷启动的，网关通过 doc_owner:{doc_id} 租约选举一个。3. 新所有者重放快照之后的持久 op，重建内存文档，只有在此之后才允许写入者接入。4. 光标状态以 {doc_id, last_acked_version, client_session_id} 形式持久化，使重连和分页翻阅 op 尾时能从正确版本恢复。

CRDT 替代方案。 每个字符获得一个唯一 ID（分数位置）。插入操作表示"在 X 和 Y 之间插入，ID 为 Z"。并发插入在无需服务器变换的情况下产生一致的顺序。服务器可以是简单的中继。每个字符的存储成本更高。

此处选择 OT 而非 CRDT 的原因：有了中央 session 所有者，OT 使操作体量小，并为服务器提供唯一的权威版本，便于 ACL、审计和撤销。CRDT 更适合离线优先的点对点编辑，但每字符 ID 和 tombstone 增加了存储开销，且在 45 分钟内难以解释清楚富文档操作。

存储选型辩护：Spanner 适合需要强一致的文档级事务和跨地域副本的场景，但对于纯追加日志来说成本较高。Cassandra 适合追加密集型 (doc_id, version) 若 session 所有者是一致性点，则该访问模式合适。单独使用 Kafka 不够，因为用户需要按文档随机回放尾部日志，且需要超出日志保留期的长期持久性。要准备好回答"为什么不只用对象存储？"——对象存储非常适合快照，但对于逐键延迟的尾部读取来说太慢且粒度太粗。

每文档单一所有者。 文档的所有编辑必须在同一节点串行化，以使 OT 权威顺序明确。通过一致性哈希将 doc_id 路由到某个 session server 实现。所有者故障时，通过 leader 选举 + 从持久 op 存储重放日志恢复。

快照。 加载文档时读取 6 个月的 op 是不可接受的。定期快照：每 1000 个 op 或 5 分钟，将 op 折叠为快照。加载 = 快照 + 快照之后的 op。

加载路径追踪： GET snapshot 读取最新的 snapshot:{doc_id}:{version} ，同时向所有者请求该版本之后的 op 尾。若所有者是冷启动的，网关从持久 op log 重放尾部，并在允许编辑者接入前选举新所有者。

离线/断线。 客户端保留本地 op log。重连时发送 {since_version: N}。服务器向客户端重放错过的 op；客户端变换并重放其缓冲的本地 op，每个 op 基于新版本作为新 op 发送。

光标/Presence。 独立通道：高频、无持久化。光标闪烁可以接受。每用户限流约 10 Hz。

ACL/共享。 权限在会话连接时检查（只读 vs 编辑者）。服务器在 op 上标注作者；客户端通过拒绝发送来强制只读。

OT 能干净处理的冲突模式。 - 相同位置的插入 + 插入：按作者优先级确定性排序。 - 相同范围的插入 + 删除：插入优先（文本保留），删除位置偏移。 - 并发的相同编辑：通过 op ID 去重。

处理不优雅的情况。 Schema 级冲突（一个用户插入表格，另一个删除周围段落）。这需要超出文本范畴的 op 类型——Google Docs 使用带结构化操作的更丰富 op 模型。

热门文档扩展。 100 名协作者在同一文档 → 单个 session server 承受 500 ops/s，对单节点来说游刃有余。超出此范围（数千名查看者），独立的只读副本从 op log 拉取；只有写入者经过所有者。

双视图模型。 持久数据集是相同的文档历史，以两种视图呈现： op:{doc_id}:{version} 是用于同步/重放的细粒度编辑索引，而 snapshot:{doc_id}:{version} 是用于快速加载的水化内容视图。快照是派生的；若过期或缺失，重放 op 索引可恢复正确性。

边缘压力测试。 一个所有者租约已过期的休眠文档从快照加尾部日志冷启动，正常文档应在 2 秒内加载完成。用户翻阅版本历史时会离开热 op 尾，读取压缩的审计存储，因此 UI 应切换到历史加载状态。有数千名查看者的热门/公开共享文档应通过只读副本扇出，写入路径保持单一所有者。多地域漂移通过将某文档所有活跃编辑路由到一个写入地域来处理；远程地域提供快照服务并订阅 op 流，存在可见延迟。

会话状态机：

阶段 模式 数据源 状态转换 延迟目标
Active 所有者在内存中 所有者文档状态 + 热 op 尾 正常编辑 节点可见性延迟 < 500 ms
热重连 所有者存活，客户端状态过期 所有者内存 + 操作日志尾部 客户端游标落后 追赶耗时 < 1 s
冷文档 无所有者租约 快照 + 持久化操作日志 第一个编辑者打开 可编辑前耗时 < 2 s
恢复 所有者故障 持久化操作日志 + 租约选举 心跳丢失 秒级；冻结写入
归档历史 已压缩版本 快照链 + 归档操作 用户打开旧版本 较慢，先只读

持久性。 每个已接受的操作在广播前均 fsync 到操作日志。快照为异步。

优雅降级。

情形 模式 数据源 用户可见行为
所有者故障 冻结写入，选举所有者，重放日志尾部 操作日志 + 所有者映射 秒级重连
客户端离线 本地乐观编辑 浏览器本地日志 重连时同步
快照过期 快照 + 较长的操作日志尾部 快照存储 + 操作日志 加载较慢，文档正确
Presence 扇出热点 丢弃/限流光标更新 presence TTL 存储 文本编辑不受影响

面试要点："单一所有者服务器在这里并非可扩展性弱点；它是使 OT 对每个文档都可理解、持久且可审计的机制。"
