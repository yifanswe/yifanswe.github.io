1. 基础知识与解题思路

本题考察点

YouTube 是行星级规模的视频平台。难点在于：每秒上传数小时的视频、转码为多种格式、EB 级别的持久化存储、通过自适应码率向数十亿观众分发内容，以及使内容可被发现的推荐/排序系统。

其不变量如下：

每个上传的视频都被持久化存储，转码为合适的格式，通过 CDN 以自适应码率向观众分发以匹配其带宽和设备，并保持可用；互动计数器、推荐和搜索收敛至一致状态；删除和可见性变更在数秒内传播生效。

在 ICT5 / L6+ 层级，面试官在考察你是否能设计上传 + 转码流水线、PB 级出口流量的 CDN 策略、自适应码率流媒体、推荐系统架构，以及应对热点内容爆发（视频病毒式传播）的能力。

60 秒面试框架

口述版本：

我会将 YouTube 设计为四个主要系统：上传 + 异步转码流水线、基于 CDN 的视频分发层（支持自适应码率流媒体）、用于内容发现的信息流与搜索服务，以及驱动首页和相关视频的推荐流水线。有趣的决策包括：针对多种格式的转码策略、面向成本优化的 CDN 分层、通过 HLS/DASH 实现的自适应码率、推荐内容的新鲜度，以及病毒式热点内容的处理。

首先明确需求

功能性需求：

  • 创作者上传视频（任意时长、常见格式）。
  • 观众在网页端、移动端、智能电视上观看，支持自适应画质。
  • 搜索、推荐、订阅功能。
  • 评论、点赞、不喜欢（或现在仅保留点赞）。
  • 频道与订阅按钮。
  • 直播（除非被问及，否则超出范围）。
  • 变现、广告（大部分超出范围）。

非功能性需求：

  • 每分钟上传 500 小时以上的视频。
  • 每日数十亿次观看。
  • 亚秒级视频启动时间（第一帧的 TTFB）。
  • PB 级存储。
  • 支持 4K 及更高分辨率。
  • 多区域分发。

以下内容超出范围，除非被特别询问：

  • 直播流媒体流水线。
  • 广告服务和竞价。
  • 变现分成。
  • 版权检测（Content ID）。

规模假设

领域 假设
每日上传量 数十万小时。
每日观看量 50 亿次以上。
并发观看峰值 数千万。
平均视频大小 转码前 50 MB-5 GB；多分辨率转码后达数 GB。
存储 数百 EB。
出站流量费 PB 级/天。
Latency（延迟） 视频启动 < 1 秒；卡顿缓冲极少发生。

瓶颈在于出口带宽成本和 CDN 命中率。存储量同样巨大，但带宽是主要开销。

上传与转码

上传：

  1. 创作者客户端请求上传 URL，获取签名 URL + video_id。
  2. 客户端直接将分块上传至对象存储（可断点续传）。
  3. 客户端调用完成接口。
  4. 上传服务将视频标记为 status=processing；将转码任务加入队列。

转码：

  1. 转码 Worker 拉取任务。
  2. 原始视频按时间切分为片段（例如 6 秒）。
  3. 并行将各片段转码为每种目标分辨率和编解码格式。
  4. 生成 ABR 清单（HLS .m3u8 或 DASH .mpd），列出每种分辨率对应的片段。
  5. 生成缩略图（多帧）。
  6. 将派生文件写入派生存储。
  7. 更新视频状态 status=ready 至少一个可用派生文件完成后即更新。
  8. 更高分辨率可稍后完成。

成本优化：

  • 使用竞价实例进行转码。
  • 跳过稀少请求的分辨率（4K），直到被请求时再处理。
  • 随着经济性变化，重新转码至新编解码格式（AV1）。
  • 派生文件稳定后将原始文件归档至冷存储。

面试用语：

转码是 YouTube 花费巨资的环节。我们按片段并行处理，按预期需求优先处理质量级别，并大量使用竞价/抢占式计算资源。

自适应码率流媒体（HLS / DASH）

观看者的播放器根据当前带宽、缓冲状态和设备能力选择最佳分辨率。

  • 清单列出多种分辨率下的片段。
  • 播放器逐段下载，每段约 6 秒。
  • 每段之间，播放器测量网络状况，可能切换分辨率。
  • 预缓冲 30-60 秒；缓冲耗尽时触发重新缓冲。

服务端：

  • 所有片段均从 CDN 分发。
  • 清单文件较小；获取一次后直播内容会定期刷新。

CDN 策略

出口带宽是主要成本，命中率是关键指标。

多层架构：

  • 设备缓存：浏览器/应用保存最近观看的内容。
  • 边缘 POP：全球数百个节点，地理位置靠近观看者。
  • 区域缓存：规模更大，服务来自边缘的缓存未命中请求。
  • 源站：派生存储；命中代价高昂。

缓存策略：

  • 派生文件不可变，因此设置较长的 TTL。
  • LRU 淘汰；热门视频保持热缓存状态。
  • 针对预测的爆款内容提前预热 POP。
  • 按 video_id 片段进行缓存分区以扩展规模。

命中率目标：90%+，10% 的未命中为源站带宽预算。

自适应预取：播放器在用户观看当前片段时预下载下一段。

推荐

两阶段：

离线（每天/每小时）： - 为每位用户构建候选集："与最近观看相似"、"你所在地区的热门内容"、"来自你的订阅"。\n- 存储为预计算列表。

在线（每次请求）： - 排序器在超时时间内（约 100 ms）对候选集评分。\n- 特征：观看者历史、互动度、新鲜度、视频表现。\n- 返回有序列表。

架构：

  • 观看事件喂入特征存储。
  • 每日批处理任务更新 Embedding。
  • 实时信号（观看、点赞）更新在线特征。

搜索

视频元数据（标题、描述、频道）+ 字幕的倒排索引。

  • 索引按 token 哈希分片。
  • 按相关性、热度、新鲜度、频道权威性评分。
  • 最终一致性：新视频在数分钟内完成索引。

互动计数器

与 YouTube 点赞设计（独立文档）模式相同。针对爆款视频采用按视频子计数器分片；定期重新计数以保证正确性。

数据 分区键 原因
视频元数据 video_id 哈希值 直接查找。
频道信息 channel_id 哈希值 频道页面。
派生文件 video_id 哈希值 与视频相同。
观看事件 video_id 哈希 + 时间桶 按视频聚合。
搜索索引 Token 哈希 查询驱动。

订阅 user_id 哈希用于出向； channel_id 用于入向 双向操作。

一致性模型

层级 一致性 原因
视频元数据 强一致性 真实数据来源。
派生文件 不可变 基于内容寻址。
观看次数 最终一致 用户体验。
搜索索引 最终一致 索引延迟。
推荐 最终一致 预计算；定期刷新。
订阅 添加/移除时强一致性。 避免漏发通知。

多区域策略

  • 派生文件的对象存储进行地理复制；原始文件可留在上传区域。
  • CDN 边缘节点遍布全球。
  • 视频元数据已复制；每个视频有归属区域。
  • 推荐特征按区域存储；全球热门内容跨区域复制。

故障模式与缓解措施

故障模式 缓解措施 监控指标
转码失败。 重试；降级为更少分辨率。 转码失败率。
爆款视频 CDN 未命中风暴。 提前预热边缘节点；多层缓存。 源站出口带宽；命中率。
派生文件损坏。 读取时校验 checksum；从原始文件重建。 哈希不匹配。
源站过载。 限流；提升分发层级；请求合并。 源站 QPS。
观看者设备较慢。 ABR 降至更低分辨率。 分辨率分布。
热门上传的搜索延迟。 异步索引；对已知频道进行预索引。 索引延迟。
推荐器响应慢。 严格超时；降级为热门内容。 排名器超时率。
存储成本增长。 冷层存储原始文件；极少访问的派生文件才删除。 存储增长速率。
已删除视频仍被分发。 CDN 清除 + 缓存失效。 过时服务计数。
违反地域限制。 边缘节点规则强制执行。 地域违规次数。

运维指标

正确性：

  • 删除后仍分发旧内容。
  • 转码失败次数。
  • 互动计数器漂移。

容量：

  • 按区域统计的出口带宽。
  • CDN 命中率。
  • 源站 QPS。
  • 转码队列深度。

业务层面：

  • 观看时长。
  • 每视频互动量。
  • 推荐点击率（CTR）。

需要指出的权衡

  • 预生成所有分辨率 vs 按需生成：预生成节省首次观看延迟，按需节省存储空间。
  • 严格 CDN TTL vs 灵活缓存失效。
  • 激进的 ABR 分辨率选择 vs 平滑播放体验。
  • 推荐新鲜度 vs 计算成本。
  • 搜索索引大小 vs 查询延迟。
  • 为预测的爆款内容提前预热边缘节点 vs 预测失误的代价。

常见陷阱与更优做法

陷阱 更优做法
单一视频文件分发。 多个派生文件；ABR 清单。
发布前同步转码。 首个派生文件可见；其余异步处理。
单层 CDN。 带请求合并的多层架构。
单行计数器。 子计数器分片。
永久存储所有原始文件。 派生文件稳定后归档至冷存储。
单一全局推荐器。 离线批处理 + 带超时的在线排序器。

  • 上传 + 异步转码；首个派生文件发布。
  • 以命中率为核心的多层 CDN。
  • 通过 HLS/DASH 实现 ABR；播放器驱动。
  • 两阶段推荐流水线。
  • 独立倒排索引实现搜索。
  • 互动计数器使用子分片。

  • 分层存储以控制成本。

面试用语

YouTube 是视频版 Instagram 加推荐引擎加搜索引擎。主要成本是出口带宽；CDN 策略是设计的核心之一。转码按片段并行处理。自适应码率无论观看者带宽如何均能保证流畅播放。推荐系统是离线候选集加带严格超时的在线排序器。

2. 模拟面试

面试官： 设计 YouTube。

候选人： 我将设计四大系统：上传 + 转码流水线、用于分发的多层 CDN、元数据 + 搜索服务，以及推荐流水线。核心不变式是每次上传的持久化存储，转码为支持 ABR 播放的格式，以亚秒级启动延迟分发，互动计数器和搜索在数分钟内收敛。

面试官： 带我了解一下上传流程。

候选人： 创作者客户端请求上传 URL 和 video_id。通过签名 URL 将分块直接上传至对象存储——可断点续传，能应对客户端网络抖动。完成后，视频标记为 processing。转码任务入队。Worker 将原始视频切分为 6 秒片段，并行转码为多种分辨率（144p 至 4K）和编解码格式（H.264、VP9、AV1）。生成 HLS 和 DASH 清单。至少一种可用分辨率就绪后，视频标记为 ready 并可发布。

面试官： 为什么要按片段转码？

候选人： 并行性。一段1小时的视频按顺序转码需要一个小时。将其拆分为600个6秒的片段，在600个工作节点上并行转码，几分钟内即可完成。输出结果也天然适用于HLS/DASH的分段传输。

面试官： 自适应码率——播放器如何决策？

候选人： 播放器下载清单，然后逐个获取片段。在片段之间，它通过测量下载时间与片段时长的比值来估算可用带宽，同时检查当前缓冲区水平，选择能持续播放而不发生缓冲区下溢的最高分辨率。若网络条件变差则降低分辨率，改善后则提升。服务器只负责提供片段，智能逻辑在客户端。

面试官： CDN 策略？

候选人： 多级架构。边缘 POP 是距离观众最近的一层——全球数百个。区域缓存位于边缘层之后。源站是派生文件存储。命中率目标超过90%，因为源站出流量费用高昂。由于派生文件不可变，缓存 TTL 设置较长。对预测会爆火的内容提前预热边缘节点。缓存未命中时进行请求合并，避免惊群效应。每个视频的片段按 id 分片，不同片段可能由不同的缓存节点提供服务。

面试官： 一个视频爆火——某地区10万个并发观看。

候选人： 边缘 POP 缓存承接大部分流量。每个片段的首次未命中会依次访问区域缓存，再到源站。边缘层的请求合并：对同一片段的100个并发未命中会合并为一个上游请求，响应返回后扇出给所有等待的客户端。我们监控出流量并提前预热更多边缘节点。推荐引擎可能将内容推送给更多观众，进一步叠加负载——若源站过载，我们会对推荐系统施加背压。

面试官： 推荐系统是如何工作的？

候选人： 两阶段。离线阶段：每日批处理通过协同过滤、相似内容向量嵌入、订阅关系和热门趋势为每个用户构建候选集，每个用户都有一份存储好的候选列表。在线阶段：当用户请求首页时，获取候选集并连同观看者上下文一起发送给排序器。排序器根据特征（观看历史、互动情况、内容新鲜度、视频表现）打分。严格的100ms超时；若排序器超时，则回退到最近的热门内容。

面试官： 搜索？

候选人： 对标题、描述、标签、频道和字幕的倒排索引。索引器消费视频事件，按词条分片。查询对相关性、热度、新鲜度和频道权威性进行综合评分。最终一致性——新视频在几分钟内可被搜索到。高价值频道可能会加快索引速度。

面试官： 存储？

候选人： EB 级别。分层存储。派生文件属于热数据——有 CDN 重度支撑。原始文件在派生文件稳定后可下沉至冷存储，以备需要时重新转码。低播放量的旧视频进一步下沉。对派生文件进行引用计数，以便淘汰未使用的文件。随着时间推移，我们可能将老旧派生文件重新编码为更廉价的编解码器（AV1）以节省带宽。

面试官： 互动计数器？

候选人： 与 YouTube 点赞设计相同。对每个视频的计数器进行子分片以应对爆火时的高计数量。定期从每用户状态重新统计以保证正确性。展示为最终一致性；用户可以接受秒级延迟。

面试官： 订阅？

候选人： 频道有订阅者列表。新上传事件通过通知系统向订阅者触发通知。扇出规模随订阅者数量增长；大 V 频道可能使用类似 Twitter 的推拉混合模式。首页"订阅"选项卡从每用户订阅状态读取数据。

面试官： 多区域？

候选人： 派生文件进行地理多副本复制以支持边缘快速服务。原始文件存储在上传所在地区。视频元数据多副本复制；每个视频有归属主区域用于写入。CDN 边缘节点覆盖全球。推荐特征按地区维护，跨地区共享热门趋势。地理限制在边缘层执行。

面试官： 直播？

候选人： 管道不同。通过 RTMP 或 WebRTC 接入；实时转码并生成低延迟 HLS 片段（LL-HLS，约2-4秒端到端延迟）。CDN 传输模型相同，但缓存 TTL 非常短。支持最近内容的 DVR 功能。与 VOD 的区别在于清单会持续更新。

面试官： 总结。

候选人： 上传至对象存储；异步转码集群生成多分辨率派生文件。多级 CDN 通过 HLS/DASH 自适应码率流式传输提供服务。首个派生文件发布后视频即可快速可见；更高分辨率异步上线。推荐系统由离线候选集加在线排序器组成。搜索使用异步倒排索引。互动计数器对爆火内容进行子分片。存储从热态派生文件分层至冷态原始文件。多地区架构使用 CDN 边缘节点结合每视频归属主区域。最主要的成本是出流量；CDN 策略和命中率决定了整体经济效益。

3. 练习备注

五分钟版本

  1. 明确范围：上传、观看、搜索、推荐。
  2. 不变约束：持久存储、转码、CDN 传输、秒级以内启播。
  3. 上传 + 按片段并行转码。
  4. 多级 CDN；命中率高于90%。
  5. HLS/DASH 自适应码率；由播放器驱动。
  6. 推荐：离线候选集 + 带超时的在线排序器。
  7. 搜索：异步倒排索引。
  8. 互动计数子分片；存储分层。

一分钟结尾陈述

YouTube 是一个以重度 CDN 策略为核心的视频管道。上传写入原始文件；转码集群对每个片段并行处理以生成多分辨率派生文件。多级 CDN 通过 HLS/DASH 自适应码率传输，命中率高于90%是使出流量经济可行的关键。推荐系统由离线候选集加带严格超时的在线排序器组成。搜索为异步倒排索引。互动计数器进行子分片。存储从热态分层至冷态。最主要的成本是出流量；最核心的正确性问题是持久存储和可见性传播。

最佳面试用语

  • "转码按片段并行；清单将其拼接以实现 ABR。"
  • "命中率90%以上的多级 CDN 是使出流量经济可行的关键。"
  • "自适应码率由播放器驱动；服务器只负责提供片段。"
  • "推荐系统由离线候选集加在线排序器组成，并有严格超时控制。"
  • "从热态派生文件到冷态原始文件的存储分层在 EB 规模下不可或缺。"
