§1. 功能开关系统的本质

一个 feature flag system 是程序行为的运行时配置。给定一个开关键和某些上下文（用户、设备、请求），系统返回一个值——布尔值、变体名称、JSON blob 或数字——供调用代码进行分支判断。其核心目的是将代码的 deployment （缓慢、不可逆、在 CI 中测试）与行为的 release （必须快速、可在数秒内回滚、可按用户/分群/区域定向）分离开来。

一个 功能开关 从狭义上讲，这是一种 operational control：熔断开关、灰度发布、断路器、按租户覆盖。成功标准是"变更无害"——可观测性、快速回滚和定向精度是重中之重。

一个 experiment 是附带了 statistical analysis on top的功能开关：确定性地将用户随机分配到各变体、记录曝光日志、关联结果指标、进行显著性检验。成功标准从"无害"转变为"估算因果效应"——方差缩减、功效分析和分析严谨性成为主角。

两者共享同一条热路径。两者都在 (user, flag) → variant 每个触及门控代码的请求上进行计算。在规模化场景下，它们共享同一个进程内评估器、同一个配置分发平面以及同一条曝光流水线。这就是为什么 Gatekeeper、LIX、Statsig、LaunchDarkly 等系统能同时服务于两种用途。

读者需要仔细区分的相邻类别:

  • 静态配置管理 （Consul、etcd、Spring Cloud Config、随二进制部署的 YAML）：变更不频繁、对整个机群统一生效、无按用户定向、无实验能力。控制面与开关系统类似，但评估契约弱得多——是"机群最终收敛"，而非"某个具体用户此刻确定性地获得某个变体"。
  • 绑定单一层的 A/B 测试框架 （面向 Web 的 Google Optimize、面向移动端的 Firebase Remote Config、浏览器端的 Optimizely 脚本）：是完整开关/实验平台的子集，通常只面向特定层（仅客户端）。缺乏服务端评估、缺乏跨平台一致性、缺乏深度指标集成。
  • 基于部署的灰度发布 （Kubernetes 滚动更新、Spinnaker 金丝雀发布、ECS 任务百分比）：发布的是 代码，而非 behavior。无法定向特定用户，无法在同一个二进制中运行对照组与实验组，无法在不重新部署的情况下关闭某个功能。
  • 授权 / 权限管理 （Open Policy Agent、Zanzibar 风格的权限系统）：表面上同样是"给定用户和键，返回一个决策"，但数据来源是关系图，而非灰度发布策略。两种系统经常集成，但不可互换。

功能开关系统不擅长的场景:

  • 跨服务的事务语义。 开关不是分布式事务。两个服务在同一时刻对同一个开关求值，在传播窗口期内可能看到不同的值。如果你需要"所有微服务在同一纳秒原子切换行为"，你需要不同的原语（通常是带显式版本握手的协调式部署）。
  • 回溯性变体更改。 一旦用户被分配到某个变体且曝光事件已记录，就无法重写历史。用新的灰度比例重新计算分配，也无法让用户"时光倒流"。
  • 高基数查找。 "返回每用户特征 embedding"是一个特征 store，而非开关系统。开关基于属性进行定向；它们不会记忆每个用户的值。
  • 长期业务状态。 如果你用一个开关来判断"该用户是否在高级计划中"，那是权限管理，而非开关。开关应当有过期时间；权限则永久存在。

该技术所解决的问题类别： 使行为变更成为一等公民、受治理、可观测、可回滚的运行时操作，区别于代码部署，同时由同一原语支持运营（熔断开关）和科学（A/B）两种模式。

§2. 固有保证

该技术按设计提供的能力，以及需要你额外构建的能力。

设计保证 （任何生产级开关系统都应具备以下特性，否则不算生产级）：

  1. 进程内评估，p99 低于毫秒级。 评估器作为库运行于应用进程内部。评估是纯函数：哈希 + 规则树遍历 + 桶表查找。热路径上无网络调用。典型数值：p50 为 300–700 ns，每次评估 p99 为 5–10 µs。如果系统在每次请求时通过远程过程调用（RPC）访问中央服务来评估开关，则架构形态有误，在任何非平凡规模下都会失败。

  2. 按……的确定性分配 (user_id, flag_key). 同一用户在同一开关上，跨请求、会话、设备、应用服务器和 SDK 语言，始终获得相同的变体。分配本身即存储——无需查询每用户表，因为 hash(user_id || flag_key) mod 10000 仅凭输入即可复现。这是让系统在不存储 100,000 个开关 × 10 亿用户的变体决策的情况下，支持数亿用户的关键技巧。

  3. 灰度发布安全性。 开关在 1% 时恰好对 1% 的用户生效；在 10% 时包含原来的 1% 再加 9% 更多（原来的 1% 不会翻转出去）；在 50% 时包含 10% 再加 40% 更多。随着灰度比例增长，桶只会向实验侧流动，用户不会来回震荡。

  4. 即时关闭。 从运营人员在控制台点击"关闭"到全球 100% 的生产服务器采纳新值：全球范围内不超过 60 秒，通常不超过 10 秒。没有这个特性，开关系统不过是修复 bug 更慢的手段。

  5. 默认安全回退。 如果 SDK 无法访问配置平面（冷启动、分区、边缘节点故障），它返回调用方提供的默认值。它永远不会抛出异常，不会阻塞请求，不会返回任意值。默认值是保守的、已有的行为。

  6. 曝光日志作为评估契约的一部分。 每次 getVariant(user, flag) 调用会自动向分析流水线发送一条曝光事件。调用方不可能"忘记"记录日志。如果日志记录是调用方的责任，那么半数实验最终都会数据损坏。

  7. 可审计性。 每次开关变更都以持久化方式记录操作者、时间戳、变更前后状态和理由。"谁在凌晨 2 点为客户 X 打开了 enterprise_sso_v3 ""这一问题必须在一分钟内可查。

必须由系统设计者额外构建 ——开关系统不会自动提供这些能力：

  • 跨服务的事务性行为变更。 如果服务 A 和服务 B 都检查同一个开关且必须同步切换，你需要一个编排器（发布协调器、链式依赖）。开关层保证最终收敛到新值，而非原子性。
  • 实验中的方差缩减。 统计显著性流水线给你置信区间；降低噪声（CUPED——利用实验前数据的受控实验方差缩减、分层、交错排列）需要显式的建模工作。
  • 开关生命周期管理。 系统可以标记过期开关，但无法删除代码中已失效的 if 分支。那需要静态分析 + PR 机器人 + 团队规范。
  • 多实验交互处理。 分层分配只有在实验作者 使用这些分层的前提下才能防止同页面冲突。系统提供原语；接入是你的工作。
  • 跨平台身份稳定性。 如果 iOS SDK 对 device_id 进行哈希，而服务端 SDK 对 user_id进行哈希，同一个人会落入不同的桶。开关系统强制 SDK 间的哈希可移植性，但无法为你的业务选择正确的身份标识；那是你的决策。

需要内化的契约： 开关系统为你提供确定性、进程内、快速、可回滚、可审计的查找，以及扇出分发平面和曝光分析流水线。其余一切——治理、清理、跨服务协调、统计严谨性——是你的问题，也是平台团队需要额外构建的问题。

§3. 设计空间

真实实现中的三个主要变体维度。

维度 A：SaaS 还是内部自建。

  • SaaS ——LaunchDarkly、Statsig、Optimizely、Split.io、ConfigCat、GrowthBook Cloud。按月活跃用户（MAU）付费，获得托管控制台、各语言 SDK、推送式传播、实验分析器和销售支持。传播延迟通常为 100 ms–1 s；评估是进程内的，因此是微秒级。
  • 内部 ——Facebook Gatekeeper、LinkedIn LIX、Uber XP、Netflix ABlaze、Airbnb ERF（实验报告框架）、Pinterest Helium、Etsy Catapult。之所以自建，是因为在数亿用户规模下 SaaS 费用极高，传播要求比任何 SaaS 愿意承诺的都要严格，且实验平台必须深度集成内部指标流水线（Hive、Pinot、BigQuery、Druid），而任何外部供应商都无法原生查询这些系统。
  • 开源自托管 ——Unleash、GrowthBook OSS、FF4J、Flagsmith（也提供 SaaS 版）。适合希望避免 SaaS 定价但工程积累不足以从零构建的中型团队。在实验分析器方面通常较弱；需要自带指标平台。

盈亏平衡点大约是 约 1000 万 MAU 或每年约 20 万美元的 SaaS 支出，超过这个点自建在成本上才合理。低于此值，SaaS 或开源占优。

维度 B：本地评估（SDK 将配置推送到每个应用）还是 RPC 评估（每次评估都是一次网络调用）。

  • Local eval 是标准做法。SDK 在内存中保存所有开关规则的编译快照；评估即函数调用。所有现代系统在服务端都采用这种方式。
  • RPC eval 在规模化时架构形态有误。一些早期系统和部分客户端开关产品（将规则下发到浏览器会泄露正在进行的实验信息）使用 RPC。在服务端代价是致命的：每次请求 30 次开关评估 × 1 ms RPC = 每个端点 30 ms 延迟税。默认禁止。
  • Hybrid client eval ——对于本地配置快照会包含不想泄露的规则的浏览器/移动客户端，客户端 SDK 调用服务端评估端点 一次 每个会话调用一次并接收解析后的变体。这以延迟换取保密性，是客户端开关的正确架构形态。

维度 C：仅开关 还是 开关 + 实验 + 指标。

  • Flags-only ——老派的 FF4J、基础版 Unleash、简单的 Consul 支撑的开关服务器。仅支持带定向的键值配置。对熔断开关和灰度发布已足够。
  • Flags + experimentation ——LaunchDarkly Experimentation、Statsig、Optimizely、Split.io。新增曝光流水线、指标定义和统计引擎。
  • Flags + experimentation + metric platform ——LIX、Uber XP、Netflix ABlaze、Statsig（含其指标仓库）。实验系统是内部指标平台的主要消费者；两者协同设计。在此场景下，你可以询问"7 天留存率的提升是否在 95% 置信度下显著"，并在 50 ms 内获得经过 CUPED 方差缩减的可靠答案。

§4. 字节级原理

这是区分开关系统浅层概述与深度理解的章节。三个结构至关重要：每台应用服务器上的 compiled rule tree 、提供确定性的 bucketing function ，以及驱动实验分析的 columnar exposure store 。另外还有主机端配置分发。

4a. 编译规则树

控制台将开关呈现为人类可读的规则。SDK 看到的是编译后的树。编译之所以重要，是因为评估是热路径。

源规则（人类编写的内容）：

flag: checkout-v2
  variations: [control, treatment_A, treatment_B]
  rules (evaluated in order):
    1. IF email endsWith "@linkedin.com"               -> treatment_A
    2. IF country IN {US, CA} AND app_version >= 5.0
         AND tenure_days > 30
         ramp: {control: 80%, treatment_A: 10%, treatment_B: 10%}
    3. ELSE                                            -> control

编译后（SDK 实际遍历的内容）：

RuleNode {
  conditions: [
    Cond { attr: "email", op: ENDS_WITH, values: ["@linkedin.com"] }
  ],
  outcome: SINGLE(variation_id=1)        // treatment_A
} -> RuleNode {
  conditions: [
    Cond { attr: "country",      op: IN_SET,     values_hash: 0xA3F9... },
    Cond { attr: "app_version",  op: GTE_SEMVER, value: 5_000_000_000 },
    Cond { attr: "tenure_days",  op: GT,         value: 30 }
  ],
  outcome: BUCKETED(
    bucket_table: [
      (0..7999,    variation_id=0),  // 80% control
      (8000..8999, variation_id=1),  // 10% treatment_A
      (9000..9999, variation_id=2)   // 10% treatment_B
    ],
    salt: "checkout-v2:rule2"
  )
} -> RuleNode {
  conditions: [],                       // fallthrough
  outcome: SINGLE(variation_id=0)       // control
}

编译器预先嵌入的属性：

  • IN_SET 值数量 > 8 的子句会被预先哈希到完美哈希表或 Roaring bitmap 中——O(1) 查找，而非列表扫描。
  • country IN {US, CA} 变为针对内部化国家代码的 2 位 bitmap 测试。
  • Semver 比较被预解析为 packed int64（major*1e10 + minor*1e6 + patch).
  • 桶表是一个有序数组，由 (end_bucket, variation_id) 构成——对桶号进行二分查找，时间复杂度为 O(log K)，其中 K 为变体数（通常为 2–5）。

每个开关编译后约为 200–500 字节。10 万个开关约占 SDK 堆内存 20–50 MB。可 mmap、缓存友好，热门开关可驻留于 L2/L3 缓存。

为何使用编译树，而非解释执行 JSON。 每次评估时重新解析 JSON 并通过字符串匹配属性名，每次评估耗时 5–10 µs。编译树仅需约 300 ns。这是热路径上 30 倍的差距，乘以 LIX 规模下机群每秒约 18 亿次评估。

与替代方案的权衡：存储预计算的 (user, flag) → 变体 映射。 O(1) 评估，但需要 400 TB 存储（10 亿用户 × 10 万开关 × 4 字节），加上每次评估的远程查找（已禁止）。规则树以极少量 CPU 换取零每用户存储和零网络开销。在规模化场景下，这显然是正确的选择。

4b. 分桶函数——确定性的核心

给定 (user_id, flag_salt)，在……中生成一个整数 [0, 9999]:

fn bucket(user_id: &str, salt: &str) -> u16 {
    // 1. Concatenate with a sentinel that can't appear in either input
    let input = format!("{}\x00{}", salt, user_id);
    // 2. Hash with a stable, language-portable function
    let h = murmur3_128(input.as_bytes());
    // 3. Take the low 64 bits, modulo 10000 -> 0.01% resolution bucket
    (h.low64() % 10_000) as u16
}

每个选择的原因：

  • Salt 是 flag_key （或 flag_key:rule_id)，而非空字符串。两个开关不得产生相关联的桶。如果两个开关都只对 user_id进行哈希，那么处于开关 A 10% 灰度中的用户 也会 处于开关 B 的 10% 灰度中，B 的"实验效果"会悄悄地携带 A 的效果。加盐可以使桶之间解相关。

  • 稳定、跨语言可移植的哈希。 MurmurHash3、xxHash 或截断的 SHA-1。Java、Python、Go、JavaScript、Rust、Swift、Kotlin 中的同一函数必须产生相同的数字。LaunchDarkly 使用 SHA-1，Unleash 使用 MurmurHash3，LIX 有自己基于 MD5 的哈希，并附有文档规范以确保客户端和服务端 SDK 一致。每个平台都会发布 测试向量文件 ——几百条 (user_id, flag_key) -> bucket 三元组——每个 SDK 都有 CI 检查，一旦实现出现偏差即报错。

  • 10,000 个桶，而非 100 个。 100 是旧标准；它将灰度粒度限制在 1%，对于"最安全的 0.1% 金丝雀"而言粒度太粗。10,000 个桶 = 0.01% 粒度。Statsig、LaunchDarkly、LIX、Gatekeeper、Optimizely 都使用 10,000+。一些以实验为主的平台使用 1,000,000 个桶，用于即便 0.01% 仍嫌粒度过粗的超大规模实验。

  • 取模，而非范围划分。 hash % 10000 当哈希充分混合时，能均匀分布。一些较旧的系统使用 hash / (UINT_MAX / 10000) 在混合不充分的哈希上，范围边界附近存在细微偏差。

确定性的实际应用——灰度递增场景：

Rollout starts at 1%:
  bucket 0..99      -> treatment
  bucket 100..9999  -> control
  user u_42 bucket 4271 -> control

Rollout bumps to 10%:
  bucket 0..999     -> treatment
  bucket 1000..9999 -> control
  user u_42 bucket 4271 -> control (still)

Rollout at 50%:
  bucket 0..4999    -> treatment
  bucket 5000..9999 -> control
  user u_42 bucket 4271 -> TREATMENT (now flipped, once and only once)

Rollout at 100%:
  all buckets -> treatment
  user u_42 -> treatment

每个用户恰好越过一次阈值。 关键不变量：已在实验组中的桶始终留在实验组。 一个在每次灰度步骤都重新随机化的朴素实现会导致用户来回震荡——这会破坏实验有效性， 。 并使用户感到困惑（"新功能在页面加载之间时有时无"）。

对于需要在实验进行中重新平衡（以受控方式在变体之间移动用户）的少数高级实验设计，分配必须持久化存储或整个灰度过程重新运行。该功能仅在专业平台（Uber XP、Netflix ABlaze）中提供，普通团队很少能用到。

4c. 主机端配置分发

约 30 MB 的编译规则集如何送达 30,000+ 台应用服务器并保持新鲜？

朴素扇出：每台服务器每 60 秒向 flag 服务轮询完整规则集。带宽： 30,000 × 30 MB / 60 s = 15 GB/sec 来自源节点。源节点过载崩溃。此外，在 60 秒轮询窗口内，一半机群使用旧规则集，另一半使用新规则集——kill-switch 传播最坏情况为 60 秒。不可接受。

真实设计——通过边缘扇出层进行增量推送：

control plane (MySQL/Postgres)
   |
   v
ruleset compiler (rule tree -> binary blob + diff vs prior version)
   |
   v
origin service (signed snapshots + diffs in S3)
   |
   v
regional edge PoPs (Points of Presence) — terminate SSE / long-poll, hold 100k+ connections each
   |
   v hybrid pull+push
   v  (initial: pull 30 MB snapshot; steady-state: receive 500-byte diffs)
   |
SDK in app server (in-process eval, atomic pointer-swap on update)

LIX 规模下的稳态数据（~100k 个 flag，~30k 台应用服务器，~500 次变更/分钟）： - 一个 diff：~500 字节（典型的 flag 变更只涉及少量规则）。 - 一次变更通过 N 个边缘 PoP 扇出至 30k 台服务器：每个边缘推送 500 B × (30k / N) 每次变更。 - 总计： 500 changes/min × 500 B × 30k = ~125 MB/s 的扇出流量，分布在各边缘节点。每个边缘只处理一小部分。 - Kill switch 传播：dashboard 点击 -> DB 提交（~10 ms）-> 变更流（~100 ms）-> 边缘扇出（~500 ms）-> SDK 应用（~10 ms）。 端到端 ~1.1 s p50，< 10 s p99。

冷启动成本： 30,000 servers × 30 MB = 900 GB。机群启动时发生一次，代价较高；稳态下可忽略不计。边缘节点缓存来自 S3 的快照，因此源节点每个快照版本每个边缘只被访问一次，而不是每台服务器都访问。

4d. 曝光存储——列式 OLAP

每次 eval 后，SDK 推送一个曝光事件：

{
  "user_id": "u_42",
  "flag_key": "checkout-v2",
  "variant": "treatment_A",
  "rule_matched": "rule_2",
  "ts": 1715712000123,
  "country": "US",
  "app_version": "5.3.1",
  "session_id": "s_abc..."
}

这些事件写入 Kafka（按 user_id）并落入列式 OLAP（Online Analytical Processing）存储——Pinot、Druid、ClickHouse、BigQuery。为何选择列式：

主要查询是变体对比：

SELECT variant, COUNT(DISTINCT user_id), AVG(metric_value)
FROM exposures_joined_with_outcomes
WHERE flag_key = 'checkout-v2'
  AND date BETWEEN '2026-05-01' AND '2026-05-22'
GROUP BY variant

这涉及潜在数十亿行中的约 5 列。行式存储（Postgres、MySQL）每行扫描 200 字节以读取 30 字节——浪费 7× 的 I/O。列式存储仅读取所需的 4 列，且经过字典压缩。

Pinot 中的 segment 文件如下所示：

exposures_2026-05-22.segment
  metadata.json           (schema, time range, row count)
  columns/
    user_id.dict          (dictionary: distinct user IDs -> small ints)
    user_id.fwd           (forward index: row -> dict_id)
    flag_key.dict
    flag_key.fwd
    flag_key.inv          (HOT — used by every WHERE flag_key = X)
    variant.dict
    variant.fwd
    ts.raw                (sorted; range index)
  startree.idx            (optional: pre-aggregated cube)

其倒排索引在 flag_key 是关键所在。 WHERE flag_key = 'checkout-v2' 变为字典查找 + bitmap 读取——亚毫秒级，与总表大小无关。实验 dashboard 在 50–200 ms 内返回提升量、置信区间和 p-value。

与 LSM（Log-Structured Merge tree）行式存储的权衡： Cassandra 和 RocksDB 擅长按主键进行点读/写。变体对比不按主键读取；它扫描某个 flag 的行并跨用户聚合。列式存储在该访问模式下胜出 10–100 倍。反之，列式存储不擅长单行 upsert——但我们从不 upsert 曝光记录，只做追加。

4e. 字节级别的完整 eval 流程详解

User u_42 访问 checkout API。处理器调用 flagSDK.getVariant("checkout-v2", userContext):

1. SDK reads the compiled rule tree from its in-memory cache.
   Address: heap pointer to a ~400-byte structure.
   Cache line hit, since this flag is used often.

2. SDK walks the rule list in order:

   Rule 1: email endsWith "@linkedin.com"?
     userContext.email = "alice@gmail.com" — NO MATCH.

   Rule 2: country IN {US, CA} AND app_version >= 5.0 AND tenure_days > 30?
     country = "US" — passes bitmap test.
     app_version = 5.3.1 (parsed 5_003_001_000) >= 5_000_000_000 — passes.
     tenure_days = 142 > 30 — passes.
     All clauses pass -> enter BUCKETED outcome.

3. Bucket computation:
   salt = "checkout-v2:rule_2"
   input bytes = "checkout-v2:rule_2\x00u_42"
   murmur3_128(input) -> 128-bit hash; low 64 bits = 0x9F32A0...
   hash.low64 % 10_000 = 4271

4. Binary search the bucket table:
   4271 falls in range [0, 7999] -> variation_id = 0 (control).

5. Resolve variation_id 0 -> "control" string. Return.

6. SDK appends exposure event to a thread-local ring buffer:
   {user_id:"u_42", flag_key:"checkout-v2", variant:"control",
    rule_matched:"rule_2", ts:1715712000123, ...}
   (~50 ns; no syscall, no allocation)

7. A background flusher drains the ring buffer to Kafka every 1 second
   or 1000 events, whichever comes first.

8. Total CPU time steps 1–6: ~400 ns p50, ~5 µs p99.

与朴素的"RPC 到 flag 服务"路径相比：p99 为 1–5 ms，线程被阻塞，且依赖 flag 服务作为故障模式。进程内设计快约 1000 倍。

4f. 持久性与崩溃语义

从宿主机的角度看，SDK 是无状态的。若应用服务器在 eval 中途崩溃，该 eval 丢失；用户的下一个请求将确定性地重新求值。没有需要恢复的每用户状态。

SDK 保存的唯一持久状态：

  • 本地磁盘上的最后已知良好规则集 （例如， /var/cache/lix/ruleset.bin）。在配置平面不可达时用于冷启动。通过 rename(2) ——部分写入不会损坏文件。启动时通过 mmap 映射，以实现快速可用。
  • 曝光环形缓冲区 仅在内存中。崩溃时，未刷新的事件会丢失。这是可接受的，因为曝光事件是尽力而为的统计信号，而非事务性的。丢失 0.001% 的曝光不会使实验失效。

真正重要的持久性存在于 控制平面：当 dashboard 提交 flag 变更时，MySQL/PostgreSQL Write-Ahead Log（WAL）在任何传播开始之前先完成提交。若变更流 worker 在提交和发布之间崩溃，则在重启时从 binlog/WAL 偏移量处重放。审计日志单独复制到 S3 + Kafka，即使元数据 DB 发生灾难性故障也能存活。

§5. 容量上限

该技术跨越 7+ 个数量级的范围。请引用多个真实部署案例。

小规模——LaunchDarkly 免费版客户，OSS Unleash 自托管： - 1k–50k MAU，~100 个 flag，单区域。 - eval 为微秒级（进程内 SDK 在每种规模下工作方式相同）。 - 传播：SSE（Server-Sent Events）推送，典型 < 1 秒。 - 存储：一个 Postgres，一个 Redis 缓存，无需分析流水线。 - 具体数字：使用 LaunchDarkly 免费版的典型 Series-A 初创公司，所有 flag 流量在每月 $20 的配置上处理；flag 系统在其成本结构中几乎不可见。

中等规模——Statsig 客户、Stripe 级别公司、GrowthBook 自托管： - 1M–50M MAU，1k–5k 个 flag，多区域。 - 进程内 eval p50 约 500 ns。 - 传播：边缘推送 diff，全球 < 500 ms。 - 存储：专用 config-plane DB，Kafka 用于曝光，列式数据仓库（BigQuery 或 Snowflake）用于实验分析器。 - 具体数字：典型的 10M MAU 公司运行约 50 个并发实验，每天产生约 20 亿次曝光，可在约 20 节点的单个 Pinot 或 Druid 集群中管理。

大规模——LinkedIn LIX、Uber XP、Netflix ABlaze、Airbnb ERF（Experimentation Reporting Framework）： - 100M–1B MAU，10k–100k 个 flag，全球部署。 - LIX：~40,000 个活跃 LIX key，每天约数十亿次 eval，跨区域约 30,000+ 台应用服务器。每台应用服务器峰值时约每秒 60k 次 eval，占一个核心的约 4%。 - Uber XP：同时运行约 1,000+ 个并发实验，采用互斥的分层分配。构建于 Cassandra + Kafka + Pinot 之上。 - Netflix ABlaze：数百个并发实验，具有多年保留期以衡量累积产品影响。 - 传播：通过 SSE 进行边缘扇出，全球 p50 < 2 秒，p99 < 10 秒。

超大规模——Facebook Gatekeeper： - 每个 HHVM（Hip-Hop Virtual Machine）请求会访问 Gatekeeper 一次或多次。公开演讲显示其规模在 每秒数亿次 eval 的范围内，覆盖整个机群。 - 最热的 flag 检查被 JIT 编译（Just-In-Time 编译）直接注入 HHVM 字节码，使 eval 成本降至纳秒级别。 - 他们首先遇到的瓶颈是曝光日志量——在每秒 1 亿次 eval 时，无法将每次曝光都记录到 Kafka。采样变为强制要求（1-in-N 日志），变体决策对每个用户仍是确定性的，但 log 事件会被采样。

各规模层级的瓶颈演变：

变化内容 首要瓶颈 解决方案
100k MAU 无，任何方案都可行 YAML 或 Unleash 即可
10M MAU 轮询导致的传播延迟 切换至 SSE/边缘推送
100M MAU 配置快照大小 + 冷启动 编译规则树 + 边缘缓存
1B MAU 曝光量 采样，分区 Pinot 表
10B 次请求/天 热点 flag 的 eval CPU JIT 编译热规则（Gatekeeper）

该技术在整个范围内保持相同形态。变化的是：规则编译、传播推送与轮询的选择、曝光采样、边缘扇出——每个层级在前一个层级成为瓶颈时启用。

§7. 疑难问题

内在于该技术，不特定于某一工作负载。每个问题都有一句话陈述、一个朴素修复方案、带具体状态的故障演练以及真实修复方案。示例涵盖后端、移动端、ML、定价、排序。

问题 1：在数千台服务器上保持一致的低延迟 eval（发布/kill 传播）

后端 kill-switch 场景：值班工程师将 checkout-v2 从 50% 调整到 0%，因为 iOS 应用在 10% 的用户上崩溃。

朴素的解决方案： SDK 每 60 秒轮询 flag 服务以获取完整规则集。

为什么会出问题： 在 30k 台服务器 × 30 MB 规则集 / 60 s 的情况下，源节点看到 15 GB/sec 的出口流量并立即过载崩溃。更糟的是，kill-switch 传播最坏情况等于轮询间隔——60 s——平均 30 s。在此窗口内，被路由到尚未轮询的服务器的新用户会命中有问题的流程。kill 操作后 30 秒以上，iOS 崩溃报告仍在持续涌入。

真正的修法： 通过边缘扇出层进行基于增量的推送。SDK 与区域边缘 PoP 保持持久 SSE 连接。源节点向每个边缘发出紧凑 diff（flag_id, version, ~500 B blob）；边缘在 < 500 ms 内扇出至其连接的 SDK。SDK 原子性地交换指针，切换到新编译的规则。kill-switch 传播：p50 < 2 s，p99 < 10 s。带宽：500 B × 30k = 每次变更共 15 MB，分布在各边缘节点。

移动应用发布（iOS/Android 应用拉取其配置）和后端服务 flag 也面临相同问题。修复方案相同。

问题 2：跨 Java、Python、Go 和 JS SDK 的一致分桶

ML 模型 A/B 场景：推荐系统有一个用 Go 编写的服务端排序器、一个用 Python 编写的实时评分器，以及一个用 TypeScript 编写的客户端 UI。三者都需要知道"该用户对应哪个模型变体"。

朴素的解决方案： 每个 SDK 使用该语言内置的 hashCode().

为什么会出问题： Java String.hashCode() 是与 Python 不同的函数 hash() 与 Go 不同 hash/fnv 与 V8 的字符串哈希不同。相同的 user_id 在不同服务中落入完全不同的分桶。用户 u_42 从 Go 排序器的角度看在实验组，从 Python 评分器的角度看在对照组，在客户端又在另一个完全不同的分桶中。A/B 测试数据一片混乱。

具体状态：用户 u_42 打开 iOS 应用，看到实验组推荐（TypeScript JS 哈希分桶 23，在 10% 组内）。他们滚动时，Go 后端重新排序（Go FNV 分桶 7421，不在该组），返回对照组排名。新模型的推荐随着用户滚动时出现时消失。

真正的修法： 一个由平台指定的稳定、语言无关的哈希，附有已发布的测试向量。MurmurHash3、xxHash 或截断的 SHA-1——所有这些在每种语言中都有字节一致的实现。平台发布测试向量文件（[(user_id, flag_key, expected_bucket), ...]），每个 SDK 都有一个 CI 任务，若实现产生偏差则失败。LIX 通过将跨语言测试向量检入 SDK 仓库来强制执行这一点。

问题 3：跨会话和身份切换的处理稳定性

定价实验场景：一个电商网站正在测试 10% 对比 15% 的批量折扣。

朴素的解决方案： 在请求时使用 hash(user_id, flag_key)。结构上保证稳定。

为何会出问题（比看起来更微妙）：

  • 用户未登录时浏览： user_id = anonymous。他们的分桶取决于使用哪个匿名身份。他们看到 10% 折扣。
  • 用户登录后：分桶从其真实 user_id重新计算。他们看到 15%。同一商品的价格在他们面前瞬间变化了 5 个百分点。他们截图并在社交媒体上投诉。
  • 一个 B2B 租户在实验进行中将账户计划从"free"更改为"pro"。若分桶规则依赖于计划，则分桶在实验中途发生变化。

真正的修法： 分层身份 + 长期实验的粘性覆盖。

  • 按用例定义稳定的身份。对于消费者应用： user_id 用于已登录用户， device_id 用于匿名用户，并在首次登录时进行关联迁移（系统将两者关联）。
  • 按 稳定 属性（user_id、account_id）分桶。永远不要按 可变 属性（计划、国家、应用版本）分桶。可变属性是 条件，而不是分桶键。
  • 对于罕见的长期实验（年度保留期、监管队列），在粘性分配存储中显式持久化分配。表很小，只包含那些分配必须在深层变更后仍存活的用户。

问题 4：交叉（重叠）实验

排序实验场景：Feed Team A 正在测试新的 tie-breaker；Feed Team B 正在测试新的时效性提升。两者同时在 Feed 上运行。

朴素的解决方案： 每个实验按 (user_id, flag_key) 哈希，且相互独立。全部并行运行。

为什么会出问题： 同一界面上的两个实验可能相互影响。随机处于（A_new_tiebreaker, B_freshness_boost）对比（A_old, B_no_boost）的用户会混淆两个实验——两者都无法干净地隔离各自的效果。来自 Uber/Netflix 复盘的真实示例模式：两个首页实验各自单独测试时均显示 1% 提升，但两种处理同时触发时的联合效果实际上是 -3%（改动在视觉上相互冲突）。每个团队都发布了，总体业务指标下降，没有人能解释原因。

真正的修法： 互斥层。定义层级（feed_ranker_layer, feed_layout_layer, checkout_layer，等等）。在一个层内，每个用户被分配到 恰好一个 实验。分层分配使用不同的分桶方案： bucket(user_id, layer_id) -> segment -> experiment。同一层内的两个实验不能在同一用户上运行。跨层时，实验是相互独立的。

Google 的"Overlapping Experiment Infrastructure"论文（Tang 等，2010 年）是权威参考。Netflix ABlaze、Uber XP、LIX 和 Statsig 都实现了分层分配的变体。

问题 5：发布安全性

移动功能发布场景：随 App Store 二进制更新在 iOS 上发布的新应用内支付流程。

朴素的解决方案： dashboard 滑块 0% -> 100%，值班工程师拖动它。

为什么会出问题： - 没有自动护栏。1% 时的 bug 仍会损失 1% 的收入，以及 100% 受影响用户的信任。 - 手动发布发生在工作时间；bug 在凌晨 3 点才暴露。没有人负责回滚。 - 变更瞬间全球生效——没有区域节奏控制，因此仅限 US 的 canary 无法在 APAC 用户遇到 bug 之前发现问题。 - 移动应用版本是一个混淆因素：10% 使用 app v5.2 的用户可能都会看到新流程；下一个 App Store 的 v5.3 版本会悄悄地更换不同的用户群体。

真正的修法： 自动化发布策略 + 护栏指标 + 自动回滚。发布步骤是数据： [1%, 5%, 25%, 50%, 100%] 与 hold_duration_minutes 步骤之间。护栏指标（5xx 错误率、p99 延迟、转化率、崩溃率）附带阈值。发布控制器在每个步骤轮询指标；阈值触发时，自动回滚至上一步骤（或 0%）。发布按 逐区域顺序 ——先 us-west，然后 eu-west，再 ap-southeast。

LIX 有明确的"auto-ramp"功能，附带指标护栏。Statsig 称之为"Pulse"实验。LaunchDarkly 有"Release Guardrails"。模式相同，名称不同。

问题 6：Flag 债务——死于千个陈旧 flag

所有 flag 系统都面临的通用场景。

朴素的解决方案： 依靠工程师自行清理。

为什么会出问题： 在 100k 个 flag 中，约 70k 个在 100% 发布状态下停留超过六个月。代码分支仍然存在。每次代码变更都需要考虑两个分支。测试矩阵急剧膨胀。初级工程师不知道哪个分支是"活跃"的。最终，一个在 100% 状态停留两年的 flag 被意外关闭——生产立即中断，因为 OFF 分支多年未经测试，且引用了一个已不存在的数据库列。

来自多家公司的真实复盘：一个功能 flag 在 100% 状态停留 18 个月；一位新工程师将其关闭"以测试某些功能"，生产环境中断。

真正的修法： 将生命周期管理作为一等公民概念。

  • 每个 flag 在创建时都有 Time-to-Live（TTL）。在 100% 发布状态下 30 天后，自动标记为待清理。
  • 夜间机器人开启 PR，移除死亡分支和 flag。
  • dashboard 显示每个团队的 flag 债务排行榜。拥有最多陈旧 flag 的团队在清理之前被阻止创建新 flag。
  • 静态分析：linter 对任何 100% 状态超过 N 天的 flag 检查进行标记。引用已清理 flag 的新代码会导致 CI 失败。

这是大多数 flag 系统教程所忽略的部分。在规模化场景下，这是 #1 的维护负担。

§8. 实验的统计分析

eval 引擎和曝光流水线为你提供了确定性的 A/B 机制。它们 非 不会为你提供正确的因果答案。其上的统计层——显著性检验、功效分析、多重检验校正——才是将"我们运行了一个实验"转变为"我们有信心发布"的关键。这一层出错是技术上完美的实验平台产生错误业务决策的最常见原因。

8a. 频率主义 t 检验 vs 贝叶斯后验

两种分析流派，均在主要平台的生产中使用。

频率主义（双样本 t 检验） ——大多数平台的默认方案。计算实验组和对照组均值之差，除以合并标准误，得到 t 统计量；转换为 p-value（在零假设——即无效果——成立的情况下，观察到至少这么大差异的概率）。若 p < α （通常是 α = 0.05).

优点：简单、易于理解、有数十年文献支持，监管机构认可。 缺点：p-value 不是"效果真实存在的概率"（常见误解），且阈值是任意的。置信区间需要仔细解释——"95% CI 为 [0.2%, 1.8%] 提升"意味着"如果我们无限次运行该实验，95% 的结果区间将包含真实效果"，而不是"真实效果在该区间内的概率为 95%"。

贝叶斯后验概率 ——GrowthBook（默认）、Netflix 以及越来越多的 LinkedIn LIX 所青睐。从效果大小的先验分布开始（通常是以 0 为中心的弱信息正态分布）。通过贝叶斯规则用观测数据更新。报告 P(treatment > control | data) ——这正是决策者真正想要的概率。

优点：直接回答"我应该发布吗？"—— P(lift > 0) > 0.95 是一个可辩护的决策规则。自然地处理早停（无 p-hacking 惩罚）。可跨实验组合。 缺点：依赖先验选择；小样本可能较敏感。对需要 p-value 的监管机构来说更难解释。

何时使用哪种方法。 当决策是顺序进行的，且希望清晰表达"我对提升量有多大把握"时，使用贝叶斯方法。当需要符合监管要求的答案（金融监管下的定价实验、医疗/健康功能），需要预先注册 alpha 阈值，或受众习惯于频率主义输出时，使用频率主义方法。

混合方法在规模化场景下是常态——LIX、Netflix 和 Statsig 都在同一实验 dashboard 上报告 新旧两个 频率主义置信区间和贝叶斯后验概率，让实验者相互验证。

8b. 功效分析：检测 1% 提升量在 95% 置信度下需要多少用户？

每个实验者都应能在白板面试中推导出的公式。为了检测最小可检测效应量（MDE） δ ，具有功效 1 - β （通常为 0.8），在显著性水平 α （通常为 0.05）下，对基线转化率为 p:

n_per_variant ≈ ( z_{α/2} + z_β )² × 2 × p × (1 - p) / δ²

具体数字：基线 checkout 转化率 p = 5%，MDE δ = 0.05 × 0.01 = 0.0005 （1% 相对提升）， z_{0.025} = 1.96, z_{0.2} = 0.84:

n ≈ (1.96 + 0.84)² × 2 × 0.05 × 0.95 / (0.0005)²
n ≈ 7.84 × 2 × 0.0475 / 0.00000025
n ≈ ~2.97M per variant ≈ ~5.94M total exposures

在 5% 基线转化率上检测 1% 相对提升需要约 600 万次曝光。在 1M 日活跃用户（DAU）中 30% 暴露于 checkout 的情况下，这需要 约 20 天 的运行时间。那些因"一周后无显著结果"而被终止的实验，有一半根本没有足够的样本量来检测其 MDE。

操作规则：每个实验设计必须包含 预先计算的样本量估计。若平台的功效计算器显示需要 20 天，请勿在第 7 天就宣判实验无效。你在第 7 天观察到的 0.5% 提升可能就是真实效果——你只是还没有收集到足够的证据。

对于连续指标（每用户收入、会话时长），将 p(1-p) 替换为观测方差 σ²。对于比率指标（CTR = 点击次数 / 展示次数），使用 delta 方法计算比率的方差。现代平台（LIX、Statsig）已将此自动化。

8c. "0.5% 提升，不显著"问题

实验者的常见抱怨： "我们运行了一周，看到 0.5% 的提升，但不显著。我们应该发布吗？"

诚实的答案需要将"不显著"转化为其背后的真相，这有三种可能性：

  1. 真实效果为零。 0.5% 是噪声。发布在预期上什么都不做，你已经花了一个迭代周期在一个无效操作上。
  2. 真实效果很小（例如 0.3%）但真实存在。 实验没有足够的功效来检测它。置信区间大致为 [-0.4%, 1.4%] ——较宽且包含零，但也包含有意义的正向提升。
  3. 真实效果很大（例如 1.5%），但观测到的点估计不幸偏低。 样本量较小时区间较宽；你恰好抽到了一个偏低的实现。

平台应该显示 置信区间，而不仅仅是"显著：是/否"。置信区间为 [-0.4%, 1.4%] 所传达的信息与 [0.45%, 0.55%] （两者的点估计均为 0.5%，但第二个是精确测量的零效应，而第一个则反映了对可能存在实质性效应的不确定性）。

可扩展的决策框架："要使这个实验净值为正并值得上线，我需要对真实效应持有怎样的信念？"结合功能维护成本综合考量。对于价值 1 亿美元/年指标上 0.5% 的提升，意味着每年增收 50 万美元；若功能维护成本为 20 万美元/年，即便证据不充分也应上线。对于价值 100 万美元/年指标上 0.5% 的提升，意味着每年仅增收 5000 美元；即便有充分证据也不应上线。

8d. 偷看数据——频率主义检验的大忌

朴素的工作流程：运行实验，每天查看 p 值，当 p 值低于 0.05 时上线。

为何会出问题：每次偷看都是一次假设检验。若真实效应为零，p 值会上下波动——有时低于 0.05，大多数时候高于。偷看 N 次并在第一次低于阈值时停下， p < 0.05 会将误判率从 5% 膨胀至 N=10 次偷看时的约 30%（若每天偷看一个月则高达约 70%）。

具体示例：100 个 A/A 测试（实验组与对照组完全相同，真实效应恰好为零）。在结束时仅做一次观察，按定义约有 5 个误判。若 30 天内每天偷看， 100 个 A/A 测试中约有 70 个 在其运行过程中某时刻显示出"显著性"。你将上线 70% 的无效变更。

真正的修法： 三种方法之一。

  1. 预先注册分析日期。 预先决定"我们在第 14 天进行分析，之前不做任何分析。"在运行期间，仪表板上只显示"实验剩余运行时间"计数器，不展示 p 值。LinkedIn 在高风险定价实验中采用此方式。

  2. 序贯检验 / 始终有效的 p 值。 使用专为持续监控设计的检验方法——序贯概率比检验（SPRT）、msPRT（混合 SPRT）或始终有效的置信序列。这些方法动态调整阈值，使偷看不会膨胀错误率。Statsig、Optimizely 和 LIX 已推出相关变体实现。代价是：与固定期望检验相比，每个固定样本量的统计功效略低。

  3. 贝叶斯决策规则与显式损失函数。 贝叶斯后验概率不会以相同方式受到偷看膨胀问题的影响；你可以持续更新。这是 GrowthBook 的默认工作流程。

平台应使偷看要么不可能，要么安全无害。在实时实验仪表板上展示原始频率主义 p 值且不加任何警告，是一种不负责任的做法。

8e. 多重检验校正

若你在一个实验中检验 20 个指标并对每个检验使用 α = 0.05 每次检验的显著性水平，则全局误判率为 1 - (1 - 0.05)^20 ≈ 64%。即使什么都没出问题，在大多数实验中你也会偶然发现某个"显著"的护栏指标违规。

Bonferroni 校正： 将 α 除以检验次数。 α' = 0.05 / 20 = 0.0025。方法简单但保守——会损失一些统计功效。

Benjamini-Hochberg（BH）方法： 控制错误发现率（FDR）而非全局误判率。将 p 值从小到大排序；若第 k 个 p 值满足 p_k ≤ (k/m) × α，则拒绝零假设，其中 m 是检验总数。当存在许多真实效应时，其统计功效远高于 Bonferroni 方法。

真实平台（LIX、Statsig、Netflix ABlaze）默认对护栏指标面板自动应用 BH 校正。仪表板同时显示"BH 校正后 p 值"和原始值，决策规则应始终基于校正后的数值。

多重检验的另一个维度： 同时运行多个实验。 若你的平台每季度在 α = 0.05的显著性水平下运行 1000 个实验，即便没有任何功能产生任何效果，你每季度也预计会出现约 50 个误判。这就是为什么护栏专属指标、A/A 测试脚手架，以及将上线决策与 实际 效应量（而非仅统计显著性）挂钩，在规模化场景下至关重要。

§9. 方差缩减——最低成本的统计功效翻倍方法

若检测 MDE 所需的样本量与 1/variance成正比，那么将方差减半即可将所需样本量减半——相当于实验速度提升 2 倍。最常用且最有效的技术是 CUPED （使用实验前数据的受控实验，发音"cup-ed"），由 Microsoft Bing 提出。

9a. CUPED：一段话说清数学原理

对于每个用户，你拥有目标指标的实验前值 X （例如，实验前 4 周的人均收入）和实验中的值 Y。原始实验估计量为 Y_treatment - Y_control。CUPED 估计量为 Y' = Y - θ(X - E[X])，则拒绝零假设，其中 θ = Cov(Y, X) / Var(X) （Y 对 X 的 OLS 斜率）。处理效应为 Y'_treatment - Y'_control。因为 X 与处理分配无关（它早于实验），减去 θX 可以消除源于用户预先存在差异的方差，而不会引入偏差。

方差缩减因子： 1 - ρ²，则拒绝零假设，其中 ρ 是 X 。 Y之间的相关系数。对于 ρ ≈ 0.7 的收入指标，CUPED 将方差缩减 50%——在相同 MDE 下所需样本量减半。对于 ρ ≈ 0.5 的会话次数指标，方差缩减为 25%。

Microsoft Bing 公布的结果： CUPED 将其许多实验所需的样本量缩减了约 2 倍。Netflix 和 LinkedIn 也公布了类似数字（部分指标可达 3 倍缩减）。对于一个每年运行 1000 个实验的平台而言，这意味着全面将实验运行时间成本减半。

9b. 分层抽样

在实验前，将用户划分为若干层（例如，高/中/低活跃度、地区、设备平台）。对随机分配进行分层，使每层都获得目标比例的实验组与对照组（而非仅基于总体人口）。层内方差小于全局方差，分层估计量的方差大致为 Σ (n_s / n) × var_s 而非 var_global.

方差缩减程度取决于指标方差有多少来自层间与层内。对于收入指标按"已登录用户 vs 匿名用户"分层，层间方差巨大——分层可大幅降低噪声。对于基于随机哈希的分层，则毫无效果。

大多数实验平台默认对一小组稳定属性进行分层：国家、平台（iOS/Android/web）和活跃度分位数。将分层与 CUPED 结合使用，可获得乘法式方差缩减效果。

9c. 其他方差缩减技术

  • 方差加权回归 ——对于比率指标（CTR），用各观测值方差的倒数进行加权。
  • Interleaving ——用于排序实验，在同一页面上混合来自变体 A 和变体 B 的结果，观察用户点击哪个。用户充当自身的对照，完全消除用户间方差。Netflix 将其用于个性化实验。
  • 合成控制 / 差分法（Difference-in-differences） ——用于无法随机化的准实验（例如，按国家逐步推出功能），从未处理地区构造合成对照，比较趋势。

方差缩减的收益大致相当于将 DAU 翻倍的收益：每种将方差减半的技术，都能让你更快地上线功能、更快地终止不良实验，并能同时运行更多实验。 在 Microsoft Bing，单独使用 CUPED 所带来的经量化的收益，相当于额外获取了 50%–100% 的用户流量。

§10. 样本比率不匹配（SRM）检测

若你以 50/50 的比例将用户分配到对照组和实验组，你预期在曝光日志中观察到大约 50/50 的比例。若你观察到 48/52，说明出了问题——无论指标数据显示什么，该实验均 无效，无论指标数字如何显示。

10a. SRM 为何重要

SDK 中的分桶 bug、过滤掉部分实验组曝光的日志过滤器、将部分实验组用户回退到对照组的路由 bug、下游服务对两个变体请求处理方式不同——这些问题都会表现为样本比率不匹配。下游指标数据随后成为 不可比人群 的比较（实验组现在是存活于过滤器的用户子集；对照组是未经过滤的人群）。任何差异都可能是选择效应，而非功能本身的效果。

真实案例：某实验平台团队发现其 treatment 结账流程的某个变体仅占总曝光量的 47%，而非预期的 50%。调查发现一个 bug：若用户使用 app 版本 v5.2.1 且语言设置不为"en"，SDK 就会未能记录曝光。仅实验组代码路径触发了该 bug；对照组代码路径未受影响。实验组因此系统性地排除了一个子群体。在指标中观察到的 1% 提升，是实验组剩余用户构成的产物，并非真实的因果效应。

10b. SRM 检验

该检验是卡方拟合优度检验。给定分配比例（50/50、33/33/34 等）以及各变体的实际曝光计数：

χ² = Σ (observed - expected)² / expected

对于分配比例 50/50、实际观测到的两个变体， (48k, 52k):

expected = (50k, 50k)
χ² = (48k - 50k)² / 50k + (52k - 50k)² / 50k = 80 + 80 = 160

自由度为 1，p 值 ≈ 10^-36。 在真实 50/50 分配下观察到如此大的不平衡的概率基本为零。说明出了问题。

平台应在展示任何指标结果之前 自动 运行该检验，并在检测到 SRM 时 阻断 阻止实验仪表板显示结果。Microsoft、Netflix 和 LIX 均采用此做法。Statsig 会在任何受影响实验的顶部显示 SRM 警告横幅。

10c. SRM 的常见原因

  • 分桶 bug ——盐值错误、哈希函数错误、差一错误（off-by-one）。
  • 日志过滤器 ——去重过滤器或限速器对不同变体的事件丢弃方式不同。
  • 不对称错误率 ——实验组崩溃更频繁，导致实验组用户无法到达曝光记录点。
  • 不对称资格检查 ——对照组能访问某功能，但实验组还需通过额外检查，部分用户被过滤掉。
  • 前序实验的遗留效应 ——被前序实验固定分配的用户，因前序分配仍然生效，落入了不同的变体桶。
  • 时段偏斜 ——若实验在某地区高峰时段启动，实验组可能不成比例地接触到来自某一地理位置的大量用户。

修复模式：修复 bug， 丢弃在故障期间收集的所有数据，以相同配置重启实验（这样重新曝光的用户会回到相同的变体——确定性在此帮了你大忙）。一个允许你"忽略"SRM 并直接上线的平台，实际上是在上线有问题的实验。

§11. 网络效应与干扰

A/B 测试从根本上假设稳定单元处理值假设（SUTVA）成立：用户 A 的结果仅取决于用户 A 的处理分配。社交、市场和内容平台系统性地违反了这一假设。

11a. 干扰问题

LinkedIn / Facebook / Twitter 中的具体模式：

  • Alice 在 实验组：一个新的"长帖子"功能让她可以撰写 5000 字的帖子。她发帖更频繁了。
  • Bob 在 对照组：他仍受旧的 1000 字限制。但 Bob 与 Alice 相连，因此他的信息流现在显示了更多（更长的）帖子，因为 Alice 发帖更多了。
  • Bob 花更多时间阅读 Alice 的帖子。Bob 的会话时长增加了。
  • 实验仪表板显示："对照组会话时长相对于实验组增加了 2%。"结论：该功能 损害了 用户参与度。上线被否决。

这是错误的结论。该功能实际上 提升了 用户参与度——Bob 因为 Alice 处于实验组而阅读更多内容。但 Bob 在对照组中，所以他的改善被计入了对照组。处理效应通过社交图谱 泄漏 到了对照组。

市场平台中存在类似模式：实验组买家看到更好的定价，购买更多，耗尽库存；对照组买家看到耗尽的库存，购买减少。实验组看起来"很好"；平台整体看起来"中性"，因为它只是在不同桶之间转移购买行为，而非创造新的购买。

11b. 集群随机化

解决方案：将用户 集群 而非个体分配到某个变体。集群是一组内部交互为主的用户群体——好友群、地理区域、社区。

  • 地理随机化 ——按国家、州或城市分配。德国用户为实验组；法国用户为对照组。实验组与对照组之间的干扰现在被限制在（少量的）跨境活动范围内。
  • 社交集群随机化 ——运行图分区算法（例如 Louvain 模块度算法）找出社区，将整个社区分配到同一变体。
  • 基于时间的随机化 ——实验组运行一周，对照组运行一周，然后再运行实验组，比较地区内趋势。避免溢出效应，代价是与星期内时间效应产生混淆。

代价：集群随机化在单位流量下的统计功效大幅下降。若你有 100 个地理集群，想在 95% 置信度下检测效应，你实际上只有 100 个数据点，而非 1 亿用户。所需样本量大约按集群数与用户数之比进行放大。

11c. "无法干净地对信息流算法做 A/B 测试"的痛点

这是 Facebook、Twitter、LinkedIn 和 Pinterest 均记录在案的行业痛点。信息流排序变更会影响内容生产（创作者看到参与度，调整发帖行为）和内容消费（观看者看到不同信息流，调整会话行为）。双方通过社交图谱相互影响。

行业应对方案：

  • 长期实验 ——针对信息流变更进行多周或多月的实验，让网络达到均衡状态。
  • 生产者侧指标 ——与消费者侧分开监控——将创作者行为作为护栏指标追踪；若实验组导致创作者减少发帖，该抑制效应会掩盖真实的消费者影响。
  • 切换回测（Switchback）实验 ——每隔 N 小时全局交替算法；比较相邻时间窗口。信息流适用此方法，因为均衡状态能快速重建。
  • 反事实日志 ——在实际服务变体的同时记录"另一变体会展示什么"；对反事实数据的离线分析完全避免了干扰问题。存储和计算开销大，Netflix 和 Spotify 在推荐实验中普遍采用。

血泪经验：任何在对社交图谱或市场功能做 A/B 测试时不承认干扰问题的平台，都会产生系统性错误的结论。主流实验平台现已将干扰感知分析模式作为一级功能推出，当实验范围涉及触碰社交图谱的界面时，仪表板会显示明确警告。

§12. Holdout 与全局对照

每个实验都是局部优化："这个变更比当前状态更好吗？"在每年数百个实验的累积下，平台会产生漂移。每次单独的上线决策都有据可查，但累积效应却未可知。Holdout 用来回答累积层面的问题。

12a. 全局 holdout 模式

将 1% 的用户 排除在 所有 所有实验之外。这 1% 的用户看到的是平台在 holdout 创建时的状态——过去一年所有已推广实验对他们均不适用。其余 99% 的用户则看到每个已上线实验的处理版本。

一年后，将 99% 与 1% 进行比较。差异即为 过去一年所有已上线变更的累积影响。若 99% 组的人均收入高出 5%，则实验整体净正向。若 99% 组的人均收入低 2%，则平台已陷入局部最优——每个单独实验看起来都不错，但整体表现更差。

12b. 为何重要

局部最优确实存在：一个 UX 简化上线了，一个通知频率调整上线了，一个排序微调上线了——每次都有局部提升的支撑，但合在一起可能会降低用户体验（通知疲劳、推荐信息茧房、内容发现减少）。上线流程天然偏向于展示短期指标改善的变更；长期来看，那些低参与度但更健康的行为模式被系统性地过滤掉了。

Netflix 公开讨论 holdout 的做法：他们维护多年期 holdout，并用其识别出那些在局部实验中表现为正向、但在 6 个月维度上为负向的变更。

12c. 实现方式

holdout 本身是一个标记—— global_holdout ——分配给 1% 的用户。其他每个实验的资格规则都隐含了 AND user NOT IN global_holdout这一条件。平台在仪表板层面强制执行：实验创建者无法选择退出 holdout 排除机制。

用户身份必须保持多年稳定——分桶盐值固定，user_id 是稳定身份标识，holdout 永不重新随机化。年中加入的新用户在首次曝光时被确定性分配。

Holdout 与实验者的需求存在冲突（所有人都希望实验覆盖 100% 的符合条件用户），也与功能工程需求存在冲突（大型功能的灰度发布希望达到 100%）。平台必须将 holdout 上限作为不可商量的规则强制执行，并清晰传达其背后的权衡：你用 1% 用户的即时体验，换取对长期方向的诚实度量。

§13. 实验元数据与设计规范

机构层面的失败模式：一个团队一年内运行了 100 个实验，一年后却无法回忆起其中大多数实验测试了什么、为什么测试。平台保留了变体数据，却丢失了 科学背景。实验变成了无法解读的历史遗产。

13a. 预注册：设计文档作为门禁

每个实验在启动前，必须在平台本身填写一份设计文档。字段包括：

  • 假设 ——一句话："我们相信展示 X 将因 Z 而导致 Y。"强制进行因果思考，而不仅仅是"看看会发生什么"。
  • 主要指标 ——决定上线/不上线的单一数字。必须是指标平台中已有的、定义上达成共识的指标（不允许即席 SQL）。
  • 护栏指标 ——即便主要指标正向移动，这些指标也不能负向变动（例如崩溃率、p99 延迟、客服工单、收入、公平性指标）。
  • 样本量估计 / MDE ——由平台的功效计算器基于主要指标的基准方差预先计算得出。
  • 运行时长 ——基于样本量和 DAU 计算得出。平台拒绝让你在此日期之前停止实验（偷看锁定）。
  • 决策标准 ——明确指定："若主要指标提升 ≥0.5%，且 p < 0.05 Bonferroni 校正后无护栏违规则上线。若主要指标移动但护栏违规则迭代。若主要指标未移动或负向移动则终止。"
  • 负责人与评审人 ——工程负责人、数据科学评审人（不同人员）、产品经理发起人。

13b. 平台强制执行

若设计文档不完整，"启动实验"按钮处于禁用状态。平台将文档与实验一同存储；文档可查询、可链接，并与指标结果并排展示在仪表板上。

实验完成后，平台要求填写 决策报告："主要指标提升 +0.3%（低于我们 0.5% 的阈值），无护栏违规。我们正在使用更强的处理方案进行迭代。"与实验记录一同存档。

LinkedIn 内部实践：每个已上线实验都有一个"决策依据"条目，可供第三方审计。审计内容可查询，例如"显示 2025 年所有已上线定价实验及其假设、主要指标和决策依据"。

13c. 取证价值

六个月后，有人问道："为什么结账转化率下降了？"调查人员查询："显示过去 6 个月内所有涉及结账界面的实验。"有了预注册，他们会得到 N 个实验的列表，每个实验均包含假设、指标和决策。他们可以快速定位候选原因。

若没有预注册，同样的调查需要依靠团队成员的记忆进行访谈："你还记得大概四月份有个实验吗？它测试了什么？指标是什么？"大部分机构知识已经消失了。

该模式与医学中的临床试验预注册（NIH ClinicalTrials.gov）如出一辙，原因也相同：它防止了事后重新叙事、p 值操纵和指标筛选。 没有强制预注册的实验平台，产出的是测量表演，而非科学。

§14. 故障模式全程剖析

前几节涵盖了统计层面的故障模式（偷看、SRM、干扰）。本节涵盖 系统故障模式：在评估/分发/曝光层面发生故障时，架构如何吸收这些故障。

故障1：部署了错误的规则（经典的"哦不"场景）

有人将 checkout-v2 从 50% 提升至 100%，而新代码存在一个 bug，导致 app v5.2 的 iOS 用户受到影响。

检测方式：实时护栏监控器（Kafka 曝光流上的 Flink 作业）+ 崩溃报告 + 来自应用性能监控（APM）的 5xx 错误率。三者应在 60 秒内达成一致。

恢复：值班人员在控制台点击"kill switch"。 checkout-v2 规则被重写为 ELSE -> control。通过 SSE 扇出，在 10 秒内完成传播。iOS 客户端在下次请求时将看到对照组版本。

崩溃前最后一个完整写入的批次。 记录 kill 操作的审计日志条目。在传播开始前已写入 S3 + Kafka，因此即使传播中途失败，该 意图 也是持久的，运维人员可以验证状态恢复情况。

故障2：App 服务器在 eval 过程中崩溃

App 服务器在执行过程中遭到 SIGKILL getVariant("checkout-v2", user).

恢复：微不足道。Eval 是一个纯函数，除环形缓冲区写入外没有任何副作用。用户请求超时，负载均衡器将流量路由到另一台服务器，eval 确定性地重新计算。相同的 user_id + 相同的 flag_key + 相同的规则集 = 相同的结果。无状态丢失。

崩溃前最后一个完整写入的批次。 无需任何措施。确定性本身就是持久性的保障。

故障3：App 服务器无法访问配置平面（冷启动、分区）

新启动的 App 服务器，其路由所指向的配置平面 edge PoP 宕机。

恢复流程：1. SDK 尝试主 edge PoP — 2 秒后超时。2. SDK 尝试另一区域的备用 edge PoP — 假设同样不可达。3. SDK 回退至 本地磁盘规则集缓存 处传递 /var/cache/<system>/ruleset.bin。最后已知良好的快照，每日同步保持更新。4. 如果连这也失败（全新机器，无缓存），SDK 进入 纯默认模式 — 每次 getVariant 均返回调用方提供的默认值。App 服务器降级运行（无定向，无实验），但仍可处理请求。5. 后台重连循环，每 30 秒进行一次指数退避。

崩溃前最后一个完整写入的批次。 本地磁盘缓存文件。通过 rename(2)进行原子替换。即使部分写入也不会导致数据损坏——最坏情况下，文件保留的是上一个有效快照。

故障4：配置平面数据库主节点丢失

标志元数据存储中的 MySQL/Postgres 主节点崩溃。半同步副本的数据落后仅数毫秒。

恢复：1. 数据库集群管理器（Orchestrator、MHA、Vitess、Patroni）提升副本。控制台写入被阻塞 5–30 秒。2. 变更流工作进程重新连接到新主节点的 binlog/WAL 位置。若位置不匹配（半同步模式下极为罕见），工作进程从最后提交的偏移量读取并重放遗漏的事件。3. SDK 不受影响——它们从 edge 缓存读取，而非直接访问数据库。

崩溃前最后一个完整写入的批次。 binlog/WAL。一旦变更提交到 N 个副本（半同步）的日志中，即可在主节点丢失后存活。

故障5：网络分区——某区域无法访问全局控制平面

亚太区与全局控制平面断开连接。控制台中的新标志变更无法到达 AP edge 节点。

恢复：AP edge 节点继续向 AP App 服务器提供 最后已知良好的规则集 。AP 区域以分区发生时刻冻结的配置运行。读操作持续正常；新变更在分区恢复前无法传播到 AP。恢复后，edge 节点从最后的版本游标开始重放变更流。不存在合并冲突，因为标志定义由全局控制平面拥有——AP edge 节点只读不写。

明确说明的权衡： 在分区期间，AP 使用的是过时的规则。分区期间触发的 kill switch 在恢复之前无法到达 AP。对于标志系统而言，这是正确的选择—— 分区期间可用性优先于一致性 ，因为另一种选择（"AP 服务器拒绝评估标志"）会导致 AP 的所有请求失败。

对于极少数 AP 特定的分区期间 kill switch 需求，存在带手动覆盖功能的区域控制平面镜像（实际上几乎从不使用，但可满足合规要求）。

崩溃前最后一个完整写入的批次。 每个 edge 节点上的版本游标。"我当前处于版本 17421"使每个 edge 节点能够确定性地重新同步。

故障6：edge PoP 永久丢失

eu-west edge PoP 硬件损毁。

恢复：微不足道。Edge 节点是全局控制平面的无状态缓存。新的 edge 实例启动，从源站（或其他 edge 节点）同步规则集，接受 SDK 连接。SDK 通过 SDK 内置的 edge 列表进行故障转移。

崩溃前最后一个完整写入的批次。 edge 节点不持有任何持久状态。源站中由 S3 支持的规则集快照是唯一的真实来源。

§15. 回滚安全性与 kill-switch 延迟

一个需要 12 分钟才能传播的 kill switch 根本不算 kill switch——它只不过是一次稍微快一点的部署。围绕回滚的操作 SLA 决定了整个传播平面的设计。

15a. 延迟预算

行业 SLA 目标，通常在故障复盘时向高管承诺：

  • p50 传播延迟：< 2 秒 ——从值班人员点击控制台到超过 99% 的生产服务器提供新值。
  • p99 传播延迟：< 10 秒 ——考虑网络慢、垃圾回收暂停以及 edge PoP 故障转移的情况。
  • 最坏情况传播延迟：< 60 秒 ——即使发生分区恢复或 edge PoP 冷启动。

凌晨3点的故障模式使其成为硬性 SLA："我们凌晨 3 点部署，指标出现异常，回滚耗时 12 分钟。这意味着 12 分钟的结账中断 × 亚洲凌晨 3 点的峰值流量 = 损失 $X 营收。"复盘结论：kill-switch 工具需要提交 JIRA 工单并执行部署。修复方案：在控制台构建一键 kill 功能，并为值班人员提供审计记录和审批绕过机制。

15b. 延迟预算带来的收益

传播路径上的每个组件都有自己的预算：

On-caller click → control plane write:        50 ms
Control plane write → change stream:           200 ms
Change stream → regional edge PoPs:            500 ms
Edge PoPs → SDKs via SSE:                       1000 ms
SDK applies new ruleset (pointer swap):        10 ms
Application code observes new value:           per-request
TOTAL p50:                                     ~1.8 seconds

任何超出预算的组件都需要调查。如果控制平面写入耗时 2 秒（可能是数据库积压），仅这一个组件就会击穿 SLA。控制台必须展示"距上次传播的时间"，使运维人员能感知系统健康状况。

15c. Kill-switch 的用户体验

Kill switch 不仅仅是一个 API 调用——它是一个能够在凌晨 3 点肾上腺素飙升的值班人员手中正常工作的 UX。

  • 大红按钮。 在视觉上与"灰度至 50%"或"修改变体比例"明显区分。
  • 无强制确认弹窗 用于 kill 操作。将"决定 kill"到"完成 kill"的延迟缩短 5 秒，在故障期间至关重要。审计在事后处理问责。
  • 为 kill 操作绕过审批流程。 初级值班工程师不应该需要叫醒高级工程师来批准 kill 操作。审计日志记录谁执行了 kill；审批是事后进行的。
  • 对 un-kill 操作进行频率限制。 Kill 只需一键；un-kill（kill 之后重新启用）需要第二人审批或 5 分钟冷却期。不对称摩擦：易于 kill，难以 un-kill。
  • 可见状态。 每个标志的控制台显著展示当前灰度比例、最后变更时间、最后操作者及当前健康状态。无需深入菜单。

LinkedIn 的 LIX 有一个有据可查的事故：kill UX 是设置面板中的一个复选框，需要三次点击才能访问。在一次持续 4 分钟并造成营收影响的故障后，团队将其重新设计为标志主页顶部显著的"禁用"按钮。从故障检测到点击 kill 的时间从约 3 分钟缩短至 30 秒以内。

15d. 自动回滚与指标护栏循环

手动 kill 是最后手段。自动回滚才是第一道防线。

模式：每个实验注册 护栏指标 与 自动 kill 阈值。实时监控器（Kafka 曝光流上的 Flink）在 1 分钟窗口内计算指标。若护栏被触发，平台将自动：

  1. 暂停实验灰度（不再向处理组添加用户）。
  2. 通知值班人员（PagerDuty、Slack）。
  3. 在可配置的延迟（通常 5 分钟供人工审查）后，自动将灰度降至 0%。

从"处理组出现问题"到"处理组关闭"的完整循环通常在 10 分钟内完成，全程支持人工干预。LIX 的自动灰度、Statsig 的 Pulse 以及 LaunchDarkly 的 Release Guardrails 均实现了这一模式。

权衡：误报。真实的指标抖动（例如区域性 CDN 问题）可能触发对一个无辜实验的自动回滚。平台必须允许团队配置灵敏度，并提供合理的默认值（3 倍标准差偏差持续 5 分钟，而非 30 秒）。

§16. 额外的故障模式——长尾问题

除经典场景外，成熟平台还通过经验积累，针对一长串细微故障部署了防护措施。

16a. 大规模发布期间配置服务过载

场景：一个新功能以 100% 比例向所有用户发布。每个 SDK 同时请求新的规则集版本。源站遭遇同步洪峰。

朴素的后果：源站过载，传播停滞，SDK 回退到默认值，整个集群短暂地提供过时或降级的行为。

真实的修复方案：- Edge 扇出吸收洪峰 ——源站只发出一次变更，由各 edge 节点扇出。正常运行期间，SDK 从不直接访问源站。- SDK 连接抖动 ——SDK 在配置变更后重连时，在拉取前添加一个小的随机延迟（0–5 秒）。防止 edge 节点上的惊群效应。- edge 节点的背压 ——连接数过载的 edge 节点优雅地拒绝新连接，将 SDK 重定向到负载较低的 edge 节点。

16b. SDK bug 导致向所有用户返回错误变体

场景：新 SDK 版本在规则遍历器中存在 bug，对每个标志都返回回退变体，忽略所有规则。已部署至整个集群。

真实后果：每个实验都显示"对照组 100%"，因为所有用户都在回退路径中。整个平台在实验人员看来完全失效。

真实的修复方案：- 跨语言测试向量 ——每个 SDK 在通过 CI 之前都对已发布的 (user_id, flag_key) -> expected_variant 测试集运行 canary 测试。规则遍历器的回归会立即导致测试失败。- 分阶段 SDK 发布 ——新 SDK 版本首先部署到 canary 集群，并运行合成 A/A 实验以确认集群仍能产生 50/50 的分流。真实生产发布跟随 canary 的健康状态进行。- SDK 版本上报 ——每次曝光事件都包含 SDK 版本信息。平台控制台展示"按 SDK 版本划分的曝光量"，使变体分布异常的异常版本清晰可见。

16c. 粘性缓存导致过时变体持续数小时

场景：实验从 50% 灰度降至 0%（kill）。大多数用户立即看到新的（对照组）变体。一小部分用户持续数小时看到旧的（处理组）变体，因为其会话缓存的变体仍存在于浏览器/App 状态中。

真实后果：kill switch 在 eval 层已生效，但下游缓存（HTTP 响应缓存、JavaScript 变量、React 状态）仍持有过时变体。拥有缓存页面的未登录匿名用户会继续体验到已被 kill 的功能。

真实的修复方案：- 将变体查询视为缓存未命中 ——不要在请求之外缓存变体结果。始终在下一个请求边界重新评估。- 标志变更时的 edge 缓存失效 ——标志变更时，精确清除受该标志控制的所有 CDN 缓存内容。要求缓存层了解哪些标志影响了被缓存的响应。- 每次导航时的客户端 eval ——移动/Web SDK 在每次界面入口处重新评估标志，而非每个会话只评估一次。

16d. 基于时间的属性评估漂移

场景：标志规则是 tenure_days > 30关系。其中 tenure_days 由以下内容计算得出 now() - user.created_at。集群中不同服务器计算出的 now() 存在细微差异——时钟偏差从数秒到数十秒不等。

真实后果： created_at 恰好 30 天前的用户在某些服务器上看到处理组（tenure_days = 30.01），在另一些服务器上看到对照组（tenure_days = 29.99）。用户在会话中来回切换变体。实验曝光显示重复计数；指标归因变得嘈杂。

真实的修复方案：- 在服务端对基于时间的属性进行量化 — tenure_days 在曝光时被四舍五入到整天。条件在每个（用户, 天）组合下变得确定性。- 禁止 now() 在面向用户的规则定义中使用 ——规则编译器拒绝引用实时时间的规则。基于时间的资格判断使用预计算属性（is_new_user, tenure_bucket），这些属性在整个集群中保持稳定。- NTP / PTP 时钟同步 ——在运维层面强制要求所有生产服务器之间的时间差在 100 毫秒以内。对漂移超出阈值的主机发出告警。

16e. 分桶 salt 碰撞

场景：两个实验创建时使用了相同的标志键前缀（例如 checkout-v2 。 checkout-v2-mobile）。由于 salt 构造中的 bug，两者使用了相同的有效 salt。在 checkout-v2 处理组中的用户也不成比例地出现在 checkout-v2-mobile 处理组中，导致两者相互干扰。

真实的修复方案：- Salt 碰撞检测 ——平台按计算出的 salt 对所有标志进行索引。创建新标志时，若其 salt 与现有标志冲突，则操作失败并返回明确的错误信息。- Salt 始终包含唯一的标志标识符 ——UUID、单调递增 ID 或带命名空间的键——而不仅仅是人类可读的标志名称。

16f. 曝光日志丢失与重复

场景：曝光事件被推送到 Kafka。生产者批次在传输过程中失败。生产者重试；部分事件被记录两次。

真实后果：实验指标对用户数进行了多计。一个只有一次曝光的用户出现两次；其指标值被双倍加权。

真实的修复方案：- 按 (user_id, flag_key, ts_minute) 去重 ——分析层应用去重窗口。Pinot/Druid 可在数据入库时通过唯一键分段进行去重。- 至少一次投递配合幂等下游 ——生产者采用至少一次投递语义，但消费方负责处理重复数据。反向设计（Kafka 的恰好一次）是可行的，但代价高昂；在分析层进行去重是标准模式。- 采样容忍度 ——在曝光采样生效的高评估频率场景下，去重的重要性相对较低，因为采样本身就会丢失部分事件。平台的设计必须确保丢失/重复的噪声被限制在 MDE 以下。

§17. 为什么不用硬编码 if 语句加 WHITELIST

代码评审中经常被提及的经典朴素替代方案：

if env == "prod" and customer_id in WHITELIST:
    use_new_checkout = True
else:
    use_new_checkout = False

或其部署在二进制文件中的 YAML 亲戚：

features:
  checkout_v2:
    enabled: true
    allowed_customers: [acme, foo_corp, bar_inc]

对一个团队、一个标志、一个产品来说运行良好。但在规模化场景下完全失效。通过六个具体故障来逐一分析。

故障1：无法即时回滚

T=0：部署并携带 checkout_v2.enabled = true。Bug 随之上线。

T=5m：值班人员看到错误率飙升。想要切换标志。

T=5m–25m：编辑 YAML，提交 PR，等待代码审查，合并，触发部署，等待 canary，等待向数千台服务器的全量发布。 超过 20 分钟 才能恢复到安全状态。这段时间内数百万用户受到影响。

标志系统通过 部署与发布解耦解决了这个问题：kill 是一个耗时 1 秒的配置平面操作，而不是部署。

故障2：无法进行百分比灰度发布

T=0： checkout_v2.enabled = true 部署后，立即触达 100% 的用户。

T=5m：bug 影响了 100% 的用户，而非 1%。爆炸半径是整个用户群。

使用标志系统，同样的 bug 在 1% 灰度时就会被发现，用户影响降低 100 倍。

故障3：无法进行定向

PM："能否仅对德国 app 5.3+ 版本的付费用户开启此功能？"

朴素的回答：编写一个自定义 if ，包含三层嵌套条件、硬编码国家代码、版本检查。下次 PM 想要"德国或法国"时，又是一次代码变更 + 部署周期。

标志系统将这变成一次控制台编辑，在数秒内完成传播。

故障4：无法进行实验

PM："新版结账流程实际上转化率更高吗？"

朴素的回答：你不知道。你已经向 100% 的用户发布了。你可以与历史数据进行比较，但你没有同期对照组——任何差异都可能是混淆因素（季节、营销活动、天气）。

标志系统可同时运行对照组与处理组，明确无误地衡量提升效果。

故障5：部署与发布紧耦合

每次发布都伴随着部署。你无法在不向所有人部署的情况下，仅对内部用户在生产环境测试功能 X。你无法在不让用户可见的情况下，暗启动新代码路径以验证性能。

标志系统将这两者解耦：将代码部署至 100% 的服务器，同时将功能保持在 0%；或在仅后端行为层面将功能运行在 100%（收集性能数据），同时将 UI 界面控制在 0%。

故障6：审计与治理

"谁在周六为我们最大的客户开启了 enterprise_sso_v3 ？"在朴素的 YAML 模型中：不翻 YAML 的 git blame 就没人知道，即便如此也只能定位到合并者，而非其意图。标志系统：控制台审计日志，包含操作者、时间戳、理由和工单链接。

这些故障会叠加。在拥有 100 名以上工程师的公司，每一个都是潜在的生产事故。这就是为什么每个超过这一规模的公司都会自建或购买一套标志系统。

§18. 扩展维度

两种不同类型的增长需要不同的解决方案。混淆两者会导致错误的投入方向。

类型1：更多标志、更多应用（分发问题）

从 1k 个标志扩展至 100k 个标志，跨越 100 → 10k → 30k 台 App 服务器。

  • 1k 标志 / 100 服务 / 1 个区域： SDK 每分钟轮询一次，获取完整快照。源站是单个 Postgres + 一个轻量 REST API。带宽消耗微乎其微。
  • 10k 标志 / 1000 服务 / 3 个区域： 基于轮询的快照方案失效（快照约 50 MB，全集群轮询约数十 GB/分钟）。改用通过 SSE 传输的 基于差异的推送 。添加区域 edge 扇出，使源站每次变更只发出一次规范差异。编译规则树以保持 SDK 堆内存占用平稳。
  • 100k 标志 / 30k 服务 / 全球： 分片控制平面（Postgres 按命名空间分区）、各区域 edge PoP、用于冷启动的 S3 签名快照。编译后的规则树变得不可或缺——JSON 解释的 eval 成本将是其 30 倍。分层分配可防止交叉实验相互干扰。

约 10k 标志时的拐点： 基于轮询的分发方案失效。架构必须从"SDK 拉取"转变为"控制平面通过扇出层推送差异"。这是一个结构性变化——SDK 从无状态轮询器转变为有状态订阅者。

类型2：每个标志的评估频率更高（Gatekeeper 场景——eval 成本问题）

某一个标志——比如 news_feed_ranking_v2 ——在每次 Feed 渲染时求值，每天 10 亿次请求 = 每个标志每秒约 12,000 次求值。

  • 每个标志每秒 100 次求值： 内存中的规则树，求值几乎无成本，无需特殊处理。
  • 每个标志每秒 1 万次求值： 进程内仍微不足道。MurmurHash3 加 3 层规则遍历低于微秒级，约占 CPU 的 0.1%。
  • 每个标志每秒 10 万次求值（每台服务器）： 进程内仍可接受；这大致是 Gatekeeper 最热检查的速率，约占一个核心的 10%。
  • 每个标志每秒 100 万次求值（Gatekeeper 级热标志）： 求值成本不可忽视。当一个请求中多个下游调用使用同一标志时，批量求值（求值一次，传播变体）。考虑 JIT 编译规则求值 ——Facebook 实际上在 HHVM 中这样做，为热标志规则生成字节码，使求值仅需寥寥几条指令。

临界点约为每个标志每秒 10 万次求值： 曝光日志成本变得不可忽视。曝光事件必须采样（1/N 日志记录，变体选择对每个用户仍是确定性的，但 log 事件被采样）。这是实验与成本的权衡——在极端速率下，无法负担记录每次求值。

临界点约为每个标志每秒 100 万次求值（全机群范围）： 该标志的 Kafka 曝光 topic 需要独立的分区集；Pinot/Druid 分析表需要按 (flag_key, hour) 子分区，以将查询延迟控制在一秒以内。

两种修复方式的分歧

  • 类型 1 是 分发问题：通过基于推送的差量、编译后的规则树、分片控制平面和边缘扇出来解决。
  • 类型 2 是 求值成本与曝光成本 问题：通过 JIT 求值、曝光采样和分区分析表来解决。

分片元数据存储来解决热标志问题毫无帮助，因为元数据存储不在求值路径上。反之，JIT 编译规则来解决传播缓慢也毫无帮助，因为编译只发生一次而传播是多次的。

带阈值的决策规则：

  • 需要按用户定向？ → 标志系统。无阈值；只要有一次需求，标志系统就是正确的原语。
  • 需要将单一行为变更逐步发布至少于 50% 的用户？ → 标志系统。基于部署的发布无法定向到比部署单元粒度更细的范围（通常按集群，最细约为机群的 10–25%）。
  • 需要衡量某变更对指标的因果效应？ → 带实验功能的标志系统。具备统计显著性流水线的 A/B 框架。无法通过配置管理或部署实现。
  • 需要小于 10 秒的回滚 SLA？ → 标志系统。基于部署的回滚需要数分钟到数十分钟。静态配置可以很快，但缺乏定向能力。
  • 需要强一致性（所有服务在同一时刻看到相同的值）？ → 不适合用标志系统。使用协调部署或版本化配置握手。
  • 月活用户少于 10 万、同时使用的标志少于 5 个、单体应用？ → YAML 或环境变量已够用，无需构建标志平台。
  • 月活用户超过 1000 万或同时运行的实验超过 50 个？ → SaaS 标志平台（Statsig、LaunchDarkly）或自托管（Unleash、GrowthBook）。当 SaaS 费用超过每年 20 万美元时，考虑内部自建。

矩阵迫使人们诚实面对：标志系统是 带按用户定向和分析功能的行为变更的正确原语。它不适合用于 基础设施配置, 强一致性，以及 纯代码发布.

§20. 使用场景图谱

五个领域以五种不同方式使用相同的标志/实验技术。

使用场景 1：后端熔断开关（通用）

领域：任何生产后端服务。具体需求：代码路径的即时回滚，全局范围内小于 10 秒。技术变体：简单布尔标志，无实验，定向仅为"关闭 vs 开启"。仪表板提供一个醒目的红色按钮；审计日志记录操作者。在技术栈中的位置：所有涉及高风险代码路径的请求处理器均受 if (flags.isEnabled("new-pricing-engine", ctx))保护。默认走安全路径。具体案例：Stripe 使用 LaunchDarkly 将每个新支付处理器集成置于熔断开关之后。新 Adyen 集成中的漏洞在仪表板点击后 5 秒内被关闭。

使用场景 2：移动端功能发布（iOS/Android 分阶段发布）

领域：二进制文件通过应用商店延迟触达用户的消费者移动应用。具体需求：对 v5.3 二进制中已存在的功能进行渐进式发布，定向已更新的用户，在监控崩溃率的同时在数天/数周内分步推进。技术变体：带百分比发布 + app_version 条件 + 崩溃率护栏 + 分区域节奏控制的标志。在技术栈中的位置：iOS/Android 应用中的 SDK 持有规则集；应用在功能入口处检查标志。崩溃报告反馈给护栏监控器。具体案例：Uber 通过 Uber XP 发布新拼车 UX。5.3 及以上版本的应用符合条件；发布节奏为 1% → 5% → 25% → 50% → 100%，每步保持 24 小时；崩溃率绝对值上升 0.5 时自动回滚。

使用场景 3：ML 模型 A/B 测试（推荐、排序、评分）

领域：服务模型变体以比较提升效果的 ML 系统。具体需求：确定性的加盐分桶，使同一用户在每次请求中看到相同的模型变体；曝光日志与模型输出指标（点击、观看时长、转化率）关联，用于离线统计显著性分析。技术变体：带多变体的标志，每个变体指向一个模型版本；与指标平台深度集成；分析器中采用 CUPED 方差缩减。在技术栈中的位置：推荐服务在请求时评估标志，调度至模型变体 A 或 B，并记录带请求 ID 的曝光；指标流水线按 user_id + ts将曝光与结果关联。具体案例：Netflix ABlaze、LIX、Uber XP 均常态化运行数十个并发 ML 模型 A/B 测试——Netflix 已公开讨论过在首页、搜索和预告片中同时运行数百个推荐实验。

使用场景 4：定价实验（电商、平台）

领域：测试价格弹性的电商或平台。具体需求：跨多个会话对每位用户保持稳定分桶（避免用户看到价格波动）；细致的身份处理（未登录用户通过 device_id，已登录通过 user_id）；严格的审计日志，因为定价决策在许多司法管辖区受到监管。技术变体：带粘性变体分配、登录时身份延续和合规级审计追踪的标志。在技术栈中的位置：定价服务在价格展示时评估标志；结账服务在购买时再次评估，必须返回相同变体；曝光日志输入价格弹性分析器。具体案例：Airbnb 通过 ERF（实验报告框架）进行定价实验。Booking.com 同时运行约 1,000 个实验，其中许多涉及定价。

使用场景 5：排序实验（Feed、搜索）

领域：排序型产品——新闻 Feed、搜索结果、通知排序。具体需求：分层分配，使同一产品面上的两个排序实验互不干扰；深度指标集成，包括反事实指标（"如果施加处理会带来多少点击"）；带持续数月保留组的长期分析。技术变体：分层分配系统 + 实验调度器 + 离线反事实评估器 + 多月保留组队列存储。在技术栈中的位置：排序服务每次请求向实验平台查询一次用户在所有相关层的分配；记录分配和排序结果；离线任务使用方差缩减计算提升效果。具体案例：Facebook 新闻 Feed 和 LinkedIn Feed 每季度运行数千个排序实验。据公开讨论，LinkedIn 的 Feed 排序产品面在任意时刻有 100+ 个并发实验，分层分配防止相互干扰。

使用场景 6：用户分群（企业付费墙、渐进式启用）

领域：按套餐层级或按租户发布功能的 B2B SaaS。具体需求：基于稳定属性（套餐、tenant_id、客户成功组群）的定向；支持早期访问的细粒度白名单；多租户审计（"高盛在 X 到 Y 日期之间看到了哪些功能"）。技术变体：带丰富属性定向、大型白名单（数千客户）和可供合规审查查询的按租户审计日志的标志。具体案例：Salesforce、Atlassian、Notion 均通过内部标志系统以租户级定向来管控企业功能。此处标志系统兼具功能发布和权益管理的双重职能——但对于硬性权益（付费 vs 免费），记录系统通常是独立的权益服务，而非标志系统。

使用场景 7：区域发布（合规、GDPR、数据驻留）

领域：受区域监管的任何产品。具体需求：按区域定向，并严格保证在一个区域启用的标志不会渗漏到另一个区域；按区域审计；能够在监管审查期间冻结某区域的标志状态。技术变体：将国家/地区条件作为一等公民的标志，以及可配置为拒绝特定标志发布的区域边缘 PoP。具体案例：某全球社交网络先在美国推出新内容推荐算法，待法律审查后再仅在欧盟推出。标志的 country IN {EU_set} 条件在仪表板中置于合规审批工作流之后。

以上七种场景底层技术相同，规则语法、身份处理、曝光流水线深度和统计显著性集成各有不同。

§21. 标志的权限模型与基于角色的访问控制（RBAC）

一家拥有 1,000 名工程师的公司中，标志系统既是代码依赖，也是 治理基础设施 。任何有能力翻转标志的人实际上都在向生产环境部署。权限设计必须反映这一点。

21a. 角色层级

典型大型平台的权限模型：

  • 工程师 ——可在所属团队的命名空间中创建新标志；可自由调整非生产标志；无需审批即可将生产标志发布至 5% 以内；可翻转熔断开关。
  • 产品经理/工程经理 ——可审批 5% 以上的生产发布；可修改运行中实验的变体比例。
  • 值班人员（任意团队） ——无需审批即可翻转所属团队命名空间中的任何熔断开关；翻转跨团队熔断开关需附有书面说明（事后审计）。
  • 平台管理员 ——可创建新命名空间、修改全局发布策略、覆盖锁定。全组织仅 5–10 人。
  • 审计员/SRE ——对所有标志和审计日志具有只读权限；可订阅审计告警。

21b. 生产影响变更的审批工作流

高爆炸半径操作需要多方审批：

  • 发布至 100% （涉及货币化的标志，如定价、结账、广告）——需要产品经理审批加指定审核人。
  • 修改变体比例 （在实验运行中途）——需要数据科学审核（修改会使统计假设失效）。
  • 删除标志 （已在 100%）——需要验证代码分支已移除（静态分析检查），并经过 7 天"墓地"期后才能真正删除。
  • 修改保留组 ——需要平台管理员审批；保留组至关重要，不可随意更改。

平台在仪表板内以内联工作流形式实现审批。点击"发布至 100%" → 弹出表单要求填写理由和审核人姓名 → 审核人收到 Slack/邮件通知 → 审核人批准或拒绝 → 批准后执行操作。

21c. 命名空间与团队隔离

标志按 命名空间 分组（通常按团队或按产品划分）。团队可读取所有标志，但只能在自己的命名空间内写入。跨命名空间写入需要平台管理员覆盖权限。

这解决了以下问题：搜索团队的初级工程师不会意外关闭支付团队命名空间中的标志。命名空间是信任单元。

21d. "任何人均可熔断"原则

熔断开关是角色层级的例外。任何值班工程师可在任何时间无需审批地熔断所属团队命名空间中的任何标志。原因：在故障期间，延迟的代价（每分钟损失 X 元收入）远超过度谨慎熔断的代价（事后审查后很容易恢复）。

审计日志记录熔断操作：谁、何时、针对哪个标志、原因（自由文本字段）。事后回顾读取审计日志。问责是事后的，而非预防性的。

反模式：在小公司，熔断开关被设置了审批门控。在凌晨 3 点的真实故障中，值班人员联系不到审批者，熔断被延迟，故障时间延长。这是糟糕的设计。

21e. 基于审计的访问审查

平台将访问日志导出到 SIEM（安全信息和事件管理）系统。季度访问审查暴露异常："用户 X 6 个月内未翻转任何标志——撤销其访问？"或"用户 Y 上月翻转了 47 个标志，但不属于这些标志的任何所属团队——需调查。"

为满足 SOC 2/ISO 27001 合规要求，平台必须能在数分钟内回答"展示用户 X 在过去 90 天内所做的所有变更"。§22 的审计基础设施使这成为可能。

§22. 审计日志需求

每次标志变更都是向生产环境的一次部署，每次部署都必须可审计。

22a. 日志记录内容

对于每次变更，平台写入包含以下内容的记录：

  • 操作者 ——用户 ID、认证方式（SSO、API 密钥）、源 IP。对于程序化变更，记录服务身份（Kubernetes Pod、CI 作业）。
  • 时间戳 ——挂钟时间和单调时间，均采用 UTC。
  • 标志标识符 ——完整命名空间路径： payments.checkout.checkout-v2.
  • 操作 — CREATE, UPDATE_RULE, CHANGE_VARIATION_RATIO, KILL, DELETE, RAMP_TO_X_PERCENT.
  • 变更前状态 ——变更前标志的 JSON 快照。
  • 变更后状态 ——变更后标志的 JSON 快照。
  • 差量 ——预计算的结构化差量，便于人工快速审查。
  • 原因/理由 ——自由文本，非trivial变更为必填项。
  • 关联工单 ——生产影响变更的 JIRA/Linear/Asana ID。
  • 审批链 ——对于需要多方审批的变更，附审批记录。

22b. 保留期与持久性

审计日志在变更生效前被持久写入 两个独立存储 ：

  1. S3（对象存储） ，跨区域复制，不可变保存 7 年（合规窗口）。对象锁防止篡改，即使是平台管理员也不例外。
  2. Kafka topic ，无限保留，跨区域复制。为下游 SIEM、告警和分析消费者提供数据。

仅当两次写入均成功后，变更才会应用到控制平面。如果审计日志写入失败，变更被拒绝。这是 不可妥协的 约束：无法被审计的变更不允许被执行。

保留期：7 年是 SOC 2/SOX（萨班斯-奥克斯利法案）标准。医疗（HIPAA）或金融（巴塞尔 III）可能要求更长；平台必须支持受监管工作负载按命名空间配置延长保留期。

22c. 取证查询

审计能力的核心是取证查询： "实验 X 导致昨天下午 2:14 收入下降 10%。是谁开启的？"

查询读取审计日志：

SELECT actor, ts, before_state, after_state, reason
FROM audit_log
WHERE flag_namespace = 'payments.checkout.checkout-v2'
  AND ts BETWEEN '2026-05-22 14:00' AND '2026-05-22 14:15'
ORDER BY ts DESC;

数秒内返回：操作者 = alice@linkedin.com，操作 = RAMP_TO_100_PERCENT，原因 = "已获产品经理审批；设计文档注明此日期完成 100% 发布。"调查以秒而非小时为单位锁定操作者。

没有审计日志，调查需要 git blame YAML、部署日志、Slack 历史——数小时的"考古挖掘"且结果不佳。有了审计日志，答案只需一次查询。

22d. "实验 X 导致收入下降 10%"演练

源自已发布事后分析的真实模式（已去除具体信息）：

  1. 00:00 ——工程师将 discount_widget_v3 从 5% 提升至 25%，理由为"参与度指标提升，可进入下一步"。自动审批（低于阈值）。
  2. 00:08 ——每次会话收入下降 10%。实时护栏监控器（Flink）标记异常。
  3. 00:09 ——自动暂停触发；发布停在 25%。
  4. 00:14 ——值班人员被告警唤醒。
  5. 00:15 ——值班人员查询审计日志，发现近期的发布操作。熔断标志（回滚至 0%）。
  6. 00:16 ——收入恢复正常。
  7. 事后分析 ——审计日志显示工程师的理由（"可进入下一步"）不充分：实验的主要指标是参与度（已提升），但护栏指标"每次会话收入"未在小样本量时被监控。流程变更：要求在每个发布步骤（而非仅在实验结束时）进行护栏审查。

从问题发生到解决共耗时 16 分钟。若无审计日志，此过程可能需要数小时，且事后流程改进将无从实现。

§23. 移动端特有注意事项

移动端标志求值与服务端有本质不同的约束。应用可能处于离线状态，二进制文件可能已有六个月之久，带宽十分宝贵，且 Apple/Google 控制部署周期。

23a. 离线标志求值

移动设备会有相当多时间处于断网状态（地铁、飞机、偏远地区、Wi-Fi 不可用）。SDK 必须在这些时段仍能求值标志。

设计：

  • 在本地缓存完整规则集 于 NSUserDefaults /SharedPreferences/SQLite。应用打开时，SDK 使用缓存的规则集立即求值标志，无需等待任何网络请求。
  • 缓存的 Time-To-Live (TTL) ——通常为 24–72 小时。TTL 到期后，SDK 会向分析系统发出警告，但继续返回已缓存的变体。（离线时直接报错会导致应用崩溃。）
  • 应用回到前台时刷新 ——当应用回到前台时，SDK 异步拉取最新规则集并应用于后续评估。当前会话为保持一致性，继续使用旧变体。
  • 加密缓存 ——对于包含用户分群定义（例如，允许列表中的用户 ID）的规则集，设备端缓存使用应用级密钥进行加密。防止通过静态分析应用本地存储来逆向推断正在进行的实验。

23b. 应用版本碎片化

移动端用户群跨越多个应用版本。旧版本无法理解在其发布之后引入的新 flag 类型或新规则运算符。

前向兼容契约：

  • 旧版本对未知 flag 安全降级。 如果 v5.2 应用遇到一个在 v5.2 构建时不存在的 flag，SDK 返回调用方提供的默认值（v5.2 代码本来也不知道如何处理该新 flag）。
  • 旧版本忽略未知运算符。 如果某条规则使用了 op: REGEX_MATCH 而 v5.2 SDK 只认识 EQUALS, IN_SET, GT/LT，SDK 会跳过该规则并继续处理下一条。永不抛出异常。
  • 规则集的 Schema 版本控制。 每个规则集都有一个 version 字段；旧版 SDK 会拒绝高于其支持版本的新规则集（回退到本地缓存）。防止旧版 SDK 误解新 schema。
  • 规则设计者需考虑旧客户端。 针对 v5.3+ 的新 flag 应通过将规则中的 app_version >= 5.3 作为强制条件而非"可选条件"，将 v5.2 及以下版本导向安全路径。

制度纪律：每个 flag 规则设计都要进行"这在 app v5.0 上会怎样运行？"的检查。移动平台工程师为新运算符设置一个部署过渡期——通常 v5.0 SDK 可以读取为 v5.2 生成的规则集（6 个月的前向兼容）。

23c. 应用商店审核延迟——"Stripe 暗代码发布"模式

iOS App Store 和 Google Play 的审核很慢（首次应用需要数天到数周，更新需要数小时到数天）。你无法在 5 分钟内将代码修复推送到移动端，但可以在 5 秒内推送一次 flag 变更。

这是移动端团队发布 暗代码的根本原因：每个新功能都在二进制包中构建并发布，通过一个设置为 0% 的 flag 进行控制。在构建和审核时代码就已存在，满足 Apple 的审核要求（假设该功能本身是被允许的）。审核通过后，flag 在数周内从 0% 逐步提升到 100%。如果出现 bug，flag 可以在数秒内恢复到 0%——无需再走一次应用商店发布流程。

Stripe 的移动 SDK 是一个典型案例：SDK 中的大多数功能在对商户可见之前很久就已存在于代码库中。Apple 审核了二进制包；功能在数周后通过后端 flag 切换发布给商户。

权衡取舍：二进制包体积。每个暗功能都会增加应用的安装体积。移动团队会积极控制二进制体积（例如，"应用总体积不超过 50 MB"），并针对低端市场在发布时精简功能。部分团队使用 App Thinning 加按需资源，仅在 flag 激活时下载暗功能的资源包。

23d. 配置 payload 大小至关重要

移动端带宽弥足珍贵。一个大型移动应用的典型 SDK 规则集未压缩可能达到 5–10 MB。在慢速网络上，这是一笔可感知的启动开销。

优化措施：

  • 压缩二进制格式 ——使用 protobuf 或 Cap'n Proto 代替 JSON，体积缩小 3–5 倍。
  • 增量更新 ——首次完整下载规则集后，后续更新采用增量方式。SDK 将增量应用于本地副本。稳定态带宽约为完整下载的 1%。
  • 按应用版本过滤规则集 ——SDK 上报自身的应用版本，服务端仅发送与该版本相关的规则。v5.0 SDK 不会收到 v5.3 专属规则。
  • 按需拉取 ——对于很少使用的 flag，SDK 仅在首次评估该 flag 时才拉取规则。权衡：首次调用的延迟 vs 二进制体积。
  • 延迟加载非必要 flag ——将规则集拆分为"启动关键"和"懒加载"两类。启动关键项（登录、导航、付费墙）同步下载；懒加载项在后台拉取。

23e. 移动端审核周期

移动团队将 flag 系统视为非二进制变更的 应用商店旁路。每周工作节奏可能如下：

  • 周一： 将新实验从 1% 提升到 5%。
  • 周三： 复盘实验。下线或暂停表现不佳的实验。
  • 周五： 将通过验证的实验提升到 25% 或 100%。提交新版本（如有代码变更）以备周一审核。

二进制发布节奏为每周到每月；flag 发布节奏为每天。大部分产品变更速度存在于 flag 层。

§24. 服务端渲染（SSR）与 flag 评估

现代 Web 应用将 SSR（Next.js、Remix、Nuxt）与客户端注水（hydration）混合使用。flag 评估必须在两个阶段保持一致，这比看起来要难得多。

24a. SSR flag 评估问题

在纯服务端渲染中，HTML 在服务端计算时已将用户的变体烘焙进去。客户端收到的是完整渲染好的页面。

1. Request arrives at SSR server.
2. SDK evaluates flag: variant = "treatment_A".
3. SSR renders HTML with treatment_A's variation.
4. Server sends HTML to client.
5. Client displays page.

首次渲染没有问题。但存在以下情况：

  • CDN 缓存的 SSR 页面 ——为保持低延迟，渲染好的 HTML 会在 CDN 边缘缓存，例如 60 秒。命中缓存的用户看到的是缓存填充时的变体。如果 flag 在这 60 秒内从 50% 降至 0%，该用户仍会看到实验组，尽管新鲜评估应将其分配到对照组。
  • 回滚后的陈旧变体 ——缓存成为陈旧真值的来源。关闭 flag 并不能使已缓存的页面失效。

解决方案：

  • 缓存键包含变体 ——不缓存 (url) → HTML，而是缓存 (url, variant) → HTML。每个变体都有自己的缓存版本。权衡：缓存键基数增大，缓存命中率降低。
  • flag 变更时使缓存失效 ——当 flag 发生变更时，精准清除受该 flag 影响的缓存页面。需要对缓存页面打上影响其渲染的 flag 标签。
  • 混合渲染 ——服务端渲染静态框架（不含依赖 flag 的内容）；客户端在注水后通过客户端评估填入依赖 flag 的槽位。权衡：变体填入时会出现可见的内容偏移。

Vercel 和 Next.js 对此有明确指引。其解决方案是：使用 cookies() 。 headers() 在请求层评估 flag，但将响应标记为 Cache-Control: private 或对依赖 flag 的页面使用 dynamic = "force-dynamic" 。这以牺牲 CDN 缓存为代价，换取正确性。

24b. 注水（Hydration）不一致

SSR 渲染完成后，客户端对页面进行注水——运行与服务端相同的 React/Vue 代码，期望输出一致。如果服务端评估了 getVariant() = "treatment" 而客户端 SDK 尚未加载规则集，客户端评估可能返回默认值，从而产生注水不一致警告（以及 UI 异常）。

解决方案：

  • 将变体烘焙进 HTML ——服务端嵌入一个包含已解析变体的 <script>window.__variant_state = {...}</script> blob。客户端 SDK 从该 blob 初始化，在初始注水阶段不重新评估。后续客户端导航则正常评估。
  • 基于 Cookie 的变体锁定 ——SSR 服务端在 Cookie 中设置变体；客户端 SDK 读取该 Cookie 以保证一致的评估。Cookie 在页面加载间传递变体。

核心不变量： 服务端渲染的变体必须与客户端注水时使用的变体一致。 任何偏差都会导致 UI bug。

24c. 边缘评估 flag 作为 SSR 问题的解决方案

当 SSR 缓存对高流量公开页面至关重要，同时又需要个性化时， 边缘评估 flag 模式 是现代的答案。详见 §25。

§25. 边缘评估 flag

Cloudflare Workers、Vercel Edge Config、Fastly Compute@Edge、AWS Lambda@Edge——这些都允许你在 CDN 边缘运行代码，在请求到达源站之前处理。在边缘进行 flag 评估可实现亚毫秒级的个性化，无需访问源站服务器。

25a. 架构

User request → CDN edge → Edge Function:
  1. Read cookies / headers for user identity.
  2. Look up flag ruleset (cached at edge, refreshed every 30s).
  3. Evaluate flag → variant.
  4. Either:
     a. Rewrite request to a variant-specific origin URL.
     b. Set a header / cookie that origin uses for variant-specific rendering.
     c. Fully render the variant-specific response (for static A/B content).

总延迟：边缘 1–5 ms，相比源站往返的 50–500 ms。

25b. 使用场景

后端版本间的 A/B 路由 — if variant == "treatment_A": rewrite host to backend-v2.example.com else: route to backend-v1.example.com。将变体路由与应用代码解耦。

地理限制/合规 — if country == "EU": apply GDPR-compliant variant。在任何数据离开用户所在地区之前，在边缘完成决策。

简单个性化 — if user_tier == "premium": serve premium hero image。主图变体直接从边缘提供，完全不涉及源站。

多 CDN A/B — if hash(user_id) % 2 == 0: serve from Fastly else: serve from Cloudflare。用真实用户流量对比 CDN 性能。

25c. 局限性

  • 规则集大小限制 ——边缘 Worker 内存预算较低（Cloudflare Workers：128 MB；Vercel Edge Config：上次测量为 512 KB）。大型平台的完整规则集无法放入。只有评估频率高、规则简单的 flag 才在边缘评估；包含大型允许列表的复杂规则仍在源站处理。
  • 边缘的用户身份 ——边缘 Worker 可以看到 Cookie、Header 和 IP，但看不到源站服务器可以计算的丰富用户属性（账户等级、内部分群）。边缘评估 flag 只能基于 Cookie/IP 属性。
  • 来自边缘的曝光日志 ——将曝光事件从边缘 Worker 发送到 Kafka 并不简单。部分平台在边缘缓冲曝光事件并批量发送；另一些则进行大量采样。
  • 边缘的统计显著性 ——曝光事件的保真度通常低于源站曝光。边缘评估的实验需要明确关注曝光样本的代表性。

Cloudflare 的"Workers and the Edge Config"产品、Vercel 的"Edge Config"以及 LaunchDarkly 的"Edge Functions"集成是产品化的形态。被 Vercel 营销网站、Cloudflare 自身以及运行 CDN 前置 A/B 测试的公司所使用（Booking.com 在营销页面运行边缘评估实验）。

§26. CI 中的 flag 测试

未经测试的 flag 代码是定时炸弹。从未测试过的熔断开关，在故障发生时同样无法发挥作用。

26a. 锁定 flag 值的单元测试

每个触及 flag 控制代码路径的单元测试都必须显式锁定 flag 的值。常用模式：

def test_checkout_with_v2_treatment():
    with flag_override("checkout-v2", "treatment_A"):
        result = run_checkout(user)
        assert result.discount == 0.15

def test_checkout_with_v2_control():
    with flag_override("checkout-v2", "control"):
        result = run_checkout(user)
        assert result.discount == 0.10

大多数 flag SDK 都提供带有 override / setVariant API 的测试模式。测试框架集成这些 API，使每个测试能独立控制 flag 状态。

26b. 切换 flag 的集成测试

覆盖 flag 实际评估路径的端到端测试：

1. Start app with real flag SDK pointed at a test control plane.
2. Test setup: set flag to 100% treatment via dashboard API.
3. Run user flow → assert treatment behavior.
4. Test setup: set flag to 100% control.
5. Run user flow → assert control behavior.
6. Test setup: set flag with a rule (e.g., country == "US" → treatment).
7. Run user flow with US context → assert treatment.
8. Run user flow with EU context → assert control.

这些测试能捕获：SDK 传播延迟（测试等待传播完成；若过慢则测试不稳定）、规则编译错误（编译错误的规则会被测试捕获）、曝光日志错误（测试断言曝光事件已发出）、身份处理（测试覆盖已登录/未登录/跨设备流程）。

26c. "忘记测试 OFF 路径"的灾难

经典事故：一个熔断开关在 100% 状态下保持了两年。OFF 路径的测试没有得到维护。一位新工程师在 ON 路径中添加了一个数据库查询，依赖于最近新增的一列。当该熔断开关后来在一次无关故障中被触发时，OFF 路径运行并报错，因为它不知道那个新列的存在。

真正的修法：

  • 在每次 PR 的 CI 中测试两条路径。 触及 flag 控制代码的 PR 必须同时运行 ON 和 OFF 变体的测试。覆盖率工具会暴露未经测试的变体。
  • 定期对 flag OFF 状态进行混沌测试。 夜间任务在预发布环境中随机禁用每个 flag 并运行完整测试套件。在 OFF 路径变成生产事故之前提前发现问题。
  • 针对不可达分支的静态分析。 一个 linter，对 if (flag.isEnabled('X')) 进行标记——其中 flag X 已在 100% 状态超过 90 天，建议删除该死代码分支。
  • 对 OFF 状态进行合成监控。 一个持续运行的合成用户，设置 force_variant = "control" ，在生产环境中验证 OFF 路径，一旦出现异常立即暴露。

原则：熔断开关是一份契约。只有经过测试，契约才有效。没有测试强制执行，契约会逐渐退化为"我们希望紧急情况下这能用"。

§27. OpenFeature——供应商中立的标准

OpenFeature（Linux Foundation / CNCF，云原生计算基金会）是一个正在兴起的标准，提供供应商中立的 feature flag API。其动机是：避免对 LaunchDarkly、Statsig、Optimizely 或任何单一 SaaS 供应商产生锁定依赖。

27a. SDK + Provider 模式

OpenFeature 跨语言定义了统一的 SDK API： client.getBooleanValue(flagKey, defaultValue, evaluationContext)。应用代码只使用这一 API。

在 SDK 背后，一个 provider（提供方） 插入实际的 flag 后端：

  • LaunchDarklyProvider 用于 LaunchDarkly。
  • StatsigProvider 用于 Statsig。
  • FlipDefaultProvider 用于 Flagsmith。
  • InternalProvider 用于内部自研 flag 系统。

切换 provider 只需在应用初始化时修改一行代码。应用代码不感知后端是哪一个。

27b. OpenFeature 的适用场景

  • 多云 ——不同云地域根据地区能力或合同使用不同的 flag 供应商。OpenFeature 让应用代码保持统一。
  • 供应商灵活性 ——今天采用 LaunchDarkly、两年后想迁移到 Statsig 的公司，面临的是已知的迁移成本（更换 provider），而不是未知成本（重写所有 flag 评估点）。
  • 开源友好 ——希望支持多个 flag 后端但不想全部打包的开源应用。
  • 混合技术栈环境 ——内部 flag 系统用于某类工作负载（低延迟热路径），LaunchDarkly 用于另一类（营销界面）。统一 API 贯穿两者。
  • 混合本地/SaaS ——从 SaaS 起步，随着规模增长迁移到内部系统，应用代码无需变更。

27c. 不适用的场景

对于大多数只使用一个 flag 供应商且没有迁移计划的公司，OpenFeature 只增加了一层抽象，却没有明显回报。原生 LaunchDarkly 或 Statsig SDK 功能更丰富（其定制化评估上下文、自定义分析集成、供应商专属规则运算符）。

OpenFeature 生态系统正在走向成熟。截至 2026 年，服务端 SDK（Java、Go、Node.js、Python）已趋于稳定；客户端和移动端尚不成熟。大多数大型平台正在评估，但尚未正式落地。

§28. 目标属性存储与个人身份信息（PII）

flag 规则需要用户属性：邮件域名、套餐等级、国家、设备。这些属性存储在哪里以及如何传递到评估点，涉及隐私和安全问题。

28a. 两种模式：服务端评估 vs 供应商评估

本地评估（服务端，进程内 SDK） ——应用代码已经可以访问用户属性（它知道当前用户以 alice@linkedin.com身份登录，使用 Pro 套餐，位于美国）。应用将这些属性作为 evaluationContext 传递给 SDK，由 SDK 在本地评估规则。 PII 不会离开应用。

供应商端评估（RPC 调用 flag 供应商） ——应用将用户身份和属性发送给 SaaS 供应商；供应商的服务评估规则并返回变体。供应商的日志中现在包含了用户的邮箱、套餐、位置、IP。PII 转入供应商手中。

28b. "我们将 PII 发送到了 LaunchDarkly 日志"问题

一种有据可查的模式：某团队使用带完整用户属性的客户端 JavaScript SDK 接入 LaunchDarkly。LaunchDarkly 的评估日志中包含 email = alice@linkedin.com, plan = Pro, country = US, lastLogin = ...。六个月后，一次隐私审计询问"我们是否将用户邮箱共享给任何第三方？"答案是肯定的，嵌入在 flag 评估日志中。

修复方案：

  • 使用带本地评估的服务端 SDK。 应用在不向 LaunchDarkly 发送 PII 的情况下评估规则。只有已解析的变体会被记录到 LaunchDarkly 用于诊断。
  • 使用"私有属性"功能。 LaunchDarkly、Statsig 及其他供应商允许将属性标记为"私有"——这些属性用于评估但在日志中被脱敏。
  • 发送前对标识符进行哈希处理。 发送 hash(email + salt) 代替原始邮箱。供应商可以按用户一致地定向，但无法反推出邮箱。

28c. 本地评估是符合 GDPR 的默认选择

对于任何服务欧盟用户或在严格隐私法规（GDPR、CCPA、LGPD）约束下运营的平台， 本地评估是强制要求。隐私论点是：

  • 本地评估将用户数据保留在应用自身的处理上下文中。无第三方数据传输（Schrems II / 跨境传输问题）。
  • flag 供应商只存储规则集，永远不接触用户数据。
  • 审计追踪更简单——数据路径完全在平台的控制之下。

大多数企业级 SaaS flag 供应商将本地评估作为服务端 SDK 的默认模式。客户端（浏览器、移动端）更为复杂——客户端上的规则集可能泄露实验设计，且部分属性（如 IP）无法对网络隐藏。

28d. 曝光日志中的 PII 脱敏

曝光事件包含用户标识符（user_id）及用于评估的属性。这些数据经由 Kafka 流入分析存储系统。这些存储中的 PII 会产生合规风险。

缓解措施：

  • 用户 ID 哈希化 ——曝光日志存储 hash(user_id) 而非原始用户 ID。查询通过与权威用户身份存储进行联表完成，后者有独立的访问控制。
  • 属性白名单 ——只记录规则实际引用的属性。如果规则针对 country，则只记录 country ，而非完整上下文。
  • 用户删除时清除 PII ——当用户行使"被遗忘权"（GDPR）时，该用户的曝光日志必须可被清除。分析存储系统必须在合规时间窗口内（通常为 30 天）支持定向删除。

§29. 指标定义与计算流水线

实验通过指标来计算提升效果。指标流水线是一个与功能开关系统并列的核心基础设施。对"'转化'意味着什么"的混淆，是导致实验结论错误的最常见原因。

29a. 流水线

1. SDK emits exposure event → Kafka (exposures topic).
2. Application emits outcome event → Kafka (outcomes topic), e.g.,
   {user_id, ts, event_type: "purchase", amount: 49.99}.
3. Stream processor (Flink, Spark Streaming, or Kafka Streams) joins
   exposures with outcomes on (user_id, time window).
4. Joined records land in a columnar store (Pinot, Druid, BigQuery).
5. Hourly/daily aggregation job computes per-variant metric values:
   - conversion_rate = count(purchases) / count(exposures), per variant.
   - revenue_per_user = sum(amount) / count(exposures), per variant.
6. Stat engine (frequentist t-test, Bayesian model, or both) computes
   lift, confidence interval, p-value, posterior probability.
7. Experiment dashboard renders results.

该流水线持续运行以进行实时监控（延迟1分钟），同时作为夜间批处理作业生成最终数字（包含去重、schema规范化和CUPED调整）。

29b. 指标定义即代码

"转化"指标可以表示：点击、注册、完成首次购买、7日留存、30日留存。不同定义会产生不同的提升效果。实验必须使用 经过明确定义达成共识的 指标。

模式：每个指标都在指标仓库中以SQL或DSL形式定义：

metric: checkout_conversion
definition: |
  SELECT
    COUNT(DISTINCT user_id) FILTER (WHERE event_type = 'purchase'
      AND event_ts BETWEEN exposure_ts AND exposure_ts + INTERVAL '7 days')
    /
    COUNT(DISTINCT user_id) AS conversion_rate
  FROM exposures e
  JOIN events o ON e.user_id = o.user_id
type: ratio
unit: percentage
owners: [revenue-data-science]

指标在用于实验之前需经过审查和审批。实验仪表板通过ID获取指标，而非通过临时SQL。

LinkedIn的Unified Metric Platform（UMP）、Airbnb的Minerva、Spotify的Metric Catalog以及Statsig的Metric Catalog都是这一模式的产品化实现。

29c. "什么是转化？"的歧义问题

真实的失败案例：团队A说"结账转化提升了1.2%。"团队B说"同一实验中结账转化下降了0.3%。"两者都是正确的。团队A衡量的是7日转化，团队B衡量的是会话当日转化。不同的分母、不同的时间窗口定义，得出截然相反的方向。

指标仓库可以防止这种情况：对于 checkout_conversion，由一个团队负责维护，任何人均可查询，只有一个定义。争议通过阅读定义来解决，而不是通过询问团队成员。

29d. 存储与分区

曝光-结果关联表规模巨大——对于大型平台来说有数十亿行。针对主要查询"显示过去14天实验X的变体提升效果"，按 (experiment_id, date) 进行分区，并按变体进行子分区以加快group-by速度。

Pinot和Druid使用star-tree索引进行预聚合汇总。实验仪表板的"显示实验X的变体对比"查询无论底层数据量多大都能在50–200毫秒内返回，因为汇总数据是预先计算好的。

权衡点：预聚合在segment构建时就已固定。新的指标定义或新的切片维度需要重新聚合，对于大型实验可能需要数小时。因此各平台会谨慎地对指标定义进行版本管理——在实验进行中途更改定义实际上会使之前的数据失效。

§30. 多臂老虎机与上下文老虎机——超越固定A/B测试

标准A/B测试在整个运行期间分配固定比例（50/50或33/33/33）。 老虎机（Bandits） 随着证据的积累动态地将更多流量分配给表现更好的变体，优化 探索 （学习哪个变体最优）和 利用 （将用户导向胜出变体）之间的权衡。

30a. Thompson Sampling

最简单的生产就绪老虎机算法。对每个变体，维护其预期奖励的概率分布（例如，对于二元结果使用Beta(α, β)，其中α=成功次数，β=失败次数）。

分配：对每个新用户，从每个变体的分布中各抽取一个样本；将用户分配给样本值最高的变体。

对于探索不足的变体，分布会变宽；对于已充分探索的变体，分布会变窄。早期，所有变体的采样频率大致相同。随着某个变体表现出更优越的效果，其样本值往往更高，被分配到的用户也会更多。最终，几乎全部流量都流向胜出者。

可证明的性质：Thompson Sampling实现对数级遗憾——与将所有人分配给最优方案相比，累计差距为用户数N的O(log N)。

30b. 置信上界（UCB）

对每个变体，计算 mean_reward + c × sqrt(log(N) / n_variant)。第一项是经验均值；第二项是奖励探索采样不足变体的置信奖励。

分配：将每个新用户发送到UCB值最高的变体。

UCB在给定历史记录下是确定性的，而Thompson Sampling是随机性的。两者都能实现相似的遗憾上界。Thompson Sampling在生产中更常见，因为其随机性提供了自然的流量限制，并避免了两个变体打平的退化情况。

30c. 上下文老虎机

普通老虎机假设所有用户都是相同的。 上下文老虎机（Contextual bandits） 以用户特征作为输入，并学习一个策略：给定用户上下文 x，哪个变体能最大化预期奖励 r(x, variant)?

实现：每个变体有一个模型（例如，逻辑回归、梯度提升树），基于该变体过去曝光的 (context → reward) 数据对进行训练。新用户带着上下文 x进来；分配给模型预测该用户最高奖励的变体 x （加上探索奖励）。

上下文老虎机天然地处理异质性处理效果：变体A可能对新用户最优，变体B对回访用户最优；上下文老虎机会学习这一规律并进行相应路由。

30d. 老虎机的适用场景

  • 推荐 — Netflix首页、Spotify播放列表、YouTube自动播放：数十个候选变体，流量巨大，每次展示都有可测量的奖励。老虎机实时地将更多用户分配给表现更好的变体。
  • 广告投放 — Google/Meta广告竞价部分由老虎机驱动：探索新创意，利用已知效果良好的创意。
  • 标题/缩略图测试 — Upworthy、BuzzFeed、新闻网站：众多标题变体，希望快速找到最优方案。老虎机比固定分配A/B测试收敛更快。
  • 推荐中的冷启动 — 当新内容被添加时，老虎机通过探索来了解其相对于已知内容的吸引力。

30e. 老虎机不适用的场景

  • 当需要干净的因果估计时。 老虎机给出的是"事后来看哪个变体最好"，而非"处理组相对于对照组的提升幅度"。如果实验目标是做出有据可查的商业决策，固定A/B加显著性统计分析才是正确的工具。
  • 当变体具有长尾或延迟奖励时。 老虎机在奖励信号快速时效果最好（几秒内的点击，而非数周后的留存）。当奖励延迟较长时，老虎机学习速度不够快。
  • 当公平性或法规要求平等对待时。 某些领域（定价、借贷）在法律上不允许基于观察到的早期信号将用户不成比例地分配到各变体。
  • 当变体相互破坏稳定时。 如果变体A的效果依赖于处于变体A中的用户比例（网络效应），老虎机可能会收敛到错误的胜出者。

LinkedIn、Netflix和Meta都将老虎机用于推荐和广告投放，并有明确的共识：老虎机回答"现在部署哪个最好"，而并行的A/B测试回答"用于发布/不发布决策的真实因果效果是什么"。

§31. 带具体数字的真实世界实现

需要记忆的具体数字；这些在面试中经常出现，并为所有关于规模的论断提供支撑。

  • Facebook Gatekeeper （Tang等人，公开演讲）：全集群每秒数亿次评估。进程内评估。热点开关检查被JIT编译为HHVM（Hip-Hop Virtual Machine）字节码。公开演讲引用了评估开销在热路径上主导了网络栈开销——他们通过JIT将评估时间压缩到纳秒级以下以应对最热的开关。每次页面加载、每次Graph API调用、每次News Feed排序调用都会触发Gatekeeper一次或多次。

  • LinkedIn LIX （LinkedIn eXperimentation）：任意时刻有超过 40,000个活跃LIX key ， 每天数十亿次评估。与指标定义仓库（UMP——Unified Metric Platform）和实验平台（历史上称为T-REX，现已并入LIX）集成，支持基于队列和在线实验。基于推送的SSE传播。统计引擎支持频率派+贝叶斯以及序贯检验。维护了持续数月的holdout群组系统。

  • Uber XP （eXPerimentation Platform）： 1,000+个并发实验 同时运行，采用互斥层，覆盖全球乘客/司机群体。基于Cassandra + Kafka + Apache Pinot构建。分层分配。具备自动回滚的实时护栏监控。CUPED（Controlled-experiment Using Pre-Experiment Data）方差缩减。

  • Netflix ABlaze：数百个并发实验。长达一年的holdout以衡量累积产品影响。Netflix Tech Blog公开介绍了交错分层分配和分层随机化以提高统计功效。

  • Airbnb ERF （Experimentation Reporting Framework）：2010年代中期标志性的公开参考实现，支撑数百个同时进行的实验，详细公布了其显著性统计技术栈。

  • Statsig：由前Facebook Gatekeeper团队创立。亚秒级传播，进程内评估，完整的实验套件。公开基准声称p99评估延迟 < 100微秒。被希望拥有Facebook级DNA而无需自建的中大型公司所采用。

  • LaunchDarkly：企业级SaaS参考实现。基于SSE的推送（典型延迟 < 200毫秒）。服务端和客户端SDK覆盖所有主流语言。CDN集成用于边缘配置。按MAU定价，在数百万低位封顶具有成本效益；超过该规模后，内部自建方案占主导。

  • Optimizely：传统上通过JavaScript注入进行基于Web的A/B测试。已扩展至服务端。在Web优化方面实力强劲，在后端kill-switch用例上深度不足。销售给希望使用UX A/B工具的企业团队。

  • GrowthBook：开源，统计严谨，默认贝叶斯。可自托管。被希望获得LaunchDarkly用户体验同时保留内部数据的中型公司所采用。

  • Unleash：开源，以功能开关为主（实验功能较轻量）。SDK默认轮询（可配置为SSE）。在欧盟市场因数据驻留规则倾向于自托管而颇为流行。

  • Pinterest Helium、Etsy Catapult、Spotify GoFlag、Slack Houston、Twitter Mantis flag layer：每家营收超过约10亿美元的大型互联网公司都有一个内部开关平台，形态基本一致。这一形态本身就是标准解决方案。

如果在面试中被问到，请记住以下数字： Gatekeeper约100M+次评估/秒，LIX 40k+个开关+每天数十亿次评估，Uber XP 1k+个并发实验，Netflix年度holdout，Statsig评估延迟低于100微秒，LaunchDarkly SSE传播 < 200毫秒。

§32. 总结

功能开关与实验系统本质上是一个确定性的、进程内的纯函数—— hash(user_id, salt) -> bucket -> walk compiled rule tree -> variant ——通过基于推送的配置平面复制到每台应用服务器，全球kill-switch SLA在60秒以内，同时配备一个列式曝光流水线，将确定性分配转化为因果A/B推断引擎；进程内评估不变性、确定性分桶函数和10秒以内的传播要求，是驱动所有其他设计选择的三大设计约束（编译后的规则树、边缘扇出、分层分配、粘性覆盖、列式曝光存储、自动回滚护栏、极端采样率、开关债务生命周期自动化）。

附录A：标准开关评估，一段话概括

每台应用服务器上的SDK持有一个已编译、内存映射的规则集（约30 MB，对应10万个开关）。在 getVariant(flag_key, user_context) 调用时，SDK查找开关的已编译规则树（堆指针，缓存命中热），按顺序遍历每条规则，检查内化属性条件（国家位图，semver压缩整数），在第一条匹配规则处或返回单一变体（允许列表情况），或计算 bucket = murmur3_128(flag_key:rule_id || user_id) % 10000，二分搜索分桶表得到变体ID，返回该变体名称，并异步将一条曝光事件（用户、开关、变体、规则、时间戳、属性）追加到线程本地环形缓冲区，每秒刷新到Kafka。总CPU开销：p50约400纳秒，p99约5微秒。规则集通过订阅区域边缘接入点（PoP）的SSE保持实时更新，在仪表板变更后数秒内从全局控制平面扇出差异。如果SSE连接断开，SDK回退到本地磁盘上最后一个已知良好的规则集；如果该方案也失败，则返回调用方提供的默认值。曝光事件流入一个按 (flag_key, date)分区的列式OLAP表（Pinot、Druid、ClickHouse）；实验分析UI在50–200毫秒内查询出带置信区间的变体级指标提升，无论底层曝光数据有多少TB。
