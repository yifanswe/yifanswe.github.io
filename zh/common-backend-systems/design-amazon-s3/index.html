1. 基础知识与解题思路

本题考察点

S3 是典型的星球级对象存储。难点在于在支持跨 EB 量级存储的数万亿对象的同时，提供 11 个 9 的持久性和接近 100% 的可用性，而磁盘和节点故障则持续不断发生。

其不变量如下：

每个返回成功的 PUT 请求，其数据均已跨多个故障域持久存储；每个针对已存储 key 的 GET 请求均返回最新已提交版本（或对于启用版本控制的 bucket，返回所请求的版本）；列举、key 唯一性以及 ACL 强制执行均正确；数据在任意单 AZ 故障下不丢失。

在 ICT5 / L6+ 级别，面试官在考察你能否设计具有高持久性的纠删码存储、将元数据与数据分离、处理高吞吐量的 key 列举、管理热点 key，以及推导一致性（自 2020 年更新后为强一致性，此前为最终一致性）。

60 秒面试框架

口述版本：

我将把 S3 设计为两个服务：一个将对象 key 映射到数据位置的元数据服务，以及一个使用纠删码或跨可用区副本存储对象内容的数据平面。关键决策包括：通过纠删码实现持久性、key 命名空间分区、一致性（现已为强读后写一致性）、热前缀处理，以及针对大对象的分片上传。

首先明确需求

功能性需求：

  • 通过 key 在 bucket 中执行 PUT、GET、DELETE 对象操作。
  • 按前缀列举对象；按字典序排序。
  • 针对大对象的分片上传。
  • Bucket 版本控制（每个 bucket 可选）。
  • 生命周期策略：N 天后分层或过期。
  • 通过 IAM 和 bucket 策略进行访问控制。
  • 预签名 URL。
  • 跨区域复制（可选）。

非功能性需求：

  • 11 个 9 的持久性（99.999999999%）。
  • 每个区域高可用。
  • 强读后写一致性（现代 S3）。
  • 对象大小：1 字节至 5 TB。
  • 吞吐量：每前缀每秒数千请求。
  • 每客户 PB 至 EB 规模。
  • 异地复制与容灾。

以下内容超出范围，除非被特别询问：

  • 与计算直接集成（例如 S3 Select）。
  • 查询对象内容（Athena、Glue）。
  • 对象转换（Lambda 触发器）。

规模假设

领域 假设
对象 全球范围内数万亿个；大型客户每人数十亿个。
Bucket 数百万。
对象大小 双峰分布：大量小 bucket，大量超大 bucket。
吞吐量 每前缀最高数千 QPS；每 bucket 数百万 QPS。
持久性 11 个 9 = 每 10^11 个对象中丢失约 1 个。
可用性 每区域 99.99% 以上。
Latency（延迟） GET 首字节延迟低于 100ms；小对象 PUT 延迟低于 200ms。
存储 每区域 EB 量级存储。

瓶颈在于因前缀元数据分区导致的每前缀吞吐量，以及相关故障下的持久性保障。

核心组件

组件 职责 关键细节
前端 TLS、鉴权、签名验证、请求路由。 无状态；水平扩展。

复制服务 跨区域复制。 每个对象在 PUT 后执行。

持久性与纠删码

11 个 9 意味着每 10^11 个对象中丢失 1 个。实现这一目标需要：

  • 将 chunk 跨可用区分散（独立故障域）。
  • 纠删码（例如 10+4 Reed-Solomon）：将对象分成 10 个数据 chunk + 4 个奇偶校验 chunk；可容忍任意 4 个 chunk 丢失；存储开销为 1.4 倍，而三副本为 3 倍。
  • 对于小对象，副本方式可能比纠删码开销更低。

放置策略：

  • 对于每个对象，在至少 3 个 AZ 中选取 N+M 个存储节点。
  • 确保同一对象的任意两个 chunk 不共享同一 AZ 内的机架。
  • 后台修复：当节点故障时，从存活 chunk 在健康节点上重新生成其 chunk。

面试用语：

持久性通过将 chunk 跨故障域分散以及纠删码共同实现。数学上：chunk 跨 3 个 AZ 分布并采用 Reed-Solomon 10+4，在修复完成前丢失超过 4 个 chunk 的概率极小。

写入路径

对于典型的 100 MB 以下 PUT：

  1. 客户端带 key、内容和 header 向前端发起 PUT 请求。
  2. 前端鉴权、检查 ACL，生成 request_id。
  3. 在内容流式传入时计算内容哈希（etag）。
  4. 纠删编码为 k+m 个 chunk。
  5. 放置服务跨 AZ 选取 k+m 个存储节点。
  6. 前端并行写入 chunk；等待 quorum（或全部完成）。
  7. 所有 chunk 写入并确认后，前端提交至元数据服务： (bucket, key, version) -> chunk_refs.
  8. 元数据写入返回；向客户端发送 ACK。

为实现强读后写一致性，元数据提交必须在客户端收到成功响应之前对所有读取方可见。这通常意味着元数据服务通过共识协议进行写入。

分片上传（针对大对象）

对于 100 MB 至 5 TB 的对象：

  1. 客户端发起分片上传，获取 uploadId.
  2. 并行上传各分片；每个分片独立存储。
  3. 每个分片有自己的放置位置和 etag。
  4. 所有分片上传完成后，客户端调用 complete 并传入（partNumber, etag）列表。
  5. 元数据服务将该对象记录为多个分片的组合。
  6. 客户端 GET 时检索组合信息并按顺序流式传输各分片。

优点：

  • 可恢复：失败的分片可独立重试。
  • 并行：多个分片可同时上传。
  • 内存有界：服务端无需缓冲整个对象。

读取路径

  1. 客户端向前端发起 GET 请求。
  2. 认证与 ACL 检查。
  3. 前端向元数据服务查询对象的分块布局。
  4. 前端并行从存储节点读取分块。
  5. 若使用 EC：从 k+m 个分块中的任意 k 个重建；可容忍节点缓慢或故障。
  6. 将字节流传输给客户端。

对于热门对象，前端可短暂缓存分块内容；或在前方部署 CDN。

房源

LIST 是最痛苦的操作：按键的字典序排列，分页返回。

实现方式：

  • 每个存储桶维护一个以对象键为索引的类 B-tree 索引。
  • 范围扫描从 marker 处对 max-keys 条目。
  • 按键范围分片；范围扫描在必要时跨分片进行。

性能：

  • LIST 远慢于 GET/PUT。
  • 建议客户尽量减少 LIST 操作，并对结果进行缓存。

热点存储桶（数百万个键）：将存储桶的键空间划分为多个范围，每个范围对应独立的元数据分片。

一致性模型

现代 S3（2020 年后）：

  • 新对象的强读后写一致性。
  • 覆盖写和删除的强一致性。
  • 不再出现"列表显示旧数据、GET 返回新数据"的问题。

实现方式：

  • 元数据服务具备强一致性（每个范围通过共识协议维护）。
  • LIST 查询强一致性索引。
  • 前端协调写入，仅在元数据提交后才向客户端返回 ACK。

热点前缀处理

元数据服务按键前缀进行分区。当存储桶中数百万次写入的键均以 logs/2024/01/ 开头时，会将某一分片固定下来。

缓解措施：

  • 客户指导：对高吞吐量场景使用哈希前缀。
  • 自适应分区：S3 自动拆分热点前缀。
  • 预热：针对已知高吞吐量的存储桶联系 AWS。

扩展与分区

数据 分区键 原因
对象元数据 每个存储桶的键范围 按前缀列举；范围扫描。
对象分块 存储节点哈希 分散负载。
存储桶级元数据 存储桶 ID 每存储桶配置。
ACL 账号 + 存储桶 IAM 查询。
复制队列 时间 + 存储桶 按存储桶处理。

故障模式与缓解措施

故障模式 缓解措施 监控指标
磁盘故障。 纠删码可容忍故障；后台修复。 磁盘故障率；重建积压。
可用区故障。 分块跨可用区分布；服务从剩余节点继续提供。 可用区可用性；仲裁大小。
热点前缀。 自适应分区；客户指导。 每前缀 QPS。
元数据分片热点。 子分片；缓存热门读取。 每分片 QPS。
存储节点缓慢。 从 EC 仲裁读取，跳过慢节点。 每节点尾延迟。
相关性故障（固件漏洞）。 硬件多样化；分阶段滚动升级。 跨可用区故障相关性。
对象版本冲突。 版本控制生成唯一的 version_id。 版本冲突计数。
分块上传未完成。 垃圾回收过期的分片。 过期上传时长。
跨区域复制延迟。 异步执行并进行监控；对延迟告警。 复制延迟分布。
ACL 漏洞导致数据泄露。 纵深防御检查；审计日志。 未授权访问尝试。

运维指标

持久性：

  • 修复积压；磁盘故障后的修复时长。
  • 丢失对象事件（必须接近零）。
  • 跨可用区分块分布。

容量：

  • 按地区的 PUT/GET QPS。
  • 按操作的延迟 p50/p99。
  • 存储增长。
  • 每前缀 QPS。

业务层面：

  • 每客户存储量。
  • 每客户请求量。
  • 出口带宽。

多区域策略

  • 每个地区独立运行。
  • 跨地区复制按存储桶选择性开启；PUT 操作异步复制到对等地区。
  • 复制可以是单向或双向的。
  • 存储桶以地区为范围；每个地区每个存储桶独立维护键命名空间。

需要指出的权衡

  • 纠删码相比复制具有更高的持久性且存储开销更低，但会增加 CPU 消耗和读取复杂度。
  • 强一致性会增加元数据写入延迟，但可消除一类用户错误。
  • 按前缀分区使列举操作更快，但会产生热点前缀瓶颈。
  • 版本控制以存储成本换取历史保留。
  • 跨地区复制代价高昂；通常按需选择性开启。
  • 小对象的元数据开销占比偏高；聚合小对象有助于改善。

常见陷阱与更优做法

陷阱 更优做法
全量三副本复制。 大对象使用纠删码；小对象使用复制。
单一元数据服务。 按键范围分片；每分片通过共识协议维护。
最终一致性。 通过元数据共识实现强读后写一致性。
像对待 GET 一样对待 LIST。 LIST 代价高昂；鼓励稀少使用。
忽视相关性故障。 硬件多样化；分阶段固件滚动升级。
单可用区分块。 至少跨 3 个可用区分散。

面试官可能深入探讨的方向

深入方向 优秀回答要点
持久性数学 纠删码比率；故障概率；修复时间。
分块上传 发起、上传分片、完成；并行性；可恢复性。
强一致性 元数据共识；提交后再 ACK。
热点前缀 自适应分区；客户端哈希键。
纠删码 vs 复制 按对象大小权衡。
跨地区复制 异步 PUT；冲突处理。

Staff 级别回答检查清单

  • 两个服务：元数据服务 + 数据平面。
  • 跨可用区纠删码保障持久性。
  • 通过共识协议实现强一致性元数据。
  • 针对大对象的分片上传。
  • 自适应分区处理热点前缀。
  • 生命周期与存储分层。
  • 异步跨地区复制。
  • 后台修复应对磁盘故障。

面试用语

S3 由元数据服务和数据平面组成。数据平面通过跨可用区纠删码和持续后台修复实现 11 个九的持久性。元数据服务在每个键范围内强一致。最难的扩展问题是热点前缀和相关性故障下的持久性；解决方案是自适应分区和多样化故障域。

2. 模拟面试

面试官： 设计 S3。

候选人： 我将设计两个服务：元数据服务（将对象键映射到数据布局）和数据平面（使用跨可用区纠删码存储对象字节）。核心不变量是持久性——每次成功的 PUT 都能在任意单可用区故障中存活，目标是 11 个九。自 2020 年更新以来支持读后写强一致性。我将逐一介绍 PUT、GET、分块上传和持久性计算。

面试官： 带我了解一次 PUT 操作。

候选人： 客户端发送携带键和字节的 PUT 请求。前端认证、检查 ACL、流式接收数据并计算 etag。字节被纠删编码为 k+m 个分块——例如 10 个数据分块加 4 个奇偶校验分块。布局服务选取 14 个存储节点，分散在至少 3 个可用区中。前端并行写入分块；全部确认后，向元数据服务提交： (bucket, key, version_id) -> chunk_refs。元数据提交在每个键范围内通过共识协议完成。提交成功后，向客户端返回 ACK。

面试官： 为何选择纠删码而非三副本复制？

候选人： 存储开销。三副本复制是 3× 开销。Reed-Solomon 10+4 是 1.4× 开销，持久性相当。对于大对象，在 PB 级别差异巨大。对于小对象，EC 的元数据开销可能超过节省；有时复制更优。现代 S3 采用混合方案。

面试官： 11 个九——如何实现？

候选人： 将分块跨 3 个可用区分散。使用 Reed-Solomon 10+4，在数据丢失前可容忍 4 个分块损失。可用区是独立的故障域。在持续后台修复的情况下，修复完成前丢失 5 个分块的概率极低。具体计算：假设磁盘 MTBF 为 100 万小时，修复时间 1 小时，14 个分块跨 3 个可用区——一年内的综合概率远低于 10^-11。硬件和固件多样化可降低相关性故障风险。

面试官： 带我过一遍 GET 流程。

候选人： 客户端向前端发起 GET。认证 + ACL 校验。元数据查询返回 chunk_refs。前端并行从存储节点读取分块。由于使用了 EC，任意 k 个分块（14 个中的 10 个）即可重建。我们向全部 14 个节点发起读取，取最先完成的 10 个，这自然地缓解了尾延迟。将字节流传输给客户端。

面试官： 1 TB 的对象——如何处理？

候选人： 分块上传。客户端发起请求并获取 uploadId。并行上传分片（例如，每片 100 MB，共 10,000 片）。每个分片独立存储——拥有自己的纠删编码分块。客户端重试失败的分片。所有分片完成后，客户端发送包含 (partNumber, etag) 列表的完成请求。元数据服务将对象记录为多分片的组合。GET 通过顺序或并行读取各分片来重建。

面试官： 强一致性——如何实现？

候选人： 元数据服务在每个键范围内通过共识协议（键范围副本集内的 Paxos 或 Raft）实现强一致性。成功的 PUT 提交后，任何后续 GET 都能看到新版本。LIST 查询同一个强一致性索引。2020 年以前，S3 对新键删除后采用最终一致性；现代 S3 已修复此问题。

面试官： 某客户每秒写入 100,000 个对象，所有键均以 logs/2024/01/开头。会发生什么？

候选人： 所有这些写入都打到拥有该前缀范围的元数据分片。热点出现。缓解措施：自适应分区——S3 自动检测热点前缀并将范围拆分到更多分片。客户指导：使用哈希前缀分散负载。对于已知的高吞吐量存储桶，提前预分区。我们监控每前缀 QPS 并告警。

面试官： 磁盘发生故障。会发生什么？

候选人： 该磁盘上的分块丢失。后台修复检测到并从每个受影响对象的剩余 k 个分块中重建，写入健康节点。修复持续进行；积压受到监控。在修复完成之前，GET 仍然有效，因为我们还有剩余的 k+m-1 个分块（仍高于 k 的阈值）。

面试官： 版本控制？

候选人： 按存储桶开关控制。启用后，对已有键的每次 PUT 都会创建新的 version_id，而非覆盖。元数据服务按 (bucket, key, version_id)建立索引。DELETE 创建删除标记；对象可以被恢复。生命周期策略可以使旧版本过期。

面试官： 生命周期？

候选人： 按存储桶策略：例如，"30 天后迁移至低频访问，1 年后迁移至 Glacier，7 年后删除。"生命周期服务每日运行，扫描存储桶索引，执行迁移。迁移至 Glacier 可能涉及针对不同存储类别以不同的持久性/延迟权衡重新编码。

面试官： 跨地区复制？

候选人： 按存储桶配置：源地区 → 目标地区。每次 PUT 在元数据提交后，复制服务将一个副本加入队列。对等地区通过内部 API 接收对象。异步执行；延迟通常为秒级到分钟级。监控复制延迟。

面试官： 列举拥有数十亿对象的存储桶？

候选人： LIST 是最难的操作。元数据服务为每个存储桶维护一个有序的类 B-tree 索引。从标记位置进行最多 max-keys 条目的范围扫描。若键跨越多个分片，前端负责拼接结果。我们强烈建议客户使用特定前缀来限制扫描范围，并进行分页。

面试官： 总结。

候选人： S3 = 元数据 + 数据平面。跨可用区纠删码实现 11 个九。元数据通过共识协议在每个键范围内强一致。分块上传处理大对象。自适应分区处理热点前缀。后台修复应对持续的磁盘故障。生命周期和复制是异步流水线。强一致性于 2020 年通过在 ACK 前同步提交元数据引入。

3. 练习备注

五分钟版本

  1. 明确：持久性、吞吐量、分块上传、一致性。
  2. 不变量：11 个九持久性；强读后写。
  3. 两个服务：元数据服务 + 数据平面。
  4. 跨 3 个可用区的纠删码；例如 RS 10+4。
  5. 针对大对象的分片上传。
  6. 通过元数据共识实现强一致性。
  7. 自适应分区处理热点前缀。
  8. 后台修复；生命周期；跨地区复制。

一分钟结尾陈述

S3 由元数据服务和数据平面组成。数据平面通过跨可用区纠删码和持续后台修复实现 11 个九。元数据服务在每个键范围内强一致；PUT 仅在元数据提交后才返回 ACK。分块上传支持最大 5 TB 的对象。自适应分区处理热点前缀；客户端使用哈希键的指导有所帮助。生命周期和跨地区复制是异步流水线。持久性在相关性故障下是最难的正确性问题；解决方案是多样化故障域和持续修复。

最佳面试用语

  • "跨可用区纠删码以低于复制的存储开销实现持久性。"
  • "强一致性要求在 PUT ACK 前完成元数据提交。"
  • "分块上传是大对象的必要方式；并行且可恢复。"
  • "热点前缀通过自适应分区和客户端键哈希来缓解。"
  • "持续后台修复使磁盘的持续故障对用户不可见。"
