1. 基础知识与解题思路

本题考察点

Twitch 聊天是一个高扇出的直播评论系统：数千到数百万观众连接到同一个直播间，每个人既能发送消息，也能以亚秒级延迟看到其他所有人的消息。难点在于不对称扇出——一条消息必须到达数百万观众——同时还需处理审核、慢速模式、表情渲染和临时保留。

其不变量如下：

每位观众发送的聊天消息，在亚秒级延迟内到达同一频道的所有其他在线观众，在接收方之间保持一致的顺序；审核操作（禁言、封禁、删除）在数秒内生效；系统能承受观众数量的大幅涌入以及特定频道的审核规则。

在 ICT5 / L6+ 级别，面试官在考察你能否设计基于 WebSocket 的 pub/sub 扇出，将每频道会话扩展到数百万观众，处理审核工作流，并就临时存储和限速进行推理。

60 秒面试框架

口述版本：

我将把 Twitch 聊天设计为每频道的 pub/sub：观众发布的每条消息通过 WebSocket 扇出到所有在线观众。频道会话运行在一批聊天服务器上；观众通过 WebSocket 连接到处理其所在频道的服务器。审核、慢速模式和表情替换在服务端应用。关键决策在于百万观众频道的扇出架构、审核以及限速。

功能需求

功能性需求：

  • 观众加入频道；发送消息。
  • 消息实时广播给所有观众。
  • 审核：禁言、封禁、消息删除。
  • 表情（按名称替换为图片）。
  • 慢速模式、仅订阅者、仅关注者。
  • （重新）加入时可见聊天历史（最近 N 条消息）。
  • 回复、@ 提及。

非功能性需求：

  • 亚秒级消息投递。
  • 热门频道每频道数百万观众。
  • 承受特定频道的流量峰值（热门主播上线）。
  • 审核操作在数秒内生效。
  • 多地区。

以下内容超出范围，除非被特别询问：

  • 语音/视频聊天。
  • 私信（Whispers）。
  • 频道积分兑换详情。

规模假设

领域 假设
频道 数百万个；数千个活跃。
每频道观众数 1 到 100 万；长尾分布。
消息/秒 热门频道每秒数千条。
Latency（延迟） < 500ms 广播。
持久性 临时保留；最近几百条消息。
并发连接 数千万。

瓶颈：百万观众频道的每频道扇出。

核心产品模型

审核服务 服务端强制执行。 快速。
表情服务 替换表情 token。 已缓存。
限流 每观众 + 每频道。 内联处理。
持久性 每频道最近 N 条消息。 内存 + 短期存储。

核心组件

组件 职责 关键细节
WebSocket 网关 持有观众连接；接收发送请求；投递接收消息。 有状态；数量众多。
频道协调器 每频道状态：速率、慢速模式、审核操作。 每频道一个。
Pub/sub 代理 在频道内广播。 Redis pub/sub 或自定义实现。
审核服务 应用审核规则。 实时。
表情服务 Token → 图片；在客户端渲染。 已缓存。

自动审核 / AI 检测骚扰、垃圾信息。 异步 + 内联。

发送路径

  1. 观众通过 WebSocket 发送消息。
  2. 网关验证：观众已加入频道；未被封禁；未被禁言。
  3. 限流检查：每位观众每个频道的令牌桶。
  4. 内容审核：关键词过滤；垃圾信息检测；表情替换。
  5. 将消息发布到频道的 pub/sub 主题。
  6. 持久化到最近消息缓冲区（有界）。
  7. 所有订阅该频道的网关将消息转发给其本地观众的 WebSocket。

接收路径

  1. 每个网关订阅其观众所在的频道。
  2. pub/sub 代理推送新消息。
  3. 网关将消息发送给该频道的所有本地观众 WebSocket。
  4. 客户端进行表情替换后渲染。

百万观众频道的扇出

一个拥有 100 万观众、每秒 1000 条消息的频道，每秒需要 10 亿次消息投递。只要分布式处理，这是可控的。

策略：

  • 每个频道配置多个网关。
  • 每个网关持有一部分观众。
  • pub/sub 广播至所有网关，每个网关再转发给本地观众。
  • 网关扇出：100 万 / N 个网关 ≈ 每网关 10k。单节点可控。

面试用语：

每个网关只向其本地观众转发。pub/sub 广播至所有网关；扇出成本分摊到整个网关集群。百万观众频道只需 100 个网关，每个网关服务 10k 观众。

审核服务

  • 封禁： 永久屏蔽。用户无法发送消息。在发送检查时强制执行。
  • 禁言： 临时限制；N 秒后过期。
  • 慢速模式： 用户每 N 秒只能发送一条消息。
  • 仅限订阅者： 只有付费观众可以发送。
  • 仅限粉丝： 只有粉丝（具有最低关注时长）可以发送。
  • 删除： 移除特定消息；广播删除事件。
  • 自动审核： AI 标记可疑内容；审核员审批后才展示。

版主（志愿者）可以执行操作。操作通过 pub/sub 传播到该频道的所有网关。

限流

每位观众每个频道：- 普通用户：约每秒 1 条消息。- 订阅者：可能放宽限制。- 垃圾信息检测：相同或相似的消息将被标记。

每个频道：- 慢速模式统一应用。

在网关以内存形式维护每个 (channel, user) 的令牌桶，并定期同步。

表情

频道和全局表情。类似 :LUL: 的文本 token 替换为图片。

  • 表情字典按频道缓存。
  • 在客户端渲染时或在网关处进行替换。
  • 每个频道的自定义订阅者表情。

最近消息缓冲区

频道在内存缓冲区中保留最近 N 条消息（例如 100-500 条）。

  • 观众加入时回放缓冲区，使其能看到上下文。
  • 有界；旧消息会被丢弃。
  • 可选：使用带有 TTL（小时级）的临时存储。

频道协调器

每个频道的状态（模式、封禁名单、版主、慢速模式计时器）由协调器服务维护。

  • 网关读取协调器状态。
  • 版主操作更新协调器并广播。

扩展与分区

数据 分区策略
连接数 分布在各网关
频道 pub/sub 主题 每个频道一个主题
频道状态 按 channel_id 分片
最近消息 每个频道独立的内存存储
版主操作 每个频道

对于超热门频道：对 pub/sub 主题本身进行分片；部署多个协调器；观众连接到任意可用实例。

一致性模型

层级 一致性
消息投递 每个频道至少一次 + 有序投递
版主操作 强最终一致性
频道状态 强一致性（变更时使用 CAS）
最近缓冲区 最终一致
跨区域 有界延迟

多区域策略

  • 频道有归属区域（即主播所在的区域）。
  • 其他区域的观众连接到本地网关；网关订阅该频道的 pub/sub，可能跨区域。
  • 跨区域延迟增加约 50-100ms。
  • 对于大型全球直播，采用区域扇出：每个区域的 pub/sub 镜像该频道；消息跨区域复制。

故障模式与缓解措施

故障模式 缓解措施 监控指标
网关崩溃。 观众重连；重新订阅。 重连率。
pub/sub 代理故障。 故障转移副本；重新订阅。 代理健康状况。
百万观众突涌。 自动扩展网关；预热。 每个频道的在线人数。
垃圾信息攻击。 自动审核 + 限流 + 封禁。 垃圾信息发送率。
版主操作延迟。 pub/sub 传播；告警。 从版主操作到生效的延迟。
频道热门 pub/sub 主题。 主题分片；多区域。 主题吞吐量。
跨区域消息丢失。 至少一次；重试。 丢包率。
消息乱序。 频道级单调序号。 乱序事件数。
限流绕过。 在网关聚合 + 中央合理性检查。 绕过次数。
表情字典过期。 定期刷新；变更时失效。 表情过期率。

运维指标

正确性：

  • 消息丢失。
  • 乱序投递。
  • 版主操作传播延迟。

容量：

  • 每网关连接数。
  • 每个频道每秒消息数。
  • pub/sub 扇出延迟。

业务层面：

  • 每个频道的并发观众数。
  • 聊天参与率。
  • 每个频道的版主操作次数。

需要指出的权衡

  • 频道级有序投递增加了复杂性，但符合用户体验预期。
  • 持久化深度与成本的权衡：缓冲区越大，回放效果越好，但内存消耗也越大。
  • 激进的自动审核有助于管理，但可能误伤合规用户。
  • 多区域增加延迟，但提升了韧性。
  • 每个频道一个 pub/sub 主题结构简单；对热门频道进行分片会增加复杂性。

常见陷阱与更优做法

陷阱 更优做法
每个频道单一代理。 热门频道使用分片代理。
在发送路径中进行同步审核。 内联快速过滤 + 异步深度自动审核。
持久化所有聊天记录。 有界最近缓冲区；临时存储。
单一网关。 分布式；通过 pub/sub 互联。
无限流。 每个 (channel, user) 的令牌桶。
信任客户端表情渲染。 服务端表情替换。

面试官可能深入探讨的方向

深入方向 优秀回答要点
百万观众扇出 网关分布；pub/sub 广播。
审核服务 内联 + 异步；传播。
慢速模式 每个频道的令牌桶。
多区域 每个区域的 pub/sub；复制。
订单 频道级序号；每个接收方保证顺序。
自动审核 ML 分类器；内联快速路径 + 异步深度处理。

Staff 级别回答检查清单

  • WebSocket 网关 + 每个频道的 pub/sub。
  • 分布式扇出（每个网关仅转发给本地观众）。
  • 内联审核 + 异步自动审核。
  • 每位观众 + 每个频道的限流。
  • 有界最近消息缓冲区。
  • 多区域，频道有归属地。
  • 突涌时自动扩展。

面试用语

Twitch 聊天是基于每个频道的 pub/sub，观众通过 WebSocket 连接到分布式网关集群。pub/sub 将每条消息广播至所有订阅网关；每个网关转发给其本地观众。百万观众频道通过分布式扇出处理。审核采用内联（快速过滤 + 封禁检查）加异步深度自动审核的方式。最近消息缓冲区有界。

2. 模拟面试

面试官： 设计 Twitch 聊天系统。

候选人： 基于每个频道的 pub/sub 系统。观众通过 WebSocket 连接到聊天网关。每个频道有一个 pub/sub 主题；所有拥有该频道观众的网关均订阅该主题。当有人发送消息时，网关验证后将其发布到主题；pub/sub 广播至所有订阅网关，每个网关转发给其本地观众。百万观众频道的扇出成本分摊到整个网关集群。审核在内联（快速过滤、封禁检查）加异步深度自动审核阶段完成。

面试官： 带我走一遍消息发送的流程。

候选人： 观众输入消息；网关通过 WebSocket 接收。网关检查：观众是否在频道中、是否被封禁、是否被禁言、是否在限流内（每用户每频道的令牌桶）、是否满足频道模式（仅限订阅者、慢速模式）。内联内容过滤（违禁词、垃圾信息）。以频道级单调序号发布到频道 pub/sub 主题。添加到最近消息缓冲区（有界，约 500 条）。pub/sub 广播至所有订阅该频道的网关。每个网关将消息转发给该频道的本地观众 WebSocket。端到端延迟低于 1 秒。

面试官： 百万观众的频道。

候选人： 每个网关只向其本地观众转发。若有 100 个网关，每个网关服务 10k 观众，则每个网关每秒发送 1000 条消息 × 10k 观众 = 每秒 1000 万次操作——数量较大但单网关可承载。pub/sub 广播至 100 个网关：每条消息 100 次投递。总扇出（每条消息 100 万次投递）均摊到整个网关集群。通过增加网关来扩展。

面试官： 跨观众的消息顺序如何保证。

候选人： 频道级单调序号由频道协调器（或持有协调器分配范围的发布网关）分配。所有网关按序号顺序转发。接收方看到一致的顺序。在多区域场景中，每个区域的本地顺序与全局一致；跨区域观众可能略有延迟，但与同区域接收方的顺序一致。

面试官： 版主封禁。

候选人： 版主点击封禁按钮。REST 请求发送到聊天服务。频道协调器更新封禁名单。封禁事件通过 pub/sub 广播至所有网关。网关更新本地状态；被封禁用户的后续发送请求在网关层被拒绝。已发布的消息可能仍然可见；可选择广播删除事件以移除其最近消息。端到端传播时间为几秒。

面试官： 慢速模式。

候选人： 频道级设置：每位用户每 N 秒最多发送一条消息。网关处的令牌桶：每位用户有一个 N 秒补充一次的桶。桶为空时拒绝发送。订阅者可豁免。

面试官： 自动审核。

候选人： 两层架构。内联快速过滤：违禁词、正则表达式、垃圾信息模式。成本低；对每次发送均执行。异步深度分类器：ML 模型对消息打分；若不安全，事后广播删除事件，并可能触发禁言。采用异步是因为深度模型开销较大；我们接受临界消息被短暂展示的风险。

面试官： 最近消息缓冲区。

候选人： 每个频道保留最近约 500 条消息的环形缓冲区，存储在内存中（由协调器维护或进行复制）。新观众加入时回放缓冲区，使其能看到上下文。有界；旧消息被丢弃。临时存储；不长期持久化，但可能保留数小时用于审核审计。

面试官： 连接中途断开。

候选人： 观众通过 WebSocket 重新连接。网关重新将该连接订阅到频道。网关发送最近 N 条消息（回放缓冲区），使观众快速追上进度。由于聊天主要是临时性的，不会有状态丢失。

面试官： 多地区。

候选人： 频道归属区域。pub/sub 主题位于归属区域。全球观众连接到本地网关；网关订阅归属区域的主题（或区域镜像）。对于大型全球直播，每个区域运行自己的 pub/sub 来镜像归属区域的流——观众本地接入，跨区域复制增加约 50-100ms 的延迟。

面试官： 垃圾信息攻击：1 万个机器人账号狂轰频道。

候选人： 每用户限流可拦截大多数请求。模式检测：来自大量新账号的相似消息 → 自动审核介入；封禁操作传播。频道模式可切换为仅限粉丝或仅限订阅者以防御攻击。pub/sub 能吸收突涌流量，但审核可降低信噪比。

面试官： 总结。

候选人： 基于每个频道的 pub/sub 与 WebSocket 网关。百万观众频道的分布式扇出。内联审核（限流、违禁词、封禁检查）+ 异步深度自动审核。有界最近消息缓冲区。慢速模式和频道模式在网关层强制执行。多区域，频道有归属地 + 区域镜像，适用于全球直播。版主操作通过 pub/sub 在数秒内传播。

3. 练习备注

五分钟版本

  1. 澄清：规模、延迟、审核。
  2. 不变量：亚秒级广播；频道内有序；版主操作快速生效。
  3. WebSocket 网关 + 每个频道的 pub/sub。
  4. 分布式扇出：网关 → 本地观众。
  5. 内联审核 + 异步自动审核。
  6. 每用户每频道限流。
  7. 有界最近消息缓冲区。
  8. 多区域，频道有归属地 + 镜像。

一分钟结尾陈述

Twitch 聊天是基于每个频道的 pub/sub 与 WebSocket 网关的组合。每个网关只向其本地观众转发；扇出分摊到整个网关集群，因此百万观众频道只需一百个各服务 10k 观众的网关。审核采用内联（限流、封禁检查、违禁词）加异步深度自动审核。有界最近缓冲区供新加入者使用。频道级序号保证顺序。多区域，频道有归属地。

最佳面试用语

  • "pub/sub 广播至所有网关；每个网关转发给本地观众。"
  • "百万观众的扇出分摊到整个网关集群。"
  • "审核采用内联快速过滤 + 异步深度自动审核。"
  • "频道级单调序号保证跨观众的消息顺序。"
  • "最近消息缓冲区是有界的；聊天记录在设计上是临时性的。"
