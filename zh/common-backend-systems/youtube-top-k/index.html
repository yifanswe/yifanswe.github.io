YouTube Top-K Videos (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Hard · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建一个服务，返回某时间窗口（最近 1 小时、1 天、30 天）内 YouTube 播放量最高的 Top-K 视频。活跃视频 1 亿个，峰值每小时 10 亿次观看事件，K 最大 1000，响应 p99 < 200 ms。

1. 需求（约 5 分钟）

功能需求 - 查询给定窗口内的 Top-K 视频： 1h, 1d, 7d, 30d。以及"全部时间"。 - 可选按分类或地区筛选。

我会暂不考虑每用户个性化推荐以及超出原始计数的热门算法调整。

非功能需求 - 查询 p99 < 200 ms。 - 摄入吞吐量维持每小时 10 亿事件，峰值约 28 万事件/秒。 - 最终一致性——Top-K 在约 1 分钟内反映事件（分钟级粒度即可）。 - 近似 Top-K 可接受；若精确代价过高，可使用 sketch 算法。

容量估算 - 1 亿个活跃视频。若全部物化，4 个窗口的精确计数器需约 4 亿条热数据。 - 观看事件约 100 字节；每小时 10 亿 × 24 = 每天 2.4 TB 原始数据（复制前），但压缩后的分钟级计数远小于此。 - 若选择合适粒度，精确桶状态可以放入内存： 1h 60 个分钟桶 × 1 亿 × 4 B ≈ 24 GB； 1d 24 个小时桶 ≈ 9.6 GB； 30d 30 个天桶 ≈ 12 GB。副本及分类/地区维度会成倍放大，因此热数据的主要成本是流处理器中的状态大小，而非 API 读取。

2. 核心实体（约 2 分钟）

  • ViewEvent — video_id, user_id, ts, region, category.
  • WindowCounter — (video_id, window) → count.
  • TopKList — (window, [video_id]_K)，即预计算排名。

3. API（约 5 分钟）

GET /v1/topk?window=1h&k=100&category=music&region=us   → { videos[] }

窗口已枚举（1h, 1d, 7d, 30d, all）；不支持任意时间窗口。

响应游标仅用于在预计算列表内分页；它编码了 (window, facet, version, offset) ，确保每一页都来自同一个已发布的快照。

4. 数据流（约 5 分钟）

观看事件 → Kafka → 流处理器 → 更新各窗口计数器（滑动）→ 维护每个窗口的 Top-K 堆 → 将 Top-K 列表发布到读存储 → 从 Redis 读取提供服务。

5. 高层设计（约 10–15 分钟）

  1. 事件采集。 YouTube 播放器向采集层发送观看事件。事件进入 Kafka topic views.raw，按 video_id 分区（使同一视频的所有事件落在同一分区，便于聚合）。
  2. 流聚合器。 Flink 作业消费 views.raw，在多个窗口中维护计数器。两种模式： - 滚动 1 分钟计数器 （始终物化）。 - 1h/1d/7d/30d 滑动窗口 通过累加相应数量的滚动桶计算得出。
  3. 单视频状态。 Flink 中的 keyed state： video_id → ring buffer of last N minute-buckets。内存成本 = 1 亿个视频 × 固定小常量。
  4. Top-K 维护。 在每个任务槽（拥有一段 video_id哈希范围）内，为每个窗口维护一个局部 Top-K 的最小堆。每到分钟边界，向合并阶段发射局部 Top-K。
  5. 合并阶段。 协调者从每个任务接收局部 Top-K 并计算全局 Top-K。在 N 个分区、K=1000 的情况下，每个分区的局部 Top-K 可适当增大（如 5K），以保证当刚好低于局部 K 的项可能合并进全局 Top-K 时的正确性。
  6. 读存储。 全局 Top-K 写入 Redis（每个 (window, region, category) 组合一个列表）。读服务直接获取预计算列表。
  7. 冷存储回填。 原始事件同时归档到 S3 / 数据仓库，用于即席分析。

端到端叙述：视频被观看 → 事件落入该 video_id 对应的 Kafka 分区 → Flink 任务递增当前分钟桶 → 到达分钟边界时滑动缓冲区（丢弃最旧，添加最新）→ 重新计算每个窗口的滑动计数 → 若更新后的计数进入局部 Top-K，则更新局部堆 → 每分钟合并局部堆 → 全局 Top-K 写入 Redis → API 直接提供服务。

6. 深度探讨（约 10 分钟）

具体存储契约。

存储 键/数据结构 条目大小 + TTL 分片键
原始事件流 Kafka views.raw 事件 {video_id, ts, region, category} 约 100 B，保留 24–72 小时 video_id 或加盐热 key
桶状态 Flink/RocksDB video_id → ring buffers 每视频每 facet 集约 50–200 B video_id
局部堆 每个任务 topk:{window}:{facet} 最小堆 K' × ~16 B，每分钟刷新 任务分区
全局列表 Redis topk:{window}:{facet}:{version} 视频 ID / 计数列表 K × ~24 B，TTL 按窗口/版本设置 window+facet
全时计数器 KV/OLAP views_total:{video_id} 每视频 8–16 B，长期保留 video_id
原始归档 按小时分区的对象存储 / 数据仓库 每天约 2.4 TB 原始数据，冷保留 date/hour

各存储容量：Kafka 热保留 2.4 TB/天 × 3 副本 × 3 天约 21.6 TB（压缩前）。核心窗口的精确活跃状态为数十 GB，但随 facet 快速倍增，因此需限制支持的 facet 数量。Redis 服务列表极小：400 个 facet/窗口列表 × 1000 行 × 24 B 不足 10 MB（含开销前）；读路径不是成本中心。

摄入与查询路径追踪。 播放器信标 → 采集层验证并写入 Kafka → Flink 任务按 video_id 键控，更新当前桶与总计数 → 到达分钟边界时推进环形缓冲区和局部堆 → 合并阶段将局部 K' 列表合并为全局 K → 读存储写入新的 Redis 版本 → API 读取 topk:{window}:{facet}:{version} 并并行补全视频元数据。查询路径刻意设计为单次 Redis 读取加元数据填充；用户请求不触发任何扫描。

热路径与冷路径：

路径 涉及的存储 并行度 游标 / 版本
热 API 读取 Redis Top-K 列表 + 元数据缓存 元数据填充作为一批执行 {window, facet, version, offset}
Redis 未命中 内存中最新快照，然后是前一个 Redis 版本 元数据填充仍然批量执行 若可用则使用同一版本
回填查询 OLAP / 对象归档 离线扫描，不面向用户 作业 ID，非 API 游标

游标固定某个已发布的 version；否则第 2 页可能来自更新的分钟并导致重复或跳过视频。

精确 vs 近似。 在 1 亿 key / 28 万 QPS 下的精确 Top-K 计算量大但可行，需合理分片。近似替代方案： - Count-min sketch 用于有界误差计数器；节省内存但损失精确性。 - Space-saving 算法 用于流式 Top-K：维护 k×c 个桶，当新项占主导时替换最小项。误差有理论上界。 - 默认选精确（Flink 中的单视频状态），该问题规模可行；若内存溢出则回退到 space-saving。

为什么选 Flink/RocksDB 而非 Spark 微批：我们需要分钟级时效、keyed state、水位线和低持续延迟的故障恢复。Spark Structured Streaming 也可行，但 Flink 对长寿命 keyed state 和事件时间窗口更为原生契合。

为什么用 Redis 提供服务而非直接查询 Flink 状态：Redis 提供稳定的、版本化的快照，在 p99 200 ms 内可预期。Flink 状态针对流式更新和 checkpoint 优化，而非数千个 API 读取。

Top-K 聚合正确性。 朴素的"全局 Top-K = 局部 Top-K 的并集"是错误的：某个项在每个分区均刚好低于 K，但全局可能超过 K。解决方案：发送局部 Top-K'，其中 K' > K（如 5K），使全局合并能看到足够候选。或通过第二遍扫描纳入全局计数。

Flink task 0 -> local top 5K --\
Flink task 1 -> local top 5K ----> merge coordinator -> global top K + version
Flink task N -> local top 5K --/                         |
                                                           v
                                                Redis published snapshot

滑动窗口内存。 30 天窗口按分钟分辨率计算：每视频 43200 个桶 × 1 亿视频 = 4.3 万亿条记录。数量过大。缓解方案：对较旧的窗口使用粗粒度桶——1 小时内分钟级、1 天内小时级、7d/30d 使用天级。以牺牲窗口尾端精度换取内存。

面试关键句："难点不在于计算 Top-K；难点在于当产品只需要当前时刻附近的分钟级时效时，拒绝以分钟粒度存储 30 天数据。"

热 key 倾斜。 病毒视频会占据不成比例的事件量。持有该视频的 Kafka 分区产生背压。缓解方案：对热 key 使用加盐分区，下游再合并。检测方式：上游追踪器监控分区消费延迟。

延迟事件。 网络不稳定导致的观看事件可能延迟 5 分钟到达。Flink 通过水位线处理——超出水位线的事件被丢弃或发送到 side output 以供对账。权衡点是时效性与完整性；对于 Top-K 而言，丢弃 0.01% 的事件是可以接受的。

多地区。 在每个地区内计算地区级 Top-K；对于全局 Top-K，运行一个消费全局 Kafka mirror 的单一全局 Flink 作业，或在协调者处合并各地区 Top-K。全局视图会增加延迟成本。

分面 Top-K（按分类 / 地区）。 每条 (window, dimension) 每个组合都是一个独立的预计算列表。基数较小（如 10 个分类 × 10 个地区 × 4 个窗口 = 400 个列表）。计算成本随维度线性增长。

这是另一种双视图模式：计数器按 video_id 键控用于更新，已发布列表按 (window, facet) 键控用于读取。它们是同一份视图数据以两种方式物化，使写入和读取均高效。

故障恢复。 Flink 使用 RocksDB 支持的状态，并在任务故障时从 Kafka 源 offset 重放。通过 Flink checkpoint + Kafka 事务性读取实现精确一次语义。

双视图模式在此处显式呈现：计数器按 video_id键控用于更新，列表按 (window, facet, version)键控用于读取。合并阶段是将一种视图转换为另一种视图的唯一之处。

情形 模式 数据源 延迟预算 回退
病毒热 key 加盐计数 Kafka 加盐分区 + 合并 <1 分钟 将 video_id 拆分为多个盐值，在堆合并前重新合并
延迟事件 水位线模式 Flink 事件时间 1–5 分钟延迟 丢弃 / side-output 极晚到达的事件；离线对账

我会推迟的内容。 每用户"我想看的 Top-K"个性化推荐，热门 = 基于增速而非原始计数，机器人观看的欺诈过滤。
