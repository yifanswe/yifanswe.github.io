1. 基础知识与解题思路

本题考察点

Twitter（X）是经典的"社交信息流"问题：短帖子、关注图谱、主页时间线，均需在全球规模下运行。难点在于不对称性——一个拥有 1 亿粉丝的明星发一条帖子，但普通用户的时间线读取需要在亚秒级完成。朴素的写时扇出或读时扇出在极端情况下都会失败。

其不变量如下：

每条推文恰好存储一次于权威存储中；每位关注者的主页时间线最终包含其关注作者的所有推文（遵循可见性、屏蔽、删除规则）；主页时间线读取的 p99 延迟在亚秒级；系统能处理明星和病毒式传播场景而不阻塞普通用户。

在 ICT5 / L6+ 级别，面试官在考察你是否能阐明混合扇出策略（普通作者推送，明星拉取）、将权威存储与时间线投影分离、正确处理删除和可见性变更，以及将排名/搜索设计为独立于信息流分发的关注点。

60 秒面试框架

口述版本：

我将围绕混合扇出策略来设计 Twitter。权威推文存储和关注图谱是事实来源。对于普通作者，我们将推文 id 推送（写时扇出）到关注者的时间线收件箱。对于拥有数百万粉丝的明星，我们在主页时间线构建时拉取（读时扇出），将明星推文与预计算收件箱合并。关键决策在于明星阈值、删除与可见性传播、排名，以及热点事件流量激增的处理。

首先明确需求

功能性需求：

  • 用户发布推文（文字、可选媒体）；回复、转推、引用、点赞。
  • 用户互相关注；取消关注；屏蔽；静音。
  • 主页时间线显示来自已关注用户的推文。
  • 用户时间线显示用户自己的帖子。
  • 按文字、话题标签、用户搜索。
  • 针对提及、点赞、回复的实时通知。
  • 删除推文；如支持则可编辑。

非功能性需求：

  • 数亿日活跃用户。
  • 主页时间线读取延迟 p99 低于 200 ms。
  • 推文发布延迟低于 1 秒。
  • 时间线新鲜度有一定延迟可接受（秒级）。
  • 高可用性：写入中断期间时间线读取必须正常工作。
  • 删除和屏蔽必须快速生效（秒级，而非分钟级）。

以下内容超出范围，除非被特别询问：

  • 广告插入与排名。
  • 热门话题检测（独立模块）。
  • 私信（独立模块，参见 Facebook Messenger 设计）。
  • Spaces / 直播音频（独立模块）。

规模假设

领域 假设

瓶颈在于读取路径：主页时间线读取数量比写入多 4 个数量级。

写入路径：发布推文

  1. 客户端 POST /tweets.
  2. API 进行身份验证、校验、运行策略过滤器（垃圾信息、内容）。
  3. 生成 tweet_id （snowflake：时间戳 + 机器 + 序列号）。
  4. 写入推文存储，按以下方式分区 tweet_id.
  5. 向流发送 TweetCreated 事件写入流。
  6. 更新作者的 AuthorTimeline 缓存。
  7. 扇出决策： - 如果作者是"普通用户"（关注者低于阈值，例如 1 万到 10 万）：扇出 worker 消费事件并将 tweet_id 追加到每位关注者的 TimelineInbox。- 如果作者是"明星"（超过阈值）：跳过扇出；依赖读取时合并。
  8. 返回 tweet_id。

帖子响应不等待扇出完成。扇出是异步的；关注者可能在数秒后才能看到推文。

读取路径：主页时间线

  1. 客户端请求 /home.
  2. 主页时间线服务从缓存中获取用户的 TimelineInbox （最近 N 个推文 id）。
  3. 对于查看者关注的明星用户，从 AuthorTimeline 缓存。
  4. 合并收件箱与明星推文，按时间排序（或按排名分数排序）。
  5. 通过推文存储将推文 id 批量水合为完整推文内容。
  6. 应用可见性过滤：删除、屏蔽、静音、安全策略。
  7. 可选通过排名服务排名（超时后回退到按时间排序）。
  8. 返回页面。

复合读取速度快，因为收件箱是预计算的，而明星作者时间线很小且已缓存。推文水合是批量键值读取。

面试用语：

主页时间线是预计算收件箱与少量明星作者时间线的合并。收件箱承担扇出成本；明星合并使大账户的发帖路径保持低成本。

混合扇出：为何拆分

纯写时扇出（推送）：读取成本低（只需读取收件箱）。明星的写入成本高（一条推文 → 1 亿条收件箱追加）。

纯读时扇出（拉取）：写入成本低（只需存储）。读取成本高（每次请求需查询查看者关注的所有作者）。

混合方案：普通作者推送。明星在读取时拉取。一个关注了 200 个普通账户和 10 个明星的用户获取：预计算收件箱 + 10 次小型明星时间线读取。

阈值调优：设置使扇出写入成本等于读取放大收益的阈值。通常为 1 万到 100 万粉丝；取决于平台。

边界情况：一个非明星用户帖子病毒式传播。根据粉丝增长动态将其移入明星桶。

删除与可见性

被删除的推文必须快速消失。朴素方法：扫描所有时间线并删除。更优方案：

  • 将推文在推文存储中标记为 deleted 。
  • 读取路径的水合步骤对已删除推文不返回内容（或返回墓碑标记）。
  • 收件箱条目保留（存储开销小），但不显示任何内容。

对于屏蔽：当 A 屏蔽 B 时，读取路径在水合时从 A 的主页时间线过滤掉 B 的推文。无需重写收件箱。

对于受保护账户：推文上设置可见性标志；读取路径检查查看者是否为已批准的关注者。

面试用语：

删除和屏蔽在读取时应用，而非在写入时。收件箱是快速索引；事实存储在推文存储和可见性服务中。

排序

默认主页时间线按时间倒序排列。排名可选地基于互动信号重新排序：

  • 时效性。
  • 作者亲密度（查看者与作者的互动频率）。
  • 推文互动量（点赞、回复、转推）。
  • 话题相关性。

实现方式：

  • 排名是候选集生成后调用的独立服务。
  • 严格超时（例如 100 ms）。超时后回退到按时间排序。
  • 每用户缓存特征。

搜索

通过消费推文事件流构建的倒排索引：

  • 对推文内容进行分词；按 token 索引到 (tweet_id, posted_at).
  • 按 token hash 或词项 + 时间分片。
  • 实时近在线索引（秒级延迟）。
  • 独立于信息流分发；在自己的集群中运行。

扩展与分区

数据 分区键 原因
推文存储 tweet_id hash（snowflake 包含时间） 均匀分布；按 id 查找。
关注图 follower_id 用于出向； followee_id 用于入向 需要双向查询。
时间线收件箱 user_id 哈希值 按用户分区。
作者时间线 user_id 哈希值 按作者缓存。

一致性模型

层级 一致性 原因
推文存储 强写入 真实数据来源。
关注图 强写入 用于驱动扇出和可见性。

故障模式与缓解措施

故障模式 缓解措施 监控指标
热门帖子的扇出延迟。 自动扩缩扇出 worker；对明星进行采样；基于阈值的推送/拉取分流。 扇出积压；每作者延迟。
明星被错误分类导致推文泛滥收件箱。 动态明星检测；阈值调整；突发时回退。 收件箱写入速率激增。
已删除推文仍可见。 读取时水合检查；推文存储软删除；缓存 TTL。 过期读取告警。
屏蔽绕过。 读取路径的最终可见性关卡。 拉黑逃逸计数。
排名服务缓慢。 硬超时 + 按时间排序回退。 排名器超时率。
推文存储热分片（一条热门推文）。 副本；缓存层；嵌入媒体使用 CDN。 每分片 QPS。
收件箱缓存驱逐导致条目丢失。 从作者时间线 + 关注图谱重建（慢路径）。 缓存未命中率。
热门趋势事件。 扇出背压；优雅降级（无排名、更少推荐）。 按话题统计事件 QPS。
搜索延迟。 异步索引；显示"无结果"或部分结果。 索引延迟。

运维指标

正确性：

  • 过期帖子可见性（显示已删除推文）。
  • 屏蔽违规次数。
  • 收件箱与真实值的偏差（抽样）。
  • 扇出延迟分布。

容量：

  • 按地区统计读取 p50/p99。
  • 写入 p50/p99。
  • 扇出 worker 积压。
  • 缓存命中率。

业务层面：

  • 每用户每天推文数。
  • 主页时间线 DAU。
  • 互动率（每次时间线曝光的点赞/回复/转推数）。

多区域策略

  • 推文存储和关注图谱在全球复制；每位用户在主要写入时有一个归属地区。
  • 收件箱缓存按地区存储；通过事件流更新。
  • 从本地区域读取；仅在必要时跨区域获取尾部数据。
  • 每个区域维护独立的搜索索引，并复制推文数据。

需要指出的权衡

  • 混合扇出在写入和读取成本之间取得平衡，但比任何单一纯方案都更为复杂。
  • 预物化存储收件箱会占用大量内存；限制为最近 N 条可降低但无法消除成本。
  • 排序可提升用户参与度，但会增加延迟；严格超时机制可保护 Feed 的响应性。
  • 读取时的可见性过滤逻辑正确，但会增加额外的查询开销。
  • Snowflake ID 支持无需协调的时间排序，但需要谨慎管理时钟。
  • 最终一致性收件箱意味着新发布的推文可能不会立即出现；通常可以接受。

常见陷阱与更优做法

陷阱 更优做法
纯写入时扇出。 带名人拉取路径的混合方案。
在收件箱中存储完整推文内容。 存储推文 ID；在读取时进行内容填充。
在写入时应用屏蔽规则。 在读取时应用；收件箱仅存储内容 ID。
同步排序。 严格超时并回退至按时间顺序排列。
不设名人阈值。 基于关注者数量和发帖频率的动态阈值。
跨所有收件箱复制推文内容。 推文存储是唯一数据源；收件箱仅保存 ID。

深入方向 优秀回答要点
扇出策略 混合方式；阈值调优；病毒式晋升。
可见性与删除 读取时内容填充；存储中标记墓碑。
排序 带超时的旁路服务；特征缓存。

Staff 级别回答检查清单

  • 混合扇出：普通用户推送，名人用户拉取。
  • 推文存储为权威来源；收件箱保存 ID。
  • 读取时内容填充并进行可见性过滤。
  • 排序作为带降级的旁路服务。
  • Snowflake ID 实现有序且无需协调的 ID 生成。
  • 多区域部署，配合区域缓存。
  • 异步搜索索引。
  • 监测正确性漂移的运营指标。

面试用语

Twitter 的设计核心是扇出策略。普通作者将推文推送到预计算的收件箱；名人则在读取时拉取。推文内容存储于唯一的权威存储中；收件箱是 ID 列表。可见性、删除和屏蔽在读取时应用，而非写入时。排序和搜索是独立的旁路服务。

2. 模拟面试

面试官： 设计 Twitter。

候选人： 我将围绕混合扇出策略进行设计。权威推文存储和关注图谱是数据的唯一真实来源。对于普通作者，我们将推文 ID 推送至关注者收件箱（写时扇出）。对于拥有大量关注者的名人，我们跳过扇出，在读取时拉取。主页时间线将预计算的收件箱与名人作者的时间线合并。不变量为：每位关注者最终都能看到所有已发布的推文，受可见性规则约束。

面试官： 请介绍一下发布推文的流程。

候选人： 客户端以 POST 方式提交推文文本。系统进行身份验证、执行策略检查、分配 Snowflake ID（带时间戳前缀以实现自然时间排序），并将数据持久化至按 tweet_id 分片的推文存储。向数据流发出一个 TweetCreated 事件，更新作者最近推文缓存。若作者的关注者数低于名人阈值，扇出 Worker 将 tweet_id 追加至每位关注者的收件箱。若作者为名人，则跳过扇出。响应返回 tweet_id，无需等待扇出完成。

面试官： 请介绍一下读取主页时间线的流程。

候选人： 从 Redis 获取用户收件箱——包含最近推文 ID 的列表，上限约为 800 条。识别被关注的名人用户，从缓存中获取其最近的作者时间线。将收件箱和名人推文按时间排序合并。通过对推文存储的批量读取，将推文 ID 填充为完整内容。应用可见性过滤：删除、屏蔽、静音、年龄限制。可选地调用排序服务并设置严格超时；超时时回退至按时间顺序排列。返回当前页面。

面试官： 一位名人拥有 1 亿关注者，发帖时会发生什么？

候选人： 若采用纯写时扇出，我们需要追加 1 亿次——耗时数分钟且写放大极为严重。混合方案可避免这一问题：名人不进行扇出。其关注者的主页时间线在读取时，通过每位作者的小型缓存来拉取名人推文。代价是读放大——对于关注了 10 位名人的用户，只需 10 次小型缓存查询——远比每次发帖 1 亿次写入便宜。

面试官： 如何设定名人阈值？

候选人： 调优目标是使总体工作量最小化。若某用户有 F 个关注者且每天发 P 条推文，扇出成本为 F * P 次写入/天。若有 R 个关注者每天读取时间线 R_r 次，拉取成本为 R * R_r 次读取/天。找到推送成本超过拉取成本的 F 值。实践中通常在 1 万到 100 万之间。动态调整：快速增长关注者的用户会被晋升；不活跃账号则被降级。

面试官： 某用户突然爆红，但仍被归类为普通用户。

候选人： 检测到关注者快速增长时，将其晋升为名人状态。晋升前已积累的扇出积压正常完成；新发推文使用拉取方式。此外，若某普通用户的单条推文病毒式传播（转发级联），扇出 Worker 可能会积压。我们采用背压机制：队列深度告警、动态 Worker 扩容，以及一个降级方案——若队列过深则停止推送，对这些关注者改用拉取模式。

面试官： 如何处理删除？

候选人： 在推文存储中将推文标记为 deleted 已删除状态。不扫描并清理收件箱——收件箱保存的是 tweet_id，读取时的内容填充会感知到已删除状态并对该条目返回空内容。缓存遵守 TTL 或失效事件。因此，删除后数秒内推文即从 Feed 中消失，无需触碰数十亿条收件箱记录。

面试官： 屏蔽呢？

候选人： 同样的模式。屏蔽是关注边上的一个状态，或存储在独立的屏蔽列表中。在读取时内容填充阶段，可见性服务检查：该作者是否在我的屏蔽列表中？该作者是否设置了保护且我未获批准？过滤掉。当屏蔽关系变更时无需重写收件箱；下次读取时即可感知到新的屏蔽状态。

面试官： 搜索？

候选人： 独立的服务。索引器消费推文事件流，构建倒排索引——词条 → 包含 (tweet_id, posted_at) 的倒排列表。按词条哈希或按词条 + 时间分片。最终一致性：推文发布后数秒内可被搜索到。读取模式为：查询 → 词条查找 → 评分 → 返回推文 ID → 通过推文存储填充内容。搜索服务不与 Feed 分发路径交互。

面试官： 排序？

候选人： 带严格超时的旁路服务。主页时间线服务生成候选集（收件箱 + 名人推文）。将候选 ID 和查看者上下文发送至排序服务。排序器返回带分数的 ID。若超时，则回退至按时间顺序排列。排序器使用的特征包括：查看者与作者的亲密度、推文互动量、时效性。重量级特征是预计算的。

面试官： 多区域？

候选人： 推文存储和关注图谱在全球范围内复制，每位用户在其归属区域进行主写入。收件箱缓存是区域性的，并从事件流中重建。读取来自本地区域。跨区域复制存在有界延迟，通常为数秒。对于发帖，我们接受一小段时间窗口内不同区域的用户在略有不同的时间看到新推文。

面试官： 热点事件——世界杯进球。接下来会发生什么？

候选人： 大量发帖、转发、点赞涌入。扇出 Worker 扩容。我们启用背压：若队列积压，暂时推迟部分扇出，对受影响的作者临时切换为拉取模式。排序服务在高负载下可能降级为按时间顺序排列。搜索索引优先处理热点词条。热门推文周围的缓存吸收大量读取请求。我们进行监控并优雅降级——Feed 响应略有变慢，但保持正确性。

面试官： 总结。

候选人： 混合扇出。普通作者将推文 ID 推送至关注者收件箱；名人在读取时拉取。推文内容存储于唯一的权威存储中；收件箱仅保存 ID。读取路径填充 ID、应用可见性过滤、在超时限制内可选地进行排序。Snowflake ID 无需协调即可实现时间有序的推文 ID。搜索和排序是旁路服务。多区域部署使用每用户归属区域及最终一致性副本。最难处理的场景是名人和病毒式传播峰值，通过动态阈值晋升和背压来应对。

3. 练习备注

五分钟版本

  1. 明确：规模、延迟、病毒案例、删除处理。
  2. 不变量：每条已发布的推文最终对关注者可见，受可见性规则约束。
  3. 混合扇出：普通用户推送，名人用户拉取。
  4. 权威推文存储；收件箱仅保存 ID。
  5. Snowflake ID 用于时间排序。
  6. 读取时可见性处理：删除、屏蔽、静音。
  7. 排序 + 搜索作为带超时的旁路服务。
  8. 多区域部署，每用户绑定归属区域。

一分钟结尾陈述

Twitter 是混合扇出问题的经典案例。普通作者将推文 ID 推送到预计算的收件箱；名人在读取时拉取。推文存储是唯一的真实来源；收件箱和缓存是其投影。删除和屏蔽通过可见性门在读取时应用，而非重写收件箱。排序和搜索是带超时和异步索引的旁路服务。多区域部署使用每用户归属区域和最终一致性副本收件箱缓存。最难处理的场景——名人、病毒式流量峰值——通过动态阈值和背压来应对。

最佳面试用语

  • "混合扇出：普通作者推送，名人拉取。"
  • "收件箱保存推文 ID；推文存储是内容的唯一来源。"
  • "删除和屏蔽在读取时应用；不要重写数十亿条收件箱记录。"
  • "排序是一个带硬超时和按时间顺序降级的旁路服务。"
  • "Snowflake ID 无需全局协调即可实现时间有序的推文 ID。"
