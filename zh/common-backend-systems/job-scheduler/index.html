Distributed Job Scheduler — Cron at Scale (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Hard · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建分布式 Cron / 任务调度器。用户注册带有调度计划（Cron 表达式或一次性时间戳）和负载（待 POST 的 URL 或待调用的函数）的任务。系统在计划时间精确触发每个任务一次，失败时重试。已注册任务 1000 万个，顶峰时每分钟整点约 10 万个任务同时触发，平均任务运行时长 5 秒。

1. 需求（约 5 分钟）

功能需求 - 注册带有 Cron 表达式和处理器的周期性或一次性任务。\n- 在计划时间的约 1 秒内触发每个任务。\n- 至多一次或至少一次语义（可配置）；绝不静默漏触发。\n- 失败时按退避策略重试。\n- 可见性：查看任务历史及成功/失败状态。

我会推迟多租户深度和处理器运行时沙箱的设计（假设目标为 HTTP Webhook）。

非功能需求 - 触发时间精度：p99 在计划时间 1 秒以内。\n- 调度器节点故障不丢失触发记录。\n- 不得静默漏触发。 宁可双重触发（要求处理器幂等）也不能漏触发。\n- 线性扩展：1000 万个任务，峰值 10 万次触发/秒。

容量估算 - 1000 万个已注册任务 × 约 200 字节元数据 = 2 GB。\n- 10 万次触发/秒 × 约 5 秒处理器运行时长 = 50 万个并发执行中实例。\n- 调度元数据经复制后体量极小（约 2 GB × RF=3 = 6 GB）。主要成本来自运行历史及重试/审计事件：如果峰值 10 万次触发/秒持续一小时，以约 300 字节/事件 × RF=3 计，仅该小时就会产生约 324 GB 数据（不含索引）。

2. 核心实体（约 2 分钟）

  • Job — job_id，调度计划（Cron / 一次性），目标（URL + 负载），重试策略，所有者。
  • JobInstance ——具体的一次触发： instance_id，job_id，scheduled_at，status（queued/running/done/failed）。
  • Lease ——防止两个 Worker 运行同一实例的租约。

3. API（约 5 分钟）

POST  /v1/jobs              { schedule, target, retry_policy }  → { job_id }
GET   /v1/jobs/{id}/runs?cursor=...
DELETE /v1/jobs/{id}
POST  /v1/jobs/{id}/pause / resume

从 Token 获取用户身份。任务创建对可选字段幂等。 client_id.

运行历史游标编码 {job_id, last_scheduled_at, last_instance_id}。处理器调用包含 fire_id ，供客户在自身侧对重试进行去重。

4. 数据流（约 5 分钟）

任务注册 → 计算 next_fire_at → 在时间桶中建立索引。时间轮逐秒遍历时间桶 → Worker 通过租约认领到期任务 → 调用处理器 → 记录结果 → 重新计算 next_fire_at。

5. 高层设计（约 10–15 分钟）

  1. 任务目录。 以 job_id为键的分片 MySQL。行存储调度计划、目标和 next_fire_at.
  2. 调度索引。 用于"未来 N 秒内到期"的时间桶索引。有两种适合该问题的方案：\n - 时间轮： 内存中的时间桶数组，每秒（或更细粒度）一个槽位，每个槽位持有 job_id在该槽位到期的任务。每个槽位 tick 成本为 O(1)。状态持久化在后备存储中，以防崩溃丢失状态。\n - 有序集合 / 堆，按 next_fire_at: Redis ZSET 以 next_fire_at为键。Worker 执行 ZRANGEBYSCORE 0 now 并消费。
  3. 调度器（Dispatcher）。 每次 tick（1 Hz）遍历时间轮/ZSET。对每个到期任务，写入一行 JobInstance （status=queued）并向任务队列（Kafka 或 SQS）推送触发事件。
  4. Worker 池。 拉取触发事件。每个 Worker：\n - 获取租约（Redis SET instance:I worker:W NX EX 60）。\n - 调用处理器（向目标 HTTP POST，携带负载 + 签名重试 Token）。\n - 记录结果；释放租约。
  5. 重试策略。 失败时，计算下次重试时间 = now + backoff(attempt)。用该触发时间调度一个新的 JobInstance 。限制重试次数。
  6. 下次触发时间重算。 成功触发后，根据 Cron 表达式重新计算 next_fire_at ，并同时更新目录行和调度索引。
  7. 协调。 多个 Dispatcher 实例。每个实例负责一部分 job_id的分片（一致性哈希）。分片内通过主节点选举确定活跃 Dispatcher；备节点在故障时接管。

端到端叙述：09:59:55，Dispatcher 遍历 10:00:00 槽位的时间轮，发现 10 万个到期任务 → 批量写入 10 万行 JobInstance → 推送到 Kafka。10:00:00，Worker 从 Kafka 拉取，对每个实例加租约，向目标发起 HTTP POST。处理器在 200 ms 内响应 200 → 记录结果 → Dispatcher 重新计算下次触发时间（例如， 10:01:00 对于每分钟任务）并重新插入时间轮。

具体存储方案：

存储 我会明确说出的具体存储方案： 条目大小 TTL / 保留期 分片键
作业 catalog job:{job_id} 行，含调度计划、目标、next_fire_at ~200-500 B 直至删除 job_id
调度索引 due:{epoch_second}:{shard} 列表或 ZSET 成员=job_id 约 24 字节/成员 直至触发/重建 job_id % N
实例数据库 instance:{job_id}:{scheduled_at} 状态行 ~300 B 30-90 天 job_id
Lease lease:{instance_id} -> Worker + 防护令牌 ~100 B 60 秒 TTL，按需续期 instance_id
触发队列 主题 fires.{shard} 事件=instance_id 约 200 B 24 小时重放 job_id % N

按存储容量估算：

存储 Math 主要调节项
作业 catalog 1000 万任务 * 500 字节 * RF=3 ≈ 15 GB 字节数不是关键，分片数量更重要
调度索引 1000 万条到期记录 * 24 字节 * RF=2 ≈ 480 MB 时间桶宽度和重建来源
实例数据库 5000 万次触发/天 * 300 字节 * RF=3 ≈ 45 GB/天；10 万次/秒持续 1 小时 ≈ 324 GB 历史保留与汇总策略
租约 50 万并发执行 * 100 字节 * RF=2 ≈ 100 MB TTL/心跳间隔
触发队列 5000 万/天 * 200 字节 * RF=3 ≈ 30 GB/天；持续峰值会远高于此 保留时长与批处理策略

6. 深度探讨（约 10 分钟）

为什么选择时间轮。 在 1000 万个任务、峰值 10 万次/秒触发的场景下，每秒扫描堆中到期任务的代价很高。时间轮每次 tick 的成本为 O(当前槽位内的活跃任务数)。对于 1 Hz 的 1 小时时间轮，共有 3600 个槽位；超出轮盘范围的任务由分层第二时间轮处理（例如，日轮向小时轮投递）。分层时间轮是经典答案。

为什么不只用 Redis ZSET：以 next_fire_at 为键的 ZSET 是一个良好的持久化索引，v1 也更简单，但在每分钟整点时会产生热点范围扫描和锁竞争。时间轮提供可预期的每槽位工作量；ZSET/Kafka 日志是恢复数据来源。

技术选型依据：MySQL/Postgres 非常适合任务目录，因为更新是按任务进行的，需要支持创建/暂停/删除的事务语义；如果面试偏向云原生，DynamoDB 同样可用，但按所有者/历史进行二级访问更为繁琐。Kafka 优于直接 Worker RPC，因为调度突发需要缓冲和重放；SQS 运维上更简单，但对分区归属和按 job_id % N有序重放的控制力较弱。Redis 租约在秒级协调上足够快，而 etcd/ZooKeeper 只应管理 Dispatcher 分片的主节点选举，而非每个任务实例。

持久化与恢复。 时间轮位于内存中；若进程崩溃，正在处理的槽位会丢失。两种防御措施：\n- 以 Redis 有序集合作为后备存储（重启时即可重建时间轮）。\n- 定期快照到持久化存储 + 从 Kafka Topic 重放近期事件。

注册路径跟踪：API 校验 Cron 并检查所有者配额 -> 写入 job:{job_id} 与 next_fire_at -> 写入 due:{epoch_second}:{shard} -> 发出 job.registered 审计事件。目录行与调度索引的更新必须原子完成，或通过修复任务进行对账（扫描目录中未出现在索引里的行）。

触发路径跟踪：Dispatcher 拥有分片 -> 读取时间轮槽位 -> 以幂等插入方式创建 instance:{job_id}:{scheduled_at} -> 发布 fires.{shard} -> Worker 获取 lease:{instance_id} -> 调用目标 -> 写入终态状态 -> 重新计算下次触发时间并重新插入任务。

热触发路径，逐步说明：\n1. Dispatcher 通过协调器租约拥有哈希范围 job_id % N = shard 。\n2. 在 tick T时，它从内存中排空 due:{T}:{shard} ；若时间轮槽位缺失，则回退到 Redis/ZSET 恢复索引以处理 [T - grace, T]。\n3. 以幂等方式批量插入 instance:{job_id}:{scheduled_at} ，幂等键为 (job_id, scheduled_at, attempt)。\n4. 将紧凑的触发消息发布到 fires.{shard} ，并提交槽位水位线。\n5. Worker 按分片拉取，读取实例行，以防护令牌获取 lease:{instance_id} ，调用目标，仅在防护令牌有效时写入终态状态。

冷启动/恢复路径：Dispatcher 故障后，备节点认领分片，读取最后提交的水位线，扫描调度索引以处理 [watermark, now]，并重建缺失的实例行。重复触发事件是允许的，因为实例插入和客户可见的 fire_id 均为幂等；漏触发则不被允许。

运行历史分页使用 {job_id, last_scheduled_at, last_instance_id} ，使 UI 能按时间顺序遍历 job_id 分片，无需偏移量扫描。

精确一次 vs 至少一次。 在分布式调度器中实现真正的精确一次是不切实际的目标。承诺至少一次 + 要求处理器幂等（通过处理器可用于去重的 fire_id ）。在 SDK 中清晰说明这一点。

基于租约的双重触发防护。 Worker 获取 SET instance:I worker:W NX EX 60。若 Worker 宕机，租约到期，其他 Worker 可重新认领。Worker 定期续期租约（心跳），用于长时间运行的任务。

热秒问题。 每分钟整点是惊群现象——所有人的 */1 * * * * 任务都落在 :00。两种缓解措施：\n- 注册时添加抖动： 自动对调度计划添加 ±5 秒的抖动（为严格调度的用户提供可选关闭）。\n- 槽位内平滑： 在调度时，使用一个小抖动缓冲区，在整秒内均匀分散触发。

扩展调度分发。 在 N 个 Dispatcher 实例间对 job_id 进行分片。每个实例拥有一个哈希范围；任务只从一个所有者触发。协调器（etcd / ZooKeeper）追踪所有权；故障时，范围进行故障转移。

Worker 池的背压。 若目标处理较慢，队列增长。根据队列深度自动扩缩 Worker。对于突发目标，允许设置每目标并发限制（不要向单个端点同时发起 1 万次并发调用）。

双视图模型。 调度器以两种视图保存同一个未来触发：目录行（job:{job_id}.next_fire_at）是修复的真相来源，而调度索引（due:{second}:{shard} 或 ZSET）是下次 tick 的服务索引。修复循环持续对二者进行对账，确保丢失的时间桶不会变成静默漏触发。

任务实例状态机：

阶段 模式 数据源 状态转换 延迟/行为
已调度 未来触发 调度索引 + 目录 tick 到达 scheduled_at p99 在 1 秒目标以内
已入队 触发事件已发出 触发队列 + 实例行 Worker 拉取 正常负载下为秒级
运行中 持有租约 租约 + worker 心跳 目标返回或超时 每 15-30 秒续约
重试中 失败的尝试 实例 DB + 重试策略 退避到期 带抖动的重试
终态 完成/失败/死信 实例 DB 成功或尝试次数耗尽 保留 30-90 天

边缘压力测试。 分钟级突发流量由抖动、槽位分散和批量实例写入来处理。时间轮条目已过期的休眠任务可从 job:{job_id}.next_fire_at 中由修复扫描恢复。若用户翻阅非常久远的运行历史，API 会读取归档实例分区而非热数据行，并返回较慢的游标。多区域主动-主动调度风险较高，因为两个区域可能同时触发同一任务；我会为每个任务保持单一写入/分发区域，并将历史数据全局复制。冷启动新的调度器时只需重建其分片窗口，而非全部 1000 万个任务。

Webhook 可靠性。 目标可能不稳定。使用指数退避 + 抖动进行重试，最多重试 N 次。重试耗尽后，路由到死信队列并在 UI 中展示。

Cron 漂移 / 时区。 调度计划以用户指定的时区存储。服务器在解析时将时间转换为 UTC 以计算 next_fire_at。夏令时切换需要特别处理（跳过的小时、重复的小时）——需明确约定规范。

可审计性。 每次触发都会发出一条结构化事件。JobInstance 历史保留 30 天以上，便于调试和 SLA 报告。

优雅降级。

情形 模式 数据源 延迟/行为
调度器宕机 备用节点接管分片并重放到期区间 调度索引 + Kafka 可能产生重复，但不会丢失
分钟级热点槽位 分散并批量写入实例 时间轮槽位 根据 SLA 在 1-5 秒内触发
Worker 池饱和 队列增长，自动扩容，按目标设置上限 触发队列 延迟，积压可见
租约在运行中途到期 新 worker 可能以相同 ID 重试 fire_id 租约 + 实例 DB 至少一次语义
调度索引损坏 从目录重建 next_fire_at 任务目录 重建期间精度下降

面试要点："我宁愿以稳定的 fire_id 也不假装实现精确一次；产品承诺是不出现静默丢失。"
