1. 基础知识与解题思路

本题考察点

支付系统用于转移资金。每个 bug 都可能导致监管投诉、拒付，或某个会计师需要追查的丢失金额。难点不在于吞吐量，而在于每笔交易在众多外部系统的生命周期内的正确性，这些系统可能超时、返回歧义响应，或在不通知你的情况下改变状态。

其不变量如下：

对于每个支付意图，系统记录且仅记录一个终态结果（已结算、已退款、已撤销、已拒绝），每一步的资金流动以复式记账方式记录在不可变账本中，每个外部副作用使用幂等性，并通过与 provider 对账单的核对来检测内部日志遗漏的漂移。

在 ICT5 / L6+ 级别，面试官在测试你是否能将资金建模为状态机，为 provider 设计幂等调用，处理超时的"未知状态"结果，构建复式记账账本，以及针对外部真相来源进行每日对账。

60 秒面试框架

口述版本：

我会围绕三个核心设计支付系统：支付意图状态机、作为资金流动真相来源的复式记账账本，以及包裹在对账循环中的幂等 provider 集成。有趣的决策在于如何处理 provider 超时、如何建模退款和拒付、如何端到端强制幂等，以及当我们的记录与 provider 的记录不一致时如何对账。

首先明确需求

功能性需求：

  • 客户通过多种方式付款：卡、ACH、电子钱包、BNPL、银行转账。
  • 商户在结算后收到资金。
  • 退款、部分退款、撤销、拒付。
  • 多货币。
  • 循环计费 / 订阅。
  • 来自客户端和 provider 的幂等重试。
  • 来自 provider 的 Webhook（成功、失败、拒付、争议）。
  • 面向财务团队的报表和对账。

非功能性需求：

  • 强正确性：不重复扣款，不丢失交易。
  • 对 provider 故障具有弹性。
  • 可审计性：每次状态变更均可追溯。
  • 合规：PCI-DSS，地区性法规（PSD2 SCA、KYC、AML）。
  • 结算可能需要几秒（卡）到几天（ACH）。
  • 授权步骤（与结账流程内联）的高可用性。

以下内容超出范围，除非被特别询问：

  • 欺诈评分算法（视为来自欺诈服务的输入）。
  • 税费计算（独立系统）。
  • 商户入驻 KYC（独立系统）。
  • 卡号保险库 / PAN 存储（委托给保险库）。

规模假设

领域 假设
交易量 峰值每天最多数百万笔交易；跨市场累计数千万笔。
Latency（延迟） 授权 p99 低于 2 秒（主要由 provider 往返时间决定）。
结算窗口 卡当天捕获；ACH 1-5 个工作日。
退款窗口 扣款后最长 180 天。
拒付窗口 某些网络最长 540 天。
多区域 按区域划分的发卡行和收单行；中央账本。
对账节奏 provider 结算每日一次；高风险场景接近实时。
存储 多年的交易历史；不可变。

瓶颈在于 provider 集成可靠性和对账——而非原始 QPS。处理 1000 TPS 的支付系统已经是很大的规模。

核心产品模型

三层：

层级 职责 需求
支付意图状态机 管理每笔支付从创建到终态的生命周期。 强一致性；显式状态转换。
资金账本 对所有内部账户的每笔贷记/借记进行复式记录。 仅追加、不可变、平衡。
Provider 集成 对卡网络、银行、电子钱包的调用。 幂等；与 provider 对账单核对。

最大的设计错误是将意图状态与账本条目混为一谈，或让应用跳过账本直接信任 provider 响应。

核心组件

组件 职责 关键细节
Payment API 接收来自商户的意图创建、捕获、退款请求。 幂等，基于 Idempotency-Key.
支付意图状态机 管理完整生命周期。 条件转换；每次变更均有审计。
风控服务 返回欺诈评分和允许的操作。 根据延迟预算选择异步或同步。
卡号保险库 存储 token 化的卡片数据。 PCI-DSS 隔离；返回 token，而非 PAN。
幂等性存储 缓存请求 id → 响应，以支持安全重试。 以...为键 (merchant_id, idempotency_key).
Provider 适配器 将内部 API 转换为 provider 的 API。 每个 provider 一个；熔断器；重试策略。
Webhook 接收器 消费 provider 的异步通知。 验证签名；基于事件 id 保证幂等。

结算服务 捕获完成后向商户付款。 持有资金；遵守持有期。
争议处理器 管理拒付和争议。 提供证据，更新意图状态。

允许的状态转换是显式的；每次转换都以当前状态为条件并基于转换键保证幂等。账本仅在特定转换时更新：

  • authorized：持有，但资金不移动（取决于支付轨道）。
  • captured：账本条目：借记商户待结算 / 贷记收单行。
  • settled：账本条目：provider 资金已到账。
  • refunded：账本条目：冲销。
  • charged_back：账本条目：冲销 + 手续费。

面试用语：

资金永不隐式移动。每条账本条目对应一个显式状态转换，并附有记录的原因和幂等键。

端到端幂等性

三层幂等性：

  1. 客户端 → 支付 API： Idempotency-Key 请求头。幂等性存储以该键为索引缓存响应， (merchant_id, idempotency_key) 有效期为 24-48 小时。使用相同键重试将返回相同响应。
  2. 支付 API → 服务商： 特定服务商的幂等机制（Stripe 的 Idempotency-Key，其他服务商使用请求 ID）。适配器为每次尝试生成确定性键。
  3. 状态机转换： 每次转换对 (intent_id, from_state)执行一次 CAS 更新。相同的转换请求始终产生相同结果。

关键情况：向服务商发起的扣款超时，我们无法得知扣款是否成功。

  • 将支付意图状态标记为 processing 与 provider_status_unknown.
  • 对账 Worker 使用相同的幂等键向服务商查询结果。
  • 如果服务商确认成功，则转换至 authorized 或 captured 并写入账本条目。
  • 如果失败，则转换至 failed.
  • 我们不会对同一笔支付意图向客户重复扣款。

面试用语：

超时是核心正确性问题。系统必须将"未知"作为一等状态来处理，并通过对账而非猜测来解决。

双式记账账本

每笔资金流动都记录两条分录：

debit  account_A   amount   transaction_id
credit account_B   amount   transaction_id

内部账户：

  • 商户待结算账户（介于扣款与结算之间）。
  • 商户可用账户（结算后）。
  • 服务商在途账户。
  • 已收手续费账户。
  • 退款待处理账户。
  • 拒付欠款账户。

客户向商户 M 支付 $100：

At capture:
  debit  customer_funding             $100
  credit merchant_M_pending           $97
  credit platform_fee_revenue         $3

At settlement (provider deposits):
  debit  provider_in_transit          $97
  credit merchant_M_available         $97

在任意时间点，每笔交易的借方 == 贷方（双式记账平衡），所有余额之和等于零。任何偏差都意味着缺少分录，对账引擎会检测到这一情况。

特性：

  • 仅追加写入；不支持更新或删除。
  • 每条分录包含 transaction_id, account, amount, currency, direction, timestamp, reference_intent.
  • 按账户建立索引以支持余额查询，按支付意图建立索引以支持可追溯性。

写入路径：授权 / 扣款

  1. 商户携带幂等键发起 POST 创建支付意图。
  2. 支付 API 检查幂等性存储；若缓存响应存在，则直接返回。
  3. 校验金额、币种、支付方式。
  4. 风控服务打分；可能提前拒绝。
  5. 生成 intent_id；持久化，同时写入 state=created.
  6. 携带确定性幂等键调用服务商适配器。
  7. 服务商响应：\n - 成功：转换至 authorized；若为自动扣款模式，则写入账本分录。\n - 失败：转换至 failed。\n - 超时：转换至 processing，调度对账任务。\n - SCA 验证：转换至 requires_action；返回验证 URL。
  8. 将响应持久化至幂等性存储。
  9. 向商户返回响应。

对于手动扣款，后续单独的 /capture 调用触发实际资金流动。

Webhook 路径：异步更新

服务商发送 Webhook 传递异步事件：

  1. Webhook 接收器验证签名。
  2. 按 provider_event_id去重；忽略重复请求。
  3. 持久化原始事件。
  4. 查找受影响的支付意图；通过 CAS 执行状态转换。
  5. 如适用，写入账本分录。
  6. 通过商户侧 Webhook 通知商户。

Webhook 可能乱序到达或重复投递。两种情况均通过 provider_event_id 上的幂等性和支付意图状态上的 CAS 来处理。

对账

每日与服务商结算文件对账：

  1. 从服务商拉取结算文件（CSV / SFTP / API）。
  2. 对于每条服务商记录，找到匹配的支付意图和账本分录。
  3. 匹配 provider_charge_id 以及金额。
  4. 不匹配的记录进入人工审核队列。
  5. 我方账本总和与服务商预期总和之间的偏差触发告警。

此机制可捕获以下问题：

  • 我方漏收的 Webhook。
  • 我方已记录但服务商未确认的扣款。
  • 货币换算错误。
  • 手续费差异。

面试用语：

对账是对基于事件的流水线可能遗漏内容的安全兜底。它运行较慢，但具有权威性。我方账本与服务商结算数据之间的偏差是系统性缺陷的预警指标。

退款、撤销与拒付

退款： 将资金退还给客户。从 captured 或 settled 与 refunded发起新的状态转换。账本分录冲销原始贷方条目。

撤销： 在扣款前取消。转换至 authorized → voided。无账本分录（未发生资金流动）。

拒付： 客户通过卡组织发起争议。我方收到 Webhook；转换至 captured → disputed 再转至 charged_back ，直至争议解决。账本分录冲销原始金额并追加拒付手续费。我方介入争议处理流程以提供证据。

多币种

每个支付意图关联一种币种。账本分录以其原始币种记录金额。跨币种换算（例如，商户以 USD 结算、客户以 EUR 付款）涉及一个独立的外汇腿：

debit  customer_funding_EUR        €100
credit fx_holding_EUR              €100

debit  fx_holding_USD              $108
credit merchant_pending_USD        $108

FX 汇率在扣款时锁定；由独立的外汇服务提供汇率。

层级 一致性 原因
支付意图状态转换 强一致性（CAS） 资金正确性。
账本写入 强一致性；与状态转换事务性绑定 双式记账平衡。

故障模式与缓解措施

故障模式 缓解措施 监控指标
服务商超时。 支付意图处于 processing 状态；对账 Worker 查询服务商。 未知状态计数；对账延迟。
服务商故障。 熔断器；队列缓存支付意图；向商户返回可见错误。 服务商可用性；队列深度。
重复 Webhook。 在 provider_event_id. 重复事件率。
Webhook 乱序。 对支付意图状态执行 CAS；幂等性转换。 乱序转换尝试次数。
账本偏差。 每日对账；告警；不匹配时隔离处理。 每商户/每币种的偏差量。
Webhook 丢失。 通过结算文件由对账捕获。 缺失的支付意图更新。
未扣款即退款。 状态机拒绝无效转换。 无效转换尝试次数。
重试导致重复扣款。 API 层幂等性；服务商侧幂等性。 重复扣款投诉量。
账本中的币种不匹配。 Schema 强制校验；按币种独立账户。 跨币种分录数量。
拒付量激增。 风险评分调优；人工审核队列。 每商户拒付率。

运维指标

正确性：

  • 每账户每日账本偏差。
  • 超过阈值的未知状态支付意图数量。
  • 未匹配的服务商结算条目。
  • 重复 Webhook 率。
  • 未经扣款的退款（不应发生，如出现则告警）。

容量：

  • 支付意图创建速率。
  • 每服务商的调用延迟与错误率。
  • 对账耗时。
  • Webhook 队列深度。

业务层面：

  • 授权成功率。
  • 从扣款到结算的时间。
  • 拒付比率。
  • 退款比率。
  • 按服务商、按币种划分的收入。

多区域策略

  • 发卡行与收单行具有地域性；我们在每个地区维护一个服务商。
  • 支付意图状态和账本写入路由至商户的归属地区。
  • 跨地区复制用于只读副本和灾难恢复。
  • Webhook 落地于服务商发送的目标地区；再路由至商户所在地区。

需要指出的权衡

  • 同步服务商调用会增加延迟，但可为用户提供即时反馈。异步 Webhook 提供最终一致性更新。
  • 激进的风险拒绝可降低欺诈，但会损害转化率。
  • 每日对账运行较慢但具有权威性；实时对账速度快但噪声较多。
  • 按商户的幂等性是安全的，但存储会不断增长；通过 TTL 限制保留时长。
  • 单一账本具有可审计性；分片账本可扩展，但会使余额查询变复杂。
  • 将卡片 Token 化存储于 Vault 会增加延迟，但属于 PCI 合规要求。

常见陷阱与更优做法

陷阱 更优做法
在未以事务方式写入账本的情况下信任服务商的成功响应。 将账本分录作为状态转换 CAS 的一部分写入。
忽略超时。 原生一等支持 unknown 状态；对账 Worker。
每笔交易单行账本记录。 双式记账：每笔交易均有匹配的借贷分录。
将退款视为新的扣款处理。 退款是一次状态转换，附带反向账本分录。
忘记 Webhook 可能重复投递。 在 provider_event_id；幂等性转换。
无对账机制。 每日服务商结算匹配；偏差告警。

面试官可能深入探讨的方向

深入方向 优秀回答要点
超时处理 未知状态；对账；在未获服务商确认前绝不自动重试扣款。

Staff 级别回答检查清单

  • 将支付建模为具有显式转换的状态机。
  • 以双式记账账本作为资金流动的唯一真实来源。
  • 端到端幂等性：API、服务商、状态 CAS。
  • 将超时的未知状态作为一等状态；通过对账解决。

面试用语

支付系统的核心是正确性，而非规模。每笔扣款都是状态机转换，并写入平衡的双式记账账本；幂等性贯穿端到端；服务商超时导致的未知结果是一等状态，通过对账解决。对账是对基于事件的流水线可能遗漏任何偏差的预警机制。

2. 模拟面试

面试官： 设计一个支付系统。

候选人： 我将设计三个层次：支付意图状态机、作为资金流动唯一真实来源的双式记账账本，以及包裹了幂等性和对账机制的服务商集成。核心不变式是：每个支付意图有且仅有一个终态结果，每笔资金流动由两条平衡的账本分录记录，外部超时通过对账解决而非凭猜测重试。

面试官： 请描述一笔卡片扣款的完整流程。

候选人： 商户通过 POST 请求携带幂等键提交支付意图。Payment API 检查幂等存储。若为新请求，则以 created 状态创建意图，执行风险评分，然后携带确定性幂等键调用卡片提供商。成功后，转换为 authorized 并写入分类账条目（借记客户资金账户，贷记商户待结算账户）。若超时，则转换为 processing 并由对账工作进程向提供商查询。提供商推送的 Webhook 更新（授权确认、结算）驱动后续状态转换。

面试官： 提供商超时，接下来怎么处理？

候选人： 这是核心正确性场景。意图状态转为 processing 与 provider_status_unknown。对账工作进程使用相同幂等键向提供商发起查询。若提供商返回成功，则转换为已授权状态并写入分类账条目；若返回失败，则转换为失败状态。我们绝不盲目重试扣款，那可能导致重复扣费；也绝不假设失败，那可能导致客户已被扣款但我们的记录为空。

面试官： 请展示一笔 $100 扣款的复式记账分类账。

候选人： 授权时，两条分录：借记 customer_funding $100，贷记 merchant_X_pending $97，贷记 platform_fee_revenue $3。结算时：借记 provider_in_transit $97，贷记 merchant_X_available $97。每笔交易平衡，所有余额之和为零。每日试算平衡检查确保无分录缺失。

面试官： 对账机制如何运作？

候选人： 每日拉取提供商的结算文件。逐行通过 provider_charge_id 匹配对应意图和分类账条目。不匹配的条目进入审核队列。我方分类账与提供商预期汇总之间的聚合偏差触发告警。这可捕获 Webhook 流水线遗漏的一切——重复事件、丢失事件、手续费差异。分类账是内部真相，提供商结算是外部真相，对账使两者达成一致或上报升级。

面试官： 同一 Webhook 到达两次。

候选人： 在 provider_event_id。Webhook 接收器存储每个事件 id；首次处理后重复事件将被忽略。状态转换本身也是幂等的：通过 CAS 操作作用于 (intent_id, from_state)。即使去重被绕过，第二次转换也会因状态已推进而失败。

面试官： 退款如何处理？

候选人： 商户调用 /refund 对已授权或已结算的意图发起退款。将状态转换为 refunded （或 partial_refunded）。分类账条目逆转原始操作：贷记客户资金账户，借记商户待结算或可用账户。我们携带新幂等键调用提供商的退款 API。确认后，退款完成。原始授权记录保留在分类账中，退款作为引用原始意图的新交易存在。

面试官： 如何防止重复扣费？

候选人： 每一层都实施幂等性。客户端 → Payment API 使用 Idempotency-Key 请求头；幂等存储缓存响应。Payment API → 提供商使用提供商特定的幂等机制。意图状态转换通过 CAS 操作作用于当前状态。因此，重试 API 调用将返回相同响应而不重新扣费；重试提供商调用落在同一笔扣款 id 上；重试状态转换因状态已推进而失败。没有任何路径会为同一意图生成两条分类账条目。

面试官： 拒付如何处理？

候选人： 客户通过银行对某笔扣款提出异议，银行向卡网络发起拒付请求，我们收到 Webhook 通知。意图从 settled 与 disputed，随后若败诉则转换为 charged_back 若胜诉则回到 settled 若胜诉则分类账条目逆转原始扣款并增加拒付手续费。争议处理人员收集证据（派送确认、客户沟通记录）并通过提供商的争议 API 提交。

面试官： 多币种如何处理？

候选人： 每条分类账条目都有币种标识。我们绝不在同一账户中混合币种，而是将 FX 兑换作为显式交易腿。USD 商户向 EUR 客户完成授权：借记 customer_funding_EUR €100，贷记 fx_holding_EUR €100，然后在配对交易中：借记 fx_holding_USD $108，贷记 merchant_pending_USD $108。FX 汇率在授权时从 FX 服务锁定。按币种分别对账。

面试官： 如何实现高可用？

候选人： API 层无状态，部署于多个地区。数据库采用多地区部署，商户归属地区为主节点。跨地区只读副本用于报表。提供商集成本质上是外部依赖，熔断器和重试机制应对提供商抖动。Webhook 可落地于任意地区，路由至商户归属地区处理。

面试官： PCI 合规如何保障？

候选人： 我们从不存储原始卡片数据。卡片保险库是独立隔离系统，处于 PCI-DSS 范围内。客户对卡片进行令牌化，我们仅存储令牌。我们的广泛系统处于 PCI-DSS 范围之外或受限范围。SCA（PSD2）合规通过在需要时触发提供商的 3DS 流程并将意图转换为相应状态来实现。 requires_action.

面试官： 总结。

候选人： 支付意图状态机与显式转换；以复式记账分类账作为资金真相来源；每一层实施幂等性；提供商超时的一等公民未知状态通过对账解决；Webhook 去重；每日结算对账。多币种使用按币种独立账户加显式 FX 交易腿。合规通过卡片保险库和提供商原生 SCA 实现。最难的正确性问题是超时，安全网是每日对账。

3. 练习备注

五分钟版本

  1. 明确：意图、授权、退款、拒付；幂等性；合规性。
  2. 不变量：每个意图唯一的终态；平衡的复式记账分类账。
  3. 状态机：created → authorized → captured → settled，带 void/refund/dispute 分支。
  4. 端到端幂等性：client → API → provider → 状态 CAS。
  5. 复式记账分类账；按币种独立账户。
  6. 超时的未知状态；对账工作进程。
  7. Webhook 去重；CAS 状态转换。
  8. 每日与提供商结算文件进行对账。

一分钟结尾陈述

支付系统是运行在复式记账分类账之上的状态机。每次资金流动都是由显式状态转换触发的两条平衡分类账条目。端到端幂等性使重试安全；提供商超时的一等公民 unknown 状态通过对账而非猜测来解决。Webhook 基于事件 id 去重；每日结算对账捕获事件流水线遗漏的一切。多币种使用按币种独立账户加显式 FX 交易腿。最难的正确性问题是超时。

最佳面试用语

  • "资金永远不会隐式流动；每条分类账条目都对应一个带幂等键的状态转换。"
  • "超时是一等公民状态，而非重试决策。"
  • "复式记账分类账意味着试算平衡是每日的健全性检查。"
  • "与提供商结算文件的对账是我们事件流水线可能遗漏的一切的安全网。"
  • "三层幂等性：客户端、提供商、状态机。"
