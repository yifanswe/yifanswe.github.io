通知系统接收一个领域事件（"Alice 回复了你的评论"、"你的 $43.21 付款成功"、"PagerDuty 事件 #4421 已升级"、"你的 Stripe 付款已到账"），并将其转化为一条或多条异步的、经渠道路由的、面向特定用户的触达消息。它是一类基础设施，位于事件生产者（业务中的任意服务）与外部投递端点（APNs（苹果推送通知服务）、FCM（Firebase 云消息）、SES（简单邮件服务）、SendGrid、Twilio、Web Push、应用内收件箱）之间。

本文是对这类系统的参考资料——它是什么、它固有地保证什么、扇出模式的设计空间、各渠道的字节级协议机制、疑难问题、故障模式，以及同一技术如何服务于交易支付、社交信息流、营销活动、告警、2FA（双因素认证）短信和游戏内推送等不同场景。多个业务域使用方式各异，本文全面覆盖。

§1. 通知系统是什么（以及它不是什么）

通知系统是一个 异步扇出投递平台 它从上游服务接收一个逻辑事件，并生成 N 条经异构渠道（推送、邮件、短信、应用内收件箱、Web Push、语音）路由的目标消息。它解决了四个任何单一渠道 API 都无法解决的问题：

  1. 定向：将事件映射到接收者集合（单个用户、扇出列表、用户分群）。
  2. 路由：根据偏好、设备可用性、紧急程度和成本，为每位接收者选择渠道。
  3. 限流 / 去重 / 定时：用户粒度的配额、重复消息抑制、免打扰时段。
  4. 外部渠道适配：HTTP/2 对接 APNs，SMTP/HTTPS 对接邮件，REST 对接短信提供商——管理连接池、重试和背压。

它位于 事件生产者 （信息流、支付、身份、监控、营销自动化）和 投递端点 （APNs、FCM、SES、Twilio）之间，并管理一个 应用内收件箱存储.

与相邻技术类别的区分

  • vs. 分布式消息队列 （Kafka、RabbitMQ）：队列移动字节，没有"用户"的概念。通知系统在内部使用队列，但其抽象是"用户 × 渠道 × 内容"。你在队列之上构建通知系统，而不是将其作为队列来使用。
  • vs. 应用内实时推送 （WebSocket、SSE、XMPP）：实时渠道仅向当前在线的客户端投递。完整的平台同时使用两者——客户端在线时用 WebSocket，离线时用 APNs/FCM。WebSocket 只是众多渠道之一。
  • vs. 事务性邮件服务 （SES、SendGrid 单独使用）：SES 只会发一封邮件。通知平台负责编排 SES + APNs + Twilio + 收件箱 + 偏好 + 去重。SES 是原语；平台负责组合。
  • vs. 营销自动化 （Mailchimp、Iterable、Braze）：营销平台在同一扇出-路由-限流底层能力之上叠加了活动分群和 A/B 测试。
  • vs. 告警 （PagerDuty、Opsgenie）：告警是专为高紧急度、值班轮换和升级策略定制的通知平台。底层原语相同。

通知系统不擅长什么

  • 同步的带内确认。渠道 ACK 的含义是"我们已接受"，而非"用户已看到"。
  • 跨所有渠道的严格精确一次投递。推送是尽力而为的；邮件可能进垃圾箱；短信提供商有自己的重试机制。平台保证至少一次投递；设备或收件箱负责去重。
  • 跨渠道排序。即使按顺序投递，推送可能比邮件先到——APNs 和 SES 之间没有全局时钟。
  • 长篇富文本内容。APNs 将 payload 上限限制为 4KB；短信每段 160 个字符。发送链接，将内容托管在别处。
  • 双向对话。通知是单向的。请使用消息系统处理回复。

§2. 固有保证（以及哪些需要额外分层实现）

平台在设计上提供：

  • 每个渠道的至少一次投递，其中"投递"指"已被渠道 API 接受"——而非"已在屏幕上渲染"。APNs 返回 200 意味着 APNs 已接受；设备可能仍处于离线状态。
  • 持久化收件箱 作为唯一真相来源。应用内收件箱行一旦写入，即可在崩溃后存活。推送和邮件只是辅助提示。
  • 基于事件/投递键的幂等性。相同的 event_id + recipient_id 可在重试时也只解析为一条收件箱记录。
  • 特定渠道的可靠性:
  • 推送（APNs/FCM）：尽力而为。Apple 对未投递的推送在 iOS 上保留约 30 天，Android 上保留数小时，之后丢弃。 apns-collapse-id 保证即使发送了十次推送，也只出现一条横幅通知。
  • 邮件（SES/SendGrid）：对于信誉良好的发送者，投递率约 99%；延迟从数秒到数分钟不等；内容可能进入垃圾箱。
  • 短信（Twilio）：对运营商（MNOs）的投递率约 95-99%；运营商的队列不透明。
  • 站内信收件箱：100%（我们已持久化写入）。
  • 背压：队列吸收生产者的流量突发；分发器的速率与之解耦。

平台在设计上不提供：

  • 跨渠道的精确一次投递。分发器与 APNs 之间的网络抖动会导致重复推送；设备侧的 collapse_id 会合并它们。这是分层实现的，并非固有能力。
  • 严格的跨渠道排序。"推送先于邮件到达"是惯例。分发器并行扇出；APNs 约 200ms，邮件约 5s。只有收件箱（按事件时间戳排序）才是有序的来源。
  • 向生产者实时确认投递。生产者发布后即完成。用户是否看到它 = 异步回调（Twilio webhook、SES SNS bounce），可能在数小时后才收到。
  • 垃圾邮件合规。CAN-SPAM、GDPR、CCPA 要求处理退订和跟踪同意。平台暴露机制；业务方负责制定策略。

必须由系统设计者额外构建:

  • 生产者侧幂等键 ——若支付服务重试"支付成功"，我们将收到两个事件。在 Redis 中对生产者生成的 event_id 进行去重是强制要求。
  • 跨渠道协调 ——"先推送，若 5 分钟内无 ACK 则发邮件"——通过延迟队列构建。
  • 合规审计 ——"GDPR 数据导出通知是否于 2025-08-13 发出？"——必须支持 90 天以上的可查询性。

§3. 设计空间：扇出模式

通知系统中最主要的变量维度是 如何计算和物化接收者集合。三种模式，各有明显的权衡。

写时扇出（推模型）

当生产者发出事件时，系统 立即向每位接收者的收件箱写入一条记录。读取极为简单——用户打开应用，从 SELECT * FROM inbox WHERE user_id=X LIMIT 50 自己的分区获取。

  • 写入成本：O(F)，其中 F = 接收者数量。对于 1:1 事务事件（收款通知），F=1 → 成本极低。对于 1:N 社交事件（你的朋友发帖了），F = 关注者数量。对于名人账号，F 可达 1 亿。
  • 读取成本：O(1) 页面加载——一次分区查找。
  • 存储：O(事件数 × 平均关注者数)——每位接收者各存一份副本。
  • 适用场景：1:1 事务消息（收款通知、密码重置、告警）、小扇出社交场景。
  • 不适用场景：高扇出社交场景（名人发帖）、病毒式内容。

读时扇出（拉模型）

生产者仅向 生产者自己的时间线/事件日志写入。读取时，接收者的客户端/服务端查询所有相关生产者的时间线并合并。

  • 写入成本：每个事件 O(1)。
  • 读取成本：O(F')，其中 F' = 接收者订阅的生产者数量。对于关注了 300 人的普通用户，每次打开收件箱都会触发 300 次分区读取。
  • 存储：O(事件数)——单份副本。
  • 适用场景：极高扇出来源（名人时间线、RSS 式信息流）、写密集型工作负载。
  • 不适用场景：读密集型工作负载、关注了大量来源的接收者、对延迟敏感的 UI。

混合模式（非平凡系统的标准选择）

按来源选择策略。低于 名人阈值 （约 1 万–10 万关注者），使用写时扇出。超过阈值，使用读时扇出——不预填充收件箱；读取时从名人的近期时间线拉取并合并。

if poster.follower_count < THRESHOLD:
    fanout_on_write(poster, followers)
else:
    fanout_on_read_only(poster)

# Reader path:
inbox  = pre-pushed inbox (top N)
celebs = user.celebrity_followees  # bounded, ~10-30
fresh  = [read recent_timeline(c, since=24h) for c in celebs]
return merge(inbox, *fresh).sorted_by_ts().take(50)

Twitter 在 2010–2012 年因名人写入风暴引发级联故障后转向此方案。Facebook、Instagram、LinkedIn 均使用其变体。

对于事务型通知（支付、安全、告警），该问题并不存在——fanout 为 1，因此写时 fanout 显然是正确的。混合策略的问题仅对社交 fanout 才有意义。

§4. 字节级原理

这里是浅层文档的终点，也是真正深度的起点。通知平台有四个关键存储/传输结构： inbox store （Cassandra 中的 LSM tree）， device-token store （MySQL 中的 B+ tree）， dedup/rate-limit store （Redis 哈希表），以及 APNs/FCM wire protocol （带 HPACK 帧的 HTTP/2）。我逐一介绍。

4.1. Inbox store：Cassandra LSM tree，按 user_id 分区

访问模式：\n- 写入密集：每条通知 = ≥1 次 inbox 写入。每天 5B 条 = 持续约 58k 次写入/秒，病毒式爆发峰值约 1.75M 次/秒。\n- 读取模式："用户 X 的最近 50 条通知"——在单个分区内进行范围扫描。\n- 跨用户查询：几乎不存在。99.9% 的 inbox 查询范围限于单个 user_id.

LSM（Log-Structured Merge）tree 是教科书级的选择。B+ tree（MySQL/InnoDB）在 1B 用户、58k 次写入/秒的场景下会遭遇页分裂风暴；随机写入模式会频繁抖动 buffer pool。LSM 以约 10 倍写放大（压缩开销）换取纯顺序写入，每节点每 SSD 可达约 100k 次写入/秒。

Schema

CREATE TABLE notifications_inbox (
  user_id          bigint,
  created_at_ms    bigint,
  notification_id  uuid,
  type             text,
  payload          blob,    -- protobuf-encoded
  read_at_ms       bigint,
  PRIMARY KEY ((user_id), created_at_ms, notification_id)
) WITH CLUSTERING ORDER BY (created_at_ms DESC, notification_id ASC)
  AND default_time_to_live = 2592000;  -- 30 days
  • (user_id) 是分区键 → 某用户的所有通知共置于相同的 RF=3（复制因子）副本上。
  • created_at_ms DESC clustering → 最新优先；"最近 50 条"读取该分区的前 50 个 cell。
  • default_time_to_live = 30d → Cassandra 在过期时写入墓碑（tombstone）；旧数据由压缩（compaction）回收。

LSM 磁盘布局

每个 Cassandra 节点将数据存储为 磁盘上不可变的 SSTable（Sorted String Table） 以及 可变的内存 memtable.

                         WRITE PATH
                         ─────────
  client write
      │
      ▼
  commit log (append-only, fsync periodic 10ms or per-batch)  ── DURABILITY POINT
      │
      ▼
  memtable  (skip-list or trie-map in heap, sorted by clustering key)
      │
      │  when memtable hits flush threshold (~256 MB or 15 min)
      ▼
  flush to disk as a new SSTable (immutable, sorted)
      │
      ▼
  background compaction merges SSTables (size-tiered or leveled)

                         READ PATH
                         ─────────
  read for (user_id=12345, last 50)
      │
      ▼
  1. memtable lookup
  2. bloom filter check on each SSTable for partition (user_id=12345)
  3. for SSTables that pass: read partition index → seek to offset
  4. merge results in clustering-key order, take top 50
  5. apply tombstones (expired TTLs)

SSTable 文件布局

┌─────────────────────────────────────────────────────────────────┐
│  Data.db        (rows, sorted by partition_key then clustering) │
│  Index.db       (partition-key → byte offset into Data.db)      │
│  Filter.db      (bloom filter for partition keys, ~1% FP)       │
│  Summary.db     (sampled Index.db entries, kept in heap)        │
│  Statistics.db  (min/max clustering keys, tombstone counts)     │
│  CompressionInfo.db                                              │
└─────────────────────────────────────────────────────────────────┘

对于 SELECT * FROM notifications_inbox WHERE user_id=X LIMIT 50：\n1. Hash user_id → 通过一致性哈希 token ring 识别归属副本。\n2. 在副本上，Filter.db 中的布隆过滤器返回"是，该 SSTable 可能包含 user_id=X"——假阳性率约 1%。\n3. Index.db（或 Summary.db，驻内存，采样）给出 Data.db 中的字节偏移量。\n4. 定位到 Data.db，按 clustering 顺序向前扫描，取 50 个 cell。

每个 SSTable 最坏情况两次寻址。若用户 X 的数据跨越 4 个 SSTable（部分行最近刚刷入，其他来自更早的压缩），则需访问 4 个 SSTable 并在内存中合并。 Leveled compaction 将数量维持在较低水平（任意一行约 5 个 SSTable/层）； size-tiered compaction 数量更高，是 Cassandra 未经调优时的默认策略。

与 B+ tree 的权衡对比

维度 LSM（Cassandra） B+ tree（MySQL）
写吞吐量 ~100k 次/秒/节点（顺序写） ~20k 次/秒/节点（随机分裂）
读延迟 p99 ~5-10ms（多 SSTable 合并） ~1-2ms（单树遍历）
压缩开销 ~10 倍写放大 ~2-3 倍（分裂 + buffer pool）
分区内范围扫描 快速 快速（叶链）
跨分区查询 极差（无全局索引） 可通过二级索引支持
TTL 过期 原生支持 手动清理任务

对于写入率高、读取范围限于分区的 1B 用户 inbox，LSM 胜出。跨用户分析（"帖子 X 的所有用户通知"）= 添加 Pinot/Druid 作为辅助存储。

持久性与恢复

提交日志是持久性的保障点。Cassandra 在更新 memtable 之前先追加写入提交日志。崩溃后：重启 → 回放提交日志 → 提供读取服务。Fsync 为 周期性 （默认 10ms）以保证吞吐量，或 批次 （每次写入）以获得更严格的持久性——将事务型通知单独放入一个 keyspace，并启用 batch.

4.2. Device-token store：MySQL B+ tree，按 user_id 分片

CREATE TABLE device_tokens (
  user_id        BIGINT      NOT NULL,
  device_id      VARCHAR(64) NOT NULL,
  platform       ENUM('IOS','ANDROID','WEB') NOT NULL,
  token          VARCHAR(256) NOT NULL,
  app_version    VARCHAR(32),
  locale         VARCHAR(16),
  last_seen_ms   BIGINT,
  active         BOOLEAN     NOT NULL DEFAULT TRUE,
  PRIMARY KEY (user_id, device_id),
  INDEX idx_token (token(64))
) ENGINE=InnoDB;

为什么选 B+ tree：\n- 点查询或窄范围查询（每用户 ≤10 行）。B+ tree 分支因子约 100 → O(log_100 N) 查询。对于 10 亿行，约 5 层深；上层 3 层驻 InnoDB buffer pool → p99 仅需 1 次磁盘寻址。\n- 变更可预测：token 轮换 = 对 (user_id, device_id)执行 UPSERT；无数据抖动。\n- 按 user_id 分片到 32 个 MySQL 集群 → 每个分片约 40 GB。每个分片 = 1 主节点 + 2 副本。运维简单。

Token 轮换反馈循环（§8 中描述的 410/404 路径）将回写该表以停用失效 token。

4.3. Redis：dedup、rate-limit、preference cache

Dedup: SETNX dedup:{event_id}:{user_id} 1 EX 86400。SETNX 返回 0 → 已分发 → 丢弃。每秒 58k 事件 × 约 50 次投递/事件 ≈ 290 万次操作/秒 → 64 节点 Redis 集群分摊约 45k 次/秒/节点（远低于约 10 万次操作/秒/核的上限）。约 2.5 亿活跃键 × 50 字节 ≈ 12 GB，完全可以承载。

Rate limit（滑动令牌桶）: ratelimit:push:{user_id} → { tokens, last_refill_ms }。Lua 脚本原子性地按经过时间补充令牌，减 1，返回是否允许。各渠道的桶相互独立（push 5 次/分钟，email 3 次/小时，SMS 1 次/小时）。

Preference cache: prefs:{user_id} → { push, email, sms, quiet_hours, tz, muted_types }。读穿透，TTL=60s。Unsubscribe API 同时写入 MySQL 和 DEL prefs:{user_id} → 下次读取时数据即为最新。

4.4. APNs/FCM wire protocol：带 HPACK 的 HTTP/2 与持久连接

深度信号：你并非只是"向 APNs 发 POST 请求"——你需要维护 持久的 HTTP/2 连接，通过每条连接多路复用数千个并发推送，管理 HPACK 头部压缩，并处理 GOAWAY 以实现连接的优雅生命周期管理。这正是 APNs 和 FCM 通过摊销 TLS 建立开销来大幅提升吞吐量的地方。

APNs over HTTP/2

TCP connection                                    [persistent, keep-alive]
  └─ TLS 1.3 handshake (or session resumption)   [client cert or JWT auth]
     └─ HTTP/2 connection
        ├─ Stream 1: POST /3/device/{token1}    payload1
        ├─ Stream 3: POST /3/device/{token2}    payload2
        ├─ Stream 5: POST /3/device/{token3}    payload3
        ├─ ... (up to ~1000 concurrent streams per connection)
        └─ Stream 2001: POST /3/device/{tokenN} payloadN

  Responses arrive interleaved on the same connection:
        Stream 1:    200 OK
        Stream 3:    410 Unregistered      ← device gone, mark token dead
        Stream 5:    429 Too Many Requests ← back off
        Stream 2001: 200 OK

帧级机制：

  • HEADERS 帧：HPACK 压缩。在同一连接上发出第一个请求后，重复出现的头部（如 apns-topic: com.foo.app, apns-push-type: alert, authorization: bearer <JWT> 每次仅需约 5 字节（HPACK 动态表索引，而非完整字符串）。在大规模场景下这很重要：10k 次推送 × 节省 200 字节/头部 = 每秒每连接节省 2 MB。
  • DATA 帧：JSON payload（普通推送 ≤4KB，voip-pushtype 为 5KB）。
  • WINDOW_UPDATE 帧：HTTP/2 流量控制。APNs 公告一个窗口（例如初始 64KB）；如果你的在途数据超过该窗口，则需等待 WINDOW_UPDATE。具有背压感知的客户端会遵守此机制；激进的客户端则会导致 TCP 层面的停顿。
  • GOAWAY 帧：APNs 在需要优雅关闭连接时发送此帧——通常在 Apple 边缘节点部署期间，或连接存活时间"过长"时。你的客户端必须耗尽在途流并在新连接上重连。未处理 GOAWAY 是一个常见 bug——推送会无声失败或挂起。
  • PING 帧：HTTP/2 存活检查。客户端周期性发送；未收到 PONG → 连接断开 → 重连。

吞吐量计算：持久连接的重要性

Cold connection (per-push reconnect):
  TLS handshake = ~50-100ms
  One push per handshake → ~10 pushes/sec/conn

Persistent connection with multiplexing:
  ~1000 concurrent streams per HTTP/2 connection
  Each stream completes in ~80ms (Apple round-trip)
  Throughput per connection ≈ 1000 streams / 0.08s = ~12.5k pushes/sec

  Sustained throughput limited by APNs server-side ≈ 1000-2000 pushes/sec/conn

病毒式爆发时 175 万次推送/秒：

1.75M/sec ÷ 1000/sec/conn = 1,750 persistent APNs connections

分布在约 50 台分发主机上，每台持有 30-40 条连接。Apple 每个 provider 证书最多允许约 100 条连接；大客户（Twitter、LinkedIn、Meta）可协商更高上限。

FCM 特性

FCM（Firebase Cloud Messaging）同样使用 HTTP/2，但提供了显式的 批量端点:

POST https://fcm.googleapis.com/v1/projects/{pid}/messages:send  (single)
POST https://fcm.googleapis.com/batch                            (batch, up to 500)

批量端点接受一个包含最多 500 条独立消息的 multipart 请求体，通过单次 HTTPS 往返返回一个响应数组。APNs 没有等效的批量端点——你需要依赖 HTTP/2 流多路复用。

应用层批处理技巧：即使在 APNs 上，也应将推送分组，在约 10ms 刷新窗口内合并约 500 个流。这样可以摊销加密开销并提升 HPACK 复用率。

§5. 容量规模：从小到大

相同架构可跨越 6+ 个数量级，具体取决于规模。底层结构（队列 → fanout → dispatcher → 渠道路由）相同，只是规模不同。

变化内容 吞吐量 占用规模 真实案例

  • 极小/小型 → 瓶颈是 生产者速率.
  • 中型 → 瓶颈是 单一 MySQL inbox；迁移到 Cassandra。
  • 大型 → 瓶颈是 名人写入风暴；迁移到混合写时/读时 fanout。
  • 超大型 → 瓶颈是 持久连接管理 （数百万 socket）；在 Erlang/Go 中构建专用 socket 层。

§7. 该技术固有的难题

问题 1：名人 fanout（单一来源，1 亿接收者）

出现于 社交 （名人发帖）、 营销 （广播给所有用户）、 告警 （向所有值班人员发送事故通知）、 事务型 （向某地区所有用户发送监管通知）。

简单粗暴的解法：写时 fanout 到所有 1 亿用户。同步 = 帖子挂起数小时。通过 Kafka 异步 = 队列积压，普通用户等待。

为什么行不通：来源在 T=0 时发布；fanout 将 1 亿条投递加入队列。全系统每秒 100 万次投递 = 100 秒的纯来源负载。Dispatcher 池容量为 20 万次/秒 = 500 秒（约 8 分钟）。普通用户的事件被同一 Kafka 分区（若分区策略不佳）或 dispatcher 容量阻塞。

真正的解决方案：混合写时 fanout + 读时 fanout，并设定来源阈值（见 §3）。当 source.recipient_count > 10k时，跳过 inbox 预填充；接收者在读取时查询来源的近期时间线并合并。

对于推送通知， 采样 ——参与度评分将接收者降至实际与该来源互动的用户（总数的 1%-10%）。每用户通知预算（每天 ≤5 次推送）进一步削减。净效果：名人帖子推送给约 100 万人，而非 1 亿人。

对于营销广播， 分时段分片 ——通过延迟队列在 30 分钟内逐步发送 1 亿封邮件。保护 SES/SendGrid 配额；避免下游 IMAP 服务器不堪重负。

问题 2：精确一次 vs 至少一次

出现于 支付 （不重复发送收据）、 2FA （不发送两个验证码）、 告警 （不重复呼叫值班人员）、 社交 （不重复通知"Alice 已回复"）。

简单粗暴的解法：在 dispatcher 和 APNs 之间使用分布式锁或 2PC（两阶段提交）。

为什么行不通：APNs 是外部系统，无法参与 2PC。APNs 返回 200 → 也许已投递，也许 APNs 崩溃了。即使 APNs 在 T=80ms 时确认，dispatcher 也可能在 T=81ms 记录 DELIVERED 之前崩溃——重启时从 Kafka offset 重新处理，再次分发，两次推送。

真正的解决方案：至少一次投递，在 设备层实现幂等性。\n- 设备侧去重通过 apns-collapse-id / FCM collapse_key：相同 collapse ID 的推送会合并；用户只看到一条横幅。\n- Inbox 幂等性通过 event_id （Cassandra UPSERT）：无论重试多少次，行只写入一次。\n- 对于事务型事件（密码重置、支付收据），生产者附加 idempotency_key；相同键在 24 小时内返回相同的投递结果。Stripe 的收据流水线即采用此方式。

对于大多数非事务型通知，至少一次投递 + 设备侧去重是正确做法；精确一次是错误的目标。

问题 3：跨渠道顺序

出现于 chat (1→3→7 未读；用户必须看到"7"，而非"1"), 告警 (告警从打开→已解决；用户不能先看到"已解决"), 电子商务 (已下单→已发货→已送达)。

简单粗暴的解法：依赖 Kafka 分区排序，按 user_id.

为什么行不通：T=10 的"1 条新消息"和 T=12 的"3 条新消息"都在同一分区中。Fanout 按顺序读取；dispatcher 也按顺序读取。但 dispatcher 将每条消息拆分为 push/邮件/站内信。Push #1 在连接 A 上排在 50 条消息之后；push #2 落到空闲的连接 B 上。手机在 T=15 收到"3"，在 T=17 收到"1"。

真正的解决方案：1. 按(user_id, channel) dispatcher 内部针对每个分片进行确定性路由，使用户 42 的所有 push 都经过同一 APNs 路由器——每用户 FIFO。2. apns-collapse-id / collapse_key：设备端以新状态覆盖旧状态。线序错误 → 设备仍显示最新状态。3. Inbox Cassandra 按 created_at_ms 保证无论分发并行度如何都按时间顺序排列，前提是每行携带 事件的 时间戳，而非 dispatcher 的写入时间。

应用层按用户串行化 + 设备端 collapse_id + 存储事件时间排序 = 各通道内有效有序。

问题 4：多层限流与垃圾消息防护

出现于 营销 （活动配额）， 事务型 （50 次重试不要连发 50 条回执）， 告警 （告警抖动 → 抑制）， 社交 （批量标签垃圾消息）。

简单粗暴的解法：信任生产者自行限流。

为什么行不通：生产者 bug 在循环中重试，为同一用户发送 10k 条事件，每条产生约 5 条通知。用户被骚扰，卸载应用。

真正的解决方案：在 dispatcher 层实施多层限流。

Layer 1: per-user, per-channel:   5 push/min, 10 push/hr
Layer 2: per-user, per-type:      3 NEW_FOLLOWER/day
Layer 3: per-user global cap:     20 notif/day across channels
Layer 4: per-campaign kill switch (operator-controlled)
Layer 5: provider circuit breaker (APNs/FCM err > 5% → open)

每层 = Redis 令牌桶；Lua 脚本原子性地执行检查并递减。拒绝时记录 RATE_LIMITED_LAYER_2 于 delivery_status.

更深层反垃圾：滚动 通知质量分 ——高打开率类型计费更低；低互动类型节流更严。来自 feedback.events → 偏好评分。

问题 5：时区感知的免打扰时段

出现于 营销 （不要在本地时间凌晨 3 点发邮件）， 社交 （"点赞了你的帖子"延迟发送，而聊天消息始终允许）， 告警 （低优先级告警延至值班人员的免打扰时段结束后发送）。

简单粗暴的解法：以 UTC 偏移存储免打扰时段；检查 now_utc.

为什么行不通：用户在东京（UTC+9），免打扰时段为"本地 22:00-08:00"。以 UTC 存储 = "13:00-23:00"。用户飞往纽约（UTC-5）；本地时间相同，但 UTC 变为"03:00-13:00"。UTC 表示法需在出行时更新。

真正的解决方案：- 以 IANA 名称存储时区（Asia/Tokyo）。App SDK 在设备时区变更时刷新。- 以本地时间存储免打扰时段（{start_local: "22:00", end_local: "08:00"}）。- Dispatcher 计算 now_in_user_tz。若在窗口内：通过延迟队列（Redis ZSET，调度器轮询到期项并重新入队）调度至窗口结束时发送。

夏令时（DST）由 IANA TZ 数据库处理。边缘情况：跨时区旅行者可能短暂出现不一致；可接受此情况。

问题 6：退订传播延迟

出现于 营销 （GDPR/CAN-SPAM 法律违规）， 事务型 （用户取消回执订阅）， 社交 （用户静音会话）。

简单粗暴的解法：写入 MySQL，等待缓存 TTL（60s）追上。

为什么行不通：用户在 T=0 退订。通知在 T=10 触发；dispatcher 读取过期缓存并发送。对于受监管通道，属于法律违规。

真正的解决方案：1. 退订 API 同时写入 MySQL 和 DEL prefs:{user_id} 在 Redis 中。下次读取即为最新数据。2. 发布 unsubscribe.events 到 Kafka。Fanout 侧消费者更新内存中的布隆过滤器；系统停止为该用户创建投递任务。3. 对于 deliveries.queued中正在处理的消息：dispatcher 在分发时重新检查 Redis DEL 之后的偏好设置。

综合效果：中位传播延迟 <500ms；最坏情况 <5s。

§8. 故障模式演练

8.1. Dispatcher 在处理中途崩溃

Dispatcher 从 Kafka 消费了事件，将 inbox 写入 Cassandra，将 push 入队到 APNs 路由器，然后在提交 Kafka offset 前崩溃。

恢复：- Kafka 消费者从上次提交的 offset 重启。- 重新处理事件。- Inbox INSERT：在 (user_id, event_id) 上幂等 → 不会产生重复行。- Push 到 APNs：重新分发；APNs 收到重复消息； apns-collapse-id 在设备上抑制第二次横幅通知。- delivery_status UPSERT；最终状态为 DELIVERED。

持久性要点：Kafka offset。仅在 Cassandra 写入和 APNs 发送均返回成功后才提交。提交过早 = 丢失工作；提交过晚 = 重复工作（至少一次契约可处理此情况）。

8.2. APNs 节流/不可用

APNs 返回 429 / 503。若 30s 内 429/503 比例 > 5%，APNs 路由器上的断路器打开。打开状态：停止发送；HTTP/2 连接空闲但保持连接。30s 后进入半开状态：发送 10 条测试 push；若 9/10 成功 → 关闭断路器。

打开期间：push 排队在 deliveries.apns.retry。恢复后以 20% 容量排空队列并逐步提升。Inbox 已写入；用户下次打开应用时可见。若 push 是唯一通道且超过新鲜度窗口（例如聊天 >1h），在用户已订阅的情况下升级为邮件/SMS 发送。

8.3. 设备 token 轮换

用户重新安装应用 → 旧 token 失效。APNs 返回 410 Gone；FCM 返回 404 或带 NOT_REGISTERED.

Dispatcher 发布 feedback.events: { TOKEN_DEAD, ... }。反馈处理器： UPDATE device_tokens SET active=FALSE WHERE token=?。该用户的下次 push：active=TRUE 过滤器将其排除。每周清理任务硬删除超过 30 天未活跃的 token。重新安装的应用在首次启动时通过 POST /devices 注册新 token。

8.4. 邮件地址无效但 push 正常

SES 退信 → SNS webhook → feedback.events: { EMAIL_BOUNCE, hard }. UPDATE user_emails SET bounced=TRUE；偏好缓存失效；后续通知跳过邮件；push 和站内信继续发送。站内信横幅提示用户更新邮件地址。

软退信（邮箱已满）：按退避策略重试（1h、4h、24h）；3 次软退信视为硬退信。

8.5. Cassandra 节点宕机

该节点上的分区集群在有效 RF=2 的情况下继续运行。运维人员执行 nodetool removenode，添加新节点。新节点引导启动：通过 gossip 协议通信，获取 token 范围，从存活的 2 个副本流式同步数据（1TB 约需数小时）。引导期间，写入同时发往新节点和旧副本；引导完成前读取使用旧副本。

持久性要点：RF=3 配合 QUORUM。丢失 3 个节点中的 1 个 ≠ 数据丢失。

8.6. Kafka broker 宕机/主节点变更

标准 Kafka 恢复：ISR（同步副本集）中的副本被晋升；生产者重新发现主节点；消费者从上次提交的 offset 继续。生产者通过 event_id实现幂等性；主节点变更后的重试不会重复创建消息（去重层会捕获）。

8.7. 网络分区（脑裂）

Cassandra 是 AP 系统。3 副本分区分裂为 2+1；多数侧提供 QUORUM 读/写；少数侧拒绝写入。恢复后：通过 hinted handoff + read-repair 合并。

Kafka 使用 controller quorum（KRaft/ZooKeeper）。少数侧 broker 失去主节点地位；生产者重定向到多数侧。无脑裂。

少数侧的 Dispatcher pod 因本地 Kafka 失去主节点地位而停滞。由于去重机制，不会发生双重投递。

§9. 为何不在处理器中同步做 fanout？

最简单的替代方案：当生产者服务处理请求时（Alice 发帖），同一处理器在返回之前同步调用每个接收者的 APNs/SES/Twilio。

  • T=0：Alice 发送 POST 请求更新动态。
  • T=1ms：feed-service 插入帖子。解析出 5,000 个关注者，每人 3 台设备。
  • T=2ms：Handler 向 APNs 发起 15,000 条 HTTP/2 流。每条 p99 约 100ms。
  • 1000 并发情况下 → 15 批 × 100ms = 1500ms 同步等待。Alice 看到 1.5s 的等待动画，而这本应是一个 50ms 的发帖操作。若 APNs 出现故障，她的发帖请求将在 5s 后超时。

名人场景：@cristiano 拥有 1 亿关注者 → 100,000 批 × 100ms = 10,000s = 2.8 小时。发推显示"未发送"（超时）。

简单实现混淆了 写入路径 封装为 fanout 路径: - 写入路径：同步，毫秒级预算，只写入 Alice 的帖子并向 Kafka 发出一个事件。- Fanout 路径：异步，秒到分钟级预算，在后台处理 N 个接收者的扩展。

此方案同样缺少限流、去重、重试、免打扰时段、偏好检查和 inbox 持久化。一旦 fanout 脱离关键路径，所有这些功能都必然被纳入架构。同样的逻辑适用于事务型 1:1 场景：即使是支付回执也不应将 SES 延迟带入支付 API。始终通过队列将生产者与投递解耦。

§13. 真实世界实现与数据

  • Twitter 时间线 fanout：约在 2010-2012 之后从 fanout-on-write 切换至混合模式，原因是名人账号（Lady Gaga、Justin Bieber 突破 1000 万关注者+）引发写入风暴。记录于 Twitter 工程博客及《数据密集型应用系统设计》第 1 章。当前阈值据报道约为100万粉丝 超过该阈值则对收件箱采用读时扩散（fanout-on-read）。

  • Snap：峰值约每天 100 亿条 snap。管道运行在 AWS 上（DynamoDB + SQS + SNS），并实现了自定义去重和互动过滤。
  • Discord：每天超过 40 亿个事件。这是规模最大的 Cassandra 部署之一——参见"How Discord Stores Trillions of Messages"。每用户分区，配合滚动时间桶聚簇键。

§14. 邮件送达率深度解析

通过 SES（Simple Email Service）或 SendGrid 发送邮件是容易的部分。难的是：在规模化、可持续的条件下，让邮件落入收件人的 收件箱 而非垃圾箱——跨 Gmail / Outlook / Yahoo Mail / 企业邮件服务器皆然。这正是 送达率 领域的核心挑战，大多数工程师在 90% 的营销邮件进了垃圾箱之前根本意识不到这个问题。

整个体系建立在三个分层的认证标准之上：SPF、DKIM、DMARC。在此之上还有：发件人信誉（按 IP、按域名）、反馈循环和 IP 预热规程。缺少任何一项，大批量发件方都会被默认视为垃圾邮件发送者。

14.1. SPF——发件人策略框架

SPF（Sender Policy Framework，发件人策略框架）是发信域名上的一条 DNS TXT 记录，列出了哪些 IP 允许以该域名发送邮件。接收端 MTA（Mail Transfer Agent，邮件传输代理）会检查：发起连接的 SMTP 服务器 IP 是 1.2.3.4； MAIL FROM 信封发件人是 bounce@example.com；接收方查询 example.com TXT 并看到：

example.com.  IN  TXT  "v=spf1 ip4:54.240.0.0/18 include:_spf.mailprovider.com -all"

-all （硬拒绝）表示"如果连接 IP 不在此列表中，则拒绝"。 ~all （软拒绝）表示"接受但标记为可疑"。大多数大规模发件方使用 -all ，因为在 Gmail 看来，软拒绝本身就偏向垃圾邮件。

这个陷阱在于：SPF 只验证 信封 MAIL FROM （退信地址），而不验证 请求头 From: 用户看到的 From 头。垃圾邮件发送者可以让 SPF 通过 bounce.spammer.com ，同时在可见头部显示 From: ceo@yourbank.com ——SPF 对此无能为力。这就是为什么仅靠 SPF 不够，还需要叠加 DKIM 和 DMARC。

14.2. DKIM——域名密钥识别邮件

DKIM（DomainKeys Identified Mail，域名密钥识别邮件）使用私钥对邮件 正文 + 选定头部 进行签名。收件方从 DNS 获取公钥（位于 selector._domainkey.example.com ），并验证签名。若正文或已签名头部在传输中被修改，签名验证将失败。

线路机制：发出邮件的 MTA 插入一个 DKIM-Signature: 头部：

DKIM-Signature: v=1; a=rsa-sha256; c=relaxed/relaxed;
    d=example.com; s=s2025;
    h=from:to:subject:date:message-id;
    bh=MTIzNDU2Nzg5MGFiY2RlZg==;
    b=Dw1z8e7K6L3M2N1O5P4Q9R8S7T6U5V4W3X2Y1Z0A9B8C7D6E5F4G3H2I1J0...
  • d= 签名域名
  • s= 选择器——允许轮换密钥（今年用s2025 ，明年用 s2026 ），而不会导致旧邮件失效
  • h= 覆盖的头部字段
  • bh= 正文哈希
  • b= 签名值

验证方查询 s2025._domainkey.example.com TXT → RSA 公钥（或更新配置下的 Ed25519）。RFC 8463 允许使用 Ed25519-SHA256。

陷阱：宽松规范化（c=relaxed/relaxed）是默认设置，因为邮件服务器会修改空白字符； simple/simple 在大多数中继下会导致验签失败。正文哈希在 Content-Transfer-Encoding解码后的正文上计算，并进行行尾规范化；弄错这一点的结果是"我看不出问题，但 Gmail 拒绝每一封邮件"。

密钥轮换：每 6-12 个月进行一次。生成新密钥对、新选择器，发布公钥，切换签名，在无签名邮件满 1 个月后停用旧选择器。部分合规制度要求按季度轮换。

14.3. DMARC——基于域名的邮件认证、报告与一致性

DMARC（Domain-based Message Authentication, Reporting, and Conformance，基于域名的邮件认证、报告与一致性）将 SPF 和 DKIM 绑定在一起，并增加了 策略 + 报告 层。接收方检查：SPF 是否通过，且 SPF 域名是否与可见的 From: 域名"对齐"？DKIM 是否通过，签名 d= 是否与 From:对齐？只要其中一项通过认证且对齐，DMARC 即通过。若两项均失败，则执行策略。

DNS 记录：

_dmarc.example.com.  IN  TXT  "v=DMARC1; p=reject; rua=mailto:dmarc-agg@example.com;
                               ruf=mailto:dmarc-forensic@example.com; pct=100; aspf=s; adkim=s"
  • p=none （仅监控——从这里开始）
  • p=quarantine （投入垃圾箱）
  • p=reject （完全丢弃）
  • rua= 汇总报告（每日 XML 摘要，记录谁尝试以你的域名发信）
  • ruf= 每封邮件的取证报告（较少使用，涉及隐私）
  • aspf=s, adkim=s 严格对齐—— subdomain.example.com 不满足 example.com的对齐要求。大多数发件方使用宽松对齐 r.

落地步骤：先设置 p=none 运行 30-60 天；分析 rua 报告，找出被遗忘的合法发件方（营销工具、HR 供应商、财务 ERP）；将其加入 SPF / DKIM；收紧至 p=quarantine；最终切换至 p=reject。部分财富 500 强企业花费 18 个月才能达到 p=reject ，原因是存在大量以其域名发信的"影子 IT"。

灾难场景：某金融公司在 p=none 状态下通过 DMARC 报告发现，每天有 4 万封仿冒其品牌的钓鱼邮件被发出。他们一夜之间收紧至 p=reject ——却发现 HR 供应商发送密码重置邮件时从未完成 SPF 对齐。密码重置邮件开始退信。修复方案： p=reject 需要完整清点所有合法发件方 第一个.

14.4. 发件人信誉——按 IP 和按域名

Gmail、Outlook、Yahoo 各自为每个发送 IP 和每个发送域名维护内部信誉分数。输入信号包括：退信率（24 小时内 >5% 为差）、垃圾邮件投诉率（>0.1% 为差；即用户点击"举报垃圾邮件"按钮）、垃圾邮件陷阱命中（向非真实用户的地址发信——反垃圾邮件厂商用此捕获邮件列表购买者）、互动信号（打开、回复、"不是垃圾邮件"点击）、发送量一致性（新 IP 突然 10 倍量级增长 = 可疑）。

Gmail 的 Postmaster Tools 以 High / Medium / Low / Bad形式展示每域名的信誉。微软的 SNDS （Smart Network Data Services）提供每 IP 信誉数据。Yahoo 提供有限但可用的反馈循环。

信誉差 = 静默限速。你发出 100 万封邮件，MTA 对每封都返回 250 OK，但实际只有 20 万封进入收件箱。其余 80 万封进入垃圾箱，或者被 Gmail 以 10% 的速率"送达率限速"。没有任何报错——只有较慢的投递速度和糟糕的打开率。

14.5. 新发送 IP 的预热

一个毫无历史记录的全新 IP 在第 1 天就发送 100 万封邮件，会被所有接收方视为垃圾邮件攻击。标准预热规程如下：

Day 1:    100 emails
Day 2:    200
Day 3:    500
Day 4:    1,000
Day 5:    2,000
Week 2:   10,000-20,000/day
Week 3:   50,000-100,000/day
Week 4:   200,000-500,000/day
Week 5+:  ramp to target volume; multiple IPs in pool

预热期间，只向 高度活跃用户 （近期打开过邮件的用户）发送，以确保互动信号良好。将事务性邮件（高互动）与营销邮件混合发送。每天监控 Postmaster Tools。信誉下滑则暂停发送。

"从冷 IP 发送 100 万封邮件，90% 进了垃圾箱"的灾难案例：某初创公司在一个全新的 Mailgun IP 上启动营销活动，对所有持有地址的 100 万用户（包括多年未活跃的地址）进行群发。结果：8% 退信率，2% 投诉率，IP 被 Spamhaus 列入黑名单，所有后续邮件被封堵。恢复过程：数周的补救工作，往往无济于事——只能放弃该 IP 重新开始。

成熟的发送平台（SES、SendGrid、Mailgun）维护着已预热的 IP 池，并将新租户轮询分配到这些 IP 上，但如果你预订了专用 IP，预热工作就由你自己负责。

14.6. 反馈循环与退信处理

  • 硬退信：永久性失败（550 5.1.1 User unknown）。立即禁止发送。对硬退信地址重复发信 = 垃圾邮件信号。
  • 软退信：暂时性失败（450 4.2.1 Mailbox full）。以退避方式重试（1 小时、4 小时、24 小时、72 小时）。连续 3-5 次软退信 = 视为硬退信处理。
  • 投诉（FBL——反馈循环）：接收方告知你"用户将此邮件标记为垃圾邮件"。AOL、Yahoo、微软提供 FBL 项目。Gmail 不提供逐封邮件的反馈，只提供汇总信誉数据。流程：接收方将一封 ARF（Abuse Reporting Format，滥用举报格式）消息 POST 到你注册的端点。立即禁止向该收件人发送邮件。
  • 退订：详见 §17。

退信处理必须在数秒内更新 user_emails.bounced=TRUE 标志并传播到缓存。一个常见的 bug：退信由夜间批处理任务处理，导致 24 小时内退信地址持续被重试，进而导致信誉急剧下降。

§15. SMS 经济学与运营商过滤

SMS 看起来与推送通知是同类原语，但其经济模型、监管要求和投递机制截然不同。推送通知本质上是免费的（你为基础设施付费，而非按条付费）；SMS 每条都需要付费，且受作为公共运营商服务的监管约束。

15.2. 短号码、长号码、免费电话号码与字母数字发件人 ID

长号码 （普通 10 位电话号码，如 +1-415-555-0100）：开通费用低廉，但吞吐量有限（在美国"A2P 10DLC"规则下约 1 条/秒/号码，见下文）。适用于事务性 1:1 通信、回复及客服场景。

短号码 （5-6 位数字，如 32665 = "FBOOK"）：每个号码每月 500-1000 美元，吞吐量可达 100+ 条/秒，经运营商预批准用于大规模流量。在美国营销场景下为强制要求。审批需 4-8 周（运营商审核使用场景）。可信度高，消息几乎不被过滤。

免费电话 SMS （如 +1-800-...）：中间方案。须通过免费电话消息验证（TFMV）流程审批。吞吐量 10-25 条/秒。适用于事务性通信 + 轻量级营销。

字母数字发件人 ID （如以"Uber"、"Stripe"作为发件人名称，而非号码）：在欧盟大部分地区、印度、巴西均受支持。美国、加拿大不支持（用户无法回复）。每个国家均需提前注册；未注册的字母数字 ID 在相应国家会被封堵。

15.3. 运营商过滤与 A2P 10DLC

美国运营商（AT&T、Verizon、T-Mobile）会激进地过滤其认为疑似垃圾邮件、网络钓鱼或违禁内容（赌博、大麻、掠夺性贷款）的消息。过滤是不透明的——Twilio 返回 delivered=true ，但消息从未送达。

在 2020-2021 ，美国运营商推出了 A2P 10DLC （Application-to-Person 10-Digit Long Code，应用对人 10 位长号码）规范——所有使用长号码发送的企业必须向 TCR（The Campaign Registry）注册品牌和使用场景。未注册时，长号码消息会被大幅限速或封堵。注册后，可获得分级吞吐量（依信任评分从 1 条/秒到 75 条/秒不等）。

注册组成部分： 品牌：公司身份、EIN/DUNS、网站。审核级别（标准/个体经营/慈善/政府）决定吞吐量。 广告活动：每个使用场景的审批（2FA、营销、账户通知、客服）。不得将一个活动用于不相关的内容。 每条消息标记：每条 SMS 包含活动 ID；运营商验证一致性。

运营商过滤会拒绝：URL 短链（bit.ly, tinyurl）——在网络钓鱼中常见；请使用自有品牌短链。未遵守"STOP/HELP"回复规则——立即封堵。掠夺性关键词（大量使用 $$、"winner"、"click here"）。内容与活动类别不符。

15.4. TCPA——美国合规

TCPA （Telephone Consumer Protection Act，电话消费者保护法，1991 年）规范未经请求的通信行为。对于营销 SMS，你需要获得 事先明确书面同意 ——用户必须主动同意接收来自你特定品牌的营销短信。从"他们提供了号码用于快递送货"推断出的同意，对营销短信无效。

处罚： 每次违规 500 至 1500 美元。集体诉讼破坏性极强：100 万收件人的 TCPA 违规 = 5000 万至 1.5 亿美元。

营销 SMS 的必要内容：品牌标识（如"Acme:"）；在第一条消息中注明"回复 STOP 退订"；"Msg & data rates may apply"披露；退订必须在 24 小时内生效（行业惯例更快——即时生效）。

对于事务性 SMS（2FA 验证码、物流更新、欺诈告警），TCPA 门槛较低——交易关系中的隐含同意通常已足够——但各州规则有所不同。

15.5. "我们发送了 100 万条 SMS，但只有 80% 送达"的问题排查

Twilio 的"已投递"状态回调在运营商接受消息时触发。但用户感知的实际投递率更低，原因如下：

  • 运营商静默丢弃 (delivered=true （但从未送达）——未注册 10DLC 为 5-15%，短号码接近 0%。
  • 用户手机关机或超出信号覆盖范围 超过 72 小时——运营商使消息过期。Twilio 报告 undelivered.
  • 无效号码 — failed=true，并予以退款。
  • 区域封堵 ——部分目的地（古巴、非洲部分地区）被美国运营商过滤。

对账方法：拉取 Twilio 投递报告，与内部点击/打开数据关联（SMS 中的链接包含追踪 token）。"用户确实看到了"的真实比率通常是"已发送"的 60-80%——明显低于邮件或推送通知。

§16. Push token 生命周期

设备 token（APNs token、FCM registration token）是寻址的基本单元。它 非 永久标识符；其轮换频率远超开发者的预期，且过期 token 会快速积累。

16.1. Token 下发

1. App launches. Calls platform SDK.
   iOS: UIApplication.shared.registerForRemoteNotifications()
   Android: FirebaseMessaging.getInstance().getToken()
2. SDK negotiates with APNs / FCM (TLS connection to push platform).
3. Platform returns a token string (~64-char hex for APNs, ~150+ chars for FCM).
4. App POSTs token to your backend with user_id + device_id + app_version + platform.
5. Backend UPSERTs into device_tokens table (user_id, device_id) primary key.

Token 下发发生在每次冷启动时（本地缓存；并与平台确认）。若缓存的 token 与平台返回的不一致，则向后端注册新 token。

16.2. Token 轮换触发条件

Token 因多种原因轮换。App SDK 检测到轮换事件后，开发者必须重新注册：

  • 重新安装应用 ——旧 token 立即失效。
  • iOS 版本升级 （主版本号升级通常会触发轮换）。
  • APNs/FCM 侧轮换 ——平台可能每 6-12 个月轮换一次 token，无需用户操作。
  • 备份恢复 ——恢复到新设备或操作系统重置后会触发 token 轮换。
  • 用户登出后重新登录 （某些应用将 token 与用户身份绑定）。
  • 清除应用数据 （Android — 设置 → 清除应用数据）。
  • 特别是 FCM 轮换频率比 APNs 更激进。

轮换契约：应用的 didReceiveRegistrationToken （Android）/ application:didRegisterForRemoteNotificationsWithDeviceToken: （iOS）回调在新 token 生成时触发。 每次 冷启动时应将 token 重新 POST 给后端（幂等 UPSERT）——依赖"仅在变更时注册"的策略会在本地缓存与平台实际状态不一致时产生漏洞。

16.3. 无效 token 检测

APNs 返回：- 410 Gone + body {"reason":"Unregistered"} → token 永久失效，标记为不活跃。- 400 + {"reason":"BadDeviceToken"} → token 从未有效，标记为不活跃。- 413 payload 过大 → 非 token 问题。

FCM 返回：- 404 NOT_FOUND 或 400 与 error.details[].errorCode = UNREGISTERED → token 失效。- 400 INVALID_ARGUMENT errorCode = INVALID_REGISTRATION → token 从未有效。

这些响应触发 §8.3 中记录的反馈循环：feedback.events Kafka 消息 → device_tokens.active=FALSE → 下次推送跳过该 token。

16.4. "50% 无效 token"问题

一种常见的生产状态：一个应用在 MySQL 中注册了 1 亿个 token，但活跃用户只有 5000 万。另外 5000 万个 token 已失效（卸载、重装或系统重置）。若不及时清理，每次推送扇出需发送 1 亿次 API 调用，并承担相应成本（APNs 免费，但 FCM HTTP/2 连接负载是真实存在的；SES 按邮件发送计费）。

积累来源：- 后端从未收到无效 token 反馈（某些 FCM 路径会静默抑制——只回调"已送达"，不回调"该 token 已失效"）。- 后端收到反馈但未处理（反馈循环存在缺陷）。- 应用在注销时未注销 token（用户注销后 token 仍与其账户关联，但应用此时已切换到其他用户）。- 多账户应用未正确将 token 绑定到对应账户。

缓解措施：\n- 定期清理：60 天内无成功推送的 token → 标记为不活跃。若用户重新启动应用，重新注册即可恢复激活。- 主动探测：每周发送一条静默推送（content-available: 1）；若失败（410），标记为失效。- 按平台统计指标：按平台和应用版本跟踪 dead_token_rate ，出现峰值 = SDK 缺陷或迁移问题。- 应用端清理：注销时，应用调用 DELETE /devices/{device_id} 至后端。

可接受的稳定状态：无效 token 占比 <5%。>20% 表示积累严重；>50% 表示多年疏于维护。

§17. 退订与偏好中心

退订处理是通知系统与合规法律产生碰撞的地方。处理不当 = 监管罚款、品牌受损，以及愤怒的用户将邮件标记为垃圾邮件（进而拉低送达率）。

17.1. 一键退订

RFC 8058 的时间戳与 List-Unsubscribe + List-Unsubscribe-Post 头部定义了一键退订。Gmail（自 2024 年 2 月起）和 Yahoo 对所有每日发送量 >5k 的发件人作出要求，FTC 依据 CAN-SPAM 法案也有相应规定。

邮件头部：

List-Unsubscribe: <https://example.com/unsub?token=abc123>, <mailto:unsub+abc123@example.com>
List-Unsubscribe-Post: List-Unsubscribe=One-Click

邮件客户端识别两个头部并渲染一键"退订"按钮。用户点击后，邮件客户端向 HTTPS URL 发出 POST 请求（而非 GET——POST 可避免预取器或链接扫描器误触发退订）。POST 请求体中包含 List-Unsubscribe=One-Click 表单参数。

后端处理程序必须：1. 验证 token（HMAC 签名，包含 user_id + list_id + timestamp）。2. 同步更新偏好——POST 返回前用户必须已完成退订。3. 以 HTTP 200 确认。4. 数分钟内对该地址的后续邮件进行抑制。

常见缺陷：后端异步将退订操作加入队列 → 返回 200 → 5 分钟后营销批次再次向该用户发送邮件。RFC 8058 暗示，Gmail 强制要求退订必须"及时"生效（CAN-SPAM 将其解释为 <10 个工作日；实践中应为即时生效）。

17.2. CAN-SPAM"10 个工作日"违规

CAN-SPAM 法案（美国）要求在 10 个工作日内响应退订请求。许多公司的退订流水线构建于 T+0 click → T+nightly batch updates DB → T+next-day cache invalidates → mail service sees fresh state at T+1 to T+2.

违规模式：批处理任务存在缺陷，或批处理连续失败一周 → 已退订用户在 10 天后仍持续收到邮件。FTC 已就此类案件达成每起 100 万美元以上的和解；针对大规模营销商的集体诉讼金额可达 1000 万至 5000 万美元。

修复方案很简单：写穿式缓存失效。点击 → 写入 DB + Redis DEL → 后续读操作立即看到最新状态。

17.3. 精细化偏好中心

"一键全部退订"按钮对送达率有害。完全退订的用户 → 无互动 → 最终将未来的合规邮件标记为垃圾邮件 → 声誉下降。

精细化偏好让用户可以降低频率而不彻底离开：

Per-channel:    [push] [email] [SMS] [in-app]
Per-category:
   - Transactional        [always on, cannot opt out]
   - Account & security   [always on for some, opt-in for others]
   - Product updates      [user controlled]
   - Marketing            [user controlled]
   - Recommendations      [user controlled]
   - Social               [user controlled, granular by type]
   - Digest               [daily / weekly / off]
Per-frequency:  [real-time] [daily summary] [weekly digest]
Per-time:       [9am-5pm] [respects quiet hours]
Per-source:     [mute user X] [mute thread Y]

数据结构：

CREATE TABLE notification_preferences (
  user_id      BIGINT       NOT NULL,
  channel      VARCHAR(16)  NOT NULL,  -- push/email/sms/inapp
  category     VARCHAR(32)  NOT NULL,  -- marketing/security/social/...
  enabled      BOOLEAN      NOT NULL,
  frequency    ENUM('realtime','daily','weekly','off') DEFAULT 'realtime',
  updated_at   BIGINT       NOT NULL,
  PRIMARY KEY (user_id, channel, category)
);

每条通知在创建时携带 (channel, category) 标签。调度器对每个 (user_id, channel, category) 进行单次查询 → 启用/禁用。无论组织内部的压力如何，营销团队都无法向已关闭该类别的用户发送"营销"通知。

17.4. 退订传播：最终一致性与强一致性

营销基础设施跨越多个服务：A/B 测试平台、旅程编排器、邮件服务、短信服务、应用内收件箱。退订操作必须传播到所有服务。

最终一致性 （典型方案）：退订事件 → Kafka → 各下游服务消费 → 更新本地缓存。中位传播时延 1-5 秒，p99 为 30 秒。

强一致性 （合规所需）：偏好中心是唯一可信来源，每次发送决策都必须读取（带写穿式失效的 Redis 缓存，而非异步复制）。每个投递处理程序执行：

1. Read preferences from Redis (cache).
2. If cache miss → read from MySQL → repopulate cache.
3. If preferences indicate suppression → drop, log delivery_status=SUPPRESSED_PREFERENCE.

对于高合规性渠道（营销邮件、营销短信），最终一致性存在法律风险。修复方案：写穿式失效使缓存具备强读语义。部分公司进一步在营销路径中进行同步 MySQL 读取（牺牲 1-2 毫秒延迟以换取合规确定性）。

§18. 合规：GDPR / CAN-SPAM / TCPA / CASL

通知系统处于监管地雷区。不同司法管辖区有不同规定；平台必须在发送时强制执行。

18.1. GDPR（欧盟）

GDPR（《通用数据保护条例》，欧盟，2018 年）规范欧盟居民的个人数据处理。通知相关影响：

  • 合法依据要求 用于处理邮件/电话（第 6 条）。对于营销通知，依据几乎始终是 同意 ——明确、具体、自愿给予且可撤回。
  • 同意须具体 ——"我同意接受营销"过于宽泛。"我同意通过电子邮件接收每月产品更新"才算具体。
  • 删除权（第 17 条） ——用户请求删除 → 所有 PII 必须被移除，包括通知日志中的内容。审计记录必须证明合规。
  • 同意审计记录 ——用户何时选择加入？IP 地址？表单版本？这是证明合规的必要依据。存储在仅追加日志中，保留期 7 年以上。
  • 跨境传输（第五章） ——如果您的通知平台位于美国并向欧盟用户发送通知，则需要传输机制（标准合同条款、充分性决定或数据隐私框架）。从美国基础设施向欧盟居民发送邮件构成数据跨境传输。

处罚：最高 全球年营收的 4% 或 2000 万欧元，取较高者。真实案例：Meta 12 亿欧元（2023 年），Amazon 7.46 亿欧元（2021 年）。

18.2. CAN-SPAM（美国）

CAN-SPAM（《控制非请求色情和营销法》，2003 年）规范商业电子邮件。要求：

  • 禁止伪造/欺骗性的 From、To、Reply-To 或 Subject ——品牌必须可识别。
  • 将邮件标识为广告 ——营销邮件必须明确表明是广告。
  • 必须提供实体邮政地址 ——每封商业邮件的页脚必须包含有效邮政地址。许多公司在包含"顺便看看……"交叉销售的交易性邮件中遗漏了这一点——一旦加入营销内容，就必须添加地址。
  • 退订机制 ——清晰、醒目、便捷。10 个工作日内响应。不得收取退订费用。不得要求登录。
  • 监督代表您行事的第三方 ——如果某代理机构以您的名义发送垃圾邮件，您将承担责任。

处罚：每次违规 51,744 美元（2024 年）。向 100 万收件人发送垃圾邮件理论上可达 510 亿美元；实际和解金额远低于此。

18.3. TCPA（美国——短信和语音）

TCPA（《电话消费者保护法》，1991 年）规范基于电话的通信。短信相关影响：

  • 事先明确书面同意 用于营销短信。标准很高：须有签署的协议（数字签名有效），注明用户将在特定号码上接收营销短信。
  • 默示同意 足以用于交易性通知（您已下单 → 我们发送物流更新短信）。
  • 退订 ——STOP、UNSUBSCRIBE、CANCEL、END、QUIT、OPTOUT 均须响应。
  • 主叫方 ID 要求 ——禁止欺骗性号码。

处罚： 每次通话/短信 500 美元，每次故意违规 1,500 美元。集体诉讼因单次违规金额低而具有毁灭性——意外向 10 万名未同意用户发送短信 = 5000 万至 1.5 亿美元。

18.4. CASL（加拿大）

CASL（《加拿大反垃圾邮件法》，2014 年）比 CAN-SPAM 更为严格：

  • 必须明确选择加入 ——默示同意仅在现有业务关系中被严格允许，且在 2 年不活跃后过期。
  • 身份识别 ——必须明确标识发件人，并包含联系信息。
  • 退订 ——发送后必须有效 60 天；10 个工作日内响应。

处罚：最高 企业每次违规加拿大元 1000 万 ；个人亦享有私人诉权（个人可提起诉讼）。

18.5. "营销想发，法律不许"的现实

这是长期存在的矛盾。营销部门希望向 5000 万收件人列表发送活动。法律团队审查该列表：- 800 万用户选择加入的是"产品更新"而非"营销"。- 500 万用户是欧盟居民——其同意是否在 GDPR 框架下正确记录？营销措辞是否与其同意内容一致？- 1200 万用户超过 18 个月未互动——依据 CASL，其默示同意已失效。- 100 万用户列表中的电话号码仅用于双因素认证——TCPA 禁止将其用于营销。

结果：法律批准的列表 = 2400 万，而非 5000 万。营销部门反弹。工程团队构建平台，使 偏好中心成为唯一可信来源 的时间戳与 抑制列表在发送时强制执行 ——营销团队无法在不修改平台的情况下绕过它。

相反的失败模式：法律部门未参与，营销向所有人发送，欧盟监管机构审计，随之而来的是 GDPR 罚款。平台必须强制执行策略，而不是信赖意图。

§19. A/B 测试通知内容

通知是任何产品中可衡量性最强的事物之一——每次发送都有清晰的归因（已发送/已送达/已打开/已点击）。这是实验的沃土，而大多数公司对其利用不足。

19.1. 测试内容

  • 主题行 （邮件）："Your weekly summary" vs "Yifan, your week in review" vs "5 highlights from last week" → 打开率相差 10-30%。
  • 预览文字 （主题下方约 100 个字符）——对打开率有重大影响。
  • 发送时间 ——"9am local Tuesday" vs "11am local Tuesday" vs "模型预测的每用户最优时间"。
  • 发件人名称 ——"Acme" vs "Yifan from Acme" vs "Acme Notifications"。
  • 推送标题和正文 ——"New like" vs "Alice liked your post" vs "Alice loved your idea" — 打开率相差 3-10%。
  • 渠道组合 ——仅推送 vs 若 1 小时内未打开则推送 + 邮件。
  • 频率上限 ——每天 5 条 vs 每天 10 条 → 衡量长期留存率。

19.2. 发送时间优化

按用户个性化的 STO（Send-Time Optimization，发送时间优化）：预测特定用户最可能打开的时间。模型基于数周内每用户的互动时间戳进行训练。输出结果示例："用户 12345 在周二至周四早上 7:42 最可能打开邮件。"

实现方式：夜间批处理更新 user_preferences.optimal_send_window 每个用户-渠道对。调度时，若活动支持 STO，则使用延迟队列将其安排在用户的最优窗口。

陷阱：STO 非常适合 1 小时延迟可接受的营销场景。请勿将 STO 应用于交易性或告警通知。

19.3. 实验流水线

1. Define experiment:
   experiment_id = "weekly_digest_subject_v3"
   variants = ["control: Your weekly summary",
               "A: Yifan, your week in review",
               "B: 5 highlights from last week"]
   traffic_allocation = [0.5, 0.25, 0.25]
   metric = open_rate (primary), click_rate (secondary), unsubscribe_rate (guardrail)
   duration = 14 days
2. At send time:
   variant = hash(user_id || experiment_id) % 100
   if variant < 50: control
   elif variant < 75: A
   else: B
3. Log to experiment log:
   { experiment_id, user_id, variant, sent_ts }
4. Outcome tracking:
   { experiment_id, user_id, event_type, event_ts }
   events: SENT, DELIVERED, OPENED, CLICKED, UNSUBSCRIBED
5. Analysis:
   per-variant rates with confidence intervals
   chi-square or proportion-z test
   guardrail check: did variant tank unsubscribe rate?

按 hash(user_id || experiment_id)分桶：同一用户在本次实验中始终获得相同变体，即使跨会话也是如此。这即是 确定性分桶 ——对于干净的归因至关重要。

19.4. 打开率 vs 点击率作为成功指标

  • 打开率 （邮件）：用户打开了邮件（跟踪像素已加载）。由于邮件客户端预览以及 Apple Mail 隐私保护（MPP）会预取所有图片，即使用户从未查看邮件也会被记录为已打开，因此该指标存在虚高。自 2021 年 9 月起，iOS 上的打开率基本上是噪声。
  • 点击率：用户点击了邮件中的链接。信号更强，虚高程度更低。
  • 转化率：用户完成了您期望的动作（购买、回访等）。信号最强，但量最少——需要更大的样本量。
  • 长期互动率/留存率：此次活动是否提升了 30 天留存用户数？这是最强的信号，需要精心的实验设计（不要比较打开次数；比较群组）。

对于推送：打开率 = 用户通过推送通知打开了应用。通过启动 URL 进行直接归因。

对于短信：打开率 ≈ 送达率（几乎所有人都会阅读短信）。通过追踪 URL 统计点击率。

19.5. 护栏指标

在没有护栏的情况下优化打开率会导致灾难："紧急：立即打开"之类的主题行、操纵性紧迫感、欺骗性预览——这些都能提升打开率，但都会摧毁长期信任。护栏指标阻止以下行为：

  • 退订率
  • 垃圾邮件投诉率
  • 长期留存率
  • NPS 或情感调查
  • 跨产品互动（此通知是否蚕食了其他通知？）

如果某个变体在打开率上胜出，但退订率绝对值上升了 0.5%，则放弃该变体。

§20. 通知模板化

以 30 种语言、含个性化内容、A/B 变体和品牌一致设计，每天发送 1000 万条通知，需要一套模板系统。手工拼接 HTML 字符串无法扩展。

20.1. 模板引擎

  • Liquid （Shopify、GitHub）——开放语法，支持过滤器： {{ user.first_name | default: "there" }}。专为最终用户编写而设计；沙箱安全。
  • Handlebars （Mailchimp，在 JavaScript 生态中常见）——无逻辑模板，支持辅助函数。 {{#if user.is_premium}}...{{/if}}。Web 开发者熟悉。
  • Jinja2 / Twig ——模板中的完整编程语言；功能强大，但若允许不可信作者编写模板，很容易引入缺陷和安全漏洞。
  • MJML （Mailjet Markup Language）——声明式响应式邮件标记语言。 <mj-section><mj-column><mj-text>Hi</mj-text></mj-column></mj-section>。编译为能在 Outlook 2007、Apple Mail、Gmail 和移动客户端中正确渲染的 HTML，消除了邮件 HTML 的噩梦。
  • MJML + Liquid 组合——MJML 负责布局，Liquid 负责个性化。常见技术栈。

对于推送和短信，模板更简单——通常只是纯文本中的少量 {{ variable }} 变量替换。

20.2. 模板版本控制与回滚

模板变更频繁（营销调整、A/B 变体、产品更新）。版本控制至关重要：

template:weekly_digest
├── v1   (2025-01-15) [active until 2025-04-22]
├── v2   (2025-04-22) [active until 2025-09-10]
├── v3   (2025-09-10) [active]
└── v4   (2026-01-08) [staging — A/B at 5% traffic]

每个版本一经发布即不可变更。生产流量路由到特定版本（通常是"最新稳定版" + A/B 流量到"候选版"）。回滚 = 将流量重定向回 vN-1。

20.3. "我们发布了一个破坏了 10 万封邮件的模板变更"的回滚

灾难模式：有人直接在生产环境编辑模板（无版本控制），点击"保存"，模板现在包含格式错误的 {{ user.first_name } （缺少大括号）。后续 10 万封邮件渲染时出现字面字符串 {{ user.first_name } 在正文中。用户投诉。品牌受损。

更糟的模式：模板引用了某些用户不存在的字段（user.subscription.tier 当免费用户没有订阅时）。一半收件人收到的邮件显示"欢迎，高级会员 null！"

防御措施：1. 模板编辑须通过 PR，而非原地修改。已审核、测试并合并。 2. Lint 在部署前对所有模板进行常见错误检查（缺少大括号、未知变量、HTML 格式错误）。 3. 对测试用例进行渲染 ——每个模板包含 5-10 个示例用户（免费用户、高级用户、被封禁用户等）；CI 对所有用例进行渲染，检查缺失字段和 HTML 错误。 4. 邮件测试 ——每次模板变更自动发送至 template-qa@company.com 测试邮箱；团队在推广前进行审核。 5. 金丝雀发布 ——新模板以 1% 流量运行 1 小时，监控渲染错误率，确认无误后扩展至 100%。 6. 一键回滚 ——UI 按钮可立即回退到上一版本，恢复时间 < 1 分钟。

20.4. 模板测试基础设施

  • 渲染预览：在开发工具中，选择模板 + 示例用户 → 渲染 HTML。邮件客户端在其 UI 中提供渲染预览（Litmus、Email on Acid 提供跨客户端预览，价格在 $200-500/月之间）。
  • Litmus 测试：在 90+ 个邮件客户端（从 Outlook 2007 到 Gmail iOS）中渲染邮件并截图，可检测出 Outlook 表格布局错误。
  • 垃圾邮件评分：在发布前通过 SpamAssassin / MailTester 发送邮件；分数应低于 5 分。
  • 可访问性检查：图片需有 alt 文本、足够的颜色对比度，并提供纯文本替代内容。

§21. 多语言与本地化

对于全球化产品，每条通知都必须进行本地化。向 1 亿用户发送英文通知，而其中 6000 万人不懂英文，会导致低参与率、用户投诉以及文化不敏感问题。

21.1. 语言检测

输入来源（通常按优先级排列）：

  1. 用户设置的偏好语言 ——明确设置，优先级最高。"用户在设置中选择了西班牙语。"
  2. 应用语言环境 ——用户设备的系统语言。
  3. 浏览器语言环境 (Accept-Language （header）——用于 Web 推送、Web 邮件渲染。
  4. IP 地理位置 ——作为无偏好用户的兜底方案；精度低于语言设置。
  5. 默认配置 ——英语（美国），通常如此。

存储于 MySQL 中的每用户字段 users.locale = "es-MX" （BCP 47 语言标签），在每次登录时刷新。

21.2. 翻译流程

1. Engineer adds English string to source.
   message_key = "notif.new_follower.body"
   en: "{name} started following you"
2. Build extracts strings → POEditor / Lokalise / Crowdin / internal TMS.
3. Translators (human, sometimes machine-assisted) localize:
   es: "{name} empezó a seguirte"
   ja: "{name}さんがあなたをフォローし始めました"
   ar: "بدأ {name} في متابعتك"
4. Translations checked back into repo.
5. Build packages translations into delivery service.
6. At dispatch time, lookup string by (key, locale).

对于高并发内容（聊天消息、用户生成内容），人工翻译不可行，需使用机器翻译（Google Translate API、DeepL、内部模型），并附带质量警告。

21.3. 含语法规则的变量替换

大多数本地化问题源于此处。英语："1 message" / "2 messages"——单数与复数，两种形式。

斯拉夫语系（波兰语、俄语）有 3-4 种复数形式： - 1 条消息 - 2-4 条消息（小数形式——特殊形式） - 5 条以上消息（属格复数） - 0 条消息（有时为特殊形式）

阿拉伯语有 6 种复数形式（零、一、二、少数、多数、其他）。

模拟程序启动时会进行的重定基址操作。 ICU MessageFormat 或 Fluent （Mozilla）用于支持复数感知的模板：

{count, plural,
  =0 {No new messages}
  one {1 new message}
  few {# new messages}
  many {# new messages}
  other {# new messages}
}

本地化人员填写各语言的复数规则，渲染时由库选择正确的分支。

性别一致性（罗曼语系、斯拉夫语系、希伯来语、阿拉伯语）也需要类似的分支处理：

{user_gender, select,
  female {Es ist deine Freundin}
  male {Es ist dein Freund}
  other {Es ist deine Person}
}

21.4. RTL（从右到左）

阿拉伯语、希伯来语、波斯语、乌尔都语从右到左渲染。邮件 HTML 和推送 UI 都需要支持 RTL：

  • HTML： <html dir="rtl" lang="ar"> 。 text-align: right 加在 body 标签上。
  • 推送：大多数平台根据系统语言环境渲染，但标点符号和嵌入的英文/数字需要双向处理（Unicode Bidi 算法——使用 RLM、LRM 标记以提高清晰度）。
  • 图片方向：箭头和图标在 RTL 场景下可能需要翻转（"前进"箭头在 LTR 中指向右，在 RTL 中指向左）。
  • 表格和布局镜像——第一列变为最右侧列。

与母语为 RTL 的使用者在真实设备上进行 RTL 测试，使用伪语言环境（xx-RTL）进行工程质量验证。

21.5. 与本地化相关的通知故障案例

  • 货币格式化："$1,000.00"（美国）vs "1.000,00 €"（德国）。格式错误会导致混乱，甚至显示错误金额。
  • 日期格式化："11/04/2025"——美国解读为 11 月 4 日，欧洲解读为 4 月 11 日。应使用 ISO 8601（2025-04-11）或本地化格式库。
  • 时间格式化：14:30 vs 2:30 PM，12 小时制/24 小时制偏好因地区而异。
  • 地址格式：国家/州/邮编顺序因地而异，不要硬编码美国格式。
  • 文化禁忌：某些颜色、数字（中文语境中的 4，西方语境中的 13）和图片在某些文化中可能具有冒犯性。

§22. 深入探讨时区感知的免打扰时段

§7 的问题 5 已介绍了基础内容，以下探讨更深层的问题：

22.1. IANA TZ 数据库

不是"UTC-8"。IANA（互联网号码分配机构）TZ 数据库（也称为 Olson 数据库，tzdata）是命名时区到历史偏移量的权威映射，包含夏令时转换、历史变更和政治调整。

示例： America/Los_Angeles 当前在冬季为 UTC-8，夏季为 UTC-7。在 1942-1945 年，使用了具有不同偏移量的"战时时间"。2022 年，美国参议院通过（随后搁置）了一项使夏令时永久化的法案——若颁布，tzdata 将记录该变更。

将用户时区存储为 IANA 名称，在派发时使用库计算 now_in_user_tz （Java ZoneId、Python zoneinfo、JS Intl.DateTimeFormat）。永远不要存储忽略夏令时和政治调整的数字偏移量。

22.2. 每用户时区存储

存储：

ALTER TABLE users ADD COLUMN tz VARCHAR(64) DEFAULT 'UTC';
-- e.g. 'Asia/Tokyo', 'Europe/Berlin', 'America/Los_Angeles'

来源： - 用户显式设置（账户偏好）。 - 应用 SDK 在每次后端会话时上报设备时区（缓存 24 小时）。 - IP 地理位置作为未注册用户的兜底方案。

在每次登录时刷新，旅行中的用户可手动覆盖。

22.3. "用户出行"决策

用户平时在 America/Los_Angeles，出行至东京，设备时区变为 Asia/Tokyo，应用在下次会话时上报此变化。

我们是否使用 新的 时区来确定免打扰时段？ - 对于大多数通知：是的，免打扰时段跟随设备——用户不希望被东京凌晨 2 点的通知叫醒。 - 对于计划通知：存在疑问。"明天早上 9 点提醒我"——应该是太平洋时间 9 点（原始设置）还是东京时间 9 点（当前位置）？通常做法是：明确的 (tz_at_time_of_scheduling, local_time) 时间对，显示"9am Pacific (1am Tokyo)"，让用户可重新安排。 - 对于双因素认证：忽略免打扰时段，安全优先于睡眠。 - 对于告警（值班）：由明确的值班轮换处理，而非依据用户时区。

22.4. 大规模在用户早晨时段进行批量发送

营销团队希望在 1 亿用户各自的本地时间上午 9 点发送早间摘要。简单做法：在 UTC 时间 9 点运行定时任务发送 1 亿封邮件——但对大多数用户而言，这并非"用户的早晨"。

更优方案：按用户排程。调度器每分钟拉取符合条件的用户（1 亿 / (24*60) = 约 7 万），其 9am_local == now_utc 并进行派发。

陷阱：时区集中在人口密集地区。太平洋时间 9 点 = UTC 时间 17 点，约 1 亿太平洋时区用户在同一分钟内触达系统，形成全局雷群效应。

缓解措施：\n- 分散在时间窗口内发送："在本地时间 9-10 点之间发送"→ 分散到 60 分钟内，持续约 170 万用户/分钟。 - 容量预留：在美国东部时间 9 点前预热调度器集群，在欧洲西部时间 9 点后缩容。 - 优先级划分：事务型调度器与营销调度器分离——营销调度器进入专用池，不会挤占双因素认证验证码的资源。 - 预热缓存：在 T-15 分钟时，将即将进入时间窗口的用户偏好预加载到 Redis 中。

真实案例：Mailchimp 的"在最佳时间发送"功能会预测每个用户在每日时间窗口内的最优发送时间，并有意错峰发送以避免同时爆发。

§23. 多设备同步

现代用户拥有多台设备——iPhone、iPad、Mac、Apple Watch、多台机器上的 Web 浏览器。一条通知在逻辑上面向用户，但在物理上必须送达其所有设备，且已读状态必须同步。

23.1. "所有设备"投递

当用户 42 触发通知时：

1. Look up active devices for user 42:
   { iPhone (token_A), iPad (token_B), Mac (token_C), Apple Watch (token_D) }
2. Send to APNs with each token.
3. Inbox row written once (per user, not per device).

但请注意——APNs 提供了设备分组 API： - apns-push-type: alert 没有特殊分组 → 所有设备均会响铃。 - apns-thread-id → 在设备上对相关通知进行分组。 - Apple 的"iCloud 用户 ID" 绑定 → APNs 可向一个 iCloud 用户的所有设备投递，无需逐一枚举 token。此方式在实践中很少见，大多数应用仍自行管理每设备的 token。

Android FCM 在这方面较不成熟，没有原生的"此用户所有设备"路由——需自行管理每设备 token，发送 N 次。

23.2. 已读状态传播

用户在 iPhone 上已读，Mac、iPad、Watch 上的角标均应清除。

1. iPhone: user taps notification or opens app inbox.
2. App sends POST /notifications/{id}/read to backend.
3. Backend: UPDATE notifications_inbox SET read_at_ms = NOW() WHERE user_id=42 AND notification_id=X.
4. Backend publishes 'read.events' to Kafka: { user_id, notification_id }.
5. Per-device WebSocket workers consume; push state-change to connected clients.
6. iPad / Mac (if app open with WebSocket) update UI immediately.
7. Watch — if APNs supports state-change push (it does for newer iOS), send silent push:
   { aps: { content-available: 1 }, notification_id, action: "mark_read" }
   Watch processes silently, badge decrements.

难点在于：手表并非始终在线，iPhone 可能离线（电池耗尽），Mac 可能处于休眠状态。状态变更推送会在 APNs/FCM 中排队，TTL 约为 4 小时；如果设备在 TTL 内上线，状态即可同步。超过 TTL 后，唯一的同步途径是"下次打开应用"→ 从后端获取未读数。

23.3. "手机已清除，手表仍显示"的 UX 故障

常见故障：用户在手机上清除通知，手表仍显示横幅。原因：

  • 在清除时，APNs 无法通过 Bluetooth 或 Wi-Fi 连接到手表（与手机蓝牙断连）。
  • 通知已独立投递至手机和手表；手机的清除操作未传播出去。
  • 手表上的应用未启用后台应用刷新，无法处理静默推送。

修复方案：将应用内收件箱作为 通用已读状态的真相来源，当手表应用稍后打开时，拉取收件箱 → 发现通知 X 已读 → 清除角标。

23.4. 应用内收件箱作为通用已读状态的真相来源

推送通知是短暂的提示，持久且权威的"未读内容"存储在收件箱行的 read_at_ms 字段中。每台设备在每次打开应用时都与收件箱同步。即使推送丢失、错过或乱序，收件箱依然是权威来源。

这就是为什么收件箱写入必须在推送发送之前完成。如果推送已发出而收件箱写入失败，用户会看到一个横幅，但之后找不到对应内容。如果收件箱写入成功而推送失败，用户会在下次打开应用时看到通知——体验有所降级，但不会丢失信息。

§24. 富文本推送通知

推送通知已远超"标题 + 正文"的形式，丰富的功能特性能有效提升参与率。

24.1. 图片附件

APNs： mutable-content: 1 + Notification Service Extension，该扩展在设备上获取图片 URL 并附加图片。

{
  "aps": {
    "alert": { "title": "New photo from Alice", "body": "Check it out!" },
    "mutable-content": 1
  },
  "image_url": "https://cdn.example.com/photos/abc123.jpg"
}

FCM： image 通知 payload 中的字段原生支持。

{
  "message": {
    "token": "...",
    "notification": {
      "title": "New photo",
      "body": "From Alice",
      "image": "https://cdn.example.com/photos/abc123.jpg"
    }
  }
}

约束：APNs 的图片大小 ≤10MB，FCM ≤1MB，托管于 CDN（已缓存，速度快），支持 JPEG/PNG/GIF 格式。

24.2. 操作按钮

通知内的操作按钮允许用户回复、标记已读、删除、点赞等，无需打开应用。

APNs：在应用中定义一个 category：

let replyAction = UNTextInputNotificationAction(
    identifier: "REPLY",
    title: "Reply",
    options: [],
    textInputButtonTitle: "Send",
    textInputPlaceholder: "Type a message...")

let markReadAction = UNNotificationAction(
    identifier: "MARK_READ",
    title: "Mark as Read",
    options: [])

let category = UNNotificationCategory(
    identifier: "MESSAGE",
    actions: [replyAction, markReadAction],
    intentIdentifiers: [],
    options: [])

推送 payload 包含 "category": "MESSAGE"；操作系统渲染按钮，用户点击后触发应用的 userNotificationCenter(_:didReceive:) 回调（或针对非启动操作的后台处理器）。

FCM：操作在 payload 中逐条通知定义，机制类似。

24.3. 紧急提醒

iOS 12+： 紧急提醒 可覆盖"请勿打扰"和静音模式，无论用户设置如何，系统均会播放声音。需要具备以下条件： - Apple 颁发的特殊权限（通过开发者门户申请；仅授予合法用途——医疗、安全、紧急情况）。 - 独立于常规推送权限的用户授权对话框。 - 可通过 payload 控制音量（critical-alert-volume).

使用场景：医院呼叫系统、烟雾报警、安全告警。滥用 = Apple 吊销权限。

24.4. 临时通知（iOS）

iOS 12+ 还新增了 临时授权。应用可在未请求用户许可的情况下发送通知——但通知会静默出现在通知中心，没有横幅和声音。用户可看到后选择升级为完整通知或禁用。

使用场景：基于 AI 的可见性决策。静默发送 100 条候选通知，系统学习用户交互的通知类型，并将这些类型升级为完整提醒。

Payload：

{
  "aps": {
    "alert": "...",
    "interruption-level": "passive"
  }
}

级别： passive, active （默认）、 time-sensitive （绕过专注模式）、 critical （如上所述）。

24.5. Web 推送与 VAPID

Web 推送是相同的 APNs/FCM 基础设施对浏览器（Chrome、Firefox、iOS 16.4 起的 Safari）的开放。用户授权 → 浏览器向推送服务注册 → 应用存储订阅端点。

VAPID（自愿应用服务器标识，RFC 8292）负责鉴权：服务器使用私钥对发往推送服务的请求进行签名，推送服务用已注册的公钥验证。鉴权中无需每用户 token，只需每服务器身份。

订阅：

{
  "endpoint": "https://fcm.googleapis.com/fcm/send/eXaMpLeToKeN",
  "expirationTime": null,
  "keys": {
    "p256dh": "BNcRdreALRFXTkOOUHK1EtK2wtaz5Ry4YfYCA_0QTpQtUbVlUls0VJXg7A8u-Ts1XbjhazAkj7I99e8QcYP7DkM=",
    "auth": "tBHItJI5svbpez7KI4CCXg=="
  }
}

服务器将 payload + VAPID 头发送至 endpoint，推送服务转发至用户浏览器。详细内容见 §27。

§25. 反滥用与垃圾通知防治

通知疲劳是一个真实的生态级问题。当用户被通知淹没时，他们会卸载应用、静音，或训练自己忽略通知。

25.1. 每用户速率限制

Per-channel (push, email, SMS independently):
  - 5 push / hour
  - 30 push / day
  - 10 email / day
  - 3 SMS / day

Per-category:
  - Marketing: 1/day max, 5/week max
  - Social: 10/day max
  - Transactional: no cap
  - Security: no cap (but throttle bots)

通过 Redis 令牌桶（§4.3）强制执行。每次投递尝试均会检查；若达到限制，通知将： - 事务型：永不限速（限速会违背其目的）。 - 社交型：静默丢弃或合并为摘要。 - 营销型：静默丢弃，并计入参与率指标的统计。

25.2. 每发送方声誉

内部：每种通知类型/活动均有一个质量评分：

quality_score = w1 * open_rate + w2 * click_rate - w3 * unsubscribe_rate - w4 * complaint_rate

低质量活动受到更严格的限流；高质量（高参与率）活动获得优先处理。

营销团队的本能：每天向所有人发送。工程侧的执行机制：质量评分低于阈值 → 减少下次活动的覆盖范围，从而倒逼团队提供更优质的内容。

25.3. "营销想每天全量发送"与"用户卸载应用"之间的博弈

这一矛盾已被广泛研究，行业规范如下：

  • 推送频率：对于非必要内容，每天超过约 5 条时卸载率急剧上升。重度用户可接受 10-20 条/天（前提是全部高质量，如聊天应用）；营销密集型应用在每天 2-3 条时就会出现卸载。
  • 邮件频率：1-2 次/周是"设置后无需关注"的节奏，5+ 次/周则退订率上升。
  • 短信频率：营销类最多 1 次/周，2+ 次/周会引发用户投诉。

反直觉结论：降低频率往往 提升 总体参与度——通知疲劳减少，用户更可能打开您确实发送的通知。

25.4. 通知疲劳研究

相关研究（Google 研究、Facebook 内部研究、高校人机交互研究小组）一致表明：

  • 用户对通知的"注意力预算"有限——一旦超出，所有后续通知均会在未经查看的情况下被清除。
  • 一天中第一条通知的参与率最高，第 10 条的参与率接近零。
  • 在"闲暇时间"（傍晚、周末）收到的通知参与率低于工作时间。
  • 滥用"重要感"通知（红色角标、感叹号）会训练用户忽略它们。

设计启示：精打细算地发送、无情地排优先级、批量处理低价值更新，并以质量而非数量为优化目标。

25.5. 静默时段

除每用户免打扰时段外：对非关键内容设置系统级静默时段。示例：

  • 在任意时区的晚上 9 点到早上 9 点之间，不发送营销推送。
  • 周末不发送营销邮件。
  • "新年排毒"——在一月初收件箱疲劳度高时减少营销量。
  • 重大新闻事件期间（选举日、灾难发生时）——暂停营销，以免显得与情境格格不入。

这些是策略决策，而非技术限制，但平台会暴露相应的控制接口。

§26. 应用内收件箱模式

应用内收件箱（铃铛图标、通知界面）是持久的真相来源，存在两种架构模式。

26.1. 写时扇出至每用户收件箱

在§3和§4.1中有详细介绍。收件箱行预先写入；用户打开收件箱；快速分区扫描；行显示。

优点：O(1) 读取，UI 响应快。缺点：O(F) 写入成本；存储随用户数 × 事件数 × 重复倍数线性增长。

26.2. 收件箱打开时的懒加载水化

替代方案：不预写收件箱行。仅维护事件日志（记录哪个用户对哪条帖子做了什么操作）。在打开收件箱时，查询"最近30天内与用户X相关的所有事件"并合成收件箱内容。

1. Event log: per-event row keyed by (source_id, event_id, ts).
2. Subscription log: which user follows which sources, with start_ts.
3. Inbox open:
   - Get user's subscriptions.
   - For each subscription, scan events since (user's last_seen OR 30d ago).
   - Merge, sort, take top 50.
4. Cache result for ~5min per user.

优点：O(1) 写入（无需按用户重复存储）。存储成本低。

缺点：每次打开收件箱需 O(F') 次读取。缓存未命中代价昂贵。实时更新（用户在收件箱页面时收到新通知）需要 WebSocket 或轮询。

26.3. 混合模式：活跃用户预写，冷门用户懒加载

大多数系统采用混合模式：- 对活跃用户（30天内登录过）预写。- 对休眠用户（超过30天未登录）懒加载水化。

当休眠用户回归时，在后台异步水化其收件箱，同时展示加载状态，再切换显示。

存储节省：在典型产品中，约70%的用户处于休眠状态；为他们预写大多是浪费存储空间。

26.4. 已读状态传播

已在§23中讨论过。收件箱行的 read_at_ms 字段是规范来源。以下情况下会更新：- 用户打开收件箱（将所有可见消息标记为"已看"，但不一定是"已读"）。- 用户点击某条具体通知（将该条标记为"已读"）。- 用户触发"全部标记为已读"。

通过 WebSocket + 静默推送同步到其他设备。

26.5. 保留策略

收件箱不能无限增长。行业惯例：

  • 上限为 30 天 （Twitter、Facebook、Instagram）。
  • 上限为 100-500 条通知 每个用户（Slack、Discord）。
  • 上限为 每类别 N 条 （例如，50条社交通知、100条事务性通知、关键通知不限量）。

实现方案：- Cassandra TTL（default_time_to_live）—— 写入时自动设置过期时间。- 定期清理任务 —— DELETE WHERE user_id=X AND row_count > 100 （较复杂，每晚运行一次）。- 按类别保留 —— 对不同的 category 字段设置不同的 TTL；需要按类别分区或通过后过滤实现。

权衡取舍：保留时间越长 = 存储越多，收件箱查询越慢（需扫描更多行），压缩开销也越大。保留时间越短 = 当用户想查找历史通知时体验越差。

§27. Web 推送与渐进式 Web 应用

Web 推送将通知能力延伸至浏览器，包括未安装 App 的网站。

27.1. Service Worker 订阅

Web 推送能力要求：

  1. Service Worker 在对应域下注册 —— 这是一个 JS 后台脚本，即使没有标签页打开，浏览器也会保持其运行。
  2. 推送权限 —— 用户通过 Notification.requestPermission()主动授权；UI 提示的接受率通常约为5%–20%。
  3. 订阅 — serviceWorkerRegistration.pushManager.subscribe({ userVisibleOnly: true, applicationServerKey: VAPID_PUBLIC_KEY })。返回订阅端点 + 密钥。

App 向后端 POST 订阅信息。后端存入 device_tokens 表（platform = WEB）。

27.2. VAPID 密钥

VAPID（自愿应用服务器标识，RFC 8292）—— Web 推送的认证模型。无平台颁发的提供商证书（不同于 APNs）。服务器自行生成 ECDSA P-256 密钥对：

private key (kept on server)
public key   (registered with subscription; embedded in subscribe call)

每次推送请求中包含用私钥签名的 JWT。推送服务使用存档的公钥进行验证。

JWT 声明：

{
  "aud": "https://fcm.googleapis.com",
  "exp": 1716393600,
  "sub": "mailto:admin@example.com"
}

sub 是你的联系方式（如果你的服务器行为异常，浏览器厂商会通过此联系你）。

27.3. 载荷加密

RFC 8291：Web 推送载荷在服务器与浏览器之间进行 端到端加密。推送服务（Mozilla Push、FCM 等）无法读取内容 —— 仅转发加密字节。

加密流程：- 服务器为每次推送生成一个 ECDH 临时密钥对。- 用浏览器订阅公钥（P-256 ECDH）推导共享密钥。- HKDF 派生加密密钥。- 使用 AES-128-GCM 加密载荷。- 通过 HTTP 头发送密文 + 临时公钥。

浏览器端：- 用自身私钥 + 服务器临时公钥解密。- 触发 Service Worker 的 push 事件处理器。- 处理器调用 self.registration.showNotification(title, options).

这是少数真正实现端到端加密的通知信道之一 —— APNs/FCM 运营商可以读取你的载荷；Web 推送运营商无法读取。

27.4. "浏览器推送接受率低"的现实

Web 推送权限提示极具干扰性。Chrome 和 Firefox 的权限 UI 正变得越来越注重用户保护：

  • Chrome 80+：对重复请求使用更低调的 UI。行业分析显示默认权限接受率约为10%–20%。
  • Safari（iOS 16.4+）：要求站点以 PWA 形式安装到主屏幕。
  • Firefox：严格限制提示频率。

最佳实践：- 不要在首次页面加载时请求 —— 等用户表现出兴趣后再请求。- 明确传达价值："当你的朋友回复时获得通知"，而非"我们想向你发送通知"。- 被拒绝后不要再次提示 —— 适得其反。

在一个月活用户100万的网站上，5%的接受率 = 5万 Web 推送订阅用户。聊胜于无，但只是受众的一小部分。

27.5. 浏览器的特殊行为

  • Chrome / Edge：订阅在底层由 FCM 管理；稳定可靠。
  • Firefox：使用自己的推送服务（Mozilla）；总体可靠。
  • Safari：自 iOS 16.4 / macOS 13 起底层使用 APNs；与原生 iOS 认证方式相同（VAPID 或 APNs 证书）。
  • 移动浏览器：Android Chrome 支持 Web 推送；iOS Safari 仅支持 PWA。

§28. 通知可观测性

没有指标，通知平台就是一个黑盒。可观测性至关重要。

28.1. 按信道的投递率

频道 投递率目标 定义
推送（APNs/FCM） ~90% API 返回 2xx
邮件（SES/SendGrid） ~95% 24小时内无硬退信
短信（Twilio） ~95-99% 运营商已投递状态
站内信收件箱 ~100% 行已写入 Cassandra

低于目标即需排查。常见下降原因：

  • Push：token 轮换，失效 token 持续积累。
  • Email：可投递性问题（DKIM 轮换未完成，IP 声誉下降）。
  • SMS：运营商过滤，注册已过期。
  • In-app：少见 —— Cassandra 故障。

28.2. 打开率 / 已读率

"用户实际看到通知"的漏斗：

sent → delivered → opened → clicked → converted
100%   95%         30%       3%        0.5%

按信道和按活动细分。与基线对比；对回归发出告警。

追踪机制：- Email 打开：追踪像素（1x1 图片托管在你的 CDN 上，带有 ?token=msg_id）。图片加载 = 打开。Apple 邮件隐私保护（MPP）会预加载图片，自2021年起导致打开数虚高。- Email 点击：将所有链接重写为跟踪重定向（example.com/track?url=...&msg=...）。点击 = 重定向被记录。- Push 打开：用户点击推送后，App 记录启动来源。- SMS 点击：消息中包含追踪 URL。- In-app：从收件箱行的已读状态获取。

28.3. 从发送到转化的完整漏斗

大多数产品分析平台（Mixpanel、Amplitude）都与通知平台集成。每个 sent, delivered, opened, clicked, unsubscribed 事件流入分析管道，并带有 user_id、campaign_id、variant 标签。

各利益相关方的 KPI：- Eng：投递率、延迟、错误率、基础设施成本。- Product：打开率、点击率、留存影响。- Marketing：转化率、归因收入。- Legal/compliance：退订延迟、退信处理、投诉率。

告警规则：- 投递率绝对值下降 >2% → 呼叫值班人员。- 垃圾邮件投诉率 >0.1%（Gmail 阈值）→ 暂停活动，呼叫营销运营。- IP 声誉在 Postmaster Tools 中下降 → 呼叫可投递性团队。

28.4. 按收件人的投递审计

对于受监管内容（涉及 HIPAA、金融、GDPR 数据主体请求），你需要证明"我们在时间 Z 通过信道 C 向用户 Y 发送了通知 X"。这需要：

  • delivery_status 表，按（事件、收件人、信道）维度记录每行，金融类保留7年以上，通用类保留1年以上。
  • 只追加的审计日志（带合规锁定的 S3 或同等方案）。
  • 可按 user_id 检索（用于 GDPR 访问请求）。

审计日志与操作存储（保留期较短）分离 —— S3 / 对象存储成本低廉。

§29. 前面未涵盖的故障模式

§8 已涵盖常见故障。以下是更多补充：

29.1. APNs 限流 —— IP 隔离

如果超过 APNs 的吞吐量限制或向大量失效 token 发送，Apple 会对你的 IP 进行限流。限流可持续数小时至数天。恢复方法：指数退避，缓慢清空队列，监控成功率。

更严重的情况：反复违规 → APNs 可能将该 IP 隔离数周。届时你必须切换到新的出口 IP 并重新建立连接。缓解措施：- 严格处理 410（立即移除失效 token）。- 遇到 429 时退避。- 保持在每 IP 吞吐量限制的80%以下（粗略上限约为每秒10k请求）。- 分散到多个出口 IP。

29.2. FCM 上游变更

FCM 经历了多次破坏性变更：

  • 2014：GCM（Google Cloud Messaging）被 FCM（Firebase Cloud Messaging）取代。Token 格式发生变化。
  • 2018：XMPP 上行消息已弃用；必须使用 HTTP/2。
  • 2023-2024：旧版 HTTP API 已弃用；必须使用 FCM HTTP v1（采用 Google 服务账号认证的新版 API）。

每次迁移都导致未更新服务器代码的应用中断。运维经验：监控 Google 公告；订阅 firebase-talk；每年预留迁移时间。

29.3. Email 退信类型

  • 硬退信：硬退信（550 5.1.1 User unknown, 550 5.7.1 Blocked）。立即抑制。继续重试 = 垃圾邮件信号，会拉低你的声誉。
  • 软退信：暂时性失败（450 4.2.1 Mailbox full, 421 4.7.0 Temporary system problem）。带退避重试：1小时、4小时、24小时、72小时。若在3天内5次尝试后仍退信 → 视为硬退信处理。
  • 垃圾邮件退信：因目标服务器认为你是垃圾邮件而退信（550 5.7.1 Message rejected as spam）。单独分类；需要调查（而非简单抑制）。
  • 封锁退信：目标服务器已明确封锁你的 IP。难以恢复；在预热新 IP 后轮换旧 IP。

退信处理必须可靠 —— 退信队列积压意味着你持续向已知无效地址发送邮件，这将毁掉你的发件人声誉。

29.4. SES / SendGrid 故障

SES 曾出现部分故障（特定地域、特定收件人域）。缓解方案：

  • 多提供商冗余：SES 为主，SendGrid 为备。通过配置开关切换。
  • 按收件人域故障转移：若 SES → @gmail.com 失败但 SendGrid 正常，则将 Gmail 地址路由到 SendGrid。
  • 队列 + 重试：故障期间队列堆积 → 恢复后清空。监控队列深度。

29.5. 提供商 API 限速

每个外部 API 都有速率限制：

  • APNs：约每秒10k请求/IP/证书。
  • FCM：每个项目每秒60万 QPS（上限很高；很少触达）。
  • SES：按账户的 TPS（每秒事务数），可从1升级至10k以上。
  • Twilio：每个长号码（US 10DLC）每秒1条，高信任等级最高75条/秒。短号码100条/秒以上。

触达限制 = 429 / 4xx。退避。在稳定状态下保持在限额的80%以下。如有需要，分散到多个发送账户。

29.6. DNS 故障

提供商主机名（api.push.apple.com, fcm.googleapis.com, email-smtp.us-east-1.amazonaws.com）通过 DNS 解析。DNS 故障会导致级联失败。缓解措施：

  • 在本地长缓存 DNS 解析结果（在调度器主机上运行 resolver）。
  • 配置多个 resolver（VPC 的 DNS + Google 8.8.8.8 作为备用）。
  • 将 IP 固定作为后备（需谨慎 —— 提供商确实会轮换 IP）。
  • 使用带健康检查的服务网格 DNS（consul、内部 DNS）。

§30. 成本经济学

总结。各信道在大规模下的成本：

30.1. Push

  • APNs：免费。你为基础设施付费（调度器主机、持久连接容量）。
  • FCM：免费。同上。
  • 自托管基础设施成本：大规模下每100万条推送约 $0.05–0.10（计算 + 网络）。
  • SaaS 推送提供商 （OneSignal、Airship、Iterable、Braze）：根据量级和功能，每100万条推送约 $1–10（通常与其他信道及分析功能打包销售）。

对于每月10亿条推送的产品，自托管相比 SaaS 可节省每月 $1k–$10k —— 但你需要1–3名工程师维护。盈亏平衡点大约在每月1亿至10亿条推送之间。

30.4. 批量分级谈判

所有主要提供商都提供批量分级价格，但公开报价是标准价。实际企业谈判后的价格：

  • SES：标准价 $0.10/1k → 每月数十亿条时企业级低至 $0.04–0.06/1k。
  • SendGrid：小规模标准价 $1/1k → 每月数十亿条时约 $0.15/1k。
  • Twilio：定价 $0.0075/条短信 → 每月数亿条时约 $0.005-0.006。

谈判筹码：- 多年承诺（1-3 年）。- 多渠道捆绑（邮件 + SMS + 语音）。- 公开案例研究 / 联合营销。- 从竞争对手迁移（供应商会打折以赢得合同）。

30.5. 隐性成本

  • 独立 IP （用于发件人信誉控制）：每个 IP 每月 $50-500，另需预热时间。
  • 合规工具：DMARC 监控（Valimail、EasyDMARC）每月 $500-5000。
  • 可达性咨询：大规模场景下每月 $5-50k。
  • Analytics & A/B 平台：Iterable / Braze / Klaviyo 捆绑额外增加 $5-50k/月。
  • 工程人力：专属通知平台团队 = 每年全成本 $1-5M。

对于 B 轮公司，"自建 vs 采购"的核心从不是单条消息的成本，而是工程机会成本。对于通知是核心产品的 ARR 超过 $100M 的上市公司，自建在成本和控制力上更占优。

§31. 总结

通知投递是一个具备背压感知总线的扇出放大器：事件写入持久队列，扇出服务将其展开为每用户投递（按 recipient_id 分片），分发 worker 在紧凑的 Redis 循环中依次执行去重、偏好过滤、限流、免打扰时段处理，将收件箱持久化到 LSM-tree 存储（Cassandra，按 user_id 分区）作为真实来源，再推送至维护与 APNs/FCM 持久 HTTP/2 连接的渠道路由器，利用 collapse_id 在设备侧去重。核心设计关键在于接收方集合的分发策略：普通来源采用写时扇出，名人形态来源采用读时扇出，在此之上叠加基于互动率的推送采样——将一条 1 亿接收者的帖子转化为约 100 万次实际推送。投递语义为至少一次，幂等性在收件箱行和设备侧双重保障；恰好一次是错误的合约。收件箱是真实来源，推送只是提示，生产者永远不应感知渠道——它们将事件发送到队列，平台负责其后的一切。
