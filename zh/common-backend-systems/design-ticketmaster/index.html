1. 基础知识与解题思路

本题考察点

Ticketmaster 是一个在极端突发流量下运行的稀缺库存预订系统。

难点不在于列出活动或渲染场馆地图。难点在于以下不变量：

一个座位最多只能售出一次，即使在开售时有数百万用户同时涌入、客户端重试、支付超时、缓存过期，以及机器人攻击系统的情况下。

在 ICT5 / L6+ 级别，面试官考察的是：你能否将快速读取投影与权威预订状态分离、在流量到达库存之前控制突发流量，并解释防止超卖的精确状态转换。

60 秒面试框架

口述版本：

我会围绕稀缺座位库存和突发流量控制来设计 Ticketmaster。浏览和座位图读取可以由缓存和搜索索引提供服务，但锁座、结账和出票必须经过权威状态机。我将先澄清需求，再依次介绍读路径、预订路径、候票室、数据模型、扩展策略和故障处理。预计深入探讨的是支付重试和锁座到期下的座位锁定正确性。

首先明确需求

功能性需求：

  • 用户可按艺人、场馆、城市、日期和分类浏览活动。
  • 用户可查看活动详情、票价和座位图。
  • 用户可选择一个或多个座位并发起短时锁座。
  • 用户可完成结账并收到票。
  • 用户可查看订单历史和票务状态。
  • 系统支持活动主办方创建活动、场馆地图、开售窗口、定价层级和票务库存。
  • 系统需要机器人防护、购买限制和欺诈检查。

非功能性需求：

  • 防止座位重复售出。
  • 在热门活动开售时承受大规模突发流量。
  • 即使在开售期间，也要保持浏览和搜索的低延迟。
  • 在重试情况下保持结账的可靠性和幂等性。
  • 允许浏览视图中的可用性信息有一定延迟，但最终预订决策中绝不允许。
  • 提供资金、库存、出票、退款和争议的可审计性。

除非面试官要求，否则超出范围：

  • 完整二手票转售市场。
  • 动态定价优化。
  • 场馆入场扫码。
  • 主办方数据分析仪表板。

规模假设

大声说出假设条件，让设计有具体的形态：

领域 假设
用户 注册用户数亿。
事件 数百万场活动，但同一时刻只有少量为热门活动。
热门开售 1 到 1000 万用户在开售前后同时涌入。
库存 一个场馆有 10,000 到 100,000 个座位。部分活动采用不对号入座方式。
读写比例 浏览和座位图读取占主导；锁座和购票写入量少得多，但正确性至关重要。
Latency（延迟） 搜索需在几百毫秒以内；锁座请求在突发期间约为几百毫秒到几秒；结账可能需要几秒，因为涉及支付提供商。
一致性 座位的锁定和售出需要强一致性。活动搜索和显示的可用性可以是最终一致的。

瓶颈不在于常规数据库吞吐量，而在于热门活动中大量用户在同一时刻争抢同一小批座位。

核心产品模型

从三个独立界面来考量：

界面 用户体验 系统需求
服务发现 搜索和浏览活动。 快速、可缓存、最终一致。
选座 查看座位图并选择座位。 以读为主，可显示轻微过期的可用性信息，但需在最终验证前明确告知。
预订 锁座、支付、出票。 按座位强一致，幂等，可审计。

主要设计失误是将发现层或座位图层作为权威数据源，它们应仅作为投影层。

核心组件

组件 职责 关键细节
Edge、CDN、WAF 提供静态资源、吸收流量、阻止明显的滥用行为。 活动页面和场馆地图元数据应可缓存。
API 网关 身份验证、授权、限流、请求路由。 在流量到达库存之前执行每用户和每设备的限制。
活动目录服务 管理活动元数据、演出者、场馆、开售窗口、定价层级。 不是座位售卖状态的拥有者。
搜索服务 按文本、地理位置、时间和分类对活动进行索引，以供发现。 来自活动目录的最终一致投影。
座位图服务 提供场馆布局、区域、座位标签、无障碍信息和价格区域。 场馆几何信息变化较少；可用性变化频繁。将二者分开维护。
可用性投影 显示近似的可用、已锁定、已售出或库存紧张状态。 由库存事件派生；对 UI 有用，但非最终权威。
等候室 控制进入热门活动购票流程的准入。 保护库存服务并提供公平性保障。
座位锁定服务 创建和到期短时锁定。 座位状态转换的权威路径。
结算服务 验证锁定所有权、扣款、将锁定转为已售。 必须是幂等的，因为支付调用可能超时且用户会重试。
票务账本 发行不可变的票务记录、转让状态、条形码和审计日志。 仅在结账成功后才创建票。
事件流 发布库存、订单、支付和票务事件。 用于重建投影和调账失败。
机器人与欺诈服务 对用户、设备、IP、频率、支付风险和账号年龄进行评分。 应在接触稀缺库存之前运行。

座位状态机

这是设计的核心。

available -> held -> sold
     |         |
     |         +-> expired -> available
     |
     +-> blocked

允许的状态转换：

状态转换 操作方 条件
available -> held 座位锁定服务 座位可用、开售窗口已开启、用户持有有效准入令牌、条件写入成功。
held -> sold 结算服务 锁座属于该用户、锁座未过期、支付成功、幂等键已接受。
held -> expired 到期工作进程或锁定服务 当前时间超过 hold_expires_at，锁座未转换。
expired -> available 到期工作进程 释放座位以供后续锁定。
available -> blocked 主办方/管理员 座位被预留、移除、损坏或因合规需要保留。

重要实现细节是条件更新：

UPDATE Seat
SET status = 'held', hold_id = ?, hold_expires_at = ?, version = version + 1
WHERE event_id = ?
  AND seat_id = ?
  AND status = 'available'
  AND version = ?

对于多个相邻座位，可选方案为：

  • 若所有选中座位都在同一分区，使用单一分区和事务。
  • 使用确定性座位排序和条件写入，若任何座位失败则回滚。
  • 优先从同一分片选择座位组，以避免热路径中的分布式事务。

面试用语：

座位图可能是过期的，但锁座转换不能是过期的。最终决策是对权威座位记录进行条件状态转换。

写路径：创建锁座

  1. 用户进入活动页面并尝试购票。
  2. API 网关验证身份、开售窗口、限流和机器人评分。
  3. 若活动为热门活动，用户必须出示候票室准入令牌。
  4. 座位锁定服务验证 event_id, seat_ids，用户购买限制和幂等键。
  5. 服务路由到 event_id 或 event_id + section.
  6. 库存存储执行从 available 与 held.
  7. 创建带有短时到期时间的锁座记录，例如 2 到 10 分钟。
  8. 发出库存事件以更新可用性投影和分析数据。
  9. 客户端收到 hold_id 。 expires_at.

关键点：响应等待权威锁座完成，而非等待搜索索引或可用性缓存刷新。

写路径：结账

  1. 客户端提交 hold_id，支付方式和结账幂等键。
  2. 结账服务验证锁座属于该用户且仍处于活跃状态。
  3. 使用幂等键向支付提供商创建支付意图。
  4. 若支付成功，将锁座转换为 converted 以及座位从 held 与 sold.
  5. 在票务账本中创建订单和票务记录。
  6. 发出订单已支付和票已出票事件。
  7. 通过邮件、推送和订单页面更新通知用户。

支付超时处理：

  • 若提供商返回成功，完成订单。
  • 若提供商返回失败，释放锁座或让其到期。
  • 若提供商超时，将订单标记为 payment_unknown 或 pending_payment ，在释放座位前与提供商调账。
  • 如果支付可能已成功，不要立即将相同座位卖给其他人。

面试用语：

支付会产生未知状态问题。我宁愿临时保留锁座并进行调账，也不愿冒着向一个用户收费后又将座位卖给另一个用户的风险。

读路径：搜索和活动浏览

  1. 客户端按查询词、地理位置、分类和时间搜索活动。
  2. API 网关路由到搜索服务。
  3. 搜索服务从活动索引读取，而非从预订数据库读取。
  4. 活动详情页从目录缓存读取活动元数据。
  5. 座位图几何信息从 CDN 或缓存加载，因为其变化较少。
  6. 可用性覆盖层从具有短 TTL 的可用性投影加载。

该路径可以是最终一致的。若 UI 显示某座位看似可用但锁座失败，系统返回明确的不可用响应并刷新可用性覆盖层。

候票室设计

候票室保护权威路径免受流量突峰影响。

目标：

  • 按座位锁定服务可处理的速率准入用户。
  • 防止绕过队列。
  • 保持合理的公平性。
  • 向用户提供进度反馈。
  • 让运营人员可以暂停、减缓或停止准入。

流程：

  1. 在开售前，用户进入候票室。
  2. 系统分配包含用户 ID、活动 ID、发放时间、随机种子和签名的队列令牌。
  3. 开售时，用户按公平性策略排序，例如已在候票室中的用户随机排列，之后按先到先得顺序。
  4. 准入控制器以受控速率发放已签名的准入令牌。
  5. 热门活动的预订 API 要求提供有效的准入令牌。
  6. 令牌快速过期，并与用户、活动和设备/会话绑定。

为何不采用纯粹的先到先得？

  • 它奖励了机器人和网速最快的用户。
  • 它在开售时刻制造了边缘流量踩踏事故。
  • 随机候票室可以更公平，也更易于运营。

准入令牌：

AdmissionToken(
  token_id,
  event_id,
  user_id,
  admitted_at,
  expires_at,
  max_hold_attempts,
  signature
)

扩展与分区

分区策略选择：

数据 分区键 原因
活动目录 event_id 或主办方 常规元数据读写。
搜索索引 词条/地理位置/时间分片 查询服务与排名。
座位库存 event_id + section_id 适用于大型活动， event_id 适用于较小规模的活动 使热门活动库存保持隔离。
锁定（Holds） event_id + hold_id 或 user_id 取决于查询路径 需要在结账时按锁定查询，以及在过期时按活动查询。
订单 user_id 。 order_id 索引 用户查看订单历史；客服人员使用订单查询。
门票 event_id + seat_id 。 ticket_id 场馆入场与所有权查询。

热门活动策略：

  • 在销售开始前预先计算活动页面、场馆几何信息、价格档次及分区级别的可用性。
  • 将热门活动放入专用库存分片或隔离容量池。
  • 使用候票室来控制写入流量。
  • 将搜索、分析和推荐流量与订票数据库隔离。
  • 对于超大型场馆，按分区拆分库存，使选择楼座的用户与地面席位用户互不争抢。
  • 利用库存分片延迟产生的背压来降低候票室的放行速率。

缓存策略：

  • 积极缓存活动元数据与座位图几何信息。
  • 使用短 TTL 和版本号缓存可用性覆盖层。
  • 最终锁定或结账时绝不信任缓存。
  • 使用事件流更新快速失效热门分区的缓存。

一致性模型

需要强一致性：

  • 座位锁定创建。
  • 座位销售转换。
  • 门票签发。
  • 影响门票有效性的退款或作废操作。

可接受最终一致性：

  • 搜索结果。
  • 活动详情页面的数量统计。
  • 座位图可用性覆盖层。
  • 推荐与热度排名。
  • 分析仪表板。

这是经典的权威存储/投影分离模式：

层级 一致性 原因
库存存储 每个座位或库存桶的强一致性 防止超卖。
锁定/订单状态机 强一致状态转换 重试必须收敛。
可用性投影 最终一致 仅作为 UI 提示。
搜索索引 最终一致 发现功能可以容忍延迟。

过期与对账

锁定不能永久存在。

过期工作进程的职责：

  • 按时间桶扫描或消费座位锁定过期任务。
  • 将活跃的已过期锁定转换为 expired.
  • 将座位释放回 available 如果结账尚未进行。
  • 发送库存事件以刷新投影。

避免单次大扫描：

  • 按以下条件将锁定存储在时间桶中 expires_at.
  • 使用延迟队列或时间轮风格的调度器。
  • 使过期操作幂等，因为工作进程可能崩溃并重试。

对账作业：

  • 比较标记为 sold 与已签发的门票。
  • 将已付款订单与支付服务商的清算记录进行对比。
  • 将活跃锁定与过期截止时间进行对比。
  • 从库存事件或当前座位记录重建可用性投影。
  • 将不一致的订单隔离，供人工审核。

多区域策略

第一个版本中，每个活动使用一个主写入区域。

原因：

  • 不变量是每个座位的正确性。
  • 针对同一活动的多主写入会产生冲突解决风险。
  • 大多数热门活动天然与场馆和销售区域绑定。

设计：

  • 将活动的所有锁定和结账写入路由到其所属区域。
  • 在全球范围内复制读投影以供浏览。
  • 将 CDN 和搜索功能部署在靠近用户的位置。
  • 对权威区域使用主动-被动故障转移，并明确设定恢复点目标和恢复时间目标。

如果面试官要求主动-主动：

  • 将座位分区分配给各区域所有者，而非所有区域写入所有座位。
  • 或为大众入场预先按区域分配库存切片。
  • 不要在事后合并冲突的座位销售；避免制造冲突。

机器人防护与公平性

机器人防护需要在库存操作之前执行。

信号：

  • 账户年龄与认证状态。
  • 设备指纹与会话历史。
  • IP 信誉与代理检测。
  • 请求频率与导航模式。
  • 支付方式复用情况。
  • 队列令牌完整性。
  • 每位用户、每个家庭、每张银行卡或每台设备的购买限额。

控制措施：

  • 在边缘节点和网关进行限流。
  • 仅在风险较高时使用 CAPTCHA 或挑战验证。
  • 签名候票室令牌。
  • 每个活动的购买上限。
  • 在门票转让或转售前进行欺诈审查。

权衡：

  • 激进的机器人管控可能误拦合法用户。
  • 薄弱的机器人管控会让自动化买家耗尽稀缺库存。
  • 实际的解决方案是渐进式摩擦：低风险用户快速通过，可疑用户接受更多验证。

故障模式与缓解措施

故障模式 缓解措施 监控指标
同一座位被售出两次。 对座位状态进行条件写入，使用显式状态机、幂等性键及对账告警。 状态转换冲突率，重复 event_id + seat_id 已售记录、门票签发不匹配。
座位图显示过期的可用性信息。 将可用性视为投影，使用 TTL/版本号，在锁定时重新校验。 因 UI 过期、投影延迟、缓存版本偏差导致锁定失败。
支付服务商已向用户扣款后支付超时。 使用支付幂等性键、pending/unknown 状态，在释放座位前进行服务商对账。 未知支付时长、服务商不一致数量、延迟转换率。
结账进行中时锁定过期。 使用结账状态转换或带有隔离令牌的短暂延期。 结账后过期冲突、已过期锁定的转换尝试。
候票室被绕过，导致库存过载。 对热门活动的订票 API 要求持有签名的准入令牌。 无有效令牌的请求、库存分片饱和、准入速率与锁定延迟的关系。
过期工作进程处理落后。 按时间桶划分的过期队列、幂等工作进程、对过期延迟发出告警。 已过期的活跃锁定、过期积压时长、人为降低的可用性。
事件流丢失或延迟更新。 持久化 outbox 模式、从审计日志重放、重建投影。 Outbox 延迟、消费者延迟、投影重建时间。
机器人攻击导致登录或队列被淹没。 边缘限流、WAF、设备信誉评分、渐进式挑战验证。 机器人评分分布、挑战率、被拦截流量、误报率。
主写入区域故障。 活动所有权故障转移、故障转移期间冻结热门锁定、从权威日志恢复。 故障转移时长、写入不可用时间、复制延迟。

运维指标

正确性指标：

  • 重复售出的座位数量应为零。
  • 已付款但无门票的订单数量。
  • 有门票但无已付款订单的数量。
  • 超过过期时间的活跃锁定数量。
  • 支付未知状态的持续时间。

流量与容量指标：

  • 候票室队列大小与放行速率。
  • 按原因划分的座位锁定成功/失败率。
  • 库存分片 p95 和 p99 延迟。
  • 条件写入冲突率。
  • 结账转化率。
  • 可用性投影延迟。

业务与信任指标：

  • 机器人挑战率与通过率。
  • 购买限额违规尝试次数。
  • 退款/争议率。
  • 每个热门活动的客服联系数量。

需要指出的权衡

  • 候票室降低了即时性，但它保护了系统并提升了公平性。
  • 可用性覆盖层可能存在过期，但最终锁定必须具有权威性。
  • 每个座位的强一致性是必要的；跨所有活动的全局强一致性是不必要的。
  • 短暂锁定减少了库存占用，但增加了结账压力和用户挫败感。
  • 较长的锁定能提高结账成功率，但会使库存看起来长时间不可用。
  • 每个活动使用单一写入区域简化了正确性，但对远程用户增加了延迟。
  • 多主写入听起来很吸引人，但对于指定座位而言非常危险，除非对所有权进行分区。

常见陷阱与更优做法

陷阱 更优做法
将座位可用性缓存视为真相来源。 说明缓存只是提示；锁定服务执行权威的条件转换。
忽略候票室。 使用准入控制，确保热门活动的需求不会压垮库存服务。
声明支付成功后立即售出座位，而不处理未知状态。 将支付建模为具有 pending 和对账状态的幂等状态机。
对一个活动使用全局单一锁。 按分区或座位键进行分区；仅协调相关座位。
遗忘大众入场模式。 使用库存桶和原子预订计数器代替座位记录。
假设消息恰好被消费一次。 使用持久化 outbox、幂等消费者和可重放的审计事件。

面试用语

对于 Ticketmaster，我不会将座位图作为真相来源。座位图是一个快速投影。权威操作是从可用到已锁定、再从已锁定到已售的条件转换，并围绕支付进行幂等处理和对账。

2. 模拟面试

面试官： 设计 Ticketmaster。

候选人： 我将围绕稀缺库存和突发流量控制来进行设计。核心不变量是一个指定座位最多只能售出一次。浏览和座位图视图可以是最终一致的，但锁定、结账和门票签发需要权威性的状态转换。

面试官： 主要需求是什么？

候选人： 用户应能搜索活动、查看活动详情和座位图、锁定座位、付款并收到门票。主办方负责创建活动和库存。非功能性需求上，系统必须能承受销售开始时的巨大突发流量，防止重复销售，安全处理支付重试，并为订单和门票提供可审计性。

面试官： 您的高层设计是什么？

候选人： 我会将系统分为以读为主的发现路径和以正确性为关键的预订路径。搜索、活动详情和座位图几何信息由索引、缓存和 CDN 提供服务。对于热门活动，用户进入候票室。一旦被放行，他们可以调用座位锁定服务，该服务对权威库存存储执行条件写入。结账阶段验证锁定、幂等地处理支付、将座位转换为已售状态，并通过账本签发门票。

面试官： 您如何防止重复销售？

候选人： 每个指定座位都有一个状态机：可用、已锁定、已售、已锁定（不可售）和已过期。从可用到已锁定的转换是对权威座位记录的条件写入。如果两个用户争抢同一座位，只有一个条件写入会成功。从已锁定到已售的转换也会检查锁定所有权、锁定过期时间和订单幂等性。

面试官： 如果用户重试锁定请求会怎样？

候选人： 客户端发送幂等性键。如果第一次请求成功但响应丢失，重试将返回相同的锁定结果。如果第一次请求因座位不可用而失败，该键对应的重试也返回相同的失败结果。这防止了重复锁定，并使客户端重试变得安全。

面试官： 候票室是如何工作的？

候选人： 用户在销售开始前进入候票室。系统分配签名队列令牌。销售开始时，准入控制器以库存服务能够处理的速率放行用户。热门活动的订票 API 需要绑定用户、活动和过期时间的签名准入令牌。如果库存延迟上升，候票室会降低放行速率。

面试官： 过期的座位图怎么办？

候选人： 这是预期行为。可用性覆盖层是投影，可能存在延迟。UI 可能显示某个座位可用，但锁定服务仍会执行权威检查。如果锁定失败，客户端会刷新该分区的可用性信息。这样在不牺牲正确性的前提下实现了低延迟浏览。

面试官： 支付成功但您的服务超时了，会发生什么？

候选人： 这是支付未知状态。我会使用支付幂等性键，并将订单标记为 pending 或 payment_unknown。我不会立即释放座位。对账工作进程向服务商查询最终支付状态。如果支付成功，我们将锁定转换为已售并签发门票。如果支付失败，我们释放或过期该锁定。

面试官： 锁定是如何过期的？

候选人： 锁定有一个较短的 TTL，例如两到十分钟。我们按过期时间桶或延迟队列存储锁定。过期工作进程幂等地将活跃的已过期锁定转换为过期状态，并将座位释放回可用状态。我们监控已过期的活跃锁定和过期积压时长，因为过期锁定会人为减少库存。

面试官： 您如何为一场极受欢迎的活动进行扩展？

候选人： 我会将该活动隔离到专用容量中，预先计算其活动页面和座位图，并对大型场馆按分区拆分库存。候票室根据库存分片延迟控制准入。搜索、推荐、分析和通知与订票权威路径进行舱壁隔离。

面试官： 您如何处理多区域？

候选人： 我会对锁定和结账使用每个活动一个所有者写入区域。读投影可以全球复制。对于主动-主动的需求，我会按分区划分所有权或预先为大众入场分配库存切片。我会避免让两个区域写入同一指定座位。

面试官： 您会监控什么？

候选人： 首先是正确性：重复售出的座位、有支付无门票的订单、有门票无支付的订单、超过过期时间的活跃锁定，以及支付未知状态的时长。然后是容量：候票室放行速率、库存 p99 延迟、条件写入冲突、投影延迟和结账转化率。

面试官： 总结设计。

候选人： 该设计将快速但可能过期的发现功能与权威性的预订功能分离。候票室控制突发流量。锁定服务负责可用到已锁定的转换，结账服务负责已锁定到已售的转换，门票账本记录已签发的门票。幂等性、显式状态转换、过期工作进程和对账使系统在重试、超时和部分故障下保持正确。

3. 练习备注

五分钟版本

  1. 明确：指定座位、热门销售、禁止重复销售、需要结账。
  2. 状态不变量：每个座位最多售出一次。
  3. 将读取路径与权威路径分离。
  4. 在座位占用服务之前绘制候场室流程。
  5. 绘制座位状态机。
  6. 解释条件写入与幂等性。
  7. 解释支付未知状态与对账。
  8. 以扩展性与可观测性收尾。

一分钟结尾陈述

Ticketmaster 是一个稀缺库存系统。我会缓存活动发现和座位图视图，但将占用和销售保存在权威的库存状态机中。候场室保护热门活动，条件写入防止重复销售，结账围绕支付实现幂等，对账修复未知状态。主要权衡是用户可能看到过期的可用性，但最终预订保持正确。

最佳面试用语

  • "可用性展示是提示；占用创建才是权威。"
  • "我只需要对每个座位或每个库存桶进行强一致性，而非对整个平台全局强一致性。"
  • "候场室不仅仅是用户体验，它是库存写入路径的准入控制。"
  • "支付超时是未知状态问题，因此在释放稀缺库存之前我需要进行对账。"
  • "每个派生视图都应可从库存和订单事件中重建。"
