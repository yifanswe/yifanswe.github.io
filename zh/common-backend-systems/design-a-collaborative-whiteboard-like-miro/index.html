1. 基础知识与解题思路

本题考察点

Miro 是在无限画布上进行实时协同绘图和图表制作的工具。难点在于将 Google Docs （并发编辑收敛）与空间画布语义相结合：众多图形对象、光标存在感知、缩放级别以及跨用户的撤销操作。

其不变量如下：

任意用户执行的每个操作（创建、移动、编辑、删除图形）最终都会出现在同一白板的其他用户视图中，收敛至相同的规范状态；并发操作以确定性方式合并；光标和选中状态对其他用户实时可见；即使白板对象增长至数千个，编辑操作也不会变慢。

在 ICT5 / L6+ 级别，面试官考察的是你能否基于图形图 (shape graph) 数据模型设计基于 CRDT 或 OT 的收敛方案，处理实时存在感知、大型白板的高效渲染、跨用户的撤销/重做，并针对白板分片和扩展进行推理。

60 秒面试框架

口述版本：

我会围绕每个白板的操作日志设计 Miro，并在图形图上采用 CRDT 风格的合并语义。每个白板有一个会话服务器，对来自已连接客户端的操作进行排序并广播给所有节点。光标和存在感知是临时的、独立的通道。关键设计决策包括：空间对象的 CRDT 设计、每个白板的实时扩展、跨用户的撤销语义以及大型白板的渲染。

首先明确需求

功能性需求：

  • 多用户在共享画布上协作。
  • 创建、移动、调整大小、删除图形。
  • 文本、自由手绘、便利贴、框架。
  • 实时查看其他用户的光标和选中状态。
  • 撤销/重做。
  • 版本/历史记录。
  • 带权限控制的白板共享。
  • 锚定到对象的评论。

非功能性需求：

  • 协作者操作延迟低于 200ms。
  • 包含数千个图形的白板。
  • 离线编辑在重新连接时同步。
  • 权限强制执行。
  • 多地区。
  • 白板永久持久化。

以下内容超出范围，除非被特别询问：

  • 实时视频/音频。
  • AI 辅助功能。
  • 连接线/图表智能化。

规模假设

领域 假设
白板数量 数百万。
每个白板的并发编辑人数 典型 1-50 人；最多可达数百人。
每个白板的图形数量 数千个。
操作数 每个白板峰值每秒数百次。
Latency（延迟） 协作者操作延迟 < 200ms。
每个白板的存储空间 KB 至 MB。

瓶颈：每个白板的会话服务器（类似 Google Docs），以及客户端渲染大型白板。

图形的 CRDT 设计

图形是具有属性的对象；操作修改属性。

CRDT 设计选择：

  • 每个图形是一个按属性使用 LWW 的对象： 创建图形，设置属性。对同一属性的并发更新通过时间戳 + 平局决胜规则选出胜者。
  • 位置使用 LWW 或计数器： 当两个用户拖动同一图形时，最终位置采用最后写入者胜出。中间光标位置通过存在感知通道可见。
  • Z 轴顺序使用分数索引： 每个图形有一个有理数 z-index。在两个图形之间插入时使用分数值，避免重排时的冲突。
  • 删除操作使用墓碑标记： 被删除的图形标记为墓碑；未看到该删除操作的后续修改将被忽略。

对于自由手绘：每个笔划是独立的图形；并发笔划不会产生冲突。

面试用语：

白板图形的 CRDT 设计主要是按属性的 LWW 加分数索引（用于排序）加墓碑标记（用于删除）。Yjs 是实现此方案的常用库；Automerge 是另一种选择。

同步协议

类似 Google Docs：

  1. 客户端打开白板；加载最新快照及快照之后的操作记录。
  2. 构建本地 CRDT 状态。
  3. 客户端做出变更；发送带有当前状态向量标签的操作。
  4. 将操作发送给会话服务器。
  5. 服务器追加到操作日志；广播给所有节点（包括发起者）。
  6. 各节点将操作应用到其 CRDT 状态。

离线场景：

  • 客户端在断连期间缓冲操作。
  • 重新连接后，发送缓冲的操作；接收错过的操作。
  • CRDT 合并保证无论操作顺序如何都能收敛。

会话服务器

每个白板在一台会话服务器上运行一个主节点。

  • 在内存中维护当前 CRDT 状态。
  • 将操作持久化到操作日志。
  • 向已连接的客户端广播。
  • 是操作排序的权威节点。

分片：多个白板分布在多台会话服务器上。负载均衡器将白板分配到服务器，保持粘性。

对于超热门白板（100+ 并发编辑者）：单一会话主节点是瓶颈。缓解方案：垂直扩容；或对超大房间接受延迟惩罚。

快照

打开白板时重放数千条操作记录很慢。定期执行：

  • 将当前状态物化为快照。
  • 持久化快照。
  • 打开时，加载快照及之后的操作日志。

快照频率：每 N 次操作或每 M 分钟一次。

存在感：光标与选区

实时、短暂，通过 WebSocket 广播，不持久化。

  • 每个用户在移动时每约 100ms 广播一次光标位置。
  • 每个用户广播选区（哪些图形被选中）。
  • 其他客户端渲染带有用户名标签的彩色光标。

不持久化；重启后光标信息丢失。

撤销 / 重做

在多用户场景中较为复杂。可选方案：

  • 按用户撤销栈： 每个用户仅撤销自己的操作，不影响其他用户的操作，用户体验最直观。
  • 全局撤销栈： 任何人都可以撤销最近一次操作，容易引起混乱。

按用户撤销是标准做法。具体实现：- 每个用户在本地维护一个自己操作的栈。- 撤销时发出一个反向操作，由 CRDT 执行。

渲染大型画板

包含 10,000 个图形的画板无法在每一帧都全量重绘。

客户端优化：- 视口裁剪：仅渲染视野内的图形。- LOD（细节层次）：简化屏幕外或缩小后的图形。- WebGL / canvas 批量渲染。- 客户端空间索引（R-tree 或四叉树）。

服务端：只发送操作，而非完整状态。快照带宽受快照大小限制。

权限

按画板 ACL：用户/组拥有不同角色（查看、评论、编辑、管理员）。

在会话连接时及每次操作时检查，并在 session 服务器缓存。

扩展与分区

数据 分区策略
白板数量 board_id 哈希值
操作日志 board_id 哈希值
快照 board_id 哈希值
会话 按画板粘性
存在感知 在 session 服务器内存中
权限 按画板

一致性模型

层级 一致性 原因
通过 CRDT 管理画板状态 最终一致（收敛） 并发编辑自动合并。
操作日志 按画板强一致 持久性。
存在感知 最终一致（短暂） 正常。
权限 强一致性 安全性。

多区域策略

  • 每块画板都有一个归属区域。
  • Session 服务器位于归属区域。
  • 远程客户端通过 WebSocket 连接，接受跨区域延迟。
  • 快照和操作日志被复制以支持灾难恢复。

故障模式与缓解措施

故障模式 缓解措施 监控指标
Session 服务器故障。 副本接管；客户端重连；从日志回放。 会话重连。
并发编辑产生冲突。 CRDT 合并收敛。 收敛失败（理论上应为零）。
大型画板加载缓慢。 快照避免操作回放；懒加载远处的图形。 初始加载延迟。
存在感风暴。 限制广播速率。 存在感 QPS。
离线用户恢复时有大量操作待同步。 有界缓冲区；分块同步。 重连时的待处理操作数。
会话中途撤销权限。 每次操作时重新检查；踢出用户。 权限撤销传播。
拥有 100 名编辑者的热门画板。 垂直扩展；降低用户体验（超过 N 人后设为只读）。 按画板 QPS。
操作日志损坏。 CRC 校验 + 复制。 完整性违规。
渲染性能下降。 视口裁剪；LOD。 客户端 FPS。
跨区域复制延迟。 异步；有界。 延迟。

运维指标

正确性：

  • 收敛失败（理论上应为零）。
  • 重连时操作丢失。
  • 尝试绕过权限。

容量：

  • 每台服务器的会话数。
  • 每块画板每秒操作数。
  • 快照生成速率。

业务层面：

  • 活跃画板数，每块画板的编辑者数。
  • 平均画板大小。

需要指出的权衡

  • CRDT 自动收敛，但比 OT 使用更多元数据。
  • 按用户撤销直观，但无法撤销跨用户冲突。
  • 快照频率：越频繁 = 加载越快，存储越多。
  • 按画板粘性 session 服务器限制了单板扩展性，但简化了设计。
  • 激进的存在感广播感觉实时，但消耗带宽。
  • 跨区域按画板部署较难；通常采用单一归属区域。

常见陷阱与更优做法

陷阱 更优做法
对每个属性采用最后写入优先（LWW）。 使用分数索引实现顺序的 CRDT。
每次操作都发送完整画板状态。 只发送操作；定期生成快照。
持久化存在感信息。 通过 WebSocket 短暂存在。
全局撤销栈。 按用户撤销。
全量渲染整块画板。 视口裁剪 + LOD。
不支持离线。 缓冲操作；重连时 CRDT 合并。

面试官可能深入探讨的方向

深入方向 优秀回答要点
CRDT 设计 每属性 LWW；分数索引；墓碑标记。
同步协议 WebSocket + 操作日志 + 快照。
存在感知 通过 WebSocket 短暂存在；限速。
撤销 / 重做 按用户栈；反向操作。

多区域 按画板归属区域。

Staff 级别回答检查清单

  • 使用 LWW + 分数索引 + 墓碑标记的 CRDT 图形图。
  • 带有操作日志的按画板 session 服务器。
  • WebSocket 同步；快照用于快速加载。
  • 通过反向操作实现按用户撤销。
  • 短暂存在感。
  • 权限服务。
  • 客户端视口裁剪和 LOD。
  • 按画板归属区域的多区域部署。

面试用语

协作白板是一个带有按画板 session 服务器的 CRDT 图形图。并发编辑通过每属性 LWW、分数索引保证顺序、墓碑标记处理删除来收敛。存在感是短暂的。按用户撤销体验直观。多区域采用按画板归属区域。最难的正确性问题是并发编辑下的收敛；最难的用户体验问题是流畅渲染数千个图形。

2. 模拟面试

面试官： 设计 Miro。

候选人： 以 session 服务器为支撑的按画板 CRDT 图形图。客户端通过 WebSocket 连接；操作广播给对等节点；CRDT 收敛合并并发编辑。操作日志持久化操作；定期快照实现快速加载。存在感（光标、选区）是短暂的。不变性在于：无论并发操作的顺序如何，每位编辑者的视图最终都会收敛到相同的状态。

面试官： CRDT — 请说明设计思路。

候选人： 图形图。每个图形是一个包含属性的对象；操作用于创建、修改、删除图形。对于图形属性（位置、大小、颜色），采用带时间戳和决胜因子（作者 ID）的 LWW。对于图形顺序（z-index），采用分数索引——在两个图形之间插入时使用它们之间的有理数值。删除使用墓碑标记；未感知到删除的后续修改将被忽略。对于手绘线条，每个笔画是独立图形，并发笔画不会冲突。Yjs 和 Automerge 是实现此方案的库；自定义实现也很常见。

面试官： 两个用户同时拖动同一个图形。

候选人： 每个用户发出带时间戳的位置更新操作。CRDT 合并：对最终位置取最后写入优先。中间位置仅通过存在感可见（实时光标和"用户正在移动 X"提示），不体现在持久化状态中。最终停留位置由时间戳最新的操作决定。两个用户看到相同的最终位置。

面试官： 同步协议。

候选人： 客户端通过 WebSocket 连接到画板的 session 服务器。服务器返回最新快照及快照之后的操作。客户端构建本地 CRDT 状态。用户操作发出 op；客户端乐观应用并发送给服务器。服务器追加到操作日志并广播给所有对等节点。对等节点将操作应用到各自的 CRDT 状态。收敛由 CRDT 语义保证。

面试官： 离线编辑。

候选人： 客户端在断开连接时缓冲操作。重连后，发送缓冲操作；从服务器接收错过的操作。CRDT 合并处理所有顺序——离线产生的并发操作与在线操作合并时不会冲突。客户端本地状态最终追上并收敛。

面试官： 一块画板有 10,000 个图形，初始加载时间如何？

候选人： 快照避免了回放完整操作日志。下载最新快照后应用快照之后的操作。快照通常为 MB 级别。渲染时，客户端使用视口裁剪——只渲染可见区域内的图形。LOD：缩小后的视图显示简化图形。空间索引（R-tree 或四叉树）加速拾取和视口查询。

面试官： 多用户场景下的撤销 / 重做？

候选人： 按用户撤销。每个用户有一个本地的自己操作的栈。撤销时发出反向操作（移回图形；恢复已删除的图形等），由 CRDT 执行。其他用户的操作不受影响。任何用户都能撤销任意操作的全局撤销既令人困惑又极少使用。

面试官： 存在感 — 50 个用户在画板上移动光标。

候选人： 短暂。每个客户端在移动时每 100ms 广播一次光标位置；选区在变更时广播。服务器通过 WebSocket 转发，不持久化。对每个用户的广播速率进行限流以控制带宽。50 个用户意味着每块画板每秒 500 次光标更新，尚在可控范围内。

面试官： 权限？

候选人： 按画板 ACL——用户/组拥有不同角色（查看、评论、编辑、管理员）。在会话连接时检查，并在 session 服务器缓存。每次操作的重新验证也会缓存。撤销权限时，session 服务器踢出该用户。仅具有查看权限的用户发出的编辑操作将被拒绝。

面试官： 大型画板，100 名并发编辑者。

候选人： 单个 session 服务器按顺序处理操作，CPU 和内存限制了并发上限。垂直扩展（更强大的服务器）可支持 50-100 名编辑者。超出后性能下降；我们可能会禁用部分功能，或将画板设为"超过 N 名查看者后只读"的用户体验模式。将单块画板分片到多台服务器很困难，因为顺序是按画板维护的。

面试官： 多区域？

候选人： 按画板归属区域。Session 服务器位于归属区域。远程编辑者通过 WebSocket 连接，接受跨区域延迟（50-200ms）。操作日志和快照在全球范围复制以支持灾难恢复。我们不会为同一块画板在每个区域运行活跃的 session 服务器，因为这会破坏序列化不变性。

面试官： 总结。

候选人： 每属性 LWW、分数索引保证顺序、墓碑标记处理删除的 CRDT 图形图。按画板 session 服务器对操作排序并通过 WebSocket 广播。操作日志持久化；快照实现快速加载。存在感短暂。通过反向操作实现按用户撤销。客户端对大型画板使用视口裁剪和 LOD。多区域按画板归属区域部署。最难的正确性问题是并发编辑下的收敛；最难的用户体验问题是流畅渲染大型画板。

3. 练习备注

五分钟版本

  1. 澄清：实时多用户、图形、撤销、规模。
  2. 不变性：收敛状态；可见光标。
  3. CRDT 图形图；LWW + 分数索引 + 墓碑标记。
  4. 按画板 session 服务器；WebSocket 同步。
  5. 操作日志 + 定期快照。
  6. 短暂存在感。
  7. 通过反向操作实现按用户撤销。
  8. 视口裁剪用于渲染。

一分钟结尾陈述

协作白板是一个带有按画板 session 服务器的 CRDT 图形图。并发编辑通过每属性 LWW、分数索引保证顺序、墓碑标记处理删除来收敛。WebSocket 同步；操作日志持久化；快照实现快速加载。存在感是短暂的。通过反向操作实现按用户撤销。客户端使用视口裁剪和 LOD 进行渲染。多区域按画板归属区域部署。最难的正确性问题是收敛；最难的用户体验问题是大型画板渲染。
