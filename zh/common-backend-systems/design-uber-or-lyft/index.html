1. 基础知识与解题思路

本题考察点

打车平台实时将乘客与司机匹配。难点在于地理空间调度问题：在数百万个移动位置中找到合适的司机，原子性地完成行程交接使两名乘客不会得到同一司机，然后跟踪行程至完成。

其不变量如下：

司机最多同时分配一次进行中的行程；每个乘客请求最多匹配一名接受的司机；匹配后，双方通过行程全生命周期（接驾途中、行程中、已完成）对状态达成一致；位置和 ETA 更新及时；费用基于行程已记录事件确定性计算。

在 ICT5 / L6+ 级别，面试官考察的是您是否能建模地理空间索引、在竞争条件下设计实时匹配、原子性处理司机状态转换、将行程生命周期管理为状态机，以及对价格激增和 ETA 预测进行推理。

60 秒面试框架

口述版本：

我将 Uber 设计为三个核心系统：可用司机的实时位置索引（地理空间）、为每个请求找到并预留合适司机的匹配服务，以及拥有从请求到完成全状态机的行程生命周期服务。关键决策包括地理空间分片、原子司机分配（防止双重预订）、行程状态转换、价格激增和 ETA 预测。

首先明确需求

功能性需求：

  • 乘客请求从 A 到 B 的行程。
  • 系统匹配一名司机；乘客看到 ETA 和司机信息。
  • 司机收到通知，接受或拒绝；拒绝时系统尝试下一位。
  • 行程中乘客和司机的实时位置更新。
  • 行程生命周期：requested → matched → en_route → in_trip → completed。
  • 基于距离、时间、价格激增的费用计算。
  • 乘客或司机取消。
  • 行程后支付。
  • 评分与评价。

非功能性需求：

  • 匹配延迟在几秒内。
  • 高频位置更新（每位司机每 4–10 秒一次）。
  • 能应对突发激增（活动结束、恶劣天气、机场）。
  • 不允许司机双重预订。
  • 精确的行程记录用于计费。
  • 多区域（每个城市基本上相互独立）。

以下内容超出范围，除非被特别询问：

  • 拼车/共享出行匹配算法。
  • 长途物流（Uber Freight）。
  • 外卖配送（Uber Eats；类似但不同）。

瓶颈在于高并发下的空间查询以及原子性匹配交接。

地理空间索引

司机不断移动，索引必须快速更新。

方案：以适当分辨率将世界划分为格元（S2、H3 或 geohash）。

  • H3 分辨率 9：约 100 米的六边形格元。
  • 每个格元存储一组可用的 driver_id。
  • 位置更新时，若司机所在格元发生变化，则从旧格元移除，加入新格元。
  • 每个格元在服务中驻留内存；按 cell_id 在节点间分片。

对于"查找点 P 半径 R 内的司机"：

  • 计算覆盖点 P 周围半径 R 圆盘的格元集合。
  • 并行查询每个格元的可用司机集合。
  • 按精确距离和车辆兼容性过滤。
  • 按得分排名（距离、预计到达时间、司机评分）。

面试用语：

地理索引的本质是"每个格元中谁可用"。位置更新即格元成员变更，空间查询即格元扫描加精确距离过滤。

匹配算法

当乘客发起请求时：

  1. 计算起点周围的候选格元（例如，1-2 千米半径）。
  2. 查询地理索引以获取可用司机。
  3. 对候选人评分：预计到达时间、司机评分、车型匹配度、近期拒单率。
  4. 选取得分最高的候选人。
  5. 原子性地尝试预留该司机——对以下字段进行条件更新 Driver.status = online_available, current_trip_id = null.
  6. 若预留成功，向司机发送行程邀约。
  7. 若司机接受，将行程状态转换为 matched.
  8. 若司机拒绝或超时，释放预留并选取下一候选人。

原子预留是关键：

  • 在以下字段上使用 CAS Driver.status 。 Driver.current_trip_id.
  • 一次只能有一个行程持有一名司机。
  • 司机的"邀约待确认"是一个短暂持有（约 15-30 秒）。

面试用语：

匹配是对司机的原子条件状态转换。由于只有一个 CAS 会成功，不会有两个行程竞争获取同一名司机。

行程状态机

requested -> matched -> en_route -> arrived -> in_trip -> completed
     |             |          |          |          |
     +-> canceled  +-> driver_canceled   +-> canceled

状态转换：

状态转换 Trigger 影响
requested -> matched 司机接受。 设置 driver_id；锁定司机。
matched -> en_route 司机驶向乘客。 更新显示。
en_route -> arrived 司机到达上车点。 启动等待计时器。
arrived -> in_trip 司机开始行程。 启动计费。
in_trip -> completed 司机结束行程。 计算费用；扣款。
Any -> canceled 取消操作。 释放司机；可能收取取消费。

每次转换均通过 CAS 操作 (trip_id, from_state).

动态定价

按格元统计供需的热力图。

  • 需求 = 每格元每分钟的未接单请求数。
  • 供给 = 每格元的可用司机数。
  • 倍率 = 供需比的函数。
  • 每 30 秒至数分钟更新一次。
  • 客户端缓存用于展示。

请求时的动态定价值将记录在行程中，并用于费用计算。

预计到达时间预测

两级架构：

  • 路网预计到达时间： 基于路网和当前交通状况。
  • ML 预计到达时间： 基于历史规律、司机行为、一天中的时段进行修正。

常用起终点对预计算；任意起终点按需计算。

使用方：

  • 匹配时间：司机到上车点的预计到达时间，用于评分。
  • 行程预览：向乘客展示预计到达时间。
  • 途中：实时更新预计到达时间。

位置更新流水线

司机每 4-10 秒发送一次位置。若有 100 万名司机在线，则每秒需处理 10 万-25 万次更新。

  • 每名司机的位置更新按 driver_id 分区。
  • 流式处理流水线近实时地更新地理索引。
  • 更新允许轻微延迟（几秒钟）。
  • 我们不会永久存储每次位置更新；仅保留已完成行程的路线折线。

对于实时显示：

  • 乘客的应用在 en_route 和 in_trip 阶段订阅司机位置。
  • 服务器推送位置的频率低于司机上报频率（例如每 2-5 秒一次）。

扩展与分区

数据 分区键 原因
司机状态 driver_id 哈希值 每位司机的 CAS。
地理索引 cell_id 哈希值 共置格元成员关系。
行程 trip_id 哈希，按乘客/司机建立索引 按行程写入；按 ID 查找。
位置更新 driver_id 哈希值 流分区。
动态定价热力图 cell_id 哈希值 按格元计算。

按城市分片：

  • 各城市在运营上基本独立。
  • 旧金山的司机不会与纽约的乘客匹配。
  • 每个城市拥有自己的地理索引、派单服务，通常还有独立的数据中心。

一致性模型

层级 一致性 原因
司机状态 强一致性（CAS） 不得出现重复预订。
行程状态 强一致性（CAS） 单一权威状态。
地理索引 最终一致性（秒级延迟） 对于匹配可接受。
动态定价热力图 最终一致 定期更新。
行程历史 最终一致 报表。

多区域策略

  • 城市是天然的分片单元。
  • 每个城市的地理索引和派单服务运行在最近的区域。
  • 司机状态和行程状态在同一区域。
  • 跨城市交接（如长途机场行程）较为少见，通过专门逻辑处理。
  • 全局服务（支付、用户账户）在全球范围内复制。

故障模式与缓解措施

故障模式 缓解措施 监控指标
两名乘客同时请求同一名司机。 在司机状态上进行 CAS；只有一个成功。 条件写冲突率。
行程中司机应用断线。 基于乘客上报继续行程；重连后进行数据对账。 行程中司机断线次数。
位置更新丢失。 客户端平滑处理；在下次更新时恢复。 更新间隔分布。
场馆活动结束时的动态定价飙升。 预先分配派单容量；动态定价调节需求。 每格元供需比。
司机未响应匹配邀约（司机不在手机旁）。 邀约 15 秒后过期；重新派单。 邀约过期率。
司机接单后未出发。 有时间限制；乘客在超出阈值后可免费取消。 接单但未出发的次数。
虚假位置欺骗。 GPS 合理性检查；行为分析。 每名司机的异常分数。
地理索引热格元。 将格元子分片；或接受现状并扩展节点。 每区域 QPS。
行程卡在 in_trip. 对账 worker；行程最大持续时长。 行程时长 > N 小时。
行程结束后支付失败。 行程已记录；支付重试；复用支付系统。 支付失败率。

运维指标

正确性：

  • 重复预订尝试（必须零成功冲突）。
  • 行程状态不一致事件。
  • 前 N 秒内的取消率（用户体验问题信号）。

容量：

  • 每秒每区域的位置更新次数。
  • 匹配延迟 p99。
  • 地理索引节点的 CPU 和内存。

业务层面：

  • 活跃行程数、活跃司机数、每城市每分钟请求数。
  • 匹配成功率。
  • 动态定价倍率分布。
  • 平均等待时间。

需要指出的权衡

  • 更大的地理格元使空间查询成本更低，但会降低匹配精度。
  • 更频繁的位置更新能提供实时预计到达时间，但会消耗更多电量和带宽。
  • 更短的邀约超时能加快匹配，但会导致更多重新派单。
  • 严格的司机状态 CAS 能防止重复预订，但会限制每名司机的吞吐量。
  • 按城市分片很自然，但跨城市行程会变得棘手。
  • 激进的峰时定价抑制需求，但会引发用户投诉。

常见陷阱与更优做法

陷阱 更优做法
全局位置索引。 按格元分片的按城市地理索引。
线性扫描所有司机。 基于 H3/S2 格元的空间索引。
先来先得的派单。 基于评分的匹配；CAS 预留。
无邀约超时。 短时限邀约，失败后回退至下一候选人。
为每次位置更新持久化存储。 仅流式处理，保留时间有限；行程结束时持久化路线折线。
以简单滑块实现动态定价。 按格元频繁更新热力图。

面试官可能深入探讨的方向

深入方向 优秀回答要点
地理索引 H3 格元；格元级司机集合；空间扫描。
原子匹配 司机状态 CAS；邀约与接受握手。

Staff 级别回答检查清单

  • 基于格元的地理空间索引；按城市分片。
  • 通过 CAS 实现原子司机预留。
  • 具有显式转换的行程状态机。
  • 按单元格热力图计算峰时溢价。
  • 通过路线规划 + ML 预估 ETA。
  • 位置更新管道流式处理。
  • 按城市进行多区域部署。
  • 对卡住的行程进行对账处理。

面试用语

Uber 本质上是一个叠加在行程状态机之上的地理空间调度问题。地理索引使"查找附近司机"的开销很低；对司机状态的原子 CAS 操作保证了不会重复派单；行程是从请求到完成的状态机。峰时溢价和 ETA 作为辅助服务并行运行。

2. 模拟面试

面试官： 设计 Uber。

候选人： 我将设计三个核心系统：第一，基于 H3 单元格组织的可用司机实时地理空间索引；第二，查询索引并以原子方式锁定目标司机的匹配服务；第三，管理从请求到完成整个生命周期的行程状态机。不变量是：同一时刻每位司机最多被分配一个进行中的行程。峰时溢价、ETA 和支付均为辅助服务。

面试官： 如何找到最近的可用司机？

候选人： 将世界按分辨率 9 的 H3 单元格划分——约 100 米的六边形单元格。每个单元格维护一组可用的 driver_id。收到请求后，计算覆盖起点周围小圆盘（1-2 km）的单元格，并行查询每个单元格，获取候选 driver_id，按精确距离和车辆兼容性过滤，再按 ETA 和其他因素评分。地理索引按 cell_id 分片并保存在内存中。

面试官： 两位乘客同时发起打车请求，均获得同一个最近司机，如何防止重复派单？

候选人： 对司机状态进行原子条件更新。评分后，匹配服务执行 UPDATE Driver SET status=on_trip_pending, current_trip_id=? WHERE driver_id=? AND status=online_available AND current_trip_id IS NULL。两个请求中只有一个 CAS 会成功。失败的请求选择次优候选人。这是"用数据库原子性替代分布式锁"的经典做法。

面试官： 司机接受了请求，接下来会发生什么？

候选人： 行程转换到 matched。双方应用程序均收到通知。司机状态变为 on_trip_pending。司机开始导航；乘客看到司机正在靠近。随着司机移动，位置更新通过地理索引和每行程订阅传播。行程状态推进： matched → en_route → arrived → in_trip → completed。每次转换都是对 (trip_id, from_state) 的 CAS 操作。

面试官： 司机在 15 秒内未接受请求。

候选人： 派单超时。我们将司机的 on_trip_pending 状态释放回 online_available。匹配服务选择下一个候选司机并发起派单。我们将司机的拒单率作为未来匹配评分的信号。

面试官： 司机应用在行程中途断线。

候选人： 分两种情况。如果行程处于 in_trip状态，乘客仍可上报进度（乘客端应用发送位置）。行程依据乘客上报继续进行；司机应用重连后进行对账，位置更新恢复。若司机断线时间较长，客服可介入。行程完成可由任一方触发；我们有防止单方面提前完成的安全机制。

面试官： 峰时溢价是如何运作的？

候选人： 按单元格热力图计算。单元格内需求量（近期未响应请求数）除以供给量（可用司机数）。当比值超过阈值时，溢价倍数生效。大约每 30 秒更新一次。请求时记录的溢价倍数写入行程记录，用于费用计算。给定记录的溢价值，结果是确定性的。

面试官： 位置更新——1000 万司机每 5 秒上报一次，即每秒 200 万次更新。

候选人： 按 driver_id 分片。流式处理管道（Kafka + 有状态消费者）接收更新并应用到地理索引。若司机跨越单元格边界，从旧单元格中删除并添加到新单元格。地理索引保存在内存中；每个节点负责部分单元格。更新延迟低于 1 秒。我们不持久化每次位置更新；仅保存已完成行程的路线折线。

面试官： ETA？

候选人： 路线引擎基于道路网络和交通数据从当前位置到目的地计算 ETA。ML 模型根据历史规律、时段和司机行为进行调整。匹配评分使用到上车点的 ETA；乘客在 en_route 和 in_trip 阶段看到的是到目的地的 ETA，周期性更新。

面试官： 多区域？

候选人： 城市是天然的分片单元。每个城市拥有独立的地理索引、匹配服务和行程状态——通常部署在最近的数据中心。跨城市切换（如长途机场行程）较少见，通过显式支持处理。支付和用户账户等全局服务在全球复制。

面试官： 一个场馆散场——5 万人同时发起打车请求，怎么办？

候选人： 该单元格变为热点。该单元格对应的地理索引节点可能饱和；我们将热点单元格预先拆分成子分片。峰时溢价生效，抑制需求。司机收到高需求通知后向该区域聚集。对于已知活动，我们也会预先分配调度容量。行程积压通过"您很快将被匹配"的用户界面明确提示，而不是直接报错。

面试官： 行程卡在 in_trip 持续 12 小时。

候选人： 对账工作进程扫描超过阈值时间仍处于非终态的行程。可能的情形：司机完成了行程但未点击完成，或司机应用崩溃。我们根据检测到的运动模式或联系司机自动完成行程。费用根据已记录的事件计算。

面试官： 总结。

候选人： 地理索引使用 H3 单元格，按城市分片。对司机状态的原子 CAS 操作防止重复派单。行程状态机具有明确的转换流程。峰时溢价使用按单元格热力图。ETA 来自路线规划 + ML。位置更新流式处理；地理索引在内存中更新。对卡住的行程进行对账处理。最难的正确性问题是原子匹配；最难的扩展性问题是位置更新的数据洪流。

3. 练习备注

五分钟版本

  1. 明确范围：匹配、实时位置、行程生命周期、峰时溢价。
  2. 不变量：每位司机最多同时进行一个行程。
  3. 地理索引使用 H3 单元格；按城市分片。
  4. 匹配：对候选人评分；原子 CAS 锁定。
  5. 具有显式转换的行程状态机。
  6. 峰时溢价使用按单元格热力图；ETA 来自路线规划 + ML。
  7. 位置更新流式处理；有界保留策略。
  8. 对卡住的行程进行对账处理。

一分钟结尾陈述

Uber 本质上是一个叠加在行程状态机之上的地理空间调度问题。H3 单元格地理索引使附近司机查询开销低；对司机状态的原子 CAS 操作防止重复派单；行程生命周期是由 CAS 转换构成的状态机。峰时溢价和 ETA 作为辅助服务并行运行。城市是天然的分片单元。最难的扩展性问题是位置更新数据洪流；最难的正确性问题是原子匹配。

最佳面试用语

  • "匹配是对司机的原子条件状态转换；只有一个 CAS 能成功。"
  • "地理索引是按单元格维护的成员集合；空间查询即单元格扫描。"
  • "行程生命周期是显式状态转换，每次转换均为一个 CAS 操作。"
  • "峰时溢价是按单元格热力图，而非全局滑块。"
  • "城市是天然的分片单元；跨城市是例外情况。"
