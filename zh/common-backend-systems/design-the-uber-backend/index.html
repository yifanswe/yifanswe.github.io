1. 基础知识与解题思路

本题考察点

"Uber 后端"扩展了 设计 Uber 或 Lyft 为完整的后端平台——不仅仅是调度和匹配，还包括行程生命周期、支付、司机入驻、溢价、ETA 建模，以及支撑打车平台运营的微服务。难点在于广度：每个组件都必须在突发性地理需求和严格的延迟耦合要求下保持稳定。

其不变量如下：

每次行程遵循：请求 → 匹配 → 接驾途中 → 行程中 → 已完成 → 已支付，保证每次行程至多匹配一个司机，从记录事件中确定性计算费用，幂等支付处理，以及可审计的溯源；供需信号（溢价）反映每个城市格子的当前市场状态。

在 ICT5 / L6+ 级别，面试官考察你能否将调度、支付、溢价、ETA、司机入驻和可观测性整合为一个连贯的平台，涵盖分片策略、多服务一致性，以及能捕捉正确性偏差的运营指标。

60 秒面试框架

口述版本：

我会将 Uber 后端设计为分层平台：地理空间调度层用于匹配，行程状态机用于生命周期，支付层用于费用收取和司机结算，溢价引擎生成按格子的热力图，ETA 服务用于预测，以及大量支撑服务（司机入驻、评级、客服）。每个城市是自然的分片单元。关键决策包括调度正确性、行程状态、支付集成、溢价定价，以及微服务设计中广度与深度的权衡。

首先明确需求

功能性需求：

  • 乘客请求打车；匹配到司机。
  • 行程生命周期：请求 → 匹配 → 接驾途中 → 行程中 → 已完成 → 已支付。
  • 费用计算，包含起步价、距离、时间、溢价、过路费、税费。
  • 支付与司机结算。
  • 评级与反馈。
  • 司机入驻（背景审查、车辆审核）。
  • 客服 / 争议处理。
  • 促销与优惠券。
  • 多产品（UberX、Uber Pool、Uber Eats——聚焦核心）。

非功能性需求：

  • 匹配延迟在数秒以内。
  • 部分故障下行程状态转换正确。
  • 端到端支付幂等性。
  • 溢价定价在数分钟内反映市场信号。
  • 多区域；各城市基本独立。
  • 行程、支付、司机工时的可审计性。

以下内容超出范围，除非被特别询问：

  • 自动驾驶 / AV 集成。
  • 长途物流。
  • 地图数据 / 路由引擎内部实现。
  • 司机辅导分析。

行程生命周期（扩展）

新增取消、争议、支付的状态：

requested -> matched -> en_route -> arrived -> in_trip -> completed -> paid
   |             |         |          |          |             |
   v             v         v          v          v             v
canceled    cancelled    cancelled  cancelled  refunded    disputed

每次转换均为对 (trip_id, from_state) 的 CAS 操作，并记录审计日志。

对于支付：

  • completed -> paid：支付已捕获。
  • paid -> disputed：乘客/司机争议已开启；人工处理。
  • paid -> refunded：退款已发起（部分或全额）。

费用计算

费用由已记录事件确定性计算得出：

fare = base + (distance * per_mile) + (duration * per_minute) + tolls
fare = fare * surge_multiplier (locked at request time)
fare = fare + booking_fee - discount
total = fare + taxes
driver_payout = total - platform_fee

行程记录：- 出发地/目的地 - 路线折线（途经点 + 行驶时间）- 请求时锁定的峰时倍率 - 累计通行费 - 已应用折扣

只要有行程记录，任何人都可以重新计算费用。对账任务定期运行。

面试用语：

费用由事件确定性计算：峰时倍率在请求时锁定，行程中累计距离和时间，费用与折扣均被捕获。任何人均可重新计算；对账任务检测偏差。

支付集成

复用支付系统设计：

  1. 行程请求时：预授权预估费用（在支付方式上冻结）。
  2. 完成时：捕获实际费用。
  3. 司机结算：汇总已完成行程收入；按周期结算（每日、每周）。
  4. 退款：状态转换 + 退款流水线。
  5. 争议：冻结资金；人工处理。

关键：支付幂等性。trip_id 用作支付提供商的幂等键。

峰时定价引擎

按格子的供需热力图。

  • 需求：每分钟该格子内近期开放的请求数。
  • 供给：格子内可用司机数。
  • 倍率：供需比的函数（通常为 1.0 到 3.0+）。
  • 每 30 秒至数分钟更新一次。
  • 在客户端缓存用于展示。

请求创建时，当时的峰时值会快照写入行程。后续峰时变化不影响进行中行程的费用。

ETA 服务

两级预测：

  • 路由引擎 根据路网和当前路况计算 ETA。
  • ML 修正 根据历史规律、时段和司机行为进行调整。

用于：- 匹配评分：接驾 ETA 影响司机排名。- 展示：向乘客显示到达目的地的 ETA。- 途中：实时更新。

对常见出发-到达地对预先计算；其余按需计算。

司机入驻

异步工作流：

  1. 司机申请；提交材料（驾照、行驶证、保险、照片）。
  2. 背景调查服务运行（通常通过外部提供商）。
  3. 车辆核验。
  4. 培训材料已确认。
  5. 审核通过后，司机状态切换为 approved；可上线接单。
  6. 定期复查（每年等）。

各步骤可能耗时数天。状态机将状态更新推送给司机。

评分服务

双向。每次行程结束后：

  1. 乘客为司机评分。
  2. 司机为乘客评分。
  3. 每次评分更新汇总分数（滚动平均）。
  4. 低分标记以供审核。
  5. 双方均提交（或窗口到期）前隐藏评分，以防报复。

司机评分影响匹配优先级；乘客评分影响司机接单意愿。

促销活动

基于码或自动应用：

  • 优惠码每用户可兑换 N 次。
  • 符合条件的行程自动应用促销。
  • 原子性兑换： INSERT INTO Redemption(user, promo) IF NOT EXISTS.
  • 折扣在费用计算时应用。

反欺诈：检测滥用模式（相同支付方式、连续 IP）。

一致性模型

层级 一致性 原因
司机状态 强一致性（CAS） 不得出现重复预订。

多区域策略

  • 城市级运营在本地区域。
  • 跨城市行程罕见；通过显式逻辑处理。
  • 全局服务（支付、账户）在全球副本，区域主节点。

故障模式与缓解措施

故障模式 缓解措施 监控指标
重复派单。 对司机状态的 CAS。 冲突率。
行程卡住。 对账 worker；最大持续时长。 行程时长 > N 小时。
行程后支付失败。 重试；人工催收。 支付失败率。
活动期间峰时激增。 预分配调度；定价抑制需求。 每格子供需比。
司机应用行程中断线。 乘客上报继续行程；重连后对账。 行程中断线率。
促销滥用。 反欺诈检测；兑换次数上限。 促销兑换异常。
入驻欺诈。 背景调查；证件核验。 各来源审批率。
跨城市行程。 显式逻辑；更长的费用计算。 跨城市行程数量。
评分操纵。 异常检测；按信任度加权。 评分异常。
区域宕机。 区域内故障转移；跨区域切换罕见。 区域可用性。

运维指标

正确性：

  • 重复派单事件（零）。
  • 行程状态不一致。
  • 支付与行程的偏差。
  • 费用重算与记录总额一致。

容量：

  • 匹配延迟 p99。
  • 行程创建速率。
  • 每秒位置更新次数。

业务层面：

  • 每城市每日行程数。
  • 匹配成功率。
  • 峰时触发频率。
  • 司机利用率。
  • 取消率（乘客、司机）。

需要指出的权衡

  • 单城市分片简化了架构，但跨城市处理较复杂。
  • 更严格的预授权风控会降低匹配速度。
  • 激进的峰时定价抑制需求，但会引发用户投诉。
  • 更频繁的位置更新带来实时 ETA，但消耗更多电量。
  • 严格的司机状态 CAS 限制了每司机的吞吐量。
  • 行程开始时实时捕获支付 vs 行程结束时捕获：延迟 vs 取消友好性。

常见陷阱与更优做法

陷阱 更优做法
单体后端。 微服务，职责清晰。
行程开始时同步支付。 开始时预授权；结束时捕获。
全局峰时调节器。 按格子的热力图。
每次请求使用有状态的匹配服务。 无状态 worker 共享地理索引。
无费用对账。 基于事件的确定性费用；定期重算。
自动信任入驻。 背景调查；证件核验。

  • 调度、行程、支付、峰时、ETA、入驻的微服务。
  • 地理索引司机状态。
  • 基于 CAS 的行程状态机。
  • 复用支付系统设计。

  • 双向评分。

  • 异步入驻工作流。

面试用语

Uber 后端 = 调度 + 行程状态机 + 支付 + 峰时 + 一系列支撑微服务。每个城市独立运行；地理索引使附近查找高效；行程状态机保证正确性；支付和峰时通过确定性事件记录集成。多区域扩展通过增加城市实现，而非全球化服务。

2. 模拟面试

面试官： 设计 Uber 后端。

候选人： 我将搭建平台：带地理空间索引的调度、行程状态机、支付服务、峰时引擎、ETA、入驻、评分、客服支持。每个城市是天然的分片单元。核心不变量是：每次行程最多一位司机、确定性费用计算、幂等支付，以及端到端审计的行程状态转换。

面试官： 从调度和匹配开始。

候选人： 参见 设计 Uber 或 Lyft 深入了解。带 H3 格子的地理索引；按城市分片。对司机状态的原子 CAS 防止重复派单。行程状态机管理生命周期。

面试官： 费用计算是如何工作的？

候选人： 由已记录事件确定性计算。请求时锁定 surge_multiplier。行程中，通过位置更新追踪距离和时长。完成时，费用 = 基础费 + 距离乘以每英里单价 + 时长乘以每分钟单价 + 通行费，再乘以峰时倍率，加收预约费，减去折扣，加税。司机结算金额 = 总额 - 平台服务费。每个组成部分均记录在行程记录中；任何审计方均可重算。

面试官： 支付？

候选人： 复用支付系统设计。请求时预授权预估费用，完成时捕获实际费用。司机收益按周期（每日、每周）通过支付系统聚合结算。退款和争议是行程的状态转换。端到端幂等：trip_id 是幂等键。

面试官： 激增定价？

候选人： 按单元格生成热力图。流式处理管道按单元格追踪需求（近期请求）和供给（可用司机）。每 30 秒计算一次乘数。请求时的激增价格快照写入行程；后续激增变化不影响进行中的行程。积极的激增定价可以抑制需求并激励更多司机上线。

面试官： ETA？

候选人： 路由引擎根据路网和交通状况计算 ETA。ML 模型对历史规律进行修正。用于匹配（到达取客点的 ETA 影响评分）、行程预览（到达目的地的 ETA）以及行程中（实时更新）。热门起终点对（O-D 对）提前预计算。

面试官： 司机入驻流程？

候选人： 异步工作流。司机申请后提交证件（驾照、行驶证、保险、照片）。后台核查服务（第三方提供商）运行。车辆核验。培训材料。状态机逐步推进。 pending → approved → active。每年重新核查。失败情况（犯罪记录、驾照过期）将阻止入驻或触发停用。

面试官： 评分？

候选人： 每次行程后双向评价。乘客给司机评分，司机给乘客评分。按用户聚合。双方均提交（或窗口期到期）前隐藏，以避免报复性评分。低评分标记待审。司机评分影响匹配优先级；乘客评分影响司机接单意愿。

面试官： 优惠券？

候选人： 每用户可兑换 N 次的优惠码。原子性兑换： INSERT INTO Redemption(user, promo) IF NOT EXISTS。在费用计算时应用折扣。反欺诈：检测滥用行为，如相同银行卡、连续 IP、快速注册。超出预算的促销活动将暂停。

面试官： 跨城行程？

候选人： 较为少见但确实存在（机场长途至邻近城市、跨越多个城市的大都市区）。行程在起点城市的区域内创建；如目的地在另一城市，费用计算需考虑两个区域的规则。跨区域支付由支付系统处理。运营复杂度高，但并非架构层面的问题。

面试官： 多区域？

候选人： 每个城市使用最近的数据中心。跨城操作是显式的，并非常见情况。支付和账户等全球服务全球复制，各区域设有主节点。

面试官： 如何对账？

候选人： 每日例行任务。从行程记录重新计算费用，标记与已记录总额不符的条目。司机工时与位置更新进行核对。支付账本与行程总额进行核对。优惠券兑换与折扣使用情况进行核对。偏差触发告警和调查。

面试官： 司机完成行程，但应用在发送完成通知前崩溃。

候选人： 行程卡在 in_trip。检测方式：最大行程时长告警。调账：系统可检测到司机已开始下一单或已下线；根据启发式规则自动完成。模糊情况进行人工审核。费用根据截至检测到完成时刻所记录的事件计算。

面试官： 总结。

候选人： Uber 后端采用微服务架构：调度（地理位置 + 匹配）、行程状态机、支付（复用设计）、激增定价（按单元格热力图）、ETA（路由 + ML）、入驻、评分、客服。每个城市相互独立。费用由已记录事件确定性计算得出。调账任务捕获偏差。按城市多区域部署。最难的正确性问题是重复匹配（CAS）；最难的扩展问题是位置更新。

3. 练习备注

五分钟版本

  1. 澄清：调度、生命周期、支付、激增定价、客服的广度。
  2. 不变量：最多分配一位司机；确定性费用；幂等支付。
  3. 微服务：调度、行程、支付、激增定价、ETA、入驻、评分。
  4. 每城市地理索引；司机状态 CAS。
  5. 行程状态机；显式状态转换。
  6. 由事件确定性计算费用。
  7. 按单元格激增定价；ETA 结合路由与 ML。
  8. 异步入驻工作流；双向评分。

一分钟结尾陈述

Uber 后端以地理空间调度核心为基础，将各微服务串联起来。行程状态机保证生命周期正确性；支付复用支付设计，以 trip_id 实现幂等；激增定价是按单元格的热力图；ETA 结合路由和 ML；入驻与客服异步运行。城市是天然的分片单元。费用由已记录事件确定性计算得出；调账捕获偏差。按城市多区域部署。最难的正确性问题是重复匹配；广度挑战在于保持十几个微服务的一致性。

最佳面试用语

  • "每个城市是天然的分片单元；跨城是例外情况。"
  • "费用由事件确定性计算得出；任何人都可以重新计算，调账捕获偏差。"
  • "激增定价是按单元格的热力图，在请求时锁定写入行程。"
  • "通过 CAS 转换的行程状态机确保生命周期正确性。"
  • "通过 trip_id 实现端到端支付幂等。"
