1. 基础知识与解题思路

本题考察点

Gmail 是十亿用户规模的电子邮件服务。难点在于：与互联网其余部分的 SMTP 互操作、PB 级垃圾邮件处理、邮件线程、跨数十年邮件的全文搜索、持久的按用户收件箱存储，以及附件处理。

其不变量如下：

每封通过 SMTP 接收的邮件（或用户发送的邮件）都被持久存储在收件人邮箱中（或以明确原因拒绝），在投递前完成垃圾邮件分类，建立搜索索引，并可从用户的每台设备访问，具备一致的线程展示；发送邮件可靠投递给外部收件人，或以明确状态退信。

在 ICT5 / L6+ 级别，面试官在测试你是否能设计 SMTP 接收、反垃圾邮件、面向数十亿用户的按用户邮箱存储、大规模全文搜索，并能针对线程、基于标签的组织方式和附件存储进行合理分析。

60 秒面试框架

口述版本：

我会围绕四个核心系统设计 Gmail：带反垃圾邮件的 SMTP 接收、带线程和标签的按用户邮箱存储、全文搜索索引，以及发送用的 SMTP 出站。关键决策在于反垃圾邮件架构、按用户存储规模、搜索索引策略和附件处理。大部分工程复杂性在于这一规模下的反垃圾邮件和搜索。

首先明确需求

功能性需求：

  • 通过 SMTP 从互联网接收邮件。
  • 通过 SMTP 向互联网发送邮件。
  • 带标签、文件夹、线程的按用户收件箱。
  • 跨所有邮件搜索。
  • 附件。
  • 垃圾邮件、网络钓鱼、恶意软件过滤。
  • 自动回复、假期自动应答。
  • 多设备同步（IMAP、Web、移动端）。

非功能性需求：

  • 十亿级用户。
  • 每日投递数千亿封邮件。
  • 每用户存储从 GB 到 TB。
  • 高可用性。
  • 严格的垃圾邮件过滤，误报率低。
  • 数年邮件的搜索在一秒内完成。
  • 多地区。

以下内容超出范围，除非被特别询问：

  • 实时聊天 / Hangouts。
  • 日历集成（独立）。
  • Drive 附件集成（简要提及）。
  • 企业（Workspace）功能。

规模假设

领域 假设
用户 15 亿+。
每日投递邮件数 3000 亿+（50%+ 是垃圾邮件）。
平均邮件大小 50 KB；附件更大。
每用户存储 GB 到 TB。
搜索 QPS 10 万+/秒。
垃圾邮件比例 入站邮件 50%+；过滤后到达用户的比例 <0.1%。

瓶颈：反垃圾邮件计算和按用户存储规模。

核心产品模型

层级 职责 需求
SMTP 接收 从互联网接收。 可靠；可扩展。
反垃圾邮件 过滤垃圾邮件。 投递前。
邮箱存储 按用户的消息、线程、标签。 持久化。
搜索索引 按用户全文索引。 亚秒级查询。
附件存储 大型 blob。 CDN 友好。
出站 SMTP 发送到互联网。 重试；退信处理。

SMTP 接收

  1. 外部 MX（mail.gmail.com）接收 SMTP 连接。
  2. 对未知发件人进行灰名单 / 速率限制。
  3. SPF 检查：发送 IP 是否被 From 域授权？
  4. DKIM 验证：签名是否有效？
  5. DMARC：是否对齐？
  6. 接受或临时拒绝 / 拒绝。
  7. 传给反垃圾邮件流水线。

若在 SMTP 阶段被拒绝（5xx 响应），发件人的 MTA 退信。若已接受（2xx）但后续被过滤，邮件进入垃圾邮件文件夹（静默）或按策略丢弃。

反垃圾邮件流水线

多阶段：

  1. 信誉： 发件人域名和 IP 信誉。已知垃圾邮件发送者被屏蔽。
  2. 内容规则： 签名匹配、正则表达式、URL 黑名单。
  3. ML 分类器： 基于标注垃圾邮件训练；输出 spam_score。
  4. 用户信号： 收件人是否在地址簿中 / 是否有过往对话？
  5. 最终决策： 正常 → 收件箱，可疑 → 垃圾邮件文件夹，恶意软件 → 隔离。

ML 分类器特征：发件人行为模式。内容特征（文本、URL、附件）。收件人互动模式。跨收件人的聚类分析（群发活动检测）。

垃圾邮件占入站邮件至少 50%；Gmail 的过滤使收件箱中垃圾邮件比例低于 0.1%。

线程

使用 RFC 5322 的 References 和 In-Reply-To 头将邮件分组为会话：

  • 每封邮件都有一个 Message-ID。
  • 回复邮件在 In-Reply-To 中包含父邮件的 Message-ID，在 References 中包含先前线程。
  • 线程引擎通过引用的传递闭包对邮件分组。
  • 以主题为基础的线程作为兜底方案（Re: 原始主题）。

对每个用户而言，线程是一个包含 message_ids 的 Thread 记录。

标签与过滤器

Gmail 使用标签而非文件夹。一封邮件可有多个标签。系统标签：收件箱、已发送、草稿、垃圾邮件、垃圾桶。用户标签：任意自定义。

过滤器：用户自定义规则，根据发件人/主题/内容自动打标签、归档、删除。

智能标签：Gmail 的 ML 自动将邮件分类到收件箱、社交、推广、通知、论坛。

邮箱存储

按用户；以 user_id 分片。

  • 每个用户的邮箱是邮件记录、线程记录和标签分配的序列。
  • Bigtable 风格的行存储效果良好：行键 = user_id:thread_id:message_id.
  • 附件单独存储（内容寻址）；通过 ID 引用。
  • 存储随时间增长；冷邮件迁移到更廉价的存储层。

去重方案：附件内容寻址。若多个用户收到相同文件（如转发的图片），则只有一个 blob，多个引用。

搜索

按用户倒排索引：

  • 对主题、正文、发件人、收件人、标签、附件进行分词。
  • 按用户建立索引。
  • 新邮件实时索引；历史邮件批量补全。

查询：词元 + 过滤器 + 范围，返回排序后的邮件。按用户亚秒级响应。

对于每人有 TB 级邮件的 15 亿用户，搜索索引规模庞大——但它是按用户的，因此大量分片。

出站 SMTP

用户发送邮件：

  1. 邮件内容验证；收件人列表检查。
  2. DKIM 签名（Gmail 代表用户域名进行权威签名）。
  3. SPF/DMARC 对齐。
  4. 提交到出站 SMTP 队列。
  5. 工作节点通过 DNS MX 记录查找并连接到收件人的 MX。
  6. 通过 SMTP 投递。
  7. 收到 2xx 确认：成功。
  8. 收到 4xx 临时失败：指数退避重试。
  9. 收到 5xx 永久失败：退信给发件人。
  10. 超过重试超时：退信。

Gmail 到 Gmail：绕过外部 SMTP，内部投递以降低延迟。

退信处理

退信（NDR - 未投递报告）以 DSN 消息形式返回。解析后展示在发件人的已发送文件夹中。

扩展与分区

数据 分区键 原因
邮箱 user_id 哈希值 每用户数据。
线程 user_id + thread_id 按线程查询。
附件 content_hash 去重。
搜索索引 按用户分片 按用户的查询。
垃圾邮件分类器 全局模型 跨用户共享。
退信日志 sender_user_id 按发件人。

一致性模型

层级 一致性 原因
邮件存储 写入时强一致 权威性。
会话成员 强一致性 已关联。
标签变更 强一致性 用户期待效果生效。
搜索索引 最终一致 有界时效性。
垃圾邮件分类 最终一致 支持重新分类。
多设备同步 最终一致 有界。

多区域策略

  • 按用户划分主存储区域。
  • MX 服务器全球分布。
  • 入站邮件投递至用户所在区域。
  • 从用户所在区域发送。
  • 故障转移区域保持热备。

故障模式与缓解措施

故障模式 缓解措施 监控指标
垃圾邮件被误判为正常邮件。 用户举报；分类器重新训练。 假阴性率。
正常邮件被误判为垃圾邮件。 用户恢复；分类器重新训练；预览窗口。 假阳性率。
钓鱼邮件漏过过滤器。 多阶段检测；安全警告；用户教育。 钓鱼邮件举报率。
含恶意软件的附件。 沙箱扫描；隔离处理。 恶意软件检测率。
对 MX 服务器的 DDoS 攻击。 按 IP 限速；信誉机制。 每 IP 连接数。
单用户邮箱热点（例如，拥有数百万封邮件的用户）。 子分区；索引剪枝。 每用户 QPS。
出站投递临时失败。 带退避的重试队列。 重试队列深度。
搜索索引延迟。 异步处理；对延迟发出告警。 索引延迟。
退信循环。 检测并停止转发。 退信频率。
跨区域故障转移。 热备副本；有界延迟。 故障转移 RTO。

运维指标

正确性：

  • 垃圾邮件到达收件箱的比率。
  • 正常邮件被过滤为垃圾邮件的比率。
  • 邮件丢失率（必须接近零）。
  • 退信率。

容量：

  • SMTP 接收速率。
  • 存储增长。
  • 搜索 QPS / 延迟。

业务层面：

  • 活跃用户数。
  • 每用户邮件量。
  • 参与度（打开、回复）。

需要指出的权衡

  • 激进的垃圾邮件过滤降低了收件箱噪音，但增加了假阳性。
  • 永久存储所有邮件会消耗大量存储空间；按策略进行分层或过期处理。
  • 每用户搜索索引速度快，但存储开销大。
  • 实时会话分组与批量（缓存）方式的权衡：延迟 vs CPU 开销。
  • 附件去重节省存储空间，但增加了元数据开销。
  • DKIM/SPF/DMARC 强制执行保护用户，但会拦截配置有误的合法发件人。

常见陷阱与更优做法

陷阱 更优做法
将邮件视为即时通讯（类似 Discord）。 SMTP 是协议；围绕互操作性进行设计。
单一全局搜索索引。 按用户分片。
同步垃圾邮件检查。 多阶段，带快速路径。
文件夹而非标签。 标签（支持多重归属）。
单一 ML 模型。 多阶段分类器 + 信誉机制。
原位附件存储。 内容寻址的独立存储。

面试官可能深入探讨的方向

深入方向 优秀回答要点
反垃圾邮件 多阶段：信誉 + 内容 + ML。
存储 按用户；分层；附件去重。
搜索 每用户倒排索引；亚秒级响应。
线程 RFC 5322 References 头；Subject 兜底。
SMTP 互操作 DKIM/SPF/DMARC；重试；退信。
多区域 按用户划分主区域；MX 全球部署。

Staff 级别回答检查清单

  • 带反欺骗的 SMTP 接收。
  • 多阶段反垃圾邮件。
  • 按 user_id 分片的每用户存储。
  • 每用户搜索索引。
  • 带去重的附件存储。
  • 通过 RFC 5322 References 实现会话分组。
  • 带重试和退信的出站 SMTP。
  • 多区域部署，用户归属主区域。

面试用语

Gmail 是每用户邮箱存储加上 SMTP 互操作加上激进的反垃圾邮件。最大的工程挑战是反垃圾邮件：基于信誉、内容规则、ML 分类器和用户信号的多阶段检测。搜索是每用户倒排索引。附件采用内容寻址并去重。存储按 user_id 分片。

2. 模拟面试

面试官： 设计 Gmail。

候选人： 四个核心系统：带反欺骗的 SMTP 接收、多阶段反垃圾邮件流水线、支持会话分组和标签的每用户邮箱存储、全文搜索索引，以及用于发送的出站 SMTP。不变式为：已接收邮件的每用户持久化存储、在用户可见前完成垃圾邮件过滤、可搜索，以及可靠投递给外部收件人。

面试官： 带我了解接收一封邮件的流程。

候选人： 外部 MX 服务器接受 SMTP 连接。执行 SPF 检查（发送 IP 是否被 From 域名授权？）、DKIM 验证（签名是否有效？）、DMARC 对齐检查。若检查通过且发件人信誉可接受，接收该邮件。传递给反垃圾邮件流水线。多阶段过滤——信誉、内容规则、ML 分类器、用户信号——生成垃圾邮件得分。低于阈值：投递至用户收件箱。高于阈值：垃圾邮件文件夹。含恶意软件：隔离处理。邮箱存储写入邮件；会话分组引擎将其归入对话；搜索索引器对其进行索引。

面试官： 大规模反垃圾邮件。

候选人： 多阶段。（1）信誉：发件人 IP 和域名信誉、已知垃圾邮件黑名单、地区性垃圾邮件模式。（2）内容规则：特征匹配、正则表达式、URL 黑名单。（3）ML 分类器：基于标注垃圾邮件训练的深度模型，特征包括文本、URL、发件人行为、附件模式。（4）用户信号：发件人是否在收件人通讯录中？双方之前是否有过往来？（5）群发检测：跨收件人识别大规模群发活动。采用多阶段是因为没有任何单一检查是完美的；组合使用可获得高精确率和召回率。

面试官： 会话分组？

候选人： RFC 5322 的 References 和 In-Reply-To 头。回复邮件中包含父邮件的 Message-ID。会话分组引擎对这些引用取传递闭包以分组邮件。Subject 规范化（去除 Re:、Fwd:）作为兜底方案。每用户会话记录捕获分组；用户看到的是对话而非单封邮件。

面试官： 存储规模？

候选人： 按用户，以 user_id 分片。Bigtable 风格的行存储：行键 user_id:thread_id:message_id 使每用户读取高效。每条邮件记录包含邮件头、正文、标签、附件引用。附件作为内容寻址的 blob 单独存储并去重——许多用户可能收到同一文件，只存储一次并维护引用计数。旧邮件分层到访问延迟更长的低成本存储。

面试官： 在多年的邮件中搜索？

候选人： 每用户倒排索引。对 Subject、正文、发件人、收件人、标签、附件名称进行分词。新邮件实时建索引。按用户分片——你的搜索只访问你的分片。查询：词元 + 过滤条件（标签、日期范围、发件人），按相关性和时效性评分，返回分页结果。

面试官： 一个拥有 100 GB 邮件的用户搜索"vacation"。

候选人： 其每用户搜索索引很大，但仍有界。词元"vacation"映射到邮件 ID 的倒排列表。按其他条件过滤。评分。返回前 N 条并附带摘要。由于索引是按用户独立且经过良好调优的，响应时间在亚秒级。

面试官： 发送一封邮件。

候选人： 用户撰写邮件；客户端提交。验证收件人。DKIM 签名——Gmail 以用户域名的权威身份对邮件进行签名。提交至出站 SMTP 队列。工作进程通过 DNS 解析收件人的 MX 记录，建立 SMTP 连接，完成投递。返回 2xx：成功。返回 4xx 临时失败：进入指数退避重试队列（从分钟级到小时级，上限为数天）。返回 5xx 永久失败或最终重试耗尽：向发件人退回 NDR 退信通知。

面试官： Gmail 发给 Gmail？

候选人： 绕过外部 SMTP。接收方是同一系统；以更低的延迟进行内部投递。同样的反垃圾邮件流水线仍然适用（Gmail 内部邮件同样可能是垃圾邮件——账号被盗、机器人账号等）。

面试官： 附件去重？

候选人： 内容寻址：blob_id = 内容的 SHA-256。引用计数。多个用户引用同一附件时计数递增。所有引用归零时，GC 回收。对转发文件节省大量存储空间。注意：跨用户去重会泄露"你是否拥有该文件"的信号——对无害附件通常可以接受，但对敏感附件需特别处理。

面试官： 钓鱼邮件——一封精心伪造的邮件看起来像合法邮件。

候选人： 多信号检测：域名相似性（仿冒域名）、该收件人的异常发件人、可疑链接模式、历史钓鱼活动、针对钓鱼邮件训练的 ML 模型。在界面中显示软性警告（"此邮件疑似钓鱼"）。严重情况进入垃圾邮件文件夹或隔离区。用户举报反馈至重新训练。

面试官： 多区域？

候选人： 按用户划分主区域。MX 服务器全球分布；入站邮件路由至用户所在区域。出站邮件从用户所在区域发出。故障转移区域保持热备，复制延迟有界。搜索索引与邮件同区域。

面试官： 用户将一封合法邮件举报为垃圾邮件。

候选人： 用户举报反馈给分类器。该特定邮件移至垃圾邮件文件夹。来自同一发件人的类似邮件可能针对该用户进行追溯性重新分类。全局分类器权重根据聚合举报随时间更新。

面试官： 总结。

候选人： SMTP 接收带反欺骗检查；多阶段反垃圾邮件流水线；带会话分组和标签的每用户邮箱存储；每用户搜索索引；带去重的内容寻址附件存储；带重试和退信的出站 SMTP。多区域部署，每用户有主区域。主要的工程挑战是反垃圾邮件质量；主要的规模挑战是每用户存储和搜索索引的体量。

3. 练习备注

五分钟版本

  1. 明确范围：SMTP、反垃圾邮件、存储、搜索。
  2. 不变式：每用户持久化；垃圾邮件过滤；可搜索。
  3. MX → 反垃圾邮件 → 邮箱 → 搜索索引。
  4. 多阶段反垃圾邮件流水线。
  5. 按 user_id 分片的每用户存储（Bigtable 风格）。
  6. 每用户倒排索引用于搜索。
  7. 附件内容寻址；去重。
  8. 带重试和退信的出站 SMTP。

一分钟结尾陈述

Gmail 是每用户邮箱存储加上 SMTP 互操作加上激进的反垃圾邮件。SMTP 接收检查 SPF/DKIM/DMARC；多阶段反垃圾邮件在投递前过滤；每用户存储按 user_id 分片；每用户搜索索引。附件内容寻址并去重。出站 SMTP 带重试和退信处理。多区域部署，每用户有主区域。主要的工程挑战是十亿用户规模下的反垃圾邮件质量。

最佳面试用语

  • "反垃圾邮件是多阶段的：信誉、内容、ML、用户信号。"
  • "每用户分片使得跨多年邮件的搜索变得可行。"
  • "附件内容寻址并去重。"
  • "通过 RFC 5322 References 实现会话分组；Subject 作为兜底。"
  • "SMTP 互操作是基本要求；垃圾邮件过滤才是工程投入的重点。"
