1. 基础知识与解题思路

本题考察点

Web 爬虫是典型的大型分布式系统问题：数万亿个 URL，每台主机都是独立的受速率限制的资源，工作永无止境。难点在于保持礼貌（不对主机过度请求）、高效（不重新爬取未变更的页面），同时遵守 robots.txt。

其不变量如下：

爬虫必须以尊重每台主机礼貌预算和 robots.txt 的速率访问 URL，不得超出必要的次数获取 URL 以保持内容新鲜，并为每个成功抓取的页面生成确定性、去重的、可索引的记录。

在 ICT5 / L6+ 级别，面试官考察的是你能否设计每主机队列、在不重新获取的情况下管理数万亿 URL 的前沿队列、处理陷阱（日历链接、Session ID），以及对重要页面的新鲜度优先于完整性。

60 秒面试框架

口述版本：

我会围绕一个按主机分区的 URL 前沿队列来设计爬虫，配备严格的每主机礼貌队列。无状态抓取器集群从前沿队列中拉取任务，抓取 HTML，提取链接，通过内容和 URL 指纹去重，并将文档发送给下游解析器和索引器。关键决策在于如何处理礼貌约束、在万亿规模下去重、对前沿队列排优先级，以及检测爬虫陷阱。

首先明确需求

功能性需求：

  • 从种子 URL 开始爬取公开 Web。
  • 遵守 robots.txt 和 crawl-delay 指令。
  • 根据估算的内容新鲜度按计划重新爬取页面。
  • 检测重复和近重复页面。
  • 提取链接并在过滤条件下跟进（深度、域名白名单、内容类型）。
  • 生成适合下游索引或分析的内容。

非功能性需求：

  • 扩展至数百亿页面，数万亿已知 URL。
  • 吞吐量：每秒数十万次抓取。
  • 礼貌性：绝不过载小型站点。
  • 对爬虫陷阱和对抗性内容的鲁棒性。
  • 爬虫宕机后可恢复。

以下内容超出范围，除非被特别询问：

  • 索引服务（见"设计 Google Search"）。
  • 全规模 JavaScript 渲染（作为重型模式提及）。
  • 实时内容发现（来自发布者的推送）。

规模假设

领域 假设
页面数量 语料库中有 100 亿到 1000 亿个文档。
已知 URL 数量 是页面数量的 10 到 100 倍（大多数从未被抓取）。
主机数量 超过 2 亿个活跃主机。
抓取速率 整个集群每秒 10 万到 100 万个 URL。
平均页面大小 原始内容 50-200 KB，提取文本 10-30 KB。
礼貌性 通常每主机每秒 1 个请求；可配置。
重新爬取频率 新闻：数小时；静态页面：数周。
存储 原始 HTML 达到 PB 级别；历史版本达到数十 PB。

瓶颈在于主机并发：单个热点主机的访问速度不能超过其礼貌预算，而前沿队列必须在不违反任何主机预算的前提下保持工作进程繁忙。

核心组件

组件 职责 关键细节
URL 前沿队列 待抓取 URL 的优先级队列，按主机分片。 内部按主机 FIFO；跨主机按优先级排序。
抓取器 HTTP 客户端；抓取 URL；处理重定向、重试和 robots.txt。 无状态；水平可扩展；从前沿队列拉取。
Robots 服务 缓存并按主机提供 robots.txt 服务。 基于 TTL 刷新；每次抓取前查询。
DNS 解析器 高吞吐量 DNS，带内部缓存。 避免压垮递归解析器。
内容存储 按 URL+版本存储原始响应的追加-only Blob 存储。 版本化；大对象存于对象存储。
解析器 提取链接、文本、元数据。 流式处理；处理格式错误的 HTML。
URL 去重 跟踪"已见" URL，避免重复入队。 布隆过滤器 + 持久化集合。
内容去重 跟踪页面内容指纹。 Simhash / minhash 用于近重复检测。
调度器 根据每个 URL 决定重新抓取的频率。 Score = freshness × importance × cost。
爬取队列优先级调度器 按分数对爬取队列中的 URL 重新排序。 热门页面获得更多关注。
爬虫陷阱检测器 限制每个主机的抓取深度和 ID 参数基数。 避免无限循环。

URL 爬取队列设计

这是核心深入解析。

两级结构：

  • 每个优先级类（高、中、低）一个前置队列。
  • 每个主机一个后置队列，确保严格的每主机 FIFO 和礼貌性控制门。

工作流程：

  1. 新 URL 从提取器到达。
  2. 查找主机；将其路由到该主机的后置队列。
  3. 每个优先级类的调度器线程查看前置队列；选择礼貌性时间窗口已到期的主机；弹出下一个 URL。
  4. 将 URL 交给抓取器。

存储：

  • 前置和后置队列存储在持久化的耐久性存储中（分片日志或 KV 存储），因为数万亿个 URL 无法全部放入内存。
  • 热工作集在内存中缓存，供活跃的抓取器集群使用。

分区方式：

  • 爬取队列按 hash(host)分片。同一主机的所有 URL 位于同一分片上。
  • 每个分片管理自己的每主机队列。

这保证了：每主机的礼貌性在本地强制执行；通过让每个分片各自管理多个主机，实现跨主机并行性。

面试用语：

爬取队列是系统的核心。两级设计——每主机后置队列用于礼貌性控制，跨主机前置队列用于优先级控制——使单个分片能够从多个主机抓取，而不违反任何主机的配额。

礼貌性

对于每个主机：

  • 跟踪 last_fetch_at。下一次抓取在 last_fetch_at + politeness_interval.
  • politeness_interval 默认为一秒，可由 robots.txt 覆盖。 crawl-delay.
  • 每主机并发抓取数：通常为 1；经协商后可对大型主机放宽。
  • 对于反复出现的 5xx 或限流响应，自动增加间隔（乘法退避）。

Robots.txt 处理

在抓取 URL 之前：

  1. 检查该主机的 robots 缓存。若缺失或已过期（TTL 通常为 24 小时），则抓取 host/robots.txt.
  2. 若 robots.txt 缺失/返回 200，则解析并存储。
  3. 将 URL 与规则匹配。若对我们的 user-agent 不允许，则丢弃并触发 forbidden URLRecord 更新。
  4. 遵守 Crawl-delay （若已设置）。
  5. 缓存 sitemaps；将其加入队列作为发现来源。

存在许多解析器；规则并非正式规范，但约定很重要（按 user-agent 最长匹配等）。将 robots.txt 视为权威来源。

去重：URL 与内容

URL 去重：

  • 规范化 URL：主机名小写、排序查询参数、去除 session id、移除片段。
  • 哈希为 64 位指纹。
  • 维护一个全局 bloom filter 用于成员查询；以持久化 KV 支撑的精确集合作为真实数据源。
  • Bloom filter 未命中 → 查询 KV。集合成员关系决定是否将其加入队列。

内容去重：

  • 计算规范化文本的 SHA-256。
  • 与"已见内容"索引对比。精确匹配 → 已知重复。
  • 对于近似重复（模板页面或套版内容），计算 simhash 或 minhash 签名；将签名在 Hamming 距离阈值内的页面聚类。
  • 每个内容聚类只保留一个规范 URL。

重新抓取调度

并非所有 URL 都值得同等对待。

评分因素：

  • 重要性：域权威性、入链数量、流量估算。
  • 预估变更频率：来自历次抓取差异（新闻站每小时变化；静态文档每周变化）。
  • 上次抓取时间距今。
  • 每主机及每优先级类的容量预算。

算法：

  • 每个 URL 都有一个 refetch_score。分数越高 → 抓取越早。
  • 每次抓取后根据观测到的变更频率重新计算分数。
  • 调度器为每个主机维护一个等待重新抓取的 URL 优先级堆。

爬虫陷阱

常见陷阱：

  • 无限日历页面： /cal/2024/01/01, /cal/2024/01/02, ...
  • 多维商品 URL：查询参数组合呈指数增长。
  • URL 中包含 session id。
  • 循环的分页逻辑。

缓解措施：

  • 限制每主机的抓取深度。
  • 限制每主机每天的 URL 数量。
  • 检测参数基数爆炸并隔离该模式。
  • 在规范化时去除已知的 session id 参数。
  • 声誉系统：重复比率高的主机会被限速。

对抗性内容

  • 能检测爬虫 user-agent 并提供不同内容的页面（内容伪装）。
  • 响应中包含恶意软件载荷的页面。
  • 试图耗尽爬虫资源的页面（解压炸弹）。

防御措施：

  • 限制每页字节数；超过则中止。
  • 对解析器和渲染器进行沙箱隔离。
  • 随机抽样：偶尔以普通 user-agent 抓取以检测内容伪装。
  • 每次抓取的资源限制。

写路径与读路径

写（抓取）路径：

  1. 抓取器从爬取队列中取出 URL。
  2. 检查 robots 缓存；若不允许，丢弃并更新 URLRecord。
  3. 通过 DNS 缓存解析主机。
  4. 带有条件请求头的 HTTP GET（If-Modified-Since, ETag).
  5. 收到 304 时，更新 last_fetched_at；跳过存储。
  6. 收到 200 时，将响应流式写入内容存储；记录元数据。
  7. 解析：提取链接、文本、canonical 标签。
  8. 对每个链接：规范化、去重检查、若为新链接则加入队列。
  9. 更新内容去重索引。
  10. 将文档推送至下游索引器。

读路径： 索引器和下游消费者从文档流消费，或从内容存储读取。不属于爬虫本身的范畴。

扩展与分区

数据 分区键 原因
URL 前沿队列 按主机哈希 将同一主机的所有 URL 集中存储。
URL 去重集合 按 url_hash 哈希 均匀分布。
内容去重 按 content_hash 哈希 均匀分布。
内容存储 URL 或 document_id 大规模 blob 存储。
主机状态 按主机哈希 与爬取队列共置。

一致性模型

层级 一致性 原因
URL 去重 最终一致性（bloom filter + 精确集合） Bloom filter 误报减少工作量；偶发漏报可接受。
爬取队列调度 每主机严格顺序 礼貌性是硬性约束。
内容存储 新版本的强写入 审计与时序分析。
Robots 缓存 TTL 内最终一致性 可接受的延迟。

故障模式与缓解措施

故障模式 缓解措施 监控指标
大量请求轰炸小型站点（礼貌性漏洞）。 每主机单飞门控；对 429/5xx 执行退避。 每主机 QPS；投诉量。
爬虫陷阱消耗集群容量。 深度上限、主机上限、参数基数检测。 每主机重复比率；每主机 URL 增长率。
Robots.txt 抓取失败。 若近期不可达，默认拒绝；或默认允许但降低速率。 Robots 抓取错误率。
爬取队列分片不均衡。 将热点主机重新分片至专用分片。 每分片积压大小。
内容存储增长爆炸。 存储前去重；对未变更页面进行存储抽样。 存储增长率 vs 新内容率。
抓取器在流式传输中途崩溃。 幂等重新抓取；URL 保持在 dispatched 状态并设置超时。 飞行中 URL 过期计数。
DNS 解析器过载。 本地 DNS 缓存；按域权威性限流。 DNS QPS、错误率。
对抗性内容伪装。 混合 user-agent 抽样；声誉跟踪。 不同 agent 下每主机内容哈希的方差。

运维指标

正确性：

  • 每主机礼貌性违规次数（必须约为零）。
  • robots.txt 规则违规次数。
  • 在刷新时间窗口内同一 URL 的重复抓取次数。

容量：

  • 按优先级划分的抓取 QPS。
  • 爬取队列大小及增长率。
  • Bloom filter 误报率。
  • 每主机积压分布。

业务层面：

  • 覆盖率：在目标 SLA 内重新抓取的重要页面比例。
  • 新闻域的索引新鲜度。
  • 每天发现的 URL 数量。

多区域策略

  • 在靠近发布者的区域运行抓取器集群（美国、欧洲、亚洲）。
  • 爬取队列分片按主机分区；每个主机归属一个区域（通常根据主机托管位置选择）。
  • 内容存储按区域部署，并进行跨区域复制用于索引。
  • 避免两个区域对同一主机进行重复抓取。

需要指出的权衡

  • 礼貌性限制过严会压低吞吐量；过宽则会惹怒站长。
  • 激进去重能减少存储，但有遗漏合法更新的风险。
  • 严格遵守 robots 规则会牺牲覆盖率；宽松解释则可能引发法律问题。
  • 在万亿 URL 规模下，集中式优先级很难实现；通常接受近似的每分片优先级方案。
  • 深度抓取发现更多内容但消耗预算；浅度抓取会遗漏长尾内容。
  • 重新抓取频率应根据观测到的变更率动态调整；静态调度会浪费抓取配额。

常见陷阱与更优做法

陷阱 更优做法
一个大的全局队列。 两级爬取队列：每主机后置队列，按优先级划分前置队列。
忽略 robots.txt。 一级 robots 缓存，在每次抓取前均会查询。
仅在插入时进行 URL 去重。 Bloom filter + 精确集合；检查前重新规范化。
将所有 URL 一视同仁。 基于重要性 + 变更率为重新抓取评分。
忽略爬虫陷阱。 每主机深度上限与参数上限；基数检测。
同步索引。 仅追加文档流；索引器作为下游消费者。

面试官可能深入探讨的方向

深入方向 优秀回答要点
爬取队列设计 两级队列；每主机礼貌性控制；跨主机优先级管理。
大规模去重 Bloom filter + 精确 KV；simhash 用于近似重复检测。
重新抓取调度 基于重要性 + 变更率的评分；受容量约束。
爬虫陷阱防御 深度上限；基数检测；声誉机制。
Robots.txt 处理 缓存、解析、sitemap 发现。
JS 渲染 独立慢车道；仅用于需要的站点。

Staff 级别回答检查清单

  • 礼貌性作为硬性不变量，按主机强制执行。
  • 两级爬取队列：跨主机优先级控制，每主机 FIFO。
  • URL 规范化与 bloom filter + 精确集合去重。
  • 使用 simhash / shingles 进行内容去重。
  • 基于重要性和变更率驱动的重新抓取调度。
  • 爬虫陷阱检测与主机声誉管理。
  • robots.txt 缓存作为一等服务。
  • 仅追加文档存储；索引器位于下游。

面试用语

网络爬虫最难的约束是礼貌性，而非吞吐量。两级队列——每个主机一个后端队列，跨主机的前端队列——让我在不猛烈冲击单个主机的前提下实现并行化。使用布隆过滤器加内容指纹去重，避免无效工作；通过重要性和变化频率评分，将容量聚焦于有价值的内容。

2. 模拟面试

面试官： 设计一个网络爬虫。

候选人： 我将围绕以下要素进行设计：一个强制执行每主机礼貌性作为硬性不变量的 URL 前沿、一个无状态的抓取器集群，以及针对 URL 和内容的去重层。难点在于礼貌性、去重集合的规模、爬虫陷阱，以及确定哪些内容需要重新抓取。

面试官： URL 前沿是如何工作的？

候选人： 两级结构。每个主机拥有一个严格按 FIFO 顺序的后端队列；礼貌性规则为从该后端队列中下一次抓取距上次抓取至少间隔 T 毫秒。一组跨优先级类的前端队列决定下一个弹出哪个主机。调度线程检查哪个主机的礼貌等待窗口已到期，并选取优先级最高的那个。整个前沿按 hash(host)进行分片，使同一主机的所有 URL 位于同一分片上。

面试官： 前沿有多大？

候选人： 数以万亿计的 URL，无法全部放入内存。前沿存储在持久化存储中——日志结构 KV 或分片的 RocksDB——并缓存热点工作集。工作节点以流的方式将 URL 写入和读出磁盘驻留的队列。

面试官： 在这种规模下如何对 URL 去重？

候选人： 首先规范化：主机名转小写、查询参数排序、去除已知 session id、丢弃 fragment、规范化末尾斜杠。哈希为 64 位指纹。用分片布隆过滤器做快速成员检查。布隆命中时，查询持久化精确集合。未命中时，插入布隆和精确集合，并将 URL 入队。

面试官： 布隆过滤器的假阳性——你如何处理？

候选人： 假阳性意味着我们偶尔会跳过实际上从未见过的 URL。在布隆过滤器尺寸适当的情况下，假阳性率远低于 1%。这是可以接受的，因为该 URL 会从另一个入链中再次出现，届时会与精确集合对比检查。错过一次重新发现的代价，远低于每个 URL 都查询精确集合的代价。

面试官： 内容去重怎么做？

候选人： 对规范化后的文本计算 SHA-256。完全匹配则为已知重复，跳过存储，但更新 URL 到 content_hash 的映射。对于近似重复，计算 simhash 或 minhash。simhash 汉明距离在阈值内的页面聚为一类；每个聚类保留一个规范 URL。

面试官： 你如何遵守 robots.txt？

候选人： 设立一等 robots 服务。抓取某个 URL 之前，先查询该主机的 robots 条目。若缺失或已过期（TTL 24 小时），则抓取 /robots.txt （对该主机本身也遵守礼貌性）。解析规则并存储。将 URL 与针对我们 user-agent 的规则进行匹配。被禁止的 URL 标记为 forbidden 并跳过。我们还从 robots.txt 中提取 sitemap URL 作为发现种子，并遵守 Crawl-delay.

面试官： 爬虫陷阱示例：一个无限向前跳转的日历页面。

候选人： 三重防御。第一，每主机设置爬取深度上限（例如 20）。第二，每主机每日 URL 预算；超出后，该主机的低优先级 URL 将被丢弃。第三，参数基数检测：若某个查询参数在一个主机内取值达到数千个不同值，则标记该模式，并隔离匹配的新 URL。该主机的信誉下降，后续抓取受到限速。

面试官： 哪些内容会被重新抓取，何时抓取？

候选人： 每个 URL 维护一个重抓评分，结合重要性（PageRank 风格）、观测到的变化频率以及距上次抓取的时间。新闻域名的重抓间隔可能为一小时；静态手册可能为每月一次。调度器为每个主机维护一个优先级堆；容量按优先级类和配额分配。我们还遵守 Last-Modified 。 ETag 通过条件 GET；304 响应成本低廉且能确认内容新鲜度。

面试官： 如何处理持续返回大量 5xx 错误的主机？

候选人： 对该主机的礼貌性间隔执行指数退避。反复出错后，主机进入降级状态——每小时抓取次数减少。经过较长的冷却期后，发起探测以确认是否已恢复。若主机持续宕机，则降低优先级但不永久丢弃；故障总会发生。

面试官： 如何对大型站点在同一主机内并行抓取？

候选人： 谨慎操作。默认情况下，每次同时只从一个主机抓取一个请求。对于非常大的主机（主要新闻站点、大型电商），我们可能协商更高频率，或检测其明显能承受更多并发。若存在互惠协议，可通过多个 IP 地址向此类主机发起多个并发请求。

面试官： JS 渲染的页面怎么处理？

候选人： 两条通道。默认通道抓取原始 HTML 并解析。对于已知需要 JS 的站点，较慢且代价更高的通道使用无头浏览器。无头浏览器集群规模小得多；只有少量 URL 按重要性优先路由到该通道。

面试官： 多区域？

候选人： 在每个区域部署靠近发布方的抓取器集群。每个主机归属于一个区域——通常是地理上最近的——从而避免两个区域重复抓取同一主机。内容存储跨区域复制，供下游索引使用。

面试官： 总结。

候选人： 两级前沿按主机分片，严格执行礼貌性；无状态抓取器拉取 URL 并遵守 robots.txt。URL 去重使用布隆 + 精确集合；内容去重使用 simhash。重抓调度基于重要性 + 变化频率评分。爬虫陷阱通过深度和基数上限来限制。输出为仅追加的文档流，由下游索引器消费。礼貌性是硬性不变量，而非吞吐量；其余一切通过分片来扩展。

3. 练习备注

五分钟版本

  1. 明确：规模、礼貌性、robots、去重、新鲜度。
  2. 不变量：遵守每主机礼貌性；确定性去重。
  3. 两级前沿：每主机后端队列，按优先级的前端队列。
  4. 无状态抓取器；robots 缓存；DNS 缓存。
  5. URL 去重使用布隆 + 精确集合；先规范化。
  6. 内容去重使用哈希 + simhash。
  7. 按重要性 x 变化频率进行重抓调度。
  8. 爬虫陷阱防御；信誉机制；多区域部署。

一分钟结尾陈述

网络爬虫受礼貌性约束，而非吞吐量。两级前沿——每主机后端队列、按优先级前端队列——保证永不冲击某一主机的同时保持充分利用率。万亿级 URL 去重使用布隆过滤器加持久化精确集合兜底。内容去重使用 simhash 处理近似重复。重抓由评分驱动，将容量集中在重要且频繁变化的页面。爬虫陷阱通过深度、基数和信誉来限制。输出为仅追加的文档流，由下游索引器消费。

最佳面试用语

  • "礼貌性是硬性不变量，按主机强制执行。"
  • "两级前沿：后端队列负责礼貌性，前端队列负责优先级。"
  • "在万亿级规模下，布隆过滤器加精确集合是唯一可行的去重方式。"
  • "重抓调度 = 重要性 x 变化频率 / 容量。"
  • "爬虫陷阱通过深度上限和参数基数检测来限制。"
