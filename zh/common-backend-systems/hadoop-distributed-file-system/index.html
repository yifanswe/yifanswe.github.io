Hadoop Distributed File System (HDFS / GFS)

2026/06/01 · Common Backend Systems
目录
目录

对这些术语不熟悉？ 请参阅 Staff System Design Glossary。HDFS 是 Google File System (GFS) 的开源重实现；两者设计本质上相同。

1. 基础知识与解题思路

本题考察点

HDFS / GFS 是面向大数据的典型分布式文件系统：PB 级存储、大文件批量读写、普通商用硬件容错能力。难点在于在廉价易故障硬件上实现高持久性与高吞吐量，同时保持元数据服务的简洁性。

其不变量如下：

写入 HDFS 的每个文件均以块序列存储；每个块在多个 datanode（通常为 3 个）上复制副本；读取可来自任意副本；命名空间和块位置由 namenode 追踪；系统在不丢失数据的情况下容忍日常 datanode 故障。

在 ICT5 / L6+ 级别，面试官考察你能否清晰阐述 namenode/datanode 的分工、块复制策略、写入流水线、原始单 namenode 瓶颈及 HA / Federation 的解决方案，以及工作负载假设（以追加为主的大文件）。

60 秒面试框架

口述版本：

我会将 HDFS 描述为一个 namenode 加上一组 datanode。文件被拆分为较大的块（通常为 128 MB）；每个块按跨机架策略复制到三个 datanode。namenode 管理文件系统命名空间和块到 datanode 的映射。datanode 存储并提供块数据。写入通过副本流水线进行。该设计假设工作负载以追加为主的大文件为主。关键设计决策包括块大小、副本放置、namenode 瓶颈，以及通过备用 namenode 实现 HA。

论文设定的需求

功能性需求：

  • 分层文件系统命名空间。
  • 文件由块序列组成；大文件为主。
  • 读取与追加（原始 GFS / HDFS 不支持随机写）。
  • 高吞吐量批量读写。
  • 通过复制保证持久性。

非功能性需求：

  • PB 级数据量。
  • 数千个 datanode。
  • 商用硬件（频繁的磁盘/服务器故障）。
  • 高聚合吞吐量。
  • 简单的一致性模型。

原始设计范围之外：

  • 低延迟小文件读取。
  • 随机原地写。
  • POSIX 兼容性。

规模假设

领域 假设
文件 数百万到数十亿。
平均文件大小 数百 MB 到 TB 级。
块大小 64-128 MB（相比 Unix 的 4 KB）。
副本 通常 3x 副本。
datanode 数千个。
每个节点磁盘容量 每台数 TB。
集群总容量 PB 级。

核心概念

块

文件被拆分为固定大小的块（HDFS 默认 128 MB）。

  • 较大的块大小可减少 namenode 的元数据压力。
  • 每个块以 Unix 文件形式存储在 datanode 本地磁盘上。
  • 块是 HDFS 中的一等对象。
  • 复制因子按文件指定（默认 3）。

Namenode（主节点）

单一 namenode（最初如此；后来支持 HA）保存元数据：

  • 命名空间：目录树、文件元数据、权限。
  • 块到 datanode 的映射（在启动时从 datanode 心跳重建）。
  • 操作日志（EditLog）和命名空间镜像（FsImage）。

所有元数据保存在内存中以提升速度。EditLog 持久化变更；FsImage 是周期性检查点。

单一 namenode 是单点故障（SPOF）和扩展上限。HDFS HA 通过备用 namenode 解决；HDFS Federation 将命名空间拆分到多个 namenode。

Datanode（工作节点）

每个 datanode：

  • 将块存储在本地磁盘上（每个块一个文件）。
  • 定期向 namenode 上报已存储的块（块报告）。
  • 定期向 namenode 发送心跳。
  • 为客户端提供读写服务。

副本放置策略

对于复制因子 3：- 副本 1：通常在写入方本地节点（若为 datanode）或随机节点。- 副本 2：与副本 1 位于不同机架。- 副本 3：与副本 2 位于同一机架（不同节点）。

该策略在以下方面取得平衡：- 持久性（可容忍机架故障）。- 写入带宽（仅一次跨机架写）。- 读取本地性（大多数块可在同一机架内访问）。

核心组件

组件 职责 关键细节

Datanode 存储并提供块数据。 数量众多；使用商用硬件。

Rebalancer 移动块以均衡磁盘使用。 后台运行。

写入路径

  1. 客户端请求创建文件： create(/path/to/file).
  2. Namenode 创建文件元数据，向客户端返回租约。
  3. 客户端写入数据；在本地缓冲直到积累一个块大小的数据。
  4. 客户端请求 addBlock 来自 namenode。
  5. Namenode 按放置策略选取 3 个 datanode，返回给客户端。
  6. 客户端建立流水线：client → datanode 1 → datanode 2 → datanode 3。
  7. 数据以小数据包（约 64 KB）形式流经流水线。
  8. 每个 datanode 本地持久化后转发给下一个节点。
  9. 确认信息沿流水线回传。
  10. 三个节点全部确认后，块提交完成。
  11. 后续块重复上述流程。
  12. 关闭时：客户端通知 namenode；namenode 最终确认文件。

面试用语：

写入通过 datanode 流水线传输，每组三副本仅消耗一次跨机架带宽。数据路径上无中央协调器。

读取路径

  1. 客户端打开文件：namenode 返回块 ID + 副本位置。
  2. 对每个块，客户端选择最近的副本（机架感知）。
  3. 直接从该 datanode 读取。
  4. 若 datanode 不可达或块损坏，尝试下一个副本。

读取在初始元数据获取后完全绕过 namenode。

追加

HDFS 支持在文件关闭后追加：以追加模式重新打开；namenode 授予租约；新数据通过流水线写入文件最后一个块（或新块）。

原始 GFS 支持原子记录追加：多个写入者可并发追加；系统串行化。HDFS 追加为单写入者模式。

复制与故障处理

每个 datanode 定期发送心跳。若约 10 分钟内无心跳，namenode 将其标记为死亡。

Datanode 死亡时：- 该 datanode 持有的所有块处于副本不足状态。- Namenode 调度重新复制：选取另一个 datanode，复制该块。- 以每块的复制因子为目标。

块完整性：- Datanode 定期扫描块以检测校验和失败。- 损坏块上报给 namenode；namenode 触发重新复制。

Namenode HA

原始单一 namenode 是单点故障（SPOF）。

HDFS HA 方案：- 主 namenode + 备用 namenode。- 通过 JournalNodes（基于 Quorum）共享编辑日志。- 主节点写入 JournalNodes；备用节点读取并应用。- ZooKeeper 或类似系统协调故障转移。- 主节点故障时，备用节点在秒内提升为主节点。

Namenode Federation

单一 namenode 的内存限制了集群规模（每个文件 + 块占一条记录）。

Federation 方案：- 多个 namenode，每个管理命名空间的一个子集。- Datanode 向所有 namenode 上报。- 客户端根据路径将请求路由到正确的 namenode。- 命名空间线性扩展。

Secondary Namenode

尽管名称如此，并非热备节点。它定期将 EditLog 合并到 FsImage，以保持检查点文件较小。若无此操作，EditLog 会无限增长，拖慢启动速度。

机架感知

Datanode 配置了机架信息。Namenode 使用机架信息用于：- 副本放置（跨机架）。- 读取本地性（优先选择更近的副本）。- 重新复制（避免将两个副本放在同一机架）。

扩展与分区

关注点 策略
Namenode 内存 更大的块大小（每字节存储元数据更少）；HDFS Federation。
Datanode 容量 添加更多节点；Rebalancer 重新分配。
网络 流水线写入；机架感知放置。
热点文件 增加复制因子；使用缓存。
磁盘故障 重新复制。

一致性模型

层级 一致性
命名空间操作 强一致性（单一 namenode 串行化）
块写入（流水线期间） 按块原子化——全部 3 个确认或全部失败
追加 单写入者；读取可见已提交数据
读操作 以上次完成的流水线确认为准查看数据

故障模式与缓解措施

故障模式 缓解措施 监控指标
Datanode 死亡。 从存活副本重新复制。 副本不足块数量。
磁盘故障。 块扫描器检测；重新复制。 损坏块数量。
网络分区。 Datanode 被标记为死亡；随后以过期块重新发现。 过期 datanode 数量。
Namenode 死亡（无 HA）。 集群不可用，直至重启。 Namenode 正常运行时间。
Namenode 死亡（有 HA）。 备用节点在秒内提升。 故障转移时间。
EditLog 损坏。 JournalNodes 提供基于 Quorum 的安全保障。 Journal 节点健康状态。
流水线中 datanode 缓慢。 流水线阻塞；客户端可能重试。 慢写入检测。
热点读取。 使用缓存；增加该文件的复制因子。 每文件读取速率。
块报告风暴。 错开上报；限流。 块报告速率。
磁盘使用不均衡。 Rebalancer。 磁盘使用分布。

运维指标

正确性：

  • 副本不足的块。
  • 缺失块（零）。
  • 陈旧的块报告。

容量：

  • 集群已用与可用空间。
  • 每个 datanode 的利用率。
  • Namenode 内存。

业务层面：

  • 文件总数；总大小。
  • 读/写吞吐量。

需要指出的权衡

  • 较大的块大小可减少 namenode 负载，但对于小文件会浪费空间。
  • 单个 namenode 结构简单，但存在单点故障/瓶颈；HA + Federation 可同时解决这两个问题。
  • 仅追加模型简化了一致性维护；随机写入则需要加锁。
  • 跨机架写入消耗带宽，但提供了机架故障的容灾能力。
  • 3 倍副本持久性强但成本高；纠删码以计算成本换取空间节省。
  • 命名空间具有强一致性，但块位置信息为最终一致性。

常见陷阱与更优做法

陷阱 更优做法
将 HDFS 当作 POSIX 文件系统使用。 仅追加；大文件；不支持随机写入。
大量小文件。 合并文件，或使用 HAR/SequenceFile。
没有 HA 的单个 namenode。 Standby namenode + JournalNodes。
随机副本放置。 机架感知。
命名空间永远是单一的。 随规模增长采用 HDFS Federation。
冷数据不使用纠删码。 冷数据层用 EC，热数据层用 3 倍副本。

面试官可能深入探讨的方向

深入方向 优秀回答要点
块放置 机架感知；1+2+0 模式。
写入 pipeline 顺序经过各副本传输。
Namenode HA Active/Standby + JournalNodes + ZK。
联邦化 多命名空间；客户端按路径路由。
小文件问题 解决方案；HAR。
纠删码 对冷数据使用 Reed-Solomon 编码。

Staff 级别回答检查清单

  • Namenode + datanodes 分离。
  • 大块（128 MB）。
  • 机架感知 3 倍副本。
  • Pipeline 写入。
  • 从最近的副本读取。
  • 追加写入。
  • 通过 Standby + JournalNodes 实现 HA。
  • 通过 Federation 实现扩展。

面试用语

HDFS 采用主节点 + 工作节点设计：namenode 管理命名空间和块映射；datanodes 存储并提供数据服务。大块可降低 namenode 压力。机架感知 3 倍副本在持久性与带宽之间取得平衡。写入通过副本 pipeline 传输。仅追加模型使一致性维护简单。单 namenode 是历史上的单点故障；通过 Standby 实现的 HA 解决了可用性问题；Federation 解决了扩展问题。

2. 模拟面试

面试官： 描述 HDFS。

候选人： 主从设计：namenode 负责元数据，datanodes 负责数据。文件被切分为 128 MB 的块；每个块在机架间复制 3 份。Namenode 在内存中保存命名空间和块到 datanode 的映射；datanodes 定期通过心跳上报块报告。客户端访问 namenode 获取元数据，然后直接与 datanodes 交互读写数据。写入通过副本 pipeline 传输。读取来自最近的副本。仅追加；设计用于在商用硬件上存储 PB 级的大文件。

面试官： 介绍一次写入的完整流程。

候选人： 客户端在 namenode 上调用 create，获得一个租约。当客户端写满一个 128 MB 的块后，调用 addBlock；namenode 选取 3 个 datanodes（机架感知：1 个本地，2 个在不同机架上）。客户端建立写入 pipeline——client → datanode 1 → datanode 2 → datanode 3。数据以小数据包的形式流动；每个 datanode 在本地持久化后转发给下一个。确认响应逐级回传。所有 3 个节点确认后，块才算提交。后续的块重复此过程。关闭操作向 namenode 最终确认文件完成。

面试官： 块放置策略？

候选人： 机架感知。副本 1 通常位于写入方的本地节点（如果写入方不是 datanode 则随机选择）。副本 2 位于与副本 1 不同的机架上。副本 3 位于与副本 2 相同的机架上，但不同节点。在持久性（可容忍机架故障）、带宽（仅一次跨机架写入）和读取本地性之间取得平衡。

面试官： 为什么使用大块？

候选人： 减少 namenode 元数据。Namenode 对每个文件和每个块各保存一条记录；若使用 4 KB 的块，namenode 元数据量会急剧膨胀。使用 128 MB 的块，1 TB 的文件只需 8000 个块。权衡：小文件每个都会浪费相当于一个块的元数据。HDFS 不善于处理大量小文件。

面试官： Namenode 是单点故障。

候选人： 最初确实如此。HA 解决方案：Active + Standby namenode。通过 JournalNodes 共享编辑日志（基于 Quorum——在确认前写入大多数 JournalNodes）。Active namenode 将 EditLog 条目写入 JournalNodes；Standby 读取并重放这些条目。ZooKeeper 或类似组件协调故障转移。Active 故障后，Standby 在数秒内完成提升。

面试官： 超出内存时如何扩展 namenode。

候选人： HDFS Federation。将命名空间拆分到多个独立的 namenode 上——例如， /user 在 namenode 1 上， /data 在 namenode 2 上。每个 namenode 管理其自己的命名空间子集；datanodes 向所有 namenode 上报块信息。客户端按路径路由。命名空间线性扩展。

面试官： 读取路径。

候选人： 客户端向 namenode 请求文件的块 ID 和副本位置。对于每个块，客户端选择最近的副本（机架感知），直接从该 datanode 读取数据。如果节点不可达或校验和校验失败，则尝试下一个副本。一旦获取元数据，namenode 完全不在数据读取路径中。

面试官： Datanode 死亡。

候选人： Namenode 在丢失约 10 分钟的心跳后将其标记为死亡。该 datanode 上的所有块现在处于副本不足的状态。Namenode 安排重新复制：对于每个块，选取一个健康的 datanode，从幸存的副本中复制数据。持续进行直到达到目标副本数。此过程在后台进行，不阻塞读写操作。

面试官： 支持追加写入吗？

候选人： HDFS 支持文件关闭后追加：以追加模式重新打开文件；namenode 授予写入租约；新数据像继续写入文件一样通过 pipeline 传输。每个文件仅允许单个写入者。原始 GFS 支持原子记录追加——多个并发追加者，系统负责串行化——HDFS 不支持。

面试官： 小文件问题。

候选人： 每个文件和块都会占用 namenode 内存。大量小文件会使 namenode 元数据急剧膨胀。缓解措施：HAR（Hadoop Archive）将小文件合并；SequenceFile 或 Avro 容器文件将许多记录存储在一个 HDFS 文件中；HBase 作为 HDFS 上层来处理大量小记录。

面试官： 纠删码？

候选人： Reed-Solomon 风格。不使用 3 倍副本，而是将每个块编码为 k+m 个数据块；可容忍 m 个数据块丢失。例如，6+3 方案的开销为 1.5 倍，而非 3 倍。用于冷数据；热数据仍使用 3 倍副本以实现读取并行性。写入时增加了编码计算开销，读取时（需要重建时）也有额外计算开销。

面试官： 块完整性。

候选人： 每个块都有校验和。Datanode 的 BlockScanner 定期读取块并验证校验和。发现损坏时：上报给 namenode；触发从另一个副本重新复制。读取时也会验证校验和；不匹配时尝试另一个副本。

面试官： 总结。

候选人： Namenode（元数据）+ datanodes（数据）。大块减少元数据。机架感知 3 倍副本。Pipeline 写入。读取最近副本。仅追加。通过 Standby + JournalNodes 实现 HA；通过 Federation 实现命名空间扩展。优势在于规模、简洁性和对商用硬件故障的弹性；劣势在于小文件问题和固有的单写入者模型。

3. 练习备注

五分钟版本

  1. Namenode + datanodes。
  2. 大块（128 MB），3 倍副本。
  3. 机架感知放置。
  4. Pipeline 写入；读取最近副本。
  5. 仅追加大文件。

  8. 小文件问题；冷数据使用纠删码。

一分钟结尾陈述

HDFS 是主节点 + 工作节点架构：namenode 持有元数据，datanodes 持有数据。大块（128 MB）和 3 倍机架感知副本。写入通过副本 pipeline 传输。读取绕过 namenode。仅追加模型。原始 namenode 单点故障问题通过 HA（Standby + JournalNodes）解决；命名空间扩展问题通过 Federation 解决。优势在于商用硬件上的规模能力和对磁盘故障的弹性；劣势在于小文件问题和缺乏随机写入支持。

最佳面试用语

  • "大块缩减 namenode 元数据；权衡是小文件的浪费。"
  • "机架感知副本：通过一次跨机架写入实现跨机架的持久性。"
  • "写入通过副本 pipeline 传输；读取绕过 namenode。"
  • "通过 Standby + JournalNodes 实现 HA；通过 Federation 实现命名空间扩展。"
  • "仅追加模型是简化设计的核心假设。"
