Facebook 直播评论——直播聊天扇出（详解）

2026/06/01 · Common Backend Systems
目录
目录

难度： Medium · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建 Facebook（FB）直播或 Twitch 风格直播的实时评论系统。一位主播直播，数百万观众观看，观众发布的评论几乎实时显示给所有人。热门直播活动有 100 万并发观众，每秒 5000 条评论；评论必须在约 1 秒内显示。

1. 需求（约 5 分钟）

功能需求 - 观众在直播中发布评论。 - 所有观众在约 1 秒内看到新评论，顺序大致与发布顺序一致。 - 新加入的观众可以看到最近几百条评论作为补全内容。

我会推迟实现表情反应、投票、变现和审核员工具。

非功能需求 - p95 评论发布到显示延迟 < 1 秒。 - 每个直播支持 100 万以上并发连接；整体上有多个直播同时进行。 - 最终顺序即可——不要求所有观众看到严格的全局顺序。 - 容错性：偶尔丢失某条评论是可接受的；连续 10 秒无法投递任何评论则不可接受。 - 背压：当评论速度超过扇出速度时，丢弃最旧的评论而非拖垮系统。

容量估算 - 超级直播：100 万观众 × 每分钟 1 条评论 = 每秒 1.7 万条写入；读取扇出 100 万 × 5000 = 每秒 50 亿条消息——以高流量向每位观众发送每条评论是不现实的，详见深度解析。 - 系统必须限制每位观众显示的评论数量。若每位观众每秒接收 5 批次更新，每批约 2 KB，则单个超级直播的出口带宽仍约为每秒 10 GB。

2. 核心实体（约 2 分钟）

  • Stream — stream_id主播，started_at，状态。
  • 评论 — comment_id，stream_id，user_id，文本，时间戳。
  • 观众连接 — connection_id，stream_id，user_id。

3. API / 协议（约 5 分钟）

WS  /v1/streams/{id}/comments
  ← client sends:  { post: text }   (idempotency-key)
  → server pushes: { comment }      (batched)

HTTP:
GET /v1/streams/{id}/comments/recent?limit=200    → backfill for join

从 token 获取用户信息；按每用户每直播进行限流。

WebSocket 连接携带服务端游标 {stream_id, last_comment_id, sample_policy}。重连时，客户端请求 last_comment_id之后的近期评论，然后恢复实时推送。

5. 高层设计（约 10–15 分钟）

  1. 边缘 WebSocket 网关。 持有数百万观众连接。按直播分片：hash(stream_id) → 边缘网关集合。
  2. 评论摄入。 评论命中任意边缘网关 → 转发到 直播归属分片 （负责该直播的分区）→ 摄入服务分配服务端时间戳 + comment_id，写入持久日志（每直播一个 Kafka topic 或按直播分片）。
  3. 扇出层。 每个直播的评论 topic 由扇出服务消费，推送到所有持有该直播观众的边缘网关。边缘网关再推送到本地 WebSocket 连接。
  4. 持久化存储。 追加写入——以 (stream_id, ts) 为键的 Cassandra，或仅使用长保留期的 Kafka topic。用于补全和重放。
  5. 数据回填。 当观众加入时，边缘网关在打开 WebSocket 推送之前，通过一次独立的 HTTP 调用向其提供最近约 200 条评论。
  6. 限流/反垃圾评论管道。 发帖时使用每用户令牌桶。广播前进行脏话/黑名单过滤。审核员操作通过同一管道处理。

端到端叙述：Alice 正在观看 → 她的手机通过 WebSocket 连接到直播 S 的边缘网关 G1 → Bob 发布"🔥🔥🔥" → 他的边缘网关 G3 转发到 S 的直播归属分片 → 摄入服务打上 ts=…，comment_id=C99 → 发布到 Kafka topic stream.S.comments → 扇出服务消费，推送到 G1、G2、G3（所有持有 S 观众的网关）→ 每个网关广播到其本地 WebSocket 连接 → 500 毫秒内 Alice 看到 Bob 的评论。

发帖路径：

  1. 边缘网关验证鉴权、限流和消息大小。
  2. 它将请求转发到以 stream_id为键的流主节点分片，该分片分配 comment_id = (stream_epoch, sequence).
  3. 摄入服务在审核通过后并行写入 Kafka 和近期回填 Redis。
  4. 扇出工作节点消费流分区，并将批次推送到 stream_edges:{stream_id}.
  5. 每个网关写入本地 WS 环形缓冲区；慢速客户端丢弃旧批次，保留最新批次。

加入/读取路径：

  1. 观看者调用 GET /comments/recent?limit=200；边缘节点首先从 recent:{stream_id} 来自 Redis。
  2. Redis 未命中时，边缘节点回退到 CommentStore[(stream_id, comment_id DESC)]，过量获取 300 条，过滤已删除/已审核的评论，并重新填充 Redis。
  3. 观看者以 {stream_id, last_comment_id}打开 WS；网关在 stream_edges:{stream_id}:{region} 心跳 TTL 为 10-30 秒。
  4. 直播批次从扇出到达；网关对 last_comment_id进行去重，追加到每个连接的环形缓冲区，并推进服务端游标。
  5. 重连时，游标请求 last_comment_id之后的评论；若 id 早于保留时间，服务器返回 lossy_resume 以及最新的 200 条评论。

6. 深度探讨（约 10 分钟）

百万观看者规模下的扇出。 单一 broker 向 1M 个连接推送会崩溃。分层扇出：每个 region/AZ 一个 broker；每个 broker 持有一部分观看者（约各 1 万个）；一个有 N 个分区的 Kafka topic 供给 N 个 broker。Broker 不广播单条评论，而是每 200 毫秒批量广播累积的评论。以新鲜度换取吞吐量。

容量计算：17k 条评论/秒 × 200 B，写入 Kafka 的数据量仅约 3.4 MB/秒（针对超大直播流）；扇出才是真正的怪兽。将每条评论发送给 1M 观看者意味着每秒 170 亿次投递。将每位观看者限制为 5 批/秒 × 2 KB，出口流量约为 10 GB/秒，数量仍然庞大，但配合区域边缘网关是可行的。主要成本是网络扇出和维持连接的内存，而不是评论存储。

单个持续超大直播流的存储计算：直播 topic 写入约 3.4 MB/秒原始数据，RF=3 时约 900 GB/天，因此 7 天重放约 6 TB。持久化评论约 300 B/条，约 440 GB/天原始数据，RF=3 时约 1.3 TB/天；大多数直播流时间较短，因此保留策略比峰值大小更重要。 recent:{stream_id} 数据量很小：1,000 条评论 × 300 B 约 300 KB/流，10 万个活跃流约 30 GB 原始数据。连接内存才是关键瓶颈：1M 观看者 × 100 KB 环形缓冲上限 = 100 GB，分布在各网关上，因此环形缓冲必须有界，且批次对象应在每个网关上按流共享而非按连接复制。100 万活跃评论者的限流状态仅约 32 MB。

为什么选 Kafka 而非 Redis Pub/Sub？ Redis Pub/Sub 速度快，但不可重放，使回填/重连变得麻烦。Kafka/Pulsar 提供有序分区和从偏移量重放的能力。Cassandra 适合 (stream_id, comment_id) 回填，因为读取是近期范围扫描；MySQL 需要对热门流进行大量分片。

要做好回答"为什么不直接从 Kafka 推送到每个网关？"的准备。答案是：Kafka 是持久化流，不是面向每个连接的投递系统。扇出层可以批处理、采样、优先处理主播/审核通道，并应用区域背压，而无需将 Kafka 分区变成连接管理器。

有损投递是可接受的。 若某观看者的 WS 跟不上，则丢弃最旧的评论而非缓冲。直播流中最新的 100 条才是关键。采用带尾部丢弃策略的每连接环形缓冲。

排序。 在多百万规模下，若不通过单台机器串行化，就无法实现严格的流内有序。近似方案：摄入时对每条评论分配服务端时间戳，扇出 broker 按时间戳顺序批处理。客户端按到达顺序显示并附带时间戳——少量乱序在视觉上无妨。不要承诺全局有序。

热门流的背压。 当评论速率超过展示速率时，进行采样。对观看者展示每 5 条评论中的 1 条（随机采样）；对审核员展示全部。可提及"Twitch 聊天慢速模式"作为先例。

评论的两种视图。 持久化日志 (stream_id, comment_id) 是重放、审核和直播结束后归档的事实来源。近期 Redis 列表 recent:{stream_id} 是面向加入和重连的读优化窗口。直播扇出批次是从同一日志派生出的第三种有损投递视图；丢失一个批次只影响某位观看者的体验，不影响权威评论记录。

热门流降级：

阶段 模式 数据源 观看者效果
正常 完整批次 200 毫秒批次中包含所有评论 接近完整的聊天
hot 采样批次 随机/按分数采样 具有代表性的聊天
更热 慢速模式 每用户发帖桶收紧 写入量减少
极端 优先通道 主播/VIP/审核评论完整保真；公开评论采样 重要评论得以保留
边缘缓慢 尾部丢弃 每连接环形缓冲丢弃最旧内容 最新聊天保持直播

加入时的回填。 在 WS 打开之前从 Cassandra（以 stream_id 为键）获取最后 200 条评论。WS 打开后，边缘网关开始直播推送，通过 comment_id 对任何重叠内容去重。

重连路径：HTTP 获取 recent:{stream_id} 或 Cassandra 范围查询（条件为 comment_id > last_seen）；仅在客户端获得新游标后才打开 WS。若游标早于保留时间，返回最后 200 条并将流标记为 lossy_resume.

热门主播 / VIP 评论。 无论采样策略如何，主播自己的评论和置顶消息都以完整保真度推送。以观看者订阅的独立通道实现。

持久性。 热门流使用每流独立 Kafka topic，保留 7 天。长尾流使用单个 Kafka 集群，按分区处理多个流。7 天后将冷数据存储到对象存储，用于合规/回放。

延迟预算。 发帖到展示 1 秒预算：客户端→边缘 50 ms，边缘→分片 50 ms，摄入写入 20 ms，扇出批次窗口 200 ms，broker→边缘 50 ms，边缘→客户端批量 200 ms。符合要求。

连接扩展。 商用网关上的 1M 并发 WS 连接：每台约 5-10 万；每个流-区域约 20 台网关。在实践中，调优内核限制（文件描述符、临时端口）是真实工作量。

热键、区域和冷启动。 超大直播流会使 stream_id 成为热键，因此按区域乃至按网关桶拆分 stream_edges:{stream_id} ： stream_edges:{stream_id}:{region}:{bucket}。若单个主节点分片无法承受每秒 1.7 万条评论的盖章处理，则按 (stream_id, lane) 将摄入分为多个通道并生成 comment_id = timestamp + lane + sequence；客户端按时间戳显示并容忍小幅乱序。区域内观看者应优先接收本地扇出；跨区域评论复制可延迟几百毫秒，对公共聊天可接受，但对审核员删除操作不可接受，因此审核/控制事件使用优先通道。流开始时，在主播上线时预创建直播 topic 分区、近期列表和边缘注册条目；不要等到第一批百万观看者进入才触发冷分配。

审核。 主播侧屏蔽列表 + ML 毒性评分。被标记的评论对公开广播隐藏，但对审核员可见。

我会推迟的内容。 表情/表情爆发、超级留言/付费置顶、多语言翻译、实时分析、持久化的直播结束后评论归档 UI。

面试核心要点： 评论写入是简单的；扇出才是产品核心。我会围绕有界的每观看者投递来设计系统，并将降级设计为有意为之：对公共聊天采样，绝不丢弃主播/审核消息，并保持直播的临场感。
