Yelp — Local Business Search & Reviews (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Medium · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建 Yelp。用户打开应用，搜索"附近的拉面"，获得附近餐厅的排名列表，包含照片、评分和评价。月活跃用户 1 亿，约 1000 万家商户，累计约 2 亿条评价。读多写少；评价陆续写入。

1. 需求（约 5 分钟）

功能需求 - 按关键词 + 位置搜索商户，可选过滤（类别、评分、价格、当前营业中）。 - 查看商户页面：照片、营业时间、地址、综合评分、评价。 - 撰写和查看评价。

我会暂不考虑预订、商户工具和广告投放的设计。

非功能需求 - 搜索结果 p99 < 300 ms。 - 读多写少：平均约 10,000 次搜索/秒。 - 商户资料（营业时间、地址）强一致性；综合评分（已缓存）最终一致即可。 - 评价欺诈是真实存在的；即使不实现，标记流水线也很重要。

容量估算 - 1000 万商户 × 5 KB 资料 = 50 GB。单分片即可容纳；分片用于读取扇出。 - 2 亿条评价 × 2 KB = 400 GB。 - 照片：约 10 亿张，每张约 500 KB → 对象存储约 500 TB。 - 搜索索引规模尚可：1000 万商户 × ~2–5 KB 索引字段/词条 ≈ 主数据 20–50 GB，含副本和倒排索引开销可能 3–5 倍。主要成本是照片/CDN，而非元数据。 - 评价写入量远低于读取量：即便每天 500 万条评价，平均也只有约 60 次写入/秒；评价存储的设计目标是商户页面局部性和审核，而非写入扩展性。

2. 核心实体（约 2 分钟）

  • 商家 — business_id，name，geo，hours，categories，phone。
  • 评价 — review_id，business_id，user_id，rating，text，photos，created_at。
  • User — user_id，profile。
  • 图片 — photo_id，business_id，url，uploader。

3. API（约 5 分钟）

GET   /v1/search?q=ramen&lat=&lng=&filters=...&cursor=...   → Business[]
GET   /v1/businesses/{id}                                    → Business + recent_reviews
GET   /v1/businesses/{id}/reviews?cursor=...                 → Review[]
POST  /v1/businesses/{id}/reviews     { rating, text, photos[] }   → { review_id }

从令牌中获取 User。

搜索使用基于游标的分页；游标编码了 query_hash, geo_cell, last_score，以及 last_business_id ，因此即使评分略有漂移，第 2 页也保持稳定。

5. 高层设计（约 10–15 分钟）

  1. 商户目录服务。 权威数据存储（MySQL 或 DynamoDB，按 business_id分片）。对热门商户使用读穿透缓存。
  2. 为每个允许查看的组（公开、好友列表 ID）索引一个合成词条。查询时将用户 Token 与搜索词条做 AND 运算。\n - 我默认对好友采用查询时过滤，对公开/私密布尔值采用索引 Token。 Elasticsearch（或 OpenSearch）集群，对商户建立全文索引（名称 + 类别 + 标签 + 城市），并对位置字段建立地理空间索引。
  3. 地理 + 关键词查询。 ES 查询结合地理距离过滤 + 关键词 + 过滤器 + 自定义评分（评分、评价数、距离、时效性）。
  4. 基于 Elasticsearch 的菜品检索——对菜品名称、餐厅名称、标签进行全文搜索；对父餐厅位置进行地理距离过滤；按贝叶斯评分 + 距离 + 时效性排序。 写入评价（按 business_id 分片，使同一商户的所有评价集中存储）。向 Kafka 发布 review.created 到 Kafka。
  5. 事件。 流处理器消费评价事件，重新计算每商户评分 + 数量，写回目录。同时将欺诈评分任务入队。
  6. 照片服务。 通过 API 上传，存储到 S3，异步生成缩略图，CDN 全面前置。
  7. CDN + 边缘缓存。 搜索结果可按（查询、地理格网单元、过滤条件）短 TTL 缓存；商户页面可按 business_id.

端到端叙述：Alice 搜索"附近的拉面"→ API 网关 → 搜索服务向 ES 查询（地理距离 < 5 km，全文"ramen"，过滤 open_now=true → ES 返回前 50 条 → 服务用缓存的商户资料填充 → 返回前 20 条含照片结果。Alice 点击某商户 → 目录查询（大概率命中缓存）→ 从评价服务获取最新 20 条评价。

6. 深度探讨（约 10 分钟）

具体存储契约。

存储 键/数据结构 条目大小 + TTL 分片键
商家目录 business:{business_id} document 约 5 KB，持久保留 business_id
搜索索引 OpenSearch/ES 文档，包含文本字段 + geo_point + 过滤字段 约 2–5 KB 已索引，重建/已复制 geo_cell/business_id 如需路由
评论 reviews_by_business:{business_id} 按 created_at 约 2 KB/条评价，持久保留 business_id
用户评价索引 reviews_by_user:{user_id} pointers ~32 B/评论指针 user_id
评分聚合 rating:{business_id} → {sum,count,bayes_avg} ~32 B，保留 business_id
近期评论缓存 recent_reviews:{business_id} Redis list 前20条约40 KB，TTL 5分钟 business_id
照片 s3://photos/{business_id}/{photo_id}_{size} 原图约500 KB，缩略图保留 business_id

存储估算：商户目录和评论数据为数百GB，照片数据为数百TB。搜索是一个只读索引： 10M × 5 KB × 3 replicas 倒排索引开销前约为150 GB；与照片存储相比仍属小规模。成本的关键在于缩略图尺寸、CDN缓存命中率以及照片的生命周期分层管理，而非精简业务行字节数。

搜索路径追踪。 Client → 搜索API对查询和地理位置进行规范化 → OpenSearch执行地理距离过滤、关键词匹配、结构化过滤及前100名评分 → 搜索服务并行从缓存/KV中补全业务文档和评分聚合 → 获取缩略图URL → 返回前20条及游标。翻页时复用相同的查询哈希， last_score 因此分页基于评分/游标，而非偏移量。

冷搜索路径：若查询缓存未命中且OpenSearch响应较慢，API可降级至 popular:{geo_cell}:{category} 从Redis中取数，再补全对应商户信息。该结果个性化程度较低，但仍能在500 ms内返回有用结果。游标携带 {mode=popular_fallback, geo_cell, category, offset} 因此分页在同一降级模式下保持一致。

在单次查询中同时支持地理位置和关键词。 Elasticsearch配合 geo_point 字段可很好地处理这一场景——先执行地理距离过滤（开销低），再进行全文BM25评分，最后用script_score做自定义排名。备选方案：独立的地理索引（S2单元 → business_ids）与关键词索引取交集——更多的环节意味着更多的故障点，可作为备选方案提及。

为何选用ES/OpenSearch而非带地理扩展的关系型数据库：该工作负载在低延迟路径中同时涉及全文相关性、过滤、地理半径以及TopK排名。PostGIS在空间数据存储方面表现出色，但BM25重排和倒排索引分页才是搜索问题的核心所在。

需要准备好回答"为何不只用S2单元？"——S2非常适合附近候选集的生成，但若不重新构建一套搜索引擎，它无法对"ramen"、"ramn"等拼写容错查询以及类目同义词进行排名。

排序。 输入因素：文本相关性、距离、评分（采用贝叶斯平均并结合全局先验，避免一条5星好评胜过拥有200条评论的4.5分商户）、时效性、评论数量以及商户当前是否营业。离线训练Learning-to-Rank模型，以ES重排脚本的形式对前100条结果进行打分。

评论写入路径。 评论按 business_id 分片，因此商户页查询为单分片读取。用户可修改自己的评论；行以 (business_id, user_id) 为键，确保每位用户对每家商户最多只有一条评论。

写入路径：client → 评论服务通过唯一 (business_id,user_id) → 写入评论行 → 写入指针至 reviews_by_user → 发布 review.created → 流式聚合器异步更新 rating:{business_id} 和搜索索引 → 针对 recent_reviews:{business_id} 的缓存失效为尽力而为。

两个索引，共用同一条评论： reviews_by_business:{business_id} 针对商户页做了优化，而 reviews_by_user:{user_id} 则针对用户主页/历史记录及内容审核做了优化。两者均指向相同的评论内容；均不应复制照片二进制数据或超出查询所需的大段文本。

无需重新计算的聚合评分。 聚合管道。 sum(rating), count(review) 每家商户保存总和与计数；聚合值 = sum/count。更新仅需一次键值写入，无需扫描全部评论来计算平均值。

热点商户页面。 某热门餐厅有1000个并发访客。将商户资料缓存至CDN（TTL 60秒），近期评论缓存至Redis（TTL 5分钟）。写入时的缓存失效为尽力而为——对于评论而言，最终一致性是可接受的。

面试要点："商户资料需要强一致性；评分和近期评论需要高时效性，这是两种不同的约束。"

照片服务。 上传 → S3 → 图片处理服务生成4种尺寸 → 发布至CDN。原图保留以支持超级缩放；CDN通过URL约定按需提供合适尺寸。

欺诈检测。 启发式规则：同一IP提交多条评论、账号注册时长、评论发布频率、内容与已知模板的相似度。被标记的评论会被隐藏但不删除；人工审核队列中的待处理内容。

按"当前营业"过滤。 每家商户存储每周营业时间。在ES中将 open_hours_minute_of_week 索引为稀疏位图，并与当前分钟进行匹配。每分钟刷新一次。

优雅降级状态机。

情形 模式 数据源 延迟预算 回退
热门搜索查询 缓存结果页 query+geo+filter 缓存 Redis 有序集合 短TTL 30–60秒；点击时加载最新资料
OpenSearch响应慢 降级搜索 缓存查询或仅地理位置索引 <500 毫秒 返回附近热门商户，内部标记为陈旧数据
热点商户页面 CDN/Redis 商户目录缓存 + 近期评论缓存 <200 ms 提供最多60秒的陈旧资料；评论最终追上
评论写入突发 异步聚合 评论行数据 + Kafka 写入确认 <200 ms 评分计数有滞后；绝不丢弃评论行
照片CDN未命中 回源拉取 对象存储 几秒延迟可接受 展示占位缩略图；页面仍然可用

面试要点："Yelp搜索是一个搜索索引问题，但Yelp数据正确性是一个商户目录和评论问题；索引可以有延迟，但用户的评论不能消失。"

我会推迟的内容。 预订系统（OpenTable集成）、商户身份认证、结构化广告投放、个性化推荐（"你可能也喜欢"）、本地赞助位。
