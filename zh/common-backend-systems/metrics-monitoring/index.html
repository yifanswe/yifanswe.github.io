Metrics & Monitoring — Datadog / Prometheus Class (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Hard · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建一个指标与监控系统。主机每 10 秒上报时序指标（计数器、仪表盘、直方图）。工程师查询图表并设置条件告警。100 万台主机 × 每台约 500 个指标 = 5 亿条唯一序列 × 每分钟 6 个数据点 = 每分钟 30 亿个数据点（约 5000 万/秒）。查询跨度从分钟级到月级。

1. 需求（约 5 分钟）

功能需求 - 主机推送指标数据点 (name, tags{}, value, ts)。 - 查询：任意标签过滤 + 时间窗口聚合（avg、p99）。 - 告警：针对指标存储定期执行规则，满足条件时触发。

我会将日志、链路追踪和 APM 关联排除在外——它们是同级别的独立系统。

非功能需求 - 持续采集 5000 万数据点/秒。 - 典型仪表盘查询（最近 1 小时聚合）p99 < 1 秒。 - 单可用区故障不丢数据。 - 保留策略：原始数据 15 天，降采样数据 1 年。 - 查询允许最终一致性——轻微延迟（约 15 秒）可接受。

容量估算 - 5000 万点/秒 × 24 B/点 ≈ 1.2 GB/秒原始数据 → 约 100 TB/天 → 按 15 天原始保留约 1.5 PB。 - 压缩后（典型 10×）：约 150 TB。 - 基数：5 亿条唯一序列。基数是隐性杀手。 - 倒排索引小于原始样本，但体量仍不可忽视：若每条序列有约 6 个标签，每个 series_id 约 8 B，5 亿条序列在压缩/开销之前约占 24 GB。主要成本仍是原始/温层样本存储。

2. 核心实体（约 2 分钟）

  • Series — (metric_name, tags{}) → series_id。唯一组合。
  • Datapoint — (series_id, ts, value).
  • 聚合层 — 降采样数据（如 1 分钟、1 小时、1 天）。
  • 告警规则 — query, threshold, window.

3. API（约 5 分钟）

POST  /v1/ingest                    // host agent → batched metrics
GET   /v1/query?q=...&from=...&to=...    → time-series result
POST  /v1/alerts                    { rule }   → { alert_id }

查询语言：PromQL 风格 — avg_over_time(http_latency{service="checkout"}[5m]).

查询游标很少面向用户，但长表查询应携带 {tenant, query_hash, shard_checkpoint, time_block} 以便部分扫描可以恢复而不重复索引展开。

4. 数据流（约 5 分钟）

主机代理每 10 秒批量上报指标 → 采集层 → 写入时序存储（按 series_id + 时间分片）→ 降采样任务归并旧数据 → 查询层从合适的层级读取 → 告警运行定时查询。

5. 高层设计（约 10–15 分钟）

  1. 主机代理。 轻量级守护进程从本地 exporter 采集数据，批量处理后每 10 秒上报。采集失败时退避重试；本地磁盘缓冲（有限容量）。
  2. 采集层。 无状态采集集群，位于负载均衡器之后。校验并规范化标签，查找/创建 series_id （在 Redis 支持的基数缓存中），然后转发至 Kafka。
  3. Kafka。 按 series_id （或 hash(metric_name + tags)）。保留时间足够长，以便在存储故障时重放（24 小时）。
  4. 时序存储。 自研或开源方案（Prometheus TSDB、InfluxDB、M3、Cortex）。存储布局： - 热层：最近 24 小时存储于 SSD，细粒度（10 秒）。 - 温层：1 分钟降采样，保留 15 天。 - 冷层：1 小时降采样，存储于低成本对象存储，保留 1 年以上。 热层： 最近 24 小时存储于 SSD，细粒度（10 秒）。 - 温层： 1 分钟降采样，保留 15 天。 - 冷层： 1 小时降采样，存储于低成本对象存储，保留 1 年以上。
  5. 索引。 每分片倒排索引： tag=value → set of series_ids。支持基于标签过滤的查询。
  6. 降采样器。 流式任务消费原始 Kafka 数据，将 1 分钟和 1 小时聚合写入温层/冷层。
  7. 查询引擎。 解析 PromQL，通过索引展开标签过滤，扫描相关数据块并聚合。将过滤/聚合下推至各分片执行。
  8. 告警评估器。 定时任务驱动的 worker，定期执行每条规则的查询，应用阈值后触发通知。通知投递不在设计范围内。

端到端流程：主机上报 http_latency{service=checkout, host=web-42, route=/buy} = 0.230 → 采集层分配 series_id S99 → 写入 Kafka S99 对应的分区。存储节点消费数据，将数据点追加到热层中 S99 的块里。工程师查询 avg by service of http_latency{service=checkout} 最近 5 分钟 → 查询引擎在索引中查找标签 service=checkout → 获取约 500 条序列 → 扫描每条序列最近 5 分钟 → 聚合求平均 → 300 毫秒内返回结果。

6. 深度探讨（约 10 分钟）

基数——无声的杀手。 将 request_id 或 user_id 放入标签会导致序列数爆炸。防御手段：- 基数限制 每指标基数限制（超过阈值后拒绝新序列）。- 隔离 可疑标签键（自动检测爆炸，触发告警）。- Schema： 将标签视为有界集合；高基数数据应归入日志/链路追踪，而非指标。

存储布局。 序列行按时间列式存储。单个 block 的布局：- 时间列（delta 编码，再经 varint）。- 值列（Gorilla XOR 编码——通常低于 1 字节/数据点）。- 压缩可达原始布局的 10 倍。每个序列的 block 在时间窗口结束后即封闭且不可变；一次写入，多次读取。

为何使用时序数据库而非通用 Cassandra 行：Cassandra 能承受高写入吞吐量，但在标签过滤和时间窗口查询时，需要列式压缩、block 剪枝、降采样以及专用倒排索引。单集群下 Prometheus 表现优秀，但在 500M 序列规模下，我会选择 M3/Cortex/Thanos 风格的水平分片系统。

为何在中间引入 Kafka：从摄入直写到 TSDB 可减少跳数，但会使回放和区域故障恢复更困难。Kafka 提供 24 小时的缓冲，按 tenant + series_id分区，并让降采样消费同一数据流。代价是一个成本极高的临时缓冲区，因此保留时间必须短且需压缩。准备好回答"为何不只用 ClickHouse？"——ClickHouse 擅长分析扫描，但针对新鲜的逐序列数据进行告警以及处理高基数标签索引，通常需要 TSDB 形态的热层。

分片。 Hash series_id 到某个分片。每个分片拥有一部分序列。跨可用区 3 副本保障高可用。查询跨分片扇出；每个分片的工作并行执行。

热写与不可变读。 近期数据写入 memtable；每隔几分钟封闭为磁盘上不可变的 block（LSM 树模式）。查询层读取 memtable 与 block，并进行去重。

降采样。 滚动 1 分钟窗口将原始数据点聚合为每序列每分钟的 min/max/avg/count/sum/p50/p95/p99。再次聚合为 1 小时粒度。聚合保留了足够的统计信号以满足大多数查询需求，同时体积更小。

查询优化。 将聚合下推至分片：分片不返回原始数据点，而是返回每序列聚合结果，从而减少网络传输量。对于大规模标签过滤结果，并行执行逐序列扫描。

查询路径追踪：查询引擎解析 PromQL -> 通过倒排索引展开标签过滤器 -> 对 posting list 取交集获取 series_ids -> 按分片分组 -> 并行分片 RPC 仅扫描匹配的时间 block -> 分片返回部分聚合 -> 根节点合并并降采样至图表分辨率。对于 1 年范围的查询，规划器使用 1 小时滚动聚合而非原始 block。

热门仪表盘路径，逐步说明：1. 客户端发送 PromQL 风格查询；规划器对 {tenant, query_text, time_range} 进行哈希并检查短时查询缓存。2. 规划器通过 tag:{tenant}:{key}:{value} posting list 展开标签过滤器，取交集，并按租户限制截断结果序列集。3. 按 TSDB 分片对序列 ID 分组（series_id 分片键），仅针对请求的时间 block 发送并行 RPC。4. 分片读取 memtable 及最近一小时内封闭的原始 block，在本地计算部分聚合，返回紧凑向量。5. 根节点合并部分结果并返回图表分辨率的数据点；大型表视图的游标状态携带 {tenant, query_hash, shard_checkpoint, time_block}.

冷长范围路径：对于超出原始数据保留窗口的查询，规划器跳过原始 block，读取 rollup/{tier}/{series_id}/{day}。若用户请求的序列数超过上限，API 返回含续传游标的部分结果，而非展开无界 posting list。

摄入路径追踪：Agent 发送压缩批次 -> 摄入服务验证租户配额并规范化标签 -> 在注册表中查找或创建 series_id -> 将批次以 series_id 为键追加至 Kafka -> 存储消费者写入 memtable 条目 -> 封闭的 block 更新倒排索引与对象存储清单。

告警。 每条规则有一个评估周期（例如每 60 秒）。Worker 使用相同引擎查询指标存储。每条规则维护一个状态机（OK / 告警中 / 已恢复），带迟滞以避免抖动。通知扇出通过独立队列异步进行。

高基数"临时"标签。 若用户坚持将 URL 放入标签，则将这些指标路由至一个独立的高成本存储（设有基数上限），而非主 TSDB。

多租户。 指标名称按租户命名空间隔离；每租户基数配额；按租户限制摄入速率。

跨区域。 每个 region 运行自己的 TSDB 集群，用于本地摄入和仪表盘。跨 region 查询（全局仪表盘）在查询层跨 region 联邦。

双视图框架。 同一指标流以原始逐序列 block 形式存储，用于新鲜数据排障；以降采样滚动聚合 block 形式存储，用于长范围趋势查询。倒排索引是指向这些 block 的指针视图，而非指标内容的另一份副本；保护该指针视图免受基数爆炸的侵害，与压缩采样点同等重要。

查询降级状态机：

阶段 模式 数据源 状态转换 延迟目标
热原始 最近几分钟/小时 memtable + 原始 block 正常仪表盘 典型 < 1 秒
暖滚动聚合 天/周 1 分钟滚动聚合 范围超出原始/精细配额 < 1-2 秒
冷滚动聚合 月/年 1 小时对象存储 block 长范围 秒级，可接受异步
部分 序列过多 截断 posting 展开 序列上限触达 返回部分结果 + 游标
降级 原始分片宕机 副本分片或滚动聚合 分片超时 降低分辨率，明确告警

边缘压力测试。 热门仪表盘应命中缓存的部分聚合，时效 10-30 秒。休眠序列保留墓碑，使回收的 series_ids 不会污染旧 block。将 user_id 作为标签添加的租户，新序列在倒排索引爆炸前即被拒绝。突发写入由 Kafka 吸收，若缓冲区超出阈值则按租户配额限流。跨 region 漂移以每 region 延迟元数据形式可见；全局仪表盘不应假设所有 region 数据同等新鲜。

回放/灾难恢复。 Kafka 保留覆盖数小时；极端情况下，主机还在本地磁盘缓冲区保留近期数据点以供回放。

优雅降级。

情形 模式 数据源 用户可见行为
摄入突增 优先丢弃超配额租户 租户速率限制 部分数据并明确丢弃
新序列爆炸 隔离标签键 / 拒绝序列 基数缓存 现有指标继续运行
热门仪表盘 缓存查询结果和部分聚合 查询缓存 + 滚动聚合 图表轻微过时
原始层不可用 提供降采样数据 暖滚动聚合 分辨率降低
告警评估器延迟 保持上一状态，标记延迟 告警状态 不误触发恢复

面试要点："指标系统往往因基数问题崩溃，而非因字节数；设计必须像保护存储层一样严格保护索引。"

成本调节手段。 大部分成本来自存储。激进的降采样、列式压缩和短暂的原始数据保留是最大的节省点。

我会推迟的内容。 日志与链路追踪（问题相似但独立）、样例追踪（将指标点链接至链路追踪）、每租户成本归因与计费、超越静态阈值的异常检测。
