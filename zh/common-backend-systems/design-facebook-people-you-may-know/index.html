1. 基础知识与解题思路

本题考察点

"你可能认识的人"（PYMK）基于图信号（共同好友、共同学校/公司、共同兴趣）来推荐好友连接。难点在于：通过昂贵的图操作为数十亿用户计算个性化候选好友、随图的变化及时刷新，以及在相关性与隐私之间取得平衡。

其不变量如下：

对于每个用户，系统从社交图谱和额外信号中生成一个个性化的候选好友排序列表，定期刷新，遵守隐私控制和屏蔽设置；推荐结果不泄露隐藏信息（例如，共同好友不应暴露私密好友关系）。

在 ICT5 / L6+ 级别，面试官在考察你是否能设计离线批处理图计算来实现共同好友计算、排序、隐私强制执行，并能推理扩展性和新鲜度之间的权衡。

60 秒面试框架

口述版本：

我会将 PYMK 设计为一个离线批处理流水线，基于共同好友候选人计算及共享属性信号，同时配备在线排序器实现个性化推荐。隐私过滤在投递时生效，确保不会泄露隐藏的好友关系。图存储按 user_id 分片，流水线通过分布式图计算扩展。核心决策点包括候选人计算、排序、隐私执行和数据新鲜度。

功能需求

功能性需求：

  • 为每个用户呈现个性化的候选好友列表。
  • 信号来源：共同好友、共同学校/雇主、共同兴趣、导入的联系人。
  • 过滤条件：已是好友、已屏蔽、已隐藏的推荐。
  • 随图变化实时刷新。

非功能性需求：

  • 每日刷新（高活跃用户可更频繁）。
  • 3B+ 用户。
  • 推荐在亚秒级时延内完成。
  • 隐私：不得泄露隐藏信息。

以下内容超出范围，除非被特别询问：

  • 电话本/电子邮件联系人导入的隐私细节。
  • 详细排序模型。

规模假设

领域 假设
用户 3B。
平均好友数 200-500.
每用户的共同好友数 典型值 10k-100k。
PYMK 候选人数量 每次请求 10-50 个。
刷新频率 每日；部分信号实时更新。

瓶颈：十亿用户规模下的共同好友计算；开销大但可并行化。

离线流水线

每日批处理：

  1. 加载社交图快照。
  2. 计算每用户的共同好友： 对于用户 U 的每个好友，取其好友列表，聚合计数（共同好友数量）。
  3. 聚合其他信号： 共同学校、雇主、位置、共同兴趣。
  4. 为每用户对候选人打分： 信号的加权组合。
  5. 每用户前 K 个 存储在候选集中。

图处理框架：Spark、Apache Giraph、Google Pregel、自定义 MapReduce。

对于拥有数百个好友的 3B 用户，共同好友对的总量极其庞大（数万亿级）。激进剪枝：每用户限制为前 K 个。

共同好友计算

伪代码：

for each user U:
  candidates = {}
  for friend F of U:
    for friend G of F (where G != U and G not already U's friend):
      candidates[G] += weight(F)
  top_K = sort_and_truncate(candidates)

按 F 的连接强度或最近互动进行加权。

来自 (U, F) 与 (F, G)的二部图：类似矩阵乘积操作；在图框架中高效执行。

其他信号

  • 共同学校： 同一学校且在同一年级的用户。
  • 共同雇主： 类似模式。
  • 共同位置： 在同一城市/地区。
  • 联系人导入： 双方均已相互导入联系人（需获得同意）。
  • 主页互动： 主页访问、群组共同成员身份。

每个信号以校准权重贡献到候选人分数中。

在线排序器

在请求时：

  1. 获取用户的候选集。
  2. 获取实时特征（最近的主页访问、当前页面上下文）。
  3. 用在线模型对候选人打分。
  4. 过滤：已是好友、已屏蔽、已隐藏。
  5. 应用隐私门控。
  6. 返回前 N 个。

延迟：亚秒级。

隐私

关键。若干规则：

  • 不得显示 A 是因为 B（一个私密好友）与其相连而被推荐。
  • 共同好友计算中排除隐藏的好友关系。
  • 已屏蔽用户排除在外。
  • "不希望被发现的用户"排除在外。
  • "为何推荐此人"必须遵守展示规则——根据可见性显示共同好友数量而非列表。

隐私过滤在服务时生效；候选集可能包含私密信息，但展示时会进行脱敏。

面试用语：

隐私门控在展示时生效。隐藏的好友关系会影响排序但绝不出现在解释中。候选集可能知晓 A 和 B 共享一个隐藏好友，但 UI 必须对此进行脱敏。

反馈循环

用户忽略推荐 → "隐藏"被存储；未来候选集排除此人。

用户接受 → 创建好友边。

强信号：忽略会降低相似候选人的分数；接受会提升分数。

数据 分区策略

一致性模型

层级 一致性

多区域策略

  • 用户归属区域的图数据。
  • 候选集复制到邻近区域。
  • 每个区域部署 PYMK 服务。

故障模式与缓解措施

故障模式 缓解措施 监控指标
候选集已过期。 每日刷新；对延迟发出告警。 候选人新鲜度。
隐私泄露。 严格过滤；审计。 隐私事件报告。
共同好友计算积压。 并行处理；扩容。 作业耗时。
已隐藏用户被展示。 过滤；遵守隐藏规则。 已隐藏但被展示的计数。
垃圾推荐。 限制接受频率；检测可疑模式。 垃圾举报。
冷启动（新用户）。 使用注册信号（学校、位置）。 冷启动率。
高连接度用户影响力过大。 限制单个好友的影响力。 限制有效性。
跨区域。 异步复制；有界延迟。 延迟。
屏蔽绕过。 屏蔽列表权威。 绕过次数。
GDPR 删除请求。 从边和候选集中级联删除。 合规审计。

运维指标

正确性：

  • 隐私泄露（零）。
  • 推荐过期率。
  • 屏蔽绕过次数。

容量：

  • 作业运行时长。
  • 候选集存储量。
  • 服务 QPS。

业务层面：

  • 接受率。
  • 忽略率。
  • 每 DAU 的好友新增数。

需要指出的权衡

  • 每日刷新是可接受的；实时刷新开销会很大。
  • 共同好友是主要信号；添加更多信号会增加计算量。
  • 激进的隐私限制会降低展示相关性。
  • 每用户更多候选人 = 更多存储但在线排序效果更好。
  • 联系人导入会引发隐私问题。
  • 高连接度节点可能主导候选集——需限制其影响力。

常见陷阱与更优做法

陷阱 更优做法
实时按请求计算共同好友。 批量预计算。
无隐私门控。 展示时严格过滤。
单一信号（仅 FoF）。 多信号加权。
无反馈循环。 隐藏/接受更新候选人评分。
在解释中展示隐藏好友。 若来源为私密则脱敏。
候选人永久不变。 隐藏 TTL 或根据用户选择永久排除。

面试官可能深入探讨的方向

深入方向 优秀回答要点
共同好友计算 分布式图处理。
隐私 隐藏好友关系、屏蔽、解释脱敏。
排序 多信号；ML 排序。
冷启动 注册信号；联系人导入。
反馈 隐藏/接受循环。
变化内容 并行图处理；候选集缓存。

Staff 级别回答检查清单

  • 离线批处理流水线 + 在线排序器。
  • 多信号加权。
  • 展示时严格隐私过滤。
  • 基于忽略/接受的反馈循环。
  • 限制高连接度节点的影响力。
  • 多区域部署，用户归属主区域。
  • 冷启动回退。

面试用语

PYMK 是离线图批处理 + 在线排序器。每日通过加权共享信号（学校、雇主、位置）的共同好友计算生成候选人；在线排序器实现个性化。展示时的隐私过滤确保隐藏好友绝不泄露到解释中。忽略和接受的反馈会循环回流入流水线。

2. 模拟面试

面试官： 设计"你可能认识的人"功能。

候选人： 离线批处理流水线每日计算每用户的共同好友候选人，并融合共享属性信号（学校、雇主、位置）。每用户的前 K 个存储起来。在线排序器在请求时结合实时特征进行评分。展示时的隐私过滤器会去除隐藏好友关系、屏蔽用户，并对解释进行脱敏。忽略/接受的反馈会循环回流。不变量是在尊重隐私和屏蔽规则的前提下提供个性化候选人。

面试官： 共同好友计算。

候选人： 对每个用户，遍历其好友；对每个好友的好友进行计数和加权。伪代码： candidates[G] += weight(F) 对 F 的每个好友 G 执行。每用户截断为前 K 个。使用 Spark 或 Giraph 在批处理中完成；类似二部图矩阵乘积操作。所有用户的 (U, G) 对总计数万亿级；激进剪枝。

面试官： FoF 之外的信号。

候选人： 同年级的共同学校、共同雇主、共同位置、联系人导入（需同意）、共同兴趣、主页访问。每个信号被评分并以校准权重合并。多信号融合使候选人比纯共同好友更具相关性。

面试官： 隐私。

候选人： 若干规则。隐藏的好友关系不用于 FoF（或使用但在解释中脱敏）。已屏蔽用户排除。选择退出可发现性的用户排除。"为何推荐此人"界面对私密好友脱敏——展示"X 个共同好友"但不显示其姓名（若其为隐藏用户）。候选集可能知晓私密连接，但展示时必须隐藏。展示时的硬编码过滤器；安全审查验证。

面试官： 如果一个用户有 5000 个好友怎么办？

候选人： 高连接度节点可能主导候选集——其每个好友都会被推荐给很多人。限制贡献：按每个好友的好友数的倒数加权其影响力，或简单地限制用于 FoF 的最大好友数。

面试官： 隐藏不想看到的推荐。

候选人： 用户点击"隐藏"。隐藏事件被记录；用户的隐藏列表更新。未来候选集排除该用户。隐藏可以是永久的或有 TTL 的。我们也可以将隐藏行为作为负信号来降低相似候选人的排名。

面试官： 冷启动。

候选人： 新用户还没有好友。注册信号：学校（来自表单）、位置、联系人导入（若已授权）。从这些信号中生成初始候选人。建立几个连接后，FoF 就会发挥作用。入职流程会积极推送可能认识的人。

面试官： 用户添加了一个好友，PYMK 应该刷新吗？

候选人： 新好友应影响未来的 PYMK。每日刷新会覆盖这一变化。对于高活跃用户，可以更频繁地刷新（例如每隔几小时）。实时增量更新是可行的，但开销较大——通常每日刷新已足够。

面试官： 联系人导入——隐私问题。

候选人： 敏感。需要用户明确同意。上传的联系人通过哈希标识符与平台用户进行匹配；只有匹配到的用户才作为候选人显示。未匹配的联系人在匹配会话结束后不予保留。双方理想情况下均应相互导入联系人，以获得最高置信度的推荐。近年来隐私审查对这一信号的限制已大幅收紧。

面试官： 在线排序器。

候选人： 获取候选集 + 实时特征（最近访问、当前上下文）。使用 ML 模型评分。过滤已是好友、已屏蔽、已隐藏的用户。对"为何推荐"文本进行隐私脱敏。返回前 N 个。亚秒级延迟。

面试官： 反馈循环。

候选人： 忽略 → 记录；未来候选集排除；相似候选人可能被降权。接受 → 建立好友边；新连接在下次 FoF 周期中反馈回来。其他参与行为（主页访问、对话）提供正向信号。

面试官： 多地区。

候选人： 用户归属区域存储图数据。候选集复制到邻近区域。每个区域部署 PYMK 服务。流水线全局运行一次或按区域运行——取决于图存储架构。

面试官： 总结。

候选人： 离线批处理图流水线每日计算每用户的共同好友候选人，并融合共享属性信号。每用户的候选集存储起来。在线排序器在请求时评分。展示时的隐私过滤器去除隐藏用户、已屏蔽用户，并对解释进行脱敏。反馈循环通过忽略/接受进行更新。最难的正确性问题是隐私（绝不泄露隐藏好友关系）；最难的规模问题是数万亿对的 FoF 计算。

3. 练习备注

五分钟版本

  1. 明确：信号、时效性、隐私。
  2. 不变量：个性化；脱敏处理私人信息。
  3. 离线批量 FoF + 在线排名器。
  4. 多信号加权评分。
  5. 展示时严格隐私过滤。
  6. 基于忽略/接受的反馈循环。
  7. 通过注册信号解决冷启动问题。
  8. 限制超级连接节点的影响力。

一分钟结尾陈述

PYMK 采用离线批量好友的好友 + 在线排名器。每日流水线基于 FoF 及共同属性信号为每位用户计算候选人。在线排名器进行个性化处理。展示时进行隐私过滤：隐藏的好友关系被排除，拉黑用户被排除，说明文字脱敏处理。拒绝/接受的反馈形成循环。冷启动使用注册信号。最棘手的正确性问题是隐私——绝不将隐藏信息泄露到说明文字中。

最佳面试用语

  • "基于加权信号的好友的好友；每日批量处理。"
  • "展示时设置隐私门控；说明文字对私密好友进行脱敏。"
  • "拒绝是负向信号；接受反馈进入下一个周期。"
  • "限制超级连接节点的影响力，以保持候选人的多样性。"
  • "冷启动使用注册信号——学校、位置、可选联系人。"
