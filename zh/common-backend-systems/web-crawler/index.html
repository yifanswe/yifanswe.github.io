Web Crawler — Googlebot Class (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Hard · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建一个礼貌、大规模的网络爬虫。给定一组种子链接，爬取可达页面，提取内容用于索引，并持续刷新已知页面。目标：每天爬取 10 亿个唯一 URL，遵守 robots.txt 及每主机访问频率限制，存储原始 HTML 和提取的文本。

1. 需求（约 5 分钟）

功能需求 - 从种子集合出发，发现并抓取可达 URL。- 提取链接、文本内容和元数据；存储供下游索引使用。- 按刷新策略重新爬取已知页面（热门页面频繁，长尾页面较少）。

我会暂缓处理 JavaScript 渲染深度、图片/视频爬取和 PageRank 式排名。

非功能需求 - 礼貌性——遵守 robots.txt；按主机节流；标识自身。- 吞吐量：每天 10 亿个 URL → 约每秒 12,000 次抓取。- 避免重复工作（URL 去重）和无限循环（环路、爬虫陷阱）。- 可恢复性——若 Worker 宕机，不会丢失 URL。- 最终一致性——无实时 SLA。

容量估算 - Frontier 索引中有 1000 亿个已知 URL。- 平均页面大小 50 KB → 每天抓取 50 TB 内容。- URL 去重索引：1000 亿 URL × 32 B 规范化键 = 约 3 TB。- Frontier 元数据：1000 亿 URL × 约 80 B（url_hash, host_hash, next_fetch_at, priority, status）≈ 8 TB 主存储，加上 3 副本约 24 TB。虽然数据量大，但作为分区键值/索引存储是可管理的。- 原始内容是主要成本：每天抓取 50 TB，压缩后约 15–25 TB/天；90 天热数据保留约为 1.5–2.2 PB（不含副本），因此分层存储比队列存储更为关键。

2. 核心实体（约 2 分钟）

  • URL — url，规范化形式、主机名、协议。
  • Frontier 条目 — url, next_fetch_at, priority.
  • FetchResult — url, status, body, headers, fetched_at, content_hash.
  • HostPolicy ——已解析的 robots.txt，以及每主机的爬取延迟。

3. API / 接口（约 5 分钟）

主要为内部接口，如有外部接口则包含在内：

POST /v1/seeds            { urls[] }
GET  /v1/pages/{hash}     → fetched page
GET  /v1/stats            → crawl progress

爬虫的"API"实际上是 Frontier 与 Worker 之间的契约。

4. 数据流（约 5 分钟）

种子 URL → Frontier → 抓取器 → 解析器（提取链接 + 内容）→ URL 过滤器（去重、策略检查）→ 将新 URL 写入 Frontier；将内容写入存储。循环。

5. 高层设计（约 10–15 分钟）

  1. URL Frontier。 按主机分区的 URL 抓取优先队列。每主机队列确保礼貌性：Worker 每次抓取延迟内最多拉取 1 个该主机的 URL。实现方式为 Kafka 按主机分区 + 每主机 Redis 有序集合用于延迟控制。
  2. DNS 解析器服务。 缓存解析结果；若没有此服务，DNS 将成为瓶颈。
  3. Fetcher Worker。 拉取 URL → 查询主机策略（robots.txt + 爬取延迟）→ HTTP GET → 存储结果。对于 JS 密集型页面，发送至无头浏览器池（成本较高，谨慎使用）。
  4. 解析器管道。 从响应体中：提取链接、规范化、提取正文文本（可读性提取）、计算内容哈希值、检测语言。
  5. URL 过滤器。 - 已访问 URL 检查： Bloom filter（每分片）用于 O(1) 去重已知 URL。持久化备份在键值存储中用于确认。 - 规范化： 去除片段标识符，规范化尾部斜杠，对查询参数排序（允许列表集合），解析相对路径。 - Robots / 策略： 跳过不允许路径、站外链接（取决于爬取范围）和已知陷阱。
  6. Frontier 写入器。 插入新 URL，附带优先级 + next_fetch_at。
  7. 内容存储。 原始 HTML 存入对象存储（压缩）；提取的文本进入下游索引管道（不在本范围内）。
  8. 调度器 / 协调器。 跟踪主机队列，均衡 Worker 负载，应用全局礼貌性限制。

端到端说明：种子 URL nytimes.com/ 入队 → 抓取器拉取（主机队列为空，现在可以抓取）→ 首先 GET robots.txt（已缓存），然后获取页面 → 解析器提取 200 个链接 → URL 过滤器对 Bloom filter 进行去重 → 50 个是新 URL → Frontier 写入器以优先级 = 页面权威度 × 新鲜度将其入队。与此同时，nytimes.com 的主机队列有 1 秒爬取延迟；来自该主机的下一个 URL 需等待 1 秒后才可被处理。

6. 深度探讨（约 10 分钟）

具体存储契约。

存储 键/数据结构 条目大小 + TTL 分片键
Frontier 索引 frontier:{host_hash}:{url_hash} → {url, next_fetch_at, priority} 约 80–150 B + URL，在已知期间保留 host_hash
就绪队列 Kafka 按 host_hash分区；Redis zset ready:{partition} score=next_fetch_at 指针约 32 B，直至被抓取 host_hash
主机策略 host_policy:{host} robots + 爬取延迟 1–10 KB，TTL 24 小时 host_hash
已访问过滤器 每 URL 分片的 Bloom filter + KV 证据 seen:{url_hash} 32 B 哈希证据，保留多年 url_hash
原始内容 s3://crawl/{date}/{url_hash} 压缩 HTML 平均 10–50 KB，分层 30/90/365 天 date,url_hash
解析元数据 page:{url_hash} → 状态、content_hash、外链数量 约 200 B，保留多年 url_hash

各存储容量估算。 Frontier 元数据大小为 100B URLs × ~80 B × 3 或约 24 TB；存储完整规范化 URL 可能使主存储占用达到 15–25 TB（不含副本，取决于平均 URL 长度）。精确的已访问证据为 100B × 32 B × 3 或约 9 TB，而每条目 10 位的 Bloom filter 主存储仅约 125 GB。原始内容占主导： 1B pages/day × 50 KB 每天抓取 50 TB，压缩后约 15–25 TB/天，90 天 3 副本达数 PB。解析元数据适中： 100B × 200 B × 3 约 60 TB。主机策略数据量可能较大，但受活跃主机数量和 TTL 限制，而非 URL 数量。

抓取路径跟踪。 调度器从 ready:{partition} 读取符合条件的 URL → Worker 以可见性超时租约获取一个 URL → 读取 host_policy:{host} 。 host:next_ok_at → 若允许，原子性地推进 next_ok_at → 执行 HTTP 抓取 → 将原始 HTML 写入对象存储，将解析元数据写入 KV → 解析器发出外链 → URL 过滤器并行规范化并检查 Bloom filter/KV → 新 URL 被写入 Frontier 及其主机分区。若 Worker 在确认前宕机，租约到期后 URL 重新可见。

礼貌性——最重要的规则。 每主机爬取延迟（默认 1 秒，或来自 robots.txt）。实现方式：Redis 键 host:nytimes.com:next_ok_at。Worker 在抓取前检查此键；分发时将其设置为 now + delay 。若存在竞争（多个 Worker 争抢同一主机），失败方重新入队。

面试要点："爬虫通过在局部放慢速度来实现全局的高速；礼貌性是每主机的不变量，而非整个集群的平均值。"

Robots.txt。 每个主机每天获取一次，解析并缓存。Worker 在每次抓取前查询。遵守 Crawl-delay, User-agent, Disallow, Sitemap 指令处理跨子图的拼接。

URL 规范化。 指向同一页面的两个 URL 应哈希到相同的键。规则如下：- 小写协议 + 主机名。- 解析路径中的 .. 。 . 。- 去除默认端口。- 对查询参数排序；从允许列表中删除类似 session-id 的参数。- 去除片段标识符。 提及 <link rel="canonical"> 作为我们也会遵守的服务端提示。

大规模 URL 去重。 1000 亿个 URL 对于内存哈希集合来说太多了。可伸缩的 Bloom filter（每分区）可提供 O(1) 检查，并将假阳性率控制在约 0.1%。当 Bloom filter 报告"可能已访问"时，持久化键值存储用于硬检查。

为何选择 Bloom filter + KV 而非仅使用数据库唯一性检查：Bloom filter 在大多数重复项到达存储层之前就将其拒绝，而 KV 提供证据和可修复性。纯精确索引更简单，但在每秒 12,000 次抓取加上外链扩展的情况下，随机 I/O 成本过高；纯 Bloom filter 成本低，但因假阳性会静默丢失 URL。

Frontier 优先级排序。 并非所有 URL 平等。优先级 = f(page_rank_estimate, freshness_need, newness)。从热门枢纽新发现的 URL 获得高优先级。重爬频率：一级（如 nytimes）每小时，二级每天，长尾每周及以上。

爬虫陷阱。 无限日历、持续递增查询参数生成的 URL。防御措施：- 限制每主机的爬取深度（不超过 1000 层）。- 限制每主机每天的页面数量。- 检测内容相似性——若来自某主机的许多 URL 的 content_hash 重复出现，则将某模式加入黑名单。

分布式协调。 按主机哈希 URL → 分配到 N 个分区之一。每个分区拥有独立的 Frontier + Worker，因此每主机礼貌性集中在一处。跨分区：只有 URL 发现会向其他分区发送（来自主机 A 的已解析页面可能包含指向主机 B 的外链）。

Kafka 适合持久化交接和重放，但本身并不是优先队列。Redis zset 或 Frontier DB 存储 next_fetch_at；Kafka 传递就绪任务和解析器输出。对于"为何不直接用 Kafka？"这个问题要有所准备——因为延迟优先级调度和每主机礼貌性需要按时间索引，而不仅仅是追加顺序。

双索引框架。 URL 需要对同一规范化 URL 集合建立两种索引：按主机键控的队列用于礼貌性（frontier:{host_hash}:{url_hash}），按 URL 哈希的证据用于去重（seen:{url_hash}）。按主机键控的状态回答"我何时可以爬取该主机？"按 URL 键控的状态回答"我是否已经发现过这个页面？"明确这种二元性，可以避免设计出只优化吞吐量却违反 robots/礼貌性的方案。

JavaScript 渲染。 现代 Web 富含 JS。选择性渲染：仅在静态页面无法产生有用内容时才进行渲染。使用无头 Chrome Worker，成本较高（每页约 2 秒，而静态 HTML 仅需约 50 毫秒）。

重爬调度。 每个页面都有 next_recrawl_at。按枢纽型与叶子型、页面变更频率和重要性分级。重要页面（入链数多或新闻网站）每天重爬；长尾页面每月重爬。

故障处理。 抓取器在 5xx 时进行退避重试；遵守 Retry-After。对于持续性 4xx，将 URL 标记为失效。超时时降低优先级。

我会推迟的内容。 图片/视频爬取、结构化数据（JSON-LD）提取、反指纹规避（我们主动标识自身）、专项垂直爬虫（新闻、购物）、分布式 PageRank。
