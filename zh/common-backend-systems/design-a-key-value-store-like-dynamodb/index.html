1. 基础知识与解题思路

本题考察点

像 DynamoDB 这样的全球级 KV 存储是许多生产系统的基础。难点在于：在提供可预测的个位数毫秒级延迟和跨区域 99.999% 可用性的同时，允许运营者按请求选择所需的一致性模式。

其不变量如下：

对于任意键，一次成功的写入在至少法定数量的副本上是持久的，最终对所有一致性读取者可见，并且系统在单节点、单可用区甚至部分区域故障期间持续接受写入，冲突解决规则能够确定性地收敛。

在 ICT5 / L6+ 级别，面试官考查的是：你能否在无主节点的基于法定数量的复制（Dynamo 论文风格）和单主节点分区复制（现代 DynamoDB 风格）之间做出选择，解释一致性哈希和分区拆分，处理向量时钟或最后写入者胜出，以及设计热点分区缓解方案。

60 秒面试框架

口述版本：

我将设计一个托管 KV 存储，分为三个平面：请求路由器、拥有键空间范围的分区层，以及每个分区的复制存储层。关键决策包括：分区策略（一致性哈希 vs 带自动拆分的范围分区）、复制（无主节点法定数量 vs 基于主节点）、可调一致性（强一致 vs 最终一致），以及热点分区处理。我将先明确需求，选定分区 + 复制方案，讲解写入和读取路径，再深入探讨法定数量语义和热点键。

首先明确需求

功能性需求：

  • 通过主键进行 Put、Get、Delete（单键操作）。
  • 可选排序键，用于分区内的范围扫描。
  • 条件写入（CAS）：仅当 version = X 或条目不存在时才写入。
  • 批量 get/put、分页扫描。
  • 二级索引（全局和局部）。
  • 面向下游消费者的变更流。
  • 自动删除的每表 TTL。

非功能性需求：

  • p99 读写延迟个位数毫秒。
  • 每区域可用性 99.99% 至 99.999%。
  • 每请求可调一致性：强一致或最终一致。
  • 每表 PB 级规模；数千万 QPS。
  • 多区域，支持主动-主动模式。
  • 每租户隔离与配额。

以下内容超出范围，除非被特别询问：

  • 完整 SQL 查询规划器。
  • 跨分区的多键 ACID 事务（作为独立功能提及）。
  • 备份与 PITR 机制。

规模假设

领域 假设
表 跨租户数十万张。
每表条目数 数十亿至数万亿。
条目大小 典型为 KB 级；最大 400 KB。
吞吐量 每秒每表最多数千万读写单元。
Latency（延迟） 区域内 p50 < 5 ms，p99 < 10 ms。
副本 每分区默认 3 个副本；跨可用区分布于同一区域内。
多区域 可选主动-主动模式，采用最后写入者胜出或 CRDT。
热点分区 单分区限制约 3000 RCU / 1000 WCU；需要检测并拆分。

瓶颈在于热点分区和副本故障下的尾部延迟。

核心产品模型

界面 操作 需求
单键写入 按主键进行 Put/Update/Delete。 每键原子操作；可调持久性。
单键读取 按主键进行 Get。 可调一致性；极低延迟。
范围查询 共享分区键的条目，排序键范围扫描。 有界结果；通过游标分页。
条件写入 比较并交换（Compare-and-swap）。 在分区内原子地检查并设置。
变更流 每分区变更日志。 按分区有序；至少一次投递。

系统有意限制为每键（或范围操作的每分区）操作。跨分区事务是例外，而非常规操作。

核心组件

组件 职责 关键细节
请求路由器 解析操作，按键找到归属分区，转发至该分区的主节点。 无状态；缓存分区映射；变更时重新均衡。
元数据服务 将键范围 → 分区 → 副本集进行映射。 强一致性；通过共识协议进行复制。
分区（存储节点） 拥有连续的键范围；在本地磁盘上运行 LSM/B-tree。 每台主机数百个分区；可插拔存储引擎。
复制组 每个分区一个主节点 + 多个副本；类 Raft 日志复制。 若选择 Dynamo 论文风格，则为无主节点法定数量。
自动扩缩容器 拆分热点分区；合并冷分区；跨主机重新均衡。 在线操作；无需停机。
GSI 维护器 异步写入二级索引条目。 最终一致性。
变更流 每分区变更日志，暴露给消费者。 按分区有序；分片以支持并行消费。
跨区域复制器 在区域间复制写入，用于全局表。 默认最后写入者胜出；计数器使用 CRDT。

两种架构选择

选项 A：无主节点法定数量（原始 Dynamo 论文）

  • 每键 N 个副本，无指定主节点。
  • 写入到 N 个副本中的 W 个，从 N 个副本中的 R 个读取。 R + W > N 以实现强一致性。
  • 冲突通过向量时钟和读取时的兄弟节点协调来解决。
  • Hinted handoff：若某副本宕机，临时代理节点存储该写入。
  • 通过 Merkle 树进行反熵，定期协调分歧副本。

优点：即使副本故障仍可写入；无主节点瓶颈。缺点：兄弟节点解析复杂；客户端必须合并或系统必须应用最后写入者胜出。

选项 B：基于主节点的每分区复制（现代 DynamoDB）

  • 每个分区有一个主节点；通过共识协议（Multi-Paxos / Raft 风格）进行复制。
  • 写入仅在多数节点完成日志复制后确认。
  • 从主节点读取为强一致性；从副本读取为最终一致性。
  • 故障转移为主节点重新选举；耗时数秒。

优点：客户端模型更简单；天然支持强一致性；无兄弟节点问题。缺点：主节点是每分区的瓶颈；故障转移期间短暂写入不可用。

现代 DynamoDB 更接近基于 Multi-Paxos 的每分区主节点复制。面试回答应选择其中一种并说明理由；若被追问，对比两者。

面试用语：

我默认选择基于主节点的分区复制，因为运营体验更简单——无需合并兄弟节点、按需强一致性，且每分区主节点提供天然的分片。无主节点法定数量在副本丢失时可用性更高，但将复杂性推给了客户端。

分区策略

DynamoDB 使用 哈希分区键 作为主要分区策略：

  • 对分区键应用哈希函数，对键空间取模。
  • 每个分区拥有哈希空间中的一段连续范围。
  • 带有 (partition_key, sort_key) 的条目被路由到范围包含该值的分区。 hash(partition_key).
  • 在分区内，条目按 sort_key 排序（B-tree 或 LSM 组织）。

自动拆分： 当分区的存储或吞吐量超过阈值时，从中位键处拆分。拆分是在线进行的：拆分表选取中间点，复制在两个新分区上继续，一旦新分区追上进度，元数据服务切换路由。

热点键： 单个分区键（某一用户、某一租户）可能使单个分区饱和。缓解措施：自适应容量——临时允许该分区通过借用空闲兄弟分区的配额来提升吞吐量；写入分片——客户端在分区键后追加后缀（user_42#0, user_42#1，……）以跨分区分散，随后查询所有后缀；突发容量——短期超量供给。

面试用语：

自动拆分处理数据量不均的情况，但同一热点分区键仍会将所有请求路由到一个分区。解决方案必须针对访问模式——写入分片或在存储层之上进行缓存。

副本

每分区复制（基于主节点）：

  1. 客户端写入到达路由器 → 分区主节点。
  2. 主节点追加到本地 WAL。
  3. 主节点将日志条目发送给从节点；从节点持久化到 WAL。
  4. 一旦多数节点（如 3 中 2 个）确认，主节点应用到本地状态并响应。
  5. 从节点异步应用并确认。

共识细节：

  • 类 Raft 风格，包含 term 和日志索引，或采用稳定主节点的 Multi-Paxos。
  • 故障时在数秒内完成主节点选举；元数据服务可参与，也可由共识组内部处理。

无主节点（Dynamo 论文）：

  1. 客户端写入协调者；协调者并行转发给 N 个副本。
  2. 等待 W 个确认；响应。
  3. 读取时，协调者查询 R 个副本；解决冲突。
  4. 向量时钟追踪因果历史；冲突时将所有兄弟节点返回给客户端或通过最后写入者胜出合并。

一致性模型

读取模式 行为
最终一致 从任意副本读取；可能落后数毫秒至数秒。
强一致性读取（基于主节点） 从主节点读取；反映所有已提交的写入。
线性一致性读取（无主节点） R + W > N 的法定数量读取。
写入模式 行为
标准 put 多数副本持久化后确认。
条件 put 同上，另加在分区主节点上原子评估条件。
事务性写入 涉及的分区间两阶段提交。

写入路径

  1. 客户端 → 携带分区键的请求路由器。
  2. 路由器通过缓存的映射解析分区；转发给主节点。
  3. 主节点对当前状态检查条件表达式（如有）。
  4. 追加到 WAL；复制到从节点。
  5. 等待多数确认。
  6. 应用到本地 LSM/B-tree。
  7. 发送到每分区流；异步更新 GSI。
  8. 响应。

故障场景：

  • 条件检查失败：返回 ConditionalCheckFailed；不执行任何操作。
  • 副本不可达：若仍能达到 quorum，主节点使用剩余副本继续写入；否则返回可重试错误。
  • 主节点在写入中途宕机：客户端重试；新主节点的日志包含到上一条已提交条目为止的所有内容。

读取路径

  1. 客户端 → 路由器 → 主节点（强一致性）或任意副本（最终一致性）。
  2. 副本从本地存储返回数据项。
  3. 对于最终一致性读，无需进一步协调。
  4. 对于强一致性读，主节点确认其已应用所有已提交的条目（按照主节点的定义，这一点始终成立）。

对于范围查询，主节点扫描排序键范围，应用过滤条件，并返回不超过结果上限的数据及游标。

二级索引

本地二级索引（LSI）： 在同一分区内使用备用排序键，与基础表同步更新，与基础表保持一致。

全局二级索引（GSI）： 使用完全不同的分区键，由消费分区变更日志的 GSI 写入器异步维护，最终一致。

GSI 写入失败或延迟不会影响基础表写入，应用以最终一致性查询 GSI 来容忍这一点。

流（Change Data Capture）

每个分区的仅追加操作日志：

  • 与复制日志使用相同的序列号。
  • 以对应分区的分片形式暴露，消费者通过游标迭代。
  • 保留时间通常为 24 小时。
  • 同一分区内保证顺序；跨分区不保证。

适用于触发器、物化视图、搜索索引以及向其他系统的复制。

多区域策略

两种主要模式：

全局表（active-active）： - 每个区域在本地接受写入。 - 跨区域复制是异步的。 - 冲突解决：按时间戳 + 区域决胜的 last-writer-wins，或对计数器和集合使用 CRDT 语义。 - 跨区域读取最终一致。 - 所要求的 SLA：秒级收敛。

单区域加跨区域只读副本： - 一个区域接受写入。 - 其他区域持有只读副本。 - 更简单；故障转移需要显式晋升。

对于大多数工作负载，单区域已足够。Active-active 适用于低延迟全球写入（代价是冲突模型的复杂性）。

故障模式与缓解措施

故障模式 缓解措施 监控指标
副本宕机 以剩余 quorum 继续；恢复后重建。 副本健康状况、quorum 规模。
主节点宕机 几秒内重新选举；写入短暂暂停。 主节点选举频率、写入不可用时间。
分区过热。 自适应容量；拆分分区；对客户端限速。 每分区吞吐量、被节流的请求数。
存储卷已满。 自动拆分 + 再均衡。 每主机存储利用率。
跨区域复制延迟。 对写入方施加背压；对延迟发出告警。 复制延迟 p99。
并发写入冲突（active-active）。 带区域决胜的 LWW；计数器使用 CRDT。 每分区的冲突数量。
GSI 延迟。 异步回填；有界陈旧性告警。 每索引的 GSI 延迟。
网络分区隔离某个区域。 在本地继续运行；恢复后收敛。 分区检测时间、重新收敛时间。
单个慢副本导致尾部延迟。 向第二个副本发起带超时的对冲读取。 读取 p99 与 p50 的差距。
客户端重试放大负载。 服务端准入控制；指数退避提示。 重试后速率、被节流的 QPS。

运维指标

正确性：

  • 强一致性请求后出现陈旧读取（必须为零）。
  • 每区域的冲突解决事件数。
  • GSI 同步延迟。

容量：

  • 每分区的读写 QPS 和吞吐量消耗。
  • 存储增长。
  • 区域内及跨区域的复制延迟。
  • 每表的被节流请求率。

业务层面：

  • 每表的 p50/p99 延迟。
  • 每次操作的成本。
  • 热点键检测事件。

需要指出的权衡

  • 基于主节点的复制在故障转移时可轻松实现强一致性，但会短暂失去可用性；无主节点方案提供可用性，但将冲突解决推给了客户端。
  • 哈希分区实现均匀分布，但破坏了跨分区键的范围扫描——这是有意为之的设计。
  • 自动拆分能重新均衡数据，但无法修复热点分区键的访问模式。
  • 强一致性读需要额外的一次往返（或跳转到主节点），最终一致性读开销更低，通常已足够。
  • Active-active 多区域降低延迟，但约束了数据模型（LWW、CRDT）。
  • LSM 提供高写入吞吐量，代价是 compaction 开销；B-tree 读取可预测，但写入较慢。

常见陷阱与更优做法

陷阱 更优做法
不假思索地对所有内容按 user id 哈希。 识别热点用户，在存储层之上应用写入分片。
承诺随处提供强一致性读。 按请求区分最终一致性与强一致性；对强一致性收取费用。
忽略二级索引的一致性。 说明 GSI 是最终一致的，LSI 是强一致的。
在所有地方使用多键事务。 将事务限制在少量分区上；设计数据模型使相关数据共置。
将全局表视为"免费的魔法"。 解释 LWW / CRDT 及其后果。
忽略尾部延迟。 对冲读取、P99 告警、副本健康检查。

面试官可能深入探讨的方向

深入方向 优秀回答要点
分区 + 自动拆分 哈希分区键；每分区内按范围；在中间键处在线拆分。
主节点 vs 无主节点 权衡取舍；冲突解决；运维复杂度。
热点分区 自适应容量；写入分片；上游缓存。
GSI 设计 通过流异步维护；最终一致性；有界陈旧性。
多区域 active-active LWW vs CRDT；收敛；分区修复。
条件写入 / 事务 每分区原子性；跨分区使用 2PC。

Staff 级别回答检查清单

  • 选择分区 + 复制方案并论证。
  • 阐明每键原子性不变量和复制 quorum。
  • 介绍一致性模式下的写入和读取路径。
  • 明确处理热点分区。
  • 将 GSI 阐述为异步派生状态。
  • 讨论多区域 active-active 的冲突语义。
  • 将流作为变更捕获的接口。
  • 运维指标与尾部延迟缓解措施。

面试用语

星球级 KV 存储本质上是每分区一个已分区、已复制的日志，上层是一个键范围路由层。核心决策不在于数据结构，而在于分区（带自动拆分的哈希）、复制（主节点 vs 无主节点）以及多区域冲突语义。热点分区和尾部延迟是耗费最多工程精力的运维难题。

2. 模拟面试

面试官： 设计一个类似 DynamoDB 的键值存储。

候选人： 我将设计一个三层系统：请求路由器、拥有键范围的分区层，以及每分区的复制存储层。核心不变量是：成功的写入在大多数副本上持久化，读取可以在强一致性和最终一致性之间按请求调整。

面试官： 如何对数据进行分区？

候选人： 我对分区键进行哈希，并为每个分区分配哈希空间的连续范围。每个分区跨可用区三副本复制。当分区的存储或吞吐量超过阈值时，在中间键处自动拆分；拆分在线进行，方式是启动两个新的分区主节点，通过日志追上数据，然后切换元数据路由。

面试官： 主节点复制还是无主节点复制？

候选人： 每分区使用主节点复制。每个分区通过 Raft 风格的共识选出一个主节点，加上两个从节点。写入经由主节点，复制到多数 quorum（3 中 2），然后确认。强一致性读来自主节点；最终一致性读来自任意副本。相比 Dynamo 论文原始的无主节点 quorum 方案，我更倾向于这种方式，原因是客户端语义更简单——无需将兄弟节点解析推给应用程序。

面试官： 单个分区键变热——某位用户每秒产生 100,000 个请求，如何处理？

候选人： 三层防御。首先，自适应容量：该分区向空闲的兄弟分区借用吞吐量预算。其次，热点分区拆分无济于事，因为所有负载都集中在一个键上；自动拆分按存储拆分，而非按每键流量拆分。第三，应用程序必须对键进行分片：在写入时为分区键追加一个小的随机后缀（user_42#0 through user_42#9），并跨所有后缀扇出查询。这以少量读放大换取了写入的分散。

面试官： 如果主节点宕机怎么办？

候选人： Raft 选举启动；拥有最新日志的从节点在几秒内成为新主节点。受影响分区的写入短暂暂停。从节点在整个过程中持续提供最终一致性读服务。元数据服务获知新主节点后更新路由缓存；持有旧路由的客户端会收到"非主节点"错误，并重新向元数据服务重试。

面试官： 条件写入如何工作？

候选人： 单键 CAS 很直接——主节点读取当前值，评估条件，若通过则将检查包含在复制日志条目中，从而原子性地写入新值。对于跨分区的多项事务，使用两阶段提交：协调者准备每个分区主节点（锁定数据项并写入 prepare 条目），所有 prepare 完成后再提交。速度较慢，仅在必要时使用。

面试官： 二级索引如何工作？

候选人： 两种类型。本地二级索引在同一分区内，与基础数据项同步更新，强一致。全局二级索引使用不同的分区键，因此更新必须跨分区。GSI 维护器消费基础分区的变更流，计算 GSI 条目，并写入目标分区。GSI 是最终一致的；延迟通常低于一秒，但重建索引时会出现峰值。

面试官： 流如何工作？

候选人： 每个分区都有一个带有单调序列号的仅追加变更日志。流消费者以游标按分区分片迭代。我们保留 24 小时。同一分区内保证顺序，跨分区不保证。消费者包括 GSI 维护器、跨区域复制器和应用触发器。

面试官： 全局表如何实现跨区域的 active-active？

候选人： 每个区域接受对同一张表的写入。跨区域复制通过流异步进行。冲突使用按时间戳的 last-writer-wins，以区域 id 作为决胜项。对于计数器属性，CRDT 计数器更为正确，但大多数产品接受 LWW。收敛目标是几秒钟。代价是两个区域同时写入可能丢失其中一方的数据；用户必须针对此进行设计（例如，对"归属"数据使用每区域分区键）。

面试官： 尾部延迟如何处理？

候选人： 多种技术结合。对冲读取：同时向主节点和一个副本发送请求，取先到的响应，慢的那个被取消或忽略。推测性重试。容量余量：目标利用率保持在 60-70%，以防慢节点引发级联。每副本健康检查将慢副本从读取路径中降级。Compaction 调度避免在高峰期产生 I/O 峰值。

面试官： 存储引擎的选择？

候选人： 底层使用 LSM 树。写入先追加到 memtable，然后刷写到磁盘上的有序 run；在后台合并。可预测的写入吞吐量，良好的压缩率，但由于 merge pass，读取尾部延迟略差于 B-tree。每个 SSTable 的 Bloom filter 加速了负面查找。

面试官： 跨分区的多键事务——发生频率如何？

候选人： 按设计，极少发生。数据模型鼓励单分区操作，因为这是 ACID 开销最低的地方。跨分区事务使用 2PC，并以更高成本收费。应用架构师被鼓励重新设计，而非诉诸事务。

面试官： 总结。

候选人： 哈希分区键加自动拆分；每分区三副本 Raft 风格主节点；每请求可调一致性；通过变更流实现异步 GSI；通过 LWW 或 CRDT 实现 active-active 多区域；热点分区通过自适应容量和客户端分片处理；对冲读取和容量余量缓解尾部延迟。该架构在操作层面（单键）有意保持简单，以便系统能在星球级规模下提供可预测的性能。

3. 练习备注

五分钟版本

  1. 明确：单键操作；可调一致性；分区内范围扫描；GSI。
  2. 不变量：写入持久到 quorum；可调读取。
  3. 架构：路由器 + 分区层 + 复制存储。
  4. 哈希分区加自动拆分。
  5. 每分区基于主节点；Raft 风格共识。
  6. 写入路径：WAL → quorum → apply → 响应。
  7. 热点键处理：自适应容量 + 客户端写入分片。
  8. 多区域：LWW 或 CRDT；GSI 异步；流用于 CDC。

一分钟结尾陈述

星球级 KV 存储是每分区一个已分区、已复制的日志，上层是路由层。我选择带在线自动拆分的哈希分区、每分区三副本 Raft 风格主节点，以及可调读取一致性。热点分区通过自适应容量和客户端分片处理；多区域使用带 last-writer-wins 的异步复制；二级索引通过每分区变更流实现最终一致性。最棘手的生产问题是热点键和尾部延迟，分别通过自适应容量和对冲读取加以解决。

最佳面试用语

  • "我按哈希分区，按存储自动拆分；热点键仍需客户端分片。"
  • "主节点复制比无主节点 quorum 更简单；运维人员永远不会看到兄弟节点冲突。"
  • "GSI 是最终一致的，因为它跨分区；用户在设计时需考虑这一点。"
  • "多区域 active-active 使用带区域决胜的 LWW，或对计数器使用 CRDT。"
  • "对冲读取和容量余量是我控制 p99 不飙升的方式。"
