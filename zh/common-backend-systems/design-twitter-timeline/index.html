1. 基础知识与解题思路

本题考察点

Twitter home timeline 是 staff 级深度的混合扇出经典问题。难点在于：在不为明星推文支付推送成本、也不为普通用户支付读取成本的前提下，向数亿用户提供个性化的低延迟 feed。

其不变量如下：

每位关注者的 home timeline 最终包含其所关注作者发布的所有推文，按时效性或排序分数排列，并受可见性过滤器约束；读取的 p99 返回时间在亚秒级；写入扇出的收件箱数量不超过阈值。

在 ICT5 / L6+ 级别，面试官考察的是：你能否清晰阐述混合推/拉（扇出）决策、设计收件箱缓存、在读取时合并明星作者的 timeline、应用可见性过滤器，并对热门用户和副本进行推理。

60 秒面试框架

口述版本：

我会将 Twitter timeline 设计为一个按用户的收件箱缓存，保存来自被关注作者的近期推文 id，同时为粉丝数超过阈值的明星作者设置读取时拉取路径。读取时将收件箱与明星作者的 timeline 合并，填充推文内容，应用可见性过滤器，可选择性地进行排序。有趣的决策点在于：明星阈值、收件箱大小上限、填充成本以及可见性强制执行。

功能需求

功能性需求：

  • 用户打开主页，看到来自所关注作者的近期推文。
  • 排序：按时间倒序（或排序打分）。
  • 分页；加载更多。
  • 过滤被屏蔽/静音的作者。
  • 删除快速传播。

非功能性需求：

  • 读取 p99 < 200ms。
  • 有界陈旧性（秒级）。
  • 即使写入路径降级，服务仍可用。
  • 5 亿以上月活用户。

以下内容超出范围，除非被特别询问：

  • 推文创建（参见 设计 Twitter).
  • 深度解析推荐系统 ML 设计。
  • 搜索。

规模假设

领域 假设
用户 月活用户 5 亿。
平均关注数 100 条；长尾可达数千条。
名人关注 拥有 >1M 粉丝的用户超过 1 万。
读操作 每天 1000 亿次以上。
收件箱大小 上限约 800 条。

瓶颈：内容填充成本与名人合并。

核心产品模型

层级 职责 需求
收件箱缓存 普通作者的每用户推文 id。 读取快；大小有界。
名人合并 读取时拉取名人推文。 按作者缓存。
内容填充 推文 ID → 内容。 批量处理。
可观测性 屏蔽/静音/删除过滤器。 在读取时处理。
排序 可选重排序。 带超时的独立服务。

核心组件

组件 职责 关键细节
收件箱缓存 每用户最近推文 id 列表。 Redis 有序集合；设有上限。
收件箱写入器 发推时，针对非名人作者，将推文追加到关注者收件箱。 参见 设计 Twitter.
名人作者缓存 每位名人最近的推文 id。 在读取时拉取。
推文内容填充 推文 id → 完整内容。 从推文存储批量获取。

排序器 可选重排序。 严格超时。
推文存储 权威推文内容。 按 tweet_id 分片。
关注图 在填充时用于判断谁是名人。 读密集型。

数据模型

收件箱条目

InboxEntry(
  user_id,
  tweet_id,
  author_id,
  inserted_at
)

按 user_id 存储于 Redis 有序集合中，按 tweet_id（snowflake；时间有序）排序，上限为最近 N 条（例如 800 条）。

作者时间线缓存

AuthorTimeline(
  author_id,
  recent_tweets: [(tweet_id, posted_at)],
  cache_until
)

适用于所有作者，但对名人更为关键。

API

GET /home?cursor=&limit=
GET /users/{id}/timeline             # author timeline

读取路径

  1. 客户端请求 /home.
  2. 时间线服务为用户获取收件箱缓存（最近 N 个推文 id）。
  3. 查找关注图：哪些被关注的作者是名人？
  4. 针对每位名人，获取其最近的作者时间线缓存。
  5. 合并：按 tweet_id（嵌入时间戳）排序——最新优先。
  6. 取 K 条结果作为一页。
  7. 填充：对推文存储批量 GET 以获取推文内容。
  8. 可见性过滤：丢弃来自被屏蔽/静音用户及已删除的推文。
  9. 可选排序器处理（带超时）。
  10. 返回结果。

延迟预算：- 收件箱获取：5-10ms（Redis）。- 名人合并：10-20ms（少量缓存）。- 填充：30-50ms（从推文存储批量获取）。- 可见性：<10ms（已缓存）。- 排序：<100ms（带超时）。- 总计：<200ms。

收件箱上限与淘汰

收件箱保存最近 N 个推文 id，上限大小根据典型"加载更多"分页需求而定，不会溢出。

当用户关注新作者时，其收件箱不会回填历史内容——用户只会从此刻起看到新推文。（可选：通过回填任务填充新关注者的近期推文。）

缓存淘汰：LRU；冷收件箱被淘汰。缓存未命中时，从关注图 + 作者时间线重建（慢路径）。

名人合并

对每位被关注的名人：- 获取其最近的作者时间线缓存（小且已缓存）。- 合并到读取结果中。

若用户关注了 200 位普通作者 + 10 位名人，读取操作为 1 次收件箱读取 + 10 次作者时间线读取——可控。

对于关注了数百位名人的用户，合并成本线性增长。权衡方案：限制每次请求考虑的名人集合数量，或将顶级名人视为可写入收件箱的对象。

内容填充

推文 id → 内容。从推文存储批量获取。填充内容包括：- 推文文本与元数据。- 媒体 URL。- 作者信息（反规范化）。- 互动计数。

填充是延迟最大的贡献者。在时间线服务处缓存已填充的推文。

可见性过滤器

在读取时应用，绝不预先写入收件箱。

  • 屏蔽：用户 X 的屏蔽列表；过滤其推文。
  • 静音：类似处理。
  • 删除：推文存储中标记为已删除的推文——填充时跳过。
  • 隐私：受保护账号仅对已批准的关注者可见。
  • 安全/政策：已下架的推文。

可见性过滤器是最终关卡。即使收件箱中包含某推文 id，读取服务也可能跳过它。

排序

可选地，按 ML 评分对候选项重新排序。

  • 独立服务。
  • 严格超时（例如 100ms）。
  • 超时时，回退到逆时间顺序排列。
  • 特征：观看者-作者亲密度、推文互动量、时效性。

扩展与分区

数据 分区策略
收件箱缓存 user_id 哈希；Redis 中按用户分片
作者时间线缓存 author_id 哈希
推文存储 tweet_id 哈希（snowflake）
关注图 出向按 user_id 哈希；入向按 author_id 哈希

一致性模型

层级 一致性
推文写入 强一致性
收件箱缓存 最终一致（写入后延迟数秒）
作者时间线缓存 最终一致
可观测性 数据填充时强一致性
排序 最终一致的特征

多区域策略

  • 按用户归属地区部署。
  • 每个区域一份收件箱缓存。
  • 作者时间线缓存复制到邻近区域。
  • 从本地区域读取。
  • 推文存储与关注图的跨区域复制。

故障模式与缓解措施

故障模式 缓解措施 监控指标
收件箱缓存未命中。 从关注图 + 作者时间线重建（慢路径）。 缓存未命中率。
推文存储响应慢。 填充超时；使用骨架数据降级服务。 填充 p99 延迟。
屏蔽绕过。 读取时可见性过滤。 拉黑逃逸计数。
陈旧的删除操作。 填充时看到 deleted 状态；跳过。 陈旧删除率。
热门作者名人流量激增。 按作者缓存可扩展；CDN 化。 每作者 QPS。
排序器速度慢。 超时 + 时间顺序降级回退。 排名器超时率。
收件箱缓存淘汰风暴。 预热；持久化基线。 淘汰率。
跨区域复制。 异步；有界。 复制延迟。
关注了大量名人。 限制合并集合；将顶级名人视为可写入收件箱的对象。 合并集合大小。
收件箱已满，遗漏最近推文。 按需回填。 分页边界情况。

运维指标

正确性：

  • 可见性违规。
  • 陈旧删除展示。
  • 收件箱与真实数据的差异（抽样）。

容量：

  • 读取 p50/p99 延迟。
  • 收件箱缓存 QPS 与命中率。
  • 填充 QPS。

业务层面：

  • 主页时间线 DAU。
  • 每次曝光的互动量。

需要指出的权衡

  • 更大的收件箱上限支持更深的滚动，但占用更多内存。
  • 更激进的名人阈值降低扇出成本，但增加每次读取的名人合并次数。
  • 预物化收件箱速度快，但存储开销大。
  • 读取时进行可见性判断是准确的，但增加了查询成本。
  • 排序能提升互动量，但增加延迟和复杂性。

常见陷阱与更优做法

陷阱 更优做法
在收件箱中存储完整推文内容。 存储 id；读取时填充。
纯写入时扇出。 混合式名人拉取。
在写入时应用屏蔽。 在读取时应用。
仅逆时间顺序。 可选排序并带降级回退。
每用户单一收件箱分片。 分片；每用户独立 Redis。
无名人缓存。 按作者的时间线缓存。

面试官可能深入探讨的方向

深入方向 优秀回答要点
混合扇出 普通作者推送，名人作者拉取。
内容填充 批量处理；已缓存。
可观测性 读取时门控。
收件箱上限与淘汰 LRU + 重建。
排序 带超时的独立服务。
多区域 每用户主页；缓存副本。

Staff 级别回答检查清单

  • 混合扇出：收件箱缓存 + 名人拉取。
  • Snowflake ID 用于时间排序。
  • 读取时批量获取进行填充。
  • 读取时的可见性过滤。
  • 带超时的可选排序器。
  • 多地区按用户划分主域。

面试用语

Twitter 时间线采用混合扇出：普通作者将推文 id 推入关注者收件箱；名人推文在读取时拉取。读取操作合并收件箱与名人作者时间线，进行填充，应用可见性过滤，并可选地进行排序。推文存储是权威数据源；收件箱只保存 id。最难满足的约束是在密集关注图下实现 200ms 以内的读取。

2. 模拟面试

面试官： 设计 Twitter 首页时间线。

候选人： 混合扇出：每用户收件箱缓存保存普通作者的推文 id；名人作者在读取时拉取。读取操作合并收件箱与名人作者时间线，按时间（或评分）排序，填充推文内容，应用可见性过滤。不变量是在最终一致新鲜度下实现 200ms 以内的读取。推文写入流向扇出 Worker（普通作者）或保持仅拉取模式（名人）。

面试官： 为什么要用混合方案？

候选人： 纯推送方式对名人不可行——一个拥有 1 亿粉丝的账号发一条推文，意味着 1 亿次收件箱写入，耗时数分钟。纯拉取方式对读取路径不可行——每次读取都需要聚合数百位被关注作者的内容。混合方案按阈值划分：粉丝少的作者推送；粉丝多的作者拉取。读取操作为收件箱 + 少量名人合并。

面试官： 阈值设在哪里？

候选人： 调整以最小化总成本：扇出写入成本 vs 读取放大节省量。对于粉丝数为 F、每天发 P 条推文的作者，推送成本为 F × P。该作者的读取成本为关注者每天 R 次读取 × 每次读取的拉取次数。找到使推送成本 = 拉取成本的 F 值。通常为 1 万至 100 万粉丝。随着作者粉丝数变化动态调整。

面试官： 读取路径。

候选人： 从 Redis 有序集合（最近 N 条）获取收件箱。识别哪些被关注作者是名人（来自关注图）。针对每位名人，获取其缓存的近期作者时间线。将收件箱与名人推文按 tweet_id（snowflake = 时间有序）合并。取一页。通过批量推文存储获取填充推文内容。应用可见性过滤——丢弃被屏蔽/静音/已删除的内容。可选排序器处理。返回。

面试官： 延迟预算是多少？

候选人： 收件箱获取约 5ms。名人合并约 10-20ms。填充约 30-50ms（最大开销）。可见性约 10ms。排序 100ms（带超时）。p99 总计低于 200ms。填充是主要瓶颈；我们缓存已填充的推文以降低成本。

面试官： 收件箱上限。

候选人： 约 800 条，典型值。覆盖典型的"加载更多"分页需求。关注新作者时，收件箱不会历史回填——用户从此刻起看到新推文。深度滚动超出上限需要预缓存扩展或直接从作者处拉取。

面试官： 删除——传播方式。

候选人： 推文存储将推文标记 deleted。填充时看到该状态，返回空内容（或墓碑记录）。收件箱条目保留——推文 id 指向已删除的记录。可见性过滤器在读取时处理删除。已填充内容的缓存通过 pub/sub 在删除时失效。

面试官： 屏蔽——收件箱是否需要重写？

候选人： 不需要。屏蔽在读取时由可见性过滤器处理。收件箱中仍保留被屏蔽用户的推文 id，但读取服务在返回前会将其丢弃。每次屏蔽变更时，可节省数十亿次收件箱写入。

面试官： 用户关注了 1000 位名人。

候选人： 读取需合并 1000 个名人作者时间线——数量过多。缓解方案：每次请求按互动历史限制名人集合为前 K 位。其余名人视为可写入收件箱的对象（对该用户移出名人桶）。或限定为近期活跃的名人。属于边界情况；大多数用户关注的名人远少于此数量。

面试官： 排序？

候选人： 独立服务。候选生成后，将候选推文 id + 观看者上下文发送给排序器。排序器评分并返回有序 id。严格超时（100ms）；超时则回退到时间顺序。预计算特征：观看者-作者亲密度、推文互动量、时效性。

面试官： 多地区。

候选人： 每用户主区域。每区域一份收件箱缓存。推文存储全局复制。名人作者时间线复制到邻近区域。从本地区域读取。扇出 Worker 位于用户主区域。跨区域复制为异步，延迟有界。

面试官： 总结。

候选人： 混合扇出（普通作者推送，名人作者拉取）。Redis 中每用户收件箱缓存保存推文 id，大小为最近 N 条。读取操作合并收件箱 + 名人作者时间线，通过批量推文存储获取进行填充，读取时应用可见性过滤，并可选地带超时排序。多区域按用户主页划分。最难满足的性能约束是 200ms 以内，填充是主要成本；对已填充推文进行激进缓存可缓解此问题。

3. 练习备注

五分钟版本

  1. 混合扇出：普通用户推送，名人用户拉取。
  2. 每用户收件箱缓存（Redis 有序集合）。
  3. 名人作者时间线已缓存。
  4. 读取操作合并收件箱 + 名人时间线。
  5. 从推文存储批量填充。
  6. 读取时可见性过滤。
  7. 带超时的可选排序器。
  8. 多地区，按用户划分主域。

一分钟结尾陈述

混合扇出将推文传播分为两类：粉丝少的作者推送至关注者收件箱；名人在读取时拉取。收件箱是每用户 Redis 有序集合，存储推文 id。读取操作合并收件箱 + 名人作者时间线，填充，应用可见性过滤，可选排序。推文存储是权威数据源；收件箱是投影数据。最难满足的性能约束是 200ms 以内的读取；填充是主要瓶颈。

最佳面试用语

  • "普通作者推送，名人拉取；以总成本为阈值划分。"
  • "收件箱存储推文 id，而非内容；读取时进行批量填充。"
  • "可见性——屏蔽、静音、删除——在读取时应用，绝不预先写入收件箱。"
  • "排序器是带严格超时和时间顺序降级回退的独立服务。"
  • "按用户主区域划分；每区域独立缓存，配合全局推文存储。"
