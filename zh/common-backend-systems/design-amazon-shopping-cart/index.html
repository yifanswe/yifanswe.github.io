1. 基础知识与解题思路

本题考察点

Amazon 的购物车是一个看似简单实则复杂的功能：它是原始 Dynamo 论文中用于论证动机的典型示例。难点在于保持购物车的高可用性——即便在网络分区期间，每次"加入购物车"操作也必须成功——同时需要协调来自多个设备的并发冲突更新。

其不变量如下：

每次"加入购物车"操作最终都会出现在用户所有设备的购物车中；删除的商品不会重新出现，除非重新添加；来自多个设备的并发更新合并时不会丢失任何商品；即使部分副本不可达，购物车依然可用。

在 ICT5 / L6+ 级别，面试官考察的是你能否清晰表述 Dynamo 风格的"始终可写"权衡，通过向量时钟或 CRDT 设计冲突解决方案，处理多设备购物车合并，以及分析购物车放弃、库存检查和转换为订单等场景。

60 秒面试框架

口述版本：

我将围绕一个以 user_id 为键、始终可写的 Dynamo 风格 KV 存储来设计购物车。添加和删除操作通过向量时钟或 CRDT 语义进行合并，使得来自多个设备的并发更新永远不会静默丢失商品。关键决策在于冲突解决语义（偏向保留）、库存检查时机，以及结账时购物车转为订单的转换流程。

首先明确需求

功能性需求：

  • 将商品加入购物车并指定数量。
  • 更新数量。
  • 移除商品。
  • 在任意设备查看购物车。
  • 持久化购物车（在会话结束后保留）。
  • 应用优惠券/促销。
  • 保存以备后用（独立列表）。
  • 结账：将购物车转换为订单。

非功能性需求：

  • 高可用性：即使在部分故障期间，购物车写入也必须成功。
  • 跨设备最终一致。
  • 亚秒级响应。
  • 扩展至数亿用户。
  • 冲突解决：永不静默丢失商品。

以下内容超出范围，除非被特别询问：

  • 在加入购物车时执行库存检查（仅在结账时锁定库存）。
  • 税费计算（独立服务）。
  • 推荐商品插入（独立服务）。

冲突解决：Dynamo 的故事

场景：用户在手机上添加书 A，在笔记本上添加书 B，同时操作（或某设备离线）。

向量时钟方案：

每次写入携带一个向量时钟。购物车存储接受两次写入；读取时，若存在多个版本（兄弟节点），客户端将其合并。

对于购物车，自然的合并策略是 并集：取两个版本中的所有商品。若两个版本修改了同一商品的数量，取最大值（保留更多，避免丢失）。

结果：购物车包含 A 和 B 两本书。用户不会"丢失"任何添加操作。

对于删除操作，这更复杂。若 A 在购物车中，然后在设备 1 上被删除，而设备 2（持有过期状态）向 A 添加了数量，朴素的并集会让 A 复活。

修复方法： 使用墓碑。删除时记录带版本号的墓碑；后续使用旧版本的添加操作会被合并，但墓碑优先。

CRDT 方案：

将购物车建模为 CRDT（如 OR-Set 或 2P-Set）：

  • OR-Set：每次添加有唯一标签；删除只移除其已知的标签。
  • 结果：未看到删除操作的添加仍会出现在合并集中。

面试用语：

Dynamo 的设计以可用性优先于一致性。购物车始终可写；并发更新产生兄弟节点，由客户端合并。购物车的自然合并策略是带墓碑的集合并集——优先保留而非删除。

写入路径

  1. 用户在某设备上添加商品。
  2. Cart API 以写入法定数量 W 个副本写入购物车存储。
  3. 一旦 W 个副本确认，即返回成功。
  4. 副本异步传播到其他节点。
  5. 在另一设备上读取可能看到最终一致的状态；若存在多个版本，进行合并。

为实现高可用性： W + R > N 用于强一致性，或 W < N 。 R < N 用于最终一致性下的高可用性。

读取路径

  1. 用户在某设备上打开购物车。
  2. Cart API 从 R 个副本读取。
  3. 若版本出现分歧，客户端（或合并服务）通过向量时钟进行协调。
  4. 返回合并后的购物车。

可选地，将合并后的版本写回以修复分歧（读修复）。

多设备同步

当用户在另一设备上打开购物车时：

  1. 从购物车存储读取。
  2. 展示合并后的状态。
  3. 用户的操作（添加、删除、更新）写回。

推送更新：当写入提交时，通知已订阅的设备刷新。可选；打开时拉取即已足够。

结账时的库存处理

Amazon 历史上在加入购物车时不锁定库存。原因如下：

  • 购物车放弃率高；锁定库存会阻碍其他用户购买。
  • 库存会变成抢购式的预购争夺。
  • 昂贵的协调操作被推迟到结账时进行。

因此，加入购物车时不检查库存。结账时验证并锁定库存。若届时缺货，用户会看到"已无法购买"的提示。

对于极度稀缺的商品（如限时抢购），这会改变——参见限时抢购设计。但普通购物车：懒惰式库存检查。

结算

  1. 用户发起结账。
  2. 本次结账会话中购物车被冻结。
  3. 逐商品进行库存检查和锁定。
  4. 计算价格（含当前促销）。
  5. 创建订单（状态机）。
  6. 完成支付。
  7. 购物车标记为已转化；商品移除。

若任何商品缺货，向用户展示错误并允许调整。购物车保持活跃。

稍后购买

与购物车类似但为独立列表。相同的 KV 模式；相同的冲突解决方式。

购物车与稍后购买之间的移动是原子操作。

数据 分区键 原因

一致性模型

层级 一致性 原因
购物车写入 高可用性，最终一致性 永不丢失添加操作。
购物车读取 法定数量 + 兄弟节点解决 融合趋势。

多区域策略

  • 用户主区域用于购物车写入。
  • 跨区域异步复制。
  • 对于旅行用户，跨区域读取具有最终一致性；若存在分歧则合并。
  • 故障转移区域保持热备。

故障模式与缓解措施

故障模式 缓解措施 监控指标
多设备并发添加。 合并为并集；取最大数量。 Sibling 速率。
删除后再添加的竞争。 墓碑；OR-Set 语义。 复活缺陷数量。
购物车存储分区。 Sloppy quorum + hinted handoff。 分区事件。
副本宕机。 继续处理剩余部分。 副本健康状态。
结账竞争（两个设备）。 最后写入者优先或显式冲突处理。 并发结账次数。
结账时库存不匹配。 优雅地呈现给用户。 缺货率。
促销在购物车中途过期。 结账时重新计算。 促销过期竞争。
用户放弃购物车数月。 保留策略；软过期。 购物车年龄。
购物车合并产生不需要的商品。 墓碑；合并 UI 清晰度。 用户投诉数量。
机器人批量添加商品。 频率限制；CAPTCHA。 每用户添加频率。

运维指标

正确性：

  • 丢失添加率（必须约为零）。
  • 复活率（已删除商品重新出现）。
  • 读取时的兄弟节点率。

容量：

  • 每秒购物车写入次数。
  • 读取 QPS。
  • 每秒合并操作数。

业务层面：

  • 购物车到结账的转化率。
  • 购物车放弃率。
  • 平均购物车大小。

需要指出的权衡

  • 高可用性牺牲了严格一致性；这对购物车来说是可接受的权衡。
  • 集合并集合并保留商品，但可能复活删除的商品；墓碑以元数据为代价解决了这个问题。
  • 结账时不锁定库存节省了协调开销，但可能在结账时令用户失望。
  • 墓碑垃圾回收需要决定何时可以安全遗忘。
  • 每用户状态简化了分区。
  • CRDT 设计比向量时钟更复杂，但具有可组合性。

常见陷阱与更优做法

陷阱 更优做法
购物车写入的强一致性。 始终可写，最终合并。
最后写入者优先。 向量时钟兄弟节点 + 并集合并。
在加入购物车时锁定库存。 结账时懒惰检查。
无墓碑（删除操作会被过期添加覆盖）。 带版本号的墓碑。
单副本。 N 个副本，配置 W/R 法定数量。
严格排序。 并发操作可行；合并解决冲突。

深入方向 优秀回答要点
冲突解决 向量时钟 + 并集；墓碑；OR-Set CRDT。
多设备同步 打开时拉取 + 推送通知。
库存时机 结账时懒惰检查。
购物车到订单的流程 状态机；幂等。

Staff 级别回答检查清单

  • Dynamo 风格始终可写的购物车存储。
  • 向量时钟或 CRDT 冲突解决。
  • 带墓碑的并集合并用于删除处理。
  • 加入购物车时不检查库存；结账时检查。
  • 购物车到订单的流程作为状态机。
  • 多设备拉取 + 推送。
  • 多区域部署，用户归属主区域。

面试用语

Amazon 的购物车是 Dynamo 经典用例。可用性至上——每次"加入购物车"操作必须成功，即使在分区期间也不例外。来自多个设备的并发更新产生兄弟节点，客户端将其合并为带墓碑的并集。有趣的权衡在于：优先保留商品（永不静默丢失添加操作）而非严格一致性。库存是懒惰式的：在结账时检查，而非加入时检查。

2. 模拟面试

面试官： 设计 Amazon 的购物车。

候选人： 以 user_id 为键的 Dynamo 风格始终可写 KV 存储。即使在部分故障期间，购物车写入也能成功；读取通过向量时钟解决兄弟节点。购物车商品的自然合并策略是集合并集，每件商品取最大数量；删除使用墓碑防止复活。加入购物车时不锁定库存；库存在结账时检查并锁定。不变量是：即使在网络分区和并发设备更新下，每次添加操作都能被保留。

面试官： 带我走一遍多设备竞争场景。

候选人： 用户手机和笔记本电脑同时在线。手机添加书 A；笔记本同时添加书 B，两者均基于之前的购物车状态。购物车存储用各自的向量时钟记录两次写入。两次写入均不知道对方的存在。当用户在任一设备上打开购物车时，读取看到两个兄弟节点（并发版本）。合并：对商品集合取并集，每件商品取最大数量。结果：购物车包含 A 和 B 两本书。用户不会丢失任何添加操作。

面试官： 如果用户在手机上删除 A，然后笔记本以过期状态向 A 添加数量怎么办？

候选人： 使用墓碑。删除操作以版本 V 写入一个墓碑。笔记本的添加操作有版本 V'，但不知道墓碑的存在。合并时，按照 CRDT 语义（或追踪删除的合并规则），墓碑优先。A 保持删除状态。用户刷新笔记本后购物车中不包含 A。另一种方案——朴素并集——会让 A 复活，即"静默删除"缺陷。

面试官： 为什么加入购物车时不锁定库存？

候选人： 购物车放弃是常态——大多数购物车从不结账。在加入购物车时锁定库存会使真实买家无法购买；在高流量下，锁定的库存会超过实际买家数量。跨数亿购物车的协调成本极高。Amazon 的选择：结账时懒惰检查库存。若届时商品缺货，优雅地告知用户。例外是限时抢购，稀缺性是核心，系统在加入时锁定库存。

面试官： 结账流程。

候选人： 状态机。用户发起 → 本次结账冻结购物车（乐观锁）→ 逐商品进行库存检查和锁定 → 用当前促销计算价格 → 使用幂等键创建订单 → 完成支付 → 购物车标记为已转化；商品移除。若任何库存检查失败，向用户展示错误并允许调整；购物车保持活跃。

面试官： 如果用户从两个设备同时结账怎么办？

候选人： 幂等性。结账以一个 checkout_session_id开始。两次同时发起产生两个会话；先提交的那个创建订单。第二个会话因"购物车已结账"而失败。第二个设备上的 UI 刷新。或者，使用购物车状态上的单个 CAS 来 checkout_in_progress.

面试官： 用户快速添加 100 件商品。

候选人： 对添加操作实施每用户频率限制。设置合理阈值。若模式看起来是自动化的，进行机器人检测。购物车对正常重度用户（如批量购物者）仍然有效，但极端频率时会被标记审查。

面试官： 多区域？

候选人： 用户主区域用于购物车写入。跨区域异步复制。旅行场景：用户从另一区域访问购物车——副本可能有轻微延迟；读取返回可能过期的状态；写入时，合并解决冲突。故障转移区域保持热备。

面试官： 购物车放弃处理？

候选人： 购物车默认无限期保留（或按策略——许多电商网站保留 90 天）。营销系统可能会就放弃的购物车发送提醒邮件。软过期（N 天后标记为已放弃）用于指标；硬过期较为罕见。

面试官： 购物车中的促销？

候选人： 促销码记录在购物车商品或购物车层面。结账时验证并应用（资格、有效期、使用次数）。购物车总价在结账时计算；购物车展示显示估算总价。加入购物车和结账之间促销过期：告知用户；允许调整。

面试官： 稍后购买？

候选人： 独立列表，相同模式。在购物车和稍后购买之间移动是在一个中添加墓碑 + 在另一个中添加。相同的冲突解决方式适用。

面试官： 总结。

候选人： Dynamo 风格始终可写的购物车存储，以 user_id 为键，复制 N 份。向量时钟解决并发写入；合并是带墓碑的集合并集用于删除，碰撞时取最大数量。加入购物车时不检查库存；结账时检查。购物车到订单是具有幂等性的状态机。多区域，配置用户主区域。最难的正确性问题是保留每次添加操作同时不让删除操作复活；墓碑解决了这个问题。

3. 练习备注

五分钟版本

  1. 明确：规模、多设备、库存时机、冲突处理。
  2. 不变量：永不丢失添加操作；删除保持删除状态。
  3. Dynamo 风格始终可写存储。
  4. 向量时钟或 CRDT 用于冲突解决。
  5. 带墓碑的并集合并。
  6. 加入时不检查库存；结账时懒惰检查。
  7. 带幂等性的购物车到订单状态机。
  8. 多区域部署，用户归属主区域。

一分钟结尾陈述

Amazon 的购物车是 Dynamo 论文的经典用例。可用性至上；来自多个设备的并发更新通过向量时钟以集合并集语义和墓碑进行合并。加入购物车时不锁定库存；结账时懒惰检查。最难的正确性问题是保留每次添加操作而不复活删除操作——墓碑解决了这个问题。多区域，配置用户主区域。
