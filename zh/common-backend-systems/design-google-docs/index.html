1. 基础知识与解题思路

本题考察点

Google Docs 是实时协同编辑：多位用户同时在同一文档中输入，实时看到彼此的光标，且不会丢失任何修改。难点在于并发控制算法——操作转换（OT）或无冲突复制数据类型（CRDT）——以及持久化、离线编辑和存在感知（presence）。

其不变量如下：

每位编辑者的本地文档视图，在看到相同的操作集后，最终与其他所有人收敛到相同的规范状态；没有任何操作被静默丢失；操作顺序因果一致；离线编辑在重新连接后能正确合并；文档内容具有持久的版本历史。

在 ICT5 / L6+ 级别，面试官考察的是：你能否在 OT 和 CRDT 之间做出选择、设计同步协议、处理存在感知和光标、管理权限，以及分析单文档热点键问题（所有编辑者都固定在同一文档上）。

60 秒面试框架

口述版本：

我会围绕单文档操作日志进行设计，通过单一权威服务对操作进行序列化。编辑者发送操作；服务器使用 OT 将操作与并发操作进行转换（或通过 CRDT 合并），广播给所有对端，并持久化。存在感知和光标随之传递。关键决策在于 OT vs CRDT、单文档扩展性、离线处理以及权限执行。

首先明确需求

功能性需求：

  • 多位用户同时编辑同一文档。
  • 查看他人的光标和选区。
  • 评论与建议。
  • 编辑历史与版本。
  • 离线编辑在重新连接时同步。
  • 权限：查看、评论、编辑、所有者。
  • 富文本格式。
  • 链接共享或基于电子邮件的邀请。

非功能性需求：

  • 收敛性：所有编辑者看到相同的最终状态。
  • 延迟：自己的键击立即可见；他人的键击在 100-300 ms 内可见。
  • 每个文档的高可用性。
  • 能够承受编辑者断线。
  • 可审计性：谁做了什么修改。
  • 多区域，跨区域延迟合理。

以下内容超出范围，除非被特别询问：

  • 电子表格公式（类似问题但更专业）。
  • 幻灯片（专项）。
  • 嵌入式媒体编辑。
  • 翻译、无障碍工具。

规模假设

领域 假设
文档 数十亿。
活跃编辑会话 数千万并发。
每文档编辑者数 通常 1-3 人；热门文档最多可达数千人。
操作数 每次键击即为一个操作；每位用户每秒可有 10 次以上的突发操作。
Latency（延迟） 自身键击 <50 ms；对端键击 <300 ms。
持久性 操作日志按文档持久化。
留存期 多年版本历史。

瓶颈在于单文档热点键：一个文档的所有编辑者必须通过一个逻辑序列器串行化。

操作转换（OT）

经典方案。

概念：每个操作被重写（"转换"），以在并发操作存在的情况下保持一致性。

示例：文档内容为 HELLO。用户 A 在位置 3 插入"X"。用户 B 同时在位置 1 插入"Y"。

  • 不做转换：直接应用 A 的操作得到 HELXLO；直接应用 B 的操作得到 HYELLO。两者都应用后：结果取决于顺序。
  • 使用 OT：当 A 的操作在 B 的操作（ Insert "Y" at 1）之后到达服务器时，将 A 的位置从 3 转换为 4（被 Y 偏移）。无论顺序如何，结果均为 HYELXLO 。

实现方式：

  • 服务器作为中央序列器。
  • 每个操作携带其生成时对应的序列号。
  • 服务器将收到的操作与该序列号之后的所有操作进行转换。
  • 将转换后的操作及其新序列号广播给所有对端。

客户端侧：

  • 立即应用本地操作（乐观）。
  • 发送到服务器。
  • 当服务器以转换后的版本确认后，如有需要则进行协调。

面试用语：

OT 需要中央序列器，因为转换函数依赖于了解中间所有操作的信息。服务器是排序的真实来源。

CRDT（替代方案）

无冲突复制数据类型：操作携带足够的元数据（时间戳、唯一 ID），无需协调即可进行可交换合并。

对于文本，常见的 CRDT 有 RGA、Yjs、Logoot、Treedoc。

相比 OT 的优势：

  • 无需中央序列器；对端可直接合并。
  • 支持离线；重新连接时合并。

缺点：

  • 每个操作携带更多元数据（可能开销较大）。
  • 实现难度更高。
  • 某些 CRDT 存在"墓碑膨胀"或非平凡的垃圾回收问题。

对于 Google Docs 这样的系统，带中央服务器的 OT 是历史选择。现代系统常采用 CRDT。

同步协议

编辑者通过 WebSocket 连接到文档会话服务器。

  1. 服务器返回最新快照 + 快照之后的操作。
  2. 客户端物化本地状态。
  3. 客户端进行本地编辑；生成带有当前序列号标签的操作。
  4. 客户端在本地应用（乐观 UI）。
  5. 客户端将操作发送到服务器。
  6. 服务器对并发操作进行转换；分配新序列号；持久化到操作日志；广播给所有对端。
  7. 其他对端接收操作；如有需要则转换其本地状态。
  8. 发起操作的客户端收到确认；如本地序列号不同则进行协调。

对于离线编辑： 在本地缓冲操作，附带单调递增的 client_op_id。 重新连接后，将缓冲的操作发送到服务器。 服务器依次将每个操作与当前状态进行转换。

快照与历史

在文档打开时重放数百万次操作开销巨大。定期生成快照：

  • 每隔 N 次操作或 M 分钟，写入一个快照，包含物化后的文档及其对应的序列号。
  • 打开时，加载快照 + 快照之后的操作。
  • 旧操作可压缩，但通常为历史记录保留。

版本：

  • 操作日志即为历史记录。
  • 可在快照处标记"版本点"。
  • 命名版本是用户创建的书签。

存在感知与光标

短暂存在；通过与操作相同的 WebSocket 广播。

  • 客户端每隔几百毫秒或在选区变化时发送光标位置。
  • 服务器转发给所有对端，不做持久化。
  • 每位对端显示为一个带颜色的光标 + 姓名。

光标位置随文本移动，通过 OT/CRDT 转换更新。

评论

通过字符偏移量 + 位置 ID 锚定到一段文本（因此在编辑后仍有效）。

  • 与主文本分开存储。
  • 线程式回复。
  • 解决/取消解决状态。
  • @提及时发送通知。

扩展与分区

数据 分区键 原因
文档 doc_id 哈希值 一个文档，一个会话主节点。
操作日志 doc_id 哈希值 按文档追加。
快照 doc_id 哈希值 按文档物化。
权限 doc_id 按文档检查。
存在感知 在 session 服务器内存中 临时性。

热门文档（例如，数千人同时编辑的全员直播文档）：

  • 单一会话主节点成为瓶颈。
  • 缓解措施：
  • 只读查看者不经过主节点；通过扇出缓存流式接收操作。
  • 会话服务器的垂直扩展。
  • 对每位编辑者的操作频率进行限流。
  • 对于超大型协作会话，回退到"只读，仅一位编辑者"模式。

面试用语：

对于一个文档而言，最难解决的扩展性问题不是长尾文档，而是同时有数千人在编辑的那个病毒式传播的文档。会话主节点是中央序列器，无法按文档进行水平拆分。

一致性模型

层级 一致性 原因
每文档操作顺序 强一致性（服务器序列化） 收敛性要求。
跨对端的文档状态 最终一致性（亚秒级内） OT/CRDT 保证收敛性。
权限 强一致性 安全性。
存在感知 最终一致 临时性。
评论 写入时强一致 避免重复。
多区域复制 有界延迟 区域间异步。

故障模式与缓解措施

故障模式 缓解措施 监控指标
会话服务器宕机。 故障转移到副本；从日志重新广播操作。 会话重连率。
编辑者离线编辑。 缓冲操作；重连后通过 OT 转换重放。 缓冲操作数分布。
并发冲突编辑。 OT 转换 / CRDT 合并保证收敛性。 收敛性检查失败次数。
热门文档的操作日志分片过热。 预分片；通过扇出层限制查看者数量。 每文档操作速率。
会话中途权限变更。 重新检查操作；若权限被撤销则踢出编辑者。 撤销传播延迟。
评论风暴。 按用户限制速率。 每文档的评论 QPS。
快照服务落后。 提高快照频率；设置告警。 快照延迟。
跨区域编辑者访问远端文档。 将请求路由至文档的归属区域；接受延迟。 跨区域延迟。
网络抖动导致操作重复。 基于 client_op_id 实现幂等性。 重复操作率。
编辑者本地状态发生分歧。 定期进行状态哈希校验；不匹配时强制重新同步。 分歧检测计数。

运维指标

正确性：

  • 客户端与服务端状态分歧。
  • 收敛延迟分布。
  • 转换失败次数。

容量：

  • 每台服务器的活跃编辑会话数。
  • 每文档的操作 QPS。
  • WebSocket 连接数。
  • 快照创建速率。

业务层面：

  • 并发编辑者分布。
  • 编辑时长。
  • 每文档评论数。

多区域策略

  • 每个文档有一个归属区域（通常在用户创建时所在的区域）。
  • 会话主节点运行在归属区域。
  • 远端编辑者通过 WebSocket 连接到归属区域。
  • 快照在全球范围内复制，用于只读访问。
  • 故障转移区域保持热备。

需要指出的权衡

  • OT 需要中心化的排序器；CRDT 更具分布式特性，但实现更复杂。
  • 中心化的会话主节点是每个文档的瓶颈；对其扩容很困难。
  • 频繁快照可降低重放成本，但会消耗存储空间。
  • 存储所有按键操作数据量巨大，但可实现时光机式历史记录。
  • 本地乐观应用可提供即时反馈，但在转换时需要协调一致。
  • 每个操作都进行强权限校验会增加延迟；缓存校验则存在数据陈旧的风险。

常见陷阱与更优做法

陷阱 更优做法
最后写入者优先。 使用 OT 或 CRDT 实现收敛。
每次打开文档时都全量扫描操作日志。 使用快照 + 之后的增量操作。
每次按键都直接写入数据库。 追加写操作日志；定期生成快照。
基于锁的编辑。 无锁并基于转换的编辑。
通过轮询同步。 使用 WebSocket 广播。
多区域活跃会话主节点。 每文档单区域；跨区域只读副本。

面试官可能深入探讨的方向

深入方向 优秀回答要点
OT vs CRDT 权衡分析；中心化排序器 vs 分布式合并。
离线编辑 在本地缓冲操作；重新连接时进行转换。
每文档热点键 会话主节点扩容；观看者分流层。
快照 定期物化；有界重放。
权限 Drive 集成；每操作权限校验。
多区域 每文档归属区域；复制快照。

Staff 级别回答检查清单

  • OT 或 CRDT 的选择与论证。
  • 每文档会话主节点对操作进行串行化。
  • 追加写操作日志；定期生成快照。
  • 本地乐观应用并进行协调。
  • 离线编辑缓冲 + 重连时转换。
  • 在线状态与评论与文本操作相互独立。
  • 每操作均进行权限校验。
  • 多区域部署，每文档有归属区域。

面试用语

Google Docs 本质上是一个通过会话主节点串行化的每文档操作日志。操作变换（OT）或 CRDT 使所有编辑者在并发编辑时保持收敛。快照使加载开销降低；离线编辑通过缓冲并在重连时协调。在线状态与评论独立于主操作流水线之外。

2. 模拟面试

面试官： 设计 Google Docs。

候选人： 我将围绕一个由会话主节点串行化的每文档操作日志进行设计。编辑者通过 WebSocket 连接；主节点使用 OT 对操作进行排序，广播给所有对等节点并持久化。快照定期物化当前状态，以避免重放完整历史。在线状态与评论相互独立。不变式是所有编辑者收敛到相同状态，且没有操作被静默丢失。

面试官： OT 还是 CRDT？

候选人： 两者均有效；我选择讨论 OT，因为它是 Google Docs 的历史选择。OT 需要中心化排序器，因为转换一个操作依赖于知道之前发生了哪些操作，服务端是顺序的唯一真实来源。CRDT 更适合点对点场景，因为操作携带足够的元数据（时间戳、唯一 ID）可以无需协调地进行交换合并。CRDT 在现代系统中很流行，但每操作的开销更高。

面试官： 带我走一遍按键流程。

候选人： 用户在位置 5 输入"X"。客户端生成一个操作 Insert("X", at=5, parent_seq=42) 并在本地应用——UI 立即显示 X。通过 WebSocket 将操作发送到服务端。服务端看到当前序号为 45（自用户上次同步以来已到达三个操作）。将用户的操作与操作 43、44、45 进行转换：也许这些插入操作将位置 5 移到了 7。新操作 Insert("X", at=7) 在序号 46 处排序。服务端持久化到操作日志，广播给所有对等节点。发起方客户端收到序号 46 的确认；若本地位置不同则进行协调。

面试官： 两个用户同时编辑同一行。

候选人： OT 来处理。用户 A 在位置 3 插入"X"，parent_seq 为 42。用户 B 在位置 3 插入"Y"，parent_seq 为 42，并发发生。服务端先收到 A，分配序号 43，广播出去。然后收到 B 的操作，parent_seq 为 42；与操作 43（即 A 在位置 3 的插入）进行转换；B 的位置移至 4。新操作 Insert("Y", at=4) 在序号 44 处排序。所有对等节点根据转换规则（通常基于作者 ID 进行决胜）收敛到"AB..."或"XY..."顺序。关键点：收敛是有保证的。

面试官： 用户离线后输入了 100 段内容，然后重新上线。

候选人： 客户端使用单调递增的 client_op_id 缓冲操作。重新连接时，按顺序发送整个缓冲区。服务端将每个操作与当前状态（已向前推进）进行转换，对其排序，并广播。用户的编辑被插入到当前文档的适当位置。如果用户离线编辑的部分文档被其他人删除了，转换会处理：对已删除区域的编辑要么被跳过，要么根据策略锚定到最近的存活文本。

面试官： 5000 人同时编辑一个文档。

候选人： 这是最难的情况。该文档的会话主节点是瓶颈所在。缓解措施：5000 人中大多数可能是查看者而非编辑者。只读查看者从分流缓存而非主节点获取操作流。在实际编辑者中，对每位编辑者进行操作速率限制。对于真正的病毒式编辑会话，我们可能降级为一次只允许一个写入者（锁模式），或限制编辑者数量。

面试官： 权限？

候选人： Drive 负责权限管理。连接时，会话主节点检查用户的访问权限。每个操作通过快速缓存校验进行确认。权限被撤销时，主节点收到通知并踢出该用户；后续操作将被拒绝。缓存 TTL 较短，以保证撤销传播的及时性。

面试官： 历史记录如何存储？

候选人： 操作日志本身就是历史记录。定期快照进行压缩：每 N 个操作或 M 分钟，将当前状态物化为一个快照。打开时，加载快照 + 之后的操作。对于命名版本（"最终稿"），用户标记快照。操作日志无限期保留，用于时光机式恢复。

面试官： 评论呢？

候选人： 通过字符偏移量锚定到文本区间，编辑后依然存在。当周围文本移动时，锚点也随之移动（OT 也适用于锚点）。评论与文本操作分开存储，以避免使主日志膨胀。支持线程式回复和解决状态。

面试官： 多区域？

候选人： 每个文档有一个归属区域。会话主节点在那里运行。来自其他区域的编辑者通过 WebSocket 连接，接受跨区域延迟（几百毫秒）。快照和操作日志在全球范围内复制，以实现快速只读访问。故障转移区域保持热备。我们不尝试为每个文档运行主-主会话主节点，因为串行化不变式会被破坏。

面试官： 文档在保存过程中崩溃。

候选人： 操作在向客户端发送确认之前先持久化到操作日志。若会话主节点宕机，副本接管：它拥有最近一次复制操作之前的操作日志。客户端重新连接，重放任何尚未确认的操作。追加写日志是恢复机制。

面试官： 总结。

候选人： 每文档会话主节点通过 OT 串行化操作。每文档追加写操作日志，定期生成快照。本地乐观应用，服务端转换，广播给对等节点。离线编辑缓冲并协调。在线状态与评论相互独立。权限通过 Drive 进行每操作校验。多区域使用每文档归属区域加复制快照。最难的情况是单个非常热门的文档，其会话主节点成为串行化瓶颈。

3. 练习备注

五分钟版本

  1. 明确：实时协作、离线、历史记录、权限。
  2. 不变式：收敛；无操作丢失。
  3. 每文档会话主节点；OT 或 CRDT。
  4. 追加写操作日志；定期生成快照。
  5. 本地乐观应用；服务端转换。
  6. 离线缓冲；重连时协调。
  7. 在线状态与评论相互独立。
  8. 每操作权限校验；多区域每文档归属区域。

一分钟结尾陈述

Google Docs 是一个通过会话主节点串行化的每文档操作日志。OT（或 CRDT）使所有编辑者在并发编辑时保持收敛。本地乐观应用提供即时反馈；服务端转换协调差异。快照使加载开销降低；离线编辑在重连时缓冲并协调。在线状态与评论独立于主操作流水线。多区域使用每文档归属区域。最难的扩展情况是一个非常热门的单一文档。

最佳面试用语

  • "OT 需要中心化排序器；CRDT 支持点对点合并。"
  • "每个文档的会话主节点是该文档的中心串行化器。"
  • "本地乐观应用提供即时反馈；服务端转换进行协调。"
  • "离线编辑在本地缓冲，并在重连时通过转换进行重放。"
  • "每文档归属区域；跨区域复制为只读。"
