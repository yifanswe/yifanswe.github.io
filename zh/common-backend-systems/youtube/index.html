YouTube — Video Upload, Storage, Playback (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Hard · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建YouTube系统。创作者上传视频；观众搜索/发现并按适合其网络的码率进行流式播放。5亿创作者，20亿日活跃观众，每分钟上传500小时视频，数亿并发流。存储和CDN出流成本驱动整体设计。

1. 需求（约 5 分钟）

功能需求 - 上传一段视频。 - 以匹配观众网络的画质流式播放视频（自适应码率）。 - 查看播放量及视频详情页（标题、缩略图、作者、基础元数据）。

我会暂不考虑推荐算法、评论、直播深度以及商业化。

非功能需求 - 播放启动时间 < 2 s——这是最核心的用户体验指标。 - 在EB级规模下具备成本效益；存储和出流是预算的核心。 - 通过CDN实现多地区、低延迟视频服务。 - 播放量和元数据的最终一致性。 - 持久性——源视频达到11个9的可靠性。

容量估算 - 500 hr/min × 60 min × 24 = 每天上传720k小时。以1 GB/小时（编码后、多码率梯度）计算：每天上传约720 TB；多码率梯度使总量扩大5–10倍 → 每天约存储5 PB。 - 20亿观众 × 60分钟 × 平均5 Mbps = 约75 Pb/s的总出流带宽。 - 元数据相比媒体体量微小：即便100亿条视频行 × 2 KB 也仅约20 TB。媒体占主导：每天5 PB的新编码字节意味着每月150 PB（复制/分层前）。 - CDN出流是核心业务成本：一个热门10分钟视频以5 Mbps被观看1亿次，共需交付约375 TB。设计的核心在于减少回源读取并尽量降低每分钟存储的比特数。

2. 核心实体（约 2 分钟）

  • 视频 — video_id，owner、title、description、duration、status、source_url、ladder。
  • Ladder — 若干 (resolution, bitrate, codec) → segment_url_list的集合。可流式播放的产物。
  • 上传 — 分片上传状态。
  • 观众事件 — 观看、观看时长心跳。用于驱动排名和数据分析。

3. API（约 5 分钟）

POST  /v1/uploads                       → { upload_id, multipart_urls[] }
PUT   <multipart_url>                   (binary chunks)
POST  /v1/uploads/{upload_id}/complete  → { video_id, status: processing }
GET   /v1/videos/{id}                   → metadata + ladder manifest URL
GET   <manifest_url>                    → HLS/DASH manifest
GET   <segment_url>                     → media segment

Manifest URL经CDN前置；私密视频需签名。

上传游标为分片上传ID加已完成的分片编号；播放游标存在于播放器的manifest/segment序列中，而非API服务器。

4. 数据流（约 5 分钟）

上传（分片）→ 原始文件存入对象存储 → 转码流水线生成ABR码率梯度 → 发布manifest → CDN预热 → 视频上架。

观众请求manifest → 播放器选择码率 → 通过CDN拉取片段 → 根据缓冲区健康状况自适应调整码率。

5. 高层设计（约 10–15 分钟）

  1. 上传服务。 向对象存储（S3）发起分片上传。客户端直接上传。完成后写入Video行（status=processing），并发布 video.uploaded.
  2. 转码流水线。 - 任务调度器将源视频切分为片段分发给工作节点集群（GPU加速，使用ffmpeg或x264/x265/AV1）。 - 针对码率梯度中每个码率×分辨率×编解码器组合，生成HLS/DASH兼容片段（通常为2–6秒片段）。 - 片段和manifest存储在对象存储中，支持多CDN分发。
  3. 元数据。 视频元数据存储于分片数据库（Spanner或Cassandra），搜索索引使用Elasticsearch。
  4. 播放路径。 观众请求视频详情页 → 元数据服务返回标题、作者、manifest URL。播放器从CDN加载HLS/DASH manifest → 拉取片段。
  5. CDN 策略。 多层CDN缓存体系：边缘节点 → 区域盾构节点 → 源站（对象存储）。热门视频预热至边缘节点；长尾视频按需从源站冷拉取。
  6. 推荐/发现（超出本次设计范围）。 提及其作为独立排名服务并行存在。
  7. 播放量统计。 播放器发送观看时长心跳 → Kafka → 聚合器将计数汇总至ClickHouse + Redis实时计数器。

端到端叙述：创作者上传1 GB视频 → 分片上传至S3 → 转码任务扇出，生成1080p/720p/480p/240p × H.264及AV1码率梯度（共约源文件5倍大小）→ 发布manifest → 元数据 status=ready。东京的一位观众打开视频详情页 → 从区域缓存获取元数据 → 从东京CDN边缘节点获取manifest → 播放器初始协商720p（约3 Mbps）→ 片段从东京边缘节点提供 → 前10秒后缓冲区状态良好，播放器切换至1080p。

6. 深度探讨（约 10 分钟）

具体存储契约。

存储 键/数据结构 条目大小 + TTL 分片键
上传状态 upload:{upload_id} → 分片映射、owner、过期时间 1–10 KB，未完成状态TTL 7天 upload_id
源文件对象 s3://video-source/{video_id} 不可变原始文件 MB–GB，持久保留/冷存储 video_id
片段对象 s3://video-segments/{video_id}/{codec}/{rendition}/{segment} 0.5–4 MB/片段，分层存储 video_id, rendition
Manifest manifest:{video_id}:{codec} HLS/DASH文本 10–100 KB，每个版本不可变 video_id
元数据 video:{video_id} 数据行 {owner,title,status,manifest_version} ~2 KB，保留 video_id
转码任务 transcode:{video_id}:{rendition}:{segment} 队列条目 ~200 B，直至完成 video_id
播放量计数器 views:{video_id}:{minute} + 汇总数据 8–32 B/计数器，TTL按粒度设置 video_id

各存储容量估算。 即便存在数百万未完成的上传，上传状态也很小： 1M × 10 KB 约为10 GB。元数据同样较小：若每分钟500小时平均视频时长为10分钟，则每天约有430万次上传； 4.3M × 2 KB × 3 约为每天26 GB。每个manifest为 100 KB ，每天累计数百GB。媒体占主导：每天约720 TB的原始文件和约5 PB的编码字节（源站复制/分层前）。观看事件数据量较大但远不及媒体： 2B viewers × 60 min/day × one beacon/10s × 100 B 每天约1.2 TB（副本前）。成本的关键在于每观看分钟所传输和存储的字节数，而非行数。

播放路径追踪。 观众 → 视频详情API读取 video:{id} 和权限状态 → 返回签名manifest URL → CDN边缘节点提供manifest或从区域盾构节点拉取 → 播放器请求第一个低/中码率片段 → CDN边缘/盾构/源站通过不可变片段键进行查找 → 播放器根据缓冲区自适应码率。元数据/用户资料/推荐请求可与manifest拉取并行执行；媒体片段服务应完全绕过应用服务器。

转码的经济性。 最昂贵的基础设施。将视频切分为并行片段进行处理（切成6秒块，独立转码，再重新合并）。两个关键杠杆：- GPU流水线 （NVENC）用于快速处理常见码率梯度。- 软件编码器 （libx264、libaom-av1）用于更高质量/新编解码器，但延迟较高。- 按内容独立编码：根据内容复杂度（动画vs体育赛事）调整码率梯度——可节省20–40%的存储空间。

编解码器策略。 维护多套编解码器梯度：H.264用于兼容性，VP9/AV1用于现代设备。存储权衡——AV1可节省约30%的字节数，但需要更多计算资源。计算成本可在数月乃至数年的出流节省中摊销。

存储分层。 最近上传及热门内容：热层对象存储。长尾内容（数年前上传，鲜少观看）：温层/冷层。流量突增时按需升至热层。

为何选用对象存储而非HDFS风格的集群：片段具有不可变性、体量巨大，且对CDN源站友好，因此对象存储在无需管理存储节点的前提下提供了持久性和生命周期分层能力。HDFS可用于内部处理，但服务和数据保留更适合对象存储的语义。

对于元数据，当隐私状态、所有权和发布状态需要强一致性转换时，Spanner/分片SQL是理想选择；若将状态转换保持简单和事件驱动，Cassandra在全球读取规模上同样可行。我不会将媒体字节存入任何一者，因为媒体的访问模式是通过CDN进行不可变对象读取。

CDN缓存层级。 边缘节点（靠近用户）→ 区域盾构节点 → 源站。热门视频驻留在边缘节点；长尾内容从盾构节点再到源站冷拉取。缓存键包含片段URL（不可变，基于内容哈希）。

面试要点："应用服务器启动播放，但CDN承载产品体验；在YouTube规模下，每一次穿透到源站的缓存未命中都是一个成本缺陷。"

多CDN。 YouTube运营自有CDN（Google边缘网络），但认可多CDN策略的必要性：基于健康状态和成本进行流量调度，以实现冗余。

直播流（简要提及）。 低延迟流水线：分块CMAF片段（1–2秒），LL-HLS或DASH协议。通过RTMP摄入 → 实时转码 → CDN分发。目标端到端延迟（glass-to-glass）< 5秒。与VOD采用不同的基础设施路径。

搜索。 对标题/描述/标签建立倒排索引。发布时将新元数据流式写入ES，排名时结合互动信号。

播放量统计。 不要在关系型数据库中对每次观看递增计数。心跳 → Kafka → 聚合器 → Redis计数器 + ClickHouse汇总。展示的播放量为近似实时值，最终一致。

为何选用Kafka + ClickHouse/OLAP而非事务型计数器：观看事件是仅追加的分析型数据，而非面向用户所有权的数据。Kafka吸收突发心跳；Redis提供近似实时计数；ClickHouse提供低成本汇总。对病毒式视频而言，在关系型数据库中每个视频写一行会造成热点问题。

双索引框架。 视频包含用于播放的内容寻址媒体对象、用于视频详情页状态的元数据行、用于发现的搜索索引，以及用于播放量统计的分析汇总。这些是对同一 video_id的不同视图；媒体片段是不可变内容，而元数据/搜索/计数器是可接受滞后或重建的服务索引。

推荐系统（简述）。 双塔模型：视频塔 + 用户塔 → 向量嵌入 → 服务时进行ANN搜索。超出本次设计范围。

断点续传上传。 分片上传支持块级别重试；长时间上传可在蜂窝网络断线时恢复。服务器在超过TTL后丢弃未完成的上传。

故障模式。 转码器在任务中途崩溃：任务对片段具有幂等性；在另一工作节点上重启即可。CDN边缘节点不可用：流量切换至最近的其他边缘节点。
