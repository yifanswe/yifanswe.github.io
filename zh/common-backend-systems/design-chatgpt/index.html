1. 基础知识与解题思路

本题考察点

ChatGPT 是基于大型语言模型的流式聊天界面。难点在于 GPU 经济性——每生成一个 token 都需要 GPU 算力——同时还要处理会话状态、流式 token 投递、安全过滤，以及在有限 GPU 集群上为数百万并发会话提供服务。

其不变量如下：

每个聊天请求被路由到已加载所请求模型的推理工作节点，在模型延迟预算内生成 token，将 token 流式传输给客户端，在展示前应用安全过滤，按用户持久化对话历史，并确定性地进行计费/配额追踪；GPU 容量在用户间公平共享。

在 ICT5 / L6+ 级别，面试官在考察你是否能设计带批处理的 GPU 推理服务、通过 SSE/WebSocket 的流式 token 投递、会话状态管理、安全过滤管道，并能就 GPU 成本优化（连续批处理、KV cache、量化）进行推理。

60 秒面试框架

口述版本：

我将围绕四个核心系统设计 ChatGPT：持有会话状态的聊天会话服务、采用连续批处理为模型提供服务的 GPU 推理集群、随 token 生成即时投递的流式网关，以及输入输出双端的安全过滤管道。关键决策包括 GPU 调度与批处理、对话上下文管理、流式协议、安全集成和成本优化。

首先明确需求

功能性需求：

  • 用户发送提示，助手流式返回响应。
  • 多轮对话并携带历史记录。
  • 多个模型（GPT-4、GPT-4o 等）。
  • 文件和图片上传（视觉能力）。
  • 函数调用 / 工具使用。
  • 按用户保存对话历史。
  • 编辑 / 重新生成消息。
  • 每用户自定义指令。

非功能性需求：

  • 首 token 时间（TTFT）低于 1 秒。
  • 每秒 token 吞吐量满足流式 UX 要求。
  • 数千万个并发会话。
  • 高可用性。
  • 严格安全：过滤不安全内容。
  • 多地区。
  • 按 token 计费的精度要求。

以下内容超出范围，除非被特别询问：

  • 模型训练管道（独立的大规模系统）。
  • 自定义 GPTs / Agent 平台内部实现。
  • 记忆功能内部实现。
  • 插件 / 浏览功能细节。

推理流程

  1. 用户发送消息。
  2. API 认证 + 配额检查。
  3. 输入安全过滤：对提示分类；如违反策略则拒绝。
  4. 将用户消息追加到会话存储中的对话。
  5. 构建推理请求：完整对话历史 + 系统提示。
  6. 发送至推理调度器。
  7. 调度器选择已加载所请求模型的 GPU 工作节点。
  8. 工作节点将请求加入连续批次队列。
  9. 前向传播每次生成一个 token。
  10. 每个 token 通过 SSE 流式传输回客户端。
  11. 每个 token 或每个 chunk 应用输出安全过滤。
  12. 完成后，持久化助手消息；更新配额。

连续批处理

GPU 优化的核心。朴素方案：每个 GPU 每次处理一个请求——浪费，GPU 具有大规模并行性。

连续批处理：\n- GPU 以 N 个请求为一批运行推理。\n- 每次前向传播为批次中每个活跃请求生成一个 token。\n- 已完成的请求离开批次；新请求加入。\n- 每个请求维护 KV cache（键值注意力缓存）。\n- 有效吞吐量远高于顺序处理。

实现方案：vLLM、TGI（HuggingFace Text Generation Inference）或自研服务栈。

面试用语：

连续批处理是 GPU 的经济命脉。没有它，一台 8xH100 服务器只能服务少数用户；有了它，可服务数百人。KV cache 管理决定工作节点能维持多少并发流。

KV Cache

Transformer 注意力机制需要所有先前 token 的键和值。每个 token 重新计算极其浪费。

  • KV cache 为活跃对话中的每个 token 存储 K、V 张量。
  • 内存开销随上下文长度和并发请求数线性增长。
  • KV cache 驱逐：当内存紧张时，完成某个请求或将其换出到 CPU。
  • 每个工作节点预分配缓存池；限制每个工作节点的并发数上限。

流式协议

Server-Sent Events（SSE）是典型选择：

  • 服务端发送 data: {token}\n\n 每个 token 发送一次。
  • 客户端通过 EventSource API 接收。
  • 半双工；客户端到服务器是独立请求。

WebSocket 是全双工的替代方案；适用于中途取消流。

流式连接在整个响应期间保持打开（长生成时从数秒到数分钟）。

安全过滤

两个阶段：

输入过滤： - 分类器检查用户提示是否违反策略。\n- 拒绝明显不安全的内容（CSAM、恐怖主义等）。\n- 对边界情况（如医疗、法律建议）软警告或转化处理。

输出过滤： - 每 token 或每 chunk 分类器。\n- 如果输出看起来不安全，中止流并致歉。\n- 部分检查在完整响应上进行（越狱检测）。

对于敏感话题：模型微调强制拒绝 + RLHF；安全分类器作为双重保险。

混合方案：小型快速分类器内联 + 大型模型处理模糊情况。

对话状态管理

对话历史是模型的上下文。随着历史增长：

  • 上下文窗口限制（根据模型不同为 8k、32k、200k token）。
  • 对于长对话，需截断或总结较早的消息。
  • 摘要化：定期将早期消息总结并替换为摘要。
  • "记忆"功能：提取用户的长期信息，单独存储，注入到系统提示中。

持久化对话消息，确保用户可以恢复会话。

GPU 集群调度

集群中的 Worker：

  • 每个 Worker 拥有一块或多块 GPU。
  • 一个 Worker 加载一个（有时几个）模型。
  • Worker 按模型分组；调度器将请求路由到对应池。
  • 在一个池内，调度器选择有容量的 Worker。
  • 在整个集群中进行装箱（bin-packing）调度。

自动扩缩容：

  • 追踪每个模型的活跃流数量。
  • 需求上升时添加 Worker；空闲 Worker 则逐步下线。
  • 冷启动：加载模型权重（体积大、速度慢）。
  • 缓解措施：保持一个热备池；预加载热门模型。

多模型部署：- 较大的模型（GPT-4）每个 Worker 需要多块 GPU（模型并行）。- 较小的模型（3.5）可放在一块 GPU 上。- 量化（FP8、int8）减少内存占用并提升吞吐量。

工具/函数调用

模型决定调用某个工具（具有定义好 schema 的函数）：

  1. 模型输出： tool_call(name, args).
  2. 流式传输暂停；客户端或服务端执行工具。
  3. 工具结果作为新消息反馈回模型。
  4. 推理在上下文中携带结果后继续。

支持浏览、代码执行、图像生成、自定义工具。

视觉/多模态

图像被分词为图像 embedding：

  1. 图像上传后，视觉预处理器提取图像块（patch）。
  2. Embedding 被前置拼接到提示的文本 token 之前。
  3. 模型进行统一的注意力计算。

增加延迟和内存开销；重复使用时可预先缓存。

数据 分区键 原因

一致性模型

层级 一致性 原因
对话历史 强一致性（每条消息写入） 用户期望连续性。
配额 强一致性（CAS 递减） 资金相关。
流式 尽力而为的投递 客户端重试。

多区域策略

  • 对话存储按用户归属区域。
  • GPU 集群分布于多个区域。
  • 推理请求路由到最近的、对所请求模型有容量的区域。
  • 支持跨区域故障转移。

故障模式与缓解措施

故障模式 缓解措施 监控指标
GPU Worker 在流式传输中途崩溃。 保存部分响应；在新 Worker 上从 prefill 恢复。 Worker 故障。
容量耗尽。 使用有界等待时间的队列；超出后进行限流。 队列深度。
不安全输出泄漏。 多层安全；事后审计。 安全事件。
流式连接中断。 客户端重连；服务端从游标处恢复。 重连率。
Token 配额在流式传输中途耗尽。 停止生成；交付部分结果；给出明确提示信息。 配额耗尽。
冷启动延迟。 热备池；预加载热门模型。 TTFT 分布。
热点用户（单个用户发起数千次请求）。 每用户并发限制；限流。 每用户 RPS。
长上下文撑爆 KV cache。 截断或总结历史记录。 上下文溢出率。
模型上线回归。 金丝雀发布 + 自动回滚。 每模型错误率。
跨区域故障转移。 热备副本集群。 故障转移 RTO。

运维指标

正确性：

  • 安全过滤器误报/漏报。
  • 对话持久化成功率。
  • 配额漂移与计费差异。

容量：

  • TTFT（首 token 时间）p99。
  • 每个流的 Token/秒。
  • GPU 利用率。
  • 队列深度。

业务层面：

  • 活跃用户数；每用户 token 数。
  • 工具调用使用量。
  • 付费转化率。

需要指出的权衡

  • 批次越大 = GPU 利用率越高，但 TTFT 越差。
  • 上下文越长 = 质量越好，但内存占用越多、速度越慢。
  • 安全策略越严格 = 事故越少，但拒绝越多（用户投诉）。
  • 量化节省内存，但质量略有下降。
  • 自动扩缩容越激进 = 延迟越低，但成本越高。
  • 粘性流式连接 vs. 失败时重连。

常见陷阱与更优做法

陷阱 更优做法
每个 GPU 一个请求。 连续批处理（Continuous batching）。
每个 token 重新计算 KV cache。 维护 KV cache。
同步响应。 通过 SSE/WS 流式传输。
仅在开始时进行一次安全检查。 输入前 + 每个 token 输出过滤。
单一模型部署。 每模型一个池；按请求路由。
无限上下文。 截断或总结；强制执行限制。

面试用语

ChatGPT 是基于 GPU 推理的流式聊天服务。经济层面的挑战是 GPU 利用率——连续批处理是解决方案。用户体验层面的挑战是通过 SSE 逐 token 流式传输。安全层面的挑战是作为独立平面的输入+输出过滤。对话状态是每用户持久化的；配额和计费是强一致性计数器。

2. 模拟面试

面试官： 设计 ChatGPT。

候选人： 四大核心系统：持有对话历史的聊天会话服务、使用连续批处理的 GPU Worker 推理集群、通过 SSE 投递 token 的流式网关，以及对输入和输出进行安全过滤的模块。不变量是：每个请求都经过安全检查、在预算内生成、流式传输给客户端、持久化历史记录，并递减配额。主要成本是 GPU；主要性能挑战是 TTFT。

面试官： 带我了解一条聊天消息的流程。

候选人： 用户提交提示。网关进行身份验证，检查配额。输入安全分类器扫描是否违反策略——明显不安全的请求被拒绝。会话存储将用户消息追加到对话记录。构建推理请求：完整消息历史 + 系统提示。调度器为所请求的模型选择 GPU Worker 池，并选择有容量的 Worker。Worker 将请求加入连续批次。前向传播开始生成 token。每个 token 通过 SSE 流式返回；输出安全过滤器逐块验证。完成后，持久化助手消息并更新配额。

面试官： 解释一下连续批处理。

候选人： GPU 是大规模并行的；逐请求顺序推理会浪费算力。连续批处理将并发请求合并为一次前向传播。每次传播为批次中每个活跃请求生成一个 token。每个请求维护各自的 KV cache，因此后续传播无需重新计算之前的 token。完成的请求离开批次；新请求加入。结果：单台 H100 服务器可处理数百个并发流，而不只是寥寥几个。vLLM 和 TGI 实现了这一模式。

面试官： KV cache 是什么？

候选人： Transformer 注意力机制需要所有先前 token 的 key 和 value。每个 token 重新计算的复杂度是 O(n²)。KV cache 在每个 token 之后存储 K、V 张量，因此下一个 token 的注意力计算只需 O(n) 的边际开销。内存开销随上下文长度 × 并发请求数增长。Worker 预分配 KV cache 池；并发量受总缓存内存限制。

面试官： TTFT——如何保持低延迟？

候选人： 有几个因素。首 token 时间（TTFT）包括：排队等待、prefill（并行处理所有输入 token），以及第一个解码步骤。缓解措施：保持 GPU 集群热备并加载好模型；限制队列深度；优先保障 TTFT 而非批次大小。连续批处理也有助于改善 TTFT，因为新请求可以加入活跃批次，无需等待批次排空。

面试官： 流式传输协议？

候选人： 通常是 SSE——服务端发送 data: {token}\n\n 每个 token。客户端使用 EventSource API。半双工；客户端到服务端是独立的 POST 请求。WebSocket 作为全双工替代方案（支持中途取消）。连接在整个响应期间保持打开——对于长生成可能持续数秒到数分钟。断线后可携带游标重连以恢复。

面试官： 安全？

候选人： 两阶段。输入过滤：分类器扫描提示；拒绝明显不安全的内容（CSAM、恐怖主义等）。对边缘情况给出软性警告。输出过滤：逐 token 或逐块分类器——如果输出漂移至不安全，则中止并致歉。双重保险：模型微调通过 RLHF 强制执行拒绝策略，安全分类器是二级检查。部分检查在流式传输结束后对完整响应进行评估，以检测越狱模式。

面试官： 长对话——100 条消息，50k token。

候选人： 上下文窗口限制（取决于模型：8k、32k、128k、200k）。对于更长的对话：截断或总结。摘要化：定期将早期消息替换为摘要消息，保留最近 N 条原文。"记忆"功能：将用户的长期信息单独提取，注入到系统提示中。完整历史记录持久化供用户查阅；仅将加窗/摘要后的版本发送给模型。

面试官： 工具调用？

候选人： 模型输出中可能包含 tool_call(name, args) 以结构化格式表示的内容。流式传输暂停；服务端或客户端执行工具（浏览网页、运行代码、调用 API）。工具结果作为新消息反馈。推理在上下文中携带结果后恢复。支持插件、浏览、代码执行、图像生成、自定义工具。

面试官： GPU 集群——如何扩容？

候选人： Worker 按模型分组。每个 Worker 加载一个（或几个）模型；较大的模型使用多 GPU 模型并行。按需求对每个模型进行自动扩缩容。热备池预加载热门模型。冷启动很慢（加载数 GB 的权重）——通过保持最低热备容量来缓解。量化（FP8、int8）在轻微质量损失的代价下减少内存占用并提升吞吐量。

面试官： 一个 Worker 在流式传输中途崩溃。

候选人： 流式连接中断。客户端收到部分响应。重连后，服务端可从已持久化的部分恢复：在新 Worker 上重新 prefill 上下文至最后已投递的 token，然后继续生成。或者直接呈现部分结果并提供重新生成按钮。具体取决于 UX 设计选择。

面试官： 配额与计费？

候选人： 每用户 token 预算。每个 token 递减计数器（输入 + 输出 token）。预检查：估算响应大小；如果严重超出预算则拒绝。流式传输期间实时递减；如果耗尽，停止生成并提示"配额耗尽"。计费定期汇总使用量；按套餐收费。

面试官： 多区域？

候选人： 对话存储按用户归属区域。GPU 集群分布于多个区域。推理请求路由到最近的、对所请求模型有容量的区域。当本地容量耗尽时支持跨区域故障转移。如果用户归属区域负载较重，部分用户可能会遭遇较高延迟。

面试官： 总结。

候选人： 基于 GPU 推理的流式聊天。连续批处理是 GPU 效率的杠杆。KV cache 管理限制了每个 Worker 的并发量。SSE 向客户端投递 token。两阶段安全（输入+输出）过滤违规内容。对话历史按用户持久化，长上下文通过截断/摘要处理。工具调用集成外部执行。每用户配额与计费。多区域集群。最棘手的经济问题是 GPU 成本；最棘手的性能问题是 TTFT。

3. 练习备注

五分钟版本

  1. 澄清：聊天、流式传输、模型、安全、规模。
  2. 不变量：安全的输入/输出；流式 token；持久化历史记录。
  3. 会话 + 推理 + 流式传输 + 安全。
  4. GPU 上的连续批处理与 KV cache。
  5. SSE 流式传输协议。
  6. 两阶段安全过滤。
  7. 长上下文的对话截断/摘要化。
  8. 每用户配额；多区域。

一分钟结尾陈述

ChatGPT 是基于 GPU 推理的流式聊天服务。连续批处理与 KV cache 是 GPU 效率的杠杆——它将每台 GPU 从处理少量流的服务器变为处理数百流的服务器。SSE 投递 token。两阶段安全过滤输入和输出。对话历史按用户持久化；长上下文被截断或总结。工具调用集成外部执行。每用户配额追踪 token 使用量。多区域配合 GPU 集群自动扩缩容。主要成本是 GPU；主要性能指标是首 token 时间（TTFT）。

  • "连续批处理使 GPU 推理具有经济可行性。"
  • "KV cache 管理限制了每个 Worker 的并发量。"
  • "通过 SSE 流式传输；连接在整个响应期间保持打开。"
  • "安全是两阶段的——输入分类器加输出过滤器。"
