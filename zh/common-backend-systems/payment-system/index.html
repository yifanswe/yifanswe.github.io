Payment System — Stripe / PayPal Class (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Hard · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建一个支付系统。商家通过我们的 API 向客户卡片收费；我们路由到卡网络（Visa/MC）或钱包，结算资金，处理退款和争议，且绝不丢失或重复收费。每天 1 亿笔交易，峰值约 5k/秒，平均交易金额 $30。围绕资金的故障模式才是面试的核心。

1. 需求（约 5 分钟）

功能需求 - 商家提交收费意图（客户支付方式 + 金额）。 - 系统向卡网络发起授权，捕获资金，返回成功/失败。 - 退款和部分退款。 - 终态事件的 Webhook 回调通知商家。

我会推迟处理周期性账单、订阅、多币种转换和欺诈模型细节。

非功能需求 - 正确性高于一切。 绝不重复收费，绝不静默丢弃收费请求。 - 所有写操作均幂等——网络会重试；我们也必须重试而不重复资金流动。 - 账本余额强一致性。 - 每次状态转换的持久审计日志（合规要求）。 - p99 收费 < 2 s（大部分时间是卡网络耗时）。 - 高可用——支付中断意味着每个商家的真实营收损失。

容量估算 - 每天 1 亿笔 = 平均 1200/秒，峰值 5k/秒。 - 交易记录约 7 年保留期（PCI/监管要求）。 - 热流量并不算大；保留的资金历史才是。1 亿/天 × 7 年 ≈ 2550 亿笔交易。 - 若每笔交易产生约 4 条账本分录，每条约 250 B，账本存储约 255 TB 原始数据，含 3 副本约 765 TB（索引/冷压缩前）。账本 + 审计是主要成本。

2. 核心实体（约 2 分钟）

  • PaymentIntent ——商家订单： intent_id, merchant_id, amount, currency, customer_id, status.
  • Charge ——资金移动的一次尝试： charge_id, intent_id, status, network_ref.
  • LedgerEntry ——复式记账： (account, debit/credit, amount, ref)。权威来源。
  • PaymentMethod ——令牌化的卡片/钱包句柄。
  • 退款 ——对收费的部分或全额冲销。

3. API（约 5 分钟）

POST  /v1/payment_intents                { amount, currency, payment_method, customer }   (Idempotency-Key header required)   → PaymentIntent
POST  /v1/payment_intents/{id}/confirm                                                                                       → Charge
POST  /v1/charges/{id}/refunds            { amount }                                                                          → Refund
GET   /v1/charges/{id}
POST  /v1/webhooks/{merchant_id}/events                                                                                       // outgoing

每个改变状态的请求都必须携带 Idempotency-Key。 这是基石。

5. 高层设计（约 10–15 分钟）

  1. API Gateway。 Auth、限流、idempotency-key 中间件：查找 (merchant_id, key) → 若已见过则返回缓存响应，否则放行。
  2. PaymentIntent 服务。 在强一致性存储（Spanner 级别，或每分片有主节点的分片 MySQL）中写入 intent 行。状态机： requires_payment_method → requires_confirmation → processing → succeeded | failed.
  3. 授权服务。 确认后，构建网络消息（ISO 8583 或处理器 API 的 HTTP），路由到卡网络适配器。发送前持久化出站尝试；持久化响应。超时时标记为 unknown （而非 failed）并稍后对账——永远不要在超时时假设失败，那会导致重复收费。
  4. 账本服务。 复式记账。每次资金移动 = 一个账户借方 + 另一个账户贷方，总计必须平衡。示例： - 授权：pending_authorization（客户）↔ pending_capture（商家应付款）。 - 捕获：pending_capture → merchant_balance；客户 pending → settled。 - 退款：冲销分录。 存储于仅追加的账本表中，按账户分区。
  5. Webhook 服务。 终态时，将投递任务加入商家 Webhook URL 队列。以退避方式重试最长 3 天；持久化队列。
  6. 网络适配器。 按卡网络或处理器（如 Visa、Stripe Connect）划分。每个适配器管理自己的连接池、消息格式和重试策略。
  7. 对账。 每日批处理任务拉取当天网络结算文件，与我们的账本比对，标记差异。
  8. 审计日志。 每次状态转换追加到不可变日志（Kafka 配置超长保留期 + 冷存储）。

端到端叙述：商家携带幂等键 K 调用 POST /payment_intents 。网关检查；未见过。写入 intent 行（status=requires_confirmation）。商家调用 /confirm → 授权服务向 Visa 发送 ISO 8583 消息 → Visa 在 800 ms 内响应批准 → 账本写入复式分录（客户待处理：−$30；商家应付款：+$30）→ intent.status=succeeded → 触发 Webhook。两天后，来自 Visa 的结算文件到达 → 对账确认记录匹配 → 账本分录变为 settled。

6. 深度探讨（约 10 分钟）

具体存储与分片键。

存储 键模式 数据结构 / 条目 TTL / 保留期 分片键
幂等性 idem:{merchant_id}:{key} 状态 + 响应哈希/正文指针，1–4 KB 24–72 小时热存储，7 天持久化 (merchant_id, key)
PaymentIntent pi:{intent_id} 强一致性行，金额/状态/客户，约 1 KB 7 年 merchant_id + intent_id
收费尝试 ch:{charge_id} 网络请求/响应引用/状态，约 1 KB 7 年 charge_id
账本 ledger:{account_id}:{seq} 仅追加借贷分录，约 250 B 7 年热/温存储，更长归档 account_id
Outbox/webhook webhook:{merchant_id}:{event_id} 目标/状态/尝试次数/正文指针，约 1 KB 重试 3 天，状态保留 90 天 merchant_id
审计日志 audit/yyyy/mm/dd/{partition} 不可变事件流/对象记录，约 500 B/事件 7 年以上 日期 + 分区

账本是内容和真相来源；缓存和幂等键是控制平面的安全护栏。我会明确避免将 Redis 作为任何资金移动的权威存储。

各存储容量估算。

存储 / 索引 Math 备注
幂等性 1 亿/天 × 2 KB 均值 × 3 ≈ 600 GB（24 小时）；最多约 1.8 TB（72 小时） 热安全护栏，TTL 有界
PaymentIntent 2550 亿（7 年）× 1 KB × 3 ≈ 765 TB 冷年份压缩/归档
收费尝试 2550 亿 × 1 KB × 3 ≈ 765 TB 包含网络引用，不含完整 PAN/卡数据
账本 2550 亿笔 × 4 条分录 × 250 B × 3 ≈ 765 TB 权威资金历史
Webhook outbox 1 亿/天 × 1 KB × 90 天 × 3 ≈ 27 TB（若状态热存储） 活跃重试集合小得多
审计日志 2550 亿笔 × ~5 个事件 × 500 B × 3 ≈ 1.9 PB 原始数据 冷对象存储 + 压缩带来主要节省
结算文件 1 亿/天 × 200 B ≈ 20 GB/天原始数据 批量对账输入

Kafka 适合审计/事件扇出，但它不是账本。outbox 行首先与状态转换一起提交，然后 broker 投递可以重试。这样可确保持久支付状态先于异步通知。

确认收费路径。 1. 网关对商家进行认证并检查 idem:{merchant_id}:{key}；若同一键有其他请求正在进行，则等待或返回最终结果。 2. PaymentIntent 服务 CAS 更新 requires_confirmation → processing 对于 pi:{intent_id}。 3. 在允许的情况下并行加载保险库中的令牌化支付方式并进行欺诈/风险检查；任一失败则 intent 转为 failed 并附审计事件。 4. 授权服务在向网络发送任何内容之前写入 ch:{charge_id} ：将 network_request_persisted 。 5. 网络适配器发送授权/捕获请求。批准响应触发账本事务：客户待处理借方和商家应付款贷方必须一起提交。 6. Intent/Charge 转为终态，outbox 写入 webhook:{merchant_id}:{event_id}，幂等记录存储精确响应。 7. 若适配器超时，charge 保持 unknown；重试/对账以 network_ref为键，而非创建第二笔收费。

退款和读取路径。 1. 退款请求检查 idem:{merchant_id}:{refund_key} 并读取 ch:{charge_id} 上检查 charge_id。 2. 服务验证 refunded_amount + requested_amount <= charge.amount，然后创建 refund:{refund_id} 并在发送网络退款请求之前持久化出站尝试。 3. 网络批准后，账本在分片 account_id上写入冲销复式分录；收费的物化 refunded_amount 通过 CAS 更新。 4. GET /v1/charges/{id} 读取 ch:{charge_id}, pi:{intent_id}，以及并行的账本摘要。若收费时间超过热保留窗口，API 回落到冷存档，可能返回更慢但仍完整的审计历史。 5. 客户端携带的唯一会话状态是幂等键。商家交易导出的分页使用不透明游标 {merchant_id, created_at, charge_id} ，因此导出不依赖资金历史的偏移扫描。

双索引框架。 账本以账户为键，因为余额按账户计算。商家仪表盘以 merchant_id 。 created_at为键按交易索引。这是对同一资金事件的两个索引；仪表盘索引可以有延迟，但账本不能。

幂等性——核心所在。 每个变更端点接受一个 Idempotency-Key。服务端存储 (merchant_id, key) → response, status ，有效期 ≥24 小时。重复请求返回缓存响应而不重新执行。键存储使用 Redis 集群 + 持久化后端。关键在于：键锁定操作——相同键的飞行中请求等待（带超时）；后续请求返回结果。网络必然会重试；我们必须安全地吸收重试。

超时 = "未知"，而非"失败"。 若卡网络调用超时，真相未知。将收费标记为 failed 并接受重试可能导致重复收费。应标记为 processing/unknown；稍后通过网络的 stand-in advice 或结算文件进行对账。商家看到 pending；客户不会被收费两次。

复式账本。 每次资金移动是一对平衡的分录。借方总额 = 贷方总额，永远如此。账户余额通过派生（分录之和）或物化（为性能）获得。审计不变量：在任意快照下，所有账户之和为零。这可以发现否则会隐藏的 Bug。

状态机。 PaymentIntent 状态转换由 CAS 更新严格控制： UPDATE intent SET status='succeeded' WHERE id=X AND status='processing'。非法转换返回错误。每次转换都向审计日志发出事件。

状态 数据真实来源 允许的下一个状态 恢复行为
requires_confirmation pi:{intent_id} processing, canceled 使用相同键安全重试确认
processing intent + charge 行 succeeded, failed, unknown 超时时不暴露为失败
unknown charge 行 + 网络引用 succeeded, failed 通过网络 advice/结算文件对账
succeeded 账本已提交 仅退款/争议流程 不可变成功；无法向后转换
failed charge 终态原因 新 intent 或新确认尝试 无账本移动

为何使用 Spanner 级别的存储而非 Cassandra：我们需要事务性状态转换和账本不变量，而非写入吞吐量。若能将账户/商家写入保持在本地，每分片一个主节点的分片 MySQL 是合理替代；Cassandra 更适合仅追加的审计，而非权威余额。

精确一次 Webhook 投递。 商家的 Webhook 端点也可能失败。出站投递已持久化；以指数退避方式重试，最长 3 天；商家需根据事件 ID 去重（我们会附带）。

PCI 合规。 卡号（PAN）仅经过令牌化服务，该服务将其加密存储于隔离的保险库中。应用代码只看到令牌。这降低了我们的 PCI 范围。

故障转移与持久性。 单区域故障不得丢失任何 intent。多区域主主架构，使用无冲突状态模型：intent_id 包含区域 + 随机组件；账本按区域存储热写入，全局对账。

退款。 退款 = 一个独立状态机，镜像原始收费的逆过程。基于退款专用键幂等。网络退款调用 → 冲销原始分录的账本分录。部分退款需谨慎计算——追踪 refunded_amount，永不超过 charge.amount。

争议 / 退款纠纷。 客户报告欺诈 → 网络发送争议事件。我们从商家余额中冻结争议金额，启动证据收集流程。完整设计超出范围。

对账。 每日批处理：拉取网络结算文件 → 与我们的收费记录进行关联 → 断言匹配 → 将不匹配项标记供人工审核。这是主要的正确性检查；某天未完成对账即视为故障事件。

热点商户。 黑色星期五：某商户每秒处理 1000 笔请求。其意向单按 merchant_id 分片，导致单个分片承受全部负载。必要时对热点商户进行预分片，拆分为多个子分片。

边缘压力测试。

边缘 故障模式 兜底方案 / 不变量
网络超时 扣款真相未知 保留 unknown；通过对账来确认 network_ref，切勿盲目发起第二次扣款
幂等键已过期 商户在 TTL 后重试 要求客户端在重新确认前，先通过意向单/订单引用进行查询
热点商户 单个分片过载 按以下字段预分片 (merchant_id, sub_shard) 同时保持账本账户余额平衡
多区域漂移 同一意向单在两个区域被确认 使用区域级意向单所有者或对以下字段执行全局 CAS pi:{intent_id} 防止重复提交
冷数据归档读取 查询 5 年前的扣款记录 从归档中获取扣款记录及账本事件；p99 延迟可能达到秒级，不在结账路径上

"员工级别的核心要点：网络可以是模糊的，但我的账本不能；超时会产生未知状态，绝不应发起可能导致重复扣款的第二次尝试。"
