1. 基础知识与解题思路

本题考察点

Stripe 是典型的商户支付网关：API 优先、对开发者友好，抽象了卡网络、银行和监管制度的复杂性。它建立在 设计一个支付系统 之上，但侧重于网关层：API 设计、幂等性、Webhook、SDK 集成以及多支付方式支持。

其不变量如下：

每次对客户收费的 API 调用都会为该支付意图产生唯一的最终结果（succeeded、failed、requires_action）；幂等键使重试安全；Webhook 可靠地通知商户每次状态变更；API 在数百种支付方式和地区间表现一致可预测。

在 ICT5 / L6+ 级别，面试官考察的是你能否设计面向开发者的 API 接口、端到端的幂等性、Webhook 可靠性、支付方式抽象，以及对监管复杂性（PSD2 SCA、KYC、AML）的推理能力。

60 秒面试框架

口述版本：

我会将 Stripe 设计为面向开发者的支付网关。四个核心系统：带幂等性的 API 网关、支付意图状态机、与卡网络和替代支付方式的集成适配器，以及向商户发送的 Webhook。关键设计决策包括端到端幂等性、支付方式抽象、Webhook 可靠性，以及 3DS / SCA 流程。

首先明确需求

功能性需求：

  • 商户通过 REST API 和 SDK 集成。
  • 向客户收费（一次性、订阅）。
  • 多种支付方式：银行卡、ACH、SEPA、钱包（Apple Pay、Google Pay）、本地支付方式。
  • 所有写操作的幂等性。
  • Webhook 向商户通知事件。
  • 退款、撤销、争议处理。
  • 欧洲合规的 3DS / SCA 流程。
  • 客户 / 支付方式令牌化。
  • 多货币。

非功能性需求：

  • API 延迟 p99 低于 500ms。
  • 每个区域高可用。
  • PCI-DSS 合规。
  • 幂等重试。
  • Webhook 在数秒内送达并支持重试。
  • 可审计性。

以下内容超出范围，除非被特别询问：

  • Connect 平台（多租户市场流程）。
  • Issuing（Stripe Issuing）。
  • Treasury / 银行即服务。

Payment Intent 流程

created -> requires_payment_method -> requires_confirmation -> processing -> succeeded
   |                |                          |                |
   v                v                          v                v
 canceled        requires_action          requires_action    failed

3DS / SCA 的子状态： requires_action 表示客户必须完成挑战（3DS 重定向、推送通知等）。

幂等性

每个写操作的 API 调用都接受 Idempotency-Key 头部选择。

  • 每商户作用域。
  • TTL：通常 24 小时。
  • 使用某个 key 的第一次请求存储响应。
  • 使用相同 key 的后续请求返回相同响应（不重复执行）。
  • 使用相同 key 但不同请求体的请求返回错误。

幂等存储是一个快速 KV： (merchant_id, idempotency_key) → response.

对于 provider 调用：Stripe 对 provider 维护自己的幂等性，与商户的 key 相互独立。这意味着即使商户使用相同的 key 重试，Stripe 在 provider 层也不会重复收费。

面试用语：

幂等性是端到端的。商户的幂等键在我们的 API 层去重；我们维护独立的 provider 侧幂等键，使上游网络不会看到重复请求。三层保障：商户→Stripe、Stripe→provider，以及支付意图状态机本身。

令牌化与银行卡 Vault

原始银行卡数据永远不离开 PCI 隔离的 vault。

  • 商户通过 Stripe.js 或移动 SDK 收集银行卡信息；原始卡片数据直接发送至 Stripe（从不经过商户服务器）。
  • Stripe 将银行卡存入 vault；返回一个 payment_method_id.
  • 商户使用 pm_id 进行收费；无法访问原始卡片数据。
  • Vault 位于经过 PCI DSS Level 1 审计的高度隔离基础设施中。

这是大多数商户实现 PCI 范围缩减的关键所在。

3DS / SCA 流程

欧洲 PSD2 要求对大多数交易进行强客户认证（SCA）。

  1. 创建 PaymentIntent 时启用 payment_method 以要求 SCA。
  2. API 返回 requires_action 与 next_action 描述挑战内容。
  3. 前端 SDK 加载挑战（通常是 iframe 或重定向到发卡行）。
  4. 客户完成验证（密码、生物识别、OTP）。
  5. 银行发出认证结果。
  6. 前端确认 PaymentIntent。
  7. PaymentIntent 转换为 processing 再 succeeded 或 failed.

client_secret 由前端用于访问 PaymentIntent，而不暴露 API 密钥。

Provider 集成

每种支付方式都有其自己的 provider 适配器：

  • 银行卡： 通过收单银行接入 Visa、Mastercard；Stripe 充当收单机构。
  • ACH： 美国银行转账。
  • SEPA： 欧洲银行转账。
  • Apple Pay / Google Pay： 钱包令牌，作为银行卡处理。
  • 本地支付方式： iDEAL（荷兰）、Bancontact（比利时）、Alipay、WeChat Pay 等。

适配器可插拔。每个适配器自行处理重试、幂等性和错误映射。

Webhook 投递

Stripe 事件向商户触发 Webhook。可靠性至关重要。

  1. 事件发布到内部事件流。
  2. Webhook 投递服务消费事件；查找订阅该事件类型的商户 Webhook 端点。
  3. 对每个端点，POST 事件负载（使用商户 Webhook 密钥签名）。
  4. 在超时时间内收到 2xx 响应：成功。
  5. 失败时：以指数退避方式重试，最长持续 3 天。
  6. 重试耗尽后：标记为死信；商户可从仪表板重放。

签名防止伪造：

  • Stripe 使用商家的 webhook 密钥计算（时间戳 + 请求体）的 HMAC 签名。
  • 添加 Stripe-Signature 头部选择。
  • 商家在处理前验证签名。

事件流

所有状态变更均会发出事件。事件类型有版本控制（事件对应的 API 版本）。

示例：- payment_intent.succeeded - payment_intent.payment_failed - charge.refunded - customer.created - invoice.paid

欺诈检测（Stripe Radar）

每次扣款时进行内联 ML 评分：

  • 特征：卡片使用模式、行为特征、地理特征、历史记录。
  • 返回风险评分。
  • 高风险：拒绝或要求额外验证。
  • 对商家间的模式进行异步深度分析。

争议与退款纠纷

当客户对扣款提出争议时：

  1. 卡网络将退款纠纷发送给 Stripe。
  2. 向商家触发 Webhook。
  3. 商家提供证据（收据、通信记录、配送证明）。
  4. Stripe 提交至卡网络。
  5. 结果：失败（退款 + 手续费）或胜诉。
  6. 状态在仪表盘中可见。

对账

每日，Stripe 将内部记录与卡网络结算文件进行对账。偏差将触发调查。

多区域策略

  • 每个区域的 API 端点。
  • 每个区域的卡金库（满足数据驻留要求）。
  • 区域化提供商集成（不同区域使用不同的收单机构）。
  • 从商家所在区域投递 Webhook。
  • 跨地区复制用于容灾。

故障模式与缓解措施

故障模式 缓解措施 监控指标
服务商超时。 标记意图为处理中；进行对账。 未知状态计数。
重复 API 调用。 幂等性密钥。 重复重试率。
Webhook 端点不可用。 带退避的重试队列。 按端点失败率。
卡片拒绝。 标准拒绝原因码；向商家展示。 拒绝率。
欺诈误报。 审核队列；商家可按案例覆盖。 假阳性率。
3DS 挑战失败。 向用户展示；重试。 SCA 失败率。
卡金库不可用。 已缓存的 tokenized 支付方式继续可用。 金库可用性。
跨区域复制延迟。 异步；有界。 延迟。
Webhook 载荷过大。 截断或省略；提供 URL 供获取。 超大事件。
对账偏差。 每日对比；告警。 偏差百分比。

运维指标

正确性：

  • 双重扣款事件次数（零）。
  • 事件丢失次数。
  • Webhook 投递成功率。
  • 对账偏差。

容量：

  • 按端点统计 API QPS。
  • 每次集成的提供商延迟。
  • Webhook 投递队列深度。

业务层面：

  • 处理金额。
  • 审批率。
  • 争议率。
  • 商家留存率。

需要指出的权衡

  • 同步提供商调用增加延迟，但为商家提供即时结果。
  • 异步确认简化了 API，但增加了 Webhook 的复杂性。
  • 激进的欺诈评分降低损失，但会增加误报。
  • 严格的 3DS 提升安全性，但增加了操作摩擦。
  • Token 金库集中了 PCI 合规范围，但属于关键基础设施。
  • 多区域数据驻留要求与跨区域故障转移便捷性之间的权衡。

常见陷阱与更优做法

陷阱 更优做法
所有流程使用单一端点。 PaymentIntent 作为抽象层；3DS 使用子状态。
同步 Webhook。 带重试的异步处理。
跨商家使用相同的幂等性密钥。 每商户作用域。
API 日志中包含卡片数据。 PCI 金库；仅使用 token。
硬编码提供商。 按支付方式使用可插拔适配器。
反向时序计费。 可重放的正向时序事件流。

  • 以 PaymentIntent 作为核心抽象层。
  • 端到端幂等性。
  • 可插拔提供商适配器。
  • PCI 隔离的卡金库。
  • 包含 client_secret 的 3DS / SCA 流程。

面试用语

Stripe 是一个面向开发者的支付网关。PaymentIntent 是核心抽象层——一个状态机，涵盖从创建、SCA 挑战到成功的完整生命周期。幂等性贯穿端到端：商家 API、提供商调用、状态机。Webhook 至少投递一次，支持重试和 HMAC 签名。卡金库经过 PCI 隔离。支持数据驻留的多区域架构。

2. 模拟面试

面试官： 设计 Stripe。

候选人： Stripe 是一个围绕 PaymentIntent 抽象层构建的支付网关——一个从创建、确认、SCA 挑战、处理到成功的单次扣款状态机。四个核心系统：带幂等性的面向开发者 API、PaymentIntent 状态机、抽象卡网络和替代支付方式的提供商适配器层，以及向商家投递 Webhook。不变量是每个 PaymentIntent 有且仅有一个终态，且全程支持幂等重试。

面试官： 带我走一遍扣款流程。

候选人： 商家 POST 请求 /v1/payment_intents 与 Idempotency-Key 以及金额、货币、支付方式。幂等性检查：若已见过，返回缓存响应。新意图创建于 requires_payment_method 或直接创建于 processing 如果 confirm=true。使用支付方式调用提供商适配器（例如，通过 Visa/Mastercard 收单机构处理卡片）。提供商返回成功、失败或 3DS 挑战。成功时：意图转换为 succeeded；发出事件；触发 Webhook。3DS 时：意图移至 requires_action；前端完成挑战；商家调用 confirm；流程继续。

面试官： 幂等性——解释三层机制。

候选人： 第一层：商家的 API 调用使用 Idempotency-Key header。Stripe 的幂等性存储按 (merchant_id, key)缓存响应。重试返回相同响应而不重新执行。第二层：Stripe 调用提供商（卡网络）时使用 Stripe 自己的幂等性机制——与商家的不同。这意味着即使商家使用相同密钥重试，提供商也只看到一次扣款。第三层：PaymentIntent 状态机——转换是对 (intent_id, from_state)的 CAS 操作。重试的转换会因状态已推进而失败。

面试官： Webhook。

候选人： 每次状态变更都向内部流发出事件。Webhook 投递服务消费事件，查找商家订阅的端点，POST 带 HMAC 签名 header 的载荷。在超时内收到 2xx：成功。失败时：指数退避重试最多 3 天。耗尽后：进入死信队列；商家可从仪表盘重放。签名使用商家的 Webhook 密钥；商家在处理前验证。

面试官： 3DS / SCA？

候选人： PaymentIntent 转换为 requires_action 与 next_action 描述挑战。API 返回 client_secret 前端使用它来访问意图。前端 SDK 展示挑战（嵌入发卡行的 iframe）。客户完成（密码、生物特征、OTP）。银行发出认证结果。前端调用 confirm；意图转换为 processing 再 succeeded 或 failed.

面试官： 卡金库。

候选人： 原始卡片数据永不离开 PCI 隔离的金库。商家通过 Stripe.js 或移动 SDK 集成；原始卡片数据直接发至 Stripe（绕过商家服务器，缩小 PCI 合规范围）。金库进行 tokenize；返回 payment_method_id。商家使用 pm_id 进行扣款。金库位于经过严格审计的基础设施中（PCI DSS Level 1）。

面试官： 提供商抽象层。

候选人： 按支付方式使用可插拔适配器。卡片通过 Visa/Mastercard 收单机构。ACH 用于美国银行。SEPA 用于欧元区银行。钱包（Apple Pay、Google Pay）作为卡片处理。本地支付方式（iDEAL、Alipay 等）各有独立适配器。每个适配器处理自己的重试、幂等性映射和错误码转换。PaymentIntent 状态机与提供商无关。

面试官： 提供商超时。

候选人： 意图保持在 processing 与 unknown_state 指示符。对账 worker 使用相同幂等性密钥向提供商查询。确认成功时：转换为 succeeded。确认失败时：转换为 failed。切勿盲目重试——那可能导致双重扣款。

面试官： 争议 / 退款纠纷。

候选人： 卡网络向 Stripe 发送退款纠纷。Webhook 向商家触发 charge.dispute.created 。商家通过 API 提供证据（收据、通信记录、配送证明）。Stripe 提交至网络。结果：失败（退款 + 手续费）或胜诉。Webhook 触发最终结果。商家在仪表盘中查看争议生命周期。

面试官： 欺诈检测（Radar）？

候选人： 每次扣款进行内联 ML 评分。特征：卡片使用模式、行为信号、地理特征、账户历史。返回风险评分。高风险：拒绝或要求额外验证。商家可通过仪表盘设置自定义规则。跨商家异步深度分析，用于发现新兴欺诈模式。

面试官： 多区域？

候选人： 每个区域配置 API 端点。每个区域配置卡金库以满足数据驻留合规要求（欧盟数据留在欧盟）。区域化提供商集成（不同地区使用不同收单机构）。从商家所在区域投递 Webhook。跨区域副本用于灾难恢复。数据驻留是严格约束，而非仅作性能优化。

面试官： 如何对账？

候选人： 每日。Stripe 将内部支付记录与卡网络结算文件进行对账。不匹配进入审核。总量偏差触发调查。冷路径（对账）是计费和报告的权威来源；热路径（实时意图状态）用于操作。

面试官： 总结。

候选人： 以 PaymentIntent 状态机作为核心抽象层；可插拔提供商适配器；三层幂等性；PCI 隔离卡金库；通过 client_secret + next_action 实现 3DS；带 HMAC 签名和重试的至少一次 Webhook 投递；每日对账；支持数据驻留的多区域架构。最难的正确性问题是在重试和超时情况下每个意图有且仅有一个终态；端到端幂等性是答案。

3. 练习备注

五分钟版本

  1. 澄清：API、幂等性、支付方式、Webhook、SCA。
  2. 不变量：每个意图一个终态；幂等重试。
  3. PaymentIntent 状态机。
  4. 三层幂等性。
  5. PCI 隔离的卡金库。
  6. 可插拔提供商适配器。
  7. 通过 client_secret + next_action 实现 3DS。
  8. 至少一次带签名的 Webhook，支持重试。

一分钟结尾陈述

Stripe 是面向开发者的支付网关，围绕 PaymentIntent 状态机构建。三层幂等性（API、提供商、状态）。PCI 隔离的卡金库。按支付方式使用可插拔提供商适配器。通过 client_secret 和 next_action 实现 3DS / SCA。带 HMAC 签名和指数退避重试的至少一次 Webhook 投递。每日对账。支持数据驻留的多区域架构。最难的正确性问题是在重试和超时情况下每个意图有且仅有一个终态。

最佳面试用语

  • "PaymentIntent 是核心抽象层——一个用于单次扣款的状态机。"
  • "幂等性贯穿端到端：商家、提供商、状态机。"
  • "卡金库经过 PCI 隔离；商家获得 token，而非原始卡片数据。"
  • "3DS 流程：requires_action + client_secret + next_action。"
  • "Webhook 至少投递一次，带 HMAC 签名和重试。"
