1. 基础知识与解题思路

本题考察点

Google Search 是网络规模信息检索的经典问题。难点在于对整个公共网络建立索引，在一秒内响应任意查询，按相关性和权威度对结果排名，并持续更新以反映变化中的网络。

其不变量如下：

所有已索引页面均可通过适当查询被发现；查询结果由稳定的评分函数确定性排名；索引在有界延迟内反映语料库；查询延迟在 p99 下低于数百毫秒；系统在索引部分故障时仍可返回尽力而为的结果。

在 ICT5 / L6+ 级别，面试官考察你能否设计索引流水线（爬取 → 解析 → 建索引）、为查询服务而分片的倒排索引、带 PageRank 风格信号的排名流水线、查询规划与结果合并，以及分析新鲜度与成本的权衡。

60 秒面试框架

口述版本：

我会将 Google Search 设计为从爬虫到索引再到查询服务的流水线。爬虫抓取网络；索引器按词条分片构建倒排索引；查询服务器对分片进行 scatter-gather，排名后返回结果。排名融合静态信号（PageRank、内容质量）和查询时信号（相关性、新鲜度、个性化）。关键设计决策包括：索引分片策略、scatter-gather 查询延迟、排名架构、新鲜度流水线和服务容量。

首先明确需求

功能性需求：

  • 用户提交查询，获得相关网页的排名列表。
  • 支持布尔运算符、短语、过滤条件（site:、intext:）。
  • 基于用户历史和位置的个性化。
  • 用户输入时自动建议。
  • 知识面板和直接回答（摘要片段）。
  • 图片和视频搜索（提及即可；独立索引）。

非功能性需求：

  • 查询 p99 低于 500 ms。
  • 索引覆盖数百亿网页。
  • 新鲜度：新闻几分钟内，静态页面数小时至数天。
  • 查询服务高可用。
  • 多地区；查询由本地地区服务。

以下内容超出范围，除非被特别询问：

  • 爬虫细节（参见 Web Crawler 设计）。
  • 广告服务和竞价。
  • 垃圾信息检测细节。

规模假设

领域 假设
已索引页面 超过 1000 亿网页；长尾数量达万亿级。
查询/秒（Queries/sec） 全球超过 100k QPS。
索引大小 倒排索引需数拍字节（petabytes）。
查询延迟 p99 低于 500 ms。
新鲜度 新闻几分钟内；其余数小时至数天。
更新速率 每小时数百万新增/变更页面。
不重复词条 跨语言达数十亿。

瓶颈在于跨多个索引分片的 scatter-gather 延迟，以及查询时重新计算排名特征的代价。

核心组件

组件 职责 关键细节
爬虫 抓取并刷新网页。 参见 Web Crawler 设计。
页面存储 原始 HTML；规范 URL → 版本。 PB 级。
解析器 分词、语言检测、提取结构化数据。 流式处理。
Indexer 构建倒排索引。 批量 + 流式处理。

知识图谱 基于实体的答案；更丰富的格式。 独立检索。

倒排索引分片

一个分片拥有一部分词项的倒排列表（按词项哈希分片），或拥有一部分文档（按 doc_id 分片）。后者在 Google 规模下更为常见。

API

GET /search?q=
GET /complete?q=                 # autosuggest
GET /image-search?q=             # separate index
GET /news-search?q=              # freshness-focused

索引构建流水线

  1. 爬虫将原始 HTML 存储到页面存储中。
  2. 解析器提取文本、链接和结构化数据。
  3. 分词器进行归一化处理（小写、词干提取、特定语言处理）。
  4. 倒排列表构建器：对每个 (词项, doc_id) 对，追加包含频率和位置的条目。
  5. 索引构建器按分片将倒排索引写入排序后的段中。
  6. 周期性合并（LSM 风格）保持索引高效。

离线计算的静态信号：

  • 基于链接图的 PageRank。
  • 内容质量分数。
  • 垃圾/惩罚分数。

每个文档都有一个 static_rank 预计算值；用作与查询无关的先验。

新鲜度流水线：

  • 新闻和高优先级页面在数分钟内完成索引。
  • 一个独立的、较小的"实时索引"服务于最新内容。
  • 查询将主索引与实时索引的结果合并。

面试用语：

索引分为两层：一个批量构建的巨型主索引覆盖大部分网页，加上一个小型实时索引捕获最新内容。查询将两者的结果合并。

索引分片

两种策略：

基于文档的分片：

  • 每个分片拥有一部分文档的分区。
  • 对于一次查询，所有分片针对各自的文档处理查询并返回 top-K 结果。
  • 聚合器进行合并。
  • 优点：易于添加/移除分片；单分片工作量有界。
  • 缺点：每次查询都会命中所有分片。

基于词项的分片：

  • 每个分片拥有一部分词项的倒排列表分区。
  • 对于一次查询，只有持有查询词项的分片会被访问。
  • 优点：对于窄查询，扇出更小。
  • 缺点：热门词项会成为热点分片；跨分片的交集运算代价高昂。

实际上，在 Google 规模下，基于文档的分片更为常见，因为每次查询承受大致均等的负载。基于词项的分片在某些特定场景下使用。

查询服务

对于"best italian restaurants in NYC"这样的查询：

  1. 前端接收查询。
  2. 查询重写器进行扩展：同义词、拼写纠错、查询理解。
  3. 将查询分散到所有索引分片（或相关子集）。
  4. 每个分片返回带有打分数据的 top-N 候选 doc_ids。
  5. 聚合器合并候选结果并去重。
  6. 排序阶段：综合静态排名 + 查询时特征（BM25、语义相似度、新鲜度、个性化）→ 最终得分。
  7. Top-K 结果：从元数据存储中获取标题和摘要。
  8. 摘要生成器创建上下文相关的摘要。
  9. 知识图谱：检查查询是否有实体答案；若有则添加知识面板。
  10. 返回结果页面。

全程约 500 ms 内完成。

延迟预算：

  • 网络往返约 10 ms。
  • 索引分片查询约 100-200 ms。
  • 聚合约 50 ms。
  • 排序阶段约 50-100 ms。
  • 摘要 + 元数据获取约 50 ms。
  • 总计约 300-400 ms。

Scatter-Gather 尾部延迟

数千个分片时，最慢的分片决定了查询延迟。

缓解措施：

  • 对冲请求：向同一分片的两个副本发送请求，取最先响应的结果。
  • 尾部容忍：例如在 99% 的分片返回后即响应，丢弃最慢的 1%。
  • 单分片超时设置。
  • 健康感知的路由。

面试用语：

尾部延迟是 scatter-gather 的主要成本。对冲请求和部分响应容忍机制可将 p99 保持在合理范围内。

排序

多阶段：

  1. 轻量级的单分片打分： 每个分片使用低成本特征（BM25、静态排名）对候选文档打分，返回每分片 top-K 结果。
  2. 聚合打分： 聚合器进行合并，应用更多特征（实体匹配、新鲜度）。
  3. ML 排序器： 对 top 结果使用深度模型（BERT 风格或基于 Transformer）进行最终排序，以获得最高保真度的排名。

信号：

  • PageRank、权威性、内容质量。
  • 查询-文档相关性（BM25、语义相似度）。
  • 时效性查询的新鲜度。
  • 个性化（位置、历史点击）。
  • 点击率先验。
  • 移动友好性、页面速度。

知识图谱

对于实体查询（如"Barack Obama height"），系统从结构化知识图谱中检索，而非仅进行文本匹配。

  • 查询中的实体识别。
  • 知识图谱查询。
  • 渲染为知识面板 + 直接答案摘要。

自动补全

  • 热门查询的 Trie 结构，带频率统计。
  • 按前缀分桶。
  • 个性化重排序。
  • 延迟低于 50ms。

扩展与分区

数据 分区策略 原因
倒排索引 基于文档的分片（数千个分片） 查询负载均衡。
文档元数据 按 doc_id 哈希 快速查找。
页面存储 按 URL 哈希 每个 URL 的版本信息。
知识图谱 基于实体 按实体 ID 查询。
排序特征 按文档或按查询 尽可能预计算。
实时索引 按时间分桶 数据新鲜度。

一致性模型

层级 一致性 原因
倒排索引 最终一致（每小时批处理 + 实时层） 网页不是实时的。
文档元数据 最终一致 UI 提示。
知识图谱 实体事实强 直接答案必须正确。
排序 当前索引上的稳定函数 确定性。

多区域策略

  • 索引复制到每个区域以实现快速本地服务。
  • 查询路由到最近的区域。
  • 更新从主索引流向所有区域。
  • 新鲜度滞后：主节点与副本之间有数分钟延迟。

故障模式与缓解措施

故障模式 缓解措施 监控指标
慢速分片。 对冲请求；部分响应。 单分片尾部延迟。
故障分片。 副本回退；在缺少覆盖的情况下降级服务。 分片健康状况。
索引服务过载。 限流；速率限制；优先服务已登录用户。 QPS 与容量的权衡。
排序模型出错。 金丝雀发布；自动回滚。 点击率（Click-through rate）。
垃圾内容注入。 垃圾内容检测层；人工审核。 异常 URL 出现。
突发新闻的索引延迟。 实时索引层；预先优先化的爬取。 从发布到被索引的时间。
个性化 bug 导致私人数据泄露。 选择性加入个性化；清晰隔离。 隐私审计。
知识图谱事实错误。 人工覆盖；事实溯源。 用户举报率。
跨区域复制延迟。 异步；有界；告警。 复制延迟。

运维指标

正确性：

  • 结果质量（离线评估、点击信号）。
  • 结果中的垃圾内容。
  • 知识图谱准确性。

容量：

  • 按区域统计的查询 QPS。
  • 索引分片的 QPS 和延迟。
  • 尾部延迟 p99/p999。

业务层面：

  • 点击率（Click-through rate）。
  • Top-N 点击分布。
  • 查询改写率。

需要指出的权衡

  • 更大的索引覆盖更多页面，但会增加查询延迟。
  • 频繁重新索引提升新鲜度，但消耗更多计算资源。
  • 更多个性化提升相关性，但引发隐私问题。
  • 更深的排序模型提升质量，但消耗计算资源。
  • 对冲请求降低尾部延迟，但成本翻倍。
  • 两层索引（主索引 + 实时索引）提升新鲜度，但增加复杂性。

常见陷阱与更优做法

陷阱 更优做法
一个大型的单体索引。 分片倒排索引；基于文档的分区。
每次查询全量排序。 多阶段排序；先低成本后高成本。
忽略尾部延迟。 对冲请求；部分响应容忍。
将新鲜内容与主索引合并为一个索引。 两层：小型实时索引 + 大型主索引。
同步排序模型。 ML 模型前的轻量级处理阶段。
忽略知识图谱。 实体识别 + 结构化查询。

面试官可能深入探讨的方向

深入方向 优秀回答要点
索引分片 基于文档 vs 基于词项；权衡分析。
尾部延迟 对冲；部分响应；副本。
排序阶段 单分片低成本 → 聚合器 → ML。
新鲜度 两层索引；实时层从优先爬取中获取数据。
知识图谱 实体识别；结构化检索；溯源。
自动补全 Trie；个性化；延迟低于 50ms。

Staff 级别回答检查清单

  • 流水线：爬取 → 解析 → 索引 → 服务。
  • 基于文档的分片分区。
  • 带尾部延迟缓解的 scatter-gather。
  • 多阶段排名。
  • 用于新鲜度的两层索引。
  • 用于实体答案的知识图谱。
  • 将自动补全作为独立的快速路径。
  • 多区域副本。

面试用语

Google Search 是基于倒排索引的 scatter-gather 查询，并叠加了多阶段排序。索引按文档分片，并通过副本缓解尾部延迟；排序从单分片的低成本打分逐步升级到结果集顶部的高成本 ML 模型。新鲜度来源于与主索引合并的实时层。最难的性能问题是 scatter-gather 中的尾部延迟；最难的正确性问题是排序质量。

2. 模拟面试

面试官： 设计 Google Search。

候选人： 流水线：爬虫抓取网页，解析器分词，索引器构建分片倒排索引，查询服务器进行 scatter-gather，排序器打分，返回结果。不变式是在覆盖数百亿页面的索引上实现亚秒级查询延迟，同时具备相关性排序和有界的新鲜度。

面试官： 索引是如何分片的？

候选人： 基于文档。每个分片拥有一部分文档的分区。对于一次查询，所有分片针对各自的文档处理查询并返回 top-K 候选结果，聚合器进行合并。另一种方案是基于词项的——每个分片拥有部分词项的倒排列表——但在 Google 规模下，基于文档的方式更为常见，因为每次查询承受大致均等的负载。

面试官： 请介绍一次查询的完整流程。

候选人： 用户输入"best italian restaurants nyc"。前端进行重写：拼写纠错、同义词扩展、意图识别。将查询分散到数千个索引分片。每个分片扫描"italian"、"restaurants"、"nyc"的倒排列表，进行交集运算，计算轻量级得分（BM25 + 静态排名），返回 top-K（例如 100）个候选结果。聚合器跨分片合并为 top-1000。多阶段排序：中间层模型使用更多特征对 top-1000 打分；最终 ML 模型使用重量级特征（BERT、个性化、新鲜度）对 top-50 打分。返回 top-10 结果及摘要。若检测到实体，知识图谱添加知识面板。

面试官： 尾部延迟——数千个分片，一个慢分片就会拖垮整个查询。

候选人： 对冲请求：每个分片有多个副本；同时向两个副本发送请求，取最先响应的结果。部分响应容忍：一旦 99% 的分片响应，聚合器即返回结果，丢弃慢速的 1%，质量损失有界。单分片超时设置。健康感知路由将慢速副本降级。容量余量保持在 60-70% 利用率左右，防止慢节点引发级联问题。

面试官： 排序？

候选人： 多阶段。单分片使用 BM25 和静态信号（PageRank、权威性）进行低成本打分，每分片返回 top-K 候选结果。聚合器合并后应用更多特征（实体匹配、新鲜度）。顶部结果通过深度 ML 模型（基于 Transformer）进行最高保真度排序。两阶段结构控制计算量：低成本特征剪枝到可管理的集合；高成本特征仅对顶部结果进行评估。

面试官： PageRank？

候选人： 静态的、与查询无关的排名，基于链接图计算。拥有大量高权威性入站链接的页面获得更高的 PageRank。通过周期性批处理任务在网页链接图上离线计算。每个文档存储一个特征值，用于排序。现代 Google 拥有远超 PageRank 的众多信号，但原理相同：预计算的权威性。

面试官： 新鲜度——突发新闻报道。

候选人： 两层索引。主索引以批量方式构建（每小时或每天）；覆盖大部分网络内容。实时索引规模小得多，由新闻网站和近期更新的优先爬取驱动，可在数分钟内完成索引。查询时合并两者的结果。排名器会对时间敏感的查询提升新鲜内容的权重。编辑信号可以推广特定报道。

面试官： 知识图谱？

候选人： 对实体、属性和关系的独立结构化存储。查询中的实体识别（NER）可检测例如"Barack Obama"。在知识图谱中查找以获取属性。结果页面显示包含结构化信息的知识面板，以及针对简单事实性查询的直接答案摘要。事实来源——每条事实由哪个来源支撑——对于处理错误至关重要。

面试官： 自动建议？

候选人： 存储热门查询及其频率的 Trie 树，按前缀分片。随着用户输入，服务器返回排名靠前的补全结果。个性化层根据用户历史记录和位置重新排名。延迟低于 50ms。常见前缀在客户端缓存。

面试官： 多区域？

候选人： 索引复制到每个地区。查询路由到最近的地区。索引更新从主索引管道流向所有地区；复制延迟为数分钟。知识图谱在全球范围内复制。每个地区的容量根据本地 QPS 进行配置。

面试官： 垃圾信息？

候选人： 垃圾信息检测在解析时和索引时运行。信号包括：隐藏内容、入口页面、链接垃圾、内容质量。惩罚分数会降低页面排名。被明确认定为垃圾的内容将从索引中排除。边缘情况进行人工审核；站长工具允许申诉。

面试官： 个性化？

候选人： 选择性启用。用户的近期搜索、位置和语言偏好会影响排名。个性化特征在线从用户画像中计算，在最终排名阶段应用。有边界限制——不会过重以至于排名变得不可预测。隐私注意事项：明确的用户控制；匿名化信号。

面试官： 总结。

候选人： 从爬虫到倒排索引再到查询服务的完整管道。基于文档的分片分区。带有对冲请求的分散-聚合模式，以及针对尾部延迟的部分响应容忍机制。多阶段排名，从每个分片的廉价评分到顶部昂贵的机器学习模型。用于新鲜度的两层索引。用于实体答案的知识图谱。自动建议走独立路径。多地区副本。最难的性能问题是尾部延迟；最难的正确性问题是排名质量和垃圾信息处理。

3. 练习备注

五分钟版本

  1. 明确需求：查询延迟、新鲜度、规模。
  2. 不变量：数百亿页面上的 p99 低于一秒。
  3. 流水线：爬取 → 解析 → 索引 → 服务。
  4. 基于文档的分片分区。
  5. 带有对冲请求的分散-聚合。
  6. 多阶段排名（廉价 → 昂贵）。
  7. 用于新鲜度的两层索引。
  8. 用于实体答案的知识图谱。

一分钟结尾陈述

Google 搜索是对文档分片倒排索引的分散-聚合，采用多阶段排名。廉价的每分片评分对候选项进行剪枝；昂贵的机器学习模型对顶部候选项排名。对冲请求和部分响应容忍机制保持尾部延迟可控。两层索引合并实时层和主层以保证新鲜度。知识图谱驱动实体答案。自动建议是独立的快速路径。主要性能问题是分散-聚合尾部延迟；主要正确性问题是排名质量。

最佳面试用语

  • "对文档分片倒排索引的分散-聚合。"
  • "多阶段排名：廉价特征剪枝，昂贵特征排名。"
  • "对冲请求和部分响应容忍机制是我们处理分散-聚合尾部延迟的方式。"
  • "两层索引：主批量层 + 实时层以保证新鲜度。"
  • "知识图谱驱动直接实体答案；文本检索处理其余部分。"
