Strava — Activity Tracking & Segments (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Medium · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建 Strava。用户开始一次锻炼，手机全程流式传输 GPS 点，结束时系统保存包含地图、分段配速和赛段排行榜的活动记录。1 亿用户，每天约 1000 万个活动，每个活动约 1000 个 GPS 点。

1. 需求（约 5 分钟）

功能需求 - 记录一次锻炼：从手机流式传输 GPS 点，保存为 Activity。- 查看你的活动——地图、配速、距离、海拔、分段。- 将活动路径与预定义"赛段"（一段道路/小径）进行匹配，并更新赛段排行榜。

我会推迟处理社交动态、训练计划和 Strava Premium 分析功能。

非功能需求 - GPS 摄入在下午高峰期量很大，但对单个用户而言是突发性的；持久性比延迟更重要。- 用户点击"完成"后，活动保存的端到端 p95 < 5 秒。- 赛段排行榜更新应在活动上传后一分钟内出现。- 排行榜的最终一致性是可以接受的。

容量估算 - 1000 万活动/天 × 1000 个点 × 50 字节 = 500 GB 原始 GPS/天，全球高峰下午约 6 GB/s。- 压缩时序存储可将其降低约 10 倍。- 压缩后的原始 GPS 仍是长期存储的主导：约 50 GB/天 × 3 个副本 ≈ 150 GB/天，或约 55 TB/年（生命周期迁移至更冷存储之前）。- 活动元数据量级相当但查询频繁：1000 万/天 × 约 5 KB = 50 GB/天（不含副本）。赛段成绩较小：若每个活动匹配 5 个赛段，5000 万条成绩/天 × 约 120 B × 3 ≈ 18 GB/天。

2. 核心实体（约 2 分钟）

  • User — user_id.
  • Activity — activity_id, user_id, type, started_at, duration, distance, polyline（压缩路径）, stats。
  • GPSPoint — activity_id, ts, lat, lng, elev, hr?。作为时序数据存储在活动 blob 中。
  • Segment — segment_id, geometry（polyline），creator。
  • SegmentEffort — segment_id, activity_id, user_id, time_ms。驱动排行榜。

3. API（约 5 分钟）

POST  /v1/activities                          → { activity_id, upload_url }    // server allocates id
POST  /v1/activities/{id}/points              { points[] }                     // batched, every ~30s
POST  /v1/activities/{id}/finish              → final activity object
GET   /v1/activities/{id}
GET   /v1/segments/{id}/leaderboard?period=all|year

从 token 获取用户。点上传在 (activity_id, ts).

4. 数据流（约 5 分钟）

手机以 1 Hz 采集 GPS → 每 30 秒批量上传 → POST 至摄入服务 → 摄入服务写入每个活动的 blob（或追加至时序存储）→ 完成时，运行后台作业：计算统计数据、与赛段匹配、写入 Activity 行和 SegmentEfforts → 推送排行榜更新。

5. 高层设计（约 10–15 分钟）

  1. Ingest service。 接收批量点上传。以仅追加方式写入对象存储中的每个活动对象（每个活动一个 blob）或写入以 activity_id 为键的时序数据库（InfluxDB / TimescaleDB）。
  2. Activity service。 在 MySQL 中管理活动元数据行（按 user_id 分片）。在 finish时，启动一个处理作业。
  3. Processing pipeline。 Stream/job worker：- 读取活动的所有点。- 计算距离、时长、配速、海拔、分段配速。- 构建压缩 polyline（Google polyline 编码）并存储在活动行上。- 运行赛段匹配。- 将活动对象写回目录。
  4. Segment matching。 对每个赛段，在 S2 单元中对其边界框进行地理索引。对于每个活动，找出边界框与活动边界框相交的赛段。然后进行精确匹配：活动的 polyline 是否经过赛段起点、沿路径行进并到达终点？
  5. 排行榜服务。 Per-segment Redis sorted set： score = effort_time_ms （升序），每个用户一条记录（最佳时间）。裁剪为前 N 名加请求者的邻近区域。
  6. Storage tiers。 原始 GPS 存储在对象存储中（适合冷存储）。活动元数据 + polylines 存储在 MySQL/Cassandra 中。排行榜存储在 Redis 中。

端到端叙述：Alice 在上午 8 点开始骑行 → 手机 POST /activities → 服务器返回 activity_id A123 → 骑行过程中，每 30 秒批量上传 30 个点 → 摄入服务追加至 gs://strava/activity/A123/points.pb → 9:30 她点击完成 → 服务器将处理作业入队 → worker 读取全部 5400 个点，计算出 28.4 km/1h28m，编码 polyline，发现她的骑行路过了赛段 S55 和 S77，计算成绩时间，写入排行榜 → 推送通知"您在 Tunitas Climb 排名第 3。"

6. 深度探讨（约 10 分钟）

具体存储与分片键。

存储 键模式 数据结构 / 条目 TTL / 保留期 分片键
Raw points activities/{user_hash}/{activity_id}/points.pb protobuf/列式 blob，1000 个点压缩后约 10 KB 永久保留；30–90 天后转冷 user_hash + activity_id
Activity metadata activity:{user_id}:{activity_id} 含统计数据/polyline/blob 指针的行，2–5 KB ， user_id
Segment geo-index s2:{level}:{cell_id} list/set of segment_ids，每个繁忙单元 KB–MB no TTL cell_id
Segment effort effort:{segment_id}:{user_id}:{activity_id} 不可变成绩行，约 120 B ， segment_id
排行榜 leader:{segment_id}:{period} Redis ZSET score=time_ms，member=user_id，约 30–60 B/条目 永久热；可重建 segment_id
Upload dedupe point_seen:{activity_id}:{ts} 短暂的已见标记或块校验和，约 32 B active + 24 h activity_id

对象存储保存 GPS 内容；MySQL/Cassandra 存储可搜索的元数据和指针。Redis 排行榜是可从重建的服务索引。 SegmentEffort.

各存储容量估算。 原始点 blob 主导长期保留： 10M activities/day × 5–10 KB compressed 主存为 50–100 GB/天，3 个副本下约 55–110 TB/年（冷分层之前）。活动元数据对用户历史查询更热： 10M/day × 5 KB × 3 约 150 GB/天。赛段成绩较小但写入频繁： 50M efforts/day × 120 B × 3 约 18 GB/天。S2 赛段索引规模适中：即使 10M segments × 3 cell refs × 100 B × 3 也不到 10 GB。排行榜是上限服务索引；若热门赛段以约 60 B/成员存储前 1 万名加邻近区域，则每个热门赛段约 600 KB，无上限的真实数据仍保留在 SegmentEffort.

Point upload and finish path。 1. 手机创建活动，接收 activity_id，并上传带有序列号/校验和的数据块。2. 摄入服务验证所有权，按 (activity_id, chunk_seq) 或 (activity_id, ts)去重，并追加至原始点 blob/暂存对象。3. 在 finish时，Activity service 将活动标记为 processing 并将包含 blob 指针的任务加入队列。4. Worker 一次性读取原始点位，计算统计数据/折线/分段，写入 activity:{user_id}:{activity_id}，然后查询 S2 单元以获取候选路段。5. 精确匹配器写入 SegmentEffort 行并更新 leader:{segment_id}:{period} ，按用户尽力而为。6. 若排行榜更新失败，努力行保持持久化，修复任务会重建 Redis ZSET。

路段匹配的规模化方案。 朴素方法：对每个活动检查每个路段。每天 1000 万个活动 × 1000 万个路段 = 10¹⁴ — 不可能实现。应对策略：- 按 S2 单元（级别约 12，约 1 km²）对路段建立地理索引。- 对每个活动，计算折线经过的 S2 单元集合。- 候选路段 = 这些单元中路段的并集。- 仅对候选路段（通常只有几百个）运行精确折线匹配。

精确折线匹配。 活动必须从路段起点附近（约 10 m 内）进入，沿路径行进（Fréchet 距离低于阈值），并从终点附近退出。实现方式：遍历活动折线，寻找端点与路段对齐的窗口。

时序存储的选择。 每个活动的原始点位是仅追加的，完成后不可变，作为整体查询。对象存储（S3）在成本上胜出；将点位序列化为 protobuf 或列式（Parquet）格式以减小体积。避免将其存入行式存储。

为什么对原始点位使用 S3/对象存储而非 InfluxDB/Timescale：常见读取是“加载一个已完成的活动”，而非“按时间聚合所有用户的所有点位”。时序数据库适用于实时遥测仪表盘，但对于不可变的按活动 blob，对象存储更便宜且更简单。S2 在此优于普通 geohash，因为单元覆盖能更可预测地处理路径和边界。

对于活动元数据，若需要强用户历史分页和简单事务，MySQL 是合适选择；若全球写入量和多区域写入占主导，Cassandra 是合适选择。在面试基准中我会选择 MySQL/分片 SQL，因为高吞吐量 GPS 内容已在行式存储之外，而用户活动列表希望按 (user_id, started_at).

折线编码。 使用精度 5 的 Google 折线编码，每个点约 10 字节；1000 个点的骑行 → 10 KB，而非原始的 50 KB。

实时追踪（可选）。 “信标”功能：朋友可以实时观看骑行。为每个用户添加一个包含最新点位的 Redis cell，以及一个 SSE 通道，用于在 30 秒批次到达时进行扇出。

排行榜基数。 热门路段可能有数十万条努力记录。每次上传使用 ZADD 的 Redis 有序集合即可——每个用户只保留最佳成绩（ZADD GT）。对于“今年前 N 名”，可为每个时间窗口维护一个 Redis ZSET。

双索引框架。 同一次骑行按活动存储用于所有权（activity:{user_id}:{activity_id} + 原始 blob），按路段存储用于竞争（effort:{segment_id}:{user_id}:{activity_id}）。这两者并不矛盾：活动 blob 是内容，路段努力是从中派生的二级索引。若发现匹配器 bug，可从不可变的活动内容重建路段索引。

排行榜耗尽与冷排名。 Redis 只需保存热窗口：前 N 名、好友和查看者的邻近范围。当用户翻页超出该范围时，游标将最后的分数和 user_id 带入持久化 SegmentEffort 表中。深页的延迟预算可从 <100 ms 延伸至约 500 ms，因为用户此时已在探索长尾排名，而非加载主活动页面。

作弊检测。 GPS 欺骗、自行车活动中使用机动车。启发式检测：不合理的配速、急剧的海拔跳变、平均心率与运动量不匹配。被标记的努力从排行榜中隐藏但仍被存储。

手机离线/网络不稳定。 手机在本地缓冲点位；上传以指数退避重试。摄取服务在 (activity_id, ts) 上是幂等的，因此重复上传是安全的。

优雅降级。

情形 模式 数据源 延迟预算 回退
手机离线 本地缓冲 手机存储 直到重新连接 稍后以相同的 activity_id 上传块
