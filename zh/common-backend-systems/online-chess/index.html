Online Chess — Chess.com / Lichess Class (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Medium · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建一个在线国际象棋平台。玩家按技术评级进行匹配对局，使用计时器进行游戏，在约 200 ms 内看到对手的棋步，游戏结束后更新 Elo。每月活跃用户 1000 万，峰值时约 50 万并发，峰值约每分钟 5 万局。

1. 需求（约 5 分钟）

功能需求 - 玩家进入匹配；与评级接近的人配对。- 双方玩家实时行棋，每方有独立计时器。- 游戏结束（将死、认输、超时、和棋）时更新评级。

我会推迟锦标赛、深度反作弊（引擎检测）、教程和直播观看功能。

非功能需求 - 走棋传播 p95 < 200 ms。- 计时器公平性——双方看到相同的已用时间；没有人因网络延迟而超时落败。- 游戏状态强一致性——双方对棋局一致。- 高可用性——锦标赛对局对玩家来说是真实的。

容量估算 - 50 万并发连接。- 每分钟 5 万局 × 每局 30 步 = 每分钟 150 万步 ≈ 每秒 2.5 万步。- 走棋写入量适中：每秒 2.5 万步 × 约 80 B/步 × 3 副本 ≈ 峰值 6 MB/秒。连接状态和扩散是更大的运营压力。- 如果每天有 500 万局结束，游戏元数据约 1 KB/局 ≈ 5 GB/天；与视频/社交系统相比，长期归档成本低廉。

2. 核心实体（约 2 分钟）

  • User — user_id, rating, peak_rating。
  • Game — game_id, white_id, black_id, moves[], clock_white, clock_black, status, result.
  • 传递 — (game_id, ply, uci_or_san, ts_server, time_left).
  • 匹配票据 — user_id, rating, time_control, queued_at.

3. API / 协议（约 5 分钟）

POST  /v1/queue                       { time_control, rating_range }    → { ticket_id }
WS    /v1/queue/{ticket_id}            → push: { matched: game_id }
WS    /v1/games/{id}                   ← move: { uci }
                                       → push: { move, clock_left }
POST  /v1/games/{id}/resign

从令牌中获取 User。

5. 高层设计（约 10–15 分钟）

  1. Matchmaking service。 每种时间控制有其自己的队列（10 秒闪电战、5+5、15+10、经典）。在队列中，玩家按评级排序。匹配进程采样评级在扩展窗口内的配对（初始窗口窄，随等待时间拓宽）。配对成功后：创建游戏行，向两个客户端推送 WS 消息。
  2. Game session service。 一局游戏的所有走棋都汇聚到一个所有者节点（在 game_id上进行一致性哈希）。所有者将游戏状态保存在内存中并持久化到持久存储。
  3. 走棋验证。 服务器根据当前棋局验证每一步棋。非法棋步则拒绝。服务器是棋局和计时器的事实来源。
  4. 计时器权威。 服务器使用单调时间追踪计时器。根据服务器时间戳从当前走棋玩家的时钟中扣除时间。客户端 UI 使用自己的时钟显示，但服务器仲裁超时。
  5. 走棋广播。 所有者通过 WS 将经过验证的棋步推送给双方客户端。
  6. 游戏结束 + 评级更新。 在终局状态时，所有者运行 Elo 更新，写入最终游戏记录，向双方客户端推送 game_over 。
  7. 持久性。 走棋列表按每步追加（Cassandra 以 game_id为键）。游戏元数据在 MySQL 中（以 game_id).
  8. 观战路径（可选）。 其他用户可以以只读模式订阅游戏的 WS；所有者通过现有频道向他们扩散广播。

端到端说明：Alice 排队进行 5 分钟闪电战 → 匹配器将她与 Bob 配对（评级差距在 50 分以内）→ 创建游戏 G → 双方客户端与游戏所有者建立 WS 连接 → Alice 走 e4 → 服务器验证，扣除她 0.3 秒时间（网络延迟），将 e4 连同 clock_white: 4:59.7推送给 Bob。游戏进行 30 步；Bob 将死后，所有者运行 Elo，更新两名用户的评级，写入游戏记录。

6. 深度探讨（约 10 分钟）

具体存储与分片键。

存储 键模式 数据结构 / 条目 TTL / 保留期 分片键
匹配队列 queue:{time_control} Redis ZSET，score=rating，member=ticket_id，约 80 B 10 分钟或直到匹配成功 time_control
游戏所有者映射 game_owner:{game_id} Redis string {owner_node, epoch}，约 60 B 活跃游戏 + 5 分钟 game_id
游戏元数据 game:{game_id} MySQL 行，players/status/result/clocks，约 1 KB ， game_id
走棋日志 moves:{game_id} 仅追加列表/分区，按棋步序号排序，约 50–80 B/步 永久保存，之后冷归档 game_id
棋局检查点 state:{game_id}:{checkpoint_ply} FEN/计时器快照，约 200 B 活跃 + 7 天热数据 game_id
重连游标 ack:{game_id}:{user_id} 最后确认的棋步序号，约 40 B 活跃 + 1 小时 (game_id, user_id)
评级行 rating:{user_id} rating/provisional 标志，约 100 B ， user_id

棋局本身不在每次走棋时作为独立的权威 blob 存储；它在内存中由走棋日志推导而来，并偶尔做检查点以便快速重连/重放。

各存储容量估算。

存储 / 索引 Math 备注
匹配队列 50 万等待/并发用户 × 80 B × Redis 开销 ≈ 数十 MB 按时间控制拆分
游戏所有者映射 50 万活跃游戏 × 60 B × 3 ≈ 逻辑 90 MB 临时路由索引
游戏元数据 每天 500 万局 × 1 KB × 3 ≈ 15 GB/天 长期归档仍然适中
走棋日志 25k 次移动/秒 × 80 B × 86,400 × 3 ≈ 518 GB/天（若峰值持续） 实际每日平均值更低
Checkpoints 500k 活跃 × 10 个检查点 × 200 B × 3 ≈ 3 GB 加速恢复，非数据来源
评分 10M 用户 × 100 B × 3 ≈ 3 GB 体量小但对一致性敏感

为何在对局匹配中使用 Redis ZSET 而非 SQL 或 Kafka：操作本质是"在该时间控制队列中查找邻近评分"并原子性地移除两张票据。SQL 能做到，但热门评分段周围的行争用较为棘手；Kafka 有序但无法按评分搜索。对于棋步，Cassandra 或仅追加的 SQL 表均可使用，因为读取是单局范围扫描；关键选择在于按 game_id而非存储品牌进行序列化。

棋步路径。 1. 客户端发送 {game_id, uci, client_seq} 通过玩家 WebSocket 发送。 2. WS 网关按 game_owner:{game_id}进行路由；若缺失，则通过一致性哈希解析所有者并创建一个 epoch。 3. 所有者验证当前是该玩家的回合，根据内存中的棋盘状态校验棋步，并使用单调服务器时间戳计算已用时钟时间。 4. 所有者追加 moves:{game_id} 与 (ply, move, server_ts, time_left) 并更新 game:{game_id} 状态/时钟。 5. 所有者将已接受的棋步推送给双方玩家；观战广播在单独频道上运行，因此不会延迟玩家 ACK。 6. 到达终局状态时，所有者写入最终结果，在一次事务或 outbox 支持的工作流中更新 rating:{white_id} 。 rating:{black_id} ，然后发出 game_over.

重连/冷局路径。 1. 客户端携带 {game_id, last_seen_ply}重连；网关按 game_owner:{game_id} 路由，或在映射缺失时选举新所有者。 2. 热所有者返回内存快照以及 last_seen_ply之后的棋步；同时更新 ack:{game_id}:{user_id} ，以便后续重连知道游标位置。 3. 冷所有者加载最新的 state:{game_id}:{checkpoint_ply} 并从该半步重放 moves:{game_id} 。若无检查点，则从第 0 半步开始重放；对于正常对局来说仍然开销较低。 4. 若对局已结束并归档，则 GET /games/{id} 读取 game:{game_id} 加上棋步存档；延迟可以更高，因为已不在玩家棋步路径上。

双索引框架。 moves:{game_id} 是持久的内容日志。 game_owner:{game_id} 仅为路由索引， ack:{game_id}:{user_id} 是每客户端的重放游标， rating:{user_id} 是终局结果后派生的物化视图。这些索引均不能替代合法棋步日志。

单局单一所有者。 所有棋步和时钟决策在一个节点上序列化。将 game_id 哈希到一个会话服务器。故障时，通过主节点选举 + 从持久棋步日志回放来恢复。所有者负责强制排序、校验棋步并仲裁时钟。

为何使用单一所有者而非在每次棋步上加分布式锁：国际象棋每局吞吐量极低且排序要求极严，因此将所有者与数据共置更简单、更快速。数据库行锁在正确性上没问题，但会给热路径增加延迟，并使时钟公平性依赖于存储的尾延迟。需准备好回答"为何不用 CRDT？"——协作合并语义无法帮助解决问题，因为合法的下一步棋只有一个。

延迟下的时钟。 每位玩家都有一个服务器跟踪的挂钟。当玩家的棋步到达时，服务器从玩家时钟中减去 (server_time - last_move_received) 。若时钟将变为负数，则该玩家已超时（负旗，判负）。 - 服务器在每次棋步推送中附带 time_left ，使对手 UI 显示准确的倒计时。 - 每次棋步确认时累加增量时间控制（例如每步 +5 秒）。 - 防骚扰：标记超时前设置小的宽限窗口（~150 ms）以吸收网络抖动。

走棋验证。 服务器持有局面；对每步棋依据国际象棋规则（合法格子、不暴露己方王、易位权、吃过路兵）进行校验。遇到非法棋步则拒绝并说明原因。客户端也应在本地校验以防止提交明显非法的棋步。

断线处理。 WebSocket 在对局中断开：客户端有 30 秒宽限期；重连后，服务器重放自客户端最后一次 ACK 以来的棋步。断线期间，服务器的时钟继续对正在行棋的玩家计时。

基于评分段的对局匹配。 按时间控制维护一个 Redis 有序集合，以评分为键。匹配时扫描评分在 ±N 范围内的对，N 从 50 开始随时间扩大。在快速匹配和公平匹配之间进行权衡。

评分更新。 Elo： new = old + K × (actual - expected)。K 值因玩家试用状态而异。与对局记录写入在同一事务中完成。

反作弊（简述）。 棋步分析：与顶级引擎走法进行比较；在高搜索深度下持续一致则可疑。完整设计超出范围；真实平台使用离线批处理任务和人工审核。

观战扩展性。 热门对局（如 Carlsen vs Niemann）吸引 10 万观战者。服务器的 WS 层将"玩家"连接（改变状态）与"观战者"连接（只读）分离。对观战者采用层级扇出。

区域选择与冷启动。 对局在创建时分配到一个区域，通常是离双方玩家最近或锦标赛主场区域。若玩家相距较远，双方均承受一定延迟，但时钟保持公平，因为该区域的服务器时间戳具有权威性。所有者池完全重启后，对局所有者从检查点和棋步日志重建活跃状态，并短暂暂停时钟并附加暂停积分，避免在恢复期间有人超时判负。

对局存储。 类 PGN 格式的棋步列表，压缩后约 50 B/局。长期保留成本低廉——十亿局存档 < 100 GB。
