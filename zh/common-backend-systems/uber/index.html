Uber — Rideshare Backend (walkthrough)

2026/06/01 · Common Backend Systems
目录
目录

难度： Hard · 总耗时： ~45 min · 框架： Hello Interview Delivery

一句话场景： 构建 Uber。乘客发起用车请求；系统将其与附近司机匹配；双方可看到对方的实时位置；送达后完成支付。月活乘客1亿，司机500万，每天2000万次行程，峰值约每秒1500次行程，约100万司机并发以4 Hz频率上报GPS。

1. 需求（约 5 分钟）

功能需求 - 乘客从出发地到目的地发起用车请求；系统匹配一位可用司机。 - 行程中双方可看到对方的实时位置。 - 完成后完成支付。

我会推迟 Uber Eats、动态定价深度设计、司机入驻/背景调查，以及仅提及但不深入设计拼车（UberPool）。

非功能需求 - 匹配延迟 p99 < 5秒——乘客无法等太久。 - 实时位置更新在1秒内完成。 - 行程状态机（已请求 → 已匹配 → 进行中 → 已完成）需要强一致性。 - 区域隔离——某一城市的故障不影响其他城市。 - 高可用——涉及资金交易。

容量估算 - 100万并发司机 × 4 Hz GPS = 每秒400万次 GPS 写入。 - 峰值每秒1500次行程。 - 每次行程：约30分钟 × 240次ping/分钟 = 每次行程持久化7200次 GPS ping。 - 实时位置状态很小：100万司机 × 约80 B最新位置记录 ≈ 主节点80 MB，含副本数百 MB。主要成本在于数据流：每秒400万次ping × 约100 B ≈ 400 MB/秒，压缩/复制前约35 TB/天。 - 行程状态很小：每天2000万次行程 × 约1 KB/行程 ≈ 主节点20 GB/天；需要事务正确性，数据量不大。

2. 核心实体（约 2 分钟）

  • 乘客 — rider_id.
  • 场景 — driver_id，当前位置、状态（离线/可用/行程中）、当前行程。
  • 行程 — ride_id, rider, driver, origin, destination, status, fare.
  • GPSPing — (driver_id or rider_id), lat, lng, ts.

3. API（约 5 分钟）

POST  /v1/rides                  { origin, destination }   → { ride_id, status: requested }
GET   /v1/rides/{id}             → Ride
WS    /v1/rides/{id}/stream      → push: state changes + counterparty location

POST  /v1/drivers/location       { lat, lng }   (auth: driver)
POST  /v1/drivers/rides/{id}/{action}    // accept, pickup, dropoff

从令牌获取用户。行程状态转换按 (ride_id, action).

游标不对外暴露；服务端拥有行程状态机。客户端仅提交动作，每个动作在当前持久化状态允许时才被接受。

5. 高层设计（约 10–15 分钟）

我会按城市/区域对世界进行分区。每个区域基本上是自治的。

  1. 行程服务。 拥有行程状态机。按 ride_id分片，但路由确保同一区域的所有行程落在区域集群上。
  2. 司机位置服务。 司机以4 Hz POST GPS → 摄入写入区域 Redis 地理空间索引（GEOADD）以及 Kafka 供下游消费者使用（分析、反欺诈、行程记录）。
  3. Matching 服务。 在用车请求时触发： - 查询乘客所在 S2 单元格 + 环内的附近司机（廉价的 GEORADIUS）。 - 筛选可用司机（状态 = 可用）。 - 评分：到接客点的 ETA × 司机接单率 × 溢价意愿。 - 向顶级候选人发送邀约；拒绝/超时后转到下一位。
  4. 基于关注关系的 feed。采用类似 FB News Feed 的混合推/拉模式，但规模更小：关注时推送 → 被关注者的评论出现在关注者的 feed 列表中。大多数用户关注人数 < 100；按需读取方案可行。 司机应用通过持久 WS 或推送渠道接收邀约。司机在约15秒内接受，否则邀约重新路由。
  5. 行程追踪器。 匹配成功后，乘客和司机共享一个连接到行程频道的 WS；一方的 ping 扇出到另一方。
  6. 定价服务。 在创建行程时，根据启发式方法估算费用（基础费 + 按公里 + 按分钟 + 该单元格的溢价倍数）。送达时根据实际行程数据最终结算。
  7. 支付。 不在详细设计范围内，但在架构图中需体现。送达后通过支付网关完成扣款。

端到端叙述：Alice 点击"呼叫 UberX" → 行程服务写入行程行（状态=已请求）并询问匹配器 → 匹配器 GEORADIUS 查找附近司机，选择42号司机 → 推送邀约通知 → 42号接受 → 行程状态 → matched，司机已分配。双方应用开启 WS 连接到行程频道。42号司机位置 ping 流 → 摄入 → 扇出到 Alice 的 WS。到接客点时，司机点击"已接客" → 行程状态 → in_progress。送达时，行程状态 → completed，费用最终结算，支付完成。

6. 深度探讨（约 10 分钟）

具体存储契约。

存储 键/数据结构 条目大小 + TTL 分片键
司机地理位置 geo:{region}:{s2_cell} Redis geo/zset 成员=driver_id 约40 B/成员，TTL 10秒 region+s2_cell
司机状态 driver:{driver_id}:status KV {available,on_trip,ride_id,version} 约100 B，在线期间 TTL driver_id
邀约租约 offer:{driver_id} → ride_id 与 NX EX 15 约40 B，TTL 15秒 driver_id
行程真实状态 rides:{region} SQL/Spanner 行 约1 KB，保留多年 region, ride_id
行程事件日志 Kafka ride_events 约200 B/事件，热数据7–30天 ride_id
GPS 历史记录 按日期 + 区域分区的对象/列式存储 压缩后约20–100 B/ping，热数据30–90天 region, date

各存储容量估算。 热点匹配器状态相比数据流非常小： 1M online drivers × 40 B × 3 地理条目约120 MB，含副本的状态行约300 MB。邀约租约极小：即使是 1500 rides/sec × 15 s × 40 B × 3 也不超过3 MB。行程真实状态为 20M rides/day × 1 KB × 3 ，约60 GB/天。昂贵的路径是 GPS 历史记录和 Kafka 保留： 4M pings/sec × 100 B 压缩/复制前约35 TB/天，因此我将最新位置存储在 Redis 中，Kafka 中保留较短时间，并将行程轨迹压缩归档到列式对象存储。

用车请求路径追踪。 客户端 → 行程服务创建 rides 行，状态为 requested → 匹配器并行查询 Redis 地理索引中的附近 S2 单元格 → 并行读取顶级候选司机状态 → 写入 offer:{driver_id} 租约，附带 NX EX 15 → 推送邀约通知 → 司机接受调用行程服务 → 行程服务通过一次事务/比较并交换操作有条件地更新行程 requested→matched 和司机 available→on_trip → 向 Kafka 发出 ride.matched 事件。若第一位司机超时，租约到期，匹配器尝试下一位候选人。

每秒 400 万次 GPS 写入的地理索引。 每个区域使用 Redis 地理空间索引。若单个 Redis 达到上限，则将该区域的 Redis 分片为多个单元。写操作为 GEOADD driver:region:cell；读操作为 GEOSEARCH。每次操作亚毫秒级，适合全内存存储。

为何选择 Redis/S2 而非 PostGIS 或 Elasticsearch：此路径写入频繁、数据保留时间短，需要在数秒内找到最近的活跃司机。PostGIS 适合持久化空间查询，但无法支持每秒 400 万次的高频易变写入；Elasticsearch 增加了索引延迟和更高的刷新成本。持久化的 GPS 轨迹写入 Kafka/对象存储，而非热点匹配器。

对于行程真实状态，我会选用区域化 SQL/Spanner 风格的存储，而非 Cassandra，因为状态机需要条件转换，司机分配不能出现重复预订。Cassandra 适合仅追加的事件历史，但的真实状态来源 requested→matched→completed 需要事务或强比较并设置（compare-and-set）语义。

匹配算法。 简单的最近司机贪心策略不公平。更好的做法：每 1–2 秒进行批量匹配，收集一个单元格内的待处理请求和可用司机，运行二部图分配以最小化总 ETA。可提及匈牙利算法或拍卖式匹配作为生产环境的选择。此外，一名司机可同时向多名乘客发出邀约，但以第一个接受者为准。

司机分配的并发控制。 同一区域的两个行程请求可能选中同一名司机。使用 Redis SET driver:42:offered ride:99 NX EX 15 在邀约发出期间对司机进行 15 秒租约锁定。若司机拒绝，租约到期后重新纳入候选。

GPS 扇出。 一次行程包含 1 名司机和 1 名乘客——每次行程的扇出为 1:1。挑战在于写入速率（每秒 400 万次），而非扇出。按区域使用写入 Worker，采用无状态直写到 Redis 的方式。持久化到 Kafka 用于分析，再归档到冷存储。

行程状态机。 由行程服务负责。每次转换均为单行更新并附带状态检查（UPDATE rides SET status='completed' WHERE id=X AND status='in_progress'），以强制执行合法转换。每次转换时向 Kafka 发送事件，供下游服务消费。

阶段 允许的下一个状态 数据真实来源 延迟目标
requested matched, canceled 行程行 + 匹配器尝试 首次邀约时间 <5 秒
matched driver_arrived, canceled 行程行 + 司机状态 立即 CAS
driver_arrived in_progress, canceled_no_show 行程行 <500 毫秒
in_progress completed, support_intervention 行程行 + 行程 ping <500 毫秒
completed terminal 行程行 + 支付幂等性 支付可能异步结算

面试要点："位置可以最终一致；行程状态机不能，因为那里涉及资金、安全和司机分配。"

预计到达时间预测。 v1：haversine 距离 × 城市平均速度 + 缓冲时间。v2：来自了解实时交通状况的地图服务的路由 ETA。v3：以历史行程时间为输入的 ML 模型。每个层级都增加延迟和复杂度；我们会从 v1 开始，并说明升级路径。

价格波动（Surge pricing）。 按 S2 单元格的滚动供需比；波动倍率 = 该比率的分段函数。每分钟重新计算，写入 Redis 缓存，定价服务在创建行程时读取。在乘客发起请求前展示波动价格。

双索引框架。 司机按位置建立索引用于匹配（geo:{region}:{s2_cell}），按身份建立索引用于正确性保障（driver:{driver_id}:status）。匹配时同时读取两者：地理索引找出候选人，但司机状态行决定该司机是否真正可被分配。这是针对两种访问模式优化的同一份活跃司机数据集。

故障模式。 - 司机端应用在行程中崩溃：服务端的 last_ping > 30s 触发检查；乘客看到"正在尝试重新连接"。- 支付过程中发生网络分区：支付在 ride_id上具有幂等性，将被重试。- 乘客在匹配后、接客前取消：收取小额取消费，司机重新释放。

情形 模式 数据源 延迟预算 回退
市中心热点单元 批量匹配 Redis 地理位置 + 司机状态 匹配 <5 秒 扩大单元格范围，缩短邀约超时时间，每 1 秒进行一次批量分配
郊区稀疏区域 更大范围搜索 相邻 S2 环 + ETA 服务 匹配 <10 秒 显示较长 ETA，保持行程请求状态，允许乘客免费取消

Redis 地理位置分片宕机 保守回退 最后已知状态 + Kafka 延迟监控 本地暂停匹配 停止受影响单元格的新匹配；已有行程继续进行

区域隔离。 每个城市拥有独立的行程/匹配/Redis 技术栈。跨区域故障被隔离。全局服务仅用于身份认证、行程历史聚合和支付。
