1. 基础知识与解题思路

本题考察点

WhatsApp 与 Facebook Messenger 类似，但拥有 20 亿用户规模，以移动端优先的网络条件为前提，并将端对端加密作为不可妥协的要求。难点在于：在全球不稳定的蜂窝网络环境下，将每条消息恰好一次地投递到每台设备，同时执行服务器无法读取的 E2E 加密。

其不变量如下：

每条已发送消息以密文形式持久化存储在服务端，至少投递一次到每台接收设备，在所有设备上以相同顺序显示，并在被确认前保持可用；端对端加密强制执行，服务器永远看不到明文；群发消息在不向服务器泄露会话内容的前提下进行扇出。

在 ICT5 / L6+ 级别，面试官考察的是：你能否设计 Signal 协议风格的 E2E 加密、处理移动网络不可靠性、在不牺牲 E2E 安全的前提下支持多设备同步，并在加密约束下分析群组扇出问题。

60 秒面试框架

口述版本：

我会围绕以下核心设计 WhatsApp：服务端的每用户消息收件箱（保存待投递的加密消息）、向移动设备的长轮询或持久推送通道，以及用于端对端加密的 Signal 协议。服务端仅存储和路由密文，无法读取内容。多设备同步需要在用户设备之间进行密钥传递。关键决策点包括：多设备 E2E 加密、通过 Sender Keys 实现群组加密、移动端可靠性以及离线消息投递。

首先明确需求

功能性需求：

  • 一对一消息和群组消息（文字、媒体、语音备忘录）。
  • 语音和视频通话。
  • 状态更新（24 小时短暂内容）。
  • 强制端对端加密。
  • 已读回执、投递状态指示、正在输入提示。
  • 多设备支持（手机为主设备 + 关联设备）。
  • 离线消息投递；在服务端排队。
  • 加密备份到云端。

非功能性需求：

  • 2B+ 用户，每天数千亿条消息。
  • 移动端优先：蜂窝信号差、连接时断时续。
  • 在线投递低延迟（亚秒级）。
  • E2E 加密不可妥协。
  • 通过本地排队承受运营商级别的断线。
  • 多区域部署；跨国发送快速。

以下内容超出范围，除非被特别询问：

  • 语音/视频通话媒体服务器（WebRTC；独立模块）。
  • WhatsApp Business API 功能。
  • 支付功能。

规模假设

领域 假设
用户 20 亿以上月活用户。
每日消息数 100B+。
平均峰值 百万/秒。
群组规模 最多 1024 名成员。
每用户设备数 1 部主手机 + 最多 4 台关联设备。
Latency（延迟） 在线投递低于一秒；离线时重连后接收。
存储 服务端仅存储未投递的密文；投递后清除。

瓶颈在于每台设备收件箱的推送吞吐量以及加密约束下的群组扇出。

Signal 协议基础

每位用户拥有：

  • 身份密钥（长期，公/私钥对）。
  • 签名预置密钥（中期）。
  • 一次性预置密钥（批量上传，每个仅使用一次）。

X3DH（发起会话）：

  1. Alice 从服务器获取 Bob 的身份密钥 + 签名预置密钥 + 一个一次性预置密钥。
  2. Alice 使用所有这些密钥计算共享密钥。
  3. Alice 将初始消息 + 她的临时密钥发送给 Bob。
  4. Bob 计算出相同的共享密钥。

双重棘轮（每条消息使用独立密钥）：

  • 每条消息使用从棘轮派生的唯一对称密钥。
  • 棘轮随每条消息（链密钥）及收到新 DH 临时密钥（根密钥）时推进。
  • 提供前向保密：当前密钥泄露不会暴露历史消息。

多设备： - 每台设备拥有各自的身份密钥和预置密钥。 - 发送方获取每台接收设备的密钥包，对消息加密 N 次，每台设备一份。 - 对于群组，这会按 sender_devices × group_size × devices_per_member.

群组 Sender Keys

为 1000 人群组中每台接收设备单独加密消息的开销很大。Sender Keys 可减少这一开销：

  • 发送方为群组生成一个对称密钥。
  • 发送方通过双重棘轮将该对称密钥加密发送给每台接收设备（一次性设置或在成员变更时执行）。
  • 后续每条消息，发送方用该对称密钥加密消息；将密文 + 棘轮步进发送给所有成员。
  • 每位接收方使用此前收到的对称密钥解密。

这将每条消息的工作量从 O(N × 设备数) 降低到密钥建立后的 O(1)。

面试用语：

按设备加密对群组扩展性差。Sender Keys 让一份密文服务所有成员，密钥设置成本在成员变更事件时摊销。

发送流程

  1. 发送方客户端生成消息。
  2. 对每台接收设备，使用 Signal 会话（或群组对称密钥）加密。
  3. 将消息包发送给发送服务： {recipient_device: ciphertext}.
  4. 发送服务将每条 (recipient_device, ciphertext) 写入该设备的收件箱。
  5. 若设备在线（推送通道已连接），立即推送。
  6. 否则，密文在收件箱中等待。
  7. 发送方收到服务器确认（已接收）。
  8. 接收设备在推送时解密并显示。
  9. 接收方发送已投递回执；稍后发送已读回执。
  10. 服务器可删除已投递的密文（或短暂保留以用于跨设备同步）。

移动端可靠性

WhatsApp 的秘密武器是移动网络工程能力。

  • 具有自适应心跳的持久推送通道。
  • 后台通知触发应用唤醒。
  • 积极的压缩和消息批处理。
  • 网络质量感知的重试策略。
  • 带宽高效编码（自定义协议）。

客户端维护一个近期消息的小型本地数据库；发送时本地入队；连接恢复时清空队列。

多设备

WhatsApp 的多设备方案较为特殊：历史上手机是主设备，关联设备通过它代理。现代多设备方案：

  • 每台设备拥有独立身份。
  • 发送方向接收方所有设备发送消息。
  • 目录中保存每设备密钥包。
  • 新设备必须通过主设备验证（二维码或推送通知）。

历史消息同步到新设备：

  • 关联时，主设备通过 E2E 通道将历史消息密钥 + 密文（或明文）发送给新设备。
  • 后续消息由发送方直接发送给新设备。

群组消息

与 Messenger 相同模型，但加入 E2E 加密：

  • 群组注册表保存成员关系。
  • 发送方获取成员设备列表。
  • 发送方使用 Sender Keys（对称加密）加密。
  • 服务器将密文扇出到每个成员设备的收件箱。
  • 成员变更（添加/移除）：对群组重新生成密钥（轮换 sender key）。

对于 1024 人群组，每条消息产生 1024 次收件箱写入——仍可管理。

语音和视频通话

大部分不在本次设计范围内，但简述：信令通过消息基础设施传递，媒体通过 WebRTC 点对点或 TURN 中继服务器传输。通话的 E2E 加密使用 SRTP，密钥通过相同的 Signal 会话交换。

备份

用户加密备份到用户的 iCloud/Google Drive。

  • 本地备份文件使用从用户密码或硬件绑定密钥派生的密钥加密。
  • 云服务提供商只能看到密文。
  • 恢复需要用户密码或设备生物特征验证。

扩展与分区

数据 分区键 原因
收件箱 device_id 哈希值 每设备队列。
身份 / 密钥包 user_id 哈希值 在会话开始时查找。
群组 group_id 哈希值 群组操作。
推送连接 按 device_id 长连接。
传输中的消息 大部分为瞬态 确认后清除。

一致性模型

层级 一致性 原因
收件箱 强一致性（密文在确认前持久化存储） 不丢失。
每会话消息顺序 基于时间戳 + 发送方设备 id 的尽力而为排序 多发送方群组的严格排序难以实现；用户可接受。
密钥包 强一致性 安全设置所必需。
回执 最终一致 用户体验。
群组成员关系 添加/移除时强一致性。 避免向已移除的成员发送消息。

故障模式与缓解措施

故障模式 缓解措施 监控指标
收件人离线。 收件箱保留密文直至投递完成。 收件箱消息时间分布。
一次性预密钥已耗尽。 客户端定期上传更多预密钥。 预密钥耗尽速率。
设备密钥被攻破。 通过 Double Ratchet 实现前向保密；检测到攻击时重新生成密钥。 已报告的密钥泄露次数。
多设备密钥同步失败。 新设备需要重新验证。 设备关联失败率。
群组消息扇出延迟。 并行扇出；背压控制。 每条消息的扇出耗时。
推送通道丢失。 后台重连；回退至拉取模式。 推送重连速率。
服务器无限期保存密文。 投递后清理 + TTL。 收件箱存储增长。
需要轮换 Sender Key（成员已被移除）。 移除成员时重新生成密钥；所有成员获得新密钥。 重新生成密钥的频率。
运营商级别故障。 设备本地队列；网络恢复后投递。 按地区划分的投递成功率。
大规模垃圾邮件/诈骗问题。 信誉评分；频率限制；社区举报。 被举报的消息数量。

运维指标

正确性：

  • 消息丢失率（目标为零）。
  • 乱序投递率。
  • 解密失败率。
  • 回执准确率。

容量：

  • 按地区划分的发送 QPS。
  • 推送连接数。
  • 群组扇出延迟。
  • 收件箱存储容量。

业务层面：

  • DAU；每用户每天的消息数。
  • 群组规模分布。
  • 多设备普及率。

多区域策略

  • 用户归属地区用于收件箱和身份标识。
  • 推送网关按地区部署；用户漫游时进行跨地区路由。
  • 密钥目录在全球范围内复制，以实现快速的密钥包查询。
  • 故障转移地区保持热备状态。

需要指出的权衡

  • 端到端加密阻断了服务端内容审核；安全保障依赖举报机制和元数据分析。
  • 群组的每设备加密方案扩展性差；Sender Keys 可分摊成本但增加了复杂性。
  • 多设备场景下难以保证强顺序性；依赖时间戳和打破平局规则。
  • 激进的移动端优化可降低电池消耗，但需要自定义协议。
  • 服务器保存密文是离线投递的必要条件，但带来了额外的存储负担和风险。
  • 多设备安全需要谨慎的关联和验证机制。

常见陷阱与更优做法

陷阱 更优做法
视为普通文本消息处理。 使用 Signal Protocol 进行端到端加密；服务端只看到密文。
大型群组的每设备加密方案。 使用 Sender Keys 分摊成本。
每用户一个身份。 多设备场景下每设备独立身份。
信任客户端提供的收件人列表。 发送时由服务端校验群组成员资格。
无限期保留密文。 确认后清除 + TTL。
群组消息同步扇出。 并行写入收件箱并配合背压控制。

面试官可能深入探讨的方向

深入方向 优秀回答要点
Signal Protocol X3DH、Double Ratchet、前向保密。
多设备端到端加密 每设备密钥；发送方加密 N 次；设备关联与验证。
群组加密 Sender Keys；成员变更时轮换密钥。
移动端可靠性 持久化推送；本地消息队列；自适应心跳。
大规模垃圾消息 元数据分析；信誉评分；社区举报。
备份 用户密钥加密；云端存储密文。

Staff 级别回答检查清单

  • 通过 Signal Protocol 实现端到端加密；每设备独立密钥。
  • 服务端仅存储密文；确认后清除。
  • 使用 Sender Keys 提升群组效率。
  • 每设备独立收件箱与推送通道。
  • 具备移动网络抗扰性的设计。
  • 通过关联与验证支持多设备。
  • 加密备份到用户的云端。
  • 通过元数据（而非内容）处理垃圾消息。

面试用语

WhatsApp 是基于 Signal Protocol、采用每设备加密消息队列的系统。服务器是一个只看到密文的路由器。Sender Keys 使群组消息成为可行方案；每设备收件箱实现了多设备支持。难点在于移动端可靠性、群组加密效率以及在无法访问服务端内容的情况下保障安全。

2. 模拟面试

面试官： 设计 WhatsApp。

候选人： 我将围绕每设备密文收件箱、用于端到端加密的 Signal Protocol 以及在线投递的持久化推送通道来进行设计。服务器存储并路由密文；明文仅存在于用户设备上。多设备问题通过每设备独立身份解决。群组使用 Sender Keys 分摊加密成本。不变式是：每条消息都被持久存储，直到投递至所有收件人设备，且全程强制执行端到端加密。

面试官： 带我过一遍发送一条一对一消息的流程。

候选人： 发送方客户端从密钥目录获取收件人设备的预密钥包。运行 X3DH 建立共享密钥，然后 Double Ratchet 派生每条消息的密钥。加密消息。发送至服务器： (recipient_device_id, ciphertext, ratchet_metadata)。服务器将消息写入收件人设备的收件箱。如果收件人设备在线（推送通道已连接），则立即推送。收件人设备使用其 Double Ratchet 状态解密消息。发回已投递回执；之后发送已读回执。服务器在收到已投递确认后清除密文。

面试官： 多设备端到端加密。

候选人： 每台设备都有自己的身份密钥和预密钥包。发送方发送消息时，会获取所有收件人设备的密钥包，并分别为每台设备单独加密消息。因此，如果收件人有 3 台设备，发送方会发送 3 份密文。在发送方自己这侧，其他设备也需要一份副本——他们同样会加密给自己。服务器将消息扇出至所有相关收件箱。

面试官： 这对群组来说代价很高。

候选人： 是的——对于一个拥有 1000 名成员、每人 2-3 台设备的群组，朴素的每设备加密每条消息需要 2000-3000 次加密操作。Sender Keys 解决了这个问题。发送方为群组生成一个对称密钥；一旦该密钥建立完成（在群组创建时或成员变更时通过每设备加密分发），后续所有消息均使用该对称密钥。每条消息：对称加密一次；执行 ratchet 步进。将密文发送给所有成员。他们使用共享的对称密钥解密。这样就摊薄了成本。

面试官： 对称密钥何时更换？

候选人： 在成员变更时。当有人被移除时，重新生成密钥，使其无法解密未来的消息。当有人加入时，要么重新生成密钥，要么将当前密钥加密后传输给他。Sender Keys 通常在成员变更事件时轮换。

面试官： 收件人离线 3 天。

候选人： 密文存放在服务器上该设备的收件箱中。当收件人重新上线时，设备通过推送通道连接，获取所有排队的消息并解密。服务器收到每条消息的已投递确认后将其清除。根据 WhatsApp 的策略，未投递的密文会在一个时间窗口内保留（例如 30 天）；超过期限后丢弃，并显示"消息已过期"提示。

面试官： 你如何处理不稳定的移动网络？

候选人： 具有自适应心跳的持久化推送通道——在稳定的 WiFi 下降低 ping 频率，在移动蜂窝网络下提高频率。激进的压缩；高带宽效率协议。客户端维护一个本地待发消息数据库；离线时入队；重连后排空。服务端采用指数退避重试推送。后台推送通知唤醒应用进行投递。

面试官： 多设备关联——新关联的设备如何获取历史消息？

候选人： 用户在主设备上扫描二维码。主设备与新设备派生共享密钥，然后通过消息基础设施发送历史消息——使用派生的加密密钥加密。新设备获取近期历史记录；为保证关联速度，非常旧的消息可能不会迁移。关联完成后，新设备进入密钥目录；发送方之后会直接向其发送新消息。

面试官： 1024 人的群组。扇出是如何工作的？

候选人： 发送方使用 Sender Key 对称加密消息。向服务器发送一个请求，其中包含密文和收件人列表（群组成员）。服务器执行扇出——并行将密文写入每个成员的收件箱。每条消息产生 1024 次收件箱写入。在线成员收到推送；离线成员进入队列等待。每个成员的客户端独立解密。

面试官： 服务端垃圾消息审核怎么办？端到端加密意味着你看不到内容。

候选人： 对，我们无法检查内容。我们依赖元数据：发送频率、收件人接受情况、举报。如果用户 X 每分钟向陌生人发送数千条消息，我们会限流并可能封禁账号。社区举报可以揭示垃圾消息模式。转发消息标识符可以抑制链式传播。取舍在于：更强的端到端隐私意味着更弱的内容审核；我们依赖行为信号。

面试官： 备份呢？

候选人： 加密备份至用户云端（iCloud/Google Drive）。本地备份文件使用从用户设置的密码或设备绑定凭证派生的密钥加密。云端只看到密文。恢复需要密码。如果用户忘记密码，备份即丢失。

面试官： 多区域？

候选人： 收件箱基于用户归属地区。推送网关按地区部署；用户漫游时路由至其归属地区。密钥目录在全球复制，以实现快速的密钥包查询。故障转移地区保持热备状态。跨地区消息路由发生在服务层；密文从发送方地区传送至收件人地区的收件箱。

面试官： 总结。

候选人： WhatsApp 是采用 Signal Protocol 端到端加密、基于每设备密文收件箱的系统。服务器路由密文，从不接触明文。多设备通过每设备独立身份和每设备加密实现。群组使用 Sender Keys 分摊扇出成本。移动端可靠性通过持久化推送、本地队列和自适应协议保障。垃圾消息审核依赖元数据而非内容。备份使用用户密钥加密后存入云端。最难的约束是端到端加密与多设备支持以及群组效率的结合。

3. 练习备注

五分钟版本

  1. 明确需求：一对一、群组、端到端加密强制、多设备、移动端。
  2. 不变式：密文发送至收件人设备，服务器永远接触不到明文。
  3. Signal Protocol：X3DH、Double Ratchet。
  4. 每设备收件箱 + 持久化推送。
  5. 群组使用 Sender Keys。
  6. 多设备：每设备独立身份；发送方加密 N 次。
  7. 移动端可靠性通过持久化推送 + 本地队列实现。
  8. 垃圾消息审核依赖元数据；用户备份在客户端加密。

一分钟结尾陈述

WhatsApp 是 Signal Protocol 端到端加密消息在地球规模上的实践。服务器将密文存储并路由至每设备收件箱；持久化推送通道在设备在线时完成投递；本地队列吸收不稳定网络带来的冲击。群组使用 Sender Keys 分摊每设备加密成本。多设备通过每设备独立身份解决。垃圾消息审核依赖元数据，而非内容。备份使用用户密钥加密后存入云端。最难的约束是端到端加密与多设备支持以及群组效率的结合。

最佳面试用语

  • "服务器只存储密文；明文仅存在于设备上。"
  • "Sender Keys 使群组消息成为可行方案；没有它，每设备加密的成本会急剧膨胀。"
  • "多设备依赖每设备独立身份，而非每用户一个密钥。"
  • "移动端可靠性靠的是自定义协议，而不仅仅是更好的服务器。"
  • "端到端隐私与内容审核之间存在张力；我们依赖元数据。"
