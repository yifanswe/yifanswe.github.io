1. 基础知识与解题思路

本题考察点

限时抢购是一场蓄意制造的踩踏：一万件库存、一千万用户、十秒钟。难点在于在流量峰值会让普通电商网站崩溃的情况下，保证库存计数的正确性，同时确保结算、支付和欺诈检查仍能可靠完成。

其不变量如下：

系统保证每件稀缺库存最多售出一次，仅为正在结算的用户预留库存，在放弃或超时时释放预留，并且仅在支付成功后才将预留转为销售——即便数百万用户在抢购瞬间同时涌入、机器人发起攻击、下游服务出现降级。

在 ICT5 / L6+ 级别，面试官在测试你是否能将流量吸收（等候室、队列）与库存正确性（原子递减、预留）分离，设计幂等结算，将热门商品与网站其他部分隔离，并处理机器人和支付超时。

60 秒面试框架

口述版本：

我会将限时抢购设计为一条独立的热路径，叠加在现有电商后端之上。热路径由控制准入的等候室、具备原子递减的库存预留服务、短期预留表，以及将预留转为订单的支付完成流程组成。网站其他部分——目录、搜索、推荐——保持在正常路径上，通过舱壁隔离与突发流量分开。我将明确需求，逐步介绍准入控制、库存状态转换和支付超时处理。

首先明确需求

功能性需求：

  • 特定 SKU（或一组 SKU）在固定时间上架。
  • 库存量较小（数千至数万件）。
  • 用户将商品加入购物车、结算并支付。
  • 每用户购买限额（例如每账号限购一件）。
  • 机器人防护和欺诈检查。
  • 销售窗口可提前结束（售罄）或在截止时间结束。
  • 订单确认、履行、退款。

非功能性需求：

  • 在抢购开始时支撑 100 万至 1000 万并发用户。
  • 绝不超卖。
  • 避免使电商网站其他部分崩溃。
  • 已准入用户的可接受延迟；等待用户的退避机制。
  • 用户间的公平性（粗略，非严格）。
  • 库存、支付、欺诈争议的可审计性。

以下内容超出范围，除非被特别询问：

  • 相关商品推荐引擎。
  • 长尾库存。
  • 转售市场 / 黄牛打击执行。

规模假设

领域 假设
库存 单个热门 SKU 共 10,000 件。
用户 抢购开始时涌入 100 万至 1000 万用户。
销售时长 通常为数秒至数分钟；对于稀缺性较低的商品可达数小时。
延迟目标 已准入用户看到亚秒级响应；等待中的用户看到排队位置。
机器人占比 占进入请求的 30%-70%。
支付成功率 正常条件下为 80%-95%；突发期间更低。
结算时长 每位用户 30 秒至 2 分钟。

瓶颈在于递减时刻的库存正确性，以及用户数量与库存的不对称比例。

核心产品模型

界面 体验 需求
预售阶段 商品页、倒计时、"加入队列"CTA。 可缓存、最终一致、通过 CDN 分发。
抢购开始 等候室或立即准入。 准入控制；签名 token。
预留 库存递减；购物车预留 N 分钟。 原子操作；单用户限制；带过期时间。
结算 地址、支付、确认。 幂等；支付服务商集成。
售后 订单跟踪、履行、客服。 标准电商流程。

限时抢购是预售和结算之间的短短几分钟。将其设计为具有独立隔离的热路径，是 staff 级别的关键举措。

等候室

等候室在流量到达库存层之前对其进行整形。

流程：

  1. 用户在促销开始前或开始时点击"加入队列"。
  2. 系统签发一个已签名的队列令牌： (user_id, sku_id, issued_at, signature).
  3. 客户端轮询状态；服务端返回大致排队位置和预计等待时间（ETA）。
  4. 在准入时，服务端签发一个已签名的准入令牌： (user_id, sku_id, expires_at, max_attempts).
  5. 预订 API 需要有效的准入令牌。

准入速率控制：

  • 与库存服务的安全吞吐量相匹配。
  • 根据观察到的转化率动态调整（若大量锁定未转化为下单则加速准入；若结账出现背压则减速准入）。
  • 运营人员可以暂停、减速或停止准入。

公平性：

  • 按入队时间排序，在同一入队时间窗口内引入随机化以抵御 bot 抢跑。
  • 可选抽签模型：在促销开始时对队列成员进行随机抽取。

面试用语：

等候室并非 UX 功能，而是库存写入路径的准入控制。若没有它，即使是完美的库存服务也会在数百万并发扣减请求下崩溃。

库存状态机与原子扣减

库存是一个计数器，而非逐单位的记录（与 Ticketmaster 的座位不同）。

每次锁定的状态转换：

available -> reserved -> sold
                |
                +-> expired -> available

原子扣减：

UPDATE Inventory
SET reserved_count = reserved_count + ?,
    version = version + 1
WHERE sku_id = ?
  AND total - reserved_count - sold_count >= ?
  AND status = 'on_sale'
  AND version = ?

若条件更新成功，则写入带 TTL 的锁定行；若失败（库存不足），则返回售罄响应。

对于极高吞吐量场景，使用 Redis DECRBY 配合 Lua 脚本检查剩余数量并原子预留。定期同步回权威存储。

锁定生命周期

锁定是短暂的（通常为 2-5 分钟）：

  • 创建时：设置 TTL；写入锁定行。
  • 结账成功时：锁定转为订单；状态 converted.
  • 结账失败或用户放弃时：锁定过期；库存 reserved_count 已扣减。
  • TTL 到期时：后台工作进程（或驱逐策略）释放库存。

关键：TTL 处理必须可靠。丢失的锁定会永久减少可用库存。

选项：

  • Redis 配合 EXPIRE 以及监听键空间通知、在到期时释放库存的订阅者。
  • 以过期时间为键的有序集合，工作进程弹出并处理。
  • 按时间分桶的队列。

结账与支付

用户提交结账时：

  1. 验证锁定的所有权和有效状态。
  2. 执行欺诈检测。
  3. 向支付提供商创建支付意向，包含由 hold_id 派生的幂等键。
  4. 等待提供商响应。
  5. 成功时：将锁定转为订单；将库存状态转换 reserved → sold.
  6. 失败时：释放锁定；用户可在当前会话内重试。
  7. 超时时：支付状态未知；延长锁定；对账工作进程向提供商查询。

支付超时情况（闪购的关键场景）：

  • 库存稀缺；提前释放锁定可能让另一个用户购买到第一个用户实际已购成功的商品。
  • 保守策略：在解决支付状态期间保留锁定；对账工作进程向提供商查询最终结果。
  • 权衡：长时间挂起的支付会暂时锁定该单位；我们接受此代价以避免超卖。

Bot 防护

Bot 是主要威胁。缓解措施：

  • 预售注册：基于账户的队列，配合轻量级 KYC 检查。
  • 按账户、IP、银行卡的限购。
  • 准入前进行 CAPTCHA 验证。
  • 设备指纹识别。
  • 账户年龄与历史记录要求。
  • 边缘层速率限制：对新建/匿名 IP 采取激进限制。
  • 行为分析：时序模式、鼠标移动、请求头异常。

复杂的 bot 仍会绕过。公平性保证是"尽力而为"的；完美排除 bot 不可实现。

与正常站点的舱壁隔离

闪购不能拖垮整个电商站点。

  • 促销 SKU 的商品详情页从 CDN 提供，而非回源。
  • 与促销相关的 API 路由到专用服务层。
  • 库存服务运行在隔离容量上，与常规库存服务分开。
  • 搜索和推荐在促销期间将促销 SKU 从常规索引中排除（或显示"查看促销页"链接）。
  • 支付服务为促销流量分配专用容量。
  • 数据库连接按服务进行舱壁隔离。

面试用语：

站点其余部分甚至不应察觉到促销正在进行。在每一层隔离热点 SKU 的流量。

写路径：预订

  1. 客户端 → 携带准入令牌访问 API 网关。
  2. 网关验证令牌、速率限制、bot 评分。
  3. 预订 API 检查幂等性（按用户 × 促销）。
  4. 库存服务执行原子扣减。
  5. 成功时，写入带 TTL 的锁定记录。
  6. 向客户端返回 hold_id 和过期时间。

写路径：结账

  1. 客户端提交 hold_id、支付方式、收货地址。
  2. 结账服务验证锁定归属于该用户且处于有效状态。
  3. 欺诈检测。
  4. 创建支付意向。
  5. 成功时：转换锁定；转换库存状态；创建订单。
  6. 通知用户。

读路径：浏览与状态查询

  • 预售：商品页面从 CDN 提供。
  • 促销中：同上，另加倒计时。
  • 队列状态：轮询 API 返回大致排队位置和 ETA。
  • 购买后：订单页面来自 CDN 前置的个人中心服务。

扩展与分区

数据 分区键 原因
促销库存 sku_id （通常每场促销一个 SKU） 单一热点键；专用容量。
锁定（Holds） hold_id 哈希值 均匀分布；按 id 查找。
每用户限购计数器 user_id 哈希值 每用户原子检查。
准入令牌 无状态（已签名） 无需持久化。
订单 user_id 按用户查找的 hash； order_id 按直接查找的 hash 标准订单模式。

热点 SKU 策略：

  • 单个 SKU 的库存部署在专用容量上。
  • 可将库存计数器分片为 N 个子计数器；通过聚合汇总。每个子计数器处理 1/N 的准入量。
  • 支付服务专门为促销期间进行扩容。

一致性模型

层级 一致性 原因
库存扣减 强一致（原子条件更新） 无超卖。
锁定状态 强 CAS 幂等转换。
订单创建 强一致性 资金 + 履单。
队列位置 近似值 UX 提示，非权威信息。
库存可用性展示 最终一致 最终决策在预订时做出。

故障模式与缓解措施

故障模式 缓解措施 监控指标
库存超卖。 条件扣减；幂等键。 已售数量 vs. 总量；超出时告警。
锁定泄漏（TTL 未被执行）。 可靠的过期工作进程；对已过期但仍活跃的锁定发出告警。 已过期但仍活跃的锁定数量。
支付超时。 保留锁定；对账工作进程查询提供商。 挂起支付的锁定持续时间。
等候室绕过。 API 层要求携带准入令牌。 不携带有效令牌的预订请求数。
Bot 洪泛攻击。 边缘层速率限制；CAPTCHA；按账户/IP 限流。 Bot 评分分布；拦截率。
数据库热点分片。 子计数器分片；Redis 内存库存。 每分片 QPS；p99 延迟。
结账级联失败。 将结账与站点其余部分进行舱壁隔离。 促销期间结账错误率。
售罄速度超出预期。 优雅的售罄 UX；不返回空 5xx。 售罄后的准入尝试次数。
运营人员暂停促销。 管理 API 用于停止准入；进行中的锁定要么被兑现，要么取消并退款。 促销状态变更事件。
突发流量冲击非促销流量。 边缘层速率限制；流量隔离。 促销期间正常路径延迟。

运维指标

正确性：

  • 已售数量 vs. 总量（绝不能超出）。
  • 锁定泄漏（已过期但仍占用库存）。
  • 支付未知状态的持续时间。
  • 每用户重复购买次数（必须等于限购数量）。

容量：

  • 准入速率 vs. 库存吞吐量。
  • 按原因划分的锁定创建成功/失败数。
  • 库存扣减 p99 延迟。
  • 结账转化率。

业务层面：

  • 总销售量；售罄时长。
  • Bot 拦截率 vs. 真实用户准入率。
  • 每用户购买量分布。
  • 售后退款/争议率。

多区域策略

两种方案：

单区域库存： - 一个区域持有该 SKU 的库存计数器。\n- 其他区域对用户进行排队并转发准入请求。\n- 更简单；远程区域存在延迟损耗。

预分配区域库存： - 将 10,000 件商品跨区域分配：美国 4000、欧洲 3000、亚太 3000。\n- 各区域独立运作。\n- 风险：一个区域售罄而另一个区域仍有库存；美国用户看到售罄，但亚太区域仍有余量。\n- 缓解措施：促销期间进行跨区域库存再平衡。

大多数闪购采用单区域库存，配合各区域等候室向其输送流量。

需要指出的权衡

  • 等候室降低了 UX 的即时性，但这是在高并发冲击下保证库存正确的唯一方式。
  • 预分配区域库存可降低延迟，但可能导致区域性售罄。
  • 分片计数器可扩展，但需要聚合逻辑。
  • 短锁定时间减少库存占用；长锁定时间给用户更多余裕，但会锁定商品。
  • 激进的 bot 拦截会误伤共享 IP 后的真实用户。
  • 抽签准入竞争感较弱；先到先得（FCFS）则有利于 bot。

常见陷阱与更优做法

陷阱 更优做法
对 SKU 使用单一全局锁。 使用带聚合的子计数器分片。
在加购时预留库存。 仅在结账时预留；将"在购物车中"显示为参考信息。
促销开始时先到先得（FCFS）。 带准入控制的等候室。
先付款再扣减库存。 先预留后付款；付款失败时释放库存。
不做 bot 防护。 多层防护；假设部分 bot 仍会绕过。
与正常站点共享基础设施。 对促销的库存、结账、支付进行舱壁隔离。

面试官可能深入探讨的方向

深入方向 优秀回答要点
等候室 已签名令牌；准入速率；公平性模型。
库存正确性 条件扣减；幂等性；对账。

2. 模拟面试

面试官： 设计一个闪购系统。

候选人： 我将在电商后端之上设计一条热路径。该热路径包含：用于准入控制的等候室、执行原子条件递减的库存服务、短期持有表，以及在支付完成后将持有转为订单的结账流程。其余站点保留在正常路径上，与突发流量进行舱壁隔离。不变量是不超卖。

面试官： 请介绍一下促销开始时的流程。

候选人： 数百万用户打开商品页面，该页面由 CDN 提供服务。到达促销时间时，"购买"按钮启用，用户被引导至等候室。每位用户获得一个已签名的队列令牌。服务器以库存服务能处理的速率准入用户，并签发已签名的准入令牌。获准入的用户调用预订 API，执行原子递减。成功后，写入一条持有记录，TTL 为 3 分钟。用户进入结账流程。支付完成后，持有记录转为订单；库存的已预留数量升级为已售数量。

面试官： 如何防止超卖？

候选人： 对库存行进行条件更新：仅当满足条件时才执行递减 total - reserved - sold >= quantity。这是一个原子 CAS 操作。若两个用户同时竞争，只有一个成功。为实现高吞吐量，库存状态存储在 Redis 中，通过 Lua 脚本原子地读取并递减，并定期同步回权威存储。

面试官： 为什么需要等候室？不能让所有人直接访问库存服务吗？

候选人： 不行。即使有原子递减，库存服务也有吞吐量上限。百万并发用户同时访问会压垮数据库或 Redis 集群。等候室对流量进行整形，以库存服务能承受的速率准入用户。它还提供了公平性保障，并允许我们在稀缺资源上游部署机器人防御。

面试官： 机器人怎么处理？

候选人： 多层防御：预注册时要求账号年龄和轻量 KYC。每账号、每 IP、每张卡的购买限额。准入前进行 CAPTCHA 验证。设备指纹识别。边缘层访问频率限制。我们假设部分高级机器人仍会通过；目标是降低其占有库存的比例，而非彻底消除。在促销期间实时监控并调优。

面试官： 用户获准入场、已持有库存，但支付在支付商处超时。我们该怎么办？

候选人： 这是核心正确性场景。我们不会立即释放持有，因为如果支付实际上在支付商处已成功，我们将把第一位用户刚购买的商品分配给其他人。因此持有保持不变。对账工作进程使用支付的幂等键查询支付商。确认成功则完成订单；确认失败则释放持有；超过更长超时后仍未知则上报人工审核。代价是在对账期间一个库存单位可能被锁定数分钟；我们接受这一代价以防止超卖。

面试官： 如何处理库存服务本身的突发请求？

候选人： 分片计数器。对于 10,000 件的促销，拆分为 10 个各含 1,000 的子计数器。准入请求哈希到某个子计数器。这样可以分散负载。聚合视图（sum of sub-counter remaining）告诉我们剩余总量。风险：某个子计数器售罄而另一个仍有库存；被映射到已空子计数器的用户会收到售罄响应，即使库存实际存在。缓解措施：预先打乱分配或允许跨子计数器重试。

面试官： 如何确保热点 SKU 不会拖垮整个站点？

候选人： 在每一层进行舱壁隔离。商品页面由 CDN 提供。与促销相关的 API 路由到专用服务层。该 SKU 的库存服务运行在独立容量上。在促销期间，搜索和推荐将排除该促销 SKU。促销流量的数据库连接独立部署。正常浏览的用户甚至不应感知到促销正在进行。

面试官： 公平性——先到先得还是抽签？

候选人： 各有取舍。严格先到先得有利于机器人和网络速度最快的用户。抽签感觉随意，但能化解部分机器人优势。常见折中方案：在促销开始前的时间窗口内收集排队用户，再在该窗口内随机排序；窗口结束后，迟到者按先到先得排队。运营者可按每场促销进行调整。

面试官： 多区域？

候选人： 两种方案。单区域库存：一个区域持有计数器，其他区域对用户排队并转发准入请求。正确性更简洁，但远端区域延迟较高。或者按区域预分配库存：将 10,000 件分配到各区域。风险：某区域售罄而另一区域仍有库存。可通过促销期间跨区域再平衡来缓解。大多数闪购采用单区域库存加区域等候室的方案。

面试官： 你关注哪些指标？

候选人： 正确性：已售数量 vs 总量（绝不超出）、持有泄漏、支付未知状态的时长、每用户限额违规。容量：准入速率 vs 库存吞吐量、持有创建延迟、结账错误率。业务：总售出量、售罄时间、机器人拦截率、售后退款率。

面试官： 总结。

候选人： 闪购是对稀缺库存的一次冲击。我将促销与站点其余部分进行舱壁隔离，使用等候室进行准入控制，通过分片计数器对库存执行原子条件递减以提升吞吐量，以可靠过期释放方式对库存进行短期 TTL 持有，并将支付超时视为通过对账解决的未知状态。多层机器人防御能减少但无法消除滥用。难点在于准入控制和库存稀缺条件下的支付超时处理。

3. 练习备注

五分钟版本

  1. 明确：稀缺库存、冲击流量、机器人、公平性、支付。
  2. 不变量：不超卖。
  3. 热路径与正常站点进行舱壁隔离。
  4. 等候室：使用签名令牌进行准入控制。
  5. 原子条件递减；可选分片子计数器。
  6. 短期持有，具备可靠的 TTL 释放机制。
  7. 支付超时通过未知状态 + 对账处理。
  8. 多层机器人防御；每用户限额。

一分钟结尾陈述

闪购的核心是准入控制加原子库存递减。等候室将百万用户的突发流量整形为库存服务能处理的稳定流；条件 CAS 递减保证不超卖。带短 TTL 的持有防止库存囤积；支付超时时保持库存持有直到对账完成，而非提前释放。整个促销路径进行舱壁隔离，以防热点 SKU 拖垮站点其余部分。机器人在各层进行缓解，但无法彻底消除。

最佳面试用语

  • "在每一层将促销与站点其余部分进行舱壁隔离。"
  • "等候室是库存写入路径的准入控制，而非 UX 功能。"
  • "原子条件递减是执行'不超卖'的唯一位置。"
  • "支付超时是未知状态；绝不凭猜测释放稀缺库存。"
  • "机器人只能减少，无法消除；公平性是尽力而为。"
