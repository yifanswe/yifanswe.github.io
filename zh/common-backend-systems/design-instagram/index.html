1. 基础知识与解题思路

本题考察点

Instagram 是一个媒体密集型社交信息流：照片、视频、故事和 Reels。它将 Twitter 的扇出问题与大规模媒体存储和服务成本结合在一起。难点在于在存储数 EB 级用户内容并遵守可见性规则的同时，以数百毫秒的延迟提供个性化信息流和媒体内容。

其不变量如下：

每个上传的媒体都被持久存储，可被授权的粉丝/公众查看；每个粉丝的主页信息流最终包含他们关注账号的每一篇帖子（受排名、可见性、屏蔽和策略约束）；媒体以适合每位观看者设备和网络的分辨率呈现；删除和可见性变更在几秒内全球可见。

在 ICT5 / L6+ 级别，面试官正在测试你是否能设计照片/视频上传和 CDN 服务，处理与 Twitter 相同的混合扇出，整合排名和故事功能，并推理转码管道和存储分层。

60 秒面试框架

口述版本：

我会将 Instagram 设计为一个媒体密集型信息流系统：上传管道将原始媒体写入对象存储，异步转码管道生成多种分辨率，CDN 向观看者提供服务。信息流分发采用混合扇出（普通账号推送，名人拉取），与 Twitter 相同。故事是有时间限制的，与信息流分开。排名、搜索和推荐是旁路服务。有趣的决策是上传+转码、CDN 规模、扇出、故事和排名超时。

首先明确需求

功能性需求：

  • 用户上传照片和视频（单张、轮播、Reels）。
  • 用户关注他人，查看信息流。
  • 故事：24 小时临时帖子。
  • 点赞、评论、收藏、分享、私信。
  • 按用户、话题标签、位置搜索。
  • 带有推荐内容的探索页面。
  • 包含所有帖子的个人主页。
  • 编辑工具（滤镜、裁剪）；在客户端应用。

非功能性需求：

  • 数十亿用户；每日数亿次上传。
  • 信息流读取 p99 低于 500ms，包括媒体 URL。
  • 媒体 URL → 首字节通过 CDN 低于 100ms。
  • 存储持久性达 11 个 9。
  • 故事自动过期。
  • 多地区。
  • 删除快速传播。

以下内容超出范围，除非被特别询问：

  • 直播视频流。
  • 购物结账。
  • 广告定向和竞价。

规模假设

领域 假设
用户 20 亿以上月活用户。
上传 每天 5 亿张以上照片和视频。
信息流读取数 每天 1000 亿次以上。
报道（Stories） 每天 5 亿次以上，保留 24 小时。
平均照片 200 KB-2 MB；视频 1-100 MB。
存储 EB 级；每日持续增长。
Latency（延迟） 信息流渲染低于 1 秒；媒体启动低于 100ms。
名人阈值 与 Twitter 类似；可调节。

瓶颈是媒体存储和服务成本与信息流分发延迟的结合。

核心组件

组件 职责 关键细节
上传服务 接收媒体上传；协调存储。 可恢复上传；大型视频分块处理。
对象存储 存储原始媒体。 持久；地理复制。
转码器 生成多种分辨率、格式和缩略图。 异步；队列驱动；支持竞价实例。
帖子服务 存储帖子元数据（说明、引用、位置）。 按 post_id（snowflake）分片。
关注图 与 Twitter 相同。 按 user_id 分片。
信息流服务 通过混合扇出组合主页信息流。 收件箱 + 名人拉取 + 排名。
故事服务 有时间限制的帖子；自动过期。 与信息流分开；TTL 驱动。
CDN 全球提供媒体服务。 按媒体 URL 进行边缘缓存。
搜索索引 用户、话题标签、位置、说明的倒排索引。 最终一致性。
推荐器 为每位用户预计算探索信息流。 离线 + 在线排名。
互动计数器 点赞、评论、收藏；类似 YouTube 计数器设计。 分片；异步聚合。
审核服务 图像/视频分类、NSFW 检测、策略执行。 发布前 + 持续进行。

上传流程

  1. 客户端请求 /uploads/init 附媒体元数据。
  2. 服务器返回 upload_url 。 media_id.
  3. 客户端直接以分块方式上传媒体（可恢复）到存储网关或签名 URL。
  4. 客户端调用 /uploads/{media_id}/complete.
  5. 服务器将媒体记录为 transcoder_status=pending；将转码器任务加入队列。
  6. 客户端调用 POST /posts 与 media_ids 以及说明来创建帖子。
  7. 帖子创建时状态为 status=processing ，直到媒体至少有一个可用的派生版本。
  8. 转码器生成多种分辨率（例如，视频：240p、480p、720p、1080p；照片：缩略图 + 多分辨率）。
  9. 第一个派生版本就绪后，帖子转换为 active 状态并可见。
  10. 开始扇出到粉丝收件箱（针对非名人账号）。

面试用语：

上传和转码是基础。帖子在可用派生版本存在的那一刻变得可见；后续的派生版本为带宽更高的观看者提升质量。

转码管道

尤其对于视频：

  • 将源文件拆分为片段。
  • 并行将各片段转码为每个目标分辨率。
  • 组装。
  • 生成自适应流媒体清单（HLS / DASH）。
  • 用于拖动进度条的缩略图。

存储：

  • 保留原始文件以便重新转码（例如新格式）。
  • 派生版本单独存储；可进行存储分层。
  • 引用计数；删除级联。

成本优化：

  • 使用竞价容量运行转码工作器。
  • 跳过不常用分辨率的派生版本，直到被请求（"按需"派生）。
  • 将旧原始文件分层到冷存储。

信息流分发（与 Twitter 相同的混合扇出）

相同模型：普通账号推送，名人拉取。收件箱仅存储 post_id；读取时水化以获取帖子和媒体 URL。

排名在 Instagram 中比在 Twitter 中更为核心；默认主页信息流是经过大量排名的，而非按时间顺序排列。

  • 排名服务接收候选 post_ids + 观看者上下文。
  • 在严格超时内返回已评分的 id。
  • 特征：观看者-作者亲和力、帖子互动率、时效性、媒体类型、新鲜度。

读路径：

  1. 获取收件箱 + 名人作者时间线。
  2. 应用可见性过滤器。
  3. 排名。
  4. 水化帖子元数据 + 媒体 URL。
  5. 返回结果。

媒体 URL 指向 CDN；客户端获取。

报道（Stories）

故事不在主信息流中。它们有自己的界面（应用顶部，水平滚动）。

  • 每用户故事列表（近 24 小时）。
  • 按时效性或观看者亲和力排序。
  • TTL 驱动清理：超过 24 小时的故事自动消失。
  • 观看者状态追踪：谁查看了哪个故事（用于"已查看"指示器）。
  • 高浏览率：名人故事的计数器分片。

存储：

  • 故事存储在按 user_id 分区的独立表中。
  • 后台任务删除过期故事。
  • 如果用户选择加入（精选故事），媒体可能保留超过 24 小时。

CDN 与媒体服务

  • 媒体通过拥有全球边缘节点（POP）的 CDN 提供服务。
  • URL 签名用于短期访问（特别是私人账号）。
  • 视频采用自适应码率。
  • 缓存命中率对成本至关重要；目标超过 90%。
  • 热门内容的按地区预热缓存。

扩展与分区

数据 分区键 原因
帖子 post_id hash（snowflake） 均匀分布。
媒体 media_id 哈希值 直接查找。
报道（Stories） user_id hash，按时间分桶 每用户临时列表。
信息流收件箱 user_id 哈希值 按用户存储。
搜索索引 Token 分片 查询驱动。
互动计数器 post_id hash，热帖使用子计数器 热帖。
关注图 user_id 哈希值 与 Twitter 相同。

一致性模型

层级 一致性 原因
帖子创建 强一致性 真实数据来源。
媒体元数据 上传完成时强一致性 水化所必需。
信息流收件箱 最终一致 有界时效性。
互动计数器 最终一致 UI 提示。
故事可见性 TTL 边界处强一致性 必须按时过期。
搜索索引 最终一致 秒级延迟。

多区域策略

  • 对象存储地理复制。
  • 原始文件在主要地区；派生版本全球复制。
  • 每个地区的信息流收件箱缓存。
  • 写操作按用户归属主要地区。
  • CDN 边缘节点遍布全球。

故障模式与缓解措施

故障模式 缓解措施 监控指标
视频转码失败。 重试；降级到更低质量；将帖子标记为 processing 更长时间。 转码失败率。
热帖使互动计数器过载。 子计数器分片（与 YouTube 点赞相同）。 单计数器 QPS。
CDN 对病毒式内容的缓存未命中。 预热；多级缓存。 源站出口流量；CDN 命中率。
故事过期任务滞后。 读取时以 TTL 作为最终关卡强制执行。 过期但仍可见的帖子数量。
上传未完成（用户关闭应用）。 可恢复上传；陈旧的部分上传被垃圾回收。 未完成上传的时长。
收件箱缓存丢失。 从作者时间线 + 关注图重建。 缓存未命中率。
内容审核未能发现违规内容。 异步重新分类；用户举报。 内容审核延迟。
名人帖子扇出风暴。 混合阈值；背压；拉取路径。 扇出积压。
存储成本增长。 旧原始文件的冷存储分层；基于引用计数的去重。 存储增长速率。
多区域复制延迟。 异步；有界；告警。 复制延迟。

运维指标

正确性：

  • 删除后仍可见的过期帖子。
  • 屏蔽/可见性违规。
  • Story 已过期但仍显示的计数。
  • 转码失败率。

容量：

  • 上传 QPS；转码队列深度。
  • Feed 读取 p99 延迟。
  • CDN 命中率，源站出口流量。
  • 互动计数器延迟。

业务层面：

  • DAU；每用户发帖数；互动率。
  • Stories DAU。
  • Explore 页面互动情况。

需要指出的权衡

  • 预先生成所有分辨率浪费存储空间；按需生成首次查看较慢。
  • 排名提升互动率但增加延迟；严格超时保护 Feed。
  • 按用户物化收件箱成本较高，但加快了读取速度。
  • Stories 24 小时 TTL 是产品核心特性；过期清理必须可靠。
  • 大规模重媒体内容主导基础设施成本；CDN 策略至关重要。
  • 隐私控制（亲密朋友、私密账号）需要在读取时进行过滤。

常见陷阱与更优做法

陷阱 更优做法
同步转码，完成后才让帖子可见。 帖子在首个派生版本就绪后立即可见；更高质量版本稍后上线。
将媒体存储在帖子表中。 独立媒体存储；帖子中保存引用。
单层 CDN 缓存。 多层架构：设备 → 边缘 POP → 区域节点 → 源站。
将 Stories 当作普通帖子处理。 独立界面；TTL；过期任务 + 读取时验证。
排名无超时限制。 严格超时 + 时间倒序兜底。
原始媒体永久保存在热存储中。 随时间推移分层至冷存储。

面试官可能深入探讨的方向

深入方向 优秀回答要点
上传 + 转码 可续传上传；异步流水线；首个派生版本就绪后即可见。
Feed 扇出 混合模式，类似 Twitter；排名带超时限制。
报道（Stories） 独立界面；TTL；过期 + 读取时验证。
CDN 多层架构；私密内容使用签名 URL；热门内容预热。
互动计数器 子计数器分片；定期重新统计。
存储分层 热存储存放派生版本；冷存储存放原始文件；引用计数。

Staff 级别回答检查清单

  • 独立的上传、转码、帖子元数据、Feed、Stories 界面。
  • 首个派生版本就绪后立即可见。
  • Feed 使用混合扇出。
  • Stories 带 TTL + 读取时验证。
  • CDN 多层架构，关注命中率。
  • 互动计数器使用子分片。
  • 搜索和推荐作为旁路系统。
  • 多区域架构，每用户有主区域 + CDN 边缘节点。

面试用语

Instagram 是 Twitter 加上媒体处理流水线。扇出采用相同的混合模式；特殊之处在于上传转码流程——它会延迟全质量发布但允许快速发布，以及带有 24 小时 TTL 的 Stories。主要基础设施成本是媒体存储和 CDN 服务；主要正确性问题是 Feed 投递和可见性过滤。

2. 模拟面试

面试官： 设计 Instagram。

候选人： 我将其设计为一个重媒体社交 Feed：上传流水线将媒体写入对象存储；转码器异步生成多种分辨率；帖子元数据与媒体分离；Feed 投递使用类似 Twitter 的混合扇出；Stories 是独立的短暂界面。不变量为：已上传媒体须持久化，帖子在至少一个派生版本就绪后即可见，Feed 在数百毫秒内从关注账号投递排序内容。

面试官： 带我了解一下上传流程。

候选人： 客户端请求上传 URL 和 media_id。通过签名 URL 直接分块上传至存储。完成后调用 complete 接口。服务端将媒体标记为 transcoder_status=pending 并将转码任务入队。客户端发布引用该 media_id 的帖子元数据。帖子以 processing 状态创建，在任意派生版本就绪后变为 active ——对于照片通常在数秒内完成，视频则需更长时间。更高质量的派生版本稍后上线，但帖子已经可见。

面试官： 为什么要在首个派生版本就绪后立即可见？

候选人： 有两个原因。第一，感知发布延迟。用户希望帖子立即出现，而不是等待 30 秒的转码。第二，最低质量版本足以在 Feed 预览中渲染；高质量版本只在用户全屏查看且带宽充足时才需要。我们优化"足够显示"并异步提升质量。

面试官： Feed 投递——与 Twitter 相同？

候选人： 是的，混合扇出。普通账号将 post_id 推送到关注者收件箱；大V 在读取时拉取。Feed 服务合并并排名。排名比 Twitter 更激进——Instagram 默认为重度排名而非时间倒序。排名器有严格超时，并以时间倒序作为兜底。

面试官： Stories？

候选人： 独立界面。Stories 表按 user_id 分区，带有 expires_at = posted_at + 24h。应用顶部的 Story 栏显示关注用户的最新 Stories，按时间或亲密度排序。后台任务删除过期 Stories，读取路径将过期内容作为最终过滤条件。Story 媒体经过相同的转码器处理。

面试官： 一位大V 发布了一条被数百万人观看的病毒 Story。计数器如何避免过载？

候选人： 与 YouTube 点赞相同的模式。Story 的观看计数器被分片为子计数器；观看量哈希到随机子计数器进行原子递增。展示时聚合求和。如果需要"已被谁看过"指示器，则在独立的按 Story 按观看者的表中记录每位观看者的"已看"状态；对于非常热门的 Stories 则聚合为计数。

面试官： 存储成本？

候选人： EB 级媒体数据。我们积极分层。热层（近期帖子）使用快速对象存储。温层（仍有访问量的旧帖子）使用更廉价的存储。冷层（超过 X 个月的原始文件）使用归档存储。派生版本体积较小，在热层保留更长时间，因为它们通过 CDN 提供服务。引用计数处理删除：当没有帖子引用某个 media_id 时，即可清理。

面试官： CDN 策略？

候选人： 多层架构。设备缓存（浏览器/应用）→ 边缘 POP → 区域缓存 → 源站。命中率目标在 90% 以上。私密账号使用签名 URL。热门内容通过预测机制预热边缘缓存。视频使用 HLS/DASH 清单实现自适应码率。

面试官： 搜索？

候选人： 独立搜索服务消费帖子事件。对标题 token、话题标签、位置、用户名建立倒排索引。按 token 分片。最终一致——帖子在数秒内变为可搜索。查询扇出到相关分片、评分、返回。

面试官： Explore 页面？

候选人： 离线 + 在线推荐流水线。每日离线任务通过协同过滤、嵌入相似度和热门趋势为每位用户生成候选集。在线排名器在请求时对候选进行评分，有严格超时。候选集可达数千条；排名器返回前 50 条。

面试官： 删除？

候选人： 与 Twitter 相同。将帖子标记为 deleted；Feed 读取路径的数据填充看到删除标记后跳过。收件箱条目保留但不显示任何内容。媒体引用计数递减；存储清理为异步操作。删帖用户的 Feed 立即反映删除；关注者在数秒内看到变化。

面试官： 多区域？

候选人： 对象存储进行跨地域复制。原始文件保存在用户主区域；派生版本全球复制以加速 CDN 预热。帖子元数据进行复制。写操作指向每用户的主区域。CDN 在全球各地本地提供服务。跨区域复制延迟为数秒；用户可能短暂看到跨区域略有过时的 Feed，这是可接受的。

面试官： 总结。

候选人： Instagram 是 Twitter 加上媒体处理流水线。上传写入原始文件，异步转码器生成派生版本，帖子在首个派生版本就绪后可见。Feed 使用混合扇出；Stories 是带 TTL 的独立短暂界面。CDN 多层架构提供媒体服务；存储按年龄分层。互动计数器使用子分片。搜索和 Explore 为旁路系统。主要成本是媒体存储和 CDN；主要正确性问题是 Feed 投递和可见性。

3. 练习备注

五分钟版本

  1. 澄清：媒体类型、Feed、Stories、规模。
  2. 不变量：媒体持久化，首个派生版本就绪后可见，Feed 正常投递。
  3. 上传 + 异步转码；首个派生版本就绪后可见。
  4. 帖子元数据与媒体分离。
  5. Feed 混合扇出，类似 Twitter；排名带超时限制。
  6. Stories 独立界面，带 TTL。
  7. CDN 多层架构；存储按年龄分层。
  8. 互动计数器使用子分片；搜索/Explore 为旁路系统。

一分钟结尾陈述

Instagram 是重媒体社交 Feed。上传写入原始文件；异步转码器生成多种分辨率；帖子在首个派生版本就绪后可见。Feed 使用混合扇出——普通账号推送，大V 拉取。Stories 是带 24 小时 TTL 的独立界面。CDN 多层架构在边缘提供媒体服务。存储将内容按年龄分层至更廉价的层级。互动计数器对热门帖子使用子分片。主要成本是媒体；主要正确性问题是 Feed 可见性。

  • "帖子在首个派生版本就绪后即可见；更高质量版本稍后异步上线。"
  • "Feed 是混合扇出，结构与 Twitter 相同。"
  • "Stories 是带 TTL 和读取时验证的独立界面。"
  • "CDN 多层架构命中率 90% 以上，这是大规模媒体经济可行的关键。"
  • "互动计数器对热门帖子使用子分片；定期重新统计保证准确性。"
