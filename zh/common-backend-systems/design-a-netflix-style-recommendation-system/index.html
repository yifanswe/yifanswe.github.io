1. 基础知识与解题思路

本题考察点

Netflix 风格的推荐系统负责决定在主页展示哪些内容、以何种顺序展示。难点在于：在延迟预算内，为数亿用户在拥有 1 万部作品的目录中计算个性化排名，同时兼顾召回率、延迟、数据新鲜度与 A/B 实验。

其不变量如下：

每次用户请求主页时，在延迟预算内返回一份经过排名的行与作品列表，来源于个性化候选集加在线排序器；用户信号（观看历史、浏览行为、评分）反馈至模型重训练；A/B 实验将模型变更隔离以便进行可衡量的评估。

在 ICT5 / L6+ 级别，面试官考察的是候选人能否设计离线与在线推荐流水线、候选生成（协同过滤、Embedding、热门内容）、排序架构、特征存储、A/B 基础设施，并能权衡数据新鲜度与成本。

60 秒面试框架

口述版本：

我会设计一个两阶段推荐系统：离线批处理流水线预计算每个用户的候选集（协同过滤、Embedding 相似度、热门内容），以及在请求时对候选集进行个性化打分的在线排序器。特征存储提供离线与在线特征。A/B 实验无处不在。关键决策包括：候选生成、在线推理延迟、特征新鲜度与实验设计。

首先明确需求

功能性需求：

  • 为每个用户生成带行列分类的个性化主页。
  • 每日更新推荐，并根据互动信号实时调整。
  • 支持多种行类型：热门、继续观看、因为您观看了 X、为您推荐。
  • 新用户的冷启动处理。
  • 模型与 UX 的 A/B 测试。

非功能性需求：

  • 主页渲染时间低于 1 秒。
  • 推荐每日刷新，并叠加实时信号。
  • 用户数 2 亿以上。
  • 作品目录共 1 万部。

以下内容超出范围，除非被特别询问：

  • 搜索排序的具体实现。
  • 自动播放决策。
  • 预告片选择。

规模假设

领域 假设

离线流水线

每日（或每小时）批处理：

  1. 从过去 N 天接入互动事件 。
  2. 更新用户特征： 基于近期观看记录的偏好 Embedding；近期类型、语言、完播模式。
  3. 更新作品特征： 热度、完播率、观看趋势。
  4. 训练模型： 矩阵分解、深度协同过滤、用于 Embedding 的双塔网络。
  5. 计算每个用户的候选集： - 协同过滤：相似用户观看过的作品。\n - 基于内容：与近期观看作品相似的作品（通过 Embedding 余弦相似度）。\n - 用户所在地区的热门内容。\n - 基于近期观看记录锚定的"因为您观看了 X"行。
  6. 将候选集存入离线特征存储。

每个用户通常跨多种行类型获得数千个候选作品。

在线排序器

每次请求：

  1. 从离线特征存储获取用户候选集。
  2. 获取实时特征（刚刚观看的内容、当前会话活动）。
  3. 使用在线模型（通常为深度神经网络）对候选集打分。
  4. 应用多样性、去重与新鲜度策略。
  5. 选择行类型并对作品排序。
  6. 返回结果。

延迟预算：总计低于 500ms。排序器推理目标为 100-200ms。

多阶段排序：\n- 轻量级预排序将每行候选集剪枝至 top-K。\n- 重量级 ML 排序器对 top-K 进行打分。

面试用语：

两阶段设计：离线流水线从数十亿事件中预计算每个用户的候选集；在线排序器结合实时特征对 top 候选集打分。离线负责处理规模，在线负责保证新鲜度。

候选生成策略

策略 如何生成
协同过滤 "与您相似的用户观看了……" 矩阵分解或双塔网络。
基于内容 与近期观看作品的 Embedding 相似度。
热门内容 地区或类别中的热门作品。
"因为您观看了 X" 锚定于某部近期观看的具体作品。
编辑/人工精选 人工挑选的行。
新上线内容 目录中的近期新增作品。
继续观看 未完成的作品。
个性化搜索 查询驱动。

每种策略都贡献候选集；主页组合器进行混合搭配。

特征存储

两层架构：

离线层： 每日/每小时批量更新。用户偏好 Embedding、作品热度、历史互动数据。数据量大；批量查找。

在线层： 实时信号。当前会话、刚刚观看、刚刚浏览的内容。低延迟点查询。

特征存储提供训练与服务的一致特征（训练时使用时间点正确性以避免数据泄漏）。

A/B 实验

在 Netflix 无处不在。

  • 实验服务通过对 user_id 进行稳定哈希将用户分配至实验分组。
  • 每个分组测试一次模型变更或 UX 变更。
  • 参与事件标记了实验 ID 与变体。
  • 每日分析：对比 A 与 B 在关键指标上的表现（观看时长、完成率、留存率）。
  • 统计显著性检验；发布胜出变体。

多个实验并发运行；清晰的阶乘设计与正交变体。

冷启动

新用户没有观看历史。

  • 回退策略：推荐用户所在地区的热门内容。
  • 引导调查："选择 3 个你喜欢的内容" → 初始化偏好。
  • 快速收敛：前几次观看在数小时内影响候选生成。

参与行为捕获

客户端流式上报事件：播放、暂停、跳过、完成、评分、浏览、悬停。事件流入：- 离线流水线（次日候选集更新）。- 在线特征存储（实时信号）。- 实验结果指标。

多样性与去重

同一行内：避免展示相似内容（多部续集连续出现）。

跨行：避免同一内容出现在多行中。

多样性通过对排序输出进行后处理注入：带多样性惩罚的贪婪选择。

扩展与分区

数据 分区策略
候选集 user_id 哈希
用户档案 user_id
内容元数据 title_id
特征存储 user_id 与 title_id
参与事件 user_id + 时间
实验分配 user_id

一致性模型

层级 一致性
候选集 最终一致性（每日刷新）
在线特征 快速最终一致性（秒级）
实验分配 强一致性（稳定哈希）
参与日志 至少一次
模型服务 每个实验稳定

多区域策略

  • 用户主区域用于个人资料。
  • 候选集复制到最近的区域。
  • 每个区域部署在线排序器。
  • 参与事件按区域存储，并进行跨区域复制。

故障模式与缓解措施

故障模式 缓解措施 监控指标
排序器速度慢。 严格超时；回退到候选集顺序。 排序器 p99 延迟。
特征存储数据过期。 使用缓存数据；添加时间戳标记。 特征新鲜度。
冷启动。 热门内容 + 引导种子。 冷启动请求率。
模型回退。 A/B 测试；指标下降时自动回滚。 参与度变化量。
候选集缺失。 回退到热门度排序。 候选集缺失率。
事件流水线延迟。 背压；告警。 流水线延迟。
实验泄漏。 稳定哈希；每实验审计。 跨变体污染。
多样性失效。 后处理检查。 重复内容率。
跨区域复制。 异步；有界。 延迟。
隐私违规。 每用户退订；数据匿名化。 隐私审计。

运维指标

正确性：

  • 冷启动请求率。
  • 过期特征使用。
  • 多样性违规。
  • 实验污染。

容量：

  • 排序器 p99 延迟。
  • 特征存储 QPS。
  • 候选集新鲜度。

业务层面：

  • 每用户观看时长。
  • 完成率。
  • 每行点击率。
  • 留存管理。

需要指出的权衡

  • 离线候选集刷新成本低；在线排序器是延迟瓶颈。
  • 更深层的模型排序效果更好，但延迟和成本更高。
  • 更丰富的特征存储有助于提升效果，但会增加服务成本。
  • 激进个性化与过滤泡沫风险之间的权衡。
  • 多样性会降低表面相关性，但能提升长期参与度。
  • 冷启动回退策略与强制引导调查之间的权衡。

常见陷阱与更优做法

陷阱 更优做法
所有内容使用单一排序器。 两阶段：离线候选集 + 在线排序器。
每次请求重新计算候选集。 每日离线候选集。
单一行类型。 多种行类型，来自不同来源。
忽略 A/B 测试。 全面的实验体系。
过期特征存储。 在线 + 离线分层。
排序器异常时阻塞请求。 回退到候选集顺序。

面试官可能深入探讨的方向

深入方向 优秀回答要点
候选生成 多来源：协同过滤、内容相似度、热门度。
在线排序器 多阶段；特征存储；严格延迟要求。
特征存储 离线 + 在线分层；时间点正确性。
A/B 测试 稳定哈希；变体标记；指标分析。
冷启动 热门内容 + 引导种子。
多样性 带惩罚的后处理。

Staff 级别回答检查清单

  • 两阶段：离线候选集 + 在线排序器。
  • 多来源候选生成。
  • 离线 + 在线分层特征存储。
  • 多阶段排序，严格延迟控制。
  • A/B 测试全面覆盖。
  • 冷启动回退。
  • 多样性后处理。
  • 多区域部署，用户归属主区域。

面试用语

Netflix 风格的推荐系统是两阶段的：离线流水线从数十亿事件中为每个用户预计算候选集；在线排序器利用实时特征对候选集排分。多个候选来源（协同过滤、内容 embedding、热门度）对应不同的行类型。特征存储分离线和在线两层。A/B 实验体系全面覆盖。最严苛的性能约束是在线排序器的延迟；大规模问题由离线层承担。

2. 模拟面试

面试官： 设计 Netflix 推荐系统。

候选人： 两阶段。离线批处理流水线每天从多个来源（协同过滤、内容 embedding、热门度）为每个用户预计算候选集。在线排序器在请求时利用实时特征对候选集排分。特征存储分离线和在线两层。A/B 实验体系全面覆盖。核心不变量是：亚秒级个性化首页，候选集每日刷新，并叠加实时信号。

面试官： 带我走一遍首页的服务流程。

候选人： 用户请求 /home。后端从特征存储中查找用户的候选集——多个集合，每种行类型一个（趋势、为你推荐、因为你看了 X，等等）。每个集合包含数千个内容。在线排序器使用离线特征（用户偏好 embedding、内容特征）加上在线特征（当前会话、刚看完的内容）对每行的候选集打分。首页编排器选取行并对内容排序，同时应用多样性和去重策略。返回分页结果。参与事件回流到流水线中。

面试官： 候选生成策略。

候选人： 多种。通过矩阵分解或双塔网络进行协同过滤："与你相似的用户看了 X。"基于内容：我最近看过的内容与目录之间的 embedding 相似度。区域热门度：我所在地区的热门内容。"因为你看了 X"，锚定在最近的观看记录上。新上映内容。继续观看。编辑推荐 / 精选内容。每种行类型来自一个或多个候选生成器。混合方式是设计决策的核心。

面试官： 在线排序器。

候选人： 多阶段。廉价的预排序将候选集剪枝至 top-K（约 100 个）。昂贵的 ML 排序器（深度模型）用丰富特征对 top-K 打分。返回有序列表。严格超时（约 150ms）；超时时回退到预排序顺序。尽可能预计算特征；仅在绝对必要时在线计算。

面试官： 特征存储。

候选人： 两层。离线层：每日批量更新。包括用户偏好 embedding、内容热门度、历史参与汇总。用于排序器批量推断的批量查询。在线层：实时信号——当前会话活动、刚看完的内容。亚毫秒级点查询。关键细节：训练/服务一致性——服务时使用的特征必须在训练时可复现（时间点正确性），以避免特征泄漏。

面试官： A/B 测试。

候选人： 每次模型变更都经过 A/B 测试。实验服务通过对 user_id 进行稳定哈希将用户分配到各变体。每个变体测试一个模型或 UI。参与事件标记了实验 ID 与变体。每日分析：对比 A 与 B 在关键指标上的表现（观看时长、完成率、留存率）。统计显著性检验决定胜出者。多个实验并发运行；正交设计减少干扰。

面试官： 冷启动。

候选人： 新用户没有观看历史。引导调查：选择几个你喜欢的内容——初始化偏好特征。回退到该地区的热门内容。前几次观看后，标准流水线接管。一两个会话后，推荐开始个性化。

面试官： 过滤泡沫问题？

候选人： 多样性通过对排序输出进行后处理注入。带多样性惩罚的贪婪选择：后续选中的内容若与已选内容相似则会被惩罚。编辑推荐行会展示个性化预测范围之外的内容。我们监控多样性指标；如果某用户的推荐过于单一，则强制引入探索性内容。

面试官： 新增了一个内容。

候选人： 该内容冷启动。通过内容 embedding（由类型、演员、剧情简介计算得出）生成基于内容的候选集。初始热门度来自营销信号。有一定观看量后，协同过滤信号逐步建立。编辑推荐可能有助于冷启动引导。

面试官： 特征新鲜度。

候选人： 离线特征每日更新；在线特征在数秒内更新。"刚看完"信号是在线信号，会影响紧接着的下一次排序。每日刷新更新偏好 embedding 和候选集。对于非常新鲜的信号（刚点击了某内容），在线特征存储以亚秒级延迟承载这些信号。

面试官： 多地区。

候选人： 用户主区域用于个人资料和历史特征。候选集复制到附近区域。每个区域部署在线排序器。参与事件按区域存储，并进行跨区域复制以用于全局模型训练。

面试官： 总结。

候选人： 两阶段推荐系统：离线批处理从多个来源（协同过滤、内容 embedding、热门度）预计算每用户候选集；在线排序器利用实时特征和个性化对候选集排分。特征存储分离线和在线两层，保证训练/服务一致性。A/B 实验体系全面覆盖。冷启动通过引导 + 热门回退实现。多样性在后处理阶段注入。最严苛的性能约束是在线排序器延迟；最难的正确性问题是特征新鲜度与成本的权衡。

3. 练习备注

五分钟版本

  1. 澄清：规模、延迟、新鲜度要求。
  2. 不变量：个性化；亚秒级；可 A/B 测试。
  3. 两阶段：离线候选集 + 在线排序器。
  4. 多来源候选集：协同过滤 + 内容相似度 + 热门度。
  5. 特征存储分离线和在线两层。
  6. 多阶段排序，设置超时。
  7. 冷启动回退。
  8. A/B 测试全面覆盖；多样性后处理。

一分钟结尾陈述

Netflix 风格的推荐系统是两阶段的。离线层每天从多个来源预计算每用户候选集。在线排序器在请求时利用实时特征对候选集排分。特征存储分离线和在线两层，保证训练/服务一致性。A/B 实验体系全面覆盖。冷启动通过引导 + 热门回退实现。多样性在后处理阶段注入。多区域部署，用户归属主区域。最严苛的性能约束是在线排序器延迟。

最佳面试用语

  • "两阶段：候选集离线每日预计算；在线排序器在请求时打分。"
  • "多个候选生成器对应不同的行类型。"
  • "特征存储分离线和在线两层；训练/服务一致性至关重要。"
  • "A/B 测试全面覆盖；每次模型变更都经过测试。"
  • "多样性在后处理阶段注入，防止过滤泡沫。"
