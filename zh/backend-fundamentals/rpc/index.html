关于 RPC（远程过程调用）及更广泛的服务间通信基础设施的技术参考：gRPC、Apache Thrift、REST（表述性状态转移）、GraphQL、Connect/Twirp，以及使上述任何技术在集群规模下可靠运行的服务网格原语（Envoy Sidecar、mTLS、xDS 配置流式传输）。这不是某个具体 Mesh 产品的文档；而是关于底层技术本身的文档——它是什么、设计上提供哪些保障、硬问题在哪里，以及实际上线的字节内容。文中以支付、搜索、信息流、ML 推理、移动端和公共 API 的使用场景作为示例说明。

深度来自于： Protobuf 线格式的字节级解析, HTTP/2 帧与流多路复用，以及 级联故障机制 （重试风暴、队头阻塞、连接池耗尽），无论选择哪种框架，这些问题都会在任何 RPC 基础设施中出现。

§1. RPC 是什么，以及它不是什么

RPC 是一类服务间通信方式，其中一个进程通过类型化 Schema、显式错误码、截止时间，以及（通常）内置于线协议中的认证机制，以同步（或流式）请求-响应语义调用另一个进程中的过程。 最后这句话正是 RPC 区别于"只是发送 HTTP 并解析响应体"的关键所在。RPC 框架将操作契约内嵌到协议本身。

使某物成为 RPC 框架而非单纯传输的三个属性：

  1. Schema 即契约。 一个 .proto （Protobuf）、 .thrift或 OpenAPI 文档定义了操作、消息类型和错误空间。客户端和服务端均从同一 Schema 编译代码，线格式字节依据该 Schema 进行解释。这与原始 HTTP 有本质区别——在原始 HTTP 中，响应体是不透明字节，契约是"文档里写的那些内容"。

  2. 错误码有类型。 一次 gRPC 调用会返回 OK, DEADLINE_EXCEEDED, UNAVAILABLE, PERMISSION_DENIED等——17 个标准错误码。每个错误码都告诉客户端一个 不同的 恢复动作。相比之下，原始 HTTP 只有约 50 个状态码，基本可归并为"200 / 4xx / 5xx"，而实际含义则隐藏在 JSON 错误体中，且各服务的格式各异。

  3. 截止时间，而非超时。 截止时间是一个绝对时间戳（"必须在 UTC 14:00:00.500 之前完成"），并在调用链中逐跳传播。每一跳减去已消耗的时间，再将更小的截止时间向下游传递。逐跳超时不具备传播性，是许多持续数小时故障的根源——边缘节点已放弃用户请求后，叶子服务仍在继续执行。

RPC 在技术栈中的位置。 RPC 是微服务架构中的东西向通信基础设施。一个请求从边缘进入（负载均衡器、API 网关或 CDN），边缘执行部分逻辑后调用 5–50 个内部服务，每个服务再调用 5–50 个服务，每个服务再与数据库、缓存、搜索索引和 ML 特征存储交互。RPC 层是该调用树的 连接组织 。它不是数据库，不是队列，而是一个无状态服务向另一个无状态服务请求答案的手段。

与相邻技术类别的区分:

  • 与消息队列（Kafka、RabbitMQ、Pulsar）。 队列是 异步 。 持久的。你生产一条消息，某个消费者最终将其消费。故障语义是"至少一次投递，支持重放"。RPC 是 同步 。 非持久的。如果服务端在调用过程中崩溃，消息将丢失；客户端必须向另一个实例重试。对于可以容忍秒到分钟级延迟且必须在崩溃时保留数据的扇出任务，使用队列。对于"我需要在当前用户请求中立即得到答案"的场景，使用 RPC。

  • 与原始 HTTP。 原始 HTTP 提供带状态码和请求头的传输层。RPC 在此之上增加了 Schema、错误分类、截止时间、代码生成和流式传输。你可以在原始 HTTP 之上构建"一个 RPC 框架"——REST 大致上就是这样——但你最终会在用户代码中重新实现上述每个原语，而且十个团队会有十种不同的实现方式。

  • 与 REST。 REST 以资源为中心（GET /users/42, POST /payments）且原生基于 HTTP。它是一种使用 HTTP 的风格，并非独立协议。REST 适合公共 API（浏览器、第三方、用 curl进行 CLI 调试）。它不适合内部高频路径的集群规模场景，原因我们将在 §23 中详细分析。

  • 与 GraphQL。 GraphQL 是基于类型化 Schema 的查询语言，针对客户端驱动的响应结构进行了优化（移动应用可以精确请求所需字段）。它最适合用于 API 网关 / BFF（Backend For Frontend）层，而非作为内部东西向通信基础设施。在内部使用时，GraphQL 会成为一个臃肿的转换层，却没有任何实质性优势——内部调用方清楚地知道自己需要哪些字段。

RPC 不擅长的场景:

  • 长时间运行的操作（工作流）。如果你的"RPC"要运行 10 分钟，你实际上是在 RPC 之上构建了一个工作流引擎，并将永远与重连、幂等性和进度追踪问题作斗争。请改用工作流原语（Temporal、Cadence、Step Functions、durable saga）。
  • 持久化事件分发。RPC 是一次性交接，而非记录。请使用日志（Kafka）。
  • 批量数据移动。通过 gRPC 流式传输 GB 级数据虽然可行，但效率低下；请使用直接对象存储或数据管道。
  • 消费者数量未知或频繁变化的场景。RPC 是点对点的；pub/sub 是多对多的。

§2. 内在保证：RPC 在设计上提供了什么

这是一份清醒的清单，列明该技术免费提供了什么，以及它不提供什么（这样你就不会对它抱有不切实际的期望）。

设计上已提供:

保证 机制
类型化消息 Schema in .proto / .thrift；codegen 在每种支持的语言中生成类型化结构体。Wire 字节根据 schema 解码。
带版本的错误码 gRPC 定义了 17 个具名状态码（OK, CANCELLED, UNKNOWN, INVALID_ARGUMENT, DEADLINE_EXCEEDED, NOT_FOUND, ALREADY_EXISTS, PERMISSION_DENIED, UNAUTHENTICATED, RESOURCE_EXHAUSTED, FAILED_PRECONDITION, ABORTED, OUT_OF_RANGE, UNIMPLEMENTED, INTERNAL, UNAVAILABLE, DATA_LOSS）。每个状态码规定了不同的客户端行为。
Deadline 传播 grpc-timeout metadata header 是一个相对时长；每一跳减去已耗时间并转发更小的值。已超过 deadline 的调用会立即失败，不再分发。
流式 gRPC 支持 unary、server-streaming、client-streaming 和 bidi-streaming 模式。Wire 分帧（HTTP/2 DATA 帧 + 每条消息的长度前缀 + TRAILERS）使其成为原生特性，而非附加功能。
多路复用 HTTP/2 允许一条 TCP 连接在帧级别交错承载数百至数千个并发 RPC。无需每次调用建立连接。
每流流量控制 HTTP/2 WINDOW_UPDATE 帧意味着慢速消费者会自然地对发送方施加背压，同时不会饿死同一连接上的其他流。
Schema 演进规则 Protobuf 的字段编号约定：安全地添加字段，保留已删除的编号，永不复用编号。旧二进制会跳过未知 tag。由构造保证前向和后向兼容性 前提是你遵守规则.
紧凑的 Wire 格式 Protobuf 和 Thrift TCompact 的大小通常是等效 JSON 的 30–35%，编解码速度快约 10 倍。

未提供——必须在上层叠加:

未提供 朴素预期为何会落空
持久性 RPC 是一次性交接。如果服务器在确认后、持久化前崩溃，操作就会丢失。想要持久性？在处理器返回前持久化到数据库，或使用队列。
跨服务的事务语义 向服务 A 发出的成功 RPC 并不会"提交"任何跨服务操作。两阶段提交、saga 和 outbox 模式必须在上层叠加。RPC 是这些模式所 构建的原语之一，而非替代方案。
自动重试安全性（幂等性） 框架会欣然重试 POST /payment 并重复计费。幂等性是 应用层的 职责（幂等性键、内容寻址标识符）。RPC 重试仅在幂等调用上才是安全的。
服务发现 gRPC 不会告诉你服务 B 在哪里。你需要在其下方部署一个服务发现层（DNS、Consul、etcd、Eureka、Kubernetes endpoints）。
认证 / 鉴权 gRPC 在 header 中携带凭据（authorization，mTLS 客户端证书），但不强制执行 authn/authz。那是服务器或 sidecar 中的策略代码。
跨区域路由或故障转移 RPC 只执行你告诉它的操作。多区域故障转移、感知位置的路由以及"如果区域 A 宕机则发往 B"都是更高层的关注点。
应用层背压 HTTP/2 流量控制只覆盖单跳。跨 5 层扇出的背压是你的设计问题（令牌预算、准入控制）。
精确一次投递 与所有 RPC 一样，gRPC 是尽力而为加重试。精确一次是应用层的事（去重 + 幂等性键 + 排序）。

总结： RPC 框架给你提供了管道、schema、错误分类体系、deadline 和流式传输。其他一切——持久性、事务、幂等性、扇出、故障转移——都需要你在上层叠加实现。

有据可查的选择:

  • gRPC + Protobuf 是东西向内部流量的默认选择 在任何对 wire 效率和 CPU 效率有要求的规模下。紧凑格式、强制 schema、代码生成、deadline 传播和流式传输的组合无出其右。Google、Square、Lyft、Cloudflare、Dropbox，以及越来越多的 LinkedIn，都在热点内部路径上使用它。

  • REST + JSON 是南北向流量的正确答案 （公开 API、可被浏览器调用的端点、合作伙伴集成）。Stripe 的公开 API、GitHub 的 REST API、Twilio 的 API——全都是 REST + JSON，因为调用方是某个使用 curl 和 Python 的随机初创公司。

  • GraphQL 适合 BFF（Backend For Frontend）层 一个客户端（移动应用）只需要它想要的字段而不是更多，且不同客户端需要不同数据结构的场景。Shopify 的店面 API、GitHub 的 GraphQL API、Facebook 的移动数据层。在内部场景中并不适合，因为调用方清楚地知道自己需要什么，过度获取根本不是问题。

  • Thrift 是 Facebook、Twitter 和 Uber 的遗留答案。 在大多数工作负载中，功能上等同于 gRPC。如果你的公司在用 Thrift，就继续用 Thrift；迁移的收益很少值得付出的代价。

  • Connect / Twirp 是当你想要 Protobuf 质量的 schema、却又需要从浏览器无代理调用时的答案。 它们去掉了 Trailers 要求（gRPC 在浏览器中最大的问题），将状态码放入响应的 headers/body 中。

§4. 底层数据结构——Wire 格式

这里的深度在于 实际在 wire 上传输的字节是什么，自顶向下：Protobuf 消息 → gRPC 分帧 → HTTP/2 帧 → TCP。此处使用几个不同领域的示例（支付授权、用户档案查询、信息流条目）来保持抽象的真实性。

4a. Protobuf wire 格式：带 varint 编码的 Tag-Length-Value

Protobuf 消息是一个 (field_tag, value) 对的扁平序列。没有顶层长度、没有字段名、没有嵌入的 schema——只有根据双端已知的 schema 解码的编号字段。

字段 tag = (field_number << 3) | wire_type，以 varint 编码。Wire 类型：

Wire 类型 编码 用于
VARINT 0 int32, int64, uint32, uint64, sint32, sint64, bool, enum
FIXED64 1 fixed64, sfixed64, double
LEN 2 string, bytes, 嵌入消息, packed repeated
FIXED32 5 fixed32, sfixed32, float

（SGROUP=3 和 EGROUP=4 是已废弃的 proto2 group 标记。）

Varint 编码 是一种可变长度整数。每个字节的最高位作为续位标志，低 7 位作为数据，采用小端序：

  • 1 → 0x01 （1 字节）
  • 150 → 0x96 0x01 （二进制： 10010110 00000001 → 去掉 MSB，交换分组： 0000001 0010110 = 150)
  • 300 → 0xAC 0x02
  • 2^32 - 1 → 5 字节
  • 中的负整数 int32 / int64 被符号扩展为 64 位无符号数，因此 -1 占用 10 字节。这是经典的陷阱。有负数时请使用 sint32 / sint64 加 zigzag 编码（(n << 1) ^ (n >> 31)），这样 -1 只需 1 字节。

具体示例 1——用户档案查询 （搜索式读取）：

message GetUserProfileRequest {
  string user_id = 1;
}

编码后 GetUserProfileRequest { user_id = "42" }:

0x0A 0x02 0x34 0x32

分解：\n- 0x0A = 字段 1 的 tag，wire 类型 2（LEN）。 (1 << 3) | 2 = 10 = 0x0A. - 0x02 = 字符串的字节长度（varint）。\n- 0x34 0x32 = ASCII 编码的"42"。

总耗时： 4 字节。等效 JSON {"user_id":"42"} 为 16 字节。

响应 UserProfile { user_id = "42", display_name = "Yifan", created_at_ms = 1714400000000, roles = ["admin", "staff"] }:

field 1 (user_id, LEN):
  0x0A 0x02 0x34 0x32                                    -> "42"

field 2 (display_name, LEN):
  0x12 0x05 0x59 0x69 0x66 0x61 0x6E                     -> "Yifan"

field 4 (created_at_ms, VARINT int64):
  0x20 0x80 0xB6 0xD0 0xC5 0xC2 0x31                     -> 1714400000000

field 5 (roles, LEN) - repeated string, two entries:
  0x2A 0x05 0x61 0x64 0x6D 0x69 0x6E                     -> "admin"
  0x2A 0x05 0x73 0x74 0x61 0x66 0x66                     -> "staff"

总计：4 + 7 + 7 + 7 + 7 = 32 字节 在 wire 上传输。等效 JSON 约为 110 字节。

具体示例 2——支付授权 （不同领域，相同编码规则）：

message AuthorizeRequest {
  string idempotency_key = 1;
  int64 amount_minor_units = 2;
  string currency = 3;
}

编码后 AuthorizeRequest { idempotency_key = "abc", amount_minor_units = 4999, currency = "USD" }:

0x0A 0x03 0x61 0x62 0x63          -> field 1, "abc"
0x10 0x8F 0x27                    -> field 2 (VARINT), 4999 = 0x1387 -> varint 0x8F 0x27
0x1A 0x03 0x55 0x53 0x44          -> field 3, "USD"

13 字节。要点：Protobuf 更紧凑，原因在于：(a) 字段名是整数而非字符串；(b) 整数使用 varint，小整数只占一字节；(c) 没有分隔符、空白或引号；(d) schema（而非 wire）承载字段语义。

Schema 演进规则——字段编号约定

wire 格式 通过整数字段编号而非名称标识:

  1. 添加字段：选择一个未使用的编号；旧读取方会忽略未知 tag（proto3 默认将其保留在 unknown-fields 集合中以便往返传输）。后向 + 前向兼容。始终可选。
  2. 删除字段：停止写入并 reserve 该编号： message User { reserved 4; reserved "old_field_name"; } 不加 reserved，未来的开发者会用不同类型复用字段 4，磁盘上的旧数据解码时将产生灾难性错误。
  3. 重命名字段：免费；名称只是 codegen 产物，不出现在 wire 上。
  4. 修改类型：仅限于 wire 兼容类型之间—— int32 ↔ int64 ↔ uint32 ↔ uint64 ↔ bool （均为 VARINT）； string ↔ bytes （均为 LEN）。非 string → int32.
  5. repeated ↔ 标量：proto3 repeated 标量数组默认使用紧凑编码（一个 LEN 标签，所有值连续存储）。在 repeated 与单值标量之间切换在大多数情况下是安全的，但有若干注意事项。
  6. required 在 proto3 中不存在 因为一旦标记为 required，就永远无法在不破坏所有读取方的情况下移除该字段。proto2 有此特性；proto3 已将其移除。
  7. oneof：集合中只能设置一个字段（tagged union）。向 oneof 中添加字段是安全的；将已有字段移入 oneof 属于破坏性变更。

强制执行: buf breaking 。 protolock 在每个涉及 .proto的 PR 上运行。在大规模集群（数百个服务、数千名工程师）中，schema 审查是 Staff 工程师级别的职责。

4b. Thrift 线路格式

Thrift 提供两种二进制格式：

  • TBinaryProtocol：定宽且简单。每个字段包含 [type_byte (1)] [field_id (2)] [value]。int32 值为 1 时： 0x08 0x00 0x01 0x00 0x00 0x00 0x01 —— 7 字节完成了 Protobuf 只需 2 字节的编码。
  • TCompactProtocol：使用变长整数并对字段 ID 进行 delta 编码（每个字段存储与前一字段的差值）。int32 值为 1、字段 ID delta 为 1 时： 0x15 0x02 —— 2 字节（接近 Protobuf）。大多数生产环境 Thrift 使用 TCompact。

与 Protobuf 的权衡：Thrift IDL 原生区分 set, map, list ，并内置 service 定义和类型化异常。Protobuf 将集合视为 repeated。如今的实际选择主要取决于历史因素（Facebook 选择了 Thrift；Google 选择了 Protobuf；各公司采用其所在生态系统中的方案）。

4c. JSON

JSON 是文本格式。一个字段 "user_id": "42" 占 14 字节。解析需要分词器，速度约为 30–80 ns/字节（Java 中的 Jackson），而 Protobuf 的生成解析器约为 2–5 ns/字节。JSON 真正的优势仅在于人类可读性、无 schema 的灵活性、浏览器原生解析以及无限扩展性——这些在公共边界处都是 特性 ，在内部服务间则是 负担 。

JSON Schema 可以事后强制类型检查，但在服务间调用路径中几乎不会在运行时强制执行。

4d. HTTP/2 帧结构

单个 TCP 连接可多路复用多个并发 HTTP/2 流。每个流由一系列 帧组成，每个帧具有 9 字节头部：

+---------------+---------------+---------------+
|   Type (8)    |   Flags (8)   |
+-+-+-----------+---------------+-------------------------------+
|R|                 Stream Identifier (31)                      |
+=+=============================================================+
|                   Frame Payload (0...)                      ...
+---------------------------------------------------------------+

与 RPC 相关的帧类型：

  • HEADERS （0x1）：包含 HPACK 压缩的伪头（:method, :path, :scheme, :authority, :status）和普通头（content-type, grpc-timeout, grpc-encoding, authorization, traceparent).
  • DATA （0x0）：请求/响应体。对于 gRPC，请求体是一系列长度前缀的 Protobuf 消息——1 字节（压缩标志）+ 4 字节（消息长度，大端 uint32）+ N 字节（Protobuf 有效载荷）。
  • TRAILERS （带 END_STREAM 标志的 HEADERS 帧，在 DATA 之后发送）：包含 grpc-status, grpc-message. 这就是 gRPC 需要 HTTP/2 的原因 —— HTTP/1.1 的 trailers 几乎无法使用（浏览器不暴露它们；中间件会将其剥除）。gRPC 将调用结果状态放在 trailers 中，因为流式 RPC 可能在发现错误之前已发出多个 DATA 帧。
  • SETTINGS （0x4）：连接设置—— MAX_CONCURRENT_STREAMS （默认 100，gRPC 服务器通常协商为 1000+）， INITIAL_WINDOW_SIZE （每流流控配额，默认 65535 字节）， MAX_FRAME_SIZE （默认 16 KB）。
  • WINDOW_UPDATE （0x8）：每流和每连接的流控。接收方告知发送方"我还有 N 字节缓冲区，你可以再发送 N 字节。"
  • RST_STREAM （0x3）：取消单个流而不断开连接。当客户端截止时间到期或客户端取消时，gRPC 使用此帧。
  • GOAWAY （0x7）："我将在流 N 之后关闭此连接；请勿启动新流。"用于优雅关闭。
  • PING （0x6）：保活与 RTT 测量。

HPACK 头部压缩：双端均维护一张近期出现的头部名/值对动态表。一旦 authorization: Bearer <very long JWT> 被见过，后续引用只需 1 字节索引。静态表免费覆盖了常见头部，如 :method GET 。在已服务 1000 次请求且 auth token 相同的连接上，每次请求的头部开销约为 5 字节，而非 ASCII 下的约 800 字节。

流控：每个流初始接收窗口为 65 KB。如果接收方不消费数据，发送方将停止为该流发送 DATA 帧。按流隔离——一个慢速流不会阻塞同一连接上的兄弟流（TCP 层的队头阻塞除外，详见 §13 问题 5）。

4e. gRPC over HTTP/2：一次 unary 调用的端到端字节级分析

逐步分析 stub.GetUserProfile(GetUserProfileRequest { user_id: "42" }) 带 500 ms 截止时间。

步骤 1 —— 客户端 stub 序列化。 生成代码调用 request.toByteArray()。字节数： 0x0A 0x02 0x34 0x32。CPU 开销：约 50 ns。

步骤 2 —— gRPC 帧封装。 4 字节的 Protobuf 消息被封装在 5 字节的 gRPC 帧中：

0x00                       <- compression flag (0 = uncompressed)
0x00 0x00 0x00 0x04        <- message length (big-endian uint32 = 4)
0x0A 0x02 0x34 0x32        <- Protobuf payload

共计：9 字节。

步骤 3 —— HTTP/2 HEADERS 帧。 gRPC 客户端在现有 HTTP/2 连接上打开流 5（奇数编号，客户端发起）：

HEADERS frame, stream 5, END_HEADERS=1, END_STREAM=0
  :method = POST
  :scheme = https
  :path = /user.v1.UserService/GetUserProfile
  :authority = user-service:9090
  content-type = application/grpc+proto
  grpc-encoding = identity
  grpc-accept-encoding = gzip
  grpc-timeout = 500m                  <- deadline as relative time
  te = trailers                         <- MUST be present for gRPC
  user-agent = grpc-java/1.62.0
  authorization = Bearer eyJhbGciOi...
  traceparent = 00-abc...-def...-01     <- W3C trace context

经 HPACK 压缩后，线路上约 50–100 字节（有历史记录时更少）。

步骤 4 —— HTTP/2 DATA 帧。

DATA frame, stream 5, END_STREAM=1
  payload: 0x00 0x00 0x00 0x00 0x04 0x0A 0x02 0x34 0x32  (9 bytes)

HTTP/2 帧头：9 字节。线路上 DATA 帧总计：18 字节。

步骤 5 —— 内核与网络。 HTTP/2 编解码器写入约 70–120 字节。TLS 封装在 TLS 记录中（约 30 字节记录开销；TLS 1.3 首次连接需一个往返，之后支持 0-RTT）。TCP 每个分段增加 20 字节头部。NIC、交换机、NIC，内核 DMA 进入服务器环形缓冲区。

步骤 6 —— 服务器端解码。 服务器内核通过 read()传递字节。TLS 解密。HTTP/2 编解码器解析帧，按流 ID 解多路复用。HEADERS 进入请求元数据，DATA 进入请求体。gRPC 帧读取 [0x00][0x00 0x00 0x00 0x04] —— 紧跟 4 字节 Protobuf 消息。将 0x0A 0x02 0x34 0x32 交给生成的解析器。

步骤 7 —— Protobuf 解码。 生成的 GetUserProfileRequest.parseFrom(bytes)：- 读取标签字节 0x0A → 字段 1，wire 类型 LEN。- 读取长度 varint 0x02。- 读取 2 字节 0x34 0x32 → "42"。- 缓冲区结束。返回 { user_id = "42" }。CPU 开销：约 30 ns。

步骤 8 —— 截止时间强制执行。 gRPC 服务器注册一个截止时间到期回调。根据 grpc-timeout: 500m，服务器知道必须在 500 ms 减去已用时间内响应。如果处理器在此之前未调用 onCompleted() ，服务器发送 RST_STREAM 并以 grpc-status: DEADLINE_EXCEEDED (4).

步骤 9 —— 处理器。 应用代码执行。访问数据库，构建 UserProfile.

步骤 10 —— 响应路径。 镜像反转。服务器发出 HEADERS（:status 200, content-type），DATA（长度前缀 Protobuf），最后是 TRAILERS 与 grpc-status: 0 。 END_STREAM=1。Trailers（而非初始 HEADERS）携带状态，因为流式 RPC 可能在确定最终结果前已发出多个 DATA 帧。

步骤 11 —— 客户端接收。 HTTP/2 编解码器将帧路由至流 5。DATA 进入 gRPC 帧解析器；帧解析器解析长度前缀，将字节交给 UserProfile.parseFrom。Trailers 到达： grpc-status: 0。Stub 返回消息。

端到端：若所有组件均处于热态，p50 约 1.3 ms。其中大部分是处理器内部的数据库查询——框架在小消息 p99 时增加约 250 µs 开销。

同样的调用通过 REST + JSON 将：以约 10 倍 CPU 序列化 JSON，以 ASCII 发送 HTTP/1.1 请求行 + 头部（约 250 字节，而非约 50 字节压缩），在调用期间占用一个 TCP 连接（无多路复用），需要自行约定截止时间头部规范（无 grpc-timeout），并失去 trailer 状态模式，导致流中途出错无法干净上报。

§5. 不同规模下的容量边界

RPC 网络跨越六个数量级。不同规模遇到不同瓶颈；同一技术服务于所有规模，但拓扑结构各异。

规模层级 典型部署 RPS / 扇出 瓶颈 拓扑
小型 （初创企业） 约 10 个服务，总计约 1k QPS 每服务 10 QPS，扇出深度 2 无实际瓶颈 直接使用 HTTP/1.1 + JSON，前置负载均衡，无服务网格
中型 （类 Stripe） 约 100 个服务，内部约 50k QPS 热点服务 1k–10k QPS，扇出深度 3–5 重试风暴、连接池耗尽 热路径采用 gRPC；客户端侧负载均衡；各团队重试策略（存在分歧）；公共 API 上的幂等键
大型 （类 Netflix） 约 1000 个服务，约 5M QPS 热点服务 500k QPS，扇出深度 5–8 级联故障；sidecar 开销变得明显 服务网格（Envoy + 控制平面）、熔断器（Hystrix → resilience4j）、schema 注册中心、破坏性变更 CI 检查
超大型 （Google 内部） 1万+ 个服务，约 100 亿 RPC/秒 热点服务 100 万+ QPS，扇出深度 10+（一次搜索查询触及约 1000 个后端） sidecar CPU 与应用 CPU 相当；控制平面可扩展性 ambient mesh / 无代理 gRPC、分层服务目录、区域感知路由、对冲请求、专职 schema 审查团队

具体部署案例（覆盖上述范围）:

  • 小型/中型：Stripe 的 公共 API 采用 REST + JSON over 标准 HTTP/2；其内部服务部分是 Ruby on Rails，部分是通过 gRPC 和 Twirp 通信的 Go 服务。每个服务每秒数万次内部 RPC。Stripe 将 Idempotency-Key 作为其公共 API 的一等公民——这是"写操作重试安全"的教科书示例。

  • 中型/大型：Netflix 首页渲染每次设备会话发起 100+ 次内部 RPC 调用。在 2.5 亿会员和约 10 亿次日均播放启动的体量下，跨网格每秒有数百万次 RPC。Hystrix（现为 resilience4j）正是在此诞生；最初的重试风暴故障分析是熔断器的权威参考案例。

  • 大型：LinkedIn 的 Rest.li 框架在约 10 万个服务实例上处理约 1000 万 QPS。仅 Espresso 存储层就通过 Rest.li 承受约 300 万 QPS。D2（动态发现）负责客户端侧负载均衡。Rest.li 最初基于 HTTP/1.1 的 REST 风格，正逐步迁移至 gRPC 用于内部热路径。Feed 中间层和 Profile 中间层等中间层服务单独可承受 50 万+ QPS。

  • 大型/超大型：Uber 内部网格运行 Apache Thrift over TChannel 传输（以及其上的 RPC 框架 YARPC）。内部约 1000 万 QPS。一次行程请求扇出至约 30 个同步服务加约 40 个异步服务，p99 预算为 300 ms。

  • 超大型：Google 内部 RPC 量约为每秒 100 亿次，通过 Stubby（gRPC 的内部前身）处理。一次搜索查询扇出至约 1000 个后端，每个调用截止时间为 10–50 ms。他们运行无代理网格——gRPC 库直接消费 xDS，没有 sidecar——因为该规模下 sidecar CPU 开销将成为数十亿美元量级的独立成本项。

  • 超大型（不同形态）：Meta 全链路运行 Thrift。具体量未公开披露，但据推测与 Google 同一数量级。Thrift 的类型化异常模型在该公司比 gRPC 的扁平状态码更受青睐。

每连接容量 （Staff 工程师应掌握的数字）：

  • HTTP/1.1：每连接 1 个在途请求（流水线在实践中已死亡）。对于 50 ms 延迟下的 10k QPS，利特尔法则给出 500 个并发在途请求 = 每客户端 500 个 socket。1000 个客户端 × 500 = 服务器上 50 万个 socket——迅速耗尽临时端口范围和文件描述符。
  • HTTP/2：默认 SETTINGS_MAX_CONCURRENT_STREAMS = 100 （服务器通常协商为 1000+）。一个 TCP 连接承载 100–1000 个并发交错 RPC。同样 10k QPS / 50 ms：500 个在途 / 每连接 100 个流 = 每客户端 5 个连接。1000 个客户端 × 5 = 5000 个 socket——提升了两个数量级。
  • HTTP/3 (QUIC)：与 HTTP/2 相同的多路复用数学，但在 UDP 层进行流多路复用，因此一个丢包只阻塞一个流。在有损网络（移动网络、跨区域）中具有尾延迟优势。

延迟预算 （一次数据中心内 unary 调用，小消息）：

阶段 p50 (µs) p99 (µs)
客户端 Protobuf 编码（1 KB） 20 100
客户端内核写入 5 30
数据中心内网络（NIC → 交换机 → NIC） 100 300
服务器内核读取 5 30
HTTP/2 帧解码 5 20
服务器 Protobuf 解码 30 150
处理器逻辑（数据库查询等） 1000 3000
响应路径镜像 165 630
总计 约 1.3 ms 约 4.3 ms

处理器占主导地位。RPC 框架开销在小消息 p99 时约为 250 µs。将框架效率提升一倍可节省约 100 µs（在 4 ms 预算中）——在集群规模（CPU 和带宽）上有意义，但对单次调用延迟无实质影响。

序列化 CPU 在集群规模下的影响:

  • Protobuf 编码：约 2–5 ns/字节（生成代码，无反射）。1 KB ≈ 2–5 µs。
  • JSON（Jackson）：约 30–80 ns/字节。1 KB ≈ 30–80 µs。
  • 在 1M RPC/秒、1 KB 消息的情况下：JSON 仅解析就消耗约 50–80 个核心；Protobuf 消耗约 5 个核心。在 LinkedIn / Netflix 的规模下，这意味着序列化账单相差 100 万/年与 1000 万/年之间的差异。

连接池耗尽的数学 （反复出现在每次扇出中的隐性杀手）：

服务 B 通常在 10 ms 内响应。服务 A 有 200 连接的连接池，向 B 发起 5000 QPS。利特尔法则：在途数 = QPS × 延迟 = 5000 × 0.010 = 50 个在途，远低于连接池上限。B 降级至 1000 ms（未宕机——只是变慢）。所需在途数 = 5000 × 1.0 = 5,000。连接池为 200。A 开始在连接池获取上阻塞；约 40 ms 内，每个 A 线程均被阻塞；A 无法响应其调用方； A 已降级。波浪向外传播。我们将在 §13 中回到这一问题。

关键抽象（与具体产品无关）：

  • Sidecar 模式。 每个 Pod 运行一个代理（Envoy 是主流实现；Linkerd 是更轻量的替代方案），处理横切关注点：mTLS、重试、熔断器、流量整形、可观测性。应用代码只需使用普通 gRPC。Sidecar 运行在 localhost 上，网络跳转为回环（约 5 µs）。

  • 控制平面/数据平面分离。 控制平面（Istio Pilot、Consul Connect、Linkerd 控制器、AWS App Mesh 控制器）通过 xDS 推送配置——路由、集群定义、TLS 证书、重试策略、熔断器阈值——xDS 是一组 gRPC 流式 API（LDS=监听器、RDS=路由、CDS=集群、EDS=端点、SDS=密钥、ADS=聚合）。数据平面从不询问"服务 B 在哪里？"——它已从配置中获知。

  • 每上游连接。 A 的 sidecar 为每个后端维护 2–4 个 HTTP/2 连接（通常每个 CPU 线程一个），每个连接承载约 100–1000 个多路复用流。

  • 重试边界。 重试发生在 sidecar 而非应用代码中，因此重试策略在整个集群中统一，并可通过重试预算进行全局速率限制。

  • 无代理变体。 在 Google 规模下，sidecar 的 CPU 开销成为单独的预算项，因此 gRPC 库本身直接消费 xDS——无需 sidecar 进程——代价是库未实现的 mesh 功能。通过 gRPC 的 xDS 客户端推动开源采用不断增长。

  • Ambient mesh 变体。 不再为每个 pod 部署 sidecar，而是由内核中的 eBPF 程序（Cilium、Istio Ambient）处理数据面。每个 pod 的开销更低，成熟度较低但正在快速提升。

朴素的 REST over HTTP/1.1 拓扑——app → 负载均衡器 → app，重试/熔断器/mTLS 内置于每个 app 的库中——是每家公司的起点，也是在约 50 个服务之后无法持续维护的方案。不同团队选择不同的重试库、不同的超时策略、不同的认证方案，而 mesh 的出现成为统一标准的强制机制。

§7. 认证与服务身份

RPC fabric 在解决其他任何问题之前，首先必须解决的横切关注点是"谁在调用谁"。10 个服务时，每个团队在配置文件里放一个静态 API key 就了事了。到了 10,000 个服务、每季度轮换密钥并有 SOC 2 审计要求时，这种方案会灾难性地失败。现代的答案是 由 SPIFFE（Secure Production Identity Framework For Everyone）工作负载身份支撑的 mTLS（mutual TLS），其深度来自于深入探讨 为何在规模场景下 JWT 作为服务认证方式比 mTLS 更差。

mTLS 的本质

标准 TLS 向 服务器 验证 客户端 ——客户端检查服务器的证书是否由可信 CA（证书颁发机构）签名，以及证书的 Subject Alternative Name（SAN）是否与所连接的主机名匹配。 Mutual TLS 增加了反向验证：客户端在 TLS 握手期间也出示证书，服务器对其进行信任根验证。握手完成后，双方已通过密码学方式互相验证身份，后续连接（HTTP/2 帧、gRPC 载荷）通过同一加密通道传输。

握手成本是真实存在的（完整 TLS 1.3 握手耗费一个来回；TLS 1.2 需要两个），但该成本 分摊在长连接 HTTP/2 连接上 ，该连接在断开前可能承载数百万次 RPC。稳态下 mTLS 的单次调用开销约为 0——TLS 记录层每个记录增加约 30 字节的 MAC/填充，AES-NI 硬件加速使现代 CPU 上的加密本身几乎没有代价。

难点不在于协议，而在于 身份供给与轮换。每个工作负载都需要证书，证书会过期（最佳实践：短期证书为 1–24 小时；时间越短，泄露的爆炸半径越小），且撤销必须快速传播。

SPIFFE 与 SPIRE：作为基础设施的工作负载身份

SPIFFE（Secure Production Identity Framework For Everyone） 是工作负载身份的开放标准，其具体产物包括：

  • SPIFFE ID：格式为 URI，例如 spiffe://trust-domain/path, e.g., spiffe://prod.example.com/ns/payments/sa/balance-service。信任域是组织的身份边界；路径部分可编码任意内容（命名空间、服务账户、环境等）。
  • SVID（SPIFFE Verifiable Identity Document）：SPIFFE ID 的密码学证明，有两种形式—— X.509-SVID （将 SPIFFE ID 编码在 SAN 中的标准 X.509 证书）和 JWT-SVID （其 sub claim 为 SPIFFE ID 的 JWT，用于无法进行 TLS 终止的场景）。
  • 信任包：用于验证给定信任域 SVID 的 CA 证书集合，通过 Workload API 分发。

SPIRE 是 SPIFFE 的标准实现，包含两个组件：

  1. SPIRE Server：CA，在完成认证后向工作负载签发证书。
  2. SPIRE Agent：运行在每个节点上。工作负载通过 Unix 域套接字向 Agent 请求身份；Agent 对其进行认证（使用 Kubernetes Service Account、AWS Instance Metadata、Docker 标签等）并返回 SVID。

认证步骤是关键所在：SPIRE 不会相信"工作负载自称是 balance 服务"。它通过编排器（Kubernetes）验证，调用该 Unix 进程 确实 是 payments 命名空间中带有 balance-service 服务账户的 pod，且其所在节点本身也通过了节点认证。身份根植于运行时基础设施，而非工作负载可能撒谎的配置文件。

轮换：SVID 通常签发 1 小时的 TTL（Time To Live）。Agent 会在到期前及时在后台更新证书。工作负载通过 Workload API 接收证书流，实现无缝轮换。被泄露的 SVID 最多只能被利用不到一小时。

Istio 的自动 mTLS

Istio（主流服务 mesh）在 sidecar 之间实现了 mTLS 透明地 在 sidecar 之间。应用代码使用普通 HTTP 或 gRPC；sidecar 代理（Envoy）在 localhost 终止来自 app 的明文流量，并通过网络以 mTLS 重新加密。两个 sidecar 均出示由 Istio CA（Citadel，现已并入 Istiod）签发的证书。应用从不接触证书，也无需知道 mTLS 的存在。

数据面： - App → Envoy： localhost:9090 明文。 - Envoy → Envoy：通过网络的 mTLS。 - Envoy → App： localhost:9090 接收侧明文。

控制面： - Istiod 是 CA，签发 SVID（Istio 使用与 SPIFFE 兼容的 ID，格式为 spiffe://cluster.local/ns/payments/sa/balance-service）。 - 证书通过 SDS（Secret Discovery Service），xDS 变体之一。Envoy 通过 gRPC 流接收新证书，在内存中重新加载，无需重启进程。 - TTL：默认 24 小时；轮换在 TTL 约 50% 时触发。

最终状态：每个 pod 间的 TCP 连接均经过 mTLS 加密并完成双向认证，且没有任何应用工程师编写了一行 TLS 代码。

为何在规模场景下 JWT 作为服务认证方式比 mTLS 更差

许多公司有一种遗留模式：服务 A 调用服务 B 时在 HTTP header 中携带 Authorization: Bearer <JWT> 在 HTTP header 中，由共享密钥或中央认证服务签名。 这在规模场景下比 mTLS 更差，你应当能够列举原因：

  1. Bearer token 可被窃取。 HTTP header 中的 JWT 就像众所周知的 cookie——任何拥有这些字节的人都可以重放它。mTLS 需要私钥，而私钥永远不会离开 pod。包含 JWT 的泄露日志行会危及调用方；而包含 TLS 记录字节的泄露日志行在没有会话密钥的情况下毫无价值。
  2. 单次调用验证成本。 验证 JWT 需要解析 JSON、验证签名（RSA 或 ECDSA，每次调用约 50–200 µs）、检查过期时间，还可能访问密钥轮换端点。mTLS 仅在 每个连接上执行一次，之后每次调用无额外开销。在集群规模下（每个服务每秒数百万次 RPC），JWT 验证可能消耗 5–10% 的 CPU；mTLS 在稳态下约为 0%。
  3. Token TTL 在两难之间进退两难。 TTL 短（5 分钟）意味着调用方需要不断从认证服务刷新，该服务成为每次 RPC 的关键路径依赖。TTL 长（1 天）意味着被盗 token 可被利用整整一天。mTLS 通过将身份绑定到主机上的密钥对来规避这一问题。
  4. 从设计上缺乏双向认证。 服务器验证 JWT；客户端验证服务器的 TLS 证书。协议是非对称的。mTLS 是对称的——双方均通过密码学方式得到证明。
  5. 无传输层加密保障。 明文 HTTP 上的 header 中的 JWT 是暴露的。mTLS 加密整个通道。
  6. 跨多跳的 token 传播容易出错。 如果服务 A→B→C，B 是转发 A 的 JWT，还是使用自己的？如果 B 转发 A 的，则被攻陷的 B 可以重放它。如果 B 使用自己的，则你已失去原始调用方的身份。使用 mTLS，身份从设计上是逐跳的，原始调用方身份通过单独的、签名的、窄范围的 header 传播（见 §11 关于 header 传播的内容）。

JWT 在该 fabric 中合理的位置是 数据的分片、分区和副本复制在下层数据面进行；协调服务本身仅通过共识机制进行复制。 在调用树中传播的用户身份（人的会话），而非 服务间 身份。mTLS 认证工作负载；JWT 携带工作负载代表谁在行动。

§8. 授权策略

认证回答"你是谁"，授权回答"你被允许做什么"。即使已通过 mTLS + SPIFFE 解决了身份问题，你仍然面临这个问题：服务 A 正在调用 BalanceService.Debit(account=X, amount=$50) ——服务 A 被允许调用该账户上的那个方法吗？这是一个策略决策，现代模式是 将策略从应用代码中外化.

"我有一个 JWT，我能做什么？"这个问题

朴素模式：每个服务在自己的处理器中实现各自的授权检查。 if (caller_role == "admin") allow(); 分散在 8 种语言的 800 个服务中。六个月后，没人知道全局授权规则是什么，三个服务对"admin"有微妙的不同定义，合规审计要求单一事实来源。这就是 授权蔓延 反模式。

修复方案：由 策略决策点（PDP） 评估规则；由服务（或 sidecar）中的 策略执行点（PEP） 咨询 PDP。策略是声明式的、版本化的、可审查的。

OPA（Open Policy Agent）

OPA（Open Policy Agent） 是通用策略的事实开放标准，其设计如下：

  • Rego：一种类 Datalog 的声明式策略语言。规则描述基于输入哪些操作是 允许 （或哪些是 拒绝）的。
  • Sidecar 模式：OPA 作为 sidecar（或本地进程）与每个服务一起运行。服务通过在本地 localhost 传入描述请求的 JSON 文档来调用 OPA 获取授权决策。OPA 返回 allow: true/false 加上可选上下文。
  • Bundle 分发：策略（Rego 文件和数据）被打包成 bundle，从中央服务器分发。OPA 通过轮询或流式更新获取。策略经过版本控制、签名，像部署一样发布。
  • 决策日志：OPA 将每个决策（输入 + 结果）记录到一个存储端，审计人员可获得完整的审计追踪。

一个 Rego 规则示意：

package payments.authz

default allow = false

allow {
    input.method == "BalanceService.Debit"
    input.caller.spiffe_id == "spiffe://prod.example.com/ns/payments/sa/checkout"
    input.request.amount <= 10000
    input.account.owner == input.user.id
}

服务（或其 sidecar）构建 input 文档，来自请求元数据——调用方的 SPIFFE ID、RPC 方法、请求字段、已解析用户——并询问 OPA。OPA 对输入评估 Rego 并在微秒内返回决策（Rego 编译为高效虚拟机）。

收益：策略是单一产物，使用单一语言，由安全工程师审查，独立于其管辖的服务进行部署。一条新规则（例如"任何服务未经明确授权不得扣款超过 $10K"）只需向策略仓库提一个 PR，而非向 800 个服务分别提 800 个 PR。

Cedar（AWS）：一种更轻量的替代方案

Cedar 是 AWS 较新的策略语言，用于 Amazon Verified Permissions 和 Cognito。与 Rego 不同的设计选择：

  • 静态可分析：Cedar 策略可被检查"此策略是否可满足"以及"策略 A 是否严格比策略 B 更宽松"。Rego 是图灵完备的，难以进行形式化推理。
  • Schema 感知：Cedar 策略引用实体（User、Account、Resource）和操作（Read、Debit、Transfer）的 schema。策略语言强制要求只能引用已存在的类型。
  • 性能：Cedar 专为在 AWS 授权面实现亚毫秒级评估而设计。

Cedar 比 Rego 更轻量、约束更强——适合"谁能对哪个资源做什么"的授权，对于超出授权范围的任意策略不那么胜任。对于典型的 RPC fabric，两者均可。

集中式与分布式授权

两种部署模型：

模式 架构 权衡
集中式 PDP 一个（高可用）授权服务，每次授权检查都是对其的网络调用。 策略状态的单一事实来源。延迟耦合（每次调用增加约 1 ms）。PDP 成为关键路径依赖。
分布式 PDP（sidecar） OPA 与每个服务一起运行，策略 bundle 推送到每个实例，决策在本地完成。 亚毫秒级决策。策略更新存在传播延迟（秒级）。每个 PDP 持有完整策略集。

在规模化生产环境中，几乎总是选择 分布式 ——本地 OPA sidecar + 中央 bundle 服务器。策略传播延迟（秒级）是可接受的；每次调用的延迟则不可接受。

PEP 的位置

策略执行点有三种可能的位置：

  1. 在应用处理器中。应用在执行工作前显式调用 opa.check(request) 在开始工作前。优点：细粒度、感知业务上下文。缺点：每个开发者都要记得调用。
  2. 在 sidecar 中。Envoy 的 ext_authz 过滤器在转发给应用前对每个请求调用 OPA。优点：无需应用配合，统一强制执行。缺点：只能看到请求元数据中的内容；无法强制依赖数据库状态的业务规则。
  3. 两层叠加。sidecar 执行粗粒度授权（此服务可以调用此方法）；应用执行细粒度授权（此用户可以读取此行）。

Staff 级别的答案是"两层叠加"——sidecar 处的粗粒度授权防止被攻陷的服务调用不应调用的方法；应用处理器中的细粒度授权防止格式合法但未经授权的业务操作。

§9. 可观测性与分布式追踪

当一个用户请求扇出到 50 个服务，p99 延迟出现尖刺时，"哪个跳点慢？"是定义故障响应的核心问题。答案来自 分布式追踪，现代标准是 W3C Trace Context 经由 OpenTelemetry 插桩传播，并在多个层次进行采样决策。

什么是 trace

一个 trace 是 spans的树形结构。span 代表一个工作单元（一次 RPC 调用、一次数据库查询、一次耗时计算）。每个 span 包含：

  • trace_id （16 字节，十六进制编码为 32 个字符）：标识整个 trace 树。
  • span_id （8 字节，十六进制编码为 16 个字符）：标识此特定 span。
  • parent_span_id：链接到调用方 span。
  • start_time, end_time, duration.
  • attributes：键值标签（http.method, db.statement, grpc.code, user.id).
  • events：span 内的带时间戳日志行。
  • status: OK / ERROR / UNSET.

当请求到达边缘时，会获得一个全新的 trace_id。每个下游 RPC 创建一个具有新 span_id 但继承 trace_id 并指向其父节点。结果是一棵树；渲染为火焰图时，可精确显示哪些叶节点消耗了时间预算。

W3C Trace Context： traceparent 请求头

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 W3C Trace Context 规范标准化了 trace ID 如何在 HTTP 上传播。相关 header 为：

traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             │  │                                │                │
             │  │                                │                └── trace-flags (sampled=1)
             │  │                                └── parent-span-id
             │  └── trace-id (16 bytes)
             └── version

第二个 header， tracestate，携带特定厂商的数据（tracestate: dd=s:1;p:xyz,opentelemetry=...）。多个追踪系统通过写入各自在 tracestate.

B3 传播 是 Zipkin 的旧格式，header 名称为： X-B3-TraceId, X-B3-SpanId, X-B3-ParentSpanId, X-B3-Sampled。旧系统（Brave、Finagle、Linkerd-1）仍发出 B3。现代系统发出 W3C。两者之间存在桥接；Staff 级别的现实是"你的 mesh 在过渡窗口内会同时发出两者"。

OpenTelemetry：统一的插桩 API

OpenTelemetry（OTel） 是将追踪、指标和日志统一在单一 API 和 SDK 之下的伞形项目。它取代了早期竞争对手 OpenTracing 和 OpenCensus（两者于 2019 年合并为 OTel）。其架构为：

  • API：与语言无关的接口，供应用代码调用（tracer.startSpan(...)）。在每种语言中形态相同。
  • SDK：实现 API，配置采样器、处理器、导出器。
  • 导出器：将 span/指标序列化到后端（Jaeger、Zipkin、Tempo、Datadog、Honeycomb、X-Ray、Cloud Trace）。
  • OTLP（OpenTelemetry Protocol）：传输格式，gRPC + Protobuf。后端原生接受 OTLP。
  • Collector：接收来自应用的 OTLP 的 sidecar 或守护进程，执行处理（采样、脱敏、批处理）后转发到后端。

核心思想：针对 OTel API 一次插桩，无需触及应用代码即可切换后端。在 OTel 出现之前，每个后端（Jaeger、Zipkin、Datadog……）都有各自的客户端库；切换后端意味着在整个集群中重写插桩代码。

跨异步边界的 span 传播

难点场景：服务 A 生产一条 Kafka 消息；服务 B 在 10 秒后消费它。A 的用户请求已完成；从 A 的视角来看，trace 已经"结束"。但 B 对该消息的处理在逻辑上是同一工作流的一部分。如何将它们关联起来？

Kafka 消息头携带链路追踪上下文。 当服务 A 生产消息时，它将 traceparent （以及 tracestate）注入到 Kafka 消息头中。当服务 B 消费时，它提取这些信息并将新 span 创建为原始 span 的子节点——即使已经过了数小时且原始链路已"关闭"。OpenTelemetry 为 Kafka、RabbitMQ、SQS 等提供了专门的自动埋点，可自动完成上述操作。

在服务内部，跨线程边界的异步传播通过语言特定机制解决： Go: context.Context 上下文携带活跃 span；每个异步调用都必须将 context 穿透传递。经典的"在第 4 跳丢失 span"bug 就是忘记传递 context。 Java：ThreadLocal 默认携带 span；切换到不同线程（CompletableFuture、reactor）需要显式传播。Reactor 等框架提供了 Hooks.enableAutomaticContextPropagation() 来完成此操作。 Node.js：AsyncLocalStorage 携带 span。Promise 和 async/await 已被良好埋点；原始 callback 则未被埋点。

采样：在哪里丢弃 span

在大规模场景下对每个请求进行链路追踪是不可行的。一个 10M QPS 的服务每次 RPC 发出一个 span，每秒产生约 1 TB 的链路数据——超过了大多数公司全量遥测预算的总和。因此： 采样.

策略 运行位置 权衡
头部采样（SDK 层） 第一个接收请求的服务进行随机采样（例如，采样率 1%）。决策记录在 traceflags 比特位中，所有下游服务遵从此决策。 开销低。但你在 之前 尚不知道请求是否有价值（慢请求、错误）的情况下就做出了丢弃决定。大多数慢请求被丢弃。
头部采样（collector 层） Collector 接收所有 span，然后在转发前于 agent 层进行丢弃。 灵活性略高（可按服务设置采样率）。仍在"知道请求是否值得保留之前"丢弃。
尾部采样（后端层） 后端将所有 span 缓冲几秒钟。链路完整后，应用规则："保留所有错误链路，保留最慢的 1%，对其余的 0.1% 进行采样。" 能捕捉到有价值的链路。需要在缓冲窗口期将所有 span 保存在内存中——开销较大。常见于商业后端（Honeycomb、Datadog、Lightstep）。

生产环境大规模部署通常结合使用 针对错误的高概率头部采样 （"如果看到 5xx，设置 traceflags=01 使下游全部采样"）以及 尾部采样 用于在后端捕获慢链路。

结论："头部采样会丢弃慢请求；尾部采样以缓冲内存为代价捕获慢请求；根据目标是降低数据量还是保留有效信号来选择。"

指标与日志与链路追踪协同工作

OpenTelemetry 同样统一了指标和日志。使用模式：

  • 指标：RED（请求率、错误率、耗时）和 USE（利用率、饱和度、错误率）。按 RPC 方法统计计数器和直方图，导出为 OTLP 或 Prometheus 格式。
  • 日志：结构化日志行自动与活跃 span 关联（SDK 将 trace_id 。 span_id 注入日志记录中）。

三种信号（指标、链路追踪、日志）现在通过 trace_id关联。故障响应流程：" BalanceService.Debit 的 p99 延迟尖刺 → 找到慢链路 → 查看哪个 span 慢 → 查看该 span 的日志。"这是 Google SRE Book 之后的可观测性模型。

§10. RPC 层的限流

认证解决"你是谁"，授权解决"你能做什么"， 限速 解决"你能做多频繁"。它是服务网格抵御恶意调用方、失控循环和嘈杂邻居租户的防线。

本地限流 vs 分布式限流

两种模式：

  1. 本地（单实例）限流。 每个服务实例按调用方维护一个令牌桶。如果服务有 100 个副本，全局限流值为每个实例允许值的 100 倍。简单，但"全局速率"难以推断，且随副本扩缩容而漂移。
  2. 分布式（全局）限流。 所有副本共享一个计数器，通常由 Redis 或专用服务支撑。每次调用递增计数器；若超出限制则拒绝。单一事实来源，但增加了延迟和新的依赖。

令牌桶：经典算法

令牌桶：桶最多容纳 B 个令牌，以每秒 R 个令牌的速率补充。每次调用消耗一个令牌。若桶为空，调用被拒绝（或排队）。

突发容量: B 控制你能容忍的突发大小。 R = 100/sec, B = 200 表示持续速率 100/秒，但允许瞬间突发 200 次。

Redis 实现：每个（调用方，路由）对应一个 Redis key。Lua 脚本原子地执行：(a) 根据存储的时间戳计算自上次调用以来补充了多少令牌，(b) 将桶补充至最多 B，(c) 检查是否至少有一个令牌，(d) 递减并返回成功，否则拒绝。通过 Lua + Redis 单线程执行保证原子性。

延迟成本：每次 Redis 往返约 0.5–1 ms。在 1M QPS 下，Redis 集群需要仔细分片（按调用方 key 分片），且网络必须可靠。大多数服务网格在每个区域部署一个 Redis 集群以将往返延迟保持在区域内。

Envoy 的限流服务：sidecar 强制执行全局速率限制

Envoy 本身不实现分布式限流；它通过 envoy.filters.http.ratelimit filter 与外部服务集成。使用模式：

  1. 请求到达服务器的 sidecar（Envoy）。
  2. Envoy 向 限流服务 发起 gRPC 调用，携带描述符——例如， [("caller_spiffe_id", "spiffe://prod/ns/payments/sa/checkout"), ("route", "BalanceService.Debit")].
  3. 限流服务查阅其配置（每个描述符的限制和时间间隔）和 Redis 中的当前计数。
  4. 返回 OK 或 OVER_LIMIT.
  5. Envoy 转发请求或以 429 Too Many Requests （或 RESOURCE_EXHAUSTED （gRPC 对应状态码）拒绝请求。

参考实现是 envoyproxy/ratelimit （Lyft 的开源服务）。Lyft 在生产环境中以数十万 QPS 的规模运行它。

优势： 应用程序无需实现限流逻辑。 这是网格中的一个配置：

rate_limits:
  - descriptor_key: caller_spiffe_id
    descriptor_value: spiffe://prod/ns/marketing/sa/email-blast
    rate_limit:
      unit: SECOND
      requests_per_unit: 100

新增调用方？更新配置。发生突发？配置变更有版本控制并经过审查。

按方法、按优先级、按租户的限制

在大规模场景下，维度会相乘：

  • 按方法: BalanceService.Debit 比 BalanceService.GetBalance开销更大；对它们设置不同的限制。
  • 按优先级：关键路径的调用方（结账流程）比批处理任务（夜间对账）获得更高的配额。
  • 按租户：在多租户 SaaS 中，客户 A 的失控重试循环不能使客户 B 挨饿。租户 ID 成为限流描述符的一部分。

Stripe 以商户账户为单位在公共 API 边缘进行限流——某个商户行为异常的集成不能降低其他商户的流量。限流描述符是 (merchant_id, endpoint)，限制按商户设置。限制通过 X-RateLimit-Limit / X-RateLimit-Remaining / X-RateLimit-Reset 响应头对商户可见（见 §18）。

限流 vs 准入控制 vs 负载卸载

容易混淆，但它们是不同的原语：

  • 限流：基于 调用方的配额。若调用方超出其份额则拒绝。
  • 准入控制：基于 服务器的瞬时负载。若服务器 CPU 达到 90% 或队列深度为 N 则拒绝。不涉及"公平份额"的概念。
  • 负载卸载：过载时丢弃优先级最低的工作。需要对请求进行优先级标记。

生产环境将三者分层部署。限流在请求到达服务器前拦截恶意调用方。准入控制处理限流未能预测的负载（名义上合规的调用方引发的惊群效应）。负载卸载确保最重要的工作能够存活。

§11. 请求头传播

一个用户请求进入边缘后，在返回响应之前会扇出至 5–10 个服务跳。在此过程中，下游服务需要了解原始上下文：用户是谁、属于哪个租户、属于哪条链路、会话 ID 是什么。 请求头传播 是在各跳之间保持该上下文存活的规范，其典型故障模式是"在第 4 跳丢失用户上下文"。

传播哪些内容

每次内部 RPC 上的通用请求头：

请求头 用途 由谁设置 由谁读取
traceparent, tracestate W3C 链路追踪上下文 Tracing SDK / sidecar Tracing SDK / collector
x-request-id 幂等请求标识符（每个用户请求一个） 边缘入口 每个服务（用于日志关联）
x-user-id 已认证用户（真实用户，而非服务） 边缘认证层 每个服务（用于授权检查）
x-tenant-id 客户/组织上下文 边缘 租户范围的服务（DB 分片、计费）
x-session-id 用户会话 边缘 按会话有效性进行门控的服务
x-locale, x-timezone 本地化上下文 边缘（来自客户端） 渲染用户可见内容的服务
x-experiment-bucket A/B 测试分配 边缘实验评估器 按实验变更行为的服务
grpc-timeout 剩余截止时间 每跳重新计算 gRPC 框架
authorization （mTLS 或 token） 调用方身份（工作负载，有时包含用户） mTLS / 认证层 服务处理程序、OPA

传播规则：每个服务必须从入站请求中读取这些请求头，并将其重新附加到它发出的出站请求上。遗漏任何一个，下游服务就无从得知用户是谁。

"在第 4 跳丢失用户上下文"的故障模式

你用 Go 上线了一个新的中间层服务。六个月后，审计员问："请向我展示你们计费服务在二月份收到的所有归属于用户 X 的请求。"你查询日志，发现一半请求有 user_id 字段，另一半没有。排查发现：新中间层没有传播 x-user-id ，因为开发者使用了原始的 http.Client 而非标准内部客户端，忘记将入站上下文中的请求头复制到出站请求中。

这是经典的传播 bug。它对功能测试是不可见的（请求正常工作；只是下游服务不知道用户是谁），只有在有人尝试审计或归因时才会暴露。

各语言的标准修复方案

Go： context.Context 模式（ContextProp）。 惯用的 Go 模式是将 context.Context 穿透每个函数调用。请求头作为 context 值存储；HTTP/gRPC 客户端中间件自动从 context 中提取并将其添加到出站请求头中。规范：

func (h *Handler) Process(ctx context.Context, req *Request) (*Response, error) {
    // ctx has x-request-id, x-user-id, etc. bound to it by middleware.
    // Pass ctx through to every call:
    result, err := h.balanceClient.GetBalance(ctx, &balanceReq)
    // Client middleware reads ctx, propagates headers automatically.
}

规则： ； context.Background() 在处理程序中创建新 context。 这会断开传播链。Linter 可以发现这个问题；许多团队强制执行该规范。

Java：MDC（Mapped Diagnostic Context）。 SLF4J/Logback MDC 是一个存储关联 ID 的 ThreadLocal map。服务端 filter 从入站请求中提取请求头并调用 MDC.put("user.id", userId)。日志记录器在每行日志中包含 MDC key。出站 HTTP/gRPC 客户端拦截器读取 MDC 并重新附加为请求头。现代框架（Spring Cloud Sleuth、OpenTelemetry Java agent）已自动化此过程。异步边界（CompletableFuture, @Async）需要显式的 MDC 传播（框架通常提供封装器）。

其他语言: - Python: contextvars （PEP 567）是现代等价物，支持 ThreadLocal 和 asyncio。 Node.js: AsyncLocalStorage （Node 13+）用于异步感知的上下文。 Rust: tokio-tracing 提供了 Span 带属性传播的上下文。

Sidecar 辅助传播

配置完善的服务网格可以减轻应用程序的负担。 Envoy 的 tracing filter 和请求头转发 filter 可以配置为自动将入站请求头白名单中的内容复制到出站请求中。应用程序代码可以省略对白名单中请求头的显式传播（sidecar 在出站时处理）。

限制：sidecar 原样复制 正是 入站请求头。它无法进行转换（例如，解密用户 token、提取 user_id、为下游重新编码）。任何需要应用逻辑的操作仍需在应用中完成。

需要指出的反模式

  • 在每个服务跳重新推导用户身份：不要在 6 个服务中对用户进行 6 次认证。边缘认证一次并签发内部 token，下游服务信任该 token。Staff 级别的模式是"边缘签名身份请求头"——边缘盖上 x-user-id 及其短期 HMAC；下游服务以低成本验证 HMAC（约 10 µs vs JWT 的约 1 ms）。这是 GitHub、Shopify 和类 Slack 系统内部的做法。
  • 在请求头中放太多内容：每个请求头都消耗 HPACK 表空间和带宽。不要传输整个用户对象——传输 ID，让服务在需要时获取完整状态。
  • 忘记在出口处清除内部请求头：内部请求头（x-internal-debug, x-canary-shard）必须在响应发送给客户端之前在边缘被剥离，否则会将内部架构泄露给公众。

§12. 连接管理

RPC 服务网格所依赖的"物理"层是 TCP 连接。这些连接如何被池化、多路复用、保活和回收，大多数时候是不可见的，而在其余时间则是最严重故障的根源。

客户端连接池

一个朴素的客户端为每次 RPC 调用建立一个 TCP 连接。在 5,000 QPS 下，每秒每台服务器有 5,000 次 TCP 握手，每次握手消耗一个往返时间（数据中心内约 500 µs）加上 TLS 握手（TLS 1.3 再额外一次往返，TLS 1.2 则两次）。在任何应用程序工作开始之前，每次调用的延迟损耗为 1–2 ms，且内核会因临时端口压力而不堪重负（主机默认约有 28,000 个临时端口）。

连接池 是解决方案：为每个上游维护一个长连接池，复用于多次调用。连接池大小的经验法则：

  • 对于 HTTP/1.1： pool_size >= concurrent_in_flight = QPS × latency。在 5k QPS 和 10 ms 延迟下，连接池大小约为 50。
  • 对于 HTTP/2：一个连接可以多路复用 100–1000 个并发流。连接池大小缩减为"足够利用可用 CPU 的连接数"——通常每个 pod 每个上游 2–4 个连接（每个 CPU 线程一个）。

连接池本身是一种可能耗尽的资源（见 §13 问题 6）。对连接池获取的队列深度设置上限是强制要求。

HTTP/2 多路复用

HTTP/2 的主要优势： 一个 TCP 连接可以多路复用 100–1000 个并发流。 Stream 7（已传输 3 秒）不会阻塞 Stream 8（刚刚开始）。来自不同流的帧在网络上交错传输。

连接池大小的影响：- 每个上游保持一条 HTTP/2 连接，在中低规模下通常 足够了 。- 在以下情况下，多条连接会更有用： 流创建上限的可扩展性 (HTTP/2 SETTINGS_MAX_CONCURRENT_STREAMS （每条连接有流数量上限；开启第二条连接可使上限翻倍）以及 CPU 并行处理 （HTTP/2 编解码器可能成为单条连接的瓶颈；将负载分散到多条连接可分摊 CPU 压力）。- gRPC 官方建议：每个上游保持 1 条连接即可，直到连接饱和 MAX_CONCURRENT_STREAMS 或达到编解码器 CPU 上限，再考虑增加连接数。

连接绑定风险

连接绑定：一个长生命周期的流（尤其是类似“永久流式传输所有日志行”的服务端流 RPC）会绑定到某一条连接。若该连接底层的 TCP 出现停滞， 所有 共享该连接的流也会停滞（§13 问题 5 中提到的 TCP 层队头阻塞问题）。

风险：单个处理缓慢但未断开的流——例如客户端读取速度很慢的流式 RPC——会通过 HTTP/2 流量控制背压暂停该 连接的发送缓冲区，短暂阻塞同一连接上的其他流。典型症状：p99 延迟出现随机毛刺，且与请求落在哪条连接上高度相关。

缓解措施：- 积极轮换连接（每隔 N 分钟或 M 次请求后关闭并重新建立连接）。- 将“控制平面”（小型 RPC）与“数据平面”（大型流式 RPC）使用独立连接。- 调整单流流量控制参数（缩小窗口可迫使慢速消费者快速限流，而不会撑大连接缓冲区）。

Keep-alive 调优

Keep-alive 的三个层次：

  1. TCP keep-alive：内核每隔 N 秒发送一个探测包（Linux 默认为 7200 秒，即 2 小时）以检测失效对端。生产环境中响应太慢，建议调整为约 30–60 秒。
  2. HTTP/2 PING 帧：gRPC 每隔约 10–60 秒发送一个 PING。若在超时时间内未收到 PONG，则认为连接已失效并关闭重建。可通过以下配置调整： keepalive_time 。 keepalive_timeout.
  3. 应用层健康检查：带外 /healthz 轮询，与任何正在传输的流量相互独立。

Keep-alive 配置不当是神秘连接故障的常见原因。过于激进：PING 风暴会在数千条连接上消耗大量 CPU。过于宽松：失效连接在连接池中停留数小时，对其发起的每次调用都会失败，直到内核发现为止。

空闲连接回收

超过 N 分钟无流量的连接应关闭以释放资源。服务端回收（SETTINGS_MAX_IDLE_CONNECTIONS 或类似机制）是良好的工程规范。客户端侧：连接池会主动回收空闲连接。

风险：某个客户端 应该 每小时调用一次 B，但发现上一条连接已被回收。下一次调用需要支付握手成本。对于调用频率极低的调用方，可考虑通过定期 PING 保持连接活跃。

关闭时的连接排空

当服务端 Pod 正在被终止（滚动部署、缩容、节点排空）时：

  1. 服务端发送 GOAWAY ，其中 last-stream-id = N。
  2. 已有流（编号 ≤ N）正常完成。
  3. 客户端收到 GOAWAY 后，停止在该连接上开启新流，并向其他实例建立新连接。
  4. 服务端等待宽限期（通常为 30–60 秒）让流完成排空。
  5. 服务端关闭。

若无 GOAWAY，服务端终止会中断正在传输的请求；客户端会收到 UNAVAILABLE 并重试。有了 GOAWAY，关闭过程是优雅的，客户端可无缝故障转移。

同样的模式适用于 Envoy / sidecar：关闭时先发送 GOAWAY，等待排空，再退出。Kubernetes 的 preStop 钩子和终止宽限期可实现这一行为。

§13. RPC 固有的疑难问题

这些问题是该 技术固有的，无论你选择哪种框架——无论是支付、搜索、信息流还是 ML 推理——都会遇到。

问题 1：重试风暴（重试放大）

错误的解法。 "重试代价很低。对每个客户端都做三次指数退避重试，免费获得更高可靠性。"

如何出错 （支付扇出示意）。类 Stripe 规模的支付授权：一个请求扇出为 edge → 风控服务 → 余额服务 → 账本。每跳重试 3 次。余额服务发生 GC 暂停持续 2 秒，延迟飙升至 800 ms，上游调用在 500 ms 时开始超时。风控服务重试：3 次。Edge 重试风控服务：3 次。 3^4 = 81 倍放大。 余额服务此时正承受 81 倍正常负载，在流量被卸载之前始终无法恢复。

另一个场景的 t=0 追踪（搜索扇出）：搜索根节点 → 100 个叶子分片。某个分片发生 GC。p99 延迟飙升。根节点重试（3 次）。在 100 个分片并行重试的叠加效应下，叶子集群在部分容量上承受 300 倍放大 → 全集群降级。

正确的解法：重试预算。 Envoy 和现代 gRPC 客户端库对每个上游强制执行 每个上游的预算：重试量最多为成功流量的 10%（可配置）。若 B 正在以 10k QPS 成功处理请求，重试预算允许 1k 次/秒的重试；超出部分在客户端侧直接丢弃。关键在于预算是按上游计算的，而非按调用计算——否则并发失败的调用各自获得 10% 的预算，重新退化为放大问题。

重试的分层防护：- 带抖动的指数退避 backoff = min(base * 2^attempt, cap) + uniform(0, jitter)。抖动是不可或缺的；没有抖动，所有重试会在同一毫秒扎堆到达，形成惊群效应。- 只重试幂等调用 。读操作（getUserProfile）可以重试；写操作（POST /payments）不行，除非调用携带幂等键。- 遵守 deadline。 超过父请求 deadline 后的重试毫无意义。- 感知状态码的重试策略。 NOT_FOUND, INVALID_ARGUMENT, PERMISSION_DENIED 是确定性错误，永不重试。 UNAVAILABLE, DEADLINE_EXCEEDED, RESOURCE_EXHAUSTED 是重试候选。

问题 2：级联故障 / 熔断

错误的解法。 "如果 B 慢了，A 会等待。B 最终会恢复，A 也会恢复。"

如何出错 （ML 推理扇出示意）。一个推荐服务并行调用 50 个 ML 推理端点，p99 预算为 100 ms。其中一个推理端点开始耗时 5 秒（GPU 热降频，或模型部署异常）。其余 49 个健康端点在 80 ms 内响应。推荐服务等待第 50 个。它与该端点的连接池耗尽。线程池耗尽（每个线程都阻塞在慢端点上）。推荐服务此时对 所有用户都降级了，而不仅仅是请求恰好落到慢推理端点的用户。 "慢"比"宕机"更糟糕。

数学推导（Little's Law，再次登场）：连接池 200，正常在途请求 50。延迟跳升 50 倍 → 所需在途请求数跳升至 2500。连接池只有 200。每个后续调用者都阻塞在连接池获取上。应用线程在信号量上堆积。应用停止接受新请求。

正确的解法：熔断器。 在滑动窗口内追踪上游的错误率或延迟。当超过阈值时（例如最近 100 次请求有 50% 失败，或 p99 > 5 倍正常值），打开熔断器：

States:    [CLOSED] --errors above threshold--> [OPEN] --after timeout--> [HALF_OPEN]
              ^                                                                |
              |                                                                |
              +------------success on probe call-------------------------------+
  • CLOSED：正常运行，追踪错误/延迟。
  • OPEN：所有调用快速失败，返回 UNAVAILABLE ，持续 30 秒。甚至不建立连接。 错误在微秒级返回，而非秒级。 这会立即释放连接池。
  • HALF_OPEN：超时后，允许少量探测调用通过。若成功则关闭熔断器；若失败则回到 OPEN 状态。

Hystrix（Netflix，2012 年）使这一模式广为人知；resilience4j 是其现代 Java 继任者；Envoy 原生实现了这一功能。更深层的洞察： 熔断器保护的是调用方，而非被调用方。 快速失败的调用方仍能为其自身的调用方提供服务（返回默认值、缓存响应或优雅错误）。而阻塞在慢速被调用方上的调用方则无能为力。

问题 3：Deadline（而非 timeout）的传播

错误的解法。 "每个 RPC 调用都设置 1 秒超时。如果任何调用在 1 秒内未返回，就失败。"

如何出错 （信息流渲染示意）。用户打开 LinkedIn 信息流；边缘服务的预算为 500 ms。信息流中间层（由边缘调用）对排序服务有自己的 1 秒超时。排序服务对候选生成服务有 1 秒超时。候选生成服务对特征存储有 1 秒超时。特征存储对多个上游数据库有 1 秒超时。每一跳都说“我没问题，在 1 秒内返回了”，但累积深度超过了 500 ms 的用户预算 4–5 秒，用户在这一切完成之前就放弃了。更糟糕的是：当面向用户的边缘服务已经终止父请求时，整个扇出子树仍在运行，浪费 CPU 计算着没人会读取的结果。

正确的解法：传播 deadline（而非 timeout）。 Deadline 是一个绝对时间戳（“必须在 14:00:00.500 UTC 前完成”）。当 A 调用 B 时，A 传播 deadline = now + (deadline_received_from_parent - now) - safety_margin。gRPC 的 grpc-timeout 是一个 相对 値，在每跳重新计算：

Edge:     receives request at t=0, total budget 500 ms.
Edge → A: grpc-timeout = 450m (after subtracting internal time + safety)
A → B:    A spent 50ms internally, propagates grpc-timeout = 400m
B → C:    B spent 30ms, propagates grpc-timeout = 370m
...
At any hop, if (deadline - now) <= 0, the call fails immediately
with DEADLINE_EXCEEDED without being dispatched.

Deadline 是调用的一部分。 每一跳减去已耗时间。 扇出树在 deadline 耗尽时自我截断；下游调用不会浪费 CPU 计算将被丢弃的结果。同一原语解决两个问题：长时间运行的调用不会超出面向用户的预算，且僵尸调用在父请求放弃后不会继续浪费资源。

REST + JSON 不提供这一能力；你必须在每个服务中自行实现，而十个团队会以十种不同的方式实现它。

问题 4：fleet 规模下的 Schema 演进

错误的解法。 "我们手动协调 proto 变更。"

如何出错 （多租户示意）。800 个服务。一位开发者在共享的 user_id → member_id 中将 .proto 重命名，理由是“名称只是代码生成的产物”。他还将字段编号 1 复用给一个新的 is_premium bool 字段。旧的发送方将字符串写入字段 1；新的接收方尝试将其解码为 bool。解码器要么崩溃（5xx），要么静默读取到垃圾数据。

复盘：该问题在 02:00 UTC 上线。区域 A 的旧调用方二进制文件中仍有以字段 1 为字符串的排队请求。刚部署的新服务端二进制文件期望字段 1 为 bool。新服务端解码字段 1，得到乱码。状态损坏。回滚 .proto，等待二进制文件在 fleet 中重新部署，撰写事后分析。

这并非假设情景。Google 已公开发布过至少一次由单个 proto 字段编号复用引发的 fleet 范围内长达 4 小时局部中断的事后分析。

正确的解法：Schema 是契约。 在 CI 中强制执行：

  • buf breaking 或 protolock 将当前 PR 中的 proto 与 main 分支中提交的版本进行比较。对已有字段编号、字段类型的修改，或未经 reserved 的删除操作，均为 CI 阻断性错误。
  • 字段编号一旦分配并发布，即不可更改。 将其视为主键对待。
  • 保留字段编号和名称 每当删除字段时均需声明保留。
  • Schema 注册中心：每台主机上的每个二进制文件都从中央注册中心拉取统一版本的 proto 定义。部署前需通过注册中心验证。
  • CI 中的兼容性测试：启动一个基于版本 N 构建的真实客户端，向一个基于版本 N+1 构建的真实服务端发起请求，运行集成测试。反向同理。

在 fleet 规模下（数百个服务、数千名工程师），“我负责共享服务的 Schema 审查流程”是一项有说服力的 Staff 级别成就。

问题 5：队头（HoL）阻塞

错误的解法。 "HTTP/1.1 加 keep-alive 就够了。"

问题所在。 HTTP/1.1 流水线在实践中是损坏的；代理和服务端按顺序在连接上串行化请求。如果请求 1 耗时 3 秒，同一连接上的请求 2、3、4 会被阻塞。这是应用层 HoL。变通方案：为每个后端开启多条连接（50 条以上）。Socket 开销急剧膨胀；内核在临时端口压力下不堪重负；ALB / 负载均衡器的连接数限制开始成为瓶颈。

HTTP/2 的解法。 流在帧层多路复用。Stream 7 可以停滞而不阻塞同一连接上的 Stream 8、9、10。每条连接支持 100–1000 个流。

但 HTTP/2 在 TCP 层仍然存在 HoL 阻塞。 TCP 保证字节的有序交付。如果携带 Stream 7 某个片段的单个 TCP 数据包丢失，重传会停滞 所有后续字节，包括属于 Stream 8、9、10 的、已乱序到达的字节。某条流上的数据包丢失会悄无声息地延迟共享同一连接的其他不相关流。

HTTP/3（QUIC）的解法。 QUIC 运行在 UDP 之上，在应用层管理流的排序。每个 QUIC 流有独立的有序交付。Stream 7 上的数据包丢失只停滞 Stream 7，Stream 8、9、10 立即交付。在高丢包环境（移动网络、跨区域链路）中，尾延迟改善显著。Cloudflare 和 Google 在边缘使用 HTTP/3 正是出于这个原因。内部东西向流量目前仍主要使用 HTTP/2，因为数据中心内链路质量良好，HTTP/2 多路复用已经足够。

重点： "“TCP 层队头阻塞”是 HTTP/3 over QUIC 对于扇出延迟意义重大的具体原因" 。

问题 6：服务端慢但未宕机（连接池耗尽）

我们在问题 2 中提到过这一点；这里给出在各处反复出现的具体机制。

错误的解法。 "连接池按峰值配置，没问题。"

问题所在（Little's Law，再次登场）。 in_flight = QPS × latency连接池 200，正常 4k QPS、10 ms 延迟 = 40 个在途请求，5 倍余量。延迟劣化 50 倍至 500 ms（不是“宕机”——只是不健康）。所需在途请求数 = 200k。连接池大小 200。每个后续的获取操作无限期阻塞。应用线程在信号量上堆积。应用停止接受新请求。应用的调用方将其视为“宕机”。

分层的正确解法:

  1. 在连接池获取处设置队列深度上限。 如果超过 N 个线程在等待连接池，立即让新的获取操作失败，返回 RESOURCE_EXHAUSTED。快速失败；让熔断器打开。
  2. 对慢速上游使用熔断器 ——打开后快速拒绝进一步调用，排空连接池。
  3. 按上游隔离（bulkhead）。 为每个上游设置独立的线程池 / 连接池。若 B 出现问题，A 对 C 的调用不受影响。Hystrix 将此称为“舱壁隔离”——独立的防水舱，一个进水不会让整艘船沉没。
  4. 服务端负载卸除。 服务端将 SETTINGS_MAX_CONCURRENT_STREAMS 设置得足够低，使过载的服务端停止接收新流，而不是接收后变慢。
  5. 在每一层强制执行 deadline。 已超过父请求 deadline 的调用在本地被终止，不再消耗更多资源。

顺序很重要：deadline + 重试预算 + 熔断器 + 舱壁隔离 + 负载卸除是分层防护机制，每一层捕获上一层遗漏的问题。

问题 7：服务端超时短于客户端 deadline

错误的解法。 "我到处设置超时；如果有什么慢了，就会快速失败。"

如何出错客户端 A 以 1 秒 deadline 调用服务端 B。B 对其发起的数据库调用有自己的 500 ms 内部超时。数据库很慢（800 ms）。在 t=500 ms 时，B 的数据库超时触发，B 向 A 返回 DEADLINE_EXCEEDED 。 A 仍有 500 ms 的预算，于是 A 重试该调用。 B 开始处理重试——但原来的工作实际上并没有停止，因为 B 的数据库调用已经在途，数据库 仍在 以 t=800 ms 的速度慢慢返回结果（针对原来被放弃的调用），以及 t=1300 ms 的结果（针对重试调用）。 双倍工作，双倍数据库负载，而 A 可能还会再重试一次。

这种病态会累积：慢速下游导致上游重试，进一步 放大 慢速下游的负载，延长慢速时间。这与重试风暴的模式相同，但根源是 内部 超时配置不当，而非客户端的盲目重试。

正确的解法：服务端超时应长于客户端的 deadline，或者——更好的做法——直接从 grpc-timeout 中推导。模式：B 从入站请求中读取 grpc-timeout ，并将 该值 作为所有内部操作的预算。不设置比客户端 deadline 更短的独立内部超时。

Stripe 的可靠性工程团队写过这一精确的模式：他们的内部超时始终 ≥ 调用方的 deadline + 安全裕量，这样如果调用方愿意等待，服务端就不会抢先终止工作。

问题 8：Deadline 传播被遗忘

错误的解法。 "每个服务都有自己的超时配置；没问题。"

如何出错服务 A 以 grpc-timeout: 500m被调用。A 发起 5 次下游调用。每个下游都配置了自己的静态超时“1 秒”，因为这是当时的团队默认值。即使 A 的调用方在 500 ms 时就会放弃，下游服务仍会愉快地花费最多 1 秒计算没人会读取的答案。 父请求被放弃后，扇出子树继续消耗 CPU。

这在 REST over HTTP/1.1 架构中非常普遍，因为 deadline 传播并未内置于协议中。人们用 X-Deadline header 重新发明它，半途而废的实现比比皆是，而那些没有实现的部分让僵尸工作继续存活。

正确的解法：将 deadline 传播作为架构层面的不变量。每个 gRPC 服务端的框架自动从入站请求中读取 grpc-timeout ，以此计算 now + remaining 作为 deadline，并（如果发起下游调用）将 (deadline - now - safety_margin) 传播到每个下游。只要你将请求上下文透传，大多数 gRPC stub 库默认就会这样做。原则是“永远不要在 fabric 中途启动新的 deadline 树”。

对于 REST，惯例是使用 X-Request-Deadline: <unix-millis> 或 X-Request-Timeout: <ms>。两种方式都可以；关键是统一执行。

问题 9：熔断器振荡

错误的解法。 "熔断器处理故障。设置 50% 错误阈值、30 秒打开时长，搞定。"

如何出错上游 B 处于降级但未宕机状态——70% 的调用成功，30% 失败并返回 DEADLINE_EXCEEDED。超过错误阈值后，熔断器打开。30 秒后，熔断器转入 half-open 状态并发送一次探测调用。探测成功（命中那 70% 中的一次）。熔断器关闭。几秒内，失败率再次超过 50%，熔断器再次打开。 熔断器在 OPEN 和 CLOSED 之间每约 30 秒振荡一次。 每次关闭期间，调用方将全量流量打向 B 并被失败猛击；每次打开期间，调用方快速失败，不向 B 施压（这本可帮助 B 恢复），但也无法消化积压的待处理工作。

断路器 既不 保护 B（B 仍会承受全量流量的突发冲击） 也不 能很好地服务调用方（调用方在"全部快速失败"与"部分成功"之间反复震荡）。

实际修复方案:

  1. 半开状态需要 连续多次成功 才能完全闭合。Envoy 的异常检测有 success_rate_minimum_hosts 。 consecutive_5xx；半开状态需要 N 次成功（通常为 5 次）才能重新闭合。
  2. 迟滞机制：开启与关闭的阈值不同。当错误率 > 50% 时开启；仅当错误率 < 10% 时才关闭。若没有这个间隙，断路器会在阈值附近的任何速率下反复抖动。
  3. 渐进式重新接入：不再是"开启 → 关闭 = 全量流量"，而是逐步提升流量。先 1% 探测，再 10%，再 50%，最后 100%。Resilience4j 称此为"半开状态下允许的调用次数"。
  4. 按实例而非按服务配置。 每个上游实例配置一个断路器比按服务配置更有价值。一个有问题的 Pod 不会导致所有副本的断路器打开；异常检测只会将该 Pod 驱逐。

Staff 级洞察：一个频繁抖动的断路器比没有断路器更糟糕，因为调用方无法对其行为建立稳定预期。在局部故障下保持稳定才是真正的目标，而不仅仅是快速开启或关闭。

§14. 故障模式演练

服务器在请求处理过程中崩溃（位于 DATA 与 TRAILERS 之间）

客户端已发送 HEADERS + DATA。服务器开始处理，并开始写入响应 DATA，随后进程崩溃（OOM、段错误、kill -9）。

  • TCP RST 或 FIN 到达客户端。HTTP/2 编解码器检测到连接中断。该连接上所有进行中的流均收到通知。
  • gRPC stub 收到 UNAVAILABLE ，原因为"连接在未发送 TRAILERS 的情况下关闭"。
  • 恢复：如果是幂等操作且在重试预算内，客户端对另一个实例发起重试。网格的异常检测会在连续失败约 10 秒后将死亡主机驱逐。
  • 持久性要点：在 RPC 层面没有任何记录。响应从未在任何地方提交。客户端需自行承担。对于非幂等操作，客户端必须事先写入幂等性键，以防重试导致副作用被重复执行。

服务器在请求间隙崩溃（空闲状态）

  • 同样的 TCP RST。连接池检测到连接失效后将其移除，并向健康实例新建连接。
  • 异常检测将该主机驱逐。经过隔离期后，网格可能将其重新加入。

Sidecar（Envoy）崩溃

  • 应用的 gRPC stub 指向 localhost:9090。localhost 连接被拒绝。
  • 应用应快速失败并返回 UNAVAILABLE。Kubernetes 会在约 5 秒内重启 sidecar。
  • 防御手段：Envoy 以静态链接方式部署，由 supervisor 负责重启，并拥有自己的看门狗。更常见的故障是 xDS 推送配置错误导致 Envoy 拒绝配置并使用过期配置提供服务。
  • 持久性要点：Pod 是最小单元。如果 sidecar 无法恢复，删除该 Pod，调度器会拉起一个新的。

控制平面（xDS）中断

  • 各 Sidecar 缓存最后一次已知的正确配置。现有流量继续正常路由。
  • 故障影响：新服务无法注册，证书无法轮换，新路由无法下发。
  • 恢复：控制平面通过高可用副本（底层使用 etcd / Raft）实现故障转移，秒级完成。
  • 关键原则：数据平面必须在控制平面不可用时继续工作。这正是控制平面/数据平面分离的核心意义。若控制平面中断 30 分钟就导致所有 RPC 流量中断，则说明架构设计存在缺陷。

客户端与服务端机架之间的网络分区

  • 分区中的 mTLS 连接失败。异常检测将不可达一侧的服务器驱逐。
  • 服务发现通过健康检查感知到分区，并停止跨分区路由。
  • 调用快速失败并返回 UNAVAILABLE。断路器打开。重试快速失败。调用方优雅降级。
  • 脑裂注意事项：仅当被调用服务是有状态服务时才相关。无状态 RPC 不存在脑裂——两侧可以独立地为能够到达它们的调用方提供服务。
  • 持久性要点：所有持久化状态存储在数据库中，而非 RPC 层。RPC 是传输通道，数据存储在其他地方。

服务器实例永久丢失

  • 网格通过健康检查（被动：基于连续 5xx 的异常驱逐；主动：定期对 /healthz 或 gRPC reflection）进行 L7 健康检查。
  • 服务注册中心移除该实例，其他实例承接其流量。
  • 自动伸缩器可能会拉起一个替代实例。

慢而不挂（最坏情况）

通过断路器 + 异常检测 + 负载卸载进行恢复。网格必须区分"慢"与"挂"，并采用不同的补救措施：死亡实例被驱逐；慢速实例由断路器保护。框架为此提供了原语但不会自动解决——参见 §13 问题 2。

§15. 流式传输模式

gRPC 支持四种调用模式，其中三种是流式的。了解哪种模式适合哪类问题，以及何时应选择 其他 而非 gRPC 流式传输，是"理解语义"与"只懂协议"之间的分水岭。

四种 gRPC 调用模式

定义于 .proto:

service ExampleService {
  rpc UnaryCall(Request) returns (Response);
  rpc ServerStreaming(Request) returns (stream Response);
  rpc ClientStreaming(stream Request) returns (Response);
  rpc BidiStreaming(stream Request) returns (stream Response);
}

底层机制：在所有四种模式中，HTTP/2 流只建立一次，每个 Request 或 Response 是 DATA 帧中一条带长度前缀的 Protobuf 消息。框架通过在最后一帧中设置 END_STREAM 并在 TRAILERS 中携带 grpc-status.

来表示流结束。

服务端流：单次请求，多个响应形态

：客户端发送一个 Request，服务端发送一系列 Response 消息，随后是 TRAILERS。: - 使用场景推送通知 ：客户端建立长连接流；服务端在通知到达时推送。Slack、Discord 以及竞争性交易平台均采用此模式。 -日志/事件流 Tail(service="api", since=10min)：从服务器追踪日志——客户端发送 ，服务端以流的形式输出日志行。 -数据库变更流 ：复制客户端订阅来自主节点的 WAL（Write-Ahead Log）事件流。 -搜索类"全量结果"

：一个可能产生数千条匹配的查询。服务端边查找边流式输出；客户端渐进式渲染。为什么流式传输优于轮询 WINDOW_UPDATE：推送语义：没有轮询间隔带来的"尾延迟"。HTTP/2 流量控制在消费者慢时自然限制生产者（每流 CANCELLED）。单 TCP 连接、单 HTTP/2 流，编解码高效。取消路径简洁：客户端关闭流，服务端收到

URL 版本控制：

经典的 REST 方式。每个主版本使用独立的 URL 前缀。客户端显式固定到特定版本。 curl优点：简单、直观，可通过

API。 2024-12-15 使用方：Stripe（主要基于日期，例如通过 header——见下文）、Twilio（URL 版本控制）、大多数公共 REST API。

Header 版本控制： Accept content-type

IETF 推荐方式： Accept: application/vnd.example+json;version=2。URL 保持稳定；版本通过协商的 content type 传递。Stripe 以其知名的基于日期的 header 版本控制著称： Stripe-Version: 2024-12-15.

优点：URL 更简洁（一个资源，多种表示）。服务端可以按请求选择应用的版本，甚至混合版本。基于日期的版本控制与"兼容性快照"语义完美契合。

缺点：调试不直观——需要知道要设置哪个 header 才能看到正确版本。缓存层可能需要基于 header 变化（Vary: Accept 或 Vary: Stripe-Version）。工具支持参差不齐（CDN、网关）。

gRPC：字段编号稳定性 + 新增 RPC 方法

gRPC 从根本上不存在 URL 意义上的"API 版本"。版本控制规范是：

  • 消息中的字段编号不可变。 一旦发布，永不复用，永不改作他用。
  • 新增字段始终安全 （旧客户端会忽略未知 tag）。
  • 删除字段: reserved 应废弃它，永不复用该编号。
  • 变更方法语义：不允许。新增一个方法（GetUserV2）并废弃旧方法。

Protobuf 字段编号契约（§4a）是整个版本控制规范。URL 中没有版本号。隐式版本即"当前二进制所使用的 schema"。

对于 服务 级别的破坏性 schema 变更，模式如下：使用新包名添加新的服务定义（user.v2.UserService）。旧调用方继续调用 user.v1.UserService。服务端同时实现两者。待所有调用方迁移完成后，移除 v1.

废弃时间线

"v1、v2、v3 同时运行 18 个月"的现实情况。大型公司的真实废弃时间线：

阶段 时间点 操作
发布 v2 t=0 公告。迁移指南。两个版本同时存在。v1 调用方日志可见。
软废弃 t+3 个月 文档将 v1 标记为已废弃。新客户默认使用 v2。v1 仍完全支持。
警告 t+6 个月 v1 响应中包含 Deprecation HTTP header。主动联系已知 v1 调用方。
硬废弃 t+12 个月 v1 响应开始对新账户返回错误，现有调用方享有祖父条款保护。
日落 t+18 个月 v1 返回 410 Gone。迁移完成，否则客户端将出现故障。

Stripe 的"API 版本"本质上是无限的：每个客户被固定在注册时的版本，Stripe 多年来维持兼容性。内部服务以懒惰方式升级版本。AWS API 类似——你仍然可以调用 s3:PutObject 针对 2006 年的版本。

对于内部东西向流量，时间线是几个月，而非几年。但同样的纪律适用：在没有警告的情况下永远不要打破契约；提供迁移路径；保持旧版本存活足够长时间，以便每个调用方都能在不紧急的情况下完成迁移。

版本控制的陷阱

  • 耦合发布："我们将在下个冲刺部署 v2 时弃用 v1"是幻想。你将同时运行两者长达数月。
  • 隐式版本漂移：在以下情况下进行分支的代码 if (version == "v1") {} else {} 积累成意大利面条式代码。更好的做法是在独立文件中分别实现干净的 v1 处理器和干净的 v2 处理器；其中一个路由到另一个。
  • 忘记记录版本：当事故发生时，知道是哪个 API 版本产生了错误响应至关重要。始终在指标和日志中输出已解析的版本。
  • "v2 只是修复了几个问题的 v1"：不要为了表面改动就发布"v2"。向 v1 添加字段是非破坏性的。只有在存在 真正 破坏性变更时才发布新的主版本。

§17. GraphQL 深度解析

GraphQL 在 §3 和 §25 中作为适合 BFF（Backend For Frontend）层的方案被提及。更深层的问题是 为什么 ——而答案需要理解 GraphQL 的具体机制：N+1 查询问题、DataLoader 批处理、持久化查询以及 Apollo Federation。

GraphQL 简介

基于类型化模式（SDL — Schema Definition Language）的查询语言：

type User {
  id: ID!
  displayName: String!
  posts(limit: Int = 10): [Post!]!
}

type Post {
  id: ID!
  title: String!
  author: User!
  comments: [Comment!]!
}

type Query {
  user(id: ID!): User
}

客户端发送查询，指定 正是 它想要的字段：

query {
  user(id: "42") {
    displayName
    posts(limit: 3) {
      title
      comments {
        text
      }
    }
  }
}

服务器通过调用以下方式解析查询 解析器 ——每个字段一个函数。结果树与查询树完全匹配。

N+1 问题

典型的性能病态。对于上述查询，朴素的解析器执行：

  1. user(id: "42") 解析器运行 → 1 次数据库调用： SELECT * FROM users WHERE id = '42'.
  2. user.posts 解析器运行 → 1 次数据库调用： SELECT * FROM posts WHERE author_id = '42' LIMIT 3。返回 3 篇帖子。
  3. 对于每 3 篇帖子， post.comments 解析器运行 → 3 次数据库调用: SELECT * FROM comments WHERE post_id = '...'。共返回 N 条评论。

总计：单个用户 1 + 1 + 3 = 5 次数据库调用。扩展到每篇帖子都有评论的 50 篇帖子 → 1 + 1 + 50 = 52 次调用。 N+1：一个"父"查询产生了 N 个本应被批处理的"子"查询。

根本原因：GraphQL 解析器是 按字段编写的，对兄弟节点没有可见性。 post.comments 解析器对每篇帖子运行一次，完全不知道另外 49 篇帖子即将做同样的事情。

DataLoader：批处理与请求级缓存

DataLoader 是经典的解决方案，最初由 Facebook（Lee Byron）提出。它是一个请求级批处理原语：

  • 在单个请求中，当 50 个解析器各自调用 commentsLoader.load(postId)，DataLoader 缓冲这些 ID。
  • 在当前 tick 结束时（事件循环），DataLoader 调用单个用户定义的批处理函数： batchCommentsByPostId([p1, p2, ..., p50]).
  • 批处理函数发起一次数据库调用： SELECT * FROM comments WHERE post_id IN (...).
  • DataLoader 将结果返回给每个等待中的解析器。

N+1 折叠为 N+1 次 已批处理的 调用（查询树的每个"层级"一次），无论存在多少兄弟节点。

DataLoader 还会缓存 请求级：如果 userLoader.load(42) ：如果在同一请求中被两处调用，则会被去重。跨请求不缓存——每个请求获得一个新的 DataLoader，以避免陈旧缓存的风险。

Staff 级别的规范： GraphQL 解析器中的每个数据库/RPC 调用都必须通过 DataLoader。 如果开发人员添加了原始查询，N+1 问题就会重现。

持久化查询（Automatic Persisted Queries — APQ）

GraphQL 查询可能很大。单个移动端页面的查询每次请求可能有 5–10 KB 的 GraphQL 文本。在大规模场景下，每次调用都会产生带宽和解析成本。

持久化查询：客户端发送查询的 哈希值 而非查询文本。服务器预先注册了查询（通过构建工具，或通过"首次请求初始化缓存"协议）。服务器查找哈希，执行已注册的查询。

Apollo 的 APQ 流程：1. 客户端计算 SHA256(query)。 2. 第一次请求：只发送哈希。服务器返回 PersistedQueryNotFound。 3. 客户端重试，同时发送哈希和查询。服务器存储映射关系。 4. 后续请求：只发送哈希。服务器执行存储的查询。

带宽节省：5 KB 查询 → 64 字节哈希。在 1M QPS 下，5 GB/s → 64 MB/s，对移动端而言是巨大的收益。

持久化查询还可以作为安全控制——如果服务器只接受预注册的查询，攻击者就无法使用新查询探测模式。这就是 Shopify、GitHub 和 Netflix 使用的"操作白名单"模式。

Apollo Federation：分布式 GraphQL

朴素的 GraphQL 有一台服务器实现所有解析器。在大规模场景下，这就变成了一个单体——每个团队都向同一个进程贡献代码。 Apollo Federation 是"跨多个服务的 GraphQL"的答案：

  • 每个服务是一个 子图（subgraph） 拥有某些类型。服务 users 拥有 User；服务 posts 拥有 Post.
  • 一个 网关（gateway） （Apollo Router 或旧版 Apollo Gateway）将各子图组合成一个超图（supergraph）模式。
  • 客户端向网关发送一个 GraphQL 查询。网关规划为查询的每个部分调用哪些子图，在可能的情况下并行调用，并组合结果。
  • 类型可以 扩展（extended） ：（跨子图） Post ：（由 posts拥有的）可以拥有一个 author: User 字段，其 User.displayName 由 users 子图解析。网关通过 @key 指令处理跨子图的拼接。

Federation 让每个团队拥有自己的子图并独立发布。缺点：网关是一个新的关键基础设施，具有非平凡的查询规划成本。

GraphQL 的适用场景与不适用场景

适用场景: - 移动应用：蜂窝带宽宝贵，过度获取数据意味着真实的金钱成本。页面精确指定它所需要的内容。 - BFF （Backend For Frontend）：每个客户端一个 BFF（Web BFF、iOS BFF、Android BFF）是一种可行的架构；GraphQL 非常适合"将多个后端组合成客户端所需响应"的工作。 - 面向多样化客户端的外部 API：GitHub 的 GraphQL API 是因为 REST API 对仪表盘等工具存在过度获取问题而添加的。GraphQL 客户端按需请求所需字段。 - 联邦化组织：多个团队，一个统一的外部 API。Federation 让每个团队拥有一部分。

不适用场景: - 内部东西向高吞吐量：GraphQL 增加了解析、查询规划、解析器分发的开销——在低规模下无关紧要，在高规模下则难以接受。内部调用方已知道他们需要哪些字段；查询语言对他们毫无益处。 - CRUD 密集型应用：GraphQL 中的写操作（mutations）使用起来很笨拙且不可组合。REST + JSON 完全够用。 - 缓存密集型应用：HTTP 缓存基于 URL 工作；GraphQL 查询是 POST 请求体。有变通方案（持久化查询变为 GET 请求），但阻抗不匹配是真实存在的。 - 模式实际上并未统一时：federation 只在 类型 共享时才有帮助。如果各服务拥有不相交的数据，你不需要 federation——只需一个薄薄的 BFF 独立调用每个服务即可。

§18. 大规模 REST 设计

REST 是南北向流量的正确答案，但"REST"涵盖了从"相当不错"到"令人痛苦"的广泛设计范围。Staff 级别的 REST API 在资源建模、幂等性、分页和响应头方面有严格的规范。

资源建模：名词，而非动词

核心 REST 规范：URL 命名 资源；HTTP 方法命名 操作.

好的做法： - GET /users/42 ——获取用户。 - POST /users ——创建用户。 - PUT /users/42 ——替换用户。 - PATCH /users/42 ——部分更新。 - DELETE /users/42 ——删除。 - GET /users/42/posts ——用户 42 拥有的帖子集合。 - POST /users/42/posts ——为用户 42 创建一篇帖子。

不好的做法（REST 之上的 RPC 风格）： - POST /createUser, POST /getUser, POST /updateUser.

资源建模的优势：HTTP 缓存有效（对稳定 URL 的 GET 请求可缓存），幂等性定义明确（PUT/DELETE 按规范是幂等的），组合自然（集合嵌套清晰）。

幂等性键头部：Stripe 的模式

问题：POST 不是幂等的。客户端在网络抖动后重试 POST /payments 网络抖动后可能会对客户收费两次。

解决方案：使用 Idempotency-Key 头部。客户端生成一个 UUID；服务器存储 (idempotency_key) → (response)。重试时，服务器检测到该键已被见过，返回相同的响应而不重新执行。

线上模式：

POST /v1/charges
Idempotency-Key: 8e7c2a5d-3f4b-4abc-9d12-c1e5b6f8a907
Content-Type: application/json

{
  "amount": 4999,
  "currency": "usd",
  "source": "tok_visa"
}

服务端：1. 查找幂等性键。如果找到，返回已存储的响应。完成。 2. 如果未找到，对该键获取锁，执行操作。 3. 存储 (key) → (request_hash, response)。 4. 返回响应。 5. 使用相同键的未来调用看到已存储的响应。

Stripe 的具体规范：幂等性键有效期为 24 小时；如果请求体与原始请求不同，则返回错误（这样客户端就不会意外地为不同操作重用一个键）；键的作用域限定于 API 端点。

这种模式使 POST 可以安全地重试，这是任何写密集型 API 弹性的基础。它是"写操作的重试安全性"的经典真实案例。

分页：偏移量 vs 游标

偏移量分页: GET /posts?offset=1000&limit=20。实现简单；每个开发人员都熟悉。 在大规模场景下会灾难性地失败:

  • offset=1000000 需要数据库扫描 100 万行后丢弃它们。每页 O(offset+limit)。
  • 如果在第 1 页和第 2 页之间插入了新行，偏移量会发生偏移，用户会看到重复内容或遗漏条目。
  • 对缓存不友好：每一页都是唯一的。

游标分页: GET /posts?cursor=opaque_token&limit=20。服务器返回游标之后的行；响应中包含 next_cursor。游标编码最后一行的排序键（例如， created_at + id).

线上格式：

GET /v1/posts?limit=20
→ { "data": [...], "next_cursor": "eyJ0IjoxNzE0..." }

GET /v1/posts?cursor=eyJ0IjoxNzE0...&limit=20
→ { "data": [...], "next_cursor": "eyJ0IjoxNzE1..." }

优点： - 无论翻页多深，每页 O(limit)。 - 插入安全：新行不会移动游标。 - 可缓存：每个游标唯一标识一个切片。

缺点： - 不能"跳转到第 47 页"——严格顺序。 - 游标不透明性至关重要（客户端不应解析游标；服务器可以更改游标格式）。

生产环境下的大规模场景始终使用游标分页。Twitter 的 API、Stripe 的 API、GitHub 的 API——全部基于游标。"偏移量在大规模下失效"的故障模式出现在大客户深入分页列表时，数据库开始每页需要 10 秒。

速率限制响应头（X-RateLimit-*）

在响应头中返回速率限制元数据，让客户端无需试错即可自我调节：

HTTP/1.1 200 OK
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 847
X-RateLimit-Reset: 1716163200

GitHub 的约定。变体： - X-RateLimit-Limit：上限。 - X-RateLimit-Remaining：当前窗口内剩余的次数。 - X-RateLimit-Reset：窗口重置时的 Unix 时间戳。 - Retry-After: 60 在 429 响应中：重试前等待多长时间。

现代 IETF 草案头部： RateLimit-Limit, RateLimit-Remaining, RateLimit-Reset （不含 X- 前缀）。一些 API 在过渡期间同时提供两者。

HATEOAS 的理论与实践

HATEOAS （Hypermedia As The Engine Of Application State）是"真正 REST"的教条：响应中包含相关资源的链接，客户端通过跟随链接而非构建 URL 来导航 API。

理论模式：

{
  "id": 42,
  "name": "Yifan",
  "_links": {
    "self": "/users/42",
    "posts": "/users/42/posts",
    "delete": "/users/42"
  }
}

实际上，几乎没有人认真地这样做。 原因： - 客户端是根据文档编写的，而不是在运行时探索 API。这些链接是死重量。 - 工具（SDK、代码生成）不受益于 HATEOAS。 - 纯粹的 REST 遵循者是少数；"带 JSON 的 HTTP API"才是人们在 2026 年所说的 REST。

务实的例外：分页游标（"next_cursor"）是一种 HATEOAS 风格的设计， 确实 在大规模场景下存活，因为游标是不透明的，客户端确实无法构造它。除此之外，HATEOAS 仍停留在理想层面。

§19. Connect RPC 与 Twirp

在 gRPC 的完整功能集和 REST 最低公分母简洁性之间，存在一类框架，它们能获得 gRPC 的大部分优势——Protobuf 模式、类型化错误、代码生成—— 无需 gRPC 仅 HTTP/2、需要 trailers 的线上格式所带来的运营烦恼。两个值得关注的成员是 Connect RPC （Buf，2022）和 Twirp （Twitch，2018）。

Twirp：Twitch 更简单的协议

Twirp 是 Twitch 针对内部服务对 gRPC 的回应。设计选择：

  • 协议：HTTP/1.1 或 HTTP/2。
  • 传输格式：Protobuf（二进制）或 JSON，通过 Content-Type 头部选择。
  • 无 trailers：状态码通过 HTTP 状态码 + 结构化 JSON 错误体返回。
  • 无流式传输：仅支持 unary RPC。
  • 线上协议中无截止时间：依赖 HTTP 层超时。

URL 约定： POST /twirp/<package>.<Service>/<Method>。服务端和客户端均由以下来源进行代码生成 .proto.

与 gRPC 相比你放弃的内容：- 流式传输。- HTTP/2 多路复用（使用 HTTP/1.1 时）。- grpc-timeout 传播。

你获得的内容：- 可在 HTTP/1.1 上运行——每个代理、每个 CDN、每个浏览器均支持。- 可使用 curl 在 JSON 模式下调试。- 无 HTTP/2 协商难题。

Connect RPC：兼容 gRPC 但无需 trailers

Connect（来自 Buf，2022 年）是一个较新的方案，旨在解决 gRPC 的浏览器兼容性问题，同时保持与 gRPC 的互操作性。设计选择：

  • 协议：HTTP/1.1 或 HTTP/2。
  • 传输格式：Protobuf 或 JSON。
  • gRPC 线协议兼容：Connect 服务器可为需要的调用方提供原生 gRPC 服务。
  • Connect 专用协议 适用于非 gRPC 调用方：状态码映射到 HTTP 状态码 + 头部，无需 trailers。
  • 流式：服务端流式传输受支持（通过 HTTP/2，使用与 gRPC 相同的长度前缀分帧）；双向流需要 HTTP/2。
  • Deadline 传播：工作方式与 gRPC 类似。

Connect 的杀手级特性： 在浏览器中运行的 Connect 代码生成客户端 可以直接调用 Connect 服务器，无需独立的 gRPC-Web 代理。该协议与浏览器 fetch API 兼容。

适用场景

适合使用 Connect 或 Twirp 的情况：

  • 没有 gRPC-Web 的浏览器：原生 gRPC 需要 HTTP/2 trailers，而浏览器 fetch 不暴露该能力。gRPC-Web 可行但需要独立代理（Envoy 或 Buf Proxy）。Connect 无需代理。
  • 更简单的调试：JSON 模式让你可以 curl 在开发阶段调试请求。在生产环境切换为 Protobuf 以提升效率。
  • 未使用流式传输：如果你的服务 95% 都是 unary 调用，协议简化带来的收益超过你未使用的流式特性。
  • 异构代理链路：企业网络中中间代理对 HTTP/2 有异常处理时，可倾向于使用 HTTP/1.1 + 更简单的语义。

适合使用原生 gRPC 的情况：

  • 流式传输密集：双向语音/聊天、日志流等。
  • 已在使用 Istio/Envoy：trailers 问题由 sidecar 解决，使用 Connect 没有额外收益。
  • 最高吞吐量的东西向流量：gRPC 的 HTTP/2 多路复用已经成熟并经过性能优化。

Twirp 在 2026 年处于维护模式；Connect 是更活跃的项目。两者都是有效选择；对于新代码，Connect 是更稳妥的选择。

§20. HTTP/3 与 QUIC

HTTP/3——基于 QUIC 的 HTTP——是下一代传输协议。Staff 级别的深度信号在于能够准确说出 TCP 层队头阻塞 是具体的动机，以及 连接迁移 是其针对移动场景的杀手级特性。

QUIC：基于 UDP 的传输协议

QUIC （Quick UDP Internet Connections）是由 Google 开发的传输层协议（2012 年，IETF 于 2021 年标准化）。架构翻转：不再是 TCP + TLS + HTTP/2 的三层叠加，QUIC 将传输、加密和多路复用合并为基于 UDP 的单一协议。

特性：- 基于 UDP：QUIC 是基于 UDP 的应用层协议，不涉及内核 TCP 栈。- 内置加密：TLS 1.3 是强制且内置的，不是额外层，而是整体的组成部分。- 传输层流多路复用：每个 QUIC 流拥有独立的排序和流量控制，与其他流互相独立。- 0-RTT 握手：重连时，客户端可以在第一个数据包中携带应用数据（使用之前会话的密钥）。1-RTT 也比 TLS-over-TCP 更快，因为加密握手与传输握手合并进行。- 连接迁移：连接以 64 位 Connection ID 标识，而非 (src_ip, src_port, dst_ip, dst_port) 四元组。若客户端 IP 变化（移动设备从 Wi-Fi 切换到 LTE），连接依然存活。

传输层无队头阻塞

从 §13 问题 5 重申的核心架构特性：

  • HTTP/2 over TCP：TCP 保证字节的有序交付。若某个数据包丢失，内核会扣留 所有 后续字节，直到丢失的数据包被重传。即使这些后续字节属于不同的 HTTP/2 流，它们也会在内核缓冲区中被阻塞等待。 HTTP/2 多路复用在数据包丢失时会被 TCP 层队头阻塞所破坏。

  • HTTP/3 over QUIC：每个 QUIC 流独立排序。流 7 的数据包丢失只阻塞流 7，流 8、9、10 可立即交付。HTTP/2 承诺的多路复用收益 真正地 在端到端层面实现了。

在干净的数据中心内网（丢包率约 0.001%）中，这几乎无关紧要。而在有损的移动网络（切换小区时丢包率 1–5%）中，p99 延迟差异可能非常显著。

连接迁移：移动场景的杀手级特性

一位移动用户正在地铁上。手机从一个基站断开，连接到下一个基站，IP 地址发生变化。使用 TCP 时，以四元组标识的连接会断开，所有进行中的请求均失败，客户端必须重连并重试，用户会感受到短暂卡顿。

使用 QUIC 时，Connection ID 保持不变。客户端的新 IP 出现在同一个 Connection ID 下，服务器将其识别为已有连接并继续处理。 传输中的流在 IP 变更后依然存活，对应用层无可见的失败。

Cloudflare 发布的数据显示，0-RTT QUIC 对移动端页面加载时间的影响：中位数变化约 0%，p99 改善约 30%。尾部延迟才是收益所在。

生产采用现状

2026 年：

  • Cloudflare：HTTP/3 是其边缘网络的默认协议。约 50% 支持 HTTP/3 的浏览器在访问 Cloudflare 托管站点时使用 HTTP/3。
  • Google：默认通过 HTTP/3 提供搜索、YouTube 及其大多数服务。
  • Meta（Facebook、Instagram）：移动端 App 使用 HTTP/3；Web 端情况不一。
  • Apple：iOS 在服务器支持的情况下使用 HTTP/3。
  • AWS、Azure、GCP：在其 CDN/边缘产品处终止 HTTP/3。内部服务大部分仍使用 HTTP/2。

对于 内部东西向流量，HTTP/2 仍是主流选择。原因：- 数据中心内网环境干净（无丢包），QUIC 的尾部延迟优势无法体现。- 基于 UDP 的传输与某些负载均衡器和防火墙存在兼容性问题。- QUIC 实现比成熟的 TCP 更耗 CPU（硬件卸载支持较少）。- HTTP/2 成熟的工具链和可观测性尚未被 QUIC 匹敌。

对于 边缘 / 面向公网 / 移动端流量密集，HTTP/3 在 2026 年是默认选择。

Staff 级别准则： 面向客户端的最后一英里使用 HTTP/3；东西向流量使用 HTTP/2（甚至在服务网格管理连接池的场景下内部使用 HTTP/1.1）。

§21. Ambient Mesh 与 Cilium

sidecar 模式（§6）自 Istio（2018 年）和 Linkerd-2（2019 年）以来一直是服务网格数据平面的主流架构。这也是该模式的主要代价：每个 Pod 运行一个 Envoy 副本，消耗 100–300 MB RAM 和 5–15% 的 CPU。在 10,000 个 Pod 时， 仅用于 sidecar的资源就高达 1–3 TB RAM 和数百个 CPU。下一代方案是 ambient mesh （Istio）和 基于 eBPF 的网格 （Cilium）。

Istio Ambient 模式：ztunnel + waypoint

Istio 的 ambient 模式 （2022 年引入，2024 年 GA）将数据平面拆分为两层：

  • ztunnel：每节点（而非每 Pod）组件，运行在每个 Kubernetes 节点上。处理 L4 层：mTLS 终止、L4 鉴权、传输级遥测。每个节点的单个进程服务于该节点上的 所有 所有 Pod。
  • waypoint 代理：使用 可选的 每命名空间（或每 service account）组件，在需要时处理 L7 特性：HTTP 路由、重试、请求级鉴权。仅在配置了 L7 特性的命名空间部署；许多命名空间无需 waypoint。

架构节省：- 10,000 个 Pod × Envoy sidecar = 10,000 个 Envoy 实例。- 500 个节点 × ztunnel + 50 个命名空间 × waypoint = 550 个实例。- 大约 减少 20 倍的代理实例.

权衡：ztunnel 是一个更小、功能更有限的代理（仅支持 L4 + mTLS）。L7 特性需要额外经过 waypoint 代理的一跳。对于纯 L4 + mTLS，ambient 模式严格更廉价。对于复杂的 L7 路由，延迟略高于每 Pod Envoy（约 100 µs）。

迁移路径：Istio 同时支持两种模式。一个命名空间可以是"ambient"，另一个可以是"sidecar"，可逐步迁移。

Cilium Service Mesh：基于 eBPF，完全无 sidecar

Cilium 是一个 CNI（容器网络接口）插件，使用 eBPF （extended Berkeley Packet Filter）——运行在 Linux 内核中的程序——来实现网络、安全和可观测性。Cilium Service Mesh 在此基础上扩展了网格特性。

架构图：

  • 内核级数据平面：附加到内核网络栈的 eBPF 程序处理数据包、L4 负载均衡、mTLS（通过内核内置的 kTLS），流量策略和可观测性。
  • 无 sidecar 进程：不存在 Envoy 进程，应用程序直接与网络通信，由 eBPF 拦截并应用策略。
  • 按需通过内嵌 Envoy 提供 L7 特性：对于 HTTP 路由或复杂的 L7 策略，Cilium 以每节点进程方式运行 Envoy（类似于 Istio 的 ztunnel + waypoint 拆分）。

节省比 ambient Istio 更大：- 无每 Pod 代理：零每 Pod 代理 CPU/RAM 开销。- 内核内转发：避免了 sidecar 所需的用户态 ↔ 内核态上下文切换，降低延迟。

权衡：- 内核版本要求：eBPF 特性随内核版本演进，Cilium 不断推进边界，旧版内核无法运行全部特性。- 成熟度较低：与 Istio 相比，在极大规模部署 Cilium Service Mesh 的组织较少。- 不同的思维模型：内核级网络对于习惯以进程思维思考的开发者而言更难调试。

后 sidecar 时代转型的适用时机

2026 年的 Staff 级别立场：

  • 小/中规模（< 100 个服务）：sidecar 完全可行，开销是真实存在的，但运维简单性胜出。
  • 大规模（100–10000 个服务）：ambient mesh 是强力默认选择。sidecar CPU/RAM 税是预算中的一条线项；ambient 以约 20% 的资源成本提供相同的特性。
  • 超大规模（Google、Meta、Cloudflare）：无代理或 eBPF。此规模下 sidecar 资源成本每年高达数百万美元；在无代理 gRPC 库或 eBPF 数据平面上的工程投入能够回本。

历史演进弧线：从"每种语言一个库"（早期微服务）→"sidecar 代理"（Istio 时代）→"每节点代理 + 可选 L7"（ambient）→"内核驻留 eBPF"（Cilium）。每一步都以实现复杂性换取资源效率。

§22. 成本经济学

Staff 工程师应能够以美元而非仅以毫秒来讨论 RPC 架构。主要成本项：

序列化 CPU

Protobuf 解码 vs JSON 解析：- Protobuf：约 2–5 ns/byte（生成代码，无反射），1 KB ≈ 2–5 µs。- JSON（Jackson，Java）：约 30–80 ns/byte，1 KB ≈ 30–80 µs。- Protobuf 解码成本约为 JSON 的 1/10。

在 1M RPC/秒、1 KB 消息量下，CPU 差异为：- JSON：30–80 µs × 1M = 每秒墙钟时间消耗 30–80 秒的 CPU = 30–80 核。- Protobuf：2–5 µs × 1M = 2–5 核。

按 AWS 费率（按需稳定实例约 $0.04/核时），每百万 QPS 服务每年的差异：- 75 核 × 24 × 365 × $0.04 ≈ JSON 税每百万 QPS 每年约 $26,000，仅计解析开销。- 在 LinkedIn 或 Netflix 规模的集群上，这一数字累计达数百万美元/年。

编码开销类似（Protobuf 编码约 5–10 ns/byte；JSON 序列化约 50 ns/byte）。全程往返 CPU 节省：约 10 倍。

mTLS 开销

单次 TLS 1.3 握手消耗约 500 µs CPU + 1 次往返，按连接计，而非按调用计。

对于一条承载 1M RPC 后才被回收的长连接 HTTP/2 连接：- 每次 RPC 摊还握手成本：约 0.5 ns，可忽略不计。- 每次 RPC 加密（AES-NI 硬件加速）：约 1–2 ns/byte，1 KB ≈ 1–2 µs，小于框架解码成本。

仅当连接翻转率较高（短连接客户端、频繁回收）时，该成本才有影响。在规模化场景下保持连接热活，mTLS 开销即可忽略不计。

带宽与出口流量

AWS 跨可用区（同区域）数据传输：$0.01/GB。跨区域数据传输：$0.02/GB。出口至公网：$0.05–$0.09/GB，因区域而异。

在 10M RPC/秒、2 KB JSON 负载下：- 10M × 2 KB = 20 GB/s = 每方向每天 1.7 PB。- 按跨可用区费率（$0.01/GB），20 GB/s × 86400 s × $0.01 = 约 $17,000/天 = $6M/年 仅用于带宽。

切换至 Protobuf，线上数据量约为原来的 30%：- 6 GB/s = 每天 500 TB。- 成本：约 $5,000/天 = $1.8M/年。- 该服务每年节省带宽成本约 $4M.

跨区域费率有乘数效应：一个具有跨区域复制的聊天服务按 $0.02/GB 计费，在相同流量下，JSON 每年约 $12M，Protobuf 约 $3.6M。

这正是"Google 以 10 倍性能发布 Protobuf 基准测试"的现实：那些基准测试并非学术成果，规模化带来的年度节省足以支撑整支工程团队。

Sidecar 开销

每 Pod Envoy：约 100–300 MB RAM，约 5–15% CPU。

在 10,000 个 Pod 时：- 累计 RAM：1–3 TB。- 累计 CPU：500–1500 核。

按 AWS EC2 m5.xlarge 费率（4 vCPU + 16 GB RAM，$0.192/小时）：- 1000 核 × 24 × 365 × ($0.192/4) = 每年 $420,000 仅用于 sidecar CPU。- 加上 RAM，加上每 Pod 运行额外进程的开销（运维复杂性）。

这正是 ambient mesh 和基于 eBPF 的网格（§21）存在的原因：在规模化场景下将 sidecar 数量减少 20 倍，每年可节省数百万美元的成本。

连接数

每个 TCP 连接占用内核缓冲区（约 64 KB）和一个文件描述符。在 1M 并发连接时，需要约 64 GB 内核内存，并带来文件描述符压力。

HTTP/2 多路复用（每连接 100–1000 个流）相比 HTTP/1.1 可将连接数减少约 100 倍，节省效果覆盖整个基础设施。

公网客户端出口流量

对于公共 API，出口至公网是主要成本项。JSON 在此场景不可避免（浏览器、第三方）。缓解措施是 压缩 （gzip、Brotli）——JSON 压缩后体积降至原始大小的 20–30%，与 Protobuf 的线上体积相当。

一个提供 100 GB/s 压缩 JSON 的公共 REST API，按 AWS 费率每天出口成本约 $30,000。框架选择（REST vs gRPC）的影响远小于是否启用压缩。

核心数字汇总

Staff 工程师需记住的数字：

成本项 数量级
Protobuf vs JSON CPU Protobuf 约便宜 10 倍
Protobuf vs JSON 带宽 Protobuf 约小 3 倍
跨可用区带宽 $0.01/GB（持续 1 GB/s 约 $10K/天）
跨区域带宽 $0.02/GB
每个 Pod 的 Sidecar 开销 ~10% CPU，~200 MB RAM
每个长连接的 TLS 开销 可忽略不计（~0%）

在讨论"为何在大规模集群中选择 gRPC 作为东西向通信协议"时，答案很少是"因为它快"，而是"因为它帮我们节省了数百万美元的 CPU、带宽和基础设施成本。"

§23. 为什么不对所有场景都用 REST + JSON

对"如何实现服务间通信"这个问题，最朴素的回答是"直接用 REST + JSON 就好了，每种语言都有 HTTP 库，每个开发者都懂 curl。"让我们深入分析为什么这种方式在大规模场景下会失效。

1. 传输大小。 一个典型的内部 RPC 载荷，完整详细的 JSON 格式，是等效 Protobuf 大小的 3–10 倍。以 1000 万 QPS × 2 KB JSON 与 600 字节 Protobuf 相比，带宽差异约为 ~14 GB/s 对 ~6 GB/s 集群内每个方向的带宽消耗。跨可用区带宽收费为 $0.01/GB，仅 JSON 的"格式税"每年就是约 400 万美元的成本。在 Google 规模（100 亿 RPC/秒）下，差距还要大几个数量级。

2. 序列化 CPU 开销。 Java（Jackson）解析 JSON 约需 30–80 ns/字节，解析 Protobuf 约需 2–5 ns/字节。在集群规模下，这意味着 每百万 QPS 仅用于解析就需要 50 个核心对比 5 个核心 以 LinkedIn 的规模来看，每年可节省数百台主机年的 CPU 开销，这是实实在在的经济价值。

3. 不支持流式传输。 REST 有分块传输编码、SSE（Server-Sent Events）和 WebSockets，但这些都不是流式 RPC 的抽象。你需要自己实现帧分割、背压控制、keep-alive 和重连逻辑。而 gRPC 的服务端流式传输只需 returns (stream Post) 在 .proto 中定义，并在 stub 上调用一个方法。而用 REST 则需要每个服务写数百行帧处理和取消逻辑。

4. 不支持 deadline 传播。 HTTP 有 Connection: keep-alive 以及负载均衡器强制执行的任何超时设置。你可以在应用代码里加一个 X-Deadline 请求头，但每个服务都必须约定俗成并正确传播——其中有一半服务不会这么做，你会在下次故障复盘时才发现这个问题。gRPC 的 grpc-timeout deadline 机制内置于协议层，并由库强制执行。

5. HTTP/1.1 不支持多路复用。 每个连接只能有一个在途请求，高 RPS 下需要庞大的连接池。HTTP/2 解决了这个问题，但 REST over HTTP/2 仍然存在客户端库和中间代理降级或依赖 HTTP/1.1 语义的情况。gRPC 强制端到端使用 HTTP/2。

6. 没有针对每次调用的内置流量控制。 HTTP/2 提供了按流的流量控制。使用 REST 时，通常只能获得 TCP 级别的流量控制；一个消费慢的客户端会耗尽你的网络缓冲区，且你无法判断是哪个客户端造成的。

7. 没有 Schema 演进规范。 JSON 是"随心所欲"的——客户端发送任意内容，服务端解析它认识的部分。六个月后，集群中同一个"载荷"就会出现 14 种不同的形态。Protobuf 字段编号规则加上 buf breaking 强制执行规范。

8. HTTP 状态语义有损。 HTTP 只有 200 OK、4xx 和 5xx。gRPC 有 17 个具名状态码，每个状态码对应不同的客户端恢复操作。使用 REST 时，你需要在错误响应体中传达这些信息，客户端再去解析——这又是一套不一致的约定。

反面观点——何时 REST 才是正确答案：- 对外 API（公开接口、合作伙伴集成）。客户使用 curl；读取 JSON。- Web 浏览器（没有 gRPC-Web 代理的情况下）。浏览器无法生成 HTTP/2 trailers。- 低流量、调试优先的接口（管理类 API）。- 从更糟糕的方案迁移过来（SOAP、RPC over JMS）。

规模化场景下的正确答案是 新旧两个：REST + JSON 用于南北向流量，gRPC + Protobuf 用于东西向流量。Google、LinkedIn、Netflix、Uber 和 Stripe 都是这样做的。

§24. 扩展维度

类型 1：均匀扩展（更多服务，更多调用）

随着服务数量增加，拓扑结构的演变：

变化内容 拓扑
1x（10 个服务，1 万 QPS） 服务间通过 HTTP/1.1 + JSON 直接通信，前面有负载均衡器，无服务网格。
10x（100 个服务，10 万 QPS） 在热点内部路径上采用 gRPC，引入服务注册中心（Consul），在客户端库中实现客户端侧负载均衡，各团队自定义重试策略（已开始出现差异）。
100x（1000 个服务，100 万 QPS） 服务网格成为必选项。Sidecar 代理（Envoy）负责处理 mTLS、重试和熔断。控制平面通过 xDS 下发配置，引入 Schema 注册中心，并在 CI 中进行破坏性变更检查。
1000x（1 万个服务，1 亿 QPS） 服务网格成为关键资源消耗点（Sidecar CPU 开销与应用本身相当）。转向 Ambient mesh（eBPF）或无代理模式（将 xDS 集成到厚客户端库中），使用层级化服务目录，实现区域感知路由，并采用对冲请求。

拐点: - ~50–100 个服务："我们已经无法理清重试/超时策略了。"引入服务网格。- ~1000 个服务："Sidecar CPU 占 Pod 成本的 30%。"迁移到无代理模式或 eBPF。- ~1 万个服务："Schema 注册中心已是关键基础设施，Schema 审核是 Staff 级别工程师的职责。"

类型 2：某个服务成为热点

服务总数 N 不变，但服务 X 的 QPS 从 1000 增长到 100 万。

阶段 应对策略
1x 单实例，没有问题。
10x 水平扩展，轮询负载均衡，无状态最理想。
100x 客户端连接池耗尽（每个客户端都与 X 保持 N 个连接）。 子集化（Subsetting）：每个客户端随机连接到 X 的 20 个实例子集，而不是全量实例。这正是 LinkedIn 的 D2（Dynamic Discovery）所要解决的故障场景——当双方各有 1000 个实例时，朴素的全量互联负载均衡方式会导致每台主机持有 100 万个 socket。
1000x 调用方热点 key 问题：某个用户产生了 X 服务 30% 的 QPS，需要在调用方做限速，并在 X 处做准入控制。
10000x X 必须在内部进行分片。服务网格按请求 key 路由（一致性哈希），对尾延迟采用对冲请求，对读多工作负载在前面加读穿缓存。

子集化数学计算 （具体场景）：N=1000 个客户端 × M=1000 个 X 实例。朴素全互联方式：100 万个客户端→服务器连接对。子集大小为 20 时：仅 2 万个连接对，socket 数量、文件描述符和 HTTP/2 连接数减少了两个数量级。

一致性哈希路由 （具体场景）：如果 X 在内部进行了分片，则按 key 路由（user_id → 分片）。相同用户始终访问相同的分片，从而利用本地缓存并减少跨分片 join。用于特征存储、视频元数据服务（Vitess）以及任何需要会话亲和性的场景。

对冲请求 （具体场景）：将同一个幂等读请求发送到两个副本，返回最先响应的结果。Google"Tail at Scale"论文表明，中位延迟几乎不受影响，而 p99 延迟则显著降低。代价是：被调方 QPS 翻倍——仅对读多的幂等路径启用此策略。

每个扩展阶段都存在的陷阱：人们倾向于持续向现有拓扑添加容量，而不是对架构进行重构。从 100x 到 1000x 的跃升往往需要拓扑变更（子集化、一致性哈希路由、对被调服务进行分片），而不仅仅是增加 Pod 数量。

类型 2 主要暴露 被调方 ——热点服务成为瓶颈。类型 1 主要暴露 基础设施层 ——服务网格控制平面、Schema 注册中心、可观测性。

§25. 决策矩阵

场景 推荐选择 阈值 / 原因
内部微服务，高 RPS，多语言 gRPC + Protobuf 默认东西向通信，传输/CPU 高效，支持流式，有代码生成，支持 deadline 传播。
面向外部的公开 API（第三方、合作伙伴） REST + JSON 客户使用 curl；读取 JSON，可缓存。

异步扇出、持久化 不应使用 RPC——请使用队列 Kafka、RabbitMQ、Pulsar。生产者与消费者在时间上解耦。

决策阈值: - >100 万 QPS 内部流量：必须采用 Schema 规范（Protobuf + CI 破坏性变更检查），这种规模下 REST + JSON 是 CPU 和带宽的巨大浪费。- >50 个服务：必须采用服务网格，各团队的重试/超时策略已经开始出现差异，统一性必须在基础设施层面强制执行。- >10 跳扇出深度：必须强制执行 deadline 传播，逐跳超时会累积叠加，导致面向用户的时延预算超标。- 浏览器调用方：切勿使用原生 gRPC，应选择 gRPC-Web（需代理）、Connect（无需代理）、GraphQL 或 REST。

可作为 Staff 工程师技术判断的原则： gRPC 用于东西向；REST 用于南北向；当客户端字段需求各异时在 BFF 层使用 GraphQL；持久化事件流使用 Avro；凡是应该用队列或工作流的场景，切勿使用 RPC。 其他情况需要明确的理由说明。

§26. 典型案例展示

相同技术，不同场景，不同侧重点：

微服务网格——Netflix 风格的消费者端扇出。 一个设备会话的首页渲染会发出 100+ 次内部 RPC 调用。采用 Hystrix（后迁至 resilience4j）熔断器、Eureka 服务发现、Ribbon 客户端侧负载均衡。这是典型的"每个叶子节点的故障都必须被容忍"的工作负载。技术变体：采用同步 gRPC 扇出，配合激进的熔断、降级响应、对冲请求和预渲染缓存兜底。核心要求：即使 5% 的叶子服务故障，99.9% 的用户请求也必须成功。

微服务网格——Lyft 风格的运维基础设施。 Lyft 之所以构建 Envoy，正是因为他们现有的网格（Python 服务 + 各种客户端）无法统一执行重试/熔断/可观测性策略。网格不仅是 RPC 的基础设施，更是整个运维体系的支撑：流量切换、金丝雀发布、基于请求头的蓝绿部署。

微服务网格——Stripe 风格的事务性扇出。 支付授权会扇出到风控、账户余额、账本、反欺诈和外汇服务。每次调用必须成功或可安全重试，所有公开 POST 接口都使用幂等键，内部服务使用 Twirp 或 gRPC。核心要求：每次重试必须安全，不能重复扣款，deadline 传播可防止长尾延迟超出面向用户的时延预算。

移动端 API——GraphQL BFF。 Shopify 的 Storefront API 和 Facebook 的移动数据层都使用 GraphQL，因为移动端客户端所需的字段组合差异极大，过度拉取会消耗宝贵的蜂窝流量。移动端 App 发送一个精确指定所需字段的查询，BFF 层通过调用 5–20 个内部 gRPC 服务来组装响应。技术变体：边缘层使用 GraphQL，内部使用 gRPC。

公开 API——Stripe / Twilio / GitHub REST。 Stripe、Twilio 和 GitHub 的经典 REST API 都采用 REST + JSON，因为调用方可能是某个用 Python + requests 或一个 Ruby 应用或一个 Slack bot。核心要求：零工具门槛的可调试性、可缓存的 GET 响应、可预期的分页语义，以及对 CLI（curl）的友好支持。

公开 API——GitHub GraphQL。 GitHub 同样提供了 GraphQL 接口，因为集成商（CI 工具、代码分析器、仪表盘）只需要特定字段，而 REST API 的响应体存在过度拉取问题。同样的数据，两种对外接口。

内部高吞吐量——LinkedIn Rest.li。 约 1000 万 QPS，约 10 万个服务实例。Rest.li 历史上是基于 HTTP/1.1 的 REST 风格协议——Schema 使用 .pdl 文件定义（目的与 Protobuf 类似），框架会生成 Java stub。D2 负责客户端侧负载均衡。Espresso（宽列存储）通过 Rest.li 承载约 300 万 QPS。技术变体：具备 Protobuf 等效 Schema 规范的 REST 形态，加上自研客户端服务发现与负载均衡层。核心要求：1000 万 QPS、10 万个实例，以及一个早于 gRPC 成熟的 Java 生态系统。

内部高吞吐量——Google Stubby / gRPC。 约 100 亿 RPC/秒，超过 10000 个服务，一次搜索查询会扇出到约 1000 个后端，每次调用的 deadline 在 10–50 ms 之间。采用无代理服务网格（gRPC 库直接消费 xDS，无 Sidecar），因为在 100 亿 RPC/秒的规模下，Sidecar 的 CPU 开销将是另一条数十亿美元的成本线。核心要求：每次调用的框架开销最小化，严格的 Schema 规范，扇出深度 10+ 且支持 deadline 自动截断。

ML 推理扇出——Uber Michelangelo。 一次行程定价请求会并行调用 5–10 个 ML 模型（需求预测、反欺诈、预计到达时间、动态溢价）。每次推理都是对 Triton 或 TorchServe 端点的 gRPC 调用，p99 时延预算为数十毫秒。核心要求：高并发扇出并对最慢的调用使用对冲请求；通过熔断器剔除"慢"模型实例，防止一块坏的 GPU 污染整个预测链路。

数据库网络协议——Postgres、MySQL。 虽然不是 gRPC，但在概念上也是一种 RPC：客户端发送类型化的请求（SQL 语句、预处理语句、绑定参数），服务端返回类型化的响应（行数据、状态）。Postgres 网络协议定义了明确的消息类型（Q = 简单查询， P = 解析， B = 绑定， E = 执行， S = 同步），使用长度前缀和定义清晰的错误语义。这种基本原语（基于 TCP 的类型化请求-响应）随处可见，gRPC 只是其中一种经过良好打磨的实现。

流式传输——gRPC 双向流用于聊天和实时更新。 双向 gRPC 流的应用场景包括：Slack 的 WebSocket 替代方案（在可以避免浏览器限制的场景下）、Discord 的语音信令、Zoom 的通话协调，以及金融市场行情数据推送。核心要求：长时间存活的流（分钟到小时级别），流内有序交付，通过 HTTP/2 实现每流级别的流量控制。 WINDOW_UPDATE.

§27. 真实世界的实现数据

  • Google 内部：Stubby（现已大部分迁移到 gRPC）每秒约 100 亿次 RPC。基于 Borg 和服务发现，deadline 传播内置于协议层。一次搜索查询扇出到约 1000 个后端，每次调用的 deadline 为 10–50 ms。无代理服务网格——gRPC 库直接消费 xDS，无 Sidecar 进程。

  • LinkedIn Rest.li：约 10 万个服务实例，QPS 约 1000 万。Espresso（存储层）通过 Rest.li 承载约 300 万 QPS。D2（Dynamic Discovery）负责客户端侧负载均衡。Rest.li 历史上是基于 HTTP/1.1 的 REST 风格协议，正在将内部热点路径迁移到 gRPC。中间层服务（Feed 中间层、Profile 中间层）单独可承载 50 万+ QPS。

  • Netflix：Hystrix 和最初的熔断器模式，源于 2012 年的一次重试风暴事故——当时一个后端服务触发 GC，导致个性化推荐层大面积故障。服务发现采用 Eureka，每次设备会话有 100+ 次调用扇出。后迁移至 resilience4j（Hystrix 已进入维护模式），新服务采用 gRPC。

  • Uber：Apache Thrift，TChannel 传输层，后迁至 YARPC（其 RPC 抽象层）。内部约 1000 万 QPS。一次叫车请求：约 30 次同步调用 + 约 40 次异步调用，p99 时延预算 300 ms。

  • Stripe：幂等键在其公开 API 中是一等公民。 Idempotency-Key 请求头是"即使对写操作也能安全重试"这一理念在现实中最典型的体现。内部服务混合使用 Ruby、Go，以及越来越多地在热点路径上采用 gRPC + Twirp。

  • Envoy / Istio：已在 LinkedIn、Google、eBay、Salesforce、Yelp 以及绝大多数财富 100 强企业部署。仅 Envoy 在大型生产环境中每秒就处理数千万次请求，是最主流的数据平面代理。

  • Cloudflare：在边缘层使用 HTTP/3 上的 gRPC。他们曾公开撰文介绍 HTTP/3 在移动网络丢包场景下对 p99 延迟的队头阻塞优化效果。

  • Meta（Facebook）：Thrift 无处不在。内部 RPC 总量未公开，但估计与 Google 同一量级。

  • Buf（公司）：维护 buf breaking 。 Connect RPC工具链。破坏性变更检测在 CI 中成为标配，很大程度上归功于他们在行业层面推动了这一规范。

  • Discord：服务间通信采用 gRPC，语音协调使用双向流。面向浏览器的聊天协议使用 WebSockets（gRPC-Web 在浏览器端不能很好地支持双向通信）。

  • Square / Block：率先推出了 Wire（一种更轻量的类 gRPC 移动端框架）并为 Connect 做出了贡献。内部服务混合使用 Wire、gRPC 和标准 Protobuf over HTTP。

值得铭记的数字

  • 一位 Google SRE 的分享：一次 Protobuf 字段编号复用（某人用不同类型覆盖了字段 5），导致一项核心服务在整个集群范围内出现长达 4 小时的局部故障。恢复方式：回滚 .proto，等待二进制重新部署。Schema 审核不是可选项。
  • Netflix 最初的重试风暴（约 2012 年，Hystrix 诞生前）是由一个后端服务触发 GC 引发的，导致个性化推荐层大面积故障，这次事故的事后复盘催生了 Hystrix。
  • LinkedIn 之所以采用 D2 + Rest.li，部分原因是为了解决朴素客户端侧负载均衡下"1000 个服务实例各自与其他每个服务保持 1000 个连接 = 每台主机 100 万个 socket"的故障模式，这一问题通过子集化得以解决。

深入探讨要点（如有追问可展开）

  • 对冲请求。 将相同的请求发送到两个副本，返回最先响应的结果。中位延迟略有上升，p99 延迟显著降低。代价：被调方 QPS 翻倍——仅适用于读多幂等路径。
  • HTTP/2 流量控制作为背压机制。 WINDOW_UPDATE 每个流上的帧——慢速消费者自然地对其发送方形成节流，而不影响其他流。在 gRPC 流式传输中，这才是真正的背压机制，而不是"客户端降低轮询频率"。
  • 通过 SDS 进行 mTLS 证书轮换。 Secret Discovery Service 每隔几小时向 Sidecar 推送新证书，应用程序完全感知不到证书变更。SPIFFE 身份（spiffe://example.com/sa/service-a）是现代服务身份标准。
  • W3C 链路追踪上下文。 traceparent （32字符 trace id + 16字符 span id）由客户端 sidecar 注入，并由每个转发跳传递。采样决策（traceflags）必须传播，以确保 trace 的连贯性。
  • oneof 。 Any 采用 Protobuf 格式。 oneof 是一个带标签的联合体；集合中只能设置一个字段。 Any 通过类型 URL + 字节携带任意消息——灵活，但丧失了该字段的 schema 优势。
  • 服务端反射。 grpc.reflection.v1alpha 允许 grpcurl 在运行时发现服务定义。在开发环境中很有用，在生产环境中有风险（泄露 API 接口）。在 staging 环境之后应禁用。
  • xDS 变体：LDS（监听器）、RDS（路由）、CDS（集群）、EDS（端点）、SDS（密钥）、ADS（聚合——一个流处理所有）。熟悉这些术语意味着"我运行过服务网格"。
  • 无 Proxy 网格。 gRPC 库直接使用 xDS。节省了 sidecar CPU 开销，但代价是网格中库未实现的功能无法使用。
  • Ambient mesh / Cilium。 Sidecar 被内核中的 eBPF 替代。每个 Pod 的开销更低。成熟度较低，但正在快速发展。

§28. 总结

"RPC 是有类型约束、schema 绑定、截止时间传播、错误码标准化的服务间通信基础设施，它将多层服务的扇出调用树变成工程师可以推理的系统——内部使用 gRPC、Thrift 和 Connect over HTTP/2，Protobuf 载荷，公共边界用 REST + JSON，BFF 层用 GraphQL——技术的真正价值不在于'两个服务能否通信'（任何方案都能做到），而在于它所施加的契约：带字段编号演化规则的有类型消息、跨跳相减的截止时间、规定恢复动作的状态码，以及允许单个连接承载数千个飞行中调用的 HTTP/2 多路复用——因为大规模集群的真正故障模式不是'服务宕机'，而是'服务变慢但未宕机'，而阻止雪崩的分层防御（截止时间 + 重试预算 + 熔断器 + 舱壁 + 负载卸载）正是真正的 RPC 框架所标准化的原语，让每个团队都能免费获得，而不是各自重新错误地实现。"
