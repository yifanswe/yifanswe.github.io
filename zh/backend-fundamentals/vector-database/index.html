向量数据库

目录
目录

组件类别：围绕 ANN（近似最近邻）查询构建的存储与搜索系统，处理由 ML（机器学习）模型生成的高维向量嵌入。实现方式涵盖专用托管服务（Pinecone）、开源专用系统（Weaviate、Milvus、Qdrant）、Postgres 扩展（pgvector）、嵌入式库（Chroma、FAISS、Annoy），以及附加了向量能力的传统搜索引擎（Elasticsearch dense_vector、OpenSearch k-NN、Vespa、Solr）。第 §4 节中的底层机制——HNSW（层次化可导航小世界）图、IVF（倒排文件）聚类、PQ（乘积量化）压缩——在所有这些实现中均适用，因为大家都在复用 2010 年代那几个经典的学术算法。

§1. 向量数据库是什么

一个 向量数据库 存储固定维度的浮点向量——通常每个向量 256 到 3072 维——并回答如下形式的查询："在语料库中，给我在距离度量 D 下最接近该查询向量的 K 个向量。"距离度量几乎总是余弦相似度、点积或欧氏距离（L2）。相似度是在由上游嵌入模型生成的 连续且具有语义意义的空间 中计算的：一个关于"如何续签护照"的句子会落在"护照续签流程"附近，因为嵌入模型将它们在向量空间中放置得很近。

核心结构是 ANN（近似最近邻）索引，通常为以下之一：HNSW（层次化可导航小世界）、带 PQ（乘积量化）的 IVF（倒排文件），或某些工业变体如 ScaNN（可扩展最近邻）。"近似"二字至关重要。对十亿个 1536 维向量进行精确最近邻搜索，等同于对 6 TB 数据做线性扫描——每次查询耗时数分钟。ANN 以 1–5% 的召回率损失换取毫秒级响应。

向量数据库在大多数架构中充当 派生存储，从不作为数据源头（source of truth）。嵌入本身是从主系统持有的原始文本或图像中派生出来的；如果嵌入模型发生变化，整个语料库必须重新嵌入并重建索引。原始文档 ID 和元数据存储在关系型或文档数据库中；向量数据库持有以这些 ID 为键的索引，加上高维向量以及用于过滤的轻量级元数据附属信息。

与相邻类别的区别

  • 传统数据库（关系型、文档型、宽列型） 回答"给我列 X 等于值 Y 的行"—— 基于离散谓词的精确匹配。它们无法回答"给我含义与此查询相似的行"，因为它们没有语义概念，只有字节。B+ tree 在 LIKE '%word%' 谓词上退化为全表扫描，即便是 trigram 索引也只处理子字符串模糊匹配，而非语义模糊匹配。
  • 带有文本倒排索引的搜索引擎（Elasticsearch、OpenSearch、Solr） 回答"给我包含这些词项并按 BM25（Best Match 25）排序的文档"。它们匹配 离散词元（token），而非连续语义。除非分析器或同义词词典明确做了词元映射，否则它们不会将"passport renewal"与"how do I update my travel documents"的查询相匹配。基于 Lucene 的现代引擎已添加了 dense_vector 字段和 HNSW 索引，使其成为混合方案——见 §3。
  • 键值存储（Redis、DynamoDB） 回答"获取该精确键对应的值"——O(1)，无需扫描，无需评分。它们对"查找与此相似的内容"毫无用处，因为键与其他键之间不存在语义关系。Redis 已添加 RediSearch 向量模块，但访问模式仍是独立索引。
  • 图数据库（Neo4j、Neptune） 对实体之间的显式边进行建模；向量数据库对 隐式相似性 进行建模，表现为连续空间中的临近关系。图数据库存储"用户 U 关注用户 V"；向量数据库存储"用户 U 的嵌入偏好向量与用户 V 的偏好向量余弦相似度为 0.91"。
  • OLAP（在线分析处理）引擎（ClickHouse、Druid、Pinot） 扫描大型列式表以进行聚合计算；它们不将最近邻查询作为一等操作支持。

向量数据库不擅长什么

  • 按 ID 精确查找 ——比键值存储更慢；当你只需要哈希查找时，却要承担 HNSW 遍历的开销。应始终单独保存原始 ID，并从主系统获取原始文档。
  • 数据真实来源 ——嵌入是派生的；向量是对文本/图像/代码的解释层。从原始数据重建始终是恢复路径。
  • 高吞吐量点更新 ——索引构建代价高昂；HNSW 图在频繁变更下会退化；删除操作是墓碑标记，而非真正的数据擦除。
  • 返回精确的 top-K 结果 ——召回率通常为 95–99%。如果产品要求 100%，ANN 并不适用；应在小规模下使用精确 KNN 或结构化查询。
  • 跨多个向量的事务语义 ——不支持 ACID，不支持多向量原子性。
  • 当在单 GPU 上用 Python 进行线性扫描比网络往返服务更快的小语料库场景 ——向量数量低于约 10k 时，进程内的 sklearn 或 FAISS 在延迟上优于任何网络化向量数据库。
  • 字节层面的存储效率 ——单个 OpenAI text-embedding-ada-002 向量为 1536 个浮点数 × 4 字节 = 6,144 字节 每份文档，不含索引开销。1 亿份文档在不含 HNSW 边列表的情况下，原始向量就已达 600 GB。

§2. 固有保证——设计层面的契约

已提供：

  • 亚秒级 ANN（大规模）。 针对 1 亿个 768 维向量经过良好调优的 HNSW 索引，在向量驻于内存的单节点上，top-10 查询可在 5–50 毫秒内返回。Pinecone、Weaviate、Milvus 的 p99 延迟均在此范围内。
  • 可配置的召回率/延迟权衡。 HNSW 暴露了 ef_search （查询期间候选集的宽度）；IVF 暴露了 nprobe （扫描的聚类数）。值越大 = 召回率越高，速度越慢。该权衡是运维人员按使用场景调节的旋钮。
  • 元数据过滤。 几乎所有现代向量数据库都支持为每个向量附加 JSON 对象或属性集，并在 ANN 查询中通过谓词（tenant_id = X AND category = "support"）进行过滤。
  • 通过分片实现水平可扩展性。 向量按 ID 哈希或元数据进行分区；每个分片运行自己的 ANN 索引；协调器从各分片合并 top-K 结果。
  • 原始向量的持久性。 大多数实现将向量写入 WAL（预写日志）或 Parquet/RocksDB 后端存储，以便在崩溃后重建内存索引。

未提供——需额外叠加：

  • 精确结果。 ANN 以 1–5% 的召回率换取数量级的延迟提升。如果产品需要 100% 召回率（如法律文件发现），则需要暴力扫描或在其上构建结构化检索层。
  • 高写入吞吐量。 HNSW 构建的时间复杂度为 O(N log N) 次边插入；在单节点上从头重建 1 亿向量的索引需要数小时。稳态写入是可接受的（每个分片每秒数千次），但批量加载代价高昂。
  • 紧凑存储。 向量本身体积较大；PQ（乘积量化）及其他压缩手段有所帮助，但会损失召回率。没有免费午餐。
  • 幂等写入。 除非应用在 ID 层面强制唯一性，否则两次插入同一向量会创建两条索引记录。
  • 读后即写一致性。 大多数向量数据库存在异步提交阶段；新插入的向量可能在数十毫秒到数秒内不会出现在查询结果中。
  • 嵌入模型版本管理。 当上游模型发生变化时，所有现有向量都处于 旧 空间，与新查询向量不可比较。向量数据库对此一无所知；应用必须追踪模型版本并重新嵌入。
  • 权限过滤。 引擎返回匹配的向量；租户级或用户级权限控制由调用方负责。元数据过滤是可用的手段，但必须由调用方接入。

心智模型：向量数据库是一个 具备可选元数据过滤能力的水平可扩展、持久化、近实时 ANN 索引。该契约之外的一切——包括生成向量的嵌入模型——都是你自己的问题。

如何解读：实际上存在五个真实类别，选择是结构性的，而非审美性的。

  1. 专用托管服务（Pinecone） ——你只需要一个向量索引端点，不想自己运维任何东西，愿意为此付费。运维简单，但存在供应商锁定。这是 2023-2024 年 AI 初创公司的主流默认选择。
  2. 开源专用方案（Weaviate、Milvus、Qdrant） ——你需要一个真正的向量数据库，但部署在本地或自托管云上。各有不同的使用风格：Weaviate 以 GraphQL 为主且模块化；Milvus 架构厚重，专为十亿级规模构建；Qdrant 是轻量级 Rust 实现，提供简洁的 REST API，是目前崛起的中端首选。
  3. Postgres 扩展（pgvector） ——你已经在运营 Postgres；向量数量少于 5000 万；相比新增一个系统，你更愿意添加一个扩展。pgvector 现已成为主流入门方案；它在 pgvector 0.5.0（2023 年中）中获得了 HNSW 支持，目前对中等规模工作负载已具备可行性。
  4. 嵌入式库（FAISS、Chroma、Annoy） ——向量搜索在进程内完成。用于研究、原型开发，以及其他系统内部（Spotify 的推荐系统在进程内使用 Annoy；许多类 Pinecone 的引擎在底层封装了 FAISS）。
  5. 搜索引擎 + 向量（Elasticsearch、OpenSearch、Vespa、Solr） ——你已有基于 Lucene 的搜索系统，将向量检索作为另一种字段类型加入。其优势在于 原生混合检索 ——关键词与向量在同一查询中处理。代价在于 Lucene 的段模型针对倒排索引优化，并非面向密集向量搜索；HNSW 分段在 Lucene 仅追加写入的段世界中概念上显得格格不入。

当前默认选择（2020 年代中期）： 向量数量少于 1000 万、运维约束低：pgvector，搞定。 向量数量达 1 亿、需要托管服务：Pinecone 或 Weaviate Cloud。 向量数量 1 亿至 100 亿、自托管、有运维能力：Milvus 或 Qdrant。 已有 ES/OpenSearch，需要关键词与向量混合检索：Elasticsearch dense_vector 或 OpenSearch k-NN。 需要在循环中集成 ML 模型的一流排序能力：Vespa。

§4. 底层数据结构——字节级 ANN 算法

这是浅层文档常常遗漏的章节。生产级向量数据库实现了三个 ANN 算法家族；几乎所有系统都是 HNSW、IVF 和 PQ 中的一种或组合。从字节层面理解它们，是「我知道向量数据库存在」与「我知道延迟飙升时该拧哪个旋钮」之间的本质差距。

4.1 朴素基线——暴力 KNN（k 近邻）

作为参考基准：对每个查询向量 q，计算它与语料库中每个向量 v 的距离，然后排序并返回 top-K。每次查询的代价为 O(N·d)，其中 N 是语料库大小，d 是维度数。当 N=10⁹、d=1536 时，每次查询需要 1.5 × 10¹² 次浮点运算 每次查询。以 100 GFLOPS 计，耗时 15 秒。完全不可用。

即便借助 SIMD（单指令多数据）和 GPU，暴力搜索在 N ~ 10⁶ 时尚可接受（每次查询 1-100 ms），在 N=10⁸ 时已十分痛苦，在 N=10⁹+ 时完全无法使用。因此引入了 ANN。

4.2 HNSW——分层可导航小世界

HNSW 是 2020 年代占主导地位的 ANN 算法。由 Malkov 和 Yashunin 于 2016 年发表，是 Pinecone、Weaviate、Qdrant、Milvus（与 IVF-PQ 并用）、pgvector、FAISS、Elasticsearch dense_vector、OpenSearch k-NN 以及 Vespa 中的首选索引。

概念。 HNSW 构建一个多层图，每个向量是一个节点。底层（第 0 层）包含所有向量。每个更高层包含一个按几何级数递减的子集（通常每个向量有概率 1/M 被提升到下一层）。顶层只有少数节点，通过稀疏的远程边连接。

搜索从顶层的入口点开始，贪婪地朝查询向量方向移动，直到该层不再有更近的邻居，然后在当前最优节点处下降到下一层，并重复此过程。在底层，搜索返回最近的 top-K 个向量。

「小世界」洞见来自 Watts 和 Strogatz 1998 年关于小世界图的论文——在这种网络中，大多数边是局部的，但少数远程边能大幅缩短任意两节点之间的路径。在层次结构顶部添加几条长边，就能以 O(log N) 次跳跃从任意起点到达任意目标。

Layer 2:     A ────────────── F
              \              /
Layer 1:      A ─ C ──── E ─ F ─ H
                 / \    / \   \
Layer 0: A B C D E F G H I J K L M N O P  ← all vectors live here
         (dense local neighborhood graph)

字节级布局 （这是磁盘和 RAM 中实际存储的内容）：

Per vector:
  vector_data:     d × float32           (e.g., 1536 × 4 = 6144 bytes for OpenAI ada-002)
  metadata:        variable (JSON / fixed schema, e.g., 50-500 bytes)

Per layer per vector:
  edge_list:       M × uint32 neighbor IDs
                   (M typically 16-64; pgvector default M=16)
                   = 64-256 bytes per layer

一个 d=1536、M=16 的 1 亿向量 HNSW 索引，将向量和边都放在 RAM 中，其开销为： - 向量：100M × 6144 B = 614 GB - 底层边：100M × 16 × 4 B = 6.4 GB - 更高层边（按几何级数递减）：~6.4 GB - 合计： ~627 GB

这就是为什么十亿级 HNSW 需要分片多节点部署，也是为什么 PQ 压缩（后续章节）至关重要。

构建参数： - M ——每层每个节点的边数。M 越大，召回率越高，内存占用越多，构建越慢。典型值：M=16（精简），M=32（均衡），M=64（高召回率，索引较大）。 ef_construction ——构建器在插入新节点时搜索好邻居的力度。值越高，图质量越好，构建越慢。典型值 100-500。 ef_search ——查询时图遍历期间候选集的宽度。值越高，召回率越高，查询越慢。典型值 50-500。 这是运行时调优旋钮。

逐步演示一次 HNSW 搜索 端到端过程：

配置：语料库 1000 万向量，d=768，M=32，ef_search=100，查询向量 q。

  1. 入口：从顶层的全局入口点（单个节点）开始。计算 q 与入口点的距离。假设入口点在第 2 层有 4 个邻居；计算到每个邻居的距离。
  2. 顶层贪婪步进：移动到最近的邻居。重新计算该节点邻居的距离。重复此过程，直到没有任何邻居比当前节点更接近 q。
  3. 下降 至第 1 层的当前最优节点。在第 1 层重复贪婪遍历——图更密集，每个节点边更多，导航粒度更细。
  4. 下降 至第 0 层。这是包含全部 1000 万向量的底层，具有密集的局部连通性。
  5. 在第 0 层，算法不再采用严格贪婪策略，而是维护一个大小为 ef_search=100 ——一个存储当前最优节点的最小堆，以及一个待访问节点的队列。每次迭代：弹出距离最近的未访问节点；探索其 M=32 个邻居；若能改善堆的结果则将每个邻居插入候选集。持续迭代，直到队列耗尽或堆中所有候选节点都比最差的待处理节点更近。
  6. 返回 候选堆中 top-K（例如 K=10）个节点。

总计算量：每个被访问的节点需要 M 次距离计算，每次计算需要 d=768 次浮点运算。以 ef_search=100 为例，搜索大约访问 1000-5000 个节点——约为 3000 × 32 × 768 ≈ 7400 万次浮点运算。在支持 AVX-512（高级向量扩展 512 位）的现代 CPU 上，这是亚毫秒级的。

召回率通常为 95-99%（即真正 top-10 中有 9.5-9.9 个被返回）。将 ef_search 设为 500 时，召回率可超过 99%，但延迟上升约 4 倍。这就是那个旋钮。

与替代方案的权衡：HNSW 查询速度快，但 构建代价高 （每次插入都要进行完整搜索来寻找邻居），而且 删除表现差 （删除节点会破坏边；图会持续退化直至重建）。它还要求整个图都在 RAM 中——基于磁盘的 HNSW 存在（DiskANN），但速度明显更慢。

4.3 IVF——倒排文件索引

IVF 比 HNSW 早出现了十年，至今仍无处不在——它是 FAISS 默认索引的基础，并通过 ScaNN 在 YouTube 的候选生成中大量使用。

概念：使用 k-means 将所有向量聚类为 N 个 质心 （通常 N = √语料库大小；对于 1 亿向量，N ≈ 10000 个簇）。每个向量被分配到最近的质心。查询时，找到最近的 n_probe 个质心，然后对这些簇中的向量进行暴力扫描。 n_probe 是运行时调优旋钮——值越高，召回率越高，速度越慢。

                    ┌── centroid c1 ── [v1, v7, v23, v89, ...]
query vector  q ──> ├── centroid c2 ── [v4, v12, ...]         ← n_probe = 3
                    ├── centroid c3 ── [v5, v8, ...]            scan these
                    └── centroid c4 ── [v2, v15, ...]
                    ...
                    └── centroid c10000 ── [...]

字节级布局：

Centroids:        N_clusters × d × float32  (10000 × 1536 × 4 = 60 MB)
Cluster lists:    per cluster, a packed list of vector IDs and (optionally) the full vectors
Inverted file:    cluster_id → list of vector_ids

IVF 查询演示：语料库 1 亿向量，N=10000 个簇，n_probe=16。

  1. 计算 q 到全部 10000 个质心的距离：10000 × 1536 = 1500 万次浮点运算。在现代 CPU 上约 150 µs。
  2. 选取最近的 16 个质心。
  3. 对 16 个簇中的每一个，获取该簇中约 10000 个向量（100M / 10000 = 平均每簇 10000 个向量）并计算距离：16 × 10000 × 1536 = 2.46 亿次浮点运算。约 2.5 ms。
  4. 在所有已扫描簇上维护 top-K 最小堆。返回 top-K。

召回率取决于 n_probe。n_probe=1 时召回率约为 60-70%；n_probe=32 时可达 95%+。调整 n_probe 是 IVF 对应 HNSW 的 ef_search 的类比操作。

与 HNSW 的权衡：IVF 构建速度更快 （只需 k-means 和分配，而非 HNSW 的逐向量图搜索）， 内存开销更低 （因为簇列表可以存放在磁盘上），以及 更新更容易 （只需将向量重新分配到新的簇）。但查询延迟通常更高，且在相同语料库规模下，召回率/延迟的 Pareto 表现略逊于 HNSW，这就是为什么 HNSW 已成为默认选择。IVF 在极端规模（100 亿+ 向量）下胜出，因为此时 HNSW 的内存占用过大。

ScaNN （可扩展最近邻，Google 2020 年）是一种经过优化的 IVF 变体，使用各向异性向量量化——量化器经过调整，优先保留对最终排序最重要的点积幅度，而非仅仅是最近邻距离。ScaNN 支撑着 YouTube 的视频推荐候选生成，在十亿向量规模下比原始 IVFPQ 好几个百分点。

4.4 PQ——乘积量化

乘积量化解决了 内存 问题。一个 1536 维的 float32 向量占 6 KB；十亿个这样的向量共 6 TB。PQ 将每个向量压缩到通常 8-32 字节——压缩比 8 倍到 100 倍——代价是损失几个百分点的召回率。

概念：将每个 d 维向量拆分为 M 个长度为 d/M 的子向量。对每个子向量位置，通过在训练样本上运行 k-means 训练一个包含 K=256 个子向量质心的码本。每个子向量用其最近质心的索引表示——1 个字节（256 = 2⁸）。

Original vector v ∈ R^768:
  [v_1, v_2, ..., v_768]
                                ┌─ codebook 1 (256 centroids of d/M dims)
Split into M=12 chunks of 64:   ├─ codebook 2
  [v_1..v_64] [v_65..v_128] ... ├─ ...
   ↓             ↓               └─ codebook 12
   c1 (1 byte) c2 (1 byte) ...   

Compressed: 12 bytes per vector (was 768×4 = 3072 bytes)
            = 256× reduction

对于 1536 维向量，M=48 个块：每个向量压缩为 48 字节。十亿向量的索引从 6 TB 缩减为 48 GB，可以放在一台大型服务器上。

PQ 下的距离计算 使用预计算的查找表。查询时，q 的 M 个子向量分别与每个码本的 256 个质心进行比较，生成 M 张包含 256 个部分距离的表。然后，计算 q 到任意压缩向量 v 的距离时，只需进行 M 次表查找并求和——远比原始的 M×(d/M) 次浮点乘法计算代价低。

IVFPQ——主力组合：IVF 将向量划分为簇；PQ 压缩每个簇的向量。FAISS 针对大规模工作负载的默认索引就是 IVFPQ。Milvus 将 IVFPQ 作为一等索引类型对外暴露。大多数「十亿级」部署最终都落在这里，因为纯 HNSW 无法放入内存。

权衡：PQ 会损失召回率（通常比未压缩时低 2-5%）。对于召回率要求严苛的应用，可以将 PQ 用作粗过滤，再对 top 候选集使用完整未压缩向量进行重排序（即 IVFPQ 流水线中的 rerank 步骤）。

4.5 距离度量——取决于嵌入模型的选择

  • 余弦相似度 ——向量之间的夹角，与幅度无关。文本嵌入（OpenAI、Cohere、sentence-transformers）的默认选择，这类嵌入通常经过 L2 归一化，因此点积等于余弦相似度。
  • 点积 ——幅度有影响。用于非归一化嵌入，以及推荐系统等幅度编码置信度的场景。
  • 欧氏距离（L2） ——直线距离。用于图像嵌入（CLIP——对比语言-图像预训练）和一些神经检索模型。
  • 曼哈顿距离（L1） ——绝对差之和。应用场景较小众，偶尔用于稀疏模型。

选择取决于 嵌入模型的训练目标。如果度量选错了，召回率会灾难性下降——需要与模型训练时使用的度量保持一致。OpenAI text-embedding-3-* 是经过余弦归一化的；CLIP 图像嵌入使用余弦相似度；许多基于 BERT 的嵌入原生使用点积。

4.6 ANN 搜索中的过滤——难以处理的交互

生产查询通常附带元数据过滤：「查找相似的支持工单，但仅限该租户、仅限英语、仅限最近 90 天内」。按复杂程度递增，有三种策略：

  1. 后过滤 （朴素方法）：先运行 ANN 获取 top-K（比如 K=100），再过滤。问题在于：若过滤条件选择性强（例如只有 1% 的语料库通过），100 个候选中 99 个被丢弃，只剩 1 个。召回率骤降。必须超量获取 K 的倍数，有时高达 K × 1000，这会严重破坏延迟。
  2. 预过滤：先计算满足过滤条件的向量集合，再对该子集进行暴力 ANN。若过滤选择性强且子集较小（≤ 10k），效果不错。若子集较大（数百万）或过滤条件组合爆炸，则会失效。
  3. 过滤式 ANN（紧耦合）：维护辅助结构，使 ANN 遍历过程本身尊重过滤条件。Qdrant、Weaviate 和 Milvus 实现了该方案的各种变体——例如，在 HNSW 图遍历期间查询每个分段的过滤位图，只探索匹配的邻居。召回率保持高位；延迟保持可控。这是当前最先进的方案，也是真正的向量数据库与「在现有存储上加个向量索引」的核心区别。

Elasticsearch 的 dense_vector 结合 HNSW 在这方面历来表现欠佳，因为 Lucene 的段模型是不可变的，每段过滤位图处理起来较为复杂。OpenSearch 和 ES 已推出改进（8.x 中的高效过滤），但专用向量数据库在过滤式 ANN 方面仍优于它们。

4.7 持久性与持久化

大多数向量数据库持久化三个层次： 原始向量 （存储在 RocksDB、Parquet 或本地磁盘及/或对象存储的平坦文件中——作为恢复来源）， 元数据附属存储 （LSM 或关系型），以及 索引本身 （HNSW 边、IVF 簇分配），以周期性快照加 WAL 增量更新的方式持久化。

崩溃恢复：加载最新快照，重放 WAL，恢复服务。若快照损坏，则从 原始向量重建索引 ——代价高昂（HNSW 为 O(N log N)，1 亿向量需要数小时），但始终可行。原始向量是数据库内的真实来源；上游的文本/图像存储是更深层的来源。

§5. 容量上限

向量数据库的容量范围跨越约六个数量级。来自生产部署的具体数据：

  • 小规模（≤ 100 万向量） ——单个 Postgres 实例上的 pgvector。100 万向量、1536 维、M=16 的 HNSW 索引约占 6 GB RAM。查询延迟：5-20 ms。常见于 SaaS 初创公司基于客户文档的 RAG 场景。
  • 中规模（1000 万至 1 亿向量） ——Pinecone、Weaviate、Qdrant、Elasticsearch dense_vector。单节点或小型集群。向量通常在 RAM 中（分片存储）。查询 p99：10-50 ms。用于中型市场的 RAG 产品及较小消费类应用的推荐系统。
  • 大规模（10 亿+ 向量） ——Milvus、基于 FAISS 的自定义部署、ScaNN。分布式多节点，通常使用 IVFPQ 以适应内存。Spotify 的推荐系统（进程内 Annoy）、Pinterest 视觉搜索、Notion 基于企业工作区的 RAG。查询 p99：50-200 ms。
  • 超大规模（100 亿+ 向量） ——Google（通过 ScaNN 实现的 YouTube 候选生成）、Meta（Reels、Instagram 推荐，使用基于 FAISS 演进的内部系统）、Amazon（商品推荐）等的定制系统。在此规模下，没有任何现成的向量数据库可用；部署均为定制化方案，通常涉及多层索引（粗粒度 IVFPQ 用于候选生成，随后进行精确重排序）。

一个实用的近似扩展规则：一台现代单节点服务器（32 核、256-512 GB RAM、NVMe SSD）可托管约 5000 万至 2 亿个 768 维向量的 HNSW 索引，具体取决于 M 和 ef。超过这个规模就需要分片；超过约 50 亿向量则需要定制方案。

§7. 向量数据库固有的难题

问题 7.1——召回率与延迟的权衡

朴素方案：使用 HNSW 默认值（ef_search=50，M=16）直接上线。

如何出错：在负载下，召回率为 90%。用户抱怨明显的匹配结果没有出现。工程师将 ef_search 调至 500 以修复召回率；p99 延迟从 15 ms 跳升至 70 ms；SLO（服务等级目标）被突破。

具体状态：一个支持工单检索系统有 500 万个 d=768 的向量，M=16，ef_search=100。Recall@10 = 94%。机器人大约每 16 次中有 1 次遗漏相关历史工单。将 ef_search 调至 400 可将召回率提升至 99.2%，但延迟翻倍。

正确的解法：这本质上是一个 Pareto 前沿问题，无法逃脱。有三种真正可行的方案：1. 增大 M （以 M=32 或 M=64 重建索引）。更大的 M 会移动 Pareto 曲线，在相同 ef_search 下获得更高召回率。2. 添加轻量重排序步骤：用 ef_search=100 检索 top-100（速度快），再用小型交叉编码器重排序为 top-10。交叉编码器能捕获 ANN 遗漏的结果，同时保持延迟在预算内。3. 混合检索：将 ANN 与关键词搜索结合，使用 RRF（倒数排名融合）。关键词侧能捕获向量表现欠佳的精确术语匹配；向量侧能捕获关键词遗漏的语义匹配。组合召回率在相同延迟下高于任何单一方案。

问题 7.2——更新与删除

朴素方案：可以自由插入和删除向量。

如何出错：HNSW 图在删除操作下会退化。删除是一个墓碑标记——节点被标记为无效，但其边保留。随着时间推移，遍历会浪费时间在死节点上。删除 30% 的向量后，召回率明显下降，延迟上升 2-3 倍。需要批量重建索引。

具体状态：一个电商目录有 5000 万个商品向量，更新频率高——每天有 5% 的商品被添加或删除。三个月后，原始向量的一半成为墓碑；查询速度慢了 2 倍；召回率下降了 5%。

正确的解法：大多数向量数据库实现了 周期性合并/重建。Milvus 执行分段级重建。Weaviate 和 Qdrant 在后台重建 HNSW 分片。Pinecone 将此抽象化（这是他们托管服务要解决的问题）。pgvector 的 HNSW 更为朴素——大量删除会有影响；可能需要手动 REINDEX。教训是：任何存在高流失率的目录都需要索引重建策略，而不仅仅是一个删除 API。

问题 7.3——混合检索（向量 + 关键词）

朴素方案：纯向量搜索。「嵌入能捕获语义，我们不需要关键词。」

如何出错：向量搜索在专有名词、精确术语、近期行话和代码方面表现欠佳。用户搜索「ELI5 transformer architecture」——嵌入可能会拉取通用的 ML 内容，但会遗漏真正的「ELI5 transformer」教程，因为「ELI5」是一个低频 token，嵌入对其压缩较重。搜索商品代码「GTX-4090」也会输给通用的游戏 GPU 内容。

具体状态：一个文档搜索系统使用纯向量 ANN。工程师反映，搜索 kubectl apply -f 无法可靠地找到字面包含该命令的文档，因为嵌入模型将该片段视为语义内容而非字面 token。

正确的解法: 混合检索。同时运行 BM25（关键词）和向量 ANN，然后通过 RRF（Reciprocal Rank Fusion）进行融合：对每个文档 d，score(d) = Σ（对每个排名 i）1/(k + rank_i(d))，其中 k 是平滑常数（通常为 60）。RRF 鲁棒性强（无需权重调优），相比纯向量或纯关键词方式可将 MRR（Mean Reciprocal Rank）提升 5-15%，是 2024-2025 年的行业默认方案。Elastic、Weaviate、Vespa 均原生支持 RRF。Pinecone 提供稀疏-稠密混合方式，思路类似。参见 03_search.md 的搜索侧视角章节第 §9 节。

问题 7.4 — 过滤器与 ANN 的交互

朴素方案：先执行 ANN，再过滤结果。

如何出错：如果过滤条件非常严格（仅覆盖语料库的 1%），获取 top-100 后只剩 1 个结果，则过滤子集的召回率极差。为弥补这一问题需获取 top-10000，但这会超出延迟预算。

具体状态：一个多租户 SaaS RAG 产品共有 1000 万个向量，但每个租户只能看到自己的数据——平均每个租户有 10,000 个向量。后过滤查询获取 top-100 后再按 tenant_id = X 过滤，该租户可能只剩 1-2 个向量。由于过度获取，延迟达到 50 毫秒；同时大多数候选被丢弃，质量很差。

正确的解法: 带紧密耦合元数据 bitmap 索引的过滤 ANN。Qdrant 率先采用这一模式：维护每个 payload 字段的 bitmap 索引；在 HNSW 遍历过程中，跳过不满足过滤条件的节点，但将其邻居作为候选继续探索。或者， 按租户分片 ——为每个租户分配独立的 collection 或分片。若每个租户的 collection 包含 1 万个向量，ANN 的速度将极快，在单个分片上进行精确 KNN 也是可行的。

问题 7.5 — 嵌入漂移 / 模型版本偏差

朴素方案：部署新的嵌入模型后，不对语料库重新嵌入。

如何出错：查询使用模型 V2（3072 维）进行嵌入，而语料库是用 V1（1536 维）构建的。要么因维度不匹配导致查询被拒（最佳情况），要么应用代码静默截断/填充向量导致召回率灾难性下降（最坏情况——静默数据损坏）。

即使维度相同， 嵌入空间也未对齐。"passport renewal" 的 V2 向量与同一短语的 V1 向量之间的夹角可能达到 1.2 弧度，尽管两者编码的语义相同。混合使用它们在数学上毫无意义。

具体状态：某公司将 5000 万文档的语料库从 OpenAI ada-002 切换到 text-embedding-3-large（两者均为 1536 维）但未重新嵌入。搜索质量一夜之间下降 30%，因为查询与文档处于不兼容的空间中。下降是静默的——没有报错，只是结果变差了。

正确的解法: 1. 维护语料库版本元数据。 为每个向量打上 embedding_model_version标签。查询必须使用相同版本，跨版本查询将被拦截。 2. 模型变更时重新嵌入。 这代价高昂：5000 万文档 × $0.0001/1k tokens × 约 500 tokens/文档 = API 费用 $2,500，加上数天的重建索引时间。请提前规划。 3. 影子灰度发布。 同时在旧版本和新版本中索引语料库；同时查询两者；比较质量；当 V2 胜出时切换。 4. 使用 sentence-transformers / 可锁定版本的开源模型。 降低供应商模型在你不知情的情况下发生变更的风险。

问题 7.6 — 高维灾难

朴素方案：使用最新模型提供的任意嵌入维度。

如何出错：在 d=3072（OpenAI text-embedding-3-large）时，每个向量占 12 KB。1 亿向量的索引在不含边的情况下原始向量就达 1.2 TB。内存成本急剧膨胀，缓存缺失率飙升，HNSW 图的边数也成比例增长。延迟随 d 线性增加。ANN 质量在极高维度下也会有微妙下降，因为"维度灾难"使所有距离趋向相似值，导致度量区分能力减弱。

具体状态：某企业从 384 维模型切换到 3072 维模型，期待获得更好的质量。结果延迟增加 6 倍，基础设施成本增加 8 倍，而质量仅提升 2-3%。

正确的解法: 1. 使用 Matryoshka 表示学习。 最新模型（OpenAI text-embedding-3-、BGE 变体）经过专门训练，截断为较短前缀（如 256 或 512 维）后仍能保留大部分信号。以 25% 的成本获得 90% 的质量。 2. 应用 PQ 压缩。 以 2-5% 的召回率换取 8-100 倍的内存压缩。在 10 亿+规模下至关重要。 3. 根据工作负载选择维度。* 384 维对大多数语义搜索已足够。768 是一个甜蜜点。仅在边际质量提升超过边际成本时才使用 1536+。

问题 7.7 — 实时索引更新

朴素方案：向量到达时立即插入，索引即时更新。

如何出错：HNSW 支持增量插入，但每次插入都需要进行一次 ef_construction 质量的搜索。在极高插入速率（每分片每秒 1 万次）下，插入延迟上升，查询延迟也因锁争用或写放大而开始受到影响。纯 IVF 需要定期完整重新聚类；若不这样做，新向量会落入过时的聚类分配中，召回率下降。

具体状态：一个欺诈检测系统每秒嵌入 1 万个事件。向量数据库开始对插入排队，队列增长到 100 万；由于索引被持续修改，查询延迟急剧上升。

正确的解法: 1. 批量插入。 缓冲 1-5 秒，批量插入，每批次提交一次到可搜索索引。 2. 两级索引。 热向量（最近一小时）存入频繁重建的小型内存索引；冷向量存入稳定的大型 HNSW。同时查询两者，合并结果。这是大多数欺诈检测和实时推荐系统部署的做法。 3. 定期完整重建。 安排后台每晚或按分片滚动重建。大多数生产部署每周重建一次完整 HNSW，以清除墓碑节点并重新优化边的选择。

§8. 故障模式演练

8.1 插入中途崩溃

状态：HNSW 节点部分添加——向量数据已写入，部分边已设置，反向边尚未提交。内存中的图状态不一致。

恢复：WAL 中包含该插入操作。重启时，重新加载快照，重放 WAL——该部分插入要么原子性地重新应用，要么被回滚。

持久性要点：操作在 WAL 完成 fsync 后即具有持久性；内存中的图变更是尽力而为的，可从 WAL 重建。

8.2 两次操作之间发生崩溃

状态：插入 A 已提交；插入 B 尚未开始。在两者之间发生干净崩溃。

恢复：重新加载快照，重放 WAL——A 被重新应用；B 对引擎而言是未知的。如果上游生产者已持久化 B，可以重试。

8.3 主节点 / 协调器宕机

状态：负责将查询路由到分片的协调器宕机。

恢复（以 Milvus 为例）：etcd 选举新的协调器。在 5-30 秒的故障转移窗口内，数据节点仍可提供读服务。插入操作进入队列。

Pinecone 对此进行了封装；用户在托管故障转移期间会看到短暂的延迟尖峰，但 API 不会中断。

持久性要点：分片元数据存储在协调服务中（etcd、ZooKeeper）；数据存储在分片节点上。只要两者各自存活，恢复是自动的。

8.4 网络分区 / 脑裂

状态：集群发生分区，两个半区都认为自己是权威方。

向量数据库通常是 AP（可用性 + 分区容忍）的。两个分区均可提供读服务（其中一个是过时数据）；写操作要么在两个分区都允许、之后进行协调（存在隐患），要么仅限于多数方。

Milvus 使用 Pulsar+etcd 实现单一写入路径；少数方的写入被阻塞。Pinecone 通过跨可用区复制和控制平面 quorum 屏蔽了这一问题。

持久性要点：写操作通过唯一确定的日志进行；两个分区在愈合后通过重放日志收敛。

8.5 节点永久丢失

状态：某个分片的存储永久丢失（磁盘故障，所有副本均已损毁）。

恢复：从上游真实数据源重新嵌入该丢失分片上的所有文档。这正是为什么主文本/图像存储必须是最终的真实数据源——嵌入始终可以重新生成。时间成本：在十亿规模下需数小时至数天；只要上游完整，数据是可恢复的。

极端情况：引擎有向量，但应用已丢失 doc_id → raw_text 映射关系。嵌入存在，但应用无法向用户展示匹配的内容。文档 ID 和原始文本必须独立于向量数据库持久化。

8.6 静默质量漂移

状态：开发者将嵌入接口切换到新模型（或供应商静默升级了模型）。新插入的是 V2；已有向量是 V1。

症状：质量逐渐下降。没有报错。ANN 仍然返回结果。用户反映搜索"感觉变差了"。

恢复：为每个向量打上模型版本标签；定期运行质量评估（在标注查询集上计算 MRR）；当版本出现分歧时重新嵌入。这是一个质量监控问题，而非崩溃恢复问题——向量数据库本身运行正常；只是语料库的语义理解发生了漂移。

§9. 为何不使用更显而易见的简单替代方案——精确 KNN 与线性扫描

ANN 索引最直接的替代方案是暴力 KNN：将所有向量存储在一个平坦数组中；对每个查询，计算与所有向量的距离；返回 top-K。无需索引，无需调优，结果始终精确。

为何行不通:

考虑一个拥有 10 亿向量、d=1536 的 OpenAI 语料库。原始存储：1B × 1536 × 4 B = 6 TB.

一次查询需要扫描全部 6 TB。即使所有数据都在内存中，并以每核 100 GFLOPS 的持续速度进行 SIMD 加速点积运算，也需要 1B × 1536 = 1.5 × 10¹² 次浮点运算 每核每查询 15 秒。在 64 核上并行执行：每次查询 230 毫秒，这还是假设完美线性扩展的情况，而实际上由于内存带宽饱和，这种情况永远不会发生。

在有效吞吐量为 1 TFLOPS 的 GPU 上：每次查询约 1.5 秒。对于交互式产品来说仍然太慢。

ANN 的 HNSW 在 10 亿向量中每次查询大约只访问 3000-5000 个向量。这是计算量的 20 万倍压缩。每次查询亚毫秒级，可在召回率与计算之间灵活权衡。

ANN 从什么语料库规模开始划算？ 大致在 N ≈ 10⁶ 时——在此之下，在单个 CPU/GPU 上用 NumPy 或 PyTorch 进行暴力计算即可。超过 10⁶ 时，ANN 必不可少。超过 10⁸ 时，精确 KNN 已无法负担。超过 10⁹ 时，即使 ANN 也需要大量工程投入（分片、IVFPQ、多层架构）。

暴力方案在极小规模下仍有存在价值（Chroma 的默认内存索引本质上就是这样），同时也作为 ANN 之后的 重排序步骤 ——对 ANN 的 top-100 候选结果使用精确全精度距离与查询进行重排序，以找回最后几个百分点的召回率。

§10. 扩展维度

第一类——均匀增长（向量增多，查询增多）

拐点一：单节点内存。 一台 256 GB 内存的节点在 d=1536 的 HNSW 中可容纳约 3000-5000 万个向量。超过这个限度就需要分片。策略： 哈希分片 对于 vector_id （负载均匀，查询扇出到每个分片）； 元数据分片 按租户或类别（减少扇出，但在租户分布不均时会产生热点）； 混合 ——按租户分片，对热门租户进行副本复制。

拐点二：写吞吐量。 单个 HNSW 索引的上限为每分片每秒 1k-10k 次插入。超过这个限度，需要批量插入、冷热两级索引，或转向 IVF（每次插入成本更低）。

拐点三：跨分片 ANN 成本。 有 S 个分片时，一次 top-K 查询需要从每个分片获取 K 的若干倍（如 top-3K），以确保合并后全局召回率。延迟受最慢分片约束。超过约 50-100 个分片后，尾部延迟占主导地位，需要请求对冲或副本选择。

拐点四：维度。 当 d ≥ 3072 时，即使 HNSW 也会受内存带宽限制；PQ 或 Matryoshka 截断变得不可或缺。

拐点五：十亿规模。 超过 10 亿向量时，带磁盘常驻聚类的 IVFPQ 是标准架构；纯 HNSW 内存占用过大。DiskANN 是另一种选择——基于 SSD 的类 HNSW 图，采用缓存感知布局。

第二类——热点强化（某个热门查询被反复执行）

常见于产品搜索（"iPhone"）、客服（"password reset"）、助手（"what time is it"）。解决方法：

  • 查询结果缓存。 缓存 (query_vector, top_K_results)。对四舍五入后的查询向量作为 key——许多用户会稍作变化地表达同一个问题，而嵌入会映射到几乎相同的向量，因此即使粗粒度的取整也能获得缓存命中。
  • 热分片副本复制。 如果某个租户或类别占据了大量流量，将该分片复制 3-5 份。
  • 预热缓存。 在低峰期预先计算已知热门查询的 top-K 结果。
  • 重排序缓存。 缓存 (query, candidate IDs) → reranked order 对重复查询跳过慢速的交叉编码器。

热点修复主要发生在向量数据库上游——在查询嵌入和结果缓存层。

§12. Embedding 模型——向量数据库所依赖的深度解析

向量数据库处于 embedding 模型的下游。向量数据库本身不理解语义；所有语义均来自上游。

闭源/商业模型

  • OpenAI text-embedding-ada-002 （2022）——1536 维，约 $0.0001/1k tokens。2023 年事实上的行业标准。
  • OpenAI text-embedding-3-small / -3-large （2024）——1536 / 3072 维；比 ada-002 更便宜、效果更好；支持 Matryoshka 截断。
  • Cohere embed-english-v3.0, embed-multilingual-v3.0 ——1024 维；多语言能力强；性价比高。
  • Google text-embedding-004 ——768 维，属于 Gemini 系列。

开源模型

  • BGE 系列（BAAI） — bge-large/base/small，1024/768/384 维。在开源模型中位居 MTEB（Massive Text Embedding Benchmark）排行榜榜首。易于微调。
  • E5 系列（Microsoft） ——指令微调 embedding； e5-mistral-7b-instruct 质量优秀。
  • sentence-transformers — all-MiniLM-L6-v2 （384 维，速度快）， all-mpnet-base-v2 （768 维，均衡）。万能工具箱。
  • Nomic nomic-embed-text-v1 ——768 维，完全开放权重。

图像 embedding

  • CLIP (OpenAI) ——图像与文本联合空间（512/768 维）。"以文搜图"工作流的基础。
  • DINOv2 (Meta) ——自监督图像 embedding，无文本对齐。纯视觉相似度。
  • SigLIP (Google) ——改进版 CLIP，采用 sigmoid loss。

微调 vs. 零样本

零样本部署快，但在特定领域表现平庸——例如，医疗 RAG 使用通用 ada-002 的 recall@10 可能只有 70%，而在医疗问答上微调后可达 85%。对于消费级 RAG，零样本已经足够；但对于专业领域（法律、医疗、代码），微调可将 MRR 或 recall@K 提升 5-20%。方法：通过 sentence-transformers 或 FlagEmbedding 对（query, relevant doc）对使用 in-batch negatives 的对比损失进行训练。

维度选择

  • 128-384 维：速度快，内存占用低，质量较低。适合成本为主要考量的大规模语料库，或用于喂给下游重排序模型。
  • 512-768 维：主力区间。sentence-transformers、BGE-base。质量与成本的良好平衡。
  • 1024-1536 维：高质量。ada-002、Cohere v3、BGE-large。高端 RAG 的默认选择。
  • 3072+ 维：顶级。text-embedding-3-large。边际收益递减；成本显著增加。

Matryoshka 表示学习 （2022 年；已部署于 OpenAI text-embedding-3-* 及众多 2024+ 模型）训练一个单一模型，使其前 N 维本身即可作为可用的 embedding。可存储完整向量并用截断前缀查询，或存储截断版本以后再恢复。正日益成为新模型的默认方案。

§13. RAG（检索增强生成）架构

RAG 推动了 GPT-3.5 / GPT-4 之后向量数据库的商业浪潮，让人们清楚地看到 LLM 需要一种接地机制。其结构已成为规范：

INGEST
  Documents (PDFs, HTML, Markdown, transcripts...)
    │
    ▼
  Chunking (200-1000 tokens per chunk, 50-200 token overlap)
    │
    ▼
  Embedding model
    │
    ▼
  Vector DB (chunk_id, vector, metadata={doc_id, chunk_idx, source, timestamp})


QUERY
  User question
    │
    ▼
  Embedding model (SAME)
    │
    ▼
  Vector DB ANN ── top-K chunks (K = 20-100)
    │
    ▼
  Cross-encoder reranker (Cohere rerank, BGE-reranker)
    │
    ▼
  Top-K' chunks (K' = 3-10)
    │
    ▼
  Prompt: "Answer using only the following passages:
           [chunk 1]... [chunk 2]... QUESTION: {q}"
    │
    ▼
  LLM (GPT-4, Claude, Llama)
    │
    ▼
  Answer with inline citations

分块策略

  • 固定大小加重叠 （默认的朴素 RAG）：每 500 个 token 切割，重叠 50 个 token。可预测；适用于叙述性文本。
  • 语义分块：按自然边界切割（段落标题、由 embedding 模型检测到的主题转换）。鲁棒性强但更难实现。
  • 递归分块 （LangChain 默认）：依次尝试段落 → 句子 → 字符。实用的回退链。
  • 文档结构感知：遵循标题、表格、代码块——对技术文档和代码至关重要。
  • 分块大小很重要：过小（< 100 token）会丢失上下文；过大（> 1500）会撑大提示词。300-800 是最佳区间。

使用交叉编码器重排序

ANN 提供快速粗粒度的 top-K。Cross-encoder 将 （query, passage）作为联合输入 并对相关性打分——较慢（CPU 上 10-100ms/对；GPU 上 1-10ms），但更精确。标准流水线：ANN top-100 → cross-encoder 重排序 → top-10 进入 LLM 提示词。nDCG@10（归一化折损累积增益）的典型提升：5-15%。常见重排序器：Cohere rerank、BGE-reranker-large、ms-marco-MiniLM。

幻觉问题

当检索到的上下文无法回答用户问题时，LLM 会频繁产生幻觉。缓解措施：过滤低置信度检索结果（若 top-K 余弦相似度过低则跳过 LLM）；要求引用并事后验证；多跳 / 智能体检索（细化查询并重试）；评估框架（RAGAS、TruLens）衡量忠实度和答案相关性。

§14. 混合搜索——向量 + 关键词

2023-2024 年业界共识： 混合 > 纯向量 > 纯关键词 适用于大多数检索任务。机制很直接——向量和关键词互补彼此的不足。

为何混合搜索更优

  • 关键词搜索 擅长：专有名词、产品代码、精确短语、技术术语、代码符号、embedding 模型从未见过的新词。
  • 向量搜索 擅长：同义改写、语义意图、跨语言相似度、无词汇重叠的概念匹配。

搜索 "java vs scala" 得益于关键词（模型 token "java" 出现频率低且含义明确）；搜索 "how do I make my service faster" 得益于向量（没有任何精确短语能匹配所有关于优化、缓存、性能分析等主题的文档）。

Reciprocal Rank Fusion (RRF)

经典融合算法。对于每个文档 d，在所有排序来源 i（例如 BM25 排名和向量排名）上对其打分：

RRF_score(d) = Σ_i  1 / (k + rank_i(d))

其中 k 是平滑常数（通常为 60，源自 Cormack 等人 2009 年的原始论文）。在任何排名中位置靠前的文档得分高；在多个排名中均出现的文档会被进一步加权。

RRF 非常鲁棒：它 无需权重调优，无需分数归一化，无需校准。它将每个排名视为顺序投票。它是 OpenSearch、Elasticsearch、Weaviate、Vespa 和大多数现代检索框架中混合搜索的默认算法。相比纯向量或纯关键词，MRR 通常提升 5-15%。

加权组合

替代方案是 score = α × BM25_score + (1-α) × cosine_score，α 可调节。这需要分数归一化（BM25 和余弦相似度处于不同量级）和参数调优。在标注数据集上精心调优后可超越 RRF，但脆弱性较高。

稀疏-稠密混合（SPLADE, ColBERT）

前沿方向： 学习型稀疏 embedding （SPLADE）和 后期交互模型 （ColBERT）架起了关键词与向量之间的桥梁。SPLADE 输出一个稀疏高维向量，行为类似于学习型 BM25 索引。ColBERT 计算每个 token 的 embedding 并逐 token 进行匹配。这些方案在 2024-2025 年已达到研究级和生产级，但比稠密向量 + BM25 的部署更为复杂。

§15. 应用场景一览

15.1 文档语义搜索（面向支持、知识库的 RAG）

2024-2025 年最常见的使用场景。企业对内部 Confluence、Notion、Google Docs、Slack、Zendesk 工单等进行索引。用户提问；RAG 流水线检索相关分块并喂给 LLM。Glean、Notion AI、GitHub Copilot Chat 以及无数内部"与文档对话"部署均采用此模式。

需求：高召回率（不能遗漏相关文档）、强过滤能力（只展示用户有权限查看的文档）、时效性（新文档必须在几分钟内可被检索）。pgvector、Pinecone、Weaviate 或 Elasticsearch 混合搜索均可，具体取决于现有基础设施。

15.2 推荐系统

Spotify 的 Discover Weekly、Netflix 的"更多类似内容"、YouTube 的下一个视频推荐、Amazon 的"购买此商品的顾客也购买了"——均在候选生成层使用向量相似度。用户的收听历史 → embedding；track embedding 空间中的最近邻成为候选；更重量级的排序模型决定最终顺序。

需求：超大规模（10M+ 用户，100M+ 物品，但候选集中只有约 10⁶-10⁷ 个活跃物品）；极低延迟（候选生成步骤的 p99 延迟低于 100ms）；近似即可，因为重排序阶段负责精细处理。ScaNN（YouTube）、Annoy（Spotify）以及基于 FAISS 的定制系统在这一规模下占主导地位。

15.3 图像和视频搜索

CLIP 风格的模型将图像和文本嵌入同一空间。"查找红色运动鞋的图片"转化为"将'红色运动鞋'嵌入为查询向量，并在图像 embedding 语料库上执行 ANN"。Pinterest 的视觉搜索、Google Lens、Shopify 的视觉相似商品功能均采用此方案的变体。

需求：存储由图像向量主导（通常为 CLIP 的 512 维）；写入代价高，因为每张图像都需要运行模型；度量通常是 L2 归一化 embedding 上的余弦相似度。Milvus、Qdrant 和自定义 FAISS 部署较为常见。

15.4 异常检测

对正常事件进行 embedding 并聚类。新事件通过与最近聚类中心的距离，或通过 embedding 空间内的 LOF（局部离群因子）进行打分。用于欺诈检测、网络入侵检测和工业传感器监控。

需求：持续写入新事件 embedding；快速最近邻查询；语料库持续增长，因此需要带定期重聚类的 IVF 索引或支持增量添加的 HNSW。

15.5 去重

查找近似重复的文章、商品或图像。对所有内容进行 embedding；按相似度阈值聚类（例如余弦相似度 > 0.95）；标记聚类以供人工审核或自动合并。用于新闻聚合器（不展示同一 AP 通讯稿的 50 个版本）、电商（合并重复商品列表）和内容审核（查找已知垃圾邮件的变体）。

需求：近似重复检测的召回率要求极高（漏检重复项是质量缺陷）；愿意使用强力 embedding 模型；去重处理通常是批量作业，而非实时处理。

15.6 客户支持意图匹配

新工单到来，对其进行 embedding，找到 K 个最近的历史工单，利用这些工单的解决模式来进行路由、建议回复或自动响应。Zendesk Answer Bot、Intercom Resolution Bot 等系统均采用此模式。

需求：每租户隔离（各公司的工单相互隔离）；快速写入（新工单必须在几秒内可被检索）；丰富的元数据过滤（status=open、priority=high 等）。多租户是主要的运营挑战。

15.7 代码搜索

语义代码搜索——"查找解析 JSON 并处理错误的函数"。使用代码感知模型（CodeBERT、OpenAI code embeddings）对每个函数进行 embedding。用于 GitHub Copilot 的上下文获取、Sourcegraph 语义搜索以及内部开发工具。通常与基于 AST 的结构化搜索和精确字符串搜索结合使用，以提升完整性。

15.8 多模态搜索

通过 CLIP 类模型实现文本到图像和图像到文本。用于购物（从照片中查找相似商品）、图库，以及内容审核（通过语义概念查找违规图像）。

15.9 个性化推荐

用户 embedding + 物品 embedding → 通过点积预测亲和度。用户 embedding 由交互历史计算；物品通过内容进行 embedding；最近邻成为推荐结果。

15.10 抄袭检测 / 段落相似度

对每个段落进行 embedding；通过 ANN 与已知来源语料库进行比对。用于学术诚信工具、AI 内容检测和法律电子发现。

§16. 真实世界的实现案例

Pinecone

领先的专用托管向量数据库。2023 年借助 RAG 浪潮获得大量融资（1 亿美元 B 轮，估值 7.5 亿美元）。基于专有内部实现构建（最初基于 HNSW，此后不断演进）。客户数量从独立 RAG 原型到企业部署达数十万之众。向量搜索的"一键轻松"之选。

Weaviate

开源向量数据库，提供托管云选项。GraphQL 优先的 API。模块化 embedding 集成（OpenAI、Cohere、Hugging Face）。在中端市场 RAG 产品中广泛使用。

Milvus / Zilliz

开源分布式向量数据库；已捐赠给 Linux 基金会。技术栈较重（Pulsar、etcd、MinIO/S3）。专为十亿级向量规模而构建。Zilliz Cloud 是其托管版本。

Qdrant

基于 Rust，单一二进制文件，精简且现代。Apache 2.0 协议。在过滤 ANN 方面表现出色。2023-2024 年作为"没有重量级技术栈的真正向量数据库"选择，市场份额持续上升。

pgvector

Postgres 扩展。最初仅支持 IVFFlat；HNSW 在 v0.5.0（2023 年中）中加入。现已由 Supabase、Neon、AWS RDS 开箱即用地提供。当技术栈中已有 Postgres 时的务实之选。

Spotify Discover Weekly (Annoy)

Spotify 于 2015 年开源了 Annoy（Approximate Nearest Neighbors Oh Yeah）——一种随机投影森林，与 HNSW 或 IVF 不同的 ANN 算法，通过随机超平面构建二叉树。仍在生产环境中为 Spotify 数千万首歌曲服务；Discover Weekly 的候选生成对用户和歌曲进行 embedding，并查询最近邻。

YouTube 候选生成（ScaNN）

世界上部署规模最大的 ANN 系统之一。候选生成使用 ScaNN（Scalable Nearest Neighbors），这是 Google 的 IVFPQ 变体，采用各向异性向量量化。跨多台机器对数十亿视频进行分片；重量级排序器对候选进行排序。

Pinterest 视觉搜索、Notion AI、Perplexity、Copilot

Pinterest "Lens"：CLIP 类图像 embedding + 对数十亿图钉执行 ANN。Notion AI：按工作区进行严格租户隔离的 RAG。Perplexity：以实时网页 RAG 为核心产品形态。GitHub Copilot：IDE 中的语义代码搜索以及聊天的上下文获取。领域各异，核心模式相同。

Twitter/X "For You"、LinkedIn 相关功能

双塔神经网络对用户和物品进行 embedding；ANN 生成候选；排序模型进行排序。Twitter 于 2023 年开源了此技术栈的部分内容；LinkedIn 的搜索和"你可能认识的人"使用内部基础设施，结合了 LSH（局部敏感哈希）、HNSW 和 learned-to-rank 重排序器。

§17. 总结

向量数据库是一种可水平扩展的、持久化的高维 embedding ANN 索引——以牺牲精确性为代价，实现基于内容语义、支持元数据过滤的快速相似度搜索。HNSW 主导内存中中等规模的场景；IVFPQ 主导磁盘上十亿级规模的场景；真正的语义来自上游的 embedding 模型，向量数据库只是其上的"笨但快"检索层。若已有 Postgres 且向量数量 ≤ 50M，选 pgvector；需要托管服务，选 Pinecone 或 Weaviate；自托管，选 Milvus 或 Qdrant；已有搜索基础设施，选 ES/OpenSearch；构建自定义平台，选 FAISS。始终搭配一个主存储作为真相来源，始终为向量标注其 embedding 模型版本，始终为模型变更规划重新 embedding 的方案，并始终在确定纯向量方案之前对混合检索（BM25 + 向量的 RRF）进行基准测试。
