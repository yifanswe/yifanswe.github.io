负载均衡器

2026/05/22 · 后端基础组件
目录
目录

负载均衡是一类基础设施，负责将入站工作分发到后端池，并具备健康感知、流量策略和故障隔离能力。它是将"一组服务器"变为"一项服务"的那一层。其变体差异巨大——在内核中以每秒 1000 万包（Mpps）转发的第 4 层（L4）包转发器，与在用户空间以每秒 10 万请求（RPS）进行 TLS 终止路由的第 7 层（L7）HTTP 代理几乎毫无共同之处。它们共享名称和目标，却几乎不共享代码。本文将负载均衡作为一类技术进行阐述，梳理其设计空间，并结合公共 Web、内部微服务、数据库连接池、视频流、实时游戏和读副本路由等场景给出示例。

§1. 负载均衡器是什么——以及不是什么

负载均衡器（LB）是流量分发器：给定一个入站工作单元（TCP 包、HTTP 请求、UDP 数据报、DB 查询），它从池中选择一个后端并转发。它保持池的整洁——健康检查失败的后端被驱逐，新后端平滑加入，成员变化时流量重新分配，无需重置所有活跃连接。

两个主要维度划分设计空间。

L4 与 L7。 L4 LB 工作在传输层——它们看到源/目标 IP、源/目标端口和协议（五元组）。它们不解析 HTTP，不终止 TLS，不理解路由。它们速度极快（在现代 XDP——eXpress Data Path——路径上每包亚微秒级），近乎无状态（一张将五元组映射到后端的连接表），且协议无关（TCP、UDP、QUIC）。Google Maglev、Facebook Katran、Cloudflare Unimog、AWS NLB（Network Load Balancer）和 Linux IPVS 都是 L4。L7 LB 终止传输层（通常是 TCP+TLS），工作在应用层协议上——HTTP、gRPC、Kafka 线协议、Redis 协议。它们按路径、头部、请求体、JWT（JSON Web Token）声明路由；重写头部；重试；做金丝雀发布；运行 WAF（Web Application Firewall）。NGINX、HAProxy、Envoy、Traefik、AWS ALB（Application Load Balancer）、GCP HTTP(S) LB 都是 L7。每请求速度更慢（毫秒而非微秒），但能力强大得多。

硬件与软件。 传统路线是硬件设备——F5 Big-IP、Citrix NetScaler、A10——配备 ASIC 包引擎和专有操作系统。现代路线是运行在通用 x86/ARM 上的软件——NGINX、HAProxy、Envoy 运行在用户空间；IPVS、Katran、Unimog 通过 XDP/eBPF（extended Berkeley Packet Filter）运行在内核中。硬件在每比特功耗上胜出；软件在迭代速度和成本上胜出。到 2018 年，业界的裁决已经明确：超大规模厂商均用 eBPF 或 DPDK（Data Plane Development Kit）构建了软件 L4 LB。硬件设备仍然向那些希望有单一供应商负责的企业出货。

负载均衡器 不是:

  • 不是数据库。 没有持久状态。丢失 LB 后连接会重置，但不会丢失业务数据。
  • 不是服务网格。 网格是一种架构（sidecar 代理 + 控制平面 + mTLS——双向 Transport Layer Security——无处不在）；它 使用 LB（通常是 Envoy）作为构建块。
  • 不是 CDN（Content Delivery Network）。 CDN 缓存；LB 路由。CDN 内部有 LB（Cloudflare 运行 Unimog），但 CDN 的价值在于其缓存层次结构。
  • 不是 DNS。 基于 DNS 的负载均衡（多条 A 记录）是一种糟糕的替代方案——见 §9。
  • 不是重试框架。 端到端重试语义（幂等键、saga 补偿、指数退避）属于客户端和应用层的职责。

这个类别之所以重要，是因为工程师常常将 LB 用于解决它无法解决的问题（持久性、精确一次投递、跨区域一致性），却对它真正能干净解决的问题视而不见（慢启动保护、热路由卸载、蓝绿部署门控）。

§2. 固有保证

负载均衡器按设计提供：

  • 流量分发 到池中，按某种权重策略（轮询、最少连接、P2C——Power of Two Choices——随机、一致性哈希）近似均匀分配。
  • 健康感知 在检查窗口内（2–10s 主动，亚秒级被动）。
  • 池内故障转移 ——停止工作的后端不再接收新流量；部分幂等请求可被重试。
  • 活跃连接的流粘性 ——一旦 TCP 流被固定到后端 B，即使池成员变动，它仍保持在 B 上。
  • 对客户端呈现单一一致的 VIP（Virtual IP）身份 ——成员变动对外不可见。

它们 不是 提供：

  • 持久性。 崩溃后端上的飞行中请求会丢失。LB 不会缓冲或持久化。
  • 精确一次。 重试到新后端可能导致重复执行；幂等性是应用层的职责。
  • 端到端正确性。 LB 看到"后端返回 200"；它不知道 DB 写入是否已提交。
  • 跨流的顺序保证。 同一客户端的两个请求可能以任意顺序落到不同后端。
  • 强一致路由。 两台 LB 机器可能短暂地对哪个后端健康存在分歧。路由对此容忍——要么通过共享表的确定性（Maglev），要么通过应用层能够承受任意选择。
  • 事务性排水。 排水是尽力而为的；连接迁移没有两阶段提交。

与 TCP 相同的模式：下层交付字节，上层应用构建正确性保证。任何将 LB 视为持久层的人都会感到意外。

关键区分在于 L4（快速，功能简单） vs L7（较慢，功能智能）。大多数真实技术栈会 链式组合 两者：L4 负责消化每秒数据包的预算，然后将流量转发给 L7 进行 TLS 终止、路径路由和策略执行。Cloudflare：Unimog → NGINX 变种。Google：Maglev → GFE（Google 前端）。AWS：NLB + ALB 链式组合。

另一个关键维度是 内核绕过与驻内核。DPDK 在延迟下界和 PPS（每秒数据包数）上界方面表现出色，代价是丧失所有内核网络功能（iptables、tcpdump、eBPF 追踪）。XDP/eBPF 是现代折中方案：留在内核中，在网络栈其余部分之前执行一段经过验证的字节码程序，速度约为用户态的 10 倍，同时保留所有内核工具链。到 2026 年，新的 L4 LB 项目几乎普遍选择 XDP/eBPF 而非 DPDK，除非需要绝对线速确定性。

§4. 字节级原理

以下是技术落地的核心：负载分发算法、数据结构、内核绕过路径，以及一个完整的端到端数据包处理流程。

4a. 一致性哈希的几种变体

朴素取模方案 backend_id = hash(5-tuple) mod N 当 N 固定时可用，一旦 N 变化便会灾难性地失效：新增一个后端会重新映射 (N-1)/N 所有流的大部分，几乎每条存活的 TCP 连接都会 RESET。一致性哈希解决了这个问题。生产环境中有三种变体。

环形哈希（DynamoDB 风格、Cassandra 风格）。 每个后端在 2^32 的哈希环上映射到 K 个位置（虚拟节点，K=100–500 以平衡方差）。对于键 X：计算 H(X)，顺时针查找第一个 vnode，返回对应后端。通过对有序 vnode 数组进行二分查找，查找复杂度为 O(log(K·N))。K=200、N=1000 时共 20 万条目，可放入 L2 缓存，查找约 100 ns。新增后端时放置 K 个新 vnode，约 1/N 的键空间迁移。方差约为 K^(-1/2)：K=100、N=100 时负载标准差约 10%。被数据库（Cassandra、DynamoDB、Riak）采用，适用于 LB 层与存储放置协同设计的场景。

Rendezvous 哈希（HRW — 最高随机权重）。 对于键 X 和 N 个后端：计算每个后端的 H(X, B_i)，选取最大值。每次查找 O(N)——在大 N 的数据平面上过慢。但新增节点时恰好只有 1/N 的键迁移，无需调整 vnode，分布天然均匀。当 N 较小（<1000）时是合适的选择。部分 CDN 和 HDFS 的分块放置使用此方案。

Maglev（Google 的查找表变体）。 被 Maglev、Katran、Unimog 及大多数现代 XDP LB 采用。预先计算一个大小为 T（质数，通常为 65537）的固定查找表 M，其中 M[i] = 分配给槽位 i 的后端。选取后端时： slot = hash mod T，返回 M[slot]。单次数组索引—— ~10 ns，无分支，可放入 L1 缓存。

槽位分配算法（NSDI 2016 §3.4）：

  1. 对每个后端 B_i，使用两个哈希函数预计算 [0, T) 的排列 P_i：\n - offset = H1(B_i) mod T - skip = H2(B_i) mod (T-1) + 1 （T 为质数 → gcd(skip, T) = 1 → 排列覆盖 [0, T)）\n - P_i[j] = (offset + j * skip) mod T
  2. 按轮次填充 M。第 j 轮中，每个后端尝试申领槽位 P_i[j]。若已被占用，则推进其游标，在下一轮中按其偏好顺序尝试下一个槽位。

示例，T=7，后端集合 {B1, B2, B3}：

P_1 = [3, 0, 4, 1, 5, 2, 6]    // B1's preferred slot order
P_2 = [0, 2, 4, 6, 1, 3, 5]
P_3 = [3, 4, 5, 6, 0, 1, 2]

Round 0: B1 claims 3 → M[3]=B1; B2 claims 0 → M[0]=B2;
         B3 tries 3, taken, cursor advances
Round 1: B1 tries 0, taken; B2 claims 2 → M[2]=B2;
         B3 claims 4 → M[4]=B3
Round 2: B1 tries 4, taken; B2 tries 4, taken;
         B3 claims 5 → M[5]=B3
Round 3: B1 claims 1 → M[1]=B1; B2 claims 6 → M[6]=B2;
         B3 done (has T/N ≈ 2 slots)

Final: M = [B2, B1, B2, B1, B3, B3, B2]

Lookup for a 5-tuple hashing to 1234567:
  slot = 1234567 mod 7 = 4 → backend = M[4] = B3

当 B2 故障并以 {B1, B3} 重建时，B2 的槽位（0、2、6）按剩余后端的偏好顺序重新分配，而不含 B2 的槽位（1、3、4、5）大概率保持不变。在 T=65537 和实际后端数量下，每次后端变更约有 ~T/N 个槽位变更——恰好是那些 必须 迁移的流（即在故障后端上的流），存活流中只有极少部分受到扰动。

Maglev 的优势：O(1) 查找（优于 O(log N)），构造上保证均匀分布，单数组数据结构。重建复杂度为每次后端变更 O(T)，在控制平面执行一次，以约 64KB 的二进制块推送至每个数据平面节点。

4b. 加权轮询内部实现

当后端容量不同时，LB 使用权重调度器。NGINX/IPVS/HAProxy 的经典实现：

On each request:
  for each backend i:
    b[i].current_weight += b[i].effective_weight
  pick k = argmax(b[i].current_weight)
  b[k].current_weight -= sum(b[i].effective_weight)
  return k

权重 5:1:1 产生交错序列 A A B A A C A A A B A A C ... 而非 A A A A A B C A A A A A B C ... ——延迟分布更优。每次请求 O(N)，N < 10,000 时完全够用。

4c. 最少连接数（最小堆）

朴素的 O(N) 扫描在每秒 500 万次决策时过慢。实际实现： 以连接数为键的最小堆 active_connections + 倒排索引 backend_id → heap_index。新流到来：取堆顶，自增，执行 O(log N) 上浮。流结束：自减，执行 O(log N) 下沉。N=1000 时，log N = 每次决策 10 次操作。

生产优化： P2C（两次随机选择）。 随机选取两个后端，发送到负载较低的那个。每次请求 O(1)，最大负载方差从纯随机的 O(log N / log log N) 降至 O(log log N / log d)，并避免"雷群效应"——即所有 LB 节点在同一时刻独立选出同一个"最少连接"后端。Envoy 的 LEAST_REQUEST 默认算法。

4d. 连接追踪表（五元组 → 后端）

L4 LB 不仅依赖一致性哈希，还维护一张流表，将存活的五元组映射到后端，确保即使一致性哈希表刚刚变更，中途数据包仍能命中同一后端：

struct flow_entry {
    uint128_t five_tuple_hash;
    uint32_t  backend_id;
    uint64_t  last_seen_ns;
    uint32_t  packets, bytes;
};
flow_entry table[16M];  // open-addressed, 2x expected concurrent flow count

数据包到达时：计算五元组哈希，在 hash mod table_size处探测，若命中且未过期则使用缓存的后端，否则降级到 Maglev。空闲超过 60 秒的陈旧条目被回收。

流表是 粘性 层。Maglev 表是 正确性 层（新流 + 变更后的流）。两者结合：流中途稳定粘性 + 变更时有界重新分配 + 热路径 O(1) 查找。

4e. 内核绕过路径

三种方案，性能依次提升，运维复杂度依次上升：

方案一：内核网络栈。 NIC → 内核环形缓冲区 → 软中断 → IP 层 → netfilter → 用户态 socket → LB → socket → IP → NIC。 ~每包 10 µs，每核心约 50 万 pps，约 6 次系统调用/上下文切换边界。标准 NGINX、HAProxy、IPVS 路径。在 L7 场景下毫秒级解析远比微秒级 socket I/O 耗时更多，因此足够。但 L4 大规模场景下不足。

方案二：DPDK 内核绕过。 DPDK 将 NIC 完全从内核接管。用户态进程通过 PMD（轮询模式驱动）直接轮询 NIC 环形缓冲区。数据包从不经过内核。大页内存（2 MB 或 1 GB）避免 TLB（地址转换后备缓冲区）缺失。每个 LB 线程绑定到一个 CPU 核心；每个 NIC 队列映射到一个核心（RSS——接收侧扩展——在 NIC ASIC 上按五元组哈希分发数据包）。 ~每包 100 ns，每核心 10 Mpps，100 Gbps 时达到线速。 运维代价：无 tcpdump，无 iptables；LB 必须自行实现 ARP（地址解析协议）、ICMP 和分片处理。

方案三：XDP/eBPF。 内核钩子位于 之前 网络栈其余部分之前。eBPF 程序检查数据包后返回 XDP_PASS （继续上送协议栈）， XDP_DROP （丢弃——DDoS 流量清洗）， XDP_TX （改写后从同一 NIC 重新发出），或 XDP_REDIRECT （发往不同 NIC）。 ~每包 50–200 ns，每核心 5–10 Mpps。 eBPF 是经过验证的安全字节码（无无限循环，内核验证器拒绝任何无法在约 100 万条指令内证明终止的程序）。Maglev 表存储在 BPF map （驻内核的哈希/数组映射）中；用户态通过系统调用更新，eBPF 程序在处理数据包时读取。运维优势：仍在内核中——tcpdump、perf、ftrace 对 XDP 程序 PASS 的流量均有效。2026 年新 L4 LB 项目的默认方案。

4f. 端到端数据包处理

一个 TCP SYN 数据包到达基于 XDP 的 L4 LB：

T=0    ns   Packet hits NIC. Ethernet header parsed by NIC ASIC.
            RSS: NIC hashes the 5-tuple via Toeplitz hash, picks
            NIC queue 7 of 16. Queue 7 IRQ-bound to CPU core 7.

T=50   ns   Core 7 NAPI poll reads packet descriptor from queue 7's
            RX ring buffer.

T=80   ns   XDP eBPF program invoked. Walks headers:
            - bytes [0..14): Ethernet. Read ethertype at offset 12;
              0x0800 (IPv4) or 0x86DD (IPv6), else XDP_PASS.
            - bytes [14..34): IPv4. Read proto at +23 (TCP=6),
              dst_ip at +30. If dst_ip != VIP, XDP_PASS.
            - bytes [34..54): TCP. Read src_port, dst_port, flags.

T=90   ns   Compute 5-tuple hash: CRC32C with SSE4.2 instruction
            over (src_ip, src_port, dst_ip, dst_port, proto). ~5 ns.

T=100  ns   Flow table lookup (BPF map). Probe at hash mod tbl_size.
            Compare full 5-tuple. Hit: use cached backend_id, jump
            to step 6. Miss (new SYN): continue.

T=130  ns   Maglev lookup:
              slot = hash mod 65537
              backend_id = maglev_tbl[slot]  // single mem access
            Write flow table entry (5-tuple_hash, backend_id, now()).

T=150  ns   Backend info lookup (BPF map): dest_mac, dest_ip.

T=170  ns   Rewrite packet:
            - DSR (Direct Server Return): rewrite dest_mac to
              backend_mac, leave IP unchanged. Backend has VIP bound
              on loopback, accepts the packet, sends response
              directly to client. Huge return-bandwidth savings.
            - IPIP / GUE (Generic UDP Encapsulation) / GENEVE tunnel:
              prepend outer IP with dest = backend_ip. More flexible
              (crosses L3 boundaries), more overhead.

T=180  ns   Return XDP_TX. NIC transmits out the same port.

T=200  ns   Packet on wire to backend. Total LB hop ~200 ns, no
            user-space copy, no syscall, no context switch.

按此速率，一台装有 16 个 NIC 队列的 16 核服务器在 CPU 饱和之前，100 Gbps NIC 带宽早已耗尽。现代超大规模系统中的 L4 LB 层是 带宽瓶颈，而非 CPU 瓶颈.

4g. 与未选用方案的权衡对比

  • Maglev 与环形哈希：Maglev 在查找速度（O(1) vs O(log N)）和构造上均匀分布方面更优。当 LB 与存储协同设计时（如 Cassandra），环形哈希更合适。数据平面选 Maglev，存储放置选环形哈希。
  • Maglev 与 Rendezvous：在 N=1000 时，Maglev 的 O(1) 比 Rendezvous 的 O(N) 快 1000 倍。Rendezvous 适用于每分钟执行一次的控制平面决策；Maglev 适用于每微秒执行一次的数据平面决策。
  • Maglev 与 hash-mod-N：hash-mod-N 是 O(1) 且简单，但每次后端变更会重新映射 (N-1)/N 的所有流。切勿用于有状态 LB。
  • L4 与 L7 转发：L4 更快（1 次内存访问 vs HTTP 解析）且与协议无关。L7 可基于路径/头部路由。两者链式组合使用。

4h. 软状态持久性

L4 LB 在请求路径上 无持久状态。Maglev 表由后端列表确定性重建（后端列表存储在强一致性控制平面——etcd、Consul、Zookeeper——中）。流表是 软状态：若某台 LB 节点故障，存活的 LB（通过 ECMP——等价多路径——重新哈希获得这些流）会计算出 相同的 Maglev 表 并路由到 同一后端。在约 100 ms 的 ECMP 重新哈希窗口内丢弃少量数据包后，流量恢复正常。这正是使 LB 层从持久性角度看无状态的架构技巧：无需复制，无需故障转移流程。

L7 LB 也近似无状态——TLS 会话状态、HTTP/2 连接状态保存在进程内存中。若某台 L7 节点故障，进行中的请求会 RESET；客户端重试；流量通过底层 ECMP 转移。最坏情况：故障节点上进行中的请求有 5–10 秒的客户端可见降级。

  • ECMP 使单个 anycast PoP 容量弹性扩展。N 台 L4 LB 节点位于边缘路由器后端；ECMP 按五元组哈希分发数据包。当某台 LB 故障时，ECMP 重新哈希；存活的 LB 运行 相同的 Maglev 表 ，因此流量路由到 同一后端 ，无需状态迁移。这正是使 LB 层持久无状态的原因。

§7. 疑难问题

问题 1：变更期间的稳定哈希

朴素的想法: backend_id = hash(5-tuple) mod N. 失败场景：部署时 N 从 100 变为 101，99% 的流获得不同的 backend_id，全部 RESET——每次部署网站看似宕机。 解决方案：Maglev 查找表（§4）——N→N+1 时只有约 1/N 的流受扰动。加上流追踪器，即使表发生变更，也会将存活的五元组固定到其原始后端。

状态演练：100 个后端上 100 万个存活 TCP 连接，T=65537，每个后端约 655 个槽位。T=10s 时，B73 优雅移除；以 N=99 重建表。指向 B73 的约 655 个槽位分散到其余 99 个后端。流表中的存活流 → 保持原始后端直至排空。新流 → 新表。总扰动：约 1 万个原在 B73 上的流，其他后端几乎为零。1% 的后端变更带来约 1% 的扰动。

问题 2：慢启动保护——新后端被压垮

朴素的想法：新后端 B100 加入；表重建；B100 立即获得其 655 个槽位和 1% 的全量流量。 失败场景：B100 缓存冷、JIT（即时编译器）冷、连接池为空。p99 飙升至 5 秒；B100 触发自身熔断器；健康检查失败；B100 被驱逐。连接池恢复为假设 B100 在线时规划的 99 个后端。级联故障。 解决方案：加权预热。表构建时 B100 权重为 0.01 → 0.1 → 0.5 → 1.0，历时 60 秒。Envoy 的 slow_start_config 以声明式方式实现；对于 Maglev，B100 按权重比例申领部分槽位。常见于支付场景（新网关预热到慢上游（如卡组织）的连接）和推荐排序场景（内存中 ML 模型需要 30 秒 JIT 编译热路径）。

问题 3：热点后端——单台节点在均衡 LB 下仍过载

朴素的想法：LB 执行均匀的 Maglev 哈希或轮询。 失败场景：某后端托管着一位明星用户，承受 500 万次请求/秒；或磁盘退化；或运行在嘈杂的虚拟机监控器上。LB 毫不知情，持续发送其应有份额。 解决方案：P2C + 异常点检测。P2C：每次请求随机选两个后端，发送到负载较低的那个；最大负载方差大幅下降。异常点检测（Envoy outlier_detection）：驱逐错误率或延迟超过集群均值 3σ 且持续 30 秒以上的后端，经过恢复窗口后通过探测将其重新接入。搜索工作负载大量使用此机制——一个慢速分片会拉垮扇出查询的 p99。ML 推理服务对触发热量降频的 GPU 节点同样采用此策略。

问题 4：健康检查误报——误将全部后端驱逐

朴素的想法：每个后端的 /healthz 检查"能否访问 DB？"DB 发生 2 分钟故障。 失败场景：所有后端同时不健康。LB 驱逐全部后端。DB 恢复后网站仍然黑屏。 解决方案: 分层健康检查 + 恐慌模式。 /healthz = 进程存活（TCP accept 正常工作）。 /ready = 可以对外提供服务。负载均衡器使用 /healthz；编排器（Kubernetes）同时使用两者。恐慌模式（Envoy healthy_panic_threshold，默认 50%）：如果超过一半的实例不健康，则将流量路由到所有实例，包括"不健康"的——将流量发往可能还能处理的后端，总胜过发往一个保证为空的池。以一个对接 Prometheus 采集集群的可观测性后端为例：100 个 ingester，全部健康；下游存储出现抖动；到 T=30s 时，100 个实例全部报告不健康。无恐慌模式：每次采集都返回 503，告警打到错误的事情上。有恐慌模式：流量仍在流动，错误率上限约为 30% 而非 100%。

问题 5：粘性会话与故障转移

朴素的想法：用户会话存储在后端内存中；负载均衡器通过 cookie 粘性将用户固定到 B7。 失败场景：B7 宕机，用户丢失购物车。 解决方案：将会话状态迁移至 Redis/DynamoDB；后端变为无状态。粘性会话成为 性能 优化手段（暖 L1 缓存），而非正确性要求。对于确实需要粘性的场景——WebSocket（长连接 TCP，无法透明重建）、实时游戏（每个玩家的 UDP 流在对局期间必须到达同一个游戏状态权威节点）、交互式视频会议（同一 SFU——Selective Forwarding Unit——为房间混音）——故障转移语义必须明确：负载均衡器发送 RST，客户端重连，新后端处理重连（可能需要从会话存储恢复状态）。应用协议必须能够容忍这种情况。

问题 6：TLS 终止开销

朴素的想法：每个请求都进行一次全新的 TLS 1.2 RSA-2048 握手。 失败场景：以 100 万次握手/秒计，约需 500 个 CPU 核心处理加密——负载均衡器因 TLS 而非路由成为 CPU 瓶颈。 解决方案：一系列优化措施。(1) TLS 1.3 配合 X25519 ECDHE：1-RTT 握手，ECDSA P-256 约 0.1 ms vs RSA 约 1 ms——约提升 10 倍。(2) Session resumption 通过 PSK（Pre-Shared Key）票据：回访客户端跳过握手；约 95% 变为 0-RTT 或 1-RTT 简化握手——约提升 20 倍。(3) 硬件加速：AES-NI、QAT（Intel QuickAssist）、HSM（Hardware Security Module）——5 倍。(4) 连接复用：HTTP/2 复用单条 TLS 连接处理多个请求——握手成本摊销至数百个请求。综合效果：100 万次握手/秒 → 约 5000 次握手/秒，轻松应对。公网依赖此机制；若没有 HTTP/2 复用和证书缓存，网格 mTLS 会耗尽 CPU。

问题 7：SYN flood 与 L3/L4 DDoS

朴素的想法：攻击者以伪造源 IP 每秒发送 100 万个 SYN 包；负载均衡器为每个 SYN 分配一条流表项。 失败场景：流表被填满，合法流被挤出，站点宕机。 解决方案：L4 SYN cookies——将连接状态编码进 SYN-ACK 的初始序列号；负载均衡器对半开连接不存储任何状态。在分配资源前，于 ACK 到达时验证 cookie。Linux 内核原生支持此机制；基于 XDP 的负载均衡器在 eBPF 中实现。对于容量型 DDoS（100 Gbps 垃圾流量），负载均衡器触发 BGP 黑洞将攻击者前缀在上游丢弃；Cloudflare 通过其全球 DDoS 检测器自动完成此操作。

§8. 故障模式

模式 A：单台 L4 负载均衡器宕机

10 台 L4 负载均衡器，通过 ECMP 分散，各自承载约 10% 的流。LB-7 故障。边缘路由器的 ECMP 检测到其宕机（BFD——Bidirectional Forwarding Detection——每 100 ms 链路探测，或 BGP 会话中断）。ECMP 在剩余 9 台上重新哈希。原本命中 LB-7 的流现在命中 LB-3；LB-3 无对应流表项，回退到 Maglev—— 每台负载均衡器上相同的 Maglev 表 → 同一后端。流继续。过渡期间约丢失 5–10 个数据包。 持久性要点：Maglev 表本身由控制平面复制到每台负载均衡器；无需跨节点复制每条流的状态。

模式 B：整个 PoP 断电

PoP 光纤中断。BGP 检测到无对等体（或外部监控撤回路由）。在 10–60 秒内（BGP 收敛），发往 anycast VIP 的流量切换至最近的 PoP。死亡 PoP 上所有在途 TCP 连接被 RESET；客户端重试，重试落至新 PoP；新连接立即正常工作。 持久性要点：BGP 路由 + 每个 PoP 均可到达的后端池。

模式 C：控制平面宕机

etcd/Consul 集群崩溃。数据平面继续运行，使用 最后已知的 Maglev 表 ——数据平面在热路径中从不读取控制平面；表已预先推送。后端的增删操作停滞；宕机后端被负载均衡器路由 → 负载均衡器仍转发 → 后端无响应 → L7 负载均衡器通过被动健康检查（超时）发现并剔除。 持久性要点：数据平面的本地缓存表；控制平面不在请求路径上。这是一个关键设计决策——控制平面的不可用性通常高于数据平面，数据平面必须能够扛过 30 分钟的控制平面中断。

模式 D：L4 与 L7 之间的分区

L4 负载均衡器只能到达 30% 的 L7 负载均衡器（其余被隔离）。L4 的 Maglev 表仍将所有 L7 列为存活状态。70% 的流超时。 解决方案：L4 对 L7 层有独立的主动健康检查（与 L7 对后端的检查相互独立）。约 5 秒内，L4 发现宕机的 L7，重建表，继续在剩余 30% 上运行。 持久性要点：每台 L4 基于本地探测自行决定剔除——无需跨负载均衡器协调。

模式 E：永久性损失 + 同时部署

100 个后端；30 个正在滚动更新（每次 10 个）；更新进行中，硬件故障再带走 15 个。有效池从 100 → 55。每个后端流量约增加 80%。若基线容量为 60%，则推至 108% → 螺旋崩溃。 解决方案：容量冗余 N+2（一个应对故障，一个应对部署），加上若观察到错误率飙升则暂停部署的门控机制。结合慢启动（问题 2），避免恢复期间的预热实例被压垮。

§9. 为什么不用轮询 DNS

经典初级答案："直接给 DNS 多个 A 记录就好了。"几乎说对了，却完全错了。

dig api.example.com → [198.51.100.1, ..., .10]。客户端选择一个（通常是第一个），建立 TCP 连接。

故障 1——无健康感知。 .5 后端宕机。DNS 仍返回该地址。在 DNS 更新并且全球所有解析器的 TTL 过期之前，约 10% 的流量命中死亡后端。60 秒的 TTL 只是 下限；ISP 和移动运营商的缓存时间远超 TTL。

故障 2——无粘性。 客户端第一次请求可能选 .3，第二次选 .7（解析器行为由实现决定）。内存中会话、粘性上传分片、WebSocket——全部失效。

故障 3——分配不均。 轮询 DNS 在 解析器之间分配，而非在客户端之间。若 80% 的用户在同一个缓存了 .1 的大型 ISP 解析器后面，你会得到 80% 的流量打到 .1。

故障 4——无 L7 功能。 无路径路由、无 TLS 终止、无 WAF、无限流、无金丝雀发布。

故障 5——故障转移需要数分钟至数小时。 DNS 驱动的故障转移受限于全球最差的缓存 TTL。Anycast + 负载均衡器故障转移受限于 BGP 收敛——10–60 秒，有时在反射器辅助下低于 10 秒。对于 99.99% SLA，仅用 DNS 是不可行的。

DNS 负载均衡作为一层是合理的，置于 上方 负载均衡器层——用于决定将客户端发往哪个区域的 anycast VIP。作为替代品则不行， 处对 负载均衡器层。

§10. 扩展维度

类型 1：均匀增长（更多后端，更多流量）

变化内容 拓扑
1k RPS 单台 NGINX，无 L4 层
100k RPS NGINX 集群（3 台），ECMP 组合，主动-主动
1M RPS L4 + L7 分离；L7 下方 L4 启用内核旁路
10M RPS 多区域 anycast，3 个区域，BGP 驱动故障转移

拐点：约 10 万 RPS（超出单台 NGINX → 集群 + ECMP）；约 100 万 RPS（超出纯用户态 → L4 启用内核旁路）；约 1000 万 RPS（超出单区域 → 多区域）；约 1 亿 RPS（超出单 Maglev 表 → 按区域分区或使用更大的 T）。

类型 2：热点路由集中

集群可能按均匀 1000 万 RPS 设计，但某一路由—— /api/feed 面向信息流密集产品的 /api/checkout 面向限时抢购的 /v1/inference/recommend 面向热门新模型的——承担了 80% 的流量。增加机器没用；热点路由集中在固定后端池上。

修复方案：\n- 按路由的连接池 按流量份额而非均等调整大小\n- 按路由的熔断器 ——当信息流饱和时，以 503 丢弃信息流请求，保护结账（Envoy cluster.max_pending_requests) - 按路由的优先级 ——结账请求在队列中优先于信息流请求\n- 请求合并 ——当单个用户主导时（某名人主页每秒 500 万请求），识别小窗口内相同的并发请求，将一次后端响应扇出给多个客户端（Cloudflare、Varnish）

最深层的热点路由场景（名人热点、大促日过载）是负载均衡器层之上的架构问题，但负载均衡器负责执行流量降级和优先级控制。

§12. 使用场景画廊

公网流量（Cloudflare、Akamai、Fastly）

教科书级架构。来自 300+ PoP 的 Anycast VIP。L4（Cloudflare 的 Unimog、Akamai 的自定义转发器）在 XDP/eBPF 中以每台约 500 万 pps 进行包级转发，使用 Maglev 哈希。L7（Cloudflare 的 NGINX 变体、Akamai 的 Apache Traffic Server）执行 TLS 1.3 终止、HTTP/3（QUIC）、WAF、限流、CDN 缓存查询。最难的问题是 DDoS——Cloudflare 通过在 XDP 层丢包，在流量到达 L7 之前缓解多 Tbps 攻击。

内部微服务（服务网格中的 Envoy）

在 Lyft、Stripe、Airbnb，每个 Pod 都有 Envoy sidecar，负责处理出站请求，提供 mTLS、重试、熔断、可观测性（每次调用的链路追踪 + 指标）以及到 N 个上游 Pod 的连接池。集群成员通过 xDS 从控制平面（Istio Pilot、自研）推送。最难的问题是扇出——一个用户请求触及 50 个服务，因此每跳 1% 的故障率会产生 40% 的端到端故障率；重试和对冲请求至关重要。

数据库连接池化（PgBouncer、ProxySQL、vtgate）

Postgres 连接代价高昂（每连接 fork，约 10 MB RSS，建立约需 1 ms）。直接拓扑下 1 万个 Pod × 每 Pod 10 个连接 = 10 万个连接，会压垮数据库。PgBouncer 通过事务级复用将约 100 个后端连接共享给 10 万个客户端连接。ProxySQL/vtgate 更进一步——解析 SQL，将读请求路由至副本，写请求路由至主节点，通过会话固定支持读己之写，将分片查询路由至正确的 keyspace。

API 网关路由（Kong、AWS API Gateway、Apigee）

外部开发者访问统一入口；网关鉴权（OAuth2、API key、JWT），执行按客户的配额，应用请求转换（REST→gRPC、版本映射），路由至对应微服务。由于策略层的存在，延迟预算（5–50 ms）高于纯负载均衡器。具备丰富插件架构、针对策略执行优化的 L7 负载均衡器。

视频流路由（Netflix Open Connect、YouTube CDN）

Open Connect 设备（OCA）在 ISP 网络内运行基于 NGINX 的服务器。客户端请求视频片段时，经过 Netflix 的控制平面选择最佳 OCA——同 ISP 优先，否则选最近且有容量的。负载均衡分为两层：全局服务决定哪台 OCA 承载该用户的会话，再通过 DNS 或 HTTP 重定向将客户端发往该 OCA。OCA 每台支持 10 万+ 并发 HTTPS 连接，各自多 Gbps。

实时游戏（低延迟 UDP 负载均衡）

多人对局中 64 个玩家每 20–50 ms 向游戏状态权威节点发送 UDP 数据包。负载均衡器必须在对局期间将 64 个玩家的所有 UDP 流路由至同一后端（权威游戏服务器）。UDP 无连接；"粘性"是（src_ip, src_port）→ game_id 的映射，持续整个对局。难点：亚 50 ms p99 负载均衡跳转延迟，粘性故障转移（若服务器宕机，新服务器必须继承游戏状态——通常超出负载均衡器职责范围）。使用以 game_id 而非 5 元组为键的自定义会话表的 L4 UDP 负载均衡器。

数据库读副本路由

一个主节点 + 10 个读副本。负载均衡器（通常内嵌于 DB 驱动或 ProxySQL）将读请求路由至负载最低的健康副本，写请求路由至主节点。副本延迟是持续性问题——写入后立即读取可能命中尚未同步的副本。解决方案：对写后读在 N 秒内固定至主节点，或通过 LSN（Log Sequence Number）追踪，客户端请求最低 LSN，负载均衡器选择已追上的副本。具备副本延迟感知的协议感知 L7 负载均衡器。

§13. 真实世界实现与数据

  • Google Maglev （NSDI 2016）。分布式软件 L4 负载均衡器。DPDK 下每台约 100 万 pps；集群 >1 Tbps，服务于 Search、YouTube、GCP。引入了查找表一致性哈希算法，现代 L4 的实现几乎都采用此算法。
  • Facebook Katran （2018 年开源）。XDP/eBPF L4 负载均衡器。每台服务器约 500 万 pps。服务于 Meta 所有业务。在 eBPF maps 中实现 Maglev 风格的一致性哈希。
  • Cloudflare Unimog （约 2020 年）。基于 XDP 的 L4 负载均衡器。每天 >10 亿请求，峰值约 6000 万 RPS，缓解 >100 Tbps DDoS。
  • AWS Hyperplane （NLB 数据平面）。具有共享内存流表的计算节点集群。每个 NLB 数百万 RPS，每个 AZ 超 2000 万并发连接，新增延迟低于 100 µs。
  • AWS ALB。托管 L7 负载均衡器。按 LCU（Load Balancer Capacity Units）扩容。持续约 2.5–10 万 RPS/ALB。
  • Netflix Open Connect 上的 NGINX。每台 OCA 设备运行 NGINX 作为 HTTPS 终止器 + HTTP 字节范围响应器。每台设备支持数十万并发连接，各自多 Gbps。
  • GitHub 的 HAProxy。服务于 github.com，约 100 万+ RPS，基于 Lua 的路由支持 Git over HTTPS、Git over SSH 代理、API、Web UI。
  • Stack Overflow 的 HAProxy （2016 年）。持续约 3 万 RPS，峰值 7 万，约 6 台 HAProxy 服务整个站点。
  • Lyft 的 Envoy。2016 年设计并开源，兼作边缘和 sidecar。峰值每代理约 25 万 RPS。xDS 动态配置协议源于此。
  • Stripe 的 Envoy。全量 mTLS，复杂的重试 + 幂等键处理。全集群数百万 RPS。
  • GLB（GitHub Load Balancer） （2018 年开源）。基于 IPVS 的 L4 负载均衡器，通过"director"层将流一致性哈希至负载均衡器实例，解决 ECMP 重哈希导致的后端故障问题。
  • Heroku、Instagram（早期）的 PgBouncer。将数千个客户端连接复用至数十个后端连接——降低 100 倍。
  • YouTube、Slack 的 Vitess vtgate。MySQL L7 负载均衡器，负责分片查询、隐藏分片拓扑。YouTube 数百万 QPS。
  • Microsoft、Buoyant 客户的 Linkerd。基于 Rust 的 sidecar 代理，作为 Envoy 的服务网格替代方案。

差距为 约 1 万 RPS 到约 6000 万 RPS ，容量范围跨越 5–6 个数量级，不同层级有截然不同的架构选择。

§14. L7 路由规则深度解析

L4 按 5 元组路由；L7 按请求中的一切内容路由。L7 路由语言是区分包转发器与可编程应用网关的关键。Envoy 的 HTTP 路由配置（RouteConfiguration → VirtualHost → Route）是行业规范模型，大多数系统现已效仿（Istio VirtualService、Gloo、AWS App Mesh、Contour HTTPProxy、Linkerd ServiceProfile）。逐维度解析：

基于 Host 的路由。 第一层切分。 Host: api.example.com 落在某个 VirtualHost（VH）上， Host: admin.example.com 落在另一个上。通配符： *.example.com 匹配子域名；最长匹配优先（api.foo.example.com 优于 *.example.com）。在 Cloudflare 或任何多租户边缘，基于 Host 的路由可扩展至数百万 VirtualHost；路由数据结构是对反转 FQDN（Fully Qualified Domain Name）的基数前缀树，因此最长后缀匹配为 O(标签数)。一个 IP + 一个证书（通过 SNI——Server Name Indication，见 §16）+ 一个 Envoy 集群即可服务整个租户群。

基于 Path 的路由。 在 VirtualHost 内，按 URL（Uniform Resource Locator）路径匹配。三种模式：精确（/health），前缀（/api/v1/），正则（^/users/(\d+)$）。性能排序：精确（哈希查找，约 10 ns）> 前缀（前缀树遍历，约 100 ns）> 正则（RE2——Google 的安全正则引擎——约 µs）。顺序在 Envoy 中很重要——路由按从上到下首次匹配求值；将通配捕获规则 / 置于 /api/v1/users 特定路由之前会导致该路由被黑洞。常见模式：顶层按服务前缀设置通配路由 → 服务内部路由进一步细化。

基于 Header 的路由（金丝雀原语）。 匹配任意请求头。经典金丝雀场景： x-canary: true 路由至新版本集群，其余路由至稳定版本。结合在入口处注入头（移动端应用为内部员工通过用户 ID 查询发送 x-canary: true ），可获得 仅员工可见的金丝雀 ，再向公众开放。同一原语还用于 暗发布 （只读金丝雀，不影响用户状态）、 A/B 测试 （头由上游实验分配服务设置）以及 流量回放 （头由将生产流量镜像至预发布的回放工具添加）。

基于权重的路由（灰度发布）。 在单条路由匹配内，按加权集群拆分流量：99% 到 v1，1% 到 v2 → 95/5 → 50/50 → 1/99 → 0/100，历时数小时或数天。权重是控制平面的旋钮（xDS 推送或 Istio VirtualService 更新）；Envoy 在毫秒内收敛。概率拆分是按请求无状态的；若用户必须始终看到同一变体，则结合 hash policy （例如， hash on cookie userId, modulo 100, compare to weight）使选择具有粘性。这是渐进式发布的基石—— kubectl apply 在 Flagger 或 Argo Rollouts CRD（自定义资源定义）上根据来自指标提供方（Prometheus / Datadog）观测到的错误率和延迟自动调整权重。

基于查询参数的路由。 匹配 ?version=v2 或 ?region=eu。在边缘层比基于 Header 的路由少见（URL 参数对用户可见且可缓存，这对于金丝雀来说可能产生问题），但在内部 API 网关中被大量使用，因为查询字符串是合约的一部分。

镜像 / 流量影子（生产环境测试原语）。 将 N% 的真实流量发送到辅助集群， 忽略响应。客户端只看到主集群的响应，因此用户可见的行为保持不变。影子集群（通常是新版本）接收真实生产流量并生成自己的日志/指标。差异分析对比主集群与影子集群的延迟、错误率和响应体（排除不确定性因素）。Envoy request_mirror_policies 原生支持此功能——发后即忘，响应被丢弃。注意事项：副作用会执行两次（不要对真实数据库镜像写操作；影子集群需要沙箱数据存储或只读模式），且影子集群的负载会使上游依赖（缓存、搜索）的压力翻倍，除非它们也被沙箱隔离。

可组合性。 所有这些维度可以叠加使用。一条真实的 Envoy 路由可能会说："如果 Host 是 api.example.com，且路径匹配 /v2/feed，且 Header x-experiment-bucket 在 {7..9} 范围内，则 95% 流量路由到稳定版，5% 路由到金丝雀版，并将 10% 的流量镜像到集成测试集群。" Envoy 将整棵树编译为路由匹配 DAG（有向无环图），以每核约 100k RPS 的速度进行逐请求评估。服务网格控制平面（Istio Pilot、Linkerd 的 destination controller）负责高层意图（金丝雀流量 5%），并将其转换为推送到数据平面的 xDS RouteConfiguration 配置块。

§15. 健康检查策略

负载均衡器对后端健康状态的视图驱动所有路由决策；因此，健康检查的准确性对站点可用性至关重要。共有三类方法，以及几种传输协议选择。

主动健康检查。 负载均衡器按计划轮询每个后端（/healthz 每 2–5 秒轮询一次是典型配置）。通过 → 标记为健康并可路由；失败 → 超过阈值后驱逐（连续 3 次失败是 Envoy 的默认值，针对 unhealthy_threshold）；驱逐后通过 → 经过恢复阈值后重新加入。优点：能快速检测完全宕机的节点（停止响应 TCP 的节点），与真实流量无关，均匀探测整个集群。缺点：在大规模集群中，这会演变为 健康检查风暴 ——1000 个后端 × 每个负载均衡器每 10 秒检查一次 = 每个后端每秒 100 次探测；在 1000 负载均衡器 / 1000 后端的网格中，仅存活检查就产生每秒 10 万次探测。缓解措施：采样（每个负载均衡器检查随机子集的后端），通过控制平面在负载均衡器间共享结果，或使用 懒惰探测 ——仅当后端在 N 秒内没有收到真实流量时才进行探测。

被动健康检查（异常检测）。 负载均衡器监控真实流量，驱逐错误率或延迟超过阈值的后端——Envoy 的 outlier_detection 驱逐连续返回 5xx 的后端（ consecutive_5xx 次请求），并可选启用基于成功率的驱逐（驱逐低于集群均值 3σ 的后端）。优点：无额外探测流量，观测真实工作负载组合，检测"退化"后端（仍响应 /healthz 但在真实请求上缓慢）。缺点：需要实际流量才能检测故障（零流量的后端看起来永远健康），且可能被短暂的下游故障欺骗（数据库抖动 → 所有后端看起来都很糟 → 驱逐风暴 → 站点崩溃）。

混合模式（主动 + 被动）。 生产环境默认配置。主动探测检测 完全损坏 的后端（TCP RST、/healthz 返回 500、进程消失）。被动驱逐在故障发生与下一次主动探测之间的短暂窗口内检测 新近损坏 的后端——也能检测 轻微损坏 的后端，这类后端能通过 /healthz 但在真实请求上失败。Envoy 和现代 L7 负载均衡器同时运行两者：主动模式确定 候选池；被动模式根据观测到的行为从该池中剔除后端。

HTTP 健康检查。 功能最丰富的变体。负载均衡器发送一个请求（通常是 GET /healthz），期望返回 2xx 或 3xx 状态；可以要求特定 Header（例如， x-app-version: v3.2.1 用于检测部署错误的节点），可以匹配响应体中的子字符串。代价：每个后端每个检查间隔消耗一次 HTTP 请求。最适合 L7 负载均衡器检查 L7 后端。

TCP 健康检查。 仅建立 TCP 连接，发送可选字节，期待可选响应，然后关闭。比 HTTP 更轻量，但粒度更粗——仅验证进程接受 TCP，无法判断其后的应用是否健康。适用于 L4 负载均衡器，或作为 HTTP 检查前的基础验证。

gRPC 健康检查。 标准化 gRPC 服务（grpc.health.v1.Health/Check）返回 SERVING / NOT_SERVING / SERVICE_UNKNOWN。流式变体（Watch）通过维持长连接流来避免逐次探测的握手开销。天然适用于 gRPC 服务；Envoy grpc_health_check 配置直接支持。

"/healthz 返回 200 但服务已损坏"的灾难。 经典故障模式。/healthz 在进程存活时返回 200——它 非 不检查数据库连接池，不执行查询，不验证 Auth Token 缓存是否已填充。进程存活，真实请求以 500 失败。缓解措施：

  • 深度健康检查 ，触及下游依赖项。 /health/deep 执行对主数据库的单行 SELECT，ping Redis，检查 Auth 证书是否过期。运行频率宜低（每 30 秒而非每 2 秒），以避免放大下游依赖的负载。
  • 合成检查 ，执行真实的面向用户的事务（登录 + 读取 Feed + 退出），要求端到端成功。通过负载均衡器外部的独立探测器（Pingdom、Datadog Synthetics、自研）运行，结果反馈给部署门禁和告警，不直接影响负载均衡器路由。
  • 存活性与就绪性的分离。 存活性（= "进程是否存活？"）驱动 重启 决策（Kubernetes kubelet 在存活性检查失败时重启 Pod）。就绪性（= "进程是否准备好提供服务？"）驱动 路由 决策（负载均衡器仅向就绪的 Pod 发送流量）。混淆两者会导致退化的后端不断被重启，而非仅仅暂时被摘除。
  • 恐慌模式（§7 中的问题 4）。 当超过 50% 的池处于不健康状态时，无论如何都向所有人路由——下游共享故障会使 /healthz 看起来像是整个集群范围的宕机，此时尝试总好过保证黑洞。

健康检查的准确性从根本上是困难的，因为"该后端是否健康？"是由应用定义的，而负载均衡器是通用的。在生产环境中，诚实的答案是："没有任何单一检查是足够的；需要分层使用。"

§16. TLS 终止深度剖析

TLS 终止是密码学与流量工程交汇的地方。这些选择远非中立。

终止位置。 三种选项。

  • 边缘负载均衡器终止。 最外层的负载均衡器（PoP 处的 L7）终止 TLS，解密，然后将明文（或使用内部证书重新加密）路由到后端。优点：TLS 工作集中在少数具有硬件加速（AES-NI、QAT——Intel QuickAssist Technology、专用卡）的机器上；后端使用纯 HTTP；可观测性/WAF/检查均在明文上运行。缺点：数据中心内的明文对受监管的工作负载是合规问题；内部泄露会读取裸流量。

  • L7 负载均衡器终止 + 重新加密。 边缘 L4 转发到 L7 负载均衡器，后者终止 TLS，执行 L7 策略，然后使用内部 mTLS 重新加密后转发到后端。优点：WAF/金丝雀等功能仍然有效；后端仍通过 mTLS 进行身份验证。缺点：TLS 工作翻倍——在边缘一次，在 L7 到后端一次。通常可以接受，因为内部证书使用 ECDSA 且会话恢复率接近 100%。

  • 透传（TLS 桥接或 TLS 直通到后端）。 L4 负载均衡器转发加密字节；只有后端拥有证书。优点：端到端加密，合规性最简单。缺点：L7 负载均衡器无法按路径/Header 检查或路由（字节已加密）；无法在负载均衡器处部署 WAF；SNI 是唯一可用的信号，因此路由决策退化为每个证书一条路由。用于特殊场景：不需要 L7 功能的重度 WebSocket 流量，或严格的监管要求。

主流模式是在最靠近公网的负载均衡器处终止，从那里向内部使用 mTLS 重新加密。

SNI（服务器名称指示）。 没有 SNI 时，一个 IP 只能托管一个证书（证书在 TLS 握手期间、请求之前发送，因此服务器不知道客户端想要访问哪个域）。SNI 是一个 TLS 扩展，客户端在其中发送目标主机名 在 ClientHello 中，以明文形式。服务器从 SNI 映射表中选择匹配的证书。这是 anycast 多租户成为可能的原因——Cloudflare 边缘的一个 VIP 通过 SNI 查找为数百万租户的证书提供服务。大规模的 SNI 映射是以主机名为键的哈希表；证书对象（私钥、证书链、OCSP staple）缓存在内存中并独立轮换。明文 SNI 也是一种信息泄漏——中间设备（ISP、政府）即使在 HTTPS 连接上也能看到你正在访问哪个域。 加密 ClientHello（ECH） 是 2024 年后的修复方案：ClientHello 被加密到 DNS 中发布的公钥，因此 SNI 不再可见。Cloudflare 和 Google 在支持的客户端上启用 ECH；推广进展缓慢。

ALPN（应用层协议协商）。 另一个 TLS 扩展。客户端在 ClientHello 中广播支持的协议（h2, http/1.1, h3）；服务器选择一个并在 ServerHello 中包含该选择。这就是单个 TLS 端点如何在同一端口（443）上同时服务 HTTP/1.1、HTTP/2 和 HTTP/3（使用独立传输——QUIC 用于 h3）。负载均衡器按连接协商 ALPN；如果客户端支持 h2，负载均衡器会升级，将多个请求多路复用到一个 TCP 连接上，大幅减少握手摊销开销。ALPN 还在大规模场景下协商非 HTTP 协议——gRPC（在某些技术栈中基于 HTTP/2，使用 grpc-exp ALPN 标识符）、WebSocket（通过 HTTP Upgrade 在握手后协商而非使用 ALPN，但 ALPN 感知）。

TLS 1.3 中的 0-RTT。 TLS 1.3 允许回访客户端 在握手完成之前的第一个往返中发送应用数据，使用来自先前会话派生的 PSK（预共享密钥）加密。延迟收益显著：在 100ms RTT 蜂窝链路上的移动客户端，在第一次请求之后的每次请求上节省 100–200 ms（无握手往返）。代价：0-RTT 数据是 可重放的 ——捕获了 0-RTT 数据包的攻击者可以稍后重放，服务器无法辨别。缓解措施：仅对 幂等 请求允许 0-RTT（GET、OPTIONS），在负载均衡器处拒绝 POST 的 0-RTT 字节（Envoy 的 early_data_policy: HTTP1_SAFE_CRLF 或等效配置），并强制进行完整握手。

负载下的证书轮换。 证书会过期（Let's Encrypt：90 天；商业证书：1 年），且轮换过程不得丢弃流量。流程：新证书并行配置（ACME——自动证书管理环境——协议与 CA 如 Let's Encrypt 或内部 CA 通信，完成挑战，签发证书）。负载均衡器将新证书与旧证书一起加载到 SNI 映射表中。平滑切换将新握手的默认证书切换到新证书。通过 PSK 恢复的旧 TLS 会话继续有效，直到其票据过期。经过缓冲期（24–48 小时）后，旧证书被删除。在整个过程中， 没有连接被断开。Envoy 通过控制平面推送从 SDS（Secret Discovery Service）端点重新加载证书配置，原子性地交换每个监听器的证书引用。在 Cloudflare 规模下（数百万客户证书），生产轮换频率为每天到每周；负载均衡器层零停机。

HSM 与 KMS 管理的密钥方案。 TLS 需要私钥。私钥存储位置至关重要。

  • 进程内文件。 简单快速——私钥在负载均衡器的内存中；签名操作在用户空间进行。风险：内存泄露（Heartbleed、进程转储）会泄漏密钥，攻击者可以在轮换之前无限期冒充该站点。
  • HSM（硬件安全模块）。 物理或云虚拟设备将私钥保存在防篡改硅片内。负载均衡器向 HSM 发送"签名此哈希"请求；HSM 签名并返回签名；负载均衡器从不接触密钥。优点：即使完全内存泄露也不会泄漏密钥。缺点：签名延迟从微秒级增加到毫秒级（HSM 往返），吞吐量受限于 HSM 签名速率（单个 HSM 每秒数千次签名）。用于证书颁发机构、金融机构、受监管的工作负载。常见部署：HSM 在独立 VLAN 中，通过 PKCS#11 访问。
  • KMS（密钥管理服务）。 云端等效方案——AWS KMS、GCP Cloud KMS、Azure Key Vault。密钥永不离开服务；签名操作是 HTTP API 调用。延迟与 HSM 类似（每次签名毫秒级），但负载均衡器还可受益于 KMS 驱动的轮换（KMS 轮换底层密钥；客户端只看到密钥 ID）。AWS ALB 内部使用——ALB 私钥存储在 AWS KMS 中。
  • 无密钥 TLS（Cloudflare 的模式）。 Cloudflare 在其边缘为拒绝共享私钥的客户（银行、政府）终止 TLS。Cloudflare 的边缘运行 TLS 握手，但通过自定义协议将每次签名操作转发到客户的"密钥服务器"；客户的 HSM 执行签名；Cloudflare 的边缘完成握手。每次握手增加一个客户侧 RTT，但 Cloudflare 从不持有密钥。在高度受监管的租户中已投入生产。

对于 99% 的服务，正确答案是将私钥存储在负载均衡器节点的内存中，通过文件系统权限加以保护，并使用 KMS 作为 静态加密密钥 在启动时解密证书文件。仅在合规性或威胁模型要求时才使用 HSM。

§17. 服务网格集成

服务网格是这一认知的具体实现："如果 Envoy 是万物皆可用的负载均衡原语，就在每个 Pod 旁边放一个。"这使负载均衡器无处不在，而控制平面成为最重要的组件。

Envoy 作为数据平面。 每个 Pod 都有一个 Envoy Sidecar ——在同一 Pod 网络命名空间中的第二个容器，通过 iptables 或 eBPF（Cilium 的方式）拦截所有入站和出站流量。出站：应用调用 http://feed-service；iptables 将请求重定向到本地 Envoy 的 127.0.0.1:15001；Envoy 在其集群表（由 xDS 填充）中查找 feed-service ，通过 Maglev 或 P2C 选择健康的端点，建立 mTLS 连接并发送。入站：到 Pod IP 的流量被 Envoy 在 15006 端口拦截，终止 mTLS，应用入站策略（速率限制、ACL——访问控制列表），然后转发到应用的回环接口。每个 Envoy 都是 该 Pod 交互的负载均衡器；网格是所有 Envoy 决策的集合。

Istio / Linkerd 作为控制平面。 没有配置，Envoy 是聋的。控制平面负责将意图（"将 5% 的流量路由到 feed-service 的 v2"）转换为每个 Envoy 所需的集群/路由配置。组件：发现服务（监控 Kubernetes 的 Service/Endpoint 变化）、配置转换层（CRD → xDS 资源）、证书颁发机构（为每个工作负载签发短期 mTLS 证书；轮换频率通常为 24 小时）。Linkerd 的控制平面基于 Rust，简约精炼；Istio 的控制平面更大（Pilot、Citadel、Galley 等），但功能更丰富。

xDS——可扩展发现服务。 控制平面与 Envoy 之间的传输协议。一个 gRPC 双向流，控制平面推送资源更新；Envoy 进行 ACK/NACK。xDS 命名空间分解如下：

  • LDS（监听器发现服务）。 Envoy 监听的端口/地址，以及处理每个端口的过滤器链。
  • RDS（路由发现服务）。 对于每个 HTTP 监听器，对应的 RouteConfiguration（Host/路径/Header 规则——参见 §14）。
  • CDS（集群发现服务）。 存在哪些逻辑上游集群（例如，"feed-service-v2"）及其负载均衡配置（P2C、Maglev、权重）。
  • EDS（端点发现服务）。 每个集群对应的具体端点（Pod IP 和端口）成员列表。EDS 更新最频繁——每次 Pod 创建/销毁都会触发 EDS 推送。
  • SDS（Secret 发现服务）。 每个工作负载对应的 mTLS 证书/密钥/CA 包。SDS 更新驱动证书轮换。
  • ADS（聚合发现服务）。 单一流，多路复用上述所有内容——优先采用，因为它保留了因果顺序（集群在其端点之前出现）。

在大规模场景下，xDS 推送速率主导了控制平面的开销。一个拥有 100 个服务、5000 个 Pod 的网格，在稳定的变动下每分钟产生数十次 EDS 推送，再乘以订阅的 Envoy 数量。优化措施： 增量 xDS （只推送增量，而非完整快照）， 作用域 xDS （每个 Envoy 仅订阅其关心的集群）， 联邦控制平面 （每个地区一个，分别管理本地集群）。

"L7 代理附着于每个 Pod"的图景。 这在概念上是每次调用进行负载均衡，而非每条边——两个服务之间的每次调用都要经过两端的 Envoy，获得 mTLS、重试、链路追踪、速率限制、熔断，而应用本身无需做任何事。代价是每次调用两次 Envoy 跳转（每次增加约 1ms p50、约 3–5ms p99）。对于 50 个服务的扇出请求，这意味着 100 次跳转；网格开销是真实存在的（通常占端到端延迟的 10–30%）。收益是完全统一：每次调用遵守相同规则，可观测性端到端，mTLS 是基准。

Ambient 网格（后 Sidecar 演进）。 Istio Ambient（2022 年起）用两个共享组件取代了每个 Pod 的 Sidecar：

  • 一个 节点级 ztunnel （零信任隧道）——每个节点一个，处理该节点上所有 Pod 的 mTLS/身份工作。使用 Rust 实现，通过 eBPF 将 Pod 流量重定向到 ztunnel。此处没有 HTTP 解析——这是一个 L4 隧道。
  • 一个 waypoint 代理 ——每命名空间或每身份的 Envoy，处理 L7 策略（来自 §14 的路由规则）。需要 L7 功能的请求经过 waypoint；纯 L4 mTLS 流量绕过它。

收益：每个 Pod 中没有 Envoy（在 10000 Pod 的集群中每个 Pod 节省约 50–100 MB RSS，总量可观），Pod 启动更快（无 Sidecar 初始化），升级更简单（升级一次 waypoint，而非 10000 个 Sidecar）。权衡：每次调用的隔离性降低（处理节点上所有 Pod 的节点级 ztunnel 是一个共享故障域），L7 功能需要更多网络跳转（Pod → ztunnel → waypoint → ztunnel → Pod）。Linkerd 也有类似演进；Cilium Service Mesh 完全跳过 Sidecar，使用内核中的 eBPF（参见 §25）。

网格设计的演进轨迹是"负载均衡器无处不在 → 负载均衡器在处处都太昂贵 → 仅在能带来价值的地方保留负载均衡器"。Ambient 和 eBPF 网格仍然是负载均衡技术，但重塑了代理的位置。

§18. 地理路由与 Anycast 深度剖析

负载均衡器层在全球规模下的首要任务是"将客户端引导到附近的 PoP"。三种机制在生产中使用；大型 CDN 同时使用全部三种。

GeoDNS 与 ECS（EDNS 客户端子网）。 DNS 解析器传统上向权威 DNS 服务器隐藏客户端 IP——权威服务器只能看到解析器的 IP，而该 IP 可能位于不同国家（Google Public DNS 8.8.8.8 在全球范围内服务用户）。GeoDNS 按 IP 路由；没有客户端 IP，它按解析器位置路由，往往不准确。 ECS （EDNS 客户端子网，RFC 7871）允许解析器在 DNS 查询中作为 EDNS 选项传递客户端 IP 的截断形式（例如，/24 前缀）。权威服务器使用该前缀在数据库（MaxMind GeoLite、来自 RUM 的内部 IP 到地区映射——见下文）中查找地理位置，并返回最近 PoP 的 IP。ECS 对每个解析器和每个区域是可选的（隐私顾虑：它泄露的客户端信息多于仅使用解析器 IP）。主要 DNS 提供商（Google、Cloudflare 1.1.1.1 实际上默认禁用 ECS，OpenDNS）对 ECS 的支持程度不同。GeoDNS 的准确性上限取决于 IP 到地区数据库的准确性；移动运营商的 IP 段经常地理位置错误（德克萨斯州的 Verizon IP 段服务加利福尼亚州的用户等）。

BGP Anycast。 跳过 DNS 作为路由层；让网络本身路由。一个 IP（例如，1.1.1.1）通过 BGP 从多个 PoP 同时宣告。互联网的 BGP 机制基于 AS 路径长度、本地偏好和其他 BGP 属性，为每个源 AS（自治系统）选择一个 PoP——通常是拓扑上最近的，与地理上最近的高度相关。优点：无需操心 DNS；故障转移是自动的（PoP 撤回 BGP 宣告，流量在数秒到数分钟内转移）；DDoS 攻击面在所有 PoP 间 稀释 （10 Gbps 攻击分散到 50 个 PoP，每个仅 200 Mbps，可管理）。缺点：BGP 按 每路由而非每流选择，且 BGP 路由可能在流的中途发生变化（来自上游对等体的路由更新）。UDP/QUIC 的 Anycast 很简单；有状态 TCP 的 Anycast 则很困难。

基于 RUM 的路由。 真实用户监控数据——来自真实用户的实际延迟测量，通过 JavaScript 信标或应用 SDK 收集——反馈到 DNS 或 BGP 层。Cloudflare 和 Akamai 使用 RUM 来学习"运营商 X 的用户一致地从 PoP B 获得比 PoP A 更好的延迟，尽管 PoP A 看起来更近。"决策变为"对于来自该运营商 IP 段的查询，返回 PoP B。"比单独使用 IP 到地区数据库更准确，尤其适用于移动网络和新兴市场。反馈循环：RUM 数据每隔几小时更新一次；持续性变化（新海底电缆、对等关系变化）更新地图；短暂异常被平滑处理。

"Anycast 孤岛"病理。 客户端到 Anycast IP 的 BGP 路由在流的中途发生变化（例如，对等链路断开，上游路由器重新运行最优路径选择，选择了不同的下一跳，流量现在落到了不同的 Anycast PoP）。新 PoP 没有该流的连接状态——其内核发送 TCP RST。连接明显中断。对于长连接（HTTP/2、gRPC、WebSocket、大文件下载），这是不可接受的。缓解措施：

  • 流感知 Anycast 引导。 当路由发生变化时，新 PoP 识别出"我没有该流的状态"，将数据包通过 内部隧道转发到旧 PoP （旧 PoP 仍有状态），而不是发送 RST。旧 PoP 继续服务该流直至其结束；来自同一客户端的新流落到新 PoP。LinkedIn/Cloudflare 的 Unimog 架构内置了这一机制：每台 Anycast 机器上的"流引导"组件维护一个流到机器的映射表，机器之间转发数据包以在路由变动下保持状态。
  • 基于连接 ID 的路由（QUIC）。 QUIC 的连接 ID 独立于五元组。新 PoP 的负载均衡器可以读取连接 ID，在集群范围的表中查找，并转发到实际拥有该连接的机器。这是 QUIC 用于 Anycast 的杀手级特性（NAT——网络地址转换——重绑定也受益；移动 IP 变化不会中断 QUIC 流）。
  • 主动迁移 在路由发生变化之前。实践中很困难——负载均衡器无法提前收到上游 BGP 变化的通知。

Cloudflare 的 Unimog 论文详细描述了流引导方法。通过引入集群内流转发，Anycast 的"任意"变成了"正确的那一个"。

§19. 连接排水

部署是负载均衡器必须掩盖的最频繁的故障模式。连接排水是其协议。

优雅关闭流程。 当后端（或负载均衡器本身）因部署而即将下线时：

  1. 排水信号。 编排器发送"preStop" Hook 或等效信号（Kubernetes 的 terminationGracePeriodSeconds 窗口开始；Envoy 接收 /healthcheck/fail admin 端点；HAProxy 接收 SIGUSR1).
  2. 停止接受新流量。 后端将自己标记为不健康（其 /healthz 开始返回 503），和/或其在负载均衡器池中的注册被移除。负载均衡器在下一次主动探测时（或如果负载均衡器有控制平面推送通道，则立即）注意到这一情况，并停止发送新连接。
  3. 已有连接完成。 进行中的请求自然完成——响应被发送，HTTP/1.1 的 Keep-Alive 空闲超时触发，HTTP/2 流关闭，gRPC RPC 完成。
  4. 排水超时后强制关闭。 如果在，比如 30 秒后，仍有请求挂起，后端在 HTTP/2 流上发送 GOAWAY （请求客户端优雅迁移），然后发送 SIGTERM。经过最终宽限窗口（Kubernetes 默认 30 秒）后，发送 SIGKILL。

做得好，部署 不可见 ——没有 5xx 抖动，没有延迟峰值。做得不好，每次部署都是一次小规模故障，团队学会了在低流量时段部署，而不是修复问题。

"30 秒不够用"的问题。 长轮询连接、服务器发送事件（SSE）、WebSocket、gRPC 流式 RPC、文件上传以及缓慢的后端依赖项都可能使连接的存活时间远超 30 秒。固定在 chat-service-pod-7 上的 WebSocket 可能存活数小时。在 30 秒后强制关闭意味着每次部署都会断开所有活跃的聊天会话。可选方案：

  • 更长的排水超时。 terminationGracePeriodSeconds: 600 （10 分钟）。代价是部署速度变慢（需要等待长连接完成），并且对于某些流量模式可能是无界的。
  • 向客户端发送排空信号。 发送 WebSocket 关闭帧或 HTTP/2 GOAWAY 与 last_stream_id 设置为低于当前值——告知客户端优雅地迁移到新服务器。客户端重新连接到不同的后端；旧后端排空速度更快。Slack 和 Discord 在聊天服务器上采用此方式。
  • 连接重平衡。 定期（与部署无关），LB 或后端发信号让客户端重新连接，自然地迁移流量，避免任何一个后端积累过多长连接。避免了部署时的连接积压。
  • 感知粘性会话的排空。 如果会话是用户绑定的并存储在状态中，则按 会话数而非连接年龄进行排空——让用户完成当前任务，但在下次重连时重定向。

另一种故障模式：一个没有排空期就收到 SIGTERM 的后端（即"紧急终止"）在请求进行中向 LB 返回 ECONNREFUSED。LB 看到 5xx，将该后端标记为不健康，并在另一个后端上重试——大多数情况下可以透明恢复。但在全量部署期间，对用户可见的代价是真实存在的：在 1000 个 Pod 上进行无排空的滚动更新，每分钟会产生数百个 5xx。排空是零影响部署与有感知影响部署之间的分水岭。

§20. 粘性会话

粘性会话将一个逻辑会话（用户、客户、游戏对局）在跨请求间固定到特定后端。这一技术比我们所知的负载均衡历史更悠久。

基于 Cookie 的粘性。 LB 在第一次请求时设置一个 Cookie—— Set-Cookie: lb-backend=B7; Max-Age=3600 ——在后续请求中读取该 Cookie 并直接转发到 B7，绕过负载均衡决策。Envoy 和 HAProxy 通过 cookie 哈希策略实现此功能。Cookie 可以是：

  • 不透明型 （LB 内部：映射到后端 ID 的哈希值；客户端看不到有意义的内容）。
  • 签名型 （对后端 ID 加有效期进行 HMAC 签名，防止篡改）。
  • 加密型 （使用仅 LB 持有的密钥进行完整加密，防止推断）。

基于 IP 的粘性。 LB 对客户端 IP 进行哈希并一致地路由到某个后端。无需 Cookie，适用于非 HTTP 场景，但在 NAT 后会失效（多个客户端共享一个 IP，被固定到同一后端；在 Wi-Fi 和蜂窝网络之间漫游的单个客户端会被重新固定）。带有哈希策略的 IP 粘性与 L4 LB 兼容；Cookie 粘性需要 L7。

为何大多数现代设计避免使用粘性。 粘性是 应用层状态泄漏到 LB 层。一旦 LB 知道"这个用户属于 B7"，应用就失去了以下自由：

  • 重启 B7 ——用户流量被固定在那里，不易迁移。
  • 扩容节点池 ——增加 B11 对用户体验没有任何作用，因为他们仍然固定在 B7 上。
  • 对 B7 进行故障转移 ——当 B7 故障时，用户会话必须显式迁移；LB 可以做到这一点，但应用程序必须支持会话重建。

现代模式恰恰相反： 无状态后端，状态存储在共享存储中。 会话、购物车、用户偏好、进行中的游戏状态——全部存储在 Redis、DynamoDB 或会话服务中。任何后端都可以处理任何请求。LB 做纯轮询或 P2C。扩缩容、重启、部署、故障转移都变得轻而易举。

粘性适用的场景。 以下是有效使用场景的简短列表：

  • WebSocket 应用。 TCP 连接即会话；LB 别无选择，必须在连接的生命周期内固定它。（注意：这是对 某个连接的粘性，而不是对 某个用户 的粘性——这是不同的问题；重连可能落到其他地方。）
  • 服务端推送事件（SSE）、长轮询、流式 HTTP。 与 WebSocket 相同——长连接 TCP 无法透明地迁移。
  • LB 层对延迟高度敏感的缓存。 如果 LB 始终将某用户路由到已有其 L1 缓存预热的后端，则可以更智能地处理。LB 中的状态代价很小，如果它纯粹是一个性能提示（即使提示错误，系统仍然正确）。
  • 有严格存活性要求的游戏对局。 一个 64 人 UDP 对局必须在对局期间将每位玩家路由到权威游戏服务器。LB 维护一张 (game_id → 后端) 表；玩家的 UDP 流全部哈希到同一个 game_id。
  • 具有大内存缓冲区的流媒体。 一个已将电影前 10 GB 分片加载到内存的视频会话可以从粘性中受益；重新在其他地方加载需要重新获取 10 GB 数据。

当粘性使用合理时，需 显式设计故障转移行为：当 B7 故障时，其粘性用户会怎样？应用程序必须将"您的会话已结束，请重连到新后端，从 Redis 恢复状态"视为正常事件来处理。

§21. LB 处的连接池

在超大规模场景下，即使对于 LB 到后端的路径，"每个请求一个 TCP 连接"也是浪费的。计算如下：100k RPS，每个请求 1ms，需要 100 个并发连接——可控。1M RPS 在 10ms 时需要 10k 并发连接；100M RPS 时则需要 1M 并发连接，远超每个源 IP 的 65k 端口限制（TCP 源端口为 16 位），并且会压垮连接建立成本（TLS 握手、mTLS 证书校验、socket 表抖动）。

后端连接池。 LB 为每个后端维护一个预先建立好的 TCP 连接池。新请求从池中获取一个空闲连接（O(1) 操作），发送请求后将连接放回。池大小按每个后端的峰值并发请求量来调整。NGINX 的 upstream { keepalive 100; } 配置每个上游保留 100 个空闲连接；Envoy 的 http_protocol_options { max_connections: } 类似。连接池缓慢排空（空闲超过 60s 的连接由 Envoy 或后端的 keep-alive 超时关闭），按需补充。

HTTP/2 到后端的多路复用。 HTTP/1.1 无法可靠地进行流水线处理（队头阻塞）；每个请求需要独立连接或在串行 keep-alive 链中占用独立槽位。HTTP/2 在 单个 TCP 连接上多路复用多个流。LB 对后端 B7 建立 2 个 HTTP/2 连接，并在其上传输 1000 个并发流。Socket 数量：2 个而非 1000 个。连接建立成本：摊销到数千个请求上。这是 LB 到后端最大的可扩展性提升；几乎所有现代代理都使用 HTTP/2 后端连接。

HTTP/3 (QUIC) 后端连接。 对某些模式更优。QUIC over UDP 避免了 流之间 的 TCP 队头阻塞（stream 7 丢包不会阻塞 stream 8），在 LB 到后端的 RTT 不可忽略（跨区域连接池）或存在丢包时非常有用。采用率落后于 HTTP/2——后端必须支持 h3，可观测性工具必须理解 UDP 流。

"LB 到后端 Pod 的 1M 个 Socket"耗尽问题。 在非常大的规模下，即使是连接池也会耗尽资源。每源 IP 的 TCP 源端口限制为 65k。如果 LB 有 1 个源 IP 和 1000 个后端，每个后端最多约 65 个源端口（考虑预留开销）。解决方案：

  • 在 LB 上增加更多源 IP 。分配 10 个 IP，每个后端可获得 650 个端口。NIC 和内核可以处理这个问题。
  • HTTP/2 多路复用， 如上所述——将 1000 个流折叠为 1 个 socket。
  • SO_REUSEPORT 用于出站连接。 在某些内核中，多个 socket 可以共享出站的源 IP/端口；比用于监听 socket 时少见。
  • 跨后端副本的连接池共享。 如果 LB 服务 N 个后端且每个后端只看到约 10 RPS，则每个后端开 1 个连接就足够了。共享是默认行为。

第二个相关的耗尽问题： 有状态防火墙上的 conntrack 表。 Linux 的 conntrack (NF_CONNTRACK) 为连接跟踪和 NAT 保存每个流的状态。默认表大小约为 256k；在高 LB 规模下，conntrack 填满，新连接被丢弃。缓解措施：禁用 LB 出站上的 conntrack（设置 notrack 规则）、使用 XDP 绕过，或扩大表大小。Cloudflare 和 Facebook 已记录了在数据平面路径上完全禁用 conntrack 来运行 LB 的案例。

§22. 自动扩缩容交互

LB 的指标——请求速率、延迟、错误率、连接数——是自动扩缩容的输入信号。做好了，这就形成了一个闭环，使容量自我管理。

LB 指标驱动 HPA（Horizontal Pod Autoscaler）。 Kubernetes HPA 读取指标（默认：CPU、内存；使用自定义指标时：任意指标）并上下调整 Pod 数量。从 LB 获取指标的方式：

  • 每个 Pod 的每秒请求数。 如果 request_rate > 200 RPS/pod则扩容。LB 在 Prometheus 中暴露每个集群的 RPS；适配器（Prometheus Adapter、KEDA——Kubernetes 事件驱动自动扩缩容）将其提供给 HPA。
  • p95 延迟。 如果 p95 > 100 ms则扩容——连接池正在饱和。
  • 每个 Pod 的活跃连接数。 绕过 RPS 变化；"平均 Pod 有多忙？"是一个更稳定的信号。
  • 错误率。 如果错误与饱和相关而非与 Bug 相关，则扩容。

复合扩缩容：HPA 综合多个指标，使用建议副本数最高的那个。避免围绕单个噪声指标产生震荡。

"扩容时的惊群效应"问题。 扩容添加了新 Pod。LB 看到它们，将其视为池中的全容量成员，立即将其应有的流量发送给它们。新 Pod 处于 冷启动状态：JIT 编译的代码路径尚未热身，到依赖项的连接池为空，OS 页面缓存未预热，ML 模型权重未分页加载。冷启动 p99 可能是预热 p99 的 5–10 倍。全速流量导入将冷 Pod 推入熔断器触发区域，HPA 再次看到饱和，继续扩容，级联效应由此展开。

预热期。 三个层次可以解决此问题。

  • LB 处的慢启动。 Envoy 的 slow_start_config 在可配置时间窗口内（默认 60s，可延长至 5 分钟以上）将新添加端点的集群权重从 0 逐渐提升至 100%。Pod 在第一分钟内只接收正常流量的一小部分；到第 5 分钟时达到完整权重。
  • 带渐进标准的就绪探针。 Kubernetes 的就绪探针仅在 Pod 完成预热（模型加载完毕、缓存已预热、依赖项已探测）后才返回"就绪"。LB 在 Pod 真正就绪之前不会感知到它。
  • 预热钩子。 部署流水线在将新 Pod 提升为线上流量之前，会针对新 Pod 重放合成流量剖面（录制的生产请求，1% 回放速率）。当 Pod 进入 LB 的连接池时，其 JIT 和缓存都已预热。

三者结合消除了冷启动悬崖。没有它，自动扩缩容就会变成反效果扩缩容——添加 Pod 反而使情况在变好之前先变得更糟。

预测性扩缩容。 响应式扩缩容（等待饱和后扩容）会因预热窗口而落后。预测性扩缩容（预测即将到来的负载并提前预热）则胜出。广泛用于黑色星期五、体育赛事流量激增、计划性部署。输入：一天中的时间模式、营销活动日程、流量季节性。LB 不直接驱动这一过程，但其指标会输入到预测器中。

§23. DDoS 防护层

LB 层是拒绝服务防御的第一道防线。攻击按层次分为两类。

第 3 / 4 层攻击（容量型）。 攻击者生成原始数据包流量——SYN 洪泛、UDP 放大（小型 DNS 查询，大型反射响应）、TCP 反射、ICMP 洪泛。在 LB 处理任何内容之前，就在 边缘节点 进行防御。

  • Anycast 稀释。 如 §18 所述，多 PoP Anycast 网络将攻击分散到 N 个 PoP 上。针对 Cloudflare IP 的 1 Tbps 攻击，在 200 个 PoP 的情况下，每个 PoP 约承受 ~5 Gbps——可处理。网络越大，稀释效果越好。
  • 网络层 ACL（访问控制列表）。 路由器 ACL 丢弃匹配已知恶意模式的流量（来自 bogon 范围的伪造源 IP、IPv4 分片、格式错误的数据包）。
  • L4 速率限制。 按源 IP、源 AS、ASN 区域。XDP/eBPF 程序在 BPF map 中为每个源 IP 维护滑动窗口计数器，丢弃超过阈值的数据包。判决在亚微秒级完成。
  • SYN cookies。 参见 §7 问题 7——在不使用流表内存的情况下处理 SYN 洪泛。
  • 上游 BGP 协调。 在多 Tbps 规模的攻击中，LB 运营商要求其上游中转提供商通过 BGP 社区（RTBH——远程触发黑洞）或 BGP FlowSpec 丢弃攻击者的前缀。Cloudflare 的 Magic Transit 会自动将其推送到客户的自主中转决策中。

第 7 层攻击（应用滥用）。 流量体积较小，但旨在耗尽应用资源。Slowloris（建立大量 TCP 连接，每次发送一字节头部，持续占用 socket）、HTTP 慢速 POST（类似，但针对请求体）、HEAD 洪泛、高成本查询模式（?sort=relevance&fuzzy=true 针对不缓存的搜索端点）、撞库攻击（使用泄露凭据进行登录尝试）。在 L7 LB 或上游 WAF 处进行防御。

  • WAF（Web 应用防火墙）。 基于规则和基于 ML 的过滤。已知模式（SQL 注入、XSS、路径遍历）直接拦截。异常请求模式（单个 IP 大量登录失败、异常用户代理字符串、带有 Bot 特征的请求）被拦截或进行挑战验证。ModSecurity、Cloudflare WAF、AWS WAF。
  • L7 速率限制。 按用户、API 密钥、端点、地理位置。每个键使用令牌桶或滑动窗口计数器。Envoy 的 local_rate_limit 过滤器用于实例级限制， rate_limit 过滤器调用外部速率限制服务（RLS——Rate Limit Service）实现全局限制。
  • Bot 管理。 JavaScript 挑战（强制客户端执行代码，过滤简单爬虫）、CAPTCHA（强制人工参与，有效但摩擦高）、TLS 指纹识别（JA3/JA4 哈希通过 TLS 握手识别客户端；Bot 与浏览器的指纹不同）、行为分析（鼠标移动、请求时序模式）。
  • 按源限制连接数。 限制单个 IP 的并发连接数以缓解 Slowloris——100 个连接/IP 是人类流量的合理默认值。

"Cloudflare 吸收 Tbps 级攻击"的计算。 Cloudflare 发布的事件报告显示可吸收超过 100 Tbps 的攻击。计算方式：约 300 个 PoP × 每个约 400 Gbps 容量 = 约 120 Tbps 总量。Anycast 均匀分散攻击（大致上），每个 PoP 承受约 400 Gbps，可以处理。攻击者无法将攻击集中在某一个 PoP 上，因为路由决策由互联网决定，而非他们。Cloudflare 发布的 2024 年 Q3 数据显示，每季度有多次持续超过 5 Tbps 的攻击，包括一次 5.6 Tbps 的 UDP 洪泛，处理后对客户零影响。

DDoS 防御栈是分层的洋葱结构：边缘 Anycast → L4 ACL → L4 速率限制 → L7 速率限制 → WAF → 应用层。越靠近边缘，丢弃代价越低；越深入，决策所需的语义上下文越丰富。现代防御是 自动化的 ——网络在数秒内（通常通过对 netflow 数据进行流量异常检测）检测到攻击模式，并通过 BGP、控制平面更新和 WAF 规则推送在整个集群中部署对策。

§24. 之前未涵盖的故障模式

§8 中的故障模式涵盖了 LB 节点故障、PoP 故障、控制平面故障和分区。还有三种值得介绍。

模式 F：LB 自身崩溃（需要通过 ECMP 实现冗余 LB）。 单个 L4 或 L7 LB 节点不是一个部署单元。部署单元是 至少两个 LB 位于 ECMP 后端。边缘路由器通过 ECMP 在 N 个 LB 节点上公告 VIP；流量哈希分散流量。当某个节点故障（进程崩溃、内核 panic、硬件故障）时，ECMP 通过 BFD（双向转发检测——亚秒级链路探测）或 BGP 会话断开检测到故障，移除故障下一跳，并在 N-1 个节点上重新哈希。命中故障节点的流量：仍在故障节点流表中的流已丢失（少量数据包丢弃）；重新哈希后，存活 LB 通过 Maglev → 同一后端接管 → 连接继续。没有冗余，LB 就是一个 SPOF（单点故障）；有了 ECMP 冗余，LB 故障在约 100ms 内即可透明恢复。

模式 G：LB 到后端分区（LB 认为后端健康，但后端无法访问 DB）。 网络分区将后端与关键下游（DB、缓存、消息中间件）隔离，同时保持 LB↔后端路径健康。LB 对后端的 /healthz 探测成功（后端 HTTP 服务器正常运行）。真实请求失败，因为后端无法访问 DB。LB 没有信号——其探测通过，被动剔除可能会捕获一些 5xx，但速度很慢。缓解措施：

  • 深度健康检查。 /healthz 包含真实的 DB ping（以及 Redis 和其他关键依赖项）。需要更多探测流量；但在检测准确性方面物有所值。对深度检查进行速率限制，以防止 LB 的探测在局部故障时压垮 DB。
  • 合成事务。 一个定时探测器（与 LB 分离）每隔几秒运行端到端用户事务；结果反馈到 LB 连接池排除逻辑中。即使 /healthz 是浅层的，也能检测到 DB 问题。
  • 后端主动上报不健康。 当后端检测到其 DB 连接超时时，主动在 /healthz 上返回 503。LB 在下次探测时剔除该后端。速度慢（一个探测间隔），但无需额外探测流量。
  • 自适应剔除。 LB 跟踪每个后端的滑动窗口错误率；如果 30 秒内对 B7 的请求有 50% 失败，则无论 /healthz 如何都进行剔除。Envoy 的 outlier_detection 通过 consecutive_5xx 和 success_rate_minimum_hosts 实现这一功能。

这是最微妙的 LB 故障模式，因为"从 LB 视角看健康"与"从用户视角看健康"是脱节的。

模式 H：健康检查风暴（1000 个后端 × 每个 LB 每 10s 检查一次 = 每个后端 100 次/秒）。 在集群规模下，朴素的主动健康检查会在后端产生相当大的负载。1000 个后端、200 个 LB（网格 Sidecar），每个 LB 每 10 秒检查每个后端 = 集群中每秒 20000 次检查，每个后端每秒 20 次检查。如果每次检查是 HTTP 且耗时约 1ms，那么仅健康检查就消耗约 20ms/sec 的后端 CPU。在每个 Envoy 都探测可能路由到的每个集群成员的网格规模下，负载翻倍或三倍。

缓解措施：

  • 采样。 每个 LB 检查后端的随机子集；通过控制平面合并信号。每个后端的覆盖率相同，总检查次数减半或缩减至四分之一。
  • 分层健康检查。 一个小型"健康聚合器"服务检查每个后端，通过 xDS 推送向所有 LB 暴露聚合健康状态。LB 信任聚合器，永不直接探测。聚合器是 SPOF，因此以副本方式部署。
  • 高端场景仅使用被动检查。 完全跳过主动检查；依赖真实流量 + 异常值检测。当流量足够大时有效，任何故障后端在数秒内都会暴露。
  • 长间隔加事件驱动。 每 30 秒进行一次主动检查（而非每 2 秒）用于粗粒度存活性检测；事件驱动排除（基于观察到的错误）用于快速检测。

教训：每次探测都是一次请求。在集群规模下，LB 的监控流量本身就是一个需要管理的工作负载。

§25. LB 的可观测性

LB 看到每一个请求——这使它既是最重要的可观测性视角，也是最嘈杂的视角。生产可观测性有三大支柱和一种调查模式。

请求日志（采样、结构化）。 通过 LB 的每个请求 都 可以记录一行（时间戳、源 IP、方法、路径、状态、延迟、上游后端、输入/输出字节数、TLS 密码套件、响应时间、重试次数）。在 100k RPS 时，完整日志记录是每秒 100k 行——每天产生 PB 级数据。生产环境方案： 结构化 + 采样.

  • 结构化 ——JSON 或 protobuf 行，通过日志流水线（Kafka → S3 → Athena/Trino）传输，字段名一致。可搜索；非自由文本。
  • 采样 ——均匀采样 1% + 100% 的错误 + 100% 的慢请求（>p99）。在不为中位数付费的情况下捕获尾部。
  • Trace 关联 ——每条日志行都有一个 Trace ID（例如 W3C traceparent），因此可以将 LB 日志与后端 Trace、DB Trace 和下游关联起来。

四个黄金信号。 SRE 经典——每个 LB 都暴露这些指标：

  • Rate（请求速率） ——每秒请求数。脉搏。突然下降 = 上游故障或客户端消失；突然激增 = 流量涌入或攻击。
  • Errors（错误） ——5xx 速率、4xx 速率。区分：5xx = 我们的问题，4xx = 客户端问题（大多数情况下）。关注趋势；绝对数量因流量而异。
  • Latency（延迟） ——分布（p50, p95, p99, p99.9）。中位数是通常体验；p99 是仍然常见的最差体验。
  • Saturation（饱和度） — 连接数、队列深度、CPU 使用率、内存使用率。"我们距离在持续负载下崩溃有多近"的信号。如果饱和度持续攀升，应在延迟或错误率恶化之前提前扩容。

各信号按集群（例如 feed 集群）、按路由、按上游后端（便于定位某台慢机器）以及整体维度分别上报。

每个后端的指标。 LB 视角下某后端的数据，可能与该后端自身的指标不一致。LB 测量的是其 socket 处的挂钟时间，后端测量的是应用层时间。网络延迟、后端 accept 队列中的排队时间以及 TLS 开销，均体现在 LB 侧而非后端日志中。每个后端的 LB 指标：

  • 活跃连接数 — 当前已建立的连接数。
  • 发往该后端的 RPS — 每秒请求数。
  • 该后端的 p50/p95/p99 延迟。
  • 该后端的错误率。
  • 待处理请求数 — 有时 LB 会维护一个等待连接池空闲连接的请求队列；该深度是饱和度的指示指标。

"后端 p99 正常但 LB p99 很差"的排查。 经典场景。后端上报 p99 为 50 ms，LB 上报 p99 为 500 ms。多出的 450 ms 来自哪里？

  • LB 侧队列积压。 请求到达速度超过 LB 的派发能力（LB 进程 CPU 饱和）。每个请求在 LB 的 accept 队列中等待；队列时间对后端不可见。排查：LB CPU > 80%，accept 队列深度 > 0，请求速率持续上升。
  • 连接池耗尽。 LB 与后端之间维持 100 条连接，101 个请求同时到达，第 101 个请求需要等待。等待时间体现在 LB 延迟中，但不体现在后端延迟中。修复方案：增大连接池，或使用 HTTP/2 多路复用。
  • 重试。 LB 在后端故障后对请求进行了重试。后端只看到第二次（成功的）请求，而 LB 看到的是总耗时。可通过过滤 retry-count > 0 的日志来排查。
  • TLS 握手开销。 新建连接需要 1-2 个 RTT 的 TLS 握手时间。LB 从客户端发出首个字节开始计时，后端则从握手完成后才开始计时。可查看 session 复用率。
  • 慢启动爬坡。 某个新后端权重为 1%，流量速率看起来正常，但一个小突刺就超过了其降速后的容量，导致队列积压，延迟飙升。可查看慢启动状态。

解决思路是："分别向 LB 和后端索取延迟分布数据，找出差距，并追踪是哪个队列或连接池导致了问题。"有时这个差距会暴露出缺失的监控层——增加连接池等待时间和 accept 队列深度指标，往往就能找到答案。

分布式追踪。 OpenTelemetry / Zipkin / Jaeger。每个请求都会获得一个 trace ID；LB 生成一个 span（"LB hop"），后端生成其 span（"backend processing"），下游调用各自生成 span。以火焰图形式可视化后，可以精确看出每一跳耗时多少。这是跨系统延迟排查的黄金标准，如果客户端没有注入 trace ID，LB 正是注入的最佳位置。

§26. 内核级与 eBPF

L4 LB 领域的最前沿正从用户态 DPDK 向内核态 eBPF 迁移。这一转变已历经数年，且仍在加速推进。

XDP (eXpress Data Path)。 Linux 内核中位于 NIC 驱动层的钩子——是最早可介入的节点。XDP 程序是加载到内核的 eBPF 字节码；它检查每个入包并返回以下操作之一：

  • XDP_PASS — 继续沿正常内核协议栈传递（最终投递到某个 socket）。
  • XDP_DROP — 立即丢弃该数据包。这是最快的丢包方式——用于 DDoS 防护。
  • XDP_TX — 改写头部后从同一 NIC 重新发送。用于 L4 LB 的回程路径。
  • XDP_REDIRECT — 发送到另一个 NIC 或另一个 CPU。用于多 NIC LB。

XDP 在 之前 内核网络层之前运行（无需分配 sk_buff，无需路由决策，无需 netfilter），因此数据包处理速度比用户态快 1-2 个数量级。Maglev 表查找、流表查找、头部改写、重传——全部在 XDP 程序内完成，每包 50-200 ns，每核 5-10 Mpps。

内核生态系统依然健在： tcpdump 对 XDP 程序 PASS 的数据包照常生效， ftrace 可以挂载到 XDP 入口/出口，perf 事件正常上报， bpftool 可内省已加载的程序。

无 sidecar 的 L7 路由与 eBPF（Cilium）。 Cilium 是一个基于 eBPF 实现的 L7 服务网格。它不需要每个 pod 部署一个 Envoy sidecar，而是将 eBPF 程序加载到内核的 socket 层，实现以下功能：

  • L4 负载均衡 — 用 eBPF maglev 取代 kube-proxy 的 iptables/IPVS。
  • 网络策略 — 在内核中根据身份（而非 IP）丢弃不允许的数据包。
  • L7 路由 — 内核态代理（或每节点 Envoy，取决于 Cilium 版本）负责处理需要 L7 能力的 HTTP 流量路由。
  • 可观测性 — eBPF 程序无需 sidecar 即可上报指标和追踪事件。

优势：没有 sidecar 意味着没有每个 pod 的 Envoy，没有每个 pod 的资源开销，没有启动延迟，也无需执行繁琐的升级流程。局限：eBPF 无法实现 Envoy 的所有功能（扩展、自定义 Lua 过滤器等）——Cilium 在内核中处理常见场景，其余情况回退到 Envoy。

现代 Linux 负载均衡栈。 2026 年代的 L4 LB 栈大致如下：

  • NIC 卸载 — RSS 对五元组进行哈希，将数据包分发到 N 个 RX 队列；每个队列的中断绑定到一个 CPU 核。数据包处理速率随核数线性扩展。
  • XDP eBPF 程序 在每个 NIC RX 队列上运行。执行 Maglev 查找、流追踪、头部改写，并丢弃异常流量。
  • BPF maps （内核哈希表）保存 Maglev 表、流表和后端映射。由用户态控制平面通过 syscall 更新。
  • 可选的 sk_buff 路径 用于处理 XDP 无法决策的流量（通过 XDP_PASS 返回）；由标准内核网络栈处理。

对于新项目，用户态 DPDK 代码正逐渐进入历史。在 XDP 能够以 100% 的内核生态优势提供约 80% 性能的情况下，继续维护 DPDK（没有内核工具链、需要自定义 NIC 驱动、手动处理 ARP/ICMP）的代价实在过高。

§27. 无代理负载均衡

网格模式（每个 pod 部署 Envoy，所有调用经由代理）是有代价的。无代理 LB 的核心问题是：如果 客户端 直接选择后端，中间没有 LB 跳转，会怎样？

gRPC 的客户端侧负载均衡。 gRPC 内置了客户端侧 LB 支持。客户端通过名称解析器（例如 DNS、xDS 或 Consul）获悉 后端端点集合 以及负载均衡策略（round-robin、P2C、自定义）。每次调用时，客户端选择一个端点并建立（或复用）连接。服务端侧 LB 对内部调用不再参与。

  • xDS 发现后端。 客户端订阅 xDS——与 Envoy 使用的协议相同。控制平面推送端点更新，客户端的解析器随之更新。同一个 xDS 推送在更新 Envoy 的 CDS/EDS 的同时，也更新所有 gRPC 客户端。视图一致，无需额外基础设施。
  • Picker 策略。 默认为 round-robin；自定义 picker（P2C、感知地域性、基于延迟）均可插拔。

为何这能免除内部流量的 LB 跳转代价。 一次典型的网格调用的开销：

  • 客户端 → sidecar Envoy。 约 0.5 ms 跳转。（同一台机器，但仍有上下文切换。）
  • Sidecar → 远端后端。 网络 RTT（数据中心内约 0.5 ms）。
  • 远端后端 → 其 sidecar。 同机跳转。
  • Sidecar → 应用。 同机跳转。

合计：每次调用约 2 ms 的额外开销，发生在应用处理请求之前。对于扇出 50 个服务的场景，这意味着 100 ms 的额外开销。

无代理 gRPC：

  • 客户端（应用） → 远端后端（应用）。 仅网络 RTT，大约 0.5 ms。

内部调用的网格代价消除了。来自 Google（在内部大量使用无代理 gRPC）的生产数据显示，高扇出工作负载的延迟降低了 10-30%。

权衡。

  • 客户端变得更智能。 每个客户端（每种语言：Go、Java、Python、C++ 等）都需要集成 LB 库、xDS 解析器和连接池管理。"所有逻辑集中在 Envoy"的优势消失了——你需要在每种语言中重新实现这些逻辑。
  • 策略分发更难集中管控。 使用 sidecar 时，"所有人以指数退避重试 3 次"只需一份 Envoy 配置。使用无代理方案时，各客户端库的行为必须保持一致。xDS 有所帮助（控制平面推送相同配置），但库版本偏差是真实存在的问题（"我们的 Go 客户端用的是 gRPC 1.50，Java 客户端用的是 1.55，它们的重试语义略有差异"）。
  • 运维复杂性。 配置错误的客户端比配置错误的 sidecar 更难修复（更新 sidecar 无需触及应用；而在大多数情况下，更新客户端库必须重新部署应用）。
  • 可观测性更难保障。 没有 sidecar，trace/metric 埋点必须内置于客户端。gRPC 对此有良好的内置支持，但这又是一件需要保持一致性的事情。

2026 年的现状：无代理方案正在以 gRPC 为主的内部架构中胜出（Google、字节跳动、大型金融机构），而 sidecar 在混合协议（HTTP/1.1、gRPC、WebSocket、自定义协议）的多语言环境中仍占主导。Ambient mesh（Istio）和 Cilium 的 eBPF 网格进一步模糊了界限——没有 sidecar，但节点级代理仍然存在，只是跳转代价相对较小。

§28. 成本与经济性

LB 的成本无处不在——实例费用、带宽、许可证费用、工程师时间。在采用托管服务或自建之前，值得深入了解。

托管 LB 定价。 AWS ALB 定价（截至 2026 年最新数据）：

  • 固定时费：每个 ALB $0.0225/小时（约 $16.40/月固定成本）。50 个 ALB = $820/月基础费用。
  • 按 LCU（负载均衡器容量单元）计费：$0.008/LCU-小时。LCU 定义为"新连接数、活跃连接数、已处理字节数、规则评估次数中的最大值"。中等流量 ALB 持续消耗 10-50 个 LCU，低负载时每个 ALB 约 $2-11/月。峰值期间（零售 ALB 在黑色星期五），LCU 可能在一小时内飙升至 1000+，单小时费用约为 $8。
  • 数据流入免费，流出 $0.01-0.09/GB（因地区而异）。

对于使用一个 ALB 持续处理 1k RPS 的小型 SaaS：约 $25-50/月，经济实惠，直接开用即可。

对于使用同一托管 LB 处理 1M RPS 的超大规模用户：每个 LB 每月数万美元；乘以数百个 LB，年账单高达数百万美元。此时，自建 Envoy 方案开始显得更具吸引力。

NLB 定价结构类似，但使用 NLCU（网络负载均衡器容量单元）且单位成本不同。GCP 和 Azure 的定价大体相当。

自管 Envoy/HAProxy 的 TCO（总拥有成本）。 自建 LB 的真实成本是多少？

  • 硬件/云实例。 一个每实例处理 250k RPS 的 Envoy 集群需要 N 个实例加上冗余容量。假设一个由 50 台 c5.4xlarge 实例组成的集群（每台 $0.68/小时，合计 $24,500/月）。
  • 带宽。 与托管方案相同——数据无论如何都要离开数据中心。
  • 工程人力。 建设、运维、值班。现实情况下，至少需要 3-5 名工程师（年人力成本含薪资福利约 $1-2M）来规模化运营自定义 LB 层。包括故障响应、容量规划和升级测试。
  • 可观测性基础设施。 指标采集、日志存储、追踪系统。中型集群每年约 $50-200k。
  • 软件/维护。 Envoy 是开源的，但安全补丁、版本升级和集成测试都需要投入时间。HAProxy Enterprise 是商业版（每对实例每年 $5-20k）。

合计：自管集群每年约 $2-3M，可替代规模化下每年约 $5M 的托管 LB 费用。"临界点"取决于流量规模。

"我们通过自建 LB 节省 $X"的临界点。 粗略经验法则：

  • 托管 LB 年费低于 $500k 时： 继续使用托管方案。自建的工程成本高于节省的 LB 费用。应将工程师精力投入到产品建设上。
  • 年费在 $500k 至 $3M 之间： 视情况而定。特殊需求（自定义 WAF 规则、内部专用协议、合规约束）可能迫使自建；否则托管方案仍更划算。
  • 年费超过 $3M 时： 自建开始盈利。各大超大规模厂商都运营着自己的 LB（Cloudflare Unimog、AWS Hyperplane、Google Maglev、Facebook Katran），正是基于这一原因——同时因为在他们的规模下，需要托管 LB 无法提供的功能。

还有一个非货币临界点： 功能限制。如果托管 LB 不支持某项所需功能（特定时期的 HTTP/3 后端连接、特定 TLS 扩展、自定义协议），无论成本如何都需要自建。

"托管 LB 隐藏成本"模式。 常见的意外：托管 LB 看起来很便宜，直到流量增长或功能蔓延增加了 LCU 消耗。一个团队起步时每个 ALB $50/月，六个月后拥有 200 个 ALB（每个服务每个环境一个），LB 账单达到 $200k/年，而当初没有人为此做规划。最佳实践：以与流量增长相同的速度为托管 LB 成本增长做预算。考虑用一个 ALB 服务多个后端（基于路径的路由），而不是每个后端一个 ALB。

架构层面的核心真相：负载均衡已经如此商品化，以至于 错误的问题 是"Envoy vs HAProxy vs Nginx"。正确的问题是"托管还是自建"，而答案会在可预测的规模节点处翻转。

§29. 总结

负载均衡是一类流量分发基础设施，一端是线速内核旁路数据包转发器（Maglev、Katran、Unimog——XDP/eBPF、Maglev 哈希查找表，每包约 200 ns），另一端是用户态感知应用层的富策略代理（Envoy、NGINX、HAProxy——TLS 终止、路径路由、重试、网格策略）；两层在经典的 anycast + ECMP + L4 + L7 + 后端拓扑中串联，Maglev 表是唯一被复制的状态，且由后端列表确定性生成，因此数据平面是持久无状态的；设计中最难的问题——变更下的稳定哈希、慢启动、热键、分层健康检查、粘性故障转移、TLS 开销、SYN 洪泛——都可以追溯到同一根源：LB 是一个每天被重复执行数十亿次、没有第二次机会的路由决策。
