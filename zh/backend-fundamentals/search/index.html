组件类别：基于倒排索引的文本与结构化字段搜索引擎。实现涵盖 Apache Lucene（底层库）及其衍生产品（Elasticsearch、OpenSearch、Solr）、替代引擎（Vespa、Tantivy、Manticore），以及专用系统（代码搜索的 Zoekt / Sourcegraph 和 GitHub Blackbird，日志搜索的 Grafana Loki / Quickwit，推文搜索的 Twitter Earlybird，领英档案搜索的 LinkedIn Galene）。§4 中的字节级机制适用于所有基于 Lucene 的引擎，以及 Vespa 和 Tantivy 所遵循的 segment 模式。

§1. 搜索系统的本质

一个 搜索系统 接收一批文档和一个自由形式的查询，以低延迟返回 有序列表 匹配文档的有序列表。其核心数据结构是 倒排索引：一个从 词项 → 包含该词项的文档列表的映射，并附有每文档、每字段的元数据，足以计算相关性得分，以及应用过滤、分面和排序。

搜索系统是 派生存储，而非真实数据源。它位于主数据库的下游，通过变更捕获或批处理管道同步数据，用于回答主数据库无法高效处理的查询——模糊文本匹配、有序检索、大规模分面、地理距离、关键词 + 向量混合检索。

与相邻类别的区别

  • SQL LIKE '%term%' 执行全表扫描；B+ tree 无法在前导通配符查询上使用索引，超过约 100 万行后不可行。Postgres tsvector + GIN（通用倒排索引）是真正的倒排索引，但不分片时上限约 1000 万文档，缺乏 BM25（最佳匹配 25），且没有达到搜索引擎延迟级别的原生分面能力。
  • 文档数据库查询 （MongoDB、Couchbase）擅长按 ID 查询；内置的文本索引是附加在主存储上的简单倒排。未针对评分扇出或近实时刷新进行优化。
  • 向量搜索 / ANN（近似最近邻）引擎 （Pinecone、Milvus、pgvector）通过 HNSW（分层可导航小世界）或 IVF（倒排文件）图回答"语义上接近该嵌入的文档"，但 非 无法回答"包含该词项的文档"。现代基于 Lucene 的引擎和 Vespa 已融合两者，但纯向量引擎无法取代关键词检索。
  • OLAP（联机分析处理）引擎 （Druid、Pinot、ClickHouse）在分面和聚合方面有重叠，但针对分析扫描进行了优化，而非在稀疏词项空间上的有序检索。

搜索系统不擅长的场景

  • 按主键点查 ——比 KV（键值）存储慢 5–10 倍；搜索查询会扇出到每个分片。
  • 数据真实来源 ——segment 可能损坏；schema 只能向前演化；集群丢失必须可恢复。
  • 强一致性 ——刷新是近实时的（默认 1 秒）；读己之写并非免费。
  • 事务性更新 ——不支持多文档原子性。
  • 亚毫秒 p99 ——多分片扇出的 segment 搜索无法达到 <1ms。
  • 持续 >100k 写/秒/分片 ——合并代价占主导；专用引擎更胜任（Pinot upsert、Prometheus）。

§2. 固有保证——设计层面的契约

已提供：

  • 基于词项的大规模检索：时间复杂度为 O(倒排列表长度)，而非 O(N)。
  • 排序：内置 BM25 长度归一化 TF（词频）评分；可通过 function score、script score 或 learned-to-rank 重排进行扩展。
  • 近实时可见性：t=0 时索引的文档在 t=1s 时可被搜索（可配置）。有界，并非实时。
  • 分面 / 聚合 通过 doc values（按字段的列式存储）作用于已索引字段。数百万条匹配下响应时间低于 100ms。
  • 水平可扩展性 通过分片实现； 读取可用性 通过副本实现。

未提供——需额外叠加：

  • 作为主存储的持久性。 Translog（预写日志）保证单次写入的恢复，但权威记录必须保存在其他地方。模式始终是：主库 → CDC（变更数据捕获）→ 搜索引擎。
  • 强一致性。 副本可能落后数十毫秒到数秒。
  • 就地修改 schema。 Mapping 是 只能向前演化的。更改分析器或字段类型意味着需要重建索引并切换别名。
  • 默认幂等写入。 必须通过 external_version （单调递增的真实序列号）编码，或接受重试可能覆盖较新写入的风险。
  • 多文档原子性。 不支持事务。
  • 隐私 / 鉴权过滤。 引擎返回所有匹配文档；按用户授权过滤是调用方的责任。

心智模型：搜索引擎是一个 可扩展、可复制、近实时、有序检索的物化层。超出该契约范围的一切都是您自己的问题。

§4. 底层数据结构——Lucene 的字节级实现

Lucene segment 格式是理解现代搜索引擎最重要的一件事。Vespa 和 Tantivy 在细节上有所不同，但使用相同的概念原语。

4.1 一个分片就是一个 Lucene 索引

ES/OpenSearch 的一个分片 就是 磁盘上的一个 Lucene 索引。引擎是对 N 个 Lucene 索引的分布式封装。一个 Lucene 索引是一个 segment 文件的目录，每个 segment 是由一次 flush 生成的不可变小型索引。

/data/nodes/0/indices/products_v3/0/index/
├── _0.cfs          ← segment 0, compound file (all sub-files packed)
├── _0.cfe          ← segment 0 entry table
├── _0.si           ← segment 0 segment info
├── _3.fdt          ← stored fields
├── _3.fdx          ← stored fields index
├── _3.tim          ← term dictionary (FST tail)
├── _3.tip          ← term index (FST root in memory)
├── _3.doc          ← posting list
├── _3.pos          ← positions
├── _3.dvd          ← doc values data
├── _3.dvm          ← doc values metadata
├── _3.nvd          ← norms data
├── _3.fnm          ← field info
├── segments_42     ← commit point: lists live segments
└── write.lock

核心不变量： 每个 segment 一旦写入即不可变。更新和删除以墓碑的形式叠加在 segment 之上；实际的空间回收在合并时发生。合并过程之于搜索引擎，如同 compaction 之于 LSM（日志结构合并）树——这个类比是精确的，因为 Lucene 的 segment 模型在结构上就是倒排索引的 LSM 树。

4.2 单个 segment 内部

(a) 词项字典——FST（有限状态转换器）

词项字典将 词项 → 倒排列表偏移量映射。Lucene 将其编码为 FST——一个同时压缩共享前缀 。 后缀的 DAG（有向无环图），在概念上是一个最小化的 trie。

terms: "engineer", "engineers", "engineering", "england", "english"

  start ─e─►─n─►─g─┬─i─►─n─►─e─►─e─►─r─►●(eng-pointer)─┐
                   │                    └─s─►●         │
                   └─l─►─a─►─n─►─d─►●               (suffix shared)
                                  └─i─►─s─►─h─►●

为何选择 FST 而非普通 B-tree：

  • 索引部分驻留在内存中 (.tip）。尾部（.tim）在磁盘上，每次词项查找最多访问一次。
  • 对于拥有约 1000 万个唯一词项/分片的 10 亿文档索引，内存中的 FST 大小为 约 100 MB，而非 10 GB。 50–100× 压缩率 相较于平铺有序词项列表。
  • 查找复杂度为 O(term_length) 次字节比较；不依赖词项数量。
  • 前缀遍历（engin*）是单次 FST 遍历，按字典序输出所有匹配词项——自动补全代价低廉。

权衡：FST 是 只追加的，无法在 FST 中途插入词项。这没有问题，因为 segment 是不可变的：添加文档会构建一个带有 新的 FST 的新 新的 segment。

(b) 倒排列表——Frame-of-Reference + PFOR 压缩

对于每个词项，倒排列表是包含该词项的所有文档的 (doc_id, term_freq, positions...) 序列。Lucene 使用叠加压缩进行编码：

  1. 对 doc_id进行 Delta 编码 [12, 47, 89, 142] ：将 [12, 35, 42, 53]存储为
  2. 。差值比绝对值更小。 PFOR-Delta（带补丁的参考帧） 作用于差值：选择适合大多数差值的位宽（如 8 位），将 128 个差值以固定宽度块打包。离群值（"补丁"）使用旁路异常列表存储。结果：典型列表约 3 位/文档，相较未压缩的 32 位/文档，实现.
  3. 10× 压缩 每 128 个文档交错一个跳跃指针 (doc_id, file_offset) ：允许跳过中间块而无需解码。对于合取查询至关重要： q = "stainless AND bottle", stainless的下一个文档为 50000 → 跳过 1000、5000 处的块而无需解码。交集复杂度从 O(N+M) 降至趋近 O(min(N,M) × log(max)/skip_interval)。
Posting list for term "engineer":
block 0 (docs 0–127):    [bit-packed 128 deltas, 8 bits each = 128 bytes]
                          skip → (doc_id=145, offset=byte_500)
block 1 (docs 128–~300): [bit-packed 128 deltas]
                          skip → (doc_id=380, offset=byte_900)
...
Iterator asks "advance past doc 50000" — skip-list jumps straight to
the right block without decoding 50 prior blocks.

配套的 .pos 文件存储 位置信息 用于短语查询和邻近度评分；同样采用增量编码与位压缩处理。

(c) Doc values —— 按字段的列式存储

倒排索引的结构是 term → docs。对于排序、分面聚合以及基于字段值的评分，还需要 doc → field value。这就是 doc_values：按列存储的数组，每个文档每个字段一个条目。

docId:  0    1    2    3    4    5    ...
price: [42, 19, 99, 37, 19, 80, ...]   ← packed column
       (8 bits if max < 256, 16 otherwise — minimum bit width)

存储于 .dvd / .dvm，以 mmap 方式映射。这正是聚合能够高效运行的原因——若没有 doc values，每次分面统计都必须重新从倒排索引中分词。相同的列式布局也是 Druid/Pinot 能实现交互式 OLAP 的原因。

与 FieldData（旧版堆内存方案）的权衡：doc values 存储在磁盘并通过 mmap 映射，不会耗尽 JVM 内存（Out Of Memory）。首次读取需访问磁盘；后续读取由操作系统 page cache 接管。

(d) 存储字段 —— 原始文档内容块

_source （原始 JSON）存储于 .fdt / .fdx，以 16 KB 块为单位进行 LZ4 压缩。仅在搜索的第二阶段（最终 top-K 阶段）才会被取出。对非展示字段禁用 _source 可节省 30–50% 的磁盘空间，并显著加快批量重建索引的速度。日志搜索保留 _source （文档即 是 要点）；商品目录则可省略并从主数据源重新获取。

(e) Norms —— 每字段的长度归一化

.nvd 每个文档每个字段存储一个字节，用于编码字段长度（有损压缩）。BM25 使用该值进行长度归一化，并加载到内存中。

4.3 写入路径 —— 一条文档的端到端流程

无论该文档是 Booking.com 的酒店、Earlybird 的推文，还是 Galene 的用户档案，处理机制都是一样的。

Step 1: Document arrives at the shard's primary.
        Doc = {id: 12345, title: "Stainless steel bottle", ...}

Step 2: Append to translog (per-shard WAL).
        ┌─────────────────────────────────────────────────────────┐
        │ translog-{generation}.tlog                              │
        │ [op=INDEX, seq_no=12834, doc_id=12345, payload=...]     │
        └─────────────────────────────────────────────────────────┘
        Write to OS buffer. If translog.durability=request (default),
        fsync at end of bulk request. ← DURABILITY POINT.

Step 3: Apply to in-memory indexing buffer.
        - Tokenize text fields ("stainless", "steel", "bottle").
        - Append to per-thread DocumentsWriter buffer.
        - RAM only. NOT searchable yet.

Step 4: Replicate to in-sync replicas.
        Primary forwards op to each replica's translog.
        Primary waits for replicas to ack their translog write.
        ← Replicas are durable for the SAME ack.

Step 5: Refresh (every refresh_interval, default 1s).
        - Flush in-memory buffer to a NEW segment file.
        - Open a new IndexReader that includes the new segment.
        - Doc is NOW searchable.
        - Segment is NOT yet fsync'd; lives in OS page cache.

Step 6: Flush (less frequent, ~5 min or 512 MB of translog).
        - fsync all segment files.
        - Write new segments_N commit point.
        - Roll over to a new translog generation.
        - Old translog reclaimable.
        ← FSYNC POINT for segments.

Step 7: Merge (background).
        - Tiered merge policy picks similar-sized segments.
        - Concatenates, dedups tombstones, writes one larger segment.
        - Old segments deleted after merge commits.

崩溃后的数据存活情况：

  • 步骤 1–2 之间：文档丢失；客户端未收到确认。可接受。
  • 步骤 2–4 之间：主节点有 translog，副本节点没有。主节点将 translog 重放到缓冲区；文档得以恢复。若主节点宕机，副本的分片恢复流程会将其回退到步骤 2。
  • 步骤 4–5 之间：translog 已存在于主节点和副本节点。重放 translog 重建缓冲区。文档未曾可搜索便已恢复——没问题，因为确认合约是"刷新后可见"。
  • 步骤 5–6 之间：段已在 page cache 中，但尚未落盘。重启后段消失，但 translog 重放会重新创建等效的段。
  • 步骤 6 之后：段已落盘；translog 截断至提交点。引擎从提交点处打开并继续运行。

这正是 translog 作为持久性原语的原因，而非段文件。 段是不可变的，延迟进行 fsync，并可重建。translog 是 WAL，必须在每次请求时进行 fsync。

4.4 为什么刷新间隔为 1 秒

每次刷新都会创建一个新段。在每个分片 5000 次写入/秒的情况下，1 秒的刷新会生成一个 5000 文档的段——这是有意义的工作单元。每个段都有固定开销（FST 提示驻留在内存中、文件句柄、元数据）。1 秒刷新每小时累计 3600 个段， 由合并器加以控制；100 毫秒刷新每小时累计 36000 个段，合并器将跟不上节奏——合并 CPU 会挤占查询 CPU。每次搜索都会遍历 所有存活的段；一个有 100 个段的分片比有 10 个段的分片慢约 30%。1 秒是经过实践验证的最优平衡点。

特定工作负载的调优： 批量重建索引 / 日志摄取 设置 refresh_interval: -1 加载期间，完成后重新启用——同等硬件上可将吞吐量从 5 万提升至 20 万以上条/秒。 电商商品目录：5–30 秒刷新间隔即可。 实时推文 / 新闻搜索：200 毫秒，接受合并开销（Earlybird 进行了激进调优）。

4.5 段合并 —— 分层策略

分层合并将段按大小归入不同层级，从中挑选最多 max_merge_at_once （默认 10）个大小相近的段合并为下一层的一个更大段。避免了二次方级别的合并开销；段数量呈对数增长。竞争性替代方案是 大小分层合并 （Cassandra 风格）——更为激进但写放大更严重。Lucene 的分层合并在写放大（文档生命周期内约 2–3 倍）与查询侧段数量之间取得平衡。

强制合并 (_forcemerge?max_num_segments=1）将所有内容折叠为一个段。适用于只读归档（已关闭的月度日志索引、不可变的历史目录）。对活跃索引而言是灾难性操作——会独占 I/O 数分钟乃至数小时。

4.6 为什么选择倒排索引而非其他方案

  • vs B+ tree: text LIKE '%shoes%' ：需要全量扫描；前缀通配符的选择性很差。
  • vs 带二级索引的 LSM tree：LSM 提供快速的点写入，但跨二级索引的多词查询交集本质上就是倒排列表交集所做的事——而 Lucene 在 PFOR 位压缩数据上结合跳表指针完成这一操作，I/O 减少约 5–10 倍。
  • vs 内存哈希表：适用于微型索引；在十亿文档级别会撑爆堆内存。
  • vs 单独使用 HNSW 向量索引：纯 ANN 无法按精确词项过滤，分面代价高，对精确标识符（SKU、错误码、名称）的匹配精度不足。现代架构采用混合方案：精确查询使用关键词倒排列表，语义相似性使用 HNSW，通过 RRF（Reciprocal Rank Fusion）或 learned-to-rank 进行融合。

§5. 不同规模与领域的容量边界

搜索引擎在文档数量上横跨六个数量级，在查询速率上横跨四个数量级。技术栈相同，但拓扑结构差异极大。

下一个瓶颈出现的位置

  • 小规模。 Translog fsync 延迟。个人开发者将 translog.durability=async 设置为异步以突破单 SSD 的 1 万次写入/秒限制。无扇出开销。
  • 中等规模。 每个分片的查询开销。一个含 1000 万文档的 50 GB 分片，大部分预算消耗在 BM25 评分器上。
  • 大规模。 扇出尾延迟。每次查询触达所有分片；最慢的一个决定整体延迟。若每个分片有 1% 概率超过 50ms，扇出 100 个分片则有 1 - 0.99^100 ≈ 63% 超过 50ms 的概率——除非通过路由限制扇出，否则 p99 会崩溃。Galene 运行约 100 分片 × 3 副本，每个地区约 30 个数据节点，约 10 万 QPS 下 p99 约 200ms。
  • 超大规模。 存储成本与查询速率的矛盾。在 Blackbird 规模（2000 亿文档，640 TB），全量热 NVMe 存储成本极高。采用冷热分离，将可搜索快照存放在 S3；Quickwit 的计算-on-S3-segments 设计正是为此层级而生。

每分片最佳容量：20–50 GB。 低于 10 GB，扇出开销占主导；高于 50 GB，查询 p99 飙升。参考目标：

  • 100M × 2 KB/doc = 200 GB → 10 shards × 20 GB.
  • 1B × 3 KB/doc = 3 TB → 100 shards × 30 GB.
  • 200B × 3 KB/doc = 600 TB → tens of thousands of shards, federated.

数据节点配置：32–64 vCPU，128–256 GB RAM，2–4 TB NVMe。堆内存上限为 31 GB（compressed-oops 的上限）。剩余 RAM 作为 page cache——查询延迟的好坏完全取决于 page cache。需确保 倒排列表的工作集能够放入其中.

§7. 深入相关性调优 —— BM25、权重提升与 Explain API

检索功能正常运行后，下一步工作是使结果列表的顶部排序正确。搜索工程师在相关性调试上花费的实际工时比其他任何事情都多，而在 Lucene 派生引擎中，术语体系是一致的：词频、逆文档频率、长度归一化、函数评分以及 Explain API。

7.1 BM25 —— 每位搜索工程师都应铭记的公式

BM25（Best Match 25，"25"因其是伦敦城市大学 Okapi 项目中的第 25 次迭代而得名）自第 6 版起成为 Lucene 的默认评分函数。早期 Lucene 使用 TF-IDF（词频-逆文档频率），采用平方根 TF 和不同的长度归一化；BM25 修正了 TF-IDF 的两个固有缺陷。

对于单个查询词 t 针对文档 D:

score(D, t) = IDF(t) × (f(t,D) × (k1 + 1)) / (f(t,D) + k1 × (1 - b + b × |D|/avgdl))

where:
  f(t,D)  = term frequency of t in D (raw count)
  |D|     = length of D in tokens (for the queried field)
  avgdl   = average document length across the index for that field
  k1      = TF saturation knob (default 1.2)
  b       = length-normalization knob (default 0.75)

IDF(t)    = ln(1 + (N - n(t) + 0.5) / (n(t) + 0.5))
  N       = total docs in the index (for that field)
  n(t)    = number of docs containing t

对于多词查询，得分是各词项得分的 各词项之和 （技术上是跨查询子句的求和，其中单个子句本身可能是一个词项或短语）。这个总和就是 bool 。 dis_max 查询所进行的组合方式。

三个关键点有助于直观理解：

(a) 逆文档频率 —— 稀有词加分机制。对"Yifan stainless"的查询，几乎所有分数都归功于"Yifan"，因为几乎没有文档包含它。"stainless"出现在数万篇文档中，权重很低。这正是即使不做停用词过滤，"the"也几乎不贡献得分的原因。

(b) 词频饱和 — f / (f + k1) 是一条在 k1 + 1附近趋于饱和的双曲线。包含 50 次"bottle"的文档 非 比包含 1 次的文档相关 50 倍；大概只相关 2.5 倍。TF-IDF 使用 sqrt(f)，饱和更慢，使得关键词堆砌的页面得以主导结果。 k1=1.2 经过实证调优；提高到 2.0 以上会使曲线更平缓（对 TF 更敏感），降低到 0.5 会使曲线更陡峭（TF 迅速达到上限）。

(c) 长度归一化 — (1 - b + b × |D|/avgdl) 按文档长度与平均长度的比值进行除法归一化。一个包含"bottle"的 50 词商品标题，得分高于一篇同样只提到一次"bottle"的 5000 字博客文章。 b=0.75 是经验性默认值。 b=0 完全禁用长度归一化（适用于长度无关的字段，如 SKU）； b=1 表示完全归一化。

7.2 调优 k1 。 b 按字段

不同字段需要不同的参数：

字段 长度特征 推荐 k1 推荐 b
商品标题 短而紧凑，每个词都重要 1.2（默认值） 0.5–0.75
商品描述 较长，为 SEO 重复关键词 1.0 0.75
评论 长度不定，重复是噪声 0.5 0.75
作者 / 品牌 通常为单个词元 1.2 0（不进行归一化）
代码标识符 短，通常唯一 0.5 0
新闻文章正文 长篇散文 1.5 0.75

在映射中按字段覆盖设置：

{
  "settings": {
    "similarity": {
      "title_sim": { "type": "BM25", "k1": 1.0, "b": 0.5 },
      "body_sim":  { "type": "BM25", "k1": 1.5, "b": 0.85 }
    }
  },
  "mappings": {
    "properties": {
      "title": { "type": "text", "similarity": "title_sim" },
      "body":  { "type": "text", "similarity": "body_sim" }
    }
  }
}

基于点击日志进行 A/B 测试；相关性 @5 是通用指标。生产团队以 k1 步长 0.3 迭代 b 步长 0.1 迭代；预计每轮调优前可获得 1–5% 的点击提升，之后趋于平稳。

7.3 查询时 vs 索引时权重提升

为字段增加额外权重的两种方式：

索引时权重提升 —— 在索引时乘入字段的 norm 中。 建议避免使用。 Lucene 在 6.x 版本中移除了对单个文档索引时权重提升的支持，原因在于：(1) 它将排名决策烘焙进不可变的段中——更改一个权重就需要重新索引所有内容；(2) 有损量化为一个字节；(3) 与 BM25 自身的长度归一化组合效果差。

查询时权重提升 —— 在搜索时对某个子句应用乘数。易于迭代。

{
  "query": {
    "multi_match": {
      "query": "stainless steel bottle",
      "fields": ["title^3", "brand^2", "description^1"]
    }
  }
}

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 ^N 语法表示"将该字段的 BM25 得分乘以 N 后再跨字段求和"。标题中的匹配得分是描述中的 3 倍。权重提升 对得分是乘法关系，而非对排名，因此效果取决于底层 BM25 的分布。标题权重提升 3 倍通常效果明显；30 倍会使标题命中完全主导结果；1.5 倍只是轻微调整。建议通过 A/B 测试进行调优。

7.4 函数评分与自定义排名层

BM25 单独回答"文本相似度如何"。实际排名还需要回答"同时是否近期发布、是否热门、是否有库存"。 function_score 在 BM25 基础上乘以（或加上）自定义因子：

{
  "query": {
    "function_score": {
      "query": { "match": { "title": "espresso machine" } },
      "functions": [
        { "field_value_factor": {
            "field": "rating", "modifier": "log1p", "factor": 0.5
        }},
        { "gauss": {
            "release_date": {
              "origin": "now", "scale": "365d", "decay": 0.5
            }
        }},
        { "filter": { "term": { "in_stock": true }}, "weight": 2.0 }
      ],
      "score_mode": "multiply",
      "boost_mode": "multiply"
    }
  }
}

score_mode 对多个函数进行组合（multiply、sum、avg、max、min）。 boost_mode 将函数输出与基础 BM25 进行组合。乘法是常见选择——在保留强文本匹配相对顺序的同时，对陈旧或缺货的商品进行惩罚。

衰减函数（gauss, linear, exp）是"时效性很重要"的表达方式。 gauss 与 scale: 30d 表示"从原点起每 30 天得分减半"。新闻编辑室对此进行激进调优；电商对长青商品使用更长的衰减周期。

script_score （高一层）对每个候选文档运行 Painless 脚本。功能强大，代价高昂——用于"地理距离 × 评分 × 价格区间"的复合评分。避免在大规模场景的内层循环中使用（会触及每个候选文档）。

7.5 "为什么我 CEO 的档案排名靠后"的排查模式

LinkedIn 级别规模下的经典相关性 Bug：公司内有人搜索 CEO 的名字，结果却排在第 3 页之后。这在每家搜索团队都会重现。排查手册如下：

步骤 1：精确复现查询。 获取用户实际输入的查询字符串（包括自动补全、拼写错误、语言区域、过滤条件）。不要相信截图——用相同的请求体通过 curl 重新执行查询。

步骤 2：确认文档存在于索引中。 GET /profiles/_doc/{id} ——它在吗？是 _source 符合预期的吗？如果文档缺失，问题在索引管道，而非相关性。

步骤 3：确认文档能够匹配。 用相同查询配合 explain=true 以及对 _id: <ceo_id>:

GET /profiles/_search?explain=true
{
  "query": {
    "bool": {
      "filter": [{ "term": { "_id": "ceo123" }}],
      "must":   [{ "match": { "full_name": "Tim Cook" }}]
    }
  }
}

的硬过滤条件运行查询。如果 must 子句不匹配（无命中结果），则分析器是问题所在。查询 "Tim Cook" 可能被分词为 ["tim", "cook"]；文档字段 full_name 可能被索引为 keyword （仅整串匹配），因此 "Tim Cook" 能匹配，但单独的 "tim" 或 "cook" 则不能。或者该字段使用了错误的分析器（例如，本应使用带小写功能的 Standard 分析器，却用了 Whitespace 分词器）。

步骤 4：通过 Explain API 对比分数。 一旦能匹配，去掉 _id 过滤条件重新运行，并使用 explain=true 查看排名最高文档与 CEO 文档的得分明细：

{
  "explanation": {
    "value": 8.43,
    "description": "sum of:",
    "details": [
      { "value": 6.12, "description": "weight(full_name:tim in 123) [BM25]",
        "details": [
          { "value": 6.12, "description": "score(freq=1.0), computed as ...",
            "details": [
              { "value": 1.0, "description": "termFreq=1.0" },
              { "value": 9.8, "description": "idf, computed as ..." },
              { "value": 0.42, "description": "tfNorm, computed as 1/(1+k1*(1-b+b*dl/avgdl)) ..." }
            ]
          }
        ]
      },
      { "value": 2.31, "description": "weight(full_name:cook in 123) [BM25]", "details": [...] }
    ]
  }
}

从上到下阅读：总分、各子句得分、各词条明细（包括 TF、IDF 及 tfNorm 分量）。典型发现：

  • "Tim Cook" 的 IDF 低于预期，因为索引中包含 5 万个 "Tim" 和 1 万个 "Cook"——两者都很常见。
  • 排名最高的文档在标题字段中出现了 4 次 "Tim Cook"；CEO 的档案只在 full_name中出现了 1 次。TF 饱和度本应截断这一差距，但若 k1 过高则不会生效。
  • 排名最高的文档来自一个 30 词的档案；CEO 的来自一个 500 词的档案。长度归一化对较长文档有所惩罚。
  • 最坏情况：CEO 的名字在 current_company.ceo_name中，而不在 full_name中。查询只能匹配 full_name 。

步骤 5：用结构性修改来修复，而非一次性加权。 错误修复：添加 function_score 以提升 CEO 的特定文档 ID。不具通用性，会让工程师尴尬。正确修复：对 weight 的文档添加 is_executive: true加权，或添加 cross_fields multi_match 使查询同时匹配 full_name, headline与 current_company.ceo_name ，或为 "用户匹配其所在公司的知名职位" 添加 LTR 特征。三种方式均具通用性，都能提升 CEO 档案的排名—— 因为它本身的属性，而非 而非因为它是谁.

Explain API 是调试工具——切勿在生产热路径上调用（响应体会膨胀 10–100 倍）。应在相关性调试的 notebook、仪表板或 sidecar 服务中使用。

7.6 逐查询相关性监测

从长远看，相关性是一个度量问题。生产级搜索团队会构建：

  • NDCG（归一化折损累积增益）仪表板 基于点击日志计算。NDCG@10 是常用 SLA——"对于前 100 个高频查询，有多少比例的点击落在前 10 名？"
  • 并排对比工具 （例如 Yelp 的 "Sherlock"、LinkedIn 的相关性实验室），在两种引擎配置上运行相同查询并可视化展示差异。PR 需对此进行审核。
  • 采样查询回放 ——每晚将 0.1% 的生产查询在候选配置上回放；聚合得分差值是版本发布的门控指标。

若缺少这些工具，排序变更将流于感觉，且悄无声息地退步。

§8. 学习排序（LTR）——机器学习重排层

手动调优的 BM25 函数得分很快会触顶。标准升级方案是两阶段架构：以 BM25 作为快速候选生成器，再用学习模型对前 N 个结果重排。每一家严肃的消费级搜索系统（LinkedIn、Yelp、Booking、Pinterest、Etsy、Indeed）都采用这种方式。

8.1 为何需要两阶段

用复杂模型对每个文档打分是不可行的。在延迟预算内，神经重排器每次查询大约能处理 200–1000 篇文档；而 BM25 可以处理数百万篇。因此：

  1. 第 1 阶段——候选生成。 BM25（配合过滤条件和轻量函数得分）每个分片返回前 100–1000 个文档，合并为全局前 500–2000 个。优化目标是 召回率：只需确保真正相关的文档出现在这个集合中，集合内的排序不那么重要。
  2. 第 2 阶段——重排序。 学习模型对合并后的候选集打分。优化目标是 前 K 精确率：将最佳结果排在第 1 位，次佳排在第 2 位，以此类推。

关键权衡：第 1 阶段必须兼顾速度与高召回率。如果 CEO 的档案不在第 1 阶段的前 1000 名中，任何重排器都无能为力。

Query → BM25 retrieval (per shard, top 1000 each)
      → coordinator merges → top 2000 globally
      → fetch ML features per (query, doc) pair
      → LTR model scores all 2000
      → return top 10 sorted by model score

8.2 模型选择

梯度提升决策树（XGBoost、LightGBM） 占据主导。原因：数百个特征混合了稠密（数值型：BM25 得分、点击次数的对数、以天为单位的时效性）和稀疏（类别型：国家、查询意图类别）特征。GBDT 能原生处理这种混合，在数百万次曝光数据上训练只需数分钟，每对（查询, 文档）的推断仅需微秒级时间，且可解释—— predict_contrib 能说明哪些特征影响了哪些行。

神经重排器 （如 BERT cross-encoder、ColBERT、经 MS MARCO 训练的 Transformer 等交叉编码器）在相关性上胜出，但延迟代价更高。交叉编码器将 [query, doc] 联合输入并输出相关性得分——语义对齐能力更强，但比 GBDT 慢 100 倍。在预算允许时用于前 50 名的最终重排序。

混合架构 （LinkedIn、Pinterest）：GBDT 对前 2000 名评分缩减至前 200，再由交叉编码器对前 200 名评分得出最终排序。每个阶段在其延迟预算内发挥各自所长。

8.3 特征

三大类：

仅查询特征：查询长度、语言、命名实体的存在、查询意图分类器输出（"导航型" vs "信息型"）、该查询字符串的历史点击率。

仅文档特征：档案完整度的对数、以天为单位的时效性、热度（浏览量、星级、评论数）、语言、文档类型、结构化先验信息（已认证卖家、有库存、酒店 "1 小时内回复"）。

查询-文档交互特征 （高信号类）：BM25 得分本身、多字段 BM25 得分（标题、正文、品牌作为独立特征）、词汇重叠（出现在文档中的查询词条数量）、词条邻近度（包含所有查询词条的最小窗口）、该精确（查询, 文档）对的历史点击率、若混合检索参与则使用向量余弦相似度。

现代 LTR 模型在 500–2000 个特征上训练。过了某个临界点，特征工程和流水线可靠性比模型架构的选择更重要。

8.4 以点击率数据作为训练信号

主要的训练信号是隐式的： 点击日志。对每次搜索，记录曝光（哪些文档被展示、在什么位置）和点击（哪些被点击）。转化为成对偏好数据：

  • 文档 A 在第 3 位被展示，被点击。文档 B 在第 1 位被展示，未被点击。暗示：A > B（尽管位置更差）。
  • 文档 A 在第 5 位被展示，被点击。文档 B 在第 6 位被展示，被点击。两者均为正例——信号较弱。

模拟程序启动时会进行的重定基址操作。 LambdaMART （直接优化排序指标的 GBDT 变体）或神经模型中的 成对/列表式损失 。位置偏差是主要混淆因素——用户点击第 1 位仅仅因为它在第 1 位——可通过 逆倾向评分 （按位置对应的仅位置点击概率对高位点击进行降权）或 反事实评估 （交叉实验）加以修正。

冷启动问题：全新产品没有点击历史。缓解方法：对新文档依赖仅文档特征，并运行 bandit 策略，偶尔将新条目注入顶部结果以收集数据。

8.5 LinkedIn 和 Yelp 的实现方式

LinkedIn 人才搜索（Galene + LTR）：第 1 阶段是 Galene 上的 BM25，配合网络集群路由（扇出约 10 个分片）。第 1 阶段返回约 2000 个候选。第 2 阶段特征来自 在线特征存储 （Pinot、Venice），以（searcher_id, candidate_id）为键——特征包括 "关系度"、"共同雇主"、"技能重叠"、"搜索者网络最近浏览的档案"。LTR 模型是基于点击 + 接受连接数据训练的 GBDT，每周重训。单次请求延迟：第 1 阶段约 30ms，特征获取约 40ms，打分约 10ms，p99 总计在 200ms 以内。

Yelp 商家搜索：第 1 阶段是 Solr 上的 Lucene（Yelp 历史上同时使用基于 Lucene 的引擎和自定义层）。按地理位置和类目过滤。通过 BM25 + 轻量地理距离得分返回前 500 名。第 2 阶段 GBDT 使用商家评分、评论增速、当前营业状态、照片数量、该用户所在细分群体的历史点击率等特征打分。Etsy 的架构在结构上完全相同。

Booking.com：架构类似，但增加了额外复杂性——"最佳结果" 高度依赖出行日期（用户出行日期已满房的酒店无论得分如何都无关紧要）。日期可用性是 过滤，而非特征；由第 1 阶段处理。

8.6 部署模式——模型运行位置

引擎内部（Elasticsearch LTR 插件、OpenSearch Learning to Rank）：模型驻留在数据节点；重排序在分片结果返回前完成。延迟最低，耦合最紧。特征迭代（映射变更）和回滚较为痛苦。适合稳定的生产模型。

Sidecar（位于协调节点与客户端之间）：协调节点将前 N 个原始文档返回给搜索 API 层；API 调用模型服务服务。易于迭代（支持任意模型和特征，可独立部署）；因多一跳而延迟较高。大多数公司的默认模式。

引擎内张量（Vespa）：Vespa 通过其张量框架在引擎 在 内部

运行模型。将检索与排序组合在一个查询计划中，无需往返。这正是 Vespa 在 ML 密集型排序场景中备受青睐的原因。

8.7 在线评估——永远不要只信离线指标

  1. 在留存点击日志上的离线 NDCG 是必要条件，但还不够。真实的 LTR 上线总会包括： 交叉实验
  2. ——对小部分流量交叉展示生产模型和候选模型的结果，对比点击率。 A/B 测试
  3. ——在 1–5% 的流量上测试候选模型。不仅衡量 NDCG，还要衡量下游业务指标（购买量、申请开始次数、注册量）。关注公平性回退

：新模型是否系统性地降低了某些人口统计群体或卖家的排名？在 LinkedIn（求职者）和 Airbnb 式市场平台上尤为关键。

离线 NDCG 提升 2% 而线上收入提升为 0% 的情况很常见。离线提升 0.5% 而线上收入提升 3% 的情况同样常见。线上实验才是最终裁判。

§9. 混合检索——后 2023 时代的 BM25 + 向量

十年来搜索领域最具影响力的架构变革，是关键词检索与向量检索的融合。ChatGPT 使稠密检索走向主流；到 2024 年，生产系统已常规性地将两者结合。这一模式现已成为任何新搜索系统的标配。

9.1 为何选择混合，而非纯向量

  • 向量嵌入能捕获语义含义——"espresso machine" 和 "coffee maker" 在嵌入空间中很接近，即使没有任何词条重叠。但： 向量在稀有标识符上表现差。 B07XYZ1234 搜索 SKU ERR-4242 或错误码 liali或用户名。嵌入模型会归一化掉精确拼写；BM25 则能精准命中。
  • 向量在否定和运算符上表现差。 "iPhone NOT case" 需要经过查询解析；对整个短语的向量嵌入无法干净地减去 "case"。
  • 向量在大规模过滤上表现差。 "200 美元以下、4 星以上评价的浓缩咖啡机"——向量索引（HNSW）难以高效地应用后置过滤，因为图遍历到达候选时才知道过滤条件。

BM25 在同义词（"auto" vs "car"）、改写表达（"how do I reset my password" vs "I forgot my password"）和跨语言（"perro" vs "dog"）上表现差。这两种互补的失败模式正是混合检索胜出的原因。

9.2 向量存储的共置与分离

两种架构：

（a）共置：向量字段位于同一 Lucene 索引中。 Elasticsearch 和 OpenSearch 新增了 dense_vector 字段类型，支持 HNSW（分层可导航小世界）索引。映射方式：

{
  "mappings": {
    "properties": {
      "title":          { "type": "text" },
      "description":    { "type": "text" },
      "title_embedding": {
        "type": "dense_vector",
        "dims": 768,
        "index": true,
        "similarity": "cosine",
        "index_options": { "type": "hnsw", "m": 16, "ef_construction": 100 }
      }
    }
  }
}

每个 segment 现在同时包含倒排索引和 HNSW 图。查询可在一次请求中结合 kNN 、过滤条件和文本匹配，引擎在同一查询计划中交叉处理结果。

优点：单一系统、统一运维，原子更新（文本和嵌入在同一刷新边界内同步变更）。缺点：向量存储占用大（768 个浮点数 × 4 字节 = 每篇文档 3 KB，还不含 HNSW 图开销）；合并期间构建 HNSW 图代价昂贵；向量索引的压缩效果不如文本倒排索引。一个包含 7.68 维向量的 1 亿文档索引，仅向量本身就需要 300–500 GB。

（b）分离：Pinecone / Weaviate / Milvus + ES。 向量数据库存储 (doc_id, embedding)。搜索引擎存储文本索引。查询路径：

Query embedding → Pinecone returns top 1000 doc_ids by cosine
              ↘
                merge by doc_id
              ↗
Query text   → ES returns top 1000 doc_ids by BM25

两者在搜索 API 层合并。优点：各系统针对各自任务优化、可独立扩展，更换向量模型无需重建文本索引。缺点：需运维两套系统，doc_id 跨系统一致性由你负责，查询时合并增加延迟。

2024–2026 年，生产团队在索引规模允许时大多选择（a）用于新系统；当向量体量远超文本或团队已有专用向量数据库时选择（b）。LinkedIn、Pinterest、Spotify 均采用混合方式。

9.3 倒数排名融合（RRF）——简单且健壮的得分合并器

混合检索的难题：BM25 返回绝对得分（如 8.43）；向量余弦返回 [0, 1] 区间内的相似度。两者不可比。三种方案：

（a）得分归一化 ——将每个结果列表的得分除以其最大值，然后加权求和。脆弱：离群值会扭曲归一化。

（b）学习组合 ——训练一个小模型，以 bm25_score, vector_score及各特征为输入，预测相关性。效果强大，但移动部件更多。

（c）倒数排名融合（RRF） ——完全忽略得分，按 排名:

RRF(doc) = sum over all retrieval methods i of:
             1 / (k + rank_i(doc))

  k is a smoothing constant, default 60.
  rank_i is the doc's rank in method i's result list (1 = top).
  If the doc isn't in method i's top list, its contribution is 0.

对于 BM25 排第 1、向量排第 3 的文档： RRF = 1/(60+1) + 1/(60+3) = 0.0164 + 0.0159 = 0.0323。对于仅在 BM25 排第 1（不在向量前列）的文档： RRF = 1/61 = 0.0164。同时出现在两个列表中的文档即使排名居中也会被优先选择。

RRF 的优势在于不依赖得分校准，只需每个检索器产生合理的排序。它对异常得分分布鲁棒，添加新检索器无需重新调参，结果可解释（每个检索器的贡献清晰可见）。Elasticsearch 将其作为一等运算符（rrf 于 _search）正是因为生产团队一直在重复实现它。

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 k 参数控制顶部排名的主导程度。 k=60 是经验默认值；通过 A/B 调参。降低 k 会增加顶部位置的权重；提高 k 则会拉平差距。

9.4 2023 年后的语义搜索革命

变化根源：生产质量的嵌入模型变得廉价且轻量。 OpenAI text-embedding-3-small 为 1536 维，每 100 万 token 收费 0.05 美元。 Cohere embed-english-v3 为 1024 维。 开源模型 （sentence-transformers、BGE、E5）在单颗 CPU 上以 100–500 篇/秒的速度运行。对 10 亿文档目录进行嵌入，从 "不可行" 降至 "5000 美元 API 调用，一个周末的 GPU 时间"。

伴随变化：

  • RAG（检索增强生成） 作为主要用例。通过混合搜索检索到的文档块作为上下文提供给 LLM。RAG 输出质量直接依赖检索质量——推动了 LLM 团队对搜索的投入。
  • 使用交叉编码器重排序 （如 Cohere rerank、BGE-reranker），作为混合检索与最终结果之间的阶段——通常将前 100 名重排为前 10 名。
  • 后期交互模型（ColBERT） 生成逐词条嵌入，在查询时计算最大相似度之和。比单向量嵌入更具表达力，但存储和查询代价更高。部分先进生产架构使用（Vespa 原生支持）。

2026 年生产中的复合查询： (BM25 on title^3 + brand^2 + description^1) RRF-fused with (vector knn over title_embedding, ef=100) + filters on category, price, in_stock → top 200 → cross-encoder rerank → top 10。五年前这需要三个独立的研究项目；如今在 Elasticsearch 8.x 中一个查询体即可实现。

§10. 分析器、分词器与多语言搜索

分词是搜索中不起眼却至关重要的基础。错误的分析器会使其他一切（BM25、LTR、混合检索）变差，且 Bug 很隐蔽——查询无结果或将明显错误的文档排在前列，根本原因深藏于索引时文本切分为词条的方式。

10.1 分析器流水线——三个阶段

Lucene 分析器是对文本进行倒排前处理的流水线：

raw text  →  CHARACTER FILTER  →  TOKENIZER  →  TOKEN FILTERS  →  terms
  • 字符过滤器 ——对原始字符流进行处理。去除 HTML 标签，映射 &amp; 与 &，归一化 Unicode（NFC），折叠重音符号（é → e).
  • 分词器 ——将字符流切分为词条。Standard 分词器按 Unicode 边界（空白、标点）切分。其他分词器：仅空白、逐字符、基于正则、n-gram。
  • 词条过滤器 ——对词条流进行处理。转小写、去除停用词、词干提取（running → run）、同义词扩展（nyc → nyc + new york）、变音符折叠。

索引时和查询时使用的分析器必须产生兼容的词条。最常见的 Bug：索引文本被转为小写，查询却没有（或反之）——导致无匹配。始终在字段上设置 analyzer 并让其同时应用于两者，或显式分别设置 search_analyzer 并进行验证。

10.2 标准分析器与特定语言分析器

Standard — 基于 Unicode 词边界的分词器 + 小写过滤器。可作为任何拉丁字母语言的基础方案。不支持词干提取，因此 "running" 和 "ran" 不会匹配。

English — Standard 分词器 + 小写 + 英语停用词 + Porter 词干提取器（基于规则： running → run, running → run, connection → connect）。词干提取可能较为激进—— university 。 universe 两者均提取为 univers；通常是值得的权衡，但并非总是如此。

Snowball — 一系列词干提取器（Porter、KStem、Lovins），激进程度略有不同。KStem 较为保守（"只合并确定属于同一词的形态"）；Porter 较为激进。KStem 是电商场景的默认选择——对商品名称的误匹配更少。

Language-specific — 针对特定语言调优的分析器：西班牙语、法语、德语、俄语、阿拉伯语等。每种分析器都包含特定语言的停用词列表和词干规则。德语尤为重要，因为复合词（Donaudampfschifffahrtsgesellschaftskapitän）需要分解器才能找到 "Schiff" 或 "Kapitän"。

ICU（国际 Unicode 组件） — 具有更深层 Unicode 感知能力的分析器。支持亚洲语言、NFKC 规范化及区域特定大小写处理。非拉丁字母脚本的首选起点。

10.3 非拉丁字母脚本的分词

Lucene 的词边界分词器在词与词之间没有空格的语言上会失效：

中文。 词与词之间没有空白。可选方案：

  • 字符级 （长度为 1 或 2 的 n-gram）。每个字符作为一个 token 索引。简单，召回率高，精确度差——"中"出现在数百万篇文档中。
  • Jieba / IK / SmartCN 分词器。词典 + 统计模型将字符流切分为词语。标准中文索引使用 IK（内核分词）或 SmartCN。可用中文搜索的必要条件。
  • Bigram 分词器 （"CJKBigram"）。每对相邻字符作为一个 token。精确率与召回率平衡较好，无需维护词典。

日文。 类似问题但更复杂——三种字符体系（汉字、平假名、片假名）混合使用。 Kuromoji （内置 Elasticsearch 插件）是一个形态分析器，能识别汉字复合词和词形变化。若不使用 Kuromoji，"走った"（跑了）和 "走る"（跑）将被视为不同的 token。

韩文。 Nori 分词器（ES 6.4 起内置插件）。韩文会在词根上附加后缀；Nori 负责去除这些后缀。

泰语、高棉语、老挝语。 没有空格，也没有适用于所有语言的 Kuromoji 等效工具。ICU 分词器是备选方案；性能因语言而异。

阿拉伯语和希伯来语。 从右至左书写，变音符号通常被去除。ICU 能正确处理双向文本；阿拉伯语词干提取器（light10）有助于处理丰富的词形变化。

10.4 同义词

生产环境的 Elasticsearch/OpenSearch 集群由多种节点角色混合组成。每种角色承担不同的职责；在小型集群中同一虚拟机上混用角色是可行的，但在大规模场景下会出现问题。原因在于故障隔离：数据节点上的慢查询不应影响主节点上的集群状态操作。

（1）具备主节点资格的节点。

§13. 重新索引策略——别名切换模式详解

Lucene 中的映射是不可变的。你无法在实时索引上更改字段的分析器、类型或 doc_values 设置。在生产环境中演进 schema 的方式是 别名切换模式，§20.6（"疑难问题"章节）对此有简要介绍。本节深入讲解其机制。

13.1 为何映射是不可变的

段在写入时编码字段元数据。FST 词项字典、经分析的词项、doc values 列类型——所有这些都被固化在段文件中。更改分析器需要从 _source重新对每个文档进行分词。更改字段类型（从 keyword 与 text ）需要重新生成该字段的倒排索引。更改 doc_values 编解码器意味着重写整列。

Lucene 不支持原地重写。替代方案：使用新映射构建新索引，将数据复制过去，然后切换。

13.2 五步别名切换

Step 1: CREATE the new index with new mapping.
   PUT /products_v4
   { "mappings": { "properties": {
       "title": { "type": "text", "analyzer": "english_v2" },
       "title_embedding": { "type": "dense_vector", "dims": 768 }
   }}}
   Index is empty.

Step 2: START dual writes.
   Indexer publishes each doc to BOTH products_v3 AND products_v4.
   This catches all NEW edits from t=now forward.
   products_v4 is now catching up in real time.

Step 3: BACKFILL with _reindex (or external batch).
   POST /_reindex
   { "source": { "index": "products_v3" },
     "dest":   { "index": "products_v4" },
     "script": { "source": "..." }  // optional field transforms
   }
   Reindex API streams from old → new at controlled rate.
   Uses external_version so a slow backfill row cannot
   overwrite a fast live write of the same doc.

Step 4: VERIFY parity.
   Diff doc counts: GET /products_v3/_count vs /products_v4/_count
   Spot-check sampled docs by ID.
   Run query suite against both, compare top-K stability.
   This is the highest-risk gate; do not skip.

Step 5: SWAP the alias atomically.
   POST /_aliases
   { "actions": [
       { "remove": { "index": "products_v3", "alias": "products" }},
       { "add":    { "index": "products_v4", "alias": "products" }}
   ]}
   Both ops in one request = atomic. Zero downtime.

Step 6 (optional): SOAK then delete.
   Keep products_v3 read-able for 1–7 days in case rollback needed.
   Then DELETE /products_v3.

关键不变式：查询始终通过别名 products进行，而非版本化名称。生产代码永远不知道自己是否在访问 v3 或 v4。回滚只需反向执行一次别名切换调用——不到 1 秒即可回滚。

13.3 大规模 Reindex API

_reindex 支持限速。默认情况下，它使用内部滚动游标流式传输文档。使用 requests_per_second:

POST /_reindex?slices=auto&wait_for_completion=false&requests_per_second=5000

slices=auto 进行限速；在各分片间并行处理（每个源分片一个切片）。对于 100 个分片的源索引，这意味着 100 个并行重索引任务。 wait_for_completion=false 返回任务 ID；通过以下方式追踪 GET /_tasks/{id}.

1 TB 的索引在生产节流设置下通常需要 4–12 小时完成重建。瓶颈在于：源端磁盘读取吞吐量（使用 _source 直接读取，而非从 doc values 读取）、目标端合并开销（在回填期间对目标端设置 refresh_interval: -1 。 replicas: 0 在回填期间对目标端设置；完成后两者都需恢复）。

对于需要 来自其他来源的数据补充 （例如需要对每个文档调用嵌入服务），reindex API 提供了一个 script 字段，可对每个文档执行 Painless 脚本，但不支持在 Painless 中调用外部 HTTP 服务。替代方案：使用 Flink/Spark 编写自定义索引器，从主库读取、补充字段，再写入新索引。reindex API 仅用于引擎内部的复制操作。

  • 按日期标记: products-2026.05.22 ，按天滚动，适用于时序数据。
  • schema 版本 + 日期: logs-v3-2026.05.22。兼顾两者——更新 schema 版本（别名切换）与按天滚动（ILM rollover）。
  • 蓝绿颜色编码: products-blue, products-green。别名切换即为割接，回滚只需一个操作。

最重要的原则是： 生产代码通过别名读写，永远不使用原始索引名。禁止在应用中硬编码 DB 名称的模式。否则每次上线和回滚都会演变成多团队协调。

§14. 跨集群搜索与复制

当超出单个集群的限制——成本、影响范围、合规性数据本地化要求——就需要拆分为多个集群。跨集群搜索由 跨集群搜索（CCS）解决；跨集群复制由 Elasticsearch 中的 跨集群复制（CCR） ，或 OpenSearch 中的上游回放模式实现。

14.1 跨集群搜索

单次查询，扇出至 N 个集群。配置方式：

{
  "persistent": {
    "cluster": {
      "remote": {
        "us-east": { "seeds": ["us-east-coord-1:9300"] },
        "eu-west": { "seeds": ["eu-west-coord-1:9300"] }
      }
    }
  }
}

如以下查询 GET /us-east:logs-*,eu-west:logs-*/_search 会同时扇出至两个集群。每个远程集群在本地执行查询，并将各自的 top-K 结果返回给发起集群，由发起集群合并为全局 top-K。

适用场景：

  • 地理分布式部署：按地区建立索引（每个地区的数据因延迟和合规原因保留在本地）；全局查询（较少发生）跨地区扇出。
  • 租户隔离：每个租户独立集群，另设一个全局"跨租户搜索"平面用于管理/审计。
  • 跨集群冷热分离：最近 30 天使用热集群，30 天至 7 年归档使用冷集群。CCS 同时覆盖两者，用于偶发的历史查询。

核心成本是 网络：查询需通过跨地区链路访问远程集群，延迟和带宽开销随之累积。当远程集群不常被查询时，CCS 效果良好；若其成为主要查询路径，则效果变差。

14.2 跨集群复制

CCR 将主节点索引复制到一个或多个远程集群的从节点索引。从节点从主节点的 translog 中应用操作，延迟受网络和处理速度制约。

用于 灾难恢复：主集群位于 us-east-1，从节点集群位于 us-west-2。若 us-east-1 宕机，将 us-west-2 从节点提升为主节点，重新指向客户端，恢复时间约为数分钟。

还适用于 跨地区读扩展：写路径保留在单一地区，复制到其他地区的只读从节点。用户查询最近的从节点。从节点可能有数秒延迟，对大多数读取场景可接受。

注意事项：

  • Mapping 变更 会自动从主节点同步到从节点。
  • 集群间硬件差异 导致合并吞吐量不同；在写入负载较高时，从节点可能出现落后。
  • 冲突解决是单向的：主节点优先，从节点无投票权。对于多地区主动-主动写入，CCR 并不适用；需要应用层的冲突解决（如 CRDT 风格），并按实体路由写入。

OpenSearch 也有自己的跨集群复制实现，同样基于操作日志，语义几乎相同。

14.3 联邦化与集中化的选择

场景 联邦化 集中化
单集群规模超过 5 TB 或 200 个节点 是 否，优先纵向扩展
跨地区合规数据本地化 是 否
独立租户容量需求 是 否
绝大多数查询仅限于单一租户或地区 是 否
大多数查询跨租户或全局 否 是
运维简单性优先 否 是

联邦化的代价是真实存在的：集群越多，意味着需要管理更多 master 节点、快照策略、告警仪表板和值班复杂度。切勿过早联邦化。

§15. 安全——多租户、字段级安全与许可证故事

在 SaaS 或企业场景下，搜索引擎必须强制执行"谁能看到什么"。引擎负责回答"与查询匹配的文档"，而确保用户被允许查看这些文档是安全层的职责。

15.1 默认状态——无安全保障

开源 Lucene 和原生 OpenSearch 均不内置认证。原版 Elasticsearch 在 7.1 之前同样如此；现版本默认启用安全功能。将未受保护的引擎暴露于公网是典型错误——许多公司因未受保护的 ES 集群而泄露了客户数据。

生产环境至少必须具备：

  • 认证 ——基本认证、JWT、双向 TLS、SAML、OIDC。集群能够识别调用者身份。
  • 授权 ——基于角色的访问控制（RBAC）。角色映射到集群权限（读取某索引、写入另一索引）。
  • TLS 用于所有节点间和客户端与节点间的通信。

15.2 字段级安全

不同角色看到同一文档的不同字段。例如：一个 users 索引包含字段 name, email, ssn, phone。客服可以查看 name, email, phone；工程师可以查看 name, email；只有欺诈运营人员可以查看 ssn.

按角色配置：

{
  "role": {
    "indices": [{
      "names": ["users"],
      "privileges": ["read"],
      "field_security": {
        "grant": ["name", "email", "phone"],
        "except": ["ssn"]
      }
    }]
  }
}

引擎会在 _source 返回前去除不允许的字段。注意：仍可能通过聚合泄露——如果用户可以对 ssn字段做聚合，桶结果会暴露 SSN 值。字段级安全必须结合限制可聚合字段一起使用。

15.3 文档级安全

不同角色在同一索引中看到不同文档。实现方式是引擎自动应用一个查询过滤器。例如：在多租户 SaaS 中，每个文档有一个 tenant_id 字段，每个用户只能查看本租户的文档：

{
  "role": {
    "indices": [{
      "names": ["data"],
      "privileges": ["read"],
      "query": "{\"term\":{\"tenant_id\":\"{{_user.metadata.tenant}}\"}}"
    }]
  }
}

来自租户 acme_corp 的用户的每次查询都会隐式附加 AND tenant_id = "acme_corp" 过滤器。即使使用恶意构造的查询，用户也永远无法看到其他租户的数据。

运行时查询可以非常复杂——检查白名单、从元数据字段计算每用户权限、通过 runtime fields 与外部规则引擎联合查询。权衡点：复杂的文档级过滤会拖慢该用户的每次查询。

多租户模式：

  • 共享索引 + 文档级安全：成本最低、最简单，但影响范围最大。一个调优不当的查询会拖慢所有人。
  • 每租户独立索引：隔离更强，但集群状态膨胀（每个租户一个索引 = 大型 SaaS 中数千个索引）。
  • 每租户独立集群：隔离最强，成本最高。

大多数 SaaS 选择共享索引+文档级安全，直到某个租户规模大到值得拥有独立索引或集群。

15.4 开源版、Elastic Cloud 与 OpenSearch 分叉的故事

2021 年，Elastic 将 Elasticsearch 的许可证从 Apache 2.0 改为 SSPL + Elastic License 双重授权（非 OSI 认可）。导火索是 AWS 以托管服务形式转售 Elasticsearch 却未回馈贡献；Elastic 希望阻止这一模式。AWS 随即将最后一个 Apache 2.0 版本（7.10）分叉为 OpenSearch，并自行维护路线图。

当前格局（2026 年）：

  • Elasticsearch ——Elastic 的产品，双重授权（免费 Basic 版涵盖大多数用途；付费商业版包含跨集群复制、机器学习异常检测、高级安全等 x-pack 功能）。可在 Elastic Cloud（Elastic 的托管服务）上使用。
  • OpenSearch ——AWS 的 Apache 2.0 分叉。最初与 ES 7.10 功能相近，后来发展出独立功能（异常检测、可观测性、OpenSearch Dashboards）。可在 Amazon OpenSearch Service 上使用。
  • Apache Lucene ——未变，Apache 2.0，底层库。ES 和 OpenSearch 均基于它构建。

各公司的选择：

  • 全新项目，无 Elastic 合作关系：默认选择 OpenSearch。Apache 2.0 许可证明确、完全开源，在 AWS 上规模化成本更低。
  • 现有 Elastic 客户：继续使用 ES，升级路径更顺畅，x-pack 功能价值高。
  • 离线或自托管，追求自主可控：两者皆可；OpenSearch 是 Apache 2.0 下更安全的选择。

实用建议："OpenSearch 是 2026 年默认的开源选择，若已付费使用 Elastic，ES 也是强有力的选择。"了解许可证变更的存在，但无需过多纠结。

2024 年 Elastic 将部分组件重新授权为包含 AGPL3 的三重许可证——情况仍在变化中。OpenSearch 仍是安全的默认分叉选择。

§16. 运维——快照、监控与堆内存压力处置手册

生产环境的搜索集群全年 24/7 运行多年。运维是没人会写博客的部分，也是区分一个稳定集群与一个每周凌晨 2 点报警的集群的关键所在。

16.1 快照到 S3

快照/恢复子系统将索引的增量快照写入对象存储。仓库注册方式：

PUT /_snapshot/s3_repo
{
  "type": "s3",
  "settings": {
    "bucket": "company-es-snapshots",
    "region": "us-east-1",
    "compress": true,
    "base_path": "prod-cluster/"
  }
}

快照在 segment 级别是增量的。首次快照上传所有 segment 文件；后续快照仅上传自上次以来的新 segment。由于 segment 是不可变的（写入后不再修改），增量正确性显而易见——segment 要么已在 S3 中（跳过），要么是新的（上传）。

在多 TB 集群上，典型增量快照需要 1–10 分钟，首次全量快照需要数小时。通过 SLM（快照生命周期管理）策略进行调度：

{
  "policy": {
    "name": "<daily-snap-{now/d}>",
    "schedule": "0 30 1 * * ?",
    "repository": "s3_repo",
    "config": {
      "indices": ["*"],
      "ignore_unavailable": true,
      "include_global_state": true
    },
    "retention": {
      "expire_after": "30d",
      "min_count": 5,
      "max_count": 50
    }
  }
}

恢复时间取决于数据量。10 TB 集群从 S3 恢复在典型 S3 到 NVMe 带宽下需要 4–8 小时。恢复单个索引更快——只拉取该索引的文件。

仓库插件：S3（AWS 默认）、GCS（Google Cloud）、Azure（Azure）、FS（共享 NFS）、HDFS（本地部署 Hadoop 生态）。所有插件均实现相同的增量 segment 级别模式。

16.2 监控——四个黄金信号

每个搜索集群都需要针对四类指标设置告警：

（1）集群健康状态。 _cluster/health 返回状态 green （所有主分片和副本分片已分配）、 yellow （主分片已分配，部分副本分片缺失）、 red （部分主分片未分配）。Yellow 状态在恢复期间可暂时维持；Red 状态意味着在分片恢复前用户可见的数据丢失。

（2）分片分配。 分片是否在节点间均衡？是否有某节点持有的分片数量是平均值的 2 倍？分配是否因磁盘水位线停滞（cluster.routing.allocation.disk.watermark.low/high/flood_stage)?

（3）JVM 堆内存压力。 各节点的 JVM 堆内存使用率。 目标：老年代 GC 后堆使用率不超过最大值的 75%。 持续超过 85% 时，GC 暂停延长，查询超时。数据节点堆使用率达到 75% 的处置手册：

  • 检查 fielddata 缓存： GET /_nodes/stats/indices/fielddata。若较高，设置 indices.fielddata.cache.size 加以限制。
  • 检查 segment 数量：频繁 refresh 会产生过多小 segment，每个 segment 都有固定堆开销。放慢 refresh 间隔。
  • 检查查询模式：深度分页（from: 10000）会将所有结果缓冲到堆中。切换至 search_after.
  • 检查聚合：大量 cardinality 聚合和 terms 与 size: 100000 会大量消耗堆内存。设置合理的 size 限制。
  • 最后手段：增加数据节点以分散负载。

（4）GC 暂停。 10 秒的老年代 GC 暂停 = 10 秒的查询停顿。通过 GET /_nodes/stats/jvm监控。每分钟暂停超过 1 秒为警告；超过 5 秒需要人工介入。现代 G1GC（ES 7+ 起默认）基本自调优；自 ES 8 起，ZGC 作为低暂停工作负载的选项，暂停时间低于 10ms，但 CPU 开销更高。

16.3 "数据节点堆使用率达到 75%"处置手册

凌晨 2 点告警响起：三个数据节点上的 jvm.mem.heap_used_percent > 85 。排查步骤：

Step 1. Quick stabilize.
  - Throttle ingest if writes are spiking: pause one Kafka consumer group.
  - Disable expensive cron jobs (large _reindex, _update_by_query).
  - Lower refresh interval doesn't help in the moment — segments are
    already created.

Step 2. Diagnose. Hit /_nodes/stats?metric=jvm,indices.
  - field_data.memory_in_bytes? If high, someone aggregated on a `text`
    field (uses fielddata, evil). Fix: change to keyword, or block such
    aggregations with `indices.fielddata.cache.size: 10%`.
  - segments.count high per node? Too many small segments. Cause: too
    aggressive refresh, or merge stuck. Check merge stats; consider
    bumping merge scheduler threads.
  - circuit_breakers tripping? Engine is preventing OOM at the cost of
    failed requests.
  - Per-shard memory: which index is the hog? Often a single misbehaving
    index dominates.

Step 3. Mitigate.
  - Force-merge specific old indices (NOT live-writing ones) to reduce
    segment count.
  - Roll a node restart on the worst offender (full heap reclaim).
  - Move shards off the worst nodes via temporary allocation rules.

Step 4. Post-mortem and structural fix.
  - Did mapping explosion cause the heap pressure? (§18)
  - Was someone aggregating on text fields? Fix the application.
  - Was bulk index too aggressive? Cap bulk size in the indexer.
  - Are we sized correctly? If average heap is consistently >70%,
    add capacity.

结构性修复比临时处置更重要。一个每晚都达到 85% 堆使用率的集群，迟早会遇到严重故障。

16.4 分片分配洞察

GET /_cluster/allocation/explain 是分片拒绝分配时的标准调试工具。它会返回结构化原因：磁盘水位线超标、感知策略违规、节点故障历史、决策器拒绝。典型输出示例：

  • "no_attempt: no allocation can be performed because of forced awareness constraints (zone)." 在缺失的可用区添加节点。
  • "the shard cannot be allocated because of disk usage above the high watermark." 迁移分片或扩展磁盘。
  • "the node is currently throttled with concurrent recoveries." 等待或提高 cluster.routing.allocation.node_concurrent_recoveries.

Red 状态的集群几乎都可以通过查看 allocation/explain 某未分配分片的信息来解决。

§17. 成本优化——Lucene 的存储成本及应对策略

Lucene 的倒排索引、doc values、stored fields 和 FST 合计占用 约为原始 JSON 大小的 3 倍磁盘空间。1 TB 的 JSON 文档会变成 3 TB 的 Lucene 索引。细分如下：

  • Stored fields（_source)）：原始 JSON，LZ4 压缩。约为 0.5–1× 原始 JSON 大小。
  • 倒排索引（词项 + 倒排列表 + 位置信息）：约为 0.5–1× 原始文本大小。
  • Doc values（按字段列式存储）：约为 0.3–0.6× 原始结构化数据大小。
  • Norms、词向量、segment 元数据：少量固定开销。

倍数随索引字段数、存储字段数、是否保留位置信息而变化。激进优化可降至 1.5×；默认设置约为 3×。

17.1 source 过滤

默认情况下， _source 包含完整的原始 JSON。对于一个有 200 个字段而只展示 10 个字段的文档，这是一种浪费。两种精简方式：

（a） source_includes 查询时过滤 ——仅返回指定字段：

{ "_source": { "includes": ["title", "price"] }, ... }

节省带宽，但不节省磁盘。

（b） _source 索引时字段过滤 ——在存储时排除某些字段：

{ "_source": { "excludes": ["raw_html", "log_blob"] }, ... }

节省磁盘空间。权衡：这些字段无法原地重建索引（没有源数据可读）——必须从主库重新获取。对于纯噪声的日志字段来说是可接受的。

17.2 禁用 _source 用于仅日志数据

最激进的优化： _source: { enabled: false }。原始 JSON 完全不存储。倒排索引和 doc values 保留——搜索和聚合仍然有效。权衡：

  • 无法检索原始文档。 你只能看到命中结果的 ID、评分以及任何显式指定的 store: true 字段，但看不到文档主体。
  • 无法从该索引重建索引。 未来的 schema 变更需要从源头重新导入数据。
  • 高亮显示需要 store: true 在特定字段上 （摘要提取需要文本内容）。

对于纯日志搜索（源头是应用程序，且永远不会回查 ES 获取源数据，文档主体也没有展示路径），禁用 _source 可减少约 30–50% 的磁盘占用。Datadog 及类似的日志型工作负载均采用此方式。

17.3 对冷层强制合并以减少段数

一个 50 GB 的索引在热层可能有 20 个段。每个段都有固定开销（堆中的 FST tip、文件句柄、元数据）。在温层/冷层静止时，不需要快速写入——将其合并为单个段：

POST /logs-2025.04.15/_forcemerge?max_num_segments=1

权衡：需要数分钟到数小时，垄断 I/O。在低峰期执行；每次一个分片。始终在 ILM 温层/冷层过渡时执行；切勿在有实时写入的热层执行。

效果：

  • 堆内存使用下降（单个 FST tip 而非 20 个）。
  • 查询延迟下降（需访问的段更少）。
  • 磁盘使用下降 10–30%（较大段上压缩效果更好；tombstone 开销更少）。
  • 合并期间新写入被阻塞。

17.4 Codec 选择

Lucene 支持多种 codec。默认的 LZ4 对存储字段的压缩速度快且效果不错。切换到 best_compression （DEFLATE）：

{ "settings": { "index": { "codec": "best_compression" }}}

将 _source 大小减少 40–60%。检索时查询延迟增加 10–30%（解压缩开销）。适合日志工作负载；不适合低延迟电商场景。

除默认 codec 外，Lucene 9.10+ 支持 zstd（解压缩比 DEFLATE 更快，压缩比相近）以及按字段选择 codec（doc_values_format）。字段级别的专项优化是团队压榨最后 10–20% 空间的方式。

  • 每条 profile 原始 JSON：约 3 KB。总计：3 TB 原始数据。
  • 默认 ES：约 9 TB。以 NVMe SSD $300/TB-月计，每月 $2700/副本。3 个副本：$8100/月。
  • 时这不应发生——leader 在 ISR 完成复制前不会 ack。使用 _source 排除原始文本字段（仅保留可展示字段）：降至约 6 TB。3 个副本每月 $5400。
  • 时这不应发生——leader 在 ISR 完成复制前不会 ack。使用 best_compression 对于 _source：降至约 5 TB。$4500/月。
  • 通过 ILM 将 30 天未活跃的 profile 迁移到温层：热层 1 TB，温层 5 TB。(1 × $900 + 5 × $300) = $2400/月。
  • 通过 ILM 冷层将不活跃 profile 存储在 S3 上：热层 0.5 TB，温层 1.5 TB，冷层 4 TB。($450 + $450 + $200) = $1100/月。

总成本降低约 8 倍 在不改变查询语义的情况下实现。仅使用热层是最懒也是最贵的选项；激进的 ILM 策略才是省钱的所在。

§18. mapping 爆炸问题

这一问题在原始"疑难问题"章节中只是简要提及；此处进行完整深入解析，因为它是生产搜索集群让值班工程师半夜被叫起来的典型原因。

18.1 什么是 mapping 爆炸

搜索引擎会物化每个字段的元数据。对于每个唯一字段名，集群状态中存储：字段类型、分析器、doc values 设置、是否被索引、是否存储、是否有 term vector。集群状态会被复制到每个节点；每次变更都会广播出去。

如果你的导入管道允许来自不可信输入的任意字段名——用户提交的 JSON、带有结构化但不受控 key 的日志行、你无法控制的第三方 API——你可能会引爆这一问题。

18.2 一个真实的故障时间线

Day 1.  A new microservice deploys. Its logs include fields like
        attr.user_id_3f7b29a1: "..."
        attr.session_abc: "..."
        ...one new field name per call.

Day 1 evening.  Cluster has 5000 unique fields in the logs index.
                Mapping is 200 KB. No alerts.

Day 3.  100k unique fields. Mapping is 5 MB. Cluster state propagation
        slows. Background "publish cluster state" tasks take seconds
        instead of milliseconds. Coord nodes start logging warnings.

Day 5.  300k fields. Cluster state is 20 MB. Every node holds a copy;
        every change rebroadcasts. Heap pressure rising on data nodes
        (each new mapping change allocates a new full cluster state on
        the JVM heap; old ones are GC'd; allocation pressure spikes).

Day 7.  500k fields. Master OOM during a routine snapshot operation
        that tries to serialize the full cluster state. Master dies,
        election fires, new master tries to do the same thing, also
        OOMs. Cluster red. All ingest stopped.

Recovery.  Manually clean up: delete the offending indices, force-merge
           remaining cluster state, restart cluster. Hours of downtime.
           Postmortem reveals the bad ingester.

这种模式太常见了，每个搜索运维团队都有类似的故事。

18.3 缓解措施

(1) dynamic: strict 在接受外部数据的索引上使用。 带有未知字段的文档在导入时被拒绝。导入程序看到错误，你去排查。没有静默扩展。

{ "mappings": { "dynamic": "strict", "properties": { ... } }}

缺点：每个新的合法字段在文档流入之前都需要更新 mapping。如果你的 schema 稳定且变更经过协调，这是可行的；但如果开发者频繁添加字段，则会很痛苦。

(2) dynamic: false. 新字段被接受进入 _source 但不被索引。对其搜索会静默返回空结果；它们是不可见的。适用于想保留原始数据但从不搜索的场景。

(3) 动态模板。 定义新字段自动映射的规则：

{
  "mappings": {
    "dynamic_templates": [
      { "tags_as_keyword": {
          "match_mapping_type": "string",
          "match": "tag_*",
          "mapping": { "type": "keyword", "ignore_above": 256 }
      }},
      { "metric_as_float": {
          "match": "metric_*",
          "mapping": { "type": "float" }
      }}
    ]
  }
}

允许符合模式的新字段；给予它们一致的设置。结合 total_fields.limit 来限制爆炸半径。

(4) index.mapping.total_fields.limit （默认 1000）。字段数量的硬上限。达到上限后，新字段被拒绝。根据你实际的 schema 大小加上余量来调整（一个 200 字段的 schema 可以将上限设为 500）。从第 1 天起就在每个索引上设置此项。

(5) 扁平化字段。 对于真正自由形式的键值数据，使用 flattened:

{ "properties": { "attributes": { "type": "flattened" }}}

一个文档 { "attributes": { "user_id": "abc", "session": "xyz" }} 将键和值都索引到单个内部字段中。通过 attributes.user_id: abc进行搜索。权衡：对子字段的评分表达能力较弱，没有按 key 的分词。适合可变的用户提交 JSON。

(6) 结构化 + 自由形式的双索引模式。 稳定字段放入具有严格 mapping 的普通索引。自由形式数据放入带有 flattened 或 dynamic: false的独立索引。需要两者的查询在应用层进行关联。

经验教训： 永远不要允许不可信输入控制字段名。 始终设置上述任一防护措施。如果你在运营一个 SaaS 日志搜索服务，客户可以提交任意字段，你需要强制执行每个客户的字段数量限制，超出时拒绝。

§19. 超越 Lucene 的专用检索引擎

Lucene 系列（ES、OpenSearch、Solr）覆盖了约 80% 的搜索工作负载。其余 20%——极端规模、特殊分词或极端成本要求——会抛弃 Lucene。当前的替代方案全景：

19.1 Vespa——排名与检索合二为一的引擎

Vespa 是 Yahoo 的内部搜索/广告投放平台，于 2017 年开源。其关键架构差异点： ML 模型在引擎内部运行。Vespa 具有原生 tensor 类型，支持引擎可直接用于评分的运算（矩阵乘法、ReLU、attention）。一个查询计划可以在数据节点上一次完成检索（找到候选）和排名（用学习模型对其评分），无需外部服务调用。

Query → match-phase (BM25 + filters, in-engine)
     → first-phase ranking (cheap function, top 2000)
     → second-phase ranking (heavy model evaluating tensors, top 10)
     → return

每个阶段都在持有文档的内容节点上运行。不需要往返模型服务器。对于延迟预算小于 100ms 的 ML 密集型工作负载，这是决定性的——Yahoo Mail、Yahoo Shopping 广告排名，以及越来越多的 RAG 检索加重排场景，适用于那些需要比 ES + 旁路重排服务更低延迟的公司。

权衡：生态系统较小，学习曲线更陡，托管服务可用性较少（Vespa Cloud 是主要托管选项）。运维需要比 ES 更多的 Vespa 专项知识。

何时从 Lucene 切换到 Vespa：排名即产品、延迟预算低于 100ms、模型较重（cross-encoder、神经重排器或基于 tensor 的评分），且团队愿意投入 Vespa 专项知识。不是一次随意的升级。

19.2 Quickwit——Rust、S3 原生、低成本日志搜索

Quickwit 基于 Tantivy（Lucene 索引格式的 Rust 移植版）构建，并有两个架构创新：

  1. 计算与存储分离。 索引存储在 S3（或任何对象存储）上，而非本地磁盘。查询通过预取和缓存直接从 S3 读取。没有持有热数据的"数据节点"；节点是无状态的。
  2. 专为高摄入量、时间序列场景构建。 流式摄入是一等公民。索引按时间桶分区；旧桶是廉价的 S3 对象。

结果：日志搜索存储成本 $20/TB-月（S3），而非 $300/TB-月（NVMe）。对于保留期主导成本的工作负载——Binance、BlaBlaCar、大型可观测性厂商——Quickwit 在等效保留期下比 ES 便宜 5–10 倍。查询延迟更高（S3 获取耗时数百毫秒而非微秒级页面缓存），但对于查询频率由工程师驱动而非面向用户的日志搜索，这个权衡是可接受的。

何时从 Lucene 切换到 Quickwit：成本主导的多 PB 保留期日志搜索。不适用于面向用户的低延迟搜索（S3 获取的延迟预算太紧）。

19.3 Tantivy——Rust 版 Lucene 等价实现

Tantivy 是从头用 Rust 实现的 Lucene 索引格式。相同的 FST term 字典、相同的 PFOR-Delta posting 压缩、相同的段合并生命周期。区别在于实现语言：没有 JVM，没有垃圾回收，每个索引的内存开销更低。

Tantivy 是一个库，而非分布式服务。它为以下项目提供支持：

  • Meilisearch ——基于 Tantivy 的站内搜索引擎，运维比 ES 更轻量，在不足 1 亿文档规模下 p99 低于 50ms。
  • Quickwit ——基于 S3 的分布式 Tantivy（见上文）。
  • 在 Rust 应用中嵌入自定义搜索 。

何时从 Lucene 切换到 Tantivy：在 Rust 应用中嵌入搜索（进程内，无 JVM 税），或在 ES/OS 感觉过重的较小规模下运营。不适合在规模化生产分布式搜索中直接替换。

19.4 Zoekt——Google 代码搜索

Zoekt 在 Google 内部为代码搜索而构建；现在为自托管用户的 Sourcegraph 提供支持。差异点： 基于 trigram 的源代码倒排索引，专为子字符串搜索（function, getUserByEmail(）和正则表达式设计。

Lucene 的词级分词对代码来说是错误的。 getUserByEmail 对 Lucene 来说是一个 token（或使用驼峰命名分词器分割成 get, User, By, Email ——对子字符串搜索同样是错误的）。Zoekt 索引每个重叠的 3 字符序列： get, etU, tUs, Use，... 正则查询通过 trigram 交集获取候选，然后对匹配文件验证正则。

何时从 Lucene 切换到 Zoekt：用户需要子字符串和正则匹配的代码搜索。不适用于词级分词正确的英语文本。

19.5 GitHub Blackbird

Blackbird 是 GitHub 从头构建的全局代码搜索引擎，取代了基于 Lucene/Solr 的前身。 2000 亿文档，640 TB 热索引，子字符串查询 p99 低于 2 秒 ，覆盖所有公开 GitHub 内容。

架构说明（来自 GitHub 工程博客）：

  • 自定义 n-gram 倒排索引（概念上类似 Zoekt，但达到星球级规模）。
  • 按仓库哈希分片。
  • 与 Git 对象格式紧密耦合——直接索引 Git 对象而非提取的文件。
  • 用 Rust 构建，以实现内存效率和尾延迟控制。

非开源。经验教训：当 Lucene 的词级模型对你的工作负载从根本上是错误的，且你处于千亿规模时，从头构建引擎是合理的。大多数团队不会到达那个规模。

19.6 何时离开 Lucene——决策框架

Default                           → ES / OpenSearch (Lucene)
Ranking is the product, low-latency ML rerank → Vespa
Log search at PB-retention, cost dominates    → Quickwit / Loki
Embedded search in Rust app                   → Tantivy / Meilisearch
Code search (substring, regex)                → Zoekt / Blackbird
Label-filtered logs, grep-friendly            → Loki
Hosted small-to-mid site search               → Algolia / Meilisearch

2026 年，绝大多数新搜索项目仍然选择基于 Lucene 的引擎。专用引擎的存在，是因为有人遇到了瓶颈——通常围绕成本、分词或引擎内排名——而默认方案在其规模下无法解决。

§20. 该技术固有的疑难问题

20.1 相关性 vs 新鲜度——过时信号污染实时检索

问题。 多个排名信号以不同的节奏刷新：关键词（约 1s）、过滤字段（相同）、向量 embedding（批处理，数小时）、LTR 特征（在线，秒级到分钟级）。混合使用会产生不一致的结果。

示例——电商。 某商家将一瓶水的价格从 $35 降至 $19。关键词索引立即看到变更。向量 embedding（离线基于描述计算）未发生变化，因此该商品的排名与打折前相同。结果：折扣进入了索引，但在下一批次处理之前不会体现在排名中。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 将 refresh_interval 缩短至 100ms 并寄希望于此。 为什么行不通：100ms 刷新会炸掉合并过程（§4.4）——每小时 36000 个段，合并跟不上，查询降级——而且 embedding 仍然 仍在 过时。

真正的解决方案。 按信号分解新鲜度：

  • 关键词/过滤字段新鲜度：与 refresh_interval绑定。1–5 秒。
  • 向量 embedding 刷新：将新编辑的文档路由到 流式 embedding 服务，该服务将新 embedding 推送到同一索引路径。流式延迟时回退到批处理。
  • LTR（Learning to Rank）特征：由实体 ID 作为 key 的独立在线特征存储。重排器在查询时拉取新鲜特征，而非从索引中读取。将模型新鲜度与索引新鲜度解耦。

20.2 扇出成本的分片策略——更多分片并不总是带来更好的性能

问题。 索引增长导致每个分片大小超过 50 GB。本能反应是增加分片数量。

示例——可观测性日志搜索。 每日日志量从 50 TB 增长至 500 TB。每日索引从 10 个分片 × 5 TB 变为 100 个分片 × 500 GB。查询 p99 翻倍。原因：每次搜索扇出到 100 个分片；尾延迟概率 1 - (1-p)^100；每个分片有 1% 的概率超过 50ms → 整体有 63% 的概率突破 50ms。

真正的解决方案。 路由分区器 以限制扇出。日志搜索按 (tenant, hour) 分区（扇出从 100 降至 5–10）。Profile 搜索按搜索者的网络集群路由（"搜索我的人脉"查询命中约 10 个分片而非全部 100 个）。电商按 region 或 category 路由（如有）。

替代方案： 双层（热/冷） 索引。最近活跃的实体在小型热层集群（5–10 个分片）。其余在大型冷层（100 个分片）。90%+ 的查询由热层服务。Galene 和 Earlybird 的模式。

20.3 热键与热查询

问题。 少数词项出现在大量文档（或查询）中。对每个匹配项进行评分代价极高。

示例——代码搜索。 用户在 Sourcegraph 中输入 function 。该词项出现在数千万个文件中。朴素评分会触及所有文件。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 为"function"缓存前 20 个结果，TTL 为 60s。 为什么行不通：个性化使缓存失效（每个用户根据仓库和权限获得不同结果），且在索引重建时缓存会滞后。

真正的解决方案。 不要对所有内容评分。 基于排序索引的提前终止：维护按 静态先验 （新近度、仓库热度、stars）排序的 posting 列表。对于高召回查询，每个分片只对前 M = 10000 个文档评分。通过 WAND 或 Block-Max WAND 证明超过 M 的文档无法超过当前前 K 名。WAND 利用每个词项得分的上界来跳过最大可能分数低于当前阈值的文档。对于"function"，可能意味着对 5 万个文档评分而非 5000 万。

对于非个性化聚合（"有多少文件包含 function"），缓存 计数，而非结果列表。

20.4 mapping 爆炸——一个有问题的导入器字段拖垮整个集群

问题。 搜索引擎会物化每个字段的元数据。不可信输入导致字段数量爆炸，会炸掉集群状态。

示例——可观测性日志搜索。 某新微服务以 attr.<random_uuid>: <value>为字段名发送日志。每行引入一个新字段名。一周后，集群有 50 万个唯一字段。引擎在段元数据中跟踪每个字段；在每个 master 和数据节点间 gossip 传播的集群状态增长到数百 MB；master OOM；集群变红。

真正的解决方案。 "dynamic": "strict" 在接受第三方数据的索引上使用——在导入时拒绝未知字段。对于真正自由形式的键值数据，使用 flattened 字段类型：整个子对象被索引为单个字段，查询时保留 key。设置 index.mapping.total_fields.limit （默认 1000）并在超出时 大声报错 。

20.5 最终一致性 vs 主库——静默偏差

问题。 索引与主库存储产生偏差，直到用户投诉才被发现。

示例——电商库存。 某商品售罄。目录 DB 提交。CDC 事件进入 Kafka。索引器拉取。批量写入以 429 失败（请求过多）。索引器重试；成功。在失败窗口期间，索引提供了过时的"有货"结果；用户下了订单，库存系统随后拒绝了这些订单。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 重试直到被接受；监控索引器延迟。 为什么行不通：至少一次 + 非幂等写入 = 重复或乱序（重试可能覆盖 更新的 编辑），且静默偏差在引擎内部无法被检测到。

真正的解决方案。 幂等批量写入 通过 external_version 设置为源 LSN——引擎拒绝乱序写入。 定期对账：每小时对 0.1% 的 ID 进行采样，偏差超过 0.01% 时告警。将引擎视为严格的物化视图：需要权威数据的功能从主库读取。业务不变式功能（库存、隐私）在 API 层通过实时主库读取来强制执行，而非基于索引字段过滤。

20.6 不停机批量重建索引——别名交换模式

问题。 Mapping 只能向前演进。更改分析器（添加同义词、切换语言分词器）需要重建。

示例——多语言商品搜索。 某零售商在原本仅支持英语的目录中添加了日语。日语分词器（Kuromoji）需要重新分词所有文档。"停止写入"在生产环境中不是一个选项。

真正的解决方案。 使用别名交换的两阶段重建：

  1. 创建 products_v4 使用新分析器。为空。
  2. Indexer 双写（dual-writes） 与 products_v3 。 products_v4 同时进行。
  3. 回填（Backfill） products_v4 从主节点使用批量 indexer 回填。使用 external_version 因此过期的回填记录不会覆盖更新的实时编辑。
  4. 回填完成且延迟归零后， 切换别名（alias）: products → products_v4。原子操作，无停机。
  5. 浸泡测试结束后（几天），删除 products_v3.

同样适用于 LinkedIn 个人资料 schema 迁移、GitHub 仓库重索引、日志滚动以及 Booking.com 分面演进。

§21. 故障模式演练

21.1 写入过程中分片主节点宕机

主节点将 translog 操作 fsync 到磁盘后宕机，尚未转发给副本。Master 通过心跳检测到故障，提升一个同步副本。客户端要么已收到 ack（主节点崩溃前已完整转发），要么未收到（未收到 ack，重试）。使用幂等 external_version，重试是安全的。 持久性要点：对每个同步副本执行 translog fsync；主节点须等待副本 fsync 后才发送 ack。

21.2 写入间隔期间分片主节点宕机

网络抖动导致主节点心跳丢失 30 秒；master 降级该主节点，提升副本。被降级的主节点重新加入后，向新主节点查询当前 seq_no 水位线（watermark）。重放本地操作中 seq_no > watermark 的部分；幂等操作为空操作（no-op）。恢复为副本角色。 持久性要点: seq_no 每个分片单调递增，存储在 translog 中。

21.3 Master 宕机——2 master 集群中的脑裂问题

两个候选 master 节点发生网络分区：各自选举自己为主；集群状态冲突；分片分配出现分歧。 根本解决方案——quorum（多数派）。 始终运行奇数且 ≥ 3 个节点。选举配置要求多数派。1 对 2 的分区导致少数派无法选举，只读直至网络恢复。现代 ES Zen 2 本质上是集群状态的 Raft 实现；Solr 使用 ZooKeeper。ES 7 之前， discovery.zen.minimum_master_nodes = (N/2) + 1 必须显式设置——这是经典的脑裂 bug。

21.4 批量索引在批次中途失败

500 条文档的批量请求返回 HTTP 200，附带每条文档的状态：487 条成功，13 条返回 409 version conflict。冲突属于 预期行为 ——乱序 CDC 配合 external_version 低于当前值。记录日志，递增计数器，继续处理。不要重试。

若 13 条出现 503：针对这些文档使用指数退避单独重试。通过 external_version.

若整个批量请求超时：重新发送整批；版本检查会拒绝已应用的文档。

持久性要点。Kafka offset 在 after 成功批量后提交。提交前崩溃 = 下一个消费者从上一个 offset 开始，重放消息。Kafka offset + 引擎版本冲突 = 恰好一次（exactly-once）应用。

21.5 分片永久丢失

分片 42 的三个副本全部位于同一故障域（机柜、AZ——可用区），遭遇火灾。恢复方式：有快照时，恢复后从快照时间戳重放 Kafka。无快照时，从主数据源对分片 42 的文档范围执行全量重索引—— hash(entity_id) mod K 告知哪些 ID。需要数小时，但有上限。可恢复，因为索引是 派生数据。预防措施：反亲和性规则；绝不将同一分片的主节点与副本放置在同一主机、机架或 AZ 上。

21.6 级联 merge 风暴

对一个 30 个分片的线上索引意外执行 force-merge；每个分片试图合并为 1 个 segment；磁盘 I/O 饱和；p99 从 80ms 升至 8s。取消 force-merge 任务（中途无法干净中止，但新的 merge 工作会停止）；限流 indices.merge.scheduler.max_thread_count. 预防措施：不对线上索引执行 force-merge。仅对已停止写入的滚动索引（ILM 管理——Index Lifecycle Management）执行。

§22. 为什么不直接用 SQL LIKE 或 Postgres FTS

22.1 LIKE '%term%' ——直接否决

前缀通配符（Leading-wildcard） LIKE 无法使用 B-tree。全表扫描：1 亿行 × 2 KB = 200 GB，~500 MB/sec，约需 7 分钟/查询。不可行。

22.2 Postgres tsvector 配合 GIN——更值得探讨

Postgres 通过 GIN 提供真正的倒排索引。

CREATE INDEX idx ON products USING gin(to_tsvector('english', title || ' ' || description));
SELECT * FROM products
WHERE to_tsvector('english', title || ' ' || description) @@ to_tsquery('stainless & bottle')
ORDER BY ts_rank(...) DESC LIMIT 20;

适用 于 ~500 万到 1000 万条文档、单个 Postgres 节点、简单查询场景。

在 1 亿条以上文档时出现问题： （1）无原生水平分片——Citus 可行，但跨分区 ts_rank 较为繁琐。（2）无 BM25—— ts_rank 更简单；长度归一化和 TF 饱和度很重要。（3）分析器扩展性差——自定义分析器、edge-n-gram、多语言分词不是一等公民。（4）查询时无分面统计—— GROUP BY 在 2 亿候选文档后很慢。（5）无混合向量 + 关键词——pgvector 存在，但融合 BM25 + 向量 + LTR 特征比较笨拙。（6）GIN 更新代价高—— fastupdate 在大量写入下会停顿；5000–10000 次写入/秒时存在 autovacuum 压力和表膨胀。（7）无 NRT（Near-Real-Time）刷新控制。（8）无 WAND——从 2 亿文档的倒排链表中排名 top-1000 需要读取所有条目。

22.3 "用 SQL 数据库加缓存"——几乎从不奏效

缓存仅在查询完全重复时有效。有了个性化，命中率趋近于零。缓存适用于自动补全 top-K、热门查询、首页的分面统计——而不是作为搜索引擎本身。

22.4 使用 Lucene / Vespa / Tantivy 的真正价值

预构建的 FST 词典和 PFOR 压缩倒排链表；经过 20 多年优化、面向查询效率的合并/刷新策略；支持近邻、短语、BM25、函数评分、向量、地理位置、percolator 的可组合查询 DSL（领域特定语言）；丰富的分析器生态、语言插件、k-NN、LTR。在 Postgres 上构建这些 = 拙劣地重新发明一个搜索引擎。

§25. 用例展示——相同技术，不同形态

25.1 电商商品搜索（Yelp、Booking.com）

约 200 万到 2 亿件商品/商家/酒店；分面搜索（价格、类别、评分、地理距离）；个性化排序；p99 < 200ms；每隔数秒更新。 变体：基于 Lucene 的引擎，大量使用 doc values 统计分面、地理距离评分，BM25 + 函数评分实现"按评分加权"。Indexer 在索引时对关联的价格/库存/评论进行反规范化，查询时引擎无需执行关联操作。热门商品使用热层；首页分面统计进行缓存。

25.2 代码搜索（GitHub Blackbird、Sourcegraph + Zoekt）

搜索源代码，而非英文。用户需要子串匹配（func MyHelper(）、正则表达式、符号搜索。基于词的 Lucene 不适用—— getUserByEmailAddress 需要子串匹配 。 驼峰命名拆分。 变体：自定义 n-gram 倒排索引（trigram 或更长）。每个文件表示为所有重叠 3-gram 的集合；取交集找出候选文件；在匹配结果上用正则表达式验证。 GitHub Blackbird：2000 亿文档，640 TB 总量，全局子串查询 p99 低于 2 秒 ——从头构建引擎，因为 Lucene 的词级模型与子串 grep 不兼容。Sourcegraph 使用 Zoekt（源自 Google），架构相似，规模较小。

25.3 日志/可观测性搜索（Datadog、ELK、Loki、Quickwit）

每秒摄入数百万条日志，查询率较低（工程师按需搜索），保留期较短（~30 天热数据），对成本敏感。三种子形态： ELK / OpenSearch 日志 （对每个字段建全量倒排索引；最灵活，成本最高）； Loki （仅对标签建索引，日志正文以压缩块形式存储，全文搜索为"先过滤标签再 grep"——成本约为 ELK 的 1/10）； Quickwit （Tantivy segment 存储在 S3 上——计算与存储解耦，$/TB·月比 NVMe 低 10–20 倍，以数百毫秒的延迟换取数量级的成本降低）。

25.4 个人资料与人员搜索（LinkedIn Galene）

10 亿+ 个人资料；高度个性化搜索；分面丰富的 UI；10 万+ QPS；200ms 预算。 变体：Galene 是 LinkedIn 的 Lucene 分支。每位搜索者的个性化在热路径上进行——每个结果列表都使用特定于搜索者的特征进行重排。LTR 作为搜索 API 层的一部分，由在线特征存储（Pinot 或 Venice）提供支持，模型特征的刷新速度快于索引。按网络集群路由，大多数查询的 fanout 仅约 10 个分片，而非约 100 个。

25.5 实时 Feed / 推文搜索（Twitter Earlybird）

极高写入速率（历史上每天 5 亿条推文），亚秒级新鲜度，时间衰减相关性。 变体：Earlybird 是一个基于 Lucene 的内存引擎，支持激进的 100ms 刷新。热层在内存中保存最近几天的数据；冷层存储在磁盘上，很少被直接查询。按时间桶和用户分片。针对"典型查询是针对时间线小切片中非常近期的推文"进行了优化。

25.6 热路径中含 ML 的电商排序（Yahoo Shopping 上的 Vespa）

每次查询数千万候选，使用梯度提升树或神经模型对数千条结果进行 ML 重排，约 100ms 预算。 变体：Vespa 通过张量 在引擎内部 运行 ML 模型。引擎获取候选文档，在进程内将其 embedding/特征向量与模型相乘，返回 top-K——无需往返外部服务。这正是 Vespa 在 ML 密集型排序中占主导地位的原因，否则往返延迟将耗尽预算。

25.7 托管中小型站点搜索（Algolia）

最多约 1 亿条记录，全球 p99 低于 100ms，运维开销低。 变体：自定义内存引擎。每个搜索节点将全量索引加载到内存——这是延迟优势所在。成本随内存扩展；无法扩展到十亿级文档规模。对于 1 亿条以下的站点搜索，难以超越。

§26. 真实世界实现与数据

  • LinkedIn Galene ——基于 Lucene 的自定义引擎。 10 亿+ 个人资料，~10 万+ QPS，p99 < 200ms。Lucene 分支，支持个性化感知评分；联邦化、网络集群分区、热路径中的 LTR。
  • GitHub Code Search（Blackbird） ——自定义引擎（非 Lucene）。 2000 亿文档，640 TB 热索引，全局子串查询 p99 低于 2 秒 。重新设计了分词器（n-gram），因为 Lucene 的词级模型不适合代码搜索。
  • Sourcegraph + Zoekt ——开源代码搜索。Trigram 索引。典型部署规模为数万个仓库；与 Blackbird 架构相同，规模较小。
  • Yelp ——使用自定义 Lucene 进行商家搜索。 约 1000 万家商家，数百个分面，地理位置 + 分面评分，预计算首页分面，双层冷热架构。
  • Booking.com ——使用 Lucene 进行酒店搜索。 约 200 万处房源 × 数百个属性，10 个以上活跃分面下 p99 < 200ms，混合关键词 + 排序信号。
  • Twitter / X Earlybird — 自定义 Lucene 分支。 每天 5 亿条推文 历史写入速率。两层冷热架构；积极刷新。
  • Stack Overflow — Elasticsearch。 约 1 亿个问答文档。在单个 ES 集群上运行的典型中大型工作负载，具备完善的运维管理。
  • Algolia — 托管 SaaS，自定义内存引擎。面向约 1 亿条记录规模的站点，全球 p99 延迟低于 100ms。
  • Vespa at Yahoo Shopping / Vespa Cloud — 引擎内张量排序。数十亿文档，在热路径上进行 ML 重排序；应用于 Yahoo Mail/Shopping、广告排序，以及日益增多的 RAG（检索增强生成）后端。
  • Grafana Loki 大型可观测性 SaaS — 仅标签倒排索引。与同等保留期的 ES on logs 相比，成本降低约 10 倍。
  • Datadog Logs — 基于 ES。 每秒摄入数百万行日志 以 PB 级日体量运行；冷热两层架构，支持按需数据重新加载。
  • Quickwit 在 Binance / BlaBlaCar — 基于 S3 的 Tantivy。 5–10× 成本降低 相较于 ES 在等效日志保留期下的成本。
  • Apache Solr in library catalogs / enterprise CMS — 在现代 Web 中热度已下降，但在许多机构安装中仍是默认选择。

同一技术类别——基于倒排索引的排名检索——覆盖了六个数量级的文档数量、半打不同的工作负载形态，横跨从"业余 Tantivy 项目"到"GitHub 在自定义引擎上实现的全球级代码搜索"。词汇表（词条、倒排列表、doc values、段、刷新、合并、扇出）在所有系统中通用。

§27. 总结

"搜索系统是一种基于倒排索引的排名检索物化层，位于主存储的下游：词条存储在 FST 词典中，倒排列表经 PFOR 压缩并附带跳表指针，doc values 用于列式 facet/聚合，每段不可变并通过 translog WAL 保证写入持久性，1 秒刷新作为新鲜度/吞吐量旋钮；通过路由分区和冷热分层缩小扇出来提升速度，通过 WAND 进行早期终止，并利用在线特征存储喂给模型进行重排序；通过将其视为衍生数据（使得集群丢失可从真实数据源经 CDC + Kafka + 幂等 external_version 恢复）来保证正确性，通过别名切换强制映射前向演进，并与主库进行对账；通过从设计空间中选型来匹配工作负载——Lucene 衍生系（ES/Solr/OpenSearch）用于通用文本 + facet + 混合检索，Vespa 用于引擎内 ML 排序，Tantivy/Quickwit 用于成本敏感的日志搜索，专用引擎（Zoekt、Blackbird、Earlybird、Loki）则用于分词形态——代码、日志、推文、仅标签——偏离词级英文的场景。"
