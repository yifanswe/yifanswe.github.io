这是一份关于 技术类别 ——生产者与消费者之间持久、异步的消息传输——的参考资料，以 Apache Kafka、Apache Pulsar、RabbitMQ、AWS SQS (Simple Queue Service)、AWS Kinesis、Google Pub/Sub、NATS JetStream 以及更新的 S3 原生变体（WarpStream、AutoMQ）为例。读者读完后，应能深刻理解该设计空间，足以将其应用于任何系统——支付、点击流分析、IoT (Internet of Things) 遥测、CDC (Change Data Capture) 流水线、微服务扇出——并能在明显的替代方案面前为自己的选择辩护。

§1. 消息队列与分布式日志的本质

一个 消息队列 或 分布式日志 是生产者（发布记录）与消费者（读取记录）之间持久、异步的传输层，插入于两个原本通过 RPC (Remote Procedure Call) 同步通信的组件之间。它提供了同步调用无法实现的能力： 时间解耦 （消费者无需在生产者发布时处于运行状态）， 扇出 （一次发布，多个消费者），以及 背压吸收 （发布峰值被缓冲，而不是压垮消费者）。

该类别分为两种结构上截然不同的原语，文档和面试中常将其混淆。搞清楚这一分野，否则之后的每个选择都会出错。

日志模型（Kafka、Pulsar、Kinesis）。 Broker 维护一个 每分区只追加、可保留、可重放的日志。生产者追加数据；broker 从不删除单条记录，只删除超过保留期的整个 segment。消费者自行追踪读取位置（即"偏移量"）。Broker 不知道任何消费者是否读取了某条记录——它只是持有这条"磁带"。心智模型：一个事件磁带的复制版 tail -f 扇出是免费的（N 个消费者组各自追踪自己的偏移量）。重放是内置功能。

队列模型（RabbitMQ、SQS、ActiveMQ）。 Broker 维护一个 每条消息的状态机 ——每条记录处于未确认、传输中、已确认、重新投递或死信状态。消息被确认后，broker 将其删除。Broker 追踪投递状态，消费者不追踪位置。心智模型：一个带重新投递和每条消息路由的共享工作窃取队列。扇出代价高昂（broker 复制到 N 个每订阅者队列）。重放通常不可能。

两者都位于 生产者服务 （支付平台的 order-service、社交 feed 的 activity tracker、车队的 IMU publisher）和 消费者服务 （billing pipeline、feed indexer、时序存储）之间。这是基础设施，而非应用逻辑——一个带有元数据共识层的自包含 broker 集群。

不适合使用的场景： 有严格延迟预算的同步请求/响应（队列增加了一跳 + 数十 ms——对于"必须在 50 ms 内响应"的场景，请使用 gRPC）；高吞吐量下的严格全局全序（每分区可以实现，每秒百万条的跨分区全序不可行）；需要点查的主系统记录（日志是顺序读写的——请使用数据库，让日志作为变更流）；消息量小但每条消息语义丰富的场景（50 msg/秒带优先级路由 + reply-to：Postgres SKIP LOCKED 或单个 RabbitMQ exchange 比 Kafka 集群更简单）。

§2. 内置保证——哪些由系统提供，哪些需要额外分层

设计上已提供： (1) 已确认写入的持久性 ——所有变体都在复制（或在 BookKeeper 支持的 Pulsar 中通过 fsync）到足够多的节点以承受单节点故障后，才进行确认。(2) 每分区/每队列的全序 ——在单个分区或队列内，记录按 broker 接受的顺序出现；跨分区顺序无保证。(3) 默认的至少一次投递 ——生产者重试直到收到确认；消费者仅在处理完成后提交；允许重复，不允许丢失。(4) 保留作为一等概念（仅日志） ——记录独立于消费存活 N 天，支持重放。

未提供（Staff 级别陷阱）： (1) 无需显式配置的精确一次 ——Kafka 有事务协议，Pulsar 有去重 + cursor，SQS FIFO 有 5 分钟去重窗口。所有这些都需要生产者 + 消费者 + sink 共同参与；对于涉及外部 API 调用（Stripe、Postgres）的端到端精确一次，没有应用层幂等键无法实现。(2) 全局排序 ——仅限每分区；排序要求必须编码到分区键中（例如， account_id 用于账本事件）。(3) 与外部系统的原子性 ——消费者读取 + DB 写入不是原子的；需要在上层叠加 outbox 或 2PC (Two-Phase Commit)。(4) Schema 演进 ——broker 不了解 payload 内容；需要 Schema Registry + 向后兼容的 Avro/Protobuf。(5) 共享集群上的服务质量 ——嘈杂邻居 topic 会使 broker IO 饱和；需要在上层叠加配额或每租户独立集群。

系统设计者必须额外考虑的内容： ack/commit 策略、分区键策略、保留策略、schema 演进策略、死信策略、重放流程。队列或日志不是"魔法持久管道"——它是一个底层基础；应用架构才是其他一切的核心。

§3. 设计空间

五个维度区分了各命名变体。其余的差异都是在这些维度上的变化。

轴 1：日志 vs 队列（存储模型）。 见 §1。日志：保留 segment。队列：每条消息的状态机。扇出成本相差数个数量级。

轴 2：Broker 管理 vs 点对点。 Broker 管理型（Kafka、Pulsar、RabbitMQ、SQS）——专用 broker 进程拥有数据。点对点型（NATS Core、基于 gossip）——每个节点都是对等节点；延迟更低，持久性更弱。这里基本不在讨论范围内——点对点适用于超低延迟的短暂遥测数据。

轴 3：同步 vs 异步复制。 同步（Kafka acks=all, Pulsar Qa=Qw) — 生产者等待副本确认；约 5-15 ms，单节点故障不丢数据。Async（Kafka acks=1, RabbitMQ 默认）— 主节点确认后，副本异步追赶；约 1-3 ms，主节点崩溃时存在数据丢失窗口。None（acks=0）— 适用于非关键遥测数据的"即发即忘"模式。选择由工作负载决定：支付 outbox 始终使用同步；点击流分析通常使用异步；特斯拉车队 CAN 总线（Controller Area Network）遥测对故障码使用同步，对常规传感器读数使用异步采样。

Axis 4：本地磁盘与对象存储持久性。 本地 NVMe + 复制（经典 Kafka、BookKeeper bookies）— 约 1 GB/sec/broker，p99 约 5-15 ms，NVMe（Non-Volatile Memory express）约 $0.10/GB-月。对象存储为主（WarpStream、AutoMQ，约自 2023 年起）— broker 无状态，持久性依赖 S3 的 11 个 9，p99 约 200 ms（S3 分片上传），约 $0.023/GB-月，高保留量场景下成本降低 5-10 倍。该维度在 2022 年后出现，是当前领域的活跃前沿。

Axis 5：单集群与跨地域复制。 跨 3 个可用区（AZ）的单集群是默认部署方式。跨区域主动-被动复制（MirrorMaker 2、Pulsar 地理复制）可将 RPO（Recovery Point Objective）控制在秒级。跨区域主动-主动架构使冲突解决成为系统需求；Pulsar 原生支持，Kafka 则需要精心设计幂等性。

§4. 字节级机制 — 磁盘上的日志

深度内容在本节展开。自 2010 年以来，Kafka 日志是分布式日志原语的经典实现；所有后来的日志系统（Pulsar 段、Kinesis 分片、S3 原生变体）都借鉴了它。我们以 Kafka 为例，因为它是文档最为详尽的实现。

4a. 分段追加日志 + 稀疏偏移索引

单个分区的磁盘布局：

/kafka/data/clickstream-page-views-23/        (one directory per partition)
├── 00000000000000000000.log         closed segment, 1 GB, offsets 0..2_048_000
├── 00000000000000000000.index       sparse offset → byte position
├── 00000000000000000000.timeindex   sparse timestamp → offset
├── 00000000000002048000.log         closed segment, 1 GB
├── 00000000000002048000.index
├── 00000000000002048000.timeindex
├── 00000000000004096000.log         ACTIVE segment, currently appending
├── 00000000000004096000.index
├── 00000000000004096000.timeindex
├── leader-epoch-checkpoint          which epoch took over at which offset
└── partition.metadata

段轮转规则： 当活跃段达到 segment.bytes （默认 1 GB）或 segment.ms （默认 7 天）时，该段关闭，新的活跃段开始写入。 已关闭的段是不可变的。 这种不可变性是整个设计的基础——它使零拷贝读取、无锁删除、无锁压缩以及分层存储卸载成为可能。

文件命名： 文件名是补零的起始偏移量。定位偏移量 3,500,000（例如，分析师想要重放的三周前特斯拉车队遥测事件）：(1) 二分搜索段文件名 → 最大起始偏移 ≤ 3,500,000 的段是 00000000000002048000.log；(2) 打开稀疏 .index （每一条记录 index.interval.bytes = 4 KB，因此 1 GB 段约有 256K 条条目，约 2 MB 索引，保存在内存中）；(3) 二分搜索索引 → 定位到目标约 4 KB 范围内的字节位置；(4) 向前扫描 .log 解析记录批次头部，直到找到精确偏移。

索引是 有意设计为稀疏的 ——密集索引需要 64 字节 × 数十亿条记录，无法容纳。稀疏索引 + 向前扫描以 0.001 倍密集索引内存开销实现 O(log n) 定位。

4b. 记录批次布局

生产者在网络上从不逐条发送记录。它将记录累积成批次（默认最多 16 KB 或 linger.ms 的数据量），用 lz4 / zstd / snappy / gzip 端到端压缩批次，并将压缩后的批次作为一个单元发送。

值得关注的要点：(1) broker 原封不动地存储压缩批次 ——热路径不解压；消费者在客户端侧解压。这就是单个 broker 能维持约 1 GB/sec 的原因——CPU 从不接触有效载荷。(2) producerId + baseSequence ——幂等生产者的基础。broker 跟踪每个（producerId, partition）的最新序列号，拒绝重复请求。重试复用相同序列号；broker 静默去重。(3) 每批次的 CRC（Cyclic Redundancy Check） ——在每次 Fetch 和复制时验证；检测位衰减、部分写入和网络损坏。

4c. 为何该结构适合工作负载——顺序写、页面缓存、sendfile

三个特性使该设计逼近硬件极限而非软件极限。

顺序写入。 每次追加都写入活跃段的末尾。NVMe 顺序写速度达 3-7 GB/sec——原始设备吞吐量；随机写仅 100-200 MB/sec，慢 20-50 倍。即使是旋转磁盘（LinkedIn 曾使用多年），顺序写约 200 MB/sec，随机写约 5 MB/sec——相差 40 倍。 仅追加写是 Kafka 能以商用磁盘实现 1 GB/sec 的根本原因。

操作系统页面缓存作为读缓存。 broker write() 将字节写入内核页面缓存后立即返回。内核回写异步刷新脏页（每约 30 秒通过 dirty_writeback_centisecs）。页面在持久化后仍长期留存于页面缓存中。 消费最近偏移的消费者 ——这是主导模式，因为大多数消费者距日志头部仅有数秒—— 命中页面缓存，无磁盘 IO。 每个 broker 64-256 GB 内存，无需应用层缓存即可从 RAM 服务数 TB 热工作集。

sendfile() 零拷贝读取的系统调用。 broker 无需将文件读入用户空间再写入 socket。它调用 sendfile(socket_fd, log_fd, &start, length)；内核直接将页面缓存复制到 socket 缓冲区。现代网卡通过 DMA（Direct Memory Access）将页面缓存页直接传输到网络。

Without sendfile (4 copies): disk → page cache → JVM heap → kernel socket → NIC
With sendfile (2 copies):    disk → page cache → NIC

2-4 倍吞吐量提升，broker CPU 开销几乎为零。 这也是 Kafka 的 broker 可以使用 JVM（Java Virtual Machine）而不被 GC（Garbage Collection）压力拖垮热路径的原因 ——有效载荷字节从不进入堆内存。RabbitMQ 无法做到这一点，因为它需要检查每条消息以应用路由规则。

4d. 持久性：fsync 与复制

三个持久性级别：(1) 页面缓存 — write() 返回；字节在内核内存中，未落盘。内核崩溃或断电（无电池支撑的缓存）会导致数据丢失。(2) 复制确认 ——从节点的 Fetch 拉取字节，写入各自的页面缓存；一旦 min.insync.replicas （默认 2 个）确认，主节点推进 HW（High Watermark）并向生产者确认。(3) 磁盘 fsync ——内核回写刷新脏页，通常每约 30 秒一次。

Kafka 的持久性不变量基于复制，而非 fsync。 默认配置 log.flush.interval.messages = Long.MAX_VALUE ——Kafka 每次写入不执行 fsync。理由：主节点 + ≥1 个从节点页面缓存同时丢失（3 台以上机器同时断电）比单 broker fsync 失败更罕见。在跨 3 个可用区独立供电的 3 副本场景下，这是合理的。

Pulsar 采取相反的立场。 BookKeeper bookies 在确认前对每条条目执行 fsync。每节点持久性更强，因此 Pulsar 可以用更少副本确认（Qw=2, Qa=2）。权衡代价：NVMe 约 0.5 ms fsync，HDD 约 5 ms 每次写入。

面试陷阱： 候选人经常说"Kafka 每次写入都执行 fsync"。实际上并非如此。如需每次写入 fsync 语义（金融监管要求），需设置 log.flush.interval.messages=1 并承受约 10 倍吞吐量损失。或者改用 Pulsar。

4e. 字节级生产与消费全流程

具体示例：支付服务写入 order_created ，键为order_42；欺诈检测消费者（不同团队）读取它。

生产路径： - T0： 生产者的 send() 将记录缓冲到目标为分区 7（hash(order_42) % 256 = 7）的 16 KB 批次中；端到端 lz4 压缩。\n- T1： after linger.ms 或批次已满，生产者向 broker B-7（分区 7 的主节点）发送 Produce RPC。\n- T2： broker B-7 验证 CRC，通过 write() 将批次追加到活跃段的当前 EOF。字节落入页面缓存；文件位置前进 16 KB。若自上次索引以来累积字节 ≥ 4 KB，则添加稀疏索引条目。LEO 从 4,500,000 推进到 4,500,201（201 条记录）。\n- T3： broker 尚未确认——等待 ISR 从节点。\n- T4： 从节点 B-31 和 B-88 正在长轮询 B-7 的 Fetch 。下一次 Fetch 返回批次；每个从节点将批次追加（ write()）到自己的活跃段，并发送带有 fetchOffset=4,500,201的后续 Fetch，隐式确认。\n- T5： B-7 检测到两个从节点均已追上；将 HW（High Watermark）从 4,500,000 推进到 4,500,201。HW 是记录"已提交"的偏移量上界——已复制到所有 ISR 成员。\n- T6： B-7 向生产者确认；共耗时约 5 ms。\n- T7（异步，约 30 秒后）： 内核回写守护进程将页面缓存中的脏页 fsync 到 NVMe。 磁盘持久性在此处实现，而非在 T2 时。

各步骤崩溃后的数据存活情况： - T2 时崩溃（写入中途）： 生产者未收到确认，重试。重启时，broker 扫描活跃段，在末尾发现不完整批次（CRC 校验失败），截断回最后一条完整写入记录的边界。幂等生产者对重试进行去重。\n- T5 时崩溃（主节点推进 HW 前）： 生产者未收到确认。KRaft 控制器从 ISR 中选出新主节点（假设为 B-31，在 T4 时已获得批次）。生产者向新主节点重试；幂等生产者去重。\n- T6 时崩溃（确认后，T7 fsync 前）： 数据存在于三个 broker 的页面缓存中。三个 broker 同时断电（区域性灾难）将导致数据丢失。在 3 个可用区独立供电的场景下，这属于"区域性事件"故障模式——属于可接受的风险。

消费路径： - T0： 消费者发送 Fetch RPC：partition=7， fetchOffset=4,500,000, maxBytes=1 MB, maxWait=500ms, minBytes=1 KB. - T1： broker 二分搜索段文件名：4,500,000 ≥ 起始偏移 4,096,000 → 段 00000000000004096000.log. - T2： broker 读取稀疏 .index：最大已索引偏移 ≤ 4,500,000 为 4,499,840，字节位置为 67,108,000。\n- T3： broker 计算字节范围：start=67,108,000（对齐到批次边界），length=min(1 MB, HW-fetchOffset)。\n- T4： broker 调用 sendfile(socket_fd, log_fd, &start, length)。内核从页面缓存读取页面 → DMA 到网卡 → 网络传输。 字节从不进入 JVM 堆。broker CPU 开销约 0.3 ms。 - T5： 消费者接收约 1 MB 的压缩批次；对每个批次：验证 CRC，lz4 解压，从 fetchOffset 开始迭代记录，传递给应用程序。\n- T6： 消费者处理记录。\n- T7： 消费者通过 __consumer_offsets RPC 向组协调器（将 OffsetCommit 哈希到对应 broker）提交新偏移，写入 group.id 到 (group, topic, partition) → offset topic。 __consumer_offsets. - T8（T6 与 T7 之间崩溃）： 重启时，消费者从最后提交的偏移量（4,500,000）重新拉取并重新处理记录。 至少一次语义。幂等处理器必须去重。

broker 服务 1 MB 消费请求的 CPU 开销约 0.3 ms（二分搜索 + sendfile）；磁盘与网络是瓶颈。 单个 broker 在 25 Gbps 网卡上可服务约 3 GB/sec 的消费者扇出 在网卡饱和之前。这就是扇出"近乎免费"的原因——增加第 10 个消费者组的 IO 开销为零，除非多个组同时读取相同的热点偏移。

4f. 日志压缩——类 LSM 树原语

对于配置了 cleanup.policy=compact 的 topic（状态快照——CDC 当前行状态、 __consumer_offsets 本身、特征存储在线缓存），后台日志清理器扫描已关闭的段，生成新段，每个键只保留 最新 值。墓碑记录（null 值记录）表示删除。清理器构建 key → latest_offset的内存哈希映射，重写段，原子替换旧段。受 log.cleaner.io.max.bytes.per.second限速。这是内嵌于 Kafka 的类 LSM（Log-Structured Merge）树原语，但仅适用于压缩 topic。大多数生产 topic 使用 cleanup.policy=delete （删除超出保留期的整个段）。

4g. 分层存储 — 将冷段卸载到 S3

早于以下时间的已关闭段 local.retention.ms （例如 24h）将被上传到 S3 / HDFS / Azure Blob。Broker 本地只保留 .index + 元数据指针； .log 存储在对象存储中。消费者请求旧 offset 时：broker 先检查本地（约 5 ms sendfile 路径）；若未命中，则从 S3 代理给消费者进行范围读取 GetObject （约 100-200 ms，存储成本降低 10 倍）。这是通往 S3 原生设计（WarpStream、AutoMQ）的桥梁，这些设计将 broker 推向无状态，由 S3 端到端充当持久层。权衡：约 200 ms p99 生产延迟 vs 本地磁盘约 5-15 ms。

§5. 生产者端机制 — 客户端实现的另一半协议

broker 是系统的一半；生产者客户端是另一半，broker 宣传的大多数"精确一次"、"无损"和"高吞吐量"保证都依赖于生产者的正确配置。一个配置草率的 KafkaProducer 是即发即忘；一个正确配置的则是一个拥有自身状态机的小型分布式协议。本节涵盖必须熟记的几个关键参数：幂等生产者、事务性生产者、批处理、压缩以及 acks.

5a. 幂等生产者 — 通过 PID + 序列号在 broker 端去重

默认 Kafka 生产者在网络抖动时会重试相同的记录。若无保护，broker 会写入重复数据；支付用户被收费两次（§18.1）。幂等生产者（Kafka 0.11 于 2017 年引入）在 broker 端关闭了这个漏洞。

协议。 启动时，生产者客户端向某个 broker（事务协调器）发送 InitProducerId 。broker 分配唯一的 PID（Producer ID，64 位） 。 生产者纪元（16 位，非事务性时清零）。生产者随后为每个（PID，分区）维护一个 每个（PID，分区）单调递增的序列号 并在每条记录批次头部打上 producerId, producerEpoch, baseSequence （§4b 展示了字段布局）。

broker 端状态：对于每个（PID，分区），它跟踪最近接受的 5 个序列号（可通过 max.in.flight.requests.per.connection配置，默认 5）。对每个传入的 Produce 请求：- 若 incoming.sequence == lastSequence + 1 → 接受并追加。- 若 incoming.sequence <= lastSequence （且在 5 槽窗口内）→ 重复，返回成功但不再追加。生产者视角：重试"成功"。- 若 incoming.sequence > lastSequence + 1 → 乱序，返回 OutOfOrderSequenceException。生产者必须重置并重新建立状态。

Producer state machine (idempotent):
   ┌─────────────────┐
   │  Init           │── InitProducerId ──> Broker assigns PID=12345, epoch=0
   │  PID=12345      │
   │  epoch=0        │
   │  seq[part-7]=0  │
   └────────┬────────┘
            │ send(record_A)  → batch{PID:12345, epoch:0, baseSeq:0, ...}
            ▼
   ┌─────────────────┐
   │  In-flight      │   network blip; no ack received
   │  seq=0,1,2,3,4  │
   └────────┬────────┘
            │ retry batch{baseSeq:0}  → broker sees seq 0 already, returns success silently
            ▼
   ┌─────────────────┐
   │  Acked          │   producer thinks "first attempt succeeded"
   │  seq[part-7]=5  │
   └─────────────────┘

代价。 几乎免费 — 每个生产者进程生命周期内只需一次 ID 分配，每条记录批次约 24 字节元数据（PID + epoch + baseSequence）。broker 端状态有界：5 个序列号 × 约 1M 个 PID × 约 1K 个分区 = 数 MB。 为何不作为默认值？ 历史上它降低了吞吐量，因为强制了 max.in.flight.requests.per.connection=1；KIP-360（2.5+）允许最多 5 个在途请求并基于序列号重排。现代指导： enable.idempotence=true 是生产环境默认值。Confluent 自 3.0 起已将其设为隐式默认值。

它不提供什么。 幂等生产者去重 在单个生产者生命周期内，按分区。它不跨生产者重启（新 PID），不跨分区（跨 P-7 和 P-42 的写入各自可独立成功），也不提供对外部 sink 的精确一次语义。这些场景需要事务。

5b. 事务性生产者 — 跨分区和主题的原子写入

使用场景：在 §18.1 的支付流程中，消费者读取 payment.request，写入 payment.charged 到一个主题，写入 ledger.entry 到另一个主题，然后提交源 offset。若这三次写入中有任何一次未与其他写入一起完成，账目便会不一致。事务性生产者（Kafka 0.11+）将跨分区、跨主题的写入以及消费者 offset 提交包装为一个原子单元。

协议 （两阶段提交，以 事务协调器 broker 作为 TC）：

Phase 0: Producer registers transactional.id="payments-tx-7"
   Producer ──InitProducerId(transactional.id)──> TC
   TC: looks up persistent (txid → PID, epoch) mapping in __transaction_state topic.
       If exists: increment epoch (fencing — older zombie producer with prior epoch is dead).
       Returns (PID=12345, epoch=2) to producer.

Phase 1: Producer.beginTransaction()
   Producer ──AddPartitionsToTxn([orders-7, ledger-13])──> TC
   TC: writes "ongoing transaction includes orders-7, ledger-13" to __transaction_state.

Phase 2: Producer sends data
   Producer ──Produce(orders-7, batch{PID:12345, epoch:2, isTxn:true})──> broker B-7
   Producer ──Produce(ledger-13, batch{PID:12345, epoch:2, isTxn:true})──> broker B-23
   Producer ──AddOffsetsToTxn(consumer-group="payments-cg")──> TC
   Producer ──SendOffsetsToTxn(payments-cg, payment-requests-3 → offset 5000)──> TC

Phase 3: Producer.commitTransaction()
   Producer ──EndTxn(commit=true)──> TC
   TC writes "PREPARE_COMMIT" to __transaction_state (durable decision point).
   TC writes COMMIT_MARKER (control record, isTxn=true, commit=true)
       to orders-7 and ledger-13 and __consumer_offsets.
   TC writes "COMPLETE_COMMIT" to __transaction_state.

Phase 4: Cleanup
   Consumer with isolation.level=read_committed will only see records
   up to the latest COMMIT_MARKER for each partition.
   Records past an ABORT_MARKER are skipped client-side (LSO < HW).

LSO（Last Stable Offset） 是事务性消费者的新高水位线：其下方 不存在进行中事务 的最高 offset。配置了 isolation.level=read_committed 的消费者只读到 LSO； read_uncommitted 读到 HW（原有行为）。已中止事务中的记录会流式传给消费者，但 在客户端过滤 使用 broker 端的已中止事务索引 — broker 端不会重写日志。

通过 epoch 隔离生产者。 "僵尸生产者"问题：生产者 A 启动一个事务，挂起（长时间 GC 暂停），Kubernetes 将该 pod 重启为生产者 A'。如果两者继续写入，则有两个并行事务声称相同的 transactional.id。解决方案：TC 在每次 InitProducerId(txid)时递增 epoch。生产者 A' 获得 epoch=3；生产者 A 仍持有 epoch=2。当 A 的暂停结束并发送 epoch=2 的批次时，broker 拒绝并返回 InvalidProducerEpochException. A 的事务被强制中止。

代价。 每个事务增加约 50-100 ms 延迟（两次额外到 TC 的往返加上标记写入），小事务的吞吐量降低约 3-5 倍。通过 将多条记录批入一个事务 — Kafka Streams 默认每 100 ms 提交间隔一个事务。不要每条记录开一个事务。

它不提供什么。 事务性生产者提供精确一次语义 进入 Kafka 主题和 offset。它们不扩展到外部 sink（Postgres、Stripe API）。对于端到端到这些 sink 的精确一次语义，需要（a）在外部边界使用幂等键（Stripe 接受 Idempotency-Key header），或（b）消费者实现自己的去重表。

5c. 批处理 — linger.ms 。 batch.size

每次 send() 调用发送一条记录的生产者，在 100K msg/sec 下会触发 100K 次网络调用。每次网络往返约有 0.5-2 ms 的 RPC 开销，每条记录批次信封约 50-100 字节的 TCP/Kafka 头部。 批处理是 Kafka 生产者扩展的根本原因。

两个相互作用的参数：- batch.size （默认 16 KB）— 发往同一分区的每批次最大字节数。批次填满后立即发送，不论时间。- linger.ms （默认 0）— 在发送前等待更多记录填充批次的时长。 linger.ms=0 立即发送第一条记录（除非多次发送发生在一个 TCP RTT 内，否则无批处理收益）。 linger.ms=10 等待最多 10 ms 以等待更多记录。

Latency vs throughput tradeoff:

   linger.ms=0,  batch.size=16KB        linger.ms=20, batch.size=64KB
   ─────────────────────────────         ─────────────────────────────
   p50 produce:    2 ms                  p50 produce:   22 ms
   p99 produce:    8 ms                  p99 produce:   28 ms
   throughput:    50 MB/sec/producer     throughput:   800 MB/sec/producer
   network RPCs:  hundreds/sec           network RPCs:  tens/sec
   compression:   poor (small batches)   compression:   strong (large batches)

按领域的经验法则。 支付 outbox（低吞吐量，对延迟敏感）： linger.ms=0, batch.size=16KB。点击流分析（高流量，延迟容忍）： linger.ms=20-50, batch.size=128KB-1MB。CDC 管道（持续中等吞吐量，有序）： linger.ms=5-10, batch.size=64KB。Tesla 遥测（高吞吐量突发）： linger.ms=10, batch.size=256KB，热点 key 时按 key 分批。

生产者内部维护一个 RecordAccumulator — 每个（主题，分区）一个缓冲区，总计最多 buffer.memory （默认 32 MB）。 Sender 线程轮询累积器，挑选已满或已超过时限的批次 linger.ms并按目标 broker 分组发送（每个 broker 一个 ProduceRequest per broker，包含该 broker 主导的所有分区的批次）。这就是分区数量倍增生产者内存需求的原因：1000 个分区 = 最多 1000 个批次 = 若全部填满可能达 16 GB。 buffer.memory 是硬上限；一旦满了， send() 阻塞 max.block.ms.

5d. 压缩 — gzip vs lz4 vs zstd vs snappy

批次由生产者端到端压缩（broker 存储压缩后的内容，消费者解压 — §4b）。选择在 CPU、压缩率和延迟之间权衡。

编解码器 压缩率（典型 JSON） 压缩 CPU 解压 CPU 备注
none 1.0x 0 0 仅适用于极小或已预压缩的载荷（Protobuf、加密数据）
snappy ~2-3x 非常快 非常快 Google 默认；CPU 开销最低；压缩率最小
lz4 ~2-3x 最快 最快 Kafka 2.1 起的默认值；在相近 CPU 下比 snappy 压缩率约好 10%
gzip ~5-7x 慢（约 10 倍 lz4） 中等 最佳压缩率，CPU 最差；现代 Kafka 中少见
zstd ~4-6x 中等 快 2.1 起的甜蜜点：以 gzip 3-5 倍的 CPU 达到其 80% 的压缩率； 默认推荐

规模化时，压缩主导 broker 成本。 Cloudflare 的边缘日志传输（§24 提到）在从 gzip 迁移到 zstd 之前据报耗费约 10-15% 的边缘 CPU；迁移后降至约 3-4%。相同的网络字节数，生产者 CPU 大幅减少。 由于 broker 原样存储压缩批次，选择 zstd 而非 lz4 会使 broker 磁盘占用更小，这会产生复合效应 — 存储成本由保留时间 × 摄取速率 × 副本因子主导（§24）。

注意： 若载荷已经过压缩（gzip HTTP body、JPEG、带 packed fields 的 Protobuf），再次压缩毫无收益且浪费 CPU。在这些情况下默认使用 none 在这些情况下。

5e. acks — 持久性调节旋钮

最关键的生产者配置项。三种设置，语义截然不同：

  • acks=0 （即发即忘）。 生产者不等待 broker 确认。约 0.1-0.5 ms 发送延迟。 broker 崩溃写入中途的数据丢失是无声的。 仅用于容忍少量丢失窗口的遥测（指标采集、追踪采样、视频播放心跳）。
  • acks=1 （仅主节点）。 生产者等待直到主节点已追加到本地日志（页缓存）。约 1-3 ms 延迟。 数据丢失窗口： 若主节点在确认后、从节点复制前崩溃，批次将丢失。每分区，在稳态故障下概率约为 0.001%，滚动重启期间可能更高。适用于可接受 0.001% 丢失的分析工作负载（点击流 §23）。
  • acks=all （全 ISR）。 生产者等待直到 min.insync.replicas 从节点已追上。约 5-15 ms 延迟。 在任何单节点故障下均不容许丢失，配合 min.insync.replicas=2 + RF=3，可容忍一个 broker 故障并仍能提供写入服务。支付、CDC、账本、审计日志的默认选择。

min.insync.replicas 交互。 常见 bug：设置了 acks=all 但保留了 min.insync.replicas=1。此时 acks=all 意味着"当主节点的唯一 ISR 已复制时确认" — 但如果所有从节点都落后，主节点就是那个唯一的 ISR。 实际上退化为 acks=1 且没有任何警告。 LinkedIn 标准： acks=all, min.insync.replicas=2，RF=3。容忍一个 broker 故障而不中断写入；若两个 broker 宕机则拒绝写入（而非悄无声息地丢失持久性）。

acks=all 故障模式：写入不可用。 若 3 个 ISR broker 中有 2 个宕机且 min.insync.replicas=2，主节点会拒绝写入并返回 NotEnoughReplicasException。生产者端：重试或报错。 这是有意为之的设计 — 大声报错好过静默接受系统无法持久存储的写入。"不洁主节点选举"参数（§11c）是在可用性优先于持久性时的应急出口。

§6. 消费者再平衡 — 急切 vs 协作增量、静态成员资格、offset 提交策略

消费者组协议是大多数生产环境 Kafka 故障的根源。该协议概念上很简单 — N 个消费者、M 个分区，将分区分配给消费者，成员变更时重新分配 — 但其实现是一个状态机，存在令人意外的延迟悬崖。

6a. 急切再平衡（2.4 之前的遗留默认值）

在 KIP-429（Kafka 2.4，2020 年）之前，唯一的再平衡协议是 急切再平衡，也称"全局停止"。协议：

Eager rebalance, 64 consumers, 256 partitions:
T0: C5 leaves the group (deploy, crash, GC pause beyond session.timeout.ms).
T1: Group coordinator broadcasts "rebalance starting" to ALL 64 consumers.
T2: EVERY consumer (not just C5) revokes its current 4 partitions, commits offsets, returns.
T3: Coordinator collects "joins" from all 64; chooses one as leader.
T4: Group leader runs PartitionAssignor; produces new (consumer → partitions) map.
T5: Coordinator broadcasts new assignments to all 64.
T6: Each consumer resumes consumption on its new assignments.

Total stop-the-world: T2 → T6 ~ 3-10 seconds, ALL consumers idle.
At 1M msg/sec, that's 3-10M messages of lag building up.

当消费者在滚动部署期间逐一加入时，这一病症会加剧：64 次独立再平衡，每次 5 秒 = 320 秒（5 分钟以上）的间歇性停顿。 这是 LinkedIn 直至约 2018 年的主要运维痛点。

6b. 协作增量再平衡（CooperativeStickyAssignor, 2.4+)

KIP-429 引入了协作增量再平衡 — 只有失去分区的消费者暂停，其他消费者继续运行。

Cooperative rebalance, 64 consumers, 256 partitions, C5 leaves:
T0: C5 leaves.
T1: Coordinator notifies all 64 of rebalance.
T2: Each consumer reports its CURRENT assignments back.
T3: Group leader computes new assignment based on prior assignments,
    aiming to MINIMIZE movement (sticky principle).
T4: Each consumer receives instruction:
    "C18 and C42, revoke partitions P-200 and P-201 (formerly C5's)."
    The other 62 consumers receive: "no change, continue."
T5: C18 and C42 revoke; THE OTHER 62 NEVER PAUSED. Coordinator reassigns.
T6: C18 and C42 receive new partitions; consume resumes.

Stop-the-world for 62/64 consumers: 0 ms.
Stop-the-world for the 2 reassigned consumers: ~100-300 ms.

配置：设置 partition.assignment.strategy = org.apache.kafka.clients.consumer.CooperativeStickyAssignor （也可用： RangeAssignor, RoundRobinAssignor, StickyAssignor — 后者是粘性的但仍为急切模式）。迁移到协作模式需要通过混合期进行滚动重启；Kafka 3.0+ 默认包含协作协议但不优先使用它。 有状态消费者（Kafka Streams）受益最大 — 旧分区的本地状态无需重建。

6c. 静态组成员资格 — 滚动重启时避免再平衡

默认协议将每次消费者（重新）连接视为持有新 member.id的新成员。64 个消费者的滚动重启会触发 64 次再平衡（每次重启一次）。KIP-345（Kafka 2.3+）引入了 静态成员资格 通过 group.instance.id:

Without static membership:
   C5 restarts ─> coordinator sees member 'consumer-3a7f' leave (new memberId on rejoin)
   ─> rebalance (3-10 seconds for eager, 100ms for cooperative).

With group.instance.id="payments-cg-worker-5":
   C5 restarts ─> coordinator sees instance "worker-5" reconnect within session.timeout.ms
   ─> matches by group.instance.id, restores PRIOR assignments
   ─> NO REBALANCE TRIGGERED.

注意： session.timeout.ms （默认 45s）是重启的时间预算。若消费者耗时更长（JVM 慢启动、k8s 从冷镜像仓库拉取），会话超时并正常触发再平衡。生产建议：将 session.timeout.ms 调高至约 5 分钟用于有状态消费者，确保重启在窗口内完成。

LinkedIn 流处理器标准：静态成员资格 + session.timeout.ms=300000 （5 分钟）+ 协作增量再平衡。综合来看，这消除了 2020 年前观察到的约 95% 的再平衡暂停。

6d. Offset 提交策略 — 自动 vs 同步 vs 异步，至少一次 vs 至多一次

消费者如何告知 broker"我已处理到 offset X"决定了整个管道的语义保证。三种策略，四种语义模式。

enable.auto.commit=true （默认）。 后台线程每 auto.commit.interval.ms （默认 5 秒）提交当前消费者位置。 隐藏陷阱： 无论记录是否被 成功处理，提交都会发生。若消费者在 4 秒时（两次提交之间）崩溃，已被 poll()但未处理的记录的 offset 会被提交 — 无声的至多一次数据丢失。更糟的是，应用逻辑看不到这个错误。

同步提交（consumer.commitSync()). 应用在处理每批后显式调用 commitSync() 处理每批后阻塞直到 broker 确认；失败时抛出异常。强语义的至少一次：提交后崩溃不重放任何内容；提交前崩溃重放最后一批。延迟成本：每次提交约 5-10 ms。100 个消费者的集群以 1M msg/sec 处理时，若每批提交（而非每条记录），这是可接受的。

异步提交（consumer.commitAsync()). 应用处理后调用 commitAsync() 处理后在后台通过回调提交。提交延迟最低，但回调必须处理失败 — 若某次提交失败且从未重试，下一次提交会覆盖，丢失中间的处理窗口。标准模式： commitAsync() 在每批之后， commitSync() 仅在关闭/分区撤销时 （因此最终状态是持久的）。

手动偏移量存储（消费者端）。 对于向非 Kafka sink（Elasticsearch、MySQL）实现精确一次语义，消费者在外部存储中将偏移量与数据写入同一事务中。重启时，消费者从外部存储读取偏移量，并调用 consumer.seek(offset)。现在偏移量与数据在外部存储中是原子性的；Kafka 的 __consumer_offsets 被绕过。

语义对照表：

提交模式 确认后提交？ 此处崩溃 ⇒ 语义
commitSync() 处理前提交 不适用 从不重新处理，丢失该批次 至多一次
commitSync() 处理后提交 是 重启时重新处理一次 至少一次 （默认）
commitSync() 处理后提交 + 幂等消费者 是 重新处理但幂等去重 实际上精确一次
外部存储原子写入 + 偏移量 是 外部存储作为真实来源 跨 sink 精确一次

规则： 提交在 之前 处理之前 = 至多一次（数据丢失）。提交在 after 处理之后 = 至少一次（重复）。精确一次要求使用事务性生产者（§5b）或外部存储原子提交，不存在"直接开启精确一次"的开关。

§7. Schema Registry 与数据演化

Kafka broker 对其路由的字节数组内容一无所知。消息载荷是不透明的字节——这是一个刻意的设计选择（零拷贝读取依赖于 broker 从不检查字节）。这种设计很灵活，直到某天生产者团队部署了新 schema 并导致所有消费者崩溃。Schema Registry 的存在就是为了让这种情况不可能发生。

7a. 问题——"部署生产者，破坏消费者"的灾难

第 1 天：生产者将 {"user_id": 7, "event_type": "click", "url": "/foo"} 作为 JSON 发布到主题 clicks。所有五个消费者服务使用各自手写的 JSON 反序列化器进行解析。

第 30 天：生产者团队添加了 referrer 字段，不小心删除了 url （重命名为 path）。生产者在凌晨 2 点部署。到凌晨 2:05，每个消费者都抛出 NullPointerException 对于 event.url。消费者团队不知道发生了什么变化；生产者团队认为这只是一个小改名。 5 个服务宕机 4 小时，工程师排查根本原因。

Schema Registry 本可以在部署时因删除字段违反向后兼容性而拒绝生产者的 schema 注册。该 bug 在构建阶段就会被捕获，而不是在生产环境中暴露。

7b. Confluent Schema Registry——架构

Confluent Schema Registry（开源版，也有 Apicurio、Pulsar 内置 schema 服务、AWS Glue Schema Registry）是一个小型 REST 服务，以 Kafka 主题（_schemas）作为后端以保证持久性。它存储 版本化的 schema 处对 主题 （通常是 <topic-name>-value 。 <topic-name>-key).

Producer write path with Schema Registry:
   1. Producer has Avro/Protobuf/JSON-Schema definition for record class.
   2. On first send, producer calls SR: registerSchema(subject="clicks-value", schema=...).
      SR validates against compatibility mode; if OK, persists, returns schema ID (32-bit int).
      Schema ID is cached in producer JVM forever.
   3. Producer serializes record:
      ┌─────────────────────────────────────────────────────────┐
      │ magic_byte (1B = 0x00) │ schema_id (4B) │ Avro payload  │
      └─────────────────────────────────────────────────────────┘
   4. Producer ships to broker. Broker stores opaque bytes — has no idea Avro is in there.

Consumer read path:
   1. Consumer reads byte array from Kafka.
   2. Parse first 5 bytes: magic byte = 0x00, schema_id = 12345.
   3. Look up schema 12345 in SR (cached). Returns the Avro schema definition.
   4. Avro deserializer uses schema to decode binary payload into a record object.

"魔术字节 + schema ID + 载荷"格式是 wire format 约定，而非 Kafka 协议特性——schema ID 被插入到用户字节中。任何支持该约定的客户端都可以互操作（大多数语言都有 Confluent 序列化器库）。

7c. 兼容模式——防止灾难的规则

Schema Registry 对每个主题强制执行 兼容模式 。注册新 schema 时，SR 会将其与现有的最新（或所有之前）schema 进行检查，并在 API 调用时拒绝不兼容的变更。生产者在 schema 被接受之前无法发布消息。

模式 允许的变更（在新 schema 中） 使用场景
BACKWARD 删除字段。添加带默认值的字段。 默认配置 ——消费者先升级，再升级生产者。这是最常见的 Kafka 模式。
FORWARD 添加字段。删除带默认值的字段。 生产者先升级，再升级消费者。较少见。
FULL 同时满足 BACKWARD 和 FORWARD 约束。 升级顺序混合；最安全，限制最多。
NONE 任何变更均可。 生产环境中切勿使用。
BACKWARD_TRANSITIVE / FORWARD_TRANSITIVE / FULL_TRANSITIVE 与所有历史 schema 比较，而非仅与最新版本比较。 当无法保证旧消费者升级时使用。

7d. 演练：向后兼容的 schema 演化

起始 schema（Avro，v1）：

record ClickEvent {
  long user_id;
  string event_type;
  string url;
}

不良变更（被拒绝）： 重命名 url 与 path。这实际上是 删除 url + 添加必填项 path。读取新数据的旧消费者因缺少 url. 兼容性违规；SR 在注册时拒绝。

良好变更（被接受）： 添加带默认值的可选 referrer 字段。

record ClickEvent {
  long user_id;
  string event_type;
  string url;
  union {null, string} referrer = null;  // new field with default
}

这是 BACKWARD 兼容的：- 旧消费者（使用 v1）读取新生产者（v2）的数据：v1 schema 的 Avro 反序列化器会跳过 referrer 字段（v1 中不存在，但 Avro 可以用 v1 schema 读取 v2 写入的数据，因为 v1 读取器使用自己的 schema 进行投影）。 旧消费者继续工作，忽略 referrer. 新消费者（v2）读取旧生产者（v1）的数据： referrer 字段在载荷中不存在，使用默认值 null 。 新消费者可正常处理旧数据。

迁移顺序： 先将消费者升级到 v2（可同时读取 v1 和 v2 消息），再将生产者升级到 v2（开始发布 referrer）。无需停机窗口。

结论。 Schema Registry 在大规模场景下不是可选的。它是让生产者和消费者团队无需协调会议即可独立部署的契约层。LinkedIn 自 ~2015 年起运行一个分支版本（Liquid）；Confluent 内置发布；AWS 将其嵌入 Glue。成本约为每个注册实例 ~$0.50/秒（3-5 个实例用于高可用），以及几分钟的集成工作。收益是零"schema 不匹配"故障。

§8. Kafka Connect 与连接器生态系统

第一次搭建 Kafka 集群时，你立刻会问："如何将数据从 MySQL 导入它？"或"如何将数据写出到 S3？"Kafka 社区自 2015 年以来的答案是 Kafka Connect ——一个运行预构建 source 和 sink 连接器的独立工作节点集群。它的存在是因为，据统计，人们原本要编写的"消费者"中约 80% 是无状态的连接器，只是将字节从系统 A 搬到系统 B，从头开始编写每一个既浪费又容易出错。

8a. 架构——Connect 工作节点、任务、配置

Connect 集群与 Kafka 集群并行运行。工作节点是 JVM 进程；每个工作节点承载一个或多个 任务。任务是并行工作的单位（例如，一张 MySQL 表 = 一个任务；八个 S3 前缀 = 八个任务）。配置通过 REST API 管理；配置存储在压缩 Kafka 主题（connect-configs, connect-offsets, connect-status）中——Connect 是自引导式的。

8b. Source 连接器——用于 CDC 的 Debezium、JDBC source

Debezium （RedHat，开源）是事实上的 CDC source 连接器。它追踪 MySQL 的 binlog、通过逻辑复制追踪 PostgreSQL WAL、追踪 MongoDB oplog、SQL Server 事务日志、Oracle LogMiner。输出：每个 source 表对应一个 Kafka 主题（<server>.<schema>.<table>）；每条消息是 {before, after, source, op} （op = create / update / delete）。分区键是行的主键，保证了行级别的顺序（§18.5 中的 CDC 管道模式）。

Debezium 优于编写自定义消费者的原因：- 处理快照与 binlog 切换（完整表的初始快照，然后无间隙无重复地切换到流式 binlog）。- 在 Connect 的偏移量主题中追踪 binlog 位置——重启后可恢复，从上次位置继续。- 通过 GTID（全局事务标识符）追踪处理 MySQL 主节点故障转移。- Schema 发现：将 schema 变更事件发送到独立主题，与 Schema Registry 集成。- 数万次生产部署——经过充分验证。

JDBC source 连接器 是更简单的替代方案：每隔 N 秒对表进行轮询查询 SELECT * WHERE updated_at > <last_seen> ORDER BY updated_at 。优点：适用于任何 JDBC 数据库。缺点：无法捕获删除操作（行不可见），如果 updated_at 未更新则可能漏掉更新，对 source 数据库造成轮询负载。仅在无法访问 binlog 时使用。

8c. Sink 连接器——S3、Elasticsearch、Snowflake、JDBC

Sink 连接器从一个或多个 Kafka 主题消费数据并写入外部系统。标准连接器：

  • S3 sink： 按分区缓冲记录，根据时间（rotate.schedule.interval.ms）或大小（flush.size）刷新为 s3://bucket/<topic>/year=2026/month=05/day=22/hour=14/<file>处的 Parquet/JSON/Avro 文件。用于构建数据湖。
  • Elasticsearch sink： 将 Kafka 记录映射为 ES 文档；通过文档 ID = Kafka key 支持 upsert。批量索引。
  • Snowflake sink（Snowpipe Streaming 或批量模式）： 直接将记录加载到 Snowflake 表；Snowflake-Kafka 连接器处理认证、分区文件命名和重试。
  • JDBC sink： 写入任何 JDBC 数据库（Postgres、MySQL、ClickHouse）。将 Kafka 流物化为行存储的标准方式。

向 sink 的精确一次投递 是每个连接器自身关注的问题。S3 sink 通过"先写后重命名" + 在 S3 中追踪偏移量来实现。Elasticsearch sink 依赖文档 ID = Kafka key 的幂等性。JDBC sink 通常依赖基于主键的 upsert。 并非所有 sink 都是精确一次的 ——请阅读连接器文档。

8d. 单消息转换（SMT）

连接器管道通常需要轻量级转换：删除 PII（个人身份信息）字段、按 header 值路由、展平嵌套记录、更改主题名称。如果没有 SMT，就需要在中间插入一个 Kafka Streams 应用。SMT 是在工作节点中内联应用的无状态、逐记录转换。

示例：- ExtractField$Value ——将嵌套字段提升到上层。- MaskField$Value ——替换 email 与 "***@***" 以符合 PII 合规要求。- RegexRouter ——重写主题名称（cdc.prod_db.users → users). - TimestampConverter ——将毫秒/Date/Iso8601 字符串进行类型转换。

不要为"Kafka → S3 加字段重命名和 PII 脱敏"编写自定义消费者。 使用 S3 sink + 2 个 SMT，约 10 分钟的 YAML 配置，而非数天的消费者代码。

8e. Connect 存在的原因与编写自定义消费者的对比

坦率地说：编写一个自定义消费者没问题，但编写五十个是不可持续的。Kafka Connect 摊平了这些成本：

  • 偏移量管理 ——每个连接器使用相同的偏移量提交逻辑，无需每个消费者重新实现。
  • 重启与恢复 ——每个连接器从最后提交的偏移量恢复；无需自己编写该逻辑。
  • 分布式任务重平衡 ——当一个工作节点宕机时，任务自动迁移到其他工作节点。
  • Schema 集成 ——连接器开箱即与 Schema Registry 集成。
  • 单消息转换 ——可组合的转换管道。
  • 运维 REST API ——通过 HTTP 启动/停止/重启连接器，无需重新部署 JVM。

LinkedIn 的 Brooklin 在概念上是跨异构 source（Espresso、MySQL、Oracle、Couchbase）进行 CDC 的 Kafka Connect——同样的洞察：大多数"消费者"都是连接器，因此只需构建一次连接器框架并复用。

§9. 跨地域复制

单个 Kafka 集群受限于一个地域——跨可用区复制没问题（ISR 延迟 5-15 ms），但通过 ISR 进行跨地域复制则不行（50-200 ms RTT 会大幅降低吞吐量）。对于多地域部署， 跨地域复制是一个独立的管道 ，运行在两个单地域集群之间。

9b. 工具——MirrorMaker 2、Brooklin、Confluent Replicator、Cluster Linking

MirrorMaker 2（MM2） 是 Kafka 内置工具，以 Kafka Connect 连接器的形式运行。主题重命名约定：集群 A 中的 source 主题 orders 在集群 B 中变为 A.orders （以 source 集群别名为前缀）。偏移量不被保留（不同集群 = 不同偏移量）。MM2 在 B 中维护一个 __consumer_offsets-translation 主题映射 (source_offset → target_offset) ，用于迁移场景。

Brooklin （LinkedIn，2019 年开源）替换了 LinkedIn 第一代复制器。专为异构 source 设计（不仅仅是 Kafka——还包括 Espresso、Oracle、MySQL）。解耦的控制器用于编排，每个管道抽象为 datastream，在 LinkedIn 规模下跨数百对 source/destination 运行。

Confluent Replicator 是 MM2 的商业版等价物，提供偏移量转换即服务，并与 Schema Registry 集成。在 Confluent Cloud 中广泛使用。

Cluster Linking （Confluent Platform 6.0+，2020 年）是最新模型：无需运行连接器，集群 B 中的 broker 通过内部使用的同款 ISR 拉取协议直接镜像集群 A 的分区。 偏移量 1:1 保留，完全消除了偏移量转换。运维复杂度更低；耦合更紧密。这是行业发展的方向。

9c. 主动-被动 vs 主动-主动

主动-被动 （一个集群处理写入，另一个只读）：- 生产者写入发往 A。- B 是只读镜像；可在该处进行读取以实现地理就近访问。- 灾难恢复（DR）：A 故障时，将生产者重定向到 B；消费者从 B 继续（如果运行 MM2 则使用偏移量转换）。- RPO（恢复点目标）= 复制延迟，通常为 30-200 ms。- RTO（恢复时间目标）= 取决于生产者重连逻辑，通常为 30 秒至 5 分钟。

主动-主动 （两个集群都接受写入；互相复制）：- 更复杂：必须避免复制循环（B 写入 A 数据的副本，然后又镜像回去）。- 解决方案：source 集群前缀或 header replicated.from=A ，复制器根据其进行过滤。- 冲突解决：通常是基于时间戳的最后写入优先，但如果两个地域都向同一个 key 写入， 全局顺序无法协调。要么接受冲突（最终一致性），要么按 key 路由生产者（例如，欧盟用户写入欧盟，美国用户写入美国，不跨地域写入同一 key）。- Pulsar 原生的地理复制是最简洁的主动-主动方案；Kafka 需要仔细的应用层设计。

9d. 偏移量转换与主题重命名模式

棘手的细节：集群 A 上主题 orders 中的偏移量 5,000,000 并不对应集群 B 上主题 A.orders 中的偏移量 5,000,000。它们可能相差很大，因为 B 创建时间较晚、保留数据较少，或批处理方式不同。

MM2 的转换： 每个被复制的批次都会生成一条 (source_topic, source_partition, source_offset) → (target_offset) 映射，写入 <cluster>.checkpoints.internal。从 A 迁移到 B 的消费者读取转换主题，计算"我在 A 上的偏移量是 5M；在 B 上对应的偏移量是 4,827,331。"然后 consumer.seek(target_offset) 并恢复消费。

主题重命名模式（B 中的A.orders ）是业界用于明确标识的标准做法。 不要将 orders 从 A 复制到 B 中的 orders 而不加前缀。 — 如果 B 随后在其自身的生产者上运行 orders，合并是不可逆的。

RPO/RTO 跨区域预算： - 高可用服务（消费者应用）：RPO ~秒级，RTO ~分钟级。MM2 + 自动故障转移。\n- 合规级备份：RPO ~分钟级，RTO ~小时级。每日快照至 S3 + MM2 流式同步。\n- 真正的主动-主动（如全球多租户 SaaS）：RPO ~0（两个区域均接受写入），但预计约 0.01% 的冲突率需要解决。

§10. 安全性 — SASL、mTLS、ACL、加密

承载支付事件或 PII 流量却未启用认证的 Kafka 集群，不过是一个等待爆发的 CVE。生产级安全涉及四个正交维度：认证（你是谁）、授权（你能做什么）、传输加密（网络层）、静态加密（磁盘层）。可按需组合。

10a. 认证 — SASL 变体与 mTLS

Kafka broker 通过以下方式接受认证连接： SASL (Simple Authentication and Security Layer) 框架，叠加在 SSL/TLS 之上。变体如下：

  • SASL/PLAIN。 用户名 + 密码以明文形式通过 TLS 通道传输。配置最简单；密码存储于 jaas.conf broker 和客户端上。在私有网络 + 审计凭证存储场景下可接受。"PLAIN" 指的是线路格式，而非安全级别——凭证由底层 TLS 加密保护。
  • SASL/SCRAM（加盐挑战响应认证机制，RFC 5802）。 挑战-响应机制，密码永不在网络上传输。支持 SCRAM-SHA-256、SCRAM-SHA-512。凭证通过 kafka-configs.sh 管理，存储于 ZooKeeper/KRaft 中。比 PLAIN 更安全，因为 broker 永远不会看到明文密码。
  • SASL/OAUTHBEARER。 OAuth2 / OIDC (OpenID Connect) 令牌。客户端从身份提供商（Okta、Auth0、内部 SSO）获取 JWT (JSON Web Token)，传递给 broker；broker 验证签名和声明。在服务身份集中管理的云原生部署中是标准做法。
  • SASL/GSSAPI（Kerberos）。 企业级，与 Active Directory / 企业 Kerberos 集成。配置最复杂；在大型传统企业中常见。
  • mTLS（双向 TLS）。 基于客户端证书的认证。每个生产者/消费者/broker 持有由企业 CA（Certificate Authority）签发的证书；broker 在 TLS 握手时验证客户端证书。 完全无密码流量。 在云原生环境中常见（服务网格层的 mTLS 延伸至 Kafka）。

broker 可通过独立的 listener 同时提供多种认证机制——例如，内部流量使用 mTLS，外部流量使用 SASL/SCRAM。broker 间流量通常使用 mTLS 或 SASL/SCRAM 进行最简便的双向认证。

10b. 授权 — 按 topic、按主体的 ACL

认证建立身份之后， ACL (Access Control Lists) 决定该身份能执行哪些操作。Kafka 的 ACL 模型：

ACL = (principal, resource, operation, host, permission)

principal:   User:CN=payments-service.prod (from cert) or User:payments-svc (from SASL)
resource:    Topic:orders.created
operation:   Read | Write | Create | Delete | Alter | Describe | ClusterAction | ...
host:        wildcard or specific IP
permission:  Allow | Deny

ACL 配置示例：\n- 生产者 payments-service.prod 可以 Write 与 Topic:orders.* （前缀匹配）。\n- 消费者 fraud-detector.prod 可以 Read 与 Topic:orders.* 且 Read/Describe 对于 Group:fraud-detector。\n- 管理员 kafka-ops.prod 拥有 * 操作于 Cluster:kafka-cluster-prod。\n- 默认 allow.everyone.if.no.acl.found=false → 未知主体被拒绝。

存储于 ZooKeeper（旧版）或 KRaft 元数据中。通过 kafka-acls.sh CLI 或编程式 admin API 管理。 按 topic、按主体隔离是多租户集群的标准做法。

10c. 加密 — 传输层 TLS 与静态加密方案

传输中： TLS 用于客户端-broker 以及 broker-broker 之间。broker 间流量跨 AZ 传输并复制数据——至少在企业网络边界处必须加密。选择：TLS 1.2 或 1.3，配合企业 CA 签发的证书。性能成本：启用 AES-NI (Advanced Encryption Standard New Instructions) 硬件加速前吞吐量下降约 5-10%，现代硬件约 1-3%。实际影响可忽略不计。

静态： Kafka 本身不加密磁盘上的日志段。三种方案：\n- 磁盘级加密（Linux 上的 LUKS、Windows 上的 BitLocker、AWS EBS 加密）。 对 Kafka 透明；加密磁盘上的所有内容，包括页面、索引、KRaft 日志。 最常见的生产配置。 成本：启用 AES-NI 时约 1% 性能损耗。\n- 文件系统加密（eCryptfs、fscrypt）。 按目录加密；粒度更细但不常见。\n- 应用层加密。 生产者在发布前加密 payload；消费者解密。 broker 只看到密文。 最强模型；密钥由 KMS（Key Management Service — AWS KMS、HashiCorp Vault、GCP KMS）管理。适用于运维人员不应看到 payload 的多租户 broker。成本：生产者/消费者 CPU 开销；使 Kafka Streams 变复杂（无法在不逐消息解密的情况下对加密字段进行过滤）。

10d. PII 脱敏策略

即便有加密，PII（Personally Identifiable Information，个人可识别信息）法规（GDPR、CCPA、HIPAA）仍要求额外处理：

  • 通过 SMT 或生产者逻辑进行字段级脱敏。 在发布前剥离或哈希 email, ssn, credit_card ，使 PII 完全不进入 Kafka。
  • 令牌化。 用稳定令牌替换 PII（user_email → user_token_a3f9）；令牌通过独立的保险库服务映射回真实值。Kafka 只持有令牌。
  • TTL（Time-To-Live）有界保留。 将 retention.ms 设置为 GDPR 删除预算（如 30 天），使个人数据自动过期。对 PII 避免使用压缩 topic——压缩 topic 会永久保留数据。
  • 被遗忘权（RTBF）处理。 不可变日志的难题——无法外科手术式地删除单条记录。标准模式：使用用户专属密钥对每位用户的 PII 进行加密，然后 在收到 RTBF 请求时销毁该用户的密钥，使历史加密记录永久不可读 （"加密粉碎"）。LinkedIn、Twilio 等公司均采用此模式。

§11. 监控、运维与"不洁主节点选举"困境

基础设施团队的日常工作。了解哪些指标重要及其含义，是任何数据平台讨论的入门级要求。

11a. Broker JMX 指标 — 核心信号

Kafka broker 通过 JMX（Java Management Extensions，JVM 标准监控接口）暴露数百个指标。运维团队重点告警的指标：

  • BytesInPerSec / BytesOutPerSec （每 broker，每 topic）。 网络吞吐量。趋势驱动容量规划。
  • RequestQueueSize / NetworkProcessorAvgIdlePercent. 入站请求队列深度与网络线程空闲率。队列高/空闲率低 = broker CPU 瓶颈。
  • UnderReplicatedPartitions. 有一个或多个副本脱离 ISR 的分区数量。 在稳定状态下应为 0。 非零值表示存在复制延迟、broker 磁盘/网络饱和或 broker 故障。
  • UnderMinIsrPartitions. ISR 大小已降至 min.insync.replicas以下的分区，意味着 acks=all 写入被拒绝。关键告警。
  • IsrShrinksPerSec / IsrExpandsPerSec. ISR 变更速率。高收缩率而无对应扩展率是危险信号；两者均持续高速变动表示从节点处于临界不健康状态。
  • LeaderElectionRateAndTimeMs. 主节点选举频率。在 broker 重启、网络问题、控制器故障转移时出现峰值。
  • ProduceRequestRateAndTimeMs p99。 客户端端到端生产延迟。用户可见的 SLA 指标。
  • FetchRequestRateAndTimeMs p99。 消费者 fetch 延迟。
  • LogFlushRateAndTimeMs. 页面缓存刷新的频率与耗时。峰值与磁盘压力相关。
  • JVM 堆使用率与 GC 暂停时间。 GC 暂停超过 1 秒是分区主节点稳定性的风险因素。

11b. 消费者 lag 监控 — Burrow、Cruise Control

下游团队最重要的单一指标： 消费者 lag = log_end_offset - committed_offset （按消费者组、分区）。告诉你"消费者是否跟得上生产者？"

朴素监控：轮询 __consumer_offsets，与 broker offset 比较。问题：无法扩展至数千个消费者组；消费者暂停但未中断时（如批次间隔）已提交的 offset 陈旧；无法区分"消费者落后 1 万条消息正在追赶"与"消费者已在 1 万条 lag 上停滞 30 分钟"。

Burrow （LinkedIn，OSS）通过 窗口内的状态跟踪解决这一问题：\n- 追踪每（组，分区）已提交 offset 的趋势。\n- 区分"lag 高但在下降"（消费者正在追赶）与"lag 高且稳定"（消费者卡住）。\n- 输出状态：OK / WARNING / ERR / STALL / STOP。\n- 用于设置不会因良性 lag 峰值而误告警的告警规则。

Cruise Control （同为 LinkedIn，OSS）范围更广：它是集群的自动负载均衡器。监控 broker 资源利用率，在 broker 负载不均时运行分区重新分配，以重新平衡磁盘、网络和 CPU。没有 Cruise Control，负载均衡只能靠手动执行 kafka-reassign-partitions.sh ——在 LinkedIn 规模下既痛苦又容易出错。

11c. 不洁主节点选举困境

重大运维决策：当所有同步副本均故障时，是否以持久性换可用性？

Scenario: Topic 'payments' partition 7. ISR was {B-7, B-31, B-88}.
T0: B-7 (leader) has offset 5,000,000. B-31 and B-88 caught up.
T1: B-88 falls behind. ISR shrinks to {B-7, B-31}.
T2: B-31 falls behind. ISR shrinks to {B-7}.
T3: B-7 hard-crashes (disk failure, kernel panic).
T4: Controller looks at ISR = {} (empty — leader was alone).

OPTION A: "Clean" — wait for B-7 to come back.
   - Partition is UNAVAILABLE until B-7 (or another ISR-tracked replica) returns.
   - If B-7's disk is dead, partition might be down for hours.
   - Guarantees: no data loss; offsets remain consistent.

OPTION B: "Unclean leader election" — elect from non-ISR replicas (B-31, B-88).
   - Partition becomes AVAILABLE quickly.
   - Cost: B-31 was behind by some N messages; those messages are LOST.
   - Producers and consumers see offset rollback: HW was 5,000,000, now 4,990,000.

配置项： unclean.leader.election.enable 按 topic。自 Kafka 0.11 起的默认值： false （优先保证正确性而非可用性）。0.11 之前的默认值： true （优先保证可用性而非正确性）。

总结： 支付/账本/CDC = false。广告展示/心跳/非关键分析 = true。决策按 topic、按数据重要性而定。没有统一正确答案；这是业务驱动的持久性与可用性的权衡取舍。

11d. 运维节奏

  • 滚动 broker 重启 — 修补 broker，重启，等待 ISR 追赶，重复。在 LinkedIn 拥有 4000 台 broker 的规模下，这需要数周时间；自动化是必须的。
  • 分区重新分配 — 在 broker 之间迁移分区（容量再平衡、下线退役）。Cruise Control 可自动化；手动执行 kafka-reassign-partitions.sh 有风险，因为限速是按 broker 而非按分区，容易打满网络。
  • Topic 创建 — 必须遵守集群级别的分区总数预算。KRaft 可支持超过 100 万个分区，但再平衡和元数据传播仍有开销。
  • 容量规划 — 预测公式： daily_msg_rate × avg_size × RF × retention_days。为热点 key 倾斜预留 30% 余量，为副本追赶突发容量预留 50%。

§12. 成本经济学 — 存储、网络、S3 原生

成本分析在真实设计中占据对话的半壁江山，却鲜少出现在初级工程师的 PPT 里。Kafka 在大规模场景下的两大主要成本：存储与跨 AZ 带宽。

12a. 存储 — 保留期主导，副本因子成倍放大

算术公式：存储成本 = daily_ingest × retention_days × replication_factor / compression_ratio.

LinkedIn 规模示例：7T 条消息/天 × 平均 1 KB × 7 天保留 × RF=3 / 3x 压缩 = 约 49 PB 原始磁盘数据。以本地 NVMe 约 $0.10/GB-月计算，仅消息日志的磁盘费用约为 $500 万/月。（LinkedIn 的实际成本更低，因为他们自有硬件；云端等价成本高 2-3 倍。）

降低存储成本的调节手段： - 缩短保留期。 默认 7 天；能否改为 3 天？1 天？权衡：缩短重放窗口。许多生产工作负载仅需 2-3 天的运维重放；更长的保留期是"以防万一"的冗余。\n- 更好的压缩。 lz4 → zstd 通常额外减少 30-40%。无成本优化。\n- 分层存储（§4g）。 近期数据（24 小时内）使用本地 NVMe，其余使用 S3。S3 约 $0.023/GB-月 vs NVMe $0.10 → 卸载数据便宜 4 倍。在 7 天保留、24 小时本地存储的配置下，约 85% 的数据移至 S3 —— 整体成本约降低 70%。\n- 非关键 topic 将 RF 从 3 降至 2。 节省 33%，但失去单 broker 故障容忍。通常不值得。

12b. 跨 AZ 流量 — 隐性开销

云厂商对跨 AZ 流量收取约 $0.01/GB。每次 ISR 复制都跨 AZ（副本有意分布在 3 个 AZ）。每个从非本 AZ 主节点 fetch 的消费者也是如此。

实际示例：向某 topic 持续写入 500 MB/秒，RF=3 分布于 3 个 AZ。\n- 主节点在 AZ-1，从节点在 AZ-2 和 AZ-3。\n- 复制流量：500 MB/秒 × 2 个从节点 = 复制路径 1 GB/秒跨 AZ。\n- 1 GB/秒 × 86400 秒 × 30 天 = 约 2.6 PB/月的跨 AZ 流量。\n- 按 $0.01/GB = 仅复制出口费用约 $26K/月。

消费者方面：若 AZ-2 的消费者从 AZ-1 的主节点 fetch，每次 fetch 均跨 AZ（$0.01/GB）。成本量级相同。

缓解措施： - 机架感知。 配置 broker 的 broker.rack=us-east-1a 等参数。Kafka 的 RackAwareReplicaSelector （消费者侧）优先从相同机架/AZ 的副本读取。可将消费者跨 AZ 流量减少一半甚至更多。\n- AWS 于 2024 年宣布 MSK 免收跨 AZ 费用。 对 AWS 用户的成本计算影响巨大。并非普遍适用。\n- 加大压缩力度。 相同的 BytesIn 降幅同比例降低存储和跨 AZ 成本。

12c. S3 原生经济学 — WarpStream / AutoMQ

S3 原生变体（WarpStream、AutoMQ、RedPanda 的 BYOC）从根本上重构了成本方程：

Classic Kafka cost split (LinkedIn-style):
   45% NVMe storage   (replicated 3x)
   30% Cross-AZ networking (replication + consumer fetch)
   20% Compute (brokers)
    5% Operations + control plane

S3-native Kafka cost split (WarpStream-style):
    5% Local cache (small, ephemeral)
   55% S3 storage (single-copy durability, 11-nines)
   10% S3 PUT/GET costs
   25% Compute (stateless brokers, smaller and elastic)
    5% Operations + control plane

Net: 30-50% cost reduction at high retention; 200ms p99 produce latency (vs 5-15ms).

这是以延迟换成本的权衡：S3 分段上传的 p99 约 150-300ms，而本地 NVMe ISR 复制约 5-15ms。如果你的工作负载能容忍 200-500ms 的生产 p99（大多数分析、日志传送、遥测归档），S3 原生方案成本会显著更低。如果需要 <50ms（支付、实时竞价），经典 NVMe 副本 Kafka 仍是答案。

总结： S3 原生前沿正在快速演进（WarpStream 于 2024 年被 Confluent 收购）。预计"经典" Kafka 与 S3 原生将在 2020 年代后期针对不同工作负载并存；新的高保留期部署默认采用 S3 原生。

§13. KRaft 模式 — ZooKeeper 之后的新生

这是自 2011 年 Kafka 原始发布以来最重大的架构变更。值得深入理解，因为截至 2026 年，大多数生产集群的迁移工作仍在进行中。

13a. ZooKeeper 带来的问题

KRaft 之前，Kafka 使用 ZooKeeper 管理集群元数据：broker 成员关系、topic 配置、分区主节点、ACL、控制器选举。ZooKeeper 是外部依赖——独立的集群、独立的运维团队、独立的故障模式。

ZooKeeper 在大规模场景下的痛点：\n- ZNode 数量上限约 20 万。 LinkedIn 在各集群累计约 700 万个分区时逼近上限；每个分区需要多个 ZNode（主节点、ISR、配置）。\n- Watcher 扇出扩展性差。 每个 broker 监视数十个 ZNode；4000 台 broker 同时监视同一父 ZNode 导致惊群问题。\n- 控制器故障转移需要数分钟。 控制器 broker 宕机后，新控制器需从 ZooKeeper 重新加载所有分区状态——在 700 万个分区时，这意味着约 5-15 分钟的元数据变更不可用。\n- 运维复杂性。 独立的 ZK 集群需要单独配置、监控、升级和加固。是常见的故障来源。

13b. KRaft 架构 — 控制器 quorum、元数据日志

KRaft（Kafka Raft，KIP-500）用基于 Raft 的控制器 quorum 取代 ZooKeeper， 内嵌于 Kafka 本身。架构：

┌──────────────────────────────────────────────────────────────────┐
│                     KRaft-based Kafka Cluster                     │
│                                                                   │
│   ┌──────────────────────────────────────────────────────────┐   │
│   │  Controller Quorum (3 or 5 nodes, Raft-replicated)       │   │
│   │  ┌──────────┐  ┌──────────┐  ┌──────────┐                │   │
│   │  │ Ctlr-1   │  │ Ctlr-2   │  │ Ctlr-3   │                │   │
│   │  │ (leader) │  │ (follower│  │ (follower│                │   │
│   │  └──────────┘  └──────────┘  └──────────┘                │   │
│   │                                                          │   │
│   │  Metadata stored in: __cluster_metadata topic            │   │
│   │  (the controller quorum IS the Raft log holding metadata)│   │

│                            │ MetadataFetch RPC                    │

└──────────────────────────────────────────────────────────────────┘

核心变化：\n- 元数据日志是一个普通的 Kafka topic (__cluster_metadata）。每次元数据变更都是一次追加；控制器是生产者；broker 是消费此 topic 的消费者。自食其力。\n- 通过 Raft 进行控制器故障转移。 3 个或 5 个控制器组成 Raft quorum；若主节点宕机，从节点在约 1-3 秒内赢得下次选举。\n- broker 不再连接 ZooKeeper。 它们通过标准 fetch 协议从控制器拉取元数据 topic。\n- 模式： 控制器可在小集群中与 broker 合并（process.roles=broker,controller），或在大集群中独立部署（process.roles=controller 用于专用控制器节点）。

13c. KRaft 的意义 — 控制器故障转移、分区扩展性

控制器故障转移从分钟级缩短至秒级。 旧方案：控制器 broker 宕机 → 新控制器 broker 从 ZooKeeper 读取所有元数据 → 在 700 万个分区时耗时 5-15 分钟。新方案：新控制器主节点已通过 Raft 完成元数据复制 → 立即接管。

支持超过 100 万个分区。 ZooKeeper 的 ZNode 上限是硬性壁垒；KRaft 控制器 quorum 遵循常规 Kafka topic 语义（压缩、分段）进行扩展。Confluent 报告 KRaft 在基准测试中可处理每集群 200 万+ 个分区。

运维简化。 无需再运维 ZooKeeper 集群。集群启动更快（无需先配置 ZK 集群）。更紧凑的安全模型（一套认证/ACL 系统，而非两套）。故障面更小。

13d. 从 ZooKeeper 迁移至 KRaft

迁移 非 不是简单地切换一个配置项——这是一个多步骤过程：

  1. 将 broker 升级至 3.4+ ，该版本同时支持 ZK 和 KRaft（双写模式）。
  2. 配置 KRaft 控制器 quorum （3 或 5 个节点）与现有 ZooKeeper 集群并存运行。
  3. 迁移元数据。 控制器从 ZooKeeper 读取现有元数据，并将其写入 __cluster_metadata。ZK 与 KRaft 在迁移窗口期间保存相同的元数据。
  4. 滚动重启 broker 于 migration 模式 — 每个 broker 从基于 ZK 的元数据切换到基于 KRaft 的元数据。
  5. 切换完成。 在 broker 配置中禁用 ZooKeeper 依赖；ZK 集群现已下线。

Kafka 4.0（2024 年发布）完全移除了 ZooKeeper 支持；KRaft 是唯一模式。迁移路径有完善的文档，但在运维层面影响重大 — 大多数大型生产集群在 2024-2025 年期间正处于迁移进行中。

§14. 毒丸消息与死信队列

§18.2 介绍了 DLQ 的基本概念。本节深入探讨其模式、朴素 DLQ 实现的故障模式以及替代方案。

14a. 毒丸 — 什么使消息无法被处理

"毒丸"是指消费者无论重试多少次都无法成功处理的记录。来源： 格式错误的负载。 Schema 不匹配（例如，生产者在消费者升级前写入了 v2 schema）、字节损坏、编码 bug。 逻辑违规。 外键引用了已删除的实体、无效的状态转换、上游未捕获的业务规则违规。 外部系统故障。 Stripe API 返回 400 Bad Request 因为 ID 过期；Postgres 返回约束违规。 资源耗尽。 记录过大无法处理，需要经过 N 次已永久失败的数据丰富跳转。

病理现象：消费者 poll() 返回毒丸；处理器抛出异常；消费者重试；相同异常； 分区 lag 无限增长 同时组内所有消费者都在等待这一条坏记录。向生产者形成背压，最终随着保留时间延长，broker 端磁盘被填满。

14b. DLQ 基础 — 有限重试 + 重定向

标准模式：

for record in consumer.poll():
    try:
        process(record)
        commit_offset(record.offset)
    except RetriableError as e:
        if retries(record) < MAX_RETRIES:
            sleep(backoff(retries(record)))
            retry
        else:
            dlq_producer.send("orders.dlq", record, headers={"error": str(e)})
            commit_offset(record.offset)  # source partition moves on
    except PermanentError as e:
        dlq_producer.send("orders.dlq", record, headers={"error": str(e)})
        commit_offset(record.offset)

运维人员监控 orders.dlq；在分流时可以修复数据（修正行、修复 Stripe ID），重新发布到 orders.created，处理恢复正常。 DLQ 是不可或缺的运维界面。

14c. 为什么 DLQ 有时是错的 — 顺序丢失与"停车场"模式

基础 DLQ 模式隐藏的缺点： DLQ 会永久丢失按 key 的顺序。

示例：CDC 管道（按 key 有序）。记录 1： update user_42 set email=...。记录 2：同一用户， update user_42 set name=...。记录 1 失败（Postgres 短暂中断），进入 DLQ。记录 2 正常成功。运维人员随后从 DLQ 重放记录 1。 User-42 的行现在反映的是记录 1 在记录 2 之后应用 — 顺序错误，最终状态错误。

对于顺序敏感的工作负载（CDC、事件溯源），DLQ-and-skip 是不正确的。替代方案：

停车场模式。 消费者不路由到 DLQ-and-continue，而是 暂停该分区 (consumer.pause(partition)），并将坏记录停放到旁路存储（Redis、状态 DB 或人工审核队列）。分区在该 offset 处保持不动。 顺序得到保留；其他分区的下游处理继续进行。运维人员修复坏记录；消费者恢复（consumer.resume(partition)）并重试。

   Partition order-events-7 stalls at offset 5_000_000 (bad record):

   Pre-bad-record:   5_000_000 (committed)  ──> continuous flow
   Bad record:       5_000_001 (parked in Redis, partition paused)
   Post-bad-record:  5_000_002 ─ 5_000_500   (queued in broker, not consumed)

   Other partitions (0, 1, 2, ..., 6, 8, ..., N) continue processing normally.
   Operator review of parked record → fix → resume partition → records 5_000_001 onward.

代价。 被停放分区的 lag 会增长；partition-7 记录的 SLA 被打破；partition-7 的消费者实际上处于空闲状态。 对有序工作负载值得； 对无序工作负载是错误选择（DLQ-and-skip 对遥测、点击流等场景完全适用）。

带级联 TTL 的重试 topic。 Spring 的模式： orders.retry.5min, orders.retry.30min, orders.retry.2hr, orders.dlq。失败的记录以递增延迟级联流经重试 topic。适用于故障大多是瞬时性的场景（外部服务抖动）且顺序无关紧要 — 实现了"等待后重试而不阻塞源分区"的语义。 丢失顺序，与 DLQ 一样；区别仅在于重试的形态。

结论。 "DLQ"只是简称；正确的设计取决于顺序要求。有序工作负载使用停车场模式。无序工作负载使用 DLQ。不加思考地混用会导致静默的数据损坏 — 这是最糟糕的一种。

§15. 事务性发件箱模式 — 解决双写问题

§20（DB 即队列）和 §23 用例中均有提及。本节介绍其规范版本，因为该模式是一个反复出现的构建模块。

15a. 双写问题

当服务需要同时更新数据库并发布到 Kafka 时，朴素的做法是：

def create_order(order):
    db.insert("orders", order)         # write to DB
    kafka.send("orders.created", order) # publish event

故障模式：DB 写入成功；Kafka 发送失败（broker 宕机、网络抖动）。DB 有该行，但没有事件。下游消费者（账单、风控、搜索）永远看不到该订单。Kafka 发送成功；DB 写入回滚（约束违规、事务失败）。下游消费者处理了一个在 DB 中根本不存在的订单的"已创建"事件。 它们稍后尝试查找时会失败。 两者都成功；服务在两者之间崩溃；现在某些重复写入逻辑必须进行对账。

无法使对两个独立系统的两次写入保持原子性 不借助 2PC（两阶段提交），而跨 Kafka 和 Postgres 的 2PC 既不被支持，即便支持也会慢得无法接受。正确答案是将其变为一次写入，而不是两次。

15b. 发件箱模式

原子写入仅写入数据库。发件箱表位于同一数据库中，与业务数据在同一事务中写入：

BEGIN;
  INSERT INTO orders (id, customer_id, total, ...) VALUES (42, ..., ...);
  INSERT INTO outbox (event_id, aggregate_type, aggregate_id, event_type, payload, created_at)
       VALUES (uuid_v4(), 'order', 42, 'order.created',
               '{"order_id":42, "customer_id":..., "total":...}',
               NOW());
COMMIT;

两行原子提交，因为它们在同一个 Postgres 事务中。 发件箱行现在是一条必须发布事件的持久记录。 丢失它意味着同时丢失订单本身。

然后由一个独立进程 — 通常是 Debezium（§8b）— 追踪数据库的 binlog 并将发件箱行发布到 Kafka：

   ┌──────────┐         ┌──────────┐       ┌────────────────┐
   │  App     │──BEGIN──│ Postgres │       │   Kafka topic  │
   │          │ INSERT  │          │       │  orders.created│
   │          │ orders  │ ┌──────┐ │       │                │
   │          │ INSERT  │ │ WAL  │ │       └────────▲───────┘
   │          │ outbox  │ │      │ │                │
   │          │ COMMIT  │ └───┬──┘ │                │
   └──────────┘         └─────┼────┘                │
                              │                     │
                              │ tails WAL           │
                              ▼                     │
                       ┌────────────┐               │
                       │ Debezium   │──publishes────┘
                       │ CDC        │
                       └────────────┘

发布后，Debezium 将其 WAL 位置提交到自己的 Kafka offsets topic。重启时，它从上次发布的 WAL 位置恢复 — 至少一次投递（幂等生产者 + Kafka 事务可实现精确一次写入 Kafka）。

15c. 为什么发件箱优于"先写 DB 再写 Kafka"

双写问题消失了，因为根本没有双写。具体来说：如果订单 INSERT 失败，发件箱 INSERT 在同一事务中并回滚。 不会产生幻象事件。 如果订单 INSERT 成功，但服务在"写入 Kafka"之前崩溃，无关紧要 — 发件箱行是持久的，Debezium 会按自己的节奏将其处理。如果 Kafka 宕机一小时，发件箱会增长，但不会丢失任何数据。Kafka 恢复后，Debezium 会追上进度。

延迟成本： 事件在业务写入后约 50-500ms 到达 Kafka（Debezium 的轮询延迟 + WAL 追加延迟）。对于消费者异步处理的事件驱动架构，这是不可见的。

运维成本： Debezium 是需要运维的基础设施（Connect 集群、插件、schema 管理）。这是值得的，因为替代方案是永久性的数据不一致。

存储成本： 发件箱表会增长。标准做法：由独立进程（或 Debezium 内置的 outbox.event.deletion.strategy=delete）在发布确认后删除发件箱行，使表大小保持有限。

15d. 变体与替代方案

  • 内置 Debezium 支持的发件箱 — Debezium 有一个 outbox SMT，用于提取 aggregate_id 作为 Kafka key，路由到正确的 topic，并去除发件箱表的信封。省去了样板代码。
  • 基于轮询的发件箱 — 不使用 Debezium，工作进程轮询 SELECT * FROM outbox WHERE published=false ORDER BY created_at LIMIT 100 FOR UPDATE SKIP LOCKED，发布，设置 published=true。运维更简单；错过了 binlog 追踪的效率优势；在没有幂等生产者的情况下，轮询后崩溃可能产生重复。
  • 监听/通知通道（Postgres LISTEN/NOTIFY） — DB 向监听器推送通知，监听器发布到 Kafka。延迟低于轮询，但 notify 没有持久性保证（如果监听器宕机，notify 会丢失）。结合发件箱表作为数据来源。
  • Saga 模式替代方案 — 不用发件箱 + 事件，而是由 saga 执行一个工作流：每个步骤写入"待处理"记录，并分发下一步骤。对于长时间运行的多服务工作流更清晰；发件箱对"原子写入 + 事件发布"语义更清晰。

使用方： Stripe（§24 提及）、Shopify、Airbnb、LinkedIn（Brooklin 本质上是跨所有服务的托管发件箱基础设施）。 总结： 发件箱模式是解决"写入 DB + 发布事件"原子性的规范、无争议方案。任何同时执行这两项操作的生产系统都必须有发件箱或等效方案；替代方案是静默的数据漂移。

§16. 不同规模下的容量上限

吞吐量、延迟和存储上限跨越 6 个数量级以上。瓶颈在每个层级都会发生转移。

小团队 — 10K msg/sec（3 broker 集群）。 初创公司为搜索 + 账单记录活动事件。3 个 broker 分布在 3 个可用区，RF=3。平均 500 B → 写入 5 MB/sec，含副本 15 MB/sec。保留 7 天 ≈ ~9 TB。p99 生产延迟 5-10 ms。 瓶颈：技术层面无瓶颈。 在 10K msg/sec 运行 Kafka 的成本与 1M 相同；低于 ~100K msg/sec 时，正确答案通常是"使用托管服务（MSK、Confluent Cloud）"。

中等规模事件总线 — 1M msg/sec（15-30 broker）。 黑色星期五的电商微服务，12 个消费者组。写入 500 MB/sec，含副本 1.5 GB/sec，5 倍扇出时 broker 流量约 ~3.5 GB/sec。通过分层存储保留 30 天。p99 生产延迟 5-15 ms。 瓶颈：分区数量规范 + schema registry。

LinkedIn 规模 — 80M msg/sec（4,000 broker，~100 个集群）。 公开数据（2019-2024）：~7T msg/天，写入 ~18 GB/sec，读取 ~70 GB/sec，~100K 个 topic，~7M 个分区，p99 约 ~10 ms。热数据量约 ~32 PB。 瓶颈转移到存储和分区元数据。 KRaft 之前，ZooKeeper 在每集群 ~200K 个分区处触及上限；KRaft + 分层存储突破了这一限制。

超大规模 — Netflix Keystone，10 PB/天。 ~700B 事件/天，~8M 事件/sec，多个区域集群进行跨区域复制，通过 Schema Registry 严格使用 Protobuf。 瓶颈：流处理层（Flink 状态、sink 背压），而非 Kafka 本身。

S3 原生前沿 — WarpStream / AutoMQ（2023-2024）。 小型集群（10-30 节点）持续吞吐量 ~1 GB/sec，p99 ~200-500 ms，高保留场景下成本降低 30-50%。 瓶颈：S3 前缀吞吐量（每前缀每秒 ~5,500 GET / 3,500 PUT） 促使积极采用 1-2 秒批处理以摊销 S3 PUT 成本。

瓶颈会移动：小规模时是网卡，中等规模时是分区元数据，LinkedIn 规模时是存储容量，S3 原生规模时是 S3 PUT 速率。每个层级的限制都不同。

关键注解： 每次写入都要带分区 key (hash(key) % num_partitions）— 对于订单事件，key 是 order_id，对于车辆遥测是 vehicle_id，对于点击流是 user_id 或 session_id；key 的选择决定了什么是有序的以及热 key 落在哪里。 副本因子 3 是标准配置 — 能承受任何单节点损失；副本分布在多个可用区时可承受单可用区损失。 消费者组是独立的 — 新的分析团队作为自己的组订阅，从 offset 0 读取，不会影响任何人。 元数据层虽小但至关重要 — 负责 broker 成员关系、分区 leader、topic 配置；如果它宕机，稳态 IO 继续，但集群变更会暂停。

§18. 日志和队列固有的难题

任何非平凡部署都会遇到的六个问题，以不同领域为例说明。

18.1 精确一次语义 — 支付发件箱

在支付平台中，每一个 payment.charged 事件必须触发恰好一次客户扣款和恰好一次账本更新。

简单粗暴的修法： 生产者重试直到收到 ack，消费者在处理后提交 offset。

为什么会出问题： 在 T0，Stripe 服务发布 payment.charged{order=42}；broker 写入、复制、确认。在 T1，生产者的 TCP 连接在收到 ack 前断开；生产者重试。没有幂等生产者，broker 会写入重复记录 — offset 100 和 101 现在持有相同的事件；客户被扣款两次。另外，消费者读取 offset 100，完成扣款，在提交 offset 前崩溃。重启后，最后提交的 offset 是 99；消费者重新读取，再次扣款。 客户被扣款三次。

真正的修法： 需要三个组件：（1） 幂等生产者 (enable.idempotence=true）— broker 追踪（producerId, partition）→ 最高序列号，拒绝重复。（2） 事务性生产者 (transactional.id）— 在一个事务中原子地写入 N 个分区并提交消费者 offset。（3） 消费者 isolation.level=read_committed — 跳过已中止或进行中的事务。

注意： 这是精确一次 在 Kafka 内部。端到端到 Stripe 的外部 API 需要 Stripe 接受幂等 key。结论："Kafka 精确一次仅在 Kafka sink 内提供精确一次；外部边界仍然需要应用层面的幂等性。"

18.2 队头阻塞 — 点击流分析

点击流摄入 Web 事件。offset 5,000,000 处的某个事件负载格式错误；消费者每次重试都抛出异常；整个分区停滞。

简单粗暴的修法： 指数退避重试。

为什么会出问题： 在毒丸消息上无限重试。offset 5,000,000 之后该分区上的所有记录都被饿死。Lag 线性增长。

真正的修法： 死信队列（DLQ）+ 有限重试。 经过 N 次重试后，将失败记录发布到 clickstream.dlq ，在 header 中附带错误信息，提交源 offset，继续前进。运维人员驱动的流程对 DLQ 进行分流。

Kafka 没有原生 DLQ — 需要应用逻辑（或 Kafka Connect 内置的 DLQ）。RabbitMQ 将 DLX（死信交换机）作为一等公民。SQS 有每队列 DLQ。 忘记配置 DLQ 是最常见的 Kafka 生产故障之一。

18.3 热分区 — IoT 遥测

一个类似 Tesla 风格的车队遥测 topic 有 256 个分区，按 vehicle_id进行 key 分区。某辆车有 bug，以正常速率的 1000 倍发送数据。一个分区承受 50K msg/sec；其余 255 个分区只有几百。

简单粗暴的修法： 增加更多分区。

为什么会出问题： 热 key 仍然哈希到同一个分区。对正在运行的 keyed topic 重新分区非常痛苦 — 现有数据留在旧分区上，消费者必须同时处理两者。

真正的修复 — 三种方案： (1) Key 加盐 — vehicle_id || ":" || (event_seq % 8) 将一辆车的数据分散到 8 个分区。失去严格的按车辆顺序；下游按时间戳排序。（2） 两级 topic — 原始数据粗粒度分区（按地理位置），流处理器对热 key 用动态分片后缀重新分 key。（3） 将热 key 单独划到专用管道 — 幂律分布很常见；100 万辆车中排名前 100 的车辆往往产生 50% 的事件。

本能反应："实际的顺序要求是什么？"通常"按车辆有序"是过度规定的；"按时间戳对账的最终一致性按车辆"就够用了。

18.4 消费者再均衡暂停 — 微服务扇出

一个类似 Airbnb 风格的预订事件消费者组有 64 个消费者读取 256 个分区。一个消费者宕机。组协调器触发再均衡：所有 64 个停止，归还分配，重新分配，恢复。旧式"急切"再均衡是全停顿，持续 5-10 秒。在 1M msg/sec 时，这意味着 500-1000 万条消息的 lag。

简单粗暴的修法： 让消费者更可靠。没有帮助 — 伸缩事件（部署、自动伸缩）无论如何都会导致再均衡。

真正的修法： 协作增量再均衡 （KIP-429）— 只有需要移动的分区才真正移动；其他消费者不暂停。每个消费者的暂停时间从秒级降至约 ~100 ms。

更深层的修复： 静态组成员身份 (group.instance.id）。稳定的身份；在 session.timeout.ms （默认 45s）内重启可以拿回 相同的分区。对于有状态消费者（带本地状态的流处理器）至关重要。

18.5 顺序与扩展的矛盾 — CDC 管道

CDC 管道（Debezium → Kafka → 数据仓库）需要按顺序应用行变更，以使数据仓库中的行与源 DB 行保持一致。

简单粗暴的修法： "一个分区，一个消费者。"

为什么会出问题： 单分区写入约 ~100 MB/sec，单消费者约 ~50-200K msg/sec。每秒执行 100 万行变更的 DB 快 5-10 倍。

真正的修法： 按主键分区 — 同一行的所有变更都落在同一分区，保留了按行顺序。不需要跨行顺序；数据仓库独立重建各行。现实："实际的顺序范围是什么？" — 几乎总是按 key，而不是全局的。

LinkedIn 的 Brooklin 按源表主键分区，从而在数百个分区上保留了按行顺序。

18.6 生产者背压 — 日志传输

类似 Cloudflare 风格的边缘日志传输：200 个 POP 各以 500K msg/sec 向中心 Kafka 生产数据。集群出现棕色故障（控制器 GC 暂停）。

简单粗暴的修法： "生产者发送是异步的，会在本地缓冲。"

为什么会出问题： 生产者的 buffer.memory （默认 32 MB）被填满。然后 send() 阻塞或抛出异常。OOM（内存溢出）只需几分钟。

真正的修复 — 四层： (1) Producer：设置上限，将 buffer.memory设置为约100 ms，在阻塞时向上游调用方施加背压。(2) Broker 配额（ max.block.ms ）通过响应延迟来限制超额生产者。(3) 上游熔断器——若 p99 生产延迟在30秒内持续超过100 ms，则跳过 Kafka（根据数据类型选择 fail open 或 fail closed）。(4)quota.producer.defaultOutbox 模式 作为持久化兜底——在与业务写入相同的事务中将事件写入数据库 outbox；CDC 尾随 binlog 写入 Kafka。若 Kafka 宕机，outbox 积压；数据不会丢失。 §19. 故障模式演练

Broker 在写入途中（批量写入中途）崩溃。

Leader B-7 已写入一个16 KB 批次的一半；从节点尚未拉取。Controller 检测到 B-7 下线（约10秒超时），从 ISR 中选出新 leader（假设为 B-31，它未拉取到部分数据）。Producer 发往 B-7 的在途请求超时，重试至 B-31；幂等 producer 负责去重。B-7 重启后扫描活跃 segment，在尾部发现部分记录（CRC 校验失败）， 截断至最后一条完整写入的记录边界 ，成为从节点并追赶进度。持久性要点： 崩溃前最后一个完整写入的批次。 在 produce ack 与复制之间发生崩溃。

使用 时这不应发生——leader 在 ISR 完成复制前不会 ack。使用 acks=all 时则是经典数据丢失窗口：leader 写入并 ack，在复制前崩溃；新 leader 来自未见到该批次的从节点；批次丢失。 acks=1 防御措施： 对重要数据切勿使用 ； acks=1 是 LinkedIn 的默认设置。 acks=all + min.insync.replicas=2 元数据 leader 宕机（KRaft / ZooKeeper / Pulsar 元数据）。

5节点 KRaft 仲裁，controller leader 宕机。剩余4个 controller 发起 Raft 选举；新 leader 在约5-10秒内产生，回放元数据日志，广播至各 broker。 Controller 仲裁上的 Raft 日志——Raft 多数派保证已提交条目（≥3/5 节点已确认）在少数派故障时不丢失。在此期间，broker 继续服务进行中的生产和消费请求——controller 仅在元数据 崩溃前最后一个完整写入的批次。 变更 时才需要。.

网络分区（脑裂风险）。 AZ-1 拥有 leader B-7 + ISR {B-7, B-31} + controller 多数（5中的3）。AZ-2 拥有 B-88 + 2个 controller。网络分区发生。AZ-1 继续服务写入；ISR 收缩为 {B-7, B-31}。AZ-2 的 B-88 无法从 B-7 拉取数据；30秒后 controller 将其移出 ISR。AZ-2 的2个 controller 属于少数派——无法选出新 leader。 不存在脑裂。 崩溃前最后一个完整写入的批次。 Controller 仲裁上的 Raft 多数派——这正是为何5节点仲裁跨3+个 AZ 是标准做法。

永久节点丢失（broker 磁盘故障）。 B-7 宕机，磁盘不可恢复。新 broker B-7' 以相同 ID 被部署。Controller 发现 B-7 缺席；leadership 转移至其他 ISR 成员。B-7' 以无数据状态加入。对于 B-7 曾担任从节点的每个分区，leader 从 offset 0 发送 Fetch——B-7' 重放完整日志。 多 TB 级分区需要数小时的网络 IO。 分层存储有所帮助——仅复制本地层数据；旧数据驻留在 S3 中且对所有 broker 可访问（Confluent 报告分层存储使 broker 替换速度提升5-10倍）。

Consumer group 在故障时发生再平衡。 Consumer C5 挂起。Group coordinator 发起再平衡。协作式增量再平衡：仅重新分配 C5 的分区；其他分区继续处理。 微妙问题： 若 C5 正在处理某条记录但尚未提交 offset，新 consumer 将从 C5 最后已提交的 offset 重新读取→产生重复处理。 幂等 consumer 是强制要求；这个问题无法通过架构设计规避。

§20. 为何不直接用数据库表充当队列？

这是最常见的 L4/L5 设计建议；其故障模式揭示了 正是 消息队列存在的意义。

CREATE TABLE events (
  id BIGSERIAL PRIMARY KEY, type TEXT, payload JSONB, created_at TIMESTAMPTZ,
  consumed_by_billing BOOLEAN DEFAULT FALSE,
  consumed_by_fraud BOOLEAN DEFAULT FALSE
);
-- Consumer polls:
SELECT * FROM events WHERE consumed_by_billing = FALSE
  ORDER BY created_at LIMIT 100 FOR UPDATE SKIP LOCKED;
-- ... process ... then:
UPDATE events SET consumed_by_billing = TRUE WHERE id IN (...);

故障1：写入吞吐量上限。 以1M msg/sec × 5个 consumer group = 500万次状态更新/秒。Postgres 在单个主节点上峰值约5万次写入/秒——慢20-100倍。分100个分片仍慢5倍。

故障2：原地更新导致索引膨胀。 Postgres MVCC 每次更新都会创建一个新的 tuple 版本；autovacuum 在高频场景下跟不上。表中大部分成为死 tuple；查询扫描多出100倍的页；性能在数小时内崩溃。Kafka 完全规避了这个问题——broker 不跟踪每条消息的消费状态。Consumer offset 仅是每个（group, topic, partition）对应的一个整数，每秒更新一次。状态量为 O(groups × partitions)，而非 O(messages)。

故障3：无法重放。 消费后删除行→新 consumer 无法读取旧数据；consumer 逻辑中的 bug 无法通过重放来修复。永久保留行→表无限增长，索引无法放入内存。Kafka 存储多出1000倍的数据却无性能退化，因为读取是顺序的，旧数据存储在廉价存储上。

故障4：无法扇出。 五个 consumer = 五个 consumed_by_x 列。新 consumer = 修改 schema。在 Kafka 中，新 consumer group 只需一个 subscribe() 调用。

故障5：competing consumer 下的锁争用。 FOR UPDATE SKIP LOCKED （正是因为人们一直这样尝试才加入了 Postgres）提供了互斥锁，但每秒10万行级锁会造成竞争。Kafka 的"每个 group 中每个分区对应一个 consumer"消除了锁争用。

DB 作为队列的适用场景： 适度规模（< 1K msg/sec）、单个 consumer、无需重放、无需扇出。使用 FOR UPDATE SKIP LOCKED 的小型 SaaS 出站 webhook 队列加简单 worker 完全可行。 这正是 Kafka 存在的原因 ——超过约1万 msg/sec 或多 consumer 场景时，这种不匹配将是致命的。

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 相关 模式——outbox：事件与业务写入在同一事务中写入 DB outbox 表，然后 CDC 尾随 binlog 写入 Kafka——这是良好且标准的做法。Outbox 是事务性暂存，而非队列本身。面试时请加以区分。

§21. 扩展维度

两种结构上不同的增长模式，对应不同的解决方案；混淆两者是常见错误。

类型1（均匀增长）：更多 topic、更多 group、更多分区。 新产品特性→新事件类型→新 topic。演进路径：3个 broker 上 1万 msg/sec → 更多分区后同集群达到10万 msg/sec → 15-30个 broker 的专用集群加 schema registry 达到100万 msg/sec → 按领域拆分的多集群加 KRaft 加分层存储达到1000万 msg/sec → LinkedIn 的8000万 msg/sec 规模：数十个集群配备全舰队控制平面、自助接入、配额执行。

约 1000 万用户 约1K 个分区→ZooKeeper 吃力，迁移至 KRaft。约50个 broker/集群→再平衡耗时数小时，按领域拆分。10倍存储增长→分层存储必不可少。跨区域→MirrorMaker 2 或 Pulsar 地理复制；冲突解决成为系统需求。

类型2（热点集中化）：某个 topic 流量猛增10-100倍。 病毒式功能、失控 bug、黑色星期五。基线：32个分区上1万 msg/sec = 约300 msg/sec/分区，正常。10倍均匀增长 = 约3K/分区，正常。 10倍加偏斜（前1%的 key 占50%）： 某个分区承受5万 msg/sec，其他分区空闲。 增加分区无济于事 ——热点 key 仍哈希到同一分区。解决方案：加盐重新构建 key，牺牲严格的按 key 顺序。 100倍均匀增长 接近单分区上限；解决方案是增加分区，但对在线的带 key topic 重新分区需要创建新 topic + 双写 + 切换——代价高昂。 拐点： 在创建 topic 时过度配置分区数为预期需求的4-10倍；将分区数视为一次性决策。

增加节点会使写入变慢，而非变快。 类型1靠增加容量解决。类型2靠重新设计分区键解决。

§23. 使用场景展览——六个领域，不同变体

相同的原语，截然不同的需求。

支付中的事件溯源（Stripe outbox）。 支付服务在一个事务中写入 payments 行，同时在同一事务中写入 outbox 行。Debezium 尾随 binlog 并将 payment.charged 发布至 Kafka。下游：账本、通知、欺诈检测、报表。需求：严格原子性（无事件丢失）、账本对账的重放能力。适配：Kafka acks=all、事务性 consumer、 read_committed。不选队列，因为账本和报表都需要重放。

实时分析（Uber 点击流）。 移动客户端发出带有 session ID 的点击事件。Kafka 按 session_id分区。Flink 计算 session 指标；落库至 Druid（仪表盘）+ S3（长期存储）。需求：每秒100万-1000万事件，亚秒级新鲜度，容忍0.1%丢失，按 session 顺序。适配：Kafka acks=1、lz4 压缩——接受异步复制，因为分析对丢失容忍。

CDC 管道（LinkedIn Brooklin → Iceberg）。 Brooklin 尾随 MySQL/Espresso binlog，将变更事件写入 Kafka，由 Iceberg 湖表、搜索索引更新、Espresso 二级索引消费。需求：无损行变更传输、按行顺序、多下游扇出。适配：Kafka 日志 + 按 PK（Primary Key）分区 + acks=all + 压缩 topic 用于快照状态。

微服务异步（Airbnb 预订事件）。 booking.created 触发支付、日历、通知、搜索更新、房东通知。需求：约5万 msg/sec，每个 consumer 有自己的有界上下文。适配：Kafka 或 Pulsar。RabbitMQ 可行，但新增第6个 consumer 需要新建一个队列；使用日志则只需一个 subscribe() 调用。

IoT 遥测（Tesla 车队上传）。 数百万辆车辆发送 CAN-bus 事件、GPS、传感器数据、故障码。上传至区域 broker，转发至中央 Kafka 用于分析、预测性维护、OTA（Over-The-Air）追踪。需求：可承受突发、常规传感器读取容忍丢失但故障码绝不容忍、地理分布。适配：中央 Kafka + 区域集群配合跨区域复制，或边缘使用 MQTT（Message Queuing Telemetry Transport）桥接至 Kafka。安全关键场景使用 acks=all，批量场景使用 acks=1.

日志传输（Cloudflare 边缘日志）。 200+ 个 POP（Points of Presence）将请求日志、安全事件、边缘 worker 日志发送至中央 Kafka。下游：SIEM（Security Information and Event Management）、数据仓库、客户日志投递。需求：每天1T+条消息、保留30天以上、对成本敏感、约100 ms 延迟可接受。适配：Kafka + 分层存储至 S3/R2，积极使用 zstd；正在评估 WarpStream 风格的 S3 原生方案。

相同的原语——持久、分区、可重放的日志——服务于上述六个场景。旋钮（复制模式、压缩、分区键、保留策略、分层存储）有所不同；结构保持一致。

§24. 真实世界实现与数据

  • LinkedIn（Kafka 发源地，每天7T条消息）。 Kafka 于2010年在 LinkedIn 发明，用于替代 ActiveMQ。每天约7T条消息，峰值约8000万 msg/sec，约4000个 broker 分布在约100个集群中，约10万个 topic，约700万个分区，p99 约10 ms。Brooklin（CDC）为 Iceberg、Espresso 二级索引、衍生存储提供数据。
  • Netflix Keystone（每天10 PB）。 统一遥测平台。每天约7000亿事件，持续约800万事件/秒，多个区域集群，Flink 流处理。数据流向 S3、Druid、Elasticsearch、ML 特征管道。
  • Cloudflare（每天1T+条消息）。 边缘日志传输（2023年博客）。积极使用 zstd 压缩。推动其评估 S3 原生日志设计的重要驱动力。
  • Uber（每天4T条消息，从 Cherami 迁移至 Kafka）。 最初构建了 Cherami（自定义队列系统）；为获得重放能力和更高吞吐量迁移至 Kafka。uReplicator（开源）用于跨区域复制。uForwarder + Flink。
  • Stripe（约每秒500万事件）。 Outbox 模式：业务写入与 outbox 行配对；CDC 发布至 Kafka。多区域配合严格 exactly-once 写入账本，通过事务性 producer + read_committed.
  • Confluent Cloud 基准测试。 单个 c5n.4xlarge broker 在 acks=all、RF=3、lz4 配置下可持续 ~605 MB/sec 写入。端到端 p99 在中等负载下约5 ms。
  • WarpStream（2024年被 Confluent 收购）/ AutoMQ。 S3 原生 Kafka 兼容方案。在10-30节点集群上约1 GB/sec，p99 约200-500 ms，高保留场景下成本降低30-50%。
  • Pulsar（腾讯部署）。 声称单集群可达1000万 msg/sec。多租户隔离是其差异化优势。BookKeeper 按条目 fsync，副本数少于 Kafka。

不同访问模式下的稳态生产——活动事件（LinkedIn）、遥测火管（Netflix）、边缘日志（Cloudflare）、任务图（Uber）、事务性 outbox（Stripe）、高保留低成本（WarpStream/AutoMQ）、多租户 SaaS（Pulsar）。相同原语，调优各异。

§25. 总结

分布式日志是底层原语，它将"发送这条消息"转化为"将这段字节范围追加到一个多副本、分 segment 的文件中"——其余设计空间（队列 vs 日志、同步 vs 异步复制、本地磁盘 vs S3 原生、broker 管理 vs 点对点）都是围绕这一操作的不同优化程度的变体。当扇出和重放在规模上很重要时选择日志；当单消息语义在适度规模下很重要时选择队列；当保留成本占主导时选择 S3 原生；当规模尚不足以摊平引入 broker 成本时选择 DB outbox。

附录 B：值得提及的配套模式

  • 事务性 outbox： 在与业务写入相同的事务中将事件写入 DB outbox 表；CDC 读取 binlog 并发布至 Kafka。消除了双写原子性问题。被 Stripe、Shopify 采用。
  • API 边界的幂等键： 面向客户端的 API 接受幂等键 headers（Stripe 风格）。防御队列层之上的重试。
  • 压缩状态 topic： cleanup.policy=compact + 每个状态对象的 key = 物化的当前状态快照。用于 __consumer_offsets、当前行值的 CDC、特征存储在线缓存、流处理器状态备份。
  • MirrorMaker 2 / uReplicator： 用于 DR 或地理分布的跨集群复制。Offset 转换是难点——offset 在不同集群间不能1:1迁移。
  • 上层流处理： Kafka 是底层基础；Flink / Kafka Streams / Spark Structured Streaming 是计算层。Exactly-once 通常在此实现，而非在原始 consumer 中。
  • Schema Registry（Confluent、Apicurio、Pulsar 内置）： 每个 topic 都有注册的 schema（Avro / Protobuf / JSON Schema）。向后兼容性执行防止 producer 变更破坏 consumer。
