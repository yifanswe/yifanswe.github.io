组件类别：协调服务 / 元数据存储——etcd、Apache ZooKeeper、HashiCorp Consul、Google Chubby。这类服务是强一致、高可用、刻意低吞吐量的键值存储，用于保存分布式系统的控制面状态。

背景：本文探讨的是 技术类别。协调服务在存储分类中占据独特的生态位——它们介于事务型数据库与内存缓存之间，但两者都不是。它们的存在，是因为分布式系统需要对少量关键问题（"谁是主节点？"、"分片在哪里？"、"配置是什么？"、"谁持有这把锁？"）有唯一的事实来源，且该真相必须在故障时不产生歧义地保存下来。全文示例涵盖 Kubernetes、Kafka、Cassandra、CockroachDB、分布式 cron、功能开关和服务网格。

§1. 协调服务是什么

一个 协调服务 是一种强一致、高可用的分布式键值存储，针对 控制面 元数据的存储与分发进行了优化。其定义性特征如下：

  • 数据集极小，但至关重要。 从几 KB 到几 GB 不等。每个字节都举足轻重——一个错误的答案会污染整个数据面。
  • 吞吐量低，这是有意为之的设计。 单主节点共识意味着写入上限约为 10,000 次/秒。这是可以接受的，因为协调状态的 消费者 读取频率远高于写入频率，且元数据变更的速率受集群事件（部署、故障转移、配置变更）约束，而非用户流量。
  • 线性化读写。 一次成功的写入对所有副本上的所有后续读取可见。
  • Watch 原语。 客户端订阅键或前缀，并接收完全有序的变更流。
  • 租约 / 临时语义。 客户端持有的键与其存活状态挂钩——当客户端宕机时，该键随即消失。

它们解决哪类问题？

每个分布式系统都面临的五类问题：

  1. 主节点选举 ——在 N 个副本中，同一时刻恰好有一个持有租约。
  2. 服务发现 ——端点注册，消费者监听变更。
  3. 分布式锁 ——结合 fencing token 供下游资源验证。
  4. 配置分发 ——写入一次，通过 watch 机制传播。
  5. 集群成员管理 ——节点携带租约注册，故障时自动退出。

在技术栈中的位置

        ┌──────────────────────────┐
        │      Application         │  (user requests)
        └────────────┬─────────────┘
                     │
        ┌────────────▼─────────────┐
        │      Data plane          │  (DBs, caches, queues)
        │  millions of ops/sec     │  high throughput

        │  Coordination service    │  (etcd / ZK / Consul)

        │  ~GBs of metadata        │  strong consistency
        └──────────────────────────┘

与相邻类别的区别

  • 事务型数据库 （MySQL、PostgreSQL）：写吞吐量更高、数据量更大、查询能力更丰富——但没有原生的 watch、租约或协调原语。
  • 宽列 NoSQL （Cassandra、HBase）：默认最终一致，针对 TB 级用户数据进行分区，写入不具备共识保证。
  • 内存 KV （Redis、Memcached）：读取延迟低于毫秒，但没有真正的共识——Redis 的主副本复制是异步的，因此故障转移后的状态并不是故障转移前状态的一致前缀。作为缓存尚可接受，作为协调器则危险。
  • 对象存储 （S3、GCS）：容量极大，仅支持单对象的写后读一致性，不支持 watch 和租约。

不适合的场景

  • 高写入吞吐量 （>10k/秒）——每次提交都须通过 quorum fsync 门控。
  • 大数据量 （>8 GB）——快照传输和内存索引压力将成为瓶颈。
  • 低延迟全球写入 ——跨区域 quorum 最低需要 30-100 ms。
  • 丰富的查询语义 ——仅支持扁平 KV 或层级化 znode。
  • 用户数据 ——经典的反模式。

§2. 固有保证

设计上已提供

线性化。 一旦写入被确认，任意副本上的所有后续读取都将反映该写入。单一完全有序的历史记录。这是最强的单对象一致性模型。

分区下不发生脑裂。 在任意分区情况下，最多只有一侧可以提交写入。另一侧变为只读或拒绝服务。共识机制从构造上防止了两个同时存在的主节点——主节点需要多数 quorum，而一次分区不可能产生两个多数。

通过多数 quorum 实现持久性。 一次已提交的写入能够在任意少数副本故障和少数节点磁盘损坏的情况下存活。要求在多数节点上完成 fsync 后方可确认——持久性取决于 quorum 中最慢的 fsync。

Watch 有序分发。 订阅者按提交顺序接收所有匹配其谓词的已提交变更，无间隙（直至可配置的压缩时间界限）。

租约语义。 存活性由 协调服务的视角 （收到的租约心跳）决定，而非由 gossip 协议决定——这正是锁能安全地使用 fencing 的根本所在。

不保证的特性（需在上层叠加实现）

  • 高吞吐量。 每个 Raft 组的写入上限约为 10k 次/秒；如需更高吞吐量，需对工作负载进行分片。
  • 多区域可用性。 每个区域运行独立的协调服务，通过带外状态复制进行同步。
  • 客户端操作的幂等性。 如果客户端不确定其请求是否成功（提交与确认之间发生网络中断），必须重试。应用层负责使操作幂等——通常通过对版本号进行 CAS（compare-and-swap）来实现。
  • SQL 风格的多行事务。 多键条件批处理（etcd Txn、ZooKeeper multi）存在，但并非跨多键的完整 ACID 事务。
  • 快速故障转移。 在健康集群上，主节点选举耗时 150-600 ms；应用必须容忍这段短暂的不可用。

Raft（etcd、Consul、TiKV PD、CockroachDB）

现代默认选择。Ongaro 与 Ousterhout 于 2014 年提出的协议，以易于理解为设计目标。各部分均有明确规范：强主节点、随机化选举超时、日志完整性投票检查、两阶段联合共识成员变更。etcd、Consul、CockroachDB、TiKV 及 Apache Kudu 的生产实现共享相同的思维模型。

Zab（ZooKeeper Atomic Broadcast）

早于 Raft。结构上类似——主节点加从节点，单调递增的 epoch 编号（类比 Raft 的 term）。独特之处：FIFO 客户端顺序保证、显式恢复阶段（discovery → sync → broadcast）。在共识部分与 Raft 同构。差异仅在表面（epoch vs term、znode vs 扁平键）和运维层面（JVM 调优 vs 单一 Go 二进制）。

Paxos（Chubby、Spanner 内部）

Lamport 的原始协议。Multi-Paxos 在主节点稳定的常规场景下跳过 prepare 阶段。生产级 Paxos 确实存在，但 Paxos 本身不规定主节点选举、日志复制结构或成员变更方式——每个实现各自为政，产生了微妙的差异。这正是 Raft 被设计来解决的问题。

为什么 etcd 的"BBolt B+ tree + Raft"是现代默认方案

协调类工作负载具有四个共同特征：(1) 数据集小，(2) 读多写少，(3) 写入已受共识 fsync 约束，(4) 对前缀的范围扫描很常见。这些特征指向一种 B+ tree 状态机，而非 LSM（Log-Structured Merge）树。LSM 的优势——通过顺序写入吸收高写入吞吐量——在 Raft fsync 成为瓶颈时毫无意义。B+ tree 提供可预测的读延迟、干净的范围扫描、对 mmap 友好的 I/O，且无压缩停顿。

BBolt（Bolt 的 fork，本身受 LMDB 启发）实现了一种写时复制 B+ tree。结合用于日志复制的 Raft 和用于基于版本号 watch 的 MVCC（Multi-Version Concurrency Control），这是当今标准的现代技术栈。自 2015 年以来每个新的协调服务——etcd、CockroachDB 的存储、TiKV 的 range 存储——都在向这种设计的某个变体收敛。

4.1 Raft 共识日志

Raft 为每个投票节点提供相同的 完全有序的日志条目。一旦日志条目被提交（获得多数 quorum 确认），它将以相同顺序在每个节点上应用到状态机。状态机是附属品——Raft 提供日志，状态机消费它。

每条 Raft 条目：

struct LogEntry {
    uint64 term;          // election term when created
    uint64 index;         // monotonic, 1, 2, 3, ...
    bytes  command;       // serialized state-machine command
}

每个投票节点维护三个单调递增的计数器：

  • commitIndex ——已知被提交的最高索引（经多数确认）。
  • appliedIndex ——已应用到状态机的最高索引。
  • currentTerm ——该节点认为自己所处的选举 term。

不变式： appliedIndex <= commitIndex <= lastLogIndex.

主节点选举

从节点启动一个 随机化选举超时 在 150 到 300 ms 之间（随机化至关重要——没有它，所有从节点会同时超时并永远分裂投票）。当计时器触发时：

  1. 转变为候选人；递增 currentTerm；为自己投票。
  2. 发送 RequestVote(term, candidateId, lastLogIndex, lastLogTerm) 给其他节点。

每个节点投票 yes 仅当 (a) 本 term 内尚未投票，且 (b) 候选人的日志至少与自身一样新时才投票。"更新"意味着：候选人的 lastLogTerm > yours，或者 lastLogTerm == yours AND lastLogIndex >= yours。这就是 日志完整性检查 ——防止日志落后的节点成为主节点并覆盖已提交的条目。

候选人以 floor(N/2) + 1 票（含自身）胜选。转变为主节点，立即发送心跳（空的 AppendEntries）来抑制其他节点的选举计时器。票数分裂时：两个候选人均以更高 term 和新的随机超时重试；通常有一方更早触发并胜出。选举在 1-2 轮内完成，约 300-600 ms。

日志复制

针对每个从节点：

nextIndex[f]  = leader's lastLogIndex + 1 (initial)
matchIndex[f] = 0 (initial)

loop:
  prevLogIndex = nextIndex[f] - 1
  prevLogTerm  = log[prevLogIndex].term
  entries      = log[nextIndex[f] : nextIndex[f] + batch]

  send AppendEntries(term, leaderId, prevLogIndex, prevLogTerm,
                     entries, leaderCommit)

  if response.success:
      matchIndex[f] = nextIndex[f] + len(entries) - 1
      nextIndex[f]  = matchIndex[f] + 1
  else:
      nextIndex[f] -= 1   // mismatch; step back

从节点在收到 AppendEntries:

  • 如果 term < currentTerm：拒绝。
  • 如果 term > currentTerm：更新，退位为从节点。
  • 一致性检查：如果 log[prevLogIndex].term != prevLogTerm，拒绝。主节点回退并重试直至匹配；从节点随后截断并追加。
  • 将条目追加到本地日志； fsync 再响应。
  • 更新本地 commitIndex = min(leaderCommit, lastLogIndex).

提交规则 （主节点侧）： commitIndex 推进到最高索引 N 满足以下条件： N 创建于 当前 term 且多数 matchIndex[] 均为 >= N。"当前 term"限制是著名的 Raft Figure 8 修复——提交来自之前 term 的条目需要在其上提交一个当前 term 的条目，否则被废黜的主节点可以复活并覆盖数据。

Term 与"更高 term 胜出"不变式

Term 是类 Lamport 的逻辑时钟。每个 RPC 都携带发送方的 term。任何节点收到更高 term 时：更新 currentTerm，成为从节点，重置投票。这一单一不变式——更高 term 胜出，更低 term 关闭——防止两个主节点同时存在。即使被隔离的旧主节点，也会在收到更高 term 的第一条消息时退位，而且由于为新 term 投票需要先看到该 term，旧主节点无法提交任何新内容。

4.2 BBolt——写时复制 B+ tree

日志条目被提交后，etcd 将其应用到 BBolt：一个使用写时复制 B+ tree 的单文件嵌入式 KV 存储。

页布局

BBolt file (e.g., default.etcd/member/snap/db)
─────────────────────────────────────────────────
 Page 0:  meta page A (root ptr, txn id, freelist, checksum)
 Page 1:  meta page B (alternate copy — meta pages flip)
 Page 2:  freelist
 Page 3+: branch pages and leaf pages

         ┌────────────────────────────────┐
         │  Branch page (interior node)   │
         │  ┌──┬──┬──┬──┬──┬──┬──┐        │
         │  │k1│p1│k2│p2│k3│p3│..│  k+ptr │
         │  └──┴──┴──┴──┴──┴──┴──┘        │
         └─────┬───────┬──────────────────┘
               │       │
               ▼       ▼
         ┌──────────┐  ┌──────────┐
         │ Leaf     │  │ Leaf     │
         │ k:v k:v  │  │ k:v k:v  │
         └──────────┘  └──────────┘

4 KB 页。叶节点存储约 30-40 个 KV 对。分支因子 b ≈ 30-50。在 8 GB / 4 KB 页 = 2M 页的情况下，树深 4-5 层。若工作集在操作系统页缓存中（整个 DB 通常能放入 RAM），一次查找本质上就是一次内存访问。BBolt 通过 mmap 映射文件；读取由缺页中断驱动。

写时复制

当写事务修改某页时，BBolt 从空闲列表分配一个 新的 页并将新版本写入其中。父节点的指针更新为指向新子节点——但父节点本身也必须进行写时复制，如此一直传播到根节点。提交时：

  1. 写入新数据页并执行 fsync。
  2. 写入新根节点。
  3. 更新 meta 页以指向新根节点并执行 fsync。
  4. 旧页进入空闲列表（在没有活跃读者引用它们之后）。

双重 fsync——一次用于数据页，一次用于 meta 页——是持久性保证点。第二次 fsync 之后，崩溃恢复到新状态。两次 fsync 之间崩溃：下次启动时读取旧 meta 页（仍指向旧根节点）；部分写入的新页为垃圾数据。

为什么用 B+ tree 而非 LSM

对于 LSM（RocksDB、LevelDB），写路径是顺序追加到 memtable + commit log，然后异步压缩。这可以吸收 50-100k 次写/秒。但：

  • Raft fsync 已将写入上限限制在约 10k 次/秒——LSM 的写入能力被浪费。
  • LSM 的读延迟不可预测（需探测多个有序 run + bloom filter）。
  • 压缩停顿是一个运维问题。
  • 范围扫描需要跨越多个有序 run，并需要合并迭代器。

B+ tree 提供 O(log_b N) 的点查询（成本可预测），沿叶节点的范围扫描无需合并，且无后台压缩。权衡：最大写入吞吐量较低。可以接受，因为 Raft 已将其限制住了。

4.3 MVCC 层

etcd 暴露一个 Multi-Version Concurrency Control 键空间：每个 key 都有修订历史。每次 put/delete 都被分配一个全局单调递增的 版本号，即该操作对应的 Raft 日志索引：

revision 100: PUT /a = "x"        (mod=100, ver=1, create=100)
revision 101: PUT /b = "y"        (mod=101, ver=1, create=101)
revision 102: PUT /a = "x'"       (mod=102, ver=2, create=100)
revision 103: DELETE /a           (mod=103, tombstone)
revision 104: PUT /a = "x''"      (mod=104, ver=1, create=104)

BBolt 存储：

  • Key bucket: (/a, 100) -> "x", (/a, 102) -> "x'", (/a, 103) -> tombstone, (/a, 104) -> "x''".
  • Index bucket （内存中的 btree）：逻辑 key /a → 带元数据的版本号列表。

在版本号 r 处对 /a 进行读取，找到最高版本号 <= r。"从版本号 100 开始 watch"会重放版本号 > 100 的所有变更。 压缩 (etcdctl compact）丢弃低于某一阈值的版本号；每 5 分钟自动压缩是标准做法。

4.4 Watch 实现

client → etcd: WatchCreate(key="/pods/", prefix=true, start_rev=12345)

服务端：(1) 在内存中的 watcher 表里注册 watch；(2) 如果 start_rev 已过期（压缩之后），则从 MVCC 回放——即追赶阶段；(3) 追赶完毕后，watcher 进入 synced 状态，并按顺序接收后续每次匹配的提交。

扇出：10k 个 watcher × 10k 次写/秒全部触及被 watch 的前缀 = 每秒 1 亿事件——远超 etcd 的处理能力。这就是为什么系统在 etcd 前面放聚合器（即 Kubernetes apiserver）。 契约：事件按序投递；客户端携带 start_rev = last_seen + 1重连。若断连时间过长导致压缩丢弃了起始点，etcd 返回 ErrCompacted ，客户端执行全量重新同步。

4.5 一次写入的端到端流程

客户端发起 Put("/locks/job-runner", "node-7", lease=42).

第一步：客户端连接三个节点之一。若连接到从节点，从节点返回主节点地址。调用主节点 一个.

Step 2：A 将请求序列化为 Raft proposal，并作为 LogEntry { term=15, index=98234, command=<bytes> }.

Step 3 — 主节点 WAL fsync：A 将新条目追加到其 wal/ 段文件，然后 fsync。 这是主节点的持久性保证点。 延迟：NVMe 上约 1 ms。

Step 4 — 主节点发送 AppendEntries （与 step 3 并行）：

AppendEntries {
  term: 15, prevLogIndex: 98233, prevLogTerm: 15,
  entries: [new entry at 98234], leaderCommit: 98233
}

Step 5 — 从节点 WAL fsync 并响应：B 和 C 验证 prevLogIndex/prevLogTerm，追加， fsync，响应 success=true 与 matchIndex=98234.

在三节点集群中，A 需要 1 个从节点确认（A + 1 个从节点 = 3 中的多数）。一旦 B 或 C 中的任意一方 响应，该条目即被提交。

Step 6：A 看到 matchIndex[B]=98234，将 commitIndex 推进到 98234。

Step 7 — 应用到 BBolt：A 的 apply 循环打开一个写事务，插入 MVCC 条目（key_bucket[("/locks/job-runner", 98234)] = "node-7"），更新内存中的索引，将 key 与 lease 42 关联。提交 BBolt 事务 → BBolt 自身的 fsync 序列（数据页 → meta 页）。

Step 8：A 向客户端响应版本号 98234。

Step 9：从节点在下一次心跳中看到 leaderCommit=98234 后最终应用。

各步骤的崩溃恢复:

  • Step 3 fsync 之前：无操作。客户端超时。没有节点持有该条目。
  • Step 3 之后、Step 4 之前：A 持有该条目，B 和 C 没有。新选举可能传播或丢弃，取决于谁的日志更靠前。
  • Step 5 之后：条目在 3 个节点中的 2 个上持久化 → 已提交。下一任主节点持有它。
  • Step 7 和 Step 8 之间：客户端不知道结果。必须重试；基于版本号的 CAS 处理幂等性。

§5. 分布式锁 recipes——实际 API 模式

分布式锁是协调服务最常见也最容易出错的用途。本节介绍 etcd、ZooKeeper 和 Consul 中的标准锁 recipe，最后以 Kleppmann 对 Redlock 的批判作结，以说明为什么这些 recipe 呈现为现在的样子。

5.1 正确分布式锁的通用结构

任何正确的分布式锁都包含五个要素：

  1. 带超时的获取。 最多阻塞 T 秒；不要让一个慢锁永远卡住调用方。
  2. 租约（TTL）。 如果持有者宕机，锁自动释放—— 协调服务 决定活性，而非客户端。
  3. 返回给持有者的 fencing token ——这是承载安全属性的关键要素（参见 §5.5 的 Kleppmann 论述）。
  4. 监听前驱节点，而非整个前缀。等待者订阅释放锁的那个变更，形成链式结构而非惊群效应。
  5. 幂等释放。 重复释放或在租约到期后释放必须是安全的。

5.2 etcd recipe——lease + 带版本号的 key，fencing token 取自版本号

etcd v3 的锁基于三个原语构建： Lease, Txn （基于版本号的 compare-and-swap），以及 Watch。官方 concurrency.NewMutex package 实现了如下模式：

// (sketch of what etcd Mutex does)
leaseID := client.Grant(ctx, ttlSeconds).ID
go keepAlive(client, leaseID)   // refresh lease every TTL/3

// Each contender writes its own key under a shared prefix.
candidate := fmt.Sprintf("%s/%x", prefix, leaseID)
txn := client.Txn(ctx).
    If(etcd.Compare(etcd.CreateRevision(candidate), "=", 0)).
    Then(etcd.OpPut(candidate, "", etcd.WithLease(leaseID))).
    Commit()

myRev := txn.Header.Revision   // <— FENCING TOKEN

// List all candidate keys sorted by CreateRevision; lowest holds the lock.
// If we're not lowest, Watch on the predecessor (the next key below us).
// When predecessor is deleted, re-check ordering.

为何有效：

  • 客户端宕机时 Lease 自动释放。 如果 keepalive 停止——崩溃、分区、TCP 故障——租约在服务端过期，所有写入 WithLease(leaseID) 的 key 被删除。活性由 协调服务的 视角决定，而非客户端自身的判断，这正是锁在各种暂停场景下仍然安全的原因。
  • 来自以下位置的 fencing token CreateRevision. Revision 是 Raft 日志索引，即使跨主节点故障转移也永远不会回退。竞争者的 CreateRevision 是一个针对本次获取的单调计数器；将其传递给每个下游资源，下游资源会拒绝携带过期 token 的操作。
  • 监听前驱节点，而非前缀。 监听前缀意味着每次释放都会唤醒所有等待者——一次释放触发 1,000 次唤醒。监听前驱节点只唤醒下一个排队者。
  • 幂等释放 通过 Revoke(leaseID)；撤销已过期的租约是幂等操作（no-op）。

5.3 ZooKeeper 方案——使用临时顺序节点实现队列锁

# Create an ephemeral sequential znode under the lock path.
# Ephemeral: deleted when the session ends.
# Sequential: ZK appends a monotonically-increasing 10-digit counter.
my_path = zk.create(f"{lock_path}/lock-", ephemeral=True, sequence=True)
my_seq = int(my_path.split("-")[-1])   # <— FENCING TOKEN

while True:
    children = sorted(zk.get_children(lock_path))
    if my_path is lowest:
        return my_seq, lambda: zk.delete(my_path)
    # Watch predecessor (one slot below); when deleted, re-check.
    zk.exists(predecessor_path, watch=on_event)
    event.wait()

特性：

  • 临时节点通过 session 提供租约语义。 ZK session 是一个 TCP 连接加一个 session ID；若客户端在 session 超时（6-30 秒）内未发送心跳，服务器会使该 session 过期并删除其拥有的所有临时节点。这是 ZK 中与 etcd 租约等价的概念。
  • 顺序编号即 fencing token。 10 位计数器由服务器生成，单调递增，与父 znode 的 cversion。由于该值是共识提交状态的一部分，即使集群发生故障转移也不会回退。
  • 单次触发的 watch。 ZK watch 只触发 一次；必须重新注册。在触发与重新注册之间很容易错过事件——etcd 基于 revision 的 watch 可以从指定 revision 开始重放，更易于保证正确性。

5.4 Consul 方案——基于 session 的锁

Consul 在 sessions （TTL 驱动的存活检测）以及 带获取语义的 KV:

session_id = consul.session.create(ttl="30s", behavior="release", lock_delay="15s")
threading.Thread(target=renew_loop, args=(consul, session_id)).start()

# kv.put with acquire= succeeds iff no other session holds it.
acquired = consul.kv.put(key, "owner", acquire=session_id)
# If not, long-poll on the key for change to Session field, then retry.
token = consul.kv.get(key)["ModifyIndex"]   # <— FENCING TOKEN

独特之处：

  • lock_delay 是一种安全隔离机制：当持有锁的 session 异常消亡时，该 key 会进入"禁止获取"状态并持续 15 秒，之后其他 session 才能获取锁。防止"session 已过期但持有者尚不知情"的竞态条件。
  • ModifyIndex 作为 fencing token。 与 etcd revision 和 ZK cversion 类似——全局单调递增。
  • Session 可与健康检查绑定，当某个检查失败时 session 消亡——比纯 TTL 存活检测更精细。

5.5 对 Redlock 的 Kleppmann 批评

Martin Kleppmann 2016 年的文章 《如何正确使用分布式锁》 是将分布式锁视为简单问题的经典批评。文中指出三处致命缺陷：

缺陷一——进程暂停会破坏基于挂钟时间的租约。 Java 客户端在 t=0 时刻获取一个 10 秒的锁。t=2 时发生长达 15 秒的 GC 停顿。锁服务在 t=10 过期；另一个客户端在 t=11 获取锁；原客户端在 t=17 醒来时仍然以为自己持有锁。 两个客户端同时进入临界区。 如果唯一的安全保证是 TTL，任何锁服务都无法阻止这种情况。解决方案——fencing token——正是上述所有方案的核心设计思路。

缺陷二——Redlock 依赖时钟漂移假设。 Antirez 的 Redlock 依赖 N 个独立 Redis 节点之间有界的时钟漂移。NTP 步进调整、闰秒、VM 迁移后虚拟机管理程序的挂钟跳变、VM 暂停——这些都会违反该界限。一旦违反，Redlock 会在两个节点都未察觉的情况下同时授予两个租约。

缺陷三——Redlock 的 fencing token 无法在故障后保持单调性。 每个 Redis 节点有各自的计数器，故障转移会导致计数器丢失。真正的 fencing token 在所有故障下始终保持单调性——单个 Raft 组可以免费提供这一特性（revision = 日志索引），而多个独立 Redis 节点的联合在没有共识机制的情况下无法做到。

一句话总结： "Redlock 是一个快速锁；在现实故障条件下，它不是一个正确的锁。正确性的代价恰好是一个 Raft 组。"

5.6 正确的分布式锁获取——端到端完整流程

调用 etcd 支撑的锁的调用方应期望满足以下契约：

def with_lock(resource_key, work_fn, max_wait=10, lease_ttl=30):
    deadline = time.now() + max_wait
    while time.now() < deadline:
        try:
            token, release = acquire_lock(etcd, resource_key, ttl=lease_ttl)
            break
        except LockBusy:
            sleep(jittered_backoff())
    else:
        raise LockTimeout()
    try:
        work_fn(fencing_token=token)   # downstream: token >= max_token_seen
    finally:
        release()   # idempotent

§5.1 中的所有安全性质：超时、租约驱动的存活性、单调 fencing token、前驱节点监听、幂等释放。90% 的"我们自己实现分布式锁"故障的根因，都是这五项中缺少了某一项。

§6. 服务发现模式

服务发现可分为三种架构风格，各自具有截然不同的扩展特性。"使用 etcd 做服务发现"这一说法并不完整，还需要说明是 哪种 风格。

6.1 模式 A——客户端侧发现

服务的每个客户端直接查询（或监听）协调服务。Netflix 最初的 Eureka 模型、早期 Kafka 消费者从 ZooKeeper 读取分区归属权，以及 Go 生态中的"etcd 感知"客户端，都是典型例子。

协调服务负载：每个客户端至少开启一个 watch 流；流的数量 等于 客户端集群规模。若有 10 万个微服务实例，则意味着 10 万个 watch 流——远超 etcd 约 1 万流的上限一个数量级。

适用场景：小规模集群（< 1,000 个客户端）、不允许额外跳数的延迟敏感路径，或通用负载均衡器无法编码的自定义客户端侧路由（一致性哈希、粘性分片归属）。

规模化时的瓶颈：随着集群增长，协调服务成为瓶颈——可在其前端加聚合器（§6.2），或迁移至 gossip 协议（§6.3）。

6.2 模式 B——服务端侧发现

协调服务持有注册表；负载均衡器（LB）或服务网格代理代替客户端读取注册表；客户端只与 LB 通信。AWS Application Load Balancer 目标组、由 EndpointSlices 支撑的 Kubernetes Service（apiserver 从 etcd 读取）、消费控制平面 EDS（Endpoint Discovery Service）的 Envoy sidecar——均属此类变体。

   etcd → watched by → apiserver / Envoy control plane → DNS / Virtual IP (VIP) / xDS → 100k clients

协调服务负载：O(LB 集群规模) 个 watch，而非 O(客户端集群规模)——几百个 apiserver 意味着几百个 watch。聚合器吸收了扇出。

适用场景：大规模集群（10k+ 客户端）——这是规模化场景下的默认选择；多语言环境下在每种语言中嵌入协调感知的客户端库不切实际；集中式策略（限流、mTLS 终止、重试、熔断）本来就应在 LB 层处理。

6.3 模式 C——基于 gossip

每个节点周期性地将自己的视图传播给少量对等节点；经过 O(log N) 轮后集群达成收敛。Consul 的 serf 层、Cassandra 的 gossip、SWIM（Scalable Weakly-consistent Infection-style process group Membership，即 Consul 实现的协议）均属此类。 无中央注册表 ——每个节点从 gossip 消息中本地重建注册表。

协调服务负载：零——不存在单点的存活性协调服务。Consul 拆分了职责：Raft 支撑的 KV/catalog 用于强一致性状态，SWIM gossip 用于存活性检测，因此存活性探测不走 Raft。

适用场景：大规模存活性检测（SWIM 可扩展至数万节点——流量线性增长而非平方增长）；最终一致性可接受；去中心化环境（地理分布式、边缘集群）中没有合适的位置部署单一协调集群。

不适用于：任何需要线性一致性的场景（主节点选举、锁 fencing、schema 变更）——gossip 是最终一致的，两个节点可能短暂地同时认为自己是主节点。Cassandra 和 Consul 都采用混合方案——gossip 用于高频、最终一致即可的问题；嵌入式 Paxos / Raft 用于少数需要强一致性的操作。

6.4 如何选择

集群规模 新鲜度需求 模式
< 1,000 强一致性 客户端侧直连
1,000-100,000 强一致性 通过聚合器的服务端侧（默认）
> 100,000 混合 聚合器 + gossip 存活性的混合方案
任意规模 最终一致性即可 Gossip

核心观点： "客户端侧将每个客户端加入协调服务的连接表；服务端侧仅将 LB 集群加入连接表；gossip 不占用连接表中的任何位置。根据你能承受的连接表规模来做选择。"

§7. Watch 机制与扩展性

§4.4 在 API 层面介绍了 watch。本节深入探讨 watch 在 etcd v3 中的实现方式，并分析驱动每种生产架构设计的惊群问题。

7.1 etcd v3 watch 流——gRPC 多路复用

etcd v3 将 v2 的 HTTP 长轮询替换为基于 HTTP/2 的 gRPC 双向流 over HTTP/2. 一个 TCP 连接可承载多个 watch 因为 gRPC 将它们作为独立的 HTTP/2 流在单个 TCP socket 中多路复用。客户端可以在一个 TCP socket 上拥有 1,000 个逻辑 watch。服务器仍将每个 watch 作为独立订阅者在内存中追踪——TCP 是共享的，但簿记不共享。

准确的表述是： 每个 watch 对应一个 HTTP/2 流，在每个客户端连接的单个 TCP 上多路复用。 在文件描述符限制或内核 epoll 开销成为主导之前，这种方式是可行的。

7.2 Watch 分片

单个事件可能很大（默认最大 1.5 MB）；一个 WatchResponse 受 gRPC 消息大小（默认 4 MB）限制。当被监听的前缀发生突发事件——同一 Raft commit 中更新了 1000 个 Pod——etcd 会将响应分片到多个 WatchResponse 消息，并标记为 Fragment=true ，直到最后一条。

HTTP/2 流级别的流量控制生效：慢速客户端会积压其流，etcd 的每 watcher 缓冲区填满，超过阈值后 etcd 会 断开慢速 watcher 与 WATCH CANCEL。客户端必须携带新的 start_rev。这以慢速客户端获得局部视图为代价来保护集群。

7.3 Watch revision 过滤——bookmark 机制

每个 watch 携带一个 start_rev （默认：当前 + 1，即"仅未来事件"）。服务器只发送 revision >= start_rev 与 key/前缀匹配的事件。

如果 start_rev 落后于压缩水位线（例如， start_rev=10000 但 50000 之前的内容已被压缩），服务器返回 ErrCompacted ，客户端必须进行完整重同步。这种 bookmark 契约 使 etcd 的 watch 模型比 ZooKeeper 的单次触发 watch 更易于推理：只要在 之前 压缩之前重新连接，就能获取中间的所有事件。ZK 只触发一次，必须重新注册；触发与重新注册之间发生的删除操作会被遗漏。

7.4 惊群问题——同一 key 上有 10k 个 watcher

场景：10,000 个 kubelet 各自监听 /leader-election/scheduler。主节点发生变更。 所有 10,000 个 watcher 在同一 Raft commit 中被唤醒：

  • 1 次 Raft commit（锁释放 + 重新获取）。
  • 10,000 次 watch 投递——每条 1 KB，单次 commit 边界产生 10 MB 出流量。
  • 10,000 个 watcher 响应并触发次级操作——额外 10,000 次 Get加上额外 5,000 次 Put。

一个逻辑事件产生 etcd 无法承受的闪现拥塞。p99 延迟飙升，watcher 超时，重新连接重新触发 list，风暴在数秒内自我放大。缓解措施：不要在 10,000 个地方监听同一个 key（使用聚合）；在聚合器层合并事件；次级响应时使用带抖动的退避策略。

7.5 为什么 Kubernetes apiserver 引入 reflector + indexer + informer

apiserver 在 etcd watch 与 Pod 控制器之间实现了三层缓存：

etcd  ── ONE watch per resource type ──►  apiserver
                                          ├── Reflector (reads etcd watch, normalizes)
                                          ├── Indexer   (in-memory cache by name, labels)
                                          └── Informer  (exposes List + Watch to controllers,
                                                         served from cache not etcd)
                                          │
                                          ▼  thousands of controller watches
                                       controllers, kubelets (10k+)

架构规则： etcd 看到 O(resource_types) 个 watch——约 50 个。apiserver 看到 O(clients) 个——10,000+。 etcd 完全免受扇出影响；apiserver 在负载均衡器后水平扩展。

这一经验可推广： 只要 watcher 数量超过几百个，就需要引入聚合器。 Consul 通过每主机 agent 实现；apiserver 通过 reflector/informer 实现；服务网格控制平面通过 xDS 聚合实现。 协调服务不是扇出引擎。

§8. etcd MVCC 压缩——为何必须运行压缩，否则集群将崩溃

§4.3 介绍了 MVCC。本节阐述其运维风险：MVCC 历史无限增长，BBolt 持续膨胀，遗忘压缩是导致 etcd 集群崩溃最常见的原因。

8.1 每次写入都递增 revision；历史不断积累

每次 Put, Delete，或 Txn 被分配一个单调递增的 revision（= Raft 日志索引）。key/value bucket 存储 每个 key 的每个版本:

("/config/feature-X", 1001) → "off"
("/config/feature-X", 1547) → "on"
("/config/feature-X", 2104) → "on"
("/pods/payment-7",   1451) → <pod manifest>

每秒 1,000 次写入，意味着每天 8,600 万次 revision。若每个对象约 1 KB，且每个 revision 都保留，则每天约 85 GB——在不到三小时内就会触达 8 GB 上限。

8.2 压缩丢弃历史；defrag 回收空间

etcdctl compact <revision> 丢弃水位线以下的 MVCC 条目，仅保留每个仍存活 key 的最新版本。两个后果：

  1. 携带 start_rev 低于水位线的 watch 会失败，返回 ErrCompacted. 客户端必须进行完整重同步。新鲜度与存储的权衡。
  2. BBolt 空间仅在 etcdctl defrag. 压缩将 B+ tree 页面标记为空闲，但不会缩减文件大小。 压缩和 defrag 是两个独立的操作；只运行压缩而不运行 defrag，意味着即使内存中的 MVCC 索引已被裁剪，磁盘空间也不会被回收。

8.3 忘记压缩 = 集群崩溃

部署 etcd 时未启用自动压缩。写入持续流入；BBolt 持续增长。一周后文件达到 8 GB，etcd 拒绝写入：

ERROR: etcdserver: mvcc: database space exceeded

集群进入 alarm:NOSPACE。恢复步骤：积极压缩，逐节点 defrag 每次一个 （defrag 会阻塞节点；并行操作会导致失去 quorum），解除告警。这是最常见的 etcd 故障之一；通过启用自动压缩可完全避免。

8.4 Kubernetes 默认配置——每 5 分钟压缩一次

Kubernetes 内置了 --auto-compaction-retention=5m。etcd 每 5 分钟压缩超过 5 分钟前的历史——BBolt 大小与存活状态成比例，而非与写入速率成比例。5 分钟窗口是断连客户端在失去重放能力前允许落后的最大时间；对于在本地网络运行的 Kubernetes 控制器而言，断连 5 分钟的情况极为罕见。

Auto-defrag 默认 非 关闭——需手动运行或通过 operator 运行。许多故障的教训：同时启用自动压缩和 defrag 计划。

§9. 快照与备份

协调服务是控制平面的唯一真相来源；在没有备份的情况下丢失它，意味着丢失每个 Kubernetes 对象、每个 Kafka topic 配置、每条服务注册信息。快照/恢复演练是不可或缺的。

9.1 etcdctl snapshot save ——时间点 BBolt 副本

etcdctl snapshot save /backup/etcd-snapshot-$(date +%s).db

机制：etcd 在 BBolt 上开启一个读事务（MVCC 快照），拷贝 BBolt 文件，关闭事务。由于 BBolt 是写时复制的，即使有并发写入，快照也是一致的时间点视图——写入进入新页面，快照读取旧根节点。

输出： v3 备份格式 ——一个 BBolt 文件加上 etcd 元数据尾部（集群 ID、最新 revision、SHA256）。大小等于当前 BBolt 文件大小（100 MB 到 8 GB）。标准频率：每 15-30 分钟一次；这决定了 恢复点目标（RPO）.

9.2 etcdctl snapshot restore ——用于引导新集群

恢复操作 非 应用于现有集群；它从快照引导一个 新的 集群。半损坏的集群无法安全地摄入快照，因此步骤是"销毁集群，以快照作为初始状态启动新集群"：

etcdctl snapshot restore /backup/snapshot.db \
        --name=etcd-0 \
        --initial-cluster=etcd-0=...,etcd-1=...,etcd-2=... \
        --data-dir=/var/lib/etcd-new
# Then start each etcd with --initial-cluster-state=new.

恢复后的集群拥有一个 新集群 ID；出于安全考虑绑定了旧 ID 的客户端需要重新配置。

9.3 跨区域快照复制

快照是文件对象——像任何 blob 一样复制。标准 Kubernetes 控制平面模式：

  1. 每 15 分钟本地快照，保留 24 小时。
  2. 推送至 S3 并启用跨区域复制，保留 30 天。
  3. 每周推送至独立的云服务商或异地位置。

9.4 恢复时间目标（RTO）与恢复点目标（RPO）指标

对于支撑 Kubernetes 的 etcd：

  • RPO：15 分钟（快照间隔——最大可容忍数据丢失）。
  • RTO：15 分钟（在灾难发生后 15 分钟内完成恢复）。

可实现，因为快照文件较小（< 8 GB），恢复是本地 IO + 引导（2-5 分钟），且 apiserver/控制器/kubelet 能快速重新收敛。代价： 整个集群在恢复窗口期间处于离线状态 ——对数据库而言看起来宽裕的 RTO 预算，对协调服务而言却很紧张，因为所有依赖组件在协调服务恢复之前都不可用。

规范做法：每季度进行一次恢复演练。第一次在生产环境恢复时，不要在紧急情况下才发现 之前 快照已损坏、S3 的 IAM 权限有误，或 etcdctl 版本不匹配。

§10. 认证与授权

协调服务持有最敏感的元数据——服务凭证、schema 映射、主节点租约。即使在私有数据中心，这里的 AuthN/AuthZ 也不是可选的。

10.1 etcd v3 RBAC（基于角色的访问控制）

三层模型： 用户, 角色, 权限，以及可选的 双向 TLS（mTLS）身份绑定:

etcdctl role add pod-reader
etcdctl role grant-permission pod-reader read /pods/ --prefix
etcdctl user add kubelet
etcdctl user grant-role kubelet pod-reader
  • 用户 = 身份标识 （密码或 TLS 客户端证书认证）。
  • 角色 = 权限集合，每个 (read|write|readwrite, key|range|prefix).
  • 前缀级别的权限 ——而非任意模式匹配。

TLS 客户端证书认证 是生产环境的标准模式：每个组件（apiserver、kubelet 等）持有一张证书；Common Name（CN）映射到 etcd 用户。磁盘上不保存密码。

生产环境 etcd 使用两套独立的 TLS 证书： 生产环境 etcd 中：

  • 客户端-服务端 TLS (--client-cert-auth, --cert-file）：API 流量。
  • 节点间 TLS (--peer-client-cert-auth, --peer-cert-file）：Raft 流量。

建议使用不同的证书颁发机构（CA），以在某个 CA 被攻破时限制爆炸半径。

10.2 ZooKeeper ACL（访问控制列表）

ZK 按 每个 znode分配 ACL。每条记录的格式为 <scheme>:<id>:<perms>:

  • Scheme: world, auth, digest, ip, x509, sasl.
  • ID：依赖于 scheme（对于 digest: username:base64(SHA1(username:password))；对于 x509：证书 DN；对于 ip：CIDR）。
  • 权限：以下权限的子集 READ, WRITE, CREATE, DELETE, ADMIN.

注意事项： 无继承机制 ——每个 znode 有其自己的 ACL 列表（大规模使用时较为繁琐，通常以编程方式设置）。 ADMIN 权限可与 WRITE 分离

——修改 znode 的 ACL 时需要此权限。

§11. 多数据中心协调

将协调服务跨数据中心（DC）拉伸部署是最常见的设计错误之一。

11.1 5 节点 etcd 跨 3 个数据中心，按 2-2-1 分布 2-2-1 标准拓扑：5 个投票节点，

  • 分布在三个数据中心——DC-A 两个，DC-B 两个，DC-C 一个。法定人数为 5 中取 3。故障分析：
  • DC-A 完全宕机（A1+A2）：剩余 3 个节点（B1、B2、C1）——满足法定人数。
  • DC-B 完全宕机（B1+B2）：剩余 3 个节点（A1、A2、C1）——满足法定人数。
  • DC-C 宕机（C1）：剩余 4 个节点；法定人数轻松满足。
  • DC-A 和 DC-C 同时宕机：仅剩 B1+B2——5 中取 2，不满足法定人数；集群进入只读状态。

DC-B 和 DC-C 同时宕机：仅剩 A1+A2——5 中取 2，不满足法定人数。 可承受任意单个数据中心的故障；不能承受两个数据中心同时故障。

若主节点在故障数据中心，选举将从剩余 3 个节点中选出新主节点；150-300 ms 的选举超时意味着故障检测后不可用时间低于 1 秒。

11.2 延迟代价——跨数据中心往返时延（RTT）增加每次提交的耗时

  • 每次 Raft 提交都需要多数节点完成 fsync。在 2-2-1 拓扑和 20 ms 跨数据中心 RTT 的条件下：
  • DC-A 的主节点：本地 fsync 约 1 ms。
  • 向 DC-A 对等节点（A2）发送 AppendEntries：约 1 ms RTT + 约 1 ms fsync = 2 ms。

向 DC-B 对等节点（B1、B2）发送 AppendEntries：20 ms RTT + 约 1 ms fsync = 21 ms。 多数 = 3 个确认：主节点 + A2 + 第一个 DC-B 节点，共 21 ms。 提交延迟：21 ms。 无跨数据中心时：2-3 ms。 每次写入都承担 20 ms 的代价

无论客户端在何处——没有本地写优化，因为 Raft 要求多数节点确认，而多数节点跨越了数据中心。

11.3 何时不应跨数据中心运行 etcd——RTT > 50 ms

  • 当跨数据中心 RTT 超过约 50 ms 时，跨 DC 的 etcd 将变得不可用：
  • 50 ms 的提交延迟意味着吞吐量崩溃（需要约 500 个并发写入，超出合理的 Raft 流水线能力）。
  • 心跳调优变得脆弱（选举超时必须达到 500+ ms，导致故障转移需要整整 1 秒）。

跨地域网络抖动（瞬时拥塞、边界网关协议（BGP）重收敛）会触发虚假选举。 这一阈值已被业界广泛认可： 如果 RTT > 50 ms，不要在跨地域部署单个 etcd 集群。

Kubernetes 上游建议将 etcd 保持在同一地域内。

  • 11.4 替代方案 每个地域部署一套独立的协调服务。
  • Kubernetes 联邦为每个集群运行一个 etcd。 通过 Kafka topic、变更数据捕获（CDC）或 gossip 协议将跨地域状态以带外方式进行复制
  • 接受跨地域的最终一致性。 DC-A 中新创建的 pod 在 DC-B 的注册表中需等待复制延迟后才会出现。

Spanner 和 CockroachDB 通过运行多个 Raft 组来规避这一问题，每个组放置在一个地域内；跨地域事务通过 2PC 跨多个组协调。它们并不将单个 Raft 组拉伸到多个地域——这一区别值得了解。

§12. Chubby vs etcd vs ZooKeeper

这三种经典协调服务同出一脉，在表层运维特性上存在分歧。

12.1 Chubby（Burrows，2006）——始祖

Mike Burrows 的 OSDI 2006 论文 《The Chubby lock service for loosely-coupled distributed systems》 奠定了这一设计空间的基础。Chubby 是 Google 内部系统，从未开源。

设计：

  • 每个数据中心部署一个 5 节点 Paxos cell。
  • 分层文件系统风格的 API（/ls/cell/path/to/object).
  • 粗粒度锁 ——锁持有时间以 小时计，而非微秒。
  • 用于变更通知的事件通道——watch 机制的始祖。
  • KeepAlive RPC ——长轮询，在服务端有新消息时返回。

Burrows 的著名观察： Chubby 最终被更多地用作名称服务而非锁服务 ——每个 Google 服务都通过 Chubby 查询"GFS 集群 X 的主节点在哪里"。"名称服务宕机"成为 Google 内部一个独立的故障类别。

12.2 etcd 和 ZooKeeper 作为衍生系统

  • ZooKeeper （Yahoo，2008）：开源版 Chubby。分层 znode，基于 session 的存活检测，类似的 watch 机制。使用 Zab 共识而非 Multi-Paxos；基于 JVM。
  • etcd （CoreOS，2013）：扁平 KV（比 znode 更简单），Raft 共识，Go 二进制。2014 年被 Kubernetes 采用，成为事实标准。
Chubby (2006, Google)  ──► ZooKeeper (2008, Yahoo)   ──► etcd (2013, CoreOS)
   hierarchical          hierarchical znodes              flat KV with MVCC
   Multi-Paxos           Zab consensus                    Raft consensus
   KeepAlive             sessions                         leases
   coarse locking        one-shot watches                 replayable watches

12.3 KeepAlive（Chubby）vs Lease（etcd）vs Session（ZooKeeper）

三者解决同一个问题——服务端对客户端存活状态的判断——各有其实现方式：

Chubby KeepAlive：客户端长轮询；服务端在有新消息或租约即将到期时返回。若在租约窗口内未收到 KeepAlive，租约将被释放。

etcd Lease：具有 TTL 的独立对象（Grant(ctx, ttl)）；键通过 WithLease(leaseID)附加到租约；通过 gRPC 流式 KeepAlive 调用续约。租约到期将删除所有附加的键。

ZooKeeper Session：TCP 连接 + 协商超时（6-30s）的 session ID；每隔 timeout/3 发送一次心跳。session 到期将删除该 session 拥有的所有临时节点。

区别：

  • etcd 的 lease 对象独立于键。 一个 lease 可附加多个键——续约可同时维持整组键的存活。比 ZK 的"每个临时节点绑定到 session"方式更简洁。
  • ZK 的 session 绑定到 TCP 连接。 断线重连是 session 恢复操作；session 超时会丢失所有 watch。etcd 的 lease 可通过任意新连接续约——容错性更强。
  • Chubby 将存活检测与事件投递合并为一个机制。 单一机制同时承担两项职责——巧妙，但比将两者分离更难推理。

现代主流倾向于 etcd 的模型——Kafka KRaft 选择弃用 ZooKeeper 在一定程度上也反映了这一设计偏好（KRaft 更接近 etcd 而非 ZK）。

12.4 Google 构建 Chubby 的原因——将"粗粒度锁"用作 DNS

Chubby 最初为 GFS 主节点选举而构建，随后用于 BigTable tablet 所有权管理，再到 Borg。 其主要用途从锁服务转变为名称服务。 核心观点： "协调服务既是锁服务也是名称服务；API 表面看起来是前者，而生产中的主要用途是后者。"

§13. 容量上限，跨部署场景说明

现代硬件（NVMe SSD、10 Gbps 网络、数据中心内部）上，3 节点 etcd 类 Raft 集群的粗略容量上限。

写吞吐量：持续约 10k/秒，峰值约 30k

每次写入的关键路径：主节点 fsync 0.5-2 ms（NVMe），网络 RTT 从机架内 0.2 ms 到地域内 10 ms，从节点 fsync（并行）0.5-2 ms，apply fsync 约 0.3 ms。 端到端约 2 ms p50，约 8-10 ms p99。 持续超过约 10k/秒时，fsync 队列积压，Raft 心跳开始错过截止时间。

对比分片 MySQL：单个主节点可处理 20-40k 次写入/秒，64 分片集群可达 120 万+次/秒。协调服务的写入速度比分片 OLTP 数据库慢 100 倍——这是有意为之的设计，每次写入都需要为法定人数 fsync 付出代价。

数据量上限：约 8 GB

etcd 文档规定默认上限为 2 GB，硬上限为 8 GB。超出后：BBolt 难以进行压缩/快照；新从节点在 8 GB / 1 Gbps 的条件下需要 64 秒的流式传输，期间无法参与法定人数；内存中的 MVCC 索引将超出合理的 RAM 范围。

Watch 扇出：约 10k 个流

单个 etcd 集群可维持约 10k 个活跃 watch 流（通过 gRPC），以及数十万个独立键 watch（前缀 watch 算作一个流）。事件吞吐量与写吞吐量同步——每次提交都会扇出到 N 个匹配的 watcher。

跨部署场景说明

Kubernetes etcd，小规模（约 100 个节点）：100 MB 状态，100-500 次写入/秒，约 50 个 watcher。3 节点 etcd，约 1 ms 写延迟。

Kubernetes etcd，大规模（5,000+ 个节点；GKE、OpenAI 训练集群）：2-6 GB 状态，峰值 5-10k 次写入/秒，通过 apiserver 维持数百个长期 watch。3 或 5 节点，写入 p99 约 5 ms。强制执行积极的压缩策略；CRD 和事件有时被单独分片。

Kubernetes etcd，超大规模（阿里巴巴、字节跳动：100k+ 个节点）：接近 8 GB。多个 apiserver 副本承载 watch 扇出。设有专门的 SRE 轮班负责 etcd 健康监控。

Kafka KRaft（3.3+，2022 年）：在 KRaft 之前，LinkedIn 规模的 Kafka 在 200k+ 分区时遭遇 ZooKeeper 瓶颈——数十万个 znode，每个 broker 的 watch 扇出，控制器故障转移耗时 分钟级 ，因为新控制器需要扫描整个 ZK 树。KRaft 将元数据迁移到由内嵌 Raft 支持的 Kafka topic（__cluster_metadata）中。控制器故障转移降至个位数秒；Kafka 现在可扩展至 每集群 100 万+ 个分区.

HashiCorp Consul，服务网格：数万个服务，约 5k 次写入/秒，来自每个代理 sidecar 的 watch（可能有 50k+ 个代理）。内置 SWIM gossip 协议处理存活检测，避免将心跳流量压到 Raft 上。

ZooKeeper 用于遗留 Hadoop / HBase：状态仅几 MB，写入速率低，数百个 watcher。已属遗留技术，因为 JVM 的运维成本高于 etcd。

Google Chubby cell（5 节点 Paxos）：锁持有时间以 小时 计——粗粒度锁，而非微秒级互斥锁的替代品。持有 GFS 主节点租约、Bigtable tablet 所有权、Borg 主节点锁。

问题 1——写吞吐量上限

直觉上的修复方案。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： p99 延迟更高 higher p99 latency (~12 ms)，吞吐量下降或持平。适得其反。

示意图 — Kubernetes apiserver 在大规模节点变动期间。在一个 5,000 节点的集群上执行节点驱逐时，Pod 状态更新的速率超过了 etcd 10k/秒的写入上限。

真正的解决方案：(1) 在源头降低写入速率（批量处理、合并写入、将临时状态迁移出去）；(2) 在工作负载层面进行分片（Kubernetes 对事件采用此方式）；(3) 对于确实规模庞大的元数据，切换到内嵌的分片共识拓扑（CockroachDB 模式）。

重点： 一个 Raft 组的写入吞吐量从根本上受限于 quorum fsync。你无法扩展单个 Raft 组；你需要创建更多的 Raft 组。

问题 2 — Watch 风暴 / 扇出放大

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 将每个来自 etcd 的事件发送给每个感兴趣的客户端。

示意图 — Kubernetes apiserver，5,000 节点集群。20 万个 Pod；每个 kubelet、controller 和 operator 都在 watch Pod 资源。10k 次写入/秒 × 10k 个 watcher = 1 亿事件/秒。以每个事件约 1 KB 计算，出口流量达 100 GB/秒，而集群聚合网络容量仅约 3.75 GB/秒，超出预算 25 倍。

真正的解决方案。apiserver 执行 watch 聚合：每种资源类型只向 etcd 注册一个 watch，再扇出到数千个 apiserver 客户端。etcd 看到的是 O(resource_types) 个 watch，而非 O(clients) 个。同样的模式也出现在 Consul（agent 集群为 Raft 集群做屏蔽）和 Kafka（KRaft controller 与 broker 通信；broker 再与客户端通信）中。

经验教训： 协调服务不是扇出引擎。

问题 3 — 分区下的脑裂防护

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： "让两侧继续提供服务。"

示意图 — 机架交换机故障期间的服务网格控制平面。跨机架部署的 Consul 集群；顶部机架交换机故障，导致主节点被隔离。若被隔离的主节点继续确认写入，则在恢复后，分叉的历史记录将损坏服务注册信息。

逐步分析：主节点 A 处于 term 15，commitIndex=1000；分区将 A 隔离；A 继续接收写入，但无法获得多数票。若某个存在 bug 的实现"乐观地"进行确认：A 的日志中包含 1001-1050 条乐观写入，B+C 在 term 16 选出新主节点，B 的日志中有已提交的 1051-1100 条。恢复后，A 看到 term 16，退位，截断 1001-1050 条记录。 已确认 1001-1050 条记录的客户端数据是错误的。

真正的解决方案：少数派一侧 拒绝写入。在约 3 个选举超时时间内未获多数票后，即便是基于租约的线性化读取也会停止。在分区下优先保证一致性（CP 策略，Consistency over Availability）。出错比变慢更糟糕。

问题 4 — 超出 8 GB 的规模限制

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： "增加数据配额。"每个问题都会随之恶化：快照传输（16 GB / 1 Gbps = 128 秒），MVCC 索引超出内存，压缩操作停滞。

示意图 — LinkedIn 上 KRaft 之前的 Kafka。ZooKeeper 的 znode 数量突破了数十万；controller 故障转移耗时数分钟。这与不同系统中同一类问题的形态相同。

真正的解决方案 — 拓扑变更：找出正在增长的数据；将其迁移出去（事件、大型 CRD、不应存在于此处的 secret）；若元数据确实规模庞大，则切换到内嵌的分片共识方案（CockroachDB / TiKV 模式）。

问题 5 — 跨区域协调

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： "将 etcd 集群跨区域伸展——5 个节点，A 区 3 个，B 区 2 个。"

示意图 — 全球服务发现。主节点在 A 区；quorum = 3；若 A 区的两个从节点瞬时变慢，每次提交都需要等待一个 B 区从节点，RTT 超过 30 ms。p99 延迟大幅恶化。区域间分区发生时：B 区（2 个节点）无法形成 quorum，进入只读和过期状态。

真正的解决方案: 不要这样做。 协调服务应限于单一区域。每个区域拥有自己的协调服务。对于跨区域感知，通过带外方式复制状态（Kafka topic、CDC、最终一致性方案）。

重点： Raft 不是多区域协议。 Spanner / CockroachDB / YugabyteDB 通过运行多个 Raft 组（每个分片一个）并按区域分布来实现跨区域能力——它们并不是跨区域运行单个 Raft 组。

问题 6 — 磁盘损坏 / 单节点数据丢失

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 重启并通过 Raft 日志追赶。

逐步分析：节点 B 的 SSD 在 BBolt 的 meta 页上出现位翻转。WAL 完好，BBolt 不可读。B 拒绝启动。集群处于 3 节点中的 2 节点状态——再发生一次故障就会宕机。

真正的解决方案: etcdctl member remove B，清空数据目录， etcdctl member add B-prime，以 --initial-cluster-state=existing启动。B-prime 接收完整快照，通过日志重放追赶，重新加入集群。另外需要备份： etcdctl snapshot save 生成时间点备份；RPO 等于备份周期。

§16. 主节点选举风暴

一种值得单独讨论的微妙故障模式。

16.1 风暴模式

网络抖动导致主节点与从节点断开约 10 秒。从节点的选举计时器触发；term 递增：

t=0   : leader A at term 15.
t=5s  : partition; A isolated.
t=5.3s: B's election timer fires; → candidate at term 16.
t=5.4s: C's election timer fires; split vote at term 16.
t=5.6s: B and C retry at term 17 (new randomized timeouts).
t=5.7s: B wins at term 17.
t=10s : network heals; A sees term 17, steps down.

在调优良好的集群中，风暴在 500-800 ms 内完成，几乎不被察觉。

16.2 当风暴无法停止时

病理过程：分区恢复后所有客户端同时重连；大量重连触发的写入使新主节点的 WAL fsync 饱和；新主节点错过心跳；又一次选举触发。term 计数器迅速攀升；每个重连的客户端都会收到 ErrNotLeader.

症状： etcd_server_leader_changes_seen_total 每分钟从 0 跳升到数十次。 etcd_server_proposals_failed_total 上升。Apiserver 日志中充斥着"etcd member down"。

16.3 缓解措施

  • 客户端抖动重连退避 （官方 etcd 客户端库已实现此功能；许多自制客户端未实现）。
  • Raft PreVote ——在递增自身 term 之前，候选节点先询问"你会在下一个 term 投票给我吗？"若多数节点因已有主节点而回答否，则候选节点不推进其 term。可阻止"被隔离节点返回、term 跳升、当前主节点退位"的模式。
  • 选举超时调优。 默认值为 150-300 ms；在高 RTT 或高负载下，延长至 500-1000 ms。
  • 监控 term 递增。 健康的集群每周 term 递增几次；term 突然增加是红色预警。对 etcd_server_leader_changes_seen_total > 5/hour, raft_term 每 24 小时平均递增超过 1 次发出告警。

16.4 从进行中的风暴中恢复

  1. 停止输入负载（暂停 controller，清空 watch 数据流）。
  2. 重启慢节点——通常某一个节点是根源所在。
  3. 紧急增大选举超时，然后恢复原值。
  4. 保持 PreVote 开启（不要禁用它）。

操作纪律：先停止失控的反馈循环，再让系统自行稳定。

§17. 故障模式

主节点在写入中途崩溃

状态：A 是 term 15 的主节点，已将条目 5000 追加到 WAL，并发送了 AppendEntries。B 完成 fsync 并确认。A 在响应客户端之前崩溃。

恢复：B 和 C 检测到心跳缺失；选举超时；B 成为候选节点。B 拥有条目 5000；B 的 lastLogIndex=5000。B 在 term 16 赢得选举。B 将条目 5000 传播给 C（如需），在 term 16 提交。A 重启后看到 term 16，退位。

持久性要点：第 3 步中 B 的 fsync。一旦从节点完成 fsync，该条目就能在主节点故障后存活——Raft 的日志完整性规则保证下一任主节点拥有所有已提交的条目。

少数派分区

状态：5 节点集群，A 是主节点。分区将 A+B 与 C+D+E 隔离。

A 一侧（少数派）：仅 5 节点中的 2 节点——A 无法提交。读取请求（若由 A 提供服务）将继续返回过期数据，直至租约到期（默认约 10 秒），之后停止服务。

C+D+E（多数派）：发起选举，在 term 16 产生新主节点，接受写入。

恢复：A 看到 term 16，退位，截断分叉点之后所有未提交的条目，追赶进度。

持久性要点：A 一侧未提交任何内容。未提交的条目可以安全丢弃，因为它们从未被确认过。

单节点磁盘变慢

状态：B 的磁盘性能下降——fsync 从 1 ms 恶化到 200 ms。主节点 A 向 B 和 C 发送 AppendEntries；C 在 1 ms 内完成 fsync，A 获得多数票，提交成功。B 持续落后； matchIndex[B] 落后。集群正常运行，但不再能容忍进一步的故障。

检测: etcd_disk_backend_commit_duration_seconds, etcd_network_peer_send_failures_total. 缓解措施：更换 B 的磁盘；以新成员身份重新加入。

快照传输过载

状态：B 离线时间过长。主节点的日志已在 B 的 nextIndex之后被压缩。主节点必须发送快照——完整的 BBolt 文件（约 8 GB）通过网络传输。传输期间，B 不参与 quorum。若 B 是 3 节点中的一个，则以 2-of-3 状态运行，可容忍 0 次故障。

缓解措施：保持 --snapshot-count 足够小，以防节点落后太多。不要让节点长时间离线。对于状态数据量非常大的情况，每次只替换一个成员，切勿并行操作。

永久性参与者丢失

状态：永久性丢失 5 节点中的 2 个。quorum 变为 3-of-3——再发生一次故障集群就会宕机。

恢复：逐个替换丢失的成员。对每个成员： member remove，然后 member add，然后以 --initial-cluster-state=existing启动替换节点。每次替换都会触发快照传输。切勿并行操作——这会导致数据分叉；这正是 Raft 联合共识成员变更协议旨在防止的情况。

从节点慢速滞后——滞后持续增长直至崩溃

节点 B 的网络接口卡（NIC）丢弃 20% 的数据包。由于 A 和 C 构成 3 节点中的多数派，写入仍然成功——B 在技术上是投票者，但实际上未作贡献。数小时后 B 的 matchIndex 落后数千条条目。最终 B 的 nextIndex 退回到日志压缩范围内，迫使主节点发送完整快照——传输期间 B 不可用。

检测: etcd_network_peer_round_trip_time_seconds 某个 peer 的持续上升；两者之间的差距 matchIndex[B] 。 commitIndex 超过约 10,000 条条目。

缓解措施：更换 B 的 NIC；若集群已在降级状态运行，则驱逐并重新添加该节点。

磁盘写满后的启动死循环

--quota-backend-bytes 设置为 8 GB；集群达到 alarm:NOSPACE。某运维人员清空一个节点的数据目录，希望"重新开始"。该节点接收到快照，但快照大小为 8 GB，而节点配额也是 8 GB。快照安装立即再次触发 NOSPACE；节点 panic，重启，再次触发告警。 启动死循环。

"etcd 已满"的恢复流程必须包含 compact+defrag 之前 重新添加节点，或先提高配额。将"etcd 已满"视为计划性维护，而非快速修复。

部分分区下"没有 quorum 能就主节点达成共识"的死锁

一个 5 节点集群存在 部分 分区：A-B 互相可见，C-D 互相可见，E 能看到 A 和 D，但 A-B 无法看到 C-D。每个节点都试图选举自己，而每个"多数派"都依赖于 E 的投票，而 E 的票被分散在重叠但不同的多数派之间。term 迅速递增；没有主节点能稳定下来。

检测: raft_term 迅速递增，而 etcd_server_has_leader 在各节点间于 0 和 1 之间抖动。

缓解措施：这是一个网络层问题。修复分区；PreVote 和选举超时调优可以减少抖动，但无法解决根本问题。一旦任意 3 个节点形成连通分量，集群将自动恢复。

§18. 可观测性

18.1 协调服务的四个黄金信号

信号 etcd 指标 阈值
主节点存在性 etcd_server_has_leader 任意节点超过 30 秒为 0 时告警
提交延迟 etcd_disk_wal_fsync_duration_seconds p99 超过 100 ms 时告警
后端延迟 etcd_disk_backend_commit_duration_seconds p99 超过 250 ms 时告警
提案压力 etcd_server_proposals_pending 持续超过 100 时告警

18.2 "fsync 耗时 1 秒"告警可预测磁盘故障

etcd_disk_wal_fsync_duration_seconds 是预测 SSD 故障最有效的指标。健康的 NVMe 的 p99 fsync 延迟为 1-5 ms。随着 SSD 老化及其重映射表填满，延迟呈现双峰分布——大多数 fsync 在 1 ms 内完成，偶尔出现 1 秒以上的延迟。这种双峰性先于彻底故障数小时至数天出现。

- alert: EtcdSlowFsync
  expr: histogram_quantile(0.99, rate(etcd_disk_wal_fsync_duration_seconds_bucket[5m])) > 0.1
  for: 5m

当此告警触发时， 主动更换磁盘。在健康集群上进行例行维护，好过集群因硬件故障降至 2-of-3 后再处理事故。

18.3 其他关键指标

  • etcd_server_proposals_failed_total：应接近零；上升 = 风暴或配额超限。
  • etcd_server_leader_changes_seen_total：每周数次；上升 = 风暴（§16）。
  • etcd_mvcc_db_total_size_in_bytes：BBolt 大小；在 6 GB 时告警。
  • etcd_mvcc_db_total_size_in_use_in_bytes：已用空间与总空间之比；差值较大 = 执行 defrag。
  • etcd_network_peer_round_trip_time_seconds：各 peer 的 RTT；可识别慢速从节点。
  • etcd_debugging_mvcc_keys_total：应小于约 100 万。
  • grpc_server_handled_total{grpc_code=...}：API 成功率/超时率。

18.4 Kubernetes 专项

额外的 apiserver 指标： apiserver_storage_objects{resource} （每种资源的计数；峰值预示 etcd 容量问题）， apiserver_watch_events_sizes （大于 1 MB 的大事件是反模式）， apiserver_request_duration_seconds （etcd 变慢首先在此处体现）。

大多数"Kubernetes 变慢"的事故都可追溯到 etcd；apiserver 指标是症状，etcd 指标才是原因。

§19. 为何不用 Redis SETNX 实现分布式锁

一位初级工程师说："我直接用 Redis SETNX 实现分布式锁就好了。亚毫秒延迟，而且已经部署好了。"让我们逐一分析这为何是错误的。

故障 1 — GC 暂停 / 调度延迟破坏租约语义

# Naïve Redis lock
if redis.set("lock:account:X", worker_id, nx=True, ex=30):
    do_critical_section()   # transfer $500
    redis.delete("lock:account:X")

Worker A 在 t=0 时获取锁。Java GC 暂停（或内核调度器延迟，或虚拟机监控程序抢占）持续 35 秒。Redis key 在 t=30 时过期。Worker B 在 t=31 时 SETNX 成功，开始执行其临界区。A 在 t=35 时醒来，仍认为自己持有锁；A 继续执行转账操作。 A 和 B 同时执行临界区。 账户 X 被扣款两次。

修复方法： fencing token ——每次获取锁都返回一个单调递增的令牌；下游资源拒绝令牌小于其已见过的最大值的操作：

token = redis.incr("lock:counter:account:X")
if redis.set("lock:account:X", token, nx=True, ex=30):
    db.transfer(amount=500, fencing_token=token)
    # DB checks: token >= last token seen for this account

故障 2 — Redis 故障转移导致计数器丢失

Redis 通常以异步复制模式运行（primary→replica）。

  1. Worker A： incr counter → 42。以令牌 42 获取锁。
  2. primary 崩溃 之前 ，尚未复制该递增操作。
  3. replica 晋升。其计数器仍为 41。
  4. Worker B： incr counter → 42（又是 42！）。以令牌 42 获取锁。
  5. 两个 worker，都持有围栏令牌 42。 数据库的"必须 >= 最近已见值"检查无法生效——两者持有相同的令牌。

核心故障： Redis 的复制不是共识。 异步意味着故障转移后的状态不是故障转移前状态的一致前缀。计数器可能回退。锁可能被授予两次。

故障 3 — Redlock 与 Kleppmann 的批评

Antirez 提出了 Redlock——在多数 N 个独立 Redis 节点上获取锁，并设定严格的时间边界。Martin Kleppmann 在 2016 年的批评：

  • Redlock 依赖有界时钟漂移。NTP 同步、虚拟机监控程序暂停、闰秒——安全性会失效。
  • Redlock 无法提供在故障后保持全局唯一的单调围栏令牌；每个 Redis 节点有自己的计数器，且它们之间无法达成一致。
  • 客户端上的 GC 暂停仍会破坏 Redlock，因为过期是基于挂钟时间的。

为何协调服务能正确处理这个问题

  1. 每个操作都经过共识流程。 锁获取是一条 Raft 日志条目，全局有序，在多数节点上持久化。
  2. 围栏令牌来源于 Raft 提交索引 （或 etcd 的 revision，或 ZooKeeper 的 zxid）。在所有故障场景下单调递增。即便在主节点切换后，新主节点的提交索引也从旧主节点结束的地方继续。
  3. 租约语义与共识绑定。 过期由协调服务对客户端存活状态的判断决定，而非由客户端的本地时钟决定。
  4. 线性化。 锁要么持有，要么不持有。

正确答案： "Redis SETNX 是一个快速的单节点原语。协调是一个多节点共识问题。若'低延迟锁'才是真正的需求，解决方案是减少竞争，而非削弱一致性。"

§20. 迁移模式

协调服务很少从零开始构建；从 ZooKeeper 迁移到其他方案是最常见的，因为 ZK 历史最久，JVM 内存占用较大，且一次性 watch 的设计随时间推移日益老化。

20.1 ZooKeeper → KRaft（Kafka 的迁移）

Apache Kafka 从 2011 年到 2022 年使用 ZooKeeper 管理集群元数据。在 LinkedIn 规模下达到了瓶颈：20 万以上的分区对应 20 万以上的 znode；controller 故障转移需要扫描整个 ZK 树，耗时 分钟级；JVM 调优是沉重的运维负担。

KIP-500（2019 年提出），在 Kafka 3.3（2022 年）中落地，用 KRaft ——Kafka 内嵌的 Raft 机制——取代了 ZK。集群元数据迁移到了一个内部 topic __cluster_metadata，由 3 或 5 个 controller 节点上的专属 Raft 组提供支持。

迁移阶段：

阶段 1 — 过渡版本。 Kafka 2.8（2021 年）：实验性内嵌 Raft 作为备用元数据存储。元数据模型经过重写，使其可以同时由 ZK 或 KRaft 提供服务（"schema mapping 步骤）。

第二阶段 — 双写。 混合模式下的生产集群：每次写操作同时写入两端，ZK 保持权威性。在成熟环境中持续运行 6-12 个月。

第三阶段 — 切换。 KRaft 成为权威；ZK 降级为影子节点。短暂下线操作（数分钟）。

第四阶段 — 移除 ZK。 Kafka 4.0（2025 年）仅随 KRaft 发布。

经验教训：

  • 多年迁移不可避免 ——在此规模下，季度级别的一次性切换并不现实。
  • 双写阶段是最长的工期 ——在此期间发现 schema 阻抗、竞态条件和语义缺失。
  • 切换本身很短暂。 所有的工程投入都在 路径之中，而非仅仅关注 事件.

20.2 ZooKeeper → etcd（类 HBase 系统）

与 Kafka 的模式如出一辙：

  1. Schema 映射：层级式 znode → 带前缀约定的扁平键；顺序 znode 计数器 → etcd 的 revision。
  2. 双写：每个协调事件同时写入两端；读操作仍来自 ZK。
  3. 影子验证：验证器读取 etcd，与 ZK 对比，出现差异时告警。
  4. 切换：读操作切换到 etcd；写操作双写一段时间；最终停止对 ZK 的写入。

最棘手的问题： 临时节点（ephemeral node）与租约（lease）的差异。 ZK 会话绑定的临时节点语义与 etcd 租约绑定的键（会话基于 TCP 连接，而租约可通过任意连接续期）并不是 1:1 的对应关系。影子阶段可以暴露这些细微的 TTL 语义不匹配问题。

20.3 生产环境中遗留的 ZooKeeper 集群

大多数生产环境中的 ZK 集群并未被主动迁移，而是作为遗产被继承。运维规则：固定 JVM 版本并调优堆内存（ZK 对 JVM 敏感）；监控 outstanding_requests 。 latency_avg；通过快照 txnlog + snapshot；规划迁移但不要仓促行事。

核心观点： "每一次成功的协调服务迁移都历经多年，且均采用双写阶段推进；如果有人提议一次性切换，正确的回应是'给我看回滚方案'。"

§21. 何时协调服务是错误的工具

大多数协调服务的生产故障并非 bug，而是误用。三种典型的反模式。

21.1 "10k 个 Pod 读取的配置"——错误，原因在于 watch 扇出

10,000 个 Pod 监听同一个配置键意味着 10,000 条 watch 流（超过 etcd 约 1 万的上限），且每次配置变更都会在单个 Raft 提交中产生 10,000 个事件（§7 中所述的惊群效应）。

可替代的方案：

  • 带 agent 集群的 Consul KV （每个主机的 agent 缓存配置；Pod 与 localhost 通信；agent 监听 Consul）。
  • 配置服务器 + 缓存：一个专用服务读取 etcd（或数据库），通过 HTTP 长轮询或服务端推送事件（SSE）向 Pod 提供服务；etcd 只有一个 watch。
  • 基于拉取的配置：通过 apiserver 提供的 ConfigMap，LaunchDarkly 风格的特性标志服务器——Pod 每隔几秒通过条件 GET 拉取。

核心模式： 协调服务持有事实来源；向众多读取方分发是由聚合器处理的独立关注点。

21.2 "事务状态"——错误，原因在于 etcd 不是数据库

ZK 的 multi 和 etcd 的 Txn 提供原子批量操作，但不具备完整的 SQL 事务模型——没有事务中途应用失败时的回滚，没有外键完整性，没有丰富的查询能力。ZK 和 etcd 上限约为 8 GB；事务型数据库可处理 TB 级数据。

可替代的方案：使用真正的事务型数据库（PostgreSQL、CockroachDB、Spanner）存储数据，协调服务只保存 数据库的元数据 （哪个分片、哪个节点是主节点、谁是主节点）。当只需要单行原子性时，使用 DynamoDB 的行级条件写入或 Cassandra 的轻量级事务（LWT）。

核心模式： 协调服务保存数据平面的元数据，而非数据平面本身。

21.3 "我们把用户数据放进 etcd 因为方便"——反模式点名批评

诱惑："我们已经为 Kubernetes 搭建了一个 etcd 集群，不如把少量的用户元数据也放进去搭便车。"

为何错误：etcd 的容量是为控制平面数据设计的。每用户记录乘以数百万用户将远超 8 GB 上限。每笔用户写操作都与控制平面写操作竞争；某个团队的写风暴会饿死 Kubernetes 的控制循环。共享 etcd 集群会将各团队的部署爆炸半径耦合在一起。

可替代的方案：为该使用场景单独搭建一个数据库，或者——如果该场景确实需要协调——使用一个 专属的 etcd 集群。两个 3 节点的 etcd 集群比一个耦合的爆炸半径便宜得多。

21.4 总结框架

协调服务是 错误的工具，当:

  1. 读者众多、更新频率低 → 使用带缓存的配置服务器或聚合器。
  2. 写者众多、需要事务状态 → 使用数据库。
  3. 数据量持续无界增长 → 使用 Blob 存储或分片数据库。
  4. 有隔离要求的用户数据 → 使用专用数据库。

正确的使用场景比看上去要窄得多： 主节点选举、锁 fencing、服务注册、集群成员管理、小型控制平面配置。 其他一切都是等待下次故障的误用。

§22. 扩展维度

两个不同的轴，不同的拐点，不同的应对方式。

类型 1 — 均匀增长（客户端更多、watch 更多）

基线（1x）：1k 客户端，100 个 watch，1k 次写/秒，100 MB 状态。单个 3 节点集群。轻松的理想状态。

10x （1 万客户端，1k watch，5k 写/秒）：压力在于 watch 扇出，而非吞吐量。每次提交产生数百个事件。 解决方案：聚合层（apiserver 模式）。一个（或水平扩展的多个）聚合器进程维护 etcd watch，并向客户端重新广播。etcd 只有约 10 个 watch；聚合器处理 1 万个连接。etcd 层的拓扑保持不变。

100x （10 万客户端——大型集群中的每个 kubelet）：聚合器现在成为瓶颈。 水平扩展聚合器 ——多个 apiserver，每个处理一部分客户端。它们都 watch 同一个 etcd。etcd 仍为 3 个节点。 不要为客户端负载而水平扩展 etcd。

类型 2 — 状态增长（更多键、更大的值）

10x （状态从 100 MB 增至 1 GB）：压力在于快照传输时间、压缩停顿和 MVCC 索引内存。 解决方案：激进压缩（etcdctl compact），识别并剪除噪声资源类型（以 Kubernetes 事件为例）。

100x （状态超过 8 GB）： 结构性拐点。无法继续扩展单个集群。选项：

  1. 在工作负载层面分片 ——多个协调集群，每个集群管理一部分键空间，聚合层负责路由。Kubernetes 事件分片是其轻量级版本。
  2. 完全替换后端 ——k3s/k0s 通过 kine使用 SQLite/MySQL/Postgres。丧失真正的共识语义；仅适用于小型或边缘集群。
  3. 嵌入式分片共识 ——CockroachDB / TiKV 模式。每个键范围有自己的 Raft 组；数千个 Raft 组并行运行。元数据可扩展至 TB+。代价是极高的工程复杂度。

扩展协调服务的有趣之处在于 大多数操作在共识层面都是反模式。 你不会向上扩展；你会向 外.

§23. 决策矩阵与相邻类别对比

决策阈值

  • 不足 100 个服务，最终一致性可接受 → DNS 完全够用。
  • 100-1 万个服务，需要实时更新 → 带聚合器的协调服务（Consul、etcd）。
  • 需要 fencing 令牌 → 协调服务。Redis 在故障转移时无法提供单调递增的计数器。
  • 无需 fencing 的锁 → 如果真的不需要 fencing，Redis SETNX 更快。但请先暂停思考：你真的不需要 fencing 吗？大多数看似无害的分布式锁需求实际上都需要。

§24. 使用场景概览

24.1 Kubernetes API server 后端存储（etcd）

每个 Kubernetes 集群运行单个 etcd 集群，作为 所有 集群状态的唯一持久化存储——Pods、Nodes、Services、ConfigMaps、Secrets、CustomResources、租约、控制器书签。kube-apiserver 是唯一与 etcd 通信的组件。控制器通过 watch apiserver 进行协调，apiserver 将它们的 watch 请求聚合为少量针对每种资源类型的 etcd watch。教科书级别的控制平面设计：聚合器（apiserver）前置于微型共识核心（etcd）之上。

24.2 Kafka KRaft 元数据（嵌入式 Raft）

Kafka 3.3+ 将自身的 Raft 仲裁嵌入控制器节点，取代了 Kafka 从 2011 年到 2022 年使用的外部 ZooKeeper 集群。集群元数据（broker 成员、topic 配置、分区分配）保存在内部 topic（__cluster_metadata）由 Raft 支撑。控制器故障转移时间从数分钟缩短至数秒。经验教训： 在极端规模下，将共识嵌入系统本身往往优于运行外部协调服务 ——控制器状态可通过日志回放加载，而无需完整的树扫描。

24.3 HashiCorp Consul + 服务网格

Consul 将一个 5 节点的 Raft 集群（KV + 服务目录）与基于 SWIM 的 gossip 层（用于对等节点存活检测）捆绑在一起。Raft 持有权威状态；gossip 处理高频的"X 是否存活"查询，而不给 Raft 带来负担。服务网格代理（通过 Connect 的 Envoy）通过本地 Consul agent 消费目录，该 agent 充当每主机的聚合器。"协调服务加聚合器集群"的纯粹形式。

24.4 Cassandra 集群拓扑（gossip + 轻量级 Paxos）

Cassandra 将 gossip 用于大部分集群感知（节点存活、范围所有权），但依赖 嵌入式 Paxos 处理少数需要线性一致性的操作——schema 变更、轻量级事务（LWT）。LWT 直接在 Cassandra 节点之间使用 Paxos 实现比较并设置（compare-and-set）语义。混合模式：gossip 用于快速最终协调，嵌入式 Paxos 用于少数强一致性操作。

24.5 TiKV / CockroachDB 嵌入式 Raft（分片共识）

这些系统不为用户数据使用单独的协调服务。每个键范围（TiKV 中称为"region"，CockroachDB 中称为"range"）有自己的 Raft 组。一个集群并行运行 数千个 Raft 组并行运行，每个组管理约 64-96 MB 数据，并在 3-5 个节点间复制。一个独立的、更小的协调服务（TiDB 的 Placement Driver；CockroachDB 内部的 liveness 范围）处理集群级别的元数据。这种设计通过对共识问题本身进行分区，将协调语义扩展至 PB 级。

24.6 分布式 cron / 定时任务主节点选举

一个典型的小型用例：10 个工作节点协商由谁运行夜间批处理任务。每个节点尝试获取 /jobs/nightly-rollup/leader上的租约。获胜者运行任务；其他节点轮询。如果主节点在任务中途宕机，其租约过期，另一个节点获取租约并重新运行。20 行 etcd 客户端库代码即可实现。变体随处可见：Airflow 调度器选举、Kafka MirrorMaker、任何单活工作者模式。

24.7 特性标志 / 配置分发

特性标志服务将当前值保存在协调服务中。应用实例 watch 标志前缀。当运维人员更新某个标志时，每个实例都会收到 watch 事件，并在毫秒内更新进程内配置。这比每 N 秒轮询一次数据库要好得多，原因在于：(a) 传播是即时的，(b) watch 合约保证不会错过任何更新。LaunchDarkly 级别的系统和许多内部搭建的标志系统都使用 etcd 或 Consul 来实现这一功能。

§25. 真实案例数据

Kubernetes（etcd）。每个 K8s 集群运行 3 节点或 5 节点的 etcd。已知最大规模：阿里巴巴和字节跳动公开运行超过 10 万节点的集群；OpenAI 训练集群和 Google GKE 客户规模相当。状态 2-6 GB，偶尔达 8 GB。大规模变动期间峰值写速率为 5-10k 次/秒。通过 apiserver 的 watch 扇出是架构上最重要的层。

Kafka KRaft（取代 ZooKeeper，2022 年）。在 KRaft 之前，LinkedIn 规模的 Kafka 拥有 20 万+分区，遭遇了 ZooKeeper 瓶颈——数十万个 znode、每个 broker 的 watch 扇出、控制器故障转移耗时 分钟级。KRaft 用嵌入式 Raft 仲裁和 __cluster_metadata topic 取代了这一切。控制器故障转移时间降至个位数秒；Kafka 现在可在单集群内扩展至 100 万+分区。

Google Chubby（2006 年，Burrows 等人）。Google 数据中心内的 5 节点 Paxos 仲裁组。GFS master 锁定、BigTable tablet 所有权、BorgMaster 选举的底层支撑，以及（出了名的）一个被广泛使用的名称服务，以至于"名称服务不可用"成为 Google 内部的一种故障类别。锁持有时间 小时 ——粗粒度、低竞争。

HashiCorp Consul。服务发现 + KV + 健康检查。5 节点 Raft 用于 KV+目录；SWIM gossip 用于存活检测。WAN 联邦允许 Consul 跨区域用于目录（不用于跨区域的强一致性 KV）。可扩展至数万个服务和代理 sidecar。

Apache ZooKeeper（Kafka KRaft 之前、HBase、Hadoop）。最初基于 Java 的协调服务。被 Kafka（3.3 之前）、HBase（region 分配）、Hadoop YARN（RM HA）、Solr（集合元数据）所使用。处理约 1-5 万次写/秒。现代系统正在迁移离开，因为 JVM 的运维负担比 etcd 更重，且一次性 watch 模型比 etcd 基于 revision 的可重放 watch 更难推理。

TiKV / TiDB（嵌入式分片 Raft）。每个"region"（约 96 MB）有一个 Raft 组。一个 500 节点的 TiKV 集群并行运行超过 10 万个 Raft 组。Placement Driver（PD）是一个独立的 3 节点 etcd 集群，用于集群级别的元数据。

CockroachDB。每个 range（约 64 MB）有自己的 Raft 组；一个 100 节点的集群运行数万个 Raft 组；在大规模下超过 10 万个。内部 liveness 。 system ranges 持有集群级别的元数据。

§26. 总结

协调服务是一类刻意设计得极小、极慢、极端 CP 的强一致性 KV 存储——etcd、ZooKeeper、Consul、Chubby——构建在单一的 Raft（或 Zab，或 Paxos）组之上，采用 quorum-fsync 写入，上限约为 1 万次写/秒和 8 GB 状态，专门用于持有所有其他分布式组件所依赖的控制平面事实来源；它们不是数据库、缓存或队列，而是解决"谁是主节点、分片在哪里、配置是什么、谁持有这把锁"的共识日志，具备线性一致性、在故障转移后仍有效的 fencing 令牌，以及有序的 watch 流；你不会向上扩展它们——而是通过在前面放置聚合器来处理 watch 扇出、剪除不属于它的内容、以及在单个 Raft 组不再够用时切换到嵌入式分片共识（CockroachDB / TiKV / Kafka KRaft）来向外迁移——因为只有真正的共识日志才能给你一个永远不会回退的计数器。
