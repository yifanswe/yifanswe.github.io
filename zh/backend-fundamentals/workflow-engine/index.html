技术参考。作为一类技术，工作流引擎——Temporal、Cadence、AWS Step Functions、Apache Airflow、Argo Workflows、Netflix Conductor——处于后端技术栈的特定层，解决特定问题（多步骤流程的持久化编排），并具有特定的内部结构（事件溯源加确定性回放，或其某种折中变体）。本文聚焦于技术本身。我们将看到它服务于支付生命周期、打车调度、ETL（提取转换加载）流水线、ML（机器学习）训练、视频转码、CI/CD（持续集成/持续交付）以及通知活动——不同领域利用相同的基础原语。

§1. 工作流引擎究竟是什么

一个 workflow engine 是使 multi-step process 能够在崩溃、重启、部署和时间流逝中存活下去的基础设施组件。你编写代码（或声明 JSON，或绘制图表），表达"执行 A；若 A 成功，执行 B；若 A 失败，进行补偿并通知；等待三天获取人工审批；然后执行 C"。引擎保证——一旦启动——该流程 无论途中何人或何物发生故障，都能运行至完成。状态是持久化的。步骤重试由引擎管理。执行路径可审计。

三个刻意宽泛的描述：引擎 编排 （它告诉其他系统该做什么，但实际工作在那些系统中发生），它是 持久 （状态在进程重启后仍可存活，在任何回复被确认之前已持久化到数据库），并且它具备 long-running-capable （工作流可以阻塞等待信号、定时器或人工操作，时间可能是毫秒，也可能是数月）。

工作流引擎不是 非 worker，它是 coordinator ，根据 worker 已完成的历史记录告诉 worker 下一步该做什么。

与相邻类别的区分。 从远处看，有几种技术很像工作流引擎，但实际上并不是。

  • Cron / 调度器。 由时钟触发。没有多步骤依赖的概念，没有从头重跑之外的重试机制，没有持久化状态，也没有补偿机制。Cron + 状态表是朴素的替代方案；§9 详细分析了它为何会失败。
  • 消息队列（Kafka、RabbitMQ、SQS）。 传递 消息。没有"工作流当前处于哪个步骤"或"在更大流程中重试这个特定活动"的概念。你可以在其上实现编排，但你已将所有工作流关切——状态、顺序、补偿、审计——都转移到了应用代码中。
  • Saga 库（Axon、Eventuate）。 帮助构建补偿结构的代码级库。仍然需要 某个 一个拥有持久化状态的底层。引擎同时取代了两者。
  • 批量 DAG 调度器（Airflow、Luigi、Prefect、Dagster）。 也是工作流引擎，但属于 批量形态 ——针对固定时间窗口运行固定 DAG（有向无环图）的每日/每小时数据任务，一次运行就是对固定 DAG 的一次执行。不同的关注点主导：回填、基于分区键的幂等任务重跑、数据血缘。它们并不针对"数百万个并发工作流，每个都处于唯一的执行点"进行优化。设计空间在它们与 Temporal 类引擎之间存在明显分野。
  • 函数即服务事件链（Lambda + EventBridge）。 没有协调者的编排。函数间的状态存在数据库中；重试是按函数级别，而非按工作流级别。

不适合的场景：

  • 亚毫秒级请求路径。 引擎在每次状态转换时都会写入仲裁复制存储；不适合 10ms p99 的 RPC 路径。工作流触发扇出工作；触发本身是异步的。
  • 线速纯数据转换。 每秒 1000 万个事件通过转换的场景，需要流处理器（Flink）。每次状态转换的开销过高。
  • 单个工作流的任意规模。 工作流历史是有界的（Temporal：10MB/50k 个事件）。一个工作流处理十亿条目是错误的做法；正确模式是父工作流扇出到子工作流，或 ContinueAsNew (§7).
  • 绕过确定性机制。 代码即工作流的引擎会对工作流代码进行沙箱化：禁止 time.Now，禁止随机数，禁止 activities 之外的 I/O。与此对抗就是与技术本身对抗。

§2. 固有保证

由设计提供的能力。

  1. 持久化工作流状态。 每次状态转换在被确认之前，都会持久化到仲裁复制存储中。若进程在工作流中途崩溃，接手的下一个 worker 通过读取持久化历史即可恢复到相同状态。没有工作流会被悄无声息地丢失。
  2. 从引擎视角来看的精确一次活动分发。 一个给定的调用——由一个稳定的 activity_id 标识——在任意时刻最多只有一个进行中的尝试。重试以稳定身份发生，在上一次尝试被宣告失败或超时之后。绝不会对同一活动有两个并行的调度。
  3. 带可配置退避的自动重试。 声明式策略：初始间隔、倍数、最大尝试次数、最大持续时间、不可重试的错误类型。活动失败由引擎捕获，而非用户代码。
  4. 确定性回放 （Temporal/Cadence 类）。状态可通过从事件 1 开始对持久化历史重新执行工作流代码来重建。相同的代码 + 相同的历史，始终得出相同的下一个决策。持久性代价低——无需工作流中途检查点，只需回放。
  5. 长时间等待作为一等原语。 工作流可以睡眠 30 天、等待信号或等待定时器，而无需持续占用进程资源。引擎以持久化方式调度唤醒。
  6. 完整的审计历史。 每个事件——启动、决策、活动调度/完成/重试、定时器触发、信号接收、完成——均包含时间戳、载荷和失败追踪信息。
  7. 工作流启动的幂等性。 StartWorkflow(workflow_id="order-123") 两次启动产生一个工作流——第二次返回已存在的运行实例。

未由设计提供，需另行叠加实现。

  1. 精确一次 副作用. 稳定 activity_id 在重试间保持一致；但并非严格的恰好一次外部效果。如果你的 activity 向 Stripe 发起了扣款，且网络在扣款后断开，重试将再次扣款，除非你传入幂等键。这是应用层的职责。
  2. 在负载下实现低延迟调度。 p50 延迟为个位数毫秒；有意不追求亚毫秒级。这是单次 RPC 预算内的错误工具选择。
  3. 单个工作流的吞吐量无上限。 一个工作流是单个分片的持有者；一个以 10k 事件/秒速率运行的工作流会持续冲击同一个持久化存储分区。引擎通过增加工作流数量来扩展，而不是提升单个工作流的速率（§10）。
  4. 绕过确定性约束。 调用 time.Now() 在工作流代码中使用，导致重放结果发散—— NonDeterministicWorkflowError。这是关键的安全保障，但它会暴露 bug，而非掩盖它们。
  5. 跨工作流的分布式事务。 两个工作流原子性地提交一个联合结果——该功能不被原生支持。需要在上层构建 saga（一个调用补偿 activity 的工作流）。

契约：引擎负责工作流级别的持久性、重试语义、历史记录和定时器机制。应用层负责副作用的幂等性、业务正确的补偿逻辑，以及工作流之外的数据平面。

§3. 设计空间

工作流引擎家族中有四种不同的设计，各有其核心侧重点，此外 Argo 作为第五种 Kubernetes 原生变体。

A. 以代码即工作流配合确定性重放（Temporal、Cadence）。 你用真实编程语言（Go、Java、Python、TypeScript、.NET）编写工作流代码。引擎在每个异步点进行拦截—— await ChargePayment(...), workflow.Sleep(2h), signal.Receive(ctx) ——并将结果持久化为历史事件。重放通过在事件日志上重新运行代码来重建状态。优势：完整的表达能力、IDE 支持、原生补偿（defer cancelBlock）、长期等待轻而易举、signals/queries 一等公民。广泛用于 业务流程编排 ——支付、订单生命周期、KYC（了解你的客户）入驻流程。代价：确定性约束要求严格纪律；SDK 底层机制必须理解才能调试；自托管运维需要大量工程投入。

B. 声明式状态机（AWS Step Functions、Azure Durable Functions 的某些模式）。 工作流是 JSON/YAML 状态机：状态、转换、条件分支、并行分支、重试。Activity 按名称引用（例如 Lambda ARN——Amazon Resource Name）。优势：无确定性约束、强大的可视化工具、紧密的云集成（Step Functions 可直接调用约 200 个 AWS 服务）。代价：表达能力有限，复杂流程 JSON 膨胀严重，存在厂商锁定。Step Functions Standard 上限约 4k 转换/秒/账户。最适合 云原生短流程编排 。 AWS 重度 ETL.

C. 批处理 DAG 调度器（Airflow、Prefect、Dagster、Luigi）。 由任务组成的 DAG，通常用 Python 编写。引擎根据参数化配置（通常是日期分区）调度 DAG 运行。"工作流"是一次 DAG run，通常按计划执行。优势：专为批处理 ETL/ML 设计——支持回填、基于分区键的幂等任务重跑、数据血缘、SQL/Spark 集成。代价：不适合"同时存在 1000 万个各处于不同阶段的并发工作流"这种模式。长期等待处理笨拙（依赖 sensors）。每个任务开销达数百毫秒。

D. 基于 worker 轮询的 JSON 工作流 DAG（Netflix Conductor）。 JSON DAG 任务（HTTP_TASK、DECISION_TASK、FORK_JOIN、SUB_WORKFLOW）。Worker 轮询、执行并 POST 结果。状态存储在引擎的数据库中。优势：语言无关，适合多语言团队。Netflix 用它处理视频编码（Cosmos）、推荐刷新、内容摄取。代价：表达能力弱于代码即工作流；无确定性重放（状态可变）；运维成熟度落后于 Temporal。

E. Argo Workflows——Kubernetes 原生。 Kubernetes CRDs（Custom Resource Definitions）定义工作流；每个步骤是一个 pod。状态通过 k8s 存储在 etcd 中。目标场景包括 CI/CD 流水线, ML 训练, 基础设施自动化。优势：与 Kubernetes 零距离，原生支持 GitOps。代价：每个步骤都是一个 pod（开销约为秒级），不适合高频微编排场景。

§4. 底层数据结构——字节级机制

这一节是区分 Staff 级理解与功能列表阅读的分水岭。深度差异在于： 事件溯源加上确定性重放.

4a. 事件溯源历史

核心抽象： 工作流即其历史，而非一行数据。历史是一个仅追加的有类型事件序列；状态通过 派生数据 在事件日志上重新运行工作流代码来重建。

Temporal/Cadence 类引擎中的两张表（Cassandra 为标准方案；小规模部署使用 MySQL/Postgres）：

CREATE TABLE history_node (
  shard_id     int,
  workflow_id  text,
  run_id       uuid,
  branch_id    uuid,        -- supports fork-on-reset
  node_id      bigint,      -- first event id in this batch
  txn_id       bigint,      -- optimistic concurrency token
  events       blob,        -- batch of N serialized events
  PRIMARY KEY ((shard_id), workflow_id, run_id, branch_id, node_id)
);

CREATE TABLE executions (
  shard_id      int,
  namespace_id  uuid,
  workflow_id   text,
  run_id        uuid,
  execution     blob,
  next_event_id bigint,
  PRIMARY KEY ((shard_id), namespace_id, workflow_id, run_id)
);

history_node 是不可变的事件日志（仅追加）。 executions 是每个工作流的一个小型可变指针：下一个事件 ID、当前分支、最新状态。对 executions 的修改使用 txn_id compare-and-swap——每个工作流的乐观并发控制。

一个事件是一个 protobuf：

EventID:    int64    (monotonic within a run)
EventType:  enum     (WorkflowExecutionStarted, ActivityTaskScheduled,
                       ActivityTaskCompleted, TimerStarted, SignalReceived, ...)
EventTime:  timestamp
TaskID:     int64    (global ordering within shard)
Version:    int64    (multi-cluster replication version)
Attributes: oneof    (per-event-type payload)

典型事件大小：200B-2KB。 history_node 中的一个批次包含 5-20 个事件以分摊行开销。硬上限： 50,000 个事件或 10MB，以先达到者为准。

4b. 为何事件溯源契合访问模式

两种主要操作：

  1. 在末尾追加事件。 纯顺序写入。Cassandra 的 LSM（Log-Structured Merge tree）正是为此设计：追加到 memtable，刷新到 SSTable（Sorted String Table）。每节点约 10-20k 写/秒轻松应对。
  2. 读取历史范围以进行重放。 在单个 Cassandra 分区内、单个聚类范围内进行顺序扫描。由 SSTables 和 memtable 共同提供服务。

以追加为主 + 偶尔批量范围扫描 = 正是 LSM 树的设计目标。替代方案——单个可变状态行——在三方面都会失败：无审计（可变行只显示当前状态）、无确定性重放（中间决策不可恢复）、无重试语义（开始与完成必须是不同的事件）。事件溯源是唯一能同时支持这三点的结构。

4c. 权衡：事件溯源历史 vs. 可变状态行

维度 事件溯源历史 (LSM) 可变状态行 (B+ tree)
审计 构造即完整 需要单独的变更日志
重放 原生支持 无变更日志则不可能
追加吞吐量 ~10-20k/秒/节点 ~2-5k/秒/节点（页面分裂）
读取最新状态 扫描历史 + 重放（已缓存） 单点读取
存储成本 多 10-100 倍 精简
Schema 演进 困难（事件永久不可变） 容易（ALTER TABLE）

选择事件溯源的代价：廉价状态查询（通过 sticky 缓存缓解）和 schema 灵活性（事件 proto 必须永久向后兼容）。收益：持久性、重放、审计——全部免费。

这就是为何 Conductor——采用更可变模型——的重放语义弱于 Temporal。这是架构层面的选择，而非表面差异。

4d. 按 workflow_id 分片

引擎按 hash(namespace + workflow_id) mod N进行分片，通常 N=4096（超大规模部署为 16k）。一个 分片 在同一时刻只由一个 history-service 节点持有。该分片持有其中所有工作流 ID 的内存互斥锁——集群范围内不会有两个请求并发修改同一工作流。该分片工作流的所有 Cassandra 行都在同一分区内；读写都在同一副本集本地完成。该分片还将 transfer/timer/replication 队列作为行维护在同一分区中。

当一个 history 节点宕机时，成员协议（Ringpop 风格）检测并将分片重新分配给存活节点。新持有者将 executions 行重新加载到缓存——停机时间约为数十秒，由于每个事件都经过了仲裁写入，不会丢失数据。

4e. Sticky 任务队列

Cassandra 负责持久性；重放代价高昂。 sticky-execution cache：一旦 worker 重放了某个工作流，它会保持内存状态存活，并持续处理后续决策任务，直到崩溃、内存压力下被驱逐，或工作流进入空闲状态。

First decision task for workflow_id=X:
  matching picks any free worker W from the task-queue partition
  W replays full history (N events) → in-memory state
  W signals: "I am sticky for X; send future tasks to me"

Subsequent decision tasks for workflow_id=X:
  matching routes directly to W's sticky queue
  W executes the new code; no replay needed
  Per-decision cost: O(new events), not O(total history)

W crashes:
  matching detects sticky timeout
  next decision goes to the global task queue
  some worker W' picks up, replays full history, becomes new sticky owner

在约 95% 的 sticky 命中率下，平均重放开销分摊到数百次决策中。若无 sticky 缓存，每次决策都需要完整重放——在中位 50 个事件的情况下，worker CPU 消耗约多 50 倍。

4f. 确定性约束

重放必须每次产生相同的决策。工作流代码运行在严格的沙箱中：

禁止操作 原因 替代方案
time.Now() 跨重放时钟不同 workflow.Now(ctx)
rand.Float64() 重放时随机数不同 workflow.SideEffect(ctx, ...) ——结果已持久化
os.Getenv / 外部配置读取 重放 worker 上环境不同 将配置作为工作流输入传入
直接 HTTP / DB / Kafka 调用 副作用必须可重放 封装在 Activity 中
Goroutines / 原生线程 / sleeps 非确定性调度 workflow.Go, workflow.Sleep
遍历未排序的 map 迭代顺序不同 遍历前先对键排序
修改全局状态 对其他重放不可见 使用工作流局部变量

SDK 通过自定义调度器强制执行：每个 await 都经过运行时，运行时查询历史；若事件已存在，返回存储的结果；若不存在，真实分发。这使得 await ChargePayment(input) 看起来像普通的异步代码，同时具备持久性。

若重放到某个决策与历史不匹配，引擎会抛出 NonDeterministicWorkflowError 并将工作流挂起。这个错误故意设计得很醒目。

4g. 一个具体的生命周期，端到端

一个标准的支付编排工作流，字节级视角。

T=0: API server 调用 client.StartWorkflow(workflow_id="order-abc", ...). Frontend 哈希 order-abc → shard 1742。持有 shard 1742 的 history-service 节点获取每工作流的互斥锁（内存中，作用域限于该分片）并写入：

event #1: WorkflowExecutionStarted (workflow_type=OrderWorkflow, input, run_id)
event #2: WorkflowTaskScheduled    (task_queue=ORDER_TQ)

Cassandra 追加，仲裁写入 3 个副本中的 2 个。 持久性要点：当仲裁确认后，工作流正式启动。history service 将决策任务入队。

T=2ms: Matching 分发决策任务。 Worker 长轮询，获取任务。无缓存状态（首个任务）。获取 GetHistory(order-abc, 1, 2)。执行重放。工作流代码运行到第一个 workflow.ExecuteActivity(ctx, ChargePayment, input)。返回一个 future。引擎收集一个命令。

T=5ms: Worker 发送 RespondWorkflowTaskCompleted 与 ScheduleActivityTask(activity_type=ChargePayment, activity_id=1, retry_policy, timeouts)。History 追加事件 #3 WorkflowTaskStarted, #4 WorkflowTaskCompleted, #5 ActivityTaskScheduled （一批 Cassandra，仲裁写入）。将 activity 任务入队至 CHARGE_TQ.

T=10ms: Activity worker 拉取任务。 使用以下参数调用支付提供商 idempotency_key=order-abc-1 (workflow_id + activity_id)。提供商完成扣款并返回 charge_id=ch_xyz。Worker 调用 RespondActivityTaskCompleted.

T=400ms：History 追加事件 #6 ActivityTaskStarted, #7 ActivityTaskCompleted(result=ch_xyz), #8 WorkflowTaskScheduled. 将新的 decision task 分发给 sticky worker。

T=405ms：同一 Worker，无需重放。 charge_payment_future.Get(ctx, &charge_id) 返回 ch_xyz 立即执行（事件已写入 history）。代码继续执行至 ExecuteActivity(ctx, ReserveInventory, ...)。事件 9-13 已写入，依此类推。

一个包含 4 个 activity 的 workflow 的总事件数约为 25。History 大小：5-10KB。

每一步的崩溃恢复能力。

  • T=2ms 之前（history 未写入）： StartWorkflow RPC 尚未返回。调用方使用相同的 workflow_id 重试；幂等。
  • T=2ms 至 T=5ms 之间（worker 已接取任务但未响应）： ScheduleToStart 超时触发，decision task 重新入队，另一个 worker 进行重放并重新发起相同的 ScheduleActivityTask。通过以下方式保证幂等 activity_id.
  • T=10ms 至 T=400ms 之间（activity worker 在调用过程中宕机）： StartToClose 或心跳超时触发。 ActivityTaskTimedOut 已追加。重试策略安排新的尝试。该 activity 可能已执行过一次副作用——这正是幂等性键的重要所在。

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 持久化节点 每一步的关键保障是 quorum 写入。在 ack 落盘之前，一切均未改变；ack 落盘之后，任何崩溃都无法撤销。

§5. 容量上限

Workflow 引擎覆盖四个数量级。

小型：10 到 1000 个。 在以下平台启动 AWS Step Functions 用于定时任务驱动的后端作业和用户引导。免费套餐每月 4,000 次状态转换，超出后约 $25/百万次。或一个小团队在 self-hosted Temporal 上使用三台虚拟机搭建支付流 POC：约 100 个并发，约 10 次转换/秒，约 50GB 磁盘。

中型：Coinbase。 将每笔提现/存款都封装在 workflow 中（合规要求单一可审计的流程记录）。2020 年代中期： 数百万级别的并发 workflow, 约 10k 次转换/秒（持续），大量信号用于人工审批节点。MySQL 分片，约 50 个 history 节点，约 20 个 matching 节点，约 200 个 worker。历史平均约 100 个事件；合规流程等待人工审核可长达数周。

大型：Uber Cadence。 在第 2 年达到 数百万个正在运行的 workflow, 每天数十亿个 activity 遍布数百个服务——行程生命周期、欺诈审查、支付结算、司机入驻。以 Cassandra 为存储后端，每个地区数百个节点，多地区主主架构。峰值约 50 万 decision tasks/秒，约 100 亿 activities/天，热数据保留约 4 周。

超大型：Stripe Connect、Snap、Netflix Conductor。 Stripe Connect：支付流需等待 ACH（Automated Clearing House）结算，耗时数天。 数千万个正在进行的支付. Netflix Conductor: 每天数千万次 遍布 Cosmos（视频编码/转码）、内容摄取、推荐刷新。 Snap：将 Temporal 用于广告创意审核和内容审核，运行中的 workflow 达数百万量级。

在这种规模下， 联邦化（federation） 是必须的。单个集群可处理数百万量级；超出后，需通过全局命名空间异步复制将命名空间分区到多个集群。爆炸半径管理成为首要问题。

§7. 该技术固有的难题

任何使用 workflow 引擎的人都会面临的六大挑战。这些都不是 bug，而是技术本身固有的特性。

§7.1 代码变更后的确定性违规

设置。 支付 workflow 正在生产环境运行。六个月后，你在 ReserveInventory 。 NotifyWarehouse之间新增了一个欺诈检查步骤。部署。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 修改代码，重新部署。

失败。 工作流 order-abc 包含事件 5-8：ReserveInventory 已调度/已完成，然后是 WorkflowTaskScheduled。通过新代码重放时发出 ScheduleActivityTask(CheckFraud)。但下一个 history 事件是 ScheduleActivityTask(NotifyWarehouse) （旧路径）。不匹配。 NonDeterministicWorkflowError。Workflow 挂起。乘以所有正在运行的 workflow——生产流量全面中断。

真正的解决方案。 GetVersion API 修补方案：

v := workflow.GetVersion(ctx, "add-fraud-check", workflow.DefaultVersion, 1)
if v == 1 {
    workflow.ExecuteActivity(ctx, CheckFraud, input).Get(ctx, nil)
}
workflow.ExecuteActivity(ctx, NotifyWarehouse, input).Get(ctx, nil)

首次执行时记录一个 MarkerRecorded 事件并附带版本号；重放时返回相同版本。部署前的 workflow 读取 DefaultVersion 并跳过欺诈检查；新 workflow 读取版本 1。

这种情况普遍存在于每个领域：Netflix Cosmos 转码 workflow 新增感知质量检查；ML 训练流水线新增超参数调优；Argo 中的 CI/CD 流水线新增 lint 阶段。修复方案相同：基于版本标记进行分支。

§7.2 History 膨胀：10MB / 50k 事件上限

设置。 一个 客户订阅 workflow——每位客户一个——永久存在。两年后它已积累 10 万个事件。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 让存储支持更大的 history。

失败。 重放成本。每个 worker 需读取 10 万个事件并对所有事件重新执行 workflow 代码。以 100 事件/毫秒的重放速度，每次 decision 需消耗 1 秒 CPU。对于 1000 万个长期存活的 workflow，根本无法运行。除 CPU 外，行大小限制、blob 序列化以及可见性管道在超过约 10MB 后都会阻塞。

真正的解决方案。 ContinueAsNew。当 history 积累足够多（>5000 个事件、>1 天，或到达业务上有意义的边界）时，workflow 代码调用 workflow.NewContinueAsNewError(ctx, NextRun, snapshotState)。引擎以 WorkflowExecutionContinuedAsNew关闭当前运行，并以快照作为输入启动新的运行（相同的 workflow_id，新的 run_id）。新运行从单个事件开始。

订阅 workflow ContinueAsNew在每个计费周期执行一次。长期 ML 训练流水线在每个 epoch 继续执行。视频转码 workflow 在每个输出质量层级继续执行。模式相同，领域不同。

Cadence 和 Temporal 都会 拒绝 history 超过 10MB/50k——workflow 会显式失败，强制进行设计修复。该上限是有意为之的；它能防止意外的无限 workflow。

§7.3 长期运行部署中的 Workflow 版本控制

设置。 一个 审批 workflow 等待人工处理 30 天。你发布了重新调整步骤结构的 v2。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 直接重新部署，寄希望于一切正常。

失败。 v1 的 workflow 无法在 v2 下重放——不匹配风暴爆发。

真正的解决方案。 两个层次。 Patch （基于 GetVersion的现代 API）用于小范围的针对性修改。 Task-queue 版本控制 / BuildID 兼容性 用于深度重构：为 worker 二进制文件声明一个 BuildID ；matching 将任务路由到运行与启动该 workflow 的 BuildID 兼容的 worker。旧 workflow 在旧二进制上逐步耗尽；新 workflow 在新二进制上运行。

对于不兼容的 v1 → v2 迁移：保持 v1 worker 存活直到运行中的 workflow 耗尽，或以 v2 格式的输入终止并重启。没有免费的午餐。

不同引擎的表现形式各异：Step Functions 有状态机版本（别名固定执行）。Airflow 有 DAG 文件替换（每次 DAG 运行固定到其 DAG 版本）。Argo 在提交时将每个 WorkflowTemplate 执行固定到该规范。

§7.4 Activity 超时与心跳

设置。 一个 长时间运行的 ML 训练 activity ——在 GPU 上运行 4 小时。有时确实需要 4 小时；有时 GPU 会挂起。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： StartToClose = 6 hours.

失败。 两个截然不同的问题。(1) 挂起的 activity 会占用 worker 槽位 6 小时。(2) Worker 在 activity 执行中途宕机，引擎 6 小时后才能感知。

真正的解决方案。 心跳机制。长时间运行的 activity 定期执行 activity.RecordHeartbeat(ctx, progress)。如果在 HeartbeatTimeout （例如 60 秒）内没有心跳，该 activity 会被快速判定为失败并在另一个 worker 上重试。记录的进度让新的尝试可以从中断处恢复。

可全文搜索（使用文本分析器）， StartToClose = 6 hours （合理的宽松）和 HeartbeatTimeout = 60s （快速检测宕机）。四参数问题：

超时类型 边界（Bounds） 典型值
ScheduleToStart 任务入队到被接取的等待时间 10 秒至 1 分钟
StartToClose 单次尝试持续时间 分钟到小时
ScheduleToClose 含重试的总时间 通常不设置
HeartbeatTimeout 两次心跳之间的间隔 15-60 秒

最常见的生产问题： StartToClose 设置很大（activity 有时 很慢）但没有设置 HeartbeatTimeout （worker 宕机需数小时才能被检测到）。在 Airflow、Conductor、Step Functions 中问题完全相同。

§7.5 Activity 幂等性

设置。 ChargePayment 调用提供商。提供商返回 200 OK。在 worker 的响应到达引擎之前，网络中断。引擎超时并重试。银行卡被扣款两次。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： maximum_attempts = 1。不重试。

失败。 任何瞬时故障都会导致整个 workflow 失败。这是不可接受的。

真正的解决方案。 应用层幂等性键。activity 将一个稳定的键（workflow_id + activity_id）传递给外部 API。设计良好的 API 会进行去重——第二次请求返回相同的响应，而不会重复扣款。

引擎保证 activity_id 在重试之间保持稳定（在 ActivityTaskScheduled时分配，永不改变）。引擎不保证恰好执行一次；它保证 标识稳定 ，以便应用层可以在副作用边界处实现恰好执行一次语义。对于不支持幂等性的系统：在执行副作用前使用带有 processed_keys 的 INSERT IF NOT EXISTS 表。

适用于每个领域。 邮件营销 workflow 不得重复发送同一封邮件。 部署自动化 应用 Kubernetes manifest 时使用稳定的资源名称（kubectl apply 按名称具有幂等性）。 视频转码 使用确定性的键将输出写入 S3，以便重新运行时覆盖同一文件。

§7.6 Signal、Query 与 Update：同步交互

设置。 一个正在运行的 workflow 需要接收"用户点击了批准"并向监控系统暴露"当前状态"。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 在 workflow 代码中轮询数据库。

失败。 Workflow 代码是事件驱动的，而非定时轮询驱动的。DB 轮询是非确定性的（禁止使用）。通过 DB 读取状态会在引擎之外产生状态副本。

真正的解决方案。 三种原语。

  • 信号：发送给 workflow 的事件。在 history 中持久化为 WorkflowExecutionSignaled 。Workflow 代码阻塞在 workflow.GetSignalChannel("approve").Receive(ctx, &payload)上。持久、按 workflow 排序、重放安全。代价：每个 signal 都会增大 history。
  • 查询：同步读取内存中的状态。 不持久化。引擎在 sticky worker 上调用只读处理器。开销低，可频繁使用。
  • Update （较新的 Temporal 特性）：同步写入并响应。引擎写入事件，运行 workflow 逻辑，一次性返回结果。类似 RPC。

人工审批 workflow：使用 signal。 部署编排：使用 query 查询状态。 长数据流水线：使用 signal 取消。"Signal 风暴"陷阱——意外地每秒向同一 workflow 发送 1000 次 signal——是已知的生产问题。在 Conductor（事件 vs 状态查询）和 Step Functions（sendTaskHeartbeat vs DescribeExecution）中存在相同问题。

§7.7 Signal、Query、Update——三种外部交互原语详解

这三者对于持久执行引擎而言至关重要，值得进行独立深入的介绍。每种原语回答的是 外部世界 与正在运行的 workflow 进行交互的不同方式。

Signal。 即发即忘。客户端调用 SignalWorkflow(workflow_id, signal_name, payload)。Frontend 哈希到分片所有者，后者将 WorkflowExecutionSignaled 追加到 history 并将 decision task 入队。调用方在事件持久化写入后立即返回—— 非 workflow 何时做出反应。在 workflow 代码内部，协程在上等待 workflow.GetSignalChannel(name).Receive(ctx, &payload)。天然具备重放安全性：信号即事件，重放时按顺序读回。

成本模型：每个信号对历史存储进行一次法定写入，外加一个决策任务。单个工作流每秒 1k 信号的信号风暴会使一个分片饱和。实际每工作流每秒信号上限约为 100–1000，取决于持久化后端。乘以工作流数量即得集群压力。

使用场景：人工审批/拒绝（approve, reject），外部状态变更（order_cancelled, payment_received），影响流程的传入事件（司机上线、欺诈告警到达）。「每个逻辑事件对应一个信号名」的约定使工作流易于阅读； signalled_data channel 成为有类型的处理器。

查询。 同步读取工作流状态。客户端调用 QueryWorkflow(workflow_id, query_name, args)。前端将请求路由到 sticky worker（若无 sticky 归属者则路由到任意 worker；该 worker 先重放再响应）。worker 调用工作流代码中注册的只读处理器，返回基于当前状态派生的值。 关键在于：不会修改历史。 查询对重放不可见——它不出现在事件日志中，也无法改变工作流的执行路径。

让人头疼的约束：查询处理器必须是确定性的且只读的。若在查询处理器中修改工作流本地状态，不同 worker 重放时将看到两种分歧状态，下一个决策任务可能非确定性地失败。Stripe 内部有个教训：某个查询处理器在首次调用时惰性计算并缓存了一个值，之后直接返回缓存。在另一台 worker 上缓存尚未填充，行为出现分歧。修复方案：查询必须是当前状态的纯函数。

使用场景：显示「该工作流处于哪个步骤」的仪表盘、显示在途库存的监控系统、调试「为何卡住」。

Update （Temporal 1.21+，2023 年 5 月起）。带响应的同步变更。客户端调用 UpdateWorkflow(workflow_id, update_name, args) 并阻塞，直至工作流返回结果。引擎内部写入 WorkflowExecutionUpdateAccepted，在工作流中执行更新处理器（可调用 activity、等待定时器等），完成后写入 WorkflowExecutionUpdateCompleted 携带结果。引擎通过开放的 RPC 将结果流式返回。

在 Update 之前，实现同步变更并获取响应的唯一方式是「signal + query 轮询」模式：发送变更状态的信号，然后轮询查询直到状态反映该变更。丑陋且易出现竞争。Update 将其压缩为一次往返。Validator（在更新落入历史之前可同步拒绝的检查）提供廉价的准入门控——无需增长历史即可拒绝明显无效的更新。

使用场景：客户端希望 placeBid → accept/reject的竞价工作流； requestPromotion → return promoted_revision的部署工作流； applyDiscount → return new_total的订阅工作流。凡是 RPC 语义优于即发即弃的场景。

§7.8 工作流取消语义

取消是附加了额外仪式的信号。用户调用 CancelWorkflow(workflow_id)。引擎将 WorkflowExecutionCancelRequested 追加到历史并调度一个决策任务。在工作流代码内部，取消信号传播到 ctx: ctx.Done() channel 关闭（Go）， CancellationException （Java）， CancelledError （Python）。工作流代码应当处理取消——通常是执行补偿操作，然后返回。

但现实存在不对称性： 引擎无法取消正在执行中的 activity。 当工作流在某个 activity 执行期间被取消时，引擎无法深入 activity worker 并将其停止。它能做的是：停止调度新 activity，发送一个取消请求，activity 可能 通过心跳响应观察到该请求。

心跳驱动的取消契约。 希望响应取消的 activity 必须定期发送心跳。当 activity 调用 RecordHeartbeat(ctx, progress)时，引擎的响应包含一个标志—— CancelRequested = true 表示工作流已被取消或该 activity 本身已被请求取消。activity 代码必须检查该标志并干净地退出。伪代码：

for chunk := range bigDataset {
    if err := activity.RecordHeartbeat(ctx, chunk.position); err != nil {
        if errors.Is(err, activity.ErrCancelled) {
            return cleanup(chunk.position)
        }
        return err
    }
    processChunk(chunk)
}

忽略该标志的 activity——没有心跳检查的长时间 CPU 计算、未使用感知 context 客户端的阻塞 I/O——无论取消与否都会运行至完成。「我们已取消但 activity 又运行了 10 分钟」的现实是结构性的，并非 bug。常见于 ML 训练 activity（4 小时 GPU 运行，步骤间不检查）、大文件拷贝（不感知 context 的单次 io.Copy 调用），以及不感知 context 的外部 API 调用。

取消不等于终止。 TerminateWorkflow 是更强硬的手段：立即写入 WorkflowExecutionTerminated ，工作流代码没有机会执行补偿。在需要优雅回绕时使用 Cancel；当工作流已损坏且只想将其清除时使用 Terminate（例如因非确定性重放而卡住的工作流）。

取消作用域。 现代 SDK 提供 CancellationScope ——将一段工作流代码包装起来，使取消仅触发该块的补偿，而非整个工作流。Saga 补偿（§7.9）大量使用此机制：每个正向步骤都包裹在一个作用域内，其取消处理器执行对应的补偿。

操作层面的现实。 若工作流因某个忽略取消的长时间 activity 而卡住，有两个选项：等待 StartToClose 超时（可能需要数小时），或 TerminateWorkflow 并放弃补偿。一些团队采用规定：超过 60 秒的每个 activity 必须发送心跳。通过 lint 检查或 activity wrapper 强制执行。

§7.9 Saga 与补偿详解

经典的多步骤业务事务——因跨越服务、历时数小时/数天以及涉及外部系统而无法使用数据库事务——采用 saga 模式：正向步骤 + 在失败时逆序执行的补偿步骤。

结构。 一个预订工作流：

1. ChargeCard(user, $100)         → compensation: RefundCard(user, $100)
2. ReserveInventory(item, qty=1)  → compensation: ReleaseInventory(item, qty=1)
3. AssignWarehouseSlot(slot=42)   → compensation: ReleaseWarehouseSlot(slot=42)
4. NotifyCustomer(email)          → compensation: SendCancellationEmail(email)

若步骤 3 失败，按逆序补偿 2 再补偿 1。若步骤 4 失败，按逆序补偿 3、2、1。Temporal Go SDK 中的模式如下：

compensations := []func() error{}

if err := workflow.ExecuteActivity(ctx, ChargeCard, ...).Get(ctx, nil); err != nil {
    return err
}
compensations = append(compensations, func() error {
    return workflow.ExecuteActivity(disconnectedCtx, RefundCard, ...).Get(ctx, nil)
})

if err := workflow.ExecuteActivity(ctx, ReserveInventory, ...).Get(ctx, &reservationID); err != nil {
    return runCompensations(compensations)
}
compensations = append(compensations, func() error {
    return workflow.ExecuteActivity(disconnectedCtx, ReleaseInventory, reservationID).Get(ctx, nil)
})
// ... and so on

runCompensations 按逆序弹出并依次执行。 disconnectedCtx 至关重要：这是一个在工作流取消后仍存活的 context，因此即使整个工作流正在被取消，补偿操作仍会执行。

补偿幂等性要求。 每个补偿 activity 必须是幂等的。原因：补偿本身可能失败并重试。对同一笔费用退款两次必须是幂等操作（支付提供商返回「已退款」）；两次释放同一库存项必须是安全的。§7.5 中相同的 workflow_id + step_id 幂等键模式同样适用。

部分补偿的现实。 真实的资金流存在不对称性。扣款成功，发货失败：补偿运行 RefundCard。退款可能需要 5–10 个工作日才能出现在客户账单上，尽管 activity 立即返回（「退款已发起」）。客户看到扣款，联系客服，被告知「等待退款」。工作流的职责是为客服发出可审计的事件；无法让缓慢的外部系统变快。

补偿也可能失败。 更深层的恢复层。 RefundCard 本身失败：支付提供商连续 6 小时返回 500。选项：

  1. 永久重试 ——工作流无限期停留在补偿阶段，告警通知值班人员。
  2. 人工干预队列 ——在重试 N 次后升级为人工审批工作流：「补偿卡住，退款 $100 给用户 X，charge_id=ch_xyz，请手动处理。」人工在带外执行退款，发送信号标记补偿完成。
  3. 接受不对称性 ——记录一个 compensation_failed 事件，将工作流以「失败且未退款」状态完成，为财务运营团队生成工单。

大多数生产级 Saga 结合使用：带退避的重试持续 24 小时，然后升级为人工处理。审计跟踪（每次尝试均记录在工作流历史中）才是关键——财务可以证明发生了什么。

编排式 vs 协调式 Saga。 协调式：每个服务发出「我已完成我的部分」事件，下游服务监听并响应。无中央协调者。补偿要求每个服务监听「回滚」事件。在规模化时较脆弱（无全局 Saga 状态视图）。编排式（Temporal 模式）：一个工作流拥有步骤和补偿顺序。集中化、可审计、可调试。业界已对非平凡 Saga 收敛到编排式。

Saga vs 分布式事务。 跨服务的两阶段提交（2PC）也能解决这个问题，但代价是同步协调、跨服务持有锁以及全局阻塞阶段。Saga 以原子性换取可用性：系统在各步骤之间从不保持原子一致性，但最终会收敛到「全部完成」或「全部补偿」。对于 99% 的跨服务业务工作流，这种权衡是正确的。

§7.10 生产规模下的工作流版本控制——最难的问题

§7.1 介绍了 GetVersion。生产规模下的完整图景是一个长达一年的迁移手册，而非单次 API 调用。

约束。 一个 payment workflow 运行 30 天（等待 ACH 结算和人工审核）。在此期间，团队部署了 14 次。每次部署都会更改工作流代码。在当前代码下重放 30 天前的工作流，必须仍能产生与昨天相同的决策。乘以数千万个正在运行的工作流，每个都冻结在不同的代码版本。

生产中使用的三种策略。

1. GetVersion(versionId, minSupported, maxSupported) ——原地兼容。 最常见的模式。对于每处行为变更，用 GetVersion 块包裹。引擎在首次执行时记录带版本的 MarkerRecorded 事件；重放时返回相同版本。旧工作流看到 DefaultVersion，新工作流看到版本 1。

v := workflow.GetVersion(ctx, "v2-fraud-check", workflow.DefaultVersion, 2)
switch v {
case workflow.DefaultVersion:
    // original behavior, pre-fraud-check
case 1:
    // first fraud-check version
    workflow.ExecuteActivity(ctx, CheckFraudV1, ...).Get(ctx, nil)
case 2:
    // refined fraud-check with merchant heuristic
    workflow.ExecuteActivity(ctx, CheckFraudV2, ...).Get(ctx, nil)
}

minSupported 允许垃圾回收：当不存在版本 1 之前的工作流时，改为 GetVersion(ctx, "v2-fraud-check", 1, 2)。引擎拒绝重放低于 minSupported的版本，强制终止这些工作流。整洁性很重要——工作流会随着多年演化积累 GetVersion 调用，变得难以阅读。

2. 「patch」模式——显式条件逻辑。 有时部署引入了之前不存在的新检查。旧工作流跳过它；新工作流执行它。现代 Temporal SDK 提供 Patch(ctx, patchId) 。 DeprecatePatch(ctx, patchId) 作为 GetVersion. Patch 之上更简洁的别名。 DeprecatePatch 如果工作流在部署该 patch 后启动则返回 true，否则返回 false。

无条件返回 true——在旧工作流已耗尽且希望将 patch 标记为可移除时使用。 Patch 生命周期：引入 DeprecatePatch → 等待旧工作流耗尽 → 调用 GetVersion → 在下次部署中完全移除该 patch（假设已等待足够长时间）。这是

标记的 GC 方案。 3. 强制重放失败策略——最后手段。

  • 当代码变更根本不兼容时（例如，重排 activity 顺序、更改输入类型、删除某步骤）。选项： Worker BuildID 版本控制。
  • 为 worker 二进制打上 BuildID 标签。匹配规则将工作流任务路由到与该工作流启动时 BuildID 兼容的 worker。旧工作流在旧二进制上耗尽；新工作流路由到新二进制。旧工作流完成后，停用旧二进制。 终止并重启。
  • 杀死所有运行中的 v1 工作流，将可迁移的状态以 v2 重新启动。粗暴，但有时对低价值、短生命周期工作流而言是正确答案。 迁移工作流。

为每个运行中的 v1 工作流启动一个一次性工作流，读取其状态，发送信号使其优雅停止，然后以该状态启动新的 v2。

具体演练——一个运行 30 天、期间代码部署两次的 payment workflow。 PaymentWorkflow 第 0 天： order-abc启动，对象为

。代码版本为 v1。事件 1 写入。 Patch(ctx, "add-compliance-step") 工作流已推进至 7 步中的第 1–3 步。工程师发布 v2，在步骤 3 和 4 之间添加一个步骤（合规检查）。v2 用

if workflow.Patch(ctx, "add-compliance-step") {
    workflow.ExecuteActivity(ctx, ComplianceCheck, ...).Get(ctx, nil)
}

到达步骤 5 时将使用 v3 的选项。 order-abc 到达步骤 5（结算后对账）。在 v3 下重放——重试策略不同，但 activity 尚未开始，因此无重放冲突。决策完成，activity 以 v3 的选项调度。历史记录此事件。

ACH 结算完成。工作流发出信号推进至步骤 6（通知），然后步骤 7（关闭）。在 v3 下重放全部 25 天历史有效，原因如下：

  • 步骤 1–3 仍执行相同代码（无相关 GetVersion/Patch 调用）。
  • 步骤 4： Patch("add-compliance-step") 返回 false （无标记），跳过。
  • v3 的重试策略生效（activity 选项在调度时确定）。
  • 步骤 6–7：相同代码，无版本分支。

工作流完成。工程师发布 v4，弃用 add-compliance-step （所有可能命中该分支的工作流现已全部完成）。两次部署后， Patch 调用从代码库中完全移除。

长时间运行工作流的 18 个月迁移。 某些工作流长达 数年 。客户订阅、监管冻结、多年质保。将这些从 WorkflowV1 与 WorkflowV2 （结构性重写）迁移需要 18 个月，因为必须同时部署两种工作流类型，并让 v1 工作流自然耗尽。操作层面：

  1. 第 0 个月： 将 v2 与 v1 并行部署。新工作流以 v2 启动。
  2. 第 6 个月： 第 -1 个月启动的所有「年级」工作流已耗尽。v1 worker 池可以缩容。
  3. 第 12 个月： 所有「18 个月」工作流耗尽。最后一个 v1 二进制停用。
  4. 第 18 个月： 清理 v2 代码库中已死亡的 GetVersion 分支。

痛苦。不可避免。

Step Functions、Airflow、Argo 各有其版本控制方案。 Step Functions：状态机版本 + 别名将执行固定到某个版本。Airflow：DAG 文件替换是不受控的——每次 DAG 运行都固定到其启动时的 DAG 版本（在较新版本的 Airflow 中通过序列化 DAG 快照实现）。Argo： WorkflowTemplate 在提交时快照，因此已提交的工作流不受后续模板编辑影响。其形态——将运行中的执行固定到启动它的代码，自由演化新执行——是普遍的。

§7.11 工作流确定性——禁止的内容及其原因

§4f 给出了约束。完整图景：每个工作流 SDK 都提供一个确定性沙箱，约束工作流代码可以做什么。违规要么在运行时响亮地失败（被捕获），要么静默损坏重放（未捕获——危险类别）。

被捕获的违规（SDK 在注册/运行时拒绝）。

违规 重放失败原因 SDK 强制方式
time.Now() / Date.now() 墙上时钟在各次重放间不同 在 TypeScript/Python 中沙箱化；Go 在运行时 panic
rand.Float64() 重放时生成新随机数 同上
os.Getenv / 文件读取 worker 环境各异 静态分析器警告；运行时 panic
Goroutine / 线程 / 原生 async 非确定性调度 替换为 workflow.Go, workflow.NewFuture
Direct time.Sleep 不在重放中存活；阻塞工作流任务 模拟程序启动时会进行的重定基址操作。 workflow.Sleep
http.Get / db.Query direct 副作用未记录在历史中 必须通过 activity 处理

未被捕获的违规（重放中的细微 bug）。

这些才是灾难。代码在首次执行时运行正常，但重放时不匹配。

  • 以非确定性顺序迭代 map。 Go 的 map 以及 Python 的 dict （3.7 以前版本）以随机顺序迭代。如果 workflow 代码执行 for k, v := range myMap { workflow.ExecuteActivity(ctx, A, k) }，则各次执行中 activity 的调度顺序不同。Replay 会看到不同的调度顺序，从而产生不匹配。 修复方法： 在迭代前显式排序键。
  • 使用指针比较进行排序。 一个真实的灾难： sort.Slice(items, func(i, j int) bool { return items[i].ptr < items[j].ptr })。指针地址在不同进程间各不相同，排序结果是不确定的。"我们新增了一个基于指针比较的排序"这一灾难案例是经典教训——发现时 workflow 已陷入 NonDeterministicWorkflowError ，而这正是某次引入该排序的部署之后发生的。
  • 跨架构的浮点数运算。 罕见但真实存在：x86 与 ARM worker 可能产生略有差异的浮点数。workflow 代码中基于浮点数比较的分支可能发生分叉。修复方式：在 workflow 代码中使用整数运算，浮点运算放到 activity 中执行。
  • 系统调用返回值。 os.Hostname(), os.Pid() ——在不同 worker 上各不相同。在另一个 worker 上进行 replay 时会发生分叉。
  • 依赖 locale 的字符串排序。 sort.Strings(strs) 在某些平台上使用操作系统 locale。不同 worker 之间的 locale 差异会导致排序结果不一致。
  • 未使用 workflow API 的异步定时器。 启动一个 goroutine，其中调用 time.Sleep 然后修改 workflow 状态：该 goroutine 仅在首次执行时运行，replay 不会重新创建它，从而导致状态分叉。

各 SDK 中的确定性沙箱。

  • Go SDK。 workflow 代码运行在自定义协程调度器（workflow.Go 而非 go）中，时间访问通过 workflow.Now进行。SDK 通过猴子补丁捕获大多数违规行为；未捕获的违规会在下次 replay 时以 NonDeterministicWorkflowError 的形式暴露出来。
  • TypeScript SDK。 在 v8 隔离环境（独立的 JavaScript 上下文）中运行 workflow，并对 globalThis 进行了补丁：禁用 Date.now、没有 Math.random、没有 fetch。这是三者中沙箱隔离最强的。
  • Python SDK。 使用 asyncio 配合自定义事件循环来驱动 workflow 任务； time.time() 及相关函数已被补丁处理。隔离程度不及 v8 隔离环境严格。
  • Java SDK。 workflow 代码运行在 Workflow Thread（基于标准 Java 线程的自定义调度器）中；线程本地上下文跟踪 workflow 身份。禁用的 API（如 Thread.sleep）在注册时通过字节码检查进行检测。

为何要求确定性？ 当时有两种设计选择："每步之后对状态进行检查点"（Cadence 的路径：未采用）与"在事件日志上重放代码"（采用的路径：是）。检查点方式需要序列化所有内存状态——workflow 局部变量、channel 缓冲区、延迟工作。对于具有大量状态的长生命周期 workflow，这将是极大的开销。基于事件的 replay 以 CPU（重跑代码）换存储（无需状态快照），用廉价 CPU 换昂贵存储的赌注在过去十年中被证明是正确的。

为何要高声报错？ NonDeterministicWorkflowError 刻意设计为高声报错。任何静默损坏——workflow 走了不同于原有的路径——都是资金或安全方面的 bug。不如让 workflow 暂停，等待人工介入或打GetVersion 补丁，也好过默默将客户送往错误的配送中心。

§7.12 副作用与本地 activity

两种看起来像 activity 但并非完整 activity 的模式，它们的存在恰恰是因为确定性约束有时过于严格。

SideEffect。 "我只需要一次随机数。"一个 workflow 需要选取随机分区分配，或为下游生成 UUID，不能直接调用 rand.Float64() （非确定性）。封装为完整 activity 代价高昂（需经过 matching 的往返、独立的 worker、schedule + start + complete 的历史事件）。 workflow.SideEffect(ctx, func(ctx workflow.Context) interface{} { return rand.Float64() }) 是轻量级路径：

  1. 首次执行：函数在 workflow worker 中运行，结果被捕获，引擎写入包含该值的 MarkerRecorded 事件。
  2. Replay：workflow 到达 SideEffect。引擎读取 marker，返回已存储的值。函数 非 被重新执行。

与 activity 的三个事件相比，历史中只有一个事件，且保留在 workflow worker 进程内。使用场景：生成 UUID、选取随机 A/B 测试分组、在确定性节点读取 feature flag。

注意事项：该函数应快速且无副作用（不要在 SideEffect中发起 HTTP 调用；若 marker 丢失，则无法重试）。结果必须可序列化。若函数发生 panic，workflow 失败（不会重试）。

MutableSideEffect。 变体：每次调用时重新评估函数，但仅在结果发生变化时才写入新 marker。适用于"读取缓慢变化的配置值，仅在实际变化时才更新"。大多数团队直接使用普通 SideEffect 并在 workflow 状态中显式缓存。

Local Activity。 "我需要调用 HTTP 端点，但不想支付 matching 往返的开销。"普通 activity 的流程如下：

workflow worker → frontend → history → matching → activity worker → execute → respond → frontend → history → workflow worker

本地 activity 跳过 matching 调度：

workflow worker → execute in same process → respond → workflow worker

结果仍以 MarkerRecorded类型事件写入历史（确保 replay 正确），但调度延迟从约 10ms 降至亚毫秒级。使用场景：快速的本地计算，对于 SideEffect 来说过于复杂（因为可能失败并受益于重试），但又太轻量不值得走完整的 activity 机制。

权衡对比：

Activity Local Activity SideEffect
运行在独立 worker 上 是 否（与 workflow 相同） 否
记录在历史中 Schedule、Start、Complete 单个 marker 单个 marker
通过策略自动重试 是 有限（受 workflow task 预算约束） 无（无重试语义）
Latency（延迟） 约 10ms 以上的调度延迟 亚毫秒级 亚毫秒级

本地 activity 的危险之处：人们往往因其速度快而滥用于所有场景，结果导致某个 workflow task 中包含一个耗时 20 秒的本地 activity，超过了 workflow task 超时，整个决策因此失败。惯例：本地 activity 应在秒级以内，最多几秒钟。

§8. 故障模式

Worker 在 activity 执行中途崩溃。 Activity worker 在 ActivityTaskStarted 。 RespondActivityTaskCompleted之间宕机。若启用了心跳检测： HeartbeatTimeout 触发 → 引擎发出 ActivityTaskTimedOut → 重试策略调度新的尝试。若未启用： StartToClose 触发，结果相同，只是较慢。副作用可能已执行一次——应用层幂等键可使重试成为空操作。 持久性要点: ActivityTaskStarted 已以仲裁方式提交。

History service 分片丢失。 拥有分片 1700-1799 的节点崩溃，成员协议检测到后进行重新分配。新的所有者从持久化存储重新加载 executions 行。受影响的分片会有约 30 秒的暂停，但不会丢失数据，因为每个事件均经过仲裁写入。持久性依赖于 RF=3 加仲裁写入——1 个副本丢失可承受；同一分区的 2 个副本同时丢失是唯一会损失持久性的故障。

代码变更后确定性 replay 失败。 详见 §7.1。恢复方式：将 workflow 暂停至 NonDeterministicWorkflowError，部署 GetVersion 补丁，然后恢复。症状剧烈且立即显现——优于静默损坏。

Activity 永久卡住。 Worker 存活但陷入循环。无论是否发送心跳，最终 StartToClose 都会触发。操作覆盖方式：通过 UI 终止 workflow，这会写入 WorkflowExecutionTerminated.

信号丢失或重复。 客户端重试 SignalWorkflow；信号被持久化了两次。引擎默认不进行去重。在 payload 中包含确定性的信号键并在 workflow 代码中去重，或使用 SignalWithStart （在 workflow_id 上幂等）。

持久化存储仲裁丢失。 三个副本中有两个宕机：写入失败，workflow 进度停止； StartWorkflow 返回错误。由于没有任何数据被确认，不会丢失数据。恢复方式：恢复副本，workflow 随即继续。

Matching service 分区丢失。 持有 1000 个分区的 matching 节点崩溃。任务不会丢失——它们作为 transfer-queue 或 timer-queue 行存在于持久化存储中。新的 matching 节点在重新分配后加载这些任务。约 10-30 秒的短暂中断，不会有工作丢失。

客户端与 frontend 之间的网络分区。 客户端重试 StartWorkflow。通过 workflow_id 实现幂等。信号需要在应用层进行去重。

§9. 为何不用 cron + 状态表？

工程师们第一反应通常会想到的简单替代方案：

CREATE TABLE workflows (
  id VARCHAR(64) PRIMARY KEY, current_step VARCHAR(64),
  state_blob BLOB, last_updated TIMESTAMP
);

Cron 扫描 WHERE current_step != 'DONE' AND last_updated < NOW() - INTERVAL 1 MINUTE 并重新执行每行的当前步骤。

这种方式在十个 workflow 时还能用，到一百个时就会崩溃。有八种故障模式，每一种都需要你重新发明引擎的某个组件：

  1. 无重试语义。 charge_payment 遇到网络错误失败，状态仍然是 START。Cron 再次调用。第一次尝试是否已生效？除非每次调用都携带幂等键，否则会出现双重扣款——这需要你为每个步骤自行构建。
  2. 无补偿机制。 reserve_inventory 在 charge_payment 成功后失败。谁来退款？需要手写 rollback_to(state) 覆盖每个分支。
  3. 无可见性。 Workflow 在 RESERVED 处卡了 6 小时。 notify_warehouse 是否静默失败了？你需要编写自定义仪表板。
  4. 状态更新上的竞争。 两个 cron worker 选中同一行，都调用 notify_warehouse。你需要添加 version 列并使用条件 UPDATE——在拙劣地重新发明乐观并发。
  5. 状态漂移。 操作员更新 current_step 以"解卡"一个 workflow。 state_blob 状态不一致，workflow 沿错误分支继续推进，资金凭空消失。
  6. 无 replay 能力。 发现了一个 bug。"如果用修复后的代码会发生什么？"状态表只有当前状态，没有执行路径记录。
  7. 无法支持长时间等待。 "等待 30 天等待人工审批"——需要构建一个带有 scheduled_tasks 表和轮询器的调度器，在拙劣地重新发明定时器。
  8. 无法扩展。 每分钟对 1000 万行进行 cron 全表扫描，是完整的表扫描。为其建立索引；索引维护会降低写入速度。按 id mod N分片；在拙劣地重新发明分片感知引擎。

这正是 Uber 构建 Cadence 的原因：数十个内部团队各自搭建了基于状态表的 workflow 系统，都在遭受上述问题的不同子集。一旦你拥有超过少数几个复杂 workflow，采用 workflow 引擎的工程投资回报率将是压倒性的。

§9.1 从 cron + 状态表的经典迁移路径

"我们已经有一个 cron 驱动的状态表，如何迁移到 Temporal"是 workflow 引擎领域最常走的迁移路径之一。遗留系统的形态如下：

workflows table (status, current_step, state_blob, last_updated)
   ▲
   │  UPDATE current_step, state_blob WHERE id = ...
   │
cron jobs (every 1 minute)
   ├─ ProcessChargingWorkflows: SELECT * WHERE current_step = 'charging'
   ├─ ProcessShippingWorkflows: SELECT * WHERE current_step = 'shipping'
   └─ ProcessExpiredWorkflows:  SELECT * WHERE last_updated < NOW() - 1h

每个 cron 处理器运行相关步骤并更新对应行。迁移通常分为五个阶段。

第一阶段：以影子模式启动 Temporal。 部署集群，构建镜像 cron 逻辑的 v2 workflow，暂不将任何流量路由至其上。进行操作性验证——worker 能够启动，信号正常流转，可观测性工作正常。

第二阶段：影子 workflow 对比。 在每个 v1 行旁边启动一个 v2 workflow，但 v2 仅作 仅供参考 ——记录它会执行什么操作，不执行实际副作用。构建工具在每个时间节点对比 v1 与 v2 的状态，并在发生分叉时告警。运行 2-4 周，直到分叉率低于 1% 且原因已明确。

第三阶段：带协调的双轨运行。 v1 和 v2 都执行副作用，但幂等键的作用域绑定到 workflow ID。两者都尝试扣款，都尝试发货。下游服务根据幂等键进行去重——实际只有第一个请求生效。记录竞争的获胜方。构建仪表板：v2 优先的 workflow 占比、v1 优先的占比、不匹配的占比。双轨运行阶段能暴露出细微的 bug（时序差异、竞争顺序）。

第四阶段：逐步切换。 将 1% 的新 workflow 仅路由至 v2。监控一周，然后提升至 10%、50%、90%、100%。正在进行中的 v1 workflow 在 cron 路径上继续执行至完成；新 workflow 走 v2。

第五阶段：清空并下线。 等待 v1 workflow 全部清空。下线 cron 任务和 workflows 表。将状态快照历史归档以满足合规要求。

在迁移中留存的模式。

  • 幂等键约定。 如果你已经进行了双轨运行，那么每个下游调用上都已有稳定的键。将其带入 v2。
  • 状态快照 → workflow 输入映射。 遗留状态快照是 workflow 所需数据的事实来源。将相关字段提升为 workflow 输入。
  • "卡住的 workflow"仪表板。 遗留系统通常有一个仪表板用于显示"last_updated > 1h 的 workflow"。将该查询移植到 Temporal 的 visibility API："没有最近 decision task 的开放 workflow"。对运维人员来说很熟悉。
  • 补偿表。 如果遗留系统有一张跟踪退款和回滚的 compensations 表，那正好对应 saga 的补偿列表。将其作为栈结构移入 workflow 代码中。

在迁移中无法留存的模式。

  • "通过 UPDATE 某行来修复"的操作手册。 操作员习惯于通过修改数据库中的 current_step 来解卡 workflow。Temporal 中的等价操作是 ResetWorkflow （回退到较早的事件）——这是不同的思维模型。需要对运维人员进行新工具的培训。
  • 轮询状态。 轮询 workflows 表以判断"是否完成"的代码，必须改为使用 DescribeWorkflowExecution 或 query handler。
  • 批量 SELECT 并更新的策略。 "找出所有处于步骤 X 的 workflow 并推进它们"——Temporal 期望每个 workflow 自我驱动。不要尝试重建 cron 扫描；应通过信号驱动状态转换。

最终状态：更简洁的代码、更清晰的运维、真正的审计追踪。代价：需要运维一个引擎，团队需要熟悉它。在拥有约 5 种以上不同的 workflow 类型后值得采用。

§10. 扩展维度

两种从根本上不同的增长维度，需要不同的应对方式。

类型 1：更多 workflow（均匀扩展）。 从 100 万并发增长到 1 亿并发，形态大致相同（中位数 50 个事件，5 分钟时长）。

  • 100 万并发：约 10 个 history 节点、约 5 个 matching 节点、3 个 RF=3 的持久化存储节点、约 50 个 worker。4096 个分片过于充裕（约 250 个 workflow/分片）。
  • 1000 万并发：约 50 个 history 节点、约 20 个 matching 节点、20 个持久化存储节点。每个分片的缓存预热变得重要。
  • 1 亿并发：单区域单集群风险过高（爆炸半径过大）。 拐点：进行联邦化。使用 Temporal global namespace 配合异步跨区域复制，每个集群处理一个命名空间子集。

同时按 业务域拆分命名空间：订单、风控、支付、内容审核——每个域拥有独立命名空间，并设置配额和隔离。风控的 bug 不会拖垮订单系统。

持久化存储扩展：线性增长。1 亿 × 100 个事件 × 1KB ≈ 10TB 热数据。RF=3 部署在 100 个节点上，每节点约 300GB，在 NVMe 上完全可承受。

类型 2：长生命周期 workflow（热点强化）。 数量相同，但每个都是"庞然大物"。

  • 50 个事件：重放成本可忽略。Sticky 缓存命中率 95%，CPU 基本空闲。
  • 5,000 个事件：重放成本真实存在。冷重启每次决策约消耗 100ms CPU。缓存淘汰影响明显。修复方案：积极调优 sticky 参数 + ContinueAsNew 在有意义的边界处
  • 50,000 个事件：引擎硬性限制。工作流在结构上存在问题，需重新设计。 ContinueAsNew 强制要求。

嘈杂工作流反模式。 一个工作流每秒向自身发送 1000 次信号，会成为单个分片的热点：某个持久化存储分区被持续高压访问。同一分片上的其他工作流会遭受写放大。修复方案：重新设计（更频繁地 ContinueAsNew，或拆分），或将其迁移到独立分片。 单工作流热点无法通过重新分片解决 ——分片负责管理工作流；若不破坏串行化语义，则无法将一个工作流拆分到两个分片上。必须修改应用程序本身。

拐点。 1M → 10M 并发：增加分片/节点（线性扩展）。10M → 100M：联邦化（拓扑变化）。1k → 50k 事件/工作流： ContinueAsNew 强制要求。单命名空间 → 多命名空间：当某类工作流可能对其他工作流造成大范围影响时。

§12. 定时调度工作流

基于 Cron 的定时调度是工作流引擎最枯燥却最被低估的特性。Temporal 提供了 Schedules （原 Cron Schedules）—— 一种一等资源，用于声明"按此 Cron 表达式以这些输入启动该工作流"。

ScheduleSpec:
  cron_expression: "0 */4 * * *"     # every 4 hours
  workflow_type: DailyReconciliation
  task_queue: RECONCILIATION_TQ
  workflow_input: {...}
  catchup_window: 24h
  pause_on_failure: false

Schedule 资源存在于引擎中。引擎在每次触发时启动一次全新的工作流执行。每次执行相互独立（独立的 run_id，独立的历史记录）。

"宕机期间遗漏了 10 次执行，是否需要回填？"的决策。 真实案例。集群宕机了 8 小时。Schedule 每小时触发一次。有 8 个 Tick 未执行。

选项：

  1. 跳过它们。 适用于"尽力而为"的 Schedule —— 例如，每小时运行的缓存刷新。今天的数据稍显陈旧，明天就好了。
  2. 全部回填这 8 次。 按时间顺序依次或并行运行。每个 Tick 对应一个工作流，以"调度时间"作为输入。Activity 使用该时间，而非 workflow.Now。适用于分析汇总：每个 Tick 计算前一小时的数据。
  3. 仅回填最新一次。 适用于"当前状态"型 Schedule，只有最新的结果才重要。例如每日排名刷新 —— 昨天的排名已无意义，只需运行今天的。
  4. 人工分类处理。 有时回填是不安全的（例如面向用户的邮件发送）。禁用 Schedule，逐一审查每个遗漏的 Tick。

catchup_window 是引擎侧的配置项：如果某个 Tick 的应触发时间超过该窗口，则跳过。Temporal 默认值为 1 分钟（保守设置）。对于"始终回填"的 Schedule，可设置为 30 天。

Schedule 嵌套 Schedule（反模式）。 部分团队将一个 Schedule 串联起来，运行一个工作流，该工作流本身包含一个 workflow.Sleep(1h) 循环。不要这样做 —— Schedule 才是"定期触发"的正确原语。在工作流内部 Sleep 会导致历史事件永久积累。

Step Functions、Airflow。 Step Functions 使用 EventBridge Schedules 作为 Cron 机制 —— 形式相同。Airflow 的核心理念就是调度：每个 DAG 都有一个 schedule_interval。Argo 使用 CronWorkflows。该原语在此类系统中是通用的。

§13. 多租户与命名空间

单个工作流集群通常承载来自多个团队的工作负载。若缺乏隔离，一个嘈杂的团队可能会使所有人陷入资源匮乏。 命名空间（Namespaces） 是 Temporal/Cadence 中主要的隔离原语。

命名空间的含义。 一个逻辑工作流空间。每个命名空间包含：

  • 其自身的一套工作流（不存在跨命名空间的 workflow_id 冲突；可以有 order-123 于 payments 。 order-123 于 subscriptions 且它们是不同的工作流）。
  • 其自身的任务队列（Worker 轮询特定命名空间 + 任务队列）。
  • 其自身的保留策略（工作流关闭后历史记录保留多长时间）。
  • 其自身的可见性索引（Elasticsearch 命名空间）。
  • 其自身的配额（启动、信号、查询、持久化写入的速率限制）。
  • 其自身的 ACL（哪些用户/服务可以在此启动/发送信号/查询工作流）。

为何使用命名空间而非独立集群。 集群级别的隔离最强但代价最高。每个集群需要独立的持久化存储、独立的 matching/history/frontend 服务集群、独立的可见性管道。一家拥有 50 个团队的公司若为每个团队部署独立集群，运维成本将是 50 倍。命名空间共享集群，同时以低成本提供逻辑隔离。

按租户配额。 对共享集群至关重要。每个命名空间可配置：

  • MaxWorkflowExecutionsPerSecond —— 启动速率限制。
  • MaxSignalsPerSecond —— 信号速率限制。
  • MaxWorkflowTaskExecutionsPerSecond —— 决策任务限流。
  • MaxActivityExecutionsPerSecond —— Activity 调度限流。
  • RPSPerNamespace —— 粗粒度 RPC 速率。

Frontend 强制执行这些配额；超出配额的请求会被限流（等价于 HTTP 429）。若无配额，某个失控团队以每秒 10k 次的频率向工作流发送信号，可能导致整个集群饱和。

"嘈杂租户阻塞其他人"问题。 即使有配额，热分片仍可能影响邻居。某命名空间运气不佳，其 50 个工作流哈希到同一个分片；该分片的持久化分区成为热点。同一分片上其他命名空间的工作流会遭受写延迟。缓解措施：

  • 按命名空间的分片路由提示 （部分引擎支持将命名空间路由到分片子集）。
  • 持久化再均衡 （Cassandra 修复 + 手动键范围迁移）。
  • 联邦化 —— 将嘈杂的命名空间迁移到独立集群。

租户接入模式。 团队申请命名空间。运维以保守配额进行初始化，监控使用情况 2 周，随后根据情况提升配额，或在该团队的工作流异构时拆分命名空间（例如，同一命名空间中的快速/慢速流程分别设置独立配额将更为有利）。

Step Functions、Argo、Conductor 中的多租户。 Step Functions：账号 + 区域即为隔离边界；通过 IAM 实现子账号隔离，但无原生命名空间。Argo：Kubernetes 命名空间一一对应 —— 天然契合。Conductor：通过工作流名称 + 权限范围实现；不如 Temporal 严格。

跨命名空间协调。 有时，命名空间 A 中的工作流需要在命名空间 B 中启动子工作流。现代版 Temporal 通过跨命名空间子工作流支持此功能；旧版 Cadence 则需要向另一命名空间中的工作流发送信号。跨命名空间会增加延迟和耦合 —— 除非有强烈理由，否则优先采用单命名空间设计。

§14. 历史搜索与可观测性

工作流会产生数十亿个事件。查询这些事件 —— "查找所有在周二到周四之间在第 3 步失败的订单工作流" —— 是一个独立于运行工作流的关注点。

Visibility API。 Temporal 的 Visibility 是搜索/列表接口，以 Elasticsearch（或 OpenSearch）为后端。history 服务发出可见性记录的变更数据捕获流（工作流已启动、已完成、搜索属性已更新）；visibility indexer 消费该流并写入 Elasticsearch。

每个工作流都有自动索引的系统属性（WorkflowId, WorkflowType, RunId, ExecutionStatus, StartTime, CloseTime, TaskQueue）以及按命名空间声明的自定义 search attributes ：

workflow.UpsertSearchAttributes(ctx, map[string]interface{}{
    "CustomerTier":  "Gold",
    "OrderAmount":   299.99,
    "StuckInStep":   "FraudCheck",
})

每条 UpsertSearchAttributes 调用会在历史记录中写入一个标记事件 。 同时更新可见性索引。搜索属性即可被查询。

查询。 Visibility 支持类 SQL 的查询语言：

WorkflowType = "OrderWorkflow" 
  AND ExecutionStatus = "Failed" 
  AND StuckInStep = "FraudCheck"
  AND StartTime BETWEEN "2026-05-15" AND "2026-05-22"
ORDER BY StartTime DESC
LIMIT 100

被广泛用于运维仪表板、批量操作（批量重启遭遇已知 Bug 的工作流）、合规审计、客户支持（"这是我们为客户 X 运行的所有工作流"）。

自定义搜索属性是一种契约。 一旦你用属性 CustomerTier对某种工作流类型建立索引，就必须保持其含义稳定。重命名或改变用途会破坏查询。部分团队对属性名称进行版本化（CustomerTierV2).

重放调试。 事故处理的杀手级特性。当工作流存在 Bug 时，下载其历史记录（一个 ~10-100KB 的二进制文件）并在本地重放：

tctl workflow show --workflow_id order-abc --output_filename history.bin

然后在你的开发环境中：

replayer := worker.NewWorkflowReplayer()
replayer.RegisterWorkflow(OrderWorkflow)
err := replayer.ReplayWorkflowHistoryFromJSONFile(logger, "history.bin")

工作流运行时所有 Activity 均被 Mock（其结果来自历史记录）。你可以设置断点、逐步执行、精确了解代码做出了哪些决策。对于调试卡住的工作流不可或缺："为什么这个工作流走了路径 A 而不是 B？"

实践中的"查找所有在第 3 步失败的支付工作流"查询。 真实案例。一个 Bug 被发布，导致支付工作流在欺诈检测步骤失败。运维查询：

WorkflowType = "PaymentWorkflow" 
  AND ExecutionStatus = "Failed" 
  AND StuckInStep = "FraudCheck"
  AND StartTime > "2026-05-20T00:00:00Z"

返回 4,217 个工作流 ID。然后执行批量操作： tctl workflow batch terminate --query "<above>" 或 tctl workflow batch reset --reset-type LastWorkflowTask --query "<above>"。将所有匹配的工作流重置到失败步骤之前。它们在修复后的代码下重新执行。

这种批量操作模式 —— 查询 + 批量动作 —— 是工作流引擎提供的最高杠杆能力之一。若没有它，"修复 4k 个工作流"意味着团队要花整整一周时间编写临时脚本。

可见性成本。 Elasticsearch 是许多大型集群中成本最高的组件。一个在其生命周期内更新 UpsertSearchAttributes 100 次的工作流会产生 100 次索引更新。乘以 1 亿个工作流 = 100 亿次索引更新。可见性集群的成本可能超过主持久化集群。最佳实践：仅在属性实际发生变化时 Upsert ；避免使用高基数属性。

§15. 成本经济学

引擎成本分为三个维度。

历史存储。 每个事件序列化后约 200B-2KB；以 history_node 行为单位批量存储，每行 5-20 个事件。工作流历史记录通常为 ~5-50KB，长时间运行的工作流可达 ~500KB-1MB。存储成本 = (workflow rate) × (avg history size) × (retention period).

具体示例：每天 1000 万个工作流，平均历史记录 20KB，保留期 30 天。→ 10M × 20KB × 30 = 6 TB 热数据。Cassandra RF=3 → 18 TB 原始数据。以商用 NVMe ~$0.10/GB/月计算 → 仅存储成本约 $1.8k/月。超出保留期后冷归档到 S3 → 每 GB 几美分/月，接近免费。

计算集群。

  • History 服务。 按分片数 × 写入速率进行容量规划。经验法则：每个 history 节点负责 100-500 个分片，处理 5-10k 次写入/秒。4096 个分片 / 每节点 200 个分片 = 中大型集群需 20 个 history 节点。
  • Matching 服务。 按任务队列吞吐量进行容量规划。每秒 10 万个任务约需 30-100 个节点。
  • Frontend。 无状态，随 API QPS 扩展。通常 10-30 个节点。
  • Worker 集群。 随工作流执行速率 × Activity 速率扩展。长轮询 Worker 大部分时间处于空闲等待状态；按峰值并发而非平均负载进行容量规划。"长轮询，空闲成本"现实：等待任务的 Worker 会占用 matching 服务的连接槽及少量 CPU 基线。1000 个 Worker 各持有 10 个长轮询，即持续保持 10k 个开放连接。

Worker 空闲成本真实存在。过度配置 Worker（超出需求 10 倍）的团队正在为 10 倍的空闲长轮询买单。缓解措施：调优每个 Worker 的 MaxConcurrentActivityExecutionSize 。 MaxConcurrentWorkflowTaskExecutionSize ；缩减 Worker 集群规模；使用与任务队列积压绑定的自动扩缩容。

可见性 / Elasticsearch。 通常是意外的成本来源。每个工作流产生 3-10 个可见性事件（启动、状态变更、完成，以及任何 UpsertSearchAttributes）。1 亿个工作流/年 × 5 个事件 × 1KB = 500 GB。Elasticsearch 存储 + 副本 + 冷热分层。成本轻松达到主持久化成本的 2-5 倍。

"工作流速率 × 事件速率 × 保留期"的乘积。 这个乘积是核心指标。示例：

  • 小型集群：1k 个工作流/天 × 20 个事件/工作流 × 30 天保留期 = 600k 个事件存储。微不足道。
  • 中型：1M 个工作流/天 × 30 个事件 × 30 天 = 900M 个事件。适合中型 Cassandra 部署。
  • 大规模：1亿个工作流/天 × 50个事件 × 30天 = 1500亿个事件。联邦集群，多区域，专属运维团队。

降本手段。

  • 缩短保留期。 30天 → 7天。大多数工作流在完成后7天内完成调试。归档至 S3 用于合规，从热存储中删除。
  • ContinueAsNew 更激进地。 一个永久运行且增长至5万个事件的订阅工作流，其存储成本比每月运行、每轮保留100条事件历史的工作流高出百万倍。 ContinueAsNew每月运行，每轮保留100条事件历史记录。
  • 通过以下方式进行快照 ContinueAsNew. 通过关闭当前运行并以紧凑快照作为输入重新开始来截断历史记录。以下快照之前的每个事件 ContinueAsNew 在保留期结束后归档至 S3。这是长期运行工作流的标准模式。
  • 减少 UpsertSearchAttributes 频率。 每次 Upsert 对应一条历史事件和一次索引更新。
  • 合理调整 worker 机群规模。 使用与以下指标绑定的自动扩缩容 WorkflowTaskBacklog / ActivityTaskBacklog 指标。

Temporal Cloud 定价参考。 按 操作 （工作流启动、信号、查询、活动完成、定时器触发等）计费，另加存储费用。官方定价约为首档每百万操作 $25，按量递减。一个包含30个事件的工作流约等于30个操作。每天100万个工作流 = 约每天3000万操作 = 约每月9亿操作 → 按官方价格约每月 $22k。另加存储费、支持费。大客户可大幅议价。与自托管对比：一个20节点的 Cassandra 集群 + 50节点引擎机群 + 运维人员，在发布任何功能之前每月通常就超过 $50k+。Cloud 能较快回本；自托管在超大规模下才划算。

§16. 工作流与活动的幂等性

引擎内部存在两种截然不同的幂等性，各自有不同的执行点。

工作流启动幂等性。 内置支持。 StartWorkflow(workflow_id="order-123") 在以下情况下具有幂等性 workflow_id。引擎通过分片互斥锁维护“任意时刻每个 workflow_id 最多只有一个运行中工作流”的不变性。以相同的 StartWorkflow 两次 workflow_id:

  • 如果第一个仍在运行：第二个返回现有运行的句柄（或 WorkflowExecutionAlreadyStartedError，取决于 WorkflowIDReusePolicy).
  • 如果第一个已完成：行为取决于 WorkflowIDReusePolicy:
  • AllowDuplicate：使用相同 ID 启动新运行，生成新的 run_id。
  • AllowDuplicateFailedOnly：仅在上一次运行失败时允许。
  • RejectDuplicate：拒绝。
  • TerminateIfRunning：取消正在进行的运行并重新开始。

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 workflow_id 是你的业务域幂等键。常见约定： workflow_id = "order-{order_id}" 确保每个订单永久只有一个工作流。

活动幂等性。 非内置。引擎保证：

  • activity_id 在重试间保持稳定（同一逻辑活动调用，可能有多次物理尝试）。
  • attempt 每次重试时递增。

引擎不保证：

  • 副作用最多执行一次。（重试可能会重新执行已完成但未确认的副作用。）
  • 应用层键的构造是否正确。

应用层的职责：从 activity_id + attempt （或仅用 activity_id ，若希望所有重试都去重到同一外部操作）构造幂等键，并将其传递给外部系统。

func ChargePayment(ctx context.Context, input PaymentInput) (string, error) {
    activityInfo := activity.GetInfo(ctx)
    idempotencyKey := fmt.Sprintf("%s-%s", activityInfo.WorkflowExecution.ID, activityInfo.ActivityID)
    return stripeClient.Charge(input, stripe.WithIdempotencyKey(idempotencyKey))
}

如果 Stripe 收到两个相同键的请求，它会返回第一个请求的缓存结果——副作用只发生一次。没有键的情况下，第二次重试会再次扣款。这是“恰好一次”语义真正生效的边界。

外部幂等键与活动派生键。 两种模式：

  • 活动派生 （如上）：键是 workflow_id + activity_id的函数。在重试间保持稳定；在不同逻辑调用间不同。大多数情况下的默认选择。
  • 外部提供：工作流的调用方提供业务幂等键。示例：一个 去重的任务提交 API 调用方传入 request_id，工作流将其作为下游服务的幂等键。当调用方可能针对同一逻辑请求多次调用工作流时，此方式是必要的。

“同一工作流中两个活动调用相同外部效果”的陷阱。 如果同一工作流中的两个活动都调用 Stripe.Charge 与 idempotencyKey = workflow_id，它们会发生冲突——第二个返回第一个的响应。始终在键中包含 activity_id ，而不仅仅是 workflow_id.

§17. Temporal Cloud 与托管服务

何时自行运行引擎，何时交由他人运行。

自托管的理由。

  • 深度定制。 自定义持久化后端（你的内部 Cassandra 分支）、自定义匹配算法、与专有认证系统集成（你的内部 IDP——身份提供商）、自带密钥加密。
  • 本地部署要求。 受监管行业（某些地区的银行、医疗）无法将数据发送给第三方。
  • 超大规模下的成本优势。 超过约每月10亿操作后，Cloud 的按操作计费超过自托管 + 小型运维团队的综合成本。
  • 与工作负载同址部署。 延迟敏感型工作流，需要引擎、worker、持久化和下游服务都在同一网络中。

托管服务（Temporal Cloud、AWS Step Functions、通过 Orkes 托管的 Conductor）的理由。

  • 小团队。 5人团队构建支付系统，不应把一半时间花在 Cassandra 修复上。
  • 快速上线。 Cloud 只需1天入门，而自托管集群搭建需要3个月。
  • 运维保障。 全天候运维、升级管理、引擎漏洞热修复、容量规划——一切包含在内。
  • 多区域故障转移。 Temporal Cloud 将跨区域复制作为功能提供；自托管需要自行构建。

Temporal Cloud 定价模型。 按操作计费 + 存储 + 支持等级。操作按约每百万操作 $25（可议价）计费。存储约 $0.05/GB-小时。支持等级从 Bronze（尽力而为）到 Platinum（全天候电话支持）。经济可预测——费用与使用量大致成比例。

混合模式。 部分团队自托管主集群（降本 + 定制），同时对预发/开发环境使用 Cloud（低流量、快速迭代）。运维负担以次线性方式增长。

何时扭转算盘。

  • 超过约1000个命名空间（Cloud 中的每命名空间运维开销成为约束；自托管胜出）。
  • 超过约每月10亿操作（成本曲线有利于自托管）。
  • 跨越 Cloud 尚未覆盖的地理边界（如中国、某些政府区域）。

对于大多数100名工程师以下、构建业务工作流的团队，Cloud 在 TCO（总拥有成本）上更优。对于大型公司内部以工作流即服务方式提供平台的团队，自托管通常是正确答案。

§18. 持久执行作为一种运动

“持久执行”已成为一个品类标签。Temporal 首创此概念；一批更轻量的替代方案随之兴起。

Temporal 级：基于事件溯源重放的完整持久执行。 Temporal、Cadence、Restate。重量级集群，确定性 SDK，为数百万并发工作流而设计。

更轻量的无服务器持久执行。

  • Inngest。 SaaS，无服务器。工作流是部署到 Inngest 运行时的 TypeScript 函数。通过事件触发。Inngest 中的步骤函数具有持久性（每个步骤的结果都会持久化）。针对“我们希望短时工作流具备持久性”的细分场景——典型工作流是秒到分钟级，而非天级。
  • Restate。 开源。持久 RPC + Kafka 风格日志。与特定 SDK 耦合度较低；适用于遵循 Restate 协议的 HTTP 服务。目标是“持久执行作为库，而非平台”。
  • Trigger.dev。 类似 Inngest，但更注重开发者体验。工作流为 TypeScript；提供 Web UI 用于触发和监控。有免费层；中等流量后开始收费。
  • Cloudflare Workflows。 Cloudflare 的新入场者。持久执行内嵌于 Workers 运行时。仅支持 JavaScript，与 Workers 生态系统深度集成。

“我们有小型工作流但需要持久性”的细分场景。 每天运行1000个工作流、每个持续30秒的团队不需要 Temporal 集群。他们需要的是：

  • 失败时的持久重试。
  • 等效于活动的步骤边界。
  • 查看运行状态的 UI。

Inngest/Trigger.dev/Restate 都能满足。Temporal 则大材小用——无论是从运维角度（运行集群）还是概念角度（确定性重放模型对30秒工作流来说规则过多）。

何时这些工具胜过 Temporal。

  • 冷启动很重要。 Inngest 的无服务器运行时为每个工作流步骤启动一个函数；无需空闲 worker 机群。对于每天不足100个工作流的场景，这是优势。
  • JavaScript 优先的技术栈。 Trigger.dev 和 Inngest 是 TypeScript 原生的；开发体验比 Temporal 的 TS SDK 更紧凑（后者优秀，但仍需要确定性重放思维）。
  • 默认事件驱动。 Inngest 的主要触发方式是事件；Temporal 的是 StartWorkflow 调用。对于“响应用户注册”类工作流，Inngest 的思维模型更契合。

何时 Temporal 仍然胜出。

  • 长期运行的工作流。 超过1天的任何工作流都倾向于选 Temporal。Inngest 也能做到，但空闲时间需要支付无服务器开销。
  • 具有自定义重试/补偿的复杂 saga。 Temporal 的一流支持难以匹敌。
  • 自托管需求。 大多数新入场者仅提供 SaaS 模式。
  • 高吞吐量。 Temporal 可扩展至每集群每秒10万+次状态转换；Inngest 是多租户 SaaS，共享容量。

该品类正在整合。 预计未来几年将出现2-3个赢家（Temporal、Inngest，也许还有 Restate）。“带有重放或持久化步骤语义的持久函数”这一形态，正在所有这些工具中趋于收敛。

§19. Airflow 与 Temporal——挥之不去的混淆

两个引擎，两种截然不同的问题形态，却频繁被混淆。这种混淆如此普遍，值得单独成节。

Airflow 的核心场景：按调度运行的批量 DAG。 DAG 是静态任务图。一次运行是 DAG 针对某个参数化配置（通常是日期分区）的一次执行。任务通常是重量级的（Spark 作业、dbt 运行、ML 训练脚本）。每个任务的开销从数百毫秒到秒级（调度器轮询循环 + worker 分发）。

优点：

  • 回填（对过去日期重新运行 DAG）。
  • 数据血缘（哪个任务生成了哪个数据集，跨运行追溯）。
  • Sensor 式等待（轮询外部条件）。
  • 调度器宕机后的补偿逻辑。
  • 与数据生态系统集成：SQL 算子、S3 sensor、dbt、Spark 等。

弱点：

  • 长期并发工作流（每个处于独特节点）——非为此设计。Airflow 可以并发运行数千个 DAG 运行，但不是数百万个。
  • 每任务延迟过高，不适合快速编排。
  • “等待外部信号”处理笨拙——你需要构建一个轮询 sensor。

Temporal 的核心场景：具有数百万并发实例的长期有状态工作流。 工作流是一段持久运行的代码。每个工作流都是唯一的——不同的输入，不同的状态。任务（活动）可以是重量级或轻量级的（单次 HTTP 调用）。每个活动的开销以毫秒计。

优点：

  • 数百万个并发工作流，各处于不同节点。
  • 长期等待（天、月级别），不占用资源。
  • 来自外部系统的信号（而非轮询）。
  • 以代码为工作流，具备完整的语言表达能力。
  • 补偿/saga 作为一等公民模式。

弱点：

  • 对日期范围的批量回填——处理笨拙。Temporal 没有内置“为2024年每一天重新运行此工作流”的概念。
  • 数据血缘——不提供。
  • 每任务开销过高，不适合行级数据转换。
  • 确定性约束使某些数据操作模式变得丑陋。

“将1秒工作流放入 Airflow”的反模式。 一个团队想编排一个总计耗时1秒的4步 API 调用序列，将其放入 Airflow。每个任务约有500ms 的调度器开销。整个“工作流”耗时3秒。他们每天运行1万次。Airflow 的调度器被这些微型工作流的任务调度压垮；他们真正的 DAG（实际 ETL）被延迟。工具选错了。Temporal 处理这类场景每个工作流只需50ms，不会给调度器带来任何压力。

“将每日 ETL 放入 Temporal”的反模式。 一个团队将每日数据管道构建为 Temporal 工作流。每天凌晨2点，每个分区（每天1000个分区）启动一个工作流。每个工作流运行一个 Spark 作业，耗时30分钟到4小时。工作流为每个活动（Spark 作业启动、完成、失败重试）保留历史事件。一年后，可见性集群中塞满了已完成的工作流。回填十分痛苦——每次都需要触发新的工作流运行，没有原生的“重新运行上周管道”概念。工具选错了。Airflow 可以原生支持回填、展示天间依赖的 DAG 可视化，以及内置数据血缘钩子。

决策规则。

  • 时间轴是日历？ （按日、按小时分区。） Airflow。
  • 时间轴是工作流自身的生命周期？ （此工作流从订单下单到交付全程存活。） Temporal。

换言之：如果你的工作流心智模型是“今日运行 vs 昨日运行”，那是批量 DAG（Airflow）。如果心智模型是“此工作流正在处理某个客户的交易，独立于任何时钟而生灭”，那就是持久执行（Temporal）。

许多团队两者都需要。每日 ETL 管道（Airflow）向实时订单工作流（Temporal）读取的数据提供数据。它们共存于同一技术栈，互不替代。

§20. 未涵盖的故障模式

§8 涵盖了常规故障模式——worker 崩溃、分片丢失、信号重复。更深层的类别：引擎自身的故障，需要工程升级处理的那些。

历史数据库脑裂。 Cassandra 的最终一致性大多被仲裁读写所屏蔽，但仍可能泄漏。场景：两个 Cassandra DC 之间发生网络分区。双方都继续接受写入（主-主模式）。同一分片的 executions 行在两侧以不同的 txn_id 值被更新。分区恢复后，修复使用时间戳进行协调——失败方的写入被静默丢弃。如果这些写入对应于已确认的工作流事件，工作流的视图将与客户端被告知的内容产生偏差。

缓解措施：以适当一致性部署 Cassandra（读写均使用 CL_QUORUM，RF=3，理想情况下每 DC RF=3 并使用跨 DC 仲裁）。定期运行一致性检查，比较各 DC 间的历史哈希值。关键工作流的写流量固定到单个 DC。

从偏差中恢复：极为痛苦。可能涉及从备份中对单 DC 真实来源重放事件，手动协调差异。通常需要数周的取证和运维工作。

因分区期间分片迁移导致工作流卡在 NotFound。 网络分区将历史节点与分片注册表隔离。注册表认为该节点已宕机，将其分片重新分配给存活节点。原节点仍认为自己拥有这些分片（脑裂）。访问原节点的客户端看到“工作流存在”的响应；访问存活节点的客户端看到“NotFound”，因为存活节点尚未加载该分片状态。分区恢复后，原节点放弃所有权；客户端看到一致行为，但在分区期间部分信号/查询流量被错误路由。

恢复：通常在分区关闭后自愈。对于收到冲突响应的工作流（信号被旧所有者接受，查询在新所有者处返回 NotFound），重置到分区前的状态。

Temporal 版本升级后的重放中断。 引擎本身携带内部协议变更。在数月内从 Temporal 1.18 升级到 1.22，如果升级影响了某些事件类型的序列化方式，可能会引入重放差异。官方升级路径是保守的——N-2 版本兼容——但客户偶尔会遇到边缘情况，旧工作流的历史无法在新引擎版本下重放。

缓解措施：使用 WorkflowReplayer 对一批有代表性的在途工作流历史进行预生产重放测试。如有任何失败，暂停升级。

恢复：将工作流固定到特定 Temporal SDK 版本，运行旧版 worker 池，逐步耗尽工作流。可能需要工程升级（Temporal 支持团队可协助处理升级相关的重放问题）。

扩容事件期间的可见性索引滞后。 流量峰值使工作流启动速率提升3倍。可见性索引器（从持久化层到 Elasticsearch 的 CDC 流的消费者）落后。UI 和仪表盘显示过时数据——10分钟前已完成的工作流仍显示为“运行中”。对可见性的程序化查询返回错误结果。

缓解措施：监控可见性延迟（Kafka 消费者延迟指标）。提前扩容可见性索引器。对于需要新鲜数据的查询，使用 DescribeWorkflowExecution （直接从历史读取，而非可见性）代替 ListWorkflows.

恢复：扩容可见性，等待追上，审计滞后窗口期间做出的任何决策。

持久化存储 schema 迁移。 添加索引、更改列类型。Cassandra 要求逐列族运行 schema 变更，并需要协调复制因子。操作不当时，集群一半看到旧 schema，另一半看到新 schema——一方的写入对另一方不可读。实用缓解措施：在专用维护窗口内运行所有 schema 变更，暂停复制，并首先在预发环境集群上验证。

大型工作流历史导致 worker 内存耗尽。 一个历史增长至8MB（接近10MB限制）的工作流。Worker 对其进行重放；每次重放在内存中保留完整历史加工作流状态。一个 worker 上100个这样的工作流 = 1GB 重放内存。Worker OOM（内存溢出），任务转移到另一个 worker，那个 worker 也 OOM。该工作流类型的 worker 机群发生级联故障。缓解措施：根据预期历史大小调整每 worker 并发限制；激进地使用 ContinueAsNew.

“任务队列分区热点”故障。 一个拥有4个分区的任务队列。某种工作流类型意外地只针对分区0（例如，通过粘性路由漏洞）。分区0的匹配分片饱和；任务积压；新工作流启动超时。其他分区空闲。缓解措施：监控每分区任务数；重新平衡分区分配；极端情况下，增加分区数并进行迁移。

§21. 使用场景展示

同一技术服务于截然不同的领域。

支付生命周期编排——Stripe Connect、Coinbase。 每笔资金流动都是一个工作流：校验、欺诈检查、扣款/转账、等待外部清算（ACH 1-3个工作日）、对账、通知。补偿：失败时退款或冲销。要求：审计（SOX——萨班斯-奥克斯利法案，PCI-DSS——支付卡行业数据安全标准）、恰好一次的副作用、长持续时间、人工审批信号。 变体匹配：代码即工作流（Temporal）；对于声明式状态机过于复杂，对于编排模式审计要求过严。

打车生命周期——Uber、Lyft。 每次打车都是一个工作流：匹配司机、接单、导航到上车点、行程、送达、向乘客收费、向司机付款、处理纠纷。时长从几分钟到几小时不等，纠纷处理可达数天。每天处理数千万次。 变体匹配：高规模的代码即工作流；Uber 最初的 Cadence 使用场景。

订单履行 — DoorDash、Amazon。 下单 → 确认 → 批次合并 → 分配配送员 → 取餐 → 配送 → 对账。高峰期每天新增数千万个工作流实例。典型时长 30-60 分钟，预约单可达数小时。接收来自配送员、顾客、客服的信号。 变体匹配：大量使用信号的代码即工作流。

CI/CD 编排 — Argo Workflows on Kubernetes。 流水线 = 克隆 → 构建 → 测试 → 安全扫描 → 部署 → 冒烟测试 → 晋级发布。每个步骤运行在独立的容器化 Pod 中。每天数百到数千条流水线。 变体匹配：Argo。工作 是 容器；将引擎与 Kubernetes 协同部署是正确的选择。Temporal 对于每步骤的开销而言过于重量级。

ML 训练流水线 — Argo + Airflow + 有时使用 Temporal。 按计划批量训练（每日推荐模型重训）：Airflow — DAG 运行、回填、血缘追踪。容器化触发式流水线：Argo on k8s。带反馈信号的实验管理或在线学习循环：Temporal（每个实验是一个随数据到达而变更的长期工作流）。 变体匹配：取决于具体形态。

视频转码 — Netflix Cosmos（Conductor）。 每次上传触发：校验、解复用、转码为多种格式（码率、编解码器、分辨率）、QC（质量控制）、打包、发布到 CDN（内容分发网络）。每天数千万次。异构 worker 池。 变体匹配：Conductor。多语言 worker，JSON 定义，语言无关的 poll/POST。

邮件/通知活动 — Mailchimp、营销平台。 计算受众、渲染模板、批量发送、跟踪打开/点击、重试退信。每次活动收件人达数亿；活动持续数天。 变体匹配：工作流引擎用于活动编排（每个活动对应一个工作流）；流处理器用于单封邮件的发送路径。

数据 ETL DAG — Airbnb、Lyft 及各数据团队。 日常 ETL = 提取、转换、校验、加载到数据仓库、刷新下游表、质量检查。每个团队数百个 DAG；按小时/天调度。 变体匹配：Airflow。专为此类形态而生。

这种广泛适用性本身就说明了问题。工作流引擎并非"支付工具"或"数据管道工具"——它是一种编排原语，以不同的变体服务于不同形态的多步骤流程。

§22. 真实世界的实现案例与数据

Uber Cadence（2017 至今）。 由 Maxim Fateev 领导的团队构建。到 2019 年已承载 Uber 大部分有状态业务逻辑。 数百万个工作流同时在飞, 每天数十亿次 activity 执行，基于 Cassandra，每个区域数百个节点，多区域双活。代码即工作流的奠基性参考实现。

Temporal（Cadence 分支，2019 至今）。 相同的架构血统，SDK 迭代更为激进（Go、Java、Python、TypeScript、.NET、PHP）。Temporal Cloud 最大客户的处理量达到 数百万级别的并发 workflow 与 每个 namespace 数万次 actions/sec。Stripe、Snap、Datadog、Box、Coinbase 均自托管部署。

Stripe Connect。 针对付款流程的 Temporal 级持久化执行。工作流等待 ACH 结算数天，失败时必须执行补偿操作。 数千万个正在进行的支付 随时可查。

Coinbase。 将每次提现都包装为工作流，因为合规要求提供单一可审计的完整记录。同时在飞的工作流达数百万级别。

DoorDash 订单生命周期。 订单 = 下单 → 确认 → 批次合并 → 分配 → 取餐 → 配送 → 对账，典型时长 30-60 分钟。 每天数千万个新工作流 在峰值时。

Netflix Conductor（2016 至今）。 开源。驱动 Cosmos（视频编码/转码）以及内容摄入和推荐刷新。 每天数千万个工作流。Airbnb 也将其用于异步付款。

Apache Airflow（2014 至今，起源于 Airbnb）。 计划性 ETL/ML 的主流选择。最大规模部署（Airbnb、Lyft、Pinterest）每天运行 数万次 DAG 运行.

AWS Step Functions（2016 至今）。 托管的声明式状态机。标准版：4k 次转换/秒/账户，约 $25/百万次转换。快速版：至多一次语义，最长 5 分钟。在以 AWS 为主的团队中广泛使用。

Argo Workflows（2017 至今，起源于 Intuit）。 通过 CRD 原生集成 Kubernetes。大量用于 k8s 上的 CI/CD 和 ML 训练。在 CNCF（云原生计算基金会）生态系统中广泛采用（Adobe、Intuit、Tesla）。

Snap。 将 Temporal 用于广告创意审核、内容审核流水线。同时在飞的工作流达数百万级别。

规律：任何拥有多步骤业务流程、长时间运行需求和审计需求的组织，在经历一两次"只用状态表"的尝试后，都会收敛到工作流引擎。具体形态决定了哪种变体最合适——但"需要 某种 引擎"这一结论在各个领域都具有普遍性。

§23. 总结

工作流引擎是为业务流程武装化的事件溯源：每个工作流都是其仅追加的历史记录，每个 worker 都是一台确定性重放机器，每个 activity 都是具有稳定标识的至少一次执行，从而使应用层幂等键能够实现恰好一次的副作用；整个系统按 hash(workflow_id) 分片到一个基于 quorum 的副本存储上，worker 上的粘性内存缓存使得完整重放代价只在每次 worker 切换时支付一次，稳态路径上代价为零——以此换取持久状态、重试、长时间等待、重放和审计能力，代价是强制要求工作流代码的每一行都必须是确定性的。
