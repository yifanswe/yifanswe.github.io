一份技术参考文档。可观测性是指一类系统，负责收集、存储和查询生产软件的遥测数据，使人类（以及日益增多的机器）能够诊断正在发生的问题。它不是单一产品，而是由三大支柱组成的三脚架——指标、日志、链路追踪——每个支柱受其独特访问模式的驱动，拥有各自不同的存储引擎。本文档涵盖这一类别的整体内容。

§1. 什么是可观测性

可观测性是 面向生产系统诊断的遥测数据收集、存储与查询。可观测系统的输出，是无需重新部署即可在生产环境中回答那些你事先不知道会问到的问题的能力——这使其区别于传统的"监控"，后者只能回答你预先配置了仪表盘的问题。

一个可观测性技术栈做三件事： 埋点 （进程内 SDK——OpenTelemetry、Datadog agent、Prometheus 客户端库、Sentry SDK）， 传输与存储 （遥测数据异步传送至各支柱独立的存储层），以及 查询/告警 （仪表盘、临时调查、阈值或错误预算告警）。

三大支柱

并非同一数据的三种视图——而是三种不同的数据类型，具有不同的体量、结构和查询模式。

  • 指标。 预聚合的数值型时间序列。每个数据点体量小（一个浮点数 + 一个时间戳 + 一组标签），但若标签泛滥则基数会很高。访问模式：对带标签的序列进行范围扫描，通常需要聚合（sum, rate, histogram_quantile）。用于仪表盘、SLO（Service Level Objective）追踪以及大多数告警。示例： http_requests_total{service="checkout",status="500"} 每 10 秒采样一次。
  • 日志。 由应用代码输出的自由格式或结构化文本事件。体量大（可观测性账单中占比最大的一块），内容多样，查询需将 grep 风格的子字符串搜索与结构化过滤器混合使用。用于故障定位、审计追踪，以及日益增多的产品分析场景。示例：一行 JSON {"ts":"...","level":"ERROR","service":"payment","trace_id":"abc","msg":"stripe API returned 500"}.
  • 链路追踪。 跨服务的因果工作链。一条链路追踪是一棵 span 树，每个 span 是一个元组（start_time, duration, service, operation, attributes, parent_span_id）。体量随吞吐量线性增长 requests × span_count_per_request，因此在任何非平凡规模下采样都是必须的。用于"这个请求的时间都花在哪里了"。示例：用户的一次点击产生了 20 个 span，跨越 api-gateway → auth → checkout → payment → stripe。

它的位置与它不是什么

可观测性存在于 应用旁侧 而非请求路径之内。做得好，对用户延迟无感；做得差，同步日志刷写会卡死应用。它 非 不是测试的替代品， 非 不是事务性系统（遥测是尽力而为的——重启时丢失 0.01% 的指标采样完全可接受）， 非 不是单一产品问题（"安装 Datadog"是采购决策，不是系统设计；技术选型——拉取 vs 推送、全文 vs 标签索引、头部 vs 尾部采样——不论选哪家厂商都必须面对），也 非 实时 × 全量保真 × 无限制保留 × 低成本四者兼得。每个平台都只能选其中两项，放弃其余。

§2. 各支柱的固有保证

每个支柱都有其契约——设计上能提供什么，以及你无法得到什么。陷阱在于误以为某项技术提供了它实际上并不具备的保证。

指标。 你能得到： 低廉的聚合成本、可预测的存储占用、按标签快速范围查询。 rate(http_requests_total[5m]) by (service) 对数千个序列的查询能在毫秒级返回，因为 TSDB（时序数据库）在结构上为这种数据形态做了深度优化。 你得不到： 逐事件的明细——一旦递增了一个计数器，就无法追问"具体是哪个用户触发了这个？"，该信息在聚合时已被丢弃。高基数也不是免费的；每一个唯一的标签值组合都是独立的一个序列。 需要额外分层： SLO 定义、告警规则、exemplar（直方图桶指向一个采样链路追踪的指针）。

日志。 你能得到： 完整的事件载荷，持久化存储，可按任意已索引字段查询。日志是每个服务所见所为的"事实来源"。 你得不到： 成本低廉——日志体量以字节计是指标的 50-200 倍。你也不会自动获得关联性；除非应用打上了 trace_id, service, user_id. 需要额外分层： 结构化 schema、链路上下文传播、保留期分层、PII（个人身份信息）脱敏、按服务设置速率限制。

链路追踪。 你能得到： 因果结构。给定一个 trace_id，你可以重建完整的调用树，以及每一跳的耗时。这是唯一能原生表达"延迟来自第三次 gRPC 调用 payment 服务"的支柱。 你得不到： 完整覆盖。在每秒请求量超过约 1 万的情况下，不采样的链路追踪成本不可承受；未采样到的部分就等于不存在。 需要额外分层： 采样策略（头部/尾部/混合）、跨 HTTP、gRPC、Kafka、Espresso CDC（Change Data Capture）、定时任务等所有异步边界的链路上下文传播。

无法逃脱的约束： 实时 × 全量保真 × 无限制保留 × 低成本——只能选两项。 指标牺牲细节（预聚合）。日志牺牲保留（迁移至冷存储）。链路追踪牺牲覆盖率（采样）。成本决定了每个支柱为此付出的代价有多大。

§3. 设计空间

这个领域有多个维度，需要在每个维度上选定一个位置。

维度 A：采集模型——拉取 vs 推送。 拉取（Prometheus）——TSDB 按计划抓取 /metrics 按计划采集指标端点。优势：基数和速率受抓取配置约束。劣势：必须能访问每个目标——在 NAT（网络地址转换）、防火墙之后，或对于短生命周期的批处理任务，此方案会失效。推送（Datadog agent、StatsD、OTLP/HTTP）——应用将遥测数据发送至采集器。优势：适用于 serverless、批处理、跨区域场景。劣势：失控的客户端可能 DDoS 平台；通过按租户限速来缓解。大多数大型平台采用混合方式——Prometheus 拉取长期运行的服务，OTLP 接收器处理短生命周期的任务。

维度 B：日志索引策略。 - 每词倒排索引 （ELK = Elasticsearch + Logstash + Kibana、OpenSearch、Splunk）。亚秒级全文 grep。存储空间为原始数据的 1.5-3 倍；运维成本高（Lucene 分片、JVM 调优）。 - 仅标签索引 （Loki、Grafana Cloud Logs）。对少量标签（服务、级别、区域）建立索引；载荷以压缩块存储。成本低。Grep 在查询时扫描数据块——对 service=api last 1h，对全局字符串搜索则较差。 - 列式存储 （ClickHouse、用于日志的 Apache Pinot、Honeycomb 的 Retriever）。每行一条日志，每列对应一个 JSON 字段。SQL 聚合查询快，列式压缩存储成本低。对自由文本的子字符串 grep 较弱，但擅长"X 按类别发生的频率"此类分析。 - 对象存储 + Parquet （CloudWatch Logs Insights、Athena、Snowflake on logs）。存储成本最低；查询延迟为秒级到分钟级。适合合规保留，不适合故障 grep。

维度 C：链路追踪采样。 头部采样 ——在链路追踪开始时决策，随请求传播。成本低，无法优先保留错误请求。 尾部采样 ——每个 span 都会被发出，按 trace_id 缓冲约 30 秒，决策时保留错误、慢请求以及随机的 p% 请求。内存开销大；需要将一条链路追踪的所有 span 路由到同一个采集器。 混合 ——头部采样 10%，再对剩余部分进行尾部采样。大多数生产平台的选择。

维度 D：存储分层模型。 仅热存储 （Monarch 将指标按区域保存在内存中；查询成本低，扩展成本高）。 分层存储 （Prometheus + Thanos：2 小时的数据块存于 NVMe，更旧的数据迁至 S3；Loki：近期数据块存于本地 SSD，更旧的数据存于对象存储）。 原生对象存储 （Tempo：Parquet 存于 S3，布隆过滤器索引；成本最低，冷读延迟最高）。

维度 E：埋点框架。 厂商专有 SDK（Datadog agent、New Relic）——自动埋点丰富，存在厂商锁定。 OpenTelemetry（OTel） ——厂商中立，CNCF（Cloud Native Computing Foundation）项目；Google、Microsoft、AWS、Datadog、Splunk、Honeycomb、New Relic 均提供支持。若不希望被厂商锁定，2026 年的默认选择。

§4. 各支柱的字节级细节

大多数可观测性文章在这里都语焉不详，只说"存储时间序列数据"。每个支柱使用不同的存储引擎，而引擎的选择是由访问模式决定的。

4.1 指标 — 使用 Gorilla 压缩的 Prometheus TSDB

Prometheus TSDB 是一个自定义的仅追加时间序列引擎，源自 Facebook 的 Beringei / Gorilla 论文（VLDB 2015）。它是事实上的参考设计——M3DB、VictoriaMetrics、InfluxDB IOx、Mimir、Cortex 以及 Datadog 内部的 TSDB 都使用同一系列技术。

为什么使用自定义引擎而非 RocksDB 或 InnoDB？ 四个事实决定了这一选择：（1）写入是仅追加且时间戳单调递增的；（2）同一序列内的浮点值高度相关；（3）读取几乎总是对带标签序列的范围扫描；（4）高基数是主要成本驱动因素，而非吞吐量。LSM（Log-Structured Merge）树会带来压缩开销，且每条记录都会重复存储系列标识符。B+ tree 在内部节点上浪费空间，且随机写入性能极差。Prometheus TSDB 最好理解为 一个带自定义值编解码器（Gorilla）和粗粒度块级压缩（2 小时块合并）的专用仅追加 LSM.

磁盘块布局（单个 2 小时块）：

block/
  meta.json                         # block ID, time range, stats
  chunks/
    000001                          # raw chunks file (binary)
    000002
  index                             # inverted index of label values
  tombstones                        # deleted ranges (rare)

A chunk inside chunks/000001:
  +---------+-----------+---------+-----------+----------------+
  | header  | timestamp | first   | timestamp | ... compressed |
  | (series | header    | sample  | delta of  | delta-of-delta |
  |  ref)   | base      | (full)  | delta     | + XOR floats   |
  +---------+-----------+---------+-----------+----------------+
   ~10 B     ~10 B       ~16 B     ~1-4 b/sample  ~1-12 b/sample

"1.37 字节/样本"的 Gorilla 数值来自于该格式在大量真实 Facebook 样本上的稳态平均值。与朴素的 16 字节/样本编码（8B 时间戳 + 8B 浮点数）相比，Gorilla 的 压缩密度约高 12 倍。在每秒 900 万样本的集群规模下，这意味着 12 MB/s 与 144 MB/s 的差距——决定了是小型 TSDB 集群还是百节点的庞然大物。

Gorilla — 时间戳部分（delta-of-delta）。 抓取按固定计划进行。若 t1=1000，t2=1010，t3=1020，则 delta 为 10、10、10，delta-of-delta 为 0、0，可用 每样本 1 bit编码。若某次抓取延迟（10.001s），delta-of-delta 为一个小整数，用 7-9 bit 编码。最坏情况约 32 bit。

Gorilla — 浮点数部分（XOR）。 对于变化缓慢的值（CPU 为 47.3，然后是 47.5，再是 47.6），连续 IEEE 754 浮点数的 XOR 结果有大量前导零和尾随零。Gorilla 将 XOR 结果编码为 (leading_zeros, meaningful_bits, value)。与前一个值相同 → 1 bit。小幅变化 → 约 12-20 bit。大幅变化 → 最多 64 bit。平均约 7-11 bit。

倒排标签索引。 类似这样的查询 sum(rate(http_requests_total{service="api",region="us-east-1"}[5m])) 必须找到所有匹配的系列 service="api" AND region="us-east-1"。TSDB 维护一个从 (label_name, label_value) → sorted list of series_IDs构建的倒排列表索引。标签交集是倒排列表的 AND 操作——与 Lucene 搜索使用的原语相同。O(min(a, b)) 合并扫描，速度快。这也是基数爆炸带来损害的地方——每个新的唯一值都会增加一条倒排列表记录。

Head 块 + WAL（预写日志）。 当前约 2 小时窗口的内存状态存储在 series_id → most_recent_chunk的哈希表中。新样本追加到打开的 chunk。每个样本在确认前还会写入磁盘上的 WAL。fsync 每约 1s 批量执行一次（可配置）——在 fsync 之前崩溃最多丢失 1s 的样本（对监控而言可接受）。崩溃恢复时，WAL 重放会重建 head 块。

逐步了解一次完整的抓取过程。

1. App exposes /metrics:
   http_requests_total{service="api",status="200"} 4823 1716391200000

2. Scraper (HTTP GET every 10s): parse text → (metric + sorted label kv) hashes
   to series_id → append (timestamp, value) to open chunk in head block →
   append WAL record (series_id, timestamp, value).

3. WAL hits page cache; fsync batches every ~1s. Pre-fsync crash loses ≤1s.

4. Open chunk reaches 120 samples (~20 min) → close it, Gorilla-encode →
   new chunk opens. Closed chunks stay in head-block RAM.

5. Every 2h the head block is "cut" — closed chunks flush to a persistent
   block. Inverted label index rebuilt. WAL truncated.

6. After 24h, persistent blocks compact into larger blocks and ship to
   long-term storage (S3 via Thanos/Mimir).

基数是致命开关。 存储空间随基数而非采样率扩展。抓取间隔加倍，存储减半；基数加倍，存储翻倍，同时倒排索引大小、head 块内存和查询延迟也翻倍。 user_id 有 1 亿个值 → 1 亿个系列；每个打开的 chunk 约占 1 KB，仅 head 块就需要 100 GB 内存。Prometheus 早在此之前就会 OOM（内存溢出）。每实例基数在 1M-10M 以内没问题；超过 10M 则需分片（Mimir）或删除该标签。

4.2 日志 — 三种竞争存储模型

模型 A：每词全量倒排索引（ELK / OpenSearch / Splunk）。 每个词被分词后加入 Lucene 倒排索引。存储约为原始数据的 1.5-3 倍。通过倒排列表交集，查询延迟可达毫秒级。在每天 85 TB 的规模下成本极高：需要 1000+ 个节点和大量 TB 级 RAM 用于热索引。ELK 在每天 ~1 TB 以下表现出色；超过这一规模后成本曲线急剧上升。 level=ERROR AND message:"connection refused" 模型 B：索引标签，扫描其余内容（Loki）。

只对 建索引；日志内容以 zstd 压缩块的形式存储，以标签集为地址。查询 service, host, level, region ：（1）标签索引找到该时间范围内 {service="api"} |= "connection refused"对应的 chunk；（2）对每个 chunk，从 S3 拉取、解压、grep。之所以可行，是因为大多数查询首先按 service+time 缩小范围，按词索引的开销正是被省去的，且 S3 并行读取很容易实现。Loki 的标签与 Prometheus 一致——跨支柱一致性免费获得。 service="api" 模型 C：列式存储（ClickHouse、Pinot、Honeycomb 的 Retriever）。

Loki:
  /index/   # (label-set hash, time range) → chunk references
  /chunks/  # zstd-compressed payload, one chunk = one label-set + 1-2h
            # production: S3, GCS, ABS

每行对应一条日志，每列对应一个 JSON 字段。按列建索引使得 在每天 TB 级的集群上能亚秒级返回结果。LinkedIn 的日志分析正是采用日志 → Kafka → Pinot 的方式——大多数"X 发生了多少次"的查询都是 SQL 聚合，而非 grep。 SELECT count() FROM logs WHERE service='api' AND level='ERROR' GROUP BY error_code 决策规则。

以 grep 为主 → Loki。以 SQL 聚合为主 → Pinot/ClickHouse。需要即席全文检索且预算不受限 → ELK。超长期合规留存、查询频率低 → 对象存储上的 Parquet 配合 Athena/Snowflake。大规模场景下通常同时运行多套；LinkedIn 将同一条 Kafka 流分别接入 Loki 等价系统（grep）和 Pinot（分析）。 4.3 链路追踪 — 按 trace_id 分区，采样存储

存储结构。

每个 span 是一条记录： 主要查询是"给我 trace_id=X 的所有 span"。较少见的查询是："找出过去一小时内 service=payment 且 duration > 2s 且 status=ERROR 的链路"。

(trace_id, span_id, parent_span_id, service, operation,
 start_ns, duration_ns, status, attributes:{k:v}, events:[...])

对于 trace_id 查找，以 trace_id 为键的键值存储是完美方案。Jaeger 最初使用

Cassandra（LSM tree，分区键 = trace_id） 。现代 Tempo 将Parquet 文件写入 S3 并使用 Bloom filter 索引。Cassandra 和 S3 Parquet 都非常适合"大量写入、偶尔按键点查"的场景。 对于搜索查询（"找出满足条件的链路……"），需要二级索引——通常是一个小型 Lucene/Elasticsearch 索引，只存储可搜索的 span 属性以及指向廉价存储中完整 span 内容的指针。Honeycomb 采用了不同的路线——其 Retriever 将每个 span 作为列式数据库中的一行存储，不针对任何字段建索引，依靠列式扫描速度将"对链路的即席查询"作为主要工作流。

按 trace_id 分区对于尾部采样是不可或缺的。

原因：尾部采样决策是基于某个 trace_id 的完整 span 集合做出的。如果 trace abc 的 span 1 落在采集器 A，span 2 落在采集器 B，则两个采集器都无法决定是否保留该链路。Kafka 主题按 分区，确保同一链路的所有 span 哈希到同一分区并由同一消费者读取。 trace_id 头部采样与尾部采样。

头部采样： 在链路开始时抛硬币；采样标志通过 传播。开销小，无需协调，但无法优先保留错误（此时尚不知情）。 traceparent尾部采样： 每个 span 都发出，采集器按 trace_id 缓冲约 30s，然后决策——若任意 span 有 或 duration > 2× service p99，则保留全部，否则按随机概率 p 保留。能捕获有价值的链路；需要内存来缓冲，且需按 trace_id 分区以确保同一链路的所有 span 落在同一采集器。 status=ERROR 混合模式（大多数生产环境）： SDK 在头部以 10% 采样（在网络传输前丢弃 90%），采集器在此基础上进行尾部采样，保留所有错误/慢请求以及 1% 随机样本。净采样率约 1.5%，同时保留 100% 的有价值链路。 逐步了解一次完整的链路组装过程。

eBPF（extended Berkeley Packet Filter）是一项 Linux 内核特性，允许沙箱化程序在内核空间运行并附着于各类事件——syscall、kprobe、uprobe、tracepoint、网络 socket、perf event。其在可观测性方面的承诺是：

在不修改应用代码的情况下，观察每个进程的一切行为 6.1 核心能力.

传统 SDK 在进程内运行——它只能看到应用显式调用的内容。eBPF 在内核中运行——它能看到每个 syscall、每个网络包、每个 CPU 样本。从可观测性角度来看，eBPF 用

无需任何人主动接入的内核侧 hook 取代了 sidecar、agent 和 SDK 插桩.

具体而言，eBPF 程序可以：\n- 带时序地追踪每次 TCP connect、accept、send、recv——无需网络库插桩。\n- 每隔 N ms 对所有进程采样 CPU 调用栈——无需在应用中启用 profiling。\n- 从数据包中解码 HTTP/gRPC/Postgres/MySQL/Redis 协议——无需语言 SDK。\n- 捕获文件 syscall、exec 调用、signal 发送——用于安全取证。\n- 在内核中聚合指标（eBPF maps）并以 Prometheus 端点的形式暴露。

6.2 主要 eBPF 可观测性产品

Cilium Hubble（无 sidecar 的网络可观测性）。 Hubble 是 Cilium 的一部分，Cilium 是一个基于 eBPF 的 Kubernetes CNI（Container Network Interface）。每个 Pod 之间的数据包都在内核空间被观测。Hubble 生成 L3-L7（第三层到第七层）流日志——源 Pod、目标 Pod、协议、HTTP 路径、gRPC 方法、响应码、延迟。它取代了用于网络可观测性的 service mesh sidecar（Istio 的 Envoy）。每个 Pod 的 CPU 开销约为零（由内核完成工作）；零 per-pod sidecar 内存占用。

Pixie（无需代码改动，自动插桩每个服务）。 被 New Relic 收购。以 Kubernetes DaemonSet 的形式部署 eBPF agent。通过在加密前 hook 用户态 OpenSSL 函数（对 SSL_read/SSL_write的 uprobe）来解码 HTTP/2、gRPC、Kafka、MySQL、Postgres、Redis、DNS——包括 TLS 加密流量。结果：为每个服务提供链路追踪和请求/响应 payload，零代码改动。承诺是"kubectl apply，30 秒后看到一切"。

Parca / Pyroscope（通过 eBPF perf event 进行持续性能分析）。 以 100 Hz 的频率在每个节点的所有进程上采样 CPU 调用栈。调用栈以火焰图的形式记录并作为时间序列存储。通过对比部署前后的火焰图，找出"这次提交给 p99 增加了 10ms"的原因。eBPF 使这一操作成本极低（约 1% CPU 开销）。

Falco（安全可观测性）。 CNCF 运行时安全项目。eBPF 程序监控每个系统调用，应用规则（"若非 root 进程读取 /etc/shadow 则告警"），转发事件。是运行时威胁检测和合规态势的基础。

Tetragon（Cilium 的安全可观测性伴侣项目）。 基于 eBPF 的运行时执行与可观测性。监控进程执行、文件访问、网络连接，并可选择在内核层 阻断 某个系统调用——带有"牙齿"的可观测性。

6.3 "零代码改动"的承诺

这一营销主张是：无需更改代码，数分钟内即可对整个集群进行埋点，看到每一个请求。现实情况是：

无需代码改动即可实现的功能。 - L4（第四层）网络流：每条 TCP/UDP 连接，含字节数与耗时。\n- L7 协议解码：HTTP/1、HTTP/2、gRPC、Postgres wire、Kafka wire、DNS、Redis、MySQL。\n- CPU 分析：以 100 Hz 采样调用栈，通过 DWARF 调试信息映射到符号名。\n- 系统调用追踪：每次文件打开、每次 exec、每次网络连接。\n- 内存分析（有限）：通过 uprobe 挂载 malloc/free 进行分配采样。

仍需应用层埋点的功能。 - 内核无法感知的业务操作——内核没有"用户点击结算"的钩子。"compute_recommendations"这类 Span 名称需要手动调用 SDK。\n- 自定义属性—— order_id, merchant_tier。这些信息存在于应用内存中，而非网络传输中。\n- Trace 上下文传播——尽管 eBPF 可以 读取 HTTP 请求中的 traceparent header，但在跨异步边界（Kafka、后台任务）传播时，仍需 SDK 支持。\n- 业务日志行——"欺诈检查因 rule_id=42 拒绝了该交易"——由应用程序生成，对 eBPF 不可见。

因此，eBPF 是 OTel 的 补充 ，而非替代品。eBPF 在 L4-L7 网络可观测性和持续分析方面占优；OTel 在业务语义 Span 和指标方面占优。

6.4 内核版本约束

eBPF 的能力取决于内核版本，生态系统演进迅速：

  • Linux 4.9+（2016）： 基础 eBPF、kprobe、perf 事件。所有 eBPF 可观测性的最低门槛。
  • Linux 4.18+（2018）： BTF（BPF Type Format），支持可移植 eBPF 程序（CO-RE——编译一次，随处运行）。
  • Linux 5.4+（2019）： bpf_skb_load_bytes_relative，更灵活的 socket 程序。
  • Linux 5.8+（2020）： 环形缓冲区、原子操作、更广泛的 uprobe 支持。
  • Linux 5.10+（2020，LTS）： 现代生产环境的最低基准。Cilium Hubble、Pixie、Parca 均要求 5.10+。
  • Linux 5.15-6.x： 更新的钩子类型（如 bpf_loop、kfunc），某些 Cilium Tetragon 特性需要此版本。

较旧内核（带 backport 的 RHEL 7 内核 3.10、RHEL 8 内核 4.18）严重限制了可用功能。部分团队仍在运行对现代 eBPF 可观测性栈而言过于陈旧的内核——这是推广采用时的结构性制约。

6.5 eBPF 特有的故障模式

验证器拒绝程序。 每个 eBPF 程序在加载时都由内核验证器检查——循环有界、无越界内存访问。复杂程序可能被拒绝；内核版本升级导致验证器变更，影响可接受的程序范围。典型案例："在 5.10 上能跑，在 5.15 上失败"。

有缺陷的程序导致内核崩溃。 有了验证器这种情况很罕见，但历史上 JIT 编译和辅助函数中存在的 bug 确实导致过内核崩溃。新 eBPF 探针的生产发布遵循灰度发布规范。

高负载下的性能开销。 编写不当的程序（如打印每个数据包）会使 CPU 饱和。包含数百万条目的 Map 会产生哈希压力。Per-CPU Map 和环形缓冲区是标准的高吞吐量方案。

加密流量盲点。 TLS 对数据包层 eBPF 隐藏了 L7 载荷。Pixie 的解决方法是在加密前 uprobe OpenSSL；这对动态链接的 OpenSSL 有效，但对静态链接的二进制文件、Go 的 crypto/tls、或 BoringSSL 变体则无效。不同运行时的覆盖范围参差不齐。

§7. 持续分析

持续分析是近期许多可观测性讨论中的 第四支柱 。一次性分析（在工程师怀疑有问题时运行 perf 或 pprof）只能给出一个快照，而持续分析则对所有进程持续采样，并以时间序列形式存储。其产出是：对于任意历史时间戳，你可以询问"昨天下午 2:34 CPU 在做什么？"

7.1 数据模型

一个 profile 是一张 flamegraph ——一棵由 (stack_trace → cumulative_sample_count)组成的树。每个采样是一个调用栈—— [func_a, func_b, func_c] ——附带权重（1 次采样、10 ms、64 KB 内存，取决于 profile 的度量对象）。flamegraph 将数百万个采样聚合成一棵树，展示时间/内存/分配的消耗分布。

持续分析：每隔 N 秒（通常为 10 秒）为每个进程采集一张 flamegraph，并 以 flamegraph 时间序列的形式存储，可按服务、主机、时间范围查询。存储形态与 trace 类似——大量事件型数据——但载荷为树形结构，而非 span 树。

7.2 Profile 类型

  • CPU profile。 以固定速率（10-100 Hz）采样调用栈。告诉你 CPU 周期花在了哪里。
  • 堆 / 内存 profile。 在分配点采样调用栈，按分配字节数加权。告诉你内存花在了哪里。
  • 锁 / 竞争 profile。 在阻塞于互斥锁的调用点采样，按等待时间加权。告诉你线程在哪里排队等待。
  • I/O profile。 在阻塞于 I/O 的系统调用点采样。告诉你阻塞 I/O 发生在哪里。
  • Goroutine / 线程 profile。 对所有存活的 goroutine 或线程及其当前调用栈进行快照——用于诊断死锁和 goroutine 泄漏。

7.3 主要持续分析产品

Pyroscope（Grafana Labs）。 开源。最初是独立产品，现已并入 Grafana 技术栈，成为 Grafana Pyroscope。支持拉取或推送 profile 摄入；数据存储于本地 SSD + S3 以支持长期保留。提供 flamegraph diff 视图——选择两个时间范围，查看变化。

Parca（Polar Signals）。 开源、基于 eBPF 的持续分析器。eBPF 程序以 100 Hz 在每个节点的每个进程上采样调用栈，发送至 Parca 服务端。通过 DWARF 调试信息进行调用栈符号化。主打"自动分析一切"的理念。

Datadog Continuous Profiler。 SaaS 服务。支持 JVM、Go、.NET、Python、Node 代理。差异视图与 trace 集成——对于缓慢的 trace，可点击"显示该 span 期间的 profile"，查看该时间段的 CPU flamegraph。

Pyrra / Phlare。 同类的其他开源竞争者。

7.4 Flamegraph diff——定位性能回归

杀手级用例： "这次提交让 p99 增加了 10ms，调用栈中是哪部分导致的？"

取部署前一小时和部署后一小时各一张 flamegraph，对比差异：对每个 (stack_trace) 计算 samples_after − samples_before。增长的调用帧标为热色，缩减的标为冷色。读取 diff——通常有一帧会跳出来："没错，新的 JSON 序列化器每次调用多耗了 8ms。"

这将原本需要数天的排查过程（"这次部署的 17 处改动中，哪一个导致了回归？"）压缩为 10 分钟的复盘。

7.5 与一次性分析的区别

维度 一次性 profile（perf、pprof） 持续 profile
适用场景 被动触发，在怀疑有问题之后 始终运行，主动采集
覆盖范围 单进程，单时间窗口 所有进程，每个采样间隔
存储 一个磁盘文件 时间序列存储，可按时间/服务查询
工作流 SSH 登录，运行分析器，复制文件，分析 打开仪表盘，选择时间范围
成本 每次排查的工程师耗时 ~1% CPU 始终开启的开销
适用场景 在开发/预发布环境中深度排查 生产环境回归排查

从一次性到持续的转变，与从"SSH 查看日志"到"集中日志聚合"的转变如出一辙——从被动响应转向环境感知式可观测性。

7.6 故障模式

stripped 二进制文件导致符号化失败。 若没有 DWARF 调试信息或基于 build-id 的 debuginfod，eBPF 分析会显示十六进制地址而非函数名。在运维上，你必须将符号文件发送给分析器，或运行 debuginfod。

绿色线程运行时的采样偏差。 CPU 采样只能看到一个 OS 线程；对于 goroutine 或异步任务，同一个 OS 线程会运行多个逻辑上下文。运行时协作机制（Go 的 pprof label）有所帮助。

profile 存储成本。 简单实现会将每张 flamegraph 存储为树结构。现代分析器采用列式存储（Parca 风格）并结合增量压缩——在大规模集群下仍是真实的资金成本。

§8. 深入剖析采样策略

采样使 trace（以及越来越多的日志和 profile）变得经济可行。错误的采样策略会让平台要么过于昂贵，要么对关键事件视而不见。本节深入探讨采样策略空间——§3 介绍了头部/尾部/混合的轴线；此处进入运维层面的深度。

8.1 头部采样（在根 Span 处决策）

根服务的 Trace SDK 掷一枚加权硬币： random() < 0.01。若为是，则在 trace 上下文中设置 sampled=1 ；若为否，则设置 sampled=0。采样标志随 traceparent 在每个下游服务中传播。每个下游 SDK 都遵从该标志——要么生成 span，要么跳过。

优势。 廉价：99% 的 trace 在任何地方都不产生 span。零协调：SDK 独立决策。延迟中性：无需缓冲。是"设置后忽略"的首选方案。

弱点。 无法优先保留有趣的 trace。在根 Span 阶段，你还不知道请求是否会出错、耗时 5 秒，或触发罕见的 bug。等你知道的时候，决策早已锁定。"ML 团队无法调试 0.1% 的 bug"的故障（§22.3）正是源于此。

变体： - 概率头部采样。 对所有 trace 均匀 p 采样。\n- 按路由头部采样。 /api/healthcheck 以 0.001% 采样， /api/checkout 以 10% 采样。路由级策略。\n- 按租户头部采样。 免费层客户以 1% 采样，企业客户以 100% 采样。SaaS 感知分层。

8.2 尾部采样（在 trace 结束时决策）

每个服务无条件生成所有 span。收集器（或专用尾部采样层）按 trace_id 缓冲 span 约 30 秒。当 trace 被判定为"完整"（收到最后一个 span + 超时，或根 span 结束 + 超时）时，应用策略：若任意 span 包含 status=ERROR、或 trace 持续时间 > 2s、或 http.status_code >= 500，或自定义谓词，则全量保留；否则以 p=0.01 的概率随机保留。

优势。 按定义捕获有趣的 trace。错误和慢请求以 100% 保留，即使其根本原因仅占 1% 的频率。随机基线用于采样率归一化。

弱点。 - 内存。 每条 trace 的每个 span 都在收集器 RAM 中驻留约 30 秒。以每秒摄入 6000 万个 span 计，30 秒缓冲约为 18 亿个 span。每个 span 约 2 KB，尾部采样层合计约需 3.6 TB RAM。\n- 将一条 trace 的所有 span 路由到同一个收集器。 否则，span 1 落在收集器 A，span 2 落在收集器 B——两者都看不到完整 trace，都无法决策。按 trace_id 分区的 Kafka topic 是标准解决方案。\n- 迟到的 span。 作为某条 trace 一部分的长时运行 span（如 5 分钟的批处理任务），若该 trace 的根 span 在 30 秒前已结束，则会从缓冲区中被逐出，永远无法与其 trace 关联。这是缓冲窗口与尾部采样正确性之间的权衡。\n- 采样器作为集中式瓶颈。 这是必须高可用的组件。若尾部采样层宕机，所有 trace（包括错误）都面临风险。

8.3 混合策略（生产环境的最优解）

大多数生产部署将头部采样与尾部采样结合使用：

SDK head-samples at 10%    → drops 90% before any wire bytes
   ↓
Collector tail-samples on remaining → keeps 100% of errors + slow,
                                        keeps 1% random of healthy
   ↓
Net retention: ~1.5% of all traces, 100% of interesting traces

头部阶段节省了 Kafka 带宽和尾部采样器的 RAM。尾部阶段从头部采样流中挽救出有价值的 trace。

采样键至关重要。 头部采样必须使用在所有 trace 中均匀分布的键——通常是 trace_id 本身，W3C 规范要求其具有 128 位随机性。种子质量差的随机数生成器（基于时间戳，参见 §22.6 有偏采样）会产生非均匀的采样结果。

一致性采样。 sampled = trace_id_hash mod 100 < 10 确保一条 trace 的所有 span 做出相同的头部决策——这是 trace 完整性的必要条件。

8.4 自适应采样（速率随流量动态调整）

静态采样率在流量突增时会失效。以 1% 采样率遇到 10 倍流量突增时，span 摄入量也随之增加 10 倍；下游存储无法承受。自适应采样与流量成反比调整采样率：

target_spans_per_second = 100K
current_traffic = N requests/sec
new_sample_rate = target / N        (clamped to [0.001, 1.0])

实现逻辑位于 SDK 或收集器中。Datadog 的 APM 和 Honeycomb 的 auto-sample 均实现了各自的变体。

按路由自适应采样。 对 /api/checkout 采样，使其 span 速率保持在 10K/秒。对 /api/healthcheck 采样，使其 span 速率保持在 100/秒。不同路由动态获得不同的采样率。

8.5 "1% 采样但 100% 保留错误"混合方案

生产环境中最常被引用的方案：

  • SDK 端头部采样： 10%（剔除大量健康流量）。
  • 收集器端尾部采样：
  • 若存在 status=ERROR 或 http.status_code >= 500.
  • 若存在 duration > p99_of_service.
  • 若应用标记为 force-keep （调试标志、A/B 测试队列）则保留 100%。
  • 否则，以 1% 的概率保留。

最终保留率：约 1.1% 的 trace。其中"有价值"的类别相对于基线被过度代表；通过跟踪每个类别的原始采样率，仍可计算出真实的发生率。

8.6 日志和指标的采样

日志。 日志采样更难，因为每行日志往往单独具有意义（一条错误日志可能是某个事件的唯一证据）。许多团队保留 100% 的 ERROR/WARN 级别日志，并对 INFO/DEBUG 级别日志采样至 10%。有些团队应用按事件名称的采样：保留 100% 的 payment_charged，保留 1% 的 user_pageview.

指标。 指标是预聚合的，因此"采样"在同等意义上并不适用。最接近的类比是基数控制——丢弃高基数标签，仅导出排名前 N 的维度。经济性问题在于基数，而非采样。

Profile。 持续分析在每个 CPU 上以 100 Hz 采样，本身已是重度采样。进一步的"trace 关联分析"（仅在采样 trace 期间进行分析）是一种新兴技术。

§9. SLO / SLI / SLA 框架

服务级别可靠性术语借鉴自 Google SRE 一书，现已成为行业标准。混淆这三个术语会浪费大量会议时间；其背后的规范是使告警和事故响应保持连贯的关键。

9.1 定义

SLI——Service Level Indicator，服务级别指标。 度量值。 关于服务健康状况的可量化信号。示例：\n- successful_requests / total_requests （1 分钟窗口内）。\n- requests_below_500ms / total_requests. - availability = uptime / total_time. - freshness = max(now() − event_timestamp) （用于流式管道）。

SLI 是一个 指标 ——从平台已采集的可观测性原始数据中计算得出。

SLO——Service Level Objective，服务级别目标。 团队内部承诺的目标。 SLI 必须满足的阈值。\n- "30 天滚动窗口内，99.9% 的请求成功。"\n- "7 天滚动窗口内，99% 的请求在 500ms 内完成。"\n- "99% 的事件，流式管道新鲜度 < 60 秒。"

SLO 是内部的期望目标——工程团队声称能够达到的标准。

SLA——Service Level Agreement，服务级别协议。 对客户的合同承诺，通常附带经济责任。 示例：\n- AWS S3："每月 99.9% 可用时间，否则按宕机时间比例退款。"\n- Stripe："API 可用性 99.99%。"

SLA 通常 比 SLO 宽松 ——这是有意为之：若工程团队内部目标为 99.9%（SLO），但对外合同承诺为 99.5%（SLA），则差值（0.4%）是"我们未达内部目标"与"我们欠客户赔偿"之间的安全边际。SLA 通常也采用更粗粒度的时间窗口（按月而非滚动计算）。

关键点： SLA 是商业产物，SLO 是工程产物，SLI 是度量产物。 将三者混用会造成运维混乱。

9.2 错误预算 — 工程实践

若 SLO = 30 天内 99.9%，则 error budget = 1 − 0.999 = 0.1% = 43.2 分钟 即每 30 天内允许的宕机时间（或失败请求量）。团队每月有 43.2 分钟可"消耗"在故障、失败发布和高风险变更上。

为何错误预算的视角如此有效。

  • 它使风险显性化。 "我们还剩 38 分钟错误预算，同时计划了一次重大发布"——这是一个可量化的风险。
  • 它对齐了工程与产品。 产品团队为了更快推出更多功能而消耗错误预算；一旦预算耗尽，所有精力都转向可用性保障。
  • 冻结发布规则。 若错误预算耗尽，发布冻结——在下一个窗口期之前不得推送任何高风险变更。部分团队将此自动化（发布流水线检查 SLO 状态）。
  • 它避免了"全有或全无"的陷阱。 没有预算时，每一分钟宕机都是"失败"；有了预算，每月前 43 分钟是可接受的，预算完全耗尽才触发升级。

9.3 多窗口多燃烧率告警

这是与 SLO / 错误预算配套的告警创新。

朴素做法：当 SLI < SLO 时告警。 这很糟糕——等到月度 SLI 突破月度 SLO 时，当月预算早已耗尽，窗口早已错过。

正确做法：基于错误预算的 燃烧率 进行告警。

burn_rate = current_error_rate / acceptable_error_rate

若 SLO 为 99.9%（错误率 ≤ 0.1%），当前错误率为 1.0%，则 burn_rate = 1.0 / 0.1 = 10x。团队正以可持续速率的 10 倍消耗预算。以 10 倍速率，月度预算将在 3 天内耗尽。

多燃烧率、多窗口。 不要只基于单一窗口告警——将短窗口（灵敏、噪声多）与长窗口（迟钝、稳定）结合，同时覆盖快速和慢速燃烧。

Google SRE 手册的典型规则集：

严重级别 燃烧率 短窗口 长窗口 预算耗尽时间
紧急（快速燃烧） 14.4x 5 min 1 hour ~2 days
紧急（中速燃烧） 6x 30 min 6 hours ~5 days
工单（慢速燃烧） 3x 2 hour 24 hours ~10 days
工单（最慢燃烧） 1x 12 hour 72 hours budget consumed at end of window

复合条件（short_window_burn > X AND long_window_burn > X）可防止因一分钟瞬时毛刺（短窗口高、长窗口低 → 不告警）而产生抖动。

9.4 "仅在用户受影响时告警"带来的收益

与基于阈值的告警（CPU > 80%, queue_depth > 1000):

  • 告警数量大幅减少。每个服务约 5 条告警（覆盖紧急快速、紧急中速、工单慢速燃烧 × 1–2 个 SLI），而非 50–200 条。
  • 每条告警都代表真实的用户可见影响，而非某个症状。
  • 工程师不再忽视呼叫机。
  • 容量规划、发布门控和值班规模都对齐到同一个数字——错误预算。

§10. 告警架构

告警平面是可观测性中面向人工操作的一侧。架构至关重要，因为在负载下可靠触发、路由到正确的人、不将团队淹没在噪声中的告警，与一个简单的 if x > threshold: page 循环截然不同。

10.1 Prometheus AlertManager — 典型参考实现

在 Prometheus 技术栈中，AlertManager 是告警路由守护进程。流程如下：

[Prometheus] evaluates [alert rules] (PromQL expressions)
    ↓ when rule fires
[AlertManager] receives alert (HTTP POST)
    ↓
[Grouping] — combine related alerts by labels (cluster, service, alertname)
    ↓
[Inhibition] — suppress alerts implied by other alerts
                  (cluster down → don't page on every pod down)
    ↓
[Silences] — mute alerts during planned maintenance
    ↓
[Routing tree] — match labels to receivers
                  team=payments → PagerDuty key X
                  team=infra    → PagerDuty key Y, also Slack #alerts-infra
    ↓
[Receivers]
    ├─ PagerDuty / Opsgenie (paging)
    ├─ Slack / MS Teams (chat)
    ├─ Email
    ├─ Webhook (incident.io, Rootly, custom)
    └─ SMS

告警规则 以 PromQL（Prometheus Query Language）编写：

groups:
- name: payments
  rules:
  - alert: PaymentErrorBudgetBurnFast
    expr: |
      (
        rate(payment_failures_total[5m]) / rate(payment_requests_total[5m]) > 14.4 * 0.001
        AND
        rate(payment_failures_total[1h]) / rate(payment_requests_total[1h]) > 14.4 * 0.001
      )
    for: 2m
    labels:
      severity: page
      team: payments
    annotations:
      summary: "Payment SLO budget burning at 14.4x rate"
      runbook: "https://wiki.example.com/runbooks/payment-burn"
      dashboard: "https://grafana.example.com/d/payments-slo"

for: 2m ——仅在条件持续 2 分钟后告警（避免抖动）。 annotations.runbook ——在页面中附上 runbook 链接，让值班人员不必从零开始。

10.2 分组与路由 — 噪声控制层

分组。 若不分组，一次集群级故障触发 800 条 pod-down 告警，会产生 800 次呼叫。AlertManager 的 group_by: [cluster, service] 将它们折叠为一条："service=api in cluster=use1 has 200 pods down。"一次呼叫，一个事故。

路由树。 匹配器的树形结构。首次匹配生效。

route:
  group_by: [alertname, cluster, service]
  receiver: default-slack
  routes:
  - match: { severity: page }
    receiver: pagerduty-primary
    continue: false
  - match: { team: payments }
    receiver: pagerduty-payments-team
  - match: { severity: ticket }
    receiver: jira-ticket-creator

continue: false 是默认值——首次匹配即终止。 continue: true 允许多路由（同时发送到 Slack 并呼叫）。

抑制。 "若父节点宕机，则不对子节点告警。"表达为 source_match: severity=critical AND alertname=ClusterDown 抑制 severity=warning AND alertname=PodDown 匹配标签的规则，阻止级联告警。

静默。 有时限的静音，通常用于计划维护或已知事故。UI 支持值班人员一键确认并静默。

10.3 告警疲劳及其应对方法

引入 SLO 之前的团队通常每个平台有 100–500 条告警规则。大多数会短暂触发后自行恢复，让值班人员养成忽视呼叫机的习惯。工程师不再阅读告警标题，下一次真正的事故因此被错过。

导致疲劳的反模式。 - 基于原因的告警。 CPU > 80%, queue_depth > 1000, disk_usage > 75%。这些告警针对的是症状，而非客户影响。在 CPU 弹性系统上，CPU 占用 80% 是正常的。 全服务统一静态阈值。 对支付服务（必须完美）而言的"高错误率"标准，与搜索建议服务（降级模式可接受）并不相同。 单窗口告警。 rate(errors[5m]) > 0.01 会在每次 5 分钟流量异常时抖动。 每个微条件单独告警。 对服务的每个 instance 分别设置告警，意味着某服务十个实例会触发十次呼叫。

应对疲劳。 - SLO 燃烧率告警。 第 9 节和 10.1 节。将 100 条基于原因的告警替换为每个服务约 5 条基于影响的告警。 整合仪表盘。 值班人员收到呼叫后，访问一个"服务健康"仪表盘，而非 14 个不同的仪表盘。"Datadog Service Page"模式：每个服务一个 URL，所有信号汇聚。 症状和剩余错误预算是一等公民。 呼叫页面包含"本周你还有 23 分钟错误预算"——明确紧迫程度。 自动解决。 告警触发、呼叫、值班人员开始排查；条件在第 4 分钟自行恢复。AlertManager 发送恢复通知——关闭呼叫，解散作战室。 事后复盘循环。 每条触发的告警都应定期审查。"这是否代表真实的用户影响？"若否——删除规则或调整阈值。 每团队呼叫预算。 "X 团队每周被呼叫超过 5 次 → 工程经理负责减少。"将噪声视为需管理的缺陷。

10.4 值班人机工程

值班轮换工具。 PagerDuty、Opsgenie、OnCall。按周/天轮换，设置升级链，支持"跟随太阳"的多地区轮换。

Runbook 规范。 每条告警注解都链接到 runbook。Runbook 应简短——"在仪表盘 X 中验证，若满足 Z 则重启服务 Y，否则升级至团队 A。"每次事故后更新。

告警确认。 首位响应者确认 → 升级取消。解决 → 呼叫关闭。跟踪确认响应时间指标。

事后复盘反馈循环。 每次呼叫产生一条事故记录。事后复盘将预防性工程工作提取到 backlog 中。优秀团队遵循"没有 runbook 就不设告警"的规则。

§11. 日志结构与解析

"日志只是文本"的思维模型在规模化场景下失效。没有结构化字段的自由文本日志会导致搜索延迟随数据量线性增长，整类调试工作变得不可能。结构化日志是使日志可查询的工程规范。

11.1 结构化与非结构化

非结构化（旧方式）：

2026-05-22 14:32:11 INFO  Payment processed for user alice@example.com
   in 234ms total=49.99 currency=USD trace=abc123

看起来还好。要搜索"alice@example.com 的所有支付记录"，用 grep。要计算"按货币划分的平均支付金额"，用 awk 解析。要与 trace 关联，用正则表达式提取 trace=。每次查询都变成一次解析练习。

结构化（现代方式）：

{
  "ts": "2026-05-22T14:32:11.234Z",
  "level": "INFO",
  "service": "payment",
  "trace_id": "abc123",
  "span_id": "def456",
  "user_id": "alice@example.com",
  "msg": "Payment processed",
  "duration_ms": 234,
  "amount": 49.99,
  "currency": "USD"
}

每个字段都有名称、类型且可索引。查询变为 WHERE user_id='alice@example.com' 或 WHERE service='payment' AND currency='USD' ——快速，无需解析。

11.2 必需字段（Schema）

生产级日志 schema 至少包含：

  • ts （RFC 3339 时间戳，UTC，毫秒精度）——时间范围查询必需。
  • level （DEBUG、INFO、WARN、ERROR、FATAL）——按严重级别过滤必需。
  • service （服务名称）——路由和按服务查询必需。
  • trace_id （W3C trace ID 十六进制）——trace-日志关联必需。没有它，无法从慢 trace 跳转到其日志。
  • span_id （当前 span）——span 级日志关联必需。
  • host / instance / pod ——主机级调试必需。
  • version （服务版本 / git SHA）——判断"这是否始于新发布"必需。
  • msg （人类可读的消息）——日志终究由人阅读，必需。
  • 领域特定结构化字段 — user_id, order_id, merchant_tier等。

许多团队使用 OpenTelemetry 语义约定来规范日志属性名称。 http.method 而非 method. db.statement 而非 query。跨服务的一致性是实现跨服务查询的前提。

11.3 "无法跨服务搜索用户 X"的痛点

没有结构化日志时，"查看过去一小时内所有服务中与 user_id=42 相关的一切"需要：

  • 了解每个服务的日志格式。
  • 以各自格式解析自由文本日志。
  • 手动合并结果。

有了结构化日志：

SELECT * FROM logs
WHERE user_id = '42'
  AND ts >= now() - INTERVAL '1 hour'
ORDER BY ts ASC

一条查询，亚秒完成。结构化与非结构化之间的差距，就是"5 分钟内能调试完"与"根本无法调试"的差距。

11.4 日志转指标流水线

部分日志隐式地编码了指标： Payment failed 行， Order completed 行， User churned 行。每分钟统计它们即可得到指标。

流水线： 日志 → 流处理器（Flink、Kafka Streams、Materialize、Vector）→ 计数器 → 推送至 Prometheus 或 TSDB。

{"level":"ERROR","service":"payment","error_code":"insufficient_funds",...}
    ↓ Vector / Fluent Bit
    aggregator counts by (service, error_code) in 1-minute windows
    ↓
push metric: payment_errors_total{service="payment",error_code="insufficient_funds"}

这样就得到了低基数指标（去掉 user_id，保留 error_code），可驱动仪表盘和告警，同时底层日志仍可用于取证搜索。两个支柱协同工作：日志中保留高基数细节，指标中保留低基数聚合，两者来自同一数据源。

11.5 解析遗留非结构化日志

对于无法重新埋点的服务（供应商二进制、老旧代码），在采集时解析。 Grok 模式 （带名称的正则表达式）： %{TIMESTAMP_ISO8601:ts} %{LOGLEVEL:level} %{GREEDYDATA:msg}。Fluent Bit / Logstash / Vector 均支持 grok 和采集时解析。即使源数据无结构，输出也是结构化的。

权衡：解析的 CPU 开销高且脆弱（格式漂移会静默破坏解析）。尽可能优先修复数据源。

§12. 指标基数规范

基数是指标中最大的成本杠杆，也是最大的故障模式。TSDB 的存储、内存和查询成本均与 不同时间序列的数量成正比，而非采样速率。控制基数的规范是区分健康指标平台与每周二 OOM 崩溃的平台的关键。

12.1 基数的具体含义

一个 时间序列 是指标名称与标签值的唯一组合。以下两个示例产生两条时间序列：

http_requests_total{service="api", method="GET", status="200"}   ← series 1
http_requests_total{service="api", method="GET", status="500"}   ← series 2

指标的基数 = 其所有标签基数的乘积（最坏情况）。

http_requests_total{service, method, status, endpoint, region}
                     50      5       8       100         5
cardinality = 50 × 5 × 8 × 100 × 5 = 1,000,000 series

再增加一个标签：

                     ... × user_id   (1M unique users)
cardinality = 1,000,000 × 1,000,000 = 1 trillion series

这就是基数爆炸——一个错误的标签使指标从"可接受"变为"不可行"。

12.2 每条时间序列的成本

每条活跃时间序列，TSDB 需承担： 内存中的 head block。 每个打开的 chunk（当前 2 小时窗口）约 1–2 KB。5000 万条时间序列需 50–100 GB 内存。 倒排索引大小。 每个 (label_name, label_value) 组合的倒排列表条目，随基数线性增长。 查询成本。 PromQL 查询 sum by(service) (rate(metric[5m])) 对 5000 万条时间序列的指标，会扫描 5000 万条时间序列，即使大多数为零。 WAL 字节数。 每次抓取向 WAL 写入 N 个样本，其中 N = 抓取到的时间序列数量。

基数是指标的 核心成本维度 ，采样率的影响远小于此。

12.3 "100 万用户"灾难

自我制造故障的配方：

  1. 工程师向 user_id 标签添加到 http_requests_total 以调试每用户级问题。
  2. 发布上线。此后每个请求都会为该 user_id 创建一条新的时间序列。
  3. 2 小时流量后，head block 新增了 100 万条时间序列。
  4. 内存增加约 1–2 GB。在单台高配节点上或许还好。
  5. 该指标被查询—— sum by(status) (rate(http_requests_total[5m])) ——现在需要扫描 100 万条时间序列，而非 1000 条。
  6. 查询耗时从 30 毫秒变为 30 秒。仪表盘超时。AlertManager 错过评估。
  7. 更糟糕的是：Prometheus head block 耗尽内存，被 OOM 终止。

平台团队被呼叫。原本要调试的功能无法进行，因为平台本身已起火。

12.4 基数控制技术

限制用户输入维度。 永远不要用原始 user_id、request_id、session_id、查询字符串作为标签，它们的基数无界。

聚合分桶。 替换 user_id （100万个值）与 user_tier （5个值： anon, free, basic, pro, enterprise）。相同的洞察（按层级切片）在5个序列而非100万个序列上实现。

布尔投影。 替换 customer_id 与 is_high_value_customer （0/1）。依然可操作；基数为2而非100万。

在摄入时按服务配额限制。 Mimir / Cortex / VictoriaMetrics 支持 per_user_series_limit ——拒绝某服务超过N个新序列。通知所属团队；保护平台。

基数 CI 检查。 在 CI 中，在合成负载下运行服务，统计每个指标的基数。若某指标的基数增长超过2倍或超过阈值，则构建失败。在提交时捕获基数回归。

基数仪表盘。 每个租户一个 Grafana 页面："您基数最高的20个指标，按序列数排序，附增长率。"工程师可以看到自己的占用情况并自我管控。

录制规则以预聚合。 若必须在源头采集高基数，则编写录制规则进行预聚合： sum by(service, tier) (http_requests_total) → 以较低基数的新派生指标形式存储。在仪表盘中使用派生指标；仅在 trace 级别/宽事件存储中保留原始指标。

12.5 "我们记录了 credit_card_count" 的轶事

一个真实（复合）故事：一位工程师添加了 credit_card_count 标签到某个指标，以调试"为什么这个账户有这么多卡"的问题。 credit_card_count 基数为1-30（大多数账户持有<30张卡），看起来安全。但该标签与 account_id （一百万个账户）在另一个指标上进行了联合——Prometheus 的查询优化器进行了扇出。RAM在8小时内增长了三倍。事后学习：基数在标签间是乘法的——那个"小"标签可能成为触发点。

纪律。 标签的添加须经过审查。"基数是多少？跨标签后的基数是多少？若流量增长10倍基数是多少？回滚计划是什么？"

§13. 跨异步边界的 Trace 上下文传播

Trace 能够工作，是因为每个 span 都携带 (trace_id, parent_span_id)。Trace 通过联接各个 span 来重建。 一旦上下文传播中断，trace 就会中断 ——下游 span 便成为没有父节点的孤儿。

13.1 W3C 标准

W3C Trace Context（https://www.w3.org/TR/trace-context/）定义了两个 HTTP 头：

traceparent: 00-0af7651916cd43dd8448eb211c80319c-b7ad6b7169203331-01
             │  │                                │                │
             │  │                                │                └─ flags (sampled=1)
             │  │                                └─ parent span_id (8 bytes hex)
             │  └─ trace_id (16 bytes hex)
             └─ version (00)

tracestate: vendor1=value1,vendor2=value2
            (vendor-specific key-value pairs, max 32 entries, max 512 bytes)

traceparent 是必需的。 tracestate 携带供应商特定的传播信息（Datadog 的 dd-id，AWS X-Ray 的 ID）。

每个 OTel SDK 在入站 HTTP/gRPC 时读取这些头，在出站时写入它们。数百个 HTTP 库已自动插桩以实现此功能。

13.2 Kafka 生产者/消费者的常见陷阱

朴素的 Kafka 用法。 生产者发送消息，消费者接收消息。消费者的 span 没有父节点——trace 在"produce"处结束，新的 trace 在"consume"处开始。两棵不相连的树，却对应同一个逻辑请求。

Kafka 的 OTel 模式。

生产者端：

with tracer.start_as_current_span("produce_to_kafka") as span:
    carrier = {}
    propagator.inject(carrier)   # writes traceparent into carrier dict
    headers = [(k, v.encode()) for k, v in carrier.items()]
    producer.send(topic, value=payload, headers=headers)

Kafka 消息有 headers （在 Kafka 0.11 中引入）。 traceparent 。 tracestate 以头的形式写入。

消费者端：

for message in consumer:
    headers_dict = {k: v.decode() for k, v in message.headers}
    ctx = propagator.extract(headers_dict)
    with tracer.start_as_current_span("consume_from_kafka",
                                       context=ctx,
                                       kind=SpanKind.CONSUMER,
                                       links=[Link(...)]):
        process_message(message)

消费者 span：- 从生产者的上下文继承 trace_id - 将生产者的 span 作为 Link，而非父节点（因为生产者的 span 已结束—— CHILD_OF 意味着父节点仍处于打开状态）。OTel 将此称为 FOLLOWS_FROM 语义；规范将其建模为 Link。- 设置 kind=CONSUMER 以便 span 在 trace UI 中显示为消费者。

现在：produce span 和 consume span 都在同一个 trace 中，父级链接正确，trace UI 将队列跳转显示为一条边，且队列等待时间可见。

13.3 其他异步边界

Espresso / Pinot CDC 流。 CDC 信封（消息格式）必须携带 trace_id。LinkedIn 的 CDC 框架添加了 _trace_context 字段；消费者像提取 Kafka 头一样提取它。

后台任务 / cron。 计划任务没有入站 trace 上下文——它开启一个新的 trace。但如果你想关联下游，trace_id 应可从触发元数据中发现。模式：调度器在触发时记录 (job_id, trace_id) ；任务以该 trace_id 创建一个 span。

进程内的异步任务（例如，goroutine、线程池）。 OTel SDK 使用线程本地/async-local 状态来跟踪当前上下文。将工作提交到线程池时 必须 显式传递上下文：

def submit_work(executor, fn, *args):
    ctx = context.get_current()
    executor.submit(context.attach, ctx)   # attach the parent context
    executor.submit(fn, *args)

或使用 OTel 的 context.attach 包装器。遗漏这一点是"异步代码中 span 没有父节点"最常见的原因。

WebSocket / Server-Sent Events。 长连接。每条消息在逻辑上都是一个独立的请求；生产者端在自定义协议字段中为每条消息包裹一个新的 traceparent 。目前尚无标准。

HTTP/2 push、gRPC 流式传输。 每个流有一个 traceparent。在该流中，每条消息都属于同一个 span，除非应用程序创建子 span。

13.4 检测与执行

在 trace 中检测传播断点。 一个包含N个服务但每个服务只有一个 span 的 trace 是可疑的——可能是传播中断了。Honeycomb 和 Datadog 会标记"broken trace"——根 span 存在，但下一个服务没有子 span。

传播的 CI 测试。 启动服务和一个下游 mock，使用已知的 trace_id 发送请求，断言下游收到了相同的 trace_id。在每个 PR 上运行。在上线前捕获"我们忘记接入传播"的回归。

OTel 自动插桩缩小了暴露面。 大多数 HTTP/gRPC 库都被社区插桩所包裹，因此传播是自动的。剩余的传播工作在自定义协议、Kafka 和异步任务边界处。

§14. Exemplar——打通各支柱的桥梁

一个指标数据点回答"有多少"，而不是"是哪一个"。当 p99 延迟在下午2:00飙升时，指标告诉你峰值发生了——exemplar 告诉你是哪个具体的 trace 导致的。

14.1 概念

一个 exemplar 是附加到指标数据点上的指针，链接到一个示例 trace。具体而言，exemplar 附加到 histogram 桶：对于 2s-5s 的 payment_duration_seconds桶，histogram 包含一个样本 trace_id，表示某个请求落入了该桶。

payment_duration_seconds_bucket{le="5"} 12345 # 12345 requests
   # exemplar: trace_id=abc123 value=4.2 timestamp=1716391200

指标存储仍然廉价（每个桶每个序列一个浮点数）；exemplar 在每个数据点上增加了几个字节（一个 trace_id 和一个值）。

14.2 跨支柱工作流

exemplar 使统一可观测性 UI 成为可能：

  1. 工程师打开 Grafana 仪表盘，看到下午2:00出现 p99 延迟峰值。
  2. 将鼠标悬停在 histogram 热力图上；看到峰值集中在 2s-5s 桶中。
  3. 点击图表上的 exemplar 点——链接到 Tempo / Jaeger 中的一个具体 trace。
  4. 打开 trace；识别出慢速 span（payment → external-bank, 4.2s).
  5. 点击该 span 上的"查看日志"——链接到按 trace_id 过滤的 Loki / 日志搜索。
  6. 读取日志；看到"external-bank 在重试4秒后返回了503"。
  7. 根因已定位。

链路 metric → trace → log 是跨支柱的枢纽。exemplar 使第3步成为可能——没有它，工程师不得不手动搜索"2:00 PM 附近持续时间 > 2s 的 trace"，这是大海捞针式的查询。

14.3 Prometheus exemplar 的工作原理

OpenMetrics 格式（Prometheus 文本格式的继任者） 添加了 exemplar 语法：

# TYPE payment_duration_seconds histogram
payment_duration_seconds_bucket{le="5"} 12345 # {trace_id="abc123"} 4.2 1716391200.000

注释位于 # 之后，携带 exemplar。

存储。 Prometheus 按序列将 exemplar 存储在一个循环缓冲区中——容量有限（默认每个序列约150 KB），最旧的 exemplar 会被驱逐。并非所有数据点都有 exemplar 存储；缓冲区只捕获最近的数据。

查询。 Grafana 的 Explore 视图将 exemplar 显示为叠加在图表上的点。点击某个点会触发对的 trace 存储查询。 trace_id=<exemplar>.

14.4 其他关联模式

日志 ↔ Traces。 每条日志行携带 trace_id。从 trace span 出发，按 trace_id 过滤查询日志。从日志行出发，点击 trace_id → 进入 trace UI。

日志 ↔ 指标。 日志转指标管道（§11.4）使日志成为指标的上游来源；将指标追溯到日志行是通过 source 标签实现的。

Traces ↔ 性能剖析。 Datadog 和 Pyroscope 正在试验"trace 关联剖析"——对于某个具体的慢 trace，获取在该 trace 期间采样的 CPU 剖析。识别在该特定请求期间哪个函数最耗资源。

完整图谱：每个支柱都通过 trace_id与其他每个支柱相互关联。平台的存在是为了让人类（和自动化）能够遍历该图谱。

§15. 日志和 Trace 中的 PII 脱敏

遥测数据中的 PII（个人可识别信息）是一个合规和伦理问题。法规（GDPR——通用数据保护条例、HIPAA——健康保险可携带性和责任法案、PCI DSS——支付卡行业数据安全标准、CCPA——加州消费者隐私法案）对可存储的内容及其存储时长施加了严格限制。PII 脱敏纪律是任何成熟可观测性平台的结构性特征。

15.1 什么算作 PII

直接标识符：全名、电子邮件、电话号码、SSN（社会安全号码）、信用卡号、政府ID、IP地址、地理位置、设备ID、生物特征数据。

间接标识符：能唯一标识个人的组合（邮编 + 出生日期 + 性别；跨会话稳定的 user_id；Cookie ID）。

GDPR 下的特殊类别：健康数据、种族/民族来源、宗教信仰、性取向、生物特征。

HIPAA（美国医疗）下：18种特定标识符，包括病历号、IP地址、比年份更精确的日期。

PCI DSS 下：完整信用卡号（PAN）、磁条数据、PIN——绝不存储在任何地方，包括日志。

15.2 PII 在可观测性中的泄漏位置

  • 日志。 最常见的泄漏。调试日志 f"received request with body: {request.body}" 转储所有内容——包括电子邮件、电话号码，可能还有信用卡。
  • Traces / span 属性。 db.statement="SELECT * FROM users WHERE email='alice@example.com'" 在 span 中捕获电子邮件。 http.url="https://api/users?email=alice@example.com" 通过 URL 捕获它。
  • 指标。 不太常见（基数纪律通常会排除 PII 标签），但有可能。 user_email 作为标签既是基数炸弹，也是 PII。
  • 错误堆栈跟踪/异常消息。 "TypeError: cannot serialize User(email='alice@example.com')"——错误消息包含 PII。

15.3 "我们记录了信用卡号"事故类型

该类型的真实事故：- 生产环境中意外启用的调试日志打印请求体，在发现前数小时内捕获了完整的信用卡号。- 配置错误的 Lambda 的 stderr 流记录了包含 SSN 的 API 响应。- 异常处理程序记录了整个输入对象，包括未脱敏的 PII。

一旦 PII 进入日志存储，就存在于：每个备份、每个副本、每个冷存储 Parquet 文件中。被遗忘权（GDPR 第17条）赋予用户要求删除的权利——这需要扫描和重写数 PB 的日志以删除其记录。在规模化时成本高昂且往往在技术上不可行。

15.4 在 Collector 层进行脱敏

规范模式： 在持久存储之前在 Collector 进行脱敏，从而使 PII 从一开始就不会落入日志/trace 中。

OTel Collector 的 attributes 处理器支持 hash, delete，以及 update 操作：

processors:
  attributes:
    actions:
    - key: user.email
      action: hash      # SHA256 the value; consistent for joins, no PII visible
    - key: user.ssn
      action: delete
    - key: credit_card.number
      action: delete
    - key: http.url
      action: update
      value: <redacted>
      pattern: "(\\?|&)email=[^&]+"

对于自由文本字段（日志消息、异常字符串），基于正则表达式的脱敏器在 Collector 处运行。检测模式并替换：

  • 信用卡： \b\d{4}[- ]?\d{4}[- ]?\d{4}[- ]?\d{4}\b → <REDACTED_CC>
  • 美国 SSN： \b\d{3}-\d{2}-\d{4}\b → <REDACTED_SSN>
  • 电子邮件： \b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}\b → <REDACTED_EMAIL> （有时改为哈希，以保持按用户的关联性）。
  • IP 地址：对于分析，脱敏至 /24 子网；对于 PCI，完全脱敏。

工具：Vector、Fluent Bit、Cribl、自定义 Collector 处理器。部分团队使用基于 ML 的 PII 检测（Google DLP API、AWS Macie）来处理更复杂的模式。

15.5 被遗忘权的影响

GDPR 第17条要求在收到请求时删除用户数据。对于可观测性：

  • 日志。 识别所有跨存储层引用 user_id=X 的日志，包括 S3 冷存储和 Glacier。重写以清除。成本高昂，但对于结构化日志是可处理的。
  • 链路追踪。 识别所有属性包含 user.id=X的 span。重写。以 trace_id 为键的存储有助于此操作。
  • 指标。 如果基数纪律将 user_id 排除在标签之外，则基本安全。否则，有时间限制的保留期有所帮助。

架构缓解。 从一开始就不存储 PII——在可观测性数据中使用哈希/假名化的 user_id，将哈希 → 真实ID 的映射存储在支持正确删除的独立 PII 分级系统中。可观测性存储只处理哈希；被遗忘权在哈希映射层处理。

15.6 最低可行 PII 纪律

  • 默认拒绝：生产环境中不使用调试日志，每个 Collector 管道中都有脱敏处理器。
  • Schema 包含 PII 层级：每个日志/span 字段标记 pii=none|low|high|forbidden。禁止的层级在 Collector 处被拒绝。
  • 自动扫描遥测数据中的 PII 模式；对发现结果告警。
  • 对包含个人数据的任何日志，保留层级更短（高 PII 日志最多30天）。
  • PII 字段查询者的审计日志（元元可观测性）。

§16. 保留层级与经济性

遥测存储成本主导可观测性成本。在任何规模下，天真的"热存储所有内容90天"方法都是一个数百万美元的错误。保留层级——数据存储在哪里、查询速度有多快、成本是多少——是核心经济设计。

16.1 层级模型

典型的四层模型：

层级 Medium 查询延迟 典型保留期 $/GB/月（2026年，粗略估算）
热层 Prometheus / Loki / ELK / Tempo 上的 SSD < 1秒 7-30天 $0.20-0.50
温层 远程集群上的 SSD/HDD，S3 缓存 1-10秒 1-3个月 $0.05-0.15
冷层 S3 Standard / Parquet，延迟加载 10秒-分钟级 6-12个月 $0.023
归档层 S3 Glacier / Deep Archive，需要恢复 小时-天级 若干年（合规） $0.001-0.004

成本比率： 热层与归档层之间约200倍差距。在分层系统上满足7年合规保留要求，比同等全热存储保留便宜两个数量级。

16.2 各层存储什么

  • 热层。 活跃运营数据——最近7-30天，用于事故响应、仪表盘、告警。查询频率：每天数百到数万次。Loki 最近的 chunk 在 SSD 上，Prometheus head 及近期块，ELK 活跃索引，Tempo 最近几天的数据。
  • 温层。 1至3个月。用于事后分析、每周复盘、容量规划。查询频率：每天10至100次。使用低成本 SSD 上的压缩 Parquet 格式，或带缓存层的 S3。
  • Cold。 6至12个月。合规需求、偶尔的取证分析、历史趋势分析。查询频率：每天1至10次，对延迟容忍度较高。S3 Standard，上层搭配 Athena / Trino / DuckDB。
  • Archive。 数年。纯合规与监管保留需求。查询频率：接近零，法律取证场景除外。S3 Glacier Deep Archive——恢复需要数小时。

16.3 "全热存储"的经济灾难

一个场景：

  • 每天摄取 10 PB 日志。
  • 热数据保留 30 天，"因为这样比较安全"。
  • 热 SSD 存储需 300 PB。
  • 以 $0.20/GB/月计算，这意味着 $60M/月.

仅热日志一项。现实中，那些没有及早分层的公司，实际数字就在这个量级。

与分层模型对比：

  • 热存储 3 天（约 30 PB，$6M/月）。
  • 温存储 30 天（约 300 PB，$0.05/GB，$15M/月）。
  • 冷存储 1 年（3.6 EB，$0.023/GB，$83M/月——主要因为长尾数据；通过采样或在写入冷层前聚合，可减少约一半）。

逐层比较，"90 天全热"的成本大致等于"全分层保留 1 年"——而分层方案能实现 4 倍的保留时长。

16.4 各层的查询模式

热层查询。 仪表盘每 10 秒刷新，告警每 30 秒评估一次，故障响应查询覆盖最近 1 小时内 1 至 10 分钟的窗口。所有查询均在亚秒级完成。存储格式需大量使用索引。

温层查询。 事后分析，例如"上周二系统的状态是什么？"可接受延迟：5 至 30 秒。S3 上的 Parquet 格式配合 bloom filter；每次查询扫描的字节数更多，但每天查询次数少，因此扫描字节数的成本有上限。

冷层查询。 合规取证，例如"给我用户 X 过去 6 个月的所有操作记录"。对延迟容忍度高，可能需要数分钟。使用 Athena/Trino 扫描 Parquet，数据量达到 GB 级别。

归档层查询。 从 Glacier 恢复（需数小时），然后再查询。对已知法律留存案例可提前预热。极少使用。

16.5 层间迁移与降采样

数据随时间向下流转：

Hot (7 days) → Warm (3 months) → Cold (1 year) → Archive (7 years)

在每次迁移时，对数据进行转换以降低在下一层的存储成本。

  • Hot → Warm。 将指标的原始采样点聚合为 1 分钟分辨率（降采样）。对日志块进行更激进的压缩。
  • Warm → Cold。 聚合至 1 小时分辨率。丢弃 debug 级别日志（已超出其有效使用窗口）。将行式存储转换为列式 Parquet。
  • Cold → Archive。 聚合至按天粒度。迁移至 Glacier Deep Archive。

Mimir 的"长期存储"使用一系列压缩块：2h → 12h → 1d，在天边界处进行降采样。

16.6 "仓库中的幽灵"问题

一个常见错误：以完整精度摄入热层，然后迁移至温层/冷层， 无需 再进行降采样。数据依然存在，但处于查询成本高昂的层（扫描 S3 上的 Parquet），而且从不被查询——因为温层/冷层的使用场景是聚合趋势，而非原始事件。结果是：PB 级冷存储，查询次数为零——纯粹是成本浪费。

纪律。 对每一层，明确其查询使用场景。如果某层没有对应的查询场景，数据就不应该存储在那里。要么删除，要么迁移至归档层（仅用于合规）。

§17. 可观测性平台的多租户架构

平台团队负责可观测性基础栈，每个产品团队负责自身服务及其产生的遥测数据。多租户架构使这一模式得以运转——某个团队的糟糕发布不会影响其他团队的可见性。

17.1 平台团队与产品团队的职责划分

契约如下：

  • 平台团队负责： 可观测性基础栈（Prometheus/Mimir、Loki、Tempo、Grafana、Collectors、Kafka、AlertManager）。包括可靠性、扩展性、多租户隔离和成本控制。
  • 产品团队负责： 其服务的埋点、仪表盘、告警和 runbook。他们使用平台，但不运维平台。

如果没有明确的多租户机制，一个"嘈杂"的产品团队会消耗共享资源——高基数、高日志量、失控的 trace——从而影响所有其他团队的体验。有了多租户机制，"嘈杂"团队的影响就被限制在其自身租户内。

17.2 租户隔离原语

按租户配置摄入配额。 Mimir 的 per_user_series_limit, ingestion_rate_limit_mb, max_global_series_per_metric_name。Loki 有等价的按租户限流机制。当某租户超出配额时，平台拒绝其 metrics/logs——保护平台，同时告警通知该租户。

按租户配置存储配额。 每个租户的热/温/冷存储有上限。存储满后，租户必须删除数据（或付费扩容）。

按租户配置数据保留策略。 租户 A 保留 90 天（默认），租户 B 保留 30 天（自主选择/成本决策），租户 C 保留 7 年（合规需求——为此付费）。按租户独立配置。

按租户配置查询隔离。 某租户的查询负载不会挤占其他租户。按租户设置查询并发限制，以及按租户配置查询预算（每分钟 CPU 秒数）。即"查询面上的嘈杂邻居"问题。

按租户配置认证与授权。 租户只能看到自己的数据。Grafana 的租户感知数据源、基于 header 的租户识别（X-Scope-OrgID 在 Mimir/Loki 中），以及面向仪表盘/告警的 RBAC。

17.3 租户识别

平台如何判断某条遥测数据属于哪个租户？

  • 通过服务身份。 服务名 → 租户映射（来自服务目录）。OTel Collector 根据服务添加 tenant_id 基于服务的 resource 属性。
  • 通过 Kubernetes namespace。 一个 namespace = 一个团队 = 一个租户。
  • 通过显式 header。 应用在 OTLP 请求中携带 X-Tenant-ID: payments ；Collector 根据 header 进行路由。
  • 通过集群。 每个租户一个 Kubernetes 集群。隔离性强，但规模化时成本高。

LinkedIn 的做法：服务目录将服务映射到归属团队，再映射到租户。平台组件在接收数据时查询服务目录。

17.4 成本回溯模型

为了建立正确的激励机制， 将成本分配给各租户:

  • 计算每个租户在平台成本中的占比（摄入字节数、存储字节数、查询 CPU 用量）。
  • 内部计费（chargeback）或可视化追踪（showback）。
  • 高成本租户要么降低数据量，要么为此付费。

如果没有成本回溯，每个租户都有动力过度埋点和过度保留——将成本外部化到平台团队的预算中。

17.5 难点所在

共享存储上的热点租户。 即使设置了配额，查询模式也可能不均匀。单个租户的高成本 PromQL 查询可能会阻塞队列中其他租户。按租户的队列隔离有所帮助；完全隔离只能通过为每个租户提供专属基础设施来实现（成本高昂）。

跨租户查询。 某些合理的使用场景需要跨租户访问（安全审计、平台团队故障响应）。需要特殊的"超级租户"角色，并进行审计记录。

新租户接入。 自助式接入很困难。大多数平台采用手动审批流程，在授予租户配额之前进行成本预测。

配额随时间调整。 租户合理增长超出其配额。配额变更属于变更管理范畴——不能让平台团队的容量规划感到意外。

§18. 故障管理集成

可观测性是故障管理运行的基础底座。可观测性与故障管理工具之间的集成，决定了故障响应是一个拼凑而成的手动流程，还是一个协调有序的工作流。

18.1 故障生命周期及可观测性的介入点

1. Detection      ← alert fires from observability
2. Triage         ← oncall investigates; uses observability dashboards
3. Notification   ← incident-mgmt tool notifies stakeholders, escalates
4. Mitigation     ← team finds root cause via observability, mitigates
5. Resolution     ← alert auto-resolves; observability confirms recovery
6. Post-mortem    ← assemble timeline from observability artifacts

每个步骤都有特定的可观测性集成点。

18.2 工具生态

告警通知： PagerDuty、Opsgenie、VictorOps。接收来自 AlertManager（或同类工具）的告警，轮转 oncall 值班，若无响应则进行升级。

故障管理： PagerDuty Incident Response、incident.io、Rootly、FireHydrant、Blameless。将故障作为实体进行跟踪——干系人、严重级别、时间线、缓解措施、事后复盘。

ChatOps： Slack、MS Teams。托管为故障自动创建的作战室频道。

状态页面： Statuspage、status.io、内部自建。面向客户的公开通告。

复盘工具： Notion、Confluence、Blameless、FireHydrant。

18.3 从告警自动创建故障

最关键的集成点。当严重告警触发时：

  1. AlertManager → webhook 发送至故障管理工具。
  2. 故障管理工具依据告警标签中的严重级别创建故障记录。
  3. 创建专属 Slack 频道（#incident-<id>-<short-summary>).
  4. 邀请 oncall 值班人员以及自动识别到的干系人（服务 owner、依赖服务 owner）。
  5. 推送告警详情 + 仪表盘链接 + 相关 trace/log 链接。
  6. 可选：自动创建 JIRA ticket，起草状态页面。

告警触发后 30 秒内，作战室已就绪，相关人员已被通知，可观测性数据的链接已出现在聊天中。

18.4 将 trace 和日志关联到故障

故障记录应包含：

  • 仪表盘链接 展示触发违规的 SLI。
  • 具体 trace 链接 用于说明故障现象（通过 exemplar）。
  • 已保存的日志查询 过滤至故障时间窗口和相关服务。
  • 性能 profile 差异对比 如果启用了持续 profiling 且可以识别出性能回归。

这些不是截图——而是随故障演进实时更新的活跃链接。PagerDuty 和 incident.io 都支持将可观测性链接附加到故障记录中。

18.5 从可观测性产物中自动生成事后复盘时间线

故障解决后，需要撰写事后复盘。时间线部分过去通常需要从聊天记录和记忆中手动整理。现代集成可自动组装：

  • 告警触发/恢复事件 来自 AlertManager。
  • 发布事件 来自 CI/CD（Spinnaker、GitHub Actions）。
  • 聊天消息 来自作战室 Slack 频道。
  • 缓解操作 由响应人员记录（例如"已回滚提交 abc"）。
  • 恢复信号 来自可观测性数据（SLI 恢复至 SLO 范围内）。

Blameless、FireHydrant 和 incident.io 均可自动组装时间线。响应人员进行标注，事后复盘文档可在数分钟内（而非数天）准备完毕。

18.6 "此次故障是否影响了客户 SLA？"的问题

一旦 SLO/SLI/SLA 追踪机制就位，故障的客户影响即可直接从可观测性数据中计算得出：

  • "故障期间 SLI 的表现如何？"
  • "哪些客户的 SLA 遭到违反？"
  • "应退款/补偿多少？"

故障的财务后续处理，是从检测到故障的同一套遥测数据中计算得出的。SLA 管理从此变得数据驱动，而不再是"凭感觉给予善意补偿"。

§19. 移动端与前端可观测性

服务端可观测性占据了大多数文献的篇幅，但真正影响用户体验的是用户侧（浏览器、移动端应用）。前端可观测性栈在结构上与后端栈截然不同——不同的 SDK、不同的存储结构、不同的故障模式。

19.1 前端可观测性捕获的内容

  • JavaScript 错误 / 未处理的异常。 堆栈跟踪、浏览器、操作系统、URL、错误发生前的用户操作（breadcrumb）。
  • 移动端崩溃。 原生堆栈跟踪（通常在服务端符号化）、设备信息、操作系统版本、应用版本。
  • 前端性能。 首字节时间（TTFB）、可交互时间（TTI）、最大内容绘制（LCP）、帧率（FPS）、JavaScript 长任务、布局偏移（CLS — Cumulative Layout Shift）。
  • 用户操作。 点击、路由切换、表单提交、A/B 测试变体分配。
  • 前端发起的网络请求。 每条 XHR / fetch 请求，附带耗时、状态码、请求头。
  • 后端关联。 前端 HTTP 请求携带 trace_id；后端延续该 trace。这样，"用户反映页面慢"就可以进行端到端诊断。

19.2 主要产品

Sentry。 开源错误追踪，提供托管版本。支持 JS / Python / Ruby / Java / 移动端错误捕获；对堆栈进行符号化；按签名分组；分配给 git-blame 对应的作者。入门级约 $26/月，按事件量计费扩展。存储基于 Snuba（基于 ClickHouse 的宽事件存储）。

Crashlytics（Google / Firebase）。 适用于 iOS 和 Android 的移动端崩溃分析工具。免费。移动端崩溃上报的行业标准。通过上传 dSYM（iOS）/ ProGuard 映射文件（Android）进行符号化。

Datadog RUM（Real User Monitoring）。 前端 SDK，捕获性能、错误、用户会话以及延续至后端的分布式 trace。使用同一套 Datadog 后端，在统一 UI 中实现跨栈关联。

New Relic Browser、Dynatrace Real User Monitoring、AppDynamics EUM。 各厂商的等价产品。

LogRocket、FullStory。 会话回放——录制并回放用户会话（点击、导航，甚至 DOM 快照）。满足"我想看到用户当时看到的内容"这一使用场景。

OpenTelemetry JS / Mobile SDK。 OTel 提供面向浏览器（埋点 fetch / XHR / DOM 事件）和移动端（Android / iOS）的客户端 SDK。这是厂商中立的路径；截至 2026 年，相较于成熟的厂商 SDK，仍在持续演进中。

19.3 与后端可观测性的差异

采样策略不同。 后端以 1% 进行采样，因为流量大且均匀。前端事件具有突发性（一个爆款页面 → 流量激增 100 倍）。能够自动调整的自适应采样更为重要。错误通常保持 100% 采样（数量少但价值高）。

符号化。 移动端崩溃堆栈跟踪是内存地址的转储。服务端符号存储（iOS 的 dSYM、Android 的 ProGuard 映射）将其转换为函数名。"我们发布了一个符号文件损坏的版本"这一故障模式会导致该版本的所有崩溃报告全部无效。

隐私。 会话回放会捕获用户操作——可能包含表单字段值、PII。前端 SDK 需要可配置的脱敏能力（将表单字段标记为"不记录"）。

网络不稳定性。 移动端 SDK 在离线时缓冲事件，重新联网时重试上传。存在"用户在飞机上发生崩溃，崩溃报告 8 小时后才上传"的情况。

版本管理。 移动应用在线上同时存在 N 个版本。后端只有一个版本。崩溃报告必须按版本分区，且符号存储必须保留所有已发布版本的符号文件。

19.4 良好的前端可观测性能告诉团队什么

  • "v3.2.1 中按用户影响排名的前 10 大崩溃。" 不仅仅看次数——还要按受影响用户数加权。一个设备抖动的单个用户产生 1000 次崩溃；而影响 0.1% 用户的一个 bug 更为重要。
  • "按路由划分的可交互时间 p75。" 按路由统计页面加载性能，驱动优化优先级排序。
  • "过去一小时内，5% 点击了结算按钮的用户遇到了 JS 错误。" 功能性影响，而不仅仅是异常次数。
  • "这条 trace 展示了用户请求从前端穿越 8 个后端服务的全链路。" 跨栈的因果链路。

19.5 前端特有的故障模式

符号文件不匹配。 工程师为 v3.2.0 上传了符号文件；生产用户使用的是 v3.2.1。所有 v3.2.1 的崩溃均无法符号化——显示为 0x7fff8c4d3a20 而非 MyApp.PaymentScreen.confirm()。修复方案：在 CI/CD 流程中为每个发布版本自动上传符号文件。

广告拦截器。 浏览器广告拦截器会屏蔽分析域名的请求，包括部分可观测性 SDK（Datadog RUM、使用明显域名托管的 Sentry）。解决方案：通过自有域名进行代理；接受一定程度的采样偏差。

移动端的 SDK 开销。 电池和带宽都是稀缺资源。SDK 必须积极批量处理、推迟到 Wi-Fi 时传输、进行限流。

来自第三方 SDK 的误报崩溃信号。 某个有缺陷的广告 SDK 抛出未处理的异常；宿主应用上报一个"崩溃"，但实际上并不是应用本身的问题。需要具备调用栈感知能力的过滤机制。

§20. 容量规模

本节的核心要点在于，可观测性在规模上跨越了约 六个数量级 的范围，而引擎的选择取决于你所处的层级。

小型 — 初创企业、单一产品线。 10-50 个应用容器；约 30K 活跃序列，约 3K samples/sec；约 10 MB/sec、约 1 TB/day 日志；约 1K spans/sec（头部采样）。单个 Prometheus、单个 ELK、Jaeger 跑在 Cassandra 上，部署在少量 m5.large 机器上。预算 < $5K/月。一名工程师兼职维护。瓶颈：单个 ELK 节点在持续 10 MB/sec 时触及 I/O 上限。

中型 — Grafana Cloud 客户、中型 SaaS 公司。 1-5K 容器；3-15M 活跃序列，300K-1.5M samples/sec；约 1 GB/sec、约 85 TB/day 日志；100-500K spans/sec（后采样）。Mimir / Cortex 处理指标并进行分片，Loki 处理日志，Tempo 处理链路追踪，冷层使用对象存储。2-4 人团队。瓶颈：单个 Prometheus 在约 5-10M 活跃序列时触及基数上限；通过 Mimir 进行分片。

大型 — LinkedIn、Datadog 客户、中等规模超大规模云服务商。 30-100K 容器；50-500M 活跃序列，5-50M samples/sec； LinkedIn 全集群日志摄入量约 10 PB/day，持续带宽约 115 GB/sec；后采样 1-10M spans/sec，每日数十亿条。Mimir 分片，多集群 Loki + Pinot 处理日志，多地域 Tempo 加尾部采样。专属团队 10-30 人。瓶颈：如果没有为每个支柱单独划分 Kafka 集群，Kafka 就会成为瓶颈；单个 Kafka 集群处理 6 GB/sec 的日志尚可，但日志与链路追踪共用同一集群会引发背压级联。

超大型 — Google、Meta。 - Google Monarch（VLDB 2020）： 内存全局多副本 TSDB，分层区域架构， 峰值摄入量约 1 万亿 samples/sec；替代了 Borgmon。\n- Datadog： 公开披露 数万亿个指标点每天 遍布其客户聚合平台——按 10 万亿/天计算，约为 ~115M points/sec。\n- Cloudflare 每天记录约 1 万亿条日志，分析依托 ClickHouse 进行。\n- Meta 的 Scuba（VLDB 2013）： PB 级的内存宽事件存储，对数十亿事件的聚合查询能在亚秒级完成。

在这个层级，问题不再是"能否存储"，而是"是否负担得起持续保留"。留存成本远超摄入成本；积极的降采样和分层冷存储是结构性需求，而非可选项。

§22. 可观测性的固有难题

以下六大基本挑战是任何部署该技术的人都会面临的，与产品选型无关。每项均以不同领域为例加以说明。

22.1 基数爆炸（摧毁指标系统）

领域：支付故障排查。 支付团队想要"按商户统计 p99 延迟"。他们将 merchant_id 作为标签添加到 payment_duration_seconds。500 万活跃商户 × status × method × region → 基数从 50K 飙升至 5000 万。Prometheus 的 head block 达到 50 GB 并发生 OOM。平台告警触发，而非原始故障本身。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 增加 RAM。能争取到几天时间，但解决不了月级问题——基数是无上界的。

真正的解决方案。 (1) 在摄入端进行基数管控 ——TSDB 追踪每个租户的活跃序列数；超过预算（例如每个服务 100K 序列）时拒绝新序列并告警给负责人。Mimir 提供了 per_user_series_limit. (2) 聚合规则 ——在流式处理器（Flink、Materialize）中对高基数维度进行预聚合，将 top-100 商户作为指标存储，其余数据推入宽事件存储。(3) 链路追踪才是按实体切片的正确答案 ——"某商户的 p99"是一个 trace/宽事件查询，而不是指标查询。(4) 基数告警 ——当序列数在 24 小时内增长超过 2 倍时告警；在 OOM 发生前捕获错误部署。

22.2 全量日志索引的成本

领域：安全与审计日志。 安全团队需要对所有认证事件进行全文 grep："这个攻击者 IP 在 90 天内有没有出现过？"他们将 100% 的日志推入 Elasticsearch 并建立全文索引。原始 50 TB/day → 索引后 100 TB/day。ES 集群扩展到 1500 个节点。成本超过平台其他所有部分的总和。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 缩短留存时间。有帮助，但无法解决结构性问题——他们为 99% 从未被查询的数据付费建索引。

真正的解决方案。 按查询模式分层。 最近 7 天存入 ELK 以支持快速 grep（约 3 TB 热数据）。8-90 天存入 Loki + S3 分块——grep 较慢（10-30 秒），但对于取证场景可以接受。90 天以上存储为 S3 上的 Parquet，通过 Athena 查询。大多数安全工作流对较旧数据的秒级到分钟级 grep 是可以接受的；快速索引层只需覆盖活跃调查窗口即可。

22.3 头部采样 vs 尾部采样的取舍——选错的代价

领域：ML 模型推理服务。 ML 团队在排查"为何推荐模型偶尔对 0.1% 的请求返回空结果"。采用 1% 头部采样。1 万个受影响的请求中，仅采样了 100 条——这 100 条没有一条命中 bug。团队对问题一无所知。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 将头部采样提升到 50%。数据量增加 50 倍；存储和 Kafka 成本急剧膨胀。存储了大量完全无聊的 trace。

真正的解决方案。 尾部采样。 所有 span 送入 collector。按 trace_id 缓冲 30 秒。保留：100% 错误、100% 慢请求、100% 命中 recommendation_count == 0的请求，再加 1% 随机基线。存储量约为旧头部采样率的 1.5 倍，"有价值"的 trace 覆盖率达 100%。代价：collector 层必须按 trace_id 分区，且需要 RAM 来维持缓冲窗口。在 60M spans/sec 的速率下，30 秒缓冲 ≈ 18 亿条 span，约需聚合 2-4 TB RAM。

22.4 跨异步边界的 trace 上下文传播

领域：游戏服务器崩溃排查。 一位实时运营工程师在排查"用户的道具购买未到账"。调用链：HTTP → Kafka → 异步 worker → Espresso 写入 → CDC 流 → 账本对账。六个异步跳转。trace 在"请求进入 Kafka"处终止；所有下游 span 都成了孤儿。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 在 Kafka 消息体中携带 trace_id。只做了一半——丢失了 span 层级关系；SDK 无法将消费者 span 关联为子节点。

真正的解决方案。 按 OTel 规范在每个边界传播 trace 上下文。 HTTP/gRPC： traceparent header（W3C 标准）。Kafka： traceparent 。 tracestate 作为 Kafka 消息 headers。Producer SDK 在发送时附加；consumer SDK 读取并以 parent_span_id = producer's span_id、链接类型 FOLLOWS_FROM （而非 CHILD_OF ——父 span 已完成）创建 span。Espresso/Pinot CDC：在 CDC 信封中携带 trace 上下文。定时任务（cron、Flink、batch）：调度器创建根 span；任务从触发元数据中继承 trace_id。

22.5 告警疲劳

领域：任何 SRE 值班轮换。 约 200 条活跃告警规则。每周触发 40 次，大多数是在 90 秒内自行恢复的瞬时抖动。工程师停止阅读告警。下一次真实故障沉默 20 分钟才被发现。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 收紧告警阈值。能降低噪音，但无法解决结构性问题——对原因告警（CPU > 80%、队列深度 > 1000）意味着对那些不一定影响用户的症状告警。

真正的解决方案。 基于 SLO 的多窗口燃烧率告警。 定义用户痛点："滚动 28 天内 99.9% 的请求返回 2xx-3xx 且 <500ms"。计算错误预算。在 1 小时内以 14.4 倍燃烧率（一天耗尽一个月的预算）时触发告警，6 小时内 6 倍（4 天耗尽）时告警，24 小时内 3 倍时创建工单。每个服务约 5 条告警，而非 200 条；每条都代表真实的用户影响。基于原因的告警转为工单 → 仪表盘。 当用户真正受损时告警，而非某个指标偏高时告警 ——这是 SRE 原则在告警实践中的应用。

22.6 有偏差的采样

领域：移动应用崩溃分析。 Crashlytics/Sentry 以 device_id mod 10为键进行 10% 采样。数月后团队发现 iOS device_id 分布存在偏斜（基于时间戳生成的种子与启动时间相关）。这 10% 的样本现在偏向 Android。崩溃趋势显示"Android 崩溃率是 iOS 的 3 倍"，而实际两者相当——iOS 样本被系统性地丢失了。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 按平台分层采样。只解决了一个症状。

真正的解决方案。 采样键必须均匀分布。 W3C Trace Context 推荐在 trace_id 上使用 128 位随机数；使用密码学质量的 RNG，而非基于时间戳的方案。 监控偏差 ——每分区 span 速率、每平台采样率、每地域采样率。任何类别明显高于其他类别，说明哈希函数或采样键存在问题。这是可观测性平台自身的可观测性。

§23. 故障模式

平台本身也是一个分布式系统，会发生故障。以下是系统性梳理。

Collector 背压

CloudTrail + S3 Access Logs。 OTel Collector 在尾部采样时 CPU 被打满。span 摄入出现滞后。Kafka 队列堆积。

恢复措施。

  1. 在 Collector 上设置熔断器：当 CPU > 80% 持续超过 1 分钟时，切换到降级模式—— 丢弃随机采样部分 ，但保留错误/慢速 trace。在保留高价值 trace 的同时，将摄入量削减约 80%。
  2. Kafka 分层存储——磁盘满后，旧数据段卸载到 S3。trace 仍可检索，但延迟更高。
  3. 基于 Kafka 消费者积压或 CPU 对 Collector 层进行自动扩缩容。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 trace 仍在 Kafka 中。最坏情况是 trace 组装延迟，而非数据丢失。

存储层满

CloudTrail + S3 Access Logs。 由于某人部署了一个新标签，导致单个指标爆炸为 5000 万条序列，Prometheus 磁盘在一夜之间打满。

恢复措施。

  1. 存储硬限制——Prometheus 停止接受新样本（它并不会 非 崩溃）。旧数据仍可查询。
  2. 基数监控在数分钟内捕获到爆炸；告警通知负责服务的团队。
  3. 回滚服务或热修复有问题的标签。
  4. 基数恢复正常后，下一个 2h block 边界会清空 head block；磁盘压力随之消解。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 已持久化的 2h block 已落盘并复制到 S3（Thanos/Mimir）。head block 可从 WAL 重建。

有偏差的采样静默丢弃

CloudTrail + S3 Access Logs。 Trace_id 分布偏斜 → 某个 Kafka 分区负载是正常的 10 倍 → 缓冲区溢出 → 该分区的 trace 被优先丢弃。特定地域的 trace 被系统性地丢失。

恢复措施。 无法实时恢复——过去的 trace 已丢失。前向修复：通过每分区 span 速率告警主动检测偏差，如果哈希函数损坏则重新分片。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 无法恢复已丢弃的 trace。通过不变量 4（采样决策可审计）和主动偏差检测来预防。

日志中缺失 trace_id

CloudTrail + S3 Access Logs。 一个新服务在未配置 OTel logger handler 的情况下部署上线。其日志不携带 trace_id。在故障排查时，无法从 trace 跳转到该服务的日志。

恢复措施。

  1. 通过 CI 检查进行检测：每个服务的结构化日志格式必须包含 trace_id 字段。不满足条件的部署应被拒绝。
  2. 对于正在进行的故障，回退到时间范围 + 服务名称过滤器。噪音更大，但可用。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 这是一个 SDLC（软件开发生命周期）不变量，而非运行时不变量。修复应在部署流水线中进行。

多地域分区

CloudTrail + S3 Access Logs。 us-east-1 与位于 us-west-2 的中央可观测性平台失去连接，持续 20 分钟。

恢复措施。

  1. 每个地域的 OTel Collector 拥有本地 Kafka 缓冲。分区期间持续缓冲数据。
  2. 每个地域的 Prometheus 实例持续在本地 scrape；地域级 Grafana 查询保持可用。
  3. 链路恢复后，Collector 冲刷积压数据。指标存储进行回填（Thanos sidecar 接受乱序 block）。
  4. 分区期间触发的告警由 地域级 Alertmanager 通过备用告警渠道（通过不同网络路径的 PagerDuty fallback）发出。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 采用每地域独立存储。系统设计为按地域降级，而非要求全局一致视图。

Collector 背压级联影响应用

CloudTrail + S3 Access Logs。 OTel Collector 导出器（向 Tempo 发送 span）因 Tempo 过载而阻塞。Collector 内部队列填满。在默认的 OTLP exporter 阻塞行为下， 应用中的 SDK 也会阻塞 ，等待 Collector 对 OTLP 批次进行确认。应用线程在 OTel 导出器调用处堆积；请求延迟恶化；本应用于诊断故障的可观测性栈，自己引发了应用故障。

恢复措施。

  1. 配置非阻塞的 SDK 导出器。 OTel SDK BatchSpanProcessor 应在溢出时丢弃数据，而非阻塞。 maxQueueSize, scheduledDelayMillis, exporterTimeout 经过调优，确保 Collector 过载时丢弃 span，而非阻塞应用。
  2. 在 Collector 中配置内存限制处理器。 当 Collector RAM 达到阈值时，在入队前丢弃传入的 OTLP 数据。应用看到的是快速报错，而非缓慢阻塞。
  3. 异步批量日志记录。 日志同理——永远不要在日志写入时阻塞应用。 log.warn 调用应采用队列后立即返回的模式。
  4. 生产纪律。 "可观测性永远不阻塞应用，任何情况下都不行。"定期审计；违反此原则视为 P0 平台 bug。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 在极端情况下丢弃 span/日志。遥测是尽力而为的；应用存活不是。

Prometheus scrape 压垮目标服务

CloudTrail + S3 Access Logs。 Prometheus 被配置为对 /metrics 拥有 50K 指标的服务每 5 秒 scrape 一次。每次 scrape 需解析 50K 行，阻塞一个线程约 200ms。加上 10 个 Prometheus 副本为高可用（High Availability）并发 scrape，该服务每秒接收 20 次 scrape，每次消耗一个线程。指标端点成为热点——1% 的 CPU 用于指标输出。

恢复措施。

  1. 降低 scrape 频率。 大多数指标改为 30 秒，而非 5 秒。
  2. 预计算 / 预聚合。 采用每进程指标聚合，而非每次调用都进行暴露。
  3. 使用远程写入模式。 应用通过 OTLP 推送到 Collector（再由 Collector 通过 remote-write 推送到 Prometheus），消除每次 scrape 的解析开销。
  4. 基数缩减。 50K 指标本身往往就是一个未解决的基数问题的信号。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 基于推送的批量摄入在每实例超过约 10K 序列时，扩展性优于基于拉取的 scrape。

一个 bug 将日志塞满了调用栈

CloudTrail + S3 Access Logs。 一个处于关键路径上的异常开始以 1000 req/sec 的速度触发。每次触发一个 30 行的调用栈，约 3 KB 的日志行。净日志量：3 MB/sec = 单个 bug 每天产生 260 GB。ELK 摄入超限；温层填满；摄入对 所有 个服务形成积压。平台因一个服务的 bug 遭受结构性冲击。

恢复措施。

  1. 按服务限制日志速率。 每个服务设定配额（例如 100 MB/sec）。超出部分在本地 Fluent Bit / OTel Collector 处丢弃。有问题的服务日志部分丢弃；其他服务不受影响。
  2. 在 Collector 处对调用栈进行去重。 如果 1 秒内有 1000 行具有相同的调用栈特征，则只输出一条并附上 count=1000。对于重复异常，数据量缩减 1000 倍。
  3. 热修复 bug。 在此之前，速率限制是安全网。
  4. 对日志量峰值告警。 某服务的日志量超过基线 10 倍时触发告警；比等待存储层填满要更快发现问题。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 按服务限速是首要防线。没有它，一个服务的 bug 可以摧毁整个平台。

时钟偏差导致 trace 展示异常

CloudTrail + S3 Access Logs。 服务 A 的时钟比服务 B 快 200ms。一条 A→B 的 trace span 在 A 的 t=10.000时刻结束，在 B 的 t=9.950 （B 的时钟）时刻显示。trace UI 显示 B 的 span 开始于 之前 A 向 B 发起调用——这在物理上是不可能的。排查 trace 的工程师感到困惑；调查陷入停滞。

恢复措施。

  1. 严格的 NTP / PTP 规范。 每台主机运行网络时间协议（NTP）或精确时间协议（PTP）；时钟漂移 > 100ms 触发告警；时钟漂移 > 1 秒触发紧急通知。
  2. trace UI 容错。 现代 trace UI（Tempo、Jaeger）在检测到时钟偏差时，会将子 span 的开始时间对齐到父 span 的开始时间以进行补偿。无法修复数据，但可以掩盖可视化问题。
  3. span 时间以持续时间而非绝对时间报告。 OTel span 携带 start_time 和 duration。duration 是明确的；start_time 的绝对值依赖于主机时钟。UI 在 trace 内部的可视化中优先使用 duration。
  4. 通过已知的时序基准检测偏差。 应在 <10ms 内完成的跨服务请求（例如健康检查 ping）测量显示超过 100ms——标记该主机。时钟偏差的可观测性本身也是一个独立的监控项。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 时钟偏差受 NTP 规范约束。平台必须防御长尾情况。

§24. 为何不用显而易见的更简单方案——"SSH 进去 tail 日志"

这是许多工程师成长过程中形成的真实心智模型。当"应用"是单台主机上的单个进程时，它是有效的。但只要超过一个服务或一台主机，它就会彻底失效。

一位客户反馈"2:34 PM 东部时间结账失败"。你 ssh 进入 api-gateway。你不知道 800 台主机中哪台处理了他们的请求。你随机 tail 几台主机；什么都没有。在 800 台主机上 grep 了 20 分钟后，你在第 437 台主机上找到了一个线索。日志显示"下游超时"——但没说是哪个下游。调用链涉及 auth、checkout、payment、fraud、inventory 等服务。你逐一 ssh 过去。checkout 说"调用了 payment，收到 500"。Payment 有 200 台主机。你继续 grep。两小时后，你找到了："stripe API 在 30 秒后返回 5xx"。但没有 trace，你无法还原那究竟是哪个请求、哪个用户、什么参数、什么数据库状态。你放弃了。MTTR（平均修复时间）趋近于无穷大。

将此场景扩展到影响 10,000 名用户的故障，你需要计算影响半径；或者请求跨越了 Kafka 的异步跳转；又或者由于留存时间只有 24h，一半的日志已从主机上滚动删除。这就是那堵墙。可观测性技术栈的存在，就是为了用结构化、关联化、可查询的数据取代 SSH-and-grep。每个组件——集中式日志聚合、trace_id 传播、指标仪表盘、SLO 告警——都是这堵墙上的一块砖。

§25. 扩展轴

类型 1：均匀增长（更多服务、更多主机）

每台主机的遥测量相同，主机数量增加。主要靠水平扩展解决。 1x（1K 台主机）： 单台 Prometheus（约 300 万个 series）、单台 ELK（约 3 TB/天）、单台 Jaeger Cassandra，一名 SRE。 10 倍扩展（1 万台主机）： Prometheus 接近基数上限，通过联邦分片或迁移至 Mimir/Cortex；日志离开纯 ELK 或将其扩展到 200 个节点（代价高昂）；Trace 在更严格的采样下仍可放入 Cassandra。 100 倍扩展（10 万台主机）： 所有内容多地域化。各地域 Prometheus 通过 remote-write 汇总至全局 Mimir；日志经 Kafka 流入多层存储（Loki + Pinot）；Trace 采用 0.1% 头部采样加尾部采样，团队规模 5-10 名工程师。

拓扑必须发生结构性变化的临界点：单实例超过 100 万个 series → 对 TSDB 进行分片；日志摄入超过 1 GB/秒 → 离开纯全文索引；采样后超过 1000 万 span/秒 → 按 trace_id 对 Trace 分区并采用多分片尾部采样；超过 3 个地域 → 每地域独立存储并通过跨地域查询联邦聚合。

类型 2：热点激增（因错误标签导致的基数爆炸）

主机数量不变，但某个服务因一次部署添加了错误标签而产生多 100 倍的 series。水平扩展无济于事——问题集中在单个租户。修复思路： 在摄入端按服务强制限制基数配额 ——"checkout" 服务的 series 预算为 10 万个；新部署产生 500 万个 → TSDB 拒绝新 series 并通知该团队。 对日志和 span 摄入按服务进行速率限制 ——失控服务不会拖垮整个平台。 生产前在 CI 阶段进行基数检查 ——在合并前模拟指标输出并统计基数；若某服务基数翻倍则 CI 失败。类型 2 增长比类型 1 更难处理，因为它是突发性的，可追溯到特定部署，并且可能在数秒内达到稳态的 10-100 倍。TSDB 必须对"坏邻居"保持强约束。

§27. 使用场景案例集

五种不同的产品，以五种不同的方式应用相同的三支柱技术。技术是主体，这些案例是说明。

SRE 事件响应——支付平台。 Stripe 量级的支付处理器。SRE 值班人员通过 payment_success_rate。打开 Grafana 仪表板，查看 payment_duration_seconds 在 p99 桶中显示 exemplar——点击后跳转到具体 Trace。Trace 显示 payment → fraud-check → external-bank；欺诈检测是最慢的 span。带有该 trace_id 标签的日志显示"缓存未命中 → 数据库查询耗时 4 秒"。8 分钟内定位根因；MTTD（平均检测时间）不足 2 分钟，MTTR（平均恢复时间）不足 15 分钟。三支柱将原本数小时的排查压缩为十分钟的走查。

业务 KPI 监控——电商仪表板。 Shopify 量级的商家使用 Datadog 仪表板实时追踪 GMV（商品交易总额）、转化率和购物车放弃率。应用发出 orders_completed_total, cart_abandoned_total；仪表板 rate() 以 1 分钟窗口聚合。仪表板由业务团队而非工程团队维护。为 SRE 告警提供服务的 TSDB 同样服务于这些仪表板——只是标签和查询窗口不同。相同技术，不同消费者。

安全审计日志——金融服务合规。 某银行将所有认证事件、交易和管理员操作保留 7 年（法规要求），每天产生 5 TB 数据。热层（30 天，存于 Elasticsearch）用于主动调查；温层（30 天至 1 年，存于 Loki + S3）用于取证 grep；冷层（1-7 年，存于 S3 上的 Parquet + Glacier）。Splunk 是自建与购买的现有主流方案——许多银行每年支付 2000-5000 万美元用于合规认证和审计追踪功能。

ML 模型监控——推荐系统漂移。 生产中的推荐模型。监控内容：输入特征分布（漂移）、输出分布（是否退化）、每请求延迟（随索引增长是否变慢）以及业务效果（是否带动点击）。分布是高基数宽事件——适合 Honeycomb 形态的存储。延迟是指标——Prometheus 直方图。效果是与点击日志关联的指标。三支柱 × ML。当模型在凌晨 3 点退化时，三个支柱协同定位问题是特征流水线、模型还是数据质量问题。

慢请求调试的分布式追踪——Honeycomb 的主场。 某 SaaS 公司观察到 p99 延迟尖峰。工程师打开 Honeycomb。 heatmap(duration) WHERE service=api ——尖峰集中在某个时间窗口。添加 GROUP BY customer_id ——某一客户占主导。再添加 GROUP BY endpoint ——某一端点。深入查看一条慢 Trace：api → search → vector-store → object-storage。vector-store 的获取是最慢的环节。打开 vector-store 仪表板，发现 5 分钟前有一次部署，随即回滚。总耗时：4 分钟。高基数宽事件分析正是 Honeycomb 的列式 Retriever 所优化的工作流——而聚合指标仪表板无法支持这一工作流。

移动端崩溃分析——Crashlytics / Sentry。 每次崩溃都会生成一份包含堆栈追踪、设备信息、系统版本和操作路径的报告。SDK 在下次联网时上传。服务端将崩溃按堆栈签名分组，并按用户影响加权。开发者可以看到"v3.2.1 中按用户影响排名前 10 的崩溃"。这是事件形态的遥测数据（一次崩溃 = 一条结构化记录），而非指标形态。存储采用列式事件存储——与 Honeycomb 的 Retriever 同属一类，针对移动崩溃工作流进行了调优。对崩溃采用 100% 采样（崩溃稀少但价值高）。

日志数据上的产品分析——LinkedIn 的 Pinot 用于日志。 增长团队希望了解"昨天完成引导流程的用户，按国家、按设备分布"。历史上这需要自定义流水线。有了 Pinot 用于日志，这就变成了对日志流的 SQL 查询—— SELECT count(DISTINCT user_id) FROM logs WHERE event='onboarding_complete' AND date='2026-05-21' GROUP BY country, device。列式存储和按列索引在 TB/天规模的集群上可实现亚秒级响应。日志成为产品分析的底座——这是原始"日志用于事件排查"模型从未预料到的用例。

§28. 含数据的真实世界实现

  • Google Monarch（VLDB 2020）。 全球分布式内存 TSDB，分层区域架构，每个区域持有自身数据，聚合器查询时进行扇出。 摄入峰值约 1 万亿 sample/秒。 取代 Borgmon。经验教训：对于热指标，内存优于磁盘；区域自治优于全局一致性。
  • Facebook Beringei / Gorilla（VLDB 2015）。 该论文引入了 delta-of-delta + XOR 浮点压缩算法，推动了 Prometheus TSDB、M3DB、VictoriaMetrics、InfluxDB IOx 中 Gorilla 形态 chunk 的诞生。其中 1.37 字节/sample 这一稳态指标数据来自本论文。
  • Google Dapper（OSDI 2010）。 分布式追踪的开创性论文，确立了 0.01%-0.1% 的头部采样 作为默认值。现代可观测性仍沿用 Dapper 的模型——trace_id、span 层级、上下文传播。
  • LinkedIn 的日志流水线。 全集群每天约 10 PB 的日志摄入 ，持续速率约 115 GB/秒。流水线：应用 → Kafka → 分层存储。 Pinot 用于日志分析 ，每集群在 10+ TB/天规模下提供亚秒级 SQL 查询，取代 ELK 用于聚合场景。
  • Datadog。 公开披露 数万亿个指标点每天，每天数十亿条 Trace、历史上 EB 级别的日志。在 Cassandra 类 LSM 树上运行分片 TSDB，配合自定义 Gorilla 类编解码器。尾部采样是托管 APM 功能。
  • Honeycomb 的列式 Retriever。 率先提出"事件而非 Trace"——每个 span 都是自有列式存储中的一行。即席查询语言（heatmap(duration) WHERE service=api GROUP BY customer_id）原生支持极高基数。更适合"回答奇特问题"，不适合"展示一棵 Trace 树"。
  • Cloudflare 日志。 每天约 1 万亿条日志记录 ，通过基于 ClickHouse 的分析引擎处理，在此数据量下实现秒级聚合——成本比 Elasticsearch 低一个数量级。
  • Uber Jaeger。 由 Uber 开源。早期使用 trace_id 分区的 Cassandra；现支持 ClickHouse 和 Elasticsearch 后端。在 Uber 全集群默认头部采样率为 0.1%。
  • Lyft Envoy + 可观测性。 Envoy 开箱即提供丰富的指标、日志和追踪。"服务网格内置可观测性"模式（Envoy + Istio）意味着 L7（第 7 层）遥测无需应用侧埋点。
  • Sentry。 开源错误追踪——完整堆栈追踪和操作路径，SaaS 层每天处理约 1 亿+事件，存储基于 Snuba（ClickHouse 宽事件存储）。
  • OpenTelemetry。 厂商中立的埋点规范，CNCF 已毕业项目，由 Google、Microsoft、AWS、Splunk、Datadog、New Relic、Honeycomb 共同支持。战略押注：使用 OTel 埋点，随经济性变化切换后端。

§29. 总结

"可观测性是三个支柱、三种存储引擎、三个独立故障域——永远不是一个产品。指标存储于 Gorilla 压缩的 TSDB 中，按 series 哈希分片并设置严格的基数配额。日志在标签索引 chunk 存储（Loki，用于 grep）和列式存储（Pinot、ClickHouse，用于 SQL 聚合）之间拆分——在 PB 级规模下永远不要用全文索引。Trace 通过 OTLP 经 Kafka 按 trace_id 分区，进入尾部采样层，保留 100% 的错误和慢请求以及小比例的随机样本。关联性存在于打印在每个指标 exemplar、每行日志、每个 span 上的 trace_id 中。告警由 SLO 消耗驱动，这样只有在用户真正受影响时才触发，而不是当某个指标偏高时。各支柱可独立降级，平台能在它所监控的故障中存活。"
