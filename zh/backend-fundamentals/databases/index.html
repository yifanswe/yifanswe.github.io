第一部分：事务型数据库

§1. 事务型数据库是什么

一个 事务型数据库 是一种有状态的存储系统，其契约以 ACID — 原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability）— 为核心，作用于结构化数据的多行变更。它是一切不允许部分执行场景的底层支撑：银行账户余额、库存计数、用户档案、功能标志分配、工作流中的任务。其核心特征是"承诺事务要么完整发生、要么完全未发生，且结果在断电后依然存在"。

事务型数据库处于四维设计空间的一个角落：

  • 关系型行存储 (OLTP — OnLine Transaction Processing) — MySQL InnoDB、PostgreSQL、Oracle、SQL Server。B+ tree，行式存储，强 ACID，二级索引。典型的事务型数据库。适合点查和小范围扫描；不适合分析型扫描。
  • NewSQL / 分布式 SQL — Spanner、CockroachDB、YugabyteDB、OceanBase、TiDB。关系型接口与 ACID，底层通过共识算法（Paxos/Raft）实现水平扩展。以提交延迟（TrueTime 等待、Raft RTT — 往返时间）换取开箱即用的跨分片事务。
  • 宽列 / LSM (Log-Structured Merge) 存储 — Cassandra、ScyllaDB、HBase、Bigtable。LSM tree 存储，分区键访问，可调一致性。写吞吐量高、水平扩展能力强；仅支持每键线性化（通过昂贵的 Paxos 支持的轻量级事务）；多行事务支持薄弱或缺失。
  • 文档存储 — MongoDB、Couchbase、DocumentDB。每文档 JSON 式记录，无模式，历史上只支持单文档事务（MongoDB 4.0+ 新增多文档支持）。以模式严格性换取开发效率；文档是事务的基本单元。

本文聚焦于事务层 — OLTP 关系型行存储及其 NewSQL 近亲 — 因为字节级机制在此最为重要。宽列和文档变体借用了大量机制（WAL、MVCC、复制），但在存储层做出了不同的权衡；我们在 §3 和 §13 中对比分析。

事务型数据库不擅长的场景：

  • 对数十亿行进行分析型扫描。 行式 B+ tree 布局不适合此类场景。请使用列式存储（Pinot、Druid、ClickHouse、BigQuery、Snowflake）进行 OLAP（OnLine Analytical Processing）。
  • 以极高速率写入的时序数据。 仅追加、按时间戳排序的工作负载应使用基于 LSM 或时序专用引擎（Prometheus、InfluxDB、M3DB）。
  • 写主节点规模下的全局写吞吐量。 单主节点 OLTP 每个主节点约上限 20–50k 次提交/秒。超出后，需分片（除非构建 2PC，否则会失去跨分片事务）或迁移到 NewSQL。
  • 最终一致的读密集型工作负载。 如果可以接受 10 秒的数据陈旧，那么你为不必要的保证付出了额外代价。
  • 二进制大对象存储。 在 BLOB 列中存储一张 10 MB 的图片会损耗缓冲池。请使用对象存储；只存 URL。

心智模型：事务型数据库用于存储系统核心中少量的 可变、结构化、需严格守恒的状态 。围绕它分布着派生存储 — 缓存、搜索索引、分析仓库、对象存储 — 它们从数据库中获取真实数据，并针对不同访问模式进行重塑。把主库搭好，其他一切都可重建。主库一旦出错，再多的缓存也无济于事。

§2. 固有保证

该技术在设计上提供的保障，以及仍需在其上叠加的能力。

设计上已提供

  • 事务内的原子性。 一个 BEGIN ... COMMIT 块中的语句要么全部执行，要么完全不执行。这是核心原语。
  • 可配置级别的隔离性。 READ UNCOMMITTED → READ COMMITTED → REPEATABLE READ → SERIALIZABLE。MySQL 默认 REPEATABLE READ；PostgreSQL 默认 READ COMMITTED。你需要了解当前的隔离级别、每个级别允许哪些异常（脏读、不可重复读、幻读、写偏斜），以及你的业务不变式需要哪个级别。
  • 提交确认后的持久性。 使用默认配置时（innodb_flush_log_at_trx_commit=1 在 MySQL 中， synchronous_commit=on 在 PostgreSQL 中），一旦 COMMIT 返回，变更即可在崩溃后存活。其机制是 WAL（Write-Ahead Log）— 见 §4。
  • 二级索引与基表保持一致。 索引更新与行更新在同一个事务中完成。
  • 数据库内约束 — UNIQUE, FOREIGN KEY, CHECK, NOT NULL — 由数据库强制执行，而非应用层。

需在其上叠加

  • 跨集群的线性一致性。 对副本的读取可能看到陈旧快照。对于读后写场景，请使用读主节点路由，或基于会话的一致性令牌（PostgreSQL 的 pg_current_wal_lsn + 副本等待；MySQL 基于 GTID 的副本读取）。
  • 跨分片原子性。 一旦分片，数据库的事务边界止于分片内部。跨分片需要在上层叠加 2PC（两阶段提交）, sagas，或 escrow patterns 。NewSQL 以提交等待延迟为代价将此内置。
  • 客户端请求的幂等性。 数据库无法识别 HTTP 重试与原始请求是否"相同"。请使用带有 idempotency_key 约束的 UNIQUE 列，在与变更相同的事务中写入（Stripe 模式）。
  • 复制拓扑选择。 异步、同步与半同步具有截然不同的持久性特征。数据库三者均支持；选错将丧失持久性。
  • 备份与 PITR（时间点恢复）。 许多故障的根因是"备份存在，但从未测试过恢复"。
  • 在线模式变更。 在不锁表数小时的情况下为 1 TB 的表添加一列是一门艺术（pt-online-schema-change、gh-ost、PostgreSQL CONCURRENTLY).
  • 多区域灾难恢复。 跨区域异步副本能在区域故障中存活，但 RPO（恢复点目标）不为零。跨区域 RPO=0 需要基于共识的 NewSQL 或定制同步配置（延迟代价惨重）。

综合来看：数据库保证 单节点 ACID 和单集群恢复。超出此范围的一切 — 跨分片、跨区域、端到端幂等性、模式迁移 — 都是系统设计者的职责。

§3. 设计空间

事务型数据库的各变体沿若干正交维度存在差异，这些维度相互影响，无法自由独立选择。

存储引擎：B+ tree vs LSM

最具决定性的选择。 B+ tree （InnoDB、PostgreSQL、SQLite、SQL Server）是一棵由固定大小页面组成的平衡树，叶节点按主键排序存储行数据；更新就地重写页面；读取遍历 log(n) 个页面。 LSM tree （RocksDB、Cassandra、HBase、ScyllaDB；也是 CockroachDB 底层引擎）将写入缓冲在内存中的 memtable，刷新到不可变的有序文件（SSTable），并在后台合并文件；读取通过布隆过滤器查询多个文件。

权衡：B+ tree 提供 可预测的低 p99 读取延迟和便捷的范围扫描 ，代价是 较低的持续写吞吐量 （普通 NVMe 上约 20–30k 次提交/秒），因为需要随机就地写。LSM tree 提供 极高的写吞吐量 （约 100–200k 次写/秒/SSD），代价是 p99 读取延迟可预测性较差 （合并停顿；多 SSTable 读取）。对于需要 SELECT balance FOR UPDATE 在 <5ms 内返回的支付账本，B+ tree 胜出。对于 Discord 基于 Cassandra 的消息时序存储，LSM 胜出。

并发控制：MVCC vs 锁

MVCC（多版本并发控制） 按事务 ID 为每行存储多个版本；读不阻塞写，写不阻塞读。PostgreSQL 和 InnoDB 均使用 MVCC，但实现各异（PG 在元组上存储 xmin/xmax；InnoDB 存储 undo log 条目）。纯两阶段锁在现代 OLTP 中已属罕见。MVCC 是现代默认方案，因为 30 秒的分析扫描不能阻塞支付提交。代价：MVCC 垃圾（§7.5）。

拓扑：单主节点 vs 多主节点 vs 无主节点

单主节点 是 OLTP 的默认方案。一个节点接受写入，其余为只读副本。简单，主节点是吞吐量瓶颈。 多主节点 （在 OLTP 中罕见）允许在多个节点写入，但冲突解决棘手。 无主节点 （Cassandra、DynamoDB）允许任意节点接受写入，具有可调的 quorum；不适合 OLTP，因为每行线性化难以实现。

复制语义

  • Async（异步）：主节点提交后异步发送日志。延迟最低，持久性最差 — 主节点故障时可能丢失最多一个滞后窗口内已确认的写入。
  • Sync（同步）：主节点等待至少一个副本 应用 后才确认。持久性最高，尾延迟最高。
  • Semi-sync（半同步）：主节点等待副本 接收 （而非应用）。良好的平衡 — 提交时跨 AZ（可用区）RTT（约 1ms）；常见故障转移场景下 RPO=0。MySQL 的 rpl_semi_sync_master_wait_for_slave_count，PostgreSQL 的 synchronous_standby_names.
  • Consensus (Raft/Paxos)：quorum 在提交前持久化日志。NewSQL（Spanner、CockroachDB）和共识型数据库（etcd、FoundationDB）。RPO=0 且自动故障转移；每次提交需支付一次 quorum RTT 的代价。

§4. 底层数据结构：字节级机制

深度剖析章节。我们放大存储引擎，逐字节追踪单次变更，并明确对比 B+ tree 机制与 LSM 机制。这正是"用过数据库"与"知道为何选择这个数据库"的分水岭。

4.1 B+ tree 页面布局

在 InnoDB（典型的 B+ tree OLTP 引擎；PostgreSQL 在思路上与之类似）中，每张表都以 以主键为键的聚簇 B+ tree存储。主键树的叶节点包含行数据。二级索引是独立的 B+ tree，其叶节点存储主键；非覆盖查询需要额外回表查询聚簇树。

页面是 I/O 的基本单元 — InnoDB 默认 16 KB，PostgreSQL 为 8 KB。每个页面是树的一个节点。

Page = 16 KB. Internal layout (simplified, InnoDB):
┌──────────────────────────────────────────────────┐
│ Page header (38 B): page_no, prev/next page,     │
│   level, n_records, checksum, LSN                │
├──────────────────────────────────────────────────┤
│ Infimum + Supremum (system records, 26 B)        │
├──────────────────────────────────────────────────┤
│ User records (sorted by PK), each =              │
│   record header (5 B) + PK + non-PK columns      │
│   + DB_TRX_ID (6 B) + DB_ROLL_PTR (7 B)          │
├──────────────────────────────────────────────────┤
│ Free space                                       │
├──────────────────────────────────────────────────┤
│ Page directory (sparse index of slot offsets)    │
├──────────────────────────────────────────────────┤
│ FIL trailer (8 B): checksum + LSN                │
└──────────────────────────────────────────────────┘

Internal (non-leaf) pages: PK values + child page numbers, sorted.
Leaf pages: PK values + full row data + MVCC fields.
Leaf pages doubly linked (prev, next) → range scans are sequential.

4.2 分支因子与树高

使用 16 KB 页面、约 16 字节的 PK 和 4 字节的子指针，一个内部页面可容纳 约 800 个子指针。一个叶页面可容纳 约 60 行 （每行 256 B）。

举例说明。一张 700 亿行的表 — 支付宝规模的支付账本或 LinkedIn 的 profile-edges 表：

  • 叶节点：70B / 60 ≈ 1.2B 个叶页面
  • 上一层：1.2B / 800 ≈ 1.5M 个页面
  • 上一层：1.5M / 800 ≈ 1,900 个页面
  • 上一层：1,900 / 800 ≈ 3 个页面
  • 根节点：1 个页面
  • 树高 = 5 层。

点查遍历 5 个页面。有了多 GB 的缓冲池，根节点和上层内部页面始终被缓存 — 热点查询仅需 1–2 次页面读取，p50 约 0.1ms。冷查询会错过 2–3 层，每次 SSD 随机读取约 100µs，p99 约 5–8ms。

规模较小的用例 — 中等规模零售商的 1000 万行电商库存表：

  • 叶节点：10M / 60 ≈ 170k
  • 上一层：170k / 800 ≈ 200
  • 根节点：1
  • 树高 = 3 层。

规模不同，但对数上限相同。分支因子很大；B+ tree 始终保持较矮。

4.3 MVCC：PostgreSQL 与 InnoDB 的差异

PostgreSQL 和 InnoDB 均实现了 MVCC，但版本信息的存储方式不同：

  • PostgreSQL 在每个元组上存储 xmin 。 xmax — 插入和删除该行的事务 ID。更新即删除 + 插入：旧元组获得 xmax = T，新元组获得 xmin = T。其结果是 元组膨胀；死元组版本会在堆中累积，直到 autovacuum 运行。
  • InnoDB 在聚簇索引中保存 当前行 ，附带两个 MVCC 字段 — DB_TRX_ID （6 B）和 DB_ROLL_PTR （7 B）。其中 roll_ptr 指向一个独立的 undo log 存储旧行镜像的表空间。读取旧版本时，沿 roll_ptr 遍历 undo 记录。清除线程（purge thread）回收不再被任何活跃读视图所需的 undo 条目。主数据页更整洁；代价转移到 undo 表空间的增长上。

事务 T 在读取被事务 T' 更新的行 R 时：1. 检查聚簇索引中的行 R，读取 DB_TRX_ID = T'。2. 查询其 read view ——即 T 快照时刻可见的事务集合。3. 若 T' 在 T 的快照之前提交，T 读取此版本。4. 否则：沿 DB_ROLL_PTR 进入 undo log 获取前一个版本，递归执行。

这正是 MVCC 强大之处：一次耗时 30 秒的分析扫描可在热副本上运行，同时主节点以 10k/sec 的速度持续写入；该扫描通过 undo 链实现了冻结快照。代价是长事务读取问题（§7.5）。

4.4 Buffer pool：你真正需要的 LRU

Buffer pool 是数据库的内存页缓存，通常占 RAM 的 60–80%。朴素 LRU（最近最少使用）有一个已知缺陷：一次大型扫描（SELECT * FROM events WHERE ts > ...）读取数百万页，每页仅访问一次，会将热点工作集逐出缓存。

InnoDB 通过 中点插入 LRU:

buffer pool LRU list:
  ┌──────── young (5/8) ────────┬──── old (3/8) ────┐
  HEAD                          midpoint            TAIL (evict)
  ↑                             ↑
  promoted here on              new pages enter here.
  second access after           Must survive innodb_old_blocks_time
  the dwell time                ms (default 1000ms) to promote.

来防御此问题。新读入的页进入中间点。若在驻留时间内被再次访问，则晋升到年轻端。一次性扫描的页面进入旧区域，在污染年轻区域之前就被逐出。PostgreSQL 的 buffer manager 使用相同思路的时钟扫描变体。

容量规划启示：你无需将整个数据集装入 RAM，只需装入 热点工作集。一个 17 TB 的分片配合 180 GB 的 buffer pool 完全可行，前提是 1% 的账户驱动 99% 的流量。

4.5 WAL / redo log：持久性的基础原语

Write-Ahead Log（WAL，即 InnoDB 中的"redo log"）使 buffer pool 变得安全。不变量： 对每一次页面修改，WAL 中均有描述，且 WAL 在修改提交确认前已写入磁盘。 页面写入本身可在提交后滞后数分钟；WAL 是恢复的权威依据。

WAL 是一个顺序仅追加（sequential append-only）的物理-逻辑记录流："在 LSN（Log Sequence Number）L，页 P，偏移 O，写入字节 B"。顺序性是关键——单块 SSD 进行随机页写入可能只有 10k IOPS，但顺序 WAL 追加可持续达到数百 MB/sec。

提交协议（MySQL 默认， innodb_flush_log_at_trx_commit=1):

  1. 事务在 buffer pool 中修改页面；页面标记为脏页。
  2. 每次修改将一条 redo 记录追加到内存日志缓冲区。
  3. 在 COMMIT时，通过 write() + fsync().
  4. 仅在 fsync 返回后，提交才被确认。
  5. 脏页在 buffer pool 中停留数分钟，然后才写入实际位置。
  6. 定期 检查点（checkpoints） 推进"从此处重做"指针，允许 WAL 环形复用。

Group commit 至关重要。单次 NVMe fsync 约需 ~50–200µs。若以 18k commits/sec 的速率执行朴素的逐次 fsync，将使 SSD 饱和。InnoDB 将并发提交批量合并为一次 fsync——100 个事务共享一次 fsync，有效速率远超 20k/sec。PostgreSQL 实现了相同的思路（commit_delay, wal_writer_delay).

  1. 客户端 → 拥有行 42 的 DB 分片 接收 BEGIN; SELECT balance FOR UPDATE; UPDATE; COMMIT;.
  2. 锁获取：在聚簇索引中对 PK=42 加行锁。内存中的锁管理器，无磁盘 I/O。
  3. 读取现有行：B+ 树遍历，根节点 → 第 3 层 → 第 2 层 → 第 1 层 → 叶节点。若叶节点为热页，0 次磁盘 I/O；若为冷页，1 次 SSD 读取约 ~100µs。当前行： balance=500, DB_TRX_ID=T_prior, DB_ROLL_PTR=undo_ptr_5.
  4. 计算新值: balance = 400.
  5. 首先写入 undo log：在 undo_ptr_6 处写入包含旧行镜像的新 undo 记录。undo 页同样在 buffer pool 中。内存中追加一条描述此次 undo 写入的 redo 日志记录。
  6. 在 buffer pool 中修改叶节点页：写入 balance=400, DB_TRX_ID=T_current, DB_ROLL_PTR=undo_ptr_6。叶节点页变为脏页，追加 redo 记录： LSN=1042: physical write to page P_leaf, offset O, bytes <new row>.
  7. 二级索引：若存在 (account_id, ts DESC) 索引，且余额更新时 ts 未变，则该索引不受影响。若被索引列发生变化，则更多页面变脏，追加更多 redo 记录。
  8. 辅助插入 在同一事务中（日志条目、审计行）：每次都是另一次 B+ 树插入，可能触发页面分裂（目标叶节点已满 → 分配新页，重新分配；InnoDB 的右分裂启发式策略对单调递增的插入保持左页满载）。页面分裂以另一条 redo 记录记录。
  9. COMMIT：在 LSN 1080 处写入"事务提交"redo 记录，将日志缓冲区刷新到操作系统。 fsync(redo_log_fd)。NVMe 上约 ~150µs。
  10. 半同步等待 （若已配置）：binlog 事件通过 TCP 发送至副本，副本确认接收。跨可用区 RTT 约 ~500µs–1ms。
  11. 确认客户端：HTTP 200 及新值。
  12. 异步，数秒至数分钟后：页面清理线程将脏叶节点页刷写到磁盘。检查点最终推进 LSN 下限超过 1080。在该检查点之前，此次提交的 redo 日志条目对恢复而言是必需的。

崩溃场景：

  • 在步骤 9（fsync 完成）和步骤 11（确认发送）之间崩溃：客户端携带相同 idempotency_key重试。幂等性行在同一事务中已插入；查找时找到该行，返回之前的结果。 持久性点：步骤 9 的 fsync。
  • 在步骤 6（页面变脏）和步骤 9（提交已记录）之间崩溃：redo 日志中无 COMMIT 记录。重启从上次检查点扫描 redo 日志，发现不完整事务，通过 undo 回滚。锁释放，客户端超时后重试——重新发起一次新的尝试。 正确性保证：不完整的状态从未对外可见。
  • 在步骤 11（确认）和步骤 12（页面刷写）之间崩溃：redo 日志中有提交记录。重启从上次检查点向前扫描，在 LSN 1080 处找到 COMMIT，重做页面修改。脏页在 buffer pool 中重建。

4.7 B+ 树与 LSM 的对比分析

属性 B+ 树（InnoDB） LSM 树（RocksDB）
写入路径 随机页写入 + redo（顺序） 顺序 WAL + memtable，定期刷写
写放大 ~3–5x（页 + 索引 + redo + binlog） ~10–30x（compaction 重写）
读放大 1–5 次页读取（树高度） 1 个 memtable + N 个 SSTable（bloom 过滤）
空间放大 ~1.5x（填充因子 ~70%） ~1.1–2.2x
每块 SSD 持续写入速率 ~20–30k 小事务提交 ~100–200k
p99 读取延迟 低且可预测 较高，可预测性差（compaction 停顿）
范围扫描 优秀（叶节点相互链接） 良好（合并迭代器；墓碑标记增加开销）
行级锁 原生支持 通常基于乐观并发/版本控制

为何在事务型工作负载中选择 B+ 树。 当读取路径为"借记前检查余额"、"下单前检查库存"、"服务前检查 ACL"时，可预测的 p99 读取延迟比峰值写入吞吐量更重要。与 FOR UPDATE 配合的行级锁是在不基于引擎自行实现 CAS 的情况下串行化并发写操作的方式。接受 ~20–30k commits/sec/主节点，因为工作负载可以整洁地分片。

为何在高写入工作负载中选择 LSM。 当模式为"按分区键追加事件，偶尔按 partition+timestamp 进行点查询"时，LSM 在写入路径上成本约低 5 倍，且 compaction 停顿不会显现，因为读取是批量扫描而非面向用户。Cassandra 和 HBase 在时序和 IoT 场景中占主导地位；现代 NewSQL 系统在底层使用 RocksDB，同时在其上构建类似 B+ 树的事务语义。

§5. 容量上限

该技术所能实现的能力，通过不同规模的场景加以说明。

单节点 OLTP——初创规模。 在 EC2 r6i.4xlarge（16 vCPU，128 GB RAM，NVMe 本地 SSD）上的 PostgreSQL 可持续支撑 ~10k commits/sec（p99 <10ms）、从热 buffer pool 进行 ~50k 点查询/sec，工作数据量 ~1–10 TB。对于拥有 1 万客户、每客户 100 QPS 的 SaaS 完全够用。Hacker News 在约 2016 年前运行在单台 PostgreSQL 上；Basecamp 多年来运行在单台 MySQL 主节点上。下一个瓶颈：单主节点的写入吞吐量。添加只读副本可扩展读取，但无法扩展写入。在 ~30k commits/sec 时，分片看起来不可避免。

分片关系型数据库——中等规模。 应用层分片（YouTube 和 Slack 使用 Vitess；ProxySQL；或手工实现）。 Notion 于 ~2021 年将 PostgreSQL 分片为 ~32 个逻辑分区——每分片 ~5k writes/sec，合计 ~50k。 Stripe 历史上使用 MongoDB，正在迁移至分片 PostgreSQL——估计峰值 5–10k payments/sec（2024 年）。 Pinterest 将 pin 存储在 4096 个逻辑 MySQL 分片中，分布在数百个物理实例上；流量高峰期每秒写入 60 万个 pin。瓶颈： 跨分片事务。一旦需要"转账"、"在工作区之间移动文档"、"合并两个租户的数据"，应用层就需要构建 2PC 或 saga 模式。

Vitess at YouTube——高规模。 基于 MySQL 的视频元数据分片层。 数百万 QPS 聚合跨数千个分片；总数据量达 PB 级；热分片 p99 读取 <10ms。Vitess 在运维层面解决了分片路由问题，同时保留 MySQL InnoDB 作为存储引擎。

OceanBase / 支付宝——以极速处理支付。 兼容 MySQL 的分布式 SQL。 2020 年双十一峰值每秒 583,000 笔支付 ；每日约 1500 亿笔交易；账本总量约 50 PB；p99 提交延迟 <100ms（含跨分片 2PC）；RPO=0；单可用区故障 RTO <30s。这是当前生产环境的上界。架构：分片 MySQL 兼容引擎 + 每分片 Paxos + 自定义 2PC 协调器。

LinkedIn Espresso——PB 级规模的个人资料数据。 用于个人资料、帖子、评论、关系的分布式存储。 PB 级；跨数千分片聚合 ~10M QPS 。每分片 MySQL，配合自定义路由（Helix）和通过 Brooklin → Kafka → 派生存储的 CDC（变更数据捕获）。

从初创的 1 万 QPS、中等规模的 50–50 万，到 OceanBase 支持跨分片事务的数十万——这一范围 在同一技术体系内跨越了两个数量级。可扩展的是存储引擎之上的分区和复制拓扑；存储引擎本身（B+ 树、MVCC、WAL）在 Notion 的分片和 OceanBase 的分片上是相同的。

§7. 事务型数据库固有的难题

六个基本挑战。每一个都与使用场景无关，始终存在；举例来自各个领域。

7.1 热点行竞争

一句话总结。 某一行接收的写操作如此频繁，以至于行锁将其串行化，队列无限增长。

出现场景。 某 Twitch 主播的打赏系统每秒接收 10 万次打赏。某限时抢购 SKU 在上线时每秒接收 5 万次减库存操作。某热门话题的计数器每秒递增数十万次。

为何会崩溃。 单个 InnoDB 行锁每秒支持约 ~1–5k 次更新。在 10 万次/秒的情况下，锁队列不断增长；p99 从 5ms 飙升至数秒；同一分片上的无关查询也受牵连，因为 I/O 线程已饱和。

真正的解决方案：在行内进行写分片。 将热点行提升为 N=64 个子行，插入按 event_id % 64路由。读取通过 SUM 聚合。该模式出现在直播平台的打赏聚合、社交信息流服务的计数器聚合，以及 DDIA中的托管模式。提款或强一致性读取需要跨子行协调，以 2PC 代价换取不阻塞写入。

触发阈值。 当某行的写入速率持续 60 秒超过 1k/sec 时，自动触发提升。

7.2 跨分片事务

一句话总结。 一旦分片，跨越两个分片的事务就没有原生的原子性原语。

出现场景。 支付在不同分片上的账户之间转账。Notion 中的工作区迁移将页面从一个租户分片移到另一个分片。社交图中的好友请求触及两个用户的邻接表。

为何会崩溃。 具体状态（支付）：t=0，分片 17 上的 Alice 余额为 1000，分片 203 上的 Bob 余额为 500。t=1，借记 Alice → 900，提交分片 17。t=2，网络抖动，分片 203 上的 Bob 贷记超时。t=3，退款 Alice，同样超时。t=4，100 美元消失。守恒性被破坏。库存中的相同模式：扣减仓库 A，向仓库 B 增加库存失败，库存丢失。

真正的解决方案：基于每分片 XA 事务之上使用带持久化协调器的 2PC ，若允许松散排序则使用带补偿的 saga 模式。

   Coordinator                Shard 17                       Shard 203
   ───────────                ─────────                      ─────────
   1. log: PREPARING(T)
   2. ─── PREPARE ──────►     XA PREPARE: debit Alice
                              (row locked, redo+binlog fsync'd)
   3. ─── PREPARE ────────────────────────────────────►     XA PREPARE: credit Bob
                                                            (row locked, fsync'd)
   4. ◄── YES ──                                            ◄── YES ──
   5. log: COMMITTED(T)   ← the durability point
   6. ─── COMMIT ──────►     XA COMMIT
   7. ─── COMMIT ─────────────────────────────────────►     XA COMMIT

协调器的 COMMITTED 日志条目保证即使在重启后两个参与者也能提交。在 PREPARE 和 COMMIT 之间，两行均被锁定（尾部延迟约 ~16ms）。技术上称为"带参与者超时的 2PC"——若协调器静默超过超时时间，参与者单方面中止。NewSQL 通过 Paxos 复制的事务记录内置了这一机制，代价是共识 RTT 提交延迟。

7.3 复制延迟与写后读

一句话总结。 副本落后主节点 50ms–2s；写入后立即读取可能看到旧状态。

出现场景。 用户更新 LinkedIn 个人简介，刷新个人资料页，看到旧简介（读取命中了副本）。商家更新定价，重新获取产品列表，看到旧价格。用户发布 Notion 评论，重新加载页面，评论不见了。

为何会崩溃。 延迟 50ms 时，约 1% 的写后读会命中滞后副本。在 10k QPS 下，每秒有 100 次不一致读取——在用户投诉中清晰可见。

真正的解决方案。 在用户写入后 ~5s 内，通过 session cookie 跟踪，将写后读路由到主节点。或使用 session 级一致性令牌——PostgreSQL 在提交后返回 pg_current_wal_lsn ；后续读取等待任意副本追上（pg_wal_replay_wait_for_lsn）。MySQL 通过 GTID（全局事务标识符）实现类似功能。对于不变量读取（余额），始终从主节点读取；以读取可扩展性换取正确性。

7.4 高负载下的 Schema 演进

一句话总结。 添加列、修改类型、添加索引——若以持有表锁的朴素 DDL 方式执行——会阻塞生产流量。

出现场景。 某新功能向 LinkedIn 的 5 亿行 users 表添加 last_login_at 列。某监管要求向 100 亿行 accounts 表添加 kyc_status 列。Notion 的某次重新设计将 50 亿行 blocks 表的某列类型从 INT 与 BIGINT 改变。

为何会崩溃。 MySQL 历史上的原生 ALTER TABLE 会在持有元数据锁的同时重写整张表——在 5 亿行数据上会造成数小时停机。

真正的解决方案：在线 Schema 变更工具。 MySQL： pt-online-schema-change 或 gh-ost。两者均创建影子表，通过触发器（pt-osc）或 binlog 跟踪（gh-ost）双写，回填数据，然后原子切换。PostgreSQL： CREATE INDEX CONCURRENTLY; ADD COLUMN ... DEFAULT NULL （PG 11 起仅修改元数据）； pg_repack 用于重写。类型变更：双写至旧列和新列，回填，切换读取，删除旧列——这是一个耗时数周的流程。

7.5 长事务与 MVCC 膨胀

一句话总结。 一个长时间运行的 BEGIN 持有 MVCC 快照，阻止整个数据库中死行版本的垃圾回收。

出现场景。 某分析师将 psql 打开，并执行 BEGIN; SELECT ... 长达一小时。某批量 ETL 在事务内运行 30 分钟的扫描。某卡住的连接在崩溃后仍处于 idle in transaction 状态。

为何会崩溃。 PostgreSQL：T 在 LSN 1000 时启动；T 运行期间 1000 万行被更新；每次更新将旧元组标记为 xmax；autovacuum 无法回收，因为这些元组对 T 的快照仍可见；表膨胀至 10 倍；最终磁盘耗尽。InnoDB 变体：长时间的读取阻止 undo log 清除；undo 表空间无限增长。

真正的解决方案。 idle_in_transaction_session_timeout, statement_timeout 设置为几分钟；强制终止运行时间较长的事务。应用层：显式设置查询超时；在短事务之间共享快照，而非持有一个大快照。监控 pg_stat_activity 长事务并发出告警。这是生产 OLTP 系统的隐形杀手。

7.6 流量不均衡下的热分区

一句话总结。 即便采用了良好的分片策略，流量也很少均匀分布；某个分片会过热，而其他分片则处于空闲状态。

出现场景。 一个按 customer_id 发现某个卖家的活跃度是中位数的 50 倍。一个按 region 发现某个地区贡献了 80% 的 QPS。一个按 tenant_id 发现最大租户占用了总容量的 20%。

为何会崩溃。 哈希分片能均匀分布键，但无法均匀分布 流量。如果某个键的流量是其他键的 1000 倍，其所在分片的负载就会高出 1000 倍。统一扩容会导致热分片资源不足，而其他分片资源过剩。

真正的解决方案：分级处理。 对于极热键：在键内部进行写分片（§7.1）。对于中等倾斜：通过迁移分区而非单纯依赖哈希来重新均衡（Vitess 的 VReplication 可将某个键范围迁移到不同分片）。对于季节性热点：在峰值前预先扩容可能出现热点的分片——例如 Alipay 的双十一、Amazon 的黑色星期五。容量规划应以分片为单位，而非统一规划。

§8. 复制、故障转移与灾难恢复

前几节将复制视为一个可调旋钮（异步/半同步/同步）。在生产规模下，复制 是 构成了整个运维故事——数据如何在节点间流转、主节点故障如何从 6 小时的大事故变成 30 秒的短暂抖动、一条"drop table production"如何变得可恢复而非变成一次简历危机事件。本节将深入探讨。

8.1 Binlog / WAL 流式传输机制——复制究竟如何移动数据

复制是一个节点持续追踪另一个节点的持久化日志。MySQL 与 PostgreSQL 的机制在形态上有所不同，但殊途同归：传输日志，重放日志。

MySQL binlog。 binlog（二进制日志）是 独立 于 InnoDB redo log 之外的日志。redo log 是物理级别的（页面级别），binlog 是逻辑级别的（事件级别），副本消费的正是 binlog。有三种格式：

  • 基于语句的复制（SBR）。 SQL 语句被直接传输（UPDATE accounts SET balance = balance - 100 WHERE account_id = 42）。体积紧凑，但在非确定性函数（NOW(), RAND(), UUID()）、依赖行数的语句以及有副作用的触发器上会出现问题。基本已被废弃。
  • 基于行的复制（RBR）。 命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 之前 。 after 行镜像被传输。在任何语句下都具有确定性和安全性，但负载更大——单条 UPDATE ... WHERE status='pending' 影响 10 万行会产生 10 万个行事件。
  • 混合模式。 默认使用基于语句的复制；当语句具有非确定性时自动切换为行模式。自 MySQL 5.7 起为默认值。

GTID（全局事务标识符）。 每个已提交的事务都获得一个格式为 server_uuid:transaction_id的唯一 ID。副本追踪已应用的 GTID（gtid_executed）。故障转移变得简单明了：给我 GTID X:1042之后的所有内容。在 GTID 出现之前，副本追踪的是 (binlog_file, binlog_offset) ——当主节点变更时会灾难性地失效（新主节点的 binlog 文件并非旧主节点的 binlog 文件）。GTID 将标识符与物理文件解耦。

Replication slots 追踪每个副本仍需要哪些 WAL 段。若无 slot，激进的 checkpoint 可能在落后的副本消费之前就回收 WAL 段，导致副本中断。有了 slot，主节点拒绝回收副本仍需要的 WAL——代价是永久断开连接的副本的 slot 会填满主节点磁盘（这是一种反复出现的故障模式；监控 pg_replication_slots.confirmed_flush_lsn).

LSN（日志序列号）。 MySQL 和 PostgreSQL 都使用日志的单调递增位置。PostgreSQL 的 LSN 是 64 位字节偏移量（pg_current_wal_lsn() → 0/3A12F8）。MySQL 的 binlog 有 (file, position) GTID 之前的文件偏移量以及 GTID 之后的 GTID 集合。 延迟 是 primary_lsn − replica_lsn 以字节计，或以"该 LSN 在主节点提交的时间"为基准换算成秒。

逐字节地，主节点 → 副本，跟踪一个事务。 继续 §4.6 的示例（在 LSN 1042 处借记 Alice -100）：

  1. 主节点在 LSN 1042 处写入 redo 记录（页面变更）以及 GTID 为 srv-uuid:1042 的 binlog 事件，其中包含行的前/后镜像。
  2. 本地提交后，主节点的 dump thread（PG 为 walsender）读取新的 binlog/WAL 字节并写入 TCP socket。
  3. 副本的 IO thread（或 walreceiver）接收字节。副本将其写入 relay log（或本地 WAL）并执行 fsync。
  4. 在半同步模式下，副本在此时返回 ack。 主节点的提交返回给客户端。
  5. 副本的 SQL thread（或启动进程）从 relay log 读取，将行写入 InnoDB 聚簇索引（或将 WAL 记录应用到 PG heap 页面）。副本在本地提交。副本的 gtid_executed （或 pg_last_wal_replay_lsn）推进至 1042。
  6. 副本现已追上。从该副本读取快照 ≥ 1042 的数据可以看到此次借记操作。

我们所关注的"延迟"是指每个副本上步骤 1 到步骤 5 之间的时间差。步骤 4 是半同步等待的节点；步骤 5 是 sync_commit=remote_apply 等待的节点。

8.2 复制模式深度解析

四种模式及其实际代价。

异步模式（默认）。 主节点在本地 redo+binlog fsync 完成后立即向客户端返回 ack。副本按自身节奏追赶。提交延迟最低（NVMe 上约 150–300µs）。 RPO > 0 主节点故障时，尚未同步到副本的数据会丢失。

   │ COMMIT │                        │
   ├───────►│                        │

   │        │ ──── async send ──────►│   (some time later)
   │ ◄─ACK──┤                        │
   │        │                        │

            │  500ms, or 30s behind) │

半同步（rpl_semi_sync_master_wait_for_slave_count=1 在 MySQL 中， synchronous_commit=remote_write PG 中）。 主节点等待副本确认 接收 （已持久化到副本的 relay log / WAL）后再向客户端返回 ack。副本不一定已 应用 ——但数据已在持久化存储上。主节点崩溃不会丢失该提交。 RPO ≈ 0 （通常情况下）。

   │        │ ◄────── ACK ───────────┤
   │ ◄─ACK──┤                        │

延迟代价：一次跨 AZ 往返时间，通常约 0.5–2ms。跨地区：50–200ms——通常对 OLTP 不可接受。

同步/ remote_apply (synchronous_commit=remote_apply PG 中）。 主节点等待副本 应用 应用变更后再向客户端返回 ack。这是最强的保证：提交后立即从指定同步副本读取，保证能看到该提交。延迟代价极高：2× WAL fsync + 应用时间 + 往返时间。跨 AZ 提交从约 1ms 升至约 3–5ms；尾延迟叠加严重。在生产 OLTP 中极为罕见，因为应用步骤的耗时难以预测。

基于共识的模式（每分片 Raft/Paxos）。 Spanner、CockroachDB、TiDB、OceanBase、FoundationDB。每次提交必须在副本集的 多数 节点上持久化后才向客户端返回 ack。3 副本场景下，2/3 节点必须完成 fsync——该提交可以承受任何单点故障，包括主节点故障。延迟代价：一次 quorum 往返时间——同地区约 5–15ms，跨地区 50–200ms。

8.3 复制拓扑变体

主节点 → N 个副本（星形）。 默认拓扑。一个主节点直接向每个副本流式传输。每个副本消耗一个 TCP 连接、一个 dump/walsender 线程以及与写入速率相等的复制带宽。超过 5–8 个副本后，主节点的网络和 CPU 开始承压。

            ┌─► Replica A (same AZ, semi-sync)
   Primary ─┼─► Replica B (other AZ, semi-sync)
            ├─► Replica C (other AZ, async, for reads)
            └─► Replica D (other region, async, for DR)

主节点 → 中继节点 → 副本（级联）。 当扇出超过约 8 个副本时，引入一个中间节点。中继节点本身也是一个副本，它再向下游的副本集群进行流式传输。Pinterest 过去每个分片采用 1 个主节点 → 1 个中间节点 → 6 个副本，使主节点只需服务 1 个 dump 线程而非 6 个。这会增加约 30–100ms 的延迟。

   Primary ─► Intermediate Replay Replica ─┬─► Replica 1
                                           ├─► Replica 2
                                           ├─► Replica 3
                                           └─► … Replica N

多主节点（MySQL Group Replication、PostgreSQL 的 BDR）。 两个或更多节点接受写入；冲突通过认证机制解决（Group Replication 使用基于 Galera 风格认证的内存全序排列）。在 OLTP 中极为罕见，因为冲突解决机制充满风险——在两个地区同时写入同一行并通过时间戳进行协调，对于从余额减去 100 这样的语义来说是静默的数据损坏。当分区规则确保 没有任何键在两个地方被写入 （地理分区）时，多主节点才能正常工作——而这本质上只是换了个名字的分片。

在不同 AZ/地区部署读副本。 一种常见的 3-AZ 配置：主节点 + 另一个 AZ 中的 1 个半同步副本（RPO=0），加上分布在各 AZ 承担读负载的 2 个异步副本，再加 1 个用于 DR 的跨地区异步副本。

每个副本的成本。 每个副本在主节点上消耗约 1 个提交周期的 CPU（dump/walsender 线程执行网络发送）+ 与每个副本写入速率相等的网络带宽 + 与数据集大小相等的存储。8 个副本意味着 8 倍的复制带宽。一个 17 TB 的分片，写入流量为 50 MB/s，则所有副本合计产生 400 MB/s 的主节点出站流量——即 3.2 Gbps。AWS r6i.4xlarge 有 12.5 Gbps，绰绰有余；而 r6i.large 只有 0.75 Gbps，则会饱和。

8.4 复制延迟——原因、监控与缓解

延迟是核心运维指标。延迟为 0ms 时，从副本读取的数据是一致的。延迟达到 30s 时，你的 bug 追踪系统就会爆满。

原因。

  • (a) 大事务导致网络带宽饱和。 一条涉及 10 万行的 UPDATE products SET price = price * 1.05 在基于行的格式中会产生约 50 MB 的 binlog 行事件；通过 100 Mbps 链路传输，仅网络就会造成 4 秒的复制延迟。缓解措施：对大批量任务进行速率限制、安排在非高峰时段执行，或拆分成更小的事务。
  • (b) 副本上的单线程 SQL 应用（MySQL 5.7 之前的旧版本）。 5.7 版本之前，SQL 线程是一个串行应用事件的单 goroutine。使用 8 核并行提交的主节点所发送的事件，副本必须串行应用。副本无法跟上突发写入的主节点——这导致了 Twitter、Facebook 等公司在 2010 年代著名的 MySQL 复制延迟事故。
  • (c) 副本 I/O 子系统比主节点慢。 为节省成本使用更小实例的副本配备了更慢的 SSD；副本 fsync 耗时 100µs，而主节点为 50µs。一旦副本落后，就再也追不上，因为稳定状态下的应用速率低于主节点的提交速率。
  • (d) 副本上长时间运行的查询持有 MVCC 快照。 一个持续 30 分钟的分析查询会阻止副本应用任何 vacuum 或 undo 清理；最终复制线程会变慢。PostgreSQL 的 hot_standby_feedback=on 使情况更糟——副本的快照也会固定主节点的 vacuum。（将分析查询分流到副本是一种功能；在主节点的热备副本上运行 4 小时全表扫描则是一种事故。）

监控。

  • MySQL SHOW SLAVE STATUS\G — Seconds_Behind_Master. 臭名昭著的不可靠指标。 它衡量的是 当前已应用事件 的时间戳与 副本当前墙钟时间之间的差值。如果 dump 线程停滞——IO 线程空闲、SQL 线程空闲—— Seconds_Behind_Master 仍然返回 0，即便复制已经中断。使用 Performance Schema replication_applier_status_by_worker.LAST_APPLIED_TRANSACTION_END_APPLY_TIMESTAMP 获取真实数据。
  • PostgreSQL pg_stat_replication — sent_lsn, write_lsn, flush_lsn, replay_lsn。以时间表示的回放延迟： now() - pg_last_xact_replay_timestamp() 在副本上执行。
  • 以字节计算的延迟 (primary_lsn − replica_replay_lsn）比以秒计算的延迟更可靠，因为在没有数据传入时不会撒谎。
  • 告警阈值。 5s 是常见的"黄色"告警阈值；30s 为"红色"。0.5s 的延迟在跨 AZ 异步复制中属正常。超过 60s 即为事故。

缓解措施。

  • MySQL 5.7+ 多线程从库: slave_parallel_workers=8, slave_parallel_type=LOGICAL_CLOCK。副本根据主节点的提交组并行应用事件——在主节点同一 group commit 中提交的事件可以在副本上并行回放。
  • PostgreSQL 10+ 并行应用 通过逻辑复制实现。物理 WAL 的流式复制仍然是单线程的；逻辑复制支持并行工作线程。
  • 将大事务 拆分成更小的批次。将 100 万行的更新拆分为 100 次 1 万行的更新，给复制留出追赶的机会。
  • 优先保障复制网络。 在 AWS Direct Connect 或 VPC 上配置 QoS；将复制流量与应用流量隔离。
  • 将副本规格配置为不低于主节点。 在需要跟上主节点写入速率的副本上节省成本是一种错误的经济学。

一次 30 分钟复制延迟事故及滚动恢复的具体过程。

t=0:    Daily analytics job kicks off on primary's reporting replica.
        Query: UPDATE billing_summary SET rollup = (...) WHERE month = '2026-04'
        Touches 50M rows; held in one transaction.
t=2min: Replica's WAL apply slows — the long-held snapshot blocks vacuum,
        and the bulk update generates 8 GB of binlog/WAL events.
        Replication lag begins growing at ~2s per real second.
t=10min: Lag = 16 minutes. Read-after-write breaks for any user whose request
        hit this replica. Customer-facing impact: stale data in dashboards
        showing yesterday's number.
t=12min: Pager fires (replica_lag > 5min for >2min).
        On-call inspects: pg_stat_replication shows replay_lsn frozen,
        pg_stat_activity shows a 12-min-old query on the replica.
t=15min: Cancel the long query on the replica (pg_cancel_backend(pid)).
        Replication resumes. Lag plateaus at ~17min.
t=15min: Reroute read-after-write traffic to the primary (feature flag).
        Customer impact lifted; latency on primary rises 20% but tolerable.
t=15min – t=45min: Replica catches up at ~30s per real second 
        (apply rate > network rate while behind; warm caches help).
        Lag drains.

t=46min: Post-incident: move the analytics job to a dedicated reporting replica
        with hot_standby_feedback=off and statement_timeout=10min.
        Add monitoring on long-running transactions on replicas.

这一模式值得关注：延迟的根源很少是网络，几乎总是副本上的批量操作，或是落后于写入突发主节点的单线程应用。

8.5 故障转移编排

自动主节点晋升究竟是如何发生的。每个工具都实现了相同的核心流程，但在运维权衡上各有不同。

Orchestrator（GitHub 的 MySQL 拓扑管理器）。 作为独立的 HA 集群运行（本身由 etcd/Raft 支撑）。每隔几秒轮询每个 MySQL 节点。检测到主节点连续 3 次以上健康检查失败（约 30s）不可达时，执行以下操作：

   1. Detection: 3 consecutive health-check failures over 30s.
   2. Election: query all replicas for their GTID_EXECUTED.
                 Pick the one with the largest GTID (= most data).
                 Tiebreak by lag / topology preferences.
   3. Fencing: mark old primary as read-only via STONITH or 
               network fence (if reachable); reject its writes.
   4. Promotion: SET GLOBAL read_only=OFF on chosen replica.
                 Re-point all other replicas: STOP SLAVE / CHANGE MASTER TO 
                 the new primary.
   5. Service discovery: update DNS / VIP / Vitess routing layer.
   6. Re-attach old primary as a replica once recovered.

端到端墙钟时间：约 20–60s。fencing 步骤是关键所在——见 §8.6。

Patroni（PostgreSQL）。 使用 DCS（分布式配置存储） ——etcd、Consul 或 ZooKeeper——通过 DCS 自身的共识机制进行 leader 选举。每个由 Patroni 管理的 PG 节点在 DCS 中持有一个租约；leader 的租约是谁是主节点的唯一事实来源。租约到期（通常 30s 未续期）后，其他节点竞争获取租约；LSN 最高的节点胜出。

   1. Old primary's lease in etcd expires (failed to renew for 30s).
   2. All replicas observe lease expiry, query the DCS.
   3. Each replica reports its pg_last_wal_replay_lsn.
   4. Highest-LSN replica claims the leader key in etcd 
       (atomic compare-and-swap).
   5. Patroni on the winner runs pg_ctl promote.
   6. Other replicas re-point: pg_rewind or fresh basebackup,
       then standby.signal + primary_conninfo pointing at new primary.
   7. HAProxy / PgBouncer routes connections to new primary via 
       Patroni's REST API (/master endpoint).

AWS RDS Multi-AZ。 另一个 AZ 中的同步 standby。主节点故障时，AWS 将 DNS CNAME 从主节点切换到 standby（端到端约 30–60s）。对使用 RDS 端点的应用程序透明。存储通过 EBS 级别的镜像同步复制，而非在引擎层——两个节点在每次提交时都向两个 AZ 写入。

Aurora。 6 路复制的存储层（写入需 4/6 quorum，读取需 3/6 quorum），与计算层解耦。故障转移只需在现有存储卷上启动一个新的计算节点——无需数据重建，无需复制追赶。通常 <30s。

MHA（MySQL Master High Availability）——旧版工具。 一个基于 Perl 的工具，已废弃超过 5 年，但在较旧的部署中仍可见到。已基本被 Orchestrator 取代。

8.6 脑裂防护

两个可写主节点同时存在 = 数据损坏。双方各自接受冲突写入；互相无法复制；在调和时，一方的数据集会悄无声息地覆盖另一方。

Fencing tokens（隔离令牌）。 单调递增的 epoch 编号（也称 generation、term 或 fencing_token）。每次主节点写入都携带其 epoch。副本拒绝来自低于自身 epoch 的写入。晋升时，新主节点的 epoch 被设置为 old_epoch + 1。原主节点如果仍存活并在写入，其 epoch 已经过期；其写入会在副本层被拒绝。这是关键原语——没有它，快速故障转移就是不安全的。

   epoch=17, primary=A    epoch=18, primary=B (after failover)

   t=0: A writes "balance=400" with epoch=17 → replicas accept.
   t=1: Network partition. A still thinks it's primary.
   t=2: Orchestrator promotes B with epoch=18. Replicas now expect 18.
   t=3: A writes "balance=300" with epoch=17 → REJECTED.
   t=4: B writes "balance=350" with epoch=18 → ACCEPTED.

STONITH（Shoot The Other Node In The Head，击毙另一节点）。 硬件级隔离。编排器的故障转移步骤包括"通过 IPMI/BMC 对疑似主节点执行电源循环"或"通过 SNMP 拔掉其交换机端口"。简单粗暴但无歧义——断电的节点无法写入。用于传统本地部署。云端等价操作：通过 EC2 API 终止实例。

基于 Quorum 的决策（Raft/Paxos）。 共识数据库在没有多数派的情况下永远不会选出新的主节点。若分区将旧主节点隔离，拥有多数派的分区会选出新主节点；没有多数派的分区拒绝选举。旧主节点观察到自身失去 quorum 后，会主动降级（loss-of-quorum 时主动下台）。应用层无需隔离——协议本身就是隔离机制。

Watchdog 计时器。 主节点自身监控其与编排器/DCS 的连接。若心跳中断超过 N 秒，则自我降级为只读。这是对"编排器已晋升新主节点而旧主节点仍存活且毫不知情"这一场景的双重保险。

具体的脑裂场景。 AZ-A（主节点）与 AZ-B（编排器 + 副本）之间发生网络分区。AZ-A 的主节点本身健康但被隔离；AZ-B 的编排器无法连接它，宣告其宕机，并晋升一个副本。

Without fencing:
   t=0: Primary in AZ-A keeps accepting writes from local apps in AZ-A.
   t=10s: Orchestrator promotes replica in AZ-B. Apps in AZ-B write to AZ-B.
   t=10s — t=120s: Network heals. Two primaries existed for 110s.
       Conflict: account_id=42 written on AZ-A (balance=400) 
       and on AZ-B (balance=350). 
       On reconciliation, AZ-A's later commits override → losing AZ-B's debit. 
       Data corruption.

With fencing:
   t=0: Primary in AZ-A keeps accepting writes locally, but its writes 
        can't reach replicas (partition).
   t=10s: Orchestrator promotes replica in AZ-B with epoch=18.
        Apps in AZ-A still connected to AZ-A primary attempt writes.
   t=10s+: The application's connection-pool layer in AZ-A has stale 
        routing — but the binlog tail-shipping fence ensures AZ-A's 
        writes since t=0 (with stale epoch=17) cannot be replicated, 
        and the AZ-A primary's writes are stranded on local disk.
   t=120s: Partition heals. AZ-A primary observes the new epoch=18, 
        demotes itself to replica, and *discards* its post-partition writes 
        (or surfaces them as conflicts requiring manual resolution).
        No corruption — the writes that were *acked* during the partition 
        were rejected at the replica fence and never appeared as committed
        to anyone outside AZ-A.

教训：隔离机制才是让"我们有自动故障转移"不成为"我们有自动数据丢失"的保障。

8.7 备份策略

备份是数据库的最后一道时间机器。

逻辑备份（mysqldump、pg_dump）。 用于重建 schema 和数据的 SQL 语句。 pg_dump --format=custom 生成可移植的归档； mysqldump --single-transaction 生成一致性快照。可跨主要版本移植；可还原到不同引擎。 慢 恢复时：1 TB 的逻辑 dump 恢复需要 10+ 小时，因为它逐行经过 SQL 解析器、计划器和执行器重放。适合 schema 小于约 100 GB 的场景及跨版本迁移。

物理备份（MySQL 用 xtrabackup/Percona，PostgreSQL 用 pg_basebackup）。 磁盘上的数据文件副本加上足够的 WAL 以保证一致性。xtrabackup 在记录并发 WAL 写入的同时热复制 InnoDB 表空间文件；pg_basebackup 通过 walsender 接口流式传输整个 PG 数据目录。 快速恢复 ——只是磁盘 I/O。1 TB 物理备份恢复约需 ~3 小时（受限于顺序磁盘写入速率，商用 NVMe 约为 ~300 MB/s）。 与引擎版本绑定：将 InnoDB 8.0 备份还原到 8.1 实例需要特殊处理。

增量 / WAL 归档备份。 每周或每晚进行一次基础备份（完整物理备份）+ 将每个 WAL 段滚出时持续归档到 S3/GCS。恢复 = 基础备份 + 从基础备份 LSN 重放 WAL 到目标恢复点。存储成本主要由基础备份决定；WAL 归档是增量的。

3-2-1 规则。 3 份数据副本，2 种不同存储介质，1 份异地备份。对于 17 TB 支付分片：生产数据（EBS 上 1 份）+ 同区域快照（第 2 份，不同物理介质）+ 跨区域 S3 归档（第 3 份，异地）。加上实时副本作为额外冗余。

静态加密。 用于合规（PCI-DSS、HIPAA、SOC2）。可在存储层（EBS 加密、S3 SSE-KMS）或应用层（使用客户自管密钥的列加密）进行。备份加密使用相同密钥，并增加了一个额外风险——"如果在加密备份异地存储期间丢失了 KMS 密钥会怎样"——需要对 KMS 密钥本身进行严格访问控制的备份。

具体的生产配置。 一个支付数据库，17 TB/分片，256 个分片：

   Per-shard backup plan:
   - Nightly full physical backup via xtrabackup → S3 in same region.
     Cost: ~15 TB compressed (xtrabackup compresses InnoDB pages well).
     Wall-clock: ~3 hours per shard, run sequentially across shards using
     a backup orchestrator (one shard at a time per AZ to limit network).
   - Continuous binlog archive: every binlog segment rolls off → S3 within 5min.
     Storage cost: ~3 TB/day across all shards; retention 30 days = 90 TB.
   - Retention: 30 days of nightly + 7 yearly snapshots (anniversary backups)
     for legal/compliance.
   - Multi-region replication of the backup bucket itself via S3 CRR.
   - Quarterly disaster-recovery drill: restore one shard's backup into a 
     fresh cluster, replay binlog to a specific GTID, verify row counts and 
     checksums against production replica. Document RTO.

演练是关键环节。从未恢复过的备份不是备份；它们只是充满希望的加密文件。

8.8 时间点恢复（PITR）

周二上午 11 点的"drop table production"场景。一位工程师运行维护脚本时忘记设置正确的数据库上下文； DROP TABLE accounts 在生产环境中执行。副本忠实地复制了这次 drop。表在所有地方都消失了。

PITR 是恢复到 正是 坏事发生前那一刻的过程。

Goal: restore to 2026-05-22 10:59:59 UTC (1 second before DROP TABLE at 11:00:00).

Steps:
  1. Identify exact target. Inspect binlog/WAL for the bad event.
     MySQL: mysqlbinlog binlog.0001042 | grep -B5 'DROP TABLE accounts'
     → found at GTID srv-uuid:1042087 at timestamp 2026-05-22 11:00:00.123
     → recover to GTID srv-uuid:1042086 (the previous transaction)

  2. Bring up a recovery cluster (separate from production).

  3. Restore the most recent full physical backup before the event 
     (last night's 02:00 UTC backup).
     → 1 TB restore via xtrabackup-restore in 3 hours.

  4. Replay WAL/binlog forward from backup LSN to recovery target.
     MySQL: configure replication from binlog archive in S3,
            STOP SLAVE at gtid_executed = srv-uuid:1042086.
     PostgreSQL: recovery.conf / recovery_target_time = '2026-05-22 10:59:59'
            or recovery_target_lsn = '0/3A12F800'.
     Wall-clock: replay rate ~10–30 MB/s for WAL through SQL apply.
     With 9 hours of binlog (last backup at 02:00 → recovery target 11:00),
     and a 50 MB/s primary write rate, that's 1.6 TB of WAL to replay.
     ~12–15 hours.

  5. Stop precisely. Verify the dropped table exists with row count matching
     prod's state at 10:59:59 (compare to checksums or daily count audit).

  6. Promote the recovery cluster (or just extract the needed table).
     Strategy A: copy the recovered table back into the production cluster 
                 via mysqldump or pg_dump (faster than full failover).
     Strategy B: failover production to the recovery cluster (RTO = the
                 12 hours just spent).

  7. Document the gap: any writes between 11:00:00 (the drop) and "now" 
     that touched the dropped table are lost; reconstruct from upstream 
     sources or accept the gap.

总时间成本。 ~每 TB 基础恢复 1 小时 + WAL 重放时间（~10–30 MB/s 重放速率 × 自基础备份以来的 WAL 量）。对于自上次备份以来有 9 小时 WAL 的 1 TB 数据库：约 3 小时恢复 + 约 12 小时重放 = 约 15 小时。对于 PB 级，需要分片并行恢复 + WAL 重放——对 256 个分片并行恢复，每个约 3 小时，而非串行。

真实案例。 GitHub 2012 年的中断事故涉及在数据库损坏后从 binlog 恢复；恢复耗时约 6 小时。GitLab 2017 年事故（rm -rf 生产目录被删除）——由于备份长达数月静默损坏，丢失了 6 小时的工作。规律：PITR 能否奏效，当且仅当备份经过测试且 binlog 归档完整。

8.9 跨区域灾难恢复

能在失去一个 区域 ——而不仅仅是一个 AZ——后存活下来的是什么。

主动-被动（另一区域的温备）。 从主区域到另一区域被动集群的异步复制。~50–200ms 延迟（跨区域 RTT）。RPO ≈ 延迟窗口（区域丢失瞬间在途的任何数据都会丢失）。RTO = 手动故障转移过程：约 10–30 分钟，含人工决策时间。

   us-east-1 (active)                       us-west-2 (passive)
   ─────────────────                        ──────────────────
   Primary (semi-sync within region)        Standby cluster (async receive)
       │                                            ▲
       │ ─── async binlog/WAL stream ──────────────►│
       │     (cross-region, 50–80ms RTT)            │
       ▼                                            │
   Apps in us-east-1 ──── write ──────►             │
                                                    │
                                          (No app traffic; cold standby)
   On regional loss:
   1. Page on-call. Triage: is this really regional loss or just AZ?
   2. Manual: update DNS/route config to direct apps to us-west-2 primary.
   3. Promote us-west-2 standby to read-write.
   4. Reconfigure CDC pipelines to read from us-west-2 binlog/WAL.
   5. Communicate RPO (lag window) to customer; reconstruct lost txns 
      from upstream sources where possible (idempotency keys help).

主动-主动（多区域接受写入）。 写入路由到最近的区域以降低延迟；需要冲突解决。三种方案：

  • LWW（Last Write Wins）结合向量时钟。 每次写入带有一个 (region, logical_clock)向量标签。冲突尽可能按因果关系解决，退而求其次则按时间戳。对货币操作存在静默损坏——因为两个区域都写了同一行，较旧的时间戳获胜，导致一笔借记丢失。OLTP 场景中应避免使用。
  • CRDTs（无冲突复制数据类型）。 可相加的计数器（G-counters、PN-counters）；可合并的集合（G-sets、OR-sets）；具有特定字段 last-writer-wins 的 map 和寄存器。最终一致性是其内在特性。适用于点赞计数、在线状态指示器、协同文档（Yjs、Automerge）。不适用于"余额必须等于贷记减借记"。
  • 按区域分区（地理分区）。 每个区域拥有一组不相交的分区。针对区域 X 所属分区的所有写入都发往 X；冲突不可能发生，因为不会有两个区域写同一行。Spanner 的地理分区、CockroachDB 的局部性感知分区。这是最干净的主动-主动模式，但要求按每租户稳定的键（用户的归属区域）进行分区。

同步跨区域的延迟代价。 跨 us-east-1 / us-west-2 / eu-west-1 的 Raft 提交（实现真正 RPO=0 的"三区域"部署）需要承受最慢的两两 RTT。美东到美西约 ~70ms；美东到欧西约 ~100ms；在这种几何结构下，写入的 quorum 需要 3 中取 2，最优情况下约 ~70ms。每次提交约 ~70–100ms。对于 10k 提交/秒的 OLTP，这在结构上不可行——无法保持足够多的事务在途以使系统饱和。

具体的跨区域配置。

  • Stripe：主节点在美国，另一区域有异步备用节点。RPO 以秒计。面向客户的 OLTP 在单区域以低延迟提交运行。
  • Notion：PostgreSQL 跨区域异步副本用于 DR。主动-被动。
  • OceanBase at Alipay：跨北京、上海、深圳的 3 区域 Paxos quorum。RPO=0，但由于中国境内数据中心间 RTT 较短，提交延迟约 ~30–50ms。
  • CockroachDB：适用时按 region 列进行地理分区；跨区域回退使用纯异步。

"你真的需要主动-主动吗"的质疑。 对于约 95% 的 OLTP 工作负载，RPO 50ms、RTO 30 分钟的主动-被动已经足够。同步跨区域的延迟代价（每次提交 50–200ms）和主动-主动的冲突解决复杂性（损坏风险、自定义冲突处理器、应用层分区）几乎从不值得。除非有 特定 需求（监管数据驻留要求在本区域写入、两个区域用户的硬延迟目标）迫使使用主动-主动，否则选择主动-被动。

8.10 RTO/RPO 目标与架构

RTO（恢复时间目标） = 故障后恢复服务所需的时间。 RPO（恢复点目标） = 可以接受的数据丢失量。有意识地将二者配对；每降一档成本增加一个数量级。

RTO RPO 架构 成本倍数
30s 0 另一个 AZ 中的半同步副本 + 自动故障转移（Orchestrator/Patroni）。跨 AZ 提交延迟。 5–10x（副本）
5min 0 同上，但在自动检测后进行人工故障转移——由人工把关晋升。 4–8x
1hr 15min 跨区域异步副本 + 通过 WAL 归档实现 PITR。延迟 = 最坏情况下的 RPO。 2–4x
24hr 24hr 每晚逻辑备份到 S3。无复制。恢复为不含 WAL 的顺序恢复。 1.2x
30s 0 （跨区域）跨区域 Paxos/Raft quorum。NewSQL 或自定义共识。 10–20x + 延迟

95% 的选择 对于有规模的生产 OLTP：区域内 RTO=30s RPO=0（半同步 + 跨 AZ 故障转移），跨区域 RTO=15min RPO≈延迟窗口（用于 DR 的异步跨区域）。

成本直觉：RTO 每"好一档"意味着自动化好一档；RPO 每"好一档"意味着复制模式好一档。从 RPO=15min 降至 RPO=0，意味着从异步转为半同步——跨区域带宽和副本容量增加 10 倍。从区域内 RPO=0 降至跨区域 RPO=0，意味着跨区域共识——100ms 提交延迟代价，可能对吞吐量造成致命影响。

综合：RTO/RPO 是产品决策，而非技术决策。根据每分钟中断和每秒数据丢失给业务带来的损失来选定目标；按此构建满足目标的架构；不要过度工程化。

§9. 故障模式演练

每种故障逐一命名，附持久性要点。

9.1 主节点在事务中途崩溃。 T 正在写页面；redo 记录已缓冲但未刷盘。重启时，数据库从上次检查点扫描 redo 日志，发现 T 没有 COMMIT，通过 undo 日志回滚。锁释放。T 从未发生。客户端超时，用幂等键重试。 持久性要点：T 无数据丢失；结果正确。

9.2 主节点在提交的 fsync 之后、binlog ack 到副本之前崩溃。 T 在本地已提交（redo 日志在 LSN 1080 处已 fsync）。binlog 事件已生成；半同步等待进行中或刚刚完成。重启时，T 通过 redo 重放并出现在表中。binlog 也已持久化（sync_binlog=1）。副本读取事件并应用；若已接收过，复制在事件级别是幂等的。若客户端 ack 已发送，T 是持久的。若未发送，客户端用幂等键重试，看到幂等行中的 status=committed ，返回之前的结果。 持久性要点：redo 日志 + binlog 在 LSN 1080 处 fsync。

9.3 将主节点与副本 quorum 隔离的网络分区。 主节点可在本地写入，但半同步正在阻塞（没有副本可达）。有了 wait_no_slave=ON，新提交会停滞。主节点拒绝写入；客户端看到超时。分区恢复后，复制追赶上来。若持续时间较长，编排器可能会降级被隔离的主节点并晋升一个可达的副本。为防止脑裂：隔离令牌（单调 epoch）确保被降级的主节点无法 ack 新写入；基础设施层的 STONITH（击毙另一节点）是双重保险。 持久性要点：最近一次取得副本 ack 的提交；被降级主节点上超过此点的任何内容都会被回滚。

9.4 主节点永久丢失。 磁盘故障或主机起火。副本存活。编排器晋升具有最高 binlog/WAL 位置的副本（MySQL 中为SHOW SLAVE STATUS ；PostgreSQL 中为 pg_last_wal_replay_lsn() ）。其他副本重新指向。一个新节点被提供，通过快照恢复，添加为第三个副本。达到完整冗余的实际时间：约 30 分钟；服务在 <30 秒内以降级冗余恢复。 持久性要点：最后一次半同步已 ack 的提交。通常情况下 RPO=0。

9.5 整个 AZ 永久丢失。 AZ-A 中的主节点 + 1 个副本丢失。AZ-B 中 1 个副本存活。晋升 AZ-B 副本。从快照在 AZ-C 和 AZ-D 中各提供两个新副本。服务在重建期间以降级状态（单副本）运行约 30 分钟。 持久性要点：存活副本的最后一次已应用事件。

9.6 静默磁盘损坏（宇宙射线击中 B+ tree 页面）。 主节点上的某个页面有一个翻转的比特；读取返回错误数据。InnoDB 在每个页面头/尾写入校验和；PostgreSQL 在使用 --data-checksums初始化时写入每页校验和。读取时，不匹配会引发 I/O 错误；行读取失败；编排器路由到副本。损坏的页面通过将 binlog/WAL 重放到新初始化的区域来重建，或从健康副本进行单页恢复。 持久性要点：任何拥有未损坏页面的副本。没有校验和，静默损坏会悄悄违反不变量。在生产中启用，而不仅仅是开发环境。

§10. 为何不用显而易见的更简单替代方案

"直接用平面文件。" 重放从创世以来的每个事件来回答"账户 42 的余额？"需要每次查询 100ms。为了加速，你会物化当前状态——你发明了数据库。为了让物化崩溃安全，你会添加预写日志——你发明了 WAL。为了处理并发更新，你会添加锁——你发明了事务管理器。这个朴素的替代方案 变成 经过三次重建后成为了数据库。

"直接把 Redis 当主存储。" 内存成本：17 PB 的支付数据放在内存中在财务上是不合理的（磁盘便宜 50–100 倍）。持久性：默认 AOF（Append-Only File）fsync 是 everysec，崩溃时可能丢失最多 1 秒的已 ack 写入； always 会使 Redis 慢约 10 倍。事务：MULTI/EXEC 在条件失败时无回滚；Lua 提供原子性但不提供跨键隔离。复制：默认异步。Redis 是真实数据库前面正确的 fast-path layer（快速路径层） ——缓存、幂等查找、限流。不是保存关键状态的持久主存储。

"直接在所有分片上使用全局分布式事务。" 全局锁序列化整个工作负载。在 583k 写入/秒时，持有全局锁 100µs 将吞吐量上限降至 10k/秒——低于需求 60 倍。分片的意义正是打破竞争锁瓶颈。

"直接用 Cassandra，它可以水平扩展。" 没有 LWT（轻量级事务），你无法保证每键有序——并发写入按时间戳解决（LWW——Last Write Wins），因此时钟偏差会静默丢失一笔借记。有了 LWT，你需要支付 Paxos 轮次（4 条消息，数据中心内每条约 5ms），将每分区 LWT 上限限制在 200–500/秒。若要在两行之间原子转账，你需要在 LWT 之上构建 2PC——叠加昂贵的协议。真实失败案例：两个并发借记都执行 CAS balance=500 与 balance=400。一个成功，一个失败并重试。在竞争下，重试风暴将吞吐量崩溃至几百/秒——与 MySQL 行锁相同的瓶颈，但基础设施成本是 4 倍。Cassandra 适合时序数据、扇出、IoT；不适合守恒场景。

"用对象存储作为数据库。" 列举时的最终一致性，无跨对象事务，无索引，每对象开销使得按行工作负载在经济上荒谬。对象存储是正确的 archive tier（归档层） （Parquet、blob）；不是事务性底层。

§11. 扩展维度

事务性数据库在两个维度上扩展，需要不同的解决方案。

垂直扩展。 更大的机器：更多核心、RAM、更快的 SSD。单台 96-vCPU、768 GB RAM、NVMe 的 PostgreSQL 可持续支撑约 ~30–50k 提交/秒。 拐点：30–50k 是商用硬件的上限。超过此值，锁管理器和 WAL fsync 会成为结构性瓶颈。你可以在这上面运行很长时间（Notion 到 2021 年，Hacker News 到 2016 年），但最终你需要分片。

水平扩展：读副本。 添加异步或半同步副本；路由读密集型查询；主节点处理写入和读后写。 拐点：当读写比高（>10:1）时，副本才有帮助。支付场景读写比为 1:1，收益甚微；个人资料渲染场景读写比为 1000:1，8 个副本可实现约 ~8 倍读容量的优美扩展。副本延迟是运维问题。

水平扩展：分片（类型 1 — 均匀增长）。 当更多实体（用户、账户、文档）需要超过单主节点能处理的容量时，按实体键进行分区。 hash(key) % N；按 2 的幂次取模以实现干净的重新分片（N → 2N）。每个分片有自己的主节点和副本。路由器（Vitess、ProxySQL、应用层）将查询映射到分片。 拐点：当每分片写入速率达到可持续上限的 30–50% 时，安排下一次拆分。512 → 1024 通过双写 + 回填 + 切换需要数周，但全程在线。代价是永久性的：跨分片 join 是 N×M，跨分片事务需要 2PC，schema 迁移需跨分片协调。

水平扩展：热键（类型 2 — 同一实体，每实体更高速率）。 更难的维度。同一实体（Twitch 主播、秒杀 SKU、病毒式话题标签）的写入速率不断提升。仅靠分片无济于事——热行仍然在同一个分片上。 解决方案：在实体内部进行写入分片（§7.1）。 拐点：单行持续超过 1k 写入/秒。超出写入分片范围后， 基于日志的预聚合 ——追加到按实体分区的 Kafka 主题，消费者在 100ms 窗口内批处理并执行单次 SUM 更新。失去严格的逐事件排序（对打赏、点赞、点击可接受）；保留最终总和。对单实体超过 100k 写入/秒时有用。

两个旋钮都需要。 单独分片（类型 1）无法解决热键问题（类型 2）。单独写入分片（类型 2）无法解决容量增长问题（类型 1）。每个大规模 OLTP 部署最终都会同时用到两者。

11.1 分片策略深度解析

选择 如何 分片方式比选择 是否 分片影响更为深远。三大类。

Hash 分片。 shard_id = hash(partition_key) % N。无论键分布如何，都能将键均匀分散到 N 个分片上。适用于高基数键（account_id、user_id），且不需要对分区键进行范围查询。代价： N 变化时重新平衡代价高昂。从 512 → 513 个分片会重新映射约 ~99.8% 的键；几乎每行都要移动。标准缓解措施是 将 N 取整为 2 的幂次 ——按 hash(key) % 1024 分片允许将每个分片干净地对半拆分（mod 1024 → mod 2048； (hash mod 2048) < 1024 的键原地不动，其他键移到新的兄弟分片）。拆分只移动下半部分数据，而不是整个数据集。Vitess 的 KeyspaceID 以及大多数现代手写分片映射都使用这种模式。Pinterest 的 4096 个逻辑 MySQL 分片从 256 开始，翻了三次倍——每次翻倍移动一半数据。

hash sharding with power-of-two N:

   N=4                      split shard 2 into 2a + 2b → N=5? NO, → N=8
   ─────                    ────────────────────────────────────
   key → hash → mod 4       key → hash → mod 8
                            keys where (hash mod 4)==2 split:
   shard 0                     (hash mod 8)==2 → stays in old shard 2 (renamed 2a)
   shard 1                     (hash mod 8)==6 → moves to new shard 2b
   shard 2  ← splitting     other shards unchanged.
   shard 3

Range 分片。 分片拥有分区键的连续范围。 [0, 1M) → shard 0, [1M, 2M) → shard 1, ...。适用于需要范围查询的顺序键（例如时序数据、审计日志）。 热点风险 ：对于时间形态的数据——如果你按 events 以 event_id 分片，且 ID 是单调的，所有新插入都会打到最高范围分片，而旧分片逐渐冷却。这就是"最后一个分片热"问题。缓解措施：对复合键的前缀进行哈希，或按 (tenant_id, event_id) 分片，这样租户内的范围有效，但租户分散。

基于目录（查找表）的分片。 一个单独的元数据表映射 partition_key → shard_id。灵活：任何重新平衡只需更新目录。初创公司早期常用，后来往往后悔。代价：

  • 额外一跳。 每次查询都要查目录（有缓存，但缓存未命中就是多一跳）。
  • 目录变成关键节点 ——它现在是单点故障，自身往往随后也被分片（无穷无尽的乌龟）。
  • 缓存一致性。 过期的目录条目会路由到错误的分片。分片迁移时需要驱逐。

适合分区按租户划分（每个租户在目录中有一个"归属分片"）且目录足够小可以在每台应用服务器上保存在内存中的场景。

Vitess VReplication 用于在线重新分片。 从 N 个分片迁移到 2N 个（或 N → M 任意映射）的操作模式。分三个阶段：

   Phase 1 — Dual write + backfill
   ───────────────────────────────
   - Old shard (N) keeps accepting writes (and continues to be the source of truth).
   - Set up VReplication: stream binlog from old shard → new shard.
   - Backfill historical data: copy all rows from old shard to new shards 
     (partitioned by the new sharding function).
   - When backfill completes, new shards are caught up to the binlog tail.
   - Continue dual-shipping new writes (old + new).

   Phase 2 — Switch reads
   ──────────────────────
   - Verify new shards are consistent with old (row counts, checksums).
   - Update routing layer to read from new shards.
   - Old shards still receive writes (continuing dual-write for safety).
   - Observe for several days — read traffic on new, can revert if issues.

   Phase 3 — Switch writes + decommission
   ──────────────────────────────────────
   - Atomic cutover: writes now go to new shards only.
   - Old shards become read-only "archive" for a grace period (days).
   - After verification window, decommission old shards.

端到端来看，Slack 规模下的 512 → 1024 重新分片需要 数周时间 但数据库在整个过程中保持在线。Vitess 的实现负责双写、回填编排、切换和回滚机制——从零构建这套系统大约需要 6 个月的工程工作量。

11.2 分片键选择：自增 vs UUID

分片键的选择存在隐藏陷阱。

顺序分片键。 如果分片键（或其前缀）是自增的，所有新行都会堆积到最后一个分片上。一张以 posts 进行分片的表没问题——哈希会均匀分散——但一张使用 hash(post_id) % 1024 单调递增 ID 的表会导致最后一个分片成为热点。初创公司的常见做法："我们按 posts 进行分片的表没问题——哈希会均匀分散——但一张使用 post_id range 分片，基于范围"——目前没问题，但当某个租户的流量突增全部打到同一分片时， customer_id，基于范围"——目前没问题，但等到某个租户的流量突增只打到一个分片时后果会很严重。

以 UUIDv4 的哈希作为分片键。 UUIDv4 是均匀随机的； hash(uuid) % N 分布均匀。没有热点末尾分片，也不会因键生成而导致分布倾斜。代价： 无聚集性。两个连续的客户注册会落到不同的分片。跨客户查询会散射到 N 个分片。

以自然业务键的哈希作为分片键。 hash(customer_id) % N 其中 customer_id 本身是 UUIDv4 或非单调递增的标识符。某个客户的所有数据都存储在同一分片上；跨客户查询仍然是散射-聚合，但单客户工作流（如加载客户仪表盘）保持本地访问。

UUIDv7 / ULID 作为主键 + customer_id 作为分片键。 现代化模式。UUIDv7 具有 48 位时间戳前缀，后跟随机位——全局唯一、按时间排序（保留 B+ tree 插入局部性），但不可预测。将其用作主键以保持 B+ tree 友好性。使用 独立 分片键（customer_id，哈希化）来控制分布。两全其美。

   Bad:  shard by post_id range (post_id auto-increment)
         → all new posts on last shard, hot.

   OK:   shard by hash(post_id) — UUIDv4 post_id
         → spreads evenly, but per-customer queries scatter.

   Good: PK = post_id UUIDv7 (B+ tree friendly), 
         shard by hash(customer_id) — per-customer locality preserved.

11.3 多区域

最后一个扩展维度。详细内容见 §8.9；以下是与扩展相关的摘要。

主动-被动（温备用）。 主节点在一个区域，另一个区域有异步跨区域副本。RPO ≈ 复制延迟窗口（通常为秒到分钟）。RTO ≈ 手动故障转移时间（10–30 分钟）。成本：1 个额外集群，加上异步流的跨区域带宽（繁忙分片约 50 MB/s）。 95% 的选择 适用于需要跨区域灾难恢复的 OLTP。

主动-主动（多区域写入）。 三种变体——带向量时钟的 LWW（存在静默数据损坏风险，OLTP 应避免）、CRDT（最终一致性，适用于计数器/集合/协同文档，不适用于守恒性约束），或按区域分区（地理分区——每个区域拥有不相交的键，不存在冲突）。地理分区是最简洁的主动-主动方案，但需要键具有自然的区域归属。

跨区域的 NewSQL with Raft。 通过共识实现真正的跨区域线性一致性。每个事务提交延迟 50–200ms。适用于 必须 有全局序列化点的工作负载（全局限流器、跨区域共享库存）。不适用于任何对延迟敏感的 OLTP（每次提交 100ms 的上限意味着每个键的写入速率上限为 10/秒）。

反对意见。 大多数 OLTP 工作负载——支付、用户画像、电子商务、SaaS 文档——并不需要主动-主动。延迟税和冲突解决复杂性很少值得相应的工程和运营成本。默认方案是主动-被动加手动故障转移；仅在有具体需求（监管合规的数据驻留要求、两个区域内严格的延迟目标）时才升级为主动-主动。

§12. 运营关注点

前面的章节涵盖了存储引擎（§4）、难题（§7）、复制（§8）和扩展维度（§11）。本节介绍运营细节——正是这些细节决定了相关技术的部署能否在周日凌晨 3 点保持正常运行，还是在可预测的负载下崩溃。这些是数据库引擎本身无法解决的问题，但每个事务型系统的运营者都必须面对。

12.1 连接池

100–300 连接上限对于 PostgreSQL 设计来说是真实存在且至关重要的约束。

PostgreSQL 为何在 100–300 个连接附近遭遇瓶颈。 PostgreSQL 采用 每连接一个进程 的模型（fork() accept 时 fork）。每个连接是一个独立的操作系统进程，消耗约 10 MB 内存（进程栈 + 每个后端缓冲区 + work_mem 分配）。300 个连接仅进程开销就需要 3 GB，再加上每个后端的 work_mem（默认 4 MB，随并发查询扩展）。在 32 GB 实例上，超过约 300 个活跃连接后，上下文切换持续进行，OS 调度器频繁抖动，内部数据结构（proc 数组、pg_stat）上的锁竞争成为瓶颈。

MySQL 更轻量，但仍有上限。 MySQL 采用 每连接一个线程 ——线程共享进程，每个线程消耗约 256 KB 栈 + 每线程缓冲区。1000 个线程比 1000 个进程更易承受。但 MySQL 仍受制于内部锁管理器的记账开销、线程池调度器以及 InnoDB 缓冲池的 latch 竞争。默认 max_connections = 151；实际部署中会调高至 1000–5000。

会咬你一口的数学题。 一个有 100 台服务器、每台服务器连接池 50 个连接的应用 = 5000 个连接。一个设置了 max_connections=500 的 PostgreSQL 主节点在第 501 个连接时会拒绝新连接。应用开始抛出 FATAL: sorry, too many clients。典型的初创公司死亡螺旋。

解决方案。

PgBouncer 是 PostgreSQL 的标准答案。三种池化模式：

模式 一个客户端对应一个服务端连接，用于... 兼容特性 池化比例
Session（会话模式） 完整客户端会话（登录 → 登出） 所有特性（prepared stmts、SET LOCAL、临时表） 1:1 ish
Transaction（事务模式） 单个事务（BEGIN → COMMIT) 大多数特性，但破坏 prepared stmts 和 SET LOCAL 100:1
Statement（语句模式） 单条语句 仅无状态查询（无事务、无 prepared stmts） 1000:1

事务模式是大多数工作负载的最佳选择。 应用向 PgBouncer 开启 5000 个逻辑连接；PgBouncer 将它们复用到一个通向 PostgreSQL 的 50 个连接的池上，实现 100 倍缩减。注意：使用保持连接级状态的 PostgreSQL 特性的应用（SET LOCAL search_path、使用 PREPARE/EXECUTE 的服务端 prepared statements、临时表、跨事务的咨询锁）在事务模式下会出问题——它们需要会话模式，或者停止使用这些特性。

ProxySQL 用于 MySQL。原理相同——将客户端连接复用到更小的后端连接池上。还增加了查询路由（将读请求发往副本，写请求发往主节点）、查询重写和限流功能。

RDS Proxy 是 AWS 托管的 PgBouncer/ProxySQL 等价物。对 Lambda 友好（Lambda 容易冲击连接池）。

具体数字。

   Before pooling:
     100 app servers × 50 conn each = 5000 connections to PostgreSQL.
     PostgreSQL max_connections=500 → refuses 90% of connection attempts.
     OR if max_connections=5000 → ~50 GB just for process overhead.

   After pooling (transaction mode):
     100 app servers × 50 conn each = 5000 connections to PgBouncer.
     PgBouncer (on the PG host or a sidecar) → 50 connections to PostgreSQL.
     PostgreSQL handles 50 backends comfortably: ~500 MB process overhead.
     PgBouncer overhead: ~50 MB.
     100x reduction in PG-side resource usage.

为什么这很重要。 典型的初创公司故障：流量峰值期间应用从 10 个实例自动扩容到 100 个；每个实例的 30 个连接池同时打开；3000 个新连接同时连接 PostgreSQL；超过 max_connections；PG 拒绝连接；应用开始报错；告警触发。在前面加一层连接池完全可以避免这种情况。

12.2 死锁

两个事务各自持有对方所需的锁。数据库检测到循环，选出受害者，返回死锁错误。

检测。 PostgreSQL 和 MySQL 都维护一张 等待关系图：当事务 T 等待事务 T' 时，添加边 T → T'。周期性地（默认约每 1 秒； deadlock_timeout 在 PG 中）扫描是否存在循环。发现循环后，选择受害者——通常是做了最少工作的事务——并以 ERROR: deadlock detected （PG）或 Error 1213: Deadlock found （MySQL）中止该事务。

应用层策略。

  • (a) 一致的加锁顺序。 始终以规范顺序获取锁——例如，始终先锁定较小的 account_id。经典示例：
   Without consistent ordering — DEADLOCK:
     Txn A: BEGIN; UPDATE accounts WHERE id=100; UPDATE accounts WHERE id=200;
     Txn B: BEGIN; UPDATE accounts WHERE id=200; UPDATE accounts WHERE id=100;
     A holds 100, waits for 200. B holds 200, waits for 100. Cycle. Deadlock.

   With consistent ordering — SERIALIZE:
     Always lock the lower account_id first.
     Txn A wants (100, 200): locks 100 then 200.
     Txn B wants (100, 200): locks 100 then 200.
     They serialize cleanly. No deadlock.
  • (b) 死锁时重试。 死锁是暂时性的——受害者的事务被回滚，因此重试通常会成功（竞争事务届时已提交）。重试次数上限设为 3–5 次，并使用指数退避。
  • (c) 缩小事务范围。 长事务持有锁的时间更长，会增加死锁概率。将非事务性工作移到 BEGIN ... COMMIT之外。即使需要更多往返，也应使用更短的事务。
  • (d) SELECT FOR UPDATE NOWAIT / SKIP LOCKED. 对于竞争获取任务的队列工作者： SELECT FOR UPDATE SKIP LOCKED LIMIT 1 允许每个工作者获取不同的行，而不会在同一个头部竞争。PostgreSQL 在 9.5 中引入；MySQL 在 8.0 中引入。消除了队列工作者模式中的死锁。

具体示例。 一个处理 1000 笔/秒转账的支付引擎。每笔转账是 UPDATE accounts SET balance = balance - $X WHERE id = $sender; UPDATE accounts SET balance = balance + $X WHERE id = $receiver。没有一致的加锁顺序时，死锁率约为 0.1%（千分之一的转账发生死锁）；在峰值流量下，这意味着每秒 1 次死锁。使用"始终先更新 account_id 较小的账户"——零死锁。生产级支付引擎都是这样做的。

12.3 隔离级别异常详解

标准隔离级别及其各自允许的异常。

脏读 （仅限 READ UNCOMMITTED）。读者看到未提交事务的写入。如果写者回滚，读者所基于的数据"从未存在"过。PostgreSQL 拒绝实现这一级别—— SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED 被视为 READ COMMITTED。MySQL InnoDB 实现了它，但在 OLTP 中没有人使用。

不可重复读 （READ COMMITTED 允许）。在一个事务内，同一 SELECT 因另一个事务在中间提交而返回不同的行。

   T1: BEGIN; SELECT balance FROM accounts WHERE id=42;  -- returns 500
   T2: BEGIN; UPDATE accounts SET balance=400 WHERE id=42; COMMIT;
   T1: SELECT balance FROM accounts WHERE id=42;  -- returns 400 (different!)

REPEATABLE READ 可防止此问题——T1 在两次读取时看到相同的值，因为 MVCC 固定了 T1 的快照。

幻读 （READ COMMITTED 允许，标准 REPEATABLE READ 也允许；PG 和 InnoDB 的 MVCC 对 SELECT 恰好能防止此问题）。同一范围查询返回不同的 行集 ，因为另一个事务插入或删除了行。

   T1: BEGIN; SELECT COUNT(*) FROM orders WHERE customer=42;  -- returns 10
   T2: BEGIN; INSERT INTO orders (customer, ...) VALUES (42, ...); COMMIT;
   T1: SELECT COUNT(*) FROM orders WHERE customer=42;  -- standard SQL: returns 11 (phantom!)
                                                         -- PG / InnoDB MVCC: still 10

PostgreSQL 的 MVCC 和 InnoDB 的 REPEATABLE READ 均能防止普通 SELECT 的幻读。InnoDB 中的 SELECT FOR UPDATE 使用间隙锁防止幻读；PG SERIALIZABLE 使用 SSI。

丢失更新。 两个并发的 UPDATE 都读取到 100，都计算出 99，都写入 99——但从概念上讲发生了两次递减。

   T1: BEGIN; SELECT balance FROM accounts WHERE id=42;  -- 500
   T2: BEGIN; SELECT balance FROM accounts WHERE id=42;  -- 500
   T1: UPDATE accounts SET balance=400 WHERE id=42; COMMIT;  -- writes 400
   T2: UPDATE accounts SET balance=400 WHERE id=42; COMMIT;  -- writes 400

   Two debits of 100 each, but balance went 500 → 400. One debit lost.

真正的修法： UPDATE accounts SET balance = balance - 100 WHERE id=42 AND balance >= 100 （原子操作），或 SELECT FOR UPDATE 后跟计算（加锁阻塞 T2 直到 T1 提交），或 REPEATABLE READ 加上在序列化失败时显式重试。

写偏斜。 著名的异常。两个事务读取重叠数据，写入不相交的数据，但合在一起违反了某个不变量。

   Invariant: at least one doctor must be on-call.

   T1: BEGIN; SELECT COUNT(*) FROM doctors WHERE on_call=true;  -- 2 doctors on-call
   T2: BEGIN; SELECT COUNT(*) FROM doctors WHERE on_call=true;  -- still 2 (T2's snapshot)
   T1: -- Alice sees "I'm one of two, safe to go off-call"
       UPDATE doctors SET on_call=false WHERE id=Alice;  COMMIT;
   T2: -- Bob sees the same, makes the same decision
       UPDATE doctors SET on_call=false WHERE id=Bob;  COMMIT;

   Now: 0 doctors on-call. Invariant violated. No row overlap.
   REPEATABLE READ doesn't catch it — different rows updated by each txn.

REPEATABLE READ 无法捕获此问题，因为两个事务从未写入同一行。只有 SERIALIZABLE 才能防止写偏斜。PostgreSQL 的 SSI（可序列化快照隔离） 从 9.1+ 开始使用 谓词锁 ——追踪每个事务读取的谓词（WHERE on_call=true），并在某个事务的谓词与另一个事务的写入以破坏可序列化的方式相交时中止其中一个。中止表现为 ERROR: could not serialize access due to read/write dependencies ——应用重试。

SERIALIZABLE 值得付出的代价。 多行业务不变量——"至少有一名医生在岗"、"所有账户余额之和必须等于初始存款减去提款"、"同一房间内没有两个预订重叠"。如果你的不变量可以通过 UNIQUE 约束或单行 CHECK 来强制执行，则不需要 SERIALIZABLE。如果它跨越应用程序读取-决策-写入的多行数据，则需要。PG 的 SSI 与 REPEATABLE READ 相比吞吐量损耗约 10–20%；对于换来的安全性，这通常是可以接受的。

12.4 B-tree 之外的索引类型

PostgreSQL 比 MySQL 提供更丰富的索引类型，因为 GIN/GiST/BRIN 访问方法是引擎可扩展索引 API 的一部分。正确的索引类型可以使查询从 200ms 优化到 5ms。

B-tree（默认）。 等值查询和范围查询。键有序，叶页面链接以支持顺序扫描。约 80% 的情况下是正确选择。

覆盖索引。 包含 所有 查询所引用的列的 B-tree 索引，允许进行 仅索引扫描 （无需聚集树查找）。 

   Slow:
   CREATE INDEX idx_user_email ON users (email);
   SELECT name, email FROM users WHERE email = 'foo@bar';
     → index scan finds row, then heap lookup for name (2x I/O).

   Fast:
   CREATE INDEX idx_user_email_inc ON users (email) INCLUDE (name);  -- PG 11+
   SELECT name, email FROM users WHERE email = 'foo@bar';
     → all data on the index page, no heap lookup. 10x faster on hot queries.

InnoDB 的二级索引在查询列均为主键列时自然实现覆盖（因为二级叶节点包含主键）。PG 需要显式使用 INCLUDE。

部分索引。 仅对满足谓词的行建立索引。索引更小，更新更快，适用于大多数查询按该谓词过滤的场景。

   CREATE INDEX idx_active_users_email ON users (email) WHERE deleted=false;

   - Index is ~30% the size if 70% of users are non-deleted (typical).
   - Updates that mark a row deleted=true are free of index work.
   - Queries with WHERE deleted=false implicitly hit this index.

GIN（广义倒排索引）。 用于复合值——数组、JSONB、全文检索。复合值的每个"元素"都被索引；查找时找到包含该元素的匹配文档。

   CREATE INDEX idx_tags ON posts USING GIN (tags);
   SELECT * FROM posts WHERE tags @> ARRAY['postgres'];  -- "contains postgres"

   CREATE INDEX idx_meta ON posts USING GIN (meta jsonb_path_ops);
   SELECT * FROM posts WHERE meta @> '{"author":"alice"}';  -- JSONB containment

   CREATE INDEX idx_search ON posts USING GIN (to_tsvector('english', body));
   SELECT * FROM posts WHERE to_tsvector('english', body) @@ to_tsquery('claude & code');

GIN 能高效处理 @>, ?, @@ 运算符。更新比 B-tree 慢（每个被索引的值会触发多个索引条目）。

GiST（广义搜索树）。 用于几何数据（PostGIS）、范围类型、模糊匹配（三元组）。由包围盒风格结构组成的树。

   CREATE INDEX idx_geom ON locations USING GIST (geom);
   SELECT * FROM locations WHERE ST_DWithin(geom, point, 1000);  -- within 1km

   CREATE INDEX idx_period ON bookings USING GIST (period);  -- tsrange
   SELECT * FROM bookings WHERE period && '[2026-05-01, 2026-05-31)';

BRIN（块范围索引）。 用于自然聚集的数据——具有单调时间戳的时序数据。极小：每个块范围存储一个最小/最大值（例如每 128 页 = 2 MB 一块）。

   CREATE INDEX idx_created ON events USING BRIN (created_at);
   SELECT * FROM events WHERE created_at > '2026-05-01';

   - 1000x smaller than B-tree (a 10 GB B-tree → 10 MB BRIN).
   - Slightly slower point lookups (must scan an entire block range).
   - Perfect for time-series, append-only logs, IoT data.

哈希索引。 仅支持等值查询。适用场景较窄。PG 直到 PostgreSQL 10 才使哈希索引支持 WAL 日志记录和崩溃安全；在此之前是不安全的。MySQL 的 MEMORY 引擎使用哈希索引；InnoDB 的"自适应哈希索引"是自动的。实践中很少优于 B-tree，因为 B-tree 处理等值查询几乎同样快，还支持范围查询。

12.5 PostgreSQL 中的 VACUUM

PostgreSQL 的 MVCC 将死元组存储在堆中；VACUUM 回收这些空间。死元组的生命周期决定了你的表是否会单调增长或保持稳定。

死元组为何存在。 PostgreSQL 中的 UPDATE 操作是"插入新元组版本 + 将旧元组的 xmax 标记为执行更新的事务"。旧元组保留在堆中，直到 VACUUM 将其空间标记为可回收。DELETE 标记 xmax；生命周期相同。

自动清理（Autovacuum）。 在后台运行。触发条件为 n_dead_tup / n_live_tup > autovacuum_vacuum_scale_factor （默认 0.2 = 20% 死元组）。可通过 ALTER TABLE foo SET (autovacuum_vacuum_scale_factor = 0.05) 对热点表进行逐表调优。

VACUUM vs VACUUM FULL。

  • VACUUM 将死元组标记为可重用——空间可供同一堆页面内的新元组使用。 不会将空间归还给操作系统。 在线操作——无排他锁。由 autovacuum 自动运行。
  • VACUUM FULL 将表重写到仅包含活跃元组的新文件中。将空间归还给操作系统。 需要排他锁 ——阻塞所有读写操作。仅在维护窗口期间使用。如需在线表重写，请使用 pg_repack 代替。

pg_repack。 一个扩展插件，在无需排他锁的情况下完成 VACUUM FULL 的工作：创建影子表，复制活跃元组，在复制过程中通过触发器进行双写，最后原子性地完成切换。用于将膨胀的表缩减回正常大小的生产级工具。

VACUUM freeze 与 xid 回卷。 PostgreSQL 的事务 ID 是 32 位的。在约 20 亿个事务后，xid 计数器会回卷。具有非常旧的 xid 的元组在计数器回卷后会表现为"来自未来"。 Vacuum freeze 将古老的元组标记为"冻结"——对所有事务可见，无需检查 xid。只要 autovacuum 保持冻结进度，回卷就永远不会成为问题。

当 autovacuum 跟不上时（大量写入工作负载超过了冻结器的处理速度）：数据库接近 xid 临界点，然后作为安全措施 停止接受写入 作为安全措施（ERROR: database is not accepting commands to avoid wraparound data loss）。著名的故障案例：

  • Sentry（2015 年）：PostgreSQL 中的 xid 回卷，数据库只读数小时。
  • Mailchimp（2019 年）：类似事故，需要手动执行 freeze。
  • Joyent（2014 年）：SmartOS-on-Solaris 文件系统损坏 + xid 回卷。

监控指标： SELECT datname, age(datfrozenxid) FROM pg_database ——超过约 10 亿值得警惕，超过 20 亿则为严重告警。

监控 autovacuum。

  • pg_stat_user_tables.n_dead_tup ——每张表当前的死元组计数。
  • pg_stat_user_tables.last_autovacuum ——autovacuum 上次在该表上运行的时间。
  • pg_stat_progress_vacuum — 当前正在运行的 vacuum 的实时进度。

调优。 对于热表（高更新频率），降低 autovacuum_vacuum_scale_factor （例如，0.05 = 5% 的死行触发 vacuum）并增加 autovacuum_max_workers （默认值为 3 — 对于有许多热表的集群来说太少）。增大 autovacuum_vacuum_cost_limit 以使 autovacuum 运行更快（默认情况下它会自我限速以避免 I/O 独占）。

12.6 自增 vs UUID 主键

主键的选择对 B+ tree 插入局部性有重要影响。

顺序 PK（自增 / serial）。 所有新插入都命中 B+ tree 的右端边界。最右侧叶页始终在缓冲池中（最近被访问）。插入操作基本上是：追加到右端边界，偶尔分裂，然后推进。 对 CPU 和缓冲池友好。 但对于分片系统，顺序 ID 会将插入集中在拥有高范围的分片上 — 如果不加注意，就会出现"热末尾分片"问题。

UUIDv4 PK。 UUID 是均匀随机的 128 位值。每次插入落在整个树中的某个 不同的 叶页上。 对缓冲池局部性不友好 — 每次插入都会使一个可能不在缓存中的页面变脏。B+ tree 均匀扩展，但插入时会产生冷 I/O。具体影响：在插入密集型工作负载下速度降低 30% 以上。

UUIDv7 / ULID。 时间有序前缀（48 位时间戳）+ 随机位。 新插入大致落在树的右端边界 因为时间戳前缀单调递增。在保持全局唯一性和不可预测性的同时恢复了 B+ tree 的局部性。

具体基准测试 （相同硬件、相同 schema、1 亿行表）：

   Workload: 100k inserts/sec sustained, mixed with point lookups.

   PK type        Sustained insert rate    Notes
   ──────────────────────────────────────────────────────────────────
   Auto-increment 100k/sec                 right-edge hot, predictable
   UUIDv4          30k/sec                 buffer pool thrashes; many cold reads
   UUIDv7          80k/sec                 close to auto-increment, less skew

2026 年的推荐模式。 对于高插入量的表，主键优先使用自增 。 并单独使用 UUID/UUIDv7 列作为外部标识符。或者直接将 UUIDv7 用作主键 — 对 B+ tree 来说足够接近顺序，对分布式系统来说全局唯一。仅当 UUIDv7 时间戳前缀的可预测性存在安全隐患时（罕见情况）才使用 UUIDv4。

12.7 查询规划器的常见陷阱

基于代价的优化器会做出一些假设；当这些假设有误时，它会选择糟糕的执行计划。

数据倾斜会欺骗规划器。 "共有 5 个客户；customer_id=42 占所有行的 90%。"

   SELECT * FROM orders WHERE customer_id = $1;

   Planner uses statistics from pg_statistic:
   - n_distinct = 5 (5 customers)
   - Assumes uniform distribution: each customer ≈ 20% of rows.

   For customer_id=42 (actually 90% of rows): planner expects 20%, 
   picks index scan with bookmark lookup — but should have picked seq scan.
   For other customers (each ≈ 2.5% of rows): planner expects 20%, 
   picks seq scan — but should have used index.

PostgreSQL 的 CREATE STATISTICS （10+）允许你创建扩展统计信息 — 多变量分布、依赖关系、特定列的最常见值列表。通过 ANALYZE刷新。对于 非常 倾斜的数据， 执行计划提示 （通过 pg_hint_plan 扩展）或 固定计划 （MySQL 优化器在注释中使用的提示）可以锁定规划器的选择。

EXPLAIN ANALYZE 自下而上读取。 最深的节点最先执行。代价数字是 估算值 — 它们不是真实的秒数，而是抽象单位。 actual time=… 值是真实的，以毫秒为单位。始终比较估算行数与实际行数；相差 100 倍是规划器统计信息不准确的信号。

在批量数据变更后执行 ANALYZE 。 Autovacuum 会定期运行 ANALYZE，但对大量变更有滞后。加载完一百万行后立即执行 ANALYZE the_table — 否则下一次查询可能会按照表为空的情况来规划。

仅索引扫描依赖于可见性映射的新鲜度。 PG 的可见性映射跟踪哪些堆页面拥有全可见元组（不需要事务查看堆来判断可见性）。如果可见性映射显示全可见，仅索引扫描可以跳过堆查找。 Autovacuum 滞后 → 可见性映射过时 → 强制堆查找 → 性能低于预期。近期有大量写入的表的可见性映射位是冷的；仅索引扫描会降级为索引加堆查找，直到 autovacuum 更新映射为止。

综合来看：规划器在 95% 的情况下是正确的；剩下的 5% 是你花时间的地方。统计信息漂移、数据倾斜和可见性映射位过时，是造成"为什么今天这个查询变慢了？"工单的最大来源，超过其他任何类别。

  • 宽列 NoSQL 当写入速率超过每秒百万级且访问是按分区键进行时（时间序列、事件日志、IoT 遥测）使用。最多提供每键一致性。
  • 文档存储 当数据天然是文档形态且每租户可变时使用（CMS、自由格式用户数据、原型验证）。单文档事务是基本单元；支持多文档但较慢。
  • NewSQL 当你需要跨分片事务且不想自己构建 2PC 协调器时使用。提交延迟和成本更高。当你原本要在分片 MySQL 上构建自定义事务协调器时，这是正确的选择。
  • 对象存储 适用于二进制大对象、归档、快照 — 任何访问模式是"按键获取"的场景。永远不要作为主 OLTP 存储。

综合来看：每个相邻类别的存在，都是因为 OLTP SQL 在某个 特定维度 上表现不佳（地理分布、每分区写入吞吐量、schema 灵活性、归档成本）。正确选型意味着知道哪个维度最重要。

§14. 使用场景展示

五个应用场景，各自强调设计空间的不同方面。

支付账本（支付宝、Stripe、PayPal、Square）。 强调守恒不变量（余额之和 = 贷方之和 − 借方之和）、RPO=0、每账户全序、幂等性、跨分片转账。 适用：带半同步跨可用区复制的分片 MySQL InnoDB；基于 XA 事务的自定义 2PC 协调器；幂等性行与事务一起置于同一分片。支付宝的 OceanBase 是相同架构加共识复制。核心选择： 每行强一致性 — 每次扣款都读取 SELECT balance FOR UPDATE、加行锁、执行。同步路径上不存在最终一致性。 反模式：Cassandra — 不使用 LWT 会失去守恒性；使用 LWT 则变成了更差的 MySQL。

社交档案数据（LinkedIn Espresso，Facebook TAO）。 强调极端读扇出（一个档案被数百万次读取以渲染 feed）、适中的每用户写入速率、按 user_id 的干净分区、社交图谱边作为数据；边的一致性比档案字段的一致性要求低。 适用：按用户分片的 MySQL 主节点，配合大量读副本和激进缓存（Facebook 的 TAO 是 MySQL 前的缓存层）。核心选择： 通过副本和缓存扩展读能力 — 写入速率小，读取速率极大，缓存命中率决定一切。 反模式：文档存储作为主存储。档案是带有严格访问模式的小型结构化数据；文档的灵活性是额外开销。

电商库存（Amazon、Shopify、eBay）。 强调闪购期间的热门 SKU（库存计数器的单行争用）、跨仓库原子性、预留流程（提交前为购物车保留库存）、与实物盘点的对账。 适用：分片关系型主存储；促销期间热门 SKU 的写分片；带 TTL 的预留锁定；定期对账。Amazon DynamoDB 用单分区事务处理部分场景；Shopify 通过 Vitess 使用分片 MySQL（黑色星期五峰值超过 80k+ 次结账/分钟）。 反模式：峰值促销流量下的单主 OLTP — 黑色星期五是必须在促销前完成每分片容量规划的临界点。

游戏排行榜（Riot、Discord、实时 PVP）。 强调每场比赛的高写入速率（每天数百万场比赛）、"我所在区域前 100 名"的低读延迟、按分数的范围扫描、区域分区。 适用：混合方案 — 用事务型存储（PostgreSQL、MySQL）作为事实来源（比赛结果、玩家数据），加上排序集合（Redis ZSET）作为实时排行榜视图；CDC 保持两者同步。核心选择： 将持久记录与查询视图分离. 反模式：让 OLTP 以数百万 QPS 服务 top-100 排名 — ORDER BY score LIMIT 100 在十亿行表上扫描索引；Redis ZSET 正是为此而生。

SaaS 多租户数据（Notion、Slack、Figma）。 强调每租户隔离、租户规模不均（某个客户是中位值的 100 倍）、产品演进中的 schema 演化、版本历史（Notion 的块级编辑、Figma 的帧级编辑）、实时协作。 适用：以 tenant_id 为分片键的分片 PostgreSQL 或 MySQL；对超大租户进行自定义再平衡（最大的租户获得自己的分片）。Notion 的架构（2021 年以后）是典范案例；Slack 通过 Vitess 使用 MySQL，并以 team_id为键。核心选择： 以 tenant_id 作为主分区键 ，并具备应对倾斜的能力。 反模式：所有租户共用一个数据库，以 tenant_id 作为列 — 在租户规模相近且不超过约 100 个时有效；当最大租户超出单个分片容量时失败。

§16. 第一部分总结

事务型数据库本质上是一棵套上 ACID 契约的 B+ 树（或 LSM 变体）——其存储引擎通过 MVCC 和预写日志 fsync 保证单分片的原子性、隔离性与持久性，但超出单分片范围的一切（跨分片事务、地理分布、幂等重试、负载下的 schema 演化）都是系统设计者需要在上层自行构建的。分片 MySQL 或 PostgreSQL 在规模化场景下持续胜出——支撑支付宝每秒 583k 笔交易、LinkedIn PB 级用户档案数据、Notion 数百万文档、Pinterest 数十亿图钉——原因并不在于存储引擎有什么魔力，而在于 B+ 树 + MVCC + WAL 是目前已知最低成本地为开发者提供「要么发生、要么没发生」契约的方式；一旦拥有这一契约，系统中的其他所有关注点（缓存、搜索、分析、归档、派生视图）都归结为「持续从 WAL 喂数据，按需重建」。

第二部分：分析型 / 列式数据库

§17. 分析型 / 列式数据库是什么

一个 分析型数据库 是一种有状态的存储系统，其契约以 扫描吞吐量 ——在不可变或以追加为主的结构化数据上，以毫秒到秒级的速度完成对数十亿乃至数万亿行数据的聚合、过滤、分组、连接查询。它是所有需要回答「有多少」、「趋势如何」或「发生了什么变化」类问题的系统底层：营收看板、群组分析、广告归因、A/B 测试结果、异常检测、实时监控。其核心特性是「承诺对海量数据给出快速答案，即便数据存在轻微延迟」。

分析型数据库处于四维设计空间的一个角落：

  • 实时 OLAP（RTOLAP） ——Apache Pinot、Apache Druid、ClickHouse、StarRocks、Apache Doris。亚秒级 p99 扫描查询，去规范化星型 schema，以追加为主的流式摄入（Kafka 驱动），预聚合物化视图。支撑面向用户的分析场景：LinkedIn 的「谁查看了你的主页」、Uber 的波峰地图、Slack 的搜索分析、广告技术的竞价时决策。
  • 云数据仓库 ——Snowflake、BigQuery、Redshift、Databricks SQL。PB 级规模，存储（对象存储）与计算（弹性集群）分离，强 SQL 语义，写时 schema，通过批量或微批 ELT 实现 T+1 到准实时的数据新鲜度。支撑 BI 看板、ML 特征工程、财务对账、合规报告。
  • Lakehouse 引擎 ——Apache Iceberg、Delta Lake、Apache Hudi，基于 S3/GCS/ADLS，配合 Trino、Spark SQL、Presto 等查询引擎。对象存储作为事实来源，元数据层添加 ACID、时间旅行、schema 演化能力。支撑同一数据集需被多个引擎查询的 ML 流水线和分析场景。
  • HTAP / 混合存储 ——TiDB（TiFlash）、SingleStore（MemSQL）、CockroachDB 列式副本、Postgres + Citus 列式、Snowflake Unistore。单一系统通过行存主节点 + 列存副本实现 OLTP 和 OLAP 的兼顾，并提供一致视图。支撑运营分析场景——这类场景无法接受独立 OLAP 系统带来的新鲜度差距。

第二部分聚焦于实时 OLAP 和云数据仓库，因为它们代表了两个主流操作点：一端是低延迟的面向用户分析，另一端是大规模批量分析。Lakehouse 和 HTAP 变体复用了大量相同机制（列式存储、向量化执行、谓词下推），但在持久性和弹性上取舍不同；我们将在 §19 进行对比。

分析型数据库不擅长的场景：

  • 点更新 / 单行变更。 列式存储意味着单行更新会触及每个列文件。可变状态请使用 OLTP；让 OLAP 从中读取 CDC 流。
  • 亚毫秒级点查询。 即便索引良好的列存也需要承受列条带遍历开销。请使用 Redis、OLTP 主键查询或键值缓存。
  • 跨行事务一致性。 不支持跨行 ACID；聚合结果是「近似一致，误差在摄入延迟范围内」。
  • 频繁的行级 schema 差异。 无 schema 文档会浪费列式压缩。请使用文档存储，或在摄入前去规范化/展平。
  • 数据量极小。 在约 1000 万行以下，列式规划器和 shuffle 流水线的开销超过普通 Postgres 索引的收益。只在规模倒逼时才选 OLAP，而非提前使用。

心智模型：分析型数据库是一个 派生存储 通过 CDC、Kafka 或批量 ELT 与记录系统（OLTP 数据库、事件日志或两者兼有）保持同步，并针对扫描密集型查询进行优化。先把 OLTP 做好；OLAP 可以从 WAL 或事件日志重建。若 OLAP 出错（分区倾斜、缺少物化视图、数据延迟到达），看板会说谎，但真相保存在 OLTP 主节点中。

§18. 内在保证

该技术在设计上提供的保障，以及仍需在其上叠加的能力。

设计上已提供

  • 扫描吞吐量。 列式存储 + 压缩 + 向量化执行每节点每秒可扫描数十亿行。ClickHouse 在单台 32 核机器上基准测试达到 10 亿行以上/秒的扫描速度；Pinot 在去规范化星型 schema 上以 p99 < 100ms 提供 10 万+ QPS 服务。
  • 压缩。 与等效行存数据相比，磁盘占用缩小 5–100 倍。低基数字符串使用字典编码，单调递增键使用 delta + 位压缩，有序维度使用 RLE（行程编码），上层再叠加 LZ4/ZSTD。
  • 谓词下推。 查询引擎将 WHERE 子句下推至存储层；存储通过 zone map（每块 min/max）和布隆过滤器跳过整个数据块。
  • 追加持久性。 数据一旦提交到仓库（S3 分段上传、segment 合并、快照提交），便永久保留。云数据仓库继承了 S3 的 11 个 9 的持久性保证。
  • 延迟绑定 schema（因系统而异）。 Lakehouse 和 BigQuery 通过追加列而无需重写旧文件来处理 schema 演化；Pinot/Druid 需要显式重新加载。

需在其上叠加

  • 数据新鲜度。 没有任何系统能免费提供对最新 OLTP 写入的即时分析。RTOLAP 通过 Kafka 将延迟压缩至秒级；数据仓库通过微批实现分钟到小时级延迟；HTAP 是唯一能达到「数十毫秒」的选项。
  • 与 OLTP 的一致性。 如果 CDC 流水线存在延迟或分区，仓库查询可能看到与 OLTP 主节点「不同的真相」。应用层必须清楚自己处于哪种新鲜度体制下进行读取。
  • 成本控制。 Snowflake 或 BigQuery 中一次错误索引的扫描可能导致每次查询数百美元的费用。成本监控、查询预算和每租户配额都是运维人员的职责。
  • 倾斜规避。 糟糕的分区键（所有数据哈希到同一分片）会摧毁吞吐量。schema 和摄入流水线负责选择分区键；引擎本身无法拯救你。
  • 回填机制。 当指标定义发生变化时，从事件日志重新推导一年的分析数据本身就是一条独立的流水线；数据库不会对你的聚合逻辑进行版本管理。
  • 多租户隔离。 一个嘈杂客户的扫描可能让其他所有人饥饿。仓库提供资源池和按仓库的计算资源；运维人员负责对接配置。

总结：分析型数据库保证的是 对已摄入数据的扫描吞吐量。数据新鲜度、指标定义、成本和隔离，都是系统设计者的问题。

§19. 设计空间

各变体沿多个正交轴存在差异，这些轴相互影响，无法自由独立选择。

存储布局：纯列式 vs 混合行组

一个 纯列式 布局将每列存储在独立文件（或条带）中，按某个键排序。读取单列只需访问该列的字节——对窄聚合（如 SUM(amount)）的 I/O 极小。ClickHouse、Pinot、BigQuery Capacitor。

一个 混合行组 布局（PAX、Parquet、ORC）将行切分成 100k–100 万行的行组，在每个行组内按列存储数据。这让单个 S3 文件能高效支持宽（整行）和窄（单列）两种扫描模式，代价是压缩率略低于纯列式。Snowflake 微分区、Iceberg + Parquet、Delta Lake。

权衡：纯列式在窄分析扫描上胜出；混合行组在跨查询形态的灵活性和对象存储兼容性上胜出。大多数云数据仓库选择混合布局。

存储与计算的耦合方式

无共享架构 （ClickHouse、Pinot、Druid historical 节点、经典 Redshift）将数据存放在计算节点的本地磁盘上。查询延迟低，架构简单，但扩容需要重新均衡——缓慢且运维成本高。

共享存储 / 计算分离 （Snowflake、BigQuery、Redshift RA3、Databricks）将数据存放在对象存储（S3、GCS、Azure Blob）中；计算集群弹性且无状态。扩容即时——再起一个仓库即可——但查询延迟增加了对象存储 GET 的开销（冷读取 50–200ms），由 SSD 缓存层吸收。

权衡：无共享架构在亚秒级 p99 面向用户分析上胜出；共享存储在弹性大规模批量处理和多租户成本上胜出。

摄入方式：批量 vs 微批 vs 流式

Batch ELT ——Airflow / dbt / Fivetran 按小时或按天加载新数据。最便宜、最简单，新鲜度约为小时级。云数据仓库的默认选择。

微批 ——持续加载小批次（1–10 分钟）。由 Snowpipe、Databricks Auto Loader 使用。以适中成本实现分钟级新鲜度。

流式 ——Kafka 驱动的、滚动 segment 摄入（Pinot 实时服务器、Druid Kafka 索引、ClickHouse Kafka 引擎）。新鲜度达秒级；成本更高，运维复杂度提升（segment 压缩、schema 不匹配、死信处理）。

索引：zone map、布隆过滤器、倒排索引、物化视图

除列式布局本身外，分析引擎还通过以下方式加速查询：

  • Zone maps ——每块的 min/max 值，用于对选择性谓词跳过整个数据块。所有列式引擎普遍支持。
  • 布隆过滤器 ——每列的概率成员测试，用于高基数等值谓词（WHERE user_id = X）。Pinot、Druid、ClickHouse。
  • 倒排索引 ——用于 OLAP 存储内的文本搜索列（Pinot 文本索引、Druid 空间/JSON 索引）。
  • 位图索引 ——用于大量 group-by 中低基数维度列。
  • 物化视图 / 预聚合 ——在摄入时维护的预计算汇总。Pinot 的 StarTree 索引是其典型实现；Druid 在摄入时进行 rollup；BigQuery 支持物化视图；ClickHouse 支持物化视图和 projection。

Schema 模型：星型 schema vs 平铺去规范化 vs 宽表

星型 schema ——事实表 + 维度表，在查询时进行 join。经典数据仓库模式（Kimball）。可行的原因是维度表足够小，可以广播。

平铺去规范化 ——每条事实记录内联其所有维度。RTOLAP 的默认选择（Pinot、Druid），因为在亚秒延迟预算下 join 太慢。占用更多磁盘，节省查询时间。

宽表 ——每个逻辑实体使用包含数百或数千列的单张表（Snowflake Variant、带有大量列的 ClickHouse）。以 schema 规范性换取查询灵活性。

并发：每秒查询数上限

§20. 底层数据结构：列式存储

值得了解的字节级机制。

20.1 列条带布局：行主序 vs 列主序

行存（InnoDB、PostgreSQL）将每行的各列连续存放：第 1 行的 (user_id, event_type, ts, amount) 字节相邻，然后是第 2 行，依此类推。适合「给我这一行」；不适合「对十亿行中的某一列求和」，因为引擎需要读取每行的所有字节才能访问一列的数据。

列存则将每列连续存放：

Column-major (Parquet row group, simplified):
┌──────────────────────────────────────────────────────────┐
│ user_id:   [val_1, val_2, val_3, …, val_N]   contiguous  │
│ event_type:[val_1, val_2, val_3, …, val_N]   contiguous  │
│ ts:        [val_1, val_2, val_3, …, val_N]   contiguous  │
│ amount:    [val_1, val_2, val_3, …, val_N]   contiguous  │
└──────────────────────────────────────────────────────────┘

100M 行表的具体数字：

  • 行存：100M 行 × 24 字节 ≈ 2.4 GB。 SELECT SUM(amount) 读取每一页。
  • 列存（ClickHouse MergeTree）：四个文件。 user_id.bin ~800 MB， event_type.bin ~50 MB（字典编码）， ts.bin ~200 MB（delta + 位压缩）， amount.bin ~200 MB。 SUM(amount) 只读取 amount.bin ——在压缩生效前就已实现 12 倍 I/O 缩减。

另一个结论： 列内局部性极强。所有 country_code 值来自约 200 个字符串；所有 ts 值单调递增。压缩正是利用了这一特性。

20.2 压缩编解码器栈

现代列式引擎不使用单一压缩器——而是组合多种针对列数据形态定制的轻量编解码器，再在上层应用通用熵编码器。

  • 字典编码。 对于低基数列（country_code, event_type, status），将每个唯一值映射为一个小整数 ID。一个 country_code 列，1B × 8 字节 = 8 GB，变为 1B × 1 字节 + 字典 ≈ 1 GB。 8 倍缩减。
  • 行程编码（RLE）。 对于有序列，存储 (value, count) 对。按 country_code排序后，1B 行压缩为约 1k 个行程 ≈ 16 KB。
  • 位压缩。 当字典能用 K 位表示（256 个条目 → 8 位，64 个 → 6 位）时，在位级打包 ID。1B 行中一个 5 位字典每行节省 3 位 = 每列约 375 MB。
  • Delta 编码。 对于单调数值（时间戳、顺序 ID），存储 (start, delta_1, delta_2, …)。秒级精度的事件时间戳每个 delta 只需几位即可压缩。
  • Delta-of-delta。 两级 delta。用于 Gorilla（Facebook），在 Prometheus / M3DB / InfluxDB IOx 中得到推广。对于固定频率指标（「每 10 秒采样一次 CPU」），大多数 delta 恰好为 10；delta-of-delta 为 0。64 位时间戳平均压缩至约 1 位。
  • Gorilla 浮点编码。 对每个 float64 与前一个值进行 XOR，存储前导零计数 + 有效字节。对于平稳或缓慢变化的指标（内存用量、吞吐量），连续样本仅在最后几个字节不同，实现 10–20 倍压缩。
  • Zstandard / LZ4 / Snappy 顶层压缩。 对编解码器输出进行通用熵编码。Zstd level 3 是现代默认选项——压缩速度约 500 MB/秒，解压速度约 2 GB/秒，额外压缩比 2–4 倍。

在实际生产事实表（约 50 列，未压缩约 256 B/行）上的累积效果：列级编解码器约 5 倍；Zstd 顶层约 2 倍；总计约 10 倍。S3 上 10 TB 原始事实表在 Parquet 中约为 1 TB。country_code 字段的端到端路径如下：

Raw VARCHAR    8 GB     8-byte strings
Dictionary →   1 GB     8-bit IDs + small dictionary
Bit-pack →     125 MB   bit-packed to actual bit-width
Zstd →         ~15 MB   entropy coding on the bit-packed bytes
              ────
              ~530× reduction on a well-skewed column

20.3 Granule、zone map 与稀疏主键

列式引擎不对每行建索引，而是对 granule 建索引——连续行的固定大小块。典型大小：ClickHouse 中 8192 行，Pinot 中 1024 行，Parquet 中每行组约 100 万行。

每个 granule 携带逐列元数据：

  • 列在 granule 内的最小值 / 最大值 （zone map）。一个 WHERE event_date BETWEEN … 查询检查每个 granule 的 (min, max)；不相交的 granule 会被完全跳过。
  • Null 计数。
  • 布隆过滤器（可选）。 对于高基数点等值谓词（WHERE user_id = 42），每个 granule 的布隆过滤器可以排除不含匹配键的 granule。
  • 直方图（可选）。 Snowflake 为优化器在每个微分区中保存近似分布信息。
ClickHouse MergeTree, simplified granule layout:

┌────────────────────────────────────────────────────────────┐
│ Part (~10–100 GB, one logical chunk on disk)               │
│   ┌──────────────────────────────────────────────────────┐ │
│   │ Granule 0:                                           │ │
│   │   col1.bin bytes [0..L1]   col2.bin bytes [0..L2]    │ │
│   │   Mark file: col1: min=10, max=99, null=0            │ │
│   │              col2: min='2026-05-01', max='2026-05-01'│ │
│   ├──────────────────────────────────────────────────────┤ │
│   │ Granule 1: …                                         │ │
│   └──────────────────────────────────────────────────────┘ │
│                                                            │
│   Primary key index (sparse): one entry per granule        │
└────────────────────────────────────────────────────────────┘

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 主键索引 在列式 OLAP 引擎中与 OLTP B+ 树有本质区别：它是 稀疏的 ——每个 granule 一条记录，而非每行一条。一个 8192 行的 granule 使 PK 索引缩小约 8000 倍。点查询定位到一个 granule (~8000 行)在 <1 ms 内完成，随后必须在内部进行扫描。 这就是为什么 OLAP 不适合亚毫秒级行查找——其索引粒度在设计上就不匹配。

针对 zone map 的选择性谓词： SELECT SUM(amount) WHERE ts BETWEEN '2026-05-01' AND '2026-05-07' 对一张 1 亿行、粒度为 65k 行的表，需读取约 1500 个粒度页脚，找到约 7 个匹配粒度，扫描约 50 万行。 缩减 200 倍。

20.4 向量化执行与 SIMD

逐元组（Volcano）执行通过算子树每次处理一行。每行在每个算子边界都要承担虚函数调用开销、分支预测失误和缓存未命中。向量化执行每次将一个 批次 （通常为 1024 或 4096 个值）送入每个算子，将内循环转变为紧凑的标量或 SIMD 向量循环：

// Tuple-at-a-time
while (next_row(row)) {
  if (row->price > 100) emit(row);
}
// Throughput: ~10–50 M rows/sec/core

// Vectorized
for (int i = 0; i < 1024; i++) {
  selection[i] = (price[i] > 100);
}
emit(selection);
// AVX-512: 1024-wide compare + bitmask in <100 nanoseconds
// Throughput: ~1–5 GB/sec/core on integer columns

ClickHouse、DuckDB、Databricks Photon、BigQuery 的 Capacitor 引擎均大量使用 AVX-512 / AVX2。聚合（SUM、COUNT、MIN、MAX）、过滤、算术运算以及 group-by 哈希均可并行化。 向量化列式执行的扫描速度为每核 1–10 GB/sec ——整个"OLAP 速度快"的说法都建立在此之上。相同硬件，比逐元组解释器快 10–100 倍。

20.5 Parquet 文件结构

Parquet 是专有数据仓库之外最主流的磁盘列式格式。布局：

Parquet file (typical 128–1024 MB):

┌─────────────────────────────────────────────────────────────┐
│ Magic: "PAR1"                                               │
├─────────────────────────────────────────────────────────────┤
│ Row Group 0 (~128 MB uncompressed, ~1M rows)                │
│   Column chunk for col1                                     │
│     Page 0: dictionary page                                 │
│     Page 1: data page                                       │
│     …                                                       │
│   Column chunk for col2 → …                                 │
│   …                                                         │
├─────────────────────────────────────────────────────────────┤
│ Row Group 1, Row Group 2, …                                 │
├─────────────────────────────────────────────────────────────┤
│ Footer:                                                     │
│   Schema, row groups[], column metadata[], statistics       │
│   (min, max, null_count, byte_offset, sizes, bloom offset)  │
│ Footer length (4 bytes) + magic "PAR1"                      │
└─────────────────────────────────────────────────────────────┘

关键属性：自描述（页脚包含完整 schema）、行组相互独立（可通过页脚统计信息跳过，跨文件并行处理）、页是 I/O 单元（约 1 MB），且"页脚在末尾"的惯例在数据拉取开始前需要三次 S3 往返——这对小文件众多的工作负载是一个真实的性能陷阱。

20.6 Iceberg manifest 树——Parquet 之上的表语义

单独的 Parquet 只是"S3 中的文件"。要构建一张支持 INSERT / 快照隔离 / schema 演进 / 时间旅行的真实表，需要在文件之上增加一个元数据层。Iceberg 的解决方案是在对象存储中构建多级 manifest 树，并通过原子指针交换提交：

table_root/
├── metadata/
│   ├── v42.metadata.json     ← schema, snapshots[], partition spec
│   ├── snap-12345-mlist.avro ← manifest list (per-manifest partition bounds)
│   └── manifest-001.avro     ← manifest file (per-data-file column stats)
└── data/
    └── dt=2026-05-01/part-0001.parquet

链路：表元数据 → 快照 → manifest 列表 → manifest 文件 → Parquet 数据文件。

这是一个 四级 zone-map 层次结构：分区 → 文件 → 行组 → 页。每个层级可裁剪约 10–100 倍。对于带有严格谓词的 1 PB 表，通常只需扫描 <100 MB。快照隔离来自对 version-hint.text （或 catalog 行）的原子指针交换；时间旅行来自保留旧的 metadata.json 文件；schema 演进仅涉及元数据（每个快照携带一个 schema 版本）。

Delta Lake 和 Hudi 以不同格式解决同一问题（基于 _delta_log/ JSON 提交文件的 .hoodie/ timeline 目录）。Iceberg 的 manifest 树在 2026 年是主流选择，因其支持引擎最广、在 PB+ 规模下运维最简单。

20.7 物化视图与 StarTree

许多分析查询本质上是汇总： SELECT country, day, SUM(amount) FROM events GROUP BY country, day。如果 (country, day) 的基数较小，但底层数据量巨大，在写入时进行预聚合可将每次查询转化为常数时间查找。

  • ClickHouse 物化视图 ——源表上的触发器按批次将数据重新聚合到目标表中。
  • BigQuery 物化视图 ——自动维护；查询规划器在适用时会自动改写查询以使用它们。
  • Pinot StarTree 索引 ——多级预聚合树，支持任意维度和指标子集的组合。查询计划走到叶子单元并读取预聚合值，无需扫描原始数据。支撑 LinkedIn 面向用户的分析，p99 < 100 ms。

§21. 分析型数据库固有的难题

21.1 数据倾斜与热分区

糟糕的分区键——比如 country 当 60% 的流量来自同一国家时，或 user_id 对 N 取模但某个用户是中位数的 10 倍时——会导致一个节点跑满 100% 而其他节点空闲。缓解措施：使用复合键（country + user_id_bucket），对加盐字段做哈希后取模，或选择已知均匀分布的键（例如随机纳秒精度时间戳与序号的组合）。

21.2 迟到数据与乱序数据

一个时间戳为 10:00 的点击事件可能因用户网络缓慢而在 10:30 才到达数仓。如果 10:00 的分段已被封存并预聚合，迟到行要么被丢弃，要么触发重新聚合。Pinot、Druid 和 lakehouse 引擎通过可变热分段、迟到缓冲区或读时合并模式来处理这一问题。代价：增加了复杂性，以及"数据已到达"与"查询已反映"之间的滞后。

21.3 高基数与 join 膨胀

一个 user_id 具有 10 亿个不同值的列难以进行字典编码（字典会与列一样大）。两张各 10 亿行的表在 user_id 上进行 join，要么在集群中传输 32 GB 的键，要么需要广播 join 的前提条件。缓解措施：在写入时预先 join（反范式化），对两张表按 join 键分片，或使用近似技术（HyperLogLog 用于去重计数）。

21.4 Schema 演进

向一张 10 TB 的 Parquet 表新增一列很简单，前提是格式支持列追加（Iceberg、Delta）——旧文件对新列读取为 NULL。重命名或修改列类型需要重写或逻辑名称映射（Iceberg 的 schema 演进语义支持此操作）。删除列若关注存储成本则需要重写。与 OLTP 相比，schema 演进成本更低但速度更慢（需重写 GB 到 TB 级别的数据）。

21.5 实时新鲜度与查询吞吐量

一个实时 OLAP 集群每秒从 Kafka 接收 10 万个事件，必须在后台将细小的流式分段合并为更大的分段。如果合并进度落后，热分段数量增长，每次查询的开销（每个分段一次合并步骤）将呈爆炸式增长。Pinot 和 Druid 暴露分段年龄/数量指标，SRE 会对此设置告警。

21.6 云端成本失控

在 BigQuery 中对一张 10 TB 的表进行单次全表扫描 SELECT * ，成本约 $50。每天扫描整张表的仪表板每月成本高达 $1500。对每个用户仪表板渲染都执行此操作的未调优 BI 工具是已知的成本爆炸模式。缓解措施：使用聚簇表、物化视图、查询成本监控、按用户设置查询预算、以及遵守分区裁剪谓词的 BI 工具。

21.7 与 OLTP 的不一致性

用户在 10:00:00 更新了 OLTP 数据库中的个人资料。CDC 管道在 10:00:30 将变更同步至数仓。在 10:00:15 刷新的仪表板显示的是旧值。对于 BI 这无关紧要；但对于触发操作的运营仪表板（如欺诈检测），这是一个 bug。缓解措施：将运营查询路由至 OLTP 数据库，将分析查询路由至数仓，并为每个界面记录新鲜度规范。

Apache Pinot（LinkedIn）

  • 起源：由 LinkedIn 为"谁查看了你的个人资料"和广告分析而构建。
  • 规模：100k+ qps，p99 低于 100ms，每天写入 PB 级数据。

  • 部署于：LinkedIn、Uber、Walmart、Stripe。

  • 起源：由 Metamarkets（现为 Snap）为广告分析而构建。

  • 起源：由 Yandex 为 Metrica（Web 分析，每天 300 亿事件）而构建。

  • 架构：无共享集群；MergeTree 存储引擎；通过 ZooKeeper 或 ClickHouse Keeper 协调。
  • 突出特性：极致的单节点性能、支持任意 SQL 的物化视图、原生 Kafka 和 S3 引擎。

Snowflake

  • 起源：约 2014 年从零开始在云上构建。
  • 规模：每个客户 PB 级数据，每个仓库数百个并发用户，通过 Snowpipe 实现 T+1 到近实时。
  • 架构：存储在 S3（微分区）；虚拟仓库是弹性计算集群；元数据服务跟踪分区；结果缓存消除重复计算。
  • 突出特性：零拷贝克隆、时间旅行（查询历史状态）、存储与计算分离。

BigQuery

  • 起源：由 Google 基于 Dremel 构建（Snowflake 也参考了该研究论文）。
  • 规模：EB 级表；按查询付费模式；每个查询槽约 10–100 GB/sec 的扫描速度。
  • 架构：存储在 Colossus（Google 的分布式文件系统）；计算是无服务器槽位池；基于 Dremel 的查询引擎，支持向量化执行。
  • 突出特性：BigQuery ML、BI Engine 缓存、完全无服务器。
  • 部署于：Google 内部、Spotify、《纽约时报》、汇丰银行。

Lakehouse（Iceberg + Trino + Parquet）

  • 起源：Iceberg 来自 Netflix，Delta Lake 来自 Databricks，Hudi 来自 Uber。
  • 规模：S3 中的 EB 级表；查询延迟从秒级到分钟级；多个引擎可查询同一数据集。
  • 架构：对象存储中的 Parquet 文件；元数据层跟踪文件、快照、schema 版本；Trino / Spark / Flink 均可通过元数据层查询。
  • 突出特性：开放格式、多引擎、对象存储上的 ACID、时间旅行、schema 演进。
  • 部署于：Netflix、Apple、Airbnb、Adobe。

每个派生存储通过重放 Kafka 日志来重建；OLTP 主库是事实来源。这是 §6 中"主库 + 派生存储"模式的具体体现：OLTP 数据库持有可变状态；OLAP 和搜索是最终一致的派生视图，针对不同访问模式进行了优化。

§23. 总结

数据库有两种相辅相成而非相互竞争的形态。事务型形态是穿着 ACID 合约的 B+ 树（或 LSM）——其职责是保证对可变、结构化状态的小型变更要么发生、要么未发生，并在掉电后存活。分析型形态是穿着扫描吞吐量合约的列式布局——其职责是在毫秒到秒级内回答数十亿行上的聚合查询，即使数据略有陈旧。一个真实系统两者兼用：OLTP 主库保有真相，受 ACID 保护，单分片或分片；分析存储（Pinot 用于面向用户的分析，Snowflake 或 BigQuery 用于 BI，lakehouse 用于 ML 特征工程）通过 CDC 或 Kafka 与主库保持同步，回答 OLTP 主库承受不起的问题。关键在于知道每个问题属于哪种形态——"这个用户当前余额是多少"属于 OLTP；"过去一周 2 亿用户的购买趋势是什么"属于 OLAP——并以与主库同等的严谨度来工程化它们之间的管道（CDC、schema 演进、新鲜度 SLO、幂等回填）。
