关于 CDN（内容分发网络）/ 边缘缓存作为技术类别的参考文档。涵盖设计空间、内部机制、容量上限、难题和故障模式 — 无论工作负载是 SaaS 的静态图片、Stripe 的全球 API、Linux 软件包镜像、Netflix 的 4K 视频，还是 ML 模型制品，均适用。相同的原语，截然不同的使用模式；本文深入解析该原语。

§1. CDN 是什么 — 以及它不是什么

一个 CDN 是一个分布式边缘缓存加上一个全球路由层，将内容在地理位置（以及网络拓扑）上拉近到用户侧。 两个理念融合成一个产品：

  1. 一个 缓存层 ——许多小型、高速、读优化的对象存储节点运行于网络边缘，每个节点持有近期/频繁请求对象的工作集。
  2. 一个 请求调度层 ——通过 BGP（边界网关协议）发布的 anycast IP、GeoDNS（地理 DNS）以及 RUM（真实用户监控）反馈，将用户的 TCP/QUIC 连接路由到拓扑上最近的缓存节点。

单独使用其中一个则是另一回事。有调度的缓存只是反向代理缓存（单数据中心内的 Varnish 或 nginx——有用，但受限于光速，从一个区域到全球其余地区延迟较高）。没有缓存的调度只是全球 L4/L7 负载均衡器（未在其后放置任何内容之前的 AWS Global Accelerator）。CDN 是两者的融合，部署于互联网上每个主要交换节点。

它所处的位置。 从用户视角来看，CDN 是 就是源站——TLS 在 CDN 终止，HTTP 请求由 CDN 响应，只有在未命中时才会有请求到达你的服务器。从源站视角来看，CDN 是一个（庞大、分布式的）客户端，大量发起条件 GET 请求。

CDN 不擅长的场景:

  • 真实数据来源。 缓存条目是派生状态。失去一个 PoP（接入点），那里的字节就丢失了——它们会从源站重新拉取。如果源站丢失了这些字节，CDN 无法将其恢复。对象存储（S3、GCS）是持久性层；CDN 是延迟层。
  • 强一致性。 缓存新鲜度是最终一致的。清除操作在数百毫秒到数秒内传播。通过 CDN 读取自己写入的内容，需要在写入时绕过缓存，或在读取时对 URL 进行版本化。
  • 通用数据库。 即使是 CDN 托管的 KV 产品（Cloudflare KV、Fastly KV Store）也是具有 TTL 的最终一致性缓存，而非 OLTP（在线事务处理）系统。非常适合功能开关和边缘配置，但不适合账单账本。
  • 通用加速器。 部分 CDN 限制缓存内容。视频 CDN 只缓存分片形状的 HTTP。动态个性化内容（包含你姓名的已登录主页）在没有边缘计算或按桶缓存键策略的情况下是不可缓存的。

心智模型： CDN = 用于降低延迟 + 卸载流量 + 稀释 DDoS 的派生边缘状态，位于不可变或版本化的真实数据来源之前。

§2. 固有保证与需要额外分层的能力

CDN 从构造上提供以下保证：

  • 由命中率驱动的源站卸载。 将命中率从 90% 提升至 99%，源站负载下降 10 倍。保证单位为："每个在边缘缓存的对象，每个 PoP 在重新拉取前最多服务 1/TTL 个请求。"
  • 通过 anycast 实现拓扑就近接入。 用户的 SYN 包在约 1 个 RTT 内落在最近的 PoP，在有人居住的地球上通常为 5-30 ms。机制为 BGP——"按 AS 路径的拓扑最近"。通常与物理上的最近节点吻合，但并非总是如此。
  • 通过容量套利稀释 DDoS。 来自全球僵尸网络的 5 Tbps L3 洪泛攻击，由于是全球性的，会分散到 CDN 的各个 PoP。拥有 280 Tbps 出口带宽（Cloudflare 的数据）的 CDN 不需要 过滤 大多数攻击——它拥有的带宽超过攻击者。
  • TLS 在靠近用户处卸载。 TLS 在 PoP 终止。悉尼用户以约 30 ms 连接到悉尼 PoP，而不是约 300 ms 连接到 us-east-1。

CDN 不提供的能力——必须额外分层实现：

  • 读取自己写入 / 强一致性。 有界过期是有界的，但不为零。写入时绕过缓存，读取时对 URL 版本化，或通过边缘计算进行失效处理。
  • 持久性。 缓存字节随时可能被引擎的策略驱逐。源站保存权威副本。
  • Schema/协议感知。 大多数 CDN 将 HTTP 请求视为不透明的 GET 请求。它们不知道 /api/users/123/posts 。 /api/posts?user=123 是同一实体。缓存键规范化和代理键标记是设计者的职责。
  • 每用户隐私隔离。 配置错误的缓存键可能将用户 A 的认证内容提供给用户 B。CDN 不强制执行该契约—— Cache-Control: private、基于 cookie 的分桶，以及自定义 VCL（Varnish 配置语言）/ Worker 代码才负责此事。
  • 计算。 普通 CDN 提供字节服务；边缘计算（Cloudflare Workers、Fastly Compute@Edge、Lambda@Edge）是其上的附加产品。

直白地说： "只要你告诉我什么是可缓存的，提供格式良好的缓存键，并接受最终一致的新鲜度，我将把你的源站负载降低 10-100 倍，并将你的字节放置在距任意用户 5-30 ms 的位置。"

§3. 设计空间

CDN 类别分为 第三方/商业通用型 vs 自建/专用型，以及第三类 专用变体.

第三方/共享 CDN

SaaS（软件即服务）产品，多个租户共享 PoP，通过主机名/客户 ID/配置进行隔离。经济可行，因为多样化的租户流量将固定 PoP 成本摊销至数十万客户。

  • Cloudflare ——约 300 个 PoP，anycast 优先，集成 DDoS + WAF（Web 应用防火墙），通过 V8 隔离（Workers）提供边缘计算。业界最佳的 DDoS 防护能力。
  • Akamai ——约 4,200 个 PoP（通常是 ISP 内部的单机架安装）。最古老的 CDN；企业功能集最完整；流媒体和游戏行业高度依赖。
  • Fastly ——约 70 个高容量"超级 PoP"。VCL 可编程性是其差异化优势——请求处理逻辑以 Varnish 衍生 DSL 编写，可在数秒内全球生效。
  • CloudFront ——约 450 个边缘节点 + 约 13 个区域边缘缓存。与 AWS 源站（S3、ALB）紧密集成。
  • BunnyCDN ——约 120 个 PoP，面向独立开发者/注重成本。约 $0.01/GB，对比 CloudFront 定价 $0.085/GB。以企业功能和 DDoS 保证换取更低成本。
  • Google Cloud CDN、Azure Front Door ——超大规模云厂商 CDN，为各自云服务镜像 CloudFront-与-AWS 源站的模式。

私有/自建 CDN

某些工作负载规模足够大，自建 CDN 比购买更经济。当持续流量超过 100 Tbps 且工作负载足够均匀、可跳过通用功能时，经济上会发生倾斜。

  • Netflix Open Connect ——约 17,000 个缓存设备嵌入 在 ISP 网络内部。每台设备：约 250 TB SSD、100 Gbps NIC（网络接口卡），针对 4K 视频分片顺序读取优化。约 95% 的 Netflix 流量从用户自己的 ISP 内部提供——从不穿越公共互联网骨干网。该架构（在 ISP 内，而非在 IX 节点）可行，因为视频文件不可变且热门集合可存于磁盘。
  • Google 的 Google Global Cache（GGC） ——为 YouTube、Drive、Play Store 提供类似的 ISP 内部设备。
  • Meta 的边缘 PoP ——Facebook/Instagram 静态资源和 WhatsApp 媒体的内部 CDN。
  • Apple 的边缘缓存 ——iOS/macOS 更新、App Store、iCloud。多层架构，在流量高峰时大量使用 Akamai 溢出。

权衡取舍：第三方 CDN 分摊固定成本但按 GB 收费；私有 CDN 需要巨额资本支出但边际成本接近于零。低于约 10 Tbps 时第三方模式胜出；高于约 100 Tbps 时经济账翻转。中间地带则是混合搭配（Apple Edge Cache + Akamai 合同）。

边缘计算作为变体

边缘计算将 CDN 从字节缓存转变为轻量级应用平台。Cloudflare Workers（V8 隔离，约 1-5 ms 冷启动）、Fastly Compute@Edge（仅 WASM，约 1 ms 冷启动）、AWS Lambda@Edge（Node/Python/Java，约 100 ms 冷启动）、Akamai EdgeWorkers。这些在缓存查找与响应 之间 运行代码——JWT（JSON Web Token）验证、A/B 路由、图片缩放、Header 改写、个性化。不能替代源站持久性；是具有可编程行为的缓存平面扩展。

专用 CDN 变体

  • 视频 CDN ——针对 HLS（HTTP 直播流）/ DASH（基于 HTTP 的动态自适应流）分片缓存、多码率变体、manifest 操作优化。分片大小为 500 KB - 5 MB；每路流有几十个码率变体。
  • 游戏 CDN ——补丁分发为主。AAA 大作在 24 小时内向 1 亿用户推送 100 GB 数据。Steam、Epic Games、Battle.net 混合商业 CDN 与自建 P2P 辅助层。
  • 软件分发 CDN ——Linux 镜像（Ubuntu 在 Cloudflare 上）、Docker 镜像仓库透明代理（Hub 在 Cloudflare 上，ECR 在 CloudFront 上）、apt/yum 镜像。
  • 纯 DNS / 纯调度 CDN ——NS1、Cedexis。不缓存 HTTP；在 DNS 层将流量引导至最优端点。通常用作多 CDN 前端。

§4. 底层数据结构与存储引擎

这一节是区分 Staff 级候选人的关键。每个 CDN 在每个 PoP 都运行 某种 缓存引擎，由于访问模式极为一致，引擎有相当标准的形态。

(a) 基于日志结构 SSD 段的内存哈希表

缓存节点有两个物理存储层：RAM 用于热点查找，NVMe（非易失性内存总线）SSD 用于大量工作集。逻辑上是一个键值对象存储；物理上热数据在 RAM，长尾在 SSD。

这从根本上是 LSM 风格（日志结构合并） 虽然我们不这样称呼它：写入是纯顺序的，索引在 RAM 中（O(1) 查找），"压缩"是段级驱逐（没有有序运行合并，因为没有排序不变式）。

在规模上日志结构是不可或缺的，因为 SSD 写入耐久性限制了持续随机写。以每块 NVMe 3 GB/s 顺序追加是廉价的；等效的随机 4 KB 写入会在数月内烧毁存储单元。

(b) 为何适合该访问模式

在所有 CDN 工作负载中——静态 Web、API 响应、视频分片、游戏补丁、软件下载、ML 模型制品——访问模式完全一致：

  • 读主导 （即使在未命中率较高的工作负载中，读操作约占 99%）。
  • 工作集符合 Zipf 分布 / 幂律 ——1% 的键服务 80% 以上的请求。
  • 写入突发且由外部驱动 （缓存未命中时填充）。
  • 无范围扫描或有序迭代。

内存哈希表 + 日志结构 SSD 的组合胜出：约 100 ns RAM 命中，约 100 μs SSD 命中，以线速顺序写入，整段驱逐无碎片，天然契合"热数据在 RAM，温数据在 SSD，冷数据到源站"。

(c) 驱逐——缓存命中率的核心

朴素 LRU（最近最少使用）在 CDN 规模下是错误的。有两个问题：

  1. 扫描污染：一个大型爬虫遍历 10 万个冷 URL；所有热条目都被替换。普遍出现——Googlebot、archive.org、安全扫描器。
  2. 锁竞争：在每台机器 100 万 rps 时，LRU 链表头部成为全局锁。

LFU（最不常用）：计数器过时；曾经热门的键永远主导。很少单独使用。

ARC（自适应替换缓存）：IBM 经典方案；两个 LRU 列表、幽灵列表、自适应比率。更好；专利限制至 2024 年，因此多年来未出现在广泛使用的系统中。

W-TinyLFU（Window TinyLFU）：现代方案。被 Caffeine、Cloudflare 部分栈以及严肃缓存系统普遍采用。

W-TinyLFU layout:

  | window LRU   | ---filter---->  | main cache (SLRU = Segmented LRU)|
  | (1% of cache)|                  |  +----------+ +-------------+    |
  |              |                  |  | protected| |  probation  |    |
  | new entries  |                  |  |   80%    | |    20%      |    |
  | get a chance |                  |  +----------+ +-------------+    |
  +--------------+                  +----------------------------------+
                                          ^
                                          |
                            +-------------+--------------+
                            | Count-Min Sketch (4-bit    |
                            | counters, 4 hash funcs)    |
                            | — tracks key frequency     |
                            |   approximately            |
                            +----------------------------+

每次访问：

  1. 新条目进入 窗口 （小型 LRU）。新条目始终获得一次机会。
  2. 当窗口驱逐时，受害条目进入 准入过滤器：count-min sketch 将其估计频率与 main 中即将被驱逐的条目比较。频率更高者胜出。
  3. 主缓存 是 SLRU（分段 LRU）：受保护区（80%）和候选区（20%）。新条目进入候选区；第二次访问后晋升到受保护区。
  4. count-min sketch 会被 老化 （每 N 次访问将所有计数器减半），使曾经热门的条目不会永远主导。

为何在差异显著的工作负载中均胜出：

  • 静态资源 CDN 用于电商目录：过滤器阻止一次性产品变体占用永久热门的首页轮播图的位置。
  • API CDN （Stripe、GitHub Pages）：即使一次性 /api/v1/health 请求涌入，也能将 /api/v1/customers/{rare_id} 等高频低成本端点保留在缓存中。
  • 视频 CDN：热门节目主导频率；低频的罕见语言字幕请求被过滤。

实测 W-TinyLFU 在相同内存预算下比 LRU 实现高约 5-10 个百分点的命中率。在 95% 基准下提升至 99% 将使源站负载减半。在规模化场景下这意味着数百万美元的工程价值。

(d) 缓存键组成——隐藏的维度

朴素缓存键是 (method, host, path)。真实的缓存键需要更多考量：

cache_key = SHA1(
    method
    || ":" || host
    || ":" || path
    || ":" || canonicalize_query(query_params)
    || ":" || vary_fingerprint(req_headers, response_vary)
    || ":" || ( bucket(cookies, allowed_cookies) )
)

每个组成部分都是潜在的陷阱，其后果因工作负载而异：

  • Query params：是否应该 ?utm_source=twitter 对缓存进行分片？不——应将其剥离。是否应该 ?w=400 （图片宽度）进行分片？是的。CDN 配置必须明确哪些参数携带内容。一条错误的 UTM 规则可能在一个下午内将命中率暴跌 50 个百分点。
  • Vary headers: Vary: Accept-Encoding 是可以接受的（约 3 个变体）。 Vary: User-Agent 是灾难性的（每个 URL 约 1 万个变体，导致缓存雪崩）。当 API CDN 出现这种情况时，通常是因为 Vary 意外地将每个客户端的关联请求头纳入了其中。
  • Cookies: Cookie: session=... 加入缓存键 = 每用户独立分片 = 毫无意义。哈希前剥离无关 cookie；对有意义的 cookie 分桶（登录态 vs 匿名、国家分桶），并以桶值作为缓存键。

审计缓存键是任何 CDN 用户所能做的杠杆最高的优化——无论是图片托管、API 网关、新闻站点，还是任何其他场景。

(e) PoP 内按 URL 分片的一致性哈希

在一个拥有 N 台缓存机器的 PoP 中，每个 URL 应始终路由到同一台机器（从而只缓存一次，而非 N 次）。朴素的 hash(URL) % N 当机器增减时会失效——所有键都会重新映射。 Consistent hashing 每台机器约 150 个虚拟节点：

0 -------------------- ring ----------------------> 2^32

[B0 vnode1][B2 vnode1][B1 vnode1][B0 vnode2][B3 vnode1][B1 vnode2]...

hash(URL) lands on the ring; first vnode clockwise wins.

If B2 dies: only B2's vnodes are removed; keys formerly on B2 vnodes
            go to the next vnode. ~1/N of keys remap; the rest are
            unaffected.

If a new box B4 is added: vnodes inserted; ~1/N of keys move to B4.

优雅扩缩容。重新映射时的缓存缺失是有界的；其余内容保持热缓存状态。Rendezvous hashing（HRW）是一种变体——在带权重的机器上确定性略高，CPU 开销略多。整体结构相同。

(f) L2 存在性探测的 Bloom filter

当 L1 未命中并考虑向 L2（中间层）发起请求时，一种选择是使用 Bloom filter 来判断"L2 中存在哪些 key"。

  • 确定不存在 → 跳过 L2，直接访问 shield/origin（节省一跳）。
  • 可能存在 → 查询 L2。

适用于冷门长尾内容；每个 L2 key 约占 ~10 位时，误判率约 ~1%。许多 CDN 跳过此步骤——L2 探测的开销足够小，复杂度并不总是值得的。

(g) 逐字节走读一次请求

柏林的一位用户请求 https://cdn.example.com/asset.bin?v=400 通过 HTTPS 发起访问。无论资源是 JPEG、JSON API 响应、视频片段还是 ML 模型文件，流程完全相同——缓存层不关心内容类型。

1. DNS — 8 ms
   - Client resolves cdn.example.com.
   - 1.1.1.1 returns 104.16.x.x (anycast IP) within ~10ms (often cached).

2. TCP / TLS handshake — 30 ms
   - BGP routes the SYN to Frankfurt PoP (5 ms one-way).
   - SYN/SYN-ACK/ACK ~10 ms RTT.
   - TLS 1.3 1-RTT handshake — 5 more ms.
   - Session resumption → 0-RTT, skip handshake; gain ~20 ms.

3. HTTP/2 stream open — 1 ms
   - Client opens stream 1; sends GET /asset.bin?v=400 with headers.

4. PoP receives request — 0 ms (now)
   - L4 LB (XDP/eBPF on the receiving box) inspects SNI (Server Name
     Indication) or unencrypted Host header, then hashes the URL.
   - hash(URL) -> cache box B17.
   - L4 LB forwards (single hop within PoP — sub-ms).

5. Cache box B17, edge cache layer — 0.5 ms
   - Strip irrelevant query params; keep ?v=400 (declared content-bearing).
   - Compute cache_key = SHA1("GET:cdn.example.com:/asset.bin:v=400::").
   - Lookup in in-RAM hash table.
     +-- HIT (RAM): return bytes in ~0.1 ms (memcpy + framing).
     +-- HIT (SSD): RAM index -> ssd_offset -> 4 KB sector read ~100 us;
     |              larger objects -> multiple sequential reads.
     +-- MISS: proceed to step 6.

6. On miss, ask L2 (mid-tier cache) — 2 ms
   - hash(URL) -> mid-tier shard M3.
   - Cross-PoP RPC in the same region — ~1 ms RTT.
   - HIT -> return; B17 writes object to active segment + index.
   - MISS -> proceed to step 7.

7. On L2 miss, ask shield — 5 ms (or skip if no shield)
   - Request coalescing: if another concurrent request already has
     an outstanding fetch for this key, attach to that promise.
   - Shield issues conditional GET (If-None-Match) if it has stale.

8. On shield miss, go to origin — 50–200 ms
   - Frankfurt shield to us-east-1 origin ~90 ms RTT TLS-warmed.
   - Origin responds with Cache-Control: max-age=86400, immutable.

9. Propagate down the hierarchy
   - Shield caches -> L2 caches -> L1 (B17) caches.
   - Each tier writes to its active log-structured segment and updates
     in-RAM index.
   - Index update is in-memory only; no fsync (reconstructible from
     segment headers on restart).

10. Respond to client — 1 ms
    - HTTP/2 DATA frame(s); zero-copy via sendfile() where possible.

Total cache HIT:    ~ 12-15 ms (DNS + TLS + cache read).
Total cache MISS:   ~ 100-300 ms (depending on origin distance).

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 持久化节点 一致性哈希 每个节点约 150 个虚拟节点： 平滑扩缩容。重新映射时的缓存缺失是有界的，其余部分保持热态。Rendezvous 哈希（HRW）是一种变体——在带权重的节点下略微更具确定性，CPU 开销略多，整体形态相同。

（f）用于 L2 存在性探测的 Bloom filter

当 L1 未命中并考虑查询 L2（中间层）时，一个选项是 Bloom filter "L2 中存在的键"集合。

确定不存在 → 跳过 L2，直接访问 shield/源站（节省一跳）。

可能存在 → 查询 L2。

  • 适用于冷长尾内容；每个 L2 键约 10 bits 时误判率约 1%。许多 CDN 跳过这一步——L2 探测的开销足够小，复杂度未必值得。 /Products/123 。 /products/123 通常指向同一资源——但 URL 规范允许路径区分大小写，因此必须明确告知 CDN。Host 必须始终转为小写；路径通常也应如此，除非源站明确区分大小写。
  • 尾部斜杠。 /about vs /about/ ——通常是同一页面。若 CDN 不做规范化，两种变体将分别独立缓存，热门页面就会产生两个缓存条目，各自命中率减半。
  • 默认端口。 https://example.com/path 。 https://example.com:443/path ——同一资源。哈希前应去掉默认端口。
  • 百分号编码。 /foo%20bar 。 /foo+bar 。 /foo bar 通常应规范化为相同形式。RFC 3986 规范化（对百分号编码序列中的十六进制数字统一大小写，对非保留字符解码）。
  • 查询参数顺序。 ?a=1&b=2 。 ?b=2&a=1 属于相同的查询。哈希前按字母顺序排序以进行规范化。若 CDN 不做此处理，参数顺序与常规客户端不同的客户端每次请求都会缓存未命中。
  • 仅用于追踪的参数。 utm_source, utm_medium, utm_campaign, fbclid, gclid 应当被 在哈希前剥除 （但如果源站需要用于分析，则在转发至源站时予以保留）。一条将 UTM 参数纳入缓存键的错误规则会彻底击垮命中率：来自不同营销活动的每个链接点击都会产生一条新的缓存条目。一场由 Twitter 驱动的产品发布，可能在一个下午之内就将命中率从 99% 拉低至 30%。
  • 携带内容的参数。 相反， ?size=large, ?w=400, ?lang=fr, ?v=42 属于携带内容的参数， 必须 须保留在缓存键中。CDN 配置本质上是一份参数白名单，用于声明哪些参数影响内容。

一条正确的规范化流水线：

raw_url
  -> lowercase host
  -> strip default port
  -> RFC 3986 path normalize (case-fold hex, decode unreserved)
  -> resolve relative segments (. and ..)
  -> trailing-slash policy (strip or canonicalize per route)
  -> parse query string
  -> drop params not in content-bearing allowlist
  -> sort remaining params alphabetically by name
  -> stable-encode values
  -> emit canonical_url
  -> feed canonical_url into the cache_key hash

此类代码应当封装为单一函数，经过审计、测试，并在每个 PoP 之间共享。一旦各层实现出现分歧（L1 以一种方式规范化，L2 以另一种方式），就会出现某一层命中而下一层未命中的情况——缓存层级实际上形同虚设。

(b) Vary 响应头处理

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 Vary 响应头告知缓存："本响应会因请求头 X 的不同而变化，以 X 作为缓存键。"原则上没问题，但使用不当在实践中会造成灾难性后果。

  • Vary: Accept-Encoding 没有问题。实际环境中约有 ~3 个取值（gzip, br，identity）。每个 URL 三个缓存变体是可以接受的。
  • Vary: Accept-Language 处于临界值。取值达数十个；若源站确实提供本地化内容，这是正确性所需付出的代价。更好的做法是：在 CDN 配置中将 Accept-Language 分桶为小集合（en、es、fr、de、ja、"other"），并以桶值作为缓存键。
  • Vary: User-Agent 则是灾难性的。大型网站的 UA 字符串多达数万种。每个变体意味着一个独立的缓存条目。一个带有 Vary: User-Agent 的 URL 缓存命中率趋近于零。添加它的源站开发者很可能是想向移动端和桌面端提供不同的 HTML，却选错了工具。正确的做法是在边缘（Workers / VCL）使用 UA 分类器对 UA 进行分桶，并在缓存键中使用合成头。
  • Vary: Cookie 若 cookie 中包含 session ID 则更糟。每个用户都会获得独立的缓存条目，整个 CDN 退化为按用户存储的系统，命中率为零。
  • Vary: * （字面上就是一个星号）表示"此响应对每个请求都是唯一的，永不缓存。"服务器有时会发出此头，CDN 会通过完全绕过缓存来遵守它。

正确的运维规范是： CDN 运营方需审查所有来自源站的 Vary 响应头 并通过配置加以限制（将列表强制限定为已知安全值），或与源站团队协作将其移除。Vary 应是经过深思熟虑的决策，绝不应无意中引入。

在内部，缓存将 Vary 实现为两级查找：

1. URL-canonical key  -> "variant map" entry
   Variant map records which request headers Vary on (per response)
   plus a list of stored variants, each tagged by a fingerprint of
   the relevant request header values.

2. For each request, compute the variant fingerprint from the
   listed Vary headers, then look up the matching variant.

Lookup is still O(1) on a hit but two hash steps. The variant map
itself is small; the cost is the variant fragmentation when Vary is
on a high-cardinality header.

(c) 基于 Cookie 的缓存绕过与分桶策略

Cookie 是缓存键最危险的输入来源，因为它们通常携带会话身份标识。有三种策略：

  1. 在计算缓存键之前剥离所有 Cookie。 纯静态内容的默认做法。源站响应应设置 Cache-Control: public 且不包含任何用户特定数据；CDN 不会将 Cookie 纳入缓存键。
  2. 当存在敏感 Cookie 时，绕过缓存。 对于混合页面（同一 URL 同时服务于已登录用户和匿名用户），CDN 配置为："如果 Cookie: session=... 已设置，则将缓存模式设为绕过；直接回源服务，不缓存响应。」这会牺牲已登录流量的命中率，但可确保不发生跨用户数据污染。
  3. 按 cookie 分桶并以桶值作为缓存键。 当你希望对已登录用户和匿名用户分别缓存时：提取用户是否已认证（session cookie 存在则为 "logged_in_v1"，否则为 "anon_v1"），将该字符串而非 cookie 原始值写入缓存键。每种变体只缓存一次。

错误配置示例：\n- 将 Cookie: header 原文写入缓存键。每个会话都成为唯一的缓存键——既毫无意义，又同时泄露 PII。\n- 从缓存键中剥离了 cookie，却保留了 非 来自响应变体的数据——缓存中存的是用户 A 的数据；用户 B 的请求命中了该键；用户 B 看到了用户 A 的数据。这就是"我们不小心缓存了 PII"的灾难性事故。

典型事故模式：开发者为页面添加了个性化功能（一个小型的按用户展示的组件），却忘记添加 Cache-Control: private 或 no-store。CDN 的默认行为是"缓存一切"。CDN 并不将认证 cookie 计入缓存键（这本身是对的——否则会使命中率崩溃）。于是，该 URL 的每个请求都会返回最初被缓存的响应，即 第一个访问该 URL 的用户。如果那个首个用户处于已登录状态，那么此后所有用户——无论是否登录——都会看到该用户的姓名、余额乃至更多信息。多个大型网站都曾公开发生过这一事故，这是 CDN 配置灾难的典型案例。

缓解措施：\n- 对动态响应采用默认拒绝策略：只有明确标记为可缓存的路径或内容类型才会被缓存。\n- 源站必须始终设置 Cache-Control: private 或 no-store 针对每个用户的响应。\n- WAF 规则：如果响应体包含常见的用户特定标记（Set-Cookie、用户相关 HTML 模式）且请求未携带身份认证，则标记为可疑并拒绝缓存。\n- 通过综合监控分别拉取 "anonymous" 和 "logged-in" 两种变体并进行差异比对。

(d) 缓存键爆炸

与缓存投毒相对的另一种故障模式是 cache key explosion ——变体过多，命中率为零。症状：命中率无明显原因地跌破 50%；缓存填充率飙升（每个请求都触发填充）；源站负载攀升。

常见原因：\n- 在高基数请求头（User-Agent）上使用 Vary。\n- 追踪参数被纳入缓存键。\n- Cookie 被纳入缓存键。\n- Request-ID 或 correlation-ID 被意外哈希进去。\n- 来自异常客户端的随机查询参数（Chrome 的 ?_=12345 cache busters, ad SDK ?ab_segment=...）。\n- URL 本身包含每次请求独有的令牌（每次调用均带唯一签名的签名 URL）。

检测：监控仪表盘，指标为 cache-key-to-URL ratio 每个主机的情况。若某主机的缓存键数量比不同 URL 数量多出 10 倍，说明有东西在分裂缓存键。按主机统计的命中率趋势看板能在数小时内（而非数天内）发现此问题。

"每个用户拥有唯一缓存键 = 零命中"这一病态现象，从统计角度来看， the most common single cause of bad CDN bills。每次缓存未命中都消耗源站出口带宽和计算资源；一个命中率 99% 的站点若跌至 50%，相当于源站工作量增加约 50 倍，且通常看不到任何其他明显变化。

§6. 缓存失效协议

CDN 会激进地缓存内容；更难的问题是如何让它们停止缓存。失效机制处于一个从"代价高但立即生效"到"免费但响应慢"的连续谱上。

(a) 按 URL 清除 — 原子操作原语

最简单的 API： PURGE https://cdn.example.com/path/asset.json 针对 CDN 控制平面发起。流程如下：

  1. 客户 API 调用（或源站 webhook）。
  2. 控制平面验证（认证、速率限制、surrogate key 解析）。
  3. 扇出至各区域，再到各 PoP，再到各缓存节点。
  4. 每个节点从内存索引中移除该条目。磁盘上的 segment 不会被重写——该条目只是变为未引用状态，最终随 segment 一同被 GC（垃圾回收）。

端到端延迟目标：\n- Cloudflare：全球所有 PoP p99 < 150 ms（通过 Quicksilver，见 (e)）。\n- Fastly：全球 p99 < 150 ms（"即时清除"）。\n- Akamai：旧版 CCU（Content Control Utility）约 5-10 s；较新的 Fast Purge 约 150 ms。\n- CloudFront：数分钟（因其失效传播通过较为保守的控制平面进行）。

成本模型：大多数 CDN 每天提供 1000-10000 次免费清除；批量清除按量计费。清除量是一个调节旋钮——过度使用是架构异味的信号（应该改用 URL 版本化）。

(b) Surrogate keys — 按标签失效

按 URL 清除在"需要同时失效 electronics 类目下所有 50,000 个商品页"这类场景中无法扩展。 Surrogate keys 解决了这一问题：源站为每个响应打上一个或多个标签；客户按标签发起清除；CDN 的标签索引则遍历所有受影响的 URL。

Origin response:
   GET /product/123
   Cache-Tag: product:123 category:electronics brand:apple
   Cache-Control: public, max-age=300

CDN indexes:
   product:123       -> { "/product/123" }
   category:electronics -> { "/product/123", "/product/456", ... 50,000 entries }
   brand:apple       -> { "/product/123", "/product/789", ... 1200 entries }

Customer purges:
   PURGE-TAG category:electronics
   -> CDN walks the index, purges all 50,000 URLs

   PURGE-TAG product:123
   -> CDN purges just /product/123 plus any other URLs that referenced it

Cloudflare 将其称为 "Cache-Tag" 响应头；Fastly 将其称为 "Surrogate-Key"（遵循 RFC 5861 惯例）。Akamai 则将其支持为 "Edge-Cache-Tag"。模式完全相同；每个 PoP（或全局）的倒排索引是实现细节。

在实际操作中，代理键是每个电商网站处理品类更新、每个 CMS 处理标签/话题变更、每个 API 处理"某客户的数据在 N 个端点发生变更"的标准手段。没有代理键，你就必须枚举逻辑实体更新所触及的每一个 URL——对任何动态场景来说都是不可能完成的任务。

实现成本：标签索引。每个 (tag, URL) 对约占 ~10 字节。一个拥有 1000 万 URL、平均 5 个标签的站点，每个 PoP 的索引约需 ~500 MB——不可忽视，但尚在可控范围内。

(c) 按前缀清除——批量操作利器

有时你需要"清除某路径下的所有内容 /blog/." Cloudflare 在企业版方案中支持前缀清除，Fastly 支持 glob 风格的清除。其实现是对 URL 键控索引中前缀的范围遍历。

注意：前缀清除代价高昂，因为它没有预计算索引。CDN 需要在每个 PoP 遍历所有 URL。对于前缀下拥有数百万 URL 的站点，这可能耗时数分钟乃至数小时；代理键通常是更好的选择。

(d) 版本化 URL——"按部署清除"的技巧

§15 问题 1 中提到的最重要模式，值得单独展开阐述。

Bad pattern (TTL + purge):
   /static/app.js                  (TTL 1 hour, requires purge on deploy)
   Deploy new version -> purge /static/app.js across 300 PoPs
   Race: some users get old version for up to 150 ms after deploy
   Race: misconfigured PoP serves old version indefinitely

Good pattern (immutable + versioned):
   /static/app.abc123.js           (TTL 1 year, immutable)
   /static/app.def456.js           (next deploy)
   No purge ever required. The HTML simply references the new URL.

字节内容永不改变。URL 随每次构建而变化（通常使用内容哈希）。旧 URL 继续从缓存提供服务（适用于任何仍引用它们的客户端——缓存在边缘节点或浏览器中的旧 HTML）。新 URL 由新 HTML 引用，在首次请求时未命中缓存，按需填充，并永久缓存。

为什么这是正确答案：\n- 无协调问题。按 HTML 引用，部署是原子性的。\n- 无需清除扇出，节省控制平面带宽。\n- 无竞态条件。某个 URL 永远对应同一组字节。\n- 可以安全使用更长的 TTL。 max-age=31536000, immutable （1 年，不可变）完全没问题。\n- 浏览器缓存同样受益，完全消除往返请求。

Cache-Control: immutable （RFC 8246）告知浏览器："即使用户主动刷新，也不要重新验证此资源。"若没有 immutable，硬刷新会重新获取每一个资源；有了它，则只有 HTML 会被重新获取。

versioned-URL 模式几乎适用于所有静态资源：JS、CSS、图片、字体、视频分片。它不 非 适用于 HTML 本身（URL 必须保持 /about）——对于 HTML，需依赖短 TTL 加清除机制。分工如下：短生命周期的 HTML 配合 surrogate-key 清除；长生命周期的资源使用内容哈希的不可变 URL。

§6. 缓存失效协议

  • Cache-Control: max-age=N CDN 会积极地缓存内容；更难的问题是让它们停止缓存。失效机制存在于一个从"成本高但即时生效"到"免费但生效缓慢"的谱系之中。
  • Cache-Control: s-maxage=N (a) 按 URL 清除——原子操作原语
  • Cache-Control: public 最简单的 API：
  • Cache-Control: private — 仅允许浏览器缓存；CDN 不得缓存。
  • Cache-Control: no-store — 不得被缓存。
  • Cache-Control: no-cache — 可以缓存，但在复用前必须重新验证。
  • Cache-Control: must-revalidate — 内容已过期时，重新验证；不得提供过期内容。
  • Cache-Control: immutable — 该 URL 对应的字节内容不会改变；跳过重新验证。
  • Cache-Control: stale-while-revalidate=N — 在 TTL 过期后的 N 秒内，提供过期内容并在后台刷新。
  • Cache-Control: stale-if-error=N — 在 TTL 过期后的 N 秒内，若源站不可达，则提供过期内容。

隐藏的亮点是 stale-while-revalidate：立即返回一个过期响应，同时在后台触发刷新。将重新验证时的尾延迟降为零。结合 surrogate-key 清除机制，可实现"始终新鲜且响应迅速"的效果。

stale-if-error 是一种弹性原语：当源站不可达（504）时，返回过期内容而非 503。对于大多数业务场景，有界过期优于服务中断。

§8. Video CDN 分片缓存

视频是全球 CDN 上最大的单一工作负载类别（Netflix + YouTube + Twitch + TikTok 以及其余流媒体服务合计占互联网总字节数的 >70%）。其协议和缓存模式与通用 Web 缓存差异显著，有必要单独讨论。

(a) HLS — HTTP Live Streaming（Apple）

由 Apple 于 2009 年推出；已成为 iOS、macOS、Safari 及大部分其他平台事实上的标准。其核心思路是：不通过自定义协议传输单个视频文件，而是将视频 切分为约 2–10 秒的分片，每个分片是一个独立的可通过 HTTP 访问的对象，由一个清单文件（.m3u8）按顺序列出各分片。

manifest.m3u8 (master playlist — multi-bitrate)
   #EXTM3U
   #EXT-X-STREAM-INF:BANDWIDTH=400000,RESOLUTION=640x360
   low/index.m3u8
   #EXT-X-STREAM-INF:BANDWIDTH=2500000,RESOLUTION=1280x720
   med/index.m3u8
   #EXT-X-STREAM-INF:BANDWIDTH=8000000,RESOLUTION=1920x1080
   high/index.m3u8

high/index.m3u8 (media playlist — segment list for one bitrate)
   #EXTM3U
   #EXT-X-TARGETDURATION:6
   #EXT-X-VERSION:3
   #EXT-X-MEDIA-SEQUENCE:0
   #EXTINF:6.0,
   segment-0000.ts
   #EXTINF:6.0,
   segment-0001.ts
   ...
   #EXT-X-ENDLIST

客户端轮询 manifest，按顺序下载分段，并根据网络速度切换码率变体（ABR — 自适应码率）。每个分段大小通常为 500 KB - 5 MB。

为何这对 perfect HTTP 缓存而言：

  • 每个分段都是不可变的 HTTP 对象，URL 永不改变。 Cache-Control: max-age=31536000, immutable.
  • 分段足够小，可轻松放入缓存内存或 NVMe 中；许多分段可并行传输。
  • 热门视频 → 分段命中 CDN 缓存节点；一次缓存填充可服务数百万观众。
  • 95-99% 的命中率目标是可达到的，因为热门内容相对于缓存而言体量较小（冷门视频的长尾会未命中缓存，但它们并不是字节量的主要来源）。

manifest 本身是唯一无法简单缓存的内容：对于 VOD（点播） manifest 本身也是不可变的（完整的分片列表已固定）；对于 live manifest 每隔几秒更新一次，追加新的分片（旧分片则随滑动窗口直播的进行而淘汰）。

(b) DASH — Dynamic Adaptive Streaming over HTTP

MPEG-DASH，ISO/IEC 23009-1。概念上与 HLS 完全相同——manifest 加分片——但采用基于 XML 的 .mpd manifest，而非 .m3u8，支持更广泛的编解码器和容器格式（分片格式可以是 .mp4 而非 .ts），且具备更强的元数据扩展性。在 Apple 生态之外被广泛采用：YouTube、Netflix（含内部扩展）、大多数 Android 播放器以及智能电视。

从 CDN 缓存层面来看：与 HLS 完全相同。manifest 是小型文件，各分片是以 URL 为键的独立 HTTP 对象。相同的分片缓存机制对两者均适用。

许多 CDN 同时提供 来自同一源站的两种格式 — 源站打包一次，同时分发两种清单格式和共享的分片存储。CMAF（Common Media Application Format）正是为此而生的标准：一个单一分片文件（fMP4 分片式 MP4）可同时被 HLS 和 DASH 播放器使用。

(c) 清单文件缓存策略

清单文件体积极小（仅几 KB），但请求量巨大——每位观众每次会话都会拉取一次，直播时还会持续刷新。缓存策略因模式不同而有所差异：

  • VOD 清单：在视频的整个生命周期内不可变。 Cache-Control: max-age=86400 即可。命中率接近 100%，实质上是静态资源。
  • 直播清单：随着新分片的加入，每隔约 1-10 秒就会更新。 Cache-Control: max-age=1 甚至 no-cache。CDN 缓存一秒钟；每次刷新时，所有 PoP 同时重新拉取。配合请求合并与严格的 TTL 抖动，这是可行的——源站每个 PoP 每秒只会看到约 1 次清单请求。
  • 低延迟 HLS（LL-HLS — Low-Latency HTTP Live Streaming） 。 低延迟 DASH（LL-DASH） 是将直播延迟从约 30 秒降至约 3 秒的扩展方案，其原理是在分片内使用分块传输编码。CDN 必须支持部分分片缓存——在下一个 1.5 秒仍在生产时，即可缓存 2 秒分片的前 500 毫秒。具体实现是在一次传输过程中同时完成流式代理与缓存填充。

Twitch、YouTube Live 以及各大体育赛事直播平台现已在生产环境中全面使用 LL-HLS 或 LL-DASH。整个行业将直播延迟削减 25 秒的工程攻坚发生在 2020-2024 年间。

(e) Netflix Open Connect——全自建方案

Netflix 的设计与第三方视频 CDN 有本质差异：

  • 约 17,000 台缓存设备部署 在 ISP 网络内部 （而非互联网交换节点）。
  • 每台设备配置：约 250 TB SSD、100 Gbps 网卡，单用途（仅用于视频）。
  • 在非高峰时段预填充内容。 Netflix 预知明天的热门内容（新上线内容、推荐结果），并在夜间将热门内容集推送到每台设备。白天的流量几乎全部来自本地 SSD，从不穿越互联网骨干网。
  • 热门内容的命中率实际上达到 100%（因为已预先拉取）。长尾内容则依次回溯填充链路：设备 → Netflix 区域 POP → 源站。
  • 每家 ISP 免费获得硬件（Netflix 提供设备），作为回报，ISP 需在其网络内提供托管位置、供电和带宽。
  • 按每日传输字节数计算，这是全球最大的 CDN，但它只服务于 Netflix。单一客户、单一工作负载、部署在 ISP 内部、内容不可变、需求可预测。

之所以能成功，原因如下：\n- 视频文件 不可变 （无清除问题）。\n- 热门内容子集相对于磁盘而言 体量较小 （排名前 5000 的内容约为 PB 级；每台设备存储其中相关的部分）。\n- 需求 可预测 （推荐引擎在当晚就能预知明天的热门内容）。\n- 单一工作负载 = 无需通用可编程性、WAF 和边缘计算。Netflix 从头到尾优化整个技术栈。

架构启示： 当工作负载足够统一时，自建优于购买。同样的经验也适用于 Google 的 GGC（Google Global Cache）、Meta 的边缘 PoP 以及 Apple 的边缘缓存。经济效益的转折点约在持续 100 Tbps；Netflix 已超出这一门槛 5-10 倍。

(f) DRM 与签名分片

高端视频内容引入了 DRM（Digital Rights Management）：分片经过加密，客户端需单独获取解密密钥。加密后的分片可以正常缓存（拥有相应授权的用户所获得的加密字节完全相同）；密钥请求则是针对单个用户的，无法缓存。CDN 负责提供分片；独立的授权服务器负责签发密钥。

签名 URL（在 URL 中嵌入带过期时间戳的 HMAC 或 RSA 签名）是另一种高端视频常见模式。缓存键会去除签名部分（否则每个签名 URL 都是唯一的）。CDN 在边缘——Workers 或 VCL——对签名进行校验：有效签名则从缓存返回，无效或过期则拒绝请求。

§9. 边缘图片优化

图片占典型网页字节数的 50-70%。边缘图片优化——在 PoP 处对图片进行格式转换、尺寸调整和重新压缩——是现代 CDN 所提供的性价比最高的功能之一。

(a) 格式转换（JPEG 转 WebP 转 AVIF）

图片格式持续演进，每一代都带来了显著的压缩提升：

格式 年份 与 JPEG 的典型压缩对比 浏览器支持情况
JPEG 1992 基准 全平台支持
PNG 1996 无损；照片文件体积大 全平台支持
WebP 2010 体积缩小约 25-35% 所有现代浏览器
AVIF 2019 体积缩小约 50% 大多数现代浏览器；Safari < 16 不支持
JPEG XL 2021 体积缩小约 40-60% 支持有限（Chrome 已放弃，Safari 正在引入）

传统方式：源站存储 JPEG，所有人获取 JPEG。现代方式：源站存储一份源图；CDN 读取来自客户端的 Accept 头部信息，动态转换格式，并缓存转换后的版本。

边缘实现方案：\n- Cloudflare Polish / Cloudflare Image Resizing：启用功能标志即可；源站保持 JPEG；CDN 向支持的浏览器提供 WebP/AVIF。\n- Fastly Image Optimizer：原理类似，可通过 VCL 配置。\n- AWS CloudFront 通过 Lambda@Edge 函数：编写一个 Lambda 调用 Sharp/Pillow 库处理源图，返回转换后的版本。\n- ImageKit、imgproxy、Imgix：第三方 SaaS，位于源站和 CDN 之间，或作为源站运行。

缓存键包含格式标识（format=webp），以便分别缓存各种格式。协商模式：

1. Browser sends GET /photo.jpg with Accept: image/avif, image/webp, image/*
2. CDN edge inspects Accept, picks best format the browser supports.
3. CDN cache key includes the chosen format -> cache lookup.
4. On miss: CDN fetches origin's JPEG (cached separately under a "source" key),
            converts to AVIF, caches the AVIF variant.
5. Subsequent requests with same Accept profile hit the AVIF variant directly.

这实际上使每张图片的缓存条目增加了三倍（JPEG + WebP + AVIF），但每种格式仍可永久缓存（基于源图哈希，不可变）。

(b) 自动缩放

一张源图可以提供数十种尺寸：\n- 移动端缩略图：150x150\n- 平板端缩略图：300x300\n- 文章头图：1200x630\n- Hero 图：2400x1200\n- 视网膜屏变体（2x、3x）

不做边缘缩放时，源站必须存储每种变体（存储量爆炸），或者页面加载最大尺寸变体再由浏览器缩小（带宽浪费）。边缘缩放：源站存储 一张 主图；CDN URL 中包含缩放参数（?w=1200&h=630&fit=cover）；缓存键包含参数；各变体独立缓存。

Cloudflare Image Resizing URL 模式： /cdn-cgi/image/width=400,quality=85/path/to/source.jpg。Fastly：通过查询参数或 VCL 注入。AWS：Lambda@Edge 结合参数解析。

源站存储量降低 10 倍或以上。命中率保持较高，因为用户实际请求的变体是一个较小的集合（十几种尺寸即可覆盖 95% 的请求）。

(c) 根据连接速度调整图片质量

客户端发送 Save-Data: on （Chrome）或 ECT （有效连接类型—— slow-2g, 2g, 3g, 4g）客户端提示。边缘代码根据此信息调整图片质量：

if (request.headers.get("Save-Data") === "on") {
  return convertImage({ format: "webp", quality: 50, width: limit });
} else {
  return convertImage({ format: "avif", quality: 80 });
}

缓存键扩展为包含"连接质量分桶"，使各分桶单独缓存。大多数 CDN 图片服务会隐式地执行此操作。

(d) 经济效益分析

核心价值主张： "在边缘节省 50% 的图片字节数。" 具体测算：

  • 网站每月提供 100 TB JPEG 图片（质量系数 80）。
  • 启用边缘优化：格式协商（70% 的流量使用 AVIF，25% 使用 WebP，5% 保持 JPEG）。
  • 新字节量：70% × 50% + 25% × 70% + 5% × 100% = 35% + 17.5% + 5% = 约 57.5% = 约 57 TB/月。
  • 节省了每月 43 TB 的出口流量。
  • 按 CloudFront 出口费率 $0.085/GB 计算：43 TB × 1024 GB/TB × $0.085 = 约 $3,742/月的节省。
  • Cloudflare Image Resizing 约增加 $1/百万请求 = 对于该流量规模，每月增加数百美元费用。
  • 净效益：每月节省数千美元出口费用，边缘计算成本仅需数百美元。

对于每月图片流量达 1 PB 的网站（大型媒体平台、规模化 Shopify 商家），每月节省额可达数万美元。该功能的成本在数天内即可收回。

额外收益：图片体积更小，加载更快，改善 Core Web Vitals 指标（尤其是 LCP——Largest Contentful Paint）——直接影响 SEO 排名和转化率。

(e) 缓存层面面临笛卡尔积问题

负面影响：缓存条目数量爆炸。一张源图现在会产生格式 × 尺寸 × 质量 × 连接分桶的多维变体。CDN 提供商的存储成本上升，并以按图片转换计费的方式转嫁给用户。数学上仍然值得进行转换，但并非毫无代价。

缓解措施：\n- 通过白名单限制可配置的尺寸（仅允许设计系统使用的 12 种尺寸，不允许任意 ?w=N）。\n- 使用内容哈希 URL，使每张图片的变体集合显式可见。\n- 监控每个源图的变体数量；对异常增长发出告警。

§10. 边缘计算

传统 CDN 只提供字节传输。边缘计算允许你在缓存层面——即请求到达与响应之间，或替代缓存查找的位置——运行 代码 代码。这种演进源于 CDN 提供商意识到"提供缓存前的鉴权检查"或"对响应进行 JSON 转换"是其缓存节点的自然扩展。

(a) Cloudflare Workers — V8 隔离体

Workers 在 V8 隔离体（与 Chrome 标签页隔离所用的同一引擎）中运行 JavaScript 或 WebAssembly。每个 Worker 每个请求对应一个隔离体，但与数千个其他 Worker 共享 同一个 V8 进程 ——启动仅需分配一个隔离体，冷启动时间通常约为 1-5 毫秒。"Bundled"计划下每个请求内存约 128 MB，CPU 约 30-50 毫秒（"Unbound"计划更多）。

核心架构选择： 使用 V8 隔离体而非容器或虚拟机。 Docker 容器启动需要数百毫秒。V8 隔离体启动仅需个位数毫秒，因为它与运行时共享进程内存；隔离在语言虚拟机层面实现。这使 Cloudflare 能够在请求路径上运行代码而不产生延迟损耗。

权衡取舍：无法运行原生二进制文件，无法进行系统调用，I/O 受限。你可以通过 HTTP 进行请求，调用 KV、R2 或 Durable Objects，进行 CPU 计算。但无法绑定套接字、mmap 文件或派生进程。沙箱限制在 JavaScript 语言层面。

(b) Fastly Compute@Edge — WebAssembly，零冷启动

Fastly 将 WebAssembly（由 Rust、Go、AssemblyScript 等编译而来）提前编译，并在其 Lucet/Wasmtime 运行时中执行。 冷启动时间低于毫秒级 ，因为 WebAssembly 模块在首次请求时通过 mmap 从磁盘加载，且 JIT 编译后的代码在各实例间共享。

优势：更底层的语言选择（Rust 的性能），完全无冷启动。权衡：生态系统不如 JavaScript 丰富，需要编译和部署。最适合对性能敏感的转换场景。

(c) AWS Lambda@Edge — 完整 Lambda 运行时，冷启动较慢

CloudFront 的边缘计算方案，支持 Node.js、Python、Java 等——提供完整的 Lambda 运行时。冷启动时间：约 100-500 毫秒（即便在最优情况下也需要数百毫秒）。内存和 CPU 资源远大于 Workers。

冷启动的影响不可忽视：CloudFront Viewer-Request Lambda@Edge 在部署或扩容事件后的第一个请求会增加 100 毫秒以上的延迟。这对延迟敏感的路径而言是灾难性的。缓解措施包括：\n- 在边缘缓存 Lambda 执行结果，后续请求直接跳过 Lambda。\n- 改用 CloudFront Functions ——一种限制更多的运行时（仅支持 JavaScript，API 极为有限，冷启动约 5 毫秒），适合 Header 改写和简单的 URL 转换。

职责划分：Lambda@Edge 用于可接受冷启动的重量级逻辑；CloudFront Functions 用于快速路径。

(d) 边缘计算能做什么

  • 身份验证与授权。 在边缘校验 JWT 令牌、检查会话 Cookie、拒绝未授权请求，无需给源站增加负担。一次 JWT 校验约需 100 微秒；即使是繁忙的边缘节点每秒也能处理数百万次。
  • A/B 路由。 "如果用户属于 B 队列，则路由到不同的源站或设置不同的缓存键。"队列分配通过确定性 JavaScript 代码实现（对用户 ID 取哈希并设定阈值）。
  • 个性化分桶。 根据 Cookie 或地理位置添加 X-User-Bucket: logged-in-en-US 类的 Header；将该 Header 纳入缓存键。每个分桶缓存一次。
  • Header 改写。 添加安全 Header（CSP——Content Security Policy、HSTS——HTTP Strict Transport Security、X-Frame-Options），删除泄露服务器信息的 Header，在不同源站变体之间进行规范化。
  • URL 改写。 将旧 URL 重定向至新 URL，为后端路由改写路径，在遗留源站前实现简洁 URL。
  • Bot 检测。 TLS 指纹识别（JA3/JA4）、行为分析、按指纹限速。
  • 源站选择。 同一域名对应多个源站；边缘根据用户地理位置、A/B 队列或源站健康状态进行选择。
  • 轻量级个性化渲染。 提供缓存的 HTML 骨架；Worker 通过 KV 查找（如用户名）内联用户特定数据。缓存命中 + KV 读取 + 模板填充总耗时约 5 毫秒。
  • 图片与资产转换 （见第 §9 节）。
  • GraphQL 持久化查询。 即便请求体各不相同，也可通过查询哈希缓存 GraphQL 结果。

(e) 边缘计算不能做什么

  • 重量级计算。 边缘运行时通常将每个请求的 CPU 时间上限设为 30-50 毫秒。任何类似机器学习推理（大型模型的 transformer 前向传播）的任务都不适合。较小的模型（用于表格模型的数 MB 权重、简单的嵌入查找）可能适合。
  • 大型状态。 Worker KV 是最终一致性的，最适合存储配置和功能开关。Cloudflare Durable Objects 提供边缘上的有状态 Actor，但需要全局路由层（Durable Object 的查找会路由到托管该对象的 Worker，而非本地 PoP）。不符合"共享最终一致性或单个 Durable Object"模式的状态不适合放在边缘。
  • 长连接。 部分平台支持边缘 WebSocket（Cloudflare 支持）；长连接可能绑定到单个 PoP，并存在故障转移的异常行为。
  • 直接访问数据库 （传统关系型数据库）。边缘运行时无法为传统数据库维护高效的连接池——每次冷启动都需要建立新连接。替代方案：在数据库前搭建 HTTP API（Cloudflare 的 Hyperdrive），或将计算迁移到源站。
  • 任何需要真正数据持久性的场景。 边缘是缓存层；持久性位于源站或云对象存储中。

(f) "边缘计算即新型服务器"的过度宣传

2021-2023 年间曾流行一种观点：边缘计算将取代传统区域服务器。现实已经明朗：边缘计算是源站计算的 补充 ，而非替代。合理的职责划分如下：

  • 边缘：缓存、鉴权、路由、简单转换、个性化分桶、快速路径。
  • 源站（区域级）：业务逻辑、数据库访问、重量级计算、真实数据源。

当今生产环境中部署的绝大多数"边缘函数"都是不超过 100 行的 JavaScript，负责路由、鉴权或 Header 处理。繁重的计算仍然留在源站。

该图所隐藏的技术细节：

  • DNS 解析本身是任播的 ，指向 CDN 的权威 DNS。地球上任何地方的延迟都低于 10 毫秒，因为 DNS 应答器部署在各 PoP 节点。
  • EDNS Client Subnet（ECS） 至关重要：它让权威 DNS 能看到客户端子网，从而返回正确的 PoP，即使 DNS 解析器距离较远也不受影响。若没有 ECS，德国用户使用美国解析器时会被路由到美国 PoP。
  • 任播路由由 BGP 驱动 ——基于网络拓扑而非地理位置。通常二者相关，但任播孤岛（ISP 配置错误导致流量被发送到错误 PoP）是真实存在的故障模式。
  • 中间层与源站 shield 是最被低估的组件。 如果没有它们，每次 L1 缓存未命中都会直接打到源站。有了它们，一个区域内所有 PoP 的请求都会汇聚到少数几个 shield 节点——源站看到的只是每秒少量请求，而非数百万。 N_regions 未命中，而非 N_pops × N_boxes_per_pop.
  • 边缘计算位于 L4 负载均衡器和缓存查找之间。 Workers / Compute@Edge / Lambda@Edge 处理程序直接运行在缓存节点上。

§12. Anycast 故障详解

§2 将 anycast 描述为"客户端的 SYN 在约 1 RTT 内落在最近的 PoP 上"，那是营销版本。工程现实中存在若干边界情况，每个 CDN 团队最终都会踩到。

(a) TCP 连接中途的路由变更

根本问题：TCP 是一种 流程 状态分布在客户端和服务器之间的协议。Anycast 路由是 逐包的，由每一跳的 BGP 决定。如果连接中途发生路由变更——例如某运营商的 BGP 会话抖动并触发重收敛——同一 TCP 连接的报文将开始到达 不同的 PoP。新 PoP 从未见过这条连接，没有对应的 TCP 状态，于是发送 RST（重置）。用户连接中断。

在实际中这种情况较为罕见——BGP 路由通常能稳定数小时乃至数天。但在 BGP 收敛事件期间（大规模对等变更、运营商中断、偶发的网络抖动），数万条进行中的连接可能同时中断。用户看到"连接重置"；部分会自动重试，部分则直接放弃。

(b) "流内路由稳定性"要求

CDN 工程通过在 PoP 内部的 L4 层强制实现 流稳定性以及路由器级别的流稳定性来解决这一问题：

  • ECMP（等价多路径）哈希稳定性。 在一个 PoP 内，多个缓存节点通过 ECMP 共享 anycast IP。路由器根据 (src_ip, src_port, dst_ip, dst_port). 关键点的哈希选择节点：哈希必须在整个连接期间保持 稳定 。若节点增减，哈希函数必须尽可能少地重新分配流（一致性 ECMP）。朴素的取模哈希不可行——添加一个节点会重新分配所有流。
  • 路由器级别的流追踪。 现代路由器追踪流→下一跳的映射，即使底层 ECMP 集合发生变化也能保持稳定。
  • 跨 PoP 故障转移需要显式处理。 若某个 PoP 完全故障，BGP 路由将撤销，流量转移到另一个 PoP。到故障 PoP 的连接丢失；客户端重试；新连接落在新 PoP 上。不存在神奇的连续性——TCP 无法在端点故障后存活。

(c) 流感知 anycast——Cloudflare 的 Unimog

Cloudflare 于 2020 年发布了其 L4 负载均衡器"Unimog"。核心创新：在一个 PoP 内的缓存节点之间维护一张 分布式流表 ，通过 Cloudflare 内部的发布订阅机制进行复制。当某个缓存节点收到一个它不拥有的流的报文时，它通过 ECMP 将其转发给拥有该流的节点。新流通过一致性哈希分配到各节点。

效果：即使 ECMP 将报文路由到"错误"节点（因为节点集合发生了变化），该节点也能检测到并进行转发，从而保持流的连续性。

其他 CDN 也有类似设计（Google 的 Maglev 是公开的前身）。其模式是 具有分布式流表的 L4 负载均衡器，而非裸 ECMP。

(d) Anycast 孤岛

有时某个 ISP 的 BGP 路径选择会将所有 anycast 流量发送到错误的 PoP 长达数小时。CDN 观察到："该 ASN 的用户延迟很高，推测他们被路由到了 5000 英里之外的 PoP，而非附近的那个。"原因包括：

  • ISP 的上游运营商对 CDN 的 anycast IP 持有错误路径。
  • 某个交换点的 BGP 过滤器配置错误，丢弃了 CDN 的路由公告。
  • BGP 会话抖动导致在两条有效路径之间振荡，且两者均非最优。

检测方式：按 ASN 聚合的 RUM（真实用户监控）数据。缓解措施：AS 路径前置或 BGP 社区操作，使目标 PoP 更具吸引力；手动联系 ISP 的 NOC。无法仅在 CDN 层解决，因为问题出在 ISP 的自主决策中。

坦诚承认：全球约 0.5%~2% 的用户在任意时刻处于非最优 PoP 上。你需要持续度量、告警、路由调优，而不是承诺零误路由。

(e) ECS（EDNS 客户端子网）与基于 DNS 路由的替代方案

对于基于 DNS 的流量导引（较旧的 CDN 架构，或 anycast + DNS 混合方案），CDN 的权威 DNS 需要知道 客户端的 IP 子网，而不是解析器的。没有 ECS，一位使用 8.8.8.8（美国的 Google DNS）的德国用户会被路由到美国的 PoP。

ECS（RFC 7871）允许权威 DNS 看到客户端子网（通过使用 /24 或 /48 而非完整地址来保护隐私）。大多数现代解析器（Google DNS、Cloudflare 1.1.1.1、Quad9）支持 ECS。企业内部解析器通常不支持，导致企业用户被误路由。

这正是现代 CDN 倾向于 anycast 而非 DNS 路由 的原因之一：anycast 不需要 ECS，因为 BGP 路径本身就隐含了用户的位置。

§13. 源站屏蔽层机制详解

§11 的架构图提及了源站屏蔽层。它是在高负载下保护源站最重要的组件之一，值得单独论述。

(a) 屏蔽层的请求合并

屏蔽层的主要职责： 将针对同一 key 的缓存未命中合并为单次源站拉取.

Time t=0:   1000 PoPs around the world have cache miss on /api/trending
            (TTL just expired, popular endpoint).
Time t=1ms: 1000 requests arrive at the shield (mid-tier or per-region).
Time t=2ms: Shield sees "first request for /api/trending, no cache entry,
            need to fetch from origin." Marks the key as "in-flight."
            Issues ONE origin fetch.
            All 999 subsequent requests for the same key check "is there
            an in-flight fetch?" Yes → they attach to the same promise,
            wait for the origin response.
Time t=50ms: Origin responds.
Time t=51ms: Shield writes to its cache, then fans out the response to
             all 1000 waiting PoPs.

Origin saw: 1 request. Without shield: 1000 requests.

实现方式：以缓存 key 为键的哈希表，值为"进行中的 Promise + 等待者列表"。第一个请求创建 Promise；后续请求阻塞等待。当源站响应到达时，所有等待者获得相同的响应。

Nginx 通过 proxy_cache_lock + proxy_cache_lock_timeout实现这一机制。Varnish 内置了基于相同哈希的请求合并。所有自研 CDN 缓存引擎都有等效实现。

(b) 合并范围

层级结构至关重要：- 单节点合并：同一节点内的请求去重。在小规模下有效。- 单 PoP 合并：通常通过分片（每个 URL 映射到每个 PoP 中的一个节点）加上节点级合并来实现。- 跨 PoP 的屏蔽层合并：这才是真正的价值所在。屏蔽层将来自 N 个 PoP 的未命中合并为 1 次。

屏蔽层有时被称为"第 2 层"或"第 3 层"缓存；具体结构取决于 CDN：- Cloudflare：区域层（约 30 个区域）+ 可选的 Argo Smart Routing 进行路径优化。- Fastly："shielding"——将某个特定 PoP 指定为特定服务的上游。- CloudFront："Origin Shield"——按源站启用的显式区域屏蔽层。- Akamai：SureRoute + 分层分发。

(c) 屏蔽层到源站的连接池

源站通常无法承受来自 CDN 的每秒数千次 TLS 握手。屏蔽层为每个源站维护 持久连接池 ：

  • 长连接的 HTTP/2（或 HTTP/3），跨请求复用。
  • TLS 会话恢复，避免完整握手。
  • 每个源站可配置的并发限制。
  • 健康检查，快速检测源站故障。

具体来说：源站看到来自屏蔽层的少量已建立连接（10~100 个），每条连接承载多路复用的请求。从源站的角度来看，整个 CDN 看起来像一个使用持久连接的单一（大型）客户端。

(d) 屏蔽层为新 PoP 预热

§15 的问题 3（冷启动）描述了新 PoP 上线时缓存为空的情况。屏蔽层在这里起到帮助：

1. New PoP P_new opens; BGP routes ~5% of regional traffic to it (staged).
2. Cache misses on P_new flow to the regional shield.
3. Shield has the popular set already cached (it serves all PoPs in the region).
4. P_new gets responses quickly from shield (warm-cache hit at shield).
5. P_new fills its local cache; over hours its hit rate climbs.
6. Origin sees almost no extra load during P_new's warmup, because the
   shield absorbed the misses.

无屏蔽层：P_new 次未命中 → 直接打到源站 → 预热期间源站承受 100 倍正常负载。有屏蔽层：屏蔽层承受约 1 倍正常负载；源站几乎无额外压力。

(e) 屏蔽层故障处理

屏蔽层是层级中的单点。CDN 将屏蔽层以 HA（高可用）集群方式运行（通常 3~5 个节点），并配有健康检查故障转移。当主屏蔽层故障时：

  1. 各 PoP 检测到连接失败（健康检查或 TCP RST）。
  2. 各 PoP 重新路由到备用屏蔽层（集群中的其他节点或其他区域）。
  3. 备用屏蔽层的缓存可能更冷，但仍比源站暖。
  4. 已恢复的屏蔽层重新加入；流量最终恢复均衡。

故障转移期间的 2~5 秒内，源站负载会升高（部分 PoP 可能在备用屏蔽层生效前直接打到源站）。在大多数 CDN 的 SLO 范围内，但在指标中可见。

(f) 何时需要（或不需要）屏蔽层

屏蔽层对大多数企业 CDN 客户默认开启。以下情况不需要：- 源站与屏蔽层在同一数据中心（屏蔽层只是多一跳，没有收益）。- 源站本身是 CDN 或已自行合并请求（例如 S3 已自动去重）。- 工作负载量极低，屏蔽层的带宽消耗不值得。

以下情况始终需要：- 源站是区域性的（单数据中心），而流量是全球性的。- 工作负载包含热门动态内容（未命中集中在某些 key 上）。- TTL 足够短，以至于频繁重新拉取。- 源站在高负载下较为脆弱。

§14. 边缘的 HTTP/3

HTTP/3（RFC 9114）在 UDP 上通过 QUIC（RFC 9000）运行 HTTP。这一过渡正在发生：截至 2026 年，主要 CDN 上约 30% 的 Web 请求通过 HTTP/3，且还在增长。

(a) 为何选择 QUIC（简述）

TCP + TLS 在移动端和高丢包网络上存在根本性缺陷：

  • 最坏情况下需要 3 RTT（往返时间）建立连接 （TCP 1.5 RTT + TLS 1.3 1-RTT，虽然大部分可通过 TFO 和会话恢复绕过，但并不普遍）。
  • TCP 的队头阻塞：一个丢失的报文会阻塞其上所有的流。HTTP/2 将多路复用的流放在单条 TCP 连接上，因此继承了这一问题——单个报文丢失会阻塞所有流。
  • 无法在连接中途迁移：TCP 连接由 (src_ip, src_port, dst_ip, dst_port)四元组标识。任意一个发生变化（WiFi 切换到蜂窝网络），连接就会中断。

QUIC 解决了所有三个问题：

  • 0-RTT 连接建立 供回访用户使用（利用缓存的会话票据）。
  • 逐流丢包恢复：流 5 上的丢包不会阻塞流 1~4。
  • 连接 ID 与 IP/端口无关：连接可在 NAT（网络地址转换）重绑定、WiFi 切换蜂窝网络或企业 VPN 重连时存活。

(b) 回访用户的 0-RTT

持有有效会话票据的回访客户端在 第一个报文中 就将 GET 请求发送到服务器。服务器可以在任何 RTT 发生之前就开始处理请求。

First visit:
  Client SYN+ClientHello -> Server SYN-ACK+ServerHello+EncryptedExtensions -> Client GET
  (~1 RTT before request)

Repeat visit with cached ticket:
  Client SYN+ClientHello+GET (encrypted with cached key) -> Server processes immediately
  (~0 RTT before request)

对于经常访问的站点——银行应用、社交信息流、任何移动端的个人服务——0-RTT 在每次回访时将首字节时间缩短约 100 毫秒。在一次会话中累积下来，用户感知延迟可节省数百毫秒。

注意：0-RTT 数据可能被攻击者重放。服务器只能接受 幂等 请求作为 0-RTT 数据。GET 可以；POST 不行。CDN 通常只对 GET 请求允许 0-RTT。

(c) 连接迁移

用户在家中通过 WiFi 打开一个页面；读了一半后离开家切换到蜂窝网络。使用 TCP 时，所有连接中断，浏览器重新建立。使用 QUIC 时， 连接 ID 标识连接，与 IP/端口无关。客户端从新 IP 发送报文；服务器看到带有已知连接 ID 的新 IP；继续该会话。

对于移动中的用户（地铁、步行），连接迁移消除了一类卡顿。CDN 层面的指标：启用 QUIC 后，移动端 p99 交互时间下降约 50~150 毫秒。

"移动端 p99 缩短了 100 毫秒"是主要 CDN 客户在引入 QUIC 后的典型真实报告。Stripe、Reddit 和 Cloudflare 自身均报告了这一量级的提升。

(d) CDN 采用现状

  • Cloudflare：HTTP/3 自 2019 年 GA；默认为所有客户启用；截至 2025 年，超过 40% 的 Cloudflare 流量通过 HTTP/3。
  • Fastly：HTTP/3 GA；通过功能标志为大多数客户启用。
  • CloudFront：HTTP/3 自 2022 年 GA；按分发启用（可选）。
  • Akamai：HTTP/3 GA；因产品而异。
  • Google Cloud CDN：HTTP/3 GA。

部分网络（主要是企业/学校 WiFi）阻止了 443 端口的 UDP 出站流量，导致 HTTP/3 无法使用。客户端自动回退到 TCP 上的 HTTP/2。CDN 同时提供两者；客户端自行选择。

(e) 运维影响

  • UDP 路径与 TCP 路径不同。 边缘负载均衡器（XDP/eBPF）需要处理 UDP+QUIC；存在新的代码路径。
  • 有状态 UDP：连接 ID 由服务器分配；负载均衡器需要追踪连接 ID 到服务器的映射（否则来自迁移客户端的报文会落到错误的服务器上）。通过将服务器标识编码在连接 ID 本身中来解决。
  • DDoS 攻击面有所不同。 UDP 放大攻击是真实存在的隐患；QUIC 有专门的防放大机制（服务器在验证客户端之前只能发送其所接收字节数的 3 倍）。边缘过滤器需针对 QUIC 流量模式进行调优。
  • 可观测性发生变化。 诸如 tcpdump 之类的工具能处理 UDP 报文，但需要支持 QUIC 的解码器来检查协议细节。Wireshark 对此支持良好。

§15. 该技术固有的难题

问题 1：大规模缓存清除传播

朴素的想法：当源站更新时，调用清除 API，向所有 PoP 广播"丢弃 key X"。

如何出错：300 个 PoP × 100 个节点 = 每次清除需通知 30,000 个节点。每秒 1000 次清除激增 → 控制平面每秒 3000 万次 RPC（远程过程调用）。分区导致某个 PoP 与控制平面隔离——永久提供过期内容。按标签清除（"类目 X 的所有 50,000 个产品页面"）需要每个 PoP 维护倒排索引。故障在不同内容类型间表现相似： 电商静态资源 错过促销开始的清除，显示错误价格长达数小时； API 响应缓存 错过功能标志翻转，继续返回旧的响应结构，导致客户端解析崩溃； 视频片段缓存 继续提供已被责令下架的片段——面临法律风险。

真正的解决方案:

  • 通过发布订阅实现分层扇出。 源站 → 控制平面 → 区域 → PoP → 节点。Cloudflare 的"Quicksilver" KV 通过 gossip/扇出在约 150 毫秒 p99 内将配置和清除元数据分发到全球。
  • 代理 key / 标签。 "Cache-Tag"（Cloudflare）或"Surrogate-Key"（Fastly）：源站发出 Cache-Tag: post-123 category-electronics；CDN 建立标签→key 集合的索引；按标签清除时遍历该索引。
  • 软清除作为默认策略。 标记为过期；提供 stale-while-revalidate；惰性收敛。仅对法律下架和安全漏洞执行硬清除。
  • 版本化 URL 才是真正的解决方案。 app.abc123.js。新版本 = 新 URL = 无需清除。这是解决 >90% 失效问题的正确答案。

问题 2：缓存雪崩 / 惊群效应

朴素的想法：TTL 到期；下一个请求未命中；PoP 去拉取源站。

如何出错: /api/trending 在所有 PoP 上以 TTL=60s 缓存。第 60 秒时，所有 PoP 同时过期。300 个 PoP × 100 个节点 = 同一毫秒内 30,000 次同时打到源站的请求——源站崩溃。同样会影响热门图片、热门 API 端点、热门视频片段、频繁拉取的 ML 模型。

真正的解决方案:

  • 每一层的请求合并。 若某个 key"正在拉取中"，进来的请求附加到进行中的 Promise 上。Nginx 有 proxy_cache_lock；Varnish 内置此功能；Fastly 的 VCL 原生支持。
  • 以屏蔽层作为合并点。 每个 PoP 将请求合并为一次；屏蔽层将 N_pops 个请求合并为一次。源站每个区域只看到一次拉取。
  • stale-while-revalidate。 Cache-Control: max-age=60, stale-while-revalidate=3600。第一次未命中触发单次异步重新验证。
  • 抖动 TTL。 随机 ±5% 使过期时间错开。

效果：即使在 1000 万 QPS 下，源站对热门 key 也只看到 个位数的 QPS ，因为所有请求都通过层级结构合并了。

问题 3：新 PoP 冷启动

朴素的想法：上线新 PoP；让其自然预热。

如何出错：圣保罗新 PoP 上线；BGP 将巴西流量路由到该 PoP；在最初一小时内，缓存命中率约为 0%。本地源站屏蔽层尚不存在，大多数未命中直接打到源站——预热期间流量激增 100 倍。无论工作负载是图片、视频、API 还是下载，情况都相同。

真正的解决方案:

  • 分阶段 BGP 公告 ——初始时使用更长的 AS 路径，使该 PoP 只承接约 5% 的流量，再逐步提升。
  • 缓存预热 ——在开放前，从生产日志中回放该区域的前 1000 条请求；可使启动时命中率达到约 50%。
  • 分层缓存使冷 PoP 影响可控 ——L1 缓存冷但命中温热的 L2 中间层即可。
  • 从邻居 PoP 窃取 ——部分 CDN 通过 gossip 从邻近 PoP 复制 key。

问题 4：地理正确性（anycast 病理）

朴素的想法：anycast 路由到最近的 PoP。信任 BGP。

如何出错："最近"是 AS 路径意义上的最近，而非物理意义上的最近。 Anycast 孤岛：某个 ISP 中配置错误的路由器将所有 anycast 流量发送到错误的 PoP 长达数小时，而 BGP 不会察觉。当解析器不传递 ECS 时（较旧的企业解析器、部分 VPN），ECS 失效。印度 ISP 的流量因其上游存在 BGP 问题而被路由到新加坡整整一个上午——RTT 从 15 毫秒变为 250 毫秒；转化率下降。

真正的解决方案:

  • RUM 数据驱动路由决策。 按 ASN 聚合的延迟数据；CDN 调整 BGP 公告（AS 路径前置）或 DNS 响应，将流量引导正确。
  • 多 PoP 健康检查 + BGP 路由反射器调优 以在每个 ASN 上优选特定对等体。
  • GeoDNS 作为备用流量导引层 ——为不同区域使用不同的 anycast IP。
  • 坦诚承认：在大规模场景下，约有 0.5-2% 的用户处于次优 PoP 上。请做好监控和告警，不要承诺零误差。

问题 5：DDoS 吸收

朴素的想法：检测攻击，在防火墙层进行限速，封堵源 IP。

如何出错：5 Tbps 的 L3 SYN 洪泛在数秒内即可打满任何单一源站带宽——你无法拦截你根本收不到的流量。L7（第 7 层）洪泛看起来与正常流量无异；按 IP 限速会被拥有百万 IP 的僵尸网络轻松绕过。

真正的解决方案:

  • Anycast 本身就是第一道吸收层。 来自全球僵尸网络的 5 Tbps 流量分散至约 300 个 PoP，每个 PoP 约承受 17 Gbps——在容量范围之内。 被动地 稀释。
  • L3/L4 边缘清洗。 XDP/eBPF 程序在任何 TCP socket 建立之前即以线速丢弃垃圾流量（畸形 SYN、ICMP 洪泛、无关 UDP）。
  • L7 的挑战-响应机制。 JS 挑战（"I'm Under Attack Mode"）、TLS 指纹识别、行为分析。
  • WAF 规则 + 基于机器学习的异常检测 （Cloudflare Bot Management、Akamai Bot Manager Premier）。
  • 源站 IP 保密。 真实源站 IP 从不对外公开；所有流量均通过 CDN 接入。

DDoS 的经济逻辑：攻击者必须在带宽上超越防御方。280 Tbps 的攻击需要一个 280 Tbps 的僵尸网络——理论上可行，实践中从未出现过。

问题 6：缓存键敏感性与缓存投毒

朴素的想法：将所有请求头纳入缓存键，以避免 Vary 配置错误。

如何出错：攻击者发送 Host: evil.com 带有 URL 的 /app.js。CDN 在攻击者的 host 下缓存了 app.js 。下一次合法请求：若哈希错误，则返回攻击者构造的变体内容。 Web 缓存投毒 ——PortSwigger 有整套资料库。包含任意请求头会导致缓存碎片化，进而引发命中率崩溃。与内容类型无关——任何经由 CDN 前置的端点都存在此漏洞。

真正的解决方案:

  • 在哈希前对缓存键进行规范化 ：去除不在白名单中的请求头，将 host 转为小写，规范化查询参数。
  • Vary 请求头白名单 ——仅限 Vary: Accept-Encoding 及已知安全的变体。 Vary: User-Agent 规范化为 UA 类别。
  • Web 缓存投毒审计 （PortSwigger Cache Deception 扫描器）。
  • 除非已明确列入白名单且承载实质性内容，否则绝不在缓存键中信任用户可控的请求头 。

§16. Web 缓存投毒攻击深度解析

§15 中的问题 6 已从较高层面提及 Web 缓存投毒。这类攻击手法足够有趣、也足够危险，值得深入探讨。PortSwigger 研究员 James Kettle 自 2018 年起通过一系列论文使该类攻击广为人知；此后已产生多个重大 CVE（Common Vulnerabilities and Exposures）。

(a) 通用攻击模式

攻击者精心构造一个请求，使其响应可被控制——通常是利用某个请求头：源站将其反射到响应中，但 CDN 并未将其纳入缓存键。被投毒的响应被缓存后，将被返回给后续的合法用户。

具体而言：

1. Attacker observes that `/welcome` reflects an X-Forwarded-Host header
   into the rendered HTML (e.g., the origin generates absolute URLs
   like https://X-Forwarded-Host-value/...).
2. Attacker sends:
     GET /welcome HTTP/1.1
     Host: example.com
     X-Forwarded-Host: evil.com
3. Origin renders the page with evil.com baked into all link URLs and
   asset paths.
4. CDN caches the response under the URL key. X-Forwarded-Host is NOT
   in the cache key (the CDN is unaware it's content-bearing).
5. Next legitimate user requests /welcome.
6. CDN serves the poisoned cached response.
7. User's browser loads scripts from evil.com.
8. Attacker has script execution in legitimate user's context.

通用公式： 非键入输入 → 键入输出。任何影响响应但未包含在缓存键中的输入均为投毒向量。

(b) 常见变体

  • X-Forwarded-Host 被反射到绝对 URL 中。 源站使用 X-Forwarded-Host 来构造 HTML 中的规范 URL 或重定向目标。攻击者将其设置为恶意域名。
  • X-Forwarded-Scheme 被反射到绝对 URL 中。 将 HTTPS 切换为 HTTP，实施降级攻击。
  • X-Rewrite-URL 或 X-Original-URL。 某些框架解析这些头来确定"真实" URL。攻击者将其重写为不同路径；响应被缓存在原始 URL 下，但反映的是重写后路径的内容。
  • Referer 头被反射至 CSRF token 或规范 URL 中。 概率较小，但已有案可查。
  • Cookie 被反射但未纳入缓存键。 若 Cookie 被包含在响应体中（例如用于显示用户名），但未纳入缓存键，则某个用户的响应体会被缓存并返回给所有人。
  • 多层之间缓存键规范化不一致。 若 CDN 丢弃某个查询参数，而源站使用该参数，则源站响应（因参数而存在差异）会被缓存在单一键下（不含该参数）。不同参数值会产生不同响应，但只有其中一个会被缓存并返回给所有人。

(c) "X-Forwarded-Host 未纳入缓存键" CVE 模式

标准 CDN 配置将 Host 纳入缓存键，但未纳入 X-Forwarded-Host。当源站使用 X-Forwarded-Host 构造规范 URL 时，攻击者便利用这一不一致性发起攻击。

解决方案： 在转发至源站前剥离不在白名单中的请求头 。如果源站不应依据 X-Forwarded-Host 做出决策（且不应信任来自客户端的该头），则根本不应将其透传。

该 CVE 模式已影响：\n- 多个生产站点的搜索结果页与列表页。\n- 从请求头中计算 API base URL 的单页应用路由。\n- 根据 host 渲染绝对链接的文档站点。

(d) 防御措施

  • CDN 侧的请求头白名单。 仅透传源站所需的请求头；其余一律剥除。缓存键仅包含已声明的承载内容的请求头。
  • 剥除后对缓存键进行规范化。 缓存键由 URL 加白名单请求头组成，经过规范化处理。
  • 源站不信任请求头。 源站绝不应使用 X-Forwarded-Host 或类似攻击者可控的输入来构造规范 URL。若源站需要知道自身的公开域名，CDN 应以服务端托管的请求头形式注入（由 CDN 配置设置，而非直接透传）。
  • 当源站响应因请求头而变化时，应正确设置 Vary。 若源站响应确实依赖某个请求头，应设置 Vary: Header-Name 并在缓存键中显式包含该头。
  • 针对已知投毒模式的 WAF 规则。 封堵携带恶意域名的 X-Forwarded-Host、格式不符预期的请求头等。
  • 缓存投毒型 DoS 变体。 攻击者发送异常的 User-Agent 值，强制缓存碎片化并耗尽缓存空间。防御策略：在键入前对 UA 进行分组；限制每个 URL 的不同变体数量上限。
  • 缓存欺骗（反向攻击）。 攻击者诱使 CDN 缓存私有响应（例如用户的个人资料页），方式是请求 /profile/.css ——源站返回用户的个人资料 HTML，但 CDN 识别为 .css 并将其作为公共内容缓存。后续对 /profile/.css 的请求将获得缓存中的私有数据。防御策略：严格的 URL 路径校验；绝不缓存内容类型与 URL 扩展名不匹配的响应。

(e) 检测

用于检测投毒的合成监控：\n- 定期发起正常请求，捕获响应。\n- 发起携带恶意头（X-Forwarded-Host: synthetic-test-only.com）的请求，捕获响应。\n- 再次发起正常请求，与第一次进行对比。\n- 若第二次正常请求返回了受恶意头影响的响应，则投毒向量存在，触发告警。

大多数具备 CDN 感知能力的安全工具（PortSwigger 的 Param Miner、Burp 的缓存欺骗扫描器、OWASP 测试指南）均可自动化完成上述流程。

§17. 边缘安全

除缓存投毒之外，边缘节点是互联网对外安全栈中大多数组件的运行场所。CDN 已演进为将 DDoS 缓解、WAF、机器人管理以及 TLS/证书运营作为一等功能加以整合。

(a) DDoS L3/L4 — 基于 anycast 的大流量吸收

§2 和 §15 均已提及：anycast 是第一道吸收层。来自全球僵尸网络的 5 Tbps 流量分散至 300 个 PoP，每个 PoP 约承受 17 Gbps——在任何现代 PoP 带宽容量范围之内。CDN 无需过滤大部分攻击流量，仅凭带宽储备即可超越攻击者。

被动稀释之外：\n- 各 PoP 以线速运行 XDP/eBPF 程序 ，丢弃垃圾数据包——畸形 SYN、ICMP 洪泛、无关 UDP、SYN 洪泛。eBPF 在内核中 socket 创建之前运行；每台服务器丢包速率约 100 Mpps（每秒百万数据包）。\n- SYN cookies ，用于在不为半开连接分配状态的前提下应对 SYN 洪泛。\n- 按源 IP 限速，可选按 ASN 粒度，在内核 L4 层应用。\n- 基于地理位置/ASN 的过滤：若攻击流量几乎全部来自某个 ASN，则在边缘丢弃该 ASN 的流量。

Cloudflare 于 2023 年（已公开披露）成功吸收了一次 7100 万 rps 的 DDoS 攻击；该攻击几乎完全在 L4 层被消化，未对客户流量造成干扰。

(b) DDoS L7 — 应用层洪泛

L7 攻击看起来与正常 HTTP 请求别无二致。按 IP 限速会被拥有数百万 IP 的僵尸网络轻松绕过。防御体系如下：

  • WAF 规则 针对已知攻击特征（OWASP Top 10——SQL 注入、XSS 等）。
  • 限速 ：按路径、按来源、按指纹、按组合条件进行限制。
  • 挑战页面：JS 挑战（"证明你是浏览器才能继续访问"）、CAPTCHA、Cloudflare Turnstile、hCaptcha。以极低的用户摩擦成本将机器人流量减少约 99%。
  • 行为分析：按来源追踪请求模式——访问站点、时间分布、请求头分布。异常情况（某个 IP 仅请求 /login 1000 次）触发封堵。
  • 基于机器学习的检测：Cloudflare Bot Management、Akamai Bot Manager Premier、Fastly Next-Gen WAF。基于 CDN 全球流量训练，能够识别新兴机器人模式。
  • "I'm Under Attack" 模式：对所有请求发起强力挑战。在主动攻击期间以牺牲部分用户体验为代价，实现全面拒绝机器人访问。

(c) 机器人管理

机器人管理已发展成为一个重要类别。主要技术手段如下：

  • TLS 指纹识别（JA3 / JA4 / JA4+）：对 TLS ClientHello 进行哈希（密码套件、扩展、ALPN 值）。不同客户端（Chrome、Firefox、curl、Python requests、Go net/http）产生不同指纹。JA3 是最初的方案；JA4（Foxio 于 2023 年发布的更新版）对小版本库变更的鲁棒性更强。用于识别伪装成浏览器的自动化工具。
  • HTTP/2 指纹识别：类似于 TLS 指纹，但基于 HTTP/2 SETTINGS 帧与伪头部顺序，可揭示伪装行为。
  • 行为分析：鼠标移动轨迹、滚动模式、点击时序。机器人滚动有规律；人类则不同。Cloudflare Bot Management 使用页面内嵌入式信标。
  • 信誉数据库：已知机器人 IP 列表（数据中心 IP 段、Tor 出口节点、住宅代理服务）。
  • 挑战级联：对可能是合法用户的客户端使用简单挑战（JS 计算）；对可疑客户端使用更难的挑战（CAPTCHA）。

军备竞赛：机器人编写者使用真实浏览器（通过 Playwright、Puppeteer）、住宅代理（轮换真实消费者 IP）以及人工 CAPTCHA 解题服务。检测手段持续向更深层行为特征演进。

(d) WAF（Web 应用防火墙）规则

WAF 检查 HTTP 请求，拦截符合攻击特征的请求。OWASP Top 10 缓解措施：

  • A01：访问控制失效 ——WAF 可执行 URL 访问规则。
  • A02：加密失败 ——在边缘强制执行 TLS。
  • A03：注入 ——SQL 注入模式（UNION SELECT, ' OR '1'='1）、命令注入、OS 注入。WAF 规则匹配路径、查询参数、请求体、请求头中的模式。
  • A04：不安全的设计 ——WAF 可强制执行特定的 schema（例如，参数必须符合预期的正则表达式）。
  • A05：安全配置错误 ——超出 WAF 范围，但相关（例如，移除 Server 响应头）。
  • A06：自身存在漏洞的组件 ——虚拟补丁：在源站未完成修补之前，WAF 规则封堵特定已知 CVE 的利用特征。
  • A07：身份验证失败 ——对登录接口进行限速。
  • A08：软件和数据完整性失败 ——内容类型与签名校验。
  • A09：安全日志与监控缺失 ——WAF 日志接入 SIEM（安全信息与事件管理）。
  • A10：SSRF（服务端请求伪造） ——封堵在路径或请求体中包含内网 IP 段的请求。

Cloudflare 的"Managed Rules"（前身为 OWASP Core Rule Set 集成）与 Fastly Next-Gen WAF（前身为 Signal Sciences）是市场主流的托管 WAF 产品，均持续更新规则包。

WAF 规则存在误报问题。调优工作包括：偏执级别设置（低/中/高误报率）、按路由豁免、学习模式（初期仅记录不拦截）、渐进式执行上线。

(e) TLS 终止与证书管理

CDN 几乎为所有客户域名在边缘执行 TLS 终止。这意味着：

  • CDN 持有客户域名的 私钥 。信任至关重要；CDN 对密钥存储采取严格的运营安全措施（HSM——硬件安全模块、密钥封装、审计日志）。
  • 证书自动化配置 。客户将域名指向 CDN 的权威 DNS；CDN 自动签发 Let's Encrypt 或 DigiCert 证书；ACME（自动化证书管理环境，RFC 8555）负责验证与轮换。
  • SNI（服务器名称指示） 使 CDN 能够在单一 anycast IP 上托管数千个客户域名。TLS 握手中的 SNI 扩展告知 CDN 应出示哪张证书。
  • TLS 1.3 现已作为默认版本；TLS 1.2 为兼容性保留支持；TLS 1.0/1.1 已废弃。
  • OCSP Stapling （在线证书状态协议）默认开启——CDN 从 CA 获取 OCSP 响应，并在 TLS 握手中随证书一并下发。省去客户端单独发起 OCSP 请求的往返时间。
  • 证书透明度（Certificate Transparency） 日志是强制要求；所有签发的证书均对外公开。

对于不希望将私钥交给 CDN 的客户，存在以下可选方案：\n- Cloudflare Keyless SSL：CDN 执行 TLS 握手的大部分步骤，但 RSA 私钥运算在客户自有的密钥服务器上完成。握手增加约 30-50 ms 延迟，但 CDN 始终不接触私钥。\n- Encrypted Client Hello (ECH)：新兴标准，用于加密 SNI 本身，隐藏客户端所请求的域名。

证书轮换：Let's Encrypt 通常每 90 天自动续期，CDN 透明地完成续签。

§18. 故障模式演练

单个缓存节点在拉取过程中崩溃。 节点 B17 在拉取过程中发生内核崩溃。其他节点不受影响；一致性哈希重新映射 B17 的虚拟节点；各邻近节点约增加 ~1/(N-1) 的额外负载。健康检查在亚秒级内将 B17 从 L4 LB 环中移除。正在进行的源站请求被丢弃；客户端超时重试；重试命中不同节点 → 缓存 MISS → 重新拉取。B17 重启后从磁盘分片重建索引（约每 TB 需 1 秒），随后重新加入集群。 持久性要点：磁盘分片在崩溃中幸存；内存索引可从分片元数据中重建。

Shield 在热门资源冷拉取期间崩溃。 Shield 以小规模高可用（HA）集群方式运行（每个 Region 3-5 个节点）。L2 检测到连接失败后，重试至兄弟 Shield 节点，由其重新向源站拉取。新节点内部再次完成请求合并去重。中断约 2 秒；若重试失败，部分 L2 请求将看到延迟升高或 503。 持久性要点：源站是持久性层；Shield 只是缓存。

控制面宕机（清除操作传播中断）。 在一次大规模清除任务期间，Quicksilver 等效组件失去主节点地位。清除队列在前任主节点本地积压；Raft 选举出新主节点；待处理的清除操作从持久日志中重放。中断约 5-30 秒；部分 PoP 提供的内容比 SLO 允许的时间更长。 数据面对控制面故障无感知 ——请求路径的服务 SLO 不受影响。这一分离机制至关重要。 持久性要点：Raft 复制的清除日志。

网络分区将某个 Region 割裂。 海底电缆断裂导致某个大陆与控制面及源站失去联系。各 PoP 继续提供已缓存的内容。对源站的缓存未命中请求要么开放失败（stale-if-error），要么关闭失败（503）。清除操作在本地排队，待恢复连接后再下发。分区期间有界陈旧 SLO 被违反。 持久性要点：TTL——一旦分区恢复，内容最终会过期并重新拉取。

PoP 永久丢失。 地震摧毁某个 PoP。BGP 撤销该路由；流量切至下一优先 PoP；受影响用户延迟增加约 20-50 ms，但服务不中断。新 PoP 在数月内完成预置，并通过问题 3 中的分阶段上线流程投入使用。 持久性要点：PoP 相对于源站真实数据是无状态的；缓存状态可重建。

BGP 路由错误（2008 年 YouTube-巴基斯坦事件类型）。 某个错误配置的 ISP 发布 BGP 路由，声称持有 CDN 的 anycast IP；错误路由扩散；流量进入黑洞或被路由至误公告方。RPKI（资源公钥基础设施）在配置良好的对等节点处拒绝未授权的路由公告（部署日益普及）。NOC（网络运营中心）致电误公告方的上游运营商要求撤销——过程痛苦、需人工介入、耗时数小时。 持久性要点：缓存架构在此无能为力；缓解手段在路由层。

缓存局部损坏。 劣质 SSD 固件导致静默数据损坏。 对每个缓存对象进行校验和验证：返回前先校验；若校验失败则视为缓存未命中，重新拉取并驱逐问题段。对每台机器的不匹配率进行监控，超阈值则下线该节点。缓存具有 自愈能力. 持久性要点：源站是数据的权威来源。

§19. 基线方案未覆盖的故障模式

§18 中的演练涵盖了典型故障。以下是一些不在其中但足够常见、值得了解的故障：

(a) 部署时因缓存踩踏引发源站 DDoS

场景：站点部署新版本，部署操作清除了所有已缓存页面。几秒钟内，全球所有 PoP 节点对首页和其他热门页面均无缓存。流量保持正常水平——但全部命中缓存未命中。源站承受约正常请求量 10,000 倍的压力。

真实案例：某热门新闻网站在美东时间早上 9 点（正值早高峰流量）完成部署，按标签清除了首页和前 50 篇文章页面的缓存。5 秒内有 100 万个请求涌向源站（而正常稳定状态仅为 100 个）。源站数据库崩溃，网站停机 20 分钟。

缓解措施：\n- 错开清除时机。 不要一次性清除所有缓存，分批执行，在 60 秒以上的时间内逐步清除，让缓存慢慢回填。\n- 使用版本化 URL，而非主动清除缓存。 新部署生成新的资源 URL；旧的缓存响应在引用它们的 HTML 更新之前仍会继续提供服务。\n- 缓存预热。清除前先拉取新响应来预热缓存，再清除旧缓存（或直接等旧 TTL 到期）。源站感知到来自单一来源的预热流量，而非全球范围的踩踏。\n- stale-while-revalidate。清除后先提供过期内容，在后台异步刷新。源站看到的是合并后的重新验证请求，而非原始的缓存未命中请求。

(b) CDN 提供商宕机

CDN 提供商本身就是单点故障。典型事故案例：\n- Cloudflare 2022 年 6 月 21 日：BGP 配置变更错误导致 19 个 PoP 节点中断约 30 分钟，Discord、Shopify 及众多其他服务受到影响。\n- Cloudflare 2019 年 7 月 2 日：WAF 规则出错（正则表达式出现指数级回溯），导致全球所有 Cloudflare 服务器 CPU 占用率达 100%，持续约 30 分钟，半个互联网上通过 Cloudflare 提供服务的站点全部宕机。\n- Fastly 2021 年 6 月 8 日：客户推送的 VCL 配置触发了一个潜伏的服务 bug，导致全球数据平面中断约 49 分钟，BBC、Reddit、NYT、Amazon、gov.uk 等全部下线。\n- AWS CloudFront 偶发区域性中断：单次事故影响范围较小，但频率高于 Cloudflare/Fastly 的全球性中断。\n- Akamai 2021 年 7 月 22 日：DNS 服务中断，导致众多企业客户服务停用约 1 小时。

缓解方案： 多 CDN （详见 §20）。代价：复杂性提升以及两份出口流量合同。收益：49 分钟的供应商宕机不会拖垮你的服务。

(c) DNS 解析器问题

CDN 高度依赖 DNS：\n- CDN 侧权威 DNS：当 Cloudflare 的权威 DNS 故障时，所有使用 Cloudflare 前置的站点 DNS 解析均会失败。Cloudflare 在 2019 年 10 月经历过此类事故（Quad9 级联问题）。\n- ISP 处的解析器缓存投毒：行为异常的 ISP 解析器缓存了错误的答案，导致用户被路由到错误的 PoP 或非 PoP 节点，直到缓存 TTL 到期（可长达数小时）。\n- DNS over HTTPS / DNS over TLS（DoH / DoT）改变了可见性：当客户端使用 DoH 时，CDN 的 ECS 可见性下降，路由精度随之降低。\n- 长 DNS TTL 在故障期间变成负担：CNAME 记录的 24 小时 TTL 意味着故障转移需要 24 小时才能完全生效。CDN 客户的 DNS 记录通常配置非常短的 TTL（约 60 秒）。

缓解方案：为客户 DNS 记录设置短 TTL；将权威 DNS 可用性作为独立 SLO 进行监控，与 CDN 数据平面分开；制定 DNS 故障转移预案（Route 53 健康检查、NS1 流量管理等）。

(d) IPv6 边缘情况

IPv6 的部署已足够普及，不可忽视（约占美国流量的 40%、印度流量的 50%，各地区有所不同），但分布并不均匀：\n- 部分 PoP 节点存在与 IPv4 无关的 IPv6 传输问题。\n- 仅支持 IPv6 的客户源站有时无法从仅支持 IPv4 的较老 PoP 节点访问。\n- ECS 在 IPv6 上的支持有时不如 IPv4（解析器在处理 AAAA 查询时会剥离 ECS）。\n- Anycast 路径在 v4 和 v6 之间可能存在差异——同一客户端在 v6 和 v4 上可能走不同的路径。

缓解措施：全面支持双栈；按 AFI（地址族标识符）进行监控；若可靠性比轻微的延迟优化更重要，部分 CDN 客户会明确禁用 IPv6。

(e) 长尾内容压垮小型 PoP

新 PoP 节点上线后，内容抓取器/归档机器人立即开始遍历整个站点，访问大量冷门的长尾 URL。该 PoP 的小容量缓存被一次性访问的 URL 填满，热门内容被驱逐，命中率骤降至 30%，源站负载飙升。

缓解措施：\n- W-TinyLFU 准入过滤器（§4(c)）——这正是它所防御的场景。\n- 在缓存查找之前，在边缘对每个源 IP/ASN 进行速率限制。\n- 识别并绕过已知爬虫（若它们不代表正常用户流量）。

(f) 缓存层级倒置

有时 L2/Shield 层会成为瓶颈，因为所有 L1 缓存未命中都汇聚于此。若 L1 命中率下降（缓存键碎片化、部署清除等），L2 会承受一波突发流量。L2 开始丢弃请求，L1 开始直接代理到 Shield 层，Shield 层随之崩溃，源站开始承受全部负载。

缓解措施：按层级独立进行容量规划；在各层之间设置熔断器；对各层命中率进行可观测性建设，以便在降级初期及时发现问题。

(g) Slow-loris 与慢读攻击

攻击者建立大量连接，以极慢的速度发送字节以维持连接存活，从而耗尽 CDN 连接池。现代 CDN 边缘的防御手段包括：按源地址限制连接速率、空闲连接超时、HTTP/2 / HTTP/3（多路复用流比独占 TCP 连接更难被此类攻击利用）。

§20. 多 CDN 策略

单一 CDN 宕机的情况确实存在（§19）。对于高可用站点，答案是 多 CDN：与两家或以上 CDN 提供商签订合同，通过 DNS 或某种路由层在它们之间切换流量。

(a) 多 CDN 何时值得引入其复杂性

代价：双份出口流量合同（因量级梯度通常低于 2 倍，但高于 1 倍）、工程复杂度提升、需要在多个提供商上执行替代键清除操作，以及配置漂移风险。

收益：抵御提供商宕机风险、拥有议价能力（CDN 提供商知道你可以切换到竞争对手）、区域性能优化（某些 CDN 在特定地区表现更好），以及合规需求（中国需要使用国内 CDN；俄罗斯、伊朗等地区有时也有类似要求）。

值得引入的条件：\n- 站点可用性 SLO 要求很高（四个九或更高），并且你已测量到 CDN 宕机可能违反该 SLO。\n- 出口流量足够大，多供应商竞价能使每 GB 单价降低的幅度超过复杂性增加的成本。\n- 你有全球业务布局，且存在区域性能差异。\n- 你需要供应商独立性（合规要求或业务风险控制）。

不值得引入的条件：\n- 站点对可用性要求不高（小型 SaaS、内部应用）。\n- 流量较小（单供应商最低消费门槛使多供应商方案经济上不划算）。\n- 工程团队没有能力维护两套配置。

(b) 基于 DNS 的流量调度——NS1、Cedexis、AWS Route 53

多 CDN 的经典前端方案：一个"流量管理"DNS 服务按查询决定将用户路由到哪家 CDN。用户解析 cdn.example.com；流量管理器根据以下条件返回 CDN-A 或 CDN-B 的 CNAME：

  • 各 CDN 健康状态（主动健康检查；将流量从不健康的 CDN 切走）。
  • 各区域延迟（RUM 数据；将流量路由到该地区 p50 延迟更低的 CDN）。
  • 成本权重（将 80% 流量路由到较便宜的 CDN，20% 路由到较贵的以保持其预热状态）。
  • 客户层级（付费用户使用更好的 CDN）。

供应商：\n- NS1 （现属 IBM）：丰富的流量管理能力，支持按查询决策。\n- Cedexis （Citrix Intelligent Traffic Management）：基于 RUM 驱动，拥有大规模数据集。\n- AWS Route 53：简单的加权/基于延迟/健康检查路由。\n- Cloudflare Load Balancing：Cloudflare 自有的多源负载均衡，可对非 Cloudflare 后端进行负载均衡。

基于 DNS 的流量调度存在局限：DNS TTL 是故障转移速度的上限。60 秒 TTL 意味着在管理器检测到故障后，仍可能有长达 60 秒的用户被发往故障 CDN。部分解析器（尤其是较老的企业内网解析器）的缓存时间可能超过 TTL。

(c) HTTP 层面的流量调度——源站侧回退

第二种模式：用户始终优先访问 CDN-A（DNS 指向该 CDN）。若 CDN-A 宕机，客户端/浏览器通过某种信号机制回退到 CDN-B。实现方式：

  • <picture> / <img srcset> 回退 在 HTML 层面：包含多个 URL，浏览器按顺序尝试。适用于图片，但不适用于 HTML 页面本身。
  • Service Worker 拦截 fetch 失败并在备用域名上重试。适用于所有资源，但需要 Service Worker 已安装（冷访问用户没有 Service Worker）。
  • 客户端智能路由库 进行探测并选择最优节点，部分视频播放器采用此方案。

这些方案适用于站点的部分内容，但无法覆盖整个站点（初始 HTML 页面必须有确定的来源）。

(d) "若 CDN A 故障则路由至 CDN B"的自动故障转移

最常见的生产环境方案结合了 DNS 与健康检查：

1. Traffic manager (NS1, Cedexis) healthchecks CDN-A and CDN-B
   from multiple vantage points worldwide.
2. Normal: 80% to CDN-A, 20% to CDN-B (or 100% to CDN-A, 0% to B
   with B kept warm by synthetic load).
3. CDN-A fails healthcheck globally.
4. Traffic manager flips: 100% to CDN-B.
5. DNS TTL controls how fast clients see the change (60 s typical).
6. CDN-B serves; possibly degraded (its cache wasn't warm for all
   of CDN-A's URLs, so origin sees increased load).
7. CDN-A recovers; traffic manager flips back gradually.

(e) 运营注意事项

  • 配置漂移。 两家 CDN 需要保持相同的 WAF 规则、缓存设置、替代键和重定向规则。使用基础设施即代码，从单一来源统一部署到两家 CDN。
  • 清除扇出。 替代键清除操作必须同时命中两家 CDN，通常需要定制工具。
  • 可观测性。 需要跨两家 CDN 的统一流量视图；仅依赖各 CDN 自身的仪表板会存在盲区。
  • 成本报告。 两份合同，需每月对两家 CDN 的用量进行核对。

(f) 避免供应商锁定

即使不采用主动多 CDN，保有一个经过验证的备用 CDN 也能带来以下好处：\n- 在合同谈判中拥有筹码。\n- 若主 CDN 价格涨至无法承受时，有经过验证的替代方案可用。\n- 主 CDN 停止运营时的延续方案（虽属罕见，但确实发生过——一些小型 CDN 已关闭）。

许多企业采用"温备"方案：在 CDN-B 上保持完整配置，约 1% 的流量持续发往 CDN-B 以保持预热，可在数分钟内切换至 100%。

§21. 为何不直接在源站前部署 nginx

这是最常见的质疑。答案是分层次的。

第一步："直接在源站前放一个 nginx，在那里做缓存。"

为何行不通：

  1. 无 anycast，无全球部署节点。 在 us-east-1 部署一台 nginx，到东京仍需 150 ms。缓存而没有地理就近只解决了问题的一半。
  2. 通过增加区域 nginx 进行扩展：用户如何访问正确的节点？基于 DNS 的流量调度需要 GeoIP 基础设施（你已经开始自建 CDN）；anycast IP 需要 BGP 路由广播（明显在自建 CDN）。
  3. 无法抵御 DDoS。 单台 nginx 最多承受 50-100 Gbps 攻击流量；现代攻击规模超过 1 Tbps。
  4. 各区域缓存状态相互独立。 五台 nginx 实例各自独立预热；源站面临 5 倍的冷缓存拉取；没有中间层 Shield，每次缓存未命中都会直接打到源站——雪崩效应无法控制。
  5. 无边缘 TLS 终止优化。 悉尼用户仅 TLS 握手就需消耗约 300 ms。
  6. 无全局流量调度能力。 无法在区域间转移负载；CDN 提供商拥有全球 SDN（软件定义网络）层来实现这一点。
  7. 运维原语不存在。 集中式清除、替代键标记、跨 PoP 实时分析、RUM 驱动的路由——这些全部需要自行构建。

第二步——具体故障场景：一个拥有 1 亿用户的站点，采用多区域 nginx，某名人在 UTC 晚 8 点在推文中附上了站点链接。500 万并发用户通过 DNS 调度分布到 5 台 nginx 实例上，首页已缓存（命中率 100%）。但该链接携带唯一追踪参数，导致 500 万次缓存未命中，进而触发 500 万次源站请求。源站共 100 台服务器，每台需承受 50,000 rps——5 秒内崩溃。而花钱购买了 CDN 的竞争对手拥有 300 个 PoP 节点来承载同样的流量：每个 PoP 约 17k rps，每台机器 340 rps，经 Shield 合并后源站仅看到约 10 rps。他们的站点正常运行。

你并没有建成一个 CDN，而是建了一个在内容爆红时一触即溃的单区域脆弱缓存。nginx 是 CDN 的一个 组件 ，而非替代品——真正起作用的是架构本身（PoP + anycast + 分层缓存 + Shield + 控制平面），nginx 只是在每一层担当前端角色。

§22. 扩展维度

类型 1——均匀扩展（用户增多，单用户行为不变）

1x   ->  10 PoPs, 1 region, 1M rps
2x   ->  20 PoPs, 2 regions, 2M rps          ← add capacity uniformly
10x  ->  50 PoPs, 5 regions, 10M rps         ← add origin shield per region
100x ->  300 PoPs, all continents, 100M rps  ← RUM-driven routing, gossip-based
                                              purge, multi-tier cache hierarchy

约 1000 万用户

  • 约 5M rps：单区域源站开始吃力，增加源站 Shield。
  • 约 20M rps：单机缓存开始频繁换入换出，在 PoP 与 Shield 之间增加中间层缓存。
  • 约 50M rps：BGP 收敛与 anycast 路径异常主导尾延迟，从纯 anycast 转向 anycast + RUM 驱动 DNS 调度的混合模式。
  • 约 100M rps：控制平面清除扇出成为瓶颈，需要层级化 pub-sub、基于 gossip 的传播以及替代键标记。

类型 2——热点集中（某个 URL 突然爆红）

Baseline:   homepage.html at 100 rps globally
Viral:      same URL at 10M rps globally for 1 hour, then back to normal

本质上是不同的问题。如果所有人都在访问同一个 URL，增加 PoP 节点无济于事。

Per-PoP rps for the viral URL:
  10M rps / 300 PoPs = ~33k rps per PoP

Within a PoP:
  hash(URL) -> one box. That ONE BOX takes 33k rps.
  Normal box ceiling: ~50-100k rps. → fine, barely.

But: if cache misses (TTL expires, cold), every PoP's hot box
     simultaneously asks the shield. Shield sees 300 simultaneous
     requests, collapses to a single origin fetch.

类型 2 的关键拐点：

  • 在 PoP 内将热点 URL 分片到多台机器。 副本键： hash(URL || random_in_K) → K boxes。每个请求获得一个随机 K 值，将热点 URL 分散到 K 台机器上。K=10 可将 33k rps/台降至 3.3k rps/台。
  • 对极热键进行中间层合并。 将合并操作一路下推至单机层面（超过阈值时生效）。
  • CPU 饱和而非网络饱和，在极高 rps 且对象极小时出现。解决方案： TLS 卸载到硬件 NIC （Mellanox ConnectX 或同类产品）。

类型 1 和类型 2 有 截然不同的解决方案。类型 1 通过拓扑扩展（增加机器）。类型 2 通过在本地复制热键来扩展。对类型 2 说"增加更多 PoP"是错误的——它只会在所有节点均匀地放大热点 URL 的请求率，方向完全相反。

§23. 决策矩阵——CDN 与相邻技术的对比

CDN 并非总是最优答案，取决于工作负载特征。

地理分布 全球（300+ PoP） 按区域配置存储桶 按区域部署集群 按区域部署
缓存失效 全球 <150 ms 无——直接读取 应用控制 应用控制
命中率优化 调优引擎 + 准入过滤 不适用 应用级调优 有限（按路由 TTL）

决策阈值:

  • 使用 CDN 的场景：面向公共互联网的全球用户流量；可缓存内容（静态资源、GET API、视频、下载文件）；需要 DDoS 防护但不想自己工程实现；源站出口流量超过数 TB/月；持续的边缘 RPS 超出单区域源站的承载能力。
  • 使用对象存储公共读的场景：单区域用户（某一国家的 B2B SaaS）；每月出口流量低于 100 GB；无 DDoS 威胁；直接使用 S3/GCS 公共存储桶更简单且成本更低。
  • 使用应用内缓存的场景：用户私有会话数据；亚毫秒级延迟需求；缓存键为内部标识（非 URL 派生）；数据非 HTTP 格式。
  • 使用 API 网关的场景：路由、鉴权、限流是主要需求；缓存只是附带功能；按路由 TTL 粒度比全球分发更重要。

成熟的技术栈通常四者兼备：CDN 处理公共流量 + 对象存储作为权威存储 + 应用内缓存保存热点内部状态 + API 网关处理鉴权/路由。它们相互组合，而非相互替代。

§24. 使用场景概览

静态资源托管 ——经典 CDN 工作负载。图片、JS、CSS、字体、HTML。长 TTL（天到永久，使用内容哈希 URL）。源站通常为 S3。稳定状态下命中率约 99.9%。所有主流网站都采用此方案——LinkedIn、Pinterest、Etsy、Shopify 的店面。独立 Web 使用 Cloudflare，AWS 原生使用 CloudFront，企业级使用 Akamai。

API 响应缓存 ——Stripe 和 GitHub 的标准做法。偏公开数据（费率卡、GitHub Pages、包元数据），TTL 从数秒到数分钟不等。缓存键包含 API 版本 + 路径 + 查询参数，鉴权请求头被剥离。Stripe 的文档完全由 CDN 提供；GitHub Pages 由 Fastly 前置；包注册中心（npm 通过 Cloudflare，PyPI 通过 Fastly）在热门版本发布时峰值可达每分钟数百万次请求。

视频流媒体 ——Netflix Open Connect 是极致案例。视频文件以多种码率切分为约 10 秒的分片；分片文件是可缓存的单元；自适应码率客户端拉取可变质量内容。YouTube 使用 Google 的 Global Cache + 运营商内部节点。Twitch（直播）采用多个商业 CDN 的混合方案。任意时刻约有几千个标题占据 90% 的观看量。

游戏补丁分发 ——Steam 通过自有基础设施 + CDN 分发 100 GB 以上的补丁。Epic 通过 Cloudflare + 自有的 peer-assist 层将《堡垒之夜》更新推送给超过 1 亿用户。这是 §22 中纯粹的 Type-2 扩展：将补丁切分为多个 CDN 可缓存的块，并行下载，可选的同局域网客户端之间 P2P（peer-to-peer）辅助。

软件下载 ——Linux ISO（Ubuntu 通过镜像网络 + Cloudflare）、Docker registry 穿透拉取（Hub 通过 Cloudflare，ECR 通过 CloudFront）、apt/yum/Homebrew。公共 Docker registry 每天处理数十亿次拉取；热门镜像的 CDN 命中率约 99%。

DDoS 防护（反向使用场景） ——部分客户购买 CDN 服务主要为了抗 DDoS，而非缓存。Cloudflare 的核心业务可以说就是这个——流量较低但攻击面高的小型网站（游戏论坛、处于新闻管控地区的记者）。缓存只是附带功能，核心价值在于带宽容量。Project Galileo 是最公开的案例。

边缘计算 / 边缘个性化 ——CDN 对每个请求执行代码以定制响应。Workers 服务于 Stripe Press，Compute@Edge 承载 NYT 网站的部分页面，Lambda@Edge 为 Shopify 店面动态缩放图片。缓存键扩展到"用户分桶"（已登录与匿名、A/B 实验分组、地理位置），使每个分桶的内容可被缓存，但不存储个人用户状态。

ML 模型与数据制品分发 ——ML 模型文件（10 MB 到 10 GB）由全球推理工作节点拉取，完美契合"可缓存的不可变 blob"模式。HuggingFace 通过 CloudFront 托管模型。Cloudflare R2 + Workers 为部分客户在边缘提供 LLM（Large Language Model）权重和嵌入向量。

重点： 相同的基本原语，截然不同的消费方式。命中率 99.99% 的静态图片 CDN 与 PB 级工作集下命中率 99% 的视频 CDN 在运维上大相径庭，而这又与命中率 30% 处理不可缓存流量的 DDoS 防护型 CDN 截然不同。了解你的工作负载形态，决定了服务商选择、层级划分、缓存键策略以及 TTL 纪律。

§25. 真实案例数据

  • Cloudflare：约占 Web 流量的 20%。280 Tbps 容量。约 300 个 PoP，覆盖 120 余个国家。持续约 5000 万 rps；在 DDoS 事件中峰值突破每分钟 10 亿次请求。2023 年抵御的最大 DDoS 攻击记录：7100 万 rps。
  • Akamai：约占 Web 流量的 30%。约 4200 个 PoP，通常以单机架形式部署在运营商内部。在 NFL 周四橄榄球夜赛重要赛事期间承载超过 250 Tbps 的流量。企业级流媒体和游戏占据主导。
  • Netflix Open Connect：约 17000 台缓存设备部署在运营商内部。热门内容发布时峰值约 200 Tbps。约 95% 的流媒体流量来自用户所在运营商内部节点。每台设备约 250 TB SSD、100 Gbps 网卡，针对顺序 4K 读取优化。按每日字节传输量计算是全球最大的 CDN，但因为只服务单一客户而不为人知。
  • Fastly：约 70 个超级 PoP，配备 PB 级存储。支持 VCL 可编程性。2021 年 6 月 8 日故障——某客户的 VCL 推送触发了潜伏 bug，导致全球数据平面中断约 49 分钟，波及 BBC、Reddit、NYT、Amazon、gov.uk。不到 1 小时内恢复。CDN 故障影响半径的经典研究案例。清除 SLO：全球 p99 < 150 ms。
  • CloudFront：约 450 个边缘节点 + 约 13 个区域边缘缓存。理论带宽 600+ Tbps。与 AWS 源站紧密集成；通过特定模式可免费出口到 S3/ALB。为 Twitch 内容目录、亚马逊零售大量公开页面以及 Netflix 元数据 API 提供服务。
  • BunnyCDN：约 120 个 PoP，约 $0.01/GB 定价——约为 CloudFront 公开定价的 1/8。独立游戏补丁、博客托管、小型 SaaS。
  • Google Global Cache (GGC)：部署在运营商内部的 YouTube、Drive、Play Store 专用设备。YouTube 峰值全球传输超过 1 Pbps，大部分通过运营商内部层级完成。

典型缓存命中请求的延迟预算：

DNS resolve (cached at resolver)        :   2-10 ms
TCP handshake to anycast IP             :  10-15 ms (1 RTT)
TLS 1.3 1-RTT handshake (or 0-RTT)      :  10-15 ms (5 ms if resumed)
HTTP/2 request frame                    :  <1 ms
Edge cache lookup (RAM hit)             :  <1 ms
Cache response framing + send           :   1-2 ms
Network back to client                  :  5-15 ms
----------------------------------------------------------
Total p99 cache hit                     :  30-50 ms

On miss (add):
PoP -> L2 mid-tier                      :  1-3 ms
L2 -> shield                            :  5-20 ms
Shield -> origin                        :  50-200 ms (geography)
----------------------------------------------------------
Total p99 cache miss                    : 100-300 ms

缓存命中的延迟预算主要由 TLS 握手主导。将其减半需要 session 恢复（免费且已广泛部署）、0-RTT 或 QUIC 的更快握手。除此之外，下一个瓶颈是移动网络抖动——无法控制。

§26. 专用 CDN

主流 CDN 格局（Cloudflare、Akamai、Fastly、CloudFront）覆盖通用 Web + API + 图片 + 视频。但有几种工作负载存在值得了解的专用变体。

(a) 游戏补丁与资源分发

游戏补丁体积庞大（AAA 大作可达 100 GB），在窄时间窗口内同时向数千万用户发布（"发布日"）。纯 CDN 方案可行但成本高昂；游戏行业已构建出混合方案。

  • Steam Content Server Network：Valve 自有 CDN；全球约 100 余个 PoP；峰值约 150 Tbps。同时辅以 Steam Pipe，本质上是带 Tracker 的 BitTorrent：客户端从 Steam 服务器下载块， 。 也从同一网络的对等节点下载。对运营商友好，因为局域网下载可减少 WAN 流量。
  • Epic Games（堡垒之夜）：使用 Cloudflare CDN 进行分发；其 Easy Anti-Cheat 基础设施用于游戏内服务。补丁采用分块 CDN 分发。
  • Battle.net（暴雪）：自有 CDN + Akamai + 自有 P2P 客户端（"Battle.net 更新代理"，曾用于 torrent 补丁分发，随着带宽成本下降已逐步缩减）。
  • Riot Games：使用 Akamai + 自有基础设施；近期更多迁移至公有云。
  • Microsoft（Xbox + Game Pass）：Xbox Live 分发通过 Azure CDN + 自有运营商内部设备，类似于 Netflix Open Connect。

规律： 不可变的块 + 并行下载 + 当局域网对等节点拥有所需内容时进行 peer-assist。纯 BitTorrent 因 NAT 复杂性和版权问题逐渐退出主流，但其结构性思路依然延续。

(b) 软件下载——Linux 发行版、Docker registry

Linux 发行版镜像自 90 年代起便已存在。当前状况：

  • Ubuntu / Debian / Fedora 镜像：传统的志愿者运营镜像网络（高校、企业）加上 CDN 增强。Ubuntu 使用 Cloudflare 服务 releases.ubuntu.com。Debian 拥有数百个镜像；类似 apt-fast 。 mirrorbits 的元工具可自动选择最快的镜像。
  • Docker Hub：通过 Cloudflare 提供（Docker Inc. 作为 Cloudflare 客户）；由于镜像按 digest 不可变，在边缘进行激进缓存。
  • AWS ECR（Elastic Container Registry）：通过带区域边缘缓存的 CloudFront 提供。
  • GitHub Container Registry：通过 Azure / Fastly 提供。
  • Homebrew（macOS 包管理器）：bottle 通过 Cloudflare 和 GitHub LFS 提供；归档文件通过 GitHub CDN 提供。
  • PyPI / npm / Maven Central：分别通过 Fastly（PyPI）、Cloudflare（npm）、CloudFront（Maven Central）提供。热门版本发布时（例如 React 大版本发布），单个包的下载量可每分钟突破数百万次。

穿透缓存值得一提：企业环境中可能运行一个 registry 代理 （例如 Sonatype Nexus、JFrog Artifactory），从上游缓存数据。该代理对企业内网而言本身就是一个 CDN——相同的基本原语，只是规模更小。

(c) 直播流——Twitch、YouTube Live、LL-HLS

直播流是 VOD 流媒体对延迟敏感的兄弟。Twitch 和 YouTube Live 的目标是从主播到观众端到端约 3~5 秒的延迟；体育赛事直播目标约为 10~20 秒。

协议演进：

  • RTMP（Real-Time Messaging Protocol）：Flash 时代的协议；仍用于从主播到 CDN 的 推流 ，但不用于分发。
  • HLS / DASH，分片时长 2~10 秒：端到端延迟 30 秒以上。对直播而言太慢。
  • LL-HLS（Low-Latency HTTP Live Streaming）：Apple 的规范；分片的分块传输；约 2~5 秒延迟。CDN 处理部分分片的缓存。
  • LL-DASH：DASH 的等效协议。
  • WebRTC 分发：亚秒级延迟；部分平台用于交互式直播（Twitch 测试版低延迟模式、YouTube 的"超低延迟"模式）。不使用传统 CDN 缓存——采用媒体服务器架构中的 SFU（Selective Forwarding Unit）。

Twitch 架构：通过 RTMP 在推流 PoP 完成推流；转码为多种码率；切片；通过自有 CDN 分发（此前大量依赖 Akamai，目前自研比例显著提升）。近期增加了 LL-HLS。

YouTube Live：类似的流水线；使用 Google 全球基础设施；支持 LL-HLS 和超低延迟 WebRTC 模式。

(d) 纯 DNS / 仅流量调度的 CDN

NS1 和 Cedexis 在 §20 中有所提及。它们完全不缓存 HTTP——只负责调度。其在 CDN 生态中的角色：

  • 多 CDN 前端：根据每次查询选择路由到哪个 CDN。
  • 遗留架构的 GeoDNS （每个区域一个源站；DNS 选择最近的节点）。
  • 故障转移编排：监控后端；绕过故障路由。

NS1 的定价模型：按查询量 + 按流量管理功能收费。对于每月 1 亿次查询且采用多 CDN 的网站，费用约为每月数千美元——相比 CDN 出口费用而言微不足道。

(e) 私有云 / 本地 CDN

适用于无法使用公共 CDN 的工作负载（合规要求、数据驻留、内部网络超低延迟）：

  • VMware vSAN + 边缘节点：企业客户在自有数据中心部署迷你 PoP。
  • 电信边缘 （5G MEC——Multi-access Edge Computing）：移动运营商提供与基站共置的边缘计算和缓存。AT&T、Verizon 及其他运营商均有此类服务。
  • 面向 HFT（High-Frequency Trading）的专用 CDN：追求微秒级优化的场景；严格意义上并非公开的"CDN"，但采用相同架构（地理分布式缓存 + 快速路由）用于市场数据分发。

(f) 专项：广告技术与分析 CDN

广告网络和分析平台有其特殊的 CDN 需求：

  • Google Ad Manager / Doubleclick：通过 Google CDN 提供广告素材；延迟预算为个位数毫秒。
  • 面向广告技术的 Cloudflare：许多广告技术公司使用 Cloudflare 的边缘节点降低单次请求延迟。
  • Snowplow / Segment：分析 SDK 需要 CDN 托管埋点 JS；每个事件的收集通常通过 CDN 代理转发至后端。

核心约束：广告技术同时要求 极低延迟 （竞价时间预算为 100 ms）和 灵活可编程性 （频繁的 A/B 测试、实时竞价逻辑）。边缘计算非常契合；单纯的缓存层通常无法满足。

§27. 成本经济学深度解析

CDN 成本在签合同之前往往不透明。成本模型有多个维度，每个维度都有其独特的定价曲线。

(a) 出口成本（按 GB 计费）

主要成本项。实际定价（公开报价；企业折扣通常为 5~9 折）：

服务商 地区 每 GB 价格（公开报价，2025 年） 备注
CloudFront 美国、欧洲 $0.085（前 10 TB） 大流量可降至 $0.020
CloudFront 亚洲、澳大利亚 $0.114 地理溢价
Cloudflare 大多数地区 套餐包含（专业版/商业版） 按月付固定费用；出口不单独计量
Cloudflare 企业版 定制价，通常低于 $0.005/GB 量级合同
Fastly 美国、欧洲 $0.12（前 10 TB） 量级之后更便宜
Akamai 自定义 议价 规模化后通常为 $0.005~0.02
BunnyCDN 全部地区 $0.01 独立开发者定价；功能有限
Google Cloud 美国 $0.08（前 10 TB） 阶梯下降

趋势：出口价格在过去十年 已下降约 50%。AWS 于 2024 年宣布为离开 AWS 的客户提供约免费出口到互联网的服务（迟来的让步）。超大规模厂商公开定价较高；专业 CDN（Cloudflare、BunnyCDN）定价较低。

Cloudflare 的定价模式较为特殊：大多数方案将出口费用打包（按月支付专业版/商业版/企业版固定费用；不单独按 GB 计费）。这得益于其网络经济（带宽打包对等互联），也是许多高带宽网站从企业级 CDN 迁移到 Cloudflare 的原因。

(b) 请求成本（按每百万次请求计费）

独立于字节量。典型价格：

服务商 每 1 万次 HTTPS 请求 备注
CloudFront $0.0100 加出口费
Cloudflare 套餐包含 大多数方案
Fastly $0.0100 加出口费
BunnyCDN $0.0004 独立开发者定价

对于请求量大但响应体积小的网站（API CDN），请求成本可能超过出口成本。一个每月 10 亿次请求的网站，仅请求费用就达每月 $1000（按 $0.01/万次计算）。

(c) 边缘计算成本

计算定价：

  • Cloudflare Workers：每月 $5 基础费 + 每百万次请求 $0.50（套餐计划）；超出前 1000 万次后每百万 CPU-ms 收费 $0.30（按量计划）。
  • Fastly Compute@Edge：每月 $50 基础费 + 每百万次请求 $0.50。
  • AWS Lambda@Edge：每百万次请求 $0.60 + 每 GB·秒执行时长 $0.0000050001。冷启动计入计费时长。
  • AWS CloudFront Functions：每百万次请求 $0.10；非常便宜；限制极多（约 1 ms CPU 预算）。

对于每天处理 1000 万次边缘计算请求的网站：Workers 约 $50/月，CloudFront Functions 约 $60/月，Lambda@Edge 更贵。

(d) "静态便宜、动态昂贵"的经济规律

CDN 成本的经济形态：

  • 静态、不可变、内容哈希资源，无边缘计算：出口约 $0.005~0.02/GB，请求成本几乎为零（套餐包含），无计算费用。成本最低。
  • 偏动态、短 TTL、边缘有请求头逻辑：以上费用 + 每请求的边缘计算费用。单次请求仍然便宜，但累积可观。
  • 边缘个性化、重量级计算：边缘计算账单高昂。通常与在源站运行等效计算的成本相当。
  • 不可缓存 （Cache-Control: no-store，或每次请求携带鉴权）：纯透传；同时支付边缘带宽和源站计算费用。最差情况。

优化思路： 尽可能使更多内容可缓存。通过边缘计算将动态内容转化为可缓存内容（例如 GraphQL 持久化查询、为可缓存分桶变体分配 A/B 分桶），往往能通过减少源站计算来抵消边缘计算的成本。

(e) 免费套餐

主要 CDN 免费套餐：

  • Cloudflare Free：免费 TLS、DDoS 防护、基础 WAF、无限带宽。最慷慨的免费套餐；这也是许多独立网站选择 Cloudflare 的原因。
  • CloudFront Free Tier：AWS 免费套餐提供 12 个月内 1 TB 出口流量、每月 1000 万次请求。
  • Netlify、Vercel：边缘服务包含在托管方案中；每月带宽有上限。
  • BunnyCDN：无免费套餐，但每月最低 $1 已接近免费。
  • GitHub Pages：在 GitHub 基于 Fastly 的 CDN 上免费托管。

免费套餐有其限制——Cloudflare Free 的 DDoS 防护对 L3/L4 无限制，但 L7 功能有频率限制；CloudFront 免费套餐仅限一年。对于个人项目，Cloudflare Free + Vercel 几乎可以覆盖所有需求。

(f) 费用超支的常见陷阱

客户超出 CDN 预算的常见原因：

  • 缓存键碎片化导致命中率下降。 一个错误配置将命中率从 99% 降至 50%；到达源站的出口量和请求量暴增 50 倍；CDN 账单增加（更多"未命中"响应穿透至出口），同时源站成本急剧攀升。
  • 向其他区域的出口流量。 多区域服务结合跨区域复制可能在超大规模云服务商区域之间产生意外的出口流量；某些 CDN 提供商对特定地区（亚洲、南美洲）的服务收取溢价。
  • 高级功能的蔓延。 叠加使用机器人管理（$50/月）、图片优化（约 $0.10/1000 次转换）、Workers（约 $5/月/服务）会累积成可观的费用。许多企业最终在功能上的花费是带宽费用的 5-10 倍。
  • 协商价与标准定价。 在签订合同之前，你按标准价格付费。企业级承诺（最低 $10K/月）通常可解锁 50-80% 的折扣。

§28. 可观测性

在没有可观测性的情况下运行 CDN 集成，无异于盲目飞行。现代 CDN 提供商提供丰富的遥测能力；用好这些能力是运营 CDN 前置服务的关键所在。

(a) 边缘日志流

最基本的原语：每个 PoP 上的每个请求都可以被记录，这些日志可以流式传输到你的 S3 / GCS / Splunk / Datadog。实现方式如下：

  • Cloudflare Logpush：将 HTTP 请求日志（以及 Workers 日志、防火墙日志等）流式传输至 S3 / GCS / Azure / Splunk / Datadog。从请求到日志的延迟低于一分钟。
  • Fastly Real-Time Log Streaming：支持 HTTP / TCP / syslog 端点；实时记录每个请求，延迟可达亚秒级。
  • CloudFront access logs：写入 S3，延迟几分钟。实时日志（CloudFront Real-time Logs）可通过 Kinesis Data Streams 获取，延迟达亚秒级。
  • Akamai DataStream：类似的流式传输方案。

日志量可能非常庞大：一个每秒 1000 万请求的站点每秒产生约 1000 万条日志行；以每行约 500 字节计算，这意味着每秒 5 GB 的数据量。大多数用户采用采样（1% 的请求）或在边缘进行聚合。

(b) 单个 PoP 的可见性问题

跨所有 PoP 的聚合指标会掩盖特定 PoP 的故障。一个承担全球 1% 流量的 PoP 可能完全失效（报错、高延迟、低缓存命中率），而全局指标几乎不会有任何波动。

缓解方案： 单 PoP 仪表盘。Cloudflare 的分析数据按 PoP（在其术语中称为"colo"）呈现；Fastly 的单 PoP 分析与之类似；CloudFront 的单边缘位置分析能力则相对有限。

常见的单 PoP 异常：- 某个 PoP 命中率下降——提示缓存驱逐风暴或本地硬件问题。- 某个 PoP 的 p99 延迟峰值——提示本地网络饱和或热点问题。- 某个 PoP 的错误率上升——提示该 PoP 与源站的连接存在问题。

对单 PoP 异常进行告警（当某个 PoP 的指标偏离全局均值 N 个标准差时），能捕获全局聚合指标所遗漏的问题。

(c) 真实用户监控（RUM）

CDN 侧的指标是服务端视角：CDN 响应花了多长时间？真实用户延迟还包括返回客户端的网络路径（在移动端通常占主导地位）、浏览器渲染时间和页面加载时间。RUM 捕获的正是这些：

  • Cloudflare Web Analytics：通过一个轻量级 JS 信标实现的隐私友好型 RUM。
  • Akamai mPulse：企业级 RUM，功能极为全面。
  • Google Analytics / GA4 with Core Web Vitals：非 CDN 专用，但广泛使用。
  • SpeedCurve, New Relic Browser, DataDog RUM：第三方 RUM 产品。

RUM 揭示的信息：- 真实用户 p99 延迟 （相对于仅包含 CDN 侧时间的服务端 p99）。- 延迟对转化率的影响 （p99 每增加 100 ms → 转化率下降 N%）。- 各地区性能差异：某个地区的用户延迟明显偏高，提示路由问题、对等互联问题或本地网络故障。- 移动端 vs 桌面端：移动端用户的 p99 通常高出约 3 倍。- CDN 提供商对比：在多 CDN 架构中，RUM 数据显示哪家 CDN 在各地区表现更好——反馈至路由决策。

(d) 合成监控

RUM 的补充手段：从多个探测点持续探测站点；在出现故障或延迟劣化时发出告警。相关供应商：

  • Catchpoint, ThousandEyes, Pingdom, Uptrends：拥有大量探测位置的合成监控服务。
  • Datadog Synthetic Monitoring：与其可观测性技术栈集成。
  • AWS CloudWatch Synthetics：AWS 原生方案。

合成监控能捕获 RUM 无法覆盖的情况：- 低流量期间的故障 （RUM 仅在用户访问站点时才会触发）。- 特定用户流程的正确性 （合成监控可模拟复杂的多步骤流程；RUM 是单页粒度）。- 特定 PoP / 区域的健康状况 （合成监控探测特定 PoP；RUM 取决于用户的实际行为）。

(e) CDN 侧指标

除日志外，CDN 提供商还暴露聚合指标：

  • 命中率：按主机、路径、区域、时间划分。这是最重要的单一指标。
  • 出站带宽：按主机、区域、内容类型划分。
  • 请求数：按状态码、路径划分。
  • 错误率：4xx、5xx 分项统计。
  • 源站响应时间：shield 等待源站响应的时长。
  • DDoS 攻击指标：被拦截的流量/请求数量、观测到的攻击类型。
  • WAF 规则命中：哪些规则在哪些路径上触发。

Cloudflare Analytics、Fastly Insights、CloudFront CloudWatch 指标、Akamai mPulse——均暴露上述数据。这通常是 CDN 运维人员首先查看的仪表盘。

(f) 跨 CDN 与源站的链路追踪

一个经过 CDN → shield → 源站的请求应当支持端到端追踪。常见模式：

  • X-Request-ID 请求头传播：CDN 注入请求 ID；源站日志包含该 ID；客户端在响应中获取该 ID（可选）。可实现跨系统关联。
  • W3C Trace Context（traceparent 请求头）：标准化的分布式追踪上下文。CDN 将其从客户端传播至源站（若不存在则自动生成）。
  • Cloudflare 的 cf-ray 请求头：每个 Cloudflare 请求都有唯一的 ray ID；日志中包含该 ID；客户可与其源站日志进行关联。

对于复杂故障（慢请求、间歇性报错），跨 CDN + 源站 + 下游服务的链路追踪至关重要。若没有追踪上下文传播，CDN 日志与源站日志相互独立，只能靠猜测进行关联。

(g)"影子流量"模式

部分运维人员将生产流量的副本发送至影子源站以验证变更：- CDN Worker 分叉请求，同时发送至真实源站和影子源站。- 将真实源站的响应返回给用户。- 异步比较影子源站的响应，差异时发出告警。

用于新源站版本的金丝雀发布、回归测试和 A/B 对比。在边缘实现成本低廉。

§29. 总结

CDN 是一个行星级分布式直读缓存，以 anycast 路由为前端：每个 PoP 运行一个采用 W-TinyLFU 准入策略、基于一致性哈希在 PoP 内分片的日志结构对象存储，通过区域中间缓存分层汇聚至源站 shield，从而将缓存未命中请求合并，使源站即便在边缘每秒 1000 万请求的情况下也只承受个位数的 RPS。其技术价值在于：以命中率驱动的源站卸载 + 拓扑邻近性 + DDoS 稀释；持久性、强一致性、schema 感知和单用户隐私隔离则必须在上层叠加实现。无论工作负载是静态资源、API 响应、视频分片、游戏补丁、软件下载还是 ML 模型文件，同样的原语均适用；差异体现在缓存键的规范性、TTL 的纪律性，以及你在设计空间——第三方共享、私有 ISP 内建、可编程边缘、专用媒体——中所做的选择。
