对象存储技术参考——它是什么、在字节层面如何工作、设计空间在哪里分叉、何时选择哪种变体。使用场景（静态资产托管、视频/媒体目录、数据湖、ML 模型与训练数据存储、备份、用户上传文件）贯穿全文，作为同一类技术适配不同工作负载的示例。

§1. 对象存储是什么

一个 对象存储 是一个面向星球级规模的扁平 key 到不透明 blob（Binary Large OBject）的存储系统。其契约很简单：给定一个 bucket 和一个 key 字符串，交给它字节数据（最大5 TB），它将以 十一个九 的持久性保存数据，并将其提供给任何拥有读取权限的人，无论何地、无限期。它是任何以 文件大小的 blob 而非行或记录作为访问单元的系统的底层支撑。

该契约同样由它 非 做的事情定义：

  • 不支持对象内读写。 GET 返回整个对象（或字节范围）； PUT 写入整个对象。不支持原地编辑。
  • 不理解内容语义。一个5 GB 的 Apache Parquet 文件、一个2 KB 的 JPEG、一个200 GB 的 Iceberg 快照、一个4 MB 的 XGBoost（eXtreme Gradient Boosting）模型 pickle——全部是不透明字节。Schema 存储于别处（Hive Metastore、Iceberg catalog、数据库行）。
  • 不支持 POSIX（Portable Operating System Interface）语义。没有 fsync、没有 open() 返回文件描述符、没有目录 inode、没有硬链接、没有 rename()。控制台中的"文件夹"是基于 key 前缀的虚构概念。
  • 不支持多对象事务。两个 PUT是两个独立操作；没有 BEGIN; PUT a; PUT b; COMMIT;.

对象存储处于四种 静态字节 技术的设计空间的一个角落：

  • 块存储 ——AWS EBS（Elastic Block Store）、GCE Persistent Disk、Azure Managed Disks。面向单个主机的裸磁盘；在扇区（4 KB）级别随机读写；绑定到单个 VM。文件系统运行于其上。延迟约100µs–1ms；单卷容量数十 TB。适合数据库数据文件、OS 根卷、Redis AOF（Append-Only File）日志。不适合"将此视频共享给3000万观众"的场景。
  • 文件存储 ——NFS（Network File System）、AWS EFS（Elastic File System）、Azure Files、GCP Filestore。网络上的 POSIX：目录、文件句柄、字节级 I/O、锁。延迟几毫秒；PB 级容量。适合 HPC（High-Performance Computing）临时存储、遗留应用、共享配置。不适合数十亿小对象——EFS 每 GB 成本约为 S3 的10倍。
  • 对象存储 ——S3、Google Cloud Storage（GCS）、Azure Blob、R2（Cloudflare 的 S3 兼容产品）、MinIO、Ceph RADOS Gateway。本文档的主题。
  • 数据库 / 结构化存储 ——schema、事务、查询、二级索引。不适合2 MB 图像 blob（会破坏 buffer pool——参见 01_databases.md）；不适合80 GB 模型工件；不适合 PB 级 Parquet 数据。

心智模型：块存储是"面向单台服务器的快速本地磁盘"；文件存储是"面向多台服务器的共享磁盘"；数据库是"用于快速查找的结构化表"；对象存储是"容纳一切不适合前三种场景内容的无限廉价持久空间"。

对象存储不擅长的场景：

  • 低延迟点读。 S3 GET 在同一区域内的 p50 约为30-80ms，p99 约150-300ms。对于"在50ms内渲染页面"的需求，请在前面放置 CDN（Content Delivery Network）或内存缓存。
  • 事务性更新。 不支持多对象原子性。若你的不变式是"余额在两个账户间转移"，请在数据库中处理。
  • 查询对象内部内容。 S3 Select 和 Athena 会扫描整个对象，并按扫描字节数计费。如果你需要"select * where user_id = 42"，请将数据存入数据库或列式数仓。
  • 追加写入量大但单次写入较小。 100 字节的日志行，每秒 100k 条 = 100k 个独立的 PUTs = 请求费用（约 $0.50/百万次）+ 100k 个元数据槽位。请在本地缓冲，批量写入 64 MB。
  • 文件系统语义。 否 rename()。"重命名"一个 10 TB 的文件夹需要 N 次 COPY + N 次 DELETE 调用——这是数据湖中最常见的反模式。
  • 在 5 GB 文件内进行随机读写。 范围 GET 读取没问题；修改则需要重新上传整个文件。

标准角色：对象存储是架构栈底部的 被动持久层 。数据湖、模型注册中心、图片 CDN 源站、数据库的夜间备份、用户上传的文件——都存放在这里。反复出现的模式是：数据库保存 指向对象的元数据；对象保存 字节.

§2. 固有保证

该技术在设计上提供的保障，以及仍需在其上叠加的能力。

设计上已提供

  • 所有操作均具备强读写后一致性。 S3 自 2020 年 12 月起如此——在此之前，S3 对覆盖写和删除操作采用最终一致性，迫使应用层进行各种变通处理（哨兵键、不可变命名）。如今所有 S3 操作—— PUT, GET, LIST, DELETE, HEAD ——均具备强一致性。GCS、Azure Blob、R2 以及现代版本的 MinIO 自发布起即为强一致性。历史上"写后列举"的怪异行为已成过去。
  • 十一个九的持久性 （99.999999999%）。预期年均丢失量约为每 1000 亿字节 1 字节。实现机制：复制、纠删码以及跨可用区（AZ）分布。
  • 容量实际上无上限。 无 bucket 大小限制。AWS 报告称 S3 已存储超过 280 万亿个对象。
  • 按 SLA 层级衡量的单对象可用性 ——S3 标准月度可用性 99.99%；单区域低频访问（One Zone IA）99.5%；Glacier 各层级对检索操作有独立的 SLA。
  • 区域或多区域范围。 一个 bucket 存在于跨多个 AZ 的单一区域；跨区域复制需显式配置（S3 CRR——跨区域复制，GCS 双区域 bucket）。
  • 版本控制 作为可选功能：每次向同一 key 执行 PUT 操作时都会创建新版本；旧版本按策略保留。

需在其上叠加

  • 低延迟读取。 区域外 p50 延迟 50–150ms，p99 延迟数百毫秒。若需 50ms 以下的读取延迟，请在前面部署 CDN（CloudFront、Cloudflare、Fastly）或在内存中缓存。
  • 跨对象事务。 "原子性地更新对象 A 和 B"是应用层的职责。常见模式（Apache Iceberg、Delta Lake、Apache Hudi）：以唯一名称写入新文件，然后在 catalog 中原子地翻转元数据指针。
  • 在对象内部查询。 可使用 Iceberg / Delta / Hudi catalog、Athena / BigQuery / Snowflake，或将数据拉入数据库。对象存储本身不建立索引。
  • 丰富元数据。 S3 元数据是一个扁平的 KV 映射（x-amz-meta-*）。超出 key/size/last-modified 范围的内容——所有者、父级帖子、审核状态——需存入真正的数据库。
  • 配额与速率管理。 S3 不会在你花费过多时拒绝写入；它只是继续计费。成本控制在 IAM（身份与访问管理）或应用层实现。
  • 使用客户托管密钥加密。 默认 SSE-S3（使用 S3 托管密钥的服务端加密）是自动启用的；SSE-KMS（密钥管理服务）、SSE-C（客户提供密钥）以及客户端加密则需要显式配置（§13）。
  • 多区域读写语义。 S3 MRAP（多区域访问点）在区域间为最终一致性。主主架构是一个独立的设计问题（§7.3）。
  • 生命周期与分层决策。 云服务商提供工具（生命周期策略、存储类别）；决定在何处保留何种数据、保留多久是系统设计者的职责（§12）。
  • 权限管控。 IAM 策略、bucket 策略、ACL（访问控制列表，已过时）、预签名 URL、公共访问拦截（Public Access Blocks）。"S3 bucket 泄露"是最常见的云配置错误安全漏洞类型。

综合来看：该技术保证 你写入的字节能够完整地永久返回 并具备强一致性。其余所有内容——快速读取、事务语义、查询、丰富元数据、加密、成本——都是系统设计者需要解决的问题。

§3. 设计空间

各变体在定价、生态系统和一些协议细节上有所不同，但它们都遵循 S3 API的两种形态之一：AWS 原版 S3 REST API，或其重新实现版本。

超大规模云服务商托管

  • AWS S3（简单存储服务） ——2006 年 3 月上线。其他人都在复制的协议标准。十一个九的持久性；标准可用性 99.99%；六种存储类别。存储万亿级对象，EB 级数据量。自 2020 年 12 月起具备强一致性。行业基准。
  • Google Cloud Storage（GCS） ——概念模型相同。独特之处：支持透明复制的双区域/多区域 bucket（无需配置 CRR）；bucket 级别版本控制；存储层级为标准、近线（30 天最短存储期）、冷线（90 天最短）、归档（365 天最短）。从第一天起就具备强一致性。
  • Azure Blob Storage ——三种 blob 类型：块 blob（等同 S3）、追加 blob（针对日志追加优化）、页 blob（等同 EBS，用于 VM 磁盘）。热/冷/较冷/归档层级。强一致性。与 Azure Active Directory RBAC（基于角色的访问控制）、SQL Server、Office 365 深度集成。

S3 兼容替代方案

  • Cloudflare R2 ——2022 年上线。核心特性： 零出口流量费。S3 按约 $0.09/GB 收取互联网出口流量费；R2 收 $0。对于公共资产和大数据集工作负载，这意味着 10–100 倍的成本降低。存储约 $0.015/GB/月。兼容 S3 API。缺点：分层选项有限，生态系统较小。
  • Backblaze B2 ——2015 年上线。存储约 $0.006/GB/月，出口流量 $0.01/GB。兼容 S3 API。通过与 Cloudflare 的带宽联盟，访问 Cloudflare 网络的出口流量实际上为零。备份场景口碑极佳。
  • Wasabi ——单一层级，$0.0059/GB/月，无出口流量费，无 API 调用费。"一口价"可预期定价。权衡：无分层选项，全球节点较少。
  • DigitalOcean Spaces ——兼容 S3，$5/月含 250 GB 存储 + 1 TB 出口流量。适合小型工作负载。

自托管

  • MinIO ——开源，兼容 S3 API，单一 Go 二进制文件。可从家用实验室扩展至多 PB 级别。常见于隔离网络环境、本地数据湖、本地开发 S3 替代方案。纠删码可配置。AGPL（Affero 通用公共许可证）v3 社区版。
  • Ceph + RADOS Gateway ——开源分布式存储，提供对象、块和文件接口。RADOS（可靠自治分布式对象存储）是底层； radosgw 在其上提供 S3 REST API。用于 CERN（EB 级规模）、DigitalOcean Spaces（多年基于 Ceph 构建）、众多高校。运维复杂度高于 MinIO；但更灵活。
  • OpenStack Swift ——早期方案（2010 年），历史上为最终一致性。用于 Rackspace、eBay。随着该领域逐渐向 S3 API 兼容的 MinIO/Ceph 整合，其使用正在减少。

§4. 字节级别：对象存储内部的实际工作原理

深度章节。我们深入存储层，解释实际机制——纠删码、驱动吞吐量限制的前缀分片模型、分片上传的字节编排、预签名 URL 的 HMAC（基于哈希的消息认证码）数学原理、版本控制，以及逐字节的 PUT 逐字节解析。

4.1 bucket 中实际存放的内容：key 与扁平命名空间

"文件夹和文件"的思维模型是虚构的。一个 bucket 是一个 扁平 KV 映射 ，从字符串 key 映射到不透明字节加一个微小的元数据 blob。正斜杠只是普通字节； users/42/avatar.jpg 是一个 key，而非三层嵌套目录。S3 控制台解析斜杠来渲染树形结构，但存储层对此毫不知情。

影响：

  • 否 rename()。"移动" users/42/ 与 users/v2/42/ 需要 N 次复制 + N 次删除。对于包含十亿个对象的前缀，这是一项耗时数天的工作。
  • 控制台"重命名"实际上悄悄执行复制-删除操作。
  • 无文件夹级权限。bucket 策略使用前缀字符串匹配（arn:aws:s3:::my-bucket/users/42/*).

一个 key 最长为 1024 字节（S3）或 1024 个字符（GCS）。单对象元数据：key、version-id、size、content-type（MIME——多用途互联网邮件扩展）、last-modified、etag（内容的 MD5 或分片上传派生哈希）、存储类别、加密头部、用户 x-amz-meta-* 头部（几 KB）。

4.2 纠删码：十一个九是如何实现的

对象存储宣传"十一个九的持久性"。这些九从何而来？答案是 纠删码 ——Reed-Solomon——跨多块磁盘和 AZ 应用，用于替代或补充传统复制。

朴素方案：三副本复制。 存储三个副本。要丢失数据，三块磁盘必须同时故障。商用 SATA（串行 ATA）磁盘年故障率约 2%；三块同时故障的概率极低。代价： 3 倍存储开销。在 EB 级规模下，这非常可观。

纠删码更优。 Reed-Solomon 将大小为 D 的数据块生成 D + P 个分片；任意 D 个分片即可重建原始数据。常见部署配置：

  • (10, 4) ——10 个数据分片 + 4 个校验分片。容忍 4 个同时故障。开销 1.4 倍。比 3 副本更强，成本低 2 倍。
  • (6, 3) ——MinIO 默认值。1.5 倍开销。
  • (17, 3) ——用于 Meta 的 f4 冷存储。1.18 倍开销，容忍 3 个故障。

数学原理简述。 Reed-Solomon 是基于 GF(256)（256 元素的 Galois 有限域）的有限域线性代数。D 个数据字节为多项式系数；D + P 个分片是在 D + P 个点处的求值；任意 D 个点即可通过拉格朗日插值恢复多项式并还原系数。运算为 XOR（异或）和 GF 乘法，借助 AVX（高级向量扩展）每核心每秒可处理约 GB 级数据。

跨故障域分布。 纠删码是必要条件，但不充分——分片必须分布在 独立的故障域 （不同磁盘、机架、电路、AZ）中。S3 将单个对象的分片条带化分布在一个区域的多个 AZ 中。2017 年 12 月的 US-East-1 故障（§8）验证了这一机制的有效性：存活的 AZ 仍能正常服务 GET 请求。

具体数字 ——以 (9, 6) 方案跨 3 个 AZ 为例：

Object O is 100 MB. Encode into 15 fragments of ~11.1 MB each.
   AZ-1: fragments 1..5
   AZ-2: fragments 6..10
   AZ-3: fragments 11..15

A GET reconstructs from any 9 of 15 → typically reads from 2 of 3 AZs.

Failure scenarios:
   Lose 1 disk      → 14 of 15 left. Fine.
   Lose 1 rack      → fine if rack held <7 fragments.
   Lose 1 AZ        → 10 of 15 left. Still readable (need 9). Repair re-encodes lost fragments elsewhere.
   Lose 2 AZs       → 5 left, below threshold. DEGRADED. Recovery from other regions' copies if CRR.

持久性计算。 以 2%/年的磁盘故障率、以小时计的修复时间，以及跨独立故障域的 (10, 4) 配置，预期单对象年丢失概率约为 ~10⁻¹¹——即十一个九的数字。

冷热权衡。 纠删码有代价：读取一个对象需要读取 D 个分片而非一个。对于热读，这意味着带宽放大。S3 标准层对活跃层使用较低开销的方案或副本复制；Glacier Deep Archive 在不同介质（磁带库、SMR——叠瓦式磁记录——硬盘）上使用极端纠删码配置（高 D 值，非常长的修复窗口），以成本为优化目标。

4.3 前缀分片模型：5500 次 PUT / 3500 次 GET 每前缀限制的由来

S3 公布的限制是 每前缀每秒 3500 次 GET / HEAD 。 每前缀每秒 5500 次 PUT / POST / COPY / DELETE。什么是"前缀"，这些数字从何而来？

内部架构 （在 AWS re:Invent 演讲中有公开描述）：S3 通过 对 key 前缀进行哈希来对 keyspace 进行分区。一个 bucket 的命名空间被划分为多个分片（或"索引分区"）；每个分片负责哈希前缀空间中的一段连续范围，并运行在一组元数据 + 存储服务器上。 3500/5500 的数字是单个分片的上限。

分片会自动分裂——S3 检测到持续的热前缀后会分裂该范围以分散负载。分裂是渐进式的（需要数分钟到数小时，突发性宽负载需要更长时间），在此期间你会遭遇限流。

具体故障场景。 常见的数据湖反模式：

s3://my-lake/year=2026/month=05/day=22/hour=14/file-000.parquet
s3://my-lake/year=2026/month=05/day=22/hour=14/file-001.parquet
... 10,000 files written per hour, all under the same prefix

前 ~5 个字符（year=）是常量；哈希将所有文件聚集在同一个分片中。当突发请求超过 5500/秒时，你会收到带有退避提示的 503 SlowDown 。任务"随机变慢"是因为 S3 正在对你限流。

解决方案：对 key 添加哈希前缀。

s3://my-lake/a3f7/year=2026/month=05/day=22/hour=14/file-000.parquet
s3://my-lake/b9e2/year=2026/month=05/day=22/hour=14/file-001.parquet

在 MD5(filename)[:4] 的基础上前置 4 字节十六进制哈希，将 key 分散到 65k 个前缀桶中；每个分片只承受约 1/65k 的流量。bucket 的总体吞吐量实际上变得无上限。

现代表格格式（Iceberg、Delta、Hudi）写出类似 s3://lake/db/table/data/00000-12-abc123/file.parquet 的路径，其中内置了哈希段。Hive 风格的分区表则需要显式指导。

bucket 整体上限。 无文档记载的上限——"无限制，受限于不同前缀的数量。"Netflix、Lyft、Uber 均公开发布过针对单个分布良好的 bucket 持续超过 100,000 次 GET/秒的案例。

4.4 分片上传机制：单次 PUT 5 GB 上限

单次 PUT Object 请求最大支持 5 GB。若要上传超过 5 GB（最大 5 TB 对象限制）的对象，需使用 分片上传。即使对于 100 MB ~ 5 GB 的对象，也建议使用分片上传以获得并行性。

上传流程。

1. Client → S3:  CreateMultipartUpload(bucket, key)
                 ← UploadId (a server-side handle)

2. Client splits the file into N parts, each 5 MB–5 GB:
       part 1: bytes 0 .. P1
       part 2: bytes P1 .. P2
       ...
       part N: bytes Pn-1 .. EOF

3. Client uploads each part independently (in parallel, with retries):
       UploadPart(bucket, key, UploadId, partNumber=1, body=<part 1>)
            ← ETag1
       UploadPart(bucket, key, UploadId, partNumber=2, body=<part 2>)
            ← ETag2
       ...
       UploadPart(bucket, key, UploadId, partNumber=N, body=<part N>)
            ← ETagN

4. Client → S3:  CompleteMultipartUpload(bucket, key, UploadId, [(1, ETag1), (2, ETag2), ..., (N, ETagN)])
                 ← S3 atomically assembles the parts in order, stores the
                   resulting object, returns the multipart ETag and version-id.
                   ETag for multipart = MD5 of concatenated part MD5s, with -N suffix.

各分片可并行上传 以任意顺序、独立重试、可跨天暂停和恢复。直到 CompleteMultipartUpload，各部分处于"暂存"状态——仅可通过 ListParts(uploadId)，而非 GET。Complete 是原子切换：切换前，对象不存在；切换后，完整对象可见。

为何存在此机制。 三个原因：

  1. 5 GB 单次 PUT 限制。 大型 HTTP 请求体处理繁琐——TCP 连接可能中断，代理层有大小限制，且没有断点续传机制。分片上传将 5 TB 的上传拆分为 1000 个 5 GB 的分块。
  2. 并行性。 由于拥塞控制和 HTTP/1.1 单流限制，单条 TCP 连接吞吐量约为 50 MB/秒。在高速链路上，10 个 100 MB 的分片并行上传可达到约 500 MB/秒。
  3. 可恢复性。 一个 100 GB 的上传在 80 GB 处失败后，可从第 17 个分片续传，而无需从头开始。

5 MB 最小值与 etag 计算。 非最终分片的大小必须 ≥ 5 MB。分片上传的 etag 为：

multipart_etag = MD5(MD5(part1) || MD5(part2) || ... || MD5(partN)) || "-N"

并非组装后对象的 MD5——这是"客户端校验和与 etag 不匹配"错误的常见根源。

失败上传产生的垃圾数据。 发起一个分片上传，却从未 Complete 或 Abort → 分片永久暂存 ，，你将持续为此付出存储费用。常见的成本泄漏：存在缺陷的重试逻辑每次都开启新的分片上传而非续传。使用 ListMultipartUploads进行检测；通过生命周期策略自动终止超过 N 天的分片上传。

4.5 预签名 URL：基于 HMAC 签名的查询参数，用于限时访问

预签名 URL 是携带加密签名查询参数的 S3 URL。任何持有该 URL 的人均可在不提供自己 AWS 凭证的情况下执行其签名的操作（GET、PUT、DELETE），直至过期。

机制（S3 v4 签名，简化版）。

1. Application has (access_key, secret_key) and decides:
       op=PUT, bucket=uploads, key=user-42/avatar.jpg,
       expires=900s, headers=Content-Type: image/jpeg

2. Build canonical_request from (method, URI, query params, headers, payload-hash).
3. Build string_to_sign = "AWS4-HMAC-SHA256\n<date>\n<scope>\nSHA256(canonical_request)"
4. Derive signing key by HMAC-chain:
       date_key    = HMAC-SHA256("AWS4"+secret_key, "20260522")
       region_key  = HMAC-SHA256(date_key, "us-east-1")
       service_key = HMAC-SHA256(region_key, "s3")
       signing_key = HMAC-SHA256(service_key, "aws4_request")
5. signature = HMAC-SHA256(signing_key, string_to_sign)
6. Build URL:
       https://uploads.s3.amazonaws.com/user-42/avatar.jpg
         ?X-Amz-Algorithm=AWS4-HMAC-SHA256
         &X-Amz-Credential=AKIA.../20260522/us-east-1/s3/aws4_request
         &X-Amz-Date=20260522T143000Z
         &X-Amz-Expires=900
         &X-Amz-SignedHeaders=host;content-type
         &X-Amz-Signature=<hex>
7. Client PUTs directly to S3, never touching the app server.

S3 校验：规范请求相同、待签字符串相同、签名密钥相同（S3 能访问 secret_key 的派生值）。签名匹配且未过期 → 允许访问。

为何如此实用。 若不使用预签名 URL，每次上传都须经过应用服务器（客户端 → 应用 → S3）；10 MB × 10k 并发上传会使应用服务器带宽饱和。使用预签名 URL 后，流程变为：客户端 → 应用（200 字节签名 URL）→ 客户端 → 直接上传至 S3。这正是 Notion、Slack、Discord 及所有现代 Web 应用处理大文件上传的方式。

安全特性。 该 URL 是 即凭证——任何持有者在过期前均拥有相应权限。仅通过 HTTPS 传递；切勿通过邮件发送。保持较短的过期时间（上传 15 分钟，下载 1 小时）。签名包含 HTTP 方法——以 GET 签名的 URL 不能用于 PUT。

4.6 "列表操作具有最终一致性"的历史

2020 年 12 月之前，S3 以令人困惑著称：

  • PUT 写入新对象 → 强一致性。
  • PUT 覆盖写 → 最终一致性。GET 可能在数秒内返回旧值或新值。
  • DELETE → 最终一致性。GET 仍可能读取到已删除的对象。
  • LIST → 最终一致性。新对象可能在数秒内不出现在列表中。

这导致了一类 Bug：写入 → 列举确认 → 看到空列表 → 重试 → 重复写入。Hadoop/Spark 花费多年时间构建变通方案（S3Guard、EMRFS 一致视图），因为"我写了一个文件，就应该能看到它"的预期被打破了。

2020 年 12 月： AWS 发布了 所有操作的强写后读一致性。GCS 和 Azure Blob 自发布起（约 2010 年）就具备强一致性。旧代码中可能保留着 S3 专用的变通方案，现在可以移除了。

   v_xyz789               (2026-04-15, 11 KB)

All three versions stored as independent objects, each at full storage cost.

访问。版本控制下的 DELETE 行为如下： 普通的 DELETE （无 version-id）不会删除任何内容——它会添加一个 删除标记 作为新的当前版本。 GET 返回 404；旧版本可通过明确指定版本来读取。永久删除需要针对各 version-id 逐一操作，或设置生命周期规则。 DELETE 为何版本控制很重要。

意外删除/覆盖的恢复。 三个原因：

  • 错误来源的 → 不启用版本控制时，文件将丢失；启用版本控制后， cp 可回滚。 GET --version-id 审计与溯源。
  • "这个对象在 2026-04-12T14:00 时是什么状态？"查找当时有效的版本即可。 并发安全。
  • 基于 Etag 的条件写入（ ）可防止多个写入方竞争时发生更新丢失。PutObject 与 If-Match: <etag>成本影响。

一个 1 TB 的桶若启用版本控制且频繁更新，一年内可能累积 5 TB 的旧版本。通过生命周期规则在 30 / 90 / 365 天后清除非当前版本是标准做法。 4.8 一次具体的

Index entry: bucket=videos, key=user-42/clip-987.mp4, version-id=v_<...>, parts=[part-1-loc, ..., part-10-loc], size=250MB, etag="<multipart-etag>-10", last-modified=<now>

  • 某个分片的网络中断。 客户端重试 UploadPart （每个 (upload-id, part-number)幂等）；新的上传替换暂存的分片。

§5. 容量上限

该技术所能实现的能力，通过不同规模的场景加以说明。

初创阶段——100 GB。 拥有 1 万用户、每用户 10 MB 的 SaaS，总计 100 GB。在 S3 Standard 上约 $2.30/月。通过预签名 URL 直接上传。每分钟少量 PUT，每秒个位数的 GET。初期无需 CDN。"设置一个桶，分发签名 URL，无需操心。"接下来的瓶颈：多年内不会有。

中小规模——TB 级，CDN 前置。 快速增长的图片目录——Instagram 第一年、Pinterest 早期的 pins。约 10 TB 图片、数十亿对象，通过 CloudFront / Cloudflare 提供服务，缓存命中率 >95%。源站承受数千次 GET/秒；CDN 处理其余请求。约 $230/月存储+带宽。瓶颈：病毒式事件导致缓存未命中率骤增，S3 对热门前缀进行限流——通过哈希前缀键名解决。

中等规模——Dropbox、Notion、Discord 达到 PB 级。 数十 TB 到数 PB。每秒数百次持续 PUT，每秒数千次到源站的 GET。按地区或租户划分多个桶。大量使用分片上传、预签名 URL 用于浏览器上传、CDN 用于读取。$1 万-$10 万/月。

大型——Netflix、YouTube、Instagram 目录。 Netflix 目录：个位数 PB 的主文件 + 数十 PB 的编码变体。YouTube 存储量达到 EB 级别。Instagram 截至 2017 年报告有 600 亿张照片（现已超过 2000 亿张）；Pinterest 超过 1000 亿个 pins。在 S3/GCS/自有存储上达到多 EB。每秒数百万次 GET的聚合请求。自建 CDN（Netflix Open Connect）在大多数读取请求到达对象存储之前将其吸收。

超大规模——S3 自身。 公开数据：超过 2800 亿个对象 。 每秒 1 亿+请求 的聚合（2023 年）。跨多个地区、每个地区多个 AZ、每个 AZ 数千个存储节点的 EB 级数据。

从每月几美分的 100 GB 到每秒百万级 QPS 的 EB——跨越了 六个数量级。同一套 API 和存储类既服务于个人项目，也支撑着 Netflix 目录。可扩展的是：前缀分散策略、CDN 部署方式、生命周期管理策略。协议本身则无需改变。

§7. 对象存储固有的难题

六个基本挑战。每一个都与使用场景无关，始终存在；举例来自各个领域。

7.1 前缀热点问题

一句话总结。 写入集中在某一前缀下，使单个内部分片饱和并遭到限流。

出现场景。 Hive 风格的数据湖按小时写入分区： year=2026/month=05/day=22/hour=14/part-000-of-10000.parquet。日志流水线： logs/2026/05/22/14/<host>.json.gz。点击流： events/yyyy/mm/dd/hh/<batch>.avro.

为何会崩溃。 每个键都以约 20 个相同字符开头。哈希后它们落入同一分片。在 6000 次写入/秒时 → 503 SlowDown；Spark 速度降低 5 倍。AWS 支持建议"分散你的前缀"。

修复方案：哈希前缀。 在前面加上 MD5(filename)[:4]:

s3://lake/a3f7/year=2026/month=05/day=22/hour=14/part-000.parquet
s3://lake/b9e2/year=2026/month=05/day=22/hour=14/part-001.parquet

四个十六进制字符 → 65,536 个分片 → 桶吞吐量实际上无上限。现代表格格式（Iceberg、Delta、Hudi）会自动在路径中加入哈希段。S3 也会在数分钟到数小时内自动分裂长期存在的热前缀；对于突发工作负载，哈希前缀是唯一的修复手段。

7.2 列举操作引发的成本爆炸

一句话总结。 朴素的想法 ls 对庞大的目录树进行操作速度慢、成本高，往往成为瓶颈。

出现场景。 备份轮转任务执行 aws s3 ls s3://bucket/backups/ --recursive。数据湖规划器列出分区内的所有文件。监控脚本每分钟执行 s3 ls 。 ListObjectsV2 每次调用返回约 1000 个键；对于 1 亿个键，需要 10 万次调用。

为何会崩溃。 LIST 每千次调用费用 $0.005 → 全量扫描一次 $0.50，若每分钟执行一次则每月 $720。执行时间受调用速率限制——以 100 次/秒的速度，每次全量扫描耗时 17 分钟。任何等待此操作的任务都会被阻塞。

修复方案：使用辅助索引，而非 LIST。

  • 数据库索引。 你的事务型数据库已经为每个对象键保有记录，直接查询即可。
  • S3 Inventory。 S3 以约 $0.0025/百万对象的价格每日/每周发布 CSV/Parquet 格式的清单。读取清单，而非实时列举。
  • Iceberg / Delta / Hudi 清单文件。 表格格式自行维护文件列表。

原则： LIST 用于临时探索，不应出现在热路径上。 若你对非无关紧要的前缀每小时列举超过数次，说明你的做法有误。

7.3 跨地区一致性与延迟

一句话总结。 跨地区复制是异步的、最终一致的，会增加数十至数百毫秒的延迟。

出现场景。 某全球公司将上传存储于 us-east-1，通过 CDN 为欧洲提供服务；缓存未命中时需承受约 80 ms 的 RTT。DR 配置要求：写入 us-east-1 必须快速同步至 us-west-2。合规要求跨地区复制数据湖。

为何会崩溃。 S3 CRR 是异步的——在 us-east-1 写入 PUT 后，us-west-2 会滞后数分钟（使用 S3 Replication Time Control / RTC 的 SLA 为 15 分钟）。在 us-east-1 写入 GET 后立即在 us-west-2 进行 PUT 可能返回 404。不支持原生多主写入。

修复方案。

  • 单一数据源 + CDN 扇出。 写入单一地区；CloudFront 在全球缓存读取请求。缓存未命中时承受跨地区延迟。适用于静态资源的默认方案。
  • 主动-主动 + 无冲突命名。 每个地区写入自己的桶/前缀；元数据数据库记录每个对象归属哪个地区。故障转移时重放指针。
  • S3 MRAP（Multi-Region Access Point）。 通过多地区副本暴露单一端点；跨地区最终一致，写入采用最后写入优先策略。适用于读密集型全球工作负载。
  • GCS 双/多地区桶。 原生复制；任意地区均可以强一致性进行读写，写入时需承受约 50-100 ms 的跨地区仲裁延迟。

没有完美答案；选择由哪个地区承担代价。

7.4 生命周期管理——正确设置存储层

一句话总结。 数据向更低成本层迁移是一个反复出现的成本问题；策略错误意味着资金浪费或失去快速访问能力。

出现场景。 某 SaaS 将所有数据保存在 Standard 层，费用为 $0.023/GB；100 TB 每月 $2300，但 90% 的数据在 6 个月内从未被访问。备份保留 30 天的每日快照于 Standard 层；但实际上只有最新的快照会被读取。ML 数据集热用期为 2 周，之后变为冷参考数据。

为何会崩溃。 两种误区：

  • "将所有数据保持在热层。" 1 PB Standard = 每月 $23k vs Deep Archive 每月 $1k — 每年 $264k。
  • "把所有东西都迁移到 Glacier。" Deep Archive 恢复需要 12 小时；批量检索费用 $0.02/GB → 恢复 1 TB 需要 $20。

修复方案：分层策略。

0-30 days:     S3 Standard ($0.023/GB)
30-90 days:    S3 Standard-IA ($0.0125/GB)
90-365 days:   S3 Glacier Instant Retrieval
365 days+:     S3 Glacier Deep Archive

Intelligent-Tiering 可自动做出决策：S3 监控访问情况并在各层之间移动对象。小额监控费用（$0.0025/1000 个对象/月，超过 128 KB 免收）。

边界情况。 最短存储时长（IA 30 天、Glacier Instant 90 天、Deep Archive 180 天）——将一个仅存放 2 天的对象放入 Glacier，费用反而高于 Standard。生命周期规则按前缀或标签应用，而非针对单个对象。版本控制存在交互影响：对非当前版本设置的规则会清除旧版本。

7.5 权限与分层访问模型

一句话总结。 IAM 策略（基于身份）、桶策略（基于资源）、ACL（旧版）以及预签名 URL（委托授权），层层叠加。错误配置是最常见的云数据泄露原因。

  • 服务使用 IAM，跨账户访问使用存储桶策略，禁用 ACL。

§9. 为什么不直接将文件存储在文件系统上

对象存储的最直接替代方案是"一台挂载大磁盘的 Linux 机器"。下面逐一分析这种方案为何行不通。

设置。 一家拥有用户上传功能的初创公司。他们搭建了一台挂载 10 TB EBS 卷的 EC2 实例，并将其挂载于 /data，以及 cp 将上传的文件存放于此。应用程序通过 nginx 静态文件服务提供文件访问。

故障模式 1：单点故障。 EC2 实例宕机后，EBS 卷本身完好（属于独立故障域），但没有机器来挂载并提供服务。恢复步骤：启动新实例、挂载卷、重启 nginx。如果有自动化脚本，停机时间约为 5-30 分钟；否则可能长达数小时。

故障模式 2：持久性。 EBS 卷的年度持久性为 99.9999%——五个九。与 S3 的十一个九相比，差了 100,000 倍。在运行 1000 个卷的 10 年周期内，预计损失：1 个卷。卷丢失 = 数据丢失（除非你有备份，而备份需要你自己配置）。

故障模式 3：容量。 磁盘写满。扩展 EBS 需要对卷进行扩容（现代 EBS 支持在线扩容，无需停机）并扩展文件系统，但最终你会触及单卷上限（gp3 约 16 TB；io2 约 64 TB）。届时你需要多个卷，于是得手动分片——将用户分配到各个卷，记录哪个卷存放哪个文件，并做负载均衡。

故障模式 4：复制。 EBS 是单 AZ 的。要应对 AZ 故障，需要跨 AZ 复制；要应对 region 故障，则需要跨 region 复制。自己实现的话 = rsync 用 cronjob 同步到另一个 AZ；延迟取决于你设置的 cron 间隔；一致性是"rsync 同步到什么程度算什么程度"。S3 默认提供此能力，跨 AZ 的 RPO=0。

故障模式 5：读扩展。 在 EBS 上用一台 nginx 机器处理 1000 次/秒的读请求没问题。100,000 次/秒则需要多台机器，但它们都要读取相同的文件。此时你需要共享文件系统（NFS、EFS）或将文件复制到各处。无论哪种方式，延迟上升、复杂度上升、成本上升。

§10. 扩展维度

类型四：从单区域扩展到多区域

延迟敏感的全球用户；合规要求（数据驻留）；灾难恢复。

修复方案。 CRR（Cross-Region Replication）。

对象存储的应对方式。 无需干预。S3 / GCS / Azure Blob 自动扩展：对象增多 → 内部分片增加（S3 自动拆分前缀）；请求增多 → 在更多前端节点间负载均衡；存储增多 → 自动分配更多底层存储容量。用户感知不到任何变化。

潜在的隐患。

  • 桶级吞吐量限制。 文档限制（每前缀每秒 3500 次 GET、5500 次 PUT）按前缀计算。持续、分布均匀的工作负载在桶维度可达到 100k+ QPS。
  • List 操作。 随着对象数量增长，列举时间线性增加。请改用 inventory 或表格格式的清单。
  • Cost-of-LIST 与 Cost-of-PUT。 在数十亿次操作规模下，单次调用费用会累积可观。尽量使用批量操作。

拐点。 存储容量没有拐点（实际上无上限）。对于 QPS，拐点是"我们在单个前缀上持续达到 5000 QPS 并被限流"——对键进行哈希前缀处理。对于列举操作，"LIST 耗时已超过实际处理时间"——切换到旁表索引。

类型 2：单前缀热点 — 哈希前缀

修复方案。 一个日志管道将应用日志写入 logs/2026/05/22/14/。每小时整点是热前缀；每小时 100% 的写入都集中在一个前缀上，桶的其余部分处于冷状态。

对象存储的应对方式。 S3 的自动分裂会对持续热前缀生效，但延迟以分钟到小时计。对于突发型负载（每个整点都会创建新前缀），自动分裂根本来不及跟上。

真正的解决方案。 哈希前缀。详见 §4.3 和 §7.1。

拐点。 当每个前缀的持续写入速率超过约 3000/秒时，哈希前缀处理变为强制要求。突发速率超过约 5500/秒时将被限流。

类型 3：从单桶扩展到多桶

修复方案。 你的应用只有一个 bucket；生命周期和访问模式混杂在一起（热点用户上传、冷备份、多租户数据）。Bucket 策略和生命周期规则变得难以管理。

修复方案。 按访问模式拆分为多个 bucket： user-uploads-prod, backups-prod, analytics-data-lake-prod, model-artifacts-prod。每个 bucket 拥有独立的生命周期、加密设置、访问控制和标签。成本归因更清晰（按 bucket 打标签）；审计更简洁；权限管理更容易。

限制。 AWS 账号默认上限为每账号 100 个 bucket（可申请提升至 1000）；bucket 名称全局唯一。

类型 4：从单区域扩展到多区域

修复方案。 延迟敏感的全球用户；合规要求（数据驻留）；灾难恢复。

修复方案。

  • CRR (Cross-Region Replication)。 将对象从源 bucket 异步复制到一个或多个目标 bucket。DR 的标准方案。从源到目标的延迟：分钟级（RTC 层承诺 15 分钟 SLA）。
  • Multi-Region Access Point (MRAP)。 单一端点覆盖多个区域的存储桶，适用于读密集型全球分布式访问模式。
  • GCS 双区域 / 多区域存储桶。 原生副本复制；存储桶分布在两个区域或多区域地理范围内，支持对任意区域进行读写，并保证强一致性，代价是写操作存在跨区域延迟。

拐点。 当 CDN 缓存未命中时的跨区域 GET 延迟超出可接受的用户延迟（媒体密集型应用通常为 50–100ms）时；或当合规要求涉及数据驻留地时。

  • 文件存储： 适用场景：需要在多台主机间共享 POSIX 语义（open / read / write / lock）。默认选择：HPC 集群、遗留应用程序、频繁更新的共享配置。

  • 数据库 BLOB 列： 适用场景：blob 极小（<10 KB）且逻辑上属于频繁查询的行。即便如此，仍建议在行中存储 URL，将 blob 存入对象存储；数据库缓冲池对 blob 而言过于宝贵。

具体阈值。

  • 附加于 100 万条用户行的 5 KB blob：使用 BLOB 列可能可以。
  • 50 KB 图片，每年被访问 10 亿次：对象存储 + CDN。
  • 5 MB 文档，附加于某用户：对象存储。
  • 5 GB ML 模型文件：对象存储，毫无疑问。
  • 来自单台服务器每小时 10 GB 的日志流：使用对象存储并批量写入（不要每行日志对应一次 PUT）。
  • 以 10 万 QPS 进行 50 KB 随机读取：应使用缓存（Redis / Memcached / DragonflyDB），而非对象存储。

对象存储按存储类计费。在对象生命周期中选择合适的存储类是最重要的成本控制手段之一。

经济分析。

  • S3 Standard 每 GB/月 $0.023 是默认的热层，用于所有活跃提供的内容。
  • Standard-IA 每 GB/月 $0.0125 将存储成本降低一半，但每检索 1 GB 收取 $0.01。如果每月每 GB 读取次数少于 0.5 次则划算，否则检索费用会抵消节省的存储费用。
  • One Zone-IA 每 GB/月 $0.01 比 IA 便宜 20%，但仅限单可用区。若该可用区丢失则数据永久消失。仅适用于可重建的数据（从主文件转码的变体、派生的分析结果），即丢失可恢复的场景。
  • Glacier Instant 每 GB/月 $0.004 比 Standard 便宜约 5 倍，延迟为毫秒级。专为"偶尔需要快速访问"的场景设计，如季度合规报告、访问极少的用户历史数据。
  • Glacier Flexible 每 GB/月 $0.0036 检索时间为分钟至小时级，比 Standard 便宜约 6 倍。适用于每月或更低频率的访问模式，常用于备份。
  • Glacier Deep Archive 每 GB/月 $0.00099 比 Standard 便宜 23 倍，但检索需要 12 小时。用于纯归档、合规保留（7 年财务记录、数十年医疗记录），即必须保存但真心不希望读取的数据。

生命周期策略

生命周期策略是一种声明式描述，用于定义何时在分层之间迁移对象以及何时删除对象。示例：

{
  "Rules": [{
    "ID": "user-uploads-aging",
    "Filter": { "Prefix": "user-uploads/" },
    "Status": "Enabled",
    "Transitions": [
      { "Days": 30,  "StorageClass": "STANDARD_IA" },
      { "Days": 90,  "StorageClass": "GLACIER_IR" },
      { "Days": 365, "StorageClass": "DEEP_ARCHIVE" }
    ],
    "Expiration": { "Days": 2555 }   // 7 years
  }]
}

此规则： user-uploads/ 前缀下的对象在 30 天后迁移至 IA，90 天后迁移至 Glacier Instant，365 天后迁移至 Deep Archive，7 年后删除。

注意事项。

  • 迁移是需要收费的。 每次迁移都是按对象计费的操作；对于 10 亿个对象，迁移成本不容忽视。
  • 最短存储期。 将一个仅存储 10 天的对象迁移至 IA，仍会按 30 天最短存储期收费。
  • 生命周期策略按桶配置，基于前缀或标签生效。 精细化管控需要合理规划桶或标签的组织结构。

"全部保持热层"的成本灾难

一种常见的情形：一家初创公司在五年内从 100 GB 增长至 1 PB，由于"以后可能会用到"，所有数据都存在 S3 Standard 中。存储费用：1,000,000 GB × $0.023 = $23,000/月 = $276k/年。

如果其中 90% 每年访问不足一次：

100 TB hot in Standard: 100,000 × $0.023 = $2,300/mo
900 TB in Glacier Flexible: 900,000 × $0.0036 = $3,240/mo
Total: $5,540/mo (down from $23,000/mo)
Savings: $17,460/mo = $209k/yr

只需花几个小时撰写生命周期策略，每年可节省 20 万美元。这是对象存储中最大的单项成本控制手段。

访问模式未知时使用 Intelligent-Tiering

S3 Intelligent-Tiering 自动化分层决策：S3 监控每个对象的访问情况，并在高频访问 → 低频访问 → 即时归档 → 归档 → 深度归档访问层之间自动迁移。监控费用：每月每 1000 个对象 $0.0025（超过 128 KB 的对象免收此费）。

何时使用：对象数量多且访问模式不可预测（用户上传中部分内容爆火而其他长期沉寂；数据湖中大多数文件在摄入时被读取一次后再无访问，但少数成为"参考"文件）。不值得使用的情形：若已知访问模式，手动策略更经济。

§13. 安全

在云计算时代，对象存储是数据泄露事件中曝光度最高的载体。做好安全防护需要理解多个相互叠加的机制。

13.1 IAM、桶策略、ACL、预签名 URL

四个层次（摘自 §7.5）：

  • IAM（身份与访问管理）。 基于身份。"此用户/角色可以对这些资源执行这些操作。"存储于 IAM 服务中，附加于用户/角色/群组。
  • 桶策略。 基于资源。"这些主体（通过 IAM ARN 或 AWS 账户 ID 标识）可以对此桶执行这些操作。"存储于桶上。
  • ACL（访问控制列表）。 旧版的按对象或按桶授权方式，在新桶中默认已弃用（桶所有者强制模式）。
  • 预签名 URL。 嵌入于 URL 中的有时间限制的签名授权。

评估逻辑。 若（IAM 策略、桶策略、ACL）中任一明确允许，且无策略明确拒绝（拒绝优先），则请求被允许。公共访问还受到"阻止公共访问"设置（账户级别 + 桶级别）的控制。

最佳实践姿态。

  • 默认拒绝原则。 除非明确授权，否则一切访问均被拒绝。
  • 使用 IAM 进行服务间鉴权。 将应用程序角色附加到 EC2 / ECS（弹性容器服务）/ Lambda，IAM 策略精确授予所需操作。
  • 使用桶策略实现跨账户访问。 在 AWS 账户间共享数据时，在桶策略中指定账户。
  • 不要使用 ACL。 通过桶所有者强制模式禁用 ACL。
  • 账户级别阻止公共访问。 默认开启，防止任何策略将桶设为公开。

13.2 预签名 URL 与鉴权卸载

预签名 URL（§4.5）是客户端直接上传/下载的核心机制。应用服务器对用户进行鉴权，判断"是的，该用户可以上传至 users/<id>/avatar.jpg，"生成签名 URL 并返回。客户端 PUT直接向 S3 发送请求。鉴权仅在生成 URL 时发生一次，S3 在收到请求时验证签名。

特性。

  • URL 即凭证。 任何持有该 URL 的人均可在过期前执行相应操作。
  • 有时间限制。 典型设置：上传 15 分钟，下载 1 小时。不要设置超过必要的时长。
  • 操作和资源专属。 GET 签名的 URL 不能用于 PUT；针对 key1 的 URL 无法访问 key2.
  • 可固定请求头。 固定 Content-Type、Content-MD5，防止客户端上传其他内容。

13.3 服务端加密

S3 服务端加密的三种变体：

  • SSE-S3（默认）。 S3 管理密钥。每个对象使用唯一数据密钥静态加密，数据密钥由 S3 托管的主密钥加密。透明处理：客户端只需 PUT，S3 在接收时加密、在 GET时解密。无需修改代码。对于"我们希望静态加密以满足合规要求"且无需自行管理密钥的场景已足够。
  • SSE-KMS（KMS 托管）。 S3 使用客户在 AWS KMS（密钥管理服务）中控制的 CMK（客户主密钥）。每个对象使用唯一数据密钥加密，数据密钥由 KMS CMK 加密。特性：CloudTrail 中有审计追踪（每次加密/解密均有日志），KMS 访问策略（撤销密钥访问以锁定读取），密钥轮换。权衡：KMS API 调用需要收费（约 $0.03/10000 次），因此大量读取的工作负载需承担可观的附加费用。
  • SSE-C（客户提供密钥）。 客户端随每次请求提供加密密钥，S3 在接收时使用其加密后丢弃。读取时客户端需提供相同密钥。特性：密钥不由 AWS 存储（使用后即丢弃），但客户端必须自行管理密钥——丢失密钥即永久丢失数据。属于小众用法，较少使用。

13.4 客户端加密

最严格的安全姿态：在上传前加密。云服务商只看到密文。

Client:  data → AES-256(data, client_key) → ciphertext
         PUT ciphertext to S3
S3:      stores ciphertext (which is also SSE-encrypted at rest)
Client:  GET ciphertext from S3
         decrypt with client_key

何时使用。 当云服务商不得读取数据时——极端监管环境（部分医疗、部分金融），或出于安全谨慎考虑。备份工具（Borg、Restic、Tarsnap）在将客户备份上传至 S3 时即采用此方式，云服务商只看到加密后的 blob。

权衡。 无法对加密数据执行服务端处理（S3 Select、Athena 等）。密钥管理责任在您；丢失密钥则丢失一切。

13.5 VPC 端点 / Private Link

默认情况下，EC2 到 S3 的流量经由公共互联网传输（通过 TLS 加密，但仍走公网路由）。使用 S3 的 VPC 端点后，流量将在 AWS 骨干网内传输：

EC2 in VPC → VPC Gateway Endpoint for S3 → S3 in same region
(no internet gateway, no public IP, no NAT)

特性。

  • 网络隔离。 即使攻击者获取了您的 IAM 凭证，只要桶策略限制仅允许来自您的 VPC 端点的访问，他们就无法从您的 VPC 外部访问 S3。
  • 无出站数据传输费用。 通过 VPC Endpoint 访问 S3 是免费的（同一区域内）。
  • 更低的延迟 （通过避免公共互联网路由，节省约 5-15ms）。
  • 合规性 适用于要求"禁止公共互联网流量"的环境。

在注重安全的生产环境部署中，VPC endpoint 基本上是强制要求的。

13.6 Object Lock 与 WORM（一次写入多次读取）合规

某些法规要求数据不可变：财务记录（SEC Rule 17a-4）、法律保留文件、医疗审计日志。

Object Lock。 一旦对象被锁定，任何人——包括存储桶所有者，乃至 root 账户——在锁定到期之前都无法删除或修改它。

模式。

  • Governance 模式。 已锁定，但特殊 IAM 主体可以覆盖。
  • Compliance 模式。 已锁定，无法覆盖，无例外。必须等待保留期到期。

用途。 银行账本、受保留规则约束的审计日志、医疗记录，以及任何"我们承诺在 N 年内不修改此数据"具有法律约束力的场景。

13.7 威胁模型回顾

  • 意外公开暴露。 通过"阻止公共访问"功能缓解。
  • 凭证泄露。 通过短期 IAM 凭证（STS — Security Token Service）、MFA 和 CloudTrail 监控来缓解。
  • 预签名 URL 泄露。 通过设置较短的过期时间来缓解。
  • 内部人员删除。 对于合规数据，通过版本控制 + MFA Delete + Object Lock 来缓解。
  • 云服务商遭到入侵。 通过客户端加密来缓解。
  • 存储桶遭受勒索软件攻击。 通过版本控制（若启用了 MFA Delete，攻击者实际上无法删除版本）以及跨账户备份（独立的"保险库"账户，生产账户对其无访问权限）来缓解。

§14. 使用场景展示

对象存储在截然不同领域的五种典型应用场景。

14.1 静态资源托管（网站、图片、JS 包）

模式。 消费类 Web 应用——Shopify 店面、Substack 刊物、SaaS 营销网站——在全球范围内提供静态资源（图片、CSS、JS 包、字体）。

架构。 对象存储作为源站；CDN 置于前端；资源通过文件名中的哈希值进行版本化（main-a8f29c.js）因此可以永久缓存。构建时上传新版本；旧 URL 永不失效，因为旧的哈希值不会再次出现。

适用原因。 字节只需上传一次，即可被服务数十亿次。由于缓存命中率超过 99%，源站延迟无关紧要。成本在于 CDN 出流量，而非 S3 存储。URL 哈希版本化使缓存失效不再是问题。

数据规模。 中型网站：500 GB 资源，每月 5000 万次请求到 CDN，每月 100 万次请求到源站（2% 未命中）。S3：$11.50 存储费 + 极少出流量费。CloudFront：约 $4,250 出流量费。若不使用 S3+CDN，EC2 上的 nginx 需要多台服务器，且在流量激增时会崩溃。

14.2 视频与媒体存储（Netflix 级、YouTube 级）

模式。 流媒体服务存储原始视频文件及转码后的变体（分辨率 × 编码格式 × 码率）。Netflix：约 17,000 部内容，每部有数十种编码；YouTube：数十亿个视频。

架构。 原始文件存于冷层；变体文件较热，热度与流行度成正比；HLS（HTTP Live Streaming）或 DASH（Dynamic Adaptive Streaming over HTTP）切片，每段 2-10 秒/几 MB。客户端顺序请求切片；CDN 边缘节点缓存热门切片。

适用原因。 字节量大，"单次读取整个切片"的访问模式，持久性至关重要（原始文件重新生成代价高昂），分层存储必不可少（2010 年的影片存于 Glacier；新上映的内容存于 Standard）。

数据规模。 Netflix Open Connect 在峰值时段承载约 30% 的互联网流量。内容目录规模为 10-100 PB。流行度符合帕累托分布，少量内容占主导；其余内容处于冷存储状态。

14.3 数据湖（对象存储上的 Iceberg / Delta / Hudi 表）

模式。 分析 / ML 组织将 PB 级结构化数据（事件、交易、日志）以 Parquet 格式存储在对象存储中。Apache Iceberg、Delta Lake、Apache Hudi 通过清单文件在不可变 Parquet 之上提供类 ACID 语义。

架构。 s3://lake/<db>/<table>/data/<hash>/file-<id>.parquet。Catalog 元数据 s3://lake/<db>/<table>/metadata/v123.metadata.json 原子性地指向活跃文件集。写入操作以追加方式进行，并原子性地重写 catalog。Compaction 合并小文件。

适用原因。 多 PB 的数据量；以追加为主的读取模式；Trino / Spark / Flink / DuckDB / BigQuery / Snowflake 均可原生读取 S3 上的 Parquet；成本比数据仓库存储低几个数量级。Iceberg catalog + Parquet 数据 = 经典的"元数据存数据库 + 字节存对象存储"模式。

数据规模。 Uber 数据湖：约 250 PB 存储于 HDFS + S3，数十万张表。Netflix：PB 级 Iceberg on S3，所有分析均构建于此。Apple、Airbnb、Stripe 均采用此模式。

14.4 ML 模型存储与训练数据

模式。 ML 团队将训练数据集（TB 到 PB 级的图像、文本、音频、视频）和模型产物（检查点、最终模型、微调变体）存储在对象存储中。训练任务从 S3/GCS 读取，将检查点写回。推理服务从 S3/GCS 加载模型。

架构。 数据集存于前缀目录（s3://datasets/laion-5b/v2/shards/）。版本化模型注册表（s3://models/recommender/v23/model.pt）。训练任务通过 PyTorch 的 S3DataLoader 或 TensorFlow 的 tf.data 使用 S3 URL 进行并行读取。LAION-5B、Common Crawl：全量数据存储于对象存储，规模达多 PB。

适用原因。 大型且不频繁变更的字节数据；读取可并行化（来自众多 GPU 的 1000 个文件同时读取）；PB 级成本管控；版本化契合实验迭代周期。

数据规模。 OpenAI GPT 训练数据：多 PB 级。Meta LLaMA：多 PB 级数据集，多 TB 级检查点。Hugging Face hub：数万个多 GB 检查点存储于 S3，供所有运行 from transformers import AutoModel.

14.5 备份与归档

模式。 数据库备份、应用日志、系统快照持续流式传输至对象存储。绝大多数情况下从不读取；偶发的 DR（灾难恢复）事件只会读取极小的一部分。

架构。 加密转储至 s3://backups-prod/postgres/db-name/2026-05-22/。生命周期策略在 30 天后将旧数据迁移至 Glacier。偏执级别：跨账户或跨云备份目标（独立的 AWS 账户，甚至是 AWS 之外的 Backblaze B2，以确保生产环境被入侵时无法触达备份）。

适用原因。 成本低廉、持久性高、可保留多年、易于加密、可通过生命周期策略迁移至超低成本层。DR 场景下可接受的恢复延迟。

数据规模。 1 TB 数据库，每日全量备份 + 每小时增量备份，7 年保留期：使用生命周期策略约 $60-100/月，不使用则约 $1500-3000/月。

14.6 应用文件上传（Dropbox、Notion、Slack）

模式。 用户上传文件（文档、附件、照片、视频）。应用将字节存入对象存储，将元数据（所有者、共享信息、文件名、MIME 类型）存入数据库，并通过预签名 URL 提供访问。

架构。 §6 中的经典架构图。

适用原因。 用户控制、大小不可预测；多租户；CDN 用于共享/公开文件，预签名 URL 用于私有文件；生命周期策略处理回收站。

数据规模。 Notion：数百 TB。Slack：PB 级。Dropbox：EB 级（为降低成本，Magic Pocket 替代了 S3）。均采用"元数据存数据库 + 字节存对象存储"这一相同模式。

§15. 真实世界中的实现与数字

大规模落地对象存储的知名系统。

AWS S3。 参考实现。公开数据：存储对象超过 280 万亿个；每秒聚合请求超过 1 亿次；数据量达 EB 级。覆盖数十个区域，每个区域含多个可用区。是衡量"对象存储能力上限"的基准。

Netflix Open Connect + S3 源站。 Netflix 将 S3 作为视频文件的源站；Open Connect（其自建 CDN，在全球 ISP 内部署服务器）承载绝大多数字节传输。内容目录（原始文件 + 编码变体）估计为 10-100 PB。全球晚高峰期间的峰值网络吞吐量约占互联网带宽的 30%。

Dropbox Magic Pocket。 Dropbox 最初托管于 AWS S3，到 2016 年已达约 500-600 PB，随后为降低成本自建了对象存储系统"Magic Pocket"。目前规模达 EB 级，全部运行于商用硬件（SMR 磁盘、纠删码、定制控制器）。他们并非迁移"回到文件系统"，而是从零构建了自己的 S3。关于 Magic Pocket 的公开演讲是研究这一规模下系统架构的极佳资料。

Pinterest 图片。 据近期报告，约有 3000 亿个 pins，每个 pin 有多种图片分辨率存储于 S3。PB 级规模；读取频率高（每天数十亿次 GET 请求），主要通过 CloudFront 和内部缓存层提供服务。

GitHub Large File Storage（LFS）。 Git 仓库中的大型文件（二进制文件、数据集、模型）存储于 S3，Git 历史中仅保存指针。数亿个仓库中存储了数十 PB 的数据。

Cloudflare R2。 于 2022 年正式推出，明确定位于颠覆 S3 的出流量定价模式。据报告，上线一年内存储量达数十 PB，拥有数千名客户。技术层面类似（纠删码、兼容 S3 的 API）；商业创新在于"不收取出流量费，因为更广泛的 CDN 业务已覆盖带宽成本"。

Facebook Haystack 与 f4。 Meta 的图片存储系统。Haystack 是最初方案（约 2010 年代），是一个自定义 blob 存储，针对将多张照片存储于单个大文件进行了优化。f4 是冷存储层，采用极度纠删码方案（例如在机架内使用 (10, 4)，跨机架使用 (3, 1) 以实现双重容错），以更低成本存储温热但极少读取的照片。规模约 100 PB。

Microsoft Azure Storage。 Azure Blob Storage 支撑了从 Azure 虚拟机（Page blobs 作为 VHD）到 Office 365（大量文档存储）再到消费者版 OneDrive 的一切服务。数据量达数 EB 级。

OpenAI / Anthropic / Hugging Face 模型托管。 基础模型权重（每个模型从数 GB 到 TB 不等）存储于 S3 / GCS / Azure Blob，用于服务和分发。Hugging Face CDN 前置的模型 hub 提供数十亿次模型下载。

Snowflake / Databricks 数据湖。 Snowflake 的存储层构建于云对象存储之上（S3、GCS、Azure Blob）。Databricks Lakehouse 架构：Delta Lake on S3。每位客户的数据规模达 PB 级。

规模之悬殊令人叹为观止。从一个存储 10 GB 数据于 S3 的个人博客，到 Meta f4 存储 100 PB，同一个概念原语——键、字节、持久性——实现了如此跨越。

§16. 总结

对象存储是全球规模的不透明字节倾卸场：一个扁平的键到 blob 映射，具备 11 个 9 的持久性、强读后写一致性、近乎无限的容量，但延迟为 50-150ms，且语义不比 GET 。 PUT。各变体（S3、GCS、Azure Blob、R2、MinIO、Ceph）在 API 上趋于一致，在定价上各有差异——出流量而非存储才是结构性成本。在内部实现上，纠删码将每个对象分散到多个可用区；前缀分片将单前缀吞吐量限制在约每秒 3500 次 GET / 5500 次 PUT；分段上传之所以存在，是因为单次 PUT 上限为 5 GB 且并行传输效果更优。经典架构模式是：数据库中存储小型元数据，指向对象存储中的大字节文件，公开读取通过 CDN 前置，私有访问由 IAM、存储桶策略和预签名 URL 把关，并通过生命周期策略将数据迁移至更低成本的层级。把握好边界（哪些数据存于此处，哪些存于数据库，哪些存于块存储），其余的就是管道工作。
