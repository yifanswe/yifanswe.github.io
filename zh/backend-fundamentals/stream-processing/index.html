关于 流处理引擎 的技术参考，涵盖整个类别——Apache Flink、Kafka Streams、Spark Structured Streaming、Apache Beam、Materialize、RisingWave、Samza。内容包括：流处理的本质、其提供的契约、检查点与精确一次语义的字节级机制、设计空间、难点问题，以及同一技术如何分别服务于支付欺诈检测、实时 ML 特征、物联网告警、产品分析和 CDC（变更数据捕获）流水线。

§1. 流处理的本质

流处理器是一种 在无界事件序列上执行持续查询、同时维护增量更新状态的运行时。查询永不结束。每个新事件都会改变内部状态，可能产生衍生事件，任何概念性聚合的结果始终是"截至我们处理的最新事件"。

以下三个属性共同定义了这一类别，并将其与相邻概念区分开来：

  1. 输入在概念上是无界的。 不存在"数据结束"。作业运行数月乃至数年。
  2. 查询是增量式的。 当事件 e 到达时，引擎计算 f(state, e) → (new_state, output)。它不会重新扫描所有之前的事件来重新计算结果。这正是流处理器具有每事件 O(1) 成本，而批处理每次刷新需要 O(N) 成本的原因。
  3. 状态是持久的且可重放的。 引擎在事件之间维护算子本地状态（计数器、窗口缓冲区、连接哈希表、ML 模型参数），以固定节奏持久化，并能从持久化日志中重放遗漏的输入，以便在崩溃后重建状态。

它解决的问题：对到达事件的低延迟分析。输出的读取方看到的聚合和衍生信号反映的是 数秒前 的输入，有时甚至低于一秒，而不需要等待下一次批处理运行。具体而言：在刷卡后 200ms 内生成欺诈风险评分；持续刷新的实时 ML 特征（"用户 U 过去 5 分钟内的点击数"）；具有数据库读取新鲜度的衍生物化视图（"每个地区的活跃会话数"）。

流处理不是什么

  • 不是消息队列。 Kafka、Pulsar、RabbitMQ 负责传输事件，它们不进行聚合、窗口化或维护衍生状态。流处理器位于队列的 上层 。
  • 不是快速运行的批处理器。 Spark 的每日批处理每次运行都要处理 O(all_data)。即使缩短到 1 分钟微批，仍然需要重新读取每个微批的输入文件。流处理器每个事件只改变一条状态记录。
  • 不是 OLTP 数据库。 数据库为众多客户端提供随机点读和写入；流处理器执行固定的查询图，很少接受随机查询（Materialize 是边界案例）。
  • 不是 OLAP 系统。 Snowflake、BigQuery、ClickHouse 对存储数据运行交互式即席查询。流处理器在到达的数据上反复运行同一个查询。
  • 不擅长：对历史数据的随机即席查询（否，请查询数据仓库）；长期运行的同步请求/响应（否，模型是异步推送）；超出源和单一 sink 边界的跨系统分布式事务（2PC 机制范围有限）。

§2. 内在保证——以及需要额外叠加的内容

流处理器的契约比人们预想的更为狭窄。"精确一次"这个短语经常被随意抛出；请准确理解什么是有保证的，什么不是。

设计层面的保证

  • 内部状态的精确一次应用语义。 引擎保证，在恢复后，引擎的 内部 状态与未发生故障时应有的状态一致。计数器不会被重复递增。窗口聚合是正确的。这通过 一致性快照 实现（Flink 中的 Chandy–Lamport 屏障，Kafka Streams 中的 Kafka 事务，Spark 中的批次 ID）。
  • 当 sink 配合时的端到端精确一次。 如果 sink 支持事务性写入（Kafka 事务性生产者、带 XA 的 JDBC、针对键控存储的幂等 upsert），引擎可以协调两阶段提交（2PC），使记录恰好可见一次。2PC 协调者通常是流处理器的协调节点（Flink 中的 JobManager），prepare/commit 与检查点屏障绑定。
  • 基于水印的事件时间处理。 引擎将 事件时间 （记录上的时间戳）与 处理时间 （算子处的挂钟时间）分离，并提供 水印 抽象——一个单调递增的声明："我认为时间戳 ≤ W 的事件不会再到达了。"这使得基于窗口的聚合在乱序到达的情况下仍然正确。
  • 崩溃后的状态持久性。 状态以可配置的节奏检查点到持久存储（S3、HDFS、GCS）。崩溃后，作业从最后一个成功的检查点恢复，不丢失状态。
  • 键内有序性。 当事件按某个键分区并由每个键的单个算子实例处理时，引擎为该键保留源的顺序。不同键可能交错。

不提供的保证

  • 不提供跨多个无关系统的分布式事务。 "精确一次"适用于引擎状态加上一个配合的 sink（以及任何加入同一 2PC 组的第二个 sink）。同时向 Kafka 写入、调用 Stripe 支付 API 并通过 Twilio 发送短信的流水线不是精确一次的。这些外部副作用必须通过幂等化（使用确定性标识符）来处理。
  • 不免费提供有界状态。 如果应用逻辑未指定 TTL（生存时间）、驱逐策略或窗口关闭，RocksDB、Hummock 或任何状态后端都会无限增长。流处理器不会神奇地遗忘数据。这是实际中流处理作业中断的头号原因。
  • 不提供任意副作用的真正精确一次。 发送 Slack 告警、写入非事务性文件、调用外部 REST 端点——这些本质上是至少一次的。引擎无法撤销已发生的操作。
  • 不提供同步请求/响应契约。 引擎是推送式的。需要"提问并获得答案"的客户端必须查询引擎填充的下游服务存储，而不是直接查询引擎。Materialize 通过提供针对流式视图的 Postgres 协议查询打破了这一规则，但代价是架构复杂性。
  • 不提供低恢复时间（无需额外配置）。 对于大状态，从 S3 检查点冷启动可能需要数分钟到数十分钟。本地恢复、基于地区的重启以及备用 task manager 都需要配置。
  • 不适合 cron 式批量作业的调度。 请使用 Airflow、Argo Workflows 或 Dagster 来处理这类需求。

最大的错误：在下游 sink 是非幂等 REST API 的情况下，声称端到端精确一次。请如实说明契约。

§3. 设计空间

流处理器沿五个正交轴进行划分：

轴 1：执行模型——微批 vs 真流

  • 微批（Spark Structured Streaming、ksqlDB 历史模式）：引擎在极短间隔（通常 100ms–1s）内积累事件，然后对该微批运行 Spark 作业。优点：丰富的 SQL、可复用 Spark 批处理优化、语义简单。缺点：延迟下限为微批间隔，无法达到 p99 低于 100ms。
  • 真流（逐事件）（Flink、Kafka Streams、Samza、Storm、Materialize、RisingWave）：每个事件在到达时立即独立处理。优点：延迟在 10–200ms 范围内；背压是逐通道的，粒度更细。缺点：语义更复杂，故障模式更难处理。

轴 2：嵌入式库 vs 集群

  • 嵌入式库（Kafka Streams）：包含在微服务中的 JAR。无需运维集群；水平扩展即"部署更多服务 pod"。状态本地存储并通过 changelog 同步回 Kafka。仅限 Kafka 作为 source 和 sink，仅支持 JVM，单集群操作。
  • 独立集群（Flink、Spark Streaming、Samza、Storm）：由 JobManager + TaskManager（或等效组件）组成的专用集群。更适合跨 source 流水线、大状态、复杂窗口化、多语言 sink。运维成本较高。

轴 3：声明式 SQL vs 命令式 DSL

  • SQL 优先（Materialize、RisingWave、ksqlDB、Flink SQL、Spark SQL on streams）：将流处理作业定义为 SQL 视图，优化器规划数据流。分析师可以编写作业。自定义逻辑的扩展性有限。
  • DSL / API（Flink DataStream、Kafka Streams DSL、Beam SDK、Storm）：用命令式代码（Java、Scala、Python）定义数据流。完全可扩展，支持自定义窗口、自定义状态、ML 库集成。学习曲线较陡。

轴 4：通用处理器 vs 物化视图数据库

  • 通用（Flink、Spark、Kafka Streams、Samza、Storm）：编写作业、部署并监控。作业输出到 sink；客户端查询 sink。
  • 物化视图流数据库（Materialize、RisingWave、Confluent 的 KSQL pull 查询）：创建流式 SQL 视图，客户端通过 Postgres 协议直接查询视图。系统内部维护增量视图结果，以流式聚合的新鲜度和数据库的便利性提供点查询服务。

轴 5：统一批流 API vs 仅流式

  • 统一（Apache Beam、Flink Table API、Spark Structured Streaming）：一次编写流水线，可在批输入或流输入上运行。适用于数据回填和在线场景需要相同业务逻辑的情况。
  • 仅流式（Storm 历史上、ksqlDB、Materialize）：针对流式优化，批量回填是单独关注点（通常通过批处理系统处理）。

§4. 字节级机制：状态后端、检查点屏障、2PC sink、事件时间

本节将引擎内部原理变得具体而非抽象。我们以 Flink 为深度参考（它拥有最明确、文档最完善的机制），并在其他引擎有差异时加以说明。

4a. 状态后端——状态实际存储在哪里

Flink（以及 Samza、Kafka Streams）面临同一个问题：数亿个不同键的键控状态，每个键有多个值，应该存在哪里？

┌────────────────────────────────────────────────────────────────┐
│ HashMapStateBackend (formerly MemoryStateBackend)              │
│  • State lives in JVM heap as Java HashMap                     │
│  • Checkpoints serialize the whole heap state                  │
│  • Use case: state < 100 MB per slot, latency critical         │
│  • Limit: ~1 GB before OOM; checkpoints block under pressure   │
└────────────────────────────────────────────────────────────────┘
┌────────────────────────────────────────────────────────────────┐
│ EmbeddedRocksDBStateBackend  ◄── default for large state       │
│  • State is RocksDB on local NVMe, off-heap                    │
│  • LSM tree: writes → memtable → flush → L0 → ... → L6 SSTs    │
│  • Checkpoints upload SST files (incremental) to S3/HDFS       │
│  • State size limited by local SSD, not heap                   │
│  • get/put cost ~10–100 µs (block cache hit) to ~1 ms (miss)   │
└────────────────────────────────────────────────────────────────┘

原因 为何使用 LSM（日志结构合并树）而非 B+ 树来管理流处理状态？主要访问模式是对同一键"先读后写"，每分钟重复数千次（每个热门键的事件触发 get → put）。LSM 以 memtable 速度吸收写入（约每核每秒 100 万次操作，纯内存）；B+ 树每次 put 需要修改页面 + WAL fsync，写入速度慢约 10 倍。读取通过块缓存和布隆过滤器摊销（热键命中率通常 >99%）。

RocksDB write path:
  state.put("user_42|window_12:00", count=7)
        │
        ▼
  [Memtable]   ◄── skip-list in RAM, sorted by key (~64 MB default)
        │   when full → immutable, flush
        ▼
  [L0 SSTs]    ── newest, may overlap key ranges
        │
        ▼ leveled compaction
  [L1, L2, ... L6 SSTs]
        │
        ▼
  Each SST: sorted blocks (~4 KB each), block index, bloom filter
            stored on local NVMe

RocksDB read path:
  state.get("user_42|window_12:00")
    1. check memtable → if hit, return
    2. check immutable memtable
    3. check L0 SSTs (newest first), bloom filter skips most
    4. check L1, L2, ... L6 SSTs (each level: at most one SST owns the key)
    5. read 4 KB block (page cache or disk)
    6. binary search within block, return value

与 B+ 树相比放弃的是 读放大。最坏情况下，一次查询需触及 memtable + 每层 1 个 SST × 7 层 = 8 次查找。布隆过滤器将平均触及次数降至约 1.1 个 SST。对于冷数据的 OLTP 点查询，B+ 树胜出；对于流处理工作负载（约 99% 写入，热键集中），LSM 以 5–10 倍的优势胜出。

Kafka Streams 和 Samza 也使用 RocksDB。Materialize 使用自研的 Persist 层，构建于对象存储之上；RisingWave 使用 Hummock，同样是 LSM，但设计为完全存储在 S3/对象存储上，从而使计算和状态可以独立扩展——这就是云原生的技巧。Spark Structured Streaming 以 Parquet 类文件在 HDFS 上存储状态，效率较低但更简单。

4b. Chandy–Lamport 快照算法——最核心的概念

问题： 一个分布式数据流，N 个算子分布在 M 台机器上，每个都持有可变状态。你需要一个 全局一致性快照 ——对于每个事件，要么该事件已被所有应处理它的算子完整处理，要么尚未被任何算子处理。不存在半应用的事件。且要在不停止整个系统的情况下完成。

Chandy–Lamport（1985） 的方法：沿数据流的每个通道发送一个特殊标记（ 屏障）。当算子在每个输入通道上都收到屏障时，对自身状态进行快照，然后在每个输出通道上转发屏障。所有算子级快照的集合构成全局一致性切片。

Flink 的实现：

TIME ──────────────────────────────────────────────────────►

Source:    e1  e2  [BARRIER_N]  e3  e4  [BARRIER_N+1] ...
                          ▲
                          │ JobManager triggered checkpoint N
                          │ by injecting barrier_N at all sources

  When BARRIER_N flows past an operator:
    1. operator finishes processing events that arrived BEFORE the barrier
    2. operator snapshots its state (RocksDB SSTs as they are at this moment)
    3. operator emits BARRIER_N to all downstream outputs
    4. operator resumes processing events arriving AFTER the barrier

  Result: every operator's snapshot is a "consistent cut" — every event
  before barrier_N is fully applied, every event after is not.

4c. 对齐检查点与非对齐检查点

当一个算子有多个输入通道时（例如，经过 keyBy shuffle 后，每个下游槽从多个上游槽接收输入），当 barrier_N 到达通道 A 但尚未到达通道 B 时，该算子如何处理？

对齐检查点（默认）： 对已收到 barrier 的通道上的事件进行缓冲，等待所有通道的 barrier 到达，然后执行快照，再恢复处理。

Aligned:
  channel A ──e1─e2─B(N)─e3─e4───►   e3,e4 buffered until B(N) arrives on B
  channel B ──e5──────B(N)─e6──►
                       ▲
                       │ snapshot now; then drain buffered e3,e4 + e6

优点：每个算子的输入集无歧义——快照精确反映消费 barrier 之前所有数据后的状态。缺点：若通道 B 较慢（GC 暂停、网络拥塞、上游热点），通道 A 将被阻塞。在持续背压下，对齐时间可能增长至数秒，超出延迟 SLO（服务级别目标）。

非对齐检查点（Flink 1.11+）： 一旦 barrier 到达 第一个 输入通道，立即执行快照，同时捕获其他通道的在途缓冲区状态。恢复时，这些缓冲事件会在恢复处理前重新回放。

Unaligned:
  channel A ──e1─e2─B(N)─e3─e4───►
  channel B ──e5──────B(N)──e6──►
                          ▲
                          │ snapshot triggers when EITHER channel sees B(N)
                          │ in-flight e3,e4 (channel A) captured in snapshot

优点：barrier 以最快路径的速度传播；背压下检查点耗时大致恒定。缺点：快照体积增大（需保存在途缓冲区状态）；CPU 和 S3 带宽消耗更多。

超过约 10 个数据节点后，为主节点分配专用虚拟机。它们配置要求很低——4 vCPU 和 8 GB 内存即可——但对稳定性至关重要。 默认使用对齐模式。当持续出现以下情况时切换为非对齐模式： lastCompletedCheckpointDuration / checkpointInterval > 0.7 ——通常是在背压比率持续 > 0.5 时。

Kafka Streams 采用了根本不同的方案：它不使用 barrier，而是提交一个 Kafka 事务，原子性地同时更新源偏移量、状态存储变更日志主题以及输出主题。事务本身即是快照。

Spark Structured Streaming 使用 批次 ID：每个微批次都有唯一的数字 ID。批次 N 的状态变更被写入某个目录 state/N/ 在 HDFS 上。若在批次 N 执行期间发生崩溃，下次重试将从头重新执行批次 N，确定性地产生相同的状态目录。Sink 在写入前会检查批次 ID 以避免重复写入。

4d. 通过两阶段提交（2PC）Sink 实现端到端精确一次

内部精确一次（检查点 barrier）保证的是："引擎的状态是一致的。"但 sink 向外部写入——Kafka、Postgres、S3。如果仅进行普通写入，崩溃后可能会重放已写入下游的事件。因此，端到端精确一次要求 sink 参与检查点协议。

该模式为 2PC sink:

       Source         Operators           Sink
         │                │                │
         │   barrier_N    │   barrier_N    │
         │ ────────────►  │ ────────────►  │
         │                │                │ pre-commit (PREPARE):
         │                │                │   open Kafka txn (or
         │                │                │   begin XA transaction)
         │                │                │   write records, flush
         │                │                │   (NOT visible yet to
         │                │                │    read_committed consumers)
         │                │                │   ack barrier_N to JM
         │                │                │
         │            JobManager: all acks received
         │                │                │   persist ckpt N metadata
         │                │                │   to S3
         │                │                │
         │                │ notifyCheckpointComplete(N)
         │                │  ─────────────►│
         │                │                │ commit:
         │                │                │   Kafka txn commit
         │                │                │   records now visible

两个阶段：

  1. 预提交（PREPARE）： 当 barrier_N 到达 sink 时，开启一个事务（Kafka 事务性生产者，使用稳定的 transactional.id，或 JDBC XA prepare，或写入临时 S3 路径）。将上一个 barrier 以来缓冲的所有记录写入，执行刷盘，但不提交。向 JobManager 确认 barrier。
  2. 提交： 在 JobManager 确认检查点 N 已持久化到 S3 之后（所有算子已确认、元数据已持久化），发送 notifyCheckpointComplete(N) 到 sink。sink 随即提交事务，记录对外可见。

崩溃分析：

  • 预提交之前： 未写入任何记录。源端回退至检查点 N-1 的偏移量，重放后产生相同记录。精确一次。
  • 预提交期间： 已开启的事务因协调器超时而中止。恢复后，源端回退并重放。精确一次。
  • 预提交之后、提交之前： 事务状态已持久化到检查点中。恢复时，sink 重新发现该未提交事务（通过事务 ID）并提交。精确一次。
  • 提交之后： 无需任何操作。恢复时不会重放这些记录。

核心不变式： sink 事务提交与检查点元数据提交绑定在一起。JobManager 是 2PC 协调者。

Sink 分类：

  • 事务性 sink： Kafka（事务性生产者）、支持 XA 的 MySQL/Postgres、支持 2PC 的分布式数据库。在 Flink 中实现相应接口 TwoPhaseCommitSinkFunction 在其他引擎中使用类似机制。
  • 幂等 sink： 按主键进行 upsert 操作（Postgres ON CONFLICT、Cassandra upsert、使用基于 (key, timestamp) 确定性对象名称的 S3）。允许安全重放；在可见状态层面实现有效的精确一次。
  • 至少一次 sink： 其他所有情况——追加写入文件、发送邮件、调用 REST API。不要声称这些场景支持精确一次。

4e. 事件时间与水位线

事件时间 是事件负载中携带的时间戳（例如，刷卡尝试发生的时刻，或传感器读数上的时间戳）。 处理时间 是算子所在节点的系统时钟时间。两者之所以会出现偏差，原因如下：

  • 不同地区的生产者投递事件时存在 0–60 秒的偏斜。
  • 网络抖动会导致批次延迟数分钟。
  • 重放（回退到旧偏移量）会将历史时间戳注入当前的系统时钟流。

使用处理时间计算"过去 5 分钟内每个商户的交易数"，在上述任何情形下都会得出错误结果。事件时间窗口可解决这一问题。

一个 水印 是引擎的启发式断言："我认为时间戳 ≤ W 的事件不会再出现在此流上。"水位线由数据源与数据记录一同发出，沿数据流传播，并在算子处合并（算子的水位线 = 所有输入水位线的最小值）。

一种典型策略：

WatermarkStrategy
    .<MyEvent>forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withTimestampAssigner((e, ts) -> e.getEventTimeMillis());

这表示：水位线 = 当前分区上已见到的最大事件时间 − 10 秒。10 秒的预算可容忍最多晚到 10 秒的乱序事件。

4f. 窗口状态及单个事件端到端处理过程

考虑一个以 user_id 为键、用于产品分析聚合的 5 分钟滚动事件时间窗口：

For event-time tumbling window of 5 min, keyed by user_id:

  RocksDB state layout (logical):
    state["user_42|window_12:00_12:05"] = clicks = 3

  On every CLICK event for user_42 at event_time = 12:03:17:
    1. operator computes window: floor(12:03:17 / 5min) = window_12:00_12:05
    2. state.get("user_42|window_12:00_12:05") → 3
    3. state.put("user_42|window_12:00_12:05", 4)
    4. register timer at watermark = 12:05:00 (window close)

  When watermark advances past 12:05:00:
    5. timer fires
    6. window result emitted to sink
    7. window state cleared (TTL or explicit purge)

逐步追踪单个事件在 Flink 中的端到端处理（标准讲解范式）：

Event: user_42 clicks at event_time = 12:03:17.123 UTC.

Step 1: PRODUCE
  App server publishes to Kafka topic clicks.
  Producer hashes "user_42" → partition 42.
  Broker writes to log segment (sequential disk write), replicates to
  ISR (in-sync replicas), returns offset 9,765,432. (~10 ms)

Step 2: FLINK SOURCE
  TaskManager 17 owns partition 42 (parallelism = partition count).
  Source thread polls broker, deserializes event, assigns event_time.
  Updates partition watermark: max_seen = 12:03:17.123, wm = 12:03:07.123.
  Forwards event to keyBy operator.

Step 3: KEYBY SHUFFLE
  keyBy(user_id) hashes "user_42" → bucket 167.
  Bucket 167 lives on TaskManager 31, slot 5.
  Event travels through Flink's Netty-based network stack with
  credit-based flow control. (~5 ms, intra-rack)

Step 4: KEYED PROCESS — STATE READ + MUTATE
  Slot 5 on TM 31 sets thread-local key context = "user_42".
  Determines window: 12:00 – 12:05.
  state.get("user_42|w_12:00-12:05")
    → RocksDB lookup:
       (a) memtable check: miss
       (b) bloom filter on L0 SSTs: key might exist in SST 3
       (c) block index on SST 3: key falls in block 47
       (d) check page cache for block 47: HIT
       (e) binary search in 4 KB block: found, count = 3
  Returns 3.
  state.put("user_42|w_12:00-12:05", 4)
    → RocksDB insert:
       (a) append (key, 4) to memtable skip-list (~5 µs)
       (b) RocksDB's own WAL (Write-Ahead Log) is typically DISABLED
           in Flink because Flink's checkpoint is the durability mechanism
       (c) return
  Operator schedules a timer for window close at watermark = 12:05:00.

Step 5: SINK PRE-COMMIT (next barrier)
  60s later, JobManager injects barrier N+1 at all sources.
  Sources snapshot Kafka offsets, emit barrier downstream.
  Keyed operators snapshot RocksDB (incremental SST upload to S3:
    only the SSTs that have changed since the previous barrier).
  Sink receives barrier, opens a Kafka transaction, writes any
  output records buffered since the last barrier, flushes, acks
  barrier to JobManager.
  JobManager collects all acks, persists ckpt N+1 metadata to S3.
  JobManager sends notifyCheckpointComplete(N+1) to sink.
  Sink commits Kafka transaction. Records visible downstream.

Step 6: DURABILITY POINT
  The event is durable end-to-end once Kafka commits the sink transaction
  AND the checkpoint metadata is in S3. If everything crashes immediately
  after this, recovery from checkpoint N+1 will not replay the event
  (source offsets advanced past it), and any output is already in Kafka.

大多数候选人止步于"Flink 执行检查点"。只有高级候选人才能解释 sink 提交顺序为何至关重要，以及每个阶段的存活状态是什么。

§5. 深入理解窗口机制

窗口机制是事件时间语义与应用正确性的交汇点。窗口是"无界流中有限的切片，对其执行聚合才有意义"。共有四种窗口类型，选择哪种是流处理作业中影响最大的建模决策之一。

5.1 滚动窗口——不重叠、固定大小

滚动窗口将时间划分为连续的、不重叠的固定大小桶。每个事件恰好属于一个窗口，是经典的"每 1 分钟桶内的事件数"模式。

event_time axis:
  12:00       12:01       12:02       12:03       12:04
   │           │           │           │           │
   │ ╔═══════╗ │ ╔═══════╗ │ ╔═══════╗ │ ╔═══════╗ │
   │ ║ win 0 ║ │ ║ win 1 ║ │ ║ win 2 ║ │ ║ win 3 ║ │
   │ ╚═══════╝ │ ╚═══════╝ │ ╚═══════╝ │ ╚═══════╝ │
        ▲           ▲           ▲           ▲
        e1,e2       e3          e4,e5,e6    e7

使用场景： "每分钟每个商户的交易数"、"每分钟活跃设备数"、"每分钟每个端点的请求率"——适用于分桶均匀且每个事件只需计数一次的场景。

机制： 窗口分配器计算 window_start = floor(event_time / window_size) * window_size。状态以 (user_key, window_start)为键。当水位线超过 window_start + window_size + allowedLateness，窗口关闭，结果发出，状态被垃圾回收。

权衡： 边界效应。11:59:59.999 的事件落入窗口 0；12:00:00.001 的事件落入窗口 1。对于需要"跨边界平滑"的指标（例如"过去 60 秒的滚动速率"），请改用滑动窗口。

5.2 滑动窗口——重叠，滑动步长 < 窗口大小

滑动窗口具有固定大小，但以更小的步长推进。每个事件同时属于多个窗口（具体而言，属于 window_size / slide_size 个窗口）。

window size = 5 min, slide = 1 min:
   12:00              12:05
   │                  │
   ╔══════════════════╗
   ║ win [12:00, 12:05) ║
   ╚══════════════════╝
        ╔══════════════════╗
        ║ win [12:01, 12:06) ║
        ╚══════════════════╝
             ╔══════════════════╗
             ║ win [12:02, 12:07) ║
             ╚══════════════════╝

An event at 12:03:30 belongs to FIVE windows: [11:59,12:04), [12:00,12:05),
[12:01,12:06), [12:02,12:07), [12:03,12:08).

使用场景： 适用于平滑的滚动聚合——"过去 5 分钟的平均延迟，每分钟更新一次"、"欺诈特征：过去 5 分钟内的拒绝次数，每 1 分钟刷新一次"。

成本： 状态和处理量按重叠因子倍增。每 1 分钟滑动一次的 5 分钟窗口，其状态是 1 分钟滚动窗口的 5 倍。每秒滑动一次的 1 小时窗口是灾难性的（状态放大 3600 倍）。请在数据新鲜度预算允许的范围内尽量选择较粗的滑动步长。

优化： 基于窗格的增量聚合。引擎存储每分钟的局部求和结果；在发出 5 分钟窗口结果时，将 5 个窗格求和。状态增长为 O(窗格数)，而非 O(事件数)。

5.3 会话窗口——基于间隔、动态大小

会话窗口没有固定长度，它将同一键的连续事件分组，只要相邻事件之间的间隔低于阈值。当间隔超过阈值时，会话关闭。

key = user_42:

events:    e1   e2     e3                    e4   e5
time:     12:00 12:01 12:02                 12:35 12:36
                              gap = 33 min
                            > 30 min threshold

sessions:
  [12:00, 12:02 + 30min] ── session A: contains e1, e2, e3
  [12:35, 12:36 + 30min] ── session B: contains e4, e5

引擎将每个新事件视为可能开启新会话、结束上一个会话或延长当前会话的信号。若一个迟到事件到达两个现有会话之间的间隔内，两个会话会被合并为一个——这是一个细微且开销较大的操作。

具体示例——用于用户活动追踪的会话化处理：

Scenario: web analytics for an e-commerce site. Session gap = 30 minutes
of inactivity. Per-session features: page views, conversion event present?,
total dwell time, referring source.

T = 12:00:00  user_42 visits home page
  → no existing session for user_42
  → open session S1 = [12:00:00, 12:00:00 + 30min)
  → state["user_42|S1"] = {pages: 1, conv: false, dwell: 0, src: "google"}

T = 12:02:13  user_42 visits product page
  → existing session S1 still open (12:02:13 < 12:30:00)
  → extend S1 to [12:00:00, 12:02:13 + 30min) = [12:00:00, 12:32:13)
  → state["user_42|S1"] = {pages: 2, conv: false, dwell: 133s, src: "google"}

T = 12:25:00  user_42 checks out (conversion!)
  → extend S1 to [12:00:00, 12:55:00)
  → state["user_42|S1"] = {pages: 3, conv: true, dwell: 1500s, src: "google"}

T = 12:55:00  watermark advances past 12:55:00
  → S1 timer fires
  → emit session result downstream: {user: 42, pages: 3, conv: true,
                                       dwell: 1500s, src: "google",
                                       start: 12:00:00, end: 12:25:00}
  → state["user_42|S1"] cleared (TTL)

T = 13:30:00  user_42 returns
  → no open session
  → open session S2 = [13:30:00, 14:00:00)
  → state["user_42|S2"] = {pages: 1, ...}

  ── LATE EVENT TWIST ──

T = 13:35:00  a late event arrives for user_42 with event_time = 12:40:00
  → 12:40:00 falls in the gap between (S1 end + 30min) = 12:55:00 and
    S2 start = 13:30:00 — wait, no, it falls within the 30-min extension
    range of the original S1 if the gap hadn't closed.
  → Watermark already advanced past 12:55:00, so S1 was closed.
  → Goes to side-output for late-event reconciliation, OR if
    allowedLateness is configured, the engine recreates S1 state from
    the closed-but-retained window and merges.

状态特征： 会话窗口是状态开销最高的窗口类型，因为每个会话理论上可以无界延续。请显式限制最大会话时长（window.maxSize(24h)）否则具有永不结束会话的 bot 键会导致状态爆炸。

5.4 全局窗口 + 自定义触发器

全局窗口是每个键对应一个窗口，没有自动关闭机制。当内置窗口类型均不适用时使用，需通过 Trigger 实现自定义触发逻辑。

The global window:

key = sensor_42:
  [─────────────────────────────────────────────────────────────►
   single global window per key, never closes by itself
   events e1, e2, e3, e4, ... all accumulate here

Custom Trigger fires when:
  • count of events reaches N (count trigger)
  • event-time elapsed reaches T (event-time trigger)
  • a punctuation event arrives (custom predicate)
  • combination of above

使用场景： 适用于机器学习特征计算，其中"窗口"是"最近 1000 次观测"而非"最近 5 分钟"。自定义触发器还支持"当预算超额时触发"或"当哨兵事件到达时触发"等模式。这是最灵活也最容易踩坑的窗口类型——你需要自行管理 evictor 逻辑（从状态中丢弃哪些事件）和触发器逻辑（何时触发）。

5.5 水位线——引擎如何判断"T 之前不会再有事件"

水位线是引擎对 事件时间进展的单调递增断言。其语义为："作为数据源，我认为此通道上不会再到达 event_time ≤ W 的事件。"水位线与数据记录一同沿数据流传播，并在多输入算子处合并为 output_watermark = min(input_watermarks).

数据源计算水位线的方式：

Strategy 1: Bounded out-of-orderness
  watermark = max(event_time_seen_so_far) − max_lateness_budget

  Example: forBoundedOutOfOrderness(Duration.ofSeconds(10))
  After seeing event with event_time = 12:03:17, emit watermark = 12:03:07.
  Events with timestamps in (12:03:07, 12:03:17] are still tolerated;
  events with timestamps ≤ 12:03:07 are considered LATE.

Strategy 2: Monotone timestamps (in-order assumption)
  watermark = max(event_time_seen_so_far)
  Tolerates zero out-of-orderness. Faster watermark advance, more late events.

Strategy 3: Punctuated
  emit watermark only on receipt of a specific marker event
  Useful for sources like Kafka topics where the producer injects
  "I've drained partition through timestamp T" markers.

有界与无界乱序：

  • 有界： 引擎假设存在某个最大延迟上限（10 秒、60 秒、5 分钟）。水位线积极推进；结果及时发出；超出预算的事件被视为迟到事件。
  • 无界： 引擎无法确定性地声明"T 之前不会再有事件"。引擎仍会发出 某个 （来自数据源的启发式结果），但用户必须接受结果是推测性的。

迟到事件问题： 事件在其所属窗口关闭后才到达。三种处理策略：

  1. 静默丢弃 （许多引擎的默认行为）。旧窗口的聚合结果略有误差；下游消费者不会收到任何丢失信号。
  2. 允许延迟 + 重新发出： allowedLateness(5 min) 在窗口关闭后额外保留窗口状态 5 分钟。迟到事件会更新结果；引擎重新发出更新后的聚合结果（对下游采用 UPSERT 语义）。
  3. 侧输出： 将迟到事件路由到单独的流（OutputTag Flink 中的侧输出）以供离线对账。主流水线发出"足够好的实时"结果，批处理作业与侧输出进行对账。
SingleOutputStreamOperator<Result> primary = stream
    .keyBy(Event::getUserId)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .allowedLateness(Time.minutes(5))
    .sideOutputLateData(lateEventTag)   // very-late events here
    .aggregate(new MyAgg());

DataStream<Event> veryLate = primary.getSideOutput(lateEventTag);
veryLate.addSink(new ReconciliationSink());

"静默丢弃"、"允许延迟重新发出"与"侧输出到对账流"之间的权衡取决于具体应用。欺诈检测通常选择侧输出（需要审计跟踪）；互动仪表盘选择允许延迟（消费者接受偶尔的重新发出）；IoT 告警通常静默丢弃（过时的告警比遗漏更糟糕）。

§6. 状态后端调优

§4a 介绍了状态后端是什么。本节着眼于实际操作：如何调优状态后端，使其在规模化场景下真正跟得上负载。

6.1 每个状态原语对应 RocksDB 列族

Flink 将每个状态原语（ValueState, MapState, ListState，等）映射到专用的 RocksDB 列族。列族是同一个 RocksDB 实例内物理独立的 LSM 树——它们共享 WAL，但各自维护独立的 memtable、SST 和压缩调度。

Inside one TaskManager slot:
  RocksDB instance "operator_keyed_state"
   ├── column family "cf_value_state__feature_v1"  ◄── ValueState<Double>
   ├── column family "cf_map_state__friend_set"    ◄── MapState<UserId, Boolean>
   ├── column family "cf_list_state__events_buf"   ◄── ListState<Event>
   ├── column family "cf_window_state__w_5min"     ◄── window contents
   └── column family "cf_timer_state__event_time"  ◄── Flink timer service

为何这对调优至关重要：

  • 不同的状态原语具有不同的访问模式。用作每键计数器的 ValueState 对热点键读取频繁；用作好友集合的 MapState 数据量大但访问相对较少。分离列族可让引擎针对每种状态分别调整块缓存大小、写缓冲区大小和压缩策略。
  • 热点列族可能主导压缩 CPU 消耗。若某个列族承担 95% 的写入量，RocksDB 的压缩队列将被该列族占满；冷列族的压缩滞后，L0 文件不断堆积。
  • schema 演进在列族层面独立进行。迁移一个状态的 schema 不会影响其他状态。

6.2 增量检查点——仅上传变更的 SSTable

朴素的检查点方式是每隔一段时间将全部状态序列化后上传到 S3。对于状态为 100 GB、检查点间隔为 60 秒的作业，所需上传带宽为 100 GB / 60s = 1.7 GB/s（每个 TaskManager），通常不可行。

增量检查点 利用了 LSM 树的不可变性不变式：一旦 SST 文件写入完成，其内容永远不会改变。因此只需上传自上次检查点以来新增的 SST 文件。

Checkpoint N at T=60s:
  RocksDB SSTs on local NVMe:
    sst-001  ─┐
    sst-002  ─┼── existed at ckpt N-1, already in S3 ─── NOT uploaded
    sst-003  ─┘
    sst-100  ─┐
    sst-101  ─┼── created since last ckpt              ─── UPLOAD
    sst-102  ─┘

  Checkpoint metadata in S3:
    ckpt-N-metadata.json = {
      "sst_files": ["sst-001", "sst-002", "sst-003", "sst-100", "sst-101", "sst-102"],
      "kafka_offsets": {...},
      "pending_2pc_txns": {...}
    }

S3 中已有的 SST 通过其内容哈希引用；新增的才会上传。典型带宽节省：90–99%。

注意事项： S3 存储成本会增加，因为只要有检查点引用旧 SST，它们就会被保留。引擎在超过保留阈值后会对未被引用的 SST 进行垃圾回收。保留策略配置不当可能导致 S3 费用增加 10 倍。

6.3 作业升级时的状态迁移

当作业代码发生变更（新增算子、新状态原语、分区键变化）时，旧检查点可能无法直接兼容。引擎提供了跨作业版本迁移状态的工具。

兼容性升级： 新作业的算子 UID 和状态 schema 保持不变。引擎自动恢复状态。这是理想路径——通过 .uid("operator-name") 显式固定 UID，确保代码重构后 UID 的稳定性。

schema 迁移： 若状态 schema 发生变更（例如 POJO 新增字段），引擎使用 Avro/POJO 演进规则将旧字节按新 schema 反序列化。添加可空字段是安全的；删除字段是安全的（有损）；重命名字段需要显式迁移。

拓扑变更迁移： 若作业的 DAG 发生变化（新增算子、并行度大幅变化），引擎提供了 --allowNonRestoredState （丢弃不匹配状态）及显式算子状态重映射功能。这一过程对算子而言较为复杂，通常触发"保存点、编辑、恢复"工作流（参见 §10 作业升级策略）。

6.4 存储状态的 schema 演进——适用 Avro 演进规则

流处理器的状态在作业生命周期内是持久化的。schema 演进的适用方式与任何持久化格式相同：

  • 前向兼容变更： 添加带默认值的可选字段，添加新枚举值（需有"默认"回退）。读取新状态的旧代码会忽略新字段。
  • 后向兼容变更： 删除新代码不使用的字段。读取旧状态的新代码会忽略缺失字段。
  • 破坏性变更： 重命名字段、更改字段类型、删除必填字段。这些变更需要显式迁移逻辑。

Flink 支持基于 Avro 的状态（演进能力丰富）和基于 POJO 的状态（演进能力有限，支持 Java 字段的增删）。对于复杂 schema 变更，需编写迁移作业：通过 state-processor-API 读取状态，进行转换，再写入新检查点中的新状态。

6.5 RocksDB 调优为何重要——块缓存、压缩策略

RocksDB 有数百个调优参数。对于流处理负载，关键参数如下：

块缓存大小： 热点 SST 块的内存缓存。缓存未命中需要磁盘读取（约 100 µs，SSD）；命中只需内存查询（约 100 ns）。对于热点键主导的工作负载，块缓存命中率决定了每个事件的延迟。默认约 8 MB 通常偏小，建议调整为 TaskManager 内存的 20–40%。

压缩策略：

  • 分层压缩（默认）： 经典 LSM，严格的逐层大小比例。写入时写放大较低，空间放大较低，但热点更新时压缩 CPU 开销更高。
  • 分级（通用）压缩： 压缩工作量更少，空间放大更高（磁盘上的状态可能是实际大小的 2–3 倍）。适用于本地 SSD 充裕但 CPU 受限的场景。
  • FIFO 压缩： 适用于带 TTL 的临时状态——直接删除旧 SST，无需压缩。速度快，但不适用于通用键控状态。

写缓冲区大小（memtable）： 较大的 memtable（256 MB，而非默认的 64 MB）可降低 L0 刷盘频率和写放大。代价是内存消耗增加，崩溃恢复速度变慢（需要回放更多 WAL——尽管 Flink 通常禁用 RocksDB WAL，因为检查点是其持久化机制）。

每键布隆过滤器位数： 10 bits/key（默认）的假阳性率约为 1%。对于大多数读取未命中的工作负载（例如，按 sequence-id 去重且键基本唯一的场景），可增加至 15–20 bits/key，以减少 SST 访问次数。

调优循环： 通过 RocksDB 统计信息（读放大、写放大、压缩停顿、块缓存命中率）进行监控。每次只调整一个参数，然后重新测试。Flink 通过 metrics.reporter.rocksdb.factory.class 配置项暴露 RocksDB 指标。

§7. 背压

背压是引擎防止快速生产者压垮慢速消费者的机制。它也是流处理中最常被误解的概念——工程师在 Flink UI 中看到"背压：80%"时往往惊慌失措，而实际上背压是流处理引擎在负载下保持正确性的正常方式。

7.1 当 sink 变慢时会发生什么

Normal flow:
  Source → Op1 → Op2 → Sink → External system
     1M/s   1M/s  1M/s   1M/s     1M/s consumed

Sink slows (e.g., downstream Kafka becomes slow, or external REST API
adds latency):

  Source → Op1 → Op2 → Sink → External system
     1M/s   1M/s  1M/s   500k/s   500k/s consumed
                              ▲
                              └── sink's output buffer fills

Sink's input network buffer (allocated by upstream Op2) fills:

  Op2 → Sink
        [████████████████████]  ◄── buffers full
        Op2's "send" call blocks (credit-based flow control runs out of credits)

Op2 stops sending to Sink:

  Op2 cannot emit → Op2's processing loop blocks waiting on output buffer
  Op2 stops consuming from its input buffers

  Op1 → Op2
        [████████████████████]  ◄── Op2's input buffers fill

  Op1 stops sending to Op2 → Op1 stops consuming from its input

  Source → Op1
           [████████████████████]  ◄── Op1's input buffers fill

  Source stops sending → Source's Kafka consumer poll() loop slows

  Eventually: Kafka consumer lag grows. Source pulls from Kafka at the
  rate the slowest downstream operator can absorb.

核心特性： 背压从最慢的算子向源端反向传播。源端自然地降低对持久化日志的消费速度。日志缓冲未消费的数据（这正是 Kafka 保留机制的用途）。当慢速算子追上进度后，背压解除，源端恢复全速消费。

这是正常现象。这意味着：在下游出现短暂慢速时，源端不会丢弃数据，不会导致引擎 OOM，也不会静默丢失事件，而是在最慢阶段自然地进行速率限制。

7.2 检测背压

通过 Flink UI： JobGraph 视图按算子显示背压指示器。该指标通过采样获得：Flink 线程周期性地转储堆栈，检查是否阻塞在 LocalBufferPool.requestBuffer() （当输出缓冲区满时会阻塞的调用）上。阻塞样本数与总样本数之比即为"背压比率"。

Flink UI legend:
  OK        backpressure ratio < 10%
  LOW       10% – 50%
  HIGH      > 50%

Per-operator color codes propagate UPSTREAM from the bottleneck.
A HIGH on Op2 means Op2 is being slowed by something downstream (sink
or further operators). A HIGH on Source means the next operator down is
the bottleneck.

通过指标： - numRecordsInPerSecond 。 numRecordsOutPerSecond 某个算子入流 > 出流且持续一段时间，表明该算子本身是瓶颈。 outputBufferPoolUsage 接近 1.0，表明输出缓冲区已满——来自下游的背压。 inputBufferPoolUsage 接近 1.0，表明输入缓冲区已满——来自上游的背压。 idleTimeMs 接近 0，表明算子在持续工作，与输入缓冲区满的状态结合，是典型的瓶颈特征。

通过 Kafka 消费者滞后： 持续背压对终端用户最直观可见的症状。源端跟不上消费速度，滞后单调递增。滞后量的大小决定了输出的新鲜度上限。

7.3 修复方案

1. 提升瓶颈算子的并行度。 若瓶颈是 CPU 密集型转换，增加并行度可提供更多计算核心。实践方法：通过 UI 定位瓶颈，提升其并行度（.setParallelism(n)），并对输入进行重新分区。

2. 修复慢速 sink。 大多数背压由 sink 引起。请检查下游情况：Kafka 集群健康状态（Broker CPU、磁盘 IO、副本同步滞后）、外部 API 限流、数据库写锁竞争、网络出口饱和（尤其是跨可用区流量）。

修复 sink 通常是治本之策；修复引擎层面只是治标之举。

3. 使用更大的网络缓冲区。 Flink 的网络栈为每个 channel 使用固定大小的缓冲池。如果在途数据的工作集超过缓冲池大小，线程会频繁阻塞。 taskmanager.memory.network.fraction （默认约为 TM 内存的 10%）且 taskmanager.network.memory.buffers-per-channel （默认为 2）可以适当增大。当流量呈突发性而非持续高负载时，这一配置尤为有效。

4. 切换到非对齐 checkpoint。 在持续背压下，对齐 checkpoint 会在慢速 channel 上等待 barrier，导致停顿。非对齐 checkpoint 将 barrier 传播与事件处理解耦——即使算子卡在慢速下游，checkpoint 也能完成。

5. 背压触发的负载削减。 对于从根本上超出容量的过载情况，可配置的负载削减（丢弃一定比例的低优先级事件）是最后手段。大多数引擎不原生支持此功能；需在 source 端或过滤阶段通过采样决策来实现。

§8. 旁路输出与复杂事件处理（CEP）

两个相关的高级功能：从单个算子路由不同事件类别（旁路输出），以及在流上检测顺序模式（CEP——复杂事件处理）。

8.1 旁路输出——从单个算子路由不同事件类型

基本抽象：单个算子向其主输出以及一个或多个旁路输出发送数据，每个旁路输出以标签标识。 OutputTag。下游子图订阅特定标签。

                       primary output ──► main pipeline ──► OLAP sink
ProcessFunction ───┬──►
                   ├──► side output "late events"     ──► reconciliation
                   ├──► side output "malformed"       ──► dead-letter queue
                   └──► side output "high-value-user" ──► priority pipeline

旁路输出的价值：

  • 错误处理而不破坏拓扑结构。 否则，格式错误的 JSON 事件会导致反序列化器崩溃。使用旁路输出后，算子捕获解析错误并将原始字节路由到死信队列供离线检查，主流水线继续运行。
  • 延迟事件隔离。 窗口算子将“非常晚到”的事件发送到旁路输出以供批量对账；主流水线继续发出接近实时的聚合结果。
  • 多租户无需多个作业。 单个算子可将事件分类为“免费层”、“付费层”、“内部”输出，每个输出由具有不同 SLO（服务级别目标）的不同下游流水线处理。
  • CDC 路由。 带有 op=INSERT, op=UPDATE, op=DELETE 标志的 Debezium 事件可以路由到不同的旁路输出，供不同的下游消费者使用。
final OutputTag<Event> lateTag = new OutputTag<>("late") {};
final OutputTag<String> errorTag = new OutputTag<>("error") {};

SingleOutputStreamOperator<Result> main = events
    .process(new ProcessFunction<Event, Result>() {
        @Override
        public void processElement(Event e, Context ctx, Collector<Result> out) {
            try {
                if (e.eventTime < ctx.timerService().currentWatermark()) {
                    ctx.output(lateTag, e);
                    return;
                }
                out.collect(transform(e));
            } catch (Exception ex) {
                ctx.output(errorTag, e.rawBytes);
            }
        }
    });

main.getSideOutput(lateTag).addSink(new ReconciliationSink());
main.getSideOutput(errorTag).addSink(new DeadLetterSink());
main.addSink(new MainSink());

8.2 复杂事件处理（CEP）——流上的模式匹配

CEP 是一种声明式 DSL，用于检测事件流中的顺序或上下文模式。经典示例：“用户登录，然后在 1 分钟内 5 次登录失败，随后提款超过 $10,000。”

Flink 的 CEP 库将模式编译为 NFA（非确定性有限自动机），并按 key 运行。每个部分匹配的状态保存在引擎的 keyed state 中。

Pattern: login → 5×failed_login (within 60s) → large_withdrawal

NFA:
  state_0 ── login_event ──► state_1
  state_1 ── failed_login (count: 1) ──► state_2
  state_2 ── failed_login (count: 2) ──► state_3
  ...
  state_6 ── failed_login (count: 5, within 60s of state_1) ──► state_7
  state_7 ── withdrawal(amount > 10_000) ──► MATCH

  (any event in [state_1, state_7) with elapsed > 60s aborts the match)

Flink CEP API：

Pattern<Event, ?> pattern = Pattern.<Event>begin("login")
    .where(SimpleCondition.of(e -> e.type.equals("LOGIN_SUCCESS")))
    .next("failedLogins")
    .where(SimpleCondition.of(e -> e.type.equals("LOGIN_FAILED")))
    .timesOrMore(5)
    .within(Time.minutes(1))
    .next("withdrawal")
    .where(SimpleCondition.of(e -> e.type.equals("WITHDRAWAL")
                                    && e.amount > 10_000));

PatternStream<Event> patternStream = CEP.pattern(events.keyBy(Event::userId),
                                                  pattern);

DataStream<Alert> alerts = patternStream.select(
    new PatternSelectFunction<Event, Alert>() {
        @Override
        public Alert select(Map<String, List<Event>> match) {
            return new Alert(match.get("login").get(0).userId,
                             match.get("withdrawal").get(0).amount,
                             "POSSIBLE_ACCOUNT_TAKEOVER");
        }
    });

为什么 CEP 属于流处理器： 模式匹配器需要与其他流算子相同的原语——事件时间排序、watermark 感知的超时、keyed state、Exactly-Once 恢复。在外部构建将重新实现这些原语。

适用场景： 账户盗用检测（示例）、供应链 SLA 违规（“订单下达 → 48 小时内未发货”）、交易监控（“大额订单后 5 毫秒内出现反向交易”——但在此延迟级别需要硬件 FPGA，而非 Flink）、客户旅程漏斗（“页面 A → 页面 B → 30 分钟内转化”）。

局限性：

  • 并发匹配下的状态爆炸。 如果每个 key 同时存在数千个部分匹配，状态可能迅速膨胀。需谨慎调整 AfterMatchSkipStrategy 参数。
  • 延迟与正确性的权衡。 长模式超时意味着引擎需持有状态直到模式超时或匹配。24 小时的模式超时意味着每个部分匹配都要保持 24 小时的状态。
  • 无反事实推理。 CEP 难以表达“X 发生但 Y 未发生”。模式基于正向存在性；事件缺失需要使用带计时器的自定义 ProcessFunction 逻辑实现。

§9. 流上的 SQL

流处理器越来越多地支持 SQL。其动力在于：能写 SELECT 语句的分析师数量远超 Scala/Java 工程师，而且 SQL 规划器生成的数据流优化对于大多数聚合查询而言优于手写的作业图。

9.1 Flink SQL / Table API

Flink SQL 是带有流式扩展的 SQL 方言。如下查询：

SELECT
  user_id,
  TUMBLE_START(event_time, INTERVAL '5' MINUTE) AS window_start,
  COUNT(*) AS click_count,
  COUNT(DISTINCT page_id) AS unique_pages
FROM clicks
GROUP BY user_id, TUMBLE(event_time, INTERVAL '5' MINUTE);

会被编译为一个 DataStream 作业：source → keyBy(user_id) → 5 分钟滚动事件时间窗口 → 聚合器 → sink。优化器负责选择窗口实现、状态布局和并行度。

流式特定扩展：

  • TUMBLE, HOP, SESSION 窗口 TVF（Table-Valued Functions）。
  • WATERMARK FOR event_time AS event_time - INTERVAL '10' SECOND 在 DDL（Data Definition Language）中——建表时的 watermark 策略。
  • MATCH_RECOGNIZE 用于模式匹配（Flink SQL 中 CEP 的等价物）。
  • INSERT INTO ... SELECT ... 用于定义持续作业。

Table API 是其编程等价物——流式 Java/Scala/Python API，编译为与 SQL 相同的内部表示。

9.2 ksqlDB on Kafka

ksqlDB 是 Confluent 在 Kafka Streams 之上的 SQL 层。SQL 查询在 ksqlDB 服务器集群中以 Kafka Streams 拓扑运行。状态存储在 RocksDB + Kafka changelog 主题中，与原生 Kafka Streams 完全一致。

-- Define a stream from a Kafka topic:
CREATE STREAM clicks (user_id BIGINT, page_id BIGINT, event_time BIGINT)
WITH (KAFKA_TOPIC='clicks', VALUE_FORMAT='JSON',
      TIMESTAMP='event_time');

-- Continuous aggregation:
CREATE TABLE clicks_per_user AS
SELECT user_id, COUNT(*) AS click_count
FROM clicks
WINDOW TUMBLING (SIZE 5 MINUTES)
GROUP BY user_id;

-- The resulting TABLE is a Kafka topic, materialized as a queryable state
-- store. ksqlDB serves pull queries against it via REST.

优势：与 Kafka 深度集成，对 Kafka 用户运维复杂度低。劣势：仅限 Kafka 作为 source 和 sink，优化器成熟度不及 Flink SQL。

9.3 RisingWave 的 PostgreSQL 兼容流式 SQL

RisingWave 暴露 PostgreSQL 线协议。使用 psql连接，编写 SQL，引擎将其作为持续流处理作业运行。物化视图是一等公民——随着输入流到达，增量计算。

CREATE SOURCE clicks (user_id BIGINT, page_id BIGINT, event_time TIMESTAMP)
WITH (connector='kafka', topic='clicks', ...);

CREATE MATERIALIZED VIEW clicks_per_user_5min AS
SELECT user_id,
       window_start,
       window_end,
       COUNT(*) AS click_count
FROM TUMBLE(clicks, event_time, INTERVAL '5' MINUTES)
GROUP BY user_id, window_start, window_end;

-- Query the view directly:
SELECT * FROM clicks_per_user_5min
WHERE user_id = 42
ORDER BY window_start DESC
LIMIT 10;

该视图始终保持亚秒级的最新状态。查询结果由 RisingWave 内部基于 Hummock 的状态提供服务，无需重新读取原始数据。

9.4 基于流的物化视图

统一抽象：物化视图是一个聚合表达式，其结果随输入变化而增量维护。流处理器本质上是物化视图引擎：

Input event log    ─►   Streaming SQL engine    ─►   View result (queryable)
(append-only)          (incremental compute)        (point-lookup latency)
  • Flink SQL 输出到 Kafka/Postgres： 视图结果是派生数据集；消费者查询 sink。
  • ksqlDB 表： 视图结果可通过对服务器的 REST pull 查询来查询。
  • RisingWave / Materialize： 视图结果可通过 PostgreSQL 线协议查询；引擎本身充当视图的数据库。

9.5 与传统 SQL 的查询优化器差异

流式 SQL 优化器面临与批处理 SQL 优化器不同的代价模型。批处理优化器最小化“总工作量”——总 CPU、总 IO。流式优化器最小化“每事件工作量”——新事件到达时的增量代价。

区别：

  • 无全表扫描。 输入是无界流；不存在“扫描全表”的计划节点。所有输入均以增量方式订阅。
  • 状态代价主导。 join 的代价由状态中维护的哈希表大小决定，而非每事件查找代价。优化器倾向于使状态保持较小的 join 顺序。
  • Watermark 传播至关重要。 某些算子（窗口、聚合）需要 watermark 才能发出结果。优化器确保 watermark 流经每条必要路径。
  • 不可重新执行。 批处理查询可在失败后重新执行。流式查询从 checkpoint 恢复继续；优化器必须生成与状态恢复兼容的作业图。
  • 增量聚合。 优化器将聚合重写为按窗格（对于窗口）计算部分结果后再合并，而不是在每个事件上重新计算。

结论： 在批处理和流式 SQL 中看起来相同的查询，其物理计划可能截然不同。朴素的 SELECT COUNT(*) FROM big_table 在批处理中可行；流上的等价查询需要有界窗口或无界状态（规划器通常不允许后者）。

§10. 作业升级策略

流作业持续运行。在不丢失状态或重复处理的情况下升级它——新代码、新功能、修复 bug——是运行流式基础设施中操作难度最大的部分之一。

10.1 Savepoint——作业停止时手动触发的 checkpoint

一个 savepoint 是带有手动触发和无限生命周期的 checkpoint。普通 checkpoint 每 N 秒触发一次并仅保留最近几个，而 savepoint 是显式的“这是我作业的快照，永久保存（直到我删除它）”。

Upgrade workflow with savepoint:

1. Trigger savepoint:
     flink savepoint <job-id> s3://savepoints/my-job/
   → engine takes a global consistent snapshot, writes to S3, returns path.

2. Stop the job:
     flink cancel <job-id>
   → job stops. State is preserved in the savepoint.

3. Deploy new code, restart from savepoint:
     flink run --fromSavepoint s3://savepoints/my-job/snapshot-abc \
               --allowNonRestoredState \
               my-new-jar.jar
   → new job picks up from the savepoint's source offsets and state.

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 --allowNonRestoredState 该标志允许新作业忽略新代码不再需要的算子状态（例如，已删除的算子）。若不设置此标志，引擎在 savepoint 中有任何状态没有目标时将拒绝启动。

10.2 原地升级——兼容的状态 schema

当新代码的状态 schema 与旧代码兼容时（通过 Avro 演进、POJO 添加/删除字段等），升级为“原地升级”：停止作业，从 savepoint 以新代码重启，引擎透明地用新代码反序列化旧状态。

兼容的变更：- 添加无状态的新算子。- 向 POJO 状态添加可选字段。- 重命名类（通过 @TypeInfo 保留类型标识的注解）。- 添加新列族（新状态原语）——它们从空状态开始。

原地升级是快速路径；尽量采用。

10.3 并行运行——新作业与旧作业并行，然后切换流量

当新代码不兼容时（不同的 keying、不同的 sink schema、不同的算子 UID），最安全的方式是并行运行：

T = 0: only old job is running.
       old_job ──► sink (alias=v1)

T = 1: start the new job consuming the same Kafka source from a recent offset.
       old_job ──► sink_v1
       new_job ──► sink_v2     (warming up state via replay)

T = warmup: new_job has caught up; its sink_v2 is current.

T = switch: clients (downstream consumers) switch from sink_v1 to sink_v2.
            For tables: alias swap. For event streams: switch consumer group.

T = post-switch: old_job is decommissioned.
                 old_job ────► sink_v1   (paused / stopped)
                 new_job ──► sink_v2     (live)

代价：并行运行期间计算资源加倍。收益：零停机、零数据丢失，在切换前用真实流量对新作业进行完整验证。

这是主版本升级、key 变更或算子图变更的标准方式。该模式有时称为“流处理的蓝绿部署”。

10.4 “函数已更改但状态不兼容”——恢复选项

当状态不兼容且并行运行不可行时（成本过高，或状态太大无法重新预热）：

选项 1：编写状态迁移作业。 使用 Flink 的 State Processor API 从旧 savepoint 读取状态，进行转换，并写入与新代码兼容的新 savepoint。离线运行此作业，然后从新 savepoint 启动新作业。

选项 2：从辅助数据源引导。 如果状态可从持久化数据源重新计算（具有长保留期的 Kafka、S3 归档），则丢弃状态并从足够旧的偏移量重放。

选项 3：接受损失。 对于某些工作负载（例如，1 小时滚动窗口聚合），以空状态启动并让其预热 1 小时是可以接受的。

选项 4：在迁移窗口期间对状态进行双键写入。 编写代码的过渡版本，同时向旧状态 schema 和新状态 schema 写入。运行足够长时间使新状态完全填充，然后部署仅读取新状态的版本。

“反序列化了错误字节”的灾难模式在 §11（Schema 演进）中描述。作业要么立即崩溃（检测到类型不兼容），要么静默地输出垃圾（类型偶然匹配但语义不同）。后者要严重得多——需在启动时通过 schema 指纹检查来检测。

§11. 状态的 schema 演进

状态是流作业的长期资产。Schema 演进规则决定了下一次代码变更是一行更新还是耗时数天的迁移项目。

11.1 Flink 管理：Avro / POJO 及演进规则

当状态使用 Flink 的类型序列化器（POJO、Avro schema、Scala case class）声明时，引擎在演进规则的限制内自动处理演进。

Avro 演进规则：

Allowed changes (forward + backward compatible):
  • Add a field with a default value
  • Remove a field that had a default
  • Add a new enum value (with default fallback)
  • Promote int → long, float → double (numeric widening)
  • Rename via aliases (Avro feature)

Disallowed:
  • Change a field's type without aliases
  • Make a previously-optional field required
  • Rename without an alias
  • Change a record's namespace

POJO 演进规则（Flink 的 POJO 类型信息）：

  • 添加新字段：安全；旧状态中缺少的字段获得 Java 默认值（0、null 等）。
  • 删除字段：安全；读取旧状态值后丢弃。
  • 重命名字段：默认不安全；需要 @TypeInfo 映射。
  • 更改字段类型：不安全；需要迁移。

11.2 原始字节：由你处理

当状态是不透明字节（自定义序列化、字节数组、手动 serde 的深层嵌套结构）时，引擎不做任何处理。你需要编写带有 TypeSerializer 自定义升级逻辑和版本标记。

Custom serializer schema evolution skeleton:

class MyStateSerializer extends TypeSerializerSnapshot<MyState> {
  int version() { return 3; }  // current writer version

  MyState deserialize(byte[] bytes) {
    int writtenVersion = readInt(bytes, 0);
    if (writtenVersion == 1) {
      return upgradeFromV1ToV3(deserializeV1(bytes));
    } else if (writtenVersion == 2) {
      return upgradeFromV2ToV3(deserializeV2(bytes));
    } else if (writtenVersion == 3) {
      return deserializeV3(bytes);
    } else {
      throw new RuntimeException("Unknown version " + writtenVersion);
    }
  }
}

代价：高（由你编写迁移逻辑），但对磁盘格式有完全控制权。

11.3 添加字段、删除字段、重命名

添加字段： 最简单。旧状态缺少该字段；反序列化器用默认值填充。新事件会填充它。

删除字段： 安全但有损。旧状态仍包含该字段的字节；反序列化器跳过它们。

重命名字段： 没有别名则不安全。旧状态在 old_name；新代码查找 new_name 并得到默认值——数据静默丢失。重命名时务必提供别名。

11.4 “反序列化了错误字节”的灾难模式

最坏情况：引擎用新 schema 反序列化旧字节，新 schema 类型兼容但语义不同。示例：

Old state schema:
  class UserStats {
    int totalRequests;   // bytes [0..4)
    int errorCount;      // bytes [4..8)
  }

New code (developer reorders fields):
  class UserStats {
    int errorCount;      // bytes [0..4)
    int totalRequests;   // bytes [4..8)
  }

带别名的 Avro 能检测到这一问题（字段按名称而非位置）。没有显式类型信息迁移的 POJO 也能检测到（POJO 内部使用字段名）。但是，没有版本检查的自定义字节级序列化器会愉快地将旧字节重新解释为新布局：部署后的每个作业都输出垃圾——“errorCount = 旧的 totalRequests，totalRequests = 旧的 errorCount。”

垃圾数据向下游传播。聚合结果错误，但方式可能不触发告警（值看起来合理，只是不正确）。这是静默损坏模式。

防御措施：

  • 始终在序列化状态中包含 schema 版本标记。
  • 在可行的情况下，使用 Flink 的 POJO/Avro 类型信息，而非自定义序列化器。
  • 在 CI 中运行 schema 兼容性检查：用新代码反序列化旧 savepoint 测试 fixture；类型不匹配时构建失败。
  • 对于关键状态，编写一个“冒烟测试”，将新状态的聚合与部署后从原始事件计算的已知正确值进行比较。

§12. 多租户与隔离

生产环境的 Flink/Spark 集群通常运行多个作业。它们共享资源的方式决定了“嘈杂邻居”问题的影响范围。

12.1 在共享 TaskManager 上运行多个作业

最简单的模型：一个 TaskManager 有 N 个 slot；每个 slot 运行一个任务。多个作业可以将任务放在同一个 TaskManager 上——不同的 slot，但共享 JVM、共享内存池、共享 CPU。

TaskManager TM-5 (16 GB RAM, 8 slots):
  slot 0: job_fraud_features (parallelism task 0)
  slot 1: job_fraud_features (parallelism task 1)
  slot 2: job_ad_clicks (parallelism task 0)
  slot 3: job_ad_clicks (parallelism task 1)
  slot 4: job_session_aggregator (parallelism task 0)
  slot 5: job_session_aggregator (parallelism task 1)
  slot 6: empty
  slot 7: empty

优点： 高效的资源打包；一个 TaskManager 集群服务于多个作业。

缺点： 嘈杂邻居问题。行为异常作业的 GC 停顿会阻塞无关作业的 slot。slot 内存泄漏最终会导致 JVM OOM，并杀死所有 slot。

12.2 Slot 共享组

Flink 默认允许 同一作业中不同算子的任务 共享 slot。这很高效——流水线阶段共享一个 slot，避免网络 shuffle。不同作业也可以在明确允许的情况下共享 slot。

Default slot-sharing: tasks from the same job's chained pipeline stages
share a slot to enable in-memory data exchange instead of network.

Slot Sharing Group "default":
  slot 0: source(p=0) → map(p=0) → keyBy → window(p=0) → sink(p=0)
  slot 1: source(p=1) → map(p=1) → keyBy → window(p=1) → sink(p=1)
  ...

自定义 slot 共享组可以隔离作业内的算子（例如，将重状态算子放在独立的 slot 上，防止其饿死轻量算子）。在多作业层面，跨作业的 slot 共享通常被禁用。

12.3 通过独立 JobManager 实现资源隔离

对于需要作业间强隔离的场景（不同 SLO、不同安全域、不同值班团队），每个作业获得自己的 JobManager 和 TaskManager 集群。Kubernetes 模式：

Job A:
  JobManager pod (HA: 2 replicas with leader election via k8s lease)
  TaskManager pods (10 replicas with PodAntiAffinity for AZ spread)
  Dedicated namespace, resource quota, NetworkPolicy

Job B:
  JobManager pod
  TaskManager pods
  Separate namespace

这是“会话模式与应用模式”的权衡。 会话模式：一个 Flink 集群服务多个作业；配置成本低，但存在嘈杂邻居问题。 应用模式：每个作业一个 Flink 集群；成本高但隔离性好，是 Kubernetes 上的惯用方式。

对于 Staff 级别的设计，“关键作业使用应用模式，低优先级批流作业使用会话模式”是常见模式。

12.4 cgroup 限制

当多个 TaskManager 共享物理节点时，Linux cgroups（控制组）限制每个 TM 的 CPU 和内存。Kubernetes 资源限制（resources.limits.cpu, resources.limits.memory）通过 cgroups 实现。

cgroup hierarchy on a node:
  kubelet/
   ├─ pod_taskmanager_jobA_0/
   │   └─ cpu.shares = 4096, memory.limit_in_bytes = 16 GB
   ├─ pod_taskmanager_jobB_0/
   │   └─ cpu.shares = 2048, memory.limit_in_bytes = 8 GB
   └─ ...

CPU contention: cgroups distribute CPU proportional to cpu.shares.
Memory: hitting the memory limit triggers OOMKill, the kernel reaps
the cgroup. Kubernetes restarts the pod.

cgroups 提供 CPU 和内存隔离。默认情况下，它们不隔离磁盘 IO 带宽或网络带宽；对于这些，分别使用 blkio 。 tc （Traffic Control）qdiscs，或将嘈杂作业运行在专用节点上。

12.5 嘈杂邻居问题

典型症状：低优先级作业突然降低了关键作业的延迟。排查思路：

  1. CPU 争用： 两个作业在同一节点上将 CPU 跑满。cgroup shares 进行仲裁，但 CPU 密集型作业会增加延迟敏感作业的唤醒时间。
  2. 内存压力： 某个作业增长至接近其限制，JVM 进行长时间 GC 周期，slot 线程停顿。如果它们共享 JVM，另一个作业的任务也会停顿。
  3. 磁盘 IO： 某个作业的 RocksDB 压缩使本地 NVMe 饱和，另一个作业的读取请求排队等待。
  4. 网络带宽： 某个作业的跨 AZ shuffle 占满 VPC 的每 AZ 带宽预算；另一个作业的 source 拉取速度下降。
  5. Checkpoint 存储： 两个作业同时向 S3 写入；S3 每前缀限流生效；checkpoint 持续时间增长。

缓解措施： 应用模式（专用集群）、专用节点（为关键作业设置 taint/toleration）、每作业 S3 路径分片、每作业专用 Kafka 集群。

§13. 成本经济学

生产流作业的月度成本通常由三个部分主导：TaskManager 计算、checkpoint 存储和网络出口费用。了解各部分的相对贡献有助于指导优化优先级。

13.1 Checkpoint 存储——S3 + RocksDB 状态大小 × checkpoint 频率

Checkpoint storage cost model:

  cost_per_month = state_size_GB × s3_storage_$_per_GB_month
                   + retained_checkpoints × state_size_GB × s3_storage_$
                   + checkpoint_PUTs_per_month × s3_PUT_$
                   + checkpoint_GETs_per_month × s3_GET_$    (on recovery)

Worked example:
  State size: 1 TB
  Retained checkpoints: 3 (last completed + 2 historical)
  Incremental delta per checkpoint: 50 GB
  Checkpoint interval: 60s → 43,200 checkpoints/month
  S3 standard storage: $0.023/GB/month
  S3 PUT: $0.005 per 1000

  Steady-state storage: 3 × 1 TB = 3 TB → 3000 × $0.023 = $69/month
  Incremental SST upload: 50 GB × 43,200 = 2.16 PB upload/month (bandwidth)
  Per-checkpoint PUT count: ~10 (one per SST) × 43,200 = 432,000 PUTs/month
                            → 432 × $0.005 = $2.16/month (negligible)

Network egress (S3 cross-AZ to TaskManager):
  If checkpoints are in the same region but different AZ:
    50 GB × 43,200 × $0.01/GB cross-AZ = $21,600/month
  If same AZ: free, but loses AZ-failure durability.

Typical: storage cheap, PUTs negligible, NETWORK is the issue.

缓解措施：

  • 减少状态大小： 对状态设置 TTL，更激进地淘汰。
  • 降低 checkpoint 频率： 60s → 5 分钟。权衡：崩溃后的重放窗口增大。受 RPO（Recovery Point Objective）约束。
  • 同可用区 S3： 如果持久性预算允许，使用 S3 单可用区桶（生产环境可能不适用）。
  • 压缩： Flink 可以压缩 checkpoint 负载；典型 SST 内容约可压缩 3 倍。

13.2 TaskManager 集群成本

TaskManager fleet cost model:

  cost_per_month = num_taskmanagers × instance_cost_per_hour × 730 hours

Worked example:
  60 TaskManagers on m6id.4xlarge (16 vCPU, 64 GB RAM, NVMe local):
    on-demand: ~$1.00/hour × 60 × 730 = $43,800/month
    reserved 1y: ~$0.60/hour × 60 × 730 = $26,280/month
    spot: ~$0.30/hour × 60 × 730 = $13,140/month (but spot-interrupt risk)

  At 1M events/sec sustained, this is ~30k events/sec per TM. CPU
  is usually the bottleneck for stateful operators; RAM hosts the
  block cache.

缓解措施：

  • 合理调整实例规格： 如果块缓存命中率低则增加内存；如果压缩是瓶颈则增加 CPU。
  • 预留实例/竞价实例混合： 基线使用预留实例，突发使用竞价实例。竞价实例中断通过 checkpoint 重放处理。
  • 如果过度配置则减少并行度： 测量每 slot 利用率；削减多余配置。
  • 更高 IOPS 的存储： 有时更快的 NVMe 可减少 TM 数量（CPU 在等待存储时被阻塞）。

13.3 跨可用区 shuffle 的网络出口流量成本

跨可用区网络出口流量成本：AWS 约 $0.01/GB，GCP 约 $0.01/GB，Azure 约 $0.02/GB。这是高吞吐量流处理中看似不起眼、却往往是最大的费用项。

Network egress model:

  Per-event size: 1 KB serialized
  KeyBy shuffle: 100% of records cross the network (worst case)
  At 1M events/sec, shuffle traffic = 1 GB/s = 86.4 TB/day = 2.6 PB/month

  If 30% of shuffle is cross-AZ (random spreading across 3 AZs):
    0.78 PB/month × $0.01/GB = $7,800/month

  Sources reading from Kafka in the same AZ: 0 egress.
  Sources reading from Kafka in a different AZ: 100% of input is egress.

缓解措施：

  • 在源端预聚合： 在 shuffle 之前合并同一 key 的多个事件，对热点 key 可降低 10 倍流量。
  • 机架本地 Kafka broker： TaskManager 通过以下配置从同可用区的 Kafka broker 消费数据： client.rack 配置。Kafka 会尽可能返回同可用区的副本。
  • 单可用区部署： 如果可用区故障不是硬性约束（例如部分内部分析作业），单可用区部署可节省大量成本。但对生产关键路径而言是个坏主意。
  • 更小的事件 payload： 在发布到 Kafka 之前删除未使用的字段。

13.4 "激进压缩状态"模式

由于状态大小直接影响存储成本、checkpoint 时长和恢复时间，压缩状态是杠杆率最高的优化手段之一。

状态 TTL： Flink 中每种状态原语都支持 StateTtlConfig。设置明确的 TTL（例如最后访问后 24 小时过期）。后台清理会在正常 LSM 压缩期间清除过期条目。

广播状态用于小表查找： 不必让有状态算子与每个 key 的参考表进行 join，而是将小表（例如 1 GB 的功能标志配置）广播给所有算子。每个算子持有内存副本，查找为 O(1)，无需 keyed state。

近似算法： HyperLogLog 用于去重计数（每个 HLL 实例约 2 KB，而精确集合需要 O(唯一值) 的空间），Count-Min Sketch 用于频率统计，T-Digest 用于分位数计算。权衡：以有界误差率换取大幅缩减状态规模。

激进关闭窗口： 如果不需要 allowedLateness，将其设为 0。迟到事件进入 side output，窗口状态在 watermark 推进时立即关闭。

定期全量压缩： 在低流量时段调度手动 RocksDB 全量压缩，减少 tombstone 数量和过期状态。

§14. 批流统一

理想目标：一套引擎、一套 API、一套 bug。目前有三个引擎/框架声称实现了这一目标。

14.1 Flink 通过有界流实现"批+流统一引擎"

Flink 将批处理建模为"恰好会结束的流"。有界数据源有确定的最终 offset；引擎知道何时消费完所有数据；聚合操作在不等待更多事件的情况下输出最终结果。

Streaming: source is unbounded Kafka topic, watermark advances by event_time.
Batch:     source is bounded S3 file, watermark advances to infinity at EOF.

Same DataStream API. Same operators. Same state backend.
Same checkpoint/recovery (or not, if batch mode disables checkpoints).

批处理模式与流处理模式的差异：

  • Checkpoint： 批处理模式下通常禁用（重放来自输入，而非状态）。
  • 网络 shuffle： 批处理可使用"阻塞式" shuffle（中间结果落盘于各阶段之间，支持失败任务的容错重执行）。流处理使用"流水线式" shuffle。
  • 调度： 批处理可顺序调度各阶段（第一阶段完成后再启动第二阶段）。流处理各阶段并发运行。
  • 窗口： 批处理窗口在输入结束时关闭，流处理窗口在 watermark 处关闭。

支持统一的理由：将相同的业务逻辑重写两遍（Spark 批处理 + Flink 流处理，或 dbt + Kafka Streams）是 ML 系统中"线上/线下特征偏差"的第一大来源。

14.2 Spark 的"基于批处理引擎的结构化流处理"

Spark 走的是另一条路：流处理是"切分成微批的批处理"。Structured Streaming 查询与批处理查询运行相同的逻辑 SQL，引擎将每个微批视为一个小型 Spark 作业。

Logical query: SELECT user_id, COUNT(*) FROM events GROUP BY user_id

Batch execution: one job over the full input, one COUNT aggregation.
Streaming execution: one job per micro-batch over the new events, plus
                     incremental state maintenance (state in HDFS files).

优点： 优化器、类型系统、DataFrame API 完全一致。熟悉 Spark 批处理的团队几乎可以零成本掌握 Spark 流处理。

缺点： 微批模型存在延迟下限（通常 100ms–1s）。若需要真正低于 100ms 的延迟，Spark Continuous Processing 存在但处于实验性阶段且有诸多限制。

14.3 Apache Beam 作为可移植层

Beam 不是引擎，而是一套 API。你用 Beam SDK（Java、Python、Go）编写 pipeline，然后在 Beam runner上运行：Flink runner、Spark runner、Dataflow runner（Google 的托管引擎）、Samza runner、Storm runner。

            Beam SDK (Java / Python / Go)
                       │
            Pipeline → portable representation
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
  Flink Runner    Spark Runner   Dataflow Runner
        │              │              │
     Flink jobs   Spark jobs    Google Dataflow
                                     workers

优点： 一套代码，随处运行。适合在云服务商之间迁移，或需要通过 Dataflow（Google 首选的批处理路径）回填、通过 Flink（首选的流处理路径）在线运行的 pipeline。

缺点： Beam 的模型是各 runner 的最低公分母。部分 Flink 专属优化（state-processor-API 访问、自定义窗口分配器）无法通过 Beam 使用。抽象层是有代价的。

14.4 何时适合统一，何时批处理需要独立引擎

适合统一的情况：

  • 批处理和流处理的业务逻辑完全相同（ML 特征计算、派生数据集物化）。
  • "在线"和"离线"输出必须完全一致（特征一致性是模型质量要求）。
  • 团队流处理经验多于批处理经验。

不适合统一（应使用独立引擎）的情况：

  • 批处理需要流处理优化器难以处理的复杂查询优化（cross join、大规模 shuffle 聚合）。
  • 批处理的输入规模远大于流处理（全表扫描多 TB 数仓）；流处理引擎针对单事件成本优化，而非在海量有界输入上的单行成本。
  • "批处理"本身属于不同的工作负载（针对数仓的即席分析查询——那是 Snowflake 或 BigQuery 的领域，而非流处理引擎）。

实践模式：使用 Beam 或 Flink 批处理模式处理与流处理路径共享逻辑的派生数据集回填；使用 Spark + dbt 处理数仓 ETL；使用流处理引擎处理实时服务路径。

§15. CDC（变更数据捕获）与流处理

CDC 是将流处理从"小众工具"转变为"现代 ETL 基础设施"的标志性模式。其工作模式如下：

   OLTP database (Postgres / MySQL)
         │
         │ database write-ahead log
         │ (binlog, WAL)
         ▼
   ┌───────────────┐
   │   Debezium    │   ◄── Kafka Connect source, reads the DB log,
   │               │      emits change events to Kafka
   └───────┬───────┘
           │
           ▼
   ┌──────────────────────────────┐
   │   Kafka topic: db.public.    │
   │   orders                     │
   │   • op=INSERT/UPDATE/DELETE  │
   │   • before / after snapshots │
   └──────────┬───────────────────┘
              │
              ▼
   ┌──────────────────────────────┐
   │   Flink (or Materialize,     │
   │   RisingWave) consumes CDC   │

   └──────────┬───────────────────┘
              │

   └──────────────────────────────┘

15.1 为什么 CDC 是现代 ETL

传统 ETL 是"每小时将整个 OLTP 数据库导出到数仓并处理"。存在的问题：

  • 锁竞争或副本延迟。 数据导出会锁定表，或以带有复制延迟的副本为目标。
  • 延迟下限。 按小时导出意味着数据新鲜度也是按小时计算的。
  • 代价。 导出整个数据库会传输大量未发生变更的数据。
  • Schema 漂移。 导出程序需要知道 schema；schema 变更会导致其中断。

CDC 颠覆了这一模式：读取变更日志（Postgres WAL、MySQL binlog）。只有变更的行才会流动。延迟以秒计。Schema 变更以新事件的形式传播。无需表锁。OLTP 数据库正常运行。

15.2 全量快照与增量

CDC 需要处理两个阶段：

快照阶段： 源端连接器启动，需要获取每一行的初始状态。有两种策略：

  • 启动时数据库快照： 连接器在启动时以快照隔离级别执行 SELECT * FROM table ，将每行作为 INSERT 事件发出，然后切换到日志流模式。缺点：对大表来说快照可能耗时数小时；除非保存了 checkpoint，否则连接器重启会从头重新快照。
  • 增量快照（Debezium 的现代方案）： 在并发流式传输日志的同时分块进行快照。新写入实时流动；分块快照填充旧数据。无长时间阻塞阶段。

增量阶段： 连接器以增量方式读取日志。每个事务的变更以批量形式流向 Kafka。连接器追踪日志 offset（Postgres 中的 LSN，MySQL 中的 binlog 文件+位置），并将 offset 提交到 Kafka Connect。

15.3 追赶模式

一种常见故障场景：连接器宕机数小时，数据库日志保留了这期间的写入；重启后连接器需要追赶进度。追赶处理速率必须超过数据库传入写入速率，否则连接器将永远滞后。

缓解措施：

  • 日志保留时长需长于最坏情况的中断时长。 Postgres wal_keep_size，MySQL expire_logs_days。请设置得宽裕些。
  • 并行应用。 部分连接器可通过对主键进行 hash 分区来在单个数据源内并行处理，同一主键内的顺序得以保留。
  • 追赶模式预算。 追赶期间增加更多连接器实例，追赶完成后缩减。

15.4 CDC pipeline 与流处理的集成

Flink 等框架将 CDC 流作为特殊的 source 类型消费：事件携带 before 。 after 快照信息以及操作类型（c 表示创建， u 表示更新， d 表示删除， r 表示从快照阶段读取）。

Flink SQL 原生支持 CDC 流：

CREATE TABLE orders (
    order_id BIGINT PRIMARY KEY,
    user_id BIGINT,
    amount DECIMAL,
    status VARCHAR
) WITH (
    'connector' = 'kafka',
    'topic' = 'cdc.public.orders',
    'format' = 'debezium-json'
);

-- Join CDC stream with another CDC stream:
CREATE TABLE user_revenue AS
SELECT u.user_id, SUM(o.amount) AS lifetime_revenue
FROM orders o JOIN users u ON o.user_id = u.user_id
GROUP BY u.user_id;
-- ↑ this view updates incrementally as orders/users are inserted,
--   updated, or deleted in the source databases.

15.5 CDC 故障模式

  • Schema 演进不匹配。 源库新增一列；Debezium 发出包含新列的事件；Flink schema 未更新；Flink 崩溃或静默忽略新列。缓解措施：使用 schema registry，并进行自动化 schema 漂移检测。
  • 删除处理。 源端的 DELETE 操作产生 tombstone 事件（op=d ，after 状态为 null）。下游存储需要具备删除感知逻辑；追加型存储（ClickHouse）需要显式配置 collapsing-merge-tree。
  • 源库主键变更。 罕见但灾难性：源库删除主键列；CDC 事件丢失 key；下游 upsert 中断。

§16. Materialize 和 RisingWave 作为数据库

Materialize 和 RisingWave 占据独特地位：它们既是流处理引擎，也是服务数据库。你编写一个 SQL 视图；引擎随着新事件的到来增量维护该视图；客户端通过 PostgreSQL 协议以低延迟点查方式查询视图。

16.1 随数据到来持续维护查询结果的流式 SQL

核心抽象：基于流的物化视图。

CREATE MATERIALIZED VIEW top_sellers_last_hour AS
SELECT seller_id, SUM(amount) AS revenue
FROM orders
WHERE event_time > NOW() - INTERVAL '1' HOUR
GROUP BY seller_id
ORDER BY revenue DESC
LIMIT 10;

-- The view's result is maintained CONTINUOUSLY:
-- - new orders update the aggregation
-- - the 1-hour window slides forward
-- - the TOP 10 list re-evaluates

-- Client query:
SELECT * FROM top_sellers_last_hour;
-- ↑ returns instantly (point lookup against materialized state)
--   freshness: sub-second from the most recent insert

查询"过去一小时收入排名前 10 的卖家"被实时维护。视图本身是可查询的关系。Dashboard 以任意频率（每秒、每 100ms）轮询视图并获取最新结果。

16.2 兼容 PostgreSQL 协议

Materialize 和 RisingWave 都实现了 PostgreSQL 的协议。从客户端的角度来看，它们就是 Postgres：

psql -h materialize.example.com -p 6875 -U materialize
materialize=> SELECT * FROM top_sellers_last_hour;
 seller_id | revenue
-----------+---------
       42  | 18420.50
       17  | 14200.00
        9  | 12001.75
       ...

-- Use any existing Postgres client, ORM, BI tool, etc.
-- Looker, Metabase, Tableau, Grafana all connect like normal Postgres.

这对采用率意义重大。分析师和 BI 工具无需学习新的查询语言或驱动程序，流式处理的本质被隐藏在 SQL 接口之后。

16.3 何时选择 Materialize/RisingWave 而非 Flink + KV 存储

Pattern A: Flink + Cassandra/Redis
  Flink job computes derived data, writes to Cassandra.
  Application reads from Cassandra.
  Two systems to operate; two failure modes; lag between Flink output
  and Cassandra read consistency.

Pattern B: Materialize / RisingWave
  Write a SQL view.
  The engine IS the store; queries hit the maintained view.
  One system; one failure mode; consistent reads.

选择 Materialize/RisingWave 的情形：

  • 查询稳定且可以表达为 SQL 视图（而非任意的按请求转换）。
  • 分析师或 BI 工具需要用 SQL 查询数据。
  • 单集群吞吐量在 10 万–100 万事件/秒范围内（超过此量级时，基于 Flink 的定制化工程更具扩展性）。
  • 团队更看重"一套系统"而非"最佳组合"。

坚持使用 Flink + 服务存储的情形：

  • 自定义逻辑无法用 SQL 表达（ML 模型推理、业务规则引擎）。
  • 单 pipeline 吞吐量超过每秒数百万事件。
  • 多元 sink（Kafka、S3、REST API、内部系统）。
  • 需要细粒度的算子控制（自定义触发器、复杂窗口、side output）。

16.4 权衡

  • 计算与服务的耦合。 繁重查询会拖慢视图维护，反之亦然。Materialize 和 RisingWave 都提供了隔离功能（查询结果缓存、只读副本集群），但这是真实存在的问题。
  • SQL 的更高运营成本。 在 Kafka 上运行 10 个增量视图的 SQL planner，每个事件消耗的内存和 CPU 可能多于完成相同工作的手写 Flink 作业。
  • 追赶时间。 从较长的 Kafka 保留期引导新视图可能需要数小时乃至数天（视图需要处理所有历史事件以计算初始状态）。在源端进行快照有助于缩短追赶时间。

这一类别仍在演进中。三年前，这一细分领域只有 Materialize。RisingWave 以云原生存储（Hummock on S3）进入市场。Confluent 的 KSQL pull 查询是第三个参与者。预计未来会有更多选择。

§17. 流作业的可观测性

一个不暴露内部状态的流作业是无法修复的。算子指标、延迟、watermark、checkpoint、状态大小——都需要指标上报、制作 dashboard 并设置告警。

17.1 算子指标

最小可行指标集：

Per operator instance (per slot):
  numRecordsIn       — events consumed since start
  numRecordsOut      — events emitted
  numRecordsInPerSecond  — current input rate
  numRecordsOutPerSecond — current output rate
  currentInputWatermark  — most recent watermark on input
  currentOutputWatermark — watermark emitted to downstream
  idleTimeMs         — time spent waiting for input
  busyTimeMs         — time spent processing

诊断模式：

  • numRecordsIn ≫ numRecordsOut 持续 对于 filter 或聚合算子以外的算子，意味着该算子正在丢弃或跳过事件，需排查 filter 逻辑。
  • currentInputWatermark 卡住的 意味着上游 source 没有进展（分区空闲，或 watermark 策略存在 bug）。在 source 上使用 withIdleness 。
  • currentOutputWatermark < currentInputWatermark 意味着该算子正在阻碍 watermark 推进（窗口算子尚未触发）。在累积期间属于正常现象；若超过预期窗口大小仍持续，则需告警。
  • busyTimeMs / (busyTimeMs + idleTimeMs) > 0.9 持续 意味着该算子受 CPU 瓶颈限制，需增加并行度或进行优化。

17.2 背压指示器

Flink UI 对线程栈进行采样，统计阻塞在输出缓冲区上的时间。每个算子的背压指示器：

  • OK： 被阻塞的采样比例 < 10%。
  • LOW： 10–50%.
  • HIGH： > 50%.

颜色代码从瓶颈处向上游传播。从右向左（下游到上游）读取图表，找出真正慢的算子。

17.3 Checkpoint 时长趋势

lastCompletedCheckpointDuration, lastCompletedCheckpointSize, numberOfFailedCheckpoints。诊断模式：

  • 持续数周的时长增长趋势 意味着状态在增长，需审计状态大小和 TTL。
  • Duration / Interval > 0.7 持续出现意味着 checkpoint 正在挤占间隔；切换为非对齐 checkpoint 或降低频率。
  • numberOfFailedCheckpoints > 0 表示存在问题：S3 限流、对齐超时、sink 准备失败。每种失败类型有其特定原因；请检查 JobManager 日志。

17.4 "该作业的 p99 延迟从 100ms 增长到 30s"排查手册

Step 1: What changed?
  - Deploys in the last 24h?
  - Source traffic volume increase?
  - Source schema changes?
  - Downstream sink latency?
  - Cluster events (TM restarts, AZ issues)?

Step 2: Where is the time spent?
  - End-to-end latency = time-to-source + per-operator processing time
    + network shuffle + sink commit time.
  - Drill into per-operator processing time via metrics.
  - If one operator's busyTimeMs spiked, look at hot keys or state growth.

Step 3: Is it backpressure or compute?
  - Backpressure on the source → bottleneck downstream. Find the HIGH operator.
  - No backpressure but high latency → genuine per-event compute increase.

Step 4: Is state size the cause?
  - state.numKeys, state.totalSizeBytes per operator → growing?
  - RocksDB compaction stalls? (rocksdb.stall.micros metric)
  - Block cache hit rate (rocksdb.block.cache.hit.ratio)?

Step 5: Is checkpoint impact the cause?
  - Are checkpoint durations > 50% of interval?
  - During checkpoint windows, does latency spike?

Step 6: Is the sink the cause?
  - Sink commit duration metrics (sink.numRecordsCommitted.duration)?
  - Downstream system health (Kafka, DB, REST API)?

Step 7: Is it skew?
  - Per-slot processing time — is one slot 10× others?
  - Per-key throughput — is one key 100× others?

Common causes by frequency:
  1. Hot key skew (60% of cases)
  2. State growth pushed RocksDB into stalls (20%)
  3. Sink slowdown propagating backpressure (10%)
  4. Schema change increased per-event processing cost (5%)
  5. Cluster events (TM churn, AZ issues) (5%)

排查总是从指标开始，缩小到瓶颈算子，然后判断慢的原因是计算、状态、网络还是下游。

§18. 容量边界——从小到大

流处理器的吞吐量跨越六个数量级。同一技术体系可在每个量级上运行，但合适的变体和配置会有显著差异。

小规模——运营 dashboard（1k 事件/秒）

初创公司运行 ksqlDB 或单个 Kafka Streams 实例来驱动内部 dashboard。状态 < 1 GB，延迟 < 1 秒，在单台虚拟机上运行。无需关注 checkpoint；若作业崩溃，从 Kafka 最近 24 小时的保留数据重启即可。

示例：SaaS 初创公司追踪"过去 5 分钟内活跃 dashboard 数"用于计费；B2B 公司基于客户端事件构建实时告警 feed。

中等规模——实时欺诈特征（100k 事件/秒）

类似 Stripe Radar 的团队，为卡片交易维护实时风险特征。峰值约 10 万次卡授权事件/秒，状态约 50–500 GB，p99 延迟 < 200ms。Flink 作业配备约 20–60 个 TaskManager，使用 RocksDB 状态后端，S3 checkpoint，单作业拓扑。

Stripe Radar 公开表示卡授权的决策延迟低于 100ms，评估的特征来源于历史和近期活动流。该 pipeline 将 Kafka Streams 用于部分特征作业，更复杂的部分使用定制基础设施。

大规模——实时 ML 特征 pipeline（100 万–1000 万事件/秒）

Uber Michelangelo 的实时特征平台，多个 Flink 作业共处理约 1000 万事件/秒，支撑车费估算、ETA 预测和欺诈检测。状态存于 RocksDB，checkpoint 写入 HDFS，约 200 个作业共约 5000 个 TaskManager，每个作业状态从 100 GB 到 100 TB 不等。

LinkedIn 的实时派生数据集——通知排序、信息流个性化、反滥用——流经 Samza 和 Flink，平台累计每日处理数万亿事件。

超大规模——互联网级聚合（1000 万+ 事件/秒，每日数万亿）

阿里巴巴双十一购物节峰值达每秒 58.3 万笔支付，分析 dashboard 从 Flink 作业聚合订单流中实时更新。阿里巴巴将 Flink fork 为"Blink"，进行了大量优化后将改动贡献回社区。阿里巴巴流处理平台的聚合事件量每日达万亿级别。

Netflix Mantis 处理每秒数千万条运营遥测事件——日志行、指标更新、系统事件——用于整个 Netflix 流媒体基础设施的实时告警和异常检测。

下一个瓶颈出现的位置

变化内容 下一个瓶颈 缓解措施
1k/秒 无——单台机器即可容纳 保持简单
100k/秒 堆内存中的每算子状态 切换到 RocksDB
1M/秒 checkpoint 上传带宽、热点 key 增量 checkpoint、key 加盐

同一套 Flink（或 Spark、Kafka Streams）代码路径以不同的部署拓扑服务于上述所有规模。这种通用性使其成为一类技术，而非单一工具。

§19. 架构全景——标准模式

适用于任何非平凡流作业的参考拓扑，与引擎无关：

   producers (services, devices, browsers, change-data-capture)
            │
            ▼
   ┌────────────────────────────────────────────────────────┐
   │ Durable event log (Kafka, Kinesis, Pulsar, Pub/Sub)    │
   │  • partitioned by some key                             │
   │  • retention long enough to absorb downstream outages  │
   │  • replication for durability                          │
   └─────────────┬──────────────────────────────────────────┘
                 │ source consumer offsets stored in the
                 │ stream processor's state, NOT in the log's
                 │ own offset commit (so source rewind is
                 │ atomic with checkpoint recovery)
                 ▼

每处深度注解都必须指出的要点：

  • source 按与流处理器 keyBy 相同的 key 进行分区。 这使得 keyBy 在大多数情况下退化为本地查找——source 已经只持有其分区 hash 匹配的 key。跨分区 shuffle 是慢路径。
  • 并行度与分区数匹配。 并行度高于分区数会浪费槽位；低于分区数则会导致倾斜，因为一个槽位需要处理多个分区。
  • Source 偏移量存储在引擎状态中，而非日志自身的偏移量机制里。 Kafka's __consumer_offsets commit 独立于引擎的 checkpoint——因此在恢复时，两者对于从何处恢复会产生分歧。引擎在 checkpoint 时原子地提交 Source 偏移量。
  • Sink 参与 2PC。 事务性提交与 checkpoint 提交通过协调器绑定在一起。
  • 协调器具备高可用性（HA）。 checkpoint 协调存在单点故障是一种架构上的坏味道。

§20. 流处理固有的难题

通过不同领域的案例，阐述该技术与生俱来的六类问题。

20.1 跨 Source 与 Sink 的端到端 Exactly-once（经典难题）

领域案例：欺诈检测。 像"过去 5 分钟内该卡片的拒绝次数"这样的流式特征，在崩溃后不能对同一笔支付重复计数——监管机构会进行审计，重复计数会导致误报，从而拒绝正常交易。

朴素的解决方案： 使用自动提交偏移量的 Kafka 消费者，并将结果写入下游 Kafka。

故障演练：

T=0: consumer reads offset 100, processes event, writes to downstream Kafka
T=1: consumer reads offset 101, processes, writes
T=2: auto-commit commits offsets [100, 101]
T=3: consumer reads 102, writes downstream
T=4: ── CRASH before next auto-commit ──
T=5: new consumer instance picks up
T=6: reads __consumer_offsets, last committed = 101
T=7: resumes from 102 → reprocesses, writes 102 a SECOND TIME

Downstream sees event 102 twice.

真正的修法： 流处理器将 Source 偏移量存储在自身状态中；将偏移量推进与 checkpoint 屏障及 2PC Sink 绑定。恢复时，Source 回退到 checkpoint-N 的偏移量；Sink 重新发送或提交其待处理事务。两端均与 checkpoint 保持原子性。

Kafka Streams 通过单个 Kafka 事务实现相同效果，该事务涵盖偏移量提交、状态存储主题写入和输出主题写入。

20.2 乱序事件与 watermark 停滞

领域案例：来自传感器集群的 IoT 告警。 Cloudflare 的 DDoS 检测系统从边缘服务器采集遥测数据；Tesla 的车队从车辆发送传感器数据。网络抖动和区域缓冲导致事件乱序到达，偏差从数秒到数分钟不等。若按处理时间计算"每分钟失败请求数超过 X 的边缘服务器"，对于那些接收到积压突发数据的窗口将产生误报告警。

朴素的解决方案： 在算子层按挂钟时间开窗，在分钟边界处关闭。

故障演练： Tesla 某区域出现 30 秒的蜂窝网络中断。10 万个事件在生成 35 秒后才到达，被归入错误的分钟窗口。"上报故障 X 的车辆数"图表在未来 30 秒出现虚假峰值，实际计数被分割到两个分钟窗口中。基于分钟对齐特征训练的安全监控模型所读取的是脏数据。

真正的修法： 使用事件时间 + watermark。

  • Source 从事件负载而非挂钟时间分配时间戳。
  • Watermark 策略 = forBoundedOutOfOrderness(60s) ——容忍 60 秒的迟到。
  • 按事件时间进行窗口分配。
  • allowedLateness(5 min) 在 watermark 过去后，继续保持窗口状态开放额外 5 分钟，以便迟到事件更新结果并重新发送。
  • 非常迟到的事件（迟到超过 5 分钟）被发送到旁路输出，用于离线对账。

相关子问题：空闲分区会导致 watermark 停滞。 一个 Source 有 256 个分区，其中一个分区零流量（某 Tesla 区域夜间无车辆在线）。该 Source 的 watermark 永远不会推进。下游 watermark = 所有输入的最小值 = 卡住。事件时间窗口永远不会关闭。从算子的角度来看，作业陷入沉寂。

修复方法： WatermarkStrategy.withIdleness(Duration.ofSeconds(30)) ——若某个 Source 30 秒未发送数据，将其标记为空闲，并将其排除在最小值计算之外。

20.3 状态大小增长（隐形杀手）

领域案例：面向产品分析的会话化处理。 会话窗口将事件聚合为用户会话，在空闲 30 分钟后关闭。Snowplow 及类似管道在数百万用户上运行此模式。

朴素的解决方案： 让状态自由增长。RocksDB 能处理大数据，对吧？

故障演练： 一个僵尸网络攻击者在一小时内创建了 1000 万个新用户 ID，每个 ID 触发一个事件，然后沉默。会话状态持有 1000 万 × 30 分钟的缓冲事件，加上每用户约 50 个特征。状态以每小时 100 GB 的速度增长。RocksDB compaction 跟不上。checkpoint 开始耗时 5 分钟（超过 60 秒的间隔）。背压 → Kafka 积压 → SLO 违规。最终本地 SSD 撑满，RocksDB 抛出异常，TaskManager 崩溃，恢复需要数小时。

真正的解决方案（分层）：

  1. 状态 TTL（存活时间）： 每个状态都有 StateTtlConfig （例如，最后一次访问后 1 小时，后台清理）。这是兜底措施。
  2. 有界会话窗口： 将会话持续时间上限设为最长 24 小时，超过则拆分。
  3. 状态后端调优： 使用激进 level 乘数的分层 compaction；为 compaction 预留空间，将 SSD 配置为预期状态大小的 3 倍。
  4. 主动监控： 在磁盘使用率达到 70% 时触发状态大小指标告警，不要等到 compaction 失败。
  5. 上游 Bot 检测： 在 Source 处过滤掉异常的键空间。

20.4 慢速 checkpoint

领域案例：Uber 量级的实时机器学习特征管道。 一个 Flink 作业在 8 亿个实体上维护 150 个特征。一个特征键获得了总流量的 5%（可能是一个压测账户、企业联名卡，或者某位名人的 LinkedIn 主页）。拥有该键的槽位将其 checkpoint 上传到 S3 需要 90 秒。

故障演练： checkpoint 间隔为 60 秒。JobManager 在第 N 个 checkpoint 完成前就调度了第 N+1 个。待处理的 checkpoint 不断堆积，JobManager 内存增长。最终系统进入降级模式并停止 checkpoint——这意味着崩溃恢复时需要重放数小时的事件。

真正的修法：

  • 增量 checkpoint（RocksDB）： 仅传输自上次 checkpoint 以来更改的 SST 文件，将上传量从全量状态降低到增量。
  • 非对齐 checkpoint： 屏障传播不会因慢速输入通道而停滞。
  • 本地恢复： 在本地 SSD 上保留最后一个 checkpoint 的副本，使 TaskManager 重启时无需从 S3 重新下载。
  • 在键控层解决热点键倾斜问题 ——参见 §23 Type 2 扩展。
  • 对 lastCompletedCheckpointDuration / checkpointInterval > 0.7 的告警作为先行指标。

20.5 作业恢复时间

领域案例：实时仪表盘（例如 Pinterest 实时推荐）。 凌晨 3 点，一个 TaskManager 宕机。最新 checkpoint 在 S3 中，60 秒前生成。

故障演练： 触发冷启动恢复：JobManager 下载 checkpoint 元数据。60 个 TaskManager 全部从 S3 下载各自的 RocksDB SST 文件。每个 TM 7 GB，60 个 TM，即使使用分片路径，持续带宽 200 MB/s/prefix：约需 3.5 分钟。加上状态还原到 RocksDB 并重建索引：再需约 2 分钟。然后 Source 回退并重放 60 秒的事件：再需约 1 分钟。总计：约 7 分钟后才能恢复稳态。在此期间，实时 Pin 推荐功能不可用。

真正的修法：

  • 本地恢复： 每个 TM 在本地保留最后一个 checkpoint 的 RocksDB 文件。在同一台机器上重启 TM 时，恢复过程跳过 S3 下载，约 30 秒即可完成恢复。
  • 基于 Region 的重启策略： 在单个 TM 故障时，仅重启受影响的管道 Region，而不是重启整个作业。
  • 备用 TaskManager： 预留 10% 的备用容量以便立即接管。
  • 自适应调度器（Adaptive Scheduler）： 动态重新配置并行度，而非固定 DAG 的整体重启。

20.6 热点键（状态倾斜，而非吞吐量倾斜）

领域案例：游戏平台的实时排行榜。 计算"过去一小时内每款游戏的前 100 名得分"。某款游戏（病毒式爆款）的流量是其他游戏的 1000 倍。拥有该游戏键的槽位 CPU 使用率达到 100%，状态量是其他槽位的 50 倍。

朴素的解决方案： 信任哈希函数。

故障演练： 按 game_id 进行哈希分区，将游戏 G 的所有流量发送到同一个槽位。该槽位成为瓶颈。背压向上游传播。总吞吐量下降至 1 /（热点游戏数量）× 单槽位容量。

真正的修法： 使用带键加盐的两阶段聚合。第一阶段：将热点键拆分为 N 个盐桶（game_G#0, game_G#1, ..., game_G#N-1），每个处理 1/N 的流量。第二阶段：去除盐值重新键控，并聚合 N 个部分结果。

Stage 1: hot_key#salt_0, hot_key#salt_1, ..., hot_key#salt_N → partial counters
Stage 2: re-key by base hot_key (no salt) → final aggregator

对于计数、求和以及近似分位数算法，这是在第二阶段将流量降低 N 倍的线性优化。

§21. 故障模式——持久性关键点

21.1 TaskManager 崩溃

T=0     TM-17 dies (kernel panic).
T=0+1s  K8s liveness probe fails.
T=0+5s  JobManager loses heartbeat from TM-17.
T=0+10s JobManager declares TM-17 failed. Triggers restart of affected region.
T=0+15s K8s schedules replacement TM-17 on a new node.
T=0+30s New TM boots. JM assigns it the tasks TM-17 had.
T=0+45s Tasks download checkpoint SSTs from S3 (or local SSD if local
        recovery enabled).
T=0+90s Tasks restored. Source consumer rewinds to checkpoint offsets.
T=0+120s Replay through to current head. Steady state.

DURABILITY POINT: the last completed checkpoint in S3. Everything
processed and acked before that checkpoint is durable; everything
after is replayed from Kafka.

INVARIANT PRESERVED: 2PC sink either replays the open transaction
(commits) or aborts (rewinds and re-emits). Either way, exactly-once.

21.2 JobManager（协调器）崩溃

T=0     JM-leader dies.
T=0+5s  Standby JMs (HA mode via ZooKeeper or k8s lease) detect failure.
T=0+10s New JM-leader elected.
T=0+15s New JM reads job state from durable store (ZK + S3):
          • job graph
          • last completed checkpoint metadata pointer
          • TaskManager registry
T=0+20s New JM does NOT relaunch TMs (they're still running).
T=0+25s New JM resumes checkpoint coordination. Next barrier scheduled.

Job continues. TMs processed through the JM gap. No data loss because
checkpoints are persisted in S3, not in JM memory.

DURABILITY POINT: ZooKeeper / k8s ConfigMap holding pointer to "latest
completed checkpoint in S3."

21.3 Sink 在 checkpoint 中途不可用（细微之处）

下游 Kafka 集群在 2PC Sink 已预提交时变为不可达。

T=0      Barrier_N arrives at sink. Opens Kafka txn 7, writes records,
         flushes, acks barrier to JM.
T=0+5s   JM collects all acks. Writes ckpt-N metadata to S3.
T=0+5s   JM sends notifyCheckpointComplete(N) to sink.
T=0+6s   Sink calls producer.commitTransaction()
         ── Kafka cluster unreachable. Times out after 60s.
T=1m+6s  Sink throws. TaskManager fails.
T=1m+11s JM restarts the task.
T=1m+45s New task instance recovers from ckpt-N.
         Sink's TwoPhaseCommitSinkFunction sees state has a record of
         "pending transaction 7 needs commit."
         Tries commitTransaction() → still fails (Kafka still down).
         Retries with exponential backoff.
T=5m     Kafka recovers.
T=5m+30s Sink commits txn 7 successfully.
         Downstream consumers see the records, exactly once.

DURABILITY POINT: pending-transaction state persisted in checkpoint.
Stable transactional.id across restarts is the trick that lets the new
producer attach to the same Kafka coordinator state.

21.4 网络分区（脑裂）

Scenario: rack A holds JM-leader + 30 TMs; rack B holds 30 TMs +
JM-standbys. Network partitions A from B.

In A:  JM-leader sees 30 TMs reachable, 30 unreachable.
       Declares 30 TMs failed. Tries to restart on the 30 surviving TMs.
       But parallelism config requires 60 TMs. Job goes into "waiting
       for resources" state.

In B:  JM-standbys cannot reach the lease in ZK (assuming ZK has quorum
       on the A side). They cannot elect a new leader. 30 TMs in B are
       running, but cannot ack barriers. After checkpoint timeout
       (5 min default), tasks abort their checkpoints and stall.

DURABILITY POINT: ZooKeeper / k8s lease serves as the split-brain
breaker. Only the side with quorum can elect a leader. The minority
side never has an active JM, so it cannot make forward progress that
conflicts with the majority side's checkpoint chain.

INVARIANT: at most one JM can be making decisions. ZK / lease ensures
this. On partition heal, B-side TMs reconnect to JM in A; any work
they did since the partition that wasn't checkpointed is rolled back.

21.5 永久性丢失（可用区故障）

整个可用区消失。60 个 TM 中的 20 个消失。JM 可能在也可能不在该可用区上。

- If JM-leader was in lost AZ: standby elected (assuming cross-AZ
  ZK quorum). Otherwise JM-leader survives.
- 20 TMs need replacement: K8s scheduler provisions in surviving AZs.
  Capacity headroom (e.g., 50% over-provisioning) absorbs the loss.
- New TMs restore from S3 checkpoints (cross-AZ replicated).
- Source offsets in checkpoint determine replay window. Kafka retention
  (e.g., 72h) absorbs any AZ outage shorter than 3 days.

DURABILITY POINT: S3 (cross-AZ replicated) holds checkpoints.
Kafka cluster spans 3 AZs with replication=3 and min.insync.replicas=2,
so it survives one AZ loss without data loss.

INVARIANT: cross-AZ durability of BOTH checkpoints and source log.
Single-AZ for either is a single point of catastrophic loss.

21.6 checkpoint 存储 S3 故障（作业暂停）

T=0      S3 region experiences an outage (or specific bucket throttles
         heavily for a customer that has exceeded request rate).
T=0+60s  Next scheduled checkpoint.
         TaskManagers begin uploading SST files; PUT requests fail with
         S3 5xx or throttling.
T=0+120s Checkpoint times out (default 10 min).
T=0+10m  JobManager marks the checkpoint as failed.
         If max-concurrent-checkpoints = 1 (default), the next checkpoint
         starts. If S3 is still down, it also fails.
T=0+30m  Multiple failed checkpoints. Default tolerance:
         tolerable-checkpoint-failure-number = 0 → job FAILS immediately
         on first failure (default in older Flink), OR
         the job continues processing without taking new checkpoints.
T=1h     If the job continues processing, the replay window (events
         processed since last successful checkpoint) grows. RPO
         (Recovery Point Objective) is now 1 hour.

DURABILITY POINT: the LAST SUCCESSFUL checkpoint before the S3 outage.
If S3 recovers and the job is still alive, the next checkpoint succeeds.
If the job crashed during the outage, recovery from the old checkpoint
replays an hour of events.

INVARIANT: source log must retain enough history to cover the worst-case
replay window. If Kafka retention is 24h and S3 is down for 25h, replay
loses data.

MITIGATIONS:
- Multi-region S3 (cross-region replication for checkpoints).
- Higher tolerable-checkpoint-failure-number (e.g., 3) to ride through
  brief outages without job failure.
- Alert on consecutive failed checkpoints, not just job failure.
- Monitor S3 PUT error rate per checkpoint path.

21.7 背压导致 checkpoint 超时

Scenario: downstream sink is slow, backpressure propagates upstream,
aligned checkpoints stall waiting for barriers on slow channels.

T=0      Sink rate drops from 1M/sec to 100k/sec (10x slowdown).
T=0+30s  Backpressure builds. Source consumes from Kafka at 100k/sec.
         Kafka lag grows.
T=0+60s  Next aligned checkpoint scheduled.
         Barrier injected at all sources. Propagates downstream at
         the rate of slow channels.
T=0+5m   Checkpoint duration exceeds checkpoint-timeout (default 10 min).
         Some operators have received the barrier on all input channels;
         others are still waiting.
T=0+10m  Checkpoint times out. JobManager aborts.
         All in-flight 2PC pre-commits are aborted. Job continues
         processing but no checkpoint progress.
T=0+11m  Next checkpoint scheduled, same outcome.

DURABILITY POINT: last successful checkpoint (from before backpressure).
Recovery would replay all events processed since then.

DIAGNOSIS:
- Check `lastCompletedCheckpointDuration` metric — rising over time.
- Check per-operator backpressure indicator — find the HIGH operator.
- Check `checkpointAlignmentTime` — high values point to alignment as
  the bottleneck (vs actual snapshot write time).

MITIGATIONS:
- Switch to unaligned checkpoints — barrier propagation does not wait
  for input alignment.
- Fix the slow sink.
- Reduce checkpoint frequency until backpressure resolves.
- Bound checkpoint timeout to fail fast and surface the issue.

21.8 "倾斜键压垮单个 TaskManager"的诊断方法

教科书级的热点键问题表现为：一个（或少数几个）TaskManager 显示 CPU 高、状态大、checkpoint 慢，而其余 TaskManager 处于空闲状态。

Symptoms:
- Per-TM CPU graph: 5% of TMs at 100%, 95% at <20%.
- Per-TM state-size graph: same skew.
- Checkpoint duration dominated by a small number of slow TMs.
- Backpressure indicator HIGH on operators feeding the hot TMs.
- Kafka lag grows on partitions whose keys hash to hot TMs.

Diagnosis playbook:
1. Identify the hot TaskManager(s) via per-TM metrics.
2. Identify the hot operator(s) on those TMs.
3. Dump per-key state size / per-key throughput for the hot operator.
   Flink's State Processor API can read state offline; instrument with
   a custom metric: numRecordsForKey histogram.
4. The top-K keys by traffic are the culprits.
5. Cross-reference with business logic: is one user a fleet account?
   A bot? A celebrity profile? An internal test key that leaked into
   production?

Resolution:
- Filter or special-case the runaway key upstream (e.g., drop events
  with a known bot pattern at the source).
- Apply two-stage aggregation (key salting) as in §20.6.
- Provision the affected partition with a dedicated TaskManager
  (custom partition assignment).
- If the key is "legitimate but extreme" (e.g., the platform's biggest
  customer), give it its own dedicated job with custom tuning.

The hot-key failure mode is the most common cause of streaming job
incidents at scale. Bake hot-key detection into the standard alert set.

§22. 为什么不用 cron + 批量 SQL 每分钟执行一次

流处理最朴素的替代方案是"每分钟对 OLTP 数据库（或分析存储）执行一次批量查询，覆盖过去 N 分钟的数据"。五行 SQL，无需 Flink、无需集群、无需随叫随到。为什么行不通？

9.1 OLTP 数据库被打垮（或 OLAP 扫描代价过高）

在每秒 100 万事件的写入流量下，你的业务数据库正忙于处理插入操作。此时你再每分钟追加一次针对非分区键的全表扫描分组查询。无论数据存储在 MySQL、Postgres、Cassandra 还是 Snowflake 中，这种次要访问模式都会与主要访问模式产生冲突。DBA 会在一小时内呼叫你。你将其迁移到只读副本 → 副本延迟飙升 → 查询依然读取到陈旧数据。

9.2 "1 分钟延迟"只是谎言

T = 12:03:00 — cron starts
T = 12:03:45 — query returns
T = 12:03:50 — your code processes results
T = 12:03:55 — alerts written

Events at T=12:03:15 are detected at T=12:03:55. 40s latency.
Events at T=12:03:59 are detected at T=12:04:55. 56s latency.

如果你承诺了亚秒级的数据新鲜度，你实际上偏差了两个数量级。

9.3 基于处理时间窗口的结果正确性会漂移

cron 查询窗口是处理时间的。30 秒的摄取延迟（属正常现象）意味着 event_time = 12:03:15 的事件可能在 12:03:45 才落地。你在 12:03:00 执行的 cron 错过了它们，在 12:04:00 执行的 cron 又看到了它们。若窗口重叠，同样的数据会被跨多次运行重复计数；若窗口不重叠，则会完全丢失。同一份数据，根据窗口方案的不同，会以两种不同的方式出错。

9.4 没有 Exactly-once 恢复机制

cron 在运行中途崩溃。重启时从分钟边界开始，完全丢失了上一分钟的数据，制造了一个欺诈检测盲区。想通过运行 ID 和状态表来实现幂等性？恭喜你，你已经开始重新实现 Flink 了，而且还是个劣质版本，没有 checkpoint 协调。

9.5 无法扩展到多个特征

5 个特征 × 10 个窗口怎么办？每分钟 50 条昂贵的聚合查询，数据库还没到午饭时间就挂了。流处理器新增一个特征只需在已有 DAG 中增加一个算子——每个事件的增量代价仅是多一条状态记录。

为什么流处理更胜一筹

流处理颠倒了数据流向。数据库不再被用于分析查询。读取发生在流处理器内部预聚合的状态上。聚合是增量的：每个事件只更新一个计数器。每个事件的代价为 O(1)，而非每次 cron 运行的 O(N)。窗口逻辑基于事件时间、由 watermark 界定，不存在重叠双计问题。故障恢复受限于 checkpoint 间隔。

数学表达： cron + 批量 SQL 的刷新代价为 O(N)，延迟为 O(刷新间隔)。流处理的每个事件代价为 O(1)，结果延迟为 O(事件处理延迟)。在每秒超过 10 万个事件的规模下，只有第二种方案适合在数据中心内运行。

§23. 扩展维度

Type 1：均匀增长（更多事件，更多键）

工作负载相同，按一定倍数扩展。增加更多生产者、更多分区、更多并行度。

  • 1× （每秒 10 万事件，1000 万键）：16 个 TaskManager，64 个 Kafka 分区。轻而易举。
  • 2× （每秒 20 万事件，2000 万键）：将并行度扩展到 128，将 Kafka 重新分区至 128。线性扩展。
  • 10× （每秒 100 万事件，1 亿键）：256 个分区，并行度 256。开启增量 checkpoint，状态溢出到 NVMe SSD。标准操作。
  • 100× （每秒 1000 万事件，10 亿键）：
  • 分区数提升至 1024，每个 TM 拥有 1-2 个分区。
  • 使用更高 IOPS 的 NVMe（gp3 io2、i4i 实例）。
  • 按算子实例对 S3 checkpoint 前缀分片，以规避前缀限速。
  • JobManager 成为瓶颈。 按特征组拆分为多个作业，"卡片特征"、"商户特征"、"用户特征"作为独立作业共享同一 Kafka Source。

拐点： 在约每秒 500 万事件时，单作业 Flink 会触及 JobManager 和 S3 协调的上限；超过此阈值，按领域拆分。

Type 2：热点强化（状态倾斜）

键相同，但某一个键获得了不成比例的流量。哈希分区将该键的所有事件发送到同一个槽位。该槽位 CPU 使用率达到 100%，持有的状态量是其他槽位的 50 倍。背压向上游传播，总吞吐量崩溃。

这需要与 Type 1 不同的解决方案。增加更多 TM 没有帮助；热点键仍然会映射到其中一个。

按复杂度递增排列的解决方案：

  1. 在 Source 端预聚合。 在数据跨越网络之前，合并同一键的多个事件。将热点键的 shuffle 流量减少 10-100 倍。
  2. 两阶段聚合（加盐）。 第一阶段将热点键拆分为 N 个盐桶，第二阶段去除盐值重新键控并聚合。
  3. 热点键检测 + 专用槽位。 分析键空间，识别 Top-K 热点键，将其路由到配有额外资源的专用 TM。使用自定义分区器。
  4. 近似算法。 如果不需要热点键的精确计数，可使用 HyperLogLog 计算基数，使用 Count-Min Sketch 计算频率。每个槽位的状态增长为 O(精度)，而非 O(唯一值数量)。

拐点： 热点键超过每槽位每秒 5 万个事件时处理；低于此阈值可忽略，超过则加盐或预聚合。

§24. 决策矩阵——何时选择流处理而非其他方案

何时应该选择流处理器而非相邻技术？

评判标准 流处理器 批量 ETL（Spark、dbt） 基于原始数据的 OLAP（ClickHouse、Pinot、Druid） 复杂事件处理（Esper、Flink CEP）
新鲜度 秒级（或亚秒级） 分钟到小时（下一批次） 对已索引数据的查询延迟为秒级 与流处理器相同（它本身就是流处理器）
每次查询的代价 每个事件 O(1) 每批次 O(N) 每次查询 O(行扫描数) 每个事件 O(1)
聚合深度 增量，窗口大小有限 全量历史复杂聚合方便 即席查询；有状态模式匹配能力有限 模式匹配（X 然后 Y，在窗口 Z 内）
状态维护 是，跨崩溃持久化 无（输出即状态） 存储即状态；查询无状态 是
搭建成本 高（集群 + 状态调优） 低（仅需 SQL + 调度器） 中（专用集群） 高（流处理器 + CEP 库）
使用场景 持续派生数据、亚分钟级新鲜度、大状态 定期刷新可接受，历史数据复杂关联 时序类数据的交互式分析 顺序模式检测（登录后 5 分钟内发生大额提款）
应避免的场景 主要是批处理，无新鲜度需求；分析师即席查询 需要亚分钟级新鲜度；需要 Exactly-once 内部状态 需要维护派生状态，而不仅仅是回答查询 简单聚合（杀鸡用牛刀）

具体阈值：

  • 新鲜度要求 > 5 分钟时，批量 ETL 更简单、更便宜。
  • 新鲜度要求 < 1 秒且每个键的状态非平凡时，只有流处理适用。
  • 查询是针对原始事件历史的即席查询且无派生状态时，OLAP 胜出。
  • 需要"检测 A → B → C 序列在 30 秒内发生"时，在流处理器之上使用 CEP 库。

边界情形： Materialize 和 RisingWave 模糊了流处理器与 OLAP 之间的界限。 它们维护增量视图（流处理），但暴露 Postgres 协议查询接口（类 OLAP 访问）。对于构建实时仪表盘的 SQL 技术栈团队，它们可以替代"Flink + 服务存储 + 仪表盘"的组合。

§25. 使用案例图谱——相同技术，不同产品

同一类技术如何驱动截然不同的产品：

欺诈检测——Stripe Radar

需求： 对每笔卡片授权的决策 SLO 低于 100ms；每张卡片持续计算数十个特征；端到端 Exactly-once（监管机构审计）；状态量与活跃卡片数成正比（约数亿张）；来自各区域边缘授权网关的乱序事件。

方案选型： Flink 风格的真正流处理，配合事件时间窗口、RocksDB 状态和 2PC Kafka Sink。亚秒级延迟排除了微批处理。端到端 Exactly-once 排除了"At-Least-Once + 下游去重"。自定义特征逻辑（Java/Scala）排除了纯 SQL 流处理。

实时机器学习特征——Uber Michelangelo、LinkedIn

需求： 在数亿个实体（乘客、司机、会员、餐厅）上维护数百个特征，秒级更新。回填兼容性（特征必须可从历史事件重新计算，用于模型训练）。线上与线下计算的特征一致性。

方案选型： 在线路径使用 Flink 或 Samza；相同逻辑编译为 Beam 或 Spark 用于离线回填（或从一开始就通过 Beam 统一）。每个特征的状态存储在 RocksDB 中，输出到特征存储（Feast、Tecton、内部系统）。由于模型频繁变更，Schema 演进至关重要。

实时分析——Pinterest、Snapchat、Twitter

需求： 每秒超过 100 万事件流入仪表盘，展示用户互动、广告效果和内容趋势。亚分钟级新鲜度。大量重叠查询，分析师每天撰写新查询。

方案选型： 使用 Flink SQL 进行持续聚合；结果落地到 Pinot、Druid 或 ClickHouse，用于即席 OLAP 查询。当查询足够稳定可表达为物化视图时，Materialize 或 RisingWave 正越来越多地取代 Flink + OLAP 的分离架构。

会话化——Snowplow、Web 分析平台

需求： 将事件聚合为用户会话，在可配置的空闲间隔后关闭。每个会话的特征计算（访问页面、转化路径、总时长）。容忍迟到事件（移动设备离线）。

方案选型： 使用 Spark Structured Streaming 或带会话窗口的 Flink。状态大小是首要关注点；强制执行激进的 TTL 和最大会话时长上限。

IoT 告警——Tesla 车队、Cloudflare DDoS 检测

需求： 数百万台设备持续发送遥测数据。当异常超过阈值时触发告警。每设备状态（滚动基线、近期样本）。容忍设备间的时钟偏差。

方案选型： 使用事件时间窗口 + 水印空闲处理（针对离线设备）的 Flink。用 Statefun（Flink 的有状态函数）或 Beam 实现每设备状态机。通过幂等 REST 写入输出到 PagerDuty / 告警系统。

实时排行榜 — 游戏平台

需求： 按游戏、按地区、按时间窗口的 top-K 评分。得分提交后数秒内更新。热键问题（一款爆款游戏主导流量）。

方案选型： 针对热键使用 Flink 两阶段聚合；输出到 Redis 提供排行榜读服务。或由 Materialize 维护增量式"每游戏 top-100"视图，直接通过 Postgres 协议查询。

CDC 处理 — Debezium → Flink

需求： 摄取数据库的变更日志（通过 binlog 从 Postgres / MySQL 获取插入、更新、删除操作），流式处理行变更，将衍生数据集物化到独立存储。即"outbox 模式"驱动事件驱动服务。

方案选型： Debezium 读取数据库日志并发布到 Kafka。Flink 消费 Kafka CDC topic，与参考数据进行 join，写入 OLAP / OLTP / 搜索索引。变体：使用 Flink SQL 实现 UPSERT-INTO 语义；Materialize 原生消费 Debezium 并维护衍生视图。

可观测性管道 — Netflix Mantis、Honeycomb，以及大型公司内部实践

需求： 每秒 10M+ 事件的运营遥测数据（日志行、链路追踪、指标点）。实时异常检测。长尾数据低成本存储；高成本但实时的衍生指标。

方案选型： Flink/Mantis 用于实时衍生路径。输出到 Druid/Pinot 供即席查询。流处理层负责将日志行转换为实际可视化的指标。

核心在于：底层都是相同的 Chandy–Lamport 机制、相同的 LSM 状态后端、相同的 2PC sink 协议。产品场景决定了窗口选择、状态结构和 sink 类——但引擎类别是统一的。

§26. 真实世界实现与数据

  • Apache Flink at Uber（Michelangelo + AthenaX）： 约 10M events/sec，分布于约 200 个作业，约 5,000 个 TaskManagers。驱动实时票价估算、ETA 和欺诈检测。状态存于 RocksDB；checkpoint 到 HDFS。Uber 公开描述了在该规模下运行 Flink 并实现次秒级特征新鲜度延迟的实践。
  • Apache Flink at Alibaba（含 Blink fork → 回归上游）： 双十一峰值达 583k payments/sec，Flink 作业实时聚合订单流分析。平台整体每日事件量达万亿级。阿里巴巴的优化（增量 checkpoint 改进、SQL planner 增强）已贡献回 Apache Flink。
  • Apache Flink at Stripe Radar： 对信用卡授权事件进行实时评估，决策 SLO 低于 100ms。结合 Kafka Streams 与自定义基础设施，并在较重的有状态 pipeline 中使用 Flink。
  • Apache Flink at Pinterest： 约 1M events/sec 进入 Flink 作业，用于实时推荐和垃圾内容检测。与 Redis/HBase profile 存储进行流表 join 查询。
  • Apache Samza at LinkedIn： 最初由 LinkedIn（Kreps 等人）发明，用于基于 Kafka 的流处理。如今 LinkedIn 同时运行 Samza 和 Flink，用于衍生数据集、通知、信息流个性化、反滥用以及 Member Hot Lead Scoring。每日累计事件量达万亿级。
  • Netflix Mantis： 用于运营遥测的流处理平台。约每秒数千万事件。对流处理基础设施进行实时异常检测。采用类 Flink 计算模型，配合 Netflix 专属调度器。
  • Apache Spark Structured Streaming at Databricks 客户： 最常见的微批流处理部署方式。大量用于 ETL 风格 pipeline，适合 1 秒至 1 分钟延迟可接受的场景。Databricks 发布了多个百万 events/sec 部署案例（如 HSBC、Comcast）。
  • Kafka Streams at Confluent 客户： 轻量级部署，每服务实例典型吞吐量约 100k events/sec，嵌入微服务中。常见于面向服务架构内的有状态流消费者。
  • Materialize at 金融科技 / SaaS： 基于 Kafka 的增量 SQL 视图，通过 Postgres 协议查询。Ramp 等多家金融科技公司已引用其部署案例，用于实时风险视图和实时看板。每集群吞吐量通常在 100k–1M events/sec 范围内。
  • RisingWave： 支持 PostgreSQL 协议的流处理数据库，使用云原生对象存储（Hummock 存储引擎）管理状态。较新；在实时分析领域增长迅速，可替代 Flink + OLAP 组合。公开基准测试声称每集群约 1M events/sec，延迟低于一秒。
  • Apache Beam： 可移植层。在 Google 内部大量用于 Dataflow（原始流处理引擎 MillWheel 是其前身）；在外部用于让 pipeline 可跨 Flink、Spark、Dataflow 和 Samza runner 移植。

规模各异、访问模式各异、同一技术类别的不同变体。若候选人能说出其中三种，并给出大致规模和所用引擎变体，已属前十百分位。

§27. 总结

流处理是基于事件日志的常驻增量计算层：一个持续查询图，在 LSM 支持的本地存储中维护带键状态，全局一致性由 Chandy–Lamport 屏障（Flink）、或事务性 offset+state+sink 提交（Kafka Streams）、或确定性批次 ID（Spark）提供，端到端精确一次语义通过绑定到 checkpoint 协议的 2PC sink 来协调——对于次秒延迟和复杂窗口需求选择真流式（Flink），当你已经在用 Spark 且一秒延迟可接受时选微批（Spark），对于微服务内的有状态消费者选嵌入式（Kafka Streams），当消费者是分析师时选流式 SQL 数据库（Materialize、RisingWave）；按每 slot 事件吞吐量（有状态约 30k events/sec/slot）进行容量规划，用 checkpoint 间隔限制回放范围，用加盐方式防御热键，并牢记持久性的关键点是"checkpoint 元数据已持久化且 sink 事务已提交"——两者缺一，最多只有至少一次语义。
