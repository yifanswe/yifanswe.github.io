关于 OLAP（联机分析处理）数据存储的技术参考——它们是什么、其列式引擎如何在字节级别工作、设计空间如何在云仓库、实时分析引擎和湖仓查询层之间分化，以及何时选择哪种方案。各类用例（BI 仪表板、实时产品分析、面向客户的分析、ML 特征流水线、日志分析、营销归因）贯穿全文，作为同一类技术适配不同工作负载的说明。

§1. 什么是 OLAP 存储

一个 OLAP 存储 是一种数据库，其物理布局、执行引擎和持久性模型都针对同一类工作负载进行了优化： 扫描大型事实表，进行过滤、分组和聚合。OLTP（联机事务处理）数据库在 1 毫秒内提交一次小变更、在 5 毫秒内完成点查，而 OLAP 存储则能在两秒内读取 100 亿行并返回 SUM(revenue) GROUP BY country。其核心特性是"在人类可接受的时间内（而非数小时）对整个数据集回答分析性问题"。

OLAP 与 OLTP 之间存在三个正交的分野：

  • 列式存储而非行式存储。 行式存储（由 16 KB 页面组成的 B+ 树，每页约存放 60 整行）是为"给我这一行"而设计的。列式存储将每列保存在各自连续的文件或页面段中，因此一个只涉及 200 列中 3 列的查询仅需读取 1.5% 的表数据——在压缩之前即可实现 60 倍的 I/O 减少。
  • 批量写入而非小事务。 OLAP 以批量方式摄入数据（从 S3 导入百万行、一小时的流式事件、每日 ETL 加载）。带行级锁的逐行 UPDATE 是反模式；引擎围绕大批量追加和整段重写而构建。
  • 设计上不支持跨写入者的多行 ACID。 OLAP 提供按批次或按快照的原子性，而非 10,000 个并发借记事务之间的点隔离。节省下来的锁和 MVCC（多版本并发控制）垃圾回收开销，被用于提升扫描聚合的吞吐量。

OLAP 处于四向分析存储设计空间的一个角落：

  • 列式 OLAP 仓库（本文重点）。 Snowflake、BigQuery、Redshift、Databricks SQL、ClickHouse、StarRocks、Apache Doris、Pinot、Druid、Trino+Iceberg、Spark SQL+Delta、DuckDB。面向列式、压缩存储、向量化执行，以及基于大型数据统计模型的优化器。为 SELECT col1, col2, SUM(col3) FROM huge_table WHERE col4 > X GROUP BY col5.
  • OLTP 关系型数据库。 MySQL InnoDB、PostgreSQL、Oracle、SQL Server。B+ 树、行式存储、强 ACID。非常适合驱动产品运行的小型可变状态（账户余额、用户资料、购物车），但不适合 SELECT COUNT(*) FROM orders WHERE created_at > NOW() - INTERVAL '90 days' 超过 1000 亿行的场景。参见 01_databases.md §1.
  • HTAP（混合事务/分析处理）存储。 TiDB、SingleStore、Unistore（Snowflake 的实验）、Oracle HeatWave。通过增量暂存 + 列式存储的方式在 OLTP 引擎上附加列式存储，或反之。承诺：一套系统同时支持两种工作负载。现实：列式存储新鲜度仅为准实时（秒级延迟），通常一种工作负载会对另一种造成饥饿，且每行成本是 OLTP 和 OLAP 的叠加。适用于"ETL 新鲜度"是瓶颈且团队明确希望避免独立数仓的场景。
  • 搜索引擎。 Elasticsearch、OpenSearch、Solr。用于文本相关性的倒排索引 + 过滤聚合。针对"找出与自由文本查询最匹配的 10 条结果"进行了优化。可以做分面聚合，但在相同数据规模下成本是列式 OLAP 的 10-100 倍，且多列 GROUP BY性能较差。参见 03_search.md.
  • 时序数据库。 Prometheus、InfluxDB、TimescaleDB、M3DB、VictoriaMetrics、Honeycomb 的列式引擎。专为 (timestamp, metric, tags) → value 而设计，具备高基数标签索引、降采样和 TTL（生存时间）淘汰机制。OLAP 也可用于时序数据，但 TSDB（时序数据库）在每数据点存储成本和速率类查询（rate(http_requests_total[5m])）上更具优势。参见 10_observability.md §3.

本文聚焦于 列式 OLAP 角落 ，因为这是字节级机制最为关键、现代数据技术栈趋于汇聚之处。我们将把湖仓格式（Iceberg、Delta、Hudi）与集成数仓（Snowflake、BigQuery）以及单用途实时 OLAP 引擎（Pinot、Druid、ClickHouse）进行对比分析。

OLAP 不擅长的场景：

  • 低延迟点查。 从列式存储按主键读取一行，需要从 N 个独立文件中重新组装 N 列。ClickHouse 点查的 p99 约为 30-50 毫秒；Postgres B+ 树查询则不足 1 毫秒。如果你的工作负载是"根据用户 ID 渲染个人主页"，OLAP 是错误的工具。
  • 高并发小写入。 在 Snowflake 或 ClickHouse 中进行逐行 UPDATE 会重写列文件，通常采用写时复制语义。列式引擎上的逐行 OLTP 吞吐量约为 10-100 ops/秒/集群——比 OLTP 低五个数量级。应批量加载或流式写入不可变 segment，让压缩合并处理数据。
  • 跨多个写入者的 ACID。 大多数 OLAP 提供 按批次的原子性 （一次加载要么完全可见，要么完全不可见）和 每查询的快照隔离。两个并发更新相同数据时是"最后写入者胜出"或冲突中止，而非 OLTP 对数百万借贷记录的串行化处理。
  • 亚毫秒级的数据新鲜度。 湖仓流水线（Kafka → Flink → Iceberg）的数据新鲜度约为分钟级。流式 OLAP（Pinot、Druid）可达亚秒级。OLAP 领域内没有任何引擎能提供 OLTP 级别的"写入在微秒级以单行粒度即可查询"。
  • 在某列中存储 10 MB 的 blob。 充满 base64 编码 PDF 的 VARCHAR 字段会破坏压缩并损害扫描吞吐量。请使用对象存储存放 blob，OLAP 只存路径。

思维模型：OLAP 存储是存活在 OLTP 中的真实数据的 次要的、衍生的、分析性视图 。OLTP 数据源发出变更流（CDC——变更数据捕获）；流水线将其重塑为列式格式的宽反范式事实表；OLAP 引擎提供"报表查询"——BI 仪表板、实时产品分析、ML 训练特征连接、面向客户的分析仪表板、即席数据科学探索。把 OLTP 数据源做对；所有分析内容都可以从其变更日志中重建。

§2. 固有保证

列式 OLAP 在设计上提供的能力，以及仍需在其上层叠加的能力。

设计上已提供

  • 快速聚合数十亿行。 现代引擎可达每节点 1-10 GB/秒的扫描速度，并在数秒内对 100 亿行生成 GROUP BY 在数秒内生成 100 亿行的结果。ClickHouse 的聚合性能基准约为每核心约 2 GB/秒；BigQuery slot 分配通常可在数十秒内扫描 TB 级数据。
  • 高压缩比，5-20 倍。 列式布局上的按列压缩（字典 + 游程编码 + 差分 + 通用 Zstandard）对于事务型事实表通常可实现 5-10 倍压缩，对于稀疏日志或状态列则可达 20-50 倍。原因：列内数据局部相似，熵编码效果极佳。
  • 通过 zone map / 元数据实现谓词下推。 每个存储粒度（行组、微分区或 segment）携带 min, max, null_count，每列通常还有直方图/布隆过滤器。带有 WHERE event_date = '2026-05-21' 条件的查询无需读取数据即可跳过 99.9% 的表内容。
  • 每查询的快照隔离。 查询看到的是其开始时刻的表的冻结视图。并发插入和表变更仅在快照提交后才可见；正在运行的读者永远不会看到部分写入。
  • 原始数据的晚期绑定 Schema。 许多 OLAP 引擎和湖仓格式（Iceberg、Parquet）允许存储 JSON / VARIANT / Map 列，并在查询时动态提取键值，即读时定义 Schema。分析师无需 Schema 迁移即可添加新字段。
  • 按批次/快照的原子性。 1 亿行的加载要么完全可见，要么完全不可见。Iceberg 通过元数据指针交换原子性地提交新快照；Snowflake 微分区通过元数据事务提交；BigQuery 流式缓冲区在刷新时提交。

需在其上叠加

  • OLTP 级别的单行一致性。 如果单个用户的账户余额必须精确且随处立即可读，那应存储在 OLTP 中，而非 OLAP。数仓是衍生视图，本质上存在数秒至数分钟的延迟。
  • 端到端流水线的正确性。 OLAP 提供快照隔离，但不保证"来自 OLTP 系统的每笔事务都恰好到达数仓一次"。恰好一次的摄入存在于流处理器/CDC 层，依赖幂等键和偏移量检查点——参见 04_stream_processing.md §6.
  • 租户间的工作负载隔离。 来自团队 A 的重量级查询可能使共享集群饱和并降级团队 B 的服务质量。多租户需要数仓级别的"虚拟仓库"（Snowflake）、按团队的 slot 预留（BigQuery）或按集群隔离。引擎本身不会自动在租户之间公平调度。
  • 数据质量和 Schema 演化规范。 错误的列类型或上游字段被删除，会静默地损坏下游表和仪表板。读时定义 Schema 无法防范人为错误，防范手段是在摄入边界处的数据契约。
  • 成本管控。 按查询付费的引擎（BigQuery）按扫描字节计费；空闲但已分配的集群（Snowflake 虚拟仓库、Redshift）按运行时长计费。一个失控的仪表板可能在一小时内扫描 1 PB 数据并产生 5000 美元费用。配额、成本标签、slot 限制——这些都需在引擎上层进行叠加。
  • PII（个人身份信息）治理、行级安全、列掩码。 列式引擎提供 RBAC（基于角色的访问控制）和列掩码功能，但在快照、副本和下游 BI 导出中强制执行 GDPR（通用数据保护条例）删除，是引擎上层的数据治理问题。
  • 备份/时间点恢复。 Snowflake 的 Time Travel + Fail-safe 是内置能力中最强的（90 + 7 天）。Iceberg 提供快照历史。大多数引擎需要明确的保留策略以及外部"每日备份是否真正可恢复"的演练。

综合：OLAP 引擎保证对你提供的数据实现 按快照原子性、列式快速聚合以及谓词跳过 。而数据的保真度、流水线的新鲜度、租户之间的隔离、成本上限以及监管治理，都是平台团队需要负责的问题。

§3. 设计空间

五种截然不同的 OLAP 存储风格，它们在计算/存储耦合度、目标查询延迟、摄入模型和运维所有权上各有不同。

云数据仓库（Snowflake、BigQuery、Redshift、Databricks SQL）

Snowflake。 全托管的仓库即服务。解耦架构：数据以不可变微分区（每个约 16 MB，列式，压缩）存储在 S3 中；计算为"虚拟仓库"（EC2 节点集群），可随意扩缩容/暂停；元数据由 Snowflake 集中管理的 FoundationDB 存储。在时间旅行、零拷贝克隆、跨账户数据共享方面表现强劲。以读为主，对 MERGE/UPDATE 有强力支持。定价：按仓库秒数计费 + 存储费。

BigQuery。 Google 的无服务器仓库。基于 Dremel（开创列式 OLAP 浪潮的那篇论文）构建。无需集群；提交查询后，Google 的调度器从共享资源池中预留"槽位"（CPU 单元），扫描 Colossus（Google 分布式文件系统）上的列式存储（Capacitor 格式），并返回结果。存储与计算解耦，但计算对用户不可见——没有需要用户运维的节点。定价：按扫描字节数（按需）或预留槽位容量计费。

Redshift。 Amazon 的数据仓库，最初源自 ParAccel 的分支。两代产品：经典版（共享存储集群）和 RA3 节点 将计算与存储解耦（数据存储在 S3 上的 Redshift Managed Storage 中）。兼容 PostgreSQL 的 SQL 接口。定价：按节点小时 + RMS 存储费。

Databricks SQL（及更广泛的 Databricks 平台）。 湖仓（Lakehouse）方案。数据以 Delta Lake 格式存储在客户自己的 S3 / ADLS / GCS（Google Cloud Storage）上；计算使用运行在 Spark 集群上的 Photon（Databricks 的向量化 C++ 引擎）。同一引擎服务于 SQL 工作负载、Notebook 和机器学习训练。在"一个平台搞定数仓 + 湖仓 + 机器学习"方面有强有力的故事。

这四者在同一架构上殊途同归： 共享存储上的无共享计算。存储是对象存储（S3、GCS、ADLS），使用列式文件（Parquet、ORC 或专有格式）。计算是横向扩展集群，可按工作负载调整大小。元数据（表布局、快照、统计信息）存储在集中式目录中。

自托管 OLAP 引擎（ClickHouse、StarRocks、Apache Doris）

ClickHouse。 Yandex 起源（现为 ClickHouse Inc. / 开源），C++ 实现，支持单节点或集群部署。在原始 CPU 效率方面是性能之王——带有手写 SIMD（单指令多数据）内核的向量化执行，扫描速率约 2 GB/秒/核。在日志分析、面向客户的分析、时序类工作负载上表现突出。经典架构为计算与存储耦合（每个节点的本地 NVMe 存储数据）；较新的 ClickHouse Cloud 已解耦至 S3。

StarRocks。 最初由前百度工程师从 Apache Doris 分叉而来，现已独立发展为自己的项目。兼容 MySQL 协议，MPP（大规模并行处理）架构，向量化引擎。强大的实时物化视图支持；数据可从 Kafka 摄入并在数秒内可查询。

Apache Doris。 与 StarRocks 同源（共同起源）。MPP，兼容 MySQL，专注于互联网公司的实时分析（百度、字节跳动内部使用）。

这三者属于"自行运维"层级。在大规模场景下比云数仓更便宜（没有按信用点计费），但需要承担运维负担——容量规划、副本管理、版本升级、故障恢复。

实时分析引擎（Pinot、Druid）

Apache Pinot。 诞生于 LinkedIn（2014 年，最初用于"谁查看了你的主页"和信息流分析），现已进入 Apache。专为流式摄入数据上的 sub-100 ms p99 聚合查询设计，支持数十万 QPS（每秒查询数）。架构：不可变段、每列带倒排索引的列式存储、基于 Helix 的协调器。在面向客户的实时仪表盘方面表现突出（Uber 的订单地图、LinkedIn 的分析标签页）。

Apache Druid。 诞生于 Metamarkets（约 2011 年），2018 年进入 Apache。与 Pinot 定位相同——支持流式摄入的实时 OLAP。架构：段文件（带位图索引的列式存储）、Historical / MiddleManager / Broker / Coordinator 节点。被 Netflix（点击流）、Airbnb（搜索漏斗分析）、Lyft 使用。

这两者高度重叠；都以 对数秒前到达的数据进行亚秒级聚合为目标。运维故事、索引权衡和 SQL 成熟度上的差异决定最终选择。

联邦查询引擎（Trino、Presto、Spark SQL、DuckDB）

Trino（前身为 PrestoSQL）与 Presto。 源于 Facebook（2012 年），用于对 Hive 表进行即席分析。Trino 是活跃的社区分支；PrestoDB 是 Meta 主导的延续。架构： 将计算与特定存储解耦 ——Trino 拥有 Hive/Iceberg、PostgreSQL、MySQL、Kafka、Cassandra、MongoDB、S3 等连接器。可在 JOIN 在单条 SQL 语句中同时关联 Postgres 表、Iceberg 事实表和 Kafka 主题。

Spark SQL。 通用分布式计算引擎；SQL 是其能力之一。比 Trino 更适合批处理（运行时间更长的作业，通过血缘实现容错），大量用于 ETL（抽取-转换-加载）和机器学习训练管道。

DuckDB。 单节点、嵌入式（"用于分析的 SQLite"）。向量化引擎，可直接读取 Parquet / CSV / Arrow / Iceberg。被用作笔记本电脑上的数据科学主力工具，以及其他工具（dbt、Mode、Mother Duck 云端版）中的嵌入式引擎。

这些引擎不拥有存储。它们是外部存储（湖仓表、对象存储、联邦数据库）上的查询层。与 Iceberg 或 Delta 结合，便形成 湖仓 模式：在商用对象存储上以开放文件格式实现仓库级 SQL。

湖仓表格式（Iceberg、Delta Lake、Hudi）

不是引擎，而是 表格式 ——元数据层，将 S3 中的一堆 Parquet 文件变成具有快照隔离、模式演进、时间旅行和每次提交 ACID 的真正"表"。

Apache Iceberg。 源于 Netflix，供应商中立。快照 + 清单树（见第 §4 节）。2024–2026 年新建湖仓的主流选择。读/写支持：Trino、Spark、Flink、Snowflake（现已支持）、BigQuery（现已支持）、Databricks（通过互操作）、Athena、ClickHouse。

Delta Lake。 源于 Databricks，已开源。快照 + 事务日志（_delta_log）。原生于 Databricks；更广泛的引擎支持已陆续增长（Trino、Spark、Snowflake 通过 Iceberg 互操作、Flink）。

Apache Hudi。 源于 Uber。专为更新密集型工作负载构建（骑手和司机状态更新通过行程匹配持续流入）。两种表类型： Copy-on-Write （写入时重写受影响文件）和 Merge-on-Read （写入增量日志，读取时合并）。在流式 upsert 方面表现强劲；其他引擎此后已迎头赶上。

综合分析：每个单元格的值都由引擎的核心设计选择决定。Pinot 选择带位图索引的耦合存储，因为 <100 ms p99 意味着无法承受 S3 往返延迟。Snowflake 选择解耦存储，因为其工作负载是 BI，冷缓存扫描多出的 200 ms 可以接受，换取的是"5 秒内完成计算资源扩缩容"。Trino 完全不拥有存储，因为它的任务是联邦查询现有的各类数据源。

深度章节。我们深入探讨列式存储布局、压缩编解码器、区域映射、向量化执行、Parquet 文件格式以及 Iceberg 清单树，并端到端地演示一条聚合查询的执行过程。

4.1 列式布局：行优先与列优先

面向行的存储引擎（InnoDB、PostgreSQL）在页面上连续写入行。一张 200 列、每行 256 B 的表中，每行的 200 个列值在内存中相邻排列：

Row-major (one InnoDB 16 KB page):
┌───────────────────────────────────────────────────────────────┐
│ Row 1: col1 col2 col3 ... col200  Row 2: col1 col2 col3 ...   │
│        │    │    │         │                                  │
│        adjacent in memory, one row = one contiguous chunk     │
└───────────────────────────────────────────────────────────────┘

面向列的存储引擎将每一列存储在各自连续的区间中。同样是 200 列的表，你得到 200 个独立的"列文件"（或行组内 200 个连续的列块）：

Column-major (one Parquet row group, simplified):
┌─────────────────────────────────────────────────────────┐
│ col1: [val_1, val_2, val_3, ..., val_N]   ← contiguous  │
│ col2: [val_1, val_2, val_3, ..., val_N]   ← contiguous  │
│ col3: [val_1, val_2, val_3, ..., val_N]                 │
│ ...                                                     │
│ col200: [val_1, val_2, val_3, ..., val_N]               │
└─────────────────────────────────────────────────────────┘

关键结论：查询 SELECT col5, col10, SUM(col17) FROM t WHERE col5 = 'US' 读取 仅 col5、col10、col17 的字节范围。如果每列占表总字节数的 1%，该查询读取约 3% 的数据——在压缩尚未发挥作用之前，就已实现 30 倍的 I/O 缩减。

进一步的结论： 列内局部性远高于行内局部性。所有 country_code 同一列的值很可能属于约 200 个字符串之一；所有 event_type 值属于约 50 个之一；所有 timestamp 值单调递增。按列压缩正是利用了这一特性。

4.2 压缩编解码器栈

现代列式引擎不使用单一压缩算法——它会组合针对列数据形态量身定制的轻量级编解码器，然后可选地在顶层叠加通用熵编码器。

字典编码。 对于低基数列（country_code, event_type, status），为每个不同的值构建映射到小整数 ID 的字典。字典只存储一次，用 ID 替换列值。一个 country_code 1 B 个值 × 8 字节 = 8 GB 的列，变为 1 B × 1 字节（对于 256 个不同国家）+ 字典 = 约 1 GB。 8 倍压缩比.

行程编码（RLE）。 对于连续值经常重复的列（例如，按 country_code排序后：1 B 个"US"，然后 200 M 个"GB"，…）。存储 (value, count) 键值对。1 B 行中有 1000 个不同游程的列，压缩后约 16 KB。

位压缩。 对于字典编码的列，若字典大小适合用 K 位表示（256 个条目 → 8 位，64 个条目 → 6 位），则在位级别打包 ID。对 1 B 行中的 5 位字典进行位压缩，每行节省 3 位（相比完整字节）= 每列节省约 375 MB。

Delta 编码。 适用于有序或单调的数值列——最常见的是 时间戳。存储 (start, delta_1, delta_2, ...)。对于秒级精度的事件流时间戳列，增量是可用 4–8 位表示的小整数。

Delta-of-delta。 两级增量。用于 Gorilla（Facebook 的内存时序引擎，现已推广到 Prometheus/M3DB/InfluxDB IOx）。对于固定节奏的指标（"每 10 秒采样一次 CPU 使用率"），大多数时间戳增量等于 10 秒；增量的增量为 0。对于稳定节奏，可将 64 位时间戳平均压缩至约 1 位。

Gorilla 浮点编码。 Facebook Gorilla 论文的技巧：将每个 float64 与前一个值进行 XOR，然后存储前导零字节数 + 有效字节。对于静止或缓慢变化的指标（内存使用、吞吐量），连续样本仅在最后几个字节有差异，压缩比可达 10–20 倍。

Zstandard / LZ4 / Snappy 顶层压缩。 在结构编解码器之后，对结果运行通用熵编码器。Zstd 3 级是现代默认选择——压缩速度约 500 MB/秒，解压速度约 2 GB/秒，可进一步压缩 2–4 倍。Snappy 速度更快但压缩率较低；LZ4 速度最快。Parquet 默认设置因引擎而异；ZSTD 是 Iceberg/Delta 的现代默认值。

累积效果。 一张真实生产事实表（事件日志，约 50 列，未压缩约 256 B/行）：按列编解码器可得约 5 倍压缩比；Zstd 再叠加约 2 倍；合计约 10 倍。10 TB 的原始事实表在 S3 上以 Parquet 格式存储约为 1 TB。Snowflake 报告生产表的典型压缩比为 3–5 倍（其自有微分区格式与编解码器栈）。

Column "country_code" compression path (1 B rows):
Raw VARCHAR     8 GB    8-byte strings
Dictionary →    1 GB    8-bit IDs + small dictionary
Bit-pack →      125 MB  bit-packed to actual bit-width
Zstd →          ~15 MB  entropy coding on the bit-packed bytes
                ───
                ~530x reduction on a perfectly-skewed column

4.3 区域映射、MinMax 索引与稀疏主键

列式引擎不对每行建索引，而是对 granule ——固定大小的连续行块（ClickHouse 约 8192 行，Pinot 约 1024 行，Parquet 中每个行组约 100 万行）。

每个 granule 携带逐列元数据：

  • 粒度内列的最小值和最大值。 一个 WHERE event_date BETWEEN '2026-05-01' AND '2026-05-07' 查询检查每个 granule 的 min 。 max；如果粒度范围与查询条件不相交，则完全跳过该粒度。
  • Null 计数。
  • 不同值计数（有时）。
  • 布隆过滤器（可选）。 对于点等值谓词（WHERE user_id = 42），每个粒度的布隆过滤器可排除没有匹配可能的粒度。
  • 直方图（可选）。 Snowflake 的聚簇功能为优化器保留每个微分区的近似分布信息。
ClickHouse MergeTree, simplified granule layout:

┌────────────────────────────────────────────────────────────┐
│ Part (~10-100 GB, one logical chunk on disk)               │
│   ┌──────────────────────────────────────────────────────┐ │
│   │ Granule 0:                                           │ │
│   │   col1.bin bytes [0..L1]   col2.bin bytes [0..L2]    │ │
│   │   ...                                                │ │
│   │ Mark file: col1: min=10, max=99, NULL=0              │ │
│   │            col2: min='2026-05-01', max='2026-05-01'  │ │
│   ├──────────────────────────────────────────────────────┤ │
│   │ Granule 1:                                           │ │
│   │   col1.bin bytes [L1..L1+L1']                        │ │
│   │   ...                                                │ │
│   │   Mark file: col1: min=100, max=199                  │ │
│   └──────────────────────────────────────────────────────┘ │
│                                                            │
│   Primary key index (sparse): one entry per granule        │
│     granule_0 → (col1=10, col2='2026-05-01')               │
│     granule_1 → (col1=100, col2='2026-05-01')              │
│     ...                                                    │
└────────────────────────────────────────────────────────────┘

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 主键索引 在列式 OLAP 引擎中，其含义与 OLTP B+ 树根本不同。它是 稀疏的 ——每个粒度一个条目，而非每行一个条目。对于 8192 行的粒度，主键索引比 OLTP 的逐行索引小约 8000 倍。查询计划使用主键索引二分搜索到正确的粒度，然后在粒度内扫描。点查找无法用这种方式在 <1 ms 内找到单行——它们在 <1 ms 内找到一个 granule （约 8000 行），然后还需扫描约 8000 行。

这就是 OLAP 对点查找来说是反模式的原因：索引粒度在设计上就是错误的。

4.4 向量化执行与 SIMD

经典执行模型—— 逐元组处理 ——通过算子树每次处理一行（Volcano 风格迭代器）。每行在每个算子边界都要承担虚函数调用、分支预测失败和 L1 缓存未命中的开销。

现代模型—— 向量化执行 ——每次通过各算子处理一个 批次 （通常为 1024 或 4096 个值）。算子的内层循环变成对列值数组的紧凑标量或 SIMD 向量循环。

具体示例：过滤 WHERE price > 100.

// Tuple-at-a-time (slow)
while (next_row(row)) {
  if (row->price > 100) emit(row);
}
// Each iteration: virtual call, branch, possible cache miss.
// Throughput: ~10–50 M rows/sec/core.

// Vectorized (fast)
for (int i = 0; i < 1024; i++) {
  selection[i] = (price[i] > 100);
}
emit(selection);
// Inner loop: 1024-element-wide SIMD compare + bitmask in <100 nanoseconds on AVX-512.
// Throughput: ~1–5 GB/sec/core on integer columns.

SIMD 指令。 Intel AVX-512（高级向量扩展）每条指令处理 64 字节——16 个 int32 值、8 个 int64 值、64 个 byte 值。ClickHouse、DuckDB、Databricks Photon、BigQuery 的 Capacitor 引擎均密集使用 AVX-512（或旧硬件上的 AVX2）。聚合（SUM、COUNT、MIN、MAX）、过滤（谓词求值）、算术运算（价格 × 数量）和哈希（group-by）均可并行化。

向量化执行 + 紧凑列式布局 + SIMD 正是现代引擎能以 1–10 GB/秒/核扫描的原因。这比在相同硬件上的逐元组处理快 10–100 倍。整个"列式 OLAP 很快"的故事就建立在这一基础之上。

4.5 Parquet 文件结构

Parquet 是最主流的磁盘列式格式。布局（单文件）：

Parquet file (typical, 128–1024 MB on disk):

┌────────────────────────────────────────────────────────────┐
│ Magic number: "PAR1" (4 bytes)                             │
├────────────────────────────────────────────────────────────┤
│ Row Group 0  (~128 MB uncompressed, ~1 M rows)             │
│   ┌──────────────────────────────────────────────────────┐ │
│   │ Column chunk for col1                                │ │
│   │   ┌────────────────────────────────────────────────┐ │ │
│   │   │ Page 0: dictionary page                        │ │ │
│   │   │ Page 1: data page (encoded values)             │ │ │
│   │   │ Page 2: data page                              │ │ │
│   │   │ ...                                            │ │ │
│   │   └────────────────────────────────────────────────┘ │ │
│   │ Column chunk for col2  → ...                         │ │
│   │ Column chunk for col3  → ...                         │ │
│   │ ...                                                  │ │
│   └──────────────────────────────────────────────────────┘ │
├────────────────────────────────────────────────────────────┤
│ Row Group 1                                                │
│ Row Group 2                                                │
│ ...                                                        │
├────────────────────────────────────────────────────────────┤
│ Footer:                                                    │
│   File metadata:                                           │
│     Schema definition                                      │
│     Row groups[]:                                          │
│       - byte_offset, byte_length                           │
│       - num_rows                                           │
│       - column_metadata[]:                                 │
│           - col_name, encoding, codec                      │
│           - statistics: min, max, null_count               │
│           - byte_offset, compressed_size, uncompressed_size│
│           - bloom_filter offset (Parquet 1.12+)            │
│   Footer length (4 bytes)                                  │
│   Magic: "PAR1" (4 bytes)                                  │
└────────────────────────────────────────────────────────────┘

关键特性：

  • 自描述。 每个文件可独立读取。文件尾部持有完整 schema + 每个行组的统计信息。
  • 行组相互独立。 查询可根据文件尾部统计信息完全跳过某个行组，跨不同文件的行组并行处理，并仅读取投影列的列块。
  • 页是 I/O 单位。 页大小约 1 MB。如果查询只需要一个 100 万行行组中的前 1 万行，则每个投影列只需读取前约 1 MB。
  • 尾部在文件末尾。 对流式写入不友好（必须写完所有行组才能写入尾部），但惯例是先 seek 到文件末尾的 8 字节，读取尾部长度和魔术数字，然后 seek 回去读取尾部，再 seek 到数据区间。 在数据读取开始前需要 3 次 S3 往返 ——这是小文件众多场景的性能陷阱。
  • ORC（优化行列式存储）是另一种选择。 源于 Hive/Hortonworks。结构大体相同（文件含 stripe，类似 Parquet 行组；stripe 内按列存储；尾部在末尾）。针对 Hive 时代的谓词下推做了更激进的优化。现代湖仓以 Parquet 为标准，因为其引擎支持更广泛；ORC 在较旧的 Hadoop 体系中仍较常见。

4.6 Iceberg 清单树——Parquet 之上的表语义

单纯的 Parquet 只是"S3 中的文件"。要构建真正的表——具有 INSERT, UPDATE, DELETE，快照隔离、schema 演化、时间旅行——你需要在文件之上构建一个元数据层。Iceberg 的解决方案：一个 多层 manifest 树，全部存于对象存储，全部通过原子指针交换来更新。

Iceberg metadata hierarchy (in S3 alongside the data):

table_root/
├── metadata/
│   ├── v1.metadata.json  ← initial metadata
│   ├── v2.metadata.json
│   ├── ...
│   └── version-hint.text ← points to latest version number
│
│   v42.metadata.json:
│     {
│       "table_uuid": "...",
│       "current-snapshot-id": 12345,
│       "snapshots": [
│         { "snapshot-id": 12345,
│           "timestamp-ms": ...,
│           "manifest-list": "snap-12345-mlist.avro",
│           "summary": { "added-files": 17, "deleted-files": 3 } },
│         { "snapshot-id": 12344, ... },
│         ...
│       ],
│       "schema": { ... },
│       "partition-spec": { ... },
│       "sort-order": { ... }
│     }
│

│     manifest list (one entry per manifest file):
│     [
│       { "manifest_path": "manifest-001.avro",
│         "added_data_files_count": 5,

│     ]

│     manifest file (one entry per data file):
│     [

│       ...
│     ]

    ├── dt=2026-05-01/part-0001.parquet
    ├── dt=2026-05-01/part-0002.parquet
    ├── ...

调用链： table metadata → snapshot → manifest list → manifest files → data files.

为什么需要三层。 单一的数据文件扁平列表无法扩展——一张 1 PB 的表可能有数百万个 Parquet 文件。两层树结构（manifest list → manifest files）让引擎能够：

  • 读取快照的 metadata.json（1 KB）。
  • 读取 manifest list（1–10 MB）。
  • 跳过分区范围与查询谓词不相交的 manifest（大多数都会被跳过）。
  • 只读取相关的 manifest 文件（每个约 1 MB，列出约 1000 个带统计信息的数据文件）。
  • 利用每文件统计信息跳过数据文件。
  • 只读取匹配的 Parquet 文件，然后利用 Parquet 的 row-group 统计信息跳过其中的 row group。

这是一个 四级 zone-map 层次结构：分区 → 文件 → row-group → page。每层剪枝约 10–100 倍。一张 1 PB 的表在严格谓词下通常只扫描 <100 MB。

通过指针交换实现快照隔离。 一次写事务：

  1. 将新的 Parquet 文件写入 data/.
  2. 写入一个新的 manifest 文件指向这些文件。
  3. 写入一个新的 manifest list 文件，引用旧的 manifest（未变更的）加上新 manifest。
  4. 写入一个新的 v43.metadata.json 引用新的 manifest list。
  5. 原子更新 version-hint.text 为 "43" （或在基于 catalog 的部署中，原子更新该表在 catalog 中的行，使其指向 v43.metadata.json).

第 5 步是 提交点。在此之前，任何读者都看不到新数据。之后，所有新读者都能原子地看到新快照；飞行中的读者继续使用原有快照（它们持有旧的 metadata.json 引用，忽略新的 manifest list）。 无锁、基于对象存储的快照隔离。

时间旅行。 旧快照文件在超过保留阈值之前不会被删除。 SELECT * FROM t FOR VERSION AS OF 12343 在快照 12343 处打开表（manifest list、manifest 和数据文件均不可变且仍存于 S3 中）。Iceberg 默认保留快照 5 天；可配置。

Schema 演化。 每个快照携带一个 schema 版本；列的添加、重命名、删除均为仅元数据操作，会产生带有新 schema 的新快照。旧快照在其原始 schema 下仍可读。

隐式分区。 Iceberg 将分区函数（例如 bucket(16, user_id), day(event_ts)）存储在表元数据中。查询只需指定 WHERE event_ts > '2026-05-01' ，Iceberg 自动应用分区谓词。用户无需指定 WHERE event_dt = '2026-05-01' （Hive 中的分区列）。这将分区与查询语法解耦；分区演化不需要重写查询。

Delta Lake 使用类似的思路但采用不同的元数据格式——一个由 JSON 提交文件和 Parquet checkpoint 文件组成的 _delta_log/ 目录。Hudi 使用带时间轴文件的 .hoodie/ 目录。三者解决相同问题（对象存储中文件上的表语义）；在 2026 年，Iceberg 的 manifest 树因其广泛的引擎支持和 PB+ 规模下的操作简便性而成为主流选择。

4.7 一个完整的端到端查询示例

针对 Iceberg+Trino 数据湖仓的真实查询：

SELECT country, SUM(revenue) AS total_revenue
FROM clickstream_events
WHERE event_date BETWEEN '2026-05-01' AND '2026-05-07'
  AND event_type = 'purchase'
GROUP BY country
ORDER BY total_revenue DESC
LIMIT 20;

表：按 event_date分区，一年内 100 亿行，约 50 列，约 5 TB Parquet。

  1. 解析、规划。 Trino 协调节点构建执行计划： Scan → Filter → Aggregate → Sort → Limit。过滤条件下推至扫描阶段。
  2. 读取表元数据 (v_latest.metadata.json，然后 manifest list 约 5 MB）。
  3. 按分区范围剪枝 manifest。 365 天中的 7 天 → 约 99% 被跳过。
  4. 读取 7 个相关 manifest （共约 7 MB），列出带每文件列统计信息的数据文件。
  5. 剪枝数据文件 以 event_type min/max——约剔除 30%。剩余：约 1000 个文件（约 700 GB），来自约 5 万个文件（约 5 TB）。 约 85% 的字节被跳过。
  6. 调度分片。 协调节点将文件分配给各工作节点，每个 row-group 作为一个任务。
  7. 单工作节点读取。 打开 Parquet 文件（先 HTTP GET footer，再按范围获取数据）；通过 footer 统计信息跳过文件内的 row group（进一步剪枝约 30%）；只读取 4 个投影列；解压页面；解码为 Arrow 向量。
  8. 向量化过滤。 SIMD 对比 event_type == 'purchase' 跨 1024 行的批次（在 AVX-512 上每批约 100 ns）。
  9. 局部聚合。 按 country哈希分组，在每工作节点的哈希表中累积 SUM(revenue) 。
  10. Shuffle 局部结果按 hash(country) % N.
  11. 最终聚合。 按国家汇总局部结果（共约 250 个国家行）。
  12. 排序、限制。 协调节点排序并返回前 20 条。

在 50 个工作节点的 Trino 集群上的挂钟时间： 约 700 GB 有效扫描约需 3–10 秒，主要瓶颈在 S3 GET 延迟。聚合本身在并行化后以约 5 GB/秒/节点的速度仅需约 60 ms 挂钟时间。

字节级总计。 在 5 TB 的表数据中，约 700 GB Parquet 被下载 → 列投影后约 150 GB → row-group 跳过后约 30 GB。最终输出：约 480 字节（20 行 × 24 字节）。 若没有列式存储、manifest 剪枝和向量化执行，同样的查询需读取 5 TB，耗时数小时。有了这些，只需 30 GB、5 秒。

§5. 容量上限

列式 OLAP 的能力，在差异极大的规模下得到充分展示。同一技术体系覆盖六个数量级。

DuckDB 在笔记本电脑上——GB 规模。 一位数据科学家在 MacBook Pro M3（16 GB RAM、NVMe）上的 Jupyter notebook 中运行 DuckDB。从 S3 本地缓存了 50 GB Parquet；约 5 亿行数据的聚合在 1–5 秒内完成。单芯片扫描速率约 6 GB/秒；10 亿行的 TPC-H Q1 约需 2 秒。并发用户数：1。成本：零（开源）。

ClickHouse 自托管——TB 规模。 某 SaaS 分析公司在 AWS i4i.4xlarge（16 vCPU、128 GB RAM、7.5 TB NVMe）上运行一个 16 节点的 ClickHouse 集群。约 120 TB 压缩数据（约等效 5 PB 未压缩）。约 1000 QPS，p99 约 500 ms。通过 Kafka 以约 200 万事件/秒的速率摄入数据。Plausible Analytics 公开表示其整个产品运行在此规模上。

Snowflake 中大型企业——TB 到 PB 规模。 某零售商拥有数千名分析用户。虚拟仓库：4–8 个 size-large 集群（每个约 8 节点），在 BI 高峰时段扩容，夜间暂停。约 50–500 TB 压缩数据；约 5 PB 原始数据。仪表板查询：2–10 秒。ETL：5–30 分钟。并发：约 200–500 个并发查询。成本：典型为 100 万–1000 万美元/年。

Netflix Iceberg + Trino——EB 规模。 S3 上以 Iceberg 格式存储约数百 PB 至低 EB 级别（2023 年披露约 500 PB）。Trino 交互式查询（1000+ 节点），Spark 批处理（每天数万个作业），Flink 流处理。查询从 1 秒的元数据查找到多小时的 ML 训练皆有涉及。Iceberg 于 2017 年诞生于 Netflix，旨在解决该规模下 Hive 元数据扩展问题。

Meta Presto、Google BigQuery——EB 规模。 Meta 在数十万 CPU 上运行 Presto，管理超过 300 PB 数据；这是 Facebook 内部最主流的交互式引擎。BigQuery 可能是目前已知最大的 OLAP 系统——基于 Capacitor/Colossus 存储 EB 级别的客户数据，查询可在数秒内扫描多 PB 数据，每天处理数百万次查询。

LinkedIn / Uber 的 Apache Pinot——实时、多 PB 规模。 LinkedIn Pinot 用于面向成员的仪表板：约 20 万 QPS，p99 <50 ms，数十 PB。Uber Pinot + Druid：峰值约 60 万 QPS，p99 低于 100 ms。这与 Snowflake 的 OLAP 形态不同——通过存算耦合和每列位图索引实现"PB 级数十毫秒"的极速查询，而非 Snowflake 通过廉价 S3 实现的"PB 级秒级"查询。

容量范围：从笔记本电脑上的 GB 级到超大规模的 EB 级——横跨六个数量级。 存储引擎的细节（列式、压缩、向量化）在 DuckDB 进程与 BigQuery slot 上是相同的。扩展的是分区、manifest 分层以及包裹引擎的分布式协调器。

§7. OLAP 固有的难题

七个根本性挑战。无论选择哪种数仓或引擎，这些问题都会出现；示例来自各个领域。

7.1 大规模 Join

一句话总结。 一个 JOIN 两张大表之间的 Join 会触发哈希表或 shuffle，可能超出内存或网络带宽的承载能力。

出现场景。 某营销归因查询将 clicks （1000 亿行）与 purchases （10 亿行）按 user_id进行 join。某风险模型将 transactions （100 亿行）与 accounts （100 万行）按 account_id进行 join。某 ML 流水线将 events （500 亿行）与 feature_table （100 万用户特征）按 user_id.

朴素计划。 交叉积：对每一 clicks 行，扫描所有 purchases。1000 亿 × 10 亿 = 10²⁰ 次比较。宇宙热寂之前都不会有结果。

真正的解决方案。 优化器根据表大小和 join 键选择以下三种策略之一。

  • 广播（复制）join。 当一侧数据量较小（过滤后 <1 GB）时，将小侧复制到每个工作节点；每个工作节点将大侧的数据分片与内存中的小侧进行哈希 join。代价：O(大侧扫描 + N 个工作节点 × 小侧大小)。用于事实表与小维度表的 join（例如 events ⨝ products).
  • Shuffle join。 两侧都很大。按 join 键对两侧哈希分区，将行重新分发到 N 个分区，在每个分区内进行哈希 join。代价：O(两侧扫描 + 两侧网络传输)。当两侧都无法广播时使用。
  • Bucketed join。 若两张表已按 join 键预先分桶（Iceberg 的 bucket(N, user_id) 分区规范），则无需 shuffle——对应的桶直接 join。在适用的情况下，代价为零。

"1000 亿 × 100 万"交叉 join 的灾难。 一个常见的陷阱：优化器错误地将小侧估计为"巨大"，选择了 shuffle join，shuffle 阶段在网络上移动了 1000 亿行。以 10 GB/秒在集群上 shuffle 10 TB 数据 = 1000 秒的纯网络时间。解决方法：给优化器添加显式提示（/*+ BROADCAST(small_table) */），或预先收集统计信息以便优化器选择正确的计划。糟糕的 join 是"本该 10 秒完成的查询耗时 4 小时"这一问题的最大单一成因。

7.2 用于查询规划的基数估计

一句话总结。 优化器选择的执行计划取决于其对每个算子行数的估计；错误的估计导致糟糕的计划。

出现场景。 某查询带有 WHERE event_date = '2026-05-20' AND user_country = 'JP' AND event_type = 'purchase' 条件，根据日期、国家和类型不同，可能匹配 100 行或 1 亿行。优化器必须进行猜测才能选择正确的计划。

朴素估计。 假设列谓词相互独立： selectivity(date) × selectivity(country) × selectivity(type)。通常误差达数量级—— country='JP' 。 event_type='cart_view' 之间存在相关性（日本有更高的电商参与度）；联合选择性并非各自选择性的乘积。

为何会崩溃。 优化器将某 join 的左侧估计为 1000 行，选择广播 join，为哈希表分配 100 MB。实际：1 亿行。哈希表内存溢出（OOM）；工作节点崩溃；查询以不同计划从头重启；总挂钟时间从目标 5 秒增至 30 分钟。

真正的解决方案。 多管齐下：

  • 统计信息收集。 引擎维护每列直方图、NDV（不同值数量）估计和最高频值频率。Snowflake 在微分区写入时自动收集；BigQuery 通过 ZetaSQL 优化器利用元数据中的列统计信息；Spark 需要 ANALYZE TABLE.
  • 多列统计信息。 联合直方图（PostgreSQL 的 CREATE STATISTICS）捕获列间相关性。通常代价较高，选择性使用。
  • 自适应查询执行（AQE）。 Spark 3+、Snowflake 的自适应规划器、Databricks Photon 的运行时自适应。引擎启动一个计划，在前期阶段观察实际行数，然后对后续阶段重新规划。将"计划错误 = 灾难"变为"计划错误 = 缓慢启动，然后重新规划"。
  • 手动提示。 生产关键查询通常携带提示，以在已知统计信息存在误导时覆盖优化器。

7.3 数据倾斜

一句话总结。 即使使用哈希分区，少数键也会占据大多数行；一个工作节点承担了 90% 的工作量。

出现场景。 某点击流 GROUP BY user_id 发现一个账号有 100 亿事件中的 10 亿条。某 SaaS 分析 GROUP BY tenant_id 发现一个超大客户驱动了 30% 的总行数。某多人游戏 GROUP BY game_session_id 有一个病毒式传播的会话带来 100 倍中位流量。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 按 GROUP BY 键哈希分区。键均匀分布到各工作节点——但每个键对应的 行数 分布不均，持有热键的工作节点持有大多数行。

为何会崩溃。 具体示例：100 个工作节点，shuffle 聚合 100 亿行。理论平衡：每工作节点 1 亿行。实际（倾斜键）：一个工作节点持有 10 亿行（该机器人的事件）+ 约 9000 万（其他随机键）≈ 11 亿行。挂钟时间 = (11 亿行) / (1 个工作节点的吞吐量) = 预期的 10 倍。其余 99 个工作节点在 90% 的查询时间内处于空闲状态。

真正的解决方案。 对热键进行 加盐处理。对于已知的热键，在写入时或局部聚合阶段附加一个随机的 0..N-1 后缀。热键变为 N 个"分片"—— bot_user_id:0, bot_user_id:1, ..., bot_user_id:15 ——每个分片落在不同的工作节点上。局部聚合后，最终的 reduce 步骤去盐并合并：每个分片 16 个局部求和 → 每个逻辑键一个总计。

Without salting:                 With salting (N=16):
worker_0 [user_42:1B rows] ⨯     worker_0 [user_42:0, 62M rows] ✓
worker_1 [other 90M rows] ✓      worker_1 [user_42:1, 62M rows] ✓
worker_2 [other 90M rows] ✓      ...
...                              worker_15 [user_42:15, 62M rows] ✓
                                 ↓
                                 Final reduce: sum across user_42:0..15

Trino 和 Spark 具备自动倾斜检测（AQE、动态分区合并）；对于已知的病态情况，手动加盐仍然很常见。

7.4 数据新鲜度与成本

一句话总结。 从"每小时批量 ETL"到"亚秒级流式摄入"可能需要 10–100 倍的计算成本。

出现场景。 对于财务报告，24 小时延迟的每日仪表板是可以接受的；但对于欺诈检测则毫无用处。实时欺诈检测需要对交易的 <1 秒可见性。ML 模型训练需要每小时更新的新鲜特征；在线推理需要亚秒级响应。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 全量流式处理。持续运行 Flink 作业；每 30 秒提交到 Iceberg；对所有内容查询湖仓。

为何会崩溃。 持续的流式提交会产生 小文件。一张每 30 秒提交且有 100 个分区的表每天产生 100 × (86400/30) = 288,000 个文件。一年后：1 亿个小文件。仅文件列举（S3 LIST 每次调用分页返回 1000 个键）就需要数分钟；查询规划变成了"列举 1 亿个键以找到正确分区"。"小文件问题"是湖仓的第一大慢性病。

真正的解决方案。 分级新鲜度：

  • 热层（实时）： 通过 Pinot/Druid 实现亚秒级新鲜度，直接从 Kafka 摄入。保留期有限（约 7–30 天）。
  • 温层（分钟级新鲜度）： Flink → Iceberg，每 1–5 分钟提交一次。后台压缩每晚将小文件合并为较大文件（1 万个 1 MB 文件 → 100 个 100 MB 文件）。
  • 冷层（小时/天级）： 批量 ETL 生成按天分区的表，用于长历史数据；临时分析查询访问此层。
  • 联邦查询 用于统一：仪表板查询将实时层（最近一小时）、温层（最近一天）和冷层（更早数据）联合查询。Pinot 原生支持联邦查询到 Iceberg；Trino 通过连接器同时跨越两者。

成本比：实时层每 GB 的成本约是冷层的 10–100 倍，原因在于（a）需要为始终运行的流计算付费，（b）需要为支持 <100ms 查询的倒排索引付费。仅对需要该新鲜度的数据切片使用它。

7.5 多租户与工作负载隔离

一句话总结。 团队 A 的重量级查询可能使共享集群饱和，降低团队 B 的仪表板性能。

出现场景。 某数据科学家对 100 TB 表的探索性 SELECT * 查询使 Snowflake 共享仓库饱和，冻结了每一个 CFO 仪表板。某临时 Trino 查询拙劣地 join 了三张大表，消耗了集群 80% 的 CPU 长达一小时。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： "请注意操作规范。"超过 10 名用户后就无法扩展；来自行为不当来源的查询很常见。

真正的解决方案。 工作负载隔离原语：

  • Snowflake 虚拟数据仓库。 每个团队或作业类别都有专属的数据仓库——一个独立的计算集群，共享同一存储层。各仓库独立调整规模，某个仓库的负载不会影响其他仓库。常见模式：BI 仓库（自动扩缩容，高峰时段）、ETL 仓库（大规格，夜间）、数据科学仓库（小规格，按需）。
  • BigQuery 预留资源 / slot 池。 按项目或部门预留 slot 容量。空闲 slot 可弹性共享给其他项目（公平共享）；已承诺的 slot 则有保障。
  • Trino 资源组。 每个组的查询获得一定的 CPU/内存份额。行为异常的组可被限制在集群的 30% 以内，防止其耗尽资源。
  • 单查询内存限制。 Trino 的 query.max-memory-per-node，Snowflake 的 STATEMENT_TIMEOUT_IN_SECONDS，BigQuery 的 maximumBytesBilled。强制让问题查询失败，而不是拖垮整个集群。

生产环境 OLAP 平台团队最重要的日常工作是 随着负载演变持续调优这些策略.

7.6 Schema 演进

一句话总结。 在 1 PB 的表上添加、重命名或删除列——无需重写数据。

出现场景。 产品新增一个事件字段，分析侧需要追踪它。某列的类型需要从 INT 与 BIGINT （计数器溢出）修改。为清晰起见需要重命名某列。为符合 GDPR 合规要求需要删除某列（PII_email 必须被永久删除）。

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： ALTER TABLE。在 Hive（Iceberg 之前）， ALTER TABLE 仅更新元数据，但存在 按位置匹配列 的问题——在中间插入列会重新编号后续列，导致读取器误解已有的 Parquet 文件（Parquet 按位置而非名称存储列）。新增 created_at 之间 id 。 name 会使所有已有行看起来发生了偏移，造成无声的数据损坏。

真正的解决方案。 Iceberg 和 Delta 都使用列 ID 而非列位置。每列在建表时有一个稳定的 ID；重命名只改变元数据中的名称 → ID 映射；底层 Parquet 列块通过 ID 识别。在任意位置新增列仅为元数据操作，已有文件不引用新列 ID，读取时默认返回 NULL。重命名列只需更新名称 → ID 映射的元数据，无需重写文件。

对于类型变更（INT → BIGINT），Iceberg 允许将拓宽转换（INT → BIGINT、FLOAT → DOUBLE）作为纯元数据操作。缩窄或不兼容的变更则需要执行"新增列 + 回填 + 删除旧列"的流程。

对于需要 GDPR 级别删除的列删除操作（必须永久不可恢复，而非仅隐藏），需要承担重写成本——但这是一次性任务（OPTIMIZE table ... REWRITE），不会造成阻塞。

7.7 时间旅行与快照保留

一句话总结。 查询"昨天的表状态"对调试、恢复和 ML 可复现性非常有用；但旧快照会增加存储成本和元数据复杂度。

出现场景。 一个 bug 在凌晨 2 点从生产表中删除了 1000 万行数据。一位数据科学家需要复现基于昨天数据训练的模型。合规审计询问"2025-12-31 这张表的数据是什么样的？"

"增加更多节点以扩展写入能力。"分析过程：3 节点 etcd 在 10k 次写入/秒时，p99 = 8 ms。增加 2 个节点变为 5 个；法定人数 = 3；写入需要 2 个从节点确认；最慢的从节点制约每次提交。结果： 每日备份。粒度粗、恢复慢，且往往从未真正测试过。

真正的解决方案。 按快照时间旅行。 Iceberg 保存每次提交的元数据；查询 FOR VERSION AS OF <id> 或 FOR TIMESTAMP AS OF <ts> 可以在该快照的 manifest list 处打开表。代价：旧数据文件必须在保留窗口内保留（不进行垃圾回收）。Snowflake Time Travel：Enterprise+ 版支持 90 天。Snowflake Fail-safe（位于 Time Travel 之后的独立 7 天灾难恢复层）：用于灾难性恢复，需联系 Snowflake 支持团队进行还原。

运维注意事项。 时间旅行的存储成本可能远超表本身的实际大小。一张 1 TB 的表，每小时更新 1% 的行 = 每天产生 240 GB 新文件；90 天后 = 约 22 TB 的保留文件存储，对应 1 TB 的"当前"数据。调整快照保留策略是真实可行的降本手段。

§8. 故障模式

五类典型 OLAP 故障场景及各自的恢复流程。

8.1 查询内存溢出（OOM）

CloudTrail + S3 Access Logs。 一个 GROUP BY 查询的哈希表超出可用 worker 内存。常见原因：(a) 基数估算有误，(b) GROUP BY 键的基数很高（例如， GROUP BY user_id 在 10 亿用户上），(c) JOIN 导致基数爆炸。

故障模式。 worker 崩溃（OutOfMemoryError）。协调器要么让查询失败，要么重试。若多个 worker 同时 OOM，集群会陷入抖动螺旋。

恢复措施。 引擎有以下几种应对机制：

  • 溢写到磁盘。 Trino、Spark、ClickHouse、Snowflake——当内存中的哈希表超过阈值时，将部分状态溢写到本地 NVMe。Trino 的"spill-to-disk"会使大型 GROUP BY 的查询时间翻倍，但可防止 OOM。
  • 自适应分区。 检测偏斜的哈希桶，并在运行时对其重新分区。
  • 单查询内存限制。 快速响亮地失败——让用户缩减查询规模。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 查询是幂等的（只读）；使用调整后的参数从头重试。

8.2 一个坏查询导致数据仓库卡死

CloudTrail + S3 Access Logs。 无谓词的交叉连接。连接三张百亿行表的查询。扫描全量历史数据的有问题的 dbt 模型。

故障模式。 该坏查询耗尽集群 100% 的 CPU/内存/网络。其他查询要么进入等待队列（BigQuery），要么相互竞争而按比例变慢（未启用虚拟仓库隔离的 Snowflake；未设置资源组上限的 Trino）。全公司的仪表盘全线报红。

恢复措施。 两层应对：

  • 实时终止。 运维人员找到问题查询（SHOW QUERIES / Snowflake 的 QUERY_HISTORY / BigQuery 的 INFORMATION_SCHEMA.JOBS）并中止它。手动操作，由 on-call 驱动。
  • 永久修复。 设置单查询限制（最大扫描字节数、最大执行时间、最大内存）和每用户/团队的资源上限，防止问题复发。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 只读查询——中止无任何代价。写查询（MERGE、COPY）具有批量原子性——中止会回滚整个提交，部分数据永远不可见。

8.3 CDC → lakehouse 管道延迟

CloudTrail + S3 Access Logs。 Debezium → Kafka → Flink → Iceberg 管道。Flink 应每 60 秒提交一次。一次背压激增（Iceberg compaction 停滞、S3 限流、Flink GC 暂停）导致提交落后。延迟从秒级增长到分钟级再到小时级。

故障模式。 针对 lakehouse 的查询看到的是陈旧数据。仪表盘显示"今日营收 = $3 M"，而实际是"$5 M"，因为最近 2 小时的订单尚未物化。

恢复措施。 - 检测： 监控 current_time - max(commit_timestamp) 持续监控；超过 5 分钟时告警。\n- 排查： Flink 是否在 GC 抖动？Iceberg 的快照提交是否阻塞在元数据上？S3 是否在限制写入（S3 的前缀级速率限制）？\n- 应对背压： 扩容 Flink TaskManager；调整提交批次大小；必要时对上游 Kafka 进行限速。\n- 追赶： Flink 从已提交的 Kafka offset 重放；只要 Kafka 保留时间超过延迟时长，就不会丢失数据。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 Kafka 是 CDC 的持久性锚点。只要 Flink 的 offset 是持久的（Flink 的 exactly-once 检查点——参见 04_stream_processing.md §6），数据就可以恢复。

8.4 Schema 漂移导致下游表损坏

CloudTrail + S3 Access Logs。 上游服务向 purchase_events 事件新增了一个字段。新字段是不可为空的 INT（服务团队的失误）。CDC 管道将新列传播到 Bronze 层。构建 Silver 层的 dbt 模型不知道新列的存在，但其下游的 SELECT * 因此报错。

故障模式。 管道写入开始因类型错误而失败。依赖 Silver 表的仪表盘显示"0 行"，直到 schema 不一致问题得以解决。

恢复措施。 - 在生产者端强制执行 schema 契约： 服务在未经数据平台评审的情况下不应能添加不可为空的字段。\n- Iceberg 的 schema 演进： 由于 Iceberg 使用列 ID，新列以可为空的形式添加，旧行默认为 NULL。不引用新列的下游模型不受影响。只有显式使用新字段的模型才需要更新。\n- Schema-on-read 引擎（DuckDB、带 JSON 列的 Trino）： 可以将 schema 解析推迟到查询时。以牺牲结构严格性换取灵活性。\n- 排查模式： 检查 Iceberg 中的 metadata_log_entries 表，找到引发问题的提交；识别引入破坏性变更的上游管道。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 Iceberg 快照是不可变的；通过时间旅行回滚到漂移前的快照，可在团队修复上游契约期间恢复仪表盘。

8.5 S3 上的文件列举缓慢

CloudTrail + S3 Access Logs。 一张表有 50 个分区 × 100,000 次提交 × 每次提交约 10 个小文件 = 约 5000 万个 Parquet 文件。查询规划需要知道哪些文件与查询谓词匹配。朴素做法：对分区前缀执行 S3 LIST。

故障模式。 S3 的 LIST 每次调用最多返回 1000 个键；列举一个有 100 万个文件的分区需要 1000 次顺序 LIST 调用 × 约 200 ms = 在查询开始前就要花 200 秒进行规划。Iceberg 的 manifest 树对 Iceberg 管理的表避免了这个问题，但 直接查询 Hive 风格分区 Parquet（或被绕过 manifest 的外部写入者污染的 Iceberg 表） 仍可能遇到此问题。

恢复措施。 - 使用真正的表格式（Iceberg/Delta）并配合元数据 manifest；不要依赖 S3 LIST。 这是 lakehouse 从 Hive 迁移到 Iceberg 的最主要原因。\n- Compaction。 后台任务将小文件合并为大文件，将文件总数减少 100 倍。\n- 分区纪律。 不要过度分区（按天分区，而非按分钟分区）。

CloudFront → S3 内部传输费率为 $0.02/GB；CloudFront 出口费率为 $0.085。95% 的缓存命中率可将真实 S3 出口流量降低 20 倍。 S3 中的文件是不可变的；compaction 是一个写入+交换操作，通过 Iceberg 的 manifest 更新以原子方式提交。如果 compaction 中途失败，新文件会成为孤儿文件（由垃圾回收清理）；旧文件仍被当前快照所引用。

§9. 为何不直接在 OLTP 数据库上运行分析

朴素的替代方案："我们已经有 PostgreSQL 了，直接在生产副本上跑分析就行。"以下五个相互交织的原因说明这行不通：

  1. 锁与副本延迟。 一次 30 分钟的分析扫描会持有 MVCC 快照 30 分钟；PostgreSQL 的 autovacuum 无法回收死行版本；表膨胀，写吞吐量崩溃（参见 01_databases.md §7.5）。从副本读取虽可避免锁的代价，但会引入副本延迟。
  2. IOPS 耗尽。 SELECT * FROM orders 扫描超过 1000 亿行会读取约 30 TB 的 B+ tree 页面。在 100K IOPS 下，需要 50 分钟饱和 I/O。同样的扫描在 Snowflake 上只读取约 3 TB 的压缩 Parquet，zone-map 跳过 90%，实际扫描约 300 GB，以每节点 5 GB/秒 × 8 节点的速度仅需约 8 秒。 快 375 倍。
  3. 行主序存储不适合扫描。 一张 200 列的行存储表读取 3 列时，仍然需要读取每行的全部 200 列—— 89% 的 I/O 被浪费。列式存储只读取那 3 列。
  4. 无向量化。 Postgres 逐行处理；ClickHouse 通过 SIMD 内核批量处理 1024 行。即使不计 I/O，CPU 效率差距也达 10–50 倍。
  5. 并发死亡螺旋。 在 OLTP 副本上同时运行十个分析查询，会在缓冲池、WAL writer、I/O 调度器上产生串行化争用。缓冲池是为 OLTP 热数据工作集设计的；分析扫描会将其驱逐；OLTP 延迟 p99 飙升 100 倍。

具体演练。 SELECT date_trunc('day', created_at), COUNT(*) FROM orders WHERE created_at > NOW() - INTERVAL '90 days' GROUP BY 1 在 1000 亿行的 PostgreSQL 上：生产降级 1 小时，副本落后 30 分钟，p99 达到 800 ms（原为 5 ms）。同样的查询在 Snowflake 上，数据存储为 Parquet micro-partition：仅需 5 秒，对 OLTP 源零影响。"数据仓库"这个比喻是刻意的——它是你将数据转移过去处理那些会损害 OLTP 存储的工作的地方。

§10. 扩展维度

OLAP 的扩展方式，体现在两个独立的维度上。

类型 1：更多数据、更多查询（均匀增长）

数据集从 1 TB 增长到 100 TB；QPS 从 10 增长到 1000。工作负载形态不变。

现代解法：计算与存储解耦。 存储在 S3/GCS/ADLS 上几乎可以无限扩展（约 $23/TB/月，S3 标准存储，IA/Glacier 便宜 10 倍；100 PB 数据湖约 $200 万/月）。计算通过增加容量来扩展——Snowflake 可在数秒内将数据仓库从 L 调整为 XL 再到 XXL（无需迁移数据）；BigQuery 增加 slot 预留；Trino 增加 worker。

拐点。 约 1 TB：DuckDB 或小规格 Snowflake 即可满足。约 100 TB：多节点集群；分区裁剪和 compaction 至关重要。约 10 PB：lakehouse 架构不可或缺；Iceberg manifest 树取代 Hive 文件列举。约 exabyte 级（Meta、Google、Netflix）：自定义 catalog 扩展、分层存储（S3 热数据 / Glacier 冷数据）、跨多个 catalog 的联邦查询。

成本故事： 存储线性扩展且成本低廉；计算随查询负载扩展且成本高昂。 一个查询频率低的 100 PB 数据湖，成本低于一个查询频率高的 10 TB 数据湖。

类型 2：单个查询需要更多计算（热点）

一个复杂的 5 路 JOIN 跨越数百 TB，单个数据仓库在合理时间内无法完成。

解法：分布式查询执行。 所有现代 OLAP 引擎都是 MPP（大规模并行处理）。按 row group 或分区实现扫描并行；通过哈希重分布实现 shuffle 并行；通过局部再全局聚合实现聚合并行。

拐点。 增加 worker 数量在超过 shuffle 宽度后收益递减（一个有 100 个 shuffle 分区的查询，约 100 个 worker 是高效的；更多 worker 帮助有限）。网络带宽是 shuffle 密集型查询的瓶颈（10 TB shuffle 跨越 100 个 worker 以 10 Gbps 传输 = 仅网络就需要 8 秒）。解法：预分桶表可消除 shuffle，或调整 JOIN 顺序以减少中间结果规模。对于超大型查询：BigQuery 动态分配数千个 slot；Meta 的 Trino 为每个查询按需调配数百个弹性 worker 节点。

类型 1 + 类型 2 的交互。 若两个维度同时增长，负载隔离（§7.5）就变得至关重要——每个重型查询拥有独立的数据仓库；lakehouse 存储透明地服务于所有数据仓库。

§12. 数据摄入模式

向 OLAP 加载数据的五种模式。每种模式适用于特定的数据新鲜度/数据量/成本场景。

批量 ETL（Airflow → Spark → 数据仓库）

经典模式，在"昨天的数据就够用"的场景中仍占主导地位。Airflow 按每日/每小时节奏调度 DAG（有向无环图）；每个任务是一个 Spark 作业，读取源数据、进行转换，并写入数据仓库（Parquet 到 S3 / COPY INTO Snowflake / bq load）。数据新鲜度：小时到天级。成本：低廉（约 $0.50–$2/TB）。故障处理：幂等重跑。常见于银行、零售商、传统企业的每日/每周 BI。

CDC 流式处理（Debezium → Kafka → Flink → Iceberg）

在不为真正实时付出代价的前提下实现亚分钟级数据新鲜度的现代默认方案。 Debezium （或 LinkedIn 的 Brooklin、Maxwell、原生 CDC）读取 OLTP 源的逻辑复制/binlog，并将每行变更事件发送到 Kafka。 Flink 消费、转换，并以 30 秒至 5 分钟的提交节奏写入 Iceberg/Delta/Hudi。数据新鲜度：亚分钟级。成本：批量 ETL 的 5–10 倍。通过 Flink 的两阶段提交 + Iceberg 原子快照实现 exactly-once（参见 04_stream_processing.md §6）。常见于电商（实时库存）、SaaS（使用量仪表盘）、金融科技（日内对账）。

实时摄取（Kafka → Pinot/Druid 直连）

适用于实时 OLAP 引擎的亚秒级数据新鲜度。Kafka 生产者发布事件；Pinot/Druid 直接摄取，事件在数秒内即可查询。新鲜度：亚秒级。成本：最高（约为批处理的 10–50 倍）——引擎维护每个事件的内存与索引。存储分层：实时分段在内存和本地磁盘中；历史数据存于 S3。常见于面向客户的实时仪表盘（Uber、LinkedIn、Stripe）。

从 S3 批量 COPY / LOAD

适用于来自合作伙伴、供应商、外部管道的周期性大批量数据加载。数据以 Parquet/CSV 格式落入 S3；通过 COPY INTO / bq load / INSERT INTO ... SELECT FROM staged 导入。数据新鲜度取决于上游节奏。成本低廉且具有幂等性。常见于组织间数据共享以及本地部署向云端迁移的场景。

反向 ETL（数据仓库 → 操作系统）

反向操作：将聚合分析结果回写至操作系统。 dbt 物化聚合表（customer_360, lead_score, churn_risk); Hightouch / Census 将其同步至 Salesforce、Marketo、Zendesk 及内部应用数据库。分析产生洞察 → 反向 ETL 将其推送至产品功能（个性化、营销）。数据新鲜度通常为小时级。常见于采用"现代数据栈"的数据驱动产品闭环部署。

§13. 湖仓表格式深度解析

"对象存储之上的表格式"三大竞争者——Iceberg、Delta Lake、Hudi。

Apache Iceberg

起源。 Netflix，2017 年。2020 年晋升 Apache 顶级项目。旨在解决 Hive 在 PB 级规模下原子重命名和元数据扩展性问题。

架构。 元数据树（见 §4.6）：metadata.json → 清单列表 → 清单文件 → 数据文件。每次提交是对最新 metadata.json（或 catalog 行）的原子性指针交换。

优势。 厂商中立（引擎支持覆盖 Trino、Spark、Flink、Snowflake、BigQuery、Athena、ClickHouse、DuckDB、Databricks）；隐式分区（分区函数存于元数据，用户无需显式编写分区谓词）；基于列 ID 的 Schema 演进（安全地添加、重命名、删除列）；快照隔离与时间旅行；分支与标签（Iceberg 1.0+）；清单可扩展至 EB 级（已在 Netflix 和 Apple 验证）。

弱点。 在提交期间 Catalog 可能存在短暂的一致性问题（Hive Metastore、AWS Glue）；现代 Catalog（Nessie、Iceberg REST、Unity）更为健壮。压缩由用户负责——需定期执行 OPTIMIZE 作业，无内置的常驻压缩器。

2026 年主流格式。 被 Netflix、Apple、LinkedIn、腾讯、Adobe、AWS、Stripe 采用。

Delta Lake

起源。 Databricks，2019 年。Linux Foundation，2020 年。

架构。 每张表的 _delta_log/ 目录，包含有序 JSON 文件（每次提交一个），描述对数据文件的增删操作，以及定期生成的 Parquet "checkpoint" 文件，用于快照完整表状态。

优势。 与 Databricks 深度集成（Photon、Auto Loader、Live Tables、Delta Sharing——一等功能）；Z-ordering（多维聚类，在多列谓词剪枝上优于单纯的分区剪枝）；Change Data Feed（通过内置 CDC 输出 table_changes()）；带删除向量的 DELETE 和 MERGE（基于位图的软删除，Delta 3+）。

弱点。 以 Databricks 为中心的生态；可移植性不如 Iceberg 成熟。 _delta_log 在频繁更新的表上可能膨胀较大；checkpoint 频率调优至关重要。

已深度使用 Databricks，或需要 Delta 特有功能时，是强有力的选择。

Apache Hudi

起源。 Uber，2016 年。专为 大量 upsert 操作 的工作负载而生——行程状态、司机位置频繁变更。

架构。 每张表的 .hoodie/ 目录，包含时间线文件。两种表类型： Copy-on-Write (CoW) 重写受影响的 Parquet 文件（读优化，写代价高）； Merge-on-Read (MoR) 将行级 delta 日志追加到 Avro 中，读取时将基础 Parquet 与 delta 实时合并（写代价低，读需承担合并开销）。

优势。 upsert 性能业内领先；增量查询（"自时间戳 T 以来所有变更行"）高效；异步压缩。

弱点。 生态系统较小；运维模型复杂（两种表类型 + 压缩 + 清理 + 归档）。

使用方： Uber（发源地）、TikTok、以及 upsert 量占主导的大型 CDC 驱动湖仓。

格式之争（2024–2026 年现状）

  • 2017–2020: Hive 占主导；Iceberg/Delta/Hudi 同期涌现。
  • 2021–2023: 三足鼎立；各格式相继补齐功能（时间旅行、Schema 演进、删除向量）。引擎支持较为分散。
  • 2024–2026: 融合趋势。 Snowflake 新增原生 Iceberg 读写支持 (2023). Databricks 通过 UniForm 新增 Iceberg 互操作性 ——Delta 文件附加 Iceberg 兼容元数据（2023 年）。 BigQuery 新增原生 Iceberg 支持 （2024 年）。壁垒逐渐瓦解。

战略转变： 存储层正成为商品化资源。Iceberg 是主流互操作格式；各厂商在计算引擎和托管体验上竞争，而非通过存储锁定用户。2026 年典型湖仓通常在 S3 上维护一份 Iceberg 数据，由 Snowflake 用于 BI、Databricks 用于 ML、Trino 用于即席查询、Flink 用于流式写入、DuckDB 用于本地探索。

§14. 查询引擎深度解析

Trino / Presto

起源。 Facebook，2012 年。2013 年开源。 2018 年分叉为两个项目： PrestoSQL（现为 Trino）由原始创始人主导，面向社区；PrestoDB 继续在 Meta 内部发展。如今 Trino 是主流社区分支。

架构。 Coordinator + Workers 架构。查询分解为多个阶段（扫描、过滤、局部聚合、Shuffle、最终聚合、排序）。Workers 通过内存管道在阶段间流式传输元组——速度快，但单次查询不具备容错能力。

优势。 联邦查询（约 50+ 个连接器：可在一条 SQL 中查询 Postgres + Iceberg + Kafka + Cassandra）；交互式查询速度快（小查询亚秒级，大查询秒级）；经 Meta、Netflix、LinkedIn、Lyft、Uber、Pinterest、Airbnb 大规模验证。

弱点。 默认不支持查询中途容错（直到近期推出容错执行模式）；某个 Worker 在长查询过程中宕机 → 整个查询需重启。默认使用全内存 Shuffle；内存压力是重要调优参数。

规模。 Meta 以 EB 级规模运行 Presto；LinkedIn 以每集群数百个节点、PB 级规模运行 Trino。

Spark SQL

起源。 Databricks（Berkeley AMPLab），2014 年。

架构。 基于 Spark 执行器框架之上的 RDD（弹性分布式数据集）/ DataFrame 模型。查询计划转化为阶段 DAG；故障仅重启失败的分区。

优势。 通过 RDD 血统实现容错（对长时间批处理至关重要）；通用性强（SQL、DataFrames、ML、Streaming 合一）；湖仓原生支持（Iceberg、Delta、Hudi 一等支持）；自适应查询执行（基于观测到的阶段大小进行运行时重新规划）。

弱点。 短交互查询延迟高于 Trino（JVM/任务开销）；运维负担较重。

规模。 Netflix 使用 Spark 以 EB 级规模进行 ETL；Databricks 以 Photon（向量化 Spark SQL 替代引擎）运行 SQL 工作负载，速度与 Trino 相当。

DuckDB

起源。 荷兰 CWI 阿姆斯特丹，2018 年。与 MonetDB / Vectorwise 同出一门。

架构。 单进程嵌入式库。向量化引擎受 MonetDB/X100 启发。可直接从 S3 读取 Parquet、CSV、Arrow、JSON、Iceberg。

优势。 零配置（pip install duckdb）；速度快（TPC-H Q1 在 MacBook M3 上对 10 亿行执行耗时 2 秒，扫描速率约 6 GB/sec）；无需集群（仅一个库）；多语言绑定。

弱点。 单节点上限（大型机器约 10 TB）；单用户并发。

发展方向。 Mother Duck（云端 DuckDB）、无服务器分析、dbt 本地开发。DuckDB 正成为分析的"第一层"——分析师将 TB 级 Parquet 数据拉到笔记本电脑进行探索，再推送至 Snowflake/BigQuery 用于生产。

Snowflake 引擎

架构。 专有引擎。计算（虚拟数据仓库）与存储（S3 微分区）解耦。每个虚拟仓库配有本地 SSD 缓存。用 C++ 实现的自研向量化引擎。元数据由 FoundationDB 支撑。

优势。 零拷贝克隆（克隆一个 100 TB 数据库仅是元数据指针复制）；Time Travel + Fail-safe（内置 90 天快照恢复）；多集群仓库（一个虚拟仓库可有 N 个集群以支持并发）；跨账户数据共享无需复制；内置 MERGE/UPDATE/DELETE。

弱点。 历史上格式封闭（Iceberg 互操作性的引入部分缓解了此问题）；高并发下按积分计费可能较昂贵。

BigQuery

起源。 Google，2010 年（公开）。基于 Dremel （论文 2010 年）。存储基于 Capacitor （Google 的列式格式）位于 Colossus （Google 的分布式文件系统）之上。

架构。 无服务器架构。查询进入全局 Slot 调度器；Slot（CPU 单元）从共享池或预留容量中分配。存储与计算的解耦程度达到极致——即便在一次查询内，Slot 与存储也完全独立。

优势。 真正的无服务器（无集群，无节点）；单次查询可达超大规模（PB 级数据秒内扫描完毕）；与 GCP 深度集成；BigQuery ML 支持在 SQL 中训练模型。

弱点。 Capacitor 是 Google 专有格式（近期 Iceberg 支持有所缓解）；Slot 模型不透明，成本预测困难；Slot 分配的冷启动会影响亚秒级目标。

规模。 可能是全球最大的 OLAP 系统。单次查询可扫描数 PB；总管理数据量达 EB 级。

§15. 使用场景图鉴

七种不同产品/领域，相同技术家族的差异化应用。

1. 商业智能仪表盘（Tableau、Looker 基于 Snowflake/BigQuery）

某零售商的 CFO 打开 Tableau 仪表盘，查看按地区划分的周营收。Tableau 向 Snowflake 发出 5–10 条 SQL 查询；每条查询扫描数十至数百 GB 的预聚合 Gold 表。延迟：每次查询 2–10 秒。数据流：OLTP → CDC → Kafka → Flink → Iceberg Bronze → Spark → Silver → dbt → Gold；Snowflake 通过 Iceberg 互操作性读取 Gold 层数据。 模式：聚合表（Gold 层）由托管数据仓库服务于交互式仪表盘。

2. 实时分析仪表盘（Uber 订单地图、LinkedIn Feed 分析）

Uber 的运营仪表盘实时展示各城市的供需情况，每隔数秒更新一次。行程状态变更通过 Kafka 流入 Pinot，以低于 100ms 的延迟提供聚合查询服务。LinkedIn 的"谁浏览了你的主页"功能基于 Pinot 构建，规模类似——数十万 QPS，p99 延迟低于 50ms。 模式：流式摄取至实时 OLAP 引擎，实现亚秒级数据新鲜度与低于 100ms 的查询延迟。

3. ML 特征管道（数据仓库 → 特征存储 → 在线服务）

推荐系统。离线特征（user_watch_time_30d、user_genre_affinity_vector）每日由 Spark 从数据仓库计算生成。结果存入特征存储：离线（Iceberg）用于训练时关联，在线（Redis 或 DynamoDB）用于推理时查找。 模式：数据仓库作为离线特征计算层；专用存储用于在线服务。

4. 面向客户的分析（Stripe Sigma）

Stripe Sigma 允许商家对其支付数据执行 SQL 查询。引擎（Stripe 内部基于 Trino 衍生的系统）按 merchant_id 实施行级安全控制。支付数据流：Stripe Postgres → CDC → 数据仓库 → 按商家投影。延迟：每次查询数秒至数分钟。 模式：带行级过滤的多租户 OLAP；商家是最终用户。

5. 群组与漏斗分析（Amplitude、Mixpanel）

产品分析工具允许 PM 构建群组（"3 月注册且已激活且已购买的用户"）和漏斗。Amplitude 和 Mixpanel 背后均为 ClickHouse 集群（已公开披露）。事件实时摄取；群组/漏斗查询是对事件流的带时间窗口语义的 SQL 查询。 event_type/user_id 上的按列压缩可实现 20–50 倍存储节省。 模式：基于 ClickHouse 的事件流分析，配合自定义 DSL。

6. 日志分析（Honeycomb、Datadog、ClickHouse 用于日志）

日志是吞吐量最高的分析工作负载：每天数十 TB 的结构化事件。 Honeycomb 用 Go 构建了自研列式引擎，针对高基数事件属性使用稀疏索引。 Datadog 从 Elasticsearch 迁移至自研列式存储。 ClickHouse 用于日志分析 在 Cloudflare、Uber、ByteDance 广泛部署。 模式：带属性级索引的列式存储，针对"在过去一小时内查找满足谓词 X 的事件"场景进行优化。 参见 10_observability.md §5.

7. 营销归因

营销人员希望获取今日转化用户在各渠道的触点路径。需要将 events （点击、打开、展示——数十亿行）与 conversions （数百万条）按 user_id 和时间进行关联，然后应用时间衰减/马尔可夫链归因计算。实时处理代价过高；数据新鲜度要求为"次日晨报"。 模式：Spark 中的重批量 ETL 将结果物化为 customer_journey 表；BI 仪表盘在 Snowflake/BigQuery 上查询。

§16. 真实案例

在多样化应用场景中以规模化方式提供 OLAP 服务的典型系统。

Snowflake——最大的数据仓库商业公司。 截至 2024 年公开披露，约 10,000 家企业客户；年收入超 30 亿美元。客户涵盖金融（Capital One）、医疗（CVS）、零售（Walmart）。单客户部署规模通常达到数百 TB 至数 PB；最大规模接近 EB 级。Snowflake 的架构（计算与存储解耦 + S3 支撑的微分区 + FoundationDB 元数据）定义了现代云数据仓库的构建范式。

Google BigQuery——多 EB 级无服务器。 Google 内部及客户数据总量估计达数十 EB。单次查询可在数秒内扫描 PB 级数据。被 Spotify、Twitter（pre-X）、Carrefour 及大多数主要 Google 服务所使用。Dremel 论文（2010 年）是现代 OLAP 的奠基性文献。

Netflix Iceberg + Trino——EB 级湖仓。 S3 上以 Iceberg 格式存储约数百 PB（截至 2024–2026 年约为低 EB 量级）。Trino 是交互式查询层（全球约数千个节点）；Spark 用于批量 ETL；Flink 用于流式处理。Iceberg 于 2017 年在 Netflix 创建，以解决 Hive Metastore 的扩展性问题。2023 年披露约 500 PB。

Meta Presto——EB 级数据仓库。 Meta 在数十万颗 CPU 上运行 Presto。公开披露：仓库超 300 PB，每日查询量超 30 万次；Presto 是 Facebook 内部分析的主流交互式查询层。

LinkedIn Pinot——实时 OLAP 的发源地。 Pinot 于 2014 年诞生于 LinkedIn，最初用于"谁浏览了你的主页"功能：在 PB 级数据上实现数十万 QPS、p99 延迟低于 50ms。如今 LinkedIn 将 Pinot 用于面向用户的分析（主页浏览、帖子分析、招聘管理仪表盘）——内部运行数百个 Pinot 集群。

Uber Pinot + Druid。 Pinot 在峰值时为市场及乘客/司机仪表盘提供约 60 万 QPS 服务；Druid 用于时序数据（M3 指标平台）。两者均服务于面向客户的实时分析。

Pinterest Druid——广告分析。 广告主实时查看广告活动的展示量、点击量、转化量。Druid 的每列位图索引非常适合广告属性过滤；处理数十 TB 新鲜数据，查询延迟亚秒级。

ByteDance ClickHouse——互联网规模的事件分析。 公开披露：全球最大规模的 ClickHouse 部署之一——数万颗 CPU、数百 PB 数据，支撑 TikTok 和抖音的内部分析。

Cloudflare ClickHouse——日志分析。 每天摄取数百 TB 的 HTTP 请求日志；存储数万亿条历史请求用于安全/性能调查。同时支撑 DNS Analytics 和 Workers Analytics Engine 等客户产品。

Plausible Analytics ClickHouse。 完整产品（数万个客户仪表盘，数十亿次页面浏览）运行于单个 ClickHouse 集群之上。小团队，大规模，低成本——充分验证了 ClickHouse "运维简便、成本较 Snowflake 低 10 倍"的主张。

Databricks Delta + Photon。 超过 10,000 家企业客户。单个客户可运行多 PB 级的 Delta 湖仓。Photon 是 Databricks 的向量化 C++ 引擎，替代了基于 JVM 的 Spark SQL。

Stripe Sigma。 商家可对自己的支付数据执行 SQL 查询。底层为内部基于 Trino 衍生的引擎，数据以 Parquet 格式存于 S3，并按 merchant_id 实施行级安全控制。

这些数字——从 Plausible 单个 ClickHouse 节点到 Google BigQuery 的多 EB 级规模——跨越六个数量级以上，却基于同一技术家族（列式、压缩、向量化、批量写入时快照原子性）。差异在于运维层（托管 vs 自托管）、数据新鲜度要求（亚秒级 vs 小时级）以及集成方式（联邦 vs 仓库原生）。字节层面的原理如出一辙。

§17. 总结

列式 OLAP 是这样得来的：保留关系型 SQL 的接口，但摒弃行存储机制中一切为支持逐行事务而存在的部分，然后围绕批量扫描重新构建存储引擎：按列存储文件以减少 I/O、按列编解码栈实现 5–20 倍压缩、Zone Map 实现谓词驱动的数据跳过、稀疏粒度索引实现扫描而非点查、向量化 SIMD 执行实现对缓存和流水线友好的聚合，以及以快照原子批量写入语义取代 OLTP 的行级 ACID。其结果是一个能在数秒内完成 SUM ... GROUP BY 万亿行扫描的引擎——而对应地，它在 5 毫秒内按主键返回单行方面则极为糟糕。现代变体分布于：云数据仓库（Snowflake、BigQuery——托管且解耦）、自托管引擎（ClickHouse、StarRocks——自运维且高性能）、实时 OLAP（Pinot、Druid——对流式摄取数据实现低于 100ms 的查询）以及湖仓模式（Iceberg + Trino + Spark 基于对象存储——开放文件格式、多引擎、厂商中立）。它们在字节层面殊途同归：S3 上的列式 Parquet、ZSTD 压缩、组织成清单树以在四个层级对查询进行剪枝（分区 → 文件 → 行组 → 页面），由向量化引擎执行，每次在 SIMD 通道上处理 1024 个值。根据数据新鲜度 SLA、查询延迟预算、运维边界，以及团队在成本与控制权轴线上的位置来做出选择——并记住，OLAP 始终是以 OLTP 为真实来源的派生视图；把源头做好，任何分析存储都可以从其变更日志重建。
