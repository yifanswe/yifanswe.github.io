§1. 什么是认证与身份

认证与身份是一类技术，用于在每次交互中回答两个不同的问题：

  • Authentication (AuthN) ——"这是谁？"通过已知信息（密码）、持有物（TOTP — 基于时间的一次性密码、硬件令牌）、固有属性（生物特征）或链式信任（之前颁发的令牌）来证明身份。
  • Authorization (AuthZ) ——"该身份被允许执行此操作吗？"决定该操作是否被允许：RBAC（基于角色的访问控制）、ABAC（基于属性的访问控制）或 ReBAC（基于关系的访问控制）。

同一个令牌（例如签名的 JWT — JSON Web Token）通常同时承载两者： sub 表明身份； scope/role/aud 约束权限。AuthN 的难点在于凭证存储、MFA（多因素认证）和撤销。AuthZ 的难点在于大规模策略评估和跨租户影响范围隔离。

该技术位于公网与业务逻辑之间——由 API 网关在前端承接，通过令牌验证器或 sidecar 分发至各个服务。主要包含两个子问题：

  • 用户认证 ——人类通过密码 + MFA、浏览器 cookie、移动端重定向、社交登录等方式登录。登录频率较低，访问令牌有效期约 10-15 分钟，使用刷新令牌保持会话连续性。
  • 服务认证 ——微服务网格中机器对机器的认证。短生命周期（约 1 小时）、自动轮换的凭证，在 TCP 层进行加密身份验证（mTLS — 双向 TLS，SPIFFE — 面向所有人的安全生产身份框架）。无需人工介入。

不在范围内：端到端加密（Signal 协议、X3DH、双棘轮——属于独立类别，用于保护消息内容不被平台读取）；静态数据的密钥管理（用于 DEK/KEK 的 KMS——数据/密钥加密密钥；认证使用 KMS 托管签名密钥，但数据加密是独立问题）；细粒度策略语言（OPA — 开放策略代理，Rego — 身份下游的 AuthZ 引擎）。

此技术不适用的场景：高频应用内部信任场景（每个方法调用都需要独立的加密验证——认证守护的是请求边界，而非函数调用）；明文内容保护（JWT 签名保护完整性，而非机密性——任何持有令牌的人都可以读取其中的声明）。

核心特性： 认证事件很少发生；认证检查无处不在。 用户每周可能只登录十次，但他们的身份每天要在数千次 API 调用中被验证。该技术必须将验证路径压缩到微秒级，同时确保颁发路径的正确性。

§2. 各方案的固有保证

每种方案都做出了不同的权衡。

JWT（签名的 Bearer 令牌）。 提供无状态验证——任何持有 IdP（身份提供商）公钥的一方无需联系颁发者即可验证。提供完整性保护、篡改检测和内置过期机制。不提供撤销能力——一旦颁发，在 exp 之前始终有效，无论发生什么。不提供机密性——声明以 base64url 编码，并未加密。必须叠加：短 TTL + 拒绝名单用于撤销； aud 受众隔离以防止混淆代理攻击；密钥轮换基础设施。

Session cookie + 服务端存储。 提供即时撤销能力（删除会话行）。支持会话中途更新声明。便于审计。不提供廉价的横向扩展——每次 API 调用都要访问 Redis。不提供跨域有效性（除非显式联邦）。必须叠加：服务端存储、会话固定保护（登录时轮换 ID）、CSRF（跨站请求伪造）令牌。

mTLS。 在 TLS 握手期间在 TCP 层建立加密身份。身份即连接——无法从捕获的数据包中窃取，因为 TLS 会话密钥不可转移。不提供人类身份、语义范围或细粒度 AuthZ。必须叠加：PKI（公钥基础设施）运维——短生命周期证书（例如 1 小时的 SPIFFE SVID——SPIFFE 可验证身份文档）、自动轮换、CRL（证书吊销列表）或 OCSP（在线证书状态协议）。

OAuth 2.0 / OIDC (OpenID Connect)。 提供委托授权（OAuth）和联合身份（OIDC）——让用户使用 IdP B 上的账号登录应用 A，而无需将密码交给 A。标准化流程。不提供开箱即用的安全默认配置（常见陷阱：隐式流泄漏令牌、公共客户端缺少 PKCE、未正确校验 aud）。OAuth 是一种流程框架，而非撤销系统。

SAML（安全断言标记语言）。 基于 XML 的浏览器联合认证，在企业 SSO（单点登录）中已成熟应用。不适用于移动端或服务间场景。现代团队在新项目中逐步迁移至 OIDC；SAML 仍用于遗留企业 B2B 场景。

密码 + KDF（密钥派生函数）。 Argon2id/bcrypt/scrypt 通过工作因子保护，防止数据库泄漏后遭到离线破解。加盐可抵御彩虹表攻击。但无法防范网络钓鱼、凭证填充或密码复用——KDF 仅在泄漏发生后才发挥作用。必须叠加防御：MFA、已泄漏密码检测、速率限制，理想情况下使用 passkeys。

生产系统通常组合使用这些方案——JWT 用于热路径，session cookie 用于浏览器撤销，mTLS 用于服务网格，OAuth 流程用于跨应用登录，Argon2id 用于每次登录时的密码验证。

§3. 设计空间

有状态 vs 无状态 vs 混合

模式 令牌内容 每请求开销 撤销 适用场景
有状态 （session cookie） 不透明 ID（32 随机字节） Redis/DB 查询 ~1ms 删除该行记录 单域浏览器应用（银行门户、GitHub Web）
无状态 （已签名 JWT） 已签名声明 本地加密验证 ~80-150µs 困难——拒绝列表 + 短 TTL 微服务网格、移动应用
混合 （不透明 + 缓存） 不透明 ID，声明存于缓存 缓存命中 ~1ms 缓存 DEL Stripe API keys、PayPal、Google LOAS

混合模式在需要 JWT 式可移植性与即时撤销的场景中非常普遍：PayPal、Stripe、Google 内部的 LOAS（低开销认证系统）均使用短不透明令牌，并将声明存于快速分布式缓存。缓存热命中；极少情况下的缓存未命中会回退到令牌自省。

签名算法

算法 类型 密钥长度 签名 验证 签名大小 模拟程序启动时会进行的重定基址操作。
HS256 （HMAC-SHA256） 对称 256-bit ~2µs ~2µs 32 B 单一信任边界
RS256 （RSA-SHA256） 非对称 2048-bit ~3ms ~80µs 256 B 联合认证——IdP 签名，多方验证
ES256 （ECDSA P-256） 非对称 256-bit ~150µs ~150µs ~70 B 移动端、IoT、对传输大小敏感的场景
EdDSA （Ed25519） 非对称 256-bit ~80µs ~150µs 64 B 现代默认选择——确定性，免疫 nonce 重用问题

默认选择： RS256 或 EdDSA。 当多于一方需要验证时，HS256 会失败（共享密钥是致命缺陷）。ECDSA 的 nonce 重用陷阱曾造成真实灾难（索尼 PS3，2010 年，一个错误的 nonce 导致全部签名密钥泄漏）；HSM（硬件安全模块）可以规避此问题，但 EdDSA 的确定性从根本上消灭了这类 bug。

推荐选择： 所有新流程均使用 Authorization Code + PKCE（代码交换证明密钥）。 OAuth 2.1 强制要求所有客户端使用 PKCE。

令牌格式、存储与 MFA

令牌格式：JWT（自包含）、opaque（服务端查询）、PASETO（JWT 的竞争方案，无算法混淆陷阱）、Macaroons（可委托、可衰减的 caveats——Google 内部使用，Tarsnap）。Session 存储：单实例内存（~1µs）、Redis/Memcached（~1ms，默认）、MySQL/Postgres（~5-50ms，用于审计追踪）、仅 cookie 签名（0ms，无状态边缘）。

MFA 因子强度排名（弱→强）：SMS OTP（弱——存在 SIM 卡劫持风险）、TOTP（RFC 6238——Google Authenticator）、推送通知（Duo、Okta Verify）、硬件令牌（YubiKey、Titan）、WebAuthn/FIDO2/passkeys（防钓鱼——凭证绑定到来源）。现代默认选择： WebAuthn/passkeys。 Apple、Google、Microsoft 均已原生支持。

§4. 字节级原理

这才是深度所在。任何人都能说出"JWT"或"OAuth"；真正的参考资料要追溯到字节层面。

4a. JWT 结构解析

三段 base64url 编码的内容，用点号连接： header.payload.signature.

线路格式（为便于阅读已换行）：

eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6ImtleS0yMDI2LTA1In0
.
eyJpc3MiOiJodHRwczovL2lkcC5leGFtcGxlLmNvbSIsInN1YiI6InVzZXItNDIi
LCJhdWQiOiJhcGkuZXhhbXBsZS5jb20iLCJleHAiOjE3MTUwMDA5MDAsImlhdCI6
MTcxNTAwMDAwMCwianRpIjoiYjM2ZDQ4MSJ9
.
VsbGgZkYZGW5XYsTuYg5fkH3vY3p9_lKqGo7n8sZ9wW...

解码后的 header（JSON）： {"alg":"RS256","typ":"JWT","kid":"key-2026-05"} — kid （key ID）告知验证方使用哪个公钥。

解码后的 payload（声明，RFC 7519 §4.1）：

{
  "iss": "https://idp.example.com",   // issuer — required, validated
  "sub": "user-42",                    // subject — user ID
  "aud": "api.example.com",            // audience — required, MUST be validated
  "exp": 1715000900,                   // expiration epoch
  "iat": 1715000000,                   // issued at — useful for per-user revocation
  "nbf": 1715000000,                   // not valid before
  "jti": "b36d481",                    // unique token ID — key for denylist
  "scope": "read:profile write:posts"
}

签名： RS256_sign(SHA256(header_b64 + "." + payload_b64), private_key)，然后进行 base64url 编码。RS256 在 base64 之前恰好生成 256 字节；ES256 约 70 字节；HS256 32 字节。

4b. RS256 验证路径，逐字节分析

Receive: "eyJ...header.eyJ...payload.VsbGg...signature"

Step 1: split on "." → header_b64, payload_b64, signature_b64
Step 2: base64url-decode header → JSON {"alg":"RS256","kid":"key-2026-05"}
Step 3: look up public key by kid in JWKS cache (JSON Web Key Set,
        fetched from https://idp.example.com/.well-known/jwks.json,
        refreshed hourly) → RSA public key (n, e)
Step 4: compute SHA256(header_b64 + "." + payload_b64) → 32-byte digest
Step 5: base64url-decode signature_b64 → 256-byte signature bytes
Step 6: RSA verify: signature^e mod n == PKCS#1-v1.5-padded digest?
Step 7: base64url-decode payload → JSON claims
Step 8: validate claims:
   - iss == expected_issuer (constant-string compare)
   - aud includes this service
   - exp > now() - leeway (typical leeway: 60s)
   - nbf <= now() + leeway
   - jti not in revocation denylist (hashset lookup, ~50ns)
   - iat >= min_iat[sub] (per-user revocation cache lookup)
Step 9: return claims to application

总 CPU 开销：~80µs RS256 验证 + ~5µs JSON 解析 + ~50ns 哈希集查询 ≈ 每次验证约 ~100µs。

4c. RSA 数学原理（一段话）

选取两个大质数 p, q （各 1024 位用于 RSA-2048）。计算 n = p · q, φ(n) = (p-1)(q-1)，选取 e （几乎总是 65537——小、质数、二进制稀疏），计算 d = e⁻¹ mod φ(n)。公钥 = (n, e)；私钥 = d。签名： s = digest^d mod n。验证： digest =? s^e mod n。安全性依赖于整数分解的困难性——已知最优算法（NFS——数域筛法）对 2048 位需约 10¹¹ 次运算 n。HSM 将 d 存储于防篡改硅芯片中；签名在 HSM 内部完成，从不暴露 d。RS256 具体流程：SHA-256 生成 32 字节摘要，PKCS#1 v1.5 填充至 256 字节，对填充后的摘要进行 RSA 签名。

4d. 集群规模估算

系统在整个网格中每秒进行 200 万次令牌验证：

  • RS256 验证：2,000,000 × 80µs = 每墙钟秒消耗 160 CPU 秒 = 约 160 核 分布于整个集群。以 64 核主机计算，JWT 验证约消耗 ~2.5 台主机的 CPU——完全可接受。
  • ES256 验证：2M × 150µs = 约 300 核。
  • HS256 验证：2M × 2µs = 约 4 核——成本降低 40 倍，但要求每个验证方持有签名密钥。爆炸半径的毒药。

RS256 比 HMAC 慢 40 倍，但在 2M QPS 下仍只需 160 核，这正是非对称加密成为分布式验证默认选择的原因：CPU 开销微乎其微；架构上的简化（只有 IdP 持有私钥）才是真正的价值所在。

4e. JWT 撤销不变量

根本矛盾：JWT 可在 无需联系颁发方的情况下完成验证。这使其具备可扩展性。但一旦颁发，它将在 exp 之前持续有效——IdP 无法在不破坏无服务端查询模型的情况下"收回"令牌。

生产环境分层设计：

  1. 短 access token TTL + refresh token。 Access token 设置 5-15 分钟有效期。过期后，客户端使用长期有效的 refresh token（在服务端维护状态）获取新的 access token。撤销用户时：使其 refresh token 失效；access token 最多再有效 15 分钟，之后锁定。

  2. jti 通过 Kafka 维护拒绝列表。 在 Redis 中维护已撤销 JWT ID 的集合。需要立即撤销时，向 Kafka 发布 jti ；每个验证方订阅并在约 1 秒内更新本地内存拒绝列表。验证方在每次请求时检查拒绝列表（~50ns 哈希集查询）。集合大小有界：一旦超过 exp，条目自动清除，因此即使每天有 5 万次撤销，集合大小也维持在数百条。

  3. 每用户 min_iat 时间戳。 Redis： user:42 → min_iat: 1715000500。任何 iat < min_iat 早于此时间的令牌均被拒绝。要撤销用户 42 的所有令牌（"全端退出"按钮），将 min_iat 更新为当前时间。每次请求进行一次 Redis 查询（在验证方缓存；约 100MB 的 LRU 缓存可存约 100 万条记录）。

  4. 密钥轮换（核选项）。 轮换签名密钥。所有用旧密钥签名的令牌均失效。仅用于灾难性密钥泄漏场景。

完整方案组合 (1) + (2) + (3)：短 access TTL 是安全底线；通过 Kafka 广播的 jti 拒绝列表处理单个令牌撤销； min_iat 处理"全端退出"的用户级撤销。

4f. Refresh token 存储

Refresh token 是长期有效的 bearer 凭证，应像密码一样谨慎对待。

CREATE TABLE refresh_tokens (
  token_id        CHAR(36)  PRIMARY KEY,    -- visible part
  user_id         BIGINT    NOT NULL,
  hashed_secret   CHAR(64)  NOT NULL,        -- HMAC-SHA256 of secret part
  family_id       CHAR(36)  NOT NULL,        -- rotation chain
  generation      INT       NOT NULL,        -- rotation counter
  expires_at      TIMESTAMP,
  revoked_at      TIMESTAMP NULL,
  INDEX (user_id, revoked_at), INDEX (family_id)
);

向客户端返回的令牌： token_id.secret。服务端存储 token_id 已索引且 HMAC(server_key, secret)。使用时：按 .分割，按 token_id, 查找，进行 HMAC(server_key, secret) 与 hashed_secret 的常量时间比较（非常量时间会泄漏时序信息），检查 revoked_at IS NULL 。 expires_at > now()，颁发新的 access token 并轮换（新的 generation，相同的 family_id，标记旧令牌已撤销）。

family_id + generation 检测 被盗的 refresh token。如果攻击者窃取并使用了某个 refresh token，服务端会颁发新令牌并撤销旧令牌。当合法客户端尝试使用原来的（现已撤销的）refresh token 时，服务端检测到"该家族已推进到更新一代"，并撤销整个家族。OAuth 2.1 §6.1 refresh token 轮换。

4g. OAuth 2.0 authorization code + PKCE，逐字节分析

1. User clicks "Login" in webapp/mobile app.

2. Client generates:
     code_verifier  = random 43-128 char string
                      e.g., "dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk"
     code_challenge = base64url(SHA256(code_verifier))
     state          = random nonce (anti-CSRF)

3. Client 302-redirects browser to IdP /authorize?
     response_type=code&client_id=webapp&
     redirect_uri=https://app.example.com/callback&
     code_challenge=<challenge>&code_challenge_method=S256&
     state=<nonce>&scope=openid profile email

4. IdP shows login page; user submits credentials + MFA.

5. IdP:
   - verifies Argon2id hash (~80ms); MFA challenge if step-up
   - generates auth_code (1 min, single-use)
   - stores: auth_code → {user_id, client_id, code_challenge, scope}
   - 302-redirects browser to:
       https://app.example.com/callback?code=<auth_code>&state=<nonce>

6. Client callback:
   - verifies state matches stored value (anti-CSRF)
   - server-side POSTs to IdP /token:
       grant_type=authorization_code, code=<auth_code>,
       code_verifier=<original verifier>, client_id=webapp,
       client_secret=<secret> (confidential clients only),
       redirect_uri=https://app.example.com/callback

7. IdP /token:
   - retrieves stored code_challenge for auth_code
   - computes base64url(SHA256(code_verifier))
   - MUST match stored code_challenge
   - deletes auth_code (one-time use)
   - mints:
       access_token  (JWT, 15 min, aud=api.example.com)
       id_token      (JWT, OIDC identity claims about user)
       refresh_token (opaque, 30 days, server-side row)
   - returns { access_token, id_token, refresh_token, expires_in: 900 }

8. Client creates session:abc123 → {user_id, tokens} in Redis.
   Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Lax

9. Subsequent API calls: browser sends cookie → gateway resolves
   session → forwards Authorization: Bearer <access_token> →
   API validates JWT locally (no IdP call).

每个环节不可缺少的原因：

  • PKCE：防止授权码拦截。如果攻击者窃取了 auth code（浏览器历史记录、手机上的恶意应用、共享计算机），没有原始的 code_verifier就无法兑换。取代了旧的公开客户端 client_secret仅凭借的流程。
  • state 参数：防止 CSRF。否则攻击者可以将受害者登录到攻击者的账号。
  • 使用 auth code，而非令牌出现在 URL 中：code 经过用户浏览器传输（不可信，存在日志和 Referer 头问题）；令牌仅在服务端之间传递。
  • 一次性使用，auth code TTL 为 1 分钟：缩小泄漏窗口；使用一次后即关闭。
  • id_token 与 access_token（OIDC）：id_token 供客户端应用验证 谁已登录。access_token 供 API 验证 允许哪些操作。混淆两者是常见的 bug 类别。

4h. 密码存储——为何需要加盐 + 慢速 KDF

威胁模型：用户表最终会泄漏（LinkedIn 2012、Adobe 2013、Yahoo 2013、RockYou 2009、Equifax 2017）。假设哈希已公开。什么必须保持不可恢复？

差劲：SHA-256(password)。 GPU 每秒可执行约 100 亿次 SHA-256 运算。8 字符密码 = 95^8 ≈ 6.6 × 10^15 种候选 → 单台设备约 7 天可破解。彩虹表使预计算字典攻击变得轻而易举。

较好：SHA-256(salt + password)。 加盐可抵御彩虹表。但仍有 10 GH/s 的速率；字典密码数秒即可破解。

好：bcrypt(password, cost=12)。 在 2010 年硬件上约 250ms/哈希，在 2026 年硬件上约 50ms。GPU 攻击约 20 哈希/秒/核，因此 95^8 需约 300 万年。cost 因子可调；在下次登录时重新哈希。局限性：bcrypt 是 时间困难的，而非 内存-hard — ASICs（专用集成电路）和 FPGAs（现场可编程门阵列）可以以极低成本对其进行并行计算。

最佳选择：Argon2id，配合 m=64MiB, t=3, p=1. 内存密集型 — 每次哈希需要 64MiB 的 RAM，因此 ASICs 会遇到内存带宽瓶颈。这是 2015 年密码哈希竞赛的默认获胜者。 id 变体属于混合型：既能抵御侧信道攻击（即 i 血统），又能抵御 GPU 并行攻击（即 d 血统）。

数学原理：在 64MiB/hash 的条件下，一块 24GB 的 GPU 可并行运行约 375 次哈希。每次耗时 80ms，约 4700 次哈希/秒/GPU。要破解一个 10 位字符的密码（约 6 × 10^19 种组合），在一个拥有 1000 块 GPU 的集群上需要约 3 亿年。

Salt 会自动编码在输出中： $argon2id$v=19$m=65536,t=3,p=1$<base64-salt>$<base64-hash>。每位用户都会获得唯一的 salt。

集群成本：在峰值 6000 次登录/秒时，6000 × 80ms = 480 CPU 秒/墙上秒 = 480 个核心用于密码验证，外加 6000 × 64MiB = 384GiB 的瞬态 RAM。这就是为什么登录流程要严格限速——攻击者探测用户名可能会导致你的认证层 OOM。

当面试官问"密码存储用 MD5 还是 SHA-256？"时——两者都不用。只用 KDF。这就是全部答案。

4i. 端到端登录流程详解

结合 §4b + §4g + §4h 并标注时序：T=0 用户提交凭据；T=4ms IdP 读取用户行（约 2ms MySQL 分片读取）；T=84ms Argon2id 验证完成（约 80ms 计算）；T=85-2086ms MFA 挑战（约 2 秒供用户输入 TOTP）；T=2089ms IdP 使用 HSM 对 JWT 进行签名（约 3ms RS256 签名）；T=2092ms IdP 插入 refresh_token 行（约 3ms）；T=2096ms webapp 创建 Redis 会话；T=2098ms 返回 Set-Cookie。此后每次 API 调用：约 1ms 网关处的会话查找；约 80µs 后端的 RS256 验证；约 5µs 声明校验；约 50ns 拒绝名单检查。 登录后每次请求的开销：总计约 1.5ms，其中加密约 100µs。密码验证是唯一耗时的操作（80ms），而且每约 10,000 次 API 调用才发生一次。 这种不对称性正是该技术的价值所在。

§5. WebAuthn 与 Passkeys 深度解析

密码是人类有史以来发明的最糟糕的凭据。它们可被钓鱼、可被复用、可被泄露、容易忘记，并存储在暗网上的每一个数据泄露语料库中。2024-2026 年行业向 Passkeys 的转型，是首次在消费级规模上真正尝试彻底消灭密码。理解实际协议——而非仅仅了解营销说辞——是必须的。

5.1 FIDO2 的两层架构

FIDO2（快速在线身份认证 2）由两个规范协同工作：

  • CTAP2（客户端到认证器协议 v2） — 在 客户端 （浏览器、操作系统）和 认证器 （持有私钥的设备：通过 USB/NFC 连接的 YubiKey、Apple Secure Enclave、Windows Hello TPM）之间通信。CTAP2 将依赖方的挑战传递给设备，并将签名后的断言返回。线格式：通过 USB-HID、NFC 或蓝牙低功耗传输的 CBOR（简明二进制对象表示）。
  • WebAuthn（Web 认证，W3C 规范） — 浏览器向网页暴露的 JavaScript API。 navigator.credentials.create() 用于注册， navigator.credentials.get() 用于登录。WebAuthn 与操作系统/浏览器通信，后者再通过 CTAP2 与认证器通信。服务器端， 依赖方 接收一个包含签名断言的 JSON 数据包。

网页从不直接接触原始 CTAP2；认证器也从不接触 WebAuthn JSON。浏览器是两者之间的桥梁。

5.2 设备上的公钥密码学

注册时在认证器上创建全新的密钥对 在认证器上:

1. RP (Relying Party) server sends: rp_id="example.com", user_id, challenge (random 32B).
2. Browser calls navigator.credentials.create({publicKey: {...}}).
3. Browser asks OS/authenticator to:
     - generate a fresh ECDSA P-256 (or Ed25519) key pair
     - bind the pair to (rp_id, user_id) inside the secure element
     - return: { credential_id, public_key, attestation_object,
                client_data_json (contains the challenge) }
4. Authenticator may also prompt for user verification: Face ID, Touch ID, PIN.
5. Server stores: (user_id, credential_id, public_key).
   Private key NEVER LEAVES the authenticator.

登录采用挑战-响应机制：

1. Server generates challenge (32 bytes), sends to browser.
2. navigator.credentials.get(); authenticator looks up credential_id
   bound to rp_id, prompts user verification, signs:
     signature = ECDSA_sign(SHA256(authenticator_data || client_data_hash))
3. Server verifies signature with stored public_key.
4. Done. No password ever transmitted, ever.

"私钥永不离开设备"的特性由安全元件强制保证。在 Apple 设备上，是 Secure Enclave（一个拥有隔离内存和硬件 AES 引擎的独立 ARM 核心）。在 Android 上，是 StrongBox（Pixel/三星高端机型上的独立芯片）或 TEE（可信执行环境）作为备选。在 Windows 上，是 TPM（可信平台模块）。在 YubiKey 上，是 YubiKey 的安全元件硅芯片。操作系统只能获取签名后的断言，无法读取密钥。

5.3 无共享密钥，抗钓鱼

"抗钓鱼"的主张建立在 rp_id 绑定机制之上。当认证器进行签名时，它签名的内容包含 rp_id_hash — 即浏览器报告的用户实际所在 来源 的 SHA-256 哈希值。如果用户访问了 https://exarnple.com （拼写错误的域名，其 m 是 rn），浏览器将 rp_id=exarnple.com 传递给认证器。认证器查找 exarnple.com 的凭据——不存在。没有凭据，就没有签名，也就无法登录。用户 在物理上无法 被钓鱼攻击诱导使用其在 example.com 上的 exarnple.com.

凭据。

对比密码：钓鱼网站展示一个像素级完美的登录页面；用户输入密码；攻击者随即获得密码。用户是最薄弱的环节。Passkeys 将用户从信任环路中移除——浏览器通过密码学手段强制执行来源绑定。 这也是 Passkeys 能够击败凭据填充 (user, rp) 攻击的原因。没有可被填充的共享密钥。每个

对都拥有唯一的密钥。某个网站数据库被攻破后，泄露的只是公钥，对攻击者毫无用处。

§6. Passkeys 之外的无密码认证

Passkeys 是最强的无密码认证选项，但更广泛的无密码认证领域还包含几种较弱的——以及一种可接受的——替代方案。MFA 强度的层次大致如下：passkey（FIDO2/WebAuthn）> 硬件 OTP 令牌 > 带数字匹配的推送通知 > 不带数字匹配的推送通知 > 认证器应用 TOTP > 魔法链接 > SMS OTP > 基于知识的认证（安全问题）。

6.1 魔法链接 https://example.com/auth/magic?token=<random-64-char>魔法链接是一个发送到用户邮箱的一次性 URL：。点击该链接，服务器验证令牌（一次性，约 10 分钟 TTL），用户即完成登录。

  • 安全性：完全取决于用户邮箱账户的安全性。如果邮箱本身已通过 Passkey/2FA 保护，这是可以接受的。如果邮箱仅靠密码保护，魔法链接则继承了这一弱点。
  • 用户体验优势：零阻力的用户引导。无需记住密码，无需安装应用。Substack、Notion、Medium、Slack 均使用魔法链接进行登录。
  • 故障模式：邮件投递延迟（SPF/DKIM 配置错误）；链接令牌通过共享邮箱泄露，以及企业邮件过滤器预取 URL（Microsoft Defender 展开魔法链接并消耗一次性使用次数）；用户在与开始登录不同的设备上点击链接，留下半认证状态的会话。

注意：魔法链接有时被定位为"无密码"认证，但本质上是"以邮箱作为凭据"。如果威胁模型包含邮箱账户被攻陷的情况，魔法链接并无帮助。

6.2 SMS OTP（一次性密码）

通过 SMS 发送的 6 位数验证码。用户在登录页面输入验证码。服务器进行验证（一次性，约 5 分钟 TTL）。

  • 安全性：弱。SMS 并 非 端对端加密；运营商可以看到每条消息。更糟糕的是， SIM 卡劫持 攻击允许攻击者通过贿赂或社会工程学手段，说服运营商将受害者的手机号转移到攻击者控制的 SIM 卡上。2019 年 Twitter CEO 被黑事件（Jack Dorsey）以及 2022 年一系列加密货币交易所被攻陷事件（Coinbase、FTX 用户）均属于 SIM 卡劫持。
  • 用户体验：通用——每部手机都能收到 SMS。无需安装应用。按量来看，仍是部署最广泛的 MFA 因素。
  • 持续存在的原因：即使是弱 MFA，对于批量凭据填充攻击而言也远好于没有 MFA。NIST（美国国家标准与技术研究院）SP 800-63B 自 2017 年起就不鼓励使用 SMS，但未予禁止，因为替代方案并非普遍可用。

结论：作为备用因素可以接受，绝不能作为高价值账户的主要因素。那些将 SMS 伪装成"双因素认证"的银行，是在赌批量攻击者比 SIM 卡劫持团伙更懒。

6.3 认证器应用 TOTP（RFC 6238）

基于时间的一次性密码。服务器和设备共享一个密钥（一个 base32 编码的约 20 字节密钥，在注册时通过二维码下发）。每 30 秒，双方计算：

TOTP = truncate(HMAC-SHA1(secret, floor(now / 30))) mod 10^6

6 位数验证码显示在 Google Authenticator、Authy、1Password 等应用中。用户输入后，服务器计算相同的值并进行比较。

  • 安全性：比 SMS 更强——环路中没有运营商，不存在 SIM 卡劫持风险。比 Passkeys 弱——可被钓鱼（攻击者通过钓鱼页面实时代理验证码），且共享密钥可能被从设备中窃取（已 root 的手机、恶意认证器应用）。
  • 用户体验成本：需要用户安装应用并在注册时扫描二维码。换手机需要重新注册或同步（Authy 和 1Password 提供云同步；Google Authenticator 在 2023 年增加了此功能）。
  • 时钟偏差：TOTP 要求设备和服务器时钟误差在约 30 秒以内。NTP 不准确的设备会无声失败。服务器默认应接受当前时间窗口及相邻的一个窗口（±30 秒窗口）。

定位：在 Passkeys 不可用时，TOTP 是合理的 默认 MFA 因素——可被钓鱼，但远优于 SMS 或没有 MFA。

6.4 推送通知与数字匹配

IdP 向之前注册过的移动应用（Okta Verify、Duo、Microsoft Authenticator）发送推送通知。用户点击"批准"——完成。

  • MFA 疲劳：到 2021-2022 年，攻击者学会了在深夜垃圾式发送批准提示，期望用户为了让提示消失而点击"批准"。Uber 2022 年的数据泄露事件，正是通过对一名承包商实施 MFA 疲劳攻击完成的。
  • 数字匹配缓解措施：登录页面显示一个 2 位数字；推送通知要求用户将该数字 输入 到应用中，而非仅仅点击。Microsoft 和 Duo 现已默认要求此操作。消除了盲目点击的疲劳攻击。
  • 可钓鱼性：仍可通过实时代理被钓鱼（攻击者将用户输入的数字转发到合法的登录页面）。

定位：带数字匹配的推送通知是可接受的；不带数字匹配的是已知漏洞。

6.5 通过 Passkeys 实现魔法链接——融合方案

2025-2026 年的最佳实践是将以下方式结合：Passkey 作为主要凭据，向邮箱发送魔法链接或向配对设备发送 SMS 仅用于恢复，TOTP 或硬件 OTP 用于管理员操作的提权认证。这将 MFA 层次结构内置于 流程之中，而非仅仅关注 因素 ——敏感操作使用强因素，弱因素仅用于长尾的恢复场景。

§7. OAuth 2.0 安全最佳实践

OAuth 2.0 是一个流程框架，而非开箱即用的安全解决方案。它自带"坑"；实践者需要能够一眼识别这些问题。

7.1 PKCE（代码交换证明密钥，RFC 7636）— 强制要求

PKCE 解决的问题是： 我们如何确认发起授权流程的客户端与兑换授权码的是同一个客户端？

传统流程使用 client_secret。这对保密客户端（具有真实密钥后端的服务器端 Web 应用）有效，但对公开客户端（移动应用、SPA）则失效——因为它们无法保守秘密，任何嵌入二进制文件中的值都可能被提取出来。

PKCE 将密钥替换为 动态的 每次流程证明：

At /authorize:
  code_verifier  = random 43-128 char URL-safe string  (kept by client)
  code_challenge = base64url(SHA256(code_verifier))    (sent to IdP)
  → IdP stores code_challenge alongside the authorization code

At /token:
  client sends code_verifier
  IdP computes SHA256(code_verifier), compares to stored code_challenge
  → match: issue token. Mismatch: reject.

为什么有效：攻击者可以窃取授权码（浏览器历史记录、日志文件、移动端拦截重定向 URI 的恶意应用），但如果没有原始的 code_verifier （保存在客户端内存中，第一步从不发送给 IdP），他们就无法兑换。验证器与该流程绑定。

OAuth 2.1 将 PKCE 设为 所有客户端的强制要求，包括公开客户端和保密客户端。S256 挑战方法（SHA256(verifier)）是必须的；旧的 plain 方法已被弃用。

7.2 state 参数 — CSRF 防护

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 state 参数是客户端在 OAuth 流程开始时生成的随机 nonce，并在回调时进行验证。若没有它，攻击者可以自行发起一个 OAuth 流程，在中途截获授权码，然后将受害者的浏览器重定向到客户端的回调 URL，携带攻击者的授权码。受害者的浏览器提交该授权码；客户端兑换后创建会话——但该会话是 攻击者身份的会话。受害者因此登录到了攻击者的账户，其输入的任何数据都会进入攻击者的账户。

修复方法：在流程开始时生成 state （随机，32+ 字节），将其存储在客户端的 session storage 中；在回调时，将 URL 中的 state 与存储值进行比对。不匹配则中止。

7.3 nonce 参数 — OIDC 重放保护

对于 OpenID Connect，客户端还需在 nonce 处传递 /authorize。IdP 将其回显到 id_token 声明中。客户端验证 id_token.nonce == stored_nonce。这将身份断言与特定的登录流程绑定——防止攻击者将旧的 id_token 对新会话进行重放攻击。

7.4 重定向 URI 验证 — 精确匹配

IdP 必须验证请求中的 redirect_uri 与客户端预注册的 URI 完全匹配。常见缺陷：

  • 通配符 — IdP 允许 https://app.example.com/*。攻击者注册 https://app.example.com/.well-known/oauth-callback?... （或在同一来源找到开放重定向），将授权码重定向给自己。
  • 子字符串匹配 — IdP 允许任何以 https://app.example.com开头的 URI。攻击者使用 https://app.example.com.evil.com.
  • 方案不匹配 — IdP 允许 app.example.com/callback （无方案）；攻击者使用 http://... 并在传输途中拦截。
  • 客户端上的开放重定向器 — 客户端存在 /redirect?to=... 端点；攻击者将授权流程通过该端点链式传递，以窃取授权码。

修复方法： 对完整 URI 字符串进行精确匹配，包括协议方案。不允许通配符，不允许部分匹配。客户端域名上的开放重定向漏洞本身也可被独立利用。

7.5 Refresh token 轮换

已在 §4f 中介绍。每次使用 refresh token 都会返回一个新令牌并使旧令牌失效。检测家族复用以发现令牌被盗。OAuth 2.1 §6.1。

7.6 OAuth 2.1 — 整合规范

OAuth 2.1（RFC 草案持续更新至 2024-2026 年）将十年的最佳实践整合为单一的规范性规范：

  • Implicit 流程 — 已移除。 Implicit 流程将 access token 直接返回在 URL 片段中。URL 中的令牌一直是个"坑"（浏览器历史记录、Referer 头、服务器日志）。对于公开客户端，PKCE 增强的授权码流程现在是唯一允许的流程。
  • ROPC（资源所有者密码凭据）— 已移除。 客户端直接收集用户密码并发送给 IdP 的流程。完全违背了 OAuth 的初衷（密码绝不应接触客户端）。已废弃。
  • PKCE — 所有客户端均须使用。
  • Bearer token 放在 URL 查询参数中 — 禁止使用。 Token 只能放在请求头中。
  • 重定向 URI — 必须精确匹配。 不再允许使用通配符。
  • Refresh token 轮换 — 公共客户端必须使用。

建议：从第一天起就针对 OAuth 2.1 设计新系统，即使该规范尚未正式发布。2026 年没有任何已废弃的流程值得支持。

§8. SAML 与 OIDC 的权衡

SAML（安全断言标记语言）和 OIDC（OpenID Connect）解决的是同一个问题——联合身份认证，由 IdP 向下游服务提供方（SP）为用户背书。它们在年代、传输格式和生态系统方面存在差异。

8.1 两种协议概览

8.2 何时必须支持 SAML

如果你向中大型企业销售 B2B SaaS，就必须支持 SAML。每家《财富》500 强企业都有配置了 SAML 联合的 Active Directory 或 Okta。采购清单上写着"必须支持 SAML 2.0 SSO"——没有它，你就过不了安全审查。企业级 IdP 通常是 Active Directory Federation Services（ADFS）、Okta、Azure AD、Ping 或 OneLogin，这些都是原生支持 SAML 的。

对于 B2C、移动优先或 API 优先的产品，可以完全跳过 SAML，直接使用 OIDC。客户群体从不提这个需求。

8.3 常见 SAML 陷阱

SAML 的威胁模型是其最糟糕的部分。该协议涉及解析不可信的 XML、对选定 XML 子树验证签名，以及解析属性声明——而且 每个步骤都曾有过 CVE:

  • XML 签名包装 ——断言已签名，但签名仅覆盖 XML 树的一个子集。攻击者取一个合法签名的断言，将其包装在一个新的 XML 信封中，其中被签名的子树通过 ID 引用，但 可见的 断言（SP 实际解析的那个）未签名且由攻击者控制。SP 验证签名（通过），读取未签名的断言（攻击者的内容），并接受攻击者的身份。CVE-2011-1411（SAML 2.0 多个实现）、CVE-2018-1056、CVE-2022-22556。Microsoft Azure AD、Okta 和 OneLogin 均已针对此漏洞的变体发布补丁。
  • NameID 中的注释注入 ——SAML NameID 是一个文本元素，但 XML 解析器会去除注释。攻击者使用邮箱注册账户 victim@example.com<!-- -->.evil.com；IdP 签发包含该 NameID 的断言；SP 的 XML 解析器则看到 victim@example.com.evil.com 或 victim@example.com 取决于规范化顺序。跨账户身份冒充。Duo Security 于 2018 年披露了此漏洞。
  • XXE（XML 外部实体） ——旧版 SAML 解析器会加载外部 XML 实体，使得恶意断言能触发 SSRF 和文件读取攻击。
  • 时间限定断言 ——SAML 断言包含 NotBefore 。 NotOnOrAfter，但旧版 SP 库未严格执行，导致可以重放过期断言。
  • AssertionConsumerService URL 欺骗 ——类似于 OAuth 的 redirect_uri 验证。SP 必须对此进行验证。

防御手册：使用经过充分验证的库（Shibboleth、OneLogin 的 php-saml、Spring SAML），保持及时打补丁，优先使用 EncryptedAssertion（用 SP 公钥签名并加密）而非仅签名，将整个已签名树作为身份声明的唯一来源，绝不信任未签名的 XML 兄弟节点。

8.4 迁移趋势

2020-2026 年趋势：企业 IdP（Okta、Azure AD）同时支持 SAML 和 OIDC。新集成使用 OIDC，历史集成使用 SAML。要求切换到 OIDC 的客户逐年增多。大多数 SaaS 同时支持两者；部分桥接库（如 Auth0）在统一 API 后面抽象了两者。建议：将 OIDC 作为主要方式，SAML 作为备选，且不要自己编写 SAML 解析器——使用现有库。

§9. API key 与 OAuth 的权衡

API key 和 OAuth scope 解决的是一个相关但不同的问题：机器（有时也是脚本中的人）如何向你的 API 进行认证？

9.1 API key——简单易用，规模化困难

API key 是一种不透明的 bearer token，通常在注册时生成并粘贴到配置中：

curl -H "Authorization: Bearer sk_live_AbCdEfGhIjKlMnOp..." \
     https://api.example.com/v1/charges
  • 优点：发放简单，使用简单。每个开发工具都支持。Stripe、Twilio、SendGrid、Mailchimp 最初都使用 API key。从零到第一次 API 调用只需一分钟。
  • 规模化的缺点:
  • 难以限定范围 ——通常每个 key 拥有完整的账户访问权限。从构建服务器泄露的 key 可以读取所有客户数据。
  • 难以轮换 ——key 经常出现在配置文件、CI 变量、客户侧第三方集成中。轮换需要协调迁移。
  • 难以审计 ——缺乏额外基础设施时，服务器日志只能看到"key_xxx"，而无法知道"哪位工程师出于何种目的签发了这个 key"。
  • "10 万个 key"陷阱 ——成功的产品在多年集成中积累了大量 key。某公司（不具名）发现有 80 万个未撤销的 Stripe 风格 key，且不知道哪些仍在使用。批量轮换需要客户侧协调，往往不可能实现。

9.2 OAuth scope——正规的授权模型

OAuth 2.0 配合 scope 提供：

  • 按授权限定范围 ——OAuth 客户端请求特定 scope（read:profile, write:posts）；用户（或管理员）授权；token 仅限于这些 scope。API 拒绝超出 scope 的调用。
  • 按授权可撤销 ——用户可以撤销特定集成的授权，而不影响其他集成。
  • 审计跟踪 ——每个 access token 都可追溯到特定的 OAuth 客户端（即集成合作方）+ 特定的用户授权。
  • Refresh token ——短期 access token 配合显式 refresh 轮换。

成本：设置需要更多代码。每个集成合作方需注册一个 OAuth 客户端。OAuth 流程约需 5 个端点。面向客户的集成需要同意页面。从零到第一次 API 调用接近一周。

9.3 GitHub 细粒度 PAT——现代混合方案

GitHub 个人访问令牌（PAT）于 2013 年作为经典 API key（账户范围、无过期时间、无资源限制）推出。2022 年的"细粒度 PAT"重新设计保留了 API key 的 用户体验 使用体验（在设置中生成，粘贴到配置），同时添加了 OAuth 的 安全护栏 ：

  • 必须设置过期时间 ——最短 7 天，最长 1 年。不允许"永久"token。
  • 资源 scope ——精确到每个仓库（而非仅账户级别）的访问控制。"此 token 只能读取仓库 X，其他均不可访问。"
  • 权限 scope ——对特定资源（内容、issue、pull request）的读/写粒度控制。
  • 组织管理员策略 ——组织可要求使用细粒度 PAT 并禁止经典 PAT。

这是针对"我们用 API key 起步，现在希望获得 OAuth 级别的安全性而不破坏简单集成体验"的推荐方案：短期有效、资源范围限定、必须设置过期时间、有审计日志。保留粘贴到配置的简便性，去除永久 token 的"手榴弹引脚"特性。

§10. 容量规模参考

不同规模的真实部署：

小型：单个 Keycloak 实例，约 10 万用户。 某 SaaS 初创公司在单台 4 核虚拟机（配 PostgreSQL）上运行 Keycloak（开源 IdP）。负载：10 万用户，1000 日活用户，每天约 1 万次登录 → 平均约 0.1 次登录/秒。验证：约 1 万次 API 调用/天 × 1000 日活 ≈ 120 次/秒。单实例 CPU 约 10%。瓶颈出现在约 100 次登录/秒时——Argon2id 使 CPU 核心饱和。成本：约 50 美元/月。

中型：Auth0/Okta 企业版，每租户约 500 万用户。 某中型市场 SaaS 使用 Auth0（现为 Okta 旗下产品）。500 万用户，50 万日活用户，每天约 25 万次登录 → 峰值约 30 次/秒。每年约 50 亿次 API 调用 → 平均约 150 次/秒。多租户共享基础设施；每个租户有自己的 JWKS 端点，但共享验证层。瓶颈：每租户速率限制（约 1000 次登录/分钟）。成本：每月 5000 至 5 万美元，取决于 M2M（机器间通信）量。

大型：Google Identity，约 30 亿用户。 每天约 1.5 亿次登录 → 平均约 1700 次/秒，峰值约 6000 次/秒（昼夜比约 3.5 倍）。跨 Google 服务每天约 300 亿次 API 调用 → 在全球集群中平均验证约 35 万次/秒，峰值约 200 万次/秒。 登录与验证的比例约为 10,000:1。 这个比例决定了技术选型。

为处理 200 万次验证/秒：

  • JWT 验证通过 library + JWKS 缓存分散到每个服务； 无集中验证层。160 个 CPU 核心分布在数千台主机上。
  • 公共 JWKS 端点在边缘缓存，TTL 为小时级别。
  • 通过内部 pub/sub 进行每用户撤销传播，扇出延迟 < 1 秒。
  • 用户数据存储在 Bigtable 上（按 user_id 哈希分片），用于冷路径查询。
  • Argon2id 验证在专用集群上运行，预留约 5000 个 CPU 核心。
  • BeyondCorp 零信任管控内部服务——每次请求、每一跳都需验证。无 VPN，无隐式信任。

其他参考节点：LinkedIn（10 亿+ 会员，每秒数亿次会话验证，混合 session cookie + JWT）。Apple Sign-In（数亿设备，ES256）。AWS STS（Security Token Service——可能是地球上 QPS 最高的认证接口；每次 AWS API 调用都通过自定义 SigV4 验证 STS 签发的凭证）。GitHub OAuth Apps + Apps（1 亿+ 用户，每天数十亿次 API 调用）。

10,000:1 的比例决定了技术选型：任何在关键路径上向中央数据存储发起每请求查询的方案都无法承受规模压力。

§12. 认证技术固有的难题

7.1 JWT 撤销（核心问题）

朴素的想法："JWT 是无状态的。不用撤销，让用户等到 exp."

失败：用户笔记本电脑上午 9:00 被盗，活跃的 JWT 有效至 9:15。攻击者有 15 分钟的完整访问权限。对于银行应用：灾难性后果。对于企业 SSO：攻击者可清空邮箱。监管机构（HIPAA、PCI DSS、SOX、GDPR）要求可证明的即时撤销能力。

解决方案：分层机制（§4e）——短 access token TTL 限制最坏情况， jti Kafka 上的拒绝列表填补单 token 缺口（全局 < 1 秒），每用户 min_iat 用于"全局登出"，refresh token 在服务端管理。

多领域场景：医疗（数秒内撤销离职临床医生，HIPAA 要求）、消费者社交（通过设置页面"所有设备退出登录"）、B2B SaaS（注销离职员工）。同类问题，紧迫程度不同。

7.2 会话固定攻击

朴素的想法："访问时生成 session ID；登录后保持同一个 ID。"

失败：攻击者访问 app.example.com，获得 session_id=ABC。用 app.example.com/?session=ABC钓鱼受害者。受害者登录。此时 session_id=ABC 已通过认证。攻击者使用 ABC。攻击者即成为受害者。

解决方案: 登录时始终轮换 session ID。 旧 ID 失效；签发新的随机 ID（Set-Cookie）。绝不接受 URL 参数中的 session_id——只接受服务器设置的 HTTP-only cookie。

多领域场景：出现在旧版 WordPress、部分 PHP 应用、早期 Rails（在 reset_session 成为默认行为之前）。任何现代框架中一行代码即可解决，但应当能一眼识别出来。

7.3 重放攻击 / token 窃取

朴素的想法："token 已签名；只需验证签名即可。"

失败：咖啡馆 wifi 上的攻击者从受害者流量中捕获 Authorization: Bearer <jwt> （TLS 配置有误，MITM）。即使数小时后，JWT 仍在 15 分钟 TTL 内。攻击者从自己的笔记本重放相同请求。签名有效，token 有效——服务器放行。

修复方案 （纵深防御）：全程 TLS（传输中无 HTTP）；HTTP-only Secure SameSite=Lax cookie； DPoP （持有证明，RFC 9449）——客户端生成密钥对；每次请求包含一个 DPoP 头，对 URL + 方法 + 时间戳 + access token 哈希进行签名；服务器验证并在时间戳偏差时拒绝； mTLS 绑定 token （RFC 8705）——token 绑定到客户端的 TLS 证书指纹，从不同 TLS 连接发出则无效。

多领域场景：支付平台（Stripe、PayPal）对敏感操作使用发送方约束 token。移动端银行固定证书并积极轮换。消费者社交应用接受较弱的 TLS + 短 TTL 方案，因为用户体验成本超过了边际威胁。

7.4 OAuth confused deputy 问题

朴素的想法："我们向用户签发 token。他们将其发送给任意 API。API 验证签名。"

失败：API A 和 API B 都验证来自同一 IdP 的 token。为 A 签发的 token 被发送给 B；B 信任签名并接受。这样一来，仅被授权"读取日历"（A）的应用就能读取银行数据（B）。代理方（B）对于谁具有权威性感到 困惑 。

解决方案: aud 声明必须强制设置并验证。 每个 token 携带 aud 指明其所对应的 API。API B 会拒绝任何受众中 aud 不包含自身的令牌。IdP 为不同的受众铸造不同的令牌。RFC 8707 将请求受众限定令牌的 resource 参数进行了标准化。

多领域场景：在企业 SSO 场景中最为重要——一个 IdP 面向数十个内部应用，HR 自助服务令牌被重放至薪资管理 API。真实案例：多个 Salesforce CVE、BetterHelp 2023。

7.5 签名密钥泄露

朴素的想法："密钥存放在 HSM 中，无法被窃取。"

失败：内部攻击、HSM 固件漏洞、供应链攻击（真实案例：Infineon TPM ROCA 漏洞，CVE-2017-15361，导致数百万 RSA 密钥被削弱）。或更简单的情形：一条编写不当的 CI/CD 流水线短暂地将密钥提取出来。私钥一旦泄露，攻击者即可为任意用户铸造任意令牌，造成全面沦陷。

解决方案: 将密钥轮换纳入常规操作，而非应急措施。 JWKS 中公布多个密钥（不同的 kids）；验证方缓存所有密钥。使用新密钥签名；旧密钥在最长令牌 TTL 加缓冲期内继续有效。正常情况下每季度轮换一次。准备好"紧急轮换"应急预案——在数分钟内将旧密钥从 JWKS 中移除，以该密钥签发的令牌即告失效。为不同受众使用 不同的 密钥，避免单一密钥泄露殃及整个体系。对每次签名操作使用 HSM 并记录审计日志；配置异常检测（"为何在 30 秒内铸造了 1000 万个令牌？"）。

多领域场景：AWS、Microsoft 以及所有主要公共 CA 均按照 CA/Browser Forum 基线要求，严格按计划轮换签名材料。业界通行做法。

7.6 分布式验证延迟

朴素的想法："每次 API 调用都会访问 Redis 查询会话，再查用户数据库，约 5ms。"

失败：在 200 万 QPS 下，即便 1ms 的 Redis 延迟也可接受——但跨区域调用（美西用户，欧洲服务）每次请求会额外消耗 100ms 往返时延。更严重的是：Redis 成为每个请求的单点故障。

解决方案: 在请求边缘进行无状态验证。 JWT 使用缓存的 JWKS 密钥在本地验证（约 100µs CPU 开销）。仅在需要吊销证据时才进行状态查询（属于罕见情况）。吊销黑名单通过 Kafka 实现 最终一致性 ，收敛时间约 1 秒——在吊销后的 1 秒窗口内，部分验证方可能仍接受该令牌。这是明确的权衡取舍：以 1 秒的过期验证换取避免 Redis 出现在每条热路径上。

对于 1 秒容忍时间过长的场景（银行、管理员操作），在关键操作上附加"实时检查"——调用 Redis 确认会话仍然有效。高安全性流程的显式代价，仅在冷路径上使用。

多领域场景：信息流 API 可容忍 1 秒的数据过期；电汇确认需进行实时 Redis 检查；微服务网格完全无状态运行；内部管理控制台进行双重校验。

7.7 时钟偏差导致 exp 验证失败

朴素的想法："检查 exp > now()."

失败：服务器时钟存在漂移。验证方 B 的时钟比 IdP 快 90 秒。一个刚刚铸造的、设置了 exp = now() + 900 看起来对 B 已过期。导致集群中部分节点出现误拒绝级联。

解决方案: 配置时钟偏差容忍度并严格执行 NTP 同步。 JWT 库接受一个 leeway 参数（通常为 60-120 秒），对 exp 加以宽限并对 nbf/iat减去相应时间。所有主机运行 NTP（网络时间协议），策略要求漂移小于 1 秒。监控在主机漂移超过 5 秒时发出告警。注意：宽限值过大（如 600 秒）会使短期令牌的 exp 失去意义。60-120 秒是合理区间。

§13. 授权框架深度解析

AuthN 验证 谁；AuthZ 决定 能做什么。规模化场景下，三大授权体系占据主导地位，各自拥有不同的数据模型、评估成本和运维特性。

13.1 RBAC（基于角色的访问控制）

最简单的模型：用户拥有角色，角色拥有权限，权限控制操作。NIST RBAC 标准（RFC 7642），自 20 世纪 90 年代以来企业应用的基石。

user:alice → role:editor
role:editor → permission:document.write

检查逻辑是一次连接查询："Alice 是否拥有具备 document.write权限的角色？"——通常是对三张表（users, user_roles, role_permissions）进行 SQL 查询，或 Redis 中的反规范化查找。

  • 优点：易于建模，便于向合规审计人员解释（"给我看所有拥有管理员角色的人"），所有 IdP 和访问管理平台均开箱即用。层级角色（经理 ⊇ 员工）可良好组合。
  • 缺点:
  • 角色爆炸 ——对于任何具有合理复杂度的业务，"编辑者"一个角色是不够的；最终会出现"A 组文档编辑者"、"欧洲区编辑者"、"客户 X 租户编辑者"等。真实企业环境中角色数量可能超过 50,000 个。
  • 无资源级别粒度 ——Alice 是 editor。针对哪个文档？RBAC 无法回答；需要额外添加对象级检查。
  • 继承关系变得复杂 ——如果 manager 继承自 employee，当 employee 拥有"我团队所有成员"的访问权限时——经理是否会获得递归展开的权限？RBAC 并非为此而设计。

适用场景：简单的内部工具、角色数量少（<20 个）、无需按资源设置权限。如 Google Workspace 管理控制台、Atlassian 管理层级。

13.2 ABAC（基于属性的访问控制）

主体、资源、操作和环境的属性组合在策略表达式中。策略本身是 代码 ——使用 Rego 的 Open Policy Agent（OPA）、JSON 格式的 AWS IAM 策略、XML 格式的 XACML。

# OPA Rego example
package authz

allow {
  input.user.department == input.resource.department
  input.user.clearance >= input.resource.classification
  input.action == "read"
  time.now_ns() < time.parse_rfc3339_ns(input.resource.expiry)
}
  • 优点:
  • 资源级别、支持条件判断 ——"若 Alice 与文档属于同一部门，且其许可级别 ≥ 文档密级，且当前时间早于文档过期时间，则 Alice 可读取该文档。"
  • 外部化策略 ——安全团队编写 Rego；工程团队编写代码；策略可独立审查、版本控制和部署。
  • 真实适用场景 ——政府安全许可层级、基于医疗关系 + 角色 + 时间的医疗 PHI（受保护健康信息）访问控制。
  • 缺点:
  • 评估成本 ——每次检查都需要获取主体/资源/环境属性，需要进行多次查询。在 1 万 QPS 下，朴素的 ABAC 实现会被属性查询拖垮。
  • 难以审计 ——"给我看所有能读取文档 X 的人"需要对每个用户运行策略评估，无法用一条 SELECT 语句完成。
  • 策略漏洞 ——Rego 是真正的编程语言；漏洞可能悄无声息地导致过度授权或授权不足。测试至关重要。

适用场景：复杂的条件策略（许可级别、时段限制、地理围栏、合规要求）、策略由安全团队负责、愿意投资 OPA 基础设施。

13.3 ReBAC（基于关系的访问控制）

权限通过用户与资源之间的 关系 流转。Google Zanzibar 为 Drive、Calendar、YouTube 将此形式化：若从用户到资源存在一条策略图批准的关系链，则该用户拥有对该资源的访问权。

alice → member of team:platform
team:platform → editor of folder:project-x
folder:project-x → parent of doc:design.pdf

检查："Alice 能否编辑 doc:design.pdf？"遍历过程：文档继承自父文件夹；文件夹将 team:platform 设为编辑者；team:platform 将 Alice 设为成员。结果：是。

  • 优点:
  • "共享文档"的自然模型 ——Google Drive 的"任何拥有链接的人可评论，特定用户可编辑，文件夹所有者继承权限"，这 正是 ReBAC。
  • 递归评估 ——单次检查可自然地遍历任意深度的继承层级。
  • 资源实例粒度 ——每个对象的权限均由同一引擎计算。
  • 缺点:
  • 运维复杂度 ——运行 Zanzibar/SpiceDB 是真实的基础设施投入。
  • 延迟预算 ——递归遍历可能放大延迟。Zanzibar 公布的延迟达到五个九（p99 全球 ~10ms），但要实现这一目标需要具备一致性缓存的全球级基础设施。
  • 审计 ——"给我看所有能编辑文档 X 的人"需要从文档 反向 展开关系图到用户（即"expand" API）。

适用场景：任何用户之间共享资源的产品（协作工具、社交平台、具有团队层级的 B2B SaaS），且共享关系图较为复杂。Drive、Slack、Notion、Figma、Asana——均采用 ReBAC。

13.4 递归评估示例

经典示例： "若 Alice 属于团队 Y，且团队 Y 对包含文档 X 的文件夹拥有编辑权，则 Alice 对文档 X 是编辑者。"

1. Check: can alice EDIT doc:X?
2. doc:X has parent folder:F. Recurse: does anyone with editor on folder:F
   give editor on doc:X? YES (folders propagate to children by default).
3. folder:F has explicit editor: team:Y. Does alice belong to team:Y?
4. team:Y has member: alice. YES.
5. Return: allow.

每次请求都会进行这 4 步递归。朴素实现需从数据库中查询 4 行——4 次往返，耗时 5-20ms。Zanzibar 实现对分片内存图缓存执行同样的路径遍历，总耗时约 1ms。

13.5 选择合适的模型

需求 推荐选择
角色数少于 20 个，无需按资源设置权限 RBAC
条件策略（时间、地点、许可级别） ABAC
用户之间共享资源（任何协作产品） ReBAC
混合：粗粒度访问用角色，细粒度用资源级别控制 RBAC + ReBAC（大多数真实产品）
受监管场景，必须以策略代码形式可审计 结合 OPA 的 ABAC

许多真实系统会组合使用：RBAC 用于宏观控制（组织范围内的管理员/编辑者/查看者），ReBAC 用于微观控制（特定文档共享给特定用户），ABAC 层用于合规条件（非工作时间或非企业网络不得访问 PHI）。

§14. Zanzibar / SpiceDB 架构

Google Zanzibar（2019 年论文"Zanzibar: Google's Consistent, Global Authorization System"）是全球级 ReBAC 的参考实现。凡从事协作/共享类产品的工程师，都应了解它。

14.1 数据模型

Zanzibar 将权限表示为 关系元组:

<object>#<relation>@<user>

Examples:
doc:readme#owner@user:alice
doc:readme#viewer@group:engineering#member
folder:project-x#editor@team:platform
team:platform#member@user:bob

每个元组表示： 拥有 对于 关系。其中 @group:engineering#member 形式意为"在 group:engineering 上拥有 member 关系的任意用户"——用户集改写赋予了 Zanzibar 组合表达能力。

数据库存储的是 (namespace, object_id, relation, user) 行记录。在 Google 的规模下： 约 2 万亿个元组，通过 Spanner 在全球复制。

14.2 命名空间（对象类型）

命名空间定义了一种对象类型及其关系。Drive 的 doc 命名空间可能定义：

namespace doc {
  relation owner: user
  relation editor: user | doc#owner   // owners are also editors
  relation viewer: user | doc#editor | doc#parent->viewer
  // permission viewer is granted by direct grant,
  // OR by being editor, OR by being viewer on the parent (folder)
}

三种基本原语：

  • 直接关系 ——数据库中的显式元组。
  • 用户集改写 — doc#editor 包含所有 doc#owner的用户。组合。
  • 通过"元组到用户集"（TTU）计算的用户集 — doc#viewer 包含所有 viewer 基于文档的 parent 文件夹。跨对象类型的递归。

这套三规则体系足以建模 Drive、Calendar、Cloud IAM、YouTube 频道权限以及 Google Photos 共享相册—— 全部在同一引擎中.

14.3 Check API

热路径。 Check(object, relation, user) 返回 true|false:

Check("doc:design.pdf", "edit", "user:alice")

服务端遍历关系图：直接授权？用户集改写？TTU 到父对象？若任意路径命中则返回 true 。

延迟预算： p99 全局 < 10ms，即便遍历跨越多个数据中心。Zanzibar 在此达到五个九——部分依赖积极缓存（热缓存命中率约 95%），部分依赖"zookie"（快照时间戳，确保检查看到因果一致的图视图，无需全局同步读取）。

14.4 Expand API

用于审计和管理 UI。 Expand(object, relation) 返回 拥有该关系的全部用户集合 ：

Expand("doc:design.pdf", "edit") →
  { user:alice, user:bob, members-of(team:platform) }

输出是用户集的树形结构，而非扁平列表——具体的子集物化留给调用方处理（"列出所有人"可能数量巨大）。管理控制台随后遍历该树。

Expand 是"审计"模式。Check 是"决策"模式。

14.5 五个九的延迟预算

Zanzibar 公布的 p99 全局 < 10ms。在万亿级元组规模下实现这一目标需要：

  • Spanner 后端 ——全局一致性，并通过各区域本地副本优化读取延迟。
  • 积极缓存 ——Zanzibar 的"Leopard 索引"预先计算常见用户集的扁平化群组成员关系。对一个拥有 5 万名工程师的"engineering"群组进行检查时，无需遍历图，而是命中预计算的类布隆过滤器索引。
  • Zookie 快照读取 ——在为已快照读取资源的页面加载检查权限时，检查使用资源的快照时间戳，避免往返确认"此结果是否仍然有效"。
  • 副本与读取本地化 ——每次检查由最近区域的副本提供服务。

14.6 开源克隆

  • SpiceDB（AuthZed） ——最接近 Zanzibar 的开源克隆。相同的 (namespace, relation, user) 元组模型；相同的 Check/Expand API；相同的用户集改写 + TTU 原语。PostgreSQL 或 CockroachDB 作为后端。被 Reddit、Turo、Adobe、Netflix 使用。
  • OpenFGA（Auth0/Okta） ——设计相似，CNCF（云原生计算基金会）沙箱项目。使用基于 JSON 的 schema，而非 Zanzibar 的 Protobuf。
  • Permify ——较新的参与者，支持 OpenFGA 的模型以及用于临时策略的查询语言。
  • Warrant ——提供类 Zanzibar 服务的商业 SaaS。

建议：若在 2026 年从零构建协作产品，从 SpiceDB 或 OpenFGA 开始。不要自行实现——递归语义微妙，极易出错（循环、最终一致性、快照读取）。

14.7 运维注意事项

  • Schema 演进 ——新增关系需要回填派生元组，像数据库 schema 变更一样规划迁移。
  • 缓存过期 ——元组写入时，依赖该元组的缓存必须失效。SpiceDB 使用 watch API；Zanzibar 使用 zookie。
  • 列表查询性能 ——"给我看 Alice 能看到的所有文档"是 Check 的逆操作。部分实现支持 ListObjects，但在大规模场景下代价 高昂 ——这是一个全量展开操作，需谨慎使用，或将其推入独立的读模型。

§15. 大规模会话吊销

吊销是贯穿全文的核心主题。§4e 给出了概念层面的基本原语；本节是针对实际运行在 1 亿以上用户规模组织的实施手册。

15.1 三种工具及其组合方式

  1. 短期 JWT + Kafka 黑名单 ——用于按令牌吊销（特定 JWT 被曝光、泄露等）。
  2. 每用户 min_iat 缓存 ——用于"全部登出"（密码修改、账号被攻陷后重置）。
  3. 带缓存的会话存储兜底 ——用于吊销频率高的场景（管理工具、银行业务）。

生产环境中，三者同时部署，按使用场景分层叠加。

15.2 实现：基于 Kafka 的 JTI 黑名单

架构（每个数据中心）：

[Revoke API] → write to MySQL (durable) → publish to Kafka (revoked-jti)
                                              │
                                              ▼
[Verifier pods] ─ subscribe ─ maintain in-memory hashset
        │
        └─ per request: hashset.contains(jti)?  ← ~50ns

容量估算：在每天 5 万次吊销、访问令牌 TTL 为 15 分钟的情况下，黑名单在任意时刻最多包含约 520 条记录（50k/86400 × 900 ≈ 520）。520 × ~50 字节 ≈ 每个 Pod 26 KB。一旦 exp 过期，条目即可驱逐。清理任务每 60 秒运行一次。

收敛性：Kafka 端到端延迟 p99 在数据中心内约 500ms，跨区域约 1s（使用 MirrorMaker 2）。因此"撤销"操作在全球范围内约 1s 内可见。

如果某个验证节点尚未收到 Kafka 消息怎么办？在约 1s 的窗口期内，该验证节点仍会接受该 JTI。被接受的爆炸半径：1s 内某个访问令牌的使用量。对于消费者级别的撤销（用户点击"退出登录"），这是可接受的。对于高风险场景（被解雇的员工试图删除生产数据），需对关键端点包装一个 fresh Redis 检查，绕过本地缓存。

15.3 基于用户的 min_iat 用于"全部退出登录"

Redis:  user:42:min_iat → 1715000500

每个 JWT 携带 iat。验证节点读取 min_iat 令牌对应的 sub；如果 iat < min_iat，则拒绝。效果：对 user:42:min_iat = now() 使 用户 42 的所有有效令牌失效 一次性完成。

缓存策略：验证节点在 LRU 中缓存 min_iat 每用户。1M 缓存用户 × 约 50 字节 = 每个 Pod 约 50MB。缓存条目 TTL：60s，在用户定向事件时失效（订阅 user-min-iat-updates Kafka topic）。对于不在缓存中的用户，回退到 Redis（约 1ms）。

何时更新 min_iat：\n- 密码变更。\n- MFA 重置 / 设备移除。\n- 管理员强制退出。\n- 防滥用系统检测到疑似入侵。\n- 用户主动发起"从所有设备退出登录"。

15.4 带缓存的会话存储回退

对于 1s 的拒绝名单传播过慢的场景，回退到经典会话模型：每个请求检查服务端的会话行。Redis 缓存位于 MySQL 前端。缓存命中约 1ms；缓存未命中约 5ms。

适用场景：\n- 管理控制台（强制新鲜检查；对组织级设置变更绝不单独信任 JWT）。\n- 电汇确认页面。\n- 任何需要撤销后的 next 请求立即生效的场景。

成本：新增 1ms 延迟，需额外运行一个 Redis 集群。仅在关键路径上才值得。

15.5 权衡一览

工具 生效延迟 每请求开销 能力
仅短 TTL 最长 TTL（15 分钟） 无（已在验证 JWT） 最便宜，最弱
JTI 拒绝名单 约 1s（Kafka 扇出） 约 50ns 哈希集 单令牌撤销
min_iat 约 100ms（Redis 写入 + 缓存 TTL） 约 50ns 缓存命中 / 约 1ms 未命中 用户级"全部退出登录"
会话存储检查 立即生效 约 1ms Redis 特定场景保证
密钥轮换 立即生效，但全局爆炸半径 每请求无开销 核武级

LinkedIn/Stripe/PayPal 的真实生产栈：以上所有方式分层使用。

§16. 步骤升级认证与 ACR

有时一个会话对于常规浏览"认证程度足够"，但对于敏感操作则不够。步骤升级认证允许服务器在会话中途要求更强的认证，作用域限定于特定操作。

16.1 模式

一个用户在 4 小时前通过密码 + TOTP 登录。现在他点击"发起 50,000 美元电汇"。服务器的检查：

1. Is the session authenticated? Yes.
2. Does this endpoint require step-up? Yes — banking compliance: MFA within 15 min.
3. Was MFA performed within 15 min? No (it was 4 hours ago).
4. Return: 401 with WWW-Authenticate: step-up.
5. Client redirects user to MFA challenge.
6. After MFA, server issues a *new* token with elevated ACR claim.
7. Client retries the wire transfer with the new token.
8. Server accepts.

16.2 ACR — 认证上下文类引用

OIDC 将 acr 声明（认证上下文类引用）标准化，专门用于此目的。值为 URI 字符串，描述 如何 用户完成了认证：

{
  "sub": "user-42",
  "acr": "urn:mace:incommon:iap:silver",
  "amr": ["pwd", "totp"],
  "auth_time": 1715000000,
  "iat": 1715014400,
  "exp": 1715015300
}
  • acr — 宽泛的认证上下文类。NIST AAL（认证器保障级别）1、2、3 — 或厂商自定义值。
  • amr （认证方法引用）— 所使用的具体方法： pwd, totp, hwk （硬件密钥）， face, fpt （指纹）， mfa.
  • auth_time — 用户上次认证的时间。用于检查步骤升级的新鲜度。

16.3 电汇端点的代码实现

@require_acr(min_acr="aal3", max_age_seconds=900)
@require_amr_any_of(["hwk", "totp"])
def initiate_wire_transfer(request):
    ...

装饰器检查 JWT 的 acr, amr, auth_time。如果 acr < aal3 或 now - auth_time > 900s 或 amr 缺少强认证因素，则返回 401 并附带 header 说明需要何种步骤升级。

16.4 关键使用场景

  • 金融操作 — 资金转账、ACH 发起、账户注销。
  • 管理员操作 — 终止员工、变更组织级设置、轮换签名密钥。
  • PII 导出 — GDPR 数据导出、HIPAA 医疗记录下载。
  • 账户恢复 — 添加新设备、变更恢复邮箱。
  • 高信任 API 调用 — 第三方应用请求扩展作用域。

16.5 用户体验模式

步骤升级应感觉 有针对性，而非令人痛苦。银行应用：点击"发送 $5000" → 指纹提示 → 完成。指纹前的令牌具有 acr=aal2；新令牌具有 acr=aal3。用户感知到一次生物识别提示；系统已提升了保障级别。

你不想要的：重新运行整个登录流程。那会训练用户在每次操作时输入密码，导致随处输密码的网络钓鱼风险。

§17. 委托与身份模拟

真实系统中存在一个身份代表另一个身份行事的流程。两大主要类别：

  1. 管理员模拟 — 客服人员模拟用户进行调试。
  2. 服务间代表操作 — 服务 A 以保留用户身份的方式调用服务 B。

17.1 带审计的管理员模拟

一名支持工程师需要"看到客户所见的内容"。设计方案：

  • 糟糕的方案：支持工程师拥有 root 凭证，以客户身份登录。无审计；无法区分哪些操作是客户的，哪些是支持人员的。
  • 更好的方案：支持工程师通过内部管理工具请求模拟令牌。工具检查支持人员的角色，生成包含 subject 和 actor 声明的 JWT：
{
  "sub": "user-42",          // the impersonated user
  "act": {
    "sub": "support:carol",  // the actual actor
    "role": "support-tier-2"
  },
  "scope": "impersonate:read", // restricted scope
  "exp": 1715015300,           // short expiry (15 min)
  "amr": ["pwd", "totp"]       // support's own MFA, not user's
}

命令使 LLDB 的行为类似于交互式命令工具。我们可以使用 act 声明（RFC 8693 §4.1）携带操作者身份。下游服务看到 sub=user-42 （UI 显示正常），但审计日志记录 act.sub=support:carol （操作可追溯）。

防御性检查：\n- 作用域受限 — impersonate:read 只允许读操作。电汇、密码变更等需要原始用户的凭证。关键端点中硬编码。\n- 短 TTL — 最长 15 分钟。\n- 审计记录每个请求 — 为模拟操作单独记录日志流，保留 7 年。\n- 用户通知 — 用户收到邮件"支持工程师 Carol 于 14:35 查看了您的账户。"\n- 需要填写原因 — 支持人员在令牌签发前必须输入 JIRA 工单或原因。

17.2 令牌交换（RFC 8693）

OAuth 2.0 令牌交换将代表操作流程标准化。服务 A 持有用户令牌；服务 A 希望以 B 能看到用户身份的方式调用服务 B，同时调用可归因于 A。

POST /token HTTP/1.1
grant_type=urn:ietf:params:oauth:grant-type:token-exchange
subject_token=<user's access token>
subject_token_type=urn:ietf:params:oauth:token-type:access_token
audience=https://api.b.example.com
actor_token=<A's service token>            // optional, for on-behalf-of
actor_token_type=urn:ietf:params:oauth:token-type:access_token

IdP 验证两个令牌，铸造包含以下内容的新令牌：

{
  "sub": "user-42",
  "aud": "https://api.b.example.com",
  "act": {
    "sub": "service:a"
  },
  "scope": "...",
  "iat": ...,
  "exp": ...
}

服务 B 看到 sub=user-42 （正确的授权）和 act.sub=service:a （因此知道 A 是中间人）。如果 A 后来被攻陷，B 的审计日志能精确显示哪些调用是 A 代表哪些用户发起的。

17.3 "客服代表用户 X 操作"的区别

陷阱：下游服务必须区分 用户 X 本人 与 用户 X 经由客服模拟。关键在于：

  • 风险评分 — 欺诈模型不应将"用户 X 凌晨 3 点从企业 IP 登录"学习为正常用户行为。
  • 通知 — 当操作是客服发起时，不要给用户发送"您已登录！"的邮件。
  • 隐私受限操作 — 某些操作（导出个人数据）应仅限用户本人，不允许客服模拟。

实现：每个下游服务必须读取 act 声明，并基于 act 不存在（或匹配特定允许的操作者类型）来控制功能访问。

17.4 SAML / OIDC 委托声明

  • OIDC — RFC 8693 act 声明（如上）。
  • SAML — <saml:SubjectConfirmation> 与 <saml:Subject> 携带委托语义，有时为厂商特定实现。
  • JWT — act （操作者）和 may_act （允许的委托方，用于限制谁可以委托）。

始终携带操作者信息；绝不折叠为"仅用户"。这是真实系统中违反最多的授权审计原则。

§18. 多租户身份

B2B SaaS 的成败取决于多租户能力。每个客户（租户）是其自身独立的用户围墙花园，拥有自己的 IdP、自己的管理策略和自己的数据。此场景下的认证必须严格划定作用域，否则会导致跨租户数据泄漏。

18.1 按租户配置 IdP

每个租户配置其自己的记录 IdP。SaaS 按租户支持联合：

  • 租户 Acme 使用 Okta SAML。
  • 租户 Beta 使用 Azure AD OIDC。
  • 租户 Gamma 不使用 SSO；用户通过邮箱/密码注册（由 SaaS 自身管理）。
  • 租户 Delta 使用客户私有的 OpenID Connect 端点。

当用户访问 https://acme.example.com/login （租户子域）时，SaaS 读取租户配置，重定向到正确的 IdP，接受 SAML 断言或 OIDC 令牌，并创建本地会话。

实现：一张 tenant_idp_config 表：

CREATE TABLE tenant_idp_config (
  tenant_id     BIGINT PRIMARY KEY,
  idp_type      ENUM('saml','oidc','none'),
  metadata_url  TEXT,
  cert_pem      TEXT,
  client_id     VARCHAR(255),
  client_secret_kms_arn  VARCHAR(255),
  attribute_map JSON
);

属性映射至关重要：IdP 的"email"声明如何映射到 SaaS 的 email？大多数 IdP 发送 email，但有些发送 mail 或 urn:oid:0.9.2342.19200300.100.1.3。需要规范化。

18.2 租户作用域令牌

每个 token 携带 tenant_id。每个端点强制执行。

{
  "sub": "user-42",
  "tenant_id": "tenant-acme",
  "aud": "https://api.example.com",
  "scope": "read:projects"
}

适用场景 tenant-acme的用户向 https://api.example.com/projects/proj-123提交令牌时，服务器必须：

  1. 验证令牌。
  2. 加载 proj-123的 tenant_id。
  3. 如果不匹配则拒绝 proj-123.tenant_id != token.tenant_id.

第 3 步是 不可省略的。每个出现过跨租户数据泄漏的多租户 SaaS，都是因为某个端点缺少第 3 步。

18.3 跨租户模拟攻击

经典漏洞：攻击者是 tenant-acme的合法用户。他们发现 api.example.com/projects/proj-999 返回 proj-999 的数据 — 而 proj-999 属于 tenant-beta。服务器验证了令牌（签名有效，未过期），但没有检查所请求资源的租户归属。

变体： aud 混淆。如果 tenant-acme的 IdP 签发受众为 aud=https://api.example.com （共享 API）的令牌，攻击者可以拿自己的合法令牌发送给 tenant-beta的管理 API — 相同的 aud，没有租户约束。缓解措施：按租户限定 aud 作用域： aud=https://api.example.com/t/tenant-acme，并拒绝租户不匹配的情况。

18.4 租户隔离模式

  • 按租户使用子域 — acme.app.example.com, beta.app.example.com。从主机名推断租户上下文。
  • 按租户使用路径前缀 — /t/acme/..., /t/beta/...。同一域名，对 cookie 更友好；更难实现按租户的 TLS 固定。
  • 按租户使用请求头 — X-Tenant-Id: acme。在 B2B API 中常见。IdP 签发令牌；API 检查 X-Tenant-Id 是否匹配 token.tenant_id.

最干净的方案：子域 + tenant_id 声明 + 端点级强制执行。双保险。

18.5 按租户自定义 SSO — 运营成本

为每个 B2B 客户支持 SAML 意味着：

  • 按租户维护可能损坏/轮换的 IdP 元数据 URL。
  • 按租户维护按客户计划（而非你的计划）过期的证书。SAML 证书过期 → 租户被锁定 → 凌晨 2 点支持电话。
  • 按租户的属性映射错误。客户的 IdP 发送"department"，但 SaaS 期望"dept" — 静默的权限误配。

值得投入的工具：证书到期监控（在到期前30天发送告警、向租户管理员自动发送邮件）、配置中的属性映射验证器、供租户管理员使用的"模拟用户"调试工具，用于在不影响生产环境的情况下测试SSO。

§19. 认证层的反滥用与限流

认证入口天生是滥用行为的温床：密码填充攻击、暴力破解、账号劫持、通过账号锁定实施拒绝服务、机器人批量注册、OAuth授权垃圾请求。每一个上线的认证产品都需要防御措施；许多候选方案低估了这部分代码的工作量。

19.1 登录限流 — 三个维度

  • 每用户 — 每用户每分钟失败登录次数不超过N次，阈值约为5次/分钟。
  • Per-IP — 每IP每分钟登录尝试次数不超过N次，阈值约为20次/分钟。（低于按用户限制，因为攻击者会轮换用户名。）
  • Per-ASN (Autonomous System Number) — 当某个网络（云服务商、家庭ISP）出现突发流量时，对整个ASN进行限流。用于应对来自被攻陷僵尸网络的密码填充攻击活动。

实现方式：在Redis中使用滑动窗口计数器。 INCR login_fail:user:42, INCR login_fail:ip:1.2.3.4, INCR login_fail:asn:12345，所有计数器均设置TTL等于窗口大小。收到请求时，读取全部三个计数器；若任意一个超过阈值，则拒绝（附带 429 Too Many Requests).

19.2 账号锁定 — DoS反模式

朴素的防御方案：5次登录失败后，将账号锁定15分钟。这 看似 合理。

DoS滥用：攻击者对每个合法用户的账号提交5个错误密码，所有用户随即被锁定。攻击者无需破解任何密码，就已对整个用户群实施了DoS攻击。

更好的防御方案：

  • 按IP指数退避，而非按用户锁定 — 每次失败尝试后，该IP必须等待2倍时间才能发起下次尝试。来自同一IP的合法用户略感不便；而单一IP上的攻击者则被彻底阻断。
  • N次失败后触发CAPTCHA — 3至5次失败后显示CAPTCHA。机器人无法通过，人类可以，且不会锁定用户账号。
  • N次失败后触发升级MFA — 若用户已开启MFA，即使其输入了正确密码，也要求完成MFA验证（暗示可能存在凭证泄露）。
  • 仅在多次失败后成功时锁定 — 若攻击者最终猜对了密码， 再 锁定账号并通知用户，不要因为用户是攻击目标而惩罚用户。

正确答案： 绝不仅凭输入行为加锁，始终结合IP/ASN信号。 账号锁定是2005年的设计思路，在2026年已成为拒绝服务攻击向量。

19.3 机器人检测层

  • CAPTCHA — reCAPTCHA v2（复选框）、v3（隐形评分）、hCaptcha、Cloudflare Turnstile。对用户有一定干扰，但在流量漏斗顶端是最有效的机器人拦截手段。
  • 行为生物特征 — 鼠标移动轨迹、击键节奏、滚动行为。厂商包括Akamai BMP、F5 Shape、BioCatch。成本较高，对合法用户透明，但能拦截通过CAPTCHA的高级机器人。
  • 设备指纹 — User-Agent + 屏幕尺寸 + 时区 + 已安装字体 + WebGL哈希 + Canvas哈希。机器人农场的指纹往往完全相同，人类用户则不然。FingerprintJS是主流库。
  • TLS指纹 (JA3/JA4) — TLS ClientHello字节模式可揭示底层库（curl、Node.js、无头浏览器）。这是浏览器指纹而非user-agent字符串，伪造难度更高。Cloudflare和Akamai广泛使用此技术。
  • 蜜罐字段 — 在表单中隐藏名为 email 真实浏览器不会填写但朴素机器人会填写的字段。成本低廉，效果出乎意料地好。

19.4 密码填充攻击检测

与暴力破解不同：密码填充攻击中，攻击者拥有从其他网站泄露的 (email, password) 来自其他站点数据泄露的（用户名, 密码）列表，并尝试在你的网站上使用，寄望于用户重复使用了相同密码。

检测信号：

  • 大量（邮箱, 密码）组合尝试 成功 — 但每次来自 不同的 新IP，通常是住宅代理。"新IP高成功率"信号是关键特征。
  • Failed-login-from-new-IP-with-known-password — 该密码存在于泄露语料库中（Have I Been Pwned、Specops、厂商数据源）。见到此情况几乎可以确定是凭证测试，应立即告警并触发升级验证。
  • 从新设备 + 新IP + 新地理位置登录 — 对于老用户，此组合会触发邮件提醒"您刚刚从俄罗斯登录了吗？"以及升级MFA。

分层响应措施：

  1. 已知泄露密码 在登录时：强制要求修改密码。
  2. 新设备+新地理位置成功登录：发送邮件通知；要求MFA；可选地对新设备的会话设置过期，直至用户确认。
  3. 同一ASN在多个账号上出现集中成功登录：对该ASN进行限流；通知安全值班人员。

19.5 注册滥用

不那么显眼但同样重要：机器人批量创建虚假账号用于垃圾邮件、数据抓取和欺诈。缓解措施：

  • 邮箱验证 — 要求邮箱点击确认后账号才可使用，可减少约80%的一次性滥用注册。
  • 手机号验证 — 由滥用方按号码付费，提高攻击成本。Twitter、Discord、ChatGPT均采用此方式。
  • 支付验证 — 对高价值租户收取1美元（可退款），机器人没有信用卡。
  • 持有期 — 新账号在24至72小时内功能受限，可减缓垃圾信息攻击活动。

§20. 审计日志要求

在任何受监管行业——金融、医疗、政府、支付——认证事件必须被详细记录，保留数年，并以不可篡改的方式存储。审计人员一定会问，"我们没有记录"是一个合规发现项。

20.1 必须记录的内容

最低要求：

  • 登录成功/失败 — user_id, timestamp, ip, user_agent, mfa_used, result.
  • MFA挑战 — user_id, factor (totp/sms/push/webauthn), result, timestamp.
  • 密码修改 — user_id, timestamp, channel (self-service / admin-reset / forced).
  • 范围/权限授予 — 用户向第三方应用授予OAuth范围时： user_id, client_id, scopes, timestamp.
  • Token颁发 — user_id, jti, aud, scopes, expiry, issuer_kid.
  • Token撤销 — user_id, jti or family, reason, actor (user / admin / system).
  • 角色变更 — user_id, old_role, new_role, actor, reason, timestamp.
  • 模拟操作开始/结束 — actor_user_id, impersonated_user_id, ticket, start, end.
  • 管理员操作 — 管理员角色下的每一个操作。
  • 账号创建/删除 — user_id, timestamp, source (signup / import / scim).
  • MFA注册/移除 — 每次认证因子变更。
  • 会话创建/销毁 — 至少记录元数据。

20.2 保留期限

  • 金融（SOX — Sarbanes-Oxley） — 7年。
  • PCI-DSS (Payment Card Industry Data Security Standard) — 最少1年，其中3个月需立即可查。
  • HIPAA (Health Insurance Portability and Accountability Act) — 6年。
  • GDPR (General Data Protection Regulation) — 目的限制；认证日志通常1至2年。
  • SOC 2 (Service Organization Control 2) Type II — 最少1年，客户通常要求3年。
  • FedRAMP (Federal Risk and Authorization Management Program) — 系统保留3年；中/高级别可能要求更长。

实用建议：按7年冷存储规划预算，这是最长的常见合规要求，而冷存储的边际成本很低。

20.3 不可变性 — 一次写入存储

审计日志必须具备防篡改性。拥有管理员权限的攻击者不应能够事后修改日志以掩盖其行为。

实现模式：

  • WORM (Write Once Read Many) 存储 — AWS S3 Object Lock合规模式、Azure Blob不可变存储、本地WORM磁带。在保留期届满之前，无法删除或修改对象。
  • 哈希链式日志 — 每条日志条目包含前一条条目的哈希值，任何篡改都会破坏哈希链。Hyperledger/区块链风格的审计日志。
  • 仅追加Kafka — 日志发布到设置无限保留期的Kafka主题，并跨区域复制。运维人员若不丢失全部副本，则无法删除单条消息。
  • 外部审计日志服务 — Splunk Cloud、Datadog、AWS CloudTrail（用于AWS API调用）、Vanta——位于应用程序爆炸半径之外。

设计模式：将审计事件双写至（a）Kafka，供安全工具实时消费；（b）启用Object Lock的S3，用于合规保留。

20.4 不应记录的内容

审计日志中的PII和凭证是一种法律风险：

  • 永远不要记录密码，即使是在登录失败时。"密码错误"的提示信息已足够。
  • 永远不要记录完整的会话ID或完整的Token — 记录哈希值或后4位字符，否则审计日志本身将变成凭证数据库。
  • 超出user_id范围的个人可识别信息 — 尽量减少。遵循GDPR的数据最小化原则。
  • MFA验证码 — 永远不要记录实际的TOTP/SMS验证码（存在重放风险）。
  • API密钥/Secret — 绝对不要记录。日志厂商SDK有时会泄露此类信息，务必审查。

20.5 驱动审计的合规框架

  • SOX — 金融，美国上市公司，Section 302 + 404审计要求。
  • PCI-DSS — 持卡人数据；对日志内容有非常具体的要求（PCI 10.x）。
  • HIPAA — 医疗；PHI的访问日志必须可按需审阅。
  • SOC 2 — 由审计人员驱动；灵活但覆盖广泛。
  • ISO 27001 — 全球通用；控制项A.12.4要求事件日志记录。

审计人员每次必问："请展示在A至B日期之间访问过用户X记录的人员。"若无法在5分钟内作答，即为不合格。

§21. 零信任架构

2010年代的假设是企业网络是可信的（VPN内 = 安全；外部 = 危险）。零信任抛弃了这一假设：每个请求，无论其网络来源，都需经过认证和授权。基于网络的信任已消亡。

21.1 BeyondCorp — Google的参考架构

Google于2014年发布BeyondCorp，起因是Aurora攻击（2009年）表明，攻陷受信任内网上的一台企业笔记本，即可作为渗透整个网络的立足点。BeyondCorp的核心：

  • 无VPN，无网络边界。 工程师可直接从咖啡馆、家中或任何地方工作。
  • 每个请求均需认证。 Google员工从浏览器访问内部工具时，会经过一个Access Proxy，该代理验证：（a）设备是企业发放且状态健康，（b）用户已通过认证，（c）用户有权访问该资源。
  • 设备信任作为一等信号。 设备在注册时颁发证书，证书存储于设备安全元件中。设备健康状况（补丁、防恶意软件）持续被检查。
  • 持续重新评估。 若出现异常，信任评分下降；访问权限可在会话进行中被撤销。

架构图：

[Engineer's laptop] ─ HTTPS ─▶ [Access Proxy] ─ inspect ─▶ [Internal app]
        │                            │
   device cert                       ├─▶ check IdP (user identity)
   user creds                        ├─▶ check device inventory (corp-issued)
                                     ├─▶ check policy engine (allowed?)
                                     └─▶ continuous risk score

21.2 商业零信任网关

Cloudflare Access、Cisco Duo、Tailscale、Zscaler——均以服务形式实现BeyondCorp风格架构：

  • Cloudflare Access — TCP/HTTP反向代理；与客户的IdP（Okta、GSuite、AzureAD）集成；支持按应用策略（"只有使用企业发放笔记本的工程师才能访问GitHub Enterprise"）。订阅服务可替代传统VPN。
  • Tailscale — 基于WireGuard构建的VPN替代方案，采用网状网络。基于身份的访问控制（Tailscale ACL使用HuJSON格式），无共享密钥。身份绑定至Google/Microsoft/Okta IdP。
  • Cisco Duo — 主要提供MFA，但已扩展为带设备态势检查的零信任网关。
  • Zscaler Private Access (ZPA) — 面向企业，与传统VPN替代方案竞争。

21.3 VPN替代的演变

传统VPN：在企业网络之间建立隧道；隧道授予隐式网络访问权限。攻陷后的横向移动极为容易。对VPN设备（Pulse Secure、Fortinet）打补丁是一场安全噩梦。

零信任用身份取代网络。连接是到应用程序的端到端TLS，而非站点到站点VPN。每个连接均需认证；横向移动需要对每个应用重新进行认证（authN）。

建议： 对于2026年的任何新基础设施项目，零信任网关是默认选择；VPN是遗留方案。 现有VPN的风险日益增加（补丁周期慢、攻击面大）——大多数大型组织计划在2027至2028年前将其退役。

21.4 对应用层授权（authZ）的影响

应用程序对网络 一无所知 地做出假设。每个请求必须：

  1. 携带用户身份（JWT在请求头中，或会话Cookie）。
  2. 携带设备身份（mTLS证书指纹、BeyondCorp请求头）。
  3. 依据每次请求的策略进行评估（"该用户，在该设备上，此时对该资源，能否执行此操作？"）。

应用程序不存在"可信内网"模式，内网根本不存在。

§22. 影响认证设计的合规框架

在受监管行业中，合规制度不是可选项，它直接约束认证设计决策。

22.1 PCI-DSS (Payment Card Industry Data Security Standard)

适用范围：任何存储、处理或传输持卡人数据的系统。

与认证相关的要求：

  • Req 8.3 — 所有非控制台管理访问以及对持卡人数据环境（CDE）的所有远程访问均需MFA。PCI-DSS 4.0（2024年）起，新实施方案不再接受SMS OTP。
  • Req 8.2 — 密码复杂度、轮换要求；或使用硬件Token。
  • Req 10 — 全面的审计日志，包括所有认证（authN）事件。
  • Req 8.7 — 唯一用户ID；CDE内禁止共享账号。

设计影响：为支付处理的管理员访问部署无密码/Passkey方案；按账号强制MFA注册；使用不可变审计日志；工程师间绝不共享服务账号。

22.2 HIPAA (美国医疗保健)

适用范围：被覆盖实体（医院、诊所、支付方）以及业务伙伴（处理PHI的SaaS）。

与认证相关的安全保障措施：

  • §164.312(a)(2)(i) — 唯一用户标识。
  • §164.312(a)(2)(iii) — 自动注销。 无人操作会话的超时机制。
  • §164.312(d) — 人员或实体认证。 访问前需进行合理验证。
  • §164.312(b) — 审计控制。 记录 PHI 处理系统中的活动。

设计影响：会话超时时间短（15-30 分钟）；对每次 PHI 访问记录详细审计日志；每人独立账号（临床环境中不得共享登录）；紧急访问的破玻璃程序。

22.3 GDPR（欧盟隐私法规）

适用范围：任何处理欧盟居民个人数据的系统。

与认证相关的条款：

  • 第 5 条 — 数据最小化原则。 不得收集超出必要范围的身份数据。
  • 第 17 条 — 删除权（"被遗忘权"）。 用户可要求删除账号；系统必须真正删除（保留合法留存的例外情形）。
  • 第 32 条 — 处理安全性。 加密、MFA、访问控制。
  • 第 33-34 条 — 数据泄露通知。 发生泄露后须在 72 小时内上报。

设计影响：账号删除必须真正删除（而非仅禁用）；审计日志在"合法利益"依据下保留并记录保留期限；建议启用 MFA；静态凭据加密为强制要求。

GDPR 与认证最难处理的交叉点：审计日志 包含 PII（用户 ID、IP 地址、操作记录）。当用户行使被遗忘权时，是否应删除其审计日志？通常不删除——审计日志依据"法律义务"或"合法利益"保留，但用户的 user_id 必须假名化或替换为哈希值，以防止在未获额外同意的情况下将日志与用户关联。

22.4 CCPA（加利福尼亚州消费者隐私法）

GDPR 的加州简化版。包含知情权、删除权以及拒绝出售个人信息权。认证设计影响与 GDPR 类似，但要求不那么详细。

22.5 SOC 2（服务组织控制 2）

面向客户的审计认证；主要适用于 B2B SaaS 场景。信任服务标准包括：安全性、可用性、处理完整性、保密性、隐私性。

与认证相关的控制项：

  • CC6.1 — 逻辑访问控制。MFA、基于角色的访问控制、最小权限原则。
  • CC6.2-6.3 — 用户的开通与注销管理。
  • CC7.2 — 安全事件监控。

设计影响：向企业租户提供 SCIM 开通；季度访问审查；所有管理员和工程师访问均需启用 MFA；详细审计日志。

22.6 FedRAMP（美国联邦政府）

适用范围：向美国联邦机构出售的 SaaS 产品。

级别：低、中、高。认证要求逐级递增：

  • Moderate（中级） — 需要 MFA；约 325 项控制。
  • High（高级） — 基于硬件的 MFA（PIV/CAC 卡、FIDO2 硬件密钥）；约 421 项控制。

设计影响：支持硬件绑定凭据（PIV 智能卡），FedRAMP High 不得使用同步通行密钥，所有认证事件须写入政府批准的不可篡改存储。

22.7 审计操作手册

每个受监管的 SaaS 都应保留：

  • 认证事件日志，保留期 7 年。
  • 季度访问审查并附有审批签字。
  • MFA 注册报告（已注册 MFA 的用户比例）。
  • 每年对认证面进行渗透测试。
  • 包含认证泄露场景的书面应急响应计划。
  • 面向客户的合规文件（SOC 2 报告、HIPAA BAA、GDPR DPA）。

建议：将认证设计为 可举证的。审计不是"我们有 MFA"，而是"给我看 100 个随机用户账号并证明他们的 MFA 注册日期"。

§23. 认证成本

认证在高 QPS 下运行，成本不可忽视，尤其是 KMS 支持的签名和 IdP 联合认证。

23.1 KMS / HSM 签名成本

若 JWT 使用托管 KMS（AWS KMS、GCP Cloud KMS、Azure Key Vault HSM）签名，每次签名调用都会产生费用：

  • AWS KMS — 非对称（RSA/ECC）密钥每 10,000 次签名操作收费 $0.03。按 1M 次登录/秒 × 86400s × 365 = 31.5 万亿次签名/年 ≈ 若每个 JWT 都经 KMS 签名则每年约 $94M。

这显然荒谬。解决方案： 不要在 KMS 中签名每个 JWT。仅使用 KMS 签名 密钥封装 ——IdP 在进程内存中持有签名密钥；KMS 仅在密钥轮换时签名（每季度一次，全年合计数百美元）。运行时签名使用内存中的密钥，免费。

对于超高安全性场景，需权衡取舍：在 HSM 中签名（每次签名均在防篡改硅片中完成）成本约 $0.03/10k。以 100k/秒持续签名计算，约为 $258k/年。对于高风险场景（如 CA 的根签名密钥）值得投入；对于日常访问令牌则不值得。

23.2 IdP 联合认证调用成本

每次联合至第三方 IdP 的登录都会产生成本：

  • Okta — 按月活跃用户计费；B2C 场景粗估 $2-4/用户/年。
  • Auth0 — 类似的分层模型；免费额度之上约 $0.023/次登录。
  • AWS Cognito — 免费额度（50k MAU）以上 $0.0055/MAU；价格更低但企业功能较少。

100M MAU 时：Okta 级别的 IdP 每年超过 $200M。相同规模下纯自托管（小型集群上的 Keycloak）每年计算成本约 $50k + 约 5 名 SRE（站点可靠性工程师）全职人力。对于 B2C 场景，自建与采购的平衡点约在 10-30M MAU；对于有企业 IdP 需求的 B2B 场景，这一临界点更早出现。

23.3 缓存命中率目标

对于基于会话存储的验证，命中率必须 >99%。1M QPS 下的数据：

  • 100% 缓存命中：1M Redis 读取/秒，Redis 集群约 $1k/月（单个 ElastiCache 节点）。
  • 99% 命中，1% 未命中穿透到 MySQL：10k MySQL 读取/秒——接近单个 MySQL 主节点的承载上限。需要分片或读副本。
  • 90% 命中，10% 未命中：100k MySQL 读取/秒——需要 4-8 个分片。

将缓存命中率从 99.0% 提升到 99.9% 的工程投入（淘汰策略调优、预热查询、延长 TTL）：可行。从 90% 提升到 99% 的投入：巨大。性能悬崖非常陡峭。

设计原则：假设业务收入随使用量增长；为 10 倍增长设计缓存命中率，而非当前负载。

23.4 一览数学账

1M QPS 持续认证验证下：

  • JWT 本地验证（RS256）：约 80 核，约 $300/月计算成本。非常便宜。
  • Redis 会话缓存：集群约 $2k/月。
  • MySQL 会话存储（99% 缓存命中）：10k/秒，Aurora 集群约 $5k/月。
  • KMS 签名令牌（每个令牌都签名）：$94M/年。绝对不可行。
  • IdP 联合认证按登录计费：1M 次登录/天，Auth0 每年约 $230k。相比之下 Keycloak 自托管约 $50k/年 + 1 名 SRE。

核心结论：了解单位经济学。许多系统在认证上亏损，原因在于没意识到 KMS 调用或按 MAU 计费的 IdP 成本会随增长线性扩展。

§24. 故障模式详解

操作中途崩溃（登录期间 IdP 崩溃）：Argon2id 验证返回 true，但 INSERT INTO refresh_tokens 尚未提交。客户端重试；Argon2id 是幂等的；事务从未提交，无孤立行。 持久性要点：只有数据库提交才能使 refresh token 可用。

操作间崩溃（密钥轮换中断）：步骤 1（新密钥发布至 JWKS）已完成；步骤 2（将签名者切换到新密钥）尚未执行。重启时，IdP 读取"当前签名密钥"指针；若未翻转，则继续使用旧密钥签名。新密钥已在 JWKS 中但未被使用——验证器在轮换期间同时接受 新旧两个 旧密钥和新密钥。对客户端无影响。

IdP 不可用 （集群分区，5 分钟）：新登录失败，刷新失败，但 验证继续进行 ——JWKS 已被 CDN 缓存，验证器在热路径上不依赖 IdP。 恢复：多地域 IdP 配合 anycast 或 GeoDNS 故障转移。 持久性要点：CDN 上的 JWKS 设置长 TTL，在 IdP 中断期间保持验证可用。

吊销列表不同步：部分验证器上的 Kafka 消费者积压 10 分钟——在这 10 分钟内，已吊销的令牌仍然有效。缓解措施：监控每个分区的积压，超过 5 秒则告警。高安全路径启用失败关闭模式（若 feed 超过 X 秒未更新，则切换为"直接查询 Redis 拒绝名单"）。 持久性要点：Kafka 的副本日志是事实来源；本地拒绝名单是物化视图，可通过重放重建。

签名密钥泄露（事后发现）：审计日志显示该密钥上周曾被短暂访问。发布新密钥 kid 到 JWKS 并切换签名，在 10 分钟宽限期后删除旧密钥以使其签名的所有令牌失效，强制所有用户重新认证，并对可疑时间窗口内的所有活动记录审计日志。

网络分区（认证服务 ↔ 用户存储）：认证服务可以访问 Redis 但无法访问 MySQL。新登录失败（需要读取用户行）；验证继续（仅使用 JWKS）。 恢复：在认证地域部署 MySQL 读副本；故障转移至只读模式。 持久性要点：MySQL binlog 是事实来源；副本在恢复后重新应用。

用户存储分片永久丢失：一个 MySQL 分片被销毁（硬件火灾 + 副本不可用）。约 400 万用户无法登录；现有令牌在过期前仍然有效；刷新时查询 MySQL 失败。 恢复：从最近一次全量备份 + binlog 重放至故障点进行恢复。数据丢失窗口 = 备份间隔（通常 15 分钟）减去 binlog 副本的覆盖范围。 持久性要点：每日全量备份 + 持续将 binlog 传送至独立地域；对最关键的事件（密码变更、MFA 注册）进行同步复制。

IdP 停机级联为全面故障：当 IdP 宕机，而每个服务都需要调用 IdP introspection 端点进行新鲜令牌验证时——每个服务认证失败，每个下游依赖服务随之失败，整个产品无法访问。2020 年 Cloudflare-Okta 部分中断事件展示了这一模式：局部的 IdP 问题级联为数小时的客户端登录故障。 缓解措施：缓存公钥（CDN 上的 JWKS 设置数小时长 TTL）；无状态 JWT 验证，在已签发令牌的有效期内可 无需 IdP 运行；对会话查询进行积极的内存缓存，使用户感知的爆炸半径为"5 分钟内无法新登录"而非"所有功能都挂了"。 持久性要点：IdP 必须处于冷路径。任何将 IdP 置于每个 API 调用热路径上的设计，都会在 IdP 抖动时使整个产品宕机。

签名密钥轮换事故 （旧令牌在部署中途停止验证）：轮换流程将新密钥 kid 推送到一半的验证器集群。收到新 JWKS 的那一半验证器拒绝了由旧密钥签名的令牌（旧密钥仍在 JWKS 中，但该 Pod 的内存缓存是在 之前 新密钥被推送之前加载的，旧密钥不知为何被 kid 丢弃了）。症状：约 50% 的 API 请求间歇性失败。诊断：验证器库的 JWKS 缓存 TTL 为 6 小时；缓存未命中时会获取新鲜 JWKS，但缓存命中时使用的是不再包含两个密钥的过期条目。 invalid_token：密钥轮换时，强制清除所有验证器的 JWKS 缓存（滚动重启，或发布"JWKS 已更新"pub/sub 事件）；始终保持 解决方案重叠期 ——新密钥签名后，旧密钥至少在 JWKS 中保留 2 倍访问令牌 TTL 的时长。 教训 ：密钥轮换比人们想象的更容易出错；应将其视为协调部署，而非单次 API 调用。跨服务时钟偏差导致令牌误拒

"直接将会话存储在 MySQL 中，每次请求按会话 ID 查询。"

这是 2010 年前的设计。在 2M QPS 下，即使 256 个分片也会产生约 8k 次查询/秒/分片，超出 MySQL 的舒适区（约 5k/秒持续随机访问）。缓冲池频繁抖动。跨地域每请求增加 60ms RTT。在前面加一层 Redis 就变成了"会话缓存"架构——但每次登录仍有 MySQL 写操作。正确做法：本地验证 JWT；仅在需要吊销证据时才查询 Redis。 "使用 JWT 且永不吊销。"

早上 9 点笔记本被盗——窃贼拥有 15 分钟的访问令牌使用权、30 天的 refresh token 使用权。对于银行、医疗、管理工具而言：灾难性的。合规要求（GDPR、HIPAA、SOX）均要求可证明的即时吊销能力。登出、密码变更、账号注销、管理员强制重新认证均不是可选项。 "使用 HS256（对称加密）并与每个服务共享密钥。"

N 个服务、N 份签名密钥副本——任何一处泄露都可为整个系统伪造令牌。某位初级工程师将密钥复制到自己的笔记本上"用于调试"。审计变得不可能。密钥轮换需要同步部署。正确做法：RS256（或 EdDSA），HSM 支持的私钥，为验证器提供公钥 JWKS。2M QPS 分摊到数千核心上时，40 倍的验证开销微不足道。 "每次请求都调用 IdP introspection 端点验证令牌。"

IdP 成为瓶颈——处理 2M QPS 的 introspection 意味着需要与 API 层规模相当的集群。每次 API 调用都有跨地域延迟（50-100ms）。单点故障。正确做法：在每个服务中进行无状态 JWT 验证；introspection 仅用于不透明令牌，即吊销及时性比吞吐量更重要的场景。 §26. 扩展维度

§28. 使用场景图谱

消费者登录（Google、Meta）。 基于 OAuth 2.0 的 OIDC，RS256 JWT，JWKS 位于 /.well-known/openid-configuration。内部：BeyondCorp 零信任网格。通过推送或 Titan 硬件密钥进行 MFA。需求：数十亿账号、广泛的设备覆盖（Web/移动/TV/手表）、可接受分钟级吊销延迟。适配方案：联邦 OIDC + 混合 session cookie + JWT。

企业 SSO（Okta、Auth0）。 多租户 IdP。每个租户配置其主 IdP（Active Directory、Workspace 或自定义）；Okta/Auth0 负责联邦并通过 SAML 或 OIDC 向下游 SaaS 颁发令牌。使用 SCIM（跨域身份管理系统）管理用户生命周期。需求：租户间密钥隔离、细粒度管理员策略、合规性（SOC 2、ISO 27001、FedRAMP）。适配方案：租户级签名密钥 + SaaS API 使用 opaque token。

移动端认证（Apple Sign-In、OAuth + PKCE）。 授权码 + PKCE — 移动应用属于"公共客户端"，无法保存密钥，因此 PKCE 将授权码绑定到仅合法应用知晓的验证器。ES256（签名更小，更适合移动端）。每应用的假名标识符（Apple 的"隐藏我的邮件"）。需求：小报文体积、在移动芯片上快速执行加密、支持生物识别解锁。适配方案：ES256 + PKCE + 存储在 iOS Keychain / Android KeyStore 中的 refresh token。

API 认证（Stripe API 密钥、GitHub PAT、OAuth 应用）。 Opaque bearer token。Stripe 密钥（sk_live_...）通过快速分布式缓存进行验证。GitHub Personal Access Token 类似。GitHub Apps 颁发短期安装令牌（1小时）；应用本身使用由应用私钥签名的 JWT 进行认证。需求：即时吊销、细粒度权限范围、高并发单 token 请求量。适配方案：opaque token 在边缘节点缓存。

服务间网格（SPIFFE/SPIRE、mTLS）。 每个工作负载通过 SPIFFE（spiffe://prod.example.com/ns/checkout/sa/checkout-svc）获得加密身份，并持有短期 SVID（1小时 X.509 证书）。每节点的 SPIRE agent 自动轮换。Istio/Linkerd 处理 mTLS。用户身份以请求元数据中的 JWT 形式叠加。需求：零操作身份轮换、无共享密钥、连接层加密身份。适配方案：SPIFFE + JWT。

联邦身份（企业 SAML）。 企业 IdP（ADFS、Okta、Azure AD）在用户登录时向 SaaS 颁发已签名的 SAML 断言。SaaS 使用客户预上传的 IdP 证书验证签名，提取用户属性（邮箱、群组），并创建本地会话。需求：兼容遗留企业系统（每家 Fortune 500 企业都在用 SAML）、属性映射、JIT（即时）供给。适配方案：SAML 2.0 + 已签名 XML + 浏览器 POST 绑定。新 SaaS 倾向于使用 OIDC，但必须为 B2B 场景提供 SAML 支持。

无密码认证（WebAuthn / passkey）。 用户注册硬件支持的凭据（Touch ID、Face ID、YubiKey、Windows Hello）。私钥永不离开设备的安全飞地。登录时：信赖方发送挑战，设备签名，服务器用已注册的公钥验证。防钓鱼 — 凭据绑定到来源（https://example.com），无法被呈现给 https://exarnple.com。需求：防钓鱼、禁止密码复用、无需密码重置支持负担。适配方案：FIDO2/WebAuthn。消费者使用 Apple/Google passkey（云同步）；企业使用 YubiKey/Titan。

规模化下的制胜模式： 热路径使用已签名凭据（JWT 或等价物）；通过短 TTL + 黑名单实现吊销；B2B 场景使用身份联邦；服务间使用工作负载身份（SPIFFE/mTLS）；终端用户凭据使用 KDF 哈希密码并升级至 passkey。

§30. 总结

认证与身份是典型的"热路径 / 冷路径"技术范畴：每次请求的验证必须是无状态且基于加密的（在本地验证已签名 JWT，每次 RS256 验证约 ~100µs CPU），而状态 — 会话、吊销列表、用户记录 — 则存在于冷路径，吊销事件通过 Kafka 扇出至此。约 10,000:1 的登录与验证比率决定了架构方向：任何在热路径上对中央存储发起请求的设计都无法支撑规模。认证的核心难题是凭据存储（Argon2id，内存困难，不可逆）和无状态世界中的吊销（短 TTL + jti 黑名单 + 每用户 min-iat）。授权的核心难题是租户间爆炸半径隔离和关系检查的扩展性（资源共享使用 Zanzibar 风格 ReBAC，粗粒度角色使用 RBAC）。纵深防御意味着每一跳都需验证，每个令牌都携带受众，每个签名密钥都存于 HSM，每个操作都可审计 — 因为一旦用户表泄露，阻挡攻击者与用户之间的唯一屏障，就是 KDF 的工作因子。
