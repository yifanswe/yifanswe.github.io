关于 TSDBs（时序数据库） ——它们是什么，为何通用关系型引擎在其工作负载下会崩溃，字节级机制（Gorilla XOR 浮点、delta-of-delta 时间戳、2h 块、倒排标签索引），设计空间（Prometheus、VictoriaMetrics、InfluxDB、TimescaleDB、M3DB、Thanos/Mimir/Cortex、Pinot/Druid、QuestDB、ClickHouse），以及它们所服务的工作负载——监控指标、IoT（物联网）传感器遥测、金融市场数据、应用性能、业务 KPI（关键绩效指标）。用例作为示例贯穿全文——§4 的 Kubernetes 监控、§5 的 Tesla 车队遥测、§6 的交易系统、§15 的智能电表数据——从不作为叙述框架。技术本身才是主题。

§1. 时序数据库的定义

一个 TSDB（时序数据库） 是专为以下场景设计的存储引擎： 仅追加、带时间戳的数值型数据 其访问模式以（a）时间窗口范围扫描、（b）窗口内聚合（sum、avg、percentile、rate）、（c）从细粒度到粗粒度分辨率的降采样，以及（d）自动清理旧数据为主导。核心特性是"以低成本大量摄取带时间戳的数据点，快速查询时间窗口，并按计划自动丢弃旧数据"。

一个时序样本形如 (metric_name, label_set, timestamp, value) ——例如 http_requests_total{service="api", region="us-east-1", status="200"} 4823 1716391200000。该键值对 (metric_name, label_set) 定义了一条 时序 ——一个由 (timestamp, value) 数据点组成的逻辑流。该对 (timestamp, value) 是其中一个 采样。一个生产集群可能拥有 1000 万条活跃时序 每条每 10 秒接收一个样本——稳态下每秒 100 万个样本。这就是设计基准点。

TSDBs 的定位与边界

TSDBs 是存储领域中的一个细分方向，与初学者容易混淆的三个相邻类别截然不同：

  • 对比 OLTP（在线事务处理）数据库（PostgreSQL、MySQL）。 OLTP 用于处理结构化数据的 ACID（原子性、一致性、隔离性、持久性）事务性变更——银行余额、用户档案、库存计数。将指标工作负载放入 PostgreSQL 是一种浪费：没有需要提交的事务（每个样本相互独立），没有 join（一个指标是自包含的时序），除 (series, time_range)之外几乎没有点查询，而浮点值在列式编码下压缩效率高出 10 倍，行式存储显然不适合。1M samples/sec 的工作负载会压垮 PostgreSQL 主节点；同样的工作负载在 Prometheus 上仅需一台 4 核虚拟机。这种不匹配是结构性的，而非调优问题。

  • 对比 OLAP（在线分析处理）数仓（BigQuery、Snowflake、Redshift）。 OLAP 用于通用分析查询——跨 PB 级数据的 group-by、join、窗口函数。它可以存储时序数据，但需付出通用性代价：没有专用时间戳编解码、没有倒排标签索引、没有自动数据保留、没有 1.37 字节/样本的压缩率。TSDB 是 专为时序优化；OLAP 是 时序能力。交叉点约在每日 10 亿样本——低于此时 ClickHouse 或 Pinot 就够用；超过此量，专用 TSDB 在存储和查询成本上领先 5–20 倍。

  • 对比事件存储 / 事件溯源（EventStoreDB、Kafka 作为日志）。 事件存储记录离散的领域事件（OrderPlaced, PaymentCaptured）并支持 回放 日志以重建状态。TSDB 记录的是 度量值 （一个 CPU 样本、一个请求计数），目标是"时刻 T 的值"或"过去一小时的速率"，而非"重建版本 N 的聚合状态"。事件溯源关注的是 因果关系与状态重建；TSDB 关注的是 度量与聚合.

TSDB 不擅长的场景： 复杂 join（无关系型 join 算子；PromQL——Prometheus Query Language——支持标签匹配 join，但不支持任意跨指标 join）、对历史样本的事务性更新（回写对存储层不友好）、按非时间键的低延迟点查询（应使用 Redis 或键值存储）、非结构化文本载荷（日志和链路追踪应使用各自专属引擎——参见 可观测性文档).

心智模型：TSDB 是 "每秒数百万浮点数、按标签索引、按时间窗口查询、按计划丢弃"这类需求的正确答案。 凡不符合该特征的场景，TSDB 都是错误的工具。

§2. 固有保证

TSDB 在设计上提供什么，以及系统设计者仍需在上层叠加什么。

设计上已提供

  • 高写入吞吐量。 单个 Prometheus TSDB 节点在商用硬件（16 vCPU、64 GB RAM、NVMe SSD）上可支撑 每秒 100 万个样本 ；VictoriaMetrics 声称在相同硬件预算下可达 每节点 1000 万 。Datadog 内部 TSDB 每天摄取 数万亿个样本 。写入吞吐量随核心数线性扩展，直到 WAL fsync 成为瓶颈。

  • 激进压缩。 Gorilla 编码（§4）在真实生产数据上可达 每样本 1.37 字节 ——时间戳 + 64 位浮点平均压缩到不到 11 位磁盘空间。与朴素的 16 字节行存储相比，存储效率差距为 ~12×.

  • 快速时间范围聚合。 类似这样的查询 sum(rate(http_requests_total[5m])) by (service) 对 1000 条时序的查询可在毫秒内返回，因为（a）样本按时序顺序存储，（b）查询引擎将聚合下推到 chunk 解码器。

  • 自动数据保留。 删除超过 N 天的数据块。无需手动 DELETE 操作；TSDB 自行管理旧数据的生命周期。Prometheus 的 --storage.tsdb.retention.time=30d 参数即可完成全部工作。

  • 基于标签的查询索引。 从 (label_name, label_value) → series_ids 让查询能够按任意标签组合过滤，无需扫描每条时序。

需在其上叠加

  • 跨时序聚合正确性。 进程重启时会重置的计数器，若直接做差分看起来会产生负速率。应用必须使用单调递增计数器类型，同时查询语言也必须能理解计数器重置（PromQL 的 rate() 能处理此问题；对样本执行原始 SQL 则不行）。

  • 基数预算管理。 高基数标签（user_id, request_id, trace_id）会为每个唯一值生成一条时序。1000 万条时序是单节点的临界点。SDK 必须在这类标签写入 TSDB 之前先丢弃或哈希处理。

  • 长期数据保留。 原生 Prometheus 的保留期为数天到数周的全分辨率数据。一年级别的保留需要叠加 Thanos、Mimir、Cortex 或 VictoriaMetrics 的长期存储层——一个以对象存储为后端的联邦层。

  • 高可用性。 单个 Prometheus 实例是单点故障。HA（高可用性）需要运行两个 Prometheus 实例抓取相同目标，并在上层配置去重查询层。

  • 数据回填。 加载历史数据——比如去年股价的 CSV 文件——并非 Prometheus 的设计目标。TimescaleDB、InfluxDB、ClickHouse 和 QuestDB 均支持回填；Prometheus 只是最近才添加了该功能。

  • 多租户。 裸 TSDB 仅服务单一租户。Cortex/Mimir 增加了按租户的 ingester 分片、按租户的查询限制和按租户的数据保留。SaaS 可观测性平台在此基础上叠加计费功能。

  • 副本间的线性一致性。 TSDBs 通常是最终一致的——副本在抓取抖动期间可能短暂出现偏差。对于监控而言这无关紧要（1 秒的样本漂移不可感知）；但对于金融结算级时序数据则不可接受。

契约：TSDB 存储带时间戳的浮点数、查询时间范围、丢弃旧数据。其他一切——多租户、高可用性、长期保留、基数控制——均由运维人员在上层叠加。

§3. 设计空间

过去十年间，TSDB 类别分化为五个子类。它们在采集模型（拉式 vs 推式）、查询语言（PromQL vs SQL vs Flux）、保留设计（短期 vs 分层）以及访问模式（运维指标 vs 面向用户的分析）等维度上存在差异。

A. 监控 TSDB — Prometheus、VictoriaMetrics

拉式。TSDB 按计划抓取被监控目标的 HTTP /metrics 端点（默认 15–30 秒）。服务发现（Kubernetes API、Consul、基于文件）告知抓取器应轮询哪些目标。数据模型：每（metric, label_set, timestamp）一个浮点值；标签是定义唯一时序的键值对。查询语言：PromQL——函数式、标签感知，支持 rate(), histogram_quantile(), topk()、标签匹配 join。保留：默认较短（15 天）。长期保留需联邦层（Thanos/Mimir/Cortex/VictoriaMetrics 长期存储）。

优势：部署极简、Kubernetes 默认选择、生态系统丰富（Grafana、AlertManager、覆盖各类数据库/队列/代理的 exporter）。劣势：默认单节点、无内置 HA、约 1000 万活跃时序时会触碰基数上限。

VictoriaMetrics 是性能更强的 Prometheus 兼容替代方案——相同数据模型、相同 PromQL，存储密度和吞吐量高出 5–10 倍。当 Prometheus 节点触碰上限时可直接替换。

B. 通用 TSDB — InfluxDB、TimescaleDB

推式。应用和 agent（Telegraf、fluentd）通过 HTTP POST 或行协议写入样本。InfluxDB 有 tag（已索引）和 field（未索引）之分。TimescaleDB 使用特殊的 hypertable中的 PostgreSQL 行。查询语言：InfluxQL → Flux → SQL（Influx 颇为曲折的历史——Flux 曾是函数式语言，在 v3 中已弃用转而支持 SQL）。TimescaleDB 在 PostgreSQL SQL 基础上扩展了 time_bucket() 以及连续聚合功能。

优势：SQL 访问、与关系型数据 join（TimescaleDB）、任意回填、支持比数值更丰富的数据类型。劣势：单节点摄取上限低于 Prometheus 风格；对于"按标签的速率"查询，SQL 比 PromQL 更为冗长。

C. 实时分析 — Apache Pinot、Apache Druid

通过 Kafka 推送。专为 面向用户的次秒级分析查询而设计，而非运维监控。数据模型：列式 segment 中的行，按时间分区。查询语言：SQL。保留：带冷热分层存储的数周至数年。

优势：对 数十亿事件的次秒级查询。LinkedIn 的"谁看了你的主页"功能运行在 Pinot 上，对数年页面浏览事件的查询 p99 <100ms。劣势：运维复杂（controller、broker、server、deep store）；摄取速度慢于纯 TSDB；对"我的错误率是多少"监控场景过度设计。

Pinot 和 Druid 有时被称为 TSDB，有时被称为 OLAP。它们的形态像 TSDB（时间分区、仅追加、列式），但查询的是面向用户的看板，而非机器告警。

D. Prometheus 长期存储 — Thanos、Mimir、Cortex、VictoriaMetrics

接收 Prometheus 的 remote-write 流，或通过 sidecar 接入本地 2h 数据块。存储：超过约 6 小时的数据块使用 S3 级对象存储（S3、GCS——Google Cloud Storage、ABS——Azure Blob Storage）。架构：ingester 层（内存中的近期样本）+ store gateway（S3 查询路径）+ query frontend（缓存、拆分、并行化）。

优势：无限保留（S3 价格，$0.023/GB-月），通过租户分片实现水平扩展。劣势：运维面积大（5–7 个微服务）；来自 S3 的冷查询比来自 ingester 的近期数据查询慢 10–100 倍。

Mimir 是 Grafana Labs 的 Cortex fork——持续维护。 Cortex 是原始的 CNCF（云原生计算基金会）项目。 Thanos 是 sidecar 风格的替代方案——架构略有不同，效果相似。 VictoriaMetrics cluster 以更少组件完成相同工作。

E. 专用 TSDB — M3DB、OpenTSDB、QuestDB、ClickHouse-for-time-series

  • M3DB： 由 Uber 构建，针对 高基数 进行优化（Prometheus 的限制在 Uber 的规模下根本行不通）。M3 = ingester + coordinator + 查询层，以 M3DB 作为底层 TSDB。
  • OpenTSDB： 最早的（2010 年）基于 Hadoop/HBase 的 TSDB。大部分已被取代，但仍有部分公司在使用。
  • QuestDB： SQL 原生、单二进制文件，针对 金融市场数据优化。在单机上可达每秒数千万笔 tick。
  • ClickHouse for time-series： 通用列式 OLAP 数据库，配合 MergeTree 引擎 + 时间有序分区。已在 Uber、Cloudflare、eBay 作为 TSDB 部署，兼顾 SQL + 聚合分析。

§4. 字节级存储内部机制

这一节是区分"我部署过 Prometheus"和"我知道为什么 Prometheus 的磁盘占用只有朴素设计的 1/12"的分水岭。有三个关键点： 数据模型，则可使用 块布局，以及 两种压缩算法 （浮点数用 Gorilla XOR，时间戳用 delta-of-delta）。

4.1 数据模型：series、标签、样本

每个样本都存在一个逻辑 时序。一个 series 通过以下信息唯一标识：

metric_name + sorted(label_kv_pairs)
        ↓ hash
   series_id (typically uint64)

例如，指标 http_requests_total 加上标签 {service="api", region="us-east-1", status="200", method="POST"} 构成一个具有稳定 series_id的 series。不同的 status（status="500"）是 不同的 series，具有 不同的 series_id。这一点至关重要： 基数由唯一标签值组合决定，而非样本量。

样本是追加到 series 的 (timestamp_ms, float64_value) 对。稳定状态：10M series 每秒 1 亿样本 = 平均每 series 每秒 10 个样本 = 每 100ms 一个样本。

4.2 基于块的存储（Prometheus TSDB）

Prometheus 的磁盘格式是事实上的参考设计。M3DB、VictoriaMetrics、Cortex 和 Mimir 均采用相同结构的变体。

Head 块。 最近约 2 小时的数据存放在内存的 head 块中。新的抓取数据按 series 追加到内存 chunk。每个样本同时写入磁盘上的 WAL（预写日志） 用于崩溃恢复。Head 块内存大小大致为：

head_RAM ≈ active_series × (open_chunk_bytes + index_overhead)
         ≈ 10M series × (1KB open chunk + 200 bytes index)
         ≈ ~12 GB RAM for a 10M-series instance

持久化 2 小时块。 每 2 小时，head 块"切割"一次——关闭的 chunk 刷新为磁盘上的持久块，head 块重置。每个块是一个 自包含的不可变目录 存储其 2 小时窗口内的所有样本：

data/
  01HF3MRD8Y6J3QTBPGZS5RFK0E/      # ULID — universally unique time-sorted ID
    meta.json                       # min_time, max_time, stats, ulid
    chunks/
      000001                        # binary chunks (Gorilla-encoded)
      000002
    index                           # inverted label index + series table
    tombstones                      # rare — deleted ranges

块的不可变性是该设计的核心优势：块只需压缩一次、建立索引一次，之后永不重写。压缩将多个 2h 块合并为更大的块（8h → 24h → 多天），无需修改原始样本，只需重新编码为更紧凑的布局。

WAL 保障崩溃安全。 样本写入 head 块内存 chunk 之前，会先追加到磁盘上的 WAL 段。WAL 是记录的顺序流：

record_type | series_id | timestamp | value

WAL fsync 每约 100ms 批量执行一次（可配置）。崩溃后，head 块通过从最后一个检查点回放 WAL 来重建。 持久性要点：fsync 前最多 100ms 的样本可能丢失。 对监控而言可以接受（缺失一个 10s 样本不可见）；对金融结算而言不可接受（此时 TimescaleDB 的 synchronous_commit=on 或 QuestDB 的同步模式以较低吞吐量换取每次提交的持久性）。

Compaction。 每个 2h 块按固定间隔与相邻块合并——通常为 6h → 24h → 多天块。压缩过程：- 重新编码 chunk 以获得更好的压缩率（每 chunk 更多样本 → 更好的 delta-of-delta 和 XOR 局部性）。- 丢弃已标记删除的范围。- 重建倒排索引。- 减少查询引擎在长范围查询中需要访问的块数量。

4.3 倒排标签索引

PromQL 查询 sum(rate(http_requests_total{service="api", region="us-east-1"}[5m])) 必须首先找到所有匹配的 series service="api" AND region="us-east-1"。这实际上是 正是 Lucene 为全文搜索解决的问题：posting list 交集。

TSDB 维护一个 posting list 索引：

(label_name="service", label_value="api")        → [series_3, series_7, series_11, ...]
(label_name="region",  label_value="us-east-1")  → [series_3, series_11, series_19, ...]

AND merge: [series_3, series_11]  ← these series match both labels

Posting list 按 series_id 排序存储。交集运算是归并扫描——O(min(|A|, |B|)) 时间，即使列表很大也很快。查询引擎随后加载 [series_3, series_11] 在请求时间窗口内的 chunk 并解码。

基数爆炸是摧毁该索引的元凶。 每个唯一的 label_value 都会增加一条 posting list 条目。如果 user_id 是一个拥有 1000 万个唯一值的标签，该标签就有 1000 万条 posting list。仅索引的内存就超过 10 GB，按 user_id 过滤的查询从 O(log N) 退化为 O(N)。缓解方案：在抓取前于 SDK 层丢弃高基数标签（§7）。

4.4 压缩：时间戳的 delta-of-delta

监控 TSDB 中的时间戳几乎总是以固定间隔分布。每 10 秒抓取一次 → 时间戳为 T, T+10s, T+20s, T+30s, ...。相邻时间戳的差值是常数（10s、10s、10s）。 delta-of-delta 为零（10-10=0，10-10=0，10-10=0）。

Gorilla 编解码器对 delta-of-delta 进行编码：

For each sample after the first:
  delta_i = timestamp_i - timestamp_{i-1}
  delta_of_delta_i = delta_i - delta_{i-1}

If delta_of_delta == 0:          encode in 1 bit (the value '0')
If delta_of_delta in [-63, 64]:  encode in 7 bits + 2-bit tag (9 bits total)
If delta_of_delta in [-255, 256]: encode in 9 bits + 3-bit tag (12 bits total)
If delta_of_delta in [-2047, 2048]: encode in 12 bits + 4-bit tag (16 bits total)
Otherwise:                       fallback 32-bit delta + 4-bit tag (36 bits total)

稳定状态监控：99% 的样本 delta_of_delta = 0，因此时间戳压缩至 ~每样本 1 bit.

4.5 压缩：浮点数的 Gorilla XOR

同一指标的值往往高度相关。CPU 利用率的样本间变化仅为百分之零点几。内存计量值平滑变化。即使是请求速率计数器，在 rate()之后，变化也很缓慢。

Gorilla 论文（Pelkonen 等，VLDB 2015 — Facebook）观察到，两个连续 IEEE 754 双精度浮点数的 XOR 结果是一个高位和低位有大量零的小整数：

double_1 = 47.3       → IEEE 754: 0x4047A66666666666
double_2 = 47.5       → IEEE 754: 0x4047C00000000000
xor      = 47.3 ⊕ 47.5 = 0x0000666666666666
                          ↑ many leading zeros
                                                 ↑ many trailing zeros
                              ↑─ meaningful bits ─↑

Gorilla 将 XOR 结果编码为 (leading_zero_count, meaningful_bit_count, meaningful_bits):

If XOR == 0 (identical to previous): encode in 1 bit (value '0')
If meaningful_bits fit in previous range: encode in (meaningful_bits) + 1 bit (skip header)
Otherwise:                          5 bits leading + 6 bits length + meaningful_bits + 2-bit tag

生产环境中实测浮点数平均 ~每样本 7-11 bits 在生产环境中。

4.6 每样本 1.37 字节这个数字

综合来看，一个样本：- 时间戳： ~1 bit（delta-of-delta=0 情况）至最坏情况约 16 bits → 平均约 1-4 bits。- 数值： 通过 XOR 平均约 7-11 bits。- 每样本开销 （分隔符、填充）：摊销后可忽略不计。

总计：平均每样本约 11 bits = 每样本 1.37 字节。与朴素方案 (uint64 timestamp, double value) = 每样本 16 字节相比，Gorilla 的存储密度 ~提升 12 倍.

在每秒 100 万样本的工作负载下，这意味着 磁盘写入 1.4 MB/秒 vs 16 MB/秒 ——即同样一块 4 TB 磁盘，Gorilla 密度可存约 33 天，朴素方案仅约 3 天。再叠加压缩级别的重新编码，就能实现 Prometheus 在普通硬件上宣传的数月数据保留。

4.7 一次抓取的端到端流程

跟踪单个指标抓取从 HTTP 请求到字节落盘的全过程：

1. App exposes /metrics in text exposition format:
   http_requests_total{service="api",status="200"} 4823 1716391200000

2. Prometheus scraper HTTP GETs /metrics every 10s.
   Parses each line: (metric_name, label_set, value, timestamp).

3. Hash (metric_name + sorted_labels) → series_id.
   If new series_id, allocate it in the in-memory series table and
   add posting-list entries to the inverted label index.

4. Append (timestamp, value) to the open chunk for series_id in the head
   block. Chunk grows by ~1.4 bytes due to Gorilla encoding.

5. Append WAL record (series_id, timestamp, value). WAL fsync batched
   every ~100ms. Pre-fsync crash loses ≤100ms of samples.

6. After ~120 samples (~20 min at 10s scrape), the open chunk closes,
   gets Gorilla-encoded fully; a new open chunk replaces it. Closed
   chunks stay in head-block RAM until block cut.

7. Every 2h, the head block "cuts": closed chunks flush to disk in the
   block's chunks/ directory; inverted label index is built; WAL is
   truncated; meta.json written last (atomic block visibility); head
   block resets.

8. After 24h, compaction merges 2h blocks into 24h blocks; multi-day
   blocks into week-long blocks.

9. After retention (e.g. 15d), oldest blocks are deleted by rmdir.
   Atomic, no rewrite.

10. (Optional) Thanos sidecar uploads blocks to S3 for long-term
    retention; local retention can shorten once blocks are in S3.

整个流水线是"追加到内存，写入 WAL，批量 fsync，刷新块，压缩，过期"——没有随机页面重写，没有 btree 分裂，没有事务回滚。正是这种简洁性使得单节点每秒处理 100 万样本成为可能。

4.8 为何比通用数据库快 10-100 倍

朴素的 PostgreSQL 表 (metric_id, ts, value) 加上 (metric_id, ts) btree 索引：

  • 每行 16 字节（key）+ 16 字节 value + 30-40 字节 btree/MVCC 开销 = 每样本约 60 字节。
  • 随机页面写入（每次插入触发 btree 页面重写）→ SSD 天花板约 2-3 万次插入/秒。
  • 没有时间感知压缩。

专用 TSDB：

  • 每样本 1.4 字节（Gorilla）。
  • 顺序 WAL 写入 + memtable + 不可变块 → 每秒 100 万+样本。
  • 存储减少 30-50 倍，吞吐量提升 30-100 倍。

约 12 倍的存储优势和约 30 倍的吞吐量优势是结构性的——来源于访问模式（单调递增时间戳、重复值、无更新），而非精妙的调优。 通用数据库无法通过增加索引来缩小这一差距；它必须变成一个 TSDB。

§5. 容量上限

通过真实部署案例，展示该技术在截然不同规模下的能力。

小规模——单台 Prometheus 虚拟机

配置： 在 4 核 16 GB RAM 的虚拟机上运行 Prometheus，配备 100 GB 本地 SSD。抓取 50 个目标，每个目标暴露约 100 个指标。稳定状态约 1 万样本/秒。保留 7 天。

  • 存储： 1 万样本/秒 × 86400 秒 × 7 天 × 1.4 B/样本 ≈ 8 GB 覆盖整个 7 天窗口。
  • CPU： 抓取 + Gorilla 编码 + WAL 写入占用 4 核 CPU 不足 10%。
  • 查询延迟： 常见仪表盘（1 小时内按服务统计 CPU/内存）响应时间 <100ms。

对于在 Kubernetes 中运行 50 个容器的初创公司已绰绰有余。瓶颈出现在约 1 万个目标 / 100 万 series 时——head 块的内存占用。

中规模——典型 SaaS 可观测性平台

配置： 5 个 Prometheus 实例抓取 2000 节点的 Kubernetes 集群，加 Thanos sidecar 上传至 S3。稳定状态约 100 万样本/秒。本地保留 30 天 + S3 保留 1 年。

  • 活跃 series： 所有实例合计约 500 万。
  • 日样本量： 100 万 × 86400 = 约 860 亿样本/天。
  • 日存储量： 860 亿 × 1.4 B = 压缩前本地约 120 GB/天；压缩级重新编码后 S3 约 50 GB/天。
  • 本地 30 天： 压缩后约 3.5 TB。
  • S3 一年： 约 18 TB，按 $0.023/GB-月计算 = 约 $415/月。

大多数生产可观测性平台处于此规模。瓶颈：标签边界控制不当导致的基数问题，以及长时间范围查询的成本（全年 p99 延迟查询需访问 365 个日级块）。

大规模——Datadog 量级

配置： Datadog 内部 TSDB（专有，源自同一 Gorilla 系列）。根据公开资料： 峰值每天摄入超过 15 万亿个样本 在峰值时。

  • 每天 15 万亿样本 = 平均每秒 1.75 亿样本；峰值数亿样本/秒。
  • 按 1.4 B/样本的存储： 压缩后约 21 TB/天。
  • 集群规模： 数千个摄取节点，S3 长期存储达到数 PB 量级。

按租户分片（多租户 SaaS）加区域分片（数据驻留合规）加每客户速率限制，防止单一租户拖垮其他租户。

超大规模——Google Monarch

配置： Google 内部监控系统。根据公开资料（USENIX 2020）： 全球约每秒 1 万亿样本 发布时数据。

  • 每秒样本量： 约 1 万亿/秒 → 10^12 ÷ 10^6 = 每台服务器每秒 100 万样本，由全球约 100 万台摄取服务器组成的集群承载。
  • 架构： 内存内区域节点，叶节点存储近期数据，根节点聚合跨区域查询。
  • 热层无 SSD 持久化 ——Monarch 仅在内存中保存近期窗口，并在各区域间复制。可靠性依靠复制而非持久化。长期数据归档至海量对象存储。

Monarch 热层仅用内存是该设计的大胆抉择：在此规模下，SSD 回写成为瓶颈，内存 + 复制才是出路。

特定领域的规模

  • IoT——Tesla 车队遥测。 约 500 万辆车 × 数百个信号以 1-100 Hz 频率采集 → 每秒数百万样本。InfluxDB 和 ClickHouse 变体在汽车行业中较为常见。
  • 金融市场数据。 股票 tick 流（NYSE、NASDAQ 综合）峰值达到 每秒数千万 tick 的峰值。QuestDB 和 kdb+ 占据主导地位——针对单主机上的亚毫秒查询延迟优化。每个市场（含买卖价/成交量等更丰富载荷）约 50 GB/天。
  • 应用性能——LinkedIn 量级。 约 5000 个服务 × 100 个指标 × 0.1 Hz = 汇总约 5000 万样本/秒。由支持多区域联邦的内部 Prometheus 兼容 TSDB 提供支持。

规模跨越 六个数量级的样本/秒——从 4 核虚拟机的 1 万到 Monarch 全球集群的 1 万亿。存储引擎基础（Gorilla、delta-of-delta、不可变块、倒排标签索引）在各规模下完全一致。随规模扩展的是 引擎之上的拓扑结构 ——分片、联邦、复制、保留分层。

§7. TSDB 固有的难题

每个 TSDB 运维者都面临的六大结构性挑战，源自该技术的设计约束，与具体厂商或工作负载无关。

7.1 基数爆炸

问题所在。 高基数标签—— user_id, request_id, session_id, trace_id——任何拥有数百万唯一值的字段——会创建数百万条唯一时间序列。内存、索引大小和查询延迟均与基数成线性关系。

天真的解法。 "直接加标签，维度越多，可见性越好。"照做。

问题所在。 某 Web 服务发出 http_request_duration_seconds{user_id="..."} 并发送到 Prometheus。集群有 5000 万用户。TSDB 分配了 5000 万条时间序列，每条消耗约 1 KB 用于 open chunk 和索引条目。Head block 内存超过 50 GB，TSDB 发生 OOM（内存溢出）。重启后，WAL 回放时再次 OOM。生产环境的可观测性就此中断——仅仅因为有人出于好意添加了一个标签。

实际解法。 - 在 SDK 层丢弃高基数标签。Prom 客户端库 Counter.labels(user_id=...) 应当被 Counter.labels() 改用这种方式——每用户的详细信息进入 traces（每条 trace 是一个事件，而非一条时间序列），而不是 metrics。 - 对于"我需要查看某个用户的指标"，使用 exemplars ——直方图桶指向一个采样 trace_id 的指针。Exemplars 每个直方图桶只需一个 trace 指针，而非每个用户一条时间序列。 - 对于预防性防御，在 Prometheus 中配置 metric_relabel_configs ，在抓取时丢弃或哈希匹配某模式的标签。 - 设置 query.max-series-fetched ，在昂贵查询拖垮 TSDB 之前提前终止。

（在以下章节中有更深入的讨论 可观测性文档, §4.1.)

7.2 降采样与留存分层

问题所在。 原始 10 秒粒度的样本非常适合最近一小时的数据，但用于去年的数据则太浪费——没有人会在 365 天的时间窗口内查看秒级粒度。然而"年度"仪表盘很常见（容量规划、随时间变化的业务 KPI、合规报告）。

天真的解法。 永久保留原始样本，磁盘便宜。

问题所在。 100 万样本/秒 × 1 年 × 1.4 字节/样本 = 每年 44 TB 的原始数据。"显示过去一年 p99 延迟"的查询需要读取 365 个日块，每块约 120 GB，I/O 量巨大。查询耗时数分钟，仪表盘变得不可用，用户开始抱怨，你甚至开始怀疑监控是否值得。

实际解法。 分层降采样——保留多个分辨率，每个分辨率有各自的留存期：

分辨率 留存期 存储成本
原始（10s） 30 天 120 GB/天 × 30 = 3.6 TB
1m 聚合 90 天 12 GB/天 × 90 = 1.1 TB
5m 聚合 1 年 2.4 GB/天 × 365 = 880 GB
1h 聚合 5 年 200 MB/天 × 1825 = 365 GB

合计：约 6 TB 存储 5 年多分辨率数据，而原始存储需约 220 TB。 成本降低 35 倍。

实现机制： - Prometheus recording rules 预先计算聚合并将其写入新的时间序列。 rate(http_requests_total[1m]) 成为一个独立的时间序列 http_request_rate_1m ，仪表盘直接对其查询。 - Thanos / Mimir 降采样 是自动完成的——每个 2h 块会被降采样至 5m 和 1h 分辨率。 - TimescaleDB continuous aggregates 是 SQL 定义的、增量维护的汇总。 - InfluxDB continuous queries （旧版）以及 任务 （v3）实现相同功能。

查询层必须了解多个分辨率，并根据请求的时间范围选择合适的分辨率："给我 1 年的 p99"应命中 1h 分辨率的时间序列，而非原始数据。

7.3 高写入摄入

问题所在。 单个 TSDB 节点设计上支持约 100 万样本/秒；超过此限制需要分片。基数爆炸（7.1）即使在适度的样本率下也可能急剧推高摄入成本。

天真的解法。 "换一台更大的机器。"32 核、256 GB 内存、NVMe RAID。

问题所在。 在单节点超过约 300-500 万样本/秒后，WAL fsync 成为瓶颈（每次 fsync 约 50-200µs，100ms 批次限制了吞吐量）。Head block 索引超出可用内存，压缩 CPU 开始与摄入 CPU 竞争。无论硬件如何，垂直扩展在约 500 万样本/秒/节点处遇到天花板。

实际解法。 按时间序列分片。Mimir/Cortex 对 series_id 使用一致性哈希，将样本分散到 ingester ring 中。每个 ingester 拥有约 1/N 的时间序列。查询层将 PromQL 查询扇出到所有 ingester 并合并结果。M3DB 用自己的 ring 拓扑实现相同功能。VictoriaMetrics 集群按指标名称和标签指纹分片。

拐点：1 个实例 ≈ 100 万样本/秒；10 个实例 ≈ 1000 万；100 个实例 ≈ ~8000 万（因查询扇出开销呈次线性）。

7.4 长范围查询成本

问题所在。 类似这样的查询 histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket[5m])) by (le)) over the last 1 year 必须触及一年内的每个块——即使有降采样，也有数百个块。每个块都需要被打开、查询其索引、解码 chunks 并计算百分位数。

天真的解法。 直接执行，现代硬件够快。

问题所在。 在拥有 500 万时间序列的 Mimir 租户上进行朴素的 1 年查询，需扫描约 365 个日块，解码约 10 GB 数据，并执行百分位数计算。延迟：30-300 秒。一个用户点击此类查询即可使查询路径饱和；并发用户会让仪表盘完全卡死。

实际解法。 - 查询拆分与并行化。 Mimir/Thanos 查询前端将 1 年的查询拆分为 12 个月度子查询并行执行，然后合并结果。 - 结果缓存。 对稳定历史时间窗口的 PromQL 查询会被永久缓存（Mimir 的 results-cache）。重复加载的 1 年仪表盘在第二次加载时命中缓存。 - 预聚合 recording rules。 百分位数在摄入时每 5m 窗口计算一次并写入时间序列。1 年仪表盘查询预计算的时间序列，而非原始直方图桶——成本降低 100 倍。 - 自适应分辨率。 当时间范围较长时，查询前端将查询重写为使用更粗的分辨率（1h 而非 10s）。

7.5 多租户

问题所在。 一个 TSDB 为多个团队提供服务。团队 A 失控的基数和慢查询不应影响团队 B 的仪表盘。

天真的解法。 每个团队一个 TSDB。运维噩梦，成本重复，跨团队查询不可能。

问题所在。 50 个团队共享一个 TSDB 集群。团队 A 的开发者误加一个高基数标签，集群全局基数在 30 分钟内从 500 万激增至 5000 万时间序列，Head block 发生 OOM。 50 个团队同时失去监控 ，事后复盘定性为"噪音邻居"问题。

实际解法。 按租户的资源限制与分片摄入： - 每个发往 TSDB 的请求携带 tenant_id header（Mimir/Cortex 模式）。 - Ingester 按 (tenant_id, series_hash) 路由，使每个租户的时间序列驻留在租户专属的 ring 分片上。 - 每租户限制： 最大活跃时间序列数、最大样本/秒摄入速率、最大查询时间序列数、最大查询时长、最大查询内存。超出任意限制仅向该租户返回 429，其他租户不受影响。 - 按租户配置数据保留策略。 免费层保留 14 天，付费层 90 天，企业版 1 年。通过对块删除设置每租户 TTL 来实现。 - 每租户查询队列。 租户 A 的慢查询不会阻塞租户 B 的队列。

这是 Grafana Cloud、Chronosphere、Datadog、New Relic——所有 SaaS 可观测性平台的架构。同样的模式也适用于大型公司内部，其中每个工程团队即为一个"租户"。

7.6 回填

问题所在。 加载历史数据——去年的股票价格、离线车辆的归档遥测数据、从已废弃 TSDB 迁移过来的导出指标。

天真的解法。 直接向 TSDB 写入旧时间戳。

问题所在。 Prometheus（直到最近）明确拒绝时间戳超过约 1 小时的样本。原因：head block 在内存中且按时间顺序排列；乱序样本会强制重写 chunk，破坏不可变性约束。通过推送回填根本不受支持。运维人员的变通方案是停止 Prometheus，通过 promtool tsdb create-blocks-from rules 工作流加载原始块。

实际解法。 - TimescaleDB、InfluxDB、ClickHouse、QuestDB 原生支持乱序写入，因为它们不受 2h 不可变块设计的约束。 - Prometheus 现在支持乱序写入（自 v2.39，2022 年起），可配置乱序写入窗口——但存在复杂性和内存开销，并非默认行为。 - 对于 Prometheus 迁移，推荐路径是使用 promtool 离线生成块，然后将其放入数据目录。块压缩会将它们与现有数据合并。 - 对于长期回填，直接推送至长期层（Thanos 的 thanos receive、Mimir 的 distributor），该层可接受乱序数据，因为其块尚未最终化。

7.7 慢查询隔离

问题所在。 某用户的高代价查询—— topk(100, sum(rate(http_requests_total[1d])) by (user_id)) 作用于一个拥有 100 万时间序列、1 亿基数的租户——将数 GB 的 chunks 读入内存，运行数分钟，可能导致整个 TSDB OOM。

天真的解法。 信任用户写出好的查询。

问题所在。 一位初级工程师在凌晨 3 点调试问题，输入了错误的 PromQL 并按下回车。查询将 2 GB 数据读入查询引擎的工作内存，TSDB 发生 OOM。所有仪表盘和告警沉默长达 60 秒，等待重启完成。这位初级工程师现在要为一起真实故障负责。

实际解法。 - 查询超时 （默认 2 分钟，可按租户配置）。 - 每查询内存限制 (query.max-samples, query.max-fetched-series, query.max-fetched-chunks). - 查询队列 以限制并发度。 - 查询分析器 （Mimir 的读路径分析器，Grafana 的"Query Inspector"）在执行前估算成本。 - 资源配额 在查询前端拒绝预计会超出阈值的查询。

§8. 故障模式

针对具体崩溃和降级场景的恢复流程及持久性说明。

磁盘满

TSDB 实例的本地磁盘写满。WAL 写入失败，Head block 无法继续刷写。Prometheus 停止抓取新样本（可配置：丢弃或停止）。已确认写入 WAL 的样本在 WAL fsync 完成后可恢复；已有数据上的查询继续正常。 恢复： 增加磁盘，重启，WAL 在启动时回放。 预防： 设置 node_filesystem_avail_bytes / node_filesystem_size_bytes < 0.15告警，在升级前将留存期积极调低。

WAL 损坏

WAL 写入过程中断电或文件系统损坏。启动时，Prometheus 检测到 WAL 段的校验和不匹配，跳过该段并从下一条有效记录继续回放。 恢复： 丢失至多损坏段内的样本（最坏情况约 30 分钟）；Head block 从有效 WAL 记录重建。 崩溃前最后一个完整写入的批次。 WAL fsync——最后一次 fsync 之前的样本可恢复；约 100ms 批处理窗口内的样本丢失。

基数爆炸导致 TSDB 宕机

一次错误部署添加了高基数标签。时间序列数在 5 分钟内从 500 万增长至 5000 万。Head block 内存超出可用内存，Linux OOM Killer 终止进程；重启后 WAL 回放重新分配 5000 万条时间序列，再次 OOM。 恢复（优雅降级）： （1）通过 prometheus_tsdb_head_series 和"按时间序列数排行的指标"仪表盘定位问题指标；（2）添加 metric_relabel_configs 规则在抓取时丢弃该标签；（3）降低 chunk 写入队列并积极调整 --query.max-samples后重启；（4）如果 WAL 严重膨胀，删除旧段并接受样本丢失；（5）等待留存期过期以清理膨胀的块。 预防： 基数告警、抓取配置中按 job 设置基数限制、组织级别的标签命名规范。

查询 OOM

用户执行 topk(1000, ...) 且无时间过滤器，查询将 50 GB 数据读入内存。查询引擎触发每查询限制并返回错误；若无限制，则进程 OOM，其他查询在重启前全部失败。 恢复： 重启查询引擎（集群模式下为独立进程；单节点 Prometheus 模式下与主进程相同）。查询是只读操作——恢复方式仅为重启进程。

抓取目标超时级联

某个抓取目标的 /metrics 端点挂起超过 30 秒。抓取器的工作池耗尽，其他目标错过抓取窗口。每次抓取设有 scrape_timeout （默认 10s）；超时的目标被中止，并通过 up{...} == 0. 预防： 抓取超时 < 抓取间隔（始终如此）；限制每个目标的指标数量（部分导出器会暴露数万个指标）； metric_relabel_configs 在解析时丢弃噪声指标。

Ingester 副本分歧

在 Mimir 集群中，同一时间序列的两个 ingester 副本存在约 10s 的时钟偏差，或在网络抖动期间有一个副本遗漏了样本。查询前端从所有副本合并结果，并按 (series_id, timestamp). 崩溃前最后一个完整写入的批次。 Ingester 复制因子——RF=3 时，两个 ingester 可同时故障而不丢失样本。

§9. 为何不用通用数据库

诱人的论点："PostgreSQL 稳如磐石、众所周知，已在跑我的应用——为何不直接把指标也存进去？"

建一张表：

CREATE TABLE metrics (
  metric_name TEXT NOT NULL,
  labels JSONB NOT NULL,
  ts TIMESTAMPTZ NOT NULL,
  value DOUBLE PRECISION NOT NULL
);
CREATE INDEX ON metrics (metric_name, ts);

模拟 10 万样本/秒的摄入：

  1. 每条 INSERT 是一次 B-tree 页面修改。 随机页写入。NVMe SSD 随机写入：可持续约 10-30K IOPS。配合 group commit 和 prepared statements，大约 5 万行/秒。 超过 5 万/秒后，WAL fsync 会达到饱和。

  2. 每行存储开销。 8 字节时间戳 + 8 字节浮点数 + 约 50 字节用于 metric_name （TEXT）+ 约 80 字节 JSONB 标签 + 约 30 字节 B-tree 开销 = 约 180 字节/样本。 比 Gorilla 的 1.4 字节差 130 倍。

  3. 查询 SELECT avg(value) FROM metrics WHERE metric_name='cpu' AND ts > now() - '5m' GROUP BY (labels->>'service'). PostgreSQL 扫描 B-tree，获取匹配行。每行位于不同页面（不按时间序列共存）。10K 样本/秒下 5 分钟 = 300 万行，随机页读取 → 多秒延迟。

  4. 持续 INSERT 流产生的 MVCC 膨胀。 Autovacuum 跟不上；表膨胀；查询计划进一步退化。

  5. 留存管理。 DELETE FROM metrics WHERE ts < now() - '30 days' 是对数百万行的删除操作，会锁定页面、阻塞 vacuum、产生大量 WAL，整个集群随之变慢。在 TSDB 中，留存管理是 rmdir block_directory ——原子化、近乎瞬时。

这种性能劣势是 结构性的:

成本维度 PostgreSQL Prometheus 倍数对比
每样本存储 ~180 B ~1.4 B 差 130 倍
每块 SSD 持续写入速率 ~5 万 ~100 万 差 20 倍
5 分钟范围查询 秒级 毫秒级 差 100 倍
留存删除 数小时锁定 秒级（rmdir） 巨大
浮点数压缩 无/通用 Gorilla XOR 12×
时间戳压缩 none delta-of-delta 30×

在每个维度上 10-100 倍的差距，正是专用存储引擎存在的理由。PostgreSQL 是处理事务、连接和中小规模结构化查询的正确工具，但对于每秒数百万时间戳浮点数的场景，它并不适合。

TimescaleDB 通过在 PostgreSQL 之上添加带有 chunk 剪枝和压缩的 hypertable 层，弥补了部分差距——但这样做实际上已经让它成为一个 TSDB，只是恰好暴露 PostgreSQL SQL 接口。内部引擎已然不同；SQL 表面的抽象是唯一"仍是 PostgreSQL"的部分。

§10. 扩展维度

时间序列工作负载在两个本质不同的轴上扩展，需要不同的应对方法。

类型 1：更多时间序列、更多抓取目标——均匀增长

集群从 5K 扩展到 10K 个 Kubernetes pod，每个 pod 暴露约 100 个指标，活跃时间序列从 500 万增至 1000 万，样本率从 50 万/秒增至 100 万/秒。

修复方法： 水平分片。Mimir/Cortex 在 ingester ring 上使用一致性哈希，将新时间序列分散到更多 ingester 节点上。每个节点仍持有约 1/N 的时间序列；目标翻倍意味着 N 翻倍。查询前端扇出分散查询成本。

拐点： - 1 个实例 → HA 对（约 100 万时间序列/实例）： 添加第二个 Prometheus 实例抓取相同目标，查询层负责去重。 - HA 对 → 分片集群（每实例约 1000 万序列）： 过渡到 Mimir 或 VictoriaMetrics 集群。整个集群的基数超出单实例承载范围。 - 分片集群 → 多区域（集群级约 1 亿序列）： 跨区域联邦。每个区域拥有独立的 ingester 环；跨区域查询扇出至所有区域并合并结果。

类型 2：相同实体，每实体数据速率更高——热点强化

同样 5000 个 Pod，但出现了一个高基数标签。每个 Pod 的序列数从 100 增加到 50,000。总序列数从 50 万增加到 2.5 亿。采样率保持不变。

这才是难点所在。 按序列分片有助于摄入（每个 ingester 承接 1/N 的新序列），但无法优化 查询路径 ——像这样的查询 topk(10, sum(rate(...)) by (label)) 必须读取全部 2.5 亿个序列才能计算 topk。

在 Redis 中按令牌缓存 refresh token 有效性 - 在 SDK 层丢弃高基数标签。 首选且最有效的修复方案。如果不需要按用户细分的指标，就不要发出用户级标签。 - 预聚合视图。 定义一条 recording rule，在摄入时将 topk 物化为低基数派生序列。查询直接命中派生序列。 - 改用尾采样 trace。 每请求的详细数据应存入 trace，每个请求是一个事件，而非一个序列。 - 切换到支持更高基数的 TSDB。 Uber 构建 M3DB 的原因正是 Prometheus 的基数上限无法满足其使用场景。VictoriaMetrics 和 Mimir 支持比原生 Prometheus 更高的基数。

拐点： - 活跃序列 100 万–1000 万： 在大型节点上运行原生 Prometheus。 - 活跃序列 1000 万–1 亿： Mimir/Cortex 或 VictoriaMetrics 集群。 - 活跃序列 1 亿以上： M3DB 级别的系统，或通过积极的基数控制将其降回合理范围。

读放大

一个常被忽视的独立扩展维度： 单条 PromQL 查询 可能读取大量序列。 sum(rate(http_requests_total[5m])) 涵盖所有服务和所有状态码，会触及该指标匹配的每一个序列——对于适度打标签的指标可能超过 1 万个序列。乘以 N 个并发查询（Grafana 打开 30 个面板、100 个用户），查询路径会比摄入路径更早成为瓶颈。

修复方案：查询拆分、结果缓存、recording rules——与 §7.4 相同的技术。

§12. PromQL 与查询语言

查询语言是 TSDB 价值的另一半。拥有优秀存储引擎但查询语言糟糕的系统，不如引擎平庸但查询语言出色的系统受欢迎。PromQL 是监控领域的主流选择；SQL 是分析领域的主流选择。

PromQL

Prometheus 的函数式查询语言，与时间序列数据模型高度契合。

  • 即时向量 vs 范围向量。 http_requests_total 在某一时刻，每个序列返回一个样本（即时）。 http_requests_total[5m] 在过去 5 分钟内，每个序列返回一个时间窗口内的样本（范围）——不可直接展示，必须通过函数聚合后使用。

  • 聚合操作。 sum, avg, min, max, count, quantile, topk, bottomk结合 by/without 子句对序列进行聚合： sum(rate(http_requests_total[5m])) by (service) 计算 5 分钟窗口内的每秒速率，然后按服务求和。

  • rate() vs increase(). 新手最常见的困惑点。 rate(metric[5m]) = 窗口内每秒速率，自动处理计数器重置。 increase(metric[5m]) = 窗口内总增量 = rate × window_seconds。两者均能正确处理计数器重置。 rate() 适用于"每秒请求数"； increase() 适用于"窗口内总请求数"。

  • histogram_quantile(). 从 histogram 桶中计算百分位数。指标 http_request_duration_seconds_bucket{le="..."} 每个桶对应一个序列； histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket[5m])) by (le)) 跨实例聚合桶并插值计算第 99 百分位数。

  • 标签匹配 join。 metric_a * on(service) group_left() metric_b。功能有限——不是完整的 SQL join，但足以实现"用错误率乘以流量得到错误量"。

InfluxQL → Flux → SQL

InfluxDB 的查询语言在十年内演进了三次： - InfluxQL （初始版本，类 SQL）："熟悉 SQL 的人上手容易。"表达能力有限。 - Flux （v2 版本）：带有 |> 语法的函数式管道语言。功能强大，但作为新的 DSL 几乎无人了解。 - SQL （v3 IOx 版本）：基于列式引擎回归 SQL。

教训：查询语言设计很难，而且当 SQL 对时间序列查询"足够好用"时，用户不愿意学习新的 DSL。

SQL 扩展（TimescaleDB、ClickHouse、QuestDB）

带有时间序列函数的 SQL： - time_bucket('1 minute', ts) ——将样本分桶到 1 分钟窗口。 - first_value(), last_value() ——采样窗口起始/结束值。 - time_bucket_gapfill() ——用 NULL 或插值填充缺失窗口。 - LATERAL JOIN 用于窗口化查询。

TimescaleDB 示例：

SELECT time_bucket('5 minutes', ts) AS bucket,
       service,
       avg(value) AS p50,
       percentile_cont(0.99) WITHIN GROUP (ORDER BY value) AS p99
FROM metrics
WHERE metric_name = 'http_request_duration_seconds'
  AND ts > now() - INTERVAL '1 hour'
GROUP BY bucket, service
ORDER BY bucket;

ClickHouse 和 QuestDB 有类似的方言。SQL 的优势在于人人皆知，分析工具（Tableau、Looker、Superset）原生支持它。劣势在于，对于最常见的监控查询，PromQL 要简洁得多。

"指标查询的难点"教训

TSDB 新手通常在以下方面遇到困难： - 计数器重置处理。 进程重启时计数器会归零。朴素的 value_now - value_then 会产生负数。PromQL 的 rate() 能处理此问题；基于原始样本的 SQL 则不行，除非显式建模。 - 跨序列的样本对齐。 两个在略微不同时刻被抓取的序列具有偏移的时间戳。对齐它们进行 join 需要插值或窗口对齐。 - Histogram 聚合。 百分位数无法线性聚合。 avg(p99(serviceA), p99(serviceB)) 是 非 合并后总体的 p99。必须先聚合底层 histogram 桶，再计算百分位数。

这些陷阱对有经验的 SQL 从业者和初级工程师同样具有挑战性。它们不是 PromQL 的 bug——而是时间序列数据模型与查询语言相遇时必然产生的结果。

§13. 降采样与数据保留

TSDB 运营的经济学核算。

预聚合模式。 - Recording rules（Prometheus）。 按计划评估 PromQL 表达式，并将结果写回为新序列。 record: service:http_request_rate:5m / expr: sum(rate(http_requests_total[5m])) by (service) 让 dashboard 查询预聚合序列——比实时计算节省 1000 倍成本。 - 持续聚合（TimescaleDB）。 以 SQL 定义的汇总，增量维护——新的 hypertable 块会触发汇总扩展。 - 持续查询/任务（InfluxDB）。 按计划重新评估的声明式聚合。 - 物化视图（ClickHouse）。 存储聚合行的视图，通过行级触发器在插入时更新。 - Thanos / Mimir 降采样。 自动执行——每 2 小时的块降采样为 5 分钟；每 24 小时的块降采样为 1 小时。磁盘上同时保留三种分辨率的数据。

数据保留策略。 - Prometheus --storage.tsdb.retention.time=15d：超过 15 天的块由 rmdir ——原子操作，近乎即时。 - InfluxDB 保留策略：按桶对分片应用 TTL。 - TimescaleDB SELECT drop_chunks('hypertable', INTERVAL '90 days') 一次性删除整个 chunk。 - ClickHouse 按表或按列设置 TTL 子句；数据在后台合并中删除。

成本核算。 对于 SaaS 可观测性工作负载：- 原始 10 秒 × 100 个指标 × 1000 台主机 × 86400 秒/天 = 864 亿样本/天。- 以每样本 1.4 字节（Gorilla）计： 120 GB/天。- 30 天原始数据保留： 3.6 TB 本地 NVMe。- 1 年原始数据保留： 44 TB ——本地 SSD 压力较大，但存入 S3 每月约 1000 美元可接受。- 1 年 + 5 分钟降采样： 88 GB （5 分钟层）——微不足道。- 5 年 + 1 小时降采样： 约 90 GB 总计——微不足道。

核心模式： 在需要精确排查的短窗口内保留原始数据；超出此范围后积极降采样。 保留边界和分辨率的选择，是可观测性平台最大的单一成本杠杆。

§14. 专业化时间序列工作负载

不同领域对 TSDB 设计的不同部分构成压力。

  • 应用监控——RED 和 USE。 RED（Rate、Errors、Duration）用于请求驱动型服务——每个端点三个指标，大多数服务级告警的基础。USE（Utilization、Saturation、Errors）用于资源——CPU、队列深度、错误计数；大多数基础设施告警的基础。典型集群：每个服务约 50–200 个指标 × 数千个服务 × 10 秒抓取间隔 ≈ 5,000–50,000 样本/秒。

  • 基础设施监控——node_exporter。 Linux 主机指标：CPU、内存、磁盘、网络、/proc 检查。 node_exporter 每台主机暴露约 1000 个指标，标签数量有界——1000 台主机 × 1000 个指标 × 6 次抓取/分钟 ≈ 10 万样本/秒。

  • 业务 KPI。 每分钟注册数、每分钟营收、活跃用户数、转化率。由应用代码以 counter/gauge 形式发出（signup_total, revenue_dollars_total），通过 PromQL 查询或推送到 Pinot。基数由业务类别（活动、地区、层级）约束——通常不超过 1000 个唯一序列。

  • IoT 传感器遥测。 数百万设备 × 约 10–100 个传感器 × 约 1 Hz = 总计约 1000 万–10 亿样本/秒。 High（高级） 基数：每台设备是一个序列。500 万辆车的车队 × 50 个传感器 = 2.5 亿个序列——这正是 Prometheus 无法胜任的场景。回填必不可少（设备断线后重连）。适合的工作负载：InfluxDB、TimescaleDB、ClickHouse；基数极高时使用 M3DB。

  • 金融市场数据。 峰值每秒数千万 tick。摄入到查询延迟为个位数毫秒；纳秒级时间戳；值为买价/卖价/成交量元组，而非单个浮点数。适合的工作负载：QuestDB、kdb+、InfluxDB；历史 tick 分析使用 ClickHouse。

  • APM（应用性能监控）trace 摘要。 Trace 数据存储在 trace 存储（Tempo、Jaeger）中；从 trace 派生的指标——每服务 p99 延迟、每端点错误率——被摄入为时间序列。模式：尾采样 collector 将指标从 span 中提取出来（基于采样 span 计算的 RED 指标），作为 Prometheus exporter 发出。

  • 容器/Pod 指标。 Kubernetes 原生时间序列： kube-state-metrics （期望/实际 Pod 数、部署健康状态）， cAdvisor/kubelet （每容器 CPU/内存/网络）， metrics-server （HPA——Horizontal Pod Autoscaler 输入）。基数由 (namespace, pod, container) ——大型集群中通常为数十万到数百万个序列。

§15. 使用场景图鉴

不同产品、不同工作负载，均运行在 TSDB 上。

  • 服务监控——每个 Kubernetes 时代的后端服务。 最主流的 TSDB 使用场景。Prometheus（或兼容 Prometheus 的 TSDB）抓取每个服务的 /metrics 每个服务的端点。模式：约 1000 个服务 × 约 100 个指标 × 10 秒抓取 ≈ 每集群 10 万样本/秒。存储：10 万 × 86400 × 1.4 字节 ≈ 12 GB/天。对单个 Prometheus 实例轻而易举。云原生生态系统的默认选择。

  • IoT 遥测——Tesla、工厂车间、智能家居。 Tesla 超过 500 万辆车的车队，每辆车上报数百个遥测通道。工业场景：通过 OPC-UA 协议的工厂 PLC（可编程逻辑控制器）。智能家居：Nest、Ring、Hue。工作负载特征：高基数的每设备序列、每秒数百万样本的总聚合量、大量回填需求。典型技术栈：设备 → MQTT broker → Kafka → TSDB（InfluxDB 或 ClickHouse）→ Grafana。

  • 交易系统——实时股票价格。 NYSE 综合行情峰值约每秒 1000 万–5000 万笔交易。交易算法必须在微秒内获取最新 tick。典型技术栈：市场数据源（TCP/组播）→ 摄入服务 → 强劲单机上的 QuestDB 或 kdb+。查询"过去 1 分钟内 AAPL 按 100 毫秒分桶的交易"可在 1 毫秒内返回结果。

  • 云成本监控。 资源利用率 × 定价 = 成本时间序列。云厂商（AWS Cost Explorer、GCP Billing）提供数据；第三方工具（Vantage、Kubecost）进行聚合。模式：按小时获取每服务成本细分，存储为 cloud_cost_dollars{service=..., region=...}。数据量低（<1 万样本/秒），保留时间长（数年）。

  • 网络监控——SNMP 轮询。 SNMP（简单网络管理协议）是轮询路由器、交换机、防火墙已有 30 年历史的标准。NOC（网络运营中心）每 30 秒轮询约 10,000 台设备，每台设备约 100 个 OID（对象标识符）= 约 3.3 万样本/秒。技术栈：snmp_exporter / LibreNMS → Prometheus → Grafana。老式运维团队仍在使用 RRDtool（Round-Robin Database，1999 年）→ Cacti / MRTG（Multi Router Traffic Grapher）。

  • DevOps 过程指标——DORA。 部署频率、变更失败率、平均恢复时间。从 CI/CD 以 counter/gauge 形式发出；为管理层绘制"每团队每周部署次数"图表。GitHub Actions / Jenkins → Prometheus pushgateway → Grafana。数据量低，保留时间长。

  • 用户分析——DAU 与留存。 日活/周活用户数、队列留存曲线。从事件数据批量计算，结果为每个队列的小型时间序列。事件经 Kafka → Spark/Flink → Pinot 或 Prometheus 处理。仪表盘查询预计算好的时间序列。

  • 能源/电网监控——智能电表。 美国约 1.5 亿块智能电表，每块每 15 分钟（或高精度电表每 1 Hz）上报一次用电量。全国电力公司约 16.7 万样本/秒。合规保留期：7 年账单争议窗口。技术栈：电表 → AMI（高级计量基础设施）→ InfluxDB 或 TimescaleDB → 分析 + 计费。

§16. 真实案例

大规模部署 TSDB 的知名系统。

  • Prometheus。 诞生于 SoundCloud（2012 年），灵感来自 Google Borgmon。自约 2017 年起成为 Kubernetes 默认监控方案。单节点、拉取式。通过 Thanos/Mimir/Cortex 实现长期存储联邦。

  • Datadog。 SaaS 可观测性。专有内部 TSDB，基于 Gorilla 家族技术派生。 每天摄入数万亿样本 遍及整个客户群。基于推送的 agent + 每主机 collector。

  • Grafana Mimir / Cortex。 Mimir 是 Grafana Labs 对 Cortex（最初的 CNCF 水平扩展 Prometheus 项目）进行积极维护的 fork。开源。为 Grafana Cloud Metrics 提供支持。参考部署可处理数十亿个活跃序列。

  • Thanos。 基于 Sidecar 的 Prometheus 长期数据保留方案。以 S3 类存储为后端，通过查询层实现统一访问。被称为"对原生 Prometheus 的最小化升级"。

  • VictoriaMetrics。 单体或集群版本的 Prometheus 兼容 TSDB，具有更高的单节点吞吐量。提供开源版与企业版。可在更大规模下直接替换 Prometheus。

  • InfluxDB。 v1/v2 是最早的通用 TSDB。v3 IOx 是基于 Parquet 与 DataFusion（Rust）构建的全新引擎。InfluxDB Cloud 为托管服务；Aiven 等平台也提供托管版 InfluxDB。

  • TimescaleDB。 带有超表（hypertable）的 PostgreSQL 扩展。广泛应用于时序与关系混合型工作负载。 Timescale Cloud 是其托管版服务。

  • M3DB / M3。 由 Uber 构建并开源。专为 Uber 的高基数场景（约 10 亿活跃序列）而生。采用 Gorilla 衍生的块格式与自定义 Bloom filter 索引。M3 平台 = 写入器 + 协调器 + 围绕 M3DB 构建的查询层。

  • Facebook Beringei。 Gorilla 论文的原始实现。纯内存引擎，通过副本实现持久性，不依赖磁盘。该 论文本身 （Pelkonen et al., VLDB 2015）是 TSDB 领域公认的奠基性参考文献。

  • Netflix Atlas。 Netflix 内部 TSDB，针对内存中的运营仪表板进行了优化。开源。采用多层架构，包含内存热层与磁盘冷层。

  • Google Monarch。 Google 内部监控系统。根据 USENIX 2020 论文， 全球约每秒处理 1 万亿个样本。是目前公开描述中规模最大的 TSDB 部署。热层仅使用 RAM，跨可用区分布；近期数据路径中不涉及 SSD。具备跨地域的联邦查询层。

  • Apache Pinot。 LinkedIn 起源，开源。为 面向用户的分析查询 提供时序数据能力——如"谁查看了你的主页"、广告分析等。数十亿事件下 p99 延迟低于 1 秒。

  • Apache Druid。 与 Pinot 架构相近，诞生于 Metamarkets/Imply。在广告技术领域有大量使用。

  • ClickHouse。 由 Yandex 孵化，现为 ClickHouse Inc.。通用列式分析数据库；因其压缩率与写入速度而被广泛用于时序场景。Cloudflare、Uber、eBay、GitLab 均运行大规模 ClickHouse 集群。

  • QuestDB。 SQL 原生 TSDB，针对金融市场数据与 IoT 进行优化。单机可达每秒数千万次写入。

  • kdb+ / KX。 专有系统，在金融领域占据主导地位。使用 Q 编程语言。数十年来是投资银行处理 tick 数据的主流选择。

  • OpenTSDB。 2010 年代基于 HBase 的 TSDB。已基本被取代；仍在部分老旧系统中运行。

  • RRDtool。 1999 年的原始方案——轮询数据库（Round-Robin Database）。为网络监控（MRTG → Cacti）而生。采用固定大小的循环文件，内置自动降采样。因其简洁性，至今仍广泛部署于 NOC 和嵌入式系统。

§17. 总结

TSDB 是专为"仅追加、按标签索引的带时间戳浮点数、按时间范围查询"工作负载设计的专用引擎——在此类工作负载下，通用关系型数据库的性能比专用引擎落后 10 到 100 倍。其优势来自一系列结构性设计选择：单调时间戳可压缩为 delta-of-delta（约 1 bit/样本），相关浮点数可压缩为 Gorilla XOR（约 7–11 bit/样本），不可变的 2 小时数据块使数据保留成为可能 rmdir 而非 DELETE，倒排标签索引支持按任意标签组合过滤查询。主流模式——用于监控的 Prometheus + Grafana + AlertManager，加上长期存储层 Mimir/Thanos/VictoriaMetrics——是云原生的默认选择；在默认方案的设计边界之外，还存在针对特定场景的变体（Pinot/Druid 用于面向用户的分析，InfluxDB/TimescaleDB 用于 SQL + IoT，M3DB 用于万亿级序列基数，QuestDB/kdb+ 用于金融 tick 数据）。反复出现的故障模式是 基数爆炸 ——高基数标签（user_id, request_id）产生数百万个序列，导致 head block 内存溢出（OOM）。反复出现的成本优化手段是 分层降采样 ——原始样本保留数天，降采样聚合保留数年。当工作负载是每秒数百万个带时间戳的浮点数时，选择 TSDB；否则请选择其他方案。
