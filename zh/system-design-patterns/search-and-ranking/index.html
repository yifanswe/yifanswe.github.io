搜索与排序

2026/05/31 · 系统设计模式
目录
目录

通过将工作拆分为廉价的高召回率候选生成阶段和昂贵的高精度排序阶段，在分片倒排索引与近似最近邻（ANN）向量索引上并行执行，由查询规划器负责分散、汇聚、排序和数据填充，从而在 100 毫秒内从数十亿文档中返回最相关的 K 个结果。公共词汇见 Staff 级系统设计词汇表。密切相关的题目文档： 设计 Google Search, 设计 Twitter Search, 设计 Typeahead / 自动补全, 设计 Typeahead 自动补全建议, 设计 Netflix 风格的推荐系统, 设计 Facebook 信息流, 设计 Google Ads.

1. 核心难点

搜索与排序并非一个问题，而是两个共用同一界面的问题。第一个问题是 召回率：从包含 N 文档（其中 N 通常为 10^9 与 10^12）的语料库中，找出候选集合 C 该值 可能回答 查询的文档。第二个问题是 排序：从 C （通常是 10^3 与 10^5 个候选项中），对结果排序使得前 K 条（通常为 10条）最符合用户的真实意图。召回率衡量的是"我们是否遗漏了本应展示的内容"，排序衡量的是"最佳结果是否排在第一位"。这两个指标牵涉系统的不同部分，在白板面试中将二者混淆是最常见的失误。

朴素设计——将所有文档存入数据库，对每个请求执行 LIKE '%query%' 扫描——在三道独立的墙面前失败。第一道墙：线性扫描。即便对于子字符串匹配每篇文档采用极为激进的 ~10 µs 也需要 10^9 文档总计 ~10^4 秒（乃至数小时）每次查询，且在未完成数据分片前无法并行化。第二道墙：相关性。子字符串匹配无法判断哪篇文档 更好；只会返回第一个匹配项，而非最佳匹配项。第三道墙：语义。查询"最适合编程的笔记本电脑"应该能匹配标题为"开发者笔记本电脑推荐"的结果，尽管两者没有任何共同词元。单靠关键词索引会错过第三道墙；单靠向量嵌入则会错过精确匹配和结构化过滤（如"价格低于 $50"）。系统是否同时需要两者取决于负载类型——精确匹配主导的领域（电商 SKU 搜索、日志搜索、代码搜索）仅用关键词即可；语义意图主导的领域（网页搜索、推荐、"查找相似"）需要向量。面试中诚实的回答是判断你的产品属于哪种情形。

延迟使难度变得具体。消费者搜索的典型预算为 p99 端到端 200 ms，其中约 100 ms 分配给搜索后端，其余为网络 + 前端渲染。在 100 ms 内你必须：对查询进行分词，在多个分片上查找倒排列表，对其求交集/并集，用 Best Matching 25（BM25）打分，取出前N 个候选，可选地对每个候选的 N=1000 个特征运行第二阶段机器学习（ML）排序器，填充文档正文，最后合并。面对 10^12 个文档，单台机器无法完成；跨分片串行执行也同样行不通。你必须分片、并行 scatter-gather、积极剪枝，并接受这样一个事实： 最慢分片的尾延迟就是查询的延迟——S 个 S 独立分片的 p99 大约等于单分片的 p(1 - 0.01/S)，即 S=10 时单分片的 p99.9，S=100 时为 S=100, p99.99 at S=1000。这个数字正是对冲请求、按延迟选择副本以及按分片设置超时的合理依据，而非泛泛的"尾延迟很差"一句带过。整个架构是一项 跨并行 fan-out 的延迟预算分配研究，形状上与 fan-out 写放大 非常相似，但发生在读取侧。

最后一个转折：系统必须保持新鲜。三十秒前发布的推文应可搜索；刚刚降价的商品应重新排名。但重建一个 10^12文档的索引需要数小时。实际系统会对索引进行分层——一个小型、热门、近实时索引的"live"层，加上一个大型、廉价、批量构建的"main"层——查询时跨层合并。新鲜度 vs. 成本 vs. 索引构建复杂度是一个面试官必然深究的永久性权衡。

2. 如何在面试中识别背压问题

触发该模式的提示语

  • "查找与查询匹配的文档/推文/商品/视频。"
  • "返回前 K 最相关的……"
  • "按相关性、热度、新鲜度……排序。"
  • "个性化信息流 / 结果 / 推荐。"
  • "用户输入时自动补全"（专门的前缀变体）。
  • "语义搜索"/"查找相似项"（向量密集型变体）。
  • "展示与该查询/该用户相关的广告。"（排序密集型变体）。
  • "我们有数十亿条目，查询必须快速返回。"

架构异味

  • 以任意谓词（全文、过滤器、地理位置、时间）为条件的重读负载。
  • 语料库过大，无法放入单台机器；查询必须触及多个分片。
  • "最相关"由应用定义，并从反馈（点击、停留时间）中学习。
  • 新鲜度至关重要，与批量索引构建节奏并不相同。
  • 产品既有头部（热门查询、热门条目），也有庞大的长尾。
  • 读取质量由离线指标（recall@K、归一化折损累积增益（nDCG）、平均倒数排名（MRR））评判，而不仅仅是延迟。

该类型为核心的面试题示例

  1. 设计过去十年推文在 100k QPS 下的 Twitter 搜索。
  2. 为 10 亿用户设计 YouTube / Netflix 首页推荐。
  3. 设计 Web 级别的 Google 搜索。
  4. 设计支持文本、位置和评分的 Yelp 餐厅搜索。
  5. 设计亚马逊规模的电商商品搜索。
  6. 设计在 100k QPS、p99 低于 100 ms 条件下的 Google Ads 查询竞价排序。
  7. 设计 LinkedIn 的"你可能认识的人"（推荐，而非文本搜索，但使用相同的两阶段模式）。

3. 60秒口头框架

在切入架构之前，我会先估算问题规模。对于文档数低于约 1000 万、QPS 较低的语料库，单节点 Elasticsearch 配合 BM25 且不使用学习型排序器完全够用——Staff 级别的正确做法是不要 非 过度设计。有趣的场景是：语料库超出单机内存、延迟预算无法承受对全量语料库重排序，或相关性质量本身就是产品差异点。在这种情况下，我将其框定为两阶段检索问题。阶段一是候选生成：我希望高召回率、低延迟，对顺序宽松。使用分片倒排索引进行关键词查找，加上——若查询意图较强或具有语义性——基于学习嵌入的近似最近邻索引，跨分片 scatter-gather，每次查询拉回数百到一千个候选。阶段二是排序：取这些候选，使用从在线特征存储中获取的特征，运行学习型排序器——如 LambdaMART 这类梯度提升决策树（GBDT）或神经排序器——返回前十条。围绕这些，我会构建分层索引以保证新鲜度（live 层秒级新鲜，main 层批量重建）、头部查询缓存，以及在点击日志上重训排序器的离线管道。核心难题在于 scatter-gather 尾延迟、新鲜度滞后以及排序器的训练/服务偏差，我很乐意深入探讨。

4. 标准工具集

技术 适用场景 成本
倒排索引（Lucene 风格） 关键词检索、结构化过滤，是每个文本搜索系统的骨干。 索引构建代价高；查询复杂度与倒排列表长度成对数关系。
文档分片的 scatter-gather 语料库过大无法放入单机；查询 fan-out 到每个分片。 尾延迟由最慢的分片主导。
词项分片索引 超大语料库，热门词项的倒排列表极长。 热门词项存在热分片风险；不常见。
向量嵌入 + ANN（层次可导航小世界（HNSW）、倒排文件（IVF）、ScaNN） 语义搜索、推荐、"查找相似"。 召回率 < 100%；嵌入刷新成本高；内存占用大。
两阶段检索（廉价召回 + 昂贵排序） 只要排序信号对整个语料库的计算代价过高，即可采用。 阶段衔接处存在缺陷暴露面。
学习排序（LambdaMART，神经排序器） 从标注数据和点击日志中优化 nDCG / 点击率（CTR）。 训练管道、特征漂移、线上/线下一致性问题。
BM25（Best Matching 25）基准 倒排索引内的首轮打分——所有人都要与之竞争的基线。 对分词、词干提取、语言分析器选择敏感； k1 / b 超参数需按语料库调优。服务代价低（每文档微秒级）。
在线特征存储 + 离线特征管道 在排序时以微秒级延迟提供每查询/每文档/每用户特征。 线上/线下偏差是持久存在的 bug 来源。
查询改写 + 扩展 将表面形式桥接到意图（同义词、词干提取、拼写纠错）。 过度扩展会拉高召回率但增加延迟。
分层索引（live + main） 对无法频繁重建的大型语料库需要秒级新鲜度。 需要维护两条索引管道；查询时需要合并逻辑。
热门结果缓存 头部查询压力极大；缓存可消减大部分负载。 索引更新时的缓存失效，热 key 风险（见 缓存踩踏/读放大).

4.1 倒排索引构建（草图）

# Build a simple in-memory inverted index over a document corpus.
# In production, this is what Lucene / Tantivy does on disk with skip lists,
# block-compressed postings, term dictionaries, and positional info.

from collections import defaultdict
from dataclasses import dataclass, field
import math
import re

TOKEN_RE = re.compile(r"[A-Za-z0-9]+")

def tokenize(text: str) -> list[str]:
    # Real systems: language detect -> tokenizer -> lowercaser -> stemmer -> stopwords.
    return [t.lower() for t in TOKEN_RE.findall(text)]

@dataclass
class Posting:
    doc_id: int
    term_freq: int  # how many times the term appears in this doc

@dataclass
class InvertedIndex:
    # term -> list of postings (sorted by doc_id for fast intersection)
    postings: dict[str, list[Posting]] = field(default_factory=lambda: defaultdict(list))
    # doc_id -> document length in tokens (used by BM25)
    doc_len: dict[int, int] = field(default_factory=dict)
    avg_doc_len: float = 0.0
    n_docs: int = 0

    def add(self, doc_id: int, text: str) -> None:
        tokens = tokenize(text)
        self.doc_len[doc_id] = len(tokens)
        counts: dict[str, int] = defaultdict(int)
        for tok in tokens:
            counts[tok] += 1
        for term, tf in counts.items():
            self.postings[term].append(Posting(doc_id, tf))
        self.n_docs += 1

    def finalize(self) -> None:
        for term in self.postings:
            self.postings[term].sort(key=lambda p: p.doc_id)
        self.avg_doc_len = sum(self.doc_len.values()) / max(1, self.n_docs)

面试中需要注意两点：倒排列表按 doc_id 排序 保存，因此多词项查询可以通过线性归并（交集/并集）在 O(sum of postings list lengths)内回答；索引存储了足够的统计信息（词频、文档长度、文档数量），无需重新读取文档即可计算 BM25 分数。

4.2 BM25 打分（草图）

def bm25_score(idx: InvertedIndex, query_terms: list[str], doc_id: int,
               k1: float = 1.2, b: float = 0.75) -> float:
    # BM25 = sum over query terms of: IDF(t) * (tf * (k1+1)) / (tf + k1 * (1 - b + b * dl/avg_dl))
    score = 0.0
    dl = idx.doc_len.get(doc_id, 0)
    if dl == 0:
        return 0.0
    for t in query_terms:
        postings = idx.postings.get(t, [])
        df = len(postings)
        if df == 0:
            continue
        # Lucene-style IDF; smoothed to stay non-negative.
        idf = math.log(1 + (idx.n_docs - df + 0.5) / (df + 0.5))
        # Find this doc's term frequency. In production, postings are skip-listed
        # for sublinear lookup; here we do a binary search.
        lo, hi = 0, len(postings) - 1
        tf = 0
        while lo <= hi:
            mid = (lo + hi) // 2
            if postings[mid].doc_id == doc_id:
                tf = postings[mid].term_freq
                break
            if postings[mid].doc_id < doc_id:
                lo = mid + 1
            else:
                hi = mid - 1
        if tf == 0:
            continue
        denom = tf + k1 * (1 - b + b * dl / max(1.0, idx.avg_doc_len))
        score += idf * (tf * (k1 + 1)) / denom
    return score

BM25 是对词频/逆文档频率（TF-IDF）的概率改良。关键参数： k1 控制词频饱和度（使得"the"出现 100 次的得分不会是出现 1 次的 100 倍）， b 控制长度归一化（使一条 10 词的推文不会仅因原始词频而输给一篇 10000 词的页面）。它是每个学习型排序器都必须超越的基线。

4.3 跨文档分片的 scatter-gather（草图）

import asyncio
from dataclasses import dataclass

@dataclass(order=True)
class ScoredDoc:
    score: float
    doc_id: int
    shard_id: int

async def shard_search(shard_client, query: str, k: int) -> list[ScoredDoc]:
    # Each shard runs the inverted-index lookup + BM25 locally and returns its top-k.
    # In Lucene-speak: a per-shard `IndexSearcher.search(query, k)`.
    return await shard_client.search(query, k)

async def scatter_gather(shards: list, query: str, k: int,
                         per_shard_timeout_ms: int = 80) -> list[ScoredDoc]:
    # Fan out to all shards in parallel. Time-bound every shard call so one slow
    # shard cannot blow the global p99. Late shards are simply dropped; the
    # tradeoff is a small, bounded recall hit on the long tail.
    tasks = [
        asyncio.wait_for(shard_search(s, query, k), per_shard_timeout_ms / 1000.0)
        for s in shards
    ]
    results: list[ScoredDoc] = []
    for fut in asyncio.as_completed(tasks):
        try:
            results.extend(await fut)
        except asyncio.TimeoutError:
            # Record a shard-timeout metric; this is the most important search SLO signal.
            continue
    # Global merge: heap-merge top-k across all shards. We can use heapq.nlargest
    # because each shard already returned its sorted top-k.
    results.sort(reverse=True)
    return results[:k]

面试中有三个可调旋钮： 每分片超时 （值越低，尾延迟越紧；值越高，召回率越好）， 每分片 k （通常向每个分片请求 k * over_fetch_factor 个，因为全局前k 名并不均匀分布在各分片中），以及 部分结果策略 （返回现有结果，或直接失败）。对于搜索，部分结果通常是正确选择；对于支付，则绝对不能用部分结果。

4.4 两阶段排序管道（草图）

@dataclass
class Candidate:
    doc_id: int
    first_stage_score: float

def first_stage_retrieve(query: str, top_n: int = 1000) -> list[Candidate]:
    # Cheap: BM25 + ANN union, scatter-gather across shards, return ~1000 candidates.
    bm25_cands = bm25_search(query, k=top_n)
    ann_cands = ann_search(embed(query), k=top_n)
    # De-duplicate; keep max first-stage score as a feature for stage two.
    merged: dict[int, Candidate] = {}
    for c in bm25_cands + ann_cands:
        cur = merged.get(c.doc_id)
        if cur is None or c.first_stage_score > cur.first_stage_score:
            merged[c.doc_id] = c
    return list(merged.values())

def hydrate_features(candidates: list[Candidate], query: str, user_id: int) -> list[dict]:
    # Pull from an online feature store. Per-doc features (CTR, recency, quality),
    # per-(query, doc) features (BM25, semantic similarity), per-user features,
    # cross features (user x doc affinity).
    feature_store = OnlineFeatureStore()
    return feature_store.multi_get(
        doc_ids=[c.doc_id for c in candidates],
        query=query,
        user_id=user_id,
    )

def second_stage_rank(candidates: list[Candidate], features: list[dict],
                      ranker, k: int) -> list[Candidate]:
    # Expensive: GBDT or neural ranker scores each (query, doc, user) tuple.
    scores = ranker.predict(features)
    ranked = sorted(zip(candidates, scores), key=lambda x: x[1], reverse=True)
    return [c for c, _ in ranked[:k]]

def serve_query(query: str, user_id: int, k: int = 10) -> list[int]:
    cands = first_stage_retrieve(query, top_n=1000)
    feats = hydrate_features(cands, query, user_id)
    top = second_stage_rank(cands, feats, RANKER, k=k)
    return [c.doc_id for c in top]

两阶段模式之所以值得其复杂性，是因为有具体的延迟算术作为支撑，而非泛泛的最佳实践声明。阶段一（跨分片倒排列表的倒排索引查找 + BM25）每次查询约耗时 5-20 ms，因为 BM25 对每篇候选文档的打分是微秒级的。每个 (query, doc, user) 元组的学习型排序器每个候选通常需要 50-500 µs 的推理时间（GBDT）或 1-10 ms（神经网络）。如果对每篇匹配文档都运行学习型排序器，一个匹配 10^6 篇文档的查询将耗时数分钟；而对固定的 1000 个候选切片运行则需 50 ms 到数百 ms，完全在预算范围内。临界点在于： 当（阶段一的候选数）×（每候选阶段二的代价）超过延迟预算时，两阶段设计的衔接缺陷暴露面就是值得付出的代价。 对于小型语料库（< 1000 万文档）或廉价排序器（少量特征的线性模型），带 ORDER BY 的单阶段 BM25 完全足够，部署两阶段是过度设计。每个 Web 级系统——Google、Amazon、YouTube、Netflix、Twitter、TikTok、Yelp、LinkedIn Feed——都使用某种形式的两阶段 ，原因是 它们的候选数量和排序器代价使它们处于该算术的不利一侧。

5. 实际案例：朴素设计 → 崩溃 → 修复

工作负载。 Twitter 风格搜索： 100k QPS 对比 100 billion tweets （10 年历史，约 3000 万/天）。延迟服务等级目标（SLO）： p99 < 300 ms 端到端， p50 < 80 ms。新鲜度目标：现在发布的推文应在 5 seconds内可被搜索。每条推文包含全文、作者、时间戳、话题标签、语言、互动计数和嵌入向量。

5.1 朴素设计（面试反模式）

Client --> Search Service --> Postgres
                                    SELECT id FROM tweets
                                    WHERE text LIKE '%query%'
                                    ORDER BY created_at DESC LIMIT 10;

各组件的失败原因：

问题 后果
LIKE '%...%' 无法使用 B-tree 索引。 对 10^11 行进行全表扫描。单次查询：数小时。
单台 Postgres 主机。 在 100k QPS，主机在低于实际负载约 3 个数量级时就已过载。
ORDER BY created_at 仅按时间倒序排序。 没有相关性；垃圾信息横行；用户信任崩塌。
没有新鲜度管道。 若复制到只读副本，在写入峰值下延迟将无界增长。
没有过滤器：语言、屏蔽作者、已删除推文。 可见性泄漏。

此设计在召回率、排序、延迟和正确性四个维度上全部失败。

5.2 初次修复尝试（仍然有问题）

聪明的候选人会直接说："好，用 Elasticsearch。"

Client --> Search Service --> Elasticsearch (one cluster)
                                    GET /tweets/_search { query: ... }

这更好，但在这个规模下仍然是错的，面试官会深究其原因：

  • 单个集群无法容纳 10^11 文档 以合理的成本。Elastic 每分片的实际甜蜜点是 20-50 GB，因此需要数千个分片。
  • 跨数千个分片的 scatter-gather 会使尾延迟成为灾难：1000 个分片中最慢的 p99 远比集群整体 p99 差得多。
  • 排序仍然只是 BM25。 面对拥有学习型排序器的竞争对手，你将毫无胜算。
  • 没有新鲜度分层。 将每天 3000 万条推文内联索引进主索引会拖慢索引速度，并与查询负载产生竞争。
  • 没有缓存。 重复的头部查询（"世界杯"、"选举"）每秒多次冲击集群 100k * head_share 次。
  • 没有降级模式。 当某个分片宕机时，查询直接失败，而不是返回部分结果。

为何有效：

决策 原因
文档分片（而非词项分片） 无论查询内容如何，各分片负载均衡；查询会扇出到所有分片，但每个分片的工作量有界。词项分片方式在"the"这类高频词上会出现热点分片问题。
两层架构：实时层 + 主索引层 将数据新鲜度与索引构建成本解耦。实时层规模小（约10亿文档），可全部放入内存。主索引层以低成本在磁盘上保存长尾数据。
每分片超时 = 80 ms 限制尾部延迟。若某个分片超时，我们在长尾上接受部分召回，而不是让整个查询失败。
第一阶段使用 BM25 + ANN 联合检索 同时覆盖字面匹配和语义意图。单用 BM25 会遗漏同义改写；单用 ANN 会遗漏精确字符串查询（用户名、$cashtags、引号短语）。Twitter 搜索的查询组合中两类都有，这正是联合检索在此值得的原因——对于纯精确匹配场景（如用户名查找），ANN 是过度设计。
第二阶段使用 LambdaMART 基于约100个特征的 GBDT 是一个强大且服务成本低的基线。现代大规模系统通常在特征工程收益趋于平缓后迁移到神经排序器（deep & cross、带 cross-attention 的双塔、transformer reranker）；正确的选择取决于团队在训练基础设施上的能力和服务延迟预算，而不是放之四海而皆准的"GBDT 最优论"。
在线特征存储 在排序时特征读取延迟低于毫秒；离线训练，因此保持了线上/线下的特征一致性。
查询结果缓存，头部查询 TTL 约30秒 小部分查询占据了大部分流量；这将后端负载降低了约一个数量级。
排序后进行数据填充 不要为1000个候选文档支付完整的文档 I/O 成本——只对排名前10的结果进行填充。
在查询时进行可见性过滤，而非索引时 删除/屏蔽操作变化过于频繁，无法直接写入索引。

5.4 容量估算（面试风格的草算）

  • 10^11 tweets * 200 bytes posting overhead per term * 50 terms/tweet ~= 1 PB 对于主倒排索引来说，这是证明分层合理的预算。
  • 100k QPS * 1000 shards = 10^8 sub-queries/sec 集群的扇出负载。每个分片接收 10^8 / 1000 = 10^5 子查询/秒，这已远超单机可持续搜索 QPS（Lucene 分片对于非简单查询通常上限为几百到几千 QPS）。缓存 + 廉价查询拒绝 + 分片超配是让数字合理的关键。如果你没有提到缓存和每分片 QPS 目标，面试官应该会追问。
  • 散射-聚集尾延迟：在1000个分片并行扇出、每分片 p99 为20 ms 的情况下， 聚合 p99（1000个独立抽样中的最慢值）大约等于每分片的 p99.9——通常是每分片 p99 的2-5倍。在20 ms 后向副本发起 Hedged request 可以显著降低这一延迟，代价是约10%的额外集群负载。
  • 特征获取预算：1000个候选 × 约100个特征 = 每次查询10^5次特征读取。将其批量化为对内存特征存储的一次 multi-get，耗时约1-5 ms（而非"每个候选 < 1 µs"的计算量——那种算法混淆了墙上时钟时间与单项摊销成本）。准确的说法是：特征存储必须支持批量 multi-get 且 p99 在个位数毫秒内，这正是协同部署的内存键值存储（Redis、内存版 RocksDB、自定义 mmap）所能提供的。

6. 变体及如何选择

6.1 关键词检索 vs. 向量检索 vs. 混合检索

维度 BM25 / 关键词倒排索引 基于习得向量的 ANN 混合
优势 精确匹配、结构化过滤、可解释性强。 语义匹配、"查找相似"、通过共享向量空间实现多语言支持。 当查询组合中两类都存在时，两者兼具。
劣势 遗漏同义改写、同义词、语义意图。 遗漏精确短语、结构化过滤，召回率 < 100%。 两套索引流水线、两套刷新机制、融合评分复杂度高。
Web 查询上的 Recall@1000 头部查询召回率高，意图密集型长尾查询召回率低。 头部查询低于 BM25，长尾查询明显更好。 最高，但相对于更好的单一模式，提升通常有限（Recall@1000 提升5-15%）且不是免费的。
构建成本 与 token 数量线性相关。 与文档数量线性相关 + 向量模型成本（通常比分词的每文档成本高10-100倍）。 两者均有。
刷新成本 低（段合并）。 高（模型变更时向量刷新需重建整个索引）。 两者均有。
使用场景 精确匹配为主（SKU、代码、日志、用户名、法律文本）；需要可解释性/审计。 意图为主（推荐、"更多类似"、多模态）；查询组合短小/模糊。 查询组合确实混合多类，且团队有能力维护两套流水线。

混合检索是 Web 搜索 的默认选择，因为 Web 查询确实是混合的。但它并 非 非通用默认：电商 SKU 查找、日志搜索、代码搜索，或以结构化过滤为主的产品（Yelp 按菜系+现在营业+评分过滤），使用纯关键词检索在延迟、刷新成本和可预测性上均可超越混合系统。只有当你能明确指出 BM25 明显遗漏某类查询时，才选择混合检索。

6.2 文档分片 vs. 词项分片

维度 文档分片（默认） 词项分片
键到分片的映射方式 shard = hash(doc_id) % N；每个分片包含所有词项，仅持有文档的一个切片。 shard = hash(term)；一个分片拥有某个词项的所有倒排列表。
查询扇出 所有分片（高扇出）。 仅拥有查询词项的分片（低扇出）。
每分片工作量 有界、可预测。 热门词项的工作量无界——"the"会导致热点分片。
尾部延迟 受最慢分片主导；需要激进的超时策略。 受最热词项主导。
索引 每个文档只在其单一分片上索引——简单、并行。 每个文档需写入与其不同词项数相同数量的分片。
业界使用 Lucene、Elasticsearch、Solr、Vespa——行业标准。 多为历史遗留；部分早期 Web 搜索系统使用。

对于 Staff 面试： 默认使用文档分片 ，仅将词项分片作为你考虑过并放弃的替代方案提及。

6.3 离线排序并存储 vs. 在线排序

维度 离线（按用户/查询预计算） 在线（在请求时排序）
Latency（延迟） 单次键查找；低于毫秒级延迟。 第一阶段 + 第二阶段 + 数据填充；50-200 ms。
新鲜度 新鲜度受限于上次批处理（数小时到一天）。 精确到秒级。
个性化 仅限于预先计算了键的内容。 任意的每请求信号（位置、时间、A/B 分桶）。
成本 大批量任务；读取成本低。 大型在线集群；批处理规模较小。
使用场景 语料库稳定、用户可预测（Netflix 首页、Facebook PYMK）。 语料库易变（Twitter 搜索、新闻、广告）。
混合技巧 离线预计算候选集；在线排序。 相同。

混合方式（离线预计算候选集，在线用最新特征排序）是推荐信息流的行业默认方案。Twitter/新闻搜索倾向于在线排序，因为新鲜度更重要。

7. 故障模式与运维

失败 需要告警的指标 缓解措施
分片超时/丢弃 shard_timeout_rate 每分片的 p99 查询延迟。 每分片超时；部分结果；副本故障转移；当查询负载激增时自动卸载索引负载。
一个分片过热（doc_id 哈希不均） 每分片 QPS 方差。 重新分片；随机化 doc_id（Twitter 的 snowflake ID 适合这一点）；上游负载均衡器。
索引损坏（错误合并、磁盘位翻转） 校验和不匹配，Recall@K 相比金丝雀索引出现回归。 始终保留 N-1 代先前的段；可以回滚。对每个段进行校验和验证。
分片数据过期（副本错过更新） 复制延迟，分片版本漂移。 从权威日志（Kafka offset）拉取，而非从兄弟分片同步。问题形态与 大规模持久性.
新鲜度延迟（实时层落后） now - max(indexed_timestamp) ，按分片统计。 对索引器施加背压；扩容消费者；当延迟 > N 秒时发出告警。
排序模型漂移（训练/服务偏差、分布偏移） 线上 vs. 线下 nDCG 差值；按桶的 CTR；校准情况。 每日重训；上线前进行 A/B + 影子评测；若线上指标下降 X% 则自动回滚。与 idempotency-exactly-once 点击日志记录
特征存储线上/线下偏差 每个特征的分布差异。 离线使用单一数据源；复制到线上；在训练时发出特征快照；在服务时进行校验和验证。
热门查询风暴（"突发事件爆发时"） 缓存未命中率飙升，后端 QPS 激增。 对头部结果设置缓存并配置负向 TTL；请求合并；准入控制。关联 缓存踩踏/读放大.
可见性泄露（已删除推文仍可被搜索到） 清除到位的时间 SLA、抽样审计。 带严格顺序的删除 topic（顺序-因果一致性）；在查询时进行可见性过滤作为纵深防御。
恶意/垃圾查询 对每次查询成本的异常检测。 按 IP 限流；查询成本限制（背压与流量控制）；查询语法白名单。
向量模型滚动更新 相对于旧索引的召回率差值；双写窗口长度。 始终保持双索引一个窗口；在 Recall@K 达到同等水平后再切换读取方。
索引构建失败 流水线 SLO、新鲜度告警。 保留上一代索引；对"当前"索引的年龄发出告警。
ANN 召回率回归 在金丝雀集合上的离线 Recall@K；每段 HNSW 的 M / efConstruction 参数漂移。 按版本锁定 ANN 构建参数；每次模型版本更新时重新评估。
缓慢/失控的查询 每次查询延迟，倒排列表长度上限。 每次查询设置时间预算；超出则放弃；返回部分结果。

7.1 运维黄金信号

  • p50 。 p99 端到端延迟（按地区、按查询类别）。
  • 按分片 p99；分片超时率。
  • 在保留评判集上的 Recall@K 和 nDCG（离线，每日）。
  • 按桶的 CTR / 停留时长 / 放弃率（在线，每小时）。
  • 索引延迟（实时层）。
  • 索引年龄/陈旧度（主索引层）。
  • 缓存命中率及按查询类别的命中率。
  • 特征新鲜度（请求中任何特征的最大年龄）。
  • 排序器预测分布（偏斜/漂移检测）。

8. 与其他模式的组合

  • 扇出写放大 ——搜索是新闻流扇出的 读侧 镜像。扇出模式将更新推送到多个时间线；搜索模式从多个分片拉取候选集。跨多个并行调用的延迟预算分配是同一类问题。
  • 热点键偏斜 ——头部查询、突发新闻、词项"the"：都是查询层偏斜的体现。缓解措施（缓存、分片随机化、请求合并）高度重叠。
  • 缓存击穿/读放大 ——头部结果缓存、特征存储缓存；两者都必须防御缓存未命中时的惊群效应。
  • 背压/流量控制 ——当查询量激增时，必须在索引落后之前丢弃高成本查询。每次查询的成本计量至关重要。
  • 地理空间查询 ——Yelp/Uber 搜索在相关性排序之上增加了地理谓词。两阶段模式保持不变；候选集生成器从倒排索引换成了 geohash / S2 / 四叉树。
  • 实时聚合 ——用于排序的参与度特征（点赞、浏览、CTR）本身就是一个实时聚合问题。这些特征的新鲜度往往是排序器质量的关键约束。
  • 幂等性/精确一次语义 ——点击日志摄取必须去重，以防排序器训练数据被污染。同一 (query, doc, user) 的两次曝光必须被干净地合并。
  • 排序 / 因果一致性 ——删除事件必须与原始发布事件保持有序处理；否则已删除的文档会重新出现。
  • 大规模持久性 ——索引是派生状态；权威日志是输入流。你始终可以从日志重建索引；但无法从索引重建日志。
  • 协调 / 精确一次执行 ——索引构建任务（段合并、ANN 重建）需要 leader 选举，以避免重复消耗计算预算。

9. 面试官可能追问的问题

  1. "你的扇出到1000个分片，p99 预算为100 ms。请详细说明尾延迟的数学推导。你的分片级 p99 目标是多少，如何保持在该目标内？" ——重点关注每分片超时、Hedged request、按延迟选择副本，以及过度获取（over-fetching）。
  2. "如果一个分片下线会发生什么？" ——副本故障转移、部分结果、优雅降级、查询重试策略，以及为什么对搜索来说返回尽力而为的结果是正确决策，而对事务来说则不然。
  3. "你的排序器刚上线了一个新模型，线上 CTR 在一夜之间下降了3%。你如何排查？" ——A/B 与影子实验对比、特征漂移、训练/服务偏差、校准、自动回滚、holdout 集的作用。
  4. "如何处理推文被删除的情况？是在索引时还是查询时？" ——两者都要。索引时最终清除以控制成本；查询时过滤以保证正确性。删除 topic 和审计机制。
  5. "你如何在文档分片和词项分片之间做选择？" ——在真实查询组合下的负载分布；一个慢词项 vs. 一个慢分片；Lucene 在20年前是如何解决这个问题的。
  6. "跟我讲讲你的向量刷新流程。模型重新训练后，现有的 ANN 索引会怎样？" ——双索引窗口、召回率一致性门控、滚动切换；以及为何不能直接热替换。
  7. "带我走一遍离线 → 线上特征流水线。Bug 藏在哪里？" ——训练/服务偏差、定义漂移、日志并等待验证、校验和、回放。
  8. "你会如何 A/B 测试一个新的排序器？" ——交错评估 vs. 并排对比；最小可检测效应；护栏指标（延迟、错误率、公平性）；灰度策略。
  9. "如果 Kafka topic 损坏，你的索引流水线的恢复方案是什么？" ——日志是数据来源；能否从上游系统（推文存储）重建？批量回放。
  10. "你如何处理多语言查询？" ——特定语言的分析器、多语言向量、回退策略、查询语言检测。

10. 常见陷阱 → 更优方案

陷阱 更优做法
"我直接用 Elasticsearch 搞定一切就好了。" 请说明分层方案、分片数量、每分片 p99 目标、新鲜度流水线，以及第二阶段排序器。ES 只是一个组件，而非完整设计。
使用一个巨大的统一索引。 分层：实时层（内存，秒级新鲜度）+ 主索引层（磁盘，批量构建）。可能还有按文档类型划分的子索引。
默认使用词项分片。 使用文档分片；词项分片仅作为你考虑过并放弃的替代方案提及。
本能地选择混合检索。 明确说明你预期的查询类别（精确匹配、意图、结构化过滤、"更多类似"）。如果只有一类重要，就只用一种检索模式。如果存在多类且 BM25 明显遗漏其中一类，则添加 ANN 并取并集；让第二阶段排序器来仲裁。
使用 ORDER BY created_at DESC. 排序。使用 Learn-to-rank（LambdaMART、DNN）；明确说明特征和训练信号（点击+停留时长，而不仅仅是点击）。
在查询时从原始数据计算特征。 使用在线特征存储；预计算聚合值；使用缓存；在训练时发出特征快照以保持线下/线上一致性。
在一个分片故障时让整个查询失败。 每分片超时 + 部分结果 + 降级模式提示。记录召回率损失。
对头部查询使用无限 TTL 的缓存。 短 TTL（10-60 秒）；如果是个性化查询，缓存键包含用户分群；负向 TTL 吸收长尾缓存未命中；惊群时进行请求合并。
通过原地覆盖来索引删除操作。 在索引中软删除 + 查询时可见性过滤 + 后台清除。在大规模场景下，硬删除太慢。
"我们之后再做个性化。" 从第一天起就在第二阶段预留每用户特征空间和候选注入槽位；事后改造个性化代价高昂。
将排序器视为静态资产。 排序器是一个拥有独立生命周期的服务：训练、影子测试、A/B 测试、灰度发布、监控、回滚。需要指定一个负责人。
忽视离线指标。 在评测集上计算 nDCG / recall@K / MRR；每天执行；并设置回归告警。仅凭线上 CTR 会导致反馈循环。
忽视可见性/安全过滤器。 在排序后于查询时应用；将其视为硬性要求而非标志位。开销很小，但泄漏会毁掉声誉。

11. Staff 级别回答清单

  • [ ] 首先对问题规模进行估算：语料库大小、QPS、相关性质量要求。明确指出在约 1000 万文档以下/低 QPS/无语义查询的场景中，使用 BM25 的单节点 Elasticsearch 是正确答案——只有当延迟或质量的实际计算确实需要时，才升级为两阶段架构。
  • [ ] 明确说明了 召回与排序 的分工，并在前两分钟内大声说出。
  • [ ] 选择了 两阶段检索 （廉价的候选生成 + 昂贵的排序），并用延迟计算进行了论证——具体来说，对全语料库执行（候选数量 × 单候选排序成本）将超出预算。
  • [ ] 明确说明了 倒排索引 （Lucene/Elasticsearch）用于关键词召回，若查询集合包含意图/语义/"查找相似"类别——还使用 ANN （HNSW/IVF/ScaNN）进行语义召回，召回结果取并集，并以某类关键词无法覆盖的具名查询类型作为依据。若语料库仅支持精确匹配，则明确说明并跳过 ANN。
  • [ ] 选择了 按文档分片 的 scatter-gather 方式；指定了每分片超时时间和部分结果策略。
  • [ ] 指定了 实时层 + 主层 架构，明确了新鲜度目标（秒级 vs. 每晚批量），并说明了查询时的合并策略。
  • [ ] 指定了 学习型排序器 （LambdaMART 或 DNN）、特征来源，以及特征存储位置（在线特征存储）。
  • [ ] 讨论了 BM25 作为基准 ，学习型排序器必须超越该基准，以及查询改写/扩展策略。
  • [ ] 提到了 缓存 对热门查询的应用，包括 TTL 和防缓存击穿机制。
  • [ ] 讨论了 故障模式：分片超时、模型漂移、新鲜度滞后、删除顺序、训练/推理偏差。
  • [ ] 讨论了 操作：nDCG/CTR/延迟仪表盘、A/B 测试与影子测试、排序器变更的灰度发布与回滚。
  • [ ] 画了 容量估算草图：语料库大小、分片数量、每分片 p99 预算、缓存后每分片 QPS。
  • [ ] 关联了 相邻模式：扇出、热键、缓存击穿、实时聚合、地理位置、排序。
  • [ ] 说明了哪些内容 超出范围 （爬虫流水线、广告拍卖内部机制），并指明了将其纳入的触发条件。
  • [ ] 以 最大的未解风险收尾：通常是排序器漂移或训练/推理偏差，而非原始延迟。
