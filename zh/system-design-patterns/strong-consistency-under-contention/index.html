竞争下的强一致性

2026/05/31 · 系统设计模式
目录
目录

当 N 个参与者争抢同一稀缺资源时，系统必须产生且仅产生一个赢家，并向每个失败者明确返回"不行"——绝不能悄无声息地超额分配。相关术语请参见 Staff System Design Glossary。此模式是核心负载的相关问题文档： design-ticketmaster.md - design-a-flash-sale-for-an-e-commerce-site.md - design-a-stock-exchange.md - design-a-payment-system.md - design-the-stripe-payment-gateway.md - design-a-distributed-lock-manager-like-chubby.md - design-amazon-shopping-cart.md

1. 核心难点

分布式系统默认追求可用性。你将状态复制到多台机器上，使其中任何一台都能处理读请求或接受写请求。这对"发推文"、"查看个人资料"或"点赞计数加一"这类场景效果极佳，因为数据是累加性的，几秒钟的偏差无关紧要。

这个模式恰恰相反。数据是 递减且有界的。Tesla Cybertruck 发布会的第 4 排第 12 列只有一个座位。Yifan 的支票账户中只有 $143.27 。限量版显卡只剩最后一件。同一毫秒内的两个请求不可能同时获胜。ACID 的"隔离性"属性——两个事务应表现为顺序执行——正是每种分布式数据库技巧都试图放松的地方，因为串行化并发写入代价高昂。在争用场景下，你无法放松它，必须为此付出代价。

在深入讨论之前，有一点 staff 级别的重要说明："强一致性"的标准由产品需求决定，而非本能反应。银行余额、股票交易所成交记录和用户名注册绝对不能容忍任何超额分配。演唱会座位或限时抢购商品通常可以——大多数票务和电商平台会悄悄吸收极小的超卖率（不足百分之一），通过退款或升级来处理，因为补偿的业务成本低于实现完美串行化的工程成本。以下模式假设严格场景；当面试官的产品实际上允许补偿时，正确答案是明确指出这一点并选择更轻量级的设计。

不变式简单到残酷：

对于容量为 C的争用资源，成功提交的数量必须等于 min(C, demand)。绝不能 C + 1。绝不能悄无声息地丢失成功提交。每个失败者都必须收到确定性的"不行"，以便他们停止、针对不同键重试或干净地放弃。

在 staff 级别，难点不在于锁本身。难点在于在三重压力下同时实现这一目标：(1) 吞吐量——每秒 10,000 个请求（RPS）打到单个键；(2) 尾延迟——加锁层不能把 50 毫秒（ms）p99 的用户点击变成 5 秒超时；(3) 故障——持有锁的进程可能在结账途中崩溃，锁管理器可能失去 quorum，数据库可能在事务中途发生故障转移。任何一种情况都会将"只用数据库事务"的设计变成超卖、死锁、幽灵预留或完全宕机。

在争用场景下，朴素设计会以五种方式之一失败。 Lost update（丢失更新）：两个请求都读到 inventory=1，都做减法，都写回 inventory=0，都成功；你卖出了两件。 Phantom reservation（幽灵预留）：用户在会话中持有一个座位，会话悄然死亡，座位永久被占用。 Double-spend on retry（重试导致双重扣款）：支付客户端超时后重试；原始请求其实已经提交；用户被收费两次。 Lock holder dies（锁持有者崩溃）：某节点持有锁，垃圾回收（GC）将其暂停 12 秒，此时另一个节点认为锁已被放弃而将其抢走，现在两个节点都"拥有"该资源。 Cascading lock contention（级联锁竞争）：10,000 个买家都在等待同一行锁，你的数据库连接池耗尽，整个站点其他服务也随之宕机。

2. 如何在面试中识别背压问题

触发该模式的提示语

  • "库存有限" / "最后一件" / "已售罄" / "库存数量"。
  • "两个用户不能预订同一个 X。"
  • "唯一赢家" / "不超卖" / "不双重扣款"。
  • "演唱会门票" / "限时抢购" / "限量发售" / "抽签"。
  • "买卖双方撮合" / "订单簿" / "成交"。
  • "余额" / "转账" / "扣款" / "授权"。
  • "预留并确认" / "保留 N 分钟" / "结账流程"。
  • 高并发下的"先到先得"。

需求描述中的架构异味

  • 一个标量字段（stock_count, balance, seat_status），其操作是 递减 而非追加。
  • 语义上是业务规则而非仅数据库提示的唯一性约束（UNIQUE(seat_id) 于 bookings).
  • 有两个阶段的工作流（预留 → 确认，授权 → 扣款），两个阶段之间可能有几秒到几分钟的用户思考时间。
  • 库存数量的读取出现在高流量落地页上，而写入需要精确无误。（这就是读写分离——参见 cache-stampede-read-amplification.md.)
  • 单一热键——一场演唱会、一只股票代码、一次明星商品补货——集中了大量争用。与 hot-key-skew.md.
  • 幂等键要求。与 idempotency-exactly-once.md.

以此为核心难点的"设计 X"示例题

  1. Ticketmaster / Eventbrite 座位选择 ——5 万座的体育场，20 万名粉丝在前 30 秒内点击"选座"。
  2. 限量商品限时抢购 ——1,000 件显卡在中午开售，10 万人在页面上等待。
  3. 股票交易所 / 撮合引擎 ——每一笔订单与订单簿的匹配都必须按确定性的价格-时间优先级提交。
  4. 支付系统 / 钱包余额扣款 ——即使两台 ATM 和一个刷卡终端同时操作同一账户，银行余额也绝不能变为负数。
  5. Uber 司机调度 ——一名司机同一时间只能接受一个乘客；两名乘客可以分别分配给两名不同的司机，但绝不能都分配给同一名司机。
  6. 酒店 / Airbnb 按特定日期段预订 ——同一房源的重叠日期段不得同时预订成功。
  7. 用户名 / 昵称注册 ——第一个声明 @elonmusk2 的人获胜，其他人收到"已被占用"。

3. 60秒口头框架

这里的不变式是精确的。对于容量为 C的争用资源，系统必须产生恰好 min(C, demand) 次成功提交，每个失败者必须收到明确的"不行"，同时不能在争用键上阻塞整个站点。我的默认方法是将一个键的所有写入都路由到单个串行化点——要么是单写入分区，要么是数据库行锁——并用短 TTL 预留令牌保护用户体验，使座位在结账期间被保留，但在用户放弃时自动释放。在此基础上，我对低争用商品使用乐观并发控制，将数据库级唯一性约束作为最终安全网（即使应用层有 bug 也无法超卖），并在每次写入上设置幂等键以保证重试安全。我需要预防的两种故障模式是：锁持有者崩溃（使用 fencing token，永远不用裸锁）和热键崩溃（限制每键并发量，提前丢弃负载）。

4. 标准工具集

4.1 单写入分区（大规模场景的默认方案）

含义：对每个争用键 K，将所有写入路由到同一节点。该节点通过内存队列串行化写入。没有锁，没有 compare-and-swap（CAS）重试风暴——只有一个队列。这就是 Kafka 实现每分区有序性的方式，也是订单撮合引擎处理单个股票代码、LinkedIn Espresso 实现每键线性一致性的方式。

        ┌───────────────────────────────────────────┐
        │ Router (hash by seat_id or symbol)        │
        └────┬────────────┬────────────┬────────────┘
             │            │            │
       seat_id=A     seat_id=B    seat_id=C
             │            │            │
        ┌────▼───┐   ┌────▼───┐   ┌────▼───┐
        │ Node 1 │   │ Node 2 │   │ Node 3 │
        │ queue  │   │ queue  │   │ queue  │
        │  for A │   │  for B │   │  for C │
        └────────┘   └────────┘   └────────┘
        Single-threaded apply per key → no lock needed

适用场景：在有限键集合上存在高争用，且每键吞吐量适合单核处理。实际单核上限：Python 约 5-20k ops/sec（受 GIL 限制），Java 或 Go 使用紧凑内存状态机约 50-200k ops/sec，C++ 或 Rust 可达 500k+。关键数字是"每键 apply 操作的 CPU 开销是多少？"——如果减量加日志追加约需 5 微秒，单核约在 200k ops/sec 时饱和，大部分预算消耗在复制日志追加上。适用于撮合引擎、票务座位分配、广告拍卖底价。

成本：你失去了跨机器对热键进行负载均衡的能力。如果某个键比集群平均水平热 100 倍，该节点将会过载。你还需要一个路由层，将 K 一致地映射到同一节点——通常使用带版本分区映射的一致性哈希。故障转移意味着要排空在途队列并在备用节点上重放；你需要对每分区状态进行复制（通常是 Raft、Paxos 或 Kafka 风格的日志）。

故障场景：某个键变得极热，以至于单节点无法跟上。此时必须对键本身进行分片，这通常意味着改变产品设计（将座位分配到子块，将股票代码按价格区间拆分）。或者节点崩溃，需要在不丢失队列的情况下进行故障转移——你需要对内存队列输入进行同步复制。

# Sketch: single-writer partition, applied per key.
# In production this is the same idea as a Raft state machine.

import asyncio
from collections import defaultdict

class SeatPartition:
    def __init__(self, seat_ids):
        # source of truth, in memory, replicated via log
        self.seats = {s: "FREE" for s in seat_ids}
        self.queues = defaultdict(asyncio.Queue)

    async def reserve(self, seat_id, user_id, request_id):
        # Caller awaits this; per-seat queue serializes the apply step.
        future = asyncio.get_event_loop().create_future()
        await self.queues[seat_id].put((user_id, request_id, future))
        return await future

    async def run_partition(self, seat_id):
        # One coroutine per seat. No lock — there is exactly one writer.
        q = self.queues[seat_id]
        while True:
            user_id, request_id, future = await q.get()
            if self.seats[seat_id] == "FREE":
                self.seats[seat_id] = ("HELD", user_id, request_id)
                # Append to replicated log BEFORE acking the future.
                await replicate(("RESERVE", seat_id, user_id, request_id))
                future.set_result("ok")
            else:
                future.set_result("taken")

含义4.4 带 fencing token 的分布式锁（Chubby / ZooKeeper / etcd） ：一个独立的协调服务授予锁。锁持有者获得一个单调递增的fencing token

   ↓
client A is GC-paused for 12 seconds
   ↓
lock expires → client B acquires lock → token = 34
   ↓
client B writes to storage with token=34 → storage records "last=34"
   ↓
client A wakes up, tries to write with token=33 → storage rejects (33 < 34)

# Pseudocode, deliberately verbose to show the fencing step.
token = lock_service.acquire("seat:A12", ttl_sec=30)  # token = e.g. 33
if token is None:
    return "could_not_acquire"
try:
    # Storage must verify the token is the latest. The token is part of every write.
    response = storage.cas_update(
        key="seat:A12",
        expected_state="FREE",
        new_state=("HELD", user_id),
        fencing_token=token,   # storage compares to its own monotonic counter
    )
finally:
    lock_service.release("seat:A12", token=token)

def reserve(seat_id, user_id, ttl_sec=300):
    now = time.time()
    # Atomic: either we acquire the seat or someone else owns it.
    affected = db.execute(
        """
        UPDATE seats
           SET held_by = %s, held_until = %s, version = version + 1
         WHERE seat_id = %s
           AND (held_by IS NULL OR held_until < %s)
        """,
        user_id, now + ttl_sec, seat_id, now,
    )
    if affected == 1:
        return make_token(seat_id, user_id, now + ttl_sec)
    return None  # someone else holds it and the hold hasn't expired

def confirm(token):
    seat_id, user_id, expires_at = parse_token(token)
    if time.time() > expires_at:
        return "expired"
    affected = db.execute(
        """
        UPDATE seats
           SET booked_by = %s, status = 'BOOKED', held_by = NULL, held_until = NULL
         WHERE seat_id = %s AND held_by = %s AND held_until >= %s
        """,
        user_id, seat_id, user_id, time.time(),
    )
    return "ok" if affected == 1 else "lost_hold"

适用场景行。第 2 步："确认"将预留转换为永久预订。后台任务（或读取时的惰性检查）回收已过期的预留。

成本：任何在"我想要这个"和"我确认购买"之间存在用户思考时间的流程。票务购买、酒店预订、出行调度（"司机，你有 15 秒接单"）。这是 Ticketmaster、StubHub、Airbnb 的事实标准。 capacity × (TTL / median_checkout_time)：TTL 是一个伪装成数字的调优问题，而非"选一个合理值"的问题。太短会在用户还在输入信用卡信息时回收座位；太长则在限时抢购期间将有效库存上限限制为 。 ，压制了真实需求。你还需要一个清扫器加读取时的惰性回收，因为任何一方单独使用都有故障模式（清扫器在高负载下落后；惰性回收对没人再次读取的键永远不会触发）。状态空间—— FREE, HELD, BOOKED, EXPIRED ——有一个容易被忽略的不对称性： 预留和释放不是对称的。预留发生在一次原子写入中，但释放可以通过三种不同路径发生（用户显式释放、清扫器过期、下一个读者请求时的惰性回收）。每条路径都必须使用单一时间源（存储的时钟，而非任何客户端的时钟）和令牌的单一事实来源（持有者身份 加上 过期时间，而非仅身份——否则一个缓慢的 confirm 可能与成功的抢占形成竞争）来判断"这个预留现在是否真的已过期？"。

故障场景：经典的僵尸预留竞争：用户 A 的预留在 t=300 时过期；清扫器或惰性回收在 t=301 时将座位重置为 FREE；用户 B 在 t=302 时抢到该座位；用户 A 延迟到达的 confirm 请求在 t=303 携带其旧（现已过期）令牌到达。确认操作必须拒绝——这就是为什么上述代码中的 confirm 查询会重新检查 held_by = user_id AND held_until >= now() 而非单纯信任令牌。第二个故障：同一用户对自己的重试产生竞争（预留期间网络抖动），最终持有两个座位。以 reserve 为键的 (user_id, event_id, request_id) 幂等键可以解决重试问题，但无法解决合法的贪婪用户在多个会话下预留的情况——对此需要在预留服务内对每用户并发持有数量设置上限。

4.6 跨服务的两阶段预留（意图 → 提交）

含义：当提交需要多个系统共同同意时（例如，同时扣减钱包余额、标记座位为已预订、通知场馆），无法对它们进行原子更新。预留步骤是 意图：写入 Saga 状态行，不扣款，将座位标记为 HELD。提交步骤是 应用：扣减钱包、确认预订、对场馆幂等确认。如果任何步骤失败，执行补偿：退还钱包，释放座位。参见 idempotency-exactly-once.md 。 coordination-exactly-one-execution.md.

适用场景：多服务提交。支付 + 库存 + 通知。避免使用分布式两阶段提交（2PC）——它在协调者故障时会阻塞。改用带显式补偿的 Saga。

成本：可见的中间状态。每个步骤都需要仔细保证幂等性。运维复杂度高。

故障场景：补偿本身也可能失败。你需要一个永久性的对账任务来检测并告警卡住的 Saga。

4.7 数据库级约束作为最后一道防线

含义：在 UNIQUE(seat_id, event_id) 表上添加 bookings 约束。即使所有应用层检查都有 bug，数据库本身也会拒绝插入重复预订。

CREATE TABLE bookings (
  booking_id  BIGINT PRIMARY KEY,
  event_id    BIGINT NOT NULL,
  seat_id     BIGINT NOT NULL,
  user_id     BIGINT NOT NULL,
  created_at  TIMESTAMP NOT NULL,
  UNIQUE (event_id, seat_id)   -- the load-bearing constraint
);

适用场景：始终使用，当语义不变式可以表达为唯一性或检查约束时。廉价、简单、万无一失。开销基本为零。

成本：可以忽略不计。插入操作在索引上是 O(log n) 。唯一的缺点是约束违反以异常形式出现，因此应用程序必须区分"重复"（返回"已被占用"）和"真实错误"（重试或告警）。

故障场景：无法表达更复杂的不变式（例如，"日期范围不重叠"）。对于这类情况，使用排除约束（Postgres EXCLUDE USING GIST）或自定义串行化层。

5. 实际案例：朴素设计 → 崩溃 → 修复

场景：Tesla Cybertruck 预订活动。Foundation 系列共有 10,000 件。订购页面于太平洋时间上午 9:00 开放。预计在前 60 秒内有 500,000 名唯一用户访问页面。每位用户几乎同时点击"立即预订"。产品不变式： 恰好 10,000 次成功预订，不超卖，每位失败者在 5 秒内看到"已售罄"。

朴素版本（有缺陷）

# Every web request runs this. No locking.
def reserve():
    row = db.fetch_one("SELECT available FROM inventory WHERE sku='cybertruck-fs'")
    if row.available > 0:
        # ... user enters payment info, picks color, takes 30 seconds ...
        db.execute("UPDATE inventory SET available = available - 1 WHERE sku='cybertruck-fs'")
        db.execute("INSERT INTO orders (user_id, sku) VALUES (...)")
        return "confirmed"
    return "sold_out"

按严重程度排列的问题：

  1. 大规模超卖关系。其中 UPDATE 没有 WHERE available > 0 保护，且 SELECT-then-UPDATE 窗口横跨 30 秒的用户输入。每个通过 SELECT 的请求都会继续执行减量操作，无论该行此时已变成什么。即使只有约 10,000 个请求同时通过 SELECT（在同一瞬间读取），每个请求都会无条件减量；计数器会严重变负。单独对这一行的修复方法是 UPDATE ... WHERE available > 0 并依赖 affected_rows == 1，单条原子语句可以保证这一点。真正被说明的 bug 是"应用层读取，然后无条件写入"——而第 4 节中的每一种技术正是为了防止这种情况而设计的。
  2. 数据库崩溃。Web 层可能只有 50-200 个 DB 连接；数据库接受那么多并发事务，其余的在连接池层排队。在事务内持有 30 秒用户思考时间，吞吐量崩溃为 pool_size / hold_time ≈ 每秒只有几次提交，而队列无限增长。共享同一数据库的 Tesla 其他站点服务也在排队。不到一分钟，整个连接池就被卡住的结账会话占满。
  3. 从缓存读取。营销页面上的"可用"数量是缓存的。"售罄"后，缓存仍显示"92件可用"长达15分钟，引发大量用户投诉。
  4. 结账卡死。部分用户通过了 SELECT 查询后，网络中断，始终未完成结账。其库存槽位在业务层面已被占用，但订单行数据却从未写入。库存与订单意图之间出现了不匹配。

修复版本（生产级）

分层防御：

# Inventory service - single writer for sku.

class InventoryPartition:
    def __init__(self, sku, capacity):
        self.sku = sku
        self.available = capacity
        self.held = {}  # token -> (user_id, expires_at)

    def reserve(self, user_id, idempotency_key):
        # Single-threaded apply. No lock needed because there's one writer.

        # Idempotency: if same user retries, return their existing hold.
        for token, (uid, exp) in self.held.items():
            if uid == user_id and self._matches_idem(token, idempotency_key):
                return ("ok", token)

        if self.available <= 0:
            return ("sold_out", None)

        token = generate_token(self.sku, user_id, idempotency_key)
        self.available -= 1
        self.held[token] = (user_id, time.time() + 600)
        replicate_log_append({
            "op": "HELD",
            "sku": self.sku,
            "token": token,
            "user_id": user_id,
            "available_after": self.available,
        })
        return ("ok", token)

    def confirm(self, token, user_id):
        held = self.held.get(token)
        if not held or held[0] != user_id:
            return "invalid"
        if time.time() > held[1]:
            return "expired"  # will be reclaimed by sweeper
        del self.held[token]
        replicate_log_append({"op": "SOLD", "sku": self.sku, "token": token})
        return "ok"

    def release(self, token):
        held = self.held.pop(token, None)
        if held:
            self.available += 1
            replicate_log_append({"op": "RELEASE", "sku": self.sku, "token": token})

    def sweep(self):
        now = time.time()
        for token, (uid, exp) in list(self.held.items()):
            if exp < now:
                self.release(token)

订单最终化以数据库约束作为兜底：

INSERT INTO orders (user_id, sku, hold_token, status)
VALUES (:user_id, :sku, :hold_token, 'PAID');
-- UNIQUE(hold_token) ensures even if the application layer retries, only one row lands.

为何有效：

压力 设计的应对方式
超卖 SKU 的单一写入者是唯一执行递减操作的实体。原子计数器不会降至零以下。
吞吐量 在编译型语言（Java/Go/Rust）中，单核可持续支撑约 50-200k 次"递减并记录"操作每秒；日志追加的往返耗时是主要瓶颈。在60秒内我们总共只需约 1万次递减事件，因此单核的算力超出需求两个数量级。
数据库崩溃 数据库不再处于递减操作的热路径上，它只接收已确认的订单，频率与人工操作相当。
营销页面缓存失实 预订服务发出"sold_out"事件；营销服务消费该事件并更新缓存。延迟以秒计，而非分钟。
结账卡死 预占设置 TTL；清扫程序负责释放；数据库 UNIQUE 约束防止卡死流程恢复后重复提交。
锁持有者宕机 单一写入者节点的状态同步复制到从节点。故障转移后计数器得以恢复。

每位员工答案都应提及的指标

  • oversell_count （必须严格为 0；超过 0 则告警）
  • successful_reservations_per_sku
  • hold_to_confirm_ratio （比率低 = 预占被放弃；调优 TTL 或 UX）
  • cas_retry_distribution （长尾 = 竞争；考虑单一写入者）
  • lock_holder_pause_detected （任何非零值都需要围栏令牌审计）
  • partition_leader_election_count （频繁发生 = 节点不健康）
  • saga_stuck_count 。 saga_compensation_failures

8. 与其他模式的组合

该模式几乎从不单独使用。常见的搭配：

  • hot-key-skew.md ——被竞争的键按定义就是热键。按子键（座位块、价格档位）预分区，使用候场室卸载流量，或应用每键限速。限时抢购示例结合了这两种模式。
  • idempotency-exactly-once.md ——竞争下的每次写操作必须可安全重试。当网络中断时，客户端无法知晓首次尝试是否已提交。幂等键是必须的。
  • cache-stampede-read-amplification.md ——读端通常显示"还剩12件！"，该数字必须大致准确。缓存计数，但实际递减操作使用强一致性层。
  • coordination-exactly-one-execution.md ——当竞争结果必须触发恰好一次下游动作（一封确认邮件、一条账本记录）时。Outbox 模式 + 幂等消费者。
  • durability-at-scale.md ——胜出方的提交必须在节点宕机后仍能存活。同步复制分区日志（Raft）或向持久存储进行仲裁写入。
  • backpressure-flow-control.md ——当 50万用户访问同一 URL 时，必须在早期丢弃或排队。候场室模式。
  • ordering-causal-consistency.md ——股票交易所的委托簿需要严格的逐档位排序（价格-时间优先）。单一写入者分区模式使用相同的机制，附加了额外的排序约束。

9. 面试官可能追问的问题

  1. "当持有单一写入者分区的节点在预占进行中宕机时，会发生什么？" 答：同步日志复制到至少一个从节点（Raft / Multi-Paxos）；由故障检测器触发主节点选举；从节点回放已提交的日志条目；处理中的未提交请求以"unknown"状态返回，客户端携带幂等键重试；有界恢复时间目标（RTO）为秒级，而非分钟级。

  2. "你说你使用了分布式锁。如果锁持有者发生 GC 暂停长达15秒怎么办？" 答：这正是我们使用围栏令牌的原因。锁服务在每次获取时颁发一个单调递增的令牌。每个下游存储操作都包含该令牌；存储层拒绝低于其已见最高值的令牌。暂停的客户端醒来后，尝试以过期令牌写入，被拒绝。这是分布式锁的核心探测点——面试官在考察你是否了解 Martin Kleppmann 的那篇文章。

  3. "如果 50万用户在恰好 9:00:00.000 同时访问页面，你如何处理？" 答：在流量到达预占服务之前卸载。候场室模式：边缘服务器发放队列令牌，缓慢将用户放入结账流程。库存服务只以它能确认的速率接收需求。在准入时使用 CAPTCHA 或工作量证明来抵御机器人。

  4. "座位预占的正确 TTL 是多少？" 答：推导出来，不要凭感觉随意设定。在正常需求下（容量不是瓶颈），TTL 根据用户流程宽松设置——典型结账 p95 为 60-180 秒，因此 5-10 分钟的 TTL 为慢速填写卡号留有余量。在限时抢购条件下，数学逻辑反转：有效可售库存等于 capacity × (TTL / median_checkout_time)，因此10分钟 TTL 配60秒结账实际上将库存以 1/10 的容量串行化，压制了需求。将 TTL 压缩至约 60-120 秒，显示可见倒计时让用户了解竞争进度，并在活动后测量两个指标： hold_to_confirm_ratio （低意味着 TTL 过短或 UX 过慢）和 held_seat_recycle_count （高意味着对该需求水平而言 TTL 过长）。两者都要调优。

  5. "如何防止用户通过刷接口持有100个座位？" 答：在边缘进行每用户限速。在预占服务内部强制执行每用户并发预占上限（max_holds_per_user）。对 reserve 以 (user_id, request_id) 实施幂等，使重试不会倍增预占数量。

  6. "请带我梳理用户点击确认时的支付流程。如果网络在通话中途中断怎么办？" 答：幂等键在预占时由客户端生成，包含在每次确认尝试中。支付服务商通过幂等键去重。如果客户端超时后重试，第二次调用返回第一次的结果。如果支付成功但我们的订单最终化失败，后台对账作业会捕获孤立支付，并要么完成订单，要么退款。

  7. "你如何审计确保从未超卖？" 答：每次状态转换（FREE → HELD → SOLD）都记录到仅追加的事件日志中。每晚对账作业重放日志，重新计算每个 SKU 的数量，并与真实来源行进行比较。出现不匹配时告警。另外，每日报告 count(orders WHERE sku=X) ≤ capacity(X) 每个 SKU 的数据；任何违规都是 Sev-1 级别。

  8. "为什么不直接用一个原子 UPDATE inventory SET stock = stock - 1 WHERE sku = X AND stock > 0 然后收工？" 答：对于许多产品而言，那 是 正确答案。Postgres 或 MySQL 中的单个受保护 UPDATE 持有行锁约数十微秒，无需任何额外机制即可轻松处理每行数千 RPS 而不超卖，因为数据库本身会串行化写入。本文档中的技术栈只有在以下四个约束之一成立时才值得其复杂性：(a) 每键吞吐量超过单一数据库行所能串行化的能力（单一写入者分区）；(b) 工作流在意图与提交之间存在用户等待时间（预占 TTL）；(c) 提交跨越无法共享事务的多个服务（Saga）；(d) 资源不归属于任何单一数据库，外部锁管理器是唯一选项（分布式锁 + 围栏令牌）。如果这些约束均不成立，员工级别的操作是指出单行 UPDATE 已足够，并仅添加数据库 UNIQUE 约束作为双重保险。在一个每分钟处理10个订单的 B2B 库存系统上引入 Chubby，是一个危险信号。

10. 常见陷阱 → 更优方案

陷阱 更优做法
"只需使用 SELECT FOR UPDATE." 分层设置：数据库 UNIQUE 约束作为兜底，带 TTL 的预占令牌用于 UX，单一写入者分区用于扩展。说明每个层级分别吸收哪种故障。
无围栏令牌的分布式锁。 始终将锁与在存储层强制执行的单调令牌配对；否则暂停的持有者会破坏状态。
将重试视为无害操作。 对每次写入使用幂等键；对 (user, idem_key)添加 UNIQUE 约束；重试必须收敛到相同结果。
隐藏被竞争的键。 明确说出来。说："热键是 sku=cybertruck-fs。我的设计将其路由到专用的单一写入者节点。"
在任何规模下使用乐观 CAS。 低竞争下可接受；高竞争下重试会爆炸。对已知热键切换至单一写入者。
忘记幽灵预占。 始终将预占与 TTL + 清扫程序 + 下一次读取时懒回收配对。说明 TTL 数值和回收延迟目标。
清扫程序是唯一的回收机制。 在读取时添加懒回收，这样即使清扫程序落后，下一个调用者也能声明已过期的预占。
营销页面上陈旧的"有货"数量。 以竞争层作为真实来源；让营销页面成为以秒级延迟追赶的派生投影。说明延迟的 SLA。
跨服务的两阶段提交（2PC）。 2PC 在协调者故障时会阻塞。使用带有显式补偿和对账的 Saga。
对每用户预占数量没有上限。 每用户并发预占上限。在边缘进行每用户限速。
假设各服务之间的时钟一致。 使用存储自身的时钟进行 TTL 过期；绝不信任客户端时钟。或依赖单调围栏令牌，而非墙上时间。

11. Staff 级别回答清单

当你给出"竞争下的强一致性"答案时，面试官应当在你的口头设计中听到所有这些要素：

  • [ ] 你精确命名了不变量： commits == min(C, demand)，不超卖，每个落败者在规定延迟内收到"否"。
  • [ ] 你按名称标识了热键（不是"那一行"，而是" event_id=42, seat_id=A12").
  • [ ] 你选择了一种主要串行化机制（单一写入者、锁、CAS）并说明了原因。
  • [ ] 你将数据库 UNIQUE 约束作为最后一道防线分层叠加。
  • [ ] 你对任何存在用户等待时间的流程使用了带 TTL 的预占令牌。
  • [ ] 你按键命名了每次易重试写操作的幂等性。
  • [ ] 你讨论了锁持有者暂停和围栏令牌（如果设计中包含锁）。
  • [ ] 你描述了串行化节点的故障转移方案（复制、主节点选举、RTO）。
  • [ ] 你描述了读路径如何保持高速（缓存、投影）而不成为真实来源。
  • [ ] 你命名了一到两个指标，让值班工程师能够检测到超卖、卡死的预占或竞争风暴。
  • [ ] 你讨论了在边缘进行流量卸载（候场室、CAPTCHA、限速），使竞争层永远不会不堪重负。
  • [ ] 你区分了合法的 Saga 状态，并说明了卡死的 Saga 是如何被检测和解决的。
