热 Key / 数据倾斜

2026/05/31 · 系统设计模式
目录
目录

少数几个键承接了流量的大部分，导致在平均意义上数学均衡的分区方案融化了单个分片，而集群中其他节点却处于空闲状态。术语参见词汇表中的一致性哈希、Preference List、Virtual Node (vnode) 与 Quorum。核心题目文档： Ticketmaster, Flash sale, Twitter（名人扇出）, YouTube 点赞计数器, 广告点击聚合器, 输入提示自动补全, 类 Redis 分布式缓存.

1. 核心难点

大多数分区方案——一致性哈希、范围分区、hash-mod-N——都假设 key 的分布近似均匀。在该假设下，每个分片的负载收敛于 total_QPS / num_shards，容量规划只是一个乘法问题。大数定律 (LLN) 使一切在使用合成数据的压测中看起来都很平滑。

真实工作负载遵循 Zipf 分布。Web 流量、搜索查询、社交图扇出、电商 SKU 页面、票务库存行、广告展示、病毒内容——所有这些都遵循幂律分布，其中分布头部比中位数高出许多个数量级。在这些题目所涉及的规模下，一个可行的经验法则是：前 0.01% 的 key 承接约一半的请求，最热的 key 通常是中位数的 1,000 至 100,000 倍（病毒式流量峰值的极端尾部——Beyoncé、超级碗时刻——会更高，但请将 10⁶× 视为偶发事件而非规划常量）。在该分布下，一个分片会看到 0.5 × total_QPS ，而其余 N-1 个分片平分剩余的一半流量。将集群规模翻倍也无济于事：热 key 仍然只落在一个分片上。

被打破的不变量： 每个分片的负载必须低于每个分片的容量。 一致性哈希保证了 key 到分片的稳定映射 ，但对 负载倾斜下的负载均衡无能为力。Virtual nodes (vnodes) 能平滑中等频率的 key，但对单个超热 key 毫无作用——拥有该 key 的每个 vnode 都会承受其全部流量。结果是一个分片 CPU 打满、队列堆积、p99 延迟攀升，并且调用方的连接池耗尽。由于故障转移会将 key 迁移到其副本，副本随即也会过热——形成级联热分片故障。

热 key 还具有 突发性和时变性。 Beyoncé 在上午 10:00 宣布演出，昨天还是冷数据的库存行，在 10:00:01 就成为整个系统的瓶颈。缓解措施不能是静态配置，必须是由检测驱动、在数秒内做出响应的动态机制。

2. 如何在面试中识别背压问题

提示语中的信号短语

  • "名人"、"病毒式"、"热门趋势"、"黑色星期五"、"Flash sale"、"超级碗"、"Beyoncé 座位"。
  • "最热门的 X"、"排行榜"、"top-K"、"全局热榜"。
  • "某些用户有数百万粉丝"、"某些帖子获得十亿次浏览"。
  • "一个商品页面在发布期间占据了 90% 的流量"。
  • "一张票/一个 SKU 的库存"。
  • "直播聊天"场景，单个直播间有 1000 万并发观看者。
  • 任何提到 Zipf、幂律、长尾或 80/20 的场景。

需求描述中的架构异味

  • 单行、单 key 或单计数器是被争用的资源（例如， inventory[sku=PS5], likes[video_id=v1]).
  • 读 QPS 或写 QPS 引用到数百万次，但"人们实际查看的"工作集很小。
  • 在高 QPS 之上叠加了每 key 的强一致性要求（Ticketmaster、针对同一账户的支付）。
  • 一个所有人都访问的 key 上有严格的排序或计数语义。
  • 分片策略是"按 user_id 哈希"，但读取模式是"按 content_id"。

以此为核心难点的"设计 X"示例题

  1. 设计 Ticketmaster ——一个座位行，一千万人刷新；经典的写热 + 强一致性叠加场景。
  2. 设计 flash sale ——一个 SKU，禁止超卖，30 秒窗口内 1000 万购物者。
  3. 设计 Twitter ——拥有 1 亿粉丝的名人；朴素的写时扇出会崩溃。
  4. 设计 YouTube 点赞计数器 ——一个病毒视频每秒积累 100 万次点赞；单行递增风暴。
  5. 设计广告点击聚合器 ——一个广告活动承接了大部分点击；每个活动的计数器成为热点。
  6. 设计 typeahead ——"n"、"a"、"the" 等前缀的读取次数比罕见前缀高出数百万倍。
  7. 设计 Twitch 聊天/直播评论 ——单个直播间承载了整个系统的负载。

3. 60秒口头框架

我首先要指出的是，这个工作负载遵循 Zipf 分布。少数 key 会承接大部分流量，仅靠一致性哈希是救不了的——一个分片会过热，其他分片却空闲。因此我的设计分为两个部分。第一，冷路径：标准一致性哈希加 virtual nodes，外加应用层针对长尾的每 key TTL 缓存。第二，热路径：在边缘用滑动窗口计数器或 Count-Min Sketch 检测热 key，然后根据情况应用以下一种或多种方案：带抖动 TTL 的本地一级缓存 (L1) 以合并读请求、请求合并 (request coalescing) 使并发 cache miss 共用一次源站拉取、写侧子 key 拆分使热计数器变成 N 个子计数器在读取时汇总，以及将热 key 值复制到每个分片。具体选择取决于热 key 是读密集还是写密集，以及能否容忍读取过期数据。我会保留一个专用的热 key 层作为运维逃生舱口，与主存储分离，这样单个坏 key 就不会拖垮普通流量。

4. 标准工具集

七种技术。根据读写倾斜程度、一致性预算和检测延迟选择其中一部分。

4.1 带 TTL 抖动的本地 L1 缓存（读密集型热 key）

定义。 每台应用服务器在分布式缓存或存储前面维护一个进程内 LRU 缓存（例如 Caffeine、Guava 或 functools.lru_cache）。TTL 加入随机抖动，使各服务器的缓存过期不同步。

何时使用。 适用于可以容忍数秒过期的读密集型热 key（例如商品页渲染、用户资料元数据、top-K 列表）。

代价。 每台应用服务器都消耗内存。过期时间等于 TTL 加上刷新时间。缓存一致性无法保证——跨 N 台服务器的失效是尽力而为的。

失效场景。 当应用要求毫秒级读写一致性时。当数据更新频率高于 TTL 时。当热 key 的工作集过大无法放入 L1 时。

                ┌───────────────┐
client req ─►   │ App server    │   L1 hit ──► return (sub-ms, zero RPC)
                │  ├─ L1 LRU    │
                │  ├─ Coalescer │   miss ──► single fetch to L2
                │  └─ ────────► │
                └───────────────┘

# Per-server L1 cache with jittered TTL and coalescing.
import asyncio
import random
import time

class HotKeyL1Cache:
    def __init__(self, base_ttl_s=5.0, jitter_s=2.0):
        self.cache = {}                      # key -> (value, expires_at)
        self.inflight = {}                   # key -> asyncio.Future
        self.base_ttl_s = base_ttl_s
        self.jitter_s = jitter_s

    async def get(self, key, loader):
        now = time.monotonic()
        entry = self.cache.get(key)
        if entry and entry[1] > now:
            return entry[0]                  # L1 hit

        # Coalesce: only one in-flight load per key per process.
        fut = self.inflight.get(key)
        if fut is None:
            fut = asyncio.get_event_loop().create_future()
            self.inflight[key] = fut
            try:
                value = await loader(key)    # one RPC to origin
                ttl = self.base_ttl_s + random.uniform(0, self.jitter_s)
                self.cache[key] = (value, now + ttl)
                fut.set_result(value)
            except Exception as e:
                fut.set_exception(e)
            finally:
                self.inflight.pop(key, None)
        return await fut

4.2 热 key 复制（读密集型、大扇出）

定义。 不再只有一个分片拥有热 key，而是将其复制到 M 个分片（M 在极限情况下可以等于集群大小 N）。读取在副本集间轮询。写入同步或异步扇出到所有 M 个副本。

何时使用。 适用于数据变更不频繁的读密集型热 key（热门榜单、配置行、特性开关）。当 L1 缓存不足时使用—— 因为热 key 工作集过大无法放入每台应用服务器的内存中 ，或者因为需要一个每个分片都拥有该 key 的统一层。注意：复制本身 非 给你提供亚秒级的数据新鲜度——副本存在扇出延迟，因此这不能解决"无法容忍秒级过期"的问题；如有此约束，需结合同步扇出或 quorum 读取。

代价。 写入放大：一次逻辑写入变成 M 次物理写入——不只是存储层面，在 QPS 意义上也是如此。如果一个热 key 有 10K 次写入/秒且 M=64，集群为这一个 key 承载 640K 次写入/秒。每个热 key 的 M 倍存储占用通常可以忽略不计。真正的运维成本在于路由中的双代码路径以及保持副本集一致性。

失效场景。 写密集型热 key——写入放大会使你崩溃。强一致性要求——在每次写入时保持 M 个副本同步本质上是一个 M=N 的写 quorum 问题。

# Routing layer with hot-key replica fan-out.
class HotKeyRouter:
    def __init__(self, shards, hot_keys_to_replicas):
        self.shards = shards                          # list of shard ids
        self.hot = hot_keys_to_replicas               # key -> list[shard_id]

    def shards_for_read(self, key):
        replicas = self.hot.get(key)
        if replicas:
            # Random pick spreads load evenly.
            return [random.choice(replicas)]
        return [self._hash_to_shard(key)]

    def shards_for_write(self, key):
        return self.hot.get(key) or [self._hash_to_shard(key)]

4.3 请求合并/singleflight（读密集型、源站开销大）

定义。 当 N 个并发请求对同一个 key 都 cache miss 时，只有一个请求去访问源站，其他请求等待同一个 future 的结果。以热 key 为键实现为每进程锁。概念上与 Go 的 singleflight 包或 Java 的 CompletableFuture 去重机制相同。

何时使用。 当上游拉取开销高昂（数据库查询、模型推理、下游服务调用）且大量并发 miss 可以合并为一次请求时尤为适用，特别是在 key 过期后的缓存 miss 风暴场景中。

代价。 会增加（N-1）个等待请求的延迟——它们支付的是拉取时间，而非缓存命中时间。单点故障：如果领头请求挂起，所有合并的请求会一起超时。

失效场景。 当上游不稳定时——一次慢速的领头请求会阻塞所有人。缓解措施：为每个请求设置比领头拉取超时更短的超时时间，使调用方能快速失败并独立重试。

4.4 子 key 拆分/计数器分片（写密集型热 key）

定义。 将一个逻辑热 key 替换为 K 个子 key，由写入方哈希决定（例如， counter[video=v1] 变成 counter[video=v1#bucket=0..63]）。每个子 key 独立递增。读取时对 K 个子 key 求和，可选地使用小型缓存来避免每次读取都产生扇入开销。

何时使用。 适用于读取频率低或可容忍小幅扇入的写密集型热 key：如计数器、排行榜、广告展示桶、点赞计数、浏览量计数。与 实时聚合.

代价。 读取放大：一次逻辑读变为 K 次物理读（或 K 次读取定期汇总结果）。运维层面：K 的选取需谨慎——太小则子 key 仍然过热；太大则读取开销变高。

失效场景。 强一致性读（K 个子 key 可能处于不同的写入进度，因此求和是近似的，取决于 O(K × per-bucket-batch-latency)）。读-改-写语义（例如"若值小于上限则递增"）——需要协调者，参见 强一致性下的竞争.

# Write-side sub-keying for a counter.
class ShardedCounter:
    def __init__(self, kv, num_buckets=64):
        self.kv = kv
        self.k = num_buckets

    def incr(self, key, writer_hint):
        bucket = hash(writer_hint) % self.k
        self.kv.incr(f"{key}#{bucket}")

    def value(self, key):
        # Fan-in read; cache the result with short TTL if reads are hot too.
        return sum(self.kv.get(f"{key}#{i}", 0) for i in range(self.k))

4.5 客户端热 key 检测与扇出

定义。 每个客户端（应用服务器、边缘代理）维护一个小型 sketch——带滑动窗口的 Count-Min Sketch，或 top-K 流摘要——来统计每个 key 的请求速率。当某个 key 超过阈值（例如 > 总 QPS 的 1% 或绝对值 > 1K QPS），客户端在本地将该 key 切换为"热模式"：通过 L1 缓存路由、合并请求、扇出到副本集，并发出指标，以便运维人员将该 key 提升到专用热层。

何时使用。 作为上述任何技术的前置入口——你无法对所有 key 都应用这些技术（内存开销），因此检测用于缩小范围。当热 key 是突发性的（如 Beyoncé 演出宣布）时尤为关键。

代价。 每个客户端的 sketch 内存（较小，KB 级别）。误报会浪费热路径预算；漏报会让热 key 漏网。

失效场景。 高度突发的流量场景，热 key 在 sketch 窗口内快速变化。跨客户端的协调性误检可能导致同步缓存重建。

# Sliding-window Count-Min Sketch for hot-key detection.
import collections
import time

class HotKeyDetector:
    def __init__(self, window_s=10, hot_threshold_qps=1000):
        self.window_s = window_s
        self.threshold = hot_threshold_qps
        self.buckets = collections.deque()  # (ts, key_counts)

    def observe(self, key):
        now = time.time()
        if not self.buckets or self.buckets[-1][0] != int(now):
            self.buckets.append((int(now), collections.Counter()))
        self.buckets[-1][1][key] += 1
        while self.buckets and self.buckets[0][0] < now - self.window_s:
            self.buckets.popleft()

    def is_hot(self, key):
        total = sum(b[1].get(key, 0) for b in self.buckets)
        return (total / self.window_s) >= self.threshold

4.6 专用热 key 层（运维逃生舱口）

定义。 一个独立的、内存型、读优化的存储（Redis 集群、Memcached、自定义服务），热 key 被 提升 至此处——要么在检测触发时自动提升，要么由运维人员在已知事件前手动提升。主存储仍然持有真实数据；热层是一个带有写时显式失效的读副本。有些设计将热层设计为写透（write-through）；其他设计则保持只读并异步刷新。

何时使用。 作为已知热事件的锚定策略（flash sale、票务开售、体育决赛）。也作为检测提升 key 时的运维落点。

代价。 需要运维两个存储、维护两条一致的失效路径。提升后的冷读需要一次主存储的单次访问。

失效场景。 当热 key 写入密集且一致性要求高时——双写难以实现。当提升速度慢于流量峰值时（Beyoncé 宣布演出：峰值在数秒内到来）。

4.7 边缘/CDN 缓存（热读 key）

定义。 将热读 key 推送到 CDN 边缘，设置短 TTL 和 stale-while-revalidate 语义。边缘扇出到数千个接入点 (PoP)，每个接入点服务全球流量的一部分。源站只看到 ~1 / num_edges 的流量，受 TTL 刷新频率约束。

何时使用。 适用于可公开缓存的 HTTP 响应：商品详情页、公开个人主页、病毒内容。尤其适合可容忍秒级过期的读密集场景。

代价。 每用户个性化内容会破坏它。失效速度慢（CDN 清除可能需要数分钟）。缓存键工程（哪些查询参数和请求头影响缓存命中）不简单。

失效场景。 已登录用户的个性化响应。实时新鲜度要求。缓存绕过攻击——攻击者通过改变缓存键组件来强制源站穿透（参见 缓存踩踏/读放大).

4.8 每 key 准入控制（运维兜底）

定义。 独立于以上所有方案，在边缘或负载均衡器处为每个 key 安装令牌桶限流器。令牌桶的填充速率略高于热层对该 key 的测量容量。超出部分被丢弃（返回 429/排队/显示等候室）。

何时使用。 作为始终在线的安全网。其他技术优化的是 容量；准入控制保护 可用性 在容量超出速度快于缓解响应速度时。

代价。 对用户有可见影响：被丢弃的用户看到错误或排队。运维复杂性：每 key 的限流阈值需要随热 key 集合的变化进行调整和更新。

失效场景。 当用户频繁重试时——被丢弃的负载以重试风暴的形式返回。缓解措施：使用 Retry-After 请求头和客户端指数退避。参见 背压与流量控制.

# Per-key token bucket; admit only what the hot tier can handle.
import time

class PerKeyAdmission:
    def __init__(self, capacity_per_key):
        self.cap = capacity_per_key          # key -> tokens/sec
        self.state = {}                      # key -> (tokens, last_refill)

    def admit(self, key):
        rate = self.cap.get(key, float("inf"))
        tokens, last = self.state.get(key, (rate, time.monotonic()))
        now = time.monotonic()
        tokens = min(rate, tokens + (now - last) * rate)
        if tokens >= 1:
            self.state[key] = (tokens - 1, now)
            return True
        self.state[key] = (tokens, now)
        return False                          # shed

5. 实际案例：朴素设计 → 崩溃 → 修复

CloudTrail + S3 Access Logs。 YouTube 风格的点赞计数器。目录中有 100 万个视频，但某个周二下午一个音乐视频发布，吸收了整个流量峰值。

工作负载。

  • 所有视频的写入 QPS 合计：200K 点赞/秒。
  • 分布：最热视频承接 60% 的写入 = 120K 点赞/秒落在一个 key 上。
  • 读取（在每个页面浏览时渲染"点赞数"）：总计 2M 读取/秒，最热视频承接 50% = 1M 读取/秒落在一个 key 上。
  • 后端存储：32 个分片的 KV 存储；每个分片可持续处理 30K ops/秒。

没有冲正接口；每次事故的对账成本高达五位数美元。

  • 每个视频一行： likes[video_id] = count.
  • 按 video_id 一致性哈希到 32 个分片。
  • 写入： INCR likes[video_id].
  • 读取： GET likes[video_id].

故障模式

  • 热视频落在一个分片上。该分片看到 120K 写入/秒 + 1M 读取/秒 = 1.12M ops/秒，而容量仅为 30K。
  • 结果：分片 CPU 达到 100%，队列深度持续增长，p99 延迟从 5 毫秒升至 30 秒，随后超时。
  • 故障转移没有帮助——副本成为新的热分片。
  • 其余 31 个分片空闲，各约 3K ops/秒。
  • 每分片利用率：1 个分片在 37 倍容量处，31 个分片在 10% 容量处。

优化后的设计

三层缓解方案：

  1. 子 key 拆分的写计数器。 likes[v#0..63]，从 K=64 个子 key 中按 hash(user_id) % 64选择。120K 写入/秒除以 64 = 约 1.9K 写入/秒每个子 key。子 key 本身通过一致性哈希分布到 32 个分片，因此 64 个子 key 近似均匀分布。单个分片来自该热视频的写入不超过约 4K ops/秒。
  2. 读端聚合与缓存。 一个后台汇总任务每隔 likes[v#0..63] 1 秒求和一次，并将总量写入 likes_total[v] （TTL=2 秒）。读取命中总量，此时已存入普通缓存中。
  3. L1 缓存 + 边缘 CDN 用于读取。 总量还可在 CDN 缓存，TTL=1 秒，stale-while-revalidate=5 秒。100 万次读/秒中约 99% 由边缘服务。该键的源站读取约为 1 万次/秒——完全在承受范围内。

修复后的数据

层级 热键负载 容量余量
CDN 边缘 100 万次读/秒 为此专门设计；水平扩展轻而易举。
源站读取 约 1 万次读/秒 每分片 3 万次；利用率 33%。
源站写入（每子键） 约 1,900 次写/秒 每分片利用率 6%。
最热单分片 约 6,000 次总操作/秒 利用率 20%。
其余 31 个分片 不变 约 3,000 次操作/秒，约 10%。

引入的权衡：点赞计数最多延迟约 1 秒。对产品而言可以接受。

概念上发生的变化

  • 写入被 分散 （子键化），而非 集中。热行被分解为 64 个温热行。
  • 读取被 聚合一次，多次服务 （汇总 + CDN）。"读取 64 个子键/请求"的 N×M 扇入代价由每秒 1 次承担，而非每请求 1 次。
  • 检测层不在热路径上——子键化对点赞是无条件的。检测器只在 next 未知热点模式

为何不直接"缓存总量"？

读取是廉价修复；风暴来自写端。仅靠缓存，每秒 12 万次写入仍会冲击同一分片。面试的关键信号是同时解决两个维度——读放大和写放大——分别采用独立的、分层的技术。

6. 变体及如何选择

热键问题沿两个轴划分： 读 vs 写 。 一致性容忍度。根据所在象限选择工具集。

变体 热键形态 一致性预算 主要技术 次要说明 反模式
热读，最终一致 商品页、个人主页、病毒式视频元数据 秒级 CDN/边缘 + L1 缓存 + 抖动 TTL 未命中时合并请求 对每次读取都访问源站
热读，强一致 实时库存展示 亚秒级 热键复制（M=N）+ 写扇出 检测驱动提升 单分片所有权（朴素实现）
热写，无聚合 按键审计日志追加 最终一致 子键化 / 计数器分片 异步批处理 单行单写者
热写，带读聚合 点赞 / 浏览 / 点击计数器 1-5 秒 子键化 + 定期汇总 汇总读取时用 L1 缓存 单计数器上的分布式锁
热读-修改-写，严格一致 门票座位预订、库存扣减 强一致 + 有序 单写者队列（每键）+ 协调状态机 乐观并发控制（OCC） 分散写入——破坏了不变量

第五行是陷阱。对具有"不能为负"这一不变量的计数器进行子键化，会悄然破坏正确性，因为单个子键无从知晓总量。该变体请参见 强一致性下的竞争 。 coordination-exactly-once-execution.

7. 故障模式与运维

生产故障 → 指标 → 缓解措施

故障模式 信号 / 指标 即时缓解措施 永久修复
流量峰值期间某分片 CPU 达到 100% 每分片 CPU；每键 QPS Top-K 将热键提升至专属热键层；在冷路径上启用熔断器 将该键加入静态热键列表配置；调整检测器阈值
键过期后 L1 缓存未命中风暴 源站 QPS 峰值与缓存未命中率相关 增大 TTL 抖动；验证 singleflight 合并是否已生效 添加 stale-while-revalidate；使刷新异步化
副本热键层数据分歧 每副本值差异；陈旧度指标 从主存储强制重建慢副本 验证写扇出为 fire-and-forget 或 quorum；检查复制延迟 SLI
计数器子键扇入慢 聚合计数器的读延迟；每分片 p99 提高汇总频率；缓存总量 预聚合为物化总量
检测器假阴性：热键漏过 每分片延迟尾部；热分片饱和 通过运维手册手动提升该键 降低阈值；扩大 sketch 窗口；添加服务端检测作为兜底
检测器假阳性：冷键被提升 已提升键的热键层命中率 < 50% 降级该键 添加迟滞（要求持续热度）
降级不对称：键降至阈值以下，但陈旧副本/热键层副本仍在服务读取 被降级键的写后读陈旧；副本集中值不一致 通过配置变更原子地使热键层条目失效并停止向额外副本写入 降级需要显式排空：先停止向副本写入，再在等待一个 TTL 或复制延迟窗口后停止读取；迟滞可防止抖动，但无法解决排空竞争问题
CDN 边缘在 TTL 过期后仍服务陈旧数据 CDN 命中率；新鲜度滞后 发出定向清除请求；缩短 TTL 构建快速失效路径（清除 API + 软 TTL）
级联：热键拖垮分片，再拖垮副本，然后…… 集群整体错误率 呼叫值班人员；仅对热键启用负载卸除；在边缘限速 按键限速；热键隔离；自动提升

值班事故处置手册（热键级联）

  1. 识别。 topk_by_qps 热分片的指标上。确认某一个键占主导。
  2. 隔离。 在边缘对该键应用按键限速（超出 2× capacity / num_replicas时丢弃或卸除）。其他流量正常继续。
  3. 提升。 将热键移至热键层。若写入较多，为该键启用子键化。
  4. 验证。 每分片 CPU 下降；聚合错误率下降。
  5. 事后分析输入。 该键是否在检测器的 Top-K 中但未被提升？为何？检测延迟？阈值？热键层是否为此次峰值过度配置？

8. 与其他模式的组合

热键/数据倾斜很少单独出现。最常见的组合：

  • 缓存踩踏/读放大 —— 热键的 TTL 过期，导致每个缓存节点同时向源站发起请求。每层都使用 singleflight + 抖动 TTL + stale-while-revalidate 是综合防御手段。
  • fan-out-write-amplification —— 某明星的推文有 1 亿粉丝；扇出本身就是写端的热键问题。缓解方案与写扇出相对：名人推文采用读时拉取，加上此处的工具集处理读端。
  • 强一致性下的竞争 —— Ticketmaster 座位行：热键 + "一个座位只能售出一次"的不变量。子键化会破坏该不变量；正确工具是单写者队列或乐观锁，加上上游限速。
  • 实时聚合 —— 计数器子键化产生 N 个子计数器，需要聚合；聚合层本身在峰值期间会成为热路径。
  • 背压与流量控制 —— 当缓解措施无法快速部署时，保持集群运行的唯一方式是专门对热键卸除负载，而非整个集群。
  • coordination-exactly-once-execution —— 热键 + "必须恰好执行一次"（秒杀结账）需要协调模式加上按键准入控制。
  • idempotency-exactly-once —— 热写放大重试；幂等键必须以低成本强制执行（前端用布隆过滤器，后端用去重数据库）。

9. 面试官可能追问的问题

  1. "给我算算：你的工作负载有多倾斜？" 强答案：说出分布名称（参数约为 1.0 的 Zipf 分布），给出经验法则（前 0.01% 占约 50%），选一个具体的热键（某个视频、某个商品、某个座位行），在假设均匀分区的情况下计算分片负载。如果最热的单个键占总 QPS 的比例为 f ，每分片容量配置为 total_QPS / N，则热分片运行在其配置容量的 f × N 倍。当 f = 0.5 。 N = 32时，这意味着超出容量 16 倍——具体说明了为何"增加更多分片"没有帮助。

  2. "如何在运行时检测热键？" 强答案：在每台应用服务器上使用滑动窗口 Count-Min Sketch（CMS）。Sketch 尺寸设置为假阳性率 < 1%，窗口 10 秒，每秒提取一次 Top-K。分片本身的服务端检测器作为兜底。检测器向控制平面发送事件，由控制平面决定是否提升。阈值具有迟滞：持续超过 1k QPS 达 5 秒时判定为热，低于 100 QPS 持续 60 秒时降级。

  3. "如果热键写入量大且需要强一致性，比如秒杀库存行，该怎么办？" 强答案：无法使用子键化——会破坏不变量。有两种模式可用。(a) 单写者队列：将该键的所有写入路由到一个持有租约的写者进程；写者对每批次进行一次事务提交。(b) 乐观并发：每个写者读取版本，尝试条件写入；冲突时重试。两种方式的特点：写入速率被串行化，容量为 1 / commit_latency。在良好的事务存储上，实际约为每分区 1 万-3 万 tx/秒；批量提交（将 N 次扣减作为一个事务提交）以单请求延迟为代价将吞吐量提升了批次大小倍。若要超越批量单写者的上限，库存本身必须分区（例如每分区预留 100 个座位，分区即成为较小规模的热行）。

  4. "这与一致性哈希和虚拟节点如何交互？" 强答案：虚拟节点将 中等频率 的键均匀分散。对于单个超热键，虚拟节点无济于事——该键仍然哈希到唯一一个（虚拟节点，分片）对。热键复制通过向环上的多个位置写入来打破虚拟节点的不变量，要求路由器知道哪些键是特殊的。

  5. "M=N 热键复制的代价是什么？" 强答案：写放大在 QPS方面是 N 倍，不只是存储——一个 64 分片集群上每秒 1 万次写入的热键会产生 64 万次集群范围内的写入，因此只有当该键读取量远大于写入量（读写比 ≫ N）时，该技术才值得。每个热键 N 倍存储可以忽略不计（100 个键 × 64 个分片 = 6,400 条额外记录）。写入的扇出延迟等于最慢副本。此外还有运维成本：路由器中的双重代码路径、复制延迟监控、分歧检测。该技术对于写入密集的热计数器是错误选择；应优先考虑子键化。

  6. "如何处理 也会 较大（多 MB 值）的热键？" 强答案：不要将完整值复制到每个分片。对值进行分块，热键化索引，分发数据块。或将值存储一次在对象存储中并在其前置 CDN，只对元数据指针进行热键化。或将 API 设计为在客户端从可缓存的片段物化值。

  7. "带我走一遍：碧昂丝在 10:00:00 宣布巡演，系统如何保持正常运行。" 强答案：09:58 时检测 sketch 对新事件是冷的。10:00:00 公告发布，该事件行的 QPS 在 5 秒内从 0 飙升至 50 万/秒。CMS 在 10:00:02 检测到，控制平面在 10:00:03 完成提升，热键层从 10:00:04 开始服务。这 4 秒间隙是设计上真实的暴露窗口——在此期间源站分片超过容量，因此存活依赖三件事，而非提升本身：(a) 边缘的按键准入控制，将源站负载限制在正常的 5-10 倍—— 大多数 请求收到 429 或排队页面；(b) 每个缓存层的 singleflight，以合并同一键上的并发缓存未命中；(c) 优雅降级——返回排队页面或陈旧快照，而非硬错误。诚实的表述：有了这些措施，集群保持运行，而 大多数用户 会看到 4 秒的等待页面，而非成功的低延迟读取。如果提前知道该事件，预公告静态提升可以完全消除间隙；这是能做到时的正确设计，而准入控制是无法提前预知时的兜底方案。

10. 常见陷阱 → 更优方案

陷阱 更优做法
"一致性哈希解决了所有分片问题。" 一致性哈希在假设键访问均匀的情况下均衡 键 。在 Zipf 分布下，单个键就是全部负载。虚拟节点对中等频率有帮助，对头部热键无效。
对具有硬性上限（库存）的计数器进行子键化。 单写者队列或乐观并发控制；子键化违反不变量。
静态热键列表，配置一次。 运行时检测（CMS 或 Top-K sketch）+ 动态提升/降级的控制平面。
缓存使用全局 TTL，无抖动。 TTL 加均匀抖动；stale-while-revalidate；未命中时使用 singleflight。
将所有键复制到所有分片"以防万一"。 只复制检测到的热键集；分布是不对称的（分布头部占比极小）。
依赖主存储的缓存来吸收热键。 专属热键层——主存储缓存是共享资源，会被其他流量驱逐。
没有失效策略的边缘缓存。 软 TTL + 清除 API；设计缓存键以支持部分失效。
某键热时按集群限速。 按键限速；保护冷路径流量。
"直接增加更多分片。" 增加分片无济于事——热键仍在同一个分片上。独立扩展热键层。
将热键处理视为稳态配置。 将其视为具有检测、决策、执行和回滚的控制循环。
仅在应用层使用 singleflight。 在每个缓存层（L1、L2、边缘）都使用 singleflight；缓存雪崩出现在没有去重的那一层之下。

  • 在最初 60 秒内承认了 Zipf 倾斜；用具体热键和粗略负载估算加以说明。

术语表

  • 一致性哈希（Consistent Hashing）— 在数据倾斜下失效的基线方案。
  • 虚拟节点（vnode）— 平滑中频访问，无法解决头部热点问题。
  • 偏好列表（Preference List）— 热键副本复制所覆盖的对象。
  • 仲裁（Quorum）— R + W > N 与复制热键存在交互。
  • 向量时钟（Vector Clock）— 用于复制热键接受并发写入的场景。
