扇出 / 写放大

2026/05/31 · 系统设计模式
目录
目录

一次生产者写入必须对 N 个消费者可见。该模式是选择在何处承担成本的工具集——在写入时（推送）、读取时（拉取），还是混合拆分——以及应对 N 无界的明星场景。基础术语请参见 Staff 系统设计词汇表。

以该模式为重心的核心问题文档：\n- 设计 Twitter - 设计 Twitter Timeline - 设计 Facebook 新鲜事 - 设计 Instagram - 设计消息通知系统 - 设计 Discord - 设计类似 Twitch Chat 的直播评论流服务

1. 核心难点

任何社交、消息或通知系统的朴素形态，都是单个用户操作需要反映在多处的问题。一条 Tweet 是一个用户写入的一行数据；产品承诺是多达数亿粉丝能在其时间线中看到它。一条新聊天消息是一次插入；承诺是频道的每个成员——Discord 服务器中可能有数万人，1:1 对话中可能只有两台设备——都能按序收到消息并更新未读状态。一次广告点击事件是一条记录；仪表板、计费聚合、欺诈检测器和报表索引都需要各自对其衍生视图。

在朴素设计下会被打破的不变量是 每次写入的有界工作量。如果一次生产者写入的成本是 O(N)，其中 N 是无界社交图的规模，那么任何单个热门生产者都是针对自身数据库的拒绝服务向量。一条拥有 1 亿粉丝的碧昂丝推文，加上 1KB 的时间线行，是每条推文 100GB 的写入突发。一条发给拥有 50 万成员的 Discord 服务器的公告，是 50 万行的扇出。系统的 CPU 和 I/O 预算不再由用户主导，而是由谁关注谁的 图 主导。

另一个极端——通过扫描消费者关注的所有内容从头计算读取视图——将成本转移到读取时间。这解决了明星问题，但创造了对称的读放大问题（参见兄弟模式 cache-stampede-read-amplification.md）。关注了 5000 个账号的用户现在每次刷新时间线都会触发 5000 次分区读取，而平均用户刷新的频率远高于发帖频率。两个极端都不好；Staff 级别的做法是确切知道何时使用哪种，以及如何组合它们。

该模式也关乎 将延迟与工作量解耦。写了这条 Tweet 的用户必须得到快速确认（写入其自身 outbox 必须在数十毫秒内提交）。1 亿粉丝的投递必须最终完成，具有有界的新鲜度延迟，但不必在请求路径中完成。这就是 Kafka、扇出 worker 和同步权威写入与异步衍生投影明确分离的作用。

2. 如何在面试中识别背压问题

触发该模式的提示语

  • "每个粉丝都看到这篇帖子。"
  • "时间线 / 信息流 / 收件箱。"
  • "向所有订阅者发送通知。"
  • "频道成员。"
  • "向房间内所有连接的客户端广播。"
  • "向 N 个设备推送更新。"
  • "活动流 / 最新动态。"
  • "当用户发帖时，所有朋友……"

需求描述中的架构异味

  • 生产者和消费者数量严重不对称（一个写入者，多个读取者）。
  • 粉丝 / 订阅者图具有重尾特征——少数节点的边数比中位数高出数个数量级。
  • 读取新鲜度目标在秒级（不是毫秒，也不是分钟）。
  • 产品在聚合上是读密集型的（时间线刷新次数 >> 发帖次数）。
  • 存在"未读"或"已投递回执"需求，意味着每个接收方的状态必须存储在某处。
  • 面试官提到"个性化"——每个接收方的视图不同，因此共享缓存键不起作用。

以此为核心难点的"设计 X"示例题

题目 为何扇出是核心
设计 Twitter 主页时间线是"一次写入，N 个粉丝"问题的典型代表；明星账号使朴素推送设计失效。
设计 Facebook 新鲜事 同等扇出难度，但顶部加了排序——每个朋友的帖子都必须成为你信息流的候选。
设计 Instagram 照片帖子扇出给粉丝；Stories 增加了 24 小时新鲜度窗口，改变了计算方式。
设计通知系统 一个事件（降价、朋友加入、被提及）路由到成千上万或数百万个接收方收件箱及推送渠道。
设计 Discord 一条消息进入一个频道；每个已连接的成员实时看到它，离线成员还需要回填。
设计 Twitch Chat 一条聊天消息广播给某个热门直播的 10 万以上并发观众；这是通过 pub/sub 实现的读侧扇出，而非写存储。
设计 Google News 全球聚合器 一篇摄入的文章扇出到专题版块、地区版块和个性化流。

3. 60秒口头框架

难点在于写放大：一次生产者写入必须到达 N 个消费者，而对明星用户来说 N 是无界的。我会从混合推拉模型开始。对于中位数用户——几百个粉丝——我在写入时推送，通过插入每个粉丝的时间线缓存，使读取成为单个键查找。对于超过明星阈值（比如 1 万粉丝）的用户，我不推送。他们的帖子存储在按作者划分的存储中；时间线读取将预计算的推送与从读者关注的少数明星处拉取的数据合并。扇出本身通过 Kafka 异步进行，因此生产者的写延迟与投递工作解耦，扇出 worker 按接收方分片以避免热写入者。在生产中我会关注明星阈值漂移——每月根据粉丝分布重新计算。

4. 标准工具集

4.1 推送模型（写时预计算，"写时扇出"）

定义。 当生产者写入时，同步入队一个任务，将生产的条目插入每个消费者的物化收件箱 / 时间线。读取时间变为 O(1)——从自己的收件箱中获取前 K 条。

producer writes Tweet T
        │
        ▼
   tweets table  (authoritative store)
        │
        ▼
   Kafka topic  (fan-out events)
        │
        ▼
  fan-out workers
        │
        ├── insert (follower_1, T) into timeline:follower_1
        ├── insert (follower_2, T) into timeline:follower_2
        └── ... N times

何时使用。 读密集型系统中，一次读取的成本高于一次写入 乘以读频率比。如果用户每发一条 Tweet 刷新时间线 50 次，那么当读取是 O(followees) 而写入是 O(1) 时，你的数学账已经算不过来了。

代价。 存储激增——每条 Tweet 都被复制到每个粉丝的时间线中。写扇出是每次生产 O(N)。当 N=1 亿时，一个事件需要数分钟的工作。

失效场景。 明星账号。不活跃粉丝（你扇出到两年未登录的账号）。突发帖子（一个热门事件导致所有拥有百万粉丝的人同时发帖；队列积压）。

def on_tweet(tweet):
    db.tweets.insert(tweet)               # authoritative
    kafka.publish("fanout", {
        "tweet_id": tweet.id,
        "author_id": tweet.author_id,
    })

def fanout_worker(event):
    author = event["author_id"]
    for follower_id in followers_of(author):    # may be 100M
        timeline_cache.lpush(
            f"timeline:{follower_id}",
            event["tweet_id"],
        )
        timeline_cache.ltrim(f"timeline:{follower_id}", 0, 999)

4.2 拉取模型（读时计算，"读时扇出"）

定义。 仅存储生产者自己的发件箱（自己的推文/帖子）。读取时，消费者查询每个被关注者的发件箱，按时间戳合并后返回 top K。

read timeline for U:
   followees(U) = [a, b, c, ..., m]      # may be 5000
   for each f in followees(U):
       scan top K from tweets where author = f
   merge-sort by ts, take K

何时使用。 写入密集而读取较少的系统。纯名人路径（当生产者的粉丝数量过多，不宜推送时）。冷账户（读取频率极低，按需计算比维护物化视图更经济）。

代价。 读取代价为 O(followees) 次分区读取加合并开销。尾部延迟受最慢分片限制。缓存命中率是基于作者发件箱的，而非基于读者的，因此缓存对热门生产者有利，但对关注冷门账户的读者帮助有限。

失效场景。 关注数千账户的重度用户。读取 QPS 峰值（早上 9 点所有人同时刷新）。任何需要对所有候选内容打分的按读者个性化排序（排名模型）。

4.3 推拉混合模式（Twitter"名人阈值"）

定义。 设定一个粉丝数阈值 C（例如 10,000）。低于 C 的账户正常推送。高于 C 的账户 非 不推送；其推文存储在按作者划分的名人专属存储中。读取时间线时，消费者合并（a）预推送的时间线缓存与（b）对其关注的少量名人进行拉取扇出的结果。

on tweet by author A:
   if followers(A) < C:
       push to every follower's timeline cache
   else:
       store in celebrities:{A} only (do not push)

on timeline read for U:
   cached    = read timeline:{U}                   # pre-pushed posts
   celebs    = followees(U) ∩ celebrity_set        # usually < 50
   recent    = [read celebrities:{c} top K for c in celebs]
   return merge_by_ts(cached, *recent)[:K]

何时使用。 粉丝分布呈重尾特征（这是常态）。你希望 99% 的情况走低成本路径，对那 1% 可能引发写入风暴的账户走不同路径。

代价。 两条代码路径。名人集合本身是必须维护的状态（将用户晋升到阈值以上意味着停止对其推送；降级则意味着重新开始推送）。读取路径每次请求都需进行小规模扇出。

失效场景。 一个刚好低于阈值的用户突然爆红。关注了 200 个名人的时间线读者（读取侧扇出变得不可忽视）。阈值设置与实际图结构不匹配——太低浪费读取，太高则写入风暴依然存在。

4.4 Kafka（或任意持久化日志）实现异步解耦

定义。 生产者的权威写入同步提交到事实存储。第二个事件发布到 Kafka。扇出由消费 Kafka 的工作进程完成。生产者永远不会被扇出工作阻塞。

producer ──► tweets DB  (sync, ack here)
         └─► Kafka      (async, decoupled)
                  │
                  ▼
            fan-out workers (scalable, sharded, retryable)

何时使用。 始终如此，适用于任何严肃的系统设计。即使今天的扇出很简单，解耦也能让你独立扩展工作进程、独立重试，并在不修改写入路径的情况下新增消费者（搜索索引、分析、通知）。

代价。 端到端延迟增加了数百毫秒的队列延迟。精确一次语义要求工作进程具备幂等性（参见 idempotency-exactly-once.md).

失效场景。 队列在突发负载下积压，新鲜度 SLO 被违反。工作进程在扇出中途崩溃，导致重复投递。分区分配导致某个扇出工作进程独占某名人，堵塞其分区。

4.5 按接收方分片的扇出工作进程

定义。 不按作者分配 Kafka 分区（否则 Beyoncé 的所有扇出工作都压在一个工作进程上），而是按 recipient进行扇出工作分片。扇出协调器读取作者的粉丝列表，将每个粉丝哈希到某个分区，并为每个接收方（或每个接收方分片批次）发出一个事件。

              "tweet T by author A"
                       │
                       ▼
          coordinator: load followers(A)
                       │
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
   partition 0     partition 1     partition N
   followers       followers       followers
   hash%N=0        hash%N=1        hash%N=N
       │               │               │
   worker 0        worker 1        worker N
   inserts into    inserts into    inserts into
   timelines for   timelines for   timelines for
   their recipients

何时使用。 任何推送模型，且生产者粉丝数量可能很大。按接收方分片将负载均匀分散到集群中，而不是集中在拥有作者键的工作进程上。

代价。 协调器必须读取整个粉丝列表才能分发。对于 1 亿粉丝，这本身就是一个流式操作（你不能将列表加载到内存中；需要扫描粉丝索引并批量发送事件）。

失效场景。 如果粉丝表本身不支持流式读取分片，协调器就会成为瓶颈。如果批次粒度过粗，单个接收方的投递延迟会下降。

4.6 针对 VIP 接收方的优先级队列

定义。 并非所有接收方都同等重要。作者自己的设备应最先看到帖子。认证账户可能有 SLO 要求。活跃用户（当前在 App 内）享有优先投递；离线用户则等待。

fan-out queue tiers:
  P0: self + currently-active sessions     (< 1s SLO)
  P1: recently-active followers             (< 30s SLO)
  P2: long-tail dormant followers          (best effort, minutes)

何时使用。 当不同接收方对新鲜度感知差异显著时。Twitter 的"你立刻看到自己的推文"是 P0 路径；两个月未登录的远亲是 P2。

代价。 三条队列，三组工作进程。运营复杂度上升。需要实时分类接收方（缓存"过去 24 小时内活跃"状态）。

失效场景。 分类信息已过期，刚刚打开 App 的用户被低优先级处理。P0 自我饥饿（重大事件期间所有人都是"活跃"的）。

4.7 独立名人存储，读取时合并

定义。 在推拉分离之外，将名人帖子存储在专用的、高度缓存、跨地域复制的存储中，针对扇入读取进行优化。每次时间线读取时，在消费者的物化时间线与对其关注的少量名人进行小规模扇入之间执行合并。名人存储针对约 10,000 个粉丝最多的账户进行了规模设计，并可在全球范围内完全缓存在内存中。

何时使用。 当名人长尾足够小，能够保持热缓存时。当读取时合并代价可接受时（额外几次分区读取）。

代价。 名人集合的缓存必须在全球范围内保持温热且一致。集合成员（谁是名人）会随时间变化。

失效场景。 一个非名人账户一夜爆红，尚未进入名人存储。当用户关注多位名人并希望按推文排序时，读取时的归并排序并非微不足道。

5. 实际案例：朴素设计 → 崩溃 → 修复

CloudTrail + S3 Access Logs。 一个类 Twitter 系统。每日活跃用户 5 亿。中位用户有 200 个粉丝。p99 粉丝数为 5 万。前 1,000 个账户的粉丝数在 100 万到 2 亿之间。每日帖子数：5 亿。每日时间线读取次数：500 亿（每个用户刷新约 100 次）。新鲜度 SLO：非名人帖子应在 5 秒内（p95）出现在粉丝的时间线中。

没有冲正接口；每次事故的对账成本高达五位数美元。

"每条推文发出后，同步扇出到所有粉丝。读取时，从按用户划分的时间线表中获取。"

数学计算：- 每日写入量 = 5 亿。- 平均扇出 = 200，但分布呈重尾。每日总扇出写入量 ≈ 5 亿 × E[粉丝数]。重尾情况下，E[粉丝数] 由名人帖子主导。如果 0.001% 的帖子来自拥有 1 亿粉丝的账户，那仅名人就会产生约 5,000 条帖子 × 1 亿 = 5,000 亿次粉丝写入。加上中位数用户也有 5,000 亿。- 每次写入有效载荷：100 字节时间线条目。- 物化时间线每日存储：1 万亿条 × 100B = 100TB。每年：36PB。- 同步扇出在 Beyoncé 发推时阻塞生产者数分钟——写入延迟 p99 爆炸式增长。

确切的故障模式

  • 热工作进程。 拥有 Beyoncé 的 userId 的 Kafka 分区会接收她的每一条推文；其单个扇出消费者是串行的。延迟单调增长，直至消费者崩溃。
  • 存储膨胀。 非活跃用户（6 个月内未登录）的时间线仍在持续写入。一半的存储空间服务于永远不会读取它的用户。
  • 重大事件期间的写入风暴。 某大型新闻账户在美东时间早上 9 点发布突发新闻推文，触发对 1 亿粉丝的扇出，而这些粉丝也同时在刷新——读取 QPS 与写入 QPS 同步飙升。
  • 尾部延迟。 生产者的 p99 写入延迟 producer 与扇出完成时间挂钩。一条 Beyoncé 的推文需要数分钟；用户看到的是加载旋转圈。

正确的设计

  1. 将同步生产者写入与扇出解耦。 生产者提交到 tweets 表；发送 tweet_published 到 Kafka。无论粉丝数量多少，生产者 ACK 时间均 < 50ms。
  2. 推拉混合。 每月根据粉丝数分布计算名人阈值 C，设置 C = 10,000。低于 C 的作者使用推送；等于或高于 C 的作者 非 不推送——其推文保存在 celebrity_tweets:{author_id} 。
  3. 按接收方分片的扇出工作进程。 扇出协调器分块扫描粉丝索引，并为每个接收方分片发出一个 Kafka 事件（分区键为 follower_id % N）。工作进程从其分区拉取，并仅向其负责的接收方写入时间线缓存。
  4. 优先级层级。 P0 = 作者自己的设备（立即推送）。P1 = 过去 24 小时内活跃的粉丝（5 秒内推送）。P2 = 休眠粉丝（尽力推送，可容忍最多 30 分钟延迟）。
  5. 活跃粉丝过滤器。 不向 30 天以上未登录的用户进行扇出。如果他们回来，按需从各作者发件箱计算其时间线。
  6. 读取时名人存储。 时间线读取 = （a）从缓存为读者拉取预推送的时间线，加上（b）对读者关注的每位名人（通常 < 50 人），从 celebrity_tweets:{author}拉取最近帖子。按时间戳/排名合并，返回 top K。

修复后的数据

  • 中位作者扇出：每条推文 200 次时间线插入，主要面向活跃粉丝（实际投递约 80 人）。每个事件的工作量有界。
  • 名人作者帖子：每日约 5,000 条，扇出写入为零。每条仅需一次插入到名人存储，并复制到所有地域。
  • 生产者同步写入延迟：p99 < 50ms，与粉丝数量无关。
  • 扇出工作进程延迟：P0 < 1s，P1 < 5s，P2 < 5 分钟——按层级单独度量，可配置告警。
  • 存储：仅物化活跃时间线；通过排除休眠粉丝节省约 50% 存储空间。
  • 读取路径：时间线缓存命中率约 95%；与名人存储合并增加 5-20ms 读取延迟，具体取决于读者关注的名人数量。

6. 变体及如何选择

变体 写入成本 读取成本 存储 适用场景 示例
纯推送 O(N followers) O(1) High (N × posts) 读取密集，扇出有界 有成员上限的 Discord 频道
纯拉取 O(1) O(F followees) Low (1 × posts) 写入密集、低读取或完全个性化排序 每次读取全量重排的个性化新闻
推拉混合（名人阈值） O(N) for normal, O(1) for celebs O(1) + O(few celebs) Medium 粉丝图呈重尾分布 Twitter, Instagram
仅推送给活跃用户 O(active followers) O(1) + on-demand backfill for dormant 较低 休眠用户尾部较大的应用 LinkedIn feed
配合积极缓存的拉取 O(1) O(F) but cached per-author Medium 小扇出，热门生产者 Reddit subreddit listings

决策驱动因素：\n- 粉丝分布的偏斜程度。 长尾 → 混合模式。均匀 → 推送或拉取。\n- 读写比例。 > 10:1 → 推送。< 1:1 → 拉取。否则混合。\n- 读取时个性化。 在候选集上按用户排序 → 拉取，因为无论如何都无法预计算每个读者的视图。\n- 新鲜度 SLO。 秒级 → 推送或混合。分钟级可接受 → 拉取即可。\n- 活跃用户比例。 如果只有 30% 的粉丝是活跃的，就不要向另外 70% 推送。

7. 故障模式与运维

故障模式 症状/指标 缓解措施
名人写入风暴 扇出延迟飙升；某 Kafka 分区的消费者延迟持续增长 推拉混合阈值；按接收方分片的工作进程；对每个生产者限制扇出速率
突发负载下的队列积压 Kafka 消费者延迟 > SLO 根据延迟自动扩展工作进程；将 P2 工作溢出到较慢的批处理路径
休眠粉丝浪费 存储增长速度快于 DAU 增长 活跃粉丝过滤器；为回归用户按需回填
重复投递 同一条推文在某些时间线中出现两次 按 (recipient, tweet_id) 键的幂等插入；参见 idempotency-exactly-once.md
乱序投递 较旧的推文出现在较新推文之上 在时间线缓存中按推文时间戳排序（而非插入时间）；排序使用权威时间戳
阈值配置错误 粉丝分布变化后出现延迟回归 每月重新计算名人阈值；对阈值偏移设置告警
部署后冷缓存 读取路径延迟飙升 在发布时从物化存储预热时间线缓存；逐步切换
热点读者（重度用户关注所有人） 某用户的时间线读取缓慢 在读取扇出中限制每用户的"关注数"上限；提供"列表"UI 进行分组
旧时间线条目的压缩/GC 磁盘满 固定长度时间线环形缓冲区（例如最新 1000 条）；淘汰冷数据
跨地域扇出 跨地域写入风暴 地域化扇出：每个粉丝的归属地域执行插入；仅跨地域发布/订阅帖子事件

值班故障处理手册

症状："扇出延迟告警，P1 层级，延迟 = 4 分钟。"

  1. 定位生产者群体。 查询按分区划分的延迟仪表板。如果延迟集中在少数分区，则查看其粉丝分片哈希到这些分区的作者。
  2. 检查名人绕过是否遗漏。 是否有接近阈值的作者刚刚突破阈值？如果是，手动将其标记为名人，并通过名人路径重放受影响的事件。
  3. 检查工作进程健康状况。 CPU？GC 停顿？下游时间线缓存分片过热？通常真正的瓶颈是时间线缓存分片，而非工作进程本身。
  4. 施加背压。 临时降低 P2 扇出优先级（延迟休眠粉丝的投递），为 P0/P1 释放工作进程容量。
  5. 热分片修复。 如果某个时间线缓存分片是瓶颈，进行一次性重新分片，或在其前面添加一个透明读取缓存层。
  6. 事故复盘。 更新名人阈值计算逻辑。如果某个新产品功能（例如突发新闻提及）是导致峰值的原因，将其路由到单独的扇出类别中。

8. 与其他模式的组合

扇出/写入放大问题很少单独存在。典型的问题组合：

  • hot-key-skew.md ——名人是 write 侧的典型热键；其帖子是 读取 侧的典型热键。名人存储是热键缓解方案。扇出工作进程分片是队列层的热键缓解方案。
  • idempotency-exactly-once.md ——扇出工作进程失败后会重试。向接收方时间线的插入必须是幂等的（去重键为 (recipient_id, tweet_id)).
  • cache-stampede-read-amplification.md ——时间线读取会命中缓存。重大事件后大量用户同时刷新时会发生缓存击穿。使用请求合并和 stale-while-revalidate 策略。
  • backpressure-flow-control.md ——扇出队列需要背压：当消费者延迟超过 SLO 时，系统必须限流生产者或降级处理层级，而不是盲目接受工作。
  • ordering-causal-consistency.md ——聊天消息在每个频道内必须有序到达。按频道分区进行扇出；不要对同一频道并行处理多个分区。
  • durability-at-scale.md ——权威推文存储必须是持久的。物化时间线则 非 不必如此——它们是派生的投影，可以从权威存储重建。
  • real-time-aggregation.md ——计数器（"转发数"、"点赞数"）本身也是扇出问题，写入到计数器存储中；可结合扇入聚合处理。
  • coordination-exactly-one-execution.md ——当协调器分发按接收方分片的扇出任务时，即使协调器重启，也必须保证每个 (author, tweet) 恰好分发一次。
  • search-and-ranking.md ——如果时间线是 ranked （而非纯时间顺序），候选集由扇出构建，然后在读取时按用户进行排序。排序位于扇出之上，而非取代扇出。

9. 面试官可能追问的问题

Q1：请介绍名人阈值的设计。你如何确定阈值，当某人跨越阈值时会发生什么？

强答案要点：阈值是粉丝数分布与每次扇出写入成本的函数。经验上，你会在"再多一次推送的边际成本超过读取时再多一次拉取的边际成本"的点上设定阈值。Twitter 量级下实际约为 1 万至 5 万。阈值转换是非对称的：

  • 晋升（普通用户 → 名人）。 停止推送新推文；粉丝开始通过读取时的名人拉取合并看到该用户的帖子。无需回填——较旧的推文已经被推送，并保留在粉丝的时间线缓存中。
  • 降级（名人 → 普通用户）。 微妙之处：在名人期间发布的推文 during 从未被推送过；它们只能通过名人拉取合并看到。如果在用户跌破阈值的那一刻就关闭拉取路径，这些推文就会从粉丝时间线中消失。真实系统通常通过三种方式缓解：（1）降级后将用户保留在名人拉取集合中一段宽限期（例如 30 天），使粉丝在推文自然老化超出有界时间线窗口前持续合并；（2）在降级时触发对最近 N 条名人期推文的一次性回填扇出（正确但代价高昂——本身也是一次写入风暴）；（3）对阈值增加滞后性（10,000 粉丝时晋升，只有降至 5,000 时才降级），使抖动极少发生，降级路径很少被触发。

阈值本身根据当前图结构和当前成本比例定期重新计算（每月一次）。大多数生产系统结合宽限期和滞后性，并依靠有界时间线长度来关闭一致性窗口。

Q2：你的扇出工作进程在任务中途崩溃了。共有 5000 万粉丝，已投递 2000 万。你如何恢复？

强答案要点：插入必须是幂等的（在时间线缓存层按 (recipient_id, tweet_id) 去重）。扇出任务按粉丝分片做检查点——你知道哪些分片已完成。重启时，只重新发送未完成的分片。在一个分片内，重新发送会重新插入相同的 tweet_id，幂等性检查会将其丢弃。交叉参考 idempotency-exactly-once.md.

Q3：如何在不限制作者粉丝列表大小的情况下，限制单条推文的处理工作量？

强答案要点：将写入提交与扇出完成解耦。作者的写入以 O(1) 提交——单次插入到 tweets。扇出在工作进程池中异步运行，按接收方分片，可能跨多个优先级层级。作者永远不会被自己的粉丝阻塞。有界工作是每个工作进程的属性，而非每个作者的属性。如果作者有 1 亿粉丝， system 执行 1 亿个工作单元，但没有单个组件的工作量超过其分配的份额。

Q4：如何处理用户发推后立即刷新自己的时间线却看不到的情况？

强答案要点：自我可见性是 P0 路径。发帖时，写入（a）权威推文表，（b）在扇出启动之前立即插入作者自己的时间线缓存，（c）发送扇出事件。作者通过路径（b）在写入后数毫秒内看到自己的推文。读己所写是独立于通用新鲜度的一个单独关注点。

Q5：为什么不在读取时完成所有工作，完全避免写入风暴？

强答案要点：在社交 Feed 产品中，读取流量是写入流量的 50-100 倍。对于展示 50 条帖子的时间顺序 Feed，朴素拉取在 p50 下实际上是可行的——50 次内存缓存读取可在个位数毫秒内完成。推送的复杂度在三种特定场景下才值得：（1）排序 Feed 场景，对 5000 个关注对象进行候选生成会在每次刷新时重新打分；（2）严格的 p99.9 SLO 场景，scatter-gather 的尾部延迟成为问题；（3）极端刷新 QPS 场景，每次刷新的读取预算超过缓存容量。诚实的高级工程师答案是：拉取对基础时间顺序 Feed 已经足够；推送是排序/尾部延迟/QPS 的优化手段，而非"粉丝多"的通用解决方案。

Q6：如果帖子可以被删除，你的设计会如何变化？

强答案要点：删除是第二个扇出问题——也必须到达每个接收方。两种常见方式：（a）懒删除——在时间线中保留 tweet_id，但在权威存储中将其标记为已删除；读取路径过滤掉墓碑 tweet_id；（b）急删除——向相同的扇出工作进程发送删除事件，从每个接收方的时间线中移除。懒删除成本更低、更简单；急删除移除更快，但扇出成本翻倍。对于隐私敏感的删除（例如已撤销的帖子），急删除是必需的，并受 SLO 约束。

10. 常见陷阱 → 更优方案

陷阱 更优做法
纯推送，无名人分流。 基于粉丝分布计算阈值的推拉混合。
在生产者写入路径中同步扇出。 权威写入同步提交；扇出在 Kafka 后异步运行。
按作者分区扇出工作。 按接收方分区——名人作者不再固定占用某一工作进程。
平等对待所有粉丝。 优先级层级（P0 本人，P1 活跃，P2 休眠），各层级有独立 SLO。
向所有粉丝扇出，包括休眠用户。 活跃粉丝过滤器；为回归用户按需回填。
每次读取时从所有被关注者拉取内容。 缓存时间线；读取时仅拉取名人的尾部数据。
所有扇出使用单一 Kafka topic。 按优先级层级划分 topic；按扇出类型划分 topic（时间线 vs 通知 vs 搜索索引）。
失败时重试，不做去重处理。 以 (recipient_id, post_id) 为键的幂等接收方插入操作。
永久物化时间线。 保存最新 N 条的环形缓冲区；旧帖子按需从权威存储中拉取。
所有功能使用相同的全局阈值。 按功能设置阈值——时间线、通知和搜索可能各需不同的分割方式。
忽略跨区域问题。 将扇出区域化——只有帖子事件跨区域传播；每个接收方的插入操作保持在本地。
"用 Redis 就行了。" 根据访问模式选择存储层：热门时间线用内存缓存，名人存储用持久化 KV，权威推文用日志。
