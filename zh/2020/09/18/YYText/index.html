从结构上看，YYText 与 TextKit 的类边界相同——两者均为 MVC 架构，唯一区别在于 YYText 缺少 NSTextStorage 那种跨多个布局和容器复用 attributed string 的能力。NSTextStorage 是 NSAttributedString 的子类；在我看来，storage 的存在是为了扩展字符串的复用能力，使其可在多个布局和容器之间共享。

另一个结构差异：在 TextKit 中，一个 NSLayoutManager 可以管理多个 NSTextContainer；而一个 YYTextLayout 只对应一个 YYTextContainer。我猜测原因是 YYText 仅支持 iOS，不存在一个布局驱动多个容器的场景。

YYText 与 CoreText 的交互

整体流程：

  1. 首先，配置你的 NSAttributedString。
  2. 用它创建一个 CTFramesetter。
  3. 向 CTFramesetter 提供一个 CGPath，以生成 CTFrame。
  4. CTFrame 包含多个 CTLine。
  5. 每个 CTLine 包含其他 CTLine 或 CTRun。
  6. 调用 Core Graphics 渲染 CTRun。

涉及的 CoreText 概念有：CTFramesetter、CTFrame、CTLine 和 CTRun。

我们先从 CTLine 入手，了解 CTRun 的渲染流程，再回头看 CTLine 的布局方式。

CTLine & YYTextLine

YYText 将 CTLine 封装为 YYTextLine 类，主要负责：

  • 使用 YYTextLine 自身的 isVertical 属性，根据 CTLine 计算布局：
    • 基线位置
    • 边界（Bounds）
    • 上升量（Ascent）
    • 下降量（Descent）
    • 去除尾部空白后的宽度
  • 添加 YYTextAttachment

定位 attachments：

attachment 没有字体来驱动其位置，我们必须手动提供其 ascent、descent 和 leading。

对于包含文本的 CTRun，CTRunDraw 会忽略 baselineOffset。这就是为什么 YYText 不支持 baselineOffset——UILabel 支持它，推测是通过提取字形并在调用 setTextPosition 时手动添加偏移量来实现的。

但对于包含 attachment 的 CTRun，我们拥有完全的布局控制权，因为 attachment 本质上只是一个待放置的矩形，而 CTRun 已经位于 CTLine 内部——我们自己控制矩形的位置。

在文字和图片混排的默认布局中，图片和文字顶部对齐——即 YYTextAttachmentVerticalAlignTop。为了支持居中和底部对齐，我们重新设置基线，使每个 attachment 可以拥有自己的垂直属性。内边距（Insets）同样有效。

该 API 如何使用字体和对齐模式来设置 ascent 和 descent：

顶部对齐情况：

取字体的 ascender；attachment 的多余高度计入 descent。

如果 descent 为负，则截断到 0；将 ascent 设为高度。

注意：如果 attachment 比字体的 ascent 更高，我们使用字体的 ascent 作为图片的 ascent，将剩余部分放入 descent。如何填充 descent 并不重要——这只是一种定义，只要 bounds 计算正确即可。

如果 attachment 比字体的 ascent 更矮，由于 descent 不能为负，我们将 descent 设为 0，ascent 设为 attachment 高度——等价于将位置置于左下角。

如果把 YYTextAttachment 当作普通文本处理，你会发现它缺少基线处理，但实际上 attachment 并不需要基线的概念。

CTRun

经过上述 attachment 布局分析，可以清楚地看出：CTRun 是一个字符范围，其中所有字符共享相同的属性。YYText 不对 CTRun 进行封装，因为其接口已足够简单。

主要 API：

CTRunGetGlyphCount — 获取 run 中的字形数量。

CTRunGetAttributes — 返回用于创建字形 run 的属性字典。

CTRunGetStatus — 了解方向和排序有助于字符串索引分析；了解位置是否引用 identity 文字矩阵可以避免昂贵的比较操作。

CTRunGetGlyphs — 获取字形。

CTRunGetPositions — 获取每个字形的位置。

CTRunGetAdvancesPtr — 获取每个字形的 advance。

CTRunGetTypographicBounds — 获取该 run 的 ascent、descent、leading、width。

CTRunDraw — 绘制一个 run。文档指出这是一个便捷方法：你也可以逐个获取字形并单独渲染。此操作完成后，context 可能处于任意状态，不会被重置。

注意：YYText 支持的 CoreText 属性恰好就是 CTRunDraw 所支持的属性。对于 CTRunDraw 不支持的属性（如 baselineOffset），需要手动逐个获取字形并自行布局。CTRunDraw 本身也不原生支持垂直布局——这就是为什么 YYText 对于垂直布局只手动逐个布局和渲染字形，并且不支持 baselineOffset。

CTFrame

一个 frame 包含多个 line。CTFrame 是 CTFramesetter 的文本排版结果。

你可以通过 Core Graphics 接口直接绘制一个 CTFrame。CTFrame 包含一个 CTLine 数组；你可以逐行获取字形信息，并单独渲染每一行。

常用 API：

YYText 异步渲染

YYText 的另一个广受赞誉的特性是异步渲染。 Jianshu 上有一篇很好的介绍文章。 TLDR：

有一个函数 YYAsyncLayerGetDisplayQueue 用于决定将渲染任务异步提交到哪个队列。内部维护一个与 CPU 核心数等长的队列数组，每个队列均为 串行队列。如果引入了 YYKit 的 QoS 文件，可以在 Release 构建中启用 QoS。

YYTextSentinel *sentinel 是 YYLabel 的一个属性，用作计数器。每次更新都会调用 setNeedsDisplay 或以其他方式取消现有的显示任务，并递增该标记值。在异步工作中，在关键步骤之前，会将当前值与任务启动时保存的副本进行比较——相等表示没有新任务或取消，不相等表示放弃当前工作。关键步骤包括：创建 CGContext、关闭 CGContext、设置 layer.contents、从 context 获取 UIImage。

YYTextAsyncLayerDisplayTask 有三个 block：

willDisplay — 清理；检查并对比 attachments。

Display — 执行主要文本绘制工作。

  1. 检查是否已取消。
  2. 检查布局是否需要更新——重建布局；是否应收缩布局。
  3. 根据 YYTextVerticalAlignment 调整渲染图片的位置。
  4. 使用指定大小和位置在 context 中绘制。

didDisplay — 更新 YYLabel 属性。

  1. 如果收缩，赋值 shrinkLayout。
  2. 如果此渲染任务已被取消，清除所有 attachments。
  3. 添加淡入/淡出的 CATransition 动画。

实践中，"异步"帮助并不大：YYText 本身无法处理非常繁重的任务（受限于 CoreText），而对于轻量级任务，异步与同步的差异很小。另外记得禁用淡入动画，否则可能出现闪烁。
