[归档] LLDB — 由 LLVM 驱动的调试器。

之前曾有过 LLDB 的简短介绍，主要涵盖 Xcode 已提供的一些高效断点方法（符号断点、objc-exception 断点等）以及 image.

要学习 LLDB，应先了解 LLVM 的结构：

LLVM IR 是一种完整的代码表示形式。它也是 LLVM 优化器的唯一接口，因此 LLVM 前端本质上等价于 LLVM IR 是什么、LLVM IR 如何工作，以及 LLVM IR 的规则。

LLVM 是一组库的集合。与 GCC（单一整体）或 JVM / .NET（封闭不透明）不同，LLVM 被设计为一组库的集合，这使我们可以利用其丰富的接口构建新工具。

LLDB 本质上是构建在 LLVM 之上的项目。它使用 LLVM IR（Clang）作为解析器来解释命令，使用 LLVM JIT 将其转换为目标代码，并使用 LLVM 反汇编器 / LLVM Target 处理调用约定。

Target 命令主要用于解析符号。

对于动态库（iOS 上的 framework，macOS 上的 dylib），加载时有一个重定基址步骤，但符号在库内的偏移量不变。理论上，只要将 dylib 重定基址到正确地址，就能从崩溃日志中的崩溃地址恢复源代码位置。

LLDB 允许我们通过偏移量查找代码位置。

假设我们有一个包含三个 dylib 的崩溃日志，崩溃日志中有如下 Binary Images：

0x100000000 - 0x100000ff7 <A866975B-CA1E-3649-98D0-6C5FAA444ECF> /tmp/a.out

0x7fff83f32000 - 0x7fff83ffefe7 <8CBCF9B9-EBB7-365E-A3FF-2F3850763C6B> /usr/lib/system/libsystem_c.dylib

0x7fff883db000 - 0x7fff883e3ff7 <62AA0B84-188A-348B-8F9E-3E2DB08DB93C> /usr/lib/system/libsystem_dnssd.dylib

0x7fff8c0dc000 - 0x7fff8c0f7ff7 <C0535565-35D1-31A7-A744-63D9F10F12A4> /usr/lib/system/libsystem_kernel.dylib

libsystem_c.dylib 的基地址为 0x7fff83f32000，其余类似。

如何将 0x10000ff7 映射回源代码？

首先使用

(lldb) target create --no-dependents --arch x86_64 /tmp/a.out
(lldb) target modules add /usr/lib/system/libsystem_c.dylib
(lldb) target modules add /usr/lib/system/libsystem_dnssd.dylib
(lldb) target modules add /usr/lib/system/libsystem_kernel.dylib

创建目标并添加这三个 dylib。 a.out 使用

(lldb) target modules load --file a.out 0x100000000
(lldb) target modules load --file libsystem_c.dylib 0x7fff83f32000
(lldb) target modules load --file libsystem_dnssd.dylib 0x7fff883db000
(lldb) target modules load --file libsystem_kernel.dylib 0x7fff8c0dc000

模拟程序启动时会进行的重定基址操作。 target modules load 完成。

(lldb) image lookup -a 0x00007fff8a1e6d46
(lldb) image lookup -a 0x00007fff84597df0
(lldb) image lookup -a 0x00007fff84598e2a
(lldb) image lookup -a 0x0000000100000f46

Breakpoint: [br]

基于方法触发的断点。

Breakpoint set --name mainMemory br s -n main

--file test.c --line 12Memory br s -f test.c -l 12

--method mainMemory br s -m main。仅限 C++。

--name "[NSString stringWithFormat:]" — 按名称设置符号断点。

--selector <selector-name> — 按 selector 设置符号断点。

--condition：设置断点条件。

-E <source-language>: -E objc 设置 objc 异常断点； objc 可以替换为 c等。

-F <full-name>：在 C++ 中为命名空间加全部参数；在 objc 中为包含类名和 selector 的完整函数原型。

-t <thread-id>：断点仅对 TID 匹配的线程生效。

-s <shlib-name>：仅在指定动态库上设置符号断点。

-r <regular-expression>：使用正则表达式断点，例如 br s -r cellforitem.

断点命令：

Add <index of breakpoint>: breakpoint add 2 — 为第 2 号断点附加停止时的指令。

断点修改：

-G：自动继续。

-T <thread-name>：断点仅对名称匹配的线程生效。

-q <queue-name>：断点仅对指定队列中的线程生效。

Image: [im]

Image list：列出主可执行文件及所有依赖的共享库。

Image lookup --address：按地址查找信息。

Image lookup -r -n <runc_regex>：调试符号的正则查找。

Image lookup -r -s <runc_regex>：非调试符号的正则查找。

Image lookup --type <some type>：按类型名称查找信息。

(lldb) im loo --type xxCell — 在以下位置找到最佳匹配 /Users/folobe/Library/Developer/Xcode/DerivedData/xxxxxx

Image dump section <module-name>：导出模块信息。

Image dump symtab：导出主可执行文件及所有共享库的所有符号。
